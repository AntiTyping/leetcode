R1. Type-Safe Error Boundaries Runtime Type Patterns 17 days since review · rep 1 → 2
Instead of letting arbitrary throws propagate as unknown, model failure as data: a Result<T, E> (a discriminated union { ok: true; value: T } | { ok: false; error: E }) makes the error type part of the signature. A boundary function wraps risky work, catches once, and returns a typed Result so callers must handle both arms — the compiler stops them from using a value that might not exist.
Key Pattern 
type Result<T, E = Error> = { ok: true; value: T } | { ok: false; error: E };

async function attempt<T>(fn: () => Promise<T>): Promise<Result<T>> {
  try { return { ok: true, value: await fn() }; }
  catch (e) { return { ok: false, error: e instanceof Error ? e : new Error(String(e)) }; }
}

const r = await attempt(() => loadUser(id));
if (!r.ok) return respond(500, r.error.message);
console.log(r.value.name);   // r.value only exists on the ok branch
the union forces both arms; r.value is unreachable until you narrow on r.ok
Challenge Exercise

Give attempt a typed error channel: attempt<T, E>(fn, mapErr: (e: unknown) => E) so a caller can turn caught unknown into a domain error union (NotFound | Forbidden). Then write unwrapOr(result, fallback) and explain why returning a Result beats throwing when the caller must react to specific error kinds.

Pitfall: Typing a catch binding as Error — in TS it is unknown (or any pre-useUnknownInCatchVariables), because anything can be thrown. Narrow with e instanceof Error before touching .message, and never let the raw unknown leak past the boundary.
R2. Type-Safe Plugin Systems Architecture Patterns 17 days since review · rep 1 → 2
A plugin host exposes a typed hooks contract; each plugin declares which hooks it implements. Model the hook registry as an interface mapping hook name → handler signature, then a Plugin is { [K in keyof Hooks]?: Hooks[K] }. The host invokes hooks by key with typed args, so a plugin can’t register a handler with the wrong shape or listen for a hook that doesn’t exist.
Key Pattern 
interface Hooks {
  onRequest: (ctx: { url: string }) => void | Promise<void>;
  transformResponse: (body: string) => string;
}
type Plugin = { name: string } & { [K in keyof Hooks]?: Hooks[K] };

function run<K extends keyof Hooks>(plugins: Plugin[], hook: K, ...args: Parameters<Hooks[K]>) {
  for (const p of plugins) p[hook]?.(...(args as never));
}

const logger: Plugin = { name: "log", onRequest: (c) => console.log(c.url) };
// const bad: Plugin = { name: "x", onRequest: (c) => c.method };  // ✗ 'method' missing
each hook handler is pinned to Hooks[K]; unknown hook names won’t compile
Challenge Exercise

Add a hook whose handlers reduce a value in a pipeline (each receives the previous return), and type run so transformResponse plugins chain: string → string → string. Then let a plugin declare typed options via a generic Plugin<TOptions> and explain how keyof Hooks keeps the host and plugins in lockstep as hooks are added.

Pitfall: Typing the registry as Record<string, Function> — you lose per-hook argument checking and typos in hook names compile fine. Drive everything off one Hooks interface with keyof and Parameters<Hooks[K]> so adding a hook automatically updates every call site’s types.
R3. Type-Safe Internationalization Architecture Patterns 17 days since review · rep 1 → 2
Derive the set of valid message keys from the translation object with keyof typeof messages, and extract each key’s named placeholders from its template string with a template-literal type. Then t(key, params) both rejects unknown keys and requires exactly the interpolation params that key’s string mentions — a missing or extra variable is a compile error.
Key Pattern 
const en = { greeting: "Hi {name}", cart: "{count} items" } as const;
type Key = keyof typeof en;

// pull "{x}" placeholders out of a template into a params object type
type Params<S extends string> =
  S extends `${string}{${infer P}}${infer Rest}` ? { [K in P | keyof Params<Rest>]: string } : {};

function t<K extends Key>(key: K, params: Params<typeof en[K]>): string { /* interpolate */ return ""; }

t("greeting", { name: "Ada" });      // ✓
// t("greeting", {});                 // ✗ missing 'name'
// t("nope", {});                     // ✗ 'nope' is not a Key
placeholders are parsed from the string; params are required and checked per key
Challenge Exercise

Add plural support: a key whose value is { one: string; other: string } and a t overload that requires a count param. Then enforce at the type level that a second locale (de) has exactly the same keys as en using a Record<Key, string> constraint, so a missing translation fails the build.

Pitfall: Losing the literal key/placeholder types by not using as const — without it, messages widens to Record<string, string>, keyof collapses to string, and the placeholder parsing has nothing to read. The whole safety net depends on preserving the literal template strings.
R4. Mapped Type Key Remapping Advanced Type System 14 days since review · rep 1 → 2
The as clause in a mapped type lets you rename or filter keys while mapping: { [K in keyof T as NewKey]: ... }. Combine it with template-literal types to build getter names (getName), and remap a key to never to drop it entirely — the standard idiom for filtering properties by value type.
Key Pattern 
// rename each key to a getter method
type Getters<T> = { [K in keyof T & string as `get${Capitalize<K>}`]: () => T[K] };
type G = Getters<{ name: string; age: number }>;
// { getName: () => string; getAge: () => number }

// filter out function-valued props by remapping them to never
type DataOnly<T> = { [K in keyof T as T[K] extends Function ? never : K]: T[K] };
type D = DataOnly<{ id: number; save(): void }>;   // { id: number }
`as` renames via template literals; remapping to never removes a key
Challenge Exercise

Write RenameKey<T, From extends keyof T, To extends string> that renames exactly one key while leaving the rest. Then build EventHandlers<T> that maps { click: MouseEvent } to { onClick: (e: MouseEvent) => void }, and explain why K in keyof T & string is needed before Capitalize (symbol/number keys can’t be templated).

Pitfall: Applying Capitalize/template literals to keyof T directly — keys can be string | number | symbol, and template literals only accept string. Intersect with & string first. Also remember remapping to never removes the key; it doesn’t set its type to never.
R5. Type-Safe pipe() & compose() Functional Type Patterns 14 days since review · rep 1 → 2
A well-typed pipe threads the output of each function into the input of the next, so the chain’s overall type is inferred and a type mismatch between adjacent steps is a compile error. The classic implementation uses overloads per arity (2, 3, 4… functions), each pinning the intermediate types; compose is the same idea right-to-left.
Key Pattern 
function pipe<A, B>(a: A, ab: (a: A) => B): B;
function pipe<A, B, C>(a: A, ab: (a: A) => B, bc: (b: B) => C): C;
function pipe<A, B, C, D>(a: A, ab: (a: A) => B, bc: (b: B) => C, cd: (c: C) => D): D;
function pipe(a: unknown, ...fns: Array<(x: unknown) => unknown>) {
  return fns.reduce((acc, fn) => fn(acc), a);
}

const out = pipe(5, (n) => n * 2, (n) => `#${n}`, (s) => s.length);  // out: number
// pipe(5, (n) => n * 2, (s: string) => s.length);   // ✗ number not assignable to string
each overload chains B→C→D; a mismatched step breaks the chain at compile time
Challenge Exercise

Replace the overloads with a single variadic-tuple signature: pipe<T, Fns extends readonly Fn[]>(x: T, ...fns: Fns) using a recursive conditional type to validate each adjacency and infer the final output. Then write a point-free flow (compose-left that returns a function) and explain the tradeoff between the overload approach and the variadic-tuple approach (readability vs unbounded arity).

Pitfall: Typing fns as Array<(x: any) => any> and calling it a day — the chain compiles but every intermediate is any, so a broken step is silent. The type safety lives entirely in the per-arity overloads (or the variadic-tuple recursion); the implementation signature stays loose but is never what callers see.
R6. Drizzle ORM — SQL-First Type Safety Database Typing 13 days since review · rep 1 → 2
Drizzle infers types from the schema you declare in TS (no codegen step): the table definition is the source of truth, and $inferSelect/$inferInsert give the row and insert types. Its query builder mirrors SQL, so db.select().from(users).where(eq(users.id, id)) returns a fully typed row array — you write SQL-shaped code and get inference for free.
Key Pattern 
import { pgTable, serial, text } from "drizzle-orm/pg-core";
import { eq } from "drizzle-orm";

const users = pgTable("users", {
  id: serial("id").primaryKey(),
  email: text("email").notNull(),
});
type User = typeof users.$inferSelect;      // { id: number; email: string }
type NewUser = typeof users.$inferInsert;   // { id?: number; email: string }

const rows = await db.select().from(users).where(eq(users.id, 1));  // User[]
await db.insert(users).values({ email: "a@b.io" } satisfies NewUser);
$inferSelect/$inferInsert derive row + insert types straight from the schema
Challenge Exercise

Add a posts table with an authorId foreign key, then write a typed join (db.select().from(posts).innerJoin(users, eq(posts.authorId, users.id))) and inspect the inferred result shape (nested per-table). Explain how Drizzle differs from Prisma (schema-in-TS + SQL builder vs a .prisma file + generated client) and when the SQL-first model is easier to reason about.

Pitfall: Hand-writing a User interface next to the table — it drifts from the schema. Always derive with $inferSelect/$inferInsert. And note insert types differ from select types (defaulted/serial columns are optional on insert) — using the select type for values() forces you to supply columns the DB generates.

1. CORS & Security Headers Typing Node.js + TypeScript
Your API’s first line of defense is the browser policy it advertises: which origins may call it (CORS) and which protections the browser must enforce (CSP, HSTS, frame options). A string-typed allow-list invites typos and an accidental "*" that opens the door to every origin. Typing the config — a literal union of allowed origins, exhaustive header keys — turns a security policy into something the compiler checks.

Model the allowed origins as a readonly tuple (a literal union, not string[]) so an origin either is one of them or is rejected; the cors package’s origin callback narrows to that union. For response headers, declare a typed record whose keys are the exact header names you set, so a mistyped "Content-Security-Poticy" is a build error rather than a silently-ignored header.

helmet sets a sane baseline (HSTS, X-Content-Type-Options, frameguard, a starter CSP); you then tighten the CSP with a typed directive object. Keep the origin logic a pure, testable function so it can be unit-tested with the factories from 07-05.

1
Declare allowed origins as a readonly literal tuple → derive the union.
2
Type the CORS origin callback so only known origins pass.
3
Layer helmet for the baseline; type the CSP directive object.
TypeScript · example 1 of 4 — origins as a literal union, not string[]
const ALLOWED = ["https://app.acme.io", "https://admin.acme.io"] as const;
type AllowedOrigin = typeof ALLOWED[number];   // "https://app.acme.io" | "https://admin.acme.io"

function isAllowed(origin: string | undefined): origin is AllowedOrigin {
  return !!origin && (ALLOWED as readonly string[]).includes(origin);
}
the type predicate narrows a raw string to a known-origin union
TypeScript · example 2 of 4 — the cors origin callback, typed
import cors, { type CorsOptions } from "cors";

const options: CorsOptions = {
  origin(origin, cb) {
    if (origin === undefined || isAllowed(origin)) cb(null, true);   // same-origin/allowed
    else cb(new Error(`origin not allowed: ${origin}`));
  },
  credentials: true,
};
app.use(cors(options));   // reflecting a validated origin — never a blanket "*" with credentials
isAllowed gates the callback; "*" + credentials is impossible to reach here
TypeScript · example 3 of 4 — a typed security-headers record
type SecurityHeader =
  | "Strict-Transport-Security"
  | "X-Content-Type-Options"
  | "Referrer-Policy";

const headers: Record<SecurityHeader, string> = {
  "Strict-Transport-Security": "max-age=63072000; includeSubDomains",
  "X-Content-Type-Options": "nosniff",
  "Referrer-Policy": "strict-origin-when-cross-origin",
  // "X-Frmae-Options": "DENY",   // ✗ not a SecurityHeader — typo caught
};
the Record key type rejects a misspelled header name at compile time
TypeScript · example 4 of 4 — helmet baseline + typed CSP directives
import helmet from "helmet";

app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "https://cdn.acme.io"],
      imgSrc: ["'self'", "data:"],
    },
  },
}));   // directive keys/values are typed by helmet's ContentSecurityPolicyOptions
helmet supplies HSTS/nosniff/frameguard defaults; the CSP object is typed
Common Mistakes:
❌ origin: "*" together with credentials: true — browsers reject it, and a wildcard defeats the point of an allow-list.
✅ Reflect a validated origin from a typed union; only then send credentials.
❌ Setting headers via bare string keys (res.setHeader("X-Frmae-Options", ...)) — a typo is a header that never applies.
✅ Constrain header names to a literal union / Record<SecurityHeader, string> so typos fail the build.
Key Rules:
  • Declare allowed origins as a readonly literal tuple and derive the union with typeof arr[number].
  • Gate the CORS origin callback with a type predicate; never pair "*" with credentials.
  • Type response-header names as a literal union so a misspelled header is a compile error.
  • Use helmet for the baseline and tighten the CSP through its typed directive object.
Practice Exercises:
1Load ALLOWED origins from env via the type-safe config pattern (07-05 review) and keep the union by validating against a fixed literal set.
2Write a unit test (Supertest, 07-05) asserting a disallowed Origin header gets no Access-Control-Allow-Origin in the response.
3Type a cspDirectives helper as Record<"defaultSrc" | "scriptSrc" | "imgSrc", string[]> and feed it into helmet.
Connections: The literal-union-from-as const trick is the same one behind type-safe config and i18n keys (R3). The origin is AllowedOrigin predicate is type-predicate narrowing, and layering helmet as Express middleware reuses the handler/middleware typing from 06-30.
2. Input Sanitization & Output Encoding Node.js + TypeScript
Validation (Zod) proves data has the right shape; it does not prove a string is safe to render as HTML or embed in a query. XSS and injection live in that gap. The type-system move is a branded type: a SafeHtml or Sanitized string you can only produce by passing through a sanitizer, so the compiler guarantees no raw user string reaches a sink unescaped.

Brand the trusted output: type SafeHtml = string & { readonly __safe: unique symbol }. The only way to mint a SafeHtml is sanitize(dirty) (backed by DOMPurify or an escaper), so any function that renders HTML can demand SafeHtml and a raw string simply won’t type-check. This pushes the “did you sanitize?” question from code review into the compiler.

Distinguish the two jobs: sanitize when you must keep some markup (rich text), encode/escape when the value is pure data going into HTML/attribute/URL context. For SQL, the branded-string discipline is a backstop — the real fix is parameterized queries (never string-concatenate user input).

1
Define a branded SafeHtml that only sanitize() can construct.
2
Type render sinks to accept SafeHtml, not raw string.
3
Encode by context (HTML/attr/URL); parameterize all SQL.
TypeScript · example 1 of 4 — a branded SafeHtml the sanitizer alone mints
declare const _safe: unique symbol;
type SafeHtml = string & { readonly [_safe]: true };

import DOMPurify from "isomorphic-dompurify";
function sanitize(dirty: string): SafeHtml {
  return DOMPurify.sanitize(dirty) as SafeHtml;   // the one blessed constructor
}
SafeHtml is a nominal brand; no plain string is assignable to it
TypeScript · example 2 of 4 — sinks demand SafeHtml, not string
function renderComment(html: SafeHtml): string {
  return `<div class="comment">${html}</div>`;
}

const dirty: string = req.body.comment;    // untrusted
// renderComment(dirty);                    // ✗ string not assignable to SafeHtml
renderComment(sanitize(dirty));             // ✓ only sanitized markup reaches the sink
the compiler forces every value into sanitize() before it can be rendered
TypeScript · example 3 of 4 — encode by context (data, not markup)
const escapeHtml = (s: string): string =>
  s.replace(/[&<>"']/g, (c) => (
    { "&": "&amp;", "<": "&lt;", ">": "&gt;", '"': "&quot;", "'": "&#39;" }[c]!
  ));

// data into HTML text: ESCAPE (don't sanitize — you want it literal)
const label = escapeHtml(user.displayName);
// value into a URL: encodeURIComponent, not escapeHtml
encoding is context-specific; escaping ≠ sanitizing rich markup
TypeScript · example 4 of 4 — SQL: parameterize, never concatenate
// ✗ injection: a user 'name' of "'; DROP TABLE users;--" is now SQL
// db.query(`SELECT * FROM users WHERE name = '${name}'`);

// ✓ parameterized — the driver binds values; input can't become SQL
await db.query("SELECT * FROM users WHERE name = $1", [name]);
bound parameters keep user input as data; the brand is a backstop, not the fix
Common Mistakes:
❌ Treating Zod-validated as safe-to-render — z.string() proves it’s a string, not that it’s free of <script>.
✅ Brand rendered output as SafeHtml so validation and sanitization are separate, both-required steps.
❌ Sanitizing when you meant to encode (or vice-versa) — escaping rich text mangles it; sanitizing plain data is pointless.
✅ Encode data by output context (HTML/attr/URL); sanitize only when you must preserve markup.
Key Rules:
  • Make the sanitizer the only constructor of a branded SafeHtml; type render sinks to require it.
  • Validation (shape) and sanitization (safety) are distinct — do both.
  • Encode/escape by output context; don’t reach for a full sanitizer on pure data.
  • Always parameterize SQL; string interpolation of user input is the injection.
Practice Exercises:
1Add a SafeUrl brand minted only by a validator that rejects javascript: schemes, and require it on an <a href> builder.
2Write a Zod .transform that runs sanitize so the parsed output type is SafeHtml end-to-end.
3Prove the brand works: try to pass a raw request string to renderComment and read the exact compiler error.
Connections: The SafeHtml brand is branded/opaque types applied to security; the Zod .transform exercise reuses runtime validation from earlier days. Parameterized queries connect to the Drizzle/SQL-builder review (R6), where the builder parameterizes for you.
3. Idempotency Keys for Safe Retries Node.js + TypeScript
Networks retry: a client that times out on POST /payments may resend and double-charge. An idempotency key — a client-supplied unique id per logical operation — lets the server recognize a replay and return the original result instead of doing the work twice. Typing the stored record (key → status + cached response) makes the replay path explicit and keeps the cached body’s type honest.

The client sends an Idempotency-Key header (a UUID) with each mutating request. On arrival the server atomically reserves the key: if new, mark it in_progress and do the work; if it already exists, return the stored response (or a 409 if still processing). Model the stored record as a discriminated union on status so an in_progress record has no response and a completed one carries a typed, cached body.

Two subtleties: scope the key by user + endpoint so keys can’t collide across accounts, and reserve atomically (a unique DB constraint or Redis SET NX) so two concurrent replays don’t both execute.

1
Read a typed Idempotency-Key; validate it’s a UUID.
2
Atomically reserve the key (unique constraint / SET NX).
3
Replay: return the cached typed response; else run and store it.
TypeScript · example 1 of 4 — the stored record as a discriminated union
type IdempotencyRecord<T> =
  | { status: "in_progress"; key: string }
  | { status: "completed"; key: string; statusCode: number; response: T };

// only the completed arm carries a response — the compiler enforces it
function isDone<T>(r: IdempotencyRecord<T>): r is Extract<IdempotencyRecord<T>, { status: "completed" }> {
  return r.status === "completed";
}
an in_progress record has no `response`; narrowing on status reveals it
TypeScript · example 2 of 4 — a typed store interface
interface IdempotencyStore {
  reserve(key: string): Promise<"reserved" | "exists">;   // atomic: SET NX / unique insert
  get<T>(key: string): Promise<IdempotencyRecord<T> | null>;
  complete<T>(key: string, statusCode: number, response: T): Promise<void>;
}
reserve() returns a two-state literal; get/complete are generic over the body type
TypeScript · example 3 of 4 — a generic idempotent handler wrapper
async function idempotent<T>(
  store: IdempotencyStore, key: string, work: () => Promise<{ status: number; body: T }>,
): Promise<{ status: number; body: T }> {
  if (await store.reserve(key) === "exists") {
    const prior = await store.get<T>(key);
    if (prior && isDone(prior)) return { status: prior.statusCode, body: prior.response };
    return { status: 409, body: { error: "in progress" } as unknown as T };
  }
  const result = await work();                       // first time: do it once
  await store.complete(key, result.status, result.body);
  return result;
}
a replay returns the cached T; the first request is the only one that runs work()
TypeScript · example 4 of 4 — wiring it into a route, key scoped per user
app.post("/payments", async (req, res) => {
  const key = req.header("Idempotency-Key");
  if (!key || !isUuid(key)) return res.status(400).json({ error: "missing/invalid key" });

  const scoped = `${req.user.id}:payments:${key}`;   // scope so keys can't collide across users
  const { status, body } = await idempotent<Payment>(store, scoped, () => charge(req.body));
  res.status(status).json(body);
});
the same key+user replays the original Payment; a retry never double-charges
Common Mistakes:
❌ Checking “does the key exist?” then inserting in two steps — two concurrent replays both pass the check and both execute.
✅ Reserve atomically (unique constraint / SET NX); let exactly one caller win the race.
❌ Storing the cached response as any — the replay path returns an unchecked shape that may not match the real handler.
✅ Make the record generic (IdempotencyRecord<T>) so the cached body is the same type the handler produces.
Key Rules:
  • Require a UUID Idempotency-Key on mutating requests; validate it before doing work.
  • Reserve the key atomically so concurrent replays can’t both run the operation.
  • Model the record as a discriminated union on status — only completed carries a typed response.
  • Scope keys by user + endpoint so they never collide across accounts.
Practice Exercises:
1Implement IdempotencyStore over Redis using SET key val NX EX 86400 for reserve, and confirm the TTL expires stale keys.
2Add a fingerprint (hash of the request body) to the record and return 422 if the same key arrives with a different body.
3Type the replay so in_progress returns 409 via the discriminated union’s narrowing rather than a runtime string check.
Connections: The record is a discriminated union with exhaustive narrowing; the store mirrors the typed Redis caching layer, and the retry motivation pairs directly with the client-side retries/circuit breakers of 07-04. Atomic reserve is the same race-avoidance idea as optimistic concurrency (Concept 5).
2Node.js + TS: Data Integrity
4. Type-Safe Database Transactions Node.js + TypeScript
Multi-step writes must be all-or-nothing: debit one account, credit another — a crash between them corrupts data. A transaction gives atomicity, but the type-safety win is making the transactional boundary explicit so every query inside runs on the same connection. A withTransaction(fn) callback that hands you a typed transactional client makes it impossible to accidentally run a step outside the transaction.

Wrap the unit of work in a callback that receives a Tx handle typed identically to the base client, so the same query API is available but bound to the transaction. The wrapper BEGINs, runs your function, COMMITs on success, and ROLLBACKs on any throw — and it returns whatever the callback returns, fully typed. Because the queries take the passed tx, a step that forgets it (using the global pool) is visible: it simply isn’t on the transactional client.

Prefer returning a value from the callback over mutating outer variables, and keep the callback free of unrelated I/O (don’t hold a transaction open across a slow network call — it pins a connection and grows lock contention).

1
Define withTransaction<T>(fn: (tx: Tx) => Promise<T>): Promise<T>.
2
Run every step through the passed tx — not the global pool.
3
Commit on return, rollback on throw; propagate the typed result.
TypeScript · example 1 of 4 — a transactional wrapper that returns a typed result
import type { Pool, PoolClient } from "pg";
export type Tx = PoolClient;   // same query API, bound to the transaction

async function withTransaction<T>(pool: Pool, fn: (tx: Tx) => Promise<T>): Promise<T> {
  const tx = await pool.connect();
  try {
    await tx.query("BEGIN");
    const result = await fn(tx);     // T flows straight back out
    await tx.query("COMMIT");
    return result;
  } catch (e) {
    await tx.query("ROLLBACK");
    throw e;
  } finally {
    tx.release();                    // always return the connection to the pool
  }
}
the callback's return type T is the wrapper's return type; rollback on any throw
TypeScript · example 2 of 4 — an atomic transfer using the passed tx
interface TransferResult { from: number; to: number }

const result = await withTransaction(pool, async (tx): Promise<TransferResult> => {
  await tx.query("UPDATE accounts SET balance = balance - $1 WHERE id = $2", [amount, fromId]);
  await tx.query("UPDATE accounts SET balance = balance + $1 WHERE id = $2", [amount, toId]);
  return { from: fromId, to: toId };   // committed only if both succeed
});
both updates share one connection; a throw between them rolls back both
TypeScript · example 3 of 4 — the compiler nudges you to use tx
await withTransaction(pool, async (tx) => {
  await insertOrder(tx, order);        // ✓ runs in the transaction
  // await pool.query("UPDATE ...");   // ✗ conceptually wrong: NOT in the tx
  await decrementStock(tx, order.items);
});

// helper signatures demand a Tx, so a caller can't pass the raw pool:
function insertOrder(tx: Tx, o: Order): Promise<void> { /* tx.query(...) */ return Promise.resolve(); }
typing helpers to require Tx makes "forgot the transaction" a signature mismatch
TypeScript · example 4 of 4 — the same shape with Drizzle / Prisma
// Drizzle: db.transaction gives a typed tx with the full query builder
await db.transaction(async (tx) => {
  await tx.update(accounts).set({ balance: sql`balance - ${amt}` }).where(eq(accounts.id, fromId));
  await tx.update(accounts).set({ balance: sql`balance + ${amt}` }).where(eq(accounts.id, toId));
});

// Prisma: prisma.$transaction([...]) for independent ops, or an interactive callback tx
await prisma.$transaction(async (tx) => { await tx.account.update(/* ... */); });
ORMs expose the same callback shape; tx carries the full typed query API
Common Mistakes:
❌ Running one step on tx and another on the global pool — the second isn’t in the transaction and won’t roll back.
✅ Thread the passed tx through every step; type helpers to require a Tx.
❌ Forgetting release() (or awaiting a slow HTTP call mid-transaction) — you leak/pin connections and starve the pool.
✅ Release in finally; keep transactions short and free of unrelated I/O.
Key Rules:
  • Wrap units of work in withTransaction(fn); the callback’s return type is the result type.
  • Commit on return, rollback on any throw, and release() the connection in finally.
  • Run every step through the passed tx; type helpers to require a Tx so the pool can’t sneak in.
  • Keep transactions short — no slow external I/O while holding a connection and locks.
Practice Exercises:
1Add a savepoint helper for nested transactions: SAVEPOINT/ROLLBACK TO around an inner step, typed as a nested callback.
2Make withTransaction accept an isolation level ("read committed" | "serializable") as a typed option and emit the right BEGIN.
3Write an integration test (Supertest, 07-05) proving a mid-transaction throw leaves both account balances unchanged.
Connections: The callback-returns-T shape is the same resource-scoping idea as using declarations and the graceful-shutdown cleanup of 07-04. The Drizzle/Prisma tx carries the typed query builder from R6, and requiring a Tx parameter is dependency injection enforcing correctness.
5. Optimistic Concurrency & Versioned Updates Node.js + TypeScript
Two users load the same record, both edit, both save — the second silently overwrites the first (“lost update”). Optimistic concurrency avoids locking by tagging each row with a version: an update only succeeds if the version still matches what you read, otherwise you detect the conflict and retry or surface a 409. Typing the version into the entity and the update result makes the conflict a value you must handle, not a silent no-op.

Add a version: number (or a timestamp/etag) to the row. The update carries the version you read in its WHERE clause and bumps it: SET ..., version = version + 1 WHERE id = $ AND version = $. If zero rows are affected, someone else won the race — return a typed { ok: false; conflict: true } rather than pretending success. Callers then reload and retry, or return 409 Conflict to the client (often with an If-Match ETag on the API surface).

Model the outcome as a discriminated union so the success arm carries the new version and the conflict arm carries none — the compiler forces the caller to branch on it instead of assuming the write landed.

1
Store a version on the row; read it with the entity.
2
Update with WHERE version = $read and bump it; check rows affected.
3
Return a typed success/conflict union; retry or 409 on conflict.
TypeScript · example 1 of 4 — a versioned entity and a typed outcome
interface Doc { id: string; title: string; version: number }

type SaveResult =
  | { ok: true; version: number }        // the new version after a successful bump
  | { ok: false; conflict: true };        // someone else updated first
the union makes "conflict" a value the caller must branch on
TypeScript · example 2 of 4 — the compare-and-set update
async function save(tx: Tx, doc: Doc): Promise<SaveResult> {
  const res = await tx.query(
    `UPDATE docs SET title = $1, version = version + 1
     WHERE id = $2 AND version = $3`,          // only if the version is unchanged
    [doc.title, doc.id, doc.version],
  );
  return res.rowCount === 1
    ? { ok: true, version: doc.version + 1 }
    : { ok: false, conflict: true };           // 0 rows => a concurrent writer won
}
rowCount discriminates success vs conflict; no lock was held
TypeScript · example 3 of 4 — a typed retry loop on conflict
async function saveWithRetry(id: string, edit: (d: Doc) => Doc, tries = 3): Promise<Doc> {
  for (let i = 0; i < tries; i++) {
    const current = await load(id);             // reads the latest version
    const next = edit(current);
    const r = await withTransaction(pool, (tx) => save(tx, next));
    if (r.ok) return { ...next, version: r.version };
  }
  throw new Error("too many concurrent updates");
}
reload-and-retry converges; the union forces you to check r.ok before returning
TypeScript · example 4 of 4 — surface conflicts as HTTP 409 / ETag
app.put("/docs/:id", async (req, res) => {
  const expected = Number(req.header("If-Match"));   // client sends the version it saw
  const r = await withTransaction(pool, (tx) =>
    save(tx, { id: req.params.id, title: req.body.title, version: expected }));

  if (!r.ok) return res.status(409).json({ error: "version conflict — reload and retry" });
  res.set("ETag", String(r.version)).json({ version: r.version });
});
If-Match carries the version; a stale write gets 409, not a silent overwrite
Common Mistakes:
❌ Updating by id alone and assuming success — a concurrent write is silently clobbered (lost update).
✅ Include WHERE version = $read, bump the version, and treat 0 rows affected as a conflict.
❌ Returning void from save — the caller can’t tell a real write from a no-op collision.
✅ Return a discriminated SaveResult so the conflict is a typed branch the caller must handle.
Key Rules:
  • Carry a version (or ETag/timestamp) on the row; include it in the update’s WHERE and bump it.
  • Zero rows affected means a concurrent writer won — that’s a conflict, not success.
  • Return a discriminated success/conflict union so callers must branch, not assume.
  • On conflict, reload-and-retry a bounded number of times, or surface 409 (with If-Match) to the client.
Practice Exercises:
1Swap the numeric version for a updatedAt timestamp ETag; discuss the risk of clock resolution vs a monotonic counter.
2Add jittered backoff (07-04 retries) between attempts in saveWithRetry and cap total elapsed time.
3Contrast with pessimistic locking (SELECT ... FOR UPDATE) and state when each is the right call (contention level, transaction length).
Connections: The success/conflict union is another discriminated-union outcome, like the Result of R1 and the idempotency record of Concept 3. The compare-and-set is the same “atomic reserve” race-avoidance as idempotency keys, the retry loop reuses 07-04’s backoff, and the update runs inside the Tx of Concept 4.
