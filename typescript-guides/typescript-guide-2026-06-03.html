Review — 6 Concepts
R1. never & Exhaustiveness Advanced Type System rep 0 · last reviewed 2026-06-01 · 2 days ago
The never type represents values that can never occur — bottom of the type hierarchy. Use it for exhaustiveness checking in switch/if-else chains: assign the narrowed value to never, and the compiler errors if you miss a case.
Key Pattern 
type Shape = { kind: "circle"; r: number } | { kind: "square"; s: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle": return Math.PI * shape.r ** 2;
    case "square": return shape.s ** 2;
    default: {
      const _exhaustive: never = shape; // Error if a case is missing
      return _exhaustive;
    }
  }
}
Challenge Exercise

Add a "triangle" variant (with base and height) to the Shape union. Without touching the switch body, confirm the compiler flags the exhaustiveness error. Then fix it. Next, write a assertNever(x: never): never helper that throws, and use it instead of the inline assignment.

#1 Pitfall: Forgetting the default exhaustiveness guard entirely. Without it, adding a new variant to the union compiles silently and falls through to undefined at runtime. Always include the never check.
R2. Conditional Types Advanced Type System rep 1 · last reviewed 2026-06-02 · 1 day ago
T extends U ? X : Y — type-level if/else. Distributive over unions when T is a naked type parameter. Use infer to extract types from complex structures. Wrap in [T] extends [U] to prevent distribution.
Key Pattern 
// Distributive conditional + infer
type Flatten<T> = T extends Array<infer U> ? U : T;

type A = Flatten<string[]>;        // string
type B = Flatten<number>;          // number
type C = Flatten<string[] | number>; // string | number (distributes!)
Challenge Exercise

Write type DeepFlatten<T> that recursively unwraps nested arrays: DeepFlatten<number[][][]> should yield number. Hint: make the conditional type recursive with infer.

#1 Pitfall: Forgetting distributive behavior — IsString<string | number> evaluates to true | false, not false. Wrap in [T] extends [string] to prevent distribution.
R3. Mapped Types Advanced Type System rep 1 · last reviewed 2026-06-02 · 1 day ago
Transform all properties: { [K in keyof T]: NewType }. Use as for key remapping and template literal types for new key names. Add/remove readonly and ? with +/- modifiers.
Key Pattern 
// Filter keys: remove properties that aren't strings
type StringKeysOnly<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K]
};

interface Mixed { name: string; age: number; email: string }
type OnlyStrings = StringKeysOnly<Mixed>;
// { name: string; email: string }
Challenge Exercise

Write type EventMap<T> that takes an interface and produces event handler types: EventMap<{ name: string; age: number }> should become { onNameChange: (value: string) => void; onAgeChange: (value: number) => void }. Use as key remapping with template literals.

#1 Pitfall: Forgetting string & K when using template literals with keyof — keyof includes number | symbol, but template literals only work with string. Without the intersection, you get a type error.
R4. keyof & Indexed Access Types Advanced Type System rep 1 · last reviewed 2026-06-02 · 1 day ago
keyof T gets union of property names. T[K] accesses the type at key K. Combine them in generics for type-safe property access: <T, K extends keyof T>(obj: T, key: K): T[K].
Key Pattern 
// Indexed access with union keys
interface Config {
  host: string;
  port: number;
  debug: boolean;
}

type ConfigValues = Config[keyof Config]; // string | number | boolean
type NetworkConfig = Config["host" | "port"]; // string | number
Challenge Exercise

Write a type PathValue<T, P> that resolves nested indexed access: PathValue<{ a: { b: { c: number } } }, ["a", "b", "c"]> should yield number. Use recursive conditional types with infer to walk the tuple path.

#1 Pitfall: keyof returns string | number | symbol. If you use the result with Object.keys() (which returns string[]), you need keyof T & string to avoid type mismatches.
R5. Type Predicates & Assertion Functions Advanced Type System rep 1 · last reviewed 2026-06-02 · 1 day ago
param is Type for custom type guards; asserts param is Type for assertion functions that throw on failure. Both narrow types in subsequent code. The compiler trusts your predicate unconditionally.
Key Pattern 
// Discriminated union guard
interface Dog { kind: "dog"; bark(): void }
interface Cat { kind: "cat"; purr(): void }
type Pet = Dog | Cat;

function isDog(pet: Pet): pet is Dog {
  return pet.kind === "dog";
}

const pets: Pet[] = [/* ... */];
const dogs: Dog[] = pets.filter(isDog); // correctly typed as Dog[]
Challenge Exercise

Write an assertion function assertDefined<T>(val: T | undefined, msg: string): asserts val is T. Use it to access a Map.get() result without optional chaining. Then write a type guard hasProperty<K extends string>(obj: unknown, key: K): obj is Record<K, unknown> that safely checks for a property on an unknown value.

#1 Pitfall: The compiler trusts your predicate unconditionally. A wrong guard like function isString(x: unknown): x is string { return true; } silently lies to the type checker, causing runtime crashes.
R6. satisfies Operator Advanced Type System rep 1 · last reviewed 2026-06-02 · 1 day ago
expr satisfies Type validates type compatibility without widening. The inferred type stays narrow (literal types, specific methods), while ensuring conformance to a broader type. Contrast with as (overrides type, unsafe) and : Type (widens type).
Key Pattern 
type Color = { r: number; g: number; b: number };
type Palette = Record<string, Color>;

const palette = {
  red:   { r: 255, g: 0, b: 0 },
  green: { r: 0, g: 128, b: 0 },
} satisfies Palette;

// palette.red is still { r: number; g: number; b: number }
// palette.blue would be a compile error (key doesn't exist)
// With `: Palette`, palette.blue would be allowed (any string key)
Challenge Exercise

Define a type FormConfig = { label: string; required: boolean; validate?: (v: string) => boolean }. Create a const forms object with three fields using satisfies Record<string, FormConfig>. Verify that (a) accessing a nonexistent key errors, (b) each field retains its specific literal types, and (c) adding a misspelled property inside a field also errors.

#1 Pitfall: Confusing satisfies with : Type annotation. : Palette widens the type so palette.anyKey compiles (returns Color | undefined). satisfies Palette keeps the literal keys so only declared keys are valid.

1. Decorators (TC39) Functions & Patterns
TC39 Stage 3 decorators are the new standard for adding metadata and behavior to classes and their members — frameworks like Angular, Ember, and Lit already use them. Understanding the difference between legacy experimentalDecorators and the new standard is critical for modern TypeScript projects.

Decorators are functions that modify or annotate classes, methods, accessors, and fields at definition time. The TC39 proposal (supported in TypeScript 5.0+) uses a new API with a context object instead of the legacy property descriptor approach.

  • 2026-07-04 (Day 31)
  • Class decorators receive the class constructor and can return a replacement class
  • Method decorators receive the original method and a context object with metadata
  • Field decorators receive undefined (the field value isn't available at decoration time) and a context
  • Decorator factories are functions that return a decorator, enabling parameterization
  • TC39 decorators do NOT require "experimentalDecorators": true — they work natively in TS 5.0+
TypeScript — Class Decorator (Seal the Class)
// A class decorator receives the class itself and an optional context
function sealed(target: Function, _context: ClassDecoratorContext) {
  Object.seal(target);
  Object.seal(target.prototype);
}

@sealed
class BankAccount {
  owner: string;
  balance: number;

  constructor(owner: string, balance: number) {
    this.owner = owner;
    this.balance = balance;
  }

  deposit(amount: number) {
    this.balance += amount;
  }
}

// Object.seal prevents adding new properties to the class or prototype
const acct = new BankAccount("Alice", 100);
acct.deposit(50); // works
// (acct as any).newProp = "x"; // throws in strict mode (sealed)
// No runtime output — but BankAccount and its prototype are now sealed
TypeScript — Method Decorator (Logging)
function logged(
  originalMethod: Function,
  context: ClassMethodDecoratorContext
) {
  const methodName = String(context.name);

  function replacementMethod(this: any, ...args: any[]) {
    console.log(`Calling ${methodName} with`, args);
    const result = originalMethod.call(this, ...args);
    console.log(`${methodName} returned`, result);
    return result;
  }

  return replacementMethod;
}

class Calculator {
  @logged
  add(a: number, b: number): number {
    return a + b;
  }

  @logged
  multiply(a: number, b: number): number {
    return a * b;
  }
}

const calc = new Calculator();
calc.add(2, 3);
calc.multiply(4, 5);
Calling add with [2, 3] add returned 5 Calling multiply with [4, 5] multiply returned 20
TypeScript — Decorator Factory (Parameterized)
// A decorator factory is a function that RETURNS a decorator
function minLength(min: number) {
  return function (
    target: undefined,    // field decorators receive undefined
    context: ClassFieldDecoratorContext
  ) {
    return function (this: any, initialValue: string): string {
      if (initialValue.length < min) {
        throw new Error(
          `${String(context.name)} must be at least ${min} characters`
        );
      }
      return initialValue;
    };
  };
}

function maxValue(max: number) {
  return function (
    target: undefined,
    context: ClassFieldDecoratorContext
  ) {
    return function (this: any, initialValue: number): number {
      if (initialValue > max) {
        throw new Error(
          `${String(context.name)} must be at most ${max}`
        );
      }
      return initialValue;
    };
  };
}

class UserProfile {
  @minLength(3)
  username: string = "Alice";     // OK: length 5 >= 3

  @maxValue(150)
  age: number = 30;               // OK: 30 <= 150
}

const profile = new UserProfile();
console.log(profile.username, profile.age);
Alice 30
TypeScript — addInitializer for Setup Logic
// context.addInitializer runs code when the class is instantiated
function autobind(
  originalMethod: Function,
  context: ClassMethodDecoratorContext
) {
  const methodName = context.name;
  context.addInitializer(function (this: any) {
    // 'this' is the new instance being created
    this[methodName] = this[methodName].bind(this);
  });
  return originalMethod;
}

class Button {
  label = "Click me";

  @autobind
  handleClick() {
    console.log(`Button: ${this.label}`);
  }
}

const btn = new Button();
const handler = btn.handleClick; // detached from object
handler(); // "Button: Click me" — still works because of autobind
Button: Click me
TypeScript — Legacy vs TC39 Comparison
// LEGACY (experimentalDecorators: true) — DO NOT USE in new projects
// function legacyLog(target: any, key: string, descriptor: PropertyDescriptor) {
//   const original = descriptor.value;
//   descriptor.value = function(...args: any[]) {
//     console.log(`Called ${key}`);
//     return original.apply(this, args);
//   };
// }

// TC39 STANDARD (TypeScript 5.0+, no flag needed)
function tc39Log(
  originalMethod: Function,
  context: ClassMethodDecoratorContext
) {
  return function (this: any, ...args: any[]) {
    console.log(`Called ${String(context.name)}`);
    return originalMethod.apply(this, args);
  };
}

// Key differences:
// 1. Legacy receives (target, key, descriptor) — TC39 receives (method, context)
// 2. Legacy mutates descriptor.value — TC39 returns a replacement function
// 3. TC39 context has .name, .kind, .static, .private, .addInitializer
// 4. TC39 works without "experimentalDecorators" in tsconfig
Common Mistakes
❌ Mixing legacy and TC39 decorators in the same project:
"experimentalDecorators": true in tsconfig changes decorator semantics globally — TC39 decorators won't work correctly
✅ Choose one: either remove experimentalDecorators (for TC39) or keep it (for legacy Angular/NestJS projects). Don't mix.
❌ Expecting field decorator to receive the field value as first arg:
function decorate(value: string, ctx: ClassFieldDecoratorContext) { ... }
✅ Field decorators receive undefined as first arg and return an initializer function:
function decorate(target: undefined, ctx: ClassFieldDecoratorContext) { return (initial) => initial; }
Key Rules
  • TC39 decorators (TS 5.0+) use (value, context) — no property descriptors, no experimentalDecorators flag
  • Decorator factories are functions that return decorators — use them when you need parameters: @minLength(3)
  • context.addInitializer runs per-instance setup code — ideal for auto-binding, registration, or validation hooks
Exercises
1 Write a @deprecated(reason: string) decorator factory for methods that logs a warning on first call (only once per method) and then delegates to the original method. Test it on a class with two deprecated methods.
2 Write a @memoize method decorator that caches results based on JSON-stringified arguments. Use context.addInitializer to attach the cache to each instance. Verify that calling the method twice with the same args returns the cached result.
3 Write a @singleton class decorator that ensures only one instance is ever created. The second call to new should return the existing instance. Return a replacement class from the decorator.
Connections: Decorator factories use the same closure pattern as Conditional Types with infer — a function returning a function. The autobind pattern ties directly to Day 3's this types. Decorators will pair with Declaration Files when typing third-party decorator libraries.

2. Module Systems Modules & Project Structure
Every TypeScript file with a top-level import or export is a module. Understanding ESM vs CJS, type-only imports, barrel files, and dynamic imports is essential for structuring any project larger than a single file.

TypeScript supports multiple module systems, but ES Modules (ESM) are the modern standard. CJS (require/module.exports) is the Node.js legacy format. TypeScript compiles your code based on the module setting in tsconfig — but you always write ESM-style import/export in your source.

  • Named exports: export function foo(), export const bar = 1
  • Default exports: export default class Foo — one per module
  • Re-exports: export { foo } from "./module" for barrel files
  • Type-only imports: import type { Foo } from "./module" — erased at compile time
  • Dynamic imports: const mod = await import("./module") — lazy loading / code splitting
TypeScript — Named Exports (Preferred)
// math.ts — named exports
export function add(a: number, b: number): number {
  return a + b;
}

export function multiply(a: number, b: number): number {
  return a * b;
}

export const PI = 3.14159;

// consumer.ts — named imports
import { add, multiply, PI } from "./math";
console.log(add(2, 3));      // 5
console.log(multiply(4, PI)); // 12.56636

// Rename on import
import { add as sum } from "./math";
console.log(sum(1, 2)); // 3

// Import everything as namespace
import * as MathUtils from "./math";
console.log(MathUtils.add(1, 2)); // 3
TypeScript — Type-Only Imports and Exports
// types.ts
export interface User {
  id: string;
  name: string;
  email: string;
}

export type Role = "admin" | "editor" | "viewer";

// Mixing value and type exports
export class UserService {
  getUser(id: string): User { /* ... */ }
}

// consumer.ts — type-only import (erased at compile time)
import type { User, Role } from "./types";

// Inline type imports (TS 4.5+)
import { UserService, type User as UserType } from "./types";

// type-only exports (from a barrel file)
export type { User, Role } from "./types";
export { UserService } from "./types";

// Why it matters:
// - type-only imports are completely removed from JS output
// - Prevents importing values that only exist at the type level
// - Required with "verbatimModuleSyntax": true in tsconfig
// - Helps bundlers with tree-shaking
TypeScript — Barrel Files (index.ts)
// models/user.ts
export interface User { id: string; name: string }

// models/post.ts
export interface Post { id: string; title: string; authorId: string }

// models/index.ts — barrel file (re-exports everything)
export type { User } from "./user";
export type { Post } from "./post";

// Now consumers import from the directory:
import type { User, Post } from "./models";   // clean!
// Instead of:
// import type { User } from "./models/user";
// import type { Post } from "./models/post";

// WARNING: Barrel files can hurt tree-shaking and build performance
// in large projects. If you import ONE thing from a barrel,
// the bundler may load ALL re-exported modules.
// Consider: import directly from the specific file in perf-sensitive code
TypeScript — Dynamic Imports (Code Splitting)
// Dynamic import returns Promise<typeof module>
async function loadChart() {
  // Only loaded when this function is called
  const { Chart } = await import("./chart-library");
  const chart = new Chart("canvas-id");
  chart.render();
}

// Conditional loading based on environment
async function getLogger() {
  if (process.env.NODE_ENV === "development") {
    const { DevLogger } = await import("./dev-logger");
    return new DevLogger();
  }
  const { ProdLogger } = await import("./prod-logger");
  return new ProdLogger();
}

// TypeScript infers the correct types from dynamic imports
const mod = await import("./math");
// typeof mod = { add: (a: number, b: number) => number; ... }
TypeScript — CJS Interop (Legacy)
// legacy-lib.js (CommonJS)
// module.exports = { greet: function(name) { return "Hello " + name; } };

// Importing CJS from ESM TypeScript
import legacyLib from "./legacy-lib";   // default import for module.exports
legacyLib.greet("Alice");

// Or with esModuleInterop (recommended in tsconfig):
import { greet } from "./legacy-lib";   // named import works too
greet("Alice");

// tsconfig setting that makes this work:
// "esModuleInterop": true    — generates helper code for CJS/ESM compat
// "allowSyntheticDefaultImports": true  — allows default import from CJS
Common Mistakes
❌ Importing a type as a value (causes runtime errors in some setups):
import { User } from "./types"; — if User is only an interface, some bundlers try to import it at runtime
✅ Use type-only import for types and interfaces:
import type { User } from "./types";
❌ Circular barrel imports: a/index.ts re-exports from b/index.ts which re-exports from a/index.ts
✅ Import directly from the source file to break cycles, or restructure to eliminate circular dependencies
Key Rules
  • Use import type for anything that's only needed at compile time (interfaces, type aliases) — it's erased from JS output
  • Prefer named exports over default exports — they enable better refactoring, auto-imports, and tree-shaking
  • Barrel files (index.ts) simplify imports but can degrade build performance — use judiciously in large monorepos
Exercises
1 Create a mini project with 3 files: models/user.ts (exports User interface and createUser function), models/index.ts (barrel file that re-exports both — but User as type-only), and app.ts that imports from the barrel. Verify the compiled JS doesn't contain the User import.
2 Write a loadPlugin function that dynamically imports one of three modules ("chart", "table", "form") based on a string argument. Use a mapped type to ensure type safety: the return type should match the specific module's exports.
3 Refactor a file that uses import * as Utils from "./utils" to use specific named imports. Then add "verbatimModuleSyntax": true to tsconfig and fix any errors that appear (hint: some value imports will need to become import type).
Connections: Type-only imports connect to Declaration Files (next) — .d.ts files are pure type information. Barrel files interact with tsconfig paths for import aliases. Dynamic imports return Promise<typeof module>, linking back to Day 3's async patterns.
3. Declaration Files Modules & Project Structure
Declaration files (.d.ts) provide type information for JavaScript code that has no TypeScript source. They're how the ecosystem types npm packages, browser APIs, and your own legacy JS — without them, everything would be any.

A .d.ts file contains only type declarations — no implementation. TypeScript uses these files to type-check code that references JavaScript libraries. Most popular npm packages either ship their own .d.ts files or have community-maintained types in @types/* on DefinitelyTyped.

  • .d.ts files are generated by tsc --declaration or written manually
  • declare keyword tells TypeScript "this exists at runtime, trust me on the types"
  • declare module "x" for typing or augmenting third-party modules
  • declare global for extending global scope (Window, process, etc.)
  • @types/xxx packages provide types for untyped npm packages
TypeScript — Writing a Basic .d.ts File
// legacy-utils.d.ts — types for an untyped JS library
// (legacy-utils.js has no TypeScript source)

declare function formatDate(date: Date, pattern: string): string;
declare function parseDate(input: string): Date | null;

declare const VERSION: string;

declare class Logger {
  constructor(prefix: string);
  info(message: string): void;
  warn(message: string): void;
  error(message: string, error?: Error): void;
}

// Now you can use them in TypeScript:
// import { formatDate, Logger } from "legacy-utils";
// const log = new Logger("app");
// log.info(formatDate(new Date(), "yyyy-MM-dd"));
TypeScript — Module Declaration (Typing an Untyped Package)
// types/untyped-csv-lib.d.ts
declare module "untyped-csv-lib" {
  export interface CsvOptions {
    delimiter?: string;
    headers?: boolean;
    quote?: string;
  }

  export function parse(input: string, options?: CsvOptions): string[][];
  export function stringify(data: string[][], options?: CsvOptions): string;

  export default class CsvReader {
    constructor(filePath: string);
    read(): Promise<string[][]>;
    stream(): AsyncIterable<string[]>;
  }
}

// Now you can import it with full type safety:
// import CsvReader, { parse, stringify } from "untyped-csv-lib";
// const data = parse("a,b,c\n1,2,3", { headers: true });
TypeScript — Global Augmentation
// global.d.ts — extend the global scope
export {}; // This file must be a module (has export)

declare global {
  // Add a property to Window
  interface Window {
    __APP_VERSION__: string;
    analytics: {
      track(event: string, data?: Record<string, unknown>): void;
    };
  }

  // Add a global function
  function sleep(ms: number): Promise<void>;

  // Extend an existing global interface
  interface Array<T> {
    toShuffled(): T[];
  }
}

// Now these work without import:
// window.__APP_VERSION__ = "1.2.3";    // typed as string
// window.analytics.track("page_view"); // fully typed
// [1,2,3].toShuffled();                // method exists on all arrays

Common Mistakes
❌ Forgetting export {} in a declare global file:
Without it, the file is a script (not a module), and declare global has no effect
✅ Always include export {} (or any import/export) to make the file a module:
export {}; declare global { ... }
❌ Creating a .d.ts file with implementation code:
declare function add(a: number, b: number): number { return a + b; }
✅ Declarations only — no function bodies, no initializers:
declare function add(a: number, b: number): number;
Key Rules
  • .d.ts files contain only type information — no runtime code, no implementations, no side effects
  • Use declare module "name" for module-level augmentation and declare global for global scope extension
  • Install @types/xxx for community types; create custom .d.ts only when no @types package exists

4. tsconfig.json Deep Dive Modules & Project Structure
Every TypeScript project starts with tsconfig.json. Misconfigured settings cause mysterious type errors, missing types, and broken builds. Understanding the key options — especially strict, module, and paths — saves hours of debugging.

The tsconfig.json file controls how TypeScript compiles your project. It specifies which files to include, what JavaScript version to target, how modules are resolved, and which type-checking rules to enforce. The strict flag alone enables 7+ individual checks.

  • strict enables: strictNullChecks, noImplicitAny, strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, noImplicitThis, useUnknownInCatchVariables, alwaysStrict
  • target controls the output JS version; module controls the module system
  • moduleResolution controls how TypeScript finds imported files
  • paths and baseUrl enable import aliases like @/components
TypeScript — Recommended Modern tsconfig.json
{
  "compilerOptions": {
    // === Type Checking (turn everything on) ===
    "strict": true,                    // enables all strict checks
    "noUncheckedIndexedAccess": true,  // obj[key] returns T | undefined
    "noUnusedLocals": true,            // error on unused variables
    "noUnusedParameters": true,        // error on unused function params
    "noFallthroughCasesInSwitch": true, // error on switch fallthrough
    "forceConsistentCasingInFileNames": true, // prevent casing issues

    // === Module System ===
    "module": "ESNext",                // output ESM
    "moduleResolution": "bundler",     // modern bundler-style resolution
    "esModuleInterop": true,           // CJS/ESM interop helpers
    "verbatimModuleSyntax": true,      // enforce import type for types

    // === Output ===
    "target": "ES2022",                // output JS version
    "lib": ["ES2022", "DOM", "DOM.Iterable"], // available APIs
    "outDir": "./dist",                // compiled output directory
    "declaration": true,               // generate .d.ts files
    "declarationMap": true,            // sourcemaps for .d.ts
    "sourceMap": true,                 // JS sourcemaps

    // === Path Aliases ===
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],              // @/components -> ./src/components
      "@lib/*": ["./src/lib/*"]        // @lib/utils -> ./src/lib/utils
    },

    // === Performance ===
    "skipLibCheck": true,              // skip type-checking .d.ts files
    "incremental": true                // cache for faster rebuilds
  },

  "include": ["src/**/*.ts", "src/**/*.tsx"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
TypeScript — What "strict" Catches
// === strictNullChecks ===
function getLength(s: string | null): number {
  // return s.length;         // Error: s might be null
  return s?.length ?? 0;      // OK: handle null case
}

// === noImplicitAny ===
// function process(data) { }  // Error: parameter 'data' implicitly has 'any'
function process(data: unknown) { } // OK: explicit type

// === strictFunctionTypes ===
type Handler = (e: MouseEvent) => void;
// const h: Handler = (e: Event) => {}; // Error: Event not assignable to MouseEvent
const h: Handler = (e: MouseEvent) => {}; // OK

// === strictPropertyInitialization ===
class User {
  // name: string;           // Error: not initialized in constructor
  name: string;
  constructor(name: string) {
    this.name = name;        // OK: initialized
  }
}

// === noUncheckedIndexedAccess (not part of strict, but recommended) ===
const arr = [1, 2, 3];
// const first: number = arr[0]; // Error: might be undefined
const first: number | undefined = arr[0]; // OK
TypeScript — target vs module vs moduleResolution
// "target" = what JS version to output
// "target": "ES5"   -> var, function, no class, no async/await
// "target": "ES2015" -> let/const, class, arrow functions, Promise
// "target": "ES2022" -> top-level await, private fields, Error.cause
// "target": "ESNext" -> latest features (moves with each TS release)

// "module" = what module system to output
// "module": "CommonJS"   -> require(), module.exports
// "module": "ESNext"     -> import/export (static ESM)
// "module": "NodeNext"   -> Node's native ESM (respects package.json "type")

// "moduleResolution" = how TS FINDS imported modules
// "moduleResolution": "node"     -> Classic Node.js resolution
// "moduleResolution": "node16"   -> Node 16+ with ESM support
// "moduleResolution": "bundler"  -> For Vite/webpack/esbuild (most flexible)

// Common combinations:
// Library (Node):    target=ES2022, module=NodeNext, moduleResolution=NodeNext
// App (Vite/Next):   target=ES2022, module=ESNext,   moduleResolution=bundler
// Legacy (Node CJS): target=ES2020, module=CommonJS, moduleResolution=node
TypeScript — Path Aliases in Practice
// tsconfig.json paths:
// "baseUrl": ".",
// "paths": { "@/*": ["./src/*"] }

// Before: deeply nested relative imports
import { UserService } from "../../../services/user-service";
import type { Config } from "../../../../config/types";

// After: clean path aliases
import { UserService } from "@/services/user-service";
import type { Config } from "@/config/types";

// IMPORTANT: tsconfig paths only affect TypeScript compilation.
// Your bundler (Vite, webpack, etc.) needs its OWN alias config:
//
// vite.config.ts:
//   resolve: { alias: { "@": path.resolve(__dirname, "src") } }
//
// jest.config.ts:
//   moduleNameMapper: { "^@/(.*)$": "<rootDir>/src/$1" }
//
// This is the #1 source of "works in editor, breaks at runtime" issues.
TypeScript — noEmit (Type-Check Only)
// When your bundler (Vite, esbuild, SWC) handles compilation,
// use TypeScript ONLY for type checking:

{
  "compilerOptions": {
    "noEmit": true,           // don't generate any JS files
    "strict": true,            // still enforces all type checks
    "moduleResolution": "bundler",
    // ... other options
  }
}

// Run type checking:
// npx tsc --noEmit          (check types, produce no output)
// npx tsc --noEmit --watch  (continuous type checking)

// Your bundler compiles TS to JS (faster than tsc):
// vite build / esbuild / swc
// TypeScript is just the type checker, not the compiler
Common Mistakes
❌ Setting "strict": false to "fix" type errors:
This silences real bugs. Instead, fix the errors or use targeted // @ts-expect-error
✅ Always use "strict": true and add noUncheckedIndexedAccess. Fix errors at their source, not by loosening the config.
❌ Configuring paths in tsconfig but not in the bundler:
Editor shows no errors, but the build fails with "Cannot find module '@/...'"
✅ Mirror path aliases in both tsconfig (for type checking) and your bundler config (for resolution). Or use a plugin like vite-tsconfig-paths.
Key Rules
  • Always start with "strict": true and "noUncheckedIndexedAccess": true — they catch real bugs
  • target controls output JS syntax, module controls import/export style, moduleResolution controls file lookup
  • Path aliases require configuration in BOTH tsconfig and your bundler/test runner
Exercises
1 Start with an empty tsconfig.json containing only {}. Add options one at a time: first "strict": true, then compile a file with let x = null; console.log(x.toString()). Observe how each strict sub-option affects the errors. Then enable "noUncheckedIndexedAccess" and try const arr = [1,2,3]; const n: number = arr[0];.
2 Configure paths aliases for @models/*, @services/*, and @utils/* pointing to subdirectories in src/. Create stub files and verify that VS Code resolves the imports. Then set "declaration": true and inspect the generated .d.ts files.
3 Create two tsconfig files: tsconfig.json (base config) and tsconfig.build.json that extends it with "noEmit": false, "outDir": "dist", and excludes test files. Use "extends": "./tsconfig.json" for the build config. Run tsc -p tsconfig.build.json.
Connections: paths aliases interact with Module Systems and barrel files. declaration: true generates the Declaration Files we just covered. skipLibCheck skips .d.ts type checking — useful when @types packages conflict. strict mode makes Type Predicates and never exhaustiveness more valuable.
5. Namespaces vs Modules Modules & Project Structure
Namespaces were TypeScript's original module system before ES modules existed. Understanding when to use namespaces vs modules — and the powerful concept of declaration merging — prevents architectural mistakes and unlocks advanced type augmentation patterns.

Modern TypeScript should use ES modules for code organization. Namespaces (namespace X {}) are a legacy pattern from the pre-ESM era. However, namespaces still serve important roles in declaration merging, ambient declarations, and augmenting third-party types.

  • Modules: file-based, each file is a module, uses import/export, tree-shakeable
  • Namespaces: logical grouping within or across files, uses namespace X {}, not tree-shakeable
  • Declaration merging: TypeScript merges same-named interfaces, namespaces, and enums automatically
  • Module augmentation: add new types to existing modules using declare module
  • Namespaces compile to IIFEs — they're runtime objects, not just types
TypeScript — Namespace Syntax (Legacy Pattern)
// Namespace groups related code under a single name
namespace Validation {
  export interface Result {
    valid: boolean;
    errors: string[];
  }

  export function isEmail(value: string): Result {
    const valid = /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
    return {
      valid,
      errors: valid ? [] : ["Invalid email format"],
    };
  }

  export function isMinLength(value: string, min: number): Result {
    const valid = value.length >= min;
    return {
      valid,
      errors: valid ? [] : [`Must be at least ${min} characters`],
    };
  }
}

// Usage: access via namespace prefix
const result = Validation.isEmail("test@example.com");
console.log(result.valid);
true
TypeScript — Module Equivalent (Modern, Preferred)
// validation.ts — just use regular exports
export interface Result {
  valid: boolean;
  errors: string[];
}

export function isEmail(value: string): Result {
  const valid = /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
  return { valid, errors: valid ? [] : ["Invalid email format"] };
}

export function isMinLength(value: string, min: number): Result {
  const valid = value.length >= min;
  return { valid, errors: valid ? [] : [`Must be at least ${min} chars`] };
}

// consumer.ts — import what you need
import { isEmail, isMinLength } from "./validation";
// OR: import * as Validation from "./validation";

// Advantages over namespaces:
// 1. Tree-shakeable (dead code elimination)
// 2. Works with all bundlers and tools
// 3. Standard JavaScript (not TS-specific syntax)
// 4. Each file has its own scope (no global pollution)
TypeScript — Declaration Merging (Interfaces)
// Interface declarations with the same name MERGE automatically
interface Box {
  width: number;
  height: number;
}

interface Box {
  depth: number;
  color?: string;
}

// Merged result:
// interface Box {
//   width: number;
//   height: number;
//   depth: number;
//   color?: string;
// }

const box: Box = {
  width: 10,
  height: 20,
  depth: 5,
  color: "red",
};

// This is how lib.dom.d.ts gradually builds up HTMLElement, Window, etc.
// Each declaration adds more properties to the same interface.

// NOTE: type aliases CANNOT merge — duplicate names cause an error:
// type Point = { x: number };
// type Point = { y: number }; // Error: Duplicate identifier 'Point'
TypeScript — Namespace Merging with Classes and Functions
// Namespace merges with a class to add static members
class Album {
  constructor(public title: string, public artist: string) {}
}

namespace Album {
  export interface TrackInfo {
    name: string;
    duration: number;
  }

  export function create(title: string, artist: string): Album {
    return new Album(title, artist);
  }
}

// Album is both a class and has static-like namespace members
const album = new Album("OK Computer", "Radiohead"); // class constructor
const album2 = Album.create("Kid A", "Radiohead");   // namespace function
const track: Album.TrackInfo = { name: "Airbag", duration: 284 }; // namespace type

// Namespace merges with a function (adding properties)
function buildUrl(base: string): string {
  return `${buildUrl.prefix}${base}`;
}

namespace buildUrl {
  export let prefix = "https://";
  export function withQuery(base: string, params: Record<string, string>): string {
    const qs = new URLSearchParams(params).toString();
    return `${prefix}${base}?${qs}`;
  }
}

console.log(buildUrl("example.com"));
console.log(buildUrl.withQuery("api.example.com", { page: "1" }));
https://example.com https://api.example.com?page=1
TypeScript — When Namespaces Still Make Sense
// 1. Ambient declarations in .d.ts files (describing global JS)
declare namespace NodeJS {
  interface ProcessEnv {
    NODE_ENV: "development" | "production" | "test";
    PORT?: string;
    DATABASE_URL: string;
  }
}

// Now process.env.NODE_ENV is typed as a union, not just string
// process.env.DATABASE_URL is required (no ?)

// 2. Enum + namespace merging (adding utility methods to enums)
enum Color {
  Red = "RED",
  Green = "GREEN",
  Blue = "BLUE",
}

namespace Color {
  export function parse(value: string): Color | undefined {
    const upper = value.toUpperCase();
    if (upper in Color) return upper as Color;
    return undefined;
  }

  export function all(): Color[] {
    return [Color.Red, Color.Green, Color.Blue];
  }
}

const color = Color.parse("red"); // Color | undefined
const allColors = Color.all();    // Color[]
console.log(color, allColors);
RED ["RED", "GREEN", "BLUE"]
Common Mistakes
❌ Using namespaces for code organization in new projects:
namespace Models { export interface User { ... } }
✅ Use ES modules — put each model in its own file:
export interface User { ... } in models/user.ts
❌ Mixing namespace with import/export in the same file:
A file with top-level import/export is a module; namespace inside it only affects that module's scope
✅ In modules, use declare module for augmentation instead of namespace. Reserve namespace for declaration merging and ambient .d.ts files.
Key Rules
  • Use ES modules for all new code. Namespaces are legacy for code organization — modern tooling doesn't tree-shake them.
  • Declaration merging (interfaces, namespaces, enums) is how TypeScript's own lib.d.ts files work — and it's how you extend third-party types
  • Namespaces still shine for: ambient .d.ts declarations, merging with classes/functions/enums, and typing global variables like process.env
Exercises
1 Create an enum HttpMethod (GET, POST, PUT, DELETE) and merge a namespace with it that adds: HttpMethod.isIdempotent(method: HttpMethod): boolean (GET/PUT/DELETE are idempotent) and HttpMethod.hasBod(method: HttpMethod): boolean (POST/PUT have bodies). Test all combinations.
2 Use interface declaration merging to extend Array<T> in a global .d.ts file with a first(): T | undefined method, then implement it on Array.prototype in a separate file. Verify that [1,2,3].first() works with correct types.
3 Augment the NodeJS.ProcessEnv interface to add your project's environment variables (API_KEY: string, DEBUG?: "true" | "false", LOG_LEVEL: "info" | "warn" | "error"). Verify that process.env.API_KEY is typed as string (not string | undefined).
Connections: Declaration merging is the mechanism behind module augmentation with declare module. Interface merging explains why interfaces and type aliases behave differently (types can't merge). The namespace + class pattern is a TypeScript-specific way to achieve the companion object pattern from Scala/Kotlin.
