R1. Module Federation Typing Advanced Patterns & Performance rep 0 · last reviewed 2026-06-07 · overdue 4 days
Module Federation lets micro-frontends share code at runtime. TypeScript has no built-in awareness of federated modules, so you use declare module to describe the shape of remote imports. Type packages or a shared @types directory keeps multiple teams in sync.
Key Pattern 
// typings/remoteApp.d.ts
declare module 'remoteApp/Button' {
  import { FC } from 'react';
  interface ButtonProps {
    label: string;
    variant?: 'primary' | 'secondary';
    onClick?: () => void;
  }
  const Button: FC<ButtonProps>;
  export default Button;
}

declare module 'remoteApp/useAuth' {
  interface AuthState {
    user: { id: string; name: string } | null;
    token: string | null;
  }
  export function useAuth(): AuthState;
}

// Consuming the federated module — fully typed
import Button from 'remoteApp/Button';
import { useAuth } from 'remoteApp/useAuth';

const App = () => {
  const { user } = useAuth();
  return <Button label={user?.name ?? 'Guest'} variant="primary" />;
};
Button: FC<ButtonProps> useAuth(): { user: { id: string; name: string } | null; token: string | null }
Challenge Exercise

Write a declare module for a federated remoteShared/api that exports a generic fetchData<T>(endpoint: string): Promise<T> function and a Config interface with baseUrl and timeout properties. Then consume it in an app component with proper typing.

#1 Pitfall: Forgetting to include the .d.ts file in your tsconfig.json’s include or typeRoots. The declarations exist but TypeScript never sees them, so all federated imports remain any. Always verify with Ctrl+hover in your editor.
R2. Testing Types: tsd & expect-type Advanced Patterns & Performance rep 0 · last reviewed 2026-06-07 · overdue 4 days
Type-level testing tools (tsd, expect-type) let you write assertions that run at compile time to verify your library exports correct types. expectType<T>() checks that a value is exactly type T. expectError() verifies that invalid usage produces a type error.
Key Pattern 
// Using expect-type (works with vitest, jest, etc.)
import { expectTypeOf } from 'expect-type';

// Assert exact type
expectTypeOf<string>().toMatchTypeOf<string>();

// Assert function return type
declare function fetchUser(id: number): Promise<{ name: string }>;
expectTypeOf(fetchUser).returns.resolves.toEqualTypeOf<{ name: string }>();

// Assert parameter types
expectTypeOf(fetchUser).parameter(0).toBeNumber();

// Assert that something is NOT assignable
expectTypeOf<string>().not.toMatchTypeOf<number>();

// Using tsd (in __tests__/index.test-d.ts)
import { expectType, expectError } from 'tsd';
import { createStore } from '../src';

// Correct usage compiles
const store = createStore({ count: 0 });
expectType<number>(store.getState().count);

// Incorrect usage should error
expectError(store.getState().nonExistent);
Challenge Exercise

Write type tests for a Result<T, E> type (Ok/Err discriminated union). Verify: (1) Ok<number> has a value of type number, (2) Err<string> has an error of type string, (3) accessing .value on an Err produces a type error, (4) narrowing via .ok discriminant works correctly.

#1 Pitfall: Confusing toMatchTypeOf (assignability) with toEqualTypeOf (exact equality). { a: string; b: number } matches { a: string } but does not equal it. Use toEqualTypeOf when you need exact shape matching; use toMatchTypeOf for subtype checks.
R3. TypeScript Performance Advanced Patterns & Performance rep 0 · last reviewed 2026-06-07 · overdue 4 days
TypeScript compiler performance degrades with deep type recursion, wide union distributions, and excessive use of type intersections. --generateTrace produces Chrome DevTools-compatible traces to diagnose slow builds. Prefer interface extends over type & intersection for object merging — interfaces are cached and compared structurally by name.
Key Pattern 
// SLOW: type intersection creates a new anonymous type each time
type SlowMerge = BaseConfig & DatabaseConfig & CacheConfig & LogConfig;

// FAST: interface extends — TypeScript caches named interfaces
interface FastMerge extends BaseConfig, DatabaseConfig, CacheConfig, LogConfig {}

// SLOW: deeply recursive conditional type
type DeepFlatten<T> = T extends Array<infer U> ? DeepFlatten<U> : T;

// FAST: bounded recursion with a depth limit
type Flatten<T, D extends number = 5> = D extends 0
  ? T
  : T extends Array<infer U>
    ? Flatten<U, Prev[D]>
    : T;
type Prev = [never, 0, 1, 2, 3, 4];

// Diagnose with:
// tsc --generateTrace ./trace-output
// Open trace in chrome://tracing
Challenge Exercise

You have 10 config types currently merged via type FullConfig = A & B & C & D & E & F & G & H & I & J;. Refactor to use interface extends. Then write a recursive type Paths<T> that extracts all dot-notation property paths but limits depth to 3 levels to avoid performance issues.

#1 Pitfall: Using type aliases for large object merges in hot paths. Each intersection creates an anonymous structural type that TypeScript must re-evaluate every time it appears. Named interfaces are cached, making them orders of magnitude faster for types used in many places.
R4. React + TypeScript: Component Props React + TypeScript rep 0 · last reviewed 2026-06-08 · overdue 3 days
React component props can be typed with interfaces or type aliases. FC<Props> is optional — plain function declarations with typed parameters are preferred in modern React. PropsWithChildren adds the children prop. Discriminated union props enable polymorphic components where the prop shape changes based on a discriminant field.
Key Pattern 
// Discriminated union props — shape changes based on 'variant'
type AlertProps =
  | { variant: 'success'; message: string }
  | { variant: 'error'; message: string; retryAction: () => void }
  | { variant: 'loading'; progress: number };

function Alert(props: AlertProps) {
  switch (props.variant) {
    case 'success':
      return <div className="success">{props.message}</div>;
    case 'error':
      return (
        <div className="error">
          {props.message}
          <button onClick={props.retryAction}>Retry</button>
        </div>
      );
    case 'loading':
      return <progress value={props.progress} max={100} />;
  }
}

// Usage — TypeScript enforces correct props per variant
<Alert variant="error" message="Failed" retryAction={() => reload()} />
// <Alert variant="error" message="Oops" /> // Error: missing retryAction
Challenge Exercise

Create a Modal component with discriminated union props: variant: 'confirm' requires onConfirm and onCancel callbacks, variant: 'alert' requires only onDismiss, and variant: 'form' requires onSubmit with a generic data type. All variants share a title: string prop.

#1 Pitfall: Using a single flat interface with many optional props instead of discriminated unions. This creates impossible states at the type level: e.g., a component that has retryAction defined but variant: 'success'. Discriminated unions make illegal states unrepresentable.
R5. Conditional Types Advanced Type System rep 2 · last reviewed 2026-06-03 · overdue 3 days
T extends U ? X : Y creates types that choose between branches based on a condition. The infer keyword captures sub-parts of types inside the condition. When T is a bare union, the conditional distributes over each member individually.
Key Pattern 
// Extract return type using infer
type ReturnOf<T> = T extends (...args: any[]) => infer R ? R : never;

type A = ReturnOf<() => string>;       // string
type B = ReturnOf<(x: number) => boolean>; // boolean

// Distributive conditional over a union
type NonNullable<T> = T extends null | undefined ? never : T;
type C = NonNullable<string | null | undefined>; // string

// Prevent distribution with [T] wrapper
type IsString<T> = [T] extends [string] ? true : false;
type D = IsString<string | number>;     // false (not distributed)

// Extract promise value
type Awaited<T> = T extends Promise<infer V> ? Awaited<V> : T;
type E = Awaited<Promise<Promise<number>>>; // number
A = string B = boolean C = string D = false E = number
Challenge Exercise

Write a conditional type EventPayload<T> that extracts the payload type from event handler functions: given (event: { type: string; payload: infer P }) => void, it should return P. Test it with three different event handler signatures.

#1 Pitfall: Forgetting that bare union types distribute through conditional types. IsString<string | number> evaluates as IsString<string> | IsString<number>, yielding true | false (which is boolean), not false. Wrap in [T] extends [U] to prevent distribution.
R6. Mapped Types Advanced Type System rep 2 · last reviewed 2026-06-03 · overdue 3 days
{ [K in keyof T]: ... } iterates over the keys of T to create a new type. You can add/remove readonly and ? modifiers with +/- prefixes. Key remapping with as lets you transform or filter keys during mapping.
Key Pattern 
interface Config {
  host: string;
  port: number;
  debug: boolean;
}

// Make all properties optional with default values
type WithDefaults<T> = {
  [K in keyof T]: { value: T[K]; default: T[K] };
};
type ConfigWithDefaults = WithDefaults<Config>;
// { host: { value: string; default: string };
//   port: { value: number; default: number };
//   debug: { value: boolean; default: boolean } }

// Key remapping: prefix getters
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};
type ConfigGetters = Getters<Config>;
// { getHost: () => string; getPort: () => number; getDebug: () => boolean }

// Filter keys by value type
type OnlyStrings<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};
type StringConfig = OnlyStrings<Config>;
// { host: string }
ConfigWithDefaults = { host: { value: string; default: string }; port: { value: number; default: number }; debug: { value: boolean; default: boolean } } ConfigGetters = { getHost: () => string; getPort: () => number; getDebug: () => boolean } StringConfig = { host: string }
Challenge Exercise

Create a mapped type EventHandlers<T> that transforms { click: MouseEvent; keydown: KeyboardEvent; scroll: Event } into { onClick: (e: MouseEvent) => void; onKeydown: (e: KeyboardEvent) => void; onScroll: (e: Event) => void } using key remapping with as and Capitalize.

#1 Pitfall: Trying to use as key remapping with keyof T when K might be symbol. Template literal types only accept string, so you need K & string (or Extract<K, string>) in the remapping clause: [K in keyof T as `on${Capitalize<string & K>}`].
1 State Management & Architecture
#1. Redux Toolkit TypeScript Patterns redux_toolkit_ts
Redux Toolkit (RTK) is the official, opinionated Redux toolset. When combined with TypeScript, it provides type-safe global state, type-inferred reducers, and end-to-end type safety from action dispatch to selector output — critical for any dashboard or enterprise app.

RTK's createSlice infers action types from reducer names. PayloadAction<T> types the action payload. The RootState and AppDispatch types, derived from the store, flow through typed hooks to components.

For async logic, createAsyncThunk accepts type parameters for the return value, the argument, and the thunk API config (which includes the state type for getState).

The key insight is that almost all types are inferred — you declare the state interface and RTK handles the rest.

Example 1 — createSlice with typed state
import { createSlice, PayloadAction } from '@reduxjs/toolkit';

// 1. Define the state interface
interface CounterState {
  value: number;
  status: 'idle' | 'loading' | 'failed';
}

const initialState: CounterState = {
  value: 0,
  status: 'idle',
};

// 2. Create the slice — reducer action types are auto-inferred
const counterSlice = createSlice({
  name: 'counter',
  initialState,
  reducers: {
    increment(state) {
      state.value += 1;  // Immer lets you "mutate" safely
    },
    decrement(state) {
      state.value -= 1;
    },
    incrementByAmount(state, action: PayloadAction<number>) {
      state.value += action.payload;  // payload is typed as number
    },
    setStatus(state, action: PayloadAction<CounterState['status']>) {
      state.status = action.payload;  // payload: 'idle' | 'loading' | 'failed'
    },
  },
});

// 3. Export actions and reducer
export const { increment, decrement, incrementByAmount, setStatus } =
  counterSlice.actions;
export default counterSlice.reducer;
increment: ActionCreatorWithoutPayload<"counter/increment"> incrementByAmount: ActionCreatorWithPayload<number, "counter/increment"> setStatus: ActionCreatorWithPayload<"idle" | "loading" | "failed">
Example 2 — Store setup with RootState & AppDispatch
import { configureStore } from '@reduxjs/toolkit';
import counterReducer from './counterSlice';
import userReducer from './userSlice';

export const store = configureStore({
  reducer: {
    counter: counterReducer,
    user: userReducer,
  },
});

// Infer types from the store itself
export type RootState = ReturnType<typeof store.getState>;
// { counter: CounterState; user: UserState }

export type AppDispatch = typeof store.dispatch;
// ThunkDispatch & Dispatch
RootState = { counter: CounterState; user: UserState } AppDispatch = ThunkDispatch<RootState, undefined, AnyAction> & Dispatch<AnyAction>
Example 3 — Typed hooks (useAppSelector, useAppDispatch)
import { useDispatch, useSelector, TypedUseSelectorHook } from 'react-redux';
import type { RootState, AppDispatch } from './store';

// Create typed versions of the hooks
export const useAppDispatch = () => useDispatch<AppDispatch>();
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;

// Usage in a component — fully typed
function Counter() {
  const count = useAppSelector((state) => state.counter.value);
  //    ^? number
  const status = useAppSelector((state) => state.counter.status);
  //    ^? 'idle' | 'loading' | 'failed'
  const dispatch = useAppDispatch();

  return (
    <div>
      <span>{count}</span>
      <button onClick={() => dispatch(incrementByAmount(5))}>+5</button>
      {/* dispatch(incrementByAmount('5')) would be a type error */}
    </div>
  );
}
Example 4 — createAsyncThunk with proper typing
import { createAsyncThunk, createSlice } from '@reduxjs/toolkit';
import type { RootState } from './store';

interface User {
  id: number;
  name: string;
  email: string;
}

// Type parameters: <ReturnType, ArgType, ThunkApiConfig>
export const fetchUser = createAsyncThunk<
  User,                    // Return type of the payload creator
  number,                  // Argument type (userId)
  { state: RootState; rejectValue: string }  // ThunkAPI config
>(
  'user/fetchUser',
  async (userId, { getState, rejectWithValue }) => {
    const token = getState().user.token;  // typed via { state: RootState }
    const response = await fetch(`/api/users/${userId}`, {
      headers: { Authorization: `Bearer ${token}` },
    });
    if (!response.ok) {
      return rejectWithValue('Failed to fetch user'); // typed as string
    }
    return response.json() as Promise<User>;
  }
);

// Handle in extraReducers
const userSlice = createSlice({
  name: 'user',
  initialState: { data: null as User | null, token: '', loading: false, error: '' },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending, (state) => {
        state.loading = true;
      })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.data = action.payload;  // typed as User
        state.loading = false;
      })
      .addCase(fetchUser.rejected, (state, action) => {
        state.error = action.payload ?? 'Unknown error'; // typed as string
        state.loading = false;
      });
  },
});
Example 5 — Real-world: typed entity adapter for a dashboard
import { createEntityAdapter, createSlice, EntityState } from '@reduxjs/toolkit';

interface DashboardWidget {
  id: string;
  type: 'chart' | 'table' | 'metric';
  title: string;
  config: Record<string, unknown>;
  position: { x: number; y: number; w: number; h: number };
}

// Entity adapter provides normalized CRUD
const widgetAdapter = createEntityAdapter<DashboardWidget>({
  selectId: (widget) => widget.id,
  sortComparer: (a, b) => a.title.localeCompare(b.title),
});

interface WidgetState extends EntityState<DashboardWidget, string> {
  activeWidgetId: string | null;
}

const widgetSlice = createSlice({
  name: 'widgets',
  initialState: widgetAdapter.getInitialState({ activeWidgetId: null }) as WidgetState,
  reducers: {
    addWidget: widgetAdapter.addOne,
    removeWidget: widgetAdapter.removeOne,
    updateWidget: widgetAdapter.updateOne,
    setActive(state, action: PayloadAction<string | null>) {
      state.activeWidgetId = action.payload;
    },
  },
});

// Typed selectors from the adapter
const widgetSelectors = widgetAdapter.getSelectors<RootState>(
  (state) => state.widgets
);
// widgetSelectors.selectAll(state)   — DashboardWidget[]
// widgetSelectors.selectById(state, id) — DashboardWidget | undefined
Common Mistakes
Using raw useDispatch() and useSelector() instead of typed wrappers — you lose all type inference and thunk dispatch typing.
Always create useAppDispatch and useAppSelector typed hooks in a single hooks.ts file and import those everywhere.
Typing initialState inline without an explicit interface — TypeScript infers a narrower literal type that breaks when you try to assign new values in reducers.
Always declare a named state interface and annotate const initialState: MyState = { ... }.
Exercises
1 Create a todoSlice with a Todo interface (id, text, completed, priority: 'low' | 'medium' | 'high'). Implement addTodo, toggleTodo, and setPriority reducers with proper PayloadAction types.
2 Write a createAsyncThunk that fetches a paginated list of items. Type the return as { items: Item[]; total: number; page: number }, the argument as { page: number; limit: number }, and handle rejection with a typed error object { code: number; message: string }.
3 Set up a full store with two slices, derive RootState and AppDispatch, create typed hooks, and use them in a component. Verify that useAppSelector auto-completes state paths and dispatch rejects invalid payloads.
Connections: Builds on Generics (PayloadAction<T>), Discriminated Unions (action types), Type Inference (ReturnType for RootState). Compare with Zustand (next concept) for a simpler alternative.
#2. Zustand with TypeScript zustand_typescript
Zustand is a lightweight state management library that requires far less boilerplate than Redux. Its TypeScript integration is elegant: create<T>() infers the store shape, selectors are automatically typed, and the slices pattern enables modular composition without losing type safety.

Zustand stores are plain functions that return state and actions. TypeScript infers everything from the state shape you provide to create<T>(). Unlike Redux, there are no action types, reducers, or middleware to wire up — you just define state + methods.

For larger apps, the slices pattern splits a store into composable pieces, each with its own interface. Middleware like persist and devtools require type annotations to maintain inference.

Example 1 — Basic typed store
import { create } from 'zustand';

interface CounterStore {
  count: number;
  status: 'idle' | 'loading' | 'error';
  increment: () => void;
  decrement: () => void;
  incrementBy: (amount: number) => void;
  reset: () => void;
}

const useCounterStore = create<CounterStore>((set) => ({
  count: 0,
  status: 'idle',
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  incrementBy: (amount) => set((state) => ({ count: state.count + amount })),
  //           ^? (amount: number) => void  — inferred from interface
  reset: () => set({ count: 0, status: 'idle' }),
}));

// Usage in component — selector is typed
function Counter() {
  const count = useCounterStore((state) => state.count);
  //    ^? number
  const increment = useCounterStore((state) => state.increment);
  //    ^? () => void
  return <button onClick={increment}>{count}</button>;
}
Example 2 — Slices pattern for modular state
import { create, StateCreator } from 'zustand';

// Slice 1: Auth
interface AuthSlice {
  user: { id: string; name: string } | null;
  token: string | null;
  login: (user: AuthSlice['user'], token: string) => void;
  logout: () => void;
}

const createAuthSlice: StateCreator<
  AuthSlice & UISlice,  // Full store type (for cross-slice access)
  [],                    // No middleware
  [],                    // No middleware
  AuthSlice              // This slice's type
> = (set) => ({
  user: null,
  token: null,
  login: (user, token) => set({ user, token }),
  logout: () => set({ user: null, token: null }),
});

// Slice 2: UI
interface UISlice {
  theme: 'light' | 'dark';
  sidebarOpen: boolean;
  toggleTheme: () => void;
  toggleSidebar: () => void;
}

const createUISlice: StateCreator<
  AuthSlice & UISlice, [], [], UISlice
> = (set) => ({
  theme: 'dark',
  sidebarOpen: true,
  toggleTheme: () => set((s) => ({ theme: s.theme === 'dark' ? 'light' : 'dark' })),
  toggleSidebar: () => set((s) => ({ sidebarOpen: !s.sidebarOpen })),
});

// Combine slices
const useStore = create<AuthSlice & UISlice>((...args) => ({
  ...createAuthSlice(...args),
  ...createUISlice(...args),
}));
Example 3 — Middleware typing (persist + devtools)
import { create } from 'zustand';
import { devtools, persist } from 'zustand/middleware';

interface SettingsStore {
  fontSize: number;
  language: 'en' | 'es' | 'fr' | 'de';
  setFontSize: (size: number) => void;
  setLanguage: (lang: SettingsStore['language']) => void;
}

const useSettingsStore = create<SettingsStore>()(
  devtools(
    persist(
      (set) => ({
        fontSize: 16,
        language: 'en',
        setFontSize: (size) => set({ fontSize: size }, false, 'setFontSize'),
        setLanguage: (lang) => set({ language: lang }, false, 'setLanguage'),
      }),
      {
        name: 'settings-storage',  // localStorage key
        partialize: (state) => ({  // only persist these fields
          fontSize: state.fontSize,
          language: state.language,
        }),
      }
    ),
    { name: 'SettingsStore' }  // devtools display name
  )
);

// Note: create<T>()(...) — the extra () is required for middleware type inference
Example 4 — Selector typing and re-render optimization
import { useShallow } from 'zustand/react/shallow';

interface AppStore {
  bears: number;
  fish: number;
  addBear: () => void;
  addFish: () => void;
}

const useAppStore = create<AppStore>((set) => ({
  bears: 0,
  fish: 0,
  addBear: () => set((s) => ({ bears: s.bears + 1 })),
  addFish: () => set((s) => ({ fish: s.fish + 1 })),
}));

// Single selector — component only re-renders when bears changes
function BearCount() {
  const bears = useAppStore((state) => state.bears);
  return <span>{bears} bears</span>;
}

// Multiple values — useShallow prevents re-renders from other state changes
function Dashboard() {
  const { bears, fish } = useAppStore(
    useShallow((state) => ({ bears: state.bears, fish: state.fish }))
  );
  return <span>{bears} bears, {fish} fish</span>;
}
Example 5 — Redux vs Zustand side-by-side
// Redux Toolkit approach — more ceremony, more structure
// 1. Define slice (actions inferred)
// 2. Configure store (combine reducers)
// 3. Derive RootState, AppDispatch
// 4. Create typed hooks
// 5. Wrap app in Provider
// 6. Use hooks in components
// Result: ~60 lines of boilerplate for a simple counter

// Zustand approach — minimal boilerplate, same type safety
// 1. Define interface + create store (all in one)
// 2. Use hook in components (no Provider needed)
// Result: ~15 lines for the same counter

// When to choose Redux:
// - Large teams needing enforced patterns
// - Complex middleware chains (saga, RTK Query)
// - Time-travel debugging is critical
// - Existing Redux codebase

// When to choose Zustand:
// - Small-to-medium apps
// - Rapid prototyping
// - Multiple independent stores
// - Simpler mental model preferred
Common Mistakes
Forgetting the extra () after create<T> when using middleware: create<T>(devtools(...)) loses type inference.
Use the curried form: create<T>()(devtools(...)) — the extra () enables proper middleware type inference.
Selecting the entire store: const store = useStore() — every state change triggers a re-render.
Always use granular selectors: const count = useStore(s => s.count) to minimize re-renders.

2 Advanced Type Patterns
#3. HKT Simulation & Type-Level Computation hkt_type_computation
TypeScript lacks native higher-kinded types (HKTs), but you can simulate them with interface-based encoding. Combined with type-level arithmetic using tuple lengths and recursive string parsing, these techniques power libraries like fp-ts, Zod, and type-safe query builders.

A higher-kinded type is a type that takes another type constructor as a parameter — like Functor<F> where F itself takes a type argument. TypeScript cannot express F<A> directly, but you can simulate it with an interface lookup pattern: define a URI brand and a mapping from URI to concrete type.

Type-level arithmetic uses tuple lengths: a tuple of length N represents the number N, and you can add/subtract by concatenating or slicing tuples.

Example 1 — Simulating higher-kinded types
// Step 1: Define a URI-to-Type mapping
interface URItoKind<A> {
  Array: Array<A>;
  Option: Option<A>;
  Promise: Promise<A>;
}

type URIS = keyof URItoKind<any>;

// Step 2: A "Kind" helper that looks up the mapping
type Kind<F extends URIS, A> = URItoKind<A>[F];

// Step 3: Define Functor generically over any URI
interface Functor<F extends URIS> {
  map: <A, B>(fa: Kind<F, A>, f: (a: A) => B) => Kind<F, B>;
}

// Implement for Array
const arrayFunctor: Functor<'Array'> = {
  map: (fa, f) => fa.map(f),
  // fa: Array<A>, f: (a: A) => B, returns Array<B>
};

// Implement for Option
type Option<A> = { tag: 'some'; value: A } | { tag: 'none' };

const optionFunctor: Functor<'Option'> = {
  map: (fa, f) => fa.tag === 'some' ? { tag: 'some', value: f(fa.value) } : fa,
};
Kind<'Array', number> = number[] Kind<'Option', string> = { tag: 'some'; value: string } | { tag: 'none' } arrayFunctor.map([1,2,3], x => x * 2) : number[]
Example 2 — Type-level arithmetic using tuple lengths
// Build a tuple of length N
type BuildTuple<N extends number, T extends any[] = []> =
  T['length'] extends N ? T : BuildTuple<N, [...T, unknown]>;

// Add: concatenate two tuples, return the length
type Add<A extends number, B extends number> =
  [...BuildTuple<A>, ...BuildTuple<B>]['length'];

type Three = Add<1, 2>;     // 3
type Ten = Add<4, 6>;       // 10

// Subtract: remove elements from the front
type Subtract<A extends number, B extends number> =
  BuildTuple<A> extends [...BuildTuple<B>, ...infer Rest]
    ? Rest['length']
    : never;

type Five = Subtract<8, 3>;  // 5

// Comparison
type LessThan<A extends number, B extends number> =
  Subtract<A, B> extends never ? true : false;

type Yes = LessThan<3, 5>;   // true
type No = LessThan<5, 3>;    // false
Three = 3 Ten = 10 Five = 5 Yes = true No = false
Example 3 — Type-level string parsing (route matcher)
// Parse route params from a path string
type ParseParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ParseParams<Rest>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type Params1 = ParseParams<'/users/:userId/posts/:postId'>;
// 'userId' | 'postId'

type Params2 = ParseParams<'/api/:version/items/:id/details'>;
// 'version' | 'id'

// Build a typed params object from the path
type RouteParams<T extends string> = {
  [K in ParseParams<T>]: string;
};

type UserPostParams = RouteParams<'/users/:userId/posts/:postId'>;
// { userId: string; postId: string }

// Type-safe route handler
function createRoute<T extends string>(
  path: T,
  handler: (params: RouteParams<T>) => void
) { /* ... */ }

createRoute('/users/:userId/posts/:postId', (params) => {
  console.log(params.userId);  // typed as string
  console.log(params.postId);  // typed as string
  // params.typo              // Error: Property 'typo' does not exist
});
Params1 = "userId" | "postId" UserPostParams = { userId: string; postId: string }
Example 4 — Type-level state machine
// Define valid state transitions at the type level
interface StateTransitions {
  idle: 'loading';
  loading: 'success' | 'error';
  success: 'idle';
  error: 'idle' | 'loading';
}

type ValidStates = keyof StateTransitions;

// A machine that enforces valid transitions at compile time
class StateMachine<Current extends ValidStates> {
  constructor(private state: Current) {}

  transition<Next extends StateTransitions[Current]>(
    next: Next
  ): StateMachine<Next & ValidStates> {
    return new StateMachine(next as Next & ValidStates);
  }

  getState(): Current { return this.state; }
}

const machine = new StateMachine('idle');
const loading = machine.transition('loading');    // OK
const success = loading.transition('success');    // OK
const backToIdle = success.transition('idle');    // OK
// loading.transition('idle');  // Error: 'idle' not assignable to 'success' | 'error'
machine: StateMachine<"idle"> loading: StateMachine<"loading"> success: StateMachine<"success"> backToIdle: StateMachine<"idle">
Example 5 — Practical: type-safe query builder
// A simplified type-safe query builder
interface Schema {
  users: { id: number; name: string; email: string; age: number };
  posts: { id: number; title: string; body: string; authorId: number };
}

type QueryBuilder<
  Table extends keyof Schema,
  Selected extends keyof Schema[Table] = never
> = {
  select<K extends keyof Schema[Table]>(
    ...columns: K[]
  ): QueryBuilder<Table, Selected | K>;

  where(
    column: keyof Schema[Table],
    value: Schema[Table][keyof Schema[Table]]
  ): QueryBuilder<Table, Selected>;

  execute(): Promise<
    [Selected] extends [never]
      ? Schema[Table][]               // No select = all columns
      : Pick<Schema[Table], Selected>[]  // Only selected columns
  >;
};

declare function from<T extends keyof Schema>(table: T): QueryBuilder<T>;

// Usage — return type changes based on .select() calls
const allUsers = from('users').execute();
// Promise<{ id: number; name: string; email: string; age: number }[]>

const nameAndEmail = from('users').select('name', 'email').execute();
// Promise<{ name: string; email: string }[]>
allUsers: Promise<{ id: number; name: string; email: string; age: number }[]> nameAndEmail: Promise<{ name: string; email: string }[]>
Common Mistakes
Deeply recursive types without a base case or depth limit — TypeScript has a recursion limit (~50 levels) and will error with "Type instantiation is excessively deep."
Always include a recursion terminator and consider a depth counter type parameter that counts down to 0.
Using HKT simulation for simple problems — the encoding adds complexity that is only justified for library-level abstractions like functors or monads.
Prefer simple generics and conditional types for application code. Reserve HKT patterns for library APIs.

#4. Declaration Merging & Module Augmentation declaration_merging
Declaration merging is how TypeScript combines multiple declarations with the same name into a single definition. Module augmentation lets you extend third-party types without modifying their source — essential for adding custom properties to Express's Request, extending Window, or typing environment variables.

Interfaces with the same name in the same scope automatically merge. This is why you can add properties to Window or Express.Request — you are merging into the existing interface declaration.

Module augmentation uses declare module '...' to extend types from external packages. Global augmentation uses declare global within a module file to extend the global scope.

Classes, enums, and namespaces can also merge, but the rules differ: class + namespace merging adds static members, enum + namespace adds companion methods.

Example 1 — Interface merging (same-scope)
// Both declarations merge into one interface
interface User {
  id: number;
  name: string;
}

interface User {
  email: string;
  role: 'admin' | 'user';
}

// Result: User has all four properties
const user: User = {
  id: 1,
  name: 'Alice',
  email: 'alice@example.com',
  role: 'admin',
};

// This is why 'type' aliases CANNOT merge — they error on duplicate names
// type User = { id: number };
// type User = { email: string }; // Error: Duplicate identifier 'User'

// Practical use: extending a library's interface
// The library declares Window; you add to it
interface Window {
  __REDUX_DEVTOOLS_EXTENSION__?: () => any;
  gtag?: (...args: any[]) => void;
}
User = { id: number; name: string; email: string; role: "admin" | "user" }
Example 2 — Module augmentation: extending Express
// types/express.d.ts
import { User as AppUser } from '../models/user';

// Augment the express-serve-static-core module
declare module 'express-serve-static-core' {
  interface Request {
    user?: AppUser;            // Added by auth middleware
    sessionId?: string;        // Added by session middleware
    requestId: string;         // Added by request-id middleware
  }
}

// Now in your route handlers:
import { Request, Response } from 'express';

function protectedRoute(req: Request, res: Response) {
  if (!req.user) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  // req.user is typed as AppUser — auto-complete works!
  console.log(req.user.name);
  console.log(req.requestId);  // string — no type error
}
Example 3 — Global augmentation: ProcessEnv & Window
// types/env.d.ts — must have an import/export to be a module
export {};

declare global {
  namespace NodeJS {
    interface ProcessEnv {
      NODE_ENV: 'development' | 'production' | 'test';
      DATABASE_URL: string;
      API_KEY: string;
      PORT?: string;    // optional
    }
  }

  interface Window {
    __APP_CONFIG__: {
      apiUrl: string;
      version: string;
      features: Record<string, boolean>;
    };
  }
}

// Usage — fully typed
const dbUrl = process.env.DATABASE_URL;  // string (not string | undefined)
const env = process.env.NODE_ENV;        // 'development' | 'production' | 'test'
// process.env.TYPO                      // Error if strict

const config = window.__APP_CONFIG__;    // typed object
console.log(config.apiUrl);              // string
dbUrl: string env: "development" | "production" | "test" config: { apiUrl: string; version: string; features: Record<string, boolean> }
Example 4 — Namespace + class merging (static companions)
// Class declaration
class Validator<T> {
  constructor(private value: T) {}
  validate(predicate: (v: T) => boolean): boolean {
    return predicate(this.value);
  }
}

// Namespace merges as static members of the class
namespace Validator {
  export function isEmail(value: string): boolean {
    return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value);
  }
  export function isPositive(value: number): boolean {
    return value > 0;
  }
  export interface Config {
    strict: boolean;
    maxLength: number;
  }
}

// Usage — both instance methods and static helpers
const v = new Validator('test@example.com');
v.validate(Validator.isEmail);         // true — static method as predicate
Validator.isPositive(42);              // true — standalone static call

const config: Validator.Config = {     // namespace-exported interface
  strict: true,
  maxLength: 100,
};
Example 5 — When merging causes problems
// Problem 1: Accidental global pollution
// If your .d.ts file has NO import/export, everything is global!
// types/bad.d.ts (NO import/export statement)
interface User {        // This merges with EVERY User interface globally!
  debugFlag: boolean;
}
// Fix: Always include an export {} or import to make it a module

// Problem 2: Conflicting augmentations across files
// file-a.d.ts
declare module 'express-serve-static-core' {
  interface Request { user: AdminUser; }  // user is AdminUser
}
// file-b.d.ts
declare module 'express-serve-static-core' {
  interface Request { user: BasicUser; }  // user is BasicUser
}
// Result: user is AdminUser & BasicUser — probably not what you want!
// Fix: Centralize augmentations in a single file

// Problem 3: Merging function overloads across declarations
interface EventEmitter {
  on(event: 'click', handler: (e: MouseEvent) => void): void;
}
interface EventEmitter {
  on(event: 'keydown', handler: (e: KeyboardEvent) => void): void;
}
// Both overloads exist, but overloads from the FIRST declaration
// take priority over the second. Order matters!
Common Mistakes
Writing a .d.ts augmentation file without any import or export — it becomes a global script file and all interfaces pollute the global scope.
Always include at least export {} at the top of augmentation files to ensure they are treated as modules.
Trying to use type aliases for merging: type Window = { myProp: string } errors with "Duplicate identifier."
Only interface declarations merge. Use interface Window { myProp: string } instead.

2 Create a declare global block that types process.env with your project's required and optional environment variables. Verify that accessing a required variable returns string (not string | undefined).
3 Use namespace + class merging to create a Result class with instance methods (map, flatMap) and static factory methods (Result.ok(), Result.err()) in the merged namespace.
Connections: Uses Interface fundamentals. Extends Module Systems (import/export, ambient modules). Pairs with Ambient Declarations (next concept). Essential for Module Federation Typing (reviewed today).
#5. Ambient Declarations & Global Types ambient_declarations
Ambient declarations (declare) describe the shape of code that exists at runtime but was not written in TypeScript — globals injected by the browser, untyped npm packages, legacy JavaScript libraries, or values injected by bundlers. Without them, TypeScript cannot see these runtime values.

declare tells TypeScript "this exists at runtime; trust me on the shape." No JavaScript is emitted for declare statements. You use declare const, declare function, and declare class for individual globals. declare module provides types for untyped third-party packages. .d.ts files contain only declarations and are never compiled to JavaScript.

Triple-slash directives (/// <reference types="..." />) include type packages without importing them, useful for ambient type dependencies like @types/node or @types/jest.

Example 1 — declare const, function, class
// These describe runtime globals — no JS is emitted

// A global constant (e.g., injected by a build tool)
declare const __DEV__: boolean;
declare const __VERSION__: string;
declare const __BUILD_TIMESTAMP__: number;

if (__DEV__) {
  console.log(`Version: ${__VERSION__}, built: ${__BUILD_TIMESTAMP__}`);
}

// A global function (e.g., from a legacy script tag)
declare function analytics(event: string, data?: Record<string, unknown>): void;

analytics('page_view', { path: '/home' });

// A global class (e.g., from a third-party library loaded via CDN)
declare class Stripe {
  constructor(publishableKey: string);
  createToken(card: { number: string; exp_month: number; exp_year: number }): Promise<{
    token: { id: string };
    error?: { message: string };
  }>;
}

const stripe = new Stripe('pk_test_...');
const result = await stripe.createToken({
  number: '4242424242424242',
  exp_month: 12,
  exp_year: 2027,
});
// result.token.id is typed as string
Example 2 — declare module for untyped npm packages
// types/untyped-libs.d.ts
// When a package has no @types/* and no built-in types

// Minimal: just silence the import error (everything is 'any')
declare module 'legacy-csv-parser';

// Better: declare the actual exports you use
declare module 'fast-image-resize' {
  interface ResizeOptions {
    width: number;
    height: number;
    quality?: number;  // 0-100
    format?: 'jpeg' | 'png' | 'webp';
  }

  interface ResizeResult {
    buffer: Buffer;
    width: number;
    height: number;
    format: string;
  }

  export function resize(
    input: Buffer | string,
    options: ResizeOptions
  ): Promise<ResizeResult>;

  export function getMetadata(
    input: Buffer | string
  ): Promise<{ width: number; height: number; format: string }>;

  export default { resize, getMetadata };
}

// Now you get full type checking
import { resize } from 'fast-image-resize';

const result = await resize('./photo.jpg', {
  width: 800,
  height: 600,
  quality: 85,
  format: 'webp',
});
console.log(result.width, result.height);  // typed as number
Example 3 — .d.ts files: when and how to create them
// When to create .d.ts files:

Common Mistakes
Putting runtime code in a .d.ts file: export const x = 42; in a .d.ts file causes a compile error because .d.ts files cannot contain implementations.
.d.ts files are declarations only: export declare const x: number;. Implementations go in .ts files.
Using declare module '*' to suppress all import errors — this makes every import any, defeating the purpose of TypeScript.
Type specific modules: declare module 'specific-package' with at least the exports you use. Only use wildcard modules for asset imports like declare module '*.svg'.
Key Rules for Ambient Declarations
  • 2026-07-04 (Day 31)
  • declare = "this exists at runtime, here is its type" — no JS emitted
  • .d.ts files are always declarations — never contain implementations
  • A file with no import/export is a script (global scope); add export {} to make it a module
  • Use declare global inside a module file to extend global scope
  • Triple-slash directives must appear before any code in the file
  • Prefer tsconfig.json "types" over triple-slash for project-wide type packages
