R1. Type-Safe API Client Patterns Patterns overdue · rep 0 → 1
A type-safe client centralizes the contract — an endpoint map keyed by route, each value carrying its request and response shapes — so every call site infers the right body and return type from the route string alone. The boundary still validates at runtime (the bytes are external), but the call ergonomics are fully typed and refactor-safe.
Key Pattern 
interface Endpoints {
  "GET /users/:id": { params: { id: string }; response: User };
  "POST /users":    { body: { name: string }; response: User };
}

async function call<K extends keyof Endpoints>(
  key: K,
  init: Omit<Endpoints[K], "response">,
): Promise<Endpoints[K]["response"]> {
  /* build url + fetch + validate */
  return undefined as never;
}

const u = await call("POST /users", { body: { name: "Ada" } }); // u: User
route string drives both the request shape and the response type
Challenge Exercise

Extend Endpoints with "GET /users" returning User[] and no body. Make call reject a body for GET routes at compile time (hint: a conditional on whether the entry has a body key). Confirm call("GET /users", {}) is typed and call("GET /users", { body: {} }) errors.

Pitfall: Typing the client so that the response is asserted (as User) instead of validated. The endpoint map gives compile-time safety only; the server can still return the wrong shape. Parse the response with a per-route schema (Zod) and treat a mismatch as an error, not a cast.
R2. TypeScript Compiler API Tooling & Compiler overdue · rep 0 → 1
The typescript package is itself a library: ts.createProgram builds a typed view of your files, the TypeChecker answers questions about symbols and types, and you walk the AST with ts.forEachChild. This is the foundation under linters, codemods, and doc generators — anything that needs real type information, not just syntax.
Key Pattern 
import ts from "typescript";

const program = ts.createProgram(["src/index.ts"], {
  target: ts.ScriptTarget.ES2022,
});
const checker = program.getTypeChecker();

for (const sf of program.getSourceFiles()) {
  if (sf.isDeclarationFile) continue;
  ts.forEachChild(sf, function visit(node) {
    if (ts.isFunctionDeclaration(node) && node.name) {
      const sym = checker.getSymbolAtLocation(node.name);
      const type = checker.getTypeOfSymbolAtLocation(sym!, node.name);
      console.log(node.name.text, checker.typeToString(type));
    }
    ts.forEachChild(node, visit);
  });
}
greet (name: string) => string
Challenge Exercise

Write a script that prints every exported function’s name and parameter types from a file. Use checker.getSymbolAtLocation + getSignaturesOfType to read each signature, and filter to symbols whose flags include ts.SymbolFlags.Function. Bonus: emit the result as a small .d.ts-style summary.

Pitfall: Reading types off the AST node directly instead of going through the TypeChecker. The raw AST only has syntax — inferred types, resolved imports, and symbol identity all live in the checker. Also: a Program is expensive to build; reuse it (or a LanguageService) rather than recreating it per file in a watch loop.
R3. Type-Safe Dependency Injection Architecture overdue · rep 0 → 1
You can get compile-time-safe DI without decorators or reflect-metadata: model the container as a token → type map, and have resolve(token) return the mapped type. Tokens are typically unique symbols or branded keys so two services with the same shape stay distinct.
Key Pattern 
interface Services {
  logger: { log(msg: string): void };
  clock:  { now(): number };
}

class Container {
  private reg = new Map<keyof Services, unknown>();
  register<K extends keyof Services>(k: K, v: Services[K]) { this.reg.set(k, v); }
  resolve<K extends keyof Services>(k: K): Services[K] {
    const v = this.reg.get(k);
    if (!v) throw new Error(`Unregistered: ${String(k)}`);
    return v as Services[K];
  }
}

const c = new Container();
c.register("clock", { now: () => Date.now() });
const t = c.resolve("clock").now();   // t: number, fully typed
resolve("clock") => { now(): number }
Challenge Exercise

Add a factory registration so a service can be built lazily from other resolved services (register("a", (c) => new A(c.resolve("logger")))). Keep the factory’s container parameter typed against Services, and ensure a cycle (a needs b, b needs a) is detected at runtime with a clear error.

Pitfall: Leaning on decorator-based DI (@Injectable + reflect-metadata) when a typed token map would do. The metadata approach needs emitDecoratorMetadata, ties you to legacy decorators, and erases to runtime strings that the compiler can’t check. The map-based container keeps everything in the type system with zero runtime reflection.
R4. Type-Safe Routing Patterns Application Patterns overdue · rep 0 → 1
Template-literal types let a route pattern string drive its params type: "/user/:id/post/:slug" can be parsed at the type level into { id: string; slug: string }. That makes navigate / link helpers reject missing or misspelled params before runtime.
Key Pattern 
type PathParams<S extends string> =
  S extends `${string}:${infer P}/${infer Rest}`
    ? { [K in P | keyof PathParams<`/${Rest}`>]: string }
    : S extends `${string}:${infer P}`
      ? { [K in P]: string }
      : {};

type P = PathParams<"/user/:id/post/:slug">;
// { id: string; slug: string }

function link<S extends string>(pattern: S, params: PathParams<S>): string {
  return pattern.replace(/:([^/]+)/g, (_, k) => String((params as never)[k]));
}
link("/user/:id/post/:slug", { id: "1", slug: "hi" });   // ok
P = { id: string; slug: string }
Challenge Exercise

Extend PathParams to support optional segments (:id?) by making those keys optional in the result, and typed query params via a ?a&b suffix. Confirm link("/post/:id?") accepts both {} and { id: "1" }, while a required :id still demands the key.

Pitfall: Letting params widen to Record<string, string> so any key compiles. The whole point is that :slug typed as :slog at a call site fails. Keep the pattern as a const / literal type parameter through the helper, otherwise the template-literal inference never fires and you lose the per-route key checking.
R5. Effect Pattern & Functional Error Handling Functional Patterns overdue · rep 0 → 1
Instead of throwing, you return a Result (a.k.a. Either): a discriminated union of { ok: true; value } and { ok: false; error }. Errors become part of the type signature — the caller must narrow on ok before touching value, so no failure path is silently skipped.
Key Pattern 
type Result<T, E = Error> =
  | { ok: true;  value: T }
  | { ok: false; error: E };

const ok  = <T>(value: T): Result<T, never> => ({ ok: true, value });
const err = <E>(error: E): Result<never, E> => ({ ok: false, error });

function parsePort(s: string): Result<number, "NaN" | "range"> {
  const n = Number(s);
  if (Number.isNaN(n)) return err("NaN");
  if (n < 0 || n > 65535) return err("range");
  return ok(n);
}

const r = parsePort("8080");
if (r.ok) console.log(r.value + 1);   // value: number, narrowed
else console.error(r.error);          // error: "NaN" | "range"
value is only reachable inside the `r.ok` branch
Challenge Exercise

Write map and flatMap (a.k.a. andThen) over Result so you can chain fallible steps without nested ifs, short-circuiting on the first error. Then write all(results: Result<T, E>[]): Result<T[], E> that collects values or returns the first error. Keep the error channel’s union type accumulating correctly.

Pitfall: Reintroducing exceptions in the middle of a Result pipeline (a helper that throws). Once you commit to Result, a stray throw bypasses the typed error channel and the caller’s exhaustive handling. Wrap any throwing boundary (JSON.parse, fetch) in a tryCatch that converts the throw into an err(...).
R6. React + TypeScript: Generic Components React + TypeScript overdue · rep 1 → 2
A generic component threads a type parameter from its props (e.g. the item type of a List<T>) into its callbacks so renderItem receives a fully typed T. The catch is the function-component form: an arrow with React.FC can’t be generic, so you write a generic function declaration (or a generic arrow assigned without FC).
Key Pattern 
interface ListProps<T> {
  items: T[];
  renderItem: (item: T, index: number) => React.ReactNode;
  keyOf: (item: T) => React.Key;
}

function List<T>({ items, renderItem, keyOf }: ListProps<T>) {
  return <ul>{items.map((it, i) => (
    <li key={keyOf(it)}>{renderItem(it, i)}</li>
  ))}</ul>;
}

// T inferred as User from `items`; `u` is User inside renderItem
<List items={users} keyOf={(u) => u.id} renderItem={(u) => u.name} />;
T inferred from items; renderItem param typed as User
Challenge Exercise

Build a generic <Select<T>> whose options: T[], value: T, and onChange: (next: T) => void all share T, plus a getLabel: (t: T) => string. Make it work with forwardRef (which loses generics by default) by typing the wrapped component manually so T still infers from options.

Pitfall: Annotating the component as const List: React.FC<ListProps<T>> — T isn’t in scope there and FC isn’t generic, so you end up hard-coding T or falling back to any. Use a generic function declaration; if you need forwardRef/memo, cast the wrapper to a generic call signature so inference is preserved.
1Typed Object Modeling
1. Type-Safe Deep Property Paths Type-Level
Any library that takes a dotted path — form libraries ("address.city"), i18n keys, lodash get/set, ORM selectors — lives or dies on whether that string is checked. Build Paths<T> and PathValue<T, P> once and a typo in a nested key becomes a compile error instead of an undefined at runtime.

The idea: recursively walk keyof T, and for each key whose value is an object, emit both the key itself and the key joined with the paths of its child via a template literal (`${K}.${ChildPath}`). The companion PathValue<T, P> splits the path back apart with infer and indexes into T step by step, so a valid path resolves to the type sitting at its end.

The two pieces are duals: Paths produces the set of legal strings; PathValue maps a legal string to the type it points at. A typed get(obj, path) then constrains path to Paths<T> and returns PathValue<T, path> — full autocomplete on the dotted string, exact return type at the leaf.

1
Recurse over keyof T; for object values, union the key with `${K}.${child}`.
2
PathValue splits on . with infer Head/infer Rest and indexes recursively.
3
Constrain the runtime get’s path to Paths<T> and return PathValue<T, P>.
TypeScript · example 1 of 4 — generating the path union
type Paths<T> = T extends object
  ? {
      [K in keyof T & string]:
        T[K] extends object ? K | `${K}.${Paths<T[K]>}` : K;
    }[keyof T & string]
  : never;

type Data = { user: { name: string; address: { city: string } }; id: number };
type P = Paths<Data>;
// "user" | "id" | "user.name" | "user.address" | "user.address.city"
P includes every nested dotted path of Data
TypeScript · example 2 of 4 — resolving the value at a path
type PathValue<T, P extends string> =
  P extends `${infer Head}.${infer Rest}`
    ? Head extends keyof T ? PathValue<T[Head], Rest> : never
    : P extends keyof T ? T[P] : never;

type V1 = PathValue<Data, "user.address.city">;   // string
type V2 = PathValue<Data, "id">;                   // number
type V3 = PathValue<Data, "user.address">;         // { city: string }
V1 = string; V2 = number; V3 = { city: string }
TypeScript · example 3 of 4 — a typed get(obj, path)
function get<T, P extends Paths<T>>(obj: T, path: P): PathValue<T, P> {
  return (path as string)
    .split(".")
    .reduce<unknown>((acc, k) => (acc as Record<string, unknown>)[k], obj) as PathValue<T, P>;
}

const d: Data = { user: { name: "Ada", address: { city: "London" } }, id: 1 };
const city = get(d, "user.address.city");   // city: string
// get(d, "user.adress.city");               // Error: not a valid path
city: string; an invalid path is a compile error
TypeScript · example 4 of 4 — guarding arrays and depth
// arrays would otherwise spray "0" | "1" | "length" | "map" ... into Paths
type SafePaths<T> = T extends readonly unknown[]
  ? never                                  // stop at arrays (or handle [number])
  : T extends object
    ? { [K in keyof T & string]:
          T[K] extends object & { length?: never }
            ? K | `${K}.${SafePaths<T[K]>}`
            : K;
      }[keyof T & string]
    : never;

type Cfg = { db: { host: string }; tags: string[] };
type CP = SafePaths<Cfg>;   // "db" | "tags" | "db.host"
CP = "db" | "tags" | "db.host" (no array index noise)
Common Mistakes:
❌ Recursing into arrays naively, so string[] contributes "tags.0" | "tags.length" | "tags.map" … flooding the union with array internals.
✅ Special-case arrays first — stop, or deliberately emit a `${K}.${number}` index path if you actually want element access.
❌ Letting path widen to string in get, which disables both autocomplete and the leaf return type.
✅ Constrain P extends Paths<T> so the literal is preserved and PathValue<T, P> can resolve the exact leaf.
Key Rules:
  • 2026-07-04 (Day 31)
  • Paths<T> = union of K and `${K}.${Paths<child>}` for each object key.
  • PathValue<T, P> splits on . with infer and indexes recursively to the leaf.
  • Guard arrays (and other built-ins) so their members don’t pollute the path union.
  • Deeply nested shapes can approach the instantiation-depth cap — keep object depth reasonable.
Practice Exercises:
1Write a typed set<T, P extends Paths<T>>(obj: T, path: P, value: PathValue<T, P>): void (mutating) and confirm the value type is checked against the leaf.
2Add array support: extend Paths to emit `${K}.${number}` for array properties and have PathValue resolve element types.
3Write LeafPaths<T> that emits only paths ending at primitives (no intermediate object paths like "user.address").
Connections: This is template-literal infer (R4 routing, and infer … extends from Day 23) applied to objects, and it leans on keyof & Indexed Access Types. The array guard echoes the recursion guards in DeepPartial / DeepReadonly.
2. Key Selection by Value Type Type-Level
Often you want “all the keys whose value is a function” (to build a method map), or “all the string fields” (to build a search index). Pick/Omit select by key name; selecting by value type is a distinct, extremely reusable tool that underlies event maps, ORMs, and form schemas.

The core trick is a key-remapping mapped type with as: map over keyof T, and for each key conditionally remap it to K if its value matches the target type, or to never otherwise. Keys mapped to never are dropped from the result, so keyof of that object yields exactly the matching keys.

From KeysMatching<T, V> you build PickByValue (Pick<T, KeysMatching<T, V>>) and its inverse OmitByValue. Watch the matching direction: T[K] extends V is assignability, so KeysMatching<T, string> also catches string-literal fields, and a field typed any matches everything — use a stricter equality check when that bites.

1
Map K in keyof T, remapping with as [T[K]] extends [V] ? K : never.
2
never-remapped keys vanish; keyof the result is the matching key set.
3
Compose into PickByValue / OmitByValue via Pick/Omit.
TypeScript · example 1 of 4 — keys whose value matches a type
type KeysMatching<T, V> = {
  [K in keyof T]-?: [T[K]] extends [V] ? K : never;
}[keyof T];

interface Model {
  id: number;
  name: string;
  active: boolean;
  tags: string[];
}

type StringKeys = KeysMatching<Model, string>;   // "name"
type NumberKeys = KeysMatching<Model, number>;   // "id"
StringKeys = "name"; NumberKeys = "id"
TypeScript · example 2 of 4 — PickByValue / OmitByValue
type PickByValue<T, V> = Pick<T, KeysMatching<T, V>>;
type OmitByValue<T, V> = Omit<T, KeysMatching<T, V>>;

type OnlyPrimitives = PickByValue<Model, string | number | boolean>;
// { id: number; name: string; active: boolean }

type NoArrays = OmitByValue<Model, unknown[]>;
// { id: number; name: string; active: boolean }
PickByValue keeps matching fields; OmitByValue drops them
TypeScript · example 3 of 4 — extracting a method map
type Fn = (...args: any[]) => any;

class Widget {
  width = 0;
  label = "";
  resize(w: number) { this.width = w; }
  render() { return this.label; }
}

type Methods = PickByValue<Widget, Fn>;
// { resize: (w: number) => void; render: () => string }
type MethodNames = KeysMatching<Widget, Fn>;   // "resize" | "render"
Methods = { resize; render }; MethodNames = "resize" | "render"
TypeScript · example 4 of 4 — assignability vs exact match
interface Mixed { a: string; b: "x" | "y"; c: any }

// assignability: "x" | "y" IS a string, so b matches too; any matches all
type Loose = KeysMatching<Mixed, string>;   // "a" | "b" | "c"

// exact equality via mutual extends keeps only true `string` fields
type Eq<A, B> = [A] extends [B] ? ([B] extends [A] ? true : false) : false;
type ExactString = {
  [K in keyof Mixed]-?: Eq<Mixed[K], string> extends true ? K : never;
}[keyof Mixed];                              // "a"
Loose = "a" | "b" | "c"; ExactString = "a"
Common Mistakes:
❌ Forgetting the -? modifier, so optional properties leave a stray undefined in the remapped union and matching gets confused.
✅ Add -? in the mapped type to strip optionality before the conditional, then re-derive keys cleanly.
❌ Assuming KeysMatching<T, string> returns only fields typed exactly string — literals and any slip through via assignability.
✅ Use a mutual-extends equality check when you need exact value-type matching rather than assignability.
Key Rules:
  • Key-remap with as ... ? K : never; never keys drop out of the result.
  • KeysMatching + Pick/Omit gives PickByValue/OmitByValue.
  • Default matching is assignability (extends); switch to mutual-extends for exactness.
  • Wrap both sides in tuples ([T[K]] extends [V]) to avoid accidental distribution over unions.
Practice Exercises:
1Write RequiredKeys<T> and OptionalKeys<T> that select keys by their modifier (hint: compare {} extends Pick<T, K>).
2Build FunctionProperties<T> and a runtime bindAll(obj) whose return type is PickByValue<T, Fn> with every method bound to obj.
3Write NullableKeys<T> selecting keys whose value includes null or undefined, and confirm it ignores non-nullable fields.
Connections: This is key remapping (Mapped Type Key Remapping, Mapped Type Modifiers) used as a filter, and the equality check reuses the non-distributive tuple-wrap trick from Distributive Conditional Types. PickByValue<T, Fn> feeds directly into the method-binding helpers of Concept 4.
3. The Recursive JSON Type & Typed Parse Boundary Runtime Safety
Almost every app crosses the JSON boundary — fetch, files, message queues — and JSON.parse hands you any, the type system’s off-switch. A precise recursive Json type plus a validating parse step replaces that any with a value the compiler actually understands, so the boundary is the only place you reason about untrusted shape.

JSON has exactly five primitive forms plus arrays and objects, and it is recursive: arrays and objects contain more JSON. That maps to a small mutually-recursive type — JsonPrimitive, JsonArray = JsonValue[], and JsonObject = { [k: string]: JsonValue }. Typing a parse function’s return as JsonValue (instead of any) forces you to narrow before using the data, which is exactly the discipline you want at a trust boundary.

The companion idea is Jsonify<T>: a serialize step doesn’t round-trip everything. Date becomes a string, undefined properties and methods disappear, Map/Set become {}. Modeling that at the type level stops you from assuming a parsed payload still has the Date or method it had before JSON.stringify.

1
Define JsonValue as primitives ∪ JsonValue[] ∪ { [k: string]: JsonValue }.
2
Type the parse boundary as JsonValue (or validate to a known schema), never any.
3
Use Jsonify<T> to model what actually survives stringify → parse.
TypeScript · example 1 of 4 — the recursive JSON value type
type JsonPrimitive = string | number | boolean | null;
type JsonArray = JsonValue[];
type JsonObject = { [key: string]: JsonValue };
type JsonValue = JsonPrimitive | JsonArray | JsonObject;

const config: JsonValue = {
  name: "app",
  retries: 3,
  flags: [true, false],
  nested: { enabled: null },
};   // ok — every leaf is JSON-legal
// const bad: JsonValue = { when: new Date() };   // Error: Date not assignable
JsonValue admits only JSON-serializable shapes
TypeScript · example 2 of 4 — a parse boundary that returns JsonValue, not any
function parseJson(text: string): JsonValue {
  return JSON.parse(text);   // widened from any to JsonValue at the boundary
}

const data = parseJson('{"id":1,"tags":["a"]}');
// data.id;                  // Error: Property 'id' does not exist on JsonValue
if (typeof data === "object" && data !== null && !Array.isArray(data)) {
  const id = data.id;        // JsonValue — still must narrow further
}
forced narrowing instead of an `any` free-for-all
TypeScript · example 3 of 4 — validate to a real type with a guard
interface User { id: number; name: string }

function isUser(v: JsonValue): v is JsonObject & { id: number; name: string } {
  return (
    typeof v === "object" && v !== null && !Array.isArray(v) &&
    typeof v.id === "number" && typeof v.name === "string"
  );
}

const v = parseJson('{"id":1,"name":"Ada"}');
const user: User | null = isUser(v) ? { id: v.id, name: v.name } : null;
user: User | null — shape verified at the boundary
TypeScript · example 4 of 4 — Jsonify: what survives a round-trip
type Jsonify<T> =
  T extends JsonPrimitive ? T
  : T extends Date ? string                       // Date => ISO string
  : T extends (...a: any[]) => any ? never        // functions dropped
  : T extends readonly (infer E)[] ? Jsonify<E>[]
  : T extends object ? { [K in keyof T]: Jsonify<T[K]> }
  : never;

type Raw = { id: number; created: Date; greet: () => void };
type Wire = Jsonify<Raw>;
// { id: number; created: string; greet: never }  — greet effectively gone
created becomes string; the method does not survive
Common Mistakes:
❌ Typing JSON.parse’s result as your domain type (as User) and trusting it — the bytes are external, so the cast can be wrong and the bug surfaces deep in the UI.
✅ Return JsonValue (or validate with a guard/Zod) so a wrong shape is caught at the boundary, not three components later.
❌ Assuming a parsed payload still has the Date objects or methods it had before serializing — JSON.stringify silently dropped them.
✅ Model the wire shape with Jsonify<T> so the compiler knows created is a string, not a Date.
Key Rules:
  • JsonValue is mutually recursive: primitives, JsonValue[], and string-keyed JsonValue records.
  • Type parse boundaries as JsonValue (not any) so callers must narrow.
  • Validate to domain types with a type guard or schema before using the data.
  • Jsonify<T> models the lossy stringify → parse round-trip (Date→string, methods/undefined dropped).
Practice Exercises:
1Write safeParseJson(text): Result<JsonValue, "syntax"> (reuse the Result type from R5) that turns a thrown SyntaxError into an error value.
2Extend Jsonify to map undefined-valued optional properties to absent keys and bigint to a compile error (it can’t serialize).
3Write a generic fetchJson<T>(url, guard: (v: JsonValue) => v is T): Promise<T> that validates the response and rejects on a failed guard.
Connections: Validate-at-the-boundary is the same stance as Zod & Runtime Validation, Type-Safe localStorage (Day 21), and Type-Safe API Client (R1). Jsonify reuses the conditional + infer recursion patterns from DeepReadonly, and the guard ties into Type Predicates.
2Function Signature Recipes
4. Signature-Preserving Higher-Order Wrappers Functions
Wrappers like memoize, once, debounce, and logged are everywhere, and the lazy way to type them (fn: Function or (...a: any[]) => any) throws away the very signature callers depend on. Preserving the exact parameter and return types — via a generic constrained to the function type — is what keeps a wrapped function a drop-in replacement.

The pattern is a single generic F extends (...args: any[]) => any, then express the returned function’s signature with Parameters<F> and ReturnType<F> (and ThisParameterType<F> when this matters). Constraining the parameter as F — rather than spelling out (...args: A) => R with two parameters — lets the wrapped function preserve overloads and parameter names for tooltips.

When the wrapper changes the shape (e.g. debounce returns void because the result is deferred, or memoize requires the args to be usable as a cache key) you encode that in the return signature while still deriving from F. The discipline: derive everything you can from one function type parameter, and only diverge where the runtime behavior genuinely differs.

1
Take one generic F extends (...args: any[]) => any for the wrapped function.
2
Build the result type from Parameters<F> / ReturnType<F> / ThisParameterType<F>.
3
Diverge only where behavior differs (debounce → void return; memoize → key constraint).
TypeScript · example 1 of 4 — once: same signature, computed once
function once<F extends (...args: any[]) => any>(fn: F): F {
  let called = false;
  let result: ReturnType<F>;
  return function (this: unknown, ...args: Parameters<F>): ReturnType<F> {
    if (!called) { called = true; result = fn.apply(this, args); }
    return result;
  } as F;
}

const init = once((port: number) => ({ port, started: true }));
const a = init(3000);   // { port: number; started: boolean } — exact signature kept
init keeps the (port: number) => {...} signature
TypeScript · example 2 of 4 — memoize with a typed cache key
function memoize<A extends unknown[], R>(
  fn: (...args: A) => R,
  keyOf: (...args: A) => string = (...a) => JSON.stringify(a),
): (...args: A) => R {
  const cache = new Map<string, R>();
  return (...args: A): R => {
    const k = keyOf(...args);
    if (!cache.has(k)) cache.set(k, fn(...args));
    return cache.get(k)!;
  };
}

const slow = (n: number) => n * n;
const fast = memoize(slow);
fast(9);   // 81, computed once; fast still (n: number) => number
fast: (n: number) => number; args and return inferred from fn
TypeScript · example 3 of 4 — debounce returns void (result is deferred)
function debounce<F extends (...args: any[]) => void>(
  fn: F,
  ms: number,
): (...args: Parameters<F>) => void {
  let t: ReturnType<typeof setTimeout> | undefined;
  return (...args: Parameters<F>): void => {
    clearTimeout(t);
    t = setTimeout(() => fn(...args), ms);
  };
}

const onResize = debounce((w: number, h: number) => { /* ... */ }, 200);
onResize(800, 600);     // args type-checked; return is void, not the inner result
onResize: (w: number, h: number) => void
TypeScript · example 4 of 4 — preserving `this` with ThisParameterType
function logged<F extends (...args: any[]) => any>(fn: F, label: string): F {
  return function (this: ThisParameterType<F>, ...args: Parameters<F>) {
    console.log(`[${label}]`, args);
    return fn.apply(this, args) as ReturnType<F>;
  } as F;
}

const obj = {
  factor: 2,
  scale: logged(function (this: { factor: number }, n: number) {
    return n * this.factor;
  }, "scale"),
};
obj.scale(10);   // logs then returns 20; `this` stays typed as { factor: number }
[scale] [ 10 ]
Common Mistakes:
❌ Typing the parameter as Function or (...a: any[]) => any and returning the same — callers lose all argument and return-type checking on the wrapped function.
✅ Use a generic F extends (...args: any[]) => any and rebuild the signature from Parameters<F>/ReturnType<F>.
❌ Dropping this when wrapping a method, so fn.apply(this, args) is untyped and methods relying on this break.
✅ Thread ThisParameterType<F> through the returned function so the receiver type is preserved.
Key Rules:
  • One generic F extends (...args: any[]) => any captures the whole signature to preserve.
  • Parameters<F>, ReturnType<F>, ThisParameterType<F> rebuild it precisely.
  • The as F cast on the returned closure is the idiomatic escape when the body can’t prove the shape.
  • Diverge from F only where the wrapper genuinely changes behavior (debounce → void).
Practice Exercises:
1Write throttle<F>(fn, ms) preserving Parameters<F> and returning void; add a typed cancel() method on the returned function.
2Write retry<F extends (...a: any[]) => Promise<any>>(fn, n) that preserves the async signature and resolves to Awaited<ReturnType<F>>.
3Write withCache that adds a .clear() method to a memoized function, typed as an intersection of the call signature and { clear(): void }.
Connections: This is the runtime-facing side of Utility Types (Parameters/ReturnType) and this Types; the async variants pull in Recursive Awaited<T> (Day 23). The memoize key constraint connects to Type-Safe Configuration and the method extraction of Concept 2.
5. Labeled Tuple Elements & Variadic Parameter Lists Functions
Function parameter lists are tuples to TypeScript, and since 4.0 those tuples can have labels and a rest element anywhere. That unlocks readable tooltips ((start: number, ...rest: number[])), spreading typed argument tuples into calls, and helpers like partial / bind that slice off leading arguments while keeping the rest exactly typed.

A labeled tuple ([name: string, age: number]) carries names purely for display — they show up in Parameters<F> tooltips and error messages, making generic plumbing far more legible. Functionally, the label is invisible; structurally the tuple is identical to its unlabeled form.

The powerful part is the variadic tuple: a rest element ([head: A, ...tail: B[]]) lets you model “a fixed prefix then any number of the rest,” and you can spread one tuple type into another. Combined with Parameters<F>, this is how you type partial application: capture some leading args as a tuple, infer the remaining args as ...rest, and produce a new function whose parameters are exactly that remainder.

1
Parameter lists are tuples; add labels ([x: number]) for readable signatures.
2
A rest element ([a: A, ...b: B[]]) models a fixed prefix plus variadic tail.
3
Spread/infer tuples to slice leading args — the basis of typed partial/bind.
TypeScript · example 1 of 4 — labels are for humans, structure is the tuple
type Args = [first: string, second: number, ...rest: boolean[]];

function f(...args: Args): void { /* ... */ }
f("a", 1);            // ok
f("a", 1, true, false);   // ok — rest soaks up the booleans
// f(1, "a");          // Error: types in the wrong positions

type Names = Parameters<typeof f>;
// [first: string, second: number, ...rest: boolean[]]  — labels shown in tooltips
labels surface in Parameters tooltips; checking is positional
TypeScript · example 2 of 4 — spreading a typed arg tuple into a call
function makeUser(name: string, age: number, admin: boolean) {
  return { name, age, admin };
}

type MakeUserArgs = Parameters<typeof makeUser>;   // [string, number, boolean]
const args: MakeUserArgs = ["Ada", 36, true];

const u = makeUser(...args);   // spread is type-checked against the tuple
// const bad: MakeUserArgs = ["Ada", 36];   // Error: missing boolean element
a tuple typed as Parameters spreads safely into the call
TypeScript · example 3 of 4 — typed partial application
function partial<Bound extends unknown[], Rest extends unknown[], R>(
  fn: (...args: [...Bound, ...Rest]) => R,
  ...bound: Bound
): (...rest: Rest) => R {
  return (...rest: Rest) => fn(...bound, ...rest);
}

const greet = (greeting: string, name: string) => `${greeting}, ${name}!`;
const hi = partial(greet, "Hi");   // (name: string) => string
hi("Ada");                         // "Hi, Ada!"  — remaining arg fully typed
Hi, Ada!
TypeScript · example 4 of 4 — head/tail and prepending an argument
type Head<T extends unknown[]> = T extends [infer H, ...unknown[]] ? H : never;
type Tail<T extends unknown[]> = T extends [unknown, ...infer R] ? R : [];

type A = Head<[string, number, boolean]>;   // string
type B = Tail<[string, number, boolean]>;   // [number, boolean]

// inject a context argument as the new first parameter of any handler
type WithCtx<F extends (...a: any[]) => any, C> =
  (ctx: C, ...args: Parameters<F>) => ReturnType<F>;

type Handler = (id: string) => void;
type CtxHandler = WithCtx<Handler, { db: unknown }>;
// (ctx: { db: unknown }, id: string) => void
A = string; B = [number, boolean]; CtxHandler prepends ctx
Common Mistakes:
❌ Expecting tuple labels to be enforced by callers — f("a", 1) never names arguments; labels only improve tooltips and messages.
✅ Treat labels as documentation; rely on element positions and types for actual checking.
❌ Hand-spelling bind/partial with two unrelated parameters, losing the link between bound args and the remaining ones.
✅ Model the original parameters as [...Bound, ...Rest] so inferring Bound leaves Rest as the exact remaining tuple.
Key Rules:
  • Parameter lists are tuples; labels ([x: T]) are display-only, positions do the checking.
  • A rest element can sit anywhere: [a: A, ...b: B[]] models prefix + variadic tail.
  • Spreading a Parameters<F>-typed tuple into a call is type-checked element by element.
  • [...Bound, ...Rest] inference is the backbone of typed partial/bind/context-injection.
Practice Exercises:
1Write curry2 for two-arg functions using labeled tuples, and a generic flip<F> that swaps the first two parameters’ types.
2Type a pipe(...fns) where each function’s return tuple feeds the next’s single parameter; start with a fixed arity of three, then generalize.
3Write Last<T extends unknown[]> and Init<T> (everything but the last) using a trailing rest infer, and verify on [1, 2, 3].
