R1. Node.js + TypeScript Setup Node.js & APIs 19 days since review · rep 1 → 2
A modern Node + TS project runs source directly with tsx/ts-node in dev and compiles with tsc for prod. Under ESM ("type": "module" + module: "nodenext") there’s no __dirname — you reconstruct it from import.meta.url — and process.env values are all string | undefined until you parse them.
Key Pattern 
import { fileURLToPath } from "node:url";
import { dirname } from "node:path";

const __dirname = dirname(fileURLToPath(import.meta.url));  // ESM shim

// process.env is Record<string, string | undefined> — parse at the edge
const port = Number(process.env.PORT ?? 3000);

process.on("SIGTERM", () => server.close());
env values are string | undefined; import.meta.url rebuilds __dirname under ESM
Challenge Exercise

Set up a project whose tsconfig.json uses "module": "nodenext" and write a Zod env.ts that parses process.env into a typed, frozen Env object at startup (throwing on a missing var). Explain why module: "nodenext" forces explicit .js extensions on relative imports even in .ts files.

Pitfall: Trusting process.env.FOO as a string — it’s string | undefined, so a typo silently yields undefined. Parse env once at boot. And __dirname/require simply don’t exist in ESM output — reach for import.meta.url and createRequire instead.
R2. API Typing: tRPC & OpenAPI Node.js & APIs 19 days since review · rep 1 → 2
Two routes to a typed API boundary. tRPC infers the client’s types straight from the server router type — no codegen, but it couples both ends to one TS monorepo. OpenAPI is schema-first: the spec is the contract and you generate types for any language, at the cost of a build step.
Key Pattern 
const appRouter = t.router({
  getUser: t.procedure.input(z.object({ id: z.string() }))
    .query(({ input }) => db.user(input.id)),   // returns User
});
export type AppRouter = typeof appRouter;        // the ONLY export the client needs

// client — fully typed from AppRouter, zero codegen
const user = await trpc.getUser.query({ id: "1" });  // user: User
type Inputs = inferRouterInputs<AppRouter>;
the client’s call and result types flow from `typeof appRouter`
Challenge Exercise

Add a createPost mutation with a Zod input, then derive inferRouterInputs/inferRouterOutputs and assert the mutation’s output type with expectTypeOf. Then contrast: when would you choose OpenAPI + generated types over tRPC (hint: non-TS or third-party consumers)?

Pitfall: Exporting values from the server into the client bundle instead of only export type AppRouter — that drags server code (and secrets) into the client. tRPC needs only the type. And tRPC’s inference stops at the process boundary: a separate non-TS consumer still needs OpenAPI.
R3. State Management Typing: Redux Toolkit & Zustand Ecosystem Deep Dives 17 days since review · rep 1 → 2
Both stores derive their types from a single source of truth. RTK infers RootState/AppDispatch from the configured store and pre-types the hooks. Zustand uses a curried create<T>()(...) call so the store type is fixed once and middleware inference still works.
Key Pattern 
// Zustand — note the curried create<T>()(...) form
interface BearState { bears: number; add: () => void }
const useBears = create<BearState>()((set) => ({
  bears: 0,
  add: () => set((s) => ({ bears: s.bears + 1 })),
}));

// RTK — types flow from the store
export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;
store state is inferred once; selectors/actions read from it
Challenge Exercise

Build a Zustand store with the persist middleware and a typed selector useBears(s => s.bears); confirm the curried create<T>() keeps set’s partial-update type intact. Then explain why create<T>(...) (single call) breaks middleware inference where create<T>()(...) does not.

Pitfall: Writing create<BearState>((set) => ...) in one call — TypeScript can’t infer the middleware mutators, so wrapped stores mistype. Use the two-step curried form create<T>()(...). On the RTK side, hand-writing RootState instead of ReturnType<typeof store.getState> drifts from the real slice shape.
R4. Database Typing: Prisma & Drizzle Ecosystem Deep Dives 17 days since review · rep 1 → 2
Prisma generates a client + types from a schema.prisma file; a select/include narrows the returned row type to exactly the requested fields. Drizzle is SQL-first — you define tables in TS and read table.$inferSelect/$inferInsert, so the schema is the type with no generation step.
Key Pattern 
// Prisma — select narrows the result type
const u = await prisma.user.findFirst({ select: { id: true, email: true } });
//    u: { id: string; email: string } | null   — no `name`, no `password`

// Drizzle — the table definition is the type
const users = pgTable("users", { id: uuid().primaryKey(), email: text() });
type User = typeof users.$inferSelect;   // { id: string; email: string }
Prisma’s select narrows to requested fields; Drizzle infers from the table
Challenge Exercise

In Prisma, write a reusable query with Prisma.validator + Prisma.UserGetPayload<typeof args> so a selected shape has a named type. In Drizzle, define a relation and infer the joined row type. Explain how each tool guarantees a selected password-less shape never leaks the column at the type level.

Pitfall: Assuming findFirst() returns a non-null full row — it’s T | null, and a select removes fields at the type level, so reading an unselected field is a compile error (good!). Forgetting to regenerate the Prisma client after a schema edit leaves stale types.
R5. GraphQL Codegen Ecosystem Deep Dives 15 days since review · rep 1 → 2
GraphQL Code Generator turns your schema + operation documents into TS types, so each query has a precise result and variables type. The client-preset / TypedDocumentNode approach ties a query string to its types — passing it to useQuery infers data and variables with no manual generics.
Key Pattern 
// generated: graphql() returns a TypedDocumentNode<Result, Variables>
const UserDoc = graphql(`
  query User($id: ID!) { user(id: $id) { id name } }
`);

const { data } = useQuery(UserDoc, { variables: { id: "1" } });
//    data?.user: { id: string; name: string } | undefined
//    variables is checked: { id: string } required
a TypedDocumentNode carries both result and variables types to the caller
Challenge Exercise

Configure codegen.ts with the client-preset, write a query using a fragment, and confirm the generated FragmentType forces you through useFragment to read the data (masking). Explain why fragment masking keeps a component from depending on fields it didn’t itself request.

Pitfall: Letting generated types drift — codegen must re-run whenever the schema or an operation changes (wire it into CI / a watch script), or your types silently lie. And hand-writing result interfaces defeats the point: the generated TypedDocumentNode already knows the exact shape.
R6. Testing React Components Ecosystem Deep Dives 15 days since review · rep 1 → 2
React Testing Library tests behaviour through the DOM, not implementation. render returns typed queries, screen.getByRole narrows to specific HTMLElement subtypes, and userEvent methods return promises you must await. A typed custom render wrapper injects providers once for every test.
Key Pattern 
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

test("increments", async () => {
  render(<Counter />);
  const btn = screen.getByRole("button");     // HTMLElement
  await userEvent.click(btn);                 // await — it’s async
  expect(screen.getByText("1")).toBeInTheDocument();
});
queries return typed elements; userEvent actions are awaited promises
Challenge Exercise

Write a typed renderWithProviders(ui, { store?, route? }) wrapper (React Query + Router) whose options are a typed object, and use getByRole<HTMLInputElement> to read .value without a cast. Explain why preferring *ByRole over *ByTestId makes the test assert accessibility as a side effect.

Pitfall: Forgetting to await a userEvent call (or a findBy* query) — the assertion runs before the update and flakes. And getByRole returns a generic HTMLElement; when you need element-specific fields, pass the subtype (getByRole<HTMLInputElement>("textbox")) rather than casting.
1Node.js + TS: Resilience & Lifecycle
1. Graceful Shutdown & Lifecycle Signals Node.js + TypeScript
When an orchestrator (Kubernetes, a process manager) stops your service it sends SIGTERM and expects you to finish in-flight requests, close DB pools, and exit — or it kills you with SIGKILL and drops connections. A typed shutdown routine turns “hope it exits cleanly” into an ordered, testable sequence of async cleanup hooks.

The lifecycle: catch the signal → stop accepting new work (server.close()) → run cleanup hooks (drain queues, close DB/Redis) → exit. Model each hook as a typed ShutdownHook and run them in reverse registration order (last-opened, first-closed), guarding against a hook that hangs with a hard timeout.

Two rules keep this correct: make shutdown idempotent (both SIGTERM and SIGINT, plus a possible error path, can trigger it — run once) and always arm a force-exit timer so a stuck hook can’t wedge the process forever.

1
Define a typed ShutdownHook = () => Promise<void> and collect them.
2
On SIGTERM/SIGINT, stop the server, then run hooks in reverse order.
3
Guard with an idempotent flag and a force-exit timeout.
TypeScript · example 1 of 4 — a typed registry of cleanup hooks
type ShutdownHook = () => Promise<void>;

const hooks: { name: string; run: ShutdownHook }[] = [];

function onShutdown(name: string, run: ShutdownHook): void {
  hooks.push({ name, run });
}

onShutdown("db", () => prisma.$disconnect());
onShutdown("redis", async () => { await redis.quit(); });
every cleanup step is a named, awaitable ShutdownHook
TypeScript · example 2 of 4 — run hooks in reverse, idempotently
let shuttingDown = false;

async function shutdown(signal: NodeJS.Signals): Promise<void> {
  if (shuttingDown) return;      // idempotent — signals can arrive twice
  shuttingDown = true;
  console.log(`${signal} received, draining…`);

  for (const h of [...hooks].reverse()) {   // last opened, first closed
    try { await h.run(); }
    catch (err) { console.error(`hook ${h.name} failed`, err); }
  }
}
signal is typed NodeJS.Signals; the guard makes shutdown run exactly once
TypeScript · example 3 of 4 — stop the server + force-exit timer
const server = app.listen(port);

async function handle(signal: NodeJS.Signals) {
  const force = setTimeout(() => process.exit(1), 10_000).unref();  // hard cap
  await new Promise<void>((r) => server.close(() => r()));  // no new conns
  await shutdown(signal);
  clearTimeout(force);
  process.exit(0);
}
server.close stops new work; the unref’d timer guarantees an eventual exit
TypeScript · example 4 of 4 — wire the signals (and last-ditch errors)
const SIGNALS: NodeJS.Signals[] = ["SIGTERM", "SIGINT"];
for (const sig of SIGNALS) process.on(sig, () => void handle(sig));

process.on("unhandledRejection", (reason) => {
  console.error("unhandledRejection", reason);
  void handle("SIGTERM");
});
both operator signals and a fatal rejection funnel through one drain path
Common Mistakes:
❌ Calling process.exit(0) the instant SIGTERM arrives — in-flight requests are cut and DB writes may be lost mid-transaction.
✅ server.close() first to drain, run cleanup hooks, then exit.
❌ No timeout — a hook that never resolves (a hung socket) wedges the pod until SIGKILL.
✅ Arm an unref()’d force-exit timer so the process always terminates.
Key Rules:
  • 2026-07-04 (current — Day 31)
  • Handle SIGTERM and SIGINT; make the drain idempotent so a repeated signal is a no-op.
  • Stop accepting new work (server.close()) before running cleanup hooks.
  • Close resources in reverse of open order and swallow-log per-hook errors so one failure doesn’t skip the rest.
  • Always arm a hard force-exit timeout — never trust every hook to resolve.
Practice Exercises:
1Add a Promise.race per hook so a single slow hook times out at 3s instead of eating the whole budget.
2Make onShutdown return an unregister function (typed () => void) so a module can remove its hook.
3Add a readiness flag that flips to false at the start of handle so the health endpoint (Concept 2) reports “draining” before the server closes.
Connections: The NodeJS.Signals handlers extend the process-typing from Node.js + TypeScript (R1), and closing the DB/Redis pools ties back to repository pattern and Redis caching (07-01). The readiness flag feeds directly into Concept 2’s probe.
2. Health Checks & Readiness Probes Node.js + TypeScript
Orchestrators poll two distinct endpoints: liveness (“is the process alive? restart if not”) and readiness (“can it serve traffic right now?”). Conflating them causes restart loops or routing to a pod that can’t reach its database. Typing each dependency check and the aggregated report keeps the two separate and machine-readable.

Model a single dependency check as a function returning a discriminated CheckResult ("up" vs "down" with an error). Run all checks in parallel, aggregate to an overall status, and map it to HTTP: 200 when ready, 503 when not. Liveness stays trivial (return 200 if the event loop is running) so a failing dependency never triggers a pointless restart.

Readiness should also reflect the lifecycle: once the shutdown drain from Concept 1 begins, flip readiness to false so the load balancer stops sending new requests before the server closes.

1
Type a Check = () => Promise<CheckResult> with a discriminated result.
2
Run checks in parallel and aggregate to one HealthReport.
3
Map ready/not-ready to 200/503; keep liveness separate and trivial.
TypeScript · example 1 of 4 — a discriminated check result
type CheckResult =
  | { name: string; status: "up" }
  | { name: string; status: "down"; error: string };

type Check = () => Promise<CheckResult>;

const dbCheck: Check = async () => {
  try { await prisma.$queryRaw`SELECT 1`; return { name: "db", status: "up" }; }
  catch (e) { return { name: "db", status: "down", error: String(e) }; }
};
a check is up, or down with an error — never an ambiguous boolean
TypeScript · example 2 of 4 — aggregate checks in parallel
interface HealthReport {
  status: "healthy" | "unhealthy";
  checks: CheckResult[];
}

async function runChecks(checks: Check[]): Promise<HealthReport> {
  const results = await Promise.all(checks.map((c) => c()));
  const status = results.every((r) => r.status === "up") ? "healthy" : "unhealthy";
  return { status, checks: results };
}
one typed report; overall status derived from every dependency
TypeScript · example 3 of 4 — readiness maps status to 200/503
let isReady = true;   // flipped false when shutdown (Concept 1) begins

app.get("/readyz", async (_req, res) => {
  if (!isReady) return res.status(503).json({ status: "draining" });
  const report = await runChecks([dbCheck, redisCheck]);
  res.status(report.status === "healthy" ? 200 : 503).json(report);
});
503 while draining or when a dependency is down; 200 only when serveable
TypeScript · example 4 of 4 — liveness stays trivial & dependency-free
// Liveness: "is the process wedged?" — must NOT touch the DB,
// or a slow dependency triggers a needless restart loop.
app.get("/livez", (_req, res) => res.status(200).send("ok"));
liveness answers only “is the event loop alive?” — never checks dependencies
Common Mistakes:
❌ A single /health that checks the DB and is wired to the liveness probe — a brief DB blip restarts every pod, cascading the outage.
✅ Split liveness (trivial) from readiness (checks dependencies); only readiness gates traffic.
❌ Returning 200 with a { ok: false } body — the orchestrator only reads the status code, so it routes traffic anyway.
✅ Encode health in the HTTP status (503 for not-ready), not just the body.
Key Rules:
  • Keep liveness dependency-free; put dependency checks only in readiness.
  • Model each check as a discriminated CheckResult and run them with Promise.all.
  • Signal not-ready with a 503 status code — orchestrators read the code, not the body.
  • Flip readiness to not-ready as soon as graceful shutdown starts.
Practice Exercises:
1Add a per-check timeout so a hung dependency reports down after 2s instead of stalling /readyz.
2Extend CheckResult with a latencyMs: number and include it in the report for each check.
3Add a "degraded" status when non-critical checks fail but critical ones pass, and map it to 200.
Connections: The discriminated CheckResult and every-narrowing reuse discriminated unions in practice and never/exhaustiveness. The isReady flag is the other half of Concept 1’s drain, and mapping status to 503 mirrors the HTTP-boundary discipline of problem+json (07-03).
3. Typed Retries, Timeouts & Circuit Breakers Node.js + TypeScript
Every network call can be slow, flaky, or dead. Three composable primitives tame that: a timeout caps latency, a retry with backoff rides out transient blips, and a circuit breaker stops hammering a downstream that’s clearly down. Writing them as generic <T> wrappers keeps the operation’s return type intact while adding resilience.

Each wrapper takes a () => Promise<T> and returns a Promise<T>, so it’s transparent to callers. Timeout races the work against an AbortSignal.timeout; retry loops with exponential backoff and a typed should-retry predicate; the breaker is a small state machine — a discriminated union of closed / open / half-open — that short-circuits calls while a dependency is failing.

Compose them outside-in: breaker wraps retry wraps timeout. The breaker sees the final outcome of the retried, time-bounded call, so a truly-down dependency trips it instead of every retry counting separately.

1
Write a generic withTimeout<T> using AbortSignal.timeout.
2
Write a generic withRetry<T> with backoff + a typed retry predicate.
3
Model the breaker as a discriminated state machine and compose the three.
TypeScript · example 1 of 4 — generic withTimeout via AbortSignal
async function withTimeout<T>(
  fn: (signal: AbortSignal) => Promise<T>,
  ms: number,
): Promise<T> {
  const signal = AbortSignal.timeout(ms);   // aborts after ms
  return fn(signal);                        // pass to fetch/db so it truly cancels
}

const data = await withTimeout((s) => fetch(url, { signal: s }), 2000);
withTimeout preserves T; the signal lets the underlying call actually cancel
TypeScript · example 2 of 4 — generic withRetry + exponential backoff
interface RetryOpts { retries: number; baseMs: number; retryable?: (e: unknown) => boolean }

async function withRetry<T>(fn: () => Promise<T>, o: RetryOpts): Promise<T> {
  let lastErr: unknown;
  for (let attempt = 0; attempt <= o.retries; attempt++) {
    try { return await fn(); }
    catch (e) {
      lastErr = e;
      if (o.retryable && !o.retryable(e)) throw e;      // don’t retry a 400
      const delay = o.baseMs * 2 ** attempt + Math.random() * 100;  // + jitter
      await new Promise((r) => setTimeout(r, delay));
    }
  }
  throw lastErr;
}
returns T on success; backoff has jitter; a non-retryable error exits early
TypeScript · example 3 of 4 — the breaker as a discriminated state machine
type BreakerState =
  | { kind: "closed"; failures: number }
  | { kind: "open"; openedAt: number }
  | { kind: "half-open" };

class CircuitBreaker {
  private state: BreakerState = { kind: "closed", failures: 0 };
  constructor(private threshold = 5, private cooldownMs = 30_000) {}

  async call<T>(fn: () => Promise<T>): Promise<T> {
    if (this.state.kind === "open") {
      if (Date.now() - this.state.openedAt < this.cooldownMs)
        throw new Error("circuit open");
      this.state = { kind: "half-open" };   // probe once
    }
    try { const r = await fn(); this.state = { kind: "closed", failures: 0 }; return r; }
    catch (e) { this.trip(); throw e; }
  }

  private trip() {
    const failures = this.state.kind === "closed" ? this.state.failures + 1 : this.threshold;
    this.state = failures >= this.threshold
      ? { kind: "open", openedAt: Date.now() }
      : { kind: "closed", failures };
  }
}
each state carries only its own data; open short-circuits until cooldown elapses
TypeScript · example 4 of 4 — compose breaker → retry → timeout
const breaker = new CircuitBreaker();

const user = await breaker.call(() =>
  withRetry(
    () => withTimeout((s) => fetchUser(id, s), 2000),
    { retries: 3, baseMs: 100, retryable: (e) => e instanceof NetworkError },
  ),
);   // user: User — the resilient stack is transparent to the caller
the composed stack still returns User; the breaker sees the final retried outcome
Common Mistakes:
❌ Retrying everything — a 400/validation error will never succeed, so retries just amplify load and delay the failure.
✅ Gate retries with a typed retryable predicate (retry timeouts/5xx, fail fast on 4xx).
❌ A Promise.race “timeout” that leaves the real request running — you stop waiting but the socket/query keeps consuming resources.
✅ Thread an AbortSignal into the call so a timeout genuinely cancels the work.
Key Rules:
  • Write each primitive as a generic <T> wrapper so it’s transparent to the caller’s types.
  • Retry only transient failures (use a predicate); always add jitter to backoff to avoid thundering herds.
  • Model the breaker as a discriminated union so each state carries exactly its own data.
  • Compose breaker → retry → timeout, and thread an AbortSignal so timeouts truly cancel.
Practice Exercises:
1Add a maxDelayMs cap to withRetry so exponential backoff plateaus instead of growing unbounded.
2Give the breaker an onStateChange(prev, next) callback typed over BreakerState and log every transition.
3Write a withTimeout variant that rejects with a distinct TimeoutError subclass so retryable can treat it specially.
Connections: The generic <T> wrappers are generics + async/Promise typing, and the breaker’s state union is a type-safe state machine (discriminated unions). This is the outbound complement to rate limiting (07-02) and pairs with the readiness probe — an open breaker can mark a dependency down.

4. Request Context with AsyncLocalStorage Node.js + TypeScript
You often need per-request data (a request id, the authenticated user, a trace id) deep inside code that shouldn’t take it as a parameter — a logger, a repository. Node’s AsyncLocalStorage is a typed, request-scoped store that survives await boundaries, so any code in the async call tree can read the current context without threading it through every signature.

Create one AsyncLocalStorage<RequestContext> for the process. A middleware calls als.run(ctx, next) once per request; everything next triggers — across every await — can call als.getStore() to read that same ctx. Because getStore() returns T | undefined (you might be outside any request), wrap it in an accessor that throws, giving callers a non-optional type.

This is the backbone of contextual logging and trace propagation: the logger reads the current requestId itself, so you never pass it around. Keep the stored object small and treat it as read-mostly.

1
Define a typed RequestContext and one module-level AsyncLocalStorage.
2
In middleware, als.run(ctx, next) to bind context for the request’s async tree.
3
Expose a getContext() accessor that narrows away undefined.
TypeScript · example 1 of 4 — a typed store for the process
import { AsyncLocalStorage } from "node:async_hooks";

interface RequestContext {
  requestId: string;
  userId?: string;
}

const als = new AsyncLocalStorage<RequestContext>();
the store is parameterized by RequestContext — getStore() returns it or undefined
TypeScript · example 2 of 4 — bind context in middleware
app.use((req, res, next) => {
  const ctx: RequestContext = { requestId: crypto.randomUUID() };
  als.run(ctx, () => next());   // everything downstream shares this ctx
});

// later middleware can enrich the SAME object after auth
app.use((req, _res, next) => {
  const ctx = als.getStore();
  if (ctx && req.user) ctx.userId = req.user.id;
  next();
});
als.run scopes ctx to the request’s entire async call tree, across awaits
TypeScript · example 3 of 4 — an accessor that narrows away undefined
function getContext(): RequestContext {
  const ctx = als.getStore();
  if (!ctx) throw new Error("getContext() called outside a request scope");
  return ctx;   // callers get a non-optional RequestContext
}
the guard turns RequestContext | undefined into RequestContext
TypeScript · example 4 of 4 — a context-aware logger, no param threading
const log = {
  info(msg: string, extra: Record<string, unknown> = {}) {
    const { requestId, userId } = als.getStore() ?? {};
    console.log(JSON.stringify({ level: "info", msg, requestId, userId, ...extra }));
  },
};

// deep in a repository — no requestId argument needed:
async function findOrder(id: string) { log.info("loading order", { id }); /* … */ }
every log line carries the current requestId automatically
Common Mistakes:
❌ Storing request context in a module-level let currentCtx — concurrent requests clobber each other; whoever ran last wins.
✅ Use AsyncLocalStorage, which keeps a separate store per async call tree.
❌ Treating getStore() as always-defined — it’s T | undefined, and a background job or startup code runs outside any request.
✅ Wrap it in an accessor that throws (or supplies a default) so the type is honest.
Key Rules:
  • Create one AsyncLocalStorage<T> per concern and bind it once per request with als.run.
  • getStore() is T | undefined — wrap it in a guarded accessor for a non-optional type.
  • Keep the stored context small and read-mostly; mutate it only in early middleware (e.g. after auth).
  • Prefer implicit context for cross-cutting data (ids, user); pass real business args explicitly.
Practice Exercises:
1Add a traceId to RequestContext and propagate an incoming x-request-id header into it (generate one if absent).
2Write a generic runWithContext<T>(ctx, fn: () => Promise<T>): Promise<T> wrapper for use in background jobs outside HTTP.
3Type a getContext() overload that takes a default so callers can opt out of the throw.
Connections: The guarded accessor is the same narrow-away-undefined discipline as env parsing (R1) and getStore pairs with the declaration-merged req.user (07-02). This context store is exactly what OpenTelemetry (Concept 5) uses under the hood to propagate the active span.
5. OpenTelemetry Tracing with TypeScript Node.js + TypeScript
Structured logs tell you what happened; distributed tracing tells you where the time went across services. OpenTelemetry is the vendor-neutral standard: you wrap units of work in spans, attach typed attributes, and record errors — and the SDK propagates the trace across HTTP calls so one request’s spans stitch into a single waterfall.

Get a Tracer and open a span with startActiveSpan, which makes that span the active one (via the same AsyncLocalStorage mechanism from Concept 4) so nested work becomes child spans automatically. On each span set attributes (typed as string | number | boolean | arrays), record exceptions, set a status, and — critically — always span.end() in a finally.

The ergonomic move is a generic withSpan<T>(name, fn) helper that opens a span, runs your Promise<T>, marks error/OK, and ends it — keeping the operation’s return type intact while every call becomes observable. Context propagation across service boundaries is handled by the SDK’s instrumentations once you set up the provider.

1
Acquire a Tracer and open an active span for a unit of work.
2
Set typed attributes; record exceptions and a SpanStatusCode.
3
Wrap it in a generic withSpan<T> that always ends the span.
TypeScript · example 1 of 4 — open an active span
import { trace, SpanStatusCode, type Span } from "@opentelemetry/api";

const tracer = trace.getTracer("orders-service");

function instrument(span: Span, userId: string) {
  span.setAttribute("user.id", userId);        // string | number | boolean | []
  span.setAttribute("items.count", 3);
  try {
    /* … work … */
    span.setStatus({ code: SpanStatusCode.OK });
  } catch (err) {
    span.recordException(err as Error);
    span.setStatus({ code: SpanStatusCode.ERROR, message: String(err) });
    throw err;
  }
}
attribute values are constrained; a failed span is marked ERROR with the exception
TypeScript · example 3 of 4 — a generic withSpan<T> helper
async function withSpan<T>(name: string, fn: (span: Span) => Promise<T>): Promise<T> {
  return tracer.startActiveSpan(name, async (span) => {
    try {
      const result = await fn(span);
      span.setStatus({ code: SpanStatusCode.OK });
      return result;                              // return type stays T
    } catch (err) {
      span.recordException(err as Error);
      span.setStatus({ code: SpanStatusCode.ERROR });
      throw err;
    } finally {
      span.end();                                 // ALWAYS end, even on throw
    }
  });
}
withSpan preserves T and guarantees span.end() runs on every path

Common Mistakes:
❌ Forgetting span.end() (or only calling it on the success path) — unended spans never export, so traces look empty or truncated.
✅ End the span in a finally — a generic withSpan helper makes that automatic.
❌ Using startSpan and never activating it — child work isn’t parented, so you get orphan spans instead of a tree.
✅ Use startActiveSpan so nested operations attach as children via the active context.
Key Rules:
  • Prefer startActiveSpan so nested work auto-parents into a single trace.
  • Always span.end() in a finally; record exceptions and set an explicit SpanStatusCode.
  • Attributes are typed primitives/arrays — keep them low-cardinality; put ids/details there, not in the span name.
  • Wrap tracing in a generic withSpan<T> so instrumentation is transparent to your types.
Practice Exercises:
1Extend withSpan to accept an initial attributes: Record<string, string | number | boolean> and set them before running fn.
2Add the current span’s traceId (from span.spanContext()) into the RequestContext logger from Concept 4 so logs and traces correlate.
3Write a withSpanSync<T>(name, fn: (s: Span) => T): T for synchronous work and confirm the return type is preserved.
