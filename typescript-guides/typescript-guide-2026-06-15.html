Review — 6 Concepts
R1. satisfies Operator Advanced Type System rep 2 · 12 days ago
satisfies checks a value matches a type without widening it — you keep the narrower literal type while validating the shape.
Key Pattern 
type Colors = Record<string, [number, number, number] | string>;
const palette = {
  red: [255, 0, 0],
  green: "#00ff00",  // string, not tuple
} satisfies Colors;

// palette.red is still [number, number, number], not string | [number, number, number]
const r = palette.red[0]; // number (not possible with `as Colors`)
palette.red: [number, number, number] r: number palette.green: string
Challenge Exercise

#1 Pitfall: Confusing satisfies with as — as widens/overrides the type, satisfies validates without widening.
R2. React + TypeScript: Generic Components React + TypeScript rep 0 · 7 days ago
Generic components let you create reusable UI components that preserve the type of data they receive — e.g., a <List<T>> that knows the type of each item.
Key Pattern 
interface ListProps<T> {
  items: T[];
  renderItem: (item: T, index: number) => React.ReactNode;
  keyExtractor: (item: T) => string;
}

function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return <ul>{items.map((item, i) => <li key={keyExtractor(item)}>{renderItem(item, i)}</li>)}</ul>;
}

// Usage — T is inferred as { id: number; name: string }
<List items={users} renderItem={u => <span>{u.name}</span>} keyExtractor={u => String(u.id)} />
T inferred as { id: number; name: string } u: { id: number; name: string }

R5. never & Exhaustiveness Advanced Type System rep 2 · 11 days ago
never is the bottom type — no value can be assigned to it. Use it for exhaustive switch/if-else checking: if a variable narrows to never, all cases are handled.
Key Pattern 
type Shape = { kind: "circle"; r: number } | { kind: "square"; s: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle": return Math.PI * shape.r ** 2;
    case "square": return shape.s ** 2;
    default: {
      const _exhaustive: never = shape; // compile error if a case is missing
      return _exhaustive;
    }
  }
}
area({ kind: "circle", r: 5 }): number Adding a new variant to Shape without handling it causes: Type '{ kind: "triangle"; ... }' is not assignable to type 'never'

1 TypeScript Compiler & Tooling
TypeScript Compiler API Tooling & Compiler
The TypeScript Compiler API lets you programmatically analyze, type-check, and transform TypeScript code — essential for building custom linters, codemods, documentation generators, and IDE plugins.

Example 1 — Creating a program and listing source files:

TypeScript
import * as ts from "typescript";

// Create a program from tsconfig
const configPath = ts.findConfigFile("./", ts.sys.fileExists, "tsconfig.json")!;
const configFile = ts.readConfigFile(configPath, ts.sys.readFile);
const parsedConfig = ts.parseJsonConfigFileContent(configFile.config, ts.sys, "./");

const program = ts.createProgram(parsedConfig.fileNames, parsedConfig.options);
const sourceFiles = program.getSourceFiles().filter(sf => !sf.isDeclarationFile);

console.log("Source files:");
sourceFiles.forEach(sf => console.log(`  ${sf.fileName}`));
Source files: /project/src/index.ts /project/src/utils.ts

Example 2 — Walking the AST:

TypeScript
import * as ts from "typescript";

const source = `
  const greeting: string = "hello";

Example 3 — Extracting type information:

TypeScript
import * as ts from "typescript";

const source = `
  interface User { name: string; age: number; }
  const users: User[] = [];
  const firstUser = users[0];
`;

const sourceFile = ts.createSourceFile("demo.ts", source, ts.ScriptTarget.Latest, true);
const host = ts.createCompilerHost({ strict: true });
const originalGetSourceFile = host.getSourceFile;
host.getSourceFile = (fileName, target) =>
  fileName === "demo.ts" ? sourceFile : originalGetSourceFile(fileName, target);

const program = ts.createProgram(["demo.ts"], { strict: true }, host);
const checker = program.getTypeChecker();

// Find the 'firstUser' variable and get its type
ts.forEachChild(sourceFile, node => {
  if (ts.isVariableStatement(node)) {
    for (const decl of node.declarationList.declarations) {
      if (ts.isIdentifier(decl.name) && decl.name.text === "firstUser") {
        const type = checker.getTypeAtLocation(decl);
        console.log(`firstUser type: ${checker.typeToString(type)}`);
        // firstUser type: User
      }
    }
  }
});
firstUser type: User

Example 4 — Simple code transformer:

TypeScript
import * as ts from "typescript";

// Transformer: replace all string literals with uppercase versions
const uppercaseStrings: ts.TransformerFactory<ts.SourceFile> = context => {
  return sourceFile => {
    function visit(node: ts.Node): ts.Node {
      if (ts.isStringLiteral(node)) {
        return ts.factory.createStringLiteral(node.text.toUpperCase());
      }
      return ts.visitEachChild(node, visit, context);
    }
    return ts.visitNode(sourceFile, visit) as ts.SourceFile;
  };
};

const source = `const msg = "hello world";`;
const sourceFile = ts.createSourceFile("test.ts", source, ts.ScriptTarget.Latest);
const result = ts.transform(sourceFile, [uppercaseStrings]);
const printer = ts.createPrinter();
console.log(printer.printFile(result.transformed[0]));
const msg = "HELLO WORLD";
Common Mistakes
Creating a new ts.Program for every file — expensive!
Create one program and reuse it: program.getSourceFile(fileName)
Using string comparison for node types: node.kind === "FunctionDeclaration"
Use the type guards: ts.isFunctionDeclaration(node)
Mutating AST nodes directly (they're immutable)
Use ts.factory.update* or ts.factory.create* to build new nodes
Key Rules
  • 2026-07-04 (Day 31)
  • Use ts.createProgram for type-checking, ts.createSourceFile for syntax-only analysis.
  • AST nodes are immutable — use the factory API and transformers to modify code.
  • The type checker (program.getTypeChecker()) is the bridge between syntax and semantics.

Custom ESLint Rules with TypeScript Tooling & Quality
Custom ESLint rules enforce project-specific conventions at scale — no-banned-imports, naming patterns, architecture boundaries. With @typescript-eslint/utils, rules can access TypeScript type information.

Example 1 — Basic rule structure:

TypeScript
import { ESLintUtils } from "@typescript-eslint/utils";

const createRule = ESLintUtils.RuleCreator(
  name => `https://myorg.dev/rules/${name}`
);

// Rule: ban console.log (but allow console.warn/error)
export const noBannedConsole = createRule({
  name: "no-console-log",
  meta: {
    type: "suggestion",
    docs: { description: "Disallow console.log" },
    messages: {
      noConsoleLog: "Use a logger instead of console.log",
    },
    schema: [],
  },
  defaultOptions: [],
  create(context) {
    return {
      MemberExpression(node) {
        if (
          node.object.type === "Identifier" &&
          node.object.name === "console" &&
          node.property.type === "Identifier" &&
          node.property.name === "log"
        ) {
          context.report({ node, messageId: "noConsoleLog" });
        }
      },
    };
  },
});

Example 2 — Type-aware rule:

TypeScript
import { ESLintUtils } from "@typescript-eslint/utils";
import * as tsutils from "ts-api-utils";

const createRule = ESLintUtils.RuleCreator(name => `https://myorg.dev/rules/${name}`);

// Rule: warn when a Promise is not awaited or returned
export const noFloatingPromises = createRule({
  name: "no-floating-promises",
  meta: {
    type: "problem",
    docs: { description: "Require Promises to be awaited or returned" },
    messages: { floating: "This Promise must be awaited, returned, or voided." },
    schema: [],
  },
  defaultOptions: [],
  create(context) {
    const services = ESLintUtils.getParserServices(context);
    const checker = services.program.getTypeChecker();

    return {
      ExpressionStatement(node) {
        const tsNode = services.esTreeNodeToTSNodeMap.get(node.expression);
        const type = checker.getTypeAtLocation(tsNode);

        if (tsutils.isThenableType(checker, tsNode, type)) {
          context.report({ node, messageId: "floating" });
        }
      },
    };
  },
});

Example 3 — Rule with auto-fix:

TypeScript
import { ESLintUtils, AST_NODE_TYPES } from "@typescript-eslint/utils";

const createRule = ESLintUtils.RuleCreator(name => `https://myorg.dev/rules/${name}`);

// Rule: prefer 'interface' over 'type' for object types (with autofix)
export const preferInterface = createRule({
  name: "prefer-interface",
  meta: {
    type: "suggestion",
    docs: { description: "Prefer interface over type alias for object shapes" },
    fixable: "code",
    messages: { useInterface: "Use an interface instead of a type alias for object types." },
    schema: [],
  },
  defaultOptions: [],
  create(context) {
    return {
      TSTypeAliasDeclaration(node) {
        if (node.typeAnnotation.type === AST_NODE_TYPES.TSTypeLiteral) {
          context.report({
            node,
            messageId: "useInterface",
            fix(fixer) {
              const sourceCode = context.getSourceCode();
              const text = sourceCode.getText(node);
              const name = node.id.name;
              const body = sourceCode.getText(node.typeAnnotation);
              return fixer.replaceText(node, `interface ${name} ${body}`);
            },
          });
        }
      },
    };
  },
});

Example 4 — Testing rules:

TypeScript
import { RuleTester } from "@typescript-eslint/rule-tester";
import { noBannedConsole } from "./no-console-log";

const ruleTester = new RuleTester({
  parser: "@typescript-eslint/parser",
});

ruleTester.run("no-console-log", noBannedConsole, {
  valid: [
    "console.warn('warning');",
    "console.error('error');",
    "logger.log('ok');",
  ],
  invalid: [
    {
      code: "console.log('debug');",
      errors: [{ messageId: "noConsoleLog" }],
    },
  ],
});
Common Mistakes
Accessing TypeScript types without parserOptions.project in ESLint config
Set parserOptions: { project: "./tsconfig.json" } for type-aware rules
Testing rules with plain strings when they need type information
Configure the rule tester with parserOptions.project for type-aware tests
Forgetting meta.fixable: "code" when providing a fix function
Always declare fixability in meta — ESLint silently ignores fixes otherwise
Key Rules
  • Use ESLintUtils.RuleCreator from @typescript-eslint/utils for type-safe rule creation.
  • Type-aware rules (getParserServices) are powerful but slower — use them only when type info is needed.
  • Always test with RuleTester — cover both valid and invalid cases, verify fix output.

2 Advanced Application Patterns
Type-Safe Dependency Injection Architecture
DI decouples components from their dependencies, making code testable and configurable. TypeScript's type system can guarantee all dependencies are provided at compile time — no runtime "service not found" errors.

Example 1 — Manual DI with interfaces:

TypeScript
interface Logger {
  log(msg: string): void;
}
interface Database {
  query(sql: string): Promise<unknown[]>;
}

class UserService {
  constructor(
    private logger: Logger,
    private db: Database
  ) {}

  async getUser(id: string) {
    this.logger.log(`Fetching user ${id}`);
    const rows = await this.db.query(`SELECT * FROM users WHERE id = '${id}'`);
    return rows[0];
  }
}

// Production
const service = new UserService(console, realDb);
// Testing
const service2 = new UserService(mockLogger, mockDb);

Example 2 — Token-based container:

TypeScript
// Type-safe DI container using symbols as tokens
const tokens = {
  Logger: Symbol("Logger") as symbol & { __type: Logger },
  Database: Symbol("Database") as symbol & { __type: Database },
  UserService: Symbol("UserService") as symbol & { __type: UserService },
};

type TokenType<T> = T extends { __type: infer U } ? U : never;

class Container {
  private services = new Map<symbol, unknown>();

  register<T extends keyof typeof tokens>(
    token: (typeof tokens)[T],
    factory: () => TokenType<(typeof tokens)[T]>
  ): void {
    this.services.set(token, factory());
  }

  resolve<T extends symbol & { __type: unknown }>(token: T): TokenType<T> {
    const service = this.services.get(token);
    if (!service) throw new Error(`Service not registered: ${token.toString()}`);
    return service as TokenType<T>;
  }
}

const container = new Container();
container.register("Logger", () => console);
// container.resolve(tokens.Logger) — returns Logger type

Example 3 — Functional DI with a context map:

TypeScript
// Dependencies as a typed record — simple and effective
interface Deps {
  logger: Logger;
  db: Database;
  cache: Cache;
}

function createUserService(deps: Pick<Deps, "logger" | "db">) {
  return {
    async getUser(id: string) {
      deps.logger.log(`Fetching user ${id}`);
      return deps.db.query(`SELECT * FROM users WHERE id = '${id}'`);
    },
  };
}

function createOrderService(deps: Pick<Deps, "logger" | "db" | "cache">) {
  return {
    async getOrder(id: string) {
      const cached = await deps.cache.get(id);
      if (cached) return cached;
      return deps.db.query(`SELECT * FROM orders WHERE id = '${id}'`);
    },
  };
}

// Wire up — TypeScript ensures all deps are provided
const deps: Deps = { logger: console, db: realDb, cache: redisCache };
const userService = createUserService(deps);
const orderService = createOrderService(deps);

Example 4 — Generic context pattern (React-style):

TypeScript
// A typed "context" for dependency injection
type ServiceFactory<TDeps, TService> = (deps: TDeps) => TService;

function createApp<TDeps>(deps: TDeps) {
  return {
    use<TService>(factory: ServiceFactory<TDeps, TService>): TService {
      return factory(deps);
    },
  };
}

const app = createApp({ logger: console, db: realDb });
const users = app.use(({ logger, db }) => ({
  find: (id: string) => { logger.log(id); return db.query("..."); },
}));
// users.find is fully typed
Common Mistakes
Using string keys for DI tokens: container.get("UserService") — no type safety
Use branded symbols or a typed map for compile-time safety
Circular dependencies: ServiceA depends on ServiceB which depends on ServiceA
Break cycles with interfaces/factories or lazy resolution
Over-abstracting simple apps with heavyweight DI frameworks
For most apps, Pick<Deps, "needed" | "services"> is sufficient
Key Rules
  • Functional DI (Pick<Deps, ...>) is the simplest approach — use it for most TypeScript projects.
  • Token-based DI is needed when you have multiple implementations of the same interface.
  • The goal is testability — if you can swap dependencies in tests, you have enough DI.

Type-Safe Routing Patterns Application Patterns
Routes are strings at runtime but structured data at compile time. Type-safe routing catches broken links, missing parameters, and wrong handler signatures before deployment.

Example 1 — Basic route typing:

TypeScript
// Define route shapes
type Routes = {
  "/": {};
  "/users": {};
  "/users/:id": { id: string };
  "/users/:id/posts/:postId": { id: string; postId: string };
};

type RoutePath = keyof Routes;
type RouteParams<T extends RoutePath> = Routes[T];

// Type-safe link builder
function buildPath<T extends RoutePath>(
  path: T,
  ...args: RouteParams<T> extends Record<string, never>
    ? []
    : [params: RouteParams<T>]
): string {
  const [params] = args;
  if (!params) return path;
  return Object.entries(params).reduce(
    (p, [k, v]) => p.replace(`:${k}`, v as string),
    path as string
  );
}

console.log(buildPath("/"));                                    // "/"
console.log(buildPath("/users/:id", { id: "42" }));            // "/users/42"
console.log(buildPath("/users/:id/posts/:postId", { id: "1", postId: "5" })); // "/users/1/posts/5"
// buildPath("/users/:id");  // Error: expected params argument
// buildPath("/users/:id", { wrong: "x" }); // Error: 'wrong' not in params
/ /users/42 /users/1/posts/5

Example 2 — Extract params from path template:

TypeScript
// Automatically extract param names from a path string
type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof ExtractParams<Rest>]: string }
    : T extends `${string}:${infer Param}`
    ? { [K in Param]: string }
    : {};

// Test it
type P1 = ExtractParams<"/users/:id">;
// { id: string }

type P2 = ExtractParams<"/users/:id/posts/:postId">;
// { id: string; postId: string }

type P3 = ExtractParams<"/about">;
// {}
P1 = { id: string } P2 = { id: string; postId: string } P3 = {}

Example 3 — Type-safe route handler registry:

TypeScript
type Handler<TParams> = (params: TParams, req: Request) => Response | Promise<Response>;

class Router<TRoutes extends Record<string, Record<string, string>>> {
  private handlers = new Map<string, Handler<any>>();

  on<T extends keyof TRoutes & string>(
    path: T,
    handler: Handler<TRoutes[T]>
  ): this {
    this.handlers.set(path, handler);
    return this;
  }

  handle(path: string, params: Record<string, string>, req: Request): Response | Promise<Response> {
    const handler = this.handlers.get(path);
    if (!handler) return new Response("Not Found", { status: 404 });
    return handler(params, req);
  }
}

const router = new Router<Routes>()
  .on("/users/:id", (params, req) => {
    // params.id is string
    // params.foo — Error: 'foo' does not exist
    return new Response(`User ${params.id}`);
  })
  .on("/", (_params, _req) => new Response("Home"));

Example 4 — Next.js App Router typing:

TypeScript (Next.js)

Common Mistakes
Using plain string for route paths — no validation of params
Define a routes type map and use it to constrain path functions
Hardcoding route strings throughout the app — fragile to changes

Forgetting to handle undefined searchParams

  • Template literal types can auto-extract params from path strings at the type level.
  • For Next.js/React Router, lean on the framework's built-in typing rather than rolling your own.

Effect Pattern & Functional Error Handling Functional Patterns
The Effect pattern (inspired by Effect-TS/fp-ts) replaces exceptions with typed, composable error channels. Every function declares what can go wrong in its return type — no surprise throw at runtime.

Example 1 — Basic Result type:

TypeScript
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function ok<T>(value: T): Result<T, never> {
  return { ok: true, value };
}

function err<E>(error: E): Result<never, E> {
  return { ok: false, error };
}

// Usage
function divide(a: number, b: number): Result<number, "division_by_zero"> {
  if (b === 0) return err("division_by_zero");
  return ok(a / b);
}

const result = divide(10, 0);
if (result.ok) {
  console.log(result.value); // number
} else {
  console.log(result.error); // "division_by_zero"
}
division_by_zero

Example 2 — Chaining with map/flatMap:

TypeScript
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function map<T, U, E>(result: Result<T, E>, fn: (value: T) => U): Result<U, E> {
  return result.ok ? ok(fn(result.value)) : result;
}

function flatMap<T, U, E1, E2>(
  result: Result<T, E1>,
  fn: (value: T) => Result<U, E2>
): Result<U, E1 | E2> {
  return result.ok ? fn(result.value) : result;
}

// Chain operations — errors accumulate in the union
type ParseError = { type: "parse"; input: string };
type ValidationError = { type: "validation"; field: string };

function parseNumber(input: string): Result<number, ParseError> {
  const n = Number(input);
  return isNaN(n) ? err({ type: "parse", input }) : ok(n);
}

function validatePositive(n: number): Result<number, ValidationError> {
  return n > 0 ? ok(n) : err({ type: "validation", field: "must be positive" });
}

const result2 = flatMap(parseNumber("42"), validatePositive);
// Result<number, ParseError | ValidationError>
result2: Result<number, ParseError | ValidationError>

Example 3 — Async Result (TaskResult):

TypeScript
type AsyncResult<T, E> = Promise<Result<T, E>>;

async function fetchUser(id: string): AsyncResult<User, "not_found" | "network_error"> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (res.status === 404) return err("not_found" as const);
    return ok(await res.json() as User);
  } catch {
    return err("network_error" as const);
  }
}

async function getUserName(id: string): AsyncResult<string, "not_found" | "network_error"> {
  const userResult = await fetchUser(id);
  return map(userResult, user => user.name);
}

// Caller knows EXACTLY what can go wrong
const nameResult = await getUserName("123");
if (!nameResult.ok) {
  switch (nameResult.error) {
    case "not_found": console.log("User not found"); break;
    case "network_error": console.log("Network failed"); break;
    // TypeScript ensures all cases handled
  }
}

Example 4 — Collecting multiple errors:

TypeScript
function validate(input: { name: string; age: string; email: string }):
  Result<{ name: string; age: number; email: string }, ValidationError[]> {

  const errors: ValidationError[] = [];

  if (!input.name.trim()) errors.push({ type: "validation", field: "name" });

  const age = Number(input.age);
  if (isNaN(age) || age < 0) errors.push({ type: "validation", field: "age" });

  if (!input.email.includes("@")) errors.push({ type: "validation", field: "email" });

  return errors.length > 0
    ? err(errors)
    : ok({ name: input.name.trim(), age, email: input.email });
}

const valid = validate({ name: "Alice", age: "30", email: "a@b.com" });
// Result<{ name: string; age: number; email: string }, ValidationError[]>
valid: Result<{ name: string; age: number; email: string }, ValidationError[]>

Example 5 — Pattern matching with exhaustive handling:

TypeScript
type AppError =
  | { type: "not_found"; resource: string }
  | { type: "unauthorized"; reason: string }
  | { type: "validation"; fields: string[] };

function handleError(error: AppError): Response {
  switch (error.type) {
    case "not_found":
      return new Response(`${error.resource} not found`, { status: 404 });
    case "unauthorized":
      return new Response(`Unauthorized: ${error.reason}`, { status: 401 });
    case "validation":
      return new Response(`Invalid: ${error.fields.join(", ")}`, { status: 400 });
    default: {
      const _: never = error; // exhaustiveness check
      return new Response("Unknown error", { status: 500 });
    }
  }
}
Common Mistakes
Mixing Result types with thrown exceptions — inconsistent error handling
Choose one approach per layer: Result for business logic, try/catch at boundaries
Using Result<T, string> — stringly-typed errors lose discrimination
Use discriminated unions: Result<T, { type: "not_found" } | { type: "auth" }>
Deeply nesting if (result.ok) checks
Use map/flatMap to chain operations cleanly
Key Rules
  • Use Result types for business logic where you want callers to handle errors explicitly.
  • Keep try/catch at system boundaries (I/O, third-party libs) and convert to Result immediately.
  • Error types should be discriminated unions — enables exhaustive switch and precise handling.
