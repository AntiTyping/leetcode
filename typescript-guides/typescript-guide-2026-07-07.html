Review — 6 Concepts
R1. this Types & Fluent Interfaces Functions & Patterns 15 days since review · rep 3 → 4
The polymorphic this type refers to “the type of the current instance” — and it stays that type through inheritance. Returning this from a mutator lets a subclass keep chaining and still see its own methods, which a hard-coded : Builder return type would erase. It is the type-level backbone of every fluent API.
Key Pattern 
class QueryBuilder {
  private parts: string[] = [];
  select(cols: string): this { this.parts.push(`SELECT ${cols}`); return this; }
  from(t: string): this { this.parts.push(`FROM ${t}`); return this; }
  build(): string { return this.parts.join(" "); }
}
class PgQueryBuilder extends QueryBuilder {
  returning(cols: string): this { return this; }
}
// returns PgQueryBuilder, so .returning stays visible after .select().from()
new PgQueryBuilder().select("*").from("users").returning("id");
`this` return type preserves the subclass through the whole chain
Challenge Exercise

Write a this-typed type guard method isReady(): this is ReadyBuilder on a builder so that after calling it in an if, only the “ready” methods become available. Then explain why returning this beats returning the class name, and what breaks if a mixin uses ThisType<T> incorrectly.

Pitfall: Annotating a chainable method’s return type as the concrete class (: QueryBuilder) instead of this — a subclass instance is downcast to the base after the first call, and every subclass-only method vanishes from the chain.
R2. Async Patterns & Promise Typing Functions & Patterns 15 days since review · rep 3 → 4
An async function always returns Promise<T>, and Awaited<T> recursively unwraps nested promises to the settled value type. Typed concurrency helpers (Promise.all preserving a tuple, Promise.allSettled giving a result union) let you fan out work without losing per-element types.
Key Pattern 
async function load(id: string): Promise<number> { return id.length; }

type R = Awaited<ReturnType<typeof load>>;        // number (unwrapped)

// Promise.all preserves the positional tuple types
const [a, b] = await Promise.all([load("x"), Promise.resolve("hi")]);
//     ^ number   ^ string

// allSettled -> a discriminated union per element
const settled = await Promise.allSettled([load("x")]);
if (settled[0].status === "fulfilled") settled[0].value; // number
Awaited unwraps; Promise.all keeps a typed tuple; allSettled yields a union
Challenge Exercise

Type a mapConcurrent<T, R>(items: T[], limit: number, fn: (t: T) => Promise<R>): Promise<R[]> that runs at most limit tasks at once and preserves R. Then write a withTimeout<T>(p: Promise<T>, ms: number): Promise<T> using Promise.race and explain why the reject branch doesn’t widen T.

Pitfall: Typing an async function’s return as T instead of Promise<T> (or double-wrapping as Promise<Promise<T>>). Also: Promise.all rejects on the first failure — reach for allSettled when you need every outcome, and remember its elements are a union you must narrow on status.
R3. Type-Safe SQL Builders Database Typing 14 days since review · rep 1 → 2
A type-safe query builder threads a schema type through each call so the columns you select determine the row type you get back, and a bad column name is a compile error. The generic accumulates: each .select("id") narrows the result to Pick<Row, "id"> rather than the whole row.
Key Pattern 
interface Schema { users: { id: number; email: string; age: number } }

class Query<T, K extends keyof T = keyof T> {
  constructor(private cols: K[] = []) {}
  select<C extends keyof T>(...c: C[]): Query<T, C> { return new Query(c); }
  // execute resolves to only the selected columns
  run(): Pick<T, K>[] { return []; }
}

const rows = new Query<Schema["users"]>().select("id", "email").run();
//    ^ Pick<users, "id" | "email">[]  — 'age' is absent, a typo won't compile
selected keys drive the result type; unknown columns fail at compile time
Challenge Exercise

Extend the builder with a typed where<C extends keyof T>(col: C, val: T[C]) so the value must match the column’s type (where("age", "old") fails). Then add orderBy restricted to already-selected keys, and explain how Pick<T, K> keeps run() honest as the selection changes.

Pitfall: Letting select return the full row type (or any) instead of Pick<T, K> — you get columns you never queried, and downstream code reads fields that aren’t in the SQL. Keep the selected-keys generic flowing all the way to the result.
R4. Branded Types for Domain Modeling Advanced TypeScript Patterns 14 days since review · rep 1 → 2
TypeScript is structural: UserId and OrderId are both string and freely interchange. A brand attaches a phantom tag (string & { __brand: "UserId" }) so the two become incompatible, and a smart constructor is the only way to mint one — giving you nominal typing where the domain demands it.
Key Pattern 
declare const brand: unique symbol;
type Brand<T, B extends string> = T & { readonly [brand]: B };

type UserId = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;

function userId(s: string): UserId { /* validate */ return s as UserId; }

function loadUser(id: UserId) { /* ... */ }
loadUser(userId("u_1"));            // ✓
// loadUser("u_1");                  // ✗ plain string not assignable to UserId
// loadUser(orderId("o_9") as any);  // brands stop accidental id swaps
same runtime string, distinct compile-time types; only the constructor mints one
Challenge Exercise

Build a generic makeBrand<B extends string>(validate: (s: string) => boolean) factory that returns a smart constructor throwing on invalid input, plus a type guard is<B>. Use it for Email and NonEmptyString, then explain why the brand vanishes at runtime (zero cost) yet still blocks mixing ids at compile time.

Pitfall: Sprinkling as UserId casts everywhere instead of funneling construction through one validating factory — the brand then guarantees nothing, because any raw string can claim it. The value of a brand is that the cast lives in exactly one audited place.
R5. Builder Pattern with Type Narrowing Advanced TypeScript Patterns 14 days since review · rep 1 → 2
A type-state builder tracks which required fields have been set in the type, so build() only exists once every mandatory field is present. Each setter returns a builder with that field’s key flipped from missing to present, turning “you forgot to set the url” from a runtime error into a compile error.
Key Pattern 
interface Config { url: string; method: string; body?: string }

class ReqBuilder<Set extends keyof Config = never> {
  private data: Partial<Config> = {};
  url(u: string): ReqBuilder<Set | "url"> { this.data.url = u; return this as any; }
  method(m: string): ReqBuilder<Set | "method"> { this.data.method = m; return this as any; }
  // build is only callable once "url" | "method" are in Set
  build(this: ReqBuilder<"url" | "method">): Config { return this.data as Config; }
}

new ReqBuilder().url("/a").method("GET").build();   // ✓
// new ReqBuilder().url("/a").build();                // ✗ 'method' not set
the `this` parameter on build() requires the accumulated key set
Challenge Exercise

Make setters reject being called twice (calling .url() after it’s already set should not compile) by constraining the method on this: ReqBuilder<Exclude<keyof Config, "url">>-style bounds. Then explain how the this parameter (R1) and the accumulating union type combine to enforce a required-field protocol.

Pitfall: Making every field optional on a plain object builder and validating in build() at runtime — you lose the whole point. Encode required-ness in the type parameter and gate build() with a this constraint so a missing field can never reach runtime.
R6. Zustand with TypeScript State Management & Architecture 13 days since review · rep 1 → 2
Zustand’s create needs the store type provided explicitly via the curried form create<State>()(...), because TS can’t infer the state from the setter body. Once typed, selectors are fully inferred, and slices compose by intersecting their state types.
Key Pattern 
import { create } from "zustand";

interface CounterState {
  count: number;
  inc: () => void;
  addBy: (n: number) => void;
}

// note the curried create<State>()(...) form — required for correct inference
const useCounter = create<CounterState>()((set) => ({
  count: 0,
  inc: () => set((s) => ({ count: s.count + 1 })),
  addBy: (n) => set((s) => ({ count: s.count + n })),
}));

const count = useCounter((s) => s.count);   // number, inferred selector
curried create pins the state type; selectors and set() are then fully typed
Challenge Exercise

Split the store into two typed slices (createAuthSlice, createCartSlice) using the StateCreator<Full, [], [], Slice> signature and compose them into one store whose type is the intersection. Then type a persist middleware wrapper and explain why the curried create<T>() form is needed for middleware to keep inference.

Pitfall: Using create<State>((set) => ...) (non-curried) with middleware — inference collapses and set/selectors go any. Always use create<State>()(...), and type each slice with StateCreator so the composed store stays sound.
1Event-Driven Architecture
1. Event Sourcing with Typed Events Node.js + TypeScript
Instead of storing the current state of an entity, event sourcing stores the ordered list of facts that happened (AccountOpened, MoneyDeposited) and rebuilds state by folding over them. The type-system win is enormous: model your events as a discriminated union, write one exhaustive reducer, and the compiler guarantees every event kind is handled — add a new event and every fold that forgets it fails to build.

Three pieces make it type-safe. First, an event union discriminated on a type tag, each carrying its own typed payload. Second, an apply/evolve function (state, event) => state with an exhaustive switch guarded by a never default — that default is what turns “unhandled event” into a compile error. Third, a fold (events.reduce(apply, initial)) that replays history into the current state.

Because state is derived, you can rebuild it at any time, add new projections after the fact, and audit exactly what happened. The events are your source of truth and must be append-only and immutable — you never edit a past event; you append a new corrective one.

1
Define the event as a discriminated union on type.
2
Write an exhaustive apply(state, event) with a never guard.
3
Fold the event stream to derive current state; append, never mutate.
TypeScript · example 1 of 4 — events as a discriminated union
type AccountEvent =
  | { type: "Opened"; owner: string }
  | { type: "Deposited"; amount: number }
  | { type: "Withdrawn"; amount: number };

interface Account { owner: string; balance: number; open: boolean }
each event kind carries only the fields it needs; `type` is the discriminant
TypeScript · example 2 of 4 — the exhaustive apply function
const EMPTY: Account = { owner: "", balance: 0, open: false };

function apply(state: Account, e: AccountEvent): Account {
  switch (e.type) {
    case "Opened":    return { ...state, owner: e.owner, open: true };
    case "Deposited": return { ...state, balance: state.balance + e.amount };
    case "Withdrawn": return { ...state, balance: state.balance - e.amount };
    default:
      const _exhaustive: never = e;   // ← compile error if an event is unhandled
      return state;
  }
}
adding a new event kind without a case makes `_exhaustive: never` fail to compile
TypeScript · example 3 of 4 — fold history into current state
const history: AccountEvent[] = [
  { type: "Opened", owner: "Ada" },
  { type: "Deposited", amount: 100 },
  { type: "Withdrawn", amount: 30 },
];

const current = history.reduce(apply, EMPTY);
console.log(current);
{ owner: 'Ada', balance: 70, open: true }
TypeScript · example 4 of 4 — a typed decide() that produces new events
type Command = { type: "Withdraw"; amount: number };

// business rule lives here; returns events to append (never mutates state)
function decide(state: Account, cmd: Command): AccountEvent[] {
  if (!state.open) throw new Error("account closed");
  if (cmd.amount > state.balance) throw new Error("insufficient funds");
  return [{ type: "Withdrawn", amount: cmd.amount }];
}

const events = decide(current, { type: "Withdraw", amount: 20 });
const next = events.reduce(apply, current);   // balance: 50
decide() validates and emits events; apply() folds them — a clean command→event→state loop
Common Mistakes:
❌ Storing mutable current state and treating events as an afterthought log — the log drifts from reality and can’t rebuild state.
✅ Make the event stream the source of truth; derive all state by folding, so replay is always correct.
❌ A non-exhaustive switch with a silent default: return state — a new event kind is silently ignored.
✅ Assign the default case to const _: never = e so an unhandled event is a compile error.
Key Rules:
  • Model events as a discriminated union on a type tag; keep payloads minimal and immutable.
  • Derive state by folding apply(state, event) — never store mutable current state as the truth.
  • Guard the reducer’s default with a never assignment for compile-time exhaustiveness.
  • Events are append-only: correct a mistake with a new compensating event, never by editing history.
Practice Exercises:
1Add a Closed event and watch the never guard flag the missing case; then handle it so a closed account rejects deposits in decide.
2Write a generic fold<S, E>(events: E[], apply: (s: S, e: E) => S, seed: S): S and reuse it for a second aggregate (a shopping cart).
3Add an event version field and a replayTo(version) that folds only events up to a point — the basis of time-travel debugging.
Connections: The event union + never guard is exhaustive discriminated-union handling applied to persistence. The decide→apply split is the seed of CQRS (Concept 2), and folding a stream to state mirrors the reducer discipline behind Redux Toolkit and the type-safe event emitters from earlier days.
2. CQRS: Typed Commands & Queries Node.js + TypeScript
CQRS (Command Query Responsibility Segregation) splits the write side (commands that change state) from the read side (queries that return data) into separate models. Typed correctly, a single command registry maps each command name to its input and result, so a dispatcher rejects unknown commands and mismatched payloads — you get a type-safe message bus instead of a bag of any handlers.

Model both sides as a registry interface: a mapping from message name → { input; output }. A generic dispatch<K>(name: K, input: Registry[K]["input"]) then returns Promise<Registry[K]["output"]> — the command name pins both the payload and the result. Handlers are stored as { [K in keyof Registry]: (input) => output }, so a handler with the wrong signature won’t register.

Commands typically return void or an id (they cause effects — in an event-sourced system they emit events); queries return read-model data and cause no changes. Keeping the two registries separate is the whole point: the read model can be denormalized and optimized independently of the write model.

1
Declare a registry mapping message → { input; output }.
2
Type the handler map as { [K in keyof R]: Handler<R[K]> }.
3
A generic dispatch(name, input) pins payload and result by key.
TypeScript · example 1 of 4 — a command registry keyed by name
interface Commands {
  CreateOrder: { input: { items: string[] }; output: { orderId: string } };
  CancelOrder: { input: { orderId: string }; output: void };
}
each command pins its own input and output types in one place
TypeScript · example 2 of 4 — a fully typed handler map
type Handlers = { [K in keyof Commands]:
  (input: Commands[K]["input"]) => Promise<Commands[K]["output"]> };

const handlers: Handlers = {
  CreateOrder: async ({ items }) => ({ orderId: `o_${items.length}` }),
  CancelOrder: async ({ orderId }) => { /* emit OrderCancelled */ },
  // CreateOrder: async () => ({ wrong: 1 }),  // ✗ output shape mismatch
};
a handler whose return shape doesn't match Commands[K]["output"] won't compile
TypeScript · example 3 of 4 — a generic dispatch that pins types by key
async function dispatch<K extends keyof Commands>(
  name: K, input: Commands[K]["input"],
): Promise<Commands[K]["output"]> {
  return handlers[name](input as never) as Promise<Commands[K]["output"]>;
}

const { orderId } = await dispatch("CreateOrder", { items: ["a", "b"] });
// await dispatch("CreateOrder", { orderId: "x" });  // ✗ wrong input for this command
// await dispatch("Nope", {} as never);              // ✗ 'Nope' is not a command
the command name drives both the required input and the resolved output type
TypeScript · example 4 of 4 — a separate query side (read model)
interface Queries {
  GetOrder:   { input: { orderId: string }; output: { orderId: string; total: number } };
  ListOrders: { input: { userId: string };  output: Array<{ orderId: string }> };
}

// queries never mutate — they read an optimized projection
async function ask<K extends keyof Queries>(
  name: K, input: Queries[K]["input"],
): Promise<Queries[K]["output"]> { /* read from read-model store */ return null as never; }

const order = await ask("GetOrder", { orderId: "o_2" });  // { orderId; total }
reads and writes are separate registries; each side evolves independently
Common Mistakes:
❌ A single handle(msg: any): any bus — every call site loses input/output checking and typos in message names slip through.
✅ Drive dispatch off a registry interface with keyof; the name selects both payload and result types.
❌ Returning read-model data from command handlers — you re-couple the write and read sides you just separated.
✅ Keep commands effect-only (void/id) and put all data-shaping on the query side’s projection.
Key Rules:
  • Represent each side as a registry mapping message name → { input; output }.
  • Type the handler map with a mapped type so a wrong signature fails to register.
  • Make dispatch<K> generic over the key so name, payload, and result stay in lockstep.
  • Commands cause effects (return void/id); queries return data and never mutate.
Practice Exercises:
1Add middleware to dispatch (logging, validation) as a typed array of (name, input, next) => ... that preserves the per-command result type.
2Wire the command side to Concept 1: have CreateOrder return the events to append, then fold them into the read model that GetOrder reads.
3Validate command input at the boundary with Zod, and infer the input type from the schema so the registry and validator can’t drift.
Connections: The registry-keyed dispatcher is indexed-access + mapped types (Registry[K]["input"]) — the same shape as the tRPC/message-router typing and the type-safe plugin host. Commands emitting events links straight to event sourcing (Concept 1); the outbox (Concept 3) reliably ships those events.
3. Transactional Outbox Pattern Node.js + TypeScript
You update the database and want to publish an event (“order placed”) to a message broker. Do them separately and a crash between the two loses the event or double-sends it. The outbox pattern writes the event into an outbox table in the same transaction as the state change; a separate relay then publishes rows and marks them sent — giving at-least-once delivery. Typing the outbox row keeps the serialized payload honest end-to-end.

Inside one DB transaction (Concept 4 from 07-06), you both write the business change and INSERT a row into outbox capturing the event type and payload. Because they share a transaction, they commit or roll back together — the event exists iff the state change did. A background relay polls unpublished rows, sends them to the broker, and marks them published; if it crashes after sending but before marking, the row is re-sent (hence at-least-once, so consumers must be idempotent — 07-06).

Type-safety comes from a discriminated OutboxEvent union and a serializer that pins { type, payload }, so a row can only hold a known event with the right payload shape. On read, parse back into the union (Zod) before dispatching.

1
In one transaction: write state and insert a typed outbox row.
2
A relay polls unpublished rows and sends them to the broker.
3
Mark published; on retry, re-send (consumers are idempotent).
TypeScript · example 1 of 4 — a typed outbox event and row
type OutboxEvent =
  | { type: "OrderPlaced"; payload: { orderId: string; total: number } }
  | { type: "OrderShipped"; payload: { orderId: string; carrier: string } };

interface OutboxRow {
  id: string;
  event: OutboxEvent;          // stored serialized; parsed back to the union on read
  publishedAt: string | null;  // null = not yet sent
}
the row can only carry a known event whose payload matches its type
TypeScript · example 2 of 4 — write state + event atomically
async function placeOrder(tx: Tx, order: Order, event: OutboxEvent): Promise<void> {
  await tx.query("INSERT INTO orders(id, total) VALUES ($1, $2)", [order.id, order.total]);
  await tx.query(
    "INSERT INTO outbox(id, type, payload, published_at) VALUES ($1, $2, $3, NULL)",
    [crypto.randomUUID(), event.type, JSON.stringify(event.payload)],
  );
}

// both rows commit together — the event exists iff the order does
await withTransaction(pool, (tx) =>
  placeOrder(tx, order, { type: "OrderPlaced", payload: { orderId: order.id, total: order.total } }));
shared transaction: no window where the order exists but its event was lost
TypeScript · example 3 of 4 — the relay: publish then mark sent
async function relayOnce(broker: Broker): Promise<number> {
  const rows = await db.query<OutboxRow>(
    "SELECT * FROM outbox WHERE published_at IS NULL ORDER BY id LIMIT 100");
  for (const row of rows) {
    await broker.publish(row.event.type, row.event.payload);   // send
    await db.query("UPDATE outbox SET published_at = now() WHERE id = $1", [row.id]);  // then mark
  }
  return rows.length;
}
send-before-mark => at-least-once; a crash in between re-sends, never drops
TypeScript · example 4 of 4 — parse the stored row back into the union
import { z } from "zod";

const OutboxEventSchema = z.discriminatedUnion("type", [
  z.object({ type: z.literal("OrderPlaced"),  payload: z.object({ orderId: z.string(), total: z.number() }) }),
  z.object({ type: z.literal("OrderShipped"), payload: z.object({ orderId: z.string(), carrier: z.string() }) }),
]);

function toEvent(type: string, payloadJson: string): OutboxEvent {
  return OutboxEventSchema.parse({ type, payload: JSON.parse(payloadJson) });  // validated union
}
runtime parse re-establishes the compile-time union at the trust boundary
Common Mistakes:
❌ Committing the DB write, then publishing to the broker as a second step — a crash in between loses the event (dual-write problem).
✅ Insert the outbox row inside the same transaction; let a relay publish it afterward.
❌ Marking a row published before the broker confirms — a failed send is now lost forever.
✅ Publish first, mark second; accept at-least-once and make consumers idempotent (07-06).
Key Rules:
  • Write the state change and the outbox row in one transaction so they commit atomically.
  • A separate relay publishes unpublished rows, then marks them — send before mark.
  • Delivery is at-least-once: consumers must dedupe (idempotency keys / event ids).
  • Store events as a discriminated union and re-parse with Zod on read to restore type safety.
Practice Exercises:
1Add a retryCount and move rows that fail N times to a dead_letter table; type the relay result as { sent: number; deadLettered: number }.
2Give each event a stable eventId and have the consumer store processed ids so re-sends are ignored (tie into idempotency from 07-06).
3Replace polling with LISTEN/NOTIFY (Postgres) to wake the relay on insert, and keep the row types unchanged.
Connections: The shared transaction reuses withTransaction / Tx from 07-06’s data-integrity day, and at-least-once delivery makes the idempotency-key consumer essential. The stored-then-parsed event is a discriminated union crossing a trust boundary via Zod — the same round-trip you’ll use for message queues (Concept 4).
2Distributed Messaging & Coordination
4. Type-Safe Message Queues Node.js + TypeScript
Message brokers (Kafka, RabbitMQ, SQS) move opaque bytes — the SDKs hand you Buffer or string and forget the message’s shape the moment it’s serialized. The fix is a typed topic map: one interface pins each topic’s message type, and thin publish/subscribe wrappers make the producer and every consumer agree on that shape, with runtime validation at the deserialize boundary.

Declare a TopicMap interface: topic name → message type. A generic publish<T extends keyof TopicMap>(topic: T, msg: TopicMap[T]) serializes with the right type, and subscribe<T>(topic: T, handler: (msg: TopicMap[T]) => Promise<void>) hands the consumer a fully typed message. Because bytes crossing a network can be anything, the subscribe wrapper parses the payload (Zod) before calling the handler — recovering the compile-time type at runtime.

Keep the schema map next to the topic map so they can’t drift: derive the TS type from the schema with z.infer. This gives you one source of truth per topic, safe producers, safe consumers, and a validation gate on the wire.

1
Declare a TopicMap: topic name → message type.
2
Generic publish/subscribe keyed by topic pin the message type.
3
Parse the payload on receive to restore the type at the boundary.
TypeScript · example 1 of 4 — a typed topic map
interface TopicMap {
  "user.registered": { userId: string; email: string };
  "order.placed":    { orderId: string; total: number };
}
type Topic = keyof TopicMap;
one interface is the single source of truth for every topic's message shape
TypeScript · example 2 of 4 — a typed publisher over a raw broker
interface RawBroker { send(topic: string, bytes: string): Promise<void>; }

async function publish<T extends Topic>(
  broker: RawBroker, topic: T, msg: TopicMap[T],
): Promise<void> {
  await broker.send(topic, JSON.stringify(msg));      // msg is pinned to the topic
}

await publish(broker, "order.placed", { orderId: "o1", total: 42 });   // ✓
// await publish(broker, "order.placed", { orderId: "o1" });            // ✗ 'total' missing
the topic argument selects the exact message type the payload must satisfy
TypeScript · example 3 of 4 — schema map + validating subscribe
import { z } from "zod";
const schemas = {
  "user.registered": z.object({ userId: z.string(), email: z.string().email() }),
  "order.placed":    z.object({ orderId: z.string(), total: z.number() }),
} satisfies { [K in Topic]: z.ZodType<TopicMap[K]> };

function subscribe<T extends Topic>(topic: T, handler: (msg: TopicMap[T]) => Promise<void>) {
  rawConsumer.on(topic, async (bytes: string) => {
    const msg = schemas[topic].parse(JSON.parse(bytes)) as TopicMap[T];   // validate on the wire
    await handler(msg);
  });
}
`satisfies` forces each schema to match its topic's type; parse restores it at runtime
TypeScript · example 4 of 4 — a consumer sees a fully typed message
subscribe("user.registered", async (msg) => {
  //                                    ^ { userId: string; email: string }
  await sendWelcomeEmail(msg.email);
  // msg.orderId;   // ✗ not on this topic's message
});
the handler's msg is inferred from the topic — no casting, no `any`
Common Mistakes:
❌ Passing JSON.parse(bytes) straight to the handler as the message type — that’s an unchecked cast; a malformed message crashes deep in business logic.
✅ Validate with the topic’s schema at the deserialize boundary, then hand the handler a proven type.
❌ Duplicating the message shape in a TS interface and a schema that can drift apart.
✅ Keep one source of truth: derive the type via z.infer, or bind schemas to the map with satisfies.
Key Rules:
  • Model topics as an interface (topic → message type); key publish/subscribe generics off it.
  • Serialize with the pinned type; always validate on deserialize before the handler runs.
  • Bind schemas to the topic map with satisfies so the two can’t drift.
  • Assume at-least-once delivery — handlers must be idempotent and tolerate re-delivery.
Practice Exercises:
1Add a message envelope ({ id, occurredAt, data }) as a generic Envelope<TopicMap[T]> and thread it through publish/subscribe.
2Type a dead-letter path: when schemas[topic].parse throws, route the raw bytes to a <topic>.dlq topic with the error attached.
3Support schema evolution: add an optional field to order.placed and write a consumer that handles both old and new messages without breaking.
Connections: The topic-keyed generics are the same indexed-access mechanics as the CQRS dispatcher (Concept 2) and WebSocket message maps. The parse-on-receive step is runtime validation (Zod) at a trust boundary, and at-least-once delivery is exactly why the outbox (Concept 3) pairs with idempotent consumers.
5. Saga Pattern & Distributed Coordination Node.js + TypeScript
A single business transaction can span multiple services (reserve inventory, charge payment, book shipping) — you can’t hold one DB transaction across all of them. A saga runs the steps in sequence and, if a later step fails, executes compensating actions to undo the earlier ones. Pairing each step with its compensation in the type makes “every action is undoable” a compile-time guarantee, not a hopeful convention.

Model a saga as an ordered list of steps, each a { action, compensate } pair sharing a typed context. The runner executes actions forward, pushing each completed step onto an undo stack; on any failure it pops the stack and runs each compensate in reverse. Because a step’s compensate is declared right next to its action and typed against the same context, you can’t add a forward action without also saying how to undo it.

Compensations are best-effort and should be idempotent (a retry of “refund” must not double-refund). This is orchestration (a central coordinator drives the steps); the alternative is choreography (services react to each other’s events over a message queue — Concept 4). Sagas give you eventual consistency, not atomicity: there’s a window where some steps are done and others compensated.

1
Define each step as an action + compensate over a shared context.
2
Run actions forward, recording completed steps on an undo stack.
3
On failure, run compensations in reverse (idempotent, best-effort).
TypeScript · example 1 of 4 — a step pairs an action with its compensation
interface SagaStep<Ctx> {
  name: string;
  action: (ctx: Ctx) => Promise<void>;
  compensate: (ctx: Ctx) => Promise<void>;   // how to undo `action`
}

interface OrderCtx { orderId: string; reserved?: boolean; charged?: boolean }
every step must declare a compensation — undoability is structural
TypeScript · example 2 of 4 — concrete steps over a shared context
const steps: SagaStep<OrderCtx>[] = [
  { name: "reserve",
    action:     async (c) => { await inventory.reserve(c.orderId); c.reserved = true; },
    compensate: async (c) => { if (c.reserved) await inventory.release(c.orderId); } },
  { name: "charge",
    action:     async (c) => { await payments.charge(c.orderId); c.charged = true; },
    compensate: async (c) => { if (c.charged) await payments.refund(c.orderId); } },
];
each compensate guards on the flag its action set — safe to run even if partial
TypeScript · example 3 of 4 — the runner with reverse compensation
async function runSaga<Ctx>(steps: SagaStep<Ctx>[], ctx: Ctx): Promise<void> {
  const done: SagaStep<Ctx>[] = [];
  try {
    for (const step of steps) { await step.action(ctx); done.push(step); }
  } catch (err) {
    for (const step of done.reverse()) {            // undo in reverse order
      try { await step.compensate(ctx); }
      catch (e) { console.error(`compensation ${step.name} failed`, e); }  // best-effort
    }
    throw err;
  }
}
forward until a throw; then compensate completed steps last-to-first
TypeScript · example 4 of 4 — running it; a mid-saga failure rolls back
const ctx: OrderCtx = { orderId: "o_42" };
try {
  await runSaga(steps, ctx);        // if `charge` throws, `reserve` is released
} catch {
  // saga failed and compensations ran; surface eventual-consistency status to the user
}
a failure in step 2 triggers step 1's compensate — no dangling reservation
Common Mistakes:
❌ Treating a multi-service flow like one ACID transaction — there is no shared rollback across services, so a failure leaves orphaned effects.
✅ Pair each step with a compensation and run them in reverse on failure — accept eventual consistency.
❌ Non-idempotent compensations — a retried “refund” or double “release” corrupts balances/stock.
✅ Guard compensations on the flag their action set and make them idempotent so re-runs are safe.
Key Rules:
  • Structure a saga as steps, each an action paired with a compensate over one typed context.
  • Record completed steps; on failure, run compensations in reverse order.
  • Compensations are best-effort and must be idempotent — guard on state set by the action.
  • A saga yields eventual consistency, not atomicity; surface the in-between status honestly.
Practice Exercises:
1Add a per-step retry (with backoff from 07-04) before a step is considered failed and compensation kicks in.
2Persist the saga’s progress (an event-sourced log, Concept 1) so a crashed orchestrator can resume or compensate on restart.
3Rewrite the flow as choreography: each service reacts to the previous step’s event over the typed queue (Concept 4) and emits a compensation event on failure. Compare the tradeoffs with orchestration.
Connections: The saga is the distributed counterpart to 07-06’s single-DB transaction — when withTransaction can’t span services, compensations replace rollback. It builds on the whole day: steps emit events (Concept 1), are triggered by commands (Concept 2), delivered reliably via the outbox (Concept 3), and choreographed over typed queues (Concept 4).
