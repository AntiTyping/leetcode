R1. Enums Structuring Types rep 1 · last reviewed 2026-05-31 · 2 days ago
TypeScript enums: numeric (auto-increment from 0), string (explicit values), const enums (inlined at compile time). Prefer literal union types (type Dir = "up" | "down") over enums for most cases — they're simpler and tree-shake better. Use enums when you need reverse mapping or runtime iteration.
Key Pattern 
// const enum — inlined at compile time, no runtime object
const enum Direction { Up = "UP", Down = "DOWN" }
let d = Direction.Up; // compiles to: let d = "UP";

// Literal union — preferred for most use cases
type Direction2 = "up" | "down";
function move(dir: Direction2) { /* ... */ }
Challenge Exercise

Create a numeric enum HttpStatus starting at 200 (with Ok, Created, BadRequest=400, NotFound=404). Then write a isSuccess(status: HttpStatus): boolean that returns true for 2xx codes. What happens if you pass 999 as HttpStatus? Why?

#1 Pitfall: Numeric enums allow any number assignment at runtime — let s: HttpStatus = 999 compiles without error. String enums are safer because only declared values are allowed.
R2. Conditional Types Advanced Type System rep 0 · last reviewed 2026-06-01 · 1 day ago
T extends U ? X : Y — type-level if/else. Distributive over unions when T is a naked type parameter. Use infer to extract types from complex structures.
Key Pattern 
type IsString<T> = T extends string ? true : false;

type A = IsString<string>;         // true
type B = IsString<number>;         // false
type C = IsString<string | number>; // true | false (distributive!)

// Extract with infer
type Flatten<T> = T extends Array<infer U> ? U : T;
type D = Flatten<string[]>;  // string
type E = Flatten<number>;    // number
Challenge Exercise

Write type ReturnOf<T> that extracts the return type of a function type using infer. Test with ReturnOf<() => string> (should be string) and ReturnOf<(x: number) => boolean> (should be boolean).

#1 Pitfall: Forgetting distributive behavior — IsString<string | number> evaluates to true | false (distributes over each union member), not false. Wrap in [T] extends [string] to prevent distribution.
R3. Mapped Types Advanced Type System rep 0 · last reviewed 2026-06-01 · 1 day ago
Transform all properties of a type: { [K in keyof T]: NewType }. Use as for key remapping. Add/remove readonly and ? modifiers with +/-.
Key Pattern 
// Remap keys with template literal types
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};

interface Person { name: string; age: number }
type PersonGetters = Getters<Person>;
// { getName: () => string; getAge: () => number }
Challenge Exercise

Write type Mutable<T> that removes readonly from every property of T. Test with Mutable<{ readonly x: number; readonly y: string }> which should become { x: number; y: string }.

#1 Pitfall: Forgetting string & K when using template literals with keyof — keyof includes number | symbol, but template literals only work with string. Without the intersection, you get a type error.
R4. keyof & Indexed Access Types Advanced Type System rep 0 · last reviewed 2026-06-01 · 1 day ago
keyof T gets union of property names. T[K] accesses the type at key K. Combine for type-safe property access patterns.
Key Pattern 
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { name: "Alice", age: 30 };
const name = getProperty(user, "name"); // type: string
const age = getProperty(user, "age");   // type: number
// getProperty(user, "email");          // Error: "email" not in keyof
Challenge Exercise

Write a type-safe pick function: function pick<T, K extends keyof T>(obj: T, ...keys: K[]): Pick<T, K>. Implement the body and test it — the return type should narrow correctly based on the keys passed.

#1 Pitfall: keyof returns string | number | symbol. If you need to use the result in string operations (e.g., template literals, Object.keys), narrow it with & string.
R5. Type Predicates & Assertion Functions Advanced Type System rep 0 · last reviewed 2026-06-01 · 1 day ago
param is Type for custom type guards in return type. asserts param is Type for assertion functions that throw on failure. Both narrow types in subsequent code.
Key Pattern 
function isString(x: unknown): x is string {
  return typeof x === "string";
}

function assertString(x: unknown): asserts x is string {
  if (typeof x !== "string") throw new Error("Not a string");
}

const val: unknown = "hello";
if (isString(val)) {
  val.toUpperCase(); // narrowed to string
}
assertString(val);
val.toUpperCase(); // narrowed to string after assertion
Challenge Exercise

Write a type guard isNonNull<T>(val: T): val is NonNullable<T> and use it to filter an array: [1, null, 2, undefined, 3].filter(isNonNull) — the result should be typed as number[].

#1 Pitfall: The compiler trusts your predicate unconditionally. If your runtime check doesn't match the type assertion, you get silent type mismatches. Always ensure the check logic truly validates the type.
R6. satisfies Operator Advanced Type System rep 0 · last reviewed 2026-06-01 · 1 day ago
expr satisfies Type validates type compatibility without widening. Preserves literal types and specific method availability while ensuring the value conforms to a type.
Key Pattern 
const config = {
  port: 3000,
  host: "localhost"
} satisfies Record<string, string | number>;

// config.port is still number (not string | number)
// config.host is still string (not string | number)
config.port.toFixed(2); // works — type is number
config.host.toUpperCase(); // works — type is string
Challenge Exercise

Define a type Route = { path: string; method: "GET" | "POST" }. Create a routes object { home: ..., submit: ... } using satisfies Record<string, Route>. Verify that routes.home.method retains the literal type "GET" rather than widening to "GET" | "POST".

#1 Pitfall: Confusing satisfies with as. as overrides the inferred type (unsafe — can lie to the compiler). satisfies validates without changing the inferred type (safe).

1. Function Overloads Functions & Patterns
TypeScript function overloads let you describe functions that accept different argument shapes and return different types accordingly — essential for DOM APIs, parsers, and any function with polymorphic behavior.

Unlike Java or C#, TypeScript overloads are not separate implementations. You write multiple overload signatures (the public API) followed by a single implementation signature (the actual function body). The implementation signature is invisible to callers — only the overload signatures are callable.

  • 2026-07-04 (Day 31)
  • Overload signatures appear first, each with their own parameter and return types
  • The implementation signature must be compatible with all overloads (usually wider types)
  • TypeScript resolves overloads top-to-bottom — put the most specific signature first
  • Only the overload signatures are visible to callers; the implementation signature is hidden
TypeScript — Basic Overload
// Overload signatures (public API)
function greet(name: string): string;
function greet(names: string[]): string[];

// Implementation signature (hidden from callers)
function greet(input: string | string[]): string | string[] {
  if (typeof input === "string") {
    return `Hello, ${input}!`;
  }
  return input.map(name => `Hello, ${name}!`);
}

const single = greet("Alice");    // type: string
const multi = greet(["A", "B"]);  // type: string[]
// greet(42);                     // Error: no matching overload
TypeScript — DOM-style Overload (different return types based on input)
function createElement(tag: "input"): HTMLInputElement;
function createElement(tag: "canvas"): HTMLCanvasElement;
function createElement(tag: "div"): HTMLDivElement;
function createElement(tag: string): HTMLElement;

function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const input = createElement("input");   // HTMLInputElement
const canvas = createElement("canvas"); // HTMLCanvasElement
const div = createElement("div");       // HTMLDivElement
const span = createElement("span");     // HTMLElement (generic fallback)

// input.value is available — compiler knows it's HTMLInputElement
input.value = "typed correctly";
TypeScript — Event Handler Overload
interface ClickEvent { type: "click"; x: number; y: number }
interface KeyEvent { type: "key"; key: string; code: number }

function on(event: "click", handler: (e: ClickEvent) => void): void;
function on(event: "key", handler: (e: KeyEvent) => void): void;

function on(
  event: string,
  handler: (e: ClickEvent | KeyEvent) => void
): void {
  // register handler...
}

on("click", (e) => {
  console.log(e.x, e.y);   // e is ClickEvent, x/y available
});
on("key", (e) => {
  console.log(e.key);       // e is KeyEvent, key available
});
When NOT to Use Overloads

If all overload signatures have the same return type, use a union parameter instead. If the relationship between input and output is mechanical, a generic or conditional type is cleaner.

❌ Unnecessary overload — same return type:
function len(s: string): number;
function len(a: unknown[]): number;
function len(x: string | unknown[]): number { return x.length; }
✅ Simple union parameter:
function len(x: string | unknown[]): number { return x.length; }
TypeScript — Overloads vs Conditional Return Types
// Approach 1: Overloads (explicit, readable)
function parse(input: string): object;
function parse(input: Buffer): object;
function parse(input: string | Buffer): object {
  return JSON.parse(typeof input === "string" ? input : input.toString());
}

// Approach 2: Conditional return type (one signature, type-level logic)
function format<T extends string | number>(
  value: T
): T extends string ? string : number {
  if (typeof value === "string") {
    return value.toUpperCase() as any; // need assertion inside
  }
  return (value * 100) as any;
}

// Overloads: better when signatures are very different
// Conditionals: better when there's a systematic input/output mapping
Common Mistakes
❌ Implementation signature too narrow — must handle all overloads:
function foo(x: string): string;
function foo(x: number): number;
function foo(x: string): string | number { ... } // Error!
✅ Implementation signature wide enough:
function foo(x: string | number): string | number { ... }
❌ Different parameter count without optional params:
function bar(a: string): void;
function bar(a: string, b: number): void;
function bar(a: string, b: number): void { ... } // Missing overload for 1-arg case
✅ Implementation handles all arities:
function bar(a: string, b?: number): void { ... }
Key Rules
  • Overload signatures must appear directly before the implementation — no other code in between
  • The implementation signature is not callable directly — callers only see overload signatures
  • Order matters: put the most specific overload first (TypeScript resolves top-down)
  • Use overloads when return types differ based on input; use unions/generics for simpler cases
  • Method overloads in classes and interfaces follow the same pattern
Exercises
1 Write an overloaded fetchData function: when called with "json", it returns Promise<object>; when called with "text", it returns Promise<string>; when called with "blob", it returns Promise<Blob>. Write all signatures and an implementation body.
2 Write an overloaded parse function: parse(input: string, asArray: true) returns string[], parse(input: string, asArray: false) returns string, parse(input: string) returns string. The function splits on commas when asArray is true.
3 Refactor a set of overloads into a single generic with conditional return type: function convert(x: string): number; function convert(x: number): string; — write it as one generic signature using T extends string | number with a conditional return type.
Connections: Function overloads pair naturally with Conditional Types — sometimes one replaces the other. The createElement pattern connects to keyof & Indexed Access for looking up return types from a tag map. Generic Constraints (next) offer an alternative approach for polymorphic functions.
2. Generic Constraints & Defaults Functions & Patterns
Raw generics accept anything. Constraints let you require specific shapes from type parameters — enabling you to write functions that are both flexible and type-safe. Default type parameters make generics easier to use when a common case exists.

You already know <T> from Generics Basics (Day 1). Now we add constraints (extends) to require T to have certain properties, defaults (= SomeType) to provide fallbacks, and multiple constraints using intersections.

  • <T extends SomeType> — T must be assignable to SomeType
  • <T = DefaultType> — if T is not provided, it defaults to DefaultType
  • <T extends A & B> — T must satisfy both A and B simultaneously
  • Constraints work with infer in conditional types to extract nested structures
TypeScript — Basic Constraint
// Without constraint — T could be anything
function getIdUnsafe<T>(item: T): string {
  return item.id; // Error: Property 'id' does not exist on type 'T'
}

// With constraint — T must have an id property
function getId<T extends { id: string }>(item: T): string {
  return item.id; // works: T guaranteed to have id
}

getId({ id: "abc", name: "Alice" }); // OK — has id
getId({ name: "Bob" });               // Error: missing 'id'
TypeScript — Multiple Constraints & Defaults
interface Serializable {
  serialize(): string;
}

interface Loggable {
  log(level: string): void;
}

// T must satisfy BOTH interfaces
function process<T extends Serializable & Loggable>(item: T): string {
  item.log("info");
  return item.serialize();
}

// Default type parameter
interface ApiResponse<TData = unknown, TError = Error> {
  data: TData | null;
  error: TError | null;
  status: number;
}

// All three usages are valid:
type UserResponse = ApiResponse<{ name: string }>;       // TError defaults to Error
type SimpleResponse = ApiResponse;                        // both default
type CustomResponse = ApiResponse<string, CustomError>;   // both explicit
TypeScript — Type-Safe Merge Function
function merge<
  T extends Record<string, unknown>,
  U extends Record<string, unknown>
>(target: T, source: U): T & U {
  return { ...target, ...source };
}

const result = merge(
  { name: "Alice", age: 30 },
  { email: "alice@example.com", active: true }
);
// type: { name: string; age: number } & { email: string; active: boolean }

result.name;   // string
result.email;  // string
result.active; // boolean
TypeScript — Generic Repository Pattern
interface Entity {
  id: string;
  createdAt: Date;
}

class Repository<T extends Entity> {
  private items = new Map<string, T>();

  save(item: T): void {
    this.items.set(item.id, item);
  }

  findById(id: string): T | undefined {
    return this.items.get(id);
  }

  findAll(): T[] {
    return [...this.items.values()];
  }

  // Constraint lets us access id and createdAt on any T
  findRecent(since: Date): T[] {
    return this.findAll().filter(item => item.createdAt > since);
  }
}

interface User extends Entity { name: string; email: string }
const userRepo = new Repository<User>();
userRepo.save({ id: "1", name: "Alice", email: "a@b.com", createdAt: new Date() });

const user = userRepo.findById("1");
// type: User | undefined — full User type preserved
TypeScript — infer in Constraint Position
// Extract the resolved type from a Promise
type UnwrapPromise<T> = T extends Promise<infer U> ? U : T;

type A = UnwrapPromise<Promise<string>>;   // string
type B = UnwrapPromise<Promise<number[]>>; // number[]
type C = UnwrapPromise<boolean>;            // boolean (not a Promise)

// Extract function argument types
type FirstArg<T> = T extends (first: infer A, ...rest: any[]) => any ? A : never;
type D = FirstArg<(x: string, y: number) => void>; // string
Common Mistakes
❌ Over-constraining — too specific, limits reusability:
function first<T extends string[]>(arr: T): string — only works with string arrays
✅ Constraint just enough:
function first<T>(arr: T[]): T — works with any array type
❌ Under-constraining — need methods but don't require them:
function serialize<T>(item: T) { return item.toJSON(); } // Error
✅ Add the constraint you need:
function serialize<T extends { toJSON(): string }>(item: T) { return item.toJSON(); }
Key Rules
  • extends in generics means "is assignable to" — the constraint is the minimum shape required
  • Default type parameters must come after non-default ones: <T, U = string> is valid, <T = string, U> is not
  • Constraints propagate: if T extends { id: string }, then T["id"] resolves to string
  • Use keyof T in constraints for dynamic key access: <T, K extends keyof T>
  • The constraint T extends object excludes primitives; T extends Record<string, unknown> requires plain objects
Exercises
1 Write a function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] that extracts a single property from an array of objects. Test: pluck([{name: "A", age: 1}, {name: "B", age: 2}], "name") should return string[].
2 Create a generic Cache<T extends { id: string }, TTL = number> class with set(item: T, ttl?: TTL) and get(id: string): T | undefined methods. The default TTL type should allow number but let users constrain it to specific literal values.
3 Write a type DeepReadonly<T> that recursively makes all properties readonly. It should handle nested objects: DeepReadonly<{ a: { b: number } }> should make both levels readonly. Use conditional types to check if a property value is an object.
Connections: Builds directly on keyof & Indexed Access for the K extends keyof T pattern. Constraints + infer tie into Conditional Types. The repository pattern leads naturally to Branded Types (next) for distinguishing entity IDs at the type level.
3. Branded/Opaque Types Functions & Patterns
TypeScript uses structural typing — two types with the same shape are interchangeable. Branded types add a phantom tag to create nominal-like typing, preventing you from accidentally swapping a UserId for an Email even though both are strings.

The core problem: string is string. A function that takes a userId: string will happily accept an email address, a URL, or any other string. In large codebases, this causes subtle bugs. Branded types solve this by intersecting a base type with a phantom tag that exists only at compile time.

  • A brand is a property that never exists at runtime — it's purely for the type checker
  • Use unique symbol or a literal string for the brand property
  • Constructor functions validate at runtime and cast to the branded type
  • Branded types are zero-cost at runtime — no extra memory or overhead
TypeScript — The Problem (No Branding)
// Both are just strings — easily confused
function sendEmail(to: string, userId: string) { /* ... */ }

const email = "alice@example.com";
const userId = "usr_12345";

// Oops! Arguments swapped — compiles fine, breaks at runtime
sendEmail(userId, email); // no error, but wrong!
TypeScript — Branded Types with unique symbol
// Declare branded types
declare const EmailBrand: unique symbol;
declare const UserIdBrand: unique symbol;

type Email = string & { readonly [EmailBrand]: typeof EmailBrand };
type UserId = string & { readonly [UserIdBrand]: typeof UserIdBrand };

// Constructor functions — validate and brand
function Email(value: string): Email {
  if (!value.includes("@")) throw new Error(`Invalid email: ${value}`);
  return value as Email;
}

function UserId(value: string): UserId {
  if (!value.startsWith("usr_")) throw new Error(`Invalid user ID: ${value}`);
  return value as UserId;
}

// Now the function is type-safe
function sendEmail(to: Email, userId: UserId) { /* ... */ }

const email = Email("alice@example.com");
const userId = UserId("usr_12345");

sendEmail(email, userId);    // OK
sendEmail(userId, email);    // Error: UserId not assignable to Email
sendEmail("raw@string", userId); // Error: string not assignable to Email
TypeScript — Simpler Brand Pattern (string literal)
// Lighter-weight pattern using a string literal brand
type Brand<T, B extends string> = T & { readonly __brand: B };

type Meters = Brand<number, "Meters">;
type Feet = Brand<number, "Feet">;

function metersToFeet(m: Meters): Feet {
  return (m * 3.28084) as Feet;
}

function feetToMeters(f: Feet): Meters {
  return (f / 3.28084) as Meters;
}

const height = 1.8 as Meters;
const inFeet = metersToFeet(height);   // OK: Feet

// metersToFeet(inFeet);  // Error: Feet not assignable to Meters
// metersToFeet(42);       // Error: number not assignable to Meters
TypeScript — Validated Branded Types (Production Pattern)
type Brand<T, B extends string> = T & { readonly __brand: B };

type PositiveNumber = Brand<number, "PositiveNumber">;
type NonEmptyString = Brand<string, "NonEmptyString">;
type Percentage = Brand<number, "Percentage">;

// Smart constructors with validation
function PositiveNumber(n: number): PositiveNumber {
  if (n <= 0) throw new RangeError(`Expected positive, got ${n}`);
  return n as PositiveNumber;
}

function Percentage(n: number): Percentage {
  if (n < 0 || n > 100) throw new RangeError(`Expected 0-100, got ${n}`);
  return n as Percentage;
}

// Type-safe calculations
function calculateDiscount(price: PositiveNumber, pct: Percentage): PositiveNumber {
  return PositiveNumber(price * (1 - pct / 100));
}

const price = PositiveNumber(49.99);
const discount = Percentage(15);
const final = calculateDiscount(price, discount);

// calculateDiscount(49.99, 15);          // Error: number not assignable
// calculateDiscount(discount, price);    // Error: types swapped
Common Mistakes
❌ Forgetting runtime validation — the brand is compile-time only:
const bad = "not-an-email" as Email; // compiles but no validation!
✅ Always use constructor functions that validate:
const good = Email("user@example.com"); // throws if invalid
❌ Expecting brand to exist at runtime:
if (value.__brand === "Email") { ... } // __brand is undefined at runtime
✅ Use type guards for runtime checks:
function isEmail(v: string): v is Email { return v.includes("@"); }
Key Rules
  • Branded types are purely compile-time — no runtime overhead, no extra properties
  • Always pair a branded type with a constructor function that validates the value
  • as BrandedType should only appear inside the constructor function, never in application code
  • Branded types compose: type ValidatedUserId = UserId & { readonly __validated: true }
  • Libraries like Zod, io-ts, and Effect use branded types extensively for validation
Exercises
1 Create branded types for Email, Password (min 8 chars, at least one number), and Username (alphanumeric, 3-20 chars). Write constructor functions with proper validation for each. Write a registerUser(email: Email, password: Password, username: Username) function.
2 Create type USD = Brand<number, "USD"> and type EUR = Brand<number, "EUR">. Write a convert(amount: USD, rate: PositiveNumber): EUR function. Verify that convert(eurAmount, rate) produces a type error.
3 Combine branded types with type predicates: write isPositiveNumber(n: number): n is PositiveNumber and use it with Array.filter to get a PositiveNumber[] from a number[].
Connections: Branded types rely on Type Predicates for runtime narrowing and satisfies for validating branded values inline. The Brand<T, B> utility type uses the same intersection patterns from Mapped Types. Constructor functions are a natural use case for Function Overloads when you want to accept multiple input formats.
4. this Types & Fluent Interfaces Functions & Patterns
The polymorphic this type in TypeScript refers to the type of the current class — including subclasses. It enables the fluent interface pattern (method chaining) where each method returns this, and subclass methods automatically return the correct subclass type.

When a method returns this, TypeScript understands that the return type is the actual concrete class — not just the base class. This means inherited methods in subclasses return the subclass type automatically, enabling seamless chaining across inheritance hierarchies.

  • this as a type — polymorphic, changes meaning based on the concrete class
  • Fluent interface: every mutating method returns this for chaining
  • ThisParameterType<T> — extracts the this type from a function type
  • OmitThisParameter<T> — removes the this parameter from a function type
  • Arrow functions don't have their own this — they capture it from the enclosing scope
TypeScript — Basic Fluent Interface
class QueryBuilder {
  private conditions: string[] = [];
  private sortField?: string;
  private maxResults?: number;

  where(condition: string): this {
    this.conditions.push(condition);
    return this;
  }

  orderBy(field: string): this {
    this.sortField = field;
    return this;
  }

  limit(n: number): this {
    this.maxResults = n;
    return this;
  }

  build(): string {
    let query = "SELECT * FROM table";
    if (this.conditions.length) {
      query += " WHERE " + this.conditions.join(" AND ");
    }
    if (this.sortField) query += ` ORDER BY ${this.sortField}`;
    if (this.maxResults) query += ` LIMIT ${this.maxResults}`;
    return query;
  }
}

// Fluent chaining
const query = new QueryBuilder()
  .where("age > 18")
  .where("active = true")
  .orderBy("name")
  .limit(10)
  .build();
// "SELECT * FROM table WHERE age > 18 AND active = true ORDER BY name LIMIT 10"
TypeScript — this Type with Inheritance
class BasicBuilder {
  protected data: Record<string, unknown> = {};

  set(key: string, value: unknown): this {
    this.data[key] = value;
    return this;
  }

  toJSON(): string {
    return JSON.stringify(this.data);
  }
}

class TimestampedBuilder extends BasicBuilder {
  withTimestamp(): this {
    this.data.createdAt = new Date().toISOString();
    return this;
  }
}

// Because set() returns 'this', chaining works correctly on the subclass
const json = new TimestampedBuilder()
  .set("name", "Alice")       // returns TimestampedBuilder (not BasicBuilder!)
  .withTimestamp()             // still TimestampedBuilder — this method is available
  .set("role", "admin")       // still TimestampedBuilder
  .toJSON();

// Without 'this' return type, .set() would return BasicBuilder,
// and .withTimestamp() would cause a type error
TypeScript — Explicit this Parameter
// You can declare 'this' as a parameter for functions
function logName(this: { name: string }) {
  console.log(this.name);
}

const obj = { name: "Alice", logName };
obj.logName(); // "Alice"

// const fn = obj.logName;
// fn(); // Error: 'this' context of type 'void' is not assignable

// Utility types for this
type LogNameFn = typeof logName;
type ThisType = ThisParameterType<LogNameFn>; // { name: string }
type Bare = OmitThisParameter<LogNameFn>;     // () => void
TypeScript — Chainable Validation Builder
class Validator<T> {
  private checks: Array<(value: T) => string | null> = [];

  addCheck(fn: (value: T) => string | null): this {
    this.checks.push(fn);
    return this;
  }

  validate(value: T): { valid: boolean; errors: string[] } {
    const errors = this.checks
      .map(check => check(value))
      .filter((e): e is string => e !== null);
    return { valid: errors.length === 0, errors };
  }
}

class StringValidator extends Validator<string> {
  minLength(n: number): this {
    return this.addCheck(v =>
      v.length < n ? `Must be at least ${n} characters` : null
    );
  }

  matches(pattern: RegExp): this {
    return this.addCheck(v =>
      !pattern.test(v) ? `Must match ${pattern}` : null
    );
  }
}

const passwordValidator = new StringValidator()
  .minLength(8)
  .matches(/[0-9]/)
  .matches(/[A-Z]/)
  .addCheck(v => v.includes(" ") ? "Must not contain spaces" : null);

const result = passwordValidator.validate("Hello1");
// { valid: false, errors: ["Must be at least 8 characters"] }
Common Mistakes
❌ Forgetting return this in fluent methods:
where(cond: string): this { this.conditions.push(cond); } // returns undefined
✅ Always return this:
where(cond: string): this { this.conditions.push(cond); return this; }
❌ Using arrow functions for methods that need this:
class Foo { bar = () => { return this; } } // this type is Foo, not polymorphic this
✅ Use regular methods for polymorphic this:
class Foo { bar(): this { return this; } }
Key Rules
  • this as a return type is polymorphic — it refers to the concrete class, not the declaring class
  • Arrow functions capture this lexically — they don't have their own polymorphic this
  • You can declare this as a first parameter: function foo(this: MyType) — it's erased at compile time
  • ThisParameterType<T> extracts and OmitThisParameter<T> removes the this parameter type
  • The fluent/builder pattern works best when all methods return this, not a specific class name
Exercises
1 Build a fluent EventEmitter class with on(event: string, handler: Function): this, off(event: string, handler: Function): this, and emit(event: string, ...args: unknown[]): this. Create a subclass TypedEmitter that adds a once(event: string, handler: Function): this method. Verify that chaining works across both classes.
2 Write a HttpRequest builder: new HttpRequest().url("/api").method("POST").header("Content-Type", "json").body({ data: 1 }).send(). Each method except send() returns this. Create a subclass AuthenticatedRequest with .bearerToken(token: string): this.
3 Use the explicit this parameter to write a function function describe(this: { name: string; age: number }): string that returns a formatted description. Call it using .call() with a matching object. Then use OmitThisParameter to create a version without the this requirement.
Connections: The this return type works like a special case of Generic Constraints — the type is implicitly constrained to the current class. The builder pattern benefits from Branded Types for ensuring required fields are set. Fluent methods connect to Async Patterns (next) when you need to chain Promise-returning builder steps.
5. Async Patterns & Promise Typing Functions & Patterns
Nearly every TypeScript application deals with async operations — API calls, file I/O, timers. Understanding how TypeScript types Promises, async/await, and async iteration prevents an entire class of bugs where you accidentally work with Promise<T> instead of T.

async functions always return Promise<T>. TypeScript infers the inner type from your return statements. await unwraps a Promise<T> to T. The Awaited<T> utility type (built-in since TS 4.5) recursively unwraps nested Promises at the type level.

  • async function f(): Promise<T> — return type is always a Promise
  • await unwraps one level of Promise; Awaited<T> unwraps all levels
  • Promise.all preserves tuple types: [Promise<A>, Promise<B>] becomes [A, B]
  • Promise.allSettled returns PromiseSettledResult<T>[] with status discriminant
  • Async iterators: AsyncIterable<T> and for await...of for streaming data
TypeScript — Promise Basics
// Explicit return type
async function fetchUser(id: string): Promise<{ name: string; age: number }> {
  const response = await fetch(`/api/users/${id}`);
  return response.json(); // inferred as Promise<any> from json()
}

// TypeScript infers the return type from the function body
async function getGreeting() {
  return "hello"; // inferred: Promise<string>
}

// await unwraps the Promise
async function main() {
  const user = await fetchUser("1");  // type: { name: string; age: number }
  const greeting = await getGreeting(); // type: string

  // Common mistake: forgetting await
  const oops = fetchUser("1"); // type: Promise<{ name: string; age: number }>
  // oops.name; // Error: Property 'name' does not exist on type 'Promise<...>'
}
TypeScript — Awaited<T> and Promise.all
// Awaited recursively unwraps Promises
type A = Awaited<Promise<string>>;            // string
type B = Awaited<Promise<Promise<number>>>;   // number (recursive!)
type C = Awaited<string>;                      // string (non-Promise passes through)

// Promise.all preserves tuple types
async function loadDashboard() {
  const [user, posts, settings] = await Promise.all([
    fetchUser("1"),                              // Promise<User>
    fetch("/api/posts").then(r => r.json()),     // Promise<any>
    fetch("/api/settings").then(r => r.json()),  // Promise<any>
  ]);
  // user: User, posts: any, settings: any
  // Tuple type preserved — not flattened to a union

  return { user, posts, settings };
}

// Promise.allSettled — never rejects
async function loadWithFallbacks() {
  const results = await Promise.allSettled([
    fetchUser("1"),
    fetchUser("2"),
  ]);
  // type: PromiseSettledResult<{ name: string; age: number }>[]

  for (const result of results) {
    if (result.status === "fulfilled") {
      console.log(result.value.name); // narrowed: value is User
    } else {
      console.log(result.reason);      // narrowed: reason is any
    }
  }
}
TypeScript — Type-Safe API Client
interface ApiEndpoints {
  "/users": { response: { id: string; name: string }[] };
  "/users/:id": { response: { id: string; name: string; email: string } };
  "/posts": { response: { id: string; title: string }[] };
}

async function apiGet<T extends keyof ApiEndpoints>(
  endpoint: T
): Promise<ApiEndpoints[T]["response"]> {
  const response = await fetch(endpoint);
  return response.json();
}

// Return types are inferred from the endpoint map
const users = await apiGet("/users");
// type: { id: string; name: string }[]

const posts = await apiGet("/posts");
// type: { id: string; title: string }[]

// apiGet("/unknown"); // Error: "/unknown" not in keyof ApiEndpoints
TypeScript — Async Generators for Pagination
interface Page<T> {
  items: T[];
  nextCursor?: string;
}

async function* paginate<T>(
  fetchPage: (cursor?: string) => Promise<Page<T>>
): AsyncGenerator<T[], void, undefined> {
  let cursor: string | undefined;
  do {
    const page = await fetchPage(cursor);
    yield page.items;
    cursor = page.nextCursor;
  } while (cursor);
}

// Usage with for-await-of
async function getAllUsers() {
  const allUsers: { name: string }[] = [];

  for await (const batch of paginate<{ name: string }>(
    async (cursor) => {
      const res = await fetch(`/api/users?cursor=${cursor ?? ""}`);
      return res.json();
    }
  )) {
    allUsers.push(...batch); // batch is { name: string }[]
  }

  return allUsers;
}
TypeScript — Typed Async Error Handling
// TypeScript catch blocks always type error as 'unknown' (since TS 4.4)
class ApiError extends Error {
  constructor(
    message: string,
    public statusCode: number,
    public endpoint: string
  ) {
    super(message);
    this.name = "ApiError";
  }
}

async function fetchSafe<T>(url: string): Promise<T> {
  try {
    const res = await fetch(url);
    if (!res.ok) {
      throw new ApiError(`HTTP ${res.status}`, res.status, url);
    }
    return await res.json();
  } catch (error: unknown) {
    // Must narrow 'unknown' before using
    if (error instanceof ApiError) {
      console.log(`API error at ${error.endpoint}: ${error.statusCode}`);
      throw error;
    }
    if (error instanceof Error) {
      console.log(`Network error: ${error.message}`);
    }
    throw new ApiError("Unknown error", 500, url);
  }
}

// Result type pattern for avoiding try/catch
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

async function fetchResult<T>(url: string): Promise<Result<T, ApiError>> {
  try {
    const data = await fetchSafe<T>(url);
    return { ok: true, value: data };
  } catch (error) {
    return { ok: false, error: error as ApiError };
  }
}
Common Mistakes
❌ Forgetting to await — getting Promise<T> instead of T:
const user = fetchUser("1"); user.name; // Error: name not on Promise
✅ Always await async calls when you need the value:
const user = await fetchUser("1"); user.name;
❌ Returning await unnecessarily in a non-try/catch:
async function get() { return await fetchUser("1"); } // redundant await
✅ Return the Promise directly (unless inside try/catch):
async function get() { return fetchUser("1"); }
❌ Using error: Error in catch block:
catch (e: Error) { ... } // Error: only 'unknown' or 'any' allowed
✅ Use unknown and narrow:
catch (e: unknown) { if (e instanceof Error) { ... } }
Key Rules
  • async functions always return Promise<T> even if you return T directly
  • Awaited<T> recursively unwraps: Awaited<Promise<Promise<string>>> is string
  • Promise.all with a tuple literal preserves each element's type; with an array, types are unioned
  • Catch blocks type errors as unknown — always narrow before accessing properties
  • Prefer Promise.allSettled over Promise.all when you need all results regardless of individual failures
  • Async generators (async function*) yield AsyncGenerator<Yield, Return, Next>
Exercises
1 Write a typed retry<T>(fn: () => Promise<T>, attempts: number): Promise<T> function that retries a failed async operation up to N times. It should rethrow the last error if all attempts fail. Add a delay parameter with exponential backoff.
2 Write a timeout<T>(promise: Promise<T>, ms: number): Promise<T> that rejects with a custom TimeoutError if the promise doesn't resolve within ms milliseconds. Use Promise.race.
3 Create an async generator pollEndpoint<T>(url: string, intervalMs: number): AsyncGenerator<T> that fetches a URL at intervals and yields the result each time. Use it with for await...of to print updates until a status === "complete" field appears in the response.
Connections: Async error handling connects directly to Type Predicates for narrowing unknown errors. The Result type is a discriminated union pattern from Day 2.
