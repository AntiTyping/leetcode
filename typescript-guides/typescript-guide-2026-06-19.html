1 Runtime Type Patterns
#1 · Streaming & Async Iteration Typing Runtime Type Patterns

TypeScript’s AsyncIterable<T> and ReadableStream<T> types make streaming data processing type-safe — essential for real-time APIs, large file processing, and Server-Sent Events where you can’t hold the whole payload in memory.

Streaming flips the usual model: instead of awaiting a single value, you consume a sequence of values that arrive over time. TypeScript has first-class types for every layer of this.

1
AsyncIterator<T> and AsyncIterable<T> are the protocol behind for await...of. Anything with a [Symbol.asyncIterator]() method can be looped asynchronously.
2
async function* generators are the easiest way to produce an AsyncIterable<T>. Each yield expr contributes one value of type T.
3
The Web Streams API adds ReadableStream<T>, WritableStream<T>, and TransformStream<I, O> — the I/O-flavored cousins of iterators, with backpressure.
4
You consume streams with .getReader(), or compose them with .pipeTo() and .pipeThrough().
5
Server-Sent Events are just a typed event stream — parse the wire format into a discriminated union and yield narrowed events.
TypeScript · example 1 of 4
// A simple async generator with explicit return type.
// The inferred type is AsyncGenerator<number, void, unknown>.
async function* countdown(from: number): AsyncGenerator<number> {
  for (let n = from; n > 0; n--) {
    await new Promise((r) => setTimeout(r, 100));
    yield n; // each yielded value must be a number
  }
}

for await (const n of countdown(3)) {
  console.log(n); // n is inferred as number
}
3 2 1
TypeScript · example 2 of 4
// A TransformStream that parses newline-delimited JSON into typed objects.
interface LogEntry { level: "info" | "error"; msg: string }

const jsonLines = new TransformStream<string, LogEntry>({
  transform(chunk, controller) {
    for (const line of chunk.split("\n")) {
      if (line.trim()) controller.enqueue(JSON.parse(line) as LogEntry);
    }
  },
});

// Input side accepts string, output side emits LogEntry.
const writer = jsonLines.writable.getWriter();
const reader = jsonLines.readable.getReader(); // ReadableStreamDefaultReader<LogEntry>
jsonLines: TransformStream<string, LogEntry> reader.read() resolves to { value: LogEntry; done: false } | { value: undefined; done: true }
TypeScript · example 3 of 4
// A typed SSE client that yields a discriminated union of events.
type ServerEvent =
  | { type: "message"; text: string }
  | { type: "ping"; ts: number };

async function* sseEvents(url: string): AsyncGenerator<ServerEvent> {
  const res = await fetch(url);
  const reader = res.body!.pipeThrough(new TextDecoderStream()).getReader();
  let buffer = "";
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;            // narrows value to undefined
    buffer += value;            // here value is string
    let i;
    while ((i = buffer.indexOf("\n\n")) !== -1) {
      const raw = buffer.slice(0, i).replace(/^data: /, "");
      buffer = buffer.slice(i + 2);
      yield JSON.parse(raw) as ServerEvent;
    }
  }
}
sseEvents: (url: string) => AsyncGenerator<ServerEvent, void, unknown> Consumers can switch on event.type and narrow exhaustively
TypeScript · example 4 of 4
// Node 20+: convert an async iterable into a ReadableStream.
async function* lines(): AsyncGenerator<string> {
  yield "alpha";
  yield "beta";
}

// ReadableStream.from infers the element type from the iterable.
const stream: ReadableStream<string> = ReadableStream.from(lines());

for await (const line of stream) {
  console.log(line.toUpperCase()); // line is string
}
ALPHA BETA
Common Mistakes:
❌ Typing a generator’s yield result as void when consumers feed values back in via iterator.next(value) — you lose the type of what flows backward.
✅ Use the full three-parameter form AsyncGenerator<YieldType, ReturnType, NextType> for bidirectional generators so the next() argument is typed.
❌ Accessing result.value from a reader without checking done first — the type is T | undefined.
✅ Check if (done) break; (or return) before touching value so the union narrows to the present-value branch.
Key Rules:
  • 2026-07-04 (Day 31)
  • Reach for async function* + for await...of for in-process sequences; reach for Web Streams when you need backpressure or interop with fetch bodies.
  • reader.read() always returns a { value, done } union — narrow on done before reading value.
  • Use the three-arg AsyncGenerator<Y, R, N> form only when the generator is bidirectional; otherwise one arg is enough.
Practice Exercises:
1Write an async generator that yields decoded text lines from a fetch Response body stream, typed as AsyncGenerator<string>. Split on \n and skip empty lines.
2Create a TransformStream<string, LogEntry> that parses JSON log lines into typed objects and drops any line that fails to parse rather than throwing.
3Build a typed SSE helper onEvents<M>(url, map) that accepts an event-type map and returns an async iterable of narrowed event objects, so callers never see the raw string.
Connections: Builds directly on Async Patterns & Promise Typing and the WebSocket Typing review — both rely on discriminated unions to narrow streamed messages. The TransformStream<I, O> generic is a runtime echo of the type-level Conditional Types you’ve been practicing.
#2 · Type-Safe Configuration Runtime Type Patterns

Configuration bugs are runtime bombs — a missing env var or a port typed as a string can take down production at 3am. Validating config with Zod at startup turns those failures into a single clear error before the app ever serves a request.

1
The core problem: process.env.PORT is string | undefined. TypeScript can’t know whether it’s set, and it’s never a number even when it looks like one.
2
Define a Zod schema for the environment and parse process.env through it: z.object({ PORT: z.coerce.number() }).
3
Env values are always strings, so use z.coerce to convert string → number / boolean automatically.
4
Nest config objects and supply defaults so development “just works” without a fully populated .env.
5
Export the parsed result — now config.database.port is fully typed and guaranteed present.
TypeScript · example 1 of 4
import { z } from "zod";

// Basic env validation.
const EnvSchema = z.object({
  NODE_ENV: z.enum(["development", "production", "test"]),
  PORT: z.coerce.number().int().positive(),
  API_KEY: z.string().min(1),
});

// Parse eagerly at module load — throws if anything is missing/invalid.
export const env = EnvSchema.parse(process.env);
//     ^? { NODE_ENV: "development" | "production" | "test"; PORT: number; API_KEY: string }
env.PORT is number (not string), env.NODE_ENV is a literal union
TypeScript · example 2 of 4
// Nested config with defaults and coercion.
const ConfigSchema = z.object({
  database: z.object({
    host: z.string().default("localhost"),
    port: z.coerce.number().default(5432),
    ssl: z.coerce.boolean().default(false),
  }),
  cache: z.object({
    ttlSeconds: z.coerce.number().default(60),
  }),
});

// Empty input still produces a complete, typed config via defaults.
const config = ConfigSchema.parse({ database: {}, cache: {} });
console.log(config.database.port, config.cache.ttlSeconds);
5432 60
TypeScript · example 3 of 4
// Config factory with environment-specific overrides.
type Env = "development" | "production";

function loadConfig(envName: Env) {
  const base = { logLevel: "info", retries: 3 };
  const overrides: Record<Env, Partial<typeof base>> = {
    development: { logLevel: "debug" },
    production: { retries: 5 },
  };
  return { ...base, ...overrides[envName] };
}

const cfg = loadConfig("development");
//    ^? { logLevel: string; retries: number }
cfg.logLevel === "debug", cfg.retries === 3
TypeScript · example 4 of 4
// Type-safe feature flags using literal types.
const FlagsSchema = z.object({
  newCheckout: z.coerce.boolean().default(false),
  beta: z.enum(["off", "internal", "public"]).default("off"),
});
type Flags = z.infer<typeof FlagsSchema>;

function isEnabled(flags: Flags, key: keyof Flags): boolean {
  const v = flags[key];
  return v === true || (typeof v === "string" && v !== "off");
}
Flags = { newCheckout: boolean; beta: "off" | "internal" | "public" }
Common Mistakes:
❌ Using z.number() for an env var — process.env.PORT is a string, so parsing fails even for "3000".
✅ Use z.coerce.number() so Zod parses the string into a number for you.
❌ Validating config lazily on first access — the app boots fine and crashes later on the first request that touches a bad value.
✅ Validate eagerly at startup with safeParse and fail fast with a clear message before serving traffic.
Key Rules:
  • Treat all of process.env as string | undefined — coerce, never assume.
  • Parse once, at startup, and export the typed result. Don’t scatter process.env reads across the codebase.
  • Use .default() for dev ergonomics, but require critical secrets (no default) so a missing one is an immediate error.
Practice Exercises:
1Write a Zod schema that validates DATABASE_URL, PORT, NODE_ENV, and API_KEY from process.env, coercing PORT to a number and constraining NODE_ENV to a literal union.
2Create a typed config module with nested database, auth, and cache sections, each with sensible development defaults.
3Add a validate() function that throws a single formatted error listing all missing/invalid fields at once (not just the first), using safeParse and error.issues.
Connections: Leans on the Zod & Runtime Validation and Literal Types concepts, and pairs naturally with the Type-Safe Error Boundaries card below — config validation is the first place you’ll want a typed Result instead of a thrown exception.
#3 · Type-Safe Error Boundaries Runtime Type Patterns

In strict mode, every catch clause types the error as unknown — so “catch and pray” code that reads e.message won’t even compile. Discriminated error unions and a Result type give you exhaustive, compiler-checked error handling.

1
The unknown catch: catch (e) { /* e: unknown */ }. You can’t touch any property until you narrow.
2
Narrow with a type guard: e instanceof CustomError unlocks the subclass’s fields.
3
Model expected failures as a discriminated union: type AppError = NetworkError | ValidationError | AuthError.
4
For functions that fail predictably, return a Result<T, E> — { ok: true; value: T } | { ok: false; error: E } — instead of throwing.
5
In React, an ErrorBoundary component catches render-time errors and renders a typed fallback.
TypeScript · example 1 of 4
// A typed error hierarchy with instanceof narrowing.
class NetworkError extends Error { constructor(public status: number) { super("network"); } }
class ValidationError extends Error { constructor(public field: string) { super("validation"); } }

function handle(e: unknown) {
  if (e instanceof NetworkError) return `HTTP ${e.status}`; // e: NetworkError
  if (e instanceof ValidationError) return `bad field: ${e.field}`; // e: ValidationError
  if (e instanceof Error) return e.message; // e: Error
  return "unknown error";
}
Each branch narrows e to the matching subclass and exposes its fields
TypeScript · example 2 of 4
// A Result type with chaining helpers — no try/catch at call sites.
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

const ok = <T>(value: T): Result<T, never> => ({ ok: true, value });
const err = <E>(error: E): Result<never, E> => ({ ok: false, error });

function map<T, U, E>(r: Result<T, E>, f: (t: T) => U): Result<U, E> {
  return r.ok ? ok(f(r.value)) : r;
}

const parsed = map(ok(21), (n) => n * 2);
console.log(parsed.ok && parsed.value);
42
TypeScript · example 3 of 4
// Exhaustive error handling with never.
type AppError =
  | { kind: "network"; status: number }
  | { kind: "notFound"; id: string }
  | { kind: "validation"; field: string };

function describe(e: AppError): string {
  switch (e.kind) {
    case "network": return `HTTP ${e.status}`;
    case "notFound": return `missing ${e.id}`;
    case "validation": return `invalid ${e.field}`;
    default: {
      const _exhaustive: never = e; // compile error if a kind is unhandled
      return _exhaustive;
    }
  }
}
Adding a new AppError variant without a case turns _exhaustive into a type error
TypeScript (React) · example 4 of 4
// A React ErrorBoundary with typed fallback props.
import { Component, type ReactNode } from "react";

interface Props { fallback: (error: Error) => ReactNode; children: ReactNode }
interface State { error: Error | null }

class ErrorBoundary extends Component<Props, State> {
  state: State = { error: null };
  static getDerivedStateFromError(error: Error): State { return { error }; }
  render() {
    return this.state.error
      ? this.props.fallback(this.state.error) // error: Error
      : this.props.children;
  }
}
fallback receives a typed Error, not any
Common Mistakes:
❌ Annotating the catch binding as catch (e: Error) — TypeScript rejects this in strict mode; the binding is forced to unknown (or any).
✅ Leave it as unknown and narrow with a guard: if (e instanceof Error) { e.message }.
❌ Throwing plain strings or object literals — you lose stack traces and can’t use instanceof to narrow.
✅ Always throw Error subclasses so you keep stack traces and gain reliable instanceof checks.
Key Rules:
  • Catch bindings are unknown — narrow before you touch them.
  • Use Result<T, E> for expected failures (validation, lookups); reserve throw for truly exceptional, unrecoverable cases.
  • Pair discriminated error unions with a never exhaustiveness check so new variants force new handling.
Practice Exercises:
1Create an AppError union with NetworkError, NotFoundError, and ValidationError, each carrying distinct fields, and a type guard for each.
2Implement Result<T, E> with map() and flatMap() that chain computations without a single try/catch, short-circuiting on the first error.
3Write an exhaustive error handler over all AppError variants that uses a never assignment in the default branch to guarantee completeness at compile time.
Connections: A direct extension of never & Exhaustiveness, Discriminated Unions in Practice, and Type Predicates. The Result type here is the runtime counterpart to the Error Handling Patterns you reviewed earlier this week.

#4 · Type-Safe Plugin Systems Architecture Patterns

Extensible apps need plugins, but a plugin API typed with any hands every plugin author a loaded gun. Generics and mapped types let you enforce that plugins implement the required hooks and that each event carries exactly the right payload.

1
Define a plugin interface parameterized by an event map: interface Plugin<Events extends Record<string, unknown>>.
2
Register hooks keyed by event name, with the payload type pulled from the event map — on<K>(event: K, handler: (data: Events[K]) => void).
3
A plugin manager aggregates installed plugins and dispatches typed events via emit.
4
Middleware-style chains thread a typed context object through each plugin, letting later plugins read what earlier ones added.
5
Validate each plugin’s configuration with a Zod schema at registration time, before it can do any damage.
TypeScript · example 1 of 4
// A simple plugin interface with lifecycle hooks and a typed event map.
interface Plugin<Events extends Record<string, unknown>> {
  name: string;
  init?(): void;
  destroy?(): void;
  on?: { [K in keyof Events]?: (data: Events[K]) => void };
}

interface AppEvents {
  "user:login": { userId: string };
  "page:view": { path: string };
}

const analytics: Plugin<AppEvents> = {
  name: "analytics",
  on: {
    "user:login": (data) => console.log("login", data.userId), // data typed
  },
};
data in "user:login" is { userId: string }; a wrong field name is a compile error
TypeScript · example 2 of 4
// A plugin manager with a generic event map and type-safe emit.
class PluginManager<Events extends Record<string, unknown>> {
  private plugins: Plugin<Events>[] = [];

  use(plugin: Plugin<Events>): this {
    plugin.init?.();
    this.plugins.push(plugin);
    return this;
  }

  emit<K extends keyof Events>(event: K, data: Events[K]): void {
    for (const p of this.plugins) p.on?.[event]?.(data);
  }
}

const mgr = new PluginManager<AppEvents>().use(analytics);
mgr.emit("page:view", { path: "/home" }); // payload must match the event
emit("page:view", { path: 123 }) is a compile error — path must be string
TypeScript · example 3 of 4
// Augmentation pattern: a plugin adds a typed method to the shared context.
interface Context { config: Record<string, string> }

interface LoggerContext extends Context {
  log(msg: string): void;
}

function withLogger(ctx: Context): LoggerContext {
  return { ...ctx, log: (msg) => console.log("[log]", msg) };
}

// Downstream plugins receive the widened context type.
const ctx = withLogger({ config: {} });
ctx.log("ready"); // typed method available
[log] ready
TypeScript · example 4 of 4
// Validate plugin config at registration time with Zod.
import { z } from "zod";

const RateLimitConfig = z.object({ maxPerMinute: z.number().int().positive() });
type RateLimitConfig = z.infer<typeof RateLimitConfig>;

function createRateLimiter(raw: unknown): Plugin<AppEvents> {
  const config = RateLimitConfig.parse(raw); // throws on bad config
  return { name: "rate-limit", init: () => console.log("limit", config.maxPerMinute) };
}
config: { maxPerMinute: number } — invalid config fails before install
Common Mistakes:
❌ Typing plugin hooks as (...args: any[]) => any — this erases every payload type and lets typos through silently.
✅ Drive hook signatures from a generic event map: { [K in keyof Events]?: (data: Events[K]) => void } keeps each payload typed.
❌ Installing a plugin without validating its configuration — a malformed config surfaces as a confusing runtime crash later.
✅ Validate config with a Zod schema at registration, so a bad plugin is rejected at install time with a clear message.
Key Rules:
  • Parameterize the plugin system by an event map; never use bare any for hook payloads.
  • Make use() return this so plugins compose fluently and the event-map type flows through.
  • Validate untrusted plugin config at the boundary (registration), not deep inside event handlers.
Practice Exercises:
1Design a PluginManager<EventMap> that accepts plugins with typed on(event, handler) and a typed emit(event, data), rejecting mismatched payloads at compile time.
2Create a plugin that adds a logger method to the context, typed so downstream plugins can call ctx.logger(...) with autocomplete.
3Build a plugin system where each plugin declares its dependencies as a type, and the manager checks at registration that every dependency is already installed.
Connections: Combines Generic Constraints, Mapped Types, and the Type-Safe Event Emitter pattern. The fluent use() chain reuses the this Types & Fluent Interfaces idea, and config validation reuses the Type-Safe Configuration card above.
#5 · Type-Safe Internationalization (i18n) Architecture Patterns

A typo in a translation key is an invisible bug that ships straight to production as a blank label. Type-safe i18n catches missing keys, wrong interpolation variables, and incomplete locales at compile time — before a user ever sees undefined.

1
Declare the translation object as const so TypeScript infers literal types for every key instead of widening to string.
2
Use template literal types to build nested dot-path keys like 'nav.home' | 'errors.notFound'.
3
Infer interpolation params from the template string: t('greeting', { name }) where {name} is extracted from the message.
4
Use a mapped type to require every locale to have the same keys as the default locale — a missing key becomes a type error.
5
Wire it into React with a typed useTranslation hook and <Trans> component.
TypeScript · example 1 of 4
// Literal key types via `as const`, then a t() that only accepts real keys.
const en = {
  hello: "Hello",
  bye: "Goodbye",
} as const;

type Key = keyof typeof en; // "hello" | "bye"

function t(key: Key): string { return en[key]; }

t("hello"); // ok
// t("helo"); // Argument of type '"helo"' is not assignable to '"hello" | "bye"'
Key = "hello" | "bye"; a misspelled key is rejected at compile time
TypeScript · example 2 of 4
// Nested translations with dot-path access types.
const messages = {
  nav: { home: "Home", about: "About" },
  errors: { notFound: "Not found" },
} as const;

// Recursively build "nav.home" | "nav.about" | "errors.notFound".
type Paths<T, P extends string = ""> = {
  [K in keyof T & string]: T[K] extends Record<string, unknown>
    ? Paths<T[K], `${P}${K}.`>
    : `${P}${K}`;
}[keyof T & string];

type MsgKey = Paths<typeof messages>;
//   ^? "nav.home" | "nav.about" | "errors.notFound"
MsgKey = "nav.home" | "nav.about" | "errors.notFound"
TypeScript · example 3 of 4
// Extract {param} placeholders from a string literal type.
type Params<S extends string> =
  S extends `${string}{${infer P}}${infer Rest}`
    ? { [K in P | keyof Params<Rest>]: string }
    : {};

type GreetingParams = Params<"Hi {name}, you have {count} messages">;
//   ^? { name: string; count: string }

declare function tx<S extends string>(tpl: S, params: Params<S>): string;
tx("Hi {name}", { name: "Ada" }); // params required and typed
GreetingParams = { name: string; count: string }
TypeScript · example 4 of 4
// Locale completeness: every locale must match the default's keys.
type Translations = typeof en; // the default locale's shape

const fr: Translations = {
  hello: "Bonjour",
  bye: "Au revoir",
  // omit one key -> Property 'bye' is missing in type ...
};

// Extra/typo'd keys are also rejected because the type is exact in shape.
const de = { hello: "Hallo", bye: "Tschuss" } satisfies Translations;
A missing or misspelled key in fr/de is a compile error
Common Mistakes:
❌ Typing translations as Record<string, string> — you lose autocomplete and TypeScript can’t detect a missing or misspelled key.
✅ Use as const and infer literal key types so every key is checked and autocompleted.
❌ Adding a key to one locale but forgetting the others — the app falls back to a blank string in production.
✅ Derive a Translations type from the default locale and apply it to every other locale so omissions are compile errors.
Key Rules:
  • Always declare translation tables as const — widening to string defeats the whole approach.
  • Drive the key type and the interpolation params from the message data itself; never hand-maintain a parallel key list.
  • Type non-default locales against the default locale’s shape so the compiler enforces completeness.
Practice Exercises:
1Define a translations object with as const and write a t() function that only accepts valid dot-separated key paths (reuse a Paths<T> helper).
2Create a type InterpolationParams<T> that extracts every {param} placeholder from a string literal type and returns the required params object.
3Write a LocaleChecker<Default, Target> type that resolves to never (or a list of missing keys) when Target is missing any key present in Default.
Connections: A showcase for Template Literal Types, Mapped Types, keyof & Indexed Access, and the satisfies operator working together. The recursive Paths<T> helper is the same Type-Level Programming machinery you reviewed yesterday.
