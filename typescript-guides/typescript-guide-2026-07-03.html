R1. Function Overloads Functions & Patterns 15 days since review · rep 3 → 4
Several overload signatures stacked above one implementation signature let a single function present distinct, precise call shapes — useful when input and output types are correlated in a way a plain union can’t express. The implementation signature is invisible to callers; only the overloads are.
Key Pattern 
function parse(x: string): string[];
function parse(x: number): number;
function parse(x: string | number): string[] | number {
  return typeof x === "string" ? x.split(",") : x * 2;
}

const a = parse("a,b");   // string[]
const b = parse(3);       // number
each call site picks the matching overload; the impl signature isn’t callable
Challenge Exercise

Type a createElement overload set where the return depends on the tag literal ("a" → HTMLAnchorElement, "div" → HTMLDivElement) with a broad fallback signature. Then explain why the implementation signature’s return type must be assignable to every overload yet is itself never selected by a caller.

Pitfall: Order overloads most-specific first — a broad signature placed first shadows the narrower ones. And a too-loose implementation signature (e.g. any) silences real errors inside the body while callers still only see the overloads.
R2. Module Federation Typing Advanced Patterns & Performance 6 days since review · rep 2 → 3
Module Federation loads code from a separate build at runtime (import("remoteApp/Widget")), so the host has no static types for it. You bridge the gap with a declare module "remoteApp/Widget" that describes the remote’s public contract — and version the shared types so host and remote agree.
Key Pattern 
// remotes.d.ts in the host build
declare module "checkout/Cart" {
  import type { FC } from "react";
  export interface CartProps { items: string[] }
  const Cart: FC<CartProps>;
  export default Cart;
}

// usage — now typed
const Cart = lazy(() => import("checkout/Cart"));
the declare module gives the runtime-loaded remote a static contract
Challenge Exercise

Publish a shared @org/contracts package of interfaces consumed by both host and remote, then write the declare module for a remote that exposes both a default component and a named useCartStore hook. Explain exactly what breaks — and when — if the remote’s real shape drifts from the declaration.

Pitfall: The declare module is an unchecked promise — nothing verifies the remote actually matches it, so a version skew compiles fine and then explodes at runtime. Pin the shared contract package version across host and every remote.
R3. Testing Types: tsd & expect-type Advanced Patterns & Performance 6 days since review · rep 2 → 3
Type-level tests assert your types are what you expect — that a utility returns the right shape and that an invalid call is rejected. expectTypeOf(...).toEqualTypeOf<...>() and tsd’s expectType/expectError run at compile time, guarding your public type API against regressions.
Key Pattern 
import { expectTypeOf } from "expect-type";

expectTypeOf<Awaited<Promise<number>>>().toEqualTypeOf<number>();
expectTypeOf<Partial<{ a: 1 }>>().toEqualTypeOf<{ a?: 1 }>();

// @ts-expect-error — a wrong argument type is a regression test
greet(42);
assertions fail the build when a type drifts or an expected error disappears
Challenge Exercise

Write a strict Equal<A, B> helper (the classic conditional wrapping each side in a function type) and use it to assert DeepReadonly<{ a: { b: 1 } }> equals { readonly a: { readonly b: 1 } }. Explain why a plain A extends B ? true : false is too loose for an equality assertion.

Pitfall: Using extends-based checks for equality — they’re bidirectionally loose and pass for any. Use an invariant Equal, and remember @ts-expect-error fails the build if the error silently disappears (a vanished error is a regression too).
R4. React + TypeScript: Hooks Typing React + TypeScript 6 days since review · rep 2 → 3
Most hooks infer well, but a few need help: useState wants an explicit param when the initial value is null or narrower than the eventual value; useRef splits into a mutable MutableRefObject (initial value) vs a read-only RefObject (null); useReducer shines with a discriminated action union.
Key Pattern 
const [user, setUser] = useState<User | null>(null);  // widen past null
const inputRef = useRef<HTMLInputElement>(null);      // RefObject, .current: T | null
const countRef = useRef(0);                          // MutableRefObject<number>

const [state, dispatch] = useReducer(reducer, initial);
// reducer: (s: State, a: Action) => State, Action discriminated on `type`
explicit params rescue the cases inference gets wrong
Challenge Exercise

Type a useToggle custom hook returning a [boolean, () => void] tuple (use as const or an explicit tuple return), and a useFetch<T> returning a discriminated { status: "loading" } | { status: "error"; error: Error } | { status: "success"; data: T }.

Pitfall: useState(null) infers null forever — pass useState<User | null>(null). And returning [value, setValue] from a custom hook without as const/an explicit tuple type widens it to (boolean | (() => void))[], breaking destructuring.
R5. Redux Toolkit TypeScript Patterns State Management & Architecture 6 days since review · rep 2 → 3
RTK’s types flow from the store: derive RootState = ReturnType<typeof store.getState> and AppDispatch = typeof store.dispatch, then build pre-typed useAppSelector/useAppDispatch hooks. createSlice infers action-creator payloads from reducer signatures via PayloadAction<T>.
Key Pattern 
const slice = createSlice({
  name: "counter",
  initialState: { value: 0 },
  reducers: {
    addBy(state, action: PayloadAction<number>) { state.value += action.payload; },
  },
});

export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;
selector state and dispatched thunks are typed from the store shape
Challenge Exercise

Type a createAsyncThunk<Returned, Arg, { state: RootState; rejectValue: ApiError }> and handle its three lifecycle actions in extraReducers with the builder callback, narrowing action.payload in the rejected case. Explain why the builder form types better than the object-map form.

Pitfall: Using the plain untyped useSelector/useDispatch — every selector’s state is loose and thunks lose dispatch typing. Always export and use the pre-typed hooks, and supply the thunk’s generics so rejectWithValue is checked.
R6. React + TypeScript: Generic Components React + TypeScript 18 days since review · rep 1 → 2
A generic component threads a type parameter from its props into its callbacks, so a <List items={T[]}> hands renderItem a fully-typed T. Write them as generic function components; arrow-function generics need a trailing comma (<T,>) in .tsx to disambiguate from JSX.
Key Pattern 
interface ListProps<T> {
  items: T[];
  renderItem: (item: T, index: number) => React.ReactNode;
}
function List<T>({ items, renderItem }: ListProps<T>) {
  return <ul>{items.map((it, i) => <li key={i}>{renderItem(it, i)}</li>)}</ul>;
}

<List items={users} renderItem={(u) => u.name} />   // u: User, inferred
T is inferred from `items`; renderItem’s argument is fully typed
Challenge Exercise

Build a generic Select<T> whose value, options: T[], onChange: (v: T) => void, and getLabel: (v: T) => string all share T. Make it work with forwardRef (the cast-the-forwardRef trick or a typed wrapper) and confirm T is inferred from options at the call site.

Pitfall: In .tsx, a bare arrow generic <T>(…) => parses as a JSX tag — use <T,> or a function declaration. And annotating the component as React.FC erases the generic; keep it a plain generic function so T flows from props.
1Node.js + TS: Request & Response Contracts
1. Typed Pagination & Cursor Contracts Node.js + TypeScript
Every list endpoint needs pagination, and the shape of a “page” (the items plus how to fetch the next one) is a contract clients build against. Typing offset and cursor pagination — including an opaque, encoded cursor — keeps the query params, the response envelope, and the next-page logic consistent, and stops internal ordering fields from leaking.

Two families: offset/limit (simple, but rows shift under concurrent inserts, causing skips/dupes) and cursor/keyset (stable, scales to large tables). Model the request params and a generic Page<T> envelope so metadata always has a home.

The cursor is opaque to clients: encode the last-seen sort key (e.g. { id, createdAt }) as base64 so callers treat it as a black box — that lets you change its internals later without a breaking change. Because it comes back as untrusted input, validate it on decode.

1
Define request params and a generic Page<T> envelope (offset or cursor flavour).
2
Encode the sort key into an opaque cursor; decode + validate it as untrusted input.
3
Fetch limit + 1 rows to detect a next page without a second count query.
TypeScript · example 1 of 4 — generic page envelopes
interface OffsetPage<T> {
  items: T[];
  total: number;
  limit: number;
  offset: number;
}

interface CursorParams { limit: number; cursor?: string }   // cursor is opaque
interface CursorPage<T> {
  items: T[];
  nextCursor: string | null;   // null = last page
}
a Page<T> envelope gives pagination metadata a typed home
TypeScript · example 2 of 4 — an opaque, validated cursor
interface CursorData { id: string; createdAt: number }
const CursorSchema = z.object({ id: z.string(), createdAt: z.number() });

const encodeCursor = (c: CursorData): string =>
  Buffer.from(JSON.stringify(c)).toString("base64url");

function decodeCursor(raw: string): CursorData {
  const parsed = JSON.parse(Buffer.from(raw, "base64url").toString());
  return CursorSchema.parse(parsed);   // validate — it's client-supplied
}
clients see an opaque string; the server round-trips a typed CursorData
TypeScript · example 3 of 4 — a keyset query returning a CursorPage
async function listPosts(p: CursorParams): Promise<CursorPage<Post>> {
  const after = p.cursor ? decodeCursor(p.cursor) : null;
  const rows = await db.posts.findMany({
    take: p.limit + 1,                       // one extra to detect "more"
    where: after ? { createdAt: { lt: after.createdAt } } : undefined,
    orderBy: { createdAt: "desc" },
  });
  const items = rows.slice(0, p.limit);
  const last = items.at(-1);
  const nextCursor =
    rows.length > p.limit && last
      ? encodeCursor({ id: last.id, createdAt: +last.createdAt })
      : null;
  return { items, nextCursor };
}
limit + 1 reveals whether a next page exists; nextCursor is null at the end
TypeScript · example 4 of 4 — parse & clamp query params at the boundary
const PageQuery = z.object({
  limit:  z.coerce.number().int().min(1).max(100).default(20),
  cursor: z.string().optional(),
});

// req.query values are strings — parse into typed, clamped CursorParams
const params: CursorParams = PageQuery.parse(req.query);
limit is coerced, bounded to 1..100, and defaulted — never unbounded
Common Mistakes:
❌ Returning a bare T[] — clients can’t tell if there’s a next page, and adding total/nextCursor later is a breaking change.
✅ Return a generic Page<T> envelope from day one so metadata has a home.
❌ Exposing a raw DB offset or the internal sort column as the “cursor” — clients couple to your storage and you can’t change ordering.
✅ Encode an opaque cursor and validate it on decode — it’s untrusted input.
Key Rules:
  • 2026-07-04 (Day 31)
  • Wrap list results in a generic Page<T> envelope; never return a bare array.
  • Prefer keyset/cursor pagination for large or churny datasets; offset is fine for small, stable lists.
  • Keep the cursor opaque to clients but strongly typed server-side — encode, then validate on decode.
  • Fetch limit + 1 to know whether a next page exists without a second count query.
Practice Exercises:
1Add a hasMore: boolean to CursorPage<T> and derive it from the limit + 1 fetch without a separate query.
2Write a generic paginate<T>(all: T[], params: { limit: number; offset: number }): OffsetPage<T> for in-memory arrays, clamping offset/limit.
3Extend the cursor to encode sort direction (asc/desc) and confirm decodeCursor rejects a tampered value.
Connections: The cursor encode/decode is another trust-boundary parse (Zod validate-on-read, like JWT verify and Redis reads on 07-01/07-02). The generic Page<T> envelope is generic interfaces, and clamping req.query is the same boundary validation used for request bodies.
2. File Uploads & Multipart Typing Node.js + TypeScript
multipart/form-data uploads sit outside JSON typing — the body is a stream of parts, and libraries like multer bolt req.file/req.files onto the request as loosely-typed extras. Typing the file shape, narrowing single/array/fields, and validating size and mime turns an untyped, security-sensitive path into a checked one.

multer parses multipart and attaches an Express.Multer.File (fields: originalname, mimetype, size, and buffer or path). @types/multer augments Request with file/files — but you still narrow: .single() gives one file, .array() gives an array, .fields() gives a keyed record.

A file is untrusted: enforce limits.fileSize, validate mimetype, and never derive a storage path from originalname (path traversal / overwrite). Return a curated result to clients, not the raw multer object.

1
Mount the multer middleware with limits; read the typed req.file.
2
Narrow req.files (array vs keyed record) before iterating.
3
Validate mime with a type guard; store under a server-generated id, not originalname.
TypeScript · example 1 of 4 — the typed file shape & a single-file handler
import multer from "multer";
const upload = multer({ limits: { fileSize: 5 * 1024 * 1024 } });   // 5 MB

app.post("/avatar", upload.single("avatar"), (req, res) => {
  const file = req.file;         // Express.Multer.File | undefined
  if (!file) return res.status(400).end();
  file.mimetype;                 // string
  file.buffer;                   // Buffer (memory storage)
});
@types/multer augments req.file to Express.Multer.File | undefined
TypeScript · example 2 of 4 — narrow the req.files union
// upload.array("photos")           → req.files is Express.Multer.File[]
// upload.fields([{ name: "cv" }])  → req.files is { [field]: File[] }
function getFiles(req: Request): Express.Multer.File[] {
  const f = req.files;
  if (Array.isArray(f)) return f;
  if (f) return Object.values(f).flat();
  return [];
}
req.files is a union — narrow array vs keyed record before use
TypeScript · example 3 of 4 — a mime type guard from a literal union
const ALLOWED = ["image/png", "image/jpeg"] as const;
type AllowedMime = typeof ALLOWED[number];

function isAllowed(
  f: Express.Multer.File,
): f is Express.Multer.File & { mimetype: AllowedMime } {
  return (ALLOWED as readonly string[]).includes(f.mimetype);
}
the predicate narrows mimetype to the allowed literal union
TypeScript · example 4 of 4 — return a curated result, not the raw file
interface UploadResult { id: string; url: string; bytes: number }

async function store(f: Express.Multer.File): Promise<UploadResult> {
  const id = crypto.randomUUID();
  await s3.put(`uploads/${id}`, f.buffer);   // never use f.originalname as the key
  return { id, url: `/files/${id}`, bytes: f.size };
}
clients get a stable UploadResult; originalname stays display-only metadata
Common Mistakes:
❌ Reading req.body.avatar for a file field — multipart file parts aren’t on req.body; they’re on req.file/req.files.
✅ Use the multer middleware and read the typed req.file; narrow single/array/fields explicitly.
❌ Using file.originalname as the storage path/key — it’s attacker-controlled (path traversal, overwrite).
✅ Generate your own id/key and keep originalname only as display metadata.
Key Rules:
  • Multipart file parts live on req.file/req.files (via @types/multer), not req.body.
  • req.files is a union — array for .array(), a keyed record for .fields(); narrow first.
  • Enforce limits.fileSize and validate mimetype — a file is untrusted input.
  • Never derive a filesystem/storage path from originalname.
Practice Exercises:
1Write a multer fileFilter that rejects non-images, typing its callback signature from @types/multer.
2Type a handler for upload.fields([...]) that pulls exactly one required cv and an optional cover, erroring if cv is missing.
3Return a discriminated { ok: true; result: UploadResult } | { ok: false; error: string } from store and handle both branches in the route.
Connections: req.file/req.files come from the same declaration-merging augmentation as req.user (07-02). The mime literal union + is guard is literal types + type predicates, and returning a curated UploadResult mirrors the “don’t leak internal shapes” lesson from Prisma select.
2Node.js + TS: Delivery, Webhooks & Errors
3. Webhooks: Typed Payloads & Signature Verification Node.js + TypeScript
Webhooks are inbound POSTs from third parties (Stripe, GitHub) that you must verify and route by event type. A discriminated union over the event type gives each handler a precisely-typed payload, and typing the signature-verification step keeps the security-critical “is this really from the provider?” check first-class.

A webhook body is untrusted until you verify its HMAC signature against the raw bytes with a shared secret. After verifying, parse into a discriminated union keyed on type, then switch for exhaustive, per-event typing (assertNever on the default).

Crucially, the signature is computed over the exact received bytes — so mount a raw body parser on the webhook route; re-serializing parsed JSON changes bytes and breaks the HMAC. Respond 2xx fast and offload heavy work, because providers retry on non-2xx or timeout.

1
Capture the raw request body (raw parser on the webhook route only).
2
Verify the HMAC with a constant-time compare before trusting anything.
3
Parse into a discriminated union and switch with assertNever.
TypeScript · example 1 of 4 — a discriminated event union
type WebhookEvent =
  | { type: "payment.succeeded";     data: { amount: number; currency: string } }
  | { type: "payment.failed";        data: { reason: string } }
  | { type: "subscription.canceled"; data: { userId: string } };
the `type` tag discriminates each event’s `data` shape
TypeScript · example 2 of 4 — verify the signature over the raw body
import crypto from "crypto";

function verify(rawBody: Buffer, signature: string, secret: string): boolean {
  const expected = crypto.createHmac("sha256", secret).update(rawBody).digest("hex");
  const a = Buffer.from(expected), b = Buffer.from(signature);
  return a.length === b.length && crypto.timingSafeEqual(a, b);   // constant-time
}
HMAC over exact bytes + timingSafeEqual — no early-exit timing leak
TypeScript · example 3 of 4 — raw-body route, then parse into the union
app.post("/webhook",
  express.raw({ type: "application/json" }),   // req.body is a Buffer here
  (req, res) => {
    const sig = req.header("x-signature") ?? "";
    if (!verify(req.body, sig, SECRET)) return res.status(401).end();
    const event: WebhookEvent = EventSchema.parse(JSON.parse(req.body.toString()));
    handle(event);
    res.sendStatus(200);
  });
verify raw bytes first, then schema-parse into a typed WebhookEvent
TypeScript · example 4 of 4 — exhaustive dispatch with assertNever
function handle(e: WebhookEvent) {
  switch (e.type) {
    case "payment.succeeded":     return credit(e.data.amount);   // data narrowed
    case "payment.failed":        return notify(e.data.reason);
    case "subscription.canceled": return revoke(e.data.userId);
    default: { const _x: never = e; throw new Error(`unhandled ${_x}`); }
  }
}
each case sees exactly its own `data`; a new event type breaks the switch
Common Mistakes:
❌ Verifying against the parsed/re-serialized JSON — re-stringifying changes bytes and the HMAC won’t match; also using === on the digests.
✅ Verify against the exact raw bytes with crypto.timingSafeEqual, before parsing.
❌ Typing the body as any and branching on body.type — no exhaustiveness, and a new event silently falls through.
✅ Parse into a discriminated union and switch with an assertNever default so new events fail loudly.
Key Rules:
  • Verify the HMAC over the raw body (mount a raw parser) with a constant-time compare — before trusting anything.
  • Model events as a discriminated union on type; switch + assertNever gives exhaustiveness.
  • Return 2xx quickly and do heavy work async — providers retry on non-2xx/timeouts.
  • Even after the signature check, validate the payload with a schema — it’s still external input.
Practice Exercises:
1Add a charge.refunded event and watch the assertNever default force you to handle it.
2Add replay protection: include a timestamp in the signed payload and reject events older than 5 minutes.
3Make handle return a Promise<void>, respond 200 immediately, and process on a queue (tie back to BullMQ from 07-01).
Connections: The union + assertNever is discriminated unions in practice + never / exhaustiveness. Signature verification is a trust boundary like JWT verify (07-02), and the post-verify schema parse is the same validate-untrusted-input rule. Offloading heavy work echoes job queues (07-01).
4. Server-Sent Events (SSE) Typing Node.js + TypeScript
SSE is a lightweight, one-way server→client stream over plain HTTP (text/event-stream) — ideal for progress bars, notifications, and live feeds. Because the wire format is just formatted text, a typed send(event, data) wrapper over a discriminated event map keeps every message on-contract on both ends.

The server keeps the response open and writes event:/data: lines separated by a blank line; the browser’s EventSource fires the matching named events. Model a ServerEvents map (event name → payload) and make send<K extends keyof E>(name, data) serialize JSON — the same map documents what the client’s addEventListener handlers receive.

SSE auto-reconnects and supports an id: line so the browser resends Last-Event-ID on reconnect for resumable streams. Always tear down timers/subscriptions on req.on("close") or you leak resources per disconnected client.

1
Define a ServerEvents map: event name → payload type.
2
Write a send<K extends keyof E> wrapper that frames event:/data: + blank line.
3
Clean up on req.on("close"); drive client listeners off the same map.
TypeScript · example 1 of 4 — event map + a typed emitter over the response
interface ServerEvents {
  progress: { pct: number };
  message:  { text: string };
  done:     { ok: boolean };
}

function sseEmitter<E>(res: Response) {
  res.writeHead(200, {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache",
    Connection: "keep-alive",
  });
  return {
    send<K extends keyof E & string>(event: K, data: E[K]) {
      res.write(`event: ${event}\n`);
      res.write(`data: ${JSON.stringify(data)}\n\n`);
    },
    close() { res.end(); },
  };
}
send only accepts a known event name and its matching payload E[K]
TypeScript · example 2 of 4 — a typed streaming handler
app.get("/jobs/:id/stream", (req, res) => {
  const sse = sseEmitter<ServerEvents>(res);
  const timer = setInterval(() => sse.send("progress", { pct: next() }), 1000);
  // sse.send("progres", { pct: 1 });   // error: not a key of ServerEvents
  req.on("close", () => clearInterval(timer));
});
a mistyped event name is a compile error; the interval is cleaned up on close
TypeScript · example 3 of 4 — the client, typed to the same map
const es = new EventSource("/jobs/42/stream");

es.addEventListener("progress", (e) => {
  const data = JSON.parse((e as MessageEvent).data) as ServerEvents["progress"];
  bar.value = data.pct;   // data: { pct: number }
});
indexing ServerEvents["progress"] keeps client and server payloads in sync
TypeScript · example 4 of 4 — an id line for resumable streams
send<K extends keyof E & string>(event: K, data: E[K], id?: string) {
  if (id) res.write(`id: ${id}\n`);
  res.write(`event: ${event}\n`);
  res.write(`data: ${JSON.stringify(data)}\n\n`);
}
// on reconnect the browser resends the Last-Event-ID header
an id: line lets the client resume from Last-Event-ID after a drop
Common Mistakes:
❌ Writing res.write(JSON.stringify(payload)) with no event:/data: framing or trailing blank line — the client never fires and the message is malformed.
✅ Use a send() wrapper that emits event:, data:, and the \n\n terminator every time.
❌ Forgetting req.on("close") cleanup — intervals/subscriptions leak for every disconnected client.
✅ Tear down timers and listeners when the connection closes.
Key Rules:
  • SSE is one-way server→client over text/event-stream; reach for WebSockets only when you need bidirectional.
  • Drive both send and the client’s listeners off one ServerEvents map so names/payloads stay in sync.
  • Always emit the data: line + blank-line terminator; add id: to support Last-Event-ID resume.
  • Clean up on req.on("close") to avoid leaking per-connection resources.
Practice Exercises:
1Add an error: { code: string } event and a client handler; confirm send("eror", …) won’t compile.
2Write a heartbeat that sends a comment line (:\n\n) every 15s to keep proxies from closing idle connections.
3Type a broadcast(event, data) over a Set<Emitter> of connected clients, removing closed ones.
Connections: The event-map-keyed send is the type-safe event emitter pattern (07-02 review) applied to the wire, and the one-way contrast with WebSocket typing. JSON framing over a stream ties to Node streams (06-30), and per-event payloads are generic indexed access (E[K]).
5. Typed Error Responses (RFC 7807 problem+json) Node.js + TypeScript
APIs need a consistent, machine-readable error shape. RFC 7807 application/problem+json standardizes it (type, title, status, detail, instance), and typing that envelope — plus a discriminated union of your domain error kinds — means every handler returns errors the same way and clients can switch on a stable type.

Define a base Problem interface matching 7807, then extend it per error kind with extra fields (a validation problem carries an errors[] array). A discriminated union on the type slug lets a single, central error middleware map any thrown domain error to the right HTTP status and body — one typed place instead of ad-hoc shapes per route.

Keep detail client-safe: put a stable title/type on the wire and log the raw error (and any stack) server-side only. A satisfies Problem on the fallback keeps even the 500 branch honest.

1
Define the base Problem (7807 fields) and extend it per error kind.
2
Union the kinds on type; carry them on a typed HttpError.
3
Map thrown errors → status + problem+json in one central middleware.
TypeScript · example 1 of 4 — the 7807 base + a typed extension
interface Problem {
  type: string;       // a URI/slug identifying the error kind
  title: string;
  status: number;
  detail?: string;
  instance?: string;  // the specific request/resource
}

interface ValidationProblem extends Problem {
  type: "https://errors/validation";
  errors: { field: string; message: string }[];
}
a shared base every error response conforms to
TypeScript · example 2 of 4 — a discriminated union of domain errors
type ApiProblem =
  | ValidationProblem
  | (Problem & { type: "https://errors/not-found" })
  | (Problem & { type: "https://errors/unauthorized" });
the `type` slug discriminates which extra fields are present

class HttpError extends Error {
  constructor(readonly problem: ApiProblem) { super(problem.title); }

TypeScript · example 4 of 4 — one central, typed error middleware
app.use((err: unknown, req: Request, res: Response, next: NextFunction) => {
  if (err instanceof HttpError) {
    return res
      .status(err.problem.status)
      .type("application/problem+json")
      .json(err.problem);
  }
  res.status(500).type("application/problem+json").json({
    type: "about:blank", title: "Internal Server Error", status: 500,
  } satisfies Problem);
});
every error path emits problem+json; satisfies keeps the fallback honest
Common Mistakes:
❌ Returning ad-hoc { error: "..." } shapes that differ per route — clients can’t handle errors uniformly and status/body drift.
✅ Return a single typed Problem/problem+json envelope everywhere via one error middleware.
❌ Leaking stack traces or internal messages into detail — it exposes internals and varies by environment.
✅ Put a safe, stable title/type on the wire; log the raw error server-side only.
Key Rules:
  • Adopt one error envelope (RFC 7807 problem+json) and type it as a base Problem interface.
  • Model error kinds as a discriminated union on type so extra fields (e.g. validation errors) are checked.
  • Centralize the throw → status + body mapping in one typed middleware.
  • Keep detail client-safe; never serialize stack traces or internal identifiers.
Practice Exercises:
1Add a RateLimitProblem with retryAfter: number and set the Retry-After header from it in the middleware.
2Write a fromZodError(e): ValidationProblem that maps a Zod error’s issues into the errors[] array.
3Add an expectTypeOf test asserting every member of ApiProblem is assignable to Problem.
Connections: The union on type is discriminated unions + never/exhaustiveness applied to errors, and the fallback’s satisfies Problem is the satisfies operator. This is the response-side complement to request validation (Concept 1’s query parsing), and the HTTP-boundary serialization of the Result/error-handling patterns from earlier weeks.
