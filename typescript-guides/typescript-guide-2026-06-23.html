R1. tsconfig.json Deep Dive Modules & Project Structure 9 days overdue · rep 2
strict enables every strict-family check at once. paths needs baseUrl to resolve aliases. For modern bundler-based projects, prefer moduleResolution: "bundler"; for Node libraries use NodeNext.
Key Pattern 
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],
      "@/components/*": ["./src/components/*"]
    }
  }
}
Challenge Exercise

Configure a tsconfig.json that enables strict mode, defines a path alias for @/components/*, targets ES2022, and uses Node16 module resolution. Verify that import { Btn } from "@/components/Btn" resolves.

Pitfall: Setting paths without baseUrl. Path mappings are resolved relative to baseUrl — omit it and TypeScript has no anchor, so every alias fails to resolve.
R2. Module Systems Modules & Project Structure 9 days overdue · rep 2
ESM uses import/export; CJS uses require/module.exports. With "module": "NodeNext", TypeScript respects file extensions and package.json "type" to decide ESM vs CJS per file.
Key Pattern 
// Type-only re-export keeps types out of the runtime graph
export { type MyType } from "./types.js";

// Dynamic import returns a Promise of the module namespace
const { parse } = await import("./parser.js");

// In NodeNext, relative ESM imports MUST carry the .js extension
import { helper } from "./util.js";   // resolves util.ts at build
Challenge Exercise

Convert a CJS module that uses const x = require(path) dynamically into ESM with await import(path), preserving type safety on the imported namespace. Keep the function’s return type fully typed.

Pitfall: Omitting the .js extension in ESM imports under NodeNext. Even though the source file is .ts, the import specifier must reference the emitted .js path or Node throws ERR_MODULE_NOT_FOUND at runtime.
R3. Decorators (TC39) Functions & Patterns 9 days overdue · rep 2
TC39 Stage 3 decorators receive a context object as the second parameter. There are class, method, field, getter/setter, and accessor decorators. The decorator’s return value replaces the decorated element.
Key Pattern 
function log(
  target: Function,
  context: ClassMethodDecoratorContext
) {
  return function (this: unknown, ...args: unknown[]) {
    console.log(`calling ${String(context.name)}`);
    return target.call(this, ...args);
  };
}

class Service {
  @log
  fetch(id: number) { return `item ${id}`; }
}
calling fetch
Challenge Exercise

Write a @throttle(ms: number) decorator factory that limits how often a method may run. Calls inside the window should be ignored (or return the cached last result). Use the ClassMethodDecoratorContext and return a replacement function.

Pitfall: Confusing legacy experimentalDecorators (the old (target, key, descriptor) signature) with TC39 Stage 3 decorators (the (value, context) signature). The two are mutually exclusive — enabling experimentalDecorators opts you out of the standard ones.
R4. Declaration Files Modules & Project Structure 9 days overdue · rep 2
.d.ts files provide types for plain JS code with no runtime output. Use declare module for untyped packages, and pull community types from @types/* on DefinitelyTyped.
Key Pattern 
// my-lib.d.ts — types for an untyped runtime package
declare module "my-lib" {
  export interface Data { id: number; label: string }
  export function parse(s: string): Data;
}
Challenge Exercise

Write a .d.ts file for an existing JS utility that exports slugify(s: string): string, truncate(s: string, len: number): string, and a Config interface. Verify your editor gives autocomplete on the imported functions.

Pitfall: Declaring a module shape that conflicts with the library’s actual runtime exports (e.g. typing a default export when the package only has named exports). The compiler trusts your declaration, so the mismatch only blows up at runtime.
R5. Namespaces vs Modules Modules & Project Structure 7 days overdue · rep 2
Prefer ES modules over namespaces in modern code. Namespaces are still useful for two things: declaration merging (augmenting third-party types) and organizing types in global script contexts.
Key Pattern 
// Augment a third-party global type via declaration merging
declare namespace Express {
  interface Request {
    user?: { id: string; role: string };
  }
}
Challenge Exercise

Augment the Express Request type to add a custom session property of type { token: string; expiresAt: number } using declaration merging. Confirm req.session.token type-checks inside a route handler.

Pitfall: Using namespaces inside ES module files to “group” exports. In a module, a namespace compiles to an unnecessary wrapper object (an IIFE) and defeats tree-shaking. Just use named exports.
R6. Zod & Runtime Validation Runtime Patterns 6 days overdue · rep 2
A Zod schema is both a runtime validator and the single source of truth for a TypeScript type. Extract the static type with z.infer<typeof schema> so the type can never drift from the validation.
Key Pattern 
import { z } from "zod";

const UserSchema = z.object({
  name: z.string(),
  age: z.number().min(0),
});

type User = z.infer<typeof UserSchema>;
// { name: string; age: number }
{ name: string; age: number }
Challenge Exercise

Create a Zod schema for an API response modeled as a discriminated union: { status: "ok", data: T } | { status: "error", message: string }. Use z.discriminatedUnion("status", [...]) and infer the resulting TS type.

Pitfall: Forgetting that .parse() throws on invalid input. For graceful error handling without try/catch, use .safeParse(), which returns { success: true, data } or { success: false, error }.
1Database Typing
1. Prisma — Type-Safe Database Access Database Typing
Prisma generates TypeScript types directly from your database schema, so every query is fully type-checked at compile time with autocomplete on models, relations, and filters — no hand-written DTOs that drift out of sync.

The Prisma workflow has three stages: you write a declarative schema.prisma file, run prisma generate to produce a typed client, then import and use that client. Every model becomes a TypeScript type, and every method (findMany, create, update, delete) is typed against it.

Because the client is generated, the types reflect your exact schema — including relations, optional fields, and enums. When you change the schema and re-generate, mismatched queries become compile errors.

1
Define models and relations in schema.prisma.
2
Run prisma generate to emit the typed @prisma/client.
3
Import PrismaClient and write queries — results are typed automatically.
Prisma · example 1 of 4 — Schema definition
// schema.prisma
model User {
  id    Int     @id @default(autoincrement())
  email String  @unique
  name  String?
  posts Post[]
}

model Post {
  id       Int     @id @default(autoincrement())
  title    String
  likes    Int     @default(0)
  author   User    @relation(fields: [authorId], references: [id])
  authorId Int
  comments Comment[]
}

model Comment {
  id     Int    @id @default(autoincrement())
  body   String
  post   Post   @relation(fields: [postId], references: [id])
  postId Int
}
TypeScript · example 2 of 4 — Basic CRUD with inferred types
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

// create() returns a fully-typed User
const user = await prisma.user.create({
  data: { email: "ada@example.com", name: "Ada" },
});
// user: { id: number; email: string; name: string | null }

// findMany() returns User[] — name is string | null (it is optional)
const users = await prisma.user.findMany();

// update() is typed against the model; unknown fields are errors
const updated = await prisma.user.update({
  where: { id: user.id },
  data: { name: "Ada Lovelace" },
});
user: { id: number; email: string; name: string | null }
TypeScript · example 3 of 4 — Relations with include + filtered where
// include pulls in the relation AND widens the result type to match
const usersWithPosts = await prisma.user.findMany({
  where: {
    posts: { some: { likes: { gte: 5 } } },  // users with a 5+ like post
  },
  include: {
    posts: {
      where: { likes: { gte: 5 } },
      include: { comments: true },
    },
  },
});

// Type now includes posts, and each post includes comments:
// (User & { posts: (Post & { comments: Comment[] })[] })[]
for (const u of usersWithPosts) {
  for (const p of u.posts) {
    console.log(p.title, p.comments.length); // both typed
  }
}
(User & { posts: (Post & { comments: Comment[] })[] })[]
TypeScript · example 4 of 4 — Type-safe transactions
// $transaction runs all operations atomically; the tuple is typed
const [newUser, newPost] = await prisma.$transaction([
  prisma.user.create({ data: { email: "grace@example.com" } }),
  prisma.post.create({
    data: { title: "Hello", author: { connect: { id: 1 } } },
  }),
]);
// newUser: User, newPost: Post — positionally typed from the array

// Interactive transactions get a scoped, typed client
await prisma.$transaction(async (tx) => {
  const u = await tx.user.findUniqueOrThrow({ where: { id: 1 } });
  await tx.post.create({
    data: { title: "Tx post", authorId: u.id },
  });
});
Common Mistakes:
❌ Casting Prisma results to any to “simplify” dynamic include shapes: const u = result as any. You throw away every guarantee the generated client gave you.
✅ Use the generated payload helper for dynamic shapes: type UserWithPosts = Prisma.UserGetPayload<{ include: { posts: true } }>. It reconstructs the exact type for that include.
❌ Forgetting to re-run prisma generate after editing the schema — the client types go stale and queries silently reference old fields.
✅ Wire prisma generate into a postinstall script and your build step so the client is always regenerated.
Key Rules:
  • 2026-07-04 (Day 31)
  • The schema is the source of truth; the client is generated from it. Never hand-edit generated types.
  • include widens the result type to include relations; select narrows it to only chosen fields.
  • For dynamic or reusable query shapes, derive the type with Prisma.ModelGetPayload<...> instead of any.
  • $transaction with an array is positionally typed; with a callback you get a scoped, typed tx client.
Practice Exercises:
1Define a Prisma schema for a blog with User, Post, and Comment models. User has many Posts; Post has many Comments; give Post a likes field defaulting to 0.
2Write a typed query for “get all posts by a given user with 5 or more likes, including their comments.” Hover over the result and confirm comments is typed as Comment[].
3Define a reusable type alias PostWithComments using Prisma.PostGetPayload<{ include: { comments: true } }> and use it as the return type of a helper function.
Connections: Prisma’s GetPayload helpers are conditional/mapped types under the hood — the same machinery as Mapped Types and Branded Types (Concept 4 below). Pairing Prisma reads with Zod (R6) at the API boundary gives you validation on the way in and type safety on the way out.
2. Drizzle ORM — SQL-First Type Safety Database Typing
Drizzle gives you SQL-level control with full TypeScript inference and no code generation step. You define the schema in TypeScript, and types flow through every query automatically — what you write reads like SQL, but the compiler checks it.

Where Prisma generates a client from a separate schema file, Drizzle defines tables directly in TypeScript using helpers like pgTable. The table object is the schema, and Drizzle derives query result types from it on the fly.

Two inference helpers do the heavy lifting: InferSelectModel<typeof table> gives you the row type returned by a SELECT, and InferInsertModel<typeof table> gives you the shape required by an INSERT (which differs — auto-generated columns become optional).

TypeScript · example 1 of 4 — Schema definition with pgTable
import { pgTable, serial, text, integer, timestamp } from "drizzle-orm/pg-core";

export const products = pgTable("products", {
  id: serial("id").primaryKey(),
  name: text("name").notNull(),
  priceCents: integer("price_cents").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});

export const orders = pgTable("orders", {
  id: serial("id").primaryKey(),
  userId: integer("user_id").notNull(),
  total: integer("total").notNull(),
});

export const orderItems = pgTable("order_items", {
  id: serial("id").primaryKey(),
  orderId: integer("order_id").notNull(),
  productId: integer("product_id").notNull(),
  quantity: integer("quantity").notNull(),
});
TypeScript · example 2 of 4 — Inferring types from the schema
import { InferSelectModel, InferInsertModel } from "drizzle-orm";
import { products } from "./schema";

// Row type as returned by a SELECT
type Product = InferSelectModel<typeof products>;
// { id: number; name: string; priceCents: number; createdAt: Date }

// Shape required to INSERT — id and createdAt are optional (defaulted)
type NewProduct = InferInsertModel<typeof products>;
// { id?: number; name: string; priceCents: number; createdAt?: Date }

const draft: NewProduct = { name: "Mug", priceCents: 1200 }; // ok
Product: { id: number; name: string; priceCents: number; createdAt: Date } NewProduct: { id?: number; name: string; priceCents: number; createdAt?: Date }
TypeScript · example 3 of 4 — Select with a join
import { eq } from "drizzle-orm";
import { db } from "./db";
import { orders, orderItems, products } from "./schema";

// Join orders -> order_items -> products; result columns are inferred
const rows = await db
  .select({
    orderId: orders.id,
    productName: products.name,
    quantity: orderItems.quantity,
    lineTotal: products.priceCents,
  })
  .from(orders)
  .innerJoin(orderItems, eq(orderItems.orderId, orders.id))
  .innerJoin(products, eq(products.id, orderItems.productId));

// rows: { orderId: number; productName: string;
//         quantity: number; lineTotal: number }[]
for (const r of rows) console.log(r.productName, r.quantity);
{ orderId: number; productName: string; quantity: number; lineTotal: number }[]
TypeScript · example 4 of 4 — Conditional queries with where()
import { and, gte, eq, SQL } from "drizzle-orm";
import { db } from "./db";
import { products } from "./schema";

interface ProductFilter {
  minPrice?: number;
  exactName?: string;
}

async function findProducts(filter: ProductFilter) {
  const conditions: SQL[] = [];
  if (filter.minPrice !== undefined)
    conditions.push(gte(products.priceCents, filter.minPrice));
  if (filter.exactName !== undefined)
    conditions.push(eq(products.name, filter.exactName));

  return db
    .select()
    .from(products)
    .where(conditions.length ? and(...conditions) : undefined);
}
// return type inferred as Product[] from the schema
Common Mistakes:
❌ Re-declaring row types by hand: interface Product { id: number; name: string; /* ... */ }. This duplicates the schema and silently drifts when you add a column.
✅ Derive every row/insert type from the table: type Product = InferSelectModel<typeof products>. One source of truth, zero drift.
❌ Using InferSelectModel for inserts — it marks defaulted columns as required, so db.insert() rejects valid drafts.
✅ Use InferInsertModel for insert payloads; it makes auto-generated and defaulted columns optional.
Key Rules:
  • The TypeScript table object is the schema — there is no separate generation step.
  • InferSelectModel for read shapes, InferInsertModel for write shapes; they differ on defaulted columns.
  • Query result types are inferred from the .select({...}) projection, so select exactly the columns you need.
  • Build dynamic WHERE clauses by collecting SQL conditions and combining with and(...)/or(...).
Practice Exercises:
1Define a Drizzle schema for an e-commerce system with products, orders, and order_items tables. Give order_items foreign-key columns to both orders and products.
2Write a type-safe query joining orders with products through order_items, projecting order id, product name, and quantity. Confirm the inferred row type.
3Derive NewOrder = InferInsertModel<typeof orders> and write a typed createOrder(draft: NewOrder) helper. Verify the compiler rejects a draft missing total.
Connections: Drizzle’s InferSelectModel/InferInsertModel are inference helpers built on the same conditional-type foundation as Prisma’s GetPayload (Concept 1). The dynamic where() builder previews the Builder Pattern with Type Narrowing (Concept 5).
3. Type-Safe SQL Builders Database Typing
Sometimes you need raw SQL but still want safety from injection and type mismatches. Tagged-template libraries like sql-template-tag and codegen tools like @pgtyped/query bridge the gap between “just write SQL” and “keep the types.”

There are two complementary approaches. Tagged template SQL (sql-template-tag) lets you write SQL with interpolated values that are automatically turned into bound parameters — never string-concatenated — which kills injection at the source.

Codegen from SQL files (@pgtyped/query) reads your .sql files, runs them against a real database to learn their input and output column types, and generates typed query functions. You write plain SQL; the types come from the database itself.

TypeScript · example 1 of 3 — Tagged template SQL (parameterized)
import sql from "sql-template-tag";

const userId = 42;
const minLikes = 5;

// Interpolated values become bound parameters, not string literals
const query = sql`
  SELECT id, title, likes
  FROM posts
  WHERE author_id = ${userId}
    AND likes >= ${minLikes}
`;

console.log(query.text);
// "SELECT id, title, likes FROM posts
//  WHERE author_id = $1 AND likes >= $2"
console.log(query.values); // [42, 5]

// Pass to your driver: await pool.query(query.text, query.values)
text: SELECT ... WHERE author_id = $1 AND likes >= $2 values: [42, 5]
TypeScript · example 2 of 3 — A typed parameterized helper
import sql from "sql-template-tag";
import { Pool } from "pg";

const pool = new Pool();

// Typed params in, typed rows out
interface PostRow { id: number; title: string; likes: number }

async function postsByAuthor(
  authorId: number,
  minLikes: number
): Promise<PostRow[]> {
  const q = sql`
    SELECT id, title, likes FROM posts
    WHERE author_id = ${authorId} AND likes >= ${minLikes}
  `;
  const result = await pool.query<PostRow>(q.text, q.values);
  return result.rows;
}
SQL + TypeScript · example 3 of 3 — @pgtyped codegen flow
-- queries/posts.sql
/* @name FindPostsByAuthor */
SELECT id, title, likes
FROM posts
WHERE author_id = :authorId AND likes >= :minLikes;
// After running `pgtyped`, it generates posts.queries.ts with
// fully-typed params and results derived from the live DB schema:
import { FindPostsByAuthor } from "./posts.queries";

const rows = await FindPostsByAuthor.run(
  { authorId: 42, minLikes: 5 },  // params: typed
  client
);
// rows: { id: number; title: string; likes: number }[]
//        — inferred from the actual column types
rows: { id: number; title: string; likes: number }[]
Common Mistakes:
❌ String interpolation directly into SQL: `SELECT * FROM users WHERE id = ${id}`. This is a textbook SQL-injection hole — the value is concatenated into the query string.
✅ Use parameterized queries with typed placeholders: sql`... WHERE id = ${id}` emits $1 + a bound value, or use :named params with pgtyped.
❌ Trusting hand-written row interfaces to match the query forever — rename a column in SQL and the interface lies.
✅ Let @pgtyped derive result types from the real database so a column rename is a compile error, not a runtime surprise.
Key Rules:
  • Never interpolate untrusted (or any) values into a SQL string — always go through bound parameters.
  • sql-template-tag gives you safe parameterization; you still annotate result types yourself.
  • @pgtyped goes further: it infers both param and result types from the live schema via codegen.
  • Tagged templates are great for one-off dynamic queries; codegen shines for a stable set of named queries.
Practice Exercises:
1Create a type-safe query builder function that accepts a typed filter object { minLikes?: number; authorId?: number } and assembles a parameterized sql query, returning a typed PostRow[].

Connections: The injection-safety principle here is the same boundary-validation idea behind Zod (R6) and the CLI/env patterns from Day 19 — never trust raw strings. The generic findById<T> exercise leans on the Generic Constraints reviewed earlier in this series.
2Advanced TypeScript Patterns
4. Branded Types for Domain Modeling Advanced Type Patterns
TypeScript is structurally typed: a UserId and a PostId that are both string are interchangeable, so you can accidentally pass one where the other is expected. Branded types add nominal typing to catch these swaps at compile time.

The trick is to intersect a primitive with a phantom brand property — a field that exists only in the type system and is never present at runtime. Two brands that look identical (string) become incompatible because their brand symbols differ.

You pair the branded type with a constructor function (often called a smart constructor) that validates the value and returns it “cast” to the branded type. This is the only place the cast happens, so validation can’t be skipped.

TypeScript · example 1 of 4 — A basic brand with intersection
// A unique symbol makes the brand impossible to forge structurally
type Brand<T, B> = T & { readonly __brand: B };

type UserId = Brand<string, "UserId">;
type PostId = Brand<string, "PostId">;

declare function getUser(id: UserId): void;

const raw = "u_123";
getUser(raw);             // ❌ Error: string is not assignable to UserId
getUser(raw as UserId);   // ok (but unchecked — prefer a constructor)
Argument of type 'string' is not assignable to parameter of type 'UserId'.
TypeScript · example 2 of 4 — Constructor functions that validate + brand
type UserId = string & { readonly __brand: unique symbol };

// Smart constructor: validation is the only path to a UserId
function toUserId(value: string): UserId {
  if (!/^u_[a-z0-9]+$/.test(value)) {
    throw new Error(`Invalid UserId: ${value}`);
  }
  return value as UserId; // the single sanctioned cast
}

const id = toUserId("u_123"); // UserId
// const bad = toUserId("nope"); // throws at runtime

declare function loadProfile(id: UserId): void;
loadProfile(id);              // ok
loadProfile("u_123");         // ❌ raw string rejected at compile time
Argument of type 'string' is not assignable to parameter of type 'UserId'.
TypeScript · example 3 of 4 — Branded types across an API layer
type Email = string & { readonly __brand: "Email" };
type UserId = string & { readonly __brand: "UserId" };

function toEmail(s: string): Email {
  if (!s.includes("@")) throw new Error("bad email");
  return s.toLowerCase() as Email;
}

interface CreateUser {
  email: Email;     // can't be confused with name or any other string
  invitedBy?: UserId;
}

// The brand documents intent AND enforces it: the only way to fill
// `email` is to run it through toEmail(), so it's always validated.
function createUser(input: CreateUser) { /* ... */ }
createUser({ email: toEmail("ADA@EXAMPLE.COM") }); // stored lowercased
TypeScript · example 4 of 4 — Combining brands with Zod
import { z } from "zod";

type Email = string & { readonly __brand: "Email" };

// Zod validates at runtime AND brands the inferred type
const EmailSchema = z
  .string()
  .email()
  .transform((s) => s.toLowerCase() as Email);

type ValidatedEmail = z.infer<typeof EmailSchema>; // Email

const result = EmailSchema.safeParse("ADA@example.com");
if (result.success) {
  const email: Email = result.data; // branded + validated
}
ValidatedEmail = Email (string & { __brand: "Email" })
Common Mistakes:
❌ Using plain string for IDs that must never mix: function transfer(from: string, to: string) — nothing stops you swapping the arguments.
✅ Brand them: type AccountId = string & { readonly __brand: unique symbol }, so transfer(from: AccountId, to: AccountId) rejects raw strings and mismatched brands.
❌ Scattering value as Branded casts throughout the codebase — that defeats the purpose, since the brand no longer implies validation.
✅ Funnel every branded value through one smart constructor; the cast lives there and nowhere else.
Key Rules:
  • A brand is a phantom property (__brand) that exists only at the type level — zero runtime cost.
  • Use unique symbol or a unique string literal so two brands can never be structurally equal.
  • Always create branded values through a validating constructor; that constructor holds the only sanctioned cast.
  • Brands compose with Zod: validate at runtime and brand the result so “validated” is encoded in the type.
Practice Exercises:
1Create branded Email, Password, and HashedPassword types, each with a constructor function. Make it a compile error to pass a Password where a HashedPassword is required.
2Build a function linkPost(user: UserId, post: PostId) using branded IDs. Confirm the compiler rejects both a raw string and a swapped linkPost(postId, userId) call.
3Write a Zod schema that validates and brands a NonEmptyString. Infer its type and use it as the type of a required title field.
Connections: Branded types are the compile-time twin of Zod runtime validation (R6) — combine them to encode “validated” into the type itself. The intersection-with-phantom-field trick is the same shape used by Prisma/Drizzle’s internal type tagging (Concepts 1–2).
5. Builder Pattern with Type Narrowing Advanced Type Patterns
A builder that tracks what has been set at the type level can make “you forgot a required field” and “you called .build() too early” into compile errors instead of runtime bugs. The type narrows with each .set() call.

A naive builder returns this from every method and exposes .build() unconditionally — so nothing stops you from building an incomplete object. The type-safe version threads a generic type parameter that records which fields have been set; .build() is only callable once all required fields are present.

The mechanism: each setter returns a new builder type with the just-set key added to the “set” union. A conditional type then gates .build() — if any required key is still missing, .build() resolves to a function the compiler refuses to call.

TypeScript · example 1 of 3 — Naive builder (returns this)
// Works, but .build() is always available — no compile-time guard
class RequestBuilder {
  private url?: string;
  private method = "GET";

  setUrl(u: string): this { this.url = u; return this; }
  setMethod(m: string): this { this.method = m; return this; }

  build() {
    if (!this.url) throw new Error("url is required"); // runtime only
    return { url: this.url, method: this.method };
  }
}

// Compiles fine, throws at runtime — the bug escaped the type system
new RequestBuilder().setMethod("POST").build();
TypeScript · example 2 of 3 — Step builder tracking set fields
// Track which keys have been set in a generic union
class QueryBuilder<Set extends string = never> {
  private parts: Record<string, string> = {};

  select(cols: string): QueryBuilder<Set | "select"> {
    this.parts.select = cols;
    return this as QueryBuilder<Set | "select">;
  }

  from(table: string): QueryBuilder<Set | "from"> {
    this.parts.from = table;
    return this as QueryBuilder<Set | "from">;
  }

  where(cond: string): QueryBuilder<Set | "where"> {
    this.parts.where = cond;
    return this as QueryBuilder<Set | "where">;
  }

  // .build() requires both "select" and "from" in the Set union
  build(this: QueryBuilder<"select" | "from" | Set>): string {
    const w = this.parts.where ? ` WHERE ${this.parts.where}` : "";
    return `SELECT ${this.parts.select} FROM ${this.parts.from}${w}`;
  }
}
TypeScript · example 3 of 3 — The guard in action
// ✅ All required steps present — build() is callable
const ok = new QueryBuilder()
  .select("id, name")
  .from("users")
  .where("active = true")
  .build();
// "SELECT id, name FROM users WHERE active = true"

// ❌ Missing .from() — build() is not assignable
const bad = new QueryBuilder()
  .select("id")
  .build();
//        ~~~~~ The 'this' context is not assignable:
//        'from' is missing from the Set union
The 'this' context of type 'QueryBuilder<"select">' is not assignable to method's 'this' of type 'QueryBuilder<"select" | "from">'.
Common Mistakes:
❌ A builder whose setters return any or the same type regardless of state — the compiler can’t tell a complete builder from an empty one.
✅ Narrow the generic parameter on each call (QueryBuilder<Set | "select">) so the “set” union grows and gates .build().
❌ Guarding required fields only with a runtime if (!this.url) throw — the mistake still ships and only fails when that path runs.
✅ Encode the requirement in build(this: ...)’s this type so an incomplete build is a compile error you can’t ignore.
Key Rules:
  • Carry a generic type parameter that accumulates the set of fields configured so far.
  • Each setter returns the builder with its key added to that union (Set | "key").
  • Gate .build() with an explicit this parameter that requires all mandatory keys in the union.
  • Optional steps (.where(), .orderBy()) widen the union but are never required by .build().
Practice Exercises:
1Create a QueryBuilder where .select() and .from() are required before .build() compiles. Verify that calling .build() after only .select() is a type error.
2Extend the builder with optional .where() and .orderBy() steps. Confirm they can be omitted, can appear in any order, and don’t affect whether .build() is callable.
3Build an HttpRequestBuilder where .url() is required but .method(), .header(), and .body() are optional. Make .body() only available after .method("POST") (hint: narrow the method in the return type).
Connections: The type-state tracking here is the same generic-narrowing technique behind Drizzle’s chained query API (Concept 2) and the this-returning fluent interfaces reviewed in earlier guides. Pair it with Branded Types (Concept 4) to make a builder that only accepts validated, branded inputs.
