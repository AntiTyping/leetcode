R1. Decorators (TC39) Functions & Patterns 15 days since review · rep 3 → 4
The Stage-3 (TC39) decorators shipped in TS 5.0 are not the old experimentalDecorators ones. A decorator is a function (value, context) => newValueOrVoid; the context carries kind, name, addInitializer, and typed access helpers. You return a replacement (or nothing) — no reflect-metadata, no parameter decorators.
Key Pattern 
function logged<This, Args extends unknown[], R>(
  target: (this: This, ...args: Args) => R,
  ctx: ClassMethodDecoratorContext<This, (this: This, ...args: Args) => R>,
) {
  const name = String(ctx.name);
  return function (this: This, ...args: Args): R {
    console.log(`→ ${name}(${args.join(",")})`);
    return target.call(this, ...args);
  };
}

class Calc {
  @logged add(a: number, b: number) { return a + b; }
}
new Calc().add(2, 3);   // logs "→ add(2,3)" then returns 5
the replacement method keeps the original this/args/return types via generics
Challenge Exercise

Write a @bound method decorator that uses ctx.addInitializer to bind the method to the instance, and a @memoize that caches by first argument. Then explain why the new decorators can’t decorate constructor parameters, and how accessor fields interact with ClassAccessorDecoratorContext.

Pitfall: Assuming emitDecoratorMetadata / reflect-metadata works with Stage-3 decorators — it does not. That reflection-based DI style belongs to the legacy experimentalDecorators mode; the two systems are mutually exclusive and shouldn’t be mixed in one project.
R2. Module Systems Modules & Project Structure 15 days since review · rep 3 → 4
ESM and CommonJS differ in resolution, timing, and interop. Under "module": "nodenext", the nearest package.json "type" field decides how each .ts compiles, relative ESM imports need explicit extensions, and import type keeps types from leaking into emitted JS. verbatimModuleSyntax makes what you write match what is emitted.
Key Pattern 
// package.json: { "type": "module" }
// under nodenext, relative ESM imports carry the .js extension
import { parse } from "./parser.js";   // resolves parser.ts at build
import type { Config } from "./config.js";  // type-only: erased from emit

export const VERSION = "1.0.0";

// verbatimModuleSyntax on => `import type` MUST be used for types,
// and `import { x }` is emitted verbatim (no elision surprises)
nodenext ties module kind to package.json "type"; type-only imports are erased
Challenge Exercise

Set up a package that ships both ESM and CJS via the "exports" map with import/require/types conditions. Then explain why a default CJS import (import fs from "fs") needs esModuleInterop, and what __esModule markers do at runtime.

Pitfall: Omitting the .js extension on relative imports under nodenext — it type-checks in some configs but fails at runtime in native ESM. Also: mixing require() and import in the same graph without an interop story yields undefined default exports.
R3. Declaration Files Modules & Project Structure 15 days since review · rep 3 → 4
.d.ts files describe shapes without implementation. declare module "x" types an untyped package; declare global augments global scope; ambient declarations (declare const) describe things provided at runtime. Ship your own types by pointing "types" at a generated .d.ts and enabling declaration.
Key Pattern 
// types/legacy-lib.d.ts — type an untyped dependency
declare module "legacy-lib" {
  export interface Options { retries?: number }
  export function connect(url: string, opts?: Options): Promise<void>;
}

// augment the global scope
declare global {
  interface Window { __APP_VERSION__: string }
}
export {};   // makes this file a module so `declare global` is allowed
declare module types a package; declare global needs the file to be a module
Challenge Exercise

Write a .d.ts that adds a typed property to Express’s Request via module augmentation, and a wildcard declare module "*.svg" for asset imports. Then explain the difference between a script and a module .d.ts (top-level import/export) and why it changes whether declarations are global.

Pitfall: Adding a top-level import to an ambient .d.ts meant to be global — it silently becomes a module and your global declarations stop applying. Use declare global { ... } with an export {}, or keep the file import-free.
R4. tsconfig.json Deep Dive Modules & Project Structure 15 days since review · rep 3 → 4
strict is a bundle (strictNullChecks, noImplicitAny, …). moduleResolution and module should usually both be nodenext for Node. paths+baseUrl give import aliases (but the bundler must mirror them). Separate the type-check config from the emit config, and prefer tsc --noEmit for checking when a bundler emits.
Key Pattern 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "nodenext",
    "moduleResolution": "nodenext",
    "strict": true,
    "noUncheckedIndexedAccess": true,   // arr[i] is T | undefined
    "verbatimModuleSyntax": true,
    "paths": { "@app/*": ["./src/*"] },
    "declaration": true,
    "noEmit": true                       // bundler emits; tsc only checks
  },
  "include": ["src"]
}
strict + noUncheckedIndexedAccess catch the bugs; paths need bundler mirroring
Challenge Exercise

Explain what each of noUncheckedIndexedAccess, exactOptionalPropertyTypes, and useUnknownInCatchVariables changes, with a one-line example each. Then set up a project that extends a shared tsconfig.base.json and overrides outDir/rootDir per package.

Pitfall: Setting paths in tsconfig but forgetting the runtime/bundler doesn’t know them — types resolve, imports crash at run. Aliases must be mirrored in the bundler (or tsc-alias/tsconfig-paths). And strict: false hides real null bugs; turn it on from day one.
R5. Namespaces vs Modules Modules & Project Structure 15 days since review · rep 3 → 4
ES modules (files with import/export) are the standard for organizing code. namespace (the old internal module) is largely legacy — useful mainly for grouping types in a .d.ts or merging with a function/class via declaration merging. Don’t reach for namespace to structure app code.
Key Pattern 
// Legitimate namespace use: group related types + merge onto a function
function createStore() { /* ... */ }
namespace createStore {
  export interface Options { persist?: boolean }
  export type State = Record<string, unknown>;
}

const opts: createStore.Options = { persist: true };   // merged type access

// For app code, prefer modules:
// store.ts  ->  export function createStore() {}
//               export interface Options {}
namespace merges types onto a value; modules are the default for real code
Challenge Exercise

Take a file that uses nested namespaces and refactor it into ES modules with barrel index.ts re-exports. Then describe one case where a namespace in a .d.ts is still the right tool (e.g. typing a UMD global like d3.select).

Pitfall: Using namespace to avoid imports in application code — you lose tree-shaking, get implicit global-ish coupling, and fight the module system. Reserve namespaces for declaration merging and ambient UMD typings; use modules everywhere else.
R6. Zod & Runtime Validation Runtime Patterns 15 days since review · rep 3 → 4
Types vanish at runtime, so external data (HTTP bodies, env, JSON) must be validated, not asserted. Zod defines a schema once, gives a runtime parser and a static type via z.infer, so the validator and the type can never drift. parse throws; safeParse returns a discriminated result.
Key Pattern 
import { z } from "zod";

const User = z.object({
  id: z.string().uuid(),
  age: z.number().int().nonnegative(),
  role: z.enum(["admin", "user"]),
});
type User = z.infer<typeof User>;   // { id: string; age: number; role: "admin" | "user" }

const res = User.safeParse(await req.json());
if (!res.success) return reply.code(400).send(res.error.flatten());
res.data;   // fully typed User — proven at runtime
one schema yields both a runtime guard and the static type via z.infer
Challenge Exercise

Model an API response with a z.discriminatedUnion("status", [...]) and write a typed fetchJson<S extends z.ZodType>(url, schema): Promise<z.infer<S>> helper. Then use .transform() to coerce a date string into a Date and explain how input vs output types diverge (z.input vs z.output).

Pitfall: Casting untrusted data with as User instead of parsing it — the cast is a lie the compiler believes, and a missing field surfaces as a crash deep in business logic. Validate at the boundary; let the proven type flow inward.
1Reliable Message Processing
1. Idempotent Consumers & Exactly-Once Processing Node.js + TypeScript
Every real broker delivers at-least-once: the same message can arrive twice (a redelivery after a crash, a retried publish from an outbox). True “exactly-once” delivery doesn’t exist across a network — but you can get exactly-once effect by making the consumer idempotent: record each processed message id and skip duplicates. Typing the dedup store and the message envelope makes “did we handle this already?” a checked question rather than a hope.

The recipe has three typed pieces. First, every message carries a stable id in a typed Envelope<T> — the id is the deduplication key, and it must be assigned by the producer, not generated on receipt. Second, a ProcessedStore interface records which ids are done; a generic keeps the key type honest. Third, the handler runs inside a check-set boundary: mark-processed and the side effect should commit together (same DB transaction, Concept from 07-06) so a crash can’t leave one without the other.

Two dedup styles: optimistic (try to insert the id with a unique constraint; a conflict means “already handled”) or look-then-act (SELECT, branch). The optimistic insert is race-safe under concurrency; the look-then-act is not unless it holds a lock. Prefer the unique-constraint insert.

1
Give every message a producer-assigned id in a typed envelope.
2
Record processed ids in a typed store; a duplicate id short-circuits.
3
Commit the effect and the “processed” marker atomically.
TypeScript · example 1 of 4 — a typed envelope and dedup store
interface Envelope<T> {
  id: string;            // stable, producer-assigned — the dedup key
  occurredAt: string;
  data: T;
}

interface ProcessedStore {
  // returns true if newly marked, false if the id was already present
  markIfNew(id: string): Promise<boolean>;
}
the id lives on the envelope; markIfNew returns whether this is a first sighting
TypeScript · example 2 of 4 — a generic idempotent handler wrapper
function idempotent<T>(
  store: ProcessedStore,
  handle: (data: T) => Promise<void>,
): (msg: Envelope<T>) => Promise<void> {
  return async (msg) => {
    const isNew = await store.markIfNew(msg.id);
    if (!isNew) return;               // duplicate: acknowledge and skip
    await handle(msg.data);
  };
}

const onOrder = idempotent<{ orderId: string }>(store, async ({ orderId }) => {
  await chargeCustomer(orderId);      // runs at most once per message id
});
the wrapper preserves T from store to handler; duplicates never re-run the effect
TypeScript · example 3 of 4 — race-safe markIfNew via a unique constraint
// CREATE TABLE processed (id TEXT PRIMARY KEY, at TIMESTAMPTZ DEFAULT now());
class SqlProcessedStore implements ProcessedStore {
  constructor(private db: Db) {}
  async markIfNew(id: string): Promise<boolean> {
    const r = await this.db.query(
      "INSERT INTO processed(id) VALUES ($1) ON CONFLICT DO NOTHING", [id]);
    return r.rowCount === 1;          // 1 = inserted (new); 0 = conflict (dup)
  }
}
the DB's unique index is the concurrency guard — two workers can't both "win"
TypeScript · example 4 of 4 — atomic effect + marker in one transaction
async function handleAtomically<T>(
  db: Db, msg: Envelope<T>, effect: (tx: Tx, data: T) => Promise<void>,
): Promise<void> {
  await db.transaction(async (tx) => {
    const r = await tx.query(
      "INSERT INTO processed(id) VALUES ($1) ON CONFLICT DO NOTHING", [msg.id]);
    if (r.rowCount === 0) return;     // already processed — tx is a no-op
    await effect(tx, msg.data);       // effect + marker commit together
  });
}
the marker and the side effect share a transaction: no half-done processing
Common Mistakes:
❌ Generating the dedup id when the message is received — a redelivery gets a new id and is treated as fresh, defeating the whole scheme.
✅ Assign the id at the producer (or derive it from stable business keys) so redeliveries share it.
❌ Marking processed after the effect in a separate step — a crash in between double-charges on redelivery.
✅ Commit the “processed” row and the effect in one transaction so they succeed or fail together.
Key Rules:
  • Brokers are at-least-once; achieve exactly-once effect with an idempotent consumer, not a magic flag.
  • Deduplicate on a stable, producer-assigned message id carried in a typed envelope.
  • Use a unique-constraint insert (ON CONFLICT DO NOTHING) so concurrent workers can’t both process.
  • Commit the side effect and the processed-marker atomically; otherwise a crash re-runs the effect.
Practice Exercises:
1Add a TTL to the processed store (drop ids older than N days) and reason about the redelivery window you can safely forget.
2Implement a Redis ProcessedStore using SET id 1 NX and compare its guarantees with the SQL unique-constraint version.
3Extend Envelope<T> with an attempt counter and log when a message is processed on a redelivery (attempt > 1) vs skipped as a duplicate.
Connections: This is the consumer half of the outbox pattern (07-07): the outbox guarantees at-least-once delivery, and idempotent consumers turn that into exactly-once effect. The atomic marker+effect reuses withTransaction/Tx from 07-06’s data-integrity day, and the typed Envelope<T> is the same shape threaded through typed message queues.
2. Dead Letter Queues & Poison Messages Node.js + TypeScript
Some messages will never succeed — malformed payloads, a referenced record that was deleted, a bug that throws every time. Retrying such a poison message forever blocks the queue and burns resources. A dead letter queue (DLQ) is where you divert a message after N failed attempts, preserving it (with the failure reason) for inspection and replay instead of dropping it. Typing the DLQ envelope keeps the original message shape and the error metadata inspectable.

The consumer tracks an attempt count per message. On failure it decides: retry (attempt < max, usually with backoff from 07-04) or dead-letter (attempt ≥ max). Dead-lettering wraps the original message in a typed DeadLetter<T> envelope capturing the reason, the failing stage, and the attempt count, then writes it to a separate queue/table. The main queue moves on.

Crucially, distinguish transient failures (network blip — worth retrying) from permanent ones (schema validation error — dead-letter immediately; retrying can’t help). Model that decision in the type: a handler can throw a tagged PermanentError vs a TransientError, and the dispatcher routes accordingly. A DLQ is only useful if someone can replay from it after a fix, so keep the payload intact.

1
Count attempts; retry transient failures with backoff up to a max.
2
On permanent failure or exhausted retries, wrap in a typed DeadLetter<T>.
3
Persist to the DLQ with reason + payload so it can be inspected and replayed.
TypeScript · example 1 of 4 — tagged failure kinds + DLQ envelope
class TransientError extends Error { readonly kind = "transient" as const; }
class PermanentError extends Error { readonly kind = "permanent" as const; }

interface DeadLetter<T> {
  original: T;              // the untouched payload — replayable after a fix
  reason: string;
  failedStage: string;
  attempts: number;
  deadLetteredAt: string;
}
the DLQ preserves the original T plus why/where/how-many-times it failed
TypeScript · example 2 of 4 — classify failures to decide retry vs dead-letter
function isRetryable(err: unknown): boolean {
  // permanent errors are never retried; unknown/transient errors are
  if (err instanceof PermanentError) return false;
  if (err instanceof TransientError) return true;
  return true;   // default: treat unclassified errors as transient
}

// a validation failure is permanent — retrying the same bytes can't help
function validateOrThrow<T>(parse: () => T): T {
  try { return parse(); }
  catch (e) { throw new PermanentError(`bad payload: ${(e as Error).message}`); }
}
the error's tag drives routing: permanent => straight to DLQ, transient => retry
TypeScript · example 3 of 4 — a consumer with bounded retries + DLQ
interface Dlq<T> { send(letter: DeadLetter<T>): Promise<void>; }

async function consume<T>(
  msg: Envelope<T>, maxAttempts: number,
  handle: (data: T) => Promise<void>, dlq: Dlq<T>,
): Promise<void> {
  try {
    await handle(msg.data);
  } catch (err) {
    const attempts = (msg.attempt ?? 1);
    if (isRetryable(err) && attempts < maxAttempts) {
      throw err;                       // nack -> broker redelivers (backoff)
    }
    await dlq.send({
      original: msg.data, reason: (err as Error).message,
      failedStage: "handle", attempts, deadLetteredAt: new Date().toISOString(),
    });                                // give up cleanly, keep the payload
  }
}
retry while transient and under the cap; otherwise dead-letter with full context
TypeScript · example 4 of 4 — replaying from the DLQ after a fix
async function replay<T>(
  dlqRows: DeadLetter<T>[], reprocess: (data: T) => Promise<void>,
): Promise<{ ok: number; stillFailing: DeadLetter<T>[] }> {
  let ok = 0; const stillFailing: DeadLetter<T>[] = [];
  for (const row of dlqRows) {
    try { await reprocess(row.original); ok++; }
    catch { stillFailing.push(row); }  // the fix didn't cover this one
  }
  return { ok, stillFailing };
}
because `original` kept the payload, a deployed fix can drain the DLQ
Common Mistakes:
❌ Retrying a malformed message forever — it can never succeed, so it blocks the queue and floods logs (a “poison pill”).
✅ Classify permanent failures (validation) and dead-letter them immediately; only retry transient ones.
❌ Dropping a message after max retries with just a log line — the data and the reason are gone forever.
✅ Divert to a typed DLQ that keeps the original payload + reason so it can be inspected and replayed.
Key Rules:
  • Bound retries with a max-attempt count; never retry a permanent (e.g. validation) failure.
  • Model failure kind in the type (TransientError vs PermanentError) and route on it.
  • Dead-letter into a typed DeadLetter<T> that preserves the original payload and the reason.
  • A DLQ is only worthwhile if you can replay from it — keep the payload intact and build a drain tool.
Practice Exercises:
1Add exponential backoff with jitter (from 07-04) between retries and record each attempt’s delay in the envelope.
2Type a metrics counter ({ retried; deadLettered; replayed }) and emit it from consume/replay for a DLQ dashboard.
3Add a schemaVersion to DeadLetter<T> so replays can upcast old payloads (ties into Concept 4) before reprocessing.
Connections: The retry/backoff logic reuses 07-04’s typed retries & circuit breakers, and the tagged PermanentError/TransientError is discriminated-union error handling. Permanent failures are usually Zod validation errors at the boundary (R6). Replaying old dead letters is exactly why event schema versioning (Concept 4) matters.
3. Distributed Locks & Leader Election Node.js + TypeScript
When several instances of your service run, some work must happen once across the whole cluster: a nightly job, a cron tick, the outbox relay from 07-07. A distributed lock lets exactly one instance hold a named resource at a time; leader election is the same idea applied to “who runs the singleton work.” Typing the lock as a resource you acquire and must release — ideally via using (Symbol.dispose) — makes “forgot to unlock” a compile-time-visible concern rather than a production deadlock.

A correct lock has three properties: mutual exclusion (one holder), a TTL (auto-expire so a crashed holder doesn’t deadlock the cluster forever), and a fencing token (a monotonically increasing number proving you still hold it, so a slow holder whose TTL expired can’t stomp the new one). In Redis the acquire is SET key owner NX PX ttl; release must be a compare-and-delete (only delete if you own it), which requires a small Lua script to be atomic.

The TS win is modeling the lock as a disposable handle. With await using lock = await mutex.acquire(...), the lock releases automatically at scope exit — even on throw — because the handle implements [Symbol.asyncDispose]. A branded FencingToken ensures the number you pass to a write really came from a lock acquisition.

1
Acquire with SET NX PX (owner + TTL); returns a handle or null.
2
Do the singleton work; carry the fencing token to guarded writes.
3
Release via compare-and-delete — automatically through await using.
TypeScript · example 1 of 4 — a disposable, fenced lock handle
declare const fence: unique symbol;
type FencingToken = number & { readonly [fence]: true };

interface LockHandle extends AsyncDisposable {
  readonly resource: string;
  readonly token: FencingToken;      // proves ongoing ownership to writers
}

interface Mutex {
  // returns null if someone else holds the lock
  acquire(resource: string, ttlMs: number): Promise<LockHandle | null>;
}
the handle is AsyncDisposable — `await using` releases it at scope exit
TypeScript · example 2 of 4 — Redis acquire (SET NX PX) + atomic release
const RELEASE = `if redis.call("get", KEYS[1]) == ARGV[1]
                 then return redis.call("del", KEYS[1]) else return 0 end`;

class RedisMutex implements Mutex {
  constructor(private redis: Redis) {}
  async acquire(resource: string, ttlMs: number): Promise<LockHandle | null> {
    const owner = crypto.randomUUID();
    const ok = await this.redis.set(resource, owner, "PX", ttlMs, "NX");
    if (ok !== "OK") return null;                    // someone else holds it
    const token = Date.now() as FencingToken;        // monotonic-ish fence
    return {
      resource, token,
      [Symbol.asyncDispose]: async () => {
        await this.redis.eval(RELEASE, 1, resource, owner);  // delete iff mine
      },
    };
  }
}
NX gives mutual exclusion, PX gives a TTL, compare-and-delete avoids releasing another owner's lock
TypeScript · example 3 of 4 — run singleton work with automatic release
async function runExclusive(mutex: Mutex, resource: string, work: () => Promise<void>) {
  await using lock = await mutex.acquire(resource, 30_000);
  if (!lock) return;                 // another instance is already running it
  await work();                      // lock auto-releases at scope end, even on throw
}

// only one instance across the cluster actually runs the relay tick
await runExclusive(mutex, "outbox-relay", () => relayOnce(broker));
`await using` disposes the lock on every exit path — no leaked locks on error
TypeScript · example 4 of 4 — fencing tokens stop a stale holder from writing
// a write guarded by the highest fencing token seen wins;
// a paused holder whose TTL expired carries a stale (smaller) token and is rejected
async function guardedWrite(db: Db, token: FencingToken, value: string) {
  await db.query(
    `UPDATE resource SET value = $1, fence = $2 WHERE fence < $2`,
    [value, token],
  );   // no-op if a newer holder already advanced the fence
}
the branded token can only come from acquire(); stale writes are filtered by fence < token
Common Mistakes:
❌ A lock with no TTL — if the holder crashes, the resource is locked forever and the cluster stalls.
✅ Always set an expiry (PX); a crashed holder’s lock auto-releases so work can resume.
❌ Releasing with a plain DEL — you might delete a lock a different instance now holds after your TTL lapsed.
✅ Compare-and-delete (Lua) so you only release your own lock; use fencing tokens for stale-holder writes.
Key Rules:
  • A distributed lock needs mutual exclusion, a TTL (crash safety), and ideally a fencing token.
  • Acquire with SET NX PX; release with an atomic compare-and-delete, never a bare DEL.
  • Model the lock as an AsyncDisposable handle and take it with await using for leak-free release.
  • For correctness under pauses/GC, guard critical writes with a monotonic fencing token.
Practice Exercises:
1Add lock renewal: a background timer that extends the TTL (PEXPIRE) while work runs, and cancel it on dispose.
2Turn the mutex into leader election: the current leader renews a lock; followers poll and take over when it expires.
3Explain why single-Redis locks aren’t perfectly safe under failover, and sketch what Redlock adds (and its criticisms).
Connections: The await using / AsyncDisposable handle is the explicit-resource-management feature (TS 5.2) applied to a lock, and the branded FencingToken reuses branded-types (R4 from 07-07). Leader election is what makes the outbox relay (07-07) and cron ticks run exactly once across instances — complementing the idempotent consumers of Concept 1.
2Contract Evolution & Testing
4. Event Schema Versioning & Upcasting TypeScript
Events and messages are persisted — an event stored last year must still be readable by today’s code. You can’t edit history (07-07), so schemas must evolve: add fields, rename, restructure, all while old versions linger in the log and the DLQ. The type-safe answer is a versioned discriminated union plus upcasters — pure functions that migrate vN to vN+1 — so the rest of your code only ever sees the latest shape, and the compiler proves every version is handled.

Tag each event with both a type and a v (version). The stored form is a union over all versions; the current form is the newest. An upcaster chain v1 → v2 → v3 lifts any old event to current: each step is a total function typed (old: Vn) => Vn+1. Reading becomes: parse the stored bytes (Zod), match the version, run it through the remaining upcasters. Because the union is exhaustive, forgetting to upcast a version is a compile error via the never guard.

Golden rules of evolution: prefer additive, optional changes (backward compatible); when you must break, bump the version and write an upcaster rather than mutating old records. Keep upcasters pure and append-only — they’re the migration history of your event log.

1
Tag events with type + v; the stored type is a union of all versions.
2
Write pure upcasters Vn → Vn+1; chain them to reach current.
3
Consumers only see the latest shape; a never guard enforces coverage.
TypeScript · example 1 of 4 — versioned event shapes
interface UserRegisteredV1 { type: "UserRegistered"; v: 1; name: string }
// v2 splits `name` into first/last — a breaking change
interface UserRegisteredV2 { type: "UserRegistered"; v: 2; firstName: string; lastName: string }
// v3 adds an optional email — additive, backward compatible
interface UserRegisteredV3 { type: "UserRegistered"; v: 3; firstName: string; lastName: string; email?: string }

type StoredUserRegistered = UserRegisteredV1 | UserRegisteredV2 | UserRegisteredV3;
type CurrentUserRegistered = UserRegisteredV3;   // what app code consumes
the stored union spans every version ever written; `v` is the discriminant
TypeScript · example 2 of 4 — pure, total upcasters
const up1to2 = (e: UserRegisteredV1): UserRegisteredV2 => {
  const [firstName, ...rest] = e.name.split(" ");
  return { type: "UserRegistered", v: 2, firstName, lastName: rest.join(" ") };
};
const up2to3 = (e: UserRegisteredV2): UserRegisteredV3 => ({ ...e, v: 3 });
each upcaster is a typed Vn -> Vn+1 step; TS checks the shapes at each hop
TypeScript · example 3 of 4 — exhaustive upcast to current
function toCurrent(e: StoredUserRegistered): CurrentUserRegistered {
  switch (e.v) {
    case 1: return up2to3(up1to2(e));
    case 2: return up2to3(e);
    case 3: return e;
    default:
      const _exhaustive: never = e;   // a new version without a case won't compile
      return _exhaustive;
  }
}
adding UserRegisteredV4 to the union breaks this switch until you handle it
TypeScript · example 4 of 4 — parse stored bytes, then upcast

Common Mistakes:
❌ Mutating old stored events in place to “migrate” them — you rewrite history, break auditability, and risk a half-finished migration.
✅ Leave old events untouched; bump the version and add a pure upcaster that lifts them on read.
❌ A silent default in the version switch — an unhandled old version slips through as an unmigrated shape.
✅ Guard the default with const _: never = e so every version must have an upcast path.
Key Rules:
  • Version every persisted event (v); the stored type is a union of all versions, current is the newest.
  • Prefer additive, optional changes; when you must break, bump the version and write an upcaster.
  • Upcasters are pure, total Vn → Vn+1 functions; chain them and enforce coverage with never.
  • Validate stored bytes with a discriminated Zod schema before upcasting — the log is a trust boundary.

5. Consumer-Driven Contract Testing TypeScript
Types guarantee agreement within one codebase, but two independently deployed services can’t share a compiler. When the API team renames a field, the frontend’s types still compile — against a now-wrong local interface — and it breaks in production. Consumer-driven contract testing (e.g. Pact) closes that gap: the consumer declares exactly what it needs as an executable contract, the provider is verified against it in CI, and a mismatch fails the build before deploy. TypeScript keeps the contract, the mock, and the real client all typed against one shared shape.

The flow: the consumer writes a test describing a request and the minimal response it depends on — that produces a pact (a JSON contract) and runs the consumer against a mock provider that honors it. The provider then replays every interaction against the real service in its own CI; if the response no longer matches, the provider’s build fails. The contract is consumer-driven: it captures only the fields consumers actually use, so the provider is free to add fields but not remove/rename used ones.

The TS discipline is to derive one type from the contract (often via a Zod schema shared between consumer test and runtime client) so the mock, the assertion, and the production fetch can’t drift. This complements — not replaces — schema versioning (Concept 4): versioning lets the provider evolve safely; contract tests prove a given change didn’t break a real consumer.

1
Consumer declares the request + needed response fields as a contract.
2
Consumer runs against a mock honoring the contract; a pact file is emitted.
3
Provider CI verifies the real service still satisfies every interaction.
TypeScript · example 1 of 4 — one shared, typed contract shape
import { z } from "zod";

// the consumer depends only on these fields — the contract's surface
const UserResponse = z.object({
  id: z.string(),
  displayName: z.string(),      // consumer reads this; provider must keep it
});
type UserResponse = z.infer<typeof UserResponse>;

interface Contract {
  request: { method: "GET"; path: `/users/${string}` };
  response: { status: 200; body: UserResponse };
}
one Zod schema is the single source of truth for mock, assertion, and client
TypeScript · example 2 of 4 — consumer test against a mock provider
// pseudo-Pact: the consumer describes what it needs, gets a mock URL back
await provider
  .given("user 42 exists")
  .uponReceiving("a request for user 42")
  .withRequest({ method: "GET", path: "/users/42" })
  .willRespondWith({ status: 200, body: { id: "42", displayName: "Ada" } });

await provider.executeTest(async (mockUrl) => {
  const user = await getUser(mockUrl, "42");   // real consumer client code
  expect(UserResponse.parse(user)).toEqual({ id: "42", displayName: "Ada" });
});   // running this emits a pact file the provider must honor
the consumer's own client runs against the mock; success produces the contract
TypeScript · example 3 of 4 — the production client shares the type
async function getUser(baseUrl: string, id: string): Promise<UserResponse> {
  const res = await fetch(`${baseUrl}/users/${id}`);
  return UserResponse.parse(await res.json());   // same schema as the test
}

// the mock and the real call can never disagree on the shape
reusing getUser + UserResponse means the tested path IS the shipped path
TypeScript · example 4 of 4 — provider-side verification in CI
// provider CI: replay each recorded interaction against the REAL service

await new Verifier({

Common Mistakes:
❌ Trusting a hand-written local interface for a remote API — it compiles happily while the real response has drifted, breaking only in prod.
✅ Verify the real provider against a consumer-driven contract in CI so drift fails the build, not production.
❌ Putting every provider field in the contract — now harmless additive changes break the consumer test needlessly.
✅ Capture only the fields the consumer actually uses; let the provider add fields freely (tolerant reader).
Key Rules:
  • Cross-service agreement needs a runtime contract; a shared compiler doesn’t span deploy boundaries.
  • The consumer drives the contract — capture only fields it uses so providers can evolve additively.
  • Verify the real provider against the pact in its CI; a mismatch must fail before deploy.
  • Derive one type (a Zod schema) for the mock, the assertion, and the production client so they can’t drift.
Practice Exercises:
1Add a second interaction (404 for a missing user) and a matching provider state handler; assert the typed error path.
2Use Pact matchers (like, eachLike) so the contract checks shape not exact values, and keep the TS type aligned.
3Combine with Concept 4: bump the provider to a new response version and show the contract test catches a removed field a consumer still needs.
Connections: Contract testing is the cross-service complement to in-repo type testing (tsd/expect-type): types cover one codebase, pacts cover the wire between two. The shared Zod schema is R6 applied as the single source of truth, and it enforces the “don’t break used fields” promise that schema versioning (Concept 4) lets a provider keep while still evolving.
