R3. this Types & Fluent Interfaces Functions & Patterns rep 1 · last reviewed 2026-06-04 · 3 days ago
Polymorphic this type enables method chaining that preserves the subclass type. this in return position means "whatever class I'm called on".
Key Pattern 
class Builder {
  private name = "";
  private port = 3000;

  setName(n: string): this {
    this.name = n;
    return this;
  }

  setPort(p: number): this {
    this.port = p;
    return this;
  }

  describe(): string {
    return `${this.name}:${this.port}`;
  }
}

// Subclass inherits fluent chaining with correct types
class SecureBuilder extends Builder {
  private ssl = false;

  enableSSL(): this {
    this.ssl = true;
    return this;
  }
}

// Chaining works: setName returns SecureBuilder, not Builder
const config = new SecureBuilder()
  .setName("api")     // returns SecureBuilder (not Builder)
  .enableSSL()        // returns SecureBuilder
  .setPort(443);      // returns SecureBuilder

R4. Async Patterns & Promise Typing Functions & Patterns rep 1 · last reviewed 2026-06-04 · 3 days ago
async function foo(): Promise<T>, Awaited<T> unwraps nested promises. Promise.all returns tuple types with as const.
Key Pattern 
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;

// Typed retry with exponential backoff
async function retry<T>(
  fn: () => Promise<T>,
  maxRetries: number,
  delayMs = 100
): Promise<T> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === maxRetries - 1) throw err;
      await new Promise(r => setTimeout(r, delayMs * 2 ** attempt));
    }
  }
  throw new Error("Unreachable");
}

// Promise.all preserves tuple types
async function loadData() {
  const [user, posts, comments] = await Promise.all([
    fetchUser("u1"),           // Promise<User>
    fetchPosts("u1"),          // Promise<Post[]>
    fetchComments("u1"),       // Promise<Comment[]>
  ] as const);
  // user: User, posts: Post[], comments: Comment[]
}
Challenge Exercise

#1 Pitfall: Forgetting that Promise.all([p1, p2]) loses tuple types without as const in some configurations — without it, the result may be typed as an array of the union rather than a tuple of specific types.

1. Covariance & Contravariance Advanced Patterns & Performance
Understanding variance prevents subtle type errors when passing functions, arrays, or generic types. Without it, you'll silently introduce unsafe assignments that crash at runtime despite compiling cleanly. The in/out modifiers (TS 4.7+) make variance explicit in your own generic types.

Variance describes how subtyping relationships between generic types relate to the subtyping relationships of their type parameters. There are three kinds:

  • 2026-07-04 (Day 31)
  • Covariance (out) — if Dog extends Animal, then Producer<Dog> is assignable to Producer<Animal>. Applies to output positions: return types, property reads.
  • Contravariance (in) — if Dog extends Animal, then Consumer<Animal> is assignable to Consumer<Dog>. The relationship flips. Applies to input positions: function parameters.
  • Invariance — when a type parameter appears in both input and output positions, no assignment is safe in either direction.

The strictFunctionTypes flag (included in strict) enforces contravariance for function parameters. Without it, function parameters are checked bivariantly (both directions allowed), which is unsound.

TypeScript — Covariance: Output Positions
interface Animal { name: string }
interface Dog extends Animal { breed: string }
interface Cat extends Animal { indoor: boolean }

// Covariant: Producer only outputs T (return type)
interface Producer<out T> {
  produce(): T;
}

// Dog extends Animal, so Producer<Dog> extends Producer<Animal>
const dogProducer: Producer<Dog> = {
  produce() { return { name: "Rex", breed: "Labrador" }; }
};

const animalProducer: Producer<Animal> = dogProducer;  // OK: covariant
console.log(animalProducer.produce().name);

// Producer<Animal> cannot be assigned to Producer<Dog>
// const reversed: Producer<Dog> = animalProducer;  // Error!

// Arrays are covariant in TypeScript (readonly arrays are safely covariant)
const dogs: readonly Dog[] = [{ name: "Rex", breed: "Lab" }, { name: "Max", breed: "Poodle" }];
const animals: readonly Animal[] = dogs;   // OK: readonly array is safely covariant
console.log(animals[0].name);

TypeScript — Contravariance: Input Positions
// Contravariant: Consumer only accepts T (parameter)
interface Consumer<in T> {
  consume(value: T): void;
}

// A function that can handle any Animal can certainly handle a Dog
const animalConsumer: Consumer<Animal> = {
  consume(animal) { console.log(`Fed ${animal.name}`); }
};

// Animal extends ... well, Dog extends Animal
// So Consumer<Animal> is assignable to Consumer<Dog> (direction flips!)
const dogConsumer: Consumer<Dog> = animalConsumer;  // OK: contravariant
dogConsumer.consume({ name: "Rex", breed: "Labrador" });

// Consumer<Dog> cannot be assigned to Consumer<Animal>
// const reversed: Consumer<Animal> = dogConsumer;  // Error!
// Why? A Dog consumer might rely on .breed, which a Cat doesn't have

// Function parameter contravariance with strictFunctionTypes
type Handler<T> = (value: T) => void;

const handleAnimal: Handler<Animal> = (a) => console.log(a.name);
const handleDog: Handler<Dog> = handleAnimal;  // OK: contravariant

// const unsafeReverse: Handler<Animal> = handleDog;
// Error with strictFunctionTypes: Handler<Dog> not assignable to Handler<Animal>

TypeScript — Why Mutable Array Covariance Is Unsound
interface Animal { name: string }
interface Dog extends Animal { breed: string }
interface Cat extends Animal { indoor: boolean }

// TypeScript allows this (mutable array covariance) but it's UNSOUND:
const dogs: Dog[] = [{ name: "Rex", breed: "Labrador" }];
const animals: Animal[] = dogs;   // Compiles! But...

// Now we can push a Cat into what's actually a Dog[]
animals.push({ name: "Whiskers", indoor: true });  // No compile error!

// But dogs[1] is now a Cat without a 'breed' property
console.log(dogs[1].breed);  // undefined at runtime! Type system lied.

// WHY TypeScript allows this:
// Forbidding it would break too much existing code.
// Arrays are mutable AND covariant, which is technically unsound.
// This is a deliberate pragmatic tradeoff.

// SAFE alternative: use readonly arrays
const safeDogs: readonly Dog[] = [{ name: "Rex", breed: "Labrador" }];
const safeAnimals: readonly Animal[] = safeDogs;  // OK and safe
// safeAnimals.push(...)  // Error: push doesn't exist on readonly array

TypeScript — Mixed Variance and Invariance
// Transform: T appears in both input (contravariant) and output (covariant)
// With in/out modifiers, you declare the intended variance:
interface Transform<in I, out O> {
  process(input: I): O;
}

// I is contravariant (in), O is covariant (out)
const dogToString: Transform<Dog, string> = {
  process(dog) { return `${dog.name} (${dog.breed})`; }
};

// Transform<Animal, string> is assignable to Transform<Dog, string>
// because Animal is wider input (contravariant)
const animalToString: Transform<Animal, string> = {
  process(animal) { return animal.name; }
};
const alsoHandlesDogs: Transform<Dog, string> = animalToString;  // OK

// Invariant: when T appears in BOTH input and output positions
interface StateStore<T> {
  get(): T;          // covariant (output)
  set(value: T): void;  // contravariant (input)
}
// StateStore<Dog> is NOT assignable to StateStore<Animal> (or vice versa)
// because T appears in both positions, making it invariant

// const dogStore: StateStore<Dog> = { get() { return { name: "Rex", breed: "Lab" } }, set(v) {} };
// const animalStore: StateStore<Animal> = dogStore;  // Error: invariant!
// Why? animalStore.set(cat) would put a Cat in a Dog store
Common Mistakes
❌ Assuming arrays are always safely covariant: const animals: Animal[] = dogs; compiles but allows pushing a Cat into a Dog[] — this is a known unsoundness in TypeScript
✅ Use readonly arrays when you need safe covariance: readonly Dog[] assignable to readonly Animal[] is sound because you can't mutate the array.
❌ Not enabling strictFunctionTypes: without it, function parameter types are checked bivariantly (both directions), which defeats the purpose of contravariance checking
✅ Always use "strict": true (which includes strictFunctionTypes). This ensures function parameters are properly checked contravariantly.
Key Rules
  • Return types are covariant (output) — Dog extends Animal means () => Dog is assignable to () => Animal
  • Parameter types are contravariant (input) — (a: Animal) => void is assignable to (d: Dog) => void, not the other way around
  • Use in/out modifiers (TS 4.7+) to declare variance explicitly on your own generic interfaces — this catches errors early and documents intent

2. Template Literal Types — Advanced Advanced Patterns & Performance
Template literal types enable compile-time string manipulation — extracting, validating, and transforming string patterns at the type level. They power route typing in frameworks like tRPC, CSS-in-JS property validation, event emitter types, and any API where string patterns carry semantic meaning.

Template literal types build on the basic `Hello, ${string}` pattern with utility types for case manipulation, recursive pattern extraction, and union distribution. Combined with infer, they become a compile-time string parser.

  • Basic: type Greeting = `Hello, ${string}` — matches any string starting with "Hello, "
  • Utility types: Uppercase<T>, Lowercase<T>, Capitalize<T>, Uncapitalize<T>
  • Pattern extraction: S extends `user_${infer Id}` ? Id : never
  • Union distribution: template literals distribute over union types automatically
  • Event handler types: derive handler names from event names
TypeScript — CSS Units and String Validation
// CSS unit type: only accepts valid CSS length values
type CSSUnit = "px" | "em" | "rem" | "%";
type CSSLength = `${number}${CSSUnit}`;

function setWidth(el: HTMLElement, width: CSSLength): void {
  el.style.width = width;
}

// setWidth(el, "100px");   // OK
// setWidth(el, "2.5rem");  // OK
// setWidth(el, "50%");     // OK
// setWidth(el, "100");     // Error: not assignable to CSSLength
// setWidth(el, "wide");    // Error: not assignable to CSSLength

// Template literals distribute over unions
type Color = "red" | "blue" | "green";
type Shade = "light" | "dark";
type ColorScheme = `${Shade}-${Color}`;
// "light-red" | "light-blue" | "light-green" | "dark-red" | "dark-blue" | "dark-green"

// Event handler naming: from event name to handler name
type EventName = "click" | "hover" | "scroll" | "focus";
type HandlerName = `on${Capitalize<EventName>}`;
// "onClick" | "onHover" | "onScroll" | "onFocus"

// HTTP method + path combinations
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";
type ApiPath = "/users" | "/posts" | "/comments";
type Endpoint = `${HttpMethod} ${ApiPath}`;

// Extract route parameters from a URL pattern
type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<`/${Rest}`>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type UserRoute = ExtractParams<"/users/:userId">;
// "userId"

type PostRoute = ExtractParams<"/users/:userId/posts/:postId">;
// "userId" | "postId"

type CommentRoute = ExtractParams<"/users/:userId/posts/:postId/comments/:commentId">;
// "userId" | "postId" | "commentId"

// Build a params object from extracted parameter names
type RouteParams<T extends string> = {
  [K in ExtractParams<T>]: string;
};

type UserParams = RouteParams<"/users/:userId/posts/:postId">;
// { userId: string; postId: string }

// Extract the prefix from an event name
type ExtractEvent<T extends string> =
  T extends `on${infer Name}` ? Uncapitalize<Name> : T;

type ClickEvent = ExtractEvent<"onClick">;      // "click"
type HoverEvent = ExtractEvent<"onMouseEnter">;  // "mouseEnter"
type Raw = ExtractEvent<"submit">;               // "submit" (no "on" prefix)

TypeScript — CamelCase to snake_case Conversion
// Convert camelCase to snake_case at the type level
type CamelToSnake<S extends string> =
  S extends `${infer Head}${infer Tail}`
    ? Head extends Uppercase<Head>
      ? Head extends Lowercase<Head>
        ? `${Head}${CamelToSnake<Tail>}`         // not a letter (digit, etc.)
        : `_${Lowercase<Head>}${CamelToSnake<Tail>}`  // uppercase letter
      : `${Head}${CamelToSnake<Tail>}`           // lowercase letter
    : S;

type A = CamelToSnake<"myVariableName">;       // "my_variable_name"
type B = CamelToSnake<"backgroundColor">;      // "background_color"
type C = CamelToSnake<"getHTTPSUrl">;          // "get_h_t_t_p_s_url"

// Convert all object keys from camelCase to snake_case
type SnakeKeys<T> = {
  [K in keyof T as K extends string ? CamelToSnake<K> : K]: T[K];
};

TypeScript — Typed Event Emitter from Event Map
// Define events as a map
interface EventMap {
  click: { x: number; y: number };
  focus: { target: string };
  resize: { width: number; height: number };
}

// Derive handler types from event names
type HandlerMap<T> = {
  [K in keyof T & string as `on${Capitalize<K>}`]: (event: T[K]) => void;
};

type Handlers = HandlerMap<EventMap>;
// {
//   onClick: (event: { x: number; y: number }) => void;
//   onFocus: (event: { target: string }) => void;
//   onResize: (event: { width: number; height: number }) => void;
// }

// Typed event emitter class
class TypedEmitter<T extends Record<string, unknown>> {
  private handlers = new Map<string, Set<Function>>();

  on<K extends keyof T & string>(
    event: K,
    handler: (payload: T[K]) => void
  ): void {
    if (!this.handlers.has(event)) this.handlers.set(event, new Set());
    this.handlers.get(event)!.add(handler);
  }

  emit<K extends keyof T & string>(event: K, payload: T[K]): void {
    this.handlers.get(event)?.forEach(fn => fn(payload));
  }
}

const emitter = new TypedEmitter<EventMap>();
emitter.on("click", (e) => console.log(e.x, e.y));     // e: { x: number; y: number }
emitter.on("focus", (e) => console.log(e.target));       // e: { target: string }
// emitter.on("click", (e) => console.log(e.target));    // Error: no 'target' on click event
emitter.emit("click", { x: 10, y: 20 });

TypeScript — Nested Object Key Path Type
// Type-safe dot-notation access paths
type DotPath<T, Prefix extends string = ""> =
  T extends object
    ? {
        [K in keyof T & string]:
          | `${Prefix}${K}`
          | DotPath<T[K], `${Prefix}${K}.`>
      }[keyof T & string]
    : never;

interface Config {
  user: {
    name: string;
    address: {
      city: string;
      zip: number;
    };
  };
  theme: string;
}

type ConfigPath = DotPath<Config>;
// "user" | "user.name" | "user.address" | "user.address.city" | "user.address.zip" | "theme"

// Get the type at a dot-path
type PathType<T, P extends string> =
  P extends `${infer Key}.${infer Rest}`
    ? Key extends keyof T
      ? PathType<T[Key], Rest>
      : never
    : P extends keyof T
      ? T[P]
      : never;

type CityType = PathType<Config, "user.address.city">;  // string
type ZipType = PathType<Config, "user.address.zip">;    // number
type ThemeType = PathType<Config, "theme">;              // string
ConfigPath = "user" | "user.name" | "user.address" | "user.address.city" | "user.address.zip" | "theme" PathType<Config, "user.address.city"> = string PathType<Config, "user.address.zip"> = number
Common Mistakes
❌ Infinite recursion in recursive template literal types: type Parse<S> = S extends `${infer H}${infer T}` ? [H, ...Parse<T>] : [] — on long strings this exceeds recursion depth
✅ Keep recursion depth manageable. Use delimiter-based splitting (e.g., split on "/" or ".") instead of character-by-character parsing. TypeScript limits recursion to about 999 levels.
❌ Forgetting that template literals distribute over unions: `${A | B}_${C | D}` produces all 4 combinations, which can lead to combinatorial explosion with large unions
✅ Be mindful of union sizes. A template literal with 3 union slots of 10 members each produces 1000 combinations. Keep unions small or use string as a fallback for large sets.
Key Rules
  • Use template literal types for string validation and transformation at compile time — CSS units, route patterns, event names, API endpoints
  • Combine with infer for pattern extraction — pull parameter names from routes, strip prefixes from event names, parse dot-paths
  • Keep recursion depth manageable — split on delimiters rather than individual characters, and watch for combinatorial explosion with union distribution

3. TypeScript Performance Advanced Patterns & Performance
Complex types can slow the compiler to a crawl. A codebase that takes 30 seconds to type-check kills developer productivity and CI speed. Understanding how TypeScript resolves types lets you write code that compiles fast while maintaining strong safety guarantees.

TypeScript performance bottlenecks typically come from excessive type instantiation (deeply nested conditional types, recursive generics), large union types, and unnecessary re-checking. The key tools for diagnosis are --generateTrace and --extendedDiagnostics.

  • --generateTrace — produces a Chrome trace profile of type checking, showing which types are expensive
  • --extendedDiagnostics — prints timing breakdown: parse, bind, check, emit
  • interface vs type — interfaces are cached by name (nominal), type aliases are structurally re-evaluated
  • Union type limits — unions with 100+ members slow down assignability checks
  • --incremental and .tsbuildinfo — caches the dependency graph for faster rebuilds
  • @ts-expect-error vs @ts-ignore — @ts-expect-error errors if the next line has no error (safer for tests)

TypeScript — Interface vs Type Alias Performance
// SLOWER: Type alias with intersection (structurally re-evaluated each use)
type UserType = {
  id: string;
  name: string;
} & {
  email: string;
  createdAt: Date;
} & {
  permissions: string[];
  lastLogin: Date;
};

// FASTER: Interface with extends (cached by name, single declaration)
interface UserInterface {
  id: string;
  name: string;
  email: string;
  createdAt: Date;
  permissions: string[];
  lastLogin: Date;
}

// WHY: Interfaces are cached by their name in the type checker's internal cache.
// Type aliases using intersections (&) require the compiler to re-flatten
// and re-evaluate the merged properties each time the type is used.

// When intersection performance matters (large types):
// SLOW: Multiple intersection chains
type SlowConfig = BaseConfig & DatabaseConfig & CacheConfig & AuthConfig & LogConfig;

// FAST: Single interface extending
interface FastConfig extends BaseConfig, DatabaseConfig, CacheConfig, AuthConfig, LogConfig {}

// Rule of thumb:
// - Use interfaces for object shapes (especially when extended by many types)
// - Use type aliases for unions, mapped types, and conditional types
// - Avoid intersection chains with more than 3-4 types; refactor to interface extends
TypeScript — Optimizing Recursive Types
// SLOW: Unbounded recursive type
type DeepPartialSlow<T> =
  T extends object
    ? { [K in keyof T]?: DeepPartialSlow<T[K]> }
    : T;
// For deeply nested objects, this creates a new type instantiation at every level

// FASTER: Bounded-depth approach using a counter
type Depth = [never, 0, 1, 2, 3, 4, 5];  // Index = current, Value = remaining

type DeepPartialBounded<T, D extends number = 5> =
  [D] extends [never]
    ? T  // Stop recursion at depth limit
    : T extends object
      ? { [K in keyof T]?: DeepPartialBounded<T[K], Depth[D]> }
      : T;

// The bounded version stops at 5 levels deep instead of recursing infinitely

// SLOW: Large conditional type chain
type SlowLookup<T extends string> =
  T extends "a" ? 1 :
  T extends "b" ? 2 :
  T extends "c" ? 3 :
  T extends "d" ? 4 :
  // ... 50 more branches
  never;

// FAST: Object lookup (O(1) instead of O(n) conditional chain)
interface FastLookup {
  a: 1;
  b: 2;
  c: 3;
  d: 4;
  // ...
}
type Lookup<T extends keyof FastLookup> = FastLookup[T];

// The object-key approach is constant-time for the type checker,
// while the conditional chain checks each branch sequentially.
TypeScript — Practical Performance Tips
// 1. Avoid "type-level spaghetti" in production code
//    If a type requires more than 3 levels of nesting, simplify or split

// 2. Use skipLibCheck for faster builds (skip checking .d.ts files)
// tsconfig.json:
{
  "compilerOptions": {
    "skipLibCheck": true  // Skip checking all .d.ts files (node_modules)
  }
}

// 3. Avoid gigantic unions (100+ members)
// SLOW: Union of all possible CSS property values
type CSSValue = "absolute" | "relative" | "fixed" | "sticky" | /* 200 more */ string;

// FAST: Use a branded string instead
type CSSPosition = "absolute" | "relative" | "fixed" | "sticky";
// Keep the union small and focused

// 4. Use @ts-expect-error instead of @ts-ignore
// @ts-ignore suppresses the NEXT line's error silently
// @ts-expect-error suppresses the NEXT line's error BUT errors if there IS no error

// @ts-ignore
const a: number = "hello";  // Suppressed, no warning if error is fixed later

// @ts-expect-error: testing that strings are rejected
const b: number = "hello";  // Suppressed, but if this line becomes valid, TS will warn you

// 5. Use project references for monorepos (incremental per-package)
// See Project References review card above for the pattern

// 6. Isolate expensive types behind type aliases
type ExpensiveType<T> = /* complex computation */;
// Give it a name so the result is cached, rather than inlining the computation
Common Mistakes
❌ Creating deeply recursive utility types in production code: type DeepMerge<A, B> = ... with unbounded recursion that compiles slowly and fails on deep objects
✅ Bound recursive types to a maximum depth (5-10 levels is usually enough). Use the tuple-counter pattern: type Depth = [never, 0, 1, 2, 3, 4, 5] to decrement a counter at each level.
❌ Not using --incremental: every build re-checks the entire project from scratch, wasting minutes in CI
✅ Always enable "incremental": true. Cache the .tsbuildinfo file in CI for persistent speed gains across builds.
Key Rules
  • Prefer interfaces over type aliases for extensible object shapes — interfaces are cached by name while intersection types are re-evaluated
  • Avoid recursive types deeper than 5 levels — use depth counters or object-key lookups instead of long conditional chains
  • Use --incremental always and profile with --generateTrace before optimizing — measure first, then fix the actual bottleneck

4. Module Federation Typing Advanced Patterns & Performance
Micro-frontends and module federation split a large application into independently deployed units that share code at runtime. Without shared types, every boundary between these units becomes a potential runtime crash. Typed module federation ensures that remote modules conform to the expected interface before your code ever reaches production.

Module federation (introduced in webpack 5) allows separately built and deployed applications to share modules at runtime. The challenge is maintaining type safety across independently compiled bundles. There are three main approaches: declare module for ambient declarations, shared type packages, and tooling like @module-federation/typescript.

  • declare module "remoteApp/Component" — ambient declaration for a federated module
  • Shared types package — npm package consumed by all micro-frontends for shared interfaces
  • Dynamic import() typing — generic wrappers for lazy-loaded remote modules
  • @module-federation/typescript — automatic generation of .d.ts files from remote entry points
  • Type versioning — semver-compatible type contracts between host and remotes
TypeScript — Declaring Types for Federated Modules
// remote-types.d.ts — declare what the remote app exposes

// Remote: "userApp" exposes a Button component
declare module "userApp/Button" {
  import { FC } from "react";

  interface ButtonProps {
    label: string;
    variant: "primary" | "secondary" | "danger";
    onClick: () => void;
    disabled?: boolean;
  }

  const Button: FC<ButtonProps>;
  export default Button;
}

// Remote: "userApp" exposes a UserService
declare module "userApp/UserService" {
  export interface User {
    id: string;
    name: string;
    email: string;
    role: "admin" | "user";
  }

  export function getUser(id: string): Promise<User>;
  export function listUsers(): Promise<User[]>;
}

// Remote: "chartApp" exposes chart components
declare module "chartApp/LineChart" {
  import { FC } from "react";

  interface DataPoint { x: number; y: number; label?: string }
  interface LineChartProps {
    data: DataPoint[];
    width: number;
    height: number;
    color?: string;
  }

  const LineChart: FC<LineChartProps>;
  export default LineChart;

TypeScript — Typed Dynamic Import for Remote Modules
// A generic wrapper for safely loading federated modules at runtime

interface RemoteModuleConfig {
  scope: string;     // e.g., "userApp"
  module: string;    // e.g., "./Button"
  url: string;       // e.g., "https://user-app.example.com/remoteEntry.js"
}

// Type-safe module loader with error handling
async function loadRemote<T>(config: RemoteModuleConfig): Promise<T> {
  try {
    // In a real module federation setup, this uses the __webpack_init_sharing__
    // and __webpack_share_scopes__ APIs. Simplified here for clarity:
    const container = (window as any)[config.scope];
    if (!container) {
      throw new Error(`Remote container "${config.scope}" not available`);
    }
    await container.init(__webpack_share_scopes__.default);
    const factory = await container.get(config.module);
    const module = factory();
    return module as T;
  } catch (error) {
    throw new Error(
      `Failed to load ${config.scope}/${config.module}: ${(error as Error).message}`
    );
  }
}

// Usage with full type safety:
interface RemoteButton {
  default: React.FC<{ label: string; onClick: () => void }>;
}

const buttonModule = await loadRemote<RemoteButton>({
  scope: "userApp",
  module: "./Button",
  url: "https://user-app.example.com/remoteEntry.js",
});

const Button = buttonModule.default;
// Button is now fully typed: FC<{ label: string; onClick: () => void }>

// React.lazy with typed import
const LazyChart = React.lazy(async () => {
  const mod = await loadRemote<{ default: React.FC<ChartProps> }>({
    scope: "chartApp",
    module: "./LineChart",
    url: "https://chart-app.example.com/remoteEntry.js",
  });
  return mod;
});
TypeScript — Versioned Type Contracts
// Version your type contracts to prevent breaking changes

// v1: Original contract
namespace ContractV1 {
  export interface UserData {
    id: string;
    name: string;
    email: string;
  }
}

// v2: Extended contract (backwards-compatible: new optional fields only)
namespace ContractV2 {
  export interface UserData {
    id: string;
    name: string;
    email: string;
    avatar?: string;       // NEW: optional, doesn't break v1 consumers
    preferences?: {        // NEW: optional, doesn't break v1 consumers
      theme: "dark" | "light";
      language: string;
    };
  }
}

// Type-level compatibility check: v2 must be assignable to v1
type IsBackwardsCompatible = ContractV2.UserData extends ContractV1.UserData
  ? true
  : false;
// true: v2 is a superset of v1

// Runtime version negotiation
interface TypedContract<V extends number, T> {
  version: V;
  data: T;
}

type UserContractV1 = TypedContract<1, ContractV1.UserData>;
type UserContractV2 = TypedContract<2, ContractV2.UserData>;
type AnyUserContract = UserContractV1 | UserContractV2;

function processUser(contract: AnyUserContract): string {
  if (contract.version === 2) {
    // Narrowed to ContractV2.UserData
    return `${contract.data.name} (${contract.data.avatar ?? "no avatar"})`;
  }
  // Narrowed to ContractV1.UserData
  return contract.data.name;
}
IsBackwardsCompatible = true
Common Mistakes
❌ Duplicating types across micro-frontend apps by copy-pasting interfaces — they drift apart silently and cause runtime errors when contract shapes diverge
✅ Share types via an npm package (@myorg/shared-types). Both host and remote apps depend on the same package. Version it with semver to track breaking changes.
❌ Using any for remote modules: const remote = await import("remoteApp/Button") as any — defeats the purpose of TypeScript entirely
✅ Use declare module to type remote modules or create a generic loadRemote<T> wrapper. The type assertion happens once in a controlled location, not scattered throughout the codebase.
Key Rules
  • Share types via npm packages, not copy-paste — a single source of truth prevents type drift between independently deployed applications
  • Version your type contracts with semver — adding optional fields is a minor bump, removing or changing fields is a major bump
  • Use declare module for remote modules and test type compatibility in CI — catch breaking changes before deployment, not after

5. Testing Types: tsd & expect-type Advanced Patterns & Performance
Your utility types and generics need tests too. A small change to a conditional type can silently break every consumer. Type-level tests catch regressions without runtime execution — they verify that your types accept what they should and reject what they shouldn't, running in CI alongside your unit tests.

Two main tools exist for testing types: tsd (tests .d.ts files) and expect-type (fluent API, integrates with vitest/jest). Both let you assert what types resolve to, what assignments should succeed, and what should produce errors.

  • tsd: expectType<T>(value), expectError(expr), expectAssignable<T>(value)
  • expect-type: expectTypeOf(value).toBeString(), expectTypeOf(value).toEqualTypeOf<T>()
  • Testing conditional types, generic constraints, and function overloads
  • Vitest integration: assertType<T>(value) and expectTypeOf built in
  • Test both happy paths and error paths: verify that wrong types are rejected
TypeScript — tsd: Testing Declaration Files
// Install: npm install --save-dev tsd
// Run: npx tsd

// index.d.ts (your library's type declarations)
export function concat(a: string, b: string): string;
export function repeat(str: string, times: number): string;
export type Nullable<T> = T | null | undefined;

// index.test-d.ts (type tests)
import { expectType, expectError, expectAssignable } from "tsd";
import { concat, repeat, Nullable } from "./index.js";

// Test return types
expectType<string>(concat("hello", " world"));
expectType<string>(repeat("ha", 3));

// Test that wrong argument types produce errors
expectError(concat(123, "world"));        // number is not string
expectError(concat("hello"));             // missing second argument
expectError(repeat("ha", "three"));       // string is not number

// Test type aliases
expectType<Nullable<string>>(null);
expectType<Nullable<string>>(undefined);
expectType<Nullable<string>>("hello");

// Test assignability (less strict than expectType)
expectAssignable<Nullable<string>>("hello");     // string is assignable to string | null | undefined
expectAssignable<string | number>(42);            // number is assignable to string | number

// tsd runs at compile time only — no runtime execution needed
// Add to package.json scripts: "test:types": "tsd"

// Basic type checks

expectTypeOf(true).toBeBoolean();
expectTypeOf(null).toBeNull();
expectTypeOf(undefined).toBeUndefined();

import { expectTypeOf } from "expect-type";

// Custom utility type under test
type DeepReadonly<T> =
  T extends (infer U)[]
    ? ReadonlyArray<DeepReadonly<U>>
    : T extends object
      ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
      : T;

// Test primitive types pass through unchanged
expectTypeOf<DeepReadonly<string>>().toEqualTypeOf<string>();

expectTypeOf<DeepReadonly<boolean>>().toEqualTypeOf<boolean>();

// Test nested object
expectTypeOf<DeepReadonly<{ a: { b: number } }>>()
  .toEqualTypeOf<{ readonly a: { readonly b: number } }>();

// Test arrays
expectTypeOf<DeepReadonly<number[]>>()
  .toEqualTypeOf<ReadonlyArray<number>>();

// Test nested arrays
expectTypeOf<DeepReadonly<{ items: { id: number }[] }>>()

TypeScript — Vitest Integration for Type Tests
// vitest has expect-type built in!
import { describe, it, expectTypeOf, assertType } from "vitest";

// Your code under test
function parseInput(input: string): number;
function parseInput(input: number): string;
function parseInput(input: string | number): string | number {
  return typeof input === "string" ? Number(input) : String(input);
}

describe("parseInput type tests", () => {
  it("returns number when given string", () => {
    expectTypeOf(parseInput).returns.toBeNumber();  // first overload
    expectTypeOf(parseInput("42")).toBeNumber();
  });

  it("rejects boolean input", () => {
    // @ts-expect-error: boolean is not string | number
    parseInput(true);
  });
});

// Testing discriminated unions
type Result<T> = { ok: true; value: T } | { ok: false; error: string };

describe("Result type tests", () => {
  it("narrows correctly on ok check", () => {
    const result: Result<number> = { ok: true, value: 42 };

    if (result.ok) {
      expectTypeOf(result).toEqualTypeOf<{ ok: true; value: number }>();
      expectTypeOf(result.value).toBeNumber();
    } else {
      expectTypeOf(result).toEqualTypeOf<{ ok: false; error: string }>();
      expectTypeOf(result.error).toBeString();
    }
  });
});

// Testing branded types
type UserId = string & { __brand: "UserId" };

describe("branded type tests", () => {
  it("rejects plain strings", () => {
    // @ts-expect-error: string is not assignable to UserId

  it("accepts properly branded values", () => {
    const id = "u-123" as UserId;
    expectTypeOf(id).toMatchTypeOf<string>();   // UserId is still a string

  });
});

import { expectTypeOf } from "expect-type";

// CRITICAL: test that invalid usage produces errors, not just that valid usage works

  return a + b;
}

// @ts-expect-error: string args should be rejected

// toEqualTypeOf:  exact structural match (A = B and B = A)
// toMatchTypeOf:  assignability check (A extends B)

expectTypeOf<{ a: 1; b: 2 }>().toMatchTypeOf<{ a: number }>();  // passes
// expectTypeOf<{ a: 1; b: 2 }>().toEqualTypeOf<{ a: number }>();  // fails: not equal
Common Mistakes
❌ Only testing happy paths: verifying that valid types work but never testing that invalid types are rejected — a type that accepts everything is useless
✅ Test both valid and invalid usage. Use @ts-expect-error or expectError() to verify that wrong types produce compile errors. If the error goes away, your type constraint is broken.
❌ Confusing assignability (toMatchTypeOf) with equality (toEqualTypeOf): { a: string; b: number } matches { a: string } but is not equal to it
✅ Use toEqualTypeOf for strict structural equality and toMatchTypeOf for assignability checks. Default to toEqualTypeOf in type tests — it catches more regressions.
Key Rules
  • Test both valid and invalid usage — use @ts-expect-error and expectError() to verify that wrong types are rejected, not just that correct types work
  • Test edge cases with never, unknown, and any — these are the types most likely to expose flaws in conditional type logic
  • Use toEqualTypeOf for strict equality and toMatchTypeOf for assignability — default to strict equality to catch subtle regressions
