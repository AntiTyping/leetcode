R1. Node.js + TypeScript Node.js & APIs overdue · rep 1 → 2
A TS Node project is @types/node for the runtime globals plus a module decision: with "type": "module" you write ESM, lose __dirname/require, and rebuild them from import.meta.url. process.env values are typed string | undefined — the boundary is untyped input you must check, not trust.
Key Pattern 
import { fileURLToPath } from "node:url";
import { dirname, join } from "node:path";
import { readFile } from "node:fs/promises";

// ESM has no __dirname — reconstruct it
const __dirname = dirname(fileURLToPath(import.meta.url));

// process.env is Record<string, string | undefined>
const port = Number(process.env.PORT ?? "3000");   // guard the undefined

const raw = await readFile(join(__dirname, "config.json"), "utf8");
process.env.PORT: string | undefined; readFile resolves to string
Challenge Exercise

Write a typed loadEnv() that reads PORT (number), NODE_ENV ("dev" | "prod"), and DEBUG (boolean) from process.env, throwing a descriptive error for each missing or malformed value, and returns a fully typed { port: number; env: "dev" | "prod"; debug: boolean }. Confirm the return type has no undefined left in it.

Pitfall: Treating process.env.X as a guaranteed string. With the standard @types/node typings each value is string | undefined, so Number(process.env.PORT) on a missing var silently becomes NaN. Parse and validate env once at startup (a small schema or hand-written checks), and remember ESM modules don’t get __dirname/require for free.
R2. API Typing: tRPC & OpenAPI Node.js & APIs overdue · rep 1 → 2
tRPC needs no codegen: the client is typed by importing the server router’s type and inferring each procedure’s input/output (inputs validated by Zod, outputs from the resolver’s return). OpenAPI goes the other way — a spec is the source of truth and a generator (e.g. openapi-typescript) emits the types. Both still validate untrusted bytes at runtime.
Key Pattern 
// server.ts
const appRouter = router({
  getUser: publicProcedure
    .input(z.object({ id: z.string() }))
    .query(({ input }) => db.user(input.id)),   // returns User
});
export type AppRouter = typeof appRouter;       // export the TYPE only

// client.ts
import type { AppRouter } from "./server";
const trpc = createTRPCClient<AppRouter>({ /* links */ });
const user = await trpc.getUser.query({ id: "1" });  // user: User, input checked

type Outputs = inferRouterOutputs<AppRouter>;   // { getUser: User; ... }
client call is end-to-end typed from the server router type
Challenge Exercise

Add a createPost mutation with a Zod input { title: string; body: string } returning { id: string }. On the client, derive its output type with inferRouterOutputs<AppRouter>["createPost"] and its input with inferRouterInputs. Confirm calling trpc.createPost.mutate({ title: 1 }) is a compile error.

Pitfall: Importing the server router as a value into the client (import { appRouter }) instead of import type — that drags server-only code (db drivers, secrets) into the browser bundle. For OpenAPI, the opposite trap: trusting generated types as if they validate responses. The generator only describes the shape; you still parse the payload at runtime before believing it.
R3. Custom ESLint Rules with TypeScript Tooling & Quality overdue · rep 0 → 1
A custom rule is an object with a create(context) returning AST-visitor functions. The superpower of typescript-eslint is type-aware rules: through context.sourceCode.parserServices (or ESLintUtils.getParserServices) you reach the real TypeChecker and ask for the type at a node — impossible with syntax alone.
Key Pattern 
import { ESLintUtils } from "@typescript-eslint/utils";

const createRule = ESLintUtils.RuleCreator(name => `https://docs/${name}`);

export const noAnyCast = createRule({
  name: "no-any-cast",
  meta: { type: "problem", messages: { banned: "Avoid `as any`." }, schema: [] },
  defaultOptions: [],
  create(context) {
    return {
      TSAsExpression(node) {
        if (node.typeAnnotation.type === "TSAnyKeyword") {
          context.report({ node, messageId: "banned" });
        }
      },
    };
  },
});
visitor fires on each `x as any`, reporting a lint error
Challenge Exercise

Write a type-aware rule no-floating-promises-lite: on an ExpressionStatement whose expression is a call, use getParserServices(context) + checker.getTypeAtLocation to detect a Promise return type that is neither awaited nor void-ed, and report it. Note in a comment why this needs parserOptions.project set.

Pitfall: Writing a type-aware rule without parserOptions.project (or projectService) configured — getTypeAtLocation then has no program and every type comes back as any. Also weigh cost: type-aware rules build a full Program, so they’re far slower than syntax-only rules — reserve them for checks that genuinely need type information.
R4. Project References Modules & Project Structure overdue · rep 2 → 3
Project references split a codebase into independently-typecheckable units. Each referenced project sets composite: true (which forces declaration output and a .tsbuildinfo), and consumers list it under references. tsc -b (build mode) walks the dependency graph, builds in order, and skips anything unchanged.
Key Pattern 
// packages/shared/tsconfig.json
{ "compilerOptions": { "composite": true, "declaration": true, "outDir": "dist" } }

// packages/app/tsconfig.json
{
  "compilerOptions": { "outDir": "dist" },
  "references": [{ "path": "../shared" }]    // depends on shared
}

// root tsconfig.json
{ "files": [], "references": [{ "path": "./packages/shared" }, { "path": "./packages/app" }] }
$ tsc -b # builds shared, then app; reuses cache on rebuild
Challenge Exercise

Set up a two-package workspace where app imports a formatDate function from shared. Give shared composite: true, reference it from app, and verify tsc -b builds both in order. Then touch only app and confirm the rebuild skips shared. Finally, break shared’s exported type and watch the error surface at app’s call site.

Pitfall: Importing across packages with only paths aliases and no references entry — plain tsc may resolve it, but tsc -b won’t build the dependency, and editors lose go-to-def across the boundary. A referenced project must be composite and emit declarations; consumers import the built .d.ts, not the source.
R5. State Management Typing: Redux Toolkit & Zustand Ecosystem overdue · rep 1 → 2
RTK infers action and state types from createSlice and you export RootState/AppDispatch once. Zustand infers from the store creator, but the moment you add middleware you must use the curried create<T>()(...) form so the explicit state type survives. StateCreator is how you type composable slices.
Key Pattern 
interface BearState {
  bears: number;
  add: (by: number) => void;
}

// note the CURRIED create<T>()(...) form — required with middleware
const useBearStore = create<BearState>()((set) => ({
  bears: 0,
  add: (by) => set((s) => ({ bears: s.bears + by })),
}));

const bears = useBearStore((s) => s.bears);   // number, selector typed
store + selector fully typed from the BearState interface
Challenge Exercise

Split the store into two slices typed with StateCreator<Store, [], [], BearSlice> and ...FishSlice, then combine them in one create<Store>()((...a) => ({ ...bear(...a), ...fish(...a) })). Confirm cross-slice access (a bear action reading fish state) stays typed, and that the non-curried create<Store>(...) form fails once you add persist middleware.

Pitfall: Calling create<BearState>((set) => ...) (the non-curried overload) and then wrapping with middleware like devtools/persist — the type argument no longer threads through and you get cryptic errors or a widened store. Always reach for create<T>()(...) when middleware is involved; on the RTK side, never re-derive RootState by hand.
R6. Database Typing: Prisma & Drizzle Ecosystem overdue · rep 1 → 2
Prisma generates a typed client from schema.prisma, and a query’s result type narrows to the fields you select — select/include change the return type. Drizzle defines the schema in TypeScript itself and you read row types straight off the table with $inferSelect/$inferInsert, no generation step.
Key Pattern 
// Prisma: the select drives the result type
const u = await prisma.user.findUnique({
  where: { id },
  select: { id: true, email: true },
});
// u: { id: number; email: string } | null   — NOT the full User

// reusable payload type for that exact selection
type UserEmail = Prisma.UserGetPayload<{ select: { id: true; email: true } }>;

// Drizzle: types inferred from the TS table definition
const users = pgTable("users", { id: serial("id").primaryKey(), name: text("name") });
type Row = typeof users.$inferSelect;   // { id: number; name: string }
Prisma result is the selected subset; Drizzle Row from $inferSelect
Challenge Exercise

Write a Prisma helper getUserCard(id) that selects only { id, name, avatarUrl } and whose return type is expressed with Prisma.UserGetPayload (no hand-written interface). Then do the Drizzle equivalent: define a posts table and a function returning typeof posts.$inferInsert for the create path. Confirm reading an unselected field is a compile error.

Pitfall: Casting a select-ed Prisma result back to the full model (as User) so you can read fields you didn’t fetch — they’re undefined at runtime. Let the narrowed type stand and widen the select if you truly need more. With Drizzle, don’t hand-maintain a parallel row interface; derive it with $inferSelect so it can’t drift from the schema.
1Object & Value-Level Type Safety
1. Type-Safe Object.keys / Object.entries Value ↔ Type
Everyone hits this: Object.keys(obj) is typed string[], not (keyof T)[], so you can’t loop the keys and index back into the object without a cast. It’s not a bug — it’s a deliberate soundness choice — and understanding why tells you exactly when a typed wrapper is safe and when it would lie.

Because TypeScript is structurally typed, a value annotated T may carry more properties than T declares (a subtype with extra keys is still a T). If Object.keys returned (keyof T)[], code relying on that would miss those extra keys — so the standard library widens to string[] on purpose. The same reasoning applies to Object.entries and to for...in, where the loop variable is always string.

When you do control the object’s exact shape, a thin generic wrapper that asserts as (keyof T)[] recovers ergonomics. Treat the assertion as a promise you’re making to the compiler: it’s sound only if no wider object ever flows in. For entries, a mapped type produces the precise [K, T[K]] union; for building objects, a typed fromEntries recovers the key/value types that the built-in throws away.

1
Object.keys/entries widen to string keys because subtypes may add properties.
2
A generic helper can assert (keyof T)[] — sound only when you own the object’s shape.
3
Type entries/fromEntries with a mapped type so the key→value pairing survives.
TypeScript · example 1 of 4 — the default widening
const user = { id: 1, name: "Ada" };

const keys = Object.keys(user);     // string[]   (not ("id" | "name")[])
// user[keys[0]];                    // Error: implicit any — string can't index user

for (const k in user) {
  // k: string — not "id" | "name"
}
keys: string[]; for-in key: string
TypeScript · example 2 of 4 — why it must widen (extra props are legal)
interface Named { name: string }

function shout(o: Named) {
  // if keys were ("name")[], the extra runtime key would be invisible
  return Object.keys(o);            // string[]
}

// a wider object is still a Named — structural typing
shout({ name: "Ada", role: "admin" });   // ok: carries an extra "role" key
Object.keys can't promise the keys are exactly keyof Named
TypeScript · example 3 of 4 — typed wrappers you control
function objectKeys<T extends object>(o: T): (keyof T)[] {
  return Object.keys(o) as (keyof T)[];
}

function objectEntries<T extends object>(
  o: T,
): { [K in keyof T]: [K, T[K]] }[keyof T][] {
  return Object.entries(o) as { [K in keyof T]: [K, T[K]] }[keyof T][];
}

const e = objectEntries({ id: 1, name: "Ada" });
// e: (["id", number] | ["name", string])[]
for (const [k, v] of e) { /* k: "id" | "name"; v: number | string */ }
objectEntries yields ["id", number] | ["name", string]
TypeScript · example 4 of 4 — a typed fromEntries
// built-in loses the keys
const wide = Object.fromEntries([["a", 1], ["b", 2]]);   // { [k: string]: number }

function fromEntries<K extends PropertyKey, V>(
  es: readonly (readonly [K, V])[],
): Record<K, V> {
  return Object.fromEntries(es) as Record<K, V>;
}

const o = fromEntries([["a", 1], ["b", 2]] as const);    // Record<"a" | "b", 1 | 2>
o: Record<"a" | "b", 1 | 2> (keys preserved via as const)
Common Mistakes:
❌ Casting Object.keys(o) as string[] and then indexing o[k] — you still get an implicit-any / not-indexable error because string isn’t keyof o.
✅ Use a generic objectKeys<T>(o): (keyof T)[] so the loop variable indexes the object cleanly.
❌ Applying the wrapper to an object you received from elsewhere (a parameter typed T), which may have extra runtime keys — the asserted keyof T then under-reports.
✅ Only assert keyof T when you constructed the object locally; otherwise iterate Object.values or accept string keys.
Key Rules:
  • 2026-07-04 (Day 31)
  • Object.keys/entries return string keys by design — structural subtypes may add properties.
  • A generic wrapper asserting (keyof T)[] is ergonomic but only sound for objects you own.
  • Type entries with the mapped type { [K in keyof T]: [K, T[K]] }[keyof T].
  • Pair fromEntries with as const inputs to keep literal keys and values.
Practice Exercises:
1Write mapValues<T, R>(o: T, fn: (v: T[keyof T], k: keyof T) => R): Record<keyof T, R> using your typed objectEntries/fromEntries.
2Write typedForIn<T>(o: T, fn: (k: keyof T) => void) and explain in a comment when it’s unsound versus a locally-built object.
3Build invert<T extends Record<string, PropertyKey>>(o: T) that swaps keys and values, with the return type computed via a mapped type and key remapping.
Connections: This is the practical face of structural typing (and why Excess Property Checks, Concept 2, exist as a counter-measure). The wrappers lean on keyof & Indexed Access and the mapped type from Key Selection by Value (Day 22). fromEntries with as const previews typeof Type Queries (Concept 3).
2. Excess Property Checks & Object Literal Freshness Assignability
TypeScript normally lets a value with extra properties satisfy a narrower type (structural typing) — yet const o: { a: number } = { a: 1, b: 2 } errors. The difference is freshness: an object literal assigned directly gets an extra typo-catching check. Knowing the rule explains a whole class of “why does this error here but not there” confusion.

When a fresh object literal is assigned to a typed target — or passed directly as an argument — TypeScript runs an excess property check: any property not present in the target type is flagged. This is a heuristic specifically to catch misspellings and stray keys at the point you wrote them. It is not the normal assignability rule.

The literal loses its “freshness” the moment it’s stored in a variable, so assigning through an intermediate variable falls back to plain structural compatibility and the extra property is allowed. Spreads also discard freshness. With union targets the check loosens further: a property that exists in some member is accepted, only a property in no member errors. None of this is a loophole to abuse — it’s the system deliberately checking literals strictly while keeping structural flexibility everywhere else.

1
Fresh literal → typed target triggers an excess property check (unknown keys error).
2
Store the literal in a variable first — freshness is lost, structural rules apply.
3
Spreads drop freshness; union targets only reject keys absent from every member.
TypeScript · example 1 of 4 — the typo-catching check
interface Options { width: number; height?: number }

const a: Options = { width: 10, heigth: 5 };
// Error: Object literal may only specify known properties,
//        but 'heigth' does not exist in type 'Options'. Did you mean 'height'?
'heigth' does not exist in type 'Options' — typo caught at the literal
TypeScript · example 2 of 4 — a variable bypasses the check
const tmp = { width: 10, heigth: 5 };   // inferred shape, fully fine
const b: Options = tmp;                  // OK! freshness lost — structural only

// b only needs to be assignable: it has `width: number`, so it qualifies,
// and the extra `heigth` is simply ignored by the type system.
b: Options accepted — extra key tolerated once it's in a variable
TypeScript · example 3 of 4 — arguments, and opting in to extra keys
function render(o: Options) {}
render({ width: 10, color: "red" });   // Error: 'color' is excess

// deliberately allow unknown keys with an index signature
interface OpenOptions { width: number; [extra: string]: unknown }
function render2(o: OpenOptions) {}
render2({ width: 10, color: "red" });  // OK — index signature absorbs it
excess check applies to direct args; an index signature opts out
TypeScript · example 4 of 4 — spreads and union targets
// spreading produces a non-fresh object => no excess check
const c: Options = { ...{ width: 10 }, heigth: 5 };   // no error (heigth slips in)

type Shape = { circle: number } | { square: number };
const ok: Shape = { square: 4 };
const mixed: Shape = { circle: 1, square: 2 };  // allowed: each key exists in SOME member
// const bad: Shape = { triangle: 3 };          // Error: in NO member
spread + union targets weaken the excess check (key-in-some-member rule)
Common Mistakes:
❌ Believing TypeScript always rejects extra properties — it only checks fresh literals; a variable of the same shape assigns fine.
✅ Keep config/options as inline literals at the call site so you still get the typo protection the freshness check provides.
❌ Adding [k: string]: unknown just to silence an “excess property” error, which then permits any misspelling forever.
✅ Model the genuinely-optional extras explicitly (a wider type or a real index signature you intend), don’t blanket-disable the check.
Key Rules:
  • Excess property checks fire on fresh object literals assigned/passed to a typed target.
  • Assigning through a variable (or spreading) loses freshness → only structural compatibility is checked.
  • Union targets accept a key present in any member; reject only keys present in none.
  • An index signature legitimately opts a type out of the check — use it intentionally, not to mute typos.
Practice Exercises:
1Write a function configure(o: { mode: "a" | "b"; retries?: number }) and demonstrate three call sites: one that errors on a typo, one that passes via an intermediate variable, and one that passes via a spread.
2Write a generic Exact<T, Shape> helper type that rejects extra properties even through a variable (hint: map over keyof T and force unknown keys to never).
3Given type Theme = { color: string } | { gradient: string }, predict and then verify which of { color: "x", gradient: "y" } and { shadow: "z" } the compiler accepts, and explain why.
Connections: Freshness is the counterweight to the structural typing that makes Object.keys (Concept 1) widen. The Exact<T> exercise reuses key-remapping and the non-distributive equality tricks from Distributive Conditional Types (Day 24). Index signatures here lead directly into Concept 4.
3. typeof Type Queries & Deriving Types from Values Inference
The single source of truth should be the value — a default config, a const list of roles, an action map — with the type derived from it, never hand-duplicated. The typeof type query (plus as const, keyof, and indexed access) is the toolkit that turns a runtime value into a type that can never drift out of sync.

In a type position, typeof x is the type query operator: it yields the static type of the binding x — completely distinct from the runtime typeof operator that returns a string like "object". Combine it with as const to capture literal types instead of widened string/number, with keyof typeof x to get the union of keys, and with (typeof arr)[number] to get an array’s element-type union.

This is the backbone of “define once, derive everywhere.” A const object becomes both runtime data and the type of its keys/values; a const tuple becomes an enum-like union without an actual enum. For classes, typeof Cls is the constructor type while InstanceType<typeof Cls> is the instance; for whole modules, typeof import("...") gives the module’s shape.

1
Type-position typeof x reads the static type of a value binding.
2
Add as const to freeze literals; keyof typeof x gives the key union.
3
(typeof arr)[number] extracts element types; InstanceType<typeof Cls> the instance.
TypeScript · example 1 of 4 — const object → keys and values
const ROLES = { admin: 0, editor: 1, viewer: 2 } as const;

type Roles    = typeof ROLES;             // { readonly admin: 0; readonly editor: 1; readonly viewer: 2 }
type RoleName = keyof typeof ROLES;       // "admin" | "editor" | "viewer"
type RoleId   = (typeof ROLES)[RoleName]; // 0 | 1 | 2
RoleName = "admin" | "editor" | "viewer"; RoleId = 0 | 1 | 2
TypeScript · example 2 of 4 — const tuple → element union (enum-free)
const METHODS = ["GET", "POST", "PUT", "DELETE"] as const;

type Method = (typeof METHODS)[number];   // "GET" | "POST" | "PUT" | "DELETE"

function request(m: Method, url: string) {}
request("GET", "/x");      // ok
// request("PATCH", "/x"); // Error: not assignable to Method
Method = "GET" | "POST" | "PUT" | "DELETE" derived from the array
TypeScript · example 3 of 4 — derive a config type from its default
const defaultConfig = { retries: 3, timeout: 1000, verbose: false };

type Config = typeof defaultConfig;
// { retries: number; timeout: number; verbose: boolean }

function init(overrides: Partial<Config>): Config {
  return { ...defaultConfig, ...overrides };
}
init({ timeout: 5000 });   // type-checked against the derived Config
Config tracks defaultConfig automatically — no duplicated interface
TypeScript · example 4 of 4 — classes and modules
class Service { constructor(public name: string) {} }

type Ctor     = typeof Service;                 // new (name: string) => Service
type Instance = InstanceType<typeof Service>;   // Service

function makeAll(C: typeof Service, names: string[]) {
  return names.map((n) => new C(n));            // Service[]
}

// whole-module shape, types only:
// type Lib = typeof import("./lib");
Ctor is the constructor; InstanceType<typeof Service> is Service
Common Mistakes:
❌ Confusing the two typeofs: writing typeof x in an expression returns a runtime string ("number"), it does not give you a type.
✅ Use typeof x only in type positions (after :, in type/keyof); use runtime typeof solely for narrowing.
❌ Forgetting as const, so typeof ROLES widens values to number and (typeof METHODS)[number] collapses to string.
✅ Add as const to the value to preserve literal types before querying them.
Key Rules:
  • Type-position typeof value yields the value’s static type (distinct from runtime typeof).
  • as const + keyof typeof + indexed access turn a value into key/value unions.
  • (typeof arr)[number] is the idiom for a const array’s element union (an enum substitute).
  • typeof Class is the constructor; InstanceType<typeof Class> is the instance type.
Practice Exercises:
1Define const EVENTS = { click: ..., hover: ... } as const and derive an EventName union and an EventHandlerMap type purely with typeof/keyof.
2Build a const PALETTE object and a function color(name: keyof typeof PALETTE) so callers autocomplete the names and a typo fails to compile.
3Combine satisfies with typeof: write const routes = {...} satisfies Record<string, Handler> and derive a RouteKey union that keeps the literal keys.
Connections: This is the everyday application of Literal Types & const Assertions and keyof & Indexed Access, and it pairs naturally with the satisfies operator (derive a type while still validating the value). The const-array→union pattern is the type-safe alternative explored in Const Enum Alternatives.
2Indexing & Template Typing
4. Index Signatures, Record, & noUncheckedIndexedAccess Indexing
Dictionaries are everywhere — { [k: string]: V } and Record<K, V> — but by default TypeScript assumes every key lookup succeeds: dict[k] is typed V even when the key is absent. That single unsoundness is behind a huge share of “cannot read property of undefined” crashes, and the noUncheckedIndexedAccess flag closes it.

An index signature says “any key of this kind maps to V.” Record<K, V> is the same idea, and when K is a finite union the keys become required; when K is string/number it behaves like an open index signature. The hole: indexing always yields V, never V | undefined, so a missing key is invisible to the checker even though it’s undefined at runtime. The same applies to array element access by number.

Turning on noUncheckedIndexedAccess in tsconfig.json makes every index access (and array index) return V | undefined, forcing you to narrow before use. It’s more verbose, but it converts a class of runtime crashes into compile errors. Note it only affects index access — access to declared, named properties stays exactly as precise as before.

1
{ [k: string]: V } and Record<string, V> model open dictionaries of V.
2
By default a lookup is typed V even for missing keys — an unsound convenience.
3
noUncheckedIndexedAccess makes lookups V | undefined, requiring a guard.
TypeScript · example 1 of 4 — index signature and Record are equivalent
type Scores  = { [name: string]: number };
type Scores2 = Record<string, number>;     // same thing

const s: Scores = { ada: 10, lin: 7 };
const known = s.ada;                        // number
s["grace"] = 9;                             // any string key is writable
Scores and Record<string, number> are interchangeable
TypeScript · example 2 of 4 — the default unsoundness
const s: Record<string, number> = { ada: 10 };

const v = s["bob"];   // typed number — but actually undefined at runtime
v.toFixed(2);          // compiles fine... throws "Cannot read properties of undefined"
v: number (a lie) — missing key crashes at runtime
TypeScript · example 3 of 4 — with noUncheckedIndexedAccess: true
// tsconfig.json: "compilerOptions": { "noUncheckedIndexedAccess": true }
const s: Record<string, number> = { ada: 10 };

const v = s["bob"];        // now: number | undefined
// v.toFixed(2);            // Error: 'v' is possibly 'undefined'
if (v !== undefined) v.toFixed(2);   // ok after narrowing
const safe = s["bob"] ?? 0;          // or default it
v: number | undefined — the missing key is now visible to the compiler
TypeScript · example 4 of 4 — finite-key Record stays required; arrays too
type Flags = Record<"a" | "b", boolean>;
const f: Flags = { a: true, b: false };   // both keys REQUIRED (no | undefined on f.a)
const fa = f.a;                            // boolean — named/known key, always precise

const arr = [10, 20, 30];
const x = arr[99];   // with the flag: number | undefined (out-of-range is honest)
declared keys stay precise; open index + array access gain | undefined
Common Mistakes:
❌ Treating dict[key] as guaranteed present and chaining off it (dict[key].method()) — a typo’d or absent key is a runtime crash the default typings hide.
✅ Enable noUncheckedIndexedAccess, then narrow with if, ?., or ?? before using an indexed result.
❌ Reaching for Record<string, T> when the keys are actually a known, finite set — you lose required-key checking and exhaustiveness.
✅ Use a finite key union (Record<"a" | "b", T>) so all keys are required and missing ones are compile errors.
Key Rules:
  • Index signatures and Record<string, V> describe open dictionaries; access is typed V by default.
  • That default is unsound for missing keys — the value is really V | undefined at runtime.
  • noUncheckedIndexedAccess adds | undefined to index/array access, forcing a guard.
  • Named/declared property access is unaffected; a finite-key Record keeps keys required.
Practice Exercises:
1Turn on noUncheckedIndexedAccess and write a groupBy<T>(items, keyFn): Record<string, T[]>; fix every place the compiler now complains about possibly-undefined buckets.
2Write a getOr<T>(dict: Record<string, T>, key: string, fallback: T): T helper that absorbs the | undefined so call sites stay clean.
3Compare Record<Day, number> (with Day = "mon" | ... | "sun") against Record<string, number>: show one bug the finite version catches that the open one does not.
Connections: Record is itself a mapped type (Mapped Type Modifiers, Day 24), and the “always narrow indexed results” discipline is the same boundary stance as Type-Safe localStorage and Type-Safe Configuration. Finite-key Records connect back to keyof & Indexed Access and the typeof-derived unions of Concept 3.
5. Tagged Template Literal Function Typing Functions
Tagged templates power some of the most ubiquitous DSLs — styled.div\`...\`, sql\`...\`, gql\`...\`, i18n keys. Typing the tag function precisely (the TemplateStringsArray first parameter, constrained interpolations, and a meaningful return type) is the difference between a type-safe embedded language and an any-shaped string blob.

A tag is just a function called with a special argument shape: (strings: TemplateStringsArray, ...values: V[]). The strings parameter is a readonly string[] with one extra property, .raw (the un-escaped source segments). The values are the interpolated expressions, in order; there is always exactly one more string segment than there are values.

You get to choose how strict to be. Constrain ...values (e.g. (string | number)[]) to reject illegal interpolations; make the tag generic over a result type so the DSL can carry a row/result shape (a branded query, a typed style object); and, for no-interpolation tags, capture the literal text as a string-literal type to feed template-literal types. The return type is whatever your runtime produces — the point is to make it concrete, never any.

1
First param is TemplateStringsArray (a readonly string[] with .raw).
2
Constrain ...values to control what interpolations are legal.
3
Give a precise (often generic) return type so downstream code stays typed.
TypeScript · example 1 of 4 — the basic tag signature
function tag(strings: TemplateStringsArray, ...values: unknown[]): string {
  return strings.reduce(
    (acc, s, i) => acc + s + (i < values.length ? String(values[i]) : ""),
    "",
  );
}

const name = "Ada";
const out = tag`Hello ${name}, you have ${3} messages`;   // out: string
Hello Ada, you have 3 messages
TypeScript · example 2 of 4 — constraining the interpolations
function css(strings: TemplateStringsArray, ...values: (string | number)[]): string {
  return strings.reduce((a, s, i) => a + s + (values[i] ?? ""), "");
}

css`width: ${10}px; color: ${"red"};`;   // ok — number and string allowed
// css`content: ${{ x: 1 }};`             // Error: object not assignable to string | number
only string|number interpolations compile; objects are rejected
TypeScript · example 3 of 4 — a generic, result-carrying tag
interface Query<Row> { text: string; params: unknown[]; __row?: Row }

function sql<Row = unknown>(
  strings: TemplateStringsArray,
  ...params: unknown[]
): Query<Row> {
  return { text: strings.join("?"), params };
}

const q = sql<{ id: number; email: string }>`
  SELECT id, email FROM users WHERE id = ${42}
`;
// q: Query<{ id: number; email: string }> — the row type rides along
q: Query<{ id: number; email: string }>
TypeScript · example 4 of 4 — cooked vs raw, and literal capture
function inspect(strings: TemplateStringsArray) {
  return { cooked: strings[0], raw: strings.raw[0] };
}
inspect`line\nbreak`;
// { cooked: "line\nbreak" (real newline), raw: "line\\nbreak" (literal backslash-n) }

// capture the literal text of a no-interpolation tag as a string-literal type
function name<S extends string>(strings: readonly [S]): S {
  return strings[0];
}
const route = name`/users/:id`;   // route: "/users/:id"  (literal type, not string)
raw preserves escapes; literal capture yields "/users/:id"
Common Mistakes:
❌ Typing the first parameter as string[] — you lose the .raw property and the readonly guarantee that TemplateStringsArray provides.
✅ Always type it as TemplateStringsArray; reach for .raw when you need the un-escaped source (e.g. regex/path DSLs).
❌ Returning any (or a bare string when the DSL carries structure), erasing all downstream type information from the template result.
✅ Make the tag generic over its result type (a branded Query<Row>, a typed style object) so callers keep precise types.
Key Rules:
  • A tag function is (strings: TemplateStringsArray, ...values: V[]) => R.
  • TemplateStringsArray is a readonly string[] plus .raw; there’s always one more string than value.
  • Constrain ...values to reject illegal interpolations at compile time.
  • Use a generic return type to carry DSL results (rows, styles) instead of collapsing to any.

Tag functions are the runtime entry point for the type-safe SQL Query Builders and i18n patterns seen earlier in the series.
