R1. Function Overloads Functions & Patterns rep 0 · last reviewed 2026-06-02 · 2 days ago
Overloads let a single function have multiple call signatures. Write each overload signature above the implementation signature. The implementation signature must be compatible with all overloads but is not directly callable — callers only see the overload signatures.
Key Pattern 
// Overload signatures (what callers see)
function createElement(tag: "a"): HTMLAnchorElement;
function createElement(tag: "canvas"): HTMLCanvasElement;
function createElement(tag: "input"): HTMLInputElement;
function createElement(tag: string): HTMLElement;
// Implementation signature (must satisfy all overloads)
function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const link = createElement("a");       // HTMLAnchorElement
const canvas = createElement("canvas"); // HTMLCanvasElement
const div = createElement("div");       // HTMLElement
Challenge Exercise

Write an overloaded fetch wrapper: when called with { json: true } it returns Promise<unknown>; when called with { text: true } it returns Promise<string>; when called with no options it returns Promise<Response>. Test that each call site infers the correct return type.

#1 Pitfall: Ordering overloads from most specific to most general. TypeScript resolves overloads top-to-bottom and picks the first match. Putting string before "a" means the literal overload is never reached.
R2. Generic Constraints & Defaults Functions & Patterns rep 0 · last reviewed 2026-06-02 · 2 days ago
T extends SomeType constrains a generic so only compatible types are accepted. T = DefaultType provides a fallback when the caller omits the type argument. Combine both: <T extends Base = SpecificDefault>.
Key Pattern 
// Constraint: T must have a .length property
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

longest("hello", "hi");       // "hello" (string has .length)
longest([1, 2, 3], [4, 5]);   // [1, 2, 3] (array has .length)
// longest(10, 20);            // Error: number has no .length

// Default: T defaults to string if not specified
interface ApiResponse<T = string> {
  data: T;
  status: number;
}
const res: ApiResponse = { data: "ok", status: 200 };        // T = string
const res2: ApiResponse<number> = { data: 42, status: 200 }; // T = number
Challenge Exercise

Write a merge<T extends object, U extends object>(a: T, b: U): T & U function. Then add a third generic K extends keyof (T & U) = keyof (T & U) so the caller can optionally pick which keys appear in the result. Test with two different object types.

#1 Pitfall: Returning a narrower type than the constraint allows. function make<T extends object>(): T { return {}; } fails because {} is not guaranteed to satisfy every T extends object. The constraint is a minimum, not the exact type.
R3. Branded/Opaque Types Functions & Patterns rep 0 · last reviewed 2026-06-02 · 2 days ago
Branded types add a phantom property (e.g., __brand) to a primitive to prevent mixing structurally identical types. type UserId = string & { __brand: "UserId" } makes UserId and OrderId incompatible even though both are strings at runtime.
Key Pattern 
type UserId = string & { readonly __brand: unique symbol };
type OrderId = string & { readonly __brand: unique symbol };

function createUserId(id: string): UserId { return id as UserId; }
function createOrderId(id: string): OrderId { return id as OrderId; }

function getUser(id: UserId): void { /* ... */ }

const uid = createUserId("user_123");
const oid = createOrderId("order_456");
getUser(uid); // OK
// getUser(oid); // Error: OrderId not assignable to UserId
Challenge Exercise

Create branded types Celsius and Fahrenheit (both number-based). Write constructor functions and a toFahrenheit(c: Celsius): Fahrenheit converter. Verify that passing a raw number or a Fahrenheit value to toFahrenheit produces a compile error.

#1 Pitfall: Using a plain string literal for the brand instead of unique symbol. With { __brand: "UserId" }, you can accidentally create the same brand in another file. unique symbol guarantees uniqueness across the codebase.
R4. this Types & Fluent Interfaces Functions & Patterns rep 0 · last reviewed 2026-06-02 · 2 days ago
The this type in a method refers to the type of the calling instance — enabling fluent APIs (method chaining) that preserve the subclass type. Annotate the first parameter as this: ClassName to enforce the calling context.
Key Pattern 
class QueryBuilder {
  private filters: string[] = [];

  where(condition: string): this {
    this.filters.push(condition);
    return this;  // returns 'this', not 'QueryBuilder'
  }

  limit(n: number): this {
    this.filters.push(`LIMIT ${n}`);
    return this;
  }
}

class TypedQueryBuilder extends QueryBuilder {
  select(fields: string[]): this {
    // 'this' refers to TypedQueryBuilder, not QueryBuilder
    return this;
  }
}

// Chaining preserves the subclass type
const q = new TypedQueryBuilder()
  .where("age > 18")   // returns TypedQueryBuilder (not QueryBuilder)
  .select(["name"])     // still TypedQueryBuilder
  .limit(10);           // still TypedQueryBuilder
Challenge Exercise

Write a FormBuilder class with field(name: string, type: string): this and required(): this methods. Create a ValidatedFormBuilder subclass that adds validate(fn: () => boolean): this. Verify that chaining across both classes preserves the subclass type.

#1 Pitfall: Returning the class name instead of this in method signatures: where(...): QueryBuilder breaks chaining in subclasses because the return type narrows to the base class, losing subclass-specific methods.
R5. Async Patterns & Promise Typing Functions & Patterns rep 0 · last reviewed 2026-06-02 · 2 days ago
async functions always return Promise<T>. Use Awaited<T> to unwrap nested promise types. Promise.all() preserves tuple types with as const. Catch blocks receive unknown in TS 4.4+ — narrow with instanceof before accessing properties.
Key Pattern 
// Typed async function with error narrowing
async function fetchUser(id: string): Promise<{ name: string; email: string }> {
  const res = await fetch(`/api/users/${id}`);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

// Promise.all with tuple types
async function loadDashboard() {
  const [user, posts, settings] = await Promise.all([
    fetchUser("123"),                          // Promise<{ name; email }>
    fetch("/api/posts").then(r => r.json()),   // Promise<any>
    fetch("/api/settings").then(r => r.json()),// Promise<any>
  ] as const);
  // user: { name: string; email: string }
}

// Awaited unwraps nested promises
type A = Awaited<Promise<Promise<string>>>; // string
Challenge Exercise

Write a retry<T>(fn: () => Promise<T>, attempts: number): Promise<T> function that retries the async operation up to attempts times, with unknown in the catch block properly narrowed. Type the function so that calling retry(() => fetchUser("1"), 3) infers Promise<{ name: string; email: string }>.

#1 Pitfall: Forgetting that catch receives unknown (not Error). Writing catch (e) { console.log(e.message) } is a type error because unknown has no .message. Always narrow with e instanceof Error first.
R6. never & Exhaustiveness Advanced Type System rep 1 · last reviewed 2026-06-03 · 1 day ago
The never type represents values that can never occur — bottom of the type hierarchy. Use it for exhaustiveness checking in switch/if-else chains: assign the narrowed value to never, and the compiler errors if you miss a case. Functions that always throw or loop forever return never.
Key Pattern 
type Shape =
  | { kind: "circle"; r: number }
  | { kind: "square"; s: number }
  | { kind: "triangle"; base: number; height: number };

function assertNever(x: never): never {
  throw new Error(`Unexpected value: ${JSON.stringify(x)}`);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":   return Math.PI * shape.r ** 2;
    case "square":   return shape.s ** 2;
    case "triangle": return 0.5 * shape.base * shape.height;
    default:         return assertNever(shape);
  }
}
Challenge Exercise

Define a type Action = { type: "increment" } | { type: "decrement" } | { type: "reset"; value: number }. Write a reducer function with exhaustive switch using assertNever. Add a fourth action type "set" and confirm the compiler flags the missing case before you add the handler.

#1 Pitfall: Forgetting the default exhaustiveness guard entirely. Without it, adding a new variant to the union compiles silently and falls through to undefined at runtime. Always include the assertNever check in the default branch.
4 Week 4 Finish: Modules & Project Structure
1. Project References Modules & Project Structure
As codebases grow, a single tsconfig becomes a bottleneck — every edit recompiles everything. Project references split your code into independently compilable units with dependency tracking, cutting rebuild times from minutes to seconds in monorepos.

Project references let you structure a TypeScript codebase as a graph of sub-projects, each with its own tsconfig.json. TypeScript then compiles only the parts that changed, caching results via .tsbuildinfo files. This is the foundation of fast builds in monorepos.

  • 2026-07-04 (Day 31)
  • "composite": true marks a project as referenceable — requires declaration and enforces strict file inclusion
  • "references": [{ "path": "./packages/lib" }] declares dependencies between projects
  • tsc --build (or tsc -b) compiles the project graph in dependency order
  • Incremental compilation uses .tsbuildinfo to skip unchanged projects
  • A root tsconfig.json with only references orchestrates the entire build
TypeScript — Enabling composite in a Sub-Project
// packages/shared/tsconfig.json
{
  "compilerOptions": {
    "composite": true,           // Required for project references
    "declaration": true,         // Auto-enabled by composite
    "declarationMap": true,      // Go-to-definition works across projects
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler"
  },
  "include": ["src/**/*.ts"]
}

// packages/shared/src/index.ts
export interface User {
  id: string;
  name: string;
  email: string;
}

export function validateEmail(email: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}

// After `tsc -b`, this generates:
// packages/shared/dist/index.js     (compiled JS)
// packages/shared/dist/index.d.ts   (type declarations)
// packages/shared/dist/index.d.ts.map (declaration source map)
TypeScript — Referencing a Sub-Project
// packages/api/tsconfig.json
{
  "compilerOptions": {
    "composite": true,
    "declaration": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler"
  },
  "include": ["src/**/*.ts"],
  "references": [
    { "path": "../shared" }     // api depends on shared
  ]
}

// packages/api/src/handler.ts
// Import from the shared project (uses its .d.ts output)
import { User, validateEmail } from "../../shared/src";

export function createUser(name: string, email: string): User | null {
  if (!validateEmail(email)) return null;
  return { id: crypto.randomUUID(), name, email };
}

// The import resolves to shared's declaration files when built.
// In development, editors use the source files directly for
// go-to-definition (thanks to declarationMap).
TypeScript — Root tsconfig.json (Orchestrator)
// tsconfig.json (at monorepo root)
{
  "files": [],                  // Don't compile any files directly
  "references": [
    { "path": "./packages/shared" },
    { "path": "./packages/api" },
    { "path": "./packages/web" }
  ]
}

// Build commands:
// tsc -b                       Build everything in dependency order
// tsc -b --watch               Incremental watch mode
// tsc -b --clean               Remove all build outputs
// tsc -b --force               Rebuild everything from scratch
// tsc -b packages/api          Build only api and its dependencies

// Build order is automatic:
// 1. shared (no dependencies)
// 2. api (depends on shared)
// 3. web (depends on shared)
TypeScript — Incremental Compilation
// When you run `tsc -b`, TypeScript creates .tsbuildinfo files:
// packages/shared/tsconfig.tsbuildinfo
// packages/api/tsconfig.tsbuildinfo

// These files store a hash of each source file's contents.
// On the next build, TypeScript compares hashes and skips
// unchanged projects entirely.

// Example: you edit packages/api/src/handler.ts
// tsc -b will:
//   1. Check shared -> unchanged, skip (< 1ms)
//   2. Check api -> handler.ts changed, recompile api only
//   3. Check web -> unchanged if it doesn't depend on api, skip

// For a 100-file monorepo, this turns a 15-second full build
// into a 1-2 second incremental build.

// Gotcha: add .tsbuildinfo to .gitignore
// They're machine-specific cache files:
// *.tsbuildinfo
TypeScript — Real-World Monorepo Structure
// Typical monorepo layout:
//
// my-app/
// ├── tsconfig.json              (root: references only)
// ├── packages/
// │   ├── shared/                (types, utils, no dependencies)
// │   │   ├── tsconfig.json      (composite: true)
// │   │   └── src/
// │   ├── database/              (depends on shared)
// │   │   ├── tsconfig.json      (references: [shared])
// │   │   └── src/
// │   ├── api/                   (depends on shared, database)
// │   │   ├── tsconfig.json      (references: [shared, database])
// │   │   └── src/
// │   └── web/                   (depends on shared)
// │       ├── tsconfig.json      (references: [shared])
// │       └── src/

// When to split into separate tsconfigs:
// 1. Packages have different target environments (Node vs browser)
// 2. Parts of the codebase have different strict settings
// 3. Build times exceed ~10 seconds for a full rebuild
// 4. You want independent npm packages from a monorepo
// 5. Teams own different parts of the codebase
Common Mistakes
❌ Forgetting "composite": true in a referenced project:
error TS6306: Referenced project must have setting "composite": true
✅ Every project in the references array must have "composite": true in its own tsconfig. This also automatically requires "declaration": true.
❌ Running tsc instead of tsc -b on a project with references:
Regular tsc ignores the references array and compiles only the current project
✅ Always use tsc --build (or tsc -b) when working with project references. It processes the dependency graph correctly.
Key Rules
  • "composite": true + "declaration": true marks a project as referenceable — every referenced project needs both
  • Use tsc -b (not plain tsc) to compile the project graph in dependency order with incremental caching
  • Split into project references when builds are slow, environments differ, or teams need independent compilation units
Exercises
1 Create a mini monorepo with two packages: packages/types (exports interfaces) and packages/app (imports from types). Set up composite: true and references in each tsconfig. Verify that tsc -b compiles both in the correct order and generates .d.ts files.
2 Add a third package packages/utils that depends on types. Make app depend on both types and utils. Edit a file in types and run tsc -b — observe that all three packages rebuild. Edit a file in utils and confirm only utils and app rebuild (not types).
3 Set up a root tsconfig.json with "files": [] and "references" pointing to all three packages. Run tsc -b --watch and verify that editing any file triggers only the necessary rebuilds. Check the .tsbuildinfo file size and confirm it's in .gitignore.
Connections: Project references build on the tsconfig Deep Dive from Day 4. The declaration: true requirement ties directly to Declaration Files. The monorepo pattern connects to Package Publishing (next) — each referenced project can become an independently published npm package.
2. Package Publishing Modules & Project Structure
Publishing a TypeScript library that works seamlessly in both CJS and ESM environments, with proper type declarations, is one of the most error-prone tasks in the ecosystem. Getting exports, types, and dual-format builds right saves your consumers hours of debugging.

A well-published TypeScript package needs three things: compiled JavaScript (CJS, ESM, or both), type declarations (.d.ts), and correct package.json metadata so that Node.js, bundlers, and TypeScript all resolve the right files.

  • "exports" field in package.json: modern conditional exports (replaces "main" and "module")
  • Dual CJS/ESM: separate .cjs/.mjs entry points or separate directories
  • "types" field: points to the root .d.ts file for TypeScript consumers
  • "typesVersions": maps subpath imports to declaration files (pre-exports era)
  • "declaration": true, "declarationMap": true: generate .d.ts files alongside JS
  • "peerDependencies": declare version ranges your library is compatible with
TypeScript — Modern package.json with Conditional Exports
// package.json
{
  "name": "@myorg/utils",
  "version": "1.0.0",
  "type": "module",                // Default to ESM
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",    // TypeScript looks here first
      "import": "./dist/index.mjs",    // ESM consumers
      "require": "./dist/index.cjs"    // CJS consumers
    },
    "./math": {
      "types": "./dist/math.d.ts",
      "import": "./dist/math.mjs",
      "require": "./dist/math.cjs"
    },
    "./package.json": "./package.json"
  },
  "files": ["dist"],               // Only ship dist/ to npm
  "scripts": {
    "build": "tsup src/index.ts src/math.ts --format cjs,esm --dts",
    "prepublishOnly": "npm run build"
  }
}

// Consumers can now import:
// import { debounce } from "@myorg/utils";       // -> dist/index.mjs
// import { clamp } from "@myorg/utils/math";     // -> dist/math.mjs
// const { debounce } = require("@myorg/utils");  // -> dist/index.cjs
TypeScript — Dual CJS/ESM Build Setup
// tsconfig.esm.json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "module": "ESNext",
    "outDir": "./dist/esm",
    "declaration": true,
    "declarationDir": "./dist/types"
  }
}

// tsconfig.cjs.json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "module": "CommonJS",
    "outDir": "./dist/cjs",
    "declaration": false       // Types generated once from ESM build
  }
}

// Build script (in package.json):
// "build": "tsc -p tsconfig.esm.json && tsc -p tsconfig.cjs.json"

// Or use a build tool like tsup (much simpler):
// "build": "tsup src/index.ts --format cjs,esm --dts --clean"

// tsup generates:
// dist/
//   index.mjs      (ESM)
//   index.cjs      (CJS)
//   index.d.ts     (declarations)
//   index.d.mts    (ESM-specific declarations)
TypeScript — Types Field and typesVersions
// package.json — types field (simple case)
{
  "name": "my-lib",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts",     // Root type declaration
  "typesVersions": {
    "*": {
      "math": ["./dist/math.d.ts"],  // Subpath: import { add } from "my-lib/math"
      "string": ["./dist/string.d.ts"]
    }
  }
}

// When to use "types" vs "exports.types":
// - "types": legacy fallback, works with all TS versions
// - "exports.*.types": modern, requires TS 4.7+ and moduleResolution "node16" or "bundler"
// - Best practice: include BOTH for maximum compatibility

// IMPORTANT: "types" condition must come FIRST in exports:
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",  // MUST be first!
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

// If "types" comes after "import", TypeScript may resolve the
// wrong declaration file (or none at all).
TypeScript — Declaration Generation Settings
// tsconfig.json for a library
{
  "compilerOptions": {
    "declaration": true,           // Generate .d.ts files
    "declarationMap": true,        // .d.ts.map for go-to-definition
    "emitDeclarationOnly": false,  // Also emit JS (or true for type-only)
    "declarationDir": "./dist/types", // Separate dir for declarations
    "stripInternal": true,         // Remove @internal-tagged items from .d.ts

    // Control what's exported in .d.ts:
    "isolatedModules": true,       // Ensure each file can be transpiled alone
    "resolveJsonModule": true      // Allow importing .json files
  }
}

// Mark internal APIs with @internal JSDoc tag:
// src/index.ts
export function publicApi(): string {
  return internalHelper();
}

/** @internal */
export function internalHelper(): string {
  return "internal";
}

// With stripInternal: true, the .d.ts will NOT include internalHelper
// Consumers can't import it (TypeScript won't see it)
TypeScript — peerDependencies for Type Compatibility
// package.json for a React component library
{
  "name": "@myorg/ui",
  "peerDependencies": {
    "react": "^18.0.0 || ^19.0.0",
    "react-dom": "^18.0.0 || ^19.0.0"
  },
  "peerDependenciesMeta": {
    "react-dom": { "optional": true }  // Not needed for React Native
  },
  "devDependencies": {
    "react": "^19.0.0",               // Install locally for development
    "react-dom": "^19.0.0",
    "@types/react": "^19.0.0",
    "typescript": "^5.5.0"
  }
}

// Why peerDependencies matter for types:
// If your library exports React components, consumers need compatible
// @types/react. Using peerDependencies ensures:
// 1. Only one copy of React in the bundle (no duplicate React errors)
// 2. Type definitions are compatible (your JSX.Element = their JSX.Element)
// 3. Consumers control which React version they use

// Common pattern: publish types alongside the package
// NOT as a separate @types/* package (that's for DefinitelyTyped only)
Common Mistakes
❌ Putting "types" after "import" in the exports field:
TypeScript resolves conditions top-to-bottom. If "import" comes first, TS may try to use .mjs as the type source.
✅ Always put "types" as the first condition in each exports entry:
"exports": { ".": { "types": "...", "import": "...", "require": "..." } }
❌ Publishing src/ to npm along with dist/:
Consumers may accidentally import from source files, bypassing your build output
✅ Use the "files" field to whitelist only ["dist"]. Run npm pack --dry-run to verify what gets published.
Key Rules
  • Use "exports" with "types" first, then "import" and "require" for each entry point
  • Generate .d.ts files with "declaration": true and ship them alongside compiled JS — never publish raw .ts source
  • Use "peerDependencies" for shared dependencies (React, TypeScript, framework packages) to avoid version conflicts and duplicate types
Exercises
1 Create a minimal TypeScript library with two entry points: src/index.ts (exports a greet function) and src/math.ts (exports add and multiply). Configure package.json with "exports" mapping both "." and "./math" to their compiled outputs with "types" conditions. Run npm pack --dry-run to verify only dist files are included.
2 Set up a dual CJS/ESM build using tsup (or two tsconfig files). Verify that require("my-lib") resolves to the CJS output and import from "my-lib" resolves to the ESM output. Check that .d.ts files are generated for both formats.
3 Add a /** @internal */ tag to a helper function in your library. Enable "stripInternal": true in tsconfig. Rebuild and verify the function is absent from the generated .d.ts file. Then test that consumers cannot import it.
Connections: Package publishing directly builds on Project References — each composite project in a monorepo can be an independently published package. The .d.ts generation relies on Declaration Files from Day 4. Conditional exports interact with Module Systems (ESM vs CJS resolution).

3. Zod & Runtime Validation Runtime Patterns
TypeScript types are erased at runtime — a User type cannot validate an API response. Zod bridges the gap: you define a schema once and get both runtime validation and a TypeScript type via z.infer. It eliminates the "double-definition" problem where types and validators drift apart.

Zod is a TypeScript-first schema validation library. You define schemas that describe the shape of data, then use them to validate unknown input at runtime. The killer feature is z.infer<typeof schema>, which extracts a TypeScript type from any schema — so your type and your validator are always in sync.

  • z.object(), z.string(), z.number(), z.array() — primitive schema builders
  • .parse() throws ZodError on invalid input; .safeParse() returns a result object
  • z.infer<typeof schema> extracts the TypeScript type from a schema
  • .extend(), .merge(), .pick(), .omit() — schema composition
  • .transform() and .refine() — custom transformations and validations
TypeScript — Defining Schemas and Inferring Types
import { z } from "zod";

// Define a schema
const UserSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1).max(100),
  email: z.string().email(),
  age: z.number().int().min(0).max(150),
  role: z.enum(["admin", "editor", "viewer"]),
  tags: z.array(z.string()).default([]),
  bio: z.string().optional(),        // string | undefined
  createdAt: z.coerce.date(),        // coerces string/number to Date
});

// Infer the TypeScript type from the schema
type User = z.infer<typeof UserSchema>;
// {
//   id: string;
//   name: string;
//   email: string;
//   age: number;
//   role: "admin" | "editor" | "viewer";
//   tags: string[];
//   bio?: string | undefined;
//   createdAt: Date;
// }

// The type and the validator are ALWAYS in sync.
// Change the schema -> the type updates automatically.
TypeScript — .parse() vs .safeParse()
import { z } from "zod";

const EmailSchema = z.string().email();

// .parse() — throws ZodError on failure
try {
  const email = EmailSchema.parse("alice@example.com"); // "alice@example.com"
  const bad = EmailSchema.parse("not-an-email");        // throws ZodError!
} catch (err) {
  if (err instanceof z.ZodError) {
    console.log(err.issues);
    // [{ code: "invalid_string", validation: "email", message: "Invalid email", path: [] }]
  }
}

// .safeParse() — returns a discriminated union (never throws)
const result = EmailSchema.safeParse("alice@example.com");
if (result.success) {
  console.log(result.data); // "alice@example.com" (typed as string)
} else {
  console.log(result.error.issues); // ZodIssue[]
}

4. Error Handling Patterns Runtime Patterns
JavaScript's try/catch loses type information — the catch variable is unknown, and nothing in the function signature tells callers what errors to expect. TypeScript patterns like Result<T, E> and tagged error unions make error handling explicit, composable, and exhaustively checkable at compile time.

There are two schools of error handling in TypeScript: throw-based (traditional try/catch) and return-based (Result types). Each has trade-offs. The key insight is that TypeScript's type system can enforce exhaustive error handling when errors are values, not thrown exceptions.

  • Result<T, E>: { ok: true; data: T } | { ok: false; error: E } — errors in the return type
  • Tagged error types: discriminated unions for different error kinds
  • never for exhaustive error handling in switch statements
  • Custom error classes extending Error with typed properties
  • unknown in catch blocks (TS 4.4+) requires narrowing before use
TypeScript — The Result Type Pattern
// Define the Result type
type Result<T, E> =
  | { ok: true; data: T }
  | { ok: false; error: E };

// Helper constructors
function Ok<T>(data: T): Result<T, never> {
  return { ok: true, data };
}

function Err<E>(error: E): Result<never, E> {
  return { ok: false, error };
}

// Usage: parsing a config file
function parseConfig(raw: string): Result<Config, ParseError> {
  try {
    const parsed = JSON.parse(raw);
    if (!parsed.host) return Err({ kind: "missing_field", field: "host" });
    if (!parsed.port) return Err({ kind: "missing_field", field: "port" });
    return Ok({ host: parsed.host, port: parsed.port });
  } catch {
    return Err({ kind: "invalid_json" });
  }
}

interface Config { host: string; port: number }
type ParseError =
  | { kind: "invalid_json" }
  | { kind: "missing_field"; field: string };

// Caller MUST handle both cases (TypeScript enforces this)
const result = parseConfig('{"host":"localhost","port":3000}');
if (result.ok) {
  console.log(`Connecting to ${result.data.host}:${result.data.port}`);
} else {
  console.log(`Config error: ${result.error.kind}`);
}
Connecting to localhost:3000

// Define specific error types with a discriminant
type AppError =

  throw new Error(`Unhandled: ${JSON.stringify(x)}`);
}

TypeScript — Custom Error Classes
// Base application error with typed code
class AppError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly statusCode: number = 500,
    public readonly details?: Record<string, unknown>
  ) {
    super(message);
    this.name = this.constructor.name;
    // Fix prototype chain (needed for instanceof to work with TS)
    Object.setPrototypeOf(this, new.target.prototype);
  }
}

class NotFoundError extends AppError {
  constructor(resource: string, id: string) {
    super(`${resource} ${id} not found`, "NOT_FOUND", 404, { resource, id });
  }
}

class ValidationError extends AppError {
  constructor(
    public readonly field: string,
    message: string
  ) {
    super(message, "VALIDATION", 400, { field });
  }
}

// Usage with instanceof narrowing
function handleRequest() {
  try {
    throw new NotFoundError("User", "abc-123");
  } catch (err: unknown) {
    if (err instanceof NotFoundError) {
      console.log(err.statusCode, err.details);  // 404, { resource: "User", id: "abc-123" }
    } else if (err instanceof ValidationError) {
      console.log(err.field, err.message);
    } else if (err instanceof Error) {
      console.log("Unexpected:", err.message);
    } else {
      console.log("Non-Error thrown:", err);
    }
  }
}

TypeScript — Safe catch with unknown (TS 4.4+)
// Before TS 4.4: catch (err) gave err: any
// After TS 4.4: catch (err) gives err: unknown (with useUnknownInCatchVariables)

// Problem: you can't access .message on unknown
async function fetchData(url: string): Promise<string> {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.text();
  } catch (err: unknown) {
    // err.message;  // Error: 'err' is of type 'unknown'

    // Pattern 1: instanceof check
    if (err instanceof Error) {
      throw new Error(`Fetch failed: ${err.message}`, { cause: err });
    }

    // Pattern 2: type predicate helper
    throw new Error(`Fetch failed: ${String(err)}`);
  }
}

// Reusable error narrowing helper
function toError(err: unknown): Error {
  if (err instanceof Error) return err;
  if (typeof err === "string") return new Error(err);
  if (typeof err === "object" && err !== null && "message" in err) {
    return new Error(String((err as { message: unknown }).message));
  }
  return new Error(`Unknown error: ${JSON.stringify(err)}`);
}

// Usage: always safe
try {
  throw "a raw string"; // sadly, some code throws non-Errors
} catch (err: unknown) {
  const error = toError(err);
  console.log(error.message);
}

TypeScript — When to Throw vs Return Result
// THROW: for truly exceptional conditions
// - Programmer errors (null pointer, out of bounds)
// - Unrecoverable failures (database connection lost)
// - Framework boundaries (Express middleware, React error boundaries)
function divideStrict(a: number, b: number): number {
  if (b === 0) throw new Error("Division by zero"); // programmer error
  return a / b;
}

// RETURN Result: for expected business failures
// - Validation errors (user input)
// - Resource not found
// - Permission denied
// - Any error the caller is expected to handle
function divide(a: number, b: number): Result<number, "division_by_zero"> {
  if (b === 0) return Err("division_by_zero");
  return Ok(a / b);
}

// The type signature DOCUMENTS the possible errors:
// divide returns: { ok: true, data: number } | { ok: false, error: "division_by_zero" }
// Callers know exactly what can go wrong without reading the implementation.

type Result<T, E> = { ok: true; data: T } | { ok: false; error: E };
function Ok<T>(data: T): Result<T, never> { return { ok: true, data }; }
function Err<E>(error: E): Result<never, E> { return { ok: false, error }; }

// Result pattern shines in pipelines:
const result = divide(10, 2);
if (result.ok) {
  console.log(`Result: ${result.data}`);
}
Result: 5
Common Mistakes
❌ Accessing err.message directly in a catch block:
catch (err) { console.log(err.message); } — err is unknown, not Error
✅ Always narrow with instanceof Error first:
catch (err: unknown) { if (err instanceof Error) console.log(err.message); }
❌ Using Result for truly exceptional conditions (out of memory, stack overflow):
These can't be caught or returned meaningfully
✅ Use Result for expected business errors (not found, validation, permission). Let truly exceptional conditions throw and crash.
Key Rules
  • Result<T, E> makes errors part of the return type — callers can't ignore them, and the compiler enforces handling
  • Use unknown in catch blocks and narrow with instanceof before accessing properties — never assume the caught value is an Error
  • Throw for programmer errors and unrecoverable failures; return Result for expected business failures that callers should handle

5. Discriminated Unions in Practice Runtime Patterns
Discriminated unions are the backbone of type-safe state management in TypeScript. They model real-world states (loading, success, error), event systems, and message protocols so that the compiler prevents impossible states and guarantees exhaustive handling — turning runtime crashes into compile-time errors.

A discriminated union is a union of object types that share a common literal property (the "discriminant" or "tag"). TypeScript uses this tag to narrow the type in switch statements, enabling type-safe access to properties that only exist in specific variants. This pattern replaces boolean flags, optional properties, and stringly-typed APIs.

  • State machines: { status: "loading" } | { status: "success"; data: T } | { status: "error"; error: E }
  • Event systems: action types with discriminated payloads (Redux pattern)
  • API responses: success/error variants with different payloads
  • Exhaustive matching: switch + never ensures all variants are handled
  • Each variant carries only the data relevant to that state — no optional property soup
TypeScript — Fetch State Machine
// Model every possible state of an async operation
type FetchState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T; fetchedAt: Date }
  | { status: "error"; error: Error; retryCount: number };

function renderUserProfile(state: FetchState<{ name: string; email: string }>): string {
  switch (state.status) {
    case "idle":
      return "Click to load profile";
    case "loading":
      return "Loading...";
    case "success":
      // state.data is typed as { name: string; email: string }
      return `${state.data.name} (${state.data.email})`;
    case "error":
      // state.error and state.retryCount are available
      return `Error: ${state.error.message} (retried ${state.retryCount} times)`;
  }
  // No default needed: TypeScript knows all cases are covered
  // If you add a new status, the function won't compile until you handle it
}

// interface BadState { loading: boolean; data?: User; error?: Error; }
// - data and error can BOTH be set (impossible state)
// - loading can be false while data is undefined (ambiguous)
// - No compiler help when you add new states

TypeScript — Redux-Style Action Pattern
// Define actions as a discriminated union
type CounterAction =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "increment_by"; amount: number }
  | { type: "reset"; initialValue?: number };

interface CounterState {
  count: number;
  lastAction: string;
}

function counterReducer(state: CounterState, action: CounterAction): CounterState {
  switch (action.type) {
    case "increment":
      return { count: state.count + 1, lastAction: "increment" };
    case "decrement":
      return { count: state.count - 1, lastAction: "decrement" };
    case "increment_by":
      // action.amount is available (only exists on this variant)
      return { count: state.count + action.amount, lastAction: `+${action.amount}` };
    case "reset":
      return { count: action.initialValue ?? 0, lastAction: "reset" };
    default: {
      const _exhaustive: never = action;
      return _exhaustive;
    }
  }
}

let state: CounterState = { count: 0, lastAction: "init" };
state = counterReducer(state, { type: "increment" });
state = counterReducer(state, { type: "increment_by", amount: 10 });

// Typed API responses with discriminated unions
type ApiResponse<T> =
  | { status: "success"; data: T; meta: { requestId: string; duration: number } }
  | { status: "error"; code: number; message: string; details?: string[] }
  | { status: "rate_limited"; retryAfter: number };

async function callApi<T>(url: string): Promise<ApiResponse<T>> {
  const res = await fetch(url);

  if (res.status === 429) {
    return {
      status: "rate_limited",
      retryAfter: parseInt(res.headers.get("Retry-After") ?? "60", 10),
    };
  }

  if (!res.ok) {
    return {
      status: "error",
      code: res.status,
      message: res.statusText,
    };
  }

  return {
    status: "success",
    data: await res.json() as T,
    meta: {
      requestId: res.headers.get("X-Request-Id") ?? "unknown",
      duration: 0,
    },
  };
}

      break;

TypeScript — Form State Machine
// Model a form lifecycle with discriminated unions
type FormState<T> =
  | { phase: "editing"; values: Partial<T>; errors: Record<string, string> }
  | { phase: "submitting"; values: T }
  | { phase: "submitted"; values: T; response: { id: string } }
  | { phase: "failed"; values: T; error: string };

type UserForm = { name: string; email: string };

type FormAction<T> =
  | { type: "update_field"; field: keyof T; value: string }
  | { type: "submit" }
  | { type: "submit_success"; response: { id: string } }
  | { type: "submit_failure"; error: string }
  | { type: "reset" };

function formReducer(
  state: FormState<UserForm>,
  action: FormAction<UserForm>
): FormState<UserForm> {
  switch (action.type) {
    case "update_field":
      if (state.phase !== "editing") return state;  // Can only edit in editing phase
      return {
        ...state,
        values: { ...state.values, [action.field]: action.value },
      };
    case "submit":
      if (state.phase !== "editing") return state;
      const values = state.values as UserForm;      // Assume valid for brevity
      return { phase: "submitting", values };
    case "submit_success":
      if (state.phase !== "submitting") return state;
      return { phase: "submitted", values: state.values, response: action.response };
    case "submit_failure":
      if (state.phase !== "submitting") return state;
      return { phase: "failed", values: state.values, error: action.error };
    case "reset":
      return { phase: "editing", values: {}, errors: {} };
  }
}

// State transitions are explicit and type-safe:
// editing -> submitting -> submitted
//                       -> failed -> editing (via reset)

❌ Using optional properties instead of discriminated unions:
interface State { loading: boolean; data?: User; error?: Error }
Allows impossible states: { loading: true, data: user, error: err }

  • Always include an exhaustiveness check (default: assertNever(x)) so adding a new variant forces you to handle it everywhere
