R1. Discriminated Unions in Practice Runtime Patterns 14 days since review · rep 2 → 3
A discriminated (tagged) union gives every member a common literal property — the discriminant. Switching (or branching) on that tag narrows the value to exactly one member, and the compiler then knows precisely which other fields exist. It is the everyday workhorse for modelling states, events, and results.
Key Pattern 
type Msg =
  | { type: "text";   body: string }
  | { type: "image";  url: string; alt: string }
  | { type: "system"; code: number };

function render(m: Msg): string {
  switch (m.type) {
    case "text":   return m.body;            // only body is here
    case "image":  return `[img ${m.alt}]`;  // url/alt available
    case "system": return `#${m.code}`;
  }
}
each case narrows to one member; wrong-branch field access is an error
Challenge Exercise

Model a FetchState<T> union with { status: "idle" } | { status: "loading" } | { status: "success"; data: T } | { status: "error"; error: string }. Write a reducer that transitions between them and a render that reads data only in the success branch. Confirm reading state.data in the loading branch is a compile error, and add an assertNever default so a new status forces a new case.

Pitfall: Using a non-literal discriminant (a boolean flag, or overlapping shapes distinguished only by an optional field). The compiler can only narrow a union cleanly when the tag is a literal present on every member; otherwise you fall back to fragile in checks.
R2. Immutability Patterns Runtime Patterns 14 days since review · rep 2 → 3
readonly properties, readonly T[] / ReadonlyArray<T>, Readonly<T>, and as const express immutability at compile time only — they are erased at runtime and are shallow. For real runtime protection you still need Object.freeze or structural copies; to “update”, build a new value with spreads.
Key Pattern 
interface State {
  readonly id: string;
  readonly tags: readonly string[];
}

const s: State = { id: "1", tags: ["a"] };
// s.tags.push("b");          // error: push doesn't exist on readonly array
const next = { ...s, tags: [...s.tags, "b"] };   // immutable "update"

const CONFIG = { retries: 3, mode: "fast" } as const;
// CONFIG.retries = 5;        // error: readonly; mode is "fast", not string
readonly blocks mutation at compile time; spread produces a fresh value
Challenge Exercise

Write type DeepReadonly<T> that recursively marks every nested property readonly, and apply it to a nested config object. Then show that plain Readonly<T> leaves config.nested.value writable while DeepReadonly<T> does not. Bonus: explain why Object.freeze is also shallow and what a runtime deep-freeze must do.

Pitfall: Trusting readonly for runtime safety — it vanishes after compilation, so a value handed to untyped JS can still be mutated. And Readonly<T> is one level deep: nested objects/arrays stay mutable unless you recurse.
R3. infer Mastery Advanced Patterns & Performance 14 days since review · rep 2 → 3
infer X appears only inside the extends clause of a conditional type; it captures part of the matched type into a fresh variable usable in the true branch. Multiple infers, recursion, and infer X extends C constraints let you deconstruct arrays, tuples, functions, and promises.
Key Pattern 
type ElementOf<T>   = T extends (infer U)[] ? U : never;
type ReturnOf<F>    = F extends (...a: any[]) => infer R ? R : never;
type First<T extends any[]> = T extends [infer H, ...any[]] ? H : never;
type Unwrap<T>      = T extends Promise<infer V> ? Unwrap<V> : T;

type A = ElementOf<number[]>;          // number
type B = ReturnOf<() => string>;       // string
type C = First<[1, 2, 3]>;             // 1
type D = Unwrap<Promise<Promise<boolean>>>; // boolean
infer extracts the piece you matched; recursion unwraps nested shapes
Challenge Exercise

Write Parameters2<F> that infers a function’s parameter tuple, and LastParam<F> that infers just the last one using [...any[], infer L]. Then build PathParams<S> that, given a route literal like "/users/:id/posts/:postId", infers { id: string; postId: string } via recursive template-literal infer.

Pitfall: Writing infer outside a conditional’s extends position (it’s a syntax error), or forgetting that a conditional over a naked union type parameter distributes — so your single infer can yield a union of results rather than one type.
R4. Worker Threads Typing Ecosystem Deep Dives 10 days since review · rep 2 → 3
node:worker_threads runs CPU-bound work off the event loop, but postMessage / on("message", …) are effectively untyped (the payload is any). Define a message protocol as a discriminated union and wrap both ends so the request/response types are enforced across the boundary.
Key Pattern 
// protocol.ts — shared by main + worker
export type Req = { id: number; kind: "fib"; n: number };
export type Res = { id: number; value: number };

// worker.ts
import { parentPort } from "node:worker_threads";
parentPort!.on("message", (msg: Req) => {
  const value = fib(msg.n);
  parentPort!.postMessage({ id: msg.id, value } satisfies Res);
});

// main.ts
worker.postMessage({ id: 1, kind: "fib", n: 40 } satisfies Req);
worker.on("message", (r: Res) => console.log(r.value));
a shared protocol + satisfies makes both ends type-check the same shapes
Challenge Exercise

Define a Req union with two kinds ("hash" and "compress") and a matching Res union keyed by the same id. Wrap the worker in a typed call(req): Promise<Res> that resolves the right pending promise by id. Explain why the wrapper — not raw postMessage — is where the type safety actually lives.

Pitfall: Typing messages as if class instances or functions survive the trip. Values cross via the structured clone algorithm — methods, prototypes, and closures are dropped — so a type claiming a Date method or a callback made it through is a lie that crashes at runtime.
R5. Migration Strategies: JS → TS Ecosystem Deep Dives 10 days since review · rep 2 → 3
Migrate incrementally: turn on allowJs + checkJs (or // @ts-check per file) to type-check plain JS with JSDoc first, rename leaf modules to .ts before their dependents, and ratchet strictness one flag / one directory at a time rather than flipping strict everywhere on day one.
Key Pattern 
// tsconfig.json — the on-ramp
{ "compilerOptions": {
    "allowJs": true, "checkJs": true,
    "noImplicitAny": false,   // loosen first, tighten later
    "strict": false
} }

// legacy.js — typed via JSDoc, no rename needed yet
// @ts-check
/** @param {string} name @returns {string} */
function greet(name) { return `hi ${name}`; }
// greet(42);  // now flagged even in a .js file
JSDoc + checkJs catches bugs in JS before you commit to renaming
Challenge Exercise

Take a two-file JS module (a util imported by an entrypoint) and migrate it: (1) add allowJs/checkJs and annotate the util with JSDoc; (2) rename the leaf util to .ts and fix the surfaced errors; (3) enable noImplicitAny, then strictNullChecks, noting how many errors each flag adds. Explain why leaf-first ordering minimizes churn.

Pitfall: Setting "strict": true on a large JS codebase immediately — you drown in thousands of errors and stall. Ratchet: enable one strictness flag at a time, or scope strict settings to already-migrated directories, so the error count stays reviewable.
R6. Advanced Utility Type Recipes Type-Level Programming 10 days since review · rep 2 → 3
The built-in utilities are a floor, not a ceiling. Compose mapped + conditional + infer to build reusable helpers — DeepPartial, RequireAtLeastOne, Mutable, PickByValue — that encode real constraints from your domain in one named type.
Key Pattern 
type DeepPartial<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];
};

type RequireAtLeastOne<T, K extends keyof T = keyof T> =
  Omit<T, K> & { [P in K]-?: Required<Pick<T, P>> & Partial<Pick<T, Exclude<K, P>>> }[K];

type PickByValue<T, V> =
  { [K in keyof T as T[K] extends V ? K : never]: T[K] };
DeepPartial recurses; RequireAtLeastOne forces ≥1 key; PickByValue filters by type
Challenge Exercise

Implement Mutable<T> (strips readonly via the -readonly modifier) and RequireOnlyOne<T, K> (exactly one of K). Then write PickByValue<T, Function> to extract only the method keys of an interface, and verify it drops the data properties. Bonus: guard DeepPartial so it does not recurse into arrays or Date.

Pitfall: Recursive utilities that recurse into Function, Date, or arrays without a base case — DeepPartial will happily make a Date’s methods optional or turn T[] into a strange partial. Add guards (T extends (infer U)[], T extends Function) before the general object branch.
1Node.js + TS: Data & Persistence
1. Repository Pattern & Typed Data Access Layer Node.js + TypeScript
Sprinkling raw ORM/SQL calls through your handlers couples business logic to a specific database and leaks persistence shapes into the domain. A repository is a typed interface (find, save, delete…) that hides the store behind a domain-shaped contract — so you can swap Prisma for an in-memory fake in tests, and the compiler enforces the boundary.

Model the contract as a generic Repository<T, ID>. Callers depend on the interface, not the implementation, so a Postgres-backed class and an in-memory test double are interchangeable. Keep the domain type separate from the row type and map between them at the repository edge — that mapping is the one place database columns turn into domain fields.

Return types matter: findById should return T | null (a miss is expected), while save returns the persisted T (with generated id/timestamps filled in). A tiny NewEntity<T> helper (Omit<T, "id">) types the “not yet saved” shape so you can’t forget an id or invent one.

1
Define interface Repository<T, ID> with findById, findAll, save, delete — the contract every store honors.
2
Map row ↔ domain at the repository edge; keep persistence shapes out of the domain type.
3
Provide a real impl (Prisma) and an in-memory fake that share the interface, and inject whichever you need.
TypeScript · example 1 of 4 — the generic contract
interface Repository<T, ID> {
  findById(id: ID): Promise<T | null>;
  findAll(): Promise<T[]>;
  save(entity: NewEntity<T>): Promise<T>;
  delete(id: ID): Promise<void>;
}

interface User { id: string; name: string; email: string }
type NewEntity<T> = Omit<T, "id">;   // "not yet persisted" shape

type UserRepository = Repository<User, string>;
save() takes a User without id; findById() may miss and return null
TypeScript · example 2 of 4 — an in-memory fake for tests
class InMemoryUserRepo implements UserRepository {
  private store = new Map<string, User>();

  async findById(id: string)  { return this.store.get(id) ?? null; }
  async findAll()             { return [...this.store.values()]; }
  async save(u: NewEntity<User>) {
    const user: User = { id: crypto.randomUUID(), ...u };
    this.store.set(user.id, user);
    return user;
  }
  async delete(id: string)    { this.store.delete(id); }
}
implements forces every method + signature to match the interface
TypeScript · example 3 of 4 — a Prisma-backed impl mapping row → domain
class PrismaUserRepo implements UserRepository {
  constructor(private db: PrismaClient) {}

  async findById(id: string) {
    const row = await this.db.user.findUnique({ where: { id } });
    return row ? this.toDomain(row) : null;
  }
  async save(u: NewEntity<User>) {
    const row = await this.db.user.create({ data: u });
    return this.toDomain(row);
  }
  // ...findAll / delete
  private toDomain(r: { id: string; name: string; email: string }): User {
    return { id: r.id, name: r.name, email: r.email };
  }
}
the DB row shape is converted to the domain User in one place
TypeScript · example 4 of 4 — a service depends on the interface, not the store
class UserService {
  constructor(private users: UserRepository) {}   // interface, not a class

  async register(name: string, email: string): Promise<User> {
    return this.users.save({ name, email });      // id filled by the repo
  }
}

// prod:  new UserService(new PrismaUserRepo(db));
// test:  new UserService(new InMemoryUserRepo());   // no database needed
swap implementations with zero changes to UserService
Common Mistakes:
❌ Returning the raw ORM row (with DB-only columns, relations, Date vs string drift) straight out of the repository as if it were the domain type.
✅ Map row → domain at the edge so the rest of the app depends on a stable, DB-agnostic User.
❌ Having services new PrismaUserRepo(db) internally — now they can’t be tested without a real database.
✅ Depend on Repository<T, ID> and inject the implementation; tests pass an in-memory fake.
Key Rules:
  • 2026-07-04 (Day 31)
  • A repository is a typed interface; concrete stores implements it so signatures can’t drift.
  • Keep the domain type separate from the row type; convert at the repository boundary.
  • findById returns T | null (misses are expected); use Omit<T, "id"> for unsaved entities.
  • Depend on the interface and inject the impl — that’s what makes an in-memory fake possible.
Practice Exercises:
1Add a findBy(pred: (t: T) => boolean): Promise<T[]> method to the interface and implement it in the in-memory repo. Confirm forgetting it in a class that implements the interface is a compile error.
2Introduce a UserRow type with an extra created_at: Date column and a domain User without it; write the toDomain mapper and show the service never sees created_at.
3Write a generic InMemoryRepo<T extends { id: string }> so the same fake works for any entity; explain the constraint T extends { id: string }.
Connections: This is Generics Basics + interfaces + dependency injection made concrete, and it leans on module boundary types to keep persistence shapes out of the domain. It pairs directly with Prisma/Drizzle typing and sets up the caching layer in Concept 2, which often wraps a repository.
2. Type-Safe Redis Caching (ioredis) Node.js + TypeScript
Redis stores bytes: every value comes back as a string (or null), with no idea what you put in. A thin typed cache wrapper — serialize on write, parse (and optionally validate) on read — turns that stringly-typed API into get<T>(key): Promise<T | null>, so the rest of your code sees real domain types instead of raw strings.

Wrap the client so callers never touch raw commands. set JSON-serializes and attaches a TTL; get returns null for a miss and otherwise JSON.parses into T. Because the string → T step is an assertion (the bytes could be anything), validate with a schema (Zod) on read when the data crosses a trust boundary — then T is a guarantee, not a hope.

A namespaced key builder keeps keys collision-free and typed, and a getOrSet(key, ttl, loader) helper captures the cache-aside pattern in one typed place: return the cached value if present, otherwise run the loader, cache it, and return it — with the loader’s return type flowing through as T.

1
Wrap ioredis: set serializes + sets TTL; get<T> returns T | null after parsing.
2
Validate parsed values with a schema on read when the source isn’t fully trusted.
3
Expose a typed getOrSet (cache-aside) so the loader’s return type becomes the cached type.
TypeScript · example 1 of 4 — a typed cache wrapper
import Redis from "ioredis";

class Cache {
  constructor(private redis: Redis) {}

  async get<T>(key: string): Promise<T | null> {
    const raw = await this.redis.get(key);   // string | null
    return raw === null ? null : (JSON.parse(raw) as T);
  }
  async set<T>(key: string, value: T, ttlSeconds: number): Promise<void> {
    await this.redis.set(key, JSON.stringify(value), "EX", ttlSeconds);
  }
}
get<User>("u:1") is Promise<User | null>; the raw string never escapes
TypeScript · example 2 of 4 — validate on read (parse, don’t assert)
import { z } from "zod";

const UserSchema = z.object({ id: z.string(), name: z.string() });
type User = z.infer<typeof UserSchema>;

async function getUser(cache: Redis, id: string): Promise<User | null> {
  const raw = await cache.get(`user:${id}`);
  if (raw === null) return null;
  return UserSchema.parse(JSON.parse(raw));   // throws if shape drifted
}
the returned User is verified at runtime, not just cast
TypeScript · example 3 of 4 — namespaced, typed key builder
const keys = {
  user:      (id: string) => `user:${id}` as const,
  userPosts: (id: string) => `user:${id}:posts` as const,
};

await cache.set(keys.user("u1"), { id: "u1", name: "Ada" }, 60);
// keys.user(42);   // error: id must be a string
key construction is centralized and its arguments are type-checked
TypeScript · example 4 of 4 — typed cache-aside getOrSet
class Cache {
  // ...get/set from example 1
  async getOrSet<T>(
    key: string, ttl: number, loader: () => Promise<T>,
  ): Promise<T> {
    const hit = await this.get<T>(key);
    if (hit !== null) return hit;
    const value = await loader();     // T inferred from the loader
    await this.set(key, value, ttl);
    return value;
  }
}

const user = await cache.getOrSet(keys.user("u1"), 60, () => repo.findById("u1"));
user is Promise<User | null> — the loader's return type flows through as T
Common Mistakes:
❌ Casting JSON.parse(raw) as User and treating it as verified — a stale or malformed cache entry silently violates the type.
✅ parse with a schema on read for untrusted sources; reserve the bare cast for data you control end-to-end.
❌ Building keys inline as template strings everywhere — typos and collisions slip in and nothing checks the arguments.
✅ Centralize keys in a typed builder so both the format and its inputs are checked.
Key Rules:
  • Redis returns string | null; a wrapper is where serialization and the T type live.
  • get<T> should return T | null — a miss is null, not an error.
  • Parsing is an assertion unless you validate; use a schema on read across trust boundaries.
  • getOrSet captures cache-aside once and infers the cached type from the loader.
Practice Exercises:
1Add del(key: string) and getMany<T>(keys: string[]): Promise<(T | null)[]> using mget; keep the per-element null for misses.
2Make getOrSet take a Zod schema and parse both cached and freshly-loaded values, so the return type is z.infer<typeof schema>.
3Type the key builder so keys.userPosts returns the literal template type `user:${string}:posts`; explain what as const buys you here.
Connections: This is Zod & runtime validation guarding a byte-oriented boundary, plus generics to thread the value type through get/getOrSet. It commonly wraps the repository from Concept 1 (cache-aside over findById) and the key builder reuses template literal types.
3. Job Queues: BullMQ with TypeScript Node.js + TypeScript
Background work — sending email, resizing images, running reports — belongs off the request path. BullMQ (Redis-backed) is the standard Node queue, and its generics let you type a job’s payload, its return value, and its name, so producers and the worker that consumes them can’t disagree about the shape of a job.

Queue<Data, Result, Name> and Worker<Data, Result, Name> share the same three type parameters: the data you enqueue, the value your processor returns, and a union of job names. Parameterize both from one shared type and queue.add(name, data) is checked against the processor that will run it — one source of truth for the job contract.

For a queue that carries several job kinds, make Data a discriminated union keyed by the job name, then narrow inside the worker with a switch on job.name. Typed queue events (completed, failed) round it out, though note the failed callback’s error is a plain Error — the boundary where your typed Result stops.

1
Define the job contract once: Data, Result, and a Name union.
2
Parameterize both Queue and Worker with it so add and the processor agree.
3
For multi-kind queues, make Data a discriminated union and switch on job.name in the worker.
TypeScript · example 1 of 4 — a single-purpose typed queue
import { Queue, Worker } from "bullmq";

interface EmailJob { to: string; subject: string; body: string }
type EmailResult = { messageId: string };

const emailQueue = new Queue<EmailJob, EmailResult, "send">(
  "email", { connection: { host: "localhost", port: 6379 } });

await emailQueue.add("send", { to: "a@b.com", subject: "Hi", body: "..." });
// emailQueue.add("send", { to: "a@b.com" });  // error: subject/body missing
add() checks the payload against EmailJob and the name against "send"
TypeScript · example 2 of 4 — a typed worker/processor
const worker = new Worker<EmailJob, EmailResult, "send">(
  "email",
  async (job) => {
    job.data.to;       // string — typed from EmailJob
    const messageId = await sendEmail(job.data);
    return { messageId };   // must match EmailResult
  },
  { connection: { host: "localhost", port: 6379 } },
);
job.data is EmailJob; the return value is checked against EmailResult
TypeScript · example 3 of 4 — a multi-kind queue as a discriminated union
type Job =
  | { kind: "email";  to: string; subject: string }
  | { kind: "resize"; url: string; width: number };

const jobs = new Queue<Job>("work");

const w = new Worker<Job>("work", async (job) => {
  switch (job.data.kind) {
    case "email":  return sendEmail(job.data);        // to/subject here
    case "resize": return resize(job.data.url, job.data.width);
  }
});
job.data narrows by kind; each branch sees only its own fields
TypeScript · example 4 of 4 — typed queue events
import { QueueEvents } from "bullmq";

worker.on("completed", (job, result: EmailResult) => {
  console.log(`sent ${result.messageId}`);   // result is typed
});
worker.on("failed", (job, err: Error) => {
  console.error(job?.id, err.message);        // err is a plain Error
});
completed carries the typed Result; failed's error is Error, not your type
Common Mistakes:
❌ Creating new Queue("email") untyped, so add("send", anything) compiles and the worker discovers the mismatch at runtime.
✅ Parameterize Queue/Worker from one shared job type so producer and consumer share the contract.
❌ Assuming the payload arrives as rich objects — jobs are JSON-serialized into Redis, so Date/Map/class instances don’t survive.
✅ Type payloads as plain JSON-safe data (strings, numbers, arrays, plain objects) and rehydrate in the worker.
Key Rules:
  • Queue<Data, Result, Name> and Worker<Data, Result, Name> share three type parameters — keep them in sync.
  • Drive both from one shared job type so add and the processor can’t drift.
  • Multi-kind queue? Make Data a discriminated union and switch on job.name/kind.
  • Payloads are JSON-serialized through Redis — keep them plain and JSON-safe.
Practice Exercises:
1Type a reportQueue whose Data is { userId: string; range: [string, string] } and Result is { rows: number }; confirm returning {} from the processor is a type error.
2Convert a two-kind queue to a discriminated union and add an assertNever default in the worker’s switch; add a third kind and confirm it’s flagged.
3Wrap queue.add in a typed enqueue(job: Job) helper that derives the name from job.kind; explain what type safety the wrapper adds over calling add directly.
Connections: The multi-kind queue is discriminated unions (R1) applied to job payloads, and the three-parameter generics echo the schema-as-single-source idea from Fastify. Payloads crossing Redis have the same structured-serialization caveat as worker threads (R4). A worker often calls the repository (Concept 1) and warms the cache (Concept 2).

4. Structured Logging with pino Node.js + TypeScript
console.log emits unstructured strings that are painful to query in production. pino writes fast, structured JSON logs; typed with @types/pino (bundled), it gives you a Logger whose levels, child loggers (per-request context), redaction, and serializers are all checked — so your log shape is as intentional as your API shape.

A pino Logger exposes info/warn/error… that accept a merge object first, then a message: log.info({ userId }, "loaded") emits structured fields alongside the text. logger.child(bindings) returns a logger that stamps those bindings on every line — the idiomatic way to attach a requestId or userId to all logs for one request without threading it through every call.

redact removes sensitive paths (req.headers.authorization, password) from output, and serializers map known objects (an Error, a req) to a compact, consistent shape. Both are typed via pino’s LoggerOptions, so a mistyped option is caught at config time rather than swallowed.

1
Create a typed Logger with pino(options); call log.info({ fields }, "message").
2
Use logger.child({ requestId }) to bind per-request context onto every subsequent line.
3
Configure redact for secrets and serializers for errors/requests — all type-checked options.
TypeScript · example 1 of 4 — a typed logger with structured fields
import pino, { Logger } from "pino";

const log: Logger = pino({ level: "info" });

log.info({ userId: "u1", durationMs: 42 }, "user loaded");
// log.info("just a message");            // also valid
// log.wrong("oops");                     // error: no such method
merge-object-then-message is typed; unknown level methods are errors
TypeScript · example 2 of 4 — child loggers for per-request context
function handler(req: { id: string }, base: Logger) {
  const log = base.child({ requestId: req.id });   // Logger
  log.info("handling request");     // includes requestId automatically
  log.info({ step: "db" }, "query");// requestId + step both emitted
}
child() returns a Logger; bindings appear on every line it writes
TypeScript · example 3 of 4 — redaction & serializers (typed options)
const log = pino({
  redact: ["req.headers.authorization", "password"],
  serializers: {
    err: pino.stdSerializers.err,
    req: (r: { method: string; url: string }) => ({ method: r.method, url: r.url }),
  },
});

log.error({ err: new Error("boom") }, "request failed");
LoggerOptions type-checks redact paths and serializer signatures
TypeScript · example 4 of 4 — a typed context shape for bindings
interface LogContext { requestId: string; userId?: string }

function withContext(base: Logger, ctx: LogContext): Logger {
  return base.child(ctx);   // ctx shape is enforced by the caller
}

const reqLog = withContext(log, { requestId: "r1" });
// withContext(log, { userId: "u1" });   // error: requestId required
wrapping child() with a typed context makes bindings a real contract
Common Mistakes:
❌ Putting the message first and data second (log.info("loaded", { userId })) — pino’s order is merge object, then message, so the fields end up ignored.
✅ Call log.info({ userId }, "loaded") — object first — so the fields are structured, not stringified into the text.
❌ Logging whole request/error objects raw — you leak secrets and emit huge, inconsistent payloads.
✅ Configure redact and serializers so sensitive paths are stripped and known objects log a compact shape.
Key Rules:
  • Log call signature is (mergeObject?, message?) — object first, message second.
  • logger.child(bindings) stamps context on every line — the right tool for per-request fields.
  • redact and serializers are typed LoggerOptions; mistakes surface at config time.
  • Wrap child() behind a typed context interface to make your log fields a real contract.
Practice Exercises:
1Create a base logger and a request-scoped child with { requestId }; log two lines and confirm both carry the id. Then log with an extra { step } field and observe both fields appear.
2Add a redact path for headers.cookie and confirm the value is replaced with [Redacted] in output.

Connections: Child-logger bindings are the observability counterpart to Express request augmentation (attach a requestId in middleware, log it everywhere), and the typed LogContext reuses interfaces + optional properties. Serializers echo the row→domain mapping idea from Concept 1 — convert a rich object to a stable shape at the boundary.
5. gRPC + TypeScript: Schema-First Services Node.js + TypeScript
gRPC defines services in a .proto file — one language-neutral contract for messages and RPCs. A code generator like ts-proto turns that schema into TypeScript interfaces and typed service stubs, so client and server share the same generated types: the schema is the single source of truth, and drift between caller and callee becomes a compile error.

Write the contract in .proto, then run protoc with the ts-proto plugin to emit request/response interfaces and a strongly-typed service definition. You implement the server against the generated ServiceServer type (each RPC is a typed handler), and you call the server through a generated client whose method signatures match the RPCs — no hand-written DTOs to keep in sync.

Streaming RPCs (server-, client-, and bi-directional) are typed too, but mind the boundary: proto int64 maps to string or bigint (not JS number) to avoid precision loss, and proto3 scalar fields are non-optional with defaults, so an “absent” field arrives as 0/"" rather than undefined unless declared optional.

1
Define messages + service in .proto — the neutral contract.
2
Generate TS with protoc + ts-proto: interfaces for messages, a typed service definition.
3
Implement the server against the generated handler types; call it via the generated typed client.
protobuf · example 1 of 4 — the .proto contract
syntax = "proto3";
package users.v1;

message GetUserRequest  { string id = 1; }
message User            { string id = 1; string name = 2; }

service UserService {
  rpc GetUser (GetUserRequest) returns (User);
}
one schema; both client and server generate from it
TypeScript · example 2 of 4 — generated types (ts-proto output, simplified)
// users.ts — GENERATED, do not edit
export interface GetUserRequest { id: string }
export interface User { id: string; name: string }

export interface UserServiceServer {
  getUser(call: { request: GetUserRequest }): Promise<User>;
}
messages become interfaces; the service becomes a typed contract
TypeScript · example 3 of 4 — a typed server implementation
import type { UserServiceServer } from "./users";

const impl: UserServiceServer = {
  async getUser({ request }) {
    request.id;                         // string — from the schema
    return { id: request.id, name: "Ada" };   // must match User
    // return { id: request.id };       // error: name is missing
  },
};
the handler's request + return type are enforced by the generated type
TypeScript · example 4 of 4 — a typed client call
import { UserServiceClient } from "./users";

const client = new UserServiceClient(/* addr, creds */);
const user = await client.getUser({ id: "u1" });   // User
console.log(user.name);
// client.getUser({});         // error: id is required by GetUserRequest
client method signatures mirror the RPCs; request/response are typed
Common Mistakes:
❌ Hand-writing TS interfaces to mirror the .proto — they drift the instant someone edits the schema without regenerating.
✅ Generate types from the .proto (ts-proto) and treat the output as read-only; regenerate on every schema change.
❌ Typing proto int64 as number — values beyond 2^53 lose precision silently.
✅ Let the generator map int64 to string/bigint, and remember proto3 scalars default (to 0/"") rather than being undefined.
Key Rules:
  • The .proto is the single source of truth; TS types are generated, not hand-written.
  • Server implements a generated service type; the client’s methods mirror the RPCs — no manual DTOs.
  • int64 → string/bigint (not number); proto3 scalars carry defaults, not undefined, unless optional.
  • Streaming RPCs are typed too — the generated types describe uni- and bi-directional streams.
Practice Exercises:
1Add a ListUsers RPC returning repeated User; regenerate (or hand-simulate the output) and confirm the response type is User[].
2Add an optional string email = 3; field and show the generated type makes email string | undefined, unlike the non-optional scalars.
3Implement the server handler and a client call, then change the .proto (rename a field) without regenerating; explain exactly which line now fails to compile and why that’s the point.
Connections: This is the schema-as-single-source theme from Fastify/TypeBox and GraphQL codegen, generalized across service boundaries — the generated User is the same domain type your repository (Concept 1) returns. The int64/default caveats mirror the serialization boundaries seen in caching (Concept 2) and job queues (Concept 3).
