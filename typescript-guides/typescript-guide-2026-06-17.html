R1. Discriminated Unions in Practice Advanced Type System rep 1 · 7 days overdue
Use a shared literal type field to narrow unions. The compiler automatically narrows the full union to the specific variant inside each case or if branch.
Key Pattern 
// State machine with discriminated union
type AsyncState<T> =
  | { type: "idle" }
  | { type: "loading" }
  | { type: "success"; data: T }
  | { type: "error"; error: Error };

function renderState<T>(state: AsyncState<T>): string {
  switch (state.type) {
    case "idle":    return "Ready";
    case "loading": return "Loading...";
    case "success": return `Got: ${state.data}`;   // data narrowed
    case "error":   return `Err: ${state.error.message}`; // error narrowed
  }
}
case "success": state is { type: "success"; data: T } case "error": state is { type: "error"; error: Error }
Challenge Exercise

Model a payment processing pipeline with 5 states: pending, authorized (has authCode: string), captured (has amount: number and captureId: string), refunded (has refundId: string and refundedAt: Date), and failed (has reason: string). Write a describePayment function with an exhaustive switch.

#1 Pitfall: Forgetting to handle all cases without an exhaustive never check — the compiler won't warn you about missing branches unless you assign to never in the default.
WRONG vs RIGHT 
// WRONG: optional fields instead of discriminated union
type BadState = {
  loading?: boolean;
  data?: string;
  error?: Error;
};
// Nothing prevents { loading: true, error: new Error() } — contradictory!

// RIGHT: discriminated union
type GoodState =
  | { type: "loading" }
  | { type: "success"; data: string }
  | { type: "error"; error: Error };
// Each variant is self-consistent — impossible to have both data and error
R2. Error Handling Patterns Functional Patterns rep 1 · 7 days overdue
Result<T,E> type: tagged errors with discriminated unions. Replaces thrown exceptions with composable, type-safe error channels.
Key Pattern 
type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function parseJSON(input: string): Result<unknown, SyntaxError> {
  try {
    return { ok: true, value: JSON.parse(input) };
  } catch (e) {
    return { ok: false, error: e as SyntaxError };
  }
}

const result = parseJSON('{"name":"Alice"}');
if (result.ok) {
  console.log(result.value); // unknown (safe to validate next)
} else {
  console.log(result.error.message); // SyntaxError
}
result.ok === true => result.value: unknown result.ok === false => result.error: SyntaxError
Challenge Exercise

Implement a pipe function that takes an initial Result<T, E> and an array of transform functions, each returning a Result. The pipe should short-circuit on the first error and propagate it, accumulating error types in the union.

#1 Pitfall: Using throw inside library code that is supposed to return Result types — callers expect to check .ok, not catch exceptions.
WRONG vs RIGHT 
// WRONG: mixing throw with Result return type
function parseConfig(json: string): Result<Config, ParseError> {
  const data = JSON.parse(json); // THROWS on invalid JSON!
  // Caller expects to check result.ok, but this throws instead
  return { ok: true, value: data as Config };
}

// RIGHT: catch and wrap in Result
function parseConfig(json: string): Result<Config, ParseError> {
  try {
    const data = JSON.parse(json);
    return { ok: true, value: data as Config };
  } catch (e) {
    return { ok: false, error: { type: "parse", message: (e as Error).message } };
  }
}

R4. infer Mastery Advanced Type System rep 1 · 7 days overdue
Extract types from complex structures in conditional types. infer introduces a type variable that TypeScript fills in when the pattern matches.
Key Pattern 
// Extract return type from any function
type ReturnOf<T> = T extends (...args: any[]) => infer R ? R : never;

type A = ReturnOf<() => string>;          // string
type B = ReturnOf<(x: number) => boolean>; // boolean

// Extract the resolved type of a Promise
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;

type C = Awaited<Promise<string>>;                // string
type D = Awaited<Promise<Promise<number>>>;       // number
A = string B = boolean C = string D = number
Challenge Exercise

Write DeepAwaited<T> that extracts the resolved type of arbitrarily nested Promises: DeepAwaited<Promise<Promise<Promise<number>>>> should resolve to number. Then write UnpackArray<T> that extracts the element type from nested arrays: UnpackArray<number[][][]> should be number.

#1 Pitfall: infer position matters — it can only appear in the true branch of a conditional type, and its scope is limited to that branch.
WRONG vs RIGHT 
// WRONG: trying to use infer in the false branch
type Bad<T> = T extends string ? true : T extends infer U ? U : never;
// This "works" but infer U just captures T — pointless

// WRONG: using infer outside a conditional type
// type Extract<T> = infer U; // Syntax error!

// RIGHT: infer in the extends clause, use in the true branch
type GetPromiseValue<T> = T extends Promise<infer V> ? V : T;
type A = GetPromiseValue<Promise<number>>; // number
type B = GetPromiseValue<string>;           // string (fallback)

Type-Safe Builder with Phantom Types Advanced Patterns
Phantom types encode compile-time constraints that have no runtime representation. They let you enforce rules like "this builder requires fields A and B to be set before you can call build()" — all checked at compile time, zero overhead at runtime.

Example 1 — Simple phantom types for units:

A phantom type parameter appears in the type signature but not in the runtime value. This prevents mixing incompatible units:

TypeScript
// The Brand type parameter is "phantom" — it exists only at the type level
type Meters = { readonly _brand: "meters" };
type Feet = { readonly _brand: "feet" };

type Distance<Unit> = number & { readonly __unit: Unit };

function meters(n: number): Distance<Meters> {
  return n as Distance<Meters>;
}

function feet(n: number): Distance<Feet> {
  return n as Distance<Feet>;
}

function addMeters(a: Distance<Meters>, b: Distance<Meters>): Distance<Meters> {
  return (a + b) as Distance<Meters>;
}

const d1 = meters(100);
const d2 = meters(200);
const d3 = feet(50);

const total = addMeters(d1, d2);  // OK: both are Meters
// const bad = addMeters(d1, d3); // ERROR: Feet is not assignable to Meters
d1: Distance<Meters> d2: Distance<Meters> d3: Distance<Feet> total: Distance<Meters>
TypeScript — WRONG
// WRONG: plain numbers allow unit mixing
function addDistances(a: number, b: number): number {
  return a + b; // No protection — could be meters + feet!
}

const mixed = addDistances(100, 50); // Compiles fine, but semantically wrong
No compile-time error, but this is a logic bug — you're adding meters to feet.

Example 2 — Builder that tracks which required fields are set:

We use phantom type parameters to track builder state. The build() method is only available when all required fields have been set:

TypeScript
// Phantom markers — these types are never instantiated at runtime
type Unset = { readonly _state: "unset" };
type Set = { readonly _state: "set" };

// The builder carries phantom type parameters for each required field
interface RequestBuilder<
  HasUrl extends Set | Unset = Unset,
  HasMethod extends Set | Unset = Unset
> {
  url: string;
  method: string;
  headers: Record<string, string>;
  body?: unknown;
}

function createRequestBuilder(): RequestBuilder<Unset, Unset> {
  return { url: "", method: "", headers: {} };
}

function setUrl<M extends Set | Unset>(
  builder: RequestBuilder<Unset, M>,
  url: string
): RequestBuilder<Set, M> {
  return { ...builder, url };
}

function setMethod<U extends Set | Unset>(
  builder: RequestBuilder<U, Unset>,
  method: string
): RequestBuilder<U, Set> {
  return { ...builder, method };
}

function addHeader<U extends Set | Unset, M extends Set | Unset>(
  builder: RequestBuilder<U, M>,
  key: string,
  value: string
): RequestBuilder<U, M> {
  return { ...builder, headers: { ...builder.headers, [key]: value } };
}

// build() requires BOTH phantoms to be Set
function build(builder: RequestBuilder<Set, Set>): Request {
  return new Request(builder.url, {
    method: builder.method,
    headers: builder.headers,
  });
}

// Usage — step by step, type-checked at each stage
const b1 = createRequestBuilder();                    // RequestBuilder<Unset, Unset>
const b2 = setUrl(b1, "https://api.example.com");    // RequestBuilder<Set, Unset>
const b3 = setMethod(b2, "POST");                    // RequestBuilder<Set, Set>
const b4 = addHeader(b3, "Content-Type", "json");    // RequestBuilder<Set, Set>
const req = build(b4);                               // OK: both fields set

// const bad = build(b2);
// ERROR: RequestBuilder<Set, Unset> is not assignable to RequestBuilder<Set, Set>
b1: RequestBuilder<Unset, Unset> b2: RequestBuilder<Set, Unset> b3: RequestBuilder<Set, Set> b4: RequestBuilder<Set, Set> req: Request
TypeScript — WRONG
// WRONG: trying to build before setting all required fields
const incomplete = createRequestBuilder();
const withUrl = setUrl(incomplete, "https://example.com");
// build(withUrl);
// ^^^^^^^^^^^^^^
// Argument of type 'RequestBuilder<Set, Unset>' is not assignable
// to parameter of type 'RequestBuilder<Set, Set>'.
// Types of property '_state' are incompatible.
Argument of type 'RequestBuilder<Set, Unset>' is not assignable to parameter of type 'RequestBuilder<Set, Set>'.

Example 3 — Real-world: Form builder that won't compile until all required fields are configured:

TypeScript
// Track which fields have been configured with a mapped phantom type
type FieldStatus = "configured" | "pending";

interface FormBuilder<
  Fields extends Record<string, FieldStatus>
> {
  _fields: Fields;
  _config: Record<string, FormFieldConfig>;
}

interface FormFieldConfig {
  label: string;
  type: "text" | "email" | "number" | "select";
  required: boolean;
  validation?: (value: unknown) => boolean;
}

// Start with all fields pending
type PendingForm<Keys extends string> = {
  [K in Keys]: "pending";
};

function createForm<Keys extends string>(
  ...fields: Keys[]
): FormBuilder<PendingForm<Keys>> {
  const pending = Object.fromEntries(
    fields.map(f => [f, "pending"])
  ) as PendingForm<Keys>;
  return { _fields: pending, _config: {} };
}

// Configure a field — flips its status from "pending" to "configured"
function configureField<
  Fields extends Record<string, FieldStatus>,
  K extends keyof Fields & string
>(
  form: FormBuilder<Fields>,
  field: K,
  config: FormFieldConfig
): FormBuilder<Omit<Fields, K> & Record<K, "configured">> {
  return {
    _fields: { ...form._fields, [field]: "configured" } as any,
    _config: { ...form._config, [field]: config },
  };
}

// Type that checks ALL fields are "configured"
type AllConfigured<T extends Record<string, FieldStatus>> =
  T[keyof T] extends "configured" ? true : false;

// submit() only works when every field is configured
function submit<Fields extends Record<string, FieldStatus>>(
  form: AllConfigured<Fields> extends true ? FormBuilder<Fields> : never
): void {
  console.log("Submitting form:", form._config);
}

// Usage
const form1 = createForm("name", "email", "age");
// form1: FormBuilder<{ name: "pending"; email: "pending"; age: "pending" }>

const form2 = configureField(form1, "name", { label: "Name", type: "text", required: true });
const form3 = configureField(form2, "email", { label: "Email", type: "email", required: true });
const form4 = configureField(form3, "age", { label: "Age", type: "number", required: false });

submit(form4); // OK: all three fields configured
// submit(form2); // ERROR: not all fields configured — "email" and "age" still pending
form1: FormBuilder<{ name: "pending"; email: "pending"; age: "pending" }> form4: FormBuilder<{ name: "configured"; email: "configured"; age: "configured" }>

Example 4 — Fluent builder API with method chaining:

We can combine phantom types with a class-based builder for a fluent API that feels natural to use:

TypeScript
// Phantom markers
type TRUE = { readonly __flag: true };
type FALSE = { readonly __flag: false };

// EmailBuilder tracks: HasTo, HasSubject, HasBody
class EmailBuilder<
  HasTo extends TRUE | FALSE = FALSE,
  HasSubject extends TRUE | FALSE = FALSE,
  HasBody extends TRUE | FALSE = FALSE
> {
  private _to = "";
  private _subject = "";
  private _body = "";
  private _cc: string[] = [];
  private _attachments: string[] = [];

  to(address: string): EmailBuilder<TRUE, HasSubject, HasBody> {
    this._to = address;
    return this as unknown as EmailBuilder<TRUE, HasSubject, HasBody>;
  }

  subject(text: string): EmailBuilder<HasTo, TRUE, HasBody> {
    this._subject = text;
    return this as unknown as EmailBuilder<HasTo, TRUE, HasBody>;
  }

  body(html: string): EmailBuilder<HasTo, HasSubject, TRUE> {
    this._body = html;
    return this as unknown as EmailBuilder<HasTo, HasSubject, TRUE>;
  }

  // Optional fields don't change phantom types
  cc(address: string): EmailBuilder<HasTo, HasSubject, HasBody> {
    this._cc.push(address);
    return this as unknown as EmailBuilder<HasTo, HasSubject, HasBody>;
  }

  attach(filename: string): EmailBuilder<HasTo, HasSubject, HasBody> {
    this._attachments.push(filename);
    return this as unknown as EmailBuilder<HasTo, HasSubject, HasBody>;
  }

  // send() only available when ALL three required fields are set
  send(
    this: EmailBuilder<TRUE, TRUE, TRUE>
  ): Promise<{ messageId: string }> {
    return Promise.resolve({
      messageId: `msg-${Date.now()}`,
    });
  }
}

// Valid — all three required fields set
const result = await new EmailBuilder()
  .to("alice@example.com")
  .subject("Meeting")
  .body("<p>See you at 3pm</p>")
  .cc("bob@example.com")       // optional — fine
  .attach("agenda.pdf")         // optional — fine
  .send();                      // OK: HasTo, HasSubject, HasBody all TRUE

console.log(result.messageId);

// Invalid — missing body
// new EmailBuilder()
//   .to("alice@example.com")
//   .subject("Meeting")
//   .send();
// ERROR: The 'this' context of type 'EmailBuilder<TRUE, TRUE, FALSE>'
//        is not assignable to type 'EmailBuilder<TRUE, TRUE, TRUE>'
After .to(): EmailBuilder<TRUE, FALSE, FALSE> After .subject(): EmailBuilder<TRUE, TRUE, FALSE> After .body(): EmailBuilder<TRUE, TRUE, TRUE> .send() is callable: YES
TypeScript — WRONG
// WRONG: builder with no compile-time safety
class UnsafeEmailBuilder {
  private _to?: string;
  private _subject?: string;
  private _body?: string;

  to(address: string) { this._to = address; return this; }
  subject(text: string) { this._subject = text; return this; }
  body(html: string) { this._body = html; return this; }

  send() {
    // Runtime check — should have been caught at compile time!
    if (!this._to || !this._subject || !this._body) {
      throw new Error("Missing required fields");
    }
    return Promise.resolve({ messageId: "msg-1" });
  }
}

// This compiles fine, but throws at runtime:
new UnsafeEmailBuilder().to("alice@example.com").send(); // BOOM
No compile-time error. Runtime: "Missing required fields" With phantom types, this would be a compile-time error instead.
Common Mistakes
Overcomplicating with phantom types when a simple required constructor parameter would work: new Request(url, { method }) already requires both.
Use phantom types only when the builder pattern is genuinely needed (many optional fields, complex configuration, or step-by-step construction).
Exposing phantom type details in the public API: function build(b: RequestBuilder<Set, Set, Set, Set>) — unreadable.
Create a type alias: type CompleteRequest = RequestBuilder<Set, Set, Set, Set> for cleaner signatures.
Using as any excessively inside the builder to bypass the phantom checks.
Limit as any to the internal builder wiring; the public API should be fully type-safe.
Key Rules
  • 2026-07-04 (Day 31)
  • Phantom types exist only in the type system — they have zero runtime cost.
  • Use them to track state transitions at the type level: Unset → Set, Open → Closed, etc.
  • Keep the internal implementation simple; the value to users is the compile-time guarantee.
  • Consider branded types (type UserId = string & { __brand: "UserId" }) as a simpler alternative for scalar values.

Type-Safe State Machines Advanced Patterns
Complex workflows (order processing, authentication flows, UI wizards) have states where only certain transitions are valid. TypeScript can enforce that invalid transitions are compile-time errors, not runtime bugs.

Example 1 — Simple traffic light with typed transitions:

TypeScript
// Define the state machine as a type-level map
type TrafficLight = "red" | "green" | "yellow";

// Valid transitions: Red -> Green -> Yellow -> Red
type Transitions = {
  red: "green";
  green: "yellow";
  yellow: "red";
};

// Typed transition function
function transition<S extends TrafficLight>: Transitions[S] {
  const map: Record<TrafficLight, TrafficLight> = {
    red: "green",
    green: "yellow",
    yellow: "red",
  };
  return map[current] as Transitions[S];
}

const s1: "red" = "red";
const s2 = transition(s1);   // type is "green"
const s3 = transition(s2);   // type is "yellow"
const s4 = transition(s3);   // type is "red"

console.log(s1, "->", s2, "->", s3, "->", s4);
red -> green -> yellow -> red
TypeScript — WRONG
// WRONG: untyped state machine allows invalid transitions
function badTransition: string {
  if  return "yellow"; // Invalid! Red should go to Green
  if  return "red";  // Invalid! Green should go to Yellow
  return "green";
}

// No compile-time error — the bug ships to production
No error at compile time, but the transitions are logically wrong. With typed transitions this would be caught immediately.

Example 2 — Order lifecycle with typed events per state:

TypeScript
// Each state has specific data and allowed events
type OrderState =
  | { status: "draft"; items: string[] }
  | { status: "submitted"; items: string[]; submittedAt: Date }
  | { status: "paid"; items: string[]; submittedAt: Date; paidAt: Date; amount: number }
  | { status: "shipped"; items: string[]; trackingId: string }
  | { status: "delivered"; deliveredAt: Date }
  | { status: "cancelled"; reason: string };

// Events that can trigger transitions
type OrderEvent =
  | { type: "SUBMIT" }
  | { type: "PAY"; amount: number }
  | { type: "SHIP"; trackingId: string }
  | { type: "DELIVER" }
  | { type: "CANCEL"; reason: string };

// Map of which events are valid for which state
type ValidEvents = {
  draft: "SUBMIT" | "CANCEL";
  submitted: "PAY" | "CANCEL";
  paid: "SHIP" | "CANCEL";
  shipped: "DELIVER";
  delivered: never;
  cancelled: never;
};

// Type-safe dispatch — only accepts valid events for the current state
function dispatch<S extends OrderState["status"]>(
  state: Extract<OrderState, { status: S }>,
  event: Extract<OrderEvent, { type: ValidEvents[S] }>
): OrderState {
  switch (state.status) {
    case "draft":
      if (event.type === "SUBMIT") {
        return { status: "submitted", items: state.items, submittedAt: new Date() };
      }
      if (event.type === "CANCEL") {
        return { status: "cancelled", reason: (event as Extract<OrderEvent, {type:"CANCEL"}>).reason };
      }
      break;
    case "submitted":
      if (event.type === "PAY") {
        const payEvent = event as Extract<OrderEvent, { type: "PAY" }>;
        return {
          status: "paid",
          items: state.items,
          submittedAt: state.submittedAt,
          paidAt: new Date(),
          amount: payEvent.amount,
        };
      }
      break;
    case "paid":
      if (event.type === "SHIP") {
        const shipEvent = event as Extract<OrderEvent, { type: "SHIP" }>;
        return { status: "shipped", items: state.items, trackingId: shipEvent.trackingId };
      }
      break;
    case "shipped":
      if (event.type === "DELIVER") {
        return { status: "delivered", deliveredAt: new Date() };
      }
      break;
  }
  throw new Error(`Invalid transition: ${state.status} + ${event.type}`);
}

// Valid transitions
const draft: Extract<OrderState, { status: "draft" }> = { status: "draft", items: ["widget"] };
const submitted = dispatch(draft, { type: "SUBMIT" });

// Invalid — "DELIVER" is not valid for "draft" state
// dispatch(draft, { type: "DELIVER" });
// ERROR: Type '"DELIVER"' is not assignable to type '"SUBMIT" | "CANCEL"'
draft: { status: "draft"; items: string[] } dispatch(draft, { type: "SUBMIT" }): OrderState dispatch(draft, { type: "DELIVER" }): compile error

Example 3 — XState-style machine definition with fully typed transitions:

TypeScript
// Define machine config as a type-level specification
interface MachineConfig<
  TStates extends string,
  TEvents extends string
> {
  initial: TStates;
  states: {
    [S in TStates]: {
      on?: Partial<Record<TEvents, TStates>>;
    };
  };

  TStates extends string,
  TEvents extends string

      state: S,

      return config.states[state].on?.[event] !== undefined;
    },
  };

Variance Annotations (in/out) Type System Deep Dive
TypeScript 4.7+ introduced explicit variance annotations (in and out) for type parameters. Understanding variance prevents subtle assignability bugs — especially in event systems, collections, and callback-heavy APIs.

Variance describes how subtyping of a generic type relates to subtyping of its type parameter:

  • Covariant (out T): if Dog extends Animal, then Producer<Dog> is assignable to Producer<Animal>
  • Contravariant (in T): if Dog extends Animal, then Consumer<Animal> is assignable to Consumer<Dog>
  • Invariant (in out T): no assignability in either direction (both producer and consumer)

Example 1 — Producer (covariant) vs Consumer (contravariant):

TypeScript
// Covariant: only produces T (output position)
interface Producer<out T> {
  get(): T;
}

// Contravariant: only consumes T (input position)
interface Consumer<in T> {
  accept(value: T): void;
}

// Invariant: both produces and consumes T
interface Collection<in out T> {
  get(index: number): T;
  add(value: T): void;
}

class Animal { name = "animal"; }
class Dog extends Animal { breed = "unknown"; }
class Cat extends Animal { indoor = true; }

// Covariant: Dog producer CAN be used as Animal producer
const dogProducer: Producer<Dog> = { get: () => new Dog() };
const animalProducer: Producer<Animal> = dogProducer;  // OK: Dog extends Animal

// Contravariant: Animal consumer CAN be used as Dog consumer
const animalConsumer: Consumer<Animal> = { accept: (a: Animal) => console.log(a.name) };
const dogConsumer: Consumer<Dog> = animalConsumer;  // OK: reversed direction

// Invariant: neither direction works
// const dogCollection: Collection<Dog> = animalCollection; // ERROR
// const animalCollection: Collection<Animal> = dogCollection; // ERROR
dogProducer assignable to Producer<Animal>: YES (covariant) animalConsumer assignable to Consumer<Dog>: YES (contravariant) Collection: neither direction compiles (invariant)

Example 2 — Why Array<Dog> assignable to Array<Animal> is unsound:

TypeScript allows this by default (arrays are covariant), but it is technically unsound. The in out annotation would catch this:

// But dogs[1] is actually a Cat, not a Dog
console.log(dogs[1].breed); // Runtime: undefined — dogs[1] is a Cat!

// With a properly invariant collection, this would be caught:
interface SafeArray<in out T> {
  get(index: number): T;
  push(value: T): void;
}

// const safeDogs: SafeArray<Dog> = /* ... */;
// const safeAnimals: SafeArray<Animal> = safeDogs;

Example 3 — Real-world: Event system with proper variance:

TypeScript
// Event emitter — T is in both input and output positions, so invariant
interface TypedEmitter<in out TEvents extends Record<string, unknown[]>> {
  on<K extends keyof TEvents>(
    event: K,
    handler: (...args: TEvents[K]) => void
  ): void;

  emit<K extends keyof TEvents>(
    event: K,
    ...args: TEvents[K]
  ): void;
}

// Read-only view of events — only output (covariant)
interface EventSource<out TEvents extends Record<string, unknown[]>> {
  on<K extends keyof TEvents>(
    event: K,
    handler: (...args: TEvents[K]) => void
  ): void;
}

// Write-only sink — only input (contravariant)
interface EventSink<in TEvents extends Record<string, unknown[]>> {
  emit<K extends keyof TEvents>(
    event: K,
    ...args: TEvents[K]
  ): void;
}

// Define event maps
type AppEvents = {
  userLogin: [userId: string, timestamp: number];
  pageView: [url: string];
  error: [error: Error, context: string];
};

type AuditEvents = AppEvents & {
  adminAction: [action: string, adminId: string];
};

// A source of AuditEvents can be used as a source of AppEvents (covariant — broader events)
function setupLogger(source: EventSource<AppEvents>) {
  source.on("userLogin", (userId, ts) => {
    console.log(`User ${userId} logged in at ${ts}`);
  });
  source.on("error", (err, ctx) => {
    console.error(`Error in ${ctx}: ${err.message}`);
  });
}

// Usage
function createEmitter<T extends Record<string, unknown[]>>(): TypedEmitter<T> {
  const handlers = new Map<string, Function[]>();
  return {
    on(event, handler) {
      const list = handlers.get(event as string) ?? [];
      list.push(handler);
      handlers.set(event as string, list);
    },
    emit(event, ...args) {
      const list = handlers.get(event as string) ?? [];
      list.forEach(fn => fn(...args));
    },
  };
}

Example 4 — Variance with function types and callbacks:

Function parameters are contravariant, return types are covariant. This is why callback-heavy APIs need careful variance handling:

TypeScript
class Animal { name = "animal"; move() { console.log("moving"); } }

class GoldenRetriever extends Dog { color = "golden"; fetch() { console.log("fetching!"); } }

// Function return type is covariant:
// () => Dog  is assignable to  () => Animal  (Dog extends Animal)
type DogFactory = () => Dog;
type AnimalFactory = () => Animal;

const makeDog: DogFactory = () => new Dog();

console.log(makeAnimal().name); // "animal" — safe, Dog has .name

// (a: Animal) => void  is assignable to  (d: Dog) => void  (reversed!)
type AnimalHandler = (a: Animal) => void;
type DogHandler = (d: Dog) => void;

const handleAnimal: AnimalHandler = (a: Animal) => console.log(a.name);

handleDog(new Dog()); // Safe — Dog has .name

// But NOT the other direction:

// const handleAnimalBad: AnimalHandler = handleDogSpecific;
// ERROR: not all Animals have .breed — would crash at runtime

// DataProcessor<Animal> is assignable to DataProcessor<Dog> (contravariant)

Example 5 — Variance annotations as documentation and performance:

Explicit variance annotations also help the compiler — it can skip structural comparison and check variance directly, which speeds up type checking for large generic types:

// Without annotations: TypeScript infers variance by analyzing every member
// This can be slow for complex types with many members
interface SlowBox<T> {
  get(): T;
  map<U>(fn: (val: T) => U): SlowBox<U>;
  flatMap<U>(fn: (val: T) => SlowBox<U>): SlowBox<U>;
  filter(pred: (val: T) => boolean): SlowBox<T>;
  // ... 20 more methods
  // TypeScript must analyze ALL methods to determine variance
}

// No need to analyze every method — just check that T isn't used in input positions
interface FastBox<out T> {
  get(): T;
  map<U>(fn: (val: T) => U): FastBox<U>;
  flatMap<U>(fn: (val: T) => FastBox<U>): FastBox<U>;
  filter(pred: (val: T) => boolean): FastBox<T>;
  // ... 20 more methods

Common Mistakes
Misunderstanding TypeScript's default: arrays are covariant, which is technically unsound but allowed for convenience.
Use ReadonlyArray<T> when you don't need mutation — read-only collections are safely covariant.
Using out T on a type that also consumes T (e.g., has a push(value: T) method).
TypeScript will error if the annotation contradicts the actual usage. Use in out T for types that both produce and consume.
Ignoring variance entirely — it "just works" until it doesn't, usually with callbacks or mutable collections.
Add explicit in/out annotations to library-level generic interfaces — they serve as documentation and catch errors early.
Key Rules
  • out T = covariant = "produces T" (return types, getter results). Dog extends Animal means Producer<Dog> extends Producer<Animal>.
  • in T = contravariant = "consumes T" (function parameters). Direction reverses: Consumer<Animal> extends Consumer<Dog>.
  • in out T = invariant = "both produces and consumes T". No assignability in either direction.
  • Variance annotations are enforced by the compiler — if you mark out T but use T in an input position, TypeScript will error.
  • Prefer ReadonlyArray<T> over Array<T> in function parameters to get safe covariance.

Example 4 — Testing generic function inference:

One of the most valuable uses of type testing is verifying that generic functions infer types correctly in various usage patterns:

TypeScript — src/result.ts
// A Result monad implementation we want to type-test
export type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E };

export function ok<T>(value: T): Result<T, never> {
  return { ok: true, value };
}

export function err<E>(error: E): Result<never, E> {
  return { ok: false, error };
}

export function map<T, U, E>(
  result: Result<T, E>,
  fn: (value: T) => U
): Result<U, E> {
  return result.ok ? ok(fn(result.value)) : result;
}

export function flatMap<T, U, E1, E2>(
  result: Result<T, E1>,
  fn: (value: T) => Result<U, E2>
): Result<U, E1 | E2> {
  return result.ok ? fn(result.value) : result;
}

Common Mistakes
Testing runtime behavior and assuming it validates the types: expect(result).toBeDefined() says nothing about the type of result.
Use expectTypeOf(result).toEqualTypeOf<ExpectedType>() for type-level assertions.
Using toEqualTypeOf when toMatchTypeOf is more appropriate (exact vs structural match).
toEqualTypeOf is bidirectional (exact). toMatchTypeOf is one-way (the type being tested extends the expected type).
Not running type tests in CI: tsc --noEmit is often skipped or not wired into the test pipeline.
Add "test:types": "tsc --noEmit" to your scripts and run it in CI alongside unit tests.
Key Rules
  • Use *.test-d.ts files for type-only tests; run them with tsc --noEmit.
  • expectTypeOf<T>().toEqualTypeOf<U>() checks that T and U are mutually assignable (exact match).
  • expectTypeOf<T>().toMatchTypeOf<U>() checks that T extends U (one-way).
  • Test utility types, generic return types, and function signatures — not just runtime behavior.
  • Always include type tests in your CI pipeline to prevent regressions.

Assertion Functions & Type Narrowing Patterns Type Safety & Testing
TypeScript's control flow analysis narrows types automatically in if/switch blocks, but complex validation logic often lives in helper functions. Assertion functions and type predicates let you teach TypeScript that your custom validation narrows the type.

Example 1 — Basic assertion function with asserts:

An assertion function throws if the condition fails. After the call, TypeScript narrows the type:

TypeScript
// Assertion function: narrows val to NonNullable<T>
function assertDefined<T>(
  val: T,
  message?: string
): asserts val is NonNullable<T> {
  if (val === null || val === undefined) {
    throw new Error(message ?? `Expected value to be defined, got ${val}`);
  }
}

// Usage
function getUser(id: string): { name: string; email: string } | undefined {
  // Simulated lookup
  if (id === "1") return { name: "Alice", email: "alice@example.com" };
  return undefined;
}

const user = getUser("1");
// user: { name: string; email: string } | undefined

assertDefined(user, "User not found");
// After assertion, user is narrowed:
// user: { name: string; email: string }

console.log(user.name);  // OK — TypeScript knows user is defined
console.log(user.email); // OK — no optional chaining needed

Example 3 — Combining assertion functions with discriminated unions:

TypeScript
// State machine with assertions for transitions
type AppState =
  | { phase: "setup"; config: Partial<AppConfig> }
  | { phase: "ready"; config: AppConfig }
  | { phase: "running"; config: AppConfig; startedAt: Date }
  | { phase: "error"; config: AppConfig; error: Error };

interface AppConfig {
  apiUrl: string;
  apiKey: string;
  timeout: number;
}

// Assert we're in a specific phase
function assertPhase<P extends AppState["phase"]>(
  state: AppState,
  phase: P
): asserts state is Extract<AppState, { phase: P }> {
  if (state.phase !== phase) {
    throw new Error(`Expected phase "${phase}", but app is in phase "${state.phase}"`);
  }
}

// Assert config is complete
function assertConfigComplete(
  config: Partial<AppConfig>
): asserts config is AppConfig {
  const required: (keyof AppConfig)[] = ["apiUrl", "apiKey", "timeout"];
  for (const key of required) {
    if (config[key] === undefined) {
      throw new Error(`Missing required config: ${key}`);
    }
  }
}

// Usage in application lifecycle
function startApp(state: AppState): Extract<AppState, { phase: "running" }> {
  assertPhase(state, "ready");
  // state is now: { phase: "ready"; config: AppConfig }

  console.log(`Starting with API: ${state.config.apiUrl}`);
  return {
    phase: "running",
    config: state.config,
    startedAt: new Date(),
  };
}

// Type predicate (alternative to assertion function)
function isReady(state: AppState): state is Extract<AppState, { phase: "ready" }> {
  return state.phase === "ready";
}

// Type predicate returns boolean — doesn't throw
function safeStart(state: AppState): AppState {
  if (isReady(state)) {
    // state narrowed to { phase: "ready"; config: AppConfig }
    return { phase: "running", config: state.config, startedAt: new Date() };
  }
  return state; // Stay in current phase
}
assertPhase(state, "ready") narrows to: { phase: "ready"; config: AppConfig } isReady(state) narrows inside if-block to: { phase: "ready"; config: AppConfig }

Example 3b — Assertion functions in test code:

Assertion functions are especially powerful in test code, where they clean up repetitive null checks and make test failures more descriptive:

TypeScript — Test utilities
// Custom test assertions that narrow types
function assertInstanceOf<T>(
  value: unknown,
  constructor: new (...args: any[]) => T,
  message?: string
): asserts value is T {
  if (!(value instanceof constructor)) {
    throw new Error(
      message ?? `Expected instance of ${constructor.name}, got ${typeof value}`
    );
  }
}

function assertLength<T>(
  arr: T[],
  expected: number,
  message?: string
): asserts arr is T[] & { length: typeof expected } {
  if (arr.length !== expected) {
    throw new Error(
      message ?? `Expected array of length ${expected}, got ${arr.length}`
    );
  }
}

function assertMatch(
  value: string,
  pattern: RegExp,
  message?: string
): asserts value is string {
  if (!pattern.test(value)) {
    throw new Error(
      message ?? `Expected "${value}" to match ${pattern}`
    );
  }
}

// Usage in tests
function testUserCreation() {
  const result = createUser({ name: "Alice", email: "alice@test.com" });

  // Without assertion functions:
  // if (!result) throw new Error("result is null");
  // if (!(result instanceof User)) throw new Error("not a User");
  // if (result.posts.length !== 0) throw new Error("wrong length");

  // With assertion functions — cleaner and type-safe:
  assertDefined(result, "createUser should return a user");
  // result is now non-null

  assertInstanceOf(result, User, "result should be a User instance");
  // result is now typed as User

  assertLength(result.posts, 0, "new user should have no posts");
  // result.posts has known length

  assertMatch(result.id, /^usr_[a-z0-9]+$/, "user ID should match pattern");
  // result.id is confirmed to match the pattern

  console.log(`Test passed: ${result.name} created with ID ${result.id}`);
}

// Helper for async assertions
async function assertRejects<E extends Error>(
  fn: () => Promise<unknown>,
  errorType: new (...args: any[]) => E,
  messagePattern?: RegExp
): Promise<E> {
  try {
    await fn();
    throw new Error("Expected function to throw, but it resolved");
  } catch (error) {
    assertInstanceOf(error, errorType);
    if (messagePattern) {
      assertMatch(error.message, messagePattern);
    }
    return error;
  }
}

// Usage:
// const error = await assertRejects(
//   () => fetchUser("nonexistent"),
//   NotFoundError,
//   /user not found/i
// );
// error is typed as NotFoundError
assertDefined narrows: T | null | undefined -> T assertInstanceOf narrows: unknown -> T (the constructor type) assertRejects returns: the caught error typed as E

Example 4 — Assertion functions vs type predicates — when to use which:

TypeScript
// TYPE PREDICATE: returns boolean, narrows in if/else blocks
// Use when: checking is expected, and both branches are valid code paths
function isString(val: unknown): val is string {
  return typeof val === "string";
}

function processValue(val: unknown) {
  if (isString(val)) {
    console.log(val.toUpperCase()); // val is string
  } else {
    console.log("Not a string");     // val is still unknown
  }
}

// ASSERTION FUNCTION: throws on failure, narrows after the call
// Use when: the failure case is exceptional (should crash)
function assertString(val: unknown, label: string): asserts val is string {
  if (typeof val !== "string") {
    throw new TypeError(`${label} must be a string, got ${typeof val}`);
  }
}

function processRequiredString(val: unknown) {
  assertString(val, "input");
  // If we reach here, val is string — no if/else needed
  console.log(val.toUpperCase());
}

interface FormData {
  name: unknown;
  age: unknown;
  email: unknown;
}

interface ValidatedForm {
  name: string;
  age: number;
  email: string;
}

  assertString(data.name, "name");

  if (typeof data.age !== "number" || data.age < 0 || data.age > 150) {
    throw new Error(`age must be a number 0-150, got ${data.age}`);
  }

  assertString(data.email, "email");
  if (!data.email.includes("@")) {
    throw new Error(`email must contain @, got ${data.email}`);
  }
}

Common Mistakes
Writing an assertion function that doesn't actually throw when the condition fails — TypeScript trusts the asserts annotation unconditionally.
Always throw (or call a function that throws) in the failure branch. The compiler cannot verify that your assertion is correct.
Forgetting the asserts return type: function check(val: unknown): void — no narrowing happens.
Use asserts val is T for assertion functions and val is T (returns boolean) for type predicates.
Using assertion functions for optional checks where both paths are valid.
Use type predicates (is) for conditional narrowing in if/else; use assertion functions (asserts) for "must be true or crash" patterns.
Over-asserting with asserts val is T when a simple if (typeof val === "string") would work.
Inline narrowing is simpler for one-off checks. Use assertion functions when the validation logic is reusable or complex.
Key Rules
  • Type predicates (val is T): return boolean, narrow in if/else branches. Non-throwing.
  • Assertion functions (asserts val is T): throw on failure, narrow after the call. The code after the assertion assumes the condition holds.
  • TypeScript trusts your assertion — if the function doesn't throw when it should, you get runtime bugs with no compile-time warning.
  • Assertion functions must be called as statements (not inside expressions like arr.filter(assertDefined)).
  • Combine assertion functions for complex validation: assertString(x) + assertNonEmpty(x) + assertMatches(x, /regex/) — each call narrows further.
