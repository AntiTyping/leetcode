R1. Monorepo Typing: Turborepo & nx Ecosystem Deep Dives 16 days since review · rep 1 → 2
A TS monorepo shares types across packages through TypeScript project references plus workspace-protocol deps; tsc -b builds each referenced package in dependency order and emits .d.ts so consumers see types without rebuilding source. Turborepo/nx add a cached task graph on top — they orchestrate build/test/lint across packages; they don’t do the type-linking themselves.
Key Pattern 
// packages/api/tsconfig.json — depend on the shared package by reference
{ "references": [{ "path": "../core" }],
  "compilerOptions": { "composite": true } }

// root turbo.json — task graph + caching (NOT type resolution)
{ "tasks": { "build": { "dependsOn": ["^build"], "outputs": ["dist/**"] } } }

import { UserId } from "@acme/core";   // types resolved via core’s emitted .d.ts
project references link .d.ts across packages; turbo just orders & caches the tasks
Challenge Exercise

Set up two packages where @acme/api references @acme/core; mark core as composite and confirm that changing a type in core surfaces as a compile error in api after tsc -b. Then explain what "^build" in turbo.json’s dependsOn means and why it governs task ordering, not type resolution.

Pitfall: Expecting Turborepo/nx to give you cross-package types — they only schedule and cache tasks. Type sharing comes from project references + composite: true (or a bundler’s path aliases). Omitting composite: true on a referenced package means tsc -b won’t emit the .d.ts the consumer needs.
R2. Next.js + TypeScript Ecosystem Deep Dives 16 days since review · rep 1 → 2
In the App Router, components are Server Components by default (they can be async and await data directly); "use client" opts into the browser. Route handlers, generateMetadata, and page props are typed — and in Next 15 params/searchParams arrive as a Promise you must await. Server Actions are plain typed async functions marked "use server".
Key Pattern 
// app/users/[id]/page.tsx — params is a Promise in Next 15
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const user = await getUser(id);          // server component: await data inline
  return <h1>{user.name}</h1>;
}

async function createUser(data: FormData) {  // a typed Server Action
  "use server";
  await db.user.create({ name: String(data.get("name")) });
}
params/searchParams are awaited Promises; server components await data inline
Challenge Exercise

Type a route handler app/api/users/route.ts whose POST parses the body with Zod and returns NextResponse.json<User>(...). Then convert a client-side fetch data load into an async Server Component and explain exactly when you must add "use client" (hooks, event handlers, browser APIs).

Pitfall: Destructuring params synchronously in Next 15 — it’s now a Promise, so const { id } = params is wrong; await it. And calling a hook (useState) or wiring an onClick in a file without "use client" — Server Components can’t use browser-only APIs.
R3. Form Handling: React Hook Form + Zod Ecosystem Deep Dives 16 days since review · rep 1 → 2
Define the form shape once as a Zod schema; z.infer<typeof schema> is the single TS type for the form; zodResolver wires runtime validation into React Hook Form so useForm<FormValues> gives fully typed register/handleSubmit and typed errors — one schema drives both the type and the validation.
Key Pattern 
const schema = z.object({ email: z.string().email(), age: z.coerce.number().min(18) });
type FormValues = z.infer<typeof schema>;   // { email: string; age: number }

const { register, handleSubmit, formState: { errors } } =
  useForm<FormValues>({ resolver: zodResolver(schema) });

const onSubmit = handleSubmit((data) => save(data));  // data: FormValues, already valid
z.infer gives the form type; the resolver validates and narrows on submit
Challenge Exercise

Add a confirmPassword field validated with .refine(d => d.password === d.confirm) and confirm the error surfaces on the right field via errors. Explain why z.coerce.number() is needed for an <input> (its value is always a string) and how z.infer’s output type can differ from the raw input type.

Pitfall: Hand-writing a FormValues interface alongside the schema — the two drift. Derive it with z.infer. And forgetting z.coerce for numeric/date inputs: the DOM always gives strings, so a plain z.number() rejects a value that looks numeric.
R4. WebSocket Typing Ecosystem Deep Dives 16 days since review · rep 1 → 2
Type the protocol, not the socket. Model messages as a discriminated union (or, with socket.io, separate ClientToServerEvents/ServerToClientEvents maps) so each event name is tied to its payload type and on/emit are checked in both directions. On a raw ws, parse and narrow incoming JSON through a Zod schema at the boundary.
Key Pattern 
// socket.io — typed event maps in BOTH directions
interface ServerToClient { message: (m: { text: string; ts: number }) => void }
interface ClientToServer { send: (text: string, ack: (ok: boolean) => void) => void }

const io = new Server<ClientToServer, ServerToClient>();
io.on("connection", (socket) => {
  socket.on("send", (text, ack) => { io.emit("message", { text, ts: Date.now() }); ack(true); });
  // socket.emit("mesage", …)  // ✗ unknown event name is a compile error
});
event names and payloads are checked both ways; a typo’d event won’t compile
Challenge Exercise

Type a raw ws server: define a ClientMsg discriminated union ({ type: "chat" } | { type: "ping" }), parse socket.on("message") data through a Zod union, and switch exhaustively with assertNever. Explain why the wire is untyped (Buffer/string) so runtime validation is mandatory even with the compile-time union.

Pitfall: Trusting socket.on("message", (raw) => JSON.parse(raw)) as your typed message — the wire delivers arbitrary bytes, so an unvalidated cast lies. Validate at the boundary (Zod), then switch on the discriminant. With socket.io, skipping the generic event maps drops all payload checking.
R5. Streaming & Async Iteration Typing Runtime Type Patterns 16 days since review · rep 1 → 2
An async generator is typed AsyncGenerator<T> (consumable as AsyncIterable<T>) and drained with for await...of; each yielded value is T. Web Streams’ ReadableStream<T> is likewise async-iterable. Streaming keeps memory flat over large/infinite sequences and naturally models backpressure — the producer only advances when the consumer pulls.
Key Pattern 
async function* pages(url: string): AsyncGenerator<User[]> {
  let next: string | null = url;
  while (next) {
    const res = await fetch(next).then((r) => r.json());
    yield res.items as User[];       // each pull fetches exactly one page
    next = res.nextCursor;
  }
}

for await (const batch of pages("/users")) {   // batch: User[]
  await process(batch);
}
yielded values are typed T; for await pulls one at a time (backpressure)
Challenge Exercise

Write a generic async function* map<T, U>(src: AsyncIterable<T>, fn: (t: T) => U): AsyncGenerator<U> and chain it after pages(). Explain how for await...of gives backpressure for free versus loading every page into an array, and when you’d reach for a Web ReadableStream instead of a generator.

Pitfall: Collecting a stream into an array (const all = []; for await (...) all.push(x)) — you’ve thrown away the memory and backpressure benefits and can OOM on a large source. Consume incrementally. Also note AsyncGenerator<T, TReturn, TNext> has three type params; usually only the first (yield) matters.
R6. Type-Safe Configuration Runtime Type Patterns 16 days since review · rep 1 → 2
Treat config like untrusted input: parse it once at boot through a Zod schema into a frozen, fully-typed Config, applying defaults and coercions, and throw on anything invalid. Downstream code imports the typed object and never touches process.env — the type is the single source of truth and misconfig fails fast at startup, not deep in a request.
Key Pattern 
const ConfigSchema = z.object({
  PORT: z.coerce.number().default(3000),
  NODE_ENV: z.enum(["dev", "prod", "test"]).default("dev"),
  DATABASE_URL: z.string().url(),
});
export type Config = z.infer<typeof ConfigSchema>;

const parsed = ConfigSchema.safeParse(process.env);
if (!parsed.success) { console.error(parsed.error.format()); process.exit(1); }
export const config: Readonly<Config> = Object.freeze(parsed.data);
env is coerced + validated into a frozen, typed Config; a bad value crashes at boot
Challenge Exercise

Add a nested schema (e.g. redis: { host, port }) built from flat REDIS_HOST/REDIS_PORT env vars via .transform, and make config deeply readonly. Explain why parsing at boot (fail fast) beats reading process.env lazily inside request handlers.

Pitfall: Reading process.env.X ad hoc throughout the app — every read is string | undefined, defaults scatter, and a missing var surfaces as a mystery bug mid-request instead of a loud crash at boot. Parse once, freeze, and import the typed config everywhere.
1Node.js + TS: Testing Backends
1. Integration Testing with Supertest Node.js + TypeScript
Unit tests check functions in isolation; integration tests exercise the real HTTP stack — routing, middleware, Zod validation, serialization — against your Express/Fastify app. Supertest drives the app in-process (no fixed port) and returns a typed response you assert on, catching contract breaks — a wrong status code, a renamed field — that a unit test can’t see.

Pass the app object to request(app); Supertest binds an ephemeral port itself, so tests never collide on 3000. Chain .get/.post, send a typed body, await the response, then assert the status code (the real contract) and a typed body shape. Because res.body is any, cast it once to an interface at the test boundary so structural assertions are actually checked.

Keep tests order-independent by resetting shared state (the DB) in beforeEach, and tear down pools in afterAll so the process exits cleanly.

1
Wrap the app in request(app) — no listening port required.
2
Drive routes with typed .send() bodies and await the response.
3
Assert the status code + a typed body shape; reset state between tests.
TypeScript · example 1 of 4 — drive the in-process app, assert status
import request from "supertest";
import { app } from "../app.js";

test("GET /health returns 200", async () => {
  const res = await request(app).get("/health");   // no app.listen() needed
  expect(res.status).toBe(200);
  expect(res.body).toEqual({ status: "ok" });
});
request(app) binds an ephemeral port; res carries status + body
TypeScript · example 2 of 4 — typed request body & response shape
interface CreateUserBody { email: string; name: string }
interface UserResponse { id: string; email: string; name: string }

const res = await request(app)
  .post("/users")
  .send({ email: "a@b.io", name: "Ana" } satisfies CreateUserBody)
  .set("Content-Type", "application/json");

const user = res.body as UserResponse;   // one cast at the test boundary
expect(res.status).toBe(201);
expect(user.email).toBe("a@b.io");
satisfies checks the body against CreateUserBody; the cast makes body assertions typed
TypeScript · example 3 of 4 — a small typed request helper
function post<TBody>(url: string, body: TBody, token?: string) {
  const req = request(app).post(url).send(body as object);
  return token ? req.set("Authorization", `Bearer ${token}`) : req;
}

// callers get body-type checking for free:
const res = await post<CreateUserBody>("/users", { email: "a@b.io", name: "Ana" });
the generic TBody pins each call’s payload to a real request shape
TypeScript · example 4 of 4 — lifecycle: isolated, order-free state
beforeEach(async () => { await resetDb(); });   // truthful, independent state
afterAll(async () => { await pool.end(); });    // let the process exit cleanly

test("rejects a duplicate email", async () => {
  await post<CreateUserBody>("/users", { email: "dup@x.io", name: "A" });
  const res = await post<CreateUserBody>("/users", { email: "dup@x.io", name: "B" });
  expect(res.status).toBe(409);   // problem+json conflict from 07-03
});
resetting in beforeEach makes every test independent of run order
Common Mistakes:
❌ Calling app.listen(3000) in the test and hitting a real port — flaky, and leaked handles keep the process alive.
✅ Pass the app object to request(app); Supertest binds an ephemeral port automatically.
❌ Leaving res.body as any and asserting nothing structural — a renamed field slips through green.
✅ Cast res.body to a typed interface once at the boundary so field assertions are checked.
Key Rules:
  • Pass the app object to request(app); never manually listen on a fixed port in tests.
  • Cast res.body to a typed interface once at the test boundary so body assertions are type-checked.
  • Reset shared state (DB) in beforeEach so each test is independent and order-free.
  • Assert the HTTP status code, not just the body — that’s the real contract.
Practice Exercises:
1Write a typed expectJson<T>(res, status): T helper that asserts the status and returns res.body as T.
2Add a test that a 400 returns an RFC 7807 problem+json body (type it) when Zod body validation fails.
3Use a Supertest agent to persist a login cookie across two requests and assert the second is authorized.
Connections: Supertest drives the real Express/Fastify handlers (06-30) through their Zod validation and problem+json errors (07-03). The typed res.body cast is the narrow-at-the-boundary discipline again, and the request bodies come from the factories of Concept 2.
2. Typed Test Fixtures & Factories Node.js + TypeScript
Tests need realistic domain objects, but inlining a full User in every test is noisy and brittle — add a required field and dozens of tests break. A typed factory returns a complete valid object from a small Partial<T> override, so each test states only what it cares about and the compiler guarantees the result is a real T.

A factory is (overrides?: Partial<T>) => T: spread sensible defaults first, then the overrides so caller values win, and return the exact domain type. Generate unique keys (a counter or randomUUID) so persisted fixtures don’t collide on unique constraints. Compose factories — buildOrder calls buildUser — instead of duplicating nested defaults.

The payoff is maintenance: when a new required field lands on User, only the factory fails to compile, forcing a single fix rather than a hundred.

1
Write buildX(overrides: Partial<X> = {}): X merging defaults + overrides.
2
Generate unique fields (ids, emails) so rows don’t collide.
3
Compose and batch: factories call factories; buildMany maps a count.
TypeScript · example 1 of 4 — a typed factory with defaults
interface User { id: string; email: string; name: string; role: "admin" | "user"; createdAt: Date }

let seq = 0;
function buildUser(overrides: Partial<User> = {}): User {
  seq += 1;
  return {
    id: crypto.randomUUID(),
    email: `user${seq}@test.io`,
    name: "Test User",
    role: "user",
    createdAt: new Date(),
    ...overrides,        // overrides win; the result is still a FULL User
  };
}
buildUser() is a complete User; buildUser({ role: "admin" }) overrides one field
TypeScript · example 2 of 4 — state only what the test cares about
test("admins can delete", () => {
  const admin = buildUser({ role: "admin" });   // email/name/id auto-filled
  expect(canDelete(admin)).toBe(true);
});
// adding a required field to User breaks ONE place — the factory — not every test
Partial<User> lets a test set just `role`; the compiler still guarantees a full User
TypeScript · example 3 of 4 — composing factories
interface Order { id: string; user: User; total: number; status: "open" | "paid" }

function buildOrder(overrides: Partial<Order> = {}): Order {
  return {
    id: crypto.randomUUID(),
    user: buildUser(),          // reuse — no duplicated User defaults
    total: 0,
    status: "open",
    ...overrides,
  };
}
const paid = buildOrder({ status: "paid", user: buildUser({ role: "admin" }) });
nested defaults come from buildUser; overrides compose cleanly
TypeScript · example 4 of 4 — a generic buildMany helper
function buildMany<T>(factory: (o?: Partial<T>) => T, count: number, o?: Partial<T>): T[] {
  return Array.from({ length: count }, () => factory(o));
}

const admins = buildMany(buildUser, 3, { role: "admin" });   // User[]
buildMany infers T from the factory; admins: User[]
Common Mistakes:
❌ Typing the parameter as the full T (not Partial<T>) — every test is forced to pass every field, defeating the point.
✅ Accept Partial<T> and spread defaults first so a test overrides only what it needs.
❌ Sharing one mutable fixture object across tests — one test mutates it and pollutes the next.
✅ Return a fresh object per call (and unique keys) so fixtures never bleed or collide.
Key Rules:
  • Signature is (overrides?: Partial<T>) => T; spread defaults first, overrides last so overrides win.
  • Return the exact domain type so a new required field breaks only the factory — one update, not a hundred.
  • Generate unique keys (uuid/counter) so persisted fixtures don’t collide on unique constraints.
  • Compose factories rather than duplicating nested defaults.
Practice Exercises:
1Add a variant returning the DB insert shape Omit<User, "id" | "createdAt"> using a utility type.
2Make buildMany accept an (i: number) => Partial<T> so each item can vary by index.
3Write a sequence<T>(...values: T[]) helper that cycles values across calls (e.g. alternating roles).
Connections: Partial<T>, Omit, and the generic buildMany are utility types + generics. These objects feed the Supertest bodies of Concept 1 and the mocked return values of Concept 3, and mirror the branded/domain modeling of earlier days.
3. Typed Mocking with Vitest Node.js + TypeScript
Integration tests hit real collaborators; unit tests often shouldn’t (a payment gateway, a clock, an email sender). Vitest’s vi.fn/vi.mock create stand-ins, but an untyped mock will happily return the wrong shape and pass. Typing the double to the real function’s signature keeps it honest — it must accept and return exactly what the real thing does.

Parameterize vi.fn<Signature>() so args and return type are enforced, and mockResolvedValue/mockReturnValue must match. To replace a whole module, call vi.mock("../mod.js") then re-type the import with vi.mocked(fn) — that recovers the real signature on the mock API. Often cleaner still: inject a small typed interface (a Clock, a Mailer) and build a stub with satisfies, avoiding module mocking entirely.

Always clear mocks in beforeEach so call counts and implementations don’t bleed into later tests.

1
Type the double: vi.fn<(args) => Ret> (or MockedFunction<typeof real>).
2
Replace modules with vi.mock and re-type imports via vi.mocked.
3
Set typed return values and reset mocks between tests.
TypeScript · example 1 of 4 — a typed mock function
import { vi } from "vitest";

type Send = (to: string, body: string) => Promise<{ id: string }>;
const sendEmail = vi.fn<Send>();

sendEmail.mockResolvedValue({ id: "msg_1" });   // must match Promise<{ id: string }>
// sendEmail.mockResolvedValue({ id: 1 });       // ✗ number not assignable to string
the signature constrains both the call args and the resolved value
TypeScript · example 2 of 4 — mock a module, re-type with vi.mocked
import { getRate } from "../fx.js";
vi.mock("../fx.js");                    // auto-mocks every export

const mockedGetRate = vi.mocked(getRate);   // typed as the REAL signature
mockedGetRate.mockReturnValue(1.25);         // return type checked against getRate
vi.mocked recovers getRate’s signature so the mock API stays type-checked
TypeScript · example 3 of 4 — inject a typed double (no module mock)
interface Clock { now(): Date }
const fixedClock = { now: () => new Date("2026-07-05T00:00:00Z") } satisfies Clock;

const svc = new TokenService(fixedClock);   // deterministic time in tests
satisfies checks the stub against Clock while keeping its exact literal type
TypeScript · example 4 of 4 — assert calls, reset between tests
beforeEach(() => vi.clearAllMocks());   // no call-count bleed between tests

test("notifies on signup", async () => {
  await signup({ email: "a@b.io" }, sendEmail);
  expect(sendEmail).toHaveBeenCalledWith("a@b.io", expect.any(String));
  expect(sendEmail).toHaveBeenCalledTimes(1);
});
toHaveBeenCalledWith is checked against the mock’s typed parameters
Common Mistakes:
❌ A bare vi.fn() returning any — the mock can return a shape the real function never would, so the test passes but prod breaks.
✅ Give vi.fn a signature (vi.fn<Send>()) so args and return are enforced.
❌ Forgetting to reset mocks — call counts and implementations leak into later tests, causing spooky failures.
✅ clearAllMocks/resetAllMocks in beforeEach so each test starts clean.
Key Rules:
  • Parameterize vi.fn<Signature>() (or MockedFunction<typeof real>) so args and return type are enforced.
  • After vi.mock, wrap imports in vi.mocked() to recover the real signature on the mock API.
  • Prefer injecting a small typed interface (Clock, Mailer) over module mocking — satisfies keeps the stub honest.
  • Clear/reset mocks in beforeEach so each test starts from a clean slate.
Practice Exercises:
1Replace a module mock with dependency injection of a typed Mailer interface; note which is easier to type.
2Type a partial mock: vi.mock(path, factory) that mocks one export and keeps the rest with importActual.
3Use vi.spyOn(obj, "method") and confirm the spy’s return type matches the original method.
Connections: The typed signatures reuse function typing and typeof; satisfies is the same operator from the satisfies card. Mocked return values are built with the factories of Concept 2, and injecting a typed interface is dependency injection surfacing again.

4. Metrics with prom-client Node.js + TypeScript
Tracing shows one request’s path (07-04); metrics show aggregate health over time — request rate, error ratio, latency percentiles. prom-client is the standard Prometheus client; typing your Counter, Gauge, and Histogram — especially their label sets — stops the classic bug where a mislabeled metric silently splinters into cardinality dust.

Declare each metric with its label names as the generic type argument (new Counter<"method" | "status">). Then inc/observe require exactly those label keys, checked at compile time. A Histogram with explicit buckets yields latency percentiles, not just an average. Expose /metrics returning register.metrics() with the correct register.contentType.

The cardinal rule is label cardinality: label with bounded sets (method, status, route template) — never a user id or a raw URL with ids, or Prometheus creates a series per value and melts.

1
Declare each metric with its label names as the generic type argument.
2
inc/observe with a typed label object — the keys are enforced.
3
Expose register.metrics() on /metrics; keep labels low-cardinality.
TypeScript · example 1 of 4 — a label-typed counter
import { Counter, Histogram, register } from "prom-client";

const httpRequests = new Counter<"method" | "status">({
  name: "http_requests_total",
  help: "total HTTP requests",
  labelNames: ["method", "status"],
});

httpRequests.inc({ method: "GET", status: "200" });
// httpRequests.inc({ method: "GET", path: "/x" });  // ✗ 'path' is not a declared label
the label object’s keys are constrained to the declared label names
TypeScript · example 2 of 4 — a histogram for latency percentiles
const httpDuration = new Histogram<"route">({
  name: "http_request_duration_seconds",
  help: "request latency",
  labelNames: ["route"],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 2],
});

const end = httpDuration.startTimer({ route: "/orders" });
// … handle request …
end();    // records elapsed seconds into the matching bucket
explicit buckets give you p50/p95/p99, not just a mean
TypeScript · example 3 of 4 — a middleware recording both
app.use((req, res, next) => {
  const end = httpDuration.startTimer({ route: req.route?.path ?? "unknown" });
  res.on("finish", () => {
    httpRequests.inc({ method: req.method, status: String(res.statusCode) });
    end();
  });
  next();
});
route TEMPLATE (not the concrete path) keeps label cardinality bounded
TypeScript · example 4 of 4 — expose the scrape endpoint
app.get("/metrics", async (_req, res) => {
  res.set("Content-Type", register.contentType);
  res.send(await register.metrics());
});
Prometheus scrapes /metrics; register.contentType is the exposition format
Common Mistakes:
❌ Putting a high-cardinality value (user id, request id, raw URL with ids) in a label — Prometheus makes a series per value and melts.
✅ Label with the bounded route template (/orders/:id), never the concrete path.
❌ Re-constructing the same metric on every request — prom-client throws on duplicate registration.
✅ Instantiate each metric once at module scope and reuse it.
Key Rules:
  • Parameterize metrics with their label names (new Counter<Labels>) so inc/observe label objects are type-checked.
  • Keep labels low-cardinality: bounded sets (method, status, route template) only.
  • Instantiate each metric once at module scope; scrape via register.metrics() with register.contentType.
  • Use a Histogram with explicit buckets for latency so you get percentiles, not just an average.
Practice Exercises:
1Add an in-flight Gauge<"route"> that inc()s at request start and dec()s on finish.
2Type a recordDbQuery(op: "select" | "insert", ms: number) helper wrapping a labeled Histogram.
3Add default process metrics with collectDefaultMetrics and confirm they appear at /metrics.
Connections: Metrics complete the observability triad with OpenTelemetry tracing (07-04) and structured pino logs (07-01). The label-name generic is the same string-literal-union-as-a-type-parameter trick behind typed event emitters, and /metrics reuses the Express handler typing (06-30).
5. Typed Feature Flags Node.js + TypeScript
Shipping behind flags decouples deploy from release — but a stringly-typed getFlag("newChekout") typo fails silently and can ship to everyone. A typed flag registry makes the flag key, its value type, and its default compile-time facts: an unknown flag or a wrong-typed default is a build error, and boolean vs multivariate flags stay distinct.

Define one FlagSchema mapping each key to its value type; derive FlagKey = keyof FlagSchema so keys can’t be misspelled, and write evaluate<K extends FlagKey>(key: K, ctx): FlagSchema[K] whose return type is indexed from the schema. Booleans gate a feature; string/enum variants drive A/B/n rollout — and you switch on them exhaustively. Evaluation takes a typed context so targeting rules reference real attributes, with a typed default per flag.

Because the return type is FlagSchema[K], a boolean flag stays boolean and a variant flag stays its union — you can’t accidentally treat one as the other.

1
Declare a FlagSchema mapping keys → value types (+ typed defaults).
2
Type evaluate<K extends FlagKey>(key, ctx): FlagSchema[K] via indexed access.
3
Gate on booleans; switch exhaustively on variant unions.
TypeScript · example 1 of 4 — the flag registry as a typed record
interface FlagSchema {
  new_checkout: boolean;
  search_algo: "v1" | "v2" | "v3";
  max_upload_mb: number;
}

const defaults: FlagSchema = { new_checkout: false, search_algo: "v1", max_upload_mb: 10 };
type FlagKey = keyof FlagSchema;   // "new_checkout" | "search_algo" | "max_upload_mb"
one schema is the single source of truth for keys, value types, and defaults
TypeScript · example 2 of 4 — evaluation typed by indexed access
interface Ctx { userId: string; groups: string[] }

function evaluate<K extends FlagKey>(key: K, ctx: Ctx): FlagSchema[K] {
  const rule = rules[key];
  return (rule ? rule(ctx) : defaults[key]) as FlagSchema[K];
}

const algo = evaluate("search_algo", ctx);   // algo: "v1" | "v2" | "v3"
const on   = evaluate("new_checkout", ctx);  // on: boolean
// evaluate("nope", ctx);                      // ✗ 'nope' is not a FlagKey
the return type is indexed from the schema — booleans stay boolean, variants stay unions
TypeScript · example 3 of 4 — gate a boolean, switch a variant exhaustively
if (evaluate("new_checkout", ctx)) renderNewCheckout();

switch (evaluate("search_algo", ctx)) {
  case "v1": return searchV1();
  case "v2": return searchV2();
  case "v3": return searchV3();
  default: return assertNever(evaluate("search_algo", ctx));  // add a variant → compile error
}
the exhaustive switch forces every variant to be handled
TypeScript · example 4 of 4 — typed targeting rules via a mapped type
type Rule<K extends FlagKey> = (ctx: Ctx) => FlagSchema[K];

const rules: { [K in FlagKey]?: Rule<K> } = {
  new_checkout: (ctx) => ctx.groups.includes("beta"),
  max_upload_mb: (ctx) => (ctx.groups.includes("pro") ? 100 : 10),
};
each rule’s return type is pinned to its flag’s value type by the mapped type
Common Mistakes:
❌ getFlag(key: string): any — a key typo or a wrong-typed comparison (=== "V2") never errors and quietly changes behavior.
✅ Index the value type from a FlagSchema so keys and value types are compile-time checked.
❌ Treating a multivariate flag as a boolean (if (flag)) — "v1" is truthy, so the gate is always on.
✅ switch on the variant exhaustively with an assertNever default.
Key Rules:
  • Model all flags in one FlagSchema; derive FlagKey = keyof it so keys can’t be misspelled.
  • Return FlagSchema[K] via indexed access so each flag’s value type is exact (boolean vs union vs number).
  • Give every flag a typed default and switch on variants with an assertNever default.
  • Keep the evaluation context typed so targeting rules reference real attributes.

3Make defaults the source so adding a key to FlagSchema forces adding its default (no missing-key gap).
