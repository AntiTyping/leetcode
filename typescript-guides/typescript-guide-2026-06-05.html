R1. Decorators (TC39) Functions & Patterns rep 0 · last reviewed 2026-06-03 · 2 days ago
TC39 stage-3 decorators (TS 5.0+) are functions that receive the decorated value and a context object. They can wrap classes, methods, accessors, and fields. Unlike legacy experimental decorators, they don't use reflect-metadata — context provides the metadata API directly via context.metadata.
Key Pattern 
// TC39 method decorator: logs every call
function logged<This, Args extends any[], Return>(
  target: (this: This, ...args: Args) => Return,
  context: ClassMethodDecoratorContext<This, (this: This, ...args: Args) => Return>
) {
  const name = String(context.name);
  return function (this: This, ...args: Args): Return {
    console.log(`Calling ${name} with`, args);
    const result = target.call(this, ...args);
    console.log(`${name} returned`, result);
    return result;
  };
}

class Calculator {
  @logged
  add(a: number, b: number): number {
    return a + b;
  }
}

new Calculator().add(2, 3);
// Calling add with [2, 3]
// add returned 5
Challenge Exercise

Write a @throttle(ms: number) decorator factory that prevents a method from being called more than once per ms milliseconds. The decorator should return the cached result for subsequent calls within the window. Type the decorator so it works on any method signature.

#1 Pitfall: Confusing TC39 decorators with legacy experimental decorators. TC39 decorators receive (target, context) where target is the decorated value itself, not the class prototype. Legacy decorators receive (target, propertyKey, descriptor). They are not interchangeable — check "experimentalDecorators" in your tsconfig.
R2. Module Systems Modules & Project Structure rep 0 · last reviewed 2026-06-03 · 2 days ago
ESM (import/export) is the standard; CJS (require/module.exports) is legacy Node.js. TypeScript's "module" setting controls emit format. Use "moduleResolution": "bundler" for modern bundlers or "node16" for Node.js. The "type": "module" field in package.json determines whether .js files are treated as ESM or CJS.
Key Pattern 
// ESM: static imports (tree-shakeable, async-ready)
import { readFile } from "node:fs/promises";
import type { User } from "./types.js";  // type-only import (erased at runtime)

export function loadUser(path: string): Promise<User> {
  return readFile(path, "utf-8").then(JSON.parse);
}

// Dynamic import (code-splitting)
async function loadChart() {
  const { Chart } = await import("./chart.js");
  return new Chart();
}

// CJS interop: default import from a CJS module
import express from "express";  // works with esModuleInterop: true
Challenge Exercise

Create a small project with "type": "module" in package.json. Write an ESM file that dynamically imports a CJS dependency (e.g., path) and a local module. Configure tsconfig.json with "module": "node16" and "moduleResolution": "node16". Verify that .js extensions are required in import paths and that import type is erased from the output.

#1 Pitfall: Omitting the .js extension in import paths when targeting Node.js ESM. Node requires explicit extensions: import { foo } from "./utils.js" (not "./utils"). TypeScript uses .js even when the source file is .ts because imports refer to the output file.
R3. Declaration Files Modules & Project Structure rep 0 · last reviewed 2026-06-03 · 2 days ago
Declaration files (.d.ts) provide type information without runtime code. They describe the shape of JavaScript libraries so TypeScript can type-check against them. Use declare module "name" for ambient modules, declare global to extend global scope, and @types/* packages from DefinitelyTyped for community-maintained declarations.
Key Pattern 
// Ambient declaration for a JS library without types
// global.d.ts
declare module "legacy-lib" {
  export function doStuff(input: string): number;
  export interface LegacyConfig {
    verbose: boolean;
    retries: number;
  }
}

// Extending an existing module's types
declare module "express" {
  interface Request {
    userId?: string;       // Add custom property to Express Request
    sessionToken?: string;
  }
}

// Extending the global scope
declare global {
  interface Window {
    analytics: { track(event: string, data: unknown): void };
  }
}
Challenge Exercise

Write a .d.ts file that declares types for an imaginary analytics library with functions init(apiKey: string): void, track(event: string, properties?: Record<string, unknown>): void, and identify(userId: string, traits?: { email?: string; name?: string }): void. Then extend the global Window interface to include the library as window.analytics.

#1 Pitfall: Forgetting that a .d.ts file with top-level import or export becomes a module, not an ambient declaration. If you need declare global inside a module-scoped .d.ts, you must wrap it in declare global { ... }. Without any imports/exports, the file is automatically ambient.
R4. tsconfig.json Deep Dive Modules & Project Structure rep 0 · last reviewed 2026-06-03 · 2 days ago
The strict flag enables all strict-family checks (strictNullChecks, noImplicitAny, strictFunctionTypes, etc.). "moduleResolution": "bundler" for bundled apps, "node16" for Node.js libraries. "target" controls JS output level, "lib" controls available global types. Use "paths" for import aliases and "baseUrl" for non-relative module resolution.
Key Pattern 
// Modern strict tsconfig for a Node.js project
{
  "compilerOptions": {
    "strict": true,                    // All strict checks
    "target": "ES2022",                // Node 18+ native features
    "module": "node16",                // ESM + CJS interop for Node
    "moduleResolution": "node16",      // Matches module setting
    "esModuleInterop": true,           // import express from "express"
    "skipLibCheck": true,              // Faster builds, skip .d.ts checks
    "forceConsistentCasingInFileNames": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "paths": {
      "@/*": ["./src/*"]               // Import alias: @/utils -> src/utils
    }
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}
Challenge Exercise

Start from an empty tsconfig.json with only "strict": true. Progressively add settings for a library that targets both Node.js and browsers: set "target", "module", "moduleResolution", "declaration", "lib" (include both "ES2022" and "DOM"), and "paths" for a @/ alias. Verify each setting's effect by checking compiler output.

#1 Pitfall: Mismatching "module" and "moduleResolution". When "module": "node16", you must use "moduleResolution": "node16". Using "moduleResolution": "node" (the old algorithm) with modern module settings leads to incorrect resolution of exports fields in package.json.
R5. Namespaces vs Modules Modules & Project Structure rep 0 · last reviewed 2026-06-03 · 2 days ago
Modules (ESM/CJS) are the standard for organizing TypeScript code — each file is a module. Namespaces (namespace Foo { ... }) are a legacy TypeScript-only feature for grouping code without a module system. Prefer modules for all new code. Namespaces still appear in .d.ts files and legacy codebases but should not be used in modern application code.
Key Pattern 
// AVOID: Namespace (legacy pattern)
namespace Validation {
  export interface Validator {
    validate(input: string): boolean;
  }
  export class EmailValidator implements Validator {
    validate(input: string) { return input.includes("@"); }
  }
}

// PREFER: ES Module (modern pattern)
// validators.ts
export interface Validator {
  validate(input: string): boolean;
}
export class EmailValidator implements Validator {
  validate(input: string) { return input.includes("@"); }
}

// Only valid use of namespaces: declaration merging
// Extend a class or function with additional types
class MyArray<T> { /* ... */ }
namespace MyArray {
  export type Callback<T> = (item: T, index: number) => void;
}
Challenge Exercise

Take a legacy codebase snippet that uses namespace Utils { export function ... } across three files. Refactor it into ES modules with proper import/export statements. Verify that all cross-file references resolve correctly and that the namespace keyword is no longer needed.

#1 Pitfall: Using namespaces in new code to avoid import statements. This creates global scope pollution and prevents tree-shaking. Each file should be its own module with explicit imports — the bundler handles combining them.
R6. Function Overloads Functions & Patterns rep 1 · last reviewed 2026-06-04 · 1 day ago
Overloads let a single function have multiple call signatures. Write each overload signature above the implementation signature. The implementation signature must be compatible with all overloads but is not directly callable — callers only see the overload signatures. Order overloads from most specific to most general.
Key Pattern 
// Overload signatures (what callers see)
function parseValue(input: string, type: "number"): number;
function parseValue(input: string, type: "boolean"): boolean;
function parseValue(input: string, type: "date"): Date;
function parseValue(input: string, type: string): unknown;
// Implementation signature (must satisfy all overloads)
function parseValue(input: string, type: string): unknown {
  switch (type) {
    case "number":  return Number(input);
    case "boolean": return input === "true";
    case "date":    return new Date(input);
    default:        return input;
  }
}

const n = parseValue("42", "number");     // number
const b = parseValue("true", "boolean");  // boolean
const d = parseValue("2026-01-01", "date"); // Date
Challenge Exercise

Write an overloaded flatten function: when given number[][] it returns number[]; when given string[][] it returns string[]; when given a generic T[][] it returns T[]. Verify that the return type narrows correctly for each specific input type.

#1 Pitfall: Ordering overloads from most general to most specific. TypeScript resolves overloads top-to-bottom and picks the first match. Putting string before a literal type like "number" means the literal overload is never reached.

1. Immutability Patterns Runtime Patterns
Mutable state is the #1 source of bugs in large codebases — functions silently modify objects, arrays grow extra elements, and config objects change between reads. TypeScript's readonly modifier and as const assertions catch mutation at compile time, long before it reaches production. Master these and you eliminate an entire class of bugs.

TypeScript provides multiple layers of immutability, from shallow property protection to deep recursive freezing. The key is choosing the right level: readonly on individual properties, Readonly<T> for shallow object immutability, ReadonlyArray<T> for arrays, as const for deeply immutable literals, and custom DeepReadonly<T> for recursive protection.

  • 2026-07-04 (Day 31)
  • readonly modifier — prevents reassignment of individual properties
  • Readonly<T> — utility type that makes all properties of T readonly (shallow)
  • ReadonlyArray<T> / readonly T[] — array without mutating methods (push, pop, splice, etc.)
  • as const — infers the narrowest possible type with all properties readonly recursively
  • Object.freeze — runtime shallow freeze; TypeScript types it as Readonly<T>
  • DeepReadonly<T> — recursive type for deep immutability (not built-in, must be defined)
TypeScript — readonly Modifier on Properties
// readonly prevents reassignment after initialization
interface User {
  readonly id: string;
  readonly email: string;
  name: string;           // mutable
}

const user: User = { id: "u1", email: "alice@test.com", name: "Alice" };
user.name = "Bob";        // OK: name is mutable
// user.id = "u2";        // Error: Cannot assign to 'id' because it is a read-only property

// readonly in class constructors (parameter properties)
class Config {
  constructor(
    readonly host: string,
    readonly port: number,
    readonly debug: boolean = false
  ) {}
}

const cfg = new Config("localhost", 3000);
console.log(cfg.host);   // "localhost"
// cfg.port = 8080;       // Error: Cannot assign to 'port' because it is a read-only property

// readonly in function parameters prevents accidental mutation
function processUsers(users: readonly User[]): string[] {
  // users.push(newUser);  // Error: Property 'push' does not exist on type 'readonly User[]'
  // users.sort();          // Error: Property 'sort' does not exist on type 'readonly User[]'
  return users.map(u => u.name);  // OK: map returns a new array
}
TypeScript — Readonly<T> and ReadonlyArray<T>
interface Settings {
  theme: "dark" | "light";
  language: string;
  notifications: { email: boolean; push: boolean };
}

// Readonly<T> makes all top-level properties readonly (shallow!)
type FrozenSettings = Readonly<Settings>;
// {
//   readonly theme: "dark" | "light";
//   readonly language: string;
//   readonly notifications: { email: boolean; push: boolean };  // still mutable inside!
// }

const settings: FrozenSettings = {
  theme: "dark",
  language: "en",
  notifications: { email: true, push: false },
};

// settings.theme = "light";               // Error: readonly
settings.notifications.email = false;       // OK! Readonly is shallow

// ReadonlyArray<T> removes all mutating methods
const ids: ReadonlyArray<number> = [1, 2, 3];
// ids.push(4);      // Error: Property 'push' does not exist
// ids[0] = 10;      // Error: Index signature in type 'readonly number[]' only permits reading
const first = ids[0]; // OK: reading is fine

// Shorthand syntax: readonly T[]
function sum(numbers: readonly number[]): number {
  return numbers.reduce((acc, n) => acc + n, 0);
}

// ReadonlyMap and ReadonlySet
const cache: ReadonlyMap<string, number> = new Map([["a", 1], ["b", 2]]);
const visited: ReadonlySet<string> = new Set(["home", "about"]);
// cache.set("c", 3);    // Error: Property 'set' does not exist
// visited.add("contact"); // Error: Property 'add' does not exist
console.log(cache.get("a"), visited.has("home"));
1 true
TypeScript — as const for Deep Immutability
// Without as const: types are widened
const colors = ["red", "green", "blue"];
// type: string[]

// With as const: types are narrowed and readonly all the way down
const colors2 = ["red", "green", "blue"] as const;
// type: readonly ["red", "green", "blue"]

// as const on objects: all properties become readonly with literal types
const config = {
  api: {
    baseUrl: "https://api.example.com",
    timeout: 5000,
  },
  retries: 3,
  features: ["auth", "logging"] as const,
} as const;
// type: {
//   readonly api: { readonly baseUrl: "https://api.example.com"; readonly timeout: 5000 };
//   readonly retries: 3;
//   readonly features: readonly ["auth", "logging"];
// }

// config.retries = 5;             // Error: readonly
// config.api.timeout = 3000;      // Error: readonly (deep!)
// config.features.push("cache");  // Error: readonly tuple

// Practical use: derive union types from const arrays
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = typeof ROLES[number]; // "admin" | "editor" | "viewer"

// Practical use: enum-like objects
const HttpStatus = {
  OK: 200,
  NotFound: 404,
  ServerError: 500,
} as const;
type StatusCode = typeof HttpStatus[keyof typeof HttpStatus]; // 200 | 404 | 500

console.log(HttpStatus.OK);
200
TypeScript — Object.freeze and DeepReadonly<T>
// Object.freeze: runtime + compile-time protection (but shallow!)
const frozen = Object.freeze({ name: "Alice", scores: [90, 85, 92] });
// frozen.name = "Bob";    // Error: readonly at compile time + throws at runtime (strict mode)
frozen.scores.push(100);   // No compile error! freeze is shallow. Mutates at runtime.

// DeepReadonly: recursive type for full compile-time protection
type DeepReadonly<T> =
  T extends (infer U)[]
    ? ReadonlyArray<DeepReadonly<U>>
    : T extends object
      ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
      : T;

interface AppState {
  user: { name: string; preferences: { theme: string; fontSize: number } };
  items: { id: number; tags: string[] }[];
}

type ImmutableState = DeepReadonly<AppState>;

const state: ImmutableState = {
  user: { name: "Alice", preferences: { theme: "dark", fontSize: 14 } },
  items: [{ id: 1, tags: ["important"] }],
};

// state.user.name = "Bob";                      // Error: readonly
// state.user.preferences.theme = "light";       // Error: readonly (deep!)
// state.items[0].tags.push("urgent");           // Error: readonly array (deep!)
// state.items.push({ id: 2, tags: [] });        // Error: readonly array

// When to use which:
// readonly property    -> individual fields that shouldn't change
// Readonly<T>          -> shallow freeze for function params, return types
// as const             -> literal config objects, enum-like constants
// DeepReadonly<T>      -> state management (Redux store, React context)
// Object.freeze        -> runtime enforcement (defense against JS callers)
TypeScript — Readonly in Function Design
// Accept readonly, return mutable: maximum flexibility
function sortedCopy(items: readonly number[]): number[] {
  return [...items].sort((a, b) => a - b);  // spread creates a new mutable array
}

const original: readonly number[] = [3, 1, 4, 1, 5];
const sorted = sortedCopy(original);  // [1, 1, 3, 4, 5]
// original is unchanged

// readonly arrays are assignable TO mutable (but not vice versa)
const mutableArr: number[] = [1, 2, 3];
const readonlyArr: readonly number[] = mutableArr;   // OK: mutable -> readonly
// const backToMutable: number[] = readonlyArr;       // Error: readonly -> mutable

// Pattern: config objects as readonly parameters
interface DatabaseConfig {
  host: string;
  port: number;
  credentials: { user: string; password: string };
}

function connect(config: Readonly<DatabaseConfig>): void {
  // config.port = 5432;            // Error: can't modify
  console.log(`Connecting to ${config.host}:${config.port}`);
}

const dbConfig = { host: "localhost", port: 5432, credentials: { user: "admin", password: "secret" } };
connect(dbConfig);  // mutable object passed as readonly parameter: OK
Connecting to localhost:5432
Common Mistakes
❌ Assuming Readonly<T> protects nested objects:
const cfg: Readonly<Config> = ...; cfg.nested.value = "changed"; — compiles without error because Readonly is shallow
✅ Use DeepReadonly<T> or as const when you need deep immutability. Readonly<T> only protects top-level properties.
❌ Using Object.freeze and assuming TypeScript catches all mutations:
Object.freeze is shallow at both runtime and type level — nested objects are still mutable
✅ Combine Object.freeze with DeepReadonly<T> for compile-time safety, or use a deep-freeze library for runtime safety on nested data.
Key Rules
  • readonly is a compile-time-only concept — it prevents reassignment in TypeScript but has zero runtime effect (unlike Object.freeze)
  • Readonly<T>, ReadonlyArray<T>, and Object.freeze are all shallow — use as const or DeepReadonly<T> for nested structures
  • Accept readonly parameters in functions and return mutable values — this gives callers maximum flexibility while protecting the input
Exercises

Derive union types for allowed values (e.g., type Environment = typeof CONFIG.environments[number]). Write a function that accepts only valid environment strings.
3 Write a freeze<T extends object>(obj: T): DeepReadonly<T> function that recursively calls Object.freeze on all nested objects and arrays. Type the return as DeepReadonly<T> so both compile-time and runtime immutability are enforced. Test that mutating any nested property throws at runtime and fails at compile time.

2. Class Patterns Runtime Patterns
Classes are unavoidable in TypeScript — frameworks like Angular, NestJS, and TypeORM are built on them. Understanding abstract classes, private fields, static blocks, and mixins lets you model complex hierarchies correctly, while knowing when to choose plain objects keeps your code simple where classes add unnecessary ceremony.

TypeScript classes add powerful typing on top of JavaScript classes. The key patterns are: abstract classes (partial implementations that enforce a contract on subclasses), private fields (true runtime encapsulation with # vs compile-time-only private), static blocks (class initialization logic), and mixins (composition over inheritance). Knowing when to use classes vs plain typed objects is the most important decision.

  • abstract classes — can't be instantiated, define abstract methods that subclasses must implement
  • #field (ES private) — true runtime privacy, not accessible even via bracket notation
  • private field (TS private) — compile-time only, accessible at runtime via bracket notation
  • static { ... } blocks — run once when the class is defined, for complex initialization
  • Mixins — compose behaviors from multiple sources using Constructor<T> pattern
  • implements vs extends — interface conformance vs inheritance chain
TypeScript — Abstract Classes and Methods
// Abstract class: partial implementation + contract
abstract class Shape {
  constructor(public readonly color: string) {}

  // Abstract methods: subclasses MUST implement these
  abstract area(): number;
  abstract perimeter(): number;

  // Concrete methods: shared implementation
  describe(): string {
    return `${this.color} shape: area=${this.area().toFixed(2)}, perimeter=${this.perimeter().toFixed(2)}`;
  }
}

// new Shape("red");  // Error: Cannot create an instance of an abstract class

class Circle extends Shape {
  constructor(color: string, public readonly radius: number) {
    super(color);
  }
  area(): number { return Math.PI * this.radius ** 2; }
  perimeter(): number { return 2 * Math.PI * this.radius; }
}

class Rectangle extends Shape {
  constructor(color: string, public readonly width: number, public readonly height: number) {
    super(color);
  }
  area(): number { return this.width * this.height; }
  perimeter(): number { return 2 * (this.width + this.height); }
}

// Polymorphism: Shape[] accepts any subclass
const shapes: Shape[] = [new Circle("red", 5), new Rectangle("blue", 4, 6)];
shapes.forEach(s => console.log(s.describe()));
red shape: area=78.54, perimeter=31.42 blue shape: area=24.00, perimeter=20.00
TypeScript — Private Fields: # (Runtime) vs private (Compile-time)
class BankAccount {
  // ES private field: true runtime encapsulation
  #balance: number;

  // TS private: compile-time only, accessible at runtime via bracket notation
  private accountType: string;

  constructor(initialBalance: number, type: string) {
    this.#balance = initialBalance;
    this.accountType = type;
  }

  deposit(amount: number): void {
    if (amount <= 0) throw new Error("Amount must be positive");
    this.#balance += amount;
  }

  getBalance(): number {
    return this.#balance;
  }
}

const account = new BankAccount(1000, "savings");
account.deposit(500);
console.log(account.getBalance());   // 1500

// account.#balance;                 // SyntaxError at runtime: private field
// account.accountType;              // TS error, but works at runtime:
// (account as any).accountType;     // "savings" (TS private leaks!)
// (account as any)["#balance"];     // undefined (ES private is truly hidden)

// When to use which:
// #field  -> library APIs, security-sensitive data, anything exposed to JS callers
// private -> application code where compile-time checks are sufficient

TypeScript — Static Blocks and Members
class DatabasePool {
  static #instance: DatabasePool | null = null;
  static readonly MAX_CONNECTIONS = 10;
  #connections: string[] = [];

  // Static block: runs once when the class is loaded
  static {
    console.log("DatabasePool class initialized");
    // Complex initialization: read env vars, validate config, etc.
    if (!process.env.DATABASE_URL) {
      console.warn("DATABASE_URL not set, using default");
    }
  }

  private constructor() {}  // Prevent direct instantiation

  static getInstance(): DatabasePool {
    if (!DatabasePool.#instance) {
      DatabasePool.#instance = new DatabasePool();
    }
    return DatabasePool.#instance;
  }

  connect(name: string): boolean {
    if (this.#connections.length >= DatabasePool.MAX_CONNECTIONS) return false;
    this.#connections.push(name);
    return true;
  }

  get connectionCount(): number {
    return this.#connections.length;
  }
}

const pool = DatabasePool.getInstance();
pool.connect("conn-1");
console.log(pool.connectionCount, DatabasePool.MAX_CONNECTIONS);
DatabasePool class initialized DATABASE_URL not set, using default 1 10
TypeScript — Mixins: Composition over Inheritance
// The Constructor type: any class constructor
type Constructor<T = {}> = new (...args: any[]) => T;

// Mixin 1: add timestamping behavior
function Timestamped<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    createdAt = new Date();
    updatedAt = new Date();

    touch() {
      this.updatedAt = new Date();
    }
  };
}

// Mixin 2: add soft-delete behavior
function SoftDeletable<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    deletedAt: Date | null = null;

    softDelete() {
      this.deletedAt = new Date();
    }

    get isDeleted(): boolean {
      return this.deletedAt !== null;
    }
  };
}

// Base entity
class Entity {
  constructor(public id: string) {}
}

// Compose multiple mixins
const TimestampedDeletableEntity = SoftDeletable(Timestamped(Entity));

class User extends TimestampedDeletableEntity {
  constructor(id: string, public name: string) {
    super(id);
  }
}

const user = new User("u1", "Alice");
user.touch();        // from Timestamped mixin
user.softDelete();   // from SoftDeletable mixin
console.log(user.id, user.name, user.isDeleted);
u1 Alice true
TypeScript — implements vs extends, and When to Use Classes
// implements: promise to conform to an interface (no code inherited)
interface Serializable {
  serialize(): string;
}

interface Loggable {
  log(message: string): void;
}

// A class can implement multiple interfaces
class UserModel implements Serializable, Loggable {
  constructor(public name: string, public email: string) {}

  serialize(): string {
    return JSON.stringify({ name: this.name, email: this.email });
  }

  log(message: string): void {
    console.log(`[UserModel] ${message}`);
  }
}

// extends: inherit implementation (single inheritance only)
class AdminModel extends UserModel {
  constructor(name: string, email: string, public permissions: string[]) {
    super(name, email);
  }
}

// WHEN TO USE CLASSES vs PLAIN OBJECTS:
// Use classes when you need:
// - instanceof checks at runtime
// - Private state (#field)
// - Abstract contracts for subclasses
// - Framework requirements (Angular services, NestJS controllers)

// Use plain objects + types when you need:
// - Simple data transfer (DTOs, API responses)
// - Functional composition
// - Tree-shakeable code (classes aren't fully tree-shakeable)

3. Proxy & Reflect Typing Runtime Patterns
Proxies intercept fundamental object operations (get, set, has, delete) at runtime, enabling patterns like validation layers, observable objects, lazy loading, and logging — all transparent to consumers. TypeScript provides ProxyHandler<T> to type each trap, but understanding its limitations is critical: proxies add dynamic behavior that the type system cannot fully track.

A Proxy wraps a target object and intercepts operations via "traps" defined in a handler. TypeScript types the handler with ProxyHandler<T>, where each trap method receives typed arguments. The Reflect API provides the default behavior for each trap — call it when you want to perform the original operation after your custom logic.

  • ProxyHandler<T> — typed handler interface with optional trap methods
  • get trap — intercepts property reads: obj.prop, obj["prop"]
  • set trap — intercepts property writes: obj.prop = value
  • has trap — intercepts in operator: "prop" in obj
  • apply trap — intercepts function calls (proxy must wrap a function)
  • Reflect.get, Reflect.set — typed alternatives to bracket access, used inside traps for default behavior
TypeScript — Typed Proxy with get and set Traps
interface User {
  name: string;
  email: string;
  age: number;
}

// Create a logging proxy that tracks all reads and writes
function createLoggingProxy<T extends object>(target: T): T {
  const handler: ProxyHandler<T> = {
    get(target, prop, receiver) {
      console.log(`GET ${String(prop)}`);
      return Reflect.get(target, prop, receiver);
    },
    set(target, prop, value, receiver) {
      console.log(`SET ${String(prop)} = ${JSON.stringify(value)}`);
      return Reflect.set(target, prop, value, receiver);
    },
  };

  return new Proxy(target, handler);
}

const user = createLoggingProxy<User>({
  name: "Alice",
  email: "alice@test.com",
  age: 30,
});

user.name;         // logs: GET name
user.age = 31;     // logs: SET age = 31
console.log(user.age);
GET name SET age = 31 GET age 31
TypeScript — Validation Proxy
interface Product {
  name: string;
  price: number;
  quantity: number;
}

type Validator<T> = {
  [K in keyof T]?: (value: T[K]) => boolean;
};

function createValidatedProxy<T extends object>(
  target: T,
  validators: Validator<T>
): T {
  const handler: ProxyHandler<T> = {
    set(target, prop, value, receiver) {
      const key = prop as keyof T;
      const validate = validators[key];

      if (validate && !validate(value)) {
        throw new Error(`Invalid value for ${String(prop)}: ${JSON.stringify(value)}`);
      }

      return Reflect.set(target, prop, value, receiver);
    },
  };

  return new Proxy(target, handler);
}

const product = createValidatedProxy<Product>(
  { name: "Widget", price: 9.99, quantity: 100 },
  {
    price: (v) => v > 0,
    quantity: (v) => Number.isInteger(v) && v >= 0,
    name: (v) => v.length > 0 && v.length <= 100,
  }
);

product.price = 14.99;    // OK
product.quantity = 50;     // OK
console.log(product.price, product.quantity);

try {
  product.price = -5;     // Throws: Invalid value for price
} catch (e) {
  console.log((e as Error).message);
}
14.99 50 Invalid value for price: -5
TypeScript — Observable Proxy with Change Listeners
type ChangeListener<T> = (
  prop: keyof T & string,
  oldValue: T[keyof T],
  newValue: T[keyof T]
) => void;

function createObservable<T extends object>(
  target: T,
  onChange: ChangeListener<T>
): T {
  const handler: ProxyHandler<T> = {
    set(target, prop, value, receiver) {
      const key = prop as keyof T & string;
      const oldValue = Reflect.get(target, prop, receiver) as T[keyof T];
      const result = Reflect.set(target, prop, value, receiver);
      if (result && oldValue !== value) {
        onChange(key, oldValue, value as T[keyof T]);
      }
      return result;
    },
  };

  return new Proxy(target, handler);
}

interface AppSettings {
  theme: "dark" | "light";
  fontSize: number;
  language: string;
}

const settings = createObservable<AppSettings>(
  { theme: "dark", fontSize: 14, language: "en" },
  (prop, oldVal, newVal) => {
    console.log(`${prop} changed: ${JSON.stringify(oldVal)} -> ${JSON.stringify(newVal)}`);
  }
);

settings.theme = "light";
settings.fontSize = 16;
settings.language = "en";  // same value: no notification

// apply trap: intercept function calls
function createTimedFunction<Args extends any[], R>(
  fn: (...args: Args) => R
): (...args: Args) => R {
  const handler: ProxyHandler<(...args: Args) => R> = {
    apply(target, thisArg, argArray) {
      const start = performance.now();
      const result = Reflect.apply(target, thisArg, argArray);
      const elapsed = (performance.now() - start).toFixed(2);
      console.log(`${fn.name || "anonymous"} took ${elapsed}ms`);
      return result;
    },
  };

  return new Proxy(fn, handler);
}

  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}

const result = timedFib(30);
console.log(`fib(30) = ${result}`);

TypeScript — Proxy Typing Limitations
// LIMITATION 1: TypeScript can't track dynamic proxy behavior
// The proxy type is the same as the target type
interface Config { host: string; port: number }

const handler: ProxyHandler<Config> = {
  get(target, prop) {
    if (prop === "connectionString") {
      return `${target.host}:${target.port}`;  // Dynamic property
    }
    return Reflect.get(target, prop);
  },
};

const config = new Proxy({ host: "localhost", port: 3000 }, handler);
// config.connectionString;  // Error: Property 'connectionString' does not exist
// TypeScript doesn't know about the dynamic property!

// Workaround: type assertion or a broader interface
const connStr = (config as any).connectionString; // "localhost:3000" at runtime

// LIMITATION 2: Reflect.get returns any, not the property type
const val = Reflect.get({ name: "Alice", age: 30 }, "name");
// val: any (not string!)

// LIMITATION 3: Proxy doesn't intercept private fields (#field)
// Private fields bypass the proxy entirely (they use internal slots)

// BEST PRACTICE: Use proxies for:
// - Validation layers (form data, API inputs)
// - Observable state (reactive frameworks)
// - Logging and debugging (development tools)
// - Lazy loading (load data on first access)
// Avoid proxies for:
// - Performance-critical hot paths (each operation has overhead)
// - Type-safe APIs (the type system can't fully represent proxy behavior)
Common Mistakes
❌ Forgetting to call Reflect.set() and return its result in the set trap:
set(target, prop, value) { target[prop] = value; } — must return true/false
✅ Always use return Reflect.set(target, prop, value, receiver) in set traps. It handles the default behavior correctly and returns the required boolean. Direct assignment misses receiver handling.
❌ Assuming proxies intercept private field access (#field):
ES private fields use internal slots, not property access, so the get/set traps never fire
✅ Proxies only intercept operations that go through the normal property protocol. Private fields (#field) bypass the proxy entirely. Use public properties or accessor methods if you need proxy interception.
Key Rules
  • ProxyHandler<T> types the trap methods but the proxy's external type remains T — TypeScript cannot represent dynamic properties added by traps
  • Always use Reflect.* inside traps for default behavior instead of direct property access — it correctly handles the receiver parameter and prototype chains
  • Proxies add runtime overhead to every intercepted operation — use them for cross-cutting concerns (validation, logging, observation) not for high-frequency hot paths

4. Type-Level Programming Advanced Patterns & Performance
TypeScript's type system is Turing-complete — you can compute at the type level. Recursive conditional types, variadic tuple manipulation, and template literal transformations let you encode complex invariants that the compiler enforces. This is the foundation of libraries like Zod, tRPC, and Prisma, which derive types from schemas at compile time.

Type-level programming treats types as a functional programming language: you write "functions" (generic types) that take types as input and produce types as output. The building blocks are recursive conditional types, variadic tuple types, and template literal manipulation. While powerful, readability is the limiting factor — use these techniques when the safety payoff justifies the complexity.

  • Recursive conditional types — a type that references itself (e.g., DeepPartial<T>)
  • Variadic tuple types — [...T, ...U] for concatenating and manipulating tuple types
  • Template literal type manipulation — Split, Join, Replace at the type level
  • Tuple manipulation utilities — Head, Tail, Last, Length
  • Type-level arithmetic — counting with tuple length (limited but useful)
TypeScript — Recursive Conditional Types
// DeepPartial: make all properties optional, recursively
type DeepPartial<T> =
  T extends object
    ? { [K in keyof T]?: DeepPartial<T[K]> }
    : T;

interface Config {
  server: {
    host: string;
    port: number;
    ssl: { cert: string; key: string };
  };
  database: { url: string; pool: { min: number; max: number } };
}

type PartialConfig = DeepPartial<Config>;
// All nested properties are optional:
const update: PartialConfig = {
  server: { ssl: { cert: "new-cert.pem" } },  // only cert, everything else omitted
};

// DeepRequired: opposite of DeepPartial
type DeepRequired<T> =
  T extends object
    ? { [K in keyof T]-?: DeepRequired<T[K]> }
    : T;

// DeepNonNullable: remove null/undefined recursively
type DeepNonNullable<T> =
  T extends object
    ? { [K in keyof T]: DeepNonNullable<NonNullable<T[K]>> }
    : NonNullable<T>;

// Flatten nested objects into dot-notation keys
type FlattenKeys<T, Prefix extends string = ""> =
  T extends object
    ? {
        [K in keyof T & string]: T[K] extends object
          ? FlattenKeys<T[K], `${Prefix}${K}.`>
          : `${Prefix}${K}`;
      }[keyof T & string]
    : never;

type ConfigKeys = FlattenKeys<Config>;
// "server.host" | "server.port" | "server.ssl.cert" | "server.ssl.key"
// | "database.url" | "database.pool.min" | "database.pool.max"
DeepPartial<Config> = { server?: { host?: string; port?: number; ssl?: { cert?: string; key?: string } }; database?: { url?: string; pool?: { min?: number; max?: number } }; }

// Tail: everything except the first element

type Rest = Tail<[string, number, boolean]>;  // [number, boolean]

// Last: last element of a tuple
type Last<T extends unknown[]> = T extends [...any[], infer L] ? L : never;
type End = Last<[string, number, boolean]>;  // boolean

// Length: tuple length as a literal number
type Length<T extends unknown[]> = T["length"];
type Len = Length<[string, number, boolean]>;  // 3

type FirstArg<F> = F extends (arg: infer A, ...rest: any[]) => any ? A : never;
type ReturnOf<F> = F extends (...args: any[]) => infer R ? R : never;

// Split a string type into a tuple of characters/segments
type Split<S extends string, D extends string> =
  S extends `${infer Head}${D}${infer Tail}`
    ? [Head, ...Split<Tail, D>]
    : S extends ""
      ? []
      : [S];

type Parts = Split<"hello-world-foo", "-">;  // ["hello", "world", "foo"]

// Join a tuple of strings into a single string type
type Join<T extends string[], D extends string> =
  T extends []
    ? ""
    : T extends [infer Head extends string]
      ? Head
      : T extends [infer Head extends string, ...infer Tail extends string[]]
        ? `${Head}${D}${Join<Tail, D>}`
        : string;

type Joined = Join<["hello", "world"], "-">;  // "hello-world"

// Replace in a string type
type Replace<
  S extends string,
  From extends string,
  To extends string
> = S extends `${infer Before}${From}${infer After}`
  ? `${Before}${To}${After}`
  : S;

type Fixed = Replace<"hello world", " ", "-">;  // "hello-world"

  S extends `${infer Head}${infer Tail}`

      : `${Head}${CamelToKebab<Tail>}`
    : S;

5. infer Mastery Advanced Patterns & Performance
infer is the "pattern matching" of TypeScript's type system. It lets you extract types from inside other types — pulling the return type from a function, the element type from an array, the resolved type from a Promise, or any other structural position. Every advanced utility type (including TypeScript's built-in ReturnType, Parameters, and Awaited) is built on infer.

infer appears only inside the extends clause of a conditional type. It declares a type variable that TypeScript infers from the structural position. Think of it as a "capture group" in a regex — it matches a part of the type and binds it to a name you can use in the true branch of the conditional.

  • Basic: T extends Promise<infer U> ? U : never — extracts the resolved type from a Promise
  • Function return: T extends (...args: any[]) => infer R ? R : never
  • Function parameters: T extends (...args: infer P) => any ? P : never
  • Array element: T extends (infer U)[] ? U : never
  • Nested infer: extract types from deeply nested structures
  • Multiple infer in the same conditional type
  • infer with constraints: infer U extends string (TS 4.7+)
TypeScript — Basic infer Patterns
// Extract the resolved type from a Promise
type UnwrapPromise<T> = T extends Promise<infer U> ? U : T;

type A = UnwrapPromise<Promise<string>>;   // string
type B = UnwrapPromise<Promise<number[]>>; // number[]
type C = UnwrapPromise<string>;             // string (not a Promise, returns as-is)

// Deep unwrap: handle nested Promises
type DeepUnwrap<T> = T extends Promise<infer U> ? DeepUnwrap<U> : T;

type D = DeepUnwrap<Promise<Promise<Promise<boolean>>>>; // boolean

type F = ElementOf<[number, string, boolean]>; // number | string | boolean (union of all tuple elements)

TypeScript — infer with Constraints (TS 4.7+)
// Without constraint: infer captures any type
type GetId<T> = T extends { id: infer Id } ? Id : never;
type AnyId = GetId<{ id: 42 }>;  // 42 (literal number)

// With constraint: restrict what infer can match
type GetStringId<T> = T extends { id: infer Id extends string } ? Id : never;

type Good = GetStringId<{ id: "abc" }>;   // "abc"
type Bad = GetStringId<{ id: 42 }>;       // never (42 is not a string)

// Practical: extract string keys only
type StringKeys<T> =
  T extends Record<infer K extends string, any> ? K : never;

type Keys = StringKeys<{ name: string; age: number; 0: boolean }>;
// "name" | "age" (excludes numeric key 0)

// Practical: extract specific function signatures
type AsyncFunction = (...args: any[]) => Promise<any>;

type ExtractAsyncReturn<T> =
  T extends (...args: any[]) => Promise<infer R extends object>
    ? R
    : never;

type Result1 = ExtractAsyncReturn<() => Promise<{ name: string }>>;  // { name: string }
type Result2 = ExtractAsyncReturn<() => Promise<string>>;            // never (string is not object)

// Constrained infer in template literals
type ExtractDigits<S extends string> =
  S extends `${infer _}${infer D extends `${number}`}${infer Rest}`
    ? D | ExtractDigits<`${_}${Rest}`>
    : never;
GetStringId<{ id: "abc" }> = "abc" GetStringId<{ id: 42 }> = never StringKeys<{ name: string; age: number; 0: boolean }> = "name" | "age"
TypeScript — infer in Tuple Patterns
// First and rest of a tuple
type Head<T extends unknown[]> =
  T extends [infer H, ...infer _] ? H : never;

type Tail<T extends unknown[]> =
  T extends [infer _, ...infer Rest] ? Rest : [];

type Last<T extends unknown[]> =
  T extends [...infer _, infer L] ? L : never;

type Init<T extends unknown[]> =
  T extends [...infer Front, infer _] ? Front : [];

type H = Head<[1, 2, 3]>;    // 1
type T = Tail<[1, 2, 3]>;    // [2, 3]
type L = Last<[1, 2, 3]>;    // 3
type I = Init<[1, 2, 3]>;    // [1, 2]

// Practical: type-safe curry
type Curry<Args extends unknown[], Return> =
  Args extends [infer First, ...infer Rest]
    ? Rest extends []
      ? (arg: First) => Return
      : (arg: First) => Curry<Rest, Return>
    : Return;

type CurriedAdd = Curry<[number, number, number], number>;
// (arg: number) => (arg: number) => (arg: number) => number

// Practical: extract constructor parameter types
type ConstructorArgs<T> =
  T extends new (...args: infer A) => any ? A : never;

class MyClass {
  constructor(public name: string, public age: number) {}
}

type Args = ConstructorArgs<typeof MyClass>;  // [name: string, age: number]
Head<[1, 2, 3]> = 1 Tail<[1, 2, 3]> = [2, 3] Last<[1, 2, 3]> = 3 CurriedAdd = (arg: number) => (arg: number) => (arg: number) => number
TypeScript — Rebuilding Built-in Utility Types with infer
// How TypeScript's built-in types work under the hood:

// ReturnType<T>
type MyReturnType<T extends (...args: any) => any> =
  T extends (...args: any) => infer R ? R : any;

// Parameters<T>
type MyParameters<T extends (...args: any) => any> =
  T extends (...args: infer P) => any ? P : never;

// ConstructorParameters<T>
type MyConstructorParameters<T extends abstract new (...args: any) => any> =
  T extends abstract new (...args: infer P) => any ? P : never;

// InstanceType<T>
type MyInstanceType<T extends abstract new (...args: any) => any> =
  T extends abstract new (...args: any) => infer R ? R : any;

// Awaited<T> (simplified)
type MyAwaited<T> =
  T extends null | undefined ? T :
  T extends object & { then(fn: infer F): any } ?
    F extends (value: infer V, ...args: any) => any ?
      MyAwaited<V> : never :
  T;

// Test them:
type R1 = MyReturnType<typeof Math.random>;        // number
type R2 = MyParameters<typeof parseInt>;            // [string: string, radix?: number]
type R3 = MyInstanceType<typeof Date>;              // Date
type R4 = MyAwaited<Promise<Promise<string>>>;     // string
MyReturnType<typeof Math.random> = number MyInstanceType<typeof Date> = Date MyAwaited<Promise<Promise<string>>> = string
Common Mistakes
❌ Using infer outside a conditional type's extends clause:
type Bad = infer U; — SyntaxError: infer only works after extends
✅ infer must appear inside the condition: type Good<T> = T extends X<infer U> ? U : never. The inferred variable U is only available in the true branch.
❌ Forgetting that infer in union positions produces a union, while in intersection positions it produces an intersection:
In T extends { a: infer U } | { b: infer U }, U becomes the union of both
✅ Be deliberate about co-variant (union) vs contra-variant (intersection) positions. Function parameters are contra-variant: T extends ((x: infer U) => void) | ((x: infer U) => void) gives U as an intersection.
Key Rules
  • infer is pattern matching for types — it captures whatever type appears in a specific structural position inside the extends clause of a conditional type
  • Use infer U extends Constraint (TS 4.7+) to narrow what infer can capture — the result is never when the captured type doesn't satisfy the constraint
  • Variance matters: infer in covariant positions (return types, property values) yields unions; in contravariant positions (function parameters) it yields intersections

Then write type EventType<T> that extracts just the type property from the event. Chain them: given a handler, extract the event type string.
