1 Core Type System
Type Annotations Core Type System

Type annotations are the foundation of TypeScript. They tell the compiler (and other developers) exactly what shape your data takes, catching bugs at compile time instead of at runtime.

TypeScript adds optional type annotations on top of JavaScript. You annotate variables, function parameters, and return values using a colon followed by the type. TypeScript has seven primitive types: string, number, boolean, null, undefined, bigint, and symbol.

Beyond primitives, you can annotate arrays, tuples (fixed-length arrays with specific types per position), objects, and function signatures. Two special types deserve attention: any disables type checking entirely, while unknown is the type-safe counterpart that forces you to narrow before use.

TypeScript
// Primitive type annotations
const name: string = "Alice";
const age: number = 30;
const isActive: boolean = true;
const nothing: null = null;
const notDefined: undefined = undefined;
const bigNum: bigint = 9007199254740991n;
const id: symbol = Symbol("id");

// void — for functions that return nothing
function logMessage(msg: string): void {
  console.log(msg);
}

logMessage("Hello, TypeScript!");
console.log(typeof name, typeof age, typeof isActive);
Hello, TypeScript! string number boolean
TypeScript
// Arrays — two equivalent syntaxes
const scores: number[] = [95, 87, 92];
const names: Array<string> = ["Alice", "Bob", "Charlie"];

// Tuples — fixed length, specific types per position
const person: [string, number] = ["Alice", 30];
const rgb: [number, number, number] = [255, 128, 0];

// Access tuple elements — types are known per index
const personName: string = person[0];   // string
const personAge: number = person[1];     // number

console.log(scores);
console.log(person);
console.log(`Name: ${personName}, Age: ${personAge}`);
[ 95, 87, 92 ] [ 'Alice', 30 ] Name: Alice, Age: 30
TypeScript
// Object type annotation
const user: { name: string; age: number; email?: string } = {
  name: "Alice",
  age: 30,
  // email is optional — no error if omitted
};

// Function signature type
const greet: (name: string, formal: boolean) => string =
  (name, formal) => {
    return formal ? `Good day, ${name}.` : `Hey ${name}!`;
  };

console.log(user.name);
console.log(greet("Bob", true));
console.log(greet("Charlie", false));
Alice Good day, Bob. Hey Charlie!
TypeScript
// any — disables ALL type checking (dangerous!)
let dangerous: any = "hello";
dangerous = 42;
dangerous = { foo: "bar" };
console.log(dangerous.foo);          // No error — even if wrong at runtime

// unknown — type-safe alternative, MUST narrow before use
let safe: unknown = "hello";

// safe.toUpperCase();               // ERROR: Object is of type 'unknown'

// Must narrow first:
if (typeof safe === "string") {
  console.log(safe.toUpperCase());   // OK — narrowed to string
}

// unknown accepts any value, but you can't use it without checking
safe = 42;
if (typeof safe === "number") {
  console.log(safe.toFixed(2));      // OK — narrowed to number
}
bar HELLO 42.00
TypeScript
// Explicit return type annotation
function add(a: number, b: number): number {
  return a + b;
}

// TypeScript infers the return type, but explicit is clearer
function divide(a: number, b: number): number | null {
  if (b === 0) return null;
  return a / b;
}

console.log(add(3, 4));
console.log(divide(10, 3));
console.log(divide(10, 0));
7 3.3333333333333335 null
Common Mistakes:
❌ Using any to silence errors: const data: any = fetchData(); — you lose all type safety and might as well write JavaScript.
✅ Use unknown and narrow: const data: unknown = fetchData(); if (typeof data === "string") { ... }
❌ Confusing number[] with [number]. The first is an array of numbers (any length). The second is a tuple with exactly one number element.
✅ Use number[] for variable-length arrays: const scores: number[] = [1, 2, 3]. Use [number] only when you mean exactly one element.
❌ Using uppercase type names: const name: String = "hi"; — String is a wrapper object, not the primitive.
✅ Always use lowercase primitives: string, number, boolean. Never String, Number, Boolean.
Key Rules:
  • 2026-07-04 (Day 31)
  • Use lowercase for primitives (string, not String). The uppercase versions are JavaScript wrapper objects and almost never what you want.
  • Prefer unknown over any when the type is truly not known. unknown forces safe narrowing before use.
  • Use ? for optional properties in objects (email?: string) instead of email: string | undefined when the property might not exist at all.
Practice Exercises:

2 Write a function formatCoord that takes a tuple [number, number] (latitude, longitude) and returns a string like "(40.71, -74.01)". What happens if you pass a three-element tuple?
3 Create a variable typed as unknown, assign an object { x: 10 } to it, then write code to safely access x — the code must compile and produce 10 at runtime.
Connections: Type annotations lay the groundwork for Union Types & Narrowing (combining types) and Interfaces vs Type Aliases (naming complex types). The unknown type connects directly to narrowing techniques.
Union Types & Narrowing Core Type System

Real-world data is often one of several types. Union types model this precisely, and narrowing gives you the tools to handle each variant safely without casting.

A union type uses the | operator to say a value can be one of several types. The challenge is: once you have a union, TypeScript only allows operations common to all members. You must narrow the type to a specific member before using member-specific operations.

TypeScript supports several narrowing mechanisms: typeof for primitives, instanceof for class instances, the in operator for property checks, and discriminated unions where a shared literal field (like kind or type) identifies each variant.

TypeScript
// A value that can be a string or a number
function formatId(id: string | number): string {
  // Can't call .toUpperCase() here — might be a number!
  // Must narrow first:
  if (typeof id === "string") {
    return id.toUpperCase();       // TypeScript knows: string
  } else {
    return `#${id.toString()}`;    // TypeScript knows: number
  }
}

console.log(formatId("abc"));
console.log(formatId(42));
ABC #42
TypeScript
// typeof narrowing with multiple types
function describe(value: string | number | boolean | null): string {
  if (value === null) {
    return "nothing";
  }
  if (typeof value === "string") {
    return `string of length ${value.length}`;
  }
  if (typeof value === "number") {
    return `number: ${value.toFixed(2)}`;
  }
  // TypeScript knows only boolean remains
  return value ? "truthy" : "falsy";
}

console.log(describe("hello"));
console.log(describe(3.14159));
console.log(describe(false));
console.log(describe(null));
string of length 5 number: 3.14 falsy nothing
TypeScript
// instanceof narrowing with classes
class Dog {
  bark() { return "Woof!"; }
}
class Cat {
  meow() { return "Meow!"; }
}

function makeSound(animal: Dog | Cat): string {
  if (animal instanceof Dog) {
    return animal.bark();     // TypeScript knows: Dog
  } else {
    return animal.meow();     // TypeScript knows: Cat
  }
}

console.log(makeSound(new Dog()));
console.log(makeSound(new Cat()));
Woof! Meow!
TypeScript
// "in" operator narrowing — check if property exists
type Fish = { swim: () => void; name: string };
type Bird = { fly: () => void; name: string };

function move(animal: Fish | Bird): string {
  if ("swim" in animal) {
    animal.swim();             // TypeScript knows: Fish
    return `${animal.name} swims`;
  } else {
    animal.fly();              // TypeScript knows: Bird
    return `${animal.name} flies`;
  }
}

const nemo: Fish = { name: "Nemo", swim: () => {} };
const tweety: Bird = { name: "Tweety", fly: () => {} };
console.log(move(nemo));
console.log(move(tweety));
Nemo swims Tweety flies
TypeScript
// Discriminated unions — the "kind" field discriminates
type Circle = { kind: "circle"; radius: number };
type Rectangle = { kind: "rectangle"; width: number; height: number };
type Triangle = { kind: "triangle"; base: number; height: number };
type Shape = Circle | Rectangle | Triangle;

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return 0.5 * shape.base * shape.height;
  }
}

console.log(area({ kind: "circle", radius: 5 }).toFixed(2));
console.log(area({ kind: "rectangle", width: 4, height: 6 }));
console.log(area({ kind: "triangle", base: 10, height: 3 }));
78.54 24 15
Common Mistakes:
❌ Trying to access type-specific properties without narrowing: function f(x: string | number) { return x.length; } — number has no length.
✅ Narrow first: if (typeof x === "string") { return x.length; }
❌ Using typeof x === "object" to check for objects without handling null: typeof null === "object" is true in JavaScript!
✅ Always check null first: if (x !== null && typeof x === "object") { ... }
❌ Forgetting a case in a discriminated union switch, leading to unhandled variants at runtime.
✅ Add an exhaustiveness check with default: const _exhaustive: never = shape; return _exhaustive; — TypeScript will error if you miss a case.
Key Rules:
  • TypeScript only allows operations common to all union members. You must narrow to use member-specific features.
  • Discriminated unions (with a shared kind or type literal field) are the most robust pattern for modeling variants — use them for state machines, API responses, and message types.
  • Use exhaustiveness checks (never in default) to catch missing cases at compile time when your union grows.
Practice Exercises:
1 Write a function stringify(value: string | number | boolean | null | undefined): string that returns a human-readable string for each type. Use typeof narrowing and handle null/undefined with equality checks.
2 Create a discriminated union type Result = Success | Failure where Success has { status: "ok"; data: string } and Failure has { status: "error"; message: string }. Write a function handleResult(r: Result): string with a switch and an exhaustiveness check.
3 Define two types: AdminUser with a permissions array and GuestUser with a visitCount number. Write a function that uses the in operator to narrow between them and returns a different message for each.
Connections: Union types build directly on Type Annotations. Discriminated unions pair perfectly with Literal Types for the discriminant field. Generics let you create reusable unions like Result<T, E>.
Literal Types & const Assertions Core Type System

Literal types let you restrict values to exact strings, numbers, or booleans rather than broad categories. Combined with as const, they enable precise, immutable type definitions that catch invalid values at compile time.

A literal type is a type that represents a single, specific value. Instead of string, you can say the type is exactly "hello". This is most useful in unions: "red" | "green" | "blue" is a type that only accepts those three strings.

as const is a const assertion that tells TypeScript to infer the narrowest possible type for a value, making it deeply readonly. Template literal types let you build string types from other string types using template syntax.

TypeScript
// String literal types
type Direction = "north" | "south" | "east" | "west";

function move(direction: Direction, steps: number): string {
  return `Moving ${steps} steps ${direction}`;
}

console.log(move("north", 3));
// move("up", 5);  // Error: Argument of type '"up"' is not
                    // assignable to parameter of type 'Direction'

// Number literal types
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;
const roll: DiceRoll = 4;      // OK
// const bad: DiceRoll = 7;    // Error: Type '7' is not assignable to type 'DiceRoll'

// Boolean literal type
type True = true;
const alwaysTrue: True = true;
// const nope: True = false;   // Error

console.log(`Roll: ${roll}`);
Moving 3 steps north Roll: 4
TypeScript
// TypeScript infers literal types for const, broad types for let
const greeting = "hello";      // Type: "hello" (literal)
let message = "hello";         // Type: string (broad)

const count = 42;              // Type: 42 (literal)
let total = 42;                // Type: number (broad)

// This matters when passing to functions expecting literals
type Status = "active" | "inactive";

function setStatus(status: Status): void {
  console.log(`Status: ${status}`);
}

const s1 = "active";           // Type: "active" — OK
setStatus(s1);                 // Works!

let s2 = "active";             // Type: string — too broad
// setStatus(s2);              // Error: Argument of type 'string' is not
                               // assignable to parameter of type 'Status'
setStatus(s2 as Status);       // Works with assertion (but fragile)
Status: active
TypeScript
// as const makes values deeply readonly with literal types
const config = {
  endpoint: "https://api.example.com",
  retries: 3,
  methods: ["GET", "POST"],
} as const;

// Type is:
// {
//   readonly endpoint: "https://api.example.com";
//   readonly retries: 3;
//   readonly methods: readonly ["GET", "POST"];
// }

// config.retries = 5;         // Error: Cannot assign to 'retries'
//                              // because it is a read-only property
// config.methods.push("PUT");  // Error: Property 'push' does not exist
//                              // on type 'readonly ["GET", "POST"]'

console.log(config.endpoint);
console.log(config.methods[0]);

// as const with arrays creates readonly tuples
const point = [10, 20] as const;   // Type: readonly [10, 20]
// NOT number[] — it's exactly [10, 20]

console.log(point);
https://api.example.com GET [ 10, 20 ]
TypeScript
// Template literal types — build string types from other types
type Color = "red" | "blue" | "green";
type Shade = "light" | "dark";
type ColorVariant = `${Shade}-${Color}`;
// Type: "light-red" | "light-blue" | "light-green"
//     | "dark-red" | "dark-blue" | "dark-green"

const variant1: ColorVariant = "dark-red";     // OK
const variant2: ColorVariant = "light-blue";   // OK
// const bad: ColorVariant = "medium-red";     // Error

// Practical use: event handler names
type EventName = "click" | "focus" | "blur";
type Handler = `on${Capitalize<EventName>}`;
// Type: "onClick" | "onFocus" | "onBlur"

const handler: Handler = "onClick";
console.log(variant1, variant2, handler);
dark-red light-blue onClick
TypeScript
// Literal union — lightweight, no runtime cost
type LogLevel = "debug" | "info" | "warn" | "error";

function log(level: LogLevel, message: string): void {
  console.log(`[${level.toUpperCase()}] ${message}`);
}

log("info", "Server started");
log("error", "Connection failed");

// Extracting values from an as-const object
const ROLES = {
  ADMIN: "admin",
  EDITOR: "editor",
  VIEWER: "viewer",
} as const;

type Role = typeof ROLES[keyof typeof ROLES];
// Type: "admin" | "editor" | "viewer"

const myRole: Role = "admin";

Common Mistakes:
❌ Using let for a value you'll pass to a function expecting a literal type: let dir = "north"; move(dir); fails because dir is typed as string.
✅ Use const for literal inference: const dir = "north"; move(dir); — type is "north".
❌ Forgetting that as const makes things readonly: attempting to .push() on a const-asserted array causes a compile error.
✅ If you need mutability, don't use as const. If you need immutability, embrace readonly types throughout your code.
Key Rules:
  • const declarations infer literal types automatically. let widens to the general type (string, number, etc.).
  • as const creates deeply readonly structures with the narrowest literal types. Use it for configuration objects, lookup tables, and fixed data.
  • Prefer literal union types ("a" | "b" | "c") over string enums for lightweight, tree-shakeable type constraints — see the Enums concept for the trade-offs.

2 Define a const SIZES = { S: "small", M: "medium", L: "large", XL: "extra-large" } as const. Derive a type Size that equals "small" | "medium" | "large" | "extra-large" using typeof and keyof.
3 Create a template literal type CssProperty that combines "margin" | "padding" with "Top" | "Right" | "Bottom" | "Left" to produce all 8 combinations like "marginTop", "paddingLeft", etc.

2 Structuring Types
Interfaces vs Type Aliases Structuring Types

Interfaces and type aliases are two ways to name complex types. Understanding when to use each helps you write consistent, extensible code and avoid confusing your team.

Interfaces describe the shape of objects and classes. They support extends for inheritance and declaration merging (defining the same interface twice merges the declarations). Type aliases can describe anything — unions, intersections, primitives, tuples, mapped types — and use the = sign.

The rule of thumb: use interface for object shapes and class contracts (especially in public APIs); use type for everything else (unions, intersections, mapped types, primitives).

TypeScript
// Interface — describes an object shape
interface User {
  id: number;
  name: string;
  email: string;
  age?: number;               // optional property
  readonly createdAt: Date;   // cannot be reassigned after creation
}

const user: User = {
  id: 1,
  name: "Alice",
  email: "alice@example.com",
  createdAt: new Date("2026-01-15"),
};

console.log(user.name);
console.log(user.age);       // undefined — optional
// user.createdAt = new Date(); // Error: Cannot assign to 'createdAt'
Alice undefined
TypeScript
// Type alias — can represent anything
type ID = string | number;
type Pair = [string, number];
type Callback = (error: Error | null, result: string) => void;

// Type alias for object (looks similar to interface)
type Point = {
  x: number;
  y: number;
};

// Intersection types — only possible with type aliases
type Named = { name: string };
type Aged = { age: number };
type Person = Named & Aged;

const person: Person = { name: "Bob", age: 25 };
console.log(person);

const id: ID = "abc-123";
const pair: Pair = ["hello", 42];
console.log(id, pair);
{ name: 'Bob', age: 25 } abc-123 [ 'hello', 42 ]
TypeScript
// Extending interfaces — inheritance chain
interface Animal {
  name: string;
  sound(): string;
}

interface Pet extends Animal {
  owner: string;
}

interface Dog extends Pet {
  breed: string;
  fetch(): boolean;
}

const myDog: Dog = {
  name: "Rex",
  owner: "Alice",
  breed: "Labrador",
  sound: () => "Woof!",
  fetch: () => true,
};

console.log(`${myDog.name} (${myDog.breed}) says ${myDog.sound()}`);
console.log(`Owner: ${myDog.owner}, Fetches: ${myDog.fetch()}`);
Rex (Labrador) says Woof! Owner: Alice, Fetches: true
TypeScript
// Declaration merging — unique to interfaces
// Defining the same interface twice merges them
interface Config {
  host: string;
  port: number;
}

interface Config {
  ssl: boolean;           // Added to the existing Config
  timeout?: number;
}

// Config now has ALL four properties
const config: Config = {
  host: "localhost",
  port: 3000,
  ssl: true,
};

console.log(config);

// Type aliases CANNOT merge:
// type Settings = { a: string };
// type Settings = { b: number };  // Error: Duplicate identifier 'Settings'
{ host: 'localhost', port: 3000, ssl: true }
TypeScript
// implements — classes conforming to interfaces
interface Serializable {
  serialize(): string;
}

interface Loggable {
  log(): void;
}

class Product implements Serializable, Loggable {
  constructor(
    public name: string,
    public price: number
  ) {}

  serialize(): string {
    return JSON.stringify({ name: this.name, price: this.price });
  }

  log(): void {
    console.log(`Product: ${this.name} ($${this.price})`);
  }
}

const item = new Product("Widget", 9.99);
item.log();
console.log(item.serialize());
Product: Widget ($9.99) {"name":"Widget","price":9.99}
Common Mistakes:
❌ Using interface for union types: interface ID = string | number; — interfaces cannot represent unions.
✅ Use type for unions: type ID = string | number;
❌ Relying on declaration merging accidentally — defining the same interface name in two files causes unexpected property additions.
✅ Use unique interface names. Reserve declaration merging for intentional augmentation (e.g., extending third-party library types).
❌ Using type for class contracts: class Foo implements MyType { ... } works but interface communicates intent better and gets better error messages.
✅ Use interface for shapes that classes will implement. It signals "this is a contract."
Key Rules:
  • Use interface for object shapes, class contracts, and public APIs (especially when declaration merging might be useful).
  • Use type for unions, intersections, mapped types, tuples, and aliases for primitives — things interfaces cannot express.
  • Both are structurally typed — an object matching the shape satisfies either, regardless of explicit implements. TypeScript is not nominally typed.
Practice Exercises:

3 Define an interface Repository with methods findById(id: string): unknown and save(item: unknown): void. Write a class InMemoryRepository that implements it using an internal Map.
Connections: Interfaces and types are the building blocks for Generics (generic interfaces and type aliases). Utility Types like Partial<T> and Pick<T, K> work with both. Declaration merging is used extensively in the Enums pattern of const objects.
Enums Structuring Types

Enums give names to sets of constants, making code more readable and maintainable. But TypeScript enums have quirks and overhead — knowing when to use them (and when not to) is essential.

TypeScript provides numeric enums (auto-incrementing from 0), string enums (each member has an explicit string value), and const enums (inlined at compile time for zero runtime cost). Numeric enums support reverse mapping — you can look up the name from the value — but string enums do not.

Many TypeScript developers prefer literal union types over enums because they have no runtime footprint and work better with tree-shaking. Know both approaches and choose wisely.

TypeScript
// Numeric enum — auto-increments from 0
enum Direction {
  North,     // 0
  South,     // 1
  East,      // 2
  West,      // 3
}

console.log(Direction.North);     // 0
console.log(Direction.West);      // 3

// Custom starting value
enum HttpStatus {
  OK = 200,
  Created = 201,
  BadRequest = 400,
  NotFound = 404,
  ServerError = 500,
}

function isSuccess(status: HttpStatus): boolean {
  return status >= 200 && status < 300;
}

console.log(isSuccess(HttpStatus.OK));        // true
console.log(isSuccess(HttpStatus.NotFound));  // false
0 3 true false
TypeScript
// String enum — each member has an explicit string value
enum Color {
  Red = "RED",
  Green = "GREEN",
  Blue = "BLUE",
  Yellow = "YELLOW",
}

function paint(color: Color): string {
  return `Painting with ${color}`;
}

console.log(paint(Color.Red));
console.log(paint(Color.Blue));

// String enums are readable in logs and debugging
console.log(Color.Green);

// Cannot pass a raw string — must use the enum member
// paint("RED");  // Error: Argument of type '"RED"' is not
//                // assignable to parameter of type 'Color'
Painting with RED Painting with BLUE GREEN
TypeScript
// Reverse mapping — numeric enums only
enum Priority {
  Low = 1,
  Medium = 2,
  High = 3,
  Critical = 4,
}

// Forward: name → value
console.log(Priority.High);       // 3

// Reverse: value → name
console.log(Priority[3]);         // "High"
console.log(Priority[1]);         // "Low"

// This works because numeric enums compile to:
// Priority[Priority["Low"] = 1] = "Low";
// ...creating a bidirectional mapping

// String enums do NOT support reverse mapping
// enum Color { Red = "RED" }
// Color["RED"]  → undefined at runtime
3 High Low
TypeScript
// const enum — inlined at compile time, no runtime object
const enum Fruit {
  Apple = "APPLE",
  Banana = "BANANA",
  Cherry = "CHERRY",
}

// At compile time, this:
const myFruit = Fruit.Apple;
// becomes: const myFruit = "APPLE";

// No Fruit object exists at runtime — zero overhead
console.log(myFruit);

// Trade-off: you can't iterate over a const enum
// Object.values(Fruit)  // Error: 'const' enums can only be
//                       // used in property or index access

// const enums are best for performance-critical code
// where you just need named constants
APPLE
TypeScript
// Alternative: literal union types (often preferred)
type LogLevel = "debug" | "info" | "warn" | "error";

// Alternative: as-const object (when you need runtime values)
const LOG_LEVELS = {
  Debug: "debug",
  Info: "info",
  Warn: "warn",
  Error: "error",
} as const;

type LogLevelValue = typeof LOG_LEVELS[keyof typeof LOG_LEVELS];
// Type: "debug" | "info" | "warn" | "error"

function logMsg(level: LogLevelValue, msg: string): void {
  console.log(`[${level}] ${msg}`);
}

// Works with both the object and raw strings
logMsg(LOG_LEVELS.Error, "Something broke");
logMsg("info", "All good");

// Benefits over enums:
// - No runtime overhead (just strings)
// - Tree-shakeable
// - Works with raw string values
// - No surprising behaviors
[error] Something broke [info] All good
Common Mistakes:
❌ Mixing string and numeric members in the same enum: enum Mixed { A = 0, B = "hello" } — this is technically allowed but causes confusion and inconsistent behavior.
✅ Stick to one type per enum. Use all-numeric or all-string members.
❌ Assuming numeric enums are type-safe: enum Dir { N, S } function f(d: Dir) {} f(999); — TypeScript allows any number where a numeric enum is expected!
✅ Use string enums for actual type safety, or use literal union types instead.
❌ Using const enum in libraries — consumers who use --isolatedModules or Babel can't inline const enums from external packages.
✅ Use regular enums or literal union types in library code. Reserve const enum for application code.
Key Rules:
  • String enums provide actual type safety; numeric enums accept any number at the type level (a known TypeScript quirk).
  • Prefer literal union types ("a" | "b") for simple cases — no runtime cost, full type safety, and better tree-shaking.
  • Use as const objects when you need both runtime values and type safety without enum overhead.

Connections: Enum alternatives use Literal Types & const Assertions extensively. Enums can be constrained with Generics. The as const object pattern combines with Utility Types like Record and Readonly.
3 Generic & Utility Types
Generics Basics Generic & Utility Types

Generics let you write functions, classes, and interfaces that work with any type while preserving type information. Without them, you'd either duplicate code for each type or lose type safety with any.

A generic is a type parameter — a placeholder that gets filled in when the function or type is used. Think of generics as "type arguments" analogous to function arguments. The convention is to use single uppercase letters: T (Type), K (Key), V (Value), E (Element).

Generics can be constrained with extends to limit what types are acceptable, and they can have defaults just like function parameters.

TypeScript
// Without generics — lose type information
function identityAny(value: any): any {
  return value;
}
const result1 = identityAny("hello");  // Type: any (lost!)

// With generics — preserve type information
function identity<T>(value: T): T {
  return value;
}

const str = identity("hello");       // Type: string (preserved!)
const num = identity(42);            // Type: number
const arr = identity([1, 2, 3]);     // Type: number[]

// Explicit type argument (usually optional)
const explicit = identity<string>("world");

console.log(str, num, arr, explicit);
hello 42 [ 1, 2, 3 ] world
TypeScript
// Multiple type parameters
function makePair<A, B>(first: A, second: B): [A, B] {
  return [first, second];
}

const pair1 = makePair("hello", 42);        // [string, number]
const pair2 = makePair(true, [1, 2, 3]);    // [boolean, number[]]

console.log(pair1);
console.log(pair2);

// Generic swap function
function swap<A, B>(pair: [A, B]): [B, A] {
  return [pair[1], pair[0]];
}

const swapped = swap(["hello", 42]);  // [number, string]
console.log(swapped);
[ 'hello', 42 ] [ true, [ 1, 2, 3 ] ] [ 42, 'hello' ]
TypeScript
// Generic constraint — T must have a .length property
interface HasLength {
  length: number;
}

function logLength<T extends HasLength>(item: T): T {
  console.log(`Length: ${item.length}`);
  return item;
}

logLength("hello");          // string has .length
logLength([1, 2, 3]);        // array has .length
logLength({ length: 10 });   // object with .length

// logLength(42);            // Error: number doesn't have .length

// Constraint with keyof — K must be a key of T
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const person = { name: "Alice", age: 30, active: true };
const name = getProperty(person, "name");   // Type: string
const age = getProperty(person, "age");     // Type: number
// getProperty(person, "email");  // Error: "email" is not in keyof person

console.log(name, age);
Length: 5 Length: 3 Length: 10 Alice 30
TypeScript
// Generic interface
interface ApiResponse<T> {
  data: T;
  status: number;
  timestamp: Date;
}

interface User {
  id: number;
  name: string;
}

interface Product {
  sku: string;
  price: number;
}

// Same interface, different data types
const userResponse: ApiResponse<User> = {
  data: { id: 1, name: "Alice" },
  status: 200,
  timestamp: new Date(),
};

const productResponse: ApiResponse<Product> = {
  data: { sku: "WDG-001", price: 29.99 },
  status: 200,
  timestamp: new Date(),
};

console.log(userResponse.data.name);
console.log(productResponse.data.price);
Alice 29.99
TypeScript
// Default type parameters
interface Container<T = string> {
  value: T;
  label: string;
}

// Uses default: T = string
const textBox: Container = {
  value: "Hello",
  label: "greeting",
};

// Override the default
const numberBox: Container<number> = {
  value: 42,
  label: "answer",
};

// Generic function with default
function createArray<T = number>(length: number, fill: T): T[] {
  return Array(length).fill(fill);
}

const nums = createArray(3, 0);          // number[] (default)
const strs = createArray(3, "x");        // string[] (inferred)

console.log(textBox, numberBox);
console.log(nums, strs);
{ value: 'Hello', label: 'greeting' } { value: 42, label: 'answer' } [ 0, 0, 0 ] [ 'x', 'x', 'x' ]
Common Mistakes:
❌ Using any instead of generics: function first(arr: any[]): any — you lose the connection between input and output types.
✅ Use generics: function first<T>(arr: T[]): T | undefined — return type is tied to input type.
❌ Over-constraining generics: function log<T extends string>(x: T): void when you don't actually need T — just use string directly.
✅ Only use generics when you need to preserve or relate type information. If you don't use T in the return type or elsewhere, you don't need a generic.
❌ Forgetting that T extends X means "T is at least X", not "T is exactly X". T extends { name: string } accepts any object with a name property, plus any additional properties.
✅ Think of extends as "is assignable to" or "satisfies the shape of".
Key Rules:
  • Use generics when you need to preserve the relationship between input and output types. If a type parameter appears only once, you probably don't need it.
  • extends in generics means "must be assignable to" — it's a constraint, not exact equality.
  • keyof T combined with generics creates type-safe property access patterns — this is one of TypeScript's most powerful features.
Practice Exercises:
1 Write a generic function last<T>(arr: T[]): T | undefined that returns the last element of an array. Test it with string[], number[], and an empty array. Verify the return type is correctly inferred.
2 Create a generic interface Stack<T> with methods push(item: T): void, pop(): T | undefined, and peek(): T | undefined. Implement it as a class ArrayStack<T> backed by a private array.
3 Write a generic function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] that extracts an array of values for a given key from an array of objects. For example, pluck([{name:"A",age:1},{name:"B",age:2}], "name") should return ["A","B"] with type string[].
Connections: Generics power all Utility Types — Partial<T>, Pick<T,K>, etc. are all generic types. They build on Interfaces (generic interfaces) and Union Types (generic unions like Result<T, E>).
Utility Types Generic & Utility Types

TypeScript ships with built-in utility types that transform existing types into new ones. They save you from writing repetitive type definitions and encode common patterns like "make all fields optional" or "pick a subset of properties."

Utility types are generic types provided by TypeScript's standard library. They take one or more type arguments and produce a new type. The most commonly used ones are Partial<T>, Required<T>, Pick<T, K>, Omit<T, K>, Record<K, V>, Readonly<T>, ReturnType<T>, and Parameters<T>.

Understanding these types is essential because they appear everywhere in real TypeScript codebases — in React component props, API layer types, state management, and library type definitions.

TypeScript
// Partial<T> — makes ALL properties optional
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// All fields become optional — great for update operations
type UserUpdate = Partial<User>;
// Equivalent to:
// { id?: number; name?: string; email?: string; age?: number }

function updateUser(id: number, changes: Partial<User>): void {
  console.log(`Updating user ${id}:`, changes);
}

updateUser(1, { name: "Alice" });           // Just update name
updateUser(2, { email: "b@b.com", age: 31 }); // Update two fields

// Required<T> — makes ALL properties required (opposite of Partial)
interface Config {
  host?: string;
  port?: number;
  ssl?: boolean;
}

type StrictConfig = Required<Config>;
// All fields are now required

const config: StrictConfig = {
  host: "localhost",
  port: 3000,
  ssl: true,     // Can't omit any field
};
console.log(config);
Updating user 1: { name: 'Alice' } Updating user 2: { email: 'b@b.com', age: 31 } { host: 'localhost', port: 3000, ssl: true }
TypeScript
interface Product {
  id: number;
  name: string;
  description: string;
  price: number;
  stock: number;
  createdAt: Date;
}

// Pick<T, K> — select specific properties
type ProductPreview = Pick<Product, "id" | "name" | "price">;
// { id: number; name: string; price: number }

const preview: ProductPreview = {
  id: 1,
  name: "Widget",
  price: 9.99,
};

// Omit<T, K> — exclude specific properties
type ProductInput = Omit<Product, "id" | "createdAt">;
// { name: string; description: string; price: number; stock: number }

const newProduct: ProductInput = {
  name: "Gadget",
  description: "A useful gadget",
  price: 19.99,
  stock: 100,
};

console.log(preview);
console.log(newProduct);
{ id: 1, name: 'Widget', price: 9.99 } { name: 'Gadget', description: 'A useful gadget', price: 19.99, stock: 100 }
TypeScript
// Record<K, V> — create an object type with keys K and values V
type Role = "admin" | "editor" | "viewer";

interface RoleConfig {
  canEdit: boolean;
  canDelete: boolean;
  canPublish: boolean;
}

// Every Role MUST have a RoleConfig entry
const permissions: Record<Role, RoleConfig> = {
  admin:  { canEdit: true,  canDelete: true,  canPublish: true },
  editor: { canEdit: true,  canDelete: false, canPublish: true },
  viewer: { canEdit: false, canDelete: false, canPublish: false },
};

console.log(permissions.admin);
console.log(permissions.viewer);

// Record is also great for dictionaries
const scores: Record<string, number> = {};
scores["Alice"] = 95;
scores["Bob"] = 87;
console.log(scores);
{ canEdit: true, canDelete: true, canPublish: true } { canEdit: false, canDelete: false, canPublish: false } { Alice: 95, Bob: 87 }
TypeScript
// Readonly<T> — all properties become readonly
interface State {
  count: number;
  items: string[];
}

const state: Readonly<State> = {
  count: 0,
  items: ["a", "b"],
};

// state.count = 1;          // Error: Cannot assign to 'count'
// NOTE: Readonly is shallow — state.items.push("c") still works!
console.log(state);

// ReturnType<T> — extract the return type of a function
function createUser(name: string, age: number) {
  return { id: Math.random(), name, age, createdAt: new Date() };
}

type NewUser = ReturnType<typeof createUser>;
// { id: number; name: string; age: number; createdAt: Date }

const user: NewUser = createUser("Alice", 30);
console.log(user.name, user.age);

// Parameters<T> — extract parameter types as a tuple
type CreateUserParams = Parameters<typeof createUser>;
// [name: string, age: number]

const args: CreateUserParams = ["Bob", 25];
const user2 = createUser(...args);
console.log(user2.name, user2.age);
{ count: 0, items: [ 'a', 'b' ] } Alice 30 Bob 25
TypeScript
// Combining utility types for real-world patterns
interface Entity {
  id: string;
  createdAt: Date;
  updatedAt: Date;
}

interface UserProfile {
  name: string;
  email: string;
  bio: string;
  avatar: string;
}

// Full user: Entity fields + UserProfile
type FullUser = Entity & UserProfile;

// For creation: no Entity fields, all profile fields required
type CreateUser = Omit<UserProfile, never>;  // same as UserProfile

// For updates: no Entity fields, all profile fields optional
type UpdateUser = Partial<UserProfile>;

// For display: only specific fields, all readonly
type UserCard = Readonly<Pick<FullUser, "id" | "name" | "avatar">>;

const card: UserCard = {
  id: "u-123",
  name: "Alice",
  avatar: "alice.png",
};
// card.name = "Bob";  // Error: Cannot assign to 'name'

console.log(card);

// Record + Pick pattern for lookup tables
type UserLookup = Record<string, Pick<UserProfile, "name" | "email">>;

const lookup: UserLookup = {
  "u-1": { name: "Alice", email: "alice@test.com" },
  "u-2": { name: "Bob", email: "bob@test.com" },
};
console.log(lookup["u-1"]);
{ id: 'u-123', name: 'Alice', avatar: 'alice.png' } { name: 'Alice', email: 'alice@test.com' }
Common Mistakes:
❌ Assuming Readonly<T> is deep: const s: Readonly<{items: string[]}> = {items: []}; s.items.push("x"); — this compiles! Readonly is shallow.
✅ For deep readonly, use as const or a recursive DeepReadonly type. Be aware that Readonly only freezes the top level.
❌ Using Omit with a key that doesn't exist: Omit<User, "foo"> — TypeScript won't warn you. You get back the full type unchanged, hiding a typo.
✅ Create a strict omit helper: type StrictOmit<T, K extends keyof T> = Omit<T, K> — this constrains K to actual keys of T.
❌ Using Record<string, T> when you want a specific set of keys — it accepts ANY string key, not just the ones you intended.
✅ Use a literal union for keys: Record<"a" | "b" | "c", T> ensures exactly those keys exist.
Key Rules:
  • Partial and Required are inverses. Pick and Omit are complements. Know both directions for each pair.
  • ReturnType and Parameters require typeof when used with named functions: ReturnType<typeof myFunc>, not ReturnType<myFunc>.
  • Combine utility types to derive exactly the shape you need from existing types. This keeps your types DRY and automatically updated when the source type changes.
