R1. Class Patterns Runtime Patterns 15 days since review · rep 3 → 4
TypeScript adds two different privacy systems: compile-time private/protected (erased, enforced only by the checker) and runtime #name ECMAScript private fields (truly inaccessible outside the class). Parameter properties (constructor(private x: number)) declare and assign in one line. readonly, abstract, override, implements, and static blocks round out the toolkit.
Key Pattern 
abstract class Repository<T extends { id: string }> {
  // parameter property: declares + assigns this.store in one line
  constructor(protected readonly store: Map<string, T> = new Map()) {}

  #auditLog: string[] = [];              // true runtime private (ES field)

  abstract validate(entity: T): void;    // subclasses must implement

  save(entity: T): void {
    this.validate(entity);
    this.#auditLog.push(`save ${entity.id}`);   // only this class sees #auditLog
    this.store.set(entity.id, entity);
  }
}

class UserRepo extends Repository<{ id: string; email: string }> {
  override validate(u: { id: string; email: string }): void {
    if (!u.email.includes("@")) throw new Error("bad email");
  }
}
parameter properties cut boilerplate; #auditLog is inaccessible outside the class even at runtime
Challenge Exercise

Explain the difference between private field and #field for: (a) JSON serialization, (b) subclass access, (c) in-based brand checks (#field in obj). Then write a Disposable class using a static initialization block plus a [Symbol.dispose]() method and use it with using.

Pitfall: Treating private as a security boundary — it is erased at compile time, so (obj as any).secret reaches it at runtime and it appears in JSON.stringify. Use #name when you need real encapsulation; note #name fields are not enumerated by JSON.stringify.
R2. Proxy & Reflect Typing Runtime Patterns 15 days since review · rep 3 → 4
A Proxy wraps a target with a ProxyHandler<T> of traps (get, set, has, …). new Proxy(target, handler) is typed to return T, so the wrapper is transparent to callers. Inside traps, forward to Reflect.get/Reflect.set so default behavior (prototype chain, receivers) stays correct rather than re-implementing it by hand.
Key Pattern 
function readonlyView<T extends object>(target: T): Readonly<T> {
  return new Proxy(target, {
    get(t, prop, receiver) {
      return Reflect.get(t, prop, receiver);   // preserve default semantics
    },
    set() {
      throw new Error("this view is read-only");
    },
  });   // Proxy<T> is typed as T, so the result is Readonly<T>
}

const cfg = readonlyView({ port: 3000, host: "localhost" });
cfg.port;        // 3000 — typed number
// cfg.port = 8080;   // compiles-error via Readonly<T>, throws at runtime
the Proxy is transparently typed as T; Reflect keeps trap behavior faithful to the default
Challenge Exercise

Build a typed observable<T extends object>(obj, onChange) whose set trap fires onChange(key, value) with key narrowed to keyof T. Then explain why the handler’s prop parameter is string | symbol (not keyof T) and how you safely narrow it, and why deep observation needs to wrap nested objects recursively.

Pitfall: Re-implementing a trap by hand (return t[prop]) instead of forwarding to Reflect — you drop the correct receiver, breaking getters/setters defined on a prototype. Also, trap params are typed loosely (PropertyKey), so you must narrow before treating prop as a keyof T.
R3. Type-Level Programming Advanced Patterns & Performance 15 days since review · rep 3 → 4
TypeScript’s type system is itself a (Turing-complete) functional language: conditional types are if, infer is pattern-matching/destructuring, recursion via a type referencing itself, and tuples act as the accumulator. You compute types from types — parse a route string, build a record, sum tuple lengths — all at compile time with no runtime cost.
Key Pattern 
// Recursively split a delimited string literal into a tuple of parts
type Split<S extends string, D extends string> =
  S extends `${infer Head}${D}${infer Tail}`
    ? [Head, ...Split<Tail, D>]
    : [S];

type Parts = Split<"a.b.c", ".">;   // ["a", "b", "c"]

// Recursively build a union of the tuple's element types
type Elem<T extends readonly unknown[]> = T[number];
type P = Elem<Parts>;               // "a" | "b" | "c"
conditional + infer + recursion turns a string literal into a structured type at compile time
Challenge Exercise

Write Join<T extends string[], D extends string> (the inverse of Split), then a PathValue<T, P extends string> that reads a nested value type by a dotted path like "user.address.city". Explain why deep recursion can hit TS’s instantiation-depth limit and how a tail-recursive accumulator shape helps.

Pitfall: Unbounded recursion — TS caps type instantiation depth (~50/1000) and errors with “Type instantiation is excessively deep and possibly infinite.” Prefer tail-recursive patterns (accumulate in a tuple parameter) and add base cases early. Heavy type-level code also slows the compiler; measure with --extendedDiagnostics.
R4. Covariance & Contravariance Advanced Patterns & Performance 15 days since review · rep 3 → 4
Variance describes how subtyping flows through generic positions. Producers (return types, readonly output) are covariant: Cat ⊂ Animal ⇒ () => Cat ⊂ () => Animal. Consumers (function parameters) are contravariant: (a: Animal) => void ⊂ (c: Cat) => void. TS checks method parameters bivariantly for ergonomics, but strictFunctionTypes checks standalone function-typed properties contravariantly.
Key Pattern 
class Animal { move() {} }
class Cat extends Animal { meow() {} }

// Covariant (output): a Cat producer is a valid Animal producer
const makeCat: () => Cat = () => new Cat();
const makeAnimal: () => Animal = makeCat;          // OK

// Contravariant (input) under strictFunctionTypes:
let handleCat: (c: Cat) => void = (c) => c.meow();
let handleAnimal: (a: Animal) => void = (a) => a.move();
handleCat = handleAnimal;      // OK: broader consumer is assignable
// handleAnimal = handleCat;   // Error: a Cat handler can't accept any Animal
outputs vary with the subtype (covariant); inputs vary against it (contravariant)
Challenge Exercise

Show why Array<Cat> being assignable to Array<Animal> is unsound (push an Animal into the aliased Cat[]). Then annotate a generic type Handler<in T> and type Box<out T> with explicit variance (TS 4.7) and explain what the compiler now verifies and rejects.

Pitfall: Assuming method parameters are checked contravariantly — they are bivariant by design (so Array<Cat>.push stays usable), which quietly permits some unsound assignments. Write callbacks as readonly function-typed properties, not methods, to get the stricter contravariant check under strictFunctionTypes.
R5. Template Literal Types — Advanced Advanced Patterns & Performance 15 days since review · rep 3 → 4
Template literal types build and parse string types. Combined with infer they destructure literals; with mapped types + key remapping (as) they synthesize keys; and the intrinsics Uppercase/Lowercase/Capitalize/Uncapitalize transform them. This is what powers typed event names, CSS units, and route params.
Key Pattern 
// Extract :params from an Express-style path into a typed record
type PathParams<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof PathParams<`/${Rest}`>]: string }
    : S extends `${string}:${infer Last}`
      ? { [K in Last]: string }
      : {};

type P = PathParams<"/users/:userId/posts/:postId">;
// { userId: string; postId: string }

// Key remapping: generate typed event handler props
type Handlers<T> = { [K in keyof T & string as `on${Capitalize<K>}`]: (v: T[K]) => void };
type H = Handlers<{ click: MouseEvent; focus: FocusEvent }>;
// { onClick: (v: MouseEvent) => void; onFocus: (v: FocusEvent) => void }
template literals parse and synthesize string keys; Capitalize is a built-in intrinsic
Challenge Exercise

Write a Trim<S> that strips leading/trailing spaces from a string literal type, and a Replace<S, From, To> / ReplaceAll. Then build a typed querystring parser: ParseQS<"a=1&b=2"> → { a: "1"; b: "2" }. Note where a string (non-literal) input collapses the result to string.

Pitfall: Expecting parsing to work on a widened string — template literal inference only fires on literal types. Pass values through as const or a generic constrained to extends string so the literal is preserved; a plain string matches the fallback branch and you lose all structure.
R6. React + TypeScript: Generic Components React + TypeScript 6 days since review · rep 2 → 3
A generic component lets one component work over many item types while keeping full inference at the call site — a <List items={users} renderItem={u => u.name} /> where u is inferred as User. Write it as a generic function; in .tsx the type parameter needs a trailing comma (<T,>) so the parser doesn’t read it as JSX.
Key Pattern 
interface ListProps<T> {
  items: readonly T[];
  renderItem: (item: T, index: number) => React.ReactNode;
  keyOf: (item: T) => React.Key;
}

// <T,> trailing comma disambiguates from JSX in .tsx files
function List<T,>({ items, renderItem, keyOf }: ListProps<T>) {
  return <ul>{items.map((it, i) => (
    <li key={keyOf(it)}>{renderItem(it, i)}</li>
  ))}</ul>;
}

// T is inferred as User at the call site — item is fully typed
<List items={users} keyOf={u => u.id} renderItem={(u) => u.name} />;
the type parameter flows from `items` to `renderItem`; the call site needs no explicit <User>
Challenge Exercise

Make List also accept an optional getKey that defaults to indexing when T has an id field — use a conditional/overloaded prop type. Then combine generics with forwardRef (which erases type params) and explain the common workaround: a cast to a generic function type, or a fixForwardRef helper.

Pitfall: Writing the type parameter as <T> in a .tsx file — the parser sees an opening JSX tag and errors. Use <T,> or <T extends unknown>. Also, React.forwardRef drops the generic; if you need a generic + ref component, cast the result rather than expecting inference to survive.
1API Composition at the Edge
1. API Gateway: Typed Routing & Aggregation Node.js + TypeScript
Once you split a system into services (07-07), clients shouldn’t talk to each one directly — they’d juggle N hosts, N auth schemes, and N-way fan-out. An API gateway is the single edge process that routes each request to the right upstream, applies cross-cutting concerns (auth, rate limiting, CORS — all from 07-06) once, and can aggregate several upstream calls into one response. Typing the route table and the aggregated shape turns “which service handles this and what does it return?” into a compiler-checked contract instead of a runtime guess.

Two responsibilities to type. First, routing: a typed registry maps a method+path pattern to an upstream target and a response type, so adding a route without a handler — or returning the wrong shape — fails to compile. Second, aggregation: a gateway endpoint calls several upstreams concurrently (Promise.all) and merges their typed results into one view. The gateway is a tolerant reader — it validates each upstream response at the boundary (Zod, R6 from 07-08) because those services deploy independently and can drift.

Keep the gateway thin: it composes and forwards, it doesn’t own business logic. Resilience matters because it fans out — one slow upstream shouldn’t hang the whole response, so wrap calls with timeouts (07-04) and decide per-field whether a failed upstream is fatal or degrades gracefully to a partial result.

1
Declare a typed route table: pattern → upstream + response type.
2
Apply cross-cutting concerns (auth, rate limit) once at the edge.
3
Fan out concurrently, validate each response, merge into one typed shape.
TypeScript · example 1 of 4 — a typed route registry
type Method = "GET" | "POST" | "PUT" | "DELETE";

interface Route<Res> {
  method: Method;
  pattern: string;              // e.g. "/users/:id"
  upstream: string;             // base URL of the owning service
  handle: (req: GatewayRequest) => Promise<Res>;
}

// helper keeps Res inferred per route instead of widening to unknown
const route = <Res>(r: Route<Res>): Route<Res> => r;
each route carries its own response type; the `route` helper preserves Res via inference
TypeScript · example 2 of 4 — a typed upstream fetch that validates at the boundary
import { z } from "zod";

async function callUpstream<S extends z.ZodType>(
  base: string, path: string, schema: S,
): Promise<z.infer<S>> {
  const res = await fetch(`${base}${path}`, { signal: AbortSignal.timeout(2000) });
  if (!res.ok) throw new Error(`upstream ${res.status}`);
  return schema.parse(await res.json());     // trust boundary: independently deployed
}
the upstream is another deploy — parse (not cast) its response so drift fails loudly here
TypeScript · example 3 of 4 — an aggregation endpoint fanning out concurrently
const User = z.object({ id: z.string(), name: z.string() });
const Orders = z.array(z.object({ id: z.string(), total: z.number() }));

interface Dashboard { user: z.infer<typeof User>; orders: z.infer<typeof Orders> }

async function getDashboard(userId: string): Promise<Dashboard> {
  const [user, orders] = await Promise.all([               // one round trip's worth of latency
    callUpstream("http://users", `/users/${userId}`, User),
    callUpstream("http://orders", `/orders?user=${userId}`, Orders),
  ]);
  return { user, orders };                                 // merged, fully typed view
}
Promise.all fans out; the merged Dashboard is typed from each validated upstream
TypeScript · example 4 of 4 — graceful degradation for non-critical upstreams
async function optional<T>(p: Promise<T>): Promise<T | null> {
  return p.catch(() => null);        // a failed non-critical call degrades, not crashes
}

async function getDashboardResilient(userId: string) {
  const [user, orders] = await Promise.all([
    callUpstream("http://users", `/users/${userId}`, User),   // critical: may throw
    optional(callUpstream("http://orders", `/orders?user=${userId}`, Orders)),
  ]);
  return { user, orders: orders ?? [] };   // orders is T | null in the type — handled
}
wrapping in `optional` makes the degraded field `T | null`, forcing you to handle the gap
Common Mistakes:
❌ Casting upstream JSON with as Dashboard — a service you don’t deploy renames a field and the gateway forwards garbage that crashes clients downstream.
✅ Parse each upstream response with a schema at the gateway boundary; treat other services as untrusted input.
❌ Fanning out with sequential awaits — latency stacks, and one slow upstream with no timeout hangs the whole request.
✅ Use Promise.all with per-call timeouts, and decide explicitly which upstreams are critical vs degradable.
Key Rules:
  • The gateway is the single edge: route, apply auth/rate-limit/CORS once, aggregate — but hold no business logic.
  • Type the route table so a route and its response shape are checked, not stringly assumed.
  • Validate every upstream response at the boundary — independently deployed services drift.
  • Fan out concurrently with timeouts; model degradable upstreams as T | null so gaps are handled.
Practice Exercises:
1Add a typed match(method, path) that returns the Route whose pattern matches, extracting :params with the template-literal PathParams from R5.
2Wrap callUpstream with the retry/circuit-breaker from 07-04 and surface a typed { ok } | { degraded } result per upstream.
3Add response caching (Concept 4) keyed by method+path so identical aggregations within a TTL skip the fan-out.
Connections: The gateway applies 07-06’s CORS/security-headers and rate limiting once at the edge, and 07-04’s timeouts/retries protect its fan-out. Boundary validation is R6’s Zod applied to internal upstreams, and the aggregated view model is exactly what a Backend-for-Frontend (Concept 2) specializes per client.
2. Backend-for-Frontend: Typed View Models Node.js + TypeScript
A generic API gateway serves everyone the same way, but a mobile app and a web dashboard need different shapes: mobile wants a lean payload over a slow link, web wants a rich aggregate. A Backend-for-Frontend (BFF) is a gateway dedicated to one client, owning the view model that screen needs. Typing the view model separately from the domain models lets you reshape, rename, and trim per client without leaking internal entities — the frontend codes against a type built for it.

The core discipline is separating domain models (what services return) from view models (what a screen renders). The BFF calls upstreams (reusing the gateway’s typed fetch), then runs a typed assembler: a pure function (domain...) => ViewModel that picks fields, flattens nesting, computes derived values, and formats for display. Because the assembler’s output type is the client’s contract, TypeScript catches any drift between what the BFF produces and what the UI consumes.

A BFF-per-client also lets teams move independently: the mobile BFF can add a field without touching the web BFF. Keep assemblers pure and testable, and derive the shared client type from them (or a Zod schema) so the API client on the frontend and the assembler on the backend can’t disagree — the same single-source-of-truth idea as contract testing (07-08).

1
Keep domain models (upstream) distinct from view models (client-facing).
2
Assemble with a pure, typed function that trims/flattens/derives per screen.
3
Export the view-model type as the client contract; the UI codes against it.
TypeScript · example 1 of 4 — domain models vs a screen-specific view model
// Domain models — the raw shapes upstream services return
interface UserDomain { id: string; firstName: string; lastName: string; avatarKey: string }
interface OrderDomain { id: string; total: number; status: "paid" | "shipped"; createdAt: string }

// View model — exactly what the mobile "account" screen renders
interface AccountView {
  displayName: string;              // derived: firstName + lastName
  avatarUrl: string;                // derived: CDN url from avatarKey
  recentOrders: { id: string; total: string; shipped: boolean }[];  // trimmed + formatted
}
the view model is shaped for the screen, decoupled from internal domain fields
TypeScript · example 2 of 4 — a pure, typed assembler
function toAccountView(user: UserDomain, orders: OrderDomain[]): AccountView {
  return {
    displayName: `${user.firstName} ${user.lastName}`,
    avatarUrl: `https://cdn.example.com/${user.avatarKey}`,
    recentOrders: orders.slice(0, 5).map((o) => ({
      id: o.id,
      total: `$${o.total.toFixed(2)}`,          // formatted for display
      shipped: o.status === "shipped",          // derived boolean the UI wants
    })),
  };
}
a pure (domain...) => ViewModel function — trivially unit-testable, no I/O
TypeScript · example 3 of 4 — the BFF endpoint wires fetch + assemble
async function accountBff(userId: string): Promise<AccountView> {
  const [user, orders] = await Promise.all([
    callUpstream("http://users", `/users/${userId}`, UserSchema),
    callUpstream("http://orders", `/orders?user=${userId}`, OrderArraySchema),
  ]);
  return toAccountView(user, orders);     // I/O at the edge, logic in the pure assembler
}
the endpoint does I/O; the typed assembler owns the shape the client depends on
TypeScript · example 4 of 4 — the frontend shares the exact view-model type
// shared package: the BFF exports AccountView; the app imports it
import type { AccountView } from "@app/bff-account";

async function fetchAccount(): Promise<AccountView> {
  const res = await fetch("/bff/account");
  return res.json() as Promise<AccountView>;   // same type both sides — can't drift
}
// If the BFF changes AccountView, the frontend fails to compile — a caught contract break
one shared view-model type binds producer and consumer; a change breaks the build, not prod
Common Mistakes:
❌ Returning raw domain models straight to the client — internal fields leak, and every UI tweak forces a service change.
✅ Define a view model per screen and map to it in a pure assembler; keep domain shapes internal.
❌ One shared BFF trying to satisfy web and mobile at once — it accretes conditionals and couples the two clients.
✅ One BFF per client (or per team) so each evolves its view models independently.
Key Rules:
  • A BFF is a client-specific gateway: it owns the view model that one frontend renders.
  • Separate domain models (upstream) from view models (client) — map between them in a pure assembler.
  • Do I/O at the endpoint, logic in the assembler; pure functions are easy to type and test.
  • Share the view-model type (or a Zod schema) with the frontend so producer and consumer can’t drift.
Practice Exercises:
1Write a second view model (OrderDetailView) and assembler from the same domain models; note how the domain layer stays untouched.
2Derive AccountView from a Zod schema so the frontend gets both the runtime parser and the static type in one export.
3Add a Pick/Omit-based utility that proves at compile time your view model uses only fields present on the domain model.
Connections: The BFF is a specialization of the API gateway (Concept 1) — same typed upstream fetch, but with a per-client view model. The domain→view mapping reuses Pick/Omit utility types, and sharing the view-model type is contract testing’s (07-08) single-source-of-truth applied within your own stack. Pure assemblers echo the BFF-flavored pipe/compose functional patterns.
3. GraphQL Federation: Typed Subgraphs TypeScript
A single monolithic GraphQL schema doesn’t scale across teams — everyone edits one file and one deploy. Federation composes many independently owned subgraphs into one supergraph that clients query as if it were monolithic. Each team owns the types it’s responsible for and can extend types owned elsewhere. The TypeScript payoff comes from codegen: each subgraph gets generated resolver types from its schema, so a resolver that returns the wrong shape or forgets a reference resolver fails to compile.

The key primitive is the entity: a type marked @key(fields: "id") that can be referenced and extended across subgraphs. The owning subgraph defines the entity and a __resolveReference that fetches it by its key; another subgraph can add fields to that same entity, resolving them from its own data. The gateway (router) plans a query, calls each subgraph for the fields it owns, and stitches the result — the client never sees the seams.

Typing is codegen-driven: tools like GraphQL Code Generator read each subgraph’s SDL and emit Resolvers types, so your resolver map is checked against the schema field-by-field. That means the schema is the source of truth and the resolvers can’t drift from it — the same generate-types-from-schema discipline as tRPC/OpenAPI, but split across services. Federation is not a data-fetching silver bullet: naive reference resolvers cause N+1 calls, so batch with DataLoader.

1
Mark shared types as entities with @key; each is owned by one subgraph.
2
Generate resolver types per subgraph so resolvers match the schema.
3
Extend entities from other subgraphs via __resolveReference; batch to avoid N+1.
GraphQL · example 1 of 4 — two subgraphs sharing an entity
# users subgraph — OWNS the User entity
type User @key(fields: "id") {
  id: ID!
  name: String!
}

# reviews subgraph — EXTENDS User with fields it owns
extend type User @key(fields: "id") {
  id: ID! @external
  reviews: [Review!]!         # resolved from the reviews service's data
}
type Review { id: ID!  body: String! }
@key makes User a federated entity; a second subgraph extends it without owning it
TypeScript · example 2 of 4 — codegen-typed resolvers (users subgraph)
// generated by graphql-codegen from the subgraph SDL
import type { Resolvers } from "./generated/resolvers.js";

export const resolvers: Resolvers = {
  User: {
    // reference resolver: rehydrate a User from just its @key
    __resolveReference(ref) {          // ref is typed { id: string }
      return db.users.findById(ref.id);   // must return the User shape
    },
  },
  Query: {
    user: (_p, args) => db.users.findById(args.id),   // args typed from schema
  },
};
Resolvers is generated from the schema — a wrong return shape or missing field won't compile
TypeScript · example 3 of 4 — the extending subgraph resolves added fields
import type { Resolvers } from "./generated/resolvers.js";

export const resolvers: Resolvers = {
  User: {
    // this subgraph only knows the User's id (the @key); it resolves `reviews`
    reviews: (user) => reviewLoader.load(user.id),   // user: { id: string }
  },
};
the reviews subgraph fills in its own field on a User it doesn't own, keyed by id
TypeScript · example 4 of 4 — batch reference resolution to kill N+1
import DataLoader from "dataloader";

// one batched query per tick instead of one DB hit per referenced User
const reviewLoader = new DataLoader<string, Review[]>(async (userIds) => {
  const rows = await db.reviews.findByUserIds(userIds as string[]);
  const byUser = new Map<string, Review[]>();
  for (const r of rows) (byUser.get(r.userId) ?? byUser.set(r.userId, []).get(r.userId)!).push(r);
  return userIds.map((id) => byUser.get(id) ?? []);   // must return in input order
});
DataLoader coalesces per-entity reference resolves into one batched, ordered fetch
Common Mistakes:
❌ Hand-writing resolver types to match the SDL — they silently drift from the schema and break at query time, not build time.
✅ Generate Resolvers from each subgraph’s schema so resolvers are checked field-by-field against it.
❌ A reference resolver that hits the DB once per referenced entity — a classic N+1 that melts under load.
✅ Batch reference/field resolution with DataLoader so many keys become one ordered query.
Key Rules:
  • Federation composes independently owned subgraphs into one supergraph; each entity has a single owner.
  • @key makes a type a federated entity; other subgraphs extend it and resolve their own fields.
  • Generate resolver types from each subgraph’s SDL — the schema is the source of truth, resolvers are checked against it.
  • Reference/field resolvers are N+1 magnets; batch with DataLoader and return results in input order.
Practice Exercises:
1Add a products subgraph that extends Review with a product: Product field and write its __resolveReference.
2Configure graphql-codegen’s typescript-resolvers with a mappers entry so resolver parents use your DB row type, not the GraphQL type.
3Add a contract check (07-08) that fails CI if a subgraph removes a field another subgraph’s @requires depends on.
Connections: Federation is the API gateway (Concept 1) idea for GraphQL — a router stitching typed subgraphs instead of a REST gateway aggregating endpoints. Codegen’d resolver types are the schema-first, generate-types discipline of GraphQL Codegen and tRPC/OpenAPI. The N+1/DataLoader concern previews Concept 4’s caching, and cross-subgraph field ownership is the same “don’t break used fields” promise as consumer-driven contracts (07-08).
2Scaling Reads & Flow Control
4. Multi-Layer Caching Strategies Node.js + TypeScript
A gateway that fans out (Concept 1) and a federation router (Concept 3) both re-fetch the same hot data constantly. Caching turns repeat reads into cheap lookups, but a cache is a second copy of the truth — get the write path or the key wrong and you serve stale or leaked data. Typing a cache abstraction (a serializer + a typed key + a TTL) makes “what’s stored under this key and how fresh is it?” explicit, and lets you swap an in-memory layer and Redis behind one interface.

Two write strategies dominate. Cache-aside (lazy): on read, check the cache; on miss, load from the source, populate the cache, return — the application owns the cache. Write-through: writes go to the cache and the store together, so the cache is never stale for writes (at the cost of write latency). Both need a TTL so entries eventually expire, and explicit invalidation on mutation. A typed Cache<T> interface with a serializer keeps stored/loaded types honest across an in-process Map and a Redis backend.

The subtle failure is the cache stampede: a popular key expires and a thundering herd of requests all miss and hit the source at once. Guard it by coalescing concurrent misses for the same key into a single in-flight promise (single-flight). And always design the key namespace deliberately — include a version prefix so a schema change can invalidate a whole class of entries at once.

1
Model a typed Cache<T> (get/set/del + TTL) with a serializer, backend-agnostic.
2
Read via cache-aside; write via write-through or explicit invalidation.
3
Coalesce concurrent misses (single-flight) to prevent stampedes.
TypeScript · example 1 of 4 — a backend-agnostic typed cache
interface Cache<T> {
  get(key: string): Promise<T | undefined>;
  set(key: string, value: T, ttlMs: number): Promise<void>;
  del(key: string): Promise<void>;
}

// Redis-backed instance: the serializer keeps T honest across the wire
function redisCache<T>(redis: Redis, schema: { parse: (u: unknown) => T }): Cache<T> {
  return {
    async get(key) {
      const raw = await redis.get(key);
      return raw == null ? undefined : schema.parse(JSON.parse(raw));   // validate on read
    },
    async set(key, value, ttlMs) { await redis.set(key, JSON.stringify(value), "PX", ttlMs); },
    async del(key) { await redis.del(key); },
  };
}
one Cache<T> interface spans Map/Redis; the schema re-validates data crossing the process boundary
TypeScript · example 2 of 4 — cache-aside read
async function getUser(cache: Cache<User>, id: string): Promise<User> {
  const key = `user:v1:${id}`;                 // versioned namespace
  const hit = await cache.get(key);
  if (hit) return hit;                         // fast path

  const user = await db.users.findById(id);    // miss: load from source of truth
  await cache.set(key, user, 60_000);          // populate for next time
  return user;
}
check-miss-load-populate: the app owns the cache; the DB stays the source of truth
TypeScript · example 3 of 4 — write-through keeps the cache fresh on mutation
async function updateUser(cache: Cache<User>, id: string, patch: Partial<User>): Promise<User> {
  const updated = await db.users.update(id, patch);   // write the store
  await cache.set(`user:v1:${id}`, updated, 60_000);  // write the cache in the same op
  return updated;                                     // cache can't be stale for this write
}
// simpler alternative: write the store, then cache.del(key) — next read repopulates (write-invalidate)
write-through updates both copies together; write-invalidate deletes and lets the next read refill
TypeScript · example 4 of 4 — single-flight to prevent a stampede
function singleFlight<T>() {
  const inflight = new Map<string, Promise<T>>();
  return (key: string, load: () => Promise<T>): Promise<T> => {
    const existing = inflight.get(key);
    if (existing) return existing;                    // join the in-flight load
    const p = load().finally(() => inflight.delete(key));
    inflight.set(key, p);
    return p;
  };
}

const flight = singleFlight<User>();
// on a hot-key miss, 100 concurrent callers share ONE db load, not 100
const user = await flight(`user:v1:${id}`, () => db.users.findById(id));
concurrent misses for a key collapse into a single in-flight promise — no thundering herd
Common Mistakes:
❌ Caching on read but never invalidating on write — the cache serves a stale value until the TTL lapses, sometimes for minutes.
✅ Pair every mutation with a write-through update or a del(key); treat invalidation as part of the write.
❌ Letting every concurrent miss hit the database — a popular key’s expiry triggers a stampede that can topple the source.
✅ Coalesce concurrent misses with single-flight, and consider a small random TTL jitter so keys don’t all expire together.
Key Rules:
  • A cache is a second copy of the truth — every write path must update or invalidate it, not just the read path.
  • Cache-aside (lazy load on miss) is the default; write-through trades write latency for never-stale writes.
  • Type a backend-agnostic Cache<T> with a serializer that re-validates data crossing the process boundary.
  • Use versioned key namespaces and single-flight coalescing; add TTL jitter to avoid synchronized expiry.

5. Backpressure & Flow Control in Typed Pipelines Node.js + TypeScript
A fast producer feeding a slow consumer — a firehose of DB rows piped to a rate-limited API, thousands of queued jobs (07-08) hitting one worker — will buffer unbounded and eventually exhaust memory. Backpressure is the mechanism by which a slow consumer tells a fast producer to slow down. Pull-based async iteration in TypeScript gives you this for free: the producer only advances when the consumer asks for the next value, so a for await loop that does slow work naturally throttles the source.

The typed primitive is the async iterable (AsyncIterable<T> / async function*). It is pull-based: each await next() runs the generator up to the next yield and then pauses — no value is produced until requested. So a consumer that awaits slow work between pulls automatically bounds how far ahead the producer runs. This is the opposite of an event emitter (push-based), which fires regardless of whether anyone can keep up and needs a manual buffer.

To go faster than strictly sequential without going unbounded, process with a bounded concurrency window: keep at most N items in flight, and only pull the next when a slot frees. That’s the flow-control sweet spot — throughput without unbounded buffering. Node’s stream layer encodes the same idea in highWaterMark and the .pipe() / pipeline() contract, but async iterators express it more directly and with better types.

1
Model the source as a pull-based AsyncIterable<T> (async function*).
2
Consume with for await so slow work throttles the producer automatically.
3
For throughput, process with a bounded-concurrency window, not an unbounded buffer.
TypeScript · example 1 of 4 — a pull-based typed source
// paginates a DB, yielding one page at a time — nothing runs until pulled
async function* fetchAllUsers(pageSize = 100): AsyncGenerator<User, void, unknown> {
  let cursor: string | null = null;
  do {
    const { rows, next } = await db.users.page(cursor, pageSize);  // one page per pull
    for (const u of rows) yield u;
    cursor = next;
  } while (cursor);
}
AsyncGenerator<User> is lazy: the next page is fetched only when the consumer asks
TypeScript · example 2 of 4 — a slow consumer throttles the producer for free
async function syncToApi() {
  for await (const user of fetchAllUsers()) {
    await rateLimitedApi.send(user);   // slow: the loop won't pull the next user until this resolves
  }
  // memory stays flat — the generator is paused between iterations, no giant buffer
}
`for await` pulls one at a time; the producer can't outrun the consumer — backpressure by construction
TypeScript · example 3 of 4 — bounded concurrency for throughput without a firehose
async function mapBounded<T, R>(
  src: AsyncIterable<T>, limit: number, fn: (item: T) => Promise<R>,
): Promise<R[]> {
  const results: R[] = [];
  const inflight = new Set<Promise<void>>();
  for await (const item of src) {
    const p = fn(item).then((r) => { results.push(r); });
    inflight.add(p);
    p.finally(() => inflight.delete(p));
    if (inflight.size >= limit) await Promise.race(inflight);   // wait for a slot before pulling next
  }
  await Promise.all(inflight);
  return results;
}

// at most 5 sends in flight at once — faster than sequential, still bounded
await mapBounded(fetchAllUsers(), 5, (u) => rateLimitedApi.send(u));
the window caps in-flight work at `limit`; pulling pauses until a slot frees — throughput with a ceiling
TypeScript · example 4 of 4 — Node streams encode the same contract
import { pipeline } from "node:stream/promises";
import { Readable, Transform } from "node:stream";

// highWaterMark is the buffer bound; pipeline propagates backpressure + errors
await pipeline(
  Readable.from(fetchAllUsers()),                        // async iterable -> stream
  new Transform({
    objectMode: true, highWaterMark: 16,                 // at most 16 buffered objects
    transform(user: User, _enc, cb) { enrich(user).then((u) => cb(null, u), cb); },
  }),
  async function (source) { for await (const u of source) await sink.write(u); },
);
streams express backpressure via highWaterMark; pipeline() wires throttling + error propagation
Common Mistakes:
❌ Promise.all(items.map(fn)) over a huge source — it launches everything at once, unbounded, and can exhaust memory or hammer the downstream.
✅ Iterate a pull-based source and cap in-flight work with a bounded-concurrency window.
❌ Pushing from an event emitter into an unbounded array when the consumer is slower than the producer — the buffer grows without limit.
✅ Prefer pull-based AsyncIterable (or streams with highWaterMark) so the consumer’s pace controls the producer.
Key Rules:
  • Backpressure = a slow consumer throttling a fast producer; without it, buffers grow until memory dies.
  • Async iterables are pull-based: nothing is produced until requested, so for await throttles for free.
  • For throughput, use a bounded-concurrency window (at most N in flight), never an unbounded Promise.all.
  • Node streams encode the same contract via highWaterMark; pipeline() propagates both backpressure and errors.
