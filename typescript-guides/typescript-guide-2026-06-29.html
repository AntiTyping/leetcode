Review — 6 Concepts
R1. Utility Types Generic & Utility Types 18 days since review · rep 3 → 4
Partial, Required, Pick, Omit, Record, Readonly, Exclude, Extract, NonNullable, and ReturnType are built-in transformations over object and union types. Compose them instead of hand-rolling — and remember they operate at the top level unless you write a recursive version.
Key Pattern 
interface User { id: string; name: string; email: string; age: number }

type UserPatch  = Partial<Pick<User, "name" | "email">>;  // { name?; email? }
type PublicUser = Omit<User, "email">;                     // no email
type ById       = Record<string, User>;                    // dictionary
type R          = ReturnType<() => User>;                   // User
type NoNull     = NonNullable<string | null | undefined>;   // string
compose Pick/Omit/Partial/Record rather than re-declaring shapes
Challenge Exercise

Write type Mutable<T> that strips readonly using a mapped type, then compose Partial<Pick<T, K>> to type the body of a PATCH endpoint that updates a subset of fields. Finally implement update<T, K extends keyof T>(entity: T, patch: Partial<Pick<T, K>>): T and confirm an unknown field in patch is rejected.

Pitfall: Reaching for Partial<T> on a nested object and expecting deep optionality — it’s shallow. Also, Omit<T, K> does not constrain K to keyof T (it accepts any string), so a mistyped key silently omits nothing.
R2. Conditional Types Advanced Type System 17 days since review · rep 3 → 4
T extends U ? X : Y picks a branch at the type level. Over a naked union type parameter it distributes, evaluating member-by-member; infer captures a piece of the matched type for reuse in the branches.
Key Pattern 
type ElementType<T> = T extends (infer U)[] ? U : T;
type A = ElementType<string[]>;   // string
type B = ElementType<number>;     // number

type NonNull<T> = T extends null | undefined ? never : T;
type C = NonNull<string | null>;  // string  (distributes; null -> never)
infer extracts; distribution maps the conditional across union members
Challenge Exercise

Write Flatten<T> that unwraps one array level and a recursive DeepFlatten<T>. Then write Params<F> using infer to pull a function’s parameter tuple, and explain why wrapping the check as [T] extends [U] turns off distribution.

Pitfall: Forgetting that conditionals distribute over a bare union type parameter — T extends any ? T[] : never applies to each member separately, not to the whole union. Wrap both sides in a 1-tuple ([T] extends [any]) to treat the union as a single unit.
R3. Mapped Types Advanced Type System 17 days since review · rep 3 → 4
{ [K in keyof T]: ... } rebuilds a type key-by-key. Add or strip modifiers with ?/-? and readonly/-readonly, and remap keys with as (map a key to never to drop it).
Key Pattern 
type Optional<T> = { [K in keyof T]?: T[K] };
type Strip<T>    = { -readonly [K in keyof T]-?: T[K] };  // remove readonly + optional

type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};
// Getters<{ name: string }>  ->  { getName: () => string }
modifiers add/remove ?/readonly; `as` remaps (or drops) keys
Challenge Exercise

Write Nullable<T> making every property T[K] | null, then a key-remapping EventHandlers<T> that turns { click: MouseEvent } into { onClick: (e: MouseEvent) => void }. Finally, use as + a conditional never to build NonFunctionKeys<T> that drops every method-valued property.

Pitfall: Confusing “drop the key” with “make the value never.” Remapping a key to never via as removes it; returning never in the value position instead keeps the key with an unusable never type.
R4. keyof & Indexed Access Types Advanced Type System 16 days since review · rep 3 → 4
keyof T is the union of T’s keys; T[K] looks up the type at key K; T[keyof T] is the union of all value types; and T[number] on a tuple/array yields the element type.
Key Pattern 
interface User { id: number; name: string }
type Keys   = keyof User;         // "id" | "name"
type NameT  = User["name"];       // string
type Values = User[keyof User];   // number | string

const routes = ["home", "about"] as const;
type Route = typeof routes[number];   // "home" | "about"
keyof gives keys; T[K] and T[keyof T] read value types
Challenge Exercise

Write getIn<T, K extends keyof T>(obj: T, key: K): T[K], then a two-level getIn2<T, K1 extends keyof T, K2 extends keyof T[K1]>. Confirm the return type is exactly the nested value type by hovering it — e.g. getIn2(config, "db", "port") should be number, not unknown.

Pitfall: Writing T[keyof T] when you meant one specific key — it collapses distinct value types into a union. Also, keyof on a type with an index signature includes string/number, widening past your literal keys.
R5. Type Predicates & Assertion Functions Advanced Type System 16 days since review · rep 3 → 4
A user-defined type guard x is T narrows in the caller after a true return; an asserts x is T (or asserts cond) narrows by throwing on failure. Both require an explicit return-type annotation.
Key Pattern 
function isString(x: unknown): x is string {
  return typeof x === "string";
}
function assertDefined<T>(x: T): asserts x is NonNullable<T> {
  if (x == null) throw new Error("nullish");
}

const v: unknown = "hi";
if (isString(v)) v.toUpperCase();   // v: string here

let u: string | undefined;
assertDefined(u); u.length;         // u: string after the assert
`is` narrows on true; `asserts` narrows by throwing
Challenge Exercise

Write isNonEmpty<T>(a: T[]): a is [T, ...T[]] and an assertNever(x: never): never for exhaustive switches. Then write hasKey<K extends string>(o: object, k: K): o is Record<K, unknown> and use it to safely read a dynamic property.

Pitfall: A predicate whose body doesn’t actually verify the claimed type — the compiler trusts the is annotation, so a wrong guard silently lies to every caller. And an assertion function needs an explicit asserts return type; inference won’t produce it.
R6. Generic Constraints & Defaults Functions & Patterns 16 days since review · rep 3 → 4
T extends U restricts what a type parameter accepts (and unlocks property access on T); T = Default supplies a fallback when a type argument is neither passed nor inferable. Constrain to the minimum capability you need.
Key Pattern 
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}
longest([1], [1, 2]);   // number[]
longest("a", "bb");     // string

interface Box<T = string> { value: T }
const b: Box = { value: "hi" };   // T defaults to string
constrain to a capability; default only fills an un-inferable slot
Challenge Exercise

Write pluck<T, K extends keyof T = keyof T>(o: T, k: K): T[K] and a constrained merge<A extends object, B extends object>(a: A, b: B): A & B. Then define createStore<S = Record<string, unknown>>() and show exactly when the default S kicks in versus when an explicit or inferred argument overrides it.

Pitfall: Over-constraining (T extends string when you only read .length) or adding a default that masks a missing inference site. A default applies only when a type argument is neither supplied nor inferable — an inferred T always beats its default.
1React + TS: Refs & Imperative Handles
1. forwardRef & Generic Components with Refs React + TypeScript
To let a parent grab a child’s DOM node (focus an input, scroll a list) or its imperative API, a component must forward a ref. forwardRef types the second ref parameter cleanly — but it has a famous wrinkle: it erases a component’s own generic, so a generic list that also forwards a ref needs a deliberate workaround.

forwardRef<RefType, Props> turns a component into one that receives (props, ref), where ref is typed Ref<RefType>. For a DOM input, RefType is HTMLInputElement; attach it with ref={ref} and the parent’s useRef<HTMLInputElement> lines up exactly.

The wrinkle: forwardRef’s type signature isn’t generic-preserving, so a component written as forwardRef(function List<T>(...)) stops being generic — T collapses to unknown at every call site. Two recoveries: cast the result to a generic call signature, or (React 19+) skip forwardRef entirely and accept ref as a normal prop, which keeps generics natively.

1
Wrap in forwardRef<Element, Props>; the ref parameter is Ref<Element>, not Element.
2
Attach ref to the underlying element; consumers pass a matching useRef.
3
Generic components lose their type parameter — recover it with a call-signature cast, or use React 19 ref-as-prop.
TypeScript · example 1 of 4 — a basic DOM forwardRef
import { forwardRef } from "react";

type InputProps = { label: string } & React.ComponentPropsWithoutRef<"input">;

const TextInput = forwardRef<HTMLInputElement, InputProps>(
  function TextInput({ label, ...rest }, ref) {
    return <label>{label}<input ref={ref} {...rest} /></label>;
  }
);
ref param is Ref<HTMLInputElement>; rest is the native input props
TypeScript · example 2 of 4 — the consumer drives the ref
function Form() {
  const inputRef = useRef<HTMLInputElement>(null);
  useEffect(() => { inputRef.current?.focus(); }, []);
  return <TextInput ref={inputRef} label="Name" />;   // ref typed & checked
}
inputRef.current is HTMLInputElement | null
TypeScript · example 3 of 4 — the generic-loss problem
// we WANT: <List<T> items ref />, ref forwarded to the <ul>
const List = forwardRef(function List<T>(
  { items }: { items: T[] },
  ref: React.Ref<HTMLUListElement>
) {
  return <ul ref={ref}>{items.map((x, i) => <li key={i}>{String(x)}</li>)}</ul>;
});

// but List is no longer generic: T is fixed to unknown at every call site
forwardRef drops the <T> — items infers as unknown[] for all callers
TypeScript · example 4 of 4 — recovering the generic
// (a) re-assert the intended generic call signature
const GenericList = List as <T>(
  p: { items: T[] } & React.RefAttributes<HTMLUListElement>
) => React.ReactElement;

// (b) React 19: ref is just a prop — no forwardRef, generics survive
function List19<T>({ items, ref }: {
  items: T[];
  ref?: React.Ref<HTMLUListElement>;
}) {
  return <ul ref={ref}>{items.map((x, i) => <li key={i}>{String(x)}</li>)}</ul>;
}
cast restores per-call T; React 19 ref-as-prop keeps it natively
Common Mistakes:
❌ Typing the ref parameter as HTMLInputElement instead of Ref<HTMLInputElement> — a caller’s ref object can’t be assigned to a bare element type.
✅ Use React.Ref<T> / React.ForwardedRef<T> for the parameter; only ref.current is T | null.
❌ Expecting a generic forwardRef component to keep inferring its type parameter at call sites.
✅ Recover the generic with a call-signature cast, or (React 19+) drop forwardRef and accept ref as a normal prop.
Key Rules:
  • 2026-07-04 (Day 31)
  • forwardRef<Element, Props> supplies a typed second ref param; forward it to the underlying element.
  • The ref parameter’s type is Ref<Element>, not Element.
  • forwardRef erases a component’s own generic — cast to a generic call signature to restore it.
  • On React 19+, ref is a regular prop, so generic + ref components need no special handling.
Practice Exercises:
1Wrap a <Button> in forwardRef so a parent can useRef<HTMLButtonElement> and call .focus(). Confirm the ref param is Ref<HTMLButtonElement> and .current is HTMLButtonElement | null.
2Write a generic <Select<T>> that forwards a ref to its <select>; observe that T collapses to unknown, then restore inference with a call-signature cast.
3Re-implement the same <Select<T>> for React 19 by accepting ref?: Ref<HTMLSelectElement> as an ordinary prop, and show generics survive with no cast.
Connections: forwardRef is Generics Basics meeting React Component Props (Day 24), and the generic-loss cast reuses call-signature types. It sets up Concept 2 (useImperativeHandle), which customizes what the forwarded ref exposes rather than handing over the raw node.
2. useImperativeHandle: Typing the Exposed API React + TypeScript
Sometimes a parent shouldn’t touch the raw DOM node but a curated API — videoRef.current.play(), formRef.current.submit(). useImperativeHandle lets a component expose a typed handle object through its ref, and typing that handle is what gives the parent autocomplete and safety while hiding the implementation.

Pair forwardRef<Handle, Props> with useImperativeHandle(ref, () => ({ ... }), deps). The Handle interface is the contract the parent sees; the factory returns an object matching it, and the dependency array controls when the handle is recomputed. The parent uses useRef<Handle> and calls the typed methods.

Two typing wins matter. First, declare an explicit Handle interface (or derive it with ReturnType) so the exposed surface can’t drift from — or accidentally widen to — the whole DOM node. Second, treat the third argument like any hook dependency list: everything the handle’s closures read must appear there, or you ship a stale-closure bug behind a perfectly-typed method.

1
Define a Handle interface = the imperative API you want to expose.
2
forwardRef<Handle, Props> + useImperativeHandle(ref, () => ({ ... }), deps).
3
Parent uses useRef<Handle> and calls only the typed methods.
TypeScript · example 1 of 4 — define the handle + expose it
export interface VideoHandle {
  play(): void;
  pause(): void;
  seek(seconds: number): void;
}

const Video = forwardRef<VideoHandle, { src: string }>(
  function Video({ src }, ref) {
    const el = useRef<HTMLVideoElement>(null);
    useImperativeHandle(ref, () => ({
      play:  () => el.current?.play(),
      pause: () => el.current?.pause(),
      seek:  (s) => { if (el.current) el.current.currentTime = s; },
    }), []);
    return <video ref={el} src={src} />;
  }
);
the ref now exposes VideoHandle, not HTMLVideoElement
TypeScript · example 2 of 4 — the parent calls the typed API
function Player() {
  const ref = useRef<VideoHandle>(null);
  return (
    <>
      <Video ref={ref} src="/clip.mp4" />
      <button onClick={() => ref.current?.play()}>Play</button>
      <button onClick={() => ref.current?.seek(30)}>Skip 30s</button>
    </>
  );
}
// ref.current?.rewind()  // error: 'rewind' is not on VideoHandle
ref.current is VideoHandle | null; seek(30) checked, unknown methods error
TypeScript · example 3 of 4 — derive the handle type so it can’t drift
// single source of truth: the object the factory returns
function buildVideoApi(el: React.RefObject<HTMLVideoElement>) {
  return {
    play:  () => el.current?.play(),
    pause: () => el.current?.pause(),
  };
}
type VideoApi = ReturnType<typeof buildVideoApi>;   // stays in sync
VideoApi tracks the implementation via ReturnType — no duplicate interface
TypeScript · example 4 of 4 — the deps array is real
useImperativeHandle(ref, () => ({
  submit: () => doSubmit(formState),   // closes over formState
}), [formState]);   // recompute the handle whenever formState changes
omit [formState] and submit() fires with a stale snapshot
Common Mistakes:
❌ Returning a handle whose shape doesn’t match forwardRef<Handle> — or leaving Handle as the DOM element type so the parent sees the whole node.
✅ Declare an explicit Handle interface (or derive it with ReturnType) and expose only those methods.
❌ Omitting the dependency array so the handle’s closures capture stale state.
✅ List every value the handle reads in the deps array, exactly like useEffect/useCallback.
Key Rules:
  • useImperativeHandle customizes what a ref exposes; pair it with forwardRef<Handle, Props>.
  • The Handle type is the contract the parent sees — keep it minimal and intentional.
  • Derive the handle type from the implementation (ReturnType) to prevent drift.
  • The third argument is a dependency array — include everything the handle’s closures capture.
Practice Exercises:
1Build a <TextField> exposing { focus(): void; clear(): void } via useImperativeHandle; drive it from a parent using useRef<TextFieldHandle>.
2Add a getValue(): string method and confirm the parent’s ref.current?.getValue() is typed as string | undefined (because of the optional chain).
3Refactor so the handle type is ReturnType<typeof buildHandle>; then add a method to the implementation and confirm the parent sees it with no separate interface edit.
Connections: This builds directly on Concept 1 (forwardRef) and is Interfaces vs Type Aliases plus ReturnType from Utility Types (R1). The curated-API idea echoes Branded/Opaque boundaries — expose intent, hide the node.
2React + TS: Composition & State
3. Higher-Order Components (HOC) Typing React + TypeScript
An HOC wraps a component to inject props — a theme, a store slice, the current user. The typing challenge is precise: the returned component must expose the original props minus the ones the HOC supplies, so callers still pass what they must and are forbidden from passing what’s injected.

Make the HOC generic over the wrapped component’s props P, and return a component whose props are Omit<P, keyof Injected>. Constrain P extends Injected so the injection type-checks, and bridge the spread back to P with a single trusted cast at the injection seam. Everything the HOC does not inject passes straight through.

In modern React a custom hook usually beats an HOC — no prop-subtraction gymnastics, no wrapper component in the tree. Reach for an HOC when you must wrap arbitrary or class components, add cross-cutting render logic, or forward refs around a boundary. When you do, Omit is the load-bearing utility that keeps the public prop surface honest.

1
The HOC is generic over the wrapped component’s props P (constrained to include the injected props).
2
Return a component typed Omit<P, keyof Injected> — injected props vanish from the public API.
3
Bridge the merged props back to P with one cast at the seam; everything else passes through.
TypeScript · example 1 of 4 — inject a prop, subtract it from the public props
type WithUser = { user: { id: string; name: string } };

function withUser<P extends WithUser>(Cmp: React.ComponentType<P>) {
  return function Wrapped(props: Omit<P, keyof WithUser>) {
    const user = useCurrentUser();                 // injected internally
    return <Cmp {...(props as P)} user={user} />;   // bridge back to P
  };
}
Wrapped’s props are P without `user`
TypeScript · example 2 of 4 — the caller no longer passes the injected prop
type ProfileProps = WithUser & { showEmail: boolean };
function Profile({ user, showEmail }: ProfileProps) { /* ... */ return null; }

const ConnectedProfile = withUser(Profile);

<ConnectedProfile showEmail />;            // ok — user is injected
<ConnectedProfile user={u} showEmail />;   // error: 'user' was removed
Property 'user' does not exist on the wrapped component’s props
TypeScript · example 3 of 4 — a pass-through HOC keeps P intact
function withLogging<P extends object>(Cmp: React.ComponentType<P>) {
  const Wrapped = (props: P) => {
    useEffect(() => console.log(Cmp.displayName ?? Cmp.name, "mounted"), []);
    return <Cmp {...props} />;
  };
  Wrapped.displayName = `withLogging(${Cmp.displayName ?? Cmp.name})`;
  return Wrapped;
}
injects nothing, so the public props stay exactly P
TypeScript · example 4 of 4 — a hook is usually the better tool
// HOC form
const Connected = withUser(Profile);

// hook equivalent — no Omit gymnastics, no wrapper in the tree
function Profile2({ showEmail }: { showEmail: boolean }) {
  const user = useCurrentUser();   // just call the hook
  return null;
}
prefer a hook unless you must wrap arbitrary/class components
Common Mistakes:
❌ Returning ComponentType<P> so the caller is still required to pass the prop the HOC injects.
✅ Return ComponentType<Omit<P, keyof Injected>> so injected props disappear from the public API.
❌ Spreading props without the as P bridge and fighting the “Omit<P> is not assignable to P” error.
✅ Cast the merged {...props, ...injected} back to P at the single trusted seam inside the HOC.
Key Rules:
  • Make the HOC generic over the wrapped props P.
  • Subtract injected props with Omit<P, keyof Injected> in the returned component’s type.
  • Constrain P extends Injected so the injection itself type-checks.
  • Prefer a custom hook unless you must wrap arbitrary/class components or add cross-cutting render logic.
Practice Exercises:
1Write withTheme<P extends { theme: Theme }> that injects theme and returns a component typed Omit<P, "theme">. Confirm callers can’t pass theme but must still pass everything else.
2Add a second injected prop (t: TranslateFn) and widen the Omit to remove both; verify both vanish from the public props.
3Rewrite withTheme as a useTheme() hook and note which HOC use cases (class components, wrapping render output, ref forwarding) the hook can’t replace.
Connections: HOC typing is Utility Types (R1: Omit) plus Generic Constraints (R6) applied to components. It contrasts with Concept 1’s ref forwarding (HOCs often need to forward refs too) and with the render-prop / hook alternatives from Day 25.
4. Typing children: ReactNode vs ReactElement React + TypeScript
children is the most-used prop, and its type controls what a component will accept: any renderable content, exactly one element, only elements of a specific shape, or a function. Picking the right one turns “anything goes” into a compile-time guarantee about how the component may be composed.

React.ReactNode is the broadest renderable type — strings, numbers, null, boolean, arrays, and elements. It’s the right default. React.ReactElement is a single element object (not text or null); use it to require an actual element. Narrow further with ReactElement<ChildProps> to constrain a child’s prop shape, or a function type for render props.

One subtlety trips people up: constraining children to ReactElement<TabProps> checks the props shape, not the component identity — anything whose props are assignable will pass, not only your <Tab>. The PropsWithChildren<P> helper is a convenience that adds an optional children?: ReactNode; if you need children to be required, type it explicitly instead.

1
React.ReactNode — the broadest renderable (text, arrays, null, elements). The default choice.
2
React.ReactElement — a single element object; use it to require an actual element.
3
Narrow with ReactElement<P> for a specific child shape, or a function type for render props.
TypeScript · example 1 of 4 — the common case: ReactNode
type CardProps = { title: string; children: React.ReactNode };
function Card({ title, children }: CardProps) {
  return <section><h3>{title}</h3>{children}</section>;
}

<Card title="Hi">text, <b>elements</b>, {null}, or arrays</Card>;   // all ok
ReactNode accepts text, elements, arrays, null, boolean
TypeScript · example 2 of 4 — require exactly one element
type ModalProps = { children: React.ReactElement };
function Modal({ children }: ModalProps) {
  return <div className="modal">{children}</div>;
}

<Modal><Form /></Modal>;      // ok: one element
<Modal>plain text</Modal>;    // error: string is not a ReactElement
Type 'string' is not assignable to type 'ReactElement'
TypeScript · example 3 of 4 — constrain to a child’s props
type TabProps = { label: string };
function Tab(_: TabProps) { return null; }

type TabsProps = { children: React.ReactElement<TabProps>[] };
function Tabs({ children }: TabsProps) {
  return <div>{children.map((c) => c.props.label)}</div>;   // props.label: string
}

<Tabs><Tab label="A" /><Tab label="B" /></Tabs>;   // ok, props typed
c.props is TabProps — label autocompletes and is checked
TypeScript · example 4 of 4 — PropsWithChildren
import type { PropsWithChildren } from "react";

type PanelProps = PropsWithChildren<{ open: boolean }>;
// equivalent to: { open: boolean } & { children?: React.ReactNode }

function Panel({ open, children }: PanelProps) {
  return open ? <div>{children}</div> : null;
}
PropsWithChildren<P> = P & { children?: ReactNode } (children optional)
Common Mistakes:
❌ Typing children: JSX.Element (an alias of ReactElement) when you also want to allow strings/arrays/null — valid content is rejected.
✅ Use React.ReactNode for general content; reserve ReactElement for “must be a single element.”
❌ Assuming ReactElement<TabProps> guarantees the child is a <Tab> at runtime — it only checks the props shape.
✅ Treat the element-props constraint as a shape hint; if you truly depend on identity, check child.type === Tab at runtime.
Key Rules:
  • Default to React.ReactNode — it’s the true “renderable content” type.
  • Use React.ReactElement to require a single element, ReactElement<P> to constrain its props.
  • PropsWithChildren<P> adds an optional children?: ReactNode.
  • Element-props typing is structural (shape), not nominal (component identity).
Practice Exercises:
1Type a <Layout> that accepts arbitrary children, then a <Center> that requires exactly one ReactElement; prove text is rejected by the second but not the first.
2Build <RadioGroup> whose children are ReactElement<{ value: string }>[] and map over child.props.value. Confirm a child missing value is a type error.
3Compare PropsWithChildren<P> with manually writing children: React.ReactNode — when would you want children required instead, and how would you type that?
Connections: This is Type Annotations (R1) and Union Types (ReactNode is a big union) applied to the children prop, and it dovetails with the function-as-children render props from Day 25. The props-constrained element type reuses Generics + Indexed Access (R4).
5. useReducer with Discriminated Actions React + TypeScript
Once component state has several fields that change together, useReducer beats scattered useStates — and a discriminated-union Action type makes every dispatch exhaustively checked, with each action’s payload tied to its type. It’s the discriminated-union pattern applied to React state, and the same mental model as Redux.

Model Action as a union tagged by a literal type, each member carrying exactly the payload it needs. Write reducer(state: State, action: Action): State with a switch (action.type) — each case narrows action to the matching member, so its payload is fully typed. Add a never default to make the compiler force you to handle every action.

useReducer(reducer, initial) then infers dispatch’s parameter from the reducer’s action type — no manual generic needed. A wrong or missing payload at a call site is a compile error, not a runtime surprise. A lazy initializer (the optional third argument) stays typed too: its return type must match State.

1
Model Action as a discriminated union keyed on type, each member with its own payload.
2
Write reducer(state, action): State and switch (action.type) — each case sees its payload.
3
useReducer(reducer, initial) — dispatch accepts only valid actions; add a never default.
TypeScript · example 1 of 4 — state + discriminated actions
type State = { count: number; step: number };

type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "setStep"; step: number }
  | { type: "reset"; to: number };
each action carries exactly the payload its type needs
TypeScript · example 2 of 4 — the reducer narrows per case
function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "increment": return { ...state, count: state.count + state.step };
    case "decrement": return { ...state, count: state.count - state.step };
    case "setStep":   return { ...state, step: action.step };   // action.step: number
    case "reset":     return { count: action.to, step: 1 };     // action.to: number
    default: {
      const _exhaustive: never = action;   // new action -> compile error here
      return _exhaustive;
    }
  }
}
action is narrowed to the matching member in each case
TypeScript · example 3 of 4 — useReducer infers dispatch

the third arg’s return type must match State
Common Mistakes:
❌ Typing action: { type: string; payload?: any } — every dispatch compiles, payloads are any, and exhaustiveness is impossible.
✅ Make Action a discriminated union so each type locks in its own payload and the reducer can narrow.
❌ Omitting the never default, so adding a new action silently falls through without a handler.
✅ Assign action to a never-typed variable in default to force handling of every action at compile time.
Key Rules:
  • Model actions as a discriminated union keyed on a literal type.
  • Type the reducer’s return as State; the switch narrows each case’s payload automatically.
  • useReducer infers dispatch’s parameter from the reducer — no manual generic needed.
  • Add a never default for compile-time exhaustiveness as actions grow.
Practice Exercises:
1Model a <Toggle>’s state with actions on, off, toggle, and a set action carrying value: boolean. Write the reducer with a never default and confirm a set without value is an error.
2Build a fetch reducer with State = { status: "idle" | "loading" | "success" | "error"; ... } and actions start, success (carries data: T), failure (carries error: string). Make the whole thing generic over T.
3Wrap the reducer + dispatch in a typed context (from Day 25’s Context concept) so descendants can dispatch with full type-checking, and confirm an unknown action type is rejected.
Connections: This is Discriminated Unions + never & Exhaustiveness applied to React state, and it composes with Typing React Context (Day 25) to build a fully-typed store. The lazy-initializer typing reuses return-type inference from Generic Constraints (R6).
