R1. Zustand with TypeScript State Management overdue · rep 0 → 1
Zustand’s create is curried in TypeScript: you write create<State>()(setFn) — note the empty () after the type argument. That extra call is what lets the store type be inferred correctly when middleware is layered on.
Key Pattern 
import { create } from "zustand";

interface BearState {
  bears: number;
  increase: (by: number) => void;
}

// note the curried call:  create<T>()(...)
const useStore = create<BearState>()((set) => ({
  bears: 0,
  increase: (by) => set((s) => ({ bears: s.bears + by })),
}));

const bears = useStore((s) => s.bears); // number, fully typed
useStore selector is typed against BearState
Challenge Exercise

Define a useCartStore with items: Item[], addItem(item), and total derived in selectors. Wrap it in the persist middleware and give the store the correct curried type so set/get stay typed. Confirm a selector returning state.total is inferred as number.

Pitfall: Writing create<BearState>((set) => ...) without the second call. The single-call form does not give TypeScript the inference hook it needs once middleware is involved, producing confusing “not assignable” errors. Always use the curried create<T>()(...) form in TS.
R2. HKT Simulation & Type-Level Computation Advanced Type Patterns overdue · rep 0 → 1
TypeScript has no real higher-kinded types, so we simulate them: declare an interface keyed by a brand whose members map an input type to an output type, then “apply” the type constructor by indexing that interface. This is the “defunctionalization” trick (lightweight-HKT pattern).
Key Pattern 
interface HKT { readonly _A: unknown; readonly type: unknown; }
type Apply<F extends HKT, A> = (F & { readonly _A: A })["type"];

interface ArrayHKT extends HKT { type: this["_A"][]; }
interface PromiseHKT extends HKT { type: Promise<this["_A"]>; }

type T1 = Apply<ArrayHKT, string>;    // string[]
type T2 = Apply<PromiseHKT, number>;  // Promise<number>
T1 = string[] T2 = Promise<number>
Challenge Exercise

Add an OptionHKT whose type is this["_A"] | null, and write a generic mapHKT-style type alias that, given an F extends HKT and a mapping from A to B, produces Apply<F, B>. Verify Apply<OptionHKT, User> resolves to User | null.

Pitfall: Expecting full type inference of the kind F<_> the way Haskell/Scala provide. This pattern only simulates HKTs — it works for explicit Apply calls but TypeScript will not infer the type constructor F from a value, so use it sparingly where the constructor is known statically.
R3. Declaration Merging & Module Augmentation Advanced Type Patterns overdue · rep 0 → 1
Two interfaces with the same name in the same scope merge their members. declare module "x" augments an existing module’s types; declare global augments global scope. This is how you extend third-party types (e.g. Express Request) without forking them.
Key Pattern 
// augment a third-party module's interface
import "express";

declare module "express" {
  interface Request {
    user?: { id: number; role: string };  // merged into Request
  }
}

// now app.get((req) => req.user?.id) is fully typed everywhere
Request now has an optional `user` property
Challenge Exercise

Augment the global Window interface (via declare global) to add a typed __APP_CONFIG__: { apiUrl: string }. Then merge a namespace into a function so that greet is callable and exposes greet.lang: string. Show both compile and are typed.

Pitfall: Putting a module augmentation in a file with no top-level import/export. Without them the file is a script, not a module, so declare module is interpreted as ambient declaration rather than augmentation — and your additions silently fail to merge. Add an import or export {} to make it a module.
R4. Ambient Declarations & Global Types Advanced Type Patterns overdue · rep 0 → 1
declare describes the shape of something that exists at runtime but has no TypeScript source — a global from a script tag, a non-code asset import, or an untyped JS module. Ambient declarations emit no JavaScript; they are pure compile-time information, usually living in .d.ts files.
Key Pattern 
// globals.d.ts  (a .d.ts is implicitly ambient)
declare const __VERSION__: string;          // injected by the bundler
declare function gtag(cmd: string, ...args: unknown[]): void;

// type a non-code import
declare module "*.svg" {
  const url: string;
  export default url;
}

// usage in app code, fully typed, no import needed for globals:
console.log(__VERSION__);
gtag("event", "page_view");
__VERSION__: string; gtag callable; *.svg imports resolve
Challenge Exercise

Write a .d.ts that (a) declares a global process.env.NODE_ENV as "development" | "production" via module augmentation of NodeJS.ProcessEnv, and (b) declares a wildcard module for "*.json" exporting unknown. Confirm NODE_ENV narrows in a conditional.

Pitfall: Adding a value-producing statement to an ambient .d.ts — declare files must contain only declarations, never executable code. Conversely, writing declare const x = 5 with an initializer is an error: ambient declarations describe types, they never assign values.
R5. const Type Parameters TypeScript 5.x overdue · rep 0 → 1
A const type parameter (function f<const T>(x: T)) makes the inferred type as narrow as if the caller wrote as const — without forcing them to. Array literals infer as readonly tuples, strings as literals, instead of widening to string[] / string.
Key Pattern 
function tuple<const T extends readonly unknown[]>(t: T): T { return t; }

const a = tuple([1, 2, 3]);          // readonly [1, 2, 3]  (not number[])

function pluck<const K extends string>(keys: K[]) { return keys; }
const ks = pluck(["id", "name"]);    // ("id" | "name")[]  literals kept
a: readonly [1, 2, 3] ks: ("id" | "name")[]
Challenge Exercise

Write a defineRoutes<const T>(routes: T): T that, given ["/home", "/about"], infers the readonly tuple of literal paths. Then derive a RouteUnion = T[number] type from the result and confirm it is "/home" | "/about" rather than string.

Pitfall: Expecting const on the type parameter to make the runtime value immutable. It only affects inference — the array is still a mutable JS array at runtime. It also has no effect if the parameter type already widens it away (e.g. a constraint of string[] instead of readonly string[]), so pair it with readonly constraints.
R6. NoInfer Utility Type TypeScript 5.x overdue · rep 0 → 1
NoInfer<T> blocks a type parameter from being inferred at that position. You use it to pin inference to one argument and have the others merely checked against it — e.g. infer the union from a list, then validate the default against it instead of widening.
Key Pattern 
function createMachine<T extends string>(config: {
  states: T[];
  initial: NoInfer<T>;       // checked against T, not an inference source
}) { return config; }

createMachine({ states: ["on", "off"], initial: "on" });   // ok
createMachine({ states: ["on", "off"], initial: "idle" }); // Error
Argument of type '"idle"' is not assignable to '"on" | "off"'.
Challenge Exercise

Write pickDefault<T>(options: T[], fallback: NoInfer<T>): T so that T is inferred only from options and fallback must be one of them. Show that passing a fallback not present in options is a compile error rather than silently widening T.

Pitfall: Before NoInfer, people forced this with a second dummy type parameter or an intersection trick. Reaching for those now adds noise — but also remember NoInfer only removes an inference site; it cannot create a constraint. If no other argument supplies T, the parameter stays unknown.
1Type-Level Modeling
1. Tuple ↔ Object Type Conversions Type-Level
Converting between tuples and objects at the type level is the backbone of typed routers, ORMs, and form libraries: a tuple of keys becomes a record, a record becomes a tuple of entries, and back. Master it and most “magic” library types stop being magic.

A tuple indexed by [number] collapses to the union of its element types. Mapping over that union with a mapped type turns a tuple of keys into an object. Going the other way, keyof plus a mapped type reconstructs a tuple-like or record shape from an object.

The two workhorses: T[number] (tuple → union of elements) and { [K in U]: ... } (union → object). Combine with infer in template literals and you can parse and rebuild shapes the compiler tracks exactly.

1
T[number] turns a tuple type into a union of its members.
2
A mapped type { [K in Union]: V } turns that union into an object.
3
keyof O + indexing reverses an object back toward keys/values.
TypeScript · example 1 of 4 — tuple of keys to a record
type Keys = ["id", "name", "email"];

type RecordFromKeys<K extends readonly string[]> = {
  [P in K[number]]: string;
};

type User = RecordFromKeys<Keys>;
// { id: string; name: string; email: string }
User = { id: string; name: string; email: string }
TypeScript · example 2 of 4 — object to a union of entry tuples
type Entries<T> = {
  [K in keyof T]: [K, T[K]];
}[keyof T];

type O = { a: number; b: string };
type E = Entries<O>;
// ["a", number] | ["b", string]
E = ["a", number] | ["b", string]
TypeScript · example 3 of 4 — tuple of [key, value] pairs to object
type FromEntries<T extends readonly [PropertyKey, unknown][]> = {
  [P in T[number] as P[0]]: P[1];
};

type Obj = FromEntries<[["x", number], ["y", string]]>;
// { x: number; y: string }    — key remapping via `as P[0]`
Obj = { x: number; y: string }
TypeScript · example 4 of 4 — a typed Object.fromEntries wrapper
function fromEntries<const T extends readonly (readonly [PropertyKey, unknown])[]>(
  entries: T,
): { [P in T[number] as P[0]]: P[1] } {
  return Object.fromEntries(entries) as never;
}

const cfg = fromEntries([["port", 3000], ["host", "localhost"]] as const);
// cfg: { port: 3000; host: "localhost" }
console.log(cfg.port);   // 3000
3000
Common Mistakes:
❌ Using T[0] | T[1] | ... by hand to get the element union — brittle and breaks for tuples of unknown length.
✅ Use T[number], which yields the union of all elements regardless of length.
❌ Forgetting as const on the runtime tuple, so ["port", 3000] widens to (string | number)[] and the key/value literals are lost.
✅ Add as const (or a const type parameter) so element literals survive into the mapped type.
Key Rules:
  • 2026-07-04 (Day 31)
  • T[number] = union of a tuple’s element types; the key tool for tuple → union.
  • { [K in U]: V } builds an object from a union; add as ... to remap keys.
  • { [K in keyof T]: ... }[keyof T] projects an object into a union of computed members.
  • Pair these with as const / const type params so literal types are preserved.
Practice Exercises:
1Write Zip<K extends readonly string[], V extends readonly unknown[]> that pairs a tuple of keys with a tuple of values into an object type, assuming equal length.
2Write ValuesOf<T> that returns the union of an object’s value types using a mapped type and [keyof T].
3Build a typed groupBy<T, K extends keyof T> return type that maps each distinct value of T[K] to a T[] bucket.
Connections: Key remapping (as) ties back to Mapped Type Key Remapping; T[number] underpins Variadic Tuple Types. The typed fromEntries uses a const Type Parameter (R5) to keep literals.
2. Recursive DeepPartial & DeepReadonly Type-Level
Built-in Partial and Readonly only go one level deep, which is rarely what configuration, patch, and immutable-state code actually needs. The recursive versions are among the most copy-pasted utility types in real codebases — worth being able to write from memory.

A recursive mapped type applies itself to each property whose value is an object, and leaves primitives alone. The trick is the recursion guard: you must stop at non-objects (and usually at functions and arrays-of-primitives) or the type either over-applies or hits the depth limit.

DeepPartial makes every nested property optional; DeepReadonly makes every nested property readonly. Both special-case arrays so element types are recursed into without turning the array itself into an indexed object.

1
Map over keyof T with the modifier you want (? or readonly).
2
For each property, recurse if the value is an object, else keep it.
3
Special-case arrays so element types recurse but the array shape is preserved.
TypeScript · example 1 of 4 — built-ins are shallow
type Config = { db: { host: string; port: number }; debug: boolean };

type P = Partial<Config>;
// { db?: { host: string; port: number }; debug?: boolean }
//        ^ db is optional, but host/port are still REQUIRED
Partial only makes top-level keys optional
TypeScript · example 2 of 4 — DeepPartial
type DeepPartial<T> = T extends object
  ? { [K in keyof T]?: DeepPartial<T[K]> }
  : T;

type DP = DeepPartial<Config>;
// { db?: { host?: string; port?: number }; debug?: boolean }

const patch: DP = { db: { port: 5432 } };  // ok: everything optional
DP = { db?: { host?: string; port?: number }; debug?: boolean }
TypeScript · example 3 of 4 — DeepReadonly with array handling
type DeepReadonly<T> =
  T extends (infer E)[]      ? ReadonlyArray<DeepReadonly<E>>
  : T extends object         ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

type State = { user: { roles: string[] } };
type RO = DeepReadonly<State>;
// { readonly user: { readonly roles: readonly string[] } }
nested arrays become readonly too
TypeScript · example 4 of 4 — guarding functions so they are not mangled
type DeepPartialSafe<T> =
  T extends (...args: any[]) => any ? T        // leave functions intact
  : T extends object ? { [K in keyof T]?: DeepPartialSafe<T[K]> }
  : T;

type Svc = { fetch: (id: number) => void; opts: { retry: number } };
type DPS = DeepPartialSafe<Svc>;
// { fetch?: (id: number) => void; opts?: { retry?: number } }
function signature preserved, not turned into { }
Common Mistakes:
❌ Recursing without an array guard, so string[] becomes { [n: number]?: string; length?: number; ... } — every array method signature turned optional.
✅ Match T extends (infer E)[] first and map to ReadonlyArray<DeepReadonly<E>> (or DeepPartial<E>[]).
❌ Treating functions as plain objects, which strips them into {} and breaks call signatures downstream.
✅ Add a T extends (...args) => any ? T branch before the object branch to leave callables alone.
Key Rules:
  • Built-in Partial/Readonly are one level deep; recurse for nested shapes.
  • Order the conditional branches: array first, then function, then object, then base case.
  • Apply the modifier (? / readonly) in the mapped type, recurse on T[K].
  • Deeply recursive utilities can hit the instantiation-depth cap on very nested types — keep inputs bounded.
Practice Exercises:
1Write DeepRequired<T> that removes optionality recursively (the inverse of DeepPartial).
2Write DeepMutable<T> that strips readonly at every depth, including inside arrays.
3Extend DeepReadonly to also handle Map<K, V> and Set<V> by recursing into their type arguments and using the readonly variants.
Connections: These are the recursive cousins of Mapped Types and Utility Types; the array/function guards reuse the Conditional Types + infer patterns from infer Mastery. DeepPartial is the natural input type for a merge/patch helper.
3. UnionToIntersection & Distributive Tricks Type-Level
Turning a union into an intersection is the single most famous “type wizardry” one-liner. It powers merging plugin types, deriving a combined config from many modules, and extracting the last member of a union — and it teaches you how conditional-type distribution and inference positions really work.

The trick exploits contravariance: when a naked type parameter is distributed across a union and each member is placed in a function parameter position, then infer-ed back out, TypeScript must satisfy all the function signatures at once — which forces an intersection rather than a union.

The same family of tricks gives you LastInUnion (infer from the most-recently-added overload) and from there UnionToTuple. They’re fragile by nature (order isn’t guaranteed for tuple conversion) — use them for merging, not for relying on element order.

1
Distribute the union into function-parameter positions: U extends any ? (x: U) => void : never.
2
infer the parameter back out of the resulting intersection of functions.
3
Contravariance forces the inferred type to be the intersection of the members.
TypeScript · example 1 of 4 — the canonical one-liner
type UnionToIntersection<U> =
  (U extends any ? (x: U) => void : never) extends
   (x: infer I) => void ? I : never;

type R = UnionToIntersection<{ a: 1 } | { b: 2 }>;
// { a: 1 } & { b: 2 }
R = { a: 1 } & { b: 2 }
TypeScript · example 2 of 4 — why: distribution + contravariance
// Step A: distribute over the union (each member in a param position)
type Fns = ((x: { a: 1 }) => void) | ((x: { b: 2 }) => void);

// Step B: a value must satisfy BOTH call signatures, so inferring the
// single parameter type that works for both yields their intersection.
//   (x: infer I) => void   =>   I = { a: 1 } & { b: 2 }
parameter positions are contravariant => intersection
TypeScript · example 3 of 4 — merging many module configs
type AuthCfg  = { auth: { secret: string } };
type DbCfg    = { db: { url: string } };
type CacheCfg = { cache: { ttl: number } };

type AppConfig = UnionToIntersection<AuthCfg | DbCfg | CacheCfg>;
// { auth: {...} } & { db: {...} } & { cache: {...} }
//  => one object with all three sections, fully typed
AppConfig has auth, db, and cache keys
TypeScript · example 4 of 4 — LastInUnion (advanced, order-sensitive)
type LastInUnion<U> =
  UnionToIntersection<U extends any ? (x: U) => void : never> extends
    (x: infer L) => void ? L : never;

type L = LastInUnion<1 | 2 | 3>;   // 3  (compiler-dependent ordering)
// Building blocks for UnionToTuple — but treat order as unspecified.
L = 3 (do not rely on this ordering as a contract)
Common Mistakes:
❌ Putting the union in a covariant (return) position and expecting an intersection — () => U keeps it a union. The intersection only emerges from a contravariant (parameter) position.
✅ Place each distributed member as a function parameter ((x: U) => void) so contravariance forces the intersection.
❌ Relying on UnionToTuple ordering as a stable contract — it depends on internal compiler overload resolution and can change between versions.
✅ Use it only where order is irrelevant (merging), or sort/normalize explicitly if you truly need an order.
Key Rules:
  • Distribution: a naked type param in a conditional spreads across each union member.
  • Parameter positions are contravariant — inferring from them yields an intersection.
  • UnionToIntersection is the merge primitive; LastInUnion/UnionToTuple build on it.
  • Treat union-to-tuple ordering as undefined behavior; never depend on it.
Practice Exercises:
1Use UnionToIntersection to write MergeAll<T extends object[]> that intersects every object type in a tuple into one.
2Write IsUnion<T> that resolves to true when T is a union and false otherwise (hint: compare a distributed form against the whole).
3Build UnionToTuple<U> on top of LastInUnion and explain in a comment why its ordering must not be treated as a guarantee.
Connections: This leans on conditional-type distribution, infer, and the Covariance & Contravariance rules (reviewed earlier this cycle). The merge use case is the type-level analogue of combining plugin/module shapes in Type-Safe Plugin Systems.
2Runtime Safety Patterns
4. Type-Safe localStorage / Storage Wrappers Runtime Safety
localStorage is a string-only, untyped key–value bag at the most dangerous boundary there is: data persisted from a previous version of your app. A typed wrapper turns “trust me” reads into validated, schema-keyed access so stale or corrupt entries fail loudly instead of crashing deep in your UI.

The native API returns string | null and accepts any string key, so nothing the compiler knows survives a round-trip. The fix is a wrapper parameterized by a schema map: an object type whose keys are the allowed storage keys and whose values are the expected shapes.

On write you JSON.stringify; on read you JSON.parse and validate — ideally with Zod, since the bytes came from outside your program’s current type system. A get that returns T | null and validates before returning is the whole pattern.

1
Define a schema map: key → expected type (and a Zod schema per key).
2
Constrain get/set to K extends keyof Schema so only known keys compile.
3
On read, parse JSON and validate; return null on miss or invalid data.
TypeScript · example 1 of 4 — the raw API loses all types
localStorage.setItem("user", JSON.stringify({ id: 1 }));

const raw = localStorage.getItem("user");   // string | null
const user = JSON.parse(raw!);              // any  — danger
user.naem;                                  // no error, typo survives
JSON.parse returns `any` — the boundary is untyped
TypeScript · example 2 of 4 — a key-constrained wrapper
interface StoreSchema {
  user: { id: number; name: string };
  theme: "dark" | "light";
}

function getItem<K extends keyof StoreSchema>(key: K): StoreSchema[K] | null {
  const raw = localStorage.getItem(key);
  return raw === null ? null : (JSON.parse(raw) as StoreSchema[K]);
}
function setItem<K extends keyof StoreSchema>(key: K, value: StoreSchema[K]) {
  localStorage.setItem(key, JSON.stringify(value));
}

const theme = getItem("theme");   // "dark" | "light" | null
// getItem("unknown");            // Error: not a known key
theme: "dark" | "light" | null; bad keys rejected at compile time
TypeScript · example 3 of 4 — validate on read with Zod
import { z } from "zod";

const schemas = {
  user:  z.object({ id: z.number(), name: z.string() }),
  theme: z.enum(["dark", "light"]),
} as const;

function read<K extends keyof typeof schemas>(key: K) {
  const raw = localStorage.getItem(key);
  if (raw === null) return null;
  const parsed = schemas[key].safeParse(JSON.parse(raw));
  return parsed.success ? (parsed.data as z.infer<typeof schemas[K]>) : null;
}
// stale/corrupt data from an old app version => null, not a crash
read("user"): { id: number; name: string } | null (validated)
TypeScript · example 4 of 4 — guarding JSON.parse from throwing
function safeParse<T>(raw: string | null): T | null {
  if (raw === null) return null;
  try { return JSON.parse(raw) as T; }
  catch { return null; }              // malformed JSON => null, no throw
}

const data = safeParse<{ count: number }>(localStorage.getItem("stats"));
if (data) console.log(data.count);   // narrowed to the object
data: { count: number } | null
Common Mistakes:
❌ Casting JSON.parse(raw) as User and trusting it. The cast is a lie at runtime — old or hand-edited storage can hold any shape, and the bug surfaces far from the read.
✅ Validate parsed data with Zod (or a type guard) and return null on failure, so corruption fails at the boundary.
❌ Calling JSON.parse directly on possibly-malformed strings — it throws a SyntaxError that crashes the calling component.
✅ Wrap parsing in try/catch (or use a result-returning helper) so malformed input degrades to null.
Key Rules:
  • localStorage is string-only and untyped; never trust the shape of what comes back.
  • Constrain keys with K extends keyof Schema so typos and unknown keys are compile errors.
  • Validate on read — persisted data is external input, possibly from an older app version.
  • Guard JSON.parse against throwing; return null for missing and malformed entries.
Practice Exercises:
1Build a createStorage<S>(schemas) factory returning { get, set, remove } all keyed to keyof S, with Zod validation on get.
2Add a version field to stored values and have get return null (triggering a re-fetch) when the stored version is older than the current one.
3Make the wrapper work with both localStorage and sessionStorage by taking a Storage parameter, and confirm the types are identical for both.
Connections: Validate-at-the-boundary is the same discipline as Zod & Runtime Validation and Type-Safe Configuration; the schema-map keying reuses the keyof & Indexed Access machinery. The safeParse result shape echoes Type-Safe Error Boundaries.
5. Exhaustive Reducers with Discriminated Actions Runtime Safety
A reducer is a giant switch over action types — exactly the place a forgotten case ships a silent bug. Modeling actions as a discriminated union and pairing the switch with a never exhaustiveness check turns “I added an action but forgot to handle it” into a compile error.

Model the action as a discriminated union on a literal type field. In the reducer, switch (action.type) narrows each case to that specific action, so its payload is exactly typed — no casting. The closing move is a default that assigns the action to a never-typed variable: if any case is unhandled, the union isn’t fully narrowed, the residual type isn’t never, and assignment fails to compile.

Because the state transitions are typed per action, you also get autocomplete on action.payload inside each branch, and adding a new action variant immediately lights up the reducer as incomplete.

1
Define actions as a discriminated union keyed by type.
2
switch (action.type) — each case narrows to one action with its payload.
3
In default, assign to const _exhaustive: never = action to force coverage.
TypeScript · example 1 of 4 — discriminated action union
type Action =
  | { type: "increment"; by: number }
  | { type: "reset" }
  | { type: "setLabel"; label: string };

interface State { count: number; label: string }
Action is a union discriminated on `type`
TypeScript · example 2 of 4 — reducer with narrowing per case
function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "increment":           // action: { type: "increment"; by: number }
      return { ...state, count: state.count + action.by };
    case "reset":               // action: { type: "reset" }
      return { ...state, count: 0 };
    case "setLabel":            // action: { type: "setLabel"; label: string }
      return { ...state, label: action.label };
    default: {
      const _exhaustive: never = action;   // compile error if a case is missing
      return _exhaustive;
    }
  }
}
each case is narrowed; default enforces exhaustiveness
TypeScript · example 3 of 4 — adding a case you forgot to handle
type Action2 = Action | { type: "decrement"; by: number };

function reducer2(state: State, action: Action2): State {
  switch (action.type) {
    case "increment": return { ...state, count: state.count + action.by };
    case "reset":     return { ...state, count: 0 };
    case "setLabel":  return { ...state, label: action.label };
    // forgot "decrement"!
    default: {
      const _exhaustive: never = action;   // Error here points at the gap
      return _exhaustive;
    }
  }
}
Type '{ type: "decrement"; by: number }' is not assignable to type 'never'.
TypeScript · example 4 of 4 — a reusable assertNever helper
function assertNever(x: never): never {
  throw new Error(`Unhandled case: ${JSON.stringify(x)}`);
}

function handle(action: Action): string {
  switch (action.type) {
    case "increment": return `+${action.by}`;
    case "reset":     return "reset";
    case "setLabel":  return action.label;
    default:          return assertNever(action); // compile + runtime guard
  }
}
console.log(handle({ type: "setLabel", label: "hi" })); // "hi"
hi
Common Mistakes:
❌ A default that just returns state. It silently swallows unknown actions, so a forgotten case is a no-op bug, not a compile error.
✅ Assign the action to never (or call assertNever) in default so any unhandled variant fails to type-check.
❌ Using a non-literal discriminant (e.g. type: string) so the union never narrows and every action.payload needs a cast.
✅ Use string-literal type tags (a discriminated union) so each case narrows the payload automatically.
Key Rules:
  • Model actions as a discriminated union on a literal type tag for free per-case narrowing.
  • Close every reducer switch with a never assignment or assertNever in default.
  • The compile error from the never check pinpoints exactly which variant is unhandled.
  • assertNever doubles as a runtime guard if an unexpected action slips through untyped boundaries.
Practice Exercises:
1Write a formReducer over actions { type: "setField"; name; value }, { type: "submit" }, { type: "reset" } with an assertNever default; then add a fourth action and observe the error.
2Make the reducer generic over a FieldName union so setField.name is constrained to real field keys.
3Combine this with the Type-Safe localStorage wrapper (Concept 4) to persist reducer state and rehydrate it with validation on load.
Connections: This is the applied form of never & Exhaustiveness and Discriminated Unions in Practice, and the typing core of Redux Toolkit / Zustand (R1) actions. The assertNever boundary mirrors the validate-or-fail stance of Concept 4.
