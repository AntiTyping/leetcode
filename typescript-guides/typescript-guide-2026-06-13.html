R1. keyof & Indexed Access Types Advanced Type System rep 2 · 4 days overdue
keyof T produces a union of all property names of T as string literal types. T[K] accesses the type of property K in T, enabling generic property lookups that stay in sync with the source type.
Key Pattern 
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// keyof produces a union of literal key types
type UserKeys = keyof User;  // "id" | "name" | "email" | "age"

// Indexed access retrieves the type at a key
type NameType = User['name'];       // string
type IdOrAge = User['id' | 'age'];  // number

// Type-safe property access function
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user: User = { id: 1, name: 'Alice', email: 'a@b.com', age: 30 };
const name = getProperty(user, 'name');  // string
const age = getProperty(user, 'age');    // number
// getProperty(user, 'typo');            // Error: "typo" not in keyof User

// Nested indexed access
interface Config {
  db: { host: string; port: number };
  cache: { ttl: number; strategy: 'lru' | 'fifo' };
}
type DbHost = Config['db']['host'];           // string
type CacheStrategy = Config['cache']['strategy']; // 'lru' | 'fifo'
UserKeys = "id" | "name" | "email" | "age" NameType = string name: string age: number DbHost = string
Challenge Exercise

Write a generic pick function that takes an object and an array of keys, and returns a new object with only those keys. The return type must be Pick<T, K>. Then write a pluck function that extracts a single property from an array of objects: pluck(users, 'name') returns string[].

#1 Pitfall: Using keyof on a union type gives you the intersection of keys (only keys common to all members), not the union. keyof (A | B) is keyof A & keyof B. Use a distributive conditional type if you need all keys: T extends any ? keyof T : never.
R2. Type Predicates & Assertion Functions Advanced Type System rep 2 · 4 days overdue
Type predicates (x is T) narrow a parameter's type inside conditional branches. Assertion functions (asserts x is T) narrow types after the call — if the assertion fails, it throws. Both are essential for converting runtime checks into compile-time type narrowing.
Key Pattern 
// Type predicate — narrows inside if/else branches
function isString(value: unknown): value is string {
  return typeof value === 'string';
}

function process(input: unknown) {
  if (isString(input)) {
    console.log(input.toUpperCase());  // input: string
  }
}

// Discriminated union guard
interface Dog { kind: 'dog'; bark(): void; }
interface Cat { kind: 'cat'; meow(): void; }
type Animal = Dog | Cat;

function isDog(animal: Animal): animal is Dog {
  return animal.kind === 'dog';
}

// Assertion function — narrows AFTER the call
function assertIsNumber(value: unknown): asserts value is number {
  if (typeof value !== 'number') {
    throw new Error(`Expected number, got ${typeof value}`);
  }
}

function calculate(input: unknown) {
  assertIsNumber(input);
  // After this line, input is narrowed to number
  return input * 2;  // no type error
}

// assertNonNull — very common pattern
function assertDefined<T>(
  value: T | null | undefined,
  msg?: string
): asserts value is T {
  if (value == null) throw new Error(msg ?? 'Value is null/undefined');
}
Challenge Exercise

Build a isArrayOf<T> higher-order type guard: isArrayOf(isString) returns a function with signature (value: unknown) => value is string[]. Then use it to guard an unknown input as number[] using isArrayOf(isNumber).

#1 Pitfall: Type predicates lie if the runtime check is wrong. TypeScript trusts the return type annotation unconditionally. If isString returns true for a number, downstream code will treat it as a string with no runtime error until you call a string method. Always ensure the runtime logic matches the type predicate.
R3. Generic Constraints & Defaults Functions & Patterns rep 2 · due today
<T extends Constraint> limits what types can be passed as T, enabling you to access properties guaranteed by the constraint. Default type parameters (<T = string>) provide a fallback when the caller does not specify a type argument.
Key Pattern 
// Constraint: T must have a 'length' property
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

longest('abc', 'de');        // works: string has .length
longest([1, 2], [1, 2, 3]); // works: array has .length
// longest(10, 20);          // Error: number has no .length

// Multiple constraints with intersection
function merge<T extends object, U extends object>(a: T, b: U): T & U {
  return { ...a, ...b };
}

// Default type parameters
interface ApiResponse<T = unknown, E = Error> {
  data: T;
  error: E | null;
  status: number;
}

// Without specifying types — uses defaults
const res1: ApiResponse = { data: 'hello', error: null, status: 200 };
// data: unknown, error: Error | null

// With explicit types
const res2: ApiResponse<User[], string> = {
  data: [{ id: 1, name: 'Alice' }],
  error: null,
  status: 200,
};
// data: User[], error: string | null
longest('abc', 'de') : string merge({ a: 1 }, { b: 2 }) : { a: number } & { b: number } res1.data : unknown res2.data : User[]

R4. Branded/Opaque Types Functions & Patterns rep 2 · due today
Branded types use an invisible phantom property to make structurally identical types nominally distinct. type UserId = string & { readonly __brand: unique symbol } prevents accidentally passing a PostId where a UserId is expected, even though both are strings at runtime.
Key Pattern 
// Generic brand utility
declare const __brand: unique symbol;
type Brand<T, B extends string> = T & { readonly [__brand]: B };

// Create distinct ID types — all strings at runtime
type UserId = Brand<string, 'UserId'>;
type PostId = Brand<string, 'PostId'>;
type OrderId = Brand<string, 'OrderId'>;

// Smart constructors validate and brand
function toUserId(id: string): UserId {
  if (!id.startsWith('usr_')) throw new Error('Invalid UserId');
  return id as UserId;
}

function toPostId(id: string): PostId {
  if (!id.startsWith('post_')) throw new Error('Invalid PostId');
  return id as PostId;
}

// Type safety: cannot mix up IDs
function getUser(id: UserId): Promise<User> { /* ... */ }
function getPost(id: PostId): Promise<Post> { /* ... */ }

const uid = toUserId('usr_123');
const pid = toPostId('post_456');

getUser(uid);  // OK
getPost(pid);  // OK
// getUser(pid); // Error: PostId not assignable to UserId
// getUser('usr_123'); // Error: string not assignable to UserId

R5. Redux Toolkit TypeScript Patterns State Management rep 0 · due today
createSlice infers action types from reducer names. PayloadAction<T> types the payload. RootState and AppDispatch, derived from the store via ReturnType and typeof, flow through typed hooks (useAppSelector, useAppDispatch) to components for end-to-end type safety.
Key Pattern 
import { createSlice, PayloadAction, configureStore } from '@reduxjs/toolkit';

interface TodoState {
  items: { id: string; text: string; done: boolean }[];
  filter: 'all' | 'active' | 'done';
}

const todoSlice = createSlice({
  name: 'todos',
  initialState: { items: [], filter: 'all' } as TodoState,
  reducers: {
    addTodo(state, action: PayloadAction<{ id: string; text: string }>) {
      state.items.push({ ...action.payload, done: false });
    },
    toggleTodo(state, action: PayloadAction<string>) {
      const todo = state.items.find(t => t.id === action.payload);
      if (todo) todo.done = !todo.done;
    },
    setFilter(state, action: PayloadAction<TodoState['filter']>) {
      state.filter = action.payload;
    },
  },
});

const store = configureStore({ reducer: { todos: todoSlice.reducer } });
type RootState = ReturnType<typeof store.getState>;
type AppDispatch = typeof store.dispatch;

// Typed hooks
import { useDispatch, useSelector, TypedUseSelectorHook } from 'react-redux';
export const useAppDispatch = () => useDispatch<AppDispatch>();
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;
todoSlice.actions.addTodo: ActionCreatorWithPayload<{ id: string; text: string }> RootState = { todos: TodoState } useAppSelector(s => s.todos.filter) : "all" | "active" | "done"

R6. React Hooks Typing React + TS rep 0 · 4 days overdue
useState<T> types the state variable and setter. useRef<T>(null) returns RefObject<T> (read-only .current), while useRef<T>(initialValue) returns MutableRefObject<T>. useReducer is typed via the reducer function signature. Custom hooks should explicitly annotate return types, especially when returning tuples.
Key Pattern 
import { useState, useRef, useReducer, useCallback } from 'react';

// useState — explicit generic for complex types
const [user, setUser] = useState<User | null>(null);
//    ^? User | null

// useRef — DOM ref (read-only .current)
const inputRef = useRef<HTMLInputElement>(null);
//    ^? RefObject<HTMLInputElement>
// inputRef.current?.focus()

// useRef — mutable instance variable
const timerRef = useRef<number>(0);
//    ^? MutableRefObject<number>
timerRef.current = window.setTimeout(() => {}, 1000);

// useReducer — types from reducer signature
type State = { count: number; error: string | null };
type Action =
  | { type: 'increment'; payload: number }
  | { type: 'reset' }
  | { type: 'error'; payload: string };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'increment': return { ...state, count: state.count + action.payload };
    case 'reset': return { count: 0, error: null };
    case 'error': return { ...state, error: action.payload };
  }
}

const [state, dispatch] = useReducer(reducer, { count: 0, error: null });
dispatch({ type: 'increment', payload: 5 }); // OK
// dispatch({ type: 'increment' });           // Error: missing payload

// Custom hook — explicit return type for tuple
function useToggle(initial = false): [boolean, () => void] {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue(v => !v), []);
  return [value, toggle];
}
user: User | null inputRef: RefObject<HTMLInputElement> timerRef: MutableRefObject<number> state: { count: number; error: string | null } useToggle(): [boolean, () => void]

#1. const Type Parameters const_type_parameters
Before TypeScript 5.0, generic functions always widened inferred types — ['home', 'about'] became string[]. The const modifier on type parameters preserves literal and tuple types without requiring callers to write as const, making configuration objects, route definitions, and builder APIs dramatically more type-safe.

Adding const before a type parameter (<const T>) tells TypeScript to infer the narrowest possible type for T. This means string literals stay as literals, arrays become readonly tuples, and object properties retain their literal types. It is equivalent to the caller writing as const, but applied automatically.

The const modifier works with constraints: <const T extends readonly string[]> infers a readonly tuple of string literals. The readonly constraint is important because const inference produces readonly types.

Example 1 — Before vs after const type parameters
// BEFORE TS 5.0 — always widened
function getRoutes<T extends readonly string[]>(paths: T): T {
  return paths;
}

const routes = getRoutes(['home', 'about', 'contact']);
//    ^? string[]  — literals are lost!

// Caller had to add 'as const' manually
const routes2 = getRoutes(['home', 'about', 'contact'] as const);
//    ^? readonly ["home", "about", "contact"]  — works, but awkward

// AFTER TS 5.0 — const type parameter
function getRoutes2<const T extends readonly string[]>(paths: T): T {
  return paths;
}

const routes3 = getRoutes2(['home', 'about', 'contact']);
//    ^? readonly ["home", "about", "contact"]  — literals preserved!
routes: string[] routes2: readonly ["home", "about", "contact"] routes3: readonly ["home", "about", "contact"]
Example 2 — Configuration objects with const T
// Config builder that preserves exact shape
function defineConfig<const T extends {
  routes: Record<string, { path: string; method: 'GET' | 'POST' | 'PUT' | 'DELETE' }>;
  middleware: readonly string[];
}>(config: T): T {
  return config;
}

const appConfig = defineConfig({
  routes: {
    users: { path: '/api/users', method: 'GET' },
    createUser: { path: '/api/users', method: 'POST' },
  },
  middleware: ['auth', 'logging', 'cors'],
});

// Without const: method would be 'GET' | 'POST' | 'PUT' | 'DELETE'
// With const: method is exactly 'GET' or 'POST' per route
type UsersMethod = typeof appConfig.routes.users.method;
//   ^? "GET"  — not the full union!

type MiddlewareList = typeof appConfig.middleware;
//   ^? readonly ["auth", "logging", "cors"]  — exact tuple
UsersMethod = "GET" MiddlewareList = readonly ["auth", "logging", "cors"]
Example 3 — Enum-like constants from const type parameters
// Create a type-safe enum alternative
function createEnum<const T extends readonly string[]>(
  ...values: T
): { [K in T[number]]: K } & { values: T } {
  const result = {} as any;
  for (const v of values) result[v] = v;
  result.values = values;
  return result;
}

const Status = createEnum('active', 'inactive', 'pending', 'archived');

type StatusType = typeof Status.values[number];
//   ^? "active" | "inactive" | "pending" | "archived"

const s: StatusType = Status.active;  // "active"
// const bad: StatusType = 'unknown'; // Error!

// Compare with traditional enum — const T gives same safety with less ceremony
// enum Status { Active = 'active', ... }
Status.active = "active" Status.values = readonly ["active", "inactive", "pending", "archived"] StatusType = "active" | "inactive" | "pending" | "archived"
Example 4 — Type-safe builder pattern with const T
// Form builder that tracks field names at the type level
class FormBuilder<const Fields extends readonly string[] = readonly []> {
  constructor(private fields: Fields) {}

  addField<const F extends string>(name: F): FormBuilder<readonly [...Fields, F]> {
    return new FormBuilder([...this.fields, name] as any);
  }

  build(): { [K in Fields[number]]: string } {
    const form = {} as any;
    for (const f of this.fields) form[f] = '';
    return form;
  }
}

const form = new FormBuilder([] as const)
  .addField('username')
  .addField('email')
  .addField('password')
  .build();

// form is typed as { username: string; email: string; password: string }
form.username;  // OK
form.email;     // OK
// form.typo;   // Error: Property 'typo' does not exist
form: { username: string; email: string; password: string }
Example 5 — Common mistake: forgetting readonly constraint
// WRONG: const T infers readonly, but constraint expects mutable
function badRoutes<const T extends string[]>(paths: T): T {
  return paths;
}
// Error: readonly ["home", "about"] is not assignable to string[]
// The inferred readonly tuple does not satisfy the mutable array constraint

// CORRECT: constraint must accept readonly
function goodRoutes<const T extends readonly string[]>(paths: T): T {
  return paths;
}

const r = goodRoutes(['home', 'about']);
//    ^? readonly ["home", "about"]  — works!

// Also works with object constraints
function defineOptions<const T extends Readonly<{
  name: string;
  version: string;
  features: readonly string[];
}>>(opts: T): T {
  return opts;
}

const opts = defineOptions({
  name: 'myApp',
  version: '1.0.0',
  features: ['dark-mode', 'i18n'],
});
// opts.name is "myApp", not string
r: readonly ["home", "about"] opts.name: "myApp" opts.features: readonly ["dark-mode", "i18n"]
Common Mistakes
Using <const T extends string[]> — const infers readonly tuples, which are not assignable to mutable string[].
Always use <const T extends readonly string[]> or Readonly<...> in the constraint.
Applying const to type parameters that already receive widened types from other sources (e.g., state from external APIs) — it has no effect if the value is already widened.
Use const only on type parameters that receive literal values directly at the call site.
Key Rules for const Type Parameters
  • 2026-07-04 (Day 31)
  • <const T> infers the narrowest possible type — equivalent to caller writing as const
  • Constraints must accept readonly types since const produces them
  • Works with arrays (readonly tuples), objects (literal property types), and primitives (literal types)
  • Best for: config objects, route definitions, builder patterns, enum-like values
  • Available since TypeScript 5.0

#2. NoInfer Utility Type noinfer_utility
When TypeScript infers a generic type from multiple positions, it merges all candidates — which often widens the type beyond what you intended. NoInfer<T> (TS 5.4) marks a position as "do not infer from here," giving you precise control over where type inference draws its conclusions.

NoInfer<T> is a built-in utility type that wraps T to prevent TypeScript from using that position for type inference. The wrapped position still receives the inferred type and is checked against it — it just does not contribute to inference.

This solves a class of problems where you want inference to come from one specific argument (e.g., the initial state) while other arguments (e.g., valid states, default values) should be checked against the inferred type rather than influencing it.

Example 1 — The problem NoInfer solves
// WITHOUT NoInfer — inference comes from BOTH arguments
function createFSM<S extends string>(
  initial: S,
  states: S[]
) { /* ... */ }

// We want S to be "idle" | "loading" | "error"
// But inference merges both arguments:
createFSM('idle', ['idle', 'loading', 'error', 'stale']);
// S inferred as "idle" | "loading" | "error" | "stale"
// No error on 'stale' — even though we wanted 'idle' to drive inference!

// WITH NoInfer — inference only comes from 'initial'
function createFSM2<S extends string>(
  initial: S,
  states: NoInfer<S>[]
) { /* ... */ }

createFSM2('idle', ['idle', 'loading', 'error']);
// S inferred as "idle"
// Error: "loading" and "error" are not assignable to "idle"!

// The fix: use a union type explicitly
createFSM2<'idle' | 'loading' | 'error'>('idle', ['idle', 'loading', 'error']);
// OK — all states checked against the explicit union
Argument of type '"loading"' is not assignable to parameter of type '"idle"'. Argument of type '"error"' is not assignable to parameter of type '"idle"'.
Example 2 — Default values without widening
// WITHOUT NoInfer — default value widens the type
function createSignal<T>(initial: T, defaultValue: T): {
  get(): T;
  set(v: T): void;
} {
  let value = initial;
  return {
    get: () => value,
    set: (v) => { value = v; },
  };
}

// We want T to be 'red' | 'blue', but 'green' in default widens it
const signal = createSignal<'red' | 'blue'>('red', 'green');
// Without explicit annotation, T = 'red' | 'blue' | 'green'

// WITH NoInfer — default is checked, not inferred from
function createSignal2<T>(initial: T, defaultValue: NoInfer<T>): {
  get(): T;
  set(v: T): void;
} {
  let value = initial;
  return {
    get: () => value,
    set: (v) => { value = v; },
  };
}

const signal2 = createSignal2('red' as 'red' | 'blue', 'green');
// Error: 'green' is not assignable to 'red' | 'blue'

const signal3 = createSignal2('red' as 'red' | 'blue', 'blue');
// OK — 'blue' is in the inferred union
Argument of type '"green"' is not assignable to parameter of type '"red" | "blue"'.
Example 3 — Type-safe event emitter with NoInfer
// Event system where handler type is inferred from the event map, not from
// the handler argument itself
interface EventMap {
  click: { x: number; y: number };
  keydown: { key: string; code: string };
  resize: { width: number; height: number };
}

function on<E extends keyof EventMap>(
  event: E,
  handler: (data: NoInfer<EventMap[E]>) => void
): void { /* ... */ }

// Handler type is inferred from EventMap, not from the callback parameter
on('click', (data) => {
  console.log(data.x, data.y);     // OK — data is { x: number; y: number }
  // console.log(data.key);         // Error: 'key' does not exist
});

on('keydown', (data) => {
  console.log(data.key, data.code); // OK — data is { key: string; code: string }
});
on('click', handler): void — data inferred as { x: number; y: number } on('keydown', handler): void — data inferred as { key: string; code: string }
Example 4 — Constrained selection from a set
// Select items from a typed collection — selection must be a subset
function selectFrom<T extends string>(
  all: readonly T[],
  selected: NoInfer<T>[]
): NoInfer<T>[] {
  return selected.filter(s => all.includes(s));
}

const colors = ['red', 'green', 'blue', 'yellow'] as const;

selectFrom(colors, ['red', 'blue']);     // OK
selectFrom(colors, ['red', 'purple']);   // Error: 'purple' not in the set

// Without NoInfer, 'purple' would widen T to include it — no error!

// Real-world: form field selection
function createForm<const F extends readonly string[]>(
  fields: F,
  required: NoInfer<F[number]>[]
) { /* ... */ }

createForm(
  ['name', 'email', 'phone'] as const,
  ['name', 'email']  // OK — subset of fields
);

createForm(
  ['name', 'email', 'phone'] as const,
  ['name', 'address']  // Error: 'address' not in fields
);

Common Mistakes
Wrapping all parameters in NoInfer — if no position contributes to inference, TypeScript cannot infer T at all and it falls back to the constraint or unknown.
Always leave at least one parameter position unwrapped so TypeScript has a source for inference.
Expecting NoInfer to change the runtime type — it is purely a compile-time directive with no runtime effect.
NoInfer<T> resolves to T at the type level. It only affects which positions contribute to type inference.

#3. Using Declarations & Explicit Resource Management using_declarations
The using keyword (TS 5.2) brings deterministic cleanup to JavaScript/TypeScript, replacing fragile try/finally patterns. When a using variable goes out of scope, its [Symbol.dispose]() method is called automatically — like RAII in C++ or with in Python. This eliminates resource leaks for database connections, file handles, locks, and temporary files.

using declares a variable whose resource is automatically disposed when the enclosing block exits. The resource must implement Disposable (with a [Symbol.dispose]() method) or AsyncDisposable (with [Symbol.asyncDispose]() for async cleanup, used with await using).

DisposableStack and AsyncDisposableStack group multiple disposables so they are all cleaned up together, in reverse order of registration.

Example 1 — Basic using with Symbol.dispose
// A resource that implements Disposable
class TempFile implements Disposable {
  path: string;

  constructor(name: string) {
    this.path = `/tmp/${name}-${Date.now()}`;
    console.log(`Created: ${this.path}`);
    // Create the temp file on disk...
  }

  write(data: string): void {
    console.log(`Writing to ${this.path}`);
  }

  [Symbol.dispose](): void {
    console.log(`Deleted: ${this.path}`);
    // Delete the temp file from disk...
  }
}

function processData() {
  using tmpFile = new TempFile('data');
  tmpFile.write('some data');
  // ... do work with the temp file ...

  // When this block exits (normally or via exception),
  // tmpFile[Symbol.dispose]() is called automatically
}

  }

Example 4 — Mutex/Lock pattern with using
// A mutex that uses Disposable for automatic unlock
class Mutex {
  private locked = false;
  private waiters: (() => void)[] = [];

  async acquire(): Promise<Disposable> {
    while (this.locked) {
      await new Promise<void>(resolve => this.waiters.push(resolve));
    }
    this.locked = true;

    return {
      [Symbol.dispose]: () => {
        this.locked = false;
        const next = this.waiters.shift();
        if (next) next();
      },
    };
  }
}

const mutex = new Mutex();

async function criticalSection() {
  using lock = await mutex.acquire();
  // Mutex is held for this entire block
  console.log('Doing exclusive work...');
  await expensiveOperation();
  // Lock is automatically released when block exits
  // Even if expensiveOperation() throws!
}

// Multiple concurrent calls — only one enters the critical section at a time
Promise.all([
  criticalSection(),
  criticalSection(),
  criticalSection(),
]);

Common Mistakes
Using using with a value that does not implement Disposable or AsyncDisposable — TypeScript will error at compile time, but it is easy to forget the [Symbol.dispose] method.
Always implement the Disposable or AsyncDisposable interface explicitly so TypeScript enforces the contract.
Using using (synchronous) with an async disposable resource — the async cleanup will not be awaited.
Use await using for any resource with [Symbol.asyncDispose]. Use plain using only for synchronous [Symbol.dispose].
Key Rules for Using Declarations
  • using x = ... — calls x[Symbol.dispose]() when scope exits
  • await using x = ... — calls await x[Symbol.asyncDispose]() when scope exits
  • Resources are disposed in reverse order of declaration (LIFO)
  • Disposal happens even if an exception is thrown
  • DisposableStack groups multiple resources for coordinated cleanup
  • Requires TypeScript 5.2+ and "lib": ["ESNext.Disposable"] or "lib": ["ES2024"]

#4. Decorator Metadata decorator_metadata
TypeScript 5.2 introduced Symbol.metadata, which lets decorators attach metadata to classes without external libraries like reflect-metadata. This enables first-class support for dependency injection containers, ORM column mappings, validation rules, and API route registration — all with runtime introspection.

The TC39 decorator metadata proposal gives every decorator access to a context.metadata object. Multiple decorators on the same class share the same metadata object, and the metadata is accessible at runtime via MyClass[Symbol.metadata].

This replaces the reflect-metadata polyfill and Reflect.defineMetadata/Reflect.getMetadata pattern that was common in Angular, NestJS, and TypeORM. The new approach is standards-based and does not require experimental decorators.

  return function (

// Simple DI container using decorator metadata
const INJECT_KEY = Symbol('inject');

function Injectable(
  target: new (...args: any[]) => any,
  context: ClassDecoratorContext
) {
  context.metadata[INJECT_KEY] = {
    name: context.name,
    injectable: true,
  };
}

function Inject(serviceKey: string) {
  return function (
    target: undefined,
    context: ClassFieldDecoratorContext
  ) {
    const deps = (context.metadata.dependencies ??= []) as
      { field: string; serviceKey: string }[];
    deps.push({ field: String(context.name), serviceKey });
  };
}

@Injectable
class UserService {
  getUser(id: string) {
    return { id, name: 'Alice' };
  }

// The DI container reads metadata to wire dependencies
function createContainer(classes: (new (...args: any[]) => any)[]) {
  const instances = new Map<string, any>();

  for (const cls of classes) {
    const meta = cls[Symbol.metadata];
    const instance = new cls();
    const name = (meta?.[INJECT_KEY] as any)?.name ?? cls.name;
    instances.set(name, instance);
  }

  // Wire dependencies
  for (const cls of classes) {
    const meta = cls[Symbol.metadata];
    const deps = (meta?.dependencies ?? []) as
      { field: string; serviceKey: string }[];
    const name = (meta?.[INJECT_KEY] as any)?.name ?? cls.name;
    const instance = instances.get(name);

    for (const dep of deps) {
      instance[dep.field] = instances.get(dep.serviceKey);
    }
  }

  return instances;
}

#5. Type-Safe API Client Patterns typesafe_api_client
Most web apps use untyped fetch or axios calls where the response is any. A type-safe API client maps URL patterns to request/response types so that api.get('/users/:id', { id: '1' }) returns Promise<User> with full compile-time checking of the URL, parameters, and response shape.

The pattern involves three layers: (1) a route map that associates URL patterns with request/response types per HTTP method, (2) template literal types that extract path parameters from URL strings, and (3) a typed client that looks up the route map and enforces correct types at each call site.

Runtime validation with libraries like Zod can be added to bridge the gap between compile-time types and actual API responses. This pattern powers libraries like tRPC, ts-rest, and zodios.

type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<`/${Rest}`>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

  [ExtractParams<T>] extends [never]
    ? {}
    : { [K in ExtractParams<T>]: string };

  return url;
}

Example 3 — Building the typed API client
// The core typed client
type HttpMethod = 'GET' | 'POST' | 'PUT' | 'DELETE';

type RouteConfig = {
  response: unknown;
  body?: unknown;
  query?: Record<string, unknown>;
};

// Determine what arguments the client method needs
type RequestOptions<
  Route extends string,
  Config extends RouteConfig
> = PathParams<Route> extends Record<string, never>
  ? Config extends { body: infer B }
    ? { body: B; query?: Config extends { query?: infer Q } ? Q : never }
    : Config extends { query?: infer Q }
      ? { query?: Q }
      : {}
  : Config extends { body: infer B }
    ? { params: PathParams<Route>; body: B; query?: Config extends { query?: infer Q } ? Q : never }
    : { params: PathParams<Route>; query?: Config extends { query?: infer Q } ? Q : never };

// The client class
class ApiClient<Routes extends Record<string, Partial<Record<HttpMethod, RouteConfig>>>> {
  constructor(private baseUrl: string) {}

  async request<
    P extends keyof Routes & string,
    M extends keyof Routes[P] & HttpMethod
  >(
    method: M,
    path: P,
    options: RequestOptions<P, Routes[P][M] & RouteConfig>
  ): Promise<(Routes[P][M] & RouteConfig)['response']> {
    const opts = options as any;
    let url = `${this.baseUrl}${path}`;

    // Substitute path params
    if (opts.params) {
      for (const [key, val] of Object.entries(opts.params)) {
        url = url.replace(`:${key}`, val as string);
      }
    }

    // Add query params
    if (opts.query) {
      const qs = new URLSearchParams(opts.query as any).toString();
      url += `?${qs}`;
    }

    const response = await fetch(url, {
      method,
      headers: { 'Content-Type': 'application/json' },
      body: opts.body ? JSON.stringify(opts.body) : undefined,
    });

    return response.json();
  }
}

// Create a typed client instance
const api = new ApiClient<ApiRoutes>('https://api.example.com');

  name: z.string(),
  email: z.string().email(),
});

const UsersResponseSchema = z.array(UserSchema);

// Validated fetch wrapper
async function validatedFetch<T>(
  url: string,
  schema: z.ZodType<T>,
  init?: RequestInit
): Promise<T> {
  const response = await fetch(url, init);

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }

  const data = await response.json();

  // Runtime validation — throws if shape doesn't match
  return schema.parse(data);
  // Return type is T — inferred from the schema
}

// Usage — compile-time + runtime type safety
const users = await validatedFetch(
  'https://api.example.com/users',
  UsersResponseSchema
);
// users: { id: string; name: string; email: string }[]
// If API returns unexpected shape, Zod throws at runtime

// Combined: typed routes + Zod validation
interface ValidatedRoute<T> {
  path: string;
  method: HttpMethod;
  schema: z.ZodType<T>;
}

function createRoute<T>(
  path: string,
  method: HttpMethod,
  schema: z.ZodType<T>
): ValidatedRoute<T> {
  return { path, method, schema };
}

const getUserRoute = createRoute('/users/:id', 'GET', UserSchema);
// getUserRoute.schema validates User shape at runtime
// Return type is inferred as ValidatedRoute<{ id: string; name: string; email: string }>
users: { id: string; name: string; email: string }[] getUserRoute: ValidatedRoute<{ id: string; name: string; email: string }>
Common Mistakes
Trusting response.json() without runtime validation — the compile-time type is only as good as your type assertion. APIs can change, return errors, or have bugs.
Always validate API responses at runtime with Zod, io-ts, or similar. Combine compile-time types with runtime validation for true type safety.
Making the route map too complex with deeply nested conditional types — this slows down the TypeScript compiler and produces unreadable error messages.
Keep the route map flat and simple. Use helper types for path params but avoid more than 2-3 levels of conditional type nesting.
Key Rules for Type-Safe API Clients
  • Define the API contract as a route map type — URL patterns to request/response types
  • Extract path parameters using template literal types: :param segments become required properties
  • Enforce correct HTTP methods per route at the type level
  • Always add runtime validation (Zod, io-ts) for API responses — types alone cannot catch runtime mismatches
  • Consider existing libraries: tRPC (full-stack), ts-rest (REST), zodios (Zod + axios) before building from scratch
