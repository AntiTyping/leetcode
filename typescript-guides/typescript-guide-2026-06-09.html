R1. Discriminated Unions in Practice Runtime Patterns rep 0 · last reviewed 2026-06-04 · overdue 4 days
Discriminated unions use a shared literal property (e.g., type or status) to model state machines, event systems, and API responses. TypeScript narrows the union automatically inside switch/if blocks based on the discriminant value.
Key Pattern 
// File upload state machine
type UploadState =
  | { status: 'idle' }
  | { status: 'uploading'; progress: number; fileName: string }
  | { status: 'success'; url: string; size: number }
  | { status: 'error'; message: string; retryable: boolean };

function renderUpload(state: UploadState): string {
  switch (state.status) {
    case 'idle':
      return 'Select a file to upload';
    case 'uploading':
      return `Uploading ${state.fileName}: ${state.progress}%`;
    case 'success':
      return `Done! ${state.url} (${state.size} bytes)`;
    case 'error':
      return state.retryable
        ? `Failed: ${state.message} — click to retry`
        : `Failed: ${state.message}`;
    default:
      const _exhaustive: never = state;
      return _exhaustive;
  }
}

console.log(renderUpload({ status: 'uploading', progress: 42, fileName: 'photo.jpg' }));
Uploading photo.jpg: 42%
Challenge Exercise

Model a file upload state machine with four states: idle, uploading (with progress and fileName), success (with url), and error (with message and retryable). Write a renderUpload function that handles all states with proper narrowing and a never exhaustive check.

#1 Pitfall: Forgetting the exhaustive check with never in the default branch. Without it, adding a new state to the union compiles silently — the new case falls through to default with no error. Always assign the discriminant variable to never in the default case.
R2. Error Handling Patterns Runtime Patterns rep 0 · last reviewed 2026-06-04 · overdue 4 days
The Result<T, E> pattern replaces thrown exceptions with a typed return value. Tagged errors use discriminated unions to model multiple failure modes. The catch clause gives unknown — always narrow before using.
Key Pattern 
// Result type — success or failure, never both
type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function ok<T>(value: T): Result<T, never> {
  return { ok: true, value };
}

function err<E>(error: E): Result<never, E> {
  return { ok: false, error };
}

// Map over the success value
function map<T, U, E>(result: Result<T, E>, fn: (v: T) => U): Result<U, E> {
  return result.ok ? ok(fn(result.value)) : result;
}

// Chain Results (flatMap / bind)
function flatMap<T, U, E>(result: Result<T, E>, fn: (v: T) => Result<U, E>): Result<U, E> {
  return result.ok ? fn(result.value) : result;
}

// Unwrap with fallback
function unwrapOr<T, E>(result: Result<T, E>, fallback: T): T {
  return result.ok ? result.value : fallback;
}

// Usage
type ParseError = { kind: 'parse'; input: string };
type RangeError = { kind: 'range'; min: number; max: number; got: number };

function parseAge(input: string): Result<number, ParseError | RangeError> {
  const n = Number(input);
  if (Number.isNaN(n)) return err({ kind: 'parse', input });
  if (n < 0 || n > 150) return err({ kind: 'range', min: 0, max: 150, got: n });
  return ok(n);
}

const result = parseAge("25");
if (result.ok) {
  console.log(`Age: ${result.value}`);
} else if (result.error.kind === 'parse') {
  console.log(`Cannot parse: ${result.error.input}`);
}
Age: 25
Challenge Exercise

Implement a Result<T, E> type with helper functions: ok, err, map, flatMap, and unwrapOr. Then use it to write a parseAge function that returns Result<number, ParseError | RangeError> with tagged errors.

#1 Pitfall: Using unknown from a catch clause without narrowing. TypeScript types catch (e) as unknown — you must use instanceof Error or a type guard before accessing e.message. Never cast to Error directly since non-Error values can be thrown.
R3. Package Publishing Modules rep 0 · last reviewed 2026-06-04 · overdue 4 days
The exports field in package.json replaces main/types for modern packages. Use conditional exports with import, require, and types conditions. Dual CJS/ESM packages need separate build outputs and a types condition in each entry.
Key Pattern 
// package.json — dual CJS/ESM with 3 entry points
{
  "name": "@mylib/core",
  "version": "1.0.0",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./utils": {
      "types": "./dist/utils.d.ts",
      "import": "./dist/utils.mjs",
      "require": "./dist/utils.cjs"
    },
    "./react": {
      "types": "./dist/react.d.ts",
      "import": "./dist/react.mjs",
      "require": "./dist/react.cjs"
    }
  },
  "files": ["dist"],
  "typesVersions": {
    "*": {
      "utils": ["dist/utils.d.ts"],
      "react": ["dist/react.d.ts"]
    }
  }
}
Challenge Exercise

Configure a package.json with exports for a package that has three entry points: . (main), ./utils, and ./react. Each entry should support both ESM and CJS with proper types conditions.

#1 Pitfall: Missing the types condition inside exports. Without it, TypeScript cannot find type declarations for subpath imports. The types condition must come first in each entry (before import/require) because Node.js resolves conditions in order.
R4. Type-Level Programming Advanced rep 0 · last reviewed 2026-06-05 · overdue 3 days
Type-level programming uses recursive conditional types and variadic tuple types to compute at the type level. Recursive types like Flatten<T> unwrap nested arrays. Variadic tuples [...T, ...U] enable typed spread and concat operations.
Key Pattern 
// Recursive Flatten — unwraps nested arrays at the type level
type Flatten<T> = T extends Array<infer U> ? Flatten<U> : T;

type A = Flatten<number[][][]>;       // number
type B = Flatten<string[]>;            // string
type C = Flatten<boolean>;             // boolean

// Variadic tuple concat
type Concat<A extends unknown[], B extends unknown[]> = [...A, ...B];

type AB = Concat<[1, 2], [3, 4]>;     // [1, 2, 3, 4]

// Recursive tuple length
type Length<T extends unknown[]> = T['length'];

type L = Length<[string, number, boolean]>;  // 3

// TupleToUnion
type TupleToUnion<T extends unknown[]> = T[number];

type U = TupleToUnion<['a', 'b', 'c']>;  // 'a' | 'b' | 'c'
Flatten<number[][][]> = number Concat<[1,2],[3,4]> = [1, 2, 3, 4] TupleToUnion<['a','b','c']> = 'a' | 'b' | 'c'
Challenge Exercise

Implement Flatten<T> that recursively flattens nested arrays at the type level: Flatten<number[][][]> should resolve to number. Then implement TupleToUnion<T> that converts a tuple to a union of its element types.

#1 Pitfall: TypeScript has a recursion depth limit of approximately 50 levels. Deeply recursive types like Flatten<T> on extremely nested arrays will hit this limit and produce an error. Keep recursive type computations shallow or use tail-recursive patterns.
R5. infer Mastery Advanced rep 0 · last reviewed 2026-06-05 · overdue 3 days
The infer keyword extracts types from complex structures inside conditional types. It lets you "pattern match" on type shapes: T extends Promise<infer U> ? U : T unwraps a Promise. Use infer in function signatures, tuple positions, and template literal types.
Key Pattern 
// Extract return type of a method from a class
type ReturnTypeOfMethod<T, K extends keyof T> =
  T[K] extends (...args: any[]) => infer R ? R : never;

class UserService {
  getUser(id: string): { id: string; name: string } {
    return { id, name: 'Alice' };
  }
  deleteUser(id: string): boolean {
    return true;
  }
  readonly version: string;
}

type GetUserReturn = ReturnTypeOfMethod<UserService, 'getUser'>;
//   ^? { id: string; name: string }

type DeleteReturn = ReturnTypeOfMethod<UserService, 'deleteUser'>;
//   ^? boolean

type VersionReturn = ReturnTypeOfMethod<UserService, 'version'>;
//   ^? never (not a method)

// Extract first argument type
type FirstArg<T> = T extends (first: infer A, ...rest: any[]) => any ? A : never;

type IdType = FirstArg<(id: string, force: boolean) => void>;
//   ^? string
ReturnTypeOfMethod<UserService, 'getUser'> = { id: string; name: string } FirstArg<(id: string, force: boolean) => void> = string
Challenge Exercise

Write ReturnTypeOfMethod<T, K> that extracts the return type of method K from class T. It should return never for non-method properties. Then write FirstArg<T> that extracts the first parameter type from a function type.

#1 Pitfall: Forgetting to distribute infer over unions. When T is a union, conditional types distribute by default: ReturnTypeOfMethod<UserService, 'getUser' | 'deleteUser'> produces { id: string; name: string } | boolean. This is usually desired, but can surprise you if you expected a single type.
R6. Immutability Patterns Runtime rep 0 · last reviewed 2026-06-05 · overdue 3 days
readonly prevents reassignment of properties. Readonly<T> and ReadonlyArray<T> are shallow — nested objects remain mutable. Object.freeze is also shallow at runtime. Use as const for deeply immutable literal values.
Key Pattern 
// DeepReadonly — recursively makes all properties readonly
type DeepReadonly<T> =
  T extends (infer U)[]
    ? ReadonlyArray<DeepReadonly<U>>
    : T extends object
      ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
      : T;

interface Config {
  db: { host: string; port: number; replicas: string[] };
  features: { darkMode: boolean };
}

type FrozenConfig = DeepReadonly<Config>;
// {
//   readonly db: {
//     readonly host: string;
//     readonly port: number;
//     readonly replicas: ReadonlyArray<string>;
//   };
//   readonly features: { readonly darkMode: boolean };
// }

declare const config: FrozenConfig;
// config.db.port = 5433;             // Error: readonly
// config.db.replicas.push('r2');     // Error: push not on ReadonlyArray
// config.features.darkMode = false;  // Error: readonly
Cannot assign to 'port' because it is a read-only property.
Challenge Exercise

Implement DeepReadonly<T> that makes all nested properties readonly, including array elements. Test it with a nested config object that has arrays, objects, and primitive values.

#1 Pitfall: readonly only prevents reassignment, not mutation of nested objects. Readonly<{ nested: { x: number } }> still allows obj.nested.x = 5. You need DeepReadonly or as const for true deep immutability at the type level.
1 State Management Typing: Redux Toolkit & Zustand
State Management Typing Redux Toolkit Zustand Ecosystem
Typed state management prevents dispatching wrong actions, accessing nonexistent state properties, and selector type mismatches. When your store is fully typed, refactoring state shape becomes a compile-time task instead of a runtime debugging session.

Two dominant approaches in the React ecosystem, both with excellent TypeScript support:

  • 2026-07-04 (Day 31)
  • Redux Toolkit (RTK): createSlice infers action types from reducer names. PayloadAction<T> types the action payload. Typed hooks (useAppSelector, useAppDispatch) enforce correct usage everywhere
  • Zustand: create<State>()(set => ({ ... })) types the entire store. Selectors are typed automatically. Middleware like devtools and persist need explicit type annotations
  • The key difference: RTK is action-based (dispatch actions, reducers update state), Zustand is setter-based (call methods directly on the store)
  • Both prevent accessing state that does not exist and catch payload shape mismatches at compile time
rtk-slice.ts
import { createSlice, PayloadAction, configureStore } from '@reduxjs/toolkit';

// 1. Define state shape
interface CartItem {
  id: string;
  name: string;
  price: number;
  quantity: number;
}

interface CartState {
  items: CartItem[];
  coupon: string | null;
}

const initialState: CartState = {
  items: [],
  coupon: null,
};

// 2. createSlice infers action types from reducer names
const cartSlice = createSlice({
  name: 'cart',
  initialState,
  reducers: {
    addItem(state, action: PayloadAction<Omit<CartItem, 'quantity'>>) {
      const existing = state.items.find(i => i.id === action.payload.id);
      if (existing) {
        existing.quantity += 1;
      } else {
        state.items.push({ ...action.payload, quantity: 1 });
      }
    },
    removeItem(state, action: PayloadAction<string>) {
      state.items = state.items.filter(i => i.id !== action.payload);
    },
    updateQuantity(state, action: PayloadAction<{ id: string; quantity: number }>) {
      const item = state.items.find(i => i.id === action.payload.id);
      if (item) item.quantity = action.payload.quantity;
    },
    applyCoupon(state, action: PayloadAction<string>) {
      state.coupon = action.payload;
    },
    clearCart(state) {
      state.items = [];
      state.coupon = null;
    },
  },
});

// Action creators are auto-generated and typed
export const { addItem, removeItem, updateQuantity, applyCoupon, clearCart } = cartSlice.actions;

// 3. Configure store
const store = configureStore({
  reducer: { cart: cartSlice.reducer },
});

// 4. Infer root types from the store
export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;
rtk-typed-hooks.ts
import { useDispatch, useSelector, TypedUseSelectorHook } from 'react-redux';
import type { RootState, AppDispatch } from './store';

// Typed hooks — use these everywhere instead of plain useDispatch/useSelector
export const useAppDispatch: () => AppDispatch = useDispatch;
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;

// Usage in a component:
function CartSummary() {
  const items = useAppSelector(state => state.cart.items);      // CartItem[]
  const coupon = useAppSelector(state => state.cart.coupon);    // string | null
  const dispatch = useAppDispatch();

  const total = items.reduce((sum, i) => sum + i.price * i.quantity, 0);

  return (
    <div>
      <p>Items: {items.length}, Total: ${total.toFixed(2)}</p>
      <button onClick={() => dispatch(addItem({ id: 'p1', name: 'Widget', price: 9.99 }))}>
        Add Widget
      </button>
      {/* Type error: missing 'name' and 'price' */}
      {/* dispatch(addItem({ id: 'p1' })) */}
    </div>
  );
}
Argument of type '{ id: string; }' is not assignable to parameter of type 'Omit<CartItem, "quantity">'. Type '{ id: string; }' is missing properties 'name', 'price'.
zustand-store.ts
import { create } from 'zustand';
import { devtools, persist } from 'zustand/middleware';

// Define store interface
interface AuthState {
  user: { id: string; name: string; role: 'admin' | 'user' } | null;
  token: string | null;
  isLoading: boolean;
  login: (name: string, password: string) => Promise<void>;
  logout: () => void;
  refresh: () => Promise<void>;
}

// create<AuthState>() — explicit type parameter ensures full typing
const useAuthStore = create<AuthState>()(
  devtools(
    persist(
      (set, get) => ({
        user: null,
        token: null,
        isLoading: false,

        login: async (name, password) => {
          set({ isLoading: true });
          const response = await fetch('/api/login', {
            method: 'POST',
            body: JSON.stringify({ name, password }),
          });
          const data = await response.json();
          set({ user: data.user, token: data.token, isLoading: false });
        },

        logout: () => {
          set({ user: null, token: null });
        },

        refresh: async () => {
          const currentToken = get().token;
          if (!currentToken) return;
          const response = await fetch('/api/refresh', {
            headers: { Authorization: `Bearer ${currentToken}` },
          });
          const data = await response.json();
          set({ token: data.token });
        },
      }),
      { name: 'auth-storage' }
    )
  )
);

// Typed selectors — extracted for reuse and performance
const selectUser = (state: AuthState) => state.user;
const selectIsAdmin = (state: AuthState) => state.user?.role === 'admin';
const selectIsAuthenticated = (state: AuthState) => state.token !== null;

// Usage:
function UserBadge() {
  const user = useAuthStore(selectUser);           // { id; name; role } | null
  const isAdmin = useAuthStore(selectIsAdmin);     // boolean
  const logout = useAuthStore(state => state.logout);

  if (!user) return <span>Guest</span>;
  return (
    <span>
      {user.name} {isAdmin && '(admin)'}
      <button onClick={logout}>Logout</button>
    </span>
  );
}
Common Mistakes
Using plain useSelector and useDispatch without typed wrappers: useSelector(state => state.cart.items) — state is unknown
Create typed hooks: export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector
Zustand store without explicit type: create((set) => ({ count: 0 })) — loses inference for methods
Always provide the type parameter: create<StoreState>()((set) => ({ ... }))
Inline selectors in every component: useAuthStore(s => s.user?.role === 'admin') — creates a new function each render
Extract selectors as named constants outside the component to prevent unnecessary re-renders
Key Rules
  • Always create typed useAppSelector and useAppDispatch hooks for RTK
  • Use PayloadAction<T> to type reducer payloads in createSlice
  • Provide explicit type parameter to Zustand's create<State>()
  • Extract Zustand selectors as stable references outside components
  • Derive RootState and AppDispatch from the store itself with ReturnType
Exercises

2Create a Zustand store for authentication state with user, token, isLoading, and methods login, logout, refresh. Add typed selectors for selectUser, selectIsAdmin, and selectIsAuthenticated.
3Build a Zustand store that uses the persist middleware with proper TypeScript typing. The store should manage a preferences object with theme, language, and notifications settings.

2 Database Typing: Prisma & Drizzle
Database Typing Prisma Drizzle Ecosystem
Type-safe database queries catch shape mismatches at compile time. No more "column not found" or wrong field names at runtime. With Prisma or Drizzle, changing a column name or type surfaces every affected query in your IDE instantly.

Two leading approaches to typed databases in TypeScript:

  • Prisma: Schema-first. Write a .prisma file, run prisma generate, get a fully typed client. include and select narrow the return type automatically
  • Drizzle: Code-first. Define tables in TypeScript with pgTable/mysqlTable. Types are inferred from schema definitions. InferSelectModel and InferInsertModel extract types
  • Prisma generates types into node_modules/.prisma/client — you must re-run prisma generate after schema changes
  • Drizzle types are always in sync because the schema IS TypeScript — no code generation step needed
  • Both support typed transactions, relations, and aggregate queries
schema.prisma
// Prisma schema — generates TypeScript types
model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String
  role      Role     @default(USER)
  posts     Post[]
  createdAt DateTime @default(now())
}

model Post {
  id        String    @id @default(uuid())
  title     String
  body      String
  published Boolean   @default(false)
  author    User      @relation(fields: [authorId], references: [id])
  authorId  String
  comments  Comment[]
}

model Comment {
  id     String @id @default(uuid())
  text   String
  post   Post   @relation(fields: [postId], references: [id])
  postId String
}

enum Role {
  USER
  ADMIN
  MODERATOR
}
prisma-queries.ts
import { PrismaClient, Prisma } from '@prisma/client';

const prisma = new PrismaClient();

// Type-safe query — return type narrows based on include/select
async function getPostWithAuthor(postId: string) {
  // Return type includes author because of `include`
  const post = await prisma.post.findUnique({
    where: { id: postId },
    include: {
      author: true,         // adds author: User to return type
      comments: true,       // adds comments: Comment[] to return type
    },
  });
  // post is: (Post & { author: User; comments: Comment[] }) | null

  if (post) {
    console.log(post.author.name);    // string — typed!
    console.log(post.comments.length); // number
  }
  return post;
}

// Select narrows to only requested fields
async function getUserEmails() {
  const users = await prisma.user.findMany({
    select: {
      email: true,
      name: true,
      // id, role, posts, createdAt are NOT in the return type
    },
  });
  // users is: { email: string; name: string }[]

  // users[0].role  // Error: 'role' does not exist
  return users;
}

// Typed create input
async function createPost(input: Prisma.PostCreateInput) {
  // Prisma.PostCreateInput is generated:
  // { title: string; body: string; published?: boolean;
  //   author: UserCreateNestedOneWithoutPostsInput; ... }
  return prisma.post.create({ data: input });
}

// Type-safe where clauses
async function searchPosts(query: string, role?: 'USER' | 'ADMIN') {
  return prisma.post.findMany({
    where: {
      published: true,
      title: { contains: query, mode: 'insensitive' },
      author: role ? { role } : undefined,
    },
    orderBy: { createdAt: 'desc' },
    take: 20,
  });
}
Property 'role' does not exist on type '{ email: string; name: string; }'.
drizzle-schema.ts
import { pgTable, uuid, text, boolean, timestamp, pgEnum } from 'drizzle-orm/pg-core';
import { InferSelectModel, InferInsertModel, relations } from 'drizzle-orm';

// Define enum
export const roleEnum = pgEnum('role', ['USER', 'ADMIN', 'MODERATOR']);

// Define tables — types are inferred from column definitions
export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull().unique(),
  name: text('name').notNull(),
  role: roleEnum('role').notNull().default('USER'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

export const posts = pgTable('posts', {
  id: uuid('id').primaryKey().defaultRandom(),
  title: text('title').notNull(),
  body: text('body').notNull(),
  published: boolean('published').notNull().default(false),
  authorId: uuid('author_id').notNull().references(() => users.id),
});

export const comments = pgTable('comments', {
  id: uuid('id').primaryKey().defaultRandom(),
  text: text('text').notNull(),
  postId: uuid('post_id').notNull().references(() => posts.id),
});

// Define relations
export const usersRelations = relations(users, ({ many }) => ({
  posts: many(posts),
}));

export const postsRelations = relations(posts, ({ one, many }) => ({
  author: one(users, { fields: [posts.authorId], references: [users.id] }),
  comments: many(comments),
}));

// Extract TypeScript types from schema
type User = InferSelectModel<typeof users>;
//   ^? { id: string; email: string; name: string; role: 'USER'|'ADMIN'|'MODERATOR'; createdAt: Date }

type NewUser = InferInsertModel<typeof users>;
//   ^? { id?: string; email: string; name: string; role?: 'USER'|'ADMIN'|'MODERATOR'; createdAt?: Date }
User = { id: string; email: string; name: string; role: 'USER' | 'ADMIN' | 'MODERATOR'; createdAt: Date } NewUser = { id?: string; email: string; name: string; role?: 'USER' | 'ADMIN' | 'MODERATOR'; createdAt?: Date }
drizzle-queries.ts
import { eq, and, ilike, desc } from 'drizzle-orm';
import { drizzle } from 'drizzle-orm/node-postgres';
import * as schema from './drizzle-schema';

const db = drizzle(pool, { schema });

// Type-safe query builder
async function getPostWithAuthor(postId: string) {
  const result = await db.query.posts.findFirst({
    where: eq(schema.posts.id, postId),
    with: {
      author: true,
      comments: true,
    },
  });
  // result includes author and comments, fully typed
  if (result) {
    console.log(result.author.name);     // string
    console.log(result.comments.length);  // number
  }
  return result;
}

// Type-safe insert
async function createUser(email: string, name: string) {
  const [user] = await db.insert(schema.users)
    .values({ email, name })  // role and createdAt have defaults
    .returning();
  // user is fully typed: User
  return user;
}

// Type-safe SQL builder
async function searchPosts(query: string) {
  return db.select()
    .from(schema.posts)
    .where(
      and(
        eq(schema.posts.published, true),
        ilike(schema.posts.title, `%${query}%`),
      )
    )
    .orderBy(desc(schema.posts.createdAt))
    .limit(20);
}
Common Mistakes
Not regenerating the Prisma client after schema changes: npx prisma generate must run after every schema.prisma edit
Add "postinstall": "prisma generate" to package.json scripts and run it in CI
Drizzle: not using with for relations and then accessing joined fields
Always use with: { author: true } in the query to include relation data in the return type
Using raw SQL strings instead of the typed query builder: db.execute(sql`SELECT * FROM users`)
Use the query builder for type safety. Reserve sql template tags for complex queries that the builder cannot express
Key Rules

3 GraphQL Codegen
GraphQL Codegen GraphQL Ecosystem
Hand-writing TypeScript types for GraphQL queries is error-prone and falls out of sync with the schema. Codegen generates types from your actual schema and operations, so the compiler catches query/type mismatches immediately.

GraphQL codegen bridges the gap between your GraphQL schema and TypeScript types:

  • @graphql-codegen/cli reads your schema and .graphql operations, then generates TypeScript types
  • TypedDocumentNode embeds the result/variables types directly into the query document — hooks automatically infer types
  • The near-operation-file preset generates types next to each .graphql file for colocation
  • Fragments generate their own types — you compose typed fragments just like you compose GraphQL fragments
  • With typed document nodes, useQuery(GetUserDocument) automatically types data as GetUserQuery and variables as GetUserQueryVariables
codegen.ts (config)
import type { CodegenConfig } from '@graphql-codegen/cli';

const config: CodegenConfig = {
  // Schema source: URL, file, or introspection
  schema: 'http://localhost:4000/graphql',

  // Where to find .graphql operation files
  documents: ['src/**/*.graphql'],

  generates: {
    // Generate a single types file
    'src/gql/': {
      preset: 'client',  // modern preset: typed document nodes
      config: {
        // Use exact optional properties
        avoidOptionals: {
          field: true,
          inputValue: false,
          object: true,
        },
        // Enum values as string unions instead of TS enums
        enumsAsTypes: true,
        // Immutable types
        immutableTypes: true,
      },
    },
  },
};

export default config;

// Or watch mode: npx graphql-codegen --watch

4 Testing React Components
Testing React Components React Testing Library Testing
Properly typed tests catch prop mismatches and wrong query patterns at compile time, not just runtime. If a component's prop type changes, every test that constructs those props will show a type error immediately — before you even run the test.

React Testing Library (RTL) has strong TypeScript support. Key patterns:

  • render() returns a typed object with container, rerender, unmount, and query methods
  • screen provides global query methods: getByRole, getByText, queryByText (nullable), findByText (async)
  • userEvent.setup() returns a typed user instance with click, type, keyboard, clear methods
  • Mock functions use vi.fn<[...Args], Return>() (Vitest) or jest.fn<Return, [...Args]>() for typed mocks
  • Custom render wrappers provide typed context (Redux store, theme, auth) to all tests
login-form.tsx (component under test)
interface LoginFormProps {
  onSubmit: (data: { email: string; password: string }) => void;
  isLoading?: boolean;
  error?: string;
}

function LoginForm({ onSubmit, isLoading = false, error }: LoginFormProps) {
  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    const form = new FormData(e.currentTarget as HTMLFormElement);
    onSubmit({
      email: form.get('email') as string,
      password: form.get('password') as string,
    });
  };

  return (
    <form onSubmit={handleSubmit}>
      <label htmlFor="email">Email</label>
      <input id="email" name="email" type="email" required />
      <label htmlFor="password">Password</label>
      <input id="password" name="password" type="password" required />
      {error && <p role="alert">{error}</p>}
      <button type="submit" disabled={isLoading}>
        {isLoading ? 'Signing in...' : 'Sign in'}
      </button>
    </form>
  );
}

import { render, RenderOptions } from '@testing-library/react';
import { ReactNode } from 'react';
import { Provider } from 'react-redux';
import { configureStore, PreloadedState } from '@reduxjs/toolkit';
import { ThemeProvider } from './theme-context';
import type { RootState } from './store';
import { rootReducer } from './store';

// Typed render options — extends RTL's RenderOptions
interface CustomRenderOptions extends Omit<RenderOptions, 'wrapper'> {
  preloadedState?: PreloadedState<RootState>;
  theme?: 'light' | 'dark';
}

function customRender(ui: ReactNode, options: CustomRenderOptions = {}) {
  const { preloadedState, theme = 'light', ...renderOptions } = options;

  const store = configureStore({
    reducer: rootReducer,
    preloadedState,
  });

  function Wrapper({ children }: { children: ReactNode }) {
    return (
      <Provider store={store}>
        <ThemeProvider defaultTheme={theme}>
          {children}
        </ThemeProvider>
      </Provider>
    );
  }

  return {
    ...render(ui, { wrapper: Wrapper, ...renderOptions }),
    store,  // Expose store for assertions
  };
}

export { customRender as render };

5 Monorepo Typing: Turborepo & nx
Monorepo Typing Turborepo nx Ecosystem
Monorepos share types across packages. Getting the typing right prevents build-order issues and enables proper IDE support (go-to-definition, autocomplete) across package boundaries. Without correct setup, changes to shared types silently break consuming packages.

Sharing types across packages in a monorepo requires coordination between TypeScript, the package manager, and the build tool:

  • Workspace protocols: "@app/shared": "workspace:*" (pnpm) or "*" (npm/yarn) link to local packages
  • Internal packages: Set "main" and "types" in each package's package.json to point to source or built output
  • Composite tsconfig: "composite": true enables project references. The root tsconfig uses "references" to point to each package
  • Build ordering: Turborepo's pipeline / nx's targetDefaults ensure dependencies build before dependents
  • publishConfig: Overrides main/types when publishing to npm, so you can point to source during development and to dist for publishing
packages/shared/package.json

  "name": "@app/shared",
  "version": "0.0.0",
  "private": true,
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "exports": {
    ".": {
      "types": "./src/index.ts",
      "default": "./src/index.ts"
    }

    "build": "tsc --build",
