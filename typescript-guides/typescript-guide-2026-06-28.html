R1. Type Annotations Core Type System 20 days since review · rep 3 → 4
Annotations describe the shape of a value; inference fills them in when it can. The skill is knowing where an annotation earns its keep (function parameters, empty containers, public boundaries) and where it just fights inference (obvious literals, most local consts).
Key Pattern 
// annotate the boundary, let the body infer
function total(items: readonly { price: number; qty: number }[]): number {
  return items.reduce((sum, i) => sum + i.price * i.qty, 0);  // sum: number inferred
}

const point: [number, number] = [0, 0];    // tuple, not number[]
const ids: string[] = [];                   // else inferred never[]
let seen;                                    // any (implicit) — avoid
const label = "ready";                       // "ready" literal — no annotation needed
annotate params/returns/empty arrays; skip obvious literals
Challenge Exercise

Write a function groupBy<T>(items: T[], key: (item: T) => string): Record<string, T[]> and annotate only what inference can’t supply. Then create an empty accumulator inside it and show what its type would be without an annotation ({}) versus with (Record<string, T[]>), and explain which one lets acc[k].push(...) compile.

Pitfall: Declaring let x; (implicit any) or const rows = [] that infers never[], then getting cryptic errors on first use. Annotate empty containers and uninitialized let up front; conversely, don’t annotate a const whose literal already carries the exact type you want.
R2. Union Types & Narrowing Core Type System 20 days since review · rep 3 → 4
A union A | B is “one of these”; you can only touch the members common to all branches until you narrow. Control-flow narrowing (typeof, in, instanceof, truthiness, and a discriminant tag) tells the compiler which member you’re holding.
Key Pattern 
type Shape =
  | { kind: "circle"; r: number }
  | { kind: "rect"; w: number; h: number };

function area(s: Shape): number {
  switch (s.kind) {                 // discriminant narrows each branch
    case "circle": return Math.PI * s.r ** 2;   // s: circle
    case "rect":   return s.w * s.h;            // s: rect
  }
}

function len(x: string | string[]): number {
  return typeof x === "string" ? x.length : x.length;  // both branches ok
}
discriminant + typeof narrow the union to a single member
Challenge Exercise

Given type Result<T> = { ok: true; value: T } | { ok: false; error: string }, write unwrap<T>(r: Result<T>): T that narrows on r.ok and throws on the error branch. Then add a third case and use a never-typed default to prove the compiler forces you to handle it (foreshadowing exhaustiveness).

Pitfall: Reaching into a union member before narrowing (s.r when s is still Shape) — only the common properties are accessible. Also, typeof narrowing can’t distinguish two object types; use a discriminant property or an in check for those.
R3. Literal Types & const Assertions Core Type System 18 days since review · rep 3 → 4
A literal type is a single exact value ("GET", 42, true). By default TypeScript widens literals in mutable positions (let, object properties) to their base type; as const stops that, freezing values and making everything deeply readonly.
Key Pattern 
let method = "GET";                 // widened to string
const m2 = "GET";                   // "GET" (const keeps the literal)

const config = { method: "GET", retries: 3 };
// type: { method: string; retries: number }  — widened

const frozen = { method: "GET", retries: 3 } as const;
// type: { readonly method: "GET"; readonly retries: 3 }

const routes = ["home", "about"] as const;   // readonly ["home", "about"]
type Route = typeof routes[number];           // "home" | "about"
as const preserves literals + makes the value deeply readonly
Challenge Exercise

Define const HTTP_METHODS = ["GET","POST","PUT","DELETE"] as const and derive a Method union from it with typeof ... [number]. Then write request(url: string, method: Method) and confirm a typo ("PATCH") is a compile error. Finally, show what breaks if you drop as const (the derived type collapses to string).

Pitfall: Passing an object literal to a function expecting a literal-typed field and getting a widening error, or deriving a union from an array that isn’t as const (you get string, not the member union). Remember as const also makes the value readonly — you can no longer .push to it.
R4. Interfaces vs Type Aliases Structuring Types 18 days since review · rep 3 → 4
Both name object shapes. interface supports declaration merging and extends, and gives nicer error messages for public object contracts. type can express anything — unions, tuples, mapped/conditional types, primitives — which interfaces cannot.
Key Pattern 
interface User { id: string; name: string }
interface User { role: "admin" | "user" }   // merges into one User
interface Admin extends User { perms: string[] }

// type aliases do what interfaces can't:
type Id = string | number;                    // union
type Pair = [number, number];                 // tuple
type Nullable<T> = T | null;                   // generic alias
type Keys = keyof User;                        // "id" | "name" | "role"
interface = mergeable object contracts; type = anything
Challenge Exercise

Model a Shape discriminated union (only type can do this) and an Entity base object contract that three interfaces extends. Then demonstrate declaration merging by augmenting a third-party interface (e.g. add a field to Window) and explain why you couldn’t achieve that augmentation with a type alias.

Pitfall: Trying to extends a union with an interface (illegal — use type), or being surprised that two same-named interfaces silently merge (a type would instead error on redeclaration). Pick interface for extensible public object APIs, type for unions/tuples/computed types.
R5. Generics Basics Generic & Utility Types 18 days since review · rep 3 → 4
A generic parameter is a type the caller (or inference) fills in, letting one function/type work over many types without losing precision. Constraints (T extends ...) let you require capabilities while keeping T as specific as the caller passed.
Key Pattern 
function first<T>(arr: readonly T[]): T | undefined {
  return arr[0];
}
const n = first([1, 2, 3]);        // number | undefined (T inferred)

// constraint: T must have the key K
function pluck<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}
const name = pluck({ id: 1, name: "Ada" }, "name");  // string
// pluck({ id: 1 }, "nope");   // error: "nope" not a key of T
generics preserve the caller’s exact type through the call
Challenge Exercise

Write mapValues<T, U>(obj: Record<string, T>, fn: (v: T) => U): Record<string, U>, then a constrained maxBy<T>(items: T[], score: (t: T) => number): T | undefined. Prove the return type tracks the input element type by hovering it in your editor, and add a K extends keyof T overload that lets callers pass a property name instead of a scoring function.

Pitfall: Adding a type parameter that appears only once in the signature (e.g. <T>(x: T): void) — it buys nothing over unknown and often masks a design smell. A generic earns its place only when the same T links an input to an output (or two inputs) so the relationship is preserved.
R6. Enums Structuring Types 17 days since review · rep 3 → 4
Enums name a fixed set of related constants. Numeric enums are reverse-mapped and start at 0; string enums are readable and stable across serialization. In modern code a const object + as const (or a literal union) is often the leaner, tree-shakeable alternative.
Key Pattern 
enum Status { Active = "ACTIVE", Closed = "CLOSED" }   // string enum
const s: Status = Status.Active;   // "ACTIVE" at runtime

// modern alternative: const object + derived union (no runtime enum object)
const Role = { Admin: "admin", User: "user" } as const;
type Role = typeof Role[keyof typeof Role];   // "admin" | "user"

function setRole(r: Role) {}
setRole("admin");        // ok — plain string literal accepted
// setRole("root");      // error
string enum vs const-object union: pick by tree-shaking + ergonomics
Challenge Exercise

Model a LogLevel two ways: as a string enum and as a const object + derived union. Write a log(level, msg) for each and compare: which one lets a caller pass a bare string literal, which produces a runtime object, and which is fully erased by the bundler? Then note why const enum can be risky under isolatedModules.

Pitfall: Reaching for a numeric enum where the values get serialized — reordering members silently changes the persisted numbers. Prefer string enums (or a const-object union) for anything that crosses a wire or storage boundary, and remember numeric enums accept any number, weakening exhaustiveness.
1React + TypeScript: Component Patterns II
1. Polymorphic Components & the as Prop React Patterns
Design-system primitives like <Box>, <Text>, or <Button> often need to render as different elements — a button, an a, a Link — while still accepting exactly the right props for whatever they render as. A polymorphic component captures that with a single generic as prop, and typing it well is the difference between a component that autocompletes href only when it’s an anchor and one that silently accepts nonsense.

The core idea: make the component generic over an element type E extends ElementType, and merge your own props with the intrinsic props of that element (ComponentPropsWithoutRef<E>). The as prop chooses E, so the accepted props change with it — pick "a" and you get href; pick "button" and you get type/disabled.

Two refinements make it production-grade. First, subtract your own prop names from the element props so a component-level prop can’t collide with a native one (Omit<ComponentPropsWithoutRef<E>, keyof OwnProps>). Second, default as to a sensible element so callers can omit it. Because the whole thing is a generic function component, inference flows from the as value you pass at the call site to the props TypeScript then checks — no manual type argument needed.

1
Make the component generic: <E extends ElementType = "span">, with an optional as?: E prop.
2
Merge your own props with ComponentPropsWithoutRef<E>, using Omit to avoid collisions.
3
The chosen as value drives which native props are required/allowed at the call site.
TypeScript · example 1 of 4 — the prop type, built from the element
import type { ElementType, ComponentPropsWithoutRef } from "react";

type TextOwnProps<E extends ElementType> = {
  as?: E;
  size?: "sm" | "md" | "lg";
};

// own props + the intrinsic props of E, minus any collisions
type TextProps<E extends ElementType> =
  TextOwnProps<E> &
  Omit<ComponentPropsWithoutRef<E>, keyof TextOwnProps<E>>;
TextProps<"a"> includes href; TextProps<"button"> includes disabled
TypeScript · example 2 of 4 — the generic component
function Text<E extends ElementType = "span">(
  { as, size = "md", ...rest }: TextProps<E>
) {
  const Component = as ?? "span";
  return <Component className={`text-${size}`} {...rest} />;
}
as defaults to "span"; rest is the correctly-typed element props
TypeScript · example 3 of 4 — props change with the as value
<Text>plain span</Text>;                          // ok, defaults to span
<Text as="a" href="/home">a link</Text>;          // href allowed & typed
<Text as="button" onClick={() => {}} size="lg"/>; // button props allowed

<Text as="span" href="/x" />;                     // error: href not on span
<Text as="a" />;                                  // ok (href optional on anchor)
Property 'href' does not exist on span props (line 4)
TypeScript · example 4 of 4 — rendering a custom component via as
function Link({ to, children }: { to: string; children: React.ReactNode }) {
  return <a href={to}>{children}</a>;
}

// as can be your own component too — its props are required
<Text as={Link} to="/about">About</Text>;   // to is required & checked
// <Text as={Link} />;                       // error: missing 'to'
as={Link} pulls in Link's own required props (to)
Common Mistakes:
❌ Typing as as keyof JSX.IntrinsicElements only — then you can never render a custom component through it.
✅ Constrain to ElementType, which covers both intrinsic tags and component types.
❌ Spreading ComponentPropsWithoutRef<E> without Omit, so a native color/size silently clashes with your own prop of the same name.
✅ Subtract your own keys: Omit<ComponentPropsWithoutRef<E>, keyof OwnProps> so your prop always wins with its intended type.
Key Rules:
  • 2026-07-04 (Day 31)
  • Constrain the element parameter to ElementType and default it so as is optional.
  • Compose props as own props & Omit<ComponentPropsWithoutRef<E>, keyof OwnProps>.
  • Use ComponentPropsWithoutRef unless you also forward a ref (then it’s PropsWithRef + forwardRef, which is trickier to type).
  • The as value is inferred at the call site — callers never pass a manual type argument.
Practice Exercises:
1Build a polymorphic <Box as?> that defaults to "div", accepts a padding?: number own prop, and forwards all other element props. Prove <Box as="a" href="/x" /> compiles while <Box href="/x" /> (default div) errors.
2Add an own prop named color to Box and show that without Omit you get a conflicting-type error on intrinsic elements that also have color; then fix it with Omit.
3Extend Box to forward a ref using forwardRef, and describe the type challenge that appears (the generic is lost through forwardRef) and one workaround (a cast or a manually-typed wrapper).
Connections: This is Generics Basics (R5) applied to components, and the Omit/keyof composition reuses Utility Types and keyof & Indexed Access. It sets up Concept 3 (discriminated props) as the alternative when variants can’t be expressed by a single element swap, and the forwardRef exercise links to React + TypeScript: Component Props.
2. Typing React Context Safely React Patterns
Context is how you share state without prop-drilling, but the default value forces a design decision: either your context type includes undefined (and every consumer must guard) or you provide a fake default (and a component used outside the provider fails silently at runtime). The idiomatic fix — a undefined default plus a custom hook that asserts — gives consumers a non-nullable type and a loud error when the provider is missing.

createContext<T>(defaultValue) demands a default. If your real value only exists inside a provider, the honest type is T | undefined with a default of undefined. But you don’t want every consumer writing if (!ctx) throw. So you wrap the access in a custom hook that reads the context, throws if it’s undefined, and — crucially — returns the narrowed T. After the throw, control-flow narrowing removes undefined from the type, so callers get a clean non-nullable value.

This pattern also centralizes the “used outside provider” error message in one place, and keeps the context object itself un-exported so nobody bypasses the hook. For values that never change (config), you can skip state; for mutable state, put both the value and its updater in the context type so consumers get a fully-typed API. It composes naturally with the discriminated-union and reducer patterns you already know.

1
Type the context as T | undefined and default to undefined — the honest “no provider yet” state.
2
Expose a custom hook that throws when the value is undefined, so the return type narrows to T.
3
Keep the raw context private; consumers only ever call the hook.
TypeScript · example 1 of 4 — the context + asserting hook
import { createContext, useContext } from "react";

interface AuthValue { user: string | null; login(name: string): void }

const AuthContext = createContext<AuthValue | undefined>(undefined);

export function useAuth(): AuthValue {
  const ctx = useContext(AuthContext);
  if (ctx === undefined) {
    throw new Error("useAuth must be used within <AuthProvider>");
  }
  return ctx;            // narrowed: AuthValue (no | undefined)
}
useAuth() returns AuthValue — consumers never see undefined
TypeScript · example 2 of 4 — the provider owns the state
export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<string | null>(null);
  const value: AuthValue = { user, login: setUser };
  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}
value is checked against AuthValue at the provider boundary
TypeScript · example 3 of 4 — consumers get a clean, typed API
function Profile() {
  const { user, login } = useAuth();     // fully typed, non-nullable
  return user
    ? <p>Hello {user}</p>
    : <button onClick={() => login("Ada")}>Log in</button>;
}

// rendered outside <AuthProvider> -> throws the clear error at runtime
Error: useAuth must be used within <AuthProvider> (when unwrapped)
TypeScript · example 4 of 4 — a reusable createSafeContext factory
function createSafeContext<T>(name: string) {
  const Ctx = createContext<T | undefined>(undefined);
  function useCtx(): T {
    const v = useContext(Ctx);
    if (v === undefined) throw new Error(`use${name} outside ${name}Provider`);
    return v;
  }
  return [Ctx.Provider, useCtx] as const;   // tuple, positions preserved
}

const [ThemeProvider, useTheme] = createSafeContext<"dark" | "light">("Theme");
factory returns a typed [Provider, useHook] pair via as const
Common Mistakes:
❌ Passing a fake default (createContext({} as AuthValue)) so consumers seem safe — a component outside the provider then reads garbage instead of failing.
✅ Default to undefined and throw in the hook; you get both a non-nullable consumer type and a real runtime guard.
❌ Exporting the raw context and calling useContext(AuthContext) directly — every call site re-introduces the | undefined you tried to hide.
✅ Keep the context private; export only the useAuth hook so the narrowing lives in exactly one place.
Key Rules:
  • Type context as T | undefined with an undefined default when the value only exists under a provider.
  • The asserting hook’s throw narrows the return type to T via control-flow analysis.
  • Don’t fake a default value — it trades a compile error for a silent runtime bug.
  • A createSafeContext factory + as const gives a typed [Provider, useHook] pair you can reuse.
Practice Exercises:
1Build a ThemeContext holding { theme: "dark" | "light"; toggle(): void } using the undefined-default + useTheme pattern, and confirm consumers get a non-nullable value with no guard.
2Implement the generic createSafeContext<T>(name) factory and use it to spin up two independent contexts; verify the returned hook return type is exactly T (not T | undefined) with expect-type.
3Extend the auth context to carry a useReducer-based state with a discriminated Action union, exposing state and dispatch through the context type. Confirm dispatch rejects an unknown action type.
Connections: The throw-to-narrow trick is Union Types & Narrowing (R2) and assertion-style narrowing in disguise, and the factory’s as const tuple return is the same idea as the custom-hook pattern from React Hooks Typing (Day 24). Exercise 3 wires in the discriminated-union reducer you’ll formalize in Concept 3.
3. Discriminated Union Props for Component Variants React Patterns
Many components have modes where certain props only make sense together: a button that’s either a link (needs href) or a submit button (needs onClick); an input that’s "text" vs "number" with different value types. A flat props interface with everything optional lets callers pass invalid combinations. A discriminated union of prop shapes makes the illegal states unrepresentable and gives per-variant autocomplete.

Instead of one interface with a pile of optional fields, define the props as a union of the valid shapes, each tagged by a discriminant prop (often variant or kind). React checks the whole object against the union, so a caller who sets variant="link" is required to pass href and is forbidden from passing the submit-only props. Inside the component you narrow on the discriminant exactly like any other discriminated union — a switch or an early if — and each branch sees only its own props.

This is the “make impossible states impossible” principle applied to component APIs. The payoff is real: the union catches <Button variant="link" onClick={...} /> at compile time, and it drives editor autocomplete to show only the props valid for the chosen variant. Pair it with a never default in the internal switch to get exhaustiveness — add a new variant to the union and the component fails to compile until you handle it. Excess-property checking on object literals is what makes the “forbidden extra prop” half work, so keep the shapes tight.

1
Define props as a union of per-variant shapes, each with a shared discriminant prop.
2
Callers get required props per variant and are blocked from mixing another variant’s props.
3
Narrow on the discriminant inside the component; add a never default for exhaustiveness.
TypeScript · example 1 of 4 — the union of prop shapes
type ButtonProps =
  | { variant: "link"; href: string; children: React.ReactNode }
  | { variant: "submit"; onClick: () => void; children: React.ReactNode };

// note: 'href' exists ONLY on the link shape, 'onClick' ONLY on submit
two mutually-exclusive prop shapes tagged by variant
TypeScript · example 2 of 4 — narrowing inside the component
function Button(props: ButtonProps) {
  switch (props.variant) {
    case "link":
      return <a href={props.href}>{props.children}</a>;      // href in scope
    case "submit":
      return <button onClick={props.onClick}>{props.children}</button>;
    default: {
      const _exhaustive: never = props;   // new variant -> compile error here
      return _exhaustive;
    }
  }
}
each case sees only its variant’s props; default enforces exhaustiveness
TypeScript · example 3 of 4 — invalid combinations are rejected
<Button variant="link" href="/home">Home</Button>;        // ok
<Button variant="submit" onClick={save}>Save</Button>;      // ok

<Button variant="link" onClick={save}>Bad</Button>;
// error: 'onClick' does not exist on the link shape, and 'href' is missing

<Button variant="submit" href="/x">Bad</Button>;
// error: 'href' does not exist on the submit shape
Object literal may only specify known properties; 'onClick' not in link variant
TypeScript · example 4 of 4 — a typed input whose value type follows its type prop
type FieldProps =
  | { type: "text"; value: string; onChange: (v: string) => void }
  | { type: "number"; value: number; onChange: (v: number) => void };

function Field(props: FieldProps) {
  if (props.type === "number") {
    // props.value: number, props.onChange: (v: number) => void
    return <input type="number" value={props.value} />;
  }
  // props.value: string here
  return <input value={props.value} />;
}
value/onChange types are linked to the type discriminant
Common Mistakes:
❌ One flat interface with every field optional (href?, onClick?) — it accepts <Button /> with neither, or with both, and pushes validation to runtime.
✅ Model each mode as its own shape in a discriminated union so illegal prop combinations don’t type-check.
❌ Omitting the discriminant and trying to tell variants apart by which optional prop is present — the compiler can’t narrow reliably on “maybe defined” props.
✅ Give every member a shared literal discriminant (variant/type) so a switch narrows cleanly and exhaustiveness works.
Key Rules:
  • Prefer a union of prop shapes over one interface full of optionals when props are mode-specific.
  • Every union member needs a shared literal discriminant to narrow on.
  • Excess-property checks are what forbid another variant’s props on a literal — keep shapes exact.
  • Add a never default in the internal switch to force handling of every variant.
Practice Exercises:
1Type an <Alert> whose props are { severity: "info"; } vs { severity: "error"; onRetry: () => void }, so only the error variant requires onRetry. Prove passing onRetry to an info alert is an error.
2Build a <DataView> with variants loading, error (needs message), and ready (needs data: Row[]). Render each and add a never default; then add a fourth variant and observe the compile error until you handle it.
3Combine this with Concept 1: make a polymorphic <Button> that is either as="a" (requires href) or as="button" (allows onClick), using a discriminated union keyed on as. Note where the two techniques tension with each other.
Connections: This is Union Types & Narrowing (R2) and never & Exhaustiveness applied to props, with Excess Property Checks (Day 23) doing the work of rejecting foreign props. It contrasts with Concept 1: polymorphism swaps the element, discriminated props swap the contract — and Exercise 3 shows they can be combined.
2React + TypeScript: Events & Composition
4. Typing Event Handlers & Synthetic Events React Patterns
Event handling is where beginners reach for any most often — e.target.value “just works” untyped, so people stop there. But React ships precise types for every event, and using them gives you a correctly-typed currentTarget, autocomplete on key/clientX, and handler aliases you can reuse. Knowing the two ways to get the type (inline inference vs an explicit handler alias) removes the guesswork.

React wraps native DOM events in a SyntheticEvent, and provides generic types parameterized by the element: React.ChangeEvent<HTMLInputElement>, React.MouseEvent<HTMLButtonElement>, React.KeyboardEvent<HTMLInputElement>, React.FormEvent<HTMLFormElement>. The generic argument types e.currentTarget (the element the handler is attached to — always the safe one to read; e.target is only EventTarget).

There are two idioms and it’s worth being deliberate. Inline: write the handler directly in JSX (onChange={(e) => ...}) and let React infer e from the prop — no annotation needed and the type is exact. Standalone: when you extract the handler to a named function, you must annotate it, and the cleanest way is the handler-type alias React exports: React.ChangeEventHandler<HTMLInputElement> (or MouseEventHandler, KeyboardEventHandler, …). Annotate the variable with the handler type and the parameter is inferred for you. Prefer inference inline; prefer the handler-type alias when hoisting.

1
React events are generic over the element: ChangeEvent<HTMLInputElement>, MouseEvent<HTMLButtonElement>, …
2
Read e.currentTarget (typed to your element), not e.target (just EventTarget).
3
Inline handlers infer e; extracted handlers use a ...EventHandler<El> alias so the param is inferred.
TypeScript · example 1 of 4 — inline inference (no annotation needed)
function SearchBox() {
  const [q, setQ] = useState("");
  return (
    <input
      value={q}
      onChange={(e) => setQ(e.currentTarget.value)}   // e inferred: ChangeEvent<HTMLInputElement>
      onKeyDown={(e) => { if (e.key === "Enter") submit(); }}
    />
  );
}
e is fully typed from the prop; currentTarget is HTMLInputElement
TypeScript · example 2 of 4 — extracted handler via the handler-type alias
// annotate the VARIABLE with the handler type; param is inferred
const onChange: React.ChangeEventHandler<HTMLInputElement> = (e) => {
  console.log(e.currentTarget.value);   // e.currentTarget: HTMLInputElement
};

const onSubmit: React.FormEventHandler<HTMLFormElement> = (e) => {
  e.preventDefault();
  const data = new FormData(e.currentTarget);   // currentTarget: HTMLFormElement
};
handler-type alias infers the event param for you
TypeScript · example 3 of 4 — explicit event type when you annotate the param directly
function handleClick(e: React.MouseEvent<HTMLButtonElement>) {
  console.log(e.clientX, e.currentTarget.disabled);   // both typed
}

// target vs currentTarget:
function onInput(e: React.ChangeEvent<HTMLInputElement>) {
  e.currentTarget.value;   // string — safe
  // e.target is EventTarget & HTMLInputElement here for ChangeEvent specifically,
  // but for many events target is only EventTarget — prefer currentTarget
}
MouseEvent exposes clientX; currentTarget is the attached element
TypeScript · example 4 of 4 — a reusable typed handler prop on your own component
type ToggleProps = {
  checked: boolean;
  onToggle: (next: boolean, e: React.ChangeEvent<HTMLInputElement>) => void;
};

function Toggle({ checked, onToggle }: ToggleProps) {
  return (
    <input
      type="checkbox"
      checked={checked}
      onChange={(e) => onToggle(e.currentTarget.checked, e)}
    />
  );
}
your component can expose semantic + raw-event params, both typed
Common Mistakes:
❌ Annotating an inline handler param as (e: any) “to move on” — you throw away the exact event type React already inferred for free.
✅ Leave inline handlers un-annotated (React infers e); only annotate when you extract the function.
❌ Reading e.target.value on a generic event and fighting a “value does not exist on EventTarget” error, then casting.
✅ Use e.currentTarget, which the generic event type points at your specific element — no cast needed.
Key Rules:
  • React events are generic over the element type — always pass the element (<HTMLInputElement>, …).
  • Prefer e.currentTarget; e.target is typed as the wider EventTarget for most events.
  • Inline handlers infer their event — don’t annotate them; extracted handlers should use a ...EventHandler<El> alias.
  • Annotate the handler variable (not the parameter) with the handler-type alias to get parameter inference.
Practice Exercises:
1Write a controlled <form> with an extracted onSubmit: React.FormEventHandler<HTMLFormElement> that calls e.preventDefault() and reads a field via new FormData(e.currentTarget). Confirm currentTarget is HTMLFormElement.
2Type a <select> onChange both ways — once inline (inferred) and once as an extracted ChangeEventHandler<HTMLSelectElement> — and show both give e.currentTarget.value: string.
3Define a component that accepts an onKeyCommand prop typed as (key: string, e: React.KeyboardEvent) => void, wire it to an input’s onKeyDown, and forward e.key plus the raw event. Verify a caller can’t pass a handler expecting a MouseEvent.
Connections: The handler-type aliases are function-type Type Annotations (R1), and the target/currentTarget distinction is a small Narrowing story (R2). Passing typed event callbacks as props ties back to React Component Props (Day 24) and sets up render props (Concept 5), where the child is a typed function.
5. Render Props & Function-as-Children React Patterns
Before hooks, render props were the way to share stateful logic; they’re still the cleanest pattern when a component needs to own state or a resource and let the caller decide how to render it (a <Fetcher>, a <List> that renders each item, a headless dropdown). Typing them well — especially the generic “function-as-children” form — means the render callback’s argument is fully inferred from the component’s own generic.

A render prop is just a prop whose value is a function returning JSX. Two shapes exist: a named prop (render={(x) => ...}) and the special children-as-function (<C>{(x) => ...}</C>). To type either, declare the prop as a function returning React.ReactNode: render: (value: T) => React.ReactNode. When the component is generic over T, the callback’s parameter is inferred at the call site from whatever the component produces.

The function-as-children form has one wrinkle: children is normally React.ReactNode, but here you want it to be a function, so you type children explicitly as (value: T) => React.ReactNode — overriding the default. Make the component a generic function component and TypeScript threads T from the data you pass (or fetch) straight into the render callback’s parameter, giving the caller full autocomplete on item/data without any annotation. This is the headless-component pattern: the component owns behavior, the caller owns markup, and the types keep the seam honest.

1
A render prop is a prop typed as (value: T) => React.ReactNode (named) or the same on children.
2
Make the component generic over T so the callback’s argument is inferred at the call site.
3
For function-as-children, override children’s type to a function (it isn’t ReactNode anymore).
TypeScript · example 1 of 4 — a named render prop, generic over the item
type ListProps<T> = {
  items: T[];
  render: (item: T, index: number) => React.ReactNode;
};

function List<T>({ items, render }: ListProps<T>) {
  return <ul>{items.map((it, i) => <li key={i}>{render(it, i)}</li>)}</ul>;
}

// item is inferred as User — no annotation at the call site
<List items={users} render={(u) => <span>{u.name}</span>} />;
render’s item param is inferred from items (User)
TypeScript · example 2 of 4 — function-as-children (children is a function)
type FetcherProps<T> = {
  url: string;
  children: (state: { data: T | null; loading: boolean }) => React.ReactNode;
};

function Fetcher<T>({ url, children }: FetcherProps<T>) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  // ... effect that fetches url, setData, setLoading(false) ...
  return <>{children({ data, loading })}</>;
}
children is typed as a function, not the usual ReactNode
TypeScript · example 3 of 4 — the call site infers everything
<Fetcher<User> url="/me">
  {({ data, loading }) =>                 // data: User | null, loading: boolean
    loading ? <p>Loading…</p> : <p>{data?.name}</p>
  }
</Fetcher>;

// explicit <User> when it can't be inferred from props; otherwise omit it
render callback’s state param is fully typed from T = User
TypeScript · example 4 of 4 — a headless toggle exposing state via render prop
type ToggleRenderProps = {
  children: (api: { on: boolean; toggle: () => void }) => React.ReactNode;
};

function Toggle({ children }: ToggleRenderProps) {
  const [on, setOn] = useState(false);
  return <>{children({ on, toggle: () => setOn((v) => !v) })}</>;
}

<Toggle>
  {({ on, toggle }) =>                     // on: boolean, toggle: () => void
    <button onClick={toggle}>{on ? "ON" : "OFF"}</button>}
</Toggle>;
the caller owns markup; the component owns behavior; types stay tight
Common Mistakes:
❌ Leaving children as the default React.ReactNode while passing a function — TypeScript flags the function as an invalid child.
✅ Explicitly type children as (value: T) => React.ReactNode when you use function-as-children.
❌ Hard-coding the render argument type (e.g. render: (item: unknown) => ...) so callers must cast item to use it.
✅ Make the component generic over T so the callback’s parameter is inferred from items/data with zero casts.
Key Rules:
  • A render prop is a function prop returning React.ReactNode; function-as-children puts it on children.
  • Override children’s type when it’s a function — the default ReactNode won’t accept one.
  • Make the component generic over T so the callback argument is inferred at the call site.
  • Great for headless components (own behavior, delegate markup); hooks often replace the logic-sharing use case.
Practice Exercises:
1Write a generic <For items render /> component and confirm the item argument is inferred (no annotation) for both a number[] and a User[].
2Build a function-as-children <Mouse> that tracks { x: number; y: number } and passes it to children. Type children correctly and prove the caller’s destructured { x, y } are numbers.
3Convert the generic <Fetcher<T>> to expose a discriminated state ({ status: "loading" } | { status: "error"; error: Error } | { status: "ok"; data: T }) through the render callback, so the caller must narrow on status before reading data.
Connections: Render props are Generics Basics (R5) plus function-typed props (R1), and Exercise 3 folds in the Discriminated Union Props idea from Concept 3 applied to the callback’s argument. This closes the component-composition arc: polymorphism (element), discriminated props (contract), and render props (delegated rendering) are the three levers for flexible, fully-typed component APIs.
