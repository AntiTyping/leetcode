R1. Module Federation Typing Advanced Patterns & Performance due · rep 1 → 2
Module Federation loads code from another build at runtime, so the consuming compiler has no .d.ts for it by default. You restore type safety by sharing a hand-written or generated declaration for each remote (declare module "remoteApp/Widget") — the boundary is an untyped runtime fetch you must describe, not trust.
Key Pattern 
// remotes.d.ts — describe what the host imports from each remote
declare module "checkout/CartButton" {
  import type { ComponentType } from "react";
  const CartButton: ComponentType<{ sku: string }>;
  export default CartButton;
}

// host code — now fully typed despite the runtime federation
import CartButton from "checkout/CartButton";   // ComponentType<{ sku: string }>
<CartButton sku="abc" />;        // sku checked; missing prop is an error
remote import is typed from the ambient module declaration
Challenge Exercise

Set up a host that consumes analytics/track (a function (event: string, props: Record<string, unknown>) => void) from a remote. Write the ambient declare module for it, import and call it, and confirm a wrong argument type is a compile error. Then describe how you would auto-generate these declarations (e.g. the @module-federation/typescript plugin) instead of hand-writing them.

Pitfall: Letting remote imports silently resolve to any because no declaration exists — you lose every guarantee at exactly the boundary where versions can skew between independently-deployed builds. Generate the remote .d.ts as part of the remote’s build and publish it, so the host’s types track the actually-deployed contract rather than a guess.
R2. Testing Types: tsd & expect-type Advanced Patterns & Performance due · rep 1 → 2
Type-level code deserves tests too. expectTypeOf (from expect-type/Vitest) and expectType (from tsd) assert relationships at compile time — the “test” passing means tsc reported no error. Crucially these assert exact equality, which catches accidental widening that a plain assignability check would miss.
Key Pattern 
import { expectTypeOf } from "expect-type";

type Result = ReturnType<typeof parse>;

expectTypeOf<Result>().toEqualTypeOf<{ id: number; name: string }>();
expectTypeOf<Result>().not.toEqualTypeOf<{ id: number }>();
expectTypeOf(parse).parameter(0).toBeString();

// tsd flavor
import { expectType, expectError } from "tsd";
expectType<number>(add(1, 2));
expectError(add("x", 2));   // asserts this line is itself a type error
assertions fail the build (not at runtime) when the types drift
Challenge Exercise

Write a tiny Prettify<T> utility and a type test that uses toEqualTypeOf to prove Prettify<{ a: 1 } & { b: 2 }> equals { a: 1; b: 2 } exactly. Then add an assertion with toMatchTypeOf and explain in a comment how it differs from toEqualTypeOf (assignable-to vs exactly-equal), and why the exact form catches a bug the loose form lets through.

Pitfall: Reaching for toMatchTypeOf (assignability) when you mean toEqualTypeOf (exact equality) — { a: 1; b: 2 } matches { a: 1 }, so a widening regression slips past. Also remember type tests only run under tsc; a green runtime test suite tells you nothing about them, so wire the type check into CI.
R3. React + TypeScript: Hooks Typing React + TypeScript due · rep 1 → 2
useState usually infers, but an initial null or an empty array needs an explicit type argument or it widens wrongly. useReducer is typed by a discriminated-union action, useRef splits into a mutable box vs a read-only DOM ref, and a custom hook returning a pair should use as const so the tuple doesn’t widen to an array.
Key Pattern 
const [user, setUser] = useState<User | null>(null);   // else inferred null

// DOM ref: pass null, type the element — ref.current is T | null
const inputRef = useRef<HTMLInputElement>(null);

// mutable instance box: give it a value — current is mutable T
const id = useRef<number>(0);

// custom hook: as const keeps the tuple positions distinct
function useToggle(init = false) {
  const [on, setOn] = useState(init);
  return [on, () => setOn((v) => !v)] as const;   // [boolean, () => void]
}
explicit args + as const keep hook types precise
Challenge Exercise

Then write a generic useFetch<T>(url): { data: T | null; loading: boolean } and confirm data narrows to T after a loading guard.

Pitfall: Calling useState(null) or useState([]) and letting it infer null / never[], then fighting the compiler on every later setState. Supply the type argument up front. And don’t return a bare [a, b] from a custom hook without as const — it widens to (A | B)[] and destructuring loses the per-position types.
R4. Redux Toolkit TypeScript Patterns State Management & Architecture due · rep 1 → 2
RTK’s whole TS story is “infer once, export the inferred types.” createSlice derives action creators and the state shape; you compute RootState and AppDispatch from the store and build pre-typed useAppSelector/useAppDispatch hooks so components never re-annotate. createAsyncThunk takes generics for its return and arg types.
Key Pattern 
const slice = createSlice({
  name: "counter",
  initialState: { value: 0 },
  reducers: {
    add: (s, a: PayloadAction<number>) => { s.value += a.payload; },
  },
});

export const store = configureStore({ reducer: { counter: slice.reducer } });

export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;

// pre-typed hooks — defined once, used everywhere
export const useAppDispatch: () => AppDispatch = useDispatch;
export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;
RootState/AppDispatch derived from the store; hooks typed once
Challenge Exercise

R5. Using Declarations & Explicit Resource Management TypeScript 5.x due · rep 0 → 1
TS 5.2’s using binds a resource whose [Symbol.dispose]() runs automatically when the enclosing block exits — deterministic cleanup without try/finally. await using calls the async [Symbol.asyncDispose](). Disposers run in reverse order (LIFO), like a stack of finally blocks.
Key Pattern 
function openFile(path: string): Disposable & { read(): string } {
  const handle = acquire(path);
  return {
    read: () => handle.contents,
    [Symbol.dispose]() { handle.close(); },   // runs at block end
  };
}

{
  using f = openFile("a.txt");
  console.log(f.read());
}   // f[Symbol.dispose]() called here, even on throw

async function load() {
  await using conn = await openConn();   // [Symbol.asyncDispose] on exit
}
dispose runs at scope exit, in reverse declaration order
Challenge Exercise

Write a lock(mutex) that returns a Disposable releasing the mutex in [Symbol.dispose], and use it with using so the lock is always released on every exit path including a thrown error. Then declare two resources with using in one block and add a console.log in each disposer to prove they fire in LIFO order. Note which compilerOptions.lib/polyfill you need for the Symbol.dispose global.

Pitfall: Expecting using cleanup to run when the function returns a value out of the resource without the block ending — disposal is tied to block scope exit, not to the variable going unused. Also, Symbol.dispose/asyncDispose need the right lib (or a polyfill); without it the symbols are missing at runtime even though the syntax compiles.
R6. Decorator Metadata TypeScript 5.x due · rep 0 → 1
TS 5.2 implements the TC39 Symbol.metadata proposal: a standard decorator receives a context object with a metadata record it can write to, and the accumulated metadata is attached to the class at SomeClass[Symbol.metadata]. This is the modern, framework-agnostic replacement for the old reflect-metadata + emitDecoratorMetadata approach.
Key Pattern 
function logged(_value: unknown, context: ClassMethodDecoratorContext) {
  // write to the shared metadata bag for this class
  (context.metadata.logged ??= []).push(context.name);
}

class Service {
  @logged greet() {}
  @logged save() {}
}

// read it back off the class via the well-known symbol
const meta = Service[Symbol.metadata];
console.log(meta?.logged);   // ["greet", "save"]
["greet", "save"]
Challenge Exercise

1Conditional & Inference Mastery
1. Distributive vs Non-Distributive Conditional Types Conditional Types
This one rule explains a surprising amount of conditional-type behavior: why Exclude works, why a conditional “maps over” a union member-by-member, why a stray never disappears, and why wrapping a type in [ ] suddenly changes the answer. Once you can see distribution, advanced type code stops being magic.

When the checked type of a conditional T extends U ? X : Y is a naked type parameter (just T, not wrapped), and the type argument is a union, the conditional distributes: it is applied to each member separately and the results are unioned. So T extends U ? X : Y with T = A | B becomes (A extends U ? X : Y) | (B extends U ? X : Y).

Two consequences follow. First, never is the empty union — distributing over it produces never (zero members to union), which is why ToArray<never> is never, not never[]. Second, to stop distribution you make the checked type non-naked, conventionally by wrapping both sides in a one-tuple: [T] extends [U] ? X : Y. Now the union is compared as a whole. This is exactly how you write a correct IsNever check and how you compare two unions for equality.

1
A naked type parameter over a union distributes member-by-member, then unions the results.
2
Distributing over never yields never (it is the empty union).
3
Wrap in tuples — [T] extends [U] — to compare the union as a single unit.
TypeScript · example 1 of 4 — distribution in action
type ToArray<T> = T extends unknown ? T[] : never;   // T is naked

type A = ToArray<string | number>;
// distributes: (string extends unknown ? string[] : never)
//            | (number extends unknown ? number[] : never)
//            = string[] | number[]
A = string[] | number[] (NOT (string | number)[])
TypeScript · example 2 of 4 — turning distribution OFF with tuples
type ToArrayNonDist<T> = [T] extends [unknown] ? T[] : never;   // wrapped

type B = ToArrayNonDist<string | number>;   // (string | number)[]

// and the empty-union edge case:
type C = ToArray<never>;            // never  (distributing over 0 members)
type D = ToArrayNonDist<never>;     // never[]  (never compared as a whole)
B = (string | number)[]; C = never; D = never[]
TypeScript · example 3 of 4 — the built-ins rely on distribution
// Exclude / Extract / NonNullable are all distributive conditionals
type MyExclude<T, U> = T extends U ? never : T;

type E = MyExclude<"a" | "b" | "c", "b">;
// ("a" extends "b" ? never : "a")  => "a"
// ("b" extends "b" ? never : "b")  => never  (drops out of the union)
// ("c" extends "b" ? never : "c")  => "c"
// = "a" | "c"
E = "a" | "c" — matched members collapse to never and vanish
TypeScript · example 4 of 4 — a correct IsNever (needs the tuple trick)
// WRONG: distributes, so over `never` there are no members -> never (not false)
type IsNeverBad<T> = T extends never ? true : false;
type W = IsNeverBad<never>;   // never  (oops, not the boolean you wanted)

// RIGHT: wrap to compare the whole type
type IsNever<T> = [T] extends [never] ? true : false;
type X1 = IsNever<never>;      // true
type X2 = IsNever<string>;     // false
IsNeverBad<never> = never; IsNever<never> = true
Common Mistakes:
❌ Expecting T extends ... ? T[] : never to give (A | B)[] — a naked T distributes, so you actually get A[] | B[].
✅ Wrap in tuples ([T] extends [U]) when you want the union treated as one type rather than mapped member-by-member.
❌ Writing T extends never ? ... to detect never — it distributes over the empty union and the conditional never even evaluates.
✅ Use [T] extends [never] for an IsNever guard, and remember a stray never silently disappears from any distributed result.
Key Rules:
  • 2026-07-04 (Day 31)
  • Distribution happens only when the checked type is a naked type parameter and the argument is a union.
  • Distributing over never (the empty union) produces never.
  • Wrap both sides in a 1-tuple ([T] extends [U]) to disable distribution and compare the whole union.
  • Exclude, Extract, NonNullable are distributive; IsNever/union-equality checks must be non-distributive.

Connections: Distribution is the engine behind Conditional Types and the Exclude/Extract utilities, and the “collapses to never” behavior connects to never & Exhaustiveness. The tuple-wrapping trick reappears in the UnionToIntersection recipe (Day 21) and in the Equals helper used by Testing Types (R2).
2. infer ... extends Constraints Inference
Plain infer R captures whatever’s there as the widest possible type, forcing you to nest another conditional to narrow or re-check it. TS 4.7’s infer R extends C folds that check into the inference itself — cleaner code, and (4.8+) it can even coerce a string-literal capture into a number/boolean literal, which is the key to type-level parsing.

infer R extends C places a constraint directly on an inferred type variable. Two things change versus bare infer R. First, the inferred type is narrowed to C at the point of capture, so you can use it as a C immediately without a second guard. Second, since TypeScript 4.8, when C is a primitive like number or boolean and the matched type is a string literal, the compiler attempts to parse/coerce it — infer N extends number against "42" yields the literal type 42.

If the matched type does not satisfy the constraint, the conditional takes its false branch (this is the modern behavior; very old versions silently fell back to the constraint). That makes infer ... extends a precise, self-validating extraction. It shines in tuple destructuring (infer F extends string), in unwrapping return types under a bound, and above all in template-literal type parsing where you need to turn captured text into a real numeric/boolean literal.

1
infer R extends C narrows the capture to C so no second conditional is needed.
2
Against a primitive constraint, a string-literal capture is coerced to a number/boolean literal.
3
If the match fails the constraint, the conditional’s false branch is taken.
TypeScript · example 1 of 4 — bare infer vs constrained infer
// bare: F is unknown-ish, needs a second check to use as a string
type FirstLoose<T> = T extends [infer F, ...unknown[]] ? F : never;
type A = FirstLoose<[1, 2, 3]>;     // 1  (but F's type is just `unknown` shape)

// constrained: only matches when the first element is a string, F is string
type FirstString<T> =
  T extends [infer F extends string, ...unknown[]] ? F : "no string head";
type B = FirstString<["hi", 2]>;   // "hi"
type C = FirstString<[1, 2]>;      // "no string head"  (false branch)
B = "hi"; C = "no string head"
TypeScript · example 2 of 4 — coercing a string literal to a number literal
// 4.8+: "extends number" parses the captured text into a numeric literal
type ToNum<S extends string> = S extends `${infer N extends number}` ? N : never;

type X = ToNum<"42">;    // 42   (the literal type, not number)
type Y = ToNum<"3.5">;   // 3.5
type Z = ToNum<"abc">;   // never (doesn't parse as a number)
X = 42; Y = 3.5; Z = never
TypeScript · example 3 of 4 — parsing structured text without nested conditionals
// extract a route param's numeric id directly in the pattern
type IdOf<S extends string> =
  S extends `/user/${infer Id extends number}` ? Id : never;

type R1 = IdOf<"/user/7">;     // 7
type R2 = IdOf<"/user/abc">;   // never  — constraint failed, false branch

// also works for booleans
type ToBool<S extends string> = S extends `${infer B extends boolean}` ? B : never;
type T1 = ToBool<"true">;      // true
R1 = 7; R2 = never; T1 = true
TypeScript · example 4 of 4 — constrained unwrap of a return type
// only unwrap functions that return an array, and keep the element type tight
type ElemOfReturn<F> =
  F extends (...args: never[]) => (infer R extends readonly unknown[])
    ? R[number]
    : never;

type Q = ElemOfReturn<() => string[]>;     // string
type W = ElemOfReturn<() => number>;       // never  (return isn't an array)
Q = string; W = never
Common Mistakes:
❌ Capturing with bare infer N and then nesting N extends number ? ... to re-check — verbose and easy to get wrong.
✅ Fold the bound into the capture: infer N extends number narrows in one step.
❌ Expecting `${infer N extends number}` to give you a runtime number — it only produces a type-level literal; there’s no value conversion at runtime.
✅ Use it purely for type-level parsing/validation, and pair it with a real runtime parser (or Zod) when you need an actual value.
Key Rules:
  • infer R extends C narrows the captured type to C in a single step.
  • A failed constraint sends the conditional to its false branch (modern TS behavior).
  • Against number/boolean constraints, string-literal captures are coerced to literal types (4.8+).
  • It’s a compile-time construct — no runtime value conversion happens.

The false-branch-on-failure behavior leans on the conditional semantics from Concept 1, and the recursive parsing exercise previews the tuple recursion of Concept 4.
3. Recursive Awaited<T> & Promise Unwrapping Async Types
At runtime, await flattens nested promises and follows thenables — await Promise.resolve(Promise.resolve(1)) is 1, not a promise. The built-in Awaited<T> models exactly that recursion, which is why ReturnType of an async function alone isn’t enough and why understanding its shape lets you write correct async-aware utilities.

Awaited<T> (built in since TS 4.5) is a recursive conditional type that mirrors await: if T is null/undefined it passes through; if T is a thenable (has a then method whose first callback receives a value), it infers that value and recurses on it; otherwise T is returned as-is. That recursion is what unwraps Promise<Promise<X>> all the way down to X.

The everyday combination is Awaited<ReturnType<typeof asyncFn>>: ReturnType gives you Promise<X>, and Awaited peels it to X. It also distributes over unions, so Awaited<Promise<A> | B> is A | B. Knowing the definition explains the edge cases: a non-promise stays untouched, a union of resolved types is preserved, and a custom thenable is followed as long as its then signature matches. You rarely re-implement it, but you constantly compose it.

1
Awaited<T> recursively unwraps nested promises/thenables, like runtime await.
2
Compose Awaited<ReturnType<typeof fn>> to get an async function’s resolved type.
3
It distributes over unions and leaves non-promise members unchanged.
TypeScript · example 1 of 4 — recursive unwrapping
type A = Awaited<Promise<string>>;                     // string
type B = Awaited<Promise<Promise<number>>>;            // number (peeled twice)
type C = Awaited<Promise<Promise<Promise<boolean>>>>;   // boolean
type D = Awaited<number>;                              // number (non-promise: unchanged)
A = string; B = number; C = boolean; D = number
TypeScript · example 2 of 4 — the resolved type of an async function
async function loadUser(id: string) {
  return { id, name: "Ada" };
}

type Raw      = ReturnType<typeof loadUser>;          // Promise<{ id: string; name: string }>
type Resolved = Awaited<ReturnType<typeof loadUser>>; // { id: string; name: string }

// handy alias for "what an async function resolves to"
type AsyncReturn<F extends (...a: never[]) => unknown> = Awaited<ReturnType<F>>;
Resolved = { id: string; name: string }
TypeScript · example 3 of 4 — unions, Promise.all, and a thenable
type U = Awaited<Promise<number> | string>;   // number | string (distributes)

// Promise.all returns Promise<[...Awaited<each>]>
const all = Promise.all([Promise.resolve(1), Promise.resolve("x")]);
type AllResolved = Awaited<typeof all>;       // [number, string]

// custom thenable is followed too
interface MyThenable { then(cb: (v: Date) => void): void }
type T = Awaited<MyThenable>;                  // Date
U = number | string; AllResolved = [number, string]; T = Date
TypeScript · example 4 of 4 — a hand-rolled DeepAwaited to see the recursion
// illustrative — the real built-in also handles null/undefined & thenables
type DeepAwaited<T> =
  T extends Promise<infer Inner> ? DeepAwaited<Inner> : T;

type X = DeepAwaited<Promise<Promise<string>>>;   // string
type Y = DeepAwaited<string>;                      // string

// prefer the built-in Awaited in real code — it covers more edge cases
type Z = Awaited<Promise<Promise<string>>>;         // string
X = string; Y = string; Z = string
Common Mistakes:
❌ Using ReturnType<typeof asyncFn> as the resolved value — that’s the Promise<X>, not X, so downstream code accidentally types around a promise.
✅ Wrap it: Awaited<ReturnType<typeof asyncFn>> to get the actually-resolved type.
❌ Re-implementing promise unwrapping with a single non-recursive conditional, which stops after one layer and leaves Promise<Promise<X>> as Promise<X>.
✅ Use the built-in Awaited<T> — it recurses and also handles thenables and null/undefined correctly.
Key Rules:
  • Awaited<T> recursively unwraps nested promises/thenables, matching runtime await.
  • Awaited<ReturnType<typeof fn>> is the idiom for an async function’s resolved type.
  • It distributes over unions and leaves non-promise types unchanged.
  • Promise.all already produces Awaited-ed element types in its resolved tuple.

The mapped-type exercise foreshadows Concept 5’s homomorphic mapping over an object of loaders.
2Type-Level Computation
4. Type-Level Arithmetic with Tuple Lengths Recursion
TypeScript’s type system has no + operator, yet libraries do real arithmetic at the type level — fixed-length tuples, indexed path types, compile-time bounds. The trick is that a tuple’s length is a numeric literal type, so you compute by building and destructuring tuples. It’s a beautiful demonstration of how far recursion + inference can go.

The whole technique rests on one fact: for a tuple T, T["length"] is a literal number type (e.g. [unknown, unknown]["length"] is 2). So you represent the number N as a tuple of length N, do the operation by manipulating tuples, then read ["length"] to get the numeric literal back.

Addition is tuple concatenation: [...TupleOf<A>, ...TupleOf<B>]["length"]. Subtraction uses inference to strip a prefix: if TupleOf<A> matches [...TupleOf<B>, ...infer Rest], then Rest["length"] is A - B. TupleOf<N> itself is built by recursively pushing elements until the length reaches N. The catch is practical: this is genuinely recursive type instantiation, and TypeScript caps recursion depth (tail-recursive conditionals allow ~1000 iterations, but deep or naive recursion hits an instantiation-depth error). So tuple arithmetic is great for small numbers (array bounds, fixed sizes) and the wrong tool for large-number math.

1
Represent a number N as a tuple of length N; read it back with ["length"].
2
Add by concatenating tuples; subtract by inferring the leftover prefix.
3
Recursion depth is bounded — use it for small numbers, not big arithmetic.
TypeScript · example 1 of 4 — length is a literal; build a tuple of length N
type Len = [unknown, unknown, unknown]["length"];   // 3  (a literal type)

// build a tuple of length N by recursion
type TupleOf<N extends number, Acc extends unknown[] = []> =
  Acc["length"] extends N ? Acc : TupleOf<N, [...Acc, unknown]>;

type Three = TupleOf<3>;   // [unknown, unknown, unknown]
Len = 3; Three = [unknown, unknown, unknown]
TypeScript · example 2 of 4 — addition = concatenation
type Add<A extends number, B extends number> =
  [...TupleOf<A>, ...TupleOf<B>]["length"];

type S1 = Add<2, 3>;    // 5
type S2 = Add<10, 4>;   // 14
type S3 = Add<0, 7>;    // 7
S1 = 5; S2 = 14; S3 = 7
TypeScript · example 3 of 4 — subtraction via prefix inference
type Subtract<A extends number, B extends number> =
  TupleOf<A> extends [...TupleOf<B>, ...infer Rest] ? Rest["length"] : never;

type D1 = Subtract<5, 2>;   // 3
type D2 = Subtract<9, 9>;   // 0
type D3 = Subtract<3, 5>;   // never  (can't strip a longer prefix)
D1 = 3; D2 = 0; D3 = never
TypeScript · example 4 of 4 — a comparison built from subtraction (and the depth limit)
// "is A >= B?" — subtraction is never only when B > A
type GTE<A extends number, B extends number> =
  Subtract<A, B> extends never ? false : true;

type C1 = GTE<5, 3>;    // true
type C2 = GTE<2, 8>;    // false

// type Big = Add<900, 900>;   // risks "Type instantiation is excessively deep"
C1 = true; C2 = false (keep operands small to avoid depth errors)
Common Mistakes:
❌ Trying to do large-number arithmetic (hundreds/thousands) this way — you hit “Type instantiation is excessively deep and possibly infinite.”
✅ Reserve tuple arithmetic for small, bounded values (array sizes, fixed lengths); for big math, carry the number as a value, not a type.
❌ Forgetting the accumulator’s base case, so TupleOf recurses forever or never terminates for non-literal number.
✅ Always anchor the recursion on Acc["length"] extends N, and constrain inputs to literal number types (not the wide number).
Key Rules:
  • A tuple’s ["length"] is a numeric literal type — the basis of all type-level arithmetic.
  • Add by concatenating tuples; subtract by inferring the remaining prefix with ...infer Rest.
  • Build TupleOf<N> by recursive concatenation with a base case on the accumulator length.
  • Recursion depth is limited — this works for small numbers, not large-scale computation.
Practice Exercises:
1Implement Multiply<A, B> by repeated addition (recurse B times accumulating Add), and test Multiply<3, 4> equals 12.
2Write Range<N> producing the tuple [0, 1, ..., N-1] of number literals, then index it to prove Range<3>[2] is 2.
3Build a Repeat<S extends string, N extends number> that produces "ababab" for Repeat<"ab", 3>, combining tuple recursion with template-literal concatenation.
Connections: This is recursion + infer (Concept 2) applied to tuples, and the prefix-inference trick is the same one behind Variadic Tuple Types. The depth caveat ties to TypeScript Performance, and Range/indexing previews how mapped types iterate (Concept 5).
5. Mapped Type Modifiers (+/-) & Homomorphic Behavior Mapped Types
Mapped types don’t just rename keys — they can add or remove the readonly and ? modifiers, which is exactly how Partial, Required, Readonly, and Mutable are built. And whether a mapped type is homomorphic decides whether it preserves the original modifiers (and array/tuple shape) or flattens everything — the source of many “why did my optionality vanish?” surprises.

Inside a mapped type you can prefix readonly and ? with + (add, the default) or - (remove). So { -readonly [K in keyof T]: T[K] } is Mutable<T>, { [K in keyof T]-?: T[K] } is Required<T>, { readonly [K in keyof T]: T[K] } is Readonly<T>, and { [K in keyof T]?: T[K] } is Partial<T>. Removing optionality with -? also strips the | undefined that optionality implies.

A mapped type is homomorphic when it maps directly over keyof T (the form { [K in keyof T]: ... }). Homomorphic maps have two superpowers: they preserve the existing readonly/? modifiers of T unless you explicitly change them, and they preserve array and tuple-ness (mapping over an array yields an array, not a plain object). A non-homomorphic map — one that iterates an arbitrary key union like { [K in SomeUnion]: ... } — loses both: modifiers are not carried over and arrays collapse. This is why Partial<readonly string[]> stays an array but a hand-rolled map with key remapping may not.

1
+/- before readonly and ? add or remove those modifiers (+ is the default).
2
Homomorphic maps ([K in keyof T]) preserve T’s modifiers and array/tuple shape.
3
Mapping over an arbitrary union (or remapping keys) is non-homomorphic and drops both.
TypeScript · example 1 of 4 — the four standard utilities, by hand
type MyPartial<T>  = { [K in keyof T]?: T[K] };          // adds ?
type MyRequired<T> = { [K in keyof T]-?: T[K] };         // removes ? (and | undefined)
type MyReadonly<T> = { readonly [K in keyof T]: T[K] };  // adds readonly
type Mutable<T>    = { -readonly [K in keyof T]: T[K] }; // removes readonly

interface U { readonly id: number; name?: string }
type M = Mutable<U>;     // { id: number; name?: string }   (readonly gone)
type R = MyRequired<U>;  // { readonly id: number; name: string }  (? gone)
M drops readonly; R drops optionality (name: string, no undefined)
TypeScript · example 2 of 4 — -? also removes the implicit | undefined
interface Form { age?: number }   // age: number | undefined

type Loose  = Form["age"];                       // number | undefined
type Tight  = MyRequired<Form>["age"];           // number  (-? stripped undefined)

// '+' is the explicit, default-meaning form (rarely written)
type Add = { [K in keyof Form]+?: Form[K] };      // same as MyPartial<Form>
Loose = number | undefined; Tight = number
TypeScript · example 3 of 4 — homomorphic maps preserve modifiers & arrays
interface P { readonly id: number; tag?: string }

// homomorphic: [K in keyof P] — existing readonly/? are KEPT
type Same = { [K in keyof P]: P[K] };
// Same = { readonly id: number; tag?: string }   (modifiers preserved)

// arrays/tuples survive a homomorphic map
type PartialArr = Partial<readonly number[]>;     // readonly (number | undefined)[]
type RTup = { [K in keyof [string, number]]: [string, number][K] };  // still [string, number]
Same keeps readonly & ?; mapping an array yields an array
TypeScript · example 4 of 4 — non-homomorphic loses modifiers
interface P { readonly id: number; tag?: string }

// iterate an arbitrary key union -> NON-homomorphic: modifiers NOT carried
type Flat = { [K in keyof P & string]: P[K] };
// Flat = { id: number; tag: string | undefined }   (readonly & ? both dropped)

// key remapping with `as` is also non-homomorphic
type Renamed = { [K in keyof P as `_${string & K}`]: P[K] };
// { _id: number; _tag: string | undefined }  (modifiers not preserved)
Flat / Renamed drop readonly and optionality (non-homomorphic)
Common Mistakes:
❌ Assuming Required<T> only flips ? off — -? also removes the | undefined that optionality carried, changing the property’s value type.
✅ Expect { x?: number } → { x: number } (no undefined); use a separate step if you truly want to keep undefined.
❌ Adding key remapping or iterating keyof T & string and being surprised that readonly/? disappeared and arrays became plain objects.
✅ Keep the map homomorphic ([K in keyof T]) when you need to preserve modifiers and array/tuple shape; reach for remapping only when you intend to reshape.
Key Rules:
  • +/- before readonly and ? add/remove those modifiers; + is the implicit default.
  • -? removes optionality and the implied | undefined from the value type.
  • Homomorphic maps ([K in keyof T]) preserve T’s modifiers and its array/tuple shape.
  • Mapping an arbitrary key union or using as key remapping is non-homomorphic and drops both.
Practice Exercises:
1Write DeepReadonly<T> that recursively adds readonly and verify it preserves optionality on nested objects (homomorphic at each level).
2Write RequiredKeepUndefined<T> that makes properties required but keeps | undefined in the value type, and explain why plain -? can’t do this.
3Demonstrate the homomorphic difference: map readonly [1, 2, 3] with [K in keyof T] vs a remapped [K in keyof T as ...] and show with expect-type which one stays a tuple.
