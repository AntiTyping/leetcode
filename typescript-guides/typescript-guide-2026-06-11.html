R1. Utility Types Generic & Utility Types rep 2 · last reviewed 2026-06-01 · overdue 4 days
TypeScript ships with built-in utility types that transform existing types: Partial<T> makes all properties optional, Required<T> makes them all required, Pick<T, K> selects a subset, Omit<T, K> removes keys, Record<K, V> creates an object type, and Extract/Exclude operate on union members.
Key Pattern 
interface User {
  id: number;
  name: string;
  email: string;
  role: 'admin' | 'user' | 'editor';
  createdAt: Date;
}

// Partial — all optional (useful for update payloads)
type UpdateUser = Partial<User>;
// { id?: number; name?: string; email?: string; ... }

// Pick — select specific keys
type UserPreview = Pick<User, 'id' | 'name'>;
// { id: number; name: string }

// Omit — remove keys
type CreateUser = Omit<User, 'id' | 'createdAt'>;
// { name: string; email: string; role: 'admin' | 'user' | 'editor' }

// Record — typed dictionary
type RolePermissions = Record<User['role'], string[]>;
const perms: RolePermissions = {
  admin: ['read', 'write', 'delete'],
  user: ['read'],
  editor: ['read', 'write'],
};

// Extract / Exclude — filter union members
type AdminOrEditor = Extract<User['role'], 'admin' | 'editor'>;
// 'admin' | 'editor'
type NonAdmin = Exclude<User['role'], 'admin'>;
// 'user' | 'editor'

// Combining utilities
type EditableUser = Partial<Omit<User, 'id' | 'createdAt'>>;
// { name?: string; email?: string; role?: ... }
UpdateUser = { id?: number; name?: string; email?: string; role?: "admin" | "user" | "editor"; createdAt?: Date } UserPreview = { id: number; name: string } CreateUser = { name: string; email: string; role: "admin" | "user" | "editor" } AdminOrEditor = "admin" | "editor"
Challenge Exercise

Given a Product interface with id, name, price, description, category, and sku, create: (1) a ProductCard type using Pick with only display fields, (2) a ProductUpdate type using Partial and Omit that allows updating everything except id and sku, and (3) a CategoryMap using Record that maps category names to arrays of ProductCard.

#1 Pitfall: Omit does not produce an error when you omit a key that does not exist: Omit<User, 'typo'> silently returns the full User type. Always double-check the key strings match actual property names.
R2. Enums Structuring Types rep 2 · last reviewed 2026-06-02 · overdue 3 days
TypeScript enums create both a type and a runtime object. Numeric enums auto-increment and support reverse mapping (Direction[0] === 'Up'). String enums have no reverse mapping but produce more debuggable output. const enum is inlined at compile time (no runtime object). Prefer string literal unions ('up' | 'down') unless you need runtime reverse lookup.
Key Pattern 
// Numeric enum — auto-increments, has reverse mapping
enum Direction {
  Up,      // 0
  Down,    // 1
  Left,    // 2
  Right,   // 3
}
console.log(Direction.Up);        // 0
console.log(Direction[0]);        // "Up" (reverse mapping)

// String enum — explicit values, no reverse mapping
enum Status {
  Active = 'ACTIVE',
  Inactive = 'INACTIVE',
  Pending = 'PENDING',
}
console.log(Status.Active);       // "ACTIVE"
// Status['ACTIVE'] — Error: no reverse mapping for string enums

// const enum — inlined, no runtime object
const enum Color {
  Red = '#ff0000',
  Green = '#00ff00',
  Blue = '#0000ff',
}
const bg = Color.Red;  // Compiles to: const bg = "#ff0000"

// Prefer: string literal union (no runtime overhead, tree-shakeable)
type StatusUnion = 'active' | 'inactive' | 'pending';
function setStatus(s: StatusUnion) { /* ... */ }
0 "Up" "ACTIVE"
Challenge Exercise

Create a numeric enum HttpMethod and a string enum HttpStatus. Write a function that takes both and returns a formatted string. Then refactor both to string literal unions and compare the generated JavaScript output.

#1 Pitfall: Numeric enums accept any number at runtime: function move(d: Direction) {} allows move(999) without error. String enums and literal unions are stricter. This is a known TypeScript design quirk — prefer string enums or literal unions for safety.
R3. Project References Modules & Project Structure rep 1 · last reviewed 2026-06-07 · overdue 3 days
Project references let a monorepo split into multiple tsconfig.json files that reference each other. Each sub-project compiles independently, enabling incremental builds with tsc --build. The referencing project reads .d.ts outputs, not source files. composite: true must be set in each referenced project.
Key Pattern 
// packages/shared/tsconfig.json
{
  "compilerOptions": {
    "composite": true,          // Required for project references
    "declaration": true,        // Emit .d.ts files
    "declarationMap": true,     // Navigate to source in IDE
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}

// packages/app/tsconfig.json
{
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "references": [
    { "path": "../shared" }     // Reference the shared package
  ],
  "include": ["src"]
}

// Root tsconfig.json — orchestrates the build
{
  "files": [],
  "references": [
    { "path": "packages/shared" },
    { "path": "packages/app" }
  ]
}

// Build everything in dependency order:
// tsc --build
// tsc -b --watch   (incremental watch mode)
Challenge Exercise

Set up a two-package monorepo: packages/core (exports a User type and a validate function) and packages/api (imports from core). Configure composite: true, project references, and a root tsconfig.json. Build with tsc --build and verify that changing core triggers a rebuild of api.

#1 Pitfall: Forgetting "composite": true in the referenced project produces a confusing error: "Referenced project must have setting composite: true". Every project that appears in a references array must set composite: true and declaration: true.
R4. Zod & Runtime Validation Runtime Patterns rep 1 · last reviewed 2026-06-07 · overdue 3 days
Zod bridges the gap between TypeScript’s compile-time types and runtime validation. Define a schema once with z.object(), then derive the TypeScript type with z.infer<typeof schema>. Use .parse() (throws on failure) or .safeParse() (returns a result object). Zod schemas compose: .extend(), .merge(), .pick(), .omit(), .partial() mirror TypeScript utility types at runtime.
Key Pattern 
import { z } from 'zod';

const userSchema = z.object({
  id: z.number(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'editor']),
  tags: z.array(z.string()).default([]),
});

// Derive the TypeScript type — single source of truth
type User = z.infer<typeof userSchema>;
// { id: number; name: string; email: string; role: "admin" | "user" | "editor"; tags: string[] }

// .safeParse — does not throw
const result = userSchema.safeParse(apiResponse);
if (result.success) {
  console.log(result.data.name);   // typed as string
} else {
  console.error(result.error.issues);
}

// Schema composition mirrors utility types
const createUserSchema = userSchema.omit({ id: true });
type CreateUser = z.infer<typeof createUserSchema>;
// { name: string; email: string; role: ...; tags: string[] }

const updateUserSchema = userSchema.partial().required({ id: true });
type UpdateUser = z.infer<typeof updateUserSchema>;
// { id: number; name?: string; email?: string; role?: ...; tags?: string[] }
User = { id: number; name: string; email: string; role: "admin" | "user" | "editor"; tags: string[] } CreateUser = { name: string; email: string; role: "admin" | "user" | "editor"; tags: string[] }
Challenge Exercise

Create a Zod schema for an API response envelope: { success: boolean, data: T, errors?: { field: string, message: string }[] }. Make it generic by writing a function apiResponseSchema(dataSchema: ZodType) that wraps any data schema. Use safeParse on a mock response and log whether it succeeded.

#1 Pitfall: Using .parse() in a try/catch and forgetting that ZodError contains structured .issues — do not just catch (e) { console.log(e.message) }. Use safeParse or access error.issues for field-level error details.
R5. Covariance & Contravariance Advanced Patterns & Performance rep 0 · last reviewed 2026-06-07 · overdue 3 days
Variance describes how subtype relationships flow through generic types. Covariant (output position): Producer<Dog> is assignable to Producer<Animal> because Dog extends Animal. Contravariant (input position): Consumer<Animal> is assignable to Consumer<Dog> because a function that handles any Animal can handle a Dog. TypeScript uses strictFunctionTypes to enforce contravariant parameter checking. Use in and out modifiers (TS 4.7+) to declare variance explicitly.
Key Pattern 
// Covariant — output position (return types, readonly properties)
interface Producer<out T> {
  produce(): T;
}

// Contravariant — input position (function parameters)
interface Consumer<in T> {
  consume(value: T): void;
}

// Invariant — both input and output
interface Box<in out T> {
  get(): T;     // out (covariant)
  set(v: T): void; // in (contravariant)
}

class Animal { name = 'animal'; }
class Dog extends Animal { breed = 'labrador'; }

// Covariance: Producer<Dog> assignable to Producer<Animal>
const dogProducer: Producer<Dog> = { produce: () => new Dog() };
const animalProducer: Producer<Animal> = dogProducer; // OK

// Contravariance: Consumer<Animal> assignable to Consumer<Dog>
const animalConsumer: Consumer<Animal> = { consume: (a) => console.log(a.name) };
const dogConsumer: Consumer<Dog> = animalConsumer; // OK

// Invariance: Box<Dog> NOT assignable to Box<Animal>
const dogBox: Box<Dog> = { get: () => new Dog(), set: (d) => {} };
// const animalBox: Box<Animal> = dogBox; // Error!
Type 'Box<Dog>' is not assignable to type 'Box<Animal>'. Property 'set' is incompatible: Type '(v: Animal) => void' is not assignable to type '(v: Dog) => void'.
Challenge Exercise

Define Animal, Dog, and Cat classes. Create a Transformer<in A, out B> interface with a transform(input: A): B method. Verify that Transformer<Animal, Dog> is assignable to Transformer<Dog, Animal> (contravariant in first parameter, covariant in second). Try the reverse assignment and explain the error.

#1 Pitfall: Without strictFunctionTypes, TypeScript checks function parameters bivariantly (both co- and contra- are allowed), which is unsound. Always enable strictFunctionTypes (included in strict: true) to get correct contravariant checking for function parameters.
R6. Template Literal Types — Advanced Advanced Patterns & Performance rep 0 · last reviewed 2026-06-07 · overdue 3 days
Template literal types combine string literal types with ${} interpolation at the type level. They can create large unions from smaller ones (cartesian product), parse and validate string shapes, and power type-safe routing, CSS class names, and event systems. Intrinsic string manipulation types (Uppercase, Lowercase, Capitalize, Uncapitalize) transform string literals at compile time.
Key Pattern 
// Cartesian product of string unions
type Color = 'red' | 'blue' | 'green';
type Size = 'sm' | 'md' | 'lg';
type CSSClass = `${Color}-${Size}`;
// "red-sm" | "red-md" | "red-lg" | "blue-sm" | ... (9 combinations)

// Intrinsic string manipulation
type EventName = 'click' | 'focus' | 'blur';
type HandlerName = `on${Capitalize<EventName>}`;
// "onClick" | "onFocus" | "onBlur"

// Type-safe route parameters
type Route = '/users/:userId/posts/:postId';

type ExtractParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<Rest>
    : T extends `${string}:${infer Param}`
      ? Param
      : never;

type RouteParams = ExtractParams<Route>;
// "userId" | "postId"

// Dot-path accessor
type DotPath<T, Prefix extends string = ''> = {
  [K in keyof T & string]:
    T[K] extends object
      ? DotPath<T[K], `${Prefix}${K}.`>
      : `${Prefix}${K}`;
}[keyof T & string];

interface Config {
  db: { host: string; port: number };
  app: { name: string };
}
type ConfigPath = DotPath<Config>;
// "db.host" | "db.port" | "app.name"
CSSClass = "red-sm" | "red-md" | "red-lg" | "blue-sm" | "blue-md" | "blue-lg" | "green-sm" | "green-md" | "green-lg" HandlerName = "onClick" | "onFocus" | "onBlur" RouteParams = "userId" | "postId"
Challenge Exercise

Create a type ParseQueryString<S> that extracts key-value pairs from a URL query string like "name=alice&age=30&city=nyc" and produces { name: string; age: string; city: string }. Use recursive template literal types with infer to split on & and =.

#1 Pitfall: Template literal types generate combinatorial unions. `${A}-${B}-${C}` with three unions of 10 members each produces 1,000 union members. TypeScript caps unions at 100,000 members and slows down well before that. Keep your input unions small or use branded types instead.
1 Type-Level Programming
Advanced Utility Type Recipes DeepPartial DeepReadonly Recursive Types
Built-in utilities like Partial and Readonly only operate one level deep. Real-world data structures are nested — API responses, configuration objects, form state. Recursive conditional types let you build deep versions that transform an entire object tree, catching mutations and missing fields at every level.

The key insight is using conditional types to recurse into nested objects:

  • 2026-07-04 (Day 31)
  • DeepPartial<T>: Makes every property optional at every depth — essential for deeply nested form state or patch payloads
  • DeepReadonly<T>: Makes every property readonly at every depth — prevents accidental mutation of nested state
  • Paths<T>: Extracts all dot-separated paths from a nested object type — powers type-safe get(obj, 'a.b.c') functions
  • NestedKeyOf<T>: Collects all keys at every depth into a flat union
  • The base case must handle primitives, arrays, and Date/Map/Set to avoid infinite recursion on built-in types
deep-partial.ts
// DeepPartial — recursively makes all properties optional
type DeepPartial<T> =
  T extends Function ? T :
  T extends Array<infer U> ? Array<DeepPartial<U>> :
  T extends object ? { [K in keyof T]?: DeepPartial<T[K]> } :
  T;

interface AppConfig {
  server: {
    host: string;
    port: number;
    ssl: {
      enabled: boolean;
      cert: string;
      key: string;
    };
  };
  database: {
    url: string;
    pool: {
      min: number;
      max: number;
    };
  };
}

// Standard Partial — only top level is optional
type ShallowPartial = Partial<AppConfig>;
// { server?: { host: string; port: number; ssl: { ... } }; ... }
// Inner properties are still REQUIRED

// DeepPartial — every level is optional
type ConfigOverride = DeepPartial<AppConfig>;
// { server?: { host?: string; port?: number; ssl?: { enabled?: boolean; ... } }; ... }

function mergeConfig(base: AppConfig, overrides: DeepPartial<AppConfig>): AppConfig {
  return {
    server: {
      host: overrides.server?.host ?? base.server.host,
      port: overrides.server?.port ?? base.server.port,
      ssl: {
        enabled: overrides.server?.ssl?.enabled ?? base.server.ssl.enabled,
        cert: overrides.server?.ssl?.cert ?? base.server.ssl.cert,
        key: overrides.server?.ssl?.key ?? base.server.ssl.key,
      },
    },
    database: {
      url: overrides.database?.url ?? base.database.url,
      pool: {
        min: overrides.database?.pool?.min ?? base.database.pool.min,
        max: overrides.database?.pool?.max ?? base.database.pool.max,
      },
    },
  };
}

// Only override what you need — deeply nested partial is fine
const override: DeepPartial<AppConfig> = {
  server: {
    ssl: { enabled: false },   // Only change SSL enabled flag
  },
};
ConfigOverride = { server?: { host?: string; port?: number; ssl?: { enabled?: boolean; cert?: string; key?: string }; }; database?: { url?: string; pool?: { min?: number; max?: number }; }; }
deep-readonly.ts
// DeepReadonly — recursively freezes an entire object tree
type DeepReadonly<T> =
  T extends Function ? T :
  T extends Map<infer K, infer V> ? ReadonlyMap<DeepReadonly<K>, DeepReadonly<V>> :
  T extends Set<infer U> ? ReadonlySet<DeepReadonly<U>> :
  T extends Array<infer U> ? ReadonlyArray<DeepReadonly<U>> :
  T extends object ? { readonly [K in keyof T]: DeepReadonly<T[K]> } :
  T;

interface AppState {
  user: {
    name: string;
    preferences: {
      theme: 'dark' | 'light';
      notifications: boolean;
    };
  };
  items: Array<{ id: number; label: string }>;
}

type FrozenState = DeepReadonly<AppState>;

function getState(): FrozenState {
  return {
    user: {
      name: 'Alice',
      preferences: { theme: 'dark', notifications: true },
    },
    items: [{ id: 1, label: 'First' }],
  };
}

const state = getState();
// state.user.name = 'Bob';                     // Error: readonly
// state.user.preferences.theme = 'light';      // Error: readonly (DEEP)
// state.items.push({ id: 2, label: 'Second' }); // Error: ReadonlyArray
// state.items[0].label = 'Changed';            // Error: readonly (DEEP)
console.log(state.user.preferences.theme);       // OK: reading is fine
Cannot assign to 'name' because it is a read-only property. Cannot assign to 'theme' because it is a read-only property. Property 'push' does not exist on type 'readonly DeepReadonly<{ id: number; label: string }>[]'.
paths.ts (type-safe dot-path access)
// Paths<T> — all dot-separated paths through a nested object
type Paths<T, Prefix extends string = ''> = T extends object
  ? {
      [K in keyof T & string]:
        | `${Prefix}${K}`
        | Paths<T[K], `${Prefix}${K}.`>
    }[keyof T & string]
  : never;

interface FormState {
  user: {
    name: string;
    address: {
      street: string;
      city: string;
      zip: string;
    };
  };
  settings: {
    theme: 'dark' | 'light';
  };
}

type FormPath = Paths<FormState>;
// "user" | "user.name" | "user.address" | "user.address.street"
// | "user.address.city" | "user.address.zip" | "settings" | "settings.theme"

// Type-safe get function
type PathValue<T, P extends string> =
  P extends `${infer K}.${infer Rest}`
    ? K extends keyof T ? PathValue<T[K], Rest> : never
    : P extends keyof T ? T[P] : never;

function get<T, P extends Paths<T>>(obj: T, path: P): PathValue<T, P> {
  return path.split('.').reduce((acc: any, key) => acc[key], obj) as any;
}

const form: FormState = {
  user: { name: 'Alice', address: { street: '123 Main', city: 'NYC', zip: '10001' } },
  settings: { theme: 'dark' },
};

const city = get(form, 'user.address.city');   // typed as string
const theme = get(form, 'settings.theme');      // typed as 'dark' | 'light'
// get(form, 'user.address.typo');              // Error: not a valid path
FormPath = "user" | "user.name" | "user.address" | "user.address.street" | "user.address.city" | "user.address.zip" | "settings" | "settings.theme" typeof city = string typeof theme = "dark" | "light"
Common Mistakes
Forgetting to handle Date, Map, Set, and Function in the base case: DeepPartial<Date> recurses into Date’s internal properties
Always add explicit checks: T extends Function ? T : T extends Date ? T : T extends Map<...> ? ... : ...
Using Paths<T> on types with circular references — causes infinite recursion and crashes the compiler
Add a depth limiter by passing a tuple that grows: Paths<T, Prefix, [...Depth, 0]> and stop when Depth['length'] exceeds a threshold
Key Rules
  • Recursive utility types need base cases for primitives, Function, Date, Map, Set, and arrays
  • DeepPartial<T>: T extends object ? { [K in keyof T]?: DeepPartial<T[K]> } : T
  • DeepReadonly<T>: T extends object ? { readonly [K in keyof T]: DeepReadonly<T[K]> } : T
  • Paths<T> generates all dot-separated paths; pair with PathValue<T, P> for a type-safe get()
  • Add depth limits to recursive types that might hit circular references

2 Advanced Type System
Variadic Tuple Types Tuple Spread TS 4.0+
Variadic tuple types let you spread generic type parameters inside tuples: [...T, ...U]. This unlocks type-safe concat, zip, curry, and pipe at the type level — patterns that were previously impossible to type without losing precision. They are the foundation for typed function composition.

Before variadic tuples, concatenating two arrays lost their precise tuple types and collapsed to Array<T | U>. Key capabilities:

  • Tuple spread: [...A, ...B] concatenates two tuple types positionally
  • Typed concat: function concat<T extends unknown[], U extends unknown[]>(a: [...T], b: [...U]): [...T, ...U]
  • Typed pipe: Chain functions where each output matches the next input, inferred from the tuple of functions
  • Rest elements in tuples: [string, ...number[], boolean] — leading, middle, and trailing types
  • curry typing: Split a function’s parameter tuple into applied and remaining parts
variadic-basics.ts
// Type-safe concat — preserves exact tuple types
function concat<T extends unknown[], U extends unknown[]>(
  a: [...T],
  b: [...U]
): [...T, ...U] {
  return [...a, ...b] as [...T, ...U];
}

const result = concat([1, 'hello'] as const, [true, 42] as const);
// result: readonly [1, "hello", true, 42]

// Type-safe zip — pairs elements positionally
type Zip<A extends unknown[], B extends unknown[]> =
  A extends [infer AHead, ...infer ARest]
    ? B extends [infer BHead, ...infer BRest]
      ? [[AHead, BHead], ...Zip<ARest, BRest>]
      : []
    : [];

function zip<A extends unknown[], B extends unknown[]>(
  a: [...A],
  b: [...B]
): Zip<A, B> {
  return a.map((val, i) => [val, b[i]]) as any;
}

const zipped = zip(['a', 'b', 'c'], [1, 2, 3]);
// typed as [["a", 1], ["b", 2], ["c", 3]]
console.log(zipped[0]);  // ["a", 1]
console.log(zipped[1]);  // ["b", 2]
typeof result = readonly [1, "hello", true, 42] typeof zipped = [["a", 1], ["b", 2], ["c", 3]]
curry.ts (typed partial application)
// Typed curry — split parameters into applied and remaining
type Curry<F> =
  F extends (...args: infer Args) => infer R
    ? Args extends [infer First, ...infer Rest]
      ? (arg: First) => Curry<(...args: Rest) => R>
      : R
    : never;

function curry<F extends (...args: any[]) => any>(fn: F): Curry<F> {
  return function curried(...args: any[]): any {
    if (args.length >= fn.length) return fn(...args);
    return (...more: any[]) => curried(...args, ...more);
  } as any;
}

function add(a: number, b: number, c: number): number {
  return a + b + c;
}

const curriedAdd = curry(add);
// curriedAdd: (arg: number) => (arg: number) => (arg: number) => number

const add10 = curriedAdd(10);        // (arg: number) => (arg: number) => number
const add10and20 = add10(20);        // (arg: number) => number
const result = add10and20(30);       // number
console.log(result);                 // 60

// Partial application with tuple spread
type PartialApply<
  Args extends unknown[],
  Applied extends unknown[]
> = Args extends [...Applied, ...infer Rest] ? Rest : never;

function partialApply<
  F extends (...args: any[]) => any,
  A extends unknown[]
>(fn: F, ...applied: A): (...rest: PartialApply<Parameters<F>, A>) => ReturnType<F> {
  return (...rest) => fn(...applied, ...rest);
}

const add10Partial = partialApply(add, 10);
// (b: number, c: number) => number
console.log(add10Partial(20, 30));   // 60
60 60
pipe.ts (typed function composition)
// Typed pipe — each function's output must match the next's input
type PipeFns<Fns extends ((...args: any) => any)[], Acc = unknown> =
  Fns extends [infer First extends (...args: any) => any, ...infer Rest extends ((...args: any) => any)[]]
    ? [First, ...PipeFns<Rest, ReturnType<First>>]
    : [];

function pipe<A, B>(a: A, fn1: (a: A) => B): B;
function pipe<A, B, C>(a: A, fn1: (a: A) => B, fn2: (b: B) => C): C;
function pipe<A, B, C, D>(a: A, fn1: (a: A) => B, fn2: (b: B) => C, fn3: (c: C) => D): D;
function pipe(initial: unknown, ...fns: Function[]): unknown {
  return fns.reduce((acc, fn) => (fn as any)(acc), initial);
}

const result = pipe(
  '  Hello, World!  ',
  (s: string) => s.trim(),           // string => string
  (s: string) => s.toLowerCase(),    // string => string
  (s: string) => s.length,           // string => number
);
// result: number
console.log(result);  // 13

// Type error: number cannot flow into a string parameter
// pipe(
//   42,
//   (n: number) => n.toString(),
//   (s: string) => s.length,
//   (n: string) => n.toFixed(2),  // Error: string has no toFixed
// );

Common Mistakes
Forgetting as const when passing literal tuples: concat([1, 'hello'], [true]) infers (number | string)[] instead of [number, string]
Use as const or [...T] parameter syntax: function f<T extends unknown[]>(a: [...T]) — the spread hint tells TS to infer a tuple
Recursive tuple types without a base case: Zip<A, B> that never returns [] causes infinite instantiation
Always match the empty tuple as a base case: A extends [] ? [] : ... or A extends [infer H, ...infer R] ? ... : []
Key Rules
  • [...T, ...U] concatenates tuple types — the spread preserves positional types
  • Use [...T] in parameter position to hint TypeScript to infer a tuple instead of an array
  • Recursive tuple types need a base case matching [] or [infer H, ...infer R]
  • Variadic tuples enable typed concat, zip, curry, and pipe
  • Rest elements can appear at any position in a tuple: [string, ...number[], boolean]

Mapped Type Key Remapping as clause Key Filtering TS 4.1+
Mapped types with the as clause let you rename, filter, and transform keys during mapping. Combined with template literal types and conditional types, you can automatically generate getter/setter interfaces, strip prefixes, filter by value type, and create powerful type transformations — all without runtime code.

The as clause in a mapped type remaps keys during iteration:

  • Rename keys: { [K in keyof T as get${Capitalize<K>}]: ... }
  • Filter keys: Return never from the as clause to exclude a key
  • Combine with conditional types: Filter by value type, keep only functions, extract events
  • Template literal transforms: Generate getX/setX from property names, strip prefixes like on or _
key-remapping.ts
interface User {
  id: number;
  name: string;
  email: string;
  isActive: boolean;
}

// Rename keys — create getter interface
type Getters<T> = {
  [K in keyof T & string as `get${Capitalize<K>}`]: () => T[K];
};

type UserGetters = Getters<User>;
// {
//   getId: () => number;
//   getName: () => string;
//   getEmail: () => string;
//   getIsActive: () => boolean;
// }

// Create both getters and setters
type GettersAndSetters<T> = {
  [K in keyof T & string as `get${Capitalize<K>}`]: () => T[K];
} & {
  [K in keyof T & string as `set${Capitalize<K>}`]: (value: T[K]) => void;
};

type UserAccessors = GettersAndSetters<User>;
// getId: () => number
// setId: (value: number) => void
// getName: () => string
// setName: (value: string) => void
// ... etc
UserGetters = { getId: () => number; getName: () => string; getEmail: () => string; getIsActive: () => boolean; }
key-filtering.ts
interface Mixed {
  id: number;
  name: string;
  greet: (msg: string) => void;
  age: number;
  process: () => Promise<void>;
  email: string;
}

// Filter: keep only string-valued properties
type StringProps<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};

type OnlyStrings = StringProps<Mixed>;
// { name: string; email: string }

// Filter: keep only function-valued properties
type MethodsOnly<T> = {
  [K in keyof T as T[K] extends Function ? K : never]: T[K];
};

type OnlyMethods = MethodsOnly<Mixed>;
// { greet: (msg: string) => void; process: () => Promise<void> }

// Filter: keep only non-function properties (data only)
type DataOnly<T> = {
  [K in keyof T as T[K] extends Function ? never : K]: T[K];
};

type OnlyData = DataOnly<Mixed>;
// { id: number; name: string; age: number; email: string }

// Strip prefix — remove leading underscore from private keys
type StripPrefix<T, Prefix extends string> = {
  [K in keyof T & string as K extends `${Prefix}${infer Rest}` ? Rest : K]: T[K];
};

interface InternalAPI {
  _id: number;
  _secret: string;
  name: string;
  _internal: boolean;
}

type PublicAPI = StripPrefix<InternalAPI, '_'>;
// { id: number; secret: string; name: string; internal: boolean }

interface DOMHandlers {

  onFocus: (e: FocusEvent) => void;

  id: string;

  handler: (e: Events[K]) => void

  // ...
}

Common Mistakes
Forgetting & string on K in keyof T: without it, K includes number and symbol keys, which cannot be used in template literals
Always use K in keyof T & string as ... when the as clause uses template literal types
Assuming as never removes the property — it does, but returning never from a conditional like T[K] extends string ? K : never filters keys, not values
The as clause maps keys: when the as expression evaluates to never, that key is excluded from the mapped type
Key Rules
  • as clause in mapped types remaps keys: { [K in keyof T as NewKey]: T[K] }
  • Return never from the as clause to filter out keys
  • Combine with Capitalize/Uncapitalize for getter/setter generation
  • Use K extends ${Prefix}${infer Rest} ? Rest : K to strip prefixes
  • Filter by value type: T[K] extends Function ? K : never keeps only function properties
  • Always add & string to keyof T when using template literals in the as clause

3 Runtime Patterns
Type-Safe Event Emitters EventEmitter Typed Events Patterns
Event emitters are everywhere — Node.js core, DOM, state libraries, message buses. The default EventEmitter uses string event names and any payloads, so a typo in an event name or wrong payload shape silently fails at runtime. A typed event emitter catches these at compile time.

The pattern uses a map type that associates event names with their payload types:

  • Define an Events interface where keys are event names and values are payload types (or tuples of arguments)
  • Generic on<K extends keyof Events>(event: K, handler: (payload: Events[K]) => void) restricts event names and types the handler
  • Generic emit<K extends keyof Events>(event: K, payload: Events[K]) ensures the emitted data matches
  • Works with Node.js EventEmitter, DOM addEventListener, and custom implementations
typed-emitter.ts
// Generic typed event emitter
class TypedEventEmitter<Events extends Record<string, unknown>> {
  private listeners = new Map<keyof Events, Set<(payload: any) => void>>();

  on<K extends keyof Events>(
    event: K,
    handler: (payload: Events[K]) => void
  ): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(handler);

    // Return unsubscribe function
    return () => {
      this.listeners.get(event)?.delete(handler);
    };
  }

  emit<K extends keyof Events>(event: K, payload: Events[K]): void {
    this.listeners.get(event)?.forEach(fn => fn(payload));
  }

  off<K extends keyof Events>(
    event: K,
    handler: (payload: Events[K]) => void
  ): void {
    this.listeners.get(event)?.delete(handler);
  }

  once<K extends keyof Events>(
    event: K,
    handler: (payload: Events[K]) => void
  ): void {
    const wrapper = (payload: Events[K]) => {
      handler(payload);
      this.off(event, wrapper);
    };
    this.on(event, wrapper);
  }
}

// Define your event map
interface ChatEvents {
  message: { from: string; text: string; timestamp: number };
  userJoined: { username: string; roomId: string };
  userLeft: { username: string; roomId: string };
  typing: { username: string; isTyping: boolean };
  error: { code: number; message: string };
}

// Create a typed emitter
const chat = new TypedEventEmitter<ChatEvents>();

// on — event name autocompletes, handler payload is typed

  // payload is: { from: string; text: string; timestamp: number }
  console.log(`${payload.from}: ${payload.text}`);
});

chat.on('userJoined', (payload) => {
  // payload is: { username: string; roomId: string }
  console.log(`${payload.username} joined ${payload.roomId}`);
});

// emit — payload must match the event's type

  from: 'Alice',
  text: 'Hello!',
  timestamp: Date.now(),
});

// chat.emit('typo', { data: 123 });                // 'typo' not in ChatEvents

node-emitter.ts (wrapping Node.js EventEmitter)
import { EventEmitter } from 'events';

// Typed wrapper around Node.js EventEmitter
interface ServerEvents {
  connection: { clientId: string; ip: string };
  data: { clientId: string; payload: Buffer };
  disconnect: { clientId: string; reason: string };
  error: Error;
}

class TypedNodeEmitter<Events extends Record<string, unknown>> {
  private emitter = new EventEmitter();

  on<K extends keyof Events & string>(
    event: K,
    handler: (payload: Events[K]) => void
  ): this {
    this.emitter.on(event, handler as any);
    return this;
  }

  emit<K extends keyof Events & string>(
    event: K,
    payload: Events[K]
  ): boolean {
    return this.emitter.emit(event, payload);
  }

  removeAllListeners<K extends keyof Events & string>(event?: K): this {
    this.emitter.removeAllListeners(event);
    return this;
  }
}

  console.log(`Client ${payload.clientId} connected from ${payload.ip}`);
});

server.on('data', (payload) => {
  console.log(`Data from ${payload.clientId}: ${payload.payload.length} bytes`);
});

// Chaining works (returns this)
server
  .on('disconnect', (p) => console.log(`${p.clientId} left: ${p.reason}`))
  .on('error', (err) => console.error(err.message));

// Type-safe event delegation for DOM elements
interface AppEvents {
  'nav:click': { path: string; label: string };
  'form:submit': { formId: string; data: Record<string, string> };
  'modal:open': { modalId: string };
  'modal:close': { modalId: string; reason: 'user' | 'timeout' | 'api' };
}

// Custom event bus using DOM CustomEvent under the hood
class DOMEventBus<Events extends Record<string, unknown>> {
  private target = new EventTarget();

  dispatch<K extends keyof Events & string>(
    event: K,
    detail: Events[K]
  ): void {
    this.target.dispatchEvent(
      new CustomEvent(event, { detail })
    );
  }

  listen<K extends keyof Events & string>(
    event: K,
    handler: (detail: Events[K]) => void
  ): () => void {
    const wrapper = (e: Event) => {
      handler((e as CustomEvent).detail);
    };
    this.target.addEventListener(event, wrapper);
    return () => this.target.removeEventListener(event, wrapper);
  }
}

const bus = new DOMEventBus<AppEvents>();

const unsub = bus.listen('modal:close', (detail) => {
  // detail is: { modalId: string; reason: 'user' | 'timeout' | 'api' }
  if (detail.reason === 'timeout') {
    console.log(`Modal ${detail.modalId} timed out`);
  }
});

bus.dispatch('modal:close', {
  modalId: 'settings',
  reason: 'user',
});

unsub(); // Clean up listener
Common Mistakes
Defining the event map with any payloads: interface Events { message: any } — defeats the purpose entirely
Always define exact payload shapes: { message: { from: string; text: string } }
Forgetting to return an unsubscribe function from on — leads to memory leaks in long-lived components
Return a cleanup function: on(...): () => void. Callers store it and invoke on teardown
Using keyof Events without & string when interfacing with APIs that require string keys
Use K extends keyof Events & string to ensure K is a string (EventEmitter and DOM need strings)
Key Rules
  • Define an event map interface: keys are event names, values are payload types
  • Generic on<K extends keyof Events> and emit<K extends keyof Events> enforce the contract
  • The handler parameter type is automatically Events[K] — fully narrowed per event
  • Return an unsubscribe function from on for cleanup
  • Use & string on keyof Events when wrapping string-based APIs (EventEmitter, DOM)

Builder Pattern with Type-Level State Builder Phantom Types Method Chaining
The builder pattern constructs complex objects step by step. Without type-level state tracking, you can forget to set a required field and only discover it at runtime. By encoding "which fields have been set" in the type itself, TypeScript can enforce at compile time that build() is only callable after all mandatory fields are provided.

The technique uses phantom types (generic parameters that exist only at the type level) to track builder state:

  • Each set method returns a new builder type with the corresponding field marked as "set"
  • The build() method is only available when all required fields are present in the type parameter
  • This pattern uses intersection types to accumulate set fields: Builder<State & { host: true }>
  • A conditional type on build() checks whether the required fields are a subset of the current state
builder-basic.ts
// Track which fields have been set at the type level
interface RequiredFields {
  host: true;
  port: true;
  database: true;
}

interface DBConfig {
  host: string;
  port: number;
  database: string;
  user?: string;
  password?: string;
  ssl?: boolean;
  poolSize?: number;
}

class DBConfigBuilder<State extends Partial<RequiredFields> = {}> {
  private config: Partial<DBConfig> = {};

  host(host: string): DBConfigBuilder<State & { host: true }> {
    this.config.host = host;
    return this as any;
  }

  port(port: number): DBConfigBuilder<State & { port: true }> {
    this.config.port = port;
    return this as any;
  }

  database(db: string): DBConfigBuilder<State & { database: true }> {
    this.config.database = db;
    return this as any;
  }

  // Optional fields do not change the state type
  user(user: string): this {
    this.config.user = user;
    return this;
  }

  password(password: string): this {
    this.config.password = password;
    return this;
  }

  ssl(enabled: boolean): this {
    this.config.ssl = enabled;
    return this;
  }

  poolSize(size: number): this {
    this.config.poolSize = size;
    return this;
  }

  // build() is only available when all required fields are set
  build(
    this: DBConfigBuilder<RequiredFields>
  ): DBConfig {
    return this.config as DBConfig;
  }
}

// Correct usage — all required fields set
const config = new DBConfigBuilder()
  .host('localhost')       // State: { host: true }
  .port(5432)              // State: { host: true; port: true }
  .database('myapp')       // State: { host: true; port: true; database: true }
  .user('admin')           // State unchanged (optional)
  .ssl(true)               // State unchanged (optional)
  .build();                // OK — RequiredFields satisfied

console.log(config);
// { host: 'localhost', port: 5432, database: 'myapp', user: 'admin', ssl: true }

  url: true;
  method: true;

  method: HttpMethod;
  headers: Record<string, string>;

  timeout?: number;
  retries?: number;
}

  }

    this.config.method = method;
    return this as any;
  }

  // Convenience methods that set method + url together
  get(url: string): RequestBuilder<State & { url: true; method: true }> {
    this.config.url = url;
    this.config.method = 'GET';
    return this as any;
  }

  post(url: string): RequestBuilder<State & { url: true; method: true }> {
    this.config.url = url;
    this.config.method = 'POST';
    return this as any;
  }

  header(key: string, value: string): this {
    this.config.headers![key] = value;
    return this;
  }

  body(data: unknown): this {
    this.config.body = data;
    return this;
  }

  timeout(ms: number): this {
    this.config.timeout = ms;
    return this;
  }

  retries(count: number): this {
    this.config.retries = count;
    return this;
  }

  build(this: RequestBuilder<RequestRequired>): RequestConfig {
    return this.config as RequestConfig;
  }
}

  .url('/api/users')

Common Mistakes
Returning this from required-field setters: host(h: string): this { ... } — this does not update the type parameter
Return the builder with an updated state parameter: host(h: string): DBConfigBuilder<State & { host: true }>. The return this as any is safe because the runtime object is the same
Making build() a normal method and checking at runtime: if (!this.config.host) throw new Error(...)
Use the this parameter type: build(this: Builder<RequiredFields>). The error is caught at compile time, not runtime
Trying to use the builder pattern with too many required fields (10+) — the type errors become unreadable
For objects with many required fields, consider a plain constructor with a typed config object. Reserve the builder pattern for 3-5 required fields with many optional ones
Key Rules
  • Track set fields via a generic state parameter: Builder<State extends Partial<Required>>
  • Each required setter returns Builder<State & { field: true }> to accumulate state
  • Optional setters return this — they don’t change the type-level state
  • Use a this parameter on build() to constrain: build(this: Builder<RequiredFields>)
  • return this as any is the standard escape hatch — the runtime object is the same, only the type changes
