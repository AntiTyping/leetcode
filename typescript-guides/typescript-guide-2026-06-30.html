R1. Branded/Opaque Types Functions & Patterns 17 days since review · rep 3 → 4
A branded type intersects a base type with a unique tag so two structurally-identical values (a raw string, a PostId) are no longer interchangeable where a UserId is required. The brand exists only at compile time; a smart constructor is the single place that mints branded values.
Key Pattern 
declare const brand: unique symbol;
type Brand<T, B> = T & { readonly [brand]: B };

type UserId  = Brand<string, "UserId">;
type OrderId = Brand<string, "OrderId">;

const asUserId = (s: string): UserId => s as UserId;   // smart constructor

function loadUser(id: UserId) {/* ... */}
loadUser(asUserId("u_1"));   // ok
loadUser("u_1");             // error: plain string lacks the brand
declare const o: OrderId;
loadUser(o);                 // error: OrderId is not UserId
distinct branded strings are no longer assignable to each other
Challenge Exercise

Build Brand<T, B>, then model type Cents = Brand<number, "Cents"> and type Dollars = Brand<number, "Dollars">. Write toCents(d: Dollars): Cents and confirm passing a raw number or a Dollars where Cents is expected is a compile error. Bonus: write a validating parseEmail(raw: string): Email | null smart constructor that only brands after checking the format.

Pitfall: Sprinkling as to mint branded values everywhere defeats the purpose. Funnel creation through a single validating smart constructor — otherwise the brand asserts a guarantee that nothing actually verified.
R2. satisfies Operator Advanced Type System 15 days since review · rep 3 → 4
expr satisfies T checks that expr is assignable to T without widening its inferred type — you keep the precise literal type and still get an error if the shape is wrong. Contrast : T (annotation, widens to T) and as T (assertion, no real check).
Key Pattern 
type Config = Record<string, { url: string; retries: number }>;

const config = {
  auth: { url: "/auth", retries: 3 },
  data: { url: "/data", retries: 5 },
} satisfies Config;

config.auth.retries;   // number — and keys "auth" | "data" stay known
// const c: Config = {...}  would widen keys to string → config.auth errors
validated against Config, yet keys stay literal ("auth" | "data")
Challenge Exercise

Type a palette object as { primary: "#0d1117", ... } satisfies Record<string, string> and show palette.primary is still the literal hex string, not widened string. Then introduce a typo in one nested value and confirm satisfies flags it — while key autocomplete stays intact (unlike a plain : Record<string, string> annotation).

Pitfall: Reaching for as Config when you mean satisfies Config. as silently accepts a wrong shape (it’s an assertion, no check); satisfies validates the shape without discarding the narrow inferred type.
R3. never & Exhaustiveness Advanced Type System 15 days since review · rep 3 → 4
never is the bottom type (assignable to everything, nothing assignable to it). Assigning the scrutinee to a never variable in a switch default turns “unhandled union member” into a compile error — the compiler proves every case is covered.
Key Pattern 
type Shape =
  | { kind: "circle"; r: number }
  | { kind: "square"; s: number };

function area(sh: Shape): number {
  switch (sh.kind) {
    case "circle": return Math.PI * sh.r ** 2;
    case "square": return sh.s ** 2;
    default: {
      const _never: never = sh;   // add a new Shape → error here
      return _never;
    }
  }
}
adding a variant without a matching case fails to compile
Challenge Exercise

Write assertNever(x: never): never that throws, and use it in the default of a discriminated Event union with 3+ members. Add a fourth member and confirm the single assertNever call site flags it. Then explain why default: return x (without the never binding) would not catch the missing case.

Pitfall: Widening the discriminant before the check — e.g. switch (sh.kind as string) — erases the literal union and makes exhaustiveness impossible. Keep the scrutinee’s narrow type intact.
R4. Project References Modules & Project Structure 19 days since review · rep 2 → 3
references + composite: true split a codebase into independently-built TS projects. tsc --build (build mode) topologically builds each referenced project, emits .d.ts + .tsbuildinfo, and skips unchanged ones for fast incremental builds.
Key Pattern 
// packages/app/tsconfig.json
{
  "compilerOptions": { "composite": true, "outDir": "dist" },
  "references": [{ "path": "../core" }]
}

// build the whole graph, incrementally:
//   tsc --build          (a.k.a. tsc -b)
core builds first; app consumes core’s emitted .d.ts, not its source
Challenge Exercise

Set up a two-project graph (core referenced by web), give core "composite": true, and explain what breaks if you (a) omit composite, (b) run plain tsc instead of tsc -b, and (c) import a core file that core’s own tsconfig doesn’t include.

Pitfall: Expecting a referenced project’s source to be recompiled from a consumer. Build mode consumes the referenced project’s emitted .d.ts, so that project must set composite: true and be built (which -b does for you).
R5. Package Publishing Modules & Project Structure 15 days since review · rep 2 → 3
The exports map defines a package’s public entry points and, per condition (types/import/require), which file resolves — replacing the single old main/types. Put types alongside (ideally first) so node16/nodenext consumers get correct declarations per condition.
Key Pattern 
// package.json
{
  "type": "module",
  "exports": {
    ".": {
      "types":   "./dist/index.d.ts",
      "import":  "./dist/index.js",
      "require": "./dist/index.cjs"
    },
    "./utils": { "types": "./dist/utils.d.ts", "import": "./dist/utils.js" }
  }
}
"./utils" is a public subpath; everything else is encapsulated
Challenge Exercise

Write an exports map exposing "." and "./client" with distinct import/require/types targets, and explain why (a) exports blocks deep imports the map doesn’t list, and (b) a consumer on moduleResolution: "bundler" vs "nodenext" may resolve types differently. Bonus: what does a trailing "default" condition do?

Pitfall: Shipping only main/types, or listing types after import/require — condition order matters, and nodenext consumers can get wrong or missing types even when the JS resolves fine.
R6. Error Handling Patterns Runtime Patterns 13 days since review · rep 2 → 3
Prefer a typed Result<T, E> (a discriminated union { ok: true; value } | { ok: false; error }) over throwing for expected failures: the error becomes part of the signature, callers must narrow before touching value, and exhaustiveness makes unhandled error kinds a compile error.
Key Pattern 
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

type ParseError = { kind: "empty" } | { kind: "nan"; raw: string };

function parseAmount(raw: string): Result<number, ParseError> {
  if (!raw) return { ok: false, error: { kind: "empty" } };
  const n = Number(raw);
  return Number.isNaN(n)
    ? { ok: false, error: { kind: "nan", raw } }
    : { ok: true, value: n };
}

const r = parseAmount("12");
if (r.ok) r.value.toFixed(2);   // value: number only inside the guard
callers must check r.ok before touching value / error
Challenge Exercise

Define Result<T, E> plus helpers ok(v) and err(e), then write map<T, U, E>(r: Result<T, E>, f: (t: T) => U): Result<U, E> and unwrapOr(r, fallback). Model a ValidationError union and handle every error.kind in an exhaustive switch with an assertNever default.

Pitfall: Typing the error channel as Error or any — you lose the discriminant and can’t exhaustively handle failure kinds. Give E a tagged union so each failure carries its own data and the compiler enforces handling.
1Node.js + TS: Server Foundations
1. Express: Typing Handlers, Middleware & Request Augmentation Node.js + TypeScript
Express is the most common Node HTTP framework, and its handlers are effectively untyped by default — one wrong req.body field and you ship a runtime crash. @types/express lets you type params, query, body, and the response, and safely augment Request for middleware-injected data like the authenticated user.

RequestHandler<Params, ResBody, ReqBody, Query> is the generic behind every handler; the Request/Response objects thread those parameters so req.params, req.body, and res.json() are all checked in one place.

Middleware that attaches data (an authenticated req.user, a req.requestId) needs declaration merging to extend Express.Request globally — otherwise the compiler doesn’t know the property exists and you fall back to any casts. And because Express 4 doesn’t await your handler, a rejected promise won’t reach error middleware unless you wrap it; a tiny typed asyncHandler keeps the signature honest (Express 5 forwards rejections natively).

1
Type a handler with RequestHandler<P, ResBody, ReqBody, Query> so params, body, query, and response are all checked.
2
Augment Express.Request via declare global to add middleware-injected fields like req.user — no any casts.
3
Wrap async handlers (Express 4) so rejections reach error middleware; Express 5 does this for you.
TypeScript · example 1 of 4 — a fully typed route handler
import express, { RequestHandler } from "express";

interface Params    { id: string }
interface UpdateBody { name: string; email?: string }
interface UserDTO    { id: string; name: string }

const updateUser: RequestHandler<Params, UserDTO, UpdateBody> =
  (req, res) => {
    const { id }   = req.params;   // string
    const { name } = req.body;     // string (email?: string)
    res.json({ id, name });        // body must match UserDTO
  };

express().put("/users/:id", updateUser);
req.params.id & req.body.name typed; res.json checked against UserDTO
TypeScript · example 2 of 4 — typing the query string
interface ListQuery { page?: string; sort?: "asc" | "desc" }

const list: RequestHandler<{}, unknown, unknown, ListQuery> =
  (req, res) => {
    const sort = req.query.sort ?? "asc";   // "asc" | "desc"
    // res.json(undefinedVar)  // error: not assignable to ResBody
    res.status(200).json({ sort });
  };
the 4th generic overrides the default ParsedQs; sort is "asc" | "desc" | undefined
TypeScript · example 3 of 4 — augment Request for req.user
// types/express.d.ts  — a global augmentation
import "express";

declare global {
  namespace Express {
    interface Request {
      user?: { id: string; roles: string[] };
    }
  }
}

// now, anywhere:
const auth: RequestHandler = (req, _res, next) => {
  req.user = { id: "u1", roles: ["admin"] };   // typed, no `any` cast
  next();
};
req.user is available on every handler with the declared shape
TypeScript · example 4 of 4 — async handler wrapper (Express 4)
import { RequestHandler } from "express";

const asyncHandler =
  (fn: RequestHandler): RequestHandler =>
  (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next);

app.get("/users/:id", asyncHandler(async (req, res) => {
  const user = await db.users.find(req.params.id);   // may reject
  res.json(user);
}));
a rejected promise now reaches Express error middleware
Common Mistakes:
❌ Reading req.body/req.params untyped (they default to loose types) and trusting the shape you assume.
✅ Parameterize RequestHandler<P, ResBody, ReqBody, Query> — and validate the body at runtime with a schema for real trust.
❌ Writing (req as any).user in every handler because a middleware attached it.
✅ Augment Express.Request once via declare global so req.user is typed everywhere.
Key Rules:
  • 2026-07-04 (Day 31)
  • RequestHandler<Params, ResBody, ReqBody, Query> types both req and res in one place.
  • The generics describe intent; incoming req.body/query are still unvalidated at runtime — pair with a schema validator (Zod).
  • Add middleware-injected properties by augmenting Express.Request, not by casting to any.
  • On Express 4, wrap async handlers so rejections hit error middleware; Express 5 handles this natively.
Practice Exercises:
1Type a POST /login handler whose ReqBody is { email: string; password: string } and ResBody is { token: string }. Confirm returning res.json({}) is a type error.
2Create a global augmentation adding req.requestId: string, set it in a middleware, and read it in a downstream handler with no cast.
3Write a generic typedBody<T>() middleware factory that parses req.body with a validator and narrows it to T for later handlers; explain why runtime validation is still required even though the generic “types” the body.
Connections: This applies Generics Basics and declaration merging (module augmentation) to a real framework, and the Zod pairing previews runtime validation — the compile-time generic and the runtime schema are two halves of a trustworthy request. Contrast Fastify (Concept 2), which infers these types from a schema instead.
2. Fastify Type Providers: End-to-End Route Typing Node.js + TypeScript
Fastify pairs each route with a JSON Schema for validation and fast serialization; a type provider bridges that runtime schema to static types, so request.body/params/querystring are inferred from the very schema that validates them — one source of truth, zero drift.

Register a provider (e.g. @fastify/type-provider-typebox) and call .withTypeProvider<…>() on the instance; every route then infers its request/reply types from its schema. With TypeBox you write Type.Object({ … }), which is both a runtime JSON Schema and a static type via Static<typeof T> — the provider wires the inference automatically.

Prefer no dependency? Skip the provider and pass an explicit route generic (Body, Params, Querystring, Reply) to .get/.post. It’s more manual, and your types no longer come from the validator, but it needs nothing extra installed.

1
Build the app and attach a provider: Fastify().withTypeProvider<TypeBoxTypeProvider>().
2
Give each route a schema built from Type.Object(…); request.body etc. infer from it.
3
Or, provider-free, supply a route generic to type request + reply by hand.
TypeScript · example 1 of 4 — provider + schema-inferred body
import Fastify from "fastify";
import { Type } from "@sinclair/typebox";
import { TypeBoxTypeProvider } from "@fastify/type-provider-typebox";

const app = Fastify().withTypeProvider<TypeBoxTypeProvider>();

app.post("/users", {
  schema: {
    body: Type.Object({ name: Type.String(), age: Type.Number() }),
  },
}, (req, reply) => {
  req.body.name;   // string  — inferred from the schema
  req.body.age;    // number
  reply.send({ ok: true });
});
req.body typed { name: string; age: number } straight from the schema
TypeScript · example 2 of 4 — Static<> extracts the type for reuse
import { Type, Static } from "@sinclair/typebox";

const UserSchema = Type.Object({ id: Type.String(), name: Type.String() });
type User = Static<typeof UserSchema>;   // { id: string; name: string }

// one TypeBox object validates at runtime AND types your code
function greet(u: User) { return `hi ${u.name}`; }
the schema is a single source for both JSON Schema and the TS type
TypeScript · example 3 of 4 — params, query & a typed reply
app.get("/users/:id", {
  schema: {
    params:      Type.Object({ id: Type.String() }),
    querystring: Type.Object({ verbose: Type.Optional(Type.Boolean()) }),
    response:    { 200: Type.Object({ id: Type.String(), name: Type.String() }) },
  },
}, (req, reply) => {
  const { id } = req.params;      // string
  const v = req.query.verbose;    // boolean | undefined
  reply.send({ id, name: "Ada" });   // checked against the 200 schema
});
params, query, and reply.send() all inferred from the schema
TypeScript · example 4 of 4 — provider-free route generic
interface Route {
  Params: { id: string };
  Body:   { name: string };
  Reply:  { updated: boolean };
}

app.put<Route>("/users/:id", (req, reply) => {
  req.params.id;   // string
  req.body.name;   // string
  reply.send({ updated: true });   // must match Reply
});
a RouteGenericInterface types request + reply without any provider
Common Mistakes:
❌ Writing a JSON Schema for validation and a separate interface for types — they drift the moment someone edits one.
✅ Use a type provider so the validating schema is the type; derive extra types with Static<>.
❌ Forgetting .withTypeProvider<…>(), then wondering why req.body is unknown.
✅ Attach the provider to the instance and reuse that typed instance so every route infers from its schema.
Key Rules:
  • A type provider infers body/params/querystring/reply from each route’s runtime schema.
  • TypeBox objects double as JSON Schema and static types; Static<typeof S> extracts the type.
  • No provider? Pass a route generic (Body, Params, Querystring, Reply) to the route method.
  • The response schema types reply.send() and drives fast serialization — keep it accurate.
Practice Exercises:
1Define a POST /todos route with a TypeBox body of { title: string; done?: boolean } and confirm req.body.done is boolean | undefined.
2Extract type Todo = Static<typeof TodoSchema> and reuse it in a plain function; change the schema and confirm the function’s parameter type updates with no separate edit.
3Re-implement the same route provider-free with a route generic, and list one advantage and one drawback versus the provider approach.
Connections: This is Zod & runtime validation generalized — schema-first typing where validator and type share one definition — and it leans on Static<> (a conditional/mapped inference over the schema). Contrast Express (Concept 1), where types and validation are separate concerns you wire yourself.
3. Node Streams & Buffers Typing Node.js + TypeScript
Streams move large data (files, HTTP bodies, sockets) through Node without buffering it all in memory, and Buffer is its binary byte container. @types/node types both — including the object-mode channel and the Buffer/Uint8Array relationship that trips up encoding code.

Readable, Writable, and Transform are typed, but async iteration (for await (const chunk of readable)) yields any by default — so annotate the chunk (Buffer for binary, your element type in object mode). Reach for member access on an un-annotated chunk and the compiler won’t save you.

Buffer extends Uint8Array, so a Buffer flows into a Uint8Array slot but not the reverse; Buffer.from(str, enc) and buf.toString(enc) are the typed conversion points between bytes and text. For composition, the pipeline from node:stream/promises gives a typed, awaitable, self-cleaning way to wire stages together with proper error propagation.

1
Consume a Readable with for await, annotating the chunk as Buffer (binary) or your object type (object mode).
2
Convert bytes ↔ text through Buffer; remember a Buffer is a Uint8Array.
3
Compose with pipeline from node:stream/promises for typed, awaitable, error-safe piping.
TypeScript · example 1 of 4 — async-iterate a readable
import { createReadStream } from "node:fs";

const stream = createReadStream("data.bin");   // Readable

let total = 0;
for await (const chunk of stream) {
  const buf = chunk as Buffer;   // default chunk type is any → annotate
  total += buf.length;
}
console.log(total);
total number of bytes read
TypeScript · example 2 of 4 — Buffer is a Uint8Array
const buf = Buffer.from("héllo", "utf8");
const bytes: Uint8Array = buf;     // ok: Buffer extends Uint8Array
// const back: Buffer = bytes;     // error: Uint8Array is not a Buffer
buf.toString("utf8");              // "héllo"
buf.length;                        // 6  (é is 2 bytes in UTF-8)
Buffer → Uint8Array widening ok; the reverse needs Buffer.from
TypeScript · example 3 of 4 — object-mode Transform carries a type
import { Transform } from "node:stream";

interface Row { id: number; name: string }

const toName = new Transform({
  objectMode: true,
  transform(row: Row, _enc, cb) {
    cb(null, row.name + "\n");   // push a string downstream
  },
});
objectMode lets chunks be objects (Row), not just Buffers
TypeScript · example 4 of 4 — typed, awaitable pipeline
import { pipeline } from "node:stream/promises";
import { createReadStream, createWriteStream } from "node:fs";
import { createGzip } from "node:zlib";

await pipeline(
  createReadStream("in.txt"),
  createGzip(),
  createWriteStream("in.txt.gz"),
);   // resolves on success; rejects (and cleans up) on any stage error
pipeline returns Promise<void>; errors reject instead of leaking
Common Mistakes:
❌ Assuming for await (const chunk of stream) gives a Buffer — it’s any, so chunk.toString() compiles even when the chunk is an object.
✅ Annotate the chunk (Buffer in binary mode, your type in object mode) so member access is checked.
❌ Typing a byte parameter as Buffer when a plain Uint8Array is passed (e.g. from a Web API) — the assignment fails.
✅ Accept Uint8Array for input and convert with Buffer.from(view) when you need Buffer-only methods.
Key Rules:
  • Readable/Writable/Transform are typed, but async-iterated chunks default to any — annotate them.
  • Buffer extends Uint8Array: a Buffer fits a Uint8Array slot, not the reverse.
  • Convert bytes ↔ string via Buffer.from(str, enc) / buf.toString(enc); watch multi-byte lengths.
  • Prefer pipeline from node:stream/promises for typed, awaitable, auto-cleaning composition.
Practice Exercises:
1Read a text file via createReadStream, annotate each chunk as Buffer, and accumulate the decoded string with chunk.toString("utf8"). Confirm omitting the annotation lets a bogus chunk.foo compile.
2Write bytesToHex(data: Uint8Array): string that accepts either a Buffer or a Uint8Array and returns hex; explain why the parameter is Uint8Array, not Buffer.
3Build an object-mode Transform that takes { id: number } objects and emits id * 2 as numbers, then compose it with pipeline; note where the object type is (and isn’t) enforced.
Connections: This is async patterns / async iterators applied to Node’s I/O, and the object-mode channel reuses generics over a runtime stream. Buffer extends Uint8Array is a live example of structural subtyping, and it pairs with Concept 4 — node: import specifiers are ESM-resolved.

4. Native ESM & Module Resolution: nodenext Node.js + TypeScript
Running TypeScript as native ESM in Node is full of sharp edges — "type": "module", mandatory .js extensions on relative imports, and the nodenext resolver. Getting moduleResolution right is the difference between clean imports and a cryptic ERR_MODULE_NOT_FOUND at runtime.

package.json "type": "module" makes your .ts/.js files ESM; set "module": "nodenext" and "moduleResolution": "nodenext" so TS resolves exactly like Node’s ESM loader — honoring exports, conditions, and file extensions.

Under ESM, relative imports need an explicit extension, and you write .js even from a .ts file: TS type-checks the source but Node runs the emitted .js. ESM also has no __dirname, require, or module.exports — use import.meta.url, createRequire, and default imports of CommonJS modules. Top-level await is ESM-only.

1
Set "type": "module" and "module"/"moduleResolution": "nodenext" so TS mirrors Node’s ESM resolver.
2
Write relative imports with a .js extension (even though the file is .ts).
3
Interop with CJS via default import or createRequire; reserve top-level await for ESM.
TypeScript · example 1 of 4 — the configuration
// package.json
{ "type": "module" }

// tsconfig.json
{
  "compilerOptions": {
    "module": "nodenext",
    "moduleResolution": "nodenext",
    "target": "es2022"
  }
}
TS now resolves imports the way Node’s ESM loader does
TypeScript · example 2 of 4 — the .js extension rule
// src/math.ts  exports add()
// src/main.ts:
import { add } from "./math.js";   // ✅ .js — TS resolves it to math.ts
add(2, 3);

// import { add } from "./math";   // ✗ error under nodenext: needs extension
// import { add } from "./math.ts";// ✗ error: don't import the .ts extension
relative ESM imports require the emitted .js extension
TypeScript · example 3 of 4 — importing a CommonJS dependency
import cjsPkg from "legacy-cjs";     // default import = module.exports
cjsPkg.parse("...");

import { createRequire } from "node:module";
const require = createRequire(import.meta.url);
const data = require("./config.json");   // when you must use require
CJS default-imports as module.exports; createRequire for require
TypeScript · example 4 of 4 — ESM-only capabilities
// top-level await works only in an ESM module
const res  = await fetch("https://example.com/data.json");
const json = await res.json();

// __dirname doesn't exist in ESM — reconstruct it:
import { fileURLToPath } from "node:url";
import { dirname } from "node:path";
const __dirname = dirname(fileURLToPath(import.meta.url));
top-level await + import.meta.url replace CJS __dirname / require
Common Mistakes:
❌ Using "moduleResolution": "node" (the classic CJS algorithm) for an ESM Node project — it ignores exports/conditions and lets extensionless imports “resolve” then crash at runtime.
✅ Use "nodenext" for both module and moduleResolution so compile-time resolution matches Node’s runtime loader.
❌ Importing "./util" or "./util.ts" from an ESM module.
✅ Import "./util.js" — the extension is required and must be the emitted .js, even from a .ts source.
Key Rules:
  • "type": "module" + nodenext makes TS resolve like Node’s ESM loader (extensions, exports, conditions).
  • Relative ESM imports need an explicit .js extension, written from the .ts file.
  • ESM has no __dirname/require/module.exports — use import.meta.url, createRequire, default imports.
  • Top-level await and import.meta are ESM-only; "bundler" resolution is for bundlers, "nodenext" is for Node.
Practice Exercises:
1Convert a small CJS project to ESM: add "type": "module", switch to nodenext, and fix every relative import to carry a .js extension. Note each error the compiler surfaces.
2Reconstruct __dirname in an ESM module using import.meta.url, and read a sibling JSON file relative to it.
3Import one CommonJS-only dependency two ways — a default import and via createRequire — and explain when you’d need the second.
Connections: This is module systems (ESM vs CJS) made concrete for Node, and it’s the runtime counterpart to package publishing (R5): the exports/conditions you author are exactly what nodenext reads when resolving. It also underpins Concept 5, where the test runner runs your ESM .ts directly.
5. Node’s Built-in Test Runner with TypeScript Node.js + TypeScript
Node ships a zero-dependency test runner (node:test) with a spec/TAP reporter and node:assert. Combined with Node’s native TypeScript support (type stripping) or a loader like tsx, you can run typed tests with no Jest/Vitest config at all.

Import test/describe/it from node:test and assertions from node:assert/strict; the whole API is typed via @types/node, and test() returns a promise so async tests just await. A crucial distinction: running .ts directly via type stripping (node --experimental-strip-types, or tsx) only erases the types — it does not type-check. Real checking still comes from a separate tsc --noEmit.

Structure suites with describe/it, nest subtests through the t context (t.test(…)), and create typed test doubles with mock.fn / t.mock — the mock preserves the original function’s signature and exposes a typed .mock.calls.

1
Write tests with node:test + node:assert/strict; @types/node types the whole API.
2
Run via type-stripping (node --experimental-strip-types) or tsx; keep a separate tsc --noEmit for checking.
3
Group with describe/it, nest subtests with the t context, and mock with mock.fn.

import { describe, it } from "node:test";
import assert from "node:assert/strict";

describe("user service", () => {
  it("loads a user", async (t) => {
    const user = await loadUser("u1");

      assert.deepEqual(user.roles, ["member"]);

TypeScript · example 4 of 4 — typed mocking
import { test, mock } from "node:test";
import assert from "node:assert/strict";

test("calls the notifier once", () => {
  const notify = mock.fn((msg: string) => msg.length);
  notify("hi");
  assert.equal(notify.mock.callCount(), 1);
  assert.equal(notify.mock.calls[0].arguments[0], "hi");
});
mock.fn preserves the fn signature; .mock.calls is typed
Common Mistakes:
❌ Assuming node --experimental-strip-types type-checks your tests — it only erases types, so type errors run anyway.
✅ Run tsc --noEmit (in CI / pre-commit) for real checking; use stripping / tsx purely to execute.
❌ Reaching for a third-party test framework out of habit when node:test/node:assert already cover the case, adding deps and config.
✅ Start with the built-ins; adopt Vitest/Jest only when you need their extras (rich watch, jsdom, snapshots).
Key Rules:
  • node:test + node:assert/strict is a zero-dependency, fully typed testing stack via @types/node.
  • Type stripping (--experimental-strip-types / tsx) executes .ts but does NOT type-check — keep tsc --noEmit separate.
  • Use describe/it, the t context for subtests, and mock.fn/t.mock for typed test doubles.
  • Import compiled specifiers with .js under nodenext, even in test files.
Practice Exercises:
1Write a node:test file for a pure function with three assert.equal cases; run it with --experimental-strip-types (or tsx), then introduce a type error and confirm the run still executes but tsc --noEmit flags it.
2Add a describe/it suite with one async it that awaits a promise, plus one nested t.test subtest.
3Use mock.fn to stand in for a dependency, assert it was called with specific arguments, and check callCount(); confirm the mock preserves the original function’s parameter types.
