R1. Variadic Tuple Types Advanced Type System 11 days since review · rep 2 → 3
A variadic tuple lets you spread a tuple/array type parameter inside another tuple — [...T, U], [U, ...T], or [...A, ...B]. Combined with generic rest parameters, it types functions whose arity is preserved: concat, tail, curry, promisify.
Key Pattern 
type Push<T extends unknown[], V>    = [...T, V];
type Unshift<T extends unknown[], V> = [V, ...T];
type Concat<A extends unknown[], B extends unknown[]> = [...A, ...B];

function tail<T extends unknown[]>(arr: readonly [unknown, ...T]): T {
  const [, ...rest] = arr;
  return rest as T;
}

type P = Push<[1, 2], 3>;      // [1, 2, 3]
type C = Concat<[1], [2, 3]>;  // [1, 2, 3]
spreading a tuple type param preserves element types and positions
Challenge Exercise

Implement Zip<A extends any[], B extends any[]> that pairs elements positionally, and a runtime curry whose type uses [...Args] to track remaining parameters. Then write Last<T extends any[]> via T extends [...any[], infer L] ? L : never and confirm it returns 3 for [1, 2, 3].

Pitfall: A tuple may contain at most one rest element. [...A, ...B] where both are open-ended arrays collapses to a single fuzzy array type, and a rest element combined with optional/leading elements narrows in surprising ways — keep one variadic slot and put fixed elements around it.
R2. Type-Safe Event Emitters Runtime Patterns 11 days since review · rep 2 → 3
Model events as a map (eventName → payload) and make on/emit generic over K extends keyof Events so the listener’s argument is derived from the event name. That turns Node’s stringly-typed EventEmitter into a checked contract.
Key Pattern 
interface Events {
  login:  { userId: string };
  logout: void;
  error:  Error;
}
class Emitter<E> {
  private m: { [K in keyof E]?: ((p: E[K]) => void)[] } = {};
  on<K extends keyof E>(k: K, fn: (p: E[K]) => void) { (this.m[k] ??= []).push(fn); }
  emit<K extends keyof E>(k: K, p: E[K]) { this.m[k]?.forEach(f => f(p)); }
}

const e = new Emitter<Events>();
e.on("login", p => p.userId);   // p: { userId: string }
// e.emit("login", {});          // error: userId missing
the payload type flows from the event key to the listener
Challenge Exercise

Add once and off, and support void payloads so emit("logout") is callable with no second argument (hint: a conditional/overload on whether E[K] is void). Then make emit return a boolean indicating whether any listener ran.

Pitfall: Falling back to string keys or any[] listener args reintroduces the exact stringly-typed hole you were fixing. And void payloads need special handling — without it, emit("logout") demands a phantom second argument.
R3. Builder Pattern with Type-Level State Patterns 11 days since review · rep 2 → 3
Encode which required fields have been set in a type parameter (a union of key literals). Each setter returns a builder whose type-state gains the new key, and build() uses a this parameter to compile only once every required key is present.
Key Pattern 
class Req<Set extends string = never> {
  private data: Record<string, unknown> = {};
  url(u: string): Req<Set | "url">       { this.data.url = u; return this as any; }
  method(m: string): Req<Set | "method"> { this.data.method = m; return this as any; }
  build(this: Req<"url" | "method">)     { return this.data; }
}

new Req().url("/x").method("GET").build();   // ok
// new Req().url("/x").build();               // error: method not set
build() is only callable once the required keys are in the type-state
Challenge Exercise

Track required vs optional keys separately so optional setters don’t gate build. Add a header(k, v) setter that can be called many times without changing the state, and confirm calling build before both required setters is still a compile error with a readable message.

Pitfall: The this-parameter on build is what enforces the state — drop it (or return a fresh any builder) and the guarantee vanishes. Keep the internal return this as any casts inside the class; they must never leak into the public type.
R4. TypeScript Performance Advanced Patterns & Performance 10 days since review · rep 2 → 3
Type-checking cost explodes with deep conditional/recursive types, huge unions, and heavy inference. Prefer interface extends over sprawling & intersections (the compiler caches them better), name intermediate types, cap recursion depth, and lean on --incremental + project references + skipLibCheck.
Key Pattern 
// slower: a deep intersection re-computed at every use site
type Big  = A & B & C & D & E;
// faster: a single named interface the compiler caches
interface Big2 extends A, B, C, D, E {}

// cap recursion so you don't hit "Type instantiation is excessively deep"
type Repeat<T, N extends number, Acc extends T[] = []> =
  Acc["length"] extends N ? Acc : Repeat<T, N, [...Acc, T]>;
interfaces cache; bounded recursion avoids depth-limit errors
Challenge Exercise

Run tsc --extendedDiagnostics and --generateTrace trace/ on a project, find the slowest type, and refactor one nested conditional into a mapped type or a plain interface. Measure the before/after check time and explain why interface extends caches better than a repeated intersection.

Pitfall: Reaching for ever-cleverer conditional types when a plain interface would check in a fraction of the time, and leaving skipLibCheck off so every .d.ts in node_modules is re-type-checked on each build.
R5. React + TypeScript: Component Props React + TypeScript 10 days since review · rep 2 → 3
Type props with an interface/type; inherit native element props with React.ComponentPropsWithoutRef<"button">, use PropsWithChildren for children, and discriminated unions for mutually-exclusive prop sets. Prefer explicit prop types over React.FC (which has an implicit-children quirk).
Key Pattern 
interface ButtonProps extends React.ComponentPropsWithoutRef<"button"> {
  variant?: "primary" | "ghost";
}
function Button({ variant = "primary", ...rest }: ButtonProps) {
  return <button className={variant} {...rest} />;
}
// inherits onClick, disabled, type, aria-*, ... from the native button
extending ComponentPropsWithoutRef pulls in every native prop, typed
Challenge Exercise

Build an Input that inherits native input props and adds a required label: string. Then make an IconButton with a discriminated union so exactly one of icon or children (never both, never neither) must be provided, and confirm passing both is a compile error.

Pitfall: Hand-listing native props (onClick, disabled…) instead of extending ComponentPropsWithoutRef — they drift and you silently miss dozens. And React.FC’s implicit children hides whether a component actually accepts them.
R6. Prisma — Type-Safe Database Access Database Typing 8 days since review · rep 2 → 3
Prisma generates a fully-typed client from your schema. select/include narrow the return type to exactly the fields you asked for, and helper types like Prisma.UserGetPayload<…> let you name those derived shapes — the schema is the single source of truth.
Key Pattern 
const user = await prisma.user.findUnique({
  where: { id },
  select: { id: true, email: true },
});
// user: { id: string; email: string } | null   — name is NOT present

type UserWithPosts = Prisma.UserGetPayload<{ include: { posts: true } }>;
select narrows the result; GetPayload names the derived shape
Challenge Exercise

Write a function whose return type is Prisma.UserGetPayload<{ select: { id: true; email: true } }> and show that reading .name on the result is a compile error. Then extract the select object into a reusable constant typed with satisfies Prisma.UserSelect, and reuse it across two queries.

Pitfall: Widening a selected result back to the full model type (User) — you reintroduce fields that were never fetched and will be undefined at runtime. Let Prisma’s derived payload types flow through instead of casting.

1. JWT Authentication: Typing Tokens & Claims Node.js + TypeScript
A JWT carries auth state in a signed token, but jwt.verify returns string | JwtPayload — effectively untyped. Define your claims shape once and wrap sign/verify so the rest of the app sees a real AuthClaims type, and validate the decoded payload (never trust its shape) before using it.

The library hands you a broad payload type; your job is to narrow it safely. sign should accept your typed claims, and verify should return AuthClaims only after a runtime check — because a token could be malformed, expired, or forged with a completely different shape. A bare as AuthClaims cast is a lie the compiler happily believes.

Registered claims (sub, exp, iat) are standard; custom claims (role, userId) are yours. Model both in one interface, and remember a JWT is signed, not encrypted — anyone can base64-decode the payload, so keep secrets out of it.

1
Define an AuthClaims interface (your custom claims + the registered ones you rely on).
2
Wrap sign(claims) and verify(token): AuthClaims, validating the decoded payload with a schema.
3
Attach the verified claims to req.user via declaration merging so handlers read a typed user.
TypeScript · example 1 of 4 — typed claims & sign
import jwt from "jsonwebtoken";

interface AuthClaims {
  sub: string;                    // user id (registered claim)
  role: "user" | "admin";
}

function signToken(claims: AuthClaims, secret: string): string {
  return jwt.sign(claims, secret, { expiresIn: "1h" });
}
sign only accepts the AuthClaims shape; the token embeds it
TypeScript · example 2 of 4 — verify + validate (parse, don’t assert)
import { z } from "zod";

const ClaimsSchema = z.object({
  sub:  z.string(),
  role: z.enum(["user", "admin"]),
});

function verifyToken(token: string, secret: string): AuthClaims {
  const decoded = jwt.verify(token, secret);   // string | JwtPayload
  return ClaimsSchema.parse(decoded);          // throws if the shape drifted
}
the return is a verified AuthClaims, not a loose JwtPayload
TypeScript · example 3 of 4 — jose (Web Crypto, promise-based)
import { SignJWT, jwtVerify } from "jose";

const key = new TextEncoder().encode(secret);
const token = await new SignJWT({ role: "admin" })
  .setProtectedHeader({ alg: "HS256" })
  .setSubject(userId)
  .setExpirationTime("1h")
  .sign(key);

const { payload } = await jwtVerify(token, key);
// payload: JWTPayload — still validate `role` before you trust it
jose returns a JWTPayload; narrow the custom claims yourself
TypeScript · example 4 of 4 — request augmentation for a typed req.user
declare global {
  namespace Express { interface Request { user?: AuthClaims } }
}

function auth(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.replace("Bearer ", "");
  if (!token) return res.status(401).end();
  req.user = verifyToken(token, SECRET);   // typed AuthClaims
  next();
}
downstream handlers read req.user as AuthClaims | undefined
Common Mistakes:
❌ Casting jwt.verify(...)’s result to your claims type — a forged or stale token with a different shape sails straight through.
✅ Validate the decoded payload with a schema (or a type predicate) so AuthClaims is a runtime guarantee, not a hope.
❌ Putting secrets or sensitive PII in the token payload — JWTs are signed, not encrypted, so anyone can base64-decode them.
✅ Store only non-sensitive identifiers/claims in the token; keep secrets server-side.
Key Rules:
  • 2026-07-04 (Day 31)
  • jwt.verify returns string | JwtPayload — narrow and validate before treating it as your claims.
  • Model custom + registered claims in one interface; sign against it.
  • JWTs are signed, not encrypted — never put secrets in the payload.
  • Attach verified claims to req.user via declaration merging for typed handlers.
Practice Exercises:
1Add exp/iat to the picture and handle TokenExpiredError from verify in the middleware, returning 401 with a clear message.
2Replace the Zod parse with a hand-written type predicate isAuthClaims(x): x is AuthClaims and use it to narrow the decoded payload; note the tradeoffs vs a schema.
3Type a refresh-token flow: a second RefreshClaims shape with only sub, and a rotate(token): { access: string; refresh: string }.
Connections: JWT verify is a trust-boundary parse (Zod / validate-on-read, like Redis caching), and the req.user augmentation is exactly Express request augmentation. It sets up Sessions (Concept 2, the stateful alternative) and RBAC (Concept 4, which reads req.user.role).
2. Typed Sessions & Cookies Node.js + TypeScript
Session auth keeps state server-side and hands the client an opaque cookie id. With express-session the req.session object is loosely typed by default; module augmentation lets you extend SessionData so req.session.userId is checked and autocompletes — and cookie options get typed the same way, so security flags aren’t forgotten.

express-session ships a SessionData interface you can augment. Once you add your fields, every req.session.x access is type-checked — no as any, no silent typos. Logging out is req.session.destroy(cb), which the types also cover.

Cookies themselves are just strings, so a typed cookie helper — name, serialized value, and typed CookieOptions (httpOnly, sameSite, secure, maxAge) — keeps them intentional, and a schema-backed reader parses a raw cookie back into T (a cookie is untrusted input).

1
Augment express-session’s SessionData with your session fields via declare module.
2
Read/write req.session.userId fully type-checked; call req.session.destroy to log out.
3
Type CookieOptions centrally, and parse any raw cookies through a schema.
TypeScript · example 1 of 4 — augment SessionData
import "express-session";

declare module "express-session" {
  interface SessionData {
    userId?: string;
    role?: "user" | "admin";
  }
}
req.session.userId is now string | undefined, checked everywhere
TypeScript · example 2 of 4 — typed session use
app.post("/login", (req, res) => {
  req.session.userId = user.id;     // checked against SessionData
  req.session.role   = "user";
  // req.session.usrId = "x";        // error: no such property
  res.sendStatus(200);
});

app.post("/logout", (req, res) => {
  req.session.destroy(() => res.sendStatus(204));
});
session fields autocomplete; mistyped keys are compile errors
TypeScript · example 3 of 4 — typed cookie options
import type { CookieOptions } from "express";

const authCookie: CookieOptions = {
  httpOnly: true,
  sameSite: "lax",
  secure: process.env.NODE_ENV === "production",
  maxAge: 1000 * 60 * 60,   // 1 hour
};

res.cookie("sid", sessionId, authCookie);
CookieOptions type-checks sameSite/secure/maxAge values
TypeScript · example 4 of 4 — a typed, validated cookie reader
function readCookie<T>(raw: string | undefined, schema: z.ZodType<T>): T | null {
  if (!raw) return null;
  try { return schema.parse(JSON.parse(raw)); }
  catch { return null; }
}

const prefs = readCookie(req.cookies.prefs, PrefsSchema);   // Prefs | null
a raw cookie string is parsed and validated into T (or null)
Common Mistakes:
❌ Casting req.session as any to bolt on fields — you lose all checking and typos compile happily.
✅ Augment SessionData once via declare module so every access is typed and autocompleted.
❌ Setting auth cookies without httpOnly/secure — JS can read them and they leak over plain HTTP.
✅ Type a shared CookieOptions with httpOnly, sameSite, and secure so every cookie is set consistently.
Key Rules:
  • Augment express-session’s SessionData — don’t cast req.session.
  • The session id lives in an opaque cookie; the data stays server-side.
  • Type CookieOptions centrally so security flags aren’t forgotten.
  • Parse raw cookies through a schema — a cookie value is untrusted input.
Practice Exercises:
1Add createdAt: number to SessionData and write an idle-timeout middleware that destroys sessions older than N minutes.
2Write a typed setAuthCookie(res, id) helper that always applies the shared secure options; confirm callers can’t forget httpOnly.
3Sketch a short JWT-vs-session comparison (where state lives, revocation, scaling) and note which piece your readCookie<T> reuses.
Connections: This is declaration merging / module augmentation applied to express-session — the stateful counterpart to JWT (Concept 1). The typed cookie reader is another Zod trust-boundary parse (like Redis validate-on-read), and req.session.role feeds RBAC (Concept 4).
3. OAuth & Passport Strategy Typing Node.js + TypeScript
OAuth delegates login to a provider (Google, GitHub) and hands back a profile. Passport is the standard Node middleware, but its callbacks are loosely typed. @types/passport* plus a typed app User and a module-augmented Express.User make the verify callback and req.user type-safe end to end.

A strategy’s verify callback receives provider tokens + a profile and calls done(err, user). Type the user you hand to done, then augment Express.User to that shape so req.user everywhere matches — the mapping from raw profile to your domain User happens in exactly one place.

serializeUser/deserializeUser decide what goes into the session (usually just an id) and how it’s rehydrated. Type both so the stored id type and the resolved user type line up — a mismatch there is a classic source of req.user being subtly wrong.

1
Define your app User type and augment Express.User to it.
2
Type the verify callback: map provider profile → User, then done(null, user).
3
Type serializeUser (User → id) and deserializeUser (id → User) so session round-trips are checked.
TypeScript · example 1 of 4 — augment Express.User
interface AppUser {
  id: string;
  email: string;
  provider: "google" | "github";
}

declare global {
  namespace Express { interface User extends AppUser {} }
}
req.user is AppUser across every handler
TypeScript · example 2 of 4 — typed strategy verify callback
import { Strategy as GitHubStrategy } from "passport-github2";

passport.use(new GitHubStrategy(
  { clientID, clientSecret, callbackURL },
  (accessToken, refreshToken, profile, done) => {
    const user: AppUser = {
      id: profile.id,
      email: profile.emails?.[0]?.value ?? "",
      provider: "github",
    };
    done(null, user);          // done is a typed VerifyCallback
  },
));
the user handed to done() must match AppUser
TypeScript · example 3 of 4 — typed serialize / deserialize
passport.serializeUser<string>((user, done) => {
  done(null, (user as AppUser).id);        // store only the id
});

passport.deserializeUser<string>(async (id, done) => {
  const user = await repo.findById(id);    // AppUser | null
  done(null, user ?? false);               // false = "no such user"
});
the id type and the resolved user flow through both hooks
TypeScript · example 4 of 4 — a guard that relies on the typed req.user
function ensureAuth(req: Request, res: Response, next: NextFunction) {
  if (!req.user) return res.status(401).end();
  req.user.email;             // string — from AppUser
  next();
}

app.get("/me", ensureAuth, (req, res) => res.json(req.user));
req.user is AppUser | undefined; the guard narrows it
Common Mistakes:
❌ Leaving Express.User as the default {} and reading req.user.email — it’s untyped and every access is an unchecked guess.
✅ Augment Express.User to your app User so req.user is a real, checked type.
❌ Storing the whole profile in the session via serializeUser — it bloats the cookie/store and drifts from your domain shape.
✅ Serialize to an id and deserializeUser back into a full User, typing both ends.
Key Rules:
  • Augment Express.User to your app User; don’t read an untyped req.user.
  • The verify callback maps provider profile → your User; done(err, user) is typed.
  • serializeUser/deserializeUser should store an id and rehydrate — type the id and the user.
  • req.user is User | undefined; a guard narrows it before handlers use it.
Practice Exercises:
1Add a Google strategy alongside GitHub, treat provider as a discriminant, and write a function that branches per provider with an assertNever default.
2Type deserializeUser to pass false (not null) on a missing user, and explain what Passport does with each value.
3Extract a reusable mapProfile(profile): AppUser and unit-test it without Passport in the loop.
Connections: Passport sits on top of Sessions (Concept 2) — serializeUser writes the session, deserializeUser reads it. The Express.User augmentation is the same declaration-merging trick as req.user in JWT (Concept 1), and the resolved AppUser typically comes from the repository (07-01, Concept 1).

4. Role-Based Access Control with Typed Permissions Node.js + TypeScript
Once a user is authenticated you must decide what they may do. Modeling roles and permissions as literal-union types (not loose strings) lets the compiler catch a typo’d permission, and a typed can(user, permission) guard gates routes — authorization becomes checked data, not scattered if (role === "admin") branches.

Define Permission as a union of literals and a role → permissions map typed with satisfies Record<Role, readonly Permission[]>. The satisfies makes a non-existent permission a compile error and forces every role to be covered, while keeping the exact literal values for later inference.

A can(user, perm) predicate checks membership; a requirePermission(perm) middleware factory returns a guard. Because Permission is a closed union, requirePermission("post:delele") (typo) simply won’t compile — the mistake is caught at the call site, not in production.

1
Define Role and Permission as literal unions.
2
Build a rolePermissions map with satisfies Record<Role, readonly Permission[]> for exhaustiveness.
3
Write can(user, perm) and a requirePermission(perm) middleware factory.
TypeScript · example 1 of 4 — literal roles & permissions
type Role = "viewer" | "editor" | "admin";

type Permission =
  | "post:read" | "post:write" | "post:delete" | "user:manage";
Permission is a closed set — typos become compile errors
TypeScript · example 2 of 4 — an exhaustive role → permission map
const rolePermissions = {
  viewer: ["post:read"],
  editor: ["post:read", "post:write"],
  admin:  ["post:read", "post:write", "post:delete", "user:manage"],
} satisfies Record<Role, readonly Permission[]>;
// omit a role → error; write "post:remove" → error (not a Permission)
every Role must be present and only real Permissions are allowed
TypeScript · example 3 of 4 — a typed can() predicate
interface User { id: string; role: Role }

function can(user: User, perm: Permission): boolean {
  return rolePermissions[user.role].includes(perm);
}

can(admin, "user:manage");    // true
// can(admin, "user:mange");  // error: not a Permission
both the role lookup and the permission argument are checked
TypeScript · example 4 of 4 — a middleware factory
function requirePermission(perm: Permission) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.user) return res.status(401).end();
    if (!can(req.user, perm)) return res.status(403).end();
    next();
  };
}

app.delete("/posts/:id", requirePermission("post:delete"), handler);
requirePermission only accepts a real Permission literal
Common Mistakes:
❌ Checking if (req.user.role === "admin") inline across handlers — role logic scatters and a renamed role silently misses spots.
✅ Centralize in a rolePermissions map + can(); routes ask for a permission, not a role.
❌ Typing permissions as string — every typo compiles and you get no autocomplete or exhaustiveness.
✅ Use a Permission literal union with satisfies so typos fail and every role is covered.
Key Rules:
  • Model Role/Permission as literal unions, not string.
  • satisfies Record<Role, readonly Permission[]> enforces coverage + valid permissions while keeping the literals.
  • Gate routes on permissions via a requirePermission factory, not inline role checks.
  • can(user, perm) is the one place membership is decided.
Practice Exercises:
1Add a "comment:moderate" permission and confirm the map fails to compile until a role owns it; add a "guest" role and observe the missing-role error.
2Support inheritance (admin ⊇ editor ⊇ viewer) by composing the arrays, keeping the satisfies check intact.
3Write hasAll(user, perms: Permission[]) and hasAny(...), and a route that needs two permissions.
Connections: Permission unions + satisfies are literal types and the satisfies operator applied to authz, and can()’s exhaustive map echoes never / exhaustiveness. It reads the role set by JWT (Concept 1), Sessions (Concept 2), or Passport (Concept 3) — authentication answers “who”, RBAC answers “what may they do”.
5. API Keys & Rate Limiting Typing Node.js + TypeScript
Public APIs authenticate machines with API keys and protect themselves with rate limits. Typing the key metadata (owner, scopes, tier) and the rate-limit config/result turns two error-prone, stringly bits of infrastructure into checked contracts — and lets scopes reuse the same literal-union authorization pattern as RBAC.

An API key maps to a typed record: an owner id, allowed scopes (a literal union), and a tier that selects a rate-limit policy. Look the key up in a store, attach the typed record to req.apiKey, and gate on scopes exactly like permissions.

Rate limiting has a typed config (windowMs, max) and a typed result ({ allowed; remaining; resetAt }). Whether you reach for express-rate-limit or a custom token bucket, keeping the result typed lets handlers populate X-RateLimit-* headers from real fields instead of guessing.

1
Type an ApiKey record (owner, scopes, tier) and augment req.apiKey.
2
Type the rate-limit config and a RateLimitResult your limiter returns.
3
Gate on scopes (reuse the permission idea) and pick per-tier limits.
TypeScript · example 1 of 4 — a typed API key record
type Scope = "read" | "write" | "admin";
type Tier  = "free" | "pro";

interface ApiKey {
  id: string; ownerId: string;
  scopes: readonly Scope[]; tier: Tier;
}

declare global {
  namespace Express { interface Request { apiKey?: ApiKey } }
}
req.apiKey is ApiKey | undefined; scopes are a closed set
TypeScript · example 2 of 4 — a lookup middleware
async function apiKeyAuth(req: Request, res: Response, next: NextFunction) {
  const raw = req.header("x-api-key");
  if (!raw) return res.status(401).end();
  const key = await keyStore.find(raw);   // ApiKey | null
  if (!key) return res.status(401).end();
  req.apiKey = key;                        // typed
  next();
}
a valid key attaches a typed ApiKey to the request
TypeScript · example 3 of 4 — a scope guard (reusing the permission pattern)
function requireScope(scope: Scope) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.apiKey?.scopes.includes(scope)) return res.status(403).end();
    next();
  };
}

app.post("/data", apiKeyAuth, requireScope("write"), handler);
requireScope only accepts a real Scope literal
TypeScript · example 4 of 4 — typed rate-limit config & result
interface RateLimitConfig { windowMs: number; max: number }
interface RateLimitResult { allowed: boolean; remaining: number; resetAt: number }

const limits = {
  free: { windowMs: 60_000, max: 60 },
  pro:  { windowMs: 60_000, max: 600 },
} satisfies Record<Tier, RateLimitConfig>;

function check(key: ApiKey, hits: number): RateLimitResult {
  const { max, windowMs } = limits[key.tier];
  return { allowed: hits < max, remaining: Math.max(0, max - hits), resetAt: Date.now() + windowMs };
}
per-tier config is exhaustive; result fields feed X-RateLimit headers
Common Mistakes:
❌ Passing API keys as query params or typing scopes as string[] — keys leak in logs and scope typos slip through.
✅ Read keys from a header, and type scopes as a Scope union so guards and typos are checked.
❌ Returning a bare boolean from the limiter — handlers can’t populate remaining/resetAt headers accurately.
✅ Return a typed RateLimitResult so X-RateLimit-Remaining/Reset come from real fields.
Key Rules:
  • Type the ApiKey record (owner, scopes, tier) and augment req.apiKey.
  • Scopes are literal unions — gate with a requireScope factory, mirroring RBAC.
  • satisfies Record<Tier, RateLimitConfig> keeps per-tier limits exhaustive.
  • A typed RateLimitResult lets handlers emit accurate rate-limit headers.
Practice Exercises:
1Add a "metrics:read" scope and a route requiring it; confirm a key without it gets 403 and a typo’d scope won’t compile.
2Implement a simple in-memory token bucket returning RateLimitResult, and set the three X-RateLimit-* headers from it.
3Make check also return retryAfter?: number only when allowed is false, using a discriminated result union.
Connections: API-key scopes reuse Concept 4’s literal-union + satisfies authorization pattern (machine authz vs user authz), and the key lookup mirrors the auth middleware of JWT / Sessions / Passport. The per-tier satisfies Record<Tier, ...> is the exhaustiveness trick again, and a typed RateLimitResult is a small discriminated result like the Result types from discriminated unions.
