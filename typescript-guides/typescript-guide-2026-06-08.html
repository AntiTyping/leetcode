R1. Decorators (TC39) Functions & Patterns rep 1 · last reviewed 2026-06-05 · 3 days ago
TC39 stage 3 decorators replace the old experimentalDecorators. A class decorator receives the class constructor and a context object. A method decorator receives the method function and context with addInitializer.
Key Pattern 
// TC39 method decorator signature
function logged<T extends (...args: any[]) => any>(
  target: T,
  context: ClassMethodDecoratorContext
): T | void {
  const name = String(context.name);
  return function (this: ThisParameterType<T>, ...args: Parameters<T>) {
    console.log(`Entering ${name}`);
    const result = target.call(this, ...args);
    console.log(`Exiting ${name}`);
    return result;
  } as T;
}

class UserService {
  @logged
  getUser(id: string) { return { id, name: "Alice" }; }
}

new UserService().getUser("u-1");
Entering getUser Exiting getUser
Challenge Exercise

Write a @logged method decorator that logs entry/exit with the method name and timestamps. Ensure the decorator preserves the original return value.

#1 Pitfall: TC39 decorators are NOT the same as legacy experimentalDecorators. The old API uses (target, propertyKey, descriptor); the new API uses (value, context). Mixing them causes silent failures.
R2. Module Systems Modules & Project Structure rep 1 · last reviewed 2026-06-05 · 3 days ago
ESM uses import/export (static, async, tree-shakeable). CJS uses require()/module.exports (dynamic, sync). Re-exports: export { foo } from './module'.
Key Pattern 
// Re-export barrel file (index.ts)
export { UserService } from './user.service';
export { OrderService } from './order.service';
export { type Config, defaultConfig } from './config';

// Renamed re-export
export { UserService as UserSvc } from './user.service';

// Re-export everything
export * from './utils';

// Re-export with namespace
export * as validators from './validators';
Challenge Exercise

Create a barrel file that re-exports from 3 modules with at least one renamed export and one type-only re-export.

#1 Pitfall: CJS require() is synchronous and evaluated at runtime; ESM import is async and statically analyzable. Mixing them (e.g., require() in an ESM file) causes runtime errors in Node.js.

R6. Union Types & Narrowing Core Type System rep 2 · last reviewed 2026-06-01 · 7 days ago
Union types (A | B) require narrowing before accessing type-specific properties. Narrow with typeof, instanceof, in, or discriminant properties. Discriminated unions use a shared literal field (e.g., kind).
Key Pattern 
type ClickEvent  = { kind: "click";  x: number; y: number };
type KeyEvent    = { kind: "key";    key: string; code: number };
type ScrollEvent = { kind: "scroll"; deltaY: number };
type FocusEvent  = { kind: "focus";  target: string };

type AppEvent = ClickEvent | KeyEvent | ScrollEvent | FocusEvent;

function handleEvent(event: AppEvent): string {
  switch (event.kind) {
    case "click":  return `Clicked at (${event.x}, ${event.y})`;
    case "key":    return `Key pressed: ${event.key}`;
    case "scroll": return `Scrolled by ${event.deltaY}`;
    case "focus":  return `Focused on ${event.target}`;
  }
}

console.log(handleEvent({ kind: "click", x: 10, y: 20 }));
Clicked at (10, 20)
Challenge Exercise

Build a type-safe event handler using discriminated unions for 4 event types: click, key, scroll, and resize. Ensure the switch is exhaustive (add a never check).

#1 Pitfall: Forgetting that typeof null === 'object' in JavaScript — narrowing with typeof x === 'object' doesn't eliminate null. Always check x !== null && typeof x === 'object'.
1 React + TypeScript: Component Props
Component Props React + TypeScript Ecosystem
Typing React component props catches misuse at compile time — wrong prop types, missing required props, invalid children. This is where most React + TypeScript value comes from.

There are several ways to type a functional component's props in React. The key decision is whether to use React.FC or explicit type annotations.

  • 2026-07-04 (Day 31)
  • React.FC<Props> was popular but is falling out of favor — it used to implicitly include children (fixed in React 18), and it doesn't support generic components
  • Inline annotation (function Button(props: ButtonProps)) is now preferred
  • Children should be explicitly typed as React.ReactNode (the broadest type, accepting elements, strings, numbers, null, arrays)
  • Event handlers have precise types: React.MouseEvent<HTMLButtonElement>, React.ChangeEvent<HTMLInputElement>
  • ComponentPropsWithoutRef<'button'> lets you inherit all native HTML attributes for wrapper components
  • Discriminated union props create polymorphic components with type-safe conditional props
basic-props.tsx
import React from 'react';

// Explicit annotation (preferred over React.FC)
interface ButtonProps {
  label: string;
  variant?: 'primary' | 'secondary' | 'danger';
  disabled?: boolean;
  onClick: () => void;
}

function Button({ label, variant = 'primary', disabled = false, onClick }: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant}`}
      disabled={disabled}
      onClick={onClick}
    >
      {label}
    </button>
  );
}

// Usage:
<Button label="Save" onClick={() => save()} />
// <Button label="Save" />  // Error: onClick is missing
// <Button label={42} onClick={() => {}} />  // Error: number not assignable to string

function ContactForm({ onSubmit }: FormProps) {
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value);  // string, fully typed

      name: form.get('name') as string,
      email: form.get('email') as string,
    });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="name" onChange={handleChange} />
      <input name="email" type="email" onChange={handleChange} />
      <button type="submit">Send</button>
    </form>
  );
}

discriminated-union-props.tsx
// Polymorphic component: link OR button, never both
type LinkAction = {
  variant: 'link';
  href: string;
  target?: '_blank' | '_self';
};

type ButtonAction = {
  variant: 'button';
  onClick: () => void;
  disabled?: boolean;
};

type ActionProps = (LinkAction | ButtonAction) & { label: string };

function Action(props: ActionProps) {
  if (props.variant === 'link') {
    return <a href={props.href} target={props.target}>{props.label}</a>;
  }
  return <button onClick={props.onClick} disabled={props.disabled}>{props.label}</button>;
}

// Type-safe usage:
<Action variant="link" label="Docs" href="/docs" />
<Action variant="button" label="Save" onClick={() => save()} />

// Error: href not valid for button variant
// <Action variant="button" label="X" onClick={() => {}} href="/bad" />
Object literal may only specify known properties, and 'href' does not exist in type 'ButtonAction & { label: string }'.
Common Mistakes
Using React.FC<Props> — it doesn't support generic components and adds an implicit children? in older React versions
Use explicit annotation: function MyComponent(props: Props): JSX.Element
Using any for event handlers: onClick: (e: any) => void
Use precise types: onClick: (e: React.MouseEvent<HTMLButtonElement>) => void
Not using discriminated unions for conditional props (e.g., href?: string; onClick?: () => void)
Use discriminated unions so exactly one variant is enforced at compile time
Key Rules
  • Prefer explicit return types over React.FC
  • Use ComponentPropsWithoutRef for wrapper components that forward native HTML attributes
  • Use discriminated unions for polymorphic components (link vs button, modal vs drawer, etc.)
  • Always type children explicitly as React.ReactNode if your component accepts them

2 React + TypeScript: Hooks Typing
Hooks Typing React + TypeScript Ecosystem
Properly typed hooks prevent runtime errors from incorrect state shapes, wrong ref types, and mismatched reducer actions. Hooks are where most React logic lives — getting the types right here pays dividends everywhere.

React hooks have excellent TypeScript support, but several patterns require explicit generic annotations:

  • useState<T> needs an explicit generic when the initial value doesn't capture the full type (e.g., useState<User | null>(null))
  • useRef<HTMLInputElement>(null) creates a RefObject (read-only .current). useRef<number>(0) creates a MutableRefObject
  • useReducer shines with discriminated union actions — each action type gets its own payload shape
  • Custom hooks returning tuples need as const to prevent TypeScript from widening [string, () => void] to (string | (() => void))[]
  • Typed context with createContext<T | undefined>(undefined) needs a guarded hook that throws when used outside a provider
useState-typed.tsx
import { useState } from 'react';

// Simple: TypeScript infers from initial value
const [count, setCount] = useState(0);          // number
const [name, setName] = useState('');            // string

// Explicit generic needed when initial doesn't match full type
interface User {
  id: string;
  name: string;
  email: string;
}

type LoadingState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: User }
  | { status: 'error'; error: string };

function UserProfile({ userId }: { userId: string }) {
  const [state, setState] = useState<LoadingState>({ status: 'idle' });

  async function loadUser() {
    setState({ status: 'loading' });
    try {
      const user = await fetchUser(userId);
      setState({ status: 'success', data: user });
    } catch (e) {
      setState({ status: 'error', error: String(e) });
    }
  }

  // TypeScript narrows inside each case
  switch (state.status) {
    case 'loading': return <p>Loading...</p>;
    case 'success': return <p>{state.data.name}</p>;  // data is available
    case 'error':   return <p>{state.error}</p>;       // error is available
    default:        return <button onClick={loadUser}>Load</button>;
  }
}
useRef-typed.tsx
import { useRef, useEffect } from 'react';

function SearchInput() {
  // DOM ref: nullable initial = RefObject (read-only .current)
  const inputRef = useRef<HTMLInputElement>(null);

  // Mutable ref: non-null initial = MutableRefObject
  const renderCount = useRef<number>(0);

  useEffect(() => {
    // Must null-check DOM refs
    inputRef.current?.focus();
    renderCount.current += 1;  // No null check needed: MutableRefObject
    console.log(`Rendered ${renderCount.current} times`);
  });

  return <input ref={inputRef} placeholder="Search..." />;
}

// Key distinction:
// useRef<HTMLDivElement>(null)   => RefObject<HTMLDivElement>       (.current is readonly)
// useRef<number>(0)             => MutableRefObject<number>         (.current is mutable)
useReducer-typed.tsx
import { useReducer } from 'react';

interface TodoState {
  todos: { id: number; text: string; done: boolean }[];
  nextId: number;
}

// Discriminated union for actions
type TodoAction =
  | { type: 'ADD';    text: string }
  | { type: 'TOGGLE'; id: number }
  | { type: 'DELETE'; id: number }
  | { type: 'CLEAR_DONE' };

function todoReducer(state: TodoState, action: TodoAction): TodoState {
  switch (action.type) {
    case 'ADD':
      return {
        ...state,
        todos: [...state.todos, { id: state.nextId, text: action.text, done: false }],
        nextId: state.nextId + 1,
      };
    case 'TOGGLE':
      return {
        ...state,
        todos: state.todos.map(t => t.id === action.id ? { ...t, done: !t.done } : t),
      };
    case 'DELETE':
      return { ...state, todos: state.todos.filter(t => t.id !== action.id) };
    case 'CLEAR_DONE':
      return { ...state, todos: state.todos.filter(t => !t.done) };
  }
}

function TodoApp() {
  const [state, dispatch] = useReducer(todoReducer, { todos: [], nextId: 1 });

  dispatch({ type: 'ADD', text: 'Learn TypeScript' });   // OK
  dispatch({ type: 'TOGGLE', id: 1 });                    // OK
  // dispatch({ type: 'ADD', id: 1 });                    // Error: 'id' not in ADD action
  // dispatch({ type: 'UNKNOWN' });                       // Error: not assignable to TodoAction

  return null;
}
Object literal may only specify known properties, and 'id' does not exist in type '{ type: "ADD"; text: string; }'.
custom-hook-return.tsx
import { useState, useCallback } from 'react';

// Without `as const`, TS infers: (boolean | (() => void))[]
// With `as const`, TS infers: [boolean, () => void, () => void]
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue(v => !v), []);
  const reset = useCallback(() => setValue(initial), [initial]);

  return [value, toggle, reset] as const;
}

// Usage:
function Panel() {
  const [isOpen, toggle, reset] = useToggle(false);
  //     ^ boolean  ^ () => void  ^ () => void (precise types!)
  return (
    <div>
      <button onClick={toggle}>{isOpen ? 'Close' : 'Open'}</button>
      <button onClick={reset}>Reset</button>
    </div>
  );
}
typed-context.tsx
import { createContext, useContext, useState, ReactNode } from 'react';

interface AuthContextType {
  user: { id: string; name: string } | null;
  login: (name: string) => void;
  logout: () => void;
}

// Create with undefined default; guard at the hook level
const AuthContext = createContext<AuthContextType | undefined>(undefined);

// Guarded hook: throws descriptive error if used outside provider
function useAuth(): AuthContextType {
  const ctx = useContext(AuthContext);
  if (ctx === undefined) {
    throw new Error('useAuth must be used within an AuthProvider');
  }
  return ctx;  // AuthContextType, not AuthContextType | undefined
}

function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<{ id: string; name: string } | null>(null);
  const login = (name: string) => setUser({ id: crypto.randomUUID(), name });
  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// Consumer: no undefined check needed
function UserBadge() {
  const { user } = useAuth();   // AuthContextType guaranteed
  return user ? <span>{user.name}</span> : <span>Guest</span>;
}
Common Mistakes
Not providing type to useState when it starts as null: const [user, setUser] = useState(null) infers null forever
Provide explicit generic: const [user, setUser] = useState<User | null>(null)
Forgetting as const on custom hook tuple returns: return [value, setter] infers (T | SetterFn)[]
Add as const: return [value, setter] as const to get [T, SetterFn]
Using any in reducer actions: type Action = { type: string; payload: any }
Use discriminated unions with precise payloads for each action type
Key Rules
  • Always type useRef for DOM elements as nullable: useRef<HTMLElement>(null)
  • Use discriminated unions for useReducer action types
  • Use as const for custom hook tuple returns
  • Guard context hooks with a runtime check and throw if used outside the provider

3 React + TypeScript: Generic Components
Generic Components React + TypeScript Advanced
Generic components let you build reusable UI (lists, tables, selects) that maintain type safety for any data shape. Instead of using any for item types, the consumer gets full autocomplete and error checking.

A generic React component is a function component with a type parameter. The generic flows through props so the consumer gets precise types:

  • Basic pattern: function List<T>(props: { items: T[]; renderItem: (item: T) => ReactNode })
  • Constraints: <T extends { id: string }> ensures items have required fields for keys
  • forwardRef with generics is tricky — React's forwardRef loses the generic. Use a type assertion wrapper or the new React 19 ref prop
  • Generic table components use column definitions that reference keys of the data type
  • Keep generics simple — if your component only handles 2-3 fixed types, a union is clearer than a generic
generic-list.tsx
import { ReactNode } from 'react';

interface ListProps<T> {
  items: T[];
  renderItem: (item: T, index: number) => ReactNode;
  keyExtractor: (item: T) => string | number;
}

function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return (
    <ul>
      {items.map((item, i) => (
        <li key={keyExtractor(item)}>{renderItem(item, i)}</li>
      ))}
    </ul>
  );
}

// Usage: T is inferred as User
interface User { id: string; name: string; email: string }
const users: User[] = [
  { id: '1', name: 'Alice', email: 'alice@test.com' },
  { id: '2', name: 'Bob', email: 'bob@test.com' },
];

<List
  items={users}
  keyExtractor={u => u.id}      // u is User
  renderItem={u => u.name}       // u is User, full autocomplete
/>

// Type error if renderItem doesn't match item type:
// renderItem={(u) => u.age}   // Error: 'age' does not exist on type 'User'
Property 'age' does not exist on type 'User'.
generic-select.tsx
import { ReactNode } from 'react';

interface SelectProps<T extends Record<string, any>> {
  items: T[];
  labelKey: keyof T;
  valueKey: keyof T;
  selected: T[keyof T] | null;
  onChange: (value: T[keyof T]) => void;
}

function Select<T extends Record<string, any>>({
  items, labelKey, valueKey, selected, onChange
}: SelectProps<T>) {
  return (
    <select
      value={selected != null ? String(selected) : ''}
      onChange={e => {
        const item = items.find(i => String(i[valueKey]) === e.target.value);
        if (item) onChange(item[valueKey]);
      }}
    >
      <option value="">Select...</option>
      {items.map(item => (
        <option key={String(item[valueKey])} value={String(item[valueKey])}>
          {String(item[labelKey])}
        </option>
      ))}
    </select>
  );
}

// Usage: T inferred as { code: string; name: string }
const countries = [
  { code: 'US', name: 'United States' },
  { code: 'UK', name: 'United Kingdom' },
];

<Select
  items={countries}
  labelKey="name"       // autocomplete: "code" | "name"
  valueKey="code"       // autocomplete: "code" | "name"
  selected="US"
  onChange={v => console.log(v)}
/>

generic-table.tsx
import { ReactNode } from 'react';

interface Column<T> {
  key: keyof T & string;
  header: string;
  render?: (value: T[keyof T], row: T) => ReactNode;
}

interface TableProps<T extends { id: string | number }> {
  data: T[];
  columns: Column<T>[];
}

function Table<T extends { id: string | number }>({ data, columns }: TableProps<T>) {
  return (
    <table>
      <thead>
        <tr>{columns.map(col => <th key={col.key}>{col.header}</th>)}</tr>
      </thead>
      <tbody>
        {data.map(row => (
          <tr key={row.id}>
            {columns.map(col => (
              <td key={col.key}>
                {col.render ? col.render(row[col.key], row) : String(row[col.key])}
              </td>
            ))}

4 Node.js + TypeScript
Node.js + TypeScript Node.js Ecosystem
TypeScript in Node.js requires understanding @types/node, proper module configuration, and Node-specific patterns like typed streams, fs/promises, and process.env. Getting these right prevents subtle runtime bugs.

Node.js doesn't natively understand TypeScript (unless using tsx/ts-node or Node 22+ with --experimental-strip-types). Key patterns:

  • @types/node provides ambient type declarations for all Node.js built-in APIs (fs, path, http, etc.)
  • fs/promises returns properly typed Promises — prefer it over callback-based fs
  • process.env is typed as Record<string, string | undefined> — every value might be undefined
  • Declaration merging lets you augment ProcessEnv with your app's specific variables
  • Native ESM in Node requires "type": "module" in package.json and .mts/.cts extensions for disambiguation
  • Worker threads communicate via parentPort.postMessage() — type the messages with a discriminated union

async function loadConfig(path: string): Promise<Record<string, unknown>> {

  return JSON.parse(content);
}

// Typed directory listing
async function listTsFiles(dir: string): Promise<string[]> {
  const entries = await readdir(dir, { withFileTypes: true });
  // entries is Dirent[] — each has .name, .isFile(), .isDirectory()
  return entries
    .filter(e => e.isFile() && e.name.endsWith('.ts'))
    .map(e => join(dir, e.name));
}

// Error handling: Node errors have a `code` property
async function safeRead(path: string): Promise<string | null> {
  try {
    return await readFile(path, 'utf-8');
  } catch (err) {
    if (err instanceof Error && 'code' in err && err.code === 'ENOENT') {
      return null;  // File not found
    }
    throw err;  // Re-throw unexpected errors
  }
}

import { Worker, isMainThread, parentPort, workerData } from 'node:worker_threads';

// Typed messages between main thread and worker
type WorkerMessage =
  | { type: 'PROCESS'; data: number[] }
  | { type: 'CANCEL' };

type WorkerResult =
  | { type: 'DONE'; result: number }
  | { type: 'ERROR'; error: string };

if (isMainThread) {
  // Main thread: create worker
  const worker = new Worker(new URL(import.meta.url));

  worker.postMessage({ type: 'PROCESS', data: [1, 2, 3, 4, 5] } satisfies WorkerMessage);

  worker.on('message', (msg: WorkerResult) => {
    if (msg.type === 'DONE') {
      console.log(`Sum: ${msg.result}`);
    } else {
      console.error(`Worker error: ${msg.error}`);
    }
  });
} else {
  // Worker thread
  parentPort!.on('message', (msg: WorkerMessage) => {
    if (msg.type === 'PROCESS') {
      const sum = msg.data.reduce((a, b) => a + b, 0);
      parentPort!.postMessage({ type: 'DONE', result: sum } satisfies WorkerResult);
    }
  });
}

5 API Typing: tRPC & OpenAPI
API Typing: tRPC & OpenAPI Full-Stack Ecosystem
End-to-end type safety between frontend and backend eliminates an entire class of integration bugs — mismatched request/response shapes. With tRPC, you change a server procedure and the client breaks at compile time, not in production.

There are two main approaches to API type safety in the TypeScript ecosystem:

  • tRPC: Define server procedures with Zod schemas; TypeScript automatically infers client types. Zero codegen, zero config — the client "sees" the server's types through TypeScript's type inference
  • OpenAPI codegen: Generate TypeScript types from an OpenAPI/Swagger specification. Best for multi-language backends where tRPC isn't an option
  • Zod as shared contract: Define schemas once, validate at runtime, infer types at compile time with z.infer<typeof schema>
  • The key insight: types disappear at runtime, but Zod schemas persist — so Zod bridges compile-time and runtime type safety
trpc-router.ts (server)
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();

// Zod schemas define input/output validation
const UserSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(['admin', 'user', 'viewer']),
});

// Infer TypeScript type from Zod schema
type User = z.infer<typeof UserSchema>;
// Equivalent to:
// { id: string; name: string; email: string; role: 'admin' | 'user' | 'viewer' }

const appRouter = t.router({
  // Query: read data
  getUser: t.procedure
    .input(z.object({ id: z.string().uuid() }))
    .query(async ({ input }): Promise<User> => {
      // input is { id: string } — validated at runtime by Zod
      return fetchUserFromDb(input.id);
    }),

  // Mutation: write data
  createUser: t.procedure
    .input(z.object({
      name: z.string().min(1).max(100),
      email: z.string().email(),
      role: z.enum(['admin', 'user', 'viewer']).default('user'),
    }))
    .mutation(async ({ input }) => {
      // input is { name: string; email: string; role: 'admin' | 'user' | 'viewer' }
      return createUserInDb(input);
    }),

  // List with pagination
  listUsers: t.procedure
    .input(z.object({
      page: z.number().int().min(1).default(1),
      limit: z.number().int().min(1).max(100).default(20),
      role: z.enum(['admin', 'user', 'viewer']).optional(),
    }))
    .query(async ({ input }) => {
      return { users: await queryUsers(input), total: 42 };
    }),
});

// Export the router type for the client
export type AppRouter = typeof appRouter;

import { z } from 'zod';

// Define once, use for both validation and types
const CreateOrderSchema = z.object({
  productId: z.string().uuid(),
  quantity: z.number().int().positive(),
  shippingAddress: z.object({
    street: z.string(),
    city: z.string(),
    zip: z.string().regex(/^\d{5}(-\d{4})?$/),
    country: z.string().length(2),
  }),
  couponCode: z.string().optional(),
});

// {
//   productId: string;
//   quantity: number;
//   shippingAddress: { street: string; city: string; zip: string; country: string };
//   couponCode?: string;
// }

// Runtime validation (server-side)
function validateOrder(raw: unknown): CreateOrderInput {
  return CreateOrderSchema.parse(raw);  // Throws ZodError if invalid
}

// Safe parsing (no throw)
function tryValidateOrder(raw: unknown): CreateOrderInput | null {
  const result = CreateOrderSchema.safeParse(raw);
  if (result.success) return result.data;   // CreateOrderInput
  console.error(result.error.issues);        // ZodIssue[]
  return null;
}

Common Mistakes
Not using Zod for runtime validation: types disappear at runtime, so a malformed API response passes silently
Always validate at the boundary (API input, external data) with Zod .parse() or .safeParse()
Duplicating types between client and server: interface User { ... } defined in both

Use tRPC for full-stack TypeScript; use OpenAPI codegen only for multi-language backends

  • Use tRPC for full-stack TypeScript projects — zero codegen, zero config, end-to-end types
  • Use OpenAPI codegen for multi-language backends (Go, Python, Java, etc.)
  • Always validate at the boundary with Zod — z.infer bridges compile-time and runtime
