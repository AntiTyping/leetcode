R1. Type-Level Programming Advanced Patterns & Performance rep 2 · 9 days since review
TypeScript's type system is Turing-complete. Recursive types, conditional types, and mapped types combine to create type-level algorithms. Use recursive conditional types for string parsing, tuple manipulation, and deep transforms.
Key Pattern 
// Recursive type: Deep Readonly
type DeepReadonly<T> = T extends object
  ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

// Type-level string split
type Split<S extends string, D extends string> =
  S extends `${infer H}${D}${infer T}` ? [H, ...Split<T, D>] : [S];

type Parts = Split<"a.b.c", ".">; // ["a", "b", "c"]
type Parts = ["a", "b", "c"] DeepReadonly recurses through nested object properties
Challenge Exercise

Create a type FlattenObject<T> that flattens nested object types into dot-notation keys: { a: { b: number, c: { d: string } } } → { "a.b": number, "a.c.d": string }.

Pitfall: TypeScript has a recursion depth limit (~50 levels) — deeply recursive types can cause “Type instantiation is excessively deep and possibly infinite” errors.
R2. Function Overloads Functions & Patterns rep 3 · 13 days since review
Function overloads provide multiple call signatures for a single function. Declare overload signatures first (what callers see), then the implementation signature (wider, handles all cases). The implementation signature is NOT directly callable — it only guides the implementation.
Key Pattern 
function createElement(tag: "a"): HTMLAnchorElement;
function createElement(tag: "canvas"): HTMLCanvasElement;
function createElement(tag: string): HTMLElement;
function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const link = createElement("a"); // HTMLAnchorElement
const link: HTMLAnchorElement Callers only see the three overload signatures, not the implementation
Challenge Exercise

Write overloaded signatures for a fetch wrapper where fetch(url, { json: true }) returns Promise<unknown> and fetch(url) returns Promise<Response>.

Pitfall: The implementation signature must be compatible with ALL overload signatures — a common mistake is making it too narrow.
R3. Class Patterns Runtime Patterns rep 2 · 8 days since review
TypeScript classes support abstract classes (can't instantiate, define contracts), mixins (compose behaviors via functions), private fields (#field for runtime privacy vs private for compile-time only), and static blocks for initialization.
Key Pattern 
abstract class Shape {
  abstract area(): number;
  describe() { return `Area: ${this.area()}`; }
}

// Mixin pattern
type Constructor<T = {}> = new (...args: any[]) => T;
function Timestamped<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    createdAt = new Date();
  };
}

class Circle extends Timestamped(Shape) {
  constructor(private radius: number) { super(); }
  area() { return Math.PI * this.radius ** 2; }
}
Circle has: area(), describe(), createdAt Mixin composes Timestamped behavior onto Shape
Challenge Exercise

Create a mixin Serializable that adds toJSON() and static fromJSON(data) methods to any class. Use a generic constraint so fromJSON returns the correct type.

Pitfall: TypeScript's private keyword doesn't enforce privacy at runtime — use #field for true encapsulation.
R4. Proxy & Reflect Typing Runtime Patterns rep 2 · 8 days since review
Proxy intercepts object operations (get, set, has, deleteProperty, etc.). TypeScript types Proxy handlers via ProxyHandler<T>. Reflect provides the default behavior for each trap. Common patterns: validation proxies, logging proxies, reactive state.
Key Pattern 
function createValidator<T extends object>(target: T, validators: {
  [K in keyof T]?: (value: T[K]) => boolean
}): T {
  return new Proxy(target, {
    set(obj, prop, value) {
      const key = prop as keyof T;
      if (validators[key] && !validators[key]!(value)) {
        throw new Error(`Invalid value for ${String(key)}`);
      }
      return Reflect.set(obj, prop, value);
    }
  });
}
Returns Proxy<T> typed as T — transparent to callers set trap validates before delegating to Reflect.set
Challenge Exercise

Create a createReadonlyDeep<T>(obj: T) function using Proxy that intercepts both set and deleteProperty and also wraps returned objects in their own readonly proxy.

Pitfall: Proxy traps must return the correct type — set must return boolean, get can return anything. Forgetting return true from set causes silent failures in strict mode.
R5. Covariance & Contravariance Advanced Patterns & Performance rep 2 · 7 days since review
Covariance (out): subtypes can be used where supertypes are expected (return types, readonly properties). Contravariance (in): supertypes can be used where subtypes are expected (function parameters). TypeScript 4.7+ supports explicit in/out variance annotations on generics.
Key Pattern 
interface Producer<out T> { produce(): T; }
interface Consumer<in T> { consume(item: T): void; }

// Producer<Dog> assignable to Producer<Animal> (covariant)
// Consumer<Animal> assignable to Consumer<Dog> (contravariant)

type Fn<in A, out R> = (arg: A) => R;
// Fn<Animal, Dog> assignable to Fn<Dog, Animal>
// Parameters are contravariant, return types are covariant
out = covariant (produces/returns T) in = contravariant (consumes/accepts T)
Challenge Exercise

Define a typed EventBus<Events> where emit is contravariant in the event type (accepts broader events) and subscribe callback's parameter is covariant (provides specific events).

Pitfall: Without in/out annotations, TypeScript infers variance — which can be wrong for complex generics with multiple uses of the type parameter.
R6. Template Literal Types — Advanced Advanced Patterns & Performance rep 2 · 7 days since review
Template literal types combine string literals at the type level. Combined with conditional types and infer, they can parse, validate, and transform string types. Built-in utilities: Uppercase, Lowercase, Capitalize, Uncapitalize.
Key Pattern 
type Route = `/${string}`;
type EventName = `on${Capitalize<string>}`;

// Parse route params
type ExtractParams<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? Param | ExtractParams<`/${Rest}`>
    : S extends `${string}:${infer Param}`
    ? Param
    : never;

type Params = ExtractParams<"/users/:id/posts/:postId">;
// "id" | "postId"
type Params = "id" | "postId" infer captures path segments between delimiters
Challenge Exercise

Create a type CamelToSnake<S> that converts "myVariableName" → "my_variable_name" at the type level, handling consecutive uppercase letters.

Pitfall: Template literal types with unions can cause combinatorial explosion — A | B combined with C | D produces all combinations. Watch for “Expression produces a union type that is too complex to represent.”

1. Type-Safe Middleware Chains Patterns
Middleware is everywhere — Express/Koa servers, Redux, data pipelines. Making middleware chains type-safe ensures context flows correctly and each middleware's additions to the context are visible to downstream handlers.

A middleware function receives a context object and a next callback, does work, then either calls next() to continue the chain or short-circuits. The interesting part for TypeScript is context augmentation: an auth middleware adds user to the context, and every middleware that runs after it should see user in the type.

The core technique is to make each middleware generic over its input context In and declare an output context Out extends In. A compose function then accumulates the output types through the chain so the final handler receives the union of every middleware's additions.

  • 2026-07-04 (Day 31)
  • Basic middleware type: (ctx: Ctx, next: () => Promise<void>) => Promise<void>
  • Context augmentation: each middleware widens the context type via intersection
  • Composition: accumulate In & Added1 & Added2 & ... through the chain
  • Express-style mutates a shared req object; Koa-style threads a single ctx with await next()
TypeScript — Example 1: Basic middleware type
// A middleware operates over a context and can call next()
type Middleware<Ctx> = (
  ctx: Ctx,
  next: () => Promise<void>
) => Promise<void>;

interface RequestContext {
  url: string;
  method: "GET" | "POST";
}

const logger: Middleware<RequestContext> = async (ctx, next) => {
  console.log(`-> ${ctx.method} ${ctx.url}`);
  await next();
  console.log(`<- done ${ctx.url}`);
};

await logger({ url: "/api", method: "GET" }, async () => {});
-> GET /api <- done /api
TypeScript — Example 2: Middleware that augments context
interface User { id: string; role: "admin" | "user"; }

// Input context has no user; output context adds it.
type AuthMiddleware<In> = (
  ctx: In,
  next: (ctx: In & { user: User }) => Promise<void>
) => Promise<void>;

const auth: AuthMiddleware<{ token: string }> = async (ctx, next) => {
  if (ctx.token !== "secret") throw new Error("401");
  const user: User = { id: "u1", role: "admin" };
  await next({ ...ctx, user }); // downstream now sees user
};

await auth({ token: "secret" }, async (ctx) => {
  console.log(ctx.user.role); // ctx.user is typed!
});
Inside next callback: ctx is { token: string } & { user: User } ctx.user.role is "admin" | "user" — fully typed
TypeScript — Example 3: Compose with type accumulation
// Each middleware is generic over In and declares what it Adds.
type Mw<In, Add> = (
  ctx: In,
  next: (ctx: In & Add) => Promise<void>
) => Promise<void>;

// Compose two middleware: the result's downstream sees both additions.
function compose<In, A1, A2>(
  m1: Mw<In, A1>,
  m2: Mw<In & A1, A2>
): Mw<In, A1 & A2> {
  return (ctx, next) =>
    m1(ctx, (c1) => m2(c1, (c2) => next(c2)));
}

const withUser: Mw<{ token: string }, { user: string }> =
  async (ctx, next) => next({ ...ctx, user: "alice" });

const withLog: Mw<{ token: string; user: string }, { ts: number }> =
  async (ctx, next) => next({ ...ctx, ts: Date.now() });

const chain = compose(withUser, withLog);
await chain({ token: "t" }, async (ctx) => {
  console.log(ctx.user, ctx.ts); // both visible
});
chain: Mw<{ token: string }, { user: string } & { ts: number }> Final ctx: { token: string; user: string; ts: number }
TypeScript — Example 4: Express-style typed middleware (real-world)
import type { Request, Response, NextFunction } from "express";

interface User { id: string; email: string; }

// Augment Express's Request so req.user is known after auth runs.
interface AuthedRequest extends Request {
  user: User;
}

// A guard middleware narrows Request -> AuthedRequest for handlers.
function requireAuth(
  req: Request,
  res: Response,
  next: NextFunction
): void {
  const token = req.headers.authorization;
  if (!token) { res.status(401).end(); return; }
  (req as AuthedRequest).user = { id: "u1", email: "a@b.com" };
  next();
}

// Handlers downstream declare AuthedRequest to get req.user typed.
function profile(req: AuthedRequest, res: Response): void {
  res.json({ email: req.user.email }); // req.user guaranteed
}

// app.get("/me", requireAuth, profile as any);
profile receives AuthedRequest — req.user.email is string, no optional chaining requireAuth establishes the invariant the handler relies on
TypeScript — Bonus: branded context proves a check ran
// A plain { user: User } says nothing about *how* user got there.
// A branded context proves the auth middleware was applied.
declare const authed: unique symbol;
type Authenticated<Ctx> = Ctx & { readonly [authed]: true };

function authenticate<Ctx extends { token: string }>(
  ctx: Ctx
): Authenticated<Ctx & { user: { id: string } }> {
  if (ctx.token !== "secret") throw new Error("401");
  return { ...ctx, user: { id: "u1" } } as Authenticated<
    Ctx & { user: { id: string } }
  >;
}

// Downstream handlers can REQUIRE proof of authentication in the type.
function adminOnly(ctx: Authenticated<{ user: { id: string } }>) {
  return ctx.user.id;
}

const ctx = authenticate({ token: "secret" });
adminOnly(ctx);                 // OK — ctx carries the brand
// adminOnly({ user: { id: "x" } }); // ERROR: missing brand
adminOnly({ user: { id: "x" } }) Property '[authed]' is missing — an un-authenticated context can't reach adminOnly
Common Mistakes
Using any for the context (ctx: any) — this defeats the entire purpose; downstream handlers lose all type information about what was added.
Make each middleware generic over its input context and use an intersection (In & Add) for what it contributes.
Hardcoding a single concrete context type in compose, so it can't accumulate additions across the chain.
Parameterize compose over each middleware's Add type and thread the accumulated intersection forward.
Mutating req with a cast but never declaring the augmented type, so handlers still see the un-augmented Request.
Declare an augmented interface (AuthedRequest extends Request) and type the downstream handler with it.
Key Rules
  • Each middleware should be generic over its input context and declare what it adds.
  • Use intersection types (In & Add) for context accumulation.
  • Consider branded types for verified context (e.g., AuthenticatedContext) so the type itself proves a check ran.
  • Koa-style (single threaded ctx with await next()) composes more cleanly in types than Express-style mutation.
Exercises
1Define a Middleware<In, Out> type where In is the input context and Out extends In is the output context with additions. Write a compose function that chains two middleware so the second one's input is the first one's output.
2Create an auth middleware type that transforms { req: Request } into { req: Request, user: User }, and a logger middleware that adds { log: Logger }. Compose them so the final handler has all three (req, user, log) fully typed.
Connections: Builds on intersection types and generic inference. Relates to discriminated_unions_practice (modeling request state) and branded types (verified context). The same accumulation pattern powers typed Redux middleware and tRPC procedure chains.
2. Recursive Type Utilities Advanced Type System
Many real-world type transformations require recursion: deep partial, deep readonly, path-based access, flatten nested structures. Understanding recursive types unlocks powerful type-level programming for library and framework authors.

A recursive type refers to itself in its own definition, typically guarded by a conditional that stops recursion at primitives. The two essentials are a base case (T extends object ? ... : T) and a recursive case that re-applies the transform to each property's type.

The most useful family: DeepPartial, DeepRequired, DeepReadonly, and path types — generating every valid dot-notation path through an object and looking up the type at a given path. TypeScript 4.5+ added tail-call optimization for recursive conditional types, raising the practical recursion ceiling.

  • DeepPartial<T> — makes all nested properties optional
  • Path<T> — union of all valid dot-notation paths
  • PathValue<T, P> — the type located at a given path
  • Recursion limits (~50 levels) and the TS 4.5 tail-call workaround
TypeScript — Example 1: DeepPartial
type DeepPartial<T> = T extends object
  ? { [K in keyof T]?: DeepPartial<T[K]> }
  : T;

interface Config {
  server: { host: string; port: number };
  features: { auth: { enabled: boolean } };
}

// Every nested field becomes optional — great for overrides.
const override: DeepPartial<Config> = {
  server: { port: 8080 }, // host omitted, OK
};
DeepPartial<Config> = { server?: { host?: string; port?: number }; features?: { auth?: { enabled?: boolean } }; }
TypeScript — Example 2: Path<T> (all dot-notation paths)
type Path<T> = T extends object
  ? {
      [K in keyof T & string]: T[K] extends object
        ? K | `${K}.${Path<T[K]>}`
        : K;
    }[keyof T & string]
  : never;

interface Settings {
  ui: { theme: string; layout: { columns: number } };
  debug: boolean;
}

type Keys = Path<Settings>;
// "ui" | "ui.theme" | "ui.layout" | "ui.layout.columns" | "debug"
type Keys = | "ui" | "ui.theme" | "ui.layout" | "ui.layout.columns" | "debug"
TypeScript — Example 3: PathValue<T, P>
type PathValue<T, P extends string> =
  P extends `${infer Head}.${infer Rest}`
    ? Head extends keyof T
      ? PathValue<T[Head], Rest>
      : never
    : P extends keyof T
    ? T[P]
    : never;

interface Settings {
  ui: { theme: string; layout: { columns: number } };
}

type Theme = PathValue<Settings, "ui.theme">;        // string
type Cols  = PathValue<Settings, "ui.layout.columns">; // number
type Theme = string type Cols = number Invalid paths resolve to never
TypeScript — WRONG vs RIGHT: missing base case
// WRONG: no base case — recurses into string's index signature,
// number's methods, etc. Causes "excessively deep" errors.
type BadReadonly<T> = {
  readonly [K in keyof T]: BadReadonly<T[K]>;
};

// RIGHT: guard with `extends object` and special-case arrays first.
type GoodReadonly<T> =
  T extends (infer U)[]
    ? ReadonlyArray<GoodReadonly<U>>
    : T extends object
    ? { readonly [K in keyof T]: GoodReadonly<T[K]> }
    : T; // primitives stop here

type R = GoodReadonly<{ tags: string[]; meta: { x: number } }>;
type R = { readonly tags: readonly string[]; readonly meta: { readonly x: number }; }
TypeScript — Example 4: Type-safe get() (real-world lodash.get)
type Path<T> = T extends object
  ? { [K in keyof T & string]: T[K] extends object
      ? K | `${K}.${Path<T[K]>}` : K }[keyof T & string]
  : never;

type PathValue<T, P extends string> =
  P extends `${infer H}.${infer R}`
    ? H extends keyof T ? PathValue<T[H], R> : never
    : P extends keyof T ? T[P] : never;

function get<T, P extends Path<T>>(obj: T, path: P): PathValue<T, P> {
  return path.split(".").reduce(
    (acc: any, key) => acc?.[key], obj
  );
}

const cfg = { ui: { theme: "dark", layout: { columns: 3 } } };
const theme = get(cfg, "ui.theme");            // string
const cols  = get(cfg, "ui.layout.columns");   // number
// get(cfg, "ui.nope"); // ERROR: not assignable to Path
const theme: string = "dark" const cols: number = 3 Autocomplete suggests only valid paths; typos are compile errors
Common Mistakes
Infinite recursion from a missing base case — type Bad<T> = { [K in keyof T]: Bad<T[K]> } never stops at primitives.
Always guard with T extends object ? ... : T so primitives terminate the recursion.
Not handling arrays/tuples — T extends object is true for arrays, so a naive DeepReadonly may map over length, push, etc.
Special-case arrays first: T extends Array<infer U> ? ReadonlyArray<DeepReadonly<U>> : ...
Forgetting to distribute over unions, so Path<A | B> behaves unexpectedly.
Be deliberate about distribution — wrap in a tuple [T] extends [object] to prevent it, or rely on it intentionally.
Key Rules
  • Always have a base case — primitives must stop the recursion.
  • Use the T extends object guard to halt at non-object types.
  • Handle arrays, tuples, Map, and Set explicitly — they are objects but need different treatment.
  • Test recursive types with tsd / expect-type to verify boundary behavior (empty objects, optional fields, unions).
Exercises
1Implement DeepReadonly<T> that handles objects, arrays, Map, and Set — making their contents readonly too (ReadonlyArray, ReadonlyMap, ReadonlySet).
2Create Paths<T> that generates a union of all valid dot-notation paths for an object type, including nested paths that descend through arrays (e.g. "items.0.name" or a numeric index segment).
Connections: Direct application of review_type_level_programming and template literal types (R6). PathValue reuses the infer string-splitting pattern. Powers form libraries (react-hook-form field paths), i18n key typing, and ORM column references.
3. Type-Safe SQL Query Builders Database Patterns
Raw SQL strings lose type safety. Query builders like Drizzle, Kysely, and knex use TypeScript's type system to validate queries at compile time — ensuring column names exist, join conditions match, and return types are correct.

The whole approach starts with describing your schema once as TypeScript types, then deriving everything else from it. A select builder narrows the return type to only the columns you pick. A where clause restricts you to valid column names with matching value types. A join combines two table schemas into one result type.

The subtle part is nullability: an INNER JOIN produces the intersection of both row types, but a LEFT JOIN can leave the right side unmatched, so every right-hand column becomes T | null (or the whole right row becomes Partial/nullable).

  • Table schema as a type; derive column unions with keyof
  • Select narrows the return via Pick<Row, Cols>
  • where uses a mapped type so values match their column type
  • Joins: INNER → L & R, LEFT → L & Partial<R>
TypeScript — Example 1: Schema type + select narrowing
interface UsersTable {
  id: number;
  name: string;
  email: string;
  age: number;
}

// Select narrows the row to only the chosen columns.
type Select<Row, Cols extends keyof Row> = Pick<Row, Cols>;

declare function select<Row, const Cols extends readonly (keyof Row)[]>(
  cols: Cols
): Select<Row, Cols[number]>[];

const rows = select<UsersTable, ["id", "name"]>(["id", "name"]);
// rows: { id: number; name: string }[]
rows: { id: number; name: string }[] email and age are excluded from the result type
TypeScript — Example 2: Type-safe where clause
interface UsersTable { id: number; name: string; age: number; }

// Each key is optional; its value must match the column's type.
type Where<Row> = {
  [K in keyof Row]?: Row[K] | { gt?: Row[K]; lt?: Row[K] };
};

declare function where<Row>(cond: Where<Row>): Row[];

where<UsersTable>({ age: { gt: 18 }, name: "Alice" }); // OK
// where<UsersTable>({ age: "old" });   // ERROR: string not number
// where<UsersTable>({ nope: 1 });      // ERROR: unknown column
where<UsersTable>({ age: "old" }) Type 'string' is not assignable to type 'number | { gt?: number; ... }' where<UsersTable>({ nope: 1 }) Object literal may only specify known properties
TypeScript — Example 3: Joins — INNER vs LEFT nullability
interface Users { id: number; name: string; }
interface Posts { id: number; userId: number; title: string; }

// INNER JOIN: both rows always present -> intersection.
// LEFT JOIN: right side may be missing -> nullable.
type JoinResult<L, R, J extends "inner" | "left"> =
  J extends "inner"
    ? L & R
    : L & { [K in keyof R]: R[K] | null };

type Inner = JoinResult<Users, Posts, "inner">;
type Left  = JoinResult<Users, Posts, "left">;
type Inner = { id: number; name: string; userId: number; title: string } type Left = { id: number; name: string; id: number | null; ...; title: string | null }
TypeScript — Example 4: Mini Kysely-like chained builder (real-world)
interface DB {
  users: { id: number; name: string; age: number };
  posts: { id: number; userId: number; title: string };
}

class Query<Table extends keyof DB, Selected = DB[Table]> {
  constructor(private table: Table) {}

  select<const C extends readonly (keyof DB[Table])[]>(
    cols: C
  ): Query<Table, Pick<DB[Table], C[number]>> {
    return this as any;
  }

  where(cond: Partial<DB[Table]>): this { return this; }

  execute(): Selected[] { return [] as any; }
}

const result = new Query("users")
  .select(["id", "name"])
  .where({ age: 30 })  // age valid even though not selected
  .execute();
// result: { id: number; name: string }[]
result: { id: number; name: string }[] .where accepts any real column; .select narrows the returned shape
TypeScript — WRONG vs RIGHT: column typing
interface Users { id: number; name: string; age: number; }

// WRONG: string columns — typos compile, results are untyped.
declare function badSelect(cols: string[]): any[];
const bad = badSelect(["nmae"]); // typo slips through, bad: any[]

// RIGHT: keyof + const param — typos are errors, result is narrowed.
declare function goodSelect<
  const C extends readonly (keyof Users)[]
>(cols: C): Pick<Users, C[number]>[];

const good = goodSelect(["id", "name"]); // { id: number; name: string }[]
// goodSelect(["nmae"]); // ERROR: "nmae" is not a column of Users
goodSelect(["nmae"]) Type '"nmae"' is not assignable to type 'keyof Users'.
Common Mistakes
Using plain string for column names: select(cols: string[]) — you lose all column checking and autocomplete.
Constrain to keyof Row (and use const type parameters) so only real columns are accepted and the literal tuple is preserved.
Returning any (or Record<string, unknown>) from query execution, so callers get no result typing.
Thread the narrowed row type through to execute()'s return type with a Selected type parameter.
Typing LEFT JOIN columns as non-nullable, then crashing at runtime on unmatched rows.
Make every right-hand column of a LEFT JOIN T | null (or wrap the right row in Partial/nullable).
Key Rules
  • Define your schema once, derive everything else (columns, where, results) from it.
  • Use const type parameters / template literal types for column references to preserve literals.
  • LEFT JOIN columns should be T | null; INNER JOIN produces a plain intersection.
  • Narrow the return type through every builder step so execute() reflects the exact selected shape.
Exercises
1Define table types for users and posts, then create a Select<T, Cols> type that narrows the return type to only the selected columns. Verify that selecting a non-existent column is a compile error.
2Implement a JoinResult<Left, Right, JoinType> type where INNER JOIN produces Left & Right but LEFT JOIN produces Left & Partial<Right> (or with each right column made nullable).
Connections: Combines mapped types (where), Pick (select), and conditional types (join nullability). Builds on review_database_typing (Prisma/Drizzle). The chained-builder return-type threading mirrors the phantom-type builder pattern from Day 15.

4. Module Boundary Types & Opaque Exports Architecture
In large codebases, exposing internal types creates coupling. Opaque types (branded exports) let modules expose a type that callers can use but can't construct or inspect — enforcing invariants at module boundaries.

An opaque type is a type whose underlying structure is hidden from consumers. The classic technique is a branded intersection: the public type is string & { readonly __brand: unique symbol }, where the brand lives in a value the module never exports. Consumers can hold and pass the value, but cannot create one from a plain string — only the module's factory function can.

This pairs with disciplined export hygiene: export type for type-only re-exports (so they're erased and can't be instantiated), keeping the public surface minimal, and module augmentation for controlled extension points.

  • Public API types vs internal implementation types
  • Opaque types via branded intersection + a private brand
  • export type vs export for re-exports
  • Module augmentation as a sanctioned extension mechanism
TypeScript — Example 1: Branded UserId
// The brand symbol is never exported -> callers can't forge a UserId.
declare const brand: unique symbol;
export type UserId = string & { readonly [brand]: "UserId" };

export function toUserId(raw: string): UserId {
  // validation here...
  return raw as UserId;
}

function loadUser(id: UserId) { /* ... */ }

const id = toUserId("u_123");
loadUser(id);          // OK
// loadUser("u_123");  // ERROR: string not assignable to UserId
loadUser("u_123") Argument of type 'string' is not assignable to parameter of type 'UserId'.
TypeScript — Example 2: Opaque config with factory
declare const configBrand: unique symbol;
export type ValidatedConfig = {
  readonly port: number;
  readonly host: string;
} & { readonly [configBrand]: true };

export function defineConfig(input: {
  port: number; host: string;
}): ValidatedConfig {
  if (input.port < 1 || input.port > 65535) {
    throw new Error("invalid port");
  }
  return input as ValidatedConfig;
}

// Consumers can read it but can't fabricate one bypassing validation.
function startServer(cfg: ValidatedConfig) {
  console.log(`listening on ${cfg.host}:${cfg.port}`);
}
ValidatedConfig is usable (read .port/.host) but only defineConfig can mint one A raw { port, host } object is NOT assignable to ValidatedConfig
TypeScript — Example 3: Module exporting opaque type + constructor
// safe-html.ts
declare const htmlBrand: unique symbol;
export type SafeHtml = string & { readonly [htmlBrand]: true };

export function sanitize(raw: string): SafeHtml {
  const clean = raw.replace(/<script[\s\S]*?<\/script>/gi, "");
  return clean as SafeHtml;
}

export function render(html: SafeHtml): void {
  document.body.innerHTML = html; // only sanitized strings reach here
}

// consumer.ts
import { sanitize, render } from "./safe-html";
render(sanitize("<b>hi</b><script>evil()</script>")); // OK
// render("<b>hi</b>"); // ERROR: plain string is not SafeHtml
render only accepts SafeHtml — the type system proves every rendered string passed through sanitize(). A raw string is a compile error.
TypeScript — Example 4: export type vs export (re-exports)
// types.ts
export interface InternalRow { id: number; secret: string; }

// public.ts — re-export the TYPE only, never the value-level shape.
export type { InternalRow as Row } from "./types";

// With isolatedModules / verbatimModuleSyntax, mixing these is caught:
// export { InternalRow } from "./types";
//   -> ERROR if InternalRow is a type and you used a value export.

// Use plain `export` only for things that exist at runtime (functions,
// classes, consts). Use `export type` for type-only surface.
export type { ... } is fully erased at build time — zero runtime footprint verbatimModuleSyntax forces you to mark type-only exports explicitly
TypeScript — WRONG vs RIGHT: forgeable vs opaque IDs
// WRONG: a bare alias is structurally identical to string.
type WeakUserId = string;
function loadUserWeak(id: WeakUserId) {}
loadUserWeak("literally anything"); // compiles — no protection

// RIGHT: brand makes the type un-forgeable outside the module.
declare const brand: unique symbol;
type UserId = string & { readonly [brand]: "UserId" };
function loadUser(id: UserId) {}
// loadUser("anything"); // ERROR
// Two different brands also can't be swapped:
type OrderId = string & { readonly [brand]: "OrderId" };
declare const o: OrderId;
// loadUser(o); // ERROR: OrderId is not UserId
loadUser("anything") -> string not assignable to UserId loadUser(o) -> OrderId not assignable to UserId (different brand)
Common Mistakes
Exporting internal types that should be opaque, letting consumers depend on (and break against) implementation details.
Export the minimum surface; brand IDs and validated data so consumers can't reconstruct them.
Using export for a type-only symbol when verbatimModuleSyntax requires export type (or vice versa).
Use export type for types that shouldn't be instantiated outside the module; reserve export for runtime values.
Exporting the brand symbol or factory's internals, which lets callers forge opaque values.
Keep the brand declare const ... : unique symbol private to the module — only the factory can produce the branded type.
Key Rules
  • Export the minimum type surface — every exported type is a contract you must maintain.
  • Use branded types for IDs and validated data to enforce invariants at the boundary.
  • Prefer export type for types that shouldn't be instantiated outside the module.
  • Turn on isolatedModules / verbatimModuleSyntax to catch ambiguous type-vs-value exports.
Exercises
1Create a module that exports an opaque SafeHtml type. Only the module's sanitize(raw: string): SafeHtml function can create it. Consumer code should be able to use SafeHtml (pass it around, render it) but not create it from a plain string.
2Design a “token” module with opaque AccessToken and RefreshToken types that can't be accidentally swapped — passing a RefreshToken where an AccessToken is expected should be a compile error.
Connections: Generalizes branded types into a module-boundary discipline. Pairs with the phantom-type builder (Day 15) and SafeHtml mirrors the validation-proxy idea from R4. Underpins how libraries like zod brand parsed output.
5. ESM/CJS Dual Package Patterns Build & Ship
The Node.js ecosystem is migrating from CommonJS to ESM, but many consumers still need CJS. Publishing a package that works in both systems requires understanding the exports field, conditional exports, and TypeScript's moduleResolution settings.

A dual package ships both an ESM build and a CJS build and routes consumers to the right one via the package.json "exports" field. The "import" condition serves ESM (.mjs), "require" serves CJS (.cjs), and — critically — a "types" condition tells TypeScript where the declarations are for each module system.

Modern TypeScript resolution ("node16"/"nodenext" vs "bundler") follows these conditions. The biggest footgun is shipping a single .d.ts when the runtime has two module formats — you generally want both .d.mts and .d.cts. Tools like tsup or unbuild generate all four outputs (.mjs, .cjs, .d.mts, .d.cts) for you.

  • "exports" with conditional "import" / "require" / "types"
  • Separate .d.cts and .d.mts declaration files
  • tsconfig: module, "moduleResolution": "bundler" vs "node16"
  • attw (“Are The Types Wrong?”) to verify resolution
JSON — Example 1: package.json conditional exports
{
  "name": "my-lib",
  "version": "1.0.0",
  "type": "module",
  "exports": {
    ".": {
      "import": {
        "types": "./dist/index.d.mts",
        "default": "./dist/index.mjs"
      },
      "require": {
        "types": "./dist/index.d.cts",
        "default": "./dist/index.cjs"
      }
    }
  }
}
import "my-lib" -> ./dist/index.mjs (types: index.d.mts) require("my-lib") -> ./dist/index.cjs (types: index.d.cts)
TypeScript — Example 2: separate .d.cts / .d.mts declarations
// dist/index.d.mts  (ESM declarations)
export declare function greet(name: string): string;
export default greet;

// dist/index.d.cts  (CJS declarations)
export declare function greet(name: string): string;
export = greet; // CJS-style export form

// Why two files? Under moduleResolution "node16"/"nodenext",
// TypeScript treats a .d.ts differently per the resolving
// import vs require condition. `export =` is only valid for CJS.
ESM consumers see `export default` / named exports CJS consumers see `export =` — matching `const greet = require("my-lib")`
TypeScript — Example 3: tsup config for dual output
// tsup.config.ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: ["src/index.ts"],
  format: ["esm", "cjs"], // emits index.mjs + index.cjs
  dts: true,              // emits index.d.mts + index.d.cts
  clean: true,
  sourcemap: true,
  outDir: "dist",
});

// Run: tsup
// dist/index.mjs, dist/index.cjs, dist/index.d.mts, dist/index.d.cts
CLI dist/index.mjs 1.2 KB CLI dist/index.cjs 1.4 KB DTS dist/index.d.mts 0.3 KB DTS dist/index.d.cts 0.3 KB
Text — Example 4: complete dual package structure (real-world)
my-lib/
├─ src/
│  └─ index.ts          # single TS source of truth
├─ dist/                # generated by tsup
│  ├─ index.mjs         # ESM runtime
│  ├─ index.cjs         # CJS runtime
│  ├─ index.d.mts       # ESM types
│  └─ index.d.cts       # CJS types
├─ package.json         # "exports" maps conditions -> dist files
├─ tsup.config.ts
└─ tsconfig.json        # "module": "NodeNext",
                        # "moduleResolution": "NodeNext"

# Verify before publishing:
#   npx @arethetypeswrong/cli --pack
#   node -e "require('my-lib')"        # CJS works
#   node --input-type=module -e "import 'my-lib'"  # ESM works
attw: No problems found 🎉 require('my-lib') -> OK import 'my-lib' -> OK
JSON — WRONG vs RIGHT: exports field
// WRONG: single .d.ts, no per-condition types, legacy "main" only.
{
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts"
  // require() works; modern ESM + "node16" resolution may break,
  // and attw flags "masquerading" / wrong-format type errors.
}

// RIGHT: explicit conditions, types co-located per format.
{
  "type": "module",
  "main": "./dist/index.cjs",   // fallback for old tooling
  "exports": {
    ".": {
      "import": { "types": "./dist/index.d.mts", "default": "./dist/index.mjs" },
      "require": { "types": "./dist/index.d.cts", "default": "./dist/index.cjs" }
    }
  }
}
attw on RIGHT: No problems found attw on WRONG: often "Masquerading as ESM" / missing types under require
Common Mistakes
Forgetting the "types" condition inside "exports", so TypeScript can't find your declarations under modern resolution.
Always include a "types" condition (per import/require) pointing at the matching .d.mts/.d.cts.
Relying only on "main" (legacy resolution) and assuming new bundlers/Node will pick the right format.
Use the "exports" field as the source of truth; "main" is at most a fallback for very old tooling.
Testing only with import and assuming require works — the two resolve differently and either can break.
Test both entry points and run attw (“Are The Types Wrong?”) before publishing.
Key Rules
  • Always include the "types" condition in conditional exports — once per format.
  • Test your package with both import and require.
  • Use attw (Are The Types Wrong?) to verify TypeScript compatibility across resolvers.
  • Behavior differs between "moduleResolution": "node16" and "bundler" — pick based on your target consumers.
Exercises
1Write a package.json "exports" field that provides ESM for import, CJS for require, and the correct .d.mts / .d.cts declarations for TypeScript in both cases.
2Given a library with a src/index.ts entry point, write the tsup config (or an unbuild config) to produce both dist/index.mjs and dist/index.cjs with corresponding declaration files, then list the commands you'd run to verify it.
Connections: The build-and-ship counterpart to module_boundary_types — you decide the public surface, then package it so both module systems resolve it correctly. Ties to tsconfig moduleResolution knowledge and the isolatedModules discipline from Concept 4.
