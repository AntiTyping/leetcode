R1. Class Patterns Runtime rep 0 · last reviewed 2026-06-05 · overdue 4 days
Abstract classes define contracts that subclasses must implement. Mixins compose behaviors onto a base class using a function that returns a new class. Private fields with # enforce runtime privacy (not just type-level). Static blocks run once when the class is loaded.
Key Pattern 
// Mixin pattern — composable class behaviors
type Constructor<T = {}> = new (...args: any[]) => T;

function Timestamped<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    createdAt = new Date();
    updatedAt = new Date();

    touch() {
      this.updatedAt = new Date();
    }
  };
}

function Serializable<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    serialize(): string {
      return JSON.stringify(this);
    }
  };
}

// Compose mixins
class BaseEntity {
  id: string;
  constructor(id: string) {
    this.id = id;
  }
}

const TimestampedSerializable = Serializable(Timestamped(BaseEntity));

const entity = new TimestampedSerializable('abc-123');
entity.touch();
console.log(entity.serialize());
console.log(entity.createdAt instanceof Date);
{"id":"abc-123","createdAt":"2026-06-10T...","updatedAt":"2026-06-10T..."} true
Challenge Exercise

Implement a mixin pattern combining Timestamped (adds createdAt, updatedAt, touch()) and Serializable (adds serialize()) mixins. Apply both to a BaseEntity class with an id property. Verify the composed class has all methods and properties.

#1 Pitfall: #field vs private field — the # prefix is runtime enforcement (true hard private, survives even Object.keys). The private keyword is TypeScript-only and erased at compile time — the field is still accessible via (obj as any).field at runtime.
R2. Proxy & Reflect Typing Runtime rep 0 · last reviewed 2026-06-05 · overdue 4 days
Proxy wraps an object and intercepts operations via handler traps. ProxyHandler<T> types the handler. Reflect provides default implementations for each trap. The proxy’s return type is the same as the target — TypeScript cannot narrow based on trap logic.
Key Pattern 
// Validation proxy with a schema object
interface UserData {
  name: string;
  age: number;
  email: string;
}

type Validator<T> = {
  [K in keyof T]?: (value: T[K]) => boolean;
};

function validated<T extends object>(
  target: T,
  validators: Validator<T>
): T {
  const handler: ProxyHandler<T> = {
    set(obj, prop, value) {
      const key = prop as keyof T;
      const validate = validators[key];
      if (validate && !validate(value)) {
        throw new Error(`Invalid value for ${String(prop)}: ${value}`);
      }
      return Reflect.set(obj, prop, value);
    },
  };
  return new Proxy(target, handler);
}

const user = validated<UserData>(
  { name: 'Alice', age: 30, email: 'alice@test.com' },
  {
    age: (v) => v >= 0 && v <= 150,
    email: (v) => v.includes('@'),
  }
);

user.name = 'Bob';       // OK
// user.age = -5;         // throws: Invalid value for age: -5
console.log(user.name);
Bob
Challenge Exercise

Type a validation proxy that accepts a target object and a Validator<T> schema mapping each key to a validation function. The set trap should call the validator before allowing assignment. Throw on invalid values.

#1 Pitfall: Proxy type does not narrow — new Proxy(target, handler) returns the same type as target. TypeScript has no way to express that the proxy might reject certain values or add computed properties. The type system sees the proxy as identical to the original object.
R3. Namespaces vs Modules Modules rep 1 · last reviewed 2026-06-05 · overdue 4 days
ES modules (import/export) are the standard. Namespaces (namespace Foo { }) are legacy but still useful for declaration merging and augmenting third-party types. Module augmentation uses declare module to extend existing module types.
Key Pattern 
// Module augmentation — extend Express Request with a user property
import { Request, Response, NextFunction } from 'express';

// Augment the express module's types
declare module 'express-serve-static-core' {
  interface Request {
    user?: {
      id: string;
      role: 'admin' | 'user';
    };
  }
}

// Now req.user is typed across all middleware
function authMiddleware(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.split(' ')[1];
  if (token) {
    req.user = { id: 'u123', role: 'admin' };  // Typed!
  }
  next();
}

function protectedRoute(req: Request, res: Response) {
  if (req.user?.role === 'admin') {
    res.json({ admin: true, userId: req.user.id });
  } else {
    res.status(403).json({ error: 'Forbidden' });
  }
}
Challenge Exercise

Augment the Express Request type with a user property of type { id: string; role: 'admin' | 'user' }. Write an auth middleware that assigns to req.user and a protected route that reads it — both fully typed.

#1 Pitfall: Forgetting export in declaration merging makes it global. If your augmentation file has no top-level import or export, TypeScript treats it as a script (global scope) rather than a module, and the augmentation applies globally rather than to the specific module.
R4. Generics Basics Generic & Utility Types rep 2 · last reviewed 2026-06-01 · overdue 3 days
Generics parameterize types: function identity<T>(x: T): T. Constraints (T extends X) limit what T can be. Generic interfaces like Map<K, V> preserve type relationships. TypeScript usually infers generic arguments at call sites.
Key Pattern 
// groupBy — generic function with a constraint
function groupBy<T, K extends string | number | symbol>(
  arr: T[],
  fn: (item: T) => K
): Map<K, T[]> {
  const map = new Map<K, T[]>();
  for (const item of arr) {
    const key = fn(item);
    const group = map.get(key) ?? [];
    group.push(item);
    map.set(key, group);
  }
  return map;
}

interface Employee {
  name: string;
  department: 'eng' | 'sales' | 'hr';
  salary: number;
}

const employees: Employee[] = [
  { name: 'Alice', department: 'eng', salary: 120000 },
  { name: 'Bob', department: 'sales', salary: 90000 },
  { name: 'Carol', department: 'eng', salary: 130000 },
  { name: 'Dave', department: 'hr', salary: 85000 },
];

// K is inferred as 'eng' | 'sales' | 'hr'
const byDept = groupBy(employees, e => e.department);
console.log(byDept.get('eng')?.map(e => e.name));
["Alice", "Carol"]
Challenge Exercise

Write groupBy<T, K>(arr: T[], fn: (t: T) => K): Map<K, T[]> with a constraint that K extends valid Map key types. Test it with an array of objects, grouping by a string property.

#1 Pitfall: Using T extends object when T extends Record<string, unknown> is safer. The constraint extends object allows arrays and functions, which is usually not intended. Record<string, unknown> ensures a plain object shape.
R5. Interfaces vs Type Aliases Structuring Types rep 2 · last reviewed 2026-06-01 · overdue 3 days
Interfaces support declaration merging — two interface Foo declarations merge. Type aliases do not merge. Interfaces can be extended and implemented by classes. Use interfaces for public APIs (extensible), type aliases for unions, tuples, and computed types.
Key Pattern 
// Declaration merging — only possible with interfaces
interface Window {
  analytics: {
    track(event: string, data?: Record<string, unknown>): void;
    identify(userId: string): void;
  };
}

// Second declaration merges with the first
interface Window {
  featureFlags: {
    isEnabled(flag: string): boolean;
  };
}

// Window now has both analytics AND featureFlags
declare const window: Window;
window.analytics.track('click', { button: 'signup' });
window.featureFlags.isEnabled('dark-mode');

// This is IMPOSSIBLE with type aliases:
// type Window = { analytics: ... };
// type Window = { featureFlags: ... };  // Error: Duplicate identifier
Duplicate identifier 'Window'. (if using type aliases)
Challenge Exercise

Demonstrate declaration merging that is impossible with type. Create two separate interface declarations for Window that add analytics and featureFlags properties respectively. Verify both are accessible on the merged type.

#1 Pitfall: Interface merging can introduce conflicts if two declarations have the same property with different types. interface Foo { x: string } and interface Foo { x: number } will produce a type error because x cannot be both string and number. Function overloads on the same property name DO merge (they stack).
R6. Literal Types & const Assertions Core Type System rep 2 · last reviewed 2026-06-01 · overdue 3 days
Literal types narrow values to exact constants: 'get' | 'post' instead of string. as const infers the narrowest possible type: arrays become readonly tuples, objects become readonly with literal values. Template literal types compose string literals at the type level.
Key Pattern 
// Typed routing table using template literal types
type HttpMethod = 'GET' | 'POST' | 'PUT' | 'DELETE';
type ApiVersion = 'v1' | 'v2';
type Resource = 'users' | 'posts' | 'comments';

// Template literal type — composes string types
type RoutePattern = `/${ApiVersion}/${Resource}`;
//   ^? "/v1/users" | "/v1/posts" | "/v1/comments" | "/v2/users" | "/v2/posts" | "/v2/comments"

type RouteKey = `${HttpMethod} ${RoutePattern}`;
//   ^? "GET /v1/users" | "POST /v1/users" | ... (24 total combinations)

// as const for deep readonly tuple inference
const ROUTES = [
  { method: 'GET',  path: '/v1/users' },
  { method: 'POST', path: '/v1/users' },
  { method: 'GET',  path: '/v2/posts' },
] as const;

// ROUTES is: readonly [
//   { readonly method: "GET";  readonly path: "/v1/users" },
//   { readonly method: "POST"; readonly path: "/v1/users" },
//   { readonly method: "GET";  readonly path: "/v2/posts" },
// ]

type RoutePaths = (typeof ROUTES)[number]['path'];
//   ^? "/v1/users" | "/v2/posts"

console.log(ROUTES[0].method);  // "GET" (literal type, not string)
RoutePattern = "/v1/users" | "/v1/posts" | "/v1/comments" | "/v2/users" | "/v2/posts" | "/v2/comments" RoutePaths = "/v1/users" | "/v2/posts"
Challenge Exercise

Create a typed routing table using template literal types. Define HttpMethod, ApiVersion, and Resource literal unions. Compose them into a RouteKey type. Use as const to define a concrete routes array and extract path types from it.

#1 Pitfall: as const on arrays gives a readonly tuple — incompatible with mutable array params. If a function expects string[], you cannot pass a readonly ["a", "b"]. Either widen the function parameter to readonly string[] or use [...arr] to create a mutable copy.
1 Next.js + TypeScript
Next.js + TypeScript Next.js App Router Ecosystem
Next.js is the most widely used React framework, and its TypeScript integration types your data-fetching, routing, and metadata at every layer. Typed getServerSideProps, API routes, and App Router Server Components catch data shape mismatches before they reach the browser.

Next.js provides TypeScript support at every level of the framework:

  • 2026-07-04 (Day 31)
  • Pages Router: GetServerSideProps, GetStaticProps, and NextApiHandler type the data-fetching and API layers
  • App Router: Server Components are async functions with typed props. The Metadata type structures SEO data. Dynamic route params are typed via params
  • API Routes: NextRequest and NextResponse in the App Router, or NextApiRequest/NextApiResponse<T> in the Pages Router
  • The InferGetServerSidePropsType utility extracts the props type from your data-fetching function
  • Dynamic route segments like [slug] and [...catchAll] have typed params objects
pages/posts/[id].tsx (Pages Router — getServerSideProps)
import type {
  GetServerSideProps,
  InferGetServerSidePropsType,
} from 'next';

interface Post {
  id: string;
  title: string;
  body: string;
  author: { name: string; avatar: string };
  publishedAt: string;
}

// Typed data fetching — returns { props: { post: Post } }
export const getServerSideProps = (async (context) => {
  const { id } = context.params as { id: string };
  const res = await fetch(`https://api.example.com/posts/${id}`);

  if (!res.ok) {
    return { notFound: true };
  }

  const post: Post = await res.json();
  return { props: { post } };
}) satisfies GetServerSideProps<{ post: Post }>;

// InferGetServerSidePropsType extracts props from getServerSideProps
export default function PostPage({
  post,
}: InferGetServerSidePropsType<typeof getServerSideProps>) {
  // post is typed as Post
  return (
    <article>
      <h1>{post.title}</h1>
      <p>By {post.author.name}</p>
      <div>{post.body}</div>
    </article>
  );
}

app/posts/[slug]/page.tsx (App Router — Server Component)
import type { Metadata } from 'next';

// Typed params for dynamic route segments
interface PostPageProps {
  params: Promise<{ slug: string }>;
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>;
}

// Typed metadata generation
export async function generateMetadata(
  { params }: PostPageProps
): Promise<Metadata> {
  const { slug } = await params;
  const post = await getPost(slug);

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      images: [post.coverImage],
    },
  };
}

// Server Component — async function with typed props
export default async function PostPage({ params, searchParams }: PostPageProps) {
  const { slug } = await params;
  const sp = await searchParams;
  const post = await getPost(slug);        // runs on the server
  const highlight = sp.highlight;           // string | string[] | undefined

  return (
    <article>
      <h1>{post.title}</h1>
      <p>Published: {post.publishedAt}</p>
      <div>{post.body}</div>
    </article>
  );
}

// Helper — typed fetch on the server
async function getPost(slug: string): Promise<{
  title: string;
  excerpt: string;
  body: string;
  coverImage: string;
  publishedAt: string;
}> {
  const res = await fetch(`https://api.example.com/posts/${slug}`, {
    next: { revalidate: 60 },  // ISR: revalidate every 60s
  });
  if (!res.ok) throw new Error('Post not found');
  return res.json();
}
app/api/posts/route.ts (App Router — Route Handler)
import { NextRequest, NextResponse } from 'next/server';

interface CreatePostBody {
  title: string;
  body: string;
  published?: boolean;
}

// App Router route handlers use NextRequest/NextResponse
export async function GET(request: NextRequest) {
  const searchParams = request.nextUrl.searchParams;
  const page = Number(searchParams.get('page') ?? '1');

  const posts = await fetchPosts({ page });
  return NextResponse.json({ data: posts, page });
}

export async function POST(request: NextRequest) {
  const body: CreatePostBody = await request.json();

  // Validate at runtime (types don't survive JSON parsing)
  if (!body.title || !body.body) {
    return NextResponse.json(
      { error: 'title and body are required' },
      { status: 400 }
    );
  }

  const post = await createPost(body);
  return NextResponse.json({ data: post }, { status: 201 });
}
Common Mistakes
Trusting the type of await request.json() without runtime validation — JSON parsing returns any
Always validate the parsed body with Zod or a type guard before using it as a typed object
Using getServerSideProps return type manually: export default function Page(props: { post: Post })
Use InferGetServerSidePropsType<typeof getServerSideProps> to keep props in sync automatically
Forgetting that App Router params is a Promise in Next.js 15+ — must await it
Always const { slug } = await params; in Server Components and route handlers
Key Rules
  • Use InferGetServerSidePropsType to derive page props from getServerSideProps
  • Type API route responses with NextApiResponse<T> (Pages) or NextResponse.json() (App Router)
  • App Router params and searchParams are Promise objects in Next.js 15+ — await them
  • The Metadata type structures all SEO fields: title, description, openGraph, etc.
  • Always validate request.json() at runtime — TypeScript types are erased after compilation

2 Form Handling: React Hook Form + Zod
Form Handling: React Hook Form + Zod React Hook Form Zod Ecosystem
Forms are the most common source of runtime type mismatches. React Hook Form with Zod gives you compile-time type safety for field names, error messages, and submitted values — and runtime validation that matches your TypeScript types exactly.

React Hook Form (RHF) + Zod is the dominant typed form pattern in the React ecosystem:

  • useForm<T>() types every form method: register, handleSubmit, watch, setValue, formState.errors
  • zodResolver(schema) connects Zod validation to RHF — the schema IS the type definition
  • z.infer<typeof schema> extracts the TypeScript type from a Zod schema — single source of truth
  • Field registration is typed: register('email') autocompletes field names and rejects invalid ones
  • Error objects are typed per-field: errors.email?.message is string | undefined
signup-schema.ts
import { z } from 'zod';

// Zod schema — single source of truth for types AND validation
export const signupSchema = z.object({
  email: z.string().email('Invalid email address'),
  password: z
    .string()
    .min(8, 'Password must be at least 8 characters')
    .regex(/[A-Z]/, 'Must contain an uppercase letter')
    .regex(/[0-9]/, 'Must contain a number'),
  confirmPassword: z.string(),
  role: z.enum(['user', 'admin', 'editor']),
  newsletter: z.boolean().default(false),
  birthDate: z.coerce.date().optional(),
}).refine((data) => data.password === data.confirmPassword, {
  message: 'Passwords do not match',
  path: ['confirmPassword'],
});

// Extract TypeScript type from the Zod schema
export type SignupFormData = z.infer<typeof signupSchema>;
// {
//   email: string;
//   password: string;
//   confirmPassword: string;
//   role: 'user' | 'admin' | 'editor';
//   newsletter: boolean;
//   birthDate?: Date;
// }
SignupFormData = { email: string; password: string; confirmPassword: string; role: "user" | "admin" | "editor"; newsletter: boolean; birthDate?: Date | undefined; }

      method: 'POST',

dynamic-form.tsx (discriminated union forms)
import { z } from 'zod';
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';

// Discriminated union schema — form shape depends on contact method
const contactSchema = z.discriminatedUnion('method', [
  z.object({
    method: z.literal('email'),
    email: z.string().email(),
    subject: z.string().min(1),
  }),
  z.object({
    method: z.literal('phone'),
    phone: z.string().regex(/^\+?[0-9]{10,14}$/),
    preferredTime: z.enum(['morning', 'afternoon', 'evening']),
  }),
  z.object({
    method: z.literal('mail'),
    address: z.string().min(10),
    zip: z.string().regex(/^[0-9]{5}$/),
  }),
]);

type ContactForm = z.infer<typeof contactSchema>;
// | { method: 'email'; email: string; subject: string }
// | { method: 'phone'; phone: string; preferredTime: 'morning' | ... }
// | { method: 'mail'; address: string; zip: string }

function ContactFormComponent() {
  const { register, handleSubmit, watch } = useForm<ContactForm>({
    resolver: zodResolver(contactSchema),
    defaultValues: { method: 'email' } as ContactForm,
  });

  const method = watch('method');

  return (
    <form onSubmit={handleSubmit((data) => {
      // data is narrowed based on validation
      if (data.method === 'email') {
        console.log(data.email, data.subject);  // both typed
      }
    })}>
      <select {...register('method')}>
        <option value="email">Email</option>
        <option value="phone">Phone</option>
        <option value="mail">Mail</option>
      </select>

      {method === 'email' && (
        <>
          <input {...register('email')} placeholder="Email" />
          <input {...register('subject')} placeholder="Subject" />
        </>
      )}
      {method === 'phone' && (
        <>
          <input {...register('phone')} placeholder="Phone" />
          <select {...register('preferredTime')}>
            <option value="morning">Morning</option>
            <option value="afternoon">Afternoon</option>
            <option value="evening">Evening</option>
          </select>
        </>
      )}
      {method === 'mail' && (
        <>
          <input {...register('address')} placeholder="Address" />
          <input {...register('zip')} placeholder="ZIP" />
        </>
      )}

      <button type="submit">Send</button>
    </form>
  );
}
Common Mistakes
Defining the TypeScript type separately from the Zod schema: interface FormData { ... } and z.object({ ... }) can drift apart
Use z.infer<typeof schema> as the single source of truth — never hand-write the form type
Forgetting the resolver: useForm<FormData>() without zodResolver — no runtime validation happens
Always pass resolver: zodResolver(schema) to connect type-level and runtime validation
Using register('nonExistentField') without noticing — this only errors with strict typing enabled
The generic parameter useForm<SignupFormData> ensures register only accepts valid field names
Key Rules
  • Define the Zod schema first, then derive the type with z.infer<typeof schema>
  • Pass the derived type to useForm<T> and use zodResolver(schema)
  • register() autocompletes field names and rejects invalid ones when typed
  • errors.fieldName?.message is typed as string | undefined
  • Use z.discriminatedUnion for forms whose shape changes based on a selector field

3 WebSocket Typing
WebSocket Typing WebSocket Socket.IO Ecosystem
WebSocket messages are inherently untyped — they arrive as raw strings or buffers. Typed message protocols catch wrong event names, missing payloads, and shape mismatches at compile time. Without typing, a renamed event silently breaks and you discover it in production.

Typed WebSocket patterns enforce contracts between client and server:

  • Discriminated union messages: Define all message types as a union with a type discriminant, then parse incoming messages with a type guard
  • Socket.IO typed events: Define interfaces for ServerToClientEvents, ClientToServerEvents, and InterServerEvents. Pass them as generics to Server and io
  • tRPC subscriptions: Use observable to emit typed events from the server that the client receives with full type safety
  • The key pattern is bidirectional type safety: the server knows what events the client expects, and vice versa
ws-protocol.ts (shared types)
// Discriminated union — all possible WebSocket messages
export type ClientMessage =
  | { type: 'join_room'; roomId: string; username: string }
  | { type: 'leave_room'; roomId: string }
  | { type: 'send_message'; roomId: string; text: string }
  | { type: 'typing'; roomId: string; isTyping: boolean };

export type ServerMessage =
  | { type: 'room_joined'; roomId: string; members: string[] }
  | { type: 'room_left'; roomId: string }
  | { type: 'new_message'; roomId: string; from: string; text: string; timestamp: number }
  | { type: 'user_typing'; roomId: string; username: string; isTyping: boolean }
  | { type: 'error'; code: number; message: string };

// Type guard for parsing incoming messages
export function isClientMessage(data: unknown): data is ClientMessage {
  if (typeof data !== 'object' || data === null) return false;
  const msg = data as Record<string, unknown>;
  return typeof msg.type === 'string' &&
    ['join_room', 'leave_room', 'send_message', 'typing'].includes(msg.type);
}

export function isServerMessage(data: unknown): data is ServerMessage {
  if (typeof data !== 'object' || data === null) return false;
  const msg = data as Record<string, unknown>;
  return typeof msg.type === 'string' &&
    ['room_joined', 'room_left', 'new_message', 'user_typing', 'error'].includes(msg.type);
}
ws-client.ts (typed WebSocket client)
import type { ClientMessage, ServerMessage } from './ws-protocol';
import { isServerMessage } from './ws-protocol';

class TypedWebSocket {
  private ws: WebSocket;
  private handlers = new Map<ServerMessage['type'], Set<(msg: any) => void>>();

  constructor(url: string) {
    this.ws = new WebSocket(url);
    this.ws.onmessage = (event) => {
      const data: unknown = JSON.parse(event.data);
      if (!isServerMessage(data)) return;

      // data is now ServerMessage — switch narrows further
      const listeners = this.handlers.get(data.type);
      listeners?.forEach(fn => fn(data));
    };
  }

  // Typed send — only accepts valid ClientMessage shapes
  send(message: ClientMessage): void {
    this.ws.send(JSON.stringify(message));
  }

  // Typed event listener — extracts the specific message variant
  on<T extends ServerMessage['type']>(
    type: T,
    handler: (msg: Extract<ServerMessage, { type: T }>) => void
  ): void {
    if (!this.handlers.has(type)) {
      this.handlers.set(type, new Set());
    }
    this.handlers.get(type)!.add(handler);
  }
}

// Usage
const ws = new TypedWebSocket('wss://chat.example.com');

// send is typed — autocompletes message shapes
ws.send({ type: 'join_room', roomId: 'lobby', username: 'Alice' });

// on is typed — handler receives the specific message variant
ws.on('new_message', (msg) => {
  // msg is: { type: 'new_message'; roomId: string; from: string; text: string; timestamp: number }
  console.log(`${msg.from}: ${msg.text}`);
});

ws.on('error', (msg) => {
  // msg is: { type: 'error'; code: number; message: string }
  console.error(`Error ${msg.code}: ${msg.message}`);
});

// ws.on('unknown_event', () => {});

socketio-typed.ts (Socket.IO typed events)
import { Server, Socket } from 'socket.io';

// Define event interfaces for bidirectional type safety
interface ServerToClientEvents {
  message: (data: { from: string; text: string; timestamp: number }) => void;
  userJoined: (data: { username: string; roomId: string }) => void;
  typing: (data: { username: string; isTyping: boolean }) => void;
}

interface ClientToServerEvents {
  joinRoom: (roomId: string, callback: (success: boolean) => void) => void;
  sendMessage: (data: { roomId: string; text: string }) => void;
  startTyping: (roomId: string) => void;
  stopTyping: (roomId: string) => void;
}

interface InterServerEvents {
  ping: () => void;
}

interface SocketData {
  username: string;
  joinedAt: Date;
}

// Typed server — all four interfaces as generics
const io = new Server<
  ClientToServerEvents,
  ServerToClientEvents,
  InterServerEvents,
  SocketData
>();

io.on('connection', (socket) => {
  // socket.on autocompletes ClientToServerEvents
  socket.on('joinRoom', (roomId, callback) => {
    // roomId is string, callback is (success: boolean) => void
    socket.join(roomId);
    socket.data.username = 'Alice';
    socket.data.joinedAt = new Date();
    callback(true);

    // socket.emit autocompletes ServerToClientEvents
    socket.to(roomId).emit('userJoined', {
      username: socket.data.username,
      roomId,
    });
  });

  socket.on('sendMessage', (data) => {
    // data is: { roomId: string; text: string }
    io.to(data.roomId).emit('message', {
      from: socket.data.username,
      text: data.text,
      timestamp: Date.now(),
    });
  });

  // Type error: 'unknownEvent' doesn't exist on ClientToServerEvents
  // socket.on('unknownEvent', () => {});
});
Argument of type '"unknownEvent"' is not assignable to parameter of type 'keyof ClientToServerEvents'.
Common Mistakes
Trusting JSON.parse(event.data) without validation: the parsed result is any
Always validate with a type guard or Zod before narrowing: if (isServerMessage(data))
Socket.IO without type parameters: new Server() — all events accept any
Always provide all four generics: Server<C2S, S2C, Inter, Data>
Duplicating message types on client and server instead of sharing them
Define event interfaces in a shared package (@app/shared) imported by both client and server
Key Rules
  • Define all WebSocket messages as a discriminated union with a type field
  • Always validate incoming messages at runtime — JSON.parse returns any
  • Use Extract<Union, { type: T }> to narrow to a specific message variant in typed handlers
  • Socket.IO: provide all four generic parameters to Server and Socket
  • Share event type definitions between client and server via a shared package

4 Worker Threads Typing
Worker Threads Typing Worker postMessage Node.js
Worker threads run CPU-intensive tasks off the main thread, but postMessage is untyped by default. Typed message channels prevent sending the wrong data shape to a worker and catch handler mismatches at compile time instead of with silent failures at runtime.

Worker messaging is untyped by default because postMessage accepts any. Key typed patterns:

  • Discriminated union messages: Define MainToWorker and WorkerToMain unions, then wrap postMessage and onmessage with typed functions
  • Web Workers: new Worker('./worker.ts') has untyped postMessage. Wrap it in a typed facade
  • Node.js worker_threads: parentPort.postMessage and worker.postMessage are both any. Apply the same typed wrapper pattern
  • Transferable types: ArrayBuffer, MessagePort, OffscreenCanvas can be transferred (zero-copy). The transfer list is typed as Transferable[]
  • SharedArrayBuffer: Shared memory between threads. Use Atomics for thread-safe operations. TypeScript types Atomics.wait, Atomics.notify, etc.
worker-protocol.ts (shared types)
// Typed messages between main thread and worker
export type MainToWorker =
  | { type: 'process_image'; imageData: ArrayBuffer; width: number; height: number }
  | { type: 'compute_hash'; data: ArrayBuffer; algorithm: 'sha256' | 'md5' }
  | { type: 'cancel' };

export type WorkerToMain =
  | { type: 'progress'; percent: number }
  | { type: 'image_result'; processedData: ArrayBuffer; width: number; height: number }
  | { type: 'hash_result'; hash: string; algorithm: string }
  | { type: 'error'; message: string };
typed-worker.ts (typed wrapper)
import type { MainToWorker, WorkerToMain } from './worker-protocol';

// Typed Worker wrapper — enforces message shapes
class TypedWorker {
  private worker: Worker;
  private handlers = new Map<
    WorkerToMain['type'],
    Set<(msg: any) => void>
  >();

  constructor(scriptUrl: string | URL) {
    this.worker = new Worker(scriptUrl, { type: 'module' });
    this.worker.onmessage = (event: MessageEvent<WorkerToMain>) => {
      const msg = event.data;
      const listeners = this.handlers.get(msg.type);
      listeners?.forEach(fn => fn(msg));
    };
  }

  // Typed postMessage — only accepts MainToWorker shapes
  send(message: MainToWorker, transfer?: Transferable[]): void {
    this.worker.postMessage(message, transfer ?? []);
  }

  // Typed event listener with narrowing
  on<T extends WorkerToMain['type']>(
    type: T,
    handler: (msg: Extract<WorkerToMain, { type: T }>) => void
  ): void {
    if (!this.handlers.has(type)) {
      this.handlers.set(type, new Set());
    }
    this.handlers.get(type)!.add(handler);
  }

  terminate(): void {
    this.worker.terminate();
  }
}

// Usage — fully typed
const worker = new TypedWorker('./image-worker.js');

// send only accepts valid MainToWorker messages
const buffer = new ArrayBuffer(1024);
worker.send(
  { type: 'process_image', imageData: buffer, width: 100, height: 100 },
  [buffer]  // Transfer the ArrayBuffer (zero-copy)
);

// on handlers are narrowed to the specific message variant
worker.on('progress', (msg) => {
  // msg is: { type: 'progress'; percent: number }
  console.log(`Progress: ${msg.percent}%`);
});

worker.on('image_result', (msg) => {
  // msg is: { type: 'image_result'; processedData: ArrayBuffer; width: number; height: number }
  console.log(`Got ${msg.processedData.byteLength} bytes`);
});

worker.on('error', (msg) => {
  console.error(msg.message);
});

// SharedArrayBuffer — shared memory between threads
// Both threads see the same memory; Atomics prevents race conditions

// Main thread
const shared = new SharedArrayBuffer(Int32Array.BYTES_PER_ELEMENT * 4);
const view = new Int32Array(shared);

// Pass to worker (shared, not transferred)
worker.postMessage({ type: 'shared_init', buffer: shared });

// Atomic operations — typed by TypeScript
Atomics.store(view, 0, 42);        // thread-safe write
const val = Atomics.load(view, 0);  // thread-safe read: 42
Atomics.add(view, 1, 10);           // atomic increment

// Wait/notify — typed parameters
// Atomics.wait(view, 0, expectedValue, timeout?)
// Returns: 'ok' | 'not-equal' | 'timed-out'
const result: 'ok' | 'not-equal' | 'timed-out' = Atomics.wait(view, 0, 0, 1000);

// Worker side: Atomics.notify(view, 0) wakes waiting threads
Atomics.wait returns: "ok" | "not-equal" | "timed-out" Atomics.load returns: number
Common Mistakes
Using worker.postMessage(data) without typing — postMessage accepts any
Wrap postMessage in a typed function: send(msg: MainToWorker)
Forgetting to transfer ArrayBuffer ownership: postMessage({ data: buffer }) copies the buffer
Pass transferables: postMessage(msg, [buffer]) for zero-copy. The original reference becomes detached
Using SharedArrayBuffer without Atomics — race conditions on shared memory
Always use Atomics.load/Atomics.store instead of direct array access for shared buffers
Key Rules
  • Define MainToWorker and WorkerToMain discriminated unions in a shared file
  • Wrap postMessage and onmessage with typed functions on both sides
  • Transfer ArrayBuffer with the second argument to postMessage for zero-copy
  • Use Atomics for thread-safe operations on SharedArrayBuffer
  • Atomics.wait returns a typed string literal union: 'ok' | 'not-equal' | 'timed-out'

5 Migration Strategies: JS to TS
Migration Strategies: JS to TS Migration Incremental Ecosystem
Most TypeScript projects start as JavaScript. A successful migration is incremental — you never need to stop shipping features. Understanding the migration ladder from // @ts-check to full strict mode lets you adopt TypeScript at any pace without breaking your team's velocity.

The migration path from JavaScript to strict TypeScript has five stages:

  • Stage 1: // @ts-check — Add to individual JS files. TypeScript checks types via JSDoc and inference. Zero setup. Catches many bugs immediately
  • Stage 2: allowJs + checkJs — Enable in tsconfig.json. TypeScript checks all JS files in the project. Use // @ts-ignore sparingly for false positives
  • Stage 3: Rename to .ts — Rename files one by one. Use any for complex types you cannot define yet. Write .d.ts files for untyped third-party modules
  • Stage 4: any to unknown — Replace any with unknown and add proper narrowing. This is where you gain the most type safety
  • Stage 5: Enable strict flags — Turn on strict one flag at a time: noImplicitAny, strictNullChecks, strictFunctionTypes, etc.
Stage 1: // @ts-check in a JS file
// @ts-check

/**
 * @param {string} name
 * @param {number} age
 * @returns {{ name: string; age: number; greeting: string }}
 */
function createUser(name, age) {
  return {
    name,
    age,
    greeting: `Hello, ${name}!`,
  };
}

const user = createUser('Alice', 30);
console.log(user.greeting);  // string — typed via JSDoc

// createUser(42, 'not-a-number');
// Argument of type 'number' is not assignable to parameter of type 'string'.

    "strictFunctionTypes": true,      // Contravariant function parameter checking

    // "strict": true                  // Enables ALL strict flags

}

Common Mistakes
Turning on "strict": true all at once in a large codebase — thousands of errors, team loses confidence
Enable one strict flag at a time. Fix errors, commit, move to the next flag. Track error count over time
Using any as a permanent escape hatch: const data: any = ... throughout the codebase
Use unknown for truly unknown values. Use // @ts-expect-error (not @ts-ignore) for intentional suppressions — it errors when the suppression becomes unnecessary
Renaming all files from .js to .ts in a single commit
Migrate one module at a time. Start with leaf modules (no importers), then work inward. Each file rename is a separate, reviewable commit
Key Rules
  • Start with // @ts-check in individual files — zero cost, immediate type checking
  • Enable allowJs + checkJs to check the entire project without renaming files
  • Write .d.ts files for untyped third-party modules that lack @types/* packages
  • Replace any with unknown and add runtime narrowing (type guards or Zod)
  • Enable strict flags one at a time: noImplicitAny first, then strictNullChecks, then the rest
  • Use // @ts-expect-error instead of // @ts-ignore — it flags itself when no longer needed
