R1. Mapped Type Key Remapping Advanced Types 10 days overdue · rep 0
Remap keys with as in mapped types: { [K in keyof T as NewKey]: T[K] }. This lets you rename, filter, or transform property keys at the type level.
Key Pattern 
// Remap every key to a getter name using template literals.
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

interface User { name: string; age: number }

type UserGetters = Getters<User>;
// { getName: () => string; getAge: () => number }
{ getName: () => string; getAge: () => number }
Challenge Exercise

Create a type Setters<T> that maps {name: string} to {setName: (v: string) => void}. Each property foo becomes setFoo(v: T[foo]) => void.

Pitfall: Forgetting string & before K when using template literals. Keys can be string | number | symbol, but template literals only accept string. The intersection string & K filters out non-string keys.
R2. this Types & Fluent Interfaces OOP Patterns 9 days overdue · rep 2
this as a return type enables fluent method chains that preserve the subclass type. The returned this is polymorphic — it refers to the actual runtime class, not the declaring class.
Key Pattern 
class Builder {
  private name = "";
  private age = 0;

  setName(n: string): this {
    this.name = n;
    return this;  // returns the subclass type, not just Builder
  }

  setAge(a: number): this {
    this.age = a;
    return this;
  }
}

class UserBuilder extends Builder {
  private role = "";
  setRole(r: string): this {
    this.role = r;
    return this;
  }
}

// Fluent chain preserves UserBuilder through inherited methods
const user = new UserBuilder()
  .setName("Alice")   // returns UserBuilder, not Builder
  .setAge(30)          // still UserBuilder
  .setRole("admin");   // UserBuilder — would fail if setName returned Builder
Challenge Exercise

Create a FluentValidator class with .required(), .minLength(n), and .email() methods that all return this. Then create a subclass PasswordValidator that adds .hasUpperCase() and .hasDigit(). Verify that chaining inherited methods still returns the subclass type.

Pitfall: Arrow functions don’t have their own this — they capture the enclosing lexical this. You cannot use polymorphic this return types in arrow function properties. Stick with regular method declarations for fluent chains.
R3. Async Patterns & Promise Typing Async 9 days overdue · rep 2
Awaited<T> recursively unwraps nested Promises. Promise.allSettled returns PromiseSettledResult<T>[], which is a union of PromiseFulfilledResult<T> and PromiseRejectedResult.
Key Pattern 
// Awaited unwraps nested Promise types
type Nested = Promise<Promise<string>>;
type Flat = Awaited<Nested>;  // string

// Extract the resolved type from any async function
async function fetchUser() {
  return { id: 1, name: "Alice" };
}
type User = Awaited<ReturnType<typeof fetchUser>>;
// { id: number; name: string }
User = { id: number; name: string }
Challenge Exercise

Type a function parallel<T extends readonly (() => Promise<any>)[]>(...fns: T) that returns Promise<{ [K in keyof T]: Awaited<ReturnType<T[K]>> }>. Test it with functions returning different types and verify the tuple result is correctly inferred.

Pitfall: Promise.all with mixed types requires explicit tuple typing. Passing [fetchUser(), fetchPosts()] as an array loses the per-element type — use as const or declare the array as a tuple to preserve each position’s type.
R4. TypeScript Performance Tooling 9 days overdue · rep 1
--generateTrace outputs a Chrome-compatible trace for profiling type-checking. Avoid deep recursive types, excessive conditional chains, and distributing over large unions (100+ members causes exponential instantiation).
Key Pattern 
// Generate a trace, then open it in chrome://tracing
// $ tsc --generateTrace traceDir
// $ open chrome://tracing  →  Load →  traceDir/trace.json

// Bad: unbounded recursion
type DeepPartialSlow<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartialSlow<T[K]> : T[K];
};

// Better: limit recursion depth with a counter tuple
type DeepPartial<T, Depth extends unknown[] = []> =
  Depth["length"] extends 5 ? T :  // bail out at depth 5
  {
    [K in keyof T]?: T[K] extends object
      ? DeepPartial<T[K], [...Depth, unknown]>
      : T[K];
  };
Challenge Exercise

Refactor a type DeepPartial<T> that’s slow on large objects — limit recursion depth with a counter tuple. Measure with --generateTrace before and after on a 10-level nested object type.

Pitfall: Distributing over large unions (100+ members) causes exponential type instantiation. Each conditional type distributes over every member individually. Wrap the checked type in a tuple ([T] extends [U]) to disable distribution when you don’t need it.
R5. React Component Props React + TS 9 days overdue · rep 1
React.FC<Props> (optional), ComponentPropsWithRef<"button">, and PropsWithChildren<Props> are the core utility types for React component typing. Prefer ComponentPropsWithoutRef for most components.
Key Pattern 
import React from "react";

// Extend native button props with custom variants
type ButtonProps = React.ComponentPropsWithoutRef<"button"> & {
  variant: "primary" | "secondary";
  size?: "sm" | "md" | "lg";
};

function Button({ variant, size = "md", children, ...rest }: ButtonProps) {
  return (
    <button className={`btn btn-${variant} btn-${size}`} {...rest}>
      {children}
    </button>
  );
}

// All native button props work + custom props are type-checked
<Button variant="primary" onClick={handleClick} disabled />  // OK
<Button variant="tertiary" />  // Error: not in union
Challenge Exercise

Type a polymorphic <Box as={...}> component where the allowed props change based on the as prop. When as="a", it should accept href; when as="button", it should accept onClick but not href.

Pitfall: Using React.FC implicitly added children in older React types — this was removed in React 18 types. If you rely on implicit children, your component will break after upgrading. Always explicitly type children with PropsWithChildren<Props>.
R6. Type-Safe pipe() & compose() Functional Patterns Due today · rep 0
pipe(value, fn1, fn2, ...) passes a value through functions left-to-right with full type inference between each step. TypeScript can’t infer variadic function composition — you need overload signatures for each arity.
Key Pattern 
// Overloaded pipe with type inference between each step
function pipe<A>(value: A): A;
function pipe<A, B>(value: A, fn1: (a: A) => B): B;
function pipe<A, B, C>(value: A, fn1: (a: A) => B, fn2: (b: B) => C): C;
function pipe<A, B, C, D>(
  value: A, fn1: (a: A) => B, fn2: (b: B) => C, fn3: (c: C) => D
): D;
function pipe(value: unknown, ...fns: Function[]): unknown {
  return fns.reduce((acc, fn) => fn(acc), value);
}

// Usage — each step’s type flows to the next
const result = pipe(
  "  Hello World  ",
  (s) => s.trim(),         // string → string
  (s) => s.toLowerCase(),  // string → string
  (s) => s.split(" ")      // string → string[]
);
// result: string[]
result: string[]
Challenge Exercise

Implement a variadic pipe that accepts up to 6 functions with full type inference between each step. Verify that passing a mismatched function (e.g., number => string after string => number slot) produces a type error.

Pitfall: TS can’t infer variadic function composition generically — a single generic signature for N functions won’t propagate types between steps. You need one overload signature per supported arity (typically 1–6 or 1–10).
1 Practical TypeScript Tooling
1. Type-Safe CLI Tools with Commander & Yargs Tooling
Every Node.js CLI tool needs argument parsing. Type-safe parsing catches flag typos and wrong types at compile time, eliminating an entire class of runtime bugs in your dev tooling.

Two dominant libraries exist for CLI argument parsing in TypeScript: commander and yargs. Both can be used with strong types, but the approaches differ. Commander relies on explicit interface definitions and manual casting, while yargs has built-in chaining that infers option types from the .option() calls.

The key insight is to define your CLI options as an interface first, then ensure the parser produces that exact shape. This makes the rest of your CLI code downstream of a single, well-typed options object.

TypeScript · example 1 of 3 — Commander with typed options
import { Command } from "commander";

// 1. Define the shape your CLI expects
interface CliOptions {
  readonly port: number;
  readonly host: string;
  readonly verbose: boolean;
  readonly config?: string;
}

// 2. Build the command, mapping flags to the interface
const program = new Command()
  .name("my-server")
  .option("-p, --port <number>", "port to listen on", "3000")
  .option("-H, --host <string>", "hostname", "localhost")
  .option("-v, --verbose", "enable verbose logging", false)
  .option("-c, --config <path>", "path to config file")
  .parse();

// 3. Cast once at the boundary, typed everywhere after
const opts: CliOptions = {
  port: Number(program.opts().port),
  host: program.opts().host as string,
  verbose: program.opts().verbose as boolean,
  config: program.opts().config as string | undefined,
};

// 4. All downstream code uses the typed object
function startServer(options: CliOptions): void {
  console.log(`Listening on ${options.host}:${options.port}`);
  if (options.verbose) {
    console.log("Verbose mode enabled");
  }
}
Listening on localhost:3000
TypeScript · example 2 of 3 — Yargs with .option() type inference
import yargs from "yargs";
import { hideBin } from "yargs/helpers";

// Yargs infers types from .option() chains
const argv = yargs(hideBin(process.argv))
  .option("port", {
    alias: "p",
    type: "number",
    default: 3000,
    describe: "Port to listen on",
  })
  .option("host", {
    alias: "H",
    type: "string",
    default: "localhost",
    describe: "Hostname to bind",
  })
  .option("verbose", {
    alias: "v",
    type: "boolean",
    default: false,
  })
  .strict()         // reject unknown flags
  .parseSync();     // synchronous parse

// argv.port is number, argv.host is string, argv.verbose is boolean
// No manual casting needed — yargs infers from the option config
console.log(argv.port);    // number
console.log(argv.host);    // string
console.log(argv.verbose); // boolean
argv: { port: number; host: string; verbose: boolean; _: (string | number)[]; $0: string }
TypeScript · example 3 of 3 — Type-safe command registry
// Build a registry where each command is typed independently
interface CommandDef<T> {
  name: string;
  description: string;
  parse: (args: string[]) => T;
  execute: (opts: T) => Promise<void>;
}

function defineCommand<T>(def: CommandDef<T>): CommandDef<T> {
  return def;
}

// Each command defines its own option shape
const buildCmd = defineCommand({
  name: "build",
  description: "Build the project",
  parse: (args) => ({
    watch: args.includes("--watch"),
    outDir: args.find((a) => a.startsWith("--out="))?.slice(6) ?? "dist",
  }),
  execute: async (opts) => {
    // opts is { watch: boolean; outDir: string } — fully inferred
    console.log(`Building to ${opts.outDir}, watch: ${opts.watch}`);
  },
});

const commands = [buildCmd] as const;

// Dispatcher looks up by name and runs with the correct type
async function dispatch(name: string, args: string[]): Promise<void> {
  const cmd = commands.find((c) => c.name === name);
  if (!cmd) throw new Error(`Unknown command: ${name}`);
  const opts = cmd.parse(args);
  await cmd.execute(opts);
}
Common Mistakes:
❌ Using as any for parsed CLI arguments: const port = opts.port as any. This silences the compiler but hides type mismatches (e.g., port is actually a string from the parser).
✅ Define an interface and cast once at the boundary: const opts: CliOptions = { port: Number(program.opts().port) }. All downstream code stays typed.
❌ Forgetting to coerce numeric options from Commander (which returns strings by default).
✅ Always use Number() or a custom coercion function for numeric flags, or switch to yargs which handles type: "number" natively.
Key Rules:
  • 2026-07-04 (Day 31)
  • Define your CLI option interface first, then make the parser produce that shape — not the other way around.
  • Cast at the boundary (where raw args become typed objects) and nowhere else. All downstream code should receive the typed interface.
  • Use .strict() in yargs or .allowUnknownOption(false) in commander to reject typos at runtime.
  • For complex CLIs with subcommands, define each command’s options independently with a generic CommandDef<T> pattern.
Practice Exercises:
1Build a CLI tool with yargs that accepts --input <file>, --output <file>, and --format json|csv options. Make format a union type, not just a string. Verify that passing --format xml fails at runtime with .choices().
2Create a type-safe command registry using the CommandDef<T> pattern. Add three commands (init, build, deploy) with different option shapes. Write a dispatcher that routes to the correct handler.
3Refactor a CLI that uses process.argv slicing and manual parsing into a yargs-based solution. Compare the type safety: add a new flag and verify the compiler catches all call sites that need updating.
Connections: The CommandDef<T> pattern uses the same generic inference trick as Builder Pattern (Type State) from earlier guides. CLI option typing pairs naturally with Type-Safe Environment Variables (Concept 5 below) — both validate external input at the program boundary.
2. ts-morph — Programmatic TypeScript Manipulation Tooling
Generate code, refactor safely, extract types from source files. ts-morph wraps the TypeScript compiler API in a developer-friendly interface. Used by codegen tools, migration scripts, and custom transformers.

The raw TypeScript compiler API (ts.createSourceFile, etc.) is powerful but extremely verbose. ts-morph provides a fluent, high-level wrapper that makes reading, modifying, and generating TypeScript source files straightforward.

Common use cases include: generating barrel files (index.ts), adding properties to interfaces across a codebase, extracting type information for documentation, and building custom codemods that are type-aware (unlike regex-based find-and-replace).

TypeScript · example 1 of 3 — Reading exports from a source file
import { Project } from "ts-morph";

const project = new Project({
  tsConfigFilePath: "./tsconfig.json",
});

// Load a specific source file
const sourceFile = project.getSourceFileOrThrow("src/utils.ts");

// List all exported functions with their signatures
const exportedFunctions = sourceFile.getFunctions()
  .filter((fn) => fn.isExported());

for (const fn of exportedFunctions) {
  const name = fn.getName();
  const params = fn.getParameters()
    .map((p) => `${p.getName()}: ${p.getType().getText()}`)
    .join(", ");
  const returnType = fn.getReturnType().getText();

  console.log(`export function ${name}(${params}): ${returnType}`);
}

// Example output:
// export function clamp(value: number, min: number, max: number): number
// export function slugify(text: string): string
export function clamp(value: number, min: number, max: number): number export function slugify(text: string): string
TypeScript · example 2 of 3 — Adding a property to an interface
import { Project } from "ts-morph";

const project = new Project();
const sourceFile = project.addSourceFileAtPath("src/models/user.ts");

// Find the User interface
const userInterface = sourceFile.getInterfaceOrThrow("User");

// Add a new property if it doesn't already exist
if (!userInterface.getProperty("createdAt")) {
  userInterface.addProperty({
    name: "createdAt",
    type: "Date",
    hasQuestionToken: false,  // required property
  });
}

// Add a readonly optional property
if (!userInterface.getProperty("deletedAt")) {
  userInterface.addProperty({
    name: "deletedAt",
    type: "Date",
    hasQuestionToken: true,   // optional
    isReadonly: true,
  });
}

// IMPORTANT: save changes back to disk
sourceFile.saveSync();
console.log(sourceFile.getFullText());
interface User { id: string; name: string; email: string; createdAt: Date; readonly deletedAt?: Date; }
TypeScript · example 3 of 3 — Generating a module from a schema
import { Project, StructureKind } from "ts-morph";

// Schema definition (could come from an API spec, DB, etc.)
const schema = {
  name: "Product",
  fields: [
    { name: "id", type: "string", readonly: true },
    { name: "title", type: "string", readonly: false },
    { name: "price", type: "number", readonly: false },
    { name: "tags", type: "string[]", readonly: false },
  ],
};

const project = new Project();

// Create a brand-new source file
const file = project.createSourceFile(`src/generated/${schema.name.toLowerCase()}.ts`, {
  statements: [
    // Generate the interface
    {
      kind: StructureKind.Interface,
      name: schema.name,
      isExported: true,
      properties: schema.fields.map((f) => ({
        name: f.name,
        type: f.type,
        isReadonly: f.readonly,
      })),
    },
    // Generate a factory function
    {
      kind: StructureKind.Function,
      name: `create${schema.name}`,
      isExported: true,
      parameters: [{ name: "data", type: `Omit<${schema.name}, "id">` }],
      returnType: schema.name,
      statements: [
        `return { ...data, id: crypto.randomUUID() };`,
      ],
    },
  ],
});

file.saveSync();
console.log(file.getFullText());
export interface Product { readonly id: string; title: string; price: number; tags: string[]; } export function createProduct(data: Omit<Product, "id">): Product { return { ...data, id: crypto.randomUUID() }; }
Common Mistakes:
❌ Modifying the AST without calling saveSync() or save(). Changes are held in memory only — forgetting to save means your modifications are silently lost.
✅ Always call sourceFile.saveSync() (or await sourceFile.save()) after mutations. Or use project.saveSync() to save all modified files at once.
❌ Creating a Project without a tsconfig, then wondering why type resolution fails for imports.
✅ Pass { tsConfigFilePath: "./tsconfig.json" } to new Project() so ts-morph can resolve module paths and infer types correctly.
Key Rules:
  • ts-morph wraps the TS compiler API — it loads the real type checker, so getType() returns fully resolved types.
  • Use getOrThrow variants (e.g., getInterfaceOrThrow) during codegen — fail fast if the expected structure is missing.
  • For large codebases, create the Project once and reuse it. Each new Project() call re-parses the entire program.
  • Always save after mutations. Batch multiple changes, then save once for efficiency.

3. Import Attributes & JSON/CSS Modules TypeScript 5.x
import data from './config.json' with { type: 'json' } — TC39 stage 3, supported in TS 5.3+. Type-safe imports of non-JS resources eliminate require() hacks and enable bundler-native module handling.

Import attributes (formerly “import assertions”) tell the runtime and bundler the expected type of the imported module. This is critical for security (preventing a JSON file from being executed as JavaScript) and for tooling (bundlers can handle each type appropriately).

The syntax changed from assert { type: 'json' } (deprecated) to with { type: 'json' }. TypeScript 5.3+ supports the with keyword. You also need resolveJsonModule: true and typically moduleResolution: "bundler" in your tsconfig.

TypeScript · example 1 of 3 — JSON imports with import attributes
// tsconfig.json: { "compilerOptions": { "resolveJsonModule": true, "moduleResolution": "bundler" } }

// OLD (deprecated) — import assertion with "assert"
// import config from "./config.json" assert { type: "json" };

// NEW (TS 5.3+) — import attribute with "with"
import config from "./config.json" with { type: "json" };

// TypeScript infers the JSON structure as a type
// config: { port: number; host: string; debug: boolean }

console.log(config.port);    // number — fully typed
console.log(config.host);    // string — autocomplete works
// config.missing;            // Error: Property 'missing' does not exist

// Dynamic import with attributes
async function loadLocale(lang: string) {
  const messages = await import(`./locales/${lang}.json`, {
    with: { type: "json" },
  });
  return messages.default;
}
config: { port: number; host: string; debug: boolean }

TypeScript · example 3 of 3 — tsconfig for module resolution
// tsconfig.json for modern projects using import attributes
{
  "compilerOptions": {
    // Enable JSON imports
    "resolveJsonModule": true,

    // "bundler" mode: lets TS understand bare specifiers, .json,
    // .css, and other non-JS imports that your bundler resolves
    "moduleResolution": "bundler",

    // Required for import attributes
    "module": "esnext",

    // Emit import attributes in output (Node 20.10+ supports them)
    "target": "esnext",

    // Allow importing .ts files with .js extensions (bundler handles it)
    "allowImportingTsExtensions": true,
    "noEmit": true
  }
}

// With this config, all of these work:
// import data from "./data.json" with { type: "json" };
// import styles from "./App.module.css";
// import { helper } from "./utils.js";  // resolves to utils.ts
Common Mistakes:
❌ Using the deprecated assert keyword: import x from "y" assert { type: "json" }. The assert syntax was renamed to with in the TC39 proposal and is being removed from engines.
✅ Use the with keyword: import x from "y" with { type: "json" }. This is the final syntax standardized by TC39.
❌ Setting moduleResolution: "node" in a project that uses bundler-style imports (.json, .css, extensionless).
✅ Use moduleResolution: "bundler" for projects built with Vite, webpack, esbuild, or similar. Use "node16" or "nodenext" for pure Node.js projects.
Key Rules:
  • Import attributes use with, not assert. The old syntax is deprecated and will be removed.
  • resolveJsonModule: true + moduleResolution: "bundler" is the standard config for modern frontend projects.
  • For CSS modules, use a wildcard declaration (*.module.css) for basic support, or a code generator (typed-css-modules) for per-file precise types.
  • Dynamic imports support attributes too: await import("./x.json", { with: { type: "json" } }).

2 Advanced Type Patterns
4. Const Enum Alternatives & Tree-Shaking Build Optimization
const enum inlines values but breaks --isolatedModules (required by esbuild, swc, Babel, and Vite). Alternatives like literal unions and as const objects tree-shake better and work with every build tool.

const enum was designed to inline enum values at compile time, avoiding the runtime object that regular enums create. However, it requires the full TypeScript type-checker to resolve — meaning tools that transpile file-by-file (--isolatedModules mode) cannot handle them.

Since most modern build pipelines use esbuild, swc, or Babel for speed (all of which require isolatedModules), const enum is effectively unusable in modern projects. The alternatives — as const objects and string literal unions — produce the same developer experience with better tooling compatibility.

TypeScript · example 1 of 4 — The problem with const enum
// This works with tsc but BREAKS with --isolatedModules
const enum Direction {
  Up = "UP",
  Down = "DOWN",
  Left = "LEFT",
  Right = "RIGHT",
}

// tsc inlines the values:
// const d = "UP";
const d = Direction.Up;

// But esbuild/swc/Babel can't do this — they see:
// Error: Cannot access ambient const enums when
//        'isolatedModules' is enabled.

// Also broken when const enum is in a .d.ts from a library:
// consumers can't inline values they didn't compile
Cannot access ambient const enums when 'isolatedModules' is enabled. ts(2748)
TypeScript · example 2 of 4 — as const objects as enum replacement
// The "as const" object pattern — works everywhere
const Direction = {
  Up: "UP",
  Down: "DOWN",
  Left: "LEFT",
  Right: "RIGHT",
} as const;

// Extract the union type from the object values
type Direction = (typeof Direction)[keyof typeof Direction];
// type Direction = "UP" | "DOWN" | "LEFT" | "RIGHT"

// Usage is identical to an enum
function move(dir: Direction): void {
  console.log(`Moving ${dir}`);
}

move(Direction.Up);    // OK — "UP"
move(Direction.Down);  // OK — "DOWN"
move("UP");            // OK — literal value also works
move("DIAGONAL");      // Error: not in the union
Direction = "UP" | "DOWN" | "LEFT" | "RIGHT"
TypeScript · example 3 of 4 — String literal unions with exhaustive switch
// Sometimes you don't even need the object — just the union
type Status = "idle" | "loading" | "success" | "error";

// Exhaustive switch with never check
function getStatusMessage(status: Status): string {
  switch (status) {
    case "idle":    return "Waiting to start...";
    case "loading": return "Loading data...";
    case "success": return "Done!";
    case "error":   return "Something went wrong.";
    default: {
      // If you add a new Status but forget a case,
      // this line will produce a compile error
      const _exhaustive: never = status;
      return _exhaustive;
    }
  }
}

// Adding "retrying" to Status without updating the switch:
// Error: Type '"retrying"' is not assignable to type 'never'
TypeScript · example 4 of 4 — Helper function pattern
// Full pattern: object for namespacing + type for constraint
const HttpMethod = {
  Get: "GET",
  Post: "POST",
  Put: "PUT",
  Patch: "PATCH",
  Delete: "DELETE",
} as const;

type HttpMethod = (typeof HttpMethod)[keyof typeof HttpMethod];

// Helper: create a typeguard for any as-const object
function isValueOf<T extends Record<string, string>>(
  obj: T,
  value: string
): value is T[keyof T] {
  return Object.values(obj).includes(value);
}

// Runtime validation with type narrowing
function handleRequest(method: string): void {
  if (isValueOf(HttpMethod, method)) {
    // method is now HttpMethod (narrowed from string)
    console.log(`Valid method: ${method}`);
  } else {
    console.log(`Unknown method: ${method}`);
  }
}

handleRequest("GET");     // Valid method: GET
handleRequest("CONNECT"); // Unknown method: CONNECT
Valid method: GET Unknown method: CONNECT
Common Mistakes:
❌ Using const enum in a library published to npm. Consumers using esbuild, swc, or Babel (i.e., most projects) cannot resolve the inlined values and get a compile error.
✅ Use as const objects in libraries. They work with every build tool and every consumer, regardless of their isolatedModules setting.
❌ Using a regular enum and expecting tree-shaking to remove unused members. Enums compile to an IIFE that bundlers often can’t analyze.
✅ as const objects and literal unions tree-shake perfectly because they’re just plain objects and strings — no IIFE wrapper.
Key Rules:
  • const enum is incompatible with --isolatedModules (required by esbuild, swc, Babel, Vite). Avoid it in modern projects.
  • The as const object + extracted union pattern gives you the same DX as enums: dot-access, autocomplete, and exhaustive checking.
  • For simple cases, a bare string literal union (type X = "a" | "b") is the simplest and most tree-shakeable option.
  • Use the never default case pattern in switch statements to catch missing cases at compile time.
Practice Exercises:
1Refactor a codebase that uses const enum Color { Red, Green, Blue } to an as const object. Verify that all usages (Color.Red, switch statements, function parameters) still work identically.
2Create a Permission enum alternative with as const that supports bitwise flags: Read = 1, Write = 2, Execute = 4. Type a function that accepts a union of these values and performs bitwise checks.

5. Type-Safe Environment Variables Runtime Safety
process.env.DB_URL is string | undefined — easy to forget the undefined check. Type-safe env validation catches missing variables at startup, not in production at 3am when a database connection string is missing.

Environment variables are the primary configuration mechanism for deployed applications, but TypeScript’s built-in typing for them is essentially Record<string, string | undefined>. Every access returns string | undefined, making it easy to forget the null check and crash at runtime.

The solution is to validate all required env vars at application startup using a schema validation library like Zod. If any variable is missing or has the wrong format, the app fails immediately with a clear error — not minutes later in an unrelated code path.

TypeScript · example 1 of 3 — Zod schema for env vars
import { z } from "zod";

// Define the schema for your env vars
const envSchema = z.object({
  NODE_ENV: z.enum(["development", "production", "test"]),
  PORT: z.coerce.number().int().min(1).max(65535).default(3000),
  DATABASE_URL: z.string().url(),
  REDIS_URL: z.string().url().optional(),
  API_KEY: z.string().min(1, "API_KEY cannot be empty"),
  LOG_LEVEL: z.enum(["debug", "info", "warn", "error"]).default("info"),
});

// Validate at startup — fail fast if anything is wrong
function validateEnv() {
  const result = envSchema.safeParse(process.env);

  if (!result.success) {
    console.error("Invalid environment variables:");
    console.error(result.error.flatten().fieldErrors);
    process.exit(1);
  }

  return result.data;
}

// Export the validated, fully-typed env object
export const env = validateEnv();

// Now every access is fully typed — no undefined checks needed
// env.PORT         — number (not string!)
// env.DATABASE_URL — string (guaranteed to be a valid URL)
// env.REDIS_URL    — string | undefined (marked optional)
// env.NODE_ENV     — "development" | "production" | "test"
env: { NODE_ENV: "development" | "production" | "test"; PORT: number; DATABASE_URL: string; REDIS_URL?: string; API_KEY: string; LOG_LEVEL: "debug" | "info" | "warn" | "error" }
TypeScript · example 2 of 3 — createEnv() with compile-time + runtime checks
import { z, ZodType } from "zod";

// Generic env creator — reusable across projects
function createEnv<T extends Record<string, ZodType>>(
  schema: T,
  source: Record<string, string | undefined> = process.env
): z.infer<z.ZodObject<T>> {
  const zodSchema = z.object(schema);
  const result = zodSchema.safeParse(source);

  if (!result.success) {
    const errors = result.error.flatten().fieldErrors;
    const missing = Object.entries(errors)
      .map(([key, msgs]) => `  ${key}: ${msgs?.join(", ")}`)
      .join("\n");

    throw new Error(`Environment validation failed:\n${missing}`);
  }

  // Freeze to prevent accidental mutation
  return Object.freeze(result.data);
}

// Usage: define your env once at the entry point
export const env = createEnv({
  DATABASE_URL: z.string().url(),
  PORT: z.coerce.number().default(3000),
  JWT_SECRET: z.string().min(32, "JWT_SECRET must be at least 32 chars"),
  ALLOWED_ORIGINS: z.string().transform((s) => s.split(",")),
});

// env.ALLOWED_ORIGINS is string[] — parsed from comma-separated input
// env.PORT is number — coerced from string
// env.JWT_SECRET is string — guaranteed >= 32 chars
TypeScript · example 3 of 3 — t3-env pattern (server vs client separation)
import { z } from "zod";

// Separate server-only and client-safe env vars
// This prevents accidentally leaking secrets to the browser bundle

interface EnvConfig<
  TServer extends Record<string, z.ZodType>,
  TClient extends Record<string, z.ZodType>,
> {
  server: TServer;
  client: TClient;
  runtimeEnv: Record<string, string | undefined>;
}

function createT3Env<
  TServer extends Record<string, z.ZodType>,
  TClient extends Record<string, z.ZodType>,
>(config: EnvConfig<TServer, TClient>) {
  const serverSchema = z.object(config.server);
  const clientSchema = z.object(config.client);

  // Client keys must start with NEXT_PUBLIC_ (convention)
  for (const key of Object.keys(config.client)) {
    if (!key.startsWith("NEXT_PUBLIC_")) {
      throw new Error(`Client env var "${key}" must start with NEXT_PUBLIC_`);
    }
  }

  const merged = { ...config.server, ...config.client };
  const fullSchema = z.object(merged);
  const parsed = fullSchema.parse(config.runtimeEnv);

  return parsed as z.infer<typeof serverSchema> & z.infer<typeof clientSchema>;
}

// Usage in a Next.js project
export const env = createT3Env({
  server: {
    DATABASE_URL: z.string().url(),
    JWT_SECRET: z.string().min(32),
    STRIPE_SECRET_KEY: z.string().startsWith("sk_"),
  },
  client: {
    NEXT_PUBLIC_APP_URL: z.string().url(),
    NEXT_PUBLIC_STRIPE_KEY: z.string().startsWith("pk_"),
  },
  runtimeEnv: process.env,
});

// Server code: env.DATABASE_URL, env.STRIPE_SECRET_KEY
// Client code: env.NEXT_PUBLIC_APP_URL, env.NEXT_PUBLIC_STRIPE_KEY
// Accessing server vars in client bundle → undefined (bundler strips them)
Common Mistakes:
❌ Using non-null assertion: process.env.PORT!. This silences the compiler but gives you a false sense of safety. If PORT is actually missing, you get a silent undefined that crashes later.
✅ Validate with Zod at app startup. If PORT is missing, the app fails immediately with a clear error message listing exactly which variables are missing or invalid.
❌ Augmenting ProcessEnv in a global .d.ts: declare namespace NodeJS { interface ProcessEnv { PORT: string } }. This lies to the compiler — the variable might still be missing at runtime.
✅ Use runtime validation (Zod) to guarantee the values actually exist, not just type declarations that assume they do.
Key Rules:
  • Validate all env vars at startup. Fail fast with a clear error listing every missing or invalid variable.
  • z.coerce.number() converts the string to a number — critical since process.env values are always strings.
  • Use .transform() for complex parsing (comma-separated lists, JSON strings, boolean flags).
  • Freeze the validated env object to prevent accidental mutation during the application lifecycle.
  • Separate server and client env vars in frontend frameworks to prevent secret leakage.
