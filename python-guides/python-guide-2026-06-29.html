Review Section (6 items)

1. Abstract Base Classes & Protocol 11 days overdue rep 2 → 3

An ABC (abc.ABC + @abstractmethod) defines an interface subclasses must implement — instantiating an incomplete subclass raises TypeError. A Protocol (from typing) gives structural "duck" typing: any class with the right methods matches, no inheritance required.

Key Pattern
from abc import ABC, abstractmethod
from typing import Protocol

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s * self.s

print(Square(3).area())     # 9

class Sized(Protocol):           # structural: no inheritance needed
    def __len__(self) -> int: ...
def total(x: Sized) -> int: return len(x)
print(total([1, 2, 3]))       # 3  -- a list already satisfies Sized
Challenge

Define an abstract Repository with abstract save(item) and get(id). Prove that instantiating a subclass that forgets get raises TypeError. Then define a Protocol Comparable with __lt__, write a function typed against it, and pass both an int and a custom class — showing no inheritance is needed.

Pitfall

A class with an @abstractmethod can still be subclassed, but any subclass that leaves a method abstract cannot be instantiated (TypeError at construction, not at definition). A Protocol matches by structure — inheriting from it is optional and mainly helps tooling.

2. Memory Optimization 11 days overdue rep 2 → 3

The big memory wins: __slots__ (drop the per-instance __dict__), generators (stream instead of materialize), and array/bytes for homogeneous data. Measure before optimizing with sys.getsizeof and tracemalloc.

Key Pattern
import sys

class Dense:
    __slots__ = ("x", "y")          # no per-instance __dict__
    def __init__(self, x, y): self.x, self.y = x, y

# a generator streams; a list materializes everything at once
squares = (i * i for i in range(1_000_000))   # ~constant memory
print(sys.getsizeof(squares) < 1000)             # True  -- tiny, holds no data yet
print(sum(squares) > 0)                        # True  -- consumed lazily
Challenge

Create two classes holding the same two attributes, one with __slots__ and one without. Compare sys.getsizeof of the instances and explain the difference. Then rewrite a function that builds a 10-million-element list so it uses a generator instead, and argue why peak memory drops.

Pitfall

__slots__ blocks new attributes and removes __dict__ — but a subclass without its own __slots__ silently regains a __dict__, erasing the savings. A generator is single-use: once exhausted it yields nothing, so you cannot iterate it twice.

3. Mocking with unittest.mock 10 days overdue rep 2 → 3

Mock/MagicMock stand in for real objects, recording calls and returning canned values. patch() temporarily replaces a name where it is looked up (not where it is defined) for the duration of a test.

Key Pattern
from unittest.mock import MagicMock, patch

m = MagicMock(return_value=42)
print(m(1, 2))                # 42
m.assert_called_once_with(1, 2)   # raises if not called exactly so

# patch the name where it is USED
with patch("os.getcwd", return_value="/fake"):
    import os
    print(os.getcwd())        # /fake
Challenge

Use patch as a decorator to replace a function that makes a "network call" so it returns a fixed payload, then assert your code called it exactly once with the expected URL. Bonus: set side_effect to a list to return different values on successive calls, and to an exception instance to simulate a failure.

Pitfall

Patch the name in the module that uses it (patch("mymod.requests.get")), not where it is defined — patching the wrong location leaves the real object in place and the test silently passes. Also remember to assert on the calls; a bare Mock accepts any call without complaint.

4. Dunder Methods — __iter__, __next__, __call__ 8 days overdue rep 2 → 3

__iter__ returns an iterator; __next__ produces the next value and raises StopIteration when done — together they make a class usable in a for loop. __call__ makes an instance callable like a function.

Key Pattern
class Countdown:
    def __init__(self, n): self.n = n
    def __iter__(self): return self        # I am my own iterator
    def __next__(self):
        if self.n <= 0: raise StopIteration
        self.n -= 1
        return self.n + 1

print(list(Countdown(3)))    # [3, 2, 1]

class Adder:
    def __init__(self, base): self.base = base
    def __call__(self, x): return self.base + x
add5 = Adder(5)
print(add5(10))            # 15  -- instance called like a function
Challenge

Build a Repeater(value, times) that is iterable and yields value exactly times times using __iter__/__next__ (no generators). Separately, write a Multiplier class whose instances are callable and multiply their argument by a stored factor — then use one as the key= function in sorted().

Pitfall

If __iter__ returns self, the object is a one-shot iterator — a second for loop over it yields nothing. To allow repeated iteration, return a fresh iterator each time (often via a yield-based __iter__). Forgetting to raise StopIteration loops forever.

5. Descriptors 8 days overdue rep 2 → 3

A descriptor is a class implementing __get__/__set__/__delete__. Placed as a class attribute, it intercepts attribute access on instances. This is the machinery behind property, classmethod, and ORM fields.

Key Pattern
class Positive:
    def __set_name__(self, owner, name): self.key = "_" + name
    def __get__(self, obj, owner=None):
        if obj is None: return self
        return getattr(obj, self.key)
    def __set__(self, obj, value):
        if value < 0: raise ValueError("must be >= 0")
        setattr(obj, self.key, value)

class Account:
    balance = Positive()               # descriptor lives on the CLASS
    def __init__(self, b): self.balance = b

a = Account(100)
print(a.balance)                       # 100
# Account(-5) would raise ValueError: must be >= 0
Challenge

Write a Typed descriptor that enforces a given type on assignment (raising TypeError otherwise) and uses __set_name__ to auto-derive its storage key. Attach two of them to a class and confirm valid assignments work while wrong-typed ones raise. Explain why the descriptor must live on the class, not the instance.

Pitfall

A descriptor only fires when it lives on the class — assigning the descriptor to self in __init__ makes it an ordinary attribute with no interception. A data descriptor (defines __set__) takes precedence over the instance __dict__; a non-data one (only __get__) does not.

6. Exception Hierarchy & Custom Exceptions 8 days overdue rep 2 → 3

Every exception derives from BaseException; catch Exception (not BaseException, which includes KeyboardInterrupt/SystemExit). An except clause matches the class and its subclasses, checked top to bottom — so order specific-before-general. Custom exceptions subclass Exception.

Key Pattern
class AppError(Exception): ...
class NotFound(AppError): ...          # a specific subtype

try:
    raise NotFound("missing user 7")
except NotFound as e:                  # specific FIRST
    print("not found:", e)
except AppError:                        # would also match, too general here
    print("app error")

print(issubclass(NotFound, Exception))  # True
Challenge

Design a small exception tree: a base ConfigError with subclasses MissingKey and BadValue. Write a loader that raises the right subtype, and a caller that handles MissingKey specifically but lets other ConfigErrors propagate. Add a raise ... from err to chain an underlying KeyError.

Pitfall

Ordering except Exception before a specific subclass makes the specific handler dead code — Python takes the first matching clause. Catching BaseException (or a bare except:) swallows KeyboardInterrupt and SystemExit, making the program impossible to interrupt.

New Concepts — Numbers & Randomness (2 concepts)

Two standard-library modules that keep numeric code correct and fast: math (with cmath) for C-accelerated functions, exact integer helpers, and safe float comparison; and random for reproducible pseudo-randomness — sampling, shuffling, and simulation.

Numbers

Concept 1: math & cmath — Numeric Functions Done Right

Why learn this? The math module gives you C-accelerated, numerically careful functions — square roots, logs, trig, combinatorics, and float-tolerant comparison — that are faster and more correct than hand-rolled versions. cmath mirrors them for complex numbers, and integer helpers like isqrt/comb stay exact.
Example 1 — core functions
import math

print(math.sqrt(16))                 # 4.0  -- always returns a float
print(math.floor(3.7), math.ceil(3.2))  # 3 4  -- these return ints
print(math.gcd(12, 18))              # 6
print(math.factorial(5))             # 120
Output
4.0 3 4 6 120
Example 2 — constants, logs & distance
import math

print(math.pi, math.e)          # 3.141592653589793 2.718281828459045
print(math.log(math.e))         # 1.0    -- natural log (base e)
print(math.log10(1000))         # 3.0    -- dedicated, more accurate than log(x, 10)
print(round(math.dist([0, 0], [3, 4]), 1))  # 5.0  -- euclidean distance
Output
3.141592653589793 2.718281828459045 1.0 3.0 5.0
Example 3 — isclose, inf & nan
import math

print(0.1 + 0.2 == 0.3)               # False  -- binary float rounding
print(math.isclose(0.1 + 0.2, 0.3))   # True   -- tolerant comparison
print(math.inf > 10**100)             # True   -- positive infinity
print(math.isnan(float("nan")))         # True
print(math.nan == math.nan)             # False  -- NaN never equals itself
Output
False True True True False
Example 4 — exact integer helpers & cmath
import math, cmath

print(math.isqrt(17))        # 4   -- integer square root, no float rounding
print(math.comb(5, 2))       # 10  -- combinations (n choose k)
print(math.perm(5, 2))       # 20  -- permutations
print(cmath.sqrt(-1))        # 1j  -- math.sqrt(-1) would raise ValueError
Output
4 10 20 1j

Common Mistakes

✗ Comparing floats with == (or feeding a negative number to math.sqrt, which raises ValueError)

✓ Use math.isclose(a, b) for float equality; use cmath.sqrt when a complex result is acceptable


✗ Hand-rolling x ** 0.5, factorial loops, or gcd by hand for integer math

✓ Reach for math.isqrt, math.factorial, math.comb, math.gcd — C-fast and exact for ints

Key Rules

  • math works on floats and generally returns floats (but floor/ceil/isqrt return ints); it raises on domain errors like sqrt(-1)
  • Prefer math.isclose(a, b) over == for floats; NaN is never equal to anything, so test with math.isnan
  • Integer-exact helpers: isqrt, gcd, lcm, comb, perm, factorial; cmath mirrors math for complex numbers

Exercises

  1. Without using ==, test whether two floats are within tolerance using math.isclose, and explain in a comment what rel_tol controls.
  2. Compute the number of distinct 5-card hands from a 52-card deck with math.comb and confirm it equals 2598960.
  3. Given a list of (x, y) points, find the pair that is closest together using math.dist.
Connections: math.isclose echoes the precision theme of decimal/fractions; math.comb/perm relate to itertools.combinations/permutations; statistics builds on these primitives; inf/nan surface as edge cases in sorting and JSON.
Randomness

Concept 2: The random Module — Reproducible Pseudo-Randomness

Why learn this? random generates pseudo-random numbers for simulations, sampling, shuffling, and test data. Seeding makes a run reproducible, which is essential for debuggable tests. One caveat drives everything: random is not cryptographically secure — use secrets for anything sensitive.
Example 1 — seeding for reproducibility
import random

random.seed(42)
a = [random.randint(1, 100) for _ in range(3)]
random.seed(42)                         # same seed => same sequence
b = [random.randint(1, 100) for _ in range(3)]
print(a == b)      # True  -- fully reproducible
print(len(a))       # 3
Output
True 3
Example 2 — picking & sampling
import random

random.seed(0)
colors = ["red", "green", "blue", "yellow"]
print(random.choice(colors) in colors)         # True  -- one element
print(len(random.sample(colors, 2)))            # 2  -- k UNIQUE elements (no repeats)
print(len(set(random.sample(range(10), 3))))   # 3  -- guaranteed distinct
print(len(random.choices(colors, k=5)))         # 5  -- WITH replacement
Output
True 2 3 5
Example 3 — shuffle in place & weighted choices
import random

random.seed(1)
deck = list(range(5))
random.shuffle(deck)                    # shuffles IN PLACE, returns None
print(sorted(deck) == [0, 1, 2, 3, 4])  # True  -- same elements, new order

# 'a' is ten times likelier than 'b'
picks = random.choices(["a", "b"], weights=[10, 1], k=1000)
print(picks.count("a") > picks.count("b"))  # True
Output
True True
Example 4 — distributions & the shuffle gotcha
import random

random.seed(7)
print(0.0 <= random.random() < 1.0)   # True  -- half-open interval [0.0, 1.0)
g = random.gauss(mu=0, sigma=1)          # a sample from the normal distribution
print(isinstance(g, float))            # True
print(random.shuffle([3, 1, 2]))        # None  -- mutates in place, returns nothing
Output
True True None

Common Mistakes

✗ Using random for passwords, tokens, or session ids — it is predictable once the internal state is known

✓ Use the secrets module (secrets.token_hex, secrets.choice) for anything security-sensitive


✗ Writing new = random.shuffle(lst) — shuffle mutates in place and returns None

✓ Call random.shuffle(lst) for its side effect, or random.sample(lst, len(lst)) for a shuffled copy

Key Rules

  • random.seed(n) makes the sequence reproducible: same seed ⇒ same numbers (great for tests and simulations)
  • randint(a, b) is inclusive of both ends; random() is [0.0, 1.0); sample() draws unique items, choices() draws with replacement (and accepts weights)
  • random is NOT cryptographically secure — use secrets for tokens/passwords

Exercises

  1. Seed the RNG, generate a list of dice rolls, reseed with the same value, and confirm you get the identical list.
  2. Model a loaded die where 6 is three times as likely as any other face using random.choices with weights; roll it 6000 times and print the face counts.
  3. Write shuffled(seq) that returns a NEW shuffled list without mutating the original (hint: random.sample).
Connections: contrasts with secrets (crypto-secure, from the security topic); random.sample/choices echo itertools combinatorics; seeding parallels reproducible pytest runs; gauss/uniform feed the statistics module.

New Concepts — Runtime, Timing & Diagnostics (3 concepts)

Three modules for understanding the running program and its environment: time for choosing the right clock (timestamps vs. benchmarking vs. timeouts), warnings for non-fatal advisories like deprecations, and platform for reporting OS and interpreter facts in logs and bug reports.

Timing

Concept 3: The time Module — perf_counter, monotonic & sleep

Why learn this? time gives you wall-clock timestamps, monotonic clocks, sleeping, and high-resolution performance timers. Choosing the right clock matters: perf_counter for benchmarking, monotonic for timeouts, and time() only for real-world timestamps.
Example 1 — wall-clock timestamp
import time

t = time.time()                  # seconds since the Unix epoch (a float)
print(t > 1_600_000_000)         # True  -- sometime after Sept 2020
print(isinstance(t, float))      # True
print(time.gmtime(0).tm_year)     # 1970  -- break the epoch into UTC fields
Output
True True 1970
Example 2 — perf_counter for benchmarking
import time

start = time.perf_counter()       # high-res clock, for measuring DURATIONS
total = sum(range(1_000_000))
elapsed = time.perf_counter() - start
print(elapsed >= 0)               # True
print(isinstance(elapsed, float)) # True  -- fractional seconds
Output
True True
Example 3 — sleep & monotonic
import time

t0 = time.monotonic()             # never goes backwards (immune to clock changes)
time.sleep(0.01)                  # pause ~10ms (releases the GIL)
dt = time.monotonic() - t0
print(dt >= 0.01)                 # True  -- at least the slept duration
Output
True
Example 4 — nanosecond timers & formatting
import time

print(isinstance(time.perf_counter_ns(), int))  # True  -- ns, avoids float rounding
print(time.strftime("%Y-%m-%d", time.gmtime(0)))  # 1970-01-01
print(time.perf_counter_ns() > 0)                # True
Output
True 1970-01-01 True

Common Mistakes

✗ Using time.time() to measure elapsed time — the wall clock can jump (NTP sync, DST) and even move backwards

✓ Use time.perf_counter() for durations/benchmarks and time.monotonic() for timeouts


✗ Calling time.sleep() inside async code — it blocks the whole event loop

✓ In async code use await asyncio.sleep(); time.sleep is only for synchronous pauses

Key Rules

  • time.time() = wall-clock seconds since the epoch (for timestamps); it can jump — never use it to measure durations
  • time.perf_counter() = highest-resolution timer for benchmarking; time.monotonic() never goes backwards, ideal for timeouts
  • time.sleep(seconds) pauses the current thread and releases the GIL; the *_ns variants return ints to avoid float precision loss

Exercises

  1. Time how long it takes to build a list of one million squares using time.perf_counter, and print the elapsed seconds.
  2. Explain in a comment why time.time() is the wrong clock for a stopwatch, and which clock you would use instead.
  3. Use time.gmtime and time.strftime to print the Unix epoch (0) as an ISO-style date string.
Connections: perf_counter underlies timeit (from the profiling topic); monotonic backs timeout logic in threading/asyncio; time.sleep contrasts with asyncio.sleep; struct_time overlaps with datetime and zoneinfo.
Diagnostics

Concept 4: The warnings Module — Non-Fatal Advisories

Why learn this? Warnings signal non-fatal problems — deprecations, risky usage, fallbacks — without stopping the program. Libraries use them to guide callers; you can filter, silence, escalate them into errors, or capture them in tests to assert they fired.
Example 1 — emitting a warning
import warnings

def old_api():
    warnings.warn("use new_api instead", DeprecationWarning, stacklevel=2)
    return 42

print(old_api())     # 42  -- program keeps running; the warning goes to stderr
Output
42
Example 2 — filtering: silence and escalate
import warnings

with warnings.catch_warnings():
    warnings.simplefilter("ignore")   # silence everything in this block
    warnings.warn("hidden")            # produces no output
print("after block")

with warnings.catch_warnings():
    warnings.simplefilter("error")    # turn warnings INTO exceptions
    try:
        warnings.warn("boom")
    except UserWarning as e:
        print("caught:", e)          # caught: boom
Output
after block caught: boom
Example 3 — capturing warnings (for tests)
import warnings

with warnings.catch_warnings(record=True) as caught:
    warnings.simplefilter("always")
    warnings.warn("first", UserWarning)
    warnings.warn("second", DeprecationWarning)
print(len(caught))                  # 2
print(caught[0].category.__name__)   # UserWarning
print(str(caught[1].message))       # second
Output
2 UserWarning second
Example 4 — the default "once per location" behavior
import warnings

with warnings.catch_warnings():
    warnings.simplefilter("default")   # show each unique warning ONCE
    for _ in range(3):
        warnings.warn("repeated")       # printed only the first time
print("done")
Output
done

Common Mistakes

✗ Using warnings.warn for genuine errors that should stop execution

✓ raise an exception for errors; reserve warnings for advisory, recoverable conditions (deprecations, fallbacks)


✗ Leaving stacklevel=1, so the warning points at your library's internals instead of the caller

✓ Pass stacklevel=2 (or higher) so the warning blames the user's call site — the line they can actually fix

Key Rules

  • warnings.warn(msg, Category, stacklevel=n) emits a non-fatal warning; by default each unique warning shows once per source location
  • Control behavior with filters: "ignore", "error" (escalate to exception), "always", "default" — scope them with warnings.catch_warnings()
  • DeprecationWarning is hidden by default outside __main__/tests; use catch_warnings(record=True) to assert on warnings in tests

Exercises

  1. Write a deprecated function that warns with DeprecationWarning and stacklevel=2, then call it and observe the warning names the caller's line.
  2. Inside a catch_warnings block, escalate warnings to errors and prove that warnings.warn now raises.
  3. Capture warnings with record=True and assert that exactly one UserWarning with the expected message was produced.
Connections: DeprecationWarning ties into the exception hierarchy reviewed today (Warning subclasses Exception); catch_warnings is a context manager (the contextlib theme); pytest.warns wraps this exact pattern; escalating to errors pairs with logging for diagnostics.
Diagnostics

Concept 5: The platform Module — Interrogating the Environment

Why learn this? platform reports facts about the machine and interpreter — OS name and version, CPU architecture, and the Python implementation — for diagnostics, bug-report footers, and the occasional bit of OS-specific branching.
Example 1 — OS and machine
import platform

print(platform.system() in {"Linux", "Darwin", "Windows"})  # True
print(isinstance(platform.release(), str))   # True  -- OS release / kernel
print(isinstance(platform.machine(), str))   # True  -- e.g. 'x86_64', 'arm64'
Output
True True True
Example 2 — Python interpreter facts
import platform

print(platform.python_version())         # 3.12.3  -- a STRING
print(platform.python_version_tuple())   # ('3', '12', '3')
print(platform.python_implementation())  # CPython
Output
3.12.3 ('3', '12', '3') CPython
Example 3 — one-line summaries
import platform

s = platform.platform()          # single human-readable string
print(isinstance(s, str))        # True  -- e.g. 'macOS-14.5-arm64-arm-64bit'
u = platform.uname()             # named tuple of system facts
print(u.system == platform.system())  # True  -- fields: system, node, release, ...
Output
True True
Example 4 — portable branching & word size
import platform

if platform.system() == "Windows":
    clear_cmd = "cls"
else:
    clear_cmd = "clear"
print(clear_cmd in {"cls", "clear"})                   # True
print(platform.architecture()[0] in {"64bit", "32bit"})  # True
Output
True True

Common Mistakes

✗ Parsing sys.version (a free-form banner string) to detect the Python version

✓ Use platform.python_version()/python_version_tuple() or sys.version_info for reliable, structured checks


✗ Branching on platform.system() when a portable API already covers the case

✓ Prefer cross-platform modules (pathlib, os, shutil); reserve platform checks for genuinely OS-specific behavior

Key Rules

  • platform.system() returns 'Linux'/'Darwin'/'Windows'; machine()/architecture() report CPU and word size
  • Interpreter facts: python_version() (string), python_version_tuple(), python_implementation() ('CPython'/'PyPy')
  • platform.platform()/uname() give a full summary — ideal for logs and bug reports; use them for diagnostics, not core logic

Exercises

  1. Print a one-line environment banner combining OS, release, and Python version — the kind you would attach to a bug report.
  2. Choose a screen-clear command based on platform.system(), then explain in a comment why os/shutil might be a cleaner approach.
  3. Compare platform.python_version_tuple() against ('3', '10') to gate a feature that needs Python 3.10+.
Connections: overlaps with sys (sys.platform, sys.version_info — Day 28) and os (os.name); interpreter-implementation checks matter for CPython-specific topics like the GIL and reference counting (reviewed CPython internals); pairs naturally with logging for bug-report context.
