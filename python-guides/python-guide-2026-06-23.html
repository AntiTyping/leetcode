Review Section (6 items)

1. Dataclasses 10 days overdue rep 2

@dataclass auto-generates __init__, __repr__, and __eq__. Use field() for mutable defaults and __post_init__ for derived/validated fields.

Key Pattern
from dataclasses import dataclass, field

@dataclass(frozen=True)
class Config:
    name: str
    tags: list = field(default_factory=list)

    def __post_init__(self):
        if not self.name:
            raise ValueError("name required")
Challenge

Create a frozen dataclass Point3D with x, y, z floats and a method distance_to(other) that computes the Euclidean distance to another point.

Pitfall

Mutable default args — use field(default_factory=list), NOT field(default=[]) (which shares one list across all instances).

2. Type Hints — Optional, Union & Modern Syntax 8 days overdue rep 2

Python 3.10+ uses X | Y instead of Union[X, Y], and X | None instead of Optional[X]. No import needed for the pipe syntax.

Key Pattern
def greet(name: str | None = None) -> str:
    return f"Hi, {name or 'stranger'}"

def parse(x: int | float | str) -> float:
    return float(x)
Challenge

Type-annotate a function max_key(d) that takes a dict mapping str to int | float and returns the key (a str) with the maximum value.

Pitfall

Optional[X] means X | None, NOT "this parameter is optional." A parameter is optional when it has a default value — the two concepts are unrelated.

3. Classes — __eq__ and __hash__ 8 days overdue rep 2

If you define __eq__, Python sets __hash__ to None (instances become unhashable). To use objects as dict keys or in sets, define both.

Key Pattern
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __eq__(self, other):
        return (self.x, self.y) == (other.x, other.y)

    def __hash__(self):
        return hash((self.x, self.y))
Challenge

Create a Color class with r, g, b attributes that is usable as a dict key, with proper equality (two colors with the same channels are equal and collide in a set).

Pitfall

Defining __eq__ without __hash__ silently makes instances unhashable — you only discover it when adding them to a set or dict raises TypeError.

4. __slots__ 6 days overdue rep 2

__slots__ = ('x', 'y') prevents the per-instance __dict__, saving roughly 40% memory per instance and speeding up attribute access.

Key Pattern
class Point:
    __slots__ = ('x', 'y')

    def __init__(self, x, y):
        self.x, self.y = x, y

p = Point(1, 2)
# p.z = 3  -> AttributeError (no __dict__)
Challenge

Build a million-element list of Point objects with and without __slots__. Compare memory usage (use tracemalloc or sys.getsizeof on instances).

Pitfall

Slots + inheritance — a child class must declare its OWN new slots; it should not repeat the parent's slots, and any non-slotted base reintroduces __dict__.

5. Properties & @property 6 days overdue rep 2

@property creates a getter; @name.setter creates a setter. Use them for computed attributes and validation that looks like plain attribute access.

Key Pattern
class Temperature:
    def __init__(self, c):
        self._c = c

    @property
    def celsius(self):
        return self._c

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("below absolute zero")
        self._c = value
Challenge

Create a Circle class where setting radius auto-updates area and setting area auto-updates radius (each derived from the other via @property).

Pitfall

Infinite recursion — using self.x = value inside the setter for x re-invokes the same setter. Store in a different backing attribute like self._x.

6. Dunder Methods — __getitem__, __len__, __contains__ 5 days overdue rep 2

These make your class work with [], len(), and in. Implementing __getitem__ alone even makes an object iterable (Python falls back to integer indexing from 0).

Key Pattern
class Deck:
    def __init__(self, cards):
        self._cards = cards

    def __len__(self):
        return len(self._cards)

    def __getitem__(self, key):
        return self._cards[key]  # supports int AND slice

    def __contains__(self, item):
        return item in self._cards
Challenge

Build a Matrix class that supports m[row, col] (tuple key), len(m) (number of rows), and value in m (searches all cells).

Pitfall

Not handling slice objects in __getitem__ — obj[1:3] passes a slice, not an int. Delegating to a built-in list/tuple handles both for free.

New Concepts — CPython Internals (3 concepts)

Understanding how Python works under the hood helps you write faster, more predictable code.

CPython Internals

Concept 1: Reference Counting & Garbage Collection

Why learn this? Understanding memory management prevents leaks and explains performance characteristics. CPython frees most objects the instant their reference count hits zero — but reference cycles need the cyclic garbage collector, and knowing the difference tells you when a leak is possible.
Example 1 — Watching the reference count
import sys

x = [1, 2, 3]
print(sys.getrefcount(x))  # 2: x + the temp arg to getrefcount

y = x                     # another reference
print(sys.getrefcount(x))  # 3

del y
print(sys.getrefcount(x))  # 2 again
Output
2
3
2
Example 2 — Creating a reference cycle
class Node:
    def __init__(self):
        self.ref = None

a = Node()
b = Node()
a.ref = b   # a -> b
b.ref = a   # b -> a  (cycle!)

del a, b    # refcounts never hit 0 -> not freed yet
Example 3 — gc.collect() and gc.get_referrers()
import gc

gc.disable()              # turn off automatic cyclic GC
# ... create cycles as above ...
unreachable = gc.collect()  # manually run cyclic collector
print(f"collected {unreachable} objects")

obj = [1, 2]
container = {"key": obj}
print(gc.get_referrers(obj))  # shows the dict referencing obj
Example 4 — weakref to break cycles
import weakref

class Parent:
    def __init__(self):
        self.child = None

class Child:
    def __init__(self, parent):
        self.parent = weakref.ref(parent)  # weak: no cycle

p = Parent()
p.child = Child(p)
print(p.child.parent())  # call the weakref to deref -> the Parent

Common Mistakes

✗ Relying on __del__ for cleanup — with reference cycles, __del__ may run late or (historically) not at all

✓ Use context managers (with) or weakref — deterministic, cycle-proof cleanup


✗ Assuming sys.getrefcount(x) equals "how many of my variables point at x" (it counts the temporary argument too)

✓ Subtract 1 to account for the argument passed to getrefcount itself

Key Rules

  • Refcount hitting 0 frees memory immediately — deterministic, no pause
  • The cyclic gc only handles objects that form reference cycles
  • gc.collect() forces a cycle sweep and returns the count of unreachable objects collected

Exercises

  1. Show reference count changes as you create and delete variables: print sys.getrefcount of an object before assigning an alias, after, and after del of the alias. Explain each number.
  2. Create a reference cycle between two objects, del your names for them, then call gc.collect() and verify it reports them as collected (use gc.disable() first so the cycle survives until you collect).
Connections: Explains why __slots__ (Review #4) saves memory by shrinking each object. weakref pairs with caching and observer patterns. Sets up the GIL discussion (Concept 3) — the GIL also protects refcount updates from data races.
CPython Internals

Concept 2: String Interning & Integer Caching

Why learn this? This explains why is sometimes appears to work for comparing strings or small integers — and why you must never rely on it. Interning and small-int caching are CPython optimizations, not language guarantees.
Example 1 — Small integers are cached (-5 to 256)
a = 256
b = 256
print(a is b)   # True  -> same cached object
print(a == b)   # True  -> correct way to compare
Output
True
True
Example 2 — Above 256, the cache no longer applies
a = 257
b = 257
print(a is b)   # usually False -> two distinct objects
print(a == b)   # True -> values are equal
Output
False
True
Example 3 — Explicit interning with sys.intern()
import sys

s1 = sys.intern("".join(["hel", "lo"]))
s2 = sys.intern("".join(["hel", "lo"]))
print(s1 is s2)  # True -> both point at the interned copy

# Useful for huge sets of repeated strings (tokens, keys):
# interning lets dict lookups compare by identity first.
Example 4 — The peephole optimizer interns literals
a = "admin"
b = "admin"
print(a is b)         # True -> identical literals interned at compile time

c = "ad" + input()  # runtime-built string
# c is NOT guaranteed to be interned -> do not use `is`

Common Mistakes

✗ if name is "admin": — works by accident with literals, fails for runtime-built strings

✓ if name == "admin": — compares values, always correct


✗ Treating is behavior on ints/strings as a guarantee you can depend on

✓ Use is only for singletons (None, True, False) and identity checks

Key Rules

  • is checks identity (same object); == checks equality (same value)
  • CPython caches small ints -5 through 256 and interns many string literals
  • Interning is an optimization, not a contract — never write logic that depends on it

Exercises

  1. Predict the output of is comparisons for these pairs, then run them: (256, 256), (257, 257), (-5, -5), (-6, -6), ("hi", "hi"), and two runtime-built equal strings. Explain each result.
  2. Use id() to trace object identity as you reassign a variable through several values, showing when the id changes and when it reuses a cached object.
Connections: Builds on reference counting (Concept 1) — cached ints have permanently high refcounts. Reinforces the __eq__ vs identity distinction from Review #3. The is-vs-== rule is the most common real-world bug this prevents.

Concept 3: The GIL in Practice

Why learn this? The Global Interpreter Lock shapes the performance of multi-threaded Python. Knowing when it matters (CPU-bound work) and when it does not (I/O-bound work) lets you pick threading, multiprocessing, or asyncio correctly — instead of adding threads that produce no speedup.
Example 1 — CPU-bound threading: no speedup
import threading, time

def crunch(n):
    total = 0
    for i in range(n):
        total += i * i
    return total

start = time.perf_counter()
threads = [threading.Thread(target=crunch, args=(10_000_000,))
           for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"4 threads: {time.perf_counter()-start:.2f}s")
# ~ same as running them one after another: GIL serializes bytecode
Example 2 — I/O-bound threading: real speedup
import threading, time

def io_task():
    time.sleep(1)  # GIL is RELEASED during sleep/blocking I/O

start = time.perf_counter()
threads = [threading.Thread(target=io_task) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"4 threads: {time.perf_counter()-start:.2f}s")
# ~1s total, not 4s: threads overlap their waiting
Output
4 threads: 1.00s
Example 3 — multiprocessing for CPU-bound parallelism
from multiprocessing import Pool

def crunch(n):
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    with Pool(4) as pool:
        results = pool.map(crunch, [10_000_000] * 4)
    # each process has its OWN GIL -> true parallelism on 4 cores
Example 4 — concurrent.futures comparison
from concurrent.futures import (
    ThreadPoolExecutor, ProcessPoolExecutor)

# I/O-bound? -> threads
with ThreadPoolExecutor(max_workers=8) as ex:
    list(ex.map(io_task, range(8)))

# CPU-bound? -> processes
with ProcessPoolExecutor(max_workers=4) as ex:
    list(ex.map(crunch, [10_000_000] * 4))

Common Mistakes

✗ Using threading for CPU-bound work expecting a speedup — the GIL serializes bytecode, so you get none

✓ Use multiprocessing for CPU-bound work and threading/asyncio for I/O-bound


✗ Believing the GIL makes shared state automatically safe — it does not protect compound operations like x += 1

✓ Still use Lock for shared mutable state across threads

Key Rules

  • The GIL lets only one thread execute Python bytecode at a time
  • It is released during blocking I/O and many C-extension calls — that is why I/O threading works
  • For CPU parallelism use processes (or release the GIL in a C extension)

Exercises

  1. Time a CPU-bound task (e.g. summing squares to 10 million) with 1 thread vs 4 threads. Explain why the wall-clock time barely changes.
  2. Time an I/O-bound task (a function that time.sleep(1)) with 1 thread vs 4 threads. Explain why 4 threads finish in roughly 1 second instead of 4.
Connections: The GIL exists partly to keep refcounting (Concept 1) thread-safe. I/O threading motivates asyncio's single-threaded concurrency. concurrent.futures gives a unified API over both threads and processes.

Advanced Standard Library

Concept 4: abc — Abstract Base Classes in Practice

Why learn this? ABCs enforce interface contracts at instantiation time, not when a missing method is finally called. That early failure is essential for plugin systems and frameworks — you find out a class is incomplete the moment someone tries to use it.
Example 1 — Basic ABC with abstract methods
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...

    @abstractmethod
    def perimeter(self) -> float: ...

# Shape()  -> TypeError: Can't instantiate abstract class
Example 2 — A concrete implementation
import math

class Circle(Shape):
    def __init__(self, r):
        self.r = r

    def area(self):
        return math.pi * self.r ** 2

    def perimeter(self):
        return 2 * math.pi * self.r

c = Circle(3)  # OK: both abstract methods implemented
Example 3 — register() for virtual subclasses (duck typing)
from abc import ABC

class Drawable(ABC):
    pass

class Sprite:           # does NOT inherit Drawable
    def draw(self): ...

Drawable.register(Sprite)  # declare it a virtual subclass
print(issubclass(Sprite, Drawable))  # True
print(isinstance(Sprite(), Drawable)) # True
Output
True
True
Example 4 — Combining ABC with __init_subclass__
from abc import ABC, abstractmethod

class Handler(ABC):
    registry = {}

    def __init_subclass__(cls, key, **kwargs):
        super().__init_subclass__(**kwargs)
        Handler.registry[key] = cls

    @abstractmethod
    def handle(self, event): ...

class ClickHandler(Handler, key="click"):
    def handle(self, event):
        return f"clicked {event}"

print(Handler.registry)  # {'click': <class 'ClickHandler'>}

Common Mistakes

✗ Forgetting @abstractmethod — the method becomes optional, and incomplete subclasses instantiate silently

✓ Always decorate required methods with @abstractmethod so missing ones fail at instantiation


✗ Using register() expecting it to verify the class actually has the methods (it does not check)

✓ Inherit normally when you want enforcement; use register() only for trusted duck-typed classes

Key Rules

  • ABC + @abstractmethod blocks instantiation until every abstract method is overridden
  • register() creates a virtual subclass for isinstance/issubclass without inheritance or checks
  • __subclasshook__ enables structural ("has these methods") subtyping

Exercises

  1. Create a Shape ABC with abstract area() and perimeter(). Implement Circle and Rectangle, then prove that a subclass missing one method raises TypeError on instantiation.
  2. Create a plugin system using an ABC that auto-registers plugins via __init_subclass__ keyed by a name, then look one up and instantiate it by name.
Connections: __init_subclass__ was introduced in Day 22; here it pairs with ABCs. Contrasts with structural typing via typing.Protocol. The registry pattern echoes the decorator-based registration from earlier guides.

Concept 5: dataclasses — Advanced Patterns

Why learn this? Beyond the basics, dataclasses support inheritance, validation, keyword-only fields, auto-generated slots, and structural pattern matching. These features remove most of the boilerplate from real-world model classes — if you know the field-ordering rules.
Example 1 — Inheritance (parent fields come first)
from dataclasses import dataclass

@dataclass
class Shape:
    name: str

@dataclass
class Rectangle(Shape):
    width: float
    height: float

# Generated __init__: (self, name, width, height)
r = Rectangle("box", 3.0, 4.0)
Example 2 — __post_init__ validation
@dataclass
class Account:
    balance: float

    def __post_init__(self):
        if self.balance < 0:
            raise ValueError("balance must be non-negative")

# Account(-5)  -> ValueError
Example 3 — kw_only=True avoids field-ordering pain (3.10+)
@dataclass(kw_only=True)
class User:
    name: str
    role: str = "member"
    active: bool          # non-default AFTER default is fine here

u = User(name="Alice", active=True)  # must use keywords
Example 4 — slots=True for memory savings (3.10+)
@dataclass(slots=True)
class Point:
    x: float
    y: float

# Auto-generates __slots__ = ('x', 'y') -> no __dict__
p = Point(1.0, 2.0)
# p.z = 3  -> AttributeError
Example 5 — Pattern matching with dataclasses
@dataclass
class Point:
    x: float
    y: float

def describe(p):
    match p:
        case Point(x=0, y=0):
            return "origin"
        case Point(x=0, y=y):
            return f"on Y axis at {y}"
        case Point(x=x, y=y):
            return f"point {x}, {y}"

print(describe(Point(0, 5)))  # on Y axis at 5
Output
on Y axis at 5

Common Mistakes

✗ Putting a non-default field after a default field across inheritance — TypeError: non-default argument follows default argument

✓ Use kw_only=True so ordering no longer matters in the generated __init__


✗ Combining slots=True with a default field that needs __dict__-based caching

✓ Keep slotted dataclasses free of dynamically-added attributes

Key Rules

  • Inheritance: parent fields are placed first in the generated __init__
  • kw_only=True (3.10+) sidesteps the default-vs-non-default ordering rule
  • slots=True (3.10+) auto-generates __slots__; match_args enables positional pattern matching

Exercises

  1. Create a Shape → Rectangle → Square dataclass hierarchy with validation in __post_init__ (e.g. positive dimensions, and a square's width equals its height).
  2. Define a dataclass twice — once with slots=True and once without — create a million instances of each, and benchmark the memory difference.
Connections: Builds directly on Dataclasses (Review #1) and __slots__ (Review #4). Pattern matching pairs with match_args. __post_init__ is the dataclass equivalent of the validation shown in the __init_subclass__ and ABC patterns (Concept 4).
