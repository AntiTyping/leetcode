1. argparse for CLI Tools 11 days overdue rep 1

Create argument parsers with ArgumentParser(), add positional/optional args, subcommands. add_argument('--verbose', action='store_true') for boolean flags.

Key Pattern
import argparse

parser = argparse.ArgumentParser(description='Tool')
parser.add_argument('input', help='Input file')
parser.add_argument('-o', '--output', default='out.txt')
parser.add_argument('-v', '--verbose', action='store_true')
args = parser.parse_args()
Challenge

Build a CLI tool that accepts a filename, an optional --format flag (json/csv/text, default json), and a --count integer. Print the parsed args.

Pitfall

Forgetting that add_argument without a -- prefix makes it positional (required).

2. subprocess Module 11 days overdue rep 0

Run external commands with subprocess.run(). Use capture_output=True for stdout/stderr. check=True raises on non-zero exit.

Key Pattern
import subprocess

result = subprocess.run(
    ['ls', '-la'],
    capture_output=True,
    text=True,
    check=True
)
print(result.stdout)
Challenge

Run git log --oneline -5, capture output, parse each line into (hash, message) tuples.

Pitfall

Using shell=True with user input — command injection vulnerability. Always use a list of args.

3. pytest Fundamentals 10 days overdue rep 1

Test functions named test_*, fixtures with @pytest.fixture, parametrize with @pytest.mark.parametrize.

Key Pattern
import pytest

@pytest.mark.parametrize("input,expected", [
    ("hello", 5), ("", 0), ("ab", 2)
])
def test_length(input, expected):
    assert len(input) == expected
Challenge

Write a pytest fixture that creates a temporary directory with 3 files. Write a parametrized test that checks each file exists.

Pitfall

Forgetting conftest.py for shared fixtures across test files.

4. struct & Binary Data 10 days overdue rep 1

Pack/unpack binary data with struct.pack(fmt, ...) and struct.unpack(fmt, data). Format chars: i=int32, f=float, H=uint16, s=bytes.

Key Pattern
import struct

data = struct.pack('>iHf', 42, 1000, 3.14)  # big-endian
val = struct.unpack('>iHf', data)  # (42, 1000, 3.14...)
Challenge

Define a binary packet format: 4-byte magic number, 2-byte version, 4-byte payload length, then payload bytes. Write pack/unpack functions.

Pitfall

Forgetting byte order prefix (> big-endian, < little-endian, = native).

5. Enum 9 days overdue rep 2

Define named constants with class Color(Enum). Access: Color.RED, Color.RED.value, Color['RED']. Use auto() for auto-numbering.

Key Pattern
from enum import Enum, auto

class Status(Enum):
    PENDING = auto()
    ACTIVE = auto()
    CLOSED = auto()
Challenge

Create a Direction enum (N/S/E/W) with a method opposite() that returns the opposite direction.

Pitfall

Comparing enum members with == works, but is is preferred for identity. Enum members with same value are aliases, not duplicates.

6. Decorators — Function Decorators & @wraps due today rep 3

A decorator is a function that takes a function and returns a modified function. @wraps(func) preserves the original function's metadata.

Key Pattern
from functools import wraps
import time

def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter()-start:.4f}s")
        return result
    return wrapper
Challenge

Write a decorator @retry(max_attempts=3) that retries a function up to N times if it raises an exception, with exponential backoff.

Pitfall

Forgetting @wraps(func) — causes func.__name__ and func.__doc__ to be lost.

New Concepts — Testing Patterns (2 concepts)

Testing

Concept 1: Property-Based Testing with hypothesis

Why learn this? Traditional tests check specific examples you thought of. hypothesis generates hundreds of random inputs to find edge cases you missed — buffer overflows, off-by-ones, Unicode issues. It turns "I think this works" into "this works for every input I could imagine."
Example 1 — Basic: Reversing a list twice returns the original
from hypothesis import given
from hypothesis import strategies as st

@given(st.lists(st.integers()))
def test_reverse_twice(xs):
    assert list(reversed(list(reversed(xs)))) == xs
Output
runs ~100 random lists, all pass
Example 2 — Testing with constraints: sorted output
@given(st.lists(st.integers(), min_size=1))
def test_sort_is_sorted(xs):
    s = sorted(xs)
    assert all(s[i] <= s[i+1] for i in range(len(s)-1))
    assert len(s) == len(xs)  # no elements lost
Example 3 — Custom strategies & settings
from hypothesis import settings

email = st.from_regex(
    r'[a-z]{1,10}@[a-z]{1,5}\.(com|org)',
    fullmatch=True
)

@settings(max_examples=500)
@given(email)
def test_email_parse(addr):
    name, domain = addr.split('@')
    assert len(name) >= 1
    assert '.' in domain
Example 4 — Stateful testing: testing a state machine
from hypothesis.stateful import RuleBasedStateMachine, rule, initialize

class StackMachine(RuleBasedStateMachine):
    def __init__(self):
        super().__init__()
        self.stack = []

    @rule(val=st.integers())
    def push(self, val):
        self.stack.append(val)

    @rule()
    def pop_if_nonempty(self):
        if self.stack:
            val = self.stack.pop()
            assert isinstance(val, int)

TestStack = StackMachine.TestCase

Common Mistakes

✗ @given(st.lists(st.integers())) without min_size when your function requires non-empty input

✓ @given(st.lists(st.integers(), min_size=1)) when function needs at least 1 element


✗ Using the random module inside hypothesis tests (breaks reproducibility)

✓ Let hypothesis control all randomness via strategies

Key Rules

  • Think in properties, not examples: "for all X, f(f_inv(X)) == X"
  • Combine with pytest: hypothesis tests are normal test functions
  • hypothesis saves failing examples in a database and replays them on subsequent runs

Exercises

  1. Write a property test for a function compact(lst) that removes consecutive duplicates: the property should be "no consecutive equal elements in output" AND "all elements from output appear in input in order."
  2. Write a property test for a to_roman(n) / from_roman(s) pair: from_roman(to_roman(n)) == n for 1 <= n <= 3999.
Connections: Extends pytest (reviewed today). Works with @pytest.mark.parametrize. The strategies module uses generators (Week 1).
Testing

Concept 2: doctest — Testing via Documentation

Why learn this? Embed runnable examples directly in docstrings. They serve as both documentation AND tests. If the docs go stale, tests fail. One artifact, two purposes.
Example 1 — Basic doctest in a function
def factorial(n):
    """Return n!.

    >>> factorial(5)
    120
    >>> factorial(0)
    1
    >>> factorial(1)
    1
    """
    if n <= 1:
        return 1
    return n * factorial(n - 1)

if __name__ == '__main__':
    import doctest
    doctest.testmod()
Output
TestResults(failed=0, attempted=3)
Example 2 — Handling exceptions in doctest
def divide(a, b):
    """
    >>> divide(10, 3)
    3.3333333333333335
    >>> divide(10, 0)
    Traceback (most recent call last):
        ...
    ZeroDivisionError: division by zero
    """
    return a / b
Example 3 — Integration with pytest
# In conftest.py or pyproject.toml:
# [tool.pytest.ini_options]
# addopts = "--doctest-modules"

# This makes pytest discover and run all doctests
# in all .py files automatically

Common Mistakes

✗ Expecting exact float output: >>> 1/3 → 0.333... (platform-dependent precision)

✓ Use doctest.ELLIPSIS: >>> 1/3 # doctest: +ELLIPSIS → 0.333...


✗ Forgetting blank line between examples in docstring

✓ Separate independent examples with a blank line

Key Rules

  • Use doctest for simple, documentation-worthy examples
  • Use pytest for complex, edge-case, or parametrized tests
  • python -m doctest mymodule.py -v runs all doctests in a file

Exercises

  1. Write a function fizzbuzz(n) with at least 4 doctest examples including edge cases. Run it with doctest.testmod(verbose=True).
  2. Write a class Stack with push, pop, peek methods, each with doctests showing usage. Include a doctest that shows the exception when popping from an empty stack.
Connections: Pairs with pytest (Review #3) — use doctest for documentation examples, pytest for thorough testing. Works alongside hypothesis (Concept 1) for a comprehensive testing strategy.

Data Formats

Concept 3: tomllib — TOML Parsing (Python 3.11+)

Why learn this? TOML is the standard config format for Python projects (pyproject.toml). As of 3.11, Python includes tomllib for reading TOML natively — read-only; use tomli-w for writing. No more third-party deps just to read your own config.
Example 1 — Basic parsing
import tomllib

toml_str = """
[project]
name = "myapp"
version = "1.0.0"
dependencies = ["requests", "click"]

[project.scripts]
main = "myapp:cli"
"""
config = tomllib.loads(toml_str)
print(config['project']['name'])        # 'myapp'
print(config['project']['dependencies']) # ['requests', 'click']
Example 2 — Reading from file (must open as binary)
with open('pyproject.toml', 'rb') as f:
    config = tomllib.load(f)
print(config['tool']['pytest']['ini_options'])
Example 3 — Nested tables and arrays of tables
toml_str = """
[database]
host = "localhost"
port = 5432

[[database.replicas]]
host = "replica1"
port = 5432

[[database.replicas]]
host = "replica2"
port = 5433
"""
config = tomllib.loads(toml_str)
print(config['database']['replicas'])
# [{'host': 'replica1', 'port': 5432}, {'host': 'replica2', 'port': 5433}]

Common Mistakes

✗ open('config.toml', 'r') — text mode will raise an error

✓ open('config.toml', 'rb') — tomllib requires binary file objects


✗ Trying tomllib.dumps() — does not exist (read-only module)

✓ Use tomli_w.dumps() from the tomli-w package for writing TOML

Key Rules

  • tomllib is read-only — Python chose not to include a writer in the stdlib
  • Always open TOML files in binary mode ('rb')
  • TOML dates/times become datetime.date/datetime.time objects automatically

Exercises

  1. Write a function load_config(path) that reads a TOML config file and returns a dict. Handle the case where the file does not exist by returning a default config dict.
  2. Parse this TOML string and extract all server hosts into a flat list: 
    [[servers]]
name = "web"
host = "10.0.0.1"
[[servers]]
name = "db"
host = "10.0.0.2"
Connections: Replaces manual config parsing. Pairs well with pathlib for file handling. Understanding nested dicts connects to dataclasses and Pydantic for config validation.
Concurrency

Concept 4: asyncio Synchronization Primitives

Why learn this? When multiple coroutines access shared state, you need coordination. asyncio provides Lock, Semaphore, Event, Barrier, and Condition — like threading equivalents but for the event loop. Essential for any non-trivial async application.
Example 1 — asyncio.Lock: mutual exclusion for coroutines
import asyncio

lock = asyncio.Lock()
shared = {'count': 0}

async def increment(name):
    for _ in range(100):
        async with lock:
            shared['count'] += 1
    print(f"{name} done")

async def main():
    await asyncio.gather(
        increment("A"), increment("B"), increment("C")
    )
    print(f"count = {shared['count']}")  # always 300

asyncio.run(main())
Output
A done
B done
C done
count = 300
Example 2 — asyncio.Semaphore: limit concurrent access
sem = asyncio.Semaphore(3)  # max 3 concurrent

async def fetch(url):
    async with sem:
        print(f"Fetching {url}")
        await asyncio.sleep(1)  # simulate I/O
        return f"data from {url}"

async def main():
    urls = [f"https://api.example.com/{i}" for i in range(10)]
    results = await asyncio.gather(*(fetch(u) for u in urls))
Output
Only 3 fetches active at any time, completing in ~4 batches
Example 3 — asyncio.Event: signal between coroutines
event = asyncio.Event()

async def waiter():
    print("Waiting for signal...")
    await event.wait()
    print("Got signal!")

async def setter():
    await asyncio.sleep(2)
    event.set()
    print("Signal sent")

async def main():
    await asyncio.gather(waiter(), setter())
Output
Waiting for signal...
Signal sent
Got signal!
Example 4 — asyncio.Barrier (Python 3.11+): sync point
barrier = asyncio.Barrier(3)

async def worker(name):
    print(f"{name}: preparing")
    await asyncio.sleep(1)
    await barrier.wait()
    print(f"{name}: all ready, proceeding")

async def main():
    await asyncio.gather(
        worker("A"), worker("B"), worker("C")
    )
Output
A: preparing
B: preparing
C: preparing
(all 3 wait, then proceed together)
A: all ready, proceeding
B: all ready, proceeding
C: all ready, proceeding

Common Mistakes

✗ Creating sync primitives outside async context then reusing across event loops

✓ Create primitives inside async functions or as module-level with a known single event loop


✗ Using threading.Lock() in async code (blocks the event loop!)

✓ Use asyncio.Lock() for coroutines

Key Rules

  • asyncio.Lock is NOT thread-safe — for coroutines only; use threading.Lock for threads
  • Semaphore(N) limits concurrency to N coroutines — great for API rate limiting
  • Event.wait() is cancellation-safe — cancelled tasks do not corrupt state

Exercises

  1. Write a rate-limited API fetcher: use Semaphore(5) to limit concurrent HTTP requests. Fetch 20 URLs and collect results.
  2. Implement a "ready-set-go" pattern: 3 async workers prepare independently, then a Barrier synchronizes them before they all start processing together.
Connections: Builds on async/await fundamentals. Semaphore pattern connects to connection pooling (databases, HTTP clients). Event pattern used in producer-consumer architectures.
OOP / Metaprogramming

Concept 5: Advanced __init_subclass__ Patterns

Why learn this? __init_subclass__ lets a parent class hook into subclass creation without a metaclass. It is the modern, simpler alternative to metaclasses for class registration, validation, and configuration. If you are writing frameworks or plugin systems, this is essential.
Example 1 — Auto-registering plugins
import json
import xml.etree.ElementTree as ET

class Plugin:
    _registry = {}

    def __init_subclass__(cls, name=None, **kwargs):
        super().__init_subclass__(**kwargs)
        key = name or cls.__name__.lower()
        Plugin._registry[key] = cls

    @classmethod
    def get(cls, name):
        return cls._registry[name]

class JSONPlugin(Plugin, name="json"):
    def parse(self, data):
        return json.loads(data)

class XMLPlugin(Plugin, name="xml"):
    def parse(self, data):
        return ET.fromstring(data)

print(Plugin._registry)
# {'json': <class 'JSONPlugin'>, 'xml': <class 'XMLPlugin'>}
parser = Plugin.get("json")()
Output
{'json': <class 'JSONPlugin'>, 'xml': <class 'XMLPlugin'>}
Example 2 — Validation: requiring subclasses to define certain attributes
class Serializable:
    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if not hasattr(cls, 'serialization_format'):
            raise TypeError(
                f"{cls.__name__} must define 'serialization_format'"
            )

class JSONModel(Serializable):
    serialization_format = 'json'  # OK

# class BadModel(Serializable):  # TypeError!
#     pass
Example 3 — Combining with dataclasses
from dataclasses import dataclass

class Tracked:
    _instances = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        cls._instances[cls.__name__] = []

    def __post_init__(self):
        type(self)._instances[type(self).__name__].append(self)

@dataclass
class User(Tracked):
    name: str
    age: int

u1 = User("Alice", 30)
u2 = User("Bob", 25)
print(User._instances)
# {'User': [User(name='Alice', age=30), User(name='Bob', age=25)]}
Output
{'User': [User(name='Alice', age=30), User(name='Bob', age=25)]}

Common Mistakes

✗ Forgetting super().__init_subclass__(**kwargs) — breaks multiple inheritance

✓ Always call super() and pass through **kwargs


✗ Using __init_subclass__ when you need per-instance hooks — it runs at class definition, not instantiation

✓ Use __init__ or __post_init__ for per-instance logic

Key Rules

  • __init_subclass__ replaces ~80% of metaclass use cases with simpler code
  • It runs once when the class is defined, not when instances are created
  • Keyword arguments in class Foo(Base, name="x") are passed to __init_subclass__

Exercises

  1. Create a base class Command with __init_subclass__ that auto-registers subclasses by a command_name keyword argument. Implement Command.execute(name) that finds and instantiates the right subclass.
  2. Create a Validated base class whose __init_subclass__ checks that all subclasses define a validate() method (raise TypeError if missing).
Connections: Replaces metaclasses (advanced OOP). Pairs with Enum (Review #5) — Enum itself uses similar class-creation hooks. The plugin registry pattern connects to decorators (Review #6) for an alternative registration approach.
