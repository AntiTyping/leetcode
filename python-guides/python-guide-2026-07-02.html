Review Section (6 items)

1. ChainMap due today rep 3 → 4

collections.ChainMap groups several dicts into one updatable view, searching them in order for a key without merging or copying — ideal for layered configuration (command line over environment over defaults). Lookups fall through to deeper maps; writes and deletes only ever touch the first map. 15 days since last review.

Key Pattern
from collections import ChainMap

defaults = {"color": "red", "size": "M"}
user     = {"color": "blue"}
cfg = ChainMap(user, defaults)

print(cfg["color"])   # blue -- first map wins
print(cfg["size"])    # M    -- falls through to defaults

cfg["size"] = "L"     # writes go to the FIRST map only
print(user)          # {'color': 'blue', 'size': 'L'}
Challenge

Build a 3-layer config — cli over env over defaults. Show that a missing key falls through each layer, that assigning and deleting only affect the first mapping, and that del cfg[k] raises KeyError when k lives only in a deeper map. Then push a temporary scope with cfg.new_child({...}), mutate it, and drop it via .parents.

Pitfall

A ChainMap stores references to the underlying dicts, so later changes to them show through live. Mutation only affects the first mapping — you cannot update or delete a key that exists only in a deeper map (deleting raises KeyError). For a flat, independent copy, use dict(cfg).

2. Stacking & Parameterized Decorators due today rep 3 → 4

A decorator that takes arguments is a function that returns a decorator — three nested layers. Stacking @a above @b applies bottom-up: b wraps the function first, then a wraps that. Always @functools.wraps the inner wrapper to preserve the original's name and docstring. 14 days since last review.

Key Pattern
import functools

def repeat(n):                       # outer: takes the argument
    def deco(func):                    # middle: the real decorator
        @functools.wraps(func)
        def wrapper(*args, **kwargs):   # inner: the replacement
            for _ in range(n):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return deco

@repeat(3)
def greet(name):
    print(f"hi {name}")

greet("Ann")              # prints 3 times
print(greet.__name__)     # greet  -- preserved by @wraps
Challenge

Write a parameterized @retry(times, exc=Exception) that re-calls the function on failure and re-raises after the last attempt. Then stack it with a @timed decorator and reason about which one wraps the other depending on order. Confirm func.__name__ and func.__doc__ survive by using @wraps in both.

Pitfall

Omitting the third layer — writing @repeat without () — passes the decorated function in as n. Stacking order matters: the decorator nearest the def is applied innermost. Skipping @wraps silently replaces __name__/__doc__ with wrapper, breaking introspection and docs.

3. Closures & nonlocal due today rep 3 → 4

A closure is an inner function that captures variables from an enclosing scope, keeping them alive after the outer function returns. nonlocal lets the inner function rebind (not merely read) an enclosing variable — needed for +=, but not for mutating a captured list in place. 15 days since last review.

Key Pattern
def make_counter():
    count = 0
    def increment():
        nonlocal count        # rebind the enclosing variable
        count += 1
        return count
    return increment

c = make_counter()
print(c(), c(), c())           # 1 2 3
print(c.__closure__[0].cell_contents)  # 3 -- the captured cell
Challenge

Write make_accumulator() returning a function that adds its argument to a running total and returns it. Then demonstrate the late-binding trap: build [lambda: i for i in range(3)], show every call returns 2, and fix it with a default-argument capture. Explain why nonlocal is required for total += x but not for items.append(x).

Pitfall

Without nonlocal, count += 1 treats count as local and raises UnboundLocalError. Closures capture variables by reference, not by value — the classic loop-variable bug where every closure sees the final value. Mutating a captured list needs no nonlocal because no rebinding occurs.

4. Lambda Functions due today rep 3 → 4

lambda builds a small, anonymous, single-expression function. It shines as a throwaway key= or callback; it should never be assigned to a name (use def) nor stuffed with logic. Like all closures, lambdas in a loop capture the loop variable by reference. 15 days since last review.

Key Pattern
pairs = [("Ann", 30), ("Bo", 25)]
print(sorted(pairs, key=lambda p: p[1]))   # by age

print(list(map(lambda x: x * x, [1, 2, 3])))   # [1, 4, 9]

# late-binding trap: all three see the final i
fns = [lambda: i for i in range(3)]
print([f() for f in fns])   # [2, 2, 2]
Challenge

Sort a list of dicts by two fields at once using a single tuple-returning lambda. Reproduce the late-binding bug with lambdas built in a loop and fix it two ways: a default-argument capture (lambda i=i: ...) and functools.partial. Then name one situation where a proper def is strictly better than a lambda.

Pitfall

Assigning a lambda to a variable (add = lambda a, b: a + b) is a style error — def gives a real name in tracebacks (PEP 8). A lambda is a single expression: no statements, assignments, or annotations. Late binding bites when you create lambdas inside a loop.

5. functools due today rep 3 → 4

functools bundles higher-order helpers: partial pre-binds arguments, reduce folds a sequence to a single value, lru_cache/cache memoize expensive calls, and wraps preserves metadata inside decorators. 15 days since last review.

Key Pattern
from functools import partial, reduce, lru_cache

basetwo = partial(int, base=2)     # pre-bind the base=2 keyword
print(basetwo("101"))               # 5

print(reduce(lambda a, b: a * b, [1, 2, 3, 4]))   # 24

@lru_cache(maxsize=None)
def fib(n):
    return n if n < 2 else fib(n - 1) + fib(n - 2)
print(fib(30))                   # 832040
print(fib.cache_info().hits)     # >0 -- memoization at work
Challenge

Use partial to build a function that strips commas from a string (pre-bind an argument of str.strip). Memoize a recursive grid-paths counter with @cache and compare its timing to an uncached version. Then explain why lru_cache requires hashable arguments and how cache_info() and cache_clear() help you inspect and reset it.

Pitfall

lru_cache keeps references to every argument and result — it can leak memory, and its keys must be hashable (no list/dict args). reduce with no initializer raises on an empty sequence — pass a start value. partial binds positionally unless you pass the pre-set arguments as keywords.

6. dataclasses Advanced Patterns 2 days overdue rep 2 → 3

Beyond the basics, dataclasses support field(default_factory=...) for mutable defaults, frozen=True for immutability and hashability, order=True to autogenerate </> comparisons, and __post_init__ for derived or validated fields. 8 days since last review.

Key Pattern
from dataclasses import dataclass, field

@dataclass(frozen=True, order=True)
class Point:
    x: int
    y: int
    tags: list = field(default_factory=list, compare=False)
    def __post_init__(self):
        # frozen: must bypass the blocked __setattr__
        object.__setattr__(self, "mag", (self.x**2 + self.y**2) ** 0.5)

p = Point(3, 4)
print(p.mag)                    # 5.0
print(Point(1, 1) < Point(2, 0))   # True -- compares (x, y)
Challenge

Build a frozen, ordered dataclass with a default_factory list excluded from comparison (compare=False). Add a __post_init__ that validates a field and raises ValueError. Show why a plain tags: list = [] default is rejected at class creation and how default_factory fixes it. Finally, use dataclasses.replace(p, x=9) to produce a modified copy of the frozen instance.

Pitfall

A mutable default (tags: list = []) raises ValueError at class-definition time — use field(default_factory=list). In a frozen=True dataclass, normal assignment is blocked, so __post_init__ must use object.__setattr__. order=True requires eq=True (the default) to generate comparisons.

New Concepts — Sequences & Control Flow (3 concepts)

Three pieces of core Python syntax that replace loops and flag variables with a single clear line. Slicing extracts, reverses, and rewrites whole regions of a sequence; the surprising else on loops and try makes "search succeeded / failed" explicit; and conditional expressions plus chained comparisons pack small branches and range tests into expressions.

Sequences

Concept 1: Sequence Slicing Deep Dive

Why learn this? Slicing is Python's most powerful sequence operation: s[start:stop:step] extracts, samples, and reverses without a loop, and on the left of = it can replace, insert, or delete whole regions of a list. Learning the full grammar — defaults, negative indices, and the step — turns many hand-written loops into one expression that reads as intent.
Example 1 — start : stop : step with defaults
s = "abcdefg"

print(s[1:4])    # bcd  -- indices 1,2,3 (stop is EXCLUDED)
print(s[:3])     # abc  -- start defaults to 0
print(s[4:])     # efg  -- stop defaults to len
print(s[::2])    # aceg -- every 2nd item
Output
bcd abc efg aceg
Example 2 — negative indices and reversal
s = "abcdefg"

print(s[-3:])       # efg  -- last 3
print(s[:-2])       # abcde -- all but last 2
print(s[::-1])      # gfedcba -- reversed copy
print(s[-1:-4:-1])  # gfe  -- last three, backwards
Output
efg abcde gfedcba gfe
Example 3 — slice assignment: replace, delete, insert
xs = [0, 1, 2, 3, 4]

xs[1:3] = ["a", "b", "c"]   # replace 2 items with 3 -> list resizes
print(xs)   # [0, 'a', 'b', 'c', 3, 4]

xs[1:4] = []                # delete a whole region
print(xs)   # [0, 3, 4]

xs[1:1] = [9, 8]           # insert without removing anything
print(xs)   # [0, 9, 8, 3, 4]
Output
[0, 'a', 'b', 'c', 3, 4] [0, 3, 4] [0, 9, 8, 3, 4]
Example 4 — clamping, the slice object, and copies
s = "abcdef"

print(s[1:99])   # bcdef -- out-of-range bounds CLAMP, no error
print(s[99:])    # ''    -- empty, still safe

r = slice(1, None, 2)   # a reusable slice object
print(s[r])      # bdf

xs = [1, 2, 3]
ys = xs[:]        # [:] makes a shallow copy
ys.append(4)
print(xs, ys)     # [1, 2, 3] [1, 2, 3, 4]
Output
bcdef '' bdf [1, 2, 3] [1, 2, 3, 4]

Common Mistakes

✗ Reversing a sequence with a manual loop or "".join(reversed(s)) when a slice does it directly

✓ Use s[::-1] for a reversed copy — it works on str, list, tuple, and bytes


✗ Assuming s[i:j] raises when the bounds exceed the length — it silently clamps, which can hide an off-by-one bug

✓ Validate indices yourself when an out-of-range slice would be a real logic error; only single-index access raises IndexError

Key Rules

  • s[start:stop:step] — stop is exclusive; omit any part for its default (0, len, 1); a negative step walks backwards and flips the start/stop defaults
  • Negative indices count from the end; slices never raise on out-of-range bounds (they clamp), unlike single-index access which raises IndexError
  • On lists, slice assignment can grow/shrink (xs[a:b] = iterable), delete a region (xs[a:b] = []), or insert (xs[i:i] = ...); s[:] is a shallow copy

Exercises

  1. Given a list, return (a) every 3rd element starting at index 0 and (b) the list reversed — both using slicing only, no loops.
  2. Rotate a list left by k with slicing (xs[k:] + xs[:k]) and verify it against a manual rotation for a few values of k.
  3. Given a list of 10 numbers, replace the middle 4 elements with a single value using slice assignment, then delete the first two elements with another slice.
Connections: slicing complements enumerate/zip (Day 31) for sequence work; s[::-1] mirrors reversed(); slice assignment is the mutable-sequence analog of list methods like insert/extend; the slice object is exactly what a custom container receives in __getitem__ (Dunder Methods, OOP).
Control Flow

Concept 2: else Clauses on Loops & try

Why learn this? In Python, else attaches not only to if but to for, while, and try — a feature that surprises even seasoned developers. Loop-else runs only when the loop finishes without a break; try-else runs only when the try block raised nothing. Used well, they delete flag variables and make success/failure control flow explicit.
Example 1 — for/else: search and report
def find_even(nums):
    for n in nums:
        if n % 2 == 0:
            print(f"found {n}")
            break
    else:
        print("no even number")   # runs ONLY if no break fired

find_even([1, 3, 5])   # no even number
find_even([1, 4, 5])   # found 4
Output
no even number found 4
Example 2 — while/else
n = 15
while n > 1:
    if n % 20 == 0:
        print("found a multiple of 20")
        break
    n -= 1
else:
    print("no multiple of 20 in range")   # loop ended normally
Output
no multiple of 20 in range
Example 3 — try / except / else / finally
def parse(s):
    try:
        value = int(s)
    except ValueError:
        print(f"{s!r} is not an int")
    else:
        print(f"parsed {value}")   # runs ONLY if try raised nothing
    finally:
        print("done")              # ALWAYS runs

parse("42")
parse("x")
Output
parsed 42 done 'x' is not an int done
Example 4 — the subtle cases: empty loop & continue
for x in []:          # body never executes...
    print(x)
else:
    print("else still runs -- the loop 'completed' with no break")

for x in [1, 2]:
    if x == 1:
        continue          # continue does NOT count as break
    print("body", x)
else:
    print("else after continue")
Output
else still runs -- the loop 'completed' with no break body 2 else after continue

Common Mistakes

✗ Reading for/else as "else runs if the loop body never executed" — it actually runs whenever the loop ends without a break (including empty iterables)

✓ Memorize the rule: loop-else == "no break happened"; try-else == "no exception happened"


✗ Wrapping extra, non-risky code inside try where a stray exception could be caught by the wrong except

✓ Keep the try body to the single risky call and move success-only code into else

Key Rules

  • for/while ... else: the else runs iff the loop terminated normally (iterable exhausted / condition false), NOT if a break fired; continue does not skip it
  • try ... else: the else runs iff the try block raised nothing — use it to keep the try body minimal and precise
  • finally always runs (even on return, break, or exception); it is for cleanup, not success-only logic — that is what else is for

Exercises

  1. Write find_first(pred, items) that prints the first matching item or "not found" using a for/else with break — no flag variable.
  2. Write a validator with try/except/else/finally that converts a string to float: report success in else, the error in except, and print "checked" in finally.
  3. Predict, then verify, what a for/else prints when the iterable is empty, when a break fires on the last element, and when continue is used on every iteration.
Connections: loop-else pairs with break-driven linear searches (a recurring LeetCode pattern); try/else/finally builds on the exception hierarchy (Error Handling) and on context managers, which automate the cleanup finally does by hand; the "no break" semantics complement any/all short-circuiting (Day 31).

Concept 3: Conditional Expressions & Chained Comparisons

Why learn this? The ternary a if cond else b puts a small branch inside an expression — usable in comprehensions, arguments, and returns where a full if statement cannot go. Chained comparisons like lo <= x < hi read like math, evaluate each operand once, and short-circuit. Together with truthiness, they make intent compact and correct.
Example 1 — the conditional expression (ternary)
n = 7
parity = "even" if n % 2 == 0 else "odd"
print(parity)                 # odd

# inline inside a comprehension
labels = ["+" if x >= 0 else "-" for x in [-2, 3, 0]]
print(labels)                 # ['-', '+', '+']

# as an inline default
name = None
print(name if name else "anonymous")   # anonymous
Output
odd ['-', '+', '+'] anonymous
Example 2 — chained comparisons read like ranges
x = 5
print(1 <= x <= 10)   # True  -- a clean range test
print(1 < x < 3)      # False

# equivalent to (1 <= x) and (x <= 10), but x is evaluated once
lo, hi = 0, 100
print(lo <= x < hi)     # True  -- half-open range, like slicing
Output
True False True
Example 3 — each operand evaluated once, with short-circuit
def side_effect():
    print("called")
    return 5

# the middle term runs a single time despite two comparisons
print(3 < side_effect() < 10)
Output
called True
Example 4 — truthiness drives if / and / or
# falsy: 0, 0.0, "", [], {}, set(), None -- everything else is truthy
print(bool([]), bool([0]), bool(""), bool("0"))   # False True False True

# 'or' returns the FIRST truthy operand -- a common default idiom
config = {}
port = config.get("port") or 8080
print(port)              # 8080

# 'and' returns the first falsy operand, else the last
print("" and "x", "a" and "b")   # '' -> b
Output
False True False True 8080 b

Common Mistakes

✗ Overusing the ternary for nested logic: a if p else (b if q else c) — hard to read

✓ Reach for a ternary only for a short either/or; use a normal if/elif for anything that branches more than once


✗ Writing x > 0 and x < 10 by hand, duplicating x when it could be an expensive call

✓ Use the chained form 0 < x < 10 — clearer, and it evaluates x once

Key Rules

  • Conditional expression: value_if_true if condition else value_if_false — it is an expression, so it fits inside comprehensions, calls, and returns; the else is mandatory
  • Chained comparisons a < b < c mean (a < b) and (b < c) with each term evaluated once and short-circuit on the first false; you may mix operators (a <= b > c)
  • Truthiness: 0, 0.0, "", [], {}, set(), None are falsy; and/or return an operand (not a bool) and short-circuit — great for defaults, but beware valid falsy values like 0

Exercises

  1. Given a list of numbers, build a list of strings "neg"/"zero"/"pos" using a nested conditional expression inside one comprehension.
  2. Write in_range(x, lo, hi) returning True when lo <= x < hi using a single chained comparison, and test both boundaries.
  3. Explain why port = config.get("port") or 8080 is buggy if 0 is a valid port, then rewrite it correctly with an explicit None check using a conditional expression.
Connections: the ternary is the expression form of the if/else clauses in Concept 2; chained comparisons echo the half-open ranges of slicing (Concept 1) and range(); truthiness underlies any/all/filter (Day 31), and the or-default idiom recurs with dict.get and function defaults (Concept 4).

New Concepts — Function Signatures (2 concepts)

Two things every Python function author must know cold: when default arguments are evaluated (once, at definition — the source of the single most common gotcha), and how to control the way callers may pass each argument with the * and / markers. Together they make APIs safer and clearer.

Functions

Concept 4: Mutable Default Arguments

Why learn this? A default argument is evaluated once, when the function is defined — not on each call. If that default is a mutable object (list, dict, set), every call that omits the argument shares the same object, so state silently leaks between calls. This is the most common Python gotcha; the fix is a None sentinel.
Example 1 — the trap
def append_to(item, target=[]):   # default list created ONCE, at def time
    target.append(item)
    return target

print(append_to(1))   # [1]
print(append_to(2))   # [1, 2]   -- the SAME list is reused!
print(append_to(3))   # [1, 2, 3]
Output
[1] [1, 2] [1, 2, 3]
Example 2 — the fix: a None sentinel
def append_to(item, target=None):
    if target is None:
        target = []            # a fresh list every call
    target.append(item)
    return target

print(append_to(1))   # [1]
print(append_to(2))   # [2]   -- independent now
Output
[1] [2]
Example 3 — defaults are evaluated at def time
import time

def stamp(t=time.time()):   # BUG: captured once, when defined
    return t

a = stamp()
# ...some time later...
b = stamp()
print(a == b)             # True -- same frozen timestamp

def stamp2(t=None):        # correct: compute in the body
    return time.time() if t is None else t
Output
True
Example 4 — the shared object is inspectable
def count(x, seen={}):     # the dict persists across calls
    seen[x] = seen.get(x, 0) + 1
    return seen

count(1); count(1); count(2)
print(count.__defaults__)   # ({1: 3, 2: 1},) -- proof it never resets
Output
({1: 3, 2: 1},)

Common Mistakes

✗ def f(x, acc=[]) or def f(x, opts={}) — the mutable default is created once and shared by every call

✓ def f(x, acc=None): acc = acc if acc is not None else [] — allocate a fresh object inside the body


✗ Using a call like time.time() or [] as a default expecting it to run on each call

✓ Use None (or a unique sentinel) as the default and compute the real value in the body

Key Rules

  • Default-argument expressions are evaluated exactly once, at function-definition time, and stored on func.__defaults__ — never re-evaluated per call
  • Never use a mutable object (list/dict/set) or a "compute now" call as a default; use None as a sentinel and build the real value inside the function
  • To distinguish "not passed" from "passed None", use a private sentinel: _MISSING = object() and compare with is

Exercises

  1. Write a buggy accumulate(x, into=[]) that shows shared state across three calls, then rewrite it with a None sentinel and prove the calls are independent.
  2. Create a function whose default is a _MISSING = object() sentinel so it can tell f() (no arg) apart from f(None); print which branch runs for each.
  3. Given def counter(n, seen={}) that counts occurrences, explain what counter.__defaults__ shows after several calls, then rewrite it so unrelated callers do not share state.
Connections: this gotcha is exactly why dataclasses require field(default_factory=list) (reviewed today); the None-sentinel pattern reappears with Optional type hints (Type System); and "evaluated once at def time" parallels late-binding closures (Closures, reviewed today) — both are about when Python captures a value.
Functions

Concept 5: Keyword-Only & Positional-Only Parameters

Why learn this? Python lets you control how callers may pass each argument. A bare * in the signature forces the parameters after it to be keyword-only (callers must name them); a / forces the parameters before it to be positional-only (callers may not name them). This clarifies APIs, prevents accidental positional mistakes, and lets you rename internal parameters freely.
Example 1 — keyword-only with a bare *
def connect(host, *, timeout=10, retries=3):
    print(host, timeout, retries)

connect("db", timeout=5)              # ok -- named
connect("db", timeout=5, retries=1)
# connect("db", 5)  -> TypeError: too many positional arguments
Output
db 5 3 db 5 1
Example 2 — positional-only with /
def area(width, height, /):
    return width * height

print(area(3, 4))          # 12
# area(width=3, height=4) -> TypeError: positional-only
Output
12
Example 3 — combining / ... * in one signature
def f(pos_only, /, standard, *, kw_only):
    print(pos_only, standard, kw_only)

f(1, 2, kw_only=3)         # standard passed positionally
f(1, standard=2, kw_only=3)  # standard passed by name -- both allowed
# f(pos_only=1, ...) -> TypeError
Output
1 2 3 1 2 3
Example 4 — why it matters: positional-only frees the name
def make(name, /, **attrs):
    # 'name' is positional-only, so a name= keyword lands in **attrs
    return {**attrs, "name": name}

print(make("Ann", name="db-user", role="dev"))
# the positional 'name' wins; the keyword 'name' stayed as data
Output
{'name': 'Ann', 'role': 'dev'}

Common Mistakes

✗ A wide public function with all-positional params, so callers write f(True, False, None, 5) — unreadable and fragile to reordering

✓ Put optional flags after a bare * so they must be named: f(data, *, verbose=False, limit=None)


✗ Relying on callers passing a param by keyword, then renaming that param later and breaking them

✓ Mark conceptually-positional params with / (like builtins abs(x, /)); it lets you rename them safely

Key Rules

  • A bare * marks everything after it keyword-only — callers must pass those by name; ideal for optional flags and to prevent positional mistakes
  • A / marks everything before it positional-only — callers may not use those names; mirrors many C-implemented builtins (e.g. len(obj, /))
  • Order is fixed: positional-only, /, normal, *args or bare *, keyword-only, **kwargs; positional-only names are also free to reappear inside **kwargs

Exercises

  1. Write replace(text, /, *, old, new) so callers must name old and new but cannot name text; confirm replace(s, old="a", new="b") works and replace(text=s, ...) fails.
  2. Design create_user(name, /, *, admin=False, **extra) and show that calling it with name="x" routes "name" into extra while the positional stays intact.
  3. Take an existing all-positional function and refactor its optional parameters to be keyword-only with *; then explain which previously valid calls now raise TypeError.
Connections: keyword-only params pair with **kwargs and the unpacking/star topic; positional-only mirrors the C-level builtins seen in Numeric Built-ins (Day 31); forcing keywords improves readability the way named tuples and dataclasses do; and this builds directly on default arguments (Concept 4), which are common on keyword-only flags.
