super() Deep Dive

OOP • 4 days since last review

Quick Reminder

super() returns a proxy that delegates method calls to the next class in the MRO. In diamond inheritance, super().__init__() in every class ensures each __init__ is called exactly once. Use **kwargs forwarding for cooperative multiple inheritance so each class takes what it needs and passes the rest along.

Key Pattern

class Base:
    def __init__(self, **kwargs):
        print(f"Base.__init__({kwargs})")

class Left(Base):
    def __init__(self, left_val, **kwargs):
        self.left_val = left_val
        print(f"Left.__init__(left_val={left_val})")
        super().__init__(**kwargs)

class Right(Base):
    def __init__(self, right_val, **kwargs):
        self.right_val = right_val
        print(f"Right.__init__(right_val={right_val})")
        super().__init__(**kwargs)

class Diamond(Left, Right):
    def __init__(self):
        super().__init__(left_val=10, right_val=20)

d = Diamond()
# Left.__init__(left_val=10)
# Right.__init__(right_val=20)
# Base.__init__({})
print(Diamond.mro())
# [Diamond, Left, Right, Base, object]
Challenge

Create a cooperative hierarchy: Serializable (adds .to_dict()), Validatable (adds .validate()), and Model(Serializable, Validatable). Each __init__ should accept **kwargs and forward via super(). Verify that constructing a Model triggers all three __init__s exactly once.

#1 Pitfall: Forgetting **kwargs forwarding in cooperative __init__. If Left.__init__ doesn't accept and forward **kwargs, Right.__init__ never receives right_val and you get a TypeError.

Abstract Base Classes & Protocol

OOP • 3 days since last review

Quick Reminder

ABC + @abstractmethod enforces that subclasses implement specific methods (nominal subtyping). Protocol from typing defines structural subtyping — any class that has the right methods is compatible, no inheritance required. Use ABCs when you control the hierarchy; use Protocols when you want duck-typing with static type checking.

Key Pattern

from abc import ABC, abstractmethod
from typing import Protocol

# ABC -- nominal subtyping (must inherit)
class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...

class Circle(Shape):
    def __init__(self, r: float):
        self.r = r
    def area(self) -> float:
        return 3.14159 * self.r ** 2

# Protocol -- structural subtyping (just implement the method)
class Drawable(Protocol):
    def draw(self) -> str: ...

class Widget:          # no inheritance needed!
    def draw(self) -> str:
        return "Widget drawn"

def render(d: Drawable) -> None:
    print(d.draw())

render(Widget())  # Widget drawn -- type-safe, no inheritance
Challenge

Define a Comparable Protocol requiring __lt__ and __eq__. Write a generic min_value(items: list[Comparable]) -> Comparable function. Test it with both a custom class and plain integers (both satisfy the Protocol without inheriting from it).

#1 Pitfall: Trying to instantiate an ABC directly. Shape() raises TypeError. ABCs exist only to be subclassed. If you forget an abstract method in a subclass, you get the same error at instantiation time, not at class definition time.

Dunder Methods — __getitem__, __len__, __contains__

OOP • 3 days since last review

Quick Reminder

__getitem__ enables obj[key] (indexing and slicing). __len__ enables len(obj). __contains__ enables x in obj. If you define __getitem__ without __contains__, Python falls back to iterating through indices. Together they make your class behave like a built-in collection.

Key Pattern

class Playlist:
    def __init__(self, songs: list[str]):
        self._songs = songs

    def __getitem__(self, index):
        return self._songs[index]

    def __len__(self) -> int:
        return len(self._songs)

    def __contains__(self, song: str) -> bool:
        return song.lower() in (s.lower() for s in self._songs)

p = Playlist(["Imagine", "Yesterday", "Bohemian Rhapsody"])
print(p[0])            # Imagine
print(p[-1])           # Bohemian Rhapsody
print(p[0:2])          # ['Imagine', 'Yesterday']
print(len(p))          # 3
print("imagine" in p)  # True (case-insensitive)
Challenge

Build a Matrix class that supports m[row, col] indexing via __getitem__ (using tuple unpacking), len(m) returning total elements, and value in m checking all cells. Support slicing for entire rows: m[1] returns row 1 as a list.

#1 Pitfall: Forgetting that __getitem__ with slices receives a slice object, not integers. You must handle isinstance(index, slice) if you want slicing to work properly alongside integer indexing.

Dunder Methods — __iter__, __next__, __call__

OOP • 3 days since last review

Quick Reminder

__iter__ returns an iterator (often self). __next__ returns the next value or raises StopIteration. Together they make your class work in for loops. __call__ makes instances callable like functions — obj() invokes obj.__call__(). Use __call__ for stateful functions, strategies, or replacing closures.

Key Pattern

class Countdown:
    def __init__(self, start: int):
        self.current = start

    def __iter__(self):
        return self

    def __next__(self) -> int:
        if self.current < 0:
            raise StopIteration
        val = self.current
        self.current -= 1
        return val

class Multiplier:
    def __init__(self, factor: int):
        self.factor = factor

    def __call__(self, x: int) -> int:
        return x * self.factor

for n in Countdown(3):
    print(n, end=" ")  # 3 2 1 0

double = Multiplier(2)
print(double(5))   # 10
print(double(12))  # 24
Challenge

Create a MovingAverage class that is both callable and iterable. Calling avg(value) adds a value and returns the current average. Iterating yields all recorded averages. Support a window parameter for sliding-window mode.

#1 Pitfall: Making __iter__ return self for a class that should be iterable multiple times. If __iter__ returns self, the iterator is consumed after one for loop. Return a fresh iterator object instead, or use yield in __iter__.

Descriptors

OOP • 3 days since last review

Quick Reminder

Descriptors implement __get__, __set__, and/or __delete__. They are defined as class attributes and intercept access on instances. Data descriptors (with __set__) take priority over instance __dict__. Non-data descriptors (only __get__) don't. Properties, classmethods, and staticmethods are all implemented as descriptors.

Key Pattern

class Validated:
    def __init__(self, min_val=None, max_val=None):
        self.min_val = min_val
        self.max_val = max_val

    def __set_name__(self, owner, name):
        self.attr_name = f"_{name}"

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self.attr_name, None)

    def __set__(self, obj, value):
        if self.min_val is not None and value < self.min_val:
            raise ValueError(f"Must be >= {self.min_val}")
        if self.max_val is not None and value > self.max_val:
            raise ValueError(f"Must be <= {self.max_val}")
        setattr(obj, self.attr_name, value)

class Product:
    price = Validated(min_val=0)
    quantity = Validated(min_val=0, max_val=1000)

    def __init__(self, price, quantity):
        self.price = price
        self.quantity = quantity

p = Product(9.99, 5)
print(p.price, p.quantity)  # 9.99 5
Challenge

Write a TypeChecked descriptor that enforces a specific type on assignment. Usage: name = TypeChecked(str), age = TypeChecked(int). Attempting self.age = "hello" should raise TypeError. Combine it with Validated for type + range checking.

#1 Pitfall: Storing descriptor state on the descriptor itself instead of on the instance. If you write self.value = value in __set__, all instances share the same value. Use setattr(obj, self.attr_name, value) or obj.__dict__[self.attr_name] = value.

Exception Hierarchy & Custom Exceptions

Error Handling • 3 days since last review

Quick Reminder

All exceptions inherit from BaseException. Application exceptions inherit from Exception. except Exception catches most errors but not KeyboardInterrupt/SystemExit. Custom exceptions should inherit from Exception (or a domain-specific base). Use exception groups (ExceptionGroup, Python 3.11+) for multiple simultaneous errors.

Key Pattern

class AppError(Exception):
    """Base for all application errors."""

class ValidationError(AppError):
    def __init__(self, field: str, message: str):
        self.field = field
        self.message = message
        super().__init__(f"{field}: {message}")

class NotFoundError(AppError):
    def __init__(self, resource: str, id: int):
        self.resource = resource
        self.id = id
        super().__init__(f"{resource} #{id} not found")

try:
    raise ValidationError("email", "invalid format")
except ValidationError as e:
    print(f"Validation: {e.field} - {e.message}")
except AppError as e:
    print(f"App error: {e}")
# Validation: email - invalid format
Challenge

Design an exception hierarchy for a banking app: BankError base, InsufficientFundsError(amount, balance), AccountLockedError(reason), TransferError(from_acct, to_acct, cause) that chains the underlying cause with raise ... from. Write a function that demonstrates catching and re-raising with context.

#1 Pitfall: Catching Exception too broadly. A bare except Exception: swallows errors you did not expect (like TypeError from a coding bug). Catch the most specific exception type possible, and let unexpected errors propagate.

Memory Optimization

Concurrency & Performance • New

Why It Matters

Python objects carry significant overhead — a single int uses 28 bytes, a small dict over 200 bytes. When you create millions of objects, memory can balloon quickly. Understanding __slots__, generators, array, and memoryview lets you cut memory usage by 50-90%. Profiling with tracemalloc and sys.getsizeof tells you where memory is going.

Progressive Explanation

sys.getsizeof vs pympler.asizeof — shallow vs deep:

import sys

# sys.getsizeof gives SHALLOW size (just the object, not what it references)
data = {"name": "Alice", "scores": [90, 85, 92, 88]}

print(f"dict (shallow): {sys.getsizeof(data)} bytes")
print(f"str 'Alice':    {sys.getsizeof('Alice')} bytes")
print(f"list [90,...]:  {sys.getsizeof(data['scores'])} bytes")

# The shallow size of the dict doesn't include the strings and list inside!
# For deep size, use pympler (pip install pympler):
# from pympler import asizeof
# print(f"dict (deep): {asizeof.asizeof(data)} bytes")

# Manual deep size approximation:
def deep_getsizeof(obj, seen=None):
    """Recursively calculate deep size of an object."""
    if seen is None:
        seen = set()
    obj_id = id(obj)
    if obj_id in seen:
        return 0
    seen.add(obj_id)
    size = sys.getsizeof(obj)
    if isinstance(obj, dict):
        size += sum(deep_getsizeof(k, seen) + deep_getsizeof(v, seen)
                    for k, v in obj.items())
    elif isinstance(obj, (list, tuple, set, frozenset)):
        size += sum(deep_getsizeof(i, seen) for i in obj)
    return size

print(f"dict (deep):    {deep_getsizeof(data)} bytes")
Output (example, 64-bit Python)
dict (shallow): 184 bytes str 'Alice': 54 bytes list [90,...]: 120 bytes dict (deep): 530 bytes

__slots__ combined with other techniques:

import sys

class RegularPoint:
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

class SlotPoint:
    __slots__ = ("x", "y", "z")
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

rp = RegularPoint(1.0, 2.0, 3.0)
sp = SlotPoint(1.0, 2.0, 3.0)

print(f"Regular: {sys.getsizeof(rp)} + dict {sys.getsizeof(rp.__dict__)} bytes")
print(f"Slots:   {sys.getsizeof(sp)} bytes (no __dict__)")

# At scale: 1 million points
import tracemalloc
tracemalloc.start()

points_regular = [RegularPoint(i, i+1, i+2) for i in range(100_000)]
snapshot1 = tracemalloc.take_snapshot()
regular_mem = sum(stat.size for stat in snapshot1.statistics("filename"))

tracemalloc.stop()
tracemalloc.start()

points_slots = [SlotPoint(i, i+1, i+2) for i in range(100_000)]
snapshot2 = tracemalloc.take_snapshot()
slots_mem = sum(stat.size for stat in snapshot2.statistics("filename"))

tracemalloc.stop()
print(f"\n100k Regular: ~{regular_mem / 1024 / 1024:.1f} MB")
print(f"100k Slots:   ~{slots_mem / 1024 / 1024:.1f} MB")
Output (example)
Regular: 48 + dict 104 bytes Slots: 56 bytes (no __dict__) 100k Regular: ~20.8 MB 100k Slots: ~8.4 MB

Generators vs lists for large datasets:

import sys
import tracemalloc

# List approach -- all in memory at once
def squares_list(n):
    return [i * i for i in range(n)]

# Generator approach -- lazy, one at a time
def squares_gen(n):
    return (i * i for i in range(n))

N = 1_000_000

# Memory comparison
lst = squares_list(N)
gen = squares_gen(N)

print(f"List:      {sys.getsizeof(lst):>12,} bytes")
print(f"Generator: {sys.getsizeof(gen):>12,} bytes")

# Both produce the same results
print(f"Sum (list): {sum(lst)}")
print(f"Sum (gen):  {sum(squares_gen(N))}")  # re-create since consumed

# Real-world: processing a large CSV line by line
def process_large_file(path):
    """Generator -- never loads full file into memory."""
    with open(path) as f:
        for line in f:    # file object is already a generator!
            yield line.strip().split(",")

# vs loading everything at once (bad for large files):
# lines = open(path).readlines()  # entire file in RAM!
Output
List: 8,448,728 bytes Generator: 200 bytes Sum (list): 333332833333500000 Sum (gen): 333332833333500000

array module for typed numeric arrays:

import sys
from array import array

# Python list of ints: each int is a full Python object (~28 bytes)
py_list = list(range(100_000))
print(f"Python list: {sys.getsizeof(py_list):>10,} bytes")

# array.array: stores raw C-type values (4 or 8 bytes each)
# Type codes: 'i' = signed int, 'f' = float, 'd' = double, 'B' = unsigned byte
int_array = array('i', range(100_000))  # 4 bytes per int
print(f"array('i'):  {sys.getsizeof(int_array):>10,} bytes")

# For bytes/binary data
byte_array = array('B', [0] * 100_000)  # 1 byte per element
print(f"array('B'):  {sys.getsizeof(byte_array):>10,} bytes")

# array supports standard list operations
int_array.append(100_000)
int_array.extend([100_001, 100_002])
print(f"Length after append: {len(int_array)}")
print(f"Sum: {sum(int_array)}")

# Type enforcement: can only store declared type
try:
    int_array.append(3.14)  # float into int array!
except TypeError as e:
    print(f"TypeError: {e}")
Output
Python list: 800,056 bytes array('i'): 400,064 bytes array('B'): 100,064 bytes Length after append: 100003 Sum: 5000150003 TypeError: 'float' object cannot be interpreted as an integer

memoryview for zero-copy slicing:

# memoryview provides a "window" into a buffer without copying data
data = bytearray(b"Hello, World! This is a large buffer of data.")

# Normal slicing COPIES the data
slice_copy = data[7:12]
print(f"Copy:       {bytes(slice_copy)} (id: {id(slice_copy)})")

# memoryview slicing does NOT copy
mv = memoryview(data)
slice_view = mv[7:12]
print(f"View:       {bytes(slice_view)} (id: {id(slice_view)})")

# Modifying through the view modifies the original!
slice_view[0:5] = b"Earth"
print(f"Original:   {data.decode()}")

# Memory savings with large data
import sys
big_data = bytearray(1_000_000)
big_copy = big_data[100:200]       # copies 100 bytes
big_view = memoryview(big_data)[100:200]  # zero copy

print(f"\nbytearray slice: {sys.getsizeof(big_copy)} bytes")
print(f"memoryview:      {sys.getsizeof(big_view)} bytes")

mv.release()  # always release when done
Output
Copy: b'World' (id: 140234567890) View: b'World' (id: 140234567456) Original: Hello, Earth! This is a large buffer of data. bytearray slice: 157 bytes memoryview: 184 bytes

Common Mistakes

Don't
# Loading everything into a list when you only
# need to iterate once
results = [transform(x) for x in huge_dataset]
for r in results:
    process(r)
# huge_dataset AND results both in memory!
Do
# Use a generator -- process one at a time
results = (transform(x) for x in huge_dataset)
for r in results:
    process(r)
# Only one item in memory at a time
Don't
# Using sys.getsizeof on a container and
# thinking that's the total memory
d = {"a": [1,2,3], "b": [4,5,6]}
print(sys.getsizeof(d))  # ~232 bytes
# WRONG! Doesn't include the lists inside!
Do
# Use deep size calculation or tracemalloc
# for accurate memory measurement
tracemalloc.start()
d = {"a": [1,2,3], "b": [4,5,6]}
current, _ = tracemalloc.get_traced_memory()
# Or use pympler.asizeof.asizeof(d)

Key Rules

  • sys.getsizeof() measures shallow size only. For deep (recursive) size, use pympler.asizeof or tracemalloc.
  • Use generators instead of lists when you only need to iterate once — O(1) vs O(n) memory.
  • __slots__ saves ~40-50% memory per instance by eliminating the per-object __dict__.
  • array.array stores typed values compactly (like C arrays). Use for large numeric datasets where NumPy is overkill.
  • memoryview provides zero-copy slicing of buffers — critical for high-throughput binary data processing.

struct & Binary Data

Concurrency & Performance • New

Why It Matters

Binary data is everywhere: network protocols, file formats (PNG, ZIP, ELF), hardware communication, and database storage. The struct module lets you pack Python values into compact binary representations and unpack binary data into Python values. Understanding endianness and format strings is essential for interoperability with C code, network protocols, and binary files.

Progressive Explanation

struct.pack and struct.unpack basics:

import struct

# Pack Python values into bytes
# Format: '>' = big-endian, 'I' = unsigned int, 'f' = float
packed = struct.pack('>If', 42, 3.14)
print(f"Packed: {packed}")
print(f"Length: {len(packed)} bytes")  # 4 (int) + 4 (float) = 8

# Unpack bytes back into Python values
values = struct.unpack('>If', packed)
print(f"Unpacked: {values}")
print(f"Int: {values[0]}, Float: {values[1]:.2f}")

# Common format characters:
# 'b'=int8, 'B'=uint8, 'h'=int16, 'H'=uint16
# 'i'=int32, 'I'=uint32, 'q'=int64, 'Q'=uint64
# 'f'=float32, 'd'=float64, 's'=bytes, '?'=bool

# Pack a network packet header
header = struct.pack('!BBH', 1, 0, 1024)  # version, type, length
print(f"Header: {header.hex()}")
ver, typ, length = struct.unpack('!BBH', header)
print(f"Version={ver}, Type={typ}, Length={length}")
Output
Packed: b'\x00\x00\x00*@H\xf5\xc3' Length: 8 bytes Unpacked: (42, 3.140000104904175) Int: 42, Float: 3.14 Header: 01000400 Version=1, Type=0, Length=1024

Endianness explained:

import struct

value = 0x01020304  # 16909060 in decimal

# Little-endian (<): least significant byte first
le = struct.pack('<I', value)
print(f"Little-endian: {le.hex(' ')}")  # 04 03 02 01

# Big-endian (>): most significant byte first
be = struct.pack('>I', value)
print(f"Big-endian:    {be.hex(' ')}")  # 01 02 03 04

# Network byte order (!) = big-endian (standard for protocols)
net = struct.pack('!I', value)
print(f"Network:       {net.hex(' ')}")  # 01 02 03 04

# Native (@) = whatever your CPU uses (usually little-endian on x86)
native = struct.pack('@I', value)
print(f"Native:        {native.hex(' ')}")

# Why it matters: reading a file from a big-endian system on a
# little-endian machine gives wrong values if endianness is wrong!
wrong = struct.unpack('<I', be)[0]
right = struct.unpack('>I', be)[0]
print(f"\nWrong endian: {wrong:#010x} ({wrong})")
print(f"Right endian: {right:#010x} ({right})")
Output
Little-endian: 04 03 02 01 Big-endian: 01 02 03 04 Network: 01 02 03 04 Native: 04 03 02 01 Wrong endian: 0x04030201 (67305985) Right endian: 0x01020304 (16909060)

struct.Struct for repeated packing/unpacking:

import struct

# Pre-compile the format for repeated use (faster)
header_fmt = struct.Struct('!BHI')  # type(1), id(2), size(4)

print(f"Header size: {header_fmt.size} bytes")

# Pack several headers
headers = [
    header_fmt.pack(1, 100, 4096),
    header_fmt.pack(2, 101, 8192),
    header_fmt.pack(3, 102, 256),
]

# Unpack them
for i, h in enumerate(headers):
    typ, msg_id, size = header_fmt.unpack(h)
    print(f"  Header {i}: type={typ}, id={msg_id}, size={size}")

# pack_into / unpack_from for buffer manipulation
buffer = bytearray(header_fmt.size * 3)
for i, (typ, mid, sz) in enumerate([(1,100,4096), (2,101,8192), (3,102,256)]):
    header_fmt.pack_into(buffer, i * header_fmt.size, typ, mid, sz)

print(f"\nBuffer: {buffer.hex()}")
# Read back the second header
typ, mid, sz = header_fmt.unpack_from(buffer, header_fmt.size)
print(f"Second header: type={typ}, id={mid}, size={sz}")
Output
Header size: 7 bytes Header 0: type=1, id=100, size=4096 Header 1: type=2, id=101, size=8192 Header 2: type=3, id=102, size=256 Buffer: 010064000010000200650000200003006600000100 Second header: type=2, id=101, size=8192

Reading binary file formats:

import struct
import io

# Simulate a simple binary file format:
# Magic number (4 bytes) + version (2 bytes) + record count (4 bytes)
# followed by records: name_len(1 byte) + name(N bytes) + score(float)

def write_binary(stream, records):
    # Write header
    stream.write(struct.pack('>4sHI', b'MYDB', 1, len(records)))

    # Write records
    for name, score in records:
        name_bytes = name.encode('utf-8')
        stream.write(struct.pack('>B', len(name_bytes)))
        stream.write(name_bytes)
        stream.write(struct.pack('>f', score))

def read_binary(stream):
    # Read header
    magic, version, count = struct.unpack('>4sHI', stream.read(10))
    print(f"Magic: {magic}, Version: {version}, Records: {count}")

    # Read records
    records = []
    for _ in range(count):
        name_len = struct.unpack('>B', stream.read(1))[0]
        name = stream.read(name_len).decode('utf-8')
        score = struct.unpack('>f', stream.read(4))[0]
        records.append((name, score))
    return records

# Write and read back
buf = io.BytesIO()
write_binary(buf, [("Alice", 95.5), ("Bob", 87.3), ("Charlie", 92.1)])

buf.seek(0)
records = read_binary(buf)
for name, score in records:
    print(f"  {name}: {score:.1f}")
Output
Magic: b'MYDB', Version: 1, Records: 3 Alice: 95.5 Bob: 87.3 Charlie: 92.1

memoryview with struct for zero-copy access:

import struct

# Large binary buffer (simulating a network packet stream)
# Each "packet": type(1B) + length(2B) + data(variable)
raw = bytearray()
raw += struct.pack('!BH', 1, 5) + b"Hello"
raw += struct.pack('!BH', 2, 3) + b"Bye"
raw += struct.pack('!BH', 1, 6) + b"World!"

# Parse without copying using memoryview
mv = memoryview(raw)
offset = 0
packets = []

while offset < len(mv):
    ptype, plen = struct.unpack_from('!BH', mv, offset)
    offset += 3
    data = bytes(mv[offset:offset + plen])
    offset += plen
    packets.append((ptype, data.decode()))

for ptype, data in packets:
    print(f"  Type {ptype}: '{data}'")
Output
Type 1: 'Hello' Type 2: 'Bye' Type 1: 'World!'

Common Mistakes

Don't
# Forgetting endianness -- reading big-endian
# data as little-endian gives wrong values
data = struct.pack('>H', 256)  # 0x0100
wrong = struct.unpack('<H', data)[0]
# wrong = 1, not 256!
Do
# Always specify endianness explicitly
data = struct.pack('>H', 256)
correct = struct.unpack('>H', data)[0]
# correct = 256
# Use '!' for network protocols
Don't
# Packing strings directly as 's' without length
struct.pack('s', "hello")
# TypeError: a bytes-like object is required
Do
# Encode to bytes first; specify length in format
struct.pack('5s', b"hello")
# Or use length prefix:
name = b"hello"
struct.pack(f'>H{len(name)}s', len(name), name)

Key Rules

  • Always specify endianness (<, >, !, @). The default native mode (@) adds padding and varies by platform.
  • Use ! (network/big-endian) for anything that crosses machines (network protocols, file formats).
  • Pre-compile formats with struct.Struct(fmt) when packing/unpacking in loops — it's measurably faster.
  • Use struct.calcsize(fmt) to check the byte size of a format before reading.
  • Combine memoryview with unpack_from for efficient parsing of large binary buffers without copying.

Mocking with unittest.mock

Testing & Packaging • New

Why It Matters

Real code depends on external services, databases, filesystems, and time. You can't call a payment API in every test run, and tests shouldn't depend on network availability. unittest.mock lets you replace real objects with controlled fakes, so you can test your logic in isolation. It's the standard library's mocking framework and integrates seamlessly with pytest.

Progressive Explanation

MagicMock basics:

from unittest.mock import MagicMock

# Call it like any function

When to mock (and when NOT to):

# MOCK THESE (external dependencies):
# - HTTP requests (requests, httpx, aiohttp)
# - Database queries (SQLAlchemy sessions, cursor.execute)
# - File system operations (open, pathlib read/write)
# - Time (datetime.now, time.time)
# - Environment variables (os.environ)
# - Third-party API clients

# DON'T MOCK THESE (test the real behavior):
# - Your own data classes and value objects
# - Pure functions with no side effects
# - Standard library data structures (list, dict, set)
# - Simple utility functions

# Anti-pattern: mocking everything
# BAD -- you're testing the mock, not the code!
def test_bad_over_mocking():
    mock_list = MagicMock()
    mock_list.__len__ = MagicMock(return_value=3)
    assert len(mock_list) == 3  # proves nothing!

# GOOD -- only mock the external dependency
def calculate_total(items: list[dict]) -> float:
    return sum(item["price"] * item["quantity"] for item in items)

def test_calculate_total():
    # Use REAL data, no mocking needed for pure functions
    items = [
        {"price": 10.0, "quantity": 2},
        {"price": 5.0, "quantity": 3},
    ]
    assert calculate_total(items) == 35.0

# Demonstrate the real test
items = [{"price": 10.0, "quantity": 2}, {"price": 5.0, "quantity": 3}]
print(f"Total: {calculate_total(items)}")
print("Rule: mock boundaries, test behavior")
Output
Total: 35.0 Rule: mock boundaries, test behavior

Common Mistakes

Don't
# Patching the wrong location
# If user_service.py does: from db import get_user
# You must patch where it's USED, not where it's defined
with patch("db.get_user"):  # WRONG!
    ...
Do
# Patch where the name is looked up
with patch("user_service.get_user"):  # RIGHT!
    ...
# Rule: patch where it's imported, not where it lives
Don't
# Forgetting to assert the mock was called
with patch("requests.get") as mock_get:
    mock_get.return_value.json.return_value = {}
    result = do_something()
    assert result == expected
    # Never checked that requests.get was called!
Do
with patch("requests.get") as mock_get:
    mock_get.return_value.json.return_value = {}
    result = do_something()
    assert result == expected
    mock_get.assert_called_once()  # verify interaction

Key Rules

  • Patch where a name is looked up (imported), not where it's defined. If my_module imports requests, patch "my_module.requests".
  • Use return_value to control what a mock returns. Use side_effect for exceptions, sequences, or custom logic.
  • Always verify mock interactions with assert_called_once_with, assert_has_calls, etc.
  • Mock external boundaries (APIs, databases, filesystem), not internal logic. Over-mocking makes tests brittle and meaningless.
  • Use mock_open for file I/O mocking — it handles __enter__/__exit__ automatically.

Advanced typing

Testing & Packaging • New

Why It Matters

Basic type hints cover most cases, but real-world code needs generic classes, typed decorators, function overloads, and type narrowing. These advanced features let type checkers (mypy, pyright) understand complex patterns: decorators that preserve signatures, containers that work with any type, and functions with different return types based on input. This turns Python's type system from "nice annotations" into a powerful correctness tool.

Progressive Explanation

TypeVar for generic functions:

from typing import TypeVar, Sequence

# TypeVar creates a placeholder type
T = TypeVar("T")

def first(items: Sequence[T]) -> T:
    """Returns the first item. Type checker knows return type matches input."""
    return items[0]

# Type checker infers:
name = first(["Alice", "Bob"])    # name: str
num = first([1, 2, 3])            # num: int
print(f"First name: {name}")
print(f"First num: {num}")

# Bound TypeVar -- restricts to subclasses
from typing import SupportsFloat

N = TypeVar("N", bound=SupportsFloat)

def double(x: N) -> N:
    """Works with any numeric type."""
    return type(x)(float(x) * 2)  # type: ignore

print(f"double(5) = {double(5)}")
print(f"double(3.14) = {double(3.14)}")

# Constrained TypeVar -- only specific types allowed
StrOrBytes = TypeVar("StrOrBytes", str, bytes)

def concat(a: StrOrBytes, b: StrOrBytes) -> StrOrBytes:

Generic[T] for generic classes:

from typing import TypeVar, Generic, Iterator

T = TypeVar("T")

class Stack(Generic[T]):
    """A type-safe stack."""

    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        if not self._items:
            raise IndexError("pop from empty stack")
        return self._items.pop()

    def peek(self) -> T:
        if not self._items:
            raise IndexError("peek at empty stack")
        return self._items[-1]

    def __len__(self) -> int:
        return len(self._items)

    def __iter__(self) -> Iterator[T]:
        return reversed(self._items)

# Usage with specific types
int_stack: Stack[int] = Stack()
int_stack.push(1)
int_stack.push(2)
int_stack.push(3)
# int_stack.push("hello")  # type error caught by mypy!

str_stack: Stack[str] = Stack()
str_stack.push("hello")
str_stack.push("world")

print(f"Int stack top: {int_stack.peek()}")
print(f"Str stack top: {str_stack.peek()}")
print(f"Int stack items: {list(int_stack)}")

# Multiple type parameters
K = TypeVar("K")
V = TypeVar("V")

class Pair(Generic[K, V]):
    def __init__(self, key: K, value: V) -> None:
        self.key = key
        self.value = value

    def __repr__(self) -> str:
        return f"Pair({self.key!r}, {self.value!r})"

p = Pair("name", 42)
print(p)  # Pair('name', 42)
Output
Int stack top: 3 Str stack top: world Int stack items: [3, 2, 1] Pair('name', 42)

ParamSpec for preserving decorator signatures:

from typing import TypeVar, ParamSpec, Callable
import functools
import time

P = ParamSpec("P")
R = TypeVar("R")

def timer(func: Callable[P, R]) -> Callable[P, R]:
    """Decorator that preserves the original function's type signature."""
    @functools.wraps(func)
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"  {func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def greet(name: str, excited: bool = False) -> str:
    """Greet someone."""
    msg = f"Hello, {name}!"
    return msg.upper() if excited else msg

# Type checker knows: greet(name: str, excited: bool = False) -> str
result = greet("Alice", excited=True)
print(result)

# Without ParamSpec, the decorator would lose the signature:
# def bad_timer(func):
#     @functools.wraps(func)
#     def wrapper(*args, **kwargs):  # type checker sees (*Any, **Any) -> Any
#         ...
#     return wrapper
Output
greet took 0.0000s HELLO, ALICE!

@overload for typed function overloads:

from typing import overload, Literal

# @overload tells the type checker about different signatures
# Only the implementation (without @overload) actually runs

@overload
def process(data: str) -> list[str]: ...
@overload
def process(data: bytes) -> list[int]: ...
@overload
def process(data: list[int]) -> int: ...

def process(data):
    """Actual implementation handles all cases."""
    if isinstance(data, str):
        return data.split()
    elif isinstance(data, bytes):
        return list(data)
    elif isinstance(data, list):
        return sum(data)

# Type checker knows the exact return type for each input type:
words = process("hello world")       # list[str]
byte_vals = process(b"ABC")          # list[int]
total = process([1, 2, 3])           # int

print(f"str -> {words}")
print(f"bytes -> {byte_vals}")
print(f"list -> {total}")

# @overload with Literal for flag-dependent return types
@overload
def fetch(url: str, *, raw: Literal[True]) -> bytes: ...
@overload
def fetch(url: str, *, raw: Literal[False] = ...) -> str: ...

def fetch(url, *, raw=False):
    data = f"Response from {url}"
    return data.encode() if raw else data

text = fetch("https://example.com")          # str
binary = fetch("https://example.com", raw=True)  # bytes
print(f"Text type: {type(text).__name__}")
print(f"Binary type: {type(binary).__name__}")
Output
str -> ['hello', 'world'] bytes -> [65, 66, 67] list -> 6 Text type: str Binary type: bytes

TypeGuard for type narrowing:

from typing import TypeGuard, Any

def is_string_list(val: list[Any]) -> TypeGuard[list[str]]:
    """Type guard: if True, type checker narrows val to list[str]."""
    return all(isinstance(item, str) for item in val)

def is_int_or_none(val: Any) -> TypeGuard[int | None]:
    return val is None or isinstance(val, int)

# Usage
data: list[Any] = ["hello", "world", "python"]

if is_string_list(data):
    # Type checker now knows data is list[str]
    print(f"Joined: {', '.join(data)}")  # .join needs list[str]
else:
    print("Not all strings")

# Without TypeGuard, you'd need explicit casts or ignores:
# data_str: list[str] = cast(list[str], data)  # unsafe cast!

# Practical example: validating API responses
def is_valid_user(data: dict[str, Any]) -> TypeGuard[dict[str, str | int]]:
    required = {"name", "age", "email"}
    return (
        required.issubset(data.keys())
        and isinstance(data.get("name"), str)
        and isinstance(data.get("age"), int)
        and isinstance(data.get("email"), str)
    )

response: dict[str, Any] = {"name": "Alice", "age": 30, "email": "a@b.com"}
if is_valid_user(response):
    print(f"Valid user: {response['name']}, age {response['age']}")
Output
Joined: hello, world, python Valid user: Alice, age 30

Concatenate for adding parameters:

from typing import TypeVar, ParamSpec, Callable, Concatenate
import functools

P = ParamSpec("P")
R = TypeVar("R")

def with_logging(
    func: Callable[Concatenate[str, P], R]
) -> Callable[Concatenate[str, P], R]:
    """Decorator for functions whose first arg is a log prefix."""
    @functools.wraps(func)
    def wrapper(prefix: str, *args: P.args, **kwargs: P.kwargs) -> R:
        print(f"[{prefix}] Calling {func.__name__}")
        result = func(prefix, *args, **kwargs)
        print(f"[{prefix}] Done")
        return result
    return wrapper

@with_logging
def save_data(log_prefix: str, data: dict, path: str) -> bool:
    print(f"  Saving {data} to {path}")
    return True

# Type checker knows: save_data(log_prefix: str, data: dict, path: str) -> bool
result = save_data("DB", {"name": "Alice"}, "/tmp/data.json")
print(f"Saved: {result}")

# Concatenate adds/requires specific parameters at the front
# while preserving the rest of the signature via ParamSpec
Output
[DB] Calling save_data Saving {'name': 'Alice'} to /tmp/data.json [DB] Done Saved: True

Common Mistakes

Don't
from typing import Any, Callable

# Losing type information in decorators
def decorator(func: Callable[..., Any]):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper
# wrapper has type (*args, **kwargs) -> Any
Do
from typing import ParamSpec, TypeVar, Callable

P = ParamSpec("P")
R = TypeVar("R")
def decorator(func: Callable[P, R]) -> Callable[P, R]:
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        return func(*args, **kwargs)
    return wrapper
# Signature fully preserved!
Don't
# Using isinstance checks without TypeGuard
def process(items: list[Any]) -> str:
    if all(isinstance(i, str) for i in items):
        return ", ".join(items)  # type error: list[Any]
    return ""
Do
def is_str_list(v: list[Any]) -> TypeGuard[list[str]]:
    return all(isinstance(i, str) for i in v)

def process(items: list[Any]) -> str:
    if is_str_list(items):
        return ", ".join(items)  # OK: list[str]
    return ""

Key Rules

  • Use TypeVar for generic functions. Use bound= to restrict to subclasses; use constraints for a fixed set of types.
  • Inherit from Generic[T] to create generic classes that work like list[int] or dict[str, Any].
  • Use ParamSpec in decorators to preserve the decorated function's complete type signature.
  • @overload defines multiple signatures for the type checker; only the final implementation (without @overload) actually executes.
  • TypeGuard narrows types inside if blocks — use it to replace unsafe cast() calls with runtime checks.
