Type Hints — Optional, Union & Modern Syntax

Type System • 4 days since last review

Quick Reminder

Optional[X] is shorthand for Union[X, None]. Python 3.10+ supports X | Y instead of Union[X, Y]. Use Optional for parameters that can be None, never as a default argument marker. TypeAlias names complex types for readability.

Key Pattern

from typing import Optional, Union

# Pre-3.10 style
def find_user(user_id: int) -> Optional[str]:
    """Returns username or None if not found."""
    users = {1: "Alice", 2: "Bob"}
    return users.get(user_id)

# Python 3.10+ style (preferred)
def parse_value(raw: str) -> int | float | None:
    try:
        return int(raw)
    except ValueError:
        try:
            return float(raw)
        except ValueError:
            return None

print(find_user(1))       # Alice
print(find_user(99))      # None
print(parse_value("3.14"))  # 3.14
Challenge

Write a fully type-annotated function safe_get(data: dict[str, Any], path: str, default: T = None) -> T | Any that accepts a dot-separated path like "user.address.city" and traverses nested dicts. Use TypeVar for the default and return type.

#1 Pitfall: Writing def f(x: Optional[int] = None) and thinking Optional means "optional parameter." It doesn't — it means "this type or None." The = None default is what makes the parameter optional.

Classes — __eq__ and __hash__

OOP • 4 days since last review

Quick Reminder

__eq__ defines equality (==). Defining __eq__ without __hash__ makes your class unhashable (can't use in sets/dict keys). __hash__ must return the same value for objects that are equal. Use hash((self.x, self.y)) for a tuple of immutable fields.

Key Pattern

class Point:
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y

    def __eq__(self, other) -> bool:
        if not isinstance(other, Point):
            return NotImplemented
        return self.x == other.x and self.y == other.y

    def __hash__(self) -> int:
        return hash((self.x, self.y))

    def __repr__(self) -> str:
        return f"Point({self.x}, {self.y})"

p1 = Point(1, 2)
p2 = Point(1, 2)
p3 = Point(3, 4)
print(p1 == p2)              # True
print(p1 == p3)              # False
print({p1, p2, p3})          # {Point(1, 2), Point(3, 4)}
print(len({p1, p2, p3}))     # 2 -- p1 and p2 are equal
Challenge

Create a Color class with r, g, b (0-255) that is hashable and supports equality. Use it as dictionary keys mapping colors to names. Verify that Color(255, 0, 0) == Color(255, 0, 0) and they map to the same dict value.

#1 Pitfall: Defining __eq__ and forgetting __hash__. Python sets __hash__ = None when you define __eq__, making instances unhashable. Always pair them or use @dataclass(frozen=True).

Properties & @property

OOP • 3 days since last review

Quick Reminder

@property turns a method into a read-only attribute. Use @name.setter for write access with validation. Properties let you add logic (validation, computation) without changing the API. Computed properties avoid storing redundant data.

Key Pattern

class Circle:
    def __init__(self, radius: float):
        self.radius = radius  # triggers setter

    @property
    def radius(self) -> float:
        return self._radius

    @radius.setter
    def radius(self, value: float):
        if value < 0:
            raise ValueError("Radius cannot be negative")
        self._radius = value

    @property
    def area(self) -> float:
        import math
        return math.pi * self._radius ** 2

c = Circle(5)
print(c.radius)  # 5
print(f"{c.area:.2f}")  # 78.54
c.radius = 10
print(f"{c.area:.2f}")  # 314.16
Challenge

Build a Temperature class where you can set either .celsius or .fahrenheit and the other property updates automatically. Both should have setters. Add a .kelvin read-only property.

#1 Pitfall: Infinite recursion — self.radius = value inside the setter calls itself if you don't use a private _radius. Always store the actual value in a different attribute (by convention, prefixed with _).

__slots__

OOP • 3 days since last review

Quick Reminder

__slots__ replaces the per-instance __dict__ with a fixed set of attributes, saving memory (~40-50% per instance) and preventing typos. Define __slots__ = ("x", "y") as a tuple of allowed attribute names. No new attributes can be added at runtime.

Key Pattern

class Point:
    __slots__ = ("x", "y")

    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y

p = Point(1, 2)
print(p.x, p.y)   # 1 2
# p.z = 3  # AttributeError: 'Point' has no attribute 'z'

# Memory comparison
import sys
class DictPoint:
    def __init__(self, x, y):
        self.x = x
        self.y = y

print(sys.getsizeof(DictPoint(1, 2).__dict__))  # ~104 bytes
# Point with __slots__ has no __dict__ at all
Challenge

Create a __slots__-based Vector3D class with x, y, z. Add __add__, __sub__, and a magnitude property. Create 100,000 instances and compare memory usage against a non-slots version using sys.getsizeof.

#1 Pitfall: Inheriting from a __slots__ class without defining __slots__ in the subclass. The subclass gets a __dict__ anyway, defeating the purpose. Always define __slots__ in every class in the hierarchy.

Class Methods & Static Methods

OOP • 3 days since last review

Quick Reminder

@classmethod receives the class as first argument (cls) — use for alternative constructors. @staticmethod receives neither self nor cls — use for utility functions logically grouped with the class. Class methods support polymorphism; static methods don't.

Key Pattern

class Date:
    def __init__(self, year: int, month: int, day: int):
        self.year = year
        self.month = month
        self.day = day

    @classmethod
    def from_string(cls, date_str: str) -> "Date":
        y, m, d = map(int, date_str.split("-"))
        return cls(y, m, d)

    @classmethod
    def today(cls) -> "Date":
        import datetime
        t = datetime.date.today()
        return cls(t.year, t.month, t.day)

    @staticmethod
    def is_valid(year: int, month: int, day: int) -> bool:
        return 1 <= month <= 12 and 1 <= day <= 31

d = Date.from_string("2026-06-04")
print(f"{d.year}-{d.month:02d}-{d.day:02d}")  # 2026-06-04
print(Date.is_valid(2026, 13, 1))  # False
Challenge

Create a Money class with amount and currency. Add classmethods from_string("$42.50") and from_cents(4250, "USD"). Add a staticmethod convert_rate(from_curr, to_curr) that returns a hardcoded exchange rate. Test polymorphism by subclassing Money as CryptoMoney.

#1 Pitfall: Using @staticmethod when you actually need @classmethod. If a subclass overrides the method and you want it to return an instance of the subclass (not the parent), you need cls — which only @classmethod provides.

Inheritance & MRO

OOP • 3 days since last review

Quick Reminder

Python uses C3 linearization for Method Resolution Order (MRO). Call ClassName.mro() or ClassName.__mro__ to see the order. super() follows the MRO, not just the parent class. In diamond inheritance, MRO ensures each class appears only once.

Key Pattern

class Animal:
    def speak(self) -> str:
        return "..."

class Dog(Animal):
    def speak(self) -> str:
        return "Woof!"

class GuideDog(Dog):
    def speak(self) -> str:
        return f"{super().speak()} (gentle)"

# Diamond inheritance
class A:
    def method(self):
        print("A.method")

class B(A):
    def method(self):
        print("B.method")
        super().method()

class C(A):
    def method(self):
        print("C.method")
        super().method()

class D(B, C):
    def method(self):
        print("D.method")
        super().method()

d = D()
d.method()
# D.method -> B.method -> C.method -> A.method
print(D.mro())
# [D, B, C, A, object]
Challenge

Create a mixin-based design: LoggableMixin (adds .log(msg)), SerializableMixin (adds .to_dict()/.from_dict()), and a User class that inherits from both mixins plus a Base class. Print the MRO and verify all methods work.

#1 Pitfall: Assuming super() always calls the immediate parent. In diamond inheritance, it follows the MRO — which may skip to a sibling class. Always check ClassName.mro() to understand the call chain.

argparse for CLI Tools

Error Handling & I/O • New

Why It Matters

argparse is Python's built-in module for building professional command-line interfaces. Instead of manually parsing sys.argv, you declare what arguments your program accepts and argparse handles validation, help messages, type conversion, and error reporting automatically. Every Python CLI tool uses it (or a library built on top of it).

Progressive Explanation

Basic ArgumentParser — positional and optional args:

import argparse

# Create parser
parser = argparse.ArgumentParser(
    description="A simple greeting tool"
)

# Positional argument (required)
parser.add_argument("name", help="Name of the person to greet")

# Optional arguments (prefixed with --)
parser.add_argument("-c", "--count", type=int, default=1,
                    help="Number of times to greet (default: 1)")
parser.add_argument("-u", "--uppercase", action="store_true",
                    help="Print greeting in uppercase")

# Parse arguments
args = parser.parse_args(["Alice", "--count", "3", "--uppercase"])
print(args)
# Namespace(name='Alice', count=3, uppercase=True)
print(args.name)       # Alice
print(args.count)      # 3
print(args.uppercase)  # True

greeting = f"Hello, {args.name}!"
if args.uppercase:
    greeting = greeting.upper()
for _ in range(args.count):
    print(greeting)
Output
Namespace(name='Alice', count=3, uppercase=True) Alice 3 True HELLO, ALICE! HELLO, ALICE! HELLO, ALICE!

Type conversion, choices, and nargs:

import argparse

parser = argparse.ArgumentParser(description="File processor")

# type= converts the string argument automatically
parser.add_argument("--port", type=int, default=8080,
                    help="Port number (default: 8080)")

# choices= restricts valid values
parser.add_argument("--format", choices=["json", "csv", "xml"],

# nargs= controls how many values an argument consumes

                    help="Start and end values")

# required= makes an optional argument mandatory
parser.add_argument("--output", "-o", required=True,
                    help="Output file path")

args = parser.parse_args([
    "--port", "3000",
    "--format", "csv",
    "--files", "a.txt", "b.txt", "c.txt",
    "--range", "1", "100",
    "--output", "result.csv"
])

print(f"Port: {args.port}")
print(f"Format: {args.format}")
print(f"Files: {args.files}")
print(f"Range: {args.range}")
print(f"Output: {args.output}")

Port: 3000 Format: csv Files: ['a.txt', 'b.txt', 'c.txt'] Range: [1, 100] Output: result.csv

Common Mistakes

Don't
# Manual sys.argv parsing
import sys
if len(sys.argv) < 2:
    print("Usage: prog <name>")
    sys.exit(1)
name = sys.argv[1]
# No help, no validation, error-prone
Do
import argparse
parser = argparse.ArgumentParser()
parser.add_argument("name")
args = parser.parse_args()
# Auto help (-h), validation, error messages
Don't
# Forgetting type= for numeric arguments
parser.add_argument("--port")
args = parser.parse_args(["--port", "8080"])
# args.port is "8080" (a string!)
Do
parser.add_argument("--port", type=int)
args = parser.parse_args(["--port", "8080"])
# args.port is 8080 (an integer)

Key Rules

  • Arguments with -- prefix are optional; without prefix they are positional (required by default).
  • Always specify type=int or type=float for numeric args — all arguments are strings by default.
  • Use action="store_true" for boolean flags; they default to False and become True when present.
  • Use subparsers for tools with multiple commands (like git commit, git push).

threading vs multiprocessing vs asyncio

Concurrency & Performance • New

Why It Matters

Most programs wait — for network responses, disk reads, user input. Concurrency lets you do useful work while waiting. Python offers three models: threading (shared memory, limited by GIL), multiprocessing (separate processes, true parallelism), and asyncio (cooperative, single-threaded). Choosing the right one is critical for performance.

Progressive Explanation

The GIL — why it matters:

# The Global Interpreter Lock (GIL) allows only ONE thread to execute
# Python bytecode at a time. This means:
#
# - CPU-bound work: threads DON'T help (they take turns, no speedup)
# - I/O-bound work: threads DO help (they release GIL while waiting)
#
# The GIL does NOT affect:
# - multiprocessing (separate processes, separate GILs)
# - C extensions that release GIL (NumPy, etc.)
# - asyncio (single-threaded, so GIL is irrelevant)

import threading
import time

counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1  # NOT thread-safe! (race condition)

# Demonstration of GIL's effect on CPU-bound work
start = time.perf_counter()
t1 = threading.Thread(target=increment, args=(1_000_000,))
t2 = threading.Thread(target=increment, args=(1_000_000,))
t1.start()
t2.start()
t1.join()
t2.join()
elapsed = time.perf_counter() - start
print(f"Counter: {counter} (expected 2000000)")
print(f"Elapsed: {elapsed:.3f}s")
# Counter will likely be LESS than 2000000 due to race condition!
Output (varies)
Counter: 1487236 (expected 2000000) Elapsed: 0.152s

threading — for I/O-bound tasks:

import threading
import time

def download(url: str, delay: float):
    """Simulate downloading a URL."""
    print(f"[{threading.current_thread().name}] Starting {url}")
    time.sleep(delay)  # simulates I/O wait (releases GIL)
    print(f"[{threading.current_thread().name}] Finished {url}")
    return f"Data from {url}"

# Sequential -- slow
start = time.perf_counter()
for url, delay in [("page1", 1), ("page2", 1), ("page3", 1)]:
    download(url, delay)
print(f"Sequential: {time.perf_counter() - start:.1f}s\n")

# Threaded -- fast for I/O
start = time.perf_counter()
threads = []
for url, delay in [("page1", 1), ("page2", 1), ("page3", 1)]:
    t = threading.Thread(target=download, args=(url, delay), name=f"Worker-{url}")
    threads.append(t)
    t.start()

for t in threads:
    t.join()  # wait for all threads to finish
print(f"Threaded: {time.perf_counter() - start:.1f}s")
Output
[MainThread] Starting page1 [MainThread] Finished page1 [MainThread] Starting page2 [MainThread] Finished page2 [MainThread] Starting page3 [MainThread] Finished page3 Sequential: 3.0s [Worker-page1] Starting page1 [Worker-page2] Starting page2 [Worker-page3] Starting page3 [Worker-page1] Finished page1 [Worker-page2] Finished page2 [Worker-page3] Finished page3 Threaded: 1.0s

multiprocessing — for CPU-bound tasks:

import multiprocessing
import time

def cpu_work(n: int) -> int:
    """CPU-intensive computation."""
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    N = 5_000_000

    # Sequential
    start = time.perf_counter()
    results = [cpu_work(N), cpu_work(N), cpu_work(N)]
    print(f"Sequential: {time.perf_counter() - start:.2f}s")

    # Multiprocessing -- true parallelism
    start = time.perf_counter()
    with multiprocessing.Pool(3) as pool:
        results = pool.map(cpu_work, [N, N, N])
    print(f"Multiprocessing: {time.perf_counter() - start:.2f}s")
    print(f"Results match: {results[0] == results[1] == results[2]}")
Output (example, depends on CPU)
Sequential: 3.45s Multiprocessing: 1.23s Results match: True

Quick decision guide:

# WHEN TO USE EACH:
#
# +------------------+------------------+------------------+
# |   Task Type      |   Best Choice    |   Why            |
# +------------------+------------------+------------------+
# | I/O-bound        | asyncio          | lowest overhead  |
# | (network, disk)  | or threading     | GIL released     |
# +------------------+------------------+------------------+
# | CPU-bound        | multiprocessing  | bypasses GIL     |
# | (math, parsing)  |                  | true parallelism |
# +------------------+------------------+------------------+
# | Mixed I/O + CPU  | multiprocessing  | separate proc.   |
# |                  | + asyncio inside | best of both     |
# +------------------+------------------+------------------+
# | Simple scripts   | threading        | easiest to add   |
# | (few I/O waits)  |                  | shared memory    |
# +------------------+------------------+------------------+

# KEY DIFFERENCES:
# threading:        shared memory, GIL-limited, low overhead
# multiprocessing:  separate memory, true parallel, high overhead
# asyncio:          single-threaded, cooperative, very low overhead

Common Mistakes

Don't
# Using threading for CPU-bound work
# Threads take turns due to GIL -- no speedup!
threads = [threading.Thread(target=cpu_work, args=(N,))
           for _ in range(4)]
Do
# Use multiprocessing for CPU-bound work
with multiprocessing.Pool(4) as pool:
    results = pool.map(cpu_work, [N]*4)
# True parallel execution on multiple cores
Don't
# Sharing mutable state between threads without locks
counter = 0
def increment():
    global counter
    counter += 1  # race condition!
Do
lock = threading.Lock()
counter = 0
def increment():
    global counter
    with lock:  # thread-safe
        counter += 1

Key Rules

  • I/O-bound (waiting for network, disk) → asyncio or threading. CPU-bound (number crunching) → multiprocessing.
  • The GIL prevents multiple threads from running Python code simultaneously. It does NOT prevent concurrency for I/O.
  • Always protect shared mutable state with threading.Lock() (or avoid shared state entirely).
  • multiprocessing requires if __name__ == "__main__": guard on Windows/macOS.

async/await Basics

Concurrency & Performance • New

Why It Matters

asyncio is Python's modern framework for concurrent I/O. It handles thousands of network connections efficiently in a single thread. Web frameworks (FastAPI, aiohttp), database drivers, and API clients all use async/await. It's become essential for backend Python development.

Progressive Explanation

Coroutines and asyncio.run:

import asyncio

# 'async def' creates a coroutine function
async def greet(name: str, delay: float) -> str:
    print(f"Start greeting {name}")
    await asyncio.sleep(delay)  # non-blocking sleep
    print(f"Done greeting {name}")
    return f"Hello, {name}!"

# asyncio.run() starts the event loop and runs a coroutine
result = asyncio.run(greet("Alice", 1))
print(result)
Output
Start greeting Alice Done greeting Alice Hello, Alice!

asyncio.gather — running coroutines concurrently:

import asyncio
import time

async def fetch(url: str, delay: float) -> str:
    print(f"Fetching {url}...")
    await asyncio.sleep(delay)  # simulates network I/O

Common Mistakes

Don't
# Calling a coroutine without await
async def fetch():
    return "data"

async def main():
    result = fetch()  # returns coroutine object!
    print(result)  # <coroutine object fetch>
    # RuntimeWarning: coroutine never awaited
Do
async def main():
    result = await fetch()  # actually runs it
    print(result)  # "data"
Don't
# Using time.sleep in async code
async def handler():
    time.sleep(5)  # BLOCKS entire event loop!
    # No other coroutine can run during this
Do
async def handler():
    await asyncio.sleep(5)  # non-blocking
    # Other coroutines run during the wait

Key Rules

  • async def creates a coroutine; you must await it to get the result (or schedule with create_task).
  • Never use time.sleep() in async code — it blocks the entire event loop. Use asyncio.sleep() instead.
  • gather() runs coroutines concurrently and returns results in order. create_task() schedules without waiting.
  • Use asyncio.Semaphore to limit the number of concurrent operations (e.g., max 10 HTTP connections).

concurrent.futures

Concurrency & Performance • New

Why It Matters

concurrent.futures provides a high-level interface for running tasks in threads or processes with identical APIs. It's simpler than raw threading/multiprocessing, handles thread/process pools automatically, and gives you Future objects to track results. It's the pragmatic choice for most real-world concurrency needs.

Progressive Explanation

ThreadPoolExecutor — the simple way to parallelize I/O:

from concurrent.futures import ThreadPoolExecutor
import time

def download(url: str) -> str:
    """Simulate downloading a URL."""
    time.sleep(1)  # simulate network delay
    return f"Content of {url}"

urls = ["page1.html", "page2.html", "page3.html", "page4.html"]

# Using 'with' ensures cleanup of threads
start = time.perf_counter()
with ThreadPoolExecutor(max_workers=4) as executor:
    # map() applies function to each item, returns results in order
    results = list(executor.map(download, urls))

elapsed = time.perf_counter() - start
print(f"Downloaded {len(results)} pages in {elapsed:.1f}s")
for r in results:
    print(f"  {r}")
Output
Downloaded 4 pages in 1.0s Content of page1.html Content of page2.html Content of page3.html Content of page4.html

submit() and Future objects:

from concurrent.futures import ThreadPoolExecutor
import time

def process(item: str) -> dict:
    time.sleep(0.5)
    return {"item": item, "length": len(item), "upper": item.upper()}

with ThreadPoolExecutor(max_workers=3) as executor:
    # submit() returns a Future immediately
    future1 = executor.submit(process, "hello")
    future2 = executor.submit(process, "world")

    print(f"future1 done? {future1.done()}")  # probably False
    print(f"future2 done? {future2.done()}")  # probably False

    # .result() blocks until the result is ready
    result1 = future1.result()
    result2 = future2.result()
    print(f"Result 1: {result1}")
    print(f"Result 2: {result2}")

    # .result(timeout=) raises TimeoutError if not ready
    future3 = executor.submit(process, "timeout test")
    try:
        result3 = future3.result(timeout=0.1)
    except TimeoutError:
        print("future3 not ready yet!")
Output
future1 done? False future2 done? False Result 1: {'item': 'hello', 'length': 5, 'upper': 'HELLO'} Result 2: {'item': 'world', 'length': 5, 'upper': 'WORLD'} future3 not ready yet!

as_completed() — process results as they arrive:

from concurrent.futures import ThreadPoolExecutor, as_completed
import time
import random

def fetch_data(source: str) -> tuple[str, float]:
    delay = random.uniform(0.5, 2.0)
    time.sleep(delay)
    return source, delay

sources = ["database", "cache", "api", "file_system", "queue"]

with ThreadPoolExecutor(max_workers=5) as executor:
    # Submit all tasks and track which future belongs to which source
    future_to_source = {
        executor.submit(fetch_data, src): src
        for src in sources
    }

    # Process results as they complete (NOT in submission order)
    for future in as_completed(future_to_source):
        source = future_to_source[future]
        try:
            name, delay = future.result()
            print(f"  {name} completed in {delay:.2f}s")
        except Exception as e:
            print(f"  {source} failed: {e}")
Output (order varies)
cache completed in 0.52s file_system completed in 0.78s queue completed in 1.15s database completed in 1.43s api completed in 1.87s

from concurrent.futures import ProcessPoolExecutor

Common Mistakes

Don't

result = future.result()  # raises if function raised!
# Unhandled exception crashes your program
Do
future = executor.submit(risky_function)
try:
    result = future.result()
except Exception as e:
    print(f"Task failed: {e}")
    # handle gracefully
Don't
# Creating too many workers
# 1000 threads = huge memory overhead

    ...
Do
# Use sensible defaults
# ThreadPool: min(32, os.cpu_count() + 4) by default
# ProcessPool: os.cpu_count() by default
with ThreadPoolExecutor() as executor:  # use defaults
    ...

Key Rules

  • Use executor.map() for simple "apply function to each item" — results come back in input order.
  • Use executor.submit() + as_completed() when you need results as they finish or per-task error handling.
  • Always use executors as context managers (with) to ensure proper cleanup of threads/processes.
  • Swap ThreadPoolExecutor for ProcessPoolExecutor with zero code changes when switching from I/O-bound to CPU-bound.

import cProfile
import pstats
import io

def bubble_sort(arr):
    arr = arr.copy()
    n = len(arr)
    for i in range(n):
        for j in range(0, n-i-1):
            if arr[j] > arr[j+1]:
                arr[j], arr[j+1] = arr[j+1], arr[j]
    return arr

def insertion_sort(arr):
    arr = arr.copy()
    for i in range(1, len(arr)):
        key = arr[i]
        j = i - 1
        while j >= 0 and arr[j] > key:
            arr[j+1] = arr[j]
            j -= 1
        arr[j+1] = key
    return arr

import cProfile

def find_primes(limit: int) -> list[int]:
    """Sieve of Eratosthenes."""
    is_prime = [True] * (limit + 1)
    is_prime[0] = is_prime[1] = False

    for i in range(2, int(limit**0.5) + 1):
        if is_prime[i]:
            for j in range(i*i, limit + 1, i):
                is_prime[j] = False

    return [i for i in range(limit + 1) if is_prime[i]]

def analyze_primes(limit: int):
    primes = find_primes(limit)
    gaps = [primes[i+1] - primes[i] for i in range(len(primes)-1)]
    max_gap = max(gaps)
    avg_gap = sum(gaps) / len(gaps)
    return {
        "count": len(primes),
        "largest": primes[-1],
        "max_gap": max_gap,
        "avg_gap": round(avg_gap, 2)
    }

# Profile it
cProfile.run("analyze_primes(1_000_000)")
print(analyze_primes(1_000_000))

Common Mistakes

Don't
# Using time.time() for benchmarking
import time
start = time.time()
# ... code ...
print(time.time() - start)
# Low precision, affected by system load
Do
import timeit
# Runs code many times, uses best timer
t = timeit.timeit("code", number=10_000)
# Or use time.perf_counter() for single runs
import time
start = time.perf_counter()  # high precision
Don't
# Optimizing without measuring first
# "I think this is slow, let me rewrite it"
# Wasted hours on code that wasn't the bottleneck
Do
# Profile first, optimize second
cProfile.run("my_function()")
# THEN optimize the functions with highest tottime
# Re-profile to verify improvement

Key Rules

  • Use timeit for micro-benchmarks (comparing two approaches). Use cProfile for finding bottlenecks in large programs.
  • Always use timeit.repeat() and take the minimum — higher values include OS scheduling overhead.
  • In cProfile output, tottime shows where CPU time is actually spent; cumtime shows the full call chain cost.
  • Profile before optimizing. After optimizing, profile again to verify the improvement.
