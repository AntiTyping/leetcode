Review Section (6 items)

1. collections.namedtuple 1 day overdue rep 3 → 4

A namedtuple is a lightweight, immutable tuple subclass with named fields: same memory footprint as a plain tuple, but self-documenting with .field access alongside integer indexing. Ideal for small records where a full class would be overkill.

Key Pattern
from collections import namedtuple

Point = namedtuple("Point", ["x", "y"])
p = Point(1, 2)
print(p.x, p[0])          # 1 1  -- by name OR by index
print(p._asdict())        # {'x': 1, 'y': 2}
q = p._replace(y=9)       # returns a NEW instance (immutable)
print(q)                  # Point(x=1, y=9)

Vec = namedtuple("Vec", "x y z", defaults=[0])   # defaults bind rightmost
print(Vec(1, 2))            # Vec(x=1, y=2, z=0)
Challenge

Define a Color namedtuple with fields r, g, b and a default alpha=255. Unpack an instance into four variables, convert it to a dict with _asdict(), and produce a copy with a changed alpha via _replace — proving the original is untouched.

Pitfall

namedtuples are immutable — _replace builds a new instance and does not mutate. defaults apply to the rightmost fields. Do not confuse it with typing.NamedTuple (class-based syntax with type hints, same runtime object).

2. Type Hints — Basic Annotations due today rep 3 → 4

Annotations attach types to parameters, return values, and variables (def f(x: int) -> str). They are not enforced at runtime — they are stored in __annotations__ and consumed by tools like mypy and IDEs. Prefer the built-in generics list[int]/dict[str, int] (3.9+).

Key Pattern
def greet(name: str, times: int = 1) -> str:
    return (name + " ") * times

count: int = 0
scores: list[int] = []           # built-in generic (3.9+)
mapping: dict[str, float] = {}

print(greet.__annotations__)
# {'name': <class 'str'>, 'times': <class 'int'>, 'return': <class 'str'>}
print(greet(7, 2))       # still RUNS despite wrong type -- not enforced
Challenge

Annotate a function that takes a list[str] and a dict[str, int] and returns a list[tuple[str, int]]. Add a module-level annotated variable, inspect __annotations__, then call it with a deliberately wrong type and confirm it still runs (annotations do not validate).

Pitfall

Annotations never validate at runtime — passing a str where an int is hinted runs fine; only a checker complains. Use built-in list[int]/dict[str, int] for new code rather than the deprecated typing.List/typing.Dict.

3. Classes — __init__, __repr__, __str__ due today rep 3 → 4

__init__ initializes a freshly created instance. __repr__ is the unambiguous developer string (ideally eval-able); __str__ is the friendly user string. print/str() use __str__ (falling back to __repr__); the REPL and containers always use __repr__.

Key Pattern
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __repr__(self):
        return f"Point({self.x!r}, {self.y!r})"   # !r quotes strings
    def __str__(self):
        return f"({self.x}, {self.y})"

p = Point(1, 2)
print(str(p))    # (1, 2)         -- __str__
print(repr(p))   # Point(1, 2)    -- __repr__
print([p])      # [Point(1, 2)]  -- containers use __repr__
Challenge

Write a Money class that stores integer cents. Give it a __repr__ that round-trips (Money(cents=500)) and a __str__ that renders $5.00. Put several in a list and confirm the list display uses __repr__, while print(money) uses __str__.

Pitfall

If you define only __str__, the REPL and containers still show the ugly default <Point object at 0x...> — always define __repr__. Use !r inside __repr__ so nested strings appear quoted and the output is unambiguous.

4. Class Methods & Static Methods 12 days overdue rep 2 → 3

@classmethod receives the class as cls — perfect for alternative constructors. @staticmethod receives neither self nor cls — just a plain function living in the class namespace. A regular method receives the instance as self.

Key Pattern
class Pizza:
    def __init__(self, radius, ingredients):
        self.radius, self.ingredients = radius, ingredients

    @classmethod
    def margherita(cls):
        return cls(12, ["mozzarella", "tomato"])   # alt constructor via cls

    @staticmethod
    def area(r):
        return 3.14159 * r * r                     # no self / no cls

p = Pizza.margherita()
print(p.ingredients)      # ['mozzarella', 'tomato']
print(Pizza.area(10))     # 314.159
Challenge

Add a from_dict(cls, d) classmethod that builds an instance from a dict and a @staticmethod is_valid(radius) helper. Then subclass Pizza and call the classmethod constructor on the subclass — show the result is an instance of the subclass because of cls.

Pitfall

Hardcoding the class name inside an alternative constructor (instead of cls) breaks subclassing — the subclass gets a base-class instance back. Alternative constructors should be @classmethod and build with cls(...), never the literal class name.

5. Inheritance & MRO 11 days overdue rep 2 → 3

A subclass inherits attributes and methods and may override them. Attribute lookup follows the MRO (Method Resolution Order), computed by C3 linearization and visible via Cls.__mro__ or Cls.mro(). With multiple inheritance the MRO decides which implementation wins.

Key Pattern
class A:
    def who(self): return "A"
class B(A):
    def who(self): return "B"
class C(A):
    def who(self): return "C"
class D(B, C):
    pass

print(D().who())          # B  -- first match in the MRO
print([c.__name__ for c in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']
Challenge

Build a diamond: A, then B(A), C(A), D(B, C). Predict the MRO on paper, then verify with D.__mro__. Define a method on only C and confirm D resolves it through the MRO even though it directly inherits from B first.

Pitfall

The MRO is left-to-right per C3: D(B, C) checks B before C. An inconsistent hierarchy (bases whose orders conflict) raises TypeError: Cannot create a consistent method resolution order — reorder the bases to fix it.

6. super() Deep Dive 10 days overdue rep 2 → 3

super() delegates to the next class in the MRO — not necessarily the literal base class. In cooperative multiple inheritance every method calls super().method() so the whole chain runs exactly once. Bare super() inside a method is shorthand for super(CurrentClass, self).

Key Pattern
class Base:
    def __init__(self): print("Base")
class Left(Base):
    def __init__(self): print("Left"); super().__init__()
class Right(Base):
    def __init__(self): print("Right"); super().__init__()
class Child(Left, Right):
    def __init__(self): print("Child"); super().__init__()

Child()   # Child, Left, Right, Base -- each runs ONCE, in MRO order
Challenge

Build a cooperative diamond where each __init__ prints its own name and calls super().__init__(). Confirm each class's init runs exactly once and in MRO order — not the naive order you might expect from calling parents directly. Then break it by replacing one super() with a direct Base.__init__(self) and observe the duplicate/skip.

Pitfall

super() follows the MRO of the actual instance, so super().__init__() in Left may dispatch to Right, not Base — that is the whole point of cooperative inheritance. Calling a base directly (Base.__init__(self)) can run a class twice or skip siblings entirely.

New Concepts — Objects & Namespaces (3 concepts)

How Python objects actually store state: the plain __dict__ behind every attribute, the built-ins that read and write attributes by name at runtime, and the types module's ready-made building blocks for ad-hoc records and read-only views.

Objects & Namespaces

Concept 1: The __dict__ Namespace — vars() & dir()

Why learn this? Every ordinary object keeps its instance attributes in a plain dict named __dict__. Seeing this demystifies how obj.x = 1 works, why __slots__ saves memory, and how to introspect or serialize an object's state generically — without hardcoding attribute names.
Example 1 — the instance __dict__ IS the attribute storage
class User:
    species = "human"          # CLASS attribute (lives on the class)
    def __init__(self, name):
        self.name = name         # INSTANCE attribute

u = User("Ada")
print(u.__dict__)             # {'name': 'Ada'}  -- only instance attrs
u.age = 30                    # assignment just adds a dict key
print(u.__dict__)             # {'name': 'Ada', 'age': 30}
print(User.__dict__["species"]) # human  -- class attrs live on the class
Output
{'name': 'Ada'} {'name': 'Ada', 'age': 30} human
Example 2 — vars() is __dict__; dir() is the full name list
u = User("Ada")
print(vars(u))              # {'name': 'Ada'}  -- vars(obj) == obj.__dict__
print(vars(u) is u.__dict__)  # True
print("name" in dir(u))     # True
print("species" in dir(u))  # True  -- dir() includes class/inherited names
print(type(dir(u)).__name__) # list  -- a sorted list of NAMES
Output
{'name': 'Ada'} True True True list
Example 3 — mutating __dict__ directly
u = User("Ada")
u.__dict__["email"] = "ada@x.io"   # same as u.email = ...
print(u.email)                     # ada@x.io

vars(u).update(name="Grace", age=40) # bulk-set through the dict
print(u.name, u.age)               # Grace 40
Output
ada@x.io Grace 40
Example 4 — no __dict__ under __slots__
class Slim:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x, self.y = x, y

s = Slim(1, 2)
try:
    s.__dict__
except AttributeError as e:
    print("no dict:", e)     # 'Slim' object has no attribute '__dict__'
try:
    s.z = 3                  # cannot add unlisted attributes
except AttributeError as e:
    print("blocked:", e)    # 'Slim' object has no attribute 'z'
Output
no dict: 'Slim' object has no attribute '__dict__' blocked: 'Slim' object has no attribute 'z'

Common Mistakes

✗ Expecting class attributes and methods to show up in the instance __dict__ — u.__dict__ holds only instance attributes

✓ Look on type(u).__dict__ (or use dir) for class-level attributes; instance lookup falls back to the class along the MRO


✗ Treating dir(obj) as "the object's data" — it returns a sorted list of names (including inherited and dunder names), not values

✓ Use vars(obj)/obj.__dict__ for the instance data as a dict; use dir() to discover available names

Key Rules

  • Instance attributes live in obj.__dict__; vars(obj) returns that same dict. obj.x = v is just obj.__dict__["x"] = v for ordinary objects
  • Class attributes/methods live in type(obj).__dict__, shared by all instances; lookup falls back from instance to class along the MRO
  • dir(obj) returns a sorted list of all accessible names; __slots__ removes the per-instance __dict__ entirely

Exercises

  1. Write to_dict(obj) returning a shallow copy of an object's instance attributes via vars, and prove editing the copy does not touch the object.
  2. Given any object, print the names defined on the instance but not on its class: set(vars(obj)) - set(vars(type(obj))).
  3. Add an attribute by writing into obj.__dict__ directly and confirm it reads back through normal attribute access.
Connections: underpins __slots__ (removes __dict__) and the __init__ attribute model reviewed today; descriptors intercept access before __dict__ is consulted; types.SimpleNamespace (Concept 3) is essentially a thin object whose whole purpose is a friendly __dict__.
Objects & Namespaces

Concept 2: Dynamic Attributes — getattr, setattr, hasattr, delattr

Why learn this? When an attribute name is only known at runtime — from config, user input, or a loop — you cannot write obj.name. The getattr/setattr/hasattr/delattr built-ins read, write, test, and remove attributes by string name, which is the backbone of plugins, serializers, dispatchers, and ORMs.
Example 1 — read by name, with a default
class Config:
    debug = True
    level = 3

c = Config()
print(getattr(c, "level"))           # 3
print(getattr(c, "missing", "n/a"))  # n/a  -- default avoids AttributeError
name = "debug"
print(getattr(c, name))             # True  -- name computed at runtime
Output
3 n/a True
Example 2 — set and delete by name
c = Config()
setattr(c, "level", 9)      # same as c.level = 9
setattr(c, "extra", "hi")   # creates a new attribute
print(c.level, c.extra)      # 9 hi
delattr(c, "extra")         # same as del c.extra
print(hasattr(c, "extra"))  # False
Output
9 hi False
Example 3 — hasattr + build an object from a dict
class Blank: pass

b = Blank()
for k, v in {"host": "localhost", "port": 8080}.items():
    setattr(b, k, v)               # populate attributes from a mapping
print(b.host, b.port)              # localhost 8080
print(hasattr(b, "port"), hasattr(b, "user"))  # True False
Output
localhost 8080 True False
Example 4 — dispatch to a method by name
class Calc:
    def add(self, a, b): return a + b
    def sub(self, a, b): return a - b

op = "add"
calc = Calc()
method = getattr(calc, op)          # bound method, looked up by name
print(method(3, 4))               # 7
print(getattr(calc, "mul", None))  # None  -- graceful missing-method
Output
7 None

Common Mistakes

✗ Calling hasattr and then getattr — two lookups, and hasattr silently swallows unexpected errors raised inside a property

✓ Prefer a single getattr(obj, name, default); it is atomic and clearer than check-then-get


✗ Reaching for dynamic attributes to store arbitrary key/value data when a plain dict would be simpler and type-checkable

✓ Use getattr/setattr for genuinely dynamic names (plugins, serialization, dispatch); use a dict for free-form data

Key Rules

  • getattr(obj, "x") == obj.x; getattr(obj, "x", default) returns default instead of raising AttributeError
  • setattr(obj, "x", v) == obj.x = v; delattr(obj, "x") == del obj.x; hasattr(obj, "x") is a truthy existence test
  • The name is any string computed at runtime — enabling table-driven dispatch, but losing static tools' ability to catch typos

Exercises

  1. Write pluck(obj, names) returning {name: getattr(obj, name, None)} for a list of attribute names.
  2. Implement a tiny command dispatcher: given a string like "add", call the matching method via getattr, or print "unknown" when it is missing.
  3. Populate a blank object's attributes from a dict with setattr, then round-trip it back to a dict with vars.
Connections: pairs with __dict__/vars (Concept 1 — the storage these built-ins ultimately touch); the __getattr__/__setattr__ dunders let a class customize what these built-ins do; heavily used alongside types.SimpleNamespace (Concept 3) and the os/sys introspection ahead.
Objects & Namespaces

Concept 3: types Module — SimpleNamespace & MappingProxyType

Why learn this? The types module exposes building blocks the language uses internally. Two are everyday-useful: SimpleNamespace, a mutable object with attribute access and a readable repr (a quick struct/record), and MappingProxyType, a read-only view of a dict for exposing internal config safely.
Example 1 — SimpleNamespace: an ad-hoc object
from types import SimpleNamespace

cfg = SimpleNamespace(host="localhost", port=8080)
print(cfg.host, cfg.port)  # localhost 8080
cfg.port = 9090            # mutable
cfg.debug = True           # add attributes freely
print(cfg)                # namespace(host='localhost', port=9090, debug=True)
Output
localhost 8080 namespace(host='localhost', port=9090, debug=True)
Example 2 — SimpleNamespace vs dict
from types import SimpleNamespace

d = {"x": 1, "y": 2}
ns = SimpleNamespace(**d)      # dict -> attribute object
print(ns.x)                    # 1   (ns.x, NOT ns["x"])
print(vars(ns))                # {'x': 1, 'y': 2}  -- back to a dict
print(SimpleNamespace(**d) == SimpleNamespace(x=1, y=2))  # True -- value equality
Output
1 {'x': 1, 'y': 2} True
Example 3 — MappingProxyType: a read-only view
from types import MappingProxyType

_data = {"a": 1, "b": 2}
public = MappingProxyType(_data)   # read-only window onto _data
print(public["a"])                 # 1
try:
    public["c"] = 3
except TypeError as e:
    print("read-only:", e)         # does not support item assignment
_data["c"] = 3                     # the underlying dict CAN still change
print(public["c"])                 # 3  -- proxy reflects live updates
Output
1 read-only: 'mappingproxy' object does not support item assignment 3
Example 4 — other handy internal types
import types

def gen(): yield 1
print(isinstance(gen(), types.GeneratorType))     # True
print(isinstance(lambda: 0, types.FunctionType))  # True

mod = types.ModuleType("dynamic", "a docstring")   # build a module at runtime
mod.value = 42
print(mod.value, mod.__doc__)                     # 42 a docstring
Output
True True 42 a docstring

Common Mistakes

✗ Writing a full class with an __init__ just to bundle a few named values that never gain behavior

✓ Reach for SimpleNamespace (or a dataclass) for a lightweight record — less boilerplate, free repr and equality


✗ Handing out your internal dict directly and hoping callers will not mutate it

✓ Expose MappingProxyType(internal) so reads work but writes raise — but remember it is a live view, not a frozen copy

Key Rules

  • SimpleNamespace(**kwargs) is a mutable attribute-access object with a readable repr and value equality; vars(ns) gives its dict
  • MappingProxyType(d) is a read-only proxy: it forbids writes through the proxy but still reflects changes to the underlying dict (not a copy)
  • types also names internal types (FunctionType, GeneratorType, ModuleType, ...) useful for isinstance checks and metaprogramming

Exercises

  1. Parse a config dict into a SimpleNamespace, access values by attribute, then convert it back to a dict with vars.
  2. Wrap a private settings dict in a MappingProxyType, expose it, and demonstrate that writes fail while an update to the private dict shows through.
  3. Use a SimpleNamespace as a mutable accumulator inside a function (e.g., a few counters) and argue why it beats juggling several loose local variables.
Connections: SimpleNamespace is a friendly wrapper over the __dict__ model (Concept 1) and pairs with getattr/setattr (Concept 2); it overlaps with dataclasses and namedtuple (reviewed today) for records; MappingProxyType is how a class exposes its own __dict__, echoing the immutability theme of __slots__ and frozenset.

New Concepts — Talking to the System (2 concepts)

The two modules every real script eventually reaches for: os for the environment, directories, and portable path pieces, and sys for the interpreter itself — command-line arguments, the standard streams, and clean exits.

System

Concept 4: The os Module — Environment, Paths & Processes

Why learn this? os is the portable interface to the operating system: environment variables, the current directory, process IDs, and low-level file operations. Even in a pathlib world, os.environ and a handful of os helpers are indispensable for real scripts and services.
Example 1 — environment variables
import os

os.environ["APP_MODE"] = "prod"          # set (this process + children)
print(os.environ["APP_MODE"])            # prod
print(os.environ.get("MISSING", "def"))  # def  -- .get avoids KeyError
print(os.getenv("APP_MODE"))            # prod  -- getenv = get with None default
Output
prod def prod
Example 2 — current directory & directory ops
import os

cwd = os.getcwd()                          # absolute path of the working dir
os.makedirs("build/logs", exist_ok=True)  # nested; no error if it exists
print("build" in os.listdir("."))         # True  -- names in a directory
os.rmdir("build/logs"); os.rmdir("build") # remove EMPTY dirs
print(isinstance(cwd, str))             # True
Output
True True
Example 3 — process & platform facts
import os

print(os.getpid() > 0)          # True  -- this process's id
print(os.cpu_count() >= 1)      # True  -- number of logical CPUs
print(isinstance(os.sep, str))  # True  -- '/' on POSIX, '\\' on Windows
print(os.path.join("a", "b", "c.txt"))  # a/b/c.txt  -- portable join
Output
True True True a/b/c.txt
Example 4 — expanding ~ and $VARS, splitting paths
import os

os.environ["NAME"] = "ada"
print(os.path.expanduser("~") != "~")      # True  -- resolves the home dir
print(os.path.expandvars("hi $NAME"))      # hi ada
print(os.path.basename("/a/b/c.txt"))     # c.txt
print(os.path.splitext("archive.tar.gz"))  # ('archive.tar', '.gz')
Output
True hi ada c.txt ('archive.tar', '.gz')

Common Mistakes

✗ Reading os.environ["KEY"] for an optional variable — raises KeyError when it is unset

✓ Use os.environ.get("KEY", default) or os.getenv("KEY") for optional configuration


✗ Hardcoding "/" in paths or gluing paths together with string concatenation

✓ Use os.path.join (or pathlib.Path) so code works across operating systems

Key Rules

  • os.environ is a live mapping of environment variables (strings only); changes affect this process and its children, not the parent shell
  • Directory/process helpers: os.getcwd, os.chdir, os.makedirs(..., exist_ok=True), os.listdir, os.getpid, os.cpu_count
  • os.path gives portable path ops (join, basename, dirname, splitext, exists); modern code often prefers pathlib, but os.environ/os.getpid remain go-to

Exercises

  1. Read a config value from os.environ with a sensible default, then set a new variable and confirm it is visible via os.getenv.
  2. Create a nested directory tree with os.makedirs(exist_ok=True), list its contents, then remove it with os.rmdir.
  3. Given a full file path, use os.path to print its directory, base name, and extension — without touching the filesystem.
Connections: os.environ feeds the CLI-configuration story with argparse (reviewed earlier) and sys (Concept 5); os.path overlaps with the pathlib guide; spawning programs is done more safely with subprocess, and os.urandom connects to the secrets/random guides.
System

Concept 5: The sys Module — argv, exit & the Runtime

Why learn this? sys is your window into the Python runtime itself: command-line arguments, the standard streams, the interpreter version, the import search path, and how to exit with a status code. Any CLI script or diagnostic tool touches it.
Example 1 — argv: the command line
import sys
# Suppose the program was launched as:  python app.py build --fast
print(sys.argv)        # ['app.py', 'build', '--fast']
print(sys.argv[0])     # app.py           -- the script NAME
print(sys.argv[1:])    # ['build', '--fast'] -- the actual arguments
Output
['app.py', 'build', '--fast'] app.py ['build', '--fast']
Example 2 — exit codes & the stderr stream
import sys

print("normal message", file=sys.stdout)
print("error message", file=sys.stderr)   # goes to the stderr stream

def guard(ok):
    if not ok:
        sys.exit("fatal: precondition failed")  # prints to stderr, exit code 1
# sys.exit(0) = success, sys.exit(1) = failure, a str message => code 1
Output
normal message error message
Example 3 — interpreter & platform info
import sys

print(sys.version_info >= (3, 8))   # True
print(sys.version_info.major)       # 3
print(sys.platform in {"darwin", "linux", "win32"})  # True
print(sys.maxsize > 2**31)          # True on 64-bit builds
Output
True 3 True True
Example 4 — the import path & module cache
import sys

print(isinstance(sys.path, list))      # True  -- where imports are searched
print("sys" in sys.modules)            # True  -- cache of imported modules
print(sys.getrecursionlimit() > 100)  # True
sys.setrecursionlimit(3000)            # raise the ceiling for deep recursion
print(sys.getrecursionlimit())         # 3000
Output
True True True 3000

Common Mistakes

✗ Treating sys.argv[0] as the first argument — it is the script NAME; the real arguments start at sys.argv[1]

✓ Slice sys.argv[1:] for arguments, or let argparse parse them for you


✗ Ending a program with os._exit() or a bare exit() in scripts — os._exit skips cleanup and exit() is meant only for the interactive REPL

✓ Use sys.exit(code) (raises SystemExit, so finally/atexit still run) with 0 for success and non-zero for failure

Key Rules

  • sys.argv is the list of command-line tokens; sys.argv[0] is the script name and sys.argv[1:] are its arguments
  • sys.exit(status) raises SystemExit: an int is the exit code (0 = success), a string is printed to stderr with code 1. sys.stdout/sys.stderr are writable streams
  • sys also exposes runtime facts: sys.version_info, sys.platform, sys.path (import search list), sys.modules (import cache), and the recursion-limit getters/setters

Exercises

  1. Write a snippet that echoes its arguments (sys.argv[1:]) one per line, and when none are given prints a usage message to sys.stderr and exits with code 2.
  2. Print whether the interpreter is 64-bit (sys.maxsize > 2**32) and at least Python 3.10 using sys.version_info.
  3. Temporarily raise the recursion limit, run a deeply recursive function, and restore the original limit in a finally block.
Connections: sys.argv is exactly what argparse (reviewed earlier) parses for you; sys.stdout/stderr tie into the logging and print model; sys.path/sys.modules are the heart of the import system, and sys.exit cooperates with try/finally and atexit cleanup.
