pathlib for File System

Error Handling & I/O • 5 days overdue • rep 0

Quick Reminder

Path objects replace raw strings for file system paths. The / operator joins path segments: Path("src") / "main.py". Key methods: .read_text(), .write_text(), .exists(), .mkdir(parents=True), .glob("*.py"), and .iterdir(). Paths are OS-agnostic — the same code works on Windows, macOS, and Linux.

Key Pattern — Path Navigation & Globbing

from pathlib import Path

# Build paths with the / operator
project = Path("myproject")
src = project / "src"
config = project / "config" / "settings.toml"

print(config)            # myproject/config/settings.toml
print(config.parent)     # myproject/config
print(config.stem)       # settings
print(config.suffix)     # .toml
print(config.name)       # settings.toml

# Glob for files
root = Path(".")
py_files = sorted(root.glob("**/*.py"))  # recursive
print(f"Found {len(py_files)} Python files")

# Read / write
p = Path("/tmp/demo.txt")
p.write_text("hello pathlib\n")
content = p.read_text()
print(content)  # hello pathlib
Challenge

Walk a directory tree and find all .py files larger than 1 KB. Print each file's path and size in KB. Use only pathlib methods (no os.walk).

#1 Pitfall: Using string concatenation ("src" + "/" + "file.py") instead of the / operator. String concatenation breaks on Windows where the separator is \. The / operator on Path objects is cross-platform.

JSON & CSV Data Formats

Error Handling & I/O • 5 days overdue • rep 0

Quick Reminder

json.dumps(obj) serializes to a string, json.loads(s) deserializes. json.dump/load work with file objects. csv.DictReader reads rows as dicts (using headers as keys), csv.DictWriter writes dicts as rows. For non-standard types (datetime, Decimal), provide a custom encoder or default function.

Key Pattern — Round-Trip Conversion

import json
import csv
import io
from datetime import datetime

# JSON with custom encoder
data = [
    {"name": "Alice", "score": 95, "ts": datetime(2026, 6, 9)},
    {"name": "Bob", "score": 87, "ts": datetime(2026, 6, 8)},
]

def default_encoder(obj):
    if isinstance(obj, datetime):
        return obj.isoformat()
    raise TypeError(f"Not serializable: {type(obj)}")

j = json.dumps(data, default=default_encoder, indent=2)
print(j[:80] + "...")

# CSV round-trip
buf = io.StringIO()
writer = csv.DictWriter(buf, fieldnames=["name", "score"])
writer.writeheader()
for row in data:
    writer.writerow({"name": row["name"], "score": row["score"]})

buf.seek(0)
reader = csv.DictReader(buf)
for row in reader:
    print(row)  # OrderedDict with string values
Challenge

Convert a list of dicts to CSV format (as a string), then parse the CSV back to a list of dicts, and finally serialize that list to JSON. Verify the round-trip produces equivalent data.

#1 Pitfall: json.dumps does not handle datetime objects. You must supply a default function or subclass json.JSONEncoder. Forgetting this raises TypeError: Object of type datetime is not JSON serializable.

Regular Expressions

Error Handling & I/O • 5 days overdue • rep 0

Quick Reminder

re.compile(pattern) creates a reusable regex. Named groups (?P<name>...) make matches self-documenting. re.findall returns all matches as strings; re.finditer returns match objects (giving access to .group(), .start(), .end()). Use re.VERBOSE for readable multi-line patterns.

Key Pattern — Log Parsing with Named Groups

import re

log_pattern = re.compile(
    r"(?P<timestamp>\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2})"
    r"\s+(?P<level>INFO|WARNING|ERROR)"
    r"\s+(?P<message>.+)"
)

logs = [
    "2026-06-09 10:15:30 INFO Server started on port 8080",
    "2026-06-09 10:15:45 ERROR Connection refused: db-primary",
    "2026-06-09 10:16:00 WARNING Memory usage at 85%",
]

for line in logs:
    m = log_pattern.match(line)
    if m:
        print(f"[{m.group('level'):<7}] {m.group('timestamp')} | {m.group('message')}")
Output
[INFO ] 2026-06-09 10:15:30 | Server started on port 8080 [ERROR ] 2026-06-09 10:15:45 | Connection refused: db-primary [WARNING] 2026-06-09 10:16:00 | Memory usage at 85%
Challenge

Parse log lines extracting timestamp, level, and message using named groups. Process at least 5 lines and print a summary count of each log level.

#1 Pitfall: Greedy vs non-greedy matching. .* is greedy (matches as much as possible), while .*? is non-greedy (matches as little as possible). In patterns like <.*>, greedy matching captures too much: "<a>text</a>" matches the whole string, not just "<a>".

logging Module

Error Handling & I/O • 5 days overdue • rep 0

Quick Reminder

logging.getLogger(name) retrieves a named logger. Loggers form a hierarchy by dot-separated names. Handlers control output destinations (console, file, network). Formatters control the message format. Levels: DEBUG < INFO < WARNING < ERROR < CRITICAL. Always use getLogger(__name__) in modules.

Key Pattern — Dual-Handler Setup

import logging

logger = logging.getLogger("myapp")
logger.setLevel(logging.DEBUG)  # Capture everything

# Console handler: INFO and above
console = logging.StreamHandler()
console.setLevel(logging.INFO)
console.setFormatter(logging.Formatter(
    "%(levelname)-8s %(message)s"
))

# File handler: DEBUG and above
file_handler = logging.FileHandler("/tmp/myapp.log")
file_handler.setLevel(logging.DEBUG)
file_handler.setFormatter(logging.Formatter(
    "%(asctime)s %(name)s %(levelname)-8s %(message)s"
))

logger.addHandler(console)
logger.addHandler(file_handler)

logger.debug("This goes to file only")
logger.info("This goes to both console and file")
logger.error("Errors go everywhere")
Output (console)
INFO This goes to both console and file ERROR Errors go everywhere
Challenge

Set up a logging configuration where INFO+ messages go to the console and DEBUG+ messages go to a file. Include timestamps in the file format but not in the console format.

#1 Pitfall: Calling logging.basicConfig() after a handler has already been added to the root logger is a no-op. This commonly happens when a library configures logging before your basicConfig call. Always set up handlers explicitly with getLogger() for reliable control.

Classes — __eq__ and __hash__

OOP • 4 days overdue • rep 1

Quick Reminder

Defining __eq__ makes == compare by value instead of identity. However, defining __eq__ without __hash__ makes instances unhashable (cannot be used as dict keys or in sets). For hashable value objects: implement both __eq__ and __hash__, and ensure that equal objects always have equal hashes. Mutable objects generally should not be hashable.

Key Pattern — Hashable Point Class

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __eq__(self, other):
        if not isinstance(other, Point):
            return NotImplemented
        return self.x == other.x and self.y == other.y

    def __hash__(self):
        return hash((self.x, self.y))

    def __repr__(self):
        return f"Point({self.x}, {self.y})"

# Works as dict key and in sets
p1 = Point(1, 2)
p2 = Point(1, 2)
p3 = Point(3, 4)

print(p1 == p2)              # True
print({p1, p2, p3})          # {Point(1, 2), Point(3, 4)}
print({p1: "origin"}[p2])    # origin (same hash + equal)
Output
True {Point(1, 2), Point(3, 4)} origin
Challenge

Create a Point class that works correctly as a dict key. Create two Point(1, 2) instances, use one as a dict key, and look up the value using the other instance.

#1 Pitfall: Defining __eq__ without __hash__ makes your class unhashable. Python sets __hash__ = None when you define __eq__ because mutable equality comparisons and hashing are generally incompatible. You must explicitly define __hash__ to restore hashability.

Type Hints — Optional, Union

Type System • 4 days overdue • rep 1

Quick Reminder

In Python 3.10+, use X | Y instead of Union[X, Y]. Optional[X] is shorthand for X | None — it means the value can be X or None, not that the parameter is optional. Use TypeGuard for type-narrowing functions that tell the type checker a value is a specific type after a runtime check.

Key Pattern — Union Types & Type Narrowing

from typing import TypeGuard

# Python 3.10+ union syntax
def parse_id(raw: str | int) -> int:
    if isinstance(raw, str):
        return int(raw)
    return raw

# Optional means "X or None"
def find_user(user_id: int) -> dict | None:
    users = {1: {"name": "Alice"}, 2: {"name": "Bob"}}
    return users.get(user_id)

# TypeGuard for custom narrowing
def is_str_list(val: list[object]) -> TypeGuard[list[str]]:
    return all(isinstance(x, str) for x in val)

data: list[object] = ["a", "b", "c"]
if is_str_list(data):
    # type checker knows data is list[str] here
    print(", ".join(data))  # a, b, c

print(parse_id("42"))         # 42
print(find_user(1))           # {'name': 'Alice'}
print(find_user(999))         # None
Output
a, b, c 42 {'name': 'Alice'} None
Challenge

Type a function that accepts either a single string or a list of strings and always returns a list of strings. If given a string, wrap it in a list. Use the X | Y syntax.

#1 Pitfall: Optional[X] means X | None, not that the parameter can be omitted. A parameter with a default value is optional (can be omitted), but Optional[int] just means the type can be int or None.

Metaclasses

Advanced Topics • New today

Why It Matters

Metaclasses control how classes themselves are created. They are the "class of a class." Understanding metaclasses explains the deepest level of Python's object model, even though you will rarely need to write one in practice. Most use cases are better served by __init_subclass__ or class decorators.

type() as the Default Metaclass

Every class in Python is an instance of type. You can create classes dynamically by calling type(name, bases, namespace):

# Normal class creation
class Dog:
    sound = "woof"
    def speak(self):
        return self.sound

# Equivalent dynamic creation with type()
Dog2 = type("Dog2", (), {
    "sound": "woof",
    "speak": lambda self: self.sound,
})

d1 = Dog()
d2 = Dog2()
print(f"Dog: {d1.speak()}")       # woof
print(f"Dog2: {d2.speak()}")      # woof
print(f"type(Dog): {type(Dog)}")  # <class 'type'>
print(f"type(Dog2): {type(Dog2)}")  # <class 'type'>
Output
Dog: woof Dog2: woof type(Dog): <class 'type'> type(Dog2): <class 'type'>

Custom Metaclass — Auto-Registry

A custom metaclass inherits from type and overrides __new__ (called to create the class) or __init__ (called after creation). This pattern lets you intercept class creation to register, validate, or transform classes automatically.

class RegistryMeta(type):
    """Metaclass that auto-registers all subclasses."""
    _registry = {}

    def __new__(mcs, name, bases, namespace):
        cls = super().__new__(mcs, name, bases, namespace)
        # Don't register the base class itself
        if bases:
            RegistryMeta._registry[name] = cls
        return cls

class Plugin(metaclass=RegistryMeta):
    """Base class for plugins."""
    pass

class JSONPlugin(Plugin):
    format = "json"

class XMLPlugin(Plugin):
    format = "xml"

class CSVPlugin(Plugin):
    format = "csv"

print("Registered plugins:")
for name, cls in RegistryMeta._registry.items():
    print(f"  {name} -> {cls.format}")
Output
Registered plugins: JSONPlugin -> json XMLPlugin -> xml CSVPlugin -> csv

Singleton via Metaclass

class SingletonMeta(type):
    _instances = {}

    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Database(metaclass=SingletonMeta):
    def __init__(self):
        self.connection = "connected"

db1 = Database()
db2 = Database()
print(f"Same instance? {db1 is db2}")      # True
print(f"Connection: {db1.connection}")      # connected
Output
Same instance? True Connection: connected

__init_subclass__ — The Simpler Alternative

Python 3.6+ provides __init_subclass__ as a hook that runs when a class is subclassed, without needing a metaclass. Prefer this for most registry and validation patterns:

class Plugin:
    _registry = {}

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if hasattr(cls, "format"):
            Plugin._registry[cls.format] = cls

class JSONPlugin(Plugin):
    format = "json"

class XMLPlugin(Plugin):
    format = "xml"

print("Registry:", Plugin._registry)
# {'json': <class 'JSONPlugin'>, 'xml': <class 'XMLPlugin'>}
Output
Registry: {'json': <class 'JSONPlugin'>, 'xml': <class 'XMLPlugin'>}

Key Rules

  • Use __init_subclass__ or class decorators before reaching for metaclasses.
  • Metaclasses are appropriate when you need to control class creation at a fundamental level (e.g., ORMs, API frameworks).
  • Override __new__ to modify the class before it is created; override __init__ to modify it after.
  • A class can only have one metaclass — metaclass conflicts arise with multiple inheritance if bases use different metaclasses.
#1 Pitfall: Using a metaclass when __init_subclass__ or a class decorator would suffice. Metaclasses add complexity and can cause conflicts with other metaclasses during multiple inheritance. Reach for simpler tools first.
Connection: This builds on the Metaprogramming: __init_subclass__ & __class_getitem__ concept from Day 10, showing the deeper mechanism that __init_subclass__ was designed to replace. It also relates to Descriptors from Day 6 — both are mechanisms for controlling Python's object system at the meta-level.

Async Generators & async for

Concurrency • New today

Why It Matters

Async generators combine lazy iteration with asynchronous I/O. They are perfect for streaming data from APIs, databases, or message queues where each item requires an await to fetch but you want to process results one at a time without loading everything into memory.

Basic Async Generator

An async generator is defined with async def and uses yield. You consume it with async for:

import asyncio

async def countdown(n):
    """Async generator that counts down with delays."""
    while n > 0:
        await asyncio.sleep(0.1)  # simulate async work
        yield n
        n -= 1

async def main():
    async for value in countdown(5):
        print(f"  {value}...", end="")
    print(" Go!")

asyncio.run(main())
Output
5... 4... 3... 2... 1... Go!

Simulated Paginated API

import asyncio

async def fetch_pages(total_pages=3, items_per_page=2):
    """Simulate fetching paginated API results."""
    for page in range(1, total_pages + 1):
        await asyncio.sleep(0.05)  # simulate network delay
        items = [
            {"id": (page - 1) * items_per_page + i, "name": f"item_{(page - 1) * items_per_page + i}"}
            for i in range(1, items_per_page + 1)
        ]
        print(f"  Fetched page {page}")
        for item in items:
            yield item

async def main():
    results = []
    async for item in fetch_pages():
        results.append(item)
    print(f"Total items: {len(results)}")
    for r in results:
        print(f"  {r}")

asyncio.run(main())
Output
Fetched page 1 Fetched page 2 Fetched page 3 Total items: 6 {'id': 1, 'name': 'item_1'} {'id': 2, 'name': 'item_2'} {'id': 3, 'name': 'item_3'} {'id': 4, 'name': 'item_4'} {'id': 5, 'name': 'item_5'} {'id': 6, 'name': 'item_6'}

aiter() and anext() Built-ins (3.10+)

import asyncio

async def numbers():
    for i in range(1, 6):
        await asyncio.sleep(0.01)
        yield i

async def main():
    gen = numbers()
    # Manual iteration with aiter/anext
    ait = aiter(gen)
    first = await anext(ait)
    second = await anext(ait)
    print(f"First two: {first}, {second}")

    # anext with default (like next())
    rest = []
    while True:
        val = await anext(ait, None)
        if val is None:
            break
        rest.append(val)
    print(f"Rest: {rest}")

asyncio.run(main())
Output
First two: 1, 2 Rest: [3, 4, 5]

Async Context Manager with @asynccontextmanager

import asyncio
from contextlib import asynccontextmanager

@asynccontextmanager
async def managed_connection(name):
    print(f"  Opening connection to {name}...")
    await asyncio.sleep(0.05)  # simulate connection
    conn = {"name": name, "status": "open"}
    try:
        yield conn
    finally:
        await asyncio.sleep(0.02)  # simulate cleanup
        conn["status"] = "closed"
        print(f"  Closed connection to {name}")

async def main():
    async with managed_connection("database") as conn:
        print(f"  Using: {conn}")
    print("  Done!")

asyncio.run(main())
Output
Opening connection to database... Using: {'name': 'database', 'status': 'open'} Closed connection to database Done!

Key Rules

  • Use async for to consume async generators — regular for will not work.
  • Every yield in an async generator can be preceded by await calls.
  • Use async with for resources that need async setup and teardown.
  • aiter() and anext() (Python 3.10+) are the async equivalents of iter() and next().
#1 Pitfall: Forgetting await inside an async generator. If you call a coroutine without await, you get a coroutine object instead of the result, and Python will emit a "coroutine was never awaited" warning. Always await async calls inside async generators.
Connection: This extends the async/await Basics from Day 8 with lazy iteration. It also connects to Generator Expressions from Day 2 — async generators are the async equivalent of sync generators, using the same yield keyword but requiring async def and async for.

multiprocessing Patterns

Concurrency & Performance • New today

Why It Matters

multiprocessing creates separate processes, each with its own Python interpreter and GIL. This gives true CPU parallelism on multi-core machines, unlike threading which is limited by the GIL for CPU-bound work.

Basic Process

from multiprocessing import Process
import os

def worker(name):
    print(f"Worker {name}: PID={os.getpid()}, Parent={os.getppid()}")

if __name__ == "__main__":
    print(f"Main process: PID={os.getpid()}")
    procs = []
    for i in range(3):
        p = Process(target=worker, args=(f"P{i}",))
        procs.append(p)
        p.start()

    for p in procs:
        p.join()
    print("All workers done")
Output
Main process: PID=12345 Worker P0: PID=12346, Parent=12345 Worker P1: PID=12347, Parent=12345 Worker P2: PID=12348, Parent=12345 All workers done

Pool.map for Parallel Computation

Pool.map(func, iterable) distributes work across a pool of worker processes, similar to the built-in map() but parallel:

from multiprocessing import Pool
import time

def heavy_computation(n):
    """CPU-bound: sum of squares up to n."""
    return sum(i * i for i in range(n))

if __name__ == "__main__":
    numbers = [5_000_000] * 4

    # Sequential
    start = time.perf_counter()
    sequential = list(map(heavy_computation, numbers))
    seq_time = time.perf_counter() - start

    # Parallel with Pool
    start = time.perf_counter()
    with Pool(processes=4) as pool:
        parallel = pool.map(heavy_computation, numbers)
    par_time = time.perf_counter() - start

    print(f"Sequential: {seq_time:.3f}s")
    print(f"Parallel:   {par_time:.3f}s")
    print(f"Speedup:    {seq_time / par_time:.1f}x")
    print(f"Results match: {sequential == parallel}")
Output
Sequential: 1.248s Parallel: 0.387s Speedup: 3.2x Results match: True

Queue for Producer-Consumer

from multiprocessing import Process, Queue
import time

def producer(queue, items):
    for item in items:
        time.sleep(0.01)  # simulate work
        queue.put(item)
        print(f"  Produced: {item}")
    queue.put(None)  # sentinel

def consumer(queue):
    results = []
    while True:
        item = queue.get()
        if item is None:
            break
        results.append(item * 2)
        print(f"  Consumed: {item} -> {item * 2}")
    return results

if __name__ == "__main__":
    q = Queue()
    items = [1, 2, 3, 4, 5]

    prod = Process(target=producer, args=(q, items))
    cons = Process(target=consumer, args=(q,))

    prod.start()
    cons.start()

    prod.join()
    cons.join()
    print("Done!")
Output
Produced: 1 Consumed: 1 -> 2 Produced: 2 Consumed: 2 -> 4 Produced: 3 Consumed: 3 -> 6 Produced: 4 Consumed: 4 -> 8 Produced: 5 Consumed: 5 -> 10 Done!

Shared State with Value and Lock

from multiprocessing import Process, Value, Lock

def increment(counter, lock, n):
    for _ in range(n):
        with lock:
            counter.value += 1

if __name__ == "__main__":
    counter = Value('i', 0)  # shared integer, initial value 0
    lock = Lock()
    n_per_process = 100_000

    procs = [
        Process(target=increment, args=(counter, lock, n_per_process))
        for _ in range(4)
    ]

    for p in procs:
        p.start()
    for p in procs:
        p.join()

    print(f"Expected: {4 * n_per_process}")
    print(f"Actual:   {counter.value}")
    print(f"Correct:  {counter.value == 4 * n_per_process}")
Output
Expected: 400000 Actual: 400000 Correct: True

Key Rules

  • Always use if __name__ == "__main__" guard — without it, child processes re-import the module and cause infinite spawning on Windows/macOS.
  • Arguments passed to processes must be picklable (no lambdas, no open file handles).
  • Use Pool.map for simple data parallelism; use Queue for producer-consumer patterns.
  • Shared state (Value, Array) requires explicit locking to avoid race conditions.
  • Prefer concurrent.futures.ProcessPoolExecutor for a higher-level API.
#1 Pitfall: Forgetting the if __name__ == "__main__" guard. On macOS and Windows (which use "spawn" to start new processes), the child process re-imports the module. Without the guard, this triggers infinite process creation until the system runs out of resources.
Connection: This complements CPython Internals: GIL & Reference Counting from Day 11. The GIL prevents threading from achieving CPU parallelism, making multiprocessing the go-to solution. It also relates to concurrent.futures from Day 8, which provides a simpler interface over multiprocessing.Pool.

Packaging & Distribution

Tooling • New today

Why It Matters

Packaging lets you share Python code as installable libraries or command-line tools via PyPI. Understanding pyproject.toml, entry points, and build tools is essential for any Python developer who creates reusable code.

Minimal pyproject.toml (PEP 621)

The modern way to configure a Python package is pyproject.toml, replacing setup.py and setup.cfg:

[build-system]
requires = ["setuptools>=68.0", "wheel"]
build-backend = "setuptools.backends._legacy:_Backend"

[project]
name = "mypackage"
version = "0.1.0"
description = "A short description of my package"
readme = "README.md"
requires-python = ">=3.10"
license = {text = "MIT"}
authors = [
    {name = "Your Name", email = "you@example.com"},
]
dependencies = [
    "requests>=2.28",
    "click>=8.0",
]

[project.optional-dependencies]
dev = ["pytest>=7.0", "ruff"]

[project.scripts]
mycli = "mypackage.cli:main"

Package Structure

mypackage/
    pyproject.toml
    README.md
    src/
        mypackage/
            __init__.py
            cli.py
            core.py
    tests/
        test_core.py

CLI Entry Point

The [project.scripts] table creates console scripts. When the package is installed, mycli becomes available as a command:

# src/mypackage/cli.py
import click
from mypackage.core import process

@click.command()
@click.argument("name")
@click.option("--greeting", default="Hello", help="Greeting to use")
def main(name, greeting):
    """Greet someone from the CLI."""
    result = process(name, greeting)
    click.echo(result)

# After pip install: $ mycli Alice --greeting Hi
# Output: Hi, Alice!

__main__.py for python -m

Adding __main__.py lets users run your package with python -m mypackage:

# src/mypackage/__main__.py
from mypackage.cli import main

if __name__ == "__main__":
    main()

# Usage: python -m mypackage Alice --greeting Hey

twine check dist/*

# Upload to TestPyPI first
twine upload --repository testpypi dist/*

# Upload to real PyPI
twine upload dist/*

Key Rules

  • Use pyproject.toml (PEP 621) instead of setup.py for new projects.
  • Use the src/ layout to prevent accidental imports from the project root.
  • Pin minimum versions of dependencies (requests>=2.28), not exact versions.
  • Use [project.scripts] for CLI entry points and __main__.py for python -m support.
  • Always test on TestPyPI before publishing to the real PyPI.
#1 Pitfall: Forgetting __init__.py in package directories. Without it, Python will not recognize the directory as a package (in non-namespace-package mode). Also, using setup.py when pyproject.toml handles everything is outdated practice that adds unnecessary complexity.

Debugging: pdb & breakpoint()

Tooling • New today

Why It Matters

breakpoint() drops you into an interactive debugger at any point in your code. It is far more powerful than print() debugging because you can inspect variables, step through code line by line, evaluate expressions, and examine the call stack — all without restarting your program.

breakpoint() Basics (Python 3.7+)

Insert breakpoint() anywhere in your code. When execution reaches that line, Python drops into the pdb interactive debugger:

def find_bug(numbers):
    total = 0
    for i, n in enumerate(numbers):
        total += n
        if total > 10:
            breakpoint()  # Drops into debugger here
            # In the debugger, you can:
            # (Pdb) p total      -> print total
            # (Pdb) p i          -> print current index
            # (Pdb) p numbers    -> print the list
            # (Pdb) n            -> execute next line
            # (Pdb) c            -> continue to next breakpoint or end
    return total

# When running:
# find_bug([3, 5, 4, 2, 8])

Essential pdb Commands

# Navigation
n             # (next) Execute current line, step OVER function calls
s             # (step) Execute current line, step INTO function calls
c             # (continue) Run until next breakpoint or end
r             # (return) Run until current function returns
unt 42        # (until) Run until line 42

# Inspection
p expr        # Print value of expression
pp expr       # Pretty-print value of expression
l             # List 11 lines of source around current line
l 1, 20       # List lines 1-20
w             # (where) Print full stack trace
u             # (up) Move one frame up in the stack
d             # (down) Move one frame down in the stack

# Breakpoints
b 42          # Set breakpoint at line 42
b mymod.py:42 # Set breakpoint at line 42 of mymod.py
b func        # Set breakpoint at first line of func
condition 1 x > 5   # Breakpoint 1 only triggers when x > 5
cl 1          # Clear breakpoint 1
b             # List all breakpoints

# Control
q             # Quit debugger (and program)
!stmt         # Execute a Python statement

Conditional Breakpoints

def process_items(items):
    results = []
    for i, item in enumerate(items):
        value = item ** 2
        # Only break when we hit a specific condition
        if value > 50:
            breakpoint()
            # (Pdb) p i, item, value
            # (Pdb) c  -> continue to next occurrence
        results.append(value)
    return results

# Alternatively, set a conditional breakpoint in pdb:
# (Pdb) b 5, value > 50
# This is cleaner than adding if-statements in code

Post-Mortem Debugging

When an exception crashes your program, pdb.pm() lets you inspect the state at the point of failure:

import pdb

def divide_all(numbers, divisor):
    return [n / divisor for n in numbers]

# In a REPL or script:
try:
    divide_all([1, 2, 3], 0)
except ZeroDivisionError:
    pdb.pm()
    # Now you can inspect the state at the moment of the crash:
    # (Pdb) p divisor    -> 0
    # (Pdb) p numbers    -> [1, 2, 3]
    # (Pdb) w            -> see the full traceback

PYTHONBREAKPOINT Environment Variable

# Disable all breakpoints (for production)
# PYTHONBREAKPOINT=0 python my_script.py

# Use a different debugger (e.g., ipdb)
# PYTHONBREAKPOINT=ipdb.set_trace python my_script.py

# Use web debugger
# PYTHONBREAKPOINT=web_pdb.set_trace python my_script.py

import os
# Programmatically check if debugging is disabled
print(f"PYTHONBREAKPOINT = {os.environ.get('PYTHONBREAKPOINT', '(not set)')}")
# Default: calls pdb.set_trace
# "0": breakpoint() does nothing
# "module.function": calls that function instead

Key Rules

  • Use breakpoint() instead of import pdb; pdb.set_trace() — it respects PYTHONBREAKPOINT.
  • Learn the core commands: n (next), s (step), c (continue), p (print), l (list), w (where).
  • Use pdb.pm() after an exception to inspect the crash site without rerunning.
  • Set PYTHONBREAKPOINT=0 in production to disable all breakpoints.
  • Never commit breakpoint() calls to production code. Use a pre-commit hook to catch them.
#1 Pitfall: Leaving breakpoint() calls in production code. Unlike print statements (which just add noise), a breakpoint will halt your program and wait for interactive input, effectively freezing a production server. Use a linter or pre-commit hook to catch stray breakpoint() calls.
Connection: Debugging complements Profiling with timeit & cProfile from Day 8 — profiling tells you where the problem is, and pdb lets you understand why. The w (where) command shows the call stack, which connects to understanding Closures & nonlocal (Day 6) since you can inspect variables in enclosing scopes.
