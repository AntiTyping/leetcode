Generators

Generators produce values lazily — one at a time, on demand. They’re the memory-efficient alternative to lists when you don’t need all values at once.

Generator Expressions vs List Comprehensions Generators

Why it matters: When processing large datasets (millions of rows, huge files), list comprehensions create the entire list in memory. Generator expressions produce one value at a time, using constant memory regardless of input size. This is the difference between a program that works and one that crashes with MemoryError.

1. Basic syntax — identical to list comprehension but with parentheses:

Python
# List comprehension — creates the full list in memory
squares_list = [x**2 for x in range(10)]
print(type(squares_list))
print(squares_list)

# Generator expression — creates a lazy generator object
squares_gen = (x**2 for x in range(10))
print(type(squares_gen))
print(squares_gen)
<class 'list'> [0, 1, 4, 9, 16, 25, 36, 49, 64, 81] <class 'generator'> <generator object <genexpr> at 0x...>

2. Lazy evaluation — values computed on demand:

Calling next() on a generator advances it by one step, computing and returning just that value. When all values are exhausted, StopIteration is raised. You can also consume a generator with a for loop.

Python
gen = (x**2 for x in range(5))

print(next(gen))    # First value
print(next(gen))    # Second value
print(next(gen))    # Third value
print(next(gen))    # Fourth value
print(next(gen))    # Fifth value (last one)
0 1 4 9 16
Python
# After exhaustion, next() raises StopIteration
gen = (x**2 for x in range(3))
print(next(gen))    # 0
print(next(gen))    # 1
print(next(gen))    # 4

try:
    next(gen)       # No more values!
except StopIteration:
    print("Generator exhausted — no more values")
0 1 4 Generator exhausted — no more values
Python
# Using a generator in a for-loop (most common usage)
gen = (x**2 for x in range(5))
for val in gen:
    print(val, end=" ")
0 1 4 9 16

3. Memory comparison — the killer advantage:

A list comprehension allocates memory for every element at once. A generator expression uses a fixed, tiny amount of memory regardless of how many values it can produce.

Python
import sys

# Small: 1,000 elements
list_size = sys.getsizeof([x for x in range(1000)])
gen_size  = sys.getsizeof(x for x in range(1000))
print(f"List (1,000 items): {list_size:,} bytes")
print(f"Generator:          {gen_size:,} bytes")

print()

# Large: 1,000,000 elements
list_size_big = sys.getsizeof([x for x in range(1_000_000)])
gen_size_big  = sys.getsizeof(x for x in range(1_000_000))
print(f"List (1M items):    {list_size_big:,} bytes  (~{list_size_big / 1024 / 1024:.1f} MB)")
print(f"Generator:          {gen_size_big:,} bytes   (constant!)")
List (1,000 items): 8,856 bytes Generator: 112 bytes List (1M items): 8,448,728 bytes (~8.1 MB) Generator: 112 bytes (constant!)

4. Chaining with built-in functions — generators shine when composed:

Many built-in functions accept iterables, so you can pass a generator expression directly. When a generator is the only argument to a function, the outer parentheses serve double duty as both the function call and generator delimiters — no extra parens needed.

Python
# sum() — no intermediate list needed
total = sum(x**2 for x in range(1000))
print(f"Sum of squares 0-999: {total:,}")

# max() — find longest line in a file (conceptual)
# max(len(line) for line in open("file.txt"))

# any() — short-circuits! Stops as soon as True is found
data = [3, 7, 15, 42, 200, 999, 1001]
has_big = any(x > 1000 for x in data)
print(f"Has value > 1000: {has_big}")

# all() — checks every element (or short-circuits on False)
all_positive = all(x > 0 for x in data)
print(f"All positive: {all_positive}")

# ''.join() — build strings lazily
words = ["hello", "world", "python"]
upper_csv = ", ".join(w.upper() for w in words)
print(upper_csv)
Sum of squares 0-999: 332,833,500 Has value > 1000: True All positive: True HELLO, WORLD, PYTHON

5. Real-world pattern — processing pipelines:

You can chain multiple generator expressions to build a lazy processing pipeline. Each stage transforms or filters data, but nothing actually runs until a terminal operation (like sum()) pulls values through the chain. The entire pipeline runs in constant memory.

Python
# Processing pipeline — each stage is a generator
# Imagine "data.csv" has thousands of lines like:
#   "item1,electronics,29.99"
#   "item2,clothing,49.50"
#   ""  (blank lines)

# lines  = (line.strip() for line in open("data.csv"))
# non_empty = (line for line in lines if line)
# fields = (line.split(",") for line in non_empty)
# prices = (float(row[2]) for row in fields if len(row) > 2)
# total  = sum(prices)  # entire file processed in constant memory

# Demonstration with in-memory data:
raw_data = [
    "  item1,electronics,29.99  ",
    "  item2,clothing,49.50  ",
    "",
    "  item3,food,12.75  ",
    "  ",
    "  item4,electronics,89.00  ",
]

lines     = (line.strip() for line in raw_data)
non_empty = (line for line in lines if line)
fields    = (line.split(",") for line in non_empty)
prices    = (float(row[2]) for row in fields if len(row) > 2)
total     = sum(prices)

print(f"Total: ${total:.2f}")
Total: $181.24
Key Insight: Each variable above (lines, non_empty, fields, prices) is just a generator object — no data is stored. Values flow through the entire pipeline one at a time when sum() requests them. For a 10 GB file, this uses the same memory as a 10-line file.
Common Mistakes:
Trying to index a generator: gen[0] raises TypeError. Generators don’t support random access.
Use next(gen) for one value, or list(gen) to materialize (but that defeats the purpose for large data).
Trying to iterate a generator twice — second pass yields nothing (generators are exhausted after one pass):
Python
gen = (x for x in range(5))

# First iteration: works fine
first_pass = list(gen)
print(f"First pass:  {first_pass}")

# Second iteration: empty!
second_pass = list(gen)
print(f"Second pass: {second_pass}")
First pass: [0, 1, 2, 3, 4] Second pass: []
More Pitfalls:
Calling len(gen) raises TypeError. Generators don’t know their length.
Use sum(1 for _ in gen) to count, but this exhausts the generator.
Reusing an exhausted generator — it silently produces nothing.
Create a new generator expression if you need to iterate again, or use itertools.tee().
Key Rules:
  • Use generator expressions when you only need to iterate once (summing, finding max, checking any/all)
  • Use list comprehensions when you need random access, multiple iterations, or the result length
  • Generator expressions can be passed directly to functions that accept iterables: sum(), min(), max(), any(), all(), ''.join()
  • Generators are single-use — once exhausted, they produce no more values
  • When a generator expression is the only argument to a function, you don’t need double parentheses: sum(x**2 for x in data) not sum((x**2 for x in data))
Practice Exercises:
1 Write a generator expression that yields the squares of all odd numbers from 1 to 20. Use sum() to add them up. Expected result: 1 + 9 + 25 + 49 + 81 + 121 + 169 + 225 + 289 + 361 = 1330.
2 Given words = ["hello", "world", "python", "programming", "AI"], write a single expression using a generator to find the longest word. Do NOT create an intermediate list. (Hint: use max() with a key argument.) Expected result: "programming".

namedtuple — Lightweight Data Classes Collections

When you pass around tuples like (x, y) or ("Alice", 30, "NYC"), the meaning of each position is unclear. namedtuple gives you named fields while remaining immutable, hashable, and memory-efficient — perfect for data records, API responses, and config objects.

A namedtuple creates a new tuple subclass with named fields. You get all the benefits of regular tuples (immutability, hashability, memory efficiency, unpacking) plus readable attribute access. Think of it as a lightweight class that requires no boilerplate — just a name and a list of fields.

Under the hood, namedtuple dynamically generates a class with __slots__, so instances use less memory than regular objects. The generated class includes a nice __repr__, comparison methods, and several utility methods prefixed with underscore (_asdict, _replace, _make).

Python
from collections import namedtuple

Point = namedtuple('Point', ['x', 'y'])
p = Point(3, 4)

print(p.x, p.y)        # named access
print(p[0], p[1])       # index access still works
print(f"Point at ({p.x}, {p.y})")
3 4 3 4 Point at (3, 4)
Python
Color = namedtuple('Color', 'red green blue', defaults=[0, 0, 0])

white = Color(255, 255, 255)
black = Color()           # uses all defaults
red = Color(red=255)      # keyword args work

print(black)
print(red)
print(white)
Color(red=0, green=0, blue=0) Color(red=255, green=0, blue=0) Color(red=255, green=255, blue=255)
Python
Point = namedtuple('Point', 'x y')
p = Point(3, 4)

print(p._asdict())       # convert to dict
p2 = p._replace(x=10)    # creates NEW tuple (immutable!)
print(p2)

data = [5, 6]
p3 = Point._make(data)   # create from iterable
print(p3)
{'x': 3, 'y': 4} Point(x=10, y=4) Point(x=5, y=6)
Python
Record = namedtuple('Record', 'name age city')
raw = ["Alice,30,NYC", "Bob,25,LA", "Carol,35,Chicago"]

records = [Record(*line.split(",")) for line in raw]
for r in records:
    print(f"{r.name} lives in {r.city}")
Alice lives in NYC Bob lives in LA Carol lives in Chicago
Python
Point = namedtuple('Point', 'x y')

grid = {Point(0, 0): 'start', Point(1, 1): 'end'}
print(grid[Point(0, 0)])
print(grid[Point(1, 1)])
start end
Common Mistakes:
Trying to modify a field: p.x = 10 raises AttributeError because namedtuples are immutable.
Use p._replace(x=10) to create a new instance with the changed field.
Naming the type differently from the variable: P = namedtuple('Point', ...) causes confusion in repr output since the repr shows Point(...) but your variable is P.
Keep them matching: Point = namedtuple('Point', ...) so the variable name matches the type name.
Using namedtuple when you need mutability — you cannot update fields in place.
Use namedtuple for immutable records. Use dataclass (Week 2) when you need mutable objects.
Key Rules:
  • Use namedtuple for small, immutable data records (points, colors, database rows)
  • namedtuple instances are tuples — they work with tuple unpacking, indexing, and as dict keys
  • Prefer dataclass (Week 2) when you need mutability, methods, or complex initialization

namedtuple is the lightweight precursor to dataclass (coming Week 2). It builds on tuple unpacking from Day 1. The _asdict() method connects to dict comprehensions.
ChainMap — Layered Dictionary Lookups Collections

When you need to search multiple dicts in priority order (CLI args > env vars > config file > defaults), ChainMap creates a single view without copying data. It’s how Python’s own scope resolution works internally.

ChainMap groups multiple dictionaries into a single mapping. Lookups search the underlying maps one by one in order and return the first match. Writes, updates, and deletions only affect the first (highest-priority) mapping — the rest remain untouched read-only layers.

This makes ChainMap ideal for layered configuration: you stack defaults at the bottom, environment overrides in the middle, and command-line arguments on top. You can also use new_child() and parents to push and pop scope layers, mirroring how Python resolves variable names through nested scopes.

Python
from collections import ChainMap

defaults = {'color': 'red', 'size': 'medium', 'debug': False}
user_prefs = {'color': 'blue'}

config = ChainMap(user_prefs, defaults)
print(config['color'])   # user_prefs wins
print(config['size'])    # falls through to defaults
print(config['debug'])
blue medium False
Python
from collections import ChainMap

cli_args = {'verbose': True}
env_vars = {'db_host': 'prod-db', 'verbose': False}
defaults = {'db_host': 'localhost', 'db_port': 5432, 'verbose': False}

config = ChainMap(cli_args, env_vars, defaults)
print(config['verbose'])   # cli_args has highest priority
print(config['db_host'])   # env_vars, since cli_args doesn't have it
print(config['db_port'])   # defaults
True prod-db 5432
Python
from collections import ChainMap

config = ChainMap({'a': 1}, {'a': 10, 'b': 20})
config['a'] = 99    # modifies first dict only
config['c'] = 30    # adds to first dict only

print(config.maps)
[{'a': 99, 'c': 30}, {'a': 10, 'b': 20}]
Python
from collections import ChainMap

base = ChainMap({'x': 1, 'y': 2})
scope = base.new_child({'x': 10})  # new layer on top

print(scope['x'])            # child overrides
print(scope['y'])            # falls through to parent
print(scope.parents['x'])   # access parent directly
10 2 1
Python
from collections import ChainMap

global_vars = {'title': 'My Site', 'year': 2026}
page_vars = {'title': 'About Us', 'author': 'Alice'}
context = ChainMap(page_vars, global_vars)

template = "© {year} {title} by {author}"
print(template.format_map(context))
© 2026 About Us by Alice
Common Mistakes:
Expecting ChainMap to merge or copy the dicts — it creates a VIEW over them. Modifying the original dict changes the ChainMap too.
Understand that ChainMap holds references, not copies. Changes to the underlying dicts are reflected immediately.
Using del config['key'] when the key is only in a child map — it only deletes from the first map. You get a KeyError if the key isn’t in the first map even if it exists in later maps.
Only delete or modify keys in the first map. Treat deeper maps as read-only layers.
Key Rules:
  • Use ChainMap when you need priority-based lookup across multiple dicts (config layering, scope chains)
  • Writes (set, delete) only affect the first/top-most dict
  • new_child() creates a new layer; parents removes the first layer — perfect for scope push/pop
  • For simple merging where you want a flat dict, use {**d1, **d2} instead

Decorators

Decorators wrap functions to add behavior without modifying their code. They’re Python’s most elegant pattern for cross-cutting concerns like logging, timing, caching, and access control.

Function Decorators & @wraps Decorators

Decorators let you modify function behavior transparently — add logging, timing, retries, caching, or validation without touching the original function. Every Python framework uses them (@app.route, @property, @staticmethod, @pytest.fixture).

A decorator is a function that takes another function as its argument and returns a new function (usually a wrapper) that adds some behavior before or after the original. The @decorator syntax placed above a function definition is syntactic sugar that reassigns the function name to the decorator’s return value.

The critical companion to any decorator is functools.wraps, which copies the original function’s name, docstring, and other metadata onto the wrapper. Without it, debugging and introspection tools see the wrapper’s identity instead of the original function’s.

Python
def shout(func):
    def wrapper(*args, **kwargs):
        result = func(*args, **kwargs)
        return result.upper()
    return wrapper

@shout
def greet(name):
    return f"hello, {name}"

print(greet("alice"))
HELLO, ALICE
Python
# The @decorator syntax is syntactic sugar.
# These two forms are equivalent:

@shout
def greet(name):
    return f"hello, {name}"

# Same as:
def greet(name):
    return f"hello, {name}"
greet = shout(greet)
# Both produce: greet is now the wrapper returned by shout
Python
# The problem: identity loss without @wraps
def shout(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs).upper()
    return wrapper

@shout
def greet(name):
    """Greet someone politely."""
    return f"hello, {name}"

print(greet.__name__)
print(greet.__doc__)
wrapper None
Python
# The fix: functools.wraps preserves metadata
from functools import wraps

def shout(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs).upper()
    return wrapper

@shout
def greet(name):
    """Greet someone politely."""
    return f"hello, {name}"

print(greet.__name__)
print(greet.__doc__)
greet Greet someone politely.
Python
# Real-world pattern: timing decorator
import time
from functools import wraps

def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(0.1)
    return "done"

result = slow_function()
print(result)
slow_function took 0.1003s done
Python
# Simple retry decorator (retries once on failure)
from functools import wraps

def retry_once(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        try:
            return func(*args, **kwargs)
        except Exception:
            return func(*args, **kwargs)  # one retry
    return wrapper

@retry_once
def flaky():
    import random
    if random.random() < 0.5:
        raise ConnectionError("failed")
    return "success"

print(flaky())  # retries once if first attempt fails
success
Common Mistakes:
Forgetting @wraps(func) — the decorated function loses its name and docstring, breaking introspection, debugging, and documentation tools.
ALWAYS use @wraps(func) on the inner wrapper function.
Forgetting *args, **kwargs in the wrapper — the decorator only works with functions that match a specific signature.
Always use def wrapper(*args, **kwargs) to accept any function signature.
Forgetting to return the result of func(*args, **kwargs) — the decorated function silently returns None.
Always return func(*args, **kwargs) unless you intentionally want to suppress the return value.
Key Rules:
  • A decorator is a function that takes a function and returns a (usually different) function
  • ALWAYS use @functools.wraps(func) to preserve the original function’s metadata
  • Use *args, **kwargs in the wrapper to make the decorator work with any function signature
  • Decorators run at definition time (when the module loads), not at call time

Stacking & Parameterized Decorators Decorators

Real code often combines multiple decorators (logging + timing + auth) and needs configurable decorators (retry with custom attempts, cache with custom TTL). These are the advanced patterns that separate decorator users from decorator authors.

When you stack decorators, they apply bottom-to-top: the decorator closest to the def wraps the function first, and each higher decorator wraps the result of the one below. This means the outermost decorator’s wrapper runs first at call time.

Parameterized decorators add another layer of nesting. A regular decorator is decorator(func) → wrapper. A parameterized decorator is factory(params) → decorator(func) → wrapper — three nested functions. The factory returns the actual decorator, which is why you call it with parentheses: @retry(3) first calls retry(3) to get the decorator, which then wraps the function.

Python
from functools import wraps

def bold(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        return f"<b>{func(*args, **kwargs)}</b>"
    return wrapper

def italic(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        return f"<i>{func(*args, **kwargs)}</i>"
    return wrapper

@bold
@italic
def greet(name):
    return f"Hello, {name}"

print(greet("Alice"))
# italic applied first (closest to function), then bold wraps the result
<b><i>Hello, Alice</i></b>

from functools import wraps

def repeat(n):
    """Decorator factory: returns a decorator that repeats n times."""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            results = []
            for _ in range(n):
                results.append(func(*args, **kwargs))
            return results
        return wrapper
    return decorator

@repeat(3)
def say_hello():
    return "hello"

print(say_hello())
['hello', 'hello', 'hello']

# 2. decorator(say_hello)    -> returns `wrapper`
# 3. say_hello is now `wrapper`

# Real-world pattern: configurable retry decorator
from functools import wraps
import time

def retry(max_attempts=3, delay=0.5):
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None
            for attempt in range(1, max_attempts + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    last_exception = e
                    if attempt < max_attempts:
                        print(f"  Attempt {attempt} failed, retrying...")
                        time.sleep(delay)
            raise last_exception
        return wrapper
    return decorator

@retry(max_attempts=3, delay=0.1)
def unstable_api():
    import random
    if random.random() < 0.7:
        raise ConnectionError("server down")
    return {"status": "ok"}

try:
    result = unstable_api()
    print(result)
except ConnectionError as e:
    print(f"All attempts failed: {e}")
Attempt 1 failed, retrying... Attempt 2 failed, retrying... {'status': 'ok'}

Common Mistakes:
Confusing @decorator (no parens) with @decorator() (with parens). The first passes the function directly to the decorator. The second calls the decorator factory first, then passes the function to its return value.
If your decorator takes NO arguments, don’t add empty parens: use @timer not @timer() (unless you’ve specifically designed it to handle both).
Getting the nesting level wrong — parameterized decorators need THREE levels of nested functions, not two.
Remember the pattern: def factory(params): → def decorator(func): → def wrapper(*args): — each returns the next level.
Forgetting that stacked decorators apply bottom-to-top (the one closest to def runs first).
Read stacked decorators from bottom to top: the bottom one wraps the function first, and the top one’s wrapper executes first at call time.
Key Rules:
  • Stacked decorators apply bottom-to-top: @A @B def f means f = A(B(f))
  • Parameterized decorators need 3 levels: outer (takes params) → middle (takes func) → inner (takes *args)
  • @decorator and @decorator() are DIFFERENT — the first is a simple decorator, the second is a decorator factory call
  • When combining stacked + parameterized, read the execution order carefully
