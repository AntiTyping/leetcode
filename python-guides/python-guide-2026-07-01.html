1. struct & Binary Data 3 days overdue rep 2 → 3

struct packs Python values into fixed-layout bytes and unpacks them back — the bridge between Python objects and binary file formats or network protocols. A format string describes the layout: a leading byte-order/alignment char (< little, > big, ! network) followed by type codes (i int, f float, H unsigned short, s bytes).

Key Pattern
import struct

# pack: values -> bytes (big-endian: int, unsigned short)
packed = struct.pack(">iH", 1, 65535)
print(packed)              # b'\x00\x00\x00\x01\xff\xff'
print(struct.calcsize(">iH"))   # 6  -- 4 + 2 bytes

# unpack: bytes -> tuple of values
print(struct.unpack(">iH", packed))   # (1, 65535)
Challenge

Pack a small record — a 4-byte magic number, a version H, and a d (double) — into bytes with an explicit byte order, then unpack it and assert you recovered the values. Now change > to < and observe how the bytes reorder. Finally, use struct.Struct(">iH") (the compiled form) and explain why reusing it in a loop is faster than calling struct.pack each time.

Pitfall

Without a leading byte-order char, native mode adds padding for alignment — calcsize("ic") may exceed the obvious sum. Use </>/! for wire formats so layout is exact and portable. Also, i is signed; packing 4294967295 as i raises struct.error (use I).

2. pytest Fundamentals 3 days overdue rep 2 → 3

pytest finds test_* functions and runs them, using plain assert (it rewrites the bytecode to show rich failure messages). Fixtures supply reusable setup via dependency injection, and @pytest.mark.parametrize turns one test into many cases. pytest.raises asserts that a block raises.

Key Pattern
import pytest

@pytest.fixture
def numbers():
    return [1, 2, 3]        # setup, injected by name

@pytest.mark.parametrize("n, sq", [(2, 4), (3, 9)])
def test_square(n, sq):
    assert n * n == sq        # two cases from one test

def test_sum(numbers):        # fixture arrives as an argument
    assert sum(numbers) == 6

def test_raises():
    with pytest.raises(ZeroDivisionError):
        1 / 0
Challenge

Write a fixture that yields a resource and cleans it up afterward (setup before yield, teardown after). Parametrize a test with three cases including one you expect to fail, and mark that one with pytest.param(..., marks=pytest.mark.xfail). Then use pytest.raises(ValueError, match="...") to assert both the exception type and its message.

Pitfall

A fixture that returns cannot clean up — use yield and put teardown after it. Test files/functions must match the discovery pattern (test_*) or pytest silently ignores them. And assert x == y inside a helper that swallows the exception hides failures — let assertions propagate.

3. Reference Counting & Garbage Collection 1 day overdue rep 2 → 3

CPython frees most objects the instant their reference count hits zero — deterministic and immediate. A separate cyclic garbage collector exists only to reclaim reference cycles (objects that reference each other, so their counts never reach zero). sys.getrefcount peeks at the count; the gc module controls the cycle collector.

Key Pattern
import sys, gc

x = [1, 2, 3]
print(sys.getrefcount(x))   # e.g. 2 -- x plus the temporary arg to getrefcount
y = x
print(sys.getrefcount(x))   # 3 -- one more binding

# a reference cycle: refcounts never hit 0 on their own
a = {}; b = {}
a["b"] = b; b["a"] = a
del a, b
print(gc.collect())        # >0 -- the cyclic collector reclaims them
Challenge

Create two objects that reference each other, delete your names for them, and show gc.collect() returns a nonzero count. Then attach a __del__ that prints on finalization and observe when it fires with and without the cycle. Finally, rewrite the pair to use weakref for the back-reference and show the cycle no longer needs the collector.

Pitfall

sys.getrefcount(x) always reads one higher than you expect — the argument itself is a reference. Reference cycles keep memory alive until the cyclic collector runs, which is periodic, not immediate; do not rely on __del__ timing. Cycles containing objects with __del__ were only reliably collected since Python 3.4.

4. String Interning & Integer Caching 1 day overdue rep 2 → 3

CPython caches small integers (-5 to 256) and interns some strings so identical values share one object — a memory optimization, not a language guarantee. This is why is sometimes appears to work on values: it is comparing identity, which coincidentally holds for cached objects. Always compare values with ==.

Key Pattern
a = 256; b = 256
print(a is b)     # True  -- 256 is cached (small-int pool)

c = 257; d = 257
print(c is d)     # often False -- outside the cached range

import sys
s1 = sys.intern("hello world")
s2 = sys.intern("hello world")
print(s1 is s2)   # True  -- forced interning shares one object
Challenge

Demonstrate the small-int boundary: show x is y is True at 256 and (typically) False at 257 for freshly computed values. Then take two strings built at runtime (e.g. via "".join(...)) that are equal but not interned, show is is False but == is True, and fix the identity check with sys.intern. Explain one real use of interning (dictionary key speed).

Pitfall

Using is to compare values is a bug that passes in the REPL (where literals get interned/cached) and fails in real code. Reserve is for None, True, False, and sentinels. Interning behavior differs across CPython versions and builds — never depend on it for correctness.

5. The GIL in Practice 1 day overdue rep 2 → 3

The Global Interpreter Lock lets only one thread execute Python bytecode at a time, so threads do not speed up CPU-bound work — but they help I/O-bound work, because the GIL is released during blocking calls (file, socket, time.sleep). For CPU parallelism, use multiprocessing (separate interpreters, no shared GIL).

Key Pattern
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def cpu_bound(n):
    return sum(i * i for i in range(n))

# threads: GIL serializes CPU work -> little/no speedup
with ThreadPoolExecutor() as ex:
    list(ex.map(cpu_bound, [10**6] * 4))

# processes: true parallelism, one GIL per process
with ProcessPoolExecutor() as ex:
    list(ex.map(cpu_bound, [10**6] * 4))
Challenge

Time the same CPU-bound function run 4x sequentially, then across 4 threads, then across 4 processes. Confirm threads give ~no speedup while processes scale with cores. Then swap in an I/O-bound task (many time.sleep or requests) and show threads do help there. Explain why numpy-heavy code can also parallelize under threads (it releases the GIL in C).

Pitfall

Reaching for threading to speed up number-crunching is the classic mistake — the GIL serializes it. Processes have real overhead (pickling args, spawn cost), so they only pay off for substantial work. Free-threaded builds (PEP 703, Python 3.13+ experimental) change this story, but assume the GIL by default.

6. abc Module — Abstract Base Classes 1 day overdue rep 2 → 3

An abstract base class defines an interface that subclasses must implement. Subclass ABC (or set metaclass=ABCMeta) and mark required methods with @abstractmethod: instantiating a subclass that leaves any abstract method unimplemented raises TypeError. This turns "forgot to implement" from a runtime AttributeError into an early, clear error.

Key Pattern
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self): ...      # subclasses MUST define this

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s ** 2

print(Square(3).area())   # 9
# Shape()  -> TypeError: Can't instantiate abstract class
Challenge

Define an abstract Serializer with an @abstractmethod dump(obj) and a concrete helper method that calls it. Show that a subclass missing dump cannot be instantiated. Then combine @property with @abstractmethod to require an abstract read-only property, and register an unrelated class as a virtual subclass with Serializer.register(...) — noting that register does not enforce the methods.

Pitfall

Order matters when stacking: @property goes above @abstractmethod (abstractmethod must be the innermost decorator). Forgetting to inherit from ABC (or use ABCMeta) makes @abstractmethod a silent no-op — the class becomes instantiable with the method missing. register() grants isinstance without checking the interface.

New Concepts — Iteration & Aggregation Built-ins (3 concepts)

The built-ins you reach for in nearly every loop. zip/enumerate pair and index iterables without manual counters; sorted/min/max with a key= function order data by any criterion; and any/all/map/filter express aggregation and transformation lazily. Mastering these replaces piles of index bookkeeping with one clear expression.

Built-in Functions

Concept 1: zip & enumerate — Pairing and Indexing Iterables

Why learn this? The for i in range(len(xs)) anti-pattern is everywhere — and almost always wrong. enumerate gives you index and item together; zip walks several iterables in lockstep. Both are lazy iterators, so they cost nothing until consumed, and both read like the intent they express.
Example 1 — enumerate instead of a manual counter
colors = ["red", "green", "blue"]

for i, c in enumerate(colors):
    print(i, c)          # 0 red / 1 green / 2 blue

# start= sets the first index
for i, c in enumerate(colors, start=1):
    print(i, c)          # 1 red / 2 green / 3 blue
Output
0 red 1 green 2 blue 1 red 2 green 3 blue
Example 2 — zip walks iterables in parallel
names = ["Ann", "Bo", "Cy"]
ages  = [30, 25, 41]

for name, age in zip(names, ages):
    print(name, age)         # Ann 30 / Bo 25 / Cy 41

# zip stops at the SHORTEST iterable
print(list(zip([1, 2, 3], ["a", "b"])))   # [(1, 'a'), (2, 'b')]
Output
Ann 30 Bo 25 Cy 41 [(1, 'a'), (2, 'b')]
Example 3 — zip(*rows) transposes / unzips
pairs = [("Ann", 30), ("Bo", 25), ("Cy", 41)]

names, ages = zip(*pairs)      # unzip back into two tuples
print(names)                # ('Ann', 'Bo', 'Cy')
print(ages)                 # (30, 25, 41)

# the same trick transposes a matrix
matrix = [[1, 2, 3], [4, 5, 6]]
print([list(col) for col in zip(*matrix)])   # [[1, 4], [2, 5], [3, 6]]
Output
('Ann', 'Bo', 'Cy') (30, 25, 41) [[1, 4], [2, 5], [3, 6]]
Example 4 — zip(strict=True) catches length mismatches
# Python 3.10+: raise if the iterables differ in length
try:
    list(zip([1, 2, 3], ["a", "b"], strict=True))
except ValueError as e:
    print("mismatch:", e)   # zip() argument 2 is shorter than argument 1

# build a dict straight from two parallel lists
keys = ["x", "y", "z"]
vals = [1, 2, 3]
print(dict(zip(keys, vals)))   # {'x': 1, 'y': 2, 'z': 3}
Output
mismatch: zip() argument 2 is shorter than argument 1 {'x': 1, 'y': 2, 'z': 3}

Common Mistakes

✗ Looping with for i in range(len(xs)): xs[i] to get index and item

✓ Use for i, x in enumerate(xs) — clearer, and it works on any iterable, not just sequences


✗ Assuming zip pads the short iterable — it silently stops at the shortest, dropping trailing data

✓ Pass strict=True (3.10+) to fail loud on mismatched lengths, or itertools.zip_longest to pad

Key Rules

  • enumerate(it, start=0) yields (index, item) pairs lazily — use start= to offset the count
  • zip(a, b, ...) yields tuples until the shortest input is exhausted; strict=True raises on length mismatch
  • zip(*rows) transposes/unzips; both zip and enumerate return one-shot iterators — wrap in list() to reuse

Exercises

  1. Given a list of words, build a dict mapping each word to its 1-based position using enumerate(..., start=1).
  2. Given three parallel lists (names, scores, ranks), zip them and print one formatted line per person.
  3. Transpose a 3×2 nested list into a 2×3 one using zip(*rows), and confirm round-tripping it twice returns the original.
Connections: zip/enumerate feed dict() and comprehensions from Day 1–2; zip_longest lives in itertools (reviewed earlier); the lazy-iterator behavior mirrors map/filter (Concept 3); unzipping with zip(*...) uses star-unpacking from the Unpacking topic.
Built-in Functions

Concept 2: sorted, min & max with key= — Ordering by Any Criterion

Why learn this? sorted, min, and max all accept a key= function that maps each element to the value you actually want to compare. This one idea replaces custom comparison code entirely: sort objects by an attribute, find the longest string, order tuples by their second field — all with a tiny lambda. Python's sort is also stable, which unlocks multi-level sorting.
Example 1 — key= transforms what gets compared
words = ["banana", "kiwi", "cherry"]

print(sorted(words))                 # ['banana', 'cherry', 'kiwi']  (alphabetical)
print(sorted(words, key=len))        # ['kiwi', 'banana', 'cherry'] (by length)
print(max(words, key=len))           # banana  (longest)
print(min(words, key=len))           # kiwi    (shortest)
Output
['banana', 'cherry', 'kiwi'] ['kiwi', 'banana', 'cherry'] banana kiwi
Example 2 — sort objects/tuples by a field
people = [("Ann", 30), ("Bo", 25), ("Cy", 41)]

# by age (index 1), descending
print(sorted(people, key=lambda p: p[1], reverse=True))
# [('Cy', 41), ('Ann', 30), ('Bo', 25)]

# oldest person -> max by the age field
print(max(people, key=lambda p: p[1]))   # ('Cy', 41)
Output
[('Cy', 41), ('Ann', 30), ('Bo', 25)] ('Cy', 41)
Example 3 — operator.itemgetter and multi-level sort
from operator import itemgetter

rows = [("eng", "Ann"), ("eng", "Bo"), ("sci", "Al")]

# sort by dept, then name -- a tuple key sorts field by field
print(sorted(rows, key=itemgetter(0, 1)))
# [('eng', 'Ann'), ('eng', 'Bo'), ('sci', 'Al')]

# itemgetter(1) is a faster, clearer lambda p: p[1]
print(sorted(rows, key=itemgetter(1)))
Output
[('eng', 'Ann'), ('eng', 'Bo'), ('sci', 'Al')] [('sci', 'Al'), ('eng', 'Ann'), ('eng', 'Bo')]
Example 4 — stability enables layered sorting
data = [("Ann", 2), ("Bo", 1), ("Cy", 2), ("Al", 1)]

# sort by name first, THEN by the numeric group:
# equal groups keep the name order (stable sort)
step1 = sorted(data, key=itemgetter(0))
step2 = sorted(step1, key=itemgetter(1))
print(step2)
# [('Al', 1), ('Bo', 1), ('Ann', 2), ('Cy', 2)]

# empty input: min/max need a default to avoid ValueError
print(max([], default=None))   # None
Output
[('Al', 1), ('Bo', 1), ('Ann', 2), ('Cy', 2)] None

Common Mistakes

✗ Calling the key function yourself: sorted(xs, key=len(x)) — that passes a number, not a callable

✓ Pass the function object: key=len or key=lambda x: ... — Python calls it once per element


✗ Calling max([]) / min([]) on a possibly-empty iterable — raises ValueError

✓ Pass default= (e.g. max(xs, default=0)) so empty input returns a sensible value

Key Rules

  • key= is a callable applied to each element; the result (not the element) is compared — use len, str.lower, a lambda, or operator.itemgetter/attrgetter
  • sorted(...) returns a new list (leaving the input untouched); list.sort() mutates in place — both take key= and reverse=
  • Python's sort is stable: sort by the least-significant key first, then re-sort by the primary key; min/max accept default= for empty iterables

Exercises

  1. Sort a list of (name, score) tuples by score descending, breaking ties by name ascending — do it with a single tuple-returning key.
  2. Find the word with the most distinct letters in a list using max with key=lambda w: len(set(w)).
  3. Given a possibly-empty list of numbers, return the largest or 0 if empty, using max(..., default=0).
Connections: key= functions are often lambdas (Functions topic) or operator.itemgetter/attrgetter (operator module, reviewed earlier); stability matters when sorting the same data on multiple fields; sorted pairs with heapq.nlargest for top-k, and its key idea reappears in groupby.

Concept 3: any, all, map & filter — Aggregating and Transforming Lazily

Why learn this? any and all collapse an iterable of booleans into a single yes/no answer — and they short-circuit, stopping at the first decisive element. map and filter lazily transform and select. Paired with generator expressions, they let you ask "does any row match?" or "are all values valid?" in one readable line, over data of any size, without building intermediate lists.
Example 1 — any / all over a generator
nums = [2, 4, 6, 7]

print(all(n % 2 == 0 for n in nums))   # False -- 7 is odd
print(any(n > 5 for n in nums))        # True  -- 6 (and 7)

# empty-iterable identities (worth memorizing)
print(all([]))   # True  -- "no counterexample"
print(any([]))   # False -- "no witness"
Output
False True True False
Example 2 — short-circuit avoids extra work
def check(n):
    print(f"checking {n}")
    return n > 3

# any stops at the first True -> check(4) never runs
print(any(check(n) for n in [1, 2, 5, 4]))
Output
checking 1 checking 2 checking 5 True
Example 3 — map and filter (lazy iterators)
nums = [1, 2, 3, 4, 5]

squares = map(lambda x: x * x, nums)     # lazy: nothing computed yet
evens   = filter(lambda x: x % 2 == 0, nums)

print(list(squares))   # [1, 4, 9, 16, 25]
print(list(evens))     # [2, 4]

# map over multiple iterables -> like zip + apply
print(list(map(lambda a, b: a + b, [1, 2], [10, 20])))   # [11, 22]
Output
[1, 4, 9, 16, 25] [2, 4] [11, 22]
Example 4 — comprehensions are usually clearer
nums = [1, 2, 3, 4, 5]

# these two are equivalent; the comprehension reads better
print([x * x for x in nums if x % 2 == 0])          # [4, 16]
print(list(map(lambda x: x * x, filter(lambda x: x % 2 == 0, nums))))

# but map(func, xs) shines when func already exists
print(list(map(str, nums)))   # ['1', '2', '3', '4', '5']
Output
[4, 16] [4, 16] ['1', '2', '3', '4', '5']

Common Mistakes

✗ Consuming a map/filter object twice — it is a one-shot iterator, so the second pass is empty

✓ Materialize with list(...) if you need it more than once, or just use a comprehension


✗ Building a full list just to test a condition: len([x for x in xs if pred(x)]) > 0

✓ Use any(pred(x) for x in xs) — it short-circuits and never allocates the list

Key Rules

  • any returns True at the first truthy element (else False); all returns False at the first falsy element (else True) — both short-circuit; any([]) is False, all([]) is True
  • Feed any/all a generator expression, not a list comprehension, to keep the short-circuit and avoid allocating
  • map/filter return lazy one-shot iterators; a comprehension is usually clearer, but map(existing_func, xs) is idiomatic and fast

Exercises

  1. Write is_valid(records) that returns True only if every record has a non-empty "name", using all with a generator.
  2. Given a list of strings, use any to report whether at least one contains a digit — and confirm it stops early by printing inside the predicate.
  3. Rewrite list(map(lambda x: x.upper(), filter(str.isalpha, words))) as a single list comprehension and check both give the same result.
Connections: any/all pair with generator expressions (Day 2) to stay lazy; map/filter overlap with comprehensions (Comprehensions topic) and with itertools tools like starmap; the short-circuit behavior echoes Python's and/or; map(func, ...) is cleanest when func comes from operator or is a built-in like str.

New Concepts — Numbers & Sets (2 concepts)

Two corners of core Python worth knowing cold: the numeric built-ins that do integer math and base conversions without importing anything (divmod, pow, round, bin/hex/oct), and the set/frozenset types whose algebraic operators (union, intersection, difference) turn membership problems into one-liners.

Numbers

Concept 4: Numeric Built-ins & Base Conversions

Why learn this? A handful of built-ins cover the integer arithmetic you reach for constantly — divmod for quotient-and-remainder in one call, three-argument pow for efficient modular exponentiation, and bin/hex/oct/int(s, base) to move between bases. No imports, no reinventing the wheel, and they handle Python's arbitrary-precision integers natively.
Example 1 — divmod, abs, round
print(divmod(17, 5))    # (3, 2)  -- (17 // 5, 17 % 5) in one call
q, r = divmod(17, 5)
print(q, r)             # 3 2

print(abs(-4.5))       # 4.5
print(round(3.14159, 2))   # 3.14  -- round to 2 decimals
print(round(2.5), round(3.5))   # 2 4  -- banker's rounding (ties to even)
Output
(3, 2) 3 2 4.5 3.14 2 4
Example 2 — three-argument pow (modular)
print(pow(2, 10))        # 1024  -- same as 2 ** 10

# pow(base, exp, mod) == (base ** exp) % mod, but far faster
print(pow(2, 1000, 1000))   # 376  -- never builds the huge 2**1000

# modular inverse (Python 3.8+): exp = -1
print(pow(3, -1, 7))       # 5  -- because 3*5 % 7 == 1
Output
1024 376 5
Example 3 — base conversions to and from strings
n = 255
print(bin(n))    # 0b11111111
print(oct(n))    # 0o377
print(hex(n))    # 0xff

# int(str, base) parses back -- base 0 reads the prefix
print(int("ff", 16))     # 255
print(int("0b1010", 0))  # 10  -- base 0 honors the 0b prefix
Output
0b11111111 0o377 0xff 255 10
Example 4 — format() for clean base output
n = 255
# format spec drops the 0b/0x prefix and can pad
print(format(n, "b"))      # 11111111  (no 0b)
print(format(n, "08b"))    # 11111111  (zero-padded to width 8)
print(format(10, "08b"))   # 00001010
print(format(n, "x"))      # ff
print(int.bit_count(255) if hasattr(int, "bit_count") else bin(255).count("1"))   # 8 set bits
Output
11111111 11111111 00001010 ff 8

Common Mistakes

✗ Expecting round(2.5) == 3 — Python uses banker's rounding (round-half-to-even), so it is 2

✓ Know the rule (ties go to the even neighbor); for money use decimal.Decimal with an explicit rounding mode


✗ Computing (base ** exp) % mod for huge exponents — it builds an enormous integer first

✓ Use three-argument pow(base, exp, mod), which stays fast and memory-light via modular exponentiation

Key Rules

  • divmod(a, b) returns (a // b, a % b) together; pow(b, e, m) is efficient modular exponentiation and pow(b, -1, m) gives the modular inverse (3.8+)
  • bin/oct/hex return prefixed strings (0b/0o/0x); int(s, base) parses them back, and base=0 infers the base from the prefix
  • For prefix-free or padded output use format(n, "b"/"x"/"08b") or an f-string (f"{n:08b}"); round uses banker's rounding on ties

Exercises

  1. Convert 3661 seconds into hours, minutes, seconds using divmod twice, and print H:MM:SS.
  2. Compute pow(7, 256, 13) two ways — with three-arg pow and with (7 ** 256) % 13 — and confirm they match.
  3. Round-trip a number: format it as a zero-padded 8-bit binary string, then parse it back with int(s, 2) and assert equality.
Connections: three-arg pow underpins cryptography (RSA), tying into hashlib/secrets; base strings relate to binascii.hexlify and bytes.hex() (Day 30); format(n, "08b") uses the format-spec mini-language; for exact decimals prefer decimal.Decimal over round.
Data Structures

Concept 5: set & frozenset — Membership and Algebra

Why learn this? A set is an unordered collection of unique, hashable elements with O(1) membership tests — and it supports mathematical operators: union (|), intersection (&), difference (-), symmetric difference (^). Deduplicating a list, finding common items, or testing subset relationships each become a single expression. frozenset is the immutable, hashable version — usable as a dict key or set element.
Example 1 — uniqueness and fast membership
nums = [1, 2, 2, 3, 3, 3]
s = set(nums)
print(s)              # {1, 2, 3}  -- duplicates gone
print(2 in s)         # True  -- O(1) average, unlike list's O(n)
print(len(s))         # 3

s.add(4)
s.discard(99)         # discard: no error if absent (remove would raise)
print(sorted(s))      # [1, 2, 3, 4]
Output
{1, 2, 3} True 3 [1, 2, 3, 4]
Example 2 — the algebra operators
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

print(a | b)   # {1, 2, 3, 4, 5, 6}  union
print(a & b)   # {3, 4}              intersection
print(a - b)   # {1, 2}              difference (in a, not b)
print(a ^ b)   # {1, 2, 5, 6}        symmetric difference (in exactly one)
Output
{1, 2, 3, 4, 5, 6} {3, 4} {1, 2} {1, 2, 5, 6}
Example 3 — subset / superset tests
required = {"name", "email"}
provided = {"name", "email", "age"}

print(required <= provided)   # True  -- required is a subset
print(required.issubset(provided))       # True  -- same test, named
print(provided >= required)   # True  -- superset

missing = required - provided
print(missing or "nothing missing")   # nothing missing (empty set is falsy)

print({1, 2}.isdisjoint({3, 4}))   # True  -- no common elements
Output
True True True nothing missing True
Example 4 — frozenset: hashable, usable as a key
fs = frozenset([1, 2, 3])
print(fs & frozenset([2, 3, 4]))   # frozenset({2, 3})

# a normal set can't be a dict key (unhashable); frozenset can
regions = {
    frozenset({"US", "CA"}): "north-america",
    frozenset({"FR", "DE"}): "europe",
}
print(regions[frozenset({"CA", "US"})])   # north-america (order-independent key)

# a set OF sets requires frozensets as elements
print({frozenset({1}), frozenset({1})})   # {frozenset({1})}  -- deduped
Output
frozenset({2, 3}) north-america {frozenset({1})}

Common Mistakes

✗ Writing {} for an empty set — that is an empty dict

✓ Use set() for an empty set; {1, 2} only makes a set when it has elements


✗ Putting a mutable object (a list or set) into a set, or using one as a dict key — raises TypeError: unhashable

✓ Convert to a tuple or frozenset first — only hashable (effectively immutable) values can be set elements or dict keys

Key Rules

  • Sets hold unique, hashable elements with average O(1) in; they are unordered, so never rely on iteration order
  • Operators: | union, & intersection, - difference, ^ symmetric difference; comparisons <=/>= test subset/superset, isdisjoint tests no overlap
  • remove raises KeyError if absent, discard does not; frozenset is the immutable, hashable variant — use it as a dict key or set element

Exercises

  1. Given two lists of tags, print the tags common to both, the ones unique to the first, and all tags combined — using &, -, and |.
  2. Deduplicate a list while checking membership: read numbers one by one, skip any you have already seen using a set, and collect the first-seen order in a list.
  3. Build a dict keyed by frozenset of a pair of user ids to store a "friendship since" date, and show that lookup works regardless of the pair's order.
