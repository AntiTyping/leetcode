F-String Power Features

Strings • 3 days since last review • rep 1

Quick Reminder

F-strings support = for debugging (f"{x=}"), format specs (f"{pi:.2f}"), and inline expressions (f"{len(items)}").

Key Pattern

name = "Alice"
print(f"{name=}, {len(name)=}")
# Output: name='Alice', len(name)=5
Challenge

Write a function that takes a list of floats and returns a formatted string showing each value with 3 decimal places and its index, like "[0] 3.142, [1] 2.718".

#1 Pitfall: Forgetting that = includes the expression text AND the value — f"{x+1=}" prints x+1=6, not just 6.

Unpacking & Star Expressions

Unpacking • 3 days since last review • rep 1

Quick Reminder

*rest captures remaining items. a, *mid, z = [1,2,3,4,5] gives mid=[2,3,4]. Works in function args, assignments, and for loops.

Key Pattern

first, *rest = [1, 2, 3, 4]
print(first)  # 1
print(rest)   # [2, 3, 4]
Challenge

Given a list of (name, *scores) tuples like [("Alice", 90, 85, 92), ("Bob", 78, 88)], unpack each to compute the average score per person. Return a dict of {name: avg}.

#1 Pitfall: *rest always produces a list, even for a single remaining element — a, *b = [1, 2] gives b=[2] not b=2.

Counter

Collections • 3 days since last review • rep 1

Quick Reminder

Counter counts hashable objects. Supports most_common(n), arithmetic (+, -, &, |), and iteration.

Key Pattern

from collections import Counter

counts = Counter("mississippi")
print(counts.most_common(2))
# Output: [('s', 4), ('i', 4)]
Challenge

Given two strings, use Counter to determine if one is an anagram of the other, then find which characters need to be added/removed to transform one into the other.

#1 Pitfall: Counter subtraction drops zero/negative counts — use subtract() method to keep them.

Closures & nonlocal

Functions • 1 day since last review • rep 0

Quick Reminder

A closure captures variables from its enclosing scope. Use nonlocal to modify (not just read) an enclosed variable.

Key Pattern

def make_counter():
    count = 0
    def inc():
        nonlocal count
        count += 1
        return count
    return inc

c = make_counter()
print(c())  # 1
print(c())  # 2
Challenge

Write a make_accumulator() factory that returns a function. Each call to the returned function adds its argument to a running total and returns the new total. E.g., acc = make_accumulator(); acc(5) -> 5; acc(3) -> 8; acc(10) -> 18.

#1 Pitfall: Without nonlocal, assigning to a variable creates a NEW local variable instead of modifying the enclosed one, causing UnboundLocalError.

itertools

Iteration • 1 day since last review • rep 0

Quick Reminder

itertools provides lazy iteration tools: chain (concatenate), islice (slice iterators), groupby (group consecutive), product (cartesian), combinations/permutations, accumulate (running totals).

Key Pattern

from itertools import chain, islice

result = list(islice(chain([1, 2], [3, 4], [5, 6]), 4))
print(result)
# Output: [1, 2, 3, 4]
Challenge

Using itertools.groupby, take a sorted list of (category, item) tuples and produce a dict mapping each category to its list of items. E.g., [("fruit","apple"),("fruit","banana"),("veg","carrot")] becomes {"fruit":["apple","banana"],"veg":["carrot"]}.

#1 Pitfall: groupby only groups CONSECUTIVE equal elements — data must be sorted by the key first.

Function Decorators & @wraps

Decorators • 1 day since last review • rep 1

Quick Reminder

A decorator wraps a function, adding behavior before/after. Use @functools.wraps(func) to preserve the original function's __name__, __doc__, etc.

Key Pattern

import functools

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        import time
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper
Challenge

Write a @retry(max_attempts=3) parameterized decorator that retries a function up to N times if it raises an exception, with a 1-second delay between attempts.

#1 Pitfall: Forgetting @wraps(func) means the decorated function loses its name and docstring — help() and debugging tools show the wrapper's info instead.

Type System

Type Hints — Basic Annotations

Type System • New

Why It Matters

Type hints make code self-documenting, enable IDE autocompletion, and catch bugs via mypy or pyright before runtime. They communicate intent to both tools and humans.

Progressive Explanation

Simple function annotations: Add types to parameters and return values.

def greet(name: str) -> str:
    return f"Hello, {name}"

Variable annotations:

count: int = 0
names: list[str] = []
active: bool = True

Collection types: Specify what the container holds.

scores: dict[str, int] = {"alice": 95, "bob": 87}
pairs: list[tuple[str, int]] = [("alice", 95)]
unique_ids: set[float] = {1.0, 2.5}

None returns: Functions that don't return a value.

def log(msg: str) -> None:
    print(msg)

Code Examples

Example 1: Basic types

def calculate_bmi(weight_kg: float, height_m: float) -> float:
    """Calculate Body Mass Index."""
    return weight_kg / (height_m ** 2)

def is_healthy(bmi: float) -> bool:
    return 18.5 <= bmi <= 24.9

bmi = calculate_bmi(70.0, 1.75)
print(f"BMI: {bmi:.1f}, Healthy: {is_healthy(bmi)}")
# Output: BMI: 22.9, Healthy: True

Example 2: Collection types

def summarize_scores(scores: dict[str, list[int]]) -> dict[str, float]:
    """Compute average score per student."""
    return {
        name: sum(grades) / len(grades)
        for name, grades in scores.items()
    }

data = {"Alice": [90, 85, 92], "Bob": [78, 88, 95]}
print(summarize_scores(data))
# Output: {'Alice': 89.0, 'Bob': 87.0}

Example 3: Class with typed attributes

class Rectangle:
    width: float
    height: float

    def __init__(self, width: float, height: float) -> None:
        self.width = width
        self.height = height

    def area(self) -> float:
        return self.width * self.height

    def perimeter(self) -> float:
        return 2 * (self.width + self.height)

r = Rectangle(5.0, 3.0)
print(f"Area: {r.area()}, Perimeter: {r.perimeter()}")
# Output: Area: 15.0, Perimeter: 16.0

Example 4: Real-world — processing CSV-like data

def filter_records(
    records: list[dict[str, str]],
    key: str,
    value: str
) -> list[dict[str, str]]:
    """Filter records where key matches value."""
    return [r for r in records if r.get(key) == value]

def extract_column(
    records: list[dict[str, str]],
    column: str
) -> list[str]:
    """Extract a single column from all records."""
    return [r[column] for r in records if column in r]

records = [
    {"name": "Alice", "dept": "Eng"},
    {"name": "Bob", "dept": "Sales"},
    {"name": "Carol", "dept": "Eng"},
]
eng = filter_records(records, "dept", "Eng")
names = extract_column(eng, "name")
print(names)
# Output: ['Alice', 'Carol']

Common Mistakes

Don't
def foo(items: list):
    ...
# Too vague -- what's IN the list?
Do
def foo(items: list[str]):
    ...
# Specific element type
Don't (Python 3.9+)
from typing import List, Dict
def bar(x: List[str]) -> Dict[str, int]:
    ...
Do (Python 3.9+)
def bar(x: list[str]) -> dict[str, int]:
    ...
# Use lowercase built-in generics

Key Rules

  • Type hints are NOT enforced at runtime — they are for tools (mypy, pyright) and humans.
  • Use lowercase built-in generics (list, dict, set, tuple) in Python 3.9+.
  • Always annotate public function signatures; internal helpers are optional.

Type Hints — Optional, Union & Modern Syntax

Type System • New

Why It Matters

Real code has values that might be None or could be multiple types. Optional and Union (or the X | Y syntax) express these patterns clearly.

Progressive Explanation

Optional[str] means str | None — the value might be None.

from typing import Optional

def find_user(user_id: int) -> Optional[str]:
    users = {1: "Alice", 2: "Bob"}
    return users.get(user_id)  # Returns str or None

Union[int, str] means either int or str.

from typing import Union

def stringify(val: Union[int, float, str]) -> str:
    return str(val)

Python 3.10+ syntax: Use | directly — cleaner and more Pythonic.

# Python 3.10+
def find_user(user_id: int) -> str | None:
    ...

def stringify(val: int | float | str) -> str:
    return str(val)

Backporting: from __future__ import annotations enables the | syntax in Python 3.7–3.9.

Code Examples

Example 1: Function returning Optional

def find_user(user_id: int) -> str | None:
    """Look up a user by ID; returns None if not found."""
    database = {1: "Alice", 2: "Bob", 3: "Carol"}
    return database.get(user_id)

result = find_user(2)
print(result)   # Output: Bob

result = find_user(99)
print(result)   # Output: None

Example 2: Function accepting Union

def format_value(val: int | float | str) -> str:
    """Convert any supported type to a display string."""
    if isinstance(val, float):
        return f"{val:.2f}"
    return str(val)

print(format_value(42))       # Output: 42
print(format_value(3.14159))  # Output: 3.14
print(format_value("hello"))  # Output: hello

Example 3: Default None pattern

def connect(
    host: str,
    port: int = 8080,
    timeout: float | None = None
) -> str:
    """Simulate a connection with optional timeout."""
    msg = f"Connecting to {host}:{port}"
    if timeout is not None:
        msg += f" (timeout={timeout}s)"
    return msg

print(connect("localhost"))
# Output: Connecting to localhost:8080
print(connect("db.example.com", 5432, timeout=30.0))
# Output: Connecting to db.example.com:5432 (timeout=30.0s)

Example 4: Real-world database lookup

def get_setting(
    settings: dict[str, str],
    key: str,
    default: str | None = None
) -> str | None:
    """Retrieve a setting, with optional default."""
    return settings.get(key, default)

config = {"theme": "dark", "lang": "en"}
print(get_setting(config, "theme"))          # Output: dark
print(get_setting(config, "missing"))        # Output: None
print(get_setting(config, "missing", "N/A")) # Output: N/A

Common Mistakes

Don't
def foo() -> str:
    return None
# Lying about the return type!
Do
def foo() -> str | None:
    return None
# Honest Optional return
Don't
# Optional doesn't mean "has a default"
name: Optional[str] = "default"
# Misleading -- Optional is about TYPE
Do
# Optional means "could be str or None"
name: str | None = None
# Clear: the type can be None

Key Rules

  • Optional[X] is EXACTLY X | None — nothing more.
  • Prefer X | Y syntax (3.10+) over Union[X, Y] — cleaner and more Pythonic.
  • If a function NEVER returns None, don't use Optional in its return type.

Exercise 2

Data Modeling

Enum

Data Modeling • New

Why It Matters

Enums define a fixed set of named constants — more readable than magic strings or numbers, with IDE support and type safety.

Progressive Explanation

Basic enum:

from enum import Enum

class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3

Accessing members:

print(Color.RED)        # Color.RED
print(Color.RED.name)   # RED
print(Color.RED.value)  # 1

Iteration:

for c in Color:
    print(c.name, c.value)
# RED 1
# GREEN 2
# BLUE 3

IntEnum for integer comparisons, auto() for auto-numbering:

from enum import IntEnum, auto

class Priority(IntEnum):
    LOW = auto()     # 1
    MEDIUM = auto()  # 2
    HIGH = auto()    # 3

print(Priority.HIGH > Priority.LOW)  # True

String enum for serializable values:

class Status(str, Enum):
    ACTIVE = "active"
    INACTIVE = "inactive"
    PENDING = "pending"

import json
print(json.dumps({"status": Status.ACTIVE}))
# Output: {"status": "active"}

Code Examples

Example 1: Basic enum definition and access

from enum import Enum

class Season(Enum):
    SPRING = 1
    SUMMER = 2
    AUTUMN = 3
    WINTER = 4

current = Season.SUMMER
print         # Season.SUMMER
print    # SUMMER
print   # 2
print(Season(3))       # Season.AUTUMN  (lookup by value)
print(Season["WINTER"])# Season.WINTER  (lookup by name)

Example 2: Using enum in conditionals

from enum import Enum

class TrafficLight(Enum):
    RED = "stop"
    YELLOW = "caution"
    GREEN = "go"

def action(light: TrafficLight) -> str:
    if light == TrafficLight.RED:
        return "Stop immediately"
    elif light == TrafficLight.YELLOW:
        return "Slow down"
    elif light == TrafficLight.GREEN:
        return "Proceed"
    return "Unknown"

print(action(TrafficLight.GREEN))
# Output: Proceed

Example 3: String enum for API status codes

from enum import Enum

class HttpStatus(str, Enum):
    OK = "200 OK"
    NOT_FOUND = "404 Not Found"
    SERVER_ERROR = "500 Internal Server Error"

def make_response(status: HttpStatus, body: str) -> dict:
    return {"status": status.value, "body": body}

resp = make_response(HttpStatus.OK, "Hello!")
print(resp)
# Output: {'status': '200 OK', 'body': 'Hello!'}

Example 4: HTTP method enum with dispatch

from enum import Enum

class HttpMethod(str, Enum):
    GET = "GET"
    POST = "POST"
    PUT = "PUT"
    DELETE = "DELETE"

def handle_request(method: HttpMethod, path: str) -> str:
    handlers = {
        HttpMethod.GET: f"Fetching {path}",
        HttpMethod.POST: f"Creating at {path}",
        HttpMethod.PUT: f"Updating {path}",
        HttpMethod.DELETE: f"Deleting {path}",
    }
    return handlers.get(method, "Unknown method")

print(handle_request(HttpMethod.DELETE, "/users/42"))
# Output: Deleting /users/42

Common Mistakes

Don't
# Magic strings scattered through code
if status == "active":
    ...
if status == "actve":  # typo!
    ...
Do
if status == Status.ACTIVE:
    ...
# Centralized, typo-proof, IDE-completable
Don't
# Comparing enum to raw value
color = Color.RED
if color == 1:  # Works only with IntEnum!
    ...
Do
# Compare enum to enum
color = Color.RED
if color == Color.RED:
    ...

Key Rules

  • Use Enum when you have a fixed, known set of values.
  • Use str, Enum mixin when enum values need to serialize as strings (JSON, APIs).
  • Don't use Enum for open-ended sets (e.g., user names).

Dataclasses

Data Modeling • New

Why It Matters

@dataclass auto-generates __init__, __repr__, __eq__, and more — eliminating boilerplate for data-holding classes.

Progressive Explanation

Basic dataclass:

from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

p = Point(3.0, 4.0)
print(p)  # Point(x=3.0, y=4.0)

With defaults:

@dataclass
class Config:
    debug: bool = False
    retries: int = 3
    verbose: bool = True

field() for mutable defaults:

from dataclasses import dataclass, field

@dataclass
class ShoppingCart:
    items: list[str] = field(default_factory=list)
    quantities: dict[str, int] = field(default_factory=dict)

Frozen (immutable):

@dataclass(frozen=True)
class Coord:
    lat: float
    lon: float

c = Coord(40.7128, -74.0060)
# c.lat = 0  # FrozenInstanceError!

Post-init processing:

@dataclass
class User:
    first_name: str
    last_name: str
    full_name: str = field(init=False)

    def __post_init__(self):
        self.full_name = f"{self.first_name} {self.last_name}"

Serialization with asdict():

from dataclasses import asdict, astuple

p = Point(3.0, 4.0)
print(asdict(p))    # {'x': 3.0, 'y': 4.0}
print(astuple(p))   # (3.0, 4.0)

Code Examples

Example 1: Dataclass vs manual class (boilerplate savings)

# Manual -- 12 lines of boilerplate
class PointManual:
    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y
    def __repr__(self):
        return f"PointManual(x={self.x}, y={self.y})"
    def __eq__(self, other):
        if not isinstance(other, PointManual):
            return NotImplemented
        return self.x == other.x and self.y == other.y

# Dataclass -- 4 lines, same functionality
from dataclasses import dataclass

@dataclass
class PointDC:
    x: float
    y: float

p1 = PointDC(1.0, 2.0)
p2 = PointDC(1.0, 2.0)
print(p1)        # PointDC(x=1.0, y=2.0)
print(p1 == p2)  # True

Example 2: Defaults and field()

from dataclasses import dataclass, field

@dataclass
class Task:
    title: str
    priority: int = 1
    tags: list[str] = field(default_factory=list)
    metadata: dict[str, str] = field(default_factory=dict)

t1 = Task("Write report")
t2 = Task("Fix bug", priority=3, tags=["urgent"])
print(t1)  # Task(title='Write report', priority=1, tags=[], metadata={})
print(t2)  # Task(title='Fix bug', priority=3, tags=['urgent'], metadata={})

Example 3: Frozen dataclass as dict key

from dataclasses import dataclass

@dataclass(frozen=True)
class Tile:
    row: int
    col: int

visited: set[Tile] = set()
visited.add(Tile(0, 0))
visited.add(Tile(1, 2))
visited.add(Tile(0, 0))  # duplicate, ignored
print(len(visited))  # 2
print(Tile(0, 0) in visited)  # True

Example 4: Post-init validation

from dataclasses import dataclass

@dataclass
class Percentage:
    value: float

    def __post_init__(self):
        if not 0 <= self.value <= 100:
            raise ValueError(f"Percentage must be 0-100, got {self.value}")

p = Percentage(85.5)
print(p)  # Percentage(value=85.5)

try:
    bad = Percentage(150)
except ValueError as e:
    print(e)  # Percentage must be 0-100, got 150

Example 5: API response model with nested dataclasses

from dataclasses import dataclass, field, asdict

@dataclass
class Address:
    street: str
    city: str
    zip_code: str

@dataclass
class User:
    name: str
    email: str
    address: Address
    roles: list[str] = field(default_factory=list)

addr = Address("123 Main St", "Springfield", "62704")
user = User("Alice", "alice@example.com", addr, roles=["admin"])
print(user)
# User(name='Alice', email='alice@example.com',
#   address=Address(street='123 Main St', city='Springfield',
#   zip_code='62704'), roles=['admin'])
print(asdict(user))
# {'name': 'Alice', 'email': 'alice@example.com',
#  'address': {'street': '123 Main St', 'city': 'Springfield',
#   'zip_code': '62704'}, 'roles': ['admin']}

Common Mistakes

Don't
@dataclass
class Bad:
    items: list = []
# Shared mutable default across ALL instances!
Do
@dataclass
class Good:
    items: list = field(default_factory=list)
# Each instance gets its own list
Don't
@dataclass
class Wrong:
    name: str = "default"
    age: int          # Error! Non-default after default
Do
@dataclass
class Right:
    age: int          # Non-default fields first
    name: str = "default"

Key Rules

  • ALWAYS use field(default_factory=...) for mutable defaults (list, dict, set).
  • Use frozen=True when instances should be immutable (safer, hashable).
  • Dataclasses generate __eq__ by default — two instances with same field values are equal.

OOP

Classes — __init__, __repr__, __str__

OOP • New

Why It Matters

These three dunder methods define how objects are created and displayed. Every class you write should have at least __init__ and __repr__.

Progressive Explanation

__init__: The constructor — initializes instance attributes.

class Dog:
    def __init__(self, name: str, breed: str):
        self.name = name
        self.breed = breed

__repr__: The unambiguous, developer-facing representation. Used by repr(), the debugger, and the REPL.

def __repr__(self) -> str:
    return f"Dog(name={self.name!r}, breed={self.breed!r})"

__str__: The human-friendly display. Used by print(), str(), and f-strings.

def __str__(self) -> str:
    return f"{self.name} the {self.breed}"

Fallback rule: If only __repr__ is defined, it is used as the fallback for str() too.

Code Examples

Example 1: Basic class with __init__

class Person:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

p = Person("Alice", 30)
print(p.name, p.age)
# Output: Alice 30

print(p)
# Output: <__main__.Person object at 0x...>  (not helpful!)

Example 2: Adding __repr__

class Person:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self) -> str:
        return f"Person(name={self.name!r}, age={self.age!r})"

p = Person("Alice", 30)
print(repr(p))  # Person(name='Alice', age=30)
print(p)        # Person(name='Alice', age=30)  (fallback)

# Works in collections too:
people = [Person("Alice", 30), Person("Bob", 25)]
print(people)
# [Person(name='Alice', age=30), Person(name='Bob', age=25)]

Example 3: Adding __str__

class Person:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self) -> str:
        return f"Person(name={self.name!r}, age={self.age!r})"

    def __str__(self) -> str:
        return f"{self.name}, age {self.age}"

p = Person("Alice", 30)
print(str(p))     # Alice, age 30
print(repr(p))    # Person(name='Alice', age=30)
print(f"Hi {p}")  # Hi Alice, age 30

Example 4: Real-world — Money class

class Money:
    def __init__(self, amount: float, currency: str = "USD"):
        self.amount = round(amount, 2)
        self.currency = currency

    def __repr__(self) -> str:
        return f"Money(amount={self.amount}, currency={self.currency!r})"

    def __str__(self) -> str:
        symbols = {"USD": "$", "EUR": "€", "GBP": "£"}
        symbol = symbols.get(self.currency, self.currency + " ")
        return f"{symbol}{self.amount:,.2f}"

price = Money(1234.5, "USD")
print(repr(price))  # Money(amount=1234.5, currency='USD')
print(price)        # $1,234.50
print(f"Total: {price}")  # Total: $1,234.50

cost = Money(99.999, "EUR")
print(repr(cost))   # Money(amount=100.0, currency='EUR')
print(cost)         # €100.00

Common Mistakes

Don't
def __repr__(self):
    return "Point"
# Not enough info -- which point?
Do
def __repr__(self):
    return f"Point(x={self.x}, y={self.y})"
# Ideally eval-able back to an object
Don't
# Defining __str__ without __repr__
class Bad:
    def __str__(self):
        return "pretty"
    # No __repr__! Debuggers get nothing
Do
# Always define __repr__ first
class Good:
    def __repr__(self):
        return "Good(...)"  # dev-facing
    def __str__(self):
        return "pretty"     # user-facing

Key Rules

  • __repr__ should ideally return a string that could recreate the object: Point(x=3, y=4).
  • __str__ is for end users, __repr__ is for developers.
  • If you only define one, define __repr__ — it is the fallback for str() too.

Classes — __eq__ and __hash__

OOP • New

Why It Matters

Without __eq__, two objects with identical data are not "equal." Without __hash__, objects cannot be used in sets or as dict keys. These methods define object identity vs. equality.

Progressive Explanation

Default behavior: == checks identity (same object in memory), identical to is.

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

p1 = Point(1, 2)
p2 = Point(1, 2)
print(p1 == p2)  # False! (different objects)
print(p1 is p2)  # False

__eq__: Define what "equal" means for your objects.

def __eq__(self, other):
    if not isinstance(other, Point):
        return NotImplemented
    return self.x == other.x and self.y == other.y

__hash__: Required for sets/dict keys. Must be consistent with __eq__.

def __hash__(self):
    return hash((self.x, self.y))

The golden rule: If a == b, then hash(a) == hash(b) must be true (but not vice versa).

Critical side effect: Defining __eq__ without __hash__ makes instances unhashable (Python sets __hash__ to None).

Code Examples

Example 1: Default behavior — same data, not equal

class Coord:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y
    def __repr__(self):
        return f"Coord({self.x}, {self.y})"

a = Coord(3, 4)
b = Coord(3, 4)
print(a == b)    # False -- different objects in memory
print(a is b)    # False
print(id(a) == id(b))  # False

Example 2: Adding __eq__ — value-based equality

class Coord:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y

    def __repr__(self):
        return f"Coord({self.x}, {self.y})"

    def __eq__(self, other):
        if not isinstance(other, Coord):
            return NotImplemented
        return self.x == other.x and self.y == other.y

a = Coord(3, 4)
b = Coord(3, 4)
c = Coord(5, 6)
print(a == b)  # True  (same values)
print(a == c)  # False (different values)
print(a == (3, 4))  # NotImplemented -> False

Example 3: The unhashable problem

class Coord:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y
    def __eq__(self, other):
        if not isinstance(other, Coord):
            return NotImplemented
        return self.x == other.x and self.y == other.y

a = Coord(3, 4)
try:
    s = {a}  # TypeError!
except TypeError as e:
    print(e)
    # Output: unhashable type: 'Coord'

# Defining __eq__ makes Python set __hash__ = None
print(Coord.__hash__)  # None

Example 4: Implementing __hash__ correctly

class Coord:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y

    def __repr__(self):
        return f"Coord({self.x}, {self.y})"

    def __eq__(self, other):
        if not isinstance(other, Coord):
            return NotImplemented
        return self.x == other.x and self.y == other.y

    def __hash__(self):
        return hash((self.x, self.y))

# Now we can use Coords in sets and as dict keys
points = {Coord(0, 0), Coord(1, 1), Coord(0, 0)}
print(points)      # {Coord(0, 0), Coord(1, 1)}
print(len(points)) # 2 (duplicate removed)

distances = {Coord(0, 0): 0.0, Coord(3, 4): 5.0}
print(distances[Coord(3, 4)])  # 5.0

Common Mistakes

Don't
# Mutable + hashable = disaster
class Bad:
    def __init__(self, items):
        self.items = items  # mutable list!
    def __eq__(self, other):
        return self.items == other.items
    def __hash__(self):
        return hash(tuple(self.items))

b = Bad([1, 2])
s = {b}
b.items.append(3)  # hash changed!
print(b in s)  # False -- can't find it!
Do
# Only hash immutable objects
class Good:
    def __init__(self, x: int, y: int):
        self.x = x  # immutable fields
        self.y = y
    def __eq__(self, other):
        if not isinstance(other, Good):
            return NotImplemented
        return self.x == other.x and self.y == other.y
    def __hash__(self):
        return hash((self.x, self.y))
Don't
# __hash__ doesn't match __eq__
def __eq__(self, other):
    return self.x == other.x  # compares x only
def __hash__(self):
    return hash((self.x, self.y))  # uses x AND y
# Two objects can be == but have different hashes!
Do
# Use the SAME fields in both
def __eq__(self, other):
    return (self.x, self.y) == (other.x, other.y)
def __hash__(self):
    return hash((self.x, self.y))

Key Rules

  • If you define __eq__, Python sets __hash__ to None (makes objects unhashable).
  • Only add __hash__ if your objects are immutable (or you will not mutate hashed fields).
  • @dataclass(frozen=True) handles both __eq__ and __hash__ correctly.
