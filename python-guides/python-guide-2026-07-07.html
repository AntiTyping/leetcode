Review Section (6 items)

1. Enum due today rep 3 → 4

An Enum is a set of named, singleton constants. Each member has a .name and a .value; look one up by value with Color(2) and by name with Color["GREEN"]. Members are iterable, hashable, and compare by identity — ideal for a fixed vocabulary of states. 15 days since last review.

Key Pattern
from enum import Enum

class Color(Enum):
    RED = 1
    GREEN = 2
    BLUE = 3

print(Color.RED)          # Color.RED
print(Color.RED.name)     # 'RED'
print(Color.RED.value)    # 1
print(Color(2))           # Color.GREEN -- lookup by value
print(Color["BLUE"])      # Color.BLUE  -- lookup by name
print([c.name for c in Color])   # ['RED', 'GREEN', 'BLUE']
Challenge

Define a Status enum, iterate its members, and look up one member by value and another by name. Add a plain method to the enum body that returns something derived from self.value. Then create a second member with a duplicate value and explain what an alias is and how @unique forbids it.

Pitfall

Two members with the same value do NOT both exist — the second becomes an alias of the first (skipped by iteration); decorate with @enum.unique to make duplicates a ValueError. A plain Enum member is NOT equal to its raw value (Color.RED == 1 is False) — use IntEnum if you need that.

2. Async Generators & async for 15 days overdue rep 1 → 2

An async generator is an async def that yields. It can await between yields, so it streams values from an asynchronous source (a socket, a paginated API). Consume it with async for or an async comprehension — both only legal inside a coroutine. 16 days since last review.

Key Pattern
import asyncio

async def ticker(n):
    for i in range(n):
        await asyncio.sleep(0)     # await between yields
        yield i                    # async def + yield = async generator

async def main():
    out = [x async for x in ticker(3)]   # async comprehension
    print(out)                            # [0, 1, 2]

asyncio.run(main())
Challenge

Write an async generator that yields three values, awaiting asyncio.sleep(0) before each. Consume it once with a plain async for loop and once with an [... async for ...] comprehension. Then explain why the same iteration in a normal (non-async) function raises TypeError, and what aclose() is for.

Pitfall

You cannot drive an async generator with a plain for — it needs async for inside a coroutine. An async generator may NOT return a value (only bare return to stop). If you abandon one mid-stream, call await gen.aclose() (or let async for finish) so its finally/cleanup runs.

3. multiprocessing Patterns 15 days overdue rep 1 → 2

multiprocessing sidesteps the GIL by running work in separate processes. A Pool fans a function across workers with map/imap_unordered; a Queue or Pipe ships results back. Everything crossing the process boundary is pickled, so targets and arguments must be picklable. 16 days since last review.

Key Pattern
from multiprocessing import Pool

def square(x):
    return x * x

if __name__ == "__main__":      # required guard on spawn (macOS/Windows)
    with Pool(4) as pool:
        print(pool.map(square, [1, 2, 3, 4]))   # [1, 4, 9, 16]
Challenge

Use a Pool to map a CPU-bound function over a list, then switch to imap_unordered and observe results arriving out of order. Guard the entry point with if __name__ == "__main__". Explain why a lambda or a locally-defined closure fails as the target under the spawn start method.

Pitfall

On the spawn start method (the macOS/Windows default) the child re-imports your module, so the worker target must be a top-level, importable function — lambdas and closures are unpicklable. Always guard the launch behind if __name__ == "__main__", or spawning recurses and forks a bomb of processes.

4. Packaging & Distribution 15 days overdue rep 1 → 2

A modern package is described by a single pyproject.toml: a [build-system] table naming the backend and a [project] table with name, version, and dependencies. python -m build produces an sdist (source) and a wheel (built); twine upload publishes them. 16 days since last review.

Key Pattern
# pyproject.toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "mypkg"
version = "0.1.0"
dependencies = ["requests>=2"]

[project.scripts]
mycli = "mypkg.cli:main"          # console entry point

# build both artifacts, then publish:
#   python -m build        -> dist/mypkg-0.1.0.tar.gz  (sdist)
#                             dist/mypkg-0.1.0-*.whl   (wheel)
#   twine upload dist/*
Challenge

Write a minimal pyproject.toml with a [build-system] and a [project] table (name, version, one dependency) and a console entry point under [project.scripts]. Describe the difference between the sdist and the wheel that python -m build emits, and which one pip install prefers.

Pitfall

Console commands go under [project.scripts] as name = "module:function" — not a shell script you write by hand. Publish BOTH artifacts: the wheel installs without a build step, the sdist is the buildable fallback. Bumping the version is mandatory — PyPI rejects re-uploading an existing version.

5. Debugging: pdb & breakpoint() 15 days overdue rep 1 → 2

Drop breakpoint() anywhere to enter the interactive debugger at that line (it honors the PYTHONBREAKPOINT env var). Core commands: next (step over), step (into), continue, p expr, list, where, b <line> (breakpoint), quit. 16 days since last review.

Key Pattern
def accumulate(xs):
    total = 0
    for x in xs:
        breakpoint()        # enter pdb here (PYTHONBREAKPOINT=0 disables all)
        total += x
    return total

# at the (Pdb) prompt:
#   p total     -- print an expression
#   n / s       -- step over / step into
#   b 4         -- set a breakpoint at line 4
#   c           -- continue to the next breakpoint
Challenge

Add a breakpoint() inside a loop, then describe the commands you would use to step over one line, step into a called function, print a local, set a conditional breakpoint (b 12, x > 5), and continue. How do you disable every breakpoint() call in a run without editing the source?

Pitfall

Set PYTHONBREAKPOINT=0 to neutralize all breakpoint() calls (e.g. in production) — no code edits needed. At the prompt, a bare variable named like a command (n, c, l) runs the COMMAND, not a print — use p n to inspect it. next steps over calls; step descends into them.

6. subprocess Module 14 days overdue rep 1 → 2

subprocess.run launches an external command and waits: pass a list of args, add capture_output=True and text=True to get str stdout/stderr, and check=True to raise on a nonzero exit. It returns a CompletedProcess with .stdout, .stderr, and .returncode. 15 days since last review.

Key Pattern
import subprocess

r = subprocess.run(
    ["echo", "hi"],            # a LIST of args -- no shell
    capture_output=True,
    text=True,                    # decode stdout/stderr to str
    check=True,                    # raise CalledProcessError on nonzero exit
)
print(r.stdout.strip())        # hi
print(r.returncode)            # 0
Challenge

Run a command capturing stdout and stderr as text with check=True, and wrap it in a try/except subprocess.CalledProcessError that reads e.returncode and e.stderr. Then feed a string to the child's stdin via input=. Explain why shell=True with interpolated user input is a command-injection hazard.

Pitfall

Never pass untrusted input to shell=True — it invites command injection; pass an argument list instead (no shell parsing). Without capture_output=True (or stdout=PIPE) .stdout is None; without text=True you get bytes. check=True raises only on nonzero exit — a program that fails silently still returns 0.

New Concepts — System Limits & Localization (2 concepts)

Two modules that connect your program to its operating environment. resource (Unix) reads and caps a process's CPU, memory, and file-descriptor limits, and reports how much it has consumed. locale makes number, currency, and sort output follow the user's regional conventions instead of hard-coded formats.

System & Processes

Concept 1: resource — Process Resource Usage & Limits

Why learn this? When you need to know how much CPU or memory your process has burned, or to cap a resource so a runaway job can't take the box down, resource is the Unix interface. getrusage reports consumption; getrlimit/setrlimit read and tighten soft/hard limits on open files, CPU time, address space, and more.
Example 1 — how much have I used? (getrusage)
import resource

u = resource.getrusage(resource.RUSAGE_SELF)
print(u.ru_utime >= 0)     # user-mode CPU seconds
print(u.ru_stime >= 0)     # system-mode CPU seconds
print(u.ru_maxrss > 0)     # peak resident set size (bytes on macOS, KiB on Linux)
Output
True True True
Example 2 — read and tighten a limit (getrlimit / setrlimit)
import resource

soft, hard = resource.getrlimit(resource.RLIMIT_NOFILE)   # max open files
print(isinstance(soft, int) and isinstance(hard, int))   # True

resource.setrlimit(resource.RLIMIT_NOFILE, (256, hard))   # lower the SOFT limit
print(resource.getrlimit(resource.RLIMIT_NOFILE)[0])       # 256
Output
True 256
Example 3 — you cannot raise soft above hard (as non-root)
import resource

resource.setrlimit(resource.RLIMIT_NOFILE, (64, 64))   # lowering both is allowed
try:
    resource.setrlimit(resource.RLIMIT_NOFILE, (128, 64))   # soft > hard is rejected
except ValueError:
    print("soft may not exceed hard")     # unprivileged: can lower, not raise past hard
Output
soft may not exceed hard

Common Mistakes

✗ Assuming ru_maxrss is the same unit everywhere — it's bytes on macOS but kilobytes on Linux

✓ Normalize with a platform check (or a library) before comparing memory figures across OSes


✗ Trying to setrlimit the soft value above the hard cap as a normal user — raises ValueError

✓ Unprivileged processes may only lower a limit (or raise soft up to hard); raising the hard cap needs root

Key Rules

  • getrusage(RUSAGE_SELF) returns a struct with ru_utime, ru_stime, and ru_maxrss (peak memory); pass RUSAGE_CHILDREN for spawned children
  • getrlimit(res) returns (soft, hard); setrlimit(res, (soft, hard)) changes them — resource.RLIM_INFINITY (-1) means unlimited
  • This module is Unix-only (no Windows); common limits are RLIMIT_NOFILE, RLIMIT_CPU, RLIMIT_AS (address space), RLIMIT_DATA

Exercises

  1. Call getrusage(RUSAGE_SELF) and print the user CPU time and peak RSS; run a busy loop, sample again, and show the CPU time went up.
  2. Read the RLIMIT_NOFILE soft/hard pair, lower the soft limit to a smaller value, and confirm the change with a second getrlimit.
  3. Attempt to set the soft limit one above the hard limit and catch the ValueError; print a message explaining why an unprivileged process cannot do this.
Connections: resource complements the memory work in __slots__/sys.getsizeof and the gc module (it measures the OS-level footprint); its RLIMIT_CPU/RLIMIT_AS caps are the enforcement layer under sandboxing that signal alarms and multiprocessing workers rely on.
Localization

Concept 2: locale — Region-Aware Formatting & Sorting

Why learn this? "1,000,000.50" in the US is "1.000.000,50" in Germany, and "z" sorts before "ä" only in some regions. The locale module formats numbers and currency and collates strings by the user's regional conventions, so your output is correct everywhere — without you hard-coding separators.
Example 1 — the C locale baseline & localeconv
import locale

locale.setlocale(locale.LC_ALL, "C")     # portable, culture-neutral baseline
conv = locale.localeconv()                # dict of the active conventions
print(conv["decimal_point"])              # '.'
print(locale.format_string("%d", 1000000, grouping=True))   # '1000000' -- C has no grouping
Output
. 1000000
Example 2 — a real locale groups digits & formats currency
import locale

locale.setlocale(locale.LC_ALL, "en_US.UTF-8")   # must be installed on the system
print(locale.format_string("%d", 1000000, grouping=True))   # '1,000,000'
print(locale.currency(1234.5, grouping=True))            # '$1,234.50'
Output
1,000,000 $1,234.50
Example 3 — parse numbers the locale's way (atoi / atof)
import locale

locale.setlocale(locale.LC_ALL, "C")
print(locale.atoi("1000"))     # 1000 -- int per active locale
print(locale.atof("3.14"))     # 3.14 -- float per active locale
# locale.strxfrm(s) transforms a string into a locale-collation sort key:
words = ["banana", "Apple", "cherry"]
print(sorted(words, key=locale.strxfrm))   # collation-aware order
Output
1000 3.14 ['Apple', 'banana', 'cherry']

Common Mistakes

✗ setlocale(LC_ALL, "de_DE.UTF-8") crashing with locale.Error — that locale isn't installed on the machine

✓ Wrap in try/except locale.Error and fall back to "" (the user's env default) or "C"; generate locales in deployment


✗ Calling setlocale from worker threads mid-run — it mutates global process state and isn't thread-safe

✓ Set the locale once at startup; for isolated formatting prefer the babel library or explicit format specs

Key Rules

  • setlocale(LC_ALL, "") adopts the user's environment locale; "C" is the neutral default; category constants (LC_NUMERIC, LC_MONETARY, LC_COLLATE) scope one facet
  • format_string("%d", n, grouping=True) and currency(x, grouping=True) render numbers/money per the active locale; localeconv() exposes the raw conventions
  • Sort human text with key=locale.strxfrm (or compare with strcoll) — plain sorted uses code-point order, which misorders accented and non-ASCII letters

Exercises

  1. Set the "C" locale and print localeconv()["decimal_point"] and localeconv()["thousands_sep"]; note grouping is empty.
  2. Switch to "en_US.UTF-8" and format 1234567 with grouping=True and 1234.5 with currency; guard the setlocale call against locale.Error.
  3. Sort a list of mixed-case words with the default sorted and again with key=locale.strxfrm, and explain when the two orders diverge.
Connections: locale is the culture-aware layer above the format spec mini-language and str.format's ,/_ grouping (which are locale-independent); its strxfrm collation keys pair with sorted(key=...); and its UTF-8 codesets tie back to the codecs/Unicode work.

New Concepts — Compression & Package Data (2 concepts)

Two low-level building blocks you reach for under higher-level tools. zlib is the raw DEFLATE codec and checksum library behind gzip, PNG, and the zip format — one-shot or streaming compression plus crc32/adler32. importlib.resources is the correct, zip-safe way to read data files bundled inside an installed package, replacing fragile __file__ path math.

Compression

Concept 3: zlib — Raw DEFLATE & Checksums

Why learn this? zlib is the compression engine underneath gzip, the ZIP format, and PNG. Use it directly for one-shot compress/decompress of in-memory bytes, for streaming large data through a compressobj, or for fast integrity checksums (crc32, adler32) — no framing/header overhead of a full file format.
Example 1 — one-shot compress / decompress
import zlib

data = b"na" * 100                # 200 highly repetitive bytes
comp = zlib.compress(data, level=9)   # level 0..9 (9 = smallest, slowest)
print(len(comp) < len(data))       # True -- big shrink on repetitive data
print(zlib.decompress(comp) == data)   # True -- lossless round trip
Output
True True
Example 2 — checksums (and they update incrementally)
import zlib

print(zlib.crc32(b"hello"))       # 907060870
print(zlib.adler32(b"hello"))     # 103547413

part = zlib.crc32(b"hel")             # feed the running value back in:
print(zlib.crc32(b"lo", part) == zlib.crc32(b"hello"))   # True
Output
907060870 103547413 True
Example 3 — streaming with compressobj / flush
import zlib

co = zlib.compressobj()
chunks = [co.compress(b"stream "), co.compress(b"of data")]
chunks.append(co.flush())            # emit any buffered trailing output
comp = b"".join(chunks)
print(zlib.decompress(comp))       # b'stream of data'
Output
b'stream of data'

Common Mistakes

✗ Forgetting co.flush() on a compressobj — the last bytes stay buffered and the stream decompresses incomplete (or errors)

✓ Always append co.flush() to finalize; likewise drain a decompressobj and check .unused_data


✗ Treating crc32 as a security hash — it detects accidental corruption, not tampering, and collides trivially

✓ Use crc32/adler32 for integrity checks only; reach for hashlib/hmac when you need cryptographic strength

Key Rules

  • zlib.compress(data, level=-1..9) / zlib.decompress(data) handle bytes in one shot; higher level = smaller output, more CPU
  • zlib.crc32(data[, value]) and adler32 return an unsigned 32-bit checksum; pass the previous result as value to continue over chunks
  • For large/streamed data use compressobj()/decompressobj(): call .compress() per chunk, then .flush() once to finalize

Exercises

  1. Compress a repetitive bytes value at level 1 and level 9; compare the two output lengths and confirm both decompress back to the original.
  2. Compute crc32 of a message in one call, then in two calls by threading the running value through the second call, and assert they match.
  3. Use a compressobj to compress three chunks, remember to flush(), join the pieces, and decompress back to the original bytes.
Connections: zlib is the codec under gzip/bz2/lzma (which add file framing) and the zipfile format you studied; its bytes↔bytes role also appears as a codecs registry entry; and its checksums contrast with the cryptographic digests in hashlib/hmac.
Packaging

Concept 4: importlib.resources — Reading Bundled Package Data

Why learn this? A package that ships a template, a JSON schema, or a data table needs to read those files at runtime. Building paths from __file__ breaks when the package is installed as a zip or a wheel. importlib.resources.files(pkg) returns a Traversable that reads bundled resources correctly, whatever the install layout.
Example 1 — locate & read a resource with files()
from importlib.resources import files

pkg = files("email")                    # a Traversable for the 'email' package dir
init = pkg / "__init__.py"            # join with / like pathlib
print(init.is_file())               # True
print("email" in init.read_text().lower())   # True -- read the file's text
Output
True True
Example 2 — as_file() for APIs that need a real path
from importlib.resources import files, as_file

res = files("email") / "__init__.py"
with as_file(res) as path:          # guarantees a real filesystem Path
    print(path.exists())            # True
    print(path.name)                # __init__.py
Output
True __init__.py
Example 3 — iterate the resources in a package
from importlib.resources import files

pkg = files("json")                     # the json stdlib package
names = sorted(p.name for p in pkg.iterdir()
               if p.name.endswith(".py"))
print("decoder.py" in names)      # True
print("encoder.py" in names)      # True
Output
True True

Common Mistakes

✗ Building resource paths with os.path.join(os.path.dirname(__file__), "data.txt") — breaks when the package is zipped or frozen

✓ Use files(package) / "data.txt", which resolves correctly for any loader (filesystem, zip, wheel)


✗ Handing a Traversable straight to a C library or subprocess that demands a real path string

✓ Wrap it in with as_file(res) as path: — it materializes a temp file if the resource lives inside an archive

Key Rules

  • files(package) returns a Traversable; navigate with the / operator and read via .read_text()/.read_bytes() — the modern API (Python 3.9+)
  • Use as_file(resource) as a context manager whenever an API needs a genuine filesystem Path; it extracts zipped resources to a temp file and cleans up on exit
  • Prefer this over __file__ arithmetic and the deprecated read_text(pkg, name)/path() functions; it is the zip-safe, loader-agnostic way to reach package data

Exercises

  1. Use files("json") to get the package directory, join to "decoder.py", and confirm is_file() is true and its text contains the word "JSON".
  2. Wrap that same resource in as_file(...) and print the real path's .name and that .exists() is true inside the with block.
  3. Iterate a package's directory with iterdir() and collect the names of every .py resource into a sorted list.
Connections: importlib.resources is the runtime companion to the packaging/pyproject.toml work (it reads the data files you bundle); its Traversable mirrors pathlib.Path's / and read helpers; and it sits beside the broader importlib import-system machinery you studied.

New Concepts — Interactive CLIs (1 concept)

One small framework for building line-oriented, interactive command interpreters — think of a debugger prompt, a game console, or an admin shell. cmd.Cmd turns do_* methods into commands, autogenerates help from their docstrings, and runs the read-eval-print loop for you.

Interactive Tools

Concept 5: cmd — Line-Oriented Command Interpreters

Why learn this? When you want an interactive REPL — an admin console, a small game shell, a debugger-style prompt — cmd.Cmd gives you the whole loop for free. Each do_name method becomes a command, its docstring becomes the help text, and cmdloop() handles reading, parsing the first word, and dispatching. (Python's own pdb is built on it.)
Example 1 — commands are do_* methods (driven via onecmd)
import cmd

class Calc(cmd.Cmd):
    prompt = "(calc) "
    def do_add(self, arg):
        "add N N -- print the sum"
        a, b = map(int, arg.split())
        print(a + b)
    def do_quit(self, arg):
        "quit -- exit the loop"
        return True                 # a truthy return stops cmdloop()

c = Calc()
c.onecmd("add 2 3")                # dispatch one line without the interactive loop
print(c.onecmd("quit"))           # True
Output
5 True
Example 2 — redirect output; help comes from docstrings
import cmd, io

class Shell(cmd.Cmd):
    def do_greet(self, arg):
        "greet NAME -- say hello"
        print(f"hello {arg}")

s = Shell(stdout=io.StringIO())     # capture output instead of the terminal
s.onecmd("greet Ada")
print(s.stdout.getvalue().strip())   # hello Ada
Output
hello Ada
Example 3 — hooks: default() & unknown commands
import cmd

class Shell(cmd.Cmd):
    def do_ping(self, arg):
        print("pong")
    def default(self, line):        # called when no do_* matches
        print(f"unknown: {line}")

s = Shell()
s.onecmd("ping")                   # pong
s.onecmd("xyzzy")                  # unknown: xyzzy
Output
pong unknown: xyzzy

Common Mistakes

✗ Expecting a command to end the session by calling exit() — that kills the whole program, not just the loop

✓ return True from a do_* method (e.g. do_quit) — a truthy return is the signal that stops cmdloop()


✗ Naming a command handler add() and wondering why it isn't recognized

✓ Handlers MUST be prefixed do_ (the command is the suffix); help text is taken from the method's docstring or a help_name method

Key Rules

  • Subclass cmd.Cmd; each do_name(self, arg) defines command name (everything after the first word arrives as the arg string)
  • cmdloop() runs the interactive REPL; onecmd("line") dispatches a single command (handy for tests) — a truthy return from a handler stops the loop
  • Customize with prompt, intro, default(line) (unknown commands), emptyline() (blank input), and help_*/docstrings for the built-in help

Exercises

  1. Build a cmd.Cmd subclass with a do_echo that prints its argument and a do_quit that returns True; drive it with onecmd calls and confirm do_quit returns truthy.
  2. Give a command a one-line docstring, construct the shell with a StringIO stdout, run onecmd("help yourcommand"), and show the docstring appears in the captured output.
  3. Override default to report unknown commands and emptyline to do nothing, then dispatch a known command, an unknown one, and an empty line.
Connections: cmd is the framework behind pdb's prompt you reviewed today; it pairs with shlex.split when a command needs quoted arguments, with argparse for parsing each command's options, and with readline for history and tab-completion.
