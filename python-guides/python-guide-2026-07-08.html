Review Section (6 items)

1. Type Hints — Optional, Union & Modern Syntax due today rep 3 → 4

Optional[X] is exactly Union[X, None] — "an X or nothing". Since Python 3.10 the X | Y operator writes unions inline (int | str, str | None), and it also works as the second argument to isinstance. These annotations are documentation for humans and type checkers; they are not enforced at runtime. 15 days since last review.

Key Pattern
from typing import Optional, Union

def find(key: str) -> Optional[int]:      # int or None
    return {"a": 1}.get(key)

x: int | str = 3                       # 3.10+ union syntax
y: str | None = None                    # the modern Optional[str]
print(Optional[int] == Union[int, None])   # True
print(isinstance(3, int | str))          # True
Challenge

Annotate a function that returns either an int or None two ways — with Optional[int] and with int | None — and show they mean the same thing. Then explain why a wrong-typed value still runs without error at runtime, and what from __future__ import annotations changes about when annotations are evaluated.

Pitfall

Optional[int] does NOT mean "optional argument" — it means "int or None", nothing about defaults. A parameter with an Optional type still has no default unless you give it one. Annotations are not checked at runtime; passing a str where int is annotated fails only under a type checker, never on its own.

2. Dataclasses due today rep 3 → 4

@dataclass generates __init__, __repr__, and __eq__ from class-level annotated fields, so you describe the data and skip the boilerplate. Use field(default_factory=list) for mutable defaults, frozen=True for immutability (and hashability), and order=True to add comparison operators. 15 days since last review.

Key Pattern
from dataclasses import dataclass, field

@dataclass
class Point:
    x: int
    y: int = 0
    tags: list = field(default_factory=list)   # never default=[]

p = Point(1, 2)
print(p)                       # Point(x=1, y=2, tags=[])
print(p == Point(1, 2))       # True -- generated __eq__
Challenge

Write a dataclass with one required field, one field defaulting to 0, and one list field using field(default_factory=list). Print an instance to see the generated repr, and compare two equal instances. Then make a second dataclass frozen=True and show it can be used as a dict key while the non-frozen one cannot.

Pitfall

A mutable default (tags: list = []) is a ValueError in a dataclass — use field(default_factory=list). Fields without a default may not follow fields with one (same rule as function args). A plain dataclass is unhashable (because it is mutable and defines __eq__); set frozen=True to make it immutable and hashable.

3. Classes — __eq__ and __hash__ due today rep 3 → 4

Define __eq__ to compare objects by value instead of identity. But defining __eq__ sets __hash__ to None, making instances unhashable — you must add a matching __hash__ if you want to use them in sets or as dict keys. The rule: equal objects MUST have equal hashes. 15 days since last review.

Key Pattern
class Money:
    def __init__(self, cents):
        self.cents = cents
    def __eq__(self, other):
        if not isinstance(other, Money):
            return NotImplemented
        return self.cents == other.cents
    def __hash__(self):
        return hash(self.cents)   # must agree with __eq__

print(Money(100) == Money(100))          # True
print(len({Money(100), Money(100)}))    # 1 -- dedup via hash + eq
Challenge

Give a class a value-based __eq__ (returning NotImplemented for foreign types) and a matching __hash__ built from the same fields. Put two "equal" instances in a set and show it collapses to one element. Then remove __hash__ and show that adding an instance to a set now raises TypeError: unhashable type.

Pitfall

Defining __eq__ silently sets __hash__ = None — instances become unhashable until you define __hash__ too. The invariant is one-way: equal objects must hash equal, but equal hashes do NOT imply equality (collisions are fine). Never build a hash from a mutable field you also compare on — mutating it corrupts the object's slot in a set/dict.

4. Properties & @property due today rep 3 → 4

@property turns a method into a read-only computed attribute accessed without parentheses. Add a matching @name.setter to make it writable — the ideal place to validate on assignment. Properties let you start with a plain attribute and later intercept access without changing any caller. 15 days since last review.

Key Pattern
class Celsius:
    def __init__(self, temp):
        self._temp = temp
    @property
    def temp(self):
        return self._temp
    @temp.setter
    def temp(self, value):
        if value < -273.15:
            raise ValueError("below absolute zero")
        self._temp = value

c = Celsius(20)
print(c.temp)          # 20 -- no parentheses
c.temp = 25            # goes through the setter (validates)
print(c.temp)          # 25
Challenge

Write a class storing a value in self._x, expose it via a @property getter, and add a @x.setter that rejects negatives with ValueError. Read and write it with plain attribute syntax (no parens). Then explain why the backing field is named _x and what happens if you name the property and the field the same.

Pitfall

Storing the value under the SAME name as the property (self.temp = ... inside the setter) causes infinite recursion — use a distinct backing field like self._temp. A property with only a getter is read-only; assigning to it raises AttributeError until you add a .setter. The @x.setter method must reuse the property's name.

5. __slots__ due today rep 3 → 4

__slots__ replaces each instance's __dict__ with a fixed, array-like layout of named fields. It cuts per-instance memory substantially and speeds attribute access, at the cost of forbidding new attributes not listed in the slots. Ideal for small classes created in the millions. 15 days since last review.

Key Pattern
class Point:
    __slots__ = ("x", "y")      # no per-instance __dict__
    def __init__(self, x, y):
        self.x, self.y = x, y

p = Point(1, 2)
print(p.x, p.y)              # 1 2
print(hasattr(p, "__dict__"))   # False -- memory saved
try:
    p.z = 3                  # not a declared slot
except AttributeError:
    print("no new attributes")   # AttributeError
Challenge

Define a class with __slots__ = ("x", "y"), create an instance, and show hasattr(inst, "__dict__") is False. Try to set an undeclared attribute and catch the AttributeError. Then explain what happens to slots when a subclass does NOT define its own __slots__, and why __slots__ and a class-level default of the same name conflict.

Pitfall

If a subclass omits __slots__, it regains a __dict__ and the memory win is lost. You cannot give a slotted attribute a class-level default (the slot descriptor and the class attribute collide with ValueError). Multiple inheritance with non-empty slots on more than one base raises a layout conflict; keep slots to single, focused hierarchies.

6. Dunder Methods — __getitem__, __len__, __contains__ due today rep 3 → 4

Implementing the container protocol makes your object behave like a built-in sequence: __len__ powers len(), __getitem__ powers obj[i] (and, on its own, iteration and in), and __contains__ gives a fast, custom in test. Duck typing means anything with these methods is a container. 15 days since last review.

Key Pattern
class Deck:
    def __init__(self, cards):
        self._cards = list(cards)
    def __len__(self):
        return len(self._cards)
    def __getitem__(self, i):
        return self._cards[i]        # enables d[i], slicing, iteration, in
    def __contains__(self, x):
        return x in self._cards       # fast membership

d = Deck(["A", "K", "Q"])
print(len(d), d[0], "K" in d)   # 3 A True
Challenge

Build a class wrapping a list with __len__, __getitem__, and __contains__. Show that len(), indexing, and in all work. Then delete __contains__ and __iter__ and demonstrate that in and a for loop STILL work through __getitem__ alone (the old sequence-iteration fallback). Explain that fallback.

Pitfall

With only __getitem__, iteration works via the legacy integer-index protocol — but your __getitem__ must raise IndexError at the end or the for loop never stops. Slicing passes a slice object (not an int) to __getitem__; handle it if you want d[1:3]. A custom __contains__ overrides the iteration-based membership fallback.

New Concepts — Memory & Crash Diagnostics (2 concepts)

Two diagnostic modules for when something goes wrong deep in a running process. tracemalloc attributes every Python allocation to the source line that made it, so you can find what is eating memory and diff two moments to localize a leak. faulthandler prints a real traceback when the interpreter crashes at the C level (a segfault) or hangs — the situations ordinary try/except can never reach.

Diagnostics & Memory

Concept 1: tracemalloc — Tracing Memory Allocations

Why learn this? When a process quietly grows until it is killed, you need to know which line allocated the memory — not just the total. tracemalloc records a traceback for every Python allocation, reports current and peak usage, and, crucially, lets you diff two snapshots to pinpoint exactly where a leak accumulates between two points in time.
Example 1 — current vs. peak usage (get_traced_memory)
import tracemalloc

tracemalloc.start()                       # begin recording allocations
data = [b"x" * 1000 for _ in range(1000)]   # ~1 MB
current, peak = tracemalloc.get_traced_memory()
print(current > 0)      # True -- bytes currently tracked
print(peak >= current)   # True -- high-water mark
tracemalloc.stop()
Output
True True
Example 2 — top allocations by line (snapshot.statistics)
import tracemalloc

tracemalloc.start()
rows = [dict(i=i) for i in range(1000)]
snap = tracemalloc.take_snapshot()          # capture per-allocation tracebacks
top = snap.statistics("lineno")             # aggregate by source line, biggest first
print(len(top) > 0)                    # True
stat = top[0]
print(stat.size > 0, stat.count > 0)   # True True -- bytes & object count
tracemalloc.stop()
Output
True True True
Example 3 — diff two snapshots to find a leak (compare_to)
import tracemalloc

tracemalloc.start()
snap1 = tracemalloc.take_snapshot()
leak = [object() for _ in range(10000)]   # grow the heap
snap2 = tracemalloc.take_snapshot()
diff = snap2.compare_to(snap1, "lineno")     # sorted by size delta
print(diff[0].size_diff > 0)          # True -- net growth at the top line
tracemalloc.stop()
Output
True

Common Mistakes

✗ Calling get_traced_memory() or take_snapshot() without tracemalloc.start() first — it traces nothing and returns (0, 0)

✓ Call tracemalloc.start() (optionally start(nframe) for deeper tracebacks) before the code you want to measure


✗ Leaving tracing enabled in production — recording a traceback per allocation adds real CPU and memory overhead

✓ Scope it to the region under investigation and stop() (or clear_traces()) when done

Key Rules

  • tracemalloc.start(nframe=1) begins recording; get_traced_memory() returns (current, peak) bytes; reset_peak() zeroes the peak
  • take_snapshot() captures allocations with tracebacks; snapshot.statistics("lineno") (or "traceback") aggregates them, largest first
  • snapshot.compare_to(old, key) diffs two snapshots (size_diff/count_diff) — the standard way to localize a leak between two moments

Exercises

  1. Start tracing, build a large list of bytes, and print that get_traced_memory()'s peak is >= its current value.
  2. Take a snapshot after constructing a data structure, call statistics("lineno"), and print the .size and .count of the top entry.
  3. Take two snapshots around an allocation, compare_to them, and show the top size_diff is positive.
Connections: tracemalloc measures the Python-level allocations that gc, __slots__, and sys.getsizeof reason about — but unlike resource.ru_maxrss (the OS-level peak) it attributes bytes to the exact source line, making it the precise tool for the memory-optimization work.
Diagnostics

Concept 2: faulthandler — Dumping Tracebacks on Crash

Why learn this? A segfault in a C extension, a stack overflow, or a mysterious hang leaves no Python traceback — the process just dies or freezes. faulthandler installs handlers that dump the Python stack of every thread when a fatal signal fires, and it offers a watchdog timer and a signal hook so you can pull a traceback out of a stuck production process on demand.
Example 1 — enable & dump current tracebacks
import faulthandler, tempfile

faulthandler.enable()                 # dump a traceback on fatal errors (SIGSEGV, ...)
print(faulthandler.is_enabled())      # True

with tempfile.TemporaryFile("w+") as f:
    faulthandler.dump_traceback(file=f)   # write all thread stacks now
    f.seek(0)
    print("File" in f.read())         # True -- traceback text captured
Output
True True
Example 2 — watchdog: dump if code hangs (dump_traceback_later)
import faulthandler

faulthandler.dump_traceback_later(5)          # still running in 5s? dump the stack
# ... work that should finish quickly ...
faulthandler.cancel_dump_traceback_later()     # made it in time -- disarm the watchdog
print("finished before timeout")
Output
finished before timeout
Example 3 — dump on a user signal (Unix)
import faulthandler, signal

faulthandler.register(signal.SIGUSR1)   # kill -USR1 <pid> now prints a traceback
print(True)
# from another shell:  kill -USR1 <pid>
#   -> stack of every thread is written to stderr
Output
True

Common Mistakes

✗ Reaching for faulthandler to catch ordinary exceptions — it only fires on fatal, uncatchable errors (segfaults, C-level aborts)

✓ Catch normal errors with try/except; use faulthandler when the interpreter itself crashes or hangs


✗ Forgetting cancel_dump_traceback_later() after the guarded work — the timer still fires and dumps (or exits)

✓ Always cancel the watchdog once the protected section completes, or scope it tightly

Key Rules

  • faulthandler.enable(file=sys.stderr, all_threads=True) installs handlers for SIGSEGV/SIGFPE/SIGABRT; python -X faulthandler enables it from startup
  • dump_traceback(file=...) prints stacks on demand (the file must have a real fileno()); dump_traceback_later(timeout, exit=False) is a hang watchdog you disarm with cancel_dump_traceback_later()
  • register(signum) makes a chosen signal dump tracebacks — the way to inspect a stuck process without attaching a debugger

Exercises

  1. enable() the handler, assert is_enabled(), then dump_traceback into a TemporaryFile and confirm the captured text contains a File line.
  2. Arm dump_traceback_later(timeout), do a bit of quick work, and cancel_dump_traceback_later() before it fires; explain what would print if you did not cancel.
  3. Describe how register(signal.SIGUSR1) lets you inspect a hung process from another terminal, and why that beats attaching a debugger to production.
Connections: faulthandler complements traceback (formatting caught exceptions) and pdb (interactive stepping) — it is the tool for the crashes and hangs those cannot reach; its signal hooks build directly on the signal module.

New Concepts — File & System Metadata (2 concepts)

Two small modules that decode the raw values the OS hands back. stat interprets the packed st_mode from os.stat — is this a file, a directory, a symlink? what are its permission bits? errno gives stable, symbolic names to the numeric error codes carried in every OSError, so you can branch on which failure occurred without parsing English messages.

File System

Concept 3: stat — Interpreting File Metadata

Why learn this? os.stat(path).st_mode is a single integer that packs both the file type and its permission bits. The stat module decodes it: S_ISDIR/S_ISREG/S_ISLNK classify the type, S_IMODE extracts the permission bits, filemode() renders the familiar ls -l string, and named constants let you test individual permission flags.
Example 1 — classify a mode's file type
import stat

print(stat.S_ISDIR(0o040755))   # True  -- 0o040000 marks a directory
print(stat.S_ISREG(0o100644))   # True  -- 0o100000 marks a regular file
print(stat.S_ISLNK(0o120777))   # True  -- 0o120000 marks a symlink
Output
True True True
Example 2 — permission bits & the ls-style string
import stat

mode = 0o100644                    # regular file + rw-r--r--
print(stat.S_ISREG(mode))          # True
print(oct(stat.S_IMODE(mode)))     # 0o644 -- permission bits only
print(stat.filemode(mode))         # -rw-r--r-- -- like ls -l
Output
True 0o644 -rw-r--r--
Example 3 — test & combine permission flags
import stat

mode = 0o100640                       # rw-r-----
print(bool(mode & stat.S_IRUSR))     # True  -- owner may read
print(bool(mode & stat.S_IWOTH))     # False -- world may not write
rwx = stat.S_IRWXU | stat.S_IRGRP    # 0o700 | 0o040
print(oct(rwx))                     # 0o740
Output
True False 0o740

Common Mistakes

✗ Comparing st_mode directly to 0o644 — the mode also encodes the file type in its high bits, so the equality fails

✓ Mask with stat.S_IMODE(mode) to get just the permission bits, or test the type with S_ISREG/S_ISDIR


✗ Assuming os.stat reports the symlink itself — it follows links, so you see the target's metadata

✓ Use os.lstat() (or os.stat(..., follow_symlinks=False)) and S_ISLNK when you must detect the link

Key Rules

  • os.stat(path).st_mode packs file type + permission bits; decode type with S_ISREG/S_ISDIR/S_ISLNK and the low 12 permission bits with S_IMODE(mode)
  • stat.filemode(mode) renders the ls -l string (e.g. -rw-r--r--); permission constants (S_IRUSR, S_IWGRP, S_IXOTH, S_IRWXU) test or build modes by &/|
  • os.stat follows symlinks, os.lstat does not — pair S_ISLNK with lstat; st_size and st_mtime on the same result carry size and modification time

Exercises

  1. Given literal modes for a directory, a regular file, and a symlink, classify each with S_ISDIR/S_ISREG/S_ISLNK.
  2. Take 0o100644, print S_IMODE as octal and filemode() as the ls-style string.
  3. From a mode, test the owner-read and world-write bits with S_IRUSR/S_IWOTH, then OR S_IRWXU with S_IRGRP and print the octal result.
Connections: stat decodes the st_mode from os.stat/os.lstat and pathlib.Path.stat(); its type predicates back Path.is_dir()/is_file()/is_symlink(), and its permission constants pair with os.chmod and the filecmp/shutil file work.
System

Concept 4: errno — Symbolic System Error Codes

Why learn this? Every OSError carries a numeric .errno from the operating system. The errno module gives those numbers stable, symbolic names (ENOENT, EACCES, EEXIST) so you can branch on which failure happened without parsing localized message strings — and map a code back to its name or a human-readable description.
Example 1 — inspect an OSError's errno
import errno

try:
    open("/no/such/file")
except OSError as e:
    print(e.errno == errno.ENOENT)    # True -- "No such file or directory"
    print(errno.errorcode[e.errno])   # 'ENOENT' -- number back to name
Output
True ENOENT
Example 2 — names, numbers & human strings
import errno, os

print(errno.ENOENT)                    # 2   -- the numeric code
print(errno.errorcode[errno.EACCES])   # 'EACCES'
print(os.strerror(errno.ENOENT))       # No such file or directory
Output
2 EACCES No such file or directory
Example 3 — branch on the specific error
import errno

def read_or_default(path, default=""):
    try:
        with open(path) as f:
            return f.read()
    except OSError as e:
        if e.errno == errno.ENOENT:      # missing file: fall back
            return default
        raise                            # permissions etc.: propagate

print(read_or_default("/no/such/file", "fallback"))   # fallback
Output
fallback

Common Mistakes

✗ Matching errors by parsing the exception's message text — it is localized and can change; the errno number is stable

✓ Compare e.errno to errno.ENOENT/EACCES/etc. (or catch the specific subclass)


✗ Reaching for raw errno when a dedicated subclass already exists — e.g. FileNotFoundError, PermissionError

✓ Prefer the subclass when one exists; drop to e.errno for codes without a class (e.g. ENOSPC, ETIMEDOUT)

Key Rules

  • errno exposes the platform's error numbers as names (errno.ENOENT, errno.EACCES, errno.EEXIST) — the codes carried in OSError.errno
  • errno.errorcode maps a number back to its name; os.strerror(n) returns the human-readable message for a code
  • Since Python 3.3 the common ones have OSError subclasses (FileNotFoundError=ENOENT, PermissionError=EACCES/EPERM, FileExistsError=EEXIST) — use those first, errno for the rest

Exercises

  1. Trigger a FileNotFoundError by opening a missing path, catch OSError, and assert e.errno == errno.ENOENT while printing errorcode[e.errno].
  2. Print the numbers for ENOENT and EACCES and the os.strerror() message for each.
  3. Write a helper that returns a default on ENOENT but re-raises every other OSError, and demonstrate it on a missing file.
Connections: errno gives the stable codes behind OSError and its subclasses (FileNotFoundError, PermissionError) raised across os, pathlib, socket, and subprocess; os.strerror turns a code into the message, mirroring the C errno it wraps.

New Concepts — Simple Persistence (1 concept)

One module for the simplest kind of durable storage: a dictionary that lives on disk. dbm gives you a fast, byte-keyed key-value store with the familiar []/in/.get() protocol — no SQL, no schema. It is the storage layer beneath shelve, and the right tool when you need a persistent map without the weight of a database.

Persistence

Concept 5: dbm — Persistent Key-Value Stores

Why learn this? Sometimes you need values to survive a restart but a full database is overkill. dbm.open() returns a dict-like object backed by a file on disk — assign, look up, test membership, and delete just like a dict, and it is all persisted. It is fast, dependency-free, and forms the foundation that shelve builds on to store arbitrary Python objects.
Example 1 — open, write, reopen (data persists)
import dbm, tempfile, os

path = os.path.join(tempfile.mkdtemp(), "cache")
with dbm.open(path, "c") as db:      # 'c' = open/create for read+write
    db["lang"] = "python"            # str auto-encoded to bytes
    db["n"] = "42"

with dbm.open(path, "r") as db:      # reopen read-only
    print(db["lang"])                # b'python' -- values come back as bytes
    print(db["n"] == b"42")          # True
Output
b'python' True
Example 2 — dict-like membership & keys
import dbm, tempfile, os

path = os.path.join(tempfile.mkdtemp(), "kv")
with dbm.open(path, "c") as db:
    db["a"], db["b"] = "1", "2"
    print("a" in db)                   # True -- membership test
    print(sorted(db.keys()))         # [b'a', b'b'] -- keys are bytes
    print(db.get("missing", b"?"))   # b'?' -- default like a dict
Output
True [b'a', b'b'] b'?'
Example 3 — which backend & missing keys
import dbm, tempfile, os

path = os.path.join(tempfile.mkdtemp(), "store")
with dbm.open(path, "n") as db:      # 'n' = always create a new empty db
    db["x"] = "1"
print(dbm.whichdb(path) is not None)   # True -- reports the backend used

with dbm.open(path, "r") as db:
    try:
        db["nope"]
    except KeyError:
        print("missing key raises KeyError")   # like a dict
Output
True missing key raises KeyError

Common Mistakes

✗ Expecting values back as str — dbm stores and returns bytes (str is auto-encoded on write but not decoded on read)

✓ Decode on read (db[k].decode()), or use shelve (pickle on top of dbm) to store arbitrary Python objects


✗ Assuming one portable on-disk format — the concrete backend (gdbm, ndbm, dumb) varies by platform and they are not interchangeable

✓ Detect it with dbm.whichdb(path); force dbm.dumb for a pure-Python, portable (if slower) store

Key Rules

  • dbm.open(path, flag) returns a dict-like store; flags: 'r' read, 'w' read/write existing, 'c' create if needed, 'n' always new — close it (use a with block)
  • Keys and values are bytes (str is auto-encoded on write); supports [], in, keys(), get(), and del — a missing key raises KeyError
  • dbm picks a backend (gdbm/ndbm/dumb) automatically; dbm.whichdb(path) identifies it; shelve builds an object store on top by pickling values

Exercises

  1. Open a store with flag 'c', write two string values, reopen it 'r', and show the values come back as matching bytes.
  2. Populate a store and exercise the dict-like protocol: in, .get() with a default, and sorted(db.keys()).
  3. Create a fresh db with 'n', confirm dbm.whichdb() reports a backend, and show that reading an absent key raises KeyError.
Connections: dbm is the on-disk dict behind shelve (which adds pickle so you can store any object); it complements sqlite3 (relational, queryable) and the pickle/json serialization work — reach for it when you need a simple, fast, persistent key-value map.
