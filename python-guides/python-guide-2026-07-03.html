Review Section (6 items)

1. itertools due today rep 3 → 4

itertools supplies lazy iterator building blocks: chain concatenates, islice slices without indexing, groupby groups consecutive equal keys (sort first!), product/permutations/combinations enumerate, and accumulate runs a running fold. 15 days since last review.

Key Pattern
from itertools import chain, islice, groupby, accumulate, combinations

print(list(chain([1, 2], [3, 4])))          # [1, 2, 3, 4]
print(list(islice(range(100), 2, 8, 2)))    # [2, 4, 6]
print(list(accumulate([1, 2, 3, 4])))       # [1, 3, 6, 10] -- running sum
print(list(combinations("abc", 2)))         # [('a','b'),('a','c'),('b','c')]

data = [("a", 1), ("a", 2), ("b", 3)]
for k, grp in groupby(data, key=lambda t: t[0]):
    print(k, [t[1] for t in grp])            # a [1, 2] then b [3]
Challenge

Given a list of (dept, salary) tuples sorted by dept, use groupby to compute each department's total with sum. Then use islice to page through a large generator ten items at a time, and combinations to list every unique pair from a list. Explain why groupby needs the input pre-sorted on the grouping key.

Pitfall

groupby only groups consecutive equal keys — unsorted input yields fragmented groups. itertools results are one-shot iterators: once consumed they are empty, and each group iterator from groupby is invalidated as soon as you advance to the next group (materialize it with list() if you need it later).

2. operator Module due today rep 3 → 4

The operator module exposes Python's operators as functions — itemgetter, attrgetter, and methodcaller build fast, picklable key=/callback functions that replace trivial lambdas, and add/mul/lt feed reduce and functional code. 15 days since last review.

Key Pattern
from operator import itemgetter, attrgetter, methodcaller

rows = [("Ann", 30), ("Bo", 25)]
print(sorted(rows, key=itemgetter(1)))       # by 2nd field -> Bo, Ann
print(itemgetter(0, 1)(rows[0]))            # ('Ann', 30) -- a tuple of fields

words = ["hi", "HELLO", "Yo"]
print(sorted(words, key=methodcaller("lower")))   # calls x.lower() as the key

from functools import reduce
from operator import mul
print(reduce(mul, [1, 2, 3, 4]))              # 24
Challenge

Sort a list of dataclass instances by two attributes at once using a single attrgetter("a", "b"). Replace a lambda that calls .strip() on each item with methodcaller("strip"). Then sum a list with reduce(operator.add, xs), and name one advantage itemgetter has over the equivalent lambda (hint: speed and picklability for multiprocessing).

Pitfall

itemgetter(i) returns a callable — you must call it on the item (itemgetter(1)(row)), not pass the index directly. attrgetter("a.b") follows dotted paths; a single-argument getter returns a scalar, but with multiple arguments it returns a tuple.

3. threading vs multiprocessing vs asyncio 22 days overdue rep 1 → 2

Choose by workload: threading for I/O-bound blocking calls (shares memory, but the GIL serializes CPU work); multiprocessing for CPU-bound work (separate processes, true parallelism, pickling overhead); asyncio for high-concurrency I/O with cooperative single-threaded tasks. 23 days since last review.

Key Pattern
import math
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# CPU-bound -> processes bypass the GIL for real parallelism
def is_prime(n):
    return n > 1 and all(n % d for d in range(2, math.isqrt(n) + 1))

nums = [112272535095293, 112582705942171]
with ProcessPoolExecutor() as ex:
    print(list(ex.map(is_prime, nums)))   # [True, True]

# I/O-bound -> threads overlap the waiting (GIL released during blocking I/O)
with ThreadPoolExecutor(max_workers=8) as ex:
    ex.map(str.upper, ["a", "b", "c"])   # stand-in for blocking calls
Challenge

For three tasks — hashing 10 large files, downloading 100 URLs, and summing a huge list of numbers — state whether you would pick threading, multiprocessing, or asyncio, and why. Then explain how the GIL makes threads useless for CPU-bound pure-Python work but fine for I/O, and what overhead processes add.

Pitfall

Threads do NOT speed up CPU-bound pure-Python code — the GIL serializes bytecode execution; reach for processes. Processes cannot share arbitrary state and must pickle arguments and results. asyncio only helps if your libraries are async-aware; one blocking call stalls the entire event loop.

4. async/await Basics 22 days overdue rep 1 → 2

async def defines a coroutine; calling it returns a coroutine object that does nothing until awaited or scheduled. await yields control back to the event loop during I/O; asyncio.gather runs many coroutines concurrently on ONE thread. 23 days since last review.

Key Pattern
import asyncio

async def fetch(name, delay):
    await asyncio.sleep(delay)      # yields to the loop; NOT time.sleep!
    return f"{name} done"

async def main():
    # gather runs both concurrently -> ~1s total, not 1.5s
    results = await asyncio.gather(fetch("a", 1), fetch("b", 0.5))
    print(results)

asyncio.run(main())     # ['a done', 'b done']
Challenge

Write an async main that launches three fetch coroutines with asyncio.gather and measures total wall time with time.perf_counter, showing it is close to the longest single delay, not their sum. Then move a blocking call into a thread with asyncio.to_thread, and explain why a plain time.sleep inside a coroutine would block the whole loop.

Pitfall

Calling a coroutine without await (or scheduling) just builds a coroutine object and warns "coroutine was never awaited". Blocking calls (time.sleep, requests.get) inside a coroutine freeze the entire event loop — use await asyncio.sleep, async libraries, or asyncio.to_thread.

5. concurrent.futures 22 days overdue rep 1 → 2

concurrent.futures gives a uniform pool API: ThreadPoolExecutor (I/O) and ProcessPoolExecutor (CPU) share one interface. submit() returns a Future; map() applies a function across an iterable; as_completed() yields futures as they finish. 23 days since last review.

Key Pattern
from concurrent.futures import ThreadPoolExecutor, as_completed

def work(n):
    return n * n

with ThreadPoolExecutor(max_workers=4) as ex:
    futures = [ex.submit(work, i) for i in range(5)]
    for fut in as_completed(futures):        # completion order
        print(fut.result(), end=" ")          # 0 1 4 9 16 (some order)
    print()
    print(list(ex.map(work, range(5))))      # [0, 1, 4, 9, 16] -- input order
Challenge

Use submit + as_completed to process a list of jobs and print each result as soon as it finishes; then rewrite with ex.map and contrast the ordering guarantees. Show that an exception raised inside a worker surfaces only when you call fut.result(), and that the with-block waits for all pending futures on exit.

Pitfall

map() preserves INPUT order and re-raises the first exception when you iterate; as_completed() yields COMPLETION order. Worker exceptions are stored on the Future and only re-raised at .result(). ProcessPoolExecutor requires top-level, picklable functions — no lambdas or closures.

6. Profiling: timeit & cProfile 22 days overdue rep 1 → 2

Measure before optimizing. timeit micro-benchmarks a small snippet over many loops (isolating setup); cProfile profiles a whole program, ranking functions by cumulative/total time so you find the real hotspot. 23 days since last review.

Key Pattern
import timeit
# time a snippet -- setup runs once, stmt runs number= times
t = timeit.timeit("sum(range(100))", number=100_000)
print(f"{t:.4f}s")

import cProfile, pstats
def slow():
    return sum(i * i for i in range(10_000))

cProfile.run("slow()", "prof.out")
pstats.Stats("prof.out").sort_stats("cumulative").print_stats(3)
Challenge

Use timeit to compare a list comprehension vs map vs a manual loop for squaring a range, controlling for setup with the setup= argument. Then profile a function that calls two helpers with cProfile, sort by cumulative time, and identify the hotspot. Explain why you should profile before optimizing and why a single time.time() delta is a poor micro-benchmark.

Pitfall

timeit disables the garbage collector and runs many loops — a lone time.time() delta is noisy and misleading. Put imports and data in the setup= argument so you do not time them. cProfile adds overhead, so use it to find relative hotspots, not absolute production timings.

New Concepts — Generators & Coroutines (2 concepts)

Two features that turn generators from simple iterators into composable, communicating building blocks. yield from delegates iteration — and send/throw — to a sub-generator, flattening recursion and pipelines into one line; and the coroutine protocol (send(), throw(), close()) lets a paused generator receive data and clean up after itself, the very machinery classic asyncio was built on.

Generators

Concept 1: yield from — Delegating to Sub-Generators

Why learn this? Writing for x in sub: yield x re-yields a sub-generator's items, but it drops the sub-generator's return value and does not forward send()/throw(). yield from (PEP 380) does all of that in one line — making it the clean way to flatten recursive structures, compose generator pipelines, and build coroutines.
Example 1 — delegation replaces a re-yield loop
def chain_all(*iterables):
    for it in iterables:
        yield from it        # re-yields every item of it

print(list(chain_all([1, 2], (3, 4), "ab")))
# [1, 2, 3, 4, 'a', 'b']
Output
[1, 2, 3, 4, 'a', 'b']
Example 2 — recursion: flatten arbitrary nesting
def flatten(nested):
    for item in nested:
        if isinstance(item, list):
            yield from flatten(item)   # recurse into sublists
        else:
            yield item

print(list(flatten([1, [2, [3, 4], 5], [6]])))
# [1, 2, 3, 4, 5, 6]
Output
[1, 2, 3, 4, 5, 6]
Example 3 — yield from captures the sub-generator's return
def inner():
    yield 1
    yield 2
    return "done"          # PEP 380: a generator's return value

def outer():
    result = yield from inner()   # captures that return value
    print("inner returned:", result)
    yield 3

print(list(outer()))
Output
inner returned: done [1, 2, 3]
Example 4 — send() flows straight through the delegator
def averager():
    total = count = 0
    avg = None
    while True:
        x = yield avg
        total += x
        count += 1
        avg = total / count

def delegator():
    yield from averager()   # send() values pass right through

d = delegator()
next(d)                     # prime
print(d.send(10))           # 10.0
print(d.send(20))           # 15.0
Output
10.0 15.0

Common Mistakes

✗ Writing for x in sub: yield x when you also need the sub-generator's return value or send/throw forwarding

✓ Use yield from sub — it forwards values and exceptions and evaluates to the sub-generator's return value


✗ Expecting yield from x to work on a non-iterable, or thinking it only accepts generators

✓ The right-hand side may be any iterable (list, tuple, str, range) — not just a generator

Key Rules

  • yield from iterable re-yields every item; for plain iteration it equals a for-loop but is shorter and faster
  • With a sub-generator it also forwards .send()/.throw() and evaluates to the sub-generator's return value (result = yield from gen())
  • Ideal for recursion (tree/nested flattening) and for composing generator pipelines and coroutines

Exercises

  1. Write flatten(nested) handling arbitrary nesting of lists and tuples using yield from recursively. Input [1, [2, [3]], 4] should yield 1, 2, 3, 4.
  2. Write concat(*iters) that yields the items of each iterable in order using yield from; test it with a list, a range, and a string.
  3. Write a sub-generator that yields values and returns how many it produced; in a delegating generator capture that count with n = yield from sub() and yield it at the end.
Connections: yield from generalizes generator expressions and mirrors itertools.chain (today's Review 1); its send/throw forwarding is exactly what powers the coroutine protocol in Concept 2; the recursive flattening pattern echoes tree traversal in graph/DFS problems.
Coroutines

Concept 2: Generators as Coroutines — send(), throw(), close()

Why learn this? A generator paused at yield can receive values via .send(), be signalled with .throw(), and be cleaned up with .close(). This turns a generator into a coroutine — a resumable consumer used for running aggregates, pipelines, and cooperative tasks (the mechanism classic asyncio was built on before async def).
Example 1 — yield as an expression receives values
def echo():
    while True:
        received = yield          # yield expression: gets what send() passes
        print("got:", received)

e = echo()
next(e)              # prime: advance to the first yield
e.send("hello")      # got: hello
e.send(42)           # got: 42
Output
got: hello got: 42
Example 2 — a running-average coroutine that yields back
def averager():
    total = count = 0
    while True:
        x = yield total / count if count else None
        total += x
        count += 1

a = averager()
next(a)              # prime -> yields None
print(a.send(10))    # 10.0
print(a.send(30))    # 20.0
print(a.send(20))    # 20.0
Output
10.0 20.0 20.0
Example 3 — close() raises GeneratorExit for cleanup
def worker():
    try:
        while True:
            item = yield
            print("processing", item)
    finally:
        print("cleanup!")    # runs on .close()

w = worker()
next(w)
w.send("a")          # processing a
w.close()            # GeneratorExit -> finally runs -> cleanup!
Output
processing a cleanup!
Example 4 — throw() injects an exception at the pause
def resilient():
    while True:
        try:
            x = yield
            print("ok:", x)
        except ValueError:
            print("recovered from ValueError")

r = resilient()
next(r)
r.send(1)            # ok: 1
r.throw(ValueError)  # recovered from ValueError
r.send(2)            # ok: 2  -- still alive
Output
ok: 1 recovered from ValueError ok: 2

Common Mistakes

✗ Calling gen.send(value) before priming — raises TypeError: can't send non-None value to a just-started generator

✓ Prime the coroutine first with next(gen) or gen.send(None) so it reaches the first yield


✗ Relying on garbage collection to clean up an open coroutine that holds a resource

✓ Put cleanup in a finally block and call .close() explicitly (or wrap it in contextlib.closing)

Key Rules

  • x = yield makes the generator a consumer: send(v) resumes it, v becomes the value of the yield expression, and it runs to the next yield
  • Prime with next() or .send(None) before the first real send
  • .throw(exc) raises at the pause point; .close() raises GeneratorExit — put cleanup in finally

Exercises

  1. Write a coroutine accumulator() that receives numbers via send and yields the running total after each. Prime it, then send 1, 2, 3 and expect 1, 3, 6.
  2. Write a grep(pattern) coroutine that receives lines via send and prints those containing pattern; demonstrate priming and then .close().
  3. Write a coroutine that counts how many values it processed and prints the count in a finally block when .close() is called.
Connections: this is the send/throw machinery that yield from (Concept 1) transparently forwards; it is the historical foundation of the coroutines behind async/await (today's Review 4); the priming step is often automated with a small decorator (Decorators topic), and the finally cleanup mirrors context managers.

New Concepts — The Python Data Model (3 concepts)

Three dunder hooks that let your objects participate in core language operations. __format__ defines what f"{obj:spec}" means; __bool__ decides whether if obj: sees the object as truthy; and __getattr__/__setattr__ intercept attribute access for proxies, lazy defaults, and validation. Implementing them makes custom types feel built-in.

Data Model

Concept 3: __format__ — Custom Format Specifiers

Why learn this? format(obj, spec), an f-string f"{obj:spec}", and "{:spec}".format(obj) all call obj.__format__(spec). Implementing it lets your type define its own mini-format-language, so callers get meaningful, type-aware formatting instead of a TypeError on any non-empty spec.
Example 1 — delegate the spec to a wrapped value
class Money:
    def __init__(self, cents):
        self.cents = cents
    def __format__(self, spec):
        dollars = self.cents / 100
        return "$" + format(dollars, spec or ".2f")

m = Money(123456)
print(f"{m}")            # $1234.56   -- empty spec -> default .2f
print(f"{m:,.2f}")       # $1,234.56  -- thousands separator
print(format(m, ".0f"))  # $1235      -- rounds
Output
$1234.56 $1,234.56 $1235
Example 2 — invent your own spec words
import math

class Vec2:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __format__(self, spec):
        if spec == "polar":
            r = math.hypot(self.x, self.y)
            a = math.degrees(math.atan2(self.y, self.x))
            return f"({r:.2f} @ {a:.0f}deg)"
        return f"({self.x}, {self.y})"

v = Vec2(3, 4)
print(f"{v}")          # (3, 4)
print(f"{v:polar}")    # (5.00 @ 53deg)
Output
(3, 4) (5.00 @ 53deg)
Example 3 — conversions (!r/!s) bypass __format__
class Tag:
    def __format__(self, spec):
        return f"<{spec or 'span'}>"
    def __repr__(self):
        return "Tag()"

print("{:div}".format(Tag()))   # <div>  -- spec passed through
print(f"{Tag():h1}")            # <h1>
print(f"{Tag()!r}")             # Tag()  -- !r calls repr, NOT __format__
Output
<div> <h1> Tag()

Common Mistakes

✗ Overriding only __str__ and expecting f"{obj:>10}" to align it — the default object.__format__ raises TypeError on a non-empty spec

✓ Implement __format__ (often delegating with format(self.value, spec)) when you want format specs to work on your type


✗ Assuming f"{obj!r:>10}" passes the spec into __format__

✓ Order matters: the conversion (!r/!s/!a) runs first, then the :spec formats the resulting string

Key Rules

  • format(obj, spec), f"{obj:spec}", and "{:spec}".format(obj) all call obj.__format__(spec); an empty spec is the common default path
  • The default object.__format__ forwards to str(self) for an empty spec and raises TypeError for any non-empty spec
  • Conversion flags !s, !r, !a run before formatting and bypass your __format__ entirely

Exercises

  1. Write a Percent wrapping a 0..1 float whose __format__ renders "37.0%" by default and honors a precision, so f"{p:.2}" gives more decimals.
  2. Give Duration(seconds) a __format__ that returns "1h 02m 03s" for spec "hms" and the raw total seconds for an empty spec.
  3. Show that f"{obj!r}" ignores your __format__ by adding a distinct __repr__ and printing both an f-string spec and the !r form.
Connections: __format__ builds on the format-spec mini-language and f-strings (Strings topic); it is to format() what __str__ is to str() and __repr__ is to repr(); it joins __bool__ and __getattr__ below as core data-model hooks.
Data Model

Concept 4: __bool__ & the Truthiness Protocol

Why learn this? if obj, while obj, and/or, not obj, and bool(obj) all ask "is this truthy?". By default every object is truthy; define __bool__ (or __len__) so empty or zero-like instances read as falsy and work naturally in conditionals.
Example 1 — bool() falls back to __len__
class Basket:
    def __init__(self, items):
        self.items = list(items)
    def __len__(self):
        return len(self.items)     # bool() uses this when __bool__ is absent

b = Basket([])
print(bool(b))              # False  -- len 0
print(bool(Basket([1, 2])))  # True
if not b:
    print("empty basket")      # prints
Output
False True empty basket
Example 2 — explicit __bool__ decides truthiness
class Account:
    def __init__(self, balance):
        self.balance = balance
    def __bool__(self):
        return self.balance > 0     # truthy only when in the black

print(bool(Account(50)))   # True
print(bool(Account(0)))    # False
print(bool(Account(-5)))   # False
Output
True False False
Example 3 — truthiness drives the or-default idiom
class Config:
    def __init__(self, data):
        self.data = data
    def __bool__(self):
        return bool(self.data)

user = Config({})
default = Config({"theme": "dark"})
chosen = user or default          # picks first truthy -> default
print(chosen.data)                # {'theme': 'dark'}
Output
{'theme': 'dark'}
Example 4 — __bool__ takes priority over __len__
class Weird:
    def __len__(self):
        print("len called")
        return 0
    def __bool__(self):
        return True         # checked first; __len__ never consulted

print(bool(Weird()))   # True  -- 'len called' never prints
Output
True

Common Mistakes

✗ Writing if obj != None: or if len(obj) > 0: when a plain if obj: reads better once truthiness is defined

✓ Define __bool__/__len__ so if obj: means what you expect, then use it directly


✗ Assuming an object whose __len__ returns 0 is still truthy

✓ Remember the order: __bool__ is checked first; if absent, __len__ == 0 is falsy; if neither exists, the object is always truthy

Key Rules

  • Truthiness is resolved by __bool__ first, then __len__ (0 → falsy), otherwise the object is truthy
  • __bool__ must return an actual bool; and/or still return operands, using truthiness only to choose which
  • Prefer if obj: over if len(obj) > 0: or == None once truthiness is well-defined

Exercises

  1. Write a Stack whose truthiness is False when empty (via __len__) and demonstrate while stack: stack.pop().
  2. Write a Temperature whose __bool__ is True only above absolute zero (−273.15 C); test at -300, 0, and 20.
  3. Show that a class whose __len__ returns 0 becomes truthy once you add __bool__ returning True, and prove bool() never calls __len__ by adding a print to it.
Connections: truthiness underlies the and/or short-circuit and the or-default idiom from Day 32; __len__ also powers len() and container behavior (Dunder container methods, OOP); __bool__ sits alongside __format__ and __getattr__ as a core data-model hook.
Data Model

Concept 5: __getattr__ & __setattr__ — Attribute Interception

Why learn this? __getattr__ is called only when normal attribute lookup fails — perfect for lazy defaults, proxies, and friendly errors. __setattr__ intercepts every assignment — useful for validation, read-only fields, or logging. Together they let objects behave dynamically without a descriptor for each field.
Example 1 — __getattr__ as a fallback for missing names
class Defaults:
    def __init__(self, **data):
        self.data = data
    def __getattr__(self, name):
        # called ONLY when normal lookup fails
        return self.data.get(name, f"<no {name}>")

d = Defaults(color="red")
print(d.color)     # red      -- missing attr -> __getattr__ -> data
print(d.size)      # <no size>
print(d.data)      # {'color': 'red'} -- real attr, __getattr__ NOT called
Output
red <no size> {'color': 'red'}
Example 2 — __setattr__ validates every assignment
class Validated:
    def __setattr__(self, name, value):
        if name == "age" and value < 0:
            raise ValueError("age must be >= 0")
        super().__setattr__(name, value)   # avoid infinite recursion!

p = Validated()
p.age = 30
print(p.age)       # 30
try:
    p.age = -1
except ValueError as e:
    print(e)       # age must be >= 0
Output
30 age must be >= 0
Example 3 — __getattribute__ runs for EVERY access
class Trace:
    def __getattribute__(self, name):
        print("access:", name)
        return object.__getattribute__(self, name)   # delegate, avoid recursion
    def __getattr__(self, name):
        return "MISSING"      # only when __getattribute__ raises AttributeError

t = Trace()
t.x = 1
print(t.x)        # access: x  -> 1
print(t.zzz)      # access: zzz -> MISSING
Output
access: x 1 access: zzz MISSING
Example 4 — a read-only object via __setattr__
class Frozen:
    def __init__(self, **kw):
        for k, v in kw.items():
            object.__setattr__(self, k, v)   # bypass the guard during init
    def __setattr__(self, name, value):
        raise AttributeError("read-only")

f = Frozen(x=1, y=2)
print(f.x, f.y)   # 1 2
try:
    f.x = 5
except AttributeError as e:
    print(e)      # read-only
Output
1 2 read-only

Common Mistakes

✗ Writing self.name = value inside __setattr__ — each assignment re-enters __setattr__, causing infinite recursion

✓ Delegate to super().__setattr__(name, value) or object.__setattr__(self, name, value)


✗ Confusing __getattr__ (fallback, only on failure) with __getattribute__ (called for every access)

✓ Use __getattr__ for defaults/proxies; touch __getattribute__ only when you must intercept everything — and always delegate to object.__getattribute__

Key Rules

  • __getattr__(self, name) runs only when normal lookup fails; __getattribute__ runs on every access (delegate to object.__getattribute__ to avoid recursion)
  • __setattr__(self, name, value) intercepts all assignments; always store via super().__setattr__/object.__setattr__, never self.name = ...
  • Great for proxies, lazy attributes, validation, and immutability — but overuse hurts readability and performance

Exercises

  1. Write a Proxy wrapping any object that forwards unknown attribute reads to the wrapped object via __getattr__ (use getattr(self._obj, name)).
  2. Write a Typed class whose __setattr__ enforces that any attribute whose name ends in _count must be an int, raising TypeError otherwise.
  3. Build a case-insensitive record where __getattr__ lets both r.Name and r.name resolve to the same stored value.
Connections: __getattr__/__setattr__ are the dynamic counterpart to property and descriptors (OOP); they complete this section's data-model trio with __format__ and __bool__; the proxy/forwarding pattern echoes ChainMap-style layered lookup and functools.partial-style wrapping from earlier days.
