Review Section (6 items)

1. threading vs multiprocessing vs asyncio due today rep 2 → 3

Three concurrency models, three jobs. threading runs many tasks in one process sharing memory — but the GIL serializes CPU-bound Python bytecode, so it only helps I/O-bound work (the GIL is released during blocking I/O). multiprocessing spawns separate processes for true parallelism on CPU-bound work, at the cost of pickling data across the process boundary. asyncio is single-threaded cooperative multitasking on an event loop — unbeatable for huge fan-out I/O (thousands of sockets), but only with async-aware libraries. 6 days since last review.

Key Pattern
# CPU-bound      -> multiprocessing  (sidestep the GIL, real cores)
# I/O + blocking -> threading        (GIL released during I/O waits)
# I/O, massive   -> asyncio          (one thread, thousands of tasks)
import concurrent.futures as cf

with cf.ThreadPoolExecutor() as ex:      # I/O-bound fan-out
    io_results = list(ex.map(fetch, urls))
with cf.ProcessPoolExecutor() as ex:     # CPU-bound crunching
    cpu_results = list(ex.map(crunch, chunks))
Challenge

Pick the right model for each and justify it in terms of the GIL and per-worker cost: (a) SHA-256 hashing 10,000 files, (b) 5,000 HTTP requests with a blocking client, (c) serving 50,000 mostly-idle websocket connections. Then explain why threading does not speed up (a) even on a 16-core machine.

Pitfall

The GIL means threads do NOT parallelize CPU-bound pure-Python code — they time-slice one core. asyncio needs async-aware libraries: a single blocking call (time.sleep, requests.get) stalls the whole event loop. multiprocessing must pickle arguments and results, so it cannot share arbitrary live objects and pays real IPC overhead.

2. async/await Basics due today rep 2 → 3

async def defines a coroutine; calling it returns a coroutine object that does nothing until it is awaited or scheduled. await hands control back to the event loop while waiting, letting other tasks run. asyncio.run(main()) starts the loop; asyncio.gather(*coros) runs many concurrently and collects their results in order. 6 days since last review.

Key Pattern
import asyncio

async def fetch(n):
    await asyncio.sleep(0.01)     # yields to loop -- NOT time.sleep
    return n * 2

async def main():
    return await asyncio.gather(fetch(1), fetch(2), fetch(3))

print(asyncio.run(main()))    # [2, 4, 6] -- ran concurrently
Challenge

Write a coroutine that await asyncio.sleep(0.1) then returns a value; run three of them with gather and show total wall time is about 0.1s, not 0.3s. Then replace the await asyncio.sleep with a blocking time.sleep and explain why the concurrency collapses back to sequential.

Pitfall

Calling a coroutine without await produces a "coroutine was never awaited" warning and runs nothing. A blocking call inside a coroutine freezes the ENTIRE loop — offload it with await asyncio.to_thread(fn) or loop.run_in_executor. gather preserves argument order in its result list even though tasks finish out of order.

3. concurrent.futures due today rep 2 → 3

A uniform, high-level API over pools of threads or processes. Submit work with executor.submit() (returns a Future) or executor.map(); as_completed() yields futures in finish order; future.result() blocks for the value and re-raises worker exceptions. Swap ThreadPoolExecutor ↔ ProcessPoolExecutor with a one-line change. 6 days since last review.

Key Pattern
from concurrent.futures import ThreadPoolExecutor, as_completed

def work(n):
    return n * n

with ThreadPoolExecutor(max_workers=4) as ex:
    futures = [ex.submit(work, i) for i in range(5)]
    results = [f.result() for f in as_completed(futures)]

print(sorted(results))    # [0, 1, 4, 9, 16]
Challenge

Submit five tasks to a ThreadPoolExecutor, gather results via as_completed (note the order differs from submission), and show that f.result() re-raises an exception a task threw. Then state exactly what must change — and what new constraint appears — if you switch to ProcessPoolExecutor.

Pitfall

.result() blocks until the future completes and re-raises any exception from the worker — wrap it if you must keep going. With ProcessPoolExecutor the function AND its arguments must be picklable (no lambdas, no locally-defined functions). Exiting the with block waits for all pending work to finish.

4. Profiling: timeit & cProfile due today rep 2 → 3

timeit measures small snippets by running them many times, drowning out one-shot noise. cProfile profiles a whole program, attributing time to each function; pstats sorts the results. The rule: measure before you optimize — intuition about what is slow is usually wrong. 6 days since last review.

Key Pattern
import timeit, cProfile, pstats, io

t = timeit.timeit("sum(range(100))", number=10000)
print(t > 0)                    # True -- seconds for 10k runs

pr = cProfile.Profile()
pr.enable()
sum(range(100000))
pr.disable()
s = io.StringIO()
pstats.Stats(pr, stream=s).sort_stats("cumulative").print_stats(3)
print("function calls" in s.getvalue())   # True
Challenge

Use timeit to compare "".join(parts) against building the same string with repeated +=, and show join wins. Then profile a function with cProfile and print the top-3 rows by cumulative time. Explain why timeit disables the garbage collector by default and why a single time.time() delta is unreliable for fast code.

Pitfall

A lone time.time() difference is dominated by noise for fast code — timeit's repetition is the point. A timeit snippet that references your names needs globals=globals() or a setup= string. cProfile adds per-call overhead, so its absolute numbers are inflated — use it to compare relative costs, not to report real timings.

5. contextlib Deep Dive overdue 15d rep 1 → 2

contextlib turns cleanup patterns into with-statements. @contextmanager builds one from a generator (setup before yield, teardown after). suppress(exc) swallows chosen exceptions; closing(obj) guarantees .close(); ExitStack manages a dynamic number of context managers; nullcontext() is a do-nothing placeholder. 15 days since last review.

Key Pattern
from contextlib import contextmanager, suppress

@contextmanager
def tag(name):
    print(f"<{name}>")
    try:
        yield                    # body runs here
    finally:
        print(f"</{name}>")     # always runs, even on error

with tag("b"):
    print("hi")                # <b> / hi / </b>

with suppress(FileNotFoundError):
    open("/no/file")          # swallowed -- no crash
Challenge

Write a @contextmanager that prints "enter"/"exit" around its body even if the body raises (use try/finally). Then use an ExitStack to open a variable-length list of context managers inside one with-block and guarantee all get closed. Explain why the teardown must live in finally, not after the yield alone.

Pitfall

A @contextmanager generator must yield exactly once; without try/finally, an exception in the body skips your cleanup entirely. suppress silently swallows — never hand it a broad Exception. When the number of managers is only known at runtime, reach for ExitStack, not deeply nested with statements.

6. heapq & bisect Modules overdue 15d rep 1 → 2

heapq maintains a list as a binary min-heap: heappush/heappop keep the smallest at index 0 in O(log n) — the core of priority queues and top-k. bisect does O(log n) search and insertion into an already-sorted list: bisect_left finds an insertion point, insort keeps a list sorted as you add. 15 days since last review.

Key Pattern
import heapq, bisect

h = []
for x in (5, 1, 3):
    heapq.heappush(h, x)
print(heapq.heappop(h))              # 1 -- smallest first
print(heapq.nlargest(2, [5, 1, 3, 2]))  # [5, 3]

xs = [1, 3, 5]
print(bisect.bisect_left(xs, 3))    # 1 -- insertion point
bisect.insort(xs, 4)
print(xs)                           # [1, 3, 4, 5] -- stays sorted
Challenge

Use a heap to get the 3 largest values of a list without fully sorting it (heapq.nlargest). Then maintain a running sorted list with bisect.insort and use bisect to locate where a new value would go. Finally, explain how to build a max-heap out of heapq's min-heap, and why heap[0] is the only element you can trust to be in sorted position.

Pitfall

heapq is a MIN-heap; for a max-heap push negated values (or (-priority, item) tuples). heap[0] peeks the minimum, but the rest of the list is NOT sorted — iterate it and you get heap order, not sorted order. bisect assumes the list is already sorted; on an unsorted list its results are meaningless.

New Concepts — Executing & Auditing Code (2 concepts)

Two tools for controlling code as it runs. runpy is the machinery behind python -m pkg and python script.py — it lets your own program execute a module or file in a fresh namespace and read the results back out. sys.audit and audit hooks (PEP 578) give you a single choke point that observes every security-sensitive operation the interpreter performs — opening files, importing modules, spawning subprocesses — so you can log, monitor, or block them.

Runtime & Execution

Concept 1: runpy — Running Modules & Scripts Programmatically

Why learn this? runpy is the exact mechanism the interpreter uses for python -m name and python path.py. run_module(name, run_name="__main__") executes a module as if launched from the command line; run_path(path) runs a .py file (or a directory/zip containing __main__.py). Both return the resulting global namespace as a dict — ideal for launchers, task runners, plugin hosts, and test harnesses that must run code in a clean, throwaway namespace.
Example 1 — run a file, read its globals (run_path)
import runpy, tempfile, os

src = "RESULT = sum(range(10))\n"
path = os.path.join(tempfile.mkdtemp(), "job.py")
with open(path, "w") as f:
    f.write(src)

ns = runpy.run_path(path, run_name="__main__")   # execute the file
print(ns["RESULT"])       # 45 -- globals of the executed file
Output
45
Example 2 — run_name controls the __main__ guard
import runpy, tempfile, os

src = (
    "ran_main = __name__ == '__main__'\n"
    "MARK = 'guarded' if ran_main else 'imported'\n"
)
path = os.path.join(tempfile.mkdtemp(), "mod.py")
with open(path, "w") as f:
    f.write(src)

as_main = runpy.run_path(path, run_name="__main__")   # __name__ == "__main__"
as_lib  = runpy.run_path(path, run_name="lib")        # __name__ == "lib"
print(as_main["MARK"], as_lib["MARK"])   # guarded imported
Output
guarded imported
Example 3 — run a module by name (run_module)
import runpy

# Execute a stdlib module in a fresh namespace; run_name="loader"
# means its `if __name__ == "__main__"` block does NOT fire.
ns = runpy.run_module("base64", run_name="loader")
print("b64encode" in ns)     # True -- module's globals returned
print(callable(ns["b64encode"]))   # True
Output
True True

Common Mistakes

✗ Using importlib.import_module to "run a script" — import executes module-level code but sets __name__ to the module name, so the if __name__ == "__main__" block never runs

✓ Use runpy.run_module(name, run_name="__main__") to trigger that guarded block, exactly like python -m


✗ Expecting run_path/run_module to hand back a module object

✓ They return a plain dict of the resulting global namespace — read results out of it (ns["RESULT"])

Key Rules

  • runpy.run_module(mod_name, init_globals=None, run_name=None, alter_sys=False) executes a module found on sys.path; run_name sets its __name__ (pass "__main__" to fire the main guard)
  • runpy.run_path(path, run_name=None) runs a .py file, or a directory/zipfile containing __main__.py — the mechanism behind python dir_or_zip/
  • Both run in a fresh module namespace and return it as a dict; alter_sys=True also swaps sys.argv[0] and sys.modules["__main__"] for the duration of the run

Exercises

  1. Write a .py file that sets RESULT = sum(range(10)), run it with run_path(run_name="__main__"), and print ns["RESULT"].
  2. Run any stdlib module with run_module and show the returned dict contains one of its public names.
  3. Run the same file twice — once with run_name="__main__", once with run_name="lib" — and show a variable that depends on __name__ == "__main__" differs between the two runs.
Connections: runpy is the import-system counterpart to python -m — it builds on importlib to locate modules and on the __name__ == "__main__" convention; it pairs with zipapp (which runs a zip's __main__.py) and complements subprocess when you want to run code in-process instead of spawning a child.
Runtime Security

Concept 2: sys.audit & Audit Hooks (PEP 578)

Why learn this? Since Python 3.8 the interpreter emits audit events for security-sensitive operations — opening files, importing modules, compiling code, spawning subprocesses, connecting sockets. sys.addaudithook(hook) registers a callback that sees every event as (name, args), giving you one central place to log, monitor, or even block risky actions. sys.audit(event, *args) lets your own code raise custom events into the same pipeline.
Example 1 — observe a built-in event
import sys

seen = []
def hook(event, args):
    if event == "open":
        seen.append(args[0])       # args = (path, mode, flags)

sys.addaudithook(hook)
open("/etc/hostname").close()    # fires the "open" event
print("/etc/hostname" in seen)   # True
Output
True
Example 2 — raise & veto a custom event
import sys

def hook(event, args):
    if event == "myapp.charge" and args[0] > 100:
        raise PermissionError("amount too large")

sys.addaudithook(hook)
sys.audit("myapp.charge", 50)     # fine -- under the limit
print("charged 50")
try:
    sys.audit("myapp.charge", 500)   # hook raises -> blocked
except PermissionError as e:
    print(e)
Output
charged 50 amount too large
Example 3 — one hook sees many event types
import sys

counts = {}
def hook(event, args):
    counts[event] = counts.get(event, 0) + 1

sys.addaudithook(hook)
compile("1 + 1", "<s>", "eval")     # emits "compile"
__import__("json")                    # emits "import"
print(counts.get("compile", 0) >= 1)   # True
print(counts.get("import", 0) >= 1)    # True
Output
True True

Common Mistakes

✗ Trying to remove an audit hook — once added, a hook cannot be unregistered (by design, so untrusted code cannot disable auditing)

✓ Add hooks once at startup; gate their logic behind a flag if you need to turn behavior on and off


✗ Doing heavy or error-prone work inside a hook — it runs on EVERY audited event, and an unhandled exception there can abort the operation

✓ Keep hooks fast and defensive; raise deliberately only when you mean to block an action

Key Rules

  • sys.addaudithook(hook) registers hook(event: str, args: tuple); it fires for every audit event and cannot be removed
  • The interpreter raises built-in events for sensitive operations: "open", "import", "compile", "exec", "subprocess.Popen", "socket.connect", and many more
  • sys.audit(event, *args) raises your own event to all hooks; a hook that raises propagates and can veto the operation — the basis of sandboxing and audit-logging tools

Exercises

  1. Register a hook that records the path from every "open" event, open a file, and confirm the path was captured.
  2. Register a hook that raises on a custom "app.action" event when its argument exceeds a threshold; show a small value passes and a large one is blocked.
  3. Register a counting hook and show that compile() and __import__() each fire at least one built-in event.
Connections: audit hooks are the runtime-security layer beneath the import system, subprocess, and socket — they observe the very operations you drive with those modules; unlike sys.settrace (line-by-line tracing) they fire only on security-relevant events, making them the right tool for sandboxing and centralized audit logging.

New Concepts — Package & Bytecode Tooling (2 concepts)

Two modules that live inside the import machinery. marshal is the fast, internal serializer CPython uses for .pyc files — uniquely, it can serialize code objects, something pickle cannot do portably. pkgutil introspects packages at runtime: list a package's submodules, walk a whole package tree, and read bundled data files whether the package lives on disk or inside a zip — the foundation of plugin discovery and resource loading.

Serialization

Concept 3: marshal — Low-Level Object Serialization

Why learn this? marshal is the fast, internal serializer CPython uses to write .pyc bytecode caches. It handles the core built-in types and, uniquely, code objects — something pickle cannot do portably. It is not general-purpose or secure (the format is version-specific and unsafe on untrusted input), but it is exactly the right tool for caching compiled code and for understanding how .pyc files actually work.
Example 1 — round-trip built-in types
import marshal

data = {"n": 42, "xs": [1, 2, 3], "t": (True, None)}
blob = marshal.dumps(data)          # -> bytes
print(type(blob).__name__)          # bytes
print(marshal.loads(blob) == data)  # True
Output
bytes True
Example 2 — marshal a code object (what pickle can't)
import marshal

code = compile("x = sum(range(10))", "<gen>", "exec")
blob = marshal.dumps(code)          # code objects ARE supported
restored = marshal.loads(blob)
ns = {}
exec(restored, ns)                  # run the reloaded bytecode
print(ns["x"])                      # 45
Output
45
Example 3 — file I/O with dump/load (binary mode)
import marshal, tempfile, os

path = os.path.join(tempfile.mkdtemp(), "cache.bin")
with open(path, "wb") as f:
    marshal.dump([1, "two", 3.0], f)   # file MUST be binary

with open(path, "rb") as f:
    print(marshal.load(f))             # [1, 'two', 3.0]
Output
[1, 'two', 3.0]

Common Mistakes

✗ Using marshal for general persistence or exchanging data between Python versions — the format is undocumented and can change from release to release

✓ Use pickle for arbitrary Python objects or json for interop; reserve marshal for code objects and .pyc-style caches


✗ Loading marshal data from untrusted sources — like pickle it is unsafe and crafted input can crash or corrupt the interpreter

✓ Only marshal.load data you produced yourself; never accept it across a network or trust boundary

Key Rules

  • marshal.dumps(obj)/loads(bytes) and dump(obj, file)/load(file) (files must be binary) serialize a limited set of built-in types plus code objects
  • It is fast and version-specific: CPython uses it for .pyc bytecode caches, and the format carries no cross-version compatibility guarantee
  • Not secure and not general-purpose — prefer pickle (arbitrary objects) or json (interop); marshal's niche is serializing compiled code

Exercises

  1. Round-trip a dict of ints, lists, and tuples through marshal.dumps/loads and show it equals the original.
  2. compile() a snippet to a code object, marshal it to bytes, load it back, exec the result, and read a variable it defined.
  3. marshal.dump a list to a binary file and load it back, showing the values match.
Connections: marshal is the serializer behind .pyc files (see the import system and py_compile); it complements pickle (arbitrary objects, still unsafe) and json (safe interop) — its distinguishing feature is that it can serialize compiled code objects, which the others cannot.
Packaging

Concept 4: pkgutil — Package Introspection & Data

Why learn this? pkgutil discovers and introspects packages at runtime: list a package's immediate submodules, walk a whole package tree recursively, and read data files bundled inside a package regardless of whether it lives on disk or in a zip. It is how plugin systems auto-discover their plugins and how libraries load resources they ship alongside their code.
Example 1 — list a package's submodules (iter_modules)
import pkgutil, json    # json is a package (json/__init__.py, json/decoder.py, ...)

names = [m.name for m in pkgutil.iter_modules(json.__path__)]
print("decoder" in names)   # True
print("encoder" in names)   # True
Output
True True
Example 2 — read a bundled data file (get_data)
import pkgutil

# Reads a resource relative to the package -- works from a dir OR a zip.
raw = pkgutil.get_data("json", "__init__.py")   # -> bytes
print(isinstance(raw, bytes))     # True
print(len(raw) > 0)               # True
Output
True True
Example 3 — the plugin-discovery pattern (prefix + ispkg)
import pkgutil, json

def discover(package):
    # Map fully-qualified submodule name -> is-it-a-subpackage.
    prefix = package.__name__ + "."
    return {info.name: info.ispkg
            for info in pkgutil.iter_modules(package.__path__, prefix)}

found = discover(json)
print("json.decoder" in found)   # True
print(found.get("json.tool"))       # False -- tool is a module, not a package
Output
True False

Common Mistakes

✗ Reading a package's data file with open(__file__ + ...) — it breaks when the package is installed inside a zip/wheel with no real filesystem path

✓ Use pkgutil.get_data(package, resource) (or importlib.resources) so it works whether the package is on disk or zipped


✗ Expecting iter_modules to import submodules or recurse into subpackages

✓ iter_modules lists only immediate children without importing; use walk_packages for a recursive walk (it imports subpackages to descend)

Key Rules

  • pkgutil.iter_modules(path, prefix="") yields ModuleInfo(finder, name, ispkg) for the immediate submodules (no import); pass pkg.__path__ and a "pkg." prefix for dotted names
  • pkgutil.walk_packages(path, prefix="") recurses into subpackages (importing them to read their __path__) — the basis of plugin auto-discovery
  • pkgutil.get_data(package, resource) returns the bytes of a data file bundled in a package, working across dir and zip layouts (importlib.resources is the modern alternative)

Exercises

  1. Run pkgutil.iter_modules over a stdlib package's __path__ and show two of its submodule names appear.
  2. Read a file bundled inside a package with pkgutil.get_data and confirm you get non-empty bytes.
  3. Build a discover(package) helper using iter_modules with a name prefix that returns each submodule's fully-qualified name and its ispkg flag.
Connections: pkgutil sits on the import system alongside runpy and importlib — iter_modules/walk_packages power plugin discovery, and get_data reads bundled resources much as the marshal-backed .pyc cache is read; for new code, importlib.resources and importlib.metadata are the modern successors.

New Concepts — Functional Tools (1 concept)

One tool for generating families of related methods without repetition. functools.partialmethod is the descriptor-aware sibling of partial: it pre-fills some arguments of a method while still receiving self, so a single general method can spawn many small, readable specializations — on/off toggles, aliases, pre-configured queries.

Functional Tools

Concept 5: functools.partialmethod — Partial Methods on Classes

Why learn this? functools.partial pre-binds arguments to a function — but used as a class attribute it never receives the implicit self. partialmethod is the descriptor-aware version: it pre-fills some arguments of a method while still passing self through, so you can derive many tidy one-line methods (setters, flags, aliases) from a single general one, with no boilerplate.
Example 1 — flag setters from one method
from functools import partialmethod

class Cell:
    def __init__(self):
        self.state = "off"
    def set_state(self, state):
        self.state = state
    turn_on  = partialmethod(set_state, "on")    # pre-fills the "state" arg
    turn_off = partialmethod(set_state, "off")

c = Cell()
c.turn_on()
print(c.state)     # on
c.turn_off()
print(c.state)     # off
Output
on off
Example 2 — self is passed through correctly
from functools import partialmethod

class Greeter:
    def greet(self, greeting, name):
        return f"{greeting}, {name}!"
    hello = partialmethod(greet, "Hello")   # fixes greeting, still takes name
    hi    = partialmethod(greet, "Hi")

g = Greeter()
print(g.hello("Ada"))    # Hello, Ada!
print(g.hi("Bob"))       # Hi, Bob!
Output
Hello, Ada! Hi, Bob!
Example 3 — pre-bind keyword args (caller can override)
from functools import partialmethod

class Query:
    def fetch(self, *, limit, order):
        return f"limit={limit} order={order}"
    top10 = partialmethod(fetch, limit=10, order="desc")

q = Query()
print(q.top10())               # limit=10 order=desc
print(q.top10(order="asc"))    # limit=10 order=asc -- keyword overridden
Output
limit=10 order=desc limit=10 order=asc

Common Mistakes

✗ Using functools.partial as a class attribute to make a method — it is not a descriptor, so it never receives self and the call misbinds or fails

✓ Use functools.partialmethod in the class body; it is descriptor-aware and injects self correctly


✗ Assuming a pre-filled positional argument can be overridden at the call site — extra positionals are appended after the bound ones, not substituted

✓ Pre-fill with keyword arguments when you want callers to be able to override the value; positionals are fixed in place

Key Rules

  • partialmethod(method, *args, **kwargs) is the method-level counterpart of partial: define it in the class body to derive specialized methods that still receive self
  • Pre-bound positional args come first (before the call-site args); pre-bound keyword args can be overridden by the caller
  • It works with any descriptor callable (regular methods, other partialmethods) — ideal for families of one-liner methods (on/off, min/max, aliases)

Exercises

  1. Write a class with a set_state(self, state) method and derive turn_on/turn_off via partialmethod; toggle an instance and print its state.
  2. From a greet(self, greeting, name) method, build hello and hi partial methods that fix the greeting, and call each with a name.
  3. Pre-bind a keyword argument with partialmethod and show a caller can override it by passing that keyword explicitly.
Connections: partialmethod is the descriptor-aware sibling of functools.partial — it relies on the same descriptor protocol as @property and ordinary methods to bind self; it complements singledispatch (dispatch by argument type) as another way to generate related callables without repeating yourself.
