Review Section (6 items)

1. List, Dict & Set Comprehensions due today rep 3 → 4

A comprehension builds a collection from an iterable in one expression: [expr for x in it if cond] for lists, {k: v for ...} for dicts, {expr for ...} for sets. They are faster than an append loop and read top-to-bottom: the leading expression is what each item becomes, the for/if clauses read left-to-right exactly as nested statements would.

Key Pattern
nums = [1, 2, 3, 4, 5]

print([n * n for n in nums if n % 2])      # [1, 9, 25]  (filter then transform)
print({n: n * n for n in nums})          # {1: 1, 2: 4, 3: 9, 4: 16, 5: 25}
print({c for c in "banana"})              # {'b', 'a', 'n'}  (set: dedups)

# nested: flatten a grid (clauses read like nested for-loops)
grid = [[1, 2], [3, 4]]
print([x for row in grid for x in row])   # [1, 2, 3, 4]

# conditional EXPRESSION goes before the for (it is not a filter)
print(["even" if n % 2 == 0 else "odd" for n in nums])
Challenge

Given a list of words, build a dict mapping each word to its length but only for words longer than 3 characters. Then flatten a list-of-lists and dedup it in a single set comprehension. Finally, transpose a matrix with a nested comprehension using zip for comparison.

Pitfall

A trailing if filters ([x for x in it if cond]); an if/else before the for is a conditional expression that transforms every item. In nested comprehensions the for clauses run outer-to-inner left-to-right — the same order you would write the nested loops.

2. String Methods Deep Dive due today rep 3 → 4

Strings are immutable, so every method returns a new string. Beyond split/join/strip, know partition (split once into a 3-tuple), removeprefix/removesuffix (3.9+, safer than slicing), str.translate with a translation table, and the is* predicates (isdigit, isalpha, isidentifier).

Key Pattern
s = "  user@example.com  ".strip()

print(s.partition("@"))          # ('user', '@', 'example.com')
print(s.removesuffix(".com"))      # 'user@example' (no-op if missing)
print("a,b,,c".split(","))      # ['a', 'b', '', 'c'] (keeps empties)
print("a b  c".split())          # ['a', 'b', 'c'] (no-arg: collapse whitespace)

# translate: map/delete characters via a table
table = str.maketrans("aeiou", "-----", " ")  # 3rd arg = delete
print("hello world".translate(table))    # 'h-ll-w-rld'
Challenge

Parse "key=value" safely with partition so a missing = does not raise. Strip a known file extension with removesuffix and compare to the slicing approach. Build a str.maketrans table that ROT13-encodes lowercase letters and apply it with translate.

Pitfall

split(sep) keeps empty strings between consecutive separators, but no-arg split() collapses runs of whitespace and drops leading/trailing empties — they are not interchangeable. str.isdigit() is True for superscripts like "²"; use isdecimal() when you mean ordinary base-10 digits.

3. Unpacking & Star Expressions due today rep 3 → 4

Iterable unpacking binds names to elements: a, b = pair. A single starred target (first, *rest = seq) absorbs the remaining items into a list. In calls and literals, * splats an iterable and ** splats a mapping, so you can merge sequences and dicts inline.

Key Pattern
first, *middle, last = [1, 2, 3, 4, 5]
print(first, middle, last)        # 1 [2, 3, 4] 5  (middle is a list)

# splat in literals to merge
print([*[1, 2], *[3, 4]])     # [1, 2, 3, 4]
print({**{"a": 1}, **{"b": 2}})  # {'a': 1, 'b': 2}

# splat into a call; ** spreads kwargs
def point(x, y, z): return (x, y, z)
args = [1, 2]; kw = {"z": 3}
print(point(*args, **kw))         # (1, 2, 3)

a, b = b, a = 1, 2            # swap idiom uses tuple packing/unpacking
Challenge

Split a CSV row into head, *body, tail and confirm body is a list even when empty. Merge two config dicts with {**base, **override} so the override wins. Write a function that forwards *args, **kwargs to another and prove the values arrive intact.

Pitfall

Only one starred target is allowed per unpacking, and it always produces a list (never a tuple), even when the source is a tuple. In a {**a, **b} merge, later keys silently overwrite earlier ones — order decides the winner.

4. collections.Counter due today rep 3 → 4

Counter is a dict subclass for tallying hashable items. It counts an iterable in one call, returns 0 (not KeyError) for missing keys, and adds most_common(n), arithmetic (+ - & |), and elements() for reconstructing the multiset.

Key Pattern
from collections import Counter

c = Counter("abracadabra")
print(c.most_common(2))      # [('a', 5), ('b', 2)]
print(c["z"])               # 0  -- missing key, never KeyError

a = Counter(a=3, b=1)
b = Counter(a=1, b=2)
print(a + b)              # Counter({'a': 4, 'b': 3})
print(a - b)              # Counter({'a': 2}) -- drops zero/negative
print(a & b)              # Counter({'a': 1, 'b': 1}) -- min (intersection)
Challenge

Find the top-3 most frequent words in a paragraph with Counter(words).most_common(3). Detect whether two strings are anagrams by comparing their Counters. Use a - b to find which items in one bag are missing from another, and note how zero/negative counts vanish.

Pitfall

Subtraction with - drops zero and negative counts; use Counter.subtract() if you need negatives kept. Reading a missing key returns 0 but does not insert it — however assigning c[k] += 1 does create it, so a stray read-then-write can grow the dict unexpectedly.

5. defaultdict & deque due today rep 3 → 4

defaultdict(factory) calls factory() to create a value the first time a missing key is accessed — perfect for grouping (list) and counting (int). deque is a double-ended queue with O(1) append/appendleft/pop/popleft and an optional maxlen that auto-evicts from the far end.

Key Pattern
from collections import defaultdict, deque

groups = defaultdict(list)
for word in ["apple", "avocado", "banana"]:
    groups[word[0]].append(word)     # no need to pre-init the list
print(dict(groups))   # {'a': ['apple', 'avocado'], 'b': ['banana']}

dq = deque([1, 2, 3], maxlen=3)
dq.appendleft(0)         # deque([0, 1, 2]) -- the 3 fell off the right
dq.append(9)             # deque([1, 2, 9])
dq.rotate(1)            # deque([9, 1, 2]) -- rotate right by 1
print(dq)
Challenge

Group a list of (category, item) pairs into {category: [items]} with defaultdict(list). Build a fixed-size "last N events" log using deque(maxlen=N). Implement a BFS queue with popleft and explain why a list.pop(0) would be O(n) instead.

Pitfall

Merely reading d[missing] on a defaultdict inserts the default and grows the dict — use d.get(k) or k in d to probe without mutating. Random access dq[i] in the middle of a deque is O(n); reach for a list when you need fast indexing rather than end operations.

6. Generator Expressions due today rep 3 → 4

A generator expression looks like a list comprehension with parentheses — (x*x for x in it) — but it is lazy: it yields one value at a time and holds no full result in memory. It is single-use (exhausted after one pass) and ideal for piping huge or infinite streams into sum, any, max, or join.

Key Pattern
gen = (n * n for n in range(5))
print(next(gen), next(gen))   # 0 1  (produced on demand)

# bare gen-exp as a SOLE argument needs no extra parens
print(sum(n * n for n in range(5)))     # 30
print(any(n > 100 for n in range(5)))   # False  (short-circuits)
print("-".join(str(n) for n in range(4)))  # 0-1-2-3

g = (x for x in [1, 2])
print(list(g), list(g))   # [1, 2] []  -- exhausted after first pass
Challenge

Sum the squares of the first million integers without building a list. Use any(... for ...) to test a condition with early exit over a large range. Show that re-iterating an already-consumed generator yields nothing, and contrast memory use with the equivalent list comprehension.

Pitfall

A generator is exhausted after one full iteration — a second loop sees nothing; materialize with list(gen) if you need to reuse it. When a generator expression is the only argument to a function you may omit the inner parentheses (sum(x for x in it)), but with other arguments you must keep them.

New Concepts — Text & Encoding (3 concepts)

The line between human-readable text and on-the-wire bytes: how str and bytes differ and convert, the full grammar of the format spec that powers f-strings and format(), and the unicodedata module that tells you what a character actually is and how to normalize it.

Text & Encoding

Concept 1: Unicode, str & bytes

Why learn this? A str is a sequence of Unicode code points; bytes is a sequence of raw 8-bit values. Files, sockets, and hashing all deal in bytes, so you must encode on the way out and decode on the way in. Getting the boundary wrong is the source of nearly every UnicodeDecodeError and mojibake bug you will ever hit.
Example 1 — str vs bytes: code points vs raw octets
s = "café"          # 4 code points
b = s.encode("utf-8")   # bytes: é becomes TWO bytes in UTF-8
print(len(s), len(b))   # 4 5
print(b)               # b'caf\xc3\xa9'
print(b.decode("utf-8") == s)  # True  -- round trip
print(type(b[0]).__name__)   # int  -- indexing bytes gives an int (99)
Output
4 5 b'caf\xc3\xa9' True int
Example 2 — Code points: ord, chr and escapes
print(ord("A"), chr(65))       # 65 A
print(ord("€"))               # 8364  -- the euro sign's code point
print("€", "\N{EURO SIGN}")  # € €  -- numeric and by-name escapes
print(hex(ord("😀")))         # 0x1f600  -- emoji are single code points
print(len("😀"))               # 1  -- one code point (Python 3 is wide)
Output
65 A 8364 € € 0x1f600 1
Example 3 — encode/decode errors and the errors= handler
s = "café"
try:
    s.encode("ascii")
except UnicodeEncodeError as e:
    print("fail:", e.reason)        # fail: ordinal not in range(128)

print(s.encode("ascii", errors="replace"))  # b'caf?'
print(s.encode("ascii", errors="ignore"))   # b'caf'
print(s.encode("ascii", errors="backslashreplace"))  # b'caf\\xe9'
Output
fail: ordinal not in range(128) b'caf?' b'caf' b'caf\xe9'
Example 4 — bytes literals, hex, and the bytearray mutable cousin
raw = bytes([104, 105])      # b'hi'  -- from a list of ints 0..255
print(raw, raw.hex())       # b'hi' 6869
print(bytes.fromhex("6869"))  # b'hi'

ba = bytearray(b"hi")     # mutable bytes
ba[0] = 72                 # assign an int
print(ba)                  # bytearray(b'Hi')
Output
b'hi' 6869 b'hi' bytearray(b'Hi')

Common Mistakes

✗ Treating str and bytes as interchangeable — "a" + b"b" raises TypeError, and writing str to a binary file fails

✓ Decode bytes to str at input, encode str to bytes at output ("Unicode sandwich"); pick the codec explicitly


✗ Assuming len(s) equals the byte count — it counts code points, and one code point can be several UTF-8 bytes

✓ Use len(s.encode("utf-8")) when you need the on-the-wire size

Key Rules

  • str = Unicode code points; bytes = raw octets. Convert with .encode(codec) / .decode(codec)
  • Indexing bytes yields an int (0–255); slicing yields bytes. ord/chr map a char to/from its code point
  • errors= controls failures: "strict" (default), "replace", "ignore", "backslashreplace". Default codec for files is UTF-8 on modern Python

Exercises

  1. Write utf8_size(s) returning how many bytes a string occupies in UTF-8, and show it differs from len(s) for "naïve".
  2. Round-trip a string through encode/decode in UTF-8 and again in UTF-16; compare the byte lengths.
  3. Given bytes that may contain an invalid sequence, decode them with errors="replace" and count how many replacement characters appear.
Connections: the encoding boundary underlies the base64 guide (which turns bytes into ASCII-safe text), hashlib (which hashes bytes), and struct (which packs them); unicodedata (Concept 3) explains what those code points actually are.
Formatting

Concept 2: The Format Spec Mini-Language

Why learn this? Everything after the colon in f"{value:>10.2f}" is a tiny grammar shared by f-strings, str.format, and format(). Learn it once and you can align columns, pad with zeros, add thousands separators, show percentages, and switch number bases — without reaching for manual string juggling.
Example 1 — The grammar: [[fill]align][sign][#][0][width][,][.prec][type]
print(f"{42:5}")        # '   42'   -- width 5, right-aligned (numbers default right)
print(f"{42:<5}")       # '42   '   -- left align
print(f"{42:^5}")       # ' 42  '   -- center
print(f"{42:*>5}")      # '***42'   -- fill char '*' + right align
print(f"{42:05}")       # '00042'   -- zero-pad
Output
42 42 42 ***42 00042
Example 2 — Numbers: precision, percent, thousands, sign
print(f"{3.14159:.2f}")    # '3.14'   -- fixed 2 decimals
print(f"{0.1875:.1%}")     # '18.8%'  -- percent (x100 + %)
print(f"{1234567:,}")      # '1,234,567' -- thousands separator
print(f"{1234567:_}")      # '1_234_567' -- underscore grouping
print(f"{42:+}", f"{-42:+}")  # +42 -42  -- always show sign
print(f"{255_000:.2e}")    # '2.55e+05' -- scientific
Output
3.14 18.8% 1,234,567 1_234_567 +42 -42 2.55e+05
Example 3 — Bases, and the = debug specifier
n = 255
print(f"{n:b} {n:o} {n:x} {n:X}")   # 11111111 377 ff FF
print(f"{n:#x}")                   # 0xff  -- '#' adds the base prefix
print(f"{n:08b}")                  # 11111111  -- zero-padded to 8 bits

x = 7
print(f"{x=}")                     # x=7  -- self-documenting (3.8+)
print(f"{x*x=:.1f}")               # x*x=49.0  -- expr + spec together
Output
11111111 377 ff FF 0xff 11111111 x=7 x*x=49.0
Example 4 — Dynamic specs and nested fields
w, p = 10, 3
print(f"{3.14159:{w}.{p}f}")   # '     3.142' -- width/precision from variables

from datetime import date
print(f"{date(2026, 6, 27):%Y-%m-%d}")  # 2026-06-27 -- type passes spec to __format__

print("{0:>{1}}".format("hi", 6))  # '    hi' -- same grammar in str.format
Output
3.142 2026-06-27 hi

Common Mistakes

✗ Putting the fill character after the align (f"{x:>*5}") — fill comes first, then the align char

✓ Order is [fill][align]: f"{x:*>5}" pads with * on the left to width 5


✗ Using % formatting or manual str(round(x,2)) for aligned numeric tables

✓ Use one spec like f"{x:>12,.2f}" to align, group, and fix precision in a single step

Key Rules

  • Grammar order: [[fill]align][sign][#][0][width][grouping][.precision][type]
  • Align: < left, > right, ^ center, = pad after sign. Types: f e % d b o x, g (general), s (string)
  • {x=} shows name=value for debugging; specs may be nested with {...{var}...} and are dispatched to the object's __format__

Exercises

  1. Print a right-aligned column of prices to 2 decimals with thousands separators in a fixed width of 12.
  2. Format an integer as zero-padded 8-bit binary and as 0x-prefixed hex using the format spec only.
  3. Write a function that takes a value, a width, and a precision and formats the value using a nested dynamic spec.
Connections: the same mini-language drives str.format and format(); numeric types here echo the int/round built-ins; defining __format__ on your own class (an OOP dunder) lets it honor these specs, and datetime uses the spec slot for strftime codes.
Text & Encoding

Concept 3: unicodedata — Inspecting & Normalizing Text

Why learn this? The same visible character can be encoded multiple ways — "é" may be one code point or "e" plus a combining accent — so two strings that look identical can compare unequal. unicodedata lets you query a character's name, category, and numeric value, and normalize text so comparisons, dedup, and search behave correctly.
Example 1 — The same é, two encodings, normalize to compare
import unicodedata

a = "café"                         # NFC: é is one code point (U+00E9)
b = "café"                    # NFD: 'e' + combining acute accent
print(a == b)                       # False -- look identical, differ in bytes
print(len(a), len(b))               # 4 5

na = unicodedata.normalize("NFC", a)
nb = unicodedata.normalize("NFC", b)
print(na == nb)                      # True -- normalize before comparing
Output
False 4 5 True
Example 2 — Names, categories and lookup
import unicodedata

print(unicodedata.name("€"))        # EURO SIGN
print(unicodedata.category("A"))    # Lu  (Letter, uppercase)
print(unicodedata.category("5"))    # Nd  (Number, decimal digit)
print(unicodedata.category(" "))    # Zs  (Separator, space)
print(unicodedata.lookup("SNOWMAN"))  # ☃  (char by name)
Output
EURO SIGN Lu Nd Zs ☃
Example 3 — Numeric values of exotic digits
import unicodedata

print(unicodedata.numeric("½"))     # 0.5   -- vulgar fraction
print(unicodedata.numeric("Ⅷ"))     # 8.0   -- Roman numeral eight
print(unicodedata.digit("٣"))       # 3     -- Arabic-Indic digit
print("²".isdigit(), "²".isdecimal())  # True False  -- superscript is a digit, not decimal
Output
0.5 8.0 3 True False
Example 4 — Strip accents via NFD + category filter
import unicodedata

def strip_accents(s):
    decomposed = unicodedata.normalize("NFD", s)
    return "".join(c for c in decomposed
                   if unicodedata.category(c) != "Mn")  # Mn = combining mark

print(strip_accents("naïve café résumé"))  # naive cafe resume
print(unicodedata.normalize("NFKC", "ﬁ"))   # fi  -- NFKC expands the ligature
Output
naive cafe resume fi

Common Mistakes

✗ Comparing or deduping user-entered text without normalizing — visually identical strings can differ at the code-point level

✓ Normalize both sides to the same form (usually NFC) before ==, hashing, or using as dict keys


✗ Using NFKC/NFKD blindly — the K (compatibility) forms change meaning, turning "²" into "2" and ligatures into letters

✓ Use NFC/NFD for lossless canonical equivalence; reserve NFKC/NFKD for search/folding where you want that collapsing

Key Rules

  • Four normal forms: NFC (composed), NFD (decomposed), NFKC/NFKD (compatibility — lossy, collapses ligatures/superscripts)
  • name, lookup, category (e.g. Lu, Nd, Mn), numeric/digit/decimal query a character's properties
  • Combining marks have category Mn; filtering them after NFD is the standard accent-stripping trick

Exercises

  1. Write same_text(a, b) that returns True when two strings are canonically equivalent by normalizing both to NFC.
  2. Count how many combining marks (category Mn) appear in a decomposed (NFD) string.
  3. Build a slug function: strip accents via NFD + Mn filter, lowercase, and keep only ASCII letters/digits.
Connections: directly builds on the str/bytes code-point model (Concept 1); normalization matters anywhere strings are dict keys or set members; pairs with the re module and str.casefold() for robust, case- and accent-insensitive matching.

New Concepts — Types & Data Structures (2 concepts)

Two ways to extend Python's built-ins without surprises: attaching machine-readable metadata to a type with Annotated, and subclassing dict/list/str cleanly via the User* wrappers that actually call your overrides.

Type System

Concept 4: typing.Annotated — Types with Metadata

Why learn this? Sometimes a type alone is not enough — you want to say "an int that must be positive" or "a str that is a column name". Annotated[T, ...] attaches arbitrary metadata to a type that type checkers ignore but libraries (FastAPI, Pydantic, dataclasses) and your own code can read at runtime via typing.get_type_hints(..., include_extras=True).
Example 1 — The shape: type plus extra payload
from typing import Annotated, get_type_hints

Age = Annotated[int, "years, must be >= 0"]

def birthday(age: Age) -> int:
    return age + 1

print(birthday(29))                 # 30  -- to the checker it is just int
hints = get_type_hints(birthday, include_extras=True)
print(hints["age"])                  # typing.Annotated[int, 'years, must be >= 0']
Output
30 typing.Annotated[int, 'years, must be >= 0']
Example 2 — Reading metadata: __metadata__ and get_args
from typing import Annotated, get_args, get_origin

T = Annotated[str, "max_len=20", "trimmed"]   # multiple metadata items
print(T.__metadata__)        # ('max_len=20', 'trimmed')
print(get_origin(T))         # <class 'str'>  -- the real underlying type
print(get_args(T))           # (<class 'str'>, 'max_len=20', 'trimmed')
Output
('max_len=20', 'trimmed') <class 'str'> (<class 'str'>, 'max_len=20', 'trimmed')
Example 3 — A runtime validator driven by metadata
from typing import Annotated, get_type_hints, get_args
from dataclasses import dataclass

class Positive:
    def check(self, v): return v > 0

@dataclass
class Order:
    qty: Annotated[int, Positive()]

def validate(obj):
    for field, hint in get_type_hints(type(obj), include_extras=True).items():
        for meta in get_args(hint)[1:]:        # skip the type itself
            if hasattr(meta, "check") and not meta.check(getattr(obj, field)):
                raise ValueError(f"{field} failed {meta.__class__.__name__}")

validate(Order(5))               # ok
try:
    validate(Order(-1))
except ValueError as e:
    print(e)                    # qty failed Positive
Output
qty failed Positive
Example 4 — Annotated is transparent to the type checker
from typing import Annotated

UserId = Annotated[int, "db primary key"]
def fetch(uid: UserId) -> str:
    return f"user-{uid}"

print(fetch(42))            # user-42  -- behaves exactly like int
# A checker treats UserId as int: fetch("x") would be flagged,
# while the "db primary key" note is available only at runtime.
print(UserId.__origin__.__name__)  # int
Output
user-42 int

Common Mistakes

✗ Calling plain get_type_hints(obj) and wondering where the metadata went — it strips Annotated by default

✓ Pass include_extras=True to keep the metadata, then read it via get_args(hint)[1:] or hint.__metadata__


✗ Expecting Annotated to enforce anything — type checkers ignore the metadata entirely; nothing validates on its own

✓ Treat the metadata as inert data; a library or your own validator must read and act on it

Key Rules

  • Annotated[T, m1, m2, ...] — first arg is the real type, the rest is arbitrary runtime metadata (3.9+)
  • Checkers see only T; the metadata is invisible to them and never validates automatically
  • Read it with get_type_hints(..., include_extras=True), then get_origin (the type) and get_args / __metadata__ (the extras)

Exercises

  1. Define Annotated[float, "celsius"] and Annotated[float, "fahrenheit"] and write a function that inspects the unit metadata at runtime.
  2. Build a MaxLen(n) metadata class and a validator that checks any Annotated[str, MaxLen(...)] field on a dataclass.
  3. Show that get_type_hints without include_extras=True returns the bare type, while with it you get the full Annotated.
Connections: extends the basic type-hints and Optional/Union guides; the runtime introspection mirrors inspect and powers Pydantic/FastAPI dependency injection; pairs naturally with dataclasses fields and with Literal/NewType for richer domain types.
Data Structures

Concept 5: UserDict, UserList & UserString

Why learn this? Subclassing dict/list/str directly is a trap: the C implementations do not route internal calls through your overridden methods, so a custom __setitem__ is silently bypassed by update or the constructor. The collections.User* wrappers store the real data in a .data attribute and are pure-Python, so every operation honors your overrides.
Example 1 — Why subclassing dict directly surprises you
class LowerDict(dict):
    def __setitem__(self, k, v):
        super().__setitem__(k.lower(), v)

d = LowerDict()
d["AbC"] = 1            # goes through __setitem__
print(dict(d))             # {'abc': 1}  -- good

d2 = LowerDict(XyZ=9)       # constructor BYPASSES __setitem__
d2.update(MoRe=2)          # update also bypasses it
print(dict(d2))            # {'XyZ': 9, 'MoRe': 2}  -- NOT lowercased!
Output
{'abc': 1} {'XyZ': 9, 'MoRe': 2}
Example 2 — UserDict routes everything through your override
from collections import UserDict

class LowerDict(UserDict):
    def __setitem__(self, k, v):
        super().__setitem__(k.lower(), v)

d = LowerDict(XyZ=9)         # constructor uses __setitem__ now
d.update(MoRe=2)
d["AbC"] = 1
print(d.data)             # {'xyz': 9, 'more': 2, 'abc': 1} -- all lowercased
Output
{'xyz': 9, 'more': 2, 'abc': 1}
Example 3 — UserList with a validating append
from collections import UserList

class PositiveList(UserList):
    def append(self, item):
        if item <= 0:
            raise ValueError("must be positive")
        super().append(item)

pl = PositiveList([1, 2])
pl.append(3)
print(pl.data)            # [1, 2, 3]
try:
    pl.append(-5)
except ValueError as e:
    print(e)                # must be positive
Output
[1, 2, 3] must be positive
Example 4 — UserString to add a method to str-like values
from collections import UserString

class Shout(UserString):
    def excited(self):
        return self.data.upper() + "!"

s = Shout("hi")
print(s.excited())       # HI!
print(s + " there")        # hi there  -- inherited str behavior
print(len(s), s.upper())   # 2 HI  -- normal string ops still work
Output

Common Mistakes

✗ Subclassing built-in dict/list/str and expecting overridden methods to be used internally — the C constructor and bulk methods bypass them

✓ Subclass UserDict/UserList/UserString when you must override core behavior so every path goes through your code


✗ Forgetting the real storage lives in .data — iterating or copying the wrapper itself instead of inspecting self.data

✓ Access the backing store via self.data; for read-only mixins, collections.abc (Mapping/Sequence) is often a lighter alternative

Key Rules

  • UserDict/UserList/UserString wrap real data in self.data and are pure-Python, so overrides are always honored
  • Prefer them over subclassing the built-in when you need to customize __setitem__, append, etc. and want construction/bulk ops to respect it
  • For interface-only contracts (no storage tweak), collections.abc.MutableMapping/MutableSequence mixins are often cleaner

Exercises

  1. Build a case-insensitive UserDict where d["KEY"] and d["key"] hit the same entry, and confirm the constructor honors it.
  2. Write a UserList subclass that keeps itself sorted on every append using bisect.insort.
  3. Demonstrate the bypass bug: subclass plain dict with a logging __setitem__ and show the constructor does not log, then fix it with UserDict.
Connections: solves the dunder-method bypass that the OOP __getitem__/__setitem__ guides hint at; contrasts with collections.abc ABCs and defaultdict/OrderedDict (which extend dict the C way for performance); bisect pairs well with a sorted UserList.
