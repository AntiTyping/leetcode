Range-based Iteration Iteration 3 days overdue · rep 2
Iterate over collections with index and value access. Python: for i, v in enumerate(lst). Go: for i, v := range slice. TypeScript: arr.forEach((v, i) => or for...of with entries(). Rust: .iter().enumerate(). C++: range-based for (auto& x : vec). Key insight: each language provides idiomatic ways to access both index and value; avoid C-style indexed loops when the language offers a better construct.
Python
fruits = ["apple", "banana", "cherry"]

# enumerate gives (index, value) tuples
for i, fruit in enumerate(fruits):
    print(f"{i}: {fruit}")

# Dict iteration
scores = {"Alice": 95, "Bob": 87, "Carol": 92}
for name, score in scores.items():
    print(f"{name} scored {score}")
0: apple 1: banana 2: cherry Alice scored 95 Bob scored 87 Carol scored 92

Pitfalls to Remember:
  • Python: Modifying a list while iterating over it causes skipped elements or IndexError. Iterate over a copy or build a new list.
  • Go: The loop variable v in for i, v := range slice is a copy, not a reference. Modifying v does not change the original slice element.
  • TypeScript: for...in iterates over keys (as strings), not values. Always use for...of for arrays.
  • Rust: .iter() borrows, .into_iter() consumes. Using into_iter() moves the collection — you cannot use it afterward.
  • C++: Range-based for does not provide an index. You must maintain a separate counter or use C++20 views::enumerate (limited compiler support).
Higher-Order Functions Functions 3 days overdue · rep 2
Functions that take other functions as arguments or return functions. map, filter, reduce are the classic trio. Python: map(fn, lst), filter(fn, lst), functools.reduce(fn, lst). Go: pass func parameters. TypeScript: .map(), .filter(), .reduce(). Rust: .iter().map(|x|).filter(|x|).fold(init, |acc, x|). C++: std::transform, std::copy_if, std::accumulate with lambdas.
Python
from functools import reduce

nums = [1, 2, 3, 4, 5, 6, 7, 8]

squared = list(map(lambda x: x ** 2, nums))
evens = list(filter(lambda x: x % 2 == 0, nums))
total = reduce(lambda acc, x: acc + x, nums, 0)

print(f"Squared: {squared}")
print(f"Evens:   {evens}")
print(f"Sum:     {total}")
Squared: [1, 4, 9, 16, 25, 36, 49, 64] Evens: [2, 4, 6, 8] Sum: 36
Go
package main

import "fmt"

func mapInts(nums []int, f func(int) int) []int {
    result := make([]int, len(nums))
    for i, n := range nums {
        result[i] = f(n)
    }
    return result
}

func filterInts(nums []int, pred func(int) bool) []int {
    var result []int
    for _, n := range nums {
        if pred(n) { result = append(result, n) }
    }
    return result
}

func main() {
    nums := []int{1, 2, 3, 4, 5, 6, 7, 8}
    squared := mapInts(nums, func(x int) int { return x * x })
    evens := filterInts(nums, func(x int) bool { return x%2 == 0 })
    fmt.Println("Squared:", squared)
    fmt.Println("Evens:  ", evens)
}
Squared: [1 4 9 16 25 36 49 64] Evens: [2 4 6 8]

Interfaces & Traits Abstractions 3 days overdue · rep 2
Define behavior contracts without specifying implementation. Go: implicit interfaces (satisfy by implementing methods). Rust: trait with impl Trait for Type (explicit). TypeScript: interface with structural typing. Python: Protocol for structural subtyping (duck typing). C++: concepts (C++20) and abstract base classes. Key insight: Go and TS use structural typing (shape matters), Rust uses nominal (must explicitly implement), Python straddles both with Protocol + ABC.
Python
from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> str: ...

class Circle:
    def __init__(self, radius: float):
        self.radius = radius
    def draw(self) -> str:
        return f"Circle(r={self.radius})"

class Square:
    def __init__(self, side: float):
        self.side = side
    def draw(self) -> str:
        return f"Square(s={self.side})"

def render(shapes: list[Drawable]) -> None:
    for s in shapes:
        print(f"  Drawing: {s.draw()}")

render([Circle(5), Square(3)])
Drawing: Circle(r=5) Drawing: Square(s=3)

Pitfalls to Remember:
  • Python: Protocol only works with type checkers (mypy). At runtime, Python uses duck typing — any object with the right methods works, even without Protocol.
  • Go: Interfaces are satisfied implicitly. A typo in a method name silently fails to satisfy the interface — caught only when you try to assign/pass.
  • TypeScript: implements is purely a compile-time check. At runtime, there are no interfaces — any object with matching shape works.
  • Rust: Trait objects (dyn Trait) require Box or &dyn for heap/reference dispatch. Not all traits are object-safe (e.g., traits with generic methods).
  • C++: Forgetting virtual on a destructor in a base class causes undefined behavior when deleting through a base pointer. Always use virtual ~Base() = default.
FFI & Interop Advanced Patterns 3 days overdue · rep 0
Foreign Function Interface — calling code written in other languages. Python: ctypes for C libraries, cffi for cleaner bindings, pybind11 for C++. Go: cgo with // #include directives, C.function_name(). TypeScript/Node: node-ffi-napi, WebAssembly for browser, N-API for native addons. Rust: extern "C" blocks, #[no_mangle] for exporting, bindgen for auto-generating bindings. C++: extern "C" to disable name mangling, dlopen/LoadLibrary for dynamic loading.
Python
import ctypes

# Load the C standard library
libc = ctypes.CDLL("libc.dylib")  # macOS; use "libc.so.6" on Linux

# Call C's strlen
libc.strlen.argtypes = [ctypes.c_char_p]
libc.strlen.restype = ctypes.c_size_t
length = libc.strlen(b"Hello, FFI!")
print(f"strlen result: {length}")

# Call C's abs
libc.abs.argtypes = [ctypes.c_int]
libc.abs.restype = ctypes.c_int
print(f"abs(-42): {libc.abs(-42)}")
strlen result: 10 abs(-42): 42

Pitfalls to Remember:
  • Python: ctypes does not validate argument types at runtime. Passing wrong types causes segfaults, not exceptions. Always set argtypes.

Python: asyncio Internals & the GIL Language Deep Dive due today · rep 1
Python’s asyncio runs a single-threaded event loop. Coroutines (async def) are cooperative: they yield control at await points. The GIL (Global Interpreter Lock) prevents true parallel Python execution across threads, making asyncio ideal for I/O-bound work but useless for CPU-bound parallelism. Use multiprocessing or ProcessPoolExecutor for CPU-bound work. Key functions: asyncio.gather() for concurrent I/O, asyncio.create_task() for fire-and-forget, asyncio.Queue for producer-consumer patterns.
Python
import asyncio

async def fetch(name: str, delay: float) -> str:
    print(f"  [{name}] starting...")
    await asyncio.sleep(delay)  # yields control to event loop
    print(f"  [{name}] done after {delay}s")
    return f"{name}-result"

async def main():
    # Concurrent I/O with gather
    results = await asyncio.gather(
        fetch("A", 0.3),
        fetch("B", 0.1),
        fetch("C", 0.2),
    )
    print(f"Results: {results}")

asyncio.run(main())
[A] starting... [B] starting... [C] starting... [B] done after 0.1s [C] done after 0.2s [A] done after 0.3s Results: ['A-result', 'B-result', 'C-result']
Go
package main

import (
    "fmt"
    "sync"
    "time"
)

// Go comparison: goroutines + WaitGroup instead of asyncio.gather
func fetch(name string, delay time.Duration, wg *sync.WaitGroup, results chan<- string) {
    defer wg.Done()
    fmt.Printf("  [%s] starting...\n", name)
    time.Sleep(delay)
    fmt.Printf("  [%s] done after %v\n", name, delay)
    results <- name + "-result"
}

func main() {
    var wg sync.WaitGroup
    results := make(chan string, 3)

    tasks := []struct{ name string; delay time.Duration }{
        {"A", 300 * time.Millisecond},
        {"B", 100 * time.Millisecond},
        {"C", 200 * time.Millisecond},
    }
    for _, t := range tasks {
        wg.Add(1)
        go fetch(t.name, t.delay, &wg, results)
    }
    wg.Wait()
    close(results)
    for r := range results { fmt.Println(r) }
}

Go: context Package & reflect Language Deep Dive due today · rep 1
Go’s context package carries deadlines, cancellation signals, and request-scoped values across API boundaries. context.WithCancel: manual cancellation. context.WithTimeout: auto-cancel after duration. context.WithValue: attach metadata (use sparingly). The reflect package enables runtime type inspection: reflect.TypeOf() returns the type, reflect.ValueOf() returns the value. Struct tags (`json:"name"`) are accessed via field.Tag.Get("json"). Key insight: context is Go’s solution to cancellation propagation; reflect is powerful but slow and untyped.
Go
package main

import (
    "context"
    "fmt"
    "reflect"
    "time"
)

type Config struct {
    Host    string `json:"host" env:"APP_HOST"`
    Port    int    `json:"port" env:"APP_PORT"`
    Debug   bool   `json:"debug" env:"APP_DEBUG"`
}

func worker(ctx context.Context, id int) {
    select {
    case <-time.After(2 * time.Second):
        fmt.Printf("  Worker %d: completed\n", id)
    case <-ctx.Done():
        fmt.Printf("  Worker %d: cancelled (%v)\n", id, ctx.Err())
    }
}

func inspectStruct(v interface{}) {
    t := reflect.TypeOf(v)
    for i := 0; i < t.NumField(); i++ {
        f := t.Field(i)
        fmt.Printf("  %s (%s) json=%q env=%q\n",
            f.Name, f.Type, f.Tag.Get("json"), f.Tag.Get("env"))
    }
}

func main() {
    // Context with timeout
    ctx, cancel := context.WithTimeout(context.Background(), 500*time.Millisecond)
    defer cancel()
    go worker(ctx, 1)
    go worker(ctx, 2)
    time.Sleep(700 * time.Millisecond)

    // Reflect on struct tags
    fmt.Println("Config fields:")
    inspectStruct(Config{})
}

Rust: Async Runtimes & tokio Language Deep Dive

Rust’s async/await is zero-cost but requires an external runtime to drive futures. tokio is the de facto standard, providing task scheduling, I/O, timers, and channels. Understanding tokio is essential for any production Rust networking or concurrent code.

How Rust Async Works

Unlike Python or JavaScript, Rust has no built-in async runtime. async fn returns a Future that is inert until polled. A runtime like tokio provides the executor that polls futures, an I/O reactor for non-blocking I/O, and a timer for sleep/timeout.

Key tokio Features

#[tokio::main] sets up the multi-threaded runtime. tokio::spawn creates a new task (like Go goroutines). tokio::select! races multiple futures. Channels (mpsc, oneshot, broadcast) enable message passing between tasks.

use tokio::sync::mpsc;
use tokio::time::{sleep, Duration};

        }

Language Comparison: Rust requires an explicit runtime (tokio, async-std) because async is zero-cost: no hidden allocations or thread pools unless you opt in. Python’s asyncio bundles the event loop. Go’s runtime is built-in (goroutine scheduler). JavaScript’s event loop is provided by V8/Node. Common Mistakes per Language:
  • Rust: Holding a MutexGuard across an .await causes deadlocks and compilation errors with tokio’s Send requirement. Use tokio::sync::Mutex instead of std::sync::Mutex in async code.
  • Rust: tokio::spawn requires the future to be Send + 'static. Borrowing local variables across spawn boundaries requires Arc.
  • Python: asyncio.create_task() must be called from a running event loop. Calling it outside async def raises RuntimeError.
  • Go: Goroutines are preemptive; tokio tasks are cooperative. A CPU-bound tokio task blocks other tasks on the same thread — use tokio::task::spawn_blocking.
  • C++: std::async with std::launch::async creates a thread per call. High task counts exhaust OS threads. Use a thread pool for production workloads.
tokio Essentials:
  • #[tokio::main] sets up a multi-threaded runtime; use #[tokio::main(flavor = "current_thread")] for single-threaded.
  • tokio::spawn returns a JoinHandle — .await it to get the result, or drop it for fire-and-forget.
  • tokio::select! picks the first future to complete and drops the rest. The dropped futures are cancelled, not leaked.
  • Use tokio::sync::mpsc for multi-producer channels, oneshot for single-response patterns, broadcast for pub-sub.
  • Always drop senders when done to signal channel closure. Receivers will return None once all senders are dropped.

Rust: Macros — Declarative & Procedural Language Deep Dive

Macros are Rust’s metaprogramming system: they generate code at compile time, eliminating boilerplate while maintaining type safety. From vec![] to #[derive(Debug)], macros are everywhere in Rust. Understanding them unlocks the ability to read most Rust library code and write DRY, expressive APIs.

Two Kinds of Macros

Declarative macros (macro_rules!) use pattern matching on token trees. They are simpler to write and handle most code generation needs. Procedural macros operate on the AST as Rust code, enabling derive macros (#[derive(Debug)]), attribute macros (#[tokio::main]), and function-like macros.

How They Compare

Python uses decorators for similar goals (modifying functions/classes). Go uses go generate for code generation (external tooling, not language-level). TypeScript has experimental decorators. C++ uses preprocessor macros (#define) and template metaprogramming. Rust macros are uniquely powerful: they are hygienic (no name collisions), type-aware, and produce compile errors with good messages.

    ($($key:expr => $val:expr),* $(,)?) => {{

    }};
}

C++: SFINAE & Template Metaprogramming Language Deep Dive

SFINAE (Substitution Failure Is Not An Error) is the foundation of compile-time type introspection in C++. It enables writing functions that behave differently based on type properties — without runtime overhead. C++17’s if constexpr simplifies many SFINAE use cases, and C++20 concepts supersede most of them, but understanding SFINAE is essential for reading existing C++ codebases.

How SFINAE Works

When the compiler substitutes template arguments and a substitution fails (e.g., accessing a member that does not exist), instead of a hard error, that template overload is silently removed from the candidate set. This allows writing multiple template overloads that each handle different type categories.

Evolution: SFINAE → enable_if → if constexpr → Concepts

Classic SFINAE is verbose. std::enable_if (C++11) wraps SFINAE in a utility. if constexpr (C++17) enables compile-time branching inside a single function. C++20 Concepts provide clean, readable constraints. Each generation is simpler than the last.

C++: Coroutines (C++20) Language Deep Dive

C++20 coroutines bring first-class support for suspendable functions. They enable lazy generators (co_yield), async I/O (co_await), and cooperative multitasking patterns. Unlike Python or JavaScript coroutines, C++ coroutines are a low-level framework — you build the high-level abstractions yourself, giving maximum control and zero-cost overhead.

Three Coroutine Keywords

co_await suspends and waits for a result (like Python’s await). co_yield suspends and produces a value (like Python’s yield). co_return completes the coroutine (like return in an async function). Any function using these keywords becomes a coroutine.

The Coroutine Framework

C++ coroutines require a promise type (controls coroutine behavior), a coroutine handle (resumes/destroys the coroutine), and a return object (the type the caller interacts with). This is lower-level than any other language but enables zero-overhead abstractions.

C++
#include <iostream>
#include <coroutine>
#include <optional>

// A simple Generator type for co_yield
template<typename T>
struct Generator {
    struct promise_type {
        T current_value;
        auto get_return_object() {
            return Generator{std::coroutine_handle<promise_type>::from_promise(*this)};
        }
        std::suspend_always initial_suspend() { return {}; }
        std::suspend_always final_suspend() noexcept { return {}; }
        std::suspend_always yield_value(T value) {
            current_value = value;
            return {};
        }
        void return_void() {}
        void unhandled_exception() { std::terminate(); }
    };

    std::coroutine_handle<promise_type> handle;

    Generator(std::coroutine_handle<promise_type> h) : handle(h) {}
    ~Generator() { if (handle) handle.destroy(); }

    // Move-only
    Generator(const Generator&) = delete;
    Generator(Generator&& other) : handle(other.handle) { other.handle = nullptr; }

    std::optional<T> next() {
        if (!handle || handle.done()) return std::nullopt;
        handle.resume();
        if (handle.done()) return std::nullopt;
        return handle.promise().current_value;
    }
};

// Generator coroutine: Fibonacci sequence
Generator<int> fibonacci() {
    int a = 0, b = 1;
    while (true) {
        co_yield a;
        auto next = a + b;
        a = b;
        b = next;
    }
}

// Generator coroutine: range
Generator<int> range(int start, int end) {
    for (int i = start; i < end; i++) {
        co_yield i;
    }
}

    // Fibonacci: take first 10
    auto fib = fibonacci();
    std::cout << "Fibonacci:";

Concurrent Data Structures Language Deep Dive

When multiple threads or tasks access shared data, you need thread-safe data structures. Every language offers different primitives: mutexes, atomic operations, channels, lock-free structures. Choosing the right one is the difference between correct concurrent code and subtle data races or deadlocks.

Lock-based vs Lock-free

Lock-based: Mutexes guard data, ensuring exclusive access. Simple to reason about but can cause deadlocks and contention. Lock-free: Atomic operations (CAS — Compare-And-Swap) allow progress without locks. Higher throughput under contention but much harder to implement correctly. Most applications should start with locks and only move to lock-free when profiling shows contention.

Channel-based Concurrency

Go’s philosophy: “Don’t communicate by sharing memory; share memory by communicating.” Channels decouple producers from consumers, eliminating shared mutable state. Rust’s ownership system enforces this at compile time. Python’s GIL makes thread-safe collections simpler but limits true parallelism.

// Thread-safe queue
template<typename T>
class SafeQueue {
    std::queue<T> q;
    mutable std::mutex mtx;
    std::condition_variable cv;
    bool done = false;
public:
    void push(T item) {
        { std::lock_guard lock(mtx); q.push(std::move(item)); }
        cv.notify_one();
    }
    bool pop(T& item) {
        std::unique_lock lock(mtx);
        cv.wait(lock, [&]{ return !q.empty() || done; });
        if (q.empty()) return false;
        item = std::move(q.front()); q.pop();
        return true;
    }
    void finish() { { std::lock_guard lock(mtx); done = true; } cv.notify_all(); }
};

class AsyncQueue<T> {
    private items: T[] = [];
    private waiters: Array<(value: T) => void> = [];

    push(item: T): void {
        if (this.waiters.length > 0) {
            this.waiters.shift()!(item);
        } else {
            this.items.push(item);
        }
    }

    async pop(): Promise<T> {
        if (this.items.length > 0) {
            return this.items.shift()!;
        }
        return new Promise<T>(resolve => this.waiters.push(resolve));
    }
}

Language Comparison: Rust enforces data-race freedom at compile time: you cannot share Mutex<T> across threads without Arc, and you cannot access the data without locking. C++ has the tools (std::mutex, std::atomic) but no compiler enforcement — data races are undefined behavior. Common Mistakes per Language:
  • Python: The GIL makes individual operations atomic, but not compound operations. counter += 1 is NOT atomic — it is load + add + store. Always use Lock for compound operations.
  • Go: sync.Map is optimized for read-heavy workloads. For write-heavy access, a regular map with sync.RWMutex is faster. Do not use sync.Map as a default.
  • Rust: Mutex::lock() returns a MutexGuard that holds the lock until dropped. Holding the guard across a long operation starves other threads. Drop the guard as soon as possible.
  • C++: Locking order violations cause deadlocks. If thread A locks mutex1 then mutex2, and thread B locks mutex2 then mutex1, deadlock occurs. Use std::scoped_lock(m1, m2) to lock multiple mutexes safely.
  • TypeScript: SharedArrayBuffer requires Cross-Origin-Isolation headers in browsers. Without them, the constructor throws. Node.js does not have this restriction.
Concurrent Data Structure Rules:
  • Start with channels/message-passing. Only use shared memory when channels create a bottleneck.
  • Use std::atomic / AtomicU64 / atomic.AddInt64 for simple counters and flags. Do not use mutexes for single-value updates.
  • Use read-write locks (shared_mutex, RwLock, sync.RWMutex) when reads vastly outnumber writes.
  • Never hold a lock while performing I/O or sleeping. This is the #1 cause of performance degradation in concurrent code.
  • Prefer Arc<Mutex<T>> (Rust) / std::shared_ptr (C++) for shared ownership of locked data. The ownership model prevents use-after-free.
