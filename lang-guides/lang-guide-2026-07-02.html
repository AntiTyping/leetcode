1. Concurrency Primitives Concurrency rep 3 • 15 days since review
4th review (due today)
Quick Recall: The toolbox is the same everywhere — a mutex for mutual exclusion, a condition variable to wait for a predicate, a semaphore to cap concurrency, atomics for lock-free counters, and a channel/queue to hand a value from one thread to another. The philosophies split: Go says "share memory by communicating" (channels), Rust encodes thread-safety in the type system (Send/Sync), Python has real threads but a GIL, C++ gives raw primitives with UB on misuse, and JS has no shared-memory threads at all outside Workers.

Key Pattern — hand the value 42 from a spawned thread to the main thread

Go 
ch := make(chan int, 1)              // buffered handoff channel
go func() { ch <- 42 }()             // producer goroutine
fmt.Println(<-ch)                     // 42  (receive blocks until sent)
Python 
import queue, threading
q = queue.Queue()                     # thread-safe FIFO
threading.Thread(target=lambda: q.put(42)).start()
print(q.get())                        # 42  (blocks until an item arrives)
Rust 
use std::sync::mpsc;
use std::thread;
let (tx, rx) = mpsc::channel();
thread::spawn(move || tx.send(42).unwrap());
println!("{}", rx.recv().unwrap());   // 42
C++ 
#include <future><thread>
std::promise<int> p; auto f = p.get_future();
std::thread t([&]{ p.set_value(42); });
std::cout << f.get();                  // 42
t.join();
TypeScript 
// No OS threads in normal code; a Promise is the async "handoff":
const p = new Promise<number>(res => setTimeout(() => res(42), 0));
console.log(await p);                 // 42
// Cross-Worker handoff uses postMessage / MessageChannel instead.
Challenge (all 5 languages):
1
Implement a bounded semaphore that limits concurrent workers to 2 out of 5 launched, and prove (with timestamps) that at most two run at once. Use the idiomatic primitive per language: a buffered channel as a token bucket in Go, threading.Semaphore in Python, tokio::sync::Semaphore or a counting approach in Rust, std::counting_semaphore in C++, and a promise-pool in TS.
#1 Pitfall per Language:
  • Go: an unbuffered channel send blocks until someone receives; a nil channel blocks forever, and sending on a closed channel panics.
  • Python: threads don't run CPU-bound work in parallel (the GIL serializes bytecode) — reach for multiprocessing or native extensions for CPU work.
  • Rust: if every Sender is dropped, recv() returns Err; a Mutex can be poisoned if a holder panics.
  • C++: a std::promise/future is single-use; forgetting to join()/detach() a thread calls std::terminate; guard condition_variable waits with a predicate for spurious wakeups.
  • TypeScript: no shared-memory concurrency outside Workers — but await yields the event loop, so state can change across an await boundary.
2. Generic Programming Functions & Abstractions rep 3 • 15 days since review
4th review (due today)
Quick Recall: Generics = write an algorithm once, parameterized over the type, with a constraint that says what operations the type must support. The constraint mechanism differs: Go uses interface constraints (cmp.Ordered), Rust uses trait bounds (T: PartialOrd), C++20 uses concepts (std::totally_ordered), TS uses extends bounds, and Python's are type-hints only (erased at runtime). Rust and C++ monomorphize (specialized code per type); Go, TS, and Python largely erase.

Key Pattern — a generic max_of(a, b) over any orderable type

Go 
import "cmp"
func MaxOf[T cmp.Ordered](a, b T) T { if a > b { return a }; return b }
fmt.Println(MaxOf(3, 7), MaxOf("a", "z"))   // 7 z
Rust 
fn max_of<T: PartialOrd>(a: T, b: T) -> T { if a > b { a } else { b } }
println!("{} {}", max_of(3, 7), max_of("a", "z"));  // 7 z
C++ 
#include <concepts>
template <std::totally_ordered T>          // C++20 concept as a constraint
T max_of(T a, T b) { return a > b ? a : b; }
std::cout << max_of(3, 7) << " " << max_of('a', 'z'); // 7 z
TypeScript 
// `>` isn't allowed on a bare T; constrain to comparable primitives:
function maxOf<T extends number | string>(a: T, b: T): T { return a > b ? a : b; }
console.log(maxOf(3, 7), maxOf("a", "z")); // 7 z
Python 
# PEP 695 syntax (3.12+); hints are erased at runtime, duck-typed:
def max_of[T](a: T, b: T) -> T: return a if a > b else b
print(max_of(3, 7), max_of("a", "z"))      # 7 z
Challenge (all 5 languages):
1
Write a generic map_slice(xs, f) that turns a []A into a []B via f: A -> B. Then add a second constrained generic sum(xs) that requires a numeric/addable bound. Note where each language forces you to name the bound explicitly and where inference fills it in.
#1 Pitfall per Language:
  • Go: methods can't introduce their own type parameters, and constraints are interface-based — some abstractions (e.g. a generic map over method sets) simply can't be expressed.
  • Rust: monomorphization can bloat the binary; every operation you use must be named in a trait bound or it won't compile.
  • C++: pre-concepts template errors are famously deep and late; heavy template use balloons compile time and code size.
  • TypeScript: generics are erased at runtime — you can't branch on T or instanceof T; distributive conditional types surprise newcomers.
  • Python: type parameters are documentation only — nothing enforces them at runtime; use bound=/Protocol to express constraints for the checker.
3. CSV & Data Processing I/O & Data Processing rep 3 • 15 days since review
4th review (due today)
Quick Recall: The one rule: never parse CSV by splitting on commas. Real CSV has quoted fields containing commas, embedded newlines, and doubled "" escapes — only a proper parser handles them. Python and Go ship robust CSV in the stdlib; Rust has the excellent csv crate; TypeScript/JS and C++ have no stdlib CSV and need a library. Stream row-by-row for large files rather than loading everything.

Key Pattern — parse name,age\nAda,36 into structured records

Python 
import csv, io
rows = list(csv.DictReader(io.StringIO("name,age\nAda,36")))
print(rows)                          # [{'name': 'Ada', 'age': '36'}]
Go 
import ("encoding/csv"; "strings")
r := csv.NewReader(strings.NewReader("name,age\nAda,36"))
recs, _ := r.ReadAll()
fmt.Println(recs)                    // [[name age] [Ada 36]]
Rust 
// csv crate: reader is header-aware by default
let mut rdr = csv::Reader::from_reader("name,age\nAda,36".as_bytes());
for rec in rdr.records() { println!("{:?}", rec.unwrap()); } // StringRecord(["Ada", "36"])
TypeScript 
// No stdlib CSV; use csv-parse (or papaparse):
import { parse } from "csv-parse/sync";
const rows = parse("name,age\nAda,36", { columns: true });
console.log(rows);                   // [ { name: 'Ada', age: '36' } ]
C++ 
// No stdlib CSV. A naive getline+split MISHANDLES quotes:
std::string line = "Ada,36";
std::stringstream ss(line); std::string cell;
while (std::getline(ss, cell, ',')) std::cout << cell << " "; // Ada 36
// Use a library (e.g. fast-cpp-csv-parser) for real quoting/escaping.
Challenge (all 5 languages):
1
Parse a CSV whose values include a quoted field with an embedded comma and a doubled-quote escape (e.g. "Smith, Ada",36,"6"" tall") and confirm you get exactly three fields. Then stream a 1M-row file and compute a per-column sum without loading it all into memory.
#1 Pitfall per Language:
  • Python: when reading from a file, open with newline="" or embedded newlines inside quoted fields (and Windows line endings) get mangled.
  • Go: ReadAll loads everything into memory — use Read() in a loop for big files; the reader enforces a consistent field count unless you set FieldsPerRecord = -1.
  • Rust: the csv crate errors on ragged rows by default — set .flexible(true) if column counts vary.
  • TypeScript: line.split(",") silently breaks on any quoted comma or newline — always use a real parser.
  • C++: no standard parser exists; hand-rolled getline+split ignores quoting and embedded newlines — pull in a library for anything real.
4. Memory Management Deep Dive Advanced Patterns rep 1 • 20 days since review
2nd review (overdue 14 days)
Quick Recall: Three reclamation models. Manual/scope-based: C++ RAII frees at end of scope, Rust's ownership drops the value when it goes out of scope. Reference counting: Python frees immediately when the count hits zero (plus a cyclic collector for cycles). Tracing GC: Go and JS periodically find and free unreachable objects. The deterministic camps (C++/Rust) give you exact timing; the GC camps trade that for no dangling pointers.

Key Pattern — a resource whose cleanup runs deterministically at scope end

C++ 
{
    std::vector<int> v(1000);         // heap buffer owned by v
    auto p = std::make_unique<int>(5); // single heap int
}                                     // both freed HERE, deterministically (RAII)
Rust 
struct Res;
impl Drop for Res { fn drop(&mut self) { println!("freed"); } }
{ let _r = Res; }                     // "freed" prints at the closing brace
Python 
class Res:
    def __del__(self): print("freed")
r = Res(); del r                      # refcount hits 0 -> "freed" now
with open("/dev/null") as f: pass     # `with` = deterministic resource release
Go 
f, _ := os.Open("/dev/null")
defer f.Close()                       // defer handles RESOURCES; memory is GC'd
// You never free memory yourself; the GC reclaims unreachable objects.
TypeScript 
// GC-managed; no deterministic finalization. TS 5.2 `using` frees resources:
{
  using file = openSync();            // Symbol.dispose called at block end
}                                     // deterministic for RESOURCES, not memory
Challenge (all 5 languages):
1
Build a doubly-linked node structure that creates a reference cycle, then show which languages leak it: Python's cyclic GC eventually collects it, Rust needs Weak to break the Rc cycle, C++ needs weak_ptr to break the shared_ptr cycle, while Go/JS tracing GC reclaim it automatically. Instrument a destructor/finalizer to observe when (or whether) cleanup fires.
#1 Pitfall per Language:
  • C++: manual new/delete invites double-free, use-after-free, and leaks — prefer RAII and smart pointers; follow the rule of 0/3/5.
  • Rust: ownership is leak-free for trees but not for cycles — Rc<RefCell<T>> cycles leak; break them with Weak.
  • Python: plain refcounting can't free cycles; the cyclic GC handles them, but a __del__ on a cyclic object can delay or complicate collection.
  • Go: the GC only frees unreachable memory — leaked goroutines, unbounded maps/slices, and lingering references keep memory alive forever.
  • TypeScript: forgotten timers, event listeners, and closures capturing large objects pin memory; use WeakMap/WeakRef for caches that should not keep keys alive.
5. Event Systems & Reactive Patterns Advanced Patterns rep 1 • 15 days since review
2nd review (overdue 14 days)
Quick Recall: An event system decouples a producer from its consumers: subscribers register callbacks, the emitter emits and every listener runs. The reactive extension adds streams of events over time with operators (map/filter/debounce) and backpressure. Node has a first-class EventEmitter; the others assemble one from a list of callbacks (Python/C++), channels (Go), or trait objects (Rust). RxJS is the canonical reactive library.

Key Pattern — subscribe a listener, then emit an event to it

TypeScript 
import { EventEmitter } from "node:events";
const bus = new EventEmitter();
bus.on("tick", (n: number) => console.log("got", n));
bus.emit("tick", 1);                 // got 1
Python 
subs = []
def on(fn): subs.append(fn)
def emit(x): [fn(x) for fn in list(subs)]   # copy list to allow removal mid-emit
on(lambda n: print("got", n)); emit(1)      # got 1
Go 
type Handler func(int)
var handlers []Handler
handlers = append(handlers, func(n int) { fmt.Println("got", n) })
for _, h := range handlers { h(1) }  // got 1   (channels are the other idiom)
Rust 
let mut subs: Vec<Box<dyn Fn(i32)>> = Vec::new();
subs.push(Box::new(|n| println!("got {}", n)));
for h in &subs { h(1); }             // got 1
C++ 
#include <functional><vector>
std::vector<std::function<void(int)>> subs;
subs.push_back([](int n){ std::cout << "got " << n; });
for (auto& h : subs) h(1);           // got 1  (or Boost.Signals2)
Challenge (all 5 languages):
1
Build an emitter with on/off/emit that supports removing a listener during an emit without skipping or double-firing others, and that doesn't leak removed listeners. Since function values usually aren't comparable, return a subscription handle/token from on that off consumes.
#1 Pitfall per Language:
  • Node/TS: listeners you never remove leak; an unhandled "error" event throws; the default 10-listener cap logs a warning for suspected leaks.
  • Python: an exception in one subscriber aborts the whole emit loop unless you guard each call; don't mutate the list while iterating it.
  • Go: an unbuffered channel with no receiver blocks the publisher; sending on a closed channel panics — fan-out needs a receiver per subscriber.
  • Rust: boxed dyn Fn closures have no equality, so removing a specific one needs an id/handle; cross-thread emitters need Fn + Send + Sync.
  • C++: a lambda capturing this dangles if the subscriber is destroyed before it's removed; std::function has no equality, so track tokens for removal.
6. Configuration Management Advanced Patterns rep 1 • 15 days since review
2nd review (overdue 14 days)
Quick Recall: Configuration is layered with a defined precedence: built-in defaults < config file (JSON/YAML/TOML) < environment variables < CLI flags, each overriding the one before. Read it once at startup, coerce strings to typed values, validate, and fail fast. Secrets come from the environment or a secrets manager — never committed files. Libraries: pydantic-settings (Py), viper (Go), convict/zod (TS), config/figment (Rust).

Key Pattern — a port setting with a default overridden by an env var

Python 
import os
port = int(os.environ.get("PORT", "8080"))   # default < env
print(port)                                    # 8080  (pydantic-settings for layers)
Go 
port := 8080
if v := os.Getenv("PORT"); v != "" { port, _ = strconv.Atoi(v) }
fmt.Println(port)                              // 8080  (viper for full layering)
TypeScript 
const port = Number(process.env.PORT ?? 8080); // default < env
console.log(port);                             // 8080  (zod/convict to validate)
Rust 
use std::env;
let port: u16 = env::var("PORT").ok()
    .and_then(|s| s.parse().ok()).unwrap_or(8080);
println!("{}", port);                          // 8080  (config/figment crates)
C++ 
#include <cstdlib><string>
const char* raw = std::getenv("PORT");
int port = raw ? std::stoi(raw) : 8080;        // no stdlib config; parse by hand
std::cout << port;                              // 8080
Challenge (all 5 languages):
1
Implement a config loader with the precedence defaults < file < env < flags, that validates types and ranges at startup and prints a clear error naming the offending key. Add one secret field that must come from the environment and must never be echoed in logs or the config dump.
#1 Pitfall per Language:
  • All: leaving the precedence order implicit — document it and apply layers in exactly one place so overrides are predictable.
  • All: validating config deep inside request handling instead of failing fast at startup when a value is missing or malformed.
  • Python/TS: naive boolean parsing — the string "false" is truthy, so bool(os.environ["X"]) / Boolean(process.env.X) is almost always wrong.
  • Go/Rust: silently swallowing a parse error (_ / .ok()) so a typo'd env var falls back to the default without warning — log or reject bad values.
  • All: committing secrets into config files under version control — keep them in env/secret managers and out of logs and config dumps.

Ordered Collections

Beyond the flat array and the hash map lies a family of structures that maintain order — a heap keeps the extreme element ready, binary search exploits a sorted sequence, and tree-based maps/sets keep keys in order for range queries. The stdlib support here diverges sharply: Rust and C++ are rich, Python is pragmatic, and Go and TypeScript often hand you the raw materials and expect you to assemble the rest.

Heaps & Priority Queues Ordered Collections
Why it matters: whenever you repeatedly need "the smallest/largest so far" — Dijkstra's algorithm, top-k queries, event simulation, merging sorted streams, a scheduler — a heap gives O(log n) insert and extract with O(1) peek, beating re-sorting every time. The one thing to burn into memory: which direction your language's default heap points.

A binary heap is a partially ordered tree stored in an array: the root is always the min (or max). push and pop are O(log n); reading the top is O(1). The trap is polarity — Python's heapq is a min-heap, but Rust's BinaryHeap and C++'s priority_queue are max-heaps by default.

Example 1 — push 5, 1, 3 and pop the smallest

Python 
import heapq
h = []
for x in (5, 1, 3): heapq.heappush(h, x)   # min-heap over a plain list
print(heapq.heappop(h))                     # 1
1
Rust 
use std::collections::BinaryHeap;
use std::cmp::Reverse;
let mut h = BinaryHeap::new();               // BinaryHeap is a MAX-heap...
for x in [5, 1, 3] { h.push(Reverse(x)); }   // ...so wrap in Reverse for min
println!("{}", h.pop().unwrap().0);          // 1
1
C++ 
#include <queue><vector>
// default priority_queue is a MAX-heap; greater<> makes it a min-heap:
std::priority_queue<int, std::vector<int>, std::greater<int>> pq;
for (int x : {5, 1, 3}) pq.push(x);
std::cout << pq.top();                        // 1
1
Go 
import "container/heap"
type IntHeap []int                            // must implement heap.Interface
func (h IntHeap) Len() int            { return len(h) }
func (h IntHeap) Less(i, j int) bool  { return h[i] < h[j] } // < = min-heap
func (h IntHeap) Swap(i, j int)       { h[i], h[j] = h[j], h[i] }
func (h *IntHeap) Push(x any)         { *h = append(*h, x.(int)) }
func (h *IntHeap) Pop() any           { o := *h; n := len(o); x := o[n-1]; *h = o[:n-1]; return x }
// h := &IntHeap{}; heap.Init(h); heap.Push(h, 5); ... heap.Pop(h) -> 1
1
TypeScript 
// No stdlib heap; a tiny min-heap (or use a library like @datastructures-js/heap):
class MinHeap { a: number[] = [];
  push(x: number){ this.a.push(x); let i=this.a.length-1;
    while(i>0){ const p=(i-1)>>1; if(this.a[p]<=this.a[i]) break;
      [this.a[p],this.a[i]]=[this.a[i],this.a[p]]; i=p; } }
  pop(){ const top=this.a[0], last=this.a.pop()!;
    if(this.a.length){ this.a[0]=last; let i=0;
      for(;;){ let s=i,l=2*i+1,r=2*i+2;
        if(l<this.a.length&&this.a[l]<this.a[s])s=l;
        if(r<this.a.length&&this.a[r]<this.a[s])s=r;
        if(s===i)break; [this.a[s],this.a[i]]=[this.a[i],this.a[s]]; i=s; } }
    return top; } }
const h=new MinHeap(); [5,1,3].forEach(x=>h.push(x)); console.log(h.pop()); // 1

Example 2 — a priority queue keyed by a custom priority (lowest first)

Pair each item with a priority and let the heap order on that. Ties need a deterministic tie-breaker so unorderable payloads don't crash the comparison.

Python 
import heapq, itertools
pq, counter = [], itertools.count()          # counter breaks ties safely
for name, prio in [("gc", 5), ("io", 1), ("net", 3)]:
    heapq.heappush(pq, (prio, next(counter), name))
print(heapq.heappop(pq)[2])                  # io
io
C++ 
#include <queue><string><vector>
using Task = std::pair<int, std::string>;     // (priority, name)
auto cmp = [](const Task& a, const Task& b){ return a.first > b.first; }; // min by prio
std::priority_queue<Task, std::vector<Task>, decltype(cmp)> pq(cmp);
pq.push({5,"gc"}); pq.push({1,"io"}); pq.push({3,"net"});
std::cout << pq.top().second;                 // io
io
Rust 
use std::collections::BinaryHeap;
use std::cmp::Reverse;
let mut pq = BinaryHeap::new();               // min by priority via Reverse on the key
for (name, prio) in [("gc", 5), ("io", 1), ("net", 3)] { pq.push(Reverse((prio, name))); }
println!("{}", (pq.pop().unwrap().0).1);      // io
io
Go 
// container/heap with a []Task and Less comparing the priority field:
type Task struct{ name string; prio int }
type PQ []Task
func (p PQ) Less(i, j int) bool { return p[i].prio < p[j].prio } // min by prio
// ...implement Len/Swap/Push/Pop as above; heap.Pop -> Task{"io", 1}
io
TypeScript 
// Generalize the heap with a comparator over {name, prio}:
import { PriorityQueue } from "@datastructures-js/priority-queue";
const pq = new PriorityQueue<{name:string; prio:number}>((a, b) => a.prio - b.prio);
[["gc",5],["io",1],["net",3]].forEach(([n,p]) => pq.enqueue({name:n as string, prio:p as number}));
console.log(pq.dequeue().name);              // io
io
Language Comparison: Python's heapq operates on a plain list and is min-only — for a max-heap, negate the keys. Rust's BinaryHeap and C++'s std::priority_queue are max-heaps; flip with Reverse / std::greater or a custom comparator. Go ships container/heap but makes you implement five methods (Len/Less/Swap/Push/Pop) on your slice type — Less chooses the polarity. TypeScript has nothing built in; write a ~15-line binary heap or pull in a library. All give O(log n) push/pop and O(1) peek, and none support an efficient decrease-key without extra bookkeeping.
Common Mistakes:
❌ [All] Assuming the wrong polarity — reaching for "the smallest" from a max-heap (or vice-versa) and getting the opposite element.
✅ [All] Memorize your language's default (Python min; Rust/C++ max) and invert with Reverse/greater/negation deliberately.
❌ [Python] Pushing bare (priority, payload) tuples where two share a priority and the payload isn't comparable — the tie comparison raises TypeError.
✅ [Python] Insert (priority, unique_counter, payload) so ties break on the counter and the payload is never compared.
❌ [Go] Calling append or slicing the heap directly instead of heap.Push/heap.Pop — that breaks the heap invariant.
✅ [Go] Always mutate through the heap package functions after heap.Init.
❌ [C++] Reading pq.top() after pop(), or expecting pop() to return the value — it returns void.
✅ [C++] Read top() first, then pop(); check empty() before either.
Key Rules:
  • Know your default polarity: Python heapq is min; Rust BinaryHeap and C++ priority_queue are max.
  • For priority queues, store (priority, tiebreaker, payload) so equal priorities never force a comparison of the payload.
  • A heap gives O(1) peek and O(log n) push/pop but no cheap search or decrease-key — use an indexed heap or a different structure if you need those.

Binary Search & Sorted Sequences Ordered Collections
Why it matters: on a sorted sequence, binary search finds an element — or the exact spot to insert one — in O(log n) instead of O(n). It powers "binary search on the answer" optimization problems, keeps a list sorted as you insert, and answers range queries. The recurring bug is the off-by-one between "first element ≥ x" and "first element > x".

Every stdlib exposes the same two primitives: lower bound (leftmost index where the element is ≥ the target, = the insertion point that keeps order) and upper bound (leftmost index > the target). The generalization is "find the boundary where a monotonic predicate flips". The sequence must already be sorted, or the result is meaningless.

Example 1 — find the insertion point of 5 in [1, 3, 5, 7]

Python 
import bisect
a = [1, 3, 5, 7]
print(bisect.bisect_left(a, 5))       # 2  (leftmost index of/for 5)
print(bisect.bisect_right(a, 5))      # 3  (just past any existing 5s)
bisect.insort(a, 4)                    # a is now [1, 3, 4, 5, 7]
2 3
C++ 
#include <algorithm><vector>
std::vector<int> a{1, 3, 5, 7};
auto lo = std::lower_bound(a.begin(), a.end(), 5); // first >= 5
auto hi = std::upper_bound(a.begin(), a.end(), 5); // first > 5
std::cout << (lo - a.begin()) << " " << (hi - a.begin()); // 2 3
2 3
Rust 
let a = [1, 3, 5, 7];
// Ok(i) if found, Err(i) = where it would be inserted to stay sorted:
println!("{:?}", a.binary_search(&5));         // Ok(2)
println!("{:?}", a.binary_search(&4));         // Err(2)
println!("{}", a.partition_point(|&x| x < 5)); // 2  (leftmost >= 5)
Ok(2) Err(2) 2
Go 
import "sort"
a := []int{1, 3, 5, 7}
i := sort.SearchInts(a, 5)             // leftmost index where a[idx] >= 5
fmt.Println(i)                         // 2
// general form: sort.Search(len(a), func(k int) bool { return a[k] >= 5 })
2
TypeScript 
// No stdlib binary search; a lower-bound helper:
function lowerBound(a: number[], x: number): number {
  let lo = 0, hi = a.length;
  while (lo < hi) { const mid = lo + ((hi - lo) >> 1);  // avoid overflow
    if (a[mid] < x) lo = mid + 1; else hi = mid; }
  return lo;
}
console.log(lowerBound([1, 3, 5, 7], 5)); // 2

Example 2 — binary search on a predicate ("first value whose square ≥ 50")

The real power is searching a monotonic boolean function, not just a literal array — the basis of "binary search on the answer".

Python 
import bisect
# bisect over a key via a computed list, or roll a predicate search:
lo, hi = 0, 100
while lo < hi:
    mid = (lo + hi) // 2
    if mid * mid >= 50: hi = mid
    else: lo = mid + 1
print(lo)                              # 8  (8*8=64 >= 50, 7*7=49 < 50)
8
Go 
i := sort.Search(100, func(n int) bool { return n*n >= 50 })
fmt.Println(i)                         // 8   (predicate must be monotonic)
8
Rust 
// partition_point finds the boundary where the predicate flips true->false:
let ans = (0..100).collect::<Vec<_>>().partition_point(|&n| n * n < 50);
println!("{}", ans);                   // 8
8
C++ 
#include <algorithm><ranges>
auto r = std::views::iota(0, 100);
// partition_point: first element for which the predicate is FALSE
auto it = std::ranges::partition_point(r, [](int n){ return n * n < 50; });
std::cout << *it;                       // 8
8
TypeScript 
function firstTrue(lo: number, hi: number, pred: (n: number) => boolean): number {
  while (lo < hi) { const mid = lo + ((hi - lo) >> 1);
    if (pred(mid)) hi = mid; else lo = mid + 1; }
  return lo;
}
console.log(firstTrue(0, 100, n => n * n >= 50)); // 8
8
Language Comparison: Python's bisect gives bisect_left/bisect_right plus insort to keep a list sorted. C++ has the canonical lower_bound/upper_bound/binary_search/equal_range plus partition_point. Rust's binary_search returns a Result encoding both the found index and the insertion point, and partition_point handles predicate search. Go's sort.Search is purely predicate-based (with SearchInts as a helper). TypeScript has none — write the four-line loop, being careful with the < vs <= boundary. Everywhere: the input must be sorted with respect to the same order/comparator.
Common Mistakes:
❌ [All] Running binary search on an unsorted (or differently-sorted) sequence — the result is silently wrong (or UB in C++).
✅ [All] Ensure the sequence is sorted by the exact comparator you search with; sort once, then search many times.
❌ [All] Computing mid = (lo + hi) / 2 in a fixed-width integer type — lo + hi can overflow.
✅ [All] Use mid = lo + (hi - lo) / 2 (or >> 1) to avoid overflow.
❌ [Rust] Assuming binary_search returns the first match when there are duplicates — it returns an arbitrary matching index.
✅ [Rust] Use partition_point (or binary_search_by) to get the leftmost/rightmost boundary deterministically.
❌ [C++] Passing a comparator to lower_bound that differs from the one the range was sorted with — undefined behavior.
✅ [C++] Use the same comparator for sorting and searching; remember lower_bound is ≥, upper_bound is >.
Key Rules:
  • The universal primitive is "leftmost index where a monotonic predicate becomes true" — lower/upper bound are special cases.
  • Input must be sorted by the same order you search with; keep it sorted with insort/insertion at the found index.
  • Guard against integer overflow in the midpoint and be deliberate about < vs ≤ for the duplicate-handling boundary.

Ordered Maps & Sorted Sets Ordered Collections
Why it matters: a hash map gives O(1) lookup but no order; a tree-based map keeps keys sorted, enabling range scans, floor/ceiling queries, and ordered iteration in O(log n). Confusing "insertion order" (what Python dict and JS Map preserve) with "sorted order" is a classic bug that only surfaces when the data isn't already in order.

Two families: hash-based (unordered, O(1) average) and tree-based (sorted by key, O(log n)). Only Rust (BTreeMap/BTreeSet) and C++ (std::map/std::set) ship sorted associative containers in the stdlib. Python dict, Go map, and JS Map are hash/insertion-ordered — for sorted iteration you sort keys or reach for a library.

Example 1 — iterate keys in sorted order

Rust 
use std::collections::BTreeMap;
let mut m = BTreeMap::new();               // kept sorted by key
m.insert("b", 2); m.insert("a", 1); m.insert("c", 3);
for (k, v) in &m { print!("{}{} ", k, v); } // a1 b2 c3  (always sorted)
a1 b2 c3
C++ 
#include <map><string>
std::map<std::string, int> m;              // red-black tree, sorted keys
m["b"] = 2; m["a"] = 1; m["c"] = 3;
for (auto& [k, v] : m) std::cout << k << v << " "; // a1 b2 c3
a1 b2 c3
Python 
d = {"b": 2, "a": 1, "c": 3}               # dict preserves INSERTION order
for k in sorted(d):                         # sort keys for sorted iteration
    print(f"{k}{d[k]}", end=" ")            # a1 b2 c3
# For a true ordered map: sortedcontainers.SortedDict
a1 b2 c3
Go 
import ("maps"; "slices")
m := map[string]int{"b": 2, "a": 1, "c": 3} // map order is RANDOMIZED
keys := slices.Sorted(maps.Keys(m))         // Go 1.23: sort the keys
for _, k := range keys { fmt.Printf("%s%d ", k, m[k]) } // a1 b2 c3
a1 b2 c3
TypeScript 
const m = new Map([["b", 2], ["a", 1], ["c", 3]]); // preserves INSERTION order
for (const [k, v] of [...m].sort((x, y) => x[0].localeCompare(y[0])))
  process.stdout.write(`${k}${v} `);        // a1 b2 c3
a1 b2 c3

Example 2 — a range query: all entries with keys in [2, 4]

The payoff of a tree map: ask for a contiguous slice of the key space in O(log n + k) instead of scanning everything.

Rust 
use std::collections::BTreeMap;
let m: BTreeMap<i32, &str> = [(1,"a"),(2,"b"),(3,"c"),(4,"d"),(5,"e")].into();
for (k, v) in m.range(2..=4) { print!("{}:{} ", k, v); } // 2:b 3:c 4:d
2:b 3:c 4:d
C++ 
#include <map>
std::map<int, char> m{{1,'a'},{2,'b'},{3,'c'},{4,'d'},{5,'e'}};
for (auto it = m.lower_bound(2); it != m.upper_bound(4); ++it) // [2, 4]
    std::cout << it->first << ":" << it->second << " ";       // 2:b 3:c 4:d
2:b 3:c 4:d
Python 
from bisect import bisect_left, bisect_right
keys = [1, 2, 3, 4, 5]                     # a plain sorted list of keys
lo, hi = bisect_left(keys, 2), bisect_right(keys, 4)
print(keys[lo:hi])                          # [2, 3, 4]  (SortedDict.irange is cleaner)
[2, 3, 4]
Go 
import "sort"
keys := []int{1, 2, 3, 4, 5}               // sorted key slice
lo := sort.SearchInts(keys, 2)
hi := sort.SearchInts(keys, 5)             // first index > 4
fmt.Println(keys[lo:hi])                    // [2 3 4]
[2 3 4]
TypeScript 
const keys = [1, 2, 3, 4, 5];              // sorted keys
const inRange = keys.filter(k => k >= 2 && k <= 4); // O(n); binary-search for O(log n)
console.log(inRange);                       // [ 2, 3, 4 ]
[ 2, 3, 4 ]
Language Comparison: Rust (BTreeMap/BTreeSet) and C++ (std::map/std::set) are the only two with true stdlib ordered containers — both offer range/lower_bound queries and ordered iteration in O(log n). Python's dict and JS's Map preserve insertion order (not sorted); Go's map is deliberately randomized. For sorted behavior in those three you either sort the keys on demand (fine for one-off iteration) or use a dedicated structure (sortedcontainers in Python). Choose a tree map only when you actually need order/range queries — otherwise a hash map's O(1) wins.
Common Mistakes:
❌ [Python/TS] Assuming a dict/Map iterates in sorted order — it iterates in insertion order, which only looks sorted if you happened to insert sorted.
✅ [Python/TS] Sort keys explicitly, or use a real sorted structure when you need ordered iteration or range queries.
❌ [Go] Relying on map iteration order for anything — it is intentionally randomized between runs.
✅ [Go] Extract keys, sort them, then iterate; there is no stdlib ordered map.
❌ [C++] Using m[key] to read a possibly-absent key — it inserts a default-constructed value as a side effect.
✅ [C++] Use m.find(key)/m.at(key)/m.contains(key) for lookups that must not mutate the map.
❌ [All] Paying O(log n) for a tree map when you never iterate in order or do range queries.
✅ [All] Default to a hash map for O(1); switch to a tree map only when order or range access is required.
Key Rules:
  • Insertion order (dict/Map) is not sorted order — only Rust BTreeMap and C++ std::map/set keep keys sorted in the stdlib.
  • Reach for a tree map when you need ordered iteration, range scans, or floor/ceiling queries; otherwise a hash map is faster.
  • In C++, never use operator[] for lookups — it inserts; use find/at/contains.

Sequence & Set Structures

Two more everyday structures whose stdlib support is wildly uneven across languages: the double-ended queue (and its fixed-capacity cousin, the ring buffer), and the set with its algebra of unions and intersections. Python and Rust hand you batteries; Go gives you a hash map and a wish of luck; JavaScript only just grew set operations in 2025.

Deques & Ring Buffers Sequence Structures
Why it matters: a queue, a sliding window, a BFS frontier, an undo history, and a "last N events" log all want O(1) insert/remove at both ends. A plain dynamic array only gives you cheap operations at the tail — using one as a queue (repeated front-removal) is quadratic. A ring buffer is the fixed-capacity, allocation-free version that overwrites the oldest entry.

A deque supports push/pop at front and back in O(1). Python's collections.deque, Rust's VecDeque, and C++'s std::deque provide this directly; Go and JS do not, and their arrays are O(n) at the front. A ring buffer caps the size and evicts the oldest — ideal for bounded logs and streaming windows.

Example 1 — push and pop at both ends

Python 
from collections import deque
d = deque([2])
d.appendleft(1); d.append(3)          # d is deque([1, 2, 3])
print(d.popleft(), d.pop())           # 1 3  (both O(1))
1 3
Rust 
use std::collections::VecDeque;
let mut d: VecDeque<i32> = VecDeque::from([2]);
d.push_front(1); d.push_back(3);       // [1, 2, 3]
println!("{} {}", d.pop_front().unwrap(), d.pop_back().unwrap()); // 1 3
1 3
C++ 
#include <deque>
std::deque<int> d{2};
d.push_front(1); d.push_back(3);       // {1, 2, 3}
std::cout << d.front() << " " << d.back(); // 1 3
d.pop_front(); d.pop_back();            // both O(1)
1 3
Go 
// No stdlib deque. Front-removal on a slice is O(n) and can leak the backing array.
// A head-index ring avoids both. Small demo with a slice:
d := []int{2}
d = append([]int{1}, d...)             // prepend (O(n) copy)
d = append(d, 3)                       // [1 2 3]
front, back := d[0], d[len(d)-1]
fmt.Println(front, back)               // 1 3
1 3
TypeScript 
// Array has unshift/shift but they are O(n). Fine for tiny data:
const d: number[] = [2];
d.unshift(1); d.push(3);               // [1, 2, 3]
console.log(d.shift(), d.pop());       // 1 3  (shift is O(n) — use a ring for hot paths)
1 3

Example 2 — a bounded ring buffer that keeps only the last 3 items

Cap the capacity and let new pushes evict the oldest — a rolling window in constant memory.

Python 
from collections import deque
ring = deque(maxlen=3)                 # auto-evicts the oldest on overflow
for x in [1, 2, 3, 4, 5]: ring.append(x)
print(list(ring))                       # [3, 4, 5]
[3, 4, 5]
Rust 
use std::collections::VecDeque;
let mut ring: VecDeque<i32> = VecDeque::with_capacity(3);
for x in 1..=5 { if ring.len() == 3 { ring.pop_front(); } ring.push_back(x); }
println!("{:?}", ring);                 // [3, 4, 5]
[3, 4, 5]
C++ 
#include <deque>
std::deque<int> ring;
for (int x : {1, 2, 3, 4, 5}) { if (ring.size() == 3) ring.pop_front(); ring.push_back(x); }
for (int x : ring) std::cout << x << " "; // 3 4 5  (or boost::circular_buffer)
3 4 5
Go 
// Fixed-capacity ring with modular indexing, no per-push allocation:
buf := make([]int, 3); n := 0
for _, x := range []int{1, 2, 3, 4, 5} { buf[n%3] = x; n++ }
// last 3 in order: start at n%3
for i := 0; i < 3; i++ { fmt.Print(buf[(n+i)%3], " ") } // 3 4 5
3 4 5
TypeScript 
class Ring<T> { private a: T[] = []; constructor(private cap: number) {}
  push(x: T) { this.a.push(x); if (this.a.length > this.cap) this.a.shift(); }
  items() { return this.a; } }
const r = new Ring<number>(3);
[1, 2, 3, 4, 5].forEach(x => r.push(x));
console.log(r.items());                 // [ 3, 4, 5 ]
[ 3, 4, 5 ]
Language Comparison: Python's collections.deque is the star — O(1) both ends and a built-in maxlen ring buffer. Rust's VecDeque is a growable ring internally. C++'s std::deque gives O(1) ends (as a segmented, non-contiguous structure). Go and JS have no deque: a JS array's shift/unshift are O(n), and reslicing a Go slice at the front leaks its backing array — both want an explicit head/tail ring buffer for front-heavy or bounded workloads.
Common Mistakes:
❌ [JS/TS] Using an array as a queue via push+shift — every shift re-indexes the whole array, making the loop O(n²).
✅ [JS/TS] Use a ring buffer with a moving head index, or a real deque library, for FIFO on hot paths.
❌ [Go] Dequeuing with q = q[1:] repeatedly — the backing array is never freed and memory grows unbounded.
✅ [Go] Use a fixed ring buffer with modular indices, or periodically copy the live portion to a fresh slice.
❌ [C++] Treating std::deque like std::vector — it is not contiguous, so there's no single data() pointer over all elements.
✅ [C++] Use std::vector when you need contiguity/data(); use std::deque for cheap front operations.
❌ [All] Ring buffer full-vs-empty ambiguity when head == tail — you can't tell a full ring from an empty one.
✅ [All] Track an explicit size/count, or intentionally leave one slot unused to disambiguate.
Key Rules:
  • For O(1) at both ends use a real deque (Python deque, Rust VecDeque, C++ std::deque); never an array's front operations on a hot path.
  • A ring buffer is a deque with a fixed capacity that overwrites the oldest — ideal for bounded logs and sliding windows.
  • Disambiguate a full ring from an empty one with an explicit count or a reserved empty slot.

Set Algebra & Multisets Sequence Structures
Why it matters: deduplication, membership tests, "which users are in both cohorts", permission diffs, and tag intersections are all set algebra — union, intersection, difference, symmetric difference. A multiset (bag) extends this to counts: word frequencies, inventory, histogram buckets. Knowing which operations your language gives for free saves a lot of hand-rolled loops.

A set stores unique elements with O(1) membership (hash) or O(log n) ordered (tree). The four core operations are union (|), intersection (&), difference (-), and symmetric difference (^). A multiset keeps a count per element. Python is the most batteries-included; Go has no set type at all.

Example 1 — union, intersection, and difference of two sets

Python 
a, b = {1, 2, 3}, {2, 3, 4}
print(a | b, a & b, a - b)             # {1, 2, 3, 4} {2, 3} {1}
print(a ^ b)                           # {1, 4}  (symmetric difference)
{1, 2, 3, 4} {2, 3} {1} {1, 4}
Rust 
use std::collections::HashSet;
let a: HashSet<i32> = [1, 2, 3].into();
let b: HashSet<i32> = [2, 3, 4].into();
let inter: HashSet<_> = a.intersection(&b).copied().collect(); // {2, 3}
println!("{:?}", &a - &b);              // {1}  (operators also work on &HashSet)
{1}
C++ 
#include <set><algorithm><iterator>
std::set<int> a{1, 2, 3}, b{2, 3, 4}, out;   // set_* needs SORTED ranges
std::set_intersection(a.begin(), a.end(), b.begin(), b.end(),
                      std::inserter(out, out.begin()));
for (int x : out) std::cout << x << " ";     // 2 3
2 3
TypeScript 
const a = new Set([1, 2, 3]), b = new Set([2, 3, 4]);
// ES2025 / Node 22+ set methods:
console.log([...a.intersection(b)]);   // [2, 3]
console.log([...a.difference(b)]);     // [1]
// Older runtimes: [...a].filter(x => b.has(x)) for intersection
[ 2, 3 ] [ 1 ]
Go 
// No set type; the idiom is map[T]struct{} (struct{} uses zero bytes):
a := map[int]struct{}{1: {}, 2: {}, 3: {}}
b := map[int]struct{}{2: {}, 3: {}, 4: {}}
for k := range a { if _, ok := b[k]; ok { fmt.Print(k, " ") } } // 2 3 (order random)
2 3

Example 2 — a multiset / frequency count (a "bag")

When elements can repeat and you care how many times, you want counts, not just membership.

Python 
from collections import Counter
c = Counter("banana")                  # {'a': 3, 'n': 2, 'b': 1}
print(c["a"], c.most_common(1))        # 3 [('a', 3)]
print(Counter("aab") + Counter("abc")) # Counter({'a': 3, 'b': 2, 'c': 1})
3 [('a', 3)] Counter({'a': 3, 'b': 2, 'c': 1})
C++ 
#include <unordered_multiset><string>
std::unordered_multiset<char> bag;      // a real multiset container
for (char ch : std::string("banana")) bag.insert(ch);
std::cout << bag.count('a');            // 3
3
Rust 
use std::collections::HashMap;
let mut freq: HashMap<char, usize> = HashMap::new();
for ch in "banana".chars() { *freq.entry(ch).or_insert(0) += 1; } // no stdlib multiset
println!("{}", freq[&'a']);             // 3
3
Go 
freq := map[rune]int{}
for _, ch := range "banana" { freq[ch]++ } // map[T]int is the multiset idiom
fmt.Println(freq['a'])                  // 3
3
TypeScript 
const freq = new Map<string, number>();
for (const ch of "banana") freq.set(ch, (freq.get(ch) ?? 0) + 1);
console.log(freq.get("a"));            // 3
3
Language Comparison: Python is richest: set operators (| & - ^) plus Counter as a full-featured multiset with arithmetic. Rust's HashSet/BTreeSet offer both methods (union/intersection/…) and operators, but has no stdlib multiset (use a HashMap<T, usize>). C++ provides set_union/set_intersection/… over sorted ranges plus real multiset/unordered_multiset containers. JS/TS only gained Set.prototype.union/intersection/… in ES2025 (Node 22+); before that you loop. Go has no set at all — map[T]struct{} for a set and map[T]int for a multiset, with every operation hand-written.
Common Mistakes:
❌ [Python] Writing {} for an empty set — that's an empty dict.
✅ [Python] Use set() for an empty set; {1, 2} only works when non-empty.
❌ [C++] Passing unsorted ranges to set_intersection/set_union — they require sorted input and silently produce garbage otherwise.
✅ [C++] Use std::set (already sorted) or sort your vectors first, and write results via std::inserter/back_inserter.
❌ [TS/JS] Calling a.intersection(b) on a runtime without ES2025 set methods — it throws is not a function.
✅ [TS/JS] Feature-detect or target Node 22+/modern browsers; otherwise fall back to [...a].filter(x => b.has(x)).
❌ [Go] Using map[T]bool for a set and then confusing "absent" with "present but false".
✅ [Go] Use map[T]struct{} and test membership with the comma-ok form _, ok := m[k].
Key Rules:
  • The four operations are union, intersection, difference, and symmetric difference — know which your language spells as operators vs methods vs hand-written loops.
  • A multiset (bag) tracks counts: Python Counter, C++ multiset, or a map[T]int everywhere else.
  • C++ set_* algorithms require sorted ranges and an output iterator; Go has no set type — use map[T]struct{}.
