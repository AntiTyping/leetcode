1. Decorators & Middleware Functions & Abstractions rep 3 • 15 days since review
Due today • 4th review (on time)
Quick Recall: A decorator/middleware wraps a function to add behavior (logging, timing, auth, retries) without touching its body — the "wrap and delegate" shape. Python has @decorator syntax plus functools.wraps; JS/TS use higher-order functions (or the newer TC39 decorators); Go/Rust/C++ have no decorator syntax, so you wrap with a closure that returns a new function of the same signature.

Key Pattern — a logged(fn) wrapper that announces the call, then delegates

Python 
import functools
def logged(fn):
    @functools.wraps(fn)               # preserve __name__/__doc__
    def wrapper(*args, **kwargs):
        print(f"call {fn.__name__}")
        return fn(*args, **kwargs)
    return wrapper
@logged
def add(a, b): return a + b
print(add(2, 3))                       # call add / 5
call add 5
Go 
func logged(name string, fn func(int, int) int) func(int, int) int {
    return func(a, b int) int {         // same signature, wrapped
        fmt.Println("call", name)
        return fn(a, b)
    }
}
add := logged("add", func(a, b int) int { return a + b })
fmt.Println(add(2, 3))                  // call add / 5
call add 5
TypeScript 
function logged<A extends any[], R>(fn: (...a: A) => R) {
  return (...a: A): R => { console.log("call", fn.name); return fn(...a); };
}
const add = logged(function add(a: number, b: number) { return a + b; });
console.log(add(2, 3));                 // call add / 5
call add 5
Rust 
fn logged<F: Fn(i32, i32) -> i32>(name: &'static str, f: F)
    -> impl Fn(i32, i32) -> i32 {
    move |a, b| { println!("call {}", name); f(a, b) }
}
let add = logged("add", |a, b| a + b);
println!("{}", add(2, 3));              // call add / 5
call add 5
C++ 
auto logged = [](std::string name, auto fn) {
    return [=](int a, int b) {          // capture by value so it can escape
        std::cout << "call " << name << "\n"; return fn(a, b);
    };
};
auto add = logged("add", [](int a, int b){ return a + b; });
std::cout << add(2, 3);                 // call add / 5
call add 5

Write a parameterized retry(n) wrapper that re-invokes the wrapped function up to n times until it succeeds, and a memoize that caches by arguments. Note which languages need functools.wraps-style metadata preservation and which change the function's type when wrapped.
#1 Pitfall per Language:
  • Python: forgetting functools.wraps loses __name__/__doc__ and muddies tracebacks and introspection.
  • Go: no decorator syntax — the wrapper must match the exact func type (or use generics), and it changes nothing about how callers invoke it.
  • TypeScript: TC39 @decorators on classes/methods are a different mechanism from wrapping plain functions; legacy experimentalDecorators is a separate, incompatible model.
  • Rust: each move closure has a unique unnameable type — return impl Fn or box it as Box<dyn Fn>.
  • C++: capturing fn by reference [&] in an escaping lambda dangles — capture by value [=].
2. Module Systems Functions & Abstractions rep 3 • 15 days since review
Due today • 4th review (on time)
Quick Recall: A module is a unit of encapsulation and a namespace. What's exported vs private, and how names resolve, differs sharply: Python modules are files (_name convention, __all__); Go packages export by Capitalization; TS ES modules use explicit export/import; Rust has an explicit mod tree with pub; C++ uses headers (textual include) plus C++20 modules.

Key Pattern — a module exposing greet while hiding a shout helper

Python 
# greetings.py
def _shout(s): return s.upper()          # leading _ = module-private (convention)
def greet(name): return _shout(f"hi {name}")
# main.py
from greetings import greet
print(greet("sam"))                      # HI SAM
HI SAM
Go 
package greetings
func shout(s string) string { return strings.ToUpper(s) } // lowercase = unexported
func Greet(name string) string { return shout("hi " + name) } // Capital = exported
// package main:
fmt.Println(greetings.Greet("sam"))      // HI SAM
HI SAM
TypeScript 
// greetings.ts
const shout = (s: string) => s.toUpperCase();   // not exported = private
export function greet(name: string) { return shout(`hi ${name}`); }
// main.ts
import { greet } from "./greetings";
console.log(greet("sam"));               // HI SAM
HI SAM
Rust 
mod greetings {
    fn shout(s: &str) -> String { s.to_uppercase() }   // private to the module
    pub fn greet(name: &str) -> String { shout(&format!("hi {}", name)) }
}
println!("{}", greetings::greet("sam"));  // HI SAM
HI SAM
C++ 
// greetings.ixx  (C++20 module)
export module greetings;
import <string>; import <algorithm>;
static std::string shout(std::string s) {           // internal linkage = private
    std::transform(s.begin(), s.end(), s.begin(), ::toupper); return s; }
export std::string greet(std::string n) { return shout("hi " + n); }
// main.cpp:  import greetings;  std::cout << greet("sam");   // HI SAM
HI SAM

1

#1 Pitfall per Language:
  • Python: a circular import executes a half-initialized module — import the name lazily inside the function, or restructure the dependency.
  • Go: an unused import is a compile error, and import cycles between packages are forbidden entirely.
  • TypeScript: mixing CommonJS require and ES import (esModuleInterop) causes default-import surprises.
  • Rust: mod foo; declares the module (pulls the file in) — forget it and the file never compiles; use only brings names into scope.
  • C++: a header without an include guard / #pragma once yields duplicate-definition (ODR) errors; C++20 modules fix this but tooling is still uneven.
3. Async/Await Deep Dive Concurrency rep 2 • 15 days since review
3rd review (overdue 9 days)
Quick Recall: An async function returns a promise/future/coroutine that runs concurrently, not in parallel by default. await suspends at a yield point, freeing the thread for other work. The #1 mistake everywhere: awaiting sequentially in a loop instead of starting all the tasks and then awaiting them together (gather/join/Promise.all). It's for I/O concurrency — CPU-bound work doesn't benefit.

Key Pattern — run two async tasks concurrently and combine (2 + 3 = 5)

Python 
import asyncio
async def fetch(x):
    await asyncio.sleep(0.01); return x
async def main():
    a, b = await asyncio.gather(fetch(2), fetch(3))  # concurrent
    print(a + b)                                     # 5
asyncio.run(main())
5
Go 
ch := make(chan int, 2)
for _, x := range []int{2, 3} {
    go func(n int) { ch <- n }(x)    // concurrent goroutines
}
fmt.Println(<-ch + <-ch)             // 5
5
TypeScript 
const fetch = (x: number) =>
  new Promise<number>(r => setTimeout(() => r(x), 10));
const [a, b] = await Promise.all([fetch(2), fetch(3)]);  // concurrent
console.log(a + b);                  // 5
5
Rust 
use tokio::join;
async fn fetch(x: i32) -> i32 { x }
#[tokio::main]
async fn main() {
    let (a, b) = join!(fetch(2), fetch(3));  // concurrent on one task
    println!("{}", a + b);                   // 5
}
5
C++ 
#include <future>
auto a = std::async(std::launch::async, []{ return 2; });
auto b = std::async(std::launch::async, []{ return 3; });
std::cout << a.get() + b.get();       // 5
5
Challenge (all 5 languages):
1
Implement fetch_all(urls) that runs N async requests concurrently with a concurrency cap of K (a semaphore), collects results in order, and cancels the rest if any one fails. Measure wall-time for sequential awaits vs gather/join and explain the gap.
#1 Pitfall per Language:
  • Python: await in a for loop is sequential — use asyncio.gather/TaskGroup; and never call blocking I/O inside a coroutine (it stalls the whole loop).
  • Go: writing to an unbuffered channel with no reader deadlocks; a goroutine blocked forever is a resource leak.
  • TypeScript: an unawaited promise that rejects becomes an unhandled rejection, and forEach(async ...) does not await.
  • Rust: futures are lazy — a future you never .await (or spawn) does nothing; blocking calls starve the executor (use spawn_blocking).
  • C++: std::async with the default policy may run deferred (lazily on .get()), giving zero concurrency — pass std::launch::async explicitly.
4. Python: Dataclasses, __post_init__ & Slots Language Deep Dive rep 2 • 12 days since review
3rd review (overdue 6 days)
Quick Recall: @dataclass auto-generates __init__, __repr__, and __eq__ from annotated fields. __post_init__ runs after init for derived/validated fields; frozen=True makes it immutable and hashable; slots=True (3.10+) drops the per-instance __dict__ to save memory and block attribute typos. Below, each language builds the same "value struct with a derived field" its own way.

Key Pattern — a Point with a computed dist field

Python 
from dataclasses import dataclass, field
@dataclass(slots=True)
class Point:
    x: float
    y: float
    dist: float = field(init=False)
    def __post_init__(self): self.dist = (self.x**2 + self.y**2) ** 0.5
p = Point(3, 4)
print(p.dist, p)         # 5.0 Point(x=3, y=4, dist=5.0)
5.0 Point(x=3, y=4, dist=5.0)
Go 
type Point struct{ X, Y, Dist float64 }
func NewPoint(x, y float64) Point {
    return Point{X: x, Y: y, Dist: math.Hypot(x, y)}  // derive in constructor
}
p := NewPoint(3, 4)
fmt.Printf("%.1f %+v\n", p.Dist, p)   // 5.0 {X:3 Y:4 Dist:5}
5.0 {X:3 Y:4 Dist:5}
TypeScript 
class Point {
  readonly dist: number;
  constructor(readonly x: number, readonly y: number) {
    this.dist = Math.hypot(x, y);          // derived in constructor
  }
}
const p = new Point(3, 4);
console.log(p.dist, p);   // 5 Point { x: 3, y: 4, dist: 5 }
5 Point { x: 3, y: 4, dist: 5 }
Rust 
#[derive(Debug, Clone, PartialEq)]
struct Point { x: f64, y: f64, dist: f64 }
impl Point {
    fn new(x: f64, y: f64) -> Self { Self { x, y, dist: x.hypot(y) } }
}
let p = Point::new(3.0, 4.0);
println!("{} {:?}", p.dist, p);  // 5 Point { x: 3.0, y: 4.0, dist: 5.0 }
5 Point { x: 3.0, y: 4.0, dist: 5.0 }
C++ 
struct Point {
    double x, y, dist;
    Point(double x, double y) : x(x), y(y), dist(std::hypot(x, y)) {}
};
Point p(3, 4);
std::cout << p.dist;      // 5
5

5. Go: Channel Patterns & Pipeline Language Deep Dive rep 2 • 12 days since review
3rd review (overdue 6 days)
Quick Recall: Channels are typed conduits that also synchronize. Core patterns: a generator that produces onto a channel, a pipeline stage that reads–transforms–writes, close(ch) to signal done (so range exits), and fan-out/fan-in for parallelism; select multiplexes. Other languages model the same dataflow with async queues or (async) iterators.

Key Pattern — a 2-stage pipeline: generate 1..3, square, sum (→ 14)

Go 
gen := func() <-chan int {
    out := make(chan int)
    go func() { defer close(out); for i := 1; i <= 3; i++ { out <- i } }()
    return out
}
sq := func(in <-chan int) <-chan int {
    out := make(chan int)
    go func() { defer close(out); for n := range in { out <- n * n } }()
    return out
}
sum := 0
for v := range sq(gen()) { sum += v }
fmt.Println(sum)             // 14  (1 + 4 + 9)
14
Python 
import asyncio
async def gen(q):
    for i in range(1, 4): await q.put(i)
    await q.put(None)                    # sentinel = "close"
async def main():
    q = asyncio.Queue(); asyncio.create_task(gen(q)); total = 0
    while (n := await q.get()) is not None: total += n * n
    print(total)                         # 14
asyncio.run(main())
14
TypeScript 
async function* gen() { for (let i = 1; i <= 3; i++) yield i; }
async function* sq(src: AsyncIterable<number>) {
  for await (const n of src) yield n * n;
}
let sum = 0;
for await (const v of sq(gen())) sum += v;
console.log(sum);            // 14
14
Rust 
use std::sync::mpsc::channel;
use std::thread;
let (tx, rx) = channel();
thread::spawn(move || { for i in 1..=3 { tx.send(i * i).unwrap(); } });
let sum: i32 = rx.iter().sum();   // ends when the last tx drops
println!("{}", sum);              // 14
14
C++ 
// C++ has no built-in channel; a thread-safe queue + condition_variable
// is the analogue. The pipeline shape, reduced to its essence:
std::vector<int> out;
for (int i = 1; i <= 3; ++i) out.push_back(i * i);
int sum = std::accumulate(out.begin(), out.end(), 0);
std::cout << sum;            // 14
14
Challenge (all 5 languages):
1
Build a fan-out/fan-in: one generator, K worker stages squaring values concurrently, and one collector summing results, with clean shutdown (close/cancel) and no goroutine/task leak. Prove the sum is correct regardless of the worker count.
#1 Pitfall per Language:
  • Go: sending on a closed channel panics, closing twice panics, and only the sender should close; a receive from a nil channel blocks forever.
  • Python: forgetting the sentinel (or misusing task_done/join) leaves the consumer awaiting get() forever.
  • TypeScript: an async generator whose consumer stops early won't run finally unless the loop calls .return() (a break does) — clean up deliberately.
  • Rust: the mpsc receiver iterator ends only when all senders drop — a lingering tx clone keeps it blocked.
  • C++: a hand-rolled queue without a condition variable busy-waits; forgetting to notify on shutdown deadlocks consumers.
6. TypeScript: Template Literal Types & satisfies Language Deep Dive rep 2 • 9 days since review
3rd review (overdue 3 days)
Quick Recall: Template literal types build string-literal unions at the type level (`on${Capitalize<Event>}`), giving precise, autocompletable keys. satisfies checks a value against a type without widening it — you keep the narrow inferred type while proving conformance. Most languages have no type-level string manipulation; their closest analogue is a closed set modeled with typed constants or an enum.

Key Pattern — derive handler key names from an event union; type-check with satisfies

TypeScript 
type Event = "click" | "focus";
type Handler = `on${Capitalize<Event>}`;   // "onClick" | "onFocus"
const handlers = {
  onClick: () => 1,
  onFocus: () => 2,
} satisfies Record<Handler, () => number>;
console.log(handlers.onClick());   // 1  (type stays exact, not widened)
1
Python 
from typing import Literal, Callable
Handler = Literal["onClick", "onFocus"]    # closed set (no template derivation)
handlers: dict[Handler, Callable[[], int]] = {
    "onClick": lambda: 1, "onFocus": lambda: 2}
print(handlers["onClick"]())               # 1
1
Go 
type Handler string
const ( OnClick Handler = "onClick"; OnFocus Handler = "onFocus" )
handlers := map[Handler]func() int{ OnClick: func() int { return 1 } }
fmt.Println(handlers[OnClick]())           // 1
1
Rust 
enum Event { Click, Focus }
impl Event {
    const fn handler(&self) -> &'static str {
        match self { Event::Click => "onClick", Event::Focus => "onFocus" }
    }
}
println!("{}", Event::Click.handler());    // onClick
onClick
C++ 
enum class Event { Click, Focus };
constexpr const char* handler(Event e) {
    return e == Event::Click ? "onClick" : "onFocus";
}
std::cout << handler(Event::Click);         // onClick
onClick

1
In TS, build a typed event emitter where on(name, cb) only accepts keys derived via template literals from an event map, and emit infers the payload type per event. In the other languages, get as close as you can with enums/constants + a registry, and note exactly what compile-time guarantee you lose.
#1 Pitfall per Language:
  • TypeScript: a type annotation (const c: Record<...> = ...) widens/erases the literal types; satisfies checks without widening. Deeply recursive template types blow up compile time.
  • Python: Literal is enforced only by the type checker (mypy/pyright), never at runtime — a wrong string still runs.
  • Go: type Handler string is distinct, but any string literal can be converted to it — there's no exhaustiveness over the allowed set.
  • Rust: no type-level string concatenation; model the closed set as an enum and map to &'static str.
  • C++: constexpr string manipulation is limited pre-C++20; template string params (fixed_string) are advanced and compiler-dependent.

Binary Data & Encoding

Where numbers meet bytes and bytes meet text. Bit manipulation packs flags and masks; endianness decides byte order the instant data leaves the process; base64/hex make raw bytes safe to travel through text-only channels. Each language ships a different toolkit — and a different set of traps.

Bitwise Operations & Bit Manipulation Binary Data
Why it matters: bitwise operations pack flags, build masks, and power hashing, permissions, graphics, and low-level protocols. Confusing logical (&&) with bitwise (&), mishandling signed vs unsigned shifts, or shifting past a type's width produces subtle, platform-dependent bugs — sometimes undefined behavior.

AND/OR/XOR/NOT and the shifts operate per-bit. The everyday idioms: set x |= 1<<n, clear x &= ~(1<<n), toggle x ^= 1<<n, test (x>>n)&1. Watch signedness and width: a right-shift on a signed negative may be arithmetic (sign-extending) or logical depending on the language, and shifting by ≥ the type width is undefined in C/C++ and panics/wraps in Rust.

Example 1 — set, test, and clear bit 2; then popcount

Python 
flags = 0
flags |= 1 << 2                 # set bit 2  -> 0b100
print(flags, (flags >> 2) & 1)  # 4 1
flags &= ~(1 << 2)              # clear bit 2
print(flags, (0b1011).bit_count())  # 0 3   (popcount, Py 3.10+)
4 1 0 3
Go 
import "math/bits"
flags := 0
flags |= 1 << 2                 // set bit 2
fmt.Println(flags, (flags>>2)&1)      // 4 1
flags &^= 1 << 2                // AND NOT: Go's dedicated bit-clear operator
fmt.Println(flags, bits.OnesCount(0b1011)) // 0 3
4 1 0 3
TypeScript 
let flags = 0;
flags |= 1 << 2;                      // set bit 2
console.log(flags, (flags >> 2) & 1); // 4 1
flags &= ~(1 << 2);                   // clear bit 2
const popcount = (0b1011).toString(2).split("").filter(c => c === "1").length;
console.log(flags, popcount);         // 0 3
4 1 0 3
Rust 
let mut flags: u8 = 0;
flags |= 1 << 2;                       // set bit 2
println!("{} {}", flags, (flags >> 2) & 1); // 4 1
flags &= !(1 << 2);                    // clear bit 2
println!("{} {}", flags, 0b1011u8.count_ones()); // 0 3
4 1 0 3
C++ 
#include <bit>
unsigned flags = 0;
flags |= 1u << 2;                      // set bit 2
std::cout << flags << " " << ((flags >> 2) & 1) << "\n"; // 4 1
flags &= ~(1u << 2);                   // clear bit 2
std::cout << flags << " " << std::popcount(0b1011u);     // 0 3  (C++20)
4 1 0 3

Example 2 — a permissions bitset, and the power-of-two test

Combine flags with OR, test membership with AND. The classic trick n & (n-1) clears the lowest set bit — it is zero exactly when n is a power of two.

Python 
READ, WRITE, EXEC = 1, 2, 4
perms = READ | WRITE                    # 3
print(perms & WRITE != 0, perms & EXEC != 0)   # True False
def is_pow2(n): return n > 0 and n & (n - 1) == 0
print(is_pow2(8), is_pow2(6))           # True False
True False True False
Go 
const ( Read = 1 << iota; Write; Exec )  // 1, 2, 4
perms := Read | Write
fmt.Println(perms&Write != 0, perms&Exec != 0)  // true false
isPow2 := func(n int) bool { return n > 0 && n&(n-1) == 0 }
fmt.Println(isPow2(8), isPow2(6))               // true false
true false true false
TypeScript 
const READ = 1, WRITE = 2, EXEC = 4;
const perms = READ | WRITE;
console.log((perms & WRITE) !== 0, (perms & EXEC) !== 0); // true false
const isPow2 = (n: number) => n > 0 && (n & (n - 1)) === 0;
console.log(isPow2(8), isPow2(6));                        // true false
true false true false
Rust 
const READ: u8 = 1; const WRITE: u8 = 2; const EXEC: u8 = 4;
let perms = READ | WRITE;
println!("{} {}", perms & WRITE != 0, perms & EXEC != 0);  // true false
let is_pow2 = |n: u32| n > 0 && n & (n - 1) == 0;
println!("{} {}", is_pow2(8), is_pow2(6));                // true false
true false true false
C++ 
constexpr unsigned READ = 1, WRITE = 2, EXEC = 4;
unsigned perms = READ | WRITE;
std::cout << ((perms & WRITE) != 0) << " " << ((perms & EXEC) != 0) << "\n"; // 1 0
auto is_pow2 = [](unsigned n){ return n > 0 && (n & (n - 1)) == 0; };
std::cout << is_pow2(8) << " " << is_pow2(6);   // 1 0
1 0 1 0
Language Comparison: Python ints are arbitrary-precision, so there's no overflow and ~x == -(x+1) (two's complement of an unbounded int) — mask with & 0xFF to emulate a fixed width. Go, Rust, and C++ have fixed-width integer types where shift amount matters and signed vs unsigned shifts differ; Rust panics on an overflowing shift in debug, C/C++ make a shift ≥ width undefined. TypeScript/JS bitwise ops coerce operands to 32-bit signed ints (the x|0 idiom) — >>> is the only logical/unsigned shift, and values above 231 wrap; use BigInt bit ops beyond 32 bits. Popcount is built in everywhere: int.bit_count() (Py 3.10), bits.OnesCount (Go), count_ones (Rust), std::popcount (C++20).
Common Mistakes:
❌ [All] Writing &&/|| (logical) where you meant &/| (bitwise), or vice versa — compiles fine, computes garbage.
✅ [All] Bitwise for bit patterns; logical for booleans; add parentheses (bitwise ops have surprisingly low precedence).
❌ [TypeScript] Bit ops silently truncate to 32 bits — 1 << 31 is negative and 1 << 32 is 1 (the shift count is taken mod 32).
✅ [TypeScript] Use BigInt (1n << 40n) for values or shifts beyond 32 bits.
❌ [C++] Shifting a signed value that overflows, or by ≥ the type width, is undefined behavior.
✅ [C++] Twiddle bits on unsigned/uint32_t and keep the shift count in range.
❌ [Python] Expecting ~5 to be a fixed-width complement — it's -6, not 0xFA.
✅ [Python] Mask to the width you want: ~5 & 0xFF == 250.
Key Rules:
  • Set x |= 1<<n, clear x &= ~(1<<n), toggle x ^= 1<<n, test (x>>n)&1.
  • Use unsigned types for shifting and masking to avoid sign-extension and undefined behavior.
  • n & (n-1) clears the lowest set bit — zero iff n is a power of two, and the basis of an O(popcount) bit-counting loop.

Endianness & Binary Byte Buffers Binary Data
Why it matters: the instant you write a number to a file, socket, or binary format, byte order matters. Big-endian ("network order") and little-endian (most CPUs) disagree on which byte comes first; guess wrong and 1 is read back as 16777216. Explicit, fixed-width, defined-endian serialization is the only portable way to move numbers between machines.

Serialize an integer to a byte buffer with a chosen width and endianness, and deserialize by reversing exactly. Never memcpy a native in-memory struct across a boundary — struct padding and byte order differ per platform. By convention, network protocols use big-endian.

Example 1 — encode the u32 1 as 4 big-endian bytes, then decode it back

Python 
import struct
buf = struct.pack(">I", 1)          # > = big-endian, I = uint32
print(list(buf))                    # [0, 0, 0, 1]
(n,) = struct.unpack(">I", buf)
print(n)                            # 1
[0, 0, 0, 1] 1
Go 
import "encoding/binary"
buf := make([]byte, 4)
binary.BigEndian.PutUint32(buf, 1)
fmt.Println(buf)                          // [0 0 0 1]
fmt.Println(binary.BigEndian.Uint32(buf)) // 1
[0 0 0 1] 1
TypeScript 
const buf = new ArrayBuffer(4);
const view = new DataView(buf);
view.setUint32(0, 1, false);             // false = big-endian
console.log([...new Uint8Array(buf)]);   // [0, 0, 0, 1]
console.log(view.getUint32(0, false));   // 1
[ 0, 0, 0, 1 ] 1
Rust 
let buf = 1u32.to_be_bytes();            // big-endian
println!("{:?}", buf);                   // [0, 0, 0, 1]
println!("{}", u32::from_be_bytes(buf)); // 1
[0, 0, 0, 1] 1
C++ 
#include <cstdint>
uint32_t n = 1;
uint8_t buf[4] = {                       // big-endian: MSB first
    uint8_t(n >> 24), uint8_t(n >> 16), uint8_t(n >> 8), uint8_t(n) };
for (auto b : buf) std::cout << (int)b;  // 0001
uint32_t back = (uint32_t(buf[0]) << 24) | (buf[1] << 16)
              | (buf[2] << 8) | buf[3];
std::cout << " " << back;                 // 1
0001 1

Example 2 — the same value little-endian: the bytes reverse

Little-endian stores the least-significant byte first. Note 258 == 0x0102: big-endian is [0, 0, 1, 2], little-endian is [2, 1, 0, 0].

Python 
print(list(struct.pack("<I", 1)))   # [1, 0, 0, 0]   little-endian
print(list(struct.pack(">I", 258))) # [0, 0, 1, 2]   258 = 0x0102
[1, 0, 0, 0] [0, 0, 1, 2]
Go 
le := make([]byte, 4); binary.LittleEndian.PutUint32(le, 1)
fmt.Println(le)                     // [1 0 0 0]
be := make([]byte, 4); binary.BigEndian.PutUint32(be, 258)
fmt.Println(be)                     // [0 0 1 2]
[1 0 0 0] [0 0 1 2]
TypeScript 
const v = new DataView(new ArrayBuffer(4));
v.setUint32(0, 1, true);            // true = little-endian
console.log([...new Uint8Array(v.buffer)]); // [1, 0, 0, 0]
v.setUint32(0, 258, false);
console.log([...new Uint8Array(v.buffer)]); // [0, 0, 1, 2]
[ 1, 0, 0, 0 ] [ 0, 0, 1, 2 ]
Rust 
println!("{:?}", 1u32.to_le_bytes());   // [1, 0, 0, 0]
println!("{:?}", 258u32.to_be_bytes()); // [0, 0, 1, 2]
[1, 0, 0, 0] [0, 0, 1, 2]
C++ 
uint32_t v = 258;                    // 0x00000102
uint8_t le[4] = { uint8_t(v), uint8_t(v >> 8),
                  uint8_t(v >> 16), uint8_t(v >> 24) };
for (auto b : le) std::cout << (int)b;  // 2100  (little-endian bytes)
// big-endian of 258 would print bytes 0 0 1 2
2100
Language Comparison: Python's struct uses format chars for endianness (<, >, ! = network) and width; int.to_bytes(4, "big") is the modern OO form. Go's encoding/binary exposes explicit BigEndian/LittleEndian helpers plus binary.Read/Write. TypeScript/JS use DataView with an explicit littleEndian boolean — but typed-array views like Uint32Array follow the host's native order, a portability trap. Rust puts to_be_bytes/to_le_bytes/to_ne_bytes on every integer (prefer be/le over ne for I/O). C++ pre-23 has no portable helper — use shifts/masks; C++20 adds std::endian and C++23 std::byteswap.
Common Mistakes:
❌ [All] Serializing a native struct (raw memcpy) and reading it on another machine — struct padding and endianness both differ.
✅ [All] Serialize field by field with an explicit width and byte order.
❌ [Rust/C++] Using native-endian (to_ne_bytes / raw reinterpret_cast) for data that leaves the process — it breaks across architectures.
✅ [Rust/C++] Choose to_be_bytes/to_le_bytes (or explicit shifts) for anything persisted or transmitted.
❌ [TypeScript] Assuming new Uint32Array(buf) gives network order — it uses host endianness.
✅ [TypeScript] Use DataView with an explicit endian flag for any I/O.
Key Rules:
  • Always choose width + endianness explicitly for any bytes crossing a process/machine/file boundary.
  • Network protocols default to big-endian ("network byte order") — document whatever you pick.
  • Typed-array / native views follow host endianness; use an endian-aware reader/writer (DataView, struct, encoding/binary) for portability.

Base64, Hex & Text Encoding Binary Data
Why it matters: you constantly need to move binary data through text-only channels — JSON, URLs, HTTP headers, email. Base64 and hex encode bytes as ASCII; and text itself must be encoded (UTF-8) to bytes before hashing or transmission. Confusing "encoding" (a reversible byte↔text representation) with encryption, or forgetting to encode text to bytes first, corrupts data and creates security misconceptions.

Base64 maps 3 bytes → 4 ASCII chars (~33% larger); hex maps 1 byte → 2 chars (100% larger, human-readable). Neither is encryption — both are fully reversible with no key. Separately, a str must be encoded to bytes (UTF-8) before base64/hex/hashing. URL-safe base64 swaps +/ for -_ and often drops = padding.

Example 1 — UTF-8 encode a string, then base64 and hex it

Python 
import base64
data = "hi!".encode("utf-8")             # b'hi!'  -> bytes first
print(base64.b64encode(data).decode())   # aGkh
print(data.hex())                        # 686921
print(base64.b64decode("aGkh").decode()) # hi!
aGkh 686921 hi!
Go 
import ("encoding/base64"; "encoding/hex")
data := []byte("hi!")
fmt.Println(base64.StdEncoding.EncodeToString(data)) // aGkh
fmt.Println(hex.EncodeToString(data))                // 686921
dec, _ := base64.StdEncoding.DecodeString("aGkh")
fmt.Println(string(dec))                             // hi!
aGkh 686921 hi!
TypeScript 
// Node.js Buffer:
const data = Buffer.from("hi!", "utf-8");
console.log(data.toString("base64"));    // aGkh
console.log(data.toString("hex"));       // 686921
console.log(Buffer.from("aGkh", "base64").toString("utf-8")); // hi!
aGkh 686921 hi!
Rust 
use base64::{engine::general_purpose::STANDARD, Engine};
let data = "hi!".as_bytes();
println!("{}", STANDARD.encode(data));   // aGkh   (base64 crate)
println!("{}", hex::encode(data));       // 686921 (hex crate)
let back = STANDARD.decode("aGkh").unwrap();
println!("{}", String::from_utf8(back).unwrap()); // hi!
aGkh 686921 hi!
C++ 
// C++ has no standard base64 (use OpenSSL / cppcodec). Hex is trivial:
#include <cstdio>
std::string data = "hi!";
for (unsigned char c : data) std::printf("%02x", c);  // 686921
// base64 of "hi!" is "aGkh"  (cppcodec::base64_rfc4648::encode)
686921

Example 2 — URL-safe base64 for bytes that would produce +//

The bytes [251, 255] encode to +/w= in the standard alphabet — the + and / are unsafe in URLs and filenames. The URL-safe alphabet emits -_ instead, and typically drops the = padding.

Python 
raw = bytes([251, 255])
print(base64.b64encode(raw).decode())         # +/w=
print(base64.urlsafe_b64encode(raw).decode()) # -_w=   (+/ -> -_)
+/w= -_w=
Go 
raw := []byte{251, 255}
fmt.Println(base64.StdEncoding.EncodeToString(raw))    // +/w=
fmt.Println(base64.URLEncoding.EncodeToString(raw))    // -_w=
fmt.Println(base64.RawURLEncoding.EncodeToString(raw)) // -_w  (no padding)
+/w= -_w= -_w
TypeScript 
const raw = Buffer.from([251, 255]);
console.log(raw.toString("base64"));     // +/w=
console.log(raw.toString("base64url"));  // -_w   (url-safe, unpadded)
+/w= -_w
Rust 
use base64::engine::general_purpose::{STANDARD, URL_SAFE_NO_PAD};
use base64::Engine;
let raw = [251u8, 255];
println!("{}", STANDARD.encode(raw));        // +/w=
println!("{}", URL_SAFE_NO_PAD.encode(raw)); // -_w
+/w= -_w
C++ 
// Standard base64 of {0xFB, 0xFF} is "+/w="; URL-safe is "-_w".
// The alphabets differ only in the last two symbols
// ('+' '/' -> '-' '_'), and URL-safe usually drops '=' padding.
unsigned char raw[2] = {0xFB, 0xFF};
for (auto b : raw) std::printf("%02x", b);   // fbff
fbff
Language Comparison: Python has base64 (standard/urlsafe, padded or not) and bytes.hex()/bytes.fromhex(); text→bytes is an explicit str.encode("utf-8"). Go has encoding/base64 (Std/URL/Raw variants) and encoding/hex. Node uses Buffer with "base64"/"base64url"/"hex"/"utf-8" — but browsers use btoa/atob, which operate on Latin-1 "binary strings" and choke on non-ASCII (encode via TextEncoder first). Rust relies on the base64 and hex crates (not in std). C++ gives you hex for free but has no standard base64 — use OpenSSL or cppcodec. Everywhere: text → UTF-8 bytes → base64/hex; base64 and hex work on bytes, never on characters.
Common Mistakes:
❌ [All] Treating base64 as encryption — it's reversible with no key; never use it to "hide" secrets.
✅ [All] Encrypt secrets; use base64/hex only to make bytes text-safe.
❌ [All] Base64-ing a str directly instead of its UTF-8 bytes — implicit ASCII assumptions corrupt non-ASCII text.
✅ [All] Encode text to bytes (UTF-8) first, then base64/hex.
❌ [TypeScript] btoa/atob throw on characters > 0xFF (they assume Latin-1).
✅ [TypeScript] Convert via TextEncoder/Buffer before base64 in the browser.
❌ [Python] Forgetting .decode() — b64encode returns bytes, so it prints with a b'...' prefix.
✅ [Python] .decode() the result to get a real str.
Key Rules:
  • Base64/hex are representations, not encryption — they only make bytes text-safe.
  • Always go text → (UTF-8) bytes → base64/hex, and reverse exactly on the way back.
  • Match alphabet and padding on both ends; use URL-safe base64 (-_) inside URLs and filenames.

Object Semantics

Two questions every language answers differently once objects hold references: when are two things "equal" (and how must that agree with hashing so maps and sets work), and what does it mean to "copy" an object that points at other objects. Getting either wrong causes silent lookup failures and spooky action-at-a-distance.

Equality & Hashing Contracts Object Semantics
Why it matters: hash maps and sets depend on a contract between equality and hashing. If two "equal" objects hash differently, they get lost in a set; if you mutate a key after insertion, you corrupt the table. Every language ties ==/equals to a hash function — break the pairing and lookups silently miss.

The universal contract: if a == b then hash(a) == hash(b) (the converse need not hold — collisions are fine). Value types should compare and hash by content; identity/reference types compare by address. Keys placed in a hash set/map must be immutable (or at least not mutated while stored). And always distinguish identity (is, ===, pointer equality) from value equality.

Example 1 — consistent equality + hashing lets a custom type be a set/map key

Python 
class Point:
    def __init__(self, x, y): self.x, self.y = x, y
    def __eq__(self, o): return (self.x, self.y) == (o.x, o.y)
    def __hash__(self): return hash((self.x, self.y))   # must agree with __eq__
s = {Point(1, 2), Point(1, 2)}
print(len(s), Point(1, 2) in s)   # 1 True
1 True
Go 
// Structs of comparable fields are == and usable as map keys for free:
type Point struct{ X, Y int }
m := map[Point]string{{1, 2}: "a"}
fmt.Println(Point{1, 2} == Point{1, 2}, m[Point{1, 2}]) // true a
true a
TypeScript 
// JS Map/Set key objects by identity (SameValueZero), so equal-by-value
// objects are DIFFERENT keys. Canonicalize to a primitive key instead:
const key = (p: {x: number; y: number}) => `${p.x},${p.y}`;
const s = new Set([key({x:1,y:2}), key({x:1,y:2})]);
console.log(s.size, s.has(key({x:1,y:2})));   // 1 true
1 true
Rust 
use std::collections::HashSet;
#[derive(PartialEq, Eq, Hash)]        // Eq + Hash derived together
struct Point { x: i32, y: i32 }
let mut s = HashSet::new();
s.insert(Point { x: 1, y: 2 });
s.insert(Point { x: 1, y: 2 });
println!("{} {}", s.len(), s.contains(&Point { x: 1, y: 2 })); // 1 true
1 true
C++ 
#include <unordered_set>
struct Point { int x, y; bool operator==(const Point&) const = default; };
struct PointHash { size_t operator()(const Point& p) const {
    return std::hash<int>{}(p.x) ^ (std::hash<int>{}(p.y) << 1); } };
std::unordered_set<Point, PointHash> s;
s.insert({1, 2}); s.insert({1, 2});
std::cout << s.size() << " " << s.count({1, 2});  // 1 1
1 1

Example 2 — identity vs value: two distinct objects with equal contents

Equal contents do not mean the same object. Value equality compares fields; identity compares addresses/references.

Python 
a, b = [1, 2], [1, 2]
print(a == b, a is b)     # True False  (equal value, different identity)
True False
Go 
a := &Point{1, 2}; b := &Point{1, 2}
fmt.Println(*a == *b, a == b)  // true false  (values equal; pointers differ)
true false
TypeScript 
const a = {x: 1}, b = {x: 1};
console.log(a === b, JSON.stringify(a) === JSON.stringify(b)); // false true
false true
Rust 
let a = Point { x: 1, y: 2 };
let b = Point { x: 1, y: 2 };
println!("{} {}", a == b, std::ptr::eq(&a, &b));  // true false
true false
C++ 
Point a{1, 2}, b{1, 2};
std::cout << (a == b) << " " << (&a == &b);  // 1 0  (equal value; different address)
1 0
Language Comparison: Python pairs __eq__ with __hash__ — defining __eq__ alone sets __hash__ to None (the object becomes unhashable), so override both for value types (or use a frozen dataclass). Go gives struct == and map-key usability for free when all fields are comparable (no slice/map/func); there's no custom hash hook — the runtime hashes the bytes. TypeScript/JS Map/Set key objects by reference, so value-equal objects are distinct keys — canonicalize to a string/number or use a wrapper. Rust splits PartialEq/Eq (Eq = total, reflexive equality, required for hash keys) from Hash; derive them together. C++ needs operator== plus a std::hash specialization/functor for the unordered_* containers.
Common Mistakes:
❌ [All] Overriding equality but not hashing (or vice versa) — equal objects land in different buckets and lookups miss.
✅ [All] Define/derive equality and hashing together, over the same fields.
❌ [Python] Defining __eq__ without __hash__ makes instances unhashable.
✅ [Python] Define both, or use @dataclass(frozen=True) (auto-generates a consistent pair).
❌ [Go] Using a struct that contains a slice/map/func as a map key — it won't compile (non-comparable).
✅ [Go] Key on comparable fields only, or derive a canonical string key.
❌ [Rust/C++] Mutating a key's hashed fields while it's stored in a set/map — it corrupts the container.
✅ [Rust/C++] Keep keys immutable while they live in the container.
Key Rules:
  • Contract: a == b ⇒ hash(a) == hash(b); collisions (unequal items sharing a hash) are allowed.
  • Define/derive equality and hashing together, over identical fields.
  • Never mutate a value while it's a live key in a hash set/map; keep identity and value equality distinct in your head.

Shallow vs Deep Copy Object Semantics
Why it matters: copying an object that holds references can silently share inner state — mutate the "copy" and the original changes too. A shallow copy duplicates the top level but shares nested objects; a deep copy recursively clones everything. Choosing wrong causes spooky action-at-a-distance, especially with nested lists/maps and shared defaults.

A shallow copy makes a new container whose elements are the same references as the source. A deep copy recursively duplicates nested structures so the two graphs are fully independent. Value types (numbers, Rust Copy types, C++ values) copy by value and sidestep the issue; reference types need an explicit deep copy when you want independence.

Example 1 — a shallow copy shares the nested list; a deep copy does not

Python 
import copy
orig = [[1, 2], [3, 4]]
shallow = copy.copy(orig)       # top-level copy; inner lists shared
deep = copy.deepcopy(orig)      # fully independent
orig[0][0] = 99                 # mutate a nested element in place
print(shallow[0], deep[0])      # [99, 2] [1, 2]
[99, 2] [1, 2]
Go 
orig := [][]int{{1, 2}, {3, 4}}
shallow := make([][]int, len(orig)); copy(shallow, orig) // shares inner slices
deep := make([][]int, len(orig))
for i := range orig { deep[i] = append([]int{}, orig[i]...) }
orig[0][0] = 99                 // in-place: shared backing array sees it
fmt.Println(shallow[0], deep[0]) // [99 2] [1 2]
[99 2] [1 2]
TypeScript 
const orig = [[1, 2], [3, 4]];
const shallow = [...orig];              // spread = shallow; inner arrays shared
const deep = structuredClone(orig);     // deep clone
orig[0][0] = 99;
console.log(shallow[0], deep[0]);       // [99, 2] [1, 2]
[ 99, 2 ] [ 1, 2 ]
Rust 
use std::rc::Rc;
use std::cell::RefCell;
let inner = Rc::new(RefCell::new(vec![1, 2]));
let shallow = Rc::clone(&inner);        // shared: same allocation
let deep = inner.borrow().clone();      // independent Vec
inner.borrow_mut()[0] = 99;
println!("{:?} {:?}", shallow.borrow(), deep); // [99, 2] [1, 2]
[99, 2] [1, 2]
C++ 
#include <memory>
#include <vector>
std::vector<std::vector<int>> orig{{1, 2}, {3, 4}};
auto deep = orig;                       // vector copy is DEEP (value semantics)
auto shared = std::make_shared<std::vector<int>>(orig[0]);
auto shallow = shared;                  // shared_ptr copy shares the inner vector
(*shared)[0] = 99;
std::cout << shallow->at(0) << " " << deep[0][0]; // 99 1
99 1

Example 2 — the shared-default / aliasing trap

A single mutable object reused as a default (or copied by reference) accumulates state across uses — a notorious bug source. Note how ownership-based languages make accidental sharing hard.

Python 
def add(item, bucket=[]):       # DANGER: one list, created once, shared
    bucket.append(item); return bucket
print(add(1)); print(add(2))    # [1] then [1, 2]  (surprise!)
# fix: default to None, create the list inside the body
[1] [1, 2]
Go 
// A map field is reference-like; copying the struct shares the same map:
type Bag struct{ items map[string]int }
a := Bag{items: map[string]int{}}
b := a                          // shallow struct copy: same underlying map
b.items["x"] = 1
fmt.Println(a.items["x"], b.items["x"]) // 1 1  (aliased!)
1 1
TypeScript 
// A module-level default object aliases across calls:
const shared: number[] = [];
const f = (x: number, b = shared) => (b.push(x), b);
console.log(f(1), f(2));        // [1, 2] [1, 2]  (same array)
[ 1, 2 ] [ 1, 2 ]
Rust 
// Ownership prevents accidental aliasing: passing a Vec moves it,
// copying requires an explicit .clone(), sharing requires Rc/Arc.
let a = vec![1];
let mut b = a.clone();          // explicit independent copy
b.push(2);
println!("{:?} {:?}", a, b);     // [1] [1, 2]
[1] [1, 2]
C++ 
std::vector<int> a{1, 2};
auto byValue = a;               // deep copy (independent)
auto& byRef = a;                // alias (same object)
a[0] = 99;
std::cout << byValue[0] << " " << byRef[0];  // 1 99
1 99
Language Comparison: Python assignment never copies — = just binds a name; use copy.copy (shallow) / copy.deepcopy (deep). TypeScript/JS: spread and Object.assign are shallow; structuredClone (modern) or JSON.parse(JSON.stringify(...)) (lossy) are deep. Go: assigning a struct copies it, but slice/map/pointer/channel fields are reference-like, so the copy shares them (a "shallow struct copy") — deep-copy those fields by hand. Rust: no implicit copy for heap types; Clone is explicit and (for owned types like Vec/String) already deep, with sharing opt-in via Rc/Arc. C++: copy constructor/assignment do member-wise copies — value members (including std::vector/std::string) deep-copy, but raw pointers copy the pointer (shallow), so follow the Rule of Zero/Three/Five.
Common Mistakes:
❌ [All] Assuming a top-level copy is independent when it contains nested references — mutating a nested object hits both copies.
✅ [All] Deep-copy when you need independent nested state; otherwise document that it's shared.
❌ [Python] A mutable default argument (def f(x, acc=[])) is created once and shared across all calls.
✅ [Python] Default to None and create the container inside the function.
❌ [JS/TS] {...obj} / Object.assign copy only one level deep; nested objects stay shared.
✅ [JS/TS] Use structuredClone for a true deep copy.
❌ [C++] The default copy of a class holding a raw owning pointer is shallow → double-free/dangling.
✅ [C++] Own via unique_ptr/vector (Rule of Zero) or write deep-copy semantics.
Key Rules:
  • Shallow copy duplicates the container but shares nested references; deep copy recursively clones the whole graph.
  • Reach for deep copy when the copy must be independently mutable; otherwise shallow (or explicit sharing) is cheaper.
  • Know your language's default: Python = never copies, Go struct-copy is shallow over reference fields, Rust clones are explicit/deep, C++ copies are member-wise.
