Decorators & Middleware Functions & Abstractions 2 days since · rep 0
Wrap functions to add cross-cutting behavior (logging, timing, auth) without modifying the original. Python: @decorator syntax, Go: middleware functions func(http.Handler) http.Handler, TS: stage 3 decorators or HOF wrappers, Rust: procedural macros or wrapping closures, C++: function wrappers and CRTP.
Python
import functools, time

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def slow_sum(n):
    return sum(range(n))

print(slow_sum(1_000_000))

String Formatting & Interpolation I/O & Data Processing

Every program produces text output. Whether you are logging, building user messages, or formatting data for display, knowing how your language formats strings — from simple interpolation to precise numeric alignment — makes the difference between readable and messy output.

String formatting falls into three categories: (1) Interpolation — embed expressions directly in string literals, (2) Format specifiers — control width, precision, padding, and alignment, (3) Custom formatting — implement format traits/protocols so your types display nicely.

Example 1: Basic Interpolation — Embedding Values in Strings

The simplest form: insert variable values directly into a string.

class Duration:
    def __init__(self, seconds):
        self.seconds = seconds

    def __str__(self):
        h, rem = divmod(self.seconds, 3600)
        m, s = divmod(rem, 60)
        return f"{h:02d}:{m:02d}:{s:02d}"

    def __repr__(self):
        return f"Duration({self.seconds})"

    def __format__(self, spec):
        if spec == "short":
            h, rem = divmod(self.seconds, 3600)
            m, _ = divmod(rem, 60)
            return f"{h}h{m}m"
        return str(self)

print(f"Short:    {d:short}")

Language Comparison — String Formatting: Python f-strings are the gold standard for readability — expressions go right inside {}. Go uses printf-style verbs (%s, %d) and the special %v for any value; the Stringer interface controls %v output. TypeScript template literals are simple but lack format specifiers — you need .toFixed(), .padStart(), or Intl for precision. Rust’s format! macro is checked at compile time — mismatched types cause compile errors, not runtime crashes. C++20’s std::format brings Python-like syntax to C++, but older code relies on printf or ostringstream.
Common Mistakes:
  • Python: Using %-formatting with a single tuple argument — "(%s)" % (val,) needs the trailing comma or Python interprets the parens as grouping.
    ❌ "%s" % (1, 2) — TypeError: too many arguments
    ✅ "%s" % ((1, 2),) or just use f-strings
  • Go: Mismatched format verbs — fmt.Printf("%d", "hello") compiles but prints garbage. Use go vet to catch these.
    ❌ fmt.Printf("%d", myString)
    ✅ fmt.Printf("%s", myString)
  • TypeScript: Forgetting that .toFixed() returns a string, not a number — further arithmetic on it will concatenate instead of add.
    ❌ balance.toFixed(2) + 1 → "1234.571"
    ✅ parseFloat(balance.toFixed(2)) + 1
  • Rust: Confusing Display ({}) and Debug ({:?}) — Display is for user-facing output and must be manually implemented; Debug can be derived.
    ❌ Using {} without implementing Display
    ✅ #[derive(Debug)] for dev output, manual Display for user output
  • C++: printf does not check types at compile time (unlike std::format) — passing a std::string to %s is undefined behavior.
    ❌ printf("%s", myString) where myString is std::string
    ✅ printf("%s", myString.c_str()) or use std::format
Key Rules:
  • Use your language’s modern formatting — f-strings (Python), template literals (TS), format! (Rust), std::format (C++20). Avoid older approaches in new code.
  • Implement display traits for custom types — __str__ (Python), String() (Go), toString() (TS), Display (Rust), operator<< (C++). This makes types work naturally in format strings.
  • Separate user-facing from debug output — most languages distinguish between human-readable (Display/__str__) and developer-readable (Debug/__repr__) representations.

Regular Expressions I/O & Data Processing

Regular expressions are the universal tool for pattern matching in text. From validating email formats to extracting data from log files, regexes appear in every domain — knowing the syntax and each language’s API saves hours of manual string parsing.

All languages share the same core regex syntax (\d, \w, [a-z], +, *, ?, () groups) but differ in (1) how you create/compile a regex, (2) how you access match results, and (3) performance characteristics (backtracking vs. linear-time engines).

Language Comparison — Regex Engines: Go and Rust use linear-time regex engines (RE2-based / finite automaton) that guarantee no catastrophic backtracking — but they don’t support look-aheads/look-behinds. Python, TypeScript, and C++ use backtracking engines that support more features but can hang on pathological patterns. Rust’s regex crate is notably fast and safe. C++’s <regex> is the slowest of the five in most benchmarks — consider third-party libraries like RE2 for performance-critical C++ code.
Common Mistakes:
  • Python: Using re.match() when you mean re.search() — match only matches at the start of the string.
    ❌ re.match(r"\d+", "abc 123") → None
    ✅ re.search(r"\d+", "abc 123") → match at position 4
  • Go: regexp.Compile returns an error; regexp.MustCompile panics. Use MustCompile for package-level constants, Compile for user input.
    ❌ regexp.MustCompile(userInput) — panics on bad regex
    ✅ re, err := regexp.Compile(userInput)
  • TypeScript: Forgetting the /g flag with .matchAll() — it throws a TypeError without it. Also, .test() with /g has stateful lastIndex.
    ❌ text.matchAll(/\d+/) — TypeError
    ✅ text.matchAll(/\d+/g)
  • Rust: Regex compilation is slow — compiling in a loop or per-request is a performance killer. Use LazyLock or once_cell::Lazy.
    ❌ fn check(s: &str) { let re = Regex::new(...); } — recompiles every call
    ✅ static RE: LazyLock<Regex> = ...
  • C++: regex_match requires the entire string to match; regex_search finds a match anywhere. Confusing them gives unexpected false negatives.
    ❌ regex_match("abc 123", regex(R"(\d+)")) → false
    ✅ regex_search("abc 123", regex(R"(\d+)")) → true
Key Rules:
  • Compile once, reuse many times — store compiled regex patterns as module/package-level constants.
  • Use raw strings — Python r"...", Go `...`, Rust r"...", C++ R"(...)" to avoid double-escaping backslashes.
  • Prefer named groups for readability — (?P<year>\d{4}) is far clearer than (\d{4}) when extracting multiple fields.

const rawArgs = ["--name", "Alice", "--times", "3", "--loud"];

function parseArgs(args: string[]) {
    const result: Record<string, string | boolean> = {};
    for (let i = 0; i < args.length; i++) {
        if (args[i].startsWith("--")) {
            const key = args[i].slice(2);
            const next = args[i + 1];
            if (!next || next.startsWith("--")) {

                result[key] = next;
                i++;
            }

Language Comparison — CLI Parsing Approaches: Python’s argparse is built-in and battle-tested — it handles help, types, choices, and subcommands out of the box. Go’s flag package is minimal — no subcommands, no positional args. For complex CLIs, use cobra. TypeScript has no standard library for this — commander and yargs are the go-to packages. Common Mistakes:
  • Python: Confusing add_argument("name") (positional) with add_argument("--name") (optional) — the -- prefix changes the behavior entirely.
    ❌ parser.add_argument("verbose") — this is a required positional arg
    ✅ parser.add_argument("--verbose", action="store_true")
  • Go: flag.Parse() must be called before accessing flag values. Also, Go uses single-dash flags (-name) not double-dash.
    ❌ Accessing *name before flag.Parse()
    ✅ Call flag.Parse() first, then access flag values
  • TypeScript: process.argv[0] is the Node binary, [1] is the script path — actual args start at index 2.
    ❌ const name = process.argv[1] — gets the script filename
    ✅ const args = process.argv.slice(2)
  • Rust: clap derive macros generate parse() which reads from real std::env::args — use parse_from() in tests.
    ❌ Using Args::parse() in a test — reads from actual CLI
    ✅ Args::parse_from(["prog", "--name", "test"])
  • C++: argv[argc] is always NULL but argv[0] is the program name — actual arguments start at argv[1].
    ❌ string firstArg = argv[0] — gets the program name
    ✅ for (int i = 1; i < argc; ++i)
Key Rules:
  • Always provide --help — auto-generated or manual. Every CLI tool should explain itself when run with no arguments or --help.
  • Validate early, fail fast — check required arguments and valid values before doing any work. Print clear error messages.
  • Use the ecosystem’s preferred library — argparse (Python), cobra (Go), commander (TS), clap (Rust), CLI11 (C++). Don’t hand-roll parsing for anything beyond trivial scripts.

Production code must handle network errors, non-2xx status codes, and malformed responses gracefully.

import requests

def fetch_user(user_id: int) -> dict:
    url = f"https://jsonplaceholder.typicode.com/users/{user_id}"
    try:
        response = requests.get(url, timeout=5)

func fetchUser(id int) (map[string]interface{}, error) {
    url := fmt.Sprintf("https://jsonplaceholder.typicode.com/users/%d", id)
    resp, err := http.Get(url)
    if err != nil {
        return nil, fmt.Errorf("network error: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusOK {
        return nil, fmt.Errorf("HTTP error: %d", resp.StatusCode)
    }

    var user map[string]interface{}
    if err := json.NewDecoder(resp.Body).Decode(&user); err != nil {
        return nil, fmt.Errorf("JSON decode error: %w", err)
    }
    return user, nil
}

Language Comparison — HTTP Client Ecosystem: Python’s requests is the most user-friendly HTTP library in any language — .json() on the response, json= parameter for sending, automatic encoding. Go’s net/http is built-in but verbose — you must close the body, check errors at every step, and marshal/unmarshal JSON manually. TypeScript’s fetch is now built into Node.js and browsers, making it the most portable option. Note: fetch does not reject on HTTP errors — you must check response.ok. Rust’s reqwest integrates beautifully with serde for typed JSON parsing. C++ has no standard HTTP library — you must choose between libcurl (powerful but C-style), cpp-httplib (header-only), or cpr (modern wrapper).
Common Mistakes:
  • Python: Not calling response.raise_for_status() — requests does not raise on 4xx/5xx by default, so you silently process error responses.
    ❌ data = requests.get(url).json() — may parse a 404 error page
    ✅ r = requests.get(url); r.raise_for_status(); data = r.json()
  • Go: Forgetting defer resp.Body.Close() causes resource leaks — the connection is never returned to the pool.
    ❌ resp, _ := http.Get(url) and forget to close
    ✅ defer resp.Body.Close() immediately after the nil check
  • TypeScript: fetch() only rejects on network errors, not HTTP errors — a 404 resolves successfully.
    ❌ const data = await (await fetch(url)).json() — may parse 404 body
    ✅ Check response.ok or response.status before parsing
  • Rust: Using .unwrap() on HTTP operations in production — network requests can always fail.
    ❌ let data: User = client.get(url).send().unwrap().json().unwrap()
    ✅ Use ? operator and return Result
  • C++: Not setting timeouts — libcurl defaults to no timeout, so a request can hang forever on an unresponsive server.
    ❌ Omitting CURLOPT_TIMEOUT
    ✅ curl_easy_setopt(curl, CURLOPT_TIMEOUT, 10L)
Key Rules:
  • Always set timeouts — never make an HTTP request without a timeout. Network requests can hang indefinitely without one.
  • Check status codes explicitly — a successful send() does not mean the server accepted your request. Always check for 2xx status.
  • Reuse HTTP clients — creating a new client per request is expensive. All languages support persistent clients with connection pooling.
