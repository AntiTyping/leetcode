1. Logging Across Languages Testing & Error Handling rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: A logging facade separates what you record from where it goes. Prefer structured logs (key/value fields, JSON) over interpolated strings so machines can query them, and use levels (debug/info/warn/error) with a configurable threshold. Each ecosystem has a de-facto standard: Python logging, Go log/slog, TS pino/winston, Rust tracing, C++ spdlog. Log at boundaries, never log secrets, and pass a logger rather than reaching for a global.

Key Pattern — a leveled, structured log line

Python 
import logging
logging.basicConfig(level=logging.INFO, format="%(levelname)s %(name)s %(message)s")
log = logging.getLogger("svc")
log.info("user logged in uid=%s", 42)   # %-args are formatted lazily
Go 
import "log/slog"
slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, nil)))
slog.Info("user logged in", "uid", 42)   // structured key/value attrs
TypeScript (pino) 
import pino from "pino";
const log = pino({ level: "info" });
log.info({ uid: 42 }, "user logged in");  // JSON with a fields object
Rust (tracing) 
use tracing::info;
tracing_subscriber::fmt::init();
info!(uid = 42, "user logged in");         // typed fields + message
C++ (spdlog) 
#include "spdlog/spdlog.h"
spdlog::set_level(spdlog::level::info);
spdlog::info("user logged in uid={}", 42);  // fmt-style placeholders
Challenge (all 5 languages):
1
Configure JSON output with a per-request correlation id attached to every line (a child logger / slog.With / tracing span / bound fields), emit one line per level, and set the threshold from an env var so debug is suppressed in production.
#1 Pitfall per Language:
  • Python: using an f-string (log.info(f"...{x}")) formats even when the level is disabled; pass %s args instead. Calling basicConfig twice is a no-op.
  • Go: odd number of slog key/value args logs a !BADKEY; the old log package has no levels — reach for slog.
  • TypeScript: console.log is synchronous and unstructured; leaving pino at debug in prod floods logs; never log tokens/PII.
  • Rust: forgetting to install a subscriber means info! silently produces nothing; log vs tracing need a bridge to interoperate.
  • C++: the default logger isn't thread-safe unless you pick an _mt sink; an unflushed async logger can drop lines on crash — register a flush.
2. Type Systems Deep Dive Testing & Error Handling rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: Two axes describe a type system. Structural vs nominal: does compatibility depend on shape (TS, Go interfaces, C++ concepts, Python Protocol) or on the declared name (Rust traits, most of C++'s class hierarchy)? Gradual vs static: Python/TS layer optional types over dynamic values (erased at runtime), while Go/Rust/C++ check everything at compile time. Inference (Rust's Hindley–Milner-ish, C++ auto, TS flow analysis) reduces annotations without losing safety.

Key Pattern — accept "anything with a read() method"

Python (Protocol — structural) 
from typing import Protocol
class Reader(Protocol):
    def read(self) -> str: ...
def use(r: Reader) -> str: return r.read()   # any object with read() fits
Go (structural interface) 
type Reader interface{ Read() string }
// any type with `Read() string` satisfies Reader implicitly — no "implements"
func use(r Reader) string { return r.Read() }
TypeScript (structural) 
interface Reader { read(): string }
const use = (r: Reader) => r.read();          // shape-compatible objects fit
Rust (nominal traits) 
trait Reader { fn read(&self) -> String; }
impl Reader for File { fn read(&self) -> String { /* ... */ } }  // explicit impl
fn use_it<R: Reader>(r: &R) -> String { r.read() }
C++ (concepts — compile-time duck typing) 
template<class T> concept Reader = requires(T t){ { t.read() } -> std::same_as<std::string>; };
std::string use(Reader auto r){ return r.read(); }
Challenge (all 5 languages):
1
Define a Shape abstraction with an area() method and two implementations (Circle, Rect). Consume them polymorphically both ways where the language allows: static dispatch (generics/templates/impl Trait) and dynamic dispatch (dyn/virtual/interface value), and note which requires an explicit "implements".
#1 Pitfall per Language:
  • Python: type hints are erased — they don't enforce anything at runtime without a checker (mypy/pyright); Protocol needs @runtime_checkable for isinstance.
  • Go: structural interfaces make accidental satisfaction possible; a nil interface holding a typed-nil pointer is not == nil.
  • TypeScript: types vanish at runtime; any silently disables checking; excess-property checks apply to object literals but not to aliased variables.
  • Rust: the orphan rule blocks impl ForeignTrait for ForeignType; dyn Trait requires object safety (no generic methods).
  • C++: pre-concepts template errors are famously unreadable; unconstrained auto/templates accept the wrong shape until instantiation deep in the call.
3. Configuration Management Advanced Patterns rep 2 • 7 days since review
3rd review (on schedule)
Quick Recall: Config layers with a clear precedence — defaults < file < environment < flags — and gets parsed and validated once into a typed object at startup (fail fast on a bad value). The Twelve-Factor rule: environment variables for anything that varies per deploy, secrets never in the repo. Load it at the edge and pass the typed struct down; don't read getenv scattered through the code.

Key Pattern — env override with a typed default

Python (pydantic-settings) 
from pydantic_settings import BaseSettings
class Cfg(BaseSettings):
    port: int = 8080        # env PORT overrides; coerced & validated to int
    debug: bool = False
cfg = Cfg()
Go 
port := 8080
if v := os.Getenv("PORT"); v != "" {
    port, _ = strconv.Atoi(v)     // env beats default; validate the error in real code
}
TypeScript (zod) 
import { z } from "zod";
const Cfg = z.object({ PORT: z.coerce.number().default(8080) });
const cfg = Cfg.parse(process.env);   // coerce strings + validate, throws on bad
Rust (serde + envy) 
#[derive(serde::Deserialize)]
struct Cfg { #[serde(default = "def_port")] port: u16 }
fn def_port() -> u16 { 8080 }
let cfg: Cfg = envy::from_env()?;     // env vars -> typed struct
C++ 
int port = 8080;
if (const char* p = std::getenv("PORT")) port = std::stoi(p);  // manual parse/validate
Challenge (all 5 languages):
1
Load config from three layers — hard-coded defaults, a config.toml/.json file, and environment variables — with env winning. Validate that PORT is in 1..65535 and LOG_LEVEL is one of a known set, and fail at startup with a clear message when a value is invalid.
#1 Pitfall per Language:
  • Python: reading os.environ at import time freezes values before tests can set them; a bare bool(os.getenv("DEBUG")) is truthy for the string "false".
  • Go: ignoring the strconv.Atoi error silently yields 0; os.Getenv can't distinguish "unset" from "empty" (use LookupEnv).
  • TypeScript: every process.env value is string | undefined — Number(process.env.PORT) on unset gives NaN; validate/coerce with a schema.
  • Rust: env::var returns Err for unset and for non-UTF-8; forgetting a #[serde(default)] makes an optional var required.
  • C++: std::getenv returns nullptr when unset — dereferencing it is UB; std::stoi throws on non-numeric input.
4. FFI & Interop Advanced Patterns rep 2 • 6 days since review
3rd review (on schedule)
Quick Recall: The C ABI is the lingua franca: to call across languages you expose a extern "C" symbol (stable name, C calling convention) and describe its signature to the caller. The hard parts are ownership (who frees the memory?), string marshaling (NUL-terminated char* vs length-prefixed), and errors (no exceptions cross the boundary — use return codes). Each language wraps C differently: Python ctypes/cffi, Go cgo, Node ffi-napi/WASM, Rust extern blocks, C++ native.

Key Pattern — call C's abs() from each language

Python (ctypes) 
import ctypes
libc = ctypes.CDLL("libc.so.6")
libc.abs.restype = ctypes.c_int     # declare signature or you get garbage
print(libc.abs(-5))                 # 5
Go (cgo) 
/*
#include <stdlib.h>
*/
import "C"
import "fmt"
func main() { fmt.Println(int(C.abs(C.int(-5)))) }  // 5
TypeScript (ffi-napi) 
import ffi from "ffi-napi";
const libc = ffi.Library("libc", { abs: ["int", ["int"]] });
console.log(libc.abs(-5));          // 5
Rust 
extern "C" { fn abs(x: i32) -> i32; }
fn main() { println!("{}", unsafe { abs(-5) }); }   // 5 — FFI calls are unsafe
C++ 
#include <cstdlib>
#include <iostream>
extern "C" int abs(int);            // C linkage; already ABI-compatible
int main() { std::cout << abs(-5) << "\n"; }        // 5
Challenge (all 5 languages):
1
Write a tiny C function char* greet(const char* name) that mallocs a greeting, plus a free_str(char*). Call it from each language, marshal the string in and out, and free it on the C side — observe who owns the buffer and what happens if you let the language's GC/allocator free foreign memory.
#1 Pitfall per Language:
  • Python: not setting restype/argtypes defaults to int and truncates 64-bit pointers; ctypes won't free memory the C side owns.
  • Go: a Go pointer passed to C must not outlive the call (the moving GC); cgo adds build complexity and per-call overhead — batch across the boundary.
  • TypeScript: ffi-napi is native and version-fragile; a wrong type spec corrupts the stack silently — WASM is the safer modern route.
  • Rust: every FFI call is unsafe; you must uphold C's invariants; a panic unwinding across the boundary is UB — use catch_unwind or extern "C-unwind".
  • C++: without extern "C" the symbol is name-mangled and won't link; throwing a C++ exception through a C frame is undefined.
5. Python: Comprehensions & Dataclasses Language Deep Dive rep 2 • 6 days since review
3rd review (on schedule)
Quick Recall: A comprehension is Python's declarative filter+map in one expression — [f(x) for x in xs if pred(x)] — and comes in list/set/dict/generator flavors (the () generator form is lazy). A @dataclass auto-generates __init__/__repr__/__eq__ from typed fields; add frozen=True for immutability and slots=True to drop the per-instance __dict__. The comparison shows the equivalent filter/map and value-record in each language.

Key Pattern — filter-then-map, plus a lightweight value record

Python 
squares = [x*x for x in nums if x % 2 == 0]     # comprehension = filter + map
from dataclasses import dataclass
@dataclass(frozen=True, slots=True)
class Point: x: int; y: int
Go (explicit loop; struct) 
var squares []int
for _, x := range nums {
    if x%2 == 0 { squares = append(squares, x*x) }   // no comprehension syntax
}
type Point struct{ X, Y int }        // value struct, compared field-by-field
TypeScript 
const squares = nums.filter(x => x % 2 === 0).map(x => x * x);
type Point = { readonly x: number; readonly y: number };  // structural record
Rust 
let squares: Vec<i32> = nums.iter().filter(|&&x| x % 2 == 0).map(|x| x * x).collect();
#[derive(Clone, PartialEq)] struct Point { x: i32, y: i32 }
C++ (C++20 ranges) 
auto squares = nums | std::views::filter([](int x){ return x % 2 == 0; })
                    | std::views::transform([](int x){ return x * x; });  // lazy view
struct Point { int x, y; };          // aggregate; == needs =default in C++20
Challenge (all 5 languages):
1
Build a frequency map (word → count) from a list of words using the most idiomatic construct — a Python dict comprehension / Counter, Go map loop, TS reduce, Rust entry-fold, C++ unordered_map — then turn each record type into an immutable value and confirm two equal-valued instances compare equal.
#1 Pitfall per Language:
  • Python: a generator expression is single-use and lazy — iterating it twice yields nothing; a mutable default in a dataclass needs field(default_factory=...).
  • Go: no comprehensions — resist over-abstracting with generics; a struct with a slice/map field isn't deeply comparable with ==.
  • TypeScript: filter/map allocate intermediate arrays; readonly is compile-time only — it doesn't freeze the object at runtime.
  • Rust: iter() borrows, into_iter() consumes — picking the wrong one causes move/borrow errors; deriving Eq needs all fields Eq.
  • C++: a views pipeline is lazy and dangles if it outlives its source range; pre-C++20 you fall back to std::transform/copy_if.
6. Go: Goroutine Scheduling & Channel Patterns Language Deep Dive rep 2 • 6 days since review
3rd review (on schedule)
Quick Recall: Goroutines are cheap green threads multiplexed onto OS threads by the runtime's M:N scheduler (GOMAXPROCS Ps). Channels are typed, synchronized pipes: unbuffered = rendezvous, buffered = bounded queue. The canonical patterns are fan-out (many workers reading one job channel), fan-in (many producers into one channel), and done/context cancellation. Below, each language runs a bounded worker pool over a set of jobs.

Key Pattern — a bounded worker pool

Go 
var wg sync.WaitGroup
jobs := make(chan int, 100)
for w := 0; w < 3; w++ {                 // 3 workers fan out over one channel
    wg.Add(1)
    go func() { defer wg.Done(); for j := range jobs { process(j) } }()
}
for _, j := range work { jobs <- j }
close(jobs); wg.Wait()                    // close signals workers to finish
Python 
from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(max_workers=3) as ex:
    ex.map(process, work)      # threads share the GIL; use ProcessPool for CPU work
TypeScript 
await Promise.all(work.map(process));   // concurrent I/O on one event-loop thread
// real parallelism needs Worker threads
Rust 
let handles: Vec<_> = work.into_iter()
    .map(|j| std::thread::spawn(move || process(j)))
    .collect();
for h in handles { h.join().unwrap(); }  // OS threads; use rayon/tokio for pools
C++ 
std::vector<std::thread> ts;
for (int j : work) ts.emplace_back(process, j);   // one OS thread per job
for (auto& t : ts) t.join();
Challenge (all 5 languages):
1
Build a pipeline: a producer emits numbers, a pool of 3 workers squares them, and a collector sums the results — with a bounded queue applying backpressure and a cancellation path that stops all stages cleanly (Go context, Python event, JS AbortController, Rust channel drop, C++ stop flag).
#1 Pitfall per Language:
  • Go: a goroutine blocked on a channel nobody reads leaks; forgetting to close a range-d channel deadlocks; sharing loop vars pre-1.22 captured the same variable.
  • Python: a ThreadPool gives no speedup for CPU-bound work (GIL) — use ProcessPoolExecutor; exceptions in a worker surface only when you read the future.
  • TypeScript: Promise.all launches everything at once (no concurrency cap) and rejects on the first failure — use a pool/limit and allSettled when needed.
  • Rust: spawning a thread per job doesn't scale; a moved value can't be shared — wrap in Arc (and Mutex for mutation); join to observe panics.
  • C++: a std::thread that is neither joined nor detached calls std::terminate on destruction; one thread per job exhausts resources.

String Matching Algorithms

Three linear-time substring searches that beat the naive O(nm) scan by never re-examining text they've already matched. KMP precomputes a failure function so a mismatch shifts the pattern smartly; Rabin–Karp compares a rolling hash of each window; the Z-algorithm computes, for every position, how much of the prefix repeats there. All three run in O(n+m).

KMP (Knuth–Morris–Pratt) String Matching
Why it matters: the naive search re-scans the text every time a partial match fails, costing O(nm). KMP precomputes a failure function (the longest proper prefix that is also a suffix, "LPS") so that on a mismatch it slides the pattern forward without moving the text pointer back — guaranteeing O(n+m). It's the backbone of grep-style search and many string-DP setups.

The insight: after matching k characters and then failing, we already know those k characters. The LPS array tells us the longest prefix of the pattern that is also a suffix of what we matched, so we can resume at lps[k-1] instead of restarting. Building the LPS is itself a self-match of the pattern against its own prefix.

Example 1 — build the LPS (failure) array of a pattern

Python 
def build_lps(p):
    lps = [0] * len(p)
    k = 0                                  # length of current longest prefix-suffix
    for i in range(1, len(p)):
        while k and p[i] != p[k]:
            k = lps[k - 1]                 # fall back to a shorter border
        if p[i] == p[k]:
            k += 1
        lps[i] = k
    return lps
print(build_lps("ababaca"))               # [0, 0, 1, 2, 3, 0, 1]
[0, 0, 1, 2, 3, 0, 1]
Go 
func buildLPS(p string) []int {
    lps := make([]int, len(p))
    k := 0
    for i := 1; i < len(p); i++ {
        for k > 0 && p[i] != p[k] {
            k = lps[k-1]
        }
        if p[i] == p[k] {
            k++
        }
        lps[i] = k
    }
    return lps
}
// buildLPS("ababaca") -> [0 0 1 2 3 0 1]
[0 0 1 2 3 0 1]

Example 2 — search: index of the first match (or -1)

Python 
def kmp(text, pat):
    if not pat: return 0
    lps = build_lps(pat)
    k = 0
    for i, ch in enumerate(text):
        while k and ch != pat[k]:
            k = lps[k - 1]                 # text pointer i never moves back
        if ch == pat[k]:
            k += 1
        if k == len(pat):
            return i - k + 1               # start index of the match
    return -1
print(kmp("abxabcabcaby", "abcaby"))       # 6

Language Comparison: The algorithm is identical everywhere; the only friction is how you index characters. Go/Rust/C++ index the string as bytes (text[i] is a byte/u8/char), which is O(1) and correct for ASCII but splits multi-byte UTF-8; Rust makes this explicit by taking &[u8]. Python and TS index by code unit (text[i] is a 1-char string), which reads fine but each access allocates a tiny string in Python. None has KMP in the stdlib — Python's str.find, Go's strings.Index, and C++'s string::find use their own (often SIMD or Boyer–Moore-ish) engines, so hand-rolled KMP matters mainly when you need the failure function itself (periodicity, DP).
Common Mistakes:
❌ [All] On a mismatch, resetting k = 0 instead of k = lps[k-1] — that throws away the partial match and degrades to O(nm).
✅ [All] Fall back through the border chain: while k and mismatch: k = lps[k-1].
❌ [All] Moving the text index backward on a mismatch — KMP's whole point is that i only advances.
✅ [All] Advance i every iteration; only k (the pattern position) rewinds.
❌ [Go/Rust/C++] Byte-indexing a UTF-8 string and expecting per-character matching — a multi-byte rune is several indices.
✅ [Go/Rust/C++] Decode to runes/chars (or match on bytes deliberately) when the input isn't ASCII.
❌ [All] Off-by-one on the match position: returning i instead of i - len(pat) + 1.
✅ [All] The match starts at i - k + 1 when k reaches the pattern length.
Key Rules:
  • LPS[i] = length of the longest proper prefix of p[..i] that is also a suffix — both build and search reuse it.
  • On mismatch: k = lps[k-1] (never 0 blindly); on match: k++. The text pointer only moves forward.
  • Total time O(n+m); the amortized argument mirrors the monotonic stack: k increases at most n times, so it can fall back at most n times.
Practice Exercises (implement in all 5 languages):
1
Return all start indices where the pattern occurs (overlapping allowed): on a full match, record i-m+1 and continue with k = lps[m-1].
2
Shortest period: use n - lps[n-1] to find the smallest repeating unit of a string (and detect whether the whole string is a repetition of it).
3
Longest prefix of s that is also a suffix of s but not the whole string — read it straight off lps[n-1], then chain through the border array to list all such borders.
The failure function is a tiny automaton: it's the same idea as building an Aho–Corasick trie for multi-pattern search. The amortized-k analysis echoes the Monotonic Stack card, and the "border" structure feeds directly into String Periodicity and the Z-Algorithm below.
Rabin–Karp Rolling Hash String Matching
Why it matters: instead of comparing characters, Rabin–Karp compares a hash of each length-m window to the pattern's hash, updating the window hash in O(1) as it slides (a "rolling" hash). Average O(n+m). Its real superpower is multi-pattern and substring-equality queries: once you have a rolling hash you can compare any two substrings in O(1), which powers dedup, plagiarism detection, and hashing-based DP.

Treat the window as a base-B number mod a large prime. Sliding right removes the leading digit (- text[i]*B^(m-1)), shifts left (* B), and adds the new trailing digit. Because hashing can collide, a hash match must be verified with a direct compare — cheap because collisions are rare with a good modulus.

Example 1 — find the first occurrence

Python 
def rabin_karp(text, pat):
    n, m = len(text), len(pat)
    if m == 0: return 0
    if m > n: return -1
    base, mod = 256, 1_000_000_007
    high = pow(base, m - 1, mod)            # B^(m-1) mod p
    ph = th = 0
    for i in range(m):                      # hash pattern + first window
        ph = (ph * base + ord(pat[i])) % mod
        th = (th * base + ord(text[i])) % mod
    for i in range(n - m + 1):
        if ph == th and text[i:i+m] == pat: # verify to reject collisions
            return i
        if i < n - m:                        # roll the window forward
            th = ((th - ord(text[i]) * high) * base + ord(text[i+m])) % mod
    return -1
print(rabin_karp("abracadabra", "cad"))     # 4
4
Go 
func rabinKarp(text, pat string) int {
    n, m := len(text), len(pat)
    if m == 0 { return 0 }
    if m > n { return -1 }
    const base, mod = 256, 1000000007
    high := int64(1)
    for i := 0; i < m-1; i++ { high = high * base % mod }
    var ph, th int64

Language Comparison: The math is identical; the divergence is integer width and signed modulo. Python's ints are arbitrary-precision so overflow is impossible and % is always non-negative — the cleanest. Everyone else must fit in 64 bits: Go/C++ use int64/long long and add + mod before subtracting so the result stays non-negative; Rust uses u64 and the th + MOD - ... trick to avoid unsigned underflow (which would panic in debug). TS has only number (a double) — safe here because 256 * 1e9 < 2^53, but a larger base or mod would silently lose precision, so keep products under 2^53 or use BigInt. Verification (text[i:i+m] == pat) is mandatory in all five because the hash can collide.
Common Mistakes:
❌ [All] Trusting the hash: returning a match on ph == th without a direct compare — hash collisions give false positives.
✅ [All] Always verify a hash hit with an actual substring comparison (still O(n+m) amortized).
❌ [Go/C++/TS] Subtracting the leading digit and going negative before the mod — a signed/precision mess.
✅ [Go/C++/TS] Add + mod (Rust: + MOD) before the subtraction to keep the value non-negative.
❌ [All] A tiny modulus (or hashing without a mod) — frequent collisions turn it back into O(nm), or the value overflows.
✅ [All] Use a large prime modulus (and ideally double hashing) so collisions are astronomically rare.
❌ [TS] Letting base*window exceed 2^53 — silent floating-point rounding corrupts the hash.
✅ [TS] Keep every intermediate product below 2^53, or switch to BigInt.
Key Rules:
  • Rolling update: h = (h - lead*B^(m-1)) * B + next, all mod a large prime; precompute B^(m-1).
  • A hash match is a candidate, not a proof — verify with a real compare.
  • Best when you need many/overlapping patterns or O(1) substring-equality; for a single pattern, KMP avoids collision risk entirely.

Z-Algorithm String Matching
Why it matters: the Z-array gives, for every position i, the length of the longest substring starting at i that matches a prefix of the string. Computed in a single O(n) pass, it solves pattern matching (concatenate pat + sep + text), string periodicity, and counting distinct-prefix occurrences — often with simpler bookkeeping than KMP's failure function.

Maintain the rightmost matched interval [l, r) (the "Z-box"). For a new i inside the box, reuse the already-computed z[i-l] as a head start (capped at r-i), then extend by brute force only past r. Because the right boundary r only ever moves forward, total extension work is O(n).

Example 1 — compute the Z-array

Python 
def z_array(s):
    n = len(s)
    z = [0] * n
    l = r = 0
    for i in range(1, n):
        if i < r:
            z[i] = min(r - i, z[i - l])    # reuse work inside the Z-box
        while i + z[i] < n and s[z[i]] == s[i + z[i]]:
            z[i] += 1                      # extend past the box
        if i + z[i] > r:
            l, r = i, i + z[i]
    if n: z[0] = n                         # convention: whole string
    return z
print(z_array("aabcaab"))                 # [7, 1, 0, 0, 3, 1, 0]

Common Mistakes:
❌ [All] Choosing a separator that can appear in the input — a real character lets a false match bleed across the boundary.
✅ [All] Use a sentinel guaranteed outside the alphabet (NUL, or a unique symbol).
❌ [All] Forgetting to cap the reuse at r - i: z[i] = z[i-l] can overrun the Z-box and read stale matches.

  • Pattern search: run Z over pat + sentinel + text; every z[i] == len(pat) is a match at i - len(pat) - 1.
  • O(n) because r is monotonically increasing — each character is compared at most a constant number of times.

String DP & Palindromes

Two workhorses of string problems: Manacher's algorithm finds the longest palindromic substring in O(n) by mirroring palindrome radii, and the edit-distance / LCS family solves fuzzy string comparison with a 2D dynamic-programming table. Both turn quadratic (or exponential) brute force into a disciplined table sweep.

Manacher's Algorithm String DP & Palindromes
Why it matters: the longest palindromic substring is naively O(n²) (expand around each of the 2n−1 centers) or O(n²) DP. Manacher's algorithm gets it to O(n) by reusing palindrome radii across a mirror center — the same "reuse inside a known box" idea as the Z-algorithm. The #-separator trick unifies odd- and even-length palindromes so there's just one case to handle.

Transform s into ^#a#b#a#$: every real character is flanked by #, so every palindrome in the transformed string has odd length and its radius in transformed coordinates equals the palindrome's real length. Sentinels ^/$ stop the inner expansion without bounds checks. Maintain the rightmost palindrome [center, right]; for a new i, seed p[i] from its mirror p[2*center-i], then expand.

Example — longest palindromic substring in O(n)

Python 
def longest_palindrome(s):
    t = "^#" + "#".join(s) + "#$"          # e.g. "^#b#a#b#a#d#$"
    n = len(t)
    p = [0] * n
    center = right = 0
    for i in range(1, n - 1):
        if i < right:
            p[i] = min(right - i, p[2 * center - i])   # mirror
        while t[i + p[i] + 1] == t[i - p[i] - 1]:
            p[i] += 1                       # expand (sentinels stop it safely)
        if i + p[i] > right:
            center, right = i, i + p[i]
    max_len = center_idx = 0
    for i, v in enumerate(p):
        if v > max_len: max_len, center_idx = v, i
    start = (center_idx - max_len) // 2
    return s[start:start + max_len]
print(longest_palindrome("babad"))         # "bab"
print(longest_palindrome("cbbd"))          # "bb"

Common Mistakes:
❌ [All] Dropping the ^/$ sentinels — the inner while then reads out of bounds at the string ends.
✅ [All] Frame the transformed string with distinct guards so expansion always mismatches at the border.
❌ [All] Botching the coordinate back-map — getting start or max_len off by the # factor of 2.
✅ [All] In transformed coords the radius is the real length; start = (i - radius) / 2.

  • Interleave with # to unify odd/even palindromes; frame with sentinels to skip bounds checks.
  • Seed each radius from its mirror (capped at the right boundary), then expand and push the boundary forward.

Edit Distance & LCS String DP & Palindromes
Why it matters: comparing two strings "fuzzily" — how many single-character edits to turn one into the other (Levenshtein), or how long their longest common subsequence is — is the core of spell-checkers, DNA alignment, diff, and autocorrect. Both are classic 2D dynamic programs: fill a table where each cell depends on three neighbors, in O(nm) time and (with the rolling trick) O(min(n,m)) space.

Edit distance: dp[i][j] is the cost to turn a[..i] into b[..j]. If the last characters match, carry the diagonal; else 1 + min(replace, delete, insert). LCS: matching characters extend the diagonal by 1; otherwise take the better of dropping one character from either string. Both compress to one or two rows because each cell needs only the previous row/diagonal.

Example 1 — Levenshtein edit distance (rolling 1D)

Python 
def edit_distance(a, b):
    m, n = len(a), len(b)
    dp = list(range(n + 1))                # dp[j] = edits for a[:0] -> b[:j]
    for i in range(1, m + 1):
        prev = dp[0]                       # holds dp[i-1][j-1]
        dp[0] = i
        for j in range(1, n + 1):
            cur = dp[j]
            if a[i-1] == b[j-1]:
                dp[j] = prev               # characters match: free
            else:
                dp[j] = 1 + min(prev, dp[j], dp[j-1])  # replace, delete, insert
            prev = cur
    return dp[n]

            prev = cur
        }
    }
    return dp[n]
}

def lcs(a, b):
    m, n = len(a), len(b)
    dp = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if a[i-1] == b[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])
    return dp[m][n]

Language Comparison: The recurrences are identical; the differences are table allocation and the min/max call. Python initializes the first row with list(range(n+1)), Rust with (0..=n).collect(), C++ with std::iota, Go/TS with an explicit fill. Three-way min varies: Python/TS take variadic min/Math.min(a,b,c); Rust chains .min(x).min(y); Go's builtin min (1.21+) is variadic too but shown nested here; C++ needs the braced std::min({a,b,c}). The rolling 1D trick (edit distance) needs the prev/cur dance to keep the diagonal in all five; the 2D LCS is shown full-table for clarity but compresses to two rows the same way.
Common Mistakes:
❌ [All] Forgetting the base row/column: dp[0][j] = j and dp[i][0] = i for edit distance (LCS bases are 0).
✅ [All] Initialize the empty-string row/column explicitly before the sweep.
❌ [All] In the 1D rolling version, overwriting dp[j-1]/dp[j] before saving the diagonal — you need the old dp[j].
✅ [All] Stash cur = dp[j] at the top of each step and feed it into prev next iteration.
❌ [All] Confusing subsequence with substring — LCS need not be contiguous; longest common substring is a different (reset-on-mismatch) recurrence.
✅ [All] LCS keeps skipped characters; substring resets dp to 0 on a mismatch.
❌ [Rust/Go/C++] Indexing UTF-8 by byte and reporting edits over bytes, not characters, for non-ASCII input.
✅ [Rust/Go/C++] Collect to a Vec<char>/rune slice first when the alphabet is Unicode.
Key Rules:
  • Edit distance: match → diagonal; else 1 + min(replace, delete, insert); bases dp[i][0]=i, dp[0][j]=j.
  • LCS: match → diag + 1; else max(up, left); bases are 0. Both are O(nm) time.
  • Compress to O(min(n,m)) space with a rolling row; keep the diagonal via a saved prev. Backtrack the full table only if you need the actual alignment/subsequence.
Practice Exercises (implement in all 5 languages):
1
Reconstruct one actual longest common subsequence string (not just its length) by backtracking the full DP table.
2
Longest common substring (contiguous): change the recurrence to reset to 0 on a mismatch and track the running max.
3
Weighted edit distance: charge different costs for insert/delete/replace (e.g. keyboard-distance for autocorrect) and confirm it still fits the same table.
These are canonical Dynamic Programming tabulations (yesterday's DP cards); the O(min) space trick is the same rolling-row idea as Knapsack. LCS is dual to edit distance (edits = m + n − 2·LCS when only insert/delete are allowed), and the alignment backtrack mirrors Backtracking path reconstruction.
