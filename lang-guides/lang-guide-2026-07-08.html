1. Mocking & Test Doubles Testing & Error Handling rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: A test double stands in for a real collaborator. Stubs return canned answers; spies record how they were called; mocks carry pre-programmed expectations that are verified; fakes are lightweight working implementations (an in-memory DB). The split is dynamic vs static: Python (unittest.mock) and TS (vi.fn/jest.fn) patch at runtime, while Go/Rust/C++ need an interface/trait/virtual seam to substitute. Depend on an abstraction, and prefer a real fake over over-mocking.

Key Pattern — a service depends on a Store; swap in a double

Python 
from unittest.mock import Mock
db = Mock()
db.get.return_value = {"id": 1, "name": "Ada"}   # stub
assert UserService(db).username(1) == "Ada"
db.get.assert_called_once_with(1)                # spy / verify
Go 
type Store interface{ Get(id int) (string, error) }
type fakeStore struct{ calls int }                // hand-rolled fake
func (f *fakeStore) Get(id int) (string, error) { f.calls++; return "Ada", nil }
// inject &fakeStore{} into the code under test; no framework needed
TypeScript (vitest) 
import { vi, expect } from "vitest";
const get = vi.fn().mockReturnValue({ id: 1, name: "Ada" });  // stub + spy
expect(new UserService({ get }).username(1)).toBe("Ada");
expect(get).toHaveBeenCalledWith(1);
Rust (mockall) 
#[automock]
trait Store { fn get(&self, id: u32) -> String; }
let mut m = MockStore::new();
m.expect_get().with(eq(1)).returning(|_| "Ada".into());  // expectation
C++ (GoogleMock) 
class Store { public: virtual ~Store()=default; virtual std::string Get(int)=0; };
class MockStore : public Store {
 public: MOCK_METHOD(std::string, Get, (int), (override));
};
EXPECT_CALL(m, Get(1)).WillOnce(Return("Ada"));

#1 Pitfall per Language:
  • Python: patch where the name is looked up (the importing module), not where it’s defined; a bare Mock() answers any attribute truthily, hiding typos — use spec=/autospec.
  • Go: no built-in mocking — hand-write fakes or use gomock/testify; keep interfaces small and consumer-side or fakes become painful to write.
  • TypeScript: forgetting vi.clearAllMocks() between tests leaks call history; vi.mock is hoisted above imports; ESM module mocking is order-sensitive.
  • Rust: you must design against a trait to mock at all; mockall generates a MockX; the code under test needs a generic or dyn seam for static dispatch.

2. Benchmarking Testing & Error Handling rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: Micro-benchmarks measure isolated operations over many repeated runs and report a stable statistic (median / ns-per-op), never a single timing. Each ecosystem ships a harness that auto-tunes iteration count and guards against dead-code elimination: Go testing.B (the b.N loop), Rust criterion, C++ Google Benchmark, Python timeit/pytest-benchmark, TS tinybench. Always warm up, prevent the optimizer from deleting your work (black_box/DoNotOptimize), and compare relative, not absolute.

Key Pattern — benchmark a function under a harness

Python (timeit) 
import timeit
t = timeit.timeit("sum(range(1000))", number=100_000)
print(f"{t / 100_000 * 1e6:.3f} us/op")   # report per-op, not total
Go 
func BenchmarkSum(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {              // b.N is auto-tuned
        sink = sumTo(1000)                  // assign to a package var
    }
}
// go test -bench=. -benchmem
TypeScript (tinybench) 
import { Bench } from "tinybench";
const bench = new Bench({ time: 500 });    // run each task ~500ms
bench.add("sum", () => { sumTo(1000); });
await bench.run();
console.table(bench.table());              // uses performance.now()
Rust (criterion) 
fn bench(c: &mut Criterion) {
    c.bench_function("sum 1000", |b| b.iter(|| sum_to(black_box(1000))));
}
// black_box stops constant-folding / dead-code elimination; build --release
C++ (Google Benchmark) 
static void BM_Sum(benchmark::State& s){
    for (auto _ : s)
        benchmark::DoNotOptimize(sumTo(1000));   // keep the result alive
}
BENCHMARK(BM_Sum);

3. Python: asyncio Internals & the GIL Language Deep Dive rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: asyncio runs a single-threaded event loop with cooperative multitasking — tasks yield only at await points. The GIL serializes bytecode, so threads give no speedup for CPU-bound work: use asyncio for I/O-bound concurrency and processes (or another language) for CPU work. The comparison below shows the same fan-out in each language’s concurrency model.

Key Pattern — fan out concurrent work and collect results

Python 
import asyncio
async def work(n):
    await asyncio.sleep(n)                  # yields the loop while waiting
    return n * 10
async def main():
    print(await asyncio.gather(work(1), work(2), work(3)))  # [10, 20, 30]
asyncio.run(main())
Go 
// goroutines: true parallelism, no GIL
ch := make(chan int, 3)
for _, n := range []int{1, 2, 3} {
    go func(n int) { time.Sleep(time.Duration(n) * time.Second); ch <- n * 10 }(n)
}
out := []int{}
for i := 0; i < 3; i++ { out = append(out, <-ch) }   // order varies
TypeScript 
const work = (n: number) =>
  new Promise<number>(r => setTimeout(() => r(n * 10), n * 1000));
console.log(await Promise.all([work(1), work(2), work(3)]));  // [10, 20, 30]
// single-threaded event loop, like asyncio
Rust (tokio) 
async fn work(n: u64) -> u64 { sleep(Duration::from_secs(n)).await; n * 10 }
#[tokio::main]
async fn main() {
    let (a, b, c) = tokio::join!(work(1), work(2), work(3));
    println!("{:?}", [a, b, c]);            // [10, 20, 30]
}
C++ 
auto f1 = std::async(std::launch::async, work, 1);   // OS threads
auto f2 = std::async(std::launch::async, work, 2);
auto f3 = std::async(std::launch::async, work, 3);
std::cout << f1.get() << " " << f2.get() << " " << f3.get() << "\n";  // 10 20 30

1
Fetch 3 “URLs” (simulate with sleeps of differing durations) concurrently and print each result as it arrives, not in submission order. Then add a CPU-bound variant and show why Python needs a process pool (GIL) while Go/Rust/C++ scale on threads.
#1 Pitfall per Language:
  • Python: forgetting await creates a coroutine object that never runs (“coroutine was never awaited”); a blocking call inside a coroutine stalls the whole loop.
  • Go: goroutine leak — launching work that blocks forever because nobody reads the channel and there’s no context cancel.

  • Rust: awaiting on the wrong runtime (no #[tokio::main]) panics; blocking (std::thread::sleep) inside async starves the executor — use tokio::time::sleep.
  • C++: forgetting to .get() a std::async future — its destructor may block and silently serialize your “concurrent” work.
4. Go: context Package & reflect Language Deep Dive rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: context.Context carries deadlines, cancellation signals, and request-scoped values across API boundaries — pass it as the first argument and propagate it down. reflect enables runtime type inspection (struct tags, dynamic field walks) when generics aren’t enough. Below, each language times out a slow operation after 2 seconds.

Key Pattern — give up on a slow op after a deadline

Go 
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()
select {
case res := <-work:
    fmt.Println(res)
case <-ctx.Done():
    fmt.Println("timed out:", ctx.Err())   // context deadline exceeded
}
Python 
try:
    result = await asyncio.wait_for(slow(), timeout=2.0)
except asyncio.TimeoutError:
    result = "timed out"                    # wait_for cancels the coroutine
TypeScript 
const ctrl = new AbortController();
const timer = setTimeout(() => ctrl.abort(), 2000);
try { await fetch(url, { signal: ctrl.signal }); }   // AbortController = cancel
finally { clearTimeout(timer); }
Rust (tokio) 
match tokio::time::timeout(Duration::from_secs(2), slow()).await {
    Ok(v) => println!("{v}"),
    Err(_) => println!("timed out"),        // the dropped future is cancelled
}
C++ 
auto fut = std::async(std::launch::async, slow);
if (fut.wait_for(std::chrono::seconds(2)) == std::future_status::ready)
    std::cout << fut.get() << "\n";
else
    std::cout << "timed out\n";             // task keeps running in background

5. Memory Management Deep Dive Advanced Patterns rep 2 • 6 days since review
3rd review (on schedule)
Quick Recall: Three reclamation models. Manual/scope-based: C++ RAII frees at scope end, Rust’s ownership drops the value when it leaves scope. Reference counting: Python frees immediately when the count hits zero (plus a cyclic collector for cycles). Tracing GC: Go and JS periodically find and free unreachable objects. The deterministic camps (C++/Rust) give exact timing; the GC camps trade that away for no dangling pointers.

Key Pattern — a resource whose cleanup runs deterministically at scope end

C++ 
{
    std::vector<int> v(1000);           // heap buffer owned by v
    auto p = std::make_unique<int>(5);  // single heap int
}                                       // both freed HERE (RAII), deterministically
Rust 
struct Res;
impl Drop for Res { fn drop(&mut self) { println!("freed"); } }
{ let _r = Res; }                       // "freed" prints at the closing brace
Python 
class Res:
    def __del__(self): print("freed")
r = Res(); del r                        # refcount hits 0 -> "freed" now
with open("/dev/null") as f: pass       # `with` = deterministic resource release
Go 
f, _ := os.Open("/dev/null")
defer f.Close()                         // defer handles RESOURCES; memory is GC'd
// you never free memory yourself; the GC reclaims unreachable objects
TypeScript 
// GC-managed; no deterministic finalization. TS 5.2 `using` frees resources:
{
  using file = openSync();              // Symbol.dispose called at block end
}                                       // deterministic for RESOURCES, not memory

1

Instrument a destructor/finalizer to observe when (or whether) cleanup fires.
#1 Pitfall per Language:
  • C++: manual new/delete invites double-free, use-after-free, and leaks — prefer RAII and smart pointers; follow the rule of 0/3/5.
  • Rust: ownership is leak-free for trees but not for cycles — Rc<RefCell<T>> cycles leak; break them with Weak.
  • Python: plain refcounting can’t free cycles; the cyclic GC handles them, but a __del__ on a cyclic object can delay or complicate collection.
  • Go: the GC only frees unreachable memory — leaked goroutines, unbounded maps/slices, and lingering references keep memory alive forever.
  • TypeScript: forgotten timers, event listeners, and closures capturing large objects pin memory; use WeakMap/WeakRef for caches that shouldn’t keep keys alive.
6. Event Systems & Reactive Patterns Advanced Patterns rep 2 • 6 days since review
3rd review (on schedule)
Quick Recall: An event system decouples a producer from its consumers: subscribers register callbacks, the emitter emits, and every listener runs. The reactive extension adds streams of events over time with operators (map/filter/debounce) and backpressure. Node ships a first-class EventEmitter; the others assemble one from a list of callbacks (Python/C++), channels (Go), or trait objects (Rust). RxJS is the canonical reactive library.

Key Pattern — subscribe a listener, then emit an event to it

TypeScript 
import { EventEmitter } from "events";
const bus = new EventEmitter();
bus.on("order", (id: number) => console.log("shipped", id));   // subscribe
bus.emit("order", 42);                                          // -> shipped 42
Python 
from collections import defaultdict
bus = defaultdict(list)
def on(evt, fn): bus[evt].append(fn)                 # subscribe
def emit(evt, *a): [fn(*a) for fn in bus[evt]]       # publish
on("order", lambda id: print("shipped", id)); emit("order", 42)
Go 
// channels ARE the event stream; fan out to subscribers
orders := make(chan int)
go func() { for id := range orders { fmt.Println("shipped", id) } }()  // subscriber
orders <- 42                                                            // emit
Rust 
let mut subs: Vec<Box<dyn Fn(i32)>> = Vec::new();
subs.push(Box::new(|id| println!("shipped {id}")));   // subscribe (trait object)
for f in &subs { f(42); }                              // emit
// for real streams: tokio::sync::broadcast or the futures Stream trait
C++ 
#include <functional>
#include <vector>
std::vector<std::function<void(int)>> subs;
subs.push_back([](int id){ std::cout << "shipped " << id << "\n"; });  // subscribe
for (auto& f : subs) f(42);                                           // emit
Challenge (all 5 languages):
1
Build a tiny typed event bus with on/off/emit, then add a reactive operator: a debounce that suppresses bursts of the same event within a time window. Note where unsubscription is needed to avoid the listener-leak (TS/Python/C++) and where a closed channel or dropped receiver handles it (Go/Rust).
#1 Pitfall per Language:
  • TypeScript: forgetting to off/removeListener leaks listeners; Node warns at 10+ on one emitter; a throwing listener on "error" with no handler crashes the process.
  • Python: a hand-rolled bus holding strong references keeps subscribers alive forever — use weakref for object listeners; exceptions in one listener abort the rest unless caught.
  • Go: sending on a channel with no receiver blocks (or deadlocks); closing a channel twice panics; a slow subscriber applies backpressure to the producer.
  • Rust: broadcast channels drop messages for lagging receivers (a Lagged error); trait-object subscribers need consistent lifetimes/'static bounds.
  • C++: a captured this in a lambda subscriber dangles if the object dies before you unsubscribe; no built-in bus, so you own lifetime and thread safety.

Algorithmic Patterns

Three stack- and geometry-flavored patterns that recur across array, string, and grid problems: the monotonic stack that answers “next greater/smaller” queries in a single O(n) pass, interval techniques that sort then sweep to merge or schedule ranges, and 2D grid traversal that treats a matrix as a graph. Each replaces a nested-loop brute force with one ordered sweep.

Monotonic Stack & Next Greater Element Algorithmic Patterns
Why it matters: a stack whose contents stay sorted (increasing or decreasing) answers “the next/previous element greater or smaller than me” for every position in a single O(n) pass instead of the O(n²) double loop. It powers next-greater-element, daily-temperatures, stock-span, largest-rectangle-in-histogram, and trapping-rain-water.

Keep a stack of indices whose values are monotonic. When the incoming element breaks the invariant, pop every element it dominates — each pop is resolved by the current element (that’s its “next greater”). Because each index is pushed once and popped at most once, the total work is O(n) even though there’s an inner while. A decreasing stack finds next-greater; an increasing stack finds next-smaller.

Example 1 — next greater element to the right (value, or -1)

Python 
def next_greater(a):
    res = [-1] * len(a)
    stack = []                             # indices, values decreasing
    for i, x in enumerate(a):
        while stack and a[stack[-1]] < x:   # x resolves everything smaller
            res[stack.pop()] = x
        stack.append(i)
    return res
print(next_greater([2,1,2,4,3]))           # [4, 2, 4, -1, -1]
[4, 2, 4, -1, -1]
Go 
func nextGreater(a []int) []int {
    res := make([]int, len(a))
    for i := range res { res[i] = -1 }
    stack := []int{}                       // indices, values decreasing
    for i, x := range a {
        for len(stack) > 0 && a[stack[len(stack)-1]] < x {
            res[stack[len(stack)-1]] = x
            stack = stack[:len(stack)-1]
        }
        stack = append(stack, i)
    }
    return res
}
// nextGreater([]int{2,1,2,4,3}) -> [4 2 4 -1 -1]
[4 2 4 -1 -1]
TypeScript 
function nextGreater(a: number[]): number[] {
  const res = new Array(a.length).fill(-1);
  const stack: number[] = [];              // indices, values decreasing
  for (let i = 0; i < a.length; i++) {
    while (stack.length && a[stack[stack.length - 1]] < a[i])
      res[stack.pop()!] = a[i];
    stack.push(i);
  }
  return res;
}

Rust 
fn next_greater(a: &[i32]) -> Vec<i32> {
    let mut res = vec![-1; a.len()];
    let mut stack: Vec<usize> = Vec::new(); // indices, values decreasing
    for (i, &x) in a.iter().enumerate() {
        while let Some(&top) = stack.last() {
            if a[top] < x { res[top] = x; stack.pop(); } else { break; }
        }
        stack.push(i);
    }
    res
}

Example 2 — daily temperatures: days until a warmer day (distance)

Python 
def daily_temps(t):
    res = [0] * len(t)
    stack = []                             # indices, temps decreasing
    for i, x in enumerate(t):
        while stack and t[stack[-1]] < x:
            j = stack.pop(); res[j] = i - j
        stack.append(i)
    return res
print(daily_temps([73,74,75,71,69,72,76,73]))   # [1, 1, 4, 2, 1, 1, 0, 0]
[1, 1, 4, 2, 1, 1, 0, 0]

        for len(stack) > 0 && t[stack[len(stack)-1]] < x {
            j := stack[len(stack)-1]; stack = stack[:len(stack)-1]
            res[j] = i - j
        }
        stack = append(stack, i)
    }
    return res
}

Language Comparison: The algorithm is byte-for-byte the same; the only divergence is stack ergonomics. Python/TS use a plain list/array as a stack (append/pop, push/pop); Go slices it by hand (stack[:len(stack)-1]) — there is no stdlib stack; Rust’s Vec gives last()/push/pop but you store usize indices and cast the distance; C++ uses vector::back()/pop_back() (prefer it over std::stack when you also want random access). TS’s pop()! needs the non-null assertion because the type is number | undefined. Storing indices (not values) is what lets the daily-temperatures variant recover the distance.
Common Mistakes:
❌ [All] Pushing values instead of indices when you need positions/distances — you lose the ability to compute i - j.
✅ [All] Store indices; read the value with a[stack.top()] when comparing.
❌ [All] Using <= vs < without deciding how equal elements behave — it changes whether “next greater” means strictly greater.
✅ [All] Pick the comparator deliberately: < for strictly-greater, <= to also pop equals.
❌ [All] Assuming the inner while makes it O(n²) and “optimizing” it away — it’s already amortized O(n) (each index pops once).
✅ [All] Trust the amortized bound: total pushes = total pops = n.
❌ [Rust/C++] Underflow when computing a “previous smaller” distance from an empty stack — using usize/size_t subtraction with no element below.
✅ [Rust/C++] Treat an empty stack as the sentinel (index -1 conceptually); guard before subtracting.
Key Rules:
  • Decreasing stack → next greater; increasing stack → next smaller. Store indices, not values.
  • When the invariant breaks, pop and resolve every dominated element with the current one.
  • Each index is pushed and popped at most once → O(n) total despite the inner loop.

Interval Scheduling & Merging Algorithmic Patterns
Why it matters: a huge class of problems is “given ranges [start, end], merge/select/count them.” The universal move is to sort by the right key (start for merging, end for scheduling) and then sweep once, keeping O(1) state. It underlies calendar merging, meeting-room allocation, activity selection, and range coverage.

Merge overlapping intervals: sort by start, then walk left to right; if the current interval starts at or before the last kept interval’s end, extend that end, otherwise start a new block. Interval scheduling (max non-overlapping): a classic greedy — sort by end, always take the interval that finishes earliest that doesn’t conflict, because finishing early leaves the most room. The two problems look alike but sort by different keys.

Example 1 — merge overlapping intervals

Python 
def merge(intervals):
    intervals.sort(key=lambda p: p[0])          # sort by START
    out = []
    for s, e in intervals:
        if out and s <= out[-1][1]:              # overlaps the last block
            out[-1][1] = max(out[-1][1], e)
        else:
            out.append([s, e])
    return out

func merge(iv [][]int) [][]int {
    sort.Slice(iv, func(i, j int) bool { return iv[i][0] < iv[j][0] })  // by START
    out := [][]int{}
    for _, p := range iv {

        } else {
            out = append(out, []int{p[0], p[1]})
        }
    }
    return out
}

Example 2 — max non-overlapping intervals (greedy, sort by end)

Python 
def max_nonoverlap(intervals):
    intervals.sort(key=lambda p: p[1])          # sort by END time
    count, end = 0, float("-inf")
    for s, e in intervals:
        if s >= end:                             # no conflict with last chosen
            count += 1; end = e
    return count

Language Comparison: The logic is identical; the friction is how you sort by a key and represent a pair. C++ gets merging almost for free because a std::pair sorts lexicographically by first — a bare std::sort is “by start.” Rust’s sort_by_key and Python’s key= are the cleanest; Go and (for the end-key) C++/TS need an explicit comparator closure. Watch TS: sort() with no comparator is lexicographic, so [10,2].sort() misorders numbers — always pass (a,b)=>a[0]-b[0]. Sentinels differ too: -Infinity (TS), float('-inf') (Python), math.MinInt (Go), i32::MIN (Rust), INT_MIN (C++).
Common Mistakes:
❌ [All] Sorting merge by end or scheduling by start — the two problems need different sort keys; swapping them silently gives wrong answers.
✅ [All] Merge overlapping → sort by start; max non-overlapping → sort by end.
❌ [All] Getting the boundary wrong: are touching intervals [1,2],[2,3] overlapping? <= vs < decides.
✅ [All] Define whether endpoints are inclusive up front and pick the comparison to match.
❌ [TS] arr.sort() without a numeric comparator — lexicographic order breaks intervals with multi-digit bounds.
✅ [TS] Always pass an explicit comparator; remember sort mutates in place.
❌ [All] Merging without first sorting — a single unsorted pass misses overlaps that aren’t adjacent.
✅ [All] Sort first; the sweep’s correctness depends on ordered input.
Key Rules:
  • Sort, then sweep once with O(1) state — the entire family is O(n log n) dominated by the sort.
  • Merge: sort by start, extend the last block while start ≤ lastEnd.
  • Scheduling: sort by end, greedily take the earliest-finishing compatible interval.

Matrix & 2D Grid Traversal Algorithmic Patterns
Why it matters: grids are everywhere — images, game boards, maps, DP tables. Two skills cover most of it: walking a matrix in a controlled order (spiral, diagonal, layer-by-layer) using shrinking boundaries, and treating a grid as an implicit graph where each cell has up-to-four neighbors, which turns flood fill, connected components, and shortest-path-on-a-grid into ordinary BFS/DFS.

Boundary walk: keep four bounds (top,bot,left,right) and peel one edge at a time, shrinking the box until it’s empty — the spiral order. Grid as graph: the neighbors of (r,c) are (r±1,c) and (r,c±1); a stack (DFS) or queue (BFS) plus a “visited” marker (often mutating the grid in place) counts islands / floods regions. Always bounds-check before indexing.

Example 1 — spiral order of a matrix

Python 
def spiral(m):
    if not m: return []
    top, bot, left, right = 0, len(m) - 1, 0, len(m[0]) - 1
    res = []
    while top <= bot and left <= right:
        for c in range(left, right + 1): res.append(m[top][c])   # top row ->
        top += 1
        for r in range(top, bot + 1): res.append(m[r][right])    # right col v
        right -= 1
        if top <= bot:
            for c in range(right, left - 1, -1): res.append(m[bot][c])  # bottom <-
            bot -= 1
        if left <= right:
            for r in range(bot, top - 1, -1): res.append(m[r][left])    # left col ^
            left += 1
    return res

def num_islands(grid):
    if not grid: return 0
    rows, cols = len(grid), len(grid[0])
    count = 0
    for sr in range(rows):
        for sc in range(cols):
            if grid[sr][sc] == "1":
                count += 1
                stack = [(sr, sc)]; grid[sr][sc] = "0"      # sink to mark visited
                while stack:
                    r, c = stack.pop()
                    for dr, dc in ((1,0),(-1,0),(0,1),(0,-1)):
                        nr, nc = r + dr, c + dc
                        if 0 <= nr < rows and 0 <= nc < cols and grid[nr][nc] == "1":
                            grid[nr][nc] = "0"; stack.append((nr, nc))
    return count

Language Comparison: Spiral is pure index arithmetic; the real lesson is signed vs unsigned bounds. Rust makes top/bot/left/right and the DFS coordinates i32 (not usize) precisely so a boundary can go to −1 without underflow, casting back to usize only at the index — the same reason C++/Go/TS are fine with plain int/number. For the grid-DFS, everyone stores neighbor deltas in a small dirs array and mutates the grid to mark “visited” (Go/Rust/C++ use a byte/char grid, Python/TS a string cell). The one non-obvious point: bounds-check before indexing, because Rust and C++ ([]) and Go will panic / UB on an out-of-range access, whereas Python/JS would return a surprise or undefined.
Common Mistakes:
❌ [All] In spiral, redrawing the bottom row / left column when the box has collapsed to a single row or column — double-visiting the middle.
✅ [All] Guard the bottom and left passes with if top <= bot and if left <= right after shrinking.
❌ [All] Indexing a neighbor before checking it’s in bounds — off the edge of the grid.
✅ [All] Test 0 ≤ nr < rows && 0 ≤ nc < cols first, then read grid[nr][nc].
❌ [All] Marking a cell visited only when you pop it, not when you push it — the same cell gets enqueued many times (and can blow up).
✅ [All] Mark visited at push time (as here), so each cell enters the frontier once.
❌ [Rust/C++] Recursive grid DFS on a huge region overflows the call stack; unsigned index math underflows at row/col 0.
✅ [Rust/C++] Use an explicit stack (iterative) for big grids; keep coordinates signed while checking bounds.
Key Rules:
  • Boundary walks: maintain four shrinking bounds and peel one edge per pass; guard the last two passes.
  • Grid = implicit graph: neighbors are the 4 (or 8) deltas; BFS/DFS with a visited mark solves flood fill and components.
  • Always bounds-check before indexing; mark visited on push, not pop.

Combinatorics & Randomization

Two number-flavored toolkits. Combinatorics: binomial coefficients via Pascal’s rule and the multiplicative formula, plus nCr under a prime modulus using the modular inverse from yesterday’s number-theory card. Randomized algorithms: using an RNG with provable uniformity — the Fisher–Yates shuffle and reservoir sampling over a stream of unknown length. The cross-language interest is overflow, integer type, and each language’s RNG API.

Combinatorics: nCr, Pascal & Catalan Combinatorics
Why it matters: counting the number of ways — combinations C(n,r), paths in a grid, balanced-parenthesis / BST counts (Catalan) — shows up in probability, DP transitions, and competitive programming. The algorithms are one loop each; the craft is computing them without overflow and, for large answers, modulo a prime via a modular inverse.

Multiplicative formula: C(n,r) = ∏(n-i)/(i+1) for i in 0..r, using the symmetry C(n,r)=C(n,n-r) to minimize iterations. Building it incrementally keeps every partial product an exact integer (each step is a smaller binomial). Pascal’s rule C(n,r)=C(n-1,r-1)+C(n-1,r) gives the triangle by pure addition (no division). Modulo a prime p: multiply the numerator and denominator mod p, then multiply by the modular inverse den^(p-2) mod p (Fermat). Catalan: Cat(n) = C(2n,n)/(n+1).

def nCr_mod(n, r, p):
    if r < 0 or r > n: return 0
    num = den = 1
    for i in range(r):
        num = num * (n - i) % p
        den = den * (i + 1) % p
    return num * pow(den, p - 2, p) % p       # den^(p-2) = den^-1 mod p (Fermat)

}

Language Comparison: Same two loops; the entire story is how big the numbers get. Python has arbitrary-precision ints (and math.comb), so the plain formula never overflows — only the mod-p version needs the inverse, and its 3-arg pow does the fast power for free. TypeScript must switch to BigInt for the modular case (a product of two ~10⁹ values blows past 2⁵³). Go/Rust/C++ live in fixed width: in the mod-p code, num*den with both < P ≈ 10⁹ reaches ~10¹⁸, which fits in int64/u64/long long (max ~9.2×10¹⁸) but would overflow a 32-bit int — the recurring “widen before you multiply mod” rule. The plain multiplicative nCr in the fixed-width languages still overflows for large n even though the answer fits; that’s exactly when you move to the mod-p form.
Common Mistakes:
❌ [All] Computing n! / (r!(n-r)!) directly — the factorials overflow astronomically even when C(n,r) is tiny.
✅ [All] Use the incremental multiplicative form result * (n-i) / (i+1); each partial product is an exact, small binomial.
❌ [All] Dividing before multiplying ((n-i)/(i+1) first) — integer division truncates and the result is wrong.
✅ [All] Multiply into the running result then divide; result*(n-i) is always divisible by i+1.
❌ [TS] Doing the modular version with number — num*den exceeds 2⁵³ and loses precision silently.
✅ [TS] Use BigInt throughout the mod-p path; don’t mix number and bigint.
❌ [Go/Rust/C++] Using a 32-bit int for num*den % P — the pre-mod product overflows near P².
✅ [Go/Rust/C++] Use 64-bit (int64/u64/long long); for a modulus ≥ 2³² widen further to __int128/u128.
Key Rules:
  • Use the incremental multiplicative formula with the C(n,r)=C(n,n-r) symmetry; multiply before dividing.
  • Pascal’s rule builds the triangle with additions only — no division, good for many small queries.
  • For big answers mod a prime, precompute factorials mod p and use a Fermat modular inverse den^(p-2).

Reservoir Sampling & Fisher–Yates Shuffle Combinatorics
Why it matters: two tiny randomized algorithms with provable uniformity. The Fisher–Yates shuffle produces every permutation with equal probability in O(n) and O(1) extra space; reservoir sampling picks k items uniformly from a stream whose length you don’t know in advance (log lines, a huge file, an iterator) in one pass and O(k) memory. Both are exactly where a naive approach is subtly biased.

Fisher–Yates: walk from the last index down to 1; at index i pick a random j in [0, i] (inclusive) and swap. Every permutation becomes equally likely; the common bug is choosing j in [0, n) (the whole range), which is biased. Reservoir (k=1): keep the current pick; for the element at index i (0-based), replace the pick with probability 1/(i+1). By induction every seen element ends up chosen with equal probability. For k>1, keep the first k, then for the i-th element (i≥k) replace a random reservoir slot with probability k/(i+1).

Example 1 — Fisher–Yates in-place unbiased shuffle

import random
def shuffle(a):
    for i in range(len(a) - 1, 0, -1):
        j = random.randint(0, i)            # inclusive 0..i  (NOT 0..n-1)
        a[i], a[j] = a[j], a[i]
    return a

Example 2 — reservoir sampling: one uniform pick from a stream

Python 
import random
def reservoir(stream):
    chosen = None
    for i, x in enumerate(stream):          # i = 0, 1, 2, ...
        if random.randint(0, i) == 0:       # keep x with probability 1/(i+1)
            chosen = x
    return chosen
# one pass, O(1) memory, unknown stream length

Language Comparison: The algorithms are identical; the divergence is the RNG API and its quality. Python (random) and Go (math/rand) expose an inclusive/half-open integer draw directly (randint(0,i) inclusive, Intn(i+1) half-open) and both ship a stdlib shuffle. TS only has Math.random() (a float in [0,1)), so you scale with Math.floor(rnd*(i+1)) — and there’s no cryptographic guarantee (use crypto.getRandomValues if it matters). Rust needs the external rand crate (gen_range(0..=i)) and C++ composes a generator (mt19937) with a distribution (uniform_int_distribution) — never the biased rand()%n, and never the removed std::random_shuffle. All five must seed once (not per call) and, for security, swap in a CSPRNG.
Common Mistakes:
❌ [All] Fisher–Yates picking j from the whole array [0, n) each step — a classic bias that makes some permutations more likely.
✅ [All] Pick j in [0, i] (inclusive) where i is the shrinking upper bound.
❌ [All] “Shuffling” by sorting with a random comparator (sort(() => Math.random()-0.5)) — not uniform and implementation-dependent.
✅ [All] Use Fisher–Yates (or the stdlib shuffle), which is provably uniform.
❌ [C++] rand() % n for the index — low-bit bias and modulo skew; std::random_shuffle was removed in C++17.
✅ [C++] Use <random>: an engine (mt19937) plus uniform_int_distribution, or std::shuffle.
❌ [All] Re-seeding the RNG inside the loop (or on every call) — correlated or repeated “random” sequences.
✅ [All] Seed the generator once; reuse it. For security-sensitive picks use a CSPRNG, not the default PRNG.
Key Rules:
  • Fisher–Yates: for i from n-1 down to 1, swap a[i] with a[rand(0..=i)] — O(n), uniform, in place.
  • Reservoir (k=1): replace the pick with probability 1/(i+1) at the 0-based i-th item; O(1) memory, one pass, unknown length.
  • Never shuffle via a random-comparator sort or draw indices with rand()%n; both are biased.
