1. Closures & Lambdas Functions rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: A closure is a function plus the environment it captures. The subtlety is what it captures and when: Python/JS/Go capture variables by reference (late binding — a loop variable read at call time bites everyone), while Rust move and C++ [=] capture by value. A factory that returns a closure is the canonical shape.

Key Pattern — an adder factory: make_adder(5) returns a function that adds 5

Python 
def make_adder(n):
    return lambda x: x + n     # captures n by reference
add5 = make_adder(5)
print(add5(3))                 # 8
8
Go 
func makeAdder(n int) func(int) int {
    return func(x int) int { return x + n }
}
add5 := makeAdder(5)
fmt.Println(add5(3))           // 8
8
TypeScript 
const makeAdder = (n: number) => (x: number) => x + n;
const add5 = makeAdder(5);
console.log(add5(3));          // 8
8
Rust 
fn make_adder(n: i32) -> impl Fn(i32) -> i32 {
    move |x| x + n             // move: n is owned by the closure
}
let add5 = make_adder(5);
println!("{}", add5(3));       // 8
8
C++ 
auto makeAdder = [](int n) {
    return [n](int x) { return x + n; };   // [n] captures by value
};
auto add5 = makeAdder(5);
std::cout << add5(3);          // 8
8

Build a memoize(fn) that wraps a single-argument pure function and caches results in a closed-over map. Then create three closures in a for i in 0..3 loop that each print i; show which languages print 0 1 2 versus 3 3 3, and fix the broken ones by binding a fresh per-iteration copy.
#1 Pitfall per Language:
  • Python: closures created in a loop capture the loop variable by reference — all see its final value; bind it via a default arg lambda x, n=n: ....
  • Go: pre-1.22 a for loop reuses one variable, so goroutines/closures share it — copy i := i inside the loop (or use Go 1.22+ per-iteration semantics).
  • TypeScript: var in a loop is function-scoped — use let for a fresh binding each iteration.
  • Rust: a move closure takes ownership — a captured non-Copy value can't be used afterward; clone or borrow deliberately.
  • C++: capturing by reference [&] a local that outlives the scope dangles — capture by value [=] (or [n]) when the closure escapes.
2. Enums & Algebraic Data Types Type System rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: A sum type says "one of these shapes." The everyday one is Option/Maybe — a value that's present or absent, encoded in the type instead of a magic null. Rust Option<T> and C++ std::optional<T> are first-class; Python uses Optional[T], Go returns (T, ok), and TS models it with a tagged union.

Key Pattern — half(n) returns a value only when n is even

Python 
from typing import Optional
def half(n: int) -> Optional[int]:
    return n // 2 if n % 2 == 0 else None
print(half(10), half(3))       # 5 None
5 None
Go 
func half(n int) (int, bool) {
    if n%2 == 0 { return n / 2, true }
    return 0, false            // no value; ok is false
}
v, ok := half(10); fmt.Println(v, ok)  // 5 true
5 true
TypeScript 
type Option<T> = { tag: "some"; value: T } | { tag: "none" };
const half = (n: number): Option<number> =>
  n % 2 === 0 ? { tag: "some", value: n / 2 } : { tag: "none" };
console.log(half(10), half(3));
// { tag: 'some', value: 5 } { tag: 'none' }
{ tag: 'some', value: 5 } { tag: 'none' }
Rust 
fn half(n: i32) -> Option<i32> {
    if n % 2 == 0 { Some(n / 2) } else { None }
}
println!("{:?} {:?}", half(10), half(3));  // Some(5) None
Some(5) None
C++ 
#include <optional>
std::optional<int> half(int n) {
    if (n % 2 == 0) return n / 2;
    return std::nullopt;
}
// half(10).value() == 5 ; half(3).has_value() == false
5 ; false
Challenge (all 5 languages):
1
Define a data-carrying Shape sum type (Circle(r) | Rect(w, h)) and an area that matches each variant. Add a third variant Triangle(b, h) and observe which languages force you to handle it at compile time (Rust exhaustive match, C++ std::visit) versus silently falling through (Go type switch, Python match with _).
#1 Pitfall per Language:
  • Python: Optional[T] is just T | None — the type checker won't stop you from forgetting the None case at runtime; guard it.
  • Go: there are no real enums — iota constants are plain int, so an out-of-range value is representable; validate at the boundary.
  • TypeScript: prefer a discriminated union (tag) over enum; numeric enums allow arbitrary numbers to be assigned.
  • Rust: calling .unwrap() on None panics — use match, if let, or unwrap_or.
  • C++: reading a valueless optional via *opt is UB — check has_value() or use value() (which throws).
3. Range-based Iteration Iteration rep 3 • 16 days since review
Due today • 4th review (overdue 1 day)
Quick Recall: Iterate over elements, not indices, when you can — and pair each element with its position via enumerate (Python/Rust), range (Go), .entries() (JS/TS), or C++23 views::enumerate. Iterating a string yields bytes or code points depending on the language — a classic gotcha.

Key Pattern — iterate with index over the characters of "abc"

Python 
for i, c in enumerate("abc"):
    print(i, c)
# 0 a / 1 b / 2 c
0 a 1 b 2 c
Go 
for i, c := range "abc" {      // i is the BYTE offset, c is a rune
    fmt.Println(i, string(c))
}
// 0 a / 1 b / 2 c
0 a 1 b 2 c
TypeScript 
for (const [i, c] of [..."abc"].entries()) {
  console.log(i, c);
}
// 0 a / 1 b / 2 c
0 a 1 b 2 c
Rust 
for (i, c) in "abc".chars().enumerate() {
    println!("{} {}", i, c);
}
// 0 a / 1 b / 2 c
0 a 1 b 2 c
C++ 
std::string s = "abc";
for (std::size_t i = 0; i < s.size(); ++i)
    std::cout << i << " " << s[i] << "\n";
// C++23: for (auto [i, c] : s | std::views::enumerate)
0 a 1 b 2 c
Challenge (all 5 languages):
1
Iterate over "café" (the e plus a combining accent) and over "naïve", printing each iterated unit and the total count. Explain why the byte length, code-point count, and human-perceived character count differ, and show how each language exposes bytes vs code points vs grapheme clusters.
#1 Pitfall per Language:
  • Python: range(len(x)) then x[i] is un-Pythonic and error-prone — iterate the elements directly (or enumerate).
  • Go: for i, c := range s gives i as a byte index (jumps by rune width), not a sequential 0,1,2 — surprising for multibyte text.
  • TypeScript: for...in iterates keys (as strings), not values — use for...of / .entries().
  • Rust: .chars() yields Unicode scalar values, not grapheme clusters — use the unicode-segmentation crate for user-visible characters.
  • C++: indexing s[i] walks raw chars (bytes) — UTF-8 multibyte code points span several indices.
4. Higher-Order Functions Functions rep 3 • 16 days since review
Due today • 4th review (overdue 1 day)
Quick Recall: Functions that take or return functions: map transforms, filter selects, reduce/fold collapses. The trio expresses most data pipelines declaratively. Rust/C++ keep it lazy via iterator adapters/views; Python favors comprehensions; Go (pre-generics-in-stdlib) still often uses an explicit loop.

Key Pattern — sum of squares of the even numbers in [1..5] (→ 4 + 16 = 20)

Python 
nums = [1, 2, 3, 4, 5]
print(sum(x * x for x in nums if x % 2 == 0))  # 20
20
Go 
nums := []int{1, 2, 3, 4, 5}
sum := 0
for _, x := range nums {
    if x%2 == 0 { sum += x * x }
}
fmt.Println(sum)               // 20
20
TypeScript 
const nums = [1, 2, 3, 4, 5];
console.log(
  nums.filter(x => x % 2 === 0).map(x => x * x).reduce((a, b) => a + b, 0)
);                             // 20
20
Rust 
let nums = [1, 2, 3, 4, 5];
let s: i32 = nums.iter().filter(|&&x| x % 2 == 0).map(|x| x * x).sum();
println!("{}", s);             // 20
20
C++ 
#include <ranges>
#include <numeric>
std::vector<int> nums{1, 2, 3, 4, 5};
auto sq = nums | std::views::filter([](int x){ return x % 2 == 0; })
               | std::views::transform([](int x){ return x * x; });
int s = 0; for (int v : sq) s += v;
std::cout << s;                 // 20
20
Challenge (all 5 languages):
1
Implement group_by(items, keyFn) returning a map from key to the list of items, using only higher-order building blocks (no manual index loops where the language offers reduce/fold). Then write pipe(f, g, h) that returns a single composed function and use it to build a small transform pipeline.
#1 Pitfall per Language:
  • Python: chaining map()/filter() returns one-shot iterators — consume once, or just use a comprehension for clarity.
  • Go: no lazy map/filter in the core stdlib — the explicit loop is idiomatic; reaching for generic helpers can hurt readability.
  • TypeScript: reduce without an initial value throws on an empty array — always pass the seed.
  • Rust: adapters are lazy — nothing runs without a consuming call (sum, collect, for); a bare chain warns "unused".
  • C++: a view referencing a temporary range dangles once the temporary dies — keep the source alive while iterating.
5. Interfaces & Traits Functions & Abstractions rep 3 • 16 days since review
Due today • 4th review (overdue 1 day)
Quick Recall: Interfaces/traits describe behavior a type provides. Go interfaces and Python Protocols are structural (satisfied implicitly by shape); Rust traits, TS implements, and C++ abstract base classes are declared. They enable polymorphism: code against the abstraction, not the concrete type.

Key Pattern — a Shape abstraction with an area() method, implemented by Circle

Python 
from typing import Protocol
class Shape(Protocol):
    def area(self) -> float: ...
class Circle:
    def __init__(self, r): self.r = r
    def area(self) -> float: return 3.14159 * self.r ** 2
print(round(Circle(2).area(), 2))   # 12.57
12.57
Go 
type Shape interface{ Area() float64 }
type Circle struct{ R float64 }
func (c Circle) Area() float64 { return 3.14159 * c.R * c.R }
var s Shape = Circle{R: 2}      // implicit: Circle satisfies Shape
fmt.Printf("%.2f\n", s.Area())  // 12.57
12.57
TypeScript 
interface Shape { area(): number }
class Circle implements Shape {
  constructor(private r: number) {}
  area(): number { return Math.PI * this.r ** 2; }
}
console.log(new Circle(2).area().toFixed(2)); // 12.57
12.57
Rust 
trait Shape { fn area(&self) -> f64; }
struct Circle { r: f64 }
impl Shape for Circle {
    fn area(&self) -> f64 { std::f64::consts::PI * self.r * self.r }
}
println!("{:.2}", Circle { r: 2.0 }.area()); // 12.57
12.57
C++ 
struct Shape { virtual double area() const = 0; virtual ~Shape() = default; };
struct Circle : Shape {
    double r;
    explicit Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
};
std::cout << Circle{2}.area();  // 12.5664
12.5664
Challenge (all 5 languages):
1
Define Shape with area() and name(), implement Circle, Rect, Square, put them in a heterogeneous collection, and print each name+area polymorphically. In Rust/C++ show both static dispatch (generics/templates) and dynamic dispatch (dyn Shape / Shape*) and note the trade-off.
#1 Pitfall per Language:
  • Python: a Protocol is checked by the type checker, not at runtime — a missing method only fails when actually called (unless @runtime_checkable).
  • Go: a nil concrete pointer stored in an interface is not == nil — the classic "typed nil" interface trap.
  • TypeScript: implements is compile-time only; structural typing means an unrelated object with the same shape also satisfies the interface.
  • Rust: a dyn Trait must be behind a pointer (Box<dyn Shape>, &dyn Shape) and the trait must be object-safe.
  • C++: deleting a derived object through a base pointer without a virtual destructor is UB — always declare it virtual in polymorphic bases.
6. Assertion Patterns & Custom Matchers Testing & Error Handling rep 2 • 14 days since review
Due today • 3rd review (overdue 8 days)
Quick Recall: An assertion states an expected fact and fails loudly with a diagnostic when it's false. Test frameworks add matchers (toEqual, assert_eq!) that print both actual and expected on failure. Keep assert for programmer invariants (they can be compiled out), and use framework matchers for tests.

Key Pattern — assert equality with a helpful failure message

Python 
def add(a, b): return a + b
assert add(2, 2) == 4, f"expected 4, got {add(2, 2)}"
# pytest rewrites `assert add(2,2) == 5` to show both sides
(no output; passes)
Go 
func TestAdd(t *testing.T) {
    if got := add(2, 2); got != 4 {
        t.Errorf("add(2,2) = %d, want %d", got, 4)
    }
}
PASS
TypeScript 
import { expect, test } from "vitest";
test("add", () => {
  expect(add(2, 2)).toBe(4);       // toEqual for deep equality
});
✓ add
Rust 
#[test]
fn test_add() {
    assert_eq!(add(2, 2), 4, "add is broken");
    // assert_eq! prints `left` and `right` on failure
}
test test_add ... ok
C++ 
#include <gtest/gtest.h>
TEST(AddTest, Basic) {
    EXPECT_EQ(add(2, 2), 4);        // ASSERT_EQ aborts the test on fail
}
[ PASSED ] 1 test.
Challenge (all 5 languages):
1
Write a custom matcher/helper assert_close(a, b, eps) for floating-point equality within a tolerance, and one assert_raises(fn, ErrType) that passes only when fn throws/returns the expected error. Wire them into the native test runner so a failure prints both operands and the tolerance.
#1 Pitfall per Language:
  • Python: assert is stripped when running with -O — never use it for input validation in production paths, only invariants/tests.
  • Go: t.Error continues the test; t.Fatal stops it — using Error where you needed Fatal can cause cascading nil-deref panics.
  • TypeScript: toBe is reference/identity equality; use toEqual for structural comparison of objects/arrays.
  • Rust: comparing floats with assert_eq! fails on rounding — assert on an epsilon difference instead.
  • C++: ASSERT_* returns from the current function on failure — in a helper it won't stop the caller; prefer EXPECT_* or propagate.

Time & Numeric Precision

Three sources of subtle, high-severity bugs: mixing up wall-clock time with elapsed time, letting time zones corrupt dates, and trusting binary floating point with money. Each language ships a different toolkit — and a different set of traps.

Date & Time Handling Time
Why it matters: dates are deceptively hard — time zones, DST transitions, leap years, and a dozen string formats. Getting it wrong double-books meetings, mis-bills customers, and corrupts logs. The rule of thumb across every language: store and compute in UTC, convert to local time only for display, and always carry an explicit offset/zone.

A timestamp is best modeled as a timezone-aware instant. Construct it explicitly, serialize with ISO 8601 / RFC 3339 (which encodes the offset), do arithmetic with typed durations, and subtract two instants to get an elapsed span. Never build a datetime from parts without a zone — a "naive" datetime silently assumes local time and breaks the moment it crosses a machine boundary.

Example 1 — build a UTC instant, format it, add 3 days, and take the difference

Python 
from datetime import datetime, timezone, timedelta
now = datetime(2026, 6, 29, 12, 0, tzinfo=timezone.utc)
print(now.isoformat())              # 2026-06-29T12:00:00+00:00
later = now + timedelta(days=3)
print((later - now).days)           # 3
2026-06-29T12:00:00+00:00 3
Go 
t := time.Date(2026, time.June, 29, 12, 0, 0, 0, time.UTC)
fmt.Println(t.Format(time.RFC3339))     // 2026-06-29T12:00:00Z
later := t.AddDate(0, 0, 3)
fmt.Println(int(later.Sub(t).Hours()) / 24) // 3
2026-06-29T12:00:00Z 3
TypeScript 
const now = new Date("2026-06-29T12:00:00Z");
console.log(now.toISOString());         // 2026-06-29T12:00:00.000Z
const later = new Date(now.getTime() + 3 * 86_400_000);
console.log((later.getTime() - now.getTime()) / 86_400_000); // 3
2026-06-29T12:00:00.000Z 3
Rust 
use chrono::{TimeZone, Utc, Duration};
let now = Utc.with_ymd_and_hms(2026, 6, 29, 12, 0, 0).unwrap();
println!("{}", now.to_rfc3339());       // 2026-06-29T12:00:00+00:00
let later = now + Duration::days(3);
println!("{}", (later - now).num_days()); // 3
2026-06-29T12:00:00+00:00 3
C++ 
#include <chrono>
#include <format>
using namespace std::chrono;
sys_days d = 2026y / June / 29;         // a UTC calendar day
auto later = d + days{3};
std::cout << std::format("{:%Y-%m-%d}\n", d);   // 2026-06-29
std::cout << (later - d).count();               // 3
2026-06-29 3

Example 2 — parse a string, then convert a UTC instant to a fixed offset zone

Parsing must know the input's zone. Below we parse an RFC 3339 timestamp and render the same instant in UTC+2 — the wall-clock digits shift by two hours but the underlying instant is unchanged.

Python 
from datetime import datetime, timezone, timedelta
t = datetime.fromisoformat("2026-06-29T12:00:00+00:00")
berlin = t.astimezone(timezone(timedelta(hours=2)))
print(berlin.isoformat())           # 2026-06-29T14:00:00+02:00
2026-06-29T14:00:00+02:00
Go 
t, _ := time.Parse(time.RFC3339, "2026-06-29T12:00:00Z")
zone := time.FixedZone("UTC+2", 2*3600)
fmt.Println(t.In(zone).Format(time.RFC3339)) // 2026-06-29T14:00:00+02:00
2026-06-29T14:00:00+02:00
TypeScript 
const t = new Date("2026-06-29T12:00:00Z");
// same instant, rendered for a +02:00 zone:
console.log(t.toLocaleString("en-GB", { timeZone: "Europe/Berlin" }));
// 29/06/2026, 14:00:00
29/06/2026, 14:00:00
Rust 
use chrono::{DateTime, Utc, FixedOffset};
let t: DateTime<Utc> = "2026-06-29T12:00:00Z".parse().unwrap();
let plus2 = FixedOffset::east_opt(2 * 3600).unwrap();
println!("{}", t.with_timezone(&plus2).to_rfc3339());
// 2026-06-29T14:00:00+02:00
2026-06-29T14:00:00+02:00
C++ 
#include <chrono>
#include <format>
using namespace std::chrono;
sys_seconds t{sys_days{2026y/June/29} + 12h};        // 12:00 UTC
auto local = t + 2h;                                 // shift to UTC+2 wall time
std::cout << std::format("{:%Y-%m-%dT%H:%M:%S}+02:00", local);
// 2026-06-29T14:00:00+02:00
2026-06-29T14:00:00+02:00
Language Comparison: Python's datetime splits into naive vs aware — always attach tzinfo (use zoneinfo for named zones). Go bakes a monotonic reading into time.Time and uses the famous reference layout 2006-01-02 15:04:05 instead of %Y-%m-%d. TypeScript's legacy Date is a UTC millisecond instant with a clunky API — the modern answer is the Temporal proposal or a library (Luxon/date-fns). Rust has no date type in std — chrono (or time) is the ecosystem standard, with types that separate DateTime<Utc> from DateTime<Local>. C++20 finally added calendar/time-zone support to <chrono> with strong types (sys_days, zoned_time) and std::format spec strings.
Common Mistakes:
❌ [All] Storing local wall-clock times without an offset — they become ambiguous or wrong across DST and machine boundaries.
✅ [All] Store instants in UTC (or with an explicit offset); convert to local only at the display edge.
❌ [Python] Mixing naive and aware datetimes (aware - naive) raises TypeError, or worse assumes local time silently.
✅ [Python] Keep everything aware: datetime.now(timezone.utc), and use zoneinfo.ZoneInfo("Europe/Berlin") for real zones.
❌ [Go] Using %Y-%m-%d style formats — Go ignores them; only the reference date Mon Jan 2 15:04:05 2006 works.
✅ [Go] Format with the reference layout: t.Format("2006-01-02").
❌ [TypeScript] new Date("2026-06-29") parses as UTC midnight, but new Date(2026, 5, 29) uses local time (and months are 0-based!).
✅ [TypeScript] Prefer full ISO strings with an explicit Z/offset, or use Temporal/a library.
Key Rules:
  • Store/compute in UTC; convert to local only for display; always carry an explicit offset or named zone.
  • Serialize with ISO 8601 / RFC 3339 — it is unambiguous and round-trips across languages.
  • Do arithmetic with typed durations, not by adding raw seconds/milliseconds by hand (DST and leap seconds bite).

Monotonic Clocks & Durations Time
Why it matters: to measure how long something took, or to implement a timeout, you must use a monotonic clock — one that only moves forward and never jumps. The wall clock can leap backward when NTP corrects it or DST shifts, producing negative durations, phantom timeouts, and "operation finished before it started" bugs.

Two different clocks solve two different problems. The wall clock answers "what time is it?" (calendar, logs, scheduling). The monotonic clock answers "how much time has elapsed?" (benchmarks, timeouts, rate limits). Capture a start reading, do work, capture an end reading, subtract to get a Duration. Never subtract two wall-clock timestamps to measure elapsed time.

Example 1 — measure the elapsed time of an operation with a monotonic clock

Python 
import time
start = time.monotonic()            # seconds from an arbitrary epoch
total = sum(range(1_000_000))
elapsed = time.monotonic() - start
print(f"{elapsed:.4f}s")            # e.g. 0.0123s
0.0123s
Go 
start := time.Now()                 // includes a monotonic reading
total := 0
for i := 0; i < 1_000_000; i++ { total += i }
fmt.Println(time.Since(start))      // e.g. 1.2ms  (uses the monotonic part)
1.2ms
TypeScript 
const start = performance.now();    // monotonic, sub-millisecond
let total = 0;
for (let i = 0; i < 1_000_000; i++) total += i;
console.log(`${(performance.now() - start).toFixed(2)}ms`); // e.g. 3.10ms
3.10ms
Rust 
use std::time::Instant;
let start = Instant::now();          // opaque, guaranteed monotonic
let total: u64 = (0..1_000_000).sum();
println!("{:?}", start.elapsed());   // e.g. 250µs
250µs
C++ 
#include <chrono>
using namespace std::chrono;
auto start = steady_clock::now();    // steady_clock == monotonic
long total = 0;
for (int i = 0; i < 1'000'000; ++i) total += i;
auto us = duration_cast<microseconds>(steady_clock::now() - start).count();
std::cout << us << "us";              // e.g. 900us
900us

Example 2 — a deadline / timeout guard built on the monotonic clock

A deadline is "start + budget" on the monotonic clock. Poll now() < deadline to decide whether to keep working — immune to wall-clock jumps.

Python 
import time
deadline = time.monotonic() + 0.05  # 50 ms budget
ticks = 0
while time.monotonic() < deadline:
    ticks += 1
print("done, ticks > 0:", ticks > 0)  # done, ticks > 0: True
done, ticks > 0: True
Go 
deadline := time.Now().Add(50 * time.Millisecond)
ticks := 0
for time.Now().Before(deadline) { ticks++ }
fmt.Println("done, ticks > 0:", ticks > 0) // done, ticks > 0: true
done, ticks > 0: true
TypeScript 
const deadline = performance.now() + 50; // ms
let ticks = 0;
while (performance.now() < deadline) ticks++;
console.log("done, ticks > 0:", ticks > 0); // done, ticks > 0: true
done, ticks > 0: true
Rust 
use std::time::{Instant, Duration};
let deadline = Instant::now() + Duration::from_millis(50);
let mut ticks = 0u64;
while Instant::now() < deadline { ticks += 1; }
println!("done, ticks > 0: {}", ticks > 0); // done, ticks > 0: true
done, ticks > 0: true
C++ 
#include <chrono>
using namespace std::chrono;
auto deadline = steady_clock::now() + milliseconds{50};
long ticks = 0;
while (steady_clock::now() < deadline) ++ticks;
std::cout << "done, ticks > 0: " << (ticks > 0); // done, ticks > 0: 1
done, ticks > 0: 1
Language Comparison: Every language separates the two clocks. Python: time.monotonic() vs time.time() (and perf_counter() for the highest resolution). Go cleverly folds a monotonic reading into every time.Now(), so time.Since/t2.Sub(t1) are automatically monotonic — but this reading is stripped when you round-trip through time.Unix() or serialization. TypeScript/JS: performance.now() (monotonic, high-res) vs Date.now() (wall). Rust: Instant (monotonic, opaque — you can't get a calendar date from it) vs SystemTime (wall, can go backward). C++: steady_clock (monotonic) vs system_clock (wall); high_resolution_clock is often just an alias for one of them.
Common Mistakes:
❌ [All] Measuring elapsed time by subtracting two wall-clock timestamps — an NTP step or DST change can make the result negative or huge.
✅ [All] Use the monotonic clock (monotonic/Instant/steady_clock/performance.now) for durations and timeouts.
❌ [Go] Serializing a time.Time and reading it back, then diffing — the monotonic reading is dropped, so you're back on the wall clock.
✅ [Go] Keep the original time.Time in memory for Since; don't round-trip it through storage before measuring.
❌ [Rust] Trying to convert an Instant to a calendar time — it's opaque and has no epoch.
✅ [Rust] Use SystemTime/chrono for calendar times; use Instant only for elapsed measurement.
❌ [C++] Assuming high_resolution_clock is monotonic — on some implementations it aliases system_clock and can jump.
✅ [C++] Use steady_clock explicitly when you need monotonicity.
Key Rules:
  • Wall clock answers "what time is it?"; monotonic clock answers "how long did it take?" — never swap them.
  • Compute deadlines as start + budget on the monotonic clock; poll against now().
  • Monotonic readings have no calendar meaning — don't log or serialize them as timestamps.

Decimal & Fixed-Point Arithmetic Numeric Precision
Why it matters: binary floating point (f64/double) cannot represent most decimal fractions exactly — 0.1 + 0.2 is famously 0.30000000000000004. For money, tax, and billing that rounding error is a compliance bug. The fixes: an exact decimal type, or work in the smallest integer unit (cents) — fixed-point.

Floats are great for measurements and math where tiny relative error is fine, and terrible for exact base-10 quantities. Two remedies: (1) a decimal type that stores digits in base 10 (Decimal, rust_decimal) so 0.1 is exact; or (2) store the value as an integer count of the smallest unit (cents, satoshis) and format on output. Rule: never use float for money.

Example 1 — the classic 0.1 + 0.2, and the decimal fix

Python 
from decimal import Decimal
print(0.1 + 0.2)                        # 0.30000000000000004
print(Decimal("0.1") + Decimal("0.2"))  # 0.3   (exact)
0.30000000000000004 0.3
Go 
fmt.Println(0.1 + 0.2)                  // 0.30000000000000004
// No stdlib decimal: work in integer cents (fixed-point)
cents := 10 + 20
fmt.Printf("$%d.%02d\n", cents/100, cents%100) // $0.30
0.30000000000000004 $0.30
TypeScript 
console.log(0.1 + 0.2);                 // 0.30000000000000004
// Fixed-point: integer cents
const cents = 10 + 20;
console.log((cents / 100).toFixed(2));  // "0.30"
0.30000000000000004 0.30
Rust 
println!("{}", 0.1_f64 + 0.2);          // 0.30000000000000004
// rust_decimal crate:
use rust_decimal_macros::dec;
println!("{}", dec!(0.1) + dec!(0.2));  // 0.3   (exact)
0.30000000000000004 0.3
C++ 
#include <iomanip>

// Fixed-point: integer cents
long cents = 10 + 20;
std::cout << "$" << cents / 100 << "." << std::setw(2)
          << std::setfill('0') << cents % 100; // $0.30
0.30000000000000004 $0.30

Example 2 — splitting a bill: why integer cents avoid a lost penny

Split $10.00 three ways. In floats you get repeating 3.333... and must round carefully; in integer cents you divide 1000 cents by 3 and distribute the remainder deterministically so the parts sum back to exactly 1000.

Python 
total, n = 1000, 3                       # cents
base, rem = divmod(total, n)             # 333, 1
parts = [base + (1 if i < rem else 0) for i in range(n)]
print(parts, sum(parts))                 # [334, 333, 333] 1000
[334, 333, 333] 1000
Go 
total, n := 1000, 3
base, rem := total/n, total%n
parts := make([]int, n)
for i := range parts {
    parts[i] = base
    if i < rem { parts[i]++ }
}
fmt.Println(parts, base*n+rem)           // [334 333 333] 1000
[334 333 333] 1000
TypeScript 
const total = 1000, n = 3;
const base = Math.floor(total / n), rem = total % n;
const parts = Array.from({ length: n }, (_, i) => base + (i < rem ? 1 : 0));
console.log(parts, parts.reduce((a, b) => a + b, 0)); // [334,333,333] 1000
[ 334, 333, 333 ] 1000
Rust 
let (total, n) = (1000i32, 3i32);
let (base, rem) = (total / n, total % n);
let parts: Vec<i32> = (0..n).map(|i| base + if i < rem { 1 } else { 0 }).collect();
println!("{:?} {}", parts, parts.iter().sum::<i32>()); // [334, 333, 333] 1000
[334, 333, 333] 1000
C++ 
int total = 1000, n = 3;
int base = total / n, rem = total % n, sum = 0;
for (int i = 0; i < n; ++i) {
    int p = base + (i < rem ? 1 : 0);
    sum += p;
    std::cout << p << " ";               // 334 333 333
}
std::cout << "| " << sum;                // | 1000
334 333 333 | 1000
Language Comparison: Python ships an exact decimal.Decimal (base-10, configurable precision and rounding) plus fractions.Fraction for exact rationals. Rust and Go have no decimal type in std — the community uses rust_decimal / shopspring/decimal, or you work in integer minor units. TypeScript/JS only has IEEE-754 number (and BigInt for exact integers) — money means BigInt cents or a library like decimal.js/dinero.js. C++ has no standard decimal either (some compilers offer _Decimal64); use integer fixed-point or a library (Boost.Multiprecision). The one universal, dependency-free answer is integer minor units.
Common Mistakes:
❌ [All] Using float/double for currency and comparing with == — accumulated rounding makes totals drift and equality flaky.
✅ [All] Use a decimal type or integer minor units; round explicitly at defined points (e.g. banker's rounding).
❌ [Python] Building a Decimal from a float: Decimal(0.1) inherits the float error (0.1000000000000000055...).
✅ [Python] Construct from a string: Decimal("0.1").
❌ [TypeScript] (0.1 + 0.2).toFixed(2) hides the error but you already lost precision if you summed many floats first.
✅ [TypeScript] Keep the running total in integer cents (BigInt if large) and format only for display.
❌ [Go/Rust/C++] Reaching for float "just for the division" when splitting money — the remainder penny gets lost or duplicated.
✅ [Go/Rust/C++] Do integer division + explicit remainder distribution so the parts sum exactly to the total.
Key Rules:
  • Never use binary floating point for money or any exact base-10 quantity.
  • Prefer a decimal type (exact base-10) or integer minor units (fixed-point); decide rounding mode explicitly.
  • When distributing a total, allocate the remainder deterministically so the parts sum back to the original.

Memory & Reference Patterns

Two patterns for controlling object lifetime beyond plain ownership: weak references break the reference cycles that leak under reference counting, and pools/arenas amortize allocation cost by reusing or bulk-freeing memory. Both show how differently each runtime manages memory.

Weak References & Reference Cycles Memory
Why it matters: under reference counting (Python's default, Rust Rc/Arc, C++ shared_ptr), two objects that point at each other keep each other alive forever — a memory leak. A weak reference observes an object without owning it: it doesn't prevent collection and reports whether the target is still alive. Essential for parent pointers, observer lists, and caches.

A strong reference owns a share of the object and keeps it alive; a weak reference does not. To use a weak ref you must "upgrade"/"lock" it, which yields the object if it still exists or a null/None if it was already dropped. The canonical fix: in a tree with parent↔child links, make the child→parent edge weak so the cycle can't keep the subtree alive.

Example 1 — a weak reference goes empty once the last strong owner is dropped

Python 
import weakref
class Node: pass
a = Node()
w = weakref.ref(a)
print(w() is a)                 # True  (target alive)
del a
print(w())                      # None  (target collected)
True None
Go 
import "weak"                    // Go 1.24+
type Node struct{ V int }
a := &Node{V: 5}
wp := weak.Make(a)
fmt.Println(wp.Value() != nil)  // true  (still reachable)
// after a is unreachable and GC runs, wp.Value() returns nil.
// Note: Go's tracing GC already reclaims cycles automatically.
true
TypeScript 
let a: { v: number } | null = { v: 5 };
const w = new WeakRef(a);
console.log(w.deref() === a);   // true
a = null;                       // drop the only strong ref
// after GC, w.deref() becomes undefined (timing is not guaranteed)
console.log(w.deref()?.v);      // 5  (until GC runs)
true 5
Rust 
use std::rc::{Rc, Weak};
let a: Rc<i32> = Rc::new(5);
let w: Weak<i32> = Rc::downgrade(&a);
println!("{:?}", w.upgrade());  // Some(5)
drop(a);                        // last strong owner gone
println!("{:?}", w.upgrade());  // None
Some(5) None
C++ 
#include <memory>
auto a = std::make_shared<int>(5);
std::weak_ptr<int> w = a;
std::cout << (w.lock() != nullptr);  // 1  (alive)
a.reset();                          // drop last shared owner
std::cout << (w.lock() != nullptr);  // 0  (expired)
10

Example 2 — parent↔child tree: make the parent link weak to avoid a leak

Child nodes are owned strongly by the parent (parent→child = strong). The back-edge child→parent is weak, so dropping the tree's root frees everything instead of forming an uncollectable cycle.

Python 
import weakref
class Node:
    def __init__(self, val):
        self.val = val
        self.children = []
        self._parent = None
    def add(self, child):
        child._parent = weakref.ref(self)   # weak back-edge
        self.children.append(child)
    @property
    def parent(self): return self._parent() if self._parent else None

root = Node("root"); child = Node("c"); root.add(child)
print(child.parent.val)          # root
root
Go 
// Go's tracing GC reclaims parent<->child cycles automatically,
// so a plain pointer back-edge is fine and idiomatic:
type Node struct {
    Val      string
    Children []*Node
    Parent   *Node          // plain pointer; no leak under tracing GC
}
root := &Node{Val: "root"}
child := &Node{Val: "c", Parent: root}
root.Children = append(root.Children, child)
fmt.Println(child.Parent.Val)    // root
root
TypeScript 
// JS uses a tracing GC too, so cycles are collected; WeakRef is for
// caches/observers, not leak-avoidance. A weak parent link:
class Node {
  children: Node[] = [];
  private parentRef?: WeakRef<Node>;
  constructor(public val: string) {}
  add(c: Node) { c.parentRef = new WeakRef(this); this.children.push(c); }
  get parent(): Node | undefined { return this.parentRef?.deref(); }
}
const root = new Node("root"); const child = new Node("c"); root.add(child);
console.log(child.parent?.val);  // root
root
Rust 
use std::rc::{Rc, Weak};
use std::cell::RefCell;
struct Node {
    val: String,
    parent: RefCell<Weak<Node>>,       // weak back-edge breaks the cycle
    children: RefCell<Vec<Rc<Node>>>,
}
let root = Rc::new(Node { val: "root".into(),
    parent: RefCell::new(Weak::new()), children: RefCell::new(vec![]) });
let child = Rc::new(Node { val: "c".into(),
    parent: RefCell::new(Rc::downgrade(&root)), children: RefCell::new(vec![]) });
root.children.borrow_mut().push(Rc::clone(&child));
println!("{}", child.parent.borrow().upgrade().unwrap().val); // root
root
C++ 
#include <memory>
#include <vector>
struct Node {
    std::string val;
    std::weak_ptr<Node> parent;          // weak: no ownership cycle
    std::vector<std::shared_ptr<Node>> children;
};
auto root = std::make_shared<Node>(); root->val = "root";
auto child = std::make_shared<Node>(); child->val = "c";
child->parent = root;
root->children.push_back(child);
std::cout << child->parent.lock()->val;  // root
root
Language Comparison: The need for weak refs depends on the memory model. Python uses reference counting plus a cyclic garbage collector, so cycles are eventually collected — but weak refs still matter for caches and to avoid keeping large objects alive. Rust and C++ use pure reference counting (Rc/Arc, shared_ptr) with no cycle collector, so a strong cycle is a hard leak — Weak/weak_ptr is mandatory for back-edges. Go and TypeScript/JS use tracing GCs that reclaim cycles automatically, so weak references (weak.Pointer in Go 1.24, WeakRef/WeakMap in JS) exist for caches and finalizers, not to prevent leaks. Upgrade semantics are the same everywhere: upgrade() / .lock() / deref() / () returns the object or nothing.
Common Mistakes:
❌ [Rust/C++] Making both edges of a parent↔child relationship strong (Rc/shared_ptr) — the cycle never drops to zero and leaks.
✅ [Rust/C++] Own children strongly, hold the parent as Weak/weak_ptr, and upgrade()/lock() when you need it.
❌ [All] Using the target through a weak ref without checking liveness — it may already be gone.
✅ [All] Always branch on the upgrade result (Some/non-null/defined) before dereferencing.
❌ [Python] Taking a weakref.ref to a type that doesn't support it (e.g. plain list/tuple, or objects with __slots__ lacking __weakref__) — raises TypeError.
✅ [Python] Weak-reference user-defined classes (or use WeakValueDictionary for caches).
❌ [TypeScript] Relying on WeakRef.deref() timing for logic — when the target is collected is non-deterministic.
✅ [TypeScript] Treat WeakRef/WeakMap as best-effort caches; never depend on prompt collection.
Key Rules:
  • Under reference counting (Rust/C++/Python), break cycles with a weak back-edge; under tracing GC (Go/JS), cycles are collected but weak refs still help caches.
  • A weak ref owns nothing — always upgrade/lock and check for null before use.
  • Model ownership as a tree of strong edges; make any back- or cross-edges weak.

Object Pooling & Arena Allocation Memory
Why it matters: allocating and freeing many short-lived objects thrashes the allocator and, in GC languages, drives collection pressure. Two amortization strategies dominate hot paths: an object pool recycles a fixed set of reusable buffers, and an arena (bump allocator) hands out memory from a big block and frees it all at once. Both trade a little flexibility for large throughput gains in servers, parsers, and game loops.

A pool keeps a free-list: get() pops a recycled object (or makes one), put() returns it for reuse — you reset its state, not its memory. An arena allocates by simply advancing a pointer into a preallocated block; individual objects are never freed, the whole block is reclaimed at once (perfect for per-request or per-frame lifetimes). Both cut allocator calls dramatically.

Example 1 — a reusable buffer pool (get / put)

Python 
class Pool:
    def __init__(self, factory): self.factory, self.free = factory, []
    def get(self): return self.free.pop() if self.free else self.factory()
    def put(self, obj): obj.clear(); self.free.append(obj)  # reset + recycle

pool = Pool(lambda: bytearray(1024))
buf = pool.get(); buf.extend(b"hi")
pool.put(buf)                    # returned for reuse
print(len(pool.free))            # 1
1
Go 
import "sync"
var pool = sync.Pool{New: func() any { b := make([]byte, 0, 1024); return &b }}
p := pool.Get().(*[]byte)
*p = append((*p)[:0], "hi"...)   // reset length, reuse capacity
fmt.Println(string(*p))          // hi
pool.Put(p)                      // recycle (may be GC'd under pressure)
hi
TypeScript 
class Pool<T> {
  private free: T[] = [];
  constructor(private factory: () => T, private reset: (o: T) => void) {}
  get(): T { return this.free.pop() ?? this.factory(); }
  put(o: T) { this.reset(o); this.free.push(o); }
}
const pool = new Pool(() => [] as number[], a => (a.length = 0));
const buf = pool.get(); buf.push(1, 2);
pool.put(buf);
console.log(pool.get().length);  // 0  (reused, reset)
0
Rust 
struct Pool { free: Vec<String> }
impl Pool {
    fn get(&mut self) -> String { self.free.pop().unwrap_or_default() }
    fn put(&mut self, mut s: String) { s.clear(); self.free.push(s); } // keep capacity
}
let mut pool = Pool { free: vec![] };
let mut b = pool.get(); b.push_str("hi");
pool.put(b);
println!("{}", pool.free.len());  // 1
1
C++ 
#include <vector>
#include <string>
struct Pool {
    std::vector<std::string> free;
    std::string get() {
        if (free.empty()) return {};
        std::string s = std::move(free.back()); free.pop_back(); return s;
    }
    void put(std::string s) { s.clear(); free.push_back(std::move(s)); } // keep buffer
};
Pool pool;
auto b = pool.get(); b += "hi";
pool.put(std::move(b));
std::cout << pool.free.size();     // 1
1

Example 2 — an arena: bump-allocate many values, free them all at once

Allocate several values from one block; there is no per-object free — the entire arena is reclaimed together when its lifetime ends.

Python 
# Slab arena: one preallocated list, bump an index, reset in O(1)
class Arena:
    def __init__(self, n): self.slab = [None] * n; self.top = 0
    def alloc(self, v):
        i = self.top; self.slab[i] = v; self.top += 1; return i
    def reset(self): self.top = 0        # "frees" everything at once

a = Arena(8)
a.alloc(1); a.alloc(2); a.alloc(3)
print(a.top); a.reset(); print(a.top)   # 3 then 0
3 0
Go 
type Arena struct{ slab []int; top int }
func (a *Arena) Alloc(v int) int { a.slab[a.top] = v; a.top++; return a.top - 1 }
func (a *Arena) Reset()          { a.top = 0 }   // bulk free
ar := &Arena{slab: make([]int, 8)}
ar.Alloc(1); ar.Alloc(2); ar.Alloc(3)
fmt.Println(ar.top); ar.Reset(); fmt.Println(ar.top) // 3 then 0
3 0
TypeScript 
// Typed-array slab: bump an offset, reset in O(1)
class Arena {
  private slab: Int32Array; private top = 0;
  constructor(n: number) { this.slab = new Int32Array(n); }
  alloc(v: number): number { const i = this.top++; this.slab[i] = v; return i; }
  reset() { this.top = 0; }
}
const a = new Arena(8);
a.alloc(1); a.alloc(2); a.alloc(3);
console.log(a["top"]); a.reset(); // 3 then top=0
3
Rust 
use bumpalo::Bump;               // bumpalo crate
let arena = Bump::new();
let x: &mut i32 = arena.alloc(1);
let y: &mut i32 = arena.alloc(2);
let z: &mut i32 = arena.alloc(3);
println!("{}", *x + *y + *z);    // 6
// everything is freed together when `arena` is dropped
6
C++ 
#include <memory_resource>
#include <vector>
#include <array>
std::array<std::byte, 1024> buf;
std::pmr::monotonic_buffer_resource arena{buf.data(), buf.size()};
std::pmr::vector<int> v{&arena};   // allocates from the arena
v.push_back(1); v.push_back(2); v.push_back(3);
std::cout << v[0] + v[1] + v[2];  // 6
// no per-element free; the arena reclaims in bulk on scope exit

Language Comparison: Go has a first-class pool in the stdlib — sync.Pool — designed to reduce GC pressure (but the runtime may drain it at any GC, so never assume an item survives). Rust leans on arenas: bumpalo for a general bump allocator, typed-arena for one type; ownership means arena-allocated refs are borrow-checked against the arena's lifetime. C++ standardized arenas as polymorphic memory resources (std::pmr::monotonic_buffer_resource) that plug into any pmr container. Python and TypeScript/JS have GC and no manual allocator, so pooling is a hand-rolled free-list; the real win there is reusing large buffers (typed arrays, bytearray) to avoid re-allocation and GC churn. Everywhere the discipline is the same: reset state on recycle, and never touch an object after returning it to the pool.
Common Mistakes:
❌ [All] Using an object after returning it to the pool (or reading arena memory after reset) — a use-after-free / aliasing bug.
✅ [All] Drop your reference the moment you put()/reset; treat the handle as invalid afterward.
❌ [All] Forgetting to reset recycled state — stale data leaks from the previous user into the next.
✅ [All] Clear/zero the object on put (or on get) so every borrower starts clean.
❌ [Go] Assuming sync.Pool retains items — the GC can empty it between uses, and storing pointers to large objects can worsen memory.
✅ [Go] Use it only for transient, reconstructable objects; always handle a fresh New() result.
❌ [Rust] Returning an arena-allocated reference past the arena's lifetime — it won't compile, and fighting it with unsafe invites dangling refs.
✅ [Rust] Scope the arena to the batch (request/frame) and let the borrow checker bound the references.
Key Rules:
  • Pool = reuse individual objects via a free-list (reset on recycle); arena = bump-allocate many, free all at once.
  • Match the tool to the lifetime: pools for churny same-shaped buffers, arenas for per-request/per-frame batches.
  • Never use an object after returning it, and always reset recycled state — these are the two ways pools go wrong.
