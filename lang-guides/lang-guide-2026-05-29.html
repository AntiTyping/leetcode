Sequential Collections

Arrays, Slices & Vectors Collections

Sequential collections are the most fundamental data structure in programming. Every algorithm you write will manipulate arrays in some form — from simple iteration to complex sorting and searching. Mastering the differences in how each language handles dynamic arrays (creation, slicing, ownership, memory) is the foundation for writing correct, performant, idiomatic code.

Every language provides a dynamic, resizable sequence type as its workhorse collection: Python has list, Go has slices, TypeScript has Array, Rust has Vec<T>, and C++ has std::vector<T>. While they share the same core interface — indexed access, append, iterate — they differ significantly in typing, memory model, and safety guarantees. This section covers creation, indexing (including bounds checking), slicing, iteration patterns, and common mutation operations across all five languages.

package main

import "fmt"

func main() {
    nums := []int{1, 2, 3}
    nums = append(nums, 4)                          // append to end
    nums = append(nums[:2], append([]int{99}, nums[2:]...)...) // insert 99 at index 1 (shift right)
    // Wait — insert at index 1:
    // Actually, let's redo cleanly
    nums = []int{1, 2, 3}
    nums = append(nums, 4)
    // insert 99 at index 1
    nums = append(nums[:1], append([]int{99}, nums[1:]...)...)
    // remove element at index 2 (value 2)
    nums = append(nums[:2], nums[3:]...)
    fmt.Println(nums)
}

Language Comparison: Python lists are dynamic and heterogeneous — you can mix types freely, and they resize automatically. Go slices are typed and backed by fixed-size arrays; re-slicing shares the underlying array, and append may allocate a new one. TypeScript arrays are typed (when annotated) but flexible at runtime, with a rich set of built-in methods like map, filter, and splice. Rust Vec<T> is a heap-allocated, owned, growable array — the borrow checker enforces that you cannot hold a reference to elements while mutating the vector. C++ std::vector<T> is RAII-managed with value semantics; it provides iterator-based access and automatically frees memory when it goes out of scope.
Common Mistakes:
Python: Using list.append([1, 2]) when you meant list.extend([1, 2]). append adds the entire list as a single element: [..., [1, 2]]. extend adds each element individually: [..., 1, 2].
Fix: Use nums.extend([1, 2]) to flatten, or nums += [1, 2] which is equivalent to extend.
Go: Slice aliasing — modifying a slice derived from another slice affects the original backing array. b := a[1:3] shares memory with a. Appending to b may overwrite a[3] if capacity allows.
Fix: Use a full slice expression b := a[1:3:3] to limit capacity, forcing append to allocate a new array. Or copy explicitly: b := make([]int, 2); copy(b, a[1:3]).
TypeScript: [10, 2, 1].sort() sorts lexicographically, producing [1, 10, 2] instead of [1, 2, 10]. The default sort converts elements to strings.
Fix: Always provide a comparator for numeric sorts: [10, 2, 1].sort((a, b) => a - b) produces [1, 2, 10].
Rust: Cannot index a Vec with negative indices (no nums[-1]). Also, borrowing a slice (&nums[1..3]) while also mutating the vec is a compile error due to borrow rules.
Fix: Use nums[nums.len() - 1] for the last element, or nums.last().unwrap(). For mutation, finish using borrowed slices before calling mutable methods.
C++: Iterator invalidation — inserting or erasing elements during iteration with iterators invalidates those iterators, causing undefined behavior.
Fix: Use the return value of erase() which gives a valid iterator to the next element: it = vec.erase(it). Or collect indices first, then modify.
Key Rules:
  • Use dynamic arrays as your default collection — list, slices, Array, Vec, vector cover 90% of use cases. Reach for fixed-size arrays only when the size is truly constant and known at compile time.
  • Pre-allocate when you know the size for performance — Go: make([]T, 0, n), Rust: Vec::with_capacity(n), C++: v.reserve(n), Python: list comprehensions avoid repeated appends.
  • Prefer iteration over manual indexing — it is safer and more idiomatic in all five languages. Use enumerate / range / forEach / .iter().enumerate() / range-for instead of for (int i = 0; ...).

Strings — Mutability, Unicode & Common Operations Collections

Strings appear in virtually every program — from parsing input to formatting output to processing text data. But strings behave very differently across languages: some are immutable, some are mutable; some are byte-oriented, others character-oriented. Understanding these differences prevents subtle bugs around Unicode, performance traps with concatenation, and incorrect assumptions about string length.

Strings are specialized sequences of characters (or bytes). Python and TypeScript strings are immutable — every operation produces a new string. Go strings are also immutable byte slices (UTF-8 encoded), but you must use rune to work with Unicode code points. Rust distinguishes between owned String (mutable, heap-allocated UTF-8) and borrowed &str (an immutable view). C++ std::string is mutable with value semantics and provides byte-level access. This section covers creation, concatenation, splitting, joining, searching, and formatting across all five languages.

Example 1: Create strings, concatenate, get length (bytes vs chars)

Python
s = "hello"
greeting = s + " world"
print(greeting)
print(len(greeting))          # character count
emoji = "Hi 👋"
print(len(emoji))             # 4 chars (emoji is 1 char in Python)
hello world 11 4
Go
package main

import (
    "fmt"
    "unicode/utf8"
)

func main() {
    s := "hello"
    greeting := s + " world"
    fmt.Println(greeting)
    fmt.Println(len(greeting))                  // byte length
    emoji := "Hi 👋"
    fmt.Println(len(emoji))                     // 7 bytes (emoji is 4 bytes)
    fmt.Println(utf8.RuneCountInString(emoji))  // 4 runes
}
hello world 11 7 4

#include <iostream>
#include <string>
#include <algorithm>
using namespace std;

string trim(const string& s) {
    size_t start = s.find_first_not_of(" \t\n\r");
    size_t end = s.find_last_not_of(" \t\n\r");
    return (start == string::npos) ? "" : s.substr(start, end - start + 1);
}

int main() {
    string s = "  hello world  ";
    string trimmed = trim(s);
    cout << trimmed << endl;
    cout << trimmed.find("world") << endl;
    string replaced = trimmed;
    size_t pos = replaced.find("world");
    if (pos != string::npos)
        replaced.replace(pos, 5, "C++");
    cout << replaced << endl;
}

Language Comparison: Python strings are immutable Unicode sequences — every concatenation creates a new string object. Go strings are immutable byte slices encoded as UTF-8; use the rune type and for _, r := range s to iterate over Unicode code points rather than raw bytes. TypeScript strings are immutable sequences of UTF-16 code units, so surrogate pairs (like emoji) have .length of 2 even though they represent one character. Rust distinguishes between String (owned, mutable, heap-allocated UTF-8) and &str (a borrowed, immutable slice into UTF-8 data) — you cannot index by integer because UTF-8 characters are variable-width. C++ std::string is mutable with value semantics and operates at the byte level; it has no built-in Unicode awareness, so multi-byte characters require external libraries or manual handling.
Common Mistakes:
Python: Building strings with += in a loop creates a new string object on every iteration, resulting in O(n²) time for n concatenations.
Fix: Collect parts in a list and use ''.join(parts) at the end — this is O(n) and idiomatic Python.
Go: Iterating with for i := 0; i < len(s); i++ iterates over bytes, not characters. A multi-byte UTF-8 character like "é" will be split into individual bytes.
Fix: Use for i, r := range s which decodes each UTF-8 rune correctly and gives you the byte index and rune value.
TypeScript: Using == instead of === for string comparison. The == operator performs type coercion, so "0" == false is true.
Fix: Always use === (strict equality) for comparisons in TypeScript/JavaScript. Configure your linter to enforce this.
Rust: Trying to index a String by integer: s[0] does not compile because UTF-8 characters are variable-width and byte indexing could split a character.
Fix: Use s.chars().nth(0) for the first character, or s.as_bytes()[0] if you specifically need the first byte. For substrings, use &s[0..n] only when you know n is a valid UTF-8 boundary.
C++: Checking the return value of std::string::find() against -1 or 0. The method returns std::string::npos (a very large value) on failure, not -1.
Fix: Always compare against std::string::npos: if (s.find("x") != string::npos).
Key Rules:
  • Always use the language's built-in formatting for string construction — Python f-strings, Go fmt.Sprintf, TypeScript template literals, Rust format!, C++ std::format. Never build strings with repeated concatenation.
  • Be aware of bytes vs characters — Go and Rust are UTF-8 byte-oriented, so len() returns bytes, not characters. Python and TypeScript are character-oriented (though TS uses UTF-16 code units). Always use the character-counting method (utf8.RuneCountInString, .chars().count(), [...s].length) when you need the actual character count.
  • For heavy string building, use a builder/buffer — Python: ''.join(list), Go: strings.Builder, TypeScript: array + .join(), Rust: String::with_capacity(n) then push_str, C++: ostringstream or string::reserve(n).

Keyed Collections & Stacks

Hash Maps & Hash Sets Collections

Hash maps and hash sets are the workhorses of real-world programming. They give you O(1) average-time lookup, insertion, and deletion by key, making them essential for frequency counting, deduplication, caching, and the majority of coding interview problems.

Every language has a hash map and hash set, but the APIs and gotchas differ significantly. Python has built-in dict and set with clean literal syntax. Go uses map[K]V but has no built-in set type. TypeScript offers both Map/Set classes and plain objects. Rust and C++ require explicit imports from their standard libraries. Mastering the idioms for each language is critical for writing fluent, bug-free code.

package main

func main() {
    scores := map[string]int{"alice": 95, "bob": 82}

    fmt.Println(val, ok)

    fmt.Println(exists)

TypeScript has Map<K,V> and Set<T> plus plain objects Record<string, V>. Rust uses HashMap<K,V> and HashSet<T> from std::collections with the entry() API. C++ uses unordered_map<K,V> and unordered_set<T> with iterator-based access.
Common Mistakes:
Python: Modifying a dict while iterating over it raises RuntimeError.
Fix: Iterate over list(d.keys()) or work on d.copy() instead.
Go: Map iteration order is randomized by the runtime — never depend on it. Also, reading from a nil map returns the zero value, but writing to a nil map panics.
Fix: Always initialize maps with make(map[K]V) before writing. Sort keys if you need deterministic order.
TypeScript: Using plain objects as maps loses type safety for non-string keys. obj[42] silently coerces the key to the string "42".
Fix: Use Map<K,V> for non-string keys to preserve key types and get proper iteration.
Rust: Calling .get() returns Option<&V> — forgetting to unwrap or pattern match causes type errors. New Rust programmers also miss the entry().or_insert() pattern for counters.
Fix: Use if let Some(v) = map.get(&key) or *map.entry(key).or_insert(0) += 1 for frequency counting.
C++: Using operator[] on a map inserts a default-constructed value if the key doesn't exist, silently growing the map.
Fix: Use .find() or .count() for existence checks. Use at() if you want an exception on missing keys.
Key Rules:
  • Hash maps are your go-to for O(1) lookup by key — use them for frequency counting, deduplication, and caching.
  • Always handle the "key not found" case: Python raises KeyError (use .get() or in), Go returns zero value + ok bool, Rust returns Option, C++ inserts a default with [].
  • Hash sets are just maps without values — use them when you only need membership testing.

Stacks & Queues Collections

Stacks and queues are the simplest abstract data structures, but they power some of the most important algorithms in computer science. Stacks enable DFS, backtracking, and expression parsing. Queues drive BFS and level-order traversal. Knowing how to implement them idiomatically in each language — and avoiding the O(n) performance traps — is essential.

A stack is Last-In-First-Out (LIFO): the last element pushed is the first one popped. A queue is First-In-First-Out (FIFO): the first element enqueued is the first one dequeued. A deque (double-ended queue) supports efficient insertion and removal at both ends. Every language provides these in slightly different ways, and choosing the wrong underlying structure can turn O(1) operations into O(n).

package main

import "fmt"

func isValid(s string) bool {
    stack := []rune{}
    pairs := map[rune]rune{')': '(', '}': '{', ']': '['}

            }

        }
    }

Language Comparison: Python uses list as a stack (append/pop) and collections.deque for queues (appendleft/popleft in O(1)). Go uses slices as stacks and typically slices or channels for queues. TypeScript uses arrays for stacks (push/pop) but shift() for queues is O(n) — no built-in deque. Rust has Vec<T> for stacks and VecDeque<T> for efficient double-ended access. C++ has dedicated stack<T>, queue<T>, and deque<T> adapters.
Common Mistakes:
Python: Using list.pop(0) for queue operations — it's O(n) because all remaining elements must shift left.
Fix: Use collections.deque with popleft() for O(1) front removal.
Go: Memory leaks with slice-based stacks — when you reslice s[:len(s)-1], the popped element's reference remains in the underlying array.
Fix: Zero out the popped element before reslicing: s[len(s)-1] = zero then s = s[:len(s)-1], especially for pointer/interface types.
TypeScript: Using Array.shift() for queue dequeue — it's O(n) since all elements shift down one index.
Fix: For performance-critical queues, use an index-based approach or a linked list implementation.
Rust: Using Vec::remove(0) for queue dequeue — it's O(n) because all elements must shift.
Fix: Use VecDeque::pop_front() for O(1) dequeue operations.
C++: Calling top() or front() on an empty stack/queue is undefined behavior — it may crash or silently corrupt memory.
Fix: Always check empty() before accessing the top/front element.
Key Rules:
  • Use a stack when you need to process elements in reverse order or match nested structures (parentheses, HTML tags, undo/redo).
  • Use a queue for BFS, level-order traversal, and any "first-come-first-served" processing.
  • Always use a proper deque/VecDeque for queue operations — using a list/array with O(n) front removal is a common performance trap.
