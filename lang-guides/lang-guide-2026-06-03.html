#include <variant>
#include <string>
#include <iostream>
using namespace std;

struct Quit {};
struct Echo { string msg; };
struct Move { int x, y; };
using Command = variant<Quit, Echo, Move>;

string handle(const Command& cmd) {
    return visit([](auto&& arg) -> string {
        using T = decay_t<decltype(arg)>;
        if constexpr (is_same_v<T, Quit>)
            return "bye";
        else if constexpr (is_same_v<T, Echo>)
            return "echo: " + arg.msg;
        else
            return "move to (" + to_string(arg.x) + ", " + to_string(arg.y) + ")";
    }, cmd);
}

cout << handle(Move{3, 7}) << endl;

Builder Pattern & Method Chaining Functions & Abstractions

The builder pattern separates complex object construction from its representation. Method chaining creates fluent, readable APIs where each setter returns the builder itself, allowing Builder().name("X").size(10).build() instead of long constructor argument lists.

The core idea: instead of a constructor with 10 parameters (most optional), create a separate builder object that accumulates configuration, then produces the final object in one .build() call. This pattern is especially valuable when:

(1) Objects have many optional fields with sensible defaults, (2) Construction involves validation that should happen once, (3) You want a fluent, self-documenting API.

Example 1: Basic Builder — HTTP Request

    def __init__(self, method, url, headers, body):
        self.method = method
        self.url = url
        self.headers = headers
        self.body = body

    def __repr__(self):

    def __init__(self, method, url):

type HttpRequest struct {
    Method  string
    URL     string
    Headers map[string]string
    Body    string
}

// Functional options pattern — idiomatic Go
type Option func(*HttpRequest)

func WithHeader(key, value string) Option {
    return func(r *HttpRequest) {
        r.Headers[key] = value
    }
}

func WithBody(body string) Option {
    return func(r *HttpRequest) {
        r.Body = body
    }
}

func NewRequest(method, url string, opts ...Option) *HttpRequest {
    r := &HttpRequest{Method: method, URL: url, Headers: map[string]string{}}
    for _, opt := range opts {
        opt(r)
    }
    return r
}

req := NewRequest("POST", "/api/users",
    WithHeader("Content-Type", "application/json"),
    WithHeader("Authorization", "Bearer token123"),
    WithBody(`{"name": "Alice"}`),
)

use std::collections::HashMap;

#[derive(Debug)]
struct HttpRequest {
    method: String,
    url: String,
    headers: HashMap<String, String>,
    body: Option<String>,
}

struct RequestBuilder {
    method: String,
    url: String,
    headers: HashMap<String, String>,
    body: Option<String>,
}

impl RequestBuilder {
    fn new(method: &str, url: &str) -> Self {
        Self {
            method: method.into(),
            url: url.into(),
            headers: HashMap::new(),
            body: None,
        }
    }

    // Consuming builder: takes self by value, returns self
    fn header(mut self, key: &str, value: &str) -> Self {
        self.headers.insert(key.into(), value.into());
        self
    }

    fn body(mut self, data: &str) -> Self {
        self.body = Some(data.into());
        self
    }

    fn build(self) -> HttpRequest {
        HttpRequest {

Language Comparison — Builder Styles: Python uses return self for chaining; @dataclass with defaults often eliminates the need for builders entirely. Go prefers the “functional options” pattern (With* functions) over method chaining — it’s more idiomatic and works better with variadic arguments. TypeScript returns this (not the class name) to support subclass chaining. Rust has two builder styles: consuming (takes self by value, prevents reuse) and borrowing (&mut self, allows reuse). Consuming is safer because it prevents calling .build() twice. C++ returns *this by reference; be careful not to chain on a temporary that gets destroyed.
Common Mistakes:
  • Python: Forgetting return self in one setter breaks the entire chain — None is returned instead.
    ❌ def host(self, h): self._host = h — returns None
    ✅ def host(self, h): self._host = h; return self
  • Go: Using value receivers instead of pointer receivers — mutations are lost because Go copies the struct.
    ❌ func (b ServerBuilder) Port(p int) ServerBuilder
    ✅ func (b *ServerBuilder) Port(p int) *ServerBuilder
  • TypeScript: Returning the class name instead of this type breaks chaining in subclasses.
    ❌ header(...): RequestBuilder
    ✅ header(...): this
  • Rust: Using &mut self (borrowing builder) when consuming (self) is safer — consuming prevents reuse of a partially-built state.
    ❌ Reusing a builder after .build() with &mut self
    ✅ Consuming builder: fn build(self) makes reuse a compile error
  • C++: Returning a copy instead of a reference in setters creates a new builder each time, losing accumulated state.
    ❌ ServerBuilder host(...) — returns by value
    ✅ ServerBuilder& host(...) — returns by reference
Key Rules:
  • Use builders when constructors get unwieldy — more than 4-5 parameters, especially optional ones, is a good signal to switch to a builder.
  • Validate in build(), not in setters — setters should be simple and fast. Validation at build time catches invalid combinations that individual setters can’t.
  • Each language has its preferred style — Go favors functional options, Python favors dataclass with defaults for simple cases, Rust favors consuming builders. Don’t force one language’s pattern onto another.

Smart Pointers & Memory Management Functions & Abstractions

Understanding how each language manages memory is crucial for writing correct, performant code. From garbage collection to ownership, each approach has trade-offs in safety, performance, and ergonomics.

Languages fall into three memory management categories: (1) Garbage collected (Python, Go, TypeScript) — the runtime tracks references and frees memory automatically, (2) Ownership-based (Rust) — the compiler enforces memory safety at compile time with zero runtime cost, (3) Manual with smart helpers (C++) — smart pointers automate cleanup via RAII but the programmer chooses the strategy.

Language Comparison — Memory Management Philosophy: Python and TypeScript use GC and prioritize developer ergonomics — you never think about memory, but you pay with GC pauses and less predictable performance. Go uses a concurrent, low-latency GC optimized for server workloads — a pragmatic middle ground. Rust enforces memory safety at compile time through ownership and borrowing — zero runtime cost, but a steeper learning curve. C++ gives you full control with smart pointers (RAII), but mistakes lead to undefined behavior rather than compiler errors. Rust’s RefCell and C++’s mutable both provide “interior mutability” — the ability to mutate data behind an immutable reference — but Rust checks at runtime while C++ trusts the programmer.
Common Mistakes:
  • Python: Circular references defeat reference counting — the GC catches them eventually, but __del__ may not run predictably. Use weakref to break cycles.
    ❌ a.parent = b; b.child = a — circular strong references
    ✅ a.parent = weakref.ref(b)
  • Go: Assuming everything is a pointer — slices, maps, and channels are reference types, but arrays and structs are value types (copied on assignment).
    ❌ a := myStruct; b := a; b.field = 5 — a is unchanged
    ✅ Use &myStruct when you want shared mutation
  • TypeScript: Relying on WeakRef for deterministic cleanup — GC timing is unpredictable. Use explicit dispose() patterns or using (TC39 proposal) for resource management.
    ❌ new WeakRef(resource) for cleanup
    ✅ Explicit resource.close() in a try/finally
  • Rust: Reaching for Rc<RefCell<T>> as the first solution — it’s a code smell. Restructure ownership first; use interior mutability only when ownership redesign is impractical.
    ❌ Wrapping everything in Rc<RefCell<T>>
    ✅ Pass &mut T through function parameters when possible
  • C++: Using shared_ptr everywhere — it has atomic reference counting overhead. Use unique_ptr as the default; only use shared_ptr when ownership is genuinely shared.
    ❌ shared_ptr for single-owner resources
    ✅ unique_ptr by default, shared_ptr only when needed
Key Rules:
  • Understand your language’s memory model — GC languages (Python, Go, TS) handle cleanup automatically but have unpredictable timing. Rust guarantees deterministic cleanup. C++ provides it via RAII.
  • Prefer single ownership — even in GC languages, clear ownership makes code easier to reason about. In Rust, Box<T> before Rc<T>; in C++, unique_ptr before shared_ptr.
  • Use weak references to break cycles — Python’s weakref, Rust’s Weak<T>, C++’s weak_ptr, and TS’s WeakRef/WeakMap all serve this purpose.

File I/O I/O & Data Processing

Reading and writing files is one of the most common operations in any program. Getting it right means handling encoding, buffering, error conditions, and ensuring files are closed even when errors occur.

Every language provides file I/O but with different idioms for resource cleanup: Python uses with (context managers), Go uses defer, Rust uses RAII (drop), C++ uses RAII (destructors), and TypeScript uses try/finally or newer using proposals.

The critical rule: always ensure files are closed, even when an error occurs mid-read/write.

Language Comparison — Resource Cleanup: Python’s with statement is the gold standard for readability — the file is guaranteed to close even if an exception occurs. Go’s defer f.Close() achieves the same effect. Rust uses RAII — the file handle is closed when it goes out of scope (or when drop() is called). C++ also uses RAII via fstream destructors. TypeScript has no built-in resource management syntax yet (the using declaration is a Stage 3 TC39 proposal), so try/finally is the current approach for async file handles.
Common Mistakes:
  • Python: Opening a file without with — forgetting to call f.close() leaks file descriptors, which can exhaust OS limits.
    ❌ f = open("x.txt"); data = f.read() — never closed on exception
    ✅ with open("x.txt") as f: data = f.read()
  • Go: Deferring close before checking the open error — defer f.Close() on a nil file panics.
    ❌ f, err := os.Open(...); defer f.Close() before checking err
    ✅ Check err != nil first, then defer f.Close()
  • TypeScript: Using sync file operations in a server — readFileSync blocks the event loop. Use fs/promises for async I/O.
    ❌ readFileSync in a request handler
    ✅ await readFile(...)
  • Rust: Not using BufReader/BufWriter — unbuffered I/O makes a syscall per read/write operation, which is extremely slow.
    ❌ Reading line by line directly from File
    ✅ BufReader::new(file) for buffered reading
  • C++: Not checking is_open() after constructing an fstream — construction does not throw on failure by default.
    ❌ ifstream in("file.txt"); string line; getline(in, line);
    ✅ if (!in.is_open()) { /* handle error */ }
Key Rules:
  • Always close files — use with (Python), defer (Go), RAII (Rust/C++), or try/finally (TS).
  • Buffer for performance — line-by-line reading should use buffered I/O. Reading entire files is fine for small files.
  • Handle encoding explicitly — always specify encoding="utf-8" (Python, TS) or be aware of the default (Go uses byte slices, Rust uses UTF-8 strings).

Language Comparison — Type Safety in JSON: Python deserializes JSON into plain dicts/lists — maximum flexibility but zero type safety. You can use pydantic or dataclasses_json for validation. Go uses struct tags (`json:"name"`) for field mapping — unknown fields are silently ignored by default. TypeScript’s JSON.parse() returns any — the type assertion to your interface is unchecked at runtime. Use Zod or io-ts for runtime validation. Rust’s serde is the gold standard: #[derive(Serialize, Deserialize)] generates efficient, type-safe serialization code at compile time. C++’s nlohmann/json provides a convenient API but runtime type checks throw exceptions on mismatch.
Common Mistakes:
  • Python: Using json.loads() on untrusted input without validation — the parsed dict might not have the keys or types you expect.
    ❌ data = json.loads(user_input); name = data["name"] — KeyError
    ✅ Validate with data.get("name") or use pydantic
  • Go: Forgetting to export struct fields (uppercase) for JSON marshaling — lowercase fields are silently skipped.
    ❌ type User struct { name string } — name won’t appear in JSON
    ✅ type User struct { Name string `json:"name"` }
  • TypeScript: Trusting JSON.parse() output as your interface type — no runtime validation happens.
    ❌ const user: User = JSON.parse(str) — might not be a valid User
    ✅ Validate with Zod schema or manual checks
  • Rust: Using .unwrap() on deserialization — malformed JSON panics your program.
    ❌ serde_json::from_str(&input).unwrap()
    ✅ serde_json::from_str(&input)? or handle the error
  • C++: Accessing a key that doesn’t exist in nlohmann::json — data["missing"] silently creates a null entry, while data.at("missing") throws.
    ❌ data["key"] on unknown keys — silent null insertion
    ✅ data.value("key", default) or data.contains("key") first
Key Rules:
  • Validate untrusted JSON — never assume JSON from an API or user input has the right shape. Use schema validation (pydantic, Zod, serde) when possible.
  • Handle missing/null fields explicitly — use Option (Rust), pointers (Go), ? (TS), .get() (Python) to avoid crashes on missing data.
  • Map struct fields to JSON names — use struct tags (Go), #[serde(rename)] (Rust), or custom serializers to match API naming conventions (camelCase vs snake_case).
