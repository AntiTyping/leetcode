1. Rust: Async Runtimes & tokio Language Deep Dive rep 0 • 14 days since review
Due today • first scheduled review (overdue 13 days)
Quick Recall: Rust's async fn returns a lazy Future that does nothing until polled. There is no built-in runtime — you choose an executor like tokio to poll futures, schedule tasks, and drive I/O. .await yields control at suspension points; tokio::spawn runs a future concurrently on the scheduler.

Key Pattern — spawn concurrent tasks and join them

Python 
import asyncio
async def work(n):
    await asyncio.sleep(0.01)
    return n * n
async def main():
    results = await asyncio.gather(*(work(i) for i in range(4)))
    print(results)            # [0, 1, 4, 9]
asyncio.run(main())
[0, 1, 4, 9]
Go 
// Go has no async/await: goroutines + channels are the runtime
results := make([]int, 4)
var wg sync.WaitGroup
for i := 0; i < 4; i++ {
    wg.Add(1)
    go func(n int) { defer wg.Done(); results[n] = n * n }(i)
}
wg.Wait()
fmt.Println(results)          // [0 1 4 9]
[0 1 4 9]
TypeScript 
const work = (n: number) =>
  new Promise<number>(r => setTimeout(() => r(n * n), 10));
const results = await Promise.all([0,1,2,3].map(work));
console.log(results);         // [ 0, 1, 4, 9 ]
[ 0, 1, 4, 9 ]
Rust 
use tokio::task::JoinSet;
#[tokio::main]
async fn main() {
    let mut set = JoinSet::new();
    for i in 0..4 { set.spawn(async move { i * i }); }
    let mut out = vec![];
    while let Some(r) = set.join_next().await { out.push(r.unwrap()); }
    out.sort();
    println!("{:?}", out);    // [0, 1, 4, 9]
}
[0, 1, 4, 9]
C++ 
#include <future>
#include <vector>
int main() {
    std::vector<std::future<int>> fs;
    for (int i = 0; i < 4; i++)
        fs.push_back(std::async(std::launch::async, [i]{ return i * i; }));
    for (auto& f : fs) std::cout << f.get() << " "; // 0 1 4 9
}
0 1 4 9
Challenge (all 5 languages):
1
Fetch-and-aggregate: given 10 async tasks each returning an int after a random delay, run them with a concurrency limit of 3 at a time and return their sum. In Rust, use tokio::sync::Semaphore to bound concurrency; in Python a Semaphore; in Go a buffered channel; in TS a counting limiter; in C++ a thread pool of 3.
#1 Pitfall per Language:
  • Python: calling await work() sequentially in a loop is NOT concurrent — you must gather/create tasks first.
  • Go: capturing the loop variable by reference (pre-1.22) makes every goroutine see the last value — pass it as an argument.
  • TypeScript: array.forEach(async ...) does not await — use Promise.all(map(...)).
  • Rust: a Future that is never .awaited never runs — futures are lazy, unlike JS Promises which start eagerly.
  • C++: std::async without std::launch::async may run lazily on .get() (deferred), defeating concurrency.
2. Rust: Macros — Declarative & Procedural Language Deep Dive rep 0 • 14 days since review
Due today • first scheduled review (overdue 13 days)
Quick Recall: Rust macros run at compile time and operate on token streams, not values. macro_rules! (declarative) matches token patterns; procedural macros (#[derive], attribute, function-like) are functions that transform a TokenStream. They are hygienic and type-checked after expansion. Other languages reach for code-gen, reflection, or templates instead.

Key Pattern — a variadic helper at compile time vs. runtime equivalents

Python 
# No macros: use *args / decorators for metaprogramming
def make_vec(*xs):
    return list(xs)
print(make_vec(1, 2, 3))      # [1, 2, 3]
[1, 2, 3]
Go 
// No macros: variadic funcs + go:generate for code-gen
func makeSlice[T any](xs ...T) []T { return xs }
fmt.Println(makeSlice(1, 2, 3))   // [1 2 3]
[1 2 3]
TypeScript 
// No macros: rest params + decorators (experimental) / codegen
const makeArr = <T,>(...xs: T[]): T[] => xs;
console.log(makeArr(1, 2, 3));    // [ 1, 2, 3 ]
[ 1, 2, 3 ]
Rust 
// Declarative macro: matches a repetition of expressions
macro_rules! myvec {
    ( $( $x:expr ),* $(,)? ) => {{
        let mut v = Vec::new();
        $( v.push($x); )*
        v
    }};
}
fn main() {
    let v: Vec<i32> = myvec![1, 2, 3];
    println!("{:?}", v);          // [1, 2, 3]
}
[1, 2, 3]
C++ 
// Variadic templates / fold expressions are the macro alternative
#include <vector>
template <typename... Ts>
auto make_vec(Ts... xs) { return std::vector{xs...}; }
int main() {
    auto v = make_vec(1, 2, 3);
    for (int x : v) std::cout << x << " "; // 1 2 3
}
1 2 3
Challenge (all 5 languages):
1
Build a hashmap!{ "a" => 1, "b" => 2 }-style constructor. In Rust write it as a macro_rules! macro; in the other four languages produce the closest idiomatic equivalent (variadic function, object literal, builder) that yields the same map. Note in your answer when the construction happens (compile time vs. runtime).
#1 Pitfall per Language:
  • Python: overusing exec/eval for "macros" — it skips compile-time checks and is a security risk.
  • Go: expecting macros at all — there are none; use go generate + templates and commit the output.
  • TypeScript: decorators are runtime, not compile-time rewrites — they cannot change a type's shape.
  • Rust: forgetting the $(,)? trailing-comma matcher, or using :expr where you need :tt for further matching.
  • C++: preprocessor #define macros are not hygienic — prefer constexpr/templates; wrap multi-statement macros in do{...}while(0).
3. C++: SFINAE & Template Metaprogramming Language Deep Dive rep 0 • 14 days since review
Due today • first scheduled review (overdue 13 days)
Quick Recall: SFINAE = "Substitution Failure Is Not An Error." When a template's substitution produces an invalid type/expression, that overload is silently dropped instead of erroring, enabling compile-time overload selection on type properties. Modern C++20 replaces most SFINAE with concepts and requires. Other languages use traits, type guards, or duck typing.

Key Pattern — dispatch differently for integral vs. floating types

Python 
def describe(x):
    return "int" if isinstance(x, int) else "float"
print(describe(3), describe(3.0))   # int float
int float
Go 
// Type constraints on generics are Go's "SFINAE"
type Num interface{ ~int | ~float64 }
func describe[T Num](x T) string {
    switch any(x).(type) {
    case int: return "int"
    default:  return "float"
    }
}
fmt.Println(describe(3), describe(3.0)) // int float
int float
TypeScript 
// Conditional types pick a branch on type properties
type Describe<T> = T extends number ? "number" : "other";
const d: Describe<number> = "number";
console.log(d);                     // number
number
Rust 
// Trait bounds select impls at compile time
trait Describe { fn kind() -> &'static str; }
impl Describe for i32 { fn kind() -> &'static str { "int" } }
impl Describe for f64 { fn kind() -> &'static str { "float" } }
fn main() { println!("{} {}", i32::kind(), f64::kind()); }
int float
C++ 
#include <type_traits>
template <class T,
  std::enable_if_t<std::is_integral_v<T>, int> = 0>
const char* describe(T) { return "int"; }
template <class T,
  std::enable_if_t<std::is_floating_point_v<T>, int> = 0>
const char* describe(T) { return "float"; }
int main() { std::cout << describe(3) << " " << describe(3.0); }
int float
Challenge (all 5 languages):
1
Write a generic print_size(c) that compiles only for types exposing a .size()/len() member and prints it. In C++ use SFINAE (or a C++20 requires clause) to detect the member; in Rust a trait bound; in Go a constraint interface; in TS a conditional/guarded type; in Python rely on duck typing but reject types lacking __len__ with a clear error.
#1 Pitfall per Language:
  • Python: hasattr checks at runtime only — no compile-time guarantee the attribute exists.
  • Go: a type that doesn't satisfy the constraint fails to compile, but the error points at the call site, not the missing method — read carefully.
  • TypeScript: conditional types are erased at runtime; they guide the type checker, not execution.
  • Rust: overlapping blanket impls cause coherence errors — you can't have two impls that could both apply.
  • C++: placing enable_if in the return type vs. a default template arg changes overload resolution; mismatched defaults make two overloads redeclarations.
4. C++: Coroutines (C++20) Language Deep Dive rep 0 • 14 days since review
Due today • first scheduled review (overdue 13 days)
Quick Recall: A C++20 coroutine is any function containing co_await, co_yield, or co_return. The compiler rewrites it into a state machine backed by a heap-allocated frame and a promise_type. The standard library ships the machinery but few ready-made types, so you usually write or import a generator/task. Conceptually it mirrors Python generators and JS async functions.

Key Pattern — a lazy generator yielding a sequence

Python 
def counter(n):
    i = 0
    while i < n:
        yield i
        i += 1
print(list(counter(3)))       # [0, 1, 2]
[0, 1, 2]
Go 
// Go 1.23 range-over-func gives generator-like iteration
func counter(n int) func(func(int) bool) {
    return func(yield func(int) bool) {
        for i := 0; i < n; i++ { if !yield(i) { return } }
    }
}
for v := range counter(3) { fmt.Print(v, " ") } // 0 1 2
0 1 2
TypeScript 
function* counter(n: number): Generator<number> {
  for (let i = 0; i < n; i++) yield i;
}
console.log([...counter(3)]); // [ 0, 1, 2 ]
[ 0, 1, 2 ]
Rust 
// Stable Rust: implement Iterator (generators are nightly)
struct Counter { i: u32, n: u32 }
impl Iterator for Counter {
    type Item = u32;
    fn next(&mut self) -> Option<u32> {
        if self.i < self.n { let v = self.i; self.i += 1; Some(v) } else { None }
    }
}
fn main() {
    let v: Vec<_> = Counter{ i:0, n:3 }.collect();
    println!("{:?}", v);      // [0, 1, 2]
}
[0, 1, 2]
C++ 
#include <generator>   // C++23 std::generator
std::generator<int> counter(int n) {
    for (int i = 0; i < n; i++) co_yield i;
}
int main() {
    for (int v : counter(3)) std::cout << v << " "; // 0 1 2
}
0 1 2
Challenge (all 5 languages):
1
Implement a lazy fib() producer that yields Fibonacci numbers indefinitely, then take the first 10. In C++ write a coroutine generator (or hand-roll the promise_type); in Python/TS use yield; in Rust implement Iterator; in Go use a range-over-func or channel. Confirm laziness: no value is computed before it is consumed.
#1 Pitfall per Language:
  • Python: a generator is exhausted after one full iteration — re-iterating yields nothing; call the factory again.
  • Go: a channel-based generator leaks a goroutine if the consumer stops early without a cancellation signal.
  • TypeScript: spreading an infinite generator ([...gen]) hangs — bound it with take.
  • Rust: returning references from next() fights the borrow checker — yield owned values or use std::iter::from_fn.
  • C++: the coroutine frame is heap-allocated and tied to the returned object — using a yielded reference after the generator is destroyed is UB.
5. Network Programming: TCP/UDP Sockets Systems & Architecture rep 0 • 12 days since review
Due today • first scheduled review (overdue 11 days)
Quick Recall: TCP is a connection-oriented, ordered, reliable byte stream — you listen/accept then read/write bytes (no message boundaries, so you frame messages yourself). UDP is connectionless datagrams: unordered, unreliable, but message-preserving and low-latency. All five languages wrap the same BSD socket API.

Key Pattern — a minimal TCP echo server

Python 
import socket
srv = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(("127.0.0.1", 9000)); srv.listen()
conn, addr = srv.accept()
data = conn.recv(1024)        # stream: may be partial
conn.sendall(data)            # echo back
conn.close()
# client sends "hi" -> receives "hi"
Go 
ln, _ := net.Listen("tcp", "127.0.0.1:9000")
conn, _ := ln.Accept()
buf := make([]byte, 1024)
n, _ := conn.Read(buf)        // stream read
conn.Write(buf[:n])           // echo
conn.Close()
// client sends "hi" -> receives "hi"
TypeScript 
import { createServer } from "node:net";
const server = createServer(sock => {
  sock.on("data", chunk => sock.write(chunk)); // echo
});
server.listen(9000, "127.0.0.1");
// client sends "hi" -> receives "hi"
Rust 
use std::io::{Read, Write};
use std::net::TcpListener;
let listener = TcpListener::bind("127.0.0.1:9000").unwrap();
let (mut sock, _) = listener.accept().unwrap();
let mut buf = [0u8; 1024];
let n = sock.read(&mut buf).unwrap();   // stream
sock.write_all(&buf[..n]).unwrap();     // echo
// client sends "hi" -> receives "hi"
C++ 
#include <sys/socket.h>
#include <netinet/in.h>
int s = socket(AF_INET, SOCK_STREAM, 0);
sockaddr_in a{}; a.sin_family = AF_INET;
a.sin_port = htons(9000); a.sin_addr.s_addr = INADDR_ANY;
bind(s, (sockaddr*)&a, sizeof(a)); listen(s, 1);
int c = accept(s, nullptr, nullptr);
char buf[1024]; int n = recv(c, buf, sizeof(buf), 0);
send(c, buf, n, 0);          // echo
// client sends "hi" -> receives "hi"

6. Web Frameworks & HTTP Servers Systems & Architecture rep 0 • 12 days since review
Due today • first scheduled review (overdue 11 days)
Quick Recall: An HTTP server maps a (method, path) to a handler that returns a status + body. Frameworks add routing, middleware, and (de)serialization on top of the language's HTTP primitives. The spectrum: batteries-included (Python/FastAPI, JS/Express) vs. standard-library-first (Go net/http) vs. typed + compiled (Rust/axum, C++/a library like Crow).

Numeric Precision

How numbers really behave: binary floating-point rounds, and fixed-width integers run out of room. Knowing the failure modes — and the exact-arithmetic escape hatches — separates correct numeric code from subtly wrong code.

Floating-Point & IEEE 754 Numeric Precision
Why it matters: every language stores double the same IEEE-754 way, so 0.1 + 0.2 != 0.3 everywhere. Equality checks, money math, NaN comparisons, and sorting floats are recurring real-world bugs — and the fixes (epsilon comparison, decimal types, total ordering) are the same five ideas in every language.

A binary double can't represent most decimal fractions exactly — 0.1 is stored as the nearest 64-bit value, so arithmetic accumulates tiny rounding error. The rules to internalize: never compare floats with == (compare within a tolerance), never store money as a float (use integer minor units or a decimal type), and remember NaN != NaN while ±Infinity comes from overflow or division by zero.

Example 1 — the classic 0.1 + 0.2

Python 
print(0.1 + 0.2)                 # 0.30000000000000004
print(0.1 + 0.2 == 0.3)         # False
0.30000000000000004 False
Go 
a, b := 0.1, 0.2                 // force float64 (not exact constants)
fmt.Println(a + b)               // 0.30000000000000004
fmt.Println(a+b == 0.3)          // false
0.30000000000000004 false
TypeScript 
console.log(0.1 + 0.2);          // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3);  // false
0.30000000000000004 false
Rust 
let (a, b) = (0.1_f64, 0.2_f64);
println!("{}", a + b);           // 0.30000000000000004
println!("{}", a + b == 0.3);    // false
0.30000000000000004 false
C++ 
#include <iomanip>
double a = 0.1, b = 0.2;
std::cout << std::setprecision(17) << (a + b) << "\n"; // 0.30000000000000004
std::cout << ((a + b) == 0.3);                          // 0
0.30000000000000004 0

False True

false true

0 1
Language Comparison: All five compute on the same IEEE-754 binary64, so the rounding error is identical. The differences are ergonomic. Python ships math.isclose (relative + absolute tolerance) and a decimal.Decimal for exact base-10. JS/TS have only number (a double) — no integer type — so bitwise ops truncate to 32 bits and integers above 2^53 lose precision (use BigInt). Go evaluates untyped float constants at high precision, which can mask the bug until you store them in a float64. Rust forces you to pick f32/f64 and, notably, floats don't implement Ord (because of NaN) — sorting needs partial_cmp or total_cmp. C++ hides the error behind the default 6-significant-digit stream precision until you set setprecision.
Common Mistakes:
❌ [All] Comparing floats with == — accumulated rounding makes mathematically-equal values differ.
✅ [All] Compare within a tolerance: absolute and relative (isclose, abs(a-b) < eps), sized to your magnitudes.
❌ [All] Storing currency as a double — 0.1 dollars isn't exact and errors compound over many ops.
✅ [All] Use integer minor units (cents) or a decimal type (Decimal, big.Rat, fixed-point) for money.
❌ [Rust] Trying to .sort() a Vec<f64> — it won't compile because f64: !Ord.
✅ [Rust] Sort with sort_by(|a, b| a.total_cmp(b)) to get a deterministic NaN-aware ordering.
❌ [JS] Assuming all integers are exact — above Number.MAX_SAFE_INTEGER (2^53−1) they aren't.
✅ [JS] Use BigInt for exact large integers and Number.isSafeInteger to validate.
Key Rules:
  • Never compare floats with ==; compare within an epsilon (absolute + relative).
  • Money and exact decimals → integer minor units or a decimal type, never binary float.
  • NaN != NaN (test with isNaN/is_nan); overflow and x/0.0 give ±Infinity.
  • Mind the width: f32 ≈ 7 significant digits, f64 ≈ 15–17.

Arbitrary-Precision Integers (Big Integers) Numeric Precision
Why it matters: fixed-width integers silently wrap (or invoke UB) once a value exceeds 64 bits — fatal for cryptography, combinatorics, factorials, and exact financial sums. Languages diverge sharply here: Python integers are bignums by default, while others need a dedicated type or third-party library.

A big integer grows its storage to hold any magnitude exactly, at the cost of heap allocation and slower arithmetic. The cross-language spectrum runs from "free" (Python's native int) to "built-in but explicit" (JS BigInt, Go math/big) to "needs a crate/library" (Rust num-bigint, C++ Boost/GMP). The decision: reach for bignum only when the range is genuinely unbounded; otherwise a fixed type is far faster.

Go 
var x int64 = 9223372036854775807
fmt.Println(x + 1)                                   // -9223372036854775808 (wraps!)
b := big.NewInt(x)
fmt.Println(b.Add(b, big.NewInt(1)))                 // 9223372036854775808
-9223372036854775808 9223372036854775808
TypeScript 
console.log(Number.MAX_SAFE_INTEGER + 2);            // 9007199254740994 (off by one)
console.log(BigInt(Number.MAX_SAFE_INTEGER) + 2n);   // 9007199254740993n
9007199254740994 9007199254740993n
Rust 
let x: i64 = i64::MAX;
println!("{}", x.wrapping_add(1)); // -9223372036854775808  (x + 1 panics in debug)
let big = BigInt::from(x) + 1;
println!("{}", big);               // 9223372036854775808
-9223372036854775808 9223372036854775808
C++ 
long long x = 9223372036854775807LL;
std::cout << x + 1 << "\n";        // UB (signed overflow); commonly wraps
cpp_int b = x; b += 1;
std::cout << b;                    // 9223372036854775808
-9223372036854775808 9223372036854775808
Language Comparison: Python is the outlier: int is arbitrary-precision natively, so naive code "just works" (at a hidden allocation cost on huge values). JS/TS add BigInt (the n suffix), but you cannot mix BigInt and Number in one expression (a TypeError), and BigInt isn't JSON.stringify-able. Go's math/big is a value type whose methods mutate the receiver and return it — powerful but aliasing-prone. Rust and C++ have no standard bignum: pull in num-bigint or Boost.Multiprecision/GMP (C++'s non-standard __int128 only doubles the width, it isn't unbounded).
Common Mistakes:
❌ [Go] Assuming big.Int methods return a fresh value — z.Add(x, y) writes into z and aliasing the same variable corrupts results.
✅ [Go] Allocate a distinct destination (new(big.Int)) for results you intend to keep.
❌ [TypeScript] Mixing types — 1n + 1 throws; passing a BigInt to JSON.stringify throws too.
✅ [TypeScript] Keep arithmetic all-BigInt, and serialize via an explicit .toString().
❌ [Rust/C++] Forgetting bignum is a dependency — there's nothing in std to reach for.
✅ [Rust/C++] Add num-bigint / Boost.Multiprecision (or GMP) up front when exact large integers are required.
❌ [All] Defaulting to bignum "to be safe" on hot paths — it allocates and is far slower than fixed ints.
✅ [All] Use a fixed-width type when the range is bounded; reserve bignum for genuinely unbounded values.
Key Rules:
  • Exact integers beyond 64 bits: Python int, JS BigInt, Go math/big, Rust num-bigint, C++ Boost/GMP.
  • Bignum trades speed and allocation for unbounded range — don't pay for it when a fixed type fits.
  • Watch the footguns: type mixing (JS), in-place mutation/aliasing (Go), missing-from-std (Rust/C++).

Arguments & Nullability

How functions receive variable, optional, and named inputs — and how each language models the absence of a value. These are everyday mechanics where the five languages diverge the most, from Python's rich keyword args to Go having none, and from raw nullable pointers to type-checked Option.

Variadic Functions & Argument Spreading Function Mechanics
Why it matters: printf, max, loggers, and constructors all accept "any number of arguments," and you constantly need to spread an existing collection into such a call. The syntax and — crucially — the copy/allocation semantics differ per language, and Rust has no variadic functions at all.

A variadic function collects trailing arguments into one sequence; spreading does the reverse, expanding a collection back into individual arguments. Python uses *args/**kwargs, JS/TS and Go use rest (...) and spread, C++ uses variadic templates (type-safe, compile-time), and Rust deliberately omits the feature — you pass a slice or write a macro (println! itself is one).

Example 1 — define total(...) and spread a list in

Python 
def total(*nums): return sum(nums)
print(total(1, 2, 3))            # 6
xs = [4, 5, 6]
print(total(*xs))                # 15   (spread)
6 15
Go 
func total(nums ...int) int { s := 0; for _, n := range nums { s += n }; return s }
fmt.Println(total(1, 2, 3))      // 6
xs := []int{4, 5, 6}
fmt.Println(total(xs...))         // 15
6 15
TypeScript 
const total = (...nums: number[]) => nums.reduce((a, b) => a + b, 0);
console.log(total(1, 2, 3));     // 6
const xs = [4, 5, 6];
console.log(total(...xs));        // 15
6 15
Rust 
// No variadics: take a slice; callers pass &[...] or &vec
fn total(nums: &[i32]) -> i32 { nums.iter().sum() }
fn main() {
    println!("{}", total(&[1, 2, 3]));   // 6
    let xs = vec![4, 5, 6];
    println!("{}", total(&xs));          // 15
}
6 15
C++ 
// Variadic template + fold expression (type-safe, compile-time)
template <typename... Ts>
auto total(Ts... nums) { return (nums + ... + 0); }
int main() { std::cout << total(1, 2, 3); } // 6
6

Example 2 — forward args to a formatting call

Python 
def logf(fmt, *args): print(fmt.format(*args))
logf("{} + {} = {}", 1, 2, 3)    # 1 + 2 = 3
1 + 2 = 3
Go 
func logf(format string, args ...any) { fmt.Printf(format+"\n", args...) }
logf("%d + %d = %d", 1, 2, 3)    // 1 + 2 = 3
1 + 2 = 3
TypeScript 
const logf = (label: string, ...args: unknown[]) => console.log(label, ...args);
logf("sum:", 1, 2, 3);           // sum: 1 2 3
sum: 1 2 3
Rust 
// Variadic-like behavior only through macros; println! IS one
macro_rules! logf { ($($a:expr),*) => { println!($($a),*) }; }
fn main() { logf!("{} + {} = {}", 1, 2, 3); } // 1 + 2 = 3
1 + 2 = 3
C++ 
#include <cstdio>
template <typename... Ts>
void logf(const char* fmt, Ts... args) { std::printf(fmt, args...); }
int main() { logf("%d + %d\n", 1, 2); } // 1 + 2
1 + 2
Language Comparison: Python's *args packs into a tuple and **kwargs into a dict — the most flexible. JS/TS rest params build a real Array, and spread (...) works in calls and literals. Go's ...T receives a slice, and forwarding with slice... passes the same backing array (no copy) — plus ...any boxes each argument, an allocation cost. C++ variadic templates are fully type-checked and resolved at compile time (fold expressions for reduction); the legacy C va_list is type-unsafe and a format-string hazard. Rust has no variadic functions (outside extern "C") — idiomatic code takes a slice or uses a macro.
Common Mistakes:
❌ [Go] Forwarding s... and assuming a copy — the callee shares your slice's memory and can mutate it.
✅ [Go] Pass append([]T(nil), s...) when the callee must not see later mutations; remember ...any allocates.
❌ [Rust] Expecting C-style variadic functions — they don't exist for safe Rust.
✅ [Rust] Accept a &[T] slice, or define a macro_rules! macro for true variadic syntax.
❌ [C/C++] Using C va_arg/printf with a mismatched format specifier — UB and a security hole.
✅ [C++] Prefer variadic templates / std::format, which are type-checked.
❌ [TypeScript] Putting a rest parameter anywhere but last, or spreading a massive array into a call.
✅ [TypeScript] Keep ...rest last; for huge collections, pass the array itself and iterate.
Key Rules:
  • Collect many args: Python *args, JS/Go/TS rest ..., C++ template packs. Spread back with the same .../* syntax.
  • Rust has no variadic functions — take a slice or write a macro.
  • Go's slice... forwarding shares memory (no copy); C va_arg is unsafe — prefer typed alternatives.

Builds on Higher-Order Functions and Closures; the Rust "macro instead of variadic" point ties directly to review card #2 (Rust: Macros), and the named-argument alternative continues in Default & Named Arguments below.
Default & Named (Keyword) Arguments Function Mechanics
Why it matters: optional parameters and self-documenting call sites (connect(timeout=30)) reduce overloads and bugs. But support is wildly uneven: Python has first-class keyword args, Go has none by design, and the idioms to emulate them (options structs, builders, Default) are core to each ecosystem.

A default lets a caller omit a parameter; a named/keyword argument lets them pass it by name for clarity and order-independence. Python supports both natively (including keyword-only via *); TypeScript uses defaults plus the ubiquitous "options object"; C++ has trailing default arguments and C++20 designated initializers; Rust leans on Option, the Default trait with struct-update syntax, or a builder; Go offers neither and standardizes on an options struct or functional options.

Example 1 — default values

Python 
def greet(name, greeting="Hello", excited=False):
    return f"{greeting}, {name}{'!' if excited else '.'}"
print(greet("Sam"))                       # Hello, Sam.
print(greet("Sam", excited=True))         # Hello, Sam!
Hello, Sam. Hello, Sam!
Go 
// No defaults: an options struct (zero values stand in for "unset")
type GreetOpts struct{ Greeting string; Excited bool }
func greet(name string, o GreetOpts) string {
    if o.Greeting == "" { o.Greeting = "Hello" }
    end := "."; if o.Excited { end = "!" }
    return o.Greeting + ", " + name + end
}
fmt.Println(greet("Sam", GreetOpts{Excited: true})) // Hello, Sam!
Hello, Sam!
TypeScript 
function greet(name: string, greeting = "Hello", excited = false) {
  return `${greeting}, ${name}${excited ? "!" : "."}`;
}
console.log(greet("Sam"));                 // Hello, Sam.
console.log(greet("Sam", "Hi", true));     // Hi, Sam!
Hello, Sam. Hi, Sam!
Rust 
// No defaults: Option for optional params
fn greet(name: &str, greeting: Option<&str>, excited: bool) -> String {
    let g = greeting.unwrap_or("Hello");
    format!("{}, {}{}", g, name, if excited { "!" } else { "." })
}
fn main() { println!("{}", greet("Sam", None, true)); } // Hello, Sam!
Hello, Sam!
C++ 
#include <string>
std::string greet(const std::string& name,
                  const std::string& greeting = "Hello",
                  bool excited = false) {       // trailing defaults
    return greeting + ", " + name + (excited ? "!" : ".");
}
int main() { std::cout << greet("Sam"); } // Hello, Sam.
Hello, Sam.

Example 2 — named arguments for clarity

Python 
# '*' marks the following params keyword-only
def make_window(width=800, height=600, *, resizable=True):
    return (width, height, resizable)
print(make_window(height=480, resizable=False))  # (800, 480, False)
(800, 480, False)
Go 
// Struct field names give a "named argument" feel
type WinOpts struct{ Width, Height int; Resizable bool }
w := WinOpts{Height: 480, Resizable: false}      // Width defaults to 0
fmt.Println(w.Width, w.Height, w.Resizable)      // 0 480 false
0 480 false
TypeScript 
function makeWindow(o: { width?: number; height?: number; resizable?: boolean } = {}) {
  const { width = 800, height = 600, resizable = true } = o;  // destructure + defaults
  return [width, height, resizable];
}
console.log(makeWindow({ height: 480, resizable: false })); // [ 800, 480, false ]
[ 800, 480, false ]
Rust 
#[derive(Default)]
struct WinOpts { width: u32, height: u32, resizable: bool }
fn main() {
    let o = WinOpts { height: 480, ..Default::default() }; // struct-update
    println!("{} {} {}", o.width, o.height, o.resizable);  // 0 480 false
}
0 480 false
C++ 
// C++20 designated initializers name the fields
struct WinOpts { int width = 800; int height = 600; bool resizable = true; };
int main() {
    WinOpts o{ .height = 480, .resizable = false };
    std::cout << o.width << " " << o.height << " " << o.resizable; // 800 480 0
}
800 480 0
Language Comparison: Python is richest: defaults, keyword arguments, and keyword-only parameters (after *). TypeScript combines parameter defaults with the "options object" + destructuring-default idiom. C++ has trailing-only default arguments and, since C++20, designated initializers that name struct fields. Rust has neither defaults nor keyword args by language design — you reach for Option, #[derive(Default)] + ..Default::default(), or the builder pattern. Go deliberately omits both; the canonical replacements are an options struct or the "functional options" pattern (a variadic of func(*Config)).
Common Mistakes:
❌ [Python] def f(x=[]) — the mutable default is created once and shared across all calls, accumulating state.
✅ [Python] Default to None and build the mutable value inside the function body.
❌ [Go] Fighting the language by faking overloads/defaults with variadics of mixed types.
✅ [Go] Use an options struct or functional options (...Option) — idiomatic and self-documenting.
❌ [TypeScript] Expecting a defaulted parameter to trigger on null — it only fills in for undefined.
✅ [TypeScript] Pass undefined (or omit) to get the default; guard null explicitly if it's possible.
❌ [C++] Repeating a default argument in multiple declarations/headers — an ODR/redeclaration trap; defaults must be trailing.
✅ [C++] Specify each default exactly once (typically the header declaration); consider designated initializers instead.
Key Rules:
  • Defaults exist in Python/TS/C++ (C++ trailing-only); Rust and Go have none.
  • Named arguments: Python native, TS via options object, C++20 designated initializers, Rust struct-update, Go options/functional-options.
  • Never use a mutable default value in Python — sentinel None and construct inside.

Nullability & Optional Values Function Mechanics
Why it matters: "no value" is everywhere — a missing map key, a failed parse, an empty result. How a language models absence decides whether you get a runtime null-dereference crash (Tony Hoare's "billion-dollar mistake") or a compiler-checked Option you must handle. The spectrum runs from raw nullable pointers to type-level optionals.

The five languages sit at different points: C++ raw nullptr (unchecked, UB on deref) and the opt-in std::optional; Go nil plus the comma-ok idiom; Python the None singleton (only advisory typing); JS/TS null/undefined, made safe under strictNullChecks with ?. and ??; and Rust the gold standard — Option<T> in the type system, no null, with the compiler forcing you to handle None.

Example 1 — represent and check absence (missing map key)

Python 
v = {"a": 1}.get("b")            # None if missing
print(v is None)                 # True
print((v or 0) + 5)              # 5
True 5
Go 
m := map[string]int{"a": 1}
v, ok := m["b"]                  // comma-ok idiom
fmt.Println(ok, v)               // false 0
// pointers also model optional: var p *int = nil
false 0
TypeScript 
const m = new Map([["a", 1]]);
const v: number | undefined = m.get("b");
console.log(v ?? 0);             // 0   (nullish coalescing)
0
Rust 
use std::collections::HashMap;
let m = HashMap::from([("a", 1)]);
let v: Option<&i32> = m.get("b");
println!("{}", v.copied().unwrap_or(0)); // 0
0
C++ 
#include <optional>
std::optional<int> lookup(bool present) {
    return present ? std::optional<int>{1} : std::nullopt;
}
int main() { std::cout << lookup(false).value_or(0); } // 0
0

Example 2 — safe propagation / chaining

Python 
import math
def safe_sqrt(x): return math.sqrt(x) if x >= 0 else None
r = safe_sqrt(-1)
print(r if r is not None else "no real root")   # no real root
no real root
Go 
func safeSqrt(x float64) (float64, bool) {
    if x < 0 { return 0, false }
    return math.Sqrt(x), true
}
if r, ok := safeSqrt(-1); ok { fmt.Println(r) } else { fmt.Println("no real root") }
no real root
TypeScript 
const user: { profile?: { name?: string } } = {};
console.log(user.profile?.name ?? "anon"); // anon  (optional chaining + ??)
anon
Rust 
fn safe_sqrt(x: f64) -> Option<f64> {
    if x >= 0.0 { Some(x.sqrt()) } else { None }
}
fn main() {
    let msg = safe_sqrt(-1.0).map(|r| r.to_string())
        .unwrap_or_else(|| "no real root".into());
    println!("{}", msg);                 // no real root
}
no real root
C++ 
#include <optional>
#include <cmath>
std::optional<double> safe_sqrt(double x) {
    if (x >= 0) return std::sqrt(x);
    return std::nullopt;
}
int main() {
    auto r = safe_sqrt(-1);
    std::cout << (r ? std::to_string(*r) : "no real root"); // no real root
}
no real root
Language Comparison: Rust leads: Option<T> is a type the compiler forces you to unwrap (match, if let, combinators, ?), so a forgotten check is a compile error, not a crash. TypeScript under strictNullChecks tracks T | undefined in the type system, with ?. and ?? for ergonomic handling. C++ offers std::optional<T> (C++17) for value-optionality — but raw pointers and nullptr still exist and dereferencing one is UB. Python's None is a singleton with no static enforcement (Optional[T] hints are advisory). Go has no Option type; the conventions are the comma-ok idiom and nil pointers — with the infamous "nil interface" gotcha lurking.
Common Mistakes:
❌ [C++] Dereferencing a null pointer or an empty optional — *opt on empty is UB; .value() throws.
✅ [C++] Check first (if (opt)) or use value_or(default); prefer optional over raw pointers for "maybe a value."
❌ [Go] The nil-interface trap: an interface holding a typed nil pointer is not == nil, so err != nil is unexpectedly true.
✅ [Go] Return a literal nil error (not a typed nil pointer wrapped in the interface); guard pointer derefs.
❌ [JS/TS] Using || for defaults — it also overrides valid 0, "", and false.
✅ [JS/TS] Use ?? (nullish coalescing), which only falls back on null/undefined.
❌ [Rust] Sprinkling .unwrap() everywhere — it reintroduces the very panic Option was meant to prevent.
✅ [Rust] Handle with match/if let/? or unwrap_or(_else/_default); reserve unwrap for proven invariants.
Key Rules:
  • Prefer a typed optional (Rust Option, C++ std::optional, TS strict T | undefined) over sentinel nulls when available.
  • Use ?./?? (TS), ? and combinators (Rust), value_or (C++), comma-ok (Go) instead of bare dereferences.
  • null/None/nil means "absent," not 0/""/false — keep the two distinct.
