1. Python: asyncio Internals & the GIL Language Deep Dive rep 2 • 4 days overdue
4 days overdue
Quick Recall: asyncio runs a single-threaded event loop with cooperative multitasking — tasks yield at await points. The GIL prevents true parallelism for CPU-bound work in CPython, so use processes (or other languages) for CPU work, asyncio for I/O-bound concurrency.

Key Patterns — fan out concurrent work

Python 
import asyncio

async def work(n):
    await asyncio.sleep(n)
    return n * 10

async def main():
    # gather runs coroutines concurrently on one thread
    results = await asyncio.gather(work(1), work(2), work(3))
    print(results)  # [10, 20, 30]

asyncio.run(main())
Go 
// goroutines + channel: true parallelism, no GIL
func main() {
    ch := make(chan int, 3)
    for _, n := range []int{1, 2, 3} {
        go func(n int) {
            time.Sleep(time.Duration(n) * time.Second)
            ch <- n * 10
        }(n)
    }
    var out []int
    for i := 0; i < 3; i++ { out = append(out, <-ch) }
    fmt.Println(out) // order varies: e.g. [10 20 30]
}
TypeScript 
const work = (n: number) =>
  new Promise<number>(r => setTimeout(() => r(n * 10), n * 1000));

async function main() {
  const results = await Promise.all([work(1), work(2), work(3)]);
  console.log(results); // [10, 20, 30]
}
main();
Rust 
use tokio::time::{sleep, Duration};

async fn work(n: u64) -> u64 {
    sleep(Duration::from_secs(n)).await;
    n * 10
}

#[tokio::main]
async fn main() {
    let (a, b, c) = tokio::join!(work(1), work(2), work(3));
    println!("{:?}", [a, b, c]); // [10, 20, 30]
}
C++ 
#include <future>
#include <vector>
#include <iostream>

int work(int n) {
    std::this_thread::sleep_for(std::chrono::seconds(n));
    return n * 10;
}

int main() {
    auto f1 = std::async(std::launch::async, work, 1);
    auto f2 = std::async(std::launch::async, work, 2);
    auto f3 = std::async(std::launch::async, work, 3);
    std::cout << f1.get() << " " << f2.get() << " " << f3.get() << "\n"; // 10 20 30
}
Review Challenge
1
Write a program that fetches 3 URLs (simulate with sleeps of differing durations) concurrently and prints each result as it arrives (not in fixed order). Implement in all 5 languages.
#1 Pitfall per Language
  • Python: Forgetting await — calling work(1) without awaiting creates a coroutine object that never runs (and warns "coroutine was never awaited").
  • Go: Goroutine leak — launching a goroutine that blocks forever because nobody reads the channel and there's no context cancel.
  • TS: Unhandled promise rejection — a rejected promise not caught crashes the process (Node) or logs silently.
  • Rust: .await on the wrong runtime — awaiting a tokio future inside an async-std executor (or no #[tokio::main]) panics.
  • C++: Forgetting to .get() the future — the std::async destructor may block, silently serializing your "concurrent" work.
2. Go: context Package & reflect Language Deep Dive rep 2 • 4 days overdue
4 days overdue
Quick Recall: context.Context carries deadlines, cancellation signals, and request-scoped values across API boundaries — pass it as the first argument and propagate it down. reflect enables runtime type inspection when generics aren't enough.

Key Patterns — timeout after 2 seconds

Python 
import asyncio

async def slow():
    await asyncio.sleep(5)
    return "done"

async def main():
    try:
        result = await asyncio.wait_for(slow(), timeout=2.0)
    except asyncio.TimeoutError:
        result = "timed out"
    print(result)  # timed out

asyncio.run(main())
Go 
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

ch := make(chan string, 1)
go func() {
    time.Sleep(5 * time.Second)
    ch <- "done"
}()

select {
case res := <-ch:
    fmt.Println(res)
case <-ctx.Done():
    fmt.Println("timed out:", ctx.Err()) // context deadline exceeded
}
TypeScript 
function withTimeout<T>(p: Promise<T>, ms: number): Promise<T> {
  const ctrl = new AbortController();
  const timeout = new Promise<never>((_, rej) =>
    setTimeout(() => { ctrl.abort(); rej(new Error("timed out")); }, ms));
  return Promise.race([p, timeout]);
}

const slow = new Promise(r => setTimeout(() => r("done"), 5000));
withTimeout(slow, 2000).catch(e => console.log(e.message)); // timed out
Rust 
use tokio::time::{timeout, sleep, Duration};

#[tokio::main]
async fn main() {
    let slow = async {
        sleep(Duration::from_secs(5)).await;
        "done"
    };
    match timeout(Duration::from_secs(2), slow).await {
        Ok(v) => println!("{v}"),
        Err(_) => println!("timed out"),
    }
}
C++ 
#include <future>
#include <chrono>
#include <iostream>

int main() {
    auto fut = std::async(std::launch::async, []{
        std::this_thread::sleep_for(std::chrono::seconds(5));
        return std::string("done");
    });
    if (fut.wait_for(std::chrono::seconds(2)) == std::future_status::ready)
        std::cout << fut.get() << "\n";
    else
        std::cout << "timed out\n"; // (future still runs in background)
}
Review Challenge
1
Implement a function that runs a "slow" operation but gives up after 2 seconds, returning a timeout sentinel. Propagate the cancellation so the slow work actually stops where the language allows it. Implement in all 5 languages.
#1 Pitfall per Language
  • Go: Not propagating context — spawning child work with context.Background() instead of the passed ctx breaks cancellation.
  • Python: Not setting a timeout on the asyncio call — await slow() with no wait_for hangs forever.
  • TS: Not using AbortController — Promise.race rejects but the underlying fetch/work keeps running.
  • Rust: select! without a cancel branch — the dropped future cancels, but holding a guard across .await can deadlock.
  • C++: No built-in context — wait_for only times out the wait; the task keeps running. You must wire your own cancellation flag (std::atomic<bool>).
3. Mocking & Test Doubles Testing & Error Handling rep 2 • 2 days overdue
2 days overdue
Quick Recall: Mocks verify behavior (was X called, with what args?). Stubs return canned responses. Fakes are lightweight working implementations (e.g., in-memory DB). Depend on interfaces/protocols so you can swap a double in for the real collaborator.

Key Patterns — UserService depends on a Database

Python 
from unittest.mock import MagicMock

def test_get_username():
    db = MagicMock()
    db.fetch.return_value = {"id": 1, "name": "Ada"}
    svc = UserService(db)
    assert svc.username(1) == "Ada"
    db.fetch.assert_called_once_with(1)  # behavior verification
Go 
// interface-based: hand-rolled mock implementing Database
type fakeDB struct{ called int }
func (f *fakeDB) Fetch(id int) (User, error) {
    f.called++
    return User{ID: 1, Name: "Ada"}, nil
}

func TestUsername(t *testing.T) {
    db := &fakeDB{}
    svc := UserService{db}
    if got := svc.Username(1); got != "Ada" { t.Fatal(got) }
    if db.called != 1 { t.Fatalf("Fetch called %d times", db.called) }
}
TypeScript 
test("username", () => {
  const fetch = jest.fn().mockReturnValue({ id: 1, name: "Ada" });
  const db = { fetch } as unknown as Database;
  const svc = new UserService(db);
  expect(svc.username(1)).toBe("Ada");
  expect(fetch).toHaveBeenCalledWith(1);
});
Rust 
use mockall::automock;

#[automock]
trait Database { fn fetch(&self, id: u32) -> User; }

#[test]
fn username() {
    let mut db = MockDatabase::new();
    db.expect_fetch()
      .with(eq(1))
      .times(1)
      .returning(|_| User { id: 1, name: "Ada".into() });
    let svc = UserService::new(db);
    assert_eq!(svc.username(1), "Ada");
}
C++ 
#include <gmock/gmock.h>
using ::testing::Return;

class MockDB : public Database {
public:
    MOCK_METHOD(User, Fetch, (int id), (override));
};

TEST(UserServiceTest, Username) {
    MockDB db;
    EXPECT_CALL(db, Fetch(1)).Times(1).WillOnce(Return(User{1, "Ada"}));
    UserService svc(&db);
    EXPECT_EQ(svc.Username(1), "Ada");
}
Review Challenge
1
Given a UserService that depends on a Database interface, write a test using a mock that (a) stubs fetch to return a fixed user and (b) verifies fetch was called exactly once with the right id. Implement in all 5 languages.
#1 Pitfall per Language
  • Python: Patching the wrong path — mock.patch must target where the name is used, not where it's defined.
  • Go: Mocking concrete structs — you can only swap doubles if the dependency is an interface.
  • TS: Forgetting mockReset/clearMocks — call counts leak between tests.
  • Rust: mockall requires the dependency be a trait and the mock built before the SUT borrows it.
  • C++: Unmet EXPECT_CALL only fails at mock destruction — let the mock go out of scope before the test ends.
4. Benchmarking Testing & Error Handling rep 2 • 2 days overdue
2 days overdue
Quick Recall: Micro-benchmarks measure isolated operations. Beware: the compiler optimizing away dead code, not warming up the JIT, and measuring noise. Always run multiple iterations and report a stable statistic (mean/median), never a single timing.

Key Patterns

Python 
import timeit

t = timeit.timeit(
    stmt="sum(range(1000))",
    number=100_000,
)
print(f"{t/100_000*1e6:.3f} us/op")
Go 
// run: go test -bench=. -benchmem
func BenchmarkSum(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        s := 0
        for j := 0; j < 1000; j++ { s += j }
        _ = s
    }
}
TypeScript 
const N = 100_000;
const start = performance.now();
for (let k = 0; k < N; k++) {
  let s = 0;
  for (let j = 0; j < 1000; j++) s += j;
  if (s < 0) console.log(s); // prevent dead-code elimination
}
console.log(((performance.now() - start) / N * 1000).toFixed(3), "us/op");
Rust 
// Cargo.toml: criterion under [dev-dependencies]
use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn bench(c: &mut Criterion) {
    c.bench_function("sum", |b| b.iter(|| {
        let s: u64 = (0..black_box(1000)).sum();
        black_box(s)
    }));
}
criterion_group!(benches, bench);
criterion_main!(benches);
C++ 
#include <benchmark/benchmark.h>

static void BM_Sum(benchmark::State& state) {
    for (auto _ : state) {
        long s = 0;
        for (int j = 0; j < 1000; ++j) s += j;
        benchmark::DoNotOptimize(s); // keep the work
    }
}
BENCHMARK(BM_Sum);
BENCHMARK_MAIN();
Review Challenge
1
Benchmark hash-map insertion of 1M elements vs. building a sorted vector and doing binary search lookups. Report ns/op and compare. Implement in all 5 languages, guarding against dead-code elimination.
#1 Pitfall per Language
  • Python: Including import/setup cost in the timed loop — use the setup argument of timeit.
  • Go: Not using b.N — a fixed loop count ignores the framework's auto-scaling and gives noisy results.
  • TS: JIT warmup — the first iterations run interpreted; discard them or run enough to stabilize.
  • Rust: Forgetting black_box — the optimizer deletes pure computations whose result is unused.
  • C++: Forgetting DoNotOptimize — -O2 elides the entire loop body.
5. Serialization & Deserialization Advanced Patterns rep 1 • 5 days overdue
5 days overdue
Quick Recall: Serialization converts structured data to bytes/text for storage or transmission; deserialization reverses it. JSON is the lingua franca; binary formats (pickle, gob, bincode) trade portability for speed/size.

Key Patterns — round-trip a struct through JSON

Python 
import json
from dataclasses import dataclass, asdict

@dataclass
class Point: x: int; y: int

p = Point(1, 2)
s = json.dumps(asdict(p))        # '{"x": 1, "y": 2}'
back = Point(**json.loads(s))
print(back == p)                 # True
Go 
type Point struct {
    X int `json:"x"`
    Y int `json:"y"`
}

p := Point{1, 2}
b, _ := json.Marshal(p)          // {"x":1,"y":2}
var back Point
json.Unmarshal(b, &back)
fmt.Println(back == p)           // true
TypeScript 
interface Point { x: number; y: number; }

const p: Point = { x: 1, y: 2 };
const s = JSON.stringify(p);     // '{"x":1,"y":2}'
const back = JSON.parse(s) as Point;
console.log(back.x === p.x && back.y === p.y); // true
Rust 
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, PartialEq, Debug)]
struct Point { x: i32, y: i32 }

let p = Point { x: 1, y: 2 };

C++ 
#include <nlohmann/json.hpp>

#1 Pitfall per Language
  • Python: pickle can execute arbitrary code on load — never unpickle untrusted data; use JSON for interchange.

6. Design Patterns: Iterator, Observer, Strategy Advanced Patterns rep 1 • 5 days overdue
5 days overdue
Quick Recall: Iterator externalizes traversal so the consumer doesn't know the container's internals. Observer decouples event producers from consumers (publish/subscribe). Strategy encapsulates interchangeable algorithms behind a common interface.

Key Patterns — the Iterator protocol

Python 
class Countdown:
    def __init__(self, n): self.n = n
    def __iter__(self): return self
    def __next__(self):
        if self.n <= 0: raise StopIteration
        self.n -= 1
        return self.n + 1

print(list(Countdown(3)))  # [3, 2, 1]
Go 
// channels as observer/iterator
func countdown(n int) <-chan int {
    ch := make(chan int)
    go func() {
        defer close(ch)
        for ; n > 0; n-- { ch <- n }
    }()
    return ch
}

for v := range countdown(3) { fmt.Print(v, " ") } // 3 2 1
TypeScript 
class Countdown {
  constructor(private n: number) {}
  *[Symbol.iterator]() {
    for (let i = this.n; i > 0; i--) yield i;
  }
}
console.log([...new Countdown(3)]); // [3, 2, 1]
Rust 
struct Countdown(u32);
impl Iterator for Countdown {
    type Item = u32;
    fn next(&mut self) -> Option<u32> {
        if self.0 == 0 { None } else { self.0 -= 1; Some(self.0 + 1) }
    }
}
let v: Vec<_> = Countdown(3).collect(); // [3, 2, 1]
C++ 
#include <vector>
#include <iostream>
// begin()/end() makes a type range-iterable
std::vector<int> countdown(int n) {
    std::vector<int> v;
    for (; n > 0; --n) v.push_back(n);
    return v;
}
for (int x : countdown(3)) std::cout << x << " "; // 3 2 1
Review Challenge
1
Implement the Observer pattern for a simple event bus: a Bus with subscribe(handler) and publish(event), where every subscriber receives every published event. Implement in all 5 languages.
#1 Pitfall per Language
  • Python: Returning a fresh iterator from __iter__ vs. self — reusing self makes the iterable single-pass.
  • Go: Leaking the goroutine behind a channel iterator if the consumer stops ranging early — use context or a done channel.
  • TS: Confusing Symbol.iterator (sync) with Symbol.asyncIterator — for await needs the async one.
  • Rust: Observer with shared mutable subscribers needs Rc<RefCell<...>> (single-thread) or Arc<Mutex<...>> (multi-thread).
  • C++: Dangling observers — if a subscriber is destroyed without unsubscribing, the bus holds a stale pointer.

Section 1: Performance & Optimization

You can't optimize what you can't measure. Profile first, then cache, then parallelize safely.

Profiling & Performance Analysis Performance & Optimization
You can't optimize what you can't measure. Profiling identifies the actual bottleneck instead of the one you guessed at — and the guess is wrong more often than not.

A profiler samples or instruments a running program to attribute time (CPU profiling) or allocations (memory profiling) to specific functions and lines. A flame graph visualizes the call stack: width = time spent, stacked bars = call depth. The widest bars at the top are your hot paths.

Start by profiling a function that does obvious unnecessary work — here, a naive recursive Fibonacci that recomputes subproblems exponentially — and watch the profiler point straight at the recursion.

1. CPU-profile a hot function

Python 
import cProfile, pstats

def fib(n): return n if n < 2 else fib(n-1) + fib(n-2)

cProfile.run("fib(30)", "out.prof")
stats = pstats.Stats("out.prof")
stats.sort_stats("cumulative").print_stats(3)
2692537 function calls in 0.34 seconds ncalls tottime cumtime filename:lineno(function) 2692537 0.34 0.34 fib <-- hot path: 2.7M calls

Language Comparison: Python uses sampling-free instrumentation (cProfile) plus tracemalloc/memory_profiler. Go bakes pprof into the runtime — CPU, heap, and goroutine profiles with one import. TypeScript leans on the V8 engine: --prof, perf_hooks, and Chrome DevTools heap snapshots. Rust pairs cargo flamegraph/perf for CPU with allocator-based tools like dhat for memory. C++ uses external samplers (perf, gprof) and Valgrind/callgrind plus sanitizers. The recurring theme: statistical sampling (perf, pprof) scales to production; instrumentation (cProfile, Valgrind) is precise but slow.
Common Mistakes:
❌ [Python] Profiling with time.time() around a whole script and guessing which line is slow.
✅ [Python] Use cProfile + pstats.sort_stats("cumulative") to see per-function cost.
❌ [Go] Profiling a debug build or for too short a duration — too few samples to be meaningful.
✅ [Go] Profile an optimized build under realistic load; collect enough samples (seconds, not ms).
❌ [TS] Trusting console.time for micro-work dominated by JIT warmup and GC noise.
✅ [TS] Use perf_hooks measures over many iterations, or DevTools CPU profiler for call trees.
❌ [Rust] Profiling a --debug build — the numbers don't reflect release performance at all.
✅ [Rust] Profile --release (with debug = true for symbols) so the optimizer is in play.
❌ [C++] Forgetting -g, so perf report shows raw addresses instead of function names.
✅ [C++] Build with -O2 -g to keep optimizations and symbols for readable profiles.
Key Rules:
  • Measure before optimizing — profile to find the real hot path, never optimize on intuition.
  • Profile release/optimized builds under realistic input sizes; debug builds lie.
  • CPU and memory are separate profiles — a function can be cheap in time but expensive in allocations.
  • Don't micro-optimize the 1% — fix the widest flame-graph bar first.

Caching & Memoization Patterns Performance & Optimization
Caching trades space for time. Memoization — caching a function's results by its arguments — is the single most common optimization for recursive or repeated computations.

Start with the simplest cache: a dictionary keyed by the function's input. Build to a bounded LRU cache (evict the least-recently-used entry when full), then to a concurrent-safe cache for multi-threaded use. The hard part is rarely the lookup — it's invalidation (when does a cached value go stale?) and bounding (caches that grow forever become memory leaks).

1. Memoize Fibonacci

Python 
from functools import lru_cache, cache

@cache  # unbounded; or @lru_cache(maxsize=128)
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

print(fib(50))            # 12586269025 -- instant
print(fib.cache_info())   # CacheInfo(hits=48, misses=51, ...)
12586269025 CacheInfo(hits=48, misses=51, maxsize=None, currsize=51)
Go 
var memo = map[int]int{}

2. A bounded LRU cache

Python 
from collections import OrderedDict

class LRU:
    def __init__(self, cap): self.cap = cap; self.d = OrderedDict()
    def get(self, k):
        if k not in self.d: return None
        self.d.move_to_end(k)        # mark as recently used
        return self.d[k]
    def put(self, k, v):
        if k in self.d: self.d.move_to_end(k)
        self.d[k] = v
        if len(self.d) > self.cap:
            self.d.popitem(last=False) # evict LRU
put a,b,c (cap=2) -> evicts 'a'; get('a') is None

class LRU<K, V> {
  private m = new Map<K, V>();
  constructor(private cap: number) {}
  get(k: K): V | undefined {
    if (!this.m.has(k)) return undefined;
    const v = this.m.get(k)!;
    this.m.delete(k); this.m.set(k, v);  // move to newest
    return v;
  }
  put(k: K, v: V) {
    if (this.m.has(k)) this.m.delete(k);
    this.m.set(k, v);
    if (this.m.size > this.cap)
      this.m.delete(this.m.keys().next().value); // evict oldest
  }
}

Go has no built-in memoize; you write a map (or sync.Map for concurrency) by hand. Rust needs interior mutability (RefCell single-thread, Mutex multi-thread) because of the borrow checker, or reaches for the cached/lru crates. C++ uses unordered_map plus a list for LRU and shared_mutex/call_once for thread safety. The trade-off everywhere: unbounded caches are simple but leak; bounded caches need an eviction policy.
Common Mistakes:
❌ [Python] @lru_cache on a method — it keeps self alive forever (memory leak) and caches across instances.
✅ [Python] Cache module-level functions, or use cachetools/functools.cached_property for instance-scoped caching.
❌ [Go] Reading and writing a plain map from multiple goroutines — concurrent map access panics ("concurrent map writes").
✅ [Go] Guard with sync.RWMutex, or use sync.Map for read-heavy stable-key caches.
❌ [TS] Keying a cache by object with a plain Map — the entry never gets GC'd, leaking memory.
✅ [TS] Use a WeakMap when keys are objects so entries are collected when the key dies.
❌ [Rust] Trying to mutate a cache through a shared &self reference — the borrow checker rejects it.
✅ [Rust] Wrap in RefCell (single-thread) or Mutex/RwLock (multi-thread) for interior mutability.
❌ [C++] Returning a reference/iterator into the cache, then inserting — rehash invalidates it (dangling).
✅ [C++] Return by value/copy, or use node_handle/stable containers; re-lookup after any mutation.
Key Rules:
  • Memoize pure functions only — caching a function with side effects or changing inputs returns stale results.
  • Bound your caches (LRU/TTL) in long-running services; unbounded caches are slow memory leaks.
  • For concurrency, pick the right primitive: read-heavy → RWMutex/shared_mutex/sync.Map; compute-once → entry-or-insert under a lock.

Concurrent Data Structures Performance & Optimization
Multi-threaded programs need collections that survive simultaneous access. Each language solves the "two threads touch the same map" problem with a different default — locks, channels, or lock-free atomics.

The two canonical problems: a shared counter incremented from many threads (data race if unguarded), and a producer-consumer queue (one side pushes, another pops, safely). The building blocks are mutexes (mutual exclusion), atomics (lock-free single-value updates), and channels/queues (hand-off without sharing mutable state).

Language Comparison: Go's philosophy is "don't communicate by sharing memory; share memory by communicating" — channels are the idiomatic concurrent structure. Rust enforces safety at compile time: Arc<Mutex<T>> for shared state, mpsc channels for hand-off, and the borrow checker rejects data races before they run. Python's GIL serializes bytecode but does not make compound operations atomic — you still need a Lock; queue.Queue is the safe default. TypeScript only gets real parallelism via worker threads with SharedArrayBuffer+Atomics or message passing. C++ gives raw primitives (std::atomic, std::mutex, std::shared_mutex, condition_variable) and full control — and full responsibility for correctness.
Common Mistakes:
❌ [Python] Assuming the GIL makes counter += 1 atomic — it's read-modify-write and races.
✅ [Python] Guard compound mutations with a Lock, or use a queue.Queue for hand-off.
❌ [Go] Reaching for sync.Map as a general-purpose map — it's slower for write-heavy or non-stable-key workloads.
✅ [Go] Use a plain map + sync.RWMutex by default; reserve sync.Map for read-mostly, stable keys.
❌ [TS] Sharing a plain object between workers expecting mutations to propagate — only SharedArrayBuffer is shared.
✅ [TS] Use SharedArrayBuffer+Atomics for shared state, or postMessage (structured clone) for hand-off.
❌ [Rust] Locking two mutexes in inconsistent order across threads — classic deadlock.
✅ [Rust] Always acquire locks in a fixed global order, or restructure to hold one lock at a time.
❌ [C++] Waiting on a condition_variable without a predicate — spurious wakeups corrupt your logic.
✅ [C++] Always use the predicate overload: cv.wait(lk, []{ return ready; });
Key Rules:
  • Prefer message passing (channels/queues) over shared mutable state — it sidesteps most data races.
  • For a single shared counter, atomics beat a mutex; for compound invariants, you need a lock.
  • Read-heavy maps want a reader-writer lock (RWMutex/shared_mutex); write-heavy ones often want sharding.

Section 2: Security Fundamentals

Hashing protects data at rest and in transit; validation stops injection at the boundary. Get these wrong and nothing else matters.

Hashing & Cryptographic Primitives Security Fundamentals
Hashing underpins password storage, data integrity, and cache keys. Knowing which primitive to use — and which never to use — prevents the most common security mistakes.

Three distinct tools, often confused: a fast cryptographic hash (SHA-256) for integrity and content addressing; a slow password hash (bcrypt/argon2) deliberately expensive to resist brute force; and a CSPRNG (cryptographically secure RNG) for tokens, salts, and keys. Never use a fast hash for passwords, and never use a non-crypto RNG (like random/rand()) for secrets.

Common Mistakes:
❌ [Python] Hashing passwords with hashlib.sha256 — far too fast, trivially brute-forced.
✅ [Python] Use bcrypt/argon2-cffi with a per-password salt for password storage.
❌ [Go] Using math/rand for tokens or salts — predictable and seedable.
✅ [Go] Use crypto/rand for anything security-sensitive; x/crypto/bcrypt for passwords.
❌ [TS] Comparing signatures with === — leaks length/byte info via timing.
✅ [TS] Use crypto.timingSafeEqual for any secret/MAC comparison.
❌ [Rust] Rolling your own crypto or using rand::thread_rng() for keys without checking it's CSPRNG-backed.
✅ [Rust] Use vetted crates (ring, argon2) and OsRng for key/salt material.
❌ [C++] Using std::rand() or std::mt19937 for tokens — neither is cryptographically secure.
✅ [C++] Use RAND_bytes (OpenSSL) or libsodium's randombytes_buf for secrets.
Key Rules:
  • Passwords → slow, salted hashes (bcrypt/argon2). Integrity/keys → SHA-256. Authentication → HMAC.
  • Secrets, salts, tokens → always a CSPRNG, never a general-purpose RNG.
  • Compare secrets/MACs in constant time to avoid timing side-channels.
  • Don't invent crypto — use audited libraries and standard constructions.

Input Validation & Sanitization Security Fundamentals
Injection attacks (SQL, XSS, command injection) remain the most common vulnerabilities. Validating and parameterizing at the boundary is the cheapest, most effective defense you can build.

Two complementary techniques: validation rejects malformed input early (type, range, format), and parameterization keeps untrusted data out of the interpreter (SQL, shell, HTML) by separating code from data. The golden rule for SQL injection: never build queries by string concatenation — always use bound parameters so user data can never become SQL keywords.

print(valid_username("a"))           # False (too short)

fn valid_username(s: &str) -> bool {
    let len = s.chars().count();
    (3..=20).contains(&len)
        && s.chars().all(|c| c.is_ascii_alphanumeric() || c == '_')
}
println!("{}", valid_username("ada_99")); // true

Language Comparison: The high-level languages have ecosystems built around declarative validation: Python's pydantic and TypeScript's zod validate and produce typed data in one step. Rust uses the type system itself as validation (parse into a strong type, then it's always valid) plus the validator crate, and sqlx even checks SQL at compile time. Go favors small standard-library pieces (net/mail, regexp, strconv) and prepared statements. C++ relies on std::regex and manual bounds checks. Across all five, the SQL-injection defense is identical in spirit: parameterize, never concatenate.
Common Mistakes:
❌ [Python] f-string SQL: f"... WHERE name = '{name}'" — textbook SQL injection.
✅ [Python] Pass parameters as a tuple to execute(sql, (name,)).
❌ [Go] Building queries with fmt.Sprintf from user input.
✅ [Go] Use placeholders (?/$1) and pass args to db.Query.
❌ [TS] Rendering user input into HTML via innerHTML — stored/reflected XSS.
✅ [TS] Use textContent/framework escaping, validate with zod, parameterize SQL.
❌ [Rust] Validating a String then passing it around as a plain String — the validation isn't enforced by the type.
✅ [Rust] Wrap validated data in a newtype (e.g. struct Username(String)) constructed only via a validating fn.
❌ [C++] Passing user input to system() or concatenating into shell commands — command injection.
✅ [C++] Avoid system(); use exec* with an explicit argv array and validate inputs against an allowlist.
Key Rules:
  • Validate at the trust boundary (user input, network, files) — trust nothing that crosses it.
  • Prefer allowlists (what's permitted) over denylists (what's blocked); denylists always miss a case.
  • Parameterize all SQL; escape/encode output for the destination context (HTML, shell, URL).
  • Make validity a type when you can — parse into a strong type so invalid states are unrepresentable.
Practice Exercises (implement in all 5 languages):
1
Write a validator that rejects common SQL-injection payloads (e.g. ' OR '1'='1, ; DROP TABLE) for a field that should only contain a number, and demonstrate that a parameterized query is safe even without the validator.
2
Build a function that validates and sanitizes a username: alphanumeric plus underscore, 3–20 chars, returning a normalized lowercase form or a rejection reason. Input "Ada_99" → "ada_99"; input "a" → rejected ("too short").
Connections: Validation is the boundary partner of Hashing & Crypto (validate, then hash) and of Serialization (deserialized data is untrusted input — validate it). Strong-typing-as-validation echoes earlier work on enums and algebraic data types.
