1. Hashing & Cryptographic Primitives Security Fundamentals rep 1 • 12 days since review
2nd review (overdue 11 days)
Quick Recall: A cryptographic hash (SHA-256) is a one-way, fixed-length fingerprint of bytes; an HMAC keys that hash for authenticated integrity; a password hash (bcrypt/argon2/scrypt) is deliberately slow and salted. Never compare secrets with == — use a constant-time compare to avoid timing leaks. Non-cryptographic hashes (for hash maps) are a different tool: fast, not collision-resistant against an adversary.

Key Pattern — SHA-256 hex digest of a byte string

Python 
import hashlib, hmac, secrets
h = hashlib.sha256(b"hello").hexdigest()          # 2cf24dba5fb0a30e...
tag = hmac.new(b"key", b"msg", hashlib.sha256).hexdigest()
secrets.compare_digest(tag, tag)                  # True (constant-time)
Go 
import ("crypto/sha256"; "crypto/hmac"; "crypto/subtle"; "encoding/hex")
sum := sha256.Sum256([]byte("hello"))             // [32]byte
_ = hex.EncodeToString(sum[:])                    // 2cf24dba...
m := hmac.New(sha256.New, []byte("key")); m.Write([]byte("msg"))
subtle.ConstantTimeCompare(m.Sum(nil), m.Sum(nil)) == 1
TypeScript 
import { createHash, createHmac, timingSafeEqual } from "node:crypto";
const h = createHash("sha256").update("hello").digest("hex");
const tag = createHmac("sha256", "key").update("msg").digest();
timingSafeEqual(tag, tag);                         // true (constant-time)
Rust 
use sha2::{Sha256, Digest};                         // sha2 crate
let digest = Sha256::digest(b"hello");             // GenericArray<u8,32>
let hex = digest.iter().map(|b| format!("{:02x}", b)).collect::<String>();
// HMAC via the `hmac` crate; constant-time compare via `subtle::ConstantTimeEq`
C++ 
#include <openssl/sha.h>   // or EVP_Digest / OpenSSL 3.x
unsigned char out[SHA256_DIGEST_LENGTH];
SHA256(reinterpret_cast<const unsigned char*>("hello"), 5, out);
// HMAC: HMAC(EVP_sha256(), key, klen, msg, mlen, ...); compare with CRYPTO_memcmp

1
Implement a signed-token scheme: sign(payload) returns payload.HMAC_SHA256(secret, payload) and verify(token) recomputes the tag and compares it in constant time. Then add password storage with a per-user salt and a slow KDF (bcrypt/argon2/scrypt), and explain why SHA-256 alone is wrong for passwords.
#1 Pitfall per Language:
  • Python: hashing a str instead of bytes (TypeError) — encode first; and using == on digests instead of hmac.compare_digest.
  • Go: comparing MACs with bytes.Equal (early-exit, timing leak) instead of subtle.ConstantTimeCompare; and forgetting Sum256 returns an array, not a slice.
  • TypeScript: timingSafeEqual throws if the two buffers differ in length — guard length first; and rolling your own crypto instead of node:crypto/WebCrypto.
  • Rust: reaching for the hash std trait (that's for HashMap, not crypto) — use sha2/ring; compare with subtle, not ==.
  • C++: the raw SHA256 one-shot is deprecated in OpenSSL 3 (prefer the EVP API); and memcmp on secrets leaks timing — use CRYPTO_memcmp.
2. Rust: Async Runtimes & tokio Language Deep Dive rep 1 • 8 days since review
2nd review (overdue 7 days)
Quick Recall: Rust's async fn returns a lazy Future that does nothing until polled by a runtime — the language ships the Future trait and async/await syntax but no executor, so you pick one (tokio, async-std). tokio::spawn schedules a task on a multi-threaded work-stealing scheduler; .await yields at suspension points. Unlike Go (runtime built in) or Python (asyncio in stdlib), Rust makes the runtime an explicit dependency.

Key Pattern — spawn concurrent tasks and await them

Rust 
#[tokio::main]                                   // sets up the runtime
async fn main() {
    let a = tokio::spawn(async { 1 + 1 });        // task on the scheduler
    let b = tokio::spawn(async { 2 + 2 });
    let (x, y) = tokio::join!(a, b);              // await both concurrently
    println!("{} {}", x.unwrap(), y.unwrap());    // 2 4
}
Python 
import asyncio
async def work(n): return n + n
async def main():
    a, b = await asyncio.gather(work(1), work(2)) # concurrent on one event loop
    print(a, b)                                    # 2 4
asyncio.run(main())
Go 
// No async keyword; goroutines + channels, runtime is built in:
ch := make(chan int, 2)
go func(){ ch <- 1 + 1 }()
go func(){ ch <- 2 + 2 }()
fmt.Println(<-ch, <-ch)                            // 2 4 (order not guaranteed)
TypeScript 
const work = async (n: number) => n + n;
const [a, b] = await Promise.all([work(1), work(2)]); // event loop, single thread
console.log(a, b);                                 // 2 4
C++ 
#include <future>
auto a = std::async(std::launch::async, []{ return 1 + 1; });
auto b = std::async(std::launch::async, []{ return 2 + 2; });
std::cout << a.get() << " " << b.get();             // 2 4 (real threads, not tasks)
Challenge (all 5 languages):
1
Fetch three URLs concurrently with a per-request timeout and collect the results, cancelling stragglers past a deadline. Use tokio (select!/timeout) in Rust, asyncio.wait_for/gather in Python, a context-cancelled errgroup in Go, Promise.all + AbortController in TS, and std::async/a thread pool with a timed wait in C++. Note which runtimes cancel cooperatively vs. not at all.
#1 Pitfall per Language:
  • Rust: creating a future and never .await-ing it — it never runs (futures are lazy); and blocking (std::thread::sleep, sync I/O) inside an async task starves the whole worker — use tokio::time::sleep/spawn_blocking.
  • Python: calling a coroutine without await yields a coroutine object and a “never awaited” warning; and CPU-bound work in the loop blocks everything (the GIL) — offload to a process/thread pool.
  • Go: a goroutine leaking because nothing reads its channel or its context is never cancelled; and a WaitGroup/unbuffered-channel deadlock.
  • TypeScript: await-ing in a loop serializes what could be concurrent — batch with Promise.all; an unhandled rejection crashes the process.
  • C++: std::async without std::launch::async may run lazily on .get() (not concurrent); and a discarded std::future from async blocks in its destructor.
3. Rust: Macros — Declarative & Procedural Language Deep Dive rep 1 • 8 days since review
2nd review (overdue 7 days)
Quick Recall: Rust has two macro systems: declarative macro_rules! (pattern-match on token trees, expand to code — hygienic, great for variadic/DSL sugar like vec!) and procedural macros (functions over a TokenStream: #[derive(...)], attribute, and function-like), which run at compile time. Macros operate on syntax, so they can do what generics can't (generate impls, capture variadic args). C++ templates, Go go:generate, TS decorators, and Python metaclasses/decorators are the rough analogs.

Key Pattern — a variadic macro that builds a HashMap

Rust 
macro_rules! map {
    ($($k:expr => $v:expr),* $(,)?) => {{
        let mut m = std::collections::HashMap::new();
        $( m.insert($k, $v); )*                    // repetition expands per pair
        m
    }};
}
let m = map!{"a" => 1, "b" => 2};                  // {"a":1,"b":2}
// #[derive(Debug, Clone)] is a *procedural* macro generating trait impls
C++ 
// Closest analogs: variadic templates + (blunt) preprocessor macros
template<class... Kv>
auto make_map(Kv... kv){ std::map<int,int> m; (m.insert(kv), ...); return m; } // fold
#define SQUARE(x) ((x)*(x))    // text substitution, NOT hygienic (parenthesize!)
Go 
// No macros. Code generation via a directive + a separate tool:
//go:generate stringer -type=Color
// `go generate ./...` runs it; the generated .go file is committed.
TypeScript 
// No macros; decorators are the metaprogramming hook (runtime, not syntactic):
function logged(target: any, key: string, d: PropertyDescriptor) {
  const orig = d.value; d.value = function (...a: any[]) { /* wrap */ return orig.apply(this, a); };
}
Python 
# Decorators + metaclasses do at runtime what Rust macros do at compile time:
def memoize(fn):
    cache = {}
    def wrap(*a): return cache.setdefault(a, fn(*a))
    return wrap
@memoize
def fib(n): return n if n < 2 else fib(n-1) + fib(n-2)
Challenge (all 5 languages):
1
Implement a hashset!{a, b, c}-style variadic literal with macro_rules! in Rust; then reproduce the “generate a Debug/to-string impl automatically” effect via a #[derive] proc-macro (or sketch it). In the other four, achieve the closest thing: C++ variadic templates + fold, Go go:generate + stringer, TS a class decorator, Python a class decorator/metaclass — and note which run at compile time vs. runtime.
#1 Pitfall per Language:
  • Rust: wrong fragment specifier (expr vs tt vs ty) so a pattern won't match; forgetting the trailing-comma matcher $(,)?; and reaching for a proc-macro (a whole separate crate) when macro_rules! suffices.
  • C++: unparenthesized preprocessor macros (#define SQ(x) x*x mis-expands SQ(1+1)); macros ignore scope/types — prefer constexpr/templates.
  • Go: expecting macros at all — there are none by design; go:generate is manual and must be re-run and committed.
  • TypeScript: decorators need experimentalDecorators (legacy) or the TS 5 standard form, and they execute at runtime — they can't add new static types.
  • Python: a decorator that forgets functools.wraps loses the wrapped function's name/doc; metaclass magic is easy to overuse and hard to debug.
4. C++: SFINAE & Template Metaprogramming Language Deep Dive rep 1 • 8 days since review
2nd review (overdue 7 days)
Quick Recall: SFINAE (“Substitution Failure Is Not An Error”) means that when substituting template arguments makes a signature ill-formed, that overload is silently removed from the candidate set instead of erroring — the basis of compile-time overload selection via std::enable_if, the detection idiom, and void_t. C++20 concepts and if constexpr largely replace it with far cleaner syntax and errors. Rust does the same job with trait bounds, TS with conditional types.

Key Pattern — enable an overload only for integral types

C++ 
#include <type_traits>
template<class T, std::enable_if_t<std::is_integral_v<T>, int> = 0>
T half(T x){ return x / 2; }                        // only compiles for ints
// C++20, far cleaner:  template<std::integral T> T half(T x){ return x/2; }
// detection idiom:  template<class,class=void> struct has_size: std::false_type{};
Rust 
// Trait bounds do SFINAE's job with readable errors:
use std::ops::Div;
fn half<T: Div<Output = T> + From<u8>>(x: T) -> T { x / T::from(2) }
// non-satisfying types are rejected at the call site with a clear message
TypeScript 
// Conditional types are TS's compile-time "enable_if":
type IntLike<T> = T extends number ? T : never;
function half<T extends number>(x: IntLike<T>): number { return Math.floor(x / 2); }
Go 
// No SFINAE; a type-set constraint bounds the candidate types instead:
type Integer interface{ ~int | ~int32 | ~int64 }
func Half[T Integer](x T) T { return x / 2 }        // rejected for non-integers
Python 
# Duck typing + runtime dispatch; singledispatch approximates overloading:
from functools import singledispatch
@singledispatch
def half(x): raise TypeError(type(x))
@half.register
def _(x: int): return x // 2
Challenge (all 5 languages):
1
Write a generic print_all(c) that has one behavior when the argument is a container with begin()/end() and another for a scalar. In C++, select via SFINAE/void_t detection and again with a C++20 concept + if constexpr; then express the two branches with Rust trait bounds, a TS conditional type + overloads, a Go constraint, and Python singledispatch. Compare the error when the container branch's requirement isn't met.
#1 Pitfall per Language:
  • C++: only errors in the immediate context of substitution trigger SFINAE — a failure deeper in the body is a hard error; and legacy enable_if produces unreadable diagnostics (prefer concepts/if constexpr).
  • Rust: forgetting a needed bound surfaces as an error inside the body rather than at the signature — put the constraint on the signature; overlapping impls are rejected (no specialization on stable).
  • TypeScript: conditional types distribute over unions unless wrapped [T] extends [U]; and everything is erased — no runtime dispatch.
  • Go: constraints are type sets, not arbitrary predicates — you can't “enable if T has method M” beyond an interface method set.
  • Python: singledispatch keys on the first arg's runtime type only; no compile-time guarantee, and it won't dispatch on generics like list[int].
5. C++: Coroutines (C++20) Language Deep Dive rep 1 • 8 days since review
2nd review (overdue 7 days)
Quick Recall: A C++20 coroutine is any function using co_await, co_yield, or co_return; the compiler rewrites it into a state machine with a heap-allocated frame, driven by a promise type you (or a library) supply. The language ships only the low-level machinery — you need a generator/task type (C++23 std::generator, or a hand-rolled one). It's the same lazy-suspension idea as Python generators, JS generators/async, and Rust async/iterators.

Key Pattern — a lazy generator yielding a sequence

C++ 
#include <generator>   // C++23; pre-23 needs a hand-written promise type
std::generator<int> count_up(int n){
    for (int i = 0; i < n; ++i) co_yield i;         // suspends, resumes on next()
}
for (int x : count_up(3)) std::cout << x << " ";     // 0 1 2
Python 
def count_up(n):
    for i in range(n): yield i                      # generator: lazy, stateful
print(list(count_up(3)))                            # [0, 1, 2]
TypeScript 
function* countUp(n: number): Generator<number> {
  for (let i = 0; i < n; i++) yield i;              // function* + yield
}
console.log([...countUp(3)]);                       // [0, 1, 2]
Rust 
// Stable Rust has no `yield`; model lazy sequences with an iterator:
fn count_up(n: i32) -> impl Iterator<Item = i32> { (0..n) }
let v: Vec<_> = count_up(3).collect();               // [0, 1, 2]
// (generators/`gen` blocks are still unstable)
Go 
// Go 1.23 range-over-func iterators, or a channel + goroutine (classic):
func countUp(n int) func(func(int) bool) {
    return func(yield func(int) bool) {
        for i := 0; i < n; i++ { if !yield(i) { return } }
    }
}
for x := range countUp(3) { fmt.Print(x, " ") }     // 0 1 2
Challenge (all 5 languages):
1
Build a lazy, potentially-infinite Fibonacci generator and take the first 10 values without materializing the rest. Use a C++20/23 coroutine generator, a Python yield generator, a TS function*, a Rust custom Iterator (impl Iterator), and a Go 1.23 range-over-func (or channel) iterator. Then add early termination and confirm the producer stops when the consumer breaks.
#1 Pitfall per Language:
  • C++: a coroutine frame is heap-allocated and its lifetime is subtle — a dangling co_awaited reference or using a coroutine after the promise is destroyed is UB; pre-C++23 you must supply the promise boilerplate correctly.
  • Python: a generator is single-pass — iterating it twice yields nothing the second time; and an exception inside is only raised when you advance it.
  • TypeScript: forgetting * makes an ordinary function; spreading an infinite generator ([...gen]) hangs.
  • Rust: expecting yield on stable — not available; hand-roll an Iterator or use the genawaiter crate.
  • Go: the channel-based generator leaks a goroutine if the consumer stops early without a cancel signal — prefer range-over-func or pass a context/done channel.
6. Network Programming: TCP/UDP Sockets Systems & Architecture rep 1 • 8 days since review
2nd review (overdue 7 days)
Quick Recall: TCP is a reliable, ordered, connection-oriented byte stream (listen/accept/connect, read/write) — you must frame messages yourself because there are no message boundaries. UDP is connectionless datagrams (sendto/recvfrom): each packet is a discrete, possibly-lost, possibly-reordered message. Every language wraps the same BSD socket API; the difference is ergonomics and how errors/short reads surface.

Key Pattern — a minimal TCP echo server

Python 
import socket
srv = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(("127.0.0.1", 9000)); srv.listen()
conn, _ = srv.accept()
data = conn.recv(1024); conn.sendall(data)          # echo (may be a partial read!)
Go 
ln, _ := net.Listen("tcp", "127.0.0.1:9000")
conn, _ := ln.Accept()
go func() {                                          // one goroutine per connection
    io.Copy(conn, conn)                             // echo the whole stream
}()
TypeScript 
import { createServer } from "node:net";
const server = createServer(sock => {
  sock.on("data", buf => sock.write(buf));           // echo each chunk
});
server.listen(9000, "127.0.0.1");
Rust 
use std::net::TcpListener; use std::io::{Read, Write};
let listener = TcpListener::bind("127.0.0.1:9000").unwrap();
for stream in listener.incoming() {
    let mut s = stream.unwrap(); let mut buf = [0u8; 1024];
    let n = s.read(&mut buf).unwrap(); s.write_all(&buf[..n]).unwrap(); // echo
}
C++ 
#include <sys/socket.h><netinet/in.h>
int fd = socket(AF_INET, SOCK_STREAM, 0);
int yes = 1; setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, &yes, sizeof yes);
sockaddr_in a{AF_INET, htons(9000), {htonl(INADDR_LOOPBACK)}};
bind(fd, (sockaddr*)&a, sizeof a); listen(fd, 16);
int c = accept(fd, nullptr, nullptr);
char b[1024]; ssize_t n = recv(c, b, sizeof b, 0); send(c, b, n, 0); // echo
Challenge (all 5 languages):
1
Build a length-prefixed TCP protocol: the client sends a 4-byte big-endian length then that many payload bytes; the server loops reading until it has a full frame (handling short reads) and echoes it back. Then add a UDP variant with sendto/recvfrom and demonstrate that a datagram larger than the MTU or a dropped packet behaves differently from the TCP stream.
#1 Pitfall per Language:
  • Python: assuming one recv returns a whole message — TCP is a stream; loop until you have all the framed bytes, and sendall (not send) to avoid short writes.
  • Go: forgetting conn.Close() (fd leak) and not setting read/write deadlines — a stuck peer blocks the goroutine forever.
  • TypeScript: the 'data' event delivers arbitrary chunks, not messages — buffer and reframe; and forgetting backpressure (write returning false).
  • Rust: read can return fewer bytes than requested (and 0 means EOF) — loop or use read_exact; blocking sockets need threads or async for concurrency.
  • C++: ignoring recv/send return values (-1 error, 0 peer closed, or a short count), forgetting htons/htonl byte order, and not handling EINTR.

Graph Algorithms

Yesterday you built graphs (adjacency list/matrix), traversed them (BFS/DFS), and merged sets (union-find). Today those primitives combine into the three algorithms you reach for constantly: shortest paths on weighted graphs (Dijkstra), ordering under dependencies (topological sort), and cheapest connecting tree (minimum spanning tree). The cross-language interest is entirely in the supporting data structures — a min-priority-queue and a disjoint-set — and how awkward or clean each language makes them.

Dijkstra's Shortest Path Graph Algorithms
Why it matters: Dijkstra finds the shortest path from one source to every vertex in a graph with non-negative edge weights — the core of routing, network latency, game pathfinding, and anything modeled as “cheapest way to get from A to B.” With a binary heap it runs in O((V+E) log V), and getting the priority-queue idiom right in each language is the whole game.

Keep a tentative distance to every vertex (∞ initially, 0 for the source) and a min-priority-queue of (distance, vertex). Repeatedly pop the closest unsettled vertex, and for each neighbor relax the edge: if dist[u] + w < dist[v], update dist[v] and push it. A vertex is “settled” the first time it's popped (that distance is final). Because most standard heaps don't support decrease-key, the idiom is lazy deletion: push duplicates and skip a popped entry whose stored distance is stale.

Example — shortest distances from vertex 0 on a small weighted graph

Python 
import heapq
g = {0:[(1,4),(2,1)], 1:[(3,1)], 2:[(1,2),(3,5)], 3:[]}   # u -> [(v, weight)]
def dijkstra(src, n):
    dist = [float('inf')] * n; dist[src] = 0
    pq = [(0, src)]                                    # (distance, vertex) min-heap
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]: continue                        # stale entry -> skip
        for v, w in g[u]:
            if d + w < dist[v]:
                dist[v] = d + w; heapq.heappush(pq, (dist[v], v))
    return dist
print(dijkstra(0, 4))                                  # [0, 3, 1, 4]
[0, 3, 1, 4]
Go 
import ("container/heap"; "math")
type Item struct{ dist, node int }
type PQ []Item
func (p PQ) Len() int { return len(p) }
func (p PQ) Less(i, j int) bool { return p[i].dist < p[j].dist }  // min-heap
func (p PQ) Swap(i, j int) { p[i], p[j] = p[j], p[i] }
func (p *PQ) Push(x any) { *p = append(*p, x.(Item)) }
func (p *PQ) Pop() any { old := *p; n := len(old); it := old[n-1]; *p = old[:n-1]; return it }

func dijkstra(g map[int][][2]int, src, n int) []int {
    dist := make([]int, n)
    for i := range dist { dist[i] = math.MaxInt }
    dist[src] = 0
    pq := &PQ{{0, src}}
    for pq.Len() > 0 {
        it := heap.Pop(pq).(Item)
        if it.dist > dist[it.node] { continue }          // stale
        for _, e := range g[it.node] {                   // e = [v, w]
            if nd := it.dist + e[1]; nd < dist[e[0]] {
                dist[e[0]] = nd; heap.Push(pq, Item{nd, e[0]})
            }
        }
    }
    return dist                                          // [0 3 1 4]
}
[0 3 1 4]
TypeScript 
// No stdlib heap; a tiny binary min-heap of [dist, node]:
class MinHeap {
  private h: [number, number][] = [];
  push(x: [number, number]) { const h = this.h; h.push(x); let i = h.length - 1;
    while (i > 0) { const p = (i - 1) >> 1; if (h[p][0] <= h[i][0]) break;
      [h[p], h[i]] = [h[i], h[p]]; i = p; } }
  pop(): [number, number] | undefined { const h = this.h; if (!h.length) return;
    const top = h[0], last = h.pop()!; if (h.length) { h[0] = last; let i = 0;
      for (;;) { let s = i, l = 2*i+1, r = 2*i+2;
        if (l < h.length && h[l][0] < h[s][0]) s = l;
        if (r < h.length && h[r][0] < h[s][0]) s = r;
        if (s === i) break; [h[s], h[i]] = [h[i], h[s]]; i = s; } }
    return top; }
  get size() { return this.h.length; }
}
function dijkstra(g: [number, number][][], src: number): number[] {
  const dist = g.map(() => Infinity); dist[src] = 0;
  const pq = new MinHeap(); pq.push([0, src]);
  while (pq.size) {
    const [d, u] = pq.pop()!;
    if (d > dist[u]) continue;                           // stale
    for (const [v, w] of g[u]) if (d + w < dist[v]) { dist[v] = d + w; pq.push([dist[v], v]); }
  }
  return dist;
}
console.log(dijkstra([[[1,4],[2,1]], [[3,1]], [[1,2],[3,5]], []], 0)); // [0,3,1,4]
[ 0, 3, 1, 4 ]
Rust 
use std::collections::BinaryHeap;
use std::cmp::Reverse;                                   // BinaryHeap is a MAX-heap
fn dijkstra(g: &Vec<Vec<(usize, u32)>>, src: usize) -> Vec<u32> {
    let mut dist = vec![u32::MAX; g.len()];
    dist[src] = 0;
    let mut pq = BinaryHeap::new();
    pq.push(Reverse((0u32, src)));                        // Reverse -> min-heap
    while let Some(Reverse((d, u))) = pq.pop() {
        if d > dist[u] { continue; }                      // stale
        for &(v, w) in &g[u] {
            if d + w < dist[v] { dist[v] = d + w; pq.push(Reverse((dist[v], v))); }
        }
    }
    dist
}
// dijkstra(&vec![vec![(1,4),(2,1)], vec![(3,1)], vec![(1,2),(3,5)], vec![]], 0)
// -> [0, 3, 1, 4]
[0, 3, 1, 4]
C++ 
#include <queue><vector><climits>
using P = std::pair<int,int>;                            // (dist, node)
std::vector<int> dijkstra(std::vector<std::vector<P>>& g, int src){
    std::vector<int> dist(g.size(), INT_MAX); dist[src] = 0;
    std::priority_queue<P, std::vector<P>, std::greater<P>> pq;  // min-heap
    pq.push({0, src});
    while(!pq.empty()){
        auto [d, u] = pq.top(); pq.pop();
        if(d > dist[u]) continue;                         // stale
        for(auto [v, w] : g[u])
            if(d + w < dist[v]){ dist[v] = d + w; pq.push({dist[v], v}); }
    }
    return dist;                                          // 0 3 1 4
}

Language Comparison: The algorithm is identical; the priority queue is where languages diverge. Python heapq is a min-heap over tuples (compares first element) — the easiest. C++ std::priority_queue is a max-heap, so you pass std::greater to invert it. Rust BinaryHeap is also a max-heap; wrap entries in std::cmp::Reverse. Go forces you to implement the five-method heap.Interface boilerplate for every element type. TypeScript has no stdlib heap at all — you hand-roll a binary heap (or misuse a sorted array at O(n) per op). All of them use the same lazy-deletion trick (push duplicates, skip stale pops) because none offers an efficient decrease-key out of the box.
Common Mistakes:
❌ [All] Running Dijkstra on a graph with negative edge weights — it produces wrong answers; the settled-once invariant breaks.
✅ [All] Use Bellman-Ford (or SPFA) for negative edges; Dijkstra requires all weights ≥ 0.
❌ [Python] Skipping the if d > dist[u]: continue stale check — you reprocess vertices and can blow up to O(V²) or worse.
✅ [Python] Always skip a popped entry whose distance exceeds the recorded best.
❌ [C++/Rust] Forgetting the heap is a max-heap — without greater/Reverse you pop the farthest vertex and get garbage.
✅ [C++] priority_queue<P, vector<P>, greater<P>>; [Rust] BinaryHeap<Reverse<(w,v)>>.
❌ [Go] Adding INT_MAX to an edge weight during relaxation — integer overflow wraps to a negative “shorter” distance.
✅ [Go] Only relax from a vertex whose dist is finite (the stale-skip guarantees this), or guard against overflow.
Key Rules:
  • Dijkstra requires non-negative weights; a vertex's distance is final the first time it's popped from the min-heap.
  • Use lazy deletion (push duplicates, skip stale pops) since standard heaps lack decrease-key; complexity is O((V+E) log V).
  • To recover the actual path, store a prev[] parent for each relaxed vertex and walk it back from the target.

Topological Sort Graph Algorithms
Why it matters: a topological order lists the vertices of a directed acyclic graph (DAG) so every edge points “forward” — exactly what you need for build systems, task/dependency scheduling, course prerequisites, spreadsheet recalculation, and package installation order. It also doubles as a cycle detector: a DAG has a valid ordering, a graph with a cycle does not.

Two standard algorithms, both O(V+E). Kahn's algorithm (BFS-style): compute each vertex's in-degree, start from the zero-in-degree vertices, and repeatedly emit one and decrement its neighbors' in-degrees, pushing any that hit zero — if you emit fewer than V vertices, a cycle exists. DFS post-order: run DFS and prepend each vertex as it finishes; reverse the finish order. Kahn's makes cycle detection and “any valid order” natural; DFS is a few lines but needs cycle-safe marking.

Example — Kahn's algorithm on 0→1, 0→2, 1→3, 2→3

Python 
from collections import deque
def topo(g, n):
    indeg = [0]*n
    for u in range(n):
        for v in g[u]: indeg[v] += 1
    q = deque(u for u in range(n) if indeg[u] == 0)
    order = []
    while q:
        u = q.popleft(); order.append(u)
        for v in g[u]:
            indeg[v] -= 1
            if indeg[v] == 0: q.append(v)
    return order if len(order) == n else None          # None -> cycle
print(topo({0:[1,2], 1:[3], 2:[3], 3:[]}, 4))          # [0, 1, 2, 3]

func topo(g map[int][]int, n int) []int {
    indeg := make([]int, n)
    for u := 0; u < n; u++ { for _, v := range g[u] { indeg[v]++ } }
    q := []int{}
    for u := 0; u < n; u++ { if indeg[u] == 0 { q = append(q, u) } }
    order := []int{}
    for len(q) > 0 {
        u := q[0]; q = q[1:]; order = append(order, u)
        for _, v := range g[u] { indeg[v]--; if indeg[v] == 0 { q = append(q, v) } }
    }
    if len(order) != n { return nil }                   // cycle
    return order                                        // [0 1 2 3]
}

Common Mistakes:
❌ [All] Assuming the topological order is unique — a DAG usually has many valid orders; tests that compare against one fixed order are brittle.
✅ [All] Verify the output is a valid order (every edge u→v has u before v), or fix the tie-break (e.g. a min-heap for lexicographically smallest).
❌ [All] Forgetting the cycle check (len(order) != n) — on a cyclic graph you silently return a partial, wrong ordering.
✅ [All] Compare emitted count to V and report a cycle when short.
❌ [TS] Using q.shift() for the queue — O(n) per dequeue makes the whole sort O(V²).
✅ [TS] Advance a head index (shown) or use a ring buffer.
❌ [DFS variant] Marking a vertex visited but not distinguishing “in the current recursion stack” — you miss back-edges and fail to detect cycles.
✅ [DFS variant] Use three colors (white/gray/black); a gray-to-gray edge is a cycle.

  • Topological sort exists iff the graph is a DAG; use it to detect cycles (emitted count < V means a cycle).
  • Kahn's (BFS on in-degrees) is easiest for cycle detection and lexicographic tie-breaks; DFS post-order (reversed) is more compact but needs cycle-safe coloring.
  • The order is generally not unique — validate the constraint, don't compare to a single expected list.

Minimum Spanning Tree (Kruskal & Prim) Graph Algorithms
Why it matters: a minimum spanning tree connects all vertices of a weighted undirected graph with the least total edge weight — the classic model for laying cable/pipe/road cheaply, clustering, and network design. Two greedy algorithms solve it: Kruskal (sort edges + union-find) and Prim (grow a tree with a priority queue). Both directly reuse the union-find and heap structures you already know.

Kruskal sorts all edges ascending and adds each edge whose endpoints are in different components (tested with union-find), skipping any that would form a cycle, until V−1 edges are chosen. Prim starts from any vertex and repeatedly adds the cheapest edge leaving the current tree (via a min-heap), like Dijkstra but keyed on edge weight rather than path distance. Kruskal shines on sparse graphs and pairs naturally with disjoint-set; Prim shines on dense graphs.

Example — Kruskal's MST total weight on a 4-vertex graph

Python 
def kruskal(n, edges):                                 # edges: (w, u, v)
    parent = list(range(n))
    def find(x):
        while parent[x] != x: parent[x] = parent[parent[x]]; x = parent[x]
        return x
    total, used = 0, 0
    for w, u, v in sorted(edges):                      # ascending by weight
        ru, rv = find(u), find(v)
        if ru != rv:                                   # different components
            parent[ru] = rv; total += w; used += 1
            if used == n - 1: break
    return total
print(kruskal(4, [(1,0,1),(3,0,2),(4,1,2),(2,1,3),(5,2,3)]))  # 6

Language Comparison: Kruskal = sort + union-find, so the split is entirely in how each language sorts and where the disjoint-set lives. Sorting tuples/edges by weight is a one-liner everywhere, but note the traps: TypeScript's Array.sort is lexicographic by default — you must pass (a,b)=>a[0]-b[0] or 10 sorts before 2. Rust and C++ sort tuples lexicographically by their first element automatically (weight first — convenient). Go needs sort.Slice with an explicit less-func. The union-find is the same parent-array path-compression core in all five (from Day 33). For Prim, swap the disjoint-set for a min-heap keyed on edge weight — the same BinaryHeap<Reverse> / greater<> / heapq machinery as Dijkstra above.
Common Mistakes:
❌ [TS] edges.sort() without a comparator sorts by string — MST weights come out wrong.
✅ [TS] Always pass a numeric comparator: edges.sort((a, b) => a[0] - b[0]).
❌ [All] Skipping the find(u) != find(v) check and adding every edge — you create cycles and overcount weight.
✅ [All] Only union endpoints in different components; stop once V−1 edges are chosen.
❌ [All] Running MST on a disconnected graph and treating the partial result as a spanning tree.
✅ [All] If fewer than V−1 edges were used, the graph is disconnected — you have a minimum spanning forest, not a tree.
❌ [Prim] Marking a vertex in-tree when you push its edge rather than when you pop and add it — you can add the wrong (non-minimal) crossing edge.
✅ [Prim] Add a vertex to the tree only when its cheapest crossing edge is popped and it's still outside the tree.
Key Rules:
  • Kruskal = sort edges + union-find, O(E log E); Prim = grow from one vertex with a min-heap, O((V+E) log V). Kruskal favors sparse graphs, Prim dense.
  • An MST has exactly V−1 edges; if you can't reach V−1, the graph is disconnected (you built a forest).
  • Both are greedy and provably optimal via the cut property: the lightest edge crossing any cut is safe to include.

Range Queries

A recurring need: answer “sum/min/max over index range [l, r]” many times on an array that also changes. A plain array gives O(1) reads but O(n) range aggregates; a precomputed prefix-sum gives O(1) range sums but O(n) updates. Fenwick and segment trees break that trade-off, giving O(log n) for both update and query. These structures are pure index arithmetic, so the five languages differ almost only in 0- vs 1-indexing discipline and integer widths.

Prefix Sums & Fenwick Tree (BIT) Range Queries
Why it matters: prefix sums turn any range-sum query into one subtraction — the single most reused trick in array problems (subarray sums, 2D image integrals, difference arrays). When the underlying array also gets point updates, a Fenwick tree (Binary Indexed Tree) keeps both point-update and prefix-sum at O(log n) in just a few lines and one array — the most compact competitive-programming structure there is.

A prefix-sum array P where P[i] = a[0]+..+a[i-1] answers sum(l, r) = P[r+1] - P[l] in O(1), but a single update forces recomputing the tail — O(n). A Fenwick tree stores partial sums indexed by the binary representation: index i covers the block of i & (-i) elements ending at i. update walks upward adding i += i & -i; query (prefix sum) walks downward subtracting i -= i & -i. It is conventionally 1-indexed — the low-bit trick relies on it.

Example 2 — a Fenwick tree: O(log n) point update and prefix sum

Same array, but now add(i, delta) and prefix(i) both run in O(log n). Note the 1-indexing.

Python 
class Fenwick:
    def __init__(self, n): self.t = [0]*(n+1)          # 1-indexed
    def add(self, i, delta):                           # i is 1-based
        while i < len(self.t): self.t[i] += delta; i += i & -i
    def prefix(self, i):                               # sum of [1..i]
        s = 0
        while i > 0: s += self.t[i]; i -= i & -i
        return s
    def range_sum(self, l, r): return self.prefix(r) - self.prefix(l-1)
f = Fenwick(6)
for i, x in enumerate([3,1,4,1,5,9], start=1): f.add(i, x)
print(f.range_sum(2, 4)); f.add(3, 10); print(f.range_sum(2, 4))  # 6  16

Language Comparison: The Fenwick tree is almost pure integer bit-arithmetic, so all five bodies are line-for-line the same — the low-bit isolation i & -i is the heart of it. The differences are peripheral: Python is 1-liner-friendly (itertools.accumulate for static prefix sums). Rust can't negate an unsigned usize with -i, so use i & i.wrapping_neg() (two's-complement low bit). C++/Go/TS use signed ints where i & -i works directly. Everyone must respect the 1-indexed convention — index 0 would make i & -i == 0 and loop forever. Watch integer width: accumulated sums overflow 32-bit fast, so Rust/C++ use i64/long long, and JS numbers lose precision past 2&sup5;³.
Common Mistakes:
❌ [All] Using a 0-based index in the Fenwick loops — 0 & -0 == 0 so add never advances (infinite loop) and prefix(0) reads nothing.
✅ [All] Keep the tree strictly 1-indexed; map external 0-based positions to i+1.
❌ [Rust] Writing i & -i on a usize — unary negation isn't defined for unsigned types (won't compile).
✅ [Rust] Use i & i.wrapping_neg() (or make the index a signed type).
❌ [All] Confusing set value with add delta — add(i, x) increments; to set an element you must add new - old.
✅ [All] Track current values separately, or build a set(i, v) helper that applies the delta.
❌ [C++/Rust] Summing many 32-bit values into an int Fenwick — silent overflow.
✅ [C++/Rust] Store partial sums as long long/i64.
Key Rules:
  • Static array, many range-sum queries, no updates → a prefix-sum array (O(1) query). Point updates + prefix queries → a Fenwick tree (O(log n) both).
  • Fenwick trees are 1-indexed; i & -i isolates the lowest set bit — add it to walk up (update), subtract it to walk down (prefix query).
  • Fenwick is limited to invertible operations (sum, xor); for min/max or non-invertible aggregates, use a segment tree.

Prefix sums extend Arrays/Slices/Vectors; the low-bit trick applies Bitwise Operations; Fenwick is the invertible cousin of the Segment Tree below and appears in Kruskal-adjacent counting problems and 2D image integrals.
Segment Trees Range Queries
Why it matters: a segment tree answers range queries (sum, min, max, gcd — any associative operation) with point updates in O(log n), and unlike a Fenwick tree it handles non-invertible aggregates like min/max and supports lazy range updates. It's the general-purpose range-query workhorse behind interval scheduling, range-min/max queries, and many competitive and systems problems.

Store the array in the leaves of a complete binary tree; each internal node holds the aggregate of its two children over a contiguous segment. A common, cache-friendly implementation is the iterative array form: allocate 2n slots, put the array at indices [n, 2n), and fill each parent i as op(tree[2i], tree[2i+1]). A point update fixes a leaf and walks up re-combining; a range query walks the two boundaries up, folding in whichever side-nodes fall inside [l, r). Because the aggregate need not be invertible, min/max work directly.

Example — iterative segment tree for range-sum with point update

Python 
class SegTree:
    def __init__(self, a):
        self.n = len(a); self.t = [0]*(2*self.n)
        self.t[self.n:] = a                              # leaves
        for i in range(self.n-1, 0, -1): self.t[i] = self.t[2*i] + self.t[2*i+1]
    def update(self, i, val):                            # point set
        i += self.n; self.t[i] = val; i //= 2
        while i: self.t[i] = self.t[2*i] + self.t[2*i+1]; i //= 2
    def query(self, l, r):                               # sum over [l, r)
        res = 0; l += self.n; r += self.n
        while l < r:
            if l & 1: res += self.t[l]; l += 1
            if r & 1: r -= 1; res += self.t[r]
            l //= 2; r //= 2
        return res
s = SegTree([3,1,4,1,5,9]); print(s.query(1, 4)); s.update(2, 10); print(s.query(1, 4))  # 6  12

Go 
type SegTree struct{ n int; t []int }
func NewSeg(a []int) *SegTree {
    n := len(a); t := make([]int, 2*n)
    copy(t[n:], a)
    for i := n - 1; i > 0; i-- { t[i] = t[2*i] + t[2*i+1] }
    return &SegTree{n, t}
}
func (s *SegTree) Update(i, val int) {
    i += s.n; s.t[i] = val
    for i /= 2; i > 0; i /= 2 { s.t[i] = s.t[2*i] + s.t[2*i+1] }
}
func (s *SegTree) Query(l, r int) int {                  // [l, r)
    res := 0
    for l, r = l+s.n, r+s.n; l < r; l, r = l/2, r/2 {
        if l&1 == 1 { res += s.t[l]; l++ }
        if r&1 == 1 { r--; res += s.t[r] }
    }
    return res
}                                                        // Query(1,4)=6; after Update(2,10)=12

TypeScript 
class SegTree {
  private n: number; private t: number[];
  constructor(a: number[]) {
    this.n = a.length; this.t = new Array(2 * this.n).fill(0);
    for (let i = 0; i < this.n; i++) this.t[this.n + i] = a[i];
    for (let i = this.n - 1; i > 0; i--) this.t[i] = this.t[2*i] + this.t[2*i+1];
  }
  update(i: number, val: number) {
    i += this.n; this.t[i] = val;
    for (i >>= 1; i > 0; i >>= 1) this.t[i] = this.t[2*i] + this.t[2*i+1];
  }
  query(l: number, r: number): number {                 // [l, r)
    let res = 0;
    for (l += this.n, r += this.n; l < r; l >>= 1, r >>= 1) {
      if (l & 1) res += this.t[l++];
      if (r & 1) res += this.t[--r];
    }
    return res;
  }
}

struct SegTree { n: usize, t: Vec<i64> }
impl SegTree {
    fn new(a: &[i64]) -> Self {
        let n = a.len(); let mut t = vec![0i64; 2 * n];
        t[n..].copy_from_slice(a);
        for i in (1..n).rev() { t[i] = t[2*i] + t[2*i+1]; }
        SegTree { n, t }
    }
    fn update(&mut self, mut i: usize, val: i64) {
        i += self.n; self.t[i] = val; i /= 2;
        while i > 0 { self.t[i] = self.t[2*i] + self.t[2*i+1]; i /= 2; }
    }
    fn query(&self, mut l: usize, mut r: usize) -> i64 {   // [l, r)
        let mut res = 0; l += self.n; r += self.n;
        while l < r {
            if l & 1 == 1 { res += self.t[l]; l += 1; }
            if r & 1 == 1 { r -= 1; res += self.t[r]; }
            l /= 2; r /= 2;
        }
        res
    }
}

#include <vector>
struct SegTree {
    int n; std::vector<long long> t;
    SegTree(const std::vector<long long>& a): n(a.size()), t(2 * a.size()) {
        for (int i = 0; i < n; i++) t[n + i] = a[i];
        for (int i = n - 1; i > 0; i--) t[i] = t[2*i] + t[2*i+1];
    }
    void update(int i, long long val){
        for (t[i += n] = val; i >= 1; i >>= 1) t[i>>1] = t[i] + t[i^1];
    }
    long long query(int l, int r){                        // [l, r)
        long long res = 0;
        for (l += n, r += n; l < r; l >>= 1, r >>= 1) {
            if (l & 1) res += t[l++];
            if (r & 1) res += t[--r];
        }
        return res;
    }
};

Language Comparison: The iterative 2n-array segment tree is again nearly identical everywhere — index math over a flat array, no pointers or recursion. The small idiomatic touches: Python slices leaves in with t[n:] = a; Rust uses copy_from_slice and needs the aggregate stored as i64; C++ can fuse the parent update with the sibling trick t[i] + t[i^1]; Go/TS read straight from the array. To switch from sum to min/max, replace + with min/max and the identity 0 with +∞/−∞ — this is exactly where a segment tree beats a Fenwick tree, which can't do min/max because they aren't invertible. Range updates need a recursive tree with lazy propagation (a second array of pending updates).
Common Mistakes:
❌ [All] Mixing half-open and closed intervals — the iterative form queries [l, r); passing an inclusive r drops the last element.
✅ [All] Fix a convention ([l, r) here) and convert inclusive queries with r+1.
❌ [All] Using identity 0 for a min/max tree — an unfilled node of 0 corrupts a min query.
✅ [All] Use the operation's true identity: +∞ for min, −∞ for max, 0 for sum, 1 for product.
❌ [All] Attempting a range update by looping point updates — O(n log n) per range op, defeating the structure.
✅ [All] Use lazy propagation (recursive tree with a pending-update array) for O(log n) range updates.
❌ [C++/Rust] Overflowing an int aggregate on large sums, or an off-by-one when the array size isn't a power of two.
✅ [C++/Rust] Store aggregates as long long/i64; the 2n iterative layout handles non-power-of-two sizes correctly.
Key Rules:
  • Segment tree = O(log n) range query + point update for any associative op (sum, min, max, gcd); use it over Fenwick when the aggregate isn't invertible.
  • The iterative 2n-array form is compact and cache-friendly; use a recursive tree with lazy propagation when you need range updates.
  • Always pair the operation with its correct identity element, and be consistent about half-open [l, r) vs. inclusive ranges.
