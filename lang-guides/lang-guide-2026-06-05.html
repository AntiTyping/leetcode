import threading

counter = 0
lock = threading.Lock()

def increment(n):
    global counter
    for _ in range(n):

threads = [threading.Thread(target=increment, args=(100_000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()

Smart Pointers & Memory Management Functions & Abstractions 2 days since · rep 0
Manage heap memory safely without manual free. Python: reference counting + GC (automatic), Go: GC-managed (automatic), TS: GC-managed (automatic), Rust: Box (unique ownership), Rc/Arc (shared), C++: unique_ptr (sole ownership), shared_ptr (reference-counted), weak_ptr (non-owning).
Python
import sys

class Node:
    def __init__(self, val):
        self.val = val

a = Node(42)
print(f"Refcount of a: {sys.getrefcount(a) - 1}")  # -1 for the arg ref
b = a  # shared reference
print(f"Refcount after alias: {sys.getrefcount(a) - 1}")
del b
print(f"Refcount after del b: {sys.getrefcount(a) - 1}")
Refcount of a: 1 Refcount after alias: 2 Refcount after del b: 1
Go

a := &Node{Val: 42}  // heap-allocated, GC-managed

Pitfalls to Remember:
  • Python: Circular references defeat reference counting — the cyclic GC handles them, but they can delay cleanup. Use weakref for back-pointers.
  • Go: Holding references to large objects in long-lived goroutines prevents GC — set references to nil when done.

Go
type Ordered interface {
    ~int | ~float64 | ~string
}

func FindMax[T Ordered](items []T) T {
    best := items[0]
    for _, v := range items[1:] {
        if v > best { best = v }
    }
    return best
}

fmt.Println(FindMax([]int{3, 1, 4, 1, 5}))
fmt.Println(FindMax([]string{"apple", "banana", "cherry"}))

TypeScript
function findMax<T>(items: T[], key: (item: T) => number): T {
    return items.reduce((best, item) =>
        key(item) > key(best) ? item : best
    );
}

fn find_max<T: PartialOrd>(items: &[T]) -> &T {
    let mut best = &items[0];

        if item > best { best = item; }

}

CSV & Data Processing I/O & Data Processing

CSV (Comma-Separated Values) remains one of the most common data exchange formats. Whether importing spreadsheet data, processing log files, or exchanging records between systems, every developer needs to read and write CSV reliably — handling edge cases like quoted fields, embedded commas, and different delimiters.

CSV processing has two main approaches: (1) Record-by-record — read one row at a time for memory efficiency with large files, and (2) Bulk loading — read the entire file into a data structure for analysis. Most languages provide both a low-level row reader and higher-level abstractions that map rows to typed objects.

Example 1: Reading CSV — Row-by-Row Parsing

Parse a CSV string (or file) into individual records, handling the header row separately.

Python
import csv
from io import StringIO

csv_data = """name,age,city
Alice,30,New York
Bob,25,San Francisco
Carol,35,Chicago"""

# DictReader — each row becomes a dict keyed by header
reader = csv.DictReader(StringIO(csv_data))
for row in reader:
    print(f"{row['name']} is {row['age']} from {row['city']}")

# Regular reader — each row is a list
reader2 = csv.reader(StringIO(csv_data))
header = next(reader2)

string escapeCSV(const string& field) {
    if (field.find(',') != string::npos ||
        field.find('"') != string::npos ||
        field.find('\n') != string::npos) {
        string escaped = field;

Common Mistakes:
  • Python: Using .split(",") instead of csv.reader — breaks on fields containing commas like "New York, NY".
    ❌ row = line.split(",")
    ✅ row = next(csv.reader([line]))
  • Go: Not handling csv.ErrFieldCount — rows with inconsistent column counts cause silent data corruption.
    ❌ Ignoring the error from reader.Read()
    ✅ Set reader.FieldsPerRecord = -1 for variable-width or check the error
  • TypeScript: Naive split(",") does not handle quoted fields — "Smith, Jr.",30 becomes three fields instead of two.
    ❌ line.split(",") for production data
    ✅ Use papaparse or a proper parser
  • Rust: Forgetting that csv::Reader owns the header row — calling .headers() after starting iteration panics.
    ❌ Iterate first, then call .headers()
    ✅ Call .headers() before iterating or use .has_headers(false)
  • C++: getline(stream, cell, ',') does not handle quoted fields — a comma inside quotes breaks the parse.
    ❌ Simple getline loop for real-world CSV
    ✅ Use rapidcsv or a proper state-machine parser
Key Rules:
  • Never use split(",") for production CSV — always use a proper library that handles quoting, escaping, and multi-line fields.
  • Validate column counts — malformed rows with extra or missing columns cause subtle bugs that are hard to track down.
  • Handle encoding — CSV files from Excel are often encoded as UTF-8 with BOM or Latin-1. Detect and handle the encoding before parsing.

Unit Testing Testing & Error Handling

Unit tests are the first line of defense against regressions. They document expected behavior, enable refactoring with confidence, and catch bugs early. Every professional codebase has them, and interviewers expect you to write them.

Common Mistakes:
  • Python: Test files must start with test_ or end with _test.py — pytest silently ignores files that do not match.
    ❌ palindrome_tests.py
    ✅ test_palindrome.py
  • Go: Test functions must start with Test and take *testing.T — testFoo(t *testing.T) (lowercase) is silently skipped.
    ❌ func testFoo(t *testing.T)
    ✅ func TestFoo(t *testing.T)
  • TypeScript: Forgetting await on async test assertions causes tests to pass even when they should fail.
    ❌ expect(fetchData()).resolves.toBe(...)
    ✅ await expect(fetchData()).resolves.toBe(...)
  • Rust: Tests that panic are marked as failures, but assert! without a message gives unhelpful output — always add a message.
    ❌ assert_eq!(got, want)
    ✅ assert_eq!(got, want, "for input {:?}", input)
  • C++: Using ASSERT_* instead of EXPECT_* aborts the test on first failure — use EXPECT_* to see all failures at once.
    ❌ ASSERT_EQ(a, b); ASSERT_EQ(c, d); — second never runs if first fails
    ✅ EXPECT_EQ(a, b); EXPECT_EQ(c, d);
Key Rules:
  • One assertion per behavior — each test should verify one specific behavior. Multiple assertions are fine if they test the same logical concept.
  • Test names describe the scenario — test_empty_string_returns_true is better than test_1. Good names serve as documentation.
  • Test edge cases first — empty input, zero, negative numbers, maximum values, and boundary conditions catch the most bugs.

Mocking & Test Doubles Testing & Error Handling

Real code depends on databases, APIs, file systems, and clocks. Mocking replaces these external dependencies with controlled fakes so you can test your logic in isolation — without a network connection, without a database, and with deterministic results every time.

Test doubles come in several flavors: Stubs return canned data, Mocks verify that specific calls were made, Fakes are simplified working implementations (e.g., an in-memory database). The key insight is that you design your code to accept interfaces (or function parameters) rather than concrete types, making substitution easy.

Language Comparison — Mocking Approaches: Python’s unittest.mock is the most flexible — MagicMock auto-creates attributes and methods on the fly. Go deliberately avoids reflection-based mocking — you write manual mock structs that implement interfaces. This is more verbose but keeps code explicit. TypeScript’s vi.fn()/jest.fn() creates mock functions with call tracking. Rust relies on traits for testability — the mockall crate can auto-generate mocks from traits. C++’s gmock uses macros to generate mock classes from virtual interfaces, with powerful expectation syntax.
Common Mistakes:
  • Python: Using patch on the wrong module path — you must patch where the name is used, not where it is defined.
    ❌ @patch("requests.get") when the code does from requests import get
    ✅ @patch("mymodule.get")
  • Go: Not using interfaces — if your function takes a concrete http.Client, you cannot mock it. Accept an interface instead.
    ❌ func Fetch(client *http.Client)
    ✅ func Fetch(client HTTPClient) where HTTPClient is an interface
  • TypeScript: Mocking modules without vi.mock() at the top level — hoisting matters, and mocks set up inside it() may not take effect.
    ❌ Setting up module mocks inside test functions
    ✅ Call vi.mock("./module") at the top of the test file
  • Rust: Trait objects (dyn Trait) have a small runtime cost due to vtable dispatch — for hot paths, use generics with trait bounds instead.
    ❌ Always using &dyn Trait in production code for testability
    ✅ Use generics <T: Trait> for zero-cost abstraction; dyn only where needed
  • C++: MOCK_METHOD requires virtual methods — non-virtual functions cannot be mocked with gmock without template injection.
    ❌ Trying to mock a final class
    ✅ Design with abstract base classes or use dependency injection via templates
Key Rules:
  • Design for testability — accept interfaces/traits instead of concrete types. If you cannot mock it, you cannot test it in isolation.
  • Mock at the boundary — mock external services (HTTP, DB, file system), not internal helper functions. Over-mocking makes tests brittle.
  • Verify behavior, not implementation — test what happens (correct return, notification sent), not how (which internal method was called).

Benchmarking Testing & Error Handling

Performance claims without measurements are just opinions. Benchmarking gives you hard numbers to compare algorithms, validate optimizations, and prevent performance regressions. Every language has idiomatic tools for this, and knowing them is essential for writing fast code.

A good benchmark runs the code many times to amortize overhead, uses warm-up rounds to fill caches, and reports statistical summaries (mean, median, standard deviation). Crucially, you must prevent the compiler from optimizing away the code under test.

    auto start = chrono::high_resolution_clock::now();
    for (int i = 0; i < runs; ++i) f();
    return chrono::duration_cast<chrono::microseconds>(
        chrono::high_resolution_clock::now() - start).count();
}

int n = 10'000, runs = 100;
cout << "loop:    " << timeit([&]{ concatLoop(n); }, runs) << "us" << endl;
cout << "reserve: " << timeit([&]{ concatReserve(n); }, runs) << "us" << endl;

Language Comparison — Benchmarking Tools: Python’s timeit is built-in and handles GC pausing and import isolation. For profiling, use cProfile. Go has the best built-in benchmarking — testing.B auto-calibrates iteration count, reports allocations with -benchmem, and integrates with pprof for profiling. TypeScript has no standard benchmark tool — vitest bench is emerging, benchmark.js is mature. Rust’s criterion is the gold standard for microbenchmarking with statistical analysis, confidence intervals, and regression detection. C++’s Google Benchmark prevents dead code elimination with DoNotOptimize and tracks time, CPU, and throughput.
Common Mistakes:
  • Python: Benchmarking with time.time() — low resolution and affected by system clock changes. Use timeit or time.perf_counter().
    ❌ start = time.time(); ...; elapsed = time.time() - start
    ✅ timeit.repeat(stmt, number=N, repeat=R)
  • Go: Using a fixed iteration count instead of b.N — the benchmark framework auto-tunes b.N to get stable results.
    ❌ for i := 0; i < 1000; i++
    ✅ for i := 0; i < b.N; i++
  • TypeScript: V8’s JIT compiler can optimize away unused results — store and use the result to prevent dead code elimination.
    ❌ for (let i = 0; i < N; i++) compute(x);
    ✅ let result; for (let i = 0; i < N; i++) result = compute(x); return result;
  • Rust: Forgetting black_box() — the compiler can evaluate pure functions at compile time and replace the loop with a constant.
    ❌ b.iter(|| sum_formula(10_000))
    ✅ b.iter(|| sum_formula(black_box(10_000)))
  • C++: Compiler optimizing away the benchmarked code — use benchmark::DoNotOptimize(result) or volatile.
    ❌ auto r = compute(x); (compiler may eliminate)
    ✅ benchmark::DoNotOptimize(compute(x));
Key Rules:
  • Benchmark, don’t guess — intuition about performance is often wrong. Measure before and after every optimization.
  • Isolate what you measure — separate setup from the timed code. Benchmark one thing at a time.
  • Use multiple runs — a single timing is meaningless due to system noise. Report the minimum or median of many runs.
