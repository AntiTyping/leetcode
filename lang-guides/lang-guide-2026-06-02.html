Pitfalls to Remember:
  • Python: Regular dataclass is mutable by default — use @dataclass(frozen=True) if you need hashability or immutability.
  • Go: Structs are value types — passing a struct to a function copies it. Use *Point to avoid expensive copies of large structs.

  • C++: Aggregate initialization order depends on declaration order — reordering struct fields silently changes {} initialization.

Linked Lists & Trees Recursive Structures 2 days since · rep 1
Node-based recursive data structures. Python uses classes, Go uses pointer-based structs, TS uses class/interface, Rust uses Box<Node>/Option, C++ uses unique_ptr/raw pointers.
Python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def to_list(head):
    result = []
    while head:
        result.append(head.val)
        head = head.next
    return result

head = ListNode(1, ListNode(2, ListNode(3)))
print(to_list(head))

1

Pitfalls to Remember:
  • Python: Forgetting to set prev_node.next = None when detaching — creates cycles that cause infinite loops.
  • Go: Nil pointer dereference — always check if node != nil before accessing node.Val or node.Next.
  • TypeScript: null vs undefined — stick to one for “no next node”; null is conventional for linked list terminators.

  • C++: Using raw new without delete leaks memory — prefer unique_ptr for automatic cleanup.

        let val = self.a;
        (self.a, self.b) = (self.b, self.a + self.b);
        Some(val)

  • Python: Enum members are singletons — Color.RED is Color.RED is always true, but Color.RED == 1 is False (use IntEnum if you need integer comparison).

Functions & Abstractions — Week 3

Abstraction mechanisms that let you write reusable, type-safe, composable code across all five languages.

Generic Programming Functions & Abstractions

Generics let you write a single function or data structure that works with any type, without sacrificing type safety. Without generics, you either duplicate code for each type or lose compile-time checks by using any/interface{}.

Every language has a mechanism for writing code that is parameterized by type. The approaches range from duck typing (no explicit annotation needed) to full monomorphization (the compiler generates specialized code for each type used).

The key question across languages: when does the type get resolved? Python resolves at runtime (duck typing), while Go, TypeScript, Rust, and C++ resolve at compile time.

Example 1: Generic Identity Function

The simplest possible generic — a function that returns its argument unchanged, preserving the exact type.

Python
# Duck typing: works without generics
def identity(x):
    return x

# With type hints (Python 3.12+):
def identity_typed[T](x: T) -> T:
    return x

# Pre-3.12 style:
from typing import TypeVar
T = TypeVar("T")
def identity_old(x: T) -> T:
    return x

print(identity(42))
print(identity("hello"))
42 hello

// Go generics (1.18+)
func Identity[T any](x T) T {
    return x
}

Example 3: Constrained Generics — Only Comparable Types

Generics become truly powerful when you constrain the type parameter — “T must be comparable” or “T must be printable.”

Python
from typing import TypeVar, Protocol

class Comparable(Protocol):
    def __lt__(self, other: "Comparable") -> bool: ...

T = TypeVar("T", bound=Comparable)

def min_val(a: T, b: T) -> T:
    return a if a < b else b

print(min_val(3, 7))
print(min_val("apple", "banana"))
3 apple
Go
import "golang.org/x/exp/constraints"

// constraints.Ordered covers int, float, string, etc.
func MinVal[T constraints.Ordered](a, b T) T {
    if a < b {
        return a
    }
    return b
}

fmt.Println(MinVal(3, 7))
fmt.Println(MinVal("apple", "banana"))
3 apple
TypeScript
// "extends" constrains the type parameter

Rust

fn min_val<T: PartialOrd>(a: T, b: T) -> T {

Example 4: Generic Function with Multiple Type Parameters

Transform a collection using a generic mapping function — input type differs from output type.

Python
from typing import TypeVar, Callable

A = TypeVar("A")
B = TypeVar("B")

def map_list(fn: Callable[[A], B], lst: list[A]) -> list[B]:
    return [fn(x) for x in lst]

lengths = map_list(len, ["hello", "hi", "world"])
print(lengths)

squares = map_list(lambda x: x * x, [1, 2, 3, 4])
print(squares)
[5, 2, 5] [1, 4, 9, 16]
Go
func MapSlice[A any, B any](fn func(A) B, items []A) []B {
    result := make([]B, len(items))
    for i, v := range items {
        result[i] = fn(v)
    }
    return result
}

lengths := MapSlice(func(s string) int { return len(s) },
    []string{"hello", "hi", "world"})
fmt.Println(lengths)

squares := MapSlice(func(x int) int { return x * x },
    []int{1, 2, 3, 4})
fmt.Println(squares)

  • Python: Thinking type hints enforce anything at runtime — identity_typed[int]("oops") runs fine; only static checkers catch it.
    ❌ Relying on generics for runtime type safety
    ✅ Use generics for documentation and static analysis
  • Go: Trying to use [T any] with operators like < or + — you need constraints.Ordered or a custom constraint.
    ❌ func Add[T any](a, b T) T { return a + b } — won’t compile
    ✅ func Add[T constraints.Integer | constraints.Float](a, b T) T { return a + b }
  • TypeScript: Over-constraining generics when a simpler type suffices.
    ❌ function len<T extends { length: number }>(x: T): number when (x: { length: number }) => number works
    ✅ Use generics when you need to preserve the input type in the output
  • Rust: Forgetting trait bounds — fn print_it<T>(x: T) { println!("{}", x); } won’t compile without T: Display.
    ❌ Assuming T can do anything
    ✅ Add exactly the trait bounds you need: T: Display + Clone
  • C++: Template errors produce notoriously long error messages. Use C++20 concepts to get clearer errors.
    ❌ Debugging a 200-line template error
    ✅ template<typename T> requires std::integral<T> gives a clear “constraint not satisfied” message
Key Rules:
  • Use generics when the same logic applies to multiple types — if you find yourself copy-pasting a function and only changing the type, it should be generic.
  • Constrain as tightly as needed, no more — use any when you truly need no constraints, specific bounds when you need operations on T.
  • Generics are not a substitute for interfaces/traits — use generics for type-parameterized code, interfaces for behavioral abstraction.

Interfaces & Traits Functions & Abstractions

Interfaces and traits define contracts that types must fulfill. They are the primary mechanism for polymorphism — writing code that works with any type that has the right behavior, without knowing the concrete type at compile time.

The fundamental idea: separate what a type can do from how it does it. Two critical differences across languages:

Structural vs. nominal: In Go and TypeScript, a type satisfies an interface automatically if it has the right methods/fields (structural). In Rust and C++, you must explicitly implement the trait/inherit the class (nominal). Python supports both (Protocol = structural, ABC = nominal).

Static vs. dynamic dispatch: Rust defaults to static dispatch (monomorphized, zero cost) but supports dynamic dispatch via dyn Trait. Go interfaces always use dynamic dispatch. C++ virtual functions use dynamic dispatch; templates use static.

Common Mistakes:
  • Python: Using ABC when Protocol suffices — ABCs require inheritance, Protocols work structurally and are better for type checking third-party code.
    ❌ Forcing third-party classes to inherit your ABC
    ✅ Use Protocol for duck-typed interfaces
  • Go: Defining interfaces with too many methods — Go idiom is small interfaces (1-3 methods). The standard library’s io.Reader has just one method.
    ❌ A 10-method interface
    ✅ Several small interfaces composed together
  • TypeScript: Confusing interface and type — both can define object shapes, but interfaces support extends and declaration merging.
    ❌ Using type when you need to extend across files
    ✅ Use interface for public API contracts
  • Rust: Using dyn Trait everywhere — dynamic dispatch has a vtable lookup cost. Use impl Trait or generics for static dispatch when possible.
    ❌ fn area(s: &dyn HasArea) when only one type is used
    ✅ fn area(s: &impl HasArea) for static dispatch
  • C++: Forgetting virtual ~Displayable() = default; — without a virtual destructor, deleting through a base pointer leaks derived resources.
    ❌ class Base { public: string name(); }; with delete basePtr;
    ✅ Always add virtual ~Base() = default;
Key Rules:
  • Program to an interface, not an implementation — function parameters should accept the most general interface that provides the operations you need.
  • Prefer small, focused interfaces — “The bigger the interface, the weaker the abstraction” (Rob Pike). io.Reader with one method is more powerful than a 20-method File interface.
  • Default implementations reduce boilerplate — Rust traits and TypeScript interfaces (via abstract classes) let you provide default methods that implementors can override.

Decorators & Middleware Patterns Functions & Abstractions

The decorator/middleware pattern lets you layer cross-cutting concerns (logging, timing, auth, caching) onto functions without modifying their core logic. This is the “open/closed principle” in action: extend behavior without changing existing code.

The core idea: wrap a function in another function that adds behavior before and/or after the original call. The wrapper has the same signature as the original, so callers don’t know they’re talking to a decorator.

Python has built-in @decorator syntax. Other languages achieve the same pattern through higher-order functions, middleware chains, or wrapper types.

Example 3: Stacking Multiple Decorators

The real power: composing multiple decorators. Order matters — outermost decorator runs first.

Python
import functools

def log_call(fn):
    @functools.wraps(fn)
    def wrapper(*args, **kwargs):
        print(f"  -> calling {fn.__name__}{args}")
        result = fn(*args, **kwargs)
        print(f"  <- {fn.__name__} returned {result}")
        return result
    return wrapper

def cache(fn):
    memo = {}
    @functools.wraps(fn)
    def wrapper(*args):
        if args not in memo:
            memo[args] = fn(*args)
        else:
            print(f"  [cache hit for {args}]")
        return memo[args]
    return wrapper

@log_call     # outer: runs first

Language Comparison: Python has the richest decorator support with @decorator syntax — decorators are a first-class language feature. Go’s middleware pattern (func(Handler) Handler) is the standard approach for HTTP servers and is pervasive in the ecosystem. TypeScript decorators exist but are experimental; higher-order functions are the practical approach. Rust doesn’t have runtime decorators — procedural macros (compile-time) or closure wrapping are the closest equivalents. C++ achieves the pattern through templates and lambdas, though with more syntactic overhead.
Common Mistakes:
  • Python: Forgetting @functools.wraps(fn) — without it, the wrapper replaces the function’s __name__ and __doc__, making debugging harder.
    ❌ def wrapper(*args): ... — fn.__name__ becomes “wrapper”
    ✅ @functools.wraps(fn) preserves the original metadata
  • Go: Creating middleware that doesn’t call next.ServeHTTP() — this silently drops the request, causing mysterious failures.
    ❌ Returning early without calling the next handler (except for auth rejection)
    ✅ Always call next.ServeHTTP(w, r) in the happy path
  • TypeScript: Losing the this context when wrapping class methods — arrow functions capture outer this, but regular functions don’t.
    ❌ Wrapping a method and losing this binding
    ✅ Use .bind() or arrow functions to preserve context
  • Rust: Trying to return closures from functions hits lifetime issues — you need Box<dyn Fn(...)> or impl Fn(...) return types.
    ❌ fn make_adder(n: i32) -> Fn(i32) -> i32 — trait as return type
    ✅ fn make_adder(n: i32) -> impl Fn(i32) -> i32
  • C++: Capturing local variables by reference in a lambda returned from a function — the reference dangles after the function returns.
    ❌ [&name] in a returned lambda
    ✅ [=] or [name] to capture by value
Key Rules:
  • Decorators should be transparent — the wrapped function should behave identically to the original from the caller’s perspective (same signature, same return type).
  • Order matters when stacking — @A @B def f means A(B(f)). The outermost decorator’s wrapper runs first on call, last on return.
  • Keep decorators simple and composable — each should do one thing (log, time, cache, validate). Complex behavior comes from composition, not from one mega-decorator.

Module Systems Functions & Abstractions

Modules organize code into logical units, control what is public vs. private, and manage dependencies. Understanding module systems is essential for any project larger than a single file — and the rules vary dramatically between languages.

Every language answers three questions differently: (1) How do you define a module? (2) How do you import from it? (3) How do you control visibility?

The approaches range from file-based (Python, TypeScript, Rust) to directory/package-based (Go) to preprocessor-based (C++, with C++20 modules as the modern alternative).

Example 2: Controlling Visibility — Public vs. Private

Every language has a way to hide implementation details. The mechanisms differ significantly.

Python
# Python uses conventions, not enforcement

# _private: leading underscore = "please don't use this"
_cache = {}

# __mangled: double underscore in class = name-mangled
class MyClass:
    def __init__(self):
        self.__secret = 42   # becomes _MyClass__secret

    def get_secret(self):
        return self.__secret

# __all__ controls what `from module import *` exports
__all__ = ["public_func"]   # only public_func is exported by *

def public_func():
    return "I'm public"

def _private_func():
    return "I'm conventionally private"

obj = MyClass()
print(obj.get_secret())
# print(obj.__secret)            # AttributeError
print(obj._MyClass__secret)      # Works! Python doesn't truly enforce
42 42

        this.email = email;
        this.id = nextId();

// pub = public, no pub = private (to this module)
// pub(crate) = public within crate only

    pub struct User {
        pub name: String,
        pub(crate) email: String,   // crate-visible only
        id: u32,                    // private

            User {
                name: name.into(),
                email: email.into(),

        pub fn id(&self) -> u32 { self.id }
    }

}

Language Comparison — Module Philosophy: Python is file-based: each .py file is a module, directories with __init__.py are packages. Visibility is by convention (_private). Go is directory-based: every .go file in a directory shares the same package. Visibility is by naming convention (uppercase = exported). TypeScript is file-based with explicit export. Barrel files (index.ts) are a common pattern for clean public APIs. Rust has the most explicit system: mod declarations form a tree, pub controls visibility at fine granularity (pub(crate), pub(super)). C++ traditionally uses header/source separation with #include; C++20 modules are the modern replacement but adoption is still growing.
Common Mistakes:
  • Python: Circular imports — if a.py imports from b.py and b.py imports from a.py, you get ImportError. Fix by moving shared types to a third module or using late imports.
    ❌ Mutual top-level imports between two modules
    ✅ Import inside the function that needs it, or restructure
  • Go: Circular package imports are a compile error with no workaround. Go enforces a strict DAG (directed acyclic graph) of package dependencies.
    ❌ Package a imports b, package b imports a
    ✅ Extract shared types into a third package
  • TypeScript: Forgetting export on a type and wondering why another file can’t see it. Also: export default mixed with named exports causes confusion.
    ❌ Mixing default and named exports in the same file
    ✅ Stick to named exports for consistency
  • Rust: Forgetting mod mymodule; in the parent — Rust doesn’t auto-discover files. Every module must be declared.
    ❌ Creating src/utils.rs without mod utils; in main.rs
    ✅ Always add the mod declaration in the parent module
  • C++: Missing #pragma once or include guards — including the same header twice causes redefinition errors.
    ❌ #include "user.h" in two files that both get included
    ✅ Always start headers with #pragma once
Key Rules:
  • Minimize public API surface — only export what consumers need. Private-by-default (Rust) is safer than public-by-default (Python).
  • Avoid circular dependencies — every language struggles with them. Design module dependencies as a DAG (directed acyclic graph).
  • Use barrel/re-export files for clean APIs — Python’s __init__.py, TypeScript’s index.ts, and Rust’s pub use re-exports give consumers a single import point.
