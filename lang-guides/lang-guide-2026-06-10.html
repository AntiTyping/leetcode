Logging Across Languages Testing 3 days since · rep 0
logging (Python), log/slog (Go), winston/pino (TS), tracing (Rust), spdlog (C++). Structured logging with key-value context fields is preferred over string interpolation. Log levels: DEBUG, INFO, WARN, ERROR. Always include correlation IDs in distributed systems.
Python
import logging
import json

# Structured logging with custom formatter
class JSONFormatter(logging.Formatter):
    def format(self, record):
        log_obj = {
            "level": record.levelname,
            "message": record.getMessage(),
            "module": record.module,
        }
        if hasattr(record, "user_id"):
            log_obj["user_id"] = record.user_id
        return json.dumps(log_obj)

logger = logging.getLogger("app")
logger.setLevel(logging.DEBUG)
handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())
logger.addHandler(handler)

logger.info("User logged in", extra={"user_id": 42})
{"level": "INFO", "message": "User logged in", "module": "main", "user_id": 42}

Pitfalls to Remember:
  • Python: logging.basicConfig() is a no-op if called after a handler is already configured — always configure explicitly or call it first.
  • Go: slog (Go 1.21+) replaces the older log package — prefer it for structured logging. The default log.Println has no structured fields.
  • TypeScript: console.log is not structured logging — use winston or pino for production. Never log sensitive data (tokens, passwords).
  • Rust: tracing spans are async-aware; log crate is simpler but lacks span context for distributed tracing.
  • C++: spdlog is not header-only by default — link properly. Avoid cout for production logging (no levels, no rotation).
Type Systems Deep Dive Testing 3 days since · rep 0
Python: gradual typing (mypy). Go: structural interfaces, nominal for generics. TypeScript: structural, union/intersection types. Rust: nominal with traits, no subtyping. C++: nominal with templates (duck-typed at compile time). Structural = shape matters; nominal = name matters.
Python
from typing import Protocol, runtime_checkable

@runtime_checkable
class Drawable(Protocol):
    def draw(self) -> str: ...

class Circle:
    def __init__(self, r: float):
        self.r = r
    def draw(self) -> str:
        return f"Circle(r={self.r})"

class Square:
    def __init__(self, side: float):
        self.side = side
    def draw(self) -> str:
        return f"Square(side={self.side})"

def render(shapes: list[Drawable]) -> None:
    for s in shapes:
        print(s.draw())

# Structural — no explicit "implements Drawable"
render([Circle(5), Square(3)])
Circle(r=5) Square(side=3)

Iterators & Generators Iteration 8 days since · rep 2
yield generators (Python), iter.Seq (Go 1.23+), generator functions (TS), Iterator trait (Rust), ranges/coroutines (C++20). Lazy evaluation: values are computed on demand, not stored in memory. Chain iterators for composable data pipelines.
Python
from typing import Iterator, TypeVar

T = TypeVar("T")

def take(n: int, it: Iterator[T]) -> list[T]:
    """Take first n items from an iterator."""
    result = []
    for _ in range(n):
        try:
            result.append(next(it))
        except StopIteration:
            break
    return result

def fibonacci() -> Iterator[int]:
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b

# Lazy — only computes 8 values
print(take(8, fibonacci()))
[0, 1, 1, 2, 3, 5, 8, 13]
Go
package main

import "fmt"

func fibonacci(yield func(int) bool) {
    a, b := 0, 1
    for {
        if !yield(a) {
            return
        }
        a, b = b, a+b
    }
}

func take(n int, seq func(func(int) bool)) []int {
    result := make([]int, 0, n)
    seq(func(v int) bool {
        result = append(result, v)
        return len(result) < n
    })
    return result
}

func main() {
    fmt.Println(take(8, fibonacci))
}
[0 1 1 2 3 5 8 13]
TypeScript
function* fibonacci(): Generator<number> {

function take<T>(n: number, iter: Iterator<T>): T[] {
    const result: T[] = [];
    for (let i = 0; i < n; i++) {
        const { value, done } = iter.next();
        if (done) break;
        result.push(value);
    }
    return result;
}

Rust
struct Fibonacci {
    a: u64,
    b: u64,
}

impl Fibonacci {
    fn new() -> Self { Fibonacci { a: 0, b: 1 } }
}

impl Iterator for Fibonacci {
    type Item = u64;
    fn next(&mut self) -> Option<u64> {
        let val = self.a;
        let new_b = self.a + self.b;
        self.a = self.b;
        self.b = new_b;
        Some(val)
    }
}

fn main() {
    let first_8: Vec<u64> = Fibonacci::new().take(8).collect();
    println!("{:?}", first_8);
}

class Fibonacci {
    uint64_t a = 0, b = 1;

        b = new_b;

Challenge (all 5 languages):
1
Implement a take(n, iter) function that lazily takes the first n items from an infinite iterator. Test it with a Fibonacci generator.
Pitfalls to Remember:
  • Python: Generators exhaust after one pass — calling list(gen) twice gives [] the second time. Create a new generator each time.
  • Go: Go 1.23 range-over-func iterators use callbacks, not coroutines — you cannot yield from a nested function.
  • TypeScript: for...of works with generators, but Array.from() on an infinite generator will hang — always use take first.
  • Rust: .take(n) returns a new iterator (lazy) — you still need .collect() to materialize it.
  • C++: No built-in generator; C++20 coroutines exist but are verbose. A stateful class with next() is the simplest approach.
Linked Lists & Trees Recursive Structures 8 days since · rep 2
Node-based structures with pointers/references. Python: class with Optional[Node]. Go: struct with *Node. TS: class with Node | null. Rust: Option<Box<Node>> (ownership makes linked lists hard). C++: unique_ptr<Node>. Reverse a linked list: iterate with prev/curr/next pointers.
Python
from __future__ import annotations
from typing import Optional

class ListNode:
    def __init__(self, val: int, next: Optional[ListNode] = None):
        self.val = val
        self.next = next

def reverse(head: Optional[ListNode]) -> Optional[ListNode]:
    prev, curr = None, head
    while curr:
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt
    return prev

def to_list(head: Optional[ListNode]) -> list[int]:
    result = []
    while head:
        result.append(head.val)
        head = head.next
    return result

head = ListNode(1, ListNode(2, ListNode(3, ListNode(4))))
print("Before:", to_list(head))
head = reverse(head)
print("After: ", to_list(head))
Before: [1, 2, 3, 4] After: [4, 3, 2, 1]
Go
package main

import "fmt"

type ListNode struct {
    Val  int
    Next *ListNode
}

func reverse(head *ListNode) *ListNode {
    var prev *ListNode
    curr := head
    for curr != nil {
        nxt := curr.Next
        curr.Next = prev
        prev = curr
        curr = nxt
    }
    return prev
}

func toSlice(head *ListNode) []int {
    var result []int
    for head != nil {
        result = append(result, head.Val)
        head = head.Next
    }
    return result
}

func main() {

    head = reverse(head)

}

Rust
type Link = Option<Box<ListNode>>;

struct ListNode {
    val: i32,
    next: Link,
}

fn reverse(mut head: Link) -> Link {
    let mut prev: Link = None;
    while let Some(mut node) = head {
        head = node.next.take();
        node.next = prev;
        prev = Some(node);
    }
    prev
}

fn to_vec(mut head: &Link) -> Vec<i32> {
    let mut result = Vec::new();
    while let Some(ref node) = head {
        result.push(node.val);
        head = &node.next;
    }
    result
}

fn from_slice(vals: &[i32]) -> Link {
    let mut head: Link = None;
    for &v in vals.iter().rev() {
        head = Some(Box::new(ListNode { val: v, next: head }));
    }
    head
}

fn main() {
    let head = from_slice(&[1, 2, 3, 4]);
    println!("Before: {:?}", to_vec(&head));
    let head = reverse(head);
    println!("After:  {:?}", to_vec(&head));
}
Before: [1, 2, 3, 4] After: [4, 3, 2, 1]

#include <iostream>
#include <memory>
#include <vector>
using namespace std;

struct ListNode {
    int val;
    unique_ptr<ListNode> next;
    ListNode(int v, unique_ptr<ListNode> n = nullptr) : val(v), next(move(n)) {}
};

unique_ptr<ListNode> reverse(unique_ptr<ListNode> head) {
    unique_ptr<ListNode> prev;
    while (head) {
        auto nxt = move(head->next);
        head->next = move(prev);
        prev = move(head);
        head = move(nxt);
    }
    return prev;
}

vector<int> toVec(const ListNode* head) {
    vector<int> result;
    while (head) { result.push_back(head->val); head = head->next.get(); }
    return result;
}

    for (size_t i = 0; i < v.size(); ++i) {

Pitfalls to Remember:
  • Rust: Linked lists are notoriously hard because Box owns the next node — you must use .take() to temporarily extract ownership during reversal.
  • C++: unique_ptr requires std::move for ownership transfer — forgetting it causes a compile error.

Sorting & Comparators Sorting 8 days since · rep 2
sorted(key=) (Python), sort.Slice (Go), Array.sort() (TS), .sort_by() (Rust), std::sort (C++). Multi-field sorting: compare primary field first, then secondary on tie. All built-in sorts are O(n log n). Python’s Timsort is stable; Go’s is not (use sort.SliceStable).
Python

Tuples, Structs & Records Structured Data 8 days since · rep 2
tuple/NamedTuple/dataclass (Python), struct (Go), interfaces/type aliases (TS), struct/tuple (Rust), struct/tuple (C++). Destructuring: unpack fields into variables. Python dataclass auto-generates __init__, __repr__, __eq__. Go structs have zero values for all fields.
Python
from dataclasses import dataclass
import math

@dataclass
class Point:
    x: float
    y: float

    def distance_to(self, other: "Point") -> float:
        return math.hypot(self.x - other.x, self.y - other.y)

Python Deep Dive: Comprehensions & Dataclasses Language Deep Dive
Comprehensions are Python’s most expressive feature for transforming data in a single readable expression. Dataclasses eliminate boilerplate for data-holding classes. Together, they form the backbone of idiomatic Python.

List/Dict/Set Comprehensions

Comprehensions replace verbose loops with a declarative syntax: [expr for item in iterable if condition]. They are faster than equivalent for loops because the iteration happens in C under the hood. Dict comprehensions ({k: v for ...}) and set comprehensions ({expr for ...}) follow the same pattern.

Walrus Operator in Comprehensions

The walrus operator := (Python 3.8+) assigns and returns a value in one expression. In comprehensions, it avoids computing an expensive expression twice: once for filtering and once for the output.

Nested Comprehensions

Nested comprehensions flatten loops: [x for row in matrix for x in row]. The order matches the equivalent nested for loops (outer loop first). Beyond two levels of nesting, prefer explicit loops for readability.

Dataclasses

@dataclass auto-generates __init__, __repr__, __eq__ (and optionally __hash__, __order__). Use field() for default factories, __post_init__ for validation, and frozen=True for immutability.

Python
# --- List, Dict, Set Comprehensions ---

# Dict comprehension — word to length

print("Lengths:", lengths)

# Set comprehension — unique first letters

Go: No comprehension syntax. Use for loops with append. Structs are the only named-data type; no auto-generated methods — write constructors manually.

Common Mistakes:
  • Mutable default: tags: list = [] is shared across all instances! Always use field(default_factory=list).
  • Nested comprehension order: [x for row in matrix for x in row] reads left-to-right (outer first). It is not [x for x in row for row in matrix].
  • Walrus scope leak: The variable assigned by := in a comprehension leaks into the enclosing scope in Python 3.8+. This is intentional but surprising.
  • Generator vs list: (x for x in range(1000000)) (parentheses) is a generator (lazy, O(1) memory). [x for x in range(1000000)] (brackets) builds the entire list in memory.
Python Comprehension Rules:
  • Use list comprehensions when you need the full result; use generators when you only iterate once.
  • Never nest more than two levels of comprehension — use explicit loops for readability.
  • Prefer @dataclass over manual __init__ for any class that primarily holds data.
  • Always use field(default_factory=...) for mutable defaults (lists, dicts, sets).
  • Use frozen=True when instances should be immutable (e.g., configuration, cache keys).

Go Deep Dive: Goroutine Scheduling & Channel Patterns Language Deep Dive
Go’s goroutines and channels are its killer feature for concurrency. Understanding the M:N scheduler, fan-out/fan-in, and select-based patterns is essential for writing correct concurrent Go programs.

Goroutines vs OS Threads

Goroutines are green threads multiplexed onto OS threads by Go’s runtime (M:N scheduling). A goroutine starts with ~4KB stack (grows dynamically), while an OS thread uses ~1MB. You can run millions of goroutines; you cannot run millions of OS threads. GOMAXPROCS controls how many OS threads run goroutines simultaneously (defaults to the number of CPU cores).

Channel Patterns

Fan-out: multiple goroutines read from the same channel. Fan-in: multiple channels feed into one. Select with timeout: select on multiple channels with a time.After case for deadlock prevention. Channel-based semaphore: a buffered channel limits concurrency.

Go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    // GOMAXPROCS — number of OS threads for goroutines
    fmt.Println("GOMAXPROCS:", runtime.GOMAXPROCS(0))
    fmt.Println("NumCPU:", runtime.NumCPU())
    fmt.Println("NumGoroutine:", runtime.NumGoroutine())

    // Goroutines are cheap — start 10,000
    done := make(chan bool)
    for i := 0; i < 10000; i++ {
        go func() {
            // Each goroutine just signals done
            done <- true
        }()
    }

    for i := 0; i < 10000; i++ {
        <-done
    }
    fmt.Println("Launched and joined 10,000 goroutines")
}
GOMAXPROCS: 8 NumCPU: 8 NumGoroutine: 1 Launched and joined 10,000 goroutines
Go
package main

import (
    "fmt"
    "sync"
)

// Fan-out: multiple workers read from one channel
// Fan-in: all results flow into one output channel
func main() {
    jobs := make(chan int, 10)
    results := make(chan string, 10)

    // Fan-out: 3 workers
    var wg sync.WaitGroup
    for w := 0; w < 3; w++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for job := range jobs {
                results <- fmt.Sprintf("worker %d processed job %d", id, job)
            }
        }(w)
    }

    // Send 6 jobs
    for j := 1; j <= 6; j++ {
        jobs <- j
    }
    close(jobs)

    // Close results when all workers done
    go func() {
        wg.Wait()
        close(results)
    }()

    // Fan-in: collect all results
    for r := range results {
        fmt.Println(r)
    }
}
worker 0 processed job 1 worker 1 processed job 2 worker 2 processed job 3 worker 0 processed job 4 worker 1 processed job 5 worker 2 processed job 6
Go
package main

import (
    "fmt"
    "time"
)

// Select with timeout and channel-based semaphore
func main() {
    ch := make(chan string, 1)

    go func() {
        time.Sleep(50 * time.Millisecond)
        ch <- "result"
    }()

    // Select with timeout
    select {
    case msg := <-ch:
        fmt.Println("Received:", msg)
    case <-time.After(100 * time.Millisecond):
        fmt.Println("Timeout!")
    }

    // Channel-based semaphore — limit to 2 concurrent tasks
    sem := make(chan struct{}, 2)
    for i := 0; i < 5; i++ {
        sem <- struct{}{} // acquire
        go func(id int) {
            defer func() { <-sem }() // release
            fmt.Printf("task %d running (concurrency limited to 2)\n", id)
            time.Sleep(10 * time.Millisecond)
        }(i)
    }
    // Wait for all to finish
    for i := 0; i < cap(sem); i++ {
        sem <- struct{}{}
    }
    fmt.Println("All tasks complete")
}
Received: result task 0 running (concurrency limited to 2) task 1 running (concurrency limited to 2) task 2 running (concurrency limited to 2) task 3 running (concurrency limited to 2) task 4 running (concurrency limited to 2) All tasks complete
Language Comparison — Concurrency Models:

Python: asyncio is single-threaded cooperative multitasking. The GIL prevents true parallelism for CPU-bound tasks. Use multiprocessing for parallelism. asyncio.gather() is the fan-out equivalent.

TypeScript: Promise.all() for fan-out, single-threaded event loop like Python. Promise.race() is the closest to Go select. Worker threads for CPU parallelism.

Rust: tokio provides an M:N async runtime similar to Go. tokio::select! mirrors Go select. mpsc channels mirror Go channels. Ownership ensures no data races at compile time.

C++: std::async and std::thread for OS-level threads. No built-in green threads or channels. std::future/std::promise for one-shot communication. Libraries like Boost.Asio for event loops.
Common Mistakes:
  • Goroutine leak: A goroutine blocked on a channel send/receive that will never complete leaks memory forever. Always ensure channels are closed or contexts are cancelled.
  • Closing a channel twice: Panics at runtime. Only the sender should close; close only once. Use sync.Once if multiple goroutines might close.
  • Range over unbuffered channel: for v := range ch blocks forever if the channel is never closed. Always close the channel when all sends are done.
  • Sending on a closed channel: Panics. Design your pipeline so that senders finish before closing.
  • Loop variable capture: Pre-Go 1.22, go func() { use(i) }() captured the loop variable by reference. Go 1.22+ fixed this, but be aware when reading older code.
Go Concurrency Rules:
  • Do not communicate by sharing memory; share memory by communicating (channels).
  • Every goroutine must have a clear exit path — use context.Context for cancellation.
  • Use sync.WaitGroup to wait for a known number of goroutines to finish.
  • Buffered channels for decoupling producer/consumer speed; unbuffered for synchronization.
  • Use select with time.After or context.Done() to avoid blocking forever.

Rust Deep Dive: Lifetimes & Borrowing Language Deep Dive
Lifetimes are Rust’s mechanism for ensuring references are always valid — no dangling pointers, no use-after-free, no garbage collector. Understanding lifetime annotations and the borrow checker is the single biggest hurdle for Rust newcomers, and the biggest payoff for systems programming.

Lifetime Syntax: 'a

A lifetime annotation 'a is not a concrete duration — it is a constraint telling the compiler “these references must all be valid for at least the same scope.” Functions that return references must annotate which input lifetimes the output borrows from.

Lifetime Elision Rules

Rust has three elision rules that let you omit lifetime annotations in common cases: (1) each reference parameter gets its own lifetime, (2) if there’s exactly one input lifetime, it is assigned to all output lifetimes, (3) if &self or &mut self is a parameter, its lifetime is assigned to all output lifetimes.

NLL — Non-Lexical Lifetimes

Since Rust 2018, the borrow checker tracks when a reference is last used, not when the variable goes out of scope. This makes many previously-rejected programs valid.

Rust
// --- Basic lifetime annotation ---
// Returns a reference to the longer of two string slices.
// Both inputs and the output must share the same lifetime 'a.
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() >= y.len() { x } else { y }
}

fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("xyz");
        result = longest(s1.as_str(), s2.as_str());
        // result is valid here because s2 is still alive
        println!("Longest: {}", result);
    }
    // result would NOT be valid here if it pointed to s2
    // The compiler prevents this via lifetime checking
}

        format!("{} {}", self.text, suffix) // returns owned String — no lifetime needed
    }

// --- NLL (Non-Lexical Lifetimes) in action ---
fn main() {
    let mut data = vec![1, 2, 3, 4, 5];

    // Immutable borrow
    let first = &data[0];
    println!("First element: {}", first);
    // 'first' is never used after this point — NLL ends the borrow here

    // Mutable borrow is now allowed because the immutable borrow is done
    data.push(6);
    println!("After push: {:?}", data);

    // --- Common borrow checker error (explained) ---
    let mut scores = vec![100, 200, 300];
    // This would fail WITHOUT NLL:
    let biggest = scores.iter().max().copied();
    // .iter() borrows scores immutably, but .copied() returns an owned value
    // so the borrow ends immediately
    scores.push(400);
    println!("Biggest was: {:?}, now scores = {:?}", biggest, scores);
}

Language Comparison — Memory Safety:

Python: Reference counting + cycle-detecting GC. No lifetime annotations needed. Objects live as long as at least one reference exists. Memory leaks possible with reference cycles that the cycle detector misses (C extensions).

Go: Tracing garbage collector. No lifetime annotations. The GC handles all memory — at the cost of GC pauses (though Go’s GC is very low-latency, typically <1ms).

TypeScript: V8’s generational GC. No lifetime annotations. Same trade-off as Go: convenience for GC pauses. Closures can accidentally keep large objects alive.

C++: RAII (Resource Acquisition Is Initialization) with unique_ptr/shared_ptr. No borrow checker — dangling pointers and use-after-free are possible at runtime. Sanitizers (ASan, MSan) catch issues in testing but not in production.
Common Mistakes:
  • Returning a reference to a local: fn bad() -> &str { let s = String::from("x"); &s } — s is dropped at the end of the function. Return String (owned) instead.
  • Confusing 'a with a concrete scope: 'a means “at least as long as”, not “exactly this scope.” The compiler picks the shortest valid lifetime.
  • Over-annotating: Adding lifetimes where elision already works. Trust the elision rules first; only annotate when the compiler asks.
  • Struct with references: If a struct holds &str instead of String, it must declare a lifetime parameter. For owned data, prefer String over &str in structs unless performance-critical.
  • Fighting the borrow checker: If you find yourself reaching for Rc<RefCell<T>> everywhere, reconsider your data model. Often the issue is a graph structure that should be index-based instead.
Rust Lifetime Rules:
  • Every reference has a lifetime. The compiler infers most of them via elision rules.
  • A function returning a reference must borrow from an input — you cannot return a reference to a local.
  • You can have either one &mut T or any number of &T at the same time — never both.
  • NLL means the borrow ends at the last use, not at the end of the scope.
  • When in doubt, use owned types (String, Vec) and clone. Optimize to references later.

C++ Deep Dive: Move Semantics & Smart Pointers Language Deep Dive
Move semantics (C++11) eliminated unnecessary copies, making C++ both safer and faster. Smart pointers replaced raw new/delete, eliminating entire categories of memory bugs. Together with RAII, they are the foundation of modern C++ resource management.

Rvalue References & std::move

An lvalue has an identity (a name, an address). An rvalue is a temporary. T&& binds to rvalues. std::move(x) casts x to an rvalue reference, enabling the move constructor/assignment to “steal” its resources instead of copying. After a move, the source is in a “valid but unspecified” state.

Move Constructor & Assignment

A move constructor Widget(Widget&& other) transfers ownership of other’s resources (heap memory, file handles) by copying pointers and nulling the source. This is O(1) vs O(n) for a copy.

Smart Pointers

unique_ptr: sole ownership, zero overhead, move-only. shared_ptr: reference-counted shared ownership. weak_ptr: non-owning observer of shared_ptr (breaks cycles). RAII: the destructor releases resources automatically when the object leaves scope.

C++
#include <iostream>
#include <string>
#include <utility>
using namespace std;

class Buffer {
    size_t size_;
    int* data_;
public:
    // Constructor
    explicit Buffer(size_t n) : size_(n), data_(new int[n]{}) {
        cout << "  Construct " << size_ << endl;
    }

    // Copy constructor (expensive — O(n))
    Buffer(const Buffer& other) : size_(other.size_), data_(new int[other.size_]) {
        copy(other.data_, other.data_ + size_, data_);
        cout << "  Copy " << size_ << endl;
    }

    // Move constructor (cheap — O(1))
    Buffer(Buffer&& other) noexcept : size_(other.size_), data_(other.data_) {
        other.size_ = 0;
        other.data_ = nullptr;
        cout << "  Move " << size_ << endl;
    }

    // Move assignment
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;
            size_ = other.size_; data_ = other.data_;
            other.size_ = 0; other.data_ = nullptr;
            cout << "  Move-assign " << size_ << endl;
        }
        return *this;
    }

    ~Buffer() { delete[] data_; }
    size_t size() const { return size_; }
};

int main() {
    cout << "--- Create ---" << endl;
    Buffer a(1000);

    cout << "--- Copy ---" << endl;
    Buffer b = a;  // copy constructor

    cout << "--- Move ---" << endl;
    Buffer c = move(a);  // move constructor — a is now empty

    cout << "a.size=" << a.size() << " b.size=" << b.size()
         << " c.size=" << c.size() << endl;
}
--- Create --- Construct 1000 --- Copy --- Copy 1000 --- Move --- Move 1000 a.size=0 b.size=1000 c.size=1000
C++
#include <iostream>
#include <memory>
#include <string>
using namespace std;

struct Resource {
    string name;
    Resource(string n) : name(move(n)) { cout << "  Create: " << name << endl; }
    ~Resource() { cout << "  Destroy: " << name << endl; }
};

int main() {
    cout << "--- unique_ptr (sole ownership) ---" << endl;
    {
        auto r1 = make_unique<Resource>("UniqueRes");
        cout << "  Using: " << r1->name << endl;
        // auto r2 = r1;  // COMPILE ERROR — unique_ptr is move-only
        auto r2 = move(r1);  // transfer ownership
        cout << "  r1 is null: " << (r1 == nullptr) << endl;
        cout << "  r2 has: " << r2->name << endl;
    } // r2 destroyed here, r1 was already null

    cout << "\n--- shared_ptr (reference counting) ---" << endl;
    {
        auto s1 = make_shared<Resource>("SharedRes");
        cout << "  refcount: " << s1.use_count() << endl;
        {
            auto s2 = s1;  // copy — increments refcount
            cout << "  refcount: " << s1.use_count() << endl;
        } // s2 destroyed, refcount decremented
        cout << "  refcount: " << s1.use_count() << endl;
    } // s1 destroyed, refcount hits 0, Resource destroyed

    cout << "\n--- weak_ptr (breaking cycles) ---" << endl;
    {
        auto shared = make_shared<Resource>("WeakTarget");
        weak_ptr<Resource> weak = shared;
        cout << "  weak expired? " << weak.expired() << endl;
        if (auto locked = weak.lock()) {
            cout << "  locked: " << locked->name << endl;
        }
        shared.reset();  // destroy the resource
        cout << "  weak expired? " << weak.expired() << endl;
    }
}

C++
#include <iostream>
#include <memory>
#include <vector>
#include <string>
using namespace std;

// RAII — Resource Acquisition Is Initialization
// The destructor guarantees cleanup, even on exceptions
class FileHandle {
    string path_;
    bool open_ = false;
public:
    explicit FileHandle(const string& path) : path_(path), open_(true) {
        cout << "  Opened: " << path_ << endl;
    }
    ~FileHandle() {
        if (open_) cout << "  Closed: " << path_ << endl;
    }
    // Move-only (no copies)
    FileHandle(const FileHandle&) = delete;
    FileHandle& operator=(const FileHandle&) = delete;
    FileHandle(FileHandle&& other) noexcept
        : path_(move(other.path_)), open_(other.open_) {
        other.open_ = false;
    }
    const string& path() const { return path_; }
};

int main() {
    // Unique ownership of file handles
    vector<unique_ptr<FileHandle>> handles;
    handles.push_back(make_unique<FileHandle>("/tmp/a.txt"));
    handles.push_back(make_unique<FileHandle>("/tmp/b.txt"));

    for (auto& h : handles) {
        cout << "  Using: " << h->path() << endl;
    }

    cout << "  Clearing vector..." << endl;
    handles.clear();  // destructors called automatically
    cout << "  Done" << endl;
}

Language Comparison — Ownership & Resource Management:

Python: Reference counting + GC. with statements (context managers) are Python’s RAII equivalent for deterministic cleanup. No move semantics — everything is a reference.

Go: GC handles memory. defer is Go’s RAII equivalent for cleanup. No move semantics; values are copied unless you use pointers.

TypeScript: V8 GC. No RAII; use try/finally or using (TC39 Stage 3) for cleanup. No move semantics.

Rust: Ownership system is the direct successor to C++ RAII + move semantics. Box = unique_ptr, Rc = shared_ptr, Weak = weak_ptr. But Rust enforces these at compile time, while C++ relies on programmer discipline.
Common Mistakes:
  • Using after move: auto b = move(a); a.doStuff(); — undefined behavior. The compiler does not prevent this (unlike Rust).
  • Raw new/delete: Never use raw pointers for ownership in modern C++. Use make_unique or make_shared.
  • shared_ptr cycles: Two shared_ptrs pointing at each other leak memory. Use weak_ptr to break cycles (e.g., parent-child relationships).
  • Forgetting noexcept: std::vector only uses move constructors if they are noexcept — otherwise it falls back to copying for exception safety.
  • std::move does not move: It is just a cast to T&&. The actual move happens in the move constructor/assignment. Moving a const object silently copies instead.
Modern C++ Memory Rules:
  • Use unique_ptr by default for ownership. Use shared_ptr only when ownership is genuinely shared.
  • Never use raw new/delete. Use make_unique and make_shared.
  • Mark move constructors and move assignment operators noexcept.
  • After std::move(x), treat x as empty — do not read from it.
  • Follow the Rule of Five: if you define any of destructor, copy constructor, copy assignment, move constructor, move assignment — define all five (or = delete the ones you do not want).
