Assertion Patterns & Custom Matchers Testing & Error Handling
Tests need clear, descriptive assertions to pinpoint failures quickly. Each language has rich assertion libraries beyond basic assertEquals — knowing them means faster debugging and more maintainable tests.

Every language provides basic equality assertions, but real-world tests need more: checking exceptions, approximate floating-point comparisons, collection containment, and table-driven parameterized patterns. Mastering these patterns means your tests document intent and failures point directly at the problem.

Language Comparison — Parameterized Tests: Python’s @pytest.mark.parametrize is a decorator that generates separate test IDs automatically. Go uses hand-built table-driven tests with t.Run subtests — the most idiomatic Go testing pattern. TypeScript borrows Jest’s it.each pattern with template literal interpolation in test names. Rust has no built-in parametrize — you loop within a single #[test] function (or use the test-case crate). C++’s Google Test provides TEST_P with INSTANTIATE_TEST_SUITE_P, the most verbose but most flexible approach.
Common Mistakes:
  • Python: Using bare assert without pytest — you lose the rich diff output. Pytest rewrites assertions to show both sides on failure.
  • Go: Using t.Error when you mean t.Fatal — Error continues the test, Fatal stops immediately. A nil pointer after a failed check will panic if you used Error.
  • TypeScript: Forgetting toEqual vs toBe — toBe uses === (reference equality), toEqual does deep comparison. Arrays and objects need toEqual.
  • Rust: assert_eq! requires Debug trait — your type must derive Debug or the assertion won’t compile.
  • C++: ASSERT_* aborts the test on failure, EXPECT_* continues — use ASSERT for preconditions and EXPECT for the actual checks.
Key Rules:
  • One assertion per logical check — multiple assertions in one test are fine if they verify one behavior. Split different behaviors into separate tests.
  • Descriptive test names — a failing test name should tell you what broke without reading the code.
  • Test edge cases first — empty input, single element, duplicates, negative numbers, and boundary values.

Type Systems Deep Dive Testing & Error Handling
Understanding type system differences helps you write idiomatic code in each language and leverage type safety where available. The same concept can be valid in one language and a compile error in another due to structural vs nominal typing.

Languages take fundamentally different approaches to types: Python uses gradual typing (optional type hints checked by mypy), Go and TypeScript use structural typing (shape matters, not name), Rust and C++ use nominal typing (explicit declarations required). These differences affect how you design interfaces, write generic code, and think about compatibility.

Language Comparison — Type Systems:
Structural typing (Go, TS, Python Protocol): a type is compatible if it has the right methods/properties. You never write implements. Benefit: flexible, decoupled. Risk: accidental matches (a PizzaSlice accidentally satisfying a Shape interface).

Nominal typing (Rust, C++): a type is compatible only if it explicitly declares the relationship (impl Trait for Type, class Foo : public Base). Benefit: intentional, no accidental matches. Cost: more boilerplate.

Generics work differently too: Go and Rust monomorphize (generate specialized code per type at compile time). TypeScript erases types at compile time (no runtime generics). C++ templates are duck-typed at compile time — concepts (C++20) add explicit constraints. Python type hints are optional annotations checked by external tools (mypy), not by the runtime.
Common Mistakes:
  • Python: Assuming type hints are enforced at runtime — they are not. Use mypy --strict for static checking. @runtime_checkable Protocol only checks method existence, not signatures.
  • Go: Overusing interface{}/any — this defeats the type system. Use specific interfaces or generics with constraints.
  • TypeScript: Excess property checks only apply to object literals, not variables — let x: {a: number} = {a: 1, b: 2} fails, but assigning via a variable succeeds.
  • Rust: Forgetting to add trait bounds on generic functions — fn foo<T>(a: T) can do almost nothing with a without bounds.
  • C++: Pre-C++20 templates give cryptic error messages deep in template instantiation. Use concepts for clear errors.

Async/Await Deep Dive Testing & Error Handling
Modern applications need concurrent I/O. Each language takes a radically different approach to async — from Python’s single-threaded event loop to Go’s preemptive green threads to Rust’s zero-cost futures. Knowing the model prevents subtle concurrency bugs.

Key difference: Go does NOT have async/await — goroutines look synchronous but run on a scheduler. Python and TypeScript use single-threaded event loops. Rust requires an external executor (tokio). C++ has multiple competing approaches (std::async, coroutines).

Language Comparison — Async Models:
Python: Single-threaded event loop (asyncio). await yields control back to the event loop. Cannot do CPU-bound work in async code without run_in_executor.

Go: NO async/await. Goroutines are preemptive green threads managed by the Go runtime scheduler. Code looks synchronous. Channels and context handle coordination and cancellation.

TypeScript: Single-threaded event loop (like Python). Promise is the core abstraction. async/await is syntactic sugar over .then() chains. Promise.all for concurrency, Promise.race for first-wins.

Rust: Zero-cost futures that do nothing until polled. Requires an executor (tokio, async-std). .await is a suspension point. tokio::select! for racing, tokio::join! for fan-out. Dropping a future cancels it.

C++: Multiple approaches: std::async/std::future (thread-based), C++20 coroutines (co_await), or libraries like boost::asio. No built-in cancellation — threads run to completion.
Common Mistakes:
  • Python: Calling an async function without await — you get a coroutine object, not the result. Also: blocking calls (time.sleep) inside async code freeze the entire event loop.
  • Go: Goroutine leaks — launching goroutines that block forever on a channel nobody writes to. Always ensure goroutines can exit (use context or done channels).
  • TypeScript: Unhandled promise rejections — forgetting .catch() or try/catch around await. Node.js will terminate on unhandled rejections.
  • Rust: Holding a MutexGuard across an .await — the guard is !Send, causing compile errors with tokio::spawn. Use tokio::sync::Mutex instead of std::sync::Mutex in async code.
  • C++: std::async with launch::deferred is NOT concurrent — it runs lazily when you call .get(). Always specify launch::async for true concurrency.
Key Rules:
  • Never block the event loop (Python, TS) — CPU-heavy work must go to a thread pool or worker.
  • Always handle cancellation — use context.Context (Go), AbortController (TS), CancellationToken, or drop semantics (Rust).
  • Fan-out then fan-in — spawn concurrent tasks, then collect all results. This is the most common async pattern.
