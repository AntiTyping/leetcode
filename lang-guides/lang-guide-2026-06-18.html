Logging Across Languages Testing & Error Handling 8 days since review · rep 3
Each language has an idiomatic structured-logging story: Python’s logging module with handlers/formatters, Go’s log/slog with structured key-value pairs, TypeScript’s winston/pino with transports, Rust’s tracing crate with spans and subscribers, and C++’s spdlog with sinks and patterns. Prefer key-value pairs over interpolated strings so logs stay machine-parseable.

import logging

logger.info("user login", extra={"user_id": 42, "ip": "10.0.0.1"})
logger.warning("slow query", extra={"ms": 812})

Type Systems Deep Dive Testing & Error Handling 8 days since review · rep 3
Languages differ in type-system philosophy: Python uses gradual/structural typing (Protocol, TypeVar), Go uses structural interfaces (implicit satisfaction), TypeScript uses structural typing with rich inference, Rust uses nominal typing with traits and monomorphization, and C++ uses nominal typing with templates (duck-typed at instantiation, constrained by concepts).
Python
from typing import Protocol

class Sized(Protocol):
    def __len__(self) -> int: ...

def total(items: Sized) -> int:
    return len(items)

print(total([1, 2, 3]))   # list satisfies Sized
print(total("hello"))      # str satisfies Sized
3 5
Go
package main

import "fmt"

type Sizer interface{ Size() int }

type Box struct{ n int }
func (b Box) Size() int { return b.n }   // implicit satisfaction

func report(s Sizer) { fmt.Println("size:", s.Size()) }

func main() { report(Box{n: 7}) }
size: 7

fn largest<T: PartialOrd + Copy>(list: &[T]) -> T {
    let mut best = list[0];
    for &x in &list[1..] {
        if x > best { best = x; }
    }
    best
}

fn main() {
    println!("{}", largest(&[3, 9, 2, 7]));
    println!("{}", largest(&[1.5, 0.2, 4.8]));

Go: Goroutine Scheduling & Channel Patterns Language Deep Dive 8 days since review · rep 3
Go’s runtime multiplexes goroutines onto OS threads using an M:N scheduler (the G-M-P model). Goroutines start with a ~2KB stack that grows dynamically, and the scheduler uses work-stealing to balance load. Channels provide typed, synchronized communication; unbuffered channels block on both send and receive until the other side is ready.
Python (asyncio.Queue equivalent)
import asyncio

async def main():
    q: asyncio.Queue[int] = asyncio.Queue(maxsize=10)
    async def producer():
        for i in range(3):
            await q.put(i)
        await q.put(-1)  # sentinel
    async def consumer():
        while (v := await q.get()) != -1:
            print("got", v)
    await asyncio.gather(producer(), consumer())

asyncio.run(main())
got 0 got 1 got 2
Go
package main

import "fmt"

func main() {
    ch := make(chan int, 10) // buffered
    go func() {
        for i := 0; i < 3; i++ {
            ch <- i
        }
        close(ch)
    }()
    for v := range ch {
        fmt.Println("got", v)
    }
}
got 0 got 1 got 2
TypeScript (MessageChannel)
const { port1, port2 } = new MessageChannel();

port1.postMessage(null);
got 0 got 1 got 2

C++ (jthread + queue)
#include <iostream>
#include <queue>
#include <mutex>
#include <condition_variable>
#include <thread>

int main() {
    std::queue<int> q; std::mutex m; std::condition_variable cv; bool done=false;
    std::jthread prod([&]{
        for (int i=0;i<3;i++){ {std::lock_guard l(m); q.push(i);} cv.notify_one(); }
        {std::lock_guard l(m); done=true;} cv.notify_one();
    });
    std::unique_lock l(m);
    while (true) {
        cv.wait(l, [&]{ return !q.empty() || done; });
        while (!q.empty()) { std::cout << "got " << q.front() << "\n"; q.pop(); }
        if (done) break;
    }
}

C++: Move Semantics & Smart Pointers Language Deep Dive 8 days since review · rep 3
Move semantics enable efficient resource transfer without copying. std::move casts to an rvalue reference (&&), enabling move constructors/assignment. Smart pointers (unique_ptr, shared_ptr, weak_ptr) automate memory management via RAII: unique_ptr has exclusive ownership, shared_ptr uses reference counting.
Python (GC, no explicit move)
class Widget:
    def __init__(self, name): self.name = name

a = Widget("w1")
b = a              # rebinds reference; both point to same object
print(a is b, b.name)   # GC frees when refcount hits 0
True w1

Python: Dataclasses, __post_init__ & Slots Language Deep Dive 7 days since review · rep 3
@dataclass auto-generates __init__, __repr__, and __eq__. Use __post_init__ for validation/computed fields. slots=True (3.10+) prevents __dict__, reducing memory ~40% and speeding attribute access. frozen=True makes instances immutable, and field(default_factory=list) handles mutable defaults safely.
Python
from dataclasses import dataclass

@dataclass(frozen=True, slots=True)
class Point:
    x: float
    y: float
    def __post_init__(self):
        if self.x < 0 or self.y < 0:
            raise ValueError("coords must be non-negative")

p = Point(3.0, 4.0)
print(p)
# p.x = 9  # FrozenInstanceError
Point(x=3.0, y=4.0)

Go: Channel Patterns & Pipeline Language Deep Dive 7 days since review · rep 3
Go channels enable pipeline patterns: stage functions take input channels and return output channels, each running in its own goroutine. Key building blocks: a generator (produces values), fan-out (multiple goroutines reading one channel), fan-in (merging multiple channels into one), and a done channel for cancellation.
Python (async pipeline)
import asyncio

async def gen(*nums):
    for n in nums: yield n

async def sq(src):
    async for n in src: yield n * n

async def main():
    async for v in sq(gen(1, 2, 3)):
        print(v)

asyncio.run(main())
1 4 9
Go
package main

import "fmt"

func gen(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums { out <- n }
    }()
    return out
}

func sq(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in { out <- n * n }
    }()
    return out
}

func main() {
    for v := range sq(gen(1, 2, 3)) {
        fmt.Println(v)
    }
}
1 4 9
TypeScript (async generators)

🆕 New Concepts — Production Patterns

Three patterns every production service needs: observability, health probes, and rate limiting. Read each carefully, then attempt the exercises in all five languages.

1. Structured Logging & Observability Production Patterns
In production you need more than print statements. Structured logging with key-value pairs enables log aggregation, search, and alerting. Combined with metrics and tracing, it forms the three pillars of observability.

Structured vs unstructured. Unstructured logging emits free-form strings ("user 42 logged in from 10.0.0.1") that are hard to query at scale. Structured logging emits key-value pairs (level=info user_id=42 ip=10.0.0.1 msg="login") or JSON, so a log aggregator (Loki, Elasticsearch, Datadog) can filter and alert on fields.

The three pillars

Logs answer “what happened?”, metrics answer “how much/how many?” (counters, gauges, histograms), and traces answer “where did the time go?” across services (trace ID, span ID, parent span). A request flows through all three: it emits logs, increments metrics, and produces a trace.

Log levels and when to use each

DEBUG (developer detail), INFO (normal lifecycle events), WARN (recoverable anomaly), ERROR (operation failed), FATAL/CRITICAL (process must stop). Set the threshold per environment: DEBUG locally, INFO or WARN in production.

Contextual logging

Attach request IDs, user IDs, and correlation IDs to every log line within a request lifecycle. This is usually done by binding fields to a child logger and propagating it through the call stack (or via context/thread-local storage).

Common Mistakes
  • WRONG: logging sensitive data (passwords, tokens, full credit-card numbers) — logs are often stored unencrypted and widely accessible.
  • WRONG: logging at the wrong level (INFO for debug-only chatter) — it drowns out real signals and balloons cost.
  • WRONG: omitting correlation IDs, so you cannot stitch a single request across services.
  • WRONG: excessive logging in hot paths, adding measurable latency and I/O pressure.
Key Rules
  • Always use structured logging in production — key-value pairs or JSON, never bare strings
  • Include request context (request ID, user ID) in every log line within a request
  • Use log levels consistently across services so dashboards and alerts behave predictably
  • Expose metrics (counters/gauges/histograms) and propagate trace context for full observability
Language Comparison: Logging Ecosystems
  • Python: stdlib logging + python-json-logger; prometheus_client; opentelemetry.
  • Go: stdlib log/slog (structured, zero-dependency); promhttp; go.opentelemetry.io/otel.
  • TypeScript: pino/winston; prom-client; @opentelemetry/sdk-node.
  • Rust: tracing + tracing-subscriber (spans are first-class); prometheus crate; opentelemetry.
  • C++: spdlog (sinks/patterns); prometheus-cpp; opentelemetry-cpp.

2. Health Checks & Readiness Probes Production Patterns
Kubernetes and load balancers need to know if your service is alive (liveness) and ready to serve traffic (readiness). A liveness check verifies the process isn’t deadlocked; a readiness check verifies dependencies (DB, cache, external APIs) are reachable.

Liveness vs readiness vs startup. A liveness probe answers “is the process alive?” — if it fails, the orchestrator restarts the pod. A readiness probe answers “can it serve traffic right now?” — if it fails, the pod is removed from the load-balancer rotation but not killed. A startup probe gives slow-booting apps extra grace before liveness kicks in.

Endpoint patterns

Conventionally /live (or /health) for liveness, /ready for readiness. Liveness must be cheap — it should NOT touch the database. Readiness checks dependencies and can return a per-component status array.

Dependency checking & graceful degradation

Readiness pings critical dependencies (DB, cache). Run checks concurrently with a timeout so one slow dependency cannot hang the probe. For non-critical dependencies, degrade gracefully: report degraded rather than failing readiness entirely.

Response format

A common shape: { "status": "ok", "version": "1.4.2", "checks": [{"name": "db", "status": "ok"}, ...] }. Return 200 when ready, 503 when not.

Python (FastAPI-style)
import asyncio, time

async def check_db() -> bool:
    await asyncio.sleep(0)   # simulate db.ping()
    return True

async def liveness() -> dict:
    return {"status": "ok"}          # cheap, no deps

async def readiness() -> dict:
    checks = {}
    try:
        ok = await asyncio.wait_for(check_db(), timeout=1.0)
        checks["db"] = "ok" if ok else "fail"
    except asyncio.TimeoutError:
        checks["db"] = "timeout"
    status = "ok" if all(v == "ok" for v in checks.values()) else "fail"

Common Mistakes
  • WRONG: a health check that does too much work (querying the database on every liveness call) — it adds load and can falsely fail.
  • WRONG: a liveness check that verifies dependencies — a DB blip then restarts a perfectly healthy pod (use readiness for deps).
  • WRONG: not separating liveness from readiness, so the orchestrator cannot distinguish “restart me” from “hold traffic.”
  • WRONG: a probe that can hang — without a timeout, Kubernetes treats it as failed and may kill the pod.
Key Rules
  • Liveness = is the process alive (cheap, no dependencies). Readiness = can it serve traffic (checks deps)
  • Keep liveness checks cheap and deterministic
  • Run dependency checks concurrently with explicit timeouts; return partial results if some are slow
  • Return 200 when ready, 503 when not, with a per-component status array

3. Rate Limiting & Throttling Production Patterns
Rate limiting protects services from abuse and overload. Token bucket and sliding window are the most common algorithms. Limits can be applied per-client (API key), per-IP, or globally.

Token bucket

A bucket holds up to B tokens and refills at a fixed rate R tokens/second. Each request consumes one token; if the bucket is empty, the request is rejected. This allows bursts up to B while enforcing a long-run average of R — ideal for smooth limiting.

Sliding window counter

Counts requests within a rolling time window. A fixed window resets at boundaries, which allows a 2× burst straddling the boundary; a sliding window weights the previous window’s count to smooth this out, enforcing a stricter per-interval limit.

Scope & headers

Limits can be per-client, per-endpoint, or global. Always communicate state via headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset, and Retry-After on a 429. In distributed setups, store counters in Redis so all instances share the limit.

Python (token bucket)
import time

class TokenBucket:
    def __init__(self, rate: float, capacity: int):
        self.rate = rate          # tokens per second
        self.capacity = capacity
        self.tokens = float(capacity)
        self.last = time.monotonic()

    def allow(self) -> bool:
        now = time.monotonic()
        self.tokens = min(self.capacity, self.tokens + (now - self.last) * self.rate)
        self.last = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

tb = TokenBucket(rate=2, capacity=3)
print([tb.allow() for _ in range(5)])   # burst of 3, then deny
[True, True, True, False, False]

🆕 New Concepts — API Design

Two concepts for designing predictable, maintainable HTTP APIs: resource-oriented REST design and consistent error handling with proper status codes.

4. REST API Design Patterns API Design
Most backend services expose REST APIs. Good REST design means predictable URLs, proper HTTP methods, consistent response formats, and good pagination. Poor API design creates confusion and maintenance burden.

Resource naming

Use nouns, plural, and nest sub-resources: /users, /users/123, /users/123/orders. Avoid verbs in URLs — the HTTP method is the verb.

HTTP method semantics

GET reads (safe, idempotent), POST creates (not idempotent), PUT replaces (idempotent), PATCH partially updates, DELETE removes (idempotent). Idempotency matters for safe client retries.

Filtering, sorting, pagination

Use query parameters: ?status=active&sort=-created_at. Offset-based pagination (?page=2&limit=20) is simple but drifts when data changes. Cursor-based (?cursor=abc123) is stable for large, changing collections.

Versioning & HATEOAS

Version via URL path (/v1/users) or header (Accept: application/vnd.api.v1+json). HATEOAS adds navigational links in responses ("links": { "next": "...", "self": "..." }).

5. Error Responses & Status Codes API Design
Consistent error responses help API consumers handle failures gracefully. Proper HTTP status codes enable caching, retries, and monitoring. A well-structured error body tells the client exactly what went wrong and how to fix it.

Status code families

2xx success, 3xx redirect, 4xx client error, 5xx server error. Common codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests, 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable.

Error envelope & field-level details

Use a consistent shape: { "error": { "code": "VALIDATION_ERROR", "message": "...", "details": [{"field": "email", "message": "invalid format"}] } }. Field-level details let clients highlight the offending input.

Problem Details (RFC 7807/9457)

A standardized error body with type (URI identifying the problem), title, status, detail, and instance. Served with Content-Type: application/problem+json. Using a standard format means clients and gateways can parse errors uniformly.

Error middleware

A central handler catches unhandled exceptions, logs the full error server-side (with stack trace), and returns a safe 500 without leaking internals.

3 Build error middleware that catches unhandled exceptions, logs the full error server-side, and returns a safe 500 response to the client.
Builds on: REST API Design (status codes per method), Type Systems (typed error enums). Leads to: Structured Logging (log the real error before returning a safe one), Rate Limiting (429 envelope with Retry-After).
