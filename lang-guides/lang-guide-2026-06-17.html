Dependency Injection Design Patterns 7 days overdue · rep 2
Dependency Injection decouples components by injecting dependencies rather than letting components create them internally. This enables testability (swap real DB for mock), flexibility (swap implementations), and clear dependency graphs. Constructor injection is the most common and recommended form.
Python

class EmailSender(ABC):
    @abstractmethod
    def send(self, to: str, body: str) -> None: ...

class SmtpSender(EmailSender):
    def send(self, to: str, body: str) -> None:
        print(f"SMTP: sending to {to}: {body}")

class MockSender(EmailSender):
    def __init__(self):
        self.sent: list[tuple[str, str]] = []

Event Systems & Reactive Patterns Architecture 7 days overdue · rep 2
Observer/pub-sub patterns enable decoupled communication between components. Publishers emit events without knowing who listens; subscribers react without knowing who published. This is the backbone of UI frameworks, message queues, and microservice architectures.
Python
from collections import defaultdict
from typing import Callable

class EventBus:
    def __init__(self):
        self._handlers: dict[str, list[Callable]] = defaultdict(list)

    def on(self, event: str, handler: Callable) -> None:
        self._handlers[event].append(handler)

    def emit(self, event: str, **data) -> None:
        for handler in self._handlers[event]:
            handler(**data)

bus = EventBus()
bus.on("user:created", lambda name, **_: print(f"Welcome email to {name}"))
bus.on("user:created", lambda name, **_: print(f"Audit log: {name} registered"))
bus.emit("user:created", name="Alice")

package main

import "fmt"

type Handler func(data map[string]string)

type EventBus struct {
    handlers map[string][]Handler
}

func NewEventBus() *EventBus {
    return &EventBus{handlers: make(map[string][]Handler)}

func (eb *EventBus) On(event string, h Handler) {
    eb.handlers[event] = append(eb.handlers[event], h)
}

func (eb *EventBus) Emit(event string, data map[string]string) {
    for _, h := range eb.handlers[event] {

type Handler = (data: Record<string, string>) => void;

class EventBus {
  private handlers = new Map<string, Handler[]>();

  on(event: string, handler: Handler): void {
    const list = this.handlers.get(event) ?? [];
    list.push(handler);
    this.handlers.set(event, list);
  }

  emit(event: string, data: Record<string, string>): void {
    for (const h of this.handlers.get(event) ?? []) h(data);
  }
}

use std::collections::HashMap;

type Handler = Box<dyn Fn(&HashMap<&str, &str>)>;

struct EventBus {
    handlers: HashMap<String, Vec<Handler>>,
}

impl EventBus {
    fn new() -> Self { Self { handlers: HashMap::new() } }

    fn on(&mut self, event: &str, handler: Handler) {
        self.handlers.entry(event.to_string()).or_default().push(handler);
    }

    fn emit(&self, event: &str, data: &HashMap<&str, &str>) {
        if let Some(handlers) = self.handlers.get(event) {
            for h in handlers { h(data); }
        }
    }
}

fn main() {
    let mut bus = EventBus::new();
    bus.on("user:created", Box::new(|d| {
        println!("Welcome email to {}", d["name"]);
    }));
    bus.on("user:created", Box::new(|d| {
        println!("Audit log: {} registered", d["name"]);
    }));
    let mut data = HashMap::new();
    data.insert("name", "Alice");
    bus.emit("user:created", &data);
}

Concurrency Primitives Concurrency 6 days overdue · rep 2
Concurrency primitives are the building blocks for parallel and concurrent execution: goroutines and channels (Go), threads and Arc<Mutex<T>> (Rust), threading/asyncio (Python), Promise.all and workers (TypeScript), std::thread and std::mutex (C++). Choose message passing over shared state when possible.
Python
import threading

counter = 0
lock = threading.Lock()

def increment(n: int) -> None:
    global counter
    for _ in range(n):
        with lock:
            counter += 1

threads = [threading.Thread(target=increment, args=(100_000,)) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Counter: {counter}")  # exactly 400000 with lock

Generic Programming Type Systems 6 days overdue · rep 2
Generics let you write functions and data structures that work with any type while preserving type safety. Python: TypeVar and Generic[T]. Go: type parameters with constraints [T comparable]. TypeScript: <T extends Constraint>. Rust: <T: Trait> with trait bounds. C++: templates with concepts (C++20).
Python
from typing import TypeVar, Generic, Optional

T = TypeVar("T")

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> Optional[T]:
        return self._items.pop() if self._items else None

    def __len__(self) -> int:
        return len(self._items)

s: Stack[int] = Stack()
s.push(1); s.push(2); s.push(3)
print(f"Popped: {s.pop()}, size: {len(s)}")
Popped: 3, size: 2
Go
package main

import "fmt"

type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(item T) { s.items = append(s.items, item) }
func (s *Stack[T]) Pop() (T, bool) {
    if len(s.items) == 0 { var zero T; return zero, false }
    item := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return item, true
}
func (s *Stack[T]) Len() int { return len(s.items) }

func main() {
    s := &Stack[int]{}
    s.Push(1); s.Push(2); s.Push(3)
    v, _ := s.Pop()
    fmt.Printf("Popped: %d, size: %d\n", v, s.Len())
}

C++
#include <iostream>
#include <vector>
#include <optional>

template <typename T>
class Stack {
    std::vector<T> items_;
public:
    void push(T item) { items_.push_back(std::move(item)); }
    std::optional<T> pop() {
        if (items_.empty()) return std::nullopt;
        T val = std::move(items_.back());
        items_.pop_back();
        return val;
    }
    size_t size() const { return items_.size(); }
};

int main() {
    Stack<int> s;
    s.push(1); s.push(2); s.push(3);
    auto v = s.pop();
    std::cout << "Popped: " << *v << ", size: " << s.size() << "\n";
    return 0;
}

Middleware & Request Lifecycle Production Patterns Web

Every web application needs cross-cutting concerns — logging, authentication, CORS, rate limiting, request tracing. Middleware lets you compose these concerns as independent layers without polluting your business logic. Understanding the request lifecycle (how a request flows through a middleware chain) is essential for debugging and performance tuning.

Middleware is a function that wraps a handler, executing code before and/or after the inner handler runs. Middleware chains are composed by nesting wrappers — the outermost middleware runs first on the request and last on the response, creating an “onion” pattern.

Key Concepts

Request flow: Client → Middleware A (before) → Middleware B (before) → Handler → Middleware B (after) → Middleware A (after) → Client

Common middleware: Logging, authentication, CORS, compression, rate limiting, request ID injection, error recovery/panic handling.

Language Comparison: Middleware Patterns
  • Python: Function decorators with closures. Frameworks like Flask use @app.before_request; FastAPI uses dependency injection.
  • Go: func(http.Handler) http.Handler is the idiomatic pattern. The standard library supports this natively — no framework needed.
  • TypeScript: Express uses (req, res, next) => void. NestJS uses class-based interceptors and guards with decorators.
  • Rust: Tower’s Layer and Service traits are the standard. Actix-web uses Transform trait for middleware.
  • C++: No standard middleware pattern. Function composition with std::function or framework-specific (Crow, Drogon).
Common Mistakes
  • WRONG: Writing to the response body before calling the next handler, then trying to set headers after. 
    // Go: Headers must be set BEFORE WriteHeader/Write

    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        next.ServeHTTP(w, r)        // handler writes body
        w.Header().Set("X-ID", id)  // TOO LATE! Headers already sent
    })
}
  • RIGHT: Set response headers before calling the inner handler, or use a response wrapper. 
    func good(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("X-ID", id)  // Set BEFORE handler writes
        next.ServeHTTP(w, r)
    })
}
  • WRONG: Not calling the next handler, silently swallowing the request.
  • WRONG: Middleware ordering that puts auth after logging — unauthenticated requests hit business logic before being rejected.
Key Rules
  • Always call the next handler unless you intentionally short-circuit (auth failure, rate limit)
  • Set response headers before calling the inner handler or use a response wrapper
  • Order matters: recovery → request ID → logging → auth → rate limit → handler
  • Keep middleware small and focused — each should do exactly one thing
  • Use context (Go), request attributes (Python), or locals (Express) to pass data between middleware layers

Caching Patterns Production Patterns Performance

Caching avoids repeated expensive computation or I/O by storing results for reuse. An LRU (Least Recently Used) cache evicts the oldest-accessed item when full. TTL (Time To Live) caches expire entries after a fixed duration. Memoization is function-level caching of return values keyed by arguments. Getting caching right is critical — stale data and unbounded caches are the #1 and #2 production incidents.

Cache Hierarchy

Memoization: Simplest form — cache function results by arguments. Good for pure functions.

LRU Cache: Bounded size, evicts least-recently-used entries. Prevents memory leaks.

TTL Cache: Time-based expiration. Ensures freshness at the cost of occasional recomputation.

LRU + TTL: Production standard — bounded size AND time-based expiration.

Example 2: LRU Cache with Bounded Size

An LRU cache has a maximum capacity. When full, it evicts the least-recently-used entry. This prevents unbounded memory growth. The classic implementation uses a hash map + doubly linked list for O(1) get/put.

Python
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self._cache: OrderedDict[str, str] = OrderedDict()
        self._capacity = capacity
        self.hits = 0
        self.misses = 0

    def get(self, key: str) -> str | None:
        if key in self._cache:
            self.hits += 1
            self._cache.move_to_end(key)  # mark as recently used
            return self._cache[key]
        self.misses += 1
        return None

    def put(self, key: str, value: str) -> None:
        if key in self._cache:
            self._cache.move_to_end(key)
        self._cache[key] = value
        if len(self._cache) > self._capacity:
            evicted = self._cache.popitem(last=False)  # remove oldest
            print(f"  Evicted: {evicted[0]}")

    def __len__(self) -> int:
        return len(self._cache)

        c.order.MoveToFront(el)
        return el.Value.(*entry).value, true
    }
    c.misses++
    return "", false

    if el, ok := c.items[key]; ok {
        c.order.MoveToFront(el)
        el.Value.(*entry).value = value
        return

    el := c.order.PushFront(&entry{key, value})
    c.items[key] = el

        oldest := c.order.Back()

        evicted := oldest.Value.(*entry)
        delete(c.items, evicted.key)

    misses: usize,

            self.order.push_back(key.to_string());

    std::unordered_map<std::string, decltype(order_)::iterator> map_;
    int hits_ = 0, misses_ = 0;

public:
    explicit LRUCache(int cap) : capacity_(cap) {}

    std::optional<std::string> get(const std::string& key) {
        auto it = map_.find(key);
        if (it == map_.end()) { misses_++; return std::nullopt; }
        hits_++;
        order_.splice(order_.begin(), order_, it->second);  // move to front
        return it->second->second;
    }

    void put(const std::string& key, const std::string& value) {
        auto it = map_.find(key);
        if (it != map_.end()) {
            order_.erase(it->second);
            map_.erase(it);
        }
        order_.push_front({key, value});
        map_[key] = order_.begin();
        if (static_cast<int>(map_.size()) > capacity_) {
            auto& back = order_.back();
            std::cout << "  Evicted: " << back.first << "\n";
            map_.erase(back.first);
            order_.pop_back();
        }
    }

    int size() const { return map_.size(); }
    int hits() const { return hits_; }
    int misses() const { return misses_; }
};

Example 3: TTL (Time-To-Live) Cache

TTL caches automatically expire entries after a configured duration. This ensures data freshness and is essential for caching API responses, DNS lookups, or user sessions.

import time

class TTLCache:
    def __init__(self, ttl_seconds: float):
        self._cache: dict[str, tuple[float, str]] = {}
        self._ttl = ttl_seconds

        if key in self._cache:

            print(f"  Expired: {key}")
        return None

    def put(self, key: str, value: str) -> None:
        self._cache[key] = (time.monotonic() + self._ttl, value)

Language Comparison: Caching
  • Python: functools.lru_cache is built-in and production-ready. For TTL, use cachetools.TTLCache.
  • Go: No built-in LRU; use container/list + map or github.com/hashicorp/golang-lru.
  • TypeScript: JavaScript Map preserves insertion order, enabling simple LRU. For production, use lru-cache npm package.
  • Rust: No built-in LRU cache; lru crate is the standard choice. moka crate adds TTL and async support.
  • C++: No built-in cache. Use std::list + std::unordered_map for O(1) LRU. Boost.Cache exists but is niche.
Common Mistakes
  • WRONG: Unbounded cache without eviction — memory grows until OOM. 
    # Python: unbounded dict cache
cache = {}
def fetch(key):
    if key not in cache:
        cache[key] = expensive_lookup(key)  # never evicted!
    return cache[key]
  • RIGHT: Use a bounded cache with LRU or TTL eviction. 
    from functools import lru_cache

@lru_cache(maxsize=1000)  # bounded!
def fetch(key: str) -> str:
    return expensive_lookup(key)
  • WRONG: Caching mutable objects. If the cached value is modified, all callers see the mutation.
  • WRONG: Not handling cache stampede — when a popular key expires, hundreds of concurrent requests all compute the same expensive value simultaneously.
Key Rules
  • Always set a maximum size for caches — unbounded caches are memory leaks
  • Use TTL for data that can become stale (API responses, user sessions)
  • Cache immutable values, or return deep copies of mutable cached objects
  • Monitor cache hit ratio — below 80% suggests the cache is not providing value
  • Consider cache stampede protection: lock-on-miss, stale-while-revalidate, or probabilistic early expiration

Graceful Shutdown & Signal Handling Production Patterns Operations

Production services must handle OS signals (SIGTERM, SIGINT) to shut down cleanly: stop accepting new requests, drain in-flight work, flush buffers, close connections, and release resources. Without graceful shutdown, deployments cause dropped requests, corrupted data, and orphaned resources. Kubernetes and container orchestrators send SIGTERM before killing a pod — your service has a limited window to clean up.

The Shutdown Sequence

1. Signal received (SIGTERM/SIGINT) → 2. Stop accepting new work → 3. Drain in-flight requests (with timeout) → 4. Flush buffers (logs, metrics) → 5. Close connections (DB, cache) → 6. Exit cleanly

When shutdown is triggered, we must finish processing items already in the queue before exiting. New items are rejected.

        self._accepting = True

        if not self._accepting:
            print(f"  Rejected: {item} (shutting down)")

A complete graceful shutdown registers multiple cleanup callbacks and executes them in reverse order (LIFO), similar to how deferred calls work.

        self._callbacks: list[tuple[str, callable]] = []
        self._timeout = timeout
        self._shutting_down = False
        signal.signal(signal.SIGINT, self._handle_signal)
        signal.signal(signal.SIGTERM, self._handle_signal)

    def register(self, name: str, callback: callable) -> None:
        self._callbacks.append((name, callback))
        print(f"  Registered cleanup: {name}")

    def _handle_signal(self, signum: int, frame) -> None:
        if self._shutting_down:
            print("Forced exit!")
            sys.exit(1)
        self._shutting_down = True
        sig_name = signal.Signals(signum).name
        print(f"\n[{sig_name}] Starting graceful shutdown (timeout: {self._timeout}s)")
        self._execute_callbacks()
        sys.exit(0)

        for name, cb in reversed(self._callbacks):
            try:
                print(f"  Shutting down: {name}...")
                cb()
                print(f"  Done: {name}")
            except Exception as e:
                print(f"  ERROR during {name}: {e}")

Language Comparison: Signal Handling
  • Python: signal.signal() for handlers, atexit for exit hooks. Signal handlers run in the main thread only.
  • Go: signal.NotifyContext (Go 1.16+) is the modern approach — returns a context that cancels on signal. Channels via signal.Notify for manual handling.
  • TypeScript: process.on('SIGTERM', ...). No signal support in browsers. AbortController for cancellation propagation.
  • Rust: ctrlc crate for simple cases. tokio::signal for async. Raw signal handling via libc is unsafe and complex.
  • C++: std::signal() is limited (only async-signal-safe functions allowed in handler). Use std::atomic flags and poll from the main loop.
Common Mistakes
  • WRONG: Doing complex work inside the signal handler itself. 
    // C/C++: Signal handlers can only call async-signal-safe functions!
void bad_handler(int) {
    // printf, malloc, mutex lock are NOT safe here
    printf("Shutting down...\n");        // UNDEFINED BEHAVIOR
    fclose(log_file);                    // UNDEFINED BEHAVIOR
    pthread_mutex_lock(&some_mutex);     // DEADLOCK RISK
}
  • RIGHT: Set a flag in the handler, do cleanup in the main loop. 
    std::atomic<bool> shutdown{false};
void good_handler(int) {
    shutdown.store(true);  // atomic write is signal-safe
}
// Main loop checks shutdown flag and does cleanup
  • WRONG: No shutdown timeout. A stuck cleanup callback blocks the process forever, preventing orchestrators from restarting the service.
Key Rules
  • Keep signal handlers minimal — set a flag, nothing else
  • Always have a shutdown timeout — force-exit if cleanup takes too long
  • Clean up resources in reverse order of creation (LIFO)
  • Handle double-signal (second Ctrl+C) by force-exiting immediately
  • Log the shutdown process — you need to debug stuck shutdowns in production
  • Test graceful shutdown explicitly — it’s one of the most common sources of production incidents

Connection Pooling Production Patterns Infrastructure

Creating network connections (database, HTTP, cache) is expensive: TCP handshake, TLS negotiation, authentication, and protocol setup can take 10-100ms per connection. Connection pools maintain a set of reusable connections, amortizing the setup cost. Without pooling, a service handling 1000 requests/second would create and destroy 1000 connections/second — overwhelming both the service and the downstream dependency.

Pool Parameters

Min size: Connections kept warm even when idle (avoids cold start).

Max size: Upper bound to prevent exhausting downstream resources.

Idle timeout: Close connections that sit unused too long.

Max lifetime: Close connections after N minutes to prevent stale state (DNS changes, credential rotation).

Health check: Validate connections before handing them out (ping/heartbeat).

A basic pool pre-creates connections and reuses them. Callers acquire a connection, use it, then return it to the pool.

        finally:

        while not self._pool.empty():
            self._pool.get().close()

Example 3: Pool with Health Checks and Max Lifetime

Production pools validate connections before handing them out (health check) and replace connections that have lived too long (max lifetime) to handle DNS changes and credential rotation.

Python
import queue
import time
from contextlib import contextmanager

class Connection:
    _counter = 0
    def __init__(self):
        Connection._counter += 1
        self.id = Connection._counter
        self.created_at = time.monotonic()
        self._healthy = True
        time.sleep(0.01)  # fast creation for demo

    def ping(self) -> bool:
        return self._healthy

    def query(self, sql: str) -> str:
        return f"[conn#{self.id}] result"

    def close(self) -> None:
        self._healthy = False

class HealthyPool:
    def __init__(self, size: int, max_lifetime: float = 60.0):
        self._pool: queue.Queue[Connection] = queue.Queue(maxsize=size)
        self._size = size
        self._max_lifetime = max_lifetime
        for _ in range(size):
            self._pool.put(Connection())

    def _is_valid(self, conn: Connection) -> bool:
        age = time.monotonic() - conn.created_at
        if age > self._max_lifetime:
            print(f"  Conn #{conn.id} exceeded max lifetime ({age:.1f}s)")
            return False
        if not conn.ping():
            print(f"  Conn #{conn.id} failed health check")
            return False
        return True

    @contextmanager
    def acquire(self):
        conn = self._pool.get(timeout=5)
        if not self._is_valid(conn):
            conn.close()
            conn = Connection()  # replace with fresh
            print(f"  Created replacement conn #{conn.id}")
        try:
            yield conn
        finally:
            self._pool.put(conn)

Language Comparison: Built-in Pooling
  • Python: SQLAlchemy’s create_engine has built-in pooling (pool_size, max_overflow, pool_recycle). urllib3.PoolManager for HTTP.
  • Go: database/sql has built-in connection pooling via SetMaxOpenConns, SetMaxIdleConns, SetConnMaxLifetime. No separate library needed.
  • TypeScript: generic-pool npm package. Knex.js has pool config. pg (node-postgres) has built-in pooling.
  • Rust: r2d2 for sync pools, deadpool and bb8 for async pools. All support health checks and max lifetime.
  • C++: No standard library pool. Most DB drivers (libpq, mysqlclient) require manual pooling. Connection pool is often part of the application framework.
Common Mistakes
  • WRONG: Forgetting to return connections to the pool, causing pool exhaustion. 
    # Python: connection leak!
def handle_request():
    conn = pool.acquire()  # borrowed
    result = conn.query("SELECT 1")
    if result is None:
        return None  # conn never returned!
    pool.release(conn)
    return result
  • RIGHT: Use context managers / defer / RAII to guarantee return. 
    # Python: automatic return via context manager
def handle_request():
    with pool.acquire() as conn:  # returned on exit
        result = conn.query("SELECT 1")
        if result is None:
            return None  # conn still returned!
        return result
  • WRONG: Setting pool max size too high (exhausting downstream DB connections) or too low (becoming the bottleneck).
  • WRONG: No max lifetime — after a DNS change, all pooled connections still point to the old IP address.
Key Rules
  • Always use RAII / context managers / defer to return connections — never rely on manual release
  • Set max_lifetime to handle DNS changes and credential rotation (typically 5-30 minutes)
  • Set max_idle_time to free connections that sit unused (typically 1-10 minutes)
  • Pool size = number of concurrent requests your service handles; do not over-provision
  • Monitor pool metrics: active connections, wait time, timeout errors, health check failures
  • Close the pool during graceful shutdown — drain active connections first
Practice Exercises

2 Add blocking-with-timeout to the pool: when all connections are in use and pool is at max size, callers wait up to N seconds for a connection to be returned. If timeout expires, return an error instead of creating new connections.
