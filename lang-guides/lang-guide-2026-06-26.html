1. Metaprogramming Advanced Patterns rep 1 • 14 days since review
Due today • second review (overdue 8 days)
Quick Recall: Metaprogramming is code that inspects or generates code, at runtime or compile time. The mechanism differs sharply: Python uses decorators/metaclasses/reflection, Go has runtime reflect plus go generate, TypeScript uses decorators + type-level programming, Rust uses macros (compile-time), and C++ uses templates + constexpr (compile-time). Runtime reflection is flexible but slow and untyped; compile-time generation is fast and type-checked but harder to write.

Key Pattern — inspect/generate at the language's metaprogramming layer

Python 
from dataclasses import dataclass, fields
@dataclass
class P: x: int; y: int
print([f.name for f in fields(P)])   # runtime reflection
['x', 'y']
Go 
import "reflect"
type P struct{ X, Y int }
t := reflect.TypeOf(P{})
for i := 0; i < t.NumField(); i++ { fmt.Print(t.Field(i).Name, " ") } // X Y
X Y
TypeScript 
// A decorator is a metaprogramming hook that runs at class definition
const fieldNames: string[] = [];
function track(_: any, key: string) { fieldNames.push(key); }
class P { @track x = 0; @track y = 0; }
console.log(fieldNames);             // [ 'x', 'y' ]
[ 'x', 'y' ]
Rust 
// Declarative macro generates code at compile time (hygienic)
macro_rules! swap { ($a:expr, $b:expr) => {{ let t = $a; $a = $b; $b = t; }}; }
fn main() {
    let (mut x, mut y) = (1, 2);
    swap!(x, y);
    println!("{} {}", x, y);         // 2 1
}
2 1
C++ 
// Template metaprogramming: the compiler recurses to compute a value
template<class T> constexpr int rank = 0;
template<class T> constexpr int rank<T*> = rank<T> + 1;
int main() {
    std::cout << rank<int***>;        // 3, computed at compile time
}
3

1
Build a tiny auto-registry: every type/function tagged with a marker registers its name into a global list at definition time (Python decorator, Go init() + reflect, TS decorator, Rust a declarative macro that also emits a registration, C++ a static initializer). Print all registered names and note which registrations happen at compile time (Rust/C++) versus runtime (Python/Go/TS).
#1 Pitfall per Language:
  • Python: reaching for metaclasses where a decorator or __init_subclass__ would do — metaclass conflicts under multiple inheritance are brutal.
  • Go: reflect is slow and bypasses the type checker — cache reflect.Type and keep it off hot paths.
  • TypeScript: legacy experimentalDecorators and TS 5 stage-3 decorators have different semantics — pin which you use.
  • Rust: macro hygiene and the fact that proc-macros need a separate crate and slow compiles — prefer declarative macros when they suffice.
  • C++: deep template recursion explodes compile times and yields unreadable errors — cap depth, prefer constexpr/concepts.
2. Event Systems & Reactive Patterns Advanced Patterns rep 1 • 9 days since review
Due today • second review (overdue 8 days)
Quick Recall: Decouple producers of events from consumers via an observer / pub-sub bus. A subject keeps a list of subscriber callbacks and emits to all of them; subscribe returns an unsubscribe handle so you can avoid leaks. Reactive extends this to streams of values over time with operators (map/filter/debounce).

Key Pattern — a minimal event emitter (subscribe + emit)

Python 
class Emitter:
    def __init__(self): self._subs = []
    def on(self, fn): self._subs.append(fn); return lambda: self._subs.remove(fn)
    def emit(self, x):
        for fn in list(self._subs): fn(x)   # iterate a copy
e = Emitter(); off = e.on(lambda x: print("got", x))
e.emit(1); off(); e.emit(2)                 # got 1
got 1
Go 
// A channel IS an event stream; range receives until closed
events := make(chan int)
go func(){ for x := range events { fmt.Println("got", x) } }()
events <- 1
close(events)                                // got 1
got 1
TypeScript 
type Handler = (x: number) => void;
class Emitter {
  private subs = new Set<Handler>();
  on(h: Handler) { this.subs.add(h); return () => this.subs.delete(h); }
  emit(x: number) { this.subs.forEach(h => h(x)); }
}
const e = new Emitter();
const off = e.on(x => console.log("got", x));
e.emit(1); off(); e.emit(2);                 // got 1
got 1
Rust 
struct Emitter { subs: Vec<Box<dyn Fn(i32)>> }
impl Emitter {
    fn on(&mut self, f: impl Fn(i32) + 'static) { self.subs.push(Box::new(f)); }
    fn emit(&self, x: i32) { for f in &self.subs { f(x); } }
}
fn main() {
    let mut e = Emitter { subs: vec![] };
    e.on(|x| println!("got {}", x));
    e.emit(1);                               // got 1
}
got 1
C++ 
#include <functional>
#include <vector>
struct Emitter {
    std::vector<std::function<void(int)>> subs;
    void on(std::function<void(int)> f) { subs.push_back(std::move(f)); }
    void emit(int x) { for (auto& f : subs) f(x); }
};
int main() {
    Emitter e;
    e.on([](int x){ std::cout << "got " << x << "\n"; });
    e.emit(1);                               // got 1
}
got 1
Challenge (all 5 languages):
1
Build a typed event bus supporting multiple named events, once() (auto-unsubscribe after the first fire), and an unsubscribe handle. Add a debounce operator that coalesces bursts within 50ms into one emission. Prove there's no leak: after unsubscribing every handler, emit becomes a no-op.
#1 Pitfall per Language:
  • Python: holding strong refs to subscriber callbacks (esp. bound methods) keeps the owning object alive — use weakref for observers.
  • Go: sending on a closed channel panics; a slow consumer with no buffer blocks producers — size the channel and own the close.
  • TypeScript: never removing addEventListener handlers is the classic leak — capture the unsubscribe and call it on teardown.
  • Rust: handlers that mutate shared state fight the borrow checker — wrap in Rc<RefCell>/Arc<Mutex> and mind reentrancy.
  • C++: a handler that unsubscribes during emit mutates the vector mid-iteration — iterate over a copy or defer removals.
3. Configuration Management Advanced Patterns rep 1 • 9 days since review
Due today • second review (overdue 8 days)
Quick Recall: Load config from layered sources with clear precedence: defaults < file < environment variables < CLI flags. Parse into a typed struct and validate at startup so the app fails fast on missing/invalid required values. Twelve-factor: keep config in the environment, not the code.

Key Pattern — read an env var with a typed default

Python 
import os
port = int(os.getenv("PORT", "8080"))
debug = os.getenv("DEBUG", "false").lower() == "true"
print(port, debug)               # 8080 False
8080 False
Go 
port := 8080
if v := os.Getenv("PORT"); v != "" { port, _ = strconv.Atoi(v) }
debug := os.Getenv("DEBUG") == "true"
fmt.Println(port, debug)         // 8080 false
8080 false
TypeScript 
const port = Number(process.env.PORT ?? 8080);
const debug = (process.env.DEBUG ?? "false") === "true";
console.log(port, debug);        // 8080 false
8080 false
Rust 
use std::env;
let port: u16 = env::var("PORT").ok()
    .and_then(|v| v.parse().ok()).unwrap_or(8080);
let debug = env::var("DEBUG").map(|v| v == "true").unwrap_or(false);
println!("{} {}", port, debug);  // 8080 false
8080 false
C++ 
#include <cstdlib>
const char* p = std::getenv("PORT");
int port = p ? std::atoi(p) : 8080;
const char* d = std::getenv("DEBUG");
bool debug = d && std::string(d) == "true";

Challenge (all 5 languages):
1
Implement layered config: hard-coded defaults, overridden by a TOML/JSON file, overridden by env vars, overridden by CLI flags. Deserialize into a typed Config, validate (port in 1..65535, required DB URL present), and on error report which key and which source failed. Print the effective merged config plus the source of each value.

  • Python: treating "false"/"0"/"" as truthy — any non-empty string is truthy; parse booleans explicitly.

  • C++: getenv returns nullptr for unset vars — dereferencing without the null check is UB and it isn't thread-safe vs setenv.
4. Middleware & Request Lifecycle Production Patterns rep 0 • 9 days since review
Due today • first scheduled review (overdue 8 days)
Quick Recall: Middleware wraps a handler to run cross-cutting logic (logging, auth, timing) before and after the core work — an onion of layers. Each layer calls the next; the request flows inward, the response flows back outward in reverse order. It's the decorator pattern applied to HTTP.

Key Pattern — a logging middleware wrapping a handler

Python 
def logging_mw(handler):
    def wrapped(req):
        print("->", req)
        resp = handler(req)
        print("<-", resp)
        return resp
    return wrapped
@logging_mw
def hello(req): return "200 OK"
hello("GET /")
-> GET / <- 200 OK
Go 
type Handler func(string) string
func logging(next Handler) Handler {
    return func(req string) string {
        fmt.Println("->", req)
        resp := next(req)
        fmt.Println("<-", resp)
        return resp
    }
}
h := logging(func(r string) string { return "200 OK" })
h("GET /")
-> GET / <- 200 OK
TypeScript 
type Req = string; type Next = () => string;
const logging = (req: Req, next: Next): string => {
  console.log("->", req);
  const resp = next();
  console.log("<-", resp);
  return resp;
};
logging("GET /", () => "200 OK");
-> GET / <- 200 OK
Rust 
fn logging<F: Fn(&str) -> String>(next: F) -> impl Fn(&str) -> String {
    move |req| {
        println!("-> {}", req);
        let resp = next(req);
        println!("<- {}", resp);
        resp
    }
}
fn main() {
    let h = logging(|_r| "200 OK".to_string());
    h("GET /");
}
-> GET / <- 200 OK
C++ 
#include <functional>
using Handler = std::function<std::string(std::string)>;
Handler logging(Handler next) {
    return [next](std::string req) {
        std::cout << "-> " << req << "\n";
        auto resp = next(req);
        std::cout << "<- " << resp << "\n";
        return resp;
    };
}
int main() {
    auto h = logging([](std::string){ return std::string("200 OK"); });
    h("GET /");
}
-> GET / <- 200 OK
Challenge (all 5 languages):
1
Compose a chain [recover, logging, auth, timing] around a handler so the in/out order is correct (recover outermost). Add a middleware that short-circuits with 401 when auth fails — the inner handler never runs. Print the entry/exit order to prove the onion model and that timing wraps only the inner work.
#1 Pitfall per Language:
  • Python: a decorator that forgets functools.wraps loses the handler's name/docstring; stacked decorators apply bottom-up.
  • Go: not propagating context.Context (and its cancellation/deadline) through the chain breaks timeouts and tracing.
  • TypeScript: forgetting to call (or await) next() hangs the request; calling it twice double-sends the response.
  • Rust: each layer changes the return type — express bounds with generics or box to dyn Fn to keep it composable.
  • C++: capturing next by reference dangles once the builder returns — capture by value (copy the std::function).
5. Caching Patterns Production Patterns rep 0 • 9 days since review
Due today • first scheduled review (overdue 8 days)
Quick Recall: Store expensive results keyed by input to avoid recompute/refetch. Policies: TTL (expire after time), LRU (evict least-recently-used at capacity), write-through vs cache-aside. The hard parts are invalidation, staleness, and the thundering herd on a cold miss.

Key Pattern — memoize with a map (cache-aside)

Python 
from functools import lru_cache
@lru_cache(maxsize=128)
def fib(n): return n if n < 2 else fib(n-1) + fib(n-2)
print(fib(30))                   # 832040 (memoized)
832040
Go 
cache := map[int]int{}
var fib func(int) int
fib = func(n int) int {
    if n < 2 { return n }
    if v, ok := cache[n]; ok { return v }
    v := fib(n-1) + fib(n-2)
    cache[n] = v
    return v
}
fmt.Println(fib(30))             // 832040
832040
TypeScript 
const cache = new Map<number, number>();
function fib(n: number): number {
  if (n < 2) return n;
  if (cache.has(n)) return cache.get(n)!;
  const v = fib(n - 1) + fib(n - 2);
  cache.set(n, v);
  return v;
}
console.log(fib(30));            // 832040
832040
Rust 
use std::collections::HashMap;
fn fib(n: u64, memo: &mut HashMap<u64, u64>) -> u64 {
    if n < 2 { return n; }
    if let Some(&v) = memo.get(&n) { return v; }
    let v = fib(n-1, memo) + fib(n-2, memo);
    memo.insert(n, v);
    v
}
fn main() {
    let mut memo = HashMap::new();
    println!("{}", fib(30, &mut memo)); // 832040
}
832040
C++ 
#include <unordered_map>
std::unordered_map<long,long> memo;
long fib(long n) {
    if (n < 2) return n;
    if (auto it = memo.find(n); it != memo.end()) return it->second;
    return memo[n] = fib(n-1) + fib(n-2);
}
int main() { std::cout << fib(30); } // 832040
832040
Challenge (all 5 languages):
1
Implement an LRU cache with capacity K (O(1) get/put via a hash map + doubly linked list, or your language's ordered map). Add per-entry TTL so stale entries miss. Then protect a slow loader against the thundering herd with single-flight: concurrent misses for the same key wait for one in-flight load. Report hit/miss/eviction counts.
#1 Pitfall per Language:
  • Python: lru_cache holds references forever (memory growth), breaks on unhashable args, and isn't shared across processes.
  • Go: a plain map cache under concurrency races — guard with sync.Mutex or use sync.Map; unbounded maps leak.
  • TypeScript: a Map keyed by object uses reference identity — two equal-valued keys won't hit; canonicalize keys to strings.
  • Rust: a single &mut memo can't be shared across threads — use Mutex/RwLock or DashMap for concurrent caches.
  • C++: inserting into unordered_map can invalidate references/iterators on rehash — reserve ahead or re-find after insert.
6. Graceful Shutdown & Signal Handling Production Patterns rep 0 • 9 days since review
Due today • first scheduled review (overdue 8 days)
Quick Recall: On SIGINT/SIGTERM: stop accepting new work, finish in-flight requests within a deadline, release resources (DB pools, files), then exit. Trap the signal, cancel a context/flag, wait on a drain timeout, then force-exit. Never kill mid-transaction.

Key Pattern — trap the signal and clean up

Python 
import signal, sys
def handler(sig, frame):
    print("shutting down...")
    sys.exit(0)
signal.signal(signal.SIGINT, handler)
# ... main loop ...   Ctrl-C -> "shutting down..."
shutting down...
Go 
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()
<-ctx.Done()                     // blocks until a signal arrives
fmt.Println("shutting down...")
// srv.Shutdown(context.WithTimeout(...))
shutting down...
TypeScript 
process.on("SIGTERM", () => {
  console.log("shutting down...");
  server.close(() => process.exit(0));
  setTimeout(() => process.exit(1), 10_000).unref(); // force after 10s
});
shutting down...
Rust 
#[tokio::main]
async fn main() {
    tokio::signal::ctrl_c().await.unwrap();
    println!("shutting down...");
    // drain in-flight tasks, then return
}
shutting down...
C++ 
#include <csignal>
#include <atomic>
std::atomic<bool> stop{false};
void on_sig(int) { stop = true; }     // only async-signal-safe ops here
int main() {
    std::signal(SIGINT, on_sig);
    while (!stop) { /* serve */ }
    std::cout << "shutting down...\n";
}
shutting down...
Challenge (all 5 languages):
1
Build a server-ish loop that, on SIGTERM, (1) stops accepting new tasks, (2) waits up to 5s for N in-flight tasks to finish, (3) force-exits if the deadline passes, and (4) closes a fake DB pool exactly once. Log the in-flight count as it drains and prove no task is interrupted mid-work.
#1 Pitfall per Language:
  • Python: signal handlers run only in the main thread, between bytecodes — keep them tiny; KeyboardInterrupt can fire anywhere.
  • Go: not calling stop()/srv.Shutdown leaves connections open; forgetting the drain timeout blocks forever.
  • TypeScript: process.exit() kills pending async work immediately — close servers/finish writes first; unref the force-timer.
  • Rust: blocking the async runtime during cleanup stalls everything — use spawn_blocking for synchronous teardown.
  • C++: calling non-async-signal-safe functions (malloc, printf, std::cout) in the handler is UB — set an atomic flag and act in main.

Bit Manipulation & Binary Data

The same six bitwise operators in every language — but signedness, shift behavior, and byte order diverge in ways that quietly corrupt data. The foundation for flags, masks, compact sets, and binary protocols.

Bitwise Operations & Bit Manipulation Bit Manipulation
Why it matters: bitwise ops power flags, masks, hashing, compression, and low-level protocols — plus a whole genre of interview tricks (counting bits, power-of-two checks, XOR swaps). They're & | ^ ~ << >> everywhere, but signedness and shift rules differ enough to cause real bugs.

AND (&) masks bits off, OR (|) sets bits, XOR (^) toggles/finds differences, NOT (~) inverts, and shifts (<</>>) multiply or divide by powers of two while moving bit fields. The idioms worth memorizing: x & (x-1) clears the lowest set bit, x & -x isolates it, and x > 0 && (x & (x-1)) == 0 tests for a power of two.

Example 1 — set, clear, toggle, and test a single bit

Python 
x = 0b1010                       # 10
x |= (1 << 0)                    # set bit 0    -> 1011
x &= ~(1 << 1)                   # clear bit 1  -> 1001
x ^= (1 << 2)                    # toggle bit 2 -> 1101
print(bin(x), (x >> 3) & 1)      # 0b1101 1
0b1101 1
Go 
x := 0b1010
x |= 1 << 0          // set
x &^= 1 << 1         // clear (AND-NOT operator)
x ^= 1 << 2          // toggle
fmt.Printf("%b %d\n", x, (x>>3)&1) // 1101 1
1101 1
TypeScript 
let x = 0b1010;
x |= 1 << 0;                     // set
x &= ~(1 << 1);                  // clear
x ^= 1 << 2;                     // toggle
console.log(x.toString(2), (x >> 3) & 1); // 1101 1
1101 1
Rust 
let mut x: u32 = 0b1010;
x |= 1 << 0;                     // set
x &= !(1 << 1);                  // clear
x ^= 1 << 2;                     // toggle
println!("{:b} {}", x, (x >> 3) & 1); // 1101 1
1101 1
C++ 
#include <bitset>
unsigned x = 0b1010;
x |= 1u << 0;                    // set
x &= ~(1u << 1);                 // clear
x ^= 1u << 2;                    // toggle
std::cout << std::bitset<4>(x) << " " << ((x >> 3) & 1); // 1101 1
1101 1

Example 2 — population count & power-of-two check

Python 
n = 0b10110
print(n.bit_count())             # 3  (Python 3.10+)
def is_pow2(x): return x > 0 and (x & (x - 1)) == 0
print(is_pow2(16), is_pow2(18))  # True False
3 True False
Go 
import "math/bits"
n := 0b10110
fmt.Println(bits.OnesCount(uint(n)))         // 3
isPow2 := func(x int) bool { return x > 0 && x&(x-1) == 0 }
fmt.Println(isPow2(16), isPow2(18))          // true false
3 true false
TypeScript 
const popcount = (n: number) => { let c = 0; while (n) { n &= n - 1; c++; } return c; };
console.log(popcount(0b10110));              // 3
const isPow2 = (x: number) => x > 0 && (x & (x - 1)) === 0;
console.log(isPow2(16), isPow2(18));         // true false
3 true false
Rust 
let n: u32 = 0b10110;
println!("{}", n.count_ones());              // 3
let is_pow2 = |x: u32| x != 0 && x & (x - 1) == 0;
println!("{} {}", is_pow2(16), is_pow2(18)); // true false
3 true false
C++ 
#include <bit>
unsigned n = 0b10110;
std::cout << std::popcount(n) << "\n";        // 3 (C++20)
auto is_pow2 = [](unsigned x){ return std::has_single_bit(x); }; // C++20
std::cout << is_pow2(16) << " " << is_pow2(18); // 1 0
3 1 0
Language Comparison: All five share & | ^ ~ << >>, but the edges differ. Go spells "clear" with the dedicated &^ (AND-NOT) operator. JS/TS coerce operands to 32-bit signed ints, so 1 << 31 is negative and big numbers wrap — use >>> for unsigned shift or BigInt beyond 32 bits. Rust makes you choose the width and panics on overflowing shifts in debug. C++20 adds <bit> (popcount, has_single_bit, rotl/rotr, countl_zero). Python ints are arbitrary precision with no fixed width or unsigned type, so ~x behaves like infinite two's-complement.
Common Mistakes:
❌ [TypeScript] Using bitwise ops on numbers > 2^31 — they truncate to 32 bits and (1 << 31) is negative.
✅ [TypeScript] Use >>> 0 to read as unsigned, or BigInt for values wider than 32 bits.
❌ [Python] Expecting a fixed width — ~x is -(x+1), not a 32-bit complement.
✅ [Python] Mask explicitly with & 0xFFFFFFFF when you need fixed-width two's-complement behavior.
❌ [C++] Shifting a signed/negative value or by ≥ the bit-width, or shifting into the sign bit — that's UB.
✅ [C++] Use unsigned types for bit twiddling so shifts are well-defined and there's no sign extension.
❌ [Rust] An overflowing shift (x << 32 on a u32) panics in debug / wraps in release.
✅ [Rust] Use wrapping_shl/checked_shl or a wider integer type when the shift amount may reach the width.
Key Rules:
  • x & (x-1) clears the lowest set bit (loop it to popcount); x & -x isolates the lowest set bit.
  • Prefer unsigned types for bit twiddling — well-defined shifts, no sign extension.
  • Power of two: x > 0 && (x & (x-1)) == 0. Reach for the stdlib popcount/bit_count over hand loops when available.

Bit Flags & Bitsets Bit Manipulation
Why it matters: packing many booleans into one integer (permissions, feature toggles, sets of small enums) is compact and cache-friendly — a single & tests membership. OS APIs (open flags, mmap protections), graphics, and wire protocols all use bitflags. The craft is doing it type-safely.

Assign each flag a distinct power of two; combine with |, test with &, remove with &~. A bitset generalizes this to many bits, giving set algebra: union |, intersection &, difference &~, and size via popcount. Each language offers an ergonomic wrapper: Python enum.Flag, Go typed uint + iota, Rust the bitflags! crate (or plain consts), C++ std::bitset, TS numeric enums.

Example 1 — define and combine permission flags

Python 
from enum import Flag, auto
class Perm(Flag):
    READ = auto(); WRITE = auto(); EXEC = auto()
p = Perm.READ | Perm.WRITE
print(Perm.WRITE in p, Perm.EXEC in p)  # True False
print(p.value)                          # 3
True False 3
Go 
type Perm uint
const ( Read Perm = 1 << iota; Write; Exec )
p := Read | Write
fmt.Println(p&Write != 0, p&Exec != 0)  // true false
fmt.Println(uint(p))                    // 3
true false 3
TypeScript 
enum Perm { Read = 1, Write = 2, Exec = 4 }
let p = Perm.Read | Perm.Write;
console.log((p & Perm.Write) !== 0, (p & Perm.Exec) !== 0); // true false
console.log(p);                         // 3
true false 3
Rust 
// Without the bitflags crate: plain consts on a fixed width
const READ:  u8 = 1 << 0;

C++

unsigned p = Read | Write;
std::cout << ((p & Write) != 0) << " " << ((p & Exec) != 0) << "\n"; // 1 0
std::cout << p;                          // 3
1 0 3

Example 2 — a bitset as a set of small integers (union / intersection / difference)

Python

b = 0b0000_1100                  # {2,3}
print(bin(a | b), bin(a & b))    # union, intersection
print(bin(a & ~b))               # difference
0b111101 0b100 0b110001
Go 
a, b := 0b00110101, 0b00001100
fmt.Printf("%b %b %b\n", a|b, a&b, a&^b) // 111101 100 110001
111101 100 110001
TypeScript 
const a = 0b00110101, b = 0b00001100;
console.log((a | b).toString(2), (a & b).toString(2), (a & ~b).toString(2));
// 111101 100 110001
111101 100 110001
Rust 
let (a, b): (u8, u8) = (0b0011_0101, 0b0000_1100);
println!("{:b} {:b} {:b}", a | b, a & b, a & !b); // 111101 100 110001
111101 100 110001
C++ 
#include <bitset>
std::bitset<8> a(0b00110101), b(0b00001100);
std::cout << (a | b) << " " << (a & b) << " " << (a & ~b);
// 00111101 00000100 00110001
00111101 00000100 00110001
Language Comparison: Python's enum.Flag and Rust's bitflags! crate give named, type-safe flag sets with readable repr/Debug — the safest choices. Go enumerates powers of two with 1 << iota on a named uint. C++ scoped enum class won't | without a defined operator, so people use unscoped enums or operator overloads; std::bitset<N> shines for fixed-size sets with .count()/.test(). TS enums are just numbers with no exhaustiveness, so Perm.Read | 8 is silently allowed.
Common Mistakes:
❌ [All] Assigning a non-power-of-two to a flag (e.g. 3) so two flags overlap and tests misfire.
✅ [All] Always define flags as 1 << n (or auto()) so each occupies a distinct bit.
❌ [TypeScript] Treating a flags number as type-safe — any number unifies with the enum, so you can OR in garbage.
✅ [TypeScript] Validate the combined mask (or use a tagged wrapper) instead of trusting the enum type.
❌ [C++] A scoped enum class won't combine with |; unscoped enums leak names into the surrounding scope.
✅ [C++] Decide deliberately: unscoped enum (implicit int) or scoped enum with defined operator|/operator&.
❌ [Go] Forgetting != 0 in the test — p & Write is an int, not a bool.
✅ [Go] Write p & Write != 0; use an unsigned type so shifting into the top bit is well-defined.
Key Rules:
  • One flag per bit: values are distinct powers of two. Combine with |, test with & (compare to 0 in C/Go), clear with &~.
  • Prefer a named wrapper (enum.Flag, bitflags!, std::bitset, typed uint + iota) over bare ints for safety.
  • A bitset is a set of small ints: | union, & intersection, &~ difference, popcount = size.

Endianness & Byte Order Binary Data
Why it matters: the instant you write an integer to a file, socket, or binary format, byte order matters. Little-endian (x86/ARM) and big-endian ("network byte order") disagree on which byte comes first, and a mismatch silently corrupts data across machines — a classic, hard-to-spot interop bug.

Endianness is the order in which the bytes of a multi-byte integer are stored. The value 0x01020304 is stored as bytes 04 03 02 01 in little-endian and 01 02 03 04 in big-endian. Network protocols standardize on big-endian. The rule: always serialize/deserialize with an explicit byte order, and never memcpy a raw struct across the wire.

Example 1 — pack a u32 to bytes in both orders

Python 
import struct
n = 0x01020304
print(struct.pack(">I", n).hex())  # big-endian:    01020304
print(struct.pack("<I", n).hex())  # little-endian: 04030201
print(struct.unpack(">I", bytes.fromhex("01020304"))[0]) # 16909060
01020304 04030201 16909060
Go 
import ("encoding/binary"; "encoding/hex")
b := make([]byte, 4)
binary.BigEndian.PutUint32(b, 0x01020304)
fmt.Println(hex.EncodeToString(b))               // 01020304
binary.LittleEndian.PutUint32(b, 0x01020304)
fmt.Println(hex.EncodeToString(b))               // 04030201
01020304 04030201
TypeScript 
const buf = new ArrayBuffer(4);
const dv = new DataView(buf);
dv.setUint32(0, 0x01020304, false);  // false = big-endian
console.log(Buffer.from(buf).toString("hex"));   // 01020304
dv.setUint32(0, 0x01020304, true);   // true = little-endian
console.log(Buffer.from(buf).toString("hex"));   // 04030201
01020304 04030201
Rust 
let n: u32 = 0x01020304;
println!("{}", hex::encode(n.to_be_bytes())); // 01020304
println!("{}", hex::encode(n.to_le_bytes())); // 04030201
let back = u32::from_be_bytes([1, 2, 3, 4]);
println!("{}", back);                         // 16909060
01020304 04030201 16909060
C++ 
#include <cstdint>
#include <bit>
uint32_t n = 0x01020304;
uint32_t be = (std::endian::native == std::endian::little)
              ? std::byteswap(n) : n;          // C++23 byteswap
auto* p = reinterpret_cast<unsigned char*>(&be);
for (int i = 0; i < 4; i++) printf("%02x", p[i]); // 01020304
01020304

Example 2 — detect the host's native endianness

Python 
import sys
print(sys.byteorder)             # 'little' on x86/ARM
little
Go 
// Probe by writing a uint16 and reading its first byte
var i uint16 = 1
b := *(*[2]byte)(unsafe.Pointer(&i))
if b[0] == 1 { fmt.Println("little") } else { fmt.Println("big") }
little
TypeScript 
const u16 = new Uint16Array([1]);
const u8 = new Uint8Array(u16.buffer);
console.log(u8[0] === 1 ? "little" : "big"); // little
little
Rust 
println!("{}", if cfg!(target_endian = "little") { "little" } else { "big" });
little
C++ 
#include <bit>
std::cout << (std::endian::native == std::endian::little ? "little" : "big"); // C++20
little
Language Comparison: Python's struct format chars (< > ! =) make order explicit and are the go-to for binary protocols (! = network = big-endian). Go's encoding/binary exposes BigEndian/LittleEndian with typed Put/Uint helpers. JS DataView takes a littleEndian boolean per call — but typed arrays use native order, a portability trap. Rust bakes it into the integer types (to_be_bytes/to_le_bytes/from_*_bytes) with no library. C++ historically rolled its own; C++20 adds std::endian and C++23 std::byteswap.
Common Mistakes:
❌ [All] Sending raw in-memory integers/structs over a socket and assuming the other end matches.
✅ [All] Pick an explicit byte order at every serialization boundary (big-endian for interchange/network formats).
❌ [TypeScript] Using a typed array (Uint32Array) for I/O — it uses the platform's native endianness.
✅ [TypeScript] Use DataView with an explicit littleEndian flag for portable reads/writes.
❌ [C++] reinterpret_cast-ing a struct to bytes ignores padding and endianness and may break strict aliasing.
✅ [C++] Serialize field by field with explicit byte order; don't blast the struct's memory layout.
❌ [Python] Forgetting that "!" means big-endian, then mixing "=" (native) into a wire format.
✅ [Python] Pin the format string to ! or > for anything that crosses machines.
Key Rules:
  • Choose an explicit byte order at every serialization boundary; network/interchange formats are conventionally big-endian.
  • Never send raw structs across the wire — padding, alignment, and endianness all differ. Serialize each field.
  • Use the stdlib (struct, encoding/binary, to_*_bytes, DataView, std::byteswap) over manual shifting when you can.

Identifiers & Randomness

Two flavors of randomness — fast seedable PRNGs versus cryptographically secure generators — and the 128-bit identifiers built on them. Choosing wrong here is a genuine security or performance bug.

Random Number Generation Randomness
Why it matters: simulations, sampling, shuffling, tokens, and tests all need randomness — but there are two very different kinds. A fast PRNG (seedable, reproducible) is right for games and tests; a CSPRNG is mandatory for tokens, keys, and passwords. Using the wrong one is a real security bug.

A pseudo-random generator is deterministic given a seed — perfect for reproducibility, useless for secrets because the output is predictable. A cryptographically secure RNG draws from the OS entropy pool and resists prediction. Most languages ship both; pick by use case, never blur them.

Example 1 — seeded, reproducible PRNG

Python 
import random
a = random.Random(42).randint(1, 100)
b = random.Random(42).randint(1, 100)
print(a == b)                    # True (same seed -> same sequence)
True
Go 
import "math/rand"
a := rand.New(rand.NewSource(42)).Intn(100)
b := rand.New(rand.NewSource(42)).Intn(100)
fmt.Println(a == b)              // true
true
TypeScript 
// Math.random() is NOT seedable; use a tiny seeded PRNG (mulberry32)
function mulberry32(seed: number) {
  return () => {
    seed = (seed + 0x6D2B79F5) | 0;
    let t = Math.imul(seed ^ (seed >>> 15), 1 | seed);
    t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t;
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}
console.log(mulberry32(42)() === mulberry32(42)()); // true
true
Rust 
use rand::{Rng, SeedableRng};
use rand::rngs::StdRng;
let a: u32 = StdRng::seed_from_u64(42).gen_range(0..100);
let b: u32 = StdRng::seed_from_u64(42).gen_range(0..100);
println!("{}", a == b);          // true
true
C++ 
#include <random>
std::mt19937 g1(42), g2(42);     // seeded Mersenne Twister
std::uniform_int_distribution<int> d(0, 99);
std::cout << (d(g1) == d(g2));   // 1 (same seed -> same value)
1

Example 2 — cryptographically secure randomness (for tokens)

Python 
import secrets
print(len(secrets.token_hex(16)))       # 32 hex chars = 16 random bytes
print(secrets.randbelow(6) in range(6)) # True (unbiased die)
32 True
Go 
import ("crypto/rand"; "encoding/hex")
b := make([]byte, 16)
rand.Read(b)                            // crypto/rand, NOT math/rand
fmt.Println(len(hex.EncodeToString(b))) // 32
32
TypeScript 
import { randomBytes, randomInt } from "node:crypto";
console.log(randomBytes(16).toString("hex").length); // 32
console.log(randomInt(0, 6) < 6);                     // true (unbiased)
// browser: crypto.getRandomValues(new Uint8Array(16))
32 true
Rust 
use rand::RngCore;
let mut buf = [0u8; 16];
rand::rngs::OsRng.fill_bytes(&mut buf);  // OS CSPRNG
println!("{}", hex::encode(buf).len());  // 32
32
C++ 
#include <random>
std::random_device rd;           // non-deterministic source (seeds/tokens)
std::uniform_int_distribution<int> d(0, 5);
std::cout << (d(rd) < 6);        // 1
1
Language Comparison: The split is universal but the APIs differ. Python is exemplary: random for PRNG, secrets for security. Go makes it explicit with two packages — math/rand (fast, seedable, NOT secure) vs crypto/rand. Rust's rand crate offers StdRng/SmallRng plus OsRng for crypto. C++ separates engines (mt19937) from distributions (uniform_int_distribution) and exposes random_device for entropy. JS is the outlier: Math.random() is neither seedable nor secure — bring a PRNG library for reproducibility and use crypto.getRandomValues/node:crypto for secrets.
Common Mistakes:
❌ [All] Using a fast PRNG (Math.random, math/rand, mt19937) for tokens/passwords/keys — the output is predictable.
✅ [All] Use the CSPRNG (secrets, crypto/rand, OsRng, getRandomValues) for anything that must be unguessable.
❌ [TypeScript] Assuming Math.random() can be seeded or is secure — it is neither.
✅ [TypeScript] Use a seeded PRNG lib for reproducibility; node:crypto/getRandomValues for security.
❌ [C++] Re-creating std::random_device per draw (slow, possibly deterministic on some libs); rd() % n mod-bias.
✅ [C++] Use random_device once to seed an engine; draw ranges via uniform_int_distribution.
❌ [All] Modulo bias: rand() % n skews when n doesn't divide the generator's range.
✅ [All] Use the range API (randbelow, gen_range, uniform_int_distribution, randInt), not raw % n.
Key Rules:
  • Two worlds: seedable PRNG for reproducibility (games, tests, simulations) vs CSPRNG for secrets (tokens, keys, salts).
  • Avoid modulo bias — use the language's range API rather than raw % n.
  • Seed a PRNG once and reuse the generator; never seed a CSPRNG yourself — it pulls from the OS entropy pool.

UUIDs & Unique Identifiers Identifiers
Why it matters: distributed systems need IDs that are unique without a central coordinator — UUIDs let any node mint one with negligible collision risk. Choosing the version (random v4 vs time-ordered v7) affects database index locality and sortability, a real performance lever.

A UUID is a 128-bit identifier rendered as 32 hex digits in 8-4-4-4-12 groups. v4 is purely random (122 random bits) — excellent uniqueness, but its random order hurts B-tree insert locality. v7 (newer) embeds a millisecond timestamp in the high bits, so IDs sort by creation time and index efficiently. ULID/KSUID are popular alternatives with the same time-sortable idea.

Example 1 — generate a random (v4) UUID

Python 
import uuid
u = uuid.uuid4()                 # e.g. 3f2504e0-4f89-41d3-9a0c-0305e82c3301
print(u.version, len(u.hex))     # 4 32
4 32
Go 
import "github.com/google/uuid"
u := uuid.New()                  // v4 by default
fmt.Println(u.Version(), len(u.String())) // 4 36
4 36
TypeScript 
import { randomUUID } from "node:crypto";
const u = randomUUID();          // RFC 4122 v4
console.log(u.length, u[14]);    // 36 4  (the version nibble is '4')
36 4
Rust 
use uuid::Uuid;
let u = Uuid::new_v4();
println!("{} {}", u.get_version_num(), u.hyphenated().to_string().len());
// 4 36
4 36
C++ 
// No std UUID; with a library such as stduuid
#include <uuid.h>
std::random_device rd;
uuids::uuid_random_generator gen{rd};
uuids::uuid id = gen();
std::cout << uuids::to_string(id).size(); // 36
36

Example 2 — time-ordered (v7) IDs sort by creation

Python 
# v7 embeds a millisecond timestamp in the leading bits -> sortable
earlier = "0190b1c0-0000-7000-8000-000000000000"
later   = "0190b1c1-0000-7000-8000-000000000000"
print(earlier < later)           # True: lexicographic order == time order
True
Go 
earlier := "0190b1c0-0000-7000-8000-000000000000"
later   := "0190b1c1-0000-7000-8000-000000000000"
fmt.Println(earlier < later)     // true
true
TypeScript 
const earlier = "0190b1c0-0000-7000-8000-000000000000";
const later   = "0190b1c1-0000-7000-8000-000000000000";
console.log(earlier < later);    // true (string sort == chronological)
true
Rust 
let earlier = "0190b1c0-0000-7000-8000-000000000000";
let later   = "0190b1c1-0000-7000-8000-000000000000";
println!("{}", earlier < later); // true
true
C++ 
std::string earlier = "0190b1c0-0000-7000-8000-000000000000";
std::string later   = "0190b1c1-0000-7000-8000-000000000000";
std::cout << (earlier < later);  // 1
1
Language Comparison: Python (uuid) and Node (crypto.randomUUID) ship v4 in the standard library — no dependency. Go, Rust, and C++ rely on de-facto-standard libraries (google/uuid, the uuid crate, stduuid) since there's no std UUID type. v7/time-ordered support is newest: the uuid crate and recent Python expose it, else reach for ULID/KSUID libs. The cross-language point: a UUID is just 128 bits — store it as 16 bytes (or a native UUID column), not a 36-char string, to save space and speed indexes.
Common Mistakes:
❌ [Database] Using random v4 UUIDs as a clustered primary key — random inserts fragment B-tree indexes.
✅ [Database] Prefer v7/ULID for write-heavy tables so inserts stay append-mostly and indexes stay dense.
❌ [All] Storing UUIDs as 36-char strings — 2.25× the space and slower comparisons/joins than 16 bytes.
✅ [All] Store as a 16-byte / native UUID column and compare by bytes.
❌ [Security] Treating a UUID as a secret/unguessable capability — v1/v7 leak time (and v1 the MAC).
✅ [Security] Use a real CSPRNG token type for auth; a UUID guarantees uniqueness, not secrecy.
❌ [TypeScript] Pulling in a heavy uuid npm package when Node's built-in crypto.randomUUID already gives v4.
✅ [TypeScript] Use the built-in unless you specifically need v1/v5/v7 the platform doesn't provide.
Key Rules:
  • 128 bits, rendered 8-4-4-4-12. v4 = random (max uniqueness, no order); v7 = time-prefixed (sortable, index-friendly).
  • Store as 16 bytes / a native UUID column, not a string. Compare by bytes.
  • A UUID guarantees uniqueness, not secrecy — don't use it where you need an unguessable token.
