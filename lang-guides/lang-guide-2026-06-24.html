1. Hash Maps & Hash Sets Collections rep 3 • due today
Due today • 15 days since last review
Quick Recall: A hash map stores key→value with O(1) average lookup; a hash set stores unique keys. Iteration order is unspecified in most (Python dicts preserve insertion order since 3.7; Go maps are randomized; Rust HashMap is randomized for DoS resistance). Keys must be hashable/comparable.

Key Pattern — count frequencies with a default

Python 
from collections import Counter
c = Counter("banana")
print(c.most_common(1))   # [('a', 3)]
print(c["z"])             # 0 (missing key → 0, no KeyError)
[('a', 3)] 0
Go 
counts := map[rune]int{}
for _, r := range "banana" {
    counts[r]++ // zero value 0 is auto-created
}
fmt.Println(counts['a']) // 3
v, ok := counts['z']     // comma-ok idiom
fmt.Println(v, ok)       // 0 false
3 0 false
TypeScript 
const counts = new Map<string, number>();
for (const ch of "banana") {
  counts.set(ch, (counts.get(ch) ?? 0) + 1);
}
console.log(counts.get("a")); // 3
console.log(counts.has("z")); // false
3 false
Rust 
use std::collections::HashMap;
let mut counts: HashMap<char, i32> = HashMap::new();
for ch in "banana".chars() {
    *counts.entry(ch).or_insert(0) += 1;
}
println!("{}", counts[&'a']);          // 3
println!("{:?}", counts.get(&'z'));    // None
3 None
C++ 
#include <unordered_map>
#include <iostream>
std::unordered_map<char,int> counts;
for (char ch : std::string("banana")) counts[ch]++; // op[] inserts 0
std::cout << counts['a'] << "\n";         // 3
std::cout << counts.count('z') << "\n";   // 0 (don't use [] to test!)
3 0

1

#1 Pitfall per Language
  • Python: Using a mutable value (e.g. a list) as a dict key — raises TypeError: unhashable type.
  • Go: Relying on map iteration order — Go randomizes it deliberately; sort the keys if you need determinism.
  • TS: Using a plain object as a "map" and getting prototype keys (__proto__, toString) — prefer Map.
  • Rust: Holding a & from .get() while trying to .insert() — borrow checker rejects the overlapping borrows.
  • C++: Using map[key] to test existence — operator[] inserts a default-constructed entry. Use .count() / .contains() (C++20).
2. Stacks & Queues Collections rep 3 • due today
Due today • 15 days since last review
Quick Recall: A stack is LIFO (push/pop one end); a queue is FIFO. The big gotcha across languages: using a dynamic array as a queue with pop(0)/erase(begin()) is O(n). Use a real deque/ring buffer for O(1) at both ends.

Key Pattern — O(1) queue with a deque

Python 
from collections import deque
q = deque()
q.append(1); q.append(2)      # enqueue
print(q.popleft())            # 1  (O(1) dequeue)
stack = [10, 20]
print(stack.pop())            # 20 (list as stack is fine)
1 20
Go 
// idiomatic Go: slice as stack; for a queue use a head index
q := []int{1, 2}
head := 0
front := q[head]; head++ // dequeue O(1) amortized
fmt.Println(front)       // 1
stk := []int{10, 20}
top := stk[len(stk)-1]; stk = stk[:len(stk)-1]
fmt.Println(top)         // 20
1 20
TypeScript 
// Array.shift() is O(n); for hot paths use a head pointer
const stack: number[] = [10, 20];
console.log(stack.pop());     // 20
const q = [1, 2];
console.log(q.shift());       // 1 (fine for small queues)
20 1
Rust 
use std::collections::VecDeque;
let mut q: VecDeque<i32> = VecDeque::new();
q.push_back(1); q.push_back(2);
println!("{:?}", q.pop_front()); // Some(1)
let mut stk = vec![10, 20];
println!("{:?}", stk.pop());     // Some(20)
Some(1) Some(20)
C++ 
#include <stack>
#include <queue>
std::queue<int> q; q.push(1); q.push(2);
std::cout << q.front() << "\n"; q.pop(); // 1
std::stack<int> s; s.push(10); s.push(20);
std::cout << s.top() << "\n"; s.pop();   // 20
1 20

  • Python: Using list.pop(0) as a queue — it's O(n). Reach for deque.

3. Logging Across Languages Testing & Observability rep 2 • due today
Due today • 6 days since last review
Quick Recall: Log with levels (DEBUG/INFO/WARN/ERROR), to stderr (not stdout) so logs don't corrupt piped data, and prefer structured (key=value / JSON) over free text so logs are machine-parseable.

Key Pattern — leveled logger

4. Type Systems Deep Dive Testing & Type Safety rep 2 • due today
Due today • 6 days since last review
Quick Recall: Static (Go, Rust, C++, TS) vs dynamic (Python) typing; nominal (Go, Rust, C++) vs structural (TS, Python protocols). Soundness: Rust/C++ are (mostly) sound; TS is deliberately unsound in places (e.g. array covariance) to stay pragmatic.

Key Pattern — structural vs nominal compatibility

Python 
from typing import Protocol
class Sized(Protocol):
    def __len__(self) -> int: ...
def total(x: Sized) -> int: return len(x)
print(total([1,2,3]))  # 3 — list structurally matches
3
Go 
// interfaces are satisfied structurally, types are nominal
type Stringer interface{ String() string }
type ID int
func (i ID) String() string { return fmt.Sprintf("#%d", i) }
var s Stringer = ID(7)        // implicit, no "implements"
fmt.Println(s.String())       // #7
#7
TypeScript 
interface Point { x: number; y: number; }
function show(p: Point) { return `${p.x},${p.y}`; }
const obj = { x: 1, y: 2, z: 3 };  // extra prop OK via variable
console.log(show(obj));            // "1,2" — structural match
1,2
Rust 
trait Area { fn area(&self) -> f64; }
struct Sq(f64);
impl Area for Sq { fn area(&self) -> f64 { self.0 * self.0 } }
fn report(a: &dyn Area) -> f64 { a.area() }  // nominal: must impl
fn main() { println!("{}", report(&Sq(3.0))); } // 9
9
C++ 
#include <concepts>

template<HasSize T> auto sz(const T& t){ return t.size(); }
// duck-typed at compile time via concepts
int main(){ std::cout << sz(std::string("hi")); } // 2

5. Concurrent Data Structures Performance rep 1 • due today
Due today • 1 day since last review
Quick Recall: Sharing mutable state across threads needs synchronization. Choices: coarse mutex (simple, can be a bottleneck), sharded/striped locks, or purpose-built concurrent maps/queues. Prefer message passing where you can to avoid shared state entirely.

Key Pattern — thread-safe counter map

Python 
import threading
lock = threading.Lock()
counts = {}
def incr(k):
    with lock:
        counts[k] = counts.get(k, 0) + 1
incr("a"); incr("a")
print(counts)  # {'a': 2}
{'a': 2}
Go 
// sync.Map is built for concurrent access without manual locks
var m sync.Map
m.Store("a", 1)
m.Store("b", 2)
v, _ := m.Load("a")
fmt.Println(v) // 1
1
TypeScript 
// single-threaded event loop: a plain Map is "concurrency-safe"
// across awaits as long as you don't yield mid-update.
const counts = new Map<string, number>();
const incr = (k: string) => counts.set(k, (counts.get(k) ?? 0) + 1);
incr("a"); incr("a");
console.log(counts.get("a")); // 2
2
Rust 
use std::sync::{Arc, Mutex};
use std::collections::HashMap;
let m = Arc::new(Mutex::new(HashMap::new()));
{
    let mut g = m.lock().unwrap();
    *g.entry("a").or_insert(0) += 1;
}
println!("{:?}", m.lock().unwrap().get("a")); // Some(1)
Some(1)
C++ 
#include <mutex>
#include <unordered_map>
std::mutex mu;
std::unordered_map<std::string,int> counts;
void incr(const std::string& k){
    std::lock_guard<std::mutex> lk(mu);
    counts[k]++;
}
// incr("a"); incr("a"); -> counts["a"] == 2
counts["a"] == 2

6. Strings: Mutability, Unicode & Common Ops Collections rep 3 • 2 days overdue
2 days overdue • was due 2026-06-22
Quick Recall: Strings are immutable in Python/Java/TS; mutable (with care) in Rust (String) and C++ (std::string). "Length" is ambiguous: bytes vs code points vs grapheme clusters. UTF-8 indexing is not O(1) by character.

Key Pattern — counting characters, not bytes

Python 
s = "café"          # str is a sequence of code points

Section 1 — Concurrency Patterns III

Distributing work across workers and coordinating without locks.

Worker Pools & Task Queues Concurrency Patterns
Why it matters: spawning one thread per task doesn't scale — a fixed pool of workers pulling from a shared queue bounds resource use, keeps throughput high under load, and is the backbone of job systems, web servers, and data pipelines.

The pattern: a queue holds pending tasks; a fixed number of workers loop, pull a task, run it, and repeat. A bounded queue gives backpressure (producers block when full). You wait for completion with a counter/barrier, and signal "no more work" by closing the queue.

Example 1 — minimal pool that squares numbers

Python 
from concurrent.futures import ThreadPoolExecutor

def square(n): return n * n

with ThreadPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(square, range(6)))
print(results)  # [0, 1, 4, 9, 16, 25]
[0, 1, 4, 9, 16, 25]
Go 
jobs := make(chan int, 6)
results := make(chan int, 6)
var wg sync.WaitGroup
for w := 0; w < 4; w++ { // 4 workers
    wg.Add(1)
    go func() { defer wg.Done()
        for n := range jobs { results <- n * n }
    }()
}
for n := 0; n < 6; n++ { jobs <- n }
close(jobs)               // signal: no more work
wg.Wait(); close(results)
sum := 0; for r := range results { sum += r }
fmt.Println(sum)          // 55
55
TypeScript 
// concurrency-limited map (the Promise version of a pool)
async function pool<T, R>(items: T[], limit: number,
    fn: (x: T) => Promise<R>): Promise<R[]> {
  const out: R[] = []; let i = 0;
  const workers = Array.from({ length: limit }, async () => {
    while (i < items.length) { const idx = i++; out[idx] = await fn(items[idx]); }
  });
  await Promise.all(workers);
  return out;
}
const square = async (n: number) => n * n;
pool([0,1,2,3,4,5], 4, square).then(r => console.log(r));
[ 0, 1, 4, 9, 16, 25 ]
Rust 
// data parallelism with rayon's thread pool
use rayon::prelude::*;
fn main() {
    let r: Vec<i32> = (0..6).into_par_iter().map(|n| n * n).collect();
    println!("{:?}", r); // [0, 1, 4, 9, 16, 25]
}
[0, 1, 4, 9, 16, 25]
C++ 
#include <future>
#include <vector>
int square(int n){ return n*n; }
int main(){
    std::vector<std::future<int>> fs;
    for (int n=0;n<6;n++) fs.push_back(std::async(std::launch::async, square, n));
    for (auto& f : fs) std::cout << f.get() << " "; // 0 1 4 9 16 25
}
0 1 4 9 16 25

Example 4 — bounded async queue (Rust, tokio)

Rust 
use tokio::sync::mpsc;
#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::channel::<i32>(2); // bounded -> backpressure
    tokio::spawn(async move {
        for n in 0..6 { tx.send(n).await.unwrap(); }
    });
    let mut sum = 0;
    while let Some(n) = rx.recv().await { sum += n * n; }
    println!("{sum}"); // 55
}
55
Language comparison: Go and Rust bake the "queue + N workers" idiom into the runtime (channels, rayon/tokio). Python gives you batteries-included concurrent.futures — but use processes for CPU work because of the GIL. TypeScript has no real threads in the main runtime, so a "pool" is just bounded-concurrency over promises (or real worker_threads for CPU work). C++ has no standard pool — you compose std::async/std::thread + a queue yourself, or pull in a library.
Common mistakes per language
✗ Python: ThreadPoolExecutor for CPU-bound work — the GIL serializes it.
✓ Use ProcessPoolExecutor for CPU work; threads only for I/O.
✗ Go: never close(jobs) — workers ranging over the channel block forever (goroutine leak).
✓ Close the jobs channel once all work is submitted; range then exits cleanly.
✗ TS: await Promise.all(items.map(fn)) for 10k items — fires all at once, no limit.
✓ Bound concurrency with a worker loop or p-limit.
✗ Rust: unbounded mpsc::unbounded_channel() under load — memory blows up.
✓ Use a bounded channel so producers wait (backpressure).
✗ C++: spawning a std::thread per task and never joining — std::terminate.
✓ Reuse a fixed pool; join every thread (or detach deliberately).
Key Rules
  • A pool's size should match the workload: ~#cores for CPU-bound, much higher for I/O-bound.
  • Always have a clean shutdown signal (close the queue) so workers terminate.
  • Bounded queues give you backpressure; unbounded queues hide overload until you run out of memory.

Lock-Free Data Structures & Atomics Concurrency Patterns
Why it matters: atomics let multiple threads update shared state without locks, avoiding lock contention and deadlocks. They're the foundation of counters, flags, and lock-free queues in high-throughput systems — but memory ordering is subtle and easy to get wrong.

An atomic operation completes indivisibly — no other thread observes a half-done state. Key primitives: atomic load/store, fetch-and-add, and compare-and-swap (CAS): "set X to B only if it currently equals A." Lock-free algorithms loop on CAS until they win. Memory ordering (relaxed / acquire-release / sequentially-consistent) controls how surrounding reads and writes may be reordered.

Example 1 — atomic counter incremented from many threads

Python 
# Python has no user-facing atomics; a Lock is the idiom.
import threading
counter = 0
lock = threading.Lock()
def bump():
    global counter
    for _ in range(100000):
        with lock: counter += 1
ts = [threading.Thread(target=bump) for _ in range(4)]
[t.start() for t in ts]; [t.join() for t in ts]
print(counter)  # 400000
400000
Go 
import "sync/atomic"
var counter atomic.Int64
var wg sync.WaitGroup
for w := 0; w < 4; w++ {
    wg.Add(1)
    go func() { defer wg.Done()
        for i := 0; i < 100000; i++ { counter.Add(1) }
    }()
}
wg.Wait()
fmt.Println(counter.Load()) // 400000
400000
TypeScript 
// shared across Web Workers via SharedArrayBuffer + Atomics
const sab = new SharedArrayBuffer(4);
const view = new Int32Array(sab);
Atomics.add(view, 0, 1);     // atomic fetch-and-add

42
Rust 
use std::sync::atomic::{AtomicUsize, Ordering};

Example 2 — compare-and-swap loop (Rust)

Rust 
use std::sync::atomic::{AtomicI64, Ordering};
let max = AtomicI64::new(0);
fn record(max: &AtomicI64, v: i64) {
    let mut cur = max.load(Ordering::Relaxed);
    while v > cur {
        match max.compare_exchange_weak(cur, v,
                Ordering::SeqCst, Ordering::Relaxed) {
            Ok(_) => break,
            Err(actual) => cur = actual, // retry with fresh value
        }
    }
}
record(&max, 5); record(&max, 3); record(&max, 9);
println!("{}", max.load(Ordering::Relaxed)); // 9
9

Example 3 — CAS-based atomic max (C++)

C++ 
#include <atomic>
std::atomic<long> mx{0};
void record(long v){
    long cur = mx.load();
    while (v > cur && !mx.compare_exchange_weak(cur, v)) { /* cur updated */ }
}
// record(5); record(3); record(9); -> mx == 9

Go

Language comparison: Rust and C++ expose the full memory model (Relaxed/Acquire/Release/SeqCst) — powerful and dangerous. Go offers a curated, sequentially-consistent sync/atomic with typed wrappers and no orderings to pick. TypeScript's Atomics only works on a SharedArrayBuffer across workers. Python deliberately has no atomics: the GIL plus a Lock is the model, so reach for queues/locks instead.
Common mistakes per language
✗ Python: assuming counter += 1 is atomic under the GIL — it's three bytecodes; threads can interleave.
✓ Guard with a Lock (or use a queue.Queue).
✗ Go: mixing atomic and non-atomic access to the same variable.
✓ Access an atomic variable only through atomic ops.
✗ TS: using Atomics on a normal ArrayBuffer — throws; it needs SharedArrayBuffer.
✓ Allocate a SharedArrayBuffer and share it with the worker.
✗ Rust: defaulting to Relaxed when you need a happens-before edge to other data.
✓ Use Acquire/Release (or SeqCst) to publish/observe related writes.
✗ C++: reusing the expected value after a failed compare_exchange without noticing it was overwritten.
✓ On failure, expected holds the current value — loop with it.
Key Rules
  • CAS loops are the lock-free workhorse: read, compute, compare-and-swap, retry on failure.
  • Default to the strongest ordering (SeqCst) and only relax it when you've proven correctness.
  • Lock-free isn't automatically faster — benchmark; sometimes a plain mutex wins.

Section 2 — Type-Level Patterns

Using the type system to make illegal states unrepresentable.

Phantom Types & Type-Level Programming Advanced Type Patterns
Why it matters: a phantom type is a type parameter that doesn't appear in any runtime field — it exists only to let the compiler distinguish otherwise-identical values (e.g. Distance<Meters> vs Distance<Feet>), catching unit and state mix-ups at compile time with zero runtime cost.

The classic motivating bug: adding meters to feet, or passing an unvalidated input where a validated one is required. A phantom marker tags the value's "kind" in the type, so the compiler refuses the mismatch. The marker carries no data — it's erased at runtime.

Example 1 — meters vs feet that won't add

Python 
from typing import Generic, TypeVar
U = TypeVar("U")  # phantom: never stored, only checked by a type checker
class Distance(Generic[U]):
    def __init__(self, v: float): self.v = v

class Meters: ...
class Feet: ...
m: Distance[Meters] = Distance(5.0)
f: Distance[Feet]   = Distance(3.0)
# a checker (mypy/pyright) flags Distance[Meters] vs Distance[Feet] mixups
print(m.v, f.v)  # 5.0 3.0
5.0 3.0
Go 
// Go has no phantom generics in values; use distinct named types
type Meters float64
type Feet float64
var m Meters = 5
var f Feet = 3
// m + f  -> compile error: mismatched types Meters and Feet
fmt.Println(m, f) // 5 3
5 3
TypeScript 
// phantom via an unused branded property
type Distance<U> = number & { readonly __unit: U };
const meters = (n: number) => n as Distance<"m">;
const feet   = (n: number) => n as Distance<"ft">;
const m = meters(5);
const f = feet(3);
// const bad: Distance<"m"> = f; -> error: 'ft' not assignable to 'm'
console.log(m + 0, f + 0); // 5 3
5 3
Rust 
use std::marker::PhantomData;
struct Distance<U> { v: f64, _u: PhantomData<U> }
impl<U> Distance<U> { fn new(v: f64) -> Self { Self{v, _u: PhantomData} } }
struct Meters; struct Feet;
fn main() {
    let m: Distance<Meters> = Distance::new(5.0);
    let f: Distance<Feet>   = Distance::new(3.0);
    // m.v + f would compile (both f64), but a typed add() per unit won't mix
    println!("{} {}", m.v, f.v); // 5 3
}
5 3
C++ 
template<class Tag> struct Distance { double v; };
struct Meters; struct Feet;
int main(){
    Distance<Meters> m{5.0};
    Distance<Feet>   f{3.0};
    // m = f; -> compile error: distinct types
    std::cout << m.v << " " << f.v; // 5 3
}
5 3

Example 2 — validated vs raw user input (Rust)

Rust 
use std::marker::PhantomData;
struct Raw; struct Validated;
struct Email<S>(String, PhantomData<S>);
impl Email<Raw> {
    fn new(s: &str) -> Self { Email(s.into(), PhantomData) }
    fn validate(self) -> Option<Email<Validated>> {
        if self.0.contains('@') { Some(Email(self.0, PhantomData)) } else { None }
    }
}
fn send(e: Email<Validated>) { println!("sending to {}", e.0); }
fn main() {
    let e = Email::<Raw>::new("a@b.com").validate().unwrap();
    send(e); // send only accepts Validated — Raw won't compile
}
sending to a@b.com

Example 3 — phantom unit in TypeScript that blocks a mismatch

TypeScript 
type Tagged<T, Tag> = T & { readonly __tag: Tag };
type UserId = Tagged<number, "UserId">;
type PostId = Tagged<number, "PostId">;
function getUser(id: UserId) { return `user ${id}`; }
const uid = 1 as UserId;
const pid = 2 as PostId;
console.log(getUser(uid));   // ok
// getUser(pid); -> Error: PostId not assignable to UserId
user 1
Language comparison: Rust is the natural home for phantom types — PhantomData<T> exists precisely to "use" an unused type parameter while contributing zero bytes. C++ does it with empty tag types and template parameters. TypeScript fakes it with branded/intersection types (purely compile-time). Go has no generic phantom-in-value pattern, so you lean on distinct named types. Python's phantom Generic[U] is advisory only — checked by mypy/pyright, not the runtime.
Common mistakes per language
✗ Python: expecting Distance[Meters] to block Distance[Feet] at runtime.
✓ It only helps under a static checker; nothing enforces it when the program runs.
✗ Go: using a type alias (type Meters = float64) — aliases are interchangeable.
✓ Use a type definition (type Meters float64) to get a distinct type.
✗ TS: branding with a real runtime property — bloats objects and leaks the tag.
✓ Brand with a phantom readonly __tag that's never actually assigned.
✗ Rust: omitting PhantomData — "parameter U is never used" error.
✓ Add a PhantomData<U> field (zero-sized) to "use" the parameter.
✗ C++: giving the tag a member, so it stops being empty / zero-overhead.
✓ Keep tag types as empty structs used only as template arguments.
Key Rules
  • A phantom type carries no data — it exists purely to make two structurally-identical values incompatible.
  • Use it to encode state (raw vs validated) or units so misuse is a compile error.
  • It's zero-cost in Rust/C++/Go and free at runtime in TS; in Python it needs an external checker.

Newtype / Wrapper Pattern Advanced Type Patterns
Why it matters: wrapping a primitive in a one-field type (e.g. UserId(i64) instead of bare i64) gives it a distinct identity, lets you attach methods/invariants, and prevents accidentally passing the wrong "kind" of number or string — usually with zero runtime overhead.

A "newtype" is a thin wrapper around an existing type that the compiler treats as a brand-new, incompatible type. Compared to a phantom type, the newtype's parameter is the wrapped value — it's about identity, methods, and invariants on a single concrete type rather than a marker.

Example 1 — a UserId you can't confuse with a plain int

Python 
from typing import NewType
UserId = NewType("UserId", int)
def greet(uid: UserId) -> str: return f"hi {uid}"
u = UserId(7)
print(greet(u))   # hi 7
# greet(7) -> mypy flags: int is not UserId (runtime: NewType is identity)
hi 7
Go 
type UserID int64
func (u UserID) String() string { return fmt.Sprintf("U%d", u) } // attach methods
func greet(u UserID) string { return "hi " + u.String() }
fmt.Println(greet(UserID(7))) // hi U7
// greet(int64(7)) -> compile error: distinct types
hi U7
TypeScript 
type UserId = number & { readonly _brand: "UserId" };
const userId = (n: number) => n as UserId;
function greet(u: UserId) { return `hi ${u}`; }
console.log(greet(userId(7))); // hi 7
// greet(7) -> Error: number not assignable to UserId
hi 7
Rust 
struct UserId(i64);                       // tuple-struct newtype
impl UserId { fn greet(&self) -> String { format!("hi {}", self.0) } }
fn main() {
    let u = UserId(7);
    println!("{}", u.greet());            // hi 7
    // takes_userid(7i64) -> type error: i64 is not UserId
}
hi 7
C++ 
struct UserId { long value; explicit UserId(long v):value(v){} };
std::string greet(UserId u){ return "hi " + std::to_string(u.value); }
int main(){
    std::cout << greet(UserId(7)); // hi 7
    // greet(7); -> error: explicit ctor blocks implicit long->UserId
}
hi 7

Example 2 — enforce an invariant in the constructor (Rust)

Rust 
struct Percent(u8); // invariant: 0..=100
impl Percent {
    fn new(v: u8) -> Option<Percent> {
        if v <= 100 { Some(Percent(v)) } else { None }
    }
}
fn main() {
    println!("{:?}", Percent::new(150).map(|p| p.0)); // None
    println!("{:?}", Percent::new(80).map(|p| p.0));  // Some(80)
}
None Some(80)

Example 3 — Deref so the wrapper "feels like" the inner type (Rust)

Rust 
use std::ops::Deref;
struct Name(String);
impl Deref for Name { type Target = String; fn deref(&self) -> &String { &self.0 } }
fn main() {
    let n = Name("ada".into());
    println!("{}", n.len());      // 3 — String methods via Deref coercion
    println!("{}", n.to_uppercase()); // ADA
}
3 ADA

Example 4 — money type with a strong typedef (C++)

C++ 
struct USD { long cents; };
USD operator+(USD a, USD b){ return USD{a.cents + b.cents}; }
int main(){
    USD total = USD{500} + USD{250};
    std::cout << total.cents; // 750
    // total + 5; -> error: no operator+ for USD and int
}
750
Language comparison: Rust's struct Meters(f64) is the canonical newtype, often paired with Deref for ergonomics. Go's type X int is the same idea and lets you hang methods on it. C++ wraps in a struct with an explicit constructor (to block implicit conversions). TypeScript brands a primitive (compile-time only). Python's NewType is the weakest — it's an identity function at runtime, enforced only by a type checker.
Common mistakes per language
✗ Python: thinking NewType creates a real subclass — it doesn't; isinstance won't see it.
✓ Use it for static distinction; use a @dataclass/class if you need runtime behavior.
✗ Go: using a type alias instead of a definition — loses the distinct identity.
✓ type UserID int64 (no =) creates a new, method-attachable type.
✗ TS: forgetting the brand is fictional — an unguarded as can smuggle in any number.
✓ Funnel construction through a single smart constructor that validates.
✗ Rust: deriving Deref on everything — can leak the inner type and blur the abstraction.
✓ Add Deref only when transparent access is genuinely wanted.
✗ C++: a non-explicit constructor — allows silent implicit conversions, defeating the wrapper.
✓ Mark single-arg constructors explicit.
Key Rules
  • Newtype = identity + a place to hang methods and invariants; usually zero runtime cost.
  • Validate in the constructor so an instance is, by construction, always valid.
  • Block implicit conversions (Go definitions, C++ explicit, TS branding) or the wrapper buys nothing.

State Machines via Types Advanced Type Patterns
Why it matters: encoding states and transitions in the type system makes invalid transitions unrepresentable. Instead of guarding every method with "if not open, throw," the compiler refuses to even call read() on a closed connection. Bugs become compile errors.

Two common encodings: (1) a sum type / discriminated union where each variant carries the data valid in that state, matched exhaustively; and (2) the typestate pattern where each state is a distinct type and transitions consume the old state to produce the new one, so a stale state can't be reused.

Example 1 — a traffic light as a sum type with exhaustive matching

Python 
from enum import Enum, auto
class Light(Enum): RED=auto(); GREEN=auto(); YELLOW=auto()
def next_light(l: Light) -> Light:
    match l:
        case Light.RED:    return Light.GREEN
        case Light.GREEN:  return Light.YELLOW
        case Light.YELLOW: return Light.RED
print(next_light(Light.RED))  # Light.GREEN
Light.GREEN
Go 
// interface-based state: each state is a type with a Next method
type State interface{ Next() State; Name() string }
type Red struct{}; type Green struct{}; type Yellow struct{}
func (Red) Next() State { return Green{} };    func (Red) Name() string { return "RED" }
func (Green) Next() State { return Yellow{} }; func (Green) Name() string { return "GREEN" }
func (Yellow) Next() State { return Red{} };   func (Yellow) Name() string { return "YELLOW" }
func main(){ var s State = Red{}; fmt.Println(s.Next().Name()) } // GREEN
GREEN
TypeScript 
type Light = "red" | "green" | "yellow";
function next(l: Light): Light {
  switch (l) {
    case "red":    return "green";
    case "green":  return "yellow";
    case "yellow": return "red";
    // exhaustive: adding a state without a case is a compile error
  }
}
console.log(next("red")); // green
green
Rust 
enum Light { Red, Green, Yellow }
fn next(l: Light) -> Light {
    match l {
        Light::Red => Light::Green,
        Light::Green => Light::Yellow,
        Light::Yellow => Light::Red, // match must be exhaustive
    }
}
fn main(){ println!("{}", matches!(next(Light::Red), Light::Green)); } // true
true
C++ 
#include <variant>
struct Red{}; struct Green{}; struct Yellow{};
using Light = std::variant<Red, Green, Yellow>;
Light next(const Light& l){
    return std::visit([](auto s) -> Light {
        using T = decltype(s);
        if constexpr (std::is_same_v<T,Red>)   return Green{};
        else if constexpr (std::is_same_v<T,Green>) return Yellow{};
        else return Red{};
    }, l);
}
int main(){ auto n = next(Red{}); std::cout << n.index(); } // 1 (Green)
1

Example 2 — typestate: read() compiles only on an open connection (Rust)

Rust 
struct Open; struct Closed;
struct Conn<S> { url: String, _s: std::marker::PhantomData<S> }
impl Conn<Closed> {
    fn open(url: &str) -> Conn<Open> {
        Conn { url: url.into(), _s: std::marker::PhantomData }
    }
}
impl Conn<Open> {
    fn read(&self) -> String { format!("data from {}", self.url) }
    fn close(self) -> Conn<Closed> {
        Conn { url: self.url, _s: std::marker::PhantomData }
    }
}
fn main() {
    let c = Conn::<Closed>::open("db://x");
    println!("{}", c.read()); // ok — Open
    let _c = c.close();       // now Closed; c.read() would not compile
}
data from db://x

TypeScript 
type Conn =
  | { kind: "closed" }
  | { kind: "open"; socket: number }
  | { kind: "error"; reason: string };

function describe(c: Conn): string {
  switch (c.kind) {
    case "closed": return "closed";

Example 4 — door state machine (Python, match on dataclasses)

Python 
from dataclasses import dataclass
@dataclass class Locked: key: str
@dataclass class Open: pass
@dataclass class Closed: pass
Door = Locked | Open | Closed
def act(d: Door) -> str:
    match d:
        case Locked(key=k): return f"need key {k}"
        case Open():        return "walk through"
        case Closed():      return "open it"
print(act(Locked("brass")))  # need key brass
need key brass
Language comparison: Rust is uniquely strong here: typestate consumes the old value (move semantics) so a stale state is a compile error, and match is exhaustive. C++ approximates with std::variant + std::visit. TypeScript's discriminated unions + exhaustive switch are excellent and idiomatic. Go uses an interface per state (no exhaustiveness check). Python relies on Enum/dataclasses + match — expressive but checked only by a static analyzer, not the runtime.
Common mistakes per language
✗ Python: a match with no fallback that silently returns None for an unhandled state.
✓ Add case _: raise ValueError(...) or run a checker that warns on non-exhaustive matches.
✗ Go: a type switch missing a state — no compile-time exhaustiveness; bug slips through.
✓ Add a default: that panics on unknown states, and test every variant.
✗ TS: using a non-discriminated union — you can't narrow it cleanly.
✓ Give every variant a literal kind tag and rely on exhaustive switch.
✗ Rust: taking &mut self for a transition — lets the old state still be used.
✓ Take self by value so the transition consumes the prior state.
✗ C++: calling std::get<Open> on a variant in the wrong state — throws bad_variant_access.
✓ Use std::visit (or get_if) so each state is handled explicitly.
Key Rules
  • Prefer making illegal transitions unrepresentable over runtime guards.
  • Sum types (enums/unions) handle "value is in one of N states"; typestate handles "value moves through states over time."
  • Lean on exhaustive matching so adding a state forces you to handle it everywhere.
