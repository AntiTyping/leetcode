1. Python: Comprehensions & Dataclasses Language Deep Dive rep 1 • 21 days since review
2nd review (overdue 15 days)
Quick Recall: A comprehension builds a collection declaratively from an iterable with an optional filter and transform — Python's signature feature. Everyone else expresses the same idea as an iterator chain (filter then map). Separately, a @dataclass auto-generates __init__/__repr__/__eq__ from field declarations; the analog is a Go struct, a TS class/interface, a Rust #[derive], and a C++ aggregate.

Key Pattern — squares of the even numbers in 0..9

Python 
print([x*x for x in range(10) if x % 2 == 0])   # [0, 4, 16, 36, 64]
squares = {x: x*x for x in range(4)}             # dict comprehension
Go 
var out []int                          // no comprehensions; explicit loop
for x := 0; x < 10; x++ { if x%2 == 0 { out = append(out, x*x) } }
fmt.Println(out)                       // [0 4 16 36 64]
TypeScript 
const out = [...Array(10).keys()].filter(x => x % 2 === 0).map(x => x * x);
console.log(out);                      // [0, 4, 16, 36, 64]
Rust 
let out: Vec<i32> = (0..10).filter(|x| x % 2 == 0).map(|x| x * x).collect();
println!("{:?}", out);                 // [0, 4, 16, 36, 64]
C++ 
#include <ranges><vector>
auto v = std::views::iota(0, 10)
       | std::views::filter([](int x){ return x % 2 == 0; })
       | std::views::transform([](int x){ return x * x; });
for (int x : v) std::cout << x << " ";   // 0 4 16 36 64
Challenge (all 5 languages):
1
Flatten a matrix keeping only even values with a single nested comprehension in Python, then replicate it as an iterator chain in the other four. Then define an immutable Point{x, y} record with a computed magnitude using a dataclass / struct / #[derive] / aggregate, and confirm structural equality of two equal points.
#1 Pitfall per Language:
  • Python: a comprehension's variable no longer leaks in Py3, but a generator expression is lazy — consuming it twice yields nothing the second time; and mutable dataclass defaults need field(default_factory=...).
  • Go: no comprehension syntax at all — don't fake one with clever tricks; a plain for loop with append is idiomatic.
  • TypeScript: chained filter/map allocate an intermediate array each step — fine for small data, but reach for a single reduce or a generator on hot paths.
  • Rust: forgetting .collect() leaves you with a lazy iterator, not a Vec; the target type usually must be annotated.
  • C++: a views pipeline is lazy and borrows its source — the underlying range must outlive the view, and you materialize with ranges::to (C++23) or a manual loop.
2. Rust: Lifetimes & Borrowing Language Deep Dive rep 1 • 21 days since review
2nd review (overdue 15 days)
Quick Recall: A lifetime is a compile-time proof that a reference never outlives the data it points to. Rust makes you name the relationship (<'a>) when the compiler can't infer it — e.g. a function returning one of several input references. The other languages solve the same dangling-reference problem differently: GC (Go/Python/TS) keeps data alive as long as a reference exists, while C++ leaves lifetime entirely to you (dangling references are UB).

Key Pattern — return the longer of two string references

Rust 
// 'a ties the output's lifetime to BOTH inputs: result valid as long as both are
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str { if a.len() > b.len() { a } else { b } }
println!("{}", longest("hello", "hi"));   // hello
C++ 
// No lifetimes in the type system; YOU guarantee the args outlive the result:
const std::string& longest(const std::string& a, const std::string& b) {
    return a.size() > b.size() ? a : b; }
// Passing a temporary here would leave a DANGLING reference (UB).
Go 
// GC-managed; returning a value keeps it alive automatically, no lifetime needed:
func longest(a, b string) string { if len(a) > len(b) { return a }; return b }
fmt.Println(longest("hello", "hi"))       // hello
Python 
# Names are references; the GC keeps an object alive while any name binds it:
def longest(a, b): return a if len(a) > len(b) else b
print(longest("hello", "hi"))             # hello
TypeScript 
// GC-managed, no lifetimes; the returned string is reachable, so it survives:
const longest = (a: string, b: string): string => a.length > b.length ? a : b;
console.log(longest("hello", "hi"));      // hello
Challenge (all 5 languages):
1
Write a function that returns the first word of a string as a borrowed slice/view (Rust &str, C++ std::string_view, Go/TS/Python a substring). In Rust and C++ show a case that would dangle if you returned a reference into a local temporary, and fix it. Then define a struct that holds such a reference and observe how Rust forces a lifetime parameter on the struct while C++ silently allows a dangling member.
#1 Pitfall per Language:
  • Rust: trying to return a reference to a value created inside the function — it's dropped at the end of scope; return an owned String instead.
  • C++: returning const T& to a temporary or local — compiles, then dangles; prefer returning by value or string_view only into data you know outlives the call.
  • Go: taking &localVar is safe (it escapes to the heap), but capturing a loop variable by reference before Go 1.22 shared one address across iterations.
  • Python: no dangling refs, but holding a reference (in a cache/closure) unintentionally keeps large objects alive — a logical leak.
  • TypeScript: same GC caveat — closures capturing large objects, un-cleared timers, and listeners pin memory.
3. FFI & Interop Advanced Patterns rep 1 • 20 days since review
2nd review (overdue 14 days)
Quick Recall: FFI lets your code call into a native library through the C ABI — the lingua franca every language can speak. The recurring hazards are the same everywhere: matching the exact C types, who owns/frees allocations across the boundary, string encoding (C strings are NUL-terminated), and that a crash in native code takes down the whole process. Rust and C++ call C almost natively; Python/Go/TS marshal data across a heavier bridge.

Key Pattern — call C's int abs(int) from each language

Python 
import ctypes
libc = ctypes.CDLL("libc.so.6")        # or "libc.dylib" on macOS
libc.abs.restype = ctypes.c_int         # ALWAYS declare arg/return types
print(libc.abs(-42))                    # 42
Rust 
use std::os::raw::c_int;
extern "C" { fn abs(input: c_int) -> c_int; }   // link against libc
fn main() { unsafe { println!("{}", abs(-42)); } } // 42  (calls are `unsafe`)
C++ 
#include <cstdlib><iostream>
// C headers are directly usable; extern "C" disables name mangling when EXPOSING APIs
extern "C" int square(int x) { return x * x; }   // callable from other languages
int main(){ std::cout << std::abs(-42); }        // 42
Go 
/*
#include <stdlib.h>
*/
import "C"                             // cgo: the comment above is real C
import "fmt"
func main() { fmt.Println(C.abs(-42)) } // 42  (cgo has real call-boundary cost)
TypeScript 
// Node has no raw FFI; use a native addon (N-API), koffi, or WebAssembly:
import koffi from "koffi";
const lib = koffi.load("libc.so.6");
const abs = lib.func("int abs(int)");
console.log(abs(-42));                  // 42
Challenge (all 5 languages):
1
Call a C function that takes a const char* and returns a heap-allocated char* the caller must free. Correctly encode the input string to a NUL-terminated buffer, read the result, and free it across the boundary without leaking or double-freeing. Note where each language forces you to think about ownership (Rust CString, Go C.CString+C.free, ctypes c_char_p).
#1 Pitfall per Language:
  • Python: forgetting to set restype/argtypes — ctypes assumes int, silently corrupting pointer or double returns on 64-bit.
  • Go: passing a Go pointer into C that C retains violates the cgo pointer rules and can be moved by the GC; cgo calls also block a whole OS thread.
  • Rust: every extern call is unsafe; you must uphold the C function's contract, and Rust strings aren't NUL-terminated — use CString.
  • C++: forgetting extern "C" when exposing an API leaves the symbol name-mangled and unfindable from C/other languages.
  • TypeScript/Node: a native addon compiled against one Node ABI won't load on another; a crash in native code takes down the entire event loop.
4. Go: Goroutine Scheduling & Channel Patterns Language Deep Dive rep 1 • 15 days since review
2nd review (overdue 14 days)
Quick Recall: A goroutine is a cheap green thread multiplexed by Go's runtime onto a small pool of OS threads (the M:N scheduler); channels hand values between them, the CSP way. The universal fan-out/fan-in shape — launch N workers reading a jobs channel, collect on a results channel — maps to a thread pool + queue elsewhere: Python threads/asyncio, JS Promises, Rust threads+mpsc or tokio tasks, C++ std::async/threads + a future.

Key Pattern — spawn a producer, receive its value on the main thread

Go 
ch := make(chan int)
go func() { ch <- 42 }()               // goroutine sends
fmt.Println(<-ch)                       // 42  (receive blocks until sent)
// fan-out: for i:=0;i<n;i++ { go worker(jobs, results) }
Python 
import threading, queue
q = queue.Queue()
threading.Thread(target=lambda: q.put(42)).start()
print(q.get())                          # 42  (or asyncio for I/O-bound fan-out)
TypeScript 
// Single-threaded event loop; a Promise is the async handoff:
const p = new Promise<number>(res => setTimeout(() => res(42), 0));
console.log(await p);                    // 42  (Promise.all for fan-in)
Rust 
use std::sync::mpsc; use std::thread;
let (tx, rx) = mpsc::channel();
thread::spawn(move || tx.send(42).unwrap());
println!("{}", rx.recv().unwrap());      // 42  (or tokio::spawn + tasks)
C++ 
#include <future>
auto fut = std::async(std::launch::async, []{ return 42; });
std::cout << fut.get();                   // 42  (or a thread pool + queue)
Challenge (all 5 languages):
1
Build a worker pool: feed 100 jobs to N=4 workers and collect all results, then shut down cleanly with no leaked workers. In Go use a jobs channel closed by the producer and a sync.WaitGroup; elsewhere use a bounded queue + thread pool (or an async task group) and prove every job ran exactly once.
#1 Pitfall per Language:
  • Go: a leaked goroutine blocked forever on a channel is a silent memory/resource leak; sending on a closed channel panics; range over a channel needs the sender to close it.
  • Python: the GIL means threads don't parallelize CPU work — use multiprocessing or asyncio for I/O; a Queue is the safe handoff.
  • TypeScript: no shared-memory threads outside Workers; forgetting to await leaves a floating promise whose rejection is unhandled.
  • Rust: when all senders drop, recv() returns Err; sharing state across threads needs Arc<Mutex<_>>, and the borrow checker enforces it.
  • C++: a std::future from std::async blocks in its destructor; a bare std::thread not joined/detached calls std::terminate.
5. C++: Move Semantics & Smart Pointers Language Deep Dive rep 1 • 15 days since review
2nd review (overdue 14 days)
Quick Recall: A move transfers ownership of a resource instead of copying it, leaving the source in a valid-but-empty state — std::move casts an lvalue to an rvalue so the move constructor fires. Rust moves by default (the source is statically invalidated); Python/Go/TS have no moves (they copy handles and let the GC/refcount sort out lifetime). Smart pointers (unique_ptr/shared_ptr) encode ownership in the type.

Key Pattern — transfer ownership of a heap value

C++ 
#include <memory>
auto p = std::make_unique<int>(42);    // p uniquely owns the int
auto q = std::move(p);                  // ownership moves to q; p is now nullptr
std::cout << *q << " " << (p == nullptr); // 42 1
Rust 
let p = Box::new(42);                   // p owns the heap int
let q = p;                              // MOVE by default; using p now won't compile
println!("{}", q);                      // 42   (Rust makes move the default)
Go 
p := new(int); *p = 42                  // pointer to a GC'd int
q := p                                  // copies the pointer; both alias the same int
fmt.Println(*q, p == q)                 // 42 true  (no moves; GC owns lifetime)
Python 
p = [42]                               # a list on the heap, refcounted
q = p                                   # binds another name; same object, refcount 2
print(q[0], p is q)                     # 42 True  (no moves; refcounting)
TypeScript 
const p = { v: 42 };                    // object on the heap
const q = p;                            // copies the reference; both point to it
console.log(q.v, p === q);              // 42 true  (no moves; GC)
Challenge (all 5 languages):
1
Implement a Buffer that owns a heap array. In C++ give it a move constructor/assignment and delete the copies (rule of 5), and prove no deep copy happens on return; in Rust rely on default move + Drop; then show the GC languages simply share a reference. Instrument construction/move/copy/destruction to count how many allocations actually occur.
#1 Pitfall per Language:
  • C++: using a moved-from object as if it still holds its resource; declaring a move ctor without following the rule of 5; shared_ptr cycles leak (break with weak_ptr).
  • Rust: expecting to use a value after moving it — the borrow checker rejects it; Copy types (ints) are duplicated instead of moved.
  • Go: no moves means large struct copies are silent — pass pointers for big values, and beware that a slice copy shares the backing array.
  • Python: assignment never copies — b = a aliases; use copy.copy/copy.deepcopy when you need independence.
  • TypeScript: object/array assignment shares the reference; spread ({...o}) is only a shallow copy.
6. Input Validation & Sanitization Security Fundamentals rep 1 • 9 days since review
2nd review (overdue 9 days)
Quick Recall: Validate at the boundary, fail closed, and prefer an allowlist (accept known-good) over a denylist (block known-bad). Parse-and-coerce untrusted strings into typed, range-checked values as early as possible. The cardinal rule for injection: never build queries/commands/HTML by string concatenation — use parameterized queries, argument arrays, and context-aware escaping. Schema libraries (pydantic, zod, validator, serde) make this declarative.

Key Pattern — coerce and range-check an untrusted "age" input

Python 
def parse_age(s: str) -> int:
    n = int(s)                          # raises on non-numeric
    if not 0 <= n <= 150: raise ValueError("out of range")
    return n
# SQL: cur.execute("... WHERE id=%s", (uid,))  <- parameterized, never f-string
Go 
n, err := strconv.Atoi(s)
if err != nil || n < 0 || n > 150 { return fmt.Errorf("bad age") }
// SQL: db.Query("... WHERE id = $1", uid)  <- placeholders, not Sprintf
TypeScript 
import { z } from "zod";
const Age = z.coerce.number().int().min(0).max(150);
const age = Age.parse(input);           // throws with a precise error on bad input
Rust 
fn parse_age(s: &str) -> Result<u8, String> {
    let n: u8 = s.parse().map_err(|_| "not a number")?;
    if n > 150 { return Err("out of range".into()); }
    Ok(n)                               // u8 already rejects negatives
}
C++ 
#include <charconv>
int age; auto [p, ec] = std::from_chars(s.data(), s.data() + s.size(), age);
if (ec != std::errc{} || age < 0 || age > 150) throw std::invalid_argument("bad age");
// Use prepared statements (bind parameters), never build SQL by concatenation.
Challenge (all 5 languages):
1
Validate a user-registration payload (username: alphanumeric 3–20 chars; email: RFC-ish; age: 13–120) using a schema/validator, returning all field errors at once rather than the first. Then demonstrate a safe parameterized DB insert and show why "...VALUES('" + name + "')" is an injection hole.
#1 Pitfall per Language:
  • Python: f-string/`%`-building SQL or shell commands (os.system) with user input — use parameterized queries and subprocess.run([...]) with an argument list.
  • Go: using fmt.Sprintf to assemble SQL instead of placeholders; ignoring the err from a parse and proceeding with a zero value.
  • TypeScript: trusting req.body's TypeScript type at runtime — types are erased; you must validate with a schema, and beware prototype-pollution keys (__proto__).
  • Rust: reaching for .unwrap() on external input turns bad data into a panic/DoS — return a Result and handle it.
  • C++: atoi/scanf silently return 0 / read out of bounds on malformed input; prefer std::from_chars and bounds-checked buffers.

Text & String Algorithms

Strings look simple until you actually search, build, and measure them. Each language draws the line between "in the standard library" and "bring your own" in a different place: Python and Go are batteries-included for searching and splitting, Rust is precise but byte-oriented, and C++ hands you a fast mutable string but almost no higher-level text tooling. Unicode is where the naive mental model of "a string is an array of characters" breaks entirely.

String Searching & Substring Matching Text & String Algorithms
Why it matters: locating a substring, testing a prefix/suffix, splitting on a delimiter, and counting occurrences are the bread-and-butter of parsing, log processing, and routing. The traps are subtle: the "not found" sentinel differs per language (-1 vs None/Option vs npos), and indices may be measured in bytes, not characters.

Every stdlib gives you the same core verbs — find (first index of a substring), contains (boolean membership), and startsWith/endsWith. The differences are in the return type of "find" and in what unit the index counts. Get the sentinel wrong and you'll index into a string at a garbage position.

Example 1 — find, contains, and prefix/suffix tests on "hello"

Python 
s = "hello"
print(s.find("lo"), "lo" in s)         # 3 True   (find returns -1 if absent)
print(s.startswith("he"), s.endswith("lo"))  # True True
# s.index("zz") would RAISE ValueError; find returns -1 instead
3 True True True
Go 
import "strings"
s := "hello"
fmt.Println(strings.Index(s, "lo"), strings.Contains(s, "lo")) // 3 true
fmt.Println(strings.HasPrefix(s, "he"), strings.HasSuffix(s, "lo")) // true true
// Index returns a BYTE offset, and -1 when not found
3 true true true
TypeScript 
const s = "hello";
console.log(s.indexOf("lo"), s.includes("lo"));      // 3 true
console.log(s.startsWith("he"), s.endsWith("lo"));   // true true
// indexOf returns -1 when absent; index is a UTF-16 code-unit offset
3 true true true
Rust 
let s = "hello";
println!("{:?} {}", s.find("lo"), s.contains("lo")); // Some(3) true
println!("{} {}", s.starts_with("he"), s.ends_with("lo")); // true true
// find returns Option<usize> as a BYTE index (None if absent)
Some(3) true true true
C++ 
#include <string>
std::string s = "hello";
std::cout << s.find("lo") << " " << (s.find("lo") != std::string::npos); // 3 1
std::cout << " " << s.starts_with("he") << s.ends_with("lo"); // 11  (C++20)
// find returns npos (a huge size_t), NOT -1, when absent
3 1 11

Example 2 — split on a delimiter, join back, and count occurrences

Splitting and joining are stdlib one-liners in four of the five — C++ is the outlier with no built-in split or join.

Python 
parts = "a,b,c".split(",")            # ['a', 'b', 'c']
print("|".join(parts), "banana".count("a"))  # a|b|c 3
a|b|c 3
Go 
import "strings"
parts := strings.Split("a,b,c", ",")   // [a b c]
fmt.Println(strings.Join(parts, "|"), strings.Count("banana", "a")) // a|b|c 3
a|b|c 3
TypeScript 
const parts = "a,b,c".split(",");      // ['a', 'b', 'c']
const count = "banana".split("a").length - 1;  // no stdlib count
console.log(parts.join("|"), count);   // a|b|c 3
a|b|c 3
Rust 
let parts: Vec<&str> = "a,b,c".split(',').collect(); // ["a", "b", "c"]
println!("{} {}", parts.join("|"), "banana".matches("a").count()); // a|b|c 3
a|b|c 3
C++ 
#include <sstream><vector><string>
std::vector<std::string> parts; std::string cell;
std::stringstream ss("a,b,c");          // no stdlib split; roll your own
while (std::getline(ss, cell, ',')) parts.push_back(cell);
std::cout << parts.size();              // 3   (join is also manual)
3
Language Comparison: Python (find/in/split/join/count) and Go (the strings package) are the most complete — every verb is a stdlib call. TypeScript has indexOf/includes/split/join but no count (use split().length-1 or matchAll). Rust is precise but returns byte-index Options and slices by bytes. C++ has find (returning npos) and C++20 starts_with/ends_with, but no split or join — you assemble them from stringstream/getline or a library. Underlying substring search is typically the naive O(n·m) or a Boyer–Moore variant; reach for regex or an Aho–Corasick automaton for multi-pattern work.
Common Mistakes:
❌ [C++] Testing if (s.find(x) == -1) — find returns std::string::npos (the max size_t), and comparing an unsigned size_t to -1 is a trap that happens to work only by accident.
✅ [C++] Compare against std::string::npos explicitly.
❌ [Python] Calling s.index(sub) on possibly-absent input and letting the ValueError crash the request.
✅ [Python] Use s.find(sub) (returns -1) or wrap index in a guard when absence is expected.
❌ [Go/Rust] Treating the returned index as a character position on non-ASCII text — it's a byte offset, and slicing at a non-boundary panics (Rust) or splits a rune (Go).
✅ [Go/Rust] Iterate by runes/chars, or use byte offsets only for slicing at known boundaries.
❌ [TS] Using if (s.indexOf(x)) as a truthiness test — index 0 (a match at the start) is falsy.
✅ [TS] Use s.includes(x) for a boolean, or compare indexOf(x) !== -1 explicitly.
Key Rules:
  • Memorize the "not found" sentinel: Python/Go/TS use -1, Rust uses None, C++ uses npos — never compare C++'s result to -1.
  • Substring indices are byte offsets in Go and Rust; slice only at valid character boundaries on non-ASCII text.
  • Split/join are stdlib everywhere except C++ — there, use stringstream/getline or a small helper.

String Building & Efficient Concatenation Text & String Algorithms
Why it matters: building a big string by repeatedly appending to an immutable string is a classic accidental-O(n²) bug — each += copies the whole accumulated string. Rendering templates, serializing rows, assembling logs, and generating code all want an amortized-O(1) append via a dedicated builder or a buffer you flush once.

In most languages strings are immutable, so s += piece allocates a new string of the full length every time — quadratic over many appends. The fix is a growable buffer: Python's str.join over a list, Go's strings.Builder, a JS array you join, Rust's String::push_str, and C++'s already-mutable std::string (amortized-O(1) +=) or ostringstream.

Example 1 — build "0123456789" in a loop, efficiently

Python 
# Accumulate pieces in a list, then join once (the Pythonic builder):
print("".join(str(i) for i in range(10)))   # 0123456789
0123456789
Go 
import "strings"
var b strings.Builder                  // amortized O(1) appends, no re-copy
for i := 0; i < 10; i++ { fmt.Fprintf(&b, "%d", i) }
fmt.Println(b.String())                // 0123456789
0123456789
TypeScript 
const parts: string[] = [];
for (let i = 0; i < 10; i++) parts.push(String(i));
console.log(parts.join(""));           // 0123456789  (array + join beats +=)
0123456789
Rust 
let mut s = String::new();             // a growable, owned buffer
for i in 0..10 { s.push_str(&i.to_string()); }
println!("{}", s);                     // 0123456789
0123456789
C++ 
#include <string>
std::string s;                          // std::string IS mutable: += is amortized O(1)
for (int i = 0; i < 10; i++) s += std::to_string(i);
std::cout << s;                          // 0123456789
0123456789

Example 2 — pre-size the buffer and join with a separator

Reserving capacity up front avoids intermediate reallocations when you know the rough final size.

Python 
rows = [["a", "1"], ["b", "2"]]
print("\n".join(",".join(r) for r in rows))  # a,1\nb,2
a,1 b,2
Go 
import "strings"
var b strings.Builder
b.Grow(64)                             // pre-allocate to avoid regrowth
b.WriteString("a,1"); b.WriteByte('\n'); b.WriteString("b,2")
fmt.Println(b.String())                // a,1\nb,2
a,1 b,2
TypeScript 
const rows = [["a", "1"], ["b", "2"]];
console.log(rows.map(r => r.join(",")).join("\n"));  // a,1\nb,2
a,1 b,2
Rust 
let mut s = String::with_capacity(64); // reserve to avoid reallocation
s.push_str("a,1"); s.push('\n'); s.push_str("b,2");
println!("{}", s);                     // a,1\nb,2
a,1 b,2
C++ 
#include <string>
std::string s; s.reserve(64);          // reserve capacity up front
s += "a,1"; s += '\n'; s += "b,2";
std::cout << s;                          // a,1\nb,2
a,1 b,2
Language Comparison: The split is between immutable strings (Python, Go, TS, Rust's str) and C++'s mutable std::string. In the immutable camp you never mutate in place: Python collects a list and calls "".join, Go uses strings.Builder (with Grow to pre-size), TypeScript pushes to an array and joins, and Rust grows an owned String via push_str (with with_capacity). C++'s std::string is a growable buffer, so += is already amortized O(1) — reserve just skips the regrowth. Modern JS engines optimize += with rope structures, but array-join is still the safe, predictable choice.
Common Mistakes:
❌ [Go] Concatenating with s += piece inside a loop — strings are immutable, so each step copies the whole accumulator (O(n²)).
✅ [Go] Use strings.Builder (or bytes.Buffer), ideally after Grow.
❌ [Python] Building with s = s + x in a large loop, or using + to glue a list of strings.
✅ [Python] Collect pieces and call "".join(pieces) once — a single allocation of the final size.
❌ [Rust] Writing let s = a + &b + &c; in a hot loop, or calling format! per iteration — each allocates a fresh String.
✅ [Rust] Reuse one String with push_str/write!, sized with with_capacity.
❌ [C++] Reaching for ostringstream for trivial concatenation — it carries locale/formatting overhead you rarely need.
✅ [C++] Prefer std::string::+=/append with reserve; use streams only when you need formatting.
Key Rules:
  • Never build a large string with repeated += on an immutable string type — that's quadratic; use a builder/buffer or join.
  • Pre-size the buffer (Grow/with_capacity/reserve) when you can estimate the final length.
  • C++'s std::string is the one mutable string here — += is fine; the immutable languages need a separate accumulator.

Unicode: Code Points, Grapheme Clusters & Normalization Text & String Algorithms
Why it matters: "the length of a string" has at least three different answers — bytes, code points, and user-perceived characters (grapheme clusters). An emoji with a skin-tone modifier is one grapheme but several code points and many bytes. And "café" typed two different ways can compare unequal until you normalize. Getting this wrong corrupts text, breaks truncation, and causes phantom "equal" strings that aren't.

Three layers stack up: bytes (the UTF-8/UTF-16 encoding), code points (Unicode scalar values), and grapheme clusters (what a human calls a character). Languages default to different layers for len: Go and Rust count bytes, Python counts code points, JS counts UTF-16 code units. Almost none count graphemes without a library. Separately, normalization (NFC/NFD) canonicalizes equivalent sequences so they compare equal.

Example 1 — the many "lengths" of "café" (with a combining accent)

Python 
s = "café"                       # 'e' + combining acute = "café"
print(len(s), len(s.encode("utf-8")))  # 5 6   (code points vs UTF-8 bytes)
import unicodedata
print(len(unicodedata.normalize("NFC", s)))  # 4  (é becomes one code point)
5 6 4
Go 
import "unicode/utf8"
s := "café"
fmt.Println(len(s), utf8.RuneCountInString(s)) // 6 5  (bytes vs runes/code points)
// range over s yields runes; grapheme clustering needs x/text or rivo/uniseg
6 5
TypeScript 
const s = "café";
console.log(s.length, [...s].length);  // 5 5  (UTF-16 code units vs code points)
// spread/for-of iterate code points; graphemes need Intl.Segmenter
const seg = new Intl.Segmenter("en", { granularity: "grapheme" });
console.log([...seg.segment(s)].length); // 4
5 5 4
Rust 
let s = "cafe\u{301}";
println!("{} {}", s.len(), s.chars().count()); // 6 5  (bytes vs char/code points)
// grapheme clusters require the `unicode-segmentation` crate:
// use unicode_segmentation::UnicodeSegmentation;
// s.graphemes(true).count()  ==  4
6 5
C++ 
#include <string>
std::string s = "café";           // UTF-8 bytes
std::cout << s.size();                   // 6  (bytes only; no stdlib Unicode)
// std::string has NO concept of code points or graphemes: use ICU or utf8cpp.
6

Language Comparison: Default len counts different units: Go and Rust count bytes, Python counts code points, JS/TS counts UTF-16 code units (so astral characters like most emoji count as 2), and C++'s size() is bytes. For code points: iterate (Python is already code points, Go range/utf8, Rust .chars(), JS spread/for..of). Grapheme clusters need help everywhere: JS has built-in Intl.Segmenter, while Rust (unicode-segmentation), Go (x/text/uniseg), Python (regex/grapheme), and C++ (ICU) all need a library. Normalization is stdlib only in Python and JS; Go, Rust, and C++ pull in a package.
Common Mistakes:
❌ [Go/Rust] Using len(s) / s.len() as a character count — it's bytes, so any non-ASCII text over-counts.
✅ [Go/Rust] Use utf8.RuneCountInString / s.chars().count() for code points (and a grapheme library for user-visible characters).
❌ [JS/TS] Slicing with s.slice(0, n) or reversing via [...].reverse() and splitting a surrogate pair or a combining mark — producing "�" or a detached accent.
✅ [JS/TS] Segment with Intl.Segmenter at grapheme granularity before truncating or reversing.
❌ [All] Comparing or hashing user-supplied text without normalizing — visually identical strings can differ byte-for-byte (NFC vs NFD).
✅ [All] Normalize to a canonical form (usually NFC) before comparison, hashing, or storage.
❌ [C++] Assuming std::string operations are Unicode-aware — they operate on bytes, so toupper, indexing, and reversing all corrupt multibyte text.
✅ [C++] Use ICU (or utf8cpp) for any real Unicode work; treat std::string as a byte container.
Key Rules:
  • Three lengths exist — bytes, code points, grapheme clusters — and languages default to different ones; pick the layer your task actually needs.
  • Normalize (NFC) before comparing, hashing, or storing user text, or "equal" strings may differ byte-for-byte.
  • Truncating, reversing, or indexing text is safe only at grapheme boundaries; splitting a surrogate pair or combining mark corrupts the string.

Numeric Computing

Numbers are where each language's memory model shows through most sharply. Fixed-width integers wrap or trap on overflow — and the languages disagree violently on which (Python never overflows, Rust panics in debug, C++ signed overflow is undefined behavior). Floating point, by contrast, behaves identically everywhere because they all implement IEEE 754 — which means they all share the same surprises.

Integer Overflow & Safe Arithmetic Numeric Computing
Why it matters: a fixed-width integer that exceeds its range doesn't error by default — it silently wraps (or, in C++, invokes undefined behavior). This has caused real catastrophes: buffer-size miscalculations, negative durations, and the Ariane 5 rocket loss. Knowing your language's overflow behavior — and the checked/saturating/wrapping tools it offers — is the difference between a caught error and a silent corruption.

The five languages sit at very different points. Python integers are arbitrary-precision — they never overflow. Go and C++ unsigned types wrap modulo 2ⁿ; C++ signed overflow is undefined behavior. Rust panics on overflow in debug builds and wraps in release, but offers explicit checked_/wrapping_/saturating_ methods. JS/TS numbers are IEEE-754 doubles — exact only up to 2⁵³, then you need BigInt.

Example 1 — add 1 past the 32-bit signed maximum

Python 
x = 2**31 - 1                          # would be i32 max elsewhere
print(x + 1)                           # 2147483648  (ints are unbounded)
print(2**200)                          # arbitrary precision, no overflow ever
2147483648 1606938044258990275541962092341162602522202993782792835301376
Go 
import "math"
var x int32 = math.MaxInt32           // 2147483647
x++                                    // wraps around silently
fmt.Println(x)                         // -2147483648
-2147483648
TypeScript 
console.log(Number.MAX_SAFE_INTEGER + 1 === Number.MAX_SAFE_INTEGER + 2); // true (!)
// past 2^53 doubles lose integer precision; use BigInt for exact big ints:
console.log(9007199254740991n + 1n + 1n); // 9007199254740993n
true 9007199254740993n
Rust 
let x: i32 = i32::MAX;
// x + 1 would PANIC in debug ('attempt to add with overflow'); be explicit:

println!("{:?}", x.checked_add(1));    // None          (signals overflow)
-2147483648 None
C++ 
#include <cstdint><limits>
int32_t x = std::numeric_limits<int32_t>::max();
// x + 1 is UNDEFINED BEHAVIOR for signed types. Unsigned wraps predictably:
uint32_t u = std::numeric_limits<uint32_t>::max();
std::cout << (u + 1);                   // 0   (unsigned wraps mod 2^32)

Example 2 — a safe add that reports overflow instead of wrapping

When a sum could exceed the range (sizes, counters, financial totals), detect it rather than silently wrapping.

Rust 
fn safe_add(a: i32, b: i32) -> Option<i32> { a.checked_add(b) }
println!("{:?}", safe_add(i32::MAX, 1)); // None
println!("{:?}", safe_add(2, 3));        // Some(5)
None Some(5)
C++ 
#include <cstdint>
int32_t r;
bool overflow = __builtin_add_overflow(INT32_MAX, 1, &r); // GCC/Clang builtin
std::cout << overflow;                  // 1  (true: it overflowed)
1
Go 
import "math"
func safeAdd(a, b int32) (int32, bool) { // detect BEFORE adding
    if b > 0 && a > math.MaxInt32-b { return 0, false }
    if b < 0 && a < math.MinInt32-b { return 0, false }
    return a + b, true
}
fmt.Println(safeAdd(math.MaxInt32, 1)) // 0 false
0 false
Python 
# Python never overflows, but you may need to ENFORCE a width (e.g. emulating i32):
def as_i32(n): return (n + 2**31) % 2**32 - 2**31
print(as_i32(2**31 - 1 + 1))           # -2147483648  (simulate wraparound)
-2147483648
TypeScript 
function safeAdd(a: bigint, b: bigint): bigint { return a + b; } // BigInt = exact
console.log(safeAdd(9007199254740991n, 100n)); // 9007199254741091n
// For 32-bit wrap semantics: (a + b) | 0  forces a signed 32-bit result
9007199254741091n
Language Comparison: Python integers are unbounded — overflow simply doesn't exist (you opt in to fixed width). Rust is the most principled: debug builds panic, release wraps, and you choose semantics explicitly with checked_ (→ Option), wrapping_, saturating_, or overflowing_. Go wraps silently for both signed and unsigned — you detect overflow with a pre-check. C++ is the sharpest edge: unsigned wraps predictably (mod 2ⁿ), but signed overflow is undefined behavior the optimizer may exploit — use __builtin_*_overflow or check first. JS/TS has one number type (double), exact only to 2⁵³; use BigInt beyond that or | 0/>>> 0 for explicit 32-bit int semantics.
Common Mistakes:
❌ [C++] Relying on signed overflow "wrapping" — it is undefined behavior; the compiler may assume it never happens and delete your check.
✅ [C++] Use unsigned types for defined wraparound, or __builtin_add_overflow / check bounds before the operation.
❌ [Go] Assuming a computation on int32 counters can't wrap — it does, silently, producing negative sizes or lengths.
✅ [Go] Range-check before arithmetic on fixed-width types, or use a wider type (int64) for the intermediate.
❌ [Rust] Being surprised that code passing tests (debug: panics caught) corrupts silently in --release (wraps).
✅ [Rust] Pick the semantics on purpose with checked_/saturating_/wrapping_; don't rely on the debug panic.
❌ [TS/JS] Doing exact integer math past Number.MAX_SAFE_INTEGER (2⁵³−1) — results round silently.
✅ [TS/JS] Switch to BigInt for exact large integers; don't mix BigInt and number in one expression.
Key Rules:
  • Know your default: Python never overflows; Go/C++-unsigned wrap; C++-signed is UB; Rust panics (debug) / wraps (release); JS loses precision past 2⁵³.
  • When a sum/product could exceed the range, use checked arithmetic or a bounds pre-check — never assume it "can't happen".
  • Choose wrap semantics deliberately (Rust's wrapping_/saturating_/checked_, C++ unsigned, JS | 0) rather than relying on the accidental default.

Floating-Point: Precision, NaN & Comparison Numeric Computing
Why it matters: 0.1 + 0.2 != 0.3 in every language, because binary floating point can't represent those decimals exactly. Comparing floats with ==, summing many of them, or forgetting that NaN != NaN produces bugs that pass tests on small inputs and fail in production. Since all five implement IEEE 754, the surprises — and the fixes — are universal.

A 64-bit double stores about 15–17 significant decimal digits in binary, so most decimal fractions are approximations. Three rules follow: never test equality with == (use a tolerance), NaN is unequal to everything including itself, and summation order matters (add small numbers first, or use compensated summation). For exact decimals (money) use a decimal/fixed-point type instead.

Example 1 — the classic 0.1 + 0.2

Python 
print(0.1 + 0.2)                       # 0.30000000000000004
print(0.1 + 0.2 == 0.3)               # False
import math
print(math.isclose(0.1 + 0.2, 0.3))   # True  (relative + absolute tolerance)
0.30000000000000004 False True
Go 
import "math"
fmt.Println(0.1 + 0.2)                 // 0.30000000000000004
const eps = 1e-9
fmt.Println(math.Abs((0.1+0.2)-0.3) < eps) // true
0.30000000000000004 true
TypeScript 
console.log(0.1 + 0.2);               // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3);       // false
console.log(Math.abs((0.1 + 0.2) - 0.3) < Number.EPSILON); // true
0.30000000000000004 false true
Rust 
println!("{}", 0.1 + 0.2);             // 0.30000000000000004
println!("{}", 0.1 + 0.2 == 0.3);     // false
println!("{}", ((0.1_f64 + 0.2) - 0.3).abs() < f64::EPSILON); // true
0.30000000000000004 false true
C++ 
#include <cmath><iomanip>
std::cout << std::setprecision(17) << (0.1 + 0.2) << "\n"; // 0.30000000000000004
std::cout << (std::abs((0.1 + 0.2) - 0.3) < 1e-9);         // 1
0.30000000000000004 1

NaN propagates through arithmetic and compares false to everything — the only reliable test is a dedicated isNaN predicate.

Python 
import math
nan = float("nan")

print(math.isnan(nan), math.inf > 1e308) # True True
False True True
TypeScript 
const nan = NaN;
console.log(nan === nan);              // false
console.log(Number.isNaN(nan), [1, NaN].includes(NaN)); // true true
// note: indexOf(NaN) is -1, but includes(NaN) is true (SameValueZero)
false true true
Go 
import "math"
nan := math.NaN()
fmt.Println(nan == nan)                // false
fmt.Println(math.IsNaN(nan), math.IsInf(math.Inf(1), 1)) // true true
false true true
Rust 
let nan = f64::NAN;
println!("{}", nan == nan);            // false

C++

double nan = std::numeric_limits<double>::quiet_NaN();

Language Comparison: Because all five implement IEEE 754, the numeric behavior is identical — 0.1 + 0.2 and NaN semantics match everywhere. What differs is the tooling: Python ships math.isclose (relative + absolute tolerance) and a decimal/fractions module for exactness; Rust exposes f64::EPSILON, and its NaN-driven PartialOrd (not Ord) means you need total_cmp or partial_cmp to sort floats. Go, C++, and JS give you the primitives (IsNaN/isnan/Number.isNaN, an epsilon constant) and expect you to write the tolerance check. For money, everyone agrees: use a decimal/fixed-point type, never binary floats.
Common Mistakes:
❌ [All] Comparing floats with == (or using one as a loop counter / map key) — rounding makes "equal" values differ in the last bits.
✅ [All] Compare with a tolerance (abs(a-b) < eps or a relative check like math.isclose); avoid floats as exact keys.
❌ [All] Using floating-point for currency — accumulated rounding produces off-by-a-cent errors.
✅ [All] Use a decimal type (Python Decimal, integer cents, or a fixed-point library) for money.
❌ [Rust] Trying to .sort() a Vec<f64> — it won't compile because f64 is only PartialOrd (thanks to NaN).
✅ [Rust] Sort with sort_by(|a, b| a.total_cmp(b)) (or partial_cmp().unwrap() when you know there are no NaNs).
❌ [JS/TS] Testing x === NaN (always false) or using the global isNaN() which coerces (isNaN("foo") is true).
✅ [JS/TS] Use Number.isNaN(x) for a precise, coercion-free NaN test.
Key Rules:
  • Never compare floats with ==; use an absolute or relative tolerance (and never use a float as an exact key or money value).
  • NaN is unequal to everything (including itself) and propagates — detect it only with a dedicated isNaN-style predicate.
  • The IEEE-754 behavior is identical across all five languages; only the helper tooling (isclose, EPSILON, total_cmp) differs.
