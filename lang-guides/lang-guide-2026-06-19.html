TypeScript: Conditional Types & infer Language Deep Dive 7 days since review · rep 2
T extends U ? X : Y performs type-level branching. infer introduces a type variable inside the condition that TypeScript solves for: type ReturnType<T> = T extends (...args: any) => infer R ? R : never. When T is a naked type parameter, the conditional distributes over unions.
Python (closest analogue)
from typing import TypeVar, Callable, get_type_hints

# Python has no conditional types; you inspect at runtime instead.
def return_type(fn: Callable):
    return get_type_hints(fn).get("return")

def f(x: int) -> str: return str(x)
print(return_type(f))   # <class 'str'>
Go (no equivalent)
// Go generics have no conditional/inferred-result types.
// You constrain with interfaces instead:
type Number interface{ ~int | ~float64 }

func Sum[T Number](xs []T) T {
    var s T
    for _, x := range xs { s += x }
    return s
}
TypeScript
type Awaited<T> =
  T extends Promise<infer U> ? Awaited<U> : T;

type ElementType<T> = T extends (infer E)[] ? E : never;

type A = ElementType<number[]>;        // number
type B = Awaited<Promise<string>>;     // string

// distribution over unions
type ToArray<T> = T extends any ? T[] : never;
type C = ToArray<string | number>;     // string[] | number[]
Rust (associated types)
// Rust expresses "the result type" via associated types.
trait Producer { type Output; fn make(&self) -> Self::Output; }

struct IntMaker;
impl Producer for IntMaker {
    type Output = i32;
    fn make(&self) -> i32 { 42 }
}
C++ (type traits)
#include <type_traits>
// std::conditional is C++'s ternary at the type level.
template<bool B>
using Pick = std::conditional_t<B, int, double>;

static_assert(std::is_same_v<Pick<true>,  int>);
static_assert(std::is_same_v<Pick<false>, double>);
Challenge (all 5 languages)
1 Create a type-level Awaited<T> in TypeScript that recursively unwraps nested Promise types (e.g. Promise<Promise<number>> → number). In the other 4 languages, express the closest idiom (associated type, type trait, runtime hint, or interface constraint) that captures "the resolved/result type".
#1 Pitfall per Language
  • Python: there is no compile-time conditional type — don’t fake it with overloads expecting type-level branching.
  • Go: generics cannot select a result type by condition; design around interface constraints.
  • TypeScript: forgetting distribution — T extends U ? ... spreads over unions when T is a naked param; wrap in [T] extends [U] to disable it.
  • Rust: conflating associated types with generic params — associated types are chosen by the impl, not the caller.
  • C++: std::conditional evaluates both branches’ types; a branch that’s ill-formed still errors.
Rust: Trait Objects & Dynamic Dispatch Language Deep Dive 7 days since review · rep 2
dyn Trait enables runtime polymorphism via a vtable. Use Box<dyn Trait> for owned trait objects, &dyn Trait for borrowed. A trait is object-safe only if its methods don’t return Self and aren’t generic. Other languages reach the same dispatch through interfaces and virtuals.
Python
from typing import Protocol

class Animal(Protocol):
    def speak(self) -> str: ...

class Dog:
    def speak(self): return "woof"
class Cat:
    def speak(self): return "meow"

zoo: list[Animal] = [Dog(), Cat()]
print([a.speak() for a in zoo])
['woof', 'meow']
Go
package main
import "fmt"

type Animal interface{ Speak() string }
type Dog struct{}
type Cat struct{}
func (Dog) Speak() string { return "woof" }
func (Cat) Speak() string { return "meow" }

func main() {
    zoo := []Animal{Dog{}, Cat{}}
    for _, a := range zoo { fmt.Println(a.Speak()) }
}
woof meow
TypeScript
interface Animal { speak(): string; }
class Dog implements Animal { speak() { return "woof"; } }
class Cat implements Animal { speak() { return "meow"; } }

const zoo: Animal[] = [new Dog(), new Cat()];
console.log(zoo.map(a => a.speak()));
[ 'woof', 'meow' ]
Rust
trait Animal { fn speak(&self) -> &str; }
struct Dog; struct Cat;
impl Animal for Dog { fn speak(&self) -> &str { "woof" } }
impl Animal for Cat { fn speak(&self) -> &str { "meow" } }

fn main() {
    let zoo: Vec<Box<dyn Animal>> = vec![Box::new(Dog), Box::new(Cat)];
    for a in &zoo { println!("{}", a.speak()); }
}
woof meow
C++
#include <iostream>
#include <memory>
#include <vector>
struct Animal { virtual std::string speak() const = 0; virtual ~Animal() = default; };
struct Dog : Animal { std::string speak() const override { return "woof"; } };
struct Cat : Animal { std::string speak() const override { return "meow"; } };

int main() {
    std::vector<std::unique_ptr<Animal>> zoo;
    zoo.push_back(std::make_unique<Dog>());
    zoo.push_back(std::make_unique<Cat>());

C++: Concepts (C++20) & Constraints Language Deep Dive 7 days since review · rep 2
A concept names a set of type requirements; a requires clause constrains a template. Concepts replace SFINAE with readable syntax and far better diagnostics: template<typename T> concept Addable = requires(T a, T b) { a + b; };.
Python (Protocol)
from typing import Protocol

class Addable(Protocol):
    def __add__(self, other): ...

def add_all(xs: list[Addable]):
    total = xs[0]
    for x in xs[1:]: total = total + x
    return total

print(add_all([1, 2, 3]))
6
Go (type constraint)
package main
import "fmt"

type Addable interface{ ~int | ~float64 | ~string }

func Concat[T Addable](xs []T) T {
    var acc T
    for _, x := range xs { acc += x }
    return acc
}
func main() { fmt.Println(Concat([]int{1, 2, 3})) }
6
TypeScript (constraint)
// Closest: generic constraint on a structural shape.
interface Addable<T> { add(o: T): T; }

function sum<T extends Addable<T>>(xs: T[]): T {
  return xs.reduce((a, b) => a.add(b));
}
Rust (trait bound)
use std::ops::Add;

fn sum<T: Add<Output = T> + Copy>(xs: &[T], zero: T) -> T {
    let mut acc = zero;
    for &x in xs { acc = acc + x; }
    acc
}
fn main() { println!("{}", sum(&[1, 2, 3], 0)); }
6
C++
#include <iostream>
#include <concepts>
#include <vector>

template<typename T>
concept Addable = requires(T a, T b) { { a + b } -> std::convertible_to<T>; };

template<Addable T>
T sum(const std::vector<T>& xs) {
    T acc{};
    for (const auto& x : xs) acc = acc + x;
    return acc;
}
int main() { std::cout << sum(std::vector<int>{1,2,3}) << "\n"; }

Regular Expressions I/O & Data Processing 6 days since review · rep 2
Python re, Go regexp, TS RegExp, Rust regex crate, C++ <regex>. Named groups differ: Python (?P<name>...), the others use (?<name>...). Compile once, reuse the compiled pattern.
Python
import re
pat = re.compile(r"(?P<k>\w+)=(?P<v>\d+)")
m = pat.search("port=8080")
print(m.group("k"), m.group("v"))
port 8080
Go
package main
import ("fmt"; "regexp")
func main() {
    re := regexp.MustCompile(`(?P<k>\w+)=(?P<v>\d+)`)
    m := re.FindStringSubmatch("port=8080")
    fmt.Println(m[1], m[2])   // no lookbehind in RE2
}
port 8080
TypeScript
const re = /(?<k>\w+)=(?<v>\d+)/;
const m = "port=8080".match(re);
console.log(m!.groups!.k, m!.groups!.v);

Rust
use regex::Regex;
fn main() {
    let re = Regex::new(r"(?<k>\w+)=(?<v>\d+)").unwrap();
    let c = re.captures("port=8080").unwrap();
    println!("{} {}", &c["k"], &c["v"]);
}
port 8080
C++
#include <iostream>
#include <regex>
int main() {
    std::regex re(R"((\w+)=(\d+))");   // no named groups in std::regex
    std::smatch m;
    std::string s = "port=8080";
    if (std::regex_search(s, m, re))
        std::cout << m[1] << " " << m[2] << "\n";
}

#1 Pitfall per Language
  • Python: using match (anchored at start) when you meant search (anywhere).
  • Go: RE2 has no lookbehind/lookahead — restructure the pattern instead.

Distributed Systems Patterns

Three patterns that power microservice communication: typed RPC, real-time bidirectional channels, and failure isolation.

gRPC & Protocol Buffers Distributed Systems

gRPC provides strongly-typed, high-performance RPC with automatic client/server code generation across languages — the de-facto standard for internal microservice communication where REST’s overhead and loose typing become liabilities.

Protocol Buffers ("protobuf") is a binary, schema-first serialization format. You declare messages and services in a .proto file, then a compiler (protoc) generates type-safe code for every target language. gRPC layers an RPC framework over HTTP/2 on top of those generated types.

1. Protocol Buffers basics

Messages are typed records; each field has a unique, stable field number used in the wire format (never renumber a released field). Scalar types (int32, string, bool), repeated for lists, and nested messages compose into a schema.

2. Service definition

A service block names RPC methods with a request and response message: rpc SayHello (HelloRequest) returns (HelloReply);

3. Code generation & implementation

Each language plugs its own plugin into protoc (or uses build-time codegen) and implements the generated server interface; clients call generated stubs as if they were local methods.

4. Streaming

gRPC supports four call shapes: unary (1→1), server-streaming (1→N), client-streaming (N→1), and bidirectional (N↔N) using the stream keyword on either side.

Example 2 — Server implementation

Python (grpcio)
import grpc
from concurrent import futures
import greeter_pb2 as pb, greeter_pb2_grpc as rpc

class Greeter(rpc.GreeterServicer):
    def SayHello(self, request, context):
        return pb.HelloReply(message=f"Hello, {request.name}!")

server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
rpc.add_GreeterServicer_to_server(Greeter(), server)
server.add_insecure_port("[::]:50051")
server.start()
server.wait_for_termination()
# serving on :50051
Go (google.golang.org/grpc)

import (
    "context"; "net"
    "google.golang.org/grpc"

type server struct{ pb.UnimplementedGreeterServer }

use tonic::{transport::Server, Request, Response, Status};
use greet::greeter_server::{Greeter, GreeterServer};
use greet::{HelloReply, HelloRequest};
pub mod greet { tonic::include_proto!("greet"); }

#[derive(Default)]
struct MyGreeter;

#[tonic::async_trait]
impl Greeter for MyGreeter {
    async fn say_hello(&self, req: Request<HelloRequest>)
        -> Result<Response<HelloReply>, Status> {
        let msg = format!("Hello, {}!", req.into_inner().name);
        Ok(Response::new(HelloReply { message: msg }))
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    Server::builder()
        .add_service(GreeterServer::new(MyGreeter::default()))
        .serve("[::1]:50051".parse()?).await?;
    Ok(())
}

Language Comparison: All five share one .proto source of truth, but codegen ergonomics differ. Python (grpcio) and C++ (grpc++) generate verbose setter/getter APIs; Go (google.golang.org/grpc) generates plain structs and is fully synchronous-looking despite HTTP/2 underneath; Rust (tonic) is async-native on tokio with builder-style structs; TypeScript (@grpc/grpc-js) uses callback or promise stubs. Rust and Go bake codegen into the build; Python/C++ typically run protoc as a separate step.
Common Mistakes:
[Proto] Reusing or renumbering a field tag after release — breaks wire compatibility silently.
[Proto] Treat field numbers as permanent; reserve removed ones with `reserved 3;`.
[Python] Forgetting to regenerate `*_pb2.py` after editing the proto — stale stubs.
[Python] Re-run `python -m grpc_tools.protoc ...` as part of the build.
[Go] Not embedding `UnimplementedGreeterServer` — future RPCs break the build.
[Go] Embed `pb.UnimplementedGreeterServer` for forward compatibility.
[Rust] Blocking inside an async handler — starves the tokio runtime.
[Rust] Use `.await` on async work; offload CPU-bound work to `spawn_blocking`.
[C++] Leaking the server thread — calling Wait() without shutdown handling.
[C++] Hold the `unique_ptr<Server>` and call `Shutdown()` on signal.
Key Rules:
  • Use gRPC for internal, high-throughput, strongly-typed service-to-service calls; prefer REST/JSON for public, browser-facing APIs.
  • Field numbers are forever — add new fields with new numbers, never reuse old ones.
  • Pick the streaming shape that matches the data flow; don’t poll unary calls when server-streaming fits.

WebSocket & Real-Time Communication Distributed Systems

WebSockets upgrade a single HTTP connection into a persistent, full-duplex channel — the backbone of chat, live dashboards, collaborative editing, and push notifications where polling would be wasteful and laggy.

Unlike request/response HTTP, a WebSocket stays open so either side can send a frame at any time.

  1. Handshake: the client sends an HTTP request with Upgrade: websocket; the server replies 101 Switching Protocols and the TCP connection becomes a WebSocket.
  2. Server: accept connections, track them, and broadcast messages to all (or a subset of) clients.
  3. Client: handle open, message, close, and error events.
  4. Liveness: ping/pong frames detect dead peers; clients reconnect with backoff after a drop.

The examples below show a minimal echo server (sends back whatever it receives) and a client that connects, sends one message, and prints the reply.

Circuit Breaker Pattern Distributed Systems

A circuit breaker stops a caller from hammering an unhealthy downstream service: after enough failures it "opens" and fails fast, sparing both systems, then probes for recovery before resuming normal traffic. This prevents one slow dependency from cascading into a total outage.

The breaker is a small state machine wrapped around a remote call.

  1. Closed (normal): calls pass through. Consecutive failures are counted.
  2. Open (tripped): after N failures, calls fail immediately without touching the downstream. A timer runs.
  3. Half-Open (probing): when the timer expires, one trial call is allowed. Success → back to Closed (reset counters); failure → back to Open (restart timer).

Implementation needs a failure counter, a state field, a timestamp of when the breaker opened, and thread-safe access since calls are usually concurrent.

Example 2 — A working circuit breaker

Python
import time, threading

class CircuitBreaker:
    def __init__(self, threshold=3, reset_timeout=5.0):
        self.threshold = threshold
        self.reset_timeout = reset_timeout
        self.failures = 0
        self.state = State.CLOSED
        self.opened_at = 0.0
        self.lock = threading.Lock()

    def call(self, fn, *args):
        with self.lock:
            if self.state == State.OPEN:
                if time.monotonic() - self.opened_at >= self.reset_timeout:
                    self.state = State.HALF_OPEN
                else:
                    raise RuntimeError("circuit open")
        try:
            result = fn(*args)
        except Exception:
            with self.lock:
                self.failures += 1
                if self.failures >= self.threshold:
                    self.state = State.OPEN
                    self.opened_at = time.monotonic()
            raise
        with self.lock:
            self.failures = 0
            self.state = State.CLOSED
        return result
# raises "circuit open" after 3 failures, until reset_timeout

            b.state = HalfOpen
        } else {
            b.mu.Unlock()
            return errors.New("circuit open")
        }

    b.mu.Unlock()

        b.failures++
        if b.failures >= b.threshold {
            b.state, b.openedAt = Open, time.Now()
        }
        b.mu.Unlock()
        return err
    }
    b.mu.Lock()
    b.failures, b.state = 0, Closed
    b.mu.Unlock()
    return nil
}

use std::time::{Duration, Instant};

struct Breaker {
    threshold: u32,
    reset_after: Duration,
    inner: Mutex<Inner>,
}

    fn call<T, E>(&self, f: impl FnOnce() -> Result<T, E>)
        -> Result<T, String> where E: std::fmt::Display {
        {
            let mut g = self.inner.lock().unwrap();

                g.failures = 0; g.state = State::Closed;
                Ok(v)

            Err(e) => {

                    g.state = State::Open; g.opened_at = Instant::now();

            }
        }

                    throw std::runtime_error("circuit open");

            auto r = fn();
            std::lock_guard lk(m_);
            failures_ = 0; state_ = State::Closed;
            return r;
        } catch (...) {
            std::lock_guard lk(m_);
            if (++failures_ >= threshold_) {
                state_ = State::Open;
                openedAt_ = std::chrono::steady_clock::now();
            }
            throw;
        }
    }
};

Language Comparison: State and timers look similar everywhere, but thread safety diverges. Go and C++ protect shared mutable state with an explicit Mutex; Rust encodes "shared mutable state needs a lock" in the type system via Mutex<Inner> (you literally can’t forget the guard). Python needs a threading.Lock for threaded use but is single-threaded under asyncio. TypeScript on a single event loop needs no lock at all — there’s no preemption between await points other than what you write. Timers use monotonic clocks (time.monotonic, Instant, steady_clock) so they aren’t fooled by wall-clock changes.
Common Mistakes:
[Python] Using `time.time()` (wall clock) for the reset timer — NTP jumps corrupt it.
[Python] Use `time.monotonic()` for elapsed-time comparisons.
[Go] Forgetting to unlock before calling fn() — holds the lock during slow I/O.
[Go] Release the mutex before the downstream call; re-acquire to record the result.
[TypeScript] Assuming concurrency safety, then sharing one breaker across worker threads.
[TypeScript] One breaker per event loop; coordinate across threads/processes explicitly.
[Rust] Holding the MutexGuard across the user closure — serializes all calls.
[Rust] Drop the guard (scoped block) before invoking the closure.
[C++] Counting failures but never resetting on success — breaker never recovers.
[C++] Reset failures_ and state_ to Closed on every successful call.
Key Rules:
  • Open on a rate or consecutive count of failures, not a single blip; half-open lets exactly one probe through.
  • Never hold a lock across the wrapped network call — check state, release, call, re-acquire to record.
  • Combine with timeouts and retries-with-backoff; a breaker without a timeout still hangs on slow (not failing) calls.

Architecture Patterns

Two foundational patterns for building testable, maintainable applications: decoupling via injection and abstracting persistence.

Dependency Injection Across Paradigms Architecture Patterns

Dependency injection (DI) hands a component its collaborators instead of letting it construct them. This decouples code, makes it testable by swapping real implementations for fakes, and lets you change behavior (e.g. real DB vs in-memory) without touching the consumer.

DI is less a framework than a discipline: depend on abstractions, receive them from outside.

  1. Constructor injection (most common): pass dependencies as constructor/init parameters.
  2. Interface-based DI: the consumer references an interface/trait/protocol, not a concrete class, so any implementation fits.
  3. Containers vs manual wiring: small apps wire dependencies by hand in main; larger ones may use a DI container that resolves the graph for you.
  4. Testing: inject a mock/fake to assert behavior in isolation.

Language Comparison: The abstraction mechanism differs but the shape is identical. Python relies on duck typing (or Protocol) — no declaration needed. Go uses implicit interface satisfaction, so a fake just needs the right methods. TypeScript uses structural interfaces. Rust offers a choice: generics (Service<S: Store>, zero-cost static dispatch) or Box<dyn Store> (dynamic). C++ mirrors Rust’s split: templates for static, abstract base classes for dynamic. Heavy DI containers (inversify, tsyringe, Python’s dependency-injector) exist, but Go and Rust communities mostly favor explicit constructor wiring.
Common Mistakes:
[Python] Defaulting a dependency to a real impl: `def __init__(self, db=PostgresDB())` — shared mutable default + real I/O.
[Python] Default to None and construct in main, or require the dependency explicitly.
[Go] Accepting a *concrete* type instead of an interface — can't substitute a fake.
[Go] Accept the narrowest interface the function actually needs.
[TypeScript] Newing dependencies inside the class body — defeats injection.
[TypeScript] Receive collaborators via the constructor; build them at the composition root.
[Rust] Reaching for Box<dyn Trait> everywhere — pays vtable cost you may not need.
[Rust] Prefer generics for static dispatch; use dyn only when you need heterogeneous storage.
[C++] Storing an injected reference whose lifetime ends before the owner — dangling ref.
[C++] Ensure the dependency outlives the Service, or inject a shared_ptr.
Key Rules:
  • Inject through constructors/init params; build the whole graph once at the "composition root" (usually main).
  • Depend on the narrowest abstraction you need, not a concrete class — that’s what makes testing and substitution cheap.
  • Reach for a DI container only when manual wiring becomes unwieldy; for most services, explicit wiring is clearer.

Database Access Patterns Architecture Patterns

Every production app persists data. Knowing the spectrum — raw parameterized SQL, query builders, ORMs, and the repository pattern — lets you pick the right abstraction level: fast and explicit where it matters, high-level where it speeds development. And parameterized queries are your front-line defense against SQL injection.

  1. Raw SQL with parameters: never interpolate user input into SQL. Use placeholders (?, $1, :name) so the driver binds values safely.
  2. Query builders: construct SQL programmatically with type checking, avoiding string concatenation.
  3. ORMs: map rows to objects/structs; convenient but can hide expensive queries (N+1).
  4. Repository pattern: hide the data source behind an interface so the rest of the app doesn’t know (or care) whether it’s Postgres, SQLite, or an in-memory fake — pairs perfectly with DI.

Example 1 — Parameterized query (safe against injection)

Python (sqlite3)
import sqlite3
conn = sqlite3.connect(":memory:")
conn.execute("CREATE TABLE users(id INTEGER, name TEXT)")
conn.execute("INSERT INTO users VALUES (?, ?)", (1, "Ada"))

# parameterized — the value is bound, not interpolated
row = conn.execute(
    "SELECT name FROM users WHERE id = ?", (1,)).fetchone()
print(row[0])

Example 3 — Repository pattern (abstracting access)

Python
from typing import Protocol, Optional

class UserRepo(Protocol):
    def get(self, id: int) -> Optional[str]: ...
    def add(self, name: str) -> int: ...

class SqliteUserRepo:
    def __init__(self, conn): self.conn = conn
    def get(self, id):
        r = self.conn.execute(
            "SELECT name FROM users WHERE id=?", (id,)).fetchone()
        return r[0] if r else None
    def add(self, name):
        cur = self.conn.execute(
            "INSERT INTO users(name) VALUES(?)", (name,))
        return cur.lastrowid
# callers depend on UserRepo, not sqlite3
Go
type UserRepo interface {
    Get(id int) (string, error)
    Add(name string) (int, error)
}

type sqlUserRepo struct{ db *sql.DB }

func (r *sqlUserRepo) Get(id int) (string, error) {
    var name string
    err := r.db.QueryRow(
        "SELECT name FROM users WHERE id=?", id).Scan(&name)
    return name, err
}
// inject UserRepo into services
TypeScript
interface UserRepo {
  get(id: number): string | null;
  add(name: string): number;
}

class SqliteUserRepo implements UserRepo {
  constructor(private db: Database.Database) {}
  get(id: number) {
    const r = this.db.prepare(
      "SELECT name FROM users WHERE id=?").get(id) as { name: string } | undefined;
    return r?.name ?? null;
  }
  add(name: string) {
    return Number(this.db.prepare(
      "INSERT INTO users(name) VALUES(?)").run(name).lastInsertRowid);
  }
}
// service depends on UserRepo
Rust
trait UserRepo {
    fn get(&self, id: i32) -> Option<String>;
    fn add(&self, name: &str) -> i32;
}

struct SqliteUserRepo { conn: rusqlite::Connection }

impl UserRepo for SqliteUserRepo {
    fn get(&self, id: i32) -> Option<String> {
        self.conn.query_row(
            "SELECT name FROM users WHERE id=?1",
            rusqlite::params![id], |r| r.get(0)).ok()
    }
    fn add(&self, name: &str) -> i32 {
        self.conn.execute(
            "INSERT INTO users(name) VALUES(?1)",
            rusqlite::params![name]).unwrap();
        self.conn.last_insert_rowid() as i32
    }
}
// Box<dyn UserRepo> for swappable storage
C++
struct UserRepo {
    virtual std::optional<std::string> get(int id) = 0;
    virtual int add(const std::string& name) = 0;
    virtual ~UserRepo() = default;
};

class SqliteUserRepo : public UserRepo {
    sqlite3* db_;
public:
    explicit SqliteUserRepo(sqlite3* db) : db_(db) {}
    std::optional<std::string> get(int id) override { /* prepare+bind+step */ return std::nullopt; }
    int add(const std::string& name) override { /* prepare+bind+step */ return 0; }
};

Language Comparison: Placeholder syntax differs (? in Python/Go/SQLite, $1 in Postgres/sqlx, ?1 in rusqlite, :name in SOCI) but the principle is universal: bind, never interpolate. ORM maturity varies: Python (SQLAlchemy/Django) and TS (Prisma/Drizzle) have rich, ergonomic ORMs; Rust’s Diesel/sqlx give compile-time-checked queries (sqlx verifies SQL against a live schema at build time); Go leans toward thin wrappers (sqlx) plus optional GORM; C++ has the least mature ORM story (SOCI, sqlpp11) and most code uses the raw driver. The repository pattern looks the same in all five — an interface/trait/protocol with a concrete impl injected via DI.
Common Mistakes:
[Python] f-string SQL: `execute(f"... WHERE id={user_id}")` — SQL injection hole.
[Python] Always use placeholders: `execute("... WHERE id=?", (user_id,))`.
[Go] Ignoring rows.Close()/Err() — leaks connections and hides scan errors.
[Go] `defer rows.Close()` and check `rows.Err()` after the loop.
[TypeScript] String-concatenating user input into a query template — injection.
[TypeScript] Use prepared statements / the ORM's parameter binding.
[Rust] Calling sqlx without the offline cache in CI — build fails needing a live DB.
[Rust] Commit `.sqlx` query metadata (`cargo sqlx prepare`) for offline builds.
[C++] Forgetting sqlite3_finalize on a statement — leaks the prepared stmt.
[C++] RAII-wrap the sqlite3_stmt so finalize runs on scope exit.
Key Rules:
  • Parameterize every query that includes external input — this is non-negotiable for security.
  • Hide persistence behind a repository interface so business logic is testable with an in-memory fake.
  • Choose abstraction by need: raw SQL for hot/complex paths, an ORM for CRUD-heavy domains; watch for N+1 queries either way.
