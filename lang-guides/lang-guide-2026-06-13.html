Decorators & Middleware Abstractions 5 days overdue · rep 2
Decorators and middleware wrap functions to add cross-cutting behavior without modifying the original logic. Python: @decorator with functools.wraps. Go: HTTP middleware func(http.Handler) http.Handler. TypeScript: method/class decorators with @decorator syntax. Rust: procedural macros / attribute macros (#[derive(...)]) and wrapper functions. C++: CRTP pattern for static polymorphism and std::function wrappers.
Python
import functools
import time

def timing(func):
    """Decorator that measures and prints execution time."""
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timing
def slow_function(n: int) -> int:
    """Simulate a slow computation."""
    total = 0
    for i in range(n):
        total += i * i
    time.sleep(0.5)
    return total

result = slow_function(1000)
print(f"Result: {result}")
print(f"Function name preserved: {slow_function.__name__}")
print(f"Docstring preserved: {slow_function.__doc__}")

// TimingMiddleware wraps an http.Handler and logs request duration.
func TimingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        elapsed := time.Since(start)
        log.Printf("%s %s took %v", r.Method, r.URL.Path, elapsed)
    })
}

func helloHandler(w http.ResponseWriter, r *http.Request) {
    time.Sleep(50 * time.Millisecond) // simulate work
    fmt.Fprintf(w, "Hello, World!")
}

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/hello", helloHandler)

    // Wrap the entire mux with timing middleware
    wrapped := TimingMiddleware(mux)

    fmt.Println("Server starting on :8080")
    log.Fatal(http.ListenAndServe(":8080", wrapped))
}

// Method decorator that logs execution time
function timing(
    target: any,
    propertyKey: string,
    descriptor: PropertyDescriptor
): PropertyDescriptor {
    const originalMethod = descriptor.value;

    descriptor.value = function (...args: any[]) {
        const start = performance.now();
        const result = originalMethod.apply(this, args);
        const elapsed = performance.now() - start;
        console.log(`${propertyKey} took ${elapsed.toFixed(2)}ms`);
        return result;
    };

    return descriptor;
}

use std::time::Instant;

/// A timing wrapper function that measures closure execution time.
/// (Proc macros require a separate crate, so we use a generic wrapper.)
fn timed<F, R>(name: &str, f: F) -> R
where
    F: FnOnce() -> R,
{
    let start = Instant::now();
    let result = f();
    let elapsed = start.elapsed();
    println!("{} took {:.4?}", name, elapsed);
    result
}

fn slow_computation(n: u64) -> u64 {
    let mut total: u64 = 0;
    for i in 0..n {
        total = total.wrapping_add(i.wrapping_mul(i));
    }
    std::thread::sleep(std::time::Duration::from_millis(500));
    total
}

fn main() {
    let result = timed("slow_computation", || slow_computation(1000));
    println!("Result: {}", result);

    // Can also time closures inline
    let msg = timed("string_build", || {
        let mut s = String::new();
        for i in 0..100 {
            s.push_str(&format!("{} ", i));
        }
        s
    });
    println!("Built string of length: {}", msg.len());
}

// Generic timing wrapper using templates
template <typename Func, typename... Args>
auto timed(const std::string& name, Func&& func, Args&&... args) {
    auto start = std::chrono::high_resolution_clock::now();
    auto result = func(std::forward<Args>(args)...);
    auto end = std::chrono::high_resolution_clock::now();
    auto elapsed = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
    std::cout << name << " took " << elapsed.count() << "us" << std::endl;
    return result;
}

}

Common Pitfalls by Language
  • Python: Forgetting functools.wraps loses the original function’s __name__, __doc__, and __module__ attributes, breaking introspection and documentation tools.
  • Go: Middleware ordering matters — logging middleware should wrap auth middleware, not the other way around. The outermost middleware runs first on request and last on response.
  • TypeScript: Decorators are Stage 3 and may change. The legacy (experimentalDecorators) and TC39 decorator specs have different semantics. Check which your project uses.
  • Rust: Procedural macros must live in a separate crate with proc-macro = true. This adds build complexity. For simple wrapping, prefer generic functions over proc macros.
  • C++: Heavy use of CRTP and template wrappers can cause code bloat — each instantiation generates a new type. Monitor binary size in template-heavy codebases.

Common Pitfalls by Language
  • Python: Circular imports cause ImportError. If module A imports from B and B imports from A, restructure by extracting shared code into a third module C.
  • Go: Cyclic package dependencies are a compile error. Unlike Python, there is no workaround — you must redesign your package boundaries.
  • TypeScript: Barrel files (index.ts) that re-export everything can hurt tree-shaking and increase bundle size. Only re-export what consumers actually need.
  • Rust: pub(crate) exposes an item within the crate but not to external consumers. Forgetting this distinction leads to either over-exposing internals or unnecessary pub on everything.
  • C++: Header-only libraries are simple but cause long compile times on large projects. Separate declaration (header) from definition (source) for faster incremental builds.

    let lo = 0;

Common Pitfalls by Language
  • Python: Forgetting await silently returns a coroutine object instead of the result. The code runs without errors but produces wrong values. Enable RuntimeWarning for unawaited coroutines.
  • Go: Goroutine leaks occur when nobody reads from an unbuffered channel. Use buffered channels or context.Context cancellation to ensure goroutines exit cleanly.
  • TypeScript: Unhandled Promise rejections crash Node.js (v15+). Always .catch() or use Promise.allSettled() instead of Promise.all() when partial failure is acceptable.
  • Rust: tokio::spawn requires the future to be Send + 'static. You cannot borrow local variables across .await points in a spawned task — clone or use Arc.
  • C++: std::async may run deferred (lazy) unless you explicitly pass std::launch::async. Deferred futures only execute when .get() or .wait() is called, defeating the purpose of concurrency.

Network Programming — TCP/UDP Sockets Systems & Architecture
Network programming is the foundation of all distributed systems. Understanding raw sockets gives you insight into what web frameworks abstract away, and is essential for building custom protocols, debugging network issues, and writing high-performance servers.

TCP vs UDP

TCP (Transmission Control Protocol) is connection-oriented: it establishes a reliable, ordered byte stream between two endpoints. Data is guaranteed to arrive in order and without duplication. The cost is higher latency due to the three-way handshake and acknowledgment packets. Use TCP for HTTP, database connections, file transfers, and any protocol where data integrity matters.

UDP (User Datagram Protocol) is connectionless: each datagram is independent, with no delivery guarantee, no ordering, and no congestion control. UDP is faster for real-time applications like video streaming, DNS lookups, gaming, and IoT telemetry where occasional packet loss is acceptable.

Socket Lifecycle

TCP Server: Create socket → Bind to address → Listen for connections → Accept client → Read/Write data → Close connection. Each accepted client gets its own socket for bidirectional communication.

TCP Client: Create socket → Connect to server → Read/Write data → Close connection.

UDP: Create socket → Bind to address (server) or just create (client) → SendTo/RecvFrom → Close. No connection establishment needed.

Concurrency Models for Servers

Thread-per-connection: Simple but scales poorly — each thread consumes stack memory (typically 1-8 MB). Suitable for low-concurrency servers.

Event loop: Single-threaded, non-blocking I/O with callbacks (Node.js, Python asyncio). Efficient for I/O-bound work but CPU-heavy handlers block everything.

Async I/O: Rust tokio, Go goroutines. Lightweight tasks multiplexed onto a thread pool. Best scalability for high-concurrency servers.

Goroutine-per-connection (Go): Goroutines are cheap (~4 KB stack, dynamically grown). You can have millions of goroutines, making the “thread-per-connection” model practical again.

Language Comparison: Go’s goroutine-per-connection model makes concurrent servers trivially simple — just add go before the handler call. Rust’s async/tokio approach gives C-level performance with memory safety guarantees. Python threading works but is limited by the GIL for CPU-bound work in the handler. C++ gives maximum control over system calls but requires manual resource management (RAII helps). TypeScript’s event loop handles I/O concurrency naturally but is single-threaded for CPU work.
Common Pitfalls
  • Partial reads: TCP is a byte stream protocol — one send() may arrive as multiple recv() calls. Always read in a loop until you have a complete message (use length-prefixed framing or delimiters).
  • Endianness: Network byte order is big-endian. Use htons/ntohs (C/C++), binary.BigEndian (Go), or struct.pack("!H", ...) (Python) for multi-byte integers.
  • Socket leaks: Always close sockets in a finally/defer/RAII block. Leaked sockets exhaust the OS file descriptor limit (often 1024 by default).
  • Half-open connections: One side crashes without sending FIN. Use TCP keepalive (SO_KEEPALIVE) or application-level heartbeat messages to detect dead peers.
  • SIGPIPE: Writing to a closed socket sends SIGPIPE on Unix, which terminates the process by default. Handle or ignore it with signal(SIGPIPE, SIG_IGN) or use MSG_NOSIGNAL flag.
Key Rules
  • Always handle partial reads — loop until you receive the expected number of bytes or a delimiter
  • Use buffered I/O for performance (BufReader, bufio.Scanner, etc.) to reduce system call overhead

Web Frameworks & HTTP Servers Systems & Architecture
Web frameworks are the most common way to build APIs and services. Understanding the request-response lifecycle, routing, and middleware patterns helps you make better architectural decisions and debug production issues effectively.

Request-Response Lifecycle

An HTTP request arrives as a TCP connection. The server parses the request line (method, path, version), headers, and optional body. The framework routes the request to a handler based on the URL pattern and HTTP method. The handler processes the request (validates input, queries databases, applies business logic) and returns a response (status code, headers, body). Middleware functions run before and/or after the handler, adding cross-cutting concerns like logging, authentication, and error handling.

Routing

Routing maps URL patterns to handler functions. Static routes match exact paths (/users). Parameterized routes extract values from the URL (/users/:id or /users/{id}). Query parameters provide optional filters (/users?role=admin). HTTP method dispatch routes GET /users to one handler and POST /users to another. Most frameworks support route groups/prefixes for organizing related endpoints.

Middleware

Middleware wraps handlers to add functionality without modifying the handler itself. Common middleware includes: logging (record request/response details), authentication (verify tokens/sessions), CORS (add cross-origin headers), rate limiting (throttle requests), compression (gzip responses), and error handling (catch panics, format error responses). Middleware typically runs in a chain — the order of registration determines execution order.

Common Pitfalls
  • Not validating input: Always validate and sanitize request data — type, range, format. Never trust client input. Use schema validation (Pydantic, Zod, serde) rather than manual checks.
  • Blocking the event loop: In Node.js and Python async, CPU-heavy work in a handler blocks all other requests. Offload to worker threads or a task queue.
  • Missing CORS headers: Browser requests from different origins fail without proper Access-Control-Allow-Origin headers. Add CORS middleware early in development.
  • Error responses leaking internals: Never expose stack traces, SQL errors, or internal file paths in production error responses. Log the details server-side and return generic messages to clients.
  • Not setting Content-Type: Clients may misinterpret the response body. Always set Content-Type: application/json for JSON responses.

  • Always validate request input (type, range, format) before processing
  • Use middleware for cross-cutting concerns — logging, auth, CORS, rate limiting, error handling
  • Return appropriate HTTP status codes: 200 (OK), 201 (Created), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found), 500 (Internal Error)

  • Use structured logging with request IDs for traceability across services

Database Interaction — SQL Patterns Systems & Architecture

Why It Matters

Nearly every application needs persistent data. Understanding SQL interaction patterns — especially parameterized queries and connection management — is critical for security and performance. SQL injection remains one of the most common web vulnerabilities.

CRUD Operations

Create, Read, Update, Delete — the four basic operations on any data store. Every database library wraps these differently but the underlying SQL is the same: INSERT, SELECT, UPDATE, DELETE. Mastering CRUD in your language’s idiomatic style is the foundation of all database work.

Parameterized Queries

NEVER concatenate user input into SQL strings. Use parameterized queries (prepared statements) to prevent SQL injection. The database driver escapes and quotes values safely. Placeholders differ by language: Python uses ?, Go uses $1/$2, TypeScript (better-sqlite3) uses ?, Rust uses ?1/?2, and C++ uses positional sqlite3_bind_* calls.

Transactions

Group multiple operations into an atomic unit. Either all succeed or all roll back. Use BEGIN/COMMIT/ROLLBACK. Transactions are essential for data consistency — transferring money between accounts, creating an order with line items, or any operation where partial completion would leave data in a broken state.

Connection Pooling

Creating a new database connection is expensive — it involves TCP handshake, authentication, and memory allocation on both client and server. Connection pools maintain a set of reusable connections that are checked out and returned. Most production applications use a pool. SQLite is an exception since it’s an in-process database, but the pattern still applies to PostgreSQL, MySQL, and other client-server databases.

    cur = conn.cursor()
    try:
        # Check balance before transfer
        cur.execute("SELECT balance FROM accounts WHERE name = ?", (from_acct,))

Language Comparison

Python’s sqlite3 module is in the stdlib — zero dependencies. Go’s database/sql provides a universal interface; swap drivers without changing application code. Rust’s rusqlite automatically rolls back transactions on drop — RAII for database safety. TypeScript’s better-sqlite3 is synchronous, which simplifies transaction logic. C++ uses the raw SQLite C API, giving maximum control but requiring manual resource cleanup.

Common Pitfalls

  • SQL injection: The #1 database vulnerability. NEVER use string concatenation or interpolation for SQL with user input. Always use parameterized queries.
  • N+1 query problem: Fetching a list then querying for each item’s details individually. Use JOINs or batch queries instead.
  • Connection leaks: Always close connections and statements in finally/defer/Drop blocks. Leaked connections exhaust the pool and eventually bring the application to a halt.
  • Forgetting to commit: Without explicit COMMIT, some drivers auto-commit each statement, others roll back on close. Know your driver’s behavior and be explicit.
  • Prepared statement caching: Preparing the same query repeatedly wastes resources. Cache prepared statements or use a library that does it automatically.

Key Rules

  1. ALWAYS use parameterized queries for any user-provided data — no exceptions.
  2. Use transactions for operations that must be atomic — partial completion means corrupt data.
  3. Close connections and statements in finally/defer/RAII blocks — never rely on garbage collection.
  4. Use connection pooling in production — not a new connection per request.
  5. Batch related queries to avoid N+1 problems — prefer JOINs and WHERE IN clauses.
  6. Use migrations for schema changes — never modify production schemas by hand.

Why It Matters

A well-organized project structure makes code maintainable, testable, and collaborative. Build systems automate compilation, testing, and packaging. Understanding your language’s conventions prevents fights with tooling and makes onboarding new developers painless.

Project Layout Conventions

Each language community has settled on standard directory structures. Python favors the src layout with src/packagename/ keeping source separate from tests and config. Go uses a flat structure where each directory is a package — no src folder needed. TypeScript projects place source in src/ with compiled output in dist/. Rust uses src/main.rs for binaries and src/lib.rs for libraries, with Cargo enforcing the convention. C++ traditionally separates include/ for headers and src/ for implementation files, though no universal standard exists.

Dependency Management

Python’s ecosystem offers pip with requirements.txt or the more modern poetry and uv with pyproject.toml. Go has built-in module support via go mod — no external tool required. TypeScript/JavaScript relies on npm (or yarn/pnpm) with package.json. Rust’s cargo handles dependencies, building, testing, and publishing in one tool. C++ uses vcpkg or conan for package management alongside CMake for build configuration.

Build Configuration

Build configuration controls compiler flags, optimization levels, debug vs. release modes, and CI/CD integration. Debug builds include symbols and disable optimizations for easier debugging. Release builds enable optimizations and strip debug info. A proper CI/CD pipeline runs linting, testing, and building automatically on every push, catching issues before they reach production.

Cross-Language Comparison

Cargo (Rust) is widely considered the gold standard — it handles building, testing, benchmarking, documentation, and publishing in a single integrated tool with consistent conventions. Go takes a similar integrated approach with go build, go test, and go mod built into the language toolchain, trading flexibility for simplicity. npm (TypeScript/JavaScript) boasts the largest package ecosystem but relies on many third-party tools for tasks like building and linting. Python has historically been fragmented across setuptools, pip, poetry, hatch, and uv, but the ecosystem is unifying around pyproject.toml as the standard configuration file. CMake (C++) has the steepest learning curve and the most verbose configuration, but remains the de facto standard for cross-platform C++ projects.

Common Pitfalls

  • Python: Always pin dependency versions in production. Unpinned dependencies lead to “works on my machine” failures when a transitive dependency releases a breaking change.
  • Go: Circular imports between packages are a compile-time error with no workaround. Design package boundaries carefully — extract shared types into a separate package.
  • TypeScript: Path aliases defined in tsconfig.json (e.g., @app/*) are only understood by tsc. At runtime, Node.js needs a resolver like tsconfig-paths or bundler configuration to map them.
  • Rust: Commit Cargo.lock for binary projects to ensure reproducible builds, but do not commit it for library crates — let downstream consumers resolve their own dependency versions.
  • C++: Debugging find_package() failures is a rite of passage. Use cmake --debug-find and set CMAKE_PREFIX_PATH to point at your dependency install locations.

Key Rules to Remember

  1. Follow idiomatic layout — every language has conventions; fighting them wastes time and confuses contributors.
  2. Use lock files — poetry.lock, go.sum, package-lock.json, Cargo.lock ensure reproducible builds across machines and CI.
  3. Separate source and build directories — never mix generated artifacts with source code.
  4. Adopt workspace patterns early — splitting a monolith into packages later is far harder than starting with a modular layout.
  5. Automate with CI — lint, test, and build on every push so broken code never reaches the main branch.
  6. Keep it simple — resist over-engineering the build system. Start with the defaults and customize only when you have a concrete need.

Concurrency Patterns II — Producer/Consumer, Rate Limiting, Circuit Breaker Systems & Architecture
Production systems need controlled concurrency. Unbounded parallelism leads to resource exhaustion, cascading failures, and poor user experience. These patterns are the building blocks of resilient services — they appear in every production system from web servers to message queues to microservice meshes.

Producer-Consumer Pattern

The producer-consumer pattern decouples data generation from data processing. Producers push work items onto a shared queue; consumers pull items and process them independently. This enables backpressure (producers slow down when the queue is full), buffering (smooth out bursts), and independent scaling (add more consumers without changing producers). The queue is the key abstraction — it can be in-memory (channel, deque, queue) or external (Redis, Kafka, RabbitMQ).

Rate Limiting (Token Bucket)

Rate limiting controls the rate of operations to protect resources. The token bucket algorithm works as follows: a bucket holds a maximum of N tokens, refilled at a steady rate of R tokens per second. Each operation consumes one token. If the bucket is empty, the operation is either rejected immediately or delayed until a token becomes available. This is the most common algorithm used in API rate limiting, network traffic shaping, and request throttling. Variants include sliding window (counts requests in a rolling time window), leaky bucket (constant outflow rate), and fixed window (counts per fixed time interval).

Circuit Breaker State Machine

The circuit breaker pattern prevents cascading failures by failing fast when a downstream service is unhealthy. It has three states: Closed (normal operation, requests pass through to the downstream service), Open (failing fast, all requests are immediately rejected without calling the downstream service), and Half-Open (recovery testing, a limited number of requests are allowed through to check if the downstream service has recovered). The circuit transitions from Closed to Open when the failure count exceeds a threshold, from Open to Half-Open after a cooldown timer expires, and from Half-Open back to either Closed (on success) or Open (on failure).

class BoundedQueue<T> {
    private items: T[] = [];
    private maxSize: number;
    private waitingProducers: Array<{ resolve: () => void; item: T }> = [];
    private waitingConsumers: Array<{ resolve: (item: T) => void }> = [];
    private closed = false;

    constructor(maxSize: number) {
        this.maxSize = maxSize;
    }

    async enqueue(item: T): Promise<void> {
        if (this.waitingConsumers.length > 0) {
            const consumer = this.waitingConsumers.shift()!;
            consumer.resolve(item);
            return;
        }
        if (this.items.length < this.maxSize) {
            this.items.push(item);
            return;
        }
        // Queue full: wait for space (backpressure)
        return new Promise<void>((resolve) => {
            this.waitingProducers.push({ resolve, item });
        });
    }

    async dequeue(): Promise<T | null> {
        if (this.items.length > 0) {
            const item = this.items.shift()!;
            if (this.waitingProducers.length > 0) {
                const producer = this.waitingProducers.shift()!;
                this.items.push(producer.item);
                producer.resolve();
            }
            return item;
        }
        if (this.closed) return null;
        return new Promise<T | null>((resolve) => {
            this.waitingConsumers.push({ resolve: resolve as (item: T) => void });
        });
    }

    close(): void {
        this.closed = true;
        this.waitingConsumers.forEach((c) => c.resolve(null as any));
        this.waitingConsumers = [];
    }
}

    std::queue<T> queue_;
    size_t maxSize_;
    std::mutex mutex_;
    std::condition_variable notFull_;
    std::condition_variable notEmpty_;
    bool closed_ = false;

public:
    explicit BoundedQueue(size_t maxSize) : maxSize_(maxSize) {}

    void push(T item) {
        std::unique_lock<std::mutex> lock(mutex_);
        notFull_.wait(lock, [this] { return queue_.size() < maxSize_ || closed_; });
        if (closed_) return;
        queue_.push(std::move(item));
        notEmpty_.notify_one();
    }

    bool pop(T& item) {
        std::unique_lock<std::mutex> lock(mutex_);
        notEmpty_.wait(lock, [this] { return !queue_.empty() || closed_; });
        if (queue_.empty()) return false; // closed and empty
        item = std::move(queue_.front());
        queue_.pop();
        notFull_.notify_one();
        return true;
    }

    void close() {
        std::lock_guard<std::mutex> lock(mutex_);
        closed_ = true;
        notEmpty_.notify_all();
        notFull_.notify_all();
    }
};

class TokenBucket:
    """Rate limiter using the token bucket algorithm."""

    def __init__(self, rate: float, capacity: int):
        self.rate = rate          # Tokens added per second
        self.capacity = capacity  # Maximum tokens in bucket
        self.tokens = float(capacity)
        self.last_refill = time.monotonic()
        self.lock = threading.Lock()

    def _refill(self) -> None:
        now = time.monotonic()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_refill = now

    def acquire(self) -> bool:
        """Try to consume one token. Returns True if allowed."""
        with self.lock:
            self._refill()
            if self.tokens >= 1.0:
                self.tokens -= 1.0
                return True
            return False

    def wait_and_acquire(self) -> float:
        """Block until a token is available. Returns wait time."""
        while True:
            with self.lock:
                self._refill()
                if self.tokens >= 1.0:
                    self.tokens -= 1.0
                    return 0.0
                # Calculate wait time for next token
                wait_time = (1.0 - self.tokens) / self.rate
            time.sleep(wait_time)

        if self.state != new_state:
            print(f"Circuit: {self.state.value} -> {new_state.value}")
            self.state = new_state

    def call(self, func: Callable[..., T], *args, **kwargs) -> T:
        with self.lock:
            if self.state == State.OPEN:
                if time.monotonic() - self.last_failure_time >= self.recovery_timeout:
                    self._transition_to(State.HALF_OPEN)
                else:
                    raise CircuitBreakerError(
                        f"Circuit is OPEN. Retry after {self.recovery_timeout}s"
                    )

        try:
            result = func(*args, **kwargs)
            with self.lock:
                if self.state == State.HALF_OPEN:
                    self._transition_to(State.CLOSED)
                self.failure_count = 0
            return result
        except Exception as e:
            with self.lock:
                self.failure_count += 1
                self.last_failure_time = time.monotonic()
                if self.failure_count >= self.failure_threshold:
                    self._transition_to(State.OPEN)
                if self.state == State.HALF_OPEN:
                    self._transition_to(State.OPEN)
            raise

Language Comparison: Go channels are purpose-built for producer-consumer — a buffered channel IS a bounded queue with zero extra code needed. Rust’s ownership model prevents data races at compile time, but std::sync::mpsc is single-consumer; use crossbeam_channel for multi-producer multi-consumer. Python’s queue.Queue is thread-safe but GIL-limited for CPU work — use multiprocessing.Queue for true parallelism. C++ requires manual synchronization with mutexes and condition variables, offering maximum control but also maximum footgun potential. TypeScript’s single-threaded event loop means these patterns are more about async coordination than thread safety.
Common Pitfalls
  • Unbounded queues: Without a max size, producers can overwhelm consumers and exhaust memory. Always set a bound and handle the “queue full” case (block, drop, or reject).
  • Rate limiter clock drift: Using wall clock (time.time(), Date.now()) can jump during NTP adjustments. Use monotonic clocks (time.monotonic(), Instant, steady_clock).
  • Circuit breaker thrashing: Setting thresholds too low causes rapid open/close cycling. Use a half-open probe with gradual traffic increase rather than an instant all-or-nothing switch.
  • Deadlocks in producer-consumer: If producers and consumers share a lock beyond the queue itself, a full queue combined with lock contention can deadlock. Keep locks narrow and use separate locks for enqueue and dequeue paths.
  • Forgetting to handle the “open” state: Callers must handle circuit breaker rejections gracefully — return a cached response, use a fallback service, or queue the request for later retry.
Key Rules
  • Always bound your queues — unbounded queues are memory leaks waiting to happen
  • Use monotonic clocks for rate limiting and timeouts (never wall clock)
  • Circuit breaker thresholds should be tuned per service based on observed failure rates and SLAs
  • Test concurrent code with race detectors (go test -race, ThreadSanitizer, etc.)
  • Prefer language-native concurrency primitives (Go channels, Rust channels) over manual mutex + condition variable
  • Monitor and alert on circuit breaker state changes in production — an open circuit is a signal that something is wrong downstream
