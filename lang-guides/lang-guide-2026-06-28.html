1. Tuples, Structs & Records Structured Data rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: Group heterogeneous fields into one value. Tuples are positional and anonymous; structs/records add named fields. Prefer named fields once a tuple has 3+ elements or leaves the local scope. Destructuring (pattern binding / structured bindings) unpacks fields back out in every language.

Key Pattern — define a named record and destructure it

Python 
from dataclasses import dataclass
@dataclass
class Point:
    x: int
    y: int
p = Point(3, 4)
print(p.x, p.y)               # 3 4
3 4
Go 
type Point struct{ X, Y int }
p := Point{X: 3, Y: 4}
fmt.Println(p.X, p.Y)         // 3 4
3 4
TypeScript 
type Point = { x: number; y: number };
const p: Point = { x: 3, y: 4 };
const { x, y } = p;           // destructure
console.log(x, y);            // 3 4
3 4
Rust 
struct Point { x: i32, y: i32 }
let p = Point { x: 3, y: 4 };
let Point { x, y } = p;       // destructure
println!("{} {}", x, y);      // 3 4
3 4
C++ 
struct Point { int x, y; };
Point p{3, 4};
auto [x, y] = p;              // structured bindings (C++17)
std::cout << x << " " << y;   // 3 4
3 4

#1 Pitfall per Language:
  • Python: a plain @dataclass is mutable and unhashable by default — use frozen=True for a value/record type.
  • Go: structs are copied by value on assignment and function calls — pass *Point when you need to mutate the original.
  • TypeScript: a tuple type [number, number] is just an array at runtime — nothing stops .push() unless it's readonly.
  • Rust: destructuring by value moves non-Copy fields out — borrow with let Point { x, y } = &p; to keep p usable.
  • C++: structured bindings bind to copies unless you write auto& [x, y] = p; — mutating a copy won't touch p.
2. Sorting & Comparators Sorting rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: A comparator (or key function) defines the order. Multi-key sorts compose comparisons: primary, then tie-breaker. Built-in sorts are stable in Python/JS/Go(SliceStable)/Rust, but not guaranteed stable in C++ std::sort (use std::stable_sort). Prefer a key/projection when one exists — it computes each key once.

Key Pattern — sort by age descending, then name ascending

Python 
people = [("bob", 30), ("amy", 25), ("cat", 30)]
people.sort(key=lambda p: (-p[1], p[0]))
print(people)   # [('amy', 25), ('bob', 30), ('cat', 30)]
[('amy', 25), ('bob', 30), ('cat', 30)]
Go 
sort.Slice(people, func(i, j int) bool {
    if people[i].age != people[j].age {
        return people[i].age > people[j].age
    }
    return people[i].name < people[j].name
})
[{amy 25} {bob 30} {cat 30}]
TypeScript 
people.sort((a, b) => b.age - a.age || a.name.localeCompare(b.name));
console.log(people);
// [{name:'amy',age:25},{name:'bob',age:30},{name:'cat',age:30}]
[{name:'amy',age:25},{name:'bob',age:30},{name:'cat',age:30}]
Rust 
// people: Vec<(&str, i32)>
people.sort_by(|a, b| b.1.cmp(&a.1).then(a.0.cmp(b.0)));
println!("{:?}", people); // [("amy", 25), ("bob", 30), ("cat", 30)]
[("amy", 25), ("bob", 30), ("cat", 30)]
C++ 
std::sort(people.begin(), people.end(), [](auto& a, auto& b){
    return a.age != b.age ? a.age > b.age : a.name < b.name;
});
// amy 25 | bob 30 | cat 30
amy 25 | bob 30 | cat 30
Challenge (all 5 languages):
1

Verify the sort is stable for fully-equal records by tagging each with its original index and confirming the indices stay ascending among equals (in C++ prove you needed stable_sort).
#1 Pitfall per Language:
  • Python: mixing reverse=True with a multi-key tuple reverses everything — negate the numeric key instead for per-field direction.
  • Go: a comparator that returns <= (not a strict less-than) breaks the sort invariant and can panic or misorder.
  • TypeScript: arr.sort() with no comparator sorts lexicographically — [10, 2, 1] becomes [1, 10, 2].
  • Rust: sort() needs Ord; for floats or custom order use sort_by/sort_by_key (and total_cmp for f64).
  • C++: the comparator must be a strict weak ordering — using <= is UB; and std::sort isn't stable.
3. Linked Lists & Trees Recursive Structures rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: Node-based recursive structures: each node owns a link to the next/child. Ownership is the crux — Python/JS/Go rely on GC, Rust needs Option<Box<Node>> (and Rc<RefCell> for shared/back-links), C++ needs unique_ptr/manual delete. Reversing a list is the canonical pointer-rewiring drill.

Key Pattern — reverse a singly linked list iteratively (1→2→3 becomes 3→2→1)

Python 
def reverse(head):
    prev = None
    while head:
        head.next, prev, head = prev, head, head.next
    return prev
# 1->2->3  =>  3->2->1
3 -> 2 -> 1
Go 
func reverse(head *Node) *Node {
    var prev *Node
    for head != nil {
        head.Next, prev, head = prev, head, head.Next
    }
    return prev
}
3 -> 2 -> 1
TypeScript 
function reverse(head: Node | null): Node | null {
  let prev: Node | null = null;
  while (head) {
    const nxt = head.next;
    head.next = prev; prev = head; head = nxt;
  }
  return prev;
}
3 -> 2 -> 1
Rust 
fn reverse(mut head: Option<Box<Node>>) -> Option<Box<Node>> {
    let mut prev = None;
    while let Some(mut node) = head {
        head = node.next.take();   // detach rest
        node.next = prev;          // rewire
        prev = Some(node);
    }
    prev
}
3 -> 2 -> 1
C++ 
Node* reverse(Node* head) {
    Node* prev = nullptr;
    while (head) {
        Node* nxt = head->next;
        head->next = prev; prev = head; head = nxt;
    }
    return prev;
}
3 -> 2 -> 1
Challenge (all 5 languages):
1
Build a binary tree, then return its values in level order (BFS) using a queue. Next do an in-order traversal recursively. In Rust represent children as Option<Box<Node>> and traverse by reference (no moves); in C++ use unique_ptr children and confirm no leaks with a destructor that prints.
#1 Pitfall per Language:
  • Python: deep recursion on a long list/tree hits the ~1000-frame limit — iterate or raise sys.setrecursionlimit carefully.
  • Go: forgetting to advance head before rewiring Next creates a self-loop — capture next first.
  • TypeScript: losing the next reference before reassigning drops the tail — save it in a temp.
  • Rust: a doubly-linked list with Box can't have back-pointers (single ownership) — use Rc<RefCell> + Weak for parents.
  • C++: raw new nodes leak without delete — prefer unique_ptr, but then reversing needs std::move/.release().
4. Iterators & Generators Iteration rep 3 • 18 days since review
Due today • 4th review (overdue 3 days)
Quick Recall: Iterators produce values on demand; generators build lazy iterators with yield. Chaining map/filter/take stays lazy — nothing runs until you consume (collect, list(), a for). This lets you work with infinite or huge sequences in constant memory.

Key Pattern — first 3 squares of even numbers, lazily

Python 
from itertools import count, islice
gen = (x * x for x in count() if x % 2 == 0)
print(list(islice(gen, 3)))     # [0, 4, 16]
[0, 4, 16]
Go 
// Go 1.23 iterators: pull with a plain loop and break
out := []int{}
for x := 0; len(out) < 3; x++ {
    if x%2 == 0 { out = append(out, x*x) }
}
fmt.Println(out)                // [0 4 16]
[0 4 16]
TypeScript 
function* evenSquares() {
  for (let x = 0; ; x++) if (x % 2 === 0) yield x * x;
}
const g = evenSquares();
console.log([g.next().value, g.next().value, g.next().value]); // [0, 4, 16]
[ 0, 4, 16 ]
Rust 
let v: Vec<i32> = (0..)
    .filter(|x| x % 2 == 0)
    .map(|x| x * x)
    .take(3)
    .collect();
println!("{:?}", v);            // [0, 4, 16]
[0, 4, 16]
C++ 
#include <ranges>
namespace vs = std::views;
auto r = vs::iota(0)
       | vs::filter([](int x){ return x % 2 == 0; })
       | vs::transform([](int x){ return x * x; })
       | vs::take(3);
for (int v : r) std::cout << v << " "; // 0 4 16
0 4 16

#1 Pitfall per Language:
  • Python: a generator is single-use — after one pass it's exhausted; re-create it to iterate again.
  • Go: a channel-based generator leaks a goroutine if the consumer stops early — pass a context/done channel.
  • TypeScript: spreading an infinite generator ([...gen]) hangs — bound it before collecting.
  • Rust: iterator adapters are lazy and do nothing until consumed — forgetting collect()/for means no work happens (a warning fires).
  • C++: a view stores references to its source range — if the source is a temporary that dies, iterating the view is UB (dangling).
5. Pattern Matching Control Flow rep 3 • 17 days since review
Due today • 4th review (overdue 2 days)
Quick Recall: Match dispatches on a value's shape, not just equality — destructuring bindings in one step. Rust match is exhaustive (compile error if a case is missing); Python 3.10 match and C++ std::visit approximate it; Go uses type switches; TS uses discriminated unions + switch.

Key Pattern — classify a value by shape

Python 
def kind(x):
    match x:
        case 0:        return "zero"
        case int():    return "int"
        case str():    return "str"
        case _:        return "other"
print(kind(0), kind(5), kind("a"))   # zero int str
zero int str
Go 
func kind(x any) string {
    switch v := x.(type) {
    case int:    if v == 0 { return "zero" }; return "int"
    case string: return "str"
    default:     return "other"
    }
}
fmt.Println(kind(0), kind(5), kind("a")) // zero int str
zero int str
TypeScript 
type Shape =
  | { kind: "circle"; r: number }
  | { kind: "rect"; w: number; h: number };
const area = (s: Shape) =>
  s.kind === "circle" ? Math.PI * s.r ** 2 : s.w * s.h;
console.log(area({ kind: "rect", w: 2, h: 3 })); // 6
6
Rust 
enum Shape { Circle(f64), Rect(f64, f64) }
fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle(r) => std::f64::consts::PI * r * r,
        Shape::Rect(w, h) => w * h,   // exhaustive: both arms required
    }
}
println!("{}", area(&Shape::Rect(2.0, 3.0))); // 6

C++

struct Circle { double r; };
struct Rect { double w, h; };
using Shape = std::variant<Circle, Rect>;
double area(const Shape& s) {
    return std::visit([](auto& v) -> double {
        using T = std::decay_t<decltype(v)>;
        if constexpr (std::is_same_v<T, Circle>) return 3.14159 * v.r * v.r;
        else return v.w * v.h;
    }, s);
}

  • Python: case Point(x, y) requires __match_args__; a bare name like case foo: is a capture (always matches), not a constant compare.

6. Error Handling Control Flow rep 3 • 17 days since review
Due today • 4th review (overdue 2 days)
Quick Recall: Four philosophies: exceptions (Python, C++, JS/TS throw), error values (Go's (T, error)), and typed results (Rust Result<T, E>, C++23 std::expected). Rust's ? and Go's if err != nil make failure explicit in the signature; exceptions make the happy path cleaner but hide control flow.

Key Pattern — parse an int and propagate failure

Python 
def double(s):
    try:
        return int(s) * 2, None
    except ValueError as e:
        return None, str(e)
print(double("21"))   # (42, None)
print(double("x")[1]) # invalid literal for int() with base 10: 'x'
(42, None) invalid literal for int() with base 10: 'x'
Go 
func double(s string) (int, error) {
    n, err := strconv.Atoi(s)
    if err != nil { return 0, fmt.Errorf("double: %w", err) }
    return n * 2, nil
}
v, err := double("21"); fmt.Println(v, err) // 42 <nil>
42 <nil>
TypeScript 
function double(s: string): number {
  const n = Number(s);
  if (Number.isNaN(n)) throw new Error(`bad number: ${s}`);
  return n * 2;
}
console.log(double("21")); // 42
42
Rust 
fn double(s: &str) -> Result<i32, std::num::ParseIntError> {
    let n: i32 = s.parse()?;   // ? propagates the Err
    Ok(n * 2)
}
println!("{:?}", double("21")); // Ok(42)
println!("{}", double("x").is_err()); // true
Ok(42) true
C++ 
#include <expected>   // C++23
#include <string>
std::expected<int, std::string> doubler(const std::string& s) {
    try { return std::stoi(s) * 2; }
    catch (...) { return std::unexpected("bad number: " + s); }
}
// doubler("21") -> 42 ; doubler("x").has_value() -> false
42 false
Challenge (all 5 languages):
1
Write parse_config(lines) that reads key=value lines and returns either the parsed map or an error identifying the first bad line (with its line number). Use Result + ? in Rust, wrapped errors (%w) in Go, a typed exception in Python/TS, and std::expected in C++. Then add error context so a caller can tell parsing from I/O failures.
#1 Pitfall per Language:
  • Python: bare except: swallows KeyboardInterrupt/SystemExit too — catch the specific exception type.
  • Go: ignoring the error with _ or shadowing err in an inner scope hides real failures.
  • TypeScript: a caught value is typed unknown — you must narrow (e instanceof Error) before reading .message.

Probabilistic & Space-Efficient Structures

Two workhorses of high-scale systems: a Bloom filter answers "have I seen this?" in a few bits with a tunable false-positive rate, and an LRU cache serves hot data in O(1) while bounding memory. Both trade a little accuracy or history for large savings in space and time.

Bloom Filters Probabilistic Structures
Why it matters: a Bloom filter tests set membership using a tiny bit array and k hash functions — often 100× smaller than the set itself. It answers "definitely not present" or "probably present" (never a false negative). Databases (Cassandra, RocksDB), CDNs, and crawlers use it to skip expensive disk/network lookups for keys that can't exist.

Insert: hash the item with k functions and set those k bits. Query: if any of those bits is 0, the item was never added (no false negatives); if all are 1, it's probably present (a false positive is possible because other items may have set those same bits). You cannot delete from a plain Bloom filter (unsetting a bit could break other members). The examples below use a 10-bit filter with two hashes, h1(x)=x%10 and h2(x)=(x*7)%10, so every bit is hand-verifiable.

Example 1 — add {3, 14}; query a member and a non-member

Python 
class Bloom:
    def __init__(self, m=10): self.m, self.bits = m, 0
    def _idx(self, x): return (x % self.m, (x * 7) % self.m)
    def add(self, x):
        for i in self._idx(x): self.bits |= 1 << i
    def __contains__(self, x):
        return all(self.bits >> i & 1 for i in self._idx(x))

b = Bloom()
for x in (3, 14): b.add(x)      # bits set: 1,3,4,8
print(3 in b, 7 in b)           # True False
True False
Go 
type Bloom struct{ m uint; bits uint64 }
func (b *Bloom) idx(x uint) (uint, uint) { return x % b.m, (x * 7) % b.m }
func (b *Bloom) Add(x uint) { i, j := b.idx(x); b.bits |= 1<<i | 1<<j }
func (b *Bloom) Has(x uint) bool {
    i, j := b.idx(x)
    return b.bits>>i&1 == 1 && b.bits>>j&1 == 1
}
func main() {
    b := &Bloom{m: 10}
    b.Add(3); b.Add(14)
    fmt.Println(b.Has(3), b.Has(7)) // true false
}
true false

  private bits = 0n;
  constructor(private m = 10) {}
  private idx(x: number): [number, number] { return [x % this.m, (x * 7) % this.m]; }
  add(x: number) { const [i, j] = this.idx(x); this.bits |= (1n << BigInt(i)) | (1n << BigInt(j)); }

    const [i, j] = this.idx(x);
    return ((this.bits >> BigInt(i)) & 1n) === 1n && ((this.bits >> BigInt(j)) & 1n) === 1n;
  }
}

b.add(3); b.add(14);

Example 2 — a false positive: 24 was never added, yet reports "present"

With {3, 14} added, bits {1, 3, 4, 8} are set. Now h1(24)=4 and h2(24)=(24*7)%10=8 — both already set by other items, so the filter wrongly says "probably present."

Python 
print(24 in b)   # True  <-- false positive (24 was never added)
True
Go 
fmt.Println(b.Has(24)) // true  <-- false positive
true
TypeScript 
console.log(b.has(24)); // true  <-- false positive
true
Rust 
println!("{}", b.has(24)); // true  <-- false positive
true
C++ 
std::cout << b.has(24); // 1  <-- false positive
1
Language Comparison: The algorithm is identical; the ergonomics differ. Python's arbitrary-precision int doubles as an unbounded bit set, and __contains__ makes x in b read naturally. Go, Rust, and C++ use a fixed uint64 here, but a real filter needs thousands of bits — reach for a []uint64 word array, Rust's bitvec/Vec<u64>, or C++ std::vector<bool>/bitset. TypeScript must use BigInt for bit indices > 31 because number bitwise ops truncate to 32 bits. For production, size the filter with the standard formulas: m = -n·ln(p)/(ln2)² bits and k = (m/n)·ln2 hashes for n items and target false-positive rate p.
Common Mistakes:
❌ [All] Trying to delete from a plain Bloom filter by clearing bits — it can silently evict other members (false negatives).
✅ [All] Use a counting Bloom filter (per-slot counters) if deletion is required, or rebuild periodically.
❌ [All] Treating "probably present" as "present" and skipping the authoritative lookup — false positives are guaranteed at scale.
✅ [All] Use the filter only to skip work on "definitely absent"; always confirm a positive against the real store.
❌ [TypeScript] Using number bitwise (1 << i) for i ≥ 31 — it overflows/sign-flips (32-bit ops).
✅ [TypeScript] Use BigInt (1n << BigInt(i)) or a Uint8Array/typed-array word buffer.
❌ [All] Using two correlated hashes (e.g. h and 2h) — correlation spikes the false-positive rate.
✅ [All] Derive k indices via double hashing: h1 + i·h2 with independent h1, h2.
Key Rules:
  • No false negatives, tunable false positives; membership only — it stores no keys, so you can't enumerate or delete.
  • Size m and choose k from n and target p; too-small m saturates all bits to 1 (everything "present").
  • Use it as a cheap negative filter in front of an expensive authoritative lookup.

LRU Cache (O(1) Get/Put) Data Structures
Why it matters: a fixed-capacity cache that evicts the least-recently-used entry is the standard way to bound memory while keeping hot data fast. It powers page caches, DB buffer pools, HTTP caches, and memoization tables. The canonical design — a hash map for O(1) lookup plus a doubly linked list for O(1) recency updates — recurs in every language (and is LeetCode 146).

Every get and put moves the touched key to the "most recent" end; on overflow you evict from the "least recent" end. The hash map gives O(1) key→node; the linked list gives O(1) reordering. Several languages ship an ordered map that does the list part for you: Python OrderedDict.move_to_end, and JS Map preserves insertion order (delete + re-set to promote). The trace below uses capacity 2 and returns -1 on a miss.

Example 1 — capacity 2: put 1,2; get 1; put 3 (evicts 2); get 2 → miss

Python 
from collections import OrderedDict
class LRU:
    def __init__(self, cap): self.cap, self.d = cap, OrderedDict()
    def get(self, k):
        if k not in self.d: return -1
        self.d.move_to_end(k)          # mark most-recent
        return self.d[k]
    def put(self, k, v):
        if k in self.d: self.d.move_to_end(k)
        self.d[k] = v
        if len(self.d) > self.cap: self.d.popitem(last=False)  # evict LRU

c = LRU(2)
c.put(1, 1); c.put(2, 2)
print(c.get(1))   # 1
c.put(3, 3)       # evicts key 2
print(c.get(2))   # -1
print(c.get(3))   # 3
1 -1 3
Go 
import "container/list"
type entry struct{ k, v int }
type LRU struct {
    cap int
    ll  *list.List
    m   map[int]*list.Element
}
func NewLRU(cap int) *LRU { return &LRU{cap, list.New(), map[int]*list.Element{}} }
func (c *LRU) Get(k int) int {
    if e, ok := c.m[k]; ok { c.ll.MoveToFront(e); return e.Value.(*entry).v }
    return -1
}
func (c *LRU) Put(k, v int) {
    if e, ok := c.m[k]; ok { e.Value.(*entry).v = v; c.ll.MoveToFront(e); return }
    c.m[k] = c.ll.PushFront(&entry{k, v})
    if c.ll.Len() > c.cap {
        back := c.ll.Back(); c.ll.Remove(back); delete(c.m, back.Value.(*entry).k)
    }
}
// Get(1)=1 ; Put(3) evicts 2 ; Get(2)=-1 ; Get(3)=3
1 -1 3

class LRU {
  private m = new Map<number, number>();   // Map keeps insertion order
  constructor(private cap: number) {}
  get(k: number): number {
    if (!this.m.has(k)) return -1;
    const v = this.m.get(k)!;
    this.m.delete(k); this.m.set(k, v);     // re-insert = most recent
    return v;
  }
  put(k: number, v: number): void {
    if (this.m.has(k)) this.m.delete(k);
    this.m.set(k, v);
    if (this.m.size > this.cap) this.m.delete(this.m.keys().next().value); // evict oldest
  }
}

use std::collections::HashMap;
// Recency as a Vec of keys (front = LRU). O(n) touch; production: the `lru` crate is O(1).
struct Lru { cap: usize, map: HashMap<i32, i32>, order: Vec<i32> }
impl Lru {
    fn new(cap: usize) -> Self { Lru { cap, map: HashMap::new(), order: vec![] } }
    fn touch(&mut self, k: i32) {
        self.order.retain(|&x| x != k);
        self.order.push(k);            // most-recent at the back
    }
    fn get(&mut self, k: i32) -> i32 {
        if let Some(&v) = self.map.get(&k) { self.touch(k); v } else { -1 }
    }
    fn put(&mut self, k: i32, v: i32) {
        self.map.insert(k, v); self.touch(k);
        if self.map.len() > self.cap {
            let lru = self.order.remove(0);
            self.map.remove(&lru);
        }
    }
}
// get(1)=1 ; put(3) evicts 2 ; get(2)=-1 ; get(3)=3

C++ 
#include <list>
#include <unordered_map>
class LRU {
    int cap;
    std::list<std::pair<int,int>> ll;   // front = most-recent
    std::unordered_map<int, std::list<std::pair<int,int>>::iterator> m;
public:
    explicit LRU(int c) : cap(c) {}
    int get(int k) {
        auto it = m.find(k);
        if (it == m.end()) return -1;
        ll.splice(ll.begin(), ll, it->second);   // move node to front, O(1)
        return it->second->second;
    }
    void put(int k, int v) {
        auto it = m.find(k);
        if (it != m.end()) { it->second->second = v; ll.splice(ll.begin(), ll, it->second); return; }
        ll.emplace_front(k, v); m[k] = ll.begin();
        if ((int)m.size() > cap) { m.erase(ll.back().first); ll.pop_back(); }
    }
};
// get(1)=1 ; put(3) evicts 2 ; get(2)=-1 ; get(3)=3

Language Comparison: The textbook design is hash map + intrusive doubly linked list, which Go (container/list) and C++ (std::list + iterators, with O(1) splice) express directly. Python and JS/TS get it almost for free from an insertion-ordered map (OrderedDict; Map) — no manual pointers. Rust is the outlier: a safe intrusive doubly linked list fights the borrow checker (you need Rc<RefCell> or raw pointers), so the idiomatic move is to use the lru crate (or indexmap); the Vec-of-keys version above is simple but O(n) per touch. Watch the pattern: store the list node/iterator in the map, not just the value, so promotion and eviction stay O(1).
Common Mistakes:
❌ [All] Updating the value on put but forgetting to promote recency — the entry can then be wrongly evicted next.
✅ [All] Treat both get and put (hit or update) as a "touch" that moves the key to most-recent.
❌ [C++] Invalidating the stored iterator by copying/re-inserting the node — std::list iterators must stay stable; use splice, not erase+insert.
✅ [C++] ll.splice(ll.begin(), ll, it) moves the node and keeps every iterator valid — O(1) and safe.
❌ [Rust] Reaching for a hand-rolled Box doubly linked list — back-pointers violate single ownership and won't compile.
✅ [Rust] Use the lru crate for O(1), or accept an IndexMap/Vec ordering for small caches.
❌ [Go] Storing the value in the map and the list separately — they drift out of sync on eviction.
✅ [Go] Store the value in the list element and map keys → *list.Element; delete from the map using the evicted element's key.
Key Rules:
  • Map for O(1) lookup + doubly linked list for O(1) recency = O(1) get/put; keep them in lockstep on eviction.
  • Prefer the built-in ordered map (Python OrderedDict, JS Map) or a library (Rust lru) over hand-rolling pointers.
  • Both reads and writes update recency; evict from the least-recent end only when over capacity.

Distributed Coordination

When state is spread across many nodes, two questions dominate: which node owns this key? and how do I make an operation safe to repeat? Consistent hashing answers the first with minimal reshuffling on membership changes; idempotency keys answer the second so retries never double-charge.

Consistent Hashing Distributed Systems
Why it matters: naive hash(key) % N sharding remaps almost every key when N changes, causing a cache stampede or full data reshuffle. Consistent hashing places nodes and keys on a ring so that adding/removing one node moves only ~1/N of keys. It underpins Memcached clients, DynamoDB, Cassandra, and CDN request routing.

Map both nodes and keys onto a circular hash space (here 0..99). A key is owned by the first node found walking clockwise from the key's position (wrapping past the top back to the start). Add a node and only the keys between it and its predecessor move; remove one and only its keys shift to the next node. Real systems add many virtual nodes per physical node to smooth the load. The examples hash by x % 100 so ring positions are obvious.

Example 1 — ring with nodes at 10, 40, 80; route three keys

Python 
import bisect
class Ring:
    def __init__(self, nodes):
        self.pts = sorted(n % 100 for n in nodes)   # [10, 40, 80]
    def route(self, key):
        p = key % 100
        i = bisect.bisect_left(self.pts, p)
        return self.pts[i % len(self.pts)]           # wrap clockwise

r = Ring([10, 40, 80])
print(r.route(5), r.route(45), r.route(90))  # 10 80 10

Language Comparison: The core is a sorted array + binary search for the successor, and each language spells the search differently: Python bisect_left, Go sort.SearchInts, Rust partition_point, C++ lower_bound, and a manual scan (or a custom binary search) in TS. All share the same wrap-around rule: if the search runs off the end, route to index 0. For mutation-heavy rings, Go/C++/Rust would keep the ring in a balanced tree (std::map, BTreeMap) for O(log n) inserts; Python's bisect.insort is O(n) but fine for a handful of nodes × vnodes. Whatever the language, use a well-distributed hash (SHA-1/xxHash) in production — the % 100 here is only for legibility.
Common Mistakes:
❌ [All] Falling back to hash(key) % N sharding — changing N remaps ~all keys, melting caches and triggering mass data movement.
✅ [All] Use a hash ring with successor lookup so a membership change moves only ~1/N of keys.
❌ [All] One point per physical node — load is lumpy and removing a node dumps its entire share on a single neighbor.
✅ [All] Add many virtual nodes per server (e.g. 100–200) to spread keys and smooth rebalancing.
❌ [All] Forgetting the wrap-around case — a key past the last node must route to the first, not error.
✅ [All] On "search hit end," take index 0; test a key whose position exceeds every node.
❌ [All] Using a weak/biased hash (or % smallN) — clustered positions defeat the even-distribution goal.
✅ [All] Hash node ids and keys with the same strong, uniform function into a large space (e.g. 2^32).
Key Rules:
  • Place nodes and keys on a ring; a key belongs to its clockwise-successor node — membership changes touch only ~1/N of keys.
  • Use virtual nodes to balance load and make rebalancing incremental; store the ring sorted for binary-search lookup.
  • Always handle wrap-around, and hash with a strong, uniform function over a large space.

Idempotency & Request Deduplication Distributed Systems
Why it matters: networks retry. A client that times out and resends a "charge card" or "create order" request must not be charged twice. An idempotency key lets the server recognize a duplicate and return the original result instead of doing the work again — the backbone of Stripe-style payment APIs, at-least-once message queues, and safe retries everywhere.

The pattern: the client sends a unique idempotency key with the request. The server records "key → result" the first time it processes it. On a repeat with the same key, it skips the side effect and replays the stored result. The work runs exactly once even though the request arrives multiple times. A shared store (Redis/DB) with a TTL replaces the in-memory map below in production.

Example 1 — a duplicate key replays the cached result; the charge runs once

Python 
class Payments:
    def __init__(self): self.seen, self.charges = {}, 0
    def charge(self, key, amount):
        if key in self.seen:
            return self.seen[key]         # replay, no new charge
        self.charges += 1                 # side effect happens once
        result = f"charged ${amount}"
        self.seen[key] = result
        return result

p = Payments()

print(p.charge("k1", 10))   # charged $10   (deduped)

The real engineering is concurrency and durability, and here the languages' concurrency models matter: two in-flight requests with the same key can both miss the check and double-execute. The fix is an atomic "insert-if-absent": Go sync.Map.LoadOrStore or a mutex, Rust Entry API under a Mutex, a DB INSERT ... ON CONFLICT, or Redis SET key val NX. Common Mistakes:
❌ [All] Checking "seen?" and then writing in two separate steps — two concurrent duplicates both pass the check and double-execute (a race).
✅ [All] Make claim-the-key atomic: LoadOrStore, Entry+lock, DB unique constraint, or Redis SETNX.
❌ [All] Storing keys forever — the dedup store grows unbounded.
✅ [All] Give each key a TTL sized to the client's max retry window (e.g. 24h) and let it expire.
❌ [All] Caching the result before the side effect commits — a crash mid-operation replays a "success" that never happened.
✅ [All] Persist the result in the same transaction as the side effect (or mark in-progress, then complete).
❌ [All] Letting the client reuse one key for different payloads — you'd replay the wrong result.
✅ [All] Bind the key to a hash of the request body; a key with a mismatched payload is a client error (409).
Key Rules:
  • Client supplies a unique key; server executes the side effect once and replays the stored result on repeats.
  • Claim the key atomically and persist the result with the effect — otherwise races or crashes reintroduce duplicates.
  • Expire keys with a TTL matched to the retry window; bind the key to the request payload.

Resilience

Distributed calls fail transiently — a blip, a throttle, a cold start. Retrying blindly makes it worse (thundering herd); retrying with exponential backoff and jitter spreads load and recovers gracefully. Combined with the idempotency above, retries become both safe and polite.

Retry with Exponential Backoff & Jitter Resilience Patterns
Why it matters: retrying a failed call after a fixed short delay hammers an already-struggling service and synchronizes every client into a retry storm. Exponential backoff (double the wait each attempt, up to a cap) plus random jitter de-correlates clients and gives the dependency room to recover. Every cloud SDK (AWS, GCP, Stripe) ships this by default.

Delay for attempt i is min(cap, base × factor^i). Retry only transient failures (timeouts, 429, 503) — never a 400 or auth error, which will fail identically forever. Add jitter so clients don't all wake at once; "full jitter" picks a random wait in [0, delay]. Always bound the number of attempts and the total time. The first example computes the deterministic delay schedule; the second shows a retry loop succeeding on the third try.

Example 1 — exponential delays (base 100ms, factor 2, cap 1000ms)

Python 
def backoff(base, factor, n, cap):
    return [min(cap, base * factor ** i) for i in range(n)]

print(backoff(100, 2, 5, 1000))   # [100, 200, 400, 800, 1000]

func backoff(base, factor, n, cap int) []int {
    out := make([]int, n)
    d := base
    for i := 0; i < n; i++ {
        if d > cap { d = cap }
        out[i] = d
        d *= factor
    }
    return out
}

func withRetry(fn func(int) (string, error), attempts int) (string, error) {
    var err error
    for i := 0; i < attempts; i++ {
        var v string
        if v, err = fn(i); err == nil { return v, nil }
        // time.Sleep(backoff(i) + jitter)
    }
    return "", err
}
n := 0
flaky := func(int) (string, error) {
    n++; if n < 3 { return "", errors.New("transient") }; return "ok", nil
}
v, _ := withRetry(flaky, 5)
fmt.Println(v, "after", n, "tries") // ok after 3 tries

async function withRetry<T>(fn: (i: number) => Promise<T>, attempts = 5): Promise<T> {
  let last: unknown;
  for (let i = 0; i < attempts; i++) {
    try { return await fn(i); }
    catch (e) { last = e; /* await sleep(backoff(i) + jitter) */ }
  }
  throw last;
}

Rust 
fn with_retry<F>(mut fn_: F, attempts: u32) -> Result<&'static str, &'static str>
where F: FnMut(u32) -> Result<&'static str, &'static str> {
    let mut last = Err("none");
    for i in 0..attempts {
        match fn_(i) {
            Ok(v) => return Ok(v),
            Err(e) => last = Err(e),   // sleep(backoff(i) + jitter)
        }
    }
    last
}
fn main() {
    let mut n = 0;
    let flaky = |_| { n += 1; if n < 3 { Err("transient") } else { Ok("ok") } };
    println!("{:?} after {} tries", with_retry(flaky, 5), n); // Ok("ok") after 3 tries
}

std::optional<std::string> with_retry(std::function<std::optional<std::string>(int)> fn, int attempts) {
    for (int i = 0; i < attempts; i++)
        if (auto v = fn(i)) return v;   // sleep(backoff(i) + jitter)
    return std::nullopt;
}

        return ++n < 3 ? std::nullopt : std::optional<std::string>("ok");
    };
    std::cout << *with_retry(flaky, 5) << " after " << n << " tries"; // ok after 3 tries

Language Comparison: The retry loop is structurally identical; what differs is how you wait and what "failure" looks like. Python/C++ catch exceptions and sleep (blocking) — async variants use asyncio.sleep. Go returns errors and uses time.Sleep or, better, a time.After/context select so a canceled caller aborts mid-backoff. TypeScript is inherently async: the loop awaits a setTimeout-based sleep, so it never blocks the event loop. Rust distinguishes sync (std::thread::sleep) from async (tokio::time::sleep). Everywhere, jitter needs a random source (Python random, Go math/rand, JS Math.random, Rust rand, C++ <random>) — "full jitter" (rand(0, delay)) generally beats fixed exponential in reducing collisions.
Common Mistakes:
❌ [All] Retrying non-transient errors (400, 401, validation) — they'll fail identically every time and waste attempts.
✅ [All] Classify errors; retry only transient ones (timeout, 429, 502/503, connection reset), respecting Retry-After.
❌ [All] Fixed-delay or un-jittered exponential retries — clients synchronize into a thundering herd.
✅ [All] Add jitter (prefer full jitter) and cap the delay so waits de-correlate and stay bounded.
❌ [All] Unbounded retries (no max attempts / no total deadline) — a request can hang effectively forever.
✅ [All] Bound both attempt count and total elapsed time; propagate a cancellation/deadline (context, AbortSignal).
❌ [All] Retrying a non-idempotent write — a "succeeded but response was lost" case double-applies the effect.
✅ [All] Pair retries with an idempotency key (previous card) so a replay is safe.
Key Rules:
  • Delay = min(cap, base × factor^attempt); add jitter; bound attempts and total time.
  • Retry only transient failures; honor server backpressure (Retry-After, 429) and cancel on deadline.
  • Retries are safe only for idempotent operations — combine with idempotency keys for writes.
Practice Exercises (implement in all 5 languages):
1
Add full jitter (sleep = random(0, min(cap, base*2^i))) to the retry loop and a total-deadline guard that aborts once cumulative elapsed time exceeds a budget, returning the last error.
2
Make the loop respect a cancellation signal (Go context, JS AbortSignal, Rust select! on a shutdown channel, Python task cancellation) so a caller can abort mid-backoff.
3
Wrap the retrier with a circuit breaker: after K consecutive failures, "open" for a cooldown and fail fast without calling the dependency; then half-open to test recovery.
