Arrays, Slices & Vectors Collections 1 day since review
Dynamic arrays are the default collection in all 5 languages. Python list, Go slice, TS Array, Rust Vec<T>, C++ vector<T>. All support O(1) append and O(1) random access by index.
Python
nums = [10, 20, 30]
nums.append(40)
for x in nums:
    print(x)
Go
nums := []int{10, 20, 30}
nums = append(nums, 40)
for _, v := range nums {
    fmt.Println(v)
}
TypeScript
const nums: number[] = [10, 20, 30];
nums.push(40);
for (const x of nums) {
    console.log(x);
}
Rust
let mut nums = vec![10, 20, 30];
nums.push(40);
for x in &nums {
    println!("{x}");
}
C++
std::vector<int> nums = {10, 20, 30};
nums.push_back(40);
for (int x : nums) {
    std::cout << x << "\n";
}
Challenge (all 5 languages):
1
Given an array of integers, return the array with all duplicates removed, preserving the first occurrence order.
Input: [3, 1, 2, 3, 1, 4] → [3, 1, 2, 4]
#1 Pitfall per Language:
  • Python: list.append([1,2]) nests the list instead of extending it — use .extend() for concatenation.
  • Go: Slice aliasing — a sub-slice shares the same backing array, so mutations to one affect the other.
  • TypeScript: .sort() is lexicographic by default — [10,2,1].sort() gives [1, 10, 2]. Pass a comparator.
  • Rust: Cannot use negative indices — use .len() - 1 to access the last element (and watch for underflow on empty vecs).
  • C++: Iterator invalidation — inserting/erasing elements while iterating a vector invalidates iterators.
Strings — Mutability & Unicode Collections 1 day since review
Python, Go, and TS strings are immutable. Rust String is mutable (owned), &str is a borrowed slice. C++ std::string is mutable. Go iterates runes via range, Rust iterates via .chars() or .bytes().
Python
name = "World"
msg = f"Hello, {name}!"
count = msg.count("l")
print(msg)  # Hello, World!
Go
name := "World"
msg := fmt.Sprintf("Hello, %s!", name)
count := strings.Count(msg, "l")
fmt.Println(msg)
TypeScript
const name = "World";
const msg = `Hello, ${name}!`;
const count = [...msg].filter(c => c === "l").length;
console.log(msg);
Rust
let name = "World";
let msg = format!("Hello, {name}!");
let count = msg.matches('l').count();
println!("{msg}");
C++
std::string name = "World";
std::ostringstream oss;
oss << "Hello, " << name << "!";
std::string msg = oss.str();
Challenge (all 5 languages):
1
Given a string, return the most frequently occurring character. If there is a tie, return the one that appears first.
Input: "abracadabra" → 'a'
#1 Pitfall per Language:
  • Python: += in a loop creates a new string each time — O(n²). Use ''.join(parts) instead.
  • Go: Iterating with a plain index (s[i]) gives bytes, not runes — use for _, r := range s for Unicode.
  • TypeScript: == performs type coercion; always use === for string comparison.
  • Rust: s[0] does not compile — strings are UTF-8 encoded. Use .chars().nth(0) or .as_bytes()[0].
  • C++: string::find() returns std::string::npos (a large value) on failure, not -1.
Hash Maps & Hash Sets Collections 1 day since review
O(1) average lookup by key. Python dict/set, Go map[K]V, TS Map<K,V>/Set<T>, Rust HashMap/HashSet, C++ unordered_map/unordered_set.
Python
freq: dict[str, int] = {}
for ch in text:
    freq[ch] = freq.get(ch, 0) + 1
# or: from collections import Counter
# freq = Counter(text)
Go
freq := make(map[rune]int)
for _, ch := range text {
    freq[ch]++
}
// missing keys default to zero value
TypeScript
const freq = new Map<string, number>();
for (const ch of text) {
    freq.set(ch, (freq.get(ch) ?? 0) + 1);
}
Rust
use std::collections::HashMap;
let mut freq = HashMap::new();
for ch in text.chars() {
    *freq.entry(ch).or_insert(0) += 1;
}
C++
std::unordered_map<char, int> freq;
for (char ch : text) {
    freq[ch]++;
}
// operator[] inserts default (0) if missing
Challenge (all 5 languages):
1
Given two arrays, return the elements that appear in both arrays, without duplicates, sorted in ascending order.
Input: [1, 2, 2, 3], [2, 3, 4] → [2, 3]
#1 Pitfall per Language:
  • Python: Modifying a dict while iterating it raises RuntimeError — iterate over a copy or collect keys first.
  • Go: Map iteration order is randomized — never rely on insertion order.
  • TypeScript: Plain objects ({}) lose type safety and coerce keys to strings — prefer Map<K,V>.
  • Rust: .get() returns Option<&V> — you must handle None (use .unwrap(), if let, or match).
  • C++: operator[] silently inserts a default value if the key is missing — use .find() or .count() to check existence.
Stacks & Queues Collections 1 day since review
Stack = LIFO (push/pop). Queue = FIFO (enqueue/dequeue). For O(1) operations on both ends, use a deque: Python deque, Go slice (manual), TS array (or linked list), Rust VecDeque, C++ deque.
Python
stack: list[int] = []
stack.append(1)       # push
top = stack[-1]       # peek
val = stack.pop()     # pop
Go
stack := []int{}
stack = append(stack, 1)         // push
top := stack[len(stack)-1]       // peek
stack = stack[:len(stack)-1]     // pop
TypeScript
const stack: number[] = [];
stack.push(1);                     // push
const top = stack[stack.length-1]; // peek
const val = stack.pop();           // pop
Rust
let mut stack: Vec<i32> = Vec::new();
stack.push(1);                  // push
let top = stack.last();         // peek -> Option
let val = stack.pop();          // pop  -> Option
C++
std::stack<int> stk;
stk.push(1);             // push
int top = stk.top();     // peek
stk.pop();               // pop (returns void)
Challenge (all 5 languages):
1
Given a string of lowercase letters, remove all adjacent duplicates repeatedly until no more can be removed. Use a stack.
Input: "abbaca" → "ca"
#1 Pitfall per Language:
  • Python: list.pop(0) is O(n) — use collections.deque.popleft() for O(1) front removal.
  • Go: After popping from a slice, set the vacated element to its zero value to avoid memory leaks from lingering references.
  • TypeScript: Array.shift() is O(n) because it re-indexes every element — avoid in hot loops.
  • Rust: Vec::remove(0) is O(n) — use VecDeque for O(1) front and back operations.
  • C++: Calling .top() on an empty std::stack is undefined behavior — always check .empty() first.

Structured Data

Tuples, Structs & Records Structured Data

Every real program models data with multiple fields. Choosing the right container — tuple, struct, or record — determines how readable, safe, and maintainable your code will be across all five languages.

1. Tuples — Fixed-Size Heterogeneous Groupings

Tuples pack a small, fixed number of values together. They are ideal for returning multiple values from a function or grouping temporary data like coordinates.

Python
# Create a tuple and unpack it
point = (3, 7)
x, y = point  # unpacking

# Return multiple values from a function
def min_max(nums):
    return min(nums), max(nums)

lo, hi = min_max([5, 2, 8, 1, 9])
print(f"point = {point}, x = {x}, y = {y}")
print(f"min = {lo}, max = {hi}")
point = (3, 7), x = 3, y = 7 min = 1, max = 9
Go
package main

import "fmt"

// Go has no tuples — use multiple return values
func minMax(nums []int) (int, int) {
    lo, hi := nums[0], nums[0]
    for _, v := range nums {
        if v < lo { lo = v }
        if v > hi { hi = v }
    }
    return lo, hi
}

func main() {
    lo, hi := minMax([]int{5, 2, 8, 1, 9})
    fmt.Printf("min = %d, max = %d\n", lo, hi)

    // Anonymous struct as a "tuple"
    point := struct{ X, Y int }{3, 7}
    fmt.Printf("point = {%d, %d}\n", point.X, point.Y)
}
min = 1, max = 9 point = {3, 7}
TypeScript
// Typed tuple
const point: [number, number] = [3, 7];
const [x, y] = point; // destructuring

// Return multiple values via tuple
function minMax(nums: number[]): [number, number] {
    return [Math.min(...nums), Math.max(...nums)];
}

const [lo, hi] = minMax([5, 2, 8, 1, 9]);
console.log(`point = [${x}, ${y}]`);
console.log(`min = ${lo}, max = ${hi}`);
point = [3, 7] min = 1, max = 9
Rust
fn min_max(nums: &[i32]) -> (i32, i32) {
    let lo = *nums.iter().min().unwrap();
    let hi = *nums.iter().max().unwrap();
    (lo, hi)
}

fn main() {
    let point: (i32, i32) = (3, 7);
    let (x, y) = point; // pattern matching
    println!("point = ({x}, {y})");

    let (lo, hi) = min_max(&[5, 2, 8, 1, 9]);
    println!("min = {lo}, max = {hi}");
}
point = (3, 7) min = 1, max = 9
C++
#include <iostream>
#include <tuple>
#include <vector>
#include <algorithm>

std::pair<int, int> minMax(const std::vector<int>& nums) {
    auto [lo, hi] = std::minmax_element(nums.begin(), nums.end());
    return {*lo, *hi};
}

int main() {
    auto point = std::make_tuple(3, 7);
    auto [x, y] = point; // structured bindings (C++17)
    std::cout << "point = (" << x << ", " << y << ")\n";

    auto [lo, hi] = minMax({5, 2, 8, 1, 9});
    std::cout << "min = " << lo << ", max = " << hi << "\n";
}
point = (3, 7) min = 1, max = 9

2. Structs & Records — Named Fields

When data has clear, named fields, a struct (or record, or data class) is self-documenting and far safer than positional access.

Python
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

p = Point(3.0, 7.0)
print(f"Point({p.x}, {p.y})")
print(p)  # __repr__ auto-generated
Point(3.0, 7.0) Point(x=3.0, y=7.0)
Go
package main

import "fmt"

type Point struct {
    X, Y float64
}

func main() {
    p := Point{X: 3.0, Y: 7.0}
    fmt.Printf("Point(%g, %g)\n", p.X, p.Y)
    fmt.Printf("%+v\n", p) // %+v shows field names
}
Point(3, 7) {X:3 Y:7}
TypeScript
interface Point {
    x: number;
    y: number;
}

const p: Point = { x: 3, y: 7 };
console.log(`Point(${p.x}, ${p.y})`);
console.log(p);
Point(3, 7) { x: 3, y: 7 }
Rust
#[derive(Debug)]
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p = Point { x: 3.0, y: 7.0 };
    println!("Point({}, {})", p.x, p.y);
    println!("{:?}", p); // requires #[derive(Debug)]
}
Point(3, 7) Point { x: 3.0, y: 7.0 }
C++
#include <iostream>

struct Point {
    double x;
    double y;
};

int main() {
    Point p{3.0, 7.0};
    std::cout << "Point(" << p.x << ", " << p.y << ")\n";
    // No built-in debug print for structs
}
Point(3, 7)

3. Destructuring — Extracting Fields from Structured Data

Destructuring lets you bind named variables to the parts of a structured value in one step, making code more readable and concise.

Python
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

p = Point(3.0, 7.0)

# Structural pattern matching (Python 3.10+)
match p:
    case Point(x=x_val, y=y_val):
        print(f"Matched: x={x_val}, y={y_val}")

# Tuple unpacking
coords = (10, 20, 30)
x, y, z = coords
print(f"x={x}, y={y}, z={z}")
Matched: x=3.0, y=7.0 x=10, y=20, z=30
Go
package main

import (
    "fmt"
    "strings"
)

func main() {
    // Go: no struct destructuring — access fields directly
    type Point struct{ X, Y float64 }
    p := Point{3.0, 7.0}
    x, y := p.X, p.Y
    fmt.Printf("x=%g, y=%g\n", x, y)

    // Multi-return destructuring
    parts := strings.Cut("key=value", "=")
    // strings.Cut is not valid here, use SplitN
    kv := strings.SplitN("key=value", "=", 2)
    fmt.Printf("key=%s, val=%s\n", kv[0], kv[1])
}
x=3, y=7 key=key, val=value
TypeScript
interface Point { x: number; y: number }

const p: Point = { x: 3, y: 7 };
const { x, y } = p; // object destructuring
console.log(`x=${x}, y=${y}`);

// Array destructuring
const [first, ...rest] = [10, 20, 30];
console.log(`first=${first}, rest=${rest}`);

// Nested destructuring
const user = { name: "Ada", address: { city: "London" } };
const { address: { city } } = user;
console.log(`city=${city}`);
x=3, y=7 first=10, rest=20,30 city=London
Rust
#[derive(Debug)]
struct Point { x: f64, y: f64 }

fn main() {
    let p = Point { x: 3.0, y: 7.0 };

    // Struct destructuring
    let Point { x, y } = p;
    println!("x={x}, y={y}");

    // Tuple destructuring in a match
    let result: (bool, i32) = (true, 42);
    match result {
        (true, val)  => println!("OK: {val}"),
        (false, err) => println!("Err: {err}"),
    }
}
x=3, y=7 OK: 42
C++
#include <iostream>
#include <tuple>
#include <string>

struct Point { double x; double y; };

int main() {
    // Structured bindings (C++17)
    Point p{3.0, 7.0};
    auto [x, y] = p;
    std::cout << "x=" << x << ", y=" << y << "\n";

    // Works with std::tuple and std::pair too
    auto tup = std::make_tuple(42, std::string("hello"), 3.14);
    auto [i, s, d] = tup;
    std::cout << i << ", " << s << ", " << d << "\n";
}
x=3, y=7 42, hello, 3.14

4. Real-World Pattern — User Model

Model a User with name, age, and email. Create instances, modify fields (where allowed), and display them.

Python
from dataclasses import dataclass

@dataclass
class User:
    name: str
    age: int
    email: str

user = User("Alice", 30, "alice@example.com")
user.age = 31  # mutable by default
print(user)

# Immutable variant: frozen=True
from dataclasses import dataclass as dc
@dc(frozen=True)
class FrozenUser:
    name: str
    age: int
print(FrozenUser("Bob", 25))
User(name='Alice', age=31, email='alice@example.com') FrozenUser(name='Bob', age=25)
Go
package main

import "fmt"

type User struct {
    Name  string
    Age   int
    Email string
}

func (u User) String() string {
    return fmt.Sprintf("%s (age %d, %s)", u.Name, u.Age, u.Email)
}

func main() {
    user := User{Name: "Alice", Age: 30, Email: "alice@example.com"}
    user.Age = 31 // structs are mutable value types
    fmt.Println(user)

    // Zero-value initialization
    var empty User
    fmt.Printf("zero: %q, age %d\n", empty.Name, empty.Age)
}
Alice (age 31, alice@example.com) zero: "", age 0
TypeScript
interface User {
    name: string;
    age: number;
    email: string;
}

const user: User = { name: "Alice", age: 30, email: "alice@example.com" };
user.age = 31; // plain object — mutable
console.log(user);

// Readonly variant
const frozen: Readonly<User> = { name: "Bob", age: 25, email: "bob@x.com" };
// frozen.age = 26; // Error: Cannot assign to 'age'
console.log(`${frozen.name}, age ${frozen.age}`);
{ name: 'Alice', age: 31, email: 'alice@example.com' } Bob, age 25
Rust
#[derive(Debug, Clone)]
struct User {
    name: String,
    age: u32,
    email: String,
}

fn main() {
    let mut user = User {
        name: String::from("Alice"),
        age: 30,
        email: String::from("alice@example.com"),
    };
    user.age = 31; // must be declared `mut`
    println!("{:?}", user);

    // Struct update syntax
    let user2 = User { name: String::from("Bob"), ..user.clone() };
    println!("{:?}", user2);
}
User { name: "Alice", age: 31, email: "alice@example.com" } User { name: "Bob", age: 31, email: "alice@example.com" }
C++
#include <iostream>
#include <string>

struct User {
    std::string name;
    int age;
    std::string email;
};

std::ostream& operator<<(std::ostream& os, const User& u) {
    return os << u.name << " (age " << u.age << ", " << u.email << ")";
}

int main() {
    User user{"Alice", 30, "alice@example.com"};
    user.age = 31; // value semantics, mutable by default
    std::cout << user << "\n";

    User user2 = user; // copy constructor (automatic)
    user2.name = "Bob";
    std::cout << user2 << "\n";
}
Alice (age 31, alice@example.com) Bob (age 31, alice@example.com)
Language Comparison: Python treats everything as objects with dynamic attributes, but @dataclass adds structure and auto-generates __init__, __repr__, and __eq__. Go uses exported fields (capitalized) and zero-value initialization — no constructors needed. TypeScript separates types (interfaces) from runtime (plain objects); interfaces are erased after compilation. Rust enforces ownership rules with #[derive(Debug, Clone)] and requires mut to modify. C++ uses value semantics by default and automatically generates copy constructors and assignment operators.
Common Mistakes:
  • Python: Mutable default arguments in dataclass fields — field(default_factory=list) is required instead of tags: list = [], which shares one list across all instances.
  • Go: Uninitialized struct fields silently get zero values — an empty string is "", not nil. This can mask bugs when a field was never intentionally set.
  • TypeScript: Interfaces are erased at runtime — you cannot use instanceof on an interface type. Use a class or a type guard function instead.
  • Rust: Forgetting #[derive(Debug)] — structs cannot be printed with println!("{:?}", s) without it. Add it to every struct during development.
  • C++: Forgetting to initialize struct members — non-static members contain garbage values in C++ unless you provide a default or use aggregate initialization.
Key Rules:
  • Use structs when data has clear named fields — they are self-documenting and type-safe.
  • Use tuples for temporary groupings (function returns, coordinates) but not for complex data with more than 2-3 fields.
  • Prefer destructuring over index access — let (x, y) = point beats point.0, point.1.

Sorting

Sorting & Custom Comparators Sorting

Sorting is one of the most frequent operations in programming. Knowing the default behavior, gotchas, and custom comparator syntax in each language saves hours of debugging subtle ordering bugs.

1. Basic Sort — Ascending and Descending

Sort an array of integers in ascending and descending order. Pay attention to which operations are in-place vs. returning a new collection.

Python
nums = [5, 2, 8, 1, 9, 3]

# sorted() returns a NEW list (original unchanged)
asc = sorted(nums)
desc = sorted(nums, reverse=True)
print(f"original: {nums}")
print(f"asc:      {asc}")
print(f"desc:     {desc}")

# .sort() modifies IN-PLACE (returns None)
nums.sort()
print(f"in-place: {nums}")
original: [5, 2, 8, 1, 9, 3] asc: [1, 2, 3, 5, 8, 9] desc: [9, 8, 5, 3, 2, 1] in-place: [1, 2, 3, 5, 8, 9]
Go
package main

import (
    "fmt"
    "sort"
)

func main() {
    nums := []int{5, 2, 8, 1, 9, 3}

    // sort.Ints sorts in-place, ascending
    sort.Ints(nums)
    fmt.Println("asc: ", nums)

    // Descending: use sort.Slice with custom comparator
    sort.Slice(nums, func(i, j int) bool {
        return nums[i] > nums[j]
    })
    fmt.Println("desc:", nums)
}
asc: [1 2 3 5 8 9] desc: [9 8 5 3 2 1]
TypeScript
const nums = [5, 2, 8, 1, 9, 3];

// WARNING: .sort() without comparator is LEXICOGRAPHIC
console.log("default:", [...nums].sort());
// [1, 2, 3, 5, 8, 9] looks correct but try [10, 2, 1]...
console.log("gotcha:", [10, 2, 1].sort());

// ALWAYS pass a numeric comparator
const asc = [...nums].sort((a, b) => a - b);
const desc = [...nums].sort((a, b) => b - a);
console.log("asc: ", asc);
console.log("desc:", desc);
default: [ 1, 2, 3, 5, 8, 9 ] gotcha: [ 1, 10, 2 ] asc: [ 1, 2, 3, 5, 8, 9 ] desc: [ 9, 8, 5, 3, 2, 1 ]
Rust
fn main() {
    let mut nums = vec![5, 2, 8, 1, 9, 3];

    // .sort() is in-place, ascending (stable)
    nums.sort();
    println!("asc:  {:?}", nums);

    // Descending with comparator
    nums.sort_by(|a, b| b.cmp(a));
    println!("desc: {:?}", nums);

    // .sort_unstable() is faster but doesn't preserve equal order
    nums.sort_unstable();
    println!("unstable asc: {:?}", nums);
}
asc: [1, 2, 3, 5, 8, 9] desc: [9, 8, 5, 3, 2, 1] unstable asc: [1, 2, 3, 5, 8, 9]
C++
#include <iostream>
#include <vector>
#include <algorithm>
#include <functional>

int main() {
    std::vector<int> nums = {5, 2, 8, 1, 9, 3};

    // std::sort is in-place (not guaranteed stable)
    std::sort(nums.begin(), nums.end());
    std::cout << "asc: ";
    for (int n : nums) std::cout << n << " ";
    std::cout << "\n";

    // Descending with greater<>
    std::sort(nums.begin(), nums.end(), std::greater<int>());
    std::cout << "desc: ";
    for (int n : nums) std::cout << n << " ";
    std::cout << "\n";
}
asc: 1 2 3 5 8 9 desc: 9 8 5 3 2 1

2. Custom Comparator — Sort Strings by Length, Then Alphabetically

When the natural ordering is not what you need, provide a custom key or comparator. Multi-level sorting compares by a primary key first, then breaks ties with a secondary key.

Python
words = ["banana", "fig", "cherry", "apple", "kiwi"]

# key= transforms once, then sorts by result
# Tuple comparison: first by length, then alphabetically
result = sorted(words, key=lambda w: (len(w), w))
print(result)
['fig', 'kiwi', 'apple', 'banana', 'cherry']
Go
package main

import (
    "fmt"
    "sort"
)

func main() {
    words := []string{"banana", "fig", "cherry", "apple", "kiwi"}

Language Comparison: Python's key= function is the most elegant approach — transform each element once, then sort by the result. Tuple keys like (len(w), w) handle multi-level sorting effortlessly. Go requires explicit comparator functions with index-based access (i, j), which is verbose but explicit. TypeScript's default lexicographic sort is a notorious gotcha — [10, 2, 1].sort() gives [1, 10, 2]. Rust distinguishes stable .sort() from faster .sort_unstable() and provides the ergonomic .cmp().then() chaining for multi-key comparisons. C++ gives the most control with iterators and custom comparators, plus a separate std::stable_sort.
Common Mistakes:
  • Python: .sort() returns None (modifies in-place). Writing x = my_list.sort() sets x to None. Use x = sorted(my_list) to get a new sorted list.
  • Go: The comparator must implement strict weak ordering — using <= instead of < can cause infinite loops or panics in sort.Slice.
  • TypeScript: [10, 2, 1].sort() produces [1, 10, 2] (lexicographic!). Always provide a comparator for numbers: .sort((a, b) => a - b).
  • Rust: Floats (f64) do not implement Ord because of NaN. Use .sort_by(|a, b| a.partial_cmp(b).unwrap()) — but be aware this panics on NaN. Use .total_cmp() for safe float sorting.
  • C++: The comparator must return false for equal elements — using <= instead of < violates strict weak ordering and causes undefined behavior.
Key Rules:
  • Always use a numeric comparator for number sorting in TypeScript and JavaScript — default sort is lexicographic.
  • Prefer stable sort when relative order of equal elements matters (e.g., sorting by a secondary key after a primary key).
  • Use key= (Python) or sort_by_key (Rust) when the comparison is expensive — they transform each element once rather than recomputing on every comparison.

Recursive Structures

Linked Lists & Trees Recursive Structures

Linked lists and trees are the first data structures where nodes point to other nodes — making ownership, memory management, and null-safety real concerns rather than academic ones. Mastering how each language defines and traverses these recursive structures unlocks BSTs, heaps, tries, and the majority of LeetCode tree/list problems.

A linked list node holds a value and a pointer (or reference) to the next node. A binary tree node holds a value and pointers to left and right children. The challenge is that these structures are recursive: the type refers to itself. Each language handles this differently. Python and TypeScript use reference semantics with garbage collection — nodes are objects on the heap and cleanup is automatic. Go uses explicit pointers (*Node) but still relies on GC. Rust requires Box<T> for heap allocation and Option for nullable pointers — the compiler enforces single ownership, preventing dangling pointers and memory leaks at compile time. C++ gives you raw pointers (fastest, manual cleanup) or smart pointers (unique_ptr for ownership, shared_ptr for shared ownership).

        Some(Box::new(TreeNode::leaf(3))),
    )));
    preorder(&root);
    println!();
}

C++
#include <iostream>
#include <memory>
using namespace std;

struct TreeNode {
    int val;
    unique_ptr<TreeNode> left;
    unique_ptr<TreeNode> right;
    TreeNode(int v,
             unique_ptr<TreeNode> l = nullptr,
             unique_ptr<TreeNode> r = nullptr)
        : val(v), left(move(l)), right(move(r)) {}
};

void preorder(const TreeNode* node) {
    // Root -> Left -> Right
    if (!node) return;
    cout << node->val << " ";
    preorder(node->left.get());
    preorder(node->right.get());
}

int main() {
    auto root = make_unique<TreeNode>(1,
        make_unique<TreeNode>(2,
            make_unique<TreeNode>(4),
            make_unique<TreeNode>(5)
        ),
        make_unique<TreeNode>(3)
    );
    preorder(root.get());
    cout << endl;
}

Language Comparison — Ownership of Pointers: Python and TypeScript use reference semantics — nodes are objects on the heap, and the garbage collector handles cleanup automatically. Go uses explicit pointers (*TreeNode) but GC still manages memory. Rust requires Box<T> for heap allocation and Option for nullable pointers — the compiler enforces single ownership, preventing dangling pointers and memory leaks at compile time. C++ gives you raw pointers (fastest but you must delete manually) or smart pointers (unique_ptr for single ownership, shared_ptr for shared). The key insight: Rust makes the ownership model that C++ programmers must remember explicit and compiler-enforced.
Common Mistakes:
  • Python: Circular references in linked structures (e.g., a doubly-linked list) can prevent the garbage collector from reclaiming memory. Use weakref for back-pointers if cycles are necessary.
  • Go: Forgetting a nil check before dereferencing node.Next causes a nil pointer panic at runtime. Always guard with if node != nil before accessing fields.
  • TypeScript: Accessing node.next without a null check throws a runtime error. Use optional chaining (node?.next) or explicit checks (if (node !== null)).
  • Rust: Trying to create a doubly-linked list with Box fails because Box enforces single ownership — a node cannot be owned by both its parent and its child. Use Rc<RefCell<T>> for shared ownership, or use a Vec-based approach with indices instead of pointers.
  • C++: Using new ListNode() without a corresponding delete causes a memory leak. Prefer unique_ptr<ListNode> for automatic cleanup. When using raw pointers, always delete in a loop when tearing down a list.
Key Rules:
  • Use linked lists when you need O(1) insertion/deletion at known positions — for random access, use arrays. Linked lists have O(n) access by index; arrays have O(1).
  • Trees are the foundation for BSTs, heaps, tries, and expression trees — master the node definition pattern first. Every tree problem starts with the same struct: a value and two optional child pointers.
  • In Rust, Option<Box<Node>> is THE pattern for recursive structures — Box provides indirection (the compiler needs a known size for the struct, and a pointer has a known size), Option handles the null/empty case. This pair replaces nullable pointers from other languages.
