1. Dependency Injection Advanced Patterns rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: Dependency injection means a component receives its collaborators from the outside (via constructor/parameters) instead of constructing them itself — so you can swap a real implementation for a fake in tests and decouple policy from wiring. Python/TS use constructor injection (or a container); Go passes small interfaces; Rust injects via a generic type parameter (static) or a Box<dyn Trait> (dynamic); C++ injects a reference/pointer to an abstract base or a template parameter.

Key Pattern — inject a message sender into a notifier

Python 
class Notifier:
    def __init__(self, sender):        # dependency injected via constructor
        self.sender = sender
    def alert(self, msg): self.sender.send(msg)
Notifier(EmailSender()).alert("hi")    # swap FakeSender() in tests
Go 
type Sender interface{ Send(string) }
type Notifier struct{ s Sender }       // depend on the interface, not a concrete type
func (n Notifier) Alert(m string){ n.s.Send(m) }
// Notifier{s: EmailSender{}}.Alert("hi")
TypeScript 
interface Sender { send(m: string): void; }
class Notifier {
  constructor(private sender: Sender) {}   // constructor injection
  alert(m: string) { this.sender.send(m); }
}
Rust 
trait Sender { fn send(&self, m: &str); }
struct Notifier<S: Sender> { sender: S }   // generic = zero-cost static injection
impl<S: Sender> Notifier<S> {
    fn alert(&self, m: &str) { self.sender.send(m); } // or Box<dyn Sender> for dynamic
}
C++ 
struct Sender { virtual void send(const std::string&) = 0; virtual ~Sender()=default; };
class Notifier {
    Sender& s;                          // inject a reference to the abstract base
public:
    explicit Notifier(Sender& s): s(s) {}
    void alert(const std::string& m){ s.send(m); }
};

  • Go: depending on a large concrete struct instead of a small interface defined at the consumer — “accept interfaces, return structs,” and keep interfaces one or two methods wide.
  • TypeScript: calling new Concrete() inside the class or reaching for a global singleton — breaks substitution; reflection-based DI frameworks also need emitDecoratorMetadata.
  • Rust: reaching for Box<dyn Trait> (a heap alloc + vtable) when a generic type parameter would be zero-cost; over-using Rc<RefCell<_>> just to share a dependency.
  • C++: injecting the dependency by value slices a derived object — inject by reference or pointer to the base, and be explicit about who owns/outlives it.
2. TypeScript: Conditional Types & infer Language Deep Dive rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: A conditional type T extends U ? X : Y selects a type by a compile-time subtype test, and infer captures a type variable inside the extends clause (e.g. unwrap an array's element type or a function's return type). They power the stdlib utilities (ReturnType, Awaited) and distribute over unions when the checked type is a naked type parameter. Only TS has literal type-level branching; the others approximate with associated types (Rust), type_traits/specialization (C++), or plain inference (Go/Python).

Key Pattern — extract an array's element type and a function's return type

TypeScript 
type ElementOf<T> = T extends (infer E)[] ? E : never;
type A = ElementOf<number[]>;              // number
type Ret<F> = F extends (...a: any[]) => infer R ? R : never;
type B = Ret<() => string>;                // string
Rust 
// No conditional types; associated types + traits play the role:
trait Container { type Item; }
impl<T> Container for Vec<T> { type Item = T; }
// <Vec<i32> as Container>::Item == i32
C++ 
// type traits + specialization are C++'s "conditional types":
template<class T> struct element { using type = T; };
template<class T> struct element<std::vector<T>> { using type = T; };
using B = element<std::vector<int>>::type;   // int
// std::conditional_t<cond, X, Y> mirrors  T extends U ? X : Y
Go 
// Go generics have NO conditional types; constraints only bound the type set:
func First[T any](s []T) T { return s[0] }   // element type inferred, not computed
// there is no way to express "if T is X then Y"
Python 
# No conditional types; a TypeVar infers, overloads approximate at check time:
from typing import TypeVar, Sequence
T = TypeVar("T")
def first(s: Sequence[T]) -> T: return s[0]   # element type via TypeVar
Challenge (all 5 languages):
1
Implement DeepReadonly<T> and an Awaited<T>-style unwrapper recursively in TS with conditional types + infer. Then express “unwrap the element type of a container” in C++ (type traits) and Rust (associated types), and explain why Go and Python can only infer a type from a value, not compute a type from a type.
#1 Pitfall per Language:
  • TypeScript: forgetting conditional types distribute over unions when T is naked — wrap in [T] extends [U] to disable it; and placing infer in the wrong position captures the wrong type.
  • Rust: expecting TS-style type-level branching — there is no if type; model variation with associated types and trait bounds.
  • C++: std::conditional_t selects one branch but both branch types must be well-formed — an ill-formed branch still errors; guard with lazy specialization.
  • Go: trying to derive one type from another — impossible; generics only constrain, they don't compute types.
  • Python: type hints are erased at runtime — any “conditional typing” is a static-checker fiction with zero runtime effect.
3. Rust: Trait Objects & Dynamic Dispatch Language Deep Dive rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: A trait object dyn Trait (behind a pointer: &dyn, Box<dyn>) erases the concrete type and dispatches method calls through a vtable at runtime — Rust's opt-in dynamic polymorphism, versus generics which monomorphize into static dispatch. It's the direct analog of a Go interface value, a C++ base pointer + virtual, and TS/Python duck-typed interfaces.

Key Pattern — a heterogeneous list of shapes dispatched at runtime

Rust 
trait Shape { fn area(&self) -> f64; }
struct Circle(f64);
impl Shape for Circle { fn area(&self) -> f64 { 3.14 * self.0 * self.0 } }
let shapes: Vec<Box<dyn Shape>> = vec![Box::new(Circle(1.0))]; // dynamic dispatch
let total: f64 = shapes.iter().map(|s| s.area()).sum();      // 3.14
Go 
type Shape interface{ Area() float64 }
type Circle struct{ r float64 }
func (c Circle) Area() float64 { return 3.14 * c.r * c.r }
shapes := []Shape{Circle{1}}           // interface values dispatch dynamically
C++ 
struct Shape { virtual double area() const = 0; virtual ~Shape() = default; };
struct Circle : Shape { double r; double area() const override { return 3.14*r*r; } };
std::vector<std::unique_ptr<Shape>> shapes;  // base pointer -> vtable dispatch
TypeScript 
interface Shape { area(): number; }
class Circle implements Shape { constructor(public r: number){} area(){ return 3.14*this.r*this.r; } }
const shapes: Shape[] = [new Circle(1)];   // structural; method looked up at call
Python 
class Circle:
    def __init__(self, r): self.r = r
    def area(self): return 3.14 * self.r * self.r   # duck typing: any .area() works
shapes = [Circle(1)]
print(sum(s.area() for s in shapes))              # 3.14

1

Then reason about cost: show where Rust generics would monomorphize instead, and identify the object-safety rules (no generic methods, no Self-returning methods) that block dyn.
#1 Pitfall per Language:
  • Rust: making a trait non-object-safe (a generic method, or one returning Self) then trying dyn — it won't compile; and Box<dyn> costs a heap allocation plus a vtable indirection.
  • Go: the typed-nil trap — an interface holding a nil concrete pointer is not == nil; and over-wide interface method sets couple callers unnecessarily.
  • C++: forgetting virtual ~Base() — deleting a derived object via a base pointer is UB; storing polymorphic objects by value slices them.
  • TypeScript: implements is structural — a class matches an interface by shape even without the keyword, and there is no interface to check at runtime.
  • Python: no compile-time guarantee — a missing method fails only when called (AttributeError); use typing.Protocol/ABCs for an earlier signal.
4. C++: Concepts (C++20) & Constraints Language Deep Dive rep 3 • 15 days since review
4th review (on schedule)
Quick Recall: A concept is a named, compile-time predicate on template parameters that constrains which types a template accepts and produces readable errors at the call site instead of deep instantiation spew. A requires expression tests that operations are valid. It is the direct analog of a Rust trait bound, a TS extends constraint, a Go type-set constraint, and a Python Protocol.

Key Pattern — a generic sum constrained to addable/numeric types

C++ 
#include <concepts><vector>
template<class T> concept Addable = requires(T a, T b){ { a + b } -> std::same_as<T>; };
template<Addable T> T sum(const std::vector<T>& v){ T t{}; for(auto& x : v) t += x; return t; }
sum(std::vector<int>{1,2,3});          // 6; a non-addable T gives a clear error
Rust 
use std::ops::Add;
fn sum<T: Add<Output=T> + Copy + Default>(v: &[T]) -> T {
    v.iter().copied().fold(T::default(), |a, b| a + b)   // trait bound == concept
}
Go 
type Number interface{ ~int | ~int64 | ~float64 }   // a type-set constraint
func Sum[T Number](v []T) T { var t T; for _, x := range v { t += x }; return t }
TypeScript 
// constraint via `extends`; no arithmetic constraint, so bound by shape:
function sum<T extends number>(v: T[]): number { return v.reduce((a, b) => a + b, 0); }
Python 
from typing import Protocol, TypeVar
class Addable(Protocol):
    def __add__(self, other): ...
T = TypeVar("T", bound=Addable)         # structural constraint, checked statically
def total(v: list[T]) -> T: ...

  • C++: a requires expression checks syntax, not semantics — a type can satisfy it by accident; and forgetting the { expr } -> concept return-type-constraint form.

6. Caching & Memoization Patterns Performance & Optimization rep 1 • 11 days since review
2nd review (overdue 10 days)
Quick Recall: Memoization caches a pure function's result keyed by its arguments so repeated calls are O(1); a cache needs an eviction policy (LRU/LFU/TTL) once unbounded growth is a risk. Python ships functools.lru_cache/cache; the others compose a map + wrapper (Go map+sync, a TS Map, a Rust HashMap, a C++ unordered_map).

Key Pattern — memoized Fibonacci

Python 
from functools import lru_cache
@lru_cache(maxsize=None)                # built-in memoization decorator
def fib(n): return n if n < 2 else fib(n-1) + fib(n-2)
print(fib(50))                          # 12586269025 (instant)
Go 
var memo = map[int]int64{}             // no built-in; guard for concurrency if shared
func fib(n int) int64 {
    if n < 2 { return int64(n) }
    if v, ok := memo[n]; ok { return v }
    memo[n] = fib(n-1) + fib(n-2); return memo[n]
}
TypeScript 
const memo = new Map<number, number>();
function fib(n: number): number {
  if (n < 2) return n;
  if (memo.has(n)) return memo.get(n)!;
  const v = fib(n-1) + fib(n-2); memo.set(n, v); return v;
}
Rust 
use std::collections::HashMap;
fn fib(n: u64, memo: &mut HashMap<u64, u64>) -> u64 {
    if n < 2 { return n; }
    if let Some(&v) = memo.get(&n) { return v; }
    let v = fib(n-1, memo) + fib(n-2, memo); memo.insert(n, v); v
}
C++ 
#include <unordered_map>
std::unordered_map<int, long long> memo;
long long fib(int n){
    if(n < 2) return n;
    if(auto it = memo.find(n); it != memo.end()) return it->second;
    return memo[n] = fib(n-1) + fib(n-2);
}

1
Wrap an arbitrary pure function in a generic memoize (Python decorator, TS higher-order fn, Go generics, Rust closure + HashMap, C++ template) with an LRU eviction cap of N entries; prove the (N+1)th distinct key evicts the least-recently-used one. Add a TTL-expiry variant.
#1 Pitfall per Language:
  • Python: lru_cache on a method keeps self alive (a leak) and needs hashable args; a mutable-arg key silently mis-caches.
  • Go: a plain map cache isn't goroutine-safe — concurrent read/write panics; use sync.Mutex/sync.Map, and it grows unbounded without eviction.
  • TypeScript: keying a Map by object identity (not value) misses logically-equal keys; an unbounded Map leaks — use a WeakMap or an LRU.
  • Rust: the borrow checker fights self-referential memo passing — thread the cache as a &mut param or use RefCell; recursion + &mut can conflict.
  • C++: memo[n] default-inserts on a lookup (use find); a reference into the map dangles if a later insert triggers a rehash.

Graphs & Traversal

Graphs model relationships — networks, dependencies, maps, social ties — and almost every non-trivial system grows a graph somewhere. None of the five languages ship a graph type in the standard library, so you compose one from the collection primitives you already know. The interesting differences are in how each language stores nodes, edges, and the "visited" set, and how ownership/borrowing complicates the recursive traversals.

Graph Representations: Adjacency List & Matrix Graphs & Traversal
Why it matters: how you store a graph decides the cost of every operation on it. An adjacency list is O(V+E) space and iterates a node's neighbors fast; an adjacency matrix is O(V²) space but answers "is there an edge u→v?" in O(1). Choose wrong and a fast algorithm goes slow — BFS on a matrix of a sparse graph wastes nearly all its time scanning zeros.

A graph is a set of vertices plus edges between them. Two canonical encodings dominate: the adjacency list (each vertex maps to its list of neighbors — ideal for sparse graphs, which is most real graphs) and the adjacency matrix (a V×V grid where cell [i][j] marks an edge — ideal for dense graphs or when you need O(1) edge queries). Since no stdlib provides a graph, you build one from a map-of-lists or a 2D array.

Example 1 — adjacency list for a directed graph, then iterate one node's neighbors

Python 
from collections import defaultdict
g = defaultdict(list)
for u, v in [(0,1),(0,2),(1,2),(2,0)]:
    g[u].append(v)                     # adjacency list: vertex -> neighbors
print(g[0])                            # [1, 2]
[1, 2]
Go 
g := map[int][]int{}
for _, e := range [][2]int{{0,1},{0,2},{1,2},{2,0}} {
    g[e[0]] = append(g[e[0]], e[1])
}
fmt.Println(g[0])                      // [1 2]
[1 2]
TypeScript 
const g = new Map<number, number[]>();
for (const [u, v] of [[0,1],[0,2],[1,2],[2,0]]) {
  if (!g.has(u)) g.set(u, []);
  g.get(u)!.push(v);                   // adjacency list
}
console.log(g.get(0));                 // [1, 2]
[ 1, 2 ]
Rust 
use std::collections::HashMap;
let mut g: HashMap<i32, Vec<i32>> = HashMap::new();

    g.entry(u).or_default().push(v);   // entry API builds the Vec on demand
}
println!("{:?}", g[&0]);               // [1, 2]
[1, 2]
C++

for (int n : g[0]) std::cout << n << " ";  // 1 2

Language Comparison: All five compose a graph from primitives — there is no stdlib graph anywhere. Adjacency list: Python defaultdict(list), Go map[int][]int, TS Map<number,number[]>, Rust HashMap<_,Vec<_>> (the entry().or_default() idiom), C++ unordered_map<int,vector<int>> (operator[] auto-creates the vector). Adjacency matrix is a 2D array/vector everywhere — mind the row-init traps. When vertices are small integers, a Vec<Vec>/slice-of-slices indexed by id beats a hash map on both speed and cache locality; reserve the hash map for string or sparse labels. Rule of thumb: list for sparse (E << V²), matrix for dense or O(1) edge probing.
Common Mistakes:
❌ [Python] m = [[0]*n]*n creates n references to the same row — writing m[0][0] mutates every row.
✅ [Python] Build fresh rows: [[0]*n for _ in range(n)].
❌ [Go] Allocating make([][]int, n) but never making the inner slices — the rows are nil, and m[i][j] = 1 panics.
✅ [Go] Loop and m[i] = make([]int, n) for each row.
❌ [TS] Array(n).fill([]) fills with the same array reference (the same aliasing bug as Python).
✅ [TS] Array.from({length: n}, () => []) makes an independent row each time.
❌ [C++] Assuming g[u] is a read-only lookup — it default-inserts an empty vector for an absent key, silently adding vertices.
✅ [C++] Use g.find(u)/g.contains(u) to test; reserve g[u] for when auto-create is intended.
❌ [Rust] Indexing g[&u] for a possibly-absent vertex panics.
✅ [Rust] Read with g.get(&u) (an Option); insert-or-append with g.entry(u).or_default().push(v).
Key Rules:
  • Adjacency list for sparse graphs (O(V+E) space, fast neighbor iteration); adjacency matrix for dense graphs or O(1) edge-existence tests (O(V²) space).
  • Index by vertex id with an array/Vec when vertices are small integers; use a hash map only for string or sparse labels.
  • Beware 2D-array aliasing — build each row independently in every language.

Graph Traversal: BFS & DFS Graphs & Traversal
Why it matters: BFS and DFS are the workhorses behind shortest-path-in-unweighted-graphs, connectivity, cycle detection, topological sort, and flood fill. They differ only in the structure holding the frontier — a queue (BFS, level order) versus a stack/recursion (DFS, deep first) — but that one choice changes which problems each solves.

Both visit every reachable vertex exactly once, guarded by a visited set that prevents revisiting (and infinite loops on cycles). BFS uses a FIFO queue and discovers vertices in order of edge-distance, so it finds shortest paths by hop count; DFS uses a LIFO stack (usually the call stack, via recursion) and naturally expresses backtracking and topological order. The recursive form is exactly where Rust's borrow rules and C++'s recursive-lambda quirk show up.

Example 1 — BFS order from vertex 0 on 0→1, 0→2, 1→3, 2→3

Python 
from collections import deque
g = {0:[1,2], 1:[3], 2:[3], 3:[]}
def bfs(start):
    seen, q, order = {start}, deque([start]), []
    while q:
        u = q.popleft()               # FIFO -> breadth first
        order.append(u)
        for v in g[u]:
            if v not in seen: seen.add(v); q.append(v)  # mark on enqueue
    return order
print(bfs(0))                          # [0, 1, 2, 3]
[0, 1, 2, 3]
Go 
g := map[int][]int{0:{1,2}, 1:{3}, 2:{3}, 3:{}}
seen := map[int]bool{0: true}
q := []int{0}
var order []int
for len(q) > 0 {
    u := q[0]; q = q[1:]              // dequeue front

Language Comparison: The algorithm is identical everywhere; only the frontier structure and recursion ergonomics differ. BFS queue: Python collections.deque (O(1) popleft), Go a slice reslice, TS an array with shift (O(n) — use a head index on big graphs), Rust VecDeque, C++ std::queue. The insert-returns-whether-new trick makes the visited check a one-liner in Rust (HashSet::insert) and C++ (set::insert().second). DFS recursion: Python/JS/C++ recurse freely (mind stack depth), Go needs the declare-then-assign closure (var dfs func(int); dfs = ...) to self-reference, and Rust threads &mut seen/&mut order through each call because it can't re-borrow a captured mutable reference. For deep graphs, convert recursive DFS to an explicit stack.
Common Mistakes:
❌ [Python] Recursive DFS blows the ~1000 default recursion limit on deep/large graphs (RecursionError); or forgetting to add to seen before recursing, causing revisits.
✅ [Python] Use an explicit stack for deep graphs (or raise the limit judiciously), and mark visited on discovery.
❌ [Go] Using a slice queue with q = q[1:] keeps the whole backing array alive for long-running loops (a slow leak).
✅ [Go] For large/long-lived queues use container/list or periodically compact the slice.
❌ [TS] q.shift() is O(n), making BFS O(V·E); and marking visited on dequeue lets a node get enqueued twice.
✅ [TS] Mark visited when you push, and use a head-index ring buffer instead of shift.
❌ [Rust] Trying to capture &mut seen in a recursive closure — the borrow checker rejects the re-entrant mutable borrow; and g[&u] panics on a missing key.
✅ [Rust] Use a free fn taking &mut params (shown), or an explicit stack; read edges with g.get(&u).
❌ [C++] A bare recursive lambda can't name itself, and g[u] default-inserts an empty vector for an absent u during traversal.
✅ [C++] Use std::function (or a named function) for recursion, and g.find/contains to avoid accidental inserts.
Key Rules:
  • BFS (queue) gives shortest paths by edge count and level order; DFS (stack/recursion) gives depth-first order, backtracking, topological sort, and cycle detection.
  • Always guard with a visited set and mark a node visited when you first reach/enqueue it — not when you pop it — to avoid duplicate work.
  • Convert deep recursive DFS to an explicit stack to dodge stack overflow; recursion depth is a real limit (Python ~1000, others bound by the OS thread stack).

Union-Find (Disjoint Set Union) Graphs & Traversal
Why it matters: Union-Find tracks a partition of elements into disjoint sets and answers "are a and b in the same set?" and "merge these two sets" in near-constant amortized time. It's the engine behind Kruskal's minimum spanning tree, connected-components labeling, cycle detection in undirected graphs, and dynamic connectivity — anywhere you repeatedly merge groups and query membership.

Each element points to a parent; a set is a tree whose root is its representative. find(x) walks to the root; union(a, b) links one root under the other. Two optimizations bring it to nearly O(1) amortized (inverse Ackermann): path compression (flatten the walk during find) and union by rank/size (attach the smaller tree under the larger). The parent-array core is essentially identical across all five languages.

Language Comparison: Union-Find is the one structure that looks almost byte-for-byte identical in all five — it's just an integer array. The only real differences are initialization (Python list(range(n)), Go/TS loops or Array.from, Rust (0..n).collect(), C++ std::iota) and recursion for path compression: Rust and C++ do the elegant recursive p[x] = find(p[x]), Go can't self-reference a bare func literal (so it uses var find func... or the iterative halving form), and Python/TS use iterative halving to sidestep recursion-depth limits. With both path compression and union-by-size/rank, m operations on n elements run in O(m·α(n)) — α is the inverse Ackermann function, ≤ 4 for any realistic n, so effectively constant. Drop either optimization and per-op cost regresses toward O(log n).
Common Mistakes:
❌ [All] Unioning raw elements — parent[a] = b instead of parent[find(a)] = find(b) — which corrupts the forest; you must link roots.
✅ [All] Always union the representatives returned by find, and test membership with find(a) == find(b), never a == b.
❌ [Python] Recursive find with compression can exceed the recursion limit on a long chain built without union-by-size.
✅ [Python] Use iterative path halving (shown), and always pair it with union-by-size to keep trees shallow.
❌ [Go] Writing find as a bare recursive func literal won't compile — it can't reference itself.
✅ [Go] Declare var find func(int) int first, or use the iterative form.
❌ [Rust] Giving find a &self receiver — path compression mutates, so it needs &mut self; and swapping roots needs std::mem::swap.
✅ [Rust] fn find(&mut self, ...), and mem::swap(&mut ra, &mut rb) for union-by-size.
❌ [C++] Skipping union-by-size/rank — without it, adversarial unions build an O(n) chain and every find is slow.
✅ [C++] Combine size/rank with the recursive compressing find for the α(n) bound.
Key Rules:
  • Apply both path compression (in find) and union by size/rank (in union) for the near-constant α(n) amortized bound; either alone is weaker.
  • Union-Find supports merge + same-set queries but not split/delete — it's for monotonically-growing connectivity only.
  • Store elements as 0..n-1 integer ids indexing a flat array; map external keys (strings) to ids first.

Specialized Structures

Beyond the everyday map and list sit structures tuned for one job. A trie makes prefix queries and autocomplete near-free; a bitset packs boolean sets into machine words for order-of-magnitude space and speed wins. Both are simple to build from primitives, and both expose how each language handles the underlying machinery — recursive node ownership for tries, low-level bit twiddling for bitsets.

Tries (Prefix Trees) Specialized Structures
Why it matters: a trie stores strings by shared prefix, so lookup and prefix queries cost O(key length) regardless of how many keys are stored — the backbone of autocomplete, spell-checkers, IP routing tables, and dictionary problems. Where a hash set answers only "is this exact word present?", a trie also answers "how many stored words start with 'pre'?" in a single downward walk.

Each node holds a map/array of child links (one per possible next character) plus an end-of-word marker. Inserting a word walks and creates nodes character by character; searching walks them; a prefix query simply stops at the prefix's last node. The design axis is the children container: a hash map (any alphabet) versus a fixed array (fast and dense, e.g. 26 lowercase letters). Recursive ownership of child nodes is where Rust (HashMap<char, Trie>) and C++ (unique_ptr) get interesting.

Rust 
use std::collections::HashMap;
#[derive(Default)]
struct Trie { kids: HashMap<char, Trie>, words: u32 }
impl Trie {
    fn insert(&mut self, w: &str) {
        let mut n = self; n.words += 1;
        for c in w.chars() { n = n.kids.entry(c).or_default(); n.words += 1; }
    }
    fn prefix_count(&self, p: &str) -> u32 {
        let mut n = self;
        for c in p.chars() {
            match n.kids.get(&c) { Some(x) => n = x, None => return 0 }
        }
        n.words
    }
}

C++ 
#include <memory><unordered_map><string>
struct Trie {
    std::unordered_map<char, std::unique_ptr<Trie>> kids; int words = 0;
    void insert(const std::string& w) {
        Trie* n = this; n->words++;
        for (char c : w) {
            if (!n->kids[c]) n->kids[c] = std::make_unique<Trie>();
            n = n->kids[c].get(); n->words++;
        }
    }
    int prefix_count(const std::string& p) {
        Trie* n = this;
        for (char c : p) { auto it = n->kids.find(c); if (it==n->kids.end()) return 0; n = it->second.get(); }
        return n->words;
    }
};

Language Comparison: The shape — a node with a child map and a flag/count — is identical; the languages differ in how a node owns its children. The GC languages (Python, Go, TS) just hold references and let the collector reclaim the tree. Rust models it as owned values (HashMap<char, Trie> with #[derive(Default)] + the entry().or_default() idiom), so the whole trie drops recursively with its root. C++ owns children through unique_ptr, giving the same automatic recursive cleanup via RAII (though a very deep trie can overflow the stack on destruction). Children storage: a hash map handles any alphabet; for a small fixed alphabet (26 lowercase letters), a fixed array [Option<Box<Trie>>; 26] / array<unique_ptr<Trie>, 26> is faster and more cache-friendly. A trie beats a hash set precisely when you need prefix queries; for exact membership only, a hash set is simpler and usually faster.
Common Mistakes:
❌ [Python] Overwriting a branch with node.kids[c] = Trie() unconditionally; or declaring a class-level mutable kids = {} shared across all instances.
✅ [Python] Use node.kids.setdefault(c, Trie()), and initialize kids inside __init__.
❌ [Go] Writing into a nil child map panics (reading it is fine) — a zero-value Trie{} has a nil kids.
✅ [Go] Always construct nodes with New() so the child map exists before any insert.
❌ [TS] Iterating with for (i=0; i<w.length; i++) w[i] yields UTF-16 code units, splitting emoji and corrupting non-BMP keys.
✅ [TS] Iterate with for..of (code points) when the alphabet includes non-ASCII characters.
❌ [Rust] Attempting stacked borrows of parent and child at once — the borrow checker rejects it.
✅ [Rust] Walk with a single moving &mut (n = n.kids.entry(c).or_default()), advancing the borrow each step.
❌ [C++] Using n->kids[c] in a read-only walk — operator[] default-inserts an empty unique_ptr, silently growing the trie and breaking prefix counts.
✅ [C++] Use find for read-only walks; reserve operator[] for inserts.
Key Rules:
  • Use a trie when you need prefix queries (autocomplete, "count words with prefix", longest-prefix match); for exact membership only, prefer a hash set.
  • Lookup/insert are O(key length), independent of how many keys are stored — the trie's headline advantage over balanced trees.
  • Choose the child container by alphabet: a hash map for arbitrary/large alphabets, a fixed array for small dense ones (e.g. 26 letters) for speed and cache locality.

Bitsets & Bitmap Structures Specialized Structures
Why it matters: a bitset packs booleans one-per-bit into machine words, so a set of integers in 0..n uses n/8 bytes instead of n booleans — 8–64× less memory — and whole-set operations (union, intersection) become single bitwise instructions over 64 elements at a time. It's the trick behind fast small-integer sets, Bloom-filter bit arrays, chess/board state, and columnar-database filters.

Model a set of small non-negative integers as bits: element k lives in bit k — word k >> 6, bit k & 63 for 64-bit words. Insert = OR a mask in, remove = AND a cleared mask, test = AND and check non-zero. Whole-set union/intersection/difference are just |/&/& ~ applied word-by-word, processing 64 elements per instruction; popcount gives the set size. Languages split into "has a built-in bitset" and "bring your own words."

Example 2 — set algebra: union and intersection of two integer sets as bit ops

With A = {1,2,3} and B = {2,3,4}, the union is {1,2,3,4} (bits 11110 = 30) and the intersection is {2,3} (bits 1100 = 12) — one instruction each.

Python 
def mask(xs):
    m = 0
    for x in xs: m |= 1 << x
    return m
a, b = mask([1,2,3]), mask([2,3,4])
union, inter = a | b, a & b
print(union, inter, union.bit_count())  # 30 12 4  (bit_count = popcount, Py3.10+)
30 12 4

Language Comparison: Support ranges from turnkey to DIY. C++ has std::bitset<N> (fixed size; member set/test/reset/count and operators | & ^ ~) plus vector<bool> (dynamic, space-packed, but a quirky proxy type). Python integers are arbitrary-precision bitsets for free — <<, |, &, ~ all work, and int.bit_count() (3.10+) is popcount. Rust uses plain u64s (with count_ones/leading_zeros builtins) for ≤64 elements, and the bitvec crate for larger dynamic bitsets. Go has no bitset type — use uint64 words plus the math/bits package (OnesCount64, TrailingZeros64) and its distinctive &^ (AND-NOT) clear operator. JS/TS uses BigInt for arbitrary-width bit math (regular numbers are only safe for 32-bit, signed, bitwise ops) or a Uint32Array for large fixed bitmaps. Past one word, everyone loops word-by-word — still ~64 elements per instruction.
Common Mistakes:
❌ [JS/TS] Bit ops on regular numbers past bit 31 — 1 << 31 is negative and 1 << 32 wraps to 1 (the shift count is mod 32, operands coerced to signed int32).
✅ [JS/TS] Use BigInt (1n << 40n) or a typed-array bitmap for anything wider than 31 bits.
❌ [C++] 1 << k where 1 is a 32-bit int and k ≥ 31 — signed overflow/UB; and vector<bool> isn't a real container of bools (proxy refs, can't take &).
✅ [C++] Use 1ULL << k, and prefer std::bitset/vector<uint64_t> when you need real bit storage.
❌ [Go] Reaching for & ~mask to clear when Go offers a dedicated AND-NOT operator.
✅ [Go] Use b &^= 1 << k to clear a bit — idiomatic and avoids a separate NOT.
❌ [Rust] 1 << k defaults the literal to i32; shifting into the sign bit or by ≥ width panics (debug) or is a logic bug.
✅ [Rust] Type the literal to the word width: 1u64 << k.
❌ [Python] Forgetting that ~x on an int yields a negative number with infinite leading ones — masking without bounding can surprise.
✅ [Python] Read a single bit with (x >> k) & 1, and bound with an explicit width mask when emulating fixed-width sets.
Key Rules:
  • Use a bitset when your set is dense over a bounded small-integer universe — you get 8–64× space savings and whole-word union/intersection/difference in single instructions.
  • Element k maps to word k >> 6, bit k & 63 (64-bit words); set with |, clear with & ~mask (Go: &^), test with & mask.
  • Mind each language's integer width and signedness for shifts: JS numbers are 32-bit-signed for bit ops (use BigInt), C++/Rust literals default to 32-bit (use 1ULL/1u64), Python is unbounded.
