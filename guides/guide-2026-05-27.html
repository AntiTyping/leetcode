#170 · Two Sum III - Data Structure Design

Easy Hash / Design 2 days since solved

The Problem

Design a data class that supports two operations: add(number) to insert a value into the structure, and find(value) to return true if there exists any pair of stored numbers that sum to value. The pair must be two distinct elements (not the same index used twice).

Quick Reminder

Keep a Counter of added numbers. On find(v), for each key n check whether v - n exists; if n == v - n, require count ≥ 2.

Key Insight

Decide which operation should be O(1) vs O(n). For mixed workloads, the standard tradeoff is fast add (push into list/counter) and linear find using complementary-lookup. The trap is the "self-pair" case: v = 2n needs the same number twice, so you must track counts, not just presence.

Explanation: 1 + 3 = 4 so find(4) is true. No pair sums to 7.

Output: true

Input: add(0), add(-1), add(2), find(-1), find(2)
Output: true, true
Explanation: Negative numbers count: 0 + (-1) = -1. Tricky: 0 + 2 = 2 too.

class TwoSum(object):

    def __init__(self):
        self.numbers = []

    def add(self, number):
        """
        :type number: int
        :rtype: None
        """
        self.numbers.append(number)

    def find(self, value):
        """
        :type value: int
        :rtype: bool
        """
        h = set()
        for i in range(len(self.numbers)):
            n = self.numbers[i]
            diff = value - n
            if diff in h:
                return True
            else:
                h.add(n)
        return False

from collections import Counter

class TwoSum:
    def __init__(self):
        self.counts = Counter()

    def add(self, number: int) -> None:
        self.counts[number] += 1

    def find(self, value: int) -> bool:
        for n, c in self.counts.items():
            diff = value - n
            if diff == n:
                if c >= 2:
                    return True
            elif diff in self.counts:
                return True
        return False

            if c >= 2 {
                return true

            return true
        }
    }
    return false
}

#234 · Palindrome Linked List

Easy Linked List / Two Pointers 3 days since solved

The Problem

Given the head of a singly linked list, return true if it reads the same forwards and backwards. The follow-up asks for O(n) time and O(1) extra space.

Quick Reminder

Two approaches: dump values to an array and two-pointer it (O(n) space), or do slow/fast to find the middle, reverse the second half in place, then walk both halves together.

Key Insight

You cannot index backwards in a singly linked list, but you can physically flip the second half. The slow/fast pointer trick lands slow at the midpoint in one pass; reversing from there yields a backward iterator without any extra storage.

class Solution(object):
    def isPalindrome(self, head):
        """
        :type head: Optional[ListNode]
        :rtype: bool
        """
        a = []

        p = head
        while p != None:
            a.append(p.val)
            p = p.next

        l, r = 0, len(a) - 1

        while l < r:
            if a[l] != a[r]:
                return False
            l += 1
            r -= 1

        return True

        :rtype: bool

        # find middle
        # n
        while fast and fast.next:
            fast = fast.next.next
            slow = slow.next

        # reverse right
        # n

        while slow:
            tmp = slow.next
            slow.next = prev
            prev = slow
            slow = tmp

        while r:
            if l.val != r.val:
                return False
            l = l.next
            r = r.next
        return True

func isPalindrome(head *ListNode) bool {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
    }
    var prev *ListNode
    for slow != nil {
        slow.Next, prev, slow = prev, slow, slow.Next
    }
    for prev != nil {
        if prev.Val != head.Val {
            return false
        }
        prev = prev.Next
        head = head.Next
    }
    return true
}

Rotation produces a mirror, so it is naturally a two-pointer problem. Crucially, the middle character of an odd-length string must be a self-rotator (0, 1, 8) — that case falls out for free when you use l <= r instead of l < r.

#347 · Top K Frequent Elements

Medium Heap / Hash / Sorting 3 days since solved

The Problem

Given an integer array nums and an integer k, return the k most frequent elements. The answer may be returned in any order, and the test cases guarantee a unique answer.

Quick Reminder

Count frequencies into a hash, then either: (a) use heapq.nlargest for O(n log k), or (b) bucket-sort by frequency for O(n) — the latter exploits the fact that frequencies are bounded by n.

Key Insight

The frequency of any element is at most n, so we can use frequencies as array indices. That replaces the log factor of a heap or sort with pure O(n) bucket placement. This is the classic "small bounded range" trick — recognize it whenever values live in [0, n].

class Solution:
    def topKFrequent(self, nums: List[int], k: int) -> List[int]:
        """
        :type nums: List[int]
        :type k: int
        :rtype: List[int]
        """

        # build counts
        # n
        d = {}
        for n in nums:
            d[n] = 1 + d.get(n, 0)

        # buckets
        # n
        freq = [[] for x in range(len(nums) + 1)]
        for n, c in d.items():
            freq[c].append(n)

        # select top k
        # k
        ans = []
        for i in range(len(freq) - 1, 0, -1):
            for n in freq[i]:
                ans.append(n)
                if len(ans) == k:
                    return ans

        return ans
        # n + n log n + k

from collections import Counter

class Solution:
    def topKFrequent(self, nums: list[int], k: int) -> list[int]:
        counts = Counter(nums)
        buckets = [[] for _ in range(len(nums) + 1)]
        for num, c in counts.items():
            buckets[c].append(num)
        out = []
        for i in range(len(buckets) - 1, 0, -1):
            for num in buckets[i]:
                out.append(num)
                if len(out) == k:
                    return out
        return out
Go — Optimal (bucket sort) 
func topKFrequent(nums []int, k int) []int {
    counts := map[int]int{}
    for _, n := range nums {
        counts[n]++
    }
    buckets := make([][]int, len(nums)+1)
    for n, c := range counts {
        buckets[c] = append(buckets[c], n)
    }
    out := make([]int, 0, k)
    for i := len(buckets) - 1; i > 0 && len(out) < k; i-- {
        for _, n := range buckets[i] {
            out = append(out, n)
            if len(out) == k {
                return out
            }
        }
    }
    return out
}

#455 · Assign Cookies

Easy Greedy / Two Pointers 1 day since solved

The Problem

Each child i in g has greed g[i]; each cookie j in s has size s[j]. You can give at most one cookie per child, and child i is content iff s[j] >= g[i]. Maximize the count of content children.

Quick Reminder

Sort both arrays. Walk a pointer on each: if the current cookie satisfies the current child, advance both and increment the answer; otherwise discard the cookie (it cannot satisfy anyone smaller because the array is sorted).

Key Insight

Greedy with sorted inputs — always feed the least-greedy child the smallest cookie that works. The exchange argument: if an optimal solution ever uses a bigger cookie for a smaller child, you can swap to free up the small cookie without hurting the count. Sort + two-pointer is the canonical greedy template.

The Problem

class Solution(object):
    def reverseStr(self, s, k):
        """
        :type s: str
        :type k: int
        :rtype: str
        """
        l = 0
        ss = list(s)
        while l < len(s):
            if len(s) - l < k:
                a, b = l, len(s) - 1
                while a < b:
                    ss[a], ss[b] = ss[b], ss[a]
                    a += 1
                    b -= 1
            elif len(s) - l < 2 * k:
                a, b = l, l + k - 1
                while a < b:
                    ss[a], ss[b] = ss[b], ss[a]
                    a += 1
                    b -= 1
            else:
                a, b = l, l + k - 1
                while a < b:
                    ss[a], ss[b] = ss[b], ss[a]
                    a += 1
                    b -= 1
            l += 2 * k

        return "".join(ss)

#136 · Single Number

Easy Bit Manipulation XOR Trick

The Problem

Given a non-empty integer array nums where every element appears exactly twice except for one, find the lone element. Constraint: linear time and constant extra space.

Brute Force (Python)

Python — Hash count, O(n) time, O(n) space 
from collections import Counter

        for n, c in Counter(nums).items():
            if c == 1:
                return n

Brute Force (Go)

Go — Hash count, O(n) time, O(n) space 
func singleNumber(nums []int) int {
    counts := map[int]int{}
    for _, n := range nums {
        counts[n]++
    }
    for n, c := range counts {
        if c == 1 {
            return n
        }
    }
    return -1
}

Key Insight

XOR is its own inverse: a ^ a == 0 and x ^ 0 == x. XOR is also commutative and associative, so the order of operations does not matter. XOR-ing every element together cancels all the pairs and leaves the lone element.

5 Thinking Steps

  1. What property uniquely separates the lone element from the rest? It appears an odd number of times (once); all others appear an even number of times (twice).
  2. Is there an operator that distinguishes odd/even occurrence cheaply? Yes — XOR. Each pair XORs to 0; the lone element XORs with 0 to itself.
  3. Does the operator depend on element order? XOR is commutative and associative, so no.
  4. Do we need to scan twice or buffer anything? No — a single pass with one accumulator suffices.
  5. What is the loop invariant? After processing index i, the accumulator equals the XOR of all elements seen so far, which equals the unique element XOR-ed with 0 once for every cancelled pair.

Why This Works

For any integer a: a ^ a = 0 (identical bits cancel) and a ^ 0 = a. Because XOR is associative and commutative, we can rearrange the fold so each duplicate pair is adjacent: (a ^ a) ^ (b ^ b) ^ ... ^ x = 0 ^ 0 ^ ... ^ x = x. The result is exactly the lone element, with no auxiliary data and O(1) space.

Dangerous Mistake

Initialising the accumulator to 1 (or any non-zero seed) instead of 0. XOR's identity element is 0; any other seed silently corrupts the answer and the bug is invisible on small examples that happen to cancel.

3 Pitfalls

  • Assuming non-negative inputs. The problem allows negatives. XOR still works because Python and Go represent ints in two's complement — but make sure you do not branch on signs.
  • Trying to use sum arithmetic. 2 * sum(set(nums)) - sum(nums) is O(n) but O(n) space and overflows in fixed-width languages. XOR is strictly cleaner.
  • Returning early on first duplicate. The lone element can appear anywhere — first, middle, or last. You must fold over the entire array.

Python Features

  • ^ — bitwise XOR operator. Works on Python ints of arbitrary width.
  • functools.reduce(operator.xor, nums) — one-liner fold; equivalent to a manual loop.
  • 0 is the XOR identity — always seed the accumulator with it.
  • Python integers are not fixed-width, so XOR never overflows or wraps.
  • bin(x) — useful for debugging: prints the binary representation as '0b...'.

Go Features

  • ^ — bitwise XOR in Go (same symbol as Python).
  • ^= — compound XOR assignment, ideal for the accumulator loop.
  • Go's int is fixed width (32 or 64 bits), but XOR never overflows — it operates bit-by-bit independently.
  • fmt.Sprintf("%b", x) — binary format for debugging.
  • The math/bits package provides utilities like bits.OnesCount but is not needed here.
Brute: O(n) time, O(n) space
Optimal: O(n) time, O(1) space (XOR fold)

Connections

Generalizes to #137 Single Number II (every element thrice except one, requires bitwise mod-3) and #260 Single Number III (two singletons, requires grouping by a distinguishing bit). The XOR-cancellation idea also powers #268 Missing Number and parity-bit error-detection.

Easy Bit Manipulation Bit-by-Bit Traversal

The Problem

Reverse the bits of a 32-bit unsigned integer. For example, the input 00000010100101000001111010011100 should produce 00111001011110000010100101000000.

Brute Force (Python)

Python — String reverse, O(32) time 
class Solution:
    def reverseBits(self, n: int) -> int:
        b = bin(n)[2:].zfill(32)
        return int(b[::-1], 2)

Brute Force (Go)

Go — String reverse, O(32) time 
import (
    "fmt"
    "strconv"
)

func reverseBits(n uint32) uint32 {
    s := fmt.Sprintf("%032b", n)
    runes := []rune(s)
    for l, r := 0, len(runes)-1; l < r; l, r = l+1, r-1 {
        runes[l], runes[r] = runes[r], runes[l]
    }
    v, _ := strconv.ParseUint(string(runes), 2, 32)
    return uint32(v)
}

Key Insight

Reversing bits is just iteratively pulling the lowest bit off the input and pushing it onto the back of the output. (n >> 1) peels; (out << 1) | bit appends. Loop 32 times. No strings, no conversions — pure bit-pushing.

5 Thinking Steps

  1. What do "reverse bits" actually require? Bit i in the input must become bit 31 - i in the output.
  2. What is the cheapest way to read bit i of n? Either (n >> i) & 1, or repeatedly shift n right and read the lowest bit.
  3. What is the cheapest way to write bit 31 - i? Build the result by shifting it left and OR-ing in the next bit — that automatically reverses positions.
  4. How many iterations? Exactly 32 — constant for fixed-width integers.
  5. What is the loop invariant? After iteration k, out holds the first k low bits of the original input but in reversed order, sitting in out's low k positions.

Why This Works

Each step extracts the least-significant bit of the remaining input (n & 1) and shifts it into the least-significant position of the output, but the output is also shifted left first — so the bit you wrote last iteration is pushed to a higher position. After 32 iterations, the bit you pulled first (originally bit 0) has been shifted left 31 times, landing in bit 31; the bit you pulled last (originally bit 31) sits at bit 0. That is exactly the reversal.

Dangerous Mistake

Stopping the loop early when n == 0. Even after the input is exhausted, the remaining iterations still need to left-shift the result to position the high-order bits correctly. Always run all 32 iterations for a 32-bit input.

3 Pitfalls

  • Forgetting Python ints are unbounded. When porting to languages with fixed width, you must mask with 0xFFFFFFFF to keep the result in 32 bits.
  • Confusing arithmetic vs logical shift. In Python all int shifts are logical; in Go use uint32 to avoid signed-shift sign extension.
  • Off-by-one in the loop count. 32 iterations means exactly 31 left-shifts performed on the very first bit you read — not 32. The shift before the OR each iteration is the correct accounting.

Python Features

  • << — left shift (multiply by 2).
  • >> — right shift (integer divide by 2); always logical in Python.
  • & — bitwise AND, used to extract n & 1, the lowest bit.
  • | — bitwise OR, used to merge the next bit into the output.
  • bin(x) — the brute force aid, returns binary string like '0b101'.
  • int(s, 2) — parse a binary string back to an integer.

Go Features

  • uint32 — fixed-width unsigned integer, the natural type for this problem.
  • << and >> — shifts on unsigned types do not sign-extend, so logical shift comes for free.
  • &, | — bitwise AND/OR.
  • math/bits.Reverse32(n) — standard library one-liner that does exactly this operation; great to know but you should be able to implement it by hand.
  • fmt.Sprintf("%032b", n) — 32-wide zero-padded binary format string.
Brute: O(32) time, O(32) space (string buffer)
Optimal: O(32) time, O(1) space

Connections

Companion to #191 Number of 1 Bits (count set bits, same bit-by-bit pattern) and #338 Counting Bits (DP over bit-shifts). The shift-and-OR pattern reappears in encoding/decoding (Huffman, varint) and in CRC implementations.

#371 · Sum of Two Integers

Medium Bit Manipulation Half-Adder

The Problem

Given two integers a and b, return a + b without using the operators + or -. Inputs can be negative; the function must behave like two's-complement 32-bit addition.

Brute Force (Python)

Python — Increment loop (cheats with += 1) 
class Solution:
    def getSum(self, a: int, b: int) -> int:
        # Pretend "+= 1" is allowed for illustration only.
        # Loops abs(b) times; not acceptable but shows the gap.
        sign = 1 if b >= 0 else -1
        for _ in range(abs(b)):
            a += sign
        return a

Brute Force (Go)

Go — Increment loop (illustrative only) 
func getSum(a int, b int) int {
    step := 1
    if b < 0 {
        step = -1
        b = -b
    }
    for i := 0; i < b; i++ {
        a = a + step // illustrative, the real puzzle forbids +
    }
    return a
}

Key Insight

Binary addition decomposes into two pieces: the bit-by-bit sum without carry is a XOR b; the carry bits are (a AND b) << 1. Add those two values the same way recursively until carry is 0. This is the structure of every hardware adder.

5 Thinking Steps

  1. What does adding two bits produce? A sum bit and a carry bit. Sum is XOR; carry is AND.
  2. How do we combine sum-bits and carry-bits into a full sum? Shift the carry left by 1 (since it goes into the next column), then add it to the sum-bits.
  3. But "add" is what we're trying to define! Use the same procedure recursively: re-compute XOR and AND on the new pair until the carry term becomes 0.
  4. Why does it terminate? Each iteration shifts the carry left at least one position, so within 32 iterations the carry exits the word and becomes 0 (mod the mask).
  5. How do we handle negatives in Python's unbounded ints? Mask with 0xFFFFFFFF to simulate 32-bit arithmetic, and if the high bit is set after the loop, convert back via ~(x ^ 0xFFFFFFFF) to recover the Python signed value.

Why This Works

For column i, a[i] XOR b[i] equals (a[i] + b[i]) mod 2 — the sum without the carry. And a[i] AND b[i] equals 1 only when both are 1, which is exactly when column i generates a carry into column i+1. The total sum is therefore (sum-bits) + (carry-bits shifted by 1), and recursing applies the same identity until no carry remains. This is the canonical "half-adder + carry chain" decomposition.

Dangerous Mistake

In Python, omitting the 32-bit mask. Without a &= 0xFFFFFFFF, the carry chain never terminates for negative inputs — Python ints grow without bound and the loop spins forever. Always simulate fixed width when emulating hardware arithmetic.

3 Pitfalls

  • Mishandling negative results in Python. After masking, if the high bit is set, the value still looks positive to Python. Reconstruct the signed value with ~(a ^ 0xFFFFFFFF) when a > 0x7FFFFFFF.
  • Forgetting that AND-then-shift can overflow the mask. If you do not re-mask each iteration, the carry term grows beyond 32 bits and termination depends on incidental cancellation rather than a guaranteed shift-out.
  • Using arithmetic shift on signed ints in Go. A signed right shift sign-extends and can introduce spurious 1-bits. Either use uint32 throughout or be deliberate about masking.

Python Features

  • ^ — XOR computes the carryless sum.
  • & — AND computes the carry generator.
  • << — left shift propagates the carry one column.
  • ~ — bitwise NOT, used to reconstruct a signed value from a masked unsigned representation.
  • 0xFFFFFFFF — the 32-bit mask; pin this constant in muscle memory for bit problems.
  • int.bit_count() (Python 3.10+) — counts set bits; handy for sanity checks.

Go Features

  • uint32 — use it to skip the manual mask; overflow naturally wraps.
  • ^ — XOR. (Note: ^x as unary is bitwise NOT in Go, unlike Python's ~.)
  • &, |, <<, >> — standard bitwise operators.
  • math/bits.Add32(x, y, carry) — library helper returning (sum, carry-out); illustrative of how hardware exposes the operation.
  • int32(uint32(x)) — round-trip conversion to recover signed interpretation after masking.
Brute: O(|b|) time, O(1) space (increment loop)
Optimal: O(32) time, O(1) space (XOR + carry loop)

#14 · Longest Common Prefix

EasyStringVertical Scan

The Problem

Given an array of strings strs, return the longest common prefix shared by all of them. If there is no common prefix, return the empty string "". For example, ["flower", "flow", "flight"] returns "fl", while ["dog", "racecar", "car"] returns "".

Brute Force

Pick the first string as a candidate prefix and progressively shorten it until every other string starts with it. Each shrink lops off one character from the right; in the worst case we shrink to the empty string. This is the classic horizontal-scan idea: pairwise reduce, walking down the array.

Python — Brute Force 
def longestCommonPrefix(strs):
    if not strs:
        return ""
    prefix = strs[0]
    for s in strs[1:]:
        # Shrink prefix until s starts with it
        while not s.startswith(prefix):
            prefix = prefix[:-1]
            if prefix == "":
                return ""
    return prefix
Go — Brute Force 
func longestCommonPrefix(strs []string) string {
    if len(strs) == 0 {
        return ""
    }
    prefix := strs[0]
    for _, s := range strs[1:] {
        for !strings.HasPrefix(s, prefix) {
            prefix = prefix[:len(prefix)-1]
            if prefix == "" {
                return ""
            }
        }
    }
    return prefix
}

Why optimize? The brute force is already linear in total characters, but vertical scan stops at the first mismatching column without comparing entire strings.

Key Insight

Instead of shrinking a candidate prefix, walk column-by-column: at column i check that every string has the same character there. The moment any string is shorter than i or differs, you have your answer. This makes the early-exit explicit and avoids repeated startswith work.

How to Think About It

  1. Handle the empty-input edge case first — return "".
  2. Iterate column index i from 0 upward, anchored on strs[0].
  3. For each i, scan every other string: if i ≥ len(s) or s[i] != strs[0][i], stop.
  4. If the loop completes for column i, that character is in the prefix — advance.
  5. When you stop, return strs[0][:i].

Why This Works

A common prefix must agree at every position, so as soon as one string disagrees at column i, no character at position i or later can be in the answer. Walking columns finds the failure point with no wasted comparisons.

⚠ Most Dangerous Mistake

Forgetting to check that i is within bounds of every string. If strs[0] is the longest, you can read off the end of a shorter string and either crash (Go) or silently misbehave with slicing (Python returns empty slice, not error).

  • Returning a zero-length prefix as None instead of "".
  • Using strs[0][0] without checking strs[0] is non-empty.
  • Comparing strings via == on slices when you really want a column compare.

#67 · Add Binary

EasyStringBit Manipulation

The Problem

Given two binary strings a and b, return their sum as a binary string. For example, a = "11", b = "1" returns "100". The inputs can be very long (up to 10^4 characters), so converting to native integers is fine in Python but unsafe in many other languages.

Brute Force

Parse both binary strings into Python integers using int(s, 2), add them, then format back with bin(). This sidesteps the entire problem and isn't a real interview answer because it presumes arbitrary-precision integers. It is still useful as a sanity test.

Python — Brute Force 
def addBinary(a, b):
    # bin(x) returns a string like '0b101' -- strip the prefix
    return bin(int(a, 2) + int(b, 2))[2:]
Go — Brute Force 
func addBinary(a string, b string) string {
    // ParseInt with base 2; fails for very long strings (over 63 bits)
    x, _ := strconv.ParseInt(a, 2, 64)
    y, _ := strconv.ParseInt(b, 2, 64)
    return strconv.FormatInt(x+y, 2)
}

Why optimize? Real inputs can exceed 64 bits. The optimal does grade-school addition digit-by-digit with a carry, which works for any length.

Key Insight

Binary addition is just decimal addition restricted to digits 0 and 1 with carry propagating when a column sums to 2 or 3. Walk both strings from the right with two pointers and a single integer carry; emit digits in reverse, then flip.

How to Think About It

  1. Initialize two indices i = len(a)-1, j = len(b)-1, plus carry = 0.
  2. Loop while i ≥ 0 or j ≥ 0 or carry > 0.
  3. Sum equals carry plus the bit from a if i ≥ 0 plus the bit from b if j ≥ 0.
  4. Append sum % 2 to the result; set carry = sum // 2; decrement both indices.
  5. Reverse the accumulated string and return.

Why This Works

Each column produces exactly one output bit and at most one carry bit, so a single linear pass over max(len(a), len(b)) + 1 positions suffices. The final carry-only iteration handles overflow into a new high bit.

⚠ Most Dangerous Mistake

Forgetting to continue the loop while carry > 0 even after both strings are exhausted. A trailing carry must become a leading '1' — "1" + "1" needs to produce "10", not "0".

  • Building the string left-to-right and forgetting to reverse at the end.
  • Concatenating strings inside the loop in Go (O(n^2)); use a byte slice or strings.Builder.
  • Converting characters to digits as int(ch) instead of int(ch) - int('0').

#7 · Reverse Integer

MediumMathOverflow

The Problem

Given a 32-bit signed integer x, return x with its digits reversed. If reversing causes the value to fall outside the signed 32-bit range [-2^31, 2^31 - 1], return 0. You must solve this assuming the environment does not allow storing 64-bit integers (so overflow must be detected before it happens).

Brute Force

Convert the integer to a string, reverse the digits, parse back, and reapply the sign. The "no 64-bit" rule means you must explicitly check the result against INT_MAX / INT_MIN before returning. This works fine in Python because ints are unbounded, but it sidesteps the spirit of the problem.

Python — Brute Force 
def reverse(x):
    INT_MAX = 2**31 - 1
    INT_MIN = -2**31
    sign = -1 if x < 0 else 1
    # Strip sign, reverse digits, reapply sign
    rev = sign * int(str(abs(x))[::-1])
    if rev < INT_MIN or rev > INT_MAX:
        return 0
    return rev
Go — Brute Force 
func reverse(x int) int {
    sign := 1
    if x < 0 {
        sign = -1
        x = -x
    }
    s := strconv.Itoa(x)
    // Reverse the digit bytes
    b := []byte(s)
    for i, j := 0, len(b)-1; i < j; i, j = i+1, j-1 {
        b[i], b[j] = b[j], b[i]
    }
    n, _ := strconv.Atoi(string(b))
    n *= sign
    if n < math.MinInt32 || n > math.MaxInt32 {
        return 0
    }
    return n
}

Why optimize? The brute force builds a temporary that might already overflow a 32-bit register. The optimal uses pure arithmetic and checks overflow before each multiplication.

Key Insight

Build the reversed integer one digit at a time with rev = rev * 10 + digit. Before each multiply, ask: would this push us past INT_MAX/INT_MIN? If rev > INT_MAX/10, the next *10 overflows. Symmetric check for negatives. No 64-bit storage required.

How to Think About It

  1. Pull digits off x with x % 10; integer-divide x by 10. Beware language differences in modulo on negatives.
  2. Maintain a running rev initialized to 0.
  3. Before rev = rev*10 + d, check: rev > INT_MAX / 10 ⇒ overflow; rev < INT_MIN / 10 ⇒ underflow.
  4. Also handle the boundary case where rev == INT_MAX/10 but d > 7 (or d > 8 for the negative side).
  5. Loop until x == 0; return rev.

Why This Works

32-bit INT_MAX = 2147483647, last digit 7. So the only way rev*10 + d can overflow when rev == INT_MAX/10 == 214748364 is if d > 7. Any larger rev overflows regardless of d. Checking before the multiply guarantees safety without ever touching 64-bit values.

⚠ Most Dangerous Mistake

Performing the multiply and then checking. By that point, in a strict 32-bit environment, you've already overflowed and your check uses garbage. Always check the pre-condition, never the post-condition.

  • In Python, -7 % 10 == 3, not -7. Strip the sign first or use int(math.fmod(x,10)).
  • Forgetting to handle x = -2147483648 — abs() in 32-bit C would overflow.
  • Returning the reversed string instead of an int (off-by-one classic).
  • Treating leading zeros in the reversed form (e.g., 120 → 21) as a bug; they should silently disappear via integer arithmetic.

#92 · Reverse Linked List II

MediumLinked ListIn-Place Reversal

The Problem

Given the head of a singly linked list and two integers left and right where left ≤ right, reverse the nodes from position left to position right (1-indexed) and return the modified list. Positions are inclusive; for example, reversing positions 2-4 of 1->2->3->4->5 gives 1->4->3->2->5. You should do it in one pass.

Brute Force

Walk the list collecting node values into an array, reverse the slice from left-1 to right-1 in the array, then walk the list again writing values back. This is O(n) time but uses O(n) extra space and ignores the actual pointer structure — cheating by treating the list as an array.

Python — Brute Force 
def reverseBetween(head, left, right):
    vals, cur = [], head
    while cur:
        vals.append(cur.val)
        cur = cur.next
    # Reverse the slice [left-1, right-1] inclusive
    vals[left-1:right] = vals[left-1:right][::-1]
    cur = head
    for v in vals:
        cur.val = v
        cur = cur.next
    return head
Go — Brute Force 
func reverseBetween(head *ListNode, left int, right int) *ListNode {
    vals := []int{}
    for cur := head; cur != nil; cur = cur.Next {
        vals = append(vals, cur.Val)
    }
    // Reverse vals[left-1 : right]
    for i, j := left-1, right-1; i < j; i, j = i+1, j-1 {
        vals[i], vals[j] = vals[j], vals[i]
    }
    cur := head
    for _, v := range vals {
        cur.Val = v
        cur = cur.Next
    }
    return head
}

Why optimize? The optimal touches only the affected segment with O(1) extra space and rewires pointers, which is the actual skill being tested.

Key Insight

Use a dummy node before the head so that "position 0" has a real predecessor. Advance to the node just before left — call it pre. Then repeatedly take the node after the current one and splice it to the front of the reversing segment. After right - left splices, the segment is reversed in place.

How to Think About It

  1. Create dummy.next = head; advance a pointer pre by left-1 steps so it sits just before the segment.
  2. Let cur = pre.next — this becomes the tail of the reversed segment.
  3. Repeat right - left times: take nxt = cur.next, splice nxt to be pre.next, push cur down the chain.
  4. Concretely: cur.next = nxt.next; nxt.next = pre.next; pre.next = nxt.
  5. Return dummy.next.

Why This Works

Each iteration removes the node immediately after cur and inserts it directly after pre, which shifts a node one slot closer to the front of the segment. After k such operations, the (k+1)th node of the original segment now leads, achieving full reversal.

⚠ Most Dangerous Mistake

Updating cur to nxt at the end of each loop iteration. cur must not move — it stays as the original first node of the segment, which slides further back as nodes are spliced in front of it. Moving cur creates an infinite loop or a broken list.

  • Forgetting the dummy node when left == 1, then losing the head.
  • Off-by-one when counting steps to reach pre (need left-1, not left).
  • Forgetting to terminate the new tail; the original cur must still point to whatever followed the segment.

#138 · Copy List with Random Pointer

MediumLinked ListHash Map

The Problem

Given a linked list where each node has a next pointer and an additional random pointer that may point to any node in the list (or None), return a deep copy. Each random in the cloned list must point to the cloned version of the original target, not the original. The list has up to 1000 nodes.

Brute Force

Two passes with a hash map. First pass: for each original node, create a cloned node and store the mapping orig -> clone. Second pass: for each original node, set clone.next = map[orig.next] and clone.random = map[orig.random] (defaulting to None when the target is None).

Python — Brute Force 
def copyRandomList(head):
    if head is None:
        return None
    mapping = {}
    # Pass 1: create all clones
    cur = head
    while cur:
        mapping[cur] = Node(cur.val)
        cur = cur.next
    # Pass 2: wire next and random
    cur = head
    while cur:
        mapping[cur].next   = mapping.get(cur.next)
        mapping[cur].random = mapping.get(cur.random)
        cur = cur.next
    return mapping[head]

type Node struct {
    Val    int
    Next   *Node
    Random *Node
}

func copyRandomList(head *Node) *Node {
    if head == nil {
        return nil
    }
    m := map[*Node]*Node{}
    for cur := head; cur != nil; cur = cur.Next {
        m[cur] = &Node{Val: cur.Val}
    }
    for cur := head; cur != nil; cur = cur.Next {
        m[cur].Next   = m[cur.Next]    // map[nil] is the zero value: nil
        m[cur].Random = m[cur.Random]
    }
    return m[head]
}

Why optimize? The hash map costs O(n) extra space. The optimal interleave-clone trick achieves O(1) extra space by encoding the mapping in the list structure itself.

Key Insight

If you weave each clone immediately after its original (A -> A' -> B -> B' -> ...), then for any original node X, the clone is always X.next, and the clone of X.random is X.random.next. This makes the mapping implicit, eliminating the hash map.

How to Think About It

  1. Pass 1: walk the list and insert a clone right after each original node, so the list looks like A -> A' -> B -> B'.
  2. Pass 2: walk the original nodes; for each one, set cur.next.random = cur.random.next if cur.random else None.
  3. Pass 3: detach the two interleaved lists by restoring original next pointers and chaining the clones to one another.
  4. Return the cloned head, which is head.next from the original layout.
  5. Be careful: each pass uses only constant extra pointers, no auxiliary structure.

Why This Works

Interleaving uses the list itself as the mapping table. Because each original's clone sits directly after it, both next and random relationships in the cloned subset are exactly one hop further than in the original subset. Restoring the original list at the end leaves no observable side effect.

⚠ Most Dangerous Mistake

Mutating the original list and forgetting to restore it. The contract says return a deep copy — the input must remain untouched. The final pass that detaches the two lists is mandatory; skip it and you've corrupted the caller's data.

  • Handling None randoms: cur.random.next crashes if cur.random is None.
  • In Go, map[*Node]*Node returns the zero value (nil) on miss — this is actually convenient.
  • For the hash-map version, using node values as keys instead of node identities — values can repeat.

#25 · Reverse Nodes in k-Group

HardLinked ListIn-Place Reversal

The Problem

Given the head of a linked list, reverse the nodes k at a time and return the modified list. If the number of remaining nodes is less than k, leave the remaining nodes untouched. You may not alter the values of the nodes — only the node connections themselves may change. For example, with k = 3, 1->2->3->4->5 becomes 3->2->1->4->5.

Brute Force

Recursively: count the next k nodes; if there are fewer than k, return head unchanged. Otherwise reverse the first k nodes with a standard reverse routine and recurse on the rest, attaching the result as the new tail of the reversed group. Clean to write, but uses O(n/k) stack frames.

Python — Brute Force 
def reverseKGroup(head, k):
    # Check we have k nodes left
    cur, count = head, 0
    while cur and count < k:
        cur, count = cur.next, count + 1
    if count < k:
        return head
    # Reverse first k nodes
    prev, cur = None, head
    for _ in range(k):
        nxt = cur.next
        cur.next = prev
        prev, cur = cur, nxt
    # Recurse on the rest; head is now the tail of the reversed group
    head.next = reverseKGroup(cur, k)
    return prev
Go — Brute Force 
func reverseKGroup(head *ListNode, k int) *ListNode {
    cur, count := head, 0
    for cur != nil && count < k {
        cur = cur.Next
        count++
    }
    if count < k {
        return head
    }
    var prev *ListNode
    cur = head
    for i := 0; i < k; i++ {
        nxt := cur.Next
        cur.Next = prev
        prev, cur = cur, nxt
    }
    head.Next = reverseKGroup(cur, k)
    return prev
}

Why optimize? Recursion uses O(n/k) stack space; for very long lists that risks a stack overflow. The optimal is iterative with a dummy node, group counter, and explicit splicing — O(1) extra space.

Key Insight

Treat each k-group as a self-contained segment with a "group predecessor" pointer. For each group: (1) verify k nodes exist ahead, (2) reverse the group using the same splice-to-front trick from #92, (3) advance the group predecessor to what is now the tail of the reversed group. Repeat until fewer than k remain.

How to Think About It

  1. Set up dummy.next = head, with groupPrev = dummy.
  2. Find kth = groupPrev advanced by k steps; if it reaches None, stop — not enough nodes.
  3. Mark groupNext = kth.next so you know where the group ends and the rest begins.
  4. Reverse the k-segment in place (standard prev/cur/nxt loop), connecting the new tail to groupNext and groupPrev.next to the new head (originally kth).
  5. Move groupPrev forward to the old head of the segment (now the new tail) and loop.

Why This Works

Each group is reversed independently, with groupPrev serving as the stable left anchor and groupNext as the stable right anchor. Because we verify k nodes before reversing, partial groups at the tail are left in original order, satisfying the problem's contract.

⚠ Most Dangerous Mistake

Reversing first and then discovering the group is short. Once you've started swapping pointers, undoing it is painful. Always count the next k nodes first; only when you've confirmed they exist do you start reversing.

  • Forgetting to set groupPrev to the new tail (which is the original head of the segment) before the next iteration.
  • Losing the connection to groupNext — the reversed segment must stitch back to it.
  • Using k+1 hops instead of k when finding kth.
  • Treating k == 1 as a special case — it isn't; the algorithm correctly no-ops each group.
