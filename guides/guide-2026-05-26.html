#2 · Add Two Numbers

Medium Linked List Math 3 days since reviewed

The Problem

Two non-empty linked lists represent two non-negative integers stored in reverse order, one digit per node. Add the two numbers and return the sum as a linked list, also in reverse order.

Quick Reminder

Walk both lists in lockstep with a running carry. Use a dummy head node so you don't have to special-case the first digit, and remember the loop continues while either list has nodes or the carry is non-zero.

Key Insight

Reverse-order storage means the least significant digit is at the head, which is exactly the order you'd add by hand on paper — right to left. The carry from each column feeds the next, and a final carry can grow the result by one digit.

# Definition for singly-linked list.
class ListNode(object):
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next
class Solution(object):
    def addTwoNumbers(self, l1, l2):
        """
        :type l1: Optional[ListNode]
        :type l2: Optional[ListNode]
        :rtype: Optional[ListNode]
        """
        head = ListNode(0)
        current = head
        carry = 0

        while l1 or l2 or carry != 0:
            x = l1.val if l1 else 0
            y = l2.val if l2 else 0
            sum = x + y + carry
            carry = sum // 10
            newNode = ListNode(sum % 10)
            current.next = newNode
            current = newNode
            l1 = l1.next if l1 else None
            l2 = l2.next if l2 else None

        return head.next

#56 · Merge Intervals

Medium Intervals Sorting 3 days since reviewed

The Problem

Given an array of intervals where intervals[i] = [start, end], merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

Quick Reminder

Sort by start. Sweep through — either the current interval starts after the last merged interval ended (append) or it overlaps (extend the end with max).

Key Insight

Once sorted by start time, the only interval that can possibly overlap the current one is the most recent merged interval. You never need to look further back — this is what reduces an O(n²) pairwise check to O(n log n) dominated by the sort.

class Solution(object):
    def merge(self, intervals):
        """
        :type intervals: List[List[int]]
        :rtype: List[List[int]]
        """

        intervals.sort(key=lambda x: x[0])

        merged = []
        for interval in intervals:
            if not merged or merged[-1][1] < interval[0]:
                merged.append(interval)
            else:
                merged[-1][1] = max(merged[-1][1], interval[1])

        return merged

#198 · House Robber

Medium DP 3 days since reviewed

The Problem

You're a robber on a street. Each house has some money. You cannot rob two adjacent houses (alarm goes off). Return the maximum amount of money you can rob.

Quick Reminder

Classic 1D DP: at house i, either skip it (carry dp[i-1]) or rob it (nums[i] + dp[i-2]). Take the max. You can solve top-down with memoization or bottom-up with a table — both shown below.

Key Insight

The "no two adjacent" constraint means the choice at each house only depends on the previous two states. This makes the recurrence dp[i] = max(dp[i-1], nums[i] + dp[i-2]) — and means you can drop the array entirely and roll two variables for O(1) space.

class Solution:
    def rob(self, nums):
        """
        :type nums: List[int]
        :rtype: int
        """
        # maximum: func
        # max for current house: state
        # 1) dp(i) = dp(i-1)
        # 2) dp(i) = nums[i] + dp(i-2)
        # dp(i) = max(dp(i-1), nums[i]+dp(i-2))
        # 0 - num[0]
        # 1 - max(num[0], num[1])

        memo = {}
        def dp(i):
            if i == 0:
                return nums[0]
            if i == 1:
                return max(nums[0], nums[1])

            if i not in memo:
                memo[i] = max(dp(i-1), nums[i] + dp(i-2))

            return memo[i]

        return dp(len(nums) -1)

The Problem

Given an m x n 2D grid of '1's (land) and '0's (water), count the number of islands. An island is surrounded by water and formed by connecting adjacent lands horizontally or vertically.

Quick Reminder

Scan every cell. When you hit a '1' that hasn't been visited, flood-fill (BFS or DFS) from there to mark the entire island, and increment the counter once.

Key Insight

Each island is found exactly once — from whichever land cell the outer scan reaches first. The flood-fill swallows the rest of that island so it's never counted again. Two flavors: keep a separate visited set, or mutate the grid in place by setting visited cells to '0' (saves space, destroys input).

#295 · Find Median from Data Stream

Hard Heap Design 3 days since reviewed

The Problem

Design a data structure that supports addNum(int) and findMedian() — the running median of all numbers seen so far. Both operations should be fast.

Quick Reminder

Two heaps: a max-heap (small) for the lower half, a min-heap (large) for the upper half. Keep their sizes within one, and keep small's max ≤ large's min. Median is the top of the bigger heap, or the average of both tops if sizes are equal.

Key Insight

You don't need a sorted list — you just need fast access to the middle. Two heaps split the stream into "lower half" and "upper half" so the median is always at a heap top. addNum is O(log n), findMedian is O(1). Python's heapq is min-only, so negate values to fake a max-heap.

class MedianFinder(object):

    def __init__(self):
        self.small, self.large = [], []

    def addNum(self, num):
        """
        :type num: int
        :rtype: None
        """
        heapq.heappush(self.small, -num)

        if self.small and self.large and -self.small[0] > self.large[0]:
            a = - heapq.heappop(self.small)
            heapq.heappush(self.large, a)
        if len(self.small) > len(self.large) + 1:
            a = - heapq.heappop(self.small)
            heapq.heappush(self.large, a)
        if len(self.small) + 1 < len(self.large):
            a = - heapq.heappop(self.large)
            heapq.heappush(self.small, a)

    def findMedian(self):
        """
        :rtype: float
        """
        if len(self.small) > len(self.large):
            return -self.small[0]
        if len(self.small) < len(self.large):
            return self.large[0]

        return (-self.small[0] + self.large[0]) / float(2)

#322 · Coin Change

Medium DP 4 days since reviewed (overdue)

The Problem

Given coins of distinct denominations and a target amount, return the fewest number of coins to make up that amount. Return -1 if impossible. You have an unlimited supply of each coin.

Quick Reminder

Unbounded knapsack DP. dp[x] = fewest coins to make amount x. For each coin, for each amount from coin upward: dp[x] = min(dp[x], dp[x - coin] + 1). Initialize dp[0]=0 and dp[1..amount]=∞.

Key Insight

Because each coin has unlimited supply, you iterate the inner amount loop forward (low to high), letting dp[x - coin] already include the same coin used multiple times. This is the unbounded-knapsack pattern — the dual of the 0/1 knapsack which iterates backward.

class Solution(object):
    def coinChange(self, coins, amount):
        """
        :type coins: List[int]
        :type amount: int
        :rtype: int
        """
        dp = [float('inf')] * (amount + 1)
        dp[0] = 0

        for coin in coins:
            for x in range(coin, amount + 1):
                dp[x] = min(dp[x], dp[x - coin] + 1)

        if dp[amount] == float('inf'):
            return -1

        return dp[amount]

#66 · Plus One

EasyMathArray

The Problem

You are given a large integer represented as an integer array digits, where each digits[i] is the i-th digit (most significant first). Increment the large integer by one and return the resulting array of digits. The number has no leading zeros (except the number 0 itself).

Brute Force

Convert digits to an integer, add 1, then split it back into digits. This "works" but defeats the point of the problem — the input is given as an array specifically because the integer might exceed what a native integer type can hold (Java's int, Go's int64). It also costs an O(n) parse plus an O(n) stringify just to do an O(n) operation.

Python — Brute Force 
class Solution:
    def plusOne(self, digits):
        n = int("".join(str(d) for d in digits)) + 1
        return [int(c) for c in str(n)]
Go — Brute Force 
import (
    "strconv"
    "strings"
)

func plusOne(digits []int) []int {
    var sb strings.Builder
    for _, d := range digits {
        sb.WriteString(strconv.Itoa(d))
    }
    n, _ := strconv.ParseInt(sb.String(), 10, 64) // FAILS for very long inputs
    n++
    s := strconv.FormatInt(n, 10)
    out := make([]int, len(s))
    for i, c := range s {
        out[i] = int(c - '0')
    }
    return out
}

Why optimize? The brute force breaks for inputs longer than ~18 digits in Go — the whole point of the array representation is to support arbitrarily large numbers.

Key Insight

Adding 1 propagates a carry from right to left, but the carry stops at the first non-9 digit. Walk from the end: if it's 9, set to 0 and continue; otherwise increment and return. If you fall off the left edge, every digit was 9 — prepend a 1.

How to Think About It

  1. Iterate i from len(digits) - 1 down to 0.
  2. If digits[i] < 9, increment it by 1 and return digits immediately — the carry stopped here.
  3. Otherwise (it's a 9), set digits[i] = 0 and continue the loop — carry propagates left.
  4. If the loop completes without returning, every digit was 9 (e.g. [9,9,9] -> [1,0,0,0]).
  5. Return a new array: a leading 1 followed by all zeros (length n+1).

Why This Works

The loop terminates the carry at the first non-9 position because incrementing any digit less than 9 gives a value less than 10 — no carry to propagate. The all-9s case is the only one that requires a length change, and we handle it by allocating a fresh array of length n+1.

⚠ Most Dangerous Mistake

Forgetting the all-9s case and returning the mutated original array of length n — you'd get [0,0,0] for input [9,9,9] instead of [1,0,0,0]. Always check after the loop whether you ever returned early.

  • Iterating left-to-right instead of right-to-left — you can't know whether a digit needs to change until you know if a carry reaches it.
  • Allocating a new array unnecessarily on every call — you can mutate in place for the common case and only allocate on the all-9s path.
  • In Go, forgetting that append([]int{1}, digits...) creates a new slice but the original digits may now be all zeros after your loop — concatenate the right values.

#69 · Sqrt(x)

EasyMathBinary Search

The Problem

Given a non-negative integer x, return the square root of x rounded down to the nearest integer. You must not use any built-in exponent or sqrt function (e.g. pow(x, 0.5) or x ** 0.5).

Brute Force

Linear scan: try every integer i from 0 upward; return i - 1 as soon as i * i > x. This is O(√x) time. For x up to 2^31 − 1 that's ~46,000 iterations — fast enough to pass, but binary search makes it O(log x), which is about 31 iterations.

Python — Brute Force 
class Solution:
    def mySqrt(self, x):
        i = 0
        while i * i <= x:
            i += 1
        return i - 1
Go — Brute Force 
func mySqrt(x int) int {
    i := 0
    for i*i <= x {
        i++
    }
    return i - 1
}

Why optimize? O(√x) is slow for large inputs; binary search drops it to O(log x), and the technique transfers to any "find largest k such that f(k) holds" problem.

Key Insight

You're looking for the largest integer m such that m * m ≤ x. The predicate "m*m ≤ x" is monotonic in m — once it flips from true to false, it stays false. That's the textbook signal for binary search.

How to Think About It

  1. Handle the edge case x < 2 directly (returns x itself).
  2. Set lo = 1, hi = x / 2. (For x ≥ 2, √x is always at most x/2.)
  3. Loop while lo ≤ hi. Compute mid = lo + (hi - lo) / 2 (avoids overflow in Java/Go for very large bounds).
  4. If mid * mid == x, return mid. If mid * mid < x, record mid as the current best and search right (lo = mid + 1). Otherwise search left (hi = mid - 1).
  5. Return the last recorded best (the largest mid where mid*mid ≤ x).

Why This Works

Binary search on a monotonic predicate converges in O(log range). The "last value where predicate held" pattern records the candidate whenever the predicate is true and searches the higher half. Termination is guaranteed because lo always advances past mid+1 or hi retreats below mid−1.

⚠ Most Dangerous Mistake

Returning mid when mid * mid > x — you must return the last mid for which the square was still ≤ x. Either keep an ans variable or, equivalently, return hi after the loop (in the "lo ≤ hi" convention, hi ends up pointing at the floor).

  • Using (lo + hi) / 2 instead of lo + (hi - lo) / 2 — the former overflows for large lo+hi in fixed-width languages (Go's int is 64-bit so it's safe in practice, but the habit matters).
  • Off-by-one on the bounds: setting hi = x still works but wastes iterations; setting hi = x / 2 only works for x ≥ 2, hence the early return.
  • Forgetting the integer-overflow concern with mid * mid in languages with bounded ints — in Python it's fine, in Go for 32-bit ints you'd need int64.

#50 · Pow(x, n)

MediumMathRecursion

The Problem

Implement pow(x, n), which calculates x raised to the power n (i.e., x^n). x is a double, n is an integer (can be negative). Don't use the language's built-in pow.

Brute Force

Multiply x by itself n times in a loop. That's O(n) multiplications. For n up to 2^31 that's 2 billion iterations — far too slow. We need to exploit the structure of exponentiation: x^n = (x^(n/2))^2 for even n, and x^n = x · x^(n-1) for odd n. That recursion halves n each step, giving O(log n).

Python — Brute Force 
class Solution:
    def myPow(self, x, n):
        if n < 0:
            x = 1 / x
            n = -n
        result = 1.0
        for _ in range(n):
            result *= x
        return result
Go — Brute Force 
func myPow(x float64, n int) float64 {
    if n < 0 {
        x = 1 / x
        n = -n
    }
    result := 1.0
    for i := 0; i < n; i++ {
        result *= x
    }
    return result
}

Why optimize? n can be as large as 2^31 − 1 — the O(n) loop will time out. Fast exponentiation makes this 31 multiplications instead of 2 billion.

Key Insight

x^n splits naturally on the parity of n: if n is even, x^n = (x^(n/2))^2 — one squaring instead of n/2 multiplications. If n is odd, peel off one factor and recurse on n-1. This halves n every two steps in the worst case, yielding O(log n). The iterative form (binary exponentiation) reads the bits of n and squares-or-multiplies accordingly.

How to Think About It

  1. Handle the sign of n first: if n < 0, replace x with 1/x and n with -n (careful: for INT_MIN, -n overflows in 32-bit; use a 64-bit type or Python's arbitrary precision).
  2. Iterative version: initialize result = 1.0. Loop while n > 0.
  3. If the lowest bit of n is 1 (i.e. n is odd), multiply result *= x. This is the "binary expansion" trick — n = sum of powers of 2, and we multiply in x^(2^k) whenever bit k is set.
  4. Square x (x *= x) and shift n right by 1 (n //= 2). Each iteration handles one bit of n.
  5. Return result. The loop runs log₂(n) times.

Why This Works

Any positive integer n can be written in binary: n = b_k · 2^k + … + b_1 · 2 + b_0. So x^n = x^(b_k · 2^k) · … · x^(b_0). As we square x at each step we have x, x^2, x^4, x^8, … available. We multiply each one into the running result exactly when the corresponding bit of n is set. This is the same idea as multiplying by repeated doubling, just for exponents.

⚠ Most Dangerous Mistake

Negating n when n == INT_MIN (Go: math.MinInt32). -INT_MIN overflows in a fixed-width signed type because the positive range is one smaller than the negative range. Promote to a 64-bit type before negating, or in Python don't worry — arbitrary precision protects you.

  • Forgetting the base cases: x^0 = 1 for any x (including x=0 by convention in this problem), and x^1 = x. The iterative loop handles these naturally because the loop body doesn't run when n=0.
  • Mixing up integer and floating-point: the result is a double, but n is an int. Use int division (// or / in Go) on n, but float multiplication on x.
  • Recursing without memoizing the sub-result: return myPow(x, n/2) * myPow(x, n/2) is exponential because you recompute. Store it: half = myPow(x, n/2); return half * half.

#16 · 3Sum Closest

MediumTwo PointersSorting

The Problem

Given an integer array nums of length n and an integer target, find three integers in nums whose sum is closest to target. Return the sum of those three integers. You may assume that each input would have exactly one solution.

Brute Force

Try every triple (i, j, k) with i < j < k. Compute the sum, compare its distance from target to the best seen so far, and track the minimum. Three nested loops give O(n3) time which is too slow for n up to 1000 but fine for tiny inputs.

Python — Brute Force 
def threeSumClosest(nums, target):
    n = len(nums)
    best = nums[0] + nums[1] + nums[2]
    for i in range(n):
        for j in range(i + 1, n):
            for k in range(j + 1, n):
                s = nums[i] + nums[j] + nums[k]
                if abs(s - target) < abs(best - target):
                    best = s
    return best

Why optimize? Cubic time blows up at n = 1000 (1 billion ops); sorting plus two pointers brings it to O(n2). The sort cost O(n log n) is dwarfed by the n2 two-pointer phase, so it's free in the asymptotic sense.

Key Insight

Sort the array. Fix the first index i, then use two pointers (lo = i+1, hi = n-1) on the remaining slice. If the current triple sum is less than target, increment lo to grow it; if greater, decrement hi to shrink it. Track the best distance throughout. The sort enables this monotonic adjustment, just like #15 3Sum.

How to Think About It

  1. Sort nums ascending so we can move pointers monotonically. Sorting is O(n log n) but enables the O(n2) total.
  2. Initialize best = nums[0] + nums[1] + nums[2] (any valid triple). Avoid sentinel infinities — a real sum is safer.
  3. For each i from 0 to n-3, set lo = i+1, hi = n-1. The outer index fixes one element; two pointers handle the rest.
  4. While lo < hi: compute s = nums[i] + nums[lo] + nums[hi]; if |s - target| < |best - target| update best.
  5. Then if s < target do lo += 1 (need a bigger sum), else if s > target do hi -= 1 (need a smaller sum), else return s immediately (exact match means distance 0).
  6. Return best after the outer loop completes.

Why This Works

Once sorted, for a fixed i, every pair (lo, hi) covers a unique sum range. Moving lo right strictly increases the sum; moving hi left strictly decreases it. So whichever side of target we're on, the correct pointer move can only improve (or tie) the distance — we never miss the optimum because we touch every relevant boundary.

Worked Example

nums = [-1, 2, 1, -4], target = 1. Sorted: [-4, -1, 1, 2]. i = 0 (-4): lo = 1, hi = 3, s = -4 + (-1) + 2 = -3, dist 4. Move lo to 2: s = -4 + 1 + 2 = -1, dist 2. Best so far -1. Move lo to 3: stop. i = 1 (-1): lo = 2, hi = 3, s = -1 + 1 + 2 = 2, dist 1. Best now 2. Stop. Result: 2.

⚠ Most Dangerous Mistake

Returning the distance |best - target| instead of best itself. Read the prompt carefully — the problem asks for the sum of the three numbers, not how close they got.

  • Initializing best to 0 or math.inf — if all sums are large positives 0 may look closer than any real triple; use an actual triple sum.
  • Forgetting to handle the equality case s == target — you can early-return, but if you don't, ensure your loop still terminates (avoid infinite loop by always moving a pointer).
  • Integer overflow in other languages (not Python) — the sum of three 32-bit ints can overflow; use 64-bit accumulators in Go/Java if constraints demand it.
  • Re-comparing pointer values like in 3Sum (skip duplicates) — here you do NOT need duplicate skipping because we want the closest sum, even if duplicates produce it. Adding that logic can skip the actual optimum.
  • Sorting then forgetting that the returned sum is from the sorted array's indices — that's fine here because the problem only wants the sum, but be aware if the variant asks for the indices.

#75 · Sort Colors

MediumTwo PointersDutch National Flag

The Problem

Given an array nums with values 0, 1, and 2 representing red, white, and blue, sort them in place so that all 0s come first, then 1s, then 2s. You must solve this without using a library's sort, and the follow-up asks for a single-pass O(1) extra space solution.

Brute Force

Either call nums.sort() (O(n log n)) or do a two-pass counting sort: count how many of each value, then overwrite the array with that many 0s, 1s, and 2s. The counting sort is O(n) time and O(1) extra space but requires two passes, violating the single-pass follow-up.

Python — Brute Force

Why optimize? The two-pass solution is fine performance-wise, but interviewers expect the elegant single-pass Dutch National Flag using three pointers. The follow-up explicitly forbids using extra storage proportional to the alphabet.

Key Insight

Maintain three regions with pointers lo, mid, and hi: everything left of lo is 0, everything between lo and mid is 1, and everything right of hi is 2. The unknown region is [mid, hi]. Process one element at a time, shrinking the unknown until it's empty.

How to Think About It

  1. Initialize lo = 0, mid = 0, hi = n - 1. The invariant is: nums[0..lo-1] are all 0, nums[lo..mid-1] are all 1, nums[hi+1..n-1] are all 2, and nums[mid..hi] is the unknown region.
  2. While mid <= hi, inspect nums[mid]. The loop terminates when the unknown region is empty.
  3. If it's 0: swap nums[lo] and nums[mid], then lo += 1; mid += 1. The swap pushes a 0 into the 0-region and pushes whatever was at lo (a 1, by invariant) into the 1-region.
  4. If it's 1: just mid += 1 (already in the right region) — the 1 extends the 1-region naturally.
  5. If it's 2: swap nums[mid] and nums[hi], then hi -= 1 (do NOT advance mid — the swapped-in value is still unknown and must be inspected next iteration).

Why This Works

The invariant "left of lo is all 0, between lo and mid is all 1, right of hi is all 2" is preserved by each case. When mid passes hi, the unknown region is empty, so the entire array satisfies the invariant, which means it's sorted. One linear scan suffices because mid moves forward in two of the three cases and the gap (hi - mid) shrinks in the third.

Worked Example

nums = [2, 0, 2, 1, 1, 0], lo = mid = 0, hi = 5. mid=0 sees 2 → swap(0,5) gives [0,0,2,1,1,2], hi=4. mid=0 sees 0 → swap(0,0), lo=mid=1. mid=1 sees 0 → swap(1,1), lo=mid=2. mid=2 sees 2 → swap(2,4) gives [0,0,1,1,2,2], hi=3. mid=2 sees 1 → mid=3. mid=3 sees 1 → mid=4. Now mid > hi, done. Result [0,0,1,1,2,2].

⚠ Most Dangerous Mistake

Advancing mid after swapping with hi. The element you just swapped in from position hi is unknown — it could be a 0, 1, or 2 — so you must re-examine it on the next iteration. Only advance mid in the 0 and 1 cases.

  • Using mid < hi instead of mid <= hi — you'd miss the last element when the unknown region collapses to a single index.
  • Forgetting to advance lo after swapping a 0 forward — the 0 region must grow.
  • Treating this like a partition with a single pivot — three values need three regions, not two.
  • Swapping then immediately re-reading nums[mid] as if it were unchanged — the swap mutated it; either store the value first or re-read after.
  • Using lo and mid as if they always equal each other — they start equal but diverge as soon as you encounter the first 1.

The Problem

Brute Force

Rotate by one position, k times. Each single-step rotation pops the last element and prepends it (or shifts everything right by one). This is O(n*k) time with O(1) extra space — correct but painfully slow when k is large. Alternatively, allocate a new array of size n and place each element at (i + k) % n, which is O(n) time but O(n) extra space.

Why optimize? O(n*k) is too slow when k approaches n (1010 ops); the reverse-three-times trick is O(n) time with O(1) space. The new-array approach is O(n) time but uses O(n) extra space, which violates the follow-up.

Key Insight

Rotating right by k is equivalent to reversing the whole array, then reversing the first k elements, then reversing the remaining n-k elements. Three reversals, total O(n) work, no extra array. This works because reversal is its own inverse and composes cleanly across the split boundary.

How to Think About It

  1. Normalize k = k % n — rotating by n is a no-op, so we only care about k mod n. Skip out early if n == 0 or k == 0.
  2. Reverse the entire array nums[0..n-1] — this puts the last k elements at the front, but in reversed order.
  3. Reverse the first k elements nums[0..k-1] — restores their original order.
  4. Reverse the rest nums[k..n-1] — restores their original order too.
  5. Use a helper reverse(lo, hi) that swaps from both ends inward with two pointers. Total work across all three reversals is 2n swaps — strictly O(n).

Why This Works

Let A = first n-k elements, B = last k elements. The goal is to transform [A, B] into [B, A]. Reversing the whole array gives [reverse(B), reverse(A)]. Reversing the first k elements restores B; reversing the last n-k elements restores A. Result: [B, A]. The reversals are linear and disjoint, so total cost is O(n).

Worked Example

nums = [1, 2, 3, 4, 5, 6, 7], k = 3. Step 1 reverse all: [7, 6, 5, 4, 3, 2, 1]. Step 2 reverse first 3: [5, 6, 7, 4, 3, 2, 1]. Step 3 reverse last 4: [5, 6, 7, 1, 2, 3, 4]. That's the input rotated right by 3.

⚠ Most Dangerous Mistake

Forgetting k %= n. When k >= n the constraints allow k up to 105, and your reverse calls will use out-of-range indices or do redundant work. Always normalize first, and beware that n = 0 would cause a division-by-zero (guard with an early return).

  • Rotating left instead of right — the problem says right by k; reversing the whole array is correct for right rotation, but left rotation needs a slightly different split (reverse first n-k, then last k, then whole).
  • Using slice assignment in Python like nums = nums[-k:] + nums[:-k] — this rebinds the local name but does NOT mutate the caller's list. Use nums[:] = ... for in-place.
  • Off-by-one in the reverse helper — standard pattern is while lo < hi: swap; lo += 1; hi -= 1.
  • Calling reverse(nums, 0, k) with a half-open interpretation when your helper expects closed — pick one convention and document it.
  • Using the cyclic-replacement approach without tracking visited counts — when gcd(n, k) > 1 you get multiple disjoint cycles and need to start new ones from fresh indices.

#1046 · Last Stone Weight

EasyHeapSimulation

The Problem

You have an array of positive integers stones. On each turn, pick the two heaviest stones x and y with x <= y. If x == y, both are destroyed; otherwise the remaining stone has weight y - x. Repeat until one or zero stones remain. Return the weight of the last stone, or 0 if none.

Brute Force

On each iteration, sort the list (or find the two maxima with two linear scans), remove the top two, and push back the difference if nonzero. Repeating this n-1 times with an O(n log n) sort each pass gives O(n2 log n); a linear two-max scan trims it to O(n2). Both are fine for n ≤ 30 (the actual constraint) but waste work re-sorting.

Python — Brute Force 
def lastStoneWeight(stones):
    stones = list(stones)
    while len(stones) > 1:
        stones.sort()
        y = stones.pop()
        x = stones.pop()
        if x != y:
            stones.append(y - x)
    return stones[0] if stones else 0
Go — Brute Force

func lastStoneWeight(stones []int) int {
    for len(stones) > 1 {
        sort.Ints(stones)
        n := len(stones)
        y, x := stones[n-1], stones[n-2]
        stones = stones[:n-2]
        if x != y {
            stones = append(stones, y-x)
        }
    }
    if len(stones) == 0 { return 0 }
    return stones[0]

Why optimize? Re-sorting after each step is wasteful when only one element changes; a max-heap maintains the order incrementally in O(log n) per operation. The pattern generalizes — any "repeatedly extract max, modify, reinsert" workload should use a heap.

Key Insight

You only ever care about the two largest elements. A max-heap gives O(log n) extract-max and O(log n) push, so the total cost drops to O(n log n). Python's heapq is a min-heap, so push negated values to simulate a max-heap.

How to Think About It

  1. Build a max-heap from stones — in Python, negate each value and call heapify. In Go, implement heap.Interface with Less using >.
  2. While the heap has at least two elements: pop y (largest), pop x (second largest). Because the heap root is always the max, two consecutive pops give you the two heaviest stones.
  3. If y > x, push y - x back onto the heap. The remaining stone re-enters the priority queue and could be the new heaviest.
  4. If y == x, both are destroyed; push nothing. The heap loses two elements net.
  5. Return the remaining element (or 0 if empty). Remember to un-negate if you used a min-heap trick to simulate a max-heap.

Why This Works

The heap invariant guarantees the root is always the maximum, so each "pick two heaviest" step is two pops. Pushing the difference maintains the invariant. The total number of operations is at most 2(n-1) pops and (n-1) pushes, each O(log n), giving O(n log n) overall.

Worked Example

stones = [2, 7, 4, 1, 8, 1]. Max-heap: [8, 7, 4, 2, 1, 1]. Pop 8 and 7, push 1. Heap: [4, 2, 1, 1, 1]. Pop 4 and 2, push 2. Heap: [2, 1, 1, 1]. Pop 2 and 1, push 1. Heap: [1, 1, 1]. Pop 1 and 1, push nothing (equal). Heap: [1]. Return 1.

⚠ Most Dangerous Mistake

Forgetting that Python's heapq is a min-heap. If you push raw values and call heappop, you'll always get the smallest, which solves the wrong problem. Either negate on push and negate on pop, or use a separate max-heap library.

  • Pushing y - x when x == y — you'd push a 0, which contaminates the heap. Guard with if x != y.
  • Returning heap[0] without negating — if you stored negatives, the answer is -heap[0].
  • Handling the empty case — after the loop the heap may be empty (if the last two were equal). Return 0 in that case.
  • Using heapq.heappop twice without checking length — the loop guard must ensure at least two elements remain.
  • Building a min-heap and trying to "pop the largest" by scanning — defeats the purpose of using a heap; use a max-heap properly.

#973 · K Closest Points to Origin

MediumHeapTop-K

The Problem

Given an array points where points[i] = [xi, yi] and an integer k, return the k closest points to the origin (0, 0). Distance is Euclidean, but since we only compare distances we can use squared distance to avoid square roots. The answer may be returned in any order, and the answer is guaranteed unique except for ordering.

Brute Force

Compute the squared distance for every point, sort all points by distance, and return the first k. This is O(n log n) time with O(n) extra space — perfectly correct but does more work than necessary because we don't need the order of the discarded n-k points.

Python — Brute Force 
def kClosest(points, k):
    points.sort(key=lambda p: p[0] * p[0] + p[1] * p[1])
    return points[:k]
Go — Brute Force 
import "sort"

func kClosest(points [][]int, k int) [][]int {
    sort.Slice(points, func(i, j int) bool {
        di := points[i][0]*points[i][0] + points[i][1]*points[i][1]
        dj := points[j][0]*points[j][0] + points[j][1]*points[j][1]
        return di < dj
    })
    return points[:k]
}

Why optimize? Sorting all n points is O(n log n); maintaining a max-heap of size k drops it to O(n log k), which matters when k << n. Quickselect gets O(n) average. Interviewers often expect at least the heap variant since it generalizes to streaming inputs.

Key Insight

Maintain a max-heap of size at most k, keyed by squared distance. When the heap has fewer than k items, push. Once it's at capacity, push the new point only if it's closer than the heap's current max, then pop the max. After processing all points, the heap holds the k closest.

How to Think About It

  1. Define distance squared as d(p) = p[0]2 + p[1]2; never take a square root — comparing squares preserves order for non-negative quantities.
  2. Iterate through points. For each point, compute d.
  3. If the max-heap has fewer than k items, push (d, point) unconditionally — we're still filling the initial set.
  4. Otherwise compare d to the current max (heap root); if smaller, pop the max and push the new point. Use heappushpop in Python for the combined op.
  5. After the loop, extract all points from the heap and return them. Order doesn't matter per the problem statement.

Why This Works

At any moment the heap holds the k closest points seen so far. When a new point arrives, either it's farther than the current k-th closest (discard it) or it's closer (evict the current worst). The heap's root gives O(1) access to the eviction candidate and O(log k) updates. We touch each point exactly once, so total work is O(n log k).

Worked Example

points = [[1, 3], [-2, 2], [5, 8], [0, 1]], k = 2. Squared distances: 10, 8, 89, 1. Process in order — after [1,3]: heap = [(10, [1,3])]. After [-2,2]: heap = [(10, [1,3]), (8, [-2,2])]. After [5,8]: 89 > 10 so skip. After [0,1]: 1 < 10 so pop [1,3] and push [0,1]. Final heap: [(8, [-2,2]), (1, [0,1])]. Result: [[-2,2], [0,1]].

⚠ Most Dangerous Mistake

Using a min-heap of all n points and popping k times — that's still O((n + k) log n), not O(n log k). The space win comes from capping the heap at k. If k is small and n is huge, an unbounded heap wastes memory and time on points that will never be in the answer.

  • Taking the square root — pointless and slow; compare squared distances.
  • In Python, putting raw lists in the heap can crash if two distances tie because Python tries to compare the lists. Use a tuple (d, idx, point) with a unique index as tiebreaker.
  • Forgetting to negate for max-heap in Python — or in Go, forgetting to flip the Less comparator to >.
  • Sorting then slicing for a problem stated as "any order" — correct, but wastes work; the heap approach is asymptotically better when k << n.
  • Using a min-heap of size k incorrectly — with a min-heap, the root is the smallest of the kept k, which is the wrong eviction candidate; you need the largest of the kept k (max-heap).

#621 · Task Scheduler

MediumHeapGreedy

The Problem

Given a list of CPU tasks (uppercase letters) and a cooldown integer n, each unit of time the CPU can execute one task or idle. After running a task, the same task cannot run again for the next n intervals. Return the minimum number of intervals required to finish all tasks.

Brute Force

Simulate the scheduler turn by turn. Maintain a frequency count, and at each tick choose the available task with the highest remaining count (a task is "available" if its cooldown has expired). If no task is available, emit idle. Continue until all counts are zero. With careful queue/heap structures this is O(T log K) where T is total time and K is alphabet size, but a naive scan-the-frequencies brute is O(T * K).

Python — Brute Force 
from collections import Counter

def leastInterval(tasks, n):
    counts = Counter(tasks)
    cooldown = {}  # task -> next time it can run
    time = 0
    while sum(counts.values()) > 0:
        time += 1
        pick = None
        best = 0
        for task, c in counts.items():
            if c > 0 and cooldown.get(task, 0) <= time and c > best:
                pick, best = task, c
        if pick is not None:
            counts[pick] -= 1
            cooldown[pick] = time + n + 1
    return time

Why optimize? Simulation works but is verbose; a max-heap plus a queue elegantly models "run highest count, wait n turns, return it to the heap." Even better: a closed-form math formula computes the answer in O(T) without simulation. The formula is the elegant solution interviewers love.

Key Insight

Greedy choice: always run the task with the most remaining occurrences that's currently off cooldown. A max-heap holds the available tasks by count; a queue holds tasks that are cooling down with the time they can re-enter the heap. Alternatively, the answer equals max(len(tasks), (maxFreq - 1) * (n + 1) + tiesAtMax).

How to Think About It

  1. Count occurrences of each task with a frequency map. Find maxFreq (the highest count) and tiesAtMax (how many distinct tasks share that max count).
  2. Imagine arranging tasks into (maxFreq - 1) full "frames" of length n + 1, each starting with one of the most frequent tasks. The frames provide the cooldown spacing.
  3. The final frame holds tiesAtMax tasks (all the tasks tied at max frequency get placed last, one per tied task).
  4. Compute frame_total = (maxFreq - 1) * (n + 1) + tiesAtMax. This is the minimum if no extra idles are forced.
  5. If len(tasks) exceeds frame_total, no idles are needed and the answer is len(tasks). Otherwise the answer is frame_total. Return the max of the two: max(len(tasks), frame_total).

Why This Works

The most-frequent task dictates the schedule's skeleton: it must be spaced by at least n. Arrange maxFreq - 1 blocks each of width n + 1, then add one trailing block holding the tied tasks. Each leading block has the form [most_freq_task, *, *, ..., *] where the asterisks are filled by other tasks (or idle). If there are enough other tasks to fill the gaps, no idle slots are needed and the answer equals the total task count. Otherwise, the formula's idle count is exactly what's needed.

Worked Example

tasks = ["A","A","A","B","B","B"], n = 2. maxFreq = 3 (both A and B). ties = 2. Formula: (3 - 1) * (2 + 1) + 2 = 8. len(tasks) = 6. Answer = max(6, 8) = 8. A valid schedule: A B _ A B _ A B (8 slots, two idles). With more tasks, len(tasks) wins and no idles are needed.

⚠ Most Dangerous Mistake

Forgetting the max(len(tasks), formula) step. When you have many distinct tasks the formula can underestimate — for example, 26 unique tasks with n = 1 needs 26 time units, but the formula gives (1-1)*2 + 26 = 26 anyway. But with mixed frequencies and small n, the formula may compute fewer slots than the total task count; the answer can never be less than len(tasks).

  • Off-by-one in the formula — remember (maxFreq - 1) gaps, each of width n + 1, plus the trailing tied count. Not maxFreq * n.
  • Counting ties incorrectly — tiesAtMax is the number of distinct tasks whose count equals maxFreq, not the total count of those tasks.
  • Forgetting to push tasks back onto the heap after their cooldown — in the heap approach, the queue must release tasks whose ready-time has arrived.
  • Confusing n (the cooldown) with the array length — here n is the gap requirement, NOT the input size.
  • Modeling cooldown as "wait n+1 ticks" when it's actually "n other ticks between the same task" — the same task can repeat at time t and t+n+1, not t+n.
