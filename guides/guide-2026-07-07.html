The ProblemDesign a MovingAverage class initialized with a window size; each next(val) returns the average of the last size values seen (fewer if not enough have arrived yet).
Quick ReminderKeep a running sum and a fixed-size queue; add the incoming value and evict the oldest once the window overflows.
Key InsightMaintain a running sum alongside a bounded deque so each next() is O(1) — you never re-sum the window, you just add the new value and subtract the one you drop.
Try to recall: Can you implement this so each next() is O(1) rather than O(k)? What data structure gives O(1) eviction of the oldest element?
Language challenge: "You last solved this in Python. Try it in Go this time" — model the queue with a slice and track head/tail (or a ring buffer of length size); Go has no built-in deque, so a fixed-size circular array is the idiomatic move.

Explanation: only one value so far, average of [1].
Input: next(10)
Output: 5.5
Explanation: window [1, 10], (1+10)/2.

class Solution(object):
    def isHappy(self, n):
        """
        :type n: int
        :rtype: bool
        """
        visited = set()

        while n not in visited:
            visited.add(n)
            n = self.sumOfSquares(n)

            if n == 1:

        return False

Why optimize? Visiting every node is O(n); the completeness guarantee lets us skip whole perfect subtrees, so linear work is wasteful.

Key InsightThe brute force redundantly counts nodes one by one even though large chunks of a complete tree are perfect and their size is known instantly. The insight: for any subtree, walk only down the far-left spine to get the left height and only down the far-right spine to get the right height. If those two heights are equal, the subtree is perfect and contains exactly 2h − 1 nodes with no traversal needed; otherwise recurse into the two children.
How to Think About It
  1. Write a helper that measures height by following only left children until null (the left spine), and another following only right children (the right spine).
  2. At the current node compare the two spine heights; equality means every level is full, i.e. a perfect subtree.
  3. If perfect, return the closed-form node count for that height using a bit shift, without descending further.
  4. If not perfect, return 1 plus the recursive counts of the left and right subtrees, which quickly bottom out into perfect pieces.
Why This WorksIn a complete tree, at most one subtree per level is "imperfect", so the recursion follows a single root-to-leaf path, doing O(log n) height computations, each itself O(log n). When left and right spine heights match, completeness forces the last level to be entirely filled for that subtree, so 2h − 1 is exact. Every node is accounted for either by a closed-form perfect count or by exactly one recursive descent, giving O(log2 n) total.
Watch Out For
  • Confusing "complete" with "full" or "perfect" — only the last level may be partial, and it fills left to right.
  • Off-by-one in the height: decide whether a single node has height 0 or 1 and stay consistent between the two spine measurements and the 2h formula.
  • Using 2**h − 1 with the wrong h (spine node count vs edge count) flips your answer by a factor of two.

The ProblemYou are given the root of a perfect binary tree (all leaves on the same level, every parent has two children). Each node has an extra next pointer that starts as null. Brute Force — O(n) time, O(n) space

Why optimize? The queue stores an entire level, up to n/2 nodes, so the follow-up demanding O(1) extra space rules the BFS out.

Key InsightThe brute force spends O(n) memory buffering each level in a queue even though, once a level is fully linked, its own next pointers already form a left-to-right linked list of that level. The insight: after connecting level k, traverse level k through the next pointers you just set and use each node's two children to stitch together level k+1, with a dummy/leftmost anchor to remember where the next level begins. Since the tree is perfect, every node has both children, so the wiring is uniform.
How to Think About It
  1. Keep a pointer to the leftmost node of the current level; start with the root as a level of one.
  2. Walk the current level via its already-set next chain; for each node, connect its left child to its right child.
  3. Connect that node's right child to the left child of the current node's next (when a next exists), bridging across parents.
  4. Descend to the leftmost child to become the next level's entry point, and repeat until you reach the leaf level.
Why This WorksBecause the tree is perfect, if level k is fully connected then its nodes appear in strict left-to-right order via next. Their children, taken as (left, right) pairs and bridged between adjacent parents, produce exactly the left-to-right order of level k+1. By always processing a level only after it is fully linked, the invariant "current level is a complete linked list" is preserved from the root down, so every pointer ends correct while using only a couple of moving pointers.
Watch Out For
  • Forgetting to bridge the gap between a node's right child and the next parent's left child — that cross-parent link is the whole point.
  • Not stopping at the leaf level; dereferencing children of leaves will crash.
  • Losing the leftmost entry pointer before descending, which strands the next level.

The ProblemGiven the root of a binary tree, return the maximum width among all levels. Brute Force — O(n) time

Key InsightThe brute force redundantly lets indices grow without bound even though only the difference between the first and last index on a level matters. The insight: at the start of each level, subtract the first node's index from every index on that level (normalize so the leftmost becomes 0). How to Think About It
  1. Do a level-order BFS carrying each node together with its positional index (left child 2*i, right child 2*i+1).
  2. At the beginning of each level, record the index of the first (leftmost) node in the queue.
  3. Subtract that first index from each node's index as you process the level, so indices restart near zero every level.
  4. Compute the level width as last_index - first_index + 1 and keep a running maximum.
Why This WorksWidth is defined purely as a difference of positions, and subtracting a common constant from every index on a level leaves all pairwise differences unchanged, so the computed width is identical to the un-normalized version. Normalizing per level caps the magnitude of indices at roughly the level's width, which is at most n, so no overflow occurs. The BFS still visits every node once, preserving O(n) time.
Watch Out For
  • Integer overflow in Go, Rust, TypeScript, and C++ if you skip normalization — Python's arbitrary-precision ints hide the bug and can mislead you.
  • Normalizing against the wrong reference (must be the first index of the current level, captured before you start popping).
  • Forgetting the +1, which turns a width into a gap count.

Key InsightThe brute force redundantly explores independent group choices even though the dislike relation is really a graph and the question is just "is this graph bipartite?". The insight: model each person as a node and each dislike as an edge, then attempt a 2-coloring. Color a start node, and every neighbor must get the opposite color; a conflict (a disliked neighbor already sharing your color) proves no valid split exists. How to Think About It
  1. Build an adjacency list from the dislike pairs so each person knows who they conflict with.
  2. Keep a color array initialized to a sentinel (uncolored) for every person.
  3. For each uncolored person, start a BFS/DFS, assign it a color, and give every neighbor the opposite color.
  4. If you ever reach a neighbor already colored the same as the current node, return false; if all components color cleanly, return true.
Why This WorksA valid two-group split is exactly a proper 2-coloring of the dislike graph, and a graph is 2-colorable if and only if it is bipartite (contains no odd-length cycle). BFS/DFS coloring assigns forced opposite colors along every edge; the only way it can fail is when an edge connects two already-equal colors, which happens precisely when an odd cycle exists. Iterating over all start nodes guarantees disconnected components are each checked, so the algorithm is both complete and correct in O(V + E).

The ProblemThere are n computers labeled 0..n−1 and a list connections where [a, b] is a cable directly linking a and b. Brute Force — O(n^2) or worse

Repeatedly test overall connectivity: from computer 0 run a BFS/DFS over the current cables to see which computers are reachable. Re-running full traversals after each hypothetical move is wasteful and can be quadratic or worse.

class Solution:
    def makeConnected(self, n, connections):
        if len(connections) < n - 1:
            return -1

        from collections import defaultdict, deque
        adj = defaultdict(list)
        for a, b in connections:
            adj[a].append(b)
            adj[b].append(a)

                u = q.popleft()

Why optimize? Re-scanning connectivity after each hypothetical cable move is quadratic, which is far too slow for 105 nodes and edges.

Key InsightThe brute force redundantly re-checks connectivity even though the answer is a simple counting fact. The insight: to connect a graph of n nodes you need at least n−1 cables total; if you have fewer, it is impossible (return −1). Otherwise you always have enough spare (redundant) cables, and the number of moves needed equals (number of connected components) − 1 — you just count components once with union-find or a single traversal.
How to Think About It
  1. First check feasibility: if the number of cables is less than n−1, return −1 immediately.
  2. Count how many connected components the current cabling forms, using union-find (union each cable's endpoints) or one linear DFS/BFS sweep.
  3. Realize that merging k components into one requires exactly k−1 re-plugged cables.
  4. Return (component count) − 1.
Why This WorksAny connected graph on n vertices is a spanning tree plus possibly extra edges, and a spanning tree has exactly n−1 edges; so with fewer than n−1 cables total, no configuration can connect everything. When you do have at least n−1 cables, every cable beyond the ones forming the spanning forest is redundant and can be freely relocated. Bringing c components down to a single one takes c−1 new links, and you are guaranteed to have at least that many spare cables, so the minimum is exactly c−1.
Watch Out For
  • The −1 case: check len(connections) < n - 1 before anything else.
  • Counting redundant edges is unnecessary — only the component count drives the answer once feasibility holds.
  • Off-by-one: the answer is components − 1, not the raw component count.

You want every city to be able to reach city 0. Return the minimum number of roads whose direction you must reverse to achieve that. Up to 5×104 cities.
Brute Force — O(n^2) time

For each city, try to walk toward city 0 following only the current road directions; if you cannot reach 0, you know some road along the intended path is misdirected. Naively probing each city's route to 0 and re-checking after conceptual flips is quadratic because paths overlap and get re-walked repeatedly.

Python
class Solution:
    def minReorder(self, n, connections):
        # brute: for each city, follow the undirected tree toward 0,
        # counting how many edges on that path point the wrong way.
        from collections import defaultdict
        forward = set()                 # original directed edges
        adj = defaultdict(list)
        for a, b in connections:
            forward.add((a, b))
            adj[a].append(b)
            adj[b].append(a)

        def path_to_zero(city):
            # BFS to find the unique path city -> 0, count wrong edges
            from collections import deque
            parent = {city: None}
            q = deque([city])
            while q:
                u = q.popleft()
                if u == 0:
                    break
                for v in adj[u]:
                    if v not in parent:
                        parent[v] = u
                        q.append(v)
            cnt, cur = 0, 0
            while parent[cur] is not None:
                p = parent[cur]
                if (p, cur) in forward:   # road points p->cur, i.e. away from 0
                    cnt += 1
                cur = p
            return cnt

        return sum(path_to_zero(c) for c in range(1, n))

How to Think About It

  3. As you move from a visited node to an unvisited neighbor, you are going outward from 0; if that neighbor edge is tagged original, the real road points away from 0 and must be reversed — increment a counter.
  4. Return the total count after the traversal completes.
Why This WorksThe underlying structure is a tree, so from city 0 there is exactly one path to every other city, and the traversal follows each of the n−1 edges precisely once. An edge needs reversing exactly when its real direction points away from 0 along that unique outward path — which is precisely when the outward step matches the original-direction tag. Counting those tagged outward crossings therefore counts every road that must be flipped, no more and no less, in O(n).

  • Not marking nodes visited leads to infinite loops or double counting; a tree still needs a visited set because edges are stored both ways.
  • Counting reverse-tagged edges by mistake — only original-direction outward edges get flipped.

Why optimize? Enumerating 2^n subsets is hopeless beyond ~20 elements, so we need to exploit structure among the numbers.

Key InsightSort the numbers first. Divisibility is transitive: if a divides b and b divides c, then a divides c. So once sorted, a valid divisible subset only needs each element to divide the next one — this collapses the problem into a Longest Increasing Subsequence, but ordered by divisibility instead of by value.
How to Think About It
  1. Sort the array ascending so any candidate "chain" reads left to right.
  2. Define dp[i] = length of the longest divisible chain that ends at index i, and think of an LIS-style scan.
  3. For each i, look back at every j < i where nums[i] % nums[j] == 0 and extend the best chain found there.
  4. Keep a parent pointer for each i so you can walk backward from the overall best endpoint to rebuild the actual subset.
Why This WorksBecause divisibility is transitive and the array is sorted, checking only "does nums[i] divide by nums[j]" for the immediate predecessor in the chain is enough to guarantee every earlier element also divides nums[i]. Thus a chain where each term divides the next is automatically a fully mutually-divisible subset, and the longest such chain is the answer.
Watch Out For
  • Forgetting to sort first — the transitivity argument depends on it.
  • Reconstructing the path: you must track parent indices and walk back from the best endpoint, then reverse.
  • Empty input must return an empty list without crashing.

#877 Stone Game MediumDPMathGame Theory

The ProblemThere are an even number of piles of stones arranged in a row; the total number of stones is odd. Alice and Bob alternate turns, Alice first, each taking an entire pile from either the left or right end. Both play optimally to maximize their own stones. Return true if Alice ends with more stones than Bob.
Brute Force — O(2^n) time

Model the game with a minimax recursion over the current sub-range [i, j]. At each step the current player picks the left or right pile and the opponent then plays optimally on the remaining range. Return the best score difference the current player can force. Without memoization this re-explores overlapping ranges exponentially.

def stoneGame(piles):
    n = len(piles)

    def best(i, j):
        # max score difference (current player minus opponent)
        if i > j:
            return 0
        take_left = piles[i] - best(i + 1, j)
        take_right = piles[j] - best(i, j - 1)
        return max(take_left, take_right)

    return best(0, n - 1) > 0

Why optimize? The naive recursion recomputes the same [i, j] sub-ranges over and over, giving exponential time that blows up for large pile counts.

Key InsightTrack a single score difference rather than two separate scores. Define dp[i][j] as the maximum (current player's stones minus opponent's stones) achievable on piles[i..j]. Whatever the current player takes counts positively, and the opponent's optimal play on the rest is subtracted. Alice wins exactly when dp[0][n-1] > 0. (There is also a slick math fact that Alice always wins, but the interval DP is the transferable technique.)
How to Think About It
  1. Reframe the goal as a single number: the best score margin the player to move can guarantee on a sub-range.
  2. Base case: a single pile [i, i] gives a margin of piles[i]; an empty range gives 0.
  3. Transition: dp[i][j] = max(piles[i] - dp[i+1][j], piles[j] - dp[i][j-1]) — you gain your pick and inherit the negated best of the opponent.
  4. Fill by increasing interval length (or memoize the recursion), then check whether dp[0][n-1] is positive.
Why This WorksThe margin framing captures the zero-sum nature of the game: whatever one player gains, the other loses. Subtracting the opponent's best response margin correctly models "they will also play optimally against me." Since dp[0][n-1] is Alice's guaranteed lead under optimal play by both, a positive value means Alice necessarily ends ahead.
Watch Out For
  • Confusing the score-difference framing with tracking absolute scores for both players — the difference form is far cleaner.
  • Getting the base case wrong: i == j should return the pile value, not 0.
  • Filling the 2D table in the wrong order; shorter intervals must be computed before longer ones.

#1049 Last Stone Weight II MediumDPKnapsack

The ProblemYou have stones with given positive weights. Repeatedly pick any two stones a and b (a <= b), smash them, and if a != b a new stone of weight b - a returns to the pile. Return the smallest possible weight of the stone left at the end (0 if none remain).
Brute Force — O(2^n) time

def lastStoneWeightII(stones):
    n = len(stones)
    best = float('inf')

    def rec(i, total):
        nonlocal best
        if i == n:
            best = min(best, abs(total))
            return
        rec(i + 1, total + stones[i])
        rec(i + 1, total - stones[i])

    rec(0, 0)
    return best

Why optimize? Trying all 2^n sign assignments is exponential; we need to collapse the many combinations that land on the same total.

Key InsightAssigning each stone a + or - sign splits the stones into two groups; the final weight is the absolute difference between the two group sums. To minimize that difference, make each group's sum as close as possible to total/2. That is exactly a subset-sum / 0-1 knapsack: find the largest reachable subset sum s <= total/2, and the answer is total - 2*s.
How to Think About It
  1. Recognize that every play just partitions stones into a "plus" group and a "minus" group; the leftover is |sumPlus - sumMinus|.
  2. Since the two sums add up to total, minimizing their difference means pushing one group's sum as close to total/2 as possible.
  3. Compute all reachable subset sums up to total/2 using a boolean DP (or a set of achievable sums).
  4. Take the largest reachable sum s and return total - 2*s.
Why This WorksAny sequence of smashes corresponds to some ± sign assignment, and conversely any sign assignment is realizable, so the set of possible outcomes equals { |total - 2s| : s is an achievable subset sum }. This is minimized by choosing the achievable s closest to total/2 from below, giving the provably smallest remaining weight.
Watch Out For
  • Not recognizing this as subset-sum/knapsack in disguise — the smashing narrative hides it.
  • DP only needs sums up to total/2; going to the full total wastes space and can double-count.
  • Answer is total - 2*bestSum, not bestSum itself.

Why optimize? Sorting every unique word is O(n log n); when k is much smaller than the number of unique words, a bounded heap does asymptotically less work at O(n log k).

Key InsightMaintain a heap of at most k elements. Use a comparator where higher frequency ranks "better," and for equal frequency the lexicographically smaller word ranks "better." By keeping a min-heap under this ordering and evicting the worst whenever the size exceeds k, only the top k survive — each push/pop costs log k instead of log n.
How to Think About It
  1. Count word frequencies with a hash map first.
  2. Decide your heap's ordering so the element most eligible for eviction sits at the top: lowest frequency, and among ties the alphabetically larger word.
  3. Push each (word, count) pair; whenever the heap exceeds size k, pop the top to discard the current worst.
  4. Drain the heap into a list and reverse it so the strongest candidates come first in the required order.
Why This WorksA size-k min-heap under the "eviction" ordering always holds the k best seen so far: anything worse than all k current members is immediately discarded. Because the tie-break is baked into the comparator (larger word evicted first), the final drained-and-reversed order matches the required "descending frequency, ascending alphabetical" ranking exactly.
Watch Out For
  • The tie-break direction is the trap: for a min-heap of size k you must make the lexicographically LARGER word compare as "smaller" so it is evicted first.
  • Remember to reverse (or otherwise re-order) the drained heap, since popping yields worst-first.
  • Counting must be exact before you start heaping.

Why optimize? Permuting is factorial and shuffling is non-deterministic; neither scales, so we need a deterministic greedy construction.

Key InsightGreedily place the most frequent remaining character at each position, but never the character you just placed. A max-heap keyed by remaining count lets you always grab the top candidate; hold the just-used character aside for one step so it cannot repeat. It is impossible exactly when some character's count exceeds (n + 1) / 2.
How to Think About It
  1. Count each character's frequency; if any exceeds (n + 1) / 2, immediately return "" — there is no valid arrangement.
  2. Put all characters into a max-heap ordered by remaining count.
  3. Repeatedly pop the highest-count character, append it, decrement its count, and hold it aside instead of pushing it back right away.
  4. On the next iteration, push the held-back character back (if it still has count > 0) before popping the new top, guaranteeing the previous character isn't reused adjacently.
Why This WorksAlways emitting the currently most frequent allowable character prevents any single character from piling up at the end. Deferring the just-used character by exactly one slot enforces the no-adjacent-duplicates rule. If the most frequent character's count is at most (n + 1) / 2, there are always enough gaps to separate its copies, so the greedy process succeeds; otherwise the pigeonhole principle makes it impossible.
Watch Out For
  • The impossibility check: the max count must be compared against (n + 1) / 2 (integer division), not n / 2.
  • You must hold the just-placed character back for one iteration; pushing it straight back onto the heap can produce adjacent duplicates.
  • Only re-add the held character when its remaining count is still positive.

Why optimize? The triple loop wastes work exploring cut positions that a smart search would prune instantly, and it does not generalize to a variable number of segments.

Key InsightBuild the address one segment at a time with backtracking: at each step consume 1, 2, or 3 characters as the next segment, validate it immediately, and stop the branch as soon as a segment is invalid or you cannot possibly finish with exactly four segments consuming all characters.
How to Think About It
  1. Track a start index into the string and how many segments you have placed so far.
  2. When four segments are placed, accept only if the start index has reached the end of the string.
  3. For the current position, try taking 1 to 3 characters; validate each candidate segment (length, leading zero, ≤255) before recursing.
  4. Prune early: if the remaining characters cannot fill the remaining segments (too few or too many), abandon the branch.
Why This WorksEvery valid IP address corresponds to exactly one way of cutting the string into four ordered, non-empty, valid segments. Backtracking enumerates precisely those cuts: because it only extends a partial address with a valid segment and only accepts when all four parts exactly consume the string, it produces every valid address once and nothing invalid. Early pruning removes branches that could never complete, so no correct answer is lost.
Watch Out For
  • Leading-zero rule: "0" is a valid segment, but "01", "00", "012" are not.
  • You must end with exactly four segments AND have consumed the entire string; three valid segments plus leftover digits is not an answer.
  • Length bounds: strings shorter than 4 or longer than 12 digits can never form a valid address.

The ProblemGiven an integer array nums and an integer k, determine whether it is possible to partition the array into exactly k non-empty subsets whose sums are all equal. Return true or false. Every element must belong to exactly one subset.
Brute Force — O(kn) time

Maintain k running bucket sums. Assign each number, one at a time, to one of the k buckets, recursing to place the next number. If every number is placed and all buckets equal the target sum, succeed. This explores up to k choices per element, giving k^n branches with no pruning.

    buckets = [0] * k

                buckets[b] -= nums[i]
        return False
    return place(0)

Why optimize? Assigning symmetric, interchangeable buckets re-explores equivalent states endlessly; without pruning and symmetry-breaking the search blows up on modest inputs.

Key InsightFill one bucket to exactly the target before starting the next. Sort numbers descending so large elements fail fast, skip a number if the last identical number was skipped at this position, and prune the whole call once the total is not divisible by k or any element exceeds the target.
How to Think About It
  1. Compute total; if it is not divisible by k, or the largest element exceeds target = total / k, return false immediately.
  2. Sort numbers in descending order so oversized elements are rejected early and buckets fill quickly.
  3. Fill buckets one at a time: track a running sum and a used[] array; recurse to complete the current bucket before opening the next.
  4. When the current bucket reaches target exactly, decrement the bucket counter and restart filling from the beginning for the next bucket; skip duplicate values to avoid redundant branches.
Why This WorksIf a valid partition exists, some assignment fills the first bucket to the target; the order in which buckets are filled does not matter, so we may commit to completing one bucket at a time without losing any solution. Sorting descending and skipping duplicates only removes branches that are symmetric to ones already explored, and the ≤target pruning discards branches that can never reach exactly the target. Thus the search still visits at least one representative of every genuinely distinct partition, so it answers correctly while examining far fewer states.
Watch Out For
  • Early exits: total not divisible by k, and any single element greater than the target both mean an immediate false.
  • Sort descending for effective pruning; ascending order defeats the fail-fast behavior.
  • Use a used[] marker and the skip-duplicate check to avoid re-trying identical numbers in the same slot.

#735 Asteroid Collision MediumStackArraySimulation

The ProblemYou are given an array asteroids where each value's sign is its direction (positive means moving right, negative means moving left) and its absolute value is its size. When a right-moving asteroid meets a left-moving one, the smaller explodes; if they are equal, both explode. Asteroids moving in the same direction, or moving apart, never collide. Return the state of the asteroids after all collisions.
Brute Force — O(n2) time

Repeatedly scan the current list for an adjacent colliding pair (a positive immediately followed by a negative). Resolve that one collision by removing the destroyed asteroid(s), then restart the scan. Continue until a full pass finds no collisions, meaning the configuration is stable.

Python
def asteroidCollision(asteroids):
    a = list(asteroids)
    changed = True
    while changed:
        changed = False
        for i in range(len(a) - 1):
            if a[i] > 0 and a[i + 1] < 0:
                if a[i] < -a[i + 1]:
                    del a[i]
                elif a[i] > -a[i + 1]:
                    del a[i + 1]
                else:
                    del a[i:i + 2]
                changed = True
                break
    return a

Why optimize? Restarting the scan after every single collision revisits the same elements repeatedly, and list deletions in the middle are expensive, giving quadratic behavior.

Key InsightProcess asteroids left to right using a stack. A new left-moving asteroid only ever collides with the right-moving asteroids sitting on top of the stack, so resolve those collisions in a while-loop: annihilate top elements until the newcomer is destroyed, survives, or the stack no longer holds a smaller right-mover.
How to Think About It
  1. Iterate through the asteroids; keep a stack of survivors so far.
  2. A positive asteroid, or a negative one when the stack is empty or its top is negative, cannot collide now, so push it.
  3. Otherwise (negative meeting a positive top), loop: compare sizes, popping the top when it is smaller, breaking when the incoming one is destroyed, and popping both when equal.
  4. If the incoming left-mover outlives every positive on top (loop ends without destruction), push it onto the stack.
Why This WorksCollisions only happen between a right-mover and a later left-mover, and once an asteroid is to the left of the stack top it can never be reached by a future left-mover it did not already meet. Processing left to right means every survivor on the stack is either left-moving (nothing to its right can catch it) or right-moving (a candidate target for the next left-mover). The while-loop resolves each incoming left-mover against exactly the asteroids it could collide with, in order, so the final stack is the unique stable configuration.
Watch Out For
  • Equal sizes destroy both asteroids; handle that case distinctly from the strictly-smaller cases.
  • A left-moving asteroid survives when the stack is empty or its top is also left-moving (negative).
  • Use the loop's normal completion (no break/destruction) to decide whether to push the surviving left-mover.

#658 Find K Closest Elements MediumBinary SearchTwo Pointers

The ProblemGiven a sorted integer array arr, an integer k, and an integer x, return the k elements closest to x, as a sorted list. An element a is closer than b when |a - x| < |b - x|, and when the distances tie the smaller element is preferred. The result is a contiguous window of the array returned in ascending order.
Brute Force — O(n log n) time

Sort all elements by their distance to x, breaking ties in favor of the smaller value. Take the first k elements from that ordering, then sort those k back into ascending numeric order to produce the answer.

Python
def findClosestElements(arr, k, x):
    ordered = sorted(arr, key=lambda a: (abs(a - x), a))
    return sorted(ordered[:k])

Why optimize? Sorting throws away the fact that the array is already sorted and that the answer is a contiguous block, doing n log n work for what a window search settles in log n.

Key InsightThe answer is a contiguous window of length k, so search only for its left boundary. Binary search lo in [0, n - k]; at each step compare the distance of arr[mid] to x against the distance of arr[mid + k] to x, sliding the window right when the far-right candidate is strictly closer.
How to Think About It
  1. Set lo = 0 and hi = n - k; the answer window starts somewhere in this range.
  2. Take mid; compare x - arr[mid] against arr[mid + k] - x, i.e. how much closer the leftmost included element is versus the first excluded element on the right.
  3. If arr[mid + k] is strictly closer to x, the window should move right, so set lo = mid + 1; otherwise move hi = mid.
  4. When lo meets hi, return the slice arr[lo : lo + k].
Why This WorksBecause arr is sorted, comparing arr[mid] and arr[mid + k] against x tells you whether dropping the current leftmost element in favor of the next element on the right reduces total distance; the comparison x - arr[mid] > arr[mid + k] - x is exactly the condition to prefer the right neighbor, and it also encodes the tie-break toward smaller elements (ties keep the left element). This condition is monotonic in mid, so binary search converges on the unique optimal left boundary, and the k consecutive elements from there are the closest set in sorted order.
Watch Out For
  • Binary-search the window boundary by comparing mid with mid + k, not arr[mid] with x directly.
  • Tie-breaking prefers the smaller element; the > (strict) comparison keeps the left side on ties.
  • Search space for lo is [0, n - k]; the returned answer is a contiguous slice, already sorted.

The ProblemGiven a list of product names and a searchWord, simulate typing searchWord one character at a time. After each character is typed, return up to three product names from the list that share the currently typed prefix, choosing the three lexicographically smallest. The overall result is a list of lists: one suggestion list per typed prefix.
Brute Force — O(L · n log n) time

Why optimize? Re-sorting the matching products from scratch for every prefix repeats almost identical work; sorting once up front lets each prefix be answered by a narrow range lookup instead.

Key InsightSort the products once. Then either build a Trie whose every node caches up to three lexicographically smallest completions, or binary-search the sorted list for the contiguous range of products sharing each prefix and read the first three. Cumulative prefixes let you narrow the search range as characters are added.
How to Think About It
  1. Sort products lexicographically so that "first three matches" is simply the first three within a contiguous range.
  2. Trie route: insert each product; at each node along its path, append it to that node's suggestion list only while the list holds fewer than three.
  3. Binary-search route: for each growing prefix, use lower_bound to find where the prefix would start, then read forward up to three products that actually begin with the prefix.
  4. Because prefixes are cumulative, you can keep a shrinking [lo, hi) window (or walk deeper into the Trie) instead of restarting from the whole list.
Why This WorksIn a sorted list, all products sharing a given prefix occupy one contiguous block, and within that block the lexicographically smallest are exactly the first entries. So taking the first three products at or after the prefix's lower bound (that still start with the prefix) yields precisely the required suggestions. The Trie encodes the same idea structurally: each node represents a prefix, and caching its three smallest completions during a sorted insertion means every prefix query is an O(1) lookup. Both approaches respect the cumulative nature of typing, since a longer prefix's matches are always a subset of the shorter prefix's block.
Watch Out For
  • Lexicographic (dictionary) order, not length; sort before doing any lookups.
  • Return at most three suggestions per prefix, and possibly fewer (or none) when few products match.
  • Prefixes are cumulative: verify a candidate still actually starts with the full current prefix, not just that it sits past the lower bound.
