#3 · Longest Substring Without Repeating Characters

Medium Sliding Window Hash 3 days since reviewed

The Problem

Given a string s, find the length of the longest substring that contains no repeating characters. The answer is a contiguous slice of the string, not a subsequence.

Quick Reminder

Variable-size sliding window plus a hash map from char to last-seen index. When you hit a duplicate already inside the window, jump left past it.

Key Insight

You never need to shrink the window one step at a time. Store the last index where each character appeared; on collision, move left = max(left, last[c] + 1) directly. The max is critical because last[c] might be behind the current window.

The Problem

You are given two sorted arrays nums1 (with extra space at the end of length n) and nums2 of length n. Merge nums2 into nums1 in place so the result is sorted.

Quick Reminder

Two pointers from the back. Write into nums1[m+n-1] downward so you never overwrite an unread element.

Iterating from the front would force you to shift elements. Iterating from the back uses the trailing free space exactly once per write. The only remaining edge case is leftover elements in nums2 — leftover in nums1 are already in place.

#424 · Longest Repeating Character Replacement

MediumSliding WindowCharacter Frequency

The Problem

You are given a string s consisting of uppercase English letters and an integer k. You may replace at most k characters with any uppercase letters. Return the length of the longest substring you can make consisting of one repeating letter after these replacements.

Brute Force

Enumerate every substring. For each substring, count the frequency of each character, find the most common one, and check whether length - maxFreq ≤ k. Track the maximum valid length. This is O(n² · Σ) where Σ = 26.

Python — Brute Force 
from collections import Counter

def characterReplacement(s: str, k: int) -> int:
    best = 0
    n = len(s)
    for i in range(n):
        for j in range(i, n):
            cnt = Counter(s[i:j+1])
            length = j - i + 1
            if length - max(cnt.values()) <= k:
                best = max(best, length)
    return best
Go — Brute Force 
func characterReplacement(s string, k int) int {
    best, n := 0, len(s)
    for i := 0; i < n; i++ {
        var cnt [26]int
        for j := i; j < n; j++ {
            cnt[s[j]-'A']++
            maxF := 0
            for _, c := range cnt {
                if c > maxF { maxF = c }
            }
            length := j - i + 1
            if length-maxF <= k && length > best {
                best = length
            }
        }
    }
    return best
}

Why optimize? Recounting frequencies for every substring is wasteful; a sliding window can maintain the counts incrementally.

Key Insight

A window [l, r] is valid iff (r - l + 1) - maxFreq ≤ k. When the window becomes invalid you only need to shrink it by one step and never re-expand — because you only care about the maximum length ever achieved, not every valid length. So maxFreq never needs to decrease; even a stale maxFreq still upper-bounds the answer.

How to Think About It

  1. Set up count[26], l = 0, and a running maxFreq.
  2. Move r from 0 to n-1, incrementing count[s[r]] and updating maxFreq = max(maxFreq, count[s[r]]).
  3. Check whether (r - l + 1) - maxFreq > k. If so, decrement count[s[l]] and advance l by one (do not loop — one shift per step is enough).
  4. Because the window only shrinks when invalid and only by one, the window length is monotonically non-decreasing.
  5. At the end, the answer is simply r - l + 1 (i.e., the final window length), or you can track the max as you go.

Why This Works

If a window of length L is valid, any smaller window is also potentially valid — but we never want smaller, so we only ever try to grow. When the new right character would make the window invalid, sliding l by 1 keeps the window the same length, which is fine because any longer valid window must use a new maxFreq that exceeds the stale one. The stale maxFreq can only cause us to under-report length, never over-report.

⚠ Most Dangerous Mistake

Trying to “correctly” recompute maxFreq after shrinking by scanning the 26-element array. This is unnecessary and ruins the complexity intuition. The trick is that maxFreq is allowed to be stale.

  • Using a while loop on the shrink step. Use a single if — one shrink per expansion.
  • Forgetting to decrement count[s[l]] when sliding the left pointer.
  • Computing maxFreq from the array each iteration (O(26) extra work that misses the point).

#438 · Find All Anagrams in a String

MediumSliding WindowFrequency Match

The Problem

Given two strings s and p, return an array of all the start indices of p's anagrams in s. An anagram is a rearrangement; the order of indices in the output does not matter. The strings consist of lowercase English letters.

Brute Force

For every starting index i in s with at least len(p) characters remaining, take the substring of length len(p), sort it (or count its characters), and compare against p's sorted version (or count). If equal, record i. This is O(n · m · log m) with sorting, or O(n · Σ) with counts.

Python — Brute Force 
from collections import Counter

def findAnagrams(s: str, p: str) -> list[int]:
    res = []
    target = Counter(p)
    m = len(p)
    for i in range(len(s) - m + 1):
        if Counter(s[i:i+m]) == target:
            res.append(i)
    return res
Go — Brute Force 
func findAnagrams(s, p string) []int {
    res := []int{}
    m, n := len(p), len(s)
    var target [26]int
    for i := 0; i < m; i++ { target[p[i]-'a']++ }
    for i := 0; i+m <= n; i++ {
        var win [26]int
        for j := 0; j < m; j++ { win[s[i+j]-'a']++ }
        if win == target { res = append(res, i) }
    }
    return res
}

Why optimize? Rebuilding the window's frequency array from scratch at every step is O(m) per step; sliding lets us do O(1) per step.

Key Insight

A fixed-size window of length m = len(p) over s. Maintain a single 26-element frequency array; when the window slides, you do one decrement (left character leaving) and one increment (right character entering). Compare to target — in Go, comparing two [26]int arrays with == is O(26), constant. To avoid the per-step compare you can track a matches counter, but it is not required.

How to Think About It

  1. Build target[26] from p. Let m = len(p).
  2. Slide a window of length m across s; first prime it by counting the initial m characters into window[26].
  3. If window == target, push 0 to the result.
  4. For i from m to n-1: increment window[s[i]] and decrement window[s[i-m]]. If equal to target, push i - m + 1.
  5. Return the result list.

Why This Works

An anagram is fully characterized by its character multiset. Two equal-length strings are anagrams iff their 26-element frequency vectors are equal. The fixed-size window slides over s updating exactly two entries per shift, so we never reread interior characters. Array equality on [26]int is O(1) for the purposes of complexity analysis.

⚠ Most Dangerous Mistake

Confusing the “decrement before increment” order or using the wrong index for the left character. The character leaving the window when the right edge moves to i is s[i - m], not s[i - m - 1] nor s[i - m + 1]. Draw the window once and pin it down.

  • Returning early when len(s) < len(p) is not necessary, but the loop bounds must handle it — range(len(s) - m + 1) becomes empty cleanly.
  • Using a hash map instead of a 26-int array. It works but is slower and forces an explicit equality helper.
  • Forgetting that the output is start indices, not the substrings themselves.

#567 · Permutation in String

MediumSliding WindowFrequency Match

The Problem

Given two strings s1 and s2, return true if s2 contains a permutation of s1, i.e., one of s1's permutations appears as a contiguous substring of s2. Both strings consist of lowercase English letters.

Brute Force

Generate every permutation of s1 and check whether any is a substring of s2. This is O(m! · n · m) and explodes immediately. A slightly less embarrassing version: for every length-m window of s2, sort and compare to sorted s1.

Python — Brute Force 
def checkInclusion(s1: str, s2: str) -> bool:
    m = len(s1)
    target = sorted(s1)
    for i in range(len(s2) - m + 1):
        if sorted(s2[i:i+m]) == target:
            return True
    return False
Go — Brute Force 
import "sort"

func checkInclusion(s1, s2 string) bool {
    m := len(s1)
    t := []byte(s1)
    sort.Slice(t, func(i, j int) bool { return t[i] < t[j] })
    for i := 0; i+m <= len(s2); i++ {
        w := []byte(s2[i : i+m])
        sort.Slice(w, func(i, j int) bool { return w[i] < w[j] })
        if string(w) == string(t) { return true }
    }
    return false
}

Why optimize? Sorting every window is O(m log m) per shift, and we're throwing away work as the window moves.

Key Insight

Exactly the same idea as #438 but with a boolean return and an early exit. A permutation of s1 is just a substring of s2 with the same character multiset, so we maintain a fixed-size length-m sliding window's frequency vector and compare to s1's frequency vector at each step. The first match means “return true”.

How to Think About It

  1. If len(s1) > len(s2), return false immediately.
  2. Build need[26] from s1 and window[26] by counting the first m characters of s2.
  3. If window == need, return true.
  4. Slide: for i from m to n-1, do window[s2[i]]++ and window[s2[i-m]]--; check equality.
  5. If the loop finishes with no match, return false.

Why This Works

Every length-m substring of s2 is examined exactly once. Two equal-length strings are permutations of each other iff their character-count vectors match. Sliding updates the vector in O(1) per shift, and the equality check is O(26) = O(1) for constant alphabet.

⚠ Most Dangerous Mistake

Swapping the order of the two window updates such that the array equality test is performed on a window that briefly has length m+1 or m-1. Do both updates (decrement-left and increment-right) before the comparison, or restructure so that the right pointer advances first to i and then the left pointer leaves the index i - m.

  • Off-by-one in the priming step: the initial window covers indices 0..m-1, not 0..m.
  • Returning false too eagerly inside the loop instead of after it.
  • Allocating a fresh frequency array per iteration instead of mutating in place.

#387 · First Unique Character in a String

EasyHash MapCounting

The Problem

Given a string s, find the first non-repeating character in it and return its index. If it does not exist, return -1. The string contains only lowercase English letters. You must return the index of the first occurrence (in original order) whose count in the entire string equals one.

Examples:

  • "leetcode" returns 0 ('l' appears once).
  • "loveleetcode" returns 2 ('v' is the first unique character).
  • "aabb" returns -1 (no unique characters).

Brute Force

For each index i, scan the entire string and count how many times s[i] appears. If the count is exactly 1, return i. This makes a nested loop and runs in O(n^2). It works, but every character causes a full re-scan of the string. A subtle variant uses str.count in a single loop — cleaner code but still O(n^2) because count is itself a linear scan.

Python — Brute Force 
def firstUniqChar(s: str) -> int:
    n = len(s)
    for i in range(n):
        unique = True
        for j in range(n):
            if i != j and s[i] == s[j]:
                unique = False
                break
        if unique:
            return i
    return -1
Go — Brute Force 
func firstUniqChar(s string) int {
    n := len(s)
    for i := 0; i < n; i++ {
        unique := true
        for j := 0; j < n; j++ {
            if i != j && s[i] == s[j] {
                unique = false
                break
            }
        }
        if unique {
            return i
        }
    }
    return -1
}

Why optimize? O(n^2) scans the string repeatedly when a single pass to count, then a single pass to find, gives O(n). For strings of length 100,000 this is a 10000x speedup in practice — the difference between "instant" and "timed out".

Key Insight

Count every character once into a hash map (or 26-slot array). Then walk the string a second time and return the first index whose character has count exactly 1. Two linear passes beats nested scanning. Separating "compute global facts" from "scan in order" is a fundamental pattern that recurs throughout array and string problems.

How to Think About It

  1. Realize the question is really: "which characters appear exactly once?"
  2. Build a frequency map of every character in one pass.
  3. Walk the original string in order (this preserves "first" semantics).
  4. Return the index where the frequency is 1.
  5. If no such index exists after the second pass, return -1.
  6. Optional optimization: since input is guaranteed lowercase ASCII, use a 26-slot array instead of a hash map — constant factor improvement.
  7. Recognize the "two-pass over array, accumulate then scan" pattern — it's reusable.

Why This Works

Counting captures total occurrences without losing position information, because the position is recovered by re-walking the original string. The first index satisfying count==1 is necessarily the first unique character. Splitting the problem into "what" (frequency) and "where" (position) is the central trick — we use the map only as a fact base, and the original string as the order-preserving driver.

⚠ Most Dangerous Mistake

Returning the first character with count 1 from iterating the map. Map iteration order is not guaranteed in Go (and only insertion-ordered in Python 3.7+), so you must walk the original string to find the first index.

  • Iterating the dict/map instead of the string — loses positional order.
  • Forgetting the -1 return for the all-repeats case (e.g. "aabb").
  • Using s.index(c) inside the second pass — works but reintroduces O(n^2).
  • Building OrderedDict when a regular dict already preserves insertion order in modern Python.
  • Using a set instead of a counter — sets only know presence, not multiplicity.

Worked Example

Trace through s = "loveleetcode":

  • Pass 1 builds counts: {l:1, o:2, v:1, e:4, t:1, c:1, d:1}.
  • Pass 2 walks indices: l(1) — count is 1, return index 2 (the 'v')? No: index 0 is 'l' with count 1, so return 0? Re-check: 'l' appears once (only at index 0 and... actually "loveleetcode" has l only at index 0), so we return index 0... wait, the canonical answer is 2 ('v'). Let's recount: l-o-v-e-l-e-e-t-c-o-d-e. 'l' at indices 0 and 4 — count 2. So 'v' at index 2 is the first with count 1.
  • This trace illustrates why hand-counting carefully matters — off-by-one errors in frequency tables are the #1 source of wrong answers.

Follow-up Variants

  • Streaming variant: what if characters arrive one at a time and you must report the first unique so far? Use a linked-list-backed hash for O(1) amortized updates.
  • Case-insensitive: lowercase the input first or extend the counter to 52 slots.
  • Return the character itself instead of the index: trivial modification — same algorithm.
  • Unicode input: swap fixed 26-slot array for a hash map; everything else is unchanged.
  • Find ALL unique characters (return list of indices): just collect during the second pass instead of returning early.

Interview Tips

State out loud: "I'll do two passes — one to count, one to find the first index whose count is 1." This shows you've structured the problem correctly. #36 · Valid Sudoku

MediumHash SetMatrix

The Problem

Determine if a 9x9 Sudoku board is valid. Only filled cells (digits '1'-'9') need to be validated. Each row, each column, and each of the nine 3x3 sub-boxes must contain the digits without repetition. Empty cells contain '.' and are ignored. The board does not have to be solvable.

Constraints & Notes:

  • Board is fixed-size 9x9. There are 9 rows, 9 columns, and 9 boxes — 27 constraint groups total.
  • Each group must contain each digit at most once (not exactly once — partial boards are fine).
  • "Validity" here is local consistency, not global solvability.

Brute Force

Run three separate passes: one to validate each row, one for each column, one for each 3x3 box. Each pass uses its own hash set per group. It is correct but verbose and walks the board three times. The code length alone is a hint that there is structural duplication to be removed.

def isValidSudoku(board):
    # Rows
    for r in range(9):
        seen = set()

            if v in seen: return False
            seen.add(v)
    # Columns
    for c in range(9):

Why optimize? Three full board scans is wasteful. We can validate all 27 constraints in a single pass with one set. Even though board size is constant (81 cells), the elegance and reusability of the single-pass approach matters for interview impressions.

Key Insight

Use one set that holds composite keys like "row 3: 7", "col 4: 7", "box 1,1: 7". As you iterate the board once, generate three keys per filled cell. If any key already exists, the board is invalid. Composite keys collapse three independent uniqueness checks into one container with the same asymptotic cost as a single check.

How to Think About It

  1. Note that each digit belongs to exactly one row, one column, and one 3x3 box.
  2. Encode each membership as a distinct string so they never collide.
  3. Compute the box index as (r // 3, c // 3) — integer division partitions the grid.
  4. For each filled cell, attempt to add all three keys to a single shared set.
  5. If any key is already present, return false; otherwise after iteration return true.
  6. Alternative: keep three separate fixed-size arrays (rows[9][9], cols[9][9], boxes[9][9]) of bool. Same idea, no string formatting overhead.
  7. Linear box-index (r/3)*3 + c/3 maps 2D block coordinates into a single 0..8 index, useful for flat arrays.

Why This Works

Two different cells in the same row produce the same row-key, so a duplicate within that row collides. The same is true for column and box keys. Distinct prefixes ("row ", "col ", "box ") ensure no false cross-collisions. The set acts as a witness for all 27 disjoint constraint groups simultaneously — one container, three orthogonal projections of each digit's location.

⚠ Most Dangerous Mistake

Forgetting the prefix — storing just (r, v) and (c, v) as tuples without distinguishing them. A row-1 with value 5 then matches a column-1 with value 5 and you reject valid boards.

  • Computing box index as (r/3)*3 + c/3 in Python and getting a float — use //.
  • Skipping the '.' empty cell check and treating dots as a "digit" that repeats.
  • Confusing rows of bytes with rows of runes in Go (board[r][c] is byte).
  • Returning false on the first iteration before even examining all cells in the row — the continue for dots must come before the duplicate test.
  • Forgetting that "valid" doesn't mean "solvable" — some partially-filled boards have no completion but are still valid.

Worked Example

For the cell board[4][7] = '3':

  • Row key: "row 4: 3"
  • Col key: "col 7: 3"
  • Box key: "box 1,2: 3" (since 4//3=1 and 7//3=2)
  • All three are inserted into the same shared set. A collision on any of them — from any other cell's keys — means the board is invalid.

#128 · Longest Consecutive Sequence

MediumHash SetArray

The Problem

Given an unsorted array of integers nums, return the length of the longest sequence of consecutive integers. Numbers do not need to be adjacent in the array. For example, [100, 4, 200, 1, 3, 2] contains the run 1, 2, 3, 4 so the answer is 4. The algorithm must run in O(n) time.

Examples:

  • [100, 4, 200, 1, 3, 2] returns 4 (run 1,2,3,4).
  • [0, 3, 7, 2, 5, 8, 4, 6, 0, 1] returns 9 (run 0..8).
  • [] returns 0.
  • [1, 2, 0, 1] returns 3 (duplicates don't extend).

Brute Force

Sort the array, then walk it counting runs where each element is exactly one more than the previous (and skipping duplicates). This is O(n log n) due to the sort — correct, but it violates the O(n) requirement of the problem. In a real interview, mention this as the obvious starting point before pivoting to the hash-set solution.

Python — Brute Force 
def longestConsecutive(nums):
    if not nums: return 0
    nums = sorted(set(nums))
    best = run = 1
    for i in range(1, len(nums)):
        if nums[i] == nums[i-1] + 1:
            run += 1
            best = max(best, run)
        else:
            run = 1
    return best
Go — Brute Force 
func longestConsecutive(nums []int) int {
    if len(nums) == 0 { return 0 }
    sort.Ints(nums)
    best, run := 1, 1
    for i := 1; i < len(nums); i++ {

        if nums[i] == nums[i-1]+1 {
            run++
            if run > best { best = run }
        } else {
            run = 1
        }
    }
    return best
}

Why optimize? Sorting is O(n log n). The problem mandates O(n), achievable with a hash set. The trade is memory for time: we use O(n) extra space to skip the comparison-based lower bound of sorting.

Key Insight

Put every number into a set for O(1) lookup. Then only start counting a run at a number x when x - 1 is NOT in the set — meaning x is the start of its sequence. From each start, walk forward while x + 1 is present. Each element is visited at most twice total, giving O(n). This "anchor at minimum, extend forward" idea avoids the cost of sorting while still giving us the consecutive structure we'd otherwise need order to detect.

How to Think About It

  1. Build a set from nums for constant-time lookup and to de-dup.
  2. For each number, ask: "Am I the start of a run?" Yes iff n - 1 is not in the set.
  3. If yes, extend forward: count n, n+1, n+2, ... while each is in the set.
  4. Track the maximum length found across all run-starts.
  5. Convince yourself this is O(n): each element is the "end" of exactly one extension, so total inner-loop work is bounded by n.
  6. Edge case: empty input must return 0; single element must return 1.
  7. Optional: iterate the set instead of the list to avoid revisiting duplicates — same asymptotic, slightly fewer iterations.

Why This Works

The "start-only" check guarantees we count each maximal consecutive run exactly once. Without it, the run 1,2,3,4 would be re-counted starting at 2, 3, and 4 too — turning O(n) into O(n^2) in the worst case. Aggregating across all elements, the inner-loop work is at most n (each element gets "extended past" at most once across the entire algorithm), giving linear total time.

⚠ Most Dangerous Mistake

Omitting the "n - 1 not in set" guard. The algorithm still produces the right answer but degrades to O(n^2) on inputs like [1, 2, 3, ..., n] — failing the time-complexity requirement and TLE on large tests.

  • Returning 0 vs 1 for a single-element array — remember length 1 is a valid run.
  • Forgetting to handle the empty input.
  • Using nums.count(x-1) instead of set membership — O(n) per check kills complexity.
  • Mutating the set while iterating — iterate the original array (or list of keys).
  • Trying to use union-find for this problem — it works but is more code than the hash-set version for the same complexity.
  • Iterating the set instead of the array — correct but obscures the deduplication step.

Worked Example

Take nums = [100, 4, 200, 1, 3, 2]:

  • Set is {100, 4, 200, 1, 3, 2}.
  • Is 100 a run start? 99 not in set — yes. Extend: 101 not in set — run length 1.
  • Is 4 a run start? 3 IS in set — skip.
  • Is 200 a run start? Yes. Run length 1.
  • Is 1 a run start? 0 not in set — yes. Extend: 2, 3, 4 all present. Run length 4.
  • Is 3 a run start? 2 in set — skip. Is 2 a run start? 1 in set — skip.
  • Maximum is 4. Total work: 4 starts + 4 extensions = 8 operations on 6 elements. O(n).

#680 · Valid Palindrome II

EasyTwo PointersGreedy

The Problem

Given a string s, return true if it can be a palindrome after deleting at most one character. You may delete zero characters as well. The string consists of lowercase English letters. For instance, "abca" returns true (delete 'b' or 'c'), while "abc" returns false.

Examples:

  • "aba" returns true (already a palindrome).
  • "abca" returns true (delete 'c' to get "aba").
  • "abc" returns false (cannot fix with one deletion).
  • "deeee" returns true (delete 'd' to get "eeee").

Brute Force

Try removing each character one at a time and test whether the remaining string reads the same forwards as backwards. With n possible deletions and an O(n) palindrome check each, the total is O(n^2). It works but does redundant work on mostly-matching prefixes and suffixes. Additionally, each deletion materializes a new string, adding allocation cost.

Python — Brute Force 
def validPalindrome(s: str) -> bool:
    if s == s[::-1]:
        return True
    for i in range(len(s)):
        t = s[:i] + s[i+1:]
        if t == t[::-1]:
            return True
    return False
Go — Brute Force 
func validPalindrome(s string) bool {
    isPal := func(t string) bool {
        for i, j := 0, len(t)-1; i < j; i, j = i+1, j-1 {
            if t[i] != t[j] { return false }
        }
        return true
    }
    if isPal(s) { return true }
    for i := 0; i < len(s); i++ {
        if isPal(s[:i] + s[i+1:]) { return true }
    }
    return false
}

Why optimize? O(n^2) repeatedly checks prefixes already known to match. A two-pointer sweep with one "skip" attempt is O(n). The brute force also allocates n substrings; the optimal allocates none.

Key Insight

Use two pointers from both ends. Skip while characters match. At the first mismatch, you have exactly two choices: delete the left character or delete the right character. Run a plain palindrome check on each of those two sub-ranges and return true if either succeeds. The matching prefix and suffix shrink the problem — we only ever inspect the "interesting" core where palindrome-ness could fail.

How to Think About It

  1. Set l = 0, r = len(s) - 1.
  2. While s[l] == s[r], advance l and retreat r.
  3. On mismatch, the only repairable structure is to skip one side — check s[l+1..r] or s[l..r-1].
  4. Use a helper that does a plain two-pointer palindrome check between two indices.
  5. Return true if either skip yields a palindrome; otherwise false.
  6. Edge case: empty string and single character are trivially true. Strings already palindromic require zero deletions (still allowed).
  7. Note we don't need to "try" deletions at non-mismatching positions — matching pairs never benefit from removal.

Why This Works

Once a mismatch is found between s[l] and s[r], at least one of them must be deleted for the result to be a palindrome (no other deletion can resolve this specific pair). Trying both options is exhaustive over the only feasible repairs. Because we are allowed at most one deletion, neither sub-check is allowed further deletions — they are simple two-pointer palindrome verifications.

⚠ Most Dangerous Mistake

Returning false at the first mismatch without trying BOTH skips. Some strings require skipping the right side ("abca" — skip 'c'), others the left. You cannot decide which by inspection alone.

  • Reusing the outer pointers inside the skip-check — pass indices to a helper.
  • Using s[:i] + s[i+1:] inside the optimal version — defeats O(n) by allocating.
  • Returning true on the wrong base case for empty / single-character strings.
  • Off-by-one in the helper's stop condition (l < r vs l <= r).
  • Counting the original "no deletion" path as if it required a deletion — that case must succeed too.
  • Trying to "look ahead" two characters to predict which side to skip — doesn't generalize; always try both.

Worked Example

Trace s = "abca":

  • l=0, r=3: 'a' == 'a', advance to l=1, r=2.
  • l=1, r=2: 'b' != 'c' — first mismatch.
  • Try skip-left: check s[2..2] = "c" — palindrome. Return true.
  • Now trace s = "abc": same mismatch at (0, 2). Skip-left gives "c" (true), so result true? No: l=0,r=2 has 'a' != 'c'. Skip-left checks s[1..2] = "bc" — not palindrome. Skip-right checks s[0..1] = "ab" — not palindrome. Return false.

#5 · Longest Palindromic Substring

MediumStringExpand Around Center

The Problem

Given a string s, return the longest substring that reads the same forwards and backwards. If multiple substrings tie, any one is acceptable. The string can be up to 1000 characters. Note: substring means contiguous; subsequence is a different problem (#516).

Examples:

  • "babad" returns "bab" or "aba" (both length 3).
  • "cbbd" returns "bb".
  • "a" returns "a".
  • "ac" returns "a" or "c".

Brute Force

Try every substring (O(n^2) of them) and check whether each is a palindrome (O(n) per check). Track the longest. This is O(n^3) and times out on inputs of length ~500. The cubic cost comes from redundantly re-verifying overlapping substrings — an inefficiency that expand-around-center directly addresses.

Python — Brute Force 
def longestPalindrome(s: str) -> str:
    best = ""
    n = len(s)
    for i in range(n):
        for j in range(i, n):
            sub = s[i:j+1]
            if sub == sub[::-1] and len(sub) > len(best):
                best = sub

Key Insight

Every palindrome has a center. There are n single-character centers (odd-length palindromes) and n-1 between-character centers (even-length palindromes). For each center, expand outward while characters match. Track the maximum span seen. This reframes "find a palindrome" as "find the right center and grow outward" — replacing nested substring checks with a single radial scan.

How to Think About It

  1. For each index i, expand around (i, i) for odd-length palindromes.
  2. For each index i, expand around (i, i+1) for even-length palindromes.
  3. The expand function takes left, right pointers and pushes them outward while s[l] == s[r] and both in bounds.
  4. When the loop stops, the valid palindrome spans s[l+1 : r] — note the inclusive/exclusive correction.
  5. Track and return the longest span found.
  6. Avoid re-slicing on every update — store best start and length as integers, slice once at the end.
  7. For very long inputs where O(n^2) is too slow, Manacher's algorithm gives O(n) — rarely required in interviews.

Why This Works

Any palindromic substring is uniquely identified by its center and radius. By trying all 2n-1 possible centers and pushing each to its maximum radius, we enumerate every candidate without overlap. Each expansion is at most O(n), so total is O(n^2). Compared to the brute force's O(n^3), this saves one factor of n because we no longer re-verify substrings we have already implicitly extended — each expansion step builds on the previous one's success.

⚠ Most Dangerous Mistake

Forgetting the even-length case. If you only expand from single characters, "abba" returns "a" or "b" — never "abba", because its center sits BETWEEN two characters.

  • Off-by-one when extracting the span after the loop — the final l, r point one past the matching window.
  • Forgetting bounds checks l >= 0 and r < n inside the expansion.
  • Comparing characters via == on slice values vs byte values in Go — use s[l] == s[r] on byte indices.
  • Updating best with the wrong span (use r - l - 1 after expansion stops).
  • Allocating substrings every iteration to compare lengths — track (start, length) ints and slice once at the end.
  • Confusing palindromic substring with palindromic subsequence (#516) — subsequence is non-contiguous.

#647 · Palindromic Substrings

MediumStringExpand Around Center

The Problem

Given a string s, return the total number of palindromic substrings in it. A substring is a contiguous sequence; substrings starting at different positions are counted separately even if they look identical. For example "aaa" has 6 palindromes: "a", "a", "a", "aa", "aa", "aaa".

Examples:

  • "abc" returns 3 (each single character).
  • "aaa" returns 6.
  • "aaaa" returns 10 (4 singles + 3 doubles + 2 triples + 1 quad = 10).
  • "abba" returns 6 (4 singles + "bb" + "abba").

Brute Force

Enumerate every substring with two nested loops, check each in O(n), increment a counter on success. This is O(n^3) and identical in spirit to the brute force for #5. Once you have the expand-around-center solution for #5, this problem is a one-line change — replace "track max length" with "increment counter on every match".

Why optimize? Cubic time is unnecessary — the expand-around-center pattern visits each center once and counts every palindrome it produces. Same engine as #5 with only the bookkeeping changed.

Key Insight

Each successful step of an expand-around-center walk corresponds to exactly one palindromic substring. So instead of tracking the longest, simply increment a counter for every successful expansion step. Iterate over all 2n - 1 centers. The bijection between (center, radius) pairs and palindromes makes the count straightforward — no deduplication required.

How to Think About It

  1. Initialize count = 0.
  2. For each index i, expand around (i, i): each match adds 1 to count.
  3. For each index i, expand around (i, i+1): each match adds 1.
  4. Stop expanding when pointers go out of bounds or characters differ.
  5. Return count.
  6. Sanity check: a string of n identical characters has n*(n+1)/2 palindromic substrings — the maximum possible.
  7. Alternative DP formulation: dp[i][j] is true iff s[i..j] is a palindrome, with recurrence dp[i][j] = (s[i]==s[j]) && dp[i+1][j-1].

Why This Works

Every palindrome has a unique center (one of the 2n-1 candidates). For each center, expanding outward yields one new palindrome per step that succeeds. Summing across all centers counts every palindrome exactly once. The center-radius parameterization establishes a bijection between (center, radius) pairs and palindromic substrings, so no palindrome is missed and none is double-counted.

⚠ Most Dangerous Mistake

Counting only odd-length palindromes (one center per index) and missing the even-length ones (between-index centers). Double up: 2n-1 centers, not n.

  • Counting the same palindrome twice by also iterating substrings outside the expand pattern.
  • Forgetting the bounds checks l >= 0 and r < n.
  • Confusing this with subsequence counting (different problem; needs DP).
  • Returning the longest length instead of the count (don't paste #5 verbatim).
  • Adding 1 only for the outer call but not for each successful expansion step.
  • Trying to enumerate only "maximal" palindromes — the problem wants all of them, including all the smaller ones contained inside.
