Move Zeroes

Easy

The Problem: Given an integer array nums, move all 0s to the end of it while maintaining the relative order of the non-zero elements. You must do this in-place without making a copy of the array. The operation should minimize the total number of writes.
Quick Reminder: Use two pointers — a slow pointer marks where the next non-zero should go, and a fast pointer scans through the array. Swap non-zero elements into position, then the zeros naturally end up at the tail.
Key Insight: The slow pointer always points to the leftmost zero (or the next available slot). Every time the fast pointer finds a non-zero element, swapping it with the slow pointer position guarantees order preservation and minimizes operations.

The Problem: Given two strings s and t, return true if they are equal when both are typed into empty text editors. The character '#' represents a backspace that deletes the previous character. Note that after backspacing an empty text, the text will remain empty.
Quick Reminder: For O(1) space, process both strings from right to left simultaneously. Track the backspace count; when you encounter a #, increment the skip counter. When you encounter a letter with skips remaining, consume a skip. Compare the next "real" characters from each string.

Key Insight: A max-heap gives O(log n) access to the two largest elements. Python only has a min-heap (heapq), so negate all values to simulate a max-heap. Pop two, compute the difference, and push it back if non-zero. Languages like C++ and Rust have native max-heap support.
How to Think About It:
  1. Build a max-heap from all stone weights. In Python, negate values and use heapq.heapify(). In C++, use priority_queue. In Rust, use BinaryHeap.
  2. While the heap has 2 or more stones: pop the two largest with the heap's extract-max operation
  3. If the two values differ, push the difference back onto the heap
  4. When the loop ends, return the remaining stone, or 0 if the heap is empty

Why optimize? We only need the k-th largest, not a fully sorted array. A min-heap of size k keeps exactly what we need. The root of a size-k min-heap IS the k-th largest element. Each add becomes O(log k) instead of O(n log n).

Key Insight: Maintain a min-heap of exactly size k. The root (smallest element in the heap) is the k-th largest overall. When a new value arrives, only add it if it's larger than the current k-th largest (the root), then pop the root to maintain size k. This way, the heap always contains the k largest elements seen so far.
How to Think About It:
  1. In the constructor: heapify nums. While heap size exceeds k, pop the smallest element to trim down to size k
  2. In add(val): push val onto the heap
  3. If heap size exceeds k, pop the smallest (this evicts the (k+1)-th largest)
  4. Return the heap root — the root of a min-heap of size k is the k-th largest
Why This Works: A min-heap of size k always contains the k largest elements. The smallest among those k elements (the heap root) is by definition the k-th largest. When we add a new element larger than the root, it displaces the current k-th largest, and the new root becomes the updated k-th largest. Elements smaller than the root are irrelevant and never enter the heap.
Watch Out For:
  • The initial nums array might have fewer than k elements. Handle this by simply pushing and only popping when size exceeds k
  • This is a min-heap (no negation needed!) — unlike #1046, we want the smallest of the top-k, which is the natural min-heap root
  • In Python, heapq.heappushpop() is more efficient than a separate push + pop
  • In Rust, BinaryHeap is a max-heap by default — wrap with Reverse for a min-heap
  • In C++, use greater<int> as the third template parameter for a min-heap

        let i = this.data.length - 1;
        while (i > 0) {
            const parent = (i - 1) >> 1;

Complexity: Time O(total_time × 26) where total_time can be up to ~104. Space O(1) (fixed 26-letter arrays).

Why optimize? While the simulation works, there is a direct O(n) mathematical formula. The simulation approach is harder to reason about for correctness and is slower for large inputs. The formula also reveals the structural insight about why the schedule length is what it is.

Key Insight: The most frequent task creates a "frame" structure. If the most frequent task appears f times, you need (f - 1) gaps of size (n + 1) between its occurrences, plus one final group for tasks that share the maximum frequency. The answer is max((f - 1) * (n + 1) + count_of_max_freq_tasks, len(tasks)). The max with len(tasks) handles the case where n is small and there are enough distinct tasks to fill all gaps with no idle time.
How to Think About It:
  1. Count the frequency of each task using a counter or frequency array
  2. Find the maximum frequency max_freq and count how many tasks share that maximum frequency (max_count)
  3. Calculate the frame-based formula: (max_freq - 1) * (n + 1) + max_count. This represents (max_freq - 1) full chunks of size (n + 1), plus the final partial chunk containing only the tasks with max frequency
  4. Return max(formula_result, len(tasks)) because the answer can never be less than the total number of tasks (when n is small enough that no idle time is needed)
Why This Works: Imagine laying out the most frequent task with mandatory gaps: A _ _ A _ _ A (for freq=3, n=2). The gaps have size n. Other tasks fill these gaps. If gaps are fully filled, no idle time is needed and the answer is just len(tasks). If gaps remain unfilled, idle slots exist and the formula accounts for them. The max_count term handles the last position where multiple tasks with max frequency each need one more slot.
Watch Out For:
  • Forgetting the max(formula, len(tasks)) — when n=0 or there are many distinct tasks, the answer is simply len(tasks) with no idle time
  • Off-by-one in the formula: it's (max_freq - 1) gaps, not max_freq, because the last occurrence doesn't need a gap after it
  • The chunk size is (n + 1), not n, because the chunk includes the task itself plus n cooldown slots
  • max_count counts tasks with the exact maximum frequency, not tasks with frequency > 0

Alternative approach: Max-heap + cooldown queue simulation. Use a max-heap of frequencies. Each round: pop the most frequent, decrement it, and put it in a cooldown queue with its available time. When a task's cooldown expires, push it back onto the heap. This is O(n log 26) = O(n) and more intuitive for some, but the math formula is O(n) with a smaller constant.

Complexity: Time O(n · 2ⁿ) — 2ⁿ subsets, each up to length n. Space O(n · 2ⁿ) to store all subsets.

Why optimize? The iterative approach works but doesn’t teach the backtracking pattern. The recursive version introduces choose/explore/unchoose, which generalizes to permutations, combinations, and constraint satisfaction problems.

Key Insight: At each element, you make a binary choice: include it or skip it. This is the fundamental backtracking template. Build the result by exploring both branches at each index. Every node in the recursion tree is a valid subset — not just the leaves.
How to Think About It:
  1. Start with an empty path and index 0. The path represents the subset being built.
  2. At each recursive call, add a copy of the current path to the result — every partial path is a valid subset.
  3. For each remaining element from start to len(nums)-1: add the element to the path (choose), recurse with start = i + 1 (explore), remove the element (unchoose/backtrack).
  4. The loop starting from start (not 0) ensures we never revisit earlier elements, preventing duplicate subsets like [1,2] and [2,1].
Why This Works: The recursion tree has 2ⁿ leaves, one for each possible include/exclude combination. By iterating from start forward and recording at every node, we enumerate every subset exactly once. The backtrack (pop) restores state so sibling branches explore independent choices.
Watch Out For:
  • Forgetting to copy the path when adding to the result — result.append(path) appends a reference, so all entries end up pointing to the same (eventually empty) list. Use path[:] or list(path).
  • Starting the inner loop from 0 instead of start — this generates permutations of subsets, creating duplicates.
  • In Go, appending a slice without copying — later mutations can corrupt previously stored results because slices share underlying arrays.
  • In Rust, forgetting to .clone() the path before pushing to results — the borrow checker prevents moving owned data, but cloning is required for correct snapshots.

Alternative approaches: Bit manipulation — iterate from 0 to 2ⁿ−1. Each integer is a bitmask where bit j indicates whether nums[j] is included. This avoids recursion entirely and is very clean for small n.

#46 Permutations Medium Backtracking

The Problem: Given an array nums of distinct integers, return all possible permutations in any order. Constraints: 1 ≤ nums.length ≤ 6, -10 ≤ nums[i] ≤ 10, all integers are unique.

Brute Force — O(n! · n) time

Generate all n! arrangements by trying every unused element at each position. This IS the standard approach — there is no simpler brute force because generating permutations inherently requires exploring all orderings.

Python 
class Solution:
    def permute(self, nums: list[int]) -> list[list[int]]:
        result = []
        def backtrack(path, used):
            if len(path) == len(nums):
                result.append(path[:])
                return
            for i in range(len(nums)):
                if i not in used:
                    used.add(i)
                    path.append(nums[i])
                    backtrack(path, used)
                    path.pop()
                    used.remove(i)
        backtrack([], set())
        return result

Why optimize? This already IS optimal for generating all permutations. The in-place swap approach below eliminates the “used” set but has the same asymptotic complexity. The value is in learning the pattern, not reducing Big-O.

Key Insight: Unlike subsets where you choose include/exclude, permutations require using every element exactly once. Track which elements are “used” via a boolean array or set. The key structural difference: iterate over ALL elements (not from an index onward) but skip the ones already used.
How to Think About It:
  1. Maintain a used set (or boolean array) and a current permutation path.
  2. If path length equals nums length, we have a complete permutation — add a copy to the results.
  3. For each element in nums: if not used, mark it used, add to path, recurse, remove from path, unmark. This is choose/explore/unchoose.
  4. The key difference from Subsets: iterate ALL elements (index 0 to n−1) but skip used ones, rather than iterating from start onward. This allows the same element to appear at different positions across different branches.
Why This Works: Each level of recursion fills one position in the permutation. At level 0, there are n choices. At level 1, n−1 remaining choices. The total number of leaves is n · (n−1) · … · 1 = n!, and each leaf is a unique permutation because the “used” set guarantees no element is repeated.
Watch Out For:
  • Using a set for the path itself (loses insertion order — permutations are ordered sequences, not sets).
  • Not copying the path before appending to the result — same reference bug as in Subsets.
  • The swap approach requires careful index management: swap nums[start] with nums[i], recurse with start+1, then swap back. Forgetting to swap back corrupts later branches.

Alternative approaches: In-place swap approach — instead of a separate used set, swap nums[i] with nums[start], recurse with start+1, swap back. This uses the array itself as the state tracker and avoids extra space for the “used” structure.

#79 Word Search Medium Backtracking Matrix

Complexity: Time O(m · n · 3ᴸ) where L = word length. At each cell, we branch into at most 3 directions (can’t go back the way we came). Space O(L) for recursion depth.

Why optimize? This is already the optimal approach. Minor optimizations include early termination (character frequency check before DFS) and starting from the rarer end of the word, but the asymptotic complexity is the same.

Key Insight: The grid itself serves as the “visited” tracker — temporarily mark a cell (e.g., overwrite with '#') to prevent revisiting during the current path, then restore it on backtrack. No separate visited set is needed. This is the choose/explore/unchoose pattern applied to a 2D grid.
How to Think About It:
  1. For each cell (r, c) that matches word[0], start a DFS.
  2. DFS(r, c, k): if k == len(word), all characters matched — return True.
  3. If out of bounds OR board[r][c] != word[k], this path fails — return False.
  4. Save board[r][c] in a temp variable, then mark it as visited (set to '#'). This is the “choose” step.
  5. Recurse in all 4 directions with k+1. If ANY direction returns True, propagate True immediately (early termination).
  6. Restore board[r][c] from temp. This is the “unchoose” step — critical so other paths can reuse this cell.
Why This Works: Each DFS path spells out a candidate word character by character. The in-place marking ensures each cell is used at most once per path. Restoration after backtracking ensures the cell is available for other paths. Since we try every starting cell and every direction, we exhaustively check all possible word paths in the grid.
Watch Out For:
  • Not restoring the cell after backtracking — this breaks other search paths that need to use that cell.
  • Checking bounds after accessing board[r][c] — this causes an index-out-of-range error. Always check bounds first.
  • Not returning True immediately when found — continuing to search wastes time and can cause incorrect results if you overwrite the True with a later False.
  • Using 4ᴸ as the complexity instead of 3ᴸ — we never go back the way we came, so it’s 3 branches per level, not 4.

Complexity: Time O(n! · E) where E is number of prerequisites. Completely impractical for n > 10.

Why optimize? n! grows astronomically — 2000! is beyond computation. The real question is simply “does the directed graph have a cycle?” which DFS answers in O(V+E).

Key Insight: If the prerequisite graph has a cycle, it is impossible to complete all courses (circular dependency). Use DFS with three states — unvisited (0), in-progress (1), done (2) — to detect cycles. Encountering a node that is “in-progress” means you have found a back edge, which means a cycle exists.
How to Think About It:
  1. Build an adjacency list from the prerequisites: for each [a, b], add edge b → a (b must come before a).
  2. Create a state array for all courses initialized to 0 (unvisited).
  3. For each course, if unvisited, run DFS. This handles disconnected components.
  4. DFS(course): set state to 1 (in-progress). For each neighbor: if state is 1 → cycle found, return False. If state is 0 → recurse; if that returns False, propagate False.
  5. After exploring all neighbors, set state to 2 (done). Return True.
  6. If no cycle is detected from any starting node, return True.
Why This Works: Three-state DFS correctly identifies back edges — edges pointing to an ancestor in the current DFS path (state = in-progress). Back edges are the only indicator of cycles in directed graphs. Once a node is fully explored (state = done), all its descendants have been checked, so re-encountering it is a cross/forward edge, not a cycle. This is why two states (visited/unvisited) are insufficient — they cannot distinguish back edges from cross edges.
Watch Out For:
  • Using only two states (visited/unvisited) — this cannot distinguish a back edge (cycle) from a cross edge (no cycle) in directed graphs.
  • Forgetting to build an adjacency list — iterating the raw prerequisites array for each node’s neighbors is O(E) per node, making the total O(V · E) instead of O(V+E).
  • Not starting DFS from every unvisited node — the graph may be disconnected, and a cycle could exist in a component you never visited.
  • Edge direction: [a, b] means b is a prerequisite OF a, so the edge goes b → a, not a → b.

Alternative approaches: BFS topological sort (Kahn’s algorithm) — compute in-degree for every node, enqueue all nodes with in-degree 0, repeatedly dequeue and decrement neighbors’ in-degrees. If all nodes are processed, no cycle. This is iterative and avoids recursion stack limits.

Complexity: Time O(V + E) — visit every node and edge once. Space O(V) for the hash map and recursion stack.

Why optimize? This IS optimal. The only variation is BFS vs. DFS — same complexity, different traversal order. The learning value is in the pattern: using a hash map for both “visited” tracking and old→new mapping.

Key Insight: Use a hash map (old_node → new_node) that serves double duty: it is both a “visited” check AND a clone registry. When you encounter a node, if it is already in the map, return its clone. Otherwise, create the clone, add it to the map before recursing (this prevents infinite loops on cycles), then recursively clone all neighbors.
How to Think About It:
  1. Handle the null/nil input case — return null/nil.
  2. Create a hash map: old_node → cloned_node.
  3. DFS(node): if node is already in the map, return map[node] (already cloned).
  4. Create a new Node(node.val). Add it to the map immediately — before processing neighbors. This is crucial for handling cycles.
  5. For each neighbor of the original node, recursively call DFS(neighbor) and append the result to the clone’s neighbor list.
  6. Return the cloned node.
Why This Works: The hash map ensures each node is cloned exactly once. Adding the clone to the map before recursing into neighbors handles cycles: when DFS encounters an already-mapped node (via a cycle back edge), it returns the existing clone instead of recursing infinitely. Every edge is traversed once to connect cloned neighbors, producing a perfect structural replica.
Watch Out For:
  • Not adding to the map before recursing — if a graph has cycles (even just A↔B), the recursion will loop infinitely because neither node is marked as “seen” when the other tries to clone it.
  • Confusing node values with node identity — values are unique in this problem, but in general you should map by node reference, not value.
  • Forgetting the null input case — an empty graph should return null/nil, not crash.
  • In Rust, graph nodes require Rc<RefCell<..>> for shared mutable ownership — this is a common pattern for graph structures.

Alternative approaches: BFS with a queue — create the clone of the start node, add to map and queue. Process nodes level by level: for each neighbor of the current node, if not in the map, create its clone and enqueue it. Then connect the cloned neighbor to the current clone’s neighbor list. Same O(V+E) complexity.

Complexity: Time O(n²) — each city is visited once, and for each city we scan its row (n entries). Space O(n) for the visited array and queue.

Why optimize? The BFS/DFS approach is already O(n²) which is optimal (you must read the entire n×n matrix). The Union-Find alternative offers advantages for dynamic connectivity queries.

Key Insight: This is a connected components problem. Each province is one connected component. DFS/BFS from each unvisited city, marking all reachable cities as visited. Each time you start a new traversal from an unvisited city, you have found a new province.
How to Think About It:
  1. Create a visited boolean array of size n.
  2. Initialize a province counter to 0.
  3. For each city 0 to n−1: if not visited, increment the province counter and start a DFS/BFS from that city.
  4. DFS(city): mark city as visited. For each other city j: if isConnected[city][j] == 1 AND j is not visited, recurse on j.
  5. After processing all cities, return the province count.
Why This Works: DFS from a city visits its entire connected component — every city reachable through any chain of direct connections. Once the DFS finishes, all cities in that province are marked visited. Each new DFS start (from an unvisited city) discovers a new, previously unreachable component, so the counter accurately counts provinces.
Watch Out For:
  • Treating this as a grid problem — the input is an adjacency matrix, not a 2D grid. Indices represent cities, not (row, col) positions. Do not use 4-directional neighbors.
  • Confusing with #200 Number of Islands — same concept (count connected components) but different input format. Islands uses grid cells with 4-directional adjacency; Provinces uses an adjacency matrix.
  • Off-by-one with 0-indexed cities — cities are numbered 0 to n−1.
  • Forgetting that isConnected[i][i] = 1 — a city is always connected to itself. Make sure your DFS checks !visited[j] to avoid infinite self-loops.

Alternative approaches: Union-Find (Disjoint Set Union) — initialize each city as its own parent. For each pair where isConnected[i][j] == 1, union their sets. Count distinct roots at the end. With path compression and union by rank, nearly O(n² · α(n)) ≈ O(n²). Union-Find is more powerful for dynamic connectivity (adding edges over time).
