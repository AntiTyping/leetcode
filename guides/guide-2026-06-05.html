#160 · Intersection of Two Linked Lists

Easy Linked List Hash Table 14 days since last solve · last 2026-05-22

The Problem: Given the heads of two singly linked-lists headA and headB, return the node at which the two lists intersect. If the two linked lists have no intersection, return null. The lists must retain their original structure after the function returns.

Example 1: listA = [4,1,8,4,5], listB = [5,6,1,8,4,5], intersect at node with val 8 → node 8

Example 2: listA = [1,9,1,2,4], listB = [3,2,4], intersect at node with val 2 → node 2

Example 3: listA = [2,6,4], listB = [1,5], no intersection → null

Quick Reminder: Your solution added all nodes from list A to a set, then checked each node in list B against the set. This works in O(m+n) time but uses O(m) space. The optimal approach uses two pointers with zero extra space.
Key Insight: Two-pointer trick: pointer a starts at headA, pointer b starts at headB. When a reaches the end, redirect it to headB. When b reaches the end, redirect it to headA. They will meet at the intersection node (or both reach null simultaneously if no intersection) because both pointers travel exactly lenA + lenB steps.
Challenge: Your set-based approach uses O(m) space. Implement the O(1) space two-pointer approach where pointer a walks A then B, and pointer b walks B then A. They meet at the intersection because both travel the same total distance.
Your solution: O(m+n) time, O(m) space  |  Optimal: O(m+n) time, O(1) space

#209 · Minimum Size Subarray Sum

Medium Sliding Window Array 14 days since last solve · last 2026-05-22

The Problem: Given an array of positive integers nums and a positive integer target, return the minimal length of a subarray whose sum is greater than or equal to target. If there is no such subarray, return 0.

Example 1: target = 7, nums = [2,3,1,2,4,3] → 2 (subarray [4,3])

Example 2: target = 4, nums = [1,4,4] → 1 (subarray [4])

Example 3: target = 11, nums = [1,1,1,1,1,1,1,1] → 0

Quick Reminder: Your solution used prefix sums with two pointers — correct but more complex than needed. The optimal approach is a pure sliding window: maintain a running sum, expand right, and shrink left whenever sum ≥ target, tracking the minimum window length.
Key Insight: Classic sliding window on positive integers. Since all values are positive, expanding the window always increases the sum and shrinking always decreases it. This monotonicity means you never need to backtrack — just expand right and shrink left greedily.
Challenge: Your prefix-sum approach works but is more complex. Solve it in Go with a clean sliding window: one for loop expanding right, an inner for shrinking left when sum ≥ target.
Time: O(n)  |  Space: O(1)

class Solution(object):
    def minSubArrayLen(self, target, nums):
        p = [0] * len(nums)
        p[0] = nums[0]
        for i in range(1, len(nums)):
            p[i] = p[i - 1] + nums[i]

        min_l = len(nums) + 1
        i, j = 0, 0
        while i < len(nums) and j < len(nums):
            if i == 0:
                if p[j] >= target:
                    min_l = min(min_l, j - i + 1)
                    i += 1
                else:
                    j += 1
            else:
                if p[j] - p[i - 1] >= target:
                    min_l = min(min_l, j - i + 1)
                    i += 1
                else:
                    j += 1
        if min_l == len(nums) + 1:
            return 0
        else:
            return min_l

The Problem: A message containing letters A-Z can be encoded as numbers 1-26. Given a string s containing only digits, return the number of ways to decode it. For example, “12” can be decoded as “AB” (1 2) or “L” (12).

Example 1: s = "12" → 2 ("AB" or "L")

Quick Reminder: Your top-down memoized DFS solution works perfectly. At each index, you can take 1 digit (if non-zero) or 2 digits (if the pair ≤ 26). Base case: reaching end of string returns 1. A ‘0’ digit with no valid pair returns 0.

Challenge: Your top-down memoized solution uses O(n) space for the memo dict and recursion stack. Convert it to bottom-up DP with O(1) space using just two variables (dp1 and dp2).

class Solution(object):
    def numDecodings(self, s):
        memo = {}
        def dp(i):
            if i == len(s):
                return 1
            if s[i] == "0":
                return 0
            if i == len(s) - 1:
                return 1
            if i in memo:
                return memo[i]
            ans = dp(i + 1)
            if int(s[i:i + 2]) <= 26:
                ans += dp(i + 2)
            memo[i] = ans
            return ans
        return dp(0)

Key Insight: This is a connected components problem. BFS/DFS is the standard approach and already runs in O(m×n). The Union-Find alternative treats each '1' cell as a node and unions adjacent '1' cells. The number of distinct roots at the end equals the number of islands. Union-Find with path compression + union by rank achieves near O(m×n) with very small constant factors.

How to Think About It (Union-Find)

  1. Create a Union-Find structure with m × n nodes. Initialize count to the number of '1' cells.
  2. For each cell (r, c) that is '1', check its right neighbor (r, c+1) and bottom neighbor (r+1, c). If the neighbor is also '1', union them.
  3. Each successful union (where the two cells had different roots) decrements the count by 1.
  4. After processing all cells, the remaining count is the number of islands.
  5. Path compression: when calling find(x), make every node on the path point directly to the root. Union by rank: attach the shorter tree under the taller tree.
Why This Works: Every connected group of '1' cells will eventually share the same root through union operations. Since we only need to check right and down neighbors (the grid is symmetric — left and up would be redundant from already-processed cells), we cover all adjacencies in a single pass.
Watch Out For:
  • Forgetting to mark cells as visited (BFS/DFS): Without marking, you will revisit cells infinitely and loop forever or count islands multiple times.
  • 4-directional vs 8-directional: This problem uses 4-directional adjacency (up, down, left, right). Do NOT connect diagonals.
  • Union-Find indexing: Map 2D coordinates to 1D: id = r * cols + c. Do not forget to skip '0' cells.
  • Mutating the grid: BFS/DFS can mark visited cells by setting them to '0', avoiding extra space. But this modifies the input — check if that is acceptable.

Union-Find Solution (Python):

class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
        self.count = 0

    def add(self, x):
        self.count += 1

    def find(self, x):
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]  # path compression
            x = self.parent[x]
        return x

    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry:
            return
        if self.rank[rx] < self.rank[ry]:
            rx, ry = ry, rx
        self.parent[ry] = rx
        if self.rank[rx] == self.rank[ry]:
            self.rank[rx] += 1
        self.count -= 1

class Solution:

                    if c + 1 < cols and grid[r][c+1] == '1':

How to Think About It (Union-Find)

  1. Initialize Union-Find with n nodes. Set count to n (each city is its own province initially).
  2. Iterate i from 0 to n-1, and j from i+1 to n-1 (upper triangle only — the matrix is symmetric, so isConnected[i][j] == isConnected[j][i]).

Key Insight: This is a cycle detection problem in a directed graph. A valid course schedule exists if and only if the prerequisite graph is a DAG (no cycles). Two approaches: (1) BFS (Kahn’s algorithm): start with nodes that have indegree 0 (no prerequisites), remove them and their edges, repeat. If all nodes are removed, no cycle. (2) DFS with 3-color marking: white (unvisited), gray (in current DFS path), black (fully processed). A back edge to a gray node means a cycle.

How to Think About It

  1. Build an adjacency list from prerequisites. [a, b] means edge b→a (b must come before a).
  2. BFS (Kahn’s): Compute the indegree of every node. Put all indegree-0 nodes in a queue.
  3. Process the queue: dequeue a node, decrement the indegree of all its neighbors. If a neighbor’s indegree becomes 0, enqueue it. Count processed nodes.
  4. If processed count equals numCourses, all courses can be taken (no cycle). Otherwise, a cycle exists.
  5. DFS alternative: Run DFS from each unvisited node. Mark nodes gray on entry, black on exit. If DFS reaches a gray node, there’s a cycle.
Why This Works: Kahn’s algorithm peels off nodes with no remaining dependencies layer by layer. If a cycle exists, the nodes in the cycle will always have indegree ≥ 1 and will never be enqueued, so the processed count falls short of numCourses. The DFS approach detects cycles directly: a back edge (reaching a node still on the current DFS stack = gray) proves a cycle exists.
Watch Out For:
  • Prerequisite direction: [a, b] means b→a, NOT a→b. Getting this backwards gives wrong indegrees and wrong results.
  • Disconnected graph: Not all courses may be reachable from one starting point. Kahn’s handles this naturally (initializes all indegree-0 nodes). DFS must iterate over all nodes.
  • DFS without memoization: Without the 3-color scheme (or a “safe” set), you may revisit nodes that are already confirmed cycle-free, wasting time.

How to Think About It

  1. Build adjacency list and indegree array (same as #207).
  2. Initialize a queue with all indegree-0 nodes and an empty result list.
  3. Process the queue: dequeue a node, append it to result, decrement neighbors’ indegrees. Enqueue neighbors that reach indegree 0.
  4. If result.length == numCourses, return result. Otherwise return [] (cycle detected).
  5. DFS alternative: Same 3-color DFS as #207. Append nodes to result in post-order, then reverse. The reversal is needed because DFS finishes leaf-level courses first, but they should appear last in the schedule.

  • DFS: forgetting to reverse: DFS post-order gives REVERSE topological order. You must reverse the result at the end.
  • Returning wrong empty value: Return [] (empty list/array) when a cycle is found, not [-1] or None.
  • Edge direction: Same as #207 — [a, b] means edge b→a. Build the graph accordingly.
  • Disconnected components: Some courses may have no prerequisites and no dependents. Kahn’s handles them (they start with indegree 0). DFS must visit all nodes.

#236 Lowest Common Ancestor of a Binary Tree M Tree

The Problem: Given the root of a binary tree and two nodes p and q, return their lowest common ancestor (LCA). The LCA is the deepest node that is an ancestor of both p and q (a node can be an ancestor of itself).

Key Insight: Use post-order DFS. If the current node is p or q, return it. Recursively search left and right subtrees. If both return non-null, the current node is the LCA. Otherwise, propagate whichever non-null result upward. This works because the first node where both subtrees “report back” a target is exactly the LCA.

How to Think About It

  1. Base case: if root is null, or root == p, or root == q, return root. This catches both leaf-nulls and target-found cases.
  2. Recurse into left = LCA(root.left, p, q) and right = LCA(root.right, p, q).
  3. If both left and right are non-null, then p and q are on opposite sides — root is the LCA. Return root.
  4. If only one side is non-null, both targets are on that side — propagate that result up.
  5. The answer naturally bubbles up through the recursion. No extra data structures needed.
Brute Force Approach

Find the full path from root to p and from root to q using DFS. Then compare the two paths element-by-element; the last common node is the LCA. Time: O(n), Space: O(n) for storing paths.

Why This Works

The post-order DFS visits children before making a decision at the current node. When a node receives non-null results from both its left and right subtrees, it means p is in one subtree and q is in the other — making it the deepest split point, i.e., the LCA. If only one side returns non-null, both targets are in that subtree, and the result propagates upward unchanged. The recursion naturally terminates because we return immediately upon finding p or q.

Watch Out For:
  • A node can be its own ancestor: If p is an ancestor of q, then p is the LCA. The base case root == p handles this — we return immediately without searching deeper, which is correct because q must be in p’s subtree.
  • Not handling p == q: If both targets are the same node, the function correctly returns that node from the base case.
  • Confusing BST LCA with general tree LCA: In a BST you can use value comparisons to go left/right. Here, without ordering, you must search both subtrees.

Language Features That Help

  • Python: or short-circuit — return left or right elegantly returns the non-null result (or null if both are null).

  • Rust: Rc<RefCell<TreeNode>> makes this verbose. Use Rc::clone for shared ownership and .borrow() for access. Consider comparing values instead of references.

#297 Serialize and Deserialize Binary Tree H Tree

The Problem: Design an algorithm to serialize a binary tree to a string and deserialize that string back to the original tree. There is no restriction on how the serialization/deserialization algorithm should work — you just need to ensure a tree can be serialized to a string and this string can be deserialized to the original tree structure.

Key Insight: Use preorder traversal with null markers. Serialize: visit root, then left, then right, writing “null” for None nodes. Deserialize: read values one by one, recursively building left subtree then right subtree. The null markers tell you exactly when a subtree is empty, so the structure is unambiguous.

How to Think About It

  1. Choose preorder (root, left, right) as the traversal order. This naturally puts the root first, making deserialization straightforward.
  2. Serialize: DFS from root. For each node, append its value. For each null child, append a sentinel like "null". Use a delimiter (comma) between tokens.
  3. Deserialize: Split the string by delimiter into a queue/iterator of tokens. Read the first token — if it’s "null", return None. Otherwise, create a node with that value, then recursively build its left child (next tokens) and right child (remaining tokens).
  4. The key insight for deserialization is that consuming tokens from a shared iterator/queue automatically advances the position, so left subtree deserialization consumes exactly the right number of tokens before right subtree begins.
  5. Alternative: BFS (level-order) serialization also works — serialize level by level with nulls, deserialize using a queue of parent nodes.
Brute Force Approach — BFS Level-Order

Use BFS to serialize level by level, including null markers. Deserialize by reading tokens and connecting children level by level using a queue. Same O(n) complexity but more code and tokens for sparse trees.

Why This Works

Preorder traversal with null markers creates a unique encoding for any binary tree. The root comes first, followed by the complete serialization of its left subtree, then the complete serialization of its right subtree. Null markers act as “stop signals” that terminate each subtree branch. During deserialization, consuming tokens from a shared iterator/queue means the left subtree construction automatically consumes exactly the right number of tokens, leaving the remaining tokens for the right subtree. This one-to-one correspondence between tree structure and token sequence guarantees correctness.

Watch Out For:
  • Null marker handling: Every null child must be recorded. Forgetting nulls for leaf nodes makes the encoding ambiguous — you can’t tell where left subtree ends and right subtree begins.
  • Delimiter choice: Use a character that cannot appear in node values (comma works since values are integers). If values could contain your delimiter, you need escaping.
  • Negative numbers: Integer parsing must handle the minus sign. int("-5") works in Python, but be careful with custom parsers.
  • Shared iterator in deserialization: The iterator/index must be shared across recursive calls (not reset). In Python, use iter() with next(); in other languages, use a mutable index or deque.

Language Features That Help

  • Python: iter() + next() — create an iterator from the token list and pass it through recursion. Each next() call advances the shared position automatically.
  • Go: Use a slice index passed by pointer (*int) or use a closure that captures a mutable index variable for the shared deserialization position.
  • TypeScript: Use an object { index: 0 } passed by reference, or shift from an array. Array.shift() gives clean queue-like token consumption.
  • Rust: VecDeque with pop_front() works well. Alternatively, use a slice index with &mut usize for zero-allocation token consumption.
  • C++: stringstream with getline(ss, token, ',') for parsing. Or split into a queue<string> first, then pop from front during deserialization.

Intervals

Interval problems test your ability to think about overlapping ranges. The core technique is always the same: sort by start time, then scan linearly. These two problems — merging all overlapping intervals and inserting a new interval — form the complete interval manipulation toolkit.

#56 Merge Intervals M Intervals

The Problem: Given an array of intervals where intervals[i] = [start_i, end_i], merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

Key Insight: Sort intervals by start time. Then iterate: if the current interval overlaps with the last merged interval (i.e., current.start <= last_merged.end), extend the last merged interval’s end to max(last_merged.end, current.end). Otherwise, start a new merged interval. Sorting ensures you only need to compare with the most recent merged interval.

How to Think About It

  1. Sort the intervals by their start time. This guarantees that any interval that could overlap with the current merged interval comes next in sequence.
  2. Initialize a result list with the first interval.
  3. For each subsequent interval, compare its start with the end of the last interval in the result.
  4. If current.start <= result[-1].end, they overlap — update result[-1].end = max(result[-1].end, current.end). Note: use <= not < because [1,3] and [3,5] touch and should merge to [1,5].
  5. If no overlap, append the current interval to the result as a new group.
Brute Force Approach

For each pair of intervals, check if they overlap and merge them. Repeat until no more merges are possible. Time: O(n²) in the worst case due to repeated scanning. Space: O(n).

class Solution:
    def merge(self, intervals: list[list[int]]) -> list[list[int]]:
        merged = [iv[:] for iv in intervals]  # copy
        changed = True
        while changed:
            changed = False
            new_merged = []
            used = [False] * len(merged)
            for i in range(len(merged)):
                if used[i]:
                    continue
                cur = merged[i][:]
                for j in range(i + 1, len(merged)):
                    if used[j]:
                        continue
                    if cur[0] <= merged[j][1] and merged[j][0] <= cur[1]:
                        cur[0] = min(cur[0], merged[j][0])
                        cur[1] = max(cur[1], merged[j][1])
                        used[j] = True
                        changed = True
                new_merged.append(cur)
            merged = new_merged
        return merged

Why This Works

After sorting by start time, intervals that could overlap are adjacent. When we scan left to right, the last interval in our result list has the largest start time seen so far. If the next interval’s start is within the last interval’s range, they overlap and we extend. If not, there’s a gap and we start a new group. The max on the end is critical — consider [1,10], [2,3] where the second interval is fully contained. Taking max(10, 3) = 10 preserves the longer range.

Watch Out For:
  • Not sorting first: Without sorting, you’d need O(n²) pairwise comparison. Sorting is the key enabler of the linear scan.
  • Using > instead of >= for overlap: Intervals [1,4] and [4,5] touch at point 4 and should merge to [1,5]. Use <= for the overlap check.
  • Not using max for the end: When merging, don’t just take the new interval’s end — use max(last.end, current.end) to handle contained intervals.

Language Features That Help

  • Python: intervals.sort(key=lambda x: x[0]) for in-place sort by start. result[-1] for easy access to the last element.
  • Go: sort.Slice(intervals, func(i, j int) bool { return intervals[i][0] < intervals[j][0] }) for custom sorting.
  • TypeScript: intervals.sort((a, b) => a[0] - b[0]) for numeric sorting. Remember JS sort is lexicographic by default.
  • Rust: intervals.sort_by_key(|iv| iv[0]) for clean sorting. .last_mut().unwrap() to modify the last element in place.
  • C++: sort(intervals.begin(), intervals.end()) works directly since vectors compare lexicographically. result.back() for the last element.

#57 Insert Interval M Intervals

The Problem: You are given an array of non-overlapping intervals intervals sorted by start time and a newInterval. Insert newInterval into intervals such that the result is still sorted and non-overlapping (merge if necessary). Return the resulting array.

Key Insight: Three-phase linear scan. Phase 1: add all intervals that end before the new interval starts (no overlap). Phase 2: merge all intervals that overlap with the new interval by extending the new interval’s bounds. Phase 3: add all remaining intervals that start after the new interval ends. One pass, O(n).

How to Think About It

  1. Initialize result list. Let [newStart, newEnd] be the new interval.
  2. Phase 1: While the current interval’s end < newStart, it’s entirely before the new interval. Add it to result.
  3. Phase 2: While the current interval’s start <= newEnd, it overlaps. Merge: newStart = min(newStart, current.start), newEnd = max(newEnd, current.end).
  4. After Phase 2, add the merged [newStart, newEnd] to result.
  5. Phase 3: Add all remaining intervals (they’re entirely after the merged interval).
  6. Alternative shortcut: append newInterval to intervals, then call Merge Intervals (#56). Works but is O(n log n) instead of O(n) since the input is already sorted.
Brute Force Approach

Append the new interval to the list, sort, and then apply merge intervals (#56). Simple but O(n log n) due to the unnecessary sort (the input is already sorted).

Why This Works

Because the input intervals are already sorted and non-overlapping, the three phases cleanly partition them. Phase 1 intervals cannot overlap with the new interval (they end before it starts). Phase 2 intervals all overlap and are greedily merged by expanding the new interval’s bounds. Phase 3 intervals cannot overlap (they start after the merged interval ends). The sorted order guarantees these three groups are contiguous in the array, so a single left-to-right scan suffices.

Watch Out For:
  • Off-by-one in overlap detection: Phase 1 condition is current.end < newStart (strict), not <=. If current.end == newStart, they touch and should merge.
  • Forgetting to take max of ends when merging: The new interval might be contained within an existing interval, so always use max.
  • Edge cases: New interval before all existing, new interval after all existing, new interval containing all existing, empty intervals list.

Monotonic Queue

A monotonic deque maintains elements in sorted order within a sliding window, giving O(1) access to the min or max. This is the go-to technique when you need the extreme value in every window of size k.

#239 Sliding Window Maximum H Monotonic Queue

The Problem: You are given an array of integers nums and a sliding window of size k which moves from the very left to the very right. You can only see the k numbers in the window. Return the max value in each window position.

Key Insight: Maintain a monotonic decreasing deque of indices. The front of the deque always holds the index of the maximum element in the current window. When adding a new element, remove all indices from the back whose corresponding values are smaller (they can never be the maximum while the new element is in the window). Remove from the front if the index is outside the window. This gives O(1) amortized per element.

How to Think About It

  1. Create a deque that stores indices (not values). We store indices so we can check if an element has fallen out of the window.
  2. For each new element at index i: first, remove the front of the deque if deque[0] <= i - k (it’s outside the window).
  3. Then, remove all indices from the back of the deque where nums[deque[-1]] <= nums[i]. These elements are smaller than the new element and are to its left — they can never be the maximum while nums[i] is in the window.
  4. Push index i to the back of the deque.
  5. Once i >= k - 1 (the first full window), the front of the deque is the maximum: append nums[deque[0]] to the result.
  6. Each index is pushed and popped at most once, so total work is O(n).
Brute Force Approach

For each window position, scan all k elements to find the maximum. Time: O(nk), Space: O(n-k+1) for the result.

Why This Works

The deque maintains a decreasing sequence of “candidates” for the window maximum. The front is always the current max. When a new element arrives that’s larger than elements at the back, those back elements can never be the max for any future window (the new element entered later and is larger), so we safely discard them. Each element enters and leaves the deque at most once, giving O(n) total operations despite the nested-looking while loop. The deque acts as a “lazy sorted structure” that discards provably useless elements.

Watch Out For:
  • Storing indices, not values: You must store indices in the deque to check if the front element has left the window (deque[0] <= i - k). If you store values, you can’t determine window boundaries.
  • Removal order: Remove from the front first (expired elements), then remove from the back (smaller elements), then push. Getting this order wrong can cause incorrect results.
  • Using < vs <= when removing from back: Use <= (remove equal elements too) to avoid duplicates lingering. If you use <, equal elements pile up but don’t affect correctness — just waste space.
  • Off-by-one on when to start recording: The first valid window is at index k-1. Start appending results from i >= k - 1.

Language Features That Help

  • Python: collections.deque — O(1) append(), appendleft(), pop(), popleft(). Perfect for this pattern. Access front with dq[0], back with dq[-1].
  • Go: No built-in deque — use a slice as a deque. Pop front with dq = dq[1:], pop back with dq = dq[:len(dq)-1]. This is O(1) amortized with Go’s slice mechanics.
  • TypeScript: Use an array with shift() for pop-front and pop() for pop-back. Note: shift() is O(n) for arrays, but in practice it’s fast enough. For true O(1), implement a circular buffer.
  • Rust: std::collections::VecDeque — O(1) push_back(), pop_back(), push_front(), pop_front(). Access front with dq.front(), back with dq.back().
  • C++: std::deque<int> — O(1) operations on both ends. dq.front(), dq.back(), dq.push_back(), dq.pop_front(), dq.pop_back().
Brute: O(nk) time → Optimal: O(n) time, O(k) space

Connection: The monotonic deque pattern generalizes to any “sliding window min/max” problem. It also appears in #862 (Shortest Subarray with Sum at Least K) with a monotonic deque on prefix sums. The key mental model — “discard elements that can never win” — is the same principle behind monotonic stacks (#84 Largest Rectangle in Histogram, #42 Trapping Rain Water).

Backtracking

Backtracking is systematic trial-and-error: make a choice, explore, undo the choice, try the next option. N-Queens is the canonical backtracking problem — it teaches you how to prune invalid branches using constraint tracking.

#51 N-Queens H Backtracking

The Problem: Place n queens on an n × n chessboard such that no two queens attack each other (no two queens share the same row, column, diagonal, or anti-diagonal). Return all distinct solutions, where each solution is a board configuration represented as a list of strings.

Key Insight: Place queens column by column (or row by row). For each column, try each row. Track which rows, diagonals (row - col), and anti-diagonals (row + col) are occupied using sets. A placement is valid if the row, diagonal, and anti-diagonal are all free. Backtrack by removing the queen and its constraints when exploring the next option.

How to Think About It

  1. Since each column must have exactly one queen, iterate column by column (col 0, 1, ..., n-1). This eliminates column conflicts by construction.
  2. For each column, try placing a queen in each row (0 to n-1). Check three constraint sets: rows, diags (row - col), anti_diags (row + col).
  3. If the row, diagonal, and anti-diagonal are all free, place the queen: add to all three sets, record the placement, recurse to the next column.
  4. After recursion returns, backtrack: remove the queen from all three sets.
  5. Base case: when col == n, all queens are placed. Convert the board to the required format and add to results.
  6. The diagonal trick: two cells (r1, c1) and (r2, c2) are on the same diagonal iff r1 - c1 == r2 - c2, and on the same anti-diagonal iff r1 + c1 == r2 + c2.
Brute Force Approach

Generate all possible placements of n queens on n² squares and check each for validity. This is C(n², n) combinations — astronomically slow. A smarter brute force: try all n^n row assignments (one queen per column) and validate each. Still exponential but more tractable.

from itertools import permutations

class Solution:
    def solveNQueens(self, n: int) -> list[list[str]]:
        results = []
        # Try all permutations of rows (one queen per column, one per row)
        for perm in permutations(range(n)):
            # Check diagonals
            valid = True
            for i in range(n):
                for j in range(i + 1, n):
                    if abs(perm[i] - perm[j]) == abs(i - j):
                        valid = False
                        break
                if not valid:
                    break
            if valid:
                board = []
                for row in perm:
                    board.append("." * row + "Q" + "." * (n - row - 1))
                results.append(board)
        return results

            for (int i = start; i < n; i++) {

Why This Works

The backtracking with constraint sets prunes the search tree dramatically. By placing queens column by column, we ensure no two queens share a column. The row set prevents row conflicts. The diagonal set (row - col) uniquely identifies each “\” diagonal, and the anti-diagonal set (row + col) uniquely identifies each “/” diagonal. At each step, we only try rows that pass all three constraints, skipping entire subtrees of invalid placements. For n=8, this explores roughly 15,000 nodes instead of 8^8 = 16 million.

Watch Out For:
  • Diagonal math: row - col can be negative (e.g., row=0, col=3 gives -3). This is fine for set membership, but if using an array, add an offset of n-1 so indices are non-negative.
  • Forgetting anti-diagonals: Three constraints, not two. Rows and diagonals alone won’t prevent all attacks — you need anti-diagonals (row + col) as well.
  • Board representation: The output format is a list of strings where each string is a row. Build it at the end from the queen positions, not during backtracking.
  • Backtracking cleanup: Always remove from all three constraint sets when backtracking, not just one.

Language Features That Help

  • Python: set() for O(1) add/remove/lookup on rows, diags, anti-diags. String multiplication "." * n and slicing for board construction.
  • Go: map[int]bool for constraint sets. strings.Builder or byte slice for building board strings efficiently.
  • TypeScript: Set<number> for constraint tracking. ".".repeat(n) for string building. Arrow functions for clean recursive lambdas.
  • Rust: HashSet<i32> for constraints (use i32 since row - col can be negative). String::from(".").repeat(n) for board rows.
  • C++: unordered_set<int> for constraint sets. string(n, '.') constructor for board rows. Lambda with captures for clean backtracking.
Brute: O(n!) time (permutations) → Optimal: O(n!) time with heavy pruning, O(n) space

Connection: N-Queens is the template for all constraint-satisfaction backtracking. The same pattern applies to Sudoku Solver (#37), where you track row/column/box constraints with sets. The diagonal tracking technique (row ± col) is reusable in any grid problem involving diagonal relationships. Compare with #46 (Permutations) and #39 (Combination Sum) for simpler backtracking that builds up to this level.

Advanced DP

Edit Distance is a classic 2D dynamic programming problem that extends the LCS pattern. It requires defining a careful recurrence over two string indices and handling three operations (insert, delete, replace) at each mismatch.

#72 Edit Distance H DP

The Problem: Given two strings word1 and word2, return the minimum number of operations required to convert word1 into word2. You have three operations: insert a character, delete a character, or replace a character.

Key Insight: Define dp[i][j] = minimum operations to convert word1[:i] into word2[:j]. If characters match (word1[i-1] == word2[j-1]), no operation needed: dp[i][j] = dp[i-1][j-1]. Otherwise, take the minimum of three choices: insert (dp[i][j-1] + 1), delete (dp[i-1][j] + 1), or replace (dp[i-1][j-1] + 1).

How to Think About It

  1. Create a 2D table of size (m+1) × (n+1) where m = len(word1), n = len(word2).
  2. Base cases: dp[i][0] = i (delete all i characters from word1) and dp[0][j] = j (insert all j characters of word2).
  3. Fill row by row. For each cell dp[i][j], compare word1[i-1] and word2[j-1].
  4. If they match: dp[i][j] = dp[i-1][j-1] — no cost, carry forward the diagonal.
  5. If they differ: dp[i][j] = 1 + min(dp[i-1][j-1], dp[i-1][j], dp[i][j-1]). The three terms correspond to replace (change word1[i-1] to word2[j-1]), delete (remove word1[i-1]), and insert (add word2[j-1] after position i).
  6. Answer is dp[m][n]. Space optimization: only need two rows at a time (previous and current), reducing space to O(min(m,n)).
Brute Force Approach

Recursive approach: at each position, if characters match, move both pointers forward. If not, try all three operations recursively and take the minimum. Without memoization, this is O(3^(m+n)) due to overlapping subproblems.

Python Brute Force

class Solution:
    def minDistance(self, word1: str, word2: str) -> int:
        def dp(i, j):
            if i == 0:
                return j
            if j == 0:
                return i
            if word1[i - 1] == word2[j - 1]:
                return dp(i - 1, j - 1)
            return 1 + min(
                dp(i - 1, j - 1),  # replace
                dp(i - 1, j),      # delete
                dp(i, j - 1),      # insert
            )
        return dp(len(word1), len(word2))

func minDistance(word1 string, word2 string) int {
    var dp func(i, j int) int
    dp = func(i, j int) int {
        if i == 0 { return j }
        if j == 0 { return i }
        if word1[i-1] == word2[j-1] {
            return dp(i-1, j-1)
        }
        rep := dp(i-1, j-1)
        del := dp(i-1, j)
        ins := dp(i, j-1)
        mn := rep
        if del < mn { mn = del }
        if ins < mn { mn = ins }
        return 1 + mn
    }
    return dp(len(word1), len(word2))
}

Why This Works

The DP table encodes all possible alignments between prefixes of the two strings. Each cell represents the optimal cost of transforming one prefix into another. The three transitions (replace, delete, insert) correspond to the three possible “last operations”: replacing the last characters to match, removing the last character of word1 (making it shorter), or inserting the last character of word2 (making the target shorter). The recurrence is correct because optimal substructure holds — the best way to transform word1[:i] to word2[:j] depends only on the best ways to transform shorter prefixes. The base cases capture the cost of transforming from/to the empty string, which is simply the length of the other string.

Watch Out For:
  • Base cases: dp[0][j] = j and dp[i][0] = i. Transforming an empty string to a string of length j requires j insertions. Forgetting these produces wrong answers.
  • Off-by-one indexing: dp[i][j] corresponds to word1[:i] and word2[:j] (1-indexed in the DP table), but string access is 0-indexed, so compare word1[i-1] with word2[j-1].
  • Understanding the three operations: Insert = we match word2[j-1] by inserting it, so word2 shrinks by one: dp[i][j-1]. Delete = we remove word1[i-1], so word1 shrinks: dp[i-1][j]. Replace = both shrink: dp[i-1][j-1].
  • Space optimization pitfall: When reducing to 1D, you need to save the diagonal value (dp[i-1][j-1]) before overwriting it. Use a temp variable.

Language Features That Help

  • Python: @functools.lru_cache for top-down memoization as a quick optimization of the brute force. List comprehension [[0]*(n+1) for _ in range(m+1)] for 2D table.
  • Go: make([][]int, m+1) with inner make([]int, n+1) for 2D slice. No built-in min for three values before Go 1.21 — use a helper function or chain comparisons.
  • TypeScript: Math.min(a, b, c) accepts multiple arguments, making the three-way min clean. Array.from({ length: m+1 }, () => new Array(n+1).fill(0)) for 2D array.
  • Rust: .min() chains: a.min(b).min(c). vec![vec![0; n+1]; m+1] for 2D vector. String access via .as_bytes()[i] for O(1) indexing.
  • C++: min({a, b, c}) with initializer list for three-way min. vector<vector<int>>(m+1, vector<int>(n+1, 0)) for 2D table.
Brute: O(3^(m+n)) time → Optimal: O(mn) time, O(mn) space [O(min(m,n)) with rolling array]

Connection: Edit Distance extends the #1143 (Longest Common Subsequence) pattern — both use the same 2D DP structure comparing two sequences character by character. LCS has two transitions (match/skip), while Edit Distance has three (replace/delete/insert). The same framework applies to #583 (Delete Operation for Two Strings), #712 (Minimum ASCII Delete Sum), and #115 (Distinct Subsequences).
