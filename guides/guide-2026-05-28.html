class TwoSum:
    def __init__(self):
        self.counts = Counter()
    def add(self, number: int) -> None:

            if comp != num:

from collections import Counter
class Solution:
    def topKFrequent(self, nums: list[int], k: int) -> list[int]:
        count = Counter(nums)
        buckets = [[] for _ in range(len(nums) + 1)]
        for num, freq in count.items():
            buckets[freq].append(num)
        result = []
        for i in range(len(buckets) - 1, 0, -1):
            for num in buckets[i]:
                result.append(num)
                if len(result) == k:
                    return result
        return result
Go — Optimal
func topKFrequent(nums []int, k int) []int {
    count := make(map[int]int)
    for _, n := range nums {
        count[n]++
    }
    buckets := make([][]int, len(nums)+1)
    for num, freq := range count {
        buckets[freq] = append(buckets[freq], num)
    }
    var result []int
    for i := len(buckets) - 1; i > 0 && len(result) < k; i-- {
        result = append(result, buckets[i]...)
    }
    return result[:k]
}

class Solution(object):
    def reverseStr(self, s, k):
        l = 0
        ss = list(s)
        while l < len(s):
            if len(s) - l < k:
                a, b = l, len(s) - 1

                    a += 1

#20 Valid Parentheses Easy Stack String

The Problem: Given a string s containing just '(', ')', '{', '}', '[', ']', determine if the input string is valid. Valid means: open brackets are closed by the same type and in the correct order, and every close bracket has a corresponding open bracket.

Brute Force — O(n²) time

Repeatedly scan for and remove inner pairs "()", "{}", "[]" until no more replacements can be made. If the string is empty, it's valid.

Python — Brute Force
class Solution:
    def isValid(self, s: str) -> bool:
        while '()' in s or '{}' in s or '[]' in s:
            s = s.replace('()', '').replace('{}', '').replace('[]', '')
        return s == ''
Go — Brute Force
import "strings"
func isValid(s string) bool {
    for strings.Contains(s, "()") || strings.Contains(s, "{}") || strings.Contains(s, "[]") {
        s = strings.ReplaceAll(s, "()", "")
        s = strings.ReplaceAll(s, "{}", "")
        s = strings.ReplaceAll(s, "[]", "")
    }
    return s == ""
}

Complexity: Time O(n²) — each pass removes at most a few pairs and rescans the whole string. Space O(n) for new string copies.

Why optimize? Each replacement creates a new string and rescans from the start. For deeply nested inputs like "(((...)))", each pass only removes the innermost pair, requiring n/2 passes of O(n) work each.

Key Insight: The brute force rescans the entire string each iteration. A stack processes each character once: push opening brackets, pop and match for closing brackets. Each character is touched exactly once.
How to Think About It:
  1. Create an empty stack and a mapping of closing→opening brackets: ')':'(', '}':'{', ']':'['
  2. Iterate through each character in the string
  3. If it's an opening bracket, push it onto the stack
  4. If it's a closing bracket, check if the stack is non-empty AND the top matches the corresponding opening bracket. If not, return False
  5. After processing all characters, the stack must be empty (all opens were matched)
Why This Works: The stack enforces LIFO ordering — the most recently opened bracket must be the first to close. This exactly mirrors the nesting requirement of valid bracket sequences.
Watch Out For:
  • Forgetting to check if the stack is empty before popping (causes IndexError/panic on inputs like ")")
  • Not checking the stack is empty at the end (input like "(" would incorrectly return true)
  • Using the wrong mapping direction (should map closing→opening, not the reverse)

Alternative approaches: The brute-force replacement method works but is O(n²). Some solutions use a stack of expected closing brackets instead — when you see '(', push ')' — so the comparison on pop is a direct equality check. Both are O(n) time.

#155 Min Stack Medium Stack Design

The Problem: Design a stack that supports push, pop, top, and retrieving the minimum element, all in O(1) time.

Brute Force — O(n) getMin

Use a regular stack and scan the entire stack to find the minimum on each getMin() call.

Python — Brute Force
class MinStack:
    def __init__(self):
        self.stack = []
    def push(self, val: int) -> None:
        self.stack.append(val)
    def pop(self) -> None:
        self.stack.pop()
    def top(self) -> int:
        return self.stack[-1]
    def getMin(self) -> int:
        return min(self.stack)
Go — Brute Force
type MinStack struct {
    stack []int
}
func Constructor() MinStack {
    return MinStack{}
}
func (s *MinStack) Push(val int) {
    s.stack = append(s.stack, val)
}
func (s *MinStack) Pop() {
    s.stack = s.stack[:len(s.stack)-1]
}
func (s *MinStack) Top() int {
    return s.stack[len(s.stack)-1]
}
func (s *MinStack) GetMin() int {
    m := s.stack[0]
    for _, v := range s.stack {
        if v < m { m = v }
    }
    return m
}

Complexity: Time O(1) for push/pop/top, O(n) for getMin. Space O(n).

Why optimize? The problem explicitly requires O(1) for all operations including getMin. Scanning the full stack each time defeats the purpose of a stack-based design and becomes a bottleneck when getMin is called frequently.

Key Insight: Store (value, current_minimum) pairs on the stack. Each entry remembers what the minimum was at the time it was pushed. When you pop, the minimum naturally rolls back to its previous state.
How to Think About It:
  1. Maintain a main stack that stores tuples of (value, min_so_far)
  2. On push: compute new_min = min(val, current_min) and push (val, new_min)
  3. On pop: simply pop the top tuple
  4. On getMin: return the min component of the top tuple
  5. Handle the empty-to-first-push case: when stack is empty, first element's min is itself
Why This Works: Because the minimum can only change when elements are added or removed, and elements are removed in LIFO order, the minimum at each "level" of the stack is deterministic — it only depends on what's below it, which doesn't change.
Watch Out For:
  • Forgetting to handle the first push (stack is empty, so there's no previous min to compare against)
  • Using a separate min stack without syncing push/pop operations correctly
  • Alternative: two stacks approach — a main stack and a min stack. The min stack only pushes when a new minimum is found (push when val <= current min). This uses less space when there are few distinct minimums.

Alternative approaches: Two-stack design — keep a separate min-stack that only records new minimums. Push to the min-stack when val <= current min; pop from it when the popped value equals the min-stack top. Saves space when values rarely change the minimum. Both approaches are O(1) for all operations.

The Problem: Given an array of daily temperatures, return an array where answer[i] is the number of days you would have to wait after day i to get a warmer temperature. If there is no future day, answer[i] = 0.

Brute Force — O(n²) time

For each day, scan forward to find the next warmer day.

Python — Brute Force
class Solution:
    def dailyTemperatures(self, temperatures: list[int]) -> list[int]:
        n = len(temperatures)
        answer = [0] * n
        for i in range(n):
            for j in range(i + 1, n):
                if temperatures[j] > temperatures[i]:
                    answer[i] = j - i
                    break
        return answer
Go — Brute Force
func dailyTemperatures(temperatures []int) []int {
    n := len(temperatures)
    answer := make([]int, n)
    for i := 0; i < n; i++ {
        for j := i + 1; j < n; j++ {
            if temperatures[j] > temperatures[i] {
                answer[i] = j - i
                break
            }
        }
    }
    return answer
}

Complexity: Time O(n²) — for each of n days, we may scan up to n days forward. Space O(1) extra (excluding the output array).

Why optimize? With n up to 10&sup5;, the O(n²) approach can hit 10¹&sup0; operations, which is too slow. A monotonic stack reduces this to O(n) by resolving multiple pending days in a single pass.

Key Insight: The brute force re-scans forward for every element. A monotonic decreasing stack stores indices of days waiting for a warmer day. When you encounter a warmer temperature, you resolve ALL pending days in the stack that are colder — each day is pushed and popped exactly once, giving O(n) total.
How to Think About It:
  1. Initialize a stack (will store indices) and a result array of zeros
  2. Iterate through temperatures by index
  3. While the stack is non-empty AND the current temperature is greater than the temperature at the index on top of the stack: pop the top index, set result[popped_index] = current_index - popped_index
  4. Push the current index onto the stack
  5. Any indices remaining in the stack after iteration have no warmer day (their result stays 0)
Why This Works: The stack maintains a monotonically decreasing sequence of temperatures (by their indices). When a warmer temperature arrives, it resolves all pending cooler days in one batch. Each index is pushed once and popped once, so total work is O(n) even though the inner while loop can run multiple times for a single iteration.
Watch Out For:
  • Storing temperatures on the stack instead of indices (you need indices to compute day differences)
  • Using >= instead of > when comparing (equal temperatures don't count as "warmer")
  • Off-by-one: the answer is j-i (difference in indices), not the absolute temperature difference

Alternative approaches: This is the classic "monotonic stack" pattern. It appears in many problems like #42 Trapping Rain Water and #84 Largest Rectangle in Histogram. Mastering this pattern here makes those problems much more approachable. A reverse-iteration approach also works: iterate from right to left and use the answer array itself to skip ahead, giving O(n) time and O(1) extra space.

#226 Invert Binary Tree Easy Tree DFS

The Problem: Given the root of a binary tree, invert the tree (mirror it) and return its root. Every node's left and right children are swapped.

Brute Force — O(n) time

Create a completely new tree by traversing the original and building a mirrored copy. O(n) time, O(n) extra space for the new tree.

Python — Brute Force
class Solution:
    def invertTree(self, root: Optional[TreeNode]) -> Optional[TreeNode]:
        if not root:
            return None
        new_node = TreeNode(root.val)
        new_node.left = self.invertTree(root.right)
        new_node.right = self.invertTree(root.left)
        return new_node
Go — Brute Force
func invertTree(root *TreeNode) *TreeNode {
    if root == nil {
        return nil
    }
    newNode := &TreeNode{Val: root.Val}
    newNode.Left = invertTree(root.Right)
    newNode.Right = invertTree(root.Left)
    return newNode
}

Complexity: Time O(n), Space O(n) extra for the new tree

Why optimize? The brute force creates unnecessary copies. We can swap in-place with the same O(n) time but no extra tree allocation.

Key Insight: You don't need a new tree. At each node, swap its left and right children, then recurse. The swap can happen in-place — you visit every node exactly once and swap its two pointers.
How to Think About It:
  1. Base case: if the node is None/nil, return None/nil
  2. Swap the left and right children of the current node
  3. Recursively invert the left subtree (which was originally the right)
  4. Recursively invert the right subtree (which was originally the left)
  5. Return the root (same root, now with inverted structure)
Why This Works: Every subtree is a tree itself (recursive structure). Swapping children at every level mirrors the entire tree. The recursion reaches every node exactly once, and at each node the swap takes O(1), giving O(n) total.
Watch Out For:
  • Forgetting the base case (null check) — causes NullPointerException/nil dereference
  • Swapping AFTER recursing on original children instead of before (you'd recurse on the un-swapped children, inverting them in the wrong subtree). Actually both orderings work for this problem since you're swapping at every level, but it's cleaner to swap first then recurse.
  • Returning a new tree when the problem asks for in-place modification (wastes space)

Alternative approaches: BFS (level-order) with a queue — at each node dequeued, swap its children and enqueue them. Same O(n) time, O(n) space for the queue.

#98 Validate Binary Search Tree Medium Tree BST DFS

The Problem: Given the root of a binary tree, determine if it is a valid BST. A valid BST means: for every node, all values in its left subtree are strictly less than the node's value, and all values in its right subtree are strictly greater.

Brute Force — O(n) time

Do an in-order traversal, collect all values into an array, then check if the array is strictly increasing. O(n) time, O(n) space for the array.

Python — Brute Force
class Solution:
    def isValidBST(self, root: Optional[TreeNode]) -> bool:
        vals = []
        def inorder(node):
            if not node:
                return
            inorder(node.left)
            vals.append(node.val)
            inorder(node.right)
        inorder(root)
        for i in range(1, len(vals)):
            if vals[i] <= vals[i-1]:
                return False
        return True
Go — Brute Force
func isValidBST(root *TreeNode) bool {
    var vals []int
    var inorder func(*TreeNode)
    inorder = func(node *TreeNode) {
        if node == nil { return }
        inorder(node.Left)
        vals = append(vals, node.Val)
        inorder(node.Right)
    }
    inorder(root)
    for i := 1; i < len(vals); i++ {
        if vals[i] <= vals[i-1] { return false }
    }
    return true
}

Complexity: Time O(n), Space O(n) for the array

Why optimize? While the brute force is O(n), it requires O(n) extra space for the array AND makes two passes. We can do it in one pass with O(h) space by passing bounds.

Key Insight: Instead of collecting values and checking afterward, pass valid range bounds (min, max) down the recursion. Each node must satisfy min < node.val < max. When going left, update max to the current node's value. When going right, update min to the current node's value.
How to Think About It:
  1. Define a helper function that takes a node and valid range (low, high)
  2. Base case: null node is valid (return True)
  3. Check: if node.val <= low or node.val >= high, return False
  4. Recurse left with updated high bound: helper(node.left, low, node.val)
  5. Recurse right with updated low bound: helper(node.right, node.val, high)
  6. Return left AND right results
  7. Initial call: helper(root, -infinity, +infinity)
Why This Works: The bounds narrow as you descend the tree. Every node's valid range is determined by its ancestors. Going left tightens the upper bound; going right tightens the lower bound. If any node violates its bounds, the tree is not a BST.
Watch Out For:
  • Checking only against the parent (node.left.val < node.val) is WRONG — you must check against the entire valid range. Example: [5, 1, 6, null, null, 3, 7] — the 3 is in the right subtree of 5 but less than 5, which violates BST property, but it IS less than its parent 6.
  • Using <= instead of < for bounds (BST requires STRICT inequality — duplicates are not allowed in standard BST validation)
  • Using int min/max instead of -infinity/+infinity (fails for nodes with values at integer boundaries)

Alternative approaches: In-order traversal tracking just the previous value (O(1) extra space beyond the stack): maintain a prev variable and check that each visited node is strictly greater than prev.

Python Features That Help

# Chained comparison (Pythonic range check)
if not (low < node.val < high):
    return False

# Short-circuit AND for combining recursive results
return validate(node.left, low, node.val) and validate(node.right, node.val, high)
Go Features That Help
// math.MinInt / math.MaxInt as bounds
// But better: use *int (pointer) to represent "no bound"

    if node == nil { return true }
    if node.Val <= low || node.Val >= high { return false }
    return validate(node.Left, low, node.Val) && validate(node.Right, node.Val, high)
}
Brute: O(n) time, O(n) space
→
Optimal: O(n) time, O(h) space
#236 Lowest Common Ancestor of a Binary Tree Medium Tree DFS Recursion

The Problem: Given a binary tree and two nodes p and q, find their lowest common ancestor (LCA). The LCA is the deepest node that has both p and q as descendants (a node can be a descendant of itself).

Brute Force — O(n²) time

For each node, check if p is in its subtree AND q is in its subtree. The deepest such node is the LCA. Checking if a value is in a subtree takes O(n), and we do this for each node → O(n²).

Python — Brute Force
class Solution:
    def lowestCommonAncestor(self, root, p, q):
        def contains(node, target):
            if not node:
                return False
            if node == target:
                return True
            return contains(node.left, target) or contains(node.right, target)

        if not root or root == p or root == q:
            return root
        left_has_p = contains(root.left, p)
        left_has_q = contains(root.left, q)
        if left_has_p and left_has_q:
            return self.lowestCommonAncestor(root.left, p, q)
        if not left_has_p and not left_has_q:
            return self.lowestCommonAncestor(root.right, p, q)
        return root
Go — Brute Force
func lowestCommonAncestor(root, p, q *TreeNode) *TreeNode {
    var check func(*TreeNode, *TreeNode) bool
    check = func(n, t *TreeNode) bool {
        if n == nil { return false }
        if n == t { return true }
        return check(n.Left, t) || check(n.Right, t)
    }
    if root == nil || root == p || root == q { return root }
    lp := check(root.Left, p)
    lq := check(root.Left, q)
    if lp && lq { return lowestCommonAncestor(root.Left, p, q) }
    if !lp && !lq { return lowestCommonAncestor(root.Right, p, q) }
    return root
}

Complexity: Time O(n²), Space O(n) for recursion stack

Why optimize? The brute force calls contains() repeatedly at each level, doing O(n) work per level → O(n²) worst case. We can find the LCA in a single O(n) traversal.

Key Insight: Do a post-order DFS. At each node, recursively search left and right subtrees. If the left subtree returns non-null and the right subtree returns non-null, the current node is the LCA (p and q are split across subtrees). If only one side returns non-null, propagate that result up.
How to Think About It:
  1. Base case: if node is None, return None. If node IS p or q, return that node
  2. Recursively search the left subtree: left_result = recurse(node.left, p, q)
  3. Recursively search the right subtree: right_result = recurse(node.right, p, q)
  4. If BOTH left_result and right_result are non-None → current node is the LCA (p and q are in different subtrees)
  5. If only one is non-None → propagate that result upward (both p and q are in the same subtree)
  6. If both are None → neither p nor q is in this subtree, return None
Why This Works: The recursion explores the entire tree bottom-up. When we find p or q, we return it. The first node where left and right both return non-null is the split point — p is in one subtree and q is in the other. This is by definition the LCA. If p is an ancestor of q, then when we find p, we return it immediately (the base case handles "a node is a descendant of itself").
Watch Out For:
  • Forgetting that a node can be its own descendant (the problem states p and q exist in the tree, and one can be ancestor of the other)
  • Returning the wrong value when both targets are in the same subtree (should propagate the single non-null result, not the current node)
  • Not handling the case where p or q IS the root

Alternative approaches: If parent pointers exist, you can find LCA by tracing paths to root and finding the intersection (like finding the intersection of two linked lists — similar to #160!). Without parent pointers, the recursive approach above is optimal.

Connection: The pattern of "search both subtrees and combine results" is fundamental to tree problems. It's the same post-order pattern used in #104 (max depth) and #226 (invert tree), but here you're propagating node references instead of counts.

    return root
Go Features That Help

Optimal: O(n) time, O(h) space

#206 Reverse Linked List Easy Linked List Pointers

The Problem: Given the head of a singly linked list, reverse the list and return the reversed list's head. Input: 1→2→3→4→5. Output: 5→4→3→2→1.

Brute Force — O(n) time

Copy all values to an array, reverse the array, create a new linked list from the reversed values. O(n) time, O(n) extra space.

Python — Brute Force
class Solution:
    def reverseList(self, head: Optional[ListNode]) -> Optional[ListNode]:
        vals = []
        curr = head
        while curr:
            vals.append(curr.val)
            curr = curr.next
        dummy = ListNode(0)
        curr = dummy
        for v in reversed(vals):
            curr.next = ListNode(v)
            curr = curr.next
        return dummy.next
Go — Brute Force
func reverseList(head *ListNode) *ListNode {
    var vals []int
    for curr := head; curr != nil; curr = curr.Next {
        vals = append(vals, curr.Val)
    }
    dummy := &ListNode{}
    curr := dummy
    for i := len(vals) - 1; i >= 0; i-- {
        curr.Next = &ListNode{Val: vals[i]}
        curr = curr.Next
    }
    return dummy.Next
}

Complexity: Time O(n), Space O(n) extra for the array and new list

Why optimize? The array copy uses O(n) extra space. We can reverse in-place by re-pointing the next pointers.

Key Insight: Iterate through the list, and at each node, reverse the next pointer to point to the previous node instead of the next. Use three pointers: prev (starts null), curr (starts at head), and a temp to save curr.next before overwriting it.
How to Think About It:
  1. Initialize prev = None and curr = head
  2. While curr is not None: save next_temp = curr.next, then set curr.next = prev (reverse the link), then advance prev = curr, curr = next_temp
  3. After the loop, prev points to the new head (the old tail)
  4. Return prev
Why This Works: Each iteration reverses one link. After processing all nodes, every link points backward. The last node processed (original tail) becomes the new head, which is where prev ends up.
Watch Out For:
  • Forgetting to save curr.next before overwriting it (you lose the rest of the list)
  • Returning curr instead of prev at the end (curr is None after the loop)
  • Not handling empty list (head is None — prev starts as None, loop doesn't execute, return None. Works correctly.)

Alternative approaches: Recursive approach — reverse the rest of the list first, then fix the current node's link. Clean but uses O(n) stack space. The iterative approach is preferred for O(1) space.

Connection: This reversal technique is a building block for #234 (Palindrome Linked List — you reversed the second half) and #143 (Reorder List — coming next!).

#19 Remove Nth Node From End of List Medium Linked List Two Pointers

The Problem: Given the head of a linked list, remove the nth node from the end of the list and return its head. Example: [1,2,3,4,5], n=2 → [1,2,3,5] (remove 4).

Brute Force — O(n) two passes

First pass to count total length L. Second pass to reach node at position L-n-1 (the one before the target) and skip the target. Two passes, O(n) time.

Python — Brute Force
class Solution:
    def removeNthFromEnd(self, head: Optional[ListNode], n: int) -> Optional[ListNode]:
        length = 0
        curr = head
        while curr:
            length += 1
            curr = curr.next
        if length == n:
            return head.next
        curr = head
        for _ in range(length - n - 1):
            curr = curr.next
        curr.next = curr.next.next
        return head
Go — Brute Force
func removeNthFromEnd(head *ListNode, n int) *ListNode {
    length := 0
    for curr := head; curr != nil; curr = curr.Next {
        length++
    }
    if length == n {
        return head.Next
    }
    curr := head
    for i := 0; i < length-n-1; i++ {
        curr = curr.Next
    }
    curr.Next = curr.Next.Next
    return head
}

Complexity: Time O(n) two passes, Space O(1)

Why optimize? Two passes work, but we can do it in one pass using two pointers.

Key Insight: Use two pointers separated by n nodes. Advance the fast pointer n steps ahead. Then advance both pointers together until fast reaches the end. At that point, slow is at the node just before the one to remove.
How to Think About It:
  1. Create a dummy node pointing to head (handles edge case of removing the head)
  2. Set both slow and fast to the dummy node
  3. Advance fast by n+1 steps (so the gap between slow and fast is n+1 nodes, meaning slow will land BEFORE the target)
  4. Advance both slow and fast together until fast reaches None/nil
  5. Skip the target: slow.next = slow.next.next
  6. Return dummy.next (the possibly-new head)
Why This Works: When fast is at the end (None/nil), slow is exactly n+1 nodes behind. Since the list has L nodes, fast traveled L nodes from dummy, slow traveled L-(n+1) nodes from dummy. So slow is at position L-n-1 (0-indexed from dummy) — exactly the node before the target.
Watch Out For:
  • Not using a dummy node — removing the head (n = length) requires special handling without it
  • Advancing fast by n instead of n+1 — slow lands ON the target instead of before it
  • Off-by-one with n=1 (removing the last node) or n=length (removing the first node)

Alternative approaches: The two-pass approach (brute force above) is perfectly valid and often clearer. The one-pass approach is a common follow-up question.

Connection: This two-pointer gap technique is the same principle as #876 (Middle of Linked List) where slow moves at half speed. Here, both move at the same speed but start at different positions.

#143 Reorder List Medium Linked List Two Pointers Reversal

The Problem: Given the head of a singly linked list L₀→L₁→…→Lₙ₋₁→Lₙ, reorder it to L₀→Lₙ→L₁→Lₙ₋₁→L₂→Lₙ₋₂→… Modify the list in-place (don't return a new list). Example: [1,2,3,4] → [1,4,2,3]. Example: [1,2,3,4,5] → [1,5,2,4,3].

Brute Force — O(n) time

Copy all nodes to an array, then rewire using two pointers from both ends of the array. O(n) time, O(n) space.

Python — Brute Force
class Solution:
    def reorderList(self, head: Optional[ListNode]) -> None:
        nodes = []
        curr = head
        while curr:
            nodes.append(curr)
            curr = curr.next
        l, r = 0, len(nodes) - 1
        while l < r:
            nodes[l].next = nodes[r]
            l += 1
            if l == r:
                break
            nodes[r].next = nodes[l]
            r -= 1
        nodes[l].next = None
Go — Brute Force
func reorderList(head *ListNode) {
    var nodes []*ListNode
    for curr := head; curr != nil; curr = curr.Next {
        nodes = append(nodes, curr)
    }
    l, r := 0, len(nodes)-1
    for l < r {
        nodes[l].Next = nodes[r]
        l++
        if l == r { break }
        nodes[r].Next = nodes[l]
        r--
    }
    nodes[l].Next = nil
}

Complexity: Time O(n), Space O(n) for the array

Why optimize? The array uses O(n) extra space. We can do this in O(1) space by combining three linked list techniques.

Key Insight: This problem combines three classic linked list operations: (1) find the middle using slow/fast pointers, (2) reverse the second half, (3) merge the two halves by alternating nodes. Each operation is O(n) and O(1) space, giving O(n) total with O(1) space.
How to Think About It:
  1. Find the middle of the list using slow/fast pointers (slow ends at the middle)
  2. Split the list into two halves: first half ends at middle, second half starts at middle.next. Set middle.next = None to terminate the first half
  3. Reverse the second half using the iterative reversal technique from #206
  4. Merge the two halves by alternating: take one node from first half, then one from second half, repeat
  5. The merging loop continues until the second half is exhausted
Why This Works: By reversing the second half, Lₙ becomes the head of the reversed portion. Merging alternately from the first half (L₀, L₁, ...) and the reversed second half (Lₙ, Lₙ₋₁, ...) produces exactly the desired reordering.
Watch Out For:
  • Not properly terminating the first half (forgetting to set middle.next = None causes a cycle)
  • Off-by-one in finding the middle for even vs odd length lists (for [1,2,3,4], middle should be node 2 so first half is [1,2] and second half reversed is [4,3])
  • Not handling the merge correctly — the interleaving must save next pointers before rewiring

Alternative approaches: The array-based brute force is simpler to implement and works well. A stack-based approach (push all nodes, then pop from the stack while advancing from the front) also works but uses O(n) space.

Connection: This problem is the synthesis of #876 (find middle), #206 (reverse list), and basic list merging. If you've solved those, this is combining building blocks.

# Slow/fast pointer to find middle (same as #876, #234)
slow, fast = head, head
while fast.next and fast.next.next:
    slow = slow.next
    fast = fast.next.next
# slow is now at the middle

# Reverse a linked list in-place (from #206)
prev, curr = None, second_half
while curr:
    curr.next, prev, curr = prev, curr, curr.next

# Merge two lists alternately
while second:
    tmp1, tmp2 = first.next, second.next
    first.next = second
    second.next = tmp1
    first, second = tmp1, tmp2
Go Features That Help
// Slow/fast middle finding
slow, fast := head, head
for fast.Next != nil && fast.Next.Next != nil {
    slow = slow.Next
    fast = fast.Next.Next
}

// Split and reverse
second := slow.Next
slow.Next = nil
var prev *ListNode
curr := second
for curr != nil {
    next := curr.Next
    curr.Next = prev
    prev = curr
    curr = next
}
second = prev

// Merge alternating
first := head
for second != nil {
    tmp1, tmp2 := first.Next, second.Next
    first.Next = second
    second.Next = tmp1
    first = tmp1
    second = tmp2
}
