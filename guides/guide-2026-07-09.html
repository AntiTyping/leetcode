The ProblemGiven an integer x, return true if it reads the same forwards and backwards. Negative numbers are never palindromes. Bonus: solve it without converting the number to a string.
Quick ReminderRebuild the number's reverse digit by digit — but you only need to build half of it, stopping when the reversed tail meets or passes the remaining head.
Key InsightReversing the whole number risks overflow and does extra work. The insight: reverse only the second half. When the remaining x becomes ≤ the accumulated rev, you have processed half the digits — compare x == rev (even length) or x == rev // 10 (odd length, drop the middle digit).
Try to recall: Under 10 minutes, no string conversion. What two inputs must you special-case before the loop even starts?

Output: true
Input: x = -121
Output: false
Explanation: reads 121- backwards; negatives are never palindromes.
Input: x = 10
Output: false
Explanation: edge case — trailing zero, only 0 itself ends in 0 and is a palindrome.

class Solution(object):
    def isPalindrome(self, x):
        if x < 0:
            return False
        digits = []
        while x != 0:
            digits.insert(0, x % 10)
            x = x // 10
        i, j = 0, len(digits) - 1

            j -= 1
        return True

Key InsightA valid pattern is a bijection: each letter maps to exactly one word and each word maps back to exactly one letter. Keep two maps — char→word and word→char — and reject the first time either map disagrees with what you already recorded.
How to Think About It

Key InsightAnagram ≡ identical letter frequencies. The brute force rebuilds the window's counts from scratch. The insight: a fixed-size sliding window of length |p| lets you add the entering char and remove the leaving char in O(1), comparing the window's 26-count against p's count at each step.
How to Think About It

  2. If |p| > |s|, there are no windows — return empty.
  3. Compare the two count arrays; if equal, record index 0.
  4. Slide: increment the count for the new right character, decrement the count for the character leaving on the left.
  5. After each slide compare the arrays (or maintain a "matches" counter of how many of the 26 buckets agree) and record the start index on a full match.
Why This WorksTwo strings are anagrams exactly when their multiset of characters is identical, i.e. their frequency vectors are equal.   • Off-by-one on the window bounds — there are |s| − |p| + 1 windows.
  • Forgetting the early return when |p| > |s|, which can index out of bounds.
  • Comparing full 26-arrays every step is fine (O(26)); a stale "matches" counter that isn't updated on both the add and remove side is a subtle bug.

Why optimize? Counting sort works but touches the array twice; the follow-up explicitly asks for a single in-place pass, which the three-pointer partition delivers.

Key InsightThis is the Dutch National Flag partition. Keep three regions with three pointers: low (end of the 0s), mid (the scanner), and high (start of the 2s). Everything before low is 0, between low and mid is 1, after high is 2 — mid sweeps the unknown middle exactly once.
How to Think About It
  1. Set low = mid = 0 and high = n - 1.
  2. While mid ≤ high, look at nums[mid].
  3. If it is 0, swap it into the low boundary and advance both low and mid.
  4. If it is 1, it is already in place — just advance mid.
  5. If it is 2, swap it out to the high boundary and decrement high — but do not advance mid, because the value swapped in is unexamined.
Why This WorksThe three pointers maintain a loop invariant: [0, low) holds 0s, [low, mid) holds 1s, and (high, n) holds 2s. Every branch restores the invariant while shrinking the unknown region [mid, high], so when mid passes high the whole array is partitioned correctly.
Watch Out For
  • Advancing mid after swapping with high — the incoming value hasn't been classified yet, so you'd skip it.
  • Using mid < high instead of mid ≤ high, which drops the final element.
  • Forgetting that the 0-swap advances two pointers while the 2-swap advances neither mid.

Key InsightBecause every number is positive, the window product is monotonic: extending right multiplies up, shrinking left divides down. The insight: keep a window whose product is < k; when the new right element pushes the product to ≥ k, divide out from the left until it drops below again. Then every subarray ending at right and starting anywhere in [left, right] is valid — that's right - left + 1 new subarrays.
How to Think About It
  1. If k ≤ 1, no positive product can be below it — return 0.
  2. Initialize left = 0, prod = 1, count = 0.
  3. For each right, multiply prod by nums[right].
  4. While prod ≥ k, divide prod by nums[left] and advance left.
  5. Add right - left + 1 to count — the number of valid subarrays ending at right.
Why This WorksFix the right end. The valid subarrays ending there are exactly those whose start is at or after left (the smallest start keeping the product under k). Since positivity makes the product increase as the start moves left, all starts in [left, right] work and none before left do — giving exactly right - left + 1, summed over all right ends without double counting.
Watch Out For
  • The k ≤ 1 guard — without it the shrink loop can run left past right or loop forever.
  • Counting right - left + 1 per step (subarrays ending at right), not just 1.
  • Product overflow in languages with 32-bit ints — use a 64-bit accumulator in the brute force; the window version keeps it bounded.
  • This trick relies on all-positive numbers; a zero or negative breaks monotonicity.

How to Think About It

  4. Otherwise it is strictly to the right — set lo = mid + 1.
  5. When lo == hi, that single surviving index is the first bad version.
Why This WorksThe loop preserves the invariant that the answer always lies within [lo, hi]. Each step halves the range while keeping the boundary inside it: a bad mid can be the answer so hi = mid keeps it; a good mid cannot, so lo = mid + 1 discards it. The range shrinks to exactly the boundary.
Watch Out For

  • Setting hi = mid - 1 on a bad mid, which can skip the actual first bad version.
  • Looping with lo ≤ hi and returning inside — the lo < hi boundary form is cleaner and avoids an extra call.

Key InsightProcess characters left to right onto a stack. If the incoming character equals the stack's top, they cancel — pop instead of pushing. Otherwise push. The stack always holds the fully-reduced prefix, so a single pass does all the work.
How to Think About It
  1. Start with an empty stack (a list works).
  2. For each character c, peek the top of the stack.
  3. If the stack is non-empty and its top equals c, pop — the pair is removed.
  4. Otherwise push c.
  5. At the end, join the stack from bottom to top — that is the answer.
Why This WorksRemoving a pair can create a new adjacency between the characters that were on either side. The stack captures exactly this: after a pop, the next comparison is against the character that now sits directly before the current position — the same neighbor a fresh removal would expose. So one pass reaches the unique fully-reduced string.
Watch Out For
  • Peeking the top without checking the stack is non-empty first — index error on the first character.
  • Rebuilding a string on every push (O(n²)); push characters and join once at the end.
  • Comparing to the wrong element — always compare the incoming char to the current top.

Key InsightUse two stacks, in and out. Push always goes onto in. Only when out is empty do you pour all of in into out — that single reversal flips LIFO order into FIFO order. Then pop/peek read from the top of out.
How to Think About It

  2. Before any pop or peek, check whether out is empty.

  5. empty: true only when both stacks are empty.

  • Checking only one stack in empty(); elements may sit in either.
  • Peeking from in instead of out — the front lives on out after a transfer.

Two "aha" one-pass patterns to round out the day: a bit-DP that reuses smaller answers, and a greedy circuit walk that finds the unique valid start in a single sweep. Both replace an obvious O(n log n)/O(n²) with O(n).

#338 Counting Bits EasyBit ManipulationDynamic Programming

The ProblemGiven an integer n, return an array ans of length n + 1 where ans[i] is the number of set bits (1s) in the binary representation of i. Aim for O(n) time and a single pass.
Brute Force — O(n log n) time

For each i from 0 to n, count its bits independently by repeatedly testing and shifting off the lowest bit. Correct, but every number costs up to log n work.

Python
class Solution:
    def countBits(self, n):
        def popcount(x):
            c = 0
            while x:
                c += x & 1
                x >>= 1
            return c
        return [popcount(i) for i in range(n + 1)]

Why optimize? Counting each number from scratch is O(n log n); a DP recurrence reuses an already-computed smaller answer to make it O(n).

Key Insightans[i] = ans[i >> 1] + (i & 1). Dropping i's lowest bit gives the smaller index i >> 1, whose bit count is already known; then add back 1 if the bit you dropped was a one (i.e. i is odd). Every answer is built in O(1) from a previous one.
How to Think About It
  1. Allocate ans of size n + 1 with ans[0] = 0.
  2. For i from 1 to n, note that i >> 1 is strictly less than i, so ans[i >> 1] is already filled.
  3. Set ans[i] = ans[i >> 1] + (i & 1).
  4. Alternatively use ans[i] = ans[i & (i - 1)] + 1 — clearing the lowest set bit lands on a smaller, known index.
Why This WorksRight-shifting by one removes the least significant bit; the remaining bits are exactly the number i >> 1, whose popcount is stored. The removed bit contributes 1 precisely when i is odd (i & 1). Since i >> 1 < i, the DP dependency is always already computed, so the recurrence is exact and left-to-right filling is valid.
Watch Out For
  • Forgetting the ans[0] = 0 base case, which poisons every dependent entry.
  • Adding (i & 1) but shifting the wrong operand — you index with i >> 1, not i >> 1 of the count.
  • Sizing the array n instead of n + 1; the range is inclusive of n.

#134 Gas Station MediumGreedyArray

The ProblemThere are n gas stations on a circular route. At station i you gain gas[i] fuel and it costs cost[i] to reach station i+1. Starting with an empty tank at some station, return the index you must begin at to complete the full loop, or -1 if impossible. The answer is guaranteed unique when it exists.
Brute Force — O(n²) time

Try each station as the start; simulate the whole circular trip, adding gas[j] - cost[j] at each step and bailing out the moment the tank goes negative. Return the first start that survives the loop.

Python
class Solution:
    def canCompleteCircuit(self, gas, cost):
        n = len(gas)
        for start in range(n):
            tank = 0
            for i in range(n):
                j = (start + i) % n
                tank += gas[j] - cost[j]
                if tank < 0:
                    break
            else:
                return start
        return -1

Why optimize? Re-simulating from every start is O(n²); a single greedy pass decides feasibility and the start index in O(n).

Key InsightTwo facts collapse the search. First, a solution exists iff sum(gas) ≥ sum(cost). Second, if the running tank goes negative when traveling from candidate start s up to station i, then no station in [s, i] can be a valid start — so jump the candidate to i + 1 and reset the tank. One pass finds the answer.
How to Think About It
  1. Track two running sums: total (over all stations) and tank (since the current candidate start).
  2. At each station add gas[i] - cost[i] to both.
  3. If tank drops below 0, the current candidate can't work — set start = i + 1 and reset tank = 0.
  4. After the pass, if total ≥ 0 return start; otherwise return -1.
Why This WorksIf total < 0 the loop simply cannot be completed from anywhere. If total ≥ 0, a valid start must exist. When the tank fails reaching i from s, every intermediate start had an even lower prefix (since the prefix from s was non-negative up to just before the drop), so none of them survive either — the earliest feasible start is i + 1. The single candidate that never triggers a reset over the final segment is the unique answer.
Watch Out For
  • Deciding feasibility from tank instead of total — the final tank only covers the last segment; use the global total.
  • Forgetting to reset tank to 0 when you move the candidate start.
  • Overcomplicating with the modular O(n²) simulation when the greedy invariant already guarantees correctness.
