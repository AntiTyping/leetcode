Quick ReminderBinary-search a partition of the smaller array so that the left half (from both arrays) holds exactly half the elements and every left value ≤ every right value.
Key InsightYou don't need to merge. Find a cut point in the shorter array such that maxLeft ≤ minRight across both arrays; the median falls right at that balanced partition.

#101 Symmetric Tree EasyTreeDFSBFS

The ProblemGiven the root of a binary tree, return true if the tree is a mirror of itself around its center (symmetric), and false otherwise. A tree is symmetric when its left subtree is a mirror reflection of its right subtree: the values match and the children are mirrored (left's left vs right's right, left's right vs right's left). The number of nodes is in the range [1, 1000] and each node value is in [-100, 100].
Brute Force — O(n) time, O(n) space

A direct approach serializes the tree by an inorder-style traversal that records node values and explicit null markers, producing a sequence for the left half and the right half, then checks whether the whole level-order/serialized form reads the same forwards as its mirror. The simplest correct version below builds the left-to-right and right-to-left BFS encodings (including nulls) and compares them. It is linear, but it allocates full string/array encodings of the tree rather than comparing in place, which is wasteful in time constants and memory.

Python
from collections import deque

def isSymmetric(root):
    # Encode each level twice: left-to-right and right-to-left, with nulls.
    def encode(reverse):
        out = []
        q = deque([root])
        while q:
            level = []
            for _ in range(len(q)):
                node = q.popleft()
                if node is None:
                    level.append(None)
                    continue
                level.append(node.val)
                q.append(node.left)
                q.append(node.right)
            if any(v is not None for v in level):
                out.extend(level[::-1] if reverse else level)
        return out
    return encode(False) == encode(True)

Why optimize? Building two full encodings of every level (with null markers) allocates O(n) extra arrays and walks the tree twice. A single mirrored recursion compares the matching pair of nodes directly as you descend, using only the call stack and no auxiliary encoding, with the same O(n) time but far smaller constants and a cleaner invariant.

Key InsightSymmetry is not a property of one node but of a pair of nodes. Compare the left and right subtrees in lockstep: left.left mirrors right.right, and left.right mirrors right.left.
How to Think About It
  1. Reframe the question: instead of "is this tree symmetric?", ask "are these two subtrees mirror images of each other?"
  2. Write a helper that takes two nodes and returns whether they mirror each other.
  3. Base cases: two nulls are a mirror (true); one null and one node, or differing values, are not (false).
  4. Recurse on the outer pair (a.left with b.right) and the inner pair (a.right with b.left); both must hold.
  5. Kick it off by calling the helper on root.left and root.right (a single-node tree is trivially symmetric).
Why This WorksA tree is a mirror of itself exactly when its two top subtrees mirror each other, and two subtrees mirror each other exactly when their roots are equal and their cross-children mirror recursively. This recursive definition is self-similar, so the helper that checks one pair, applied down every matching pair of positions, verifies the whole tree. Each node is visited once, giving O(n) time and O(h) stack depth.
Watch Out For
  • Comparing the wrong pairs: it is outer-with-outer and inner-with-inner (left.left↔right.right), not left.left↔right.left.
  • Forgetting the asymmetric-null case: one child present and the other absent must return false, not be skipped.
  • Checking only values without recursing into structure, or recursing without first comparing values.

The ProblemGiven the root of a binary tree and an integer targetSum, return true if the tree has a root-to-leaf path such that adding up all the values along the path equals targetSum. Brute Force — O(n²) time, O(n) space

The naive method first enumerates every root-to-leaf path as an explicit list of values, collecting them all, and only then sums each path and compares to the target. Materializing each full path (copying the running list at every node and storing one list per leaf) costs up to O(n) per path across O(n) leaves, giving O(n²) time and memory, versus carrying a single running remainder down the recursion.

Why optimize? Storing and re-summing every full path duplicates work: each path is copied node by node (O(n) per path) and summed again afterward. By subtracting each node's value from the target as you descend and checking the remainder only at leaves, you replace all that bookkeeping with a single integer carried down the recursion, dropping to O(n) time and O(h) space.

Key InsightCarry the goal downward, not the history. Subtract each node's value from targetSum as you descend; the question "does a path sum to T?" becomes "does some leaf reduce the remaining target to exactly zero?"
How to Think About It
  1. Handle the empty tree up front: no root means no path, so return false.
  2. At each node, compute the remaining target after spending this node's value.
  3. If the node is a leaf (no left and no right), the path succeeds exactly when the remaining target equals this node's value (i.e. remainder hits zero).
  4. Otherwise, recurse into whichever children exist, passing the reduced target down.
  5. Combine child results with OR: the tree has a qualifying path if either subtree does.
Why This WorksA root-to-leaf path sum equals the target iff, after subtracting each value encountered, the leftover at the leaf is zero. Subtracting on the way down keeps an invariant: the parameter always holds "how much more we still need from here to the leaf." Because the decision is made only at leaves, partial sums in the interior never falsely succeed, and OR-combining children explores all paths while short-circuiting once one works. Every node is touched once: O(n) time, O(h) stack.
Watch Out For
  • Only checking the sum at leaves, not at internal nodes that happen to reach the target mid-path (the path must end at a leaf).
  • Treating a node with one child as a leaf; a leaf has both children null.
  • Negative values mean you cannot prune when the running sum exceeds the target; later nodes may bring it back down.
  • The empty-tree case must return false even when targetSum is 0.

The ProblemGiven the root of a binary tree, return the zigzag level order traversal of its node values: traverse level by level, but alternate direction each level — the first level (root) is read left to right, the second right to left, the third left to right, and so on. Return a list of lists, one per level. The number of nodes is in [0, 2000] and node values are in [-100, 100]; return an empty list for an empty tree.
Brute Force — O(n) time, O(n) space

A clumsy approach first computes each node's depth and value in a separate pass (e.g. a DFS that records (depth, value) pairs in visitation order), then groups the values by depth into per-level buckets, and finally reverses every other bucket to apply the zigzag. It produces the right answer but makes multiple passes and stores depth tags for every node; carrying direction inside one BFS does it in a single sweep.

Why optimize? Tagging every node with its depth and grouping afterward makes extra passes and stores redundant depth labels. A single BFS already processes the tree exactly one level at a time, so you can flip the read direction per level on the fly — same O(n) time, but one clean pass with no depth bookkeeping and no post-processing reversal of separate buckets.

Key InsightBFS naturally hands you one complete level at a time. Zigzag is just a presentation choice on top: keep a direction flag that flips each level and insert values at the front instead of the back on right-to-left levels.
How to Think About It
  1. Do a standard breadth-first traversal with a queue, processing the tree one level at a time.
  2. At the start of each level, record how many nodes are in it (the current queue size) so you know where the level ends.
  3. Maintain a boolean (or counter parity) telling you the current level's direction.
  4. For each node in the level, append its value left-to-right, or push to the front (or reverse at the end) when the direction is right-to-left.
  5. Enqueue children left then right as usual — the zigzag affects only output order, not traversal order.
  6. Flip the direction flag and continue until the queue empties.
Why This WorksBFS visits nodes in strict level order, and capturing the queue's size before draining a level isolates exactly that level's nodes. The traversal itself stays left-to-right at every level; only how you record each level alternates. Because children are always enqueued in left-then-right order, reversing the recorded row (or front-inserting) yields the correct mirrored reading without disturbing the next level's structure. One pass over n nodes gives O(n) time and O(n) space for the queue and output.
Watch Out For
  • Flipping the traversal/enqueue order instead of just the output order — children must still be queued left then right every level.
  • Forgetting to snapshot the level size before the inner loop; appending children mid-loop changes the queue length.
  • Front-insertion into an array list is O(k) per insert; prefer building left-to-right and reversing, or use a deque.
  • The empty tree must return an empty list, not a list containing an empty list.

Why optimize? Reversing the entire stack on every read is pure repeated work: an element can be shuffled back and forth many times. Keeping a dedicated "out" stack that holds elements already flipped into front-first order means each element is moved between stacks at most once, spreading the cost so every operation is amortized O(1).

Key InsightTwo reversals cancel out. One stack reverses insertion order; pour it into a second stack and it reverses again — back to FIFO. Keep an in stack for pushes and an out stack for pops, and only transfer when out is empty.
How to Think About It
  1. Keep two stacks: in receives every push; out serves every pop/peek.
  2. For push, simply push onto in — newest on top there.
  3. For pop/peek, first check out: if it is empty, pour all of in into out, which reverses the order so the oldest element ends up on top.
  4. Once out is non-empty, its top is the queue's front; peek reads it, pop removes it.
  5. Never transfer while out still has elements — that would interleave old and new ordering.
  6. The queue is empty only when both stacks are empty.
Why This WorksPushing onto in stores elements newest-on-top. Transferring the whole in stack into out reverses that, so out holds the elements oldest-on-top — exactly FIFO order. By transferring only when out is empty, every element is moved across the two stacks at most once (push to in, pop from in, push to out, pop from out), so n operations cost O(n) total: amortized O(1) each, even though a single transfer can be O(n).
Watch Out For
  • Refilling out while it still has elements — only transfer when it is empty, or you scramble the order.
  • Forgetting that peek must also trigger the transfer when out is empty, not just pop.
  • Checking emptiness on only one stack; the queue is empty when in AND out are both empty.
  • Transferring on every operation, which throws away the amortization and reverts to O(n) per call.

The ProblemGiven an encoded string s, return its decoded form. The input may be nested, e.g. "3[a2[c]]" decodes to "accaccacc", and "2[abc]3[cd]ef" decodes to "abcabccdcdcdef". Brute Force — O(maxK · n) time, O(n) space

The naive method repeatedly scans for the innermost k[...] — a bracket pair with no brackets inside — expands just that one by repeating its contents k times, splices the result back into the string, and starts the scan over. Because each pass rebuilds the whole string and only resolves one bracket, deeply nested or highly repeated inputs are rescanned many times. It is correct but does redundant full-string passes.

Why optimize? Re-scanning and rebuilding the entire string for each bracket repeats work proportional to the growing output many times over. A single left-to-right pass with a stack remembers each pending repeat count and the string built so far, so when a ] closes you expand once and resume — every character is processed a constant number of times relative to its appearances in the output.

Key InsightBrackets nest, and a stack is the natural model for nesting. When you hit [, push the work done so far (the string and the repeat count) and start fresh; when you hit ], pop, repeat the freshly built piece, and append it back to the parent.
How to Think About It
  1. Scan left to right, maintaining a "current string" being built and a "current number" for the pending repeat count.
  2. On a digit, fold it into the current number (multiply by ten and add, to handle multi-digit counts like 100).
  3. On [, push the current string and current number onto a stack, then reset both to start the inner segment.
  4. On a letter, append it to the current string.
  5. On ], pop the saved count k and saved prefix; set current = prefix + (current repeated k times).
  6. When the scan ends, the current string is the fully decoded result.
Why This WorksEach [ opens a new context whose contents are independent until the matching ]. Pushing the enclosing prefix and count onto a stack preserves that outer context while you build the inner one; popping on ] restores it and folds in the expanded inner segment. Because the stack mirrors bracket depth exactly, contexts close in the correct (innermost-first) order, and every character is appended or repeated a number of times equal to its multiplicity in the output — linear in the decoded length.
Watch Out For
  • Multi-digit counts: build the number across consecutive digits, do not treat each digit as a separate count.
  • Order on ]: the result is prefix + current*k, not current*k + prefix; the popped prefix comes first.
  • Resetting current string and number to empty/zero right after pushing on [.
  • Letters and digits can sit at the top level outside any bracket (e.g. trailing "ef"); the algorithm must still handle them.

Why optimize? Enumerating all 2ⁿ subsets is exponential and wasteful: most subsets overshoot the target long before they are complete, yet the brute force still builds and sums every one. Backtracking sorts once, then grows a combination element by element, pruning a branch the instant the running sum exceeds the target and skipping equal siblings to avoid duplicates — visiting only promising partial states.

Key InsightSort the candidates so equal values sit next to each other. Then do a DFS that, at each step, either includes the current candidate or skips past all copies of its value. Skipping duplicates at the same recursion depth is exactly what prevents duplicate combinations — no global set needed.
How to Think About It
  1. Sort candidates ascending so duplicate values are adjacent and you can prune by magnitude.
  2. Recurse with a start index, a running combination, and a remaining target.
  3. At each index, if the value exceeds the remaining target, stop — everything after is at least as large (pruning).
  4. Choose the current value (advancing the start index by one so it is used at most once), recurse, then remove it to backtrack.
  5. When skipping a value, also skip every later element equal to it, so no two branches at this depth start with the same number.
  6. Record the combination whenever the remaining target hits exactly zero.
Why This WorksSorting makes equal values contiguous, so “use this value or jump past all its duplicates” partitions the choices at each depth into distinct first-elements — guaranteeing every generated combination is unique without post-filtering. Advancing the index past the chosen element enforces the at-most-once rule. Because the array is sorted, the first time a candidate exceeds the remaining target you can abandon the whole tail, which makes the search visit only viable prefixes.
Watch Out For
  • Skipping duplicates only at the same depth (i > start && candidates[i] == candidates[i-1]) — skipping unconditionally would drop valid combinations that legitimately reuse a duplicated value at deeper levels.
  • Passing i + 1 (not i) as the next start index, since each element may be used at most once.
  • Forgetting to sort first, which breaks both the duplicate-skip logic and the magnitude-based pruning.
  • Appending a reference to the working list instead of a copy when recording a solution.

The ProblemGiven a string s, partition s such that every substring of the partition is a palindrome. Return all possible palindrome partitionings of s. A partition is an ordered list of contiguous substrings that, concatenated, reproduce s; each substring must read the same forwards and backwards. Constraints: 1 ≤ s.length ≤ 16, s contains only lowercase English letters.
Brute Force — O(2ⁿ · n²) time, O(n) space per partition

There are n - 1 gaps between adjacent characters, and each gap is independently either a cut or not, giving 2ⁿ⁻¹ ways to split the string. Enumerate every gap pattern with a bitmask, build the resulting list of substrings, and keep the pattern only if every substring is a palindrome. This re-checks palindromicity from scratch for each pattern and never abandons a split that has already produced a non-palindrome piece.

Why optimize? The bitmask approach builds and validates all 2ⁿ⁻¹ splits even when an early piece is already not a palindrome, wasting work on entire families of doomed patterns. Backtracking instead only ever extends a partition with a substring it has already confirmed is a palindrome, so it never enters a branch that cannot succeed — pruning the search at the first bad cut.

Key InsightBuild the partition prefix-first: from the current position, try every substring starting there, and only recurse into the remainder when that prefix is itself a palindrome. The palindrome check becomes the pruning condition, so you never descend into a partition that already contains a non-palindrome.
How to Think About It
  1. Define a DFS over a start index i and a running list of palindromic pieces.
  2. When i reaches the end of s, the running list is a complete valid partition — record a copy.
  3. For each end index j from i to the end, consider the substring s[i..j].
  4. Check whether s[i..j] is a palindrome with a two-pointer test; if not, skip this j (prune).
  5. If it is a palindrome, append it, recurse from j + 1, then pop it to backtrack and try a longer prefix.
  6. Optionally precompute an isPal[i][j] table so each palindrome check is O(1).
Why This WorksEvery recursion level fixes the next piece of the partition starting exactly where the previous piece ended, so the pieces always tile s with no gaps or overlaps. Because a piece is only appended after confirming it is a palindrome, every partial partition consists entirely of palindromes — reaching the end therefore yields a fully valid answer. Trying every possible end index for the first piece, recursively, enumerates exactly the set of palindrome partitionings, each exactly once.
Watch Out For
  • Off-by-one on substring bounds: the prefix is s[i..j] inclusive, and the recursion must continue from j + 1.
  • Single characters are palindromes, so the all-singletons partition is always valid — don't accidentally exclude length-1 pieces.
  • Storing a reference to the mutable path instead of a copy when recording a solution.
  • Re-running an O(n) palindrome check inside the recursion without memoizing can dominate the cost; precompute the table if needed.

The ProblemThe n-queens puzzle is the problem of placing n queens on an n × n chessboard so that no two queens attack each other — meaning no two share a row, column, or diagonal. Given an integer n, return all distinct solutions to the n-queens puzzle. Each solution is a board configuration represented as a list of n strings of length n, where 'Q' marks a queen and '.' marks an empty square. Constraints: 1 ≤ n ≤ 9.
Brute Force — O(nⁿ · n²) time, O(n) space

Because at most one queen can share a row in any valid solution, place exactly one queen per row and let each row independently pick a column from 0 to n-1. That gives nⁿ full assignments, enumerated here with a base-n counter held in the pos array (pos[r] = column of the queen in row r). For each assignment, check all pairs of queens for a shared column or diagonal; if none conflict, render the board. The waste is that it only validates after every queen is placed, never abandoning a partial board that is already in conflict.

Why optimize? Enumerating all nⁿ column assignments and only validating complete boards explores a staggering number of obviously dead configurations — two queens may conflict in the very first two rows, yet the brute force keeps filling the remaining n-2 rows before noticing. Backtracking places one queen at a time and checks the new queen against those already placed, abandoning a partial board the instant a conflict appears, which prunes whole subtrees of the search.

Key InsightPlace queens one row at a time. The only freedom in row r is the column, so track which columns and which two diagonal families are already occupied. A queen at (r, c) uses column c, “down-right” diagonal r - c, and “down-left” diagonal r + c — constant-time conflict checks that prune instantly.
How to Think About It
  1. Recurse row by row; the recursion depth equals the row currently being filled.
  2. Maintain three sets: occupied columns, occupied r - c diagonals, and occupied r + c diagonals.
  3. For the current row, try each column whose column and both diagonals are still free.
  4. Place the queen: mark its column and two diagonals, record its position, then recurse to the next row.
  5. On returning, unmark the column and diagonals to backtrack and try the next column.
  6. When the recursion reaches row n, all queens are placed safely — render the board and add it to the results.
Why This WorksEach cell lies on exactly one “down-right” diagonal (constant r - c) and one “down-left” diagonal (constant r + c), so two queens attack each other iff they share a column or one of those diagonal values. Placing one queen per row already rules out row conflicts; checking the three occupancy sets rules out the rest in O(1). Because a queen is only placed in a safe square, every partial board is conflict-free, so any board reaching row n is a complete valid solution — and trying every safe column at each row enumerates them all.
Watch Out For
  • The anti-diagonal index r + c ranges up to 2n - 2; size or offset your diagonal markers accordingly.
  • Forgetting to undo all three marks (column and both diagonals) when backtracking corrupts later branches.
  • Building each output row with exactly n characters and a single 'Q' at the chosen column.
  • Edge cases: n = 1 has one solution; n = 2 and n = 3 have none — the search must correctly return an empty list.

Graph & DFS

Many grid and relationship problems are graphs in disguise: a grid cell is a node with up/down/left/right edges, and a list of equations is a weighted graph where each variable is a node. The core skill is the same — build (or imagine) an adjacency structure, then flood it with DFS while tracking visited nodes so you never revisit. These two problems are deliberately Interleaved — one is an implicit grid graph, the other an explicit weighted graph — so your brain has to first recognize the graph before reaching for the traversal.

#695 Max Area of Island MediumGraphDFSGrid

The ProblemYou are given an m x n binary grid grid where 1 represents land and 0 represents water. An island is a maximal group of 1s connected 4-directionally (horizontally or vertically). The area of an island is the number of cells with value 1 in it. Return the maximum area of any island in grid; if there is no island, return 0. Constraints: 1 <= m, n <= 50, and each cell is 0 or 1.
Brute Force — O((m·n)²) time, O(m·n) space

The naive idea avoids any shared bookkeeping between cells: for every land cell, run a fresh DFS that floods its whole island into a brand-new local visited set and counts the cells, then take the max area seen. Because the visited marks are thrown away after each cell, the same island gets fully re-explored once for every land cell it contains, giving O((m·n)²) work. The optimal version marks cells visited globally so each cell is touched once.

Why optimize? Re-flooding an island once per cell it contains is pure repeated work — a single 50×50 all-land grid would explore 2500 cells 2500 times. If you mark land as visited globally (or sink it to 0) the moment you touch it, every cell is visited exactly once and the whole grid is processed in O(m·n).

Key InsightA grid is a graph: each 1 is a node with edges to its 4 orthogonal neighbors that are also 1. One DFS/BFS started from an unvisited land cell discovers an entire island and counts its size in one sweep — visited tracking shared across the whole grid is what keeps it linear.
How to Think About It
  1. Scan every cell of the grid; when you find land that has not yet been visited, it is the seed of a new island.
  2. From that seed, run a flood (DFS or BFS) that visits all 4-connected land cells, counting how many you touch.
  3. Mark each land cell as visited as soon as you enter it — either with a separate visited grid or by overwriting it to 0 — so it is never counted twice.
  4. The count returned by one flood is that island's area; keep a running maximum across all seeds.
  5. Guard every neighbor with bounds checks before reading the grid, and stop the flood at water or already-visited cells.
  6. After scanning every cell, the running maximum is the largest island area (0 if no land existed).
Why This WorksEach connected component of land is reached from exactly one seed cell — the first of its cells encountered during the scan — because once an island is flooded, all its cells are marked visited and skipped by later iterations of the outer scan. The flood explores every cell reachable through orthogonal land links, so it measures the component's full size. Taking the max over all components yields the global answer in a single linear pass.
Watch Out For
  • Only 4-directional connectivity counts — diagonally touching land cells are separate islands.
  • Bounds-check neighbors before indexing, or you will read out of range at the grid borders.
  • Forgetting to mark a cell visited before pushing its neighbors causes infinite loops or massive overcounting.
  • An all-water grid must return 0; deep recursion on a 50×50 all-land grid can overflow the stack — prefer an explicit stack or raise the recursion limit.

The ProblemYou are given an array equations where equations[i] = [A_i, B_i] and a parallel array values where values[i] is the result of A_i / B_i. Each A_i, B_i is a variable name (a string). You are also given queries where queries[j] = [C_j, D_j] asks for the value of C_j / D_j. Return an array of answers: the computed quotient for each query, or -1.0 if it cannot be determined (an unknown variable, or no chain of equations links the two). It is guaranteed there are no contradictions and no division by zero. A variable divided by itself is 1.0 only if that variable appears in the equations.
Brute Force — O(Q·(V+E)) time, O(V+E) space

Model each variable as a graph node and each equation a/b = v as two directed weighted edges: a→b with weight v and b→a with weight 1/v. To answer x/y, DFS from x to y, multiplying edge weights along the path; the accumulated product is the answer. If either variable is unknown or no path exists, return -1.0. This runs a fresh O(V+E) traversal per query. The optimal approach compresses each connected component with weighted Union-Find so queries become near-constant.

def calcEquation(equations, values, queries):
    graph = {}
    for (a, b), v in zip(equations, values):
        graph.setdefault(a, {})[b] = v
        graph.setdefault(b, {})[a] = 1.0 / v

    def dfs(src, dst, visited):
        if src not in graph or dst not in graph:
            return -1.0
        if src == dst:
            return 1.0
        visited.add(src)
        for nbr, w in graph[src].items():
            if nbr in visited:
                continue
            sub = dfs(nbr, dst, visited)
            if sub != -1.0:

        return -1.0

Why optimize? Each query reruns a full graph traversal from scratch, so Q queries over a graph with V variables and E equations cost O(Q·(V+E)). Weighted Union-Find collapses every variable to a single root with a stored ratio to that root, so once built, each query is two near-O(1) finds — answer is the ratio of the two stored weights when the roots match, -1.0 otherwise.

Key InsightDivision facts are transitive: if a/b = 2 and b/c = 3, then a/c = 6. Encode each equation as a pair of reciprocal weighted edges and the whole system becomes a graph where x/y equals the product of edge weights along any path from x to y — and the no-contradiction guarantee means every path gives the same product.
How to Think About It
  1. Build an adjacency map: for each equation a/b = v, add edge a→b with weight v and edge b→a with weight 1/v.
  2. For a query x/y, first reject it if either x or y never appeared in any equation — the answer is -1.0.
  3. DFS (or BFS) from x toward y, carrying the running product of weights traversed so far.
  4. Track visited nodes so cycles in the graph do not cause infinite loops.
  5. If you reach y, the accumulated product is the answer; the trivial case x == y (with x known) is 1.0.
  6. If the search exhausts the component without reaching y, the two variables are disconnected, so return -1.0.
Why This WorksMultiplying edge weights along a path telescopes: (a/b)·(b/c)·(c/d) = a/d, since intermediate variables cancel. Because the problem guarantees consistency, any path between two connected nodes yields the same product, so the first path DFS finds is correct. Disconnected variables have no telescoping chain, which is exactly the -1.0 ("cannot determine") case, and the reciprocal back-edges make the relation symmetric so you can travel either direction.
Watch Out For
  • A variable that never appears in the equations is unknown — even x/x must return -1.0 in that case, not 1.0.
  • Remember to add both directions per equation; missing the reciprocal edge makes half of all queries unanswerable.
  • Cycles exist (e.g. a/b, b/c, c/a) — without a visited set the DFS loops forever.
  • Use floating-point comparison carefully; here a sentinel like -1.0 for "not found" is cleaner than comparing computed doubles for equality.

The recurring move: maintain a heap of bounded size k (so you sort only what matters) or a heap holding one frontier element from each of k inputs.     freq = Counter(words)
    # -freq[w] sorts by frequency desc; w breaks ties asc
    ordered = sorted(freq.keys(), key=lambda w: (-freq[w], w))
    return ordered[:k]

Why optimize? Sorting every unique word is wasteful when k is small — you order hundreds of words just to keep a handful. A min-heap of size k (ordered so the "weakest" candidate sits on top) lets you push each word and pop the loser, holding only k elements, so the per-element cost is O(log k) instead of O(log n).

Key InsightThis is a two-stage problem: first count with a hash map, then rank by a compound ordering (frequency high-to-low, ties alphabetical). A heap shines because you only need the top k of that ranking, not a fully sorted list — bound the heap to k and you avoid ordering everything.
How to Think About It
  1. Tally each word's count in a hash map in one linear pass over words.
  2. Define the ranking precisely: a word ranks higher if its frequency is greater, and among equal frequencies the lexicographically smaller word ranks higher.
  3. To pick the top k, push candidates into a heap whose top is the worst qualifier under that ranking (a min-heap by the same comparison).
  4. Whenever the heap exceeds size k, pop the top — it is the current weakest and cannot be in the answer.
  5. After processing all unique words, the heap holds the k best, but in reverse order — drain it and reverse to get highest-frequency-first.
  6. Be careful that the heap's tie-breaking inverts correctly so alphabetical order survives the final reversal.
Why This WorksA size-k min-heap ordered by the answer's ranking always evicts the element that is currently least deserving, so once every word has been offered, only the k strongest remain — no stronger word was ever discarded, because it would have displaced a weaker one. Counting first guarantees frequencies are final before ranking. The compound comparator encodes the exact tie-break rule, so frequency dominates and alphabetical order resolves ties consistently.
Watch Out For
  • The tie-break direction is opposite for the two keys: frequency descending but word ascending — a naive single sort key gets one of them backwards.
  • With a min-heap of size k, the comparator must be inverted relative to the final order, and you must reverse the drained output.
  • Python's heapq is a min-heap only; pushing (-count, word) sorts counts the way you want but flips word ordering — handle that explicitly.
  • k can equal the number of unique words, so the answer may include every distinct word.

Why optimize? Dumping and re-sorting throws away the fact that each list is already ordered — you pay a full O(N log N) sort. Since only the smallest current head across the k lists can come next, a min-heap holding one node per list gives you that minimum in O(log k) and refills from the same list, producing the merged list in O(N log k) with O(k) extra space.

Key InsightAt any moment, the next node of the merged list is the smallest among the current heads of the k lists. A min-heap of those k heads gives that minimum in O(log k); after popping a node, push its successor from the same list. This is the classic k-way merge, the heart of external sorting.
How to Think About It
  1. Seed a min-heap with the head node of every non-empty list, ordered by node value.
  2. Use a dummy head node and a moving tail pointer to build the result without special-casing the first append.
  3. Pop the smallest node from the heap, append it to tail, and advance tail.
  4. If the popped node has a successor in its original list, push that successor onto the heap.
  5. Repeat until the heap is empty; every node has then been threaded into the result exactly once.
  6. Return dummy.next, which skips the placeholder and points at the true merged head.
Why This WorksThe heap always contains the frontier — the smallest unprocessed node of each list — so its minimum is the global smallest remaining value, exactly the next element of the sorted output. Pushing only the successor of a popped node keeps each list's contribution in order and bounds the heap to at most k elements. Since every node is pushed and popped once, total work is O(N log k), and the dummy-node pattern guarantees a clean head pointer even if some inputs are empty.
Watch Out For
  • Empty inputs: k == 0 or every list empty must return null — don't seed the heap with non-existent heads.
  • Equal values across lists need a stable tie-break; comparing raw nodes can error in languages without a default node ordering — compare by value (and a list index if needed).
  • After popping, remember to push the successor; forgetting it silently drops the rest of that list.
  • Reuse the existing nodes by relinking rather than allocating duplicates when an in-place merge is expected.

Dynamic Programming

Dynamic programming attacks problems that decompose into overlapping subproblems with optimal substructure. The recipe is always the same: define the state (what arguments uniquely describe a subproblem), write the recurrence (how an answer is built from smaller answers), and identify the base cases. Naive recursion re-solves the same states exponentially often; memoization or a bottom-up table caches each state once to make it polynomial. These problems are Interleaved.

The ProblemGiven a triangle array where row i has i + 1 elements, return the minimum path sum from the top to the bottom. From index j on row i you may step only to index j or index j + 1 on the next row (the two adjacent numbers below). There are n rows with 1 ≤ n ≤ 200 and each value fits in a 32-bit integer. Return the smallest total of any such top-to-bottom path.
Brute Force — O(2ⁿ) time, O(n) space

The naive approach mirrors the definition directly: from any cell, the best path going down is that cell's value plus the cheaper of the two recursive sub-paths (straight down and down-right). With no caching, every cell branches into two recursive calls, so the recursion tree fans out to roughly 2ⁿ leaves — one per distinct root-to-bottom path. The same (row, col) subproblem is recomputed many times, which is exactly the overlapping-subproblem signal that DP exploits.

Why optimize? The recursion recomputes each (row, col) cell once per path that reaches it, and the number of paths doubles per row — exponential work. Because there are only O(n²) distinct cells, caching each result (memoization) or filling a table bottom-up collapses the cost to O(n²).

Key InsightThe minimum path sum through a cell depends only on that cell's value plus the better of the two cells directly below it. That single relation, applied to every cell, is the whole problem — and there are only O(n²) cells, so each one needs to be solved just once.
How to Think About It
  1. Define the state: best(row, col) = minimum sum to travel from this cell down to the last row.
  2. Write the recurrence: best(row, col) = triangle[row][col] + min(best(row+1, col), best(row+1, col+1)).
  3. Identify the base case: on the last row, best(row, col) is just triangle[row][col].
  4. Notice that many paths revisit the same (row, col) — cache each result so it is computed once.
  5. Equivalently, fill the table from the bottom row upward; each cell reads the two already-solved cells beneath it.
  6. The answer is the state at the apex, best(0, 0).
Why This WorksEvery top-to-bottom path passes through a sequence of cells where each step lands on an adjacent cell below. The optimal path through a cell must continue optimally from one of its two children, so the per-cell recurrence captures every path exactly once without missing or double-counting. Solving bottom-up guarantees a cell's children are finished before it is read, and reducing to O(n²) distinct states removes the exponential redundancy.
Watch Out For
  • Adjacency is restricted: from index j you may only reach j or j + 1 below, never arbitrary columns.
  • Values can be negative, so you cannot prune based on partial sums or assume monotonic growth.
  • Off-by-one on the last row: the base case is row n - 1, and the column never exceeds the row index.
  • Plain recursion without caching will time out for large n; the overlapping subproblems are the point.

The ProblemGiven two strings word1 and word2, return the minimum number of single-character operations required to convert word1 into word2. The permitted operations are: insert a character, delete a character, and replace a character. Both strings have length between 0 and 500 and consist of lowercase English letters. This minimum count is also known as the Levenshtein distance.
Brute Force — O(3ⁿ) time, O(m + n) space

The naive recursion compares the two strings from the front via indices i and j. If the current characters match, no operation is needed and we advance both. Otherwise we try all three operations and take the cheapest: insert (advance j), delete (advance i), or replace (advance both). Each mismatch branches three ways, so the recursion tree explodes to roughly 3ⁿ nodes while revisiting the same (i, j) pairs over and over — the classic overlapping-subproblem pattern.

def minDistance(word1, word2):
    m, n = len(word1), len(word2)
    def dfs(i, j):
        if i == m: return n - j      # insert what's left of word2
        if j == n: return m - i      # delete what's left of word1
        if word1[i] == word2[j]:
            return dfs(i + 1, j + 1)
        insert = dfs(i, j + 1)
        delete = dfs(i + 1, j)
        replace = dfs(i + 1, j + 1)
        return 1 + min(insert, delete, replace)
    return dfs(0, 0)

Why optimize? Each mismatched position spawns three recursive calls, so the unmemoized tree is exponential. But the state is fully described by the pair (i, j), of which there are only (m + 1)(n + 1). Caching each state — or filling an (m + 1) × (n + 1) table — brings the work down to O(m · n).

Key InsightAlign the two strings from one end. At each step the choice is local: if the current characters match, carry on for free; otherwise the cost is 1 plus the best of three sub-alignments (insert, delete, replace). Every subproblem is just "edit distance of the two remaining suffixes," indexed by (i, j) — only O(m · n) of them exist.
How to Think About It
  1. Define the state: dist(i, j) = edit distance between word1[i:] and word2[j:].
  2. Base cases: if one suffix is empty, the answer is the length of the other (all inserts or all deletes).
  3. If word1[i] == word2[j], no operation is needed: dist(i, j) = dist(i+1, j+1).
  4. Otherwise take 1 + min of insert dist(i, j+1), delete dist(i+1, j), and replace dist(i+1, j+1).
  5. The same (i, j) recurs across many branches — memoize, or build a 2-D table.
  6. The final answer is dist(0, 0).
Why This WorksAny optimal edit sequence can be reordered so it processes the strings left to right; at the leading position exactly one of four things happens (match, insert, delete, or replace), and the rest of the sequence is an optimal edit of the remaining suffixes. Taking the minimum over those choices therefore yields the true optimum, and because each subproblem is independent of how it was reached, the (i, j) table captures every distinct case exactly once.
Watch Out For
  • Which operation maps to which index move: insert advances j, delete advances i, replace advances both.
  • Empty-string base cases must return the remaining length, not 0.
  • Replace and "match" both advance both indices, but replace adds 1 while a match adds 0.
  • Without memoization the recursion is exponential and will time out near the 500-character limit.

Linked List

Linked-list problems are about disciplined pointer manipulation: you rarely have random access, so you advance, rewire, and track nodes one reference at a time. Recurring techniques include the dummy head to simplify edge cases, careful save-then-relink ordering when reversing, and fast/slow pointers that move at different speeds to detect cycles or find midpoints in O(1) extra space. These problems are Interleaved.

#142 Linked List Cycle II MediumLinked ListTwo Pointers

The ProblemGiven the head of a singly linked list, return the node where the cycle begins. A cycle exists if some node can be reached again by continuously following the next pointer; internally a position pos indicates the index that the tail's next connects to (pos = -1 means no cycle, and pos is not passed to your function). If there is no cycle, return null. You must not modify the list. The number of nodes is in [0, 10000].
Brute Force — O(n) time, O(n) space

The straightforward approach walks the list and records every node it has visited in a hash set. The first time we arrive at a node that is already in the set, that node must be the entry point of the cycle — it is the earliest node that can be reached twice. If we instead reach the end (null) without any repeat, the list is acyclic and we return null. We store node identities (references/pointers), not values, since values may repeat without forming a cycle.

Python
class ListNode:
    def __init__(self, x):
        self.val = x
        self.next = None

def detectCycle(head):
    seen = set()
    node = head
    while node:
        if node in seen:
            return node
        seen.add(node)
        node = node.next
    return None

Why optimize? The hash-set scan is linear in time but also linear in space — it stores up to n node references. Floyd's two-phase tortoise-and-hare algorithm finds the cycle entry using only two pointers, dropping the extra space to O(1) while keeping O(n) time.

Key InsightA node belongs to the cycle if and only if it can be revisited. The first node you encounter for the second time during a forward walk is exactly the cycle's entry point. The hash set turns "have I been here before?" into an O(1) membership test, so the first repeat is the answer.
How to Think About It
  1. Decide what identifies a node: its reference/pointer, never its value (values can repeat without a cycle).
  2. Walk from head following next, recording each node in a set before advancing.
  3. Before recording, check membership: if the node is already present, it is the cycle entry — return it.
  4. If you reach null, the list terminates, so there is no cycle — return null.
  5. For O(1) space, recall the optimal idea: a slow and fast pointer meet inside the cycle, then a second walk from head locates the entry (Floyd's algorithm — reason about it, don't memorize code).
Why This WorksFollowing next repeatedly either terminates at null (acyclic) or loops forever within the cycle. In the looping case the very first node that repeats must be the entry, because every node before the entry is visited exactly once while the entry is the earliest node reachable from two different predecessors. Recording references rather than values guarantees that "already seen" means literally the same node, not merely an equal value.
Watch Out For
  • Compare node identity, not value — two distinct nodes may share a value.
  • Handle the empty list and a single node with no cycle: both should return null.
  • Do not modify the list (e.g. don't null out next as a marker); the problem forbids mutation.
  • The optimal O(1)-space solution needs two pointers in two phases; getting the second phase's start point wrong returns the wrong node.

#25 Reverse Nodes in k-Group HardLinked ListRecursion

The ProblemGiven the head of a linked list, reverse the nodes of the list k at a time and return the modified list. k is a positive integer less than or equal to the length of the list. If the number of nodes is not a multiple of k, the leftover nodes at the end (fewer than k) remain in their original order. You may not alter the values in the nodes, only the nodes themselves may be changed — though the intended in-place solution rewires next pointers rather than copying values. The list has between 1 and 5000 nodes.
Brute Force — O(n) time, O(n) space

A simple, correct-but-not-space-optimal approach copies all node values into an array, reverses each complete contiguous block of k values (leaving any trailing block shorter than k untouched), then writes the values back into the original nodes in order. This uses O(n) auxiliary space for the array and sidesteps the trickier in-place pointer surgery. The intended optimal solution instead relinks next pointers group by group in O(1) extra space.

Python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reverseKGroup(head, k):
    vals = []
    node = head
    while node:
        vals.append(node.val)
        node = node.next
    n = len(vals)
    for start in range(0, n - n % k, k):
        vals[start:start + k] = vals[start:start + k][::-1]
    node = head
    i = 0
    while node:
        node.val = vals[i]
        i += 1
        node = node.next
    return head

Why optimize? Copying values into an array costs O(n) extra space and technically sidesteps the spirit of the problem (changing nodes, not values). The optimal solution reverses each group in place by rewiring next pointers, achieving O(1) extra space while still running in a single O(n) pass.

Key InsightThe list is just a sequence of fixed-size blocks of length k, plus possibly one short trailing block. Reverse each full block independently and stitch the reversed blocks back together; a block shorter than k is left exactly as-is. The only real work is recognizing complete groups and relinking their boundaries.
How to Think About It
  1. Determine where each group of k nodes starts and ends — only fully complete groups are reversed.
  2. For each complete group, reverse the order of its k nodes (by rewiring next, or, in the brute version, by reversing values).
  3. Reconnect the boundaries: the tail of one reversed group must point to the head of the next group's result.
  4. Use a dummy node before the head so the new overall head (the first group's last node) is easy to return.
  5. When fewer than k nodes remain, stop — leave that final partial block untouched.
  6. Return the node that became the new head after the first group's reversal.
Why This WorksReversing disjoint blocks of k consecutive nodes and concatenating them in order produces exactly the required output, because reversal is local to each block and the relative order of blocks is preserved. Counting k nodes ahead before committing to a reversal guarantees we never reverse an incomplete tail, satisfying the rule that leftover nodes keep their original order. A dummy head removes the special case of the very first group changing the list's head.
Watch Out For
  • Only reverse complete groups: check that k more nodes exist before reversing.
  • The leftover tail (fewer than k nodes) must stay in original order — do not reverse it.
  • Relinking boundaries is the bug-prone part: the old group head becomes the new group tail and must connect to the next group.
  • The intended solution changes nodes, not values; the value-copy approach works but uses O(n) extra space.
