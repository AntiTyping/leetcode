Binary Search

You have 1 problem in this area (#704 Basic Binary Search). These three problems teach you to apply binary search when the array is rotated or two-dimensional. The core principle: at each step, eliminate half the search space.

#33 — Search in Rotated Sorted Array

MediumBinary Search

The Problem

Given a sorted array of unique integers rotated at an unknown pivot (e.g., [4,5,6,7,0,1,2]), and a target value, return the target's index or -1 if not present. Must run in O(log n).

Key Insight

After rotation, the array splits into two sorted halves. When you pick a midpoint, at least one half (left or right of mid) is guaranteed to be perfectly sorted. Compare nums[left] with nums[mid] to identify which half is sorted. Then check if the target falls within that sorted range — if yes, search that half; if no, search the other. This preserves O(log n) even though the whole array isn't sorted.

How to Think About It

Set up standard binary search: left = 0, right = len(nums) - 1, loop while left <= right, compute mid. If nums[mid] == target, return mid.

Determine which half is sorted. If nums[left] <= nums[mid], the left half is sorted. Otherwise the right half is sorted.

If the left half is sorted, check if target is in [nums[left], nums[mid]). If yes, search left (right = mid - 1). Otherwise search right (left = mid + 1).

If the right half is sorted, check if target is in (nums[mid], nums[right]]. If yes, search right. Otherwise search left.

Trace through [4,5,6,7,0,1,2] with target 0 to verify your understanding.

Watch Out For

  • Using nums[left] < nums[mid] (strict) instead of <= — misclassifies the two-element case when left == mid
  • Forgetting to exclude mid from the target range check (you already checked nums[mid] == target)
  • Using left < right instead of left <= right — this is a target search, not a convergence problem

Using strict < instead of <= in nums[left] <= nums[mid]. This bug only manifests with 1–2 elements remaining (when left == mid). It passes most test cases but fails on inputs like [3,1] with target 1.

Two-pass approach: First find the rotation pivot (like #153), then binary search the correct half. Also O(log n). Virtual index mapping: Once you know the pivot, use (mid + pivot) % n to run a standard binary search on the unrotated virtual array.

Python Features That Help

# Chained comparison for clean range check
if nums[left] <= target < nums[mid]:
    right = mid - 1

# Integer midpoint (no overflow in Python)
mid = (left + right) // 2

# Negative indexing for last element
nums[-1]  # quick boundary check

# len() is O(1) on Python lists
n = len(nums)
Time: O(log n)Space: O(1)

Direct successor to #704 (Basic Binary Search). Builds on #704's template by adding sorted-half detection. #153 solves a related subproblem (finding the pivot). #74 extends binary search to 2D.

#153 — Find Minimum in Rotated Sorted Array

MediumBinary Search

The Problem

Given a sorted array of unique integers that has been rotated, find the minimum element in O(log n). Example: [3,4,5,1,2] → 1.

Key Insight

You're searching for the inflection point where the array "breaks" from large values back to small. Compare nums[mid] with nums[right]: if nums[mid] > nums[right], the break is to the right of mid. If nums[mid] <= nums[right], the break is at mid or to its left. You're converging two pointers to a boundary, not searching for a target — this means a different binary search template.

How to Think About It

This is a convergence template: while left < right (strict), not <=.

Compare nums[mid] with nums[right]. If nums[mid] > nums[right], the break is in [mid+1, right], so left = mid + 1.

If nums[mid] <= nums[right], mid could be the minimum. Set right = mid (not mid - 1).

When left == right, the loop exits and nums[left] is the minimum.

Watch Out For

  • Using while left <= right causes an infinite loop (when left == right, setting right = mid changes nothing)
  • Using right = mid - 1 can skip the minimum when mid itself is the answer
  • Comparing with nums[left] instead of nums[right] requires extra handling for unrotated arrays

Mixing up the two binary search templates. In #33, you use while left <= right with right = mid - 1. Here you must use while left < right with right = mid. The templates are NOT interchangeable. Use template 1 for target search, template 2 for boundary convergence.

Python Features That Help

# Early exit for unrotated array
if nums[0] < nums[-1]:
    return nums[0]

# Convergence template
left, right = 0, len(nums) - 1
while left < right:
    mid = (left + right) // 2
    # ... narrow based on nums[mid] vs nums[right]

# divmod for index math (useful in #74)
row, col = divmod(index, num_cols)
Time: O(log n)Space: O(1)

Teaches the convergence binary search template — different from #704 and #33's target-search template. Can be used as a subroutine in #33 (find pivot first, then search). Same rotated-array concept but different comparison target.

#74 — Search a 2D Matrix

MediumBinary SearchMatrix

The Problem

Given an m×n matrix where each row is sorted and the first element of each row is greater than the last element of the previous row, determine if a target exists. Must run in O(log(m*n)).

Key Insight

The entire matrix, read left-to-right top-to-bottom, forms a single sorted sequence. Treat it as a virtual 1D array of length m * n and run standard binary search. Convert 1D index to 2D: row = mid // n, col = mid % n (where n is columns).

How to Think About It

Compute m, n = len(matrix), len(matrix[0]). Total elements: m * n.

Binary search with left = 0, right = m * n - 1.

At each step, convert mid to 2D: row, col = divmod(mid, n). Read matrix[row][col].

Compare to target and adjust bounds. Same as #704.

Watch Out For

  • Dividing by m (rows) instead of n (columns) — works for square matrices but breaks for rectangular
  • Confusing with #240 (Search a 2D Matrix II), which has weaker sorting guarantees and needs a staircase search
  • Empty matrix edge case: check not matrix or not matrix[0]

Confusing m and n in divmod(mid, n). For a 3×4 matrix, index 7 maps to row 7//4=1, col 7%4=3. Using m=3 instead gives row 7//3=2, col 7%3=1 — a completely different cell. Works for square matrices, breaks for rectangular.

Python Features That Help

# divmod for clean 1D-to-2D conversion
row, col = divmod(mid, n)
value = matrix[row][col]

# Quick dimensions
m, n = len(matrix), len(matrix[0])

# Guard against empty matrix
if not matrix or not matrix[0]:
    return False
Time: O(log(m*n))Space: O(1)

Demonstrates binary search on reshaped sorted data. Together, #704 (standard), #33/#153 (rotated), and #74 (dimensional mapping) cover the main binary search variants.

Stack

You have 0 stack problems. In Python, a list is your stack: append() pushes, pop() pops, [-1] peeks. These three problems cover the three fundamental stack patterns: matching (#20), auxiliary tracking (#155), and monotonic stack (#739).

#20 — Valid Parentheses

EasyStack

The Problem

Given a string containing only (){}[], determine if every open bracket is closed by the same type in the correct order.

Key Insight

A stack perfectly models nesting. Push each opener. When you see a closer, the most recent unmatched opener (stack top) must be its match. If it doesn't match or the stack is empty, the string is invalid. At the end, the stack must be empty.

How to Think About It

Create a mapping: {')':'(', '}':'{', ']':'['}.

Walk through each character. If it's a closer, check: is the stack non-empty and does the top match?

If it's an opener, push it.

After all characters, return len(stack) == 0.

Watch Out For

  • Popping from an empty stack (input "]") — always guard with if stack
  • Forgetting the final empty-stack check (input "(" passes all per-char tests but is invalid)

Not checking if the stack is empty before peeking/popping. Input ")(" starts with a closer. Use: if stack and stack[-1] == match[char] where the truthiness check guards against empty.

Python Features That Help

# Dictionary for O(1) bracket matching
match = {')': '(', '}': '{', ']': '['}

# List as stack
stack = []
stack.append('(')   # push
top = stack[-1]      # peek
stack.pop()          # pop

# Empty list is falsy
if stack and stack[-1] == match[char]:
    stack.pop()

# Return truthiness directly
return not stack  # True if empty
Time: O(n)Space: O(n)

Foundation for all stack problems. Teaches the matching pattern. #155 adds auxiliary state tracking. #739 introduces the completely different monotonic stack pattern.

#155 — Min Stack

MediumStackDesign

The Problem

Design a stack supporting push, pop, top, and getMin — all in O(1) time.

Key Insight

The minimum at any stack "level" depends only on what's below it and the value at that level. Store each entry as a (value, current_min) tuple. When you push, the new min is min(val, previous_min). When you pop, the previous min is automatically restored. No recalculation needed.

How to Think About It

A normal stack can't do getMin in O(1) — it requires scanning all elements.

Key realization: min for [a,b,c] only depends on min for [a,b] and value c. This is a running-minimum property.

On push: compute new_min = min(val, stack[-1][1]) if stack else val. Push (val, new_min).

On getMin: return stack[-1][1]. On top: return stack[-1][0].

Watch Out For

  • Maintaining only a single min_val variable — breaks on pop (can't restore previous min)
  • Using a sorted container — makes push/pop O(log n)
  • Empty stack on first push — no previous min to compare with

Maintaining only a single self.min_val. Works for push and getMin, but when you pop the current minimum, you have no way to know the new minimum without scanning — that's O(n). The tuple approach bakes the min into every stack frame.

Python Features That Help

# Tuple packing for stack entries
self.stack = []
self.stack.append((val, current_min))

# Conditional expression for first push
current_min = min(val, self.stack[-1][1]) if self.stack else val

# Tuple unpacking
val, min_at_level = self.stack[-1]

# float('inf') sentinel for separate min stack
self.min_stack = [float('inf')]
Time: O(1) all opsSpace: O(n)

Teaches auxiliary tracking pattern: pairing each element with metadata. #20 used the stack for matching; this uses it for state tracking. #739 will show the third pattern: monotonic stacks.

#739 — Daily Temperatures

MediumMonotonic Stack

The Problem

Given daily temperatures, for each day find how many days until a warmer temperature. Return an array of wait counts. If no warmer day exists, return 0.

Key Insight

This is a "next greater element" problem solved with a monotonic decreasing stack. The stack holds indices of days whose answer hasn't been found yet. When you hit a warmer day, pop all colder days from the stack — the distance is current_index - popped_index. Every index is pushed once and popped once, giving O(n).

How to Think About It

Initialize answer = [0] * n and an empty stack (holds indices).

Iterate left to right. While stack is non-empty and current temp > temp at stack top: pop, set answer for popped index.

Push current index onto stack.

Remaining indices on stack have no warmer day (answer stays 0).

Watch Out For

  • Storing temperatures instead of indices — you need indices to compute distance and set answer
  • Using >= instead of strict > — "warmer" means strictly greater, not equal
  • Thinking this is O(n²) due to the inner while loop — each index is pushed/popped at most once, so total is O(n)

Storing temperatures on the stack instead of indices. You push 73 and later pop it, but can't compute the distance or update the correct answer position. The stack must hold indices; look up temperature via temperatures[stack[-1]].

Python Features That Help

# List multiplication for initialization
answer = [0] * len(temperatures)

# enumerate for index + value
for i, temp in enumerate(temperatures):
    while stack and temp > temperatures[stack[-1]]:
        prev = stack.pop()
        answer[prev] = i - prev
    stack.append(i)

# Empty list is falsy for guard check
while stack and ...  # 'stack' is truthy when non-empty
Time: O(n)Space: O(n)

Introduces the monotonic stack pattern. Same pattern solves #496 Next Greater Element I, #42 Trapping Rain Water, and #84 Largest Rectangle in Histogram. Together with #20 (matching) and #155 (tracking), these cover the three major stack archetypes.

#104 — Maximum Depth of Binary Tree

EasyTreeDFS

The Problem

Given a binary tree, return its maximum depth (number of nodes along the longest root-to-leaf path). Empty tree has depth 0.

Key Insight

The depth of a tree equals 1 + max(depth of left subtree, depth of right subtree). Base case: a null node has depth 0. This single recurrence IS the solution. Trust the recursion.

How to Think About It

Base case: if root is None, return 0.

Ask: "If I knew the depth of left and right subtrees, could I compute mine?" Yes: 1 + max(left, right).

Recurse on root.left and root.right.

Return 1 + max(left_depth, right_depth). That's the whole solution.

Watch Out For

  • Confusing depth with height — LeetCode counts by nodes, not edges. Single node = depth 1.
  • Overthinking it — don't pass depth as a parameter; return it from the recursion.

Wrong base case. The base must be: None returns 0. A leaf naturally returns 1 + max(0,0) = 1. Don't special-case leaves.

Python Features That Help

# Pythonic None check
if not root:
    return 0

# max with two arguments
return 1 + max(left_depth, right_depth)

# BFS alternative (connects to your #102 knowledge)
from collections import deque
queue = deque([root])
Time: O(n)Space: O(h) recursion stack

Gateway to all DFS tree problems. Pattern: "base case for null, recurse left/right, combine." #226 modifies structure in-place. #98 passes constraints downward. #236 uses a sophisticated return value.

#226 — Invert Binary Tree

EasyTreeDFS

The Problem

Given a binary tree, invert it (mirror left-to-right) and return the root.

Key Insight

Swap the left and right children of every node. The recursion order doesn't matter: swap first then recurse (pre-order), or recurse first then swap (post-order). Both produce the same result because every node gets swapped exactly once.

How to Think About It

Base case: if root is None, return None.

Swap children: root.left, root.right = root.right, root.left.

Recurse on both (now-swapped) children.

Return root.

Watch Out For

  • Only swapping at one level — must recurse through entire tree
  • In-order traversal (recurse left, swap, recurse right) is dangerous — you end up recursing on the original left subtree twice

In-order traversal bug. If you recurse left, then swap, then recurse right, the "right" child after swap is the original left — you process it twice and never touch the original right. Use pre-order or post-order.

Python Features That Help

# Simultaneous tuple swap (no temp variable)
root.left, root.right = root.right, root.left
# Python evaluates the entire right side before assignment

# BFS approach reusing deque from #102
from collections import deque
queue = deque([root])
while queue:
    node = queue.popleft()
    node.left, node.right = node.right, node.left
Time: O(n)Space: O(h)

Builds on #104's template. #104 combined return values (depth); this modifies structure in-place. #98 will show information flowing downward through parameters.

#98 — Validate Binary Search Tree

MediumTreeBST

The Problem

Determine if a binary tree is a valid BST. Left subtree values must be strictly less than the node; right subtree values strictly greater. Recursively.

Key Insight

It's NOT enough to check each node against its direct parent. A deep node in the left subtree must be less than ALL ancestors. Pass an allowed range (low, high) downward. The root can be anything (-inf to inf). Going left, the node's value becomes the new upper bound. Going right, it becomes the new lower bound. Check: low < node.val < high.

How to Think About It

Define validate(node, low, high). Base case: None returns True.

Check: if node.val <= low or node.val >= high, return False.

Recurse left: validate(node.left, low, node.val) (tighten upper bound).

Recurse right: validate(node.right, node.val, high) (tighten lower bound).

Initial call: validate(root, float('-inf'), float('inf')).

Watch Out For

  • Only checking immediate children — tree [5,1,6,null,null,3,7] has 3 in the right subtree of 5, which should be > 5
  • Using int min/max as bounds — node values can be -2^31, so use float('-inf')
  • BST property is strict (<, not <=) — equal values are not allowed

Checking only node.left.val < node.val < node.right.val. This is the #1 incorrect approach. A node deep in the right subtree can violate the root's constraint while satisfying its parent's. The range approach propagates ALL ancestor constraints.

In-order traversal: A valid BST produces a strictly increasing sequence. Walk in-order and verify each value > previous. Avoids range parameters entirely.

Python Features That Help

# float sentinels compare correctly with any int
low, high = float('-inf'), float('inf')

# Chained comparison
if not (low < node.val < high):
    return False

# Default parameters for clean initial call
def validate(node, low=float('-inf'), high=float('inf')):
    ...

# Short-circuit with 'and'
return validate(node.left, low, node.val) and validate(node.right, node.val, high)

# Generator-based in-order (alternative)
def inorder(node):
    if node:
        yield from inorder(node.left)
        yield node.val
        yield from inorder(node.right)
Time: O(n)Space: O(h)

Teaches range propagation (information flowing downward). #104/#226 propagated info upward via return values. #236 combines both directions.

#236 — Lowest Common Ancestor of a Binary Tree

MediumTreeDFS

The Problem

Given a binary tree and two nodes p and q (guaranteed to exist), find their lowest common ancestor — the deepest node that is an ancestor of both.

Key Insight

Post-order recursion. For each node, ask: "Is p or q in my left subtree? Right subtree?" If both sides return non-null, p is in one subtree and q in the other — current node is the LCA. If only one side returns non-null, propagate that result up. If the current node IS p or q, return it immediately.

How to Think About It

Base case: if root is None, return None.

If root is p or root is q, return root. (If q is below p, p itself is the LCA.)

Recurse on left and right subtrees.

If both left and right are non-null: current node is the LCA. Return root.

Otherwise: return left or right (propagate the non-null one, or None if both are null).

Watch Out For

  • Comparing values (root.val == p.val) instead of identity (root is p) — use is for node identity
  • Confusing with #235 (LCA of BST) — that uses BST ordering; this must search both subtrees

Using root.val == p.val instead of root is p. Conceptually wrong — the problem gives you object references. In interviews, value comparison shows you don't understand identity vs. equality.

Python Features That Help

# 'is' for object identity
if root is p or root is q:
    return root

# 'or' returns first truthy value
return left or right
# None or TreeNode -> TreeNode
# None or None -> None

# 'and' for both-found check
if left and right:
    return root
Time: O(n)Space: O(h)

Culminating tree problem. Combines upward info flow (return values) with post-order processing. Same dual-purpose DFS pattern appears in #124 Binary Tree Max Path Sum (Hard section).

Backtracking

#78 — Subsets

MediumBacktracking

The Problem

Given an array of unique integers, return all possible subsets (power set). Input: [1,2,3]. Output: [[], [1], [2], [3], [1,2], [1,3], [2,3], [1,2,3]].

Key Insight

Every element presents a binary choice: include it or exclude it. Your backtracking tree is a binary decision tree of depth n, giving 2^n subsets. At each index, branch into two calls: one that adds the element to the path, one that skips it. Alternatively, use a loop from start to n, adding the current path at every node (not just leaves).

How to Think About It

Recursive function takes an index and current path. At each call: include nums[index] or skip it.

Base case: index == len(nums). Add a copy of path to results.

Include branch: append, recurse with index+1, pop (backtrack).

Skip branch: recurse with index+1 without modifying path.

Watch Out For

  • Appending the path reference instead of a copy — result.append(path[:]) not result.append(path). Without copy, every entry points to the same list.
  • Forgetting to pop (backtrack) after the include branch
  • Off-by-one on start index in the loop approach: pass i+1, not start+1

Appending the mutable path list directly to result instead of a copy. This is the most common backtracking bug in Python. Every subset in your result will be an empty list, and it's extremely confusing to debug because the recursion logic can be completely correct.

Python Features That Help

# List copy via slicing (critical for backtracking)
copy = original[:]

# Bit manipulation to enumerate subsets
for mask in range(1 << n):  # 2**n iterations
    bits = [i for i in range(n) if mask & (1 << i)]

# list.pop() for backtracking
path.pop()  # removes and returns last element

# itertools.combinations for all k-subsets
from itertools import combinations
list(combinations([1,2,3], 2))  # [(1,2), (1,3), (2,3)]
Time: O(n · 2^n)Space: O(n) recursion stack

Foundational backtracking problem. Uses start-index to avoid revisiting. #46 (Permutations) uses a used-set instead. #79 (Word Search) applies backtracking on a 2D grid. Extends the branching pattern from #22.

#46 — Permutations

MediumBacktracking

The Problem

Given an array of distinct integers, return all possible permutations. Input: [1,2,3]. Output: all 6 orderings.

Key Insight

Unlike Subsets where you advance a start index, permutations require every element exactly once and order matters. At each level, consider ALL remaining unused elements. Use a used set to track which elements are placed. The branching factor shrinks by 1 at each level: n, then n-1, then n-2, giving n! total.

How to Think About It

Maintain a path and a used set. Loop over all indices; if not used, mark used, append, recurse, pop, unmark.

Base case: len(path) == len(nums). Append a copy to results.

Alternatively: swap approach. At depth d, swap nums[d] with each nums[d..n-1], recurse, swap back.

Watch Out For

  • Using a start index like Subsets — this generates combinations, not permutations. Permutations must consider all unused elements at every level.
  • Forgetting to copy the path
  • Using a list for used instead of a set — O(n) lookup vs O(1)

Using a forward-only start index (the Subsets pattern) instead of iterating over all unused elements. This silently produces combinations instead of permutations. Ask: "Am I generating ordered arrangements (permutations) or unordered selections (subsets)?"

Python Features That Help

# set for O(1) membership testing
used = set()
used.add(2)
used.discard(2)  # safe even if not present

# Tuple unpacking for swapping
nums[0], nums[2] = nums[2], nums[0]

# Boolean array alternative
used = [False] * len(nums)

# itertools.permutations for reference
from itertools import permutations
list(permutations([1, 2, 3]))
Time: O(n · n!)Space: O(n)

Permutations and Subsets (#78) are the two fundamental backtracking templates. Subsets: start-index. Permutations: used-set. #79 (Word Search) is closer to Permutations — tracking visited cells on a grid.

#79 — Word Search

MediumBacktrackingDFS

The Problem

Given an m×n grid of characters and a word, determine if the word exists by following adjacent cells (up/down/left/right). Each cell used at most once per path.

Key Insight

This is backtracking on a 2D grid. Start from every cell matching word[0]. From each cell, explore all four directions. The critical mechanism: mark the current cell as visited before exploring neighbors, and unmark it after returning (backtrack). Unlike Number of Islands where visited cells stay visited permanently, here each path tracks its own visited cells independently.

How to Think About It

Scan for all cells matching word[0]. Each is a potential start.

Define dfs(row, col, index). Base case: index == len(word) - 1 means all matched.

Mark cell visited (change to '#'). Explore 4 neighbors that are in-bounds, unvisited, and match word[index+1].

Restore the cell after exploring (backtrack). Propagate True if any path succeeded.

Watch Out For

  • Not restoring the cell after backtracking — corrupts the board for all future DFS calls
  • Boundary check order: verify in-bounds before accessing board[nr][nc]
  • Optimization: check if board contains all characters in word with sufficient frequency before starting

Not restoring the cell after backtracking. This corrupts the board for all future DFS calls. The in-place mutation approach (board[r][c] = '#' then restore) is simpler and less error-prone than a separate visited set.

Python Features That Help

# Direction deltas for 4-directional traversal
directions = [(0,1), (0,-1), (1,0), (-1,0)]
for dr, dc in directions:
    nr, nc = row + dr, col + dc

# In-place cell marking
original = board[r][c]
board[r][c] = '#'
# ... recurse ...
board[r][c] = original  # restore

# Counter for frequency pre-check
from collections import Counter
board_count = Counter(ch for row in board for ch in row)

# any() for short-circuit
found = any(dfs(nr, nc, idx+1) for nr, nc in neighbors if valid(nr, nc))
Time: O(m·n·3^L) L=word lengthSpace: O(L)

Combines grid traversal from #200 (Number of Islands) with backtracking from #46. Key difference from Islands: here you must un-visit cells. Same backtrack-on-grid pattern appears in #212 Word Search II (which adds a Trie from #208).

#207 — Course Schedule

MediumGraphTopological Sort

The Problem

Given n courses (0 to n-1) and prerequisite pairs [a, b] (must take b before a), determine if you can finish all courses. Essentially: does the dependency graph have a cycle?

Key Insight

This asks whether a directed graph is a DAG. Use DFS with three states per node: UNVISITED, IN_PROGRESS, DONE. If you encounter an IN_PROGRESS node during DFS, you've found a cycle. Two states (visited/unvisited) can't distinguish "currently exploring" from "fully processed" — you need three. Alternatively, Kahn's BFS algorithm peels off zero-in-degree nodes; if fewer than n are processed, a cycle exists.

How to Think About It

Build adjacency list: for [a, b], edge from b to a.

DFS: for each unvisited node, mark IN_PROGRESS, visit neighbors. If neighbor is IN_PROGRESS → cycle.

After processing all neighbors, mark DONE.

BFS alternative (Kahn's): compute in-degrees, queue zero-in-degree nodes, process. Count processed; if < n, cycle exists.

Watch Out For

  • Confusing edge direction — [a, b] means b → a, not a → b
  • Using only two DFS states — can't distinguish cycle (ancestor) from cross-edge (already fully explored)
  • Not handling disconnected components — must attempt DFS from every unvisited node

Using only two DFS states (visited/unvisited). When you encounter a visited node, you can't tell if it's an ancestor on the current path (cycle) or fully explored from a different branch (safe). This causes both false positives and false negatives.

Kahn's BFS: Count processed nodes; if < n, cycle exists. Often simpler to implement correctly. DFS topological sort: Build post-order; verify it has n elements.

Python Features That Help

    graph[b].append(a)

# deque for Kahn's BFS

# Constants for three DFS states

Time: O(V + E)Space: O(V + E)

#743 — Network Delay Time

MediumGraphDijkstra

The Problem

Given a directed weighted graph and a source node k, return the minimum time for ALL nodes to receive a signal. Return -1 if any node is unreachable.

Key Insight

Single-source shortest paths with non-negative weights → Dijkstra's algorithm. The answer is the maximum shortest-path distance, because the signal spreads simultaneously and the last node to receive it determines total delay. If any node is unreachable (distance stays infinity), return -1.

How to Think About It

Build adjacency list. Initialize dist array with infinity; dist[k] = 0.

Min-heap with (0, k). Pop (d, u); if d > dist[u], skip (stale).

For each neighbor (v, w), if dist[u] + w < dist[v], update and push.

After heap is empty: answer = max(dist[1..n]). If any is infinity, return -1.

Watch Out For

  • Nodes are 1-indexed — use dist of size n+1
  • Not skipping stale heap entries (d > dist[u])
  • Using BFS for weighted graph — BFS finds shortest paths only in unweighted graphs
  • Returning sum or source distance instead of max distance

Using BFS instead of Dijkstra. BFS assumes all edges have equal cost. With different edge weights, BFS gives incorrect shortest paths.

Python Features That Help

#208 — Implement Trie (Prefix Tree)

MediumTrieDesign

The Problem

Implement a Trie with insert(word), search(word) (exact match), and startsWith(prefix). All inputs are lowercase English letters.

Key Insight

Each node has a children dict (char → TrieNode) and an is_end boolean. The difference between search and startsWith is exactly one thing: search requires the final node to be marked is_end = True, while startsWith just needs the path to exist. That single boolean flag is the entire difference.

How to Think About It

TrieNode: children = {}, is_end = False.

Insert: walk character by character, creating nodes as needed. Set is_end = True on the last node.

Search: walk the path. If any char missing, return False. At end, return node.is_end.

StartsWith: same walk, but return True at end (don't check is_end).

Watch Out For

  • Forgetting the is_end flag — makes search("app") return True when only "apple" was inserted
  • Mutating the root node — root is a sentinel, doesn't represent any character

Not checking is_end in search. Without it, search behaves identically to startsWith, defeating the entire purpose. search("app") returns True even though only "apple" was inserted.

#547 — Number of Provinces

MediumUnion-FindGraph

The Problem

Given an n×n adjacency matrix where isConnected[i][j] = 1 means cities i and j are directly connected, return the number of provinces (connected components).

Key Insight

Connected components on an adjacency matrix. Union-Find maintains a forest where each tree is a component. For each edge (i, j), union their sets. At the end, count distinct roots. The key operations: find(x) with path compression and union(x, y) with rank give nearly O(1) amortized per operation.

How to Think About It

Initialize parent = list(range(n)) (each city is its own root) and rank = [0] * n.

Iterate pairs (i, j) where i < j and isConnected[i][j] == 1. Call union(i, j).

Find: follow parent pointers to root. Apply path compression.

Union: find roots of both. Merge smaller-rank tree under larger.

Count distinct roots: len({find(i) for i in range(n)}).

Watch Out For

  • Comparing parent[x] == parent[y] instead of find(x) == find(y) — parent may point to intermediate node, not root
  • Forgetting path compression in find — degrades to O(n)
  • Confusing this with a grid problem — it's an adjacency matrix, not a 2D grid
  • Counting roots without calling find — parent[i] may not point directly to root

Comparing parent[x] == parent[y] instead of find(x) == find(y). After unions, two nodes in the same component may not share the same direct parent. Always use find to get the canonical root. This overcounts provinces.

Python Features That Help

# Initialize parent array
parent = list(range(n))
rank = [0] * n

# Path compression (path halving)
def find(x):
    while parent[x] != x:
        parent[x] = parent[parent[x]]
        x = parent[x]
    return x

# Set comprehension to count roots
num_provinces = len({find(i) for i in range(n)})

# enumerate for matrix iteration
for i, row in enumerate(isConnected):
    for j in range(i + 1, n):
        if row[j] == 1:
            union(i, j)
Time: O(n² · α(n)) ≈ O(n²)Space: O(n)

Union-Find counterpart to Number of Islands (#200). Islands uses DFS flood-fill on a grid; this uses Union-Find on an adjacency matrix. Same question: "how many connected components?" Essential for dynamic connectivity problems like #684 Redundant Connection.

#42 — Trapping Rain Water

HardTwo PointersStack

The Problem

Given n non-negative integers representing an elevation map (bar width = 1), compute how much water it can trap after raining. Input: [0,1,0,2,1,0,1,3,2,1,2,1] → 6.

Key Insight

Water above bar i equals min(left_max[i], right_max[i]) - height[i]. The water level at each position depends only on the tallest bars on both sides. Precompute prefix max (left-to-right) and suffix max (right-to-left) in two passes, then sum. No simulation needed — compute each bar's contribution independently.

How to Think About It

Precompute left_max[i]: max height from 0 to i. Left-to-right pass.

Precompute right_max[i]: max height from i to n-1. Right-to-left pass.

For each i: water = max(0, min(left_max[i], right_max[i]) - height[i]).

Sum all water values.

Optimization: two-pointer approach computes the same in one pass with O(1) space. Always advance the pointer with the smaller max.

Watch Out For

  • Computing left_max/right_max exclusive of current index — must include current bar
  • Forgetting max(0, ...) — bars taller than walls give negative values
  • Two-pointer: moving the wrong pointer (must advance the side with smaller max)

In two-pointer approach: advancing the side with the LARGER max. The correctness relies on: if left_max <= right_max, water at left pointer is determined entirely by left_max. Advancing the wrong pointer breaks this invariant.

Python Features That Help

# itertools.accumulate for prefix max
from itertools import accumulate
left_max = list(accumulate(height, max))

# reversed + accumulate for suffix max
right_max = list(accumulate(reversed(height), max))
right_max.reverse()

# zip for paired iteration
total = sum(max(0, min(l, r) - h)
    for l, r, h in zip(left_max, right_max, height))

# Two-pointer template
left, right = 0, len(height) - 1
Time: O(n)Space: O(n) or O(1) with two pointers

Combines your two-pointer strength with prefix-max technique. Monotonic stack approach connects to #739 Daily Temperatures. Often compared with #11 Container With Most Water (which you've solved).

#76 — Minimum Window Substring

HardSliding WindowHash Map

The Problem

Given strings s and t, return the minimum-length substring of s containing all characters of t (including duplicates). Return "" if impossible.

Key Insight

Sliding window with frequency matching. Expand right until window contains all of t. Contract left to minimize. The crucial optimization: track "how many distinct characters are satisfied" with a single counter (formed). When a char's count reaches its required count, increment formed. When formed equals required, the window is valid. Avoid re-checking all counts every step.

How to Think About It

Build need = Counter(t). Let required = len(need) (distinct chars).

Expand right: add s[right] to window counts. If count reaches need[char], increment formed.

While formed == required: record window if smallest. Shrink left: if removing causes count < need, decrement formed.

Return best window, or "" if none found.

Watch Out For

  • Incrementing/decrementing formed at the wrong time — only when count crosses the exact threshold
  • Rebuilding Counter inside the loop — O(window) per step, making total O(n²)
  • Not handling duplicate characters in t (e.g., t="AAB" needs 2 A's)
  • Off-by-one when extracting result substring

Incrementing formed every time you add a char from t, rather than only when that char's count first reaches its target. With t="AAB", formed should increment for 'A' only when count reaches 2, not when it reaches 1.

Python Features That Help

# Counter for frequency counting
from collections import Counter
need = Counter(t)
required = len(need)

# defaultdict(int) for window counts
from collections import defaultdict
window = defaultdict(int)
window['A'] += 1  # no KeyError

# float('inf') for minimum tracking
min_len = float('inf')

# String slicing for result
s[left:right+1]  # inclusive left, exclusive right+1
Time: O(|s| + |t|)Space: O(|s| + |t|)

Extends your strong sliding window skills to a harder "valid window" condition. The formed counter optimization is a general technique: instead of re-validating entire state, maintain one variable tracking how many constraints are met.

#124 — Binary Tree Maximum Path Sum

HardTreeDFS

The Problem

Given a binary tree, find the maximum path sum. A path is any sequence of connected nodes (doesn't need to pass through root). Node values can be negative.

Key Insight

At each node, two different questions. For the global answer: "What's the best path THROUGH this node as the apex?" (left gain + node + right gain). For the parent's recursion: "What's the best path going DOWNWARD from this node?" (node + max of one child). A path through a node uses both children, but a path returned to the parent uses at most one (paths can't fork). The recursion returns the downward value while updating a global variable with the through value.

How to Think About It

Define DFS that returns max gain extendable upward. Compute left_gain = max(0, dfs(left)) and right_gain = max(0, dfs(right)). Clamping to 0 means "don't extend into negative subtree."

Path through this node: node.val + left_gain + right_gain. Update global max.

Return to parent: node.val + max(left_gain, right_gain). Only one direction.

Base case: None returns 0. Initialize global max to float('-inf') (handles all-negative trees).

Watch Out For

  • Returning the "through" value to parent — paths are linear, can't fork. Return only one direction.
  • Not clamping gains to 0 — negative subtrees should be ignored
  • Initializing global max to 0 — fails when all values are negative

Returning node.val + left_gain + right_gain to the parent instead of node.val + max(left_gain, right_gain). This is THE crux. A path is linear — it can't branch. When reporting upward, choose ONE direction. When computing the best path AT this node, use BOTH. Mixing these up produces invalid forking paths.

Python Features That Help

    max_sum = float('-inf')
