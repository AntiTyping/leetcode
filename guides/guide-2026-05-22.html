#88 · Merge Sorted Array

Easy Two Pointers

Quick Reminder

Merge nums2 into nums1 in-place. nums1 has extra space at the end (length m+n). Use three pointers starting from the ends — fill from right to left so you never overwrite unprocessed elements.

Key Insight

By filling from the back (index m+n-1 down to 0), the "write" pointer is always at or beyond any unread element in nums1. Compare nums1[p1] and nums2[p2], place the larger one at the write pointer, and decrement. No extra array needed.

Space: O(1)

#160 · Intersection of Two Linked Lists

Easy Linked List Two Pointers

Quick Reminder

Find the node where two singly linked lists converge. Use two pointers that each traverse both lists: when one reaches the end, redirect it to the head of the other list. They meet at the intersection or both reach null.

Key Insight

Pointer A walks: listA then listB. Pointer B walks: listB then listA. Both travel the same total distance (lenA + lenB), so they arrive at the intersection simultaneously. The "redirect to the other head" trick equalizes the path lengths without counting nodes.

Try to Recall

What is the loop termination condition? Is it while pA != pB or while pA and pB? What do you return when there's no intersection?

Time: O(m + n)
Space: O(1)

Key Insight

#20 · Valid Parentheses

Easy Stack

The Problem

Given a string containing only (), {}, and [], determine if the input string is valid. Every open bracket must be closed by the same type in the correct (nested) order.

Brute Force

Repeatedly scan the string and remove any adjacent matching pair ("()", "[]", "{}"). If the string becomes empty, it was valid. Each pass removes at least one pair, so at most n/2 passes are needed.

Python 
def isValid(s: str) -> bool:
    while '()' in s or '[]' in s or '{}' in s:
        s = s.replace('()', '').replace('[]', '').replace('{}', '')
    return s == ''
Go 
func isValid(s string) bool {
    for strings.Contains(s, "()") || strings.Contains(s, "[]") || strings.Contains(s, "{}") {
        s = strings.ReplaceAll(s, "()", "")
        s = strings.ReplaceAll(s, "[]", "")
        s = strings.ReplaceAll(s, "{}", "")
    }
    return s == ""
}
Time: O(n²)
Space: O(n)

Each replace pass is O(n) and up to n/2 passes are needed. Fails on large inputs where n = 104.

Key Insight

The brute force repeatedly scans the entire string to find pairs. The insight is that a stack processes each character exactly once: push openers, pop-and-match closers. The most recent unmatched opener must match the current closer — which is exactly what LIFO gives you.

How to Think About It

  1. Create a mapping from closing brackets to their corresponding opening brackets: )→(, }→{, ]→[.
  2. Initialize an empty stack (list).
  3. Iterate through each character. If it's an opening bracket, push it onto the stack.
  4. If it's a closing bracket, check: is the stack empty? If yes, return False. Otherwise, pop and verify the popped bracket matches.
  5. After processing all characters, return whether the stack is empty — leftover opening brackets mean invalid.

Why This Works

Valid nesting means the most recent unmatched opener must match the current closer. A stack's LIFO property guarantees exactly this: pop always returns the most recent opener. If it doesn't match, the string is invalid. If the stack is empty when we try to pop, there's a closer without an opener. If the stack isn't empty at the end, there are unmatched openers.

⚠ Most Dangerous Mistake

Forgetting to check if the stack is empty before popping when you encounter a closing bracket. Strings like ")" or "](){" will cause an IndexError (Python) or panic (Go) if you pop from an empty stack.

  • Not checking stack is empty at the end — "(((" would incorrectly return True
  • Mapping in the wrong direction (opening→closing instead of closing→opening)
  • Treating all characters the same — must distinguish openers from closers

#155 · Min Stack

Medium Stack Design

The Problem

Design a stack that supports push, pop, top, and retrieving the minimum element — all in O(1) time.

Brute Force

Use a regular stack for push/pop/top. For getMin, scan the entire stack each time to find the minimum. This gives O(1) for push/pop/top but O(n) for getMin.

Python 
class MinStack:
    def __init__(self):
        self.stack = []
    def push(self, val): self.stack.append(val)
    def pop(self): self.stack.pop()
    def top(self): return self.stack[-1]
    def getMin(self): return min(self.stack)
Go 
type MinStack struct { stack []int }
func (s *MinStack) Push(val int) { s.stack = append(s.stack, val) }
func (s *MinStack) Pop()         { s.stack = s.stack[:len(s.stack)-1] }
func (s *MinStack) Top() int     { return s.stack[len(s.stack)-1] }
func (s *MinStack) GetMin() int {
    m := s.stack[0]
    for _, v := range s.stack { if v < m { m = v } }
    return m
}
Time: O(n) getMin
Space: O(n)

The constraint requires O(1) for all operations including getMin. Scanning defeats the purpose of the data structure.

Key Insight

The brute force rescans the entire stack on every getMin call. The insight is to maintain a second "min stack" alongside the main stack. Every time you push, also push the current minimum (min(val, current_min)) onto the min stack. When you pop, pop from both. The top of the min stack always holds the minimum.

How to Think About It

  1. Use two stacks: stack for values and min_stack for running minimums.
  2. Push: push value to stack. Push min(val, min_stack[-1]) to min_stack (or just val if empty).
  3. Pop: pop from both stacks.
  4. Top: return stack[-1].
  5. GetMin: return min_stack[-1].

Why This Works

The min stack records a "snapshot" of the minimum at each depth level. When elements are popped, the min stack shrinks in sync, revealing the previous minimum. Because every push/pop keeps both stacks in lockstep, the min stack's top is always the minimum of all elements currently in the main stack.

⚠ Most Dangerous Mistake

Only pushing to the min stack when the new value is less than the current minimum. If you push [2, 1, 1] and then pop once, the min stack would be wrong if you skipped the duplicate 1.

  • Forgetting to handle the empty min stack case on the first push
  • Alternative approach stores tuples (val, min) — same idea, one stack instead of two
Python 
# two lists as parallel stacks
self.stack = []
self.min_stack = []

# min() for comparing current value with running minimum
self.min_stack.append(min(val, self.min_stack[-1]))

# alternatively, store tuples: (value, current_min)
self.stack.append((val, min(val, self.stack[-1][1])))

# ternary for first-push edge case
new_min = val if not self.min_stack else min(val, self.min_stack[-1])
Go 
// two slices as parallel stacks
type MinStack struct {
    stack    []int
    minStack []int
}

// push to both stacks
func (s *MinStack) Push(val int) {
    s.stack = append(s.stack, val)
    minVal := val
    if len(s.minStack) > 0 && s.minStack[len(s.minStack)-1] < val {
        minVal = s.minStack[len(s.minStack)-1]
    }
    s.minStack = append(s.minStack, minVal)
}

// pop from both
func (s *MinStack) Pop() {
    s.stack = s.stack[:len(s.stack)-1]
    s.minStack = s.minStack[:len(s.minStack)-1]
}
Time: O(1) all operations
Space: O(n)
Builds on #20's stack fundamentals. Now the stack tracks additional state (the running minimum) alongside its values. Same "auxiliary data structure" idea appears in LRU Cache (#146).

#739 · Daily Temperatures

Medium Stack Monotonic Stack

The Problem

Given an array of daily temperatures, return an array where answer[i] is the number of days you have to wait until a warmer temperature. If no future warmer day exists, put 0.

Brute Force

For each day, scan forward through all subsequent days to find the first warmer one. If found, record the distance; otherwise, record 0.

Python 
def dailyTemperatures(temperatures):
    n = len(temperatures)
    result = [0] * n
    for i in range(n):
        for j in range(i + 1, n):
            if temperatures[j] > temperatures[i]:
                result[i] = j - i
                break
    return result
Go 
func dailyTemperatures(temperatures []int) []int {
    n := len(temperatures)
    result := make([]int, n)
    for i := 0; i < n; i++ {
        for j := i + 1; j < n; j++ {
            if temperatures[j] > temperatures[i] {
                result[i] = j - i
                break
            }
        }
    }
    return result
}
Time: O(n²)
Space: O(n)

With n up to 105, O(n²) times out. The constraint requires O(n).

Key Insight

The brute force rescans forward for every day. The insight is that a monotonic decreasing stack of indices resolves multiple days at once. When a warmer day arrives, pop all cooler days from the stack — the current day is the answer for all of them. Each index is pushed and popped at most once: O(n).

How to Think About It

  1. Initialize result = [0] * n and an empty stack.
  2. Iterate through temperatures with index. The stack holds indices of unresolved days.
  3. While the stack is non-empty AND the current temp is greater than the temp at the stack's top index: pop the index, set result[popped] = i - popped.
  4. Push the current index onto the stack.
  5. Any indices remaining after iteration keep their default 0 — no warmer day exists.

Why This Works

The stack maintains days in decreasing temperature order. When a warmer day arrives, it resolves all consecutive cooler days on the stack. Each day is pushed once and popped at most once, giving O(n). The monotonic property guarantees the first warmer day found is the nearest one.

⚠ Most Dangerous Mistake

Storing temperature values instead of indices on the stack. You need indices to compute the distance and to set the result at the correct position. Always push i, access the temperature via temperatures[stack[-1]].

  • Using ≥ instead of > in the comparison — equal temperatures don't count as "warmer"
  • Forgetting that the default value 0 already handles the "no warmer day" case

#150 · Evaluate Reverse Polish Notation

Medium Stack

The Problem

Evaluate an arithmetic expression in Reverse Polish Notation (postfix). Valid operators are +, -, *, /. Each operand may be a multi-digit integer, possibly negative. Division truncates toward zero.

Brute Force

Repeatedly find the first operator in the token list, replace it and its two preceding operands with the computed result, and repeat until one token remains. This simulates evaluation by scanning left-to-right for the first reducible pattern.

Python 
def evalRPN(tokens):
    tokens = list(tokens)
    ops = {'+', '-', '*', '/'}
    while len(tokens) > 1:
        for i, t in enumerate(tokens):
            if t in ops:
                a, b = int(tokens[i-2]), int(tokens[i-1])
                if t == '+': res = a + b
                elif t == '-': res = a - b
                elif t == '*': res = a * b
                else: res = int(a / b)
                tokens = tokens[:i-2] + [str(res)] + tokens[i+1:]
                break
    return int(tokens[0])
Go 
func evalRPN(tokens []string) int {
    ops := map[string]bool{"+":true,"-":true,"*":true,"/":true}
    for len(tokens) > 1 {
        for i, t := range tokens {
            if ops[t] {
                a, _ := strconv.Atoi(tokens[i-2])
                b, _ := strconv.Atoi(tokens[i-1])
                var res int
                switch t {
                case "+": res = a + b
                case "-": res = a - b
                case "*": res = a * b
                case "/": res = a / b
                }
                tokens = append(append(tokens[:i-2], strconv.Itoa(res)), tokens[i+1:]...)
                break
            }
        }
    }
    r, _ := strconv.Atoi(tokens[0]); return r
}
Time: O(n²)
Space: O(n)

Rescanning and rebuilding the list on each reduction is O(n) per step with up to n/2 steps. Stack-based evaluation is O(n).

Key Insight

The brute force rescans to find the first operator each time. The insight is that RPN is designed for stack evaluation: push numbers, and when you see an operator, the two operands are already on top. Pop two (b first, then a), compute a op b, push result. One pass, O(n).

How to Think About It

  1. Initialize an empty stack.
  2. Iterate through tokens. If the token is a number (can be negative like "-11"), push it as an integer.
  3. If the token is an operator: pop b (second operand), pop a (first operand). Compute a op b. Push the result.
  4. The order matters: a - b and a / b, where a was pushed first (deeper in stack).
  5. For division, truncate toward zero: use int(a / b) in Python (not //).
  6. After all tokens, the stack has exactly one element — the answer.

Why This Works

RPN encodes the evaluation order of an expression without parentheses. Each operator implicitly binds to its nearest two preceding values. The stack accumulates intermediate results in exactly the right order because operands are pushed before their operator appears.

⚠ Most Dangerous Mistake

Python's // (floor division) truncates toward negative infinity, not toward zero. -7 // 2 = -4 in Python, but the problem requires -3. Use int(a / b) instead. Go's integer division already truncates toward zero.

  • Popping operands in wrong order — b is popped first, a second, compute a op b
  • Forgetting that tokens like "-11" are negative numbers, not minus followed by 11
  • Not converting tokens to integers before pushing (comparing strings instead of numbers)
Python 
# truncate toward zero (NOT floor division)
result = int(a / b)   # correct: int(-7/2) = -3
result = a // b       # wrong:   -7//2 = -4

// strconv.Atoi for string-to-int conversion

}

Time: O(n)
Space: O(n)
Classic "evaluate with a stack" problem. Same push/pop-on-condition pattern as #20. The operand order (a before b) is the same care needed in #224 Basic Calculator.

#74 · Search a 2D Matrix

Medium Binary Search Matrix

The Problem

Given an m×n matrix where each row is sorted left-to-right and the first element of each row is greater than the last element of the previous row, determine if a target value exists in the matrix.

Brute Force

Scan every element in the matrix and check if it equals the target. Simple but ignores all the sorted structure.

Python 
def searchMatrix(matrix, target):
    for row in matrix:
        for val in row:
            if val == target:
                return True
    return False
Go 
func searchMatrix(matrix [][]int, target int) bool {
    for _, row := range matrix {
        for _, val := range row {
            if val == target { return true }
        }
    }
    return false
}
Time: O(m × n)
Space: O(1)

With m, n up to 100, this works but doesn't leverage the sorted property. Binary search achieves O(log(m·n)).

Key Insight

The brute force ignores sorted order. The insight is that the entire matrix is one sorted 1D array in row-major order. A virtual index mid in range [0, m*n - 1] maps to matrix[mid // n][mid % n]. Apply standard binary search on this virtual array.

How to Think About It

  1. Compute total elements: m * n. Set left = 0, right = m*n - 1.
  2. Standard binary search: compute mid = (left + right) // 2.
  3. Convert mid to 2D coordinates: row = mid // n, col = mid % n.
  4. Compare matrix[row][col] with target. Adjust left/right accordingly.
  5. This is exactly the #704 template with one extra step (index conversion).

Why This Works

Because each row's first element exceeds the previous row's last element, the matrix is one long sorted array in row-major order. The // n and % n conversion is just the inverse of row-major flattening. Binary search's O(log n) guarantee applies to the entire flattened array.

⚠ Most Dangerous Mistake

Using m (rows) instead of n (columns) in the index conversion. mid // n gives the row and mid % n gives the column. Swapping m and n produces wrong coordinates.

  • Edge case: empty matrix — check m == 0 or n == 0 before searching
  • Alternative: binary search for row first, then binary search within row — same complexity, more code
Python 
# divmod for simultaneous division and modulo
row, col = divmod(mid, n)

# standard binary search bounds
left, right = 0, m * n - 1

# mid calculation (overflow-safe in Python — arbitrary precision)
mid = (left + right) // 2

# len() for matrix dimensions
m, n = len(matrix), len(matrix[0])
Go 
// index conversion
row, col := mid/n, mid%n

// standard binary search
left, right := 0, m*n-1
for left <= right {
    mid := left + (right-left)/2  // overflow-safe
    val := matrix[mid/n][mid%n]
}

// matrix dimensions
m, n := len(matrix), len(matrix[0])
Time: O(log(m·n))
Space: O(1)
Direct extension of #704 Binary Search — same template, just add coordinate conversion. Also compare to #240 Search a 2D Matrix II, where rows/columns are sorted but the flattening trick doesn't apply.

#33 · Search in Rotated Sorted Array

Medium Binary Search

The Problem

Given a sorted array rotated at some unknown pivot (e.g., [4,5,6,7,0,1,2]), search for a target value. All values are unique. Return the index or -1. Must run in O(log n).

Brute Force

Linear scan through the array looking for the target. Ignores the sorted-and-rotated structure entirely.

Python 
def search(nums, target):
    for i, val in enumerate(nums):
        if val == target:
            return i
    return -1
Go 
func search(nums []int, target int) int {
    for i, val := range nums {
        if val == target { return i }
    }
    return -1
}
Time: O(n)
Space: O(1)

The constraint explicitly requires O(log n). Linear scan fails that requirement.

Key Insight

The brute force scans every element. The insight is that at any mid point in a rotated sorted array, at least one half is always properly sorted. Identify which half is sorted by comparing nums[left] with nums[mid]. Once you know which half is sorted, a simple range check tells you which half to search.

How to Think About It

  1. Set left = 0, right = n - 1. Standard binary search loop.
  2. Compute mid. If nums[mid] == target, return mid.
  3. Check if the left half is sorted: nums[left] <= nums[mid].
  4. If left half is sorted: check if target is in [nums[left], nums[mid]). If yes, search left. Otherwise, search right.
  5. If right half is sorted: check if target is in (nums[mid], nums[right]]. If yes, search right. Otherwise, search left.
  6. The <= in step 3 handles the case where left == mid (two-element subarray).

Why This Works

A rotation splits the sorted array into two sorted halves. At any mid point, the rotation break can be in at most one half — the other is guaranteed sorted. By identifying the sorted half, you can determine if the target could be in it (simple range check). If not, it must be in the other half. This preserves the O(log n) elimination guarantee.

⚠ Most Dangerous Mistake

Using < instead of <= when checking nums[left] <= nums[mid]. When left == mid (subarray of size 2), < would incorrectly classify the left half as unsorted, sending the search in the wrong direction.

  • Off-by-one in the range check: use nums[left] <= target < nums[mid] (exclusive on right) for the left-sorted case
  • Forgetting the nums[mid] == target check at the top of the loop
  • Not handling single-element arrays (works automatically with left <= right)

# standard binary search skeleton
while left <= right:
    mid = (left + right) // 2

# No need for: a <= x and x < b

#153 · Find Minimum in Rotated Sorted Array

Medium Binary Search

The Problem

Given a sorted array rotated at some pivot (all values unique), find the minimum element. Must run in O(log n).

Brute Force

Scan the entire array and track the minimum. Completely ignores the rotated-sorted structure.

Python

    return min(nums)

    for _, v := range nums {
        if v < m { m = v }
    }
    return m
}
Time: O(n)
Space: O(1)

The constraint explicitly requires O(log n). The sorted-and-rotated structure enables binary search.

Key Insight

The brute force scans everything. The insight is to compare nums[mid] with nums[right]. If nums[mid] > nums[right], the minimum is in the right half (the rotation break is there). Otherwise, the minimum is in the left half including mid. Always compare against the right endpoint, not the left.

How to Think About It

  1. Set left = 0, right = n - 1.
  2. While left < right (strict inequality, not <=).
  3. If nums[mid] > nums[right]: the rotation break is in (mid, right], so left = mid + 1.
  4. Otherwise: mid itself might be the minimum, so right = mid (not mid - 1).
  5. When left == right, that's the minimum.

Why This Works

The minimum is the point where the sorted order "breaks." Comparing nums[mid] with nums[right] tells you which side the break is on: if mid is greater than right, the break must be between mid and right. If mid is less than or equal to right, that half is properly sorted and the break is on the other side (or at mid itself). Using right = mid (not mid - 1) ensures we never skip the minimum.

⚠ Most Dangerous Mistake

Comparing with nums[left] instead of nums[right]. When the array is not rotated at all (e.g., [1,2,3,4,5]), nums[mid] > nums[left] is true but the minimum is on the left, not the right. Comparing against right handles both rotated and non-rotated cases correctly.

  • Using while left <= right instead of while left < right — causes infinite loop because right = mid never terminates when left == right
  • Using right = mid - 1 instead of right = mid — might skip the minimum element
  • Not handling the non-rotated case — the algorithm handles it automatically, no special case needed

#104 · Maximum Depth of Binary Tree

EasyTreeDFSBFS

The Problem

Given the root of a binary tree, return its maximum depth — the number of nodes along the longest path from the root down to the farthest leaf node.

Brute Force / Natural Approach

For this problem, the natural recursive DFS is the standard algorithm — there is no "worse" brute force to contrast. The straightforward recurrence depth(node) = 1 + max(depth(left), depth(right)) visits every node exactly once and is already optimal in time. However, for skewed trees (every node has only one child), the recursion stack reaches O(n) depth, which can cause a stack overflow. The iterative BFS alternative uses O(n) space in the queue but avoids deep call stacks.

Python — Recursive (Natural) 
def maxDepth(self, root):
    if not root:
        return 0
    return 1 + max(self.maxDepth(root.left), self.maxDepth(root.right))
Go — Recursive (Natural) 
func maxDepth(root *TreeNode) int {
    if root == nil { return 0 }
    left := maxDepth(root.Left)
    right := maxDepth(root.Right)
    return 1 + max(left, right)
}
Time: O(n)
Space: O(h) — O(n) for skewed trees

Why it can be insufficient: A skewed tree (linked-list shape) creates O(n) recursion depth, risking stack overflow in languages with limited stack size.

Key Insight

The recursive approach uses O(h) stack space implicitly. The insight is that BFS level-by-level counting eliminates deep recursion — each level processed adds 1 to depth, and the queue breadth stays bounded by the widest level rather than the tree height.

How to Think About It — Iterative BFS Alternative

  1. If root is None, return 0. Otherwise initialize a queue with root and depth = 0.
  2. While queue is non-empty, process all nodes at the current level.
  3. For each node in the current level, enqueue its non-None children.
  4. Increment depth after processing each complete level.
  5. When the queue empties, depth holds the maximum depth.

Why This Works

Every node adds exactly 1 to the depth. The recursive approach propagates the longest path upward via max. The BFS approach counts levels directly — each iteration of the outer loop represents one tree level. Both visit every node exactly once, so both are O(n) time. The BFS trades recursion stack for a queue, avoiding stack overflow on skewed trees while using at most O(w) queue space where w is the maximum width.

⚠ Most Dangerous Mistake

Off-by-one: returning 1 for a None node instead of 0. This makes every depth one too high. The base case must be return 0 for None.

  • Using recursion on extremely deep trees (100,000+ nodes in a chain) without checking stack limits
  • BFS: incrementing depth inside the inner loop instead of after processing the entire level
  • Confusing depth (node count) with edge count — some problems define depth by edges, this one uses node count

#226 · Invert Binary Tree

EasyTreeDFS

The Problem

Given the root of a binary tree, invert it — swap the left and right children of every node — and return the root.

Brute Force / Natural Approach

Like #104, the natural recursive approach is the standard algorithm — every node must be visited and its children swapped, so O(n) is inherently required. The recursive DFS swaps children at each node and recurses into both subtrees. For skewed trees, O(n) recursion depth can cause stack overflow, making the iterative BFS alternative preferable for robustness.

Python — Recursive (Natural)

    if not root:
        return None
    root.left, root.right = root.right, root.left
    self.invertTree(root.left)
    self.invertTree(root.right)
    return root
Go — Recursive (Natural)

    if root == nil { return nil }
    root.Left, root.Right = root.Right, root.Left
    invertTree(root.Left)
    invertTree(root.Right)
    return root

Time: O(n)
Space: O(h) — O(n) for skewed trees

Why it can be insufficient: A skewed tree produces O(n) call stack depth. The iterative BFS version uses a queue instead, avoiding deep recursion.

Key Insight

The recursive approach swaps children via the call stack. The insight is that any traversal order works (pre-order, post-order, level-order) because swapping is a local operation independent of subtree state. An iterative BFS avoids recursion depth issues by processing nodes level by level.

How to Think About It

  1. Base case: if node is None, return None.
  2. Swap the left and right children: node.left, node.right = node.right, node.left.
  3. Recursively invert the (now-swapped) left subtree.
  4. Recursively invert the (now-swapped) right subtree.
  5. Return the node (the tree is modified in place, but returning is required by the API).
  6. Alternative: use a queue (BFS). Dequeue a node, swap its children, enqueue the non-None children.

Why This Works

Inverting means mirroring: every left child becomes a right child and vice versa. Since the swap at each node is independent of the swap at any other node, any traversal order correctly produces the full mirror. The recursion visits every node exactly once, and each swap is O(1), yielding O(n) total work.

⚠ Most Dangerous Mistake

Swapping children after recursing into the original left and right, but then not accounting for the fact that pointers have changed. If you first recurse into root.left, then swap, root.left now points to what was root.right. Both swap-first and recurse-first orderings work, but you must be consistent about which subtrees you recurse into after the swap.

  • Forgetting to return root — the LeetCode API expects the root returned
  • Only swapping at one level (not recursing) — inverts only the root's immediate children
  • Trying to create new nodes instead of swapping pointers — unnecessary and error-prone

#98 · Validate Binary Search Tree

MediumTreeBSTDFS

The Problem

Given the root of a binary tree, determine if it is a valid binary search tree. A valid BST requires that for every node, all values in its left subtree are strictly less and all values in its right subtree are strictly greater.

Brute Force Approach

Perform an in-order traversal to collect all values into a list, then check if the list is strictly increasing. This works because in-order traversal of a valid BST produces sorted output. It requires O(n) extra space for the list and two full passes (traverse + verify).

Python — Brute Force 
def isValidBST(self, root):
    vals = []
    def inorder(node):
        if not node: return
        inorder(node.left)
        vals.append(node.val)
        inorder(node.right)
    inorder(root)
    return all(vals[i] < vals[i+1] for i in range(len(vals)-1))

Space: O(n) for the list

Why it's insufficient: Storing all n values wastes O(n) space. For very large trees, this extra allocation is unnecessary since validation can be done in a single pass with O(h) stack space.

Key Insight

The brute force collects all values then checks sorted order — two separate steps using O(n) space. The insight is that passing valid bounds (low, high) down during recursion validates each node in-place with O(h) space. Each node inherits constraints from every ancestor, not just its parent.

How to Think About It — Optimal

  1. Define a helper validate(node, low, high) starting with (-inf, +inf).
  2. Base case: None is valid (return True).
  3. Check: if node.val <= low or node.val >= high, return False.
  4. Recurse left with updated upper bound: validate(node.left, low, node.val).
  5. Recurse right with updated lower bound: validate(node.right, node.val, high).
  6. Return True only if both subtrees are valid.

Why This Works

The BST property is transitive: a node deep in the left subtree must be less than every ancestor it's a left descendant of. By propagating bounds downward, each node's constraint incorporates all ancestor constraints, not just the immediate parent. The bounds tighten as we descend — going left tightens the upper bound, going right tightens the lower bound. This single-pass approach verifies the global BST property with only O(h) stack space.

⚠ Most Dangerous Mistake

Only checking against the immediate parent: verifying left.val < node.val and right.val > node.val. A tree like [5, 1, 4, null, null, 3, 6] has node 3 in the right subtree of 5's right child 4 — valid locally (3 < 4) but invalid globally (3 < 5 violates the right-subtree constraint).

  • Using <= instead of < for bounds — BST requires strictly less/greater, not equal
  • Not using -inf/+inf as initial bounds — using a fixed number like -10^9 fails if node values hit those extremes
  • In-order approach: using <= instead of < when checking consecutive values — equal values are invalid in a BST
Alternative: In-order traversal checking only the previous value (O(1) extra space variant — track just the last visited value instead of the full list).

Why it's insufficient: Always traverses all n nodes even when k = 1. For k = 1, we only need the leftmost node. Also stores all n values in a list unnecessarily. When k is much smaller than n, this wastes both time and space.

Key Insight

The brute force traverses the entire tree and stores all values. The insight is that iterative in-order traversal with early termination stops after exactly k nodes, taking O(h + k) time with O(h) space. The explicit stack replaces recursion and makes early return clean.

How to Think About It — Optimal

  1. Initialize an empty stack and set current = root.
  2. While stack is non-empty or current is not None: push all left children onto the stack (go as far left as possible).
  3. Pop from the stack — this is the next smallest element in sorted order.
  4. Decrement k. If k reaches 0, return the current node's value immediately.
  5. Set current to the popped node's right child and repeat.

Why This Works

The BST property guarantees that in-order traversal visits nodes in ascending order. The iterative approach with an explicit stack simulates the recursion, and we count how many nodes we've visited. When the count reaches k, we've found the k-th smallest. The iterative structure makes early termination natural — we return immediately without unwinding a recursive call stack. Time is O(h + k): O(h) to reach the leftmost node, then O(k) pops to reach the k-th.

⚠ Most Dangerous Mistake

Off-by-one with k's indexing. k is 1-indexed (k=1 means the smallest). Decrement k after visiting each node and check k == 0, not k == 1. Alternatively, some people pre-decrement k to 0-indexed, leading to confusion about when to stop.

  • Using recursive in-order without early termination — traverses the full tree even for k=1
  • Forgetting to move to current.right after popping — this skips entire right subtrees
  • Checking k == 1 before decrementing instead of after — returns the (k-1)-th element

#236 · Lowest Common Ancestor of a Binary Tree

MediumTreeDFS

The Problem

Given a binary tree (not necessarily a BST) and two nodes p and q that are guaranteed to exist in the tree, find their lowest common ancestor (LCA). The LCA is the lowest node that has both p and q as descendants (a node can be a descendant of itself).

Brute Force Approach

Build the path from root to p and the path from root to q using DFS. Then compare the two paths from the beginning — the last common node before they diverge is the LCA. This requires two DFS traversals and storing both paths.

Python — Brute Force 
def lowestCommonAncestor(self, root, p, q):
    def findPath(node, target, path):
        if not node: return False
        path.append(node)
        if node == target: return True
        if findPath(node.left, target, path) or findPath(node.right, target, path):
            return True
        path.pop()
        return False
    path_p, path_q = [], []
    findPath(root, p, path_p)
    findPath(root, q, path_q)
    lca = root
    for a, b in zip(path_p, path_q):
        if a == b: lca = a
        else: break
    return lca

Why it's insufficient: Requires two separate DFS passes plus O(n) space for storing both paths. The optimal single-pass approach uses O(h) stack space and finds the LCA during a single traversal.

Key Insight

The brute force builds and compares two paths — two traversals and O(n) path storage. The insight is that a single post-order DFS can detect p and q bubbling up from subtrees. When a node receives non-None from both its left and right recursive calls, that node is the LCA. This eliminates path storage entirely.

How to Think About It — Optimal

  1. Base case: if node is None, return None. If node is p or q, return node (found one target).
  2. Recursively search the left subtree and the right subtree.
  3. If both return non-None: p and q are split across subtrees — current node is the LCA.
  4. If only one returns non-None: propagate that result upward (both targets are in one subtree).
  5. If both return None: neither p nor q is in this subtree — return None.

Why This Works

The LCA is the deepest node where paths to p and q diverge. The post-order DFS finds p and q and bubbles their "signals" upward. The first node to receive signals from both subtrees is exactly the divergence point. The key case to verify: if p is an ancestor of q, then p itself is the LCA. The base case if node == p or node == q: return node handles this — p is returned before q is even searched for, correctly identifying p as the LCA.

⚠ Most Dangerous Mistake

Not handling the case where p is an ancestor of q (or vice versa). Some people add extra logic to search below p for q, but the base case if node == p or node == q: return node already handles this correctly — when p is the LCA, it gets returned before the DFS descends further.

  • Searching only one subtree — both must be searched; the LCA depends on where p and q are found
  • Returning None when only one subtree returns non-None — you must propagate the non-None result
  • Assuming BST ordering — this is a general binary tree, not a BST; you cannot use value comparisons
  • Confusing this with #235 (LCA of BST) where value-based navigation is possible

Key Insight

The bitmask approach generates subsets without structure, making pruning impossible. The insight is that backtracking with an "index forward" pattern builds subsets incrementally — at each position, either include or skip the element. This tree structure allows pruning branches when extended to constrained variants.

How to Think About It — Backtracking

  1. Design a recursive function backtrack(start, current) where start is the next index to consider.
  2. At every call, immediately record a copy of current in the results (every partial subset is valid).
  3. From index start onward, try adding each remaining element: append it, recurse with i+1, then pop it (backtrack).
  4. The "index forward" pattern (starting from start, not 0) prevents generating duplicate subsets like [1,2] and [2,1].
  5. Verify: for n=3, the decision tree has 23 = 8 leaves, one per subset.

Why This Works

The algorithm traces a decision tree. At each level, we choose "include element at index i" for each i from start to n-1. Because we only look forward (never backward), each subset is generated exactly once: elements appear in the order they do in the input. The append-recurse-pop pattern ensures the current list is correctly restored for the next sibling branch. Every node in the decision tree corresponds to a valid subset, so recording at every node (not just leaves) captures all 2n subsets.

⚠ Most Dangerous Mistake

  • Only recording subsets at leaves (when start == n) — misses all subsets shorter than n

Key Insight

The brute force generates all sequences then filters for uniqueness. The insight is that a used boolean array enforces uniqueness during construction — at each position, only unused elements are candidates. This prunes the search tree from nn to n! immediately, eliminating all wasted work.

How to Think About It — Backtracking

  1. Create a used boolean array of size n, all initialized to False.
  2. Define backtrack(). Base case: if len() == n, append a copy to results.
  3. Iterate all indices 0 to n-1. Skip any index where used[i] is True.
  4. Mark used[i] = True, append nums[i] to current, recurse.
  5. After recursion returns: pop from current, set used[i] = False (backtrack).
  6. The used array replaces the "index forward" pattern from #78 — permutations need access to all elements, not just forward ones.

Why This Works

At recursion depth k, we choose from n-k unused elements. Since every element appears exactly once per permutation and we explore all valid choices at each level, every root-to-leaf path produces a distinct permutation. The total number of leaves is n! (n choices at level 0, n-1 at level 1, etc.). The used array ensures no element is selected twice within the same path, while backtracking (resetting used) ensures it's available for other paths.

⚠ Most Dangerous Mistake

Failing to reset used[i] = False after recursion. Since the used array is shared mutable state, forgetting this causes subsequent sibling branches to see elements as "taken" — producing far fewer permutations than n! with no runtime error to alert you. The bug silently drops permutations.

  • Not copying current before appending to results — same mutable reference bug as #78
  • Using "index forward" (like #78) instead of a used array — produces subsets, not permutations
  • Checking len() > n as base case — should be == n; using > wastes a recursion level
  • Go: appending to the captured slice without copying — subsequent appends may overwrite stored results

#79 · Word Search

MediumBacktrackingGridDFS

The Problem

Given an m×n grid of characters and a string word, return true if the word can be found in the grid by tracing a path through adjacent cells (up, down, left, right). Each cell may be used at most once per path.

Brute Force — DFS with Visited Set

From every starting cell, try all 4 directions at each step, using a hash set to track visited cells. The set-based approach works correctly but uses O(L) extra space for the set (where L is word length). The character-match check at each step prunes most branches.

Python — Brute Force (Visited Set) 
def exist(self, board, word):
    rows, cols = len(board), len(board[0])
    def dfs(r, c, idx, visited):
        if idx == len(word): return True
        if r < 0 or r >= rows or c < 0 or c >= cols: return False
        if (r, c) in visited: return False
        if board[r][c] != word[idx]: return False
        visited.add((r, c))
        for dr, dc in [(0,1),(0,-1),(1,0),(-1,0)]:
            if dfs(r+dr, c+dc, idx+1, visited): return True
        visited.remove((r, c))
        return False
    for r in range(rows):
        for c in range(cols):
            if dfs(r, c, 0, set()): return True
    return False

Key Insight

The basic DFS explores all paths with character-match pruning. The insight is that in-place visited marking (overwriting cells with '#' instead of using a set) saves O(L) space, and character frequency pre-checking can prune entire search attempts. The backtracking pattern (mark, recurse, restore) prevents cell reuse within a path while allowing it for other paths.

How to Think About It — Optimized Backtracking

  1. Optional pre-check: verify every character in word exists in the board with sufficient frequency. If the last character is rarer than the first, reverse the word.
  2. Iterate every cell in the grid. If it matches word[0], launch DFS from that cell.
  3. DFS takes (r, c, index). Immediately return False if: out of bounds, cell doesn't match word[index], or cell is already visited.
  4. If index == len(word) - 1 and the character matches, return True (found the complete word).
  5. Mark the cell as visited by overwriting with '#'. Try all 4 neighbors with index + 1.
  6. After exploring all neighbors, restore the cell's original character (backtrack).

Why This Works

The DFS explores every possible path of adjacent cells that could spell the word. The character-match check at each step prunes branches where the path diverges from the word — in practice, most branches terminate at the first or second character. The in-place '#' marking prevents a cell from being reused within the same path. Restoring the character after recursion (backtracking) ensures the cell is available for different paths starting from different cells. Since each step eliminates at most one neighbor (the parent), the branching factor is 3 instead of 4.

⚠ Most Dangerous Mistake

Not restoring the visited marker after a failed path. If you overwrite a cell with '#' and don't restore it, the cell becomes permanently blocked for all future DFS paths — causing false negatives. The word might exist via a different path that needs that cell, but it's been permanently removed. Always restore: board[r][c] = original_char.

  • Using a separate visited set instead of in-place marking — works but uses O(L) extra space; in-place '#' is O(1)
  • Not short-circuiting: continuing to explore neighbors after one returns True
  • Forgetting bounds checking — accessing board[-1][c] in Python silently wraps around instead of being out of bounds
  • Creating a new set() for each DFS launch instead of sharing and restoring — wastes time on allocation

Key Insight

The brute force uses a separate visited flag and path set, which is error-prone. The insight is a three-color state (0=unvisited, 1=in-progress, 2=done) merges both into one array. State 1 replaces the path set; state 2 replaces visited — cleaner and impossible to forget the path.remove() call.

How to Think About It

  1. Build an adjacency list: for [a, b], add edge b → a.
  2. Create a state array of size numCourses, all 0 (unvisited).
  3. For each unvisited node, launch DFS. Mark it as 1 (in-progress) before visiting neighbors.
  4. If a neighbor has state 1 → back edge → cycle → return False. If state 2, skip (done).
  5. After processing all neighbors, mark as 2 (done).
  6. If all nodes complete without finding a cycle, return True.

Why This Works

A directed graph has a cycle iff DFS finds a back edge (edge to a node on the current DFS stack). State 1 marks "on the stack." State 2 marks "fully explored." The three-state approach is essential: with only two states (visited/unvisited), cross-edges to completed nodes would be misidentified as cycles, producing false positives. Every edge is examined exactly once, and every node transitions through states exactly once.

⚠ Most Dangerous Mistake

Using only two states (visited/unvisited) without a separate path tracking mechanism. In a directed graph, encountering a previously-visited node does NOT mean there is a cycle — it could be a cross-edge. The three-state system distinguishes between "currently being explored" (cycle) and "fully done" (safe).

  • Building the adjacency list in the wrong direction: [a, b] means b is prerequisite OF a, so the edge is b → a
  • Not launching DFS from every unvisited node — disconnected components contain possible cycles
  • Forgetting to mark state as 2 after finishing all neighbors — makes every subsequent visit look like a cycle

The Problem

Key Insight

The brute force uses two passes — one to create nodes, one to wire neighbors. The insight is you can do both in a single DFS: create the clone, register it in a map immediately, then recursively clone neighbors. The map registration before recursion is what breaks cycles in a single pass.

How to Think About It

  1. Base case: if node is None, return None.
  2. If node is already in the clone registry (hash map), return its clone — this breaks cycles.
  3. Create a new clone node with the same value.
  4. Register it in the map immediately — before processing neighbors.
  5. For each neighbor, recursively clone and append to the clone's neighbor list.
  6. Return the clone.

Why This Works

The hash map serves as both a visited set and a clone registry. When DFS encounters a node it already cloned (via a cycle), the lookup returns the existing clone immediately, preventing infinite recursion. Registering BEFORE recursing is critical — if you wait until after processing neighbors, a cycle back to the current node finds no entry and recurses infinitely. Every node is created exactly once and every edge is traversed exactly once.

⚠ Most Dangerous Mistake

Registering the clone AFTER processing neighbors. The sequence must be: (1) create clone, (2) register in map, (3) recurse into neighbors. If steps 2 and 3 are swapped, any cycle causes infinite recursion and stack overflow.

  • Forgetting to handle the None input case (empty graph)
  • Using node values as keys instead of node references — values may not be unique
  • Creating duplicate clones by not checking the map before creating a new node

Bellman-Ford does V-1 full passes over all edges. With V up to 100 and E up to 6000, this is fine for the constraints, but Dijkstra is the standard approach for non-negative weight graphs and scales far better.

Key Insight

The brute force (Bellman-Ford) re-scans all edges V-1 times regardless of which nodes have been finalized. The insight is that with non-negative weights, Dijkstra's algorithm uses a min-heap to always process the nearest unfinalized node first — each node is finalized exactly once, and edges are only relaxed from finalized nodes.

How to Think About It

  1. Build a weighted adjacency list from the edge list.
  2. Initialize distances to infinity. Set dist[k] = 0.
  3. Push (0, k) to a min-heap.
  4. Pop minimum: (d, u). If d > dist[u], skip — this is a stale entry.
  5. For each neighbor v with weight w: if dist[u] + w < dist[v], update dist[v] and push (dist[v], v).
  6. After the heap is empty, if any node is still infinity, return -1. Otherwise return max(dist) — the last node to receive the signal.

Why This Works

Dijkstra's invariant: when a node is popped from the min-heap, its distance is finalized (optimal). This holds because all edge weights are non-negative — no future path through unprocessed nodes can be shorter than the direct path already found. The heap processes nodes in order of increasing distance. The "stale entry skip" (d > dist[u]) handles the fact that we may push multiple entries for the same node when finding progressively shorter paths.

⚠ Most Dangerous Mistake

Not skipping stale heap entries. When you find a shorter path and push a new (dist[v], v), the old (old_dist, v) still sits in the heap. If you process it later, you re-relax edges using a suboptimal distance, potentially corrupting results or causing unnecessary work.

  • Nodes are 1-indexed (1 to n), not 0-indexed — off-by-one in dist array
  • Forgetting to check if all nodes are reachable — return -1 if any dist is infinity
  • Using BFS instead of Dijkstra — BFS only works for unweighted graphs

Key Insight

The brute force scans all words for every search/startsWith call. The insight is a trie stores shared prefixes as tree paths — each node represents one character, and walking from root to a node spells out a prefix. This reduces all operations to O(L) regardless of how many words are stored, because you only follow the path for the target word.

How to Think About It

  1. Define a TrieNode with children = {} (char → child node) and is_end = False.
  2. Insert: starting from root, walk char by char. Create missing child nodes. Mark the final node's is_end = True.
  3. Search: walk from root following chars. If a char is missing → return False. At the end, return node.is_end (not just True).
  4. StartsWith: same walk as search. If all prefix chars are found, return True (no is_end check needed).
  5. Factor out a shared _traverse(word) helper that returns the node at the end of the path, or None if the path breaks.

Why This Works

The trie stores shared prefixes exactly once. "apple" and "app" share 3 nodes. Each node on a root-to-leaf path represents a prefix of one or more inserted words. The is_end flag distinguishes complete words from mere prefixes: without it, searching for "app" returns True even if only "apple" was inserted. The traversal for any operation is bounded by the word length, not the number of stored words.

⚠ Most Dangerous Mistake

Not checking is_end in search(), making it behave identically to startsWith(). If only "apple" was inserted, search("app") must return False, but without the is_end check it returns True because the path "a-p-p" exists in the trie.

  • Sharing a mutable default children = {} across all nodes (use __init__ to create a fresh dict per node)
  • Forgetting to create a root node in the Trie constructor — all operations start from root
  • Using a fixed-size array [26] when the problem may include non-lowercase characters

The Problem

DFS works well here and has the same asymptotic time. But Union-Find is the canonical technique for connected components: it handles dynamic edge additions (online connectivity), whereas DFS requires restarting from scratch. Learning Union-Find here pays off for harder problems like #130.

Key Insight

The brute force uses DFS, which processes each component as a traversal. The insight is Union-Find models connectivity declaratively: each city starts as its own component, and each connection merges two components. At the end, count the remaining distinct roots. With path compression and union by rank, each operation is nearly O(1) — and the structure naturally handles incremental edge additions.

How to Think About It

  1. Initialize parent[i] = i (each city is its own root) and rank[i] = 0.
  2. Implement find(x): follow parent pointers to the root, applying path compression (point every node on the path directly to root).
  3. Implement union(x, y): find both roots. If different, merge the smaller-rank tree under the larger.
  4. Iterate the upper triangle of the adjacency matrix (i < j). If isConnected[i][j] == 1, union(i, j).
  5. Count cities where find(i) == i — each such city is a root = one province.

Why This Works

Union-Find maintains the invariant that all nodes in the same connected component share a single root via parent pointers. Path compression flattens the tree on each find, keeping the amortized cost per operation to nearly O(1) (inverse Ackermann). After processing all edges in the adjacency matrix, the number of distinct roots exactly equals the number of connected components, because each union reduces the component count by one.

⚠ Most Dangerous Mistake

Counting roots as parent[i] == i without calling find(i) first. After multiple unions, some nodes still point to intermediate ancestors that are no longer roots. You must call find(i) to follow the chain to the actual root (and trigger path compression).

  • Processing the full matrix instead of just the upper triangle — wastes time on symmetric duplicates (and the diagonal)
  • Forgetting union by rank — without it, trees can degenerate to linked lists making find O(n)
  • Confusing this problem with #200 (Number of Islands) — that one uses a grid, this one an adjacency matrix
Alternative — DFS/BFS: Run DFS from each unvisited node, count the number of DFS launches. Same O(n²) time, equally valid for this static problem.

Brute Force

Key Insight

The brute force scans each 'O' region and checks if it touches the border — an awkward forward approach. The insight is to think in reverse: start from the borders and mark all 'O' cells reachable from the border as "safe." Then everything remaining that is 'O' must be surrounded. How to Think About It

  1. Iterate all border cells. For each border 'O', launch DFS/BFS to mark all connected 'O' cells as 'S' (safe).
  2. After marking all border-connected regions, scan the entire board.
  3. Any remaining 'O' → flip to 'X' (it was not reachable from a border, so it is surrounded).
  4. Any 'S' → restore to 'O' (it was safe).
  5. 'X' cells remain unchanged.

Why This Works

An 'O' region is NOT surrounded if and only if at least one cell in the region touches the board border. By flood-filling from every border 'O', we mark exactly the set of cells that have a path to the border. Any 'O' not reached by this process has no path to any border — it is enclosed by 'X' on all sides. The two-phase mark-then-flip ensures correctness: we never modify a cell that the DFS still needs to visit.

⚠ Most Dangerous Mistake

Trying to directly identify surrounded regions by scanning from interior cells. You end up needing to track whether a region "eventually" touches the border, which requires collecting the whole region before deciding. The reverse approach — start from borders — is dramatically simpler.

  • Flipping 'O' to 'X' during the border DFS scan — corrupts cells that other border DFS calls might need to reach
  • Not handling 1-row or 1-column boards — every cell is on the border, nothing should be flipped
  • Using recursion depth proportional to m*n on large boards — consider BFS with an explicit queue to avoid stack overflow
Alternative — Union-Find: Create a virtual "border" node. Union all border 'O' cells with it, and union all adjacent 'O' pairs. Any 'O' not in the virtual node's component is surrounded. This avoids the three-phase approach at the cost of Union-Find overhead.

#42 · Trapping Rain Water

HardTwo PointersStack

The Problem

Given n non-negative integers representing an elevation map where each bar has width 1, compute how much water it can trap after raining.

Brute Force

For each position i, scan left to find the tallest bar and scan right to find the tallest bar. Water at position i is min(max_left, max_right) - height[i]. This recalculates the maximums from scratch for every single position, resulting in O(n^2) total work.

Time: O(n²)
Space: O(1)

With n up to 20,000, the O(n²) approach of scanning left and right for max heights at every position is too slow. The max values are being redundantly recomputed.

Key Insight

The brute force re-scans left and right for max heights at every position. The insight is that two pointers from both ends, each maintaining a running maximum, can determine the water at each position in a single pass. When left_max ≤ right_max, the water at the left pointer is exactly left_max - height[left] regardless of the exact right maximum — because min(left_max, anything ≥ left_max) = left_max.

How to Think About It

  1. Initialize left = 0, right = n-1, left_max = 0, right_max = 0, result = 0.
  2. While left < right:
  3. If left_max ≤ right_max: update left_max = max(left_max, height[left]), add left_max - height[left] to result, advance left.
  4. Else: update right_max = max(right_max, height[right]), add right_max - height[right] to result, advance right.
  5. Critical: update the running max BEFORE computing water so the max includes the current bar, preventing negative results.

Why This Works

When left_max ≤ right_max, the water at the left pointer is exactly left_max - height[left]. We don't need the exact tallest bar on the right — only that it is at least right_max, which is at least left_max. Since min(left_max, anything ≥ left_max) = left_max, the left side is fully determined. Symmetrically, when right_max < left_max, the right side is fully determined. Each position is processed exactly once, giving O(n).

⚠ Most Dangerous Mistake

Updating left_max/right_max AFTER computing water instead of BEFORE. If you compute old_max - height[i] first, you get a negative value when height[i] is a new maximum. Updating first ensures the max always includes the current bar.

  • Using < vs ≤ in the comparison — either works correctly, but be consistent
  • Edge cases: empty array or single element — no water can be trapped
  • Confusing with #11 (Container With Most Water) — #11 finds the max rectangle between two bars, #42 sums individual water columns above each bar
Alternative — Prefix Arrays: Precompute left_max[i] (max of height[0..i]) and right_max[i] (max of height[i..n-1]). Then water[i] = min(left_max[i], right_max[i]) - height[i]. O(n) time, O(n) space — easier to understand but uses extra space.

Time: O(n² · |charset|) or O(n³)
Space: O(|charset|)

With |s| up to 10^5, checking all O(n^2) substrings is far too slow. We need to slide the window without rechecking character counts from scratch.

Key Insight

The brute force re-counts characters for overlapping substrings. The insight is a sliding window with a formed counter: track how many unique characters have met their required frequency. Only update formed when a character's count crosses the threshold (exactly equals requirement on the way up, drops below on the way down). This avoids re-scanning the entire window to check satisfaction.

How to Think About It

  1. Build t_count = Counter(t). Set required = len(t_count) (number of unique chars to satisfy).
  2. Expand right pointer: add s[right] to window_counts. If this char's count now exactly equals t_count, increment formed.
  3. While formed == required: the window is valid. Record it if smaller than the current best. Then shrink from left.
  4. When shrinking: if removing s[left] drops its count below t_count, decrement formed.
  5. Return the smallest recorded window, or "" if none found.

Why This Works

The formed counter tracks how many unique characters are fully satisfied. We only increment it when a character's window count exactly reaches its t_count (the threshold crossing), and decrement when it drops below. This means formed == required precisely when all character requirements are met — without scanning all characters each time. Each character enters the window at most once (right pointer) and leaves at most once (left pointer), giving O(|s|) total.

⚠ Most Dangerous Mistake

Mishandling formed with duplicate characters. If t = "AAB", incrementing formed every time 'A' enters the window (instead of only when count reaches 2) over-counts. You must increment formed only when window_counts[c] == t_count[c] (exact equality, not ≥).

  • Using ≥ instead of == to check threshold — increments formed multiple times for the same character
  • Not handling case sensitivity — the problem includes both uppercase and lowercase
  • Returning the window content instead of the substring — track start index and length, slice at the end
  • Forgetting to check c in t_count — characters not in t should never affect formed

#124 · Binary Tree Maximum Path Sum

HardTreeDFS

The Problem

Given a binary tree where node values can be negative, find the maximum path sum. A path is any sequence of connected nodes — it does not need to pass through the root, and it can start and end at any node.

Brute Force

For every node, treat it as the "peak" of a potential path: compute the max downward sum through its left subtree and right subtree separately, then combine. This requires a separate downward-traversal for each node, leading to O(n^2) work because max_down is called redundantly across overlapping subtrees.

Python — Brute Force 
def maxPathSum(root):
    result = float('-inf')
    def all_nodes(node):
        if not node: return []
        return [node] + all_nodes(node.left) + all_nodes(node.right)
    def max_down(node):  # max path sum starting from node going down
        if not node: return 0
        return node.val + max(0, max(max_down(node.left), max_down(node.right)))
    for node in all_nodes(root):  # treat each node as the "peak"
        left = max(0, max_down(node.left))
        right = max(0, max_down(node.right))
        result = max(result, node.val + left + right)
    return result

func maxPathSum(root *TreeNode) int {
    result := math.MinInt32
    var collect func(*TreeNode) []*TreeNode
    collect = func(n *TreeNode) []*TreeNode {
        if n == nil { return nil }
        return append(append([]*TreeNode{n}, collect(n.Left)...), collect(n.Right)...)
    }
    var maxDown func(*TreeNode) int
    maxDown = func(n *TreeNode) int {
        if n == nil { return 0 }
        return n.Val + max(0, max(maxDown(n.Left), maxDown(n.Right)))
    }
    for _, node := range collect(root) {
        l := max(0, maxDown(node.Left))
        r := max(0, maxDown(node.Right))
        if node.Val+l+r > result { result = node.Val + l + r }
    }
    return result
}
Time: O(n²)
Space: O(n)

For each of the n nodes treated as a "peak," max_down traverses its subtree — O(n) in the worst case. With n up to 30,000, O(n^2) may TLE. The optimal solution computes both quantities in a single DFS pass.

Key Insight

The brute force calls max_down separately for each node, recomputing subtree sums redundantly. The insight is that a single post-order DFS can compute two things simultaneously: (1) the best single-branch gain going downward (returned to parent), and (2) the best "through-path" using both subtrees (updates a global max). These are different values — a node can use both branches for a through-path, but only one branch to extend upward to its parent.

How to Think About It

  1. Initialize max_sum = -infinity (must handle all-negative trees).
  2. Define DFS(node) that returns the max downward gain from this node.
  3. Base case: None returns 0 (no gain from empty subtree).
  4. Compute left_gain = max(DFS(left), 0) and right_gain = max(DFS(right), 0) — clamping to 0 means we never follow a negative branch.
  5. Update global: max_sum = max(max_sum, node.val + left_gain + right_gain). This is the best through-path peaking at this node.
  6. Return to parent: node.val + max(left_gain, right_gain) — only the better single branch, because a path cannot fork.

Why This Works

Every maximum path has a "peak" node — the highest ancestor on that path. At that peak, the path goes down into the left subtree and down into the right subtree (either or both can be empty). By computing the through-path at every node and comparing to the global max, we consider every possible peak. The return value only includes one branch because a valid path is a sequence of nodes — it cannot fork three ways (left child, right child, AND parent). The post-order traversal ensures subtree results are available when we process each node.

⚠ Most Dangerous Mistake

Returning left_gain + node.val + right_gain to the parent. This is a V-shaped path that uses both branches — the parent cannot extend it further (it would need to go down one branch, up through this node, and down the other, creating a fork). Only return node.val + max(left_gain, right_gain).

  • Initializing max_sum = 0 instead of -inf — fails when all node values are negative (e.g., tree is [-3])
  • Not clamping subtree gains to 0 with max(gain, 0) — a negative subtree should contribute nothing, not drag down the sum
  • Confusing "downward gain" (what we return) with "through-path" (what we compare to global max)
