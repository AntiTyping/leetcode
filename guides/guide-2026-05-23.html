The Problem

You are given two non-empty linked lists representing two non-negative integers. The digits are stored in reverse order, and each node contains a single digit. Add the two numbers and return the sum as a linked list (also in reverse order).

Quick Reminder

Walk both lists in lockstep, summing digit-by-digit while threading a carry through. Use a dummy head node to simplify result construction.

#206 · Reverse Linked List

EasyLinked ListPointer Rewiring

The Problem

Given the head of a singly linked list, reverse the list and return the new head. For example, 1 -> 2 -> 3 -> 4 -> 5 becomes 5 -> 4 -> 3 -> 2 -> 1.

Brute Force

Walk the list once and copy every value into an array. Then walk the array in reverse, building a brand-new linked list. This is correct but uses O(n) extra memory and creates new nodes instead of rewiring the existing ones.

Python — Brute Force 
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reverseList(head):
    vals = []
    cur = head
    while cur:
        vals.append(cur.val)
        cur = cur.next
    dummy = ListNode()
    tail = dummy
    for v in reversed(vals):
        tail.next = ListNode(v)
        tail = tail.next
    return dummy.next
Go — Brute Force 
type ListNode struct {
    Val  int
    Next *ListNode
}

func reverseList(head *ListNode) *ListNode {
    vals := []int{}
    for cur := head; cur != nil; cur = cur.Next {
        vals = append(vals, cur.Val)
    }
    dummy := &ListNode{}
    tail := dummy
    for i := len(vals) - 1; i >= 0; i-- {
        tail.Next = &ListNode{Val: vals[i]}
        tail = tail.Next
    }
    return dummy.Next
}

Why optimize? Allocating a parallel array and a fresh list doubles memory and ignores the structural opportunity — the existing nodes can just have their next pointers flipped in place.

Key Insight

Reversal is purely a pointer-rewiring problem. Each node needs to point at its previous neighbor instead of its next one. Keep three pointers — prev, cur, next — and flip them one step at a time.

How to Think About It

  1. Initialize prev = None and cur = head.
  2. Before overwriting cur.next, save it into nxt so you don't lose the rest of the list.
  3. Flip the pointer: cur.next = prev.
  4. Advance both windows: prev = cur, then cur = nxt.
  5. When cur is null, prev is the new head.

Why This Works

At each step you maintain an invariant: the segment from head up to (but not including) cur has already been reversed, with prev as its new front. Advancing one node at a time preserves the invariant, and termination gives you a fully reversed list whose head is prev.

⚠ Most Dangerous Mistake

Forgetting to save cur.next before reassigning it. The moment you write cur.next = prev, the rest of the list becomes unreachable unless you cached it.

  • Returning head at the end instead of prev — the old head is now the tail.
  • Forgetting to handle an empty list (head is None); the loop body never runs and you correctly return None, but only if you initialized prev first.
  • Writing a recursive version and stack-overflowing on long lists. Recursion is elegant but iterative is safer for interviews.

#21 · Merge Two Sorted Lists

EasyLinked ListMerging

The Problem

Given the heads of two sorted linked lists, splice them together into a single sorted list by rewiring the existing nodes (not by copying values). Return the head of the merged list.

Brute Force

Collect all values from both lists into an array, sort it, and build a fresh linked list from the sorted values. This is O((m+n) log (m+n)) and ignores that both inputs are already sorted — the structure of the problem is wasted.

Python — Brute Force 
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def mergeTwoLists(l1, l2):
    vals = []
    for head in (l1, l2):
        cur = head
        while cur:
            vals.append(cur.val)
            cur = cur.next
    vals.sort()
    dummy = ListNode()
    tail = dummy
    for v in vals:
        tail.next = ListNode(v)
        tail = tail.next
    return dummy.next
Go — Brute Force 
type ListNode struct {
    Val  int
    Next *ListNode
}

func mergeTwoLists(l1, l2 *ListNode) *ListNode {
    vals := []int{}
    for _, h := range []*ListNode{l1, l2} {
        for cur := h; cur != nil; cur = cur.Next {
            vals = append(vals, cur.Val)
        }
    }
    sort.Ints(vals)
    dummy := &ListNode{}
    tail := dummy
    for _, v := range vals {
        tail.Next = &ListNode{Val: v}
        tail = tail.Next
    }
    return dummy.Next
}

Why optimize? Both inputs are already sorted, so any sort is wasted work. You can produce the answer in a single linear pass.

Key Insight

This is the merge half of merge-sort. Walk two cursors in parallel and always append the smaller head to the result, advancing only that cursor. A dummy head node removes the special case for the first append.

How to Think About It

  1. Create a dummy node and a tail pointer that starts at the dummy.
  2. While both l1 and l2 are non-null, compare values.
  3. Attach the smaller-valued node to tail.next, then advance that list's cursor and tail.
  4. When one list is exhausted, link the remainder of the other directly — it's already sorted.
  5. Return dummy.next.

Why This Works

The invariant: at every step, the merged list ending at tail contains all values less than or equal to both current heads, in sorted order. Because we always attach the smaller head next, no later append can violate the ordering. Linking the tail of the surviving list is safe because all of its remaining values are larger than the last appended one.

⚠ Most Dangerous Mistake

Forgetting to attach the leftover tail of the non-empty list at the end. This silently truncates the result and is hard to spot in a debugger because the early part of the output looks correct.

  • Using < when the problem allows duplicates — either is fine, but be deliberate; switching mid-loop creates bugs.
  • Allocating new nodes instead of rewiring — correct but wastes memory and misses the point of the exercise.
  • Not initializing the dummy node, leading to a null-pointer dereference on the first tail.next assignment.

#19 · Remove Nth Node From End of List

MediumLinked ListTwo Pointers

The Problem

Given the head of a linked list, remove the nth node counting from the end (1-indexed) and return the head of the resulting list. The list has at least one node and n is always valid.

Brute Force

Pass once over the list to compute its length L. Then walk again to position L - n - 1 (the node just before the target) and splice past it. Two passes, O(n) total, but feels clunky and is easy to off-by-one on the index.

Python — Brute Force 
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def removeNthFromEnd(head, n):
    length = 0
    cur = head
    while cur:
        length += 1
        cur = cur.next
    dummy = ListNode(0, head)
    prev = dummy
    for _ in range(length - n):
        prev = prev.next
    prev.next = prev.next.next
    return dummy.next
Go — Brute Force 
type ListNode struct {
    Val  int
    Next *ListNode
}

func removeNthFromEnd(head *ListNode, n int) *ListNode {
    length := 0
    for cur := head; cur != nil; cur = cur.Next {
        length++
    }
    dummy := &ListNode{Next: head}
    prev := dummy
    for i := 0; i < length-n; i++ {
        prev = prev.Next
    }
    prev.Next = prev.Next.Next
    return dummy.Next
}

Why optimize? Two passes is fine asymptotically, but interviewers often want one. There is a clean single-pass two-pointer trick.

Key Insight

Use two pointers separated by exactly n+1 nodes. When the leading pointer falls off the end, the trailing pointer is sitting on the node just before the one to remove. The gap is the trick — it encodes "n from the end" directly.

How to Think About It

  1. Create a dummy node pointing to head so removing the actual head is uniform.
  2. Initialize fast and slow both at the dummy.
  3. Advance fast by n+1 steps so the gap between fast and slow is n+1.
  4. Advance both pointers together until fast is null.
  5. Splice: slow.next = slow.next.next, then return dummy.next.

Why This Works

When fast falls off the end, it has traveled L+1 steps from the dummy. Since the gap is n+1, slow has traveled L-n steps — placing it exactly one node before the n-th-from-end. Splicing past slow.next deletes the correct node, including when it's the original head.

⚠ Most Dangerous Mistake

Off-by-one in the gap. If you make the gap n instead of n+1, slow lands on the node to delete itself, not on the predecessor — and you cannot splice it out without a back-pointer.

  • Forgetting the dummy node and crashing when n equals the list length (removing the head).
  • Not advancing fast first before the joint loop — if you advance both together, the gap collapses.
  • Returning head directly instead of dummy.next; the original head may have been removed.

#234 · Palindrome Linked List

EasyLinked ListTwo PointersReversal

The Problem

Given the head of a singly linked list, return true if the list reads the same forward and backward, false otherwise.

Brute Force

Copy all values into an array and check whether the array equals its reverse. Simple, correct, but uses O(n) extra space and ignores that linked lists can be manipulated in place.

Python — Brute Force 
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def isPalindrome(head):
    vals = []
    cur = head
    while cur:
        vals.append(cur.val)
        cur = cur.next
    return vals == vals[::-1]
Go — Brute Force 
type ListNode struct {
    Val  int
    Next *ListNode
}

func isPalindrome(head *ListNode) bool {
    vals := []int{}
    for cur := head; cur != nil; cur = cur.Next {
        vals = append(vals, cur.Val)
    }
    n := len(vals)
    for i := 0; i < n/2; i++ {
        if vals[i] != vals[n-1-i] {
            return false
        }
    }
    return true
}

Why optimize? Interview follow-ups often demand O(1) space. The array copy works but skips the elegant in-place algorithm.

Key Insight

Find the middle with slow/fast pointers, reverse the second half in place, then walk the first half and reversed second half in parallel comparing values. Two well-known patterns — midpoint finding and reversal — compose into a clean O(1) space algorithm.

How to Think About It

  1. Use slow/fast pointers: when fast reaches the end, slow is at the middle.
  2. Reverse the list starting at slow.next (or slow for odd lengths — pick a convention).
  3. Walk head and the reversed-second-half head in parallel, comparing values at each step.
  4. If any mismatch is found, return false.
  5. If the reversed-half walker hits null without mismatches, return true.

Why This Works

A palindrome reads the same forward and backward, which means the first half equals the reverse of the second half. By reversing the second half in place, comparison becomes a straight parallel walk. The middle node (if length is odd) doesn't need to be checked — it equals itself.

⚠ Most Dangerous Mistake

Mishandling odd vs even length when locating the middle. For length 5 with fast taking two steps at a time, slow ends on the middle node; for length 4 it ends on the second of the two middles. Pick one rule and stick to it — usually "reverse from slow.next" works for both.

  • Forgetting to restore the list after the comparison (if the problem says "do not modify"). Reverse the second half back before returning.
  • Comparing pointers instead of values — a == b compares node identity, not contents, in Python/Go.
  • Stopping the parallel walk when the first-half pointer hits null instead of the reversed second half; the first half can be one node longer.

#143 · Reorder List

MediumLinked ListTwo PointersReversalMerging

The Problem

Given a singly linked list L0 -> L1 -> ... -> Ln-1 -> Ln, reorder it to L0 -> Ln -> L1 -> Ln-1 -> L2 -> Ln-2 -> .... Modify the list in place; do not return anything.

Brute Force

Copy node references into an array so you have random access. Then build the new ordering by alternately popping from the front and the back of the array, rewiring next pointers as you go. Correct but uses O(n) extra space.

Python — Brute Force 
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next

def reorderList(head):
    if not head:
        return
    nodes = []
    cur = head
    while cur:
        nodes.append(cur)
        cur = cur.next
    i, j = 0, len(nodes) - 1
    while i < j:
        nodes[i].next = nodes[j]
        i += 1
        if i == j:
            break
        nodes[j].next = nodes[i]
        j -= 1
    nodes[i].next = None
Go — Brute Force 
type ListNode struct {
    Val  int
    Next *ListNode
}

func reorderList(head *ListNode) {
    if head == nil {
        return
    }
    nodes := []*ListNode{}
    for cur := head; cur != nil; cur = cur.Next {
        nodes = append(nodes, cur)
    }
    i, j := 0, len(nodes)-1
    for i < j {
        nodes[i].Next = nodes[j]
        i++
        if i == j {
            break
        }
        nodes[j].Next = nodes[i]
        j--
    }
    nodes[i].Next = nil
}

Why optimize? The array gives you random access but wastes memory. The same effect comes from reversing the second half in place — no extra storage required.

Key Insight

The target ordering is "interleave the first half with the reversed second half". So: (1) find the middle, (2) reverse the second half, (3) merge the two halves by alternating nodes. Three sub-routines you already know — composed.

How to Think About It

  1. Find the middle using slow/fast pointers; slow ends at the middle (or just past for even lengths).
  2. Split the list: set slow.next = None to terminate the first half cleanly.
  3. Reverse the second half using the prev/cur/next pattern from #206.
  4. Walk both halves with two cursors, splicing them together: first-half-node -> second-half-node -> next-first-half-node, and so on.
  5. Stop when the second-half cursor reaches null. The first half can be one node longer; that's fine — its terminal .next stays null.

Why This Works

If you label the first half A0, A1, ..., Ak and the reversed second half B0, B1, ..., Bm (where B0 is the original last node), the merge produces A0, B0, A1, B1, ... — exactly the required ordering. Splitting at the middle guarantees the second half is no longer than the first, so the interleave terminates cleanly.

⚠ Most Dangerous Mistake

Not terminating the first half before reversing. If slow.next still points into the second half when you start reversing, you'll create a cycle and the reverse loop will spin forever.

  • Saving the wrong "next" pointer during the merge step — you need both a.next and b.next before rewiring, or you'll lose a chain.
  • Off-by-one when the list has even length: the second half has equal size to the first, and the merge must stop the moment b is null.
  • Forgetting that the problem mutates in place; returning the head is a small mistake but cleaning up the original tail's .next is critical to avoid loops.

#703 · Kth Largest Element in a Stream

EasyHeapDesign

The Problem

Design a class that finds the k-th largest element in a stream. The constructor takes an integer k and an initial array of numbers. The add(val) method appends a value to the stream and returns the k-th largest element among all values seen so far. Note that this is the k-th largest in sorted order, not the k-th distinct largest.

Brute Force

Store all numbers in a list. On every add, append the new value, sort the list in descending order, and return the element at index k-1. This works but each call is O(n log n).

Python — Brute Force 
class KthLargest:
    def __init__(self, k: int, nums: list[int]):
        self.k = k
        self.nums = nums

    def add(self, val: int) -> int:
        self.nums.append(val)
        self.nums.sort(reverse=True)
        return self.nums[self.k - 1]
Go — Brute Force 
type KthLargest struct {
    k    int
    nums []int
}

func Constructor(k int, nums []int) KthLargest {
    return KthLargest{k: k, nums: nums}
}

func (kl *KthLargest) Add(val int) int {
    kl.nums = append(kl.nums, val)
    sort.Sort(sort.Reverse(sort.IntSlice(kl.nums)))
    return kl.nums[kl.k-1]
}

Why optimize? Sorting on each add is O(n log n) per call; with m adds it becomes O(m n log n), far too slow for a streaming API.

Key Insight

We never need the full sorted order — we only need the k-th largest. If we maintain a min-heap of size exactly k, then the smallest element in that heap IS the k-th largest seen so far. When a new value arrives, push it and pop if the heap exceeds size k. The root is always the answer.

How to Think About It

  1. Recognize "k-th largest" maps to maintaining the top k values.
  2. A min-heap of size k holds the k largest items; its root is the smallest of those, which is the k-th largest overall.
  3. In the constructor, push all initial numbers; after each push, if size > k, pop the root.
  4. In add, push the value, then pop if size > k.
  5. Return the root (peek) — this is the k-th largest. Each add is O(log k).

Why This Works

A min-heap of size k always contains exactly the k largest elements seen. By definition, the minimum of those k is the k-th largest in the entire stream. Pushing and popping when size exceeds k preserves this invariant.

⚠ Most Dangerous Mistake

Using a max-heap of all elements and popping k-1 times on every query — this destroys the heap and is O(n log n) per query. The whole point is a fixed-size min-heap.

  • Confusing min-heap vs max-heap: for "k-th largest" you want a min-heap of size k.
  • Forgetting that Python's heapq is min-heap only; for a max-heap you must negate values.
  • Not handling the initial array case where len(nums) < k — the heap may be smaller than k after the constructor; the first few adds simply grow it.

#1046 · Last Stone Weight

EasyHeapSimulation

The Problem

You are given an array of positive integers stones, each representing a stone's weight. Repeatedly pick the two heaviest stones and smash them together: if equal, both are destroyed; if unequal, the lighter is destroyed and the heavier becomes |x - y|. Return the weight of the last remaining stone, or 0 if none remain.

Brute Force

Sort the array, take the last two, smash them, insert the difference back, and resort. Repeat until at most one stone remains. Each sort is O(n log n) and we do up to n iterations.

Python — Brute Force 
def lastStoneWeight(stones: list[int]) -> int:
    stones = sorted(stones)
    while len(stones) > 1:
        y = stones.pop()  # heaviest
        x = stones.pop()  # second heaviest
        if y > x:
            # insert difference and resort
            stones.append(y - x)
            stones.sort()
    return stones[0] if stones else 0
Go — Brute Force 
func lastStoneWeight(stones []int) int {
    sort.Ints(stones)
    for len(stones) > 1 {
        n := len(stones)
        y, x := stones[n-1], stones[n-2]
        stones = stones[:n-2]
        if y > x {
            stones = append(stones, y-x)
            sort.Ints(stones)
        }
    }
    if len(stones) == 0 {
        return 0
    }
    return stones[0]
}

Why optimize? Resorting on every iteration is O(n^2 log n). We only need the top 2 each step — a max-heap gives this in O(log n) per operation.

Key Insight

We only ever need the two largest elements at a time. A max-heap supports "extract max" in O(log n) and "insert" in O(log n). The total work is O(n log n) instead of O(n^2 log n).

How to Think About It

  1. Recognize the access pattern: repeatedly extract the two largest elements.
  2. Build a max-heap from the stones (in Python, negate values since heapq is min-only).
  3. While heap has >= 2 stones, pop y then pop x (y >= x).
  4. If y > x, push (y - x) back onto the heap.
  5. When the loop ends, return the heap's top if non-empty, else 0.

Why This Works

Each smash operation either removes one or two stones from the heap. The heap invariant ensures we always access the current two heaviest in O(log n). The simulation directly mirrors the problem definition.

⚠ Most Dangerous Mistake

In Python, forgetting to negate when pushing/popping the max-heap. Pushing positive values into a min-heap and treating its top as max gives completely wrong answers.

  • Off-by-one when checking heap size: loop while len(h) >= 2, not > 2.
  • Pushing 0 back when y == x — harmless but wasteful; just skip the push.
  • Returning h[0] on an empty heap throws IndexError; guard with a length check.

#973 · K Closest Points to Origin

MediumHeapTop-K

The Problem

Given an array of points points[i] = [x, y] on the 2D plane and an integer k, return the k closest points to the origin (0, 0). Distance is Euclidean, but since we only compare distances, we can use squared distance x*x + y*y to avoid floating point. The answer may be returned in any order.

Brute Force

Compute the squared distance for every point, sort all points by distance ascending, and return the first k. This is O(n log n) — not bad, but more than necessary.

Python — Brute Force 
def kClosest(points: list[list[int]], k: int) -> list[list[int]]:
    # Sort by squared distance ascending; slicing returns first k
    points.sort(key=lambda p: p[0] * p[0] + p[1] * p[1])
    return points[:k]
Go — Brute Force 
func kClosest(points [][]int, k int) [][]int {
    sort.Slice(points, func(i, j int) bool {
        di := points[i][0]*points[i][0] + points[i][1]*points[i][1]
        dj := points[j][0]*points[j][0] + points[j][1]*points[j][1]
        return di < dj
    })
    return points[:k]
}

Why optimize? Sorting is O(n log n) but we only need the k smallest. A heap of size k gives O(n log k), which is better when k << n.

Key Insight

To keep the k smallest distances, use a max-heap of size k. For each new point, if the heap has fewer than k elements, push; otherwise compare against the heap's root (the largest of the current k closest). If the new point is closer, replace the root. The heap always holds the current best k.

How to Think About It

  1. Compute squared distance for each point — no need for sqrt since we're only comparing.
  2. Maintain a max-heap keyed by distance, with size capped at k.
  3. For each point: push (dist, point); if size > k, pop the largest distance.
  4. After processing all points, the heap contains the k closest.
  5. Extract them into a result list in any order.

Why This Works

At any moment, the max-heap of size k contains the k smallest distances seen so far. When a new distance is smaller than the heap's max, it belongs in the top-k and the current max is evicted. This maintains the invariant that the heap holds the global k-smallest at the end.

⚠ Most Dangerous Mistake

Using a MIN-heap and popping k times. That works (it's O(n + k log n)), but if you write it as "min-heap of size k" it's incorrect — you'd be evicting the smallest, not the largest. Match heap type to operation.

  • Computing actual Euclidean distance with math.sqrt — unnecessary and introduces float error.
  • In Python, tuples in heap compare element-wise; if distances tie, Python tries to compare lists (points) and errors. Wrap point as tuple or include an index tiebreaker.
  • Forgetting that heap order is not sorted — the returned k points are unsorted, which is fine per the problem.

#136 · Single Number

EasyBit ManipulationXOR

The Problem

Given a non-empty array of integers where every element appears exactly twice except for one element which appears once, find that single element. The solution must run in O(n) time and O(1) extra space.

Brute Force

Use a hash map (or set) to count occurrences, then iterate to find the element with count 1. This is O(n) time but O(n) extra space — violates the space constraint, but it's the most natural first approach.

Python — Brute Force 
def singleNumber(nums: list[int]) -> int:
    counts = {}
    for x in nums:
        counts[x] = counts.get(x, 0) + 1
    for k, v in counts.items():
        if v == 1:
            return k
    return -1  # unreachable per problem constraints
Go — Brute Force 
func singleNumber(nums []int) int {
    counts := make(map[int]int)
    for _, x := range nums {
        counts[x]++
    }
    for k, v := range counts {
        if v == 1 {
            return k
        }
    }
    return -1
}

Why optimize? The hash map uses O(n) extra space, violating the problem's O(1) constraint. XOR gives the same result in O(1) space.

Key Insight

XOR has two magical properties: x ^ x = 0 (a number XOR itself is zero) and x ^ 0 = x (zero is the identity). XOR is commutative and associative, so XORing all numbers together cancels every pair, leaving only the singleton.

How to Think About It

  1. Note the constraint: pairs everywhere except one element — this screams "cancel pairs."
  2. Recall XOR properties: self-cancels, identity is 0, order-independent.
  3. Initialize result = 0.
  4. XOR every element into result. All paired elements cancel.
  5. Return result — it's the single element.

Why This Works

Mathematically, for any permutation of the array, XOR cumulates to a ^ a ^ b ^ b ^ ... ^ s. Since XOR is associative/commutative, we can rearrange to (a^a) ^ (b^b) ^ ... ^ s = 0 ^ 0 ^ ... ^ s = s.

⚠ Most Dangerous Mistake

Using addition or subtraction instead of XOR. Sum-of-set-minus-sum-of-array works only if the problem guarantees distinct pair values. XOR is the bit-pattern-cancel approach that always works.

  • Initializing result to the first element vs. 0 — both work, but 0 is cleaner and matches XOR identity.
  • Confusing XOR (^) with exponentiation in Python (**) or bitwise OR (|).
  • Generalizing to "every element appears three times except one" — that needs a different bitwise trick (count bits mod 3).

#191 · Number of 1 Bits

EasyBit ManipulationPopcount

The Problem

Given an unsigned 32-bit integer n, return the number of '1' bits it has (also known as the Hamming weight or popcount). For example, the input 11 (binary 1011) has three 1-bits.

Brute Force

Convert the integer to its binary string and count the '1' characters. This is correct and O(log n), but it does string allocation.

Python — Brute Force 
def hammingWeight(n: int) -> int:
    # bin(n) returns e.g. '0b1011' — count the '1' chars
    return bin(n).count('1')
Go — Brute Force 
import "strconv"

func hammingWeight(n int) int {
    s := strconv.FormatInt(int64(n), 2) // binary string
    count := 0
    for _, c := range s {
        if c == '1' {
            count++
        }
    }
    return count
}

Why optimize? String conversion adds allocation overhead. Pure bit ops avoid allocation and clarify the underlying operation.

Key Insight

The trick n & (n - 1) clears the lowest set bit of n. Each iteration removes exactly one 1-bit, so the loop runs exactly (number of set bits) times — faster than scanning all 32 bit positions when the number has few set bits.

How to Think About It

  1. Approach 1: shift right repeatedly, mask with 1 to test the LSB, count. Loops 32 times.
  2. Approach 2 (Brian Kernighan's): repeatedly apply n = n & (n - 1), incrementing a counter. Loops only as many times as there are set bits.
  3. Initialize count = 0.
  4. While n != 0: do n &= n - 1 and count++.
  5. Return count.

Why This Works

Subtracting 1 from n flips the lowest set bit to 0 and all lower 0-bits to 1. ANDing with the original n preserves the higher bits and zeros out the lowest set bit and all bits below it. Net effect: one set bit removed per iteration.

⚠ Most Dangerous Mistake

In languages where int is signed and arithmetic right-shift is used, shifting a negative number can produce an infinite loop. Use unsigned shift (>>> in Java) or explicitly mask. Python ints are arbitrary precision so this is less of a concern, but Go's int can vary by platform.

  • Looping a fixed 32 times when you could short-circuit on n == 0.
  • Confusing n & (n - 1) with n & -n — the latter isolates the lowest set bit, doesn't clear it.
  • Reaching for the built-in (int.bit_count() in Python 3.10+ or bits.OnesCount in Go) without understanding the underlying mechanic — interviewers may ask for the manual implementation.

#338 · Counting Bits

EasyBit ManipulationDP

The Problem

Given an integer n, return an array ans of length n + 1 such that for each i (0 <= i <= n), ans[i] is the number of 1-bits in the binary representation of i. The goal is to do this in O(n) time without using built-in popcount.

Brute Force

For each i from 0 to n, count the bits using the Brian Kernighan trick (or any other per-number popcount). Total time is O(n log n) since each popcount is O(log i).

Python — Brute Force 
def countBits(n: int) -> list[int]:
    def popcount(x: int) -> int:
        c = 0
        while x:
            x &= x - 1
            c += 1
        return c
    return [popcount(i) for i in range(n + 1)]
Go — Brute Force 
func countBits(n int) []int {
    ans := make([]int, n+1)
    for i := 0; i <= n; i++ {
        c, x := 0, i
        for x != 0 {
            x &= x - 1
            c++
        }
        ans[i] = c
    }
    return ans
}

Why optimize? The brute force is O(n log n). The problem hints we can do O(n) — this requires a DP recurrence on previously computed values.

Key Insight

There's a beautiful recurrence: ans[i] = ans[i >> 1] + (i & 1). The number of 1-bits in i equals the number of 1-bits in i/2 plus 1 if i is odd. Alternative: ans[i] = ans[i & (i-1)] + 1, since clearing the lowest set bit gives a smaller number whose popcount we already know.

How to Think About It

  1. Recognize this is a DP: each ans[i] depends on a previous ans[j] for some j < i.
  2. Observation: i >> 1 drops the lowest bit, so its popcount differs from i's by exactly the LSB.
  3. Base case: ans[0] = 0.
  4. Recurrence: ans[i] = ans[i >> 1] + (i & 1).
  5. Fill the array from i = 1 to n in order — each entry is O(1) lookup. Total O(n).

Why This Works

Every integer i's binary form is (bits of i>>1) followed by the bit (i&1). The total number of 1s is simply the sum of those two parts. Because i>>1 < i, the DP fills bottom-up correctly.

⚠ Most Dangerous Mistake

Using int.bit_count() or bits.OnesCount in the solution and calling it O(1). On real CPUs popcount is a single instruction, but in the algorithmic O() it's O(log i). The intended educational answer is the DP.

  • Off-by-one on array length: must be n + 1 entries (0 through n inclusive).
  • Confusing the two recurrences: ans[i >> 1] + (i & 1) vs. ans[i & (i-1)] + 1. Both work; mixing them gives garbage.
  • Forgetting that ans[0] must be 0 — if you initialize with a different value the whole DP is wrong.

EasyIntervalsSorting

The Problem

Given an array of meeting time intervals intervals[i] = [start_i, end_i], determine if a person could attend all meetings. Equivalently, return true if no two intervals overlap. Endpoints touching (e.g., [1,5] and [5,8]) do NOT count as overlap.

Brute Force

For every pair (i, j), check whether they overlap. Two intervals overlap iff a.start < b.end and b.start < a.end. This is O(n^2).

Python — Brute Force 
def canAttendMeetings(intervals: list[list[int]]) -> bool:
    n = len(intervals)
    for i in range(n):
        for j in range(i + 1, n):
            a, b = intervals[i], intervals[j]
            if a[0] < b[1] and b[0] < a[1]:
                return False
    return True
Go — Brute Force 
func canAttendMeetings(intervals [][]int) bool {

    for i := 0; i < n; i++ {
        for j := i + 1; j < n; j++ {
            a, b := intervals[i], intervals[j]
            if a[0] < b[1] && b[0] < a[1] {
                return false
            }
        }
    }
    return true
}

Why optimize? O(n^2) is wasteful: after sorting by start time we only need to compare each interval with its neighbor.

Key Insight

Sort intervals by start time. Then any overlap must be between adjacent intervals: if interval i ends after interval i+1 starts, they conflict. This reduces the check from all pairs to consecutive pairs.

How to Think About It

  1. Sort intervals ascending by start time.
  2. Walk the sorted list from i = 1 to n-1.
  3. If intervals[i].start < intervals[i-1].end, return false.
  4. Use strict < (not <=) since touching endpoints are allowed.
  5. If the loop completes without conflict, return true.

Why This Works

After sorting by start, if any pair (i, j) with i < j overlaps, then in particular the interval immediately following i in sort order must start before i ends — otherwise a sorting argument shows no overlap is possible.

⚠ Most Dangerous Mistake

Using <= instead of < when comparing end-to-next-start. Touching endpoints (end == next start) are NOT overlaps for this problem. Read the problem definition carefully.

  • Forgetting to sort — the linear scan only works on sorted input.
  • Sorting by end time instead of start — works for some interval problems but not this comparison pattern.
  • Mutating the input array when the caller expects it unchanged — copy first if needed.

#253 · Meeting Rooms II

MediumIntervalsHeap / Sweep Line

The Problem

Given an array of meeting time intervals, return the minimum number of conference rooms required to host all meetings without conflicts. Equivalently, find the maximum number of intervals that overlap at any single point in time.

Brute Force

For every "moment in time" (sample at every integer or every endpoint), count how many intervals are active. Return the maximum count. With endpoint sampling, this is O(n^2).

Python — Brute Force 
def minMeetingRooms(intervals: list[list[int]]) -> int:
    if not intervals:
        return 0
    # Sample at each interval's start; count active intervals.
    best = 0
    for s, _ in intervals:
        active = sum(1 for (a, b) in intervals if a <= s < b)
        best = max(best, active)
    return best
Go — Brute Force 
func minMeetingRooms(intervals [][]int) int {
    if len(intervals) == 0 {
        return 0
    }
    best := 0
    for _, x := range intervals {
        s := x[0]
        active := 0
        for _, y := range intervals {
            if y[0] <= s && s < y[1] {
                active++
            }
        }
        if active > best {
            best = active
        }
    }
    return best
}

Why optimize? O(n^2) is too slow for large inputs. Two superior O(n log n) approaches exist: min-heap of end times, and sweep line on sorted starts/ends.

Key Insight

Two equivalent O(n log n) views: (1) Min-heap of room-end-times — sort by start; for each meeting, if the earliest-ending room is free, reuse it; else allocate a new room. Heap size = rooms needed. (2) Sweep line — sort all start times and all end times separately; advance two pointers and track current concurrent count.

How to Think About It

  1. Reframe: count the maximum simultaneous overlap. That's the room count.
  2. Sort intervals by start time.
  3. Maintain a min-heap of end times (one per active room).
  4. For each interval (s, e): if heap is non-empty and heap[0] <= s, pop (room frees up). Always push e.
  5. Answer = final heap size (or the running max during the loop).

Why This Works

Processing meetings in start-time order means whenever a meeting begins, all earlier meetings have been considered. The earliest-ending active meeting (heap top) is the best candidate to free a room since it ends soonest. If it has already ended by time s, we reuse that room; otherwise we open a new one. The heap size always equals current concurrent meetings.

⚠ Most Dangerous Mistake

Using < instead of <= when checking heap top against new start. If a meeting ends at time 5 and the next starts at 5, that room IS free (endpoints touching are allowed). Use heap[0] <= s.

  • Forgetting to push the current meeting's end time after potentially popping — the new meeting still needs a room.
  • Sweep-line variant: when sorting events, ties between an end-event and a start-event at the same time should put the end first (room frees before next meeting starts).
  • Treating an empty input as 1 instead of 0 — guard against empty arrays.

#435 · Non-overlapping Intervals

MediumIntervalsGreedy

The Problem

Given an array of intervals, return the minimum number of intervals you must remove to make the rest non-overlapping. Equivalently, find the maximum number of mutually non-overlapping intervals; the answer is n - that max. Touching endpoints do NOT count as overlap.

Brute Force

Try every subset of intervals; for each subset that is mutually non-overlapping, track its size; return n minus the max valid subset size. This is O(2^n) and intractable.

Python — Brute Force 
from itertools import combinations

def eraseOverlapIntervals(intervals: list[list[int]]) -> int:
    n = len(intervals)
    best = 0
    for k in range(n + 1):
        for combo in combinations(intervals, k):
            sorted_combo = sorted(combo, key=lambda x: x[0])
            ok = all(sorted_combo[i][0] >= sorted_combo[i - 1][1]
                     for i in range(1, len(sorted_combo)))
            if ok:
                best = max(best, k)
    return n - best

Why optimize? O(2^n) explodes for n > 20. There's a beautiful greedy that runs in O(n log n).

Key Insight

This is the classic "activity selection" problem in disguise. Sort intervals by END time, then greedily keep an interval if it starts at or after the previous kept interval's end. Each kept interval leaves maximum room for future picks. The answer is n - kept.

How to Think About It

  1. Reframe: find the maximum number of compatible (non-overlapping) intervals; remove the rest.
  2. Sort by END time ascending (NOT start time — this is the key difference from #252).
  3. Iterate; track prev_end initialized to negative infinity.
  4. For each interval (s, e): if s >= prev_end, keep it and set prev_end = e; else count as removed.
  5. Return the removed count (or n minus the kept count).

Why This Works

Exchange argument: any optimal solution can be transformed into the greedy one by swapping its first kept interval with the one having the smallest end time without loss. Choosing the earliest-finishing compatible interval at each step maximizes remaining options.

⚠ Most Dangerous Mistake

Sorting by start time. That gives wrong answers on inputs like [[1,100], [2,3], [4,5]] — you'd keep [1,100] first and exclude both [2,3] and [4,5]. Sorting by end time keeps the short ones first.

  • Using > instead of >= when comparing next start to prev end — touching endpoints are compatible.
  • Initializing prev_end to 0 instead of -inf — can be wrong if intervals have negative endpoints.
  • Returning the kept count instead of the removed count — the problem asks how many to remove.

#986 · Interval List Intersections

MediumIntervalsTwo Pointers

The Problem

You are given two lists of closed intervals, firstList and secondList, each pairwise disjoint and sorted in ascending order by start time. Return the intersection of these two interval lists. Each intersection is itself a closed interval; if the result is empty (touching only) it still counts.

Brute Force

For every pair (a in firstList, b in secondList), compute the intersection [max(a.start, b.start), min(a.end, b.end)]. If the start <= end, include it. Sort the result. This is O(m * n).

Python — Brute Force 
def intervalIntersection(A: list[list[int]], B: list[list[int]]) -> list[list[int]]:
    res = []
    for a in A:
        for b in B:
            lo = max(a[0], b[0])
            hi = min(a[1], b[1])
            if lo <= hi:
                res.append([lo, hi])
    res.sort()
    return res
Go — Brute Force 
func intervalIntersection(A [][]int, B [][]int) [][]int {
    res := [][]int{}
    for _, a := range A {
        for _, b := range B {
            lo, hi := a[0], a[1]
            if b[0] > lo { lo = b[0] }
            if b[1] < hi { hi = b[1] }
            if lo <= hi {
                res = append(res, []int{lo, hi})
            }
        }
    }
    sort.Slice(res, func(i, j int) bool { return res[i][0] < res[j][0] })
    return res
}

Why optimize? O(m * n) ignores that BOTH inputs are already sorted and disjoint — two pointers can do this in O(m + n).

Key Insight

Both lists are sorted and disjoint, so we can walk them with two pointers. At each step, the current intervals A[i] and B[j] either intersect (record [max(starts), min(ends)]) or don't. Advance whichever pointer points to the interval ending earliest — that interval can't intersect with anything later.

How to Think About It

  1. Two pointers i, j start at 0.
  2. Compute candidate intersection: lo = max(A[i][0], B[j][0]), hi = min(A[i][1], B[j][1]).
  3. If lo <= hi, append [lo, hi] to results.
  4. Advance the pointer whose interval ends earlier: if A[i][1] < B[j][1], increment i, else increment j.
  5. Stop when either pointer is past the end of its list.

Why This Works

Once an interval ends before the other interval's end, it cannot intersect any later interval in the other list (which all start at or after the current other interval's start). So we safely discard it by advancing its pointer. The two-pointer walk is total O(m + n) because each step advances at least one pointer.

⚠ Most Dangerous Mistake

Advancing both pointers when intervals intersect. You only advance the one whose interval ends first — the other may still intersect with the next interval on the opposite list.

  • Using < instead of <= in the intersection check — intervals are closed, so single-point intersections [x, x] are valid.
  • Tie in end times: when A[i][1] == B[j][1], advance either (or both) — advancing just one is safe and clearer.
  • Forgetting to wrap the result as a list/slice rather than a tuple/array of ints in the wrong format.
