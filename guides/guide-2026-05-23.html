#4 · Median of Two Sorted Arrays

Hard Binary Search ~375 days since solved

The Problem

Given two sorted arrays nums1 and nums2 of size m and n, return the median of the two arrays combined. The required time complexity is O(log(min(m, n))).

Quick Reminder

Binary search on the partition index of the shorter array. You're finding the split where left-half max ≤ right-half min across both arrays.

Key Insight

Don't search for the median value — search for the correct partition. Pick i elements from nums1 and (m+n+1)/2 - i from nums2, then binary-search i until nums1[i-1] ≤ nums2[j] and nums2[j-1] ≤ nums1[i].

Medium Greedy / DP ~375 days since solved

The Problem

Given an array nums where nums[i] is the max jump length from position i, return the minimum number of jumps to reach the last index. You can assume the last index is always reachable.

Quick Reminder

BFS-by-levels with a greedy twist: track current_end (frontier of this jump) and farthest (best you can reach from anything in this frontier).

Key Insight

Each jump is a BFS level. When i reaches current_end, you've exhausted the current frontier — commit a jump and set current_end = farthest. Iterate only to n-1 (not n) to avoid an off-by-one extra jump.

The Problem

A message of digits is encoded with 'A'=1, 'B'=2, ..., 'Z'=26. Given a string of digits s, return the number of ways to decode it. Leading zeros and invalid pairs (e.g. "30") contribute zero ways.

Quick Reminder

It's a Fibonacci-shaped DP: dp[i] = dp[i-1] (if s[i-1] is 1-9) + dp[i-2] (if s[i-2..i-1] is 10-26).

#48 · Rotate Image

MediumMatrixIn-Place Rotation

The Problem

You are given an n × n 2D matrix representing an image. Rotate the image by 90 degrees clockwise. You must do this in place — do not allocate another 2D matrix.

Brute Force

Allocate a fresh n × n matrix and copy each cell to its rotated destination. The cell at row i, column j in the source lands at row j, column n-1-i in the destination. Then copy back into the input. This is O(n²) time and O(n²) extra space.

Python — Brute Force 
def rotate(matrix: list[list[int]]) -> None:
    n = len(matrix)
    rotated = [[0] * n for _ in range(n)]
    for i in range(n):
        for j in range(n):
            rotated[j][n - 1 - i] = matrix[i][j]
    # copy back in place (problem demands mutation of input)
    for i in range(n):
        for j in range(n):
            matrix[i][j] = rotated[i][j]
Go — Brute Force 
func rotate(matrix [][]int) {
    n := len(matrix)
    rotated := make([][]int, n)
    for i := range rotated {
        rotated[i] = make([]int, n)
    }
    for i := 0; i < n; i++ {
        for j := 0; j < n; j++ {
            rotated[j][n-1-i] = matrix[i][j]
        }
    }
    for i := 0; i < n; i++ {
        for j := 0; j < n; j++ {
            matrix[i][j] = rotated[i][j]
        }
    }
}

Why optimize? The problem explicitly forbids the extra n² allocation — interviewers ask this exact question to test whether you can find the O(1) trick.

Key Insight

A 90° clockwise rotation equals transpose, then reverse each row. Transpose swaps (i,j) with (j,i) across the main diagonal. Reversing each row then maps row i column j to row i column n-1-j. Composing these gives (i,j) → (j, n-1-i) — exactly the rotation.

How to Think About It

  1. Recognize the destination formula: (i, j) → (j, n-1-i) for a 90° clockwise rotation.
  2. Note that doing this with a copy is trivial — the challenge is in place. Naively swapping (i,j) with (j, n-1-i) overwrites cells you still need.
  3. Decompose the rotation into two simpler in-place ops that each are safe: transpose, then reverse rows.
  4. Transpose by iterating i < j (only the upper triangle) and swapping matrix[i][j] with matrix[j][i]. Starting j at i+1 avoids undoing your work.
  5. Reverse each row using a two-pointer swap (or the language's built-in reverse) to complete the rotation.

Why This Works

Transpose is its own inverse on cells with i ≠ j and a no-op on the diagonal. After transposing, the cell originally at (i, j) lives at (j, i). Reversing row j sends column i to column n-1-i. So the original (i, j) ends up at (j, n-1-i) — the correct rotated position. Both ops mutate disjoint pairs, so no cell is clobbered before it's read.

⚠ Most Dangerous Mistake

Transposing with the inner loop running for j in range(n) instead of for j in range(i+1, n). That swaps every pair twice and leaves the matrix unchanged. Always iterate the strict upper (or strict lower) triangle.

  • Confusing the direction: clockwise vs counter-clockwise. CCW is transpose then reverse columns, or equivalently reverse rows then transpose.
  • Trying a 4-way cycle swap (rotating four corners at once) and getting indices wrong — valid technique but error-prone; transpose+reverse is cleaner.
  • In Go, forgetting that matrix[i] is a slice header sharing the backing array — you can still reverse it in place; just don't reassign matrix[i] = ... if the caller expects mutation.

#54 · Spiral Matrix

MediumMatrixSimulation

The Problem

Given an m × n matrix, return all its elements in spiral order: traverse right across the top row, down the right column, left across the bottom row, up the left column, then spiral inward and repeat until every cell is visited.

Brute Force

Simulate by walking cell-by-cell with a direction vector and a visited matrix. At each step, attempt to move in the current direction; if you'd go out of bounds or onto a visited cell, rotate the direction 90° clockwise. Stop after collecting m * n values. Uses O(m*n) extra space for the visited grid.

Python — Brute Force 
def spiralOrder(matrix: list[list[int]]) -> list[int]:
    m, n = len(matrix), len(matrix[0])
    visited = [[False] * n for _ in range(m)]
    dirs = [(0, 1), (1, 0), (0, -1), (-1, 0)]  # R, D, L, U
    r = c = d = 0
    out = []
    for _ in range(m * n):
        out.append(matrix[r][c])
        visited[r][c] = True
        nr, nc = r + dirs[d][0], c + dirs[d][1]
        if nr < 0 or nr >= m or nc < 0 or nc >= n or visited[nr][nc]:
            d = (d + 1) % 4
            nr, nc = r + dirs[d][0], c + dirs[d][1]
        r, c = nr, nc
    return out
Go — Brute Force 
func spiralOrder(matrix [][]int) []int {
    m, n := len(matrix), len(matrix[0])
    visited := make([][]bool, m)
    for i := range visited {
        visited[i] = make([]bool, n)
    }
    dirs := [4][2]int{{0, 1}, {1, 0}, {0, -1}, {-1, 0}}
    r, c, d := 0, 0, 0
    out := make([]int, 0, m*n)
    for k := 0; k < m*n; k++ {
        out = append(out, matrix[r][c])
        visited[r][c] = true
        nr, nc := r+dirs[d][0], c+dirs[d][1]
        if nr < 0 || nr >= m || nc < 0 || nc >= n || visited[nr][nc] {
            d = (d + 1) % 4
            nr, nc = r+dirs[d][0], c+dirs[d][1]
        }
        r, c = nr, nc
    }
    return out
}

Why optimize? The visited matrix doubles memory. Interviewers want the O(1)-extra-space layered approach to prove you understand the spiral structure, not just brute simulation.

Key Insight

A spiral is a sequence of shrinking rectangular frames. Maintain four boundaries — top, bottom, left, right — and after walking each side, shrink the corresponding boundary inward. You never revisit a cell because each side strictly stays inside the current frame.

How to Think About It

  1. Initialize boundaries: top=0, bottom=m-1, left=0, right=n-1.
  2. Walk left-to-right across row top, then increment top. Walk top-to-bottom down column right, then decrement right.
  3. Walk right-to-left across row bottom — but only if top ≤ bottom (otherwise this row was already consumed). Then decrement bottom.
  4. Walk bottom-to-top up column left — but only if left ≤ right. Then increment left.
  5. Repeat until top > bottom or left > right. The two extra checks in steps 3-4 handle the degenerate single-row or single-column residue that appears in non-square matrices.

Why This Works

Each side traversal walks exactly the cells of the current frame's perimeter on that side, then we shrink that boundary so those cells are excluded from all future iterations. The frame strictly shrinks each loop, so termination is guaranteed in O(m+n) loop iterations covering O(m*n) cells total. The guards in steps 3 and 4 prevent double-counting when the remaining region degenerates to a single row or single column.

⚠ Most Dangerous Mistake

Forgetting the if top ≤ bottom / if left ≤ right guards before the third and fourth side traversals. On a matrix like 3×1 or 1×3, the final inner pass will re-walk the row or column you just consumed, duplicating values in the output.

  • Off-by-one in the side ranges. The top-row walk uses left .. right inclusive; the right-column walk uses top+1 .. bottom (or you can use top .. bottom after incrementing top — pick one convention and stick to it).
  • Forgetting that matrix[0] may not exist if m == 0. Always guard with if not matrix or not matrix[0]: return [].
  • Mutating the input matrix to track visited cells (e.g., overwriting with a sentinel) — works but destroys the input; bad if the caller still needs it.

#73 · Set Matrix Zeroes

MediumMatrixIn-Place Markers

The Problem

Given an m × n matrix, if any cell is 0, set its entire row and column to 0. Do it in place. Aim for O(1) extra space.

Brute Force

Scan the matrix once and record the set of rows and columns that contain a zero. Then make a second pass and zero out every cell whose row or column appears in either set. This is O(m*n) time and O(m+n) extra space — clear and correct, but not yet O(1) space.

Python — Brute Force 
def setZeroes(matrix: list[list[int]]) -> None:
    m, n = len(matrix), len(matrix[0])
    zero_rows, zero_cols = set(), set()
    for i in range(m):
        for j in range(n):
            if matrix[i][j] == 0:
                zero_rows.add(i)
                zero_cols.add(j)
    for i in range(m):
        for j in range(n):
            if i in zero_rows or j in zero_cols:
                matrix[i][j] = 0
Go — Brute Force 
func setZeroes(matrix [][]int) {
    m, n := len(matrix), len(matrix[0])
    zeroRows := make(map[int]bool)
    zeroCols := make(map[int]bool)
    for i := 0; i < m; i++ {
        for j := 0; j < n; j++ {
            if matrix[i][j] == 0 {
                zeroRows[i] = true
                zeroCols[j] = true
            }
        }
    }
    for i := 0; i < m; i++ {
        for j := 0; j < n; j++ {
            if zeroRows[i] || zeroCols[j] {
                matrix[i][j] = 0
            }
        }
    }
}

Why optimize? The follow-up explicitly asks for O(1) extra space. The interviewer wants to see the marker trick: reuse the first row and first column as your own bookkeeping.

Key Insight

Use the matrix itself as the bookkeeping. The first row records which columns need zeroing; the first column records which rows need zeroing. Then handle the first row and first column specially with two boolean flags, since they double as both data and markers.

How to Think About It

  1. Scan row 0 and column 0 first. Record two booleans: firstRowHasZero and firstColHasZero.
  2. Scan the interior (i ≥ 1, j ≥ 1). For each cell that is zero, set matrix[i][0] = 0 and matrix[0][j] = 0 — planting markers in the borders.
  3. Second pass over the interior: zero matrix[i][j] if matrix[i][0] == 0 or matrix[0][j] == 0.
  4. If firstRowHasZero, zero out the entire first row. If firstColHasZero, zero out the entire first column. Order doesn't matter as long as you finish the interior first.
  5. Done. No auxiliary structures beyond two booleans, so O(1) extra space.

Why This Works

Every interior zero needs to propagate to its full row and full column. By writing a marker into matrix[i][0] for the row and matrix[0][j] for the column, we compress all needed information into m+n cells we know we'll eventually zero anyway. The only ambiguity is whether row 0 / column 0 themselves originally had zeros, which the two boolean flags resolve. Because the interior is processed before the first row/column are clobbered, no information is lost.

⚠ Most Dangerous Mistake

Forgetting the two boolean flags and using the markers for everything. The cell matrix[0][0] can't simultaneously encode "row 0 has a zero" and "column 0 has a zero" — it has to mean one or the other. Skipping a separate flag for either dimension causes the wrong row or column to be zeroed.

  • Processing the first row/column before the interior. Once you zero the first row, every column marker disappears, and the second pass will zero columns that shouldn't be zeroed.
  • Iterating the second pass from i = 0 or j = 0 instead of 1. Reading matrix[0][j] while i = 0 is exactly the marker, which is true even for columns that only had zeros in row 0 — double-handled if you mix in the flag logic.
  • Assuming a non-empty matrix. Always early-return when len(matrix) == 0 or len(matrix[0]) == 0.

#101 · Symmetric Tree

EasyTreeDFS/BFS

The Problem

Given the root of a binary tree, determine whether the tree is a mirror image of itself around its center axis. In other words, the left subtree should be a reflection of the right subtree. An empty tree is considered symmetric.

Brute Force

Serialize the tree using an in-order traversal that emits explicit nulls for missing children, then compare the serialized string to its reverse. This works but uses O(n) extra space for the serialization and obscures the recursive structure of the problem. It also has subtle edge cases around null markers that can produce false positives for asymmetric trees with identical values.

Python — Brute Force 
def isSymmetric(root):
    def serialize(node):
        if not node:
            return ["#"]
        return [str(node.val)] + serialize(node.left) + serialize(node.right)
    s = serialize(root)
    # Compare against mirrored serialization
    def mirror(node):
        if not node:
            return ["#"]
        return [str(node.val)] + mirror(node.right) + mirror(node.left)
    return s == mirror(root)
Go — Brute Force 
func isSymmetric(root *TreeNode) bool {
    var serialize func(n *TreeNode) []string
    serialize = func(n *TreeNode) []string {
        if n == nil { return []string{"#"} }
        out := []string{fmt.Sprint(n.Val)}
        out = append(out, serialize(n.Left)...)
        return append(out, serialize(n.Right)...)
    }
    var mirror func(n *TreeNode) []string
    mirror = func(n *TreeNode) []string {
        if n == nil { return []string{"#"} }
        out := []string{fmt.Sprint(n.Val)}
        out = append(out, mirror(n.Right)...)
        return append(out, mirror(n.Left)...)
    }
    a, b := serialize(root), mirror(root)
    return strings.Join(a, ",") == strings.Join(b, ",")
}

Why optimize? Building string arrays is wasteful when a direct paired comparison can short-circuit on the first mismatch.

Key Insight

Symmetry is a relationship between two subtrees, not a property of one. Reframe the problem as: "Are these two trees mirrors of each other?" Then walk them in lockstep, comparing left.left with right.right, and left.right with right.left.

How to Think About It

  1. Notice the problem is really about two trees being mirrors, so write a helper isMirror(a, b) instead of operating on a single root.
  2. Handle the base cases first: both null is true; only one null is false; values different is false.
  3. Recurse with the mirror pairing: isMirror(a.left, b.right) AND isMirror(a.right, b.left).
  4. Kick off the recursion with isMirror(root.left, root.right) (or return true if root itself is null).
  5. For the BFS variant, push pairs into a queue and pop two at a time, applying the same mirror comparison.

Why This Works

A tree is symmetric iff its left and right subtrees are mirrors. Two trees are mirrors iff their roots match and each tree's left subtree mirrors the other's right subtree. This recursive definition is total and well-founded because each recursive call moves strictly closer to the leaves.

⚠ Most Dangerous Mistake

Comparing left.left with right.left instead of left.left with right.right. That checks if the two subtrees are identical, not mirror images. Trees like [1,2,2,3,4,3,4] would incorrectly pass.

  • Forgetting the early return when exactly one of the two nodes is null.
  • Returning true only on value match without recursing on the children.
  • Calling the helper on root itself instead of (root.left, root.right), causing infinite recursion or always-true results.

#112 · Path Sum

EasyTreeDFS

The Problem

Given the root of a binary tree and an integer targetSum, return true if the tree has a root-to-leaf path such that adding all the values along the path equals targetSum. A leaf is a node with no children. Return false for an empty tree.

Brute Force

Enumerate every root-to-leaf path explicitly into a list of lists, then for each path compute its sum and check if any equals the target. This works but builds and stores every path, even after a match is found, and forces a separate summation pass.

Python — Brute Force 
def hasPathSum(root, targetSum):
    if not root:
        return False
    paths = []
    def dfs(node, path):
        if not node.left and not node.right:
            paths.append(path + [node.val])
            return
        if node.left:  dfs(node.left, path + [node.val])
        if node.right: dfs(node.right, path + [node.val])
    dfs(root, [])
    return any(sum(p) == targetSum for p in paths)
Go — Brute Force 
func hasPathSum(root *TreeNode, targetSum int) bool {
    if root == nil { return false }
    var paths [][]int
    var dfs func(n *TreeNode, path []int)
    dfs = func(n *TreeNode, path []int) {
        path = append(path, n.Val)
        if n.Left == nil && n.Right == nil {
            cp := make([]int, len(path))
            copy(cp, path)
            paths = append(paths, cp)
            return
        }
        if n.Left != nil  { dfs(n.Left,  path) }
        if n.Right != nil { dfs(n.Right, path) }
    }
    dfs(root, nil)
    for _, p := range paths {
        s := 0
        for _, v := range p { s += v }
        if s == targetSum { return true }
    }
    return false
}

Why optimize? We waste memory storing whole paths and we never short-circuit when an early branch already matches.

Key Insight

Instead of carrying the running path, carry the remaining sum. At each step, subtract the current node's value from the target; at a leaf, the path matches iff the remaining sum equals the leaf's value (or equivalently, the remaining sum after subtraction is 0).

How to Think About It

  1. Define the recursion as "does any root-to-leaf path starting at this node sum to remaining?"
  2. Base case 1: a null node has no paths, so return false.
  3. Base case 2: a leaf matches iff node.val == remaining.
  4. Recursive case: try the left child with remaining - node.val OR the right child with the same reduced target.
  5. The OR gives free short-circuiting: as soon as one branch finds a match the whole call returns true.

Why This Works

Every root-to-leaf path is a sequence of edges from root down. Subtracting each node's value along the way is the same as summing the path; we have a hit iff some leaf zeroes out the remaining target. The recursion enumerates exactly the set of root-to-leaf paths.

⚠ Most Dangerous Mistake

Returning true at an internal node whose value zeroes out the target. The problem specifies root-to-leaf, so the match must occur at a node with both children null. Trees like [1,2] with target 1 should return false, not true.

  • Forgetting negative values: targetSum and node values can be negative, so paths can go above and below the target. Don't prune branches early on sign assumptions.
  • Returning true for an empty tree when target is 0. An empty tree has no leaves, so the answer must be false.
  • Confusing "no children" with "left is null OR right is null". A leaf needs BOTH children null.

#199 · Binary Tree Right Side View

MediumTreeBFS

The Problem

Given the root of a binary tree, imagine yourself standing on the right side of it. Return the values of the nodes you can see, ordered from top to bottom. Equivalently, return the rightmost node at each level of the tree.

Brute Force

Run a full level-order traversal that collects every node at every depth into a list-of-lists (like #102), then take the last element of each level list. This is correct but builds the entire level-by-level structure even though we only ever need the last value per level.

Python — Brute Force 
def rightSideView(root):
    if not root: return []
    from collections import deque
    levels, q = [], deque([root])
    while q:
        level = []
        for _ in range(len(q)):
            n = q.popleft()
            level.append(n.val)
            if n.left:  q.append(n.left)
            if n.right: q.append(n.right)
        levels.append(level)
    return [lvl[-1] for lvl in levels]
Go — Brute Force 
func rightSideView(root *TreeNode) []int {
    if root == nil { return []int{} }
    var levels [][]int
    queue := []*TreeNode{root}
    for len(queue) > 0 {
        size := len(queue)
        var level []int
        for i := 0; i < size; i++ {
            n := queue[i]
            level = append(level, n.Val)
            if n.Left  != nil { queue = append(queue, n.Left) }
            if n.Right != nil { queue = append(queue, n.Right) }
        }
        queue = queue[size:]
        levels = append(levels, level)
    }
    out := make([]int, 0, len(levels))
    for _, lvl := range levels { out = append(out, lvl[len(lvl)-1]) }
    return out
}

Why optimize? Building per-level lists allocates memory we throw away; we only need one value per level.

Key Insight

During a level-order traversal, the rightmost node of each level is exactly the last node popped in that level's batch. So as we process each level, just record the last node's value instead of accumulating a list. Equivalently with DFS: visit right child before left and record the first node seen at each new depth.

How to Think About It

  1. Recognize this as a level-order problem (#102 family): we care about a per-level fact.
  2. Capture the level boundary with the standard for _ in range(len(queue)) trick.
  3. Inside the inner loop, track the last node seen at this level (or just record the value on the final iteration).
  4. Append that value to the answer once the level is done.
  5. Alternative DFS framing: pre-order with right-first; if depth == len(result), append the node's value, since right-first guarantees the first node at a depth is the rightmost.

Why This Works

A BFS that processes the queue in left-to-right insertion order pops nodes of each level in left-to-right order, so the last pop is the rightmost. For DFS, right-first traversal means each new depth is first reached via the rightmost spine, so the first node observed at any depth is the rightmost one.

⚠ Most Dangerous Mistake

Assuming the right side view is simply the chain root.right.right.right... down the right spine. If a level has nodes only on the left subtree (the right spine ended early), those left-subtree nodes still appear in the view. Example: [1,2,3,4] — the view is [1,3,4], not [1,3].

  • Capturing queue[-1] before the inner loop runs — the queue still contains the next level's enqueued nodes, polluting the result.
  • Using len(queue) directly inside the inner loop instead of caching it; the queue grows as children are pushed and the loop runs forever or scans too far.
  • For DFS: using depth > len(result) instead of depth == len(result); the strict inequality is unreachable when you increment depth by 1 each call.

#455 · Assign Cookies

EasyGreedySorting

The Problem

You have an array g of children's greed factors and an array s of cookie sizes. You can give at most one cookie to each child, and child i is content iff the cookie they receive has size ≥ g[i]. Return the maximum number of content children.

Brute Force

Try every permutation of cookies (or every assignment from children to a subset of cookies) and count the content children for each, taking the max. This is exponential and infeasible beyond tiny inputs, but it does illustrate that we're really choosing a matching between two sets.

Python — Brute Force 
from itertools import permutations
def findContentChildren(g, s):
    best = 0
    for perm in permutations(s):
        count = 0
        for i, cookie in enumerate(perm):
            if i < len(g) and cookie >= g[i]:
                count += 1
        best = max(best, count)
    return best
Go — Brute Force 
func findContentChildren(g, s []int) int {
    best := 0
    var perm func(arr []int, k int)
    perm = func(arr []int, k int) {
        if k == len(arr) {
            count := 0
            for i := 0; i < len(arr) && i < len(g); i++ {
                if arr[i] >= g[i] { count++ }
            }
            if count > best { best = count }
            return
        }
        for i := k; i < len(arr); i++ {
            arr[k], arr[i] = arr[i], arr[k]
            perm(arr, k+1)
            arr[k], arr[i] = arr[i], arr[k]
        }
    }
    perm(append([]int{}, s...), 0)
    return best
}

Why optimize? O(n!) is unusable. We need a constructive rule that picks the right matching directly.

Key Insight

Sort both arrays. Walk through cookies smallest-first: assign the current cookie to the least-greedy unsatisfied child if it fits; otherwise discard it and try the next cookie. Using the smallest cookie that satisfies a child "wastes the least" capacity, leaving bigger cookies for greedier children.

How to Think About It

  1. Sort both g (greed) and s (sizes) in ascending order.
  2. Use two pointers, i over children, j over cookies, both starting at 0.
  3. If s[j] ≥ g[i], child i is satisfied: increment both pointers and your answer counter.
  4. Otherwise the cookie is too small for even this least-greedy unsatisfied child, so discard it: increment j only.
  5. Stop when either array is exhausted; the counter is the answer.

Why This Works

An exchange argument: suppose an optimal matching assigns a larger cookie to a child who could have been satisfied by a smaller available cookie. Swapping the assignments still satisfies the same child and frees the larger cookie for someone greedier — it can only help. Repeating exchanges drives any optimal solution toward the greedy's "smallest cookie that fits" rule.

⚠ Most Dangerous Mistake

Trying to match the greediest child first with the biggest cookie. This works too, but if you mix the direction of iteration with the wrong pointer move (e.g., decrement i when the cookie is too small) you'll undercount. Pick one direction and be consistent.

  • Forgetting to sort one of the arrays.
  • Incrementing both pointers on a failed match, effectively dropping a child who could still be served.
  • Returning min(len(g), len(s)) when the cookies aren't big enough — counts impossible matches.

#122 · Best Time to Buy and Sell Stock II

MediumGreedyArray

The Problem

You are given an integer array prices where prices[i] is the price of a stock on day i. You may buy and sell as many times as you want, but you can hold at most one share at a time and must sell before buying again. Return the maximum total profit.

Brute Force

Recursively explore every state: on each day choose to buy (if not holding), sell (if holding), or skip. Take the max profit over all choice sequences. This is exponential without memoization but reflects the underlying decision tree.

Python — Brute Force 
def maxProfit(prices):
    def dfs(i, holding):
        if i == len(prices):
            return 0
        skip = dfs(i + 1, holding)
        if holding:
            sell = prices[i] + dfs(i + 1, False)
            return max(skip, sell)
        else:
            buy = -prices[i] + dfs(i + 1, True)
            return max(skip, buy)
    return dfs(0, False)
Go — Brute Force 
func maxProfit(prices []int) int {
    var dfs func(i int, holding bool) int
    dfs = func(i int, holding bool) int {
        if i == len(prices) { return 0 }
        skip := dfs(i+1, holding)
        if holding {
            sell := prices[i] + dfs(i+1, false)
            if sell > skip { return sell }
            return skip
        }
        buy := -prices[i] + dfs(i+1, true)
        if buy > skip { return buy }
        return skip
    }
    return dfs(0, false)
}

Why optimize? The recursion tree has 2^n branches. The greedy observation collapses it to a single linear scan.

Key Insight

Any profitable buy-on-day-i, sell-on-day-j trip decomposes into the sum of consecutive day-to-day gains in between, because p[j] - p[i] = (p[i+1]-p[i]) + (p[i+2]-p[i+1]) + ... + (p[j]-p[j-1]). So the maximum total profit equals the sum of all positive consecutive differences: take every uphill step and skip every downhill.

How to Think About It

  1. Notice you're allowed unlimited transactions with no fee, so every distinct uphill segment can be captured.
  2. Pair each day i with day i+1 and look at prices[i+1] - prices[i].
  3. Sum the positive differences only — the negatives correspond to dips you would skip in an optimal strategy.
  4. This is equivalent to buying at every local minimum and selling at every local maximum.
  5. Implementation is a one-line accumulator over consecutive differences.

Why This Works

Telescoping: any buy-sell interval profit equals the sum of consecutive differences inside it. Since we may transact freely, we can always restructure trades to capture each positive consecutive diff and avoid each negative one. No alternative strategy can sum to more than the sum of positives, because every interval profit is bounded above by the positives it overlaps.

⚠ Most Dangerous Mistake

Trying to identify "valleys and peaks" with explicit state machines and getting the boundaries wrong (off-by-one at the array's end, double-counting plateaus). The consecutive-diff sum sidesteps all of this.

  • Buying again before selling (the problem forbids holding more than one share).
  • Subtracting all differences instead of clamping at zero.
  • Confusing this with problem #121 (single transaction) and accidentally tracking min_price across the whole array.

#134 · Gas Station

MediumGreedyArray

The Problem

There are n gas stations on a circular route. At station i you can refuel gas[i] liters; driving from station i to i+1 costs cost[i] liters. Your tank starts empty. Return the index of the station where you should start to complete the full circle exactly once, or -1 if impossible. The answer is guaranteed unique when it exists.

Brute Force

Try every station as a starting point. From each candidate, simulate driving around the loop and verify that the tank never goes negative. If a full circle succeeds, return that start; otherwise return -1 after exhausting all candidates. This is O(n^2).

Python — Brute Force 
def canCompleteCircuit(gas, cost):
    n = len(gas)
    for start in range(n):
        tank, ok = 0, True
        for k in range(n):
            i = (start + k) % n
            tank += gas[i] - cost[i]
            if tank < 0:
                ok = False
                break
        if ok:
            return start
    return -1
Go — Brute Force 
func canCompleteCircuit(gas, cost []int) int {
    n := len(gas)
    for start := 0; start < n; start++ {
        tank, ok := 0, true
        for k := 0; k < n; k++ {
            i := (start + k) % n
            tank += gas[i] - cost[i]
            if tank < 0 { ok = false; break }
        }
        if ok { return start }
    }
    return -1
}

Why optimize? O(n^2) re-simulates work that we could have used to skip large blocks of starts in a single pass.

Key Insight

Two observations: (1) A solution exists iff sum(gas) ≥ sum(cost). (2) If we begin at some station and run out of fuel between stations i and j, then no station in [i, j] can be the answer — each of those starts inherits a non-negative tank when first visited, so they all fail at or before j too. Therefore the next candidate is j + 1, and one linear pass suffices.

How to Think About It

  1. Compute total = sum(gas[i] - cost[i]) while iterating; if it's negative, return -1 immediately.
  2. Maintain a tank running sum and a start candidate (initially 0).
  3. For each station i, add gas[i] - cost[i] to tank.
  4. If tank becomes negative, set start = i + 1 and reset tank = 0: every earlier start has been ruled out.
  5. After one pass, if total ≥ 0 return start, else return -1.

Why This Works

Existence: sum(gas) ≥ sum(cost) is necessary (obvious) and sufficient (by the skipping argument plus circular structure). Skipping: if starting at s the tank first dips below zero at station j, then for any s' ∈ [s, j] the tank at j starting from s' is at most the tank starting from s (we removed the non-negative contributions from s..s'-1), so s' also fails. The first viable start is therefore j + 1.

⚠ Most Dangerous Mistake

Resetting start = i instead of start = i + 1 on the failure step. The current station i just took us negative, so it cannot itself be a viable start — we must move past it.

  • Forgetting to reset tank to 0 after updating start.
  • Conflating tank (per-segment) with total (overall feasibility check) into a single variable.
  • Returning start without verifying total ≥ 0, which produces wrong answers when no solution exists.

Python Features
