The ProblemEach child i has a greed factor g[i] (the minimum cookie size that content them) and each cookie j has size s[j]. Assign at most one cookie per child; a child is content if their cookie's size ≥ their greed. Return the maximum number of content children.
Quick ReminderSort both arrays and walk them with two pointers, always feeding the smallest sufficient cookie to the least greedy unsatisfied child.
Key InsightGreedy by sorting: give the smallest cookie that can satisfy the least-greedy remaining child. If the current cookie is too small for the current child, it is too small for everyone left (they are greedier) — discard it. Never waste a big cookie on a child a smaller one would have satisfied.

class Solution(object):
    def nextGreaterElement(self, nums1, nums2):
        """
        :type nums1: List[int]
        :type nums2: List[int]
        :rtype: List[int]
        """
        ans = []
        for n1 in nums1:
            i = 0
            while n1 != nums2[i]:
                i += 1
            found = False
            for j in range(i, len(nums2)):
                n2 = nums2[j]
                if n2 > n1:
                    ans.append(n2)
                    found = True
                    break
            if not found:
                ans.append(-1)
        return ans

Why optimize? With n = 10⁵ a worst case (descending temps) does ~5×10⁹ comparisons, far too slow inside a typical time limit.

Key InsightBrute force rescans the future for every day. Instead, keep a stack of indices whose answer is still unknown, in decreasing temperature order. When a new warmer day arrives it resolves every colder day waiting on the stack at once — each index is pushed and popped exactly once. Same monotonic-stack pattern reused by #84 and #496.
How to Think About It
  1. Notice every day is "waiting" for the next warmer day to its right — a classic next-greater-element question.
  2. Walk left to right. Maintain a stack of indices of days that have not yet seen a warmer day.
  3. The temps of stacked indices stay strictly decreasing from bottom to top — otherwise the colder one would already be resolved.
  4. For each new day, while it is warmer than the temp at the stack's top index, that top day's answer is the distance to today; pop it.
  5. Push the current index. Anything left on the stack at the end keeps its default 0.
Why This WorksAn index stays on the stack only while no warmer day has appeared after it. When the current day is warmer than the stacked top, today is by construction the first warmer day to its right (we process days in order and resolve eagerly), so i - topIndex is exactly the wait. Each index is pushed once and popped at most once, giving linear total work despite the inner while-loop.
Watch Out For
  • Store indices on the stack, not temperatures — you need the gap j - i, and equal temperatures must not resolve each other.
  • The pop condition must be strictly greater (>); using >= would treat an equal-temperature day as "warmer."
  • Days remaining on the stack at the end have no warmer future day; leave their answer at 0, do not overwrite.

Why optimize? Each reduction restarts the scan and shifts elements, so with n tokens you do ~n²/2 work plus repeated array splicing — wasteful when a single pass suffices.

Key InsightPostfix notation is built for a stack: push operands; when you hit an operator, pop the two most recent operands, combine, and push the result. The repeated rescans of brute force vanish because the stack already remembers exactly which operands are pending — one left-to-right pass evaluates everything. Same stack discipline as #20 and #739.
How to Think About It
  1. Recognize that in postfix, an operator always applies to the two values that came just before it — the most recently seen operands.
  2. A stack naturally tracks "most recent first," so push every number you read.
  3. On an operator, pop the right operand first, then the left operand (order matters for - and /).
  4. Compute, then push the single result back so it can serve as an operand for a later operator.
  5. After consuming all tokens, exactly one value remains on the stack — the answer.
Why This WorksA valid RPN expression guarantees that whenever an operator appears, at least two operands precede it and remain unconsumed; those are exactly the top two stack elements because operands are pushed in reading order and each operator consumes the two most recent. Replacing the pair with their result preserves this invariant, so the process collapses the expression to its single value, mirroring how the recursion tree of the equivalent infix expression would fold.
Watch Out For
  • Operand order: the first pop is the right-hand operand. Computing b - a or b / a instead of a - b, a / b is a classic bug.
  • Division must truncate toward zero, not floor. -7 / 2 must be -3, so in Python use int(a / b) or math.trunc, not //.
  • A token like "-3" is a negative number, not the minus operator — check the operator set explicitly before parsing.

  2. For bar i the rectangle width is bounded by the nearest bar to the left that is shorter, and the nearest to the right that is shorter.
  3. Keep a stack of indices with non-decreasing heights. When a new bar is shorter than the stack top, the top's right boundary is found — it is the current index.
  4. Pop that top; its left boundary is the new stack top after popping; compute its area as height × (right − left − 1).
  5. Push the current index. Use a sentinel height of 0 (or process to the end) so every remaining bar gets flushed.
Why This WorksThe stack always holds indices whose heights increase from bottom to top, so when bar i is shorter than the top, every popped bar has i as its first shorter bar on the right and the element beneath it as its first shorter bar on the left. Those two boundaries define the widest rectangle in which the popped bar is the minimum height, so its area is computed exactly once and never missed. Each index is pushed and popped a single time, giving linear time.
Watch Out For
  • Width arithmetic after popping is i - stack[-1] - 1; if the stack is empty after popping, the rectangle reaches the array's left edge so width is just i.
  • Forgetting to flush bars still on the stack at the end (use a trailing sentinel of height 0 or a final drain loop) loses the tallest rectangles.
  • Areas can exceed 32-bit range conceptually (height 10⁴ × width 10⁵ = 10⁹); ensure the accumulator type is wide enough in stricter languages.

Key InsightComputing height and checking balance are the same downward pass. How to Think About It
  1. See the redundancy: the brute force computes each node's height once for the balance check and again as part of its parent's check.
  2. Combine the two jobs — let one recursive call both compute the height and verify balance on the way back up.
  3. Process children first (post-order); you need their heights before you can judge the current node.
  4. If a child subtree is already unbalanced, propagate that failure up immediately instead of doing more work.
  5. Encode "unbalanced" as an impossible height (−1) so the return value carries both pieces of information.

  • The sentinel (−1) must be a value height can never legitimately take, so the parent can distinguish "unbalanced" from a genuine height.

The ProblemGiven the root of a binary tree, return the length of its diameter: the number of edges on the longest path between any two nodes. The path need not pass through the root. Up to 10⁴ nodes.

Key InsightThe diameter through a node is left-height + right-height, and you already compute heights while traversing. This is the same height-plus-side-effect trick as #110: compute once, observe everything on the way up.
How to Think About It
  1. Realize the longest path "bends" at some single node, with one arm going into the left subtree and one into the right.
  2. The path length at that bend is left-subtree height + right-subtree height (counting edges).
  3. So the answer is the maximum of that quantity over all nodes — you just need every node's two child heights.
  4. One post-order DFS already produces each subtree's height; while returning it, record left + right against a running maximum.
  5. Return height (1 + max of children) upward so the parent can do the same with O(1) extra work.
Why This WorksAny path between two nodes has a unique highest node — its bend point. At that node the path uses the deepest descent into each child, namely the two child heights, so left + right is exactly the path's edge count. Maximizing this over every possible bend point covers every path. Because the DFS computes each height once and updates the global maximum in constant time per node, the entire search is linear.
Watch Out For
  • The answer counts edges, not nodes; a path through k nodes has k−1 edges. Don't add an extra 1 when combining child heights.
  • The diameter need not pass through the root, so a single global maximum updated at every node is required — returning only the root's value is wrong.
  • Keep the return value (height) separate from the tracked answer (max left + right); conflating them is a frequent bug.

#236 Lowest Common Ancestor of a Binary Tree MediumTreesDFS

The ProblemGiven the root of a binary tree (not a BST) and two distinct nodes p and q that both exist in the tree, return their lowest common ancestor — the deepest node that has both p and q as descendants (a node is a descendant of itself). Up to 10⁵ nodes.
Brute Force — O(n²) time, O(n) space

Write a helper that tests whether a target node lies in a given subtree. Then walk down from the root: as long as both p and q are in the same child's subtree, descend into that child. When they split across the two children (or the current node is one of them), the current node is the LCA. The subtree-membership test rescans nodes repeatedly.

Python
def lowestCommonAncestor(root, p, q):
    def contains(node, target):
        if not node:
            return False
        if node is target:
            return True
        return contains(node.left, target) or contains(node.right, target)
    cur = root
    while cur:
        if cur is p or cur is q:
            return cur
        in_left_p = contains(cur.left, p)
        in_left_q = contains(cur.left, q)
        if in_left_p and in_left_q:
            cur = cur.left
        elif not in_left_p and not in_left_q:
            cur = cur.right
        else:
            return cur

How to Think About It

  2. Base cases: an empty subtree returns null; a node equal to p or q returns itself.
  3. Recurse into both children and look at what each returns.
  4. If both children return non-null, the two targets are split beneath the current node, so the current node is the LCA.
  5. If only one side is non-null, bubble that side's result up unchanged — the LCA (or a lone target) is over there.
Why This WorksPost-order ensures each node learns its children's results before deciding. A node sees non-null from both children exactly when one target lies in its left subtree and the other in its right, which makes it their deepest common ancestor. If a node itself equals a target, returning itself lets an ancestor detect the split, correctly handling the case where one node is an ancestor of the other. Since every node is visited once and combines child results in O(1), the traversal is linear and the first qualifying node encountered on the way up is the lowest such ancestor.
Watch Out For
  • Handle the ancestor case: if p is itself an ancestor of q, returning p as soon as you match it yields the correct LCA — don't require both targets to be found in different subtrees.
  • Compare nodes by identity (reference / pointer equality), not by value; values can repeat or be irrelevant in a non-BST.
  • Both children non-null means "found something on each side," not "found p on left and q on left" — once one target is matched you stop recursing into it, so a non-null signal is enough.

Key Insight"Can finish all courses" is exactly "the prerequisite graph is a DAG (no cycle)." Rather than testing each node separately, do one global pass: either Kahn's algorithm (repeatedly remove courses with no remaining prerequisites and count how many you removed) or a single DFS with three node colors to catch a back-edge. How to Think About It
  1. Translate the requirement: finishing all courses is possible iff the directed dependency graph contains no cycle.
  2. Build the graph once, with an edge from prerequisite to dependent, and track each course's in-degree (number of unmet prerequisites).
  3. Kahn's view: any course with in-degree 0 can be taken now; take it, then decrement the in-degree of everything depending on it.
  4. Repeat, always pulling in newly-freed (in-degree 0) courses, counting how many courses you successfully scheduled.
  5. If you schedule all numCourses, there was no cycle; if some remain stuck with positive in-degree, a cycle blocks them.
Why This WorksA directed graph has a topological order iff it is acyclic. Kahn's algorithm only ever schedules a course once all its prerequisites are already scheduled, so it can place exactly the courses reachable in a valid order. If a cycle exists, every course on the cycle perpetually has at least one unscheduled prerequisite (another cycle member), so its in-degree never reaches 0 and it is never counted. Thus the count equals numCourses precisely when the graph is acyclic, and each node and edge is processed once.
Watch Out For
  • Edge direction: [a, b] means b → a (b before a). Reversing it inverts the entire dependency and breaks cycle detection.
  • Courses with no prerequisites and no dependents still count — initialize the queue with all in-degree-0 nodes, not just those that appear in edges.
  • With the DFS coloring approach, you need three states (unvisited / in-progress / done); a plain boolean visited set cannot distinguish a back edge (cycle) from a cross edge to an already-finished node.

    let mut result: Option<Vec<i32>> = None;
    fn permute(k: usize, order: &mut Vec<i32>, prereq: &[Vec<i32>], result: &mut Option<Vec<i32>>) {
        if result.is_some() { return; }
        if k == order.len() {

            for (i, &c) in order.iter().enumerate() { pos[c as usize] = i as i32; }
            if prereq.iter().all(|e| pos[e[1] as usize] < pos[e[0] as usize]) {
                *result = Some(order.clone());
            }
            return;
        }
        for i in k..order.len() {
            order.swap(k, i);
            permute(k + 1, order, prereq, result);
            order.swap(k, i);
        }
    }
    permute(0, &mut order, &prerequisites, &mut result);
    result.unwrap_or_default()
}

Key InsightThis is exactly the cycle check of #207 with one addition: record the order in which courses become schedulable. Kahn's algorithm produces a valid topological order for free — append each course as you remove it. If the produced order is shorter than numCourses, a cycle exists and you return empty. How to Think About It
  1. Reuse #207's insight: a valid order exists iff the graph is acyclic, and Kahn's algorithm both detects this and builds the order.
  2. Compute in-degrees and seed a queue with every course that has no prerequisites.
  3. Pop a course, append it to the output list — it is safe to take now — then relax its dependents.
  4. For each dependent, decrement its in-degree; when it hits 0, enqueue it as newly schedulable.
  5. When the queue empties, check the output length: if it equals numCourses, return it; otherwise a cycle blocked some courses, so return an empty array.
Why This WorksA course is appended only after all its prerequisites have already been appended (its in-degree reached 0 only because each predecessor was processed), so the emitted sequence respects every prerequisite by construction. Every node and edge is touched once: a node is enqueued exactly when its last prerequisite is removed, and each edge triggers one in-degree decrement. If a cycle exists, its members can never reach in-degree 0, so they are omitted and the length falls short — precisely the signal to return empty.
Watch Out For
  • Returning a partial order on a cyclic graph is wrong — you must return an empty array, so always compare the output length to numCourses before returning it.
  • Any valid order is accepted, but it must be complete; don't stop early after placing only the initially-free courses.
  • Direction again: append the course you remove (the prerequisite side), not its dependents; getting this backwards produces a reversed, invalid order.

The ProblemGiven a reference to a node in a connected undirected graph, return a deep copy (clone) of the entire graph. Each node holds a value and a list of neighbors. The returned graph must be structurally identical but share no nodes with the original.
Brute Force — O(V+E) time, O(V) space

You still must visit every node and every edge, so the asymptotic cost is unavoidable. The "brute" version walks the graph and naively rebuilds neighbor lists, but without remembering which originals are already cloned it would loop forever on cycles. The minimal correct naive approach is a two-pass copy: first create a clone for every reachable node, then in a second pass wire up neighbor edges by looking up clones in a map.

Why optimize? The two-pass version is already O(V+E); the real "optimization" is correctness — without a visited map a single cycle makes the recursion never terminate, and you can fuse the two passes into one to avoid walking every edge twice.

Key InsightThe hard part is not cost but cycles: a naive recursive copy revisits already-cloned nodes forever. Keep a map old node → new node. Create the clone the first time you see an original and record it before recursing into its neighbors. On any later visit, return the existing clone instead of making a new one. The map doubles as the "visited" set and the lookup table for wiring edges.
How to Think About It
  1. Handle the empty graph: if the input node is null, return null immediately.
  2. Keep a dictionary mapping each original node to its clone; this prevents infinite loops on cycles.
  3. When you reach a node, check the map: if it already has a clone, hand that back — do not recurse further.
  4. Otherwise create the clone, store it in the map first, then process its neighbors so a cycle back to this node finds the clone already present.
  5. For each neighbor, get-or-create its clone and append it to the current clone's neighbor list.
  6. Return the clone of the starting node, which now anchors the fully wired copy.
Why This WorksBecause the graph is connected, a DFS/BFS from the given node reaches every node and edge. Inserting a clone into the map before recursing guarantees that when traversal cycles back, the lookup succeeds and recursion stops, so each node is created exactly once. Wiring each neighbor through the same map ensures both endpoints of every edge point at clones, never originals, producing a structurally identical but disjoint graph.
Watch Out For
  • Record the clone in the map before recursing into neighbors — otherwise a cycle re-enters the same node and loops forever.
  • Don't forget the null-input edge case; returning a fresh node for null corrupts the answer.
  • Map keys must be node identity (the original reference), not the value — duplicate values would collide and merge distinct nodes.

Key InsightThe two ordering rules mean that reading the matrix row by row, left to right, yields one fully sorted sequence of length m·n. So the matrix is a sorted array in disguise. Run a single binary search over indices 0 .. m·n-1 and map a flat index i back to a cell with row = i / n, col = i % n.
How to Think About It
  1. Note that flattening the matrix row-major produces a globally sorted array because each row's first beats the previous row's last.
  2. Set lo = 0 and hi = m·n - 1 as bounds over the imaginary flat array.
  3. Take the midpoint mid, then convert it to a real cell using integer division and modulo by the column count n.
  4. Compare that cell's value to the target and discard the half that cannot contain it, exactly like ordinary binary search.
  5. Stop when you find the target (return true) or the window empties (return false).
Why This WorksThe row constraint guarantees that for any flat indices i < j, the mapped value at i is ≤ the mapped value at j: within a row it follows from left-to-right sorting, and across rows it follows from "first of next row > last of previous row." A monotonic sequence is exactly what binary search requires, so halving the search window each step is sound and terminates in O(log(m·n)).
Watch Out For
  • Use n (number of columns) for the index math, not m — mixing them up addresses the wrong cell.
  • Guard against an empty matrix or empty rows before computing m·n, or you divide by zero / read out of bounds.
  • Watch the half-open vs inclusive bounds convention — an off-by-one in hi can skip the last element or loop forever.

Key Insight"First index" and "last index" are two boundary problems. Use the binary-search bound template (same one used to find an insertion point): a left-bound search finds the leftmost index where nums[i] >= target; a right-bound search finds the leftmost index where nums[i] > target, whose predecessor is the last target. Run the boundary search twice with different comparisons instead of stopping at the first match.
How to Think About It
  1. Realize a plain "found it" binary search lands on some occurrence, not necessarily the edge — you need to keep searching toward the boundary.
  2. For the first position, binary search for the leftmost index whose value is >= target; if that value equals the target, it's the first index, else the target is absent.
  3. For the last position, binary search for the leftmost index whose value is > target; the index just before it is the last occurrence.
  4. When the value at mid ties with the target, do not return — shrink toward the side you want (left for first, right for last).
  5. If the first-position search fails to find the target, short-circuit and return [-1, -1].
Why This WorksOn a sorted array the predicate "value ≥ target" is monotonic: false for a prefix, then true for a suffix. Binary search reliably finds that false/true boundary, which is the first index not less than the target — the leftmost occurrence when present. The same monotonic argument on "value > target" gives one past the last occurrence. Each search halves the window, so both run in O(log n) and together pin down both edges exactly.
Watch Out For
  • Don't return as soon as nums[mid] == target — that lands on a middle occurrence, not the boundary.
  • After the left-bound search, verify the index is in range and the value equals the target before trusting it.
  • For the last position, remember the right-bound search returns one past the answer — subtract one.

Why This WorksConsider the half you keep. If you go right on an uphill step, that half starts by rising and ends at the −∞ right boundary, so somewhere it must stop rising — a peak. Watch Out For
  • Use hi = mid (not mid - 1) on the downhill branch — mid itself may be the peak.
  • Compare against the right neighbor mid + 1; make sure mid + 1 stays in range by keeping the window half-open or stopping at lo == hi.
  • Don't reach for a sorted-array assumption — this is binary search on a local property (slope), not on order.

Key InsightStore words in a trie (prefix tree): each node has up to 26 children plus an end-of-word flag. addWord walks/creates one path. search walks the trie character by character; on a normal letter follow that single child, but on a '.' recurse into every existing child (DFS over branches). How to Think About It
  1. Model each node as a small map/array of 26 child pointers plus a boolean marking that a word ends here.
  2. For addWord, descend from the root creating missing children, then flag the final node as a word end.
  3. For search, process the query character by character starting at the root.
  4. On a concrete letter, move to that one child if it exists; if it doesn't, this path fails.
  5. On a '.', try every present child by recursing — succeed if any branch matches the rest of the query.
  6. When the query is fully consumed, return whether the current node is flagged as a word end.
Why This WorksA trie path from the root spells exactly the prefixes of inserted words, and the end flag marks complete words, so a query that walks to an end-flagged node corresponds to a stored word. The wildcard turns search into a DFS: '.' branches into all children, exploring every word of the right shape, but only along trie paths that actually exist — pruning all impossible prefixes. Because shared prefixes collapse into shared paths, total work is bounded by the trie size, not the corpus size times query length.
Watch Out For
  • Only return true at the end of the query if the node's word-end flag is set — reaching an intermediate node is not a match.
  • For '.' you must try all children and OR their results; stopping at the first child is wrong.
  • A missing child on a concrete letter means immediate failure on that branch — don't index into a null/absent node.

Brute Force — O(W · m·n · 4^L) time

            let (ru, cu) = (r as usize, c as usize);
            if b[ru][cu] as u8 != w[k] { return false; }
            let ch = b[ru][cu]; b[ru][cu] = '#';
            let f = dfs(b,r+1,c,w,k+1,m,n) || dfs(b,r-1,c,w,k+1,m,n)
                 || dfs(b,r,c+1,w,k+1,m,n) || dfs(b,r,c-1,w,k+1,m,n);
            b[ru][cu] = ch;
            f
        }
        let mut res = vec![];
        for w in &words {
            let bytes = w.as_bytes();
            'outer: for r in 0..m { for c in 0..n {
                if dfs(&mut b, r as i32, c as i32, bytes, 0, m as i32, n as i32) {
                    res.push(w.clone()); break 'outer;
                }
            }}
        }
        res
    }
}

Why optimize? With up to ~3×10⁴ words on a 12×12 board, restarting a 4-way DFS per word re-walks shared prefixes thousands of times — the per-word factor explodes. Building a trie of all words lets one grid DFS match every word simultaneously.

Key InsightFlip the search around: instead of one DFS per word, build a trie of all words (extending #211's trie) and run a single backtracking DFS over the board that walks the trie in lockstep. From each cell, descend only into trie children that exist for the current letter — this prunes dead branches across all words at once. When you reach a trie node flagged as a word end, record that word.
How to Think About It
  1. Insert every target word into a trie, storing the full word string at each end node so a match is reported instantly.
  2. Start a DFS from every board cell, but only proceed if the cell's letter is a child of the current trie node.
  3. Mark the cell visited (e.g. overwrite it) before recursing and restore it after — standard backtracking so cells aren't reused within one word.
  4. Move the trie pointer to the matched child and recurse into the four neighbors with that child node.
  5. Whenever the current trie node carries a stored word, add it to the results and clear it to avoid duplicates.
  6. Optionally prune leaf trie nodes after collecting to shrink later searches.
Why This WorksThe trie collapses all words sharing a prefix into one path, so a single grid step advances the search for every word with that prefix simultaneously — no redundant re-exploration. Backtracking (mark/unmark) enforces the "each cell once per path" rule exactly. Because the DFS only follows letters that lead to real trie children, any board path that can't continue toward some word is abandoned immediately, bounding work by the trie structure rather than by words times grid.
Watch Out For
  • De-duplicate results: the same word can be reachable by multiple paths — null out the end marker after collecting, or use a set.
  • Restore the cell after recursing in all branches, including early returns, or you corrupt the board for sibling paths.
  • Match against trie children, not by re-scanning the word list — rebuilding the #211 trie is the whole point.

#261 Graph Valid Tree MediumUnion-FindGraph

The ProblemGiven n nodes labeled 0..n-1 and a list of undirected edges, determine whether they form a valid tree — a connected, acyclic graph. Return true or false.
Brute Force — O(V+E) time, O(V+E) space

Build an adjacency list and run a DFS/BFS from node 0, tracking the parent to ignore the edge you came in on. If you ever reach an already-visited node that isn't the parent, there's a cycle. After traversal, the graph is a tree iff no cycle was found and every node was visited (connected).

Python
class Solution:
    def validTree(self, n, edges):
        if len(edges) != n - 1:
            return False                      # quick necessary check
        adj = [[] for _ in range(n)]
        for a, b in edges:
            adj[a].append(b); adj[b].append(a)
        seen = set()
        stack = [0]
        while stack:
            node = stack.pop()
            seen.add(node)
            for nb in adj[node]:
                if nb not in seen:
                    stack.append(nb)
        return len(seen) == n                 # connected?

Why optimize? The DFS is already O(V+E); the cleaner "optimization" is union-find, which decides validity in nearly O(E·α(n)) without building an adjacency list — and merges the cycle and connectivity checks into one pass.

Key InsightA graph on n nodes is a tree iff it has exactly n-1 edges and is fully connected (these two together also guarantee no cycle). Use union-find (same disjoint-set tool as #323): first reject unless edges.length == n-1; then union every edge — if both endpoints are already in the same set, that edge closes a cycle, so reject. If all unions succeed, the edge count forces connectivity, so it's a tree.
How to Think About It
  1. Recall the tree identity: connected + acyclic is equivalent to connected with exactly n-1 edges.
  2. Immediately return false if the edge count isn't n-1 — too few can't connect, too many must cycle.
  3. Initialize a disjoint-set with each node as its own parent.
  4. For each edge, find the roots of both endpoints; if they already share a root, this edge creates a cycle — return false.
  5. Otherwise union the two sets and continue.
  6. Survive all edges with exactly n-1 of them, and the graph is necessarily one connected acyclic component — a tree.
Why This WorksEach successful union reduces the number of components by one. Starting from n singletons, processing n-1 edges that never join already-connected nodes ends with exactly one component — full connectivity. The cycle check catches any edge whose endpoints are already connected, which is precisely what would create a loop. So the combination "exactly n-1 edges" and "no union ever found a shared root" is equivalent to connected and acyclic, the definition of a tree.
Watch Out For
  • Don't skip the n-1 edge-count gate — without it, a graph with a cycle plus a disconnected node could slip through some checks.
  • The edge-count check alone is insufficient and connectivity alone is insufficient; you need both (union-find folds them together).
  • Use union by rank/size and path compression, or worst-case finds degrade toward O(n) each.

#323 Number of Connected Components in an Undirected Graph MediumUnion-FindGraph

The ProblemGiven n nodes labeled 0..n-1 and a list of undirected edges, return the number of connected components in the graph.
Brute Force — O(V+E) time, O(V+E) space

Build an adjacency list, then sweep over all nodes; each time you hit an unvisited node, run a DFS/BFS that marks its entire component visited and increment a counter. The counter ends as the number of components.

Why optimize? The DFS is already O(V+E) and fine; union-find is the more elegant tool that answers the count directly while processing edges, without materializing an adjacency list — the same disjoint-set machinery #261 uses.

Key InsightStart with n separate components and use union-find (the same structure as #261). Every edge that joins two different sets merges them and reduces the component count by one; an edge within an already-joined set changes nothing. After processing all edges, the count of distinct roots is the number of connected components — no traversal needed.
How to Think About It
  1. Initialize a disjoint-set with every node as its own root and a counter set to n.
  2. For each edge, find the roots of its two endpoints.
  3. If the roots differ, union the sets and decrement the component counter.
  4. If the roots already match, the edge is redundant within one component — leave the counter alone.
  5. After all edges, return the counter (equivalently, count the number of nodes that are their own root).
Why This WorksEach node begins as an isolated component, so the count starts at n. Two nodes are in the same component exactly when they're connected by some path, and union-find maintains that "same root ⇔ connected" invariant as edges are added. Since merging two distinct components into one decreases the total by exactly one, decrementing on every successful union tracks the live component count precisely; redundant edges correctly leave it unchanged.
Watch Out For
  • Only decrement the counter when the two roots actually differ — counting every edge over-merges.
  • An isolated node with no edges is still its own component; starting the counter at n handles this automatically.
  • Without path compression and union by rank/size, deep chains make find slow on large inputs.
