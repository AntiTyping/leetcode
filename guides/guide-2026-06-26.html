Key InsightBecause the input is already sorted and non-overlapping, only the contiguous block of intervals that overlap newInterval needs merging — no re-sort is required.

from functools import lru_cache

class Solution:
    def maxProfit(self, k, prices):
        @lru_cache(None)
        def dp(i, remaining, holding):
            if i == len(prices) or remaining == 0:
                return 0

            do_nothing = dp(i+1, remaining, holding)
            do_something = 0

            if holding:
                do_something = prices[i] + dp(i+1, remaining-1, 0)
            else:
                do_something = -prices[i] + dp(i+1, remaining, 1)

            return max(do_nothing, do_something)

        return dp(0, k, 0)

class Solution(object):
    def change(self, amount, coins):
        """
        :type amount: int
        :type coins: List[int]
        :rtype: int
        """
        memo = [[-1] * (amount + 1) for _ in range(len(coins))]

        def ways(i, amount):
            if amount == 0:
                return 1
            if i == len(coins):
                return 0

            if memo[i][amount] != -1:
                return memo[i][amount]

            if coins[i] > amount:
                memo[i][amount] = ways(i + 1, amount)
            else:
                memo[i][amount] = ways(i, amount - coins[i]) + ways(i + 1, amount)

            return memo[i][amount]

        return ways(0, amount)

The ProblemGiven the root of a binary tree, return true if it is a valid binary search tree. How to Think About It

  2. A node is valid iff low < val < high.
  3. Recurse left with range (low, val) — everything left must stay below val.
  4. Recurse right with range (val, high) — everything right must stay above val.
  5. If every node passes its range check, the tree is a valid BST.

Why This WorksThe BST property is global: a left descendant must be smaller than every ancestor from which it branched right (and vice versa). The in-order variant works because a BST's in-order sequence is, by definition, its values in sorted order.

  • A node can be valid relative to its parent yet invalid relative to a grandparent — local parent-only checks are wrong.
  • Values can be INT_MIN/INT_MAX; use nullable bounds or a wider type so sentinels do not cause false positives.
  • In the in-order approach, track the previous value across the whole traversal, not per subtree.

Key InsightBrute force buffers all nodes per level and then discards all but the last. The insight: you only need ONE node per depth. Either do a BFS level-order traversal and take the last node dequeued at each level (this is a level-order problem at heart), or DFS visiting the RIGHT child before the left and record the first node seen at each new depth.
How to Think About It
  1. The answer has exactly one entry per tree level, top to bottom.
  2. BFS approach: process the tree level by level with a queue; for each level, the last node you dequeue is the rightmost — append its value.
  3. Capture the queue length BEFORE draining a level so you know how many nodes belong to it.
  4. DFS alternative: recurse the right child first, carrying the depth; the FIRST time you reach a new depth, that node is the rightmost — record it.
  5. Either way you store O(1) per level instead of the whole level.

  • Snapshot the queue size BEFORE the inner loop — the queue grows as you enqueue children mid-level.
  • A node can be visible even if its parent is not on the right edge (e.g. a deep left child with no right sibling) — do not just follow right pointers.
  • Empty tree returns an empty list, not [null].
  • In DFS, record only on the FIRST visit to a depth (compare result length to the current depth).

The ProblemA path in a binary tree is any sequence of nodes where each consecutive pair is connected by an edge; a node appears at most once and the path need not pass through the root. The path sum is the sum of the node values along it. Given the root, return the maximum path sum of any non-empty path. Node count is in [1, 3·10&sup4;] and values may be negative.
Brute Force — O(n²) time, O(h) space

Treat every node as the highest (turning) point of a path. For each node, recompute the best single downward arm into its left child and into its right child (clamped at 0), and form the candidate leftArm + node.val + rightArm; track the maximum over all nodes. The arm sums are recomputed from scratch at every node, making it quadratic.

    def maxPathSum(self, root):
        # For each node, recompute best downward arm into each child (O(n^2))
        def maxDown(node):
            if node is None:
                return 0
            return node.val + max(maxDown(node.left), maxDown(node.right), 0)

            visit(node.right)
        visit(root)

Key InsightBrute force recomputes downward arm sums independently at every node. The insight: a single post-order pass where each node returns its best single downward arm to its parent, while simultaneously updating a global best with the full leftArm + node + rightArm path. Each subtree is summed exactly once.
How to Think About It
  1. Define one recursive function that returns the max sum of a downward path STARTING at this node and going down one side only (a single “arm”).
  2. At a node, recursively get the left-arm and right-arm; clamp each to ≥ 0 (a negative arm is better dropped).
  3. Update a global maximum with node.val + leftArm + rightArm — the best path that bends through this node, using both arms.
  4. Return node.val + max(leftArm, rightArm) to the parent, since an extending path can pass through only ONE arm.
  5. Initialize the global best to −∞ so all-negative trees work.
  6. The answer is the global best after the single traversal.
Why This WorksEvery path has a unique highest node — its turning point. At that node the path equals leftArm + node + rightArm; the algorithm evaluates exactly that quantity at every node, so the global optimum is guaranteed to be among them. Returning only one arm upward respects the rule that a parent's path can enter a child through a single side, never both, which prevents counting an already-forked path as still extendable.
Watch Out For
  • Clamp negative arms to 0 (drop them), but still seed the global best with −∞ (not 0) so an all-negative tree can return a single negative node.
  • Return only ONE arm to the parent — returning both would let a path fork twice, which is invalid.
  • The “through node” value (both arms) updates the global but is NOT what you return upward.
  • Clamp before adding and use a wide enough integer type; large positive subtrees can overflow 32 bits.

How to Think About It
  1. Recognize that “rot spreads to all neighbors simultaneously each minute” is precisely breadth-first expansion outward from many starting points at once.
  2. Do one initial scan: push every rotten orange into the queue together, and count how many fresh oranges exist.
  3. Run BFS in discrete levels — process the entire current frontier as a single minute, rotting fresh neighbors and enqueueing them.
  4. Each time you rot a fresh orange, decrement the fresh count so leftovers are detectable in O(1).
  5. When BFS ends, if the fresh count is still positive those oranges were unreachable, so return -1; otherwise return the number of levels elapsed.
  6. Handle the boundary case: if there were zero fresh oranges to begin with, the answer is 0 minutes regardless of rotten ones.
Why This WorksBFS visits cells in non-decreasing distance from the nearest source. Because all rotten oranges sit at distance 0 simultaneously, the level (minute) at which a fresh orange is first reached equals the earliest minute it could possibly rot. The last level processed is therefore the minimum total time, and any fresh cell never dequeued is provably unreachable from every source.
Watch Out For
  • Off-by-one on the minute counter: an empty queue right after seeding means 0 minutes, not 1.
  • Grids that start with zero fresh oranges — the answer is 0 even when no rotten ones exist.
  • Mixing per-cell expansion with per-minute levels: you must snapshot the whole frontier before incrementing the clock.
  • Counting diagonal cells as neighbors — only the 4 orthogonal directions spread rot.

Why optimize? Re-running a full O(mn) flood from every one of the mn cells is O((mn)²), far too slow on larger grids.

Key InsightBrute force asks, for every cell, “can water flow downhill from here to each ocean?” — re-running a search from all mn cells. Reverse the flow: start from the ocean borders and climb to neighbors whose height is ≥ the current one. Two flood fills (one per ocean) then intersect, visiting each cell only a constant number of times.
How to Think About It
  1. Notice the expensive part of brute force is one full traversal per source cell — flip the question to compute reachability once per ocean instead.
  2. Reverse the relation: instead of “flows down to,” traverse “is flowed-into-from,” stepping to neighbors whose height is ≥ the current cell’s.
  3. Seed one flood from every Pacific border cell (entire top row plus left column) and mark all cells it can climb to.
  4. Seed a second flood from every Atlantic border cell (bottom row plus right column) and mark its reachable set separately.
  5. A cell present in both reachable sets can drain to both oceans — collect those coordinates.
  6. Either DFS or BFS works for each flood; with a visited grid, each cell is processed once.
Why This WorksWater flows from a cell to an ocean exactly when a non-increasing-height path to that ocean exists. Reversing every edge, the ocean can reach the cell along a non-decreasing path. So the set of cells that drain into an ocean is precisely the set reachable by climbing outward from that ocean’s border. Intersecting the two climbed sets yields exactly the cells draining to both.
Watch Out For
  • Inverting the comparison wrong: from the border you move to neighbors with height ≥ current (climbing) — the opposite of the forward ≤ rule.
  • Sharing one visited grid across both oceans — keep them separate or you lose cells reachable by only one.
  • Forgetting that the four corner border cells belong to a seed set (top/left or bottom/right).
  • Equal-height neighbors are passable both ways; using strict inequality silently drops valid cells.
  • Connection: like #130 Surrounded Regions, the trick is flooding inward from the grid’s borders rather than testing each interior cell.

Key InsightBrute force checks, for every 'O', whether its region reaches a border — re-exploring the same regions repeatedly. How to Think About It
  1. Realize the only 'O's that survive are those connected to a border 'O' — so identify survivors directly instead of testing each region from scratch.
  2. Walk the four borders; from every border 'O', flood-fill its connected region and stamp each cell with a temporary sentinel such as '#'.
  3. Once all border floods finish, any cell still holding 'O' is genuinely surrounded.
  4. Make one final sweep: turn each remaining 'O' into 'X' (captured) and turn each sentinel '#' back into 'O' (restored survivor).
  5. Use DFS or BFS for the flood; the in-place sentinel doubles as the visited marker, so no separate visited structure is needed.
Why This WorksAn 'O' is captured if and only if it cannot reach the border through other 'O's. The border flood marks exactly the complement — every 'O' that is border-connected. So after marking, the unmarked 'O's are precisely the surrounded ones, making the flip correct; restoring sentinels returns the survivors to 'O'.
Watch Out For
  • Flipping survivors mid-flight: mutating before all border floods complete can sever a region partway through traversal.
  • Forgetting to restore the sentinel '#' back to 'O' in the final pass.
  • Single-row or single-column boards where every cell is a border — nothing is ever captured.
  • Connection: like #417 Pacific Atlantic Water Flow, this is a flood-fill seeded from the grid borders; spotting that shared pattern is the whole shortcut.

The ProblemGiven two words beginWord and endWord plus a dictionary wordList, return the number of words in the shortest transformation sequence from beginWord to endWord, where each adjacent pair differs by exactly one letter and every intermediate word is in wordList. Brute Force — O(N²·L) time, O(N) space

Treat words as graph nodes and run BFS from beginWord tracking depth. To find a word’s neighbors, scan every word in the dictionary and keep those differing by exactly one character. The first time endWord is dequeued, return its depth.

from typing import List
from collections import deque

def ladderLength(beginWord: str, endWord: str, wordList: List[str]) -> int:
    words = set(wordList)
    if endWord not in words:
        return 0

    def differs_by_one(a: str, b: str) -> bool:
        diff = 0
        for x, y in zip(a, b):
            if x != y:
                diff += 1
                if diff > 1:
                    return False
        return diff == 1

    queue = deque([(beginWord, 1)])
    visited = {beginWord}
    while queue:
        word, steps = queue.popleft()
        if word == endWord:
            return steps
        for cand in list(words):
            if cand not in visited and differs_by_one(word, cand):
                visited.add(cand)
                queue.append((cand, steps + 1))
    return 0

Why optimize? Finding neighbors by comparing each word against all N dictionary words costs O(N) per pop, so the whole BFS is O(N²·L) — slow for large dictionaries.

Key InsightBrute force finds a word’s neighbors by comparing it against every other word (O(N) each, O(N²) overall). Precompute wildcard patterns instead: a word of length L belongs to L buckets, one per position replaced by '*'; two words are neighbors exactly when they share a bucket. Neighbor lookup becomes a hash lookup, so BFS finds the shortest ladder fast.
How to Think About It
  1. Model words as graph nodes with an edge between any two words differing by exactly one letter; the answer is the shortest path length (in nodes) from beginWord to endWord.
  2. Since every edge has weight 1, BFS from beginWord yields the shortest transformation directly.
  3. Avoid the O(N²) all-pairs comparison: for each position i, map the pattern word[:i] + '*' + word[i+1:] to the list of words matching it.
  4. During BFS, generate a word’s neighbors by forming its L wildcard patterns and reading off the words in those buckets.
  5. Mark words visited as you enqueue them so each is processed once; track depth so the first dequeue of endWord returns its answer.
  6. Return 0 if the queue drains without reaching endWord, and early-exit when endWord is not in the dictionary.
Why This WorksBFS on an unweighted graph yields shortest paths, and the wildcard buckets define exactly the same adjacency as “differs by one letter” — two words share a starred pattern iff they match everywhere except that one position. So BFS over bucket-derived neighbors explores the identical graph, just with O(L) neighbor generation instead of O(NL), and the first arrival at endWord is the minimum number of words.
Watch Out For
  • Off-by-one on the count: the sequence length counts words (nodes), not transformations (edges), so beginWord itself is step 1.
  • Returning a nonzero answer when endWord is absent from wordList — check for it up front.
  • Not marking words visited at enqueue time, which lets the same word re-enter the queue and revisit.
  • Assuming beginWord must be in the list — it need not be, but endWord must.

The ProblemGiven two strings s1 and s2, return true if s2 contains a permutation of s1 as a substring — i.e. some contiguous window of s2 is an anagram of s1. Why optimize? Sorting every one of the n − m + 1 windows costs O(m log m) each, so long inputs (n up to 10⁴) make the repeated sorting far too slow.

Key InsightBrute force re-sorts each window from scratch, ignoring that adjacent windows differ by only one character in and one out. An anagram is fully characterized by its 26-letter frequency count, so maintain a single rolling count array and slide it in O(1) per step instead of re-examining the whole window.
How to Think About It
  1. Reduce "is this window a permutation of s1?" to "does this window's 26-letter frequency vector equal s1's frequency vector?" — order no longer matters.
  2. Build a fixed count array of size 26 for s1, and a second count array for the first window of width m in s2.
  3. Slide the window one character at a time: increment the count for the entering character on the right, decrement the count for the leaving character on the left.
  4. Decide how to compare cheaply: either compare both arrays (26 work) or track a matches counter of how many of the 26 letters currently agree, updating it as counts change.
  5. Return true the moment the window's counts equal s1's counts; if the slide finishes with no match, return false.
Why This WorksTwo strings are anagrams if and only if they have identical per-letter frequencies. A fixed-width window guarantees the window and s1 have the same length, so equal frequency vectors mean one is a permutation of the other. Sliding maintains the exact multiset of the current window incrementally, so the invariant "count array reflects the current window" holds at every step, and every candidate window is checked.
Watch Out For
  • Handle len(s1) > len(s2) up front — otherwise the first window setup reads out of bounds.
  • Off-by-one when sliding: the entering index is i and the leaving index is i − m; keep them consistent.
  • Map letters with ch − 'a' into 0..25; mixing chars and indices corrupts the counts.
  • If you track a matches counter, update it both before and after each increment/decrement so a count passing through equality is detected exactly once.

Key InsightThis is the same fixed-window count pattern as #567 Permutation in String: only this time you record every matching start index instead of returning on the first one. How to Think About It
  1. Recognize the reduction from #567: a window is an anagram of p exactly when their 26-letter frequency vectors match.
  2. Build p's count array and the count array of the first window s[0:m].
  3. For each subsequent position, add the entering character on the right and remove the leaving character on the left, keeping the array in sync with the current window.

Why optimize? The doubly nested scan is O(n²), which is too slow when fruits can hold up to 10⁵ trees.

Key InsightUnlike #567/#438 the window is variable-width, not fixed. Brute force restarts the scan from every index, but a valid window never needs to shrink from the left more than necessary: keep a count map of the current window's fruit types, and whenever it holds 3 distinct types, advance the left edge — dropping fruits — until only 2 remain. Each index enters and leaves the window once.
How to Think About It
  1. Reframe the story as: find the longest subarray with at most 2 distinct values.
  2. Maintain a left pointer and a hash map from fruit type to its count inside the current window.
  3. Advance the right pointer one tree at a time, incrementing the entering fruit's count in the map.
  4. While the map has more than 2 keys, decrement the count at the left pointer and advance left; erase a key when its count hits zero.
  5. After restoring the at-most-2 invariant, update the best answer with the current window width right − left + 1.
  6. Return the best width seen after the right pointer reaches the end.
Why This WorksThe left pointer only ever moves forward, so each element is added and removed at most once, giving linear total work. The shrink loop guarantees the window always satisfies "at most 2 distinct types" before the length is measured, so every measured width is feasible. Because the window expands maximally to the right for each left position and shrinks only the minimum required, no longer valid window is overlooked.
Watch Out For
  • Remove a fruit type from the map when its count drops to 0, or the distinct-count test stays wrong.
  • Use "while more than 2 distinct" (not "if") — introducing a third type can require dropping several trees of one type.
  • Measure the length only after shrinking back to a valid window, not during the violation.
  • A single-element or all-same array is valid; make sure the answer can be the full length.

// erase a key when its count reaches zero; size() = distinct count
if (cnt[f] == 0) cnt.erase(f);
Brute: O(n²)→Optimal: O(n)

#136 Single Number EasyBit ManipulationArray

The ProblemGiven a non-empty array nums in which every element appears exactly twice except for one element that appears once, find and return that single element. You must do it with linear runtime and, ideally, constant extra space. For example [4,1,2,1,2] returns 4.
Brute Force — O(n) time, O(n) space

Walk the array and tally how many times each value appears using a hash map (or set toggle). Then scan for the value whose tally is 1 and return it. It is linear in time but uses extra space proportional to the number of distinct elements, which the optimal solution avoids.

Python
def singleNumber(nums: list[int]) -> int:
    counts = {}
    for x in nums:
        counts[x] = counts.get(x, 0) + 1
    for x, c in counts.items():
        if c == 1:
            return x
    return -1

Why optimize? The hash map uses O(n) extra memory, but the problem explicitly asks for constant additional space.

Key InsightXOR is the perfect tool: x ^ x = 0 and x ^ 0 = x, and XOR is commutative and associative. So XOR-ing every element together cancels each duplicated pair to 0 and leaves exactly the unique value — no counting structure needed.
How to Think About It
  1. Recall the two XOR identities: a number XOR-ed with itself is 0, and any number XOR-ed with 0 is unchanged.
  2. Note XOR doesn't care about order, so the array can be combined in any sequence and the paired values still annihilate.
  3. Initialize an accumulator to 0.
  4. Fold the array with XOR: accumulate each element into the running value.
  5. After one pass, every value that appeared twice has cancelled to 0, so the accumulator holds the single number; return it.
Why This WorksBecause XOR is associative and commutative, you can mentally regroup the XOR of all elements so each duplicate pair sits together. Each pair XORs to 0, and 0 is the identity for XOR, so all pairs vanish and only the lone element survives in the accumulator — independent of the order the array was traversed.
Watch Out For
  • Start the accumulator at 0, the XOR identity — not at nums[0] unless you then skip index 0.
  • This trick only works because every other element appears an even number of times (exactly twice); it breaks if duplicates appear an odd count.
  • Negative numbers are fine — XOR operates on the two's-complement bit pattern.
  • Don't confuse XOR (^) with exponent or logical operators in languages where ^ differs.

#190 Reverse Bits EasyBit ManipulationDivide and Conquer

The ProblemReverse the bits of a given 32-bit unsigned integer and return the result. The input is treated as a fixed-width 32-bit value, so bit 0 (least significant) of the input becomes bit 31 of the output, bit 1 becomes bit 30, and so on. For example, input 0b00000010100101000001111010011100 returns 0b00111001011110000010100101000000.
Brute Force — O(32) time, O(32) space

Build the 32-character binary string of the number (zero-padded to width 32), reverse that string, then parse it back as an unsigned integer. It is conceptually simple but allocates a string and relies on string parsing rather than direct bit math.

Python
def reverseBits(n: int) -> int:
    bits = format(n, "032b")     # 32-char binary string
    return int(bits[::-1], 2)    # reverse, parse back
Go
import (
    "fmt"
    "strconv"
)

func reverseBits(num uint32) uint32 {
    s := fmt.Sprintf("%032b", num)
    r := []byte(s)
    for i, j := 0, len(r)-1; i < j; i, j = i+1, j-1 {
        r[i], r[j] = r[j], r[i]
    }
    v, _ := strconv.ParseUint(string(r), 2, 32)
    return uint32(v)
}
TypeScript
function reverseBits(n: number): number {
  // pad to 32 bits, reverse the string, parse back as unsigned
  const bits = (n >>> 0).toString(2).padStart(32, "0");
  const reversed = bits.split("").reverse().join("");
  return parseInt(reversed, 2) >>> 0;
}
Rust

        u32::from_str_radix(&rev, 2).unwrap()
    }
}

Why optimize? The string conversion allocates memory and is slow relative to direct bit shifting; if the function is called millions of times (a follow-up in the problem), the overhead matters.

Key InsightYou can reverse bits with pure arithmetic: pull off the input's least-significant bit, shift it into the most-significant side of the result, and repeat 32 times. Each step does result = (result << 1) | (n & 1) then n >>= 1 — no strings, just shifts and masks over a fixed 32-bit width.
How to Think About It
  1. Start a result accumulator at 0; you will build it bit by bit across exactly 32 iterations.
  2. Each iteration, isolate the current lowest bit of n with n & 1.
  3. Make room in the result by shifting it left by one, then OR in that isolated bit so it lands at the result's low end.
  4. Shift n right by one (logical/unsigned shift) so the next input bit becomes the lowest.
  5. After 32 rounds, the first input bit you grabbed has been pushed up to position 31 and the last sits at position 0 — the full reversal; return the result.
Why This WorksOn iteration k (0-indexed) you read input bit k and then perform 32 − 1 − k further left shifts of the result, so that bit ends up at position 31 − k — exactly the reversed position. Doing this for all 32 bits places every input bit at its mirrored output index, and the fixed loop count guarantees a full-width 32-bit reversal regardless of leading zeros.
Watch Out For
  • Always loop exactly 32 times; stopping early when n becomes 0 drops the high-order zero bits and misplaces the rest.
  • In TypeScript/JavaScript use the unsigned right shift >>> and finish with >>> 0, since bitwise ops otherwise treat numbers as signed 32-bit and can yield a negative result.
  • Use unsigned types (uint32, u32, uint32_t) so the left shift doesn't trip signed-overflow undefined behavior.
  • Mask with n & 1 for the low bit; don't accidentally grab a different bit position.

#338 Counting Bits EasyBit ManipulationDynamic Programming

The ProblemGiven an integer n, return an array ans of length n + 1 where ans[i] is the number of set bits (1s in the binary representation, the popcount) of i, for every i from 0 to n. For example, n = 5 yields [0,1,1,2,1,2]. Aim for a single linear pass.
Brute Force — O(n log n) time, O(n) space (output)

For each value i from 0 to n, independently count its set bits by repeatedly testing the lowest bit and shifting right until the number reaches 0. Store each count in the output array. Counting every number from scratch costs up to log i work per element.

Python
def countBits(n: int) -> list[int]:
    ans = []
    for i in range(n + 1):
        count, x = 0, i
        while x:
            count += x & 1
            x >>= 1
        ans.append(count)
    return ans

Why optimize? Recounting bits for every number repeats work already implied by smaller numbers; the log factor is avoidable given the array is built in increasing order.

Key InsightThe popcount of i is one plus the popcount of a strictly smaller, already-computed number, so the answer is a DP that reuses earlier entries. Two classic recurrences: ans[i] = ans[i >> 1] + (i & 1) (drop the lowest bit and add it back), or ans[i] = ans[i & (i − 1)] + 1 (clear the lowest set bit, which removes exactly one 1).
How to Think About It
  1. Observe that every number relates to a smaller number whose bit count you already know — this signals dynamic programming over 0..n.
  2. Pick a recurrence: i >> 1 is i with its last bit removed, so its popcount plus the dropped bit (i & 1) equals popcount of i.
  3. Alternatively, i & (i − 1) clears the lowest set bit of i, so popcount of i is that subresult plus 1.
  4. Allocate ans of length n + 1 with ans[0] = 0 as the base case.
  5. Fill ans left to right so each referenced subproblem (i >> 1 or i & (i−1), both < i) is already computed when you reach i.
  6. Return the completed array.
Why This WorksRight-shifting i by one discards exactly its least-significant bit, so i has the same set bits as i >> 1 except possibly that one bit, captured precisely by i & 1. Likewise i & (i − 1) turns off only the lowest 1-bit, removing exactly one from the count. Both referenced indices are strictly less than i, so a left-to-right fill always reads finished values, making the recurrence well-founded and correct.
Watch Out For
  • Size the array n + 1, not n — the range is inclusive of n.
  • Set the base case ans[0] = 0 before the loop and start filling from i = 1.
  • i & (i − 1) for i = 0 would underflow conceptually; the base case sidesteps it, so don't apply the recurrence at 0.
  • Don't confuse i >> 1 (halve / drop low bit) with i & 1 (value of the low bit) — the recurrence needs both.
