#594 Longest Harmonious Subsequence EasyHash MapSliding Window~2 days since review · rep 1 → 2, interval → 6

The ProblemA harmonious array is one where the difference between its maximum and minimum value is exactly 1. Given an integer array nums, return the length of the longest harmonious subsequence (elements need not be contiguous; order is irrelevant).
Quick ReminderOnly two distinct values can appear in a harmonious subsequence: some v and v+1. Count occurrences of each value, then for every v that also has v+1 present, the candidate length is count[v] + count[v+1].
Key InsightBecause a subsequence can freely pick elements, order doesn't matter — only how many of each value you have. Tally frequencies with a hash map; the answer is the best count[v] + count[v+1] over all keys that have a neighbour one higher. A sorted two-pointer window works too, but the counting view is O(n) with no sort.
Try to recall: Why does a subsequence collapse to "pick all copies of v and v+1"? Can you write the Counter version in under 8 minutes?

class Solution(object):
    def findLHS(self, nums):
        nums.sort()
        n = 0
        l, r = 0, 0
        while r < len(nums):
            if nums[r] - nums[l] == 1:
                if r - l + 1 > n:
                    n = r - l + 1
                r += 1
            elif nums[r] - nums[l] > 1:
                l += 1
            else:
                r += 1
        return n

Quick ReminderTrack the furthest index reachable so far. If you ever stand on an index beyond that reach, you're stuck. Alternatively (your prior approach) sweep backward, shrinking a "goal" post toward index 0.
Key InsightYou never need to try individual jump combinations. Greedily maintain the maximum reachable index as you scan; as long as the current index is within reach, extend the frontier with i + nums[i]. If the frontier ever covers the last index, you win.

class Solution(object):

        q = s[::-1]

Key InsightThe versions are a sorted boolean sequence: good good … good bad bad … bad. You are searching for the boundary — the leftmost true. That is a textbook "find first index satisfying a predicate" binary search, O(log n).
How to Think About It
  1. Recognize the answer space [1, n] is monotonic: if v is bad, everything after it is bad too.
  2. Keep a window [lo, hi] that always contains the first bad version; start lo = 1, hi = n.
  3. Check the midpoint. If mid is bad, the answer is mid or earlier, so move hi = mid. If good, the answer is strictly after, so lo = mid + 1.
  4. Loop while lo < hi; the window shrinks by half each step.
  5. When lo == hi the window is a single element — that is the first bad version.
Why This WorksThe invariant "the first bad version lies within [lo, hi]" is preserved by every branch: a bad mid can never let us discard mid itself, while a good mid lets us safely discard mid and everything before it. Halving a range of size n reaches size 1 in ⌈log₂ n⌉ steps.
Watch Out For

  • Use hi = mid (not mid - 1) when mid is bad, or you may skip the true boundary.
  • Use the half-open convergence while lo < hi; mixing <= with hi = mid can loop forever.

Why optimize? With up to 10⁴ elements, O(n²) is 10⁸ operations. A monotonic stack answers every "next greater" query in amortized O(1), giving O(n) overall.

Key InsightKeep a stack of indices whose next-greater is still unknown, kept in decreasing value order. When a larger value arrives, it resolves everything smaller on top of the stack. To handle circularity, iterate 2n times using i % n so each element gets a second pass around the wrap.
How to Think About It
  1. Initialize the answer to all -1 and an empty stack of indices.
  2. Walk i from 0 to 2n - 1; use cur = nums[i % n] as the real element (the modulo simulates wrapping).
  3. While the stack is non-empty and cur > nums[stack top], pop that index and set its answer to cur — cur is its next greater.
  4. Push the current index i % n only during the first pass (i < n), so no index is added twice.
  5. After the loop, indices still on the stack never found a greater element and keep their -1.
Why This WorksThe stack stays strictly decreasing, so the first element that pops a given index is its nearest greater neighbor to the right. Running 2n passes lets elements near the end "see" larger values that wrap to the front. Each index is pushed and popped at most once, so total work is O(n).
Watch Out For
  • Loop 2n times, not n — a single pass misses wrap-around answers.
  • Push indices only while i < n; pushing on the second pass would duplicate work and corrupt results.
  • Store indices, not values, so you can both write into the answer array and compare against nums[i % n].
  • Use strict >: equal values are not "greater," so duplicates must not resolve each other.

                string t = cur; // strip leading zeros, compare numerically
                if (best.empty() || less(t, best)) best = t;
            }
            return;
        }
        if (left > 0) rec(i + 1, left - 1, cur);   // remove
        rec(i + 1, left, cur + num[i]);            // keep
    };
    rec(0, k, "");
    return best.empty() ? "0" : best;
}

Why optimize? With num up to 10⁵ digits, enumerating subsets is astronomically large. A single greedy pass with a monotonic stack gives the minimum in O(n).

Key InsightTo make a fixed-length number small, eliminate a "peak" — a digit larger than the one right after it — as early as possible, because the leftmost high digit dominates the value. A monotonic increasing stack does exactly this: pop a larger top whenever a smaller digit arrives, spending your k budget on the most valuable removals.
How to Think About It
  1. Scan digits left to right, maintaining a stack of kept digits.
  2. Before pushing digit d, while k > 0 and the stack top is greater than d, pop it and decrement k — removing that larger high-order digit shrinks the number most.
  3. Push d. This keeps the stack non-decreasing.
  4. If k is still positive after the scan (digits were non-decreasing), drop k from the end.
  5. Strip leading zeros from the result; if it becomes empty, return "0".
Why This WorksRemoving a digit that is larger than its right neighbor always lowers the number, and doing it at the earliest (most significant) position yields the biggest drop — a greedy choice that is provably optimal. The monotonic stack finds each such removal in amortized O(1); leftover removals hit the smallest, least-significant tail digits.
Watch Out For
  • You must remove exactly k digits — if the string is already non-decreasing, trim the last k from the end.
  • Strip leading zeros at the end (e.g. "10200", k=1 → "200"), and return "0" for an empty result.
  • Stop popping once k reaches 0, even mid-string.
  • Edge case: k == len(num) removes everything → "0".

Why optimize? The hash set stores one entry per node, costing O(n) memory. Floyd's tortoise-and-hare lets you find both whether a cycle exists and where it starts using only two pointers — constant extra space.

Key InsightA slow pointer (one step) and a fast pointer (two steps) must meet inside a cycle. Once they meet, the distance from the head to the cycle start equals the distance from the meeting point to the cycle start — so a second walk pinpoints the entry.
How to Think About It
  1. Advance slow by one and fast by two until either fast falls off the end (no cycle) or the two pointers land on the same node.
  2. If they meet, a cycle exists; stop the first phase there.
  3. Reset one pointer back to the head, keep the other at the meeting point.
  4. Advance both one step at a time; the node where they meet again is the cycle's entrance.
  5. Return that node (or null if phase one exited via a null pointer).
Why This WorksLet the head-to-entry distance be a and the meeting point sit b into the loop of length c. Because fast travels twice as far, algebra gives a = c - b (mod c), meaning a pointer from the head and a pointer from the meeting point converge exactly at the entry after a steps.
Watch Out For
  • Always check fast and fast.next before dereferencing two steps, or you'll crash on acyclic lists.
  • Compare nodes by identity, not by value — duplicate values must not be mistaken for a meeting.
  • Don't forget the second phase; the meeting point itself is generally not the cycle start.

Key InsightUse a dummy node before the head so position 1 has no special case. Walk to the node just before left, then repeatedly pluck the node after it and splice it to the front of the reversed section — a "head-insertion" reversal.
How to Think About It

  4. Repeat right - left times: detach the node after cur and insert it immediately after prev.
  5. Return dummy.next, which correctly reflects any change to the head.

  • Keep cur fixed as the window tail; do not advance it during insertions or you'll miscount.
  • Reconnect the three boundary links (before-window, reversed-front, window-tail-to-after) exactly once.

Key InsightFuse the two traversals: one post-order DFS returns each node's height, and along the way it records leftHeight + rightHeight at every node into a shared "best" variable. Height and diameter are computed together in one pass.
How to Think About It
  1. Define a DFS that returns the height of the subtree rooted at a node.
  2. At each node, recursively get the left and right heights first (post-order).
  3. The path bending here has length leftHeight + rightHeight edges — update a running maximum with it.
  4. Return 1 + max(leftHeight, rightHeight) as this node's height to its parent.
  5. After the traversal, the running maximum holds the diameter.
Why This WorksEvery diameter path has a unique highest node where it bends.   • Diameter is counted in edges, not nodes — a path through k nodes has k - 1 edges, which is exactly leftHeight + rightHeight.
  • Return height to the parent but record the sum for the diameter — don't confuse the two values.
  • A single-node or empty tree has diameter 0.

The ProblemGiven the root of a binary tree, imagine standing on its right side. Return the values of the nodes you can see, ordered from top to bottom — that is, the rightmost node at each depth.

    if not root:
        return []

        levels.append([node.val for node in queue])

        for node in queue:
            if node.left:
                nxt.append(node.left)
            if node.right:
                nxt.append(node.right)
        queue = nxt
    return [level[-1] for level in levels]

Why optimize? Storing every node's value per level wastes memory — you only ever need one value per depth. A DFS that visits the right child first and records the first node it meets at each new depth needs only O(h) space for the recursion stack.

Key InsightThe right side view is simply the first node encountered at each depth when you traverse right-child-before-left-child. Track the current depth; whenever it exceeds the number of values collected so far, this is the first (rightmost) node at that depth.
How to Think About It
  1. DFS carrying a depth parameter, starting at 0 for the root.
  2. At each node, if depth equals the number of results collected so far, this node is the first seen at this depth — record its value.
  3. Recurse into the right child first, then the left child.
  4. Because right is visited first, the first node reached at any depth is the rightmost one.
  5. The result list, built in depth order, is the answer top to bottom.

A level-order BFS that keeps only the last node per level is an equally valid framing.
Watch Out For
  • Visit the right child before the left, or you'll record the leftmost node instead.
  • An empty tree returns an empty list — handle the null root.
  • Depths with only a left child still contribute: that left node is the rightmost thing at its depth.

Why optimize? Building and storing two full root-to-node paths costs O(n) time and space and throws away the tree's most useful property. In a BST, the ordering tells you which way to turn at every step, so you can find the split point by descending from the root with no extra storage.

Key InsightIn a BST, everything in the left subtree is smaller and everything in the right subtree is larger. The lowest common ancestor is the first node where p and q fall on opposite sides (or where one equals the current node) — the "split point" of the descent.
How to Think About It
  1. Start at the root as the current node.
  2. If both p and q are greater than the current value, both lie in the right subtree — move right.
  3. If both are smaller, move left.
  4. Otherwise they diverge here (one on each side, or one equals the current node) — this node is the LCA.
  5. Loop iteratively; no recursion or extra structure is required.
Why This WorksAs long as both targets sit on the same side, their common ancestor must be deeper in that direction, so descending is safe. The moment they land on different sides, the current node is the shallowest node whose subtree contains both — by definition the lowest common ancestor. Each step drops one level, so the walk costs O(h).
Watch Out For
  • Handle the case where the current node itself equals p or q — that node is the ancestor.
  • Compare against both targets each step; a single comparison isn't enough to decide direction.
  • This shortcut relies on the BST invariant — it does not work on an arbitrary binary tree.

Key InsightYou only need to visit each cell once. Walk the grid; when you hit unvisited land, flood-fill its entire island in one go, summing the cells as you sink them so they are never counted again. The area of each island is discovered exactly when you first reach it.
How to Think About It

  2. When the cell is land (1) and not yet visited, launch a flood fill (DFS or BFS) from it.

  4. Marking can reuse the grid itself — set visited land to 0 — avoiding a separate visited structure.
  5. Track the maximum returned area across all fills.
  6. Cells that were already sunk contribute 0, so the outer scan naturally skips them.
Why This WorksEach island is a connected component.   • An empty grid or a grid of all water — the answer is 0, not an error.
  • Off-by-one in bounds checks; test both the first and last row/column.
  • Recursive DFS can overflow the stack on a large single island — prefer an explicit stack or raise the recursion limit.
  • Only 4-directional connectivity counts; diagonals do not connect land.

Key InsightAn 'O' survives if and only if it connects to a border 'O'. Rather than searching outward from each interior cell, search inward from the edges: flood-fill from every border 'O', marking the whole safe region. Anything left unmarked is surrounded and must flip.
How to Think About It
  1. Walk the four borders and collect every 'O' on them as starting points.
  2. From each border 'O', flood-fill its region, temporarily marking every reached 'O' with a sentinel such as '#'.
  3. After the border sweep, do one final pass over the whole board.
  4. Turn every remaining 'O' (unmarked, hence surrounded) into 'X'.
  5. Restore every sentinel '#' back to 'O' — those were the safe cells.
Why This WorksConnectivity is symmetric: if an interior 'O' can reach the border, then the border can reach it. Marking from the border therefore captures exactly the safe set in one traversal. The complementary set — unmarked 'O's — is precisely the surrounded regions, so a single final sweep finishes the job.
Watch Out For
  • A single row or single column board — every 'O' is on the border and nothing flips.
  • Forgetting to restore the sentinel, leaving stray '#' characters on the board.
  • Recursive flood fill overflowing the stack on a large safe region; use an explicit stack/queue.
  • Modifying the board mid-traversal in a way that corrupts an in-progress search — use a distinct sentinel value.

Backtracking

Build a solution incrementally, prune invalid prefixes early, and undo each choice on the way back up. Palindrome Partitioning is a clean introduction to the choose/explore/undo template.

Blocked Practice

#131 Palindrome Partitioning MediumBacktracking

The ProblemGiven a string s, partition it so that every substring of the partition is a palindrome. Return all possible palindrome partitionings of s. For example "aab" yields [["a","a","b"],["aa","b"]].
Brute Force — O(2^n · n) time

There are n-1 gaps between the characters; each gap is either a cut or not, giving 2^(n-1) possible partitions. Enumerate every one via a bitmask, split the string accordingly, and keep it only if all resulting pieces are palindromes.

Python
def partition(s):
    n = len(s)
    result = []
    def is_pal(sub):
        return sub == sub[::-1]
    for mask in range(1 << (n - 1)):
        parts = []
        prev = 0
        for i in range(n - 1):
            if mask & (1 << i):
                parts.append(s[prev:i+1])
                prev = i + 1
        parts.append(s[prev:])
        if all(is_pal(p) for p in parts):
            result.append(parts)
    return result

Why optimize? The bitmask builds every full partition before checking it, wasting work on splits whose very first piece is not a palindrome. Building the partition incrementally and pruning as soon as a prefix fails avoids exploring huge dead branches.

Key InsightBuild partitions one piece at a time. At position start, try every prefix s[start..end]; only when that prefix is a palindrome do you commit it and recurse on the remainder. Non-palindrome prefixes are pruned instantly, so entire subtrees of impossible splits are never generated.
How to Think About It
  1. Define a recursion over a start index and a path of pieces chosen so far.
  2. When start reaches the end of the string, record a copy of path as a complete valid partition.
  3. Otherwise, for each end from start onward, test whether s[start..end] is a palindrome.
  4. If it is, push that substring onto path, recurse from end, then pop it (undo) before trying the next end.
  5. The palindrome check is a simple two-pointer scan from both ends of the candidate piece.
  6. Copy path when recording — it keeps mutating as the search continues.
Why This WorksEvery valid partition is a sequence of palindromic prefixes. By only ever extending with a palindromic prefix, the recursion enumerates exactly the valid partitions and nothing else. The undo step (backtracking) restores state so sibling choices are explored on a clean path, and pruning failed prefixes early keeps the search tree small.
Watch Out For
  • Appending the same list reference instead of a copy — all results end up empty or identical.
  • Forgetting the pop/undo after recursion, corrupting the path for later branches.
  • Single characters are palindromes, so a fully-split partition is always valid.
  • Inclusive vs exclusive substring bounds — an off-by-one drops the last character or duplicates one.

Heap

Priority queues dispatch the highest-value ready element; Task Scheduler can be solved with a max-heap or a direct greedy formula built from the busiest task's frequency.

Interleaved

#621 Task Scheduler MediumHeapGreedy

The ProblemGiven a list of CPU tasks (labeled A–Z) and an integer n, each task takes one unit of time and identical tasks must be separated by at least n units of cooldown. The CPU may sit idle. Return the minimum number of time units needed to finish all tasks.
Brute Force — O(T · k) time (T = total time, k = task types)

Simulate the clock tick by tick. At each tick, among tasks that still have work left and are off cooldown, run the one with the highest remaining count; if none is available, the CPU idles. Advance time until every task is done.

Python
def leastInterval(tasks, n):
    from collections import Counter
    counts = Counter(tasks)
    cooldown = {}          # task -> earliest time it may run again
    remaining = sum(counts.values())
    time = 0
    while remaining > 0:
        best = None
        for t, c in counts.items():
            if c > 0 and cooldown.get(t, 0) <= time:
                if best is None or c > counts[best]:
                    best = t
        if best is not None:
            counts[best] -= 1
            remaining -= 1
            cooldown[best] = time + n + 1
        time += 1
    return time

Why optimize? Ticking one unit at a time can loop over enormous idle stretches when n is large. Observing that the most frequent task dictates the schedule's skeleton lets you compute the answer directly, or drive it with a heap that only acts on real events.

Key InsightThe busiest task sets the rhythm. If the maximum frequency is maxFreq, it creates maxFreq - 1 mandatory gaps of width n + 1; every other task slots into those gaps or extends the final block. The answer is max(len(tasks), (maxFreq - 1) × (n + 1) + countOfMax).
How to Think About It
  1. Count how many times each task appears — an array of size 26 suffices.
  2. Find maxFreq, the largest of those counts.
  3. Count how many distinct tasks share that maximum frequency (countOfMax).
  4. Picture the most frequent task laid out with n slots after each occurrence: that frame has length (maxFreq - 1) × (n + 1), plus countOfMax for the final column.
  5. Other tasks fill the empty slots; if there are more tasks than slots, no idling is needed and the answer is simply the number of tasks.
  6. Return the larger of the framed length and len(tasks). (A max-heap that always dispatches the highest-count ready task yields the same schedule.)
Why This WorksSpacing the most frequent task optimally is forced — nothing can be tighter than its own cooldown requirement. That frame's idle slots are the only places idling could occur; filling them greedily with remaining tasks either consumes all idle time (when tasks are plentiful) or leaves the frame as the binding constraint. Taking the max of the two cases captures both regimes exactly.
Watch Out For
  • When n = 0, there is no cooldown and the answer is just len(tasks).
  • Ties at the maximum frequency — each such task adds one to the last column, so countOfMax matters.
  • Very large n with few tasks producing many idle slots — the framed formula still holds.
  • Off-by-one in (maxFreq - 1) vs maxFreq: the last group needs no trailing cooldown.

Dynamic Programming

You have a strong DP base (Fibonacci, Tribonacci, House Robber I, LPS). House Robber II adds a circular twist that decomposes into two linear DP passes.

Interleaved

#213 House Robber II MediumDynamic Programming

The ProblemHouses are arranged in a circle, so the first and last houses are adjacent. Each house holds nums[i] money, and robbing two adjacent houses triggers the alarm. Return the maximum amount you can rob without alerting the police.
Brute Force — O(n · 2^n) time

Enumerate every subset of houses via a bitmask. A subset is valid only if no two chosen houses are adjacent — and because the row is circular, house i and house (i+1) mod n are neighbours, which also forbids picking both the first and last house. Take the maximum sum over all valid subsets.

Python
def rob(nums):
    n = len(nums)
    if n == 1:
        return nums[0]
    best = 0
    for mask in range(1 << n):
        valid = True
        for i in range(n):
            if (mask >> i) & 1 and (mask >> ((i + 1) % n)) & 1:
                valid = False
                break
        if valid:
            total = sum(nums[i] for i in range(n) if (mask >> i) & 1)
            best = max(best, total)
    return best

Why optimize? Checking all 2^n subsets is hopeless past small inputs. The circular constraint reduces to two ordinary (linear) robbery problems, each solvable with a rolling O(1)-space scan — turning exponential work into two linear passes.

Key InsightThe only complication is that house 0 and house n-1 are adjacent, so they can never both be robbed. Split into two straight-line problems: rob houses [0 .. n-2] (excluding the last) and rob houses [1 .. n-1] (excluding the first). Solve each with the standard linear House Robber DP and take the larger result.
How to Think About It
  1. Handle the trivial case of a single house directly — it has no neighbour, so rob it.
  2. Define a helper that solves linear House Robber over a subrange: for each house keep the best total, choosing to skip it or take it plus the best from two houses back.
  3. Run the helper on the range that excludes the last house.
  4. Run the helper again on the range that excludes the first house.
  5. The circular adjacency is now impossible in either run, so return the maximum of the two totals.
  6. Each run needs only two rolling variables, so no array is required.
Why This WorksIn any valid circular selection, at least one of the first or last house is left un-robbed. The two linear cases exactly cover those possibilities: excluding the last frees the first, excluding the first frees the last. Every valid arrangement falls into (at least) one case, and each linear subproblem is optimally solved by the classic rob/skip recurrence, so the max of the two is globally optimal.
Watch Out For
  • The single-house case — slicing off both ends would leave an empty range and lose the answer.
  • Two houses: excluding one end each time correctly picks the richer single house.
  • Off-by-one when forming the two subranges; they must overlap on the interior but differ at the ends.
  • Reusing rolling variables across the two runs without resetting them.
