#739 Daily Temperatures

Medium stack monotonic-stack array

The Problem

Given an integer array temperatures, return an array answer where answer[i] is the number of days you have to wait after day i until a warmer temperature. If there is no future day for which this is possible, set answer[i] = 0.

Brute Force — O(n²) time

For every day i, scan forward through the remaining days until you find the first day with a strictly warmer temperature, then record the distance. If you reach the end without finding one, leave the answer at 0.

    n = len(temperatures)

        for j in range(i + 1, n):
            if temperatures[j] > temperatures[i]:
                answer[i] = j - i
                break
    return answer

Why optimize? With n up to 1e5, the nested scan is ~1e10 operations in the worst case (a strictly decreasing array forces every inner loop to run to the end). That times out. We need each element to do constant amortized work.

Brute force: O(n²) time, O(1) extra space (excluding output).
Key Insight: Keep a decreasing monotonic stack of indices for days still waiting for a warmer temperature. When a warmer temperature arrives, it resolves every pending colder day on top of the stack — each one popped exactly once. Since each index is pushed once and popped at most once, total work is O(n).

How to Think About It

  1. Notice the brute force re-scans the same future days over and over. Many of those days are colder than days you already looked at — wasted work.
  2. Maintain a stack of indices whose answer is not yet known. Invariant: their temperatures are strictly decreasing from bottom to top.
  3. Walk left to right. For current day i, while the stack is non-empty and temperatures[i] is warmer than the day at the stack top, that pending day is now resolved: pop it and set its answer to i - poppedIndex.
  4. After resolving, push i onto the stack as a new unresolved day.
  5. Any indices left on the stack at the end never found a warmer day — their answer stays 0 (already initialized).
  6. Convince yourself the stack stays decreasing: you only push after popping everything not strictly greater than the current day.
Why This Works: The stack holds exactly the days that are still "waiting" and, by construction, they are in decreasing-temperature order. The first warmer day to the right of any waiting day is precisely the day that pops it. Because a popped day is never reconsidered, each index is touched a constant number of times, giving O(n) total.
  • Store indices, not temperatures — you need the index to compute the day gap.
  • Use a strict comparison (>): equal temperatures are not "warmer", so they should not pop.
Watch Out: Forgetting that leftover stack entries must remain 0. If you initialize the answer array with anything other than 0, days that never warm up will be wrong — initialize to 0 and only overwrite on a pop.

The Problem

Why optimize? The whole point of this design problem is O(1) getMin. A linear scan defeats the requirement; with many getMin calls interleaved with pushes you get O(n) per call.

Brute force: O(1) push/pop/top, O(n) getMin.
Key Insight: Each stack entry can carry, alongside its value, the minimum of the entire stack at and below it. Then the minimum of the whole stack is simply the min field of the top entry — readable in O(1). When you pop, the new top already knows the correct min.

How to Think About It

  1. Realize the min only changes at push/pop boundaries, so it can be precomputed and stored, not recomputed on every query.
  2. Store pairs (value, minSoFar) on a single stack.
  3. On push of val: the new min is min(val, currentTopMin) (or just val if the stack is empty). Push (val, newMin).
  4. On pop: just remove the top pair — the entry beneath it still holds the correct running min, so nothing else needs recomputing.
  5. top() returns the top entry's value; getMin() returns the top entry's stored min.
  6. Alternative: keep two stacks — one of values, one of running mins (push to the min-stack the smaller of val and its current top).
Why This Works: The stored min at each level is an invariant: it is always the minimum of every element from the bottom up to that level. Pushing extends that prefix by one element; popping shrinks it back, and the level below already recorded the answer for the shorter prefix. No information is ever lost or stale.
  • Handle the empty stack: the first push's min is the value itself; calling getMin on an empty stack is undefined by the problem's constraints (operations are valid).
  • The min must be updated on every push, even when the value is larger than the current min (you still copy the old min forward).
Watch Out: Recomputing the min on pop. With the paired-min approach you never recompute — the entry below already carries the right min. If you try to "fix up" the min on pop by scanning, you are back to O(n).

Hard stack monotonic-stack array

The Problem

Given an array heights representing the heights of bars each of width 1, return the area of the largest rectangle that can be formed within the histogram.

Brute Force — O(n²) time

Treat each bar as the limiting height. Expand left and right while neighboring bars are at least as tall, giving the widest span for which this bar is the shortest. The area is height * width; track the maximum.

Pythondef largest_rectangle_area(heights):
    n = len(heights)
    best = 0
    for i in range(n):
        left = i
        while left > 0 and heights[left - 1] >= heights[i]:
            left -= 1
        right = i
        while right < n - 1 and heights[right + 1] >= heights[i]:
            right += 1
        best = max(best, heights[i] * (right - left + 1))
    return best

Why optimize? With n up to 1e5, expanding from every bar is O(n²) — up to 1e10 operations on a flat or staircase histogram. We need each bar's span computed in amortized O(1).

Brute force: O(n²) time, O(1) extra space.
Key Insight: Keep an increasing monotonic stack of indices. When a bar shorter than the stack top appears, that top bar can extend no further right — pop it and compute its rectangle: its height times the width spanning from just after the new stack top up to the current index. Each bar is pushed and popped exactly once, so resolving every bar's maximal width is O(n).

How to Think About It

  1. For a fixed bar to be the limiting height of a rectangle, you need its widest run where all bars are at least that tall. The brute force finds that by expanding; the stack finds it incrementally.
  2. Maintain a stack of indices with strictly increasing heights. While the current bar is shorter than the bar at the top, the top bar's right boundary is found.
  3. Pop the top index h. Its height is heights[h]. Its left boundary is the new stack top (or the array start); its right boundary is the current index i.
  4. Width after popping is i - stack[-1] - 1 (or i if the stack is now empty). Area is heights[h] * width; update the max.
  5. After popping everything taller-or-equal, push i.
  6. Append a sentinel bar of height 0 at the end (or iterate one past the array with height 0) so that every remaining bar gets flushed and measured.
Why This Works: When bar h is popped because bar i is shorter, you know two things exactly: the first shorter bar to its right is i, and the first shorter bar to its left is the element now under it on the stack. Those two boundaries define the maximal width over which heights[h] is the limiting height. Every bar is popped once, with the correct left/right boundaries, so every candidate rectangle (one per "limiting bar") is considered — in O(n) total.
  • Push a sentinel 0 at the end so no bar is left unresolved on the stack.
  • When the stack becomes empty after a pop, the width is the full prefix i (the popped bar extended all the way to the left edge).
  • Store indices, not heights — you need indices to compute widths.
Watch Out: The width formula. After popping index h, the width is i - stack[-1] - 1 (using the new top), NOT i - h. Mixing these up is the single most common bug and silently undercounts wide rectangles.

#226 Invert Binary Tree

Easy tree dfs bfs

The Problem

Given the root of a binary tree, invert the tree — swap the left and right child of every node — and return its root.

Baseline — iterative BFS with a queue, O(n) time

The recursive solution is so clean it would be the "optimal" answer, so here is the iterative BFS baseline: walk the tree level by level with a queue, swapping each node's children as you visit it. Both approaches are O(n); recursion is just cleaner and shorter.

Pythonfrom collections import deque

def invert_tree(root):
    if not root:
        return None
    queue = deque([root])
    while queue:
        node = queue.popleft()
        node.left, node.right = node.right, node.left  # swap
        if node.left:
            queue.append(node.left)
        if node.right:
            queue.append(node.right)
    return root

Key Insight: Recursively swap the left and right subtrees at every node, with the base case being a null node. Whether you swap before or after recursing into the children does not matter — every node still ends up with its children flipped.

How to Think About It

  1. State the base case first: an empty (null) tree inverts to an empty tree — return null.
  2. Otherwise, swap the node's left and right child pointers.
  3. Recurse into both children to invert each subtree.
  4. Return the (unchanged) node reference so the caller can wire it back in.
  5. Trust the recursion: assume the recursive calls correctly invert the subtrees, and the local swap plus those guarantees the whole tree is inverted.
Why This Works: Inverting a tree means inverting both subtrees and then swapping them. The recursion handles "invert both subtrees" by self-similarity, and the swap handles the rest. The null base case terminates every path, and every node is visited exactly once, giving O(n).
  • Always return the node — forgetting the return value breaks the wiring in languages where the recursive call's result is reassigned.
  • The null base case is essential: without it you dereference a null child and crash.
Watch Out: Recursing first and then swapping the results incorrectly, versus swapping the pointers. Either order works, but you must swap the actual child links, not just local copies that get discarded.

Why optimize? Collecting and sorting is O(n log n) and always visits all n nodes. Because an inorder traversal of a BST is already sorted, you can stop after k visits: O(H + k), where H is the tree height.

Brute force: O(n log n) time, O(n) space.
Key Insight: An inorder traversal (left, node, right) of a BST visits node values in strictly ascending order. So the k-th node you visit inorder is the k-th smallest — no sorting needed, and you can stop the moment you reach it.

How to Think About It

  1. Recall the BST invariant: everything in the left subtree is smaller, everything in the right is larger. Inorder therefore emits values smallest-first.
  2. Do an inorder traversal, but keep a counter of how many values you have visited.
  3. When the counter reaches k, you are at the answer — record it and stop descending.
  4. Early termination matters: once found, you do not need to visit the rest of the tree.
  5. This can be done recursively (with a shared counter/result) or iteratively with an explicit stack that simulates inorder.
  6. Iterative version: push left spine, pop a node (that is the next-smallest), decrement k; if zero return it; else move to its right child and repeat.
Why This Works: Inorder traversal of a BST is exactly a sorted enumeration of its values. Counting visited nodes lets you index into that sorted sequence directly. Because you stop at the k-th, you touch at most the left spine (height H) plus k nodes, giving O(H + k).
  • k is 1-indexed — the 1st smallest, not the 0th.
  • Stop early once the k-th is found; continuing wastes time and risks overwriting the answer.
  • The iterative stack version avoids recursion-depth limits on skewed trees.
Watch Out: Off-by-one on the counter. Decrement-then-check vs. check-then-decrement must be consistent: a clean pattern is to decrement k when visiting a node and return that node's value when k hits 0.

The Problem

Brute Force — linear scan for each root, O(n²)

The first element of the current preorder segment is the subtree root. Linear-scan the current inorder segment to find that root's index, which splits inorder into left and right subtrees. Recurse on the corresponding segments. The repeated scan is what costs O(n²).

Brute force: O(n²) time (worse with slicing), O(n) space.
Key Insight: preorder[0] is always the current subtree's root. Its position in inorder splits the inorder array into the left subtree (everything before it) and the right subtree (everything after). A precomputed value → inorder-index hashmap turns that split into an O(1) lookup.

How to Think About It

  1. Preorder visits root, then the entire left subtree, then the entire right subtree. So the first unused preorder value is the next root.
  2. Inorder visits the entire left subtree, then root, then the entire right subtree. Finding the root's index in inorder tells you the sizes of both subtrees.
  3. Build a hashmap from value to its index in inorder once, up front, so each split is O(1).
  4. Recurse with index ranges rather than slicing arrays: track the current inorder [lo, hi] window and a single moving pointer into preorder.
  5. Advance the preorder pointer as you consume roots; build the left subtree before the right (matching preorder's order).
  6. Base case: an empty inorder window (lo > hi) returns null.
Why This Works: The two traversal orders together pin down the tree uniquely (values are distinct). The hashmap removes the only repeated work — locating the root — so every node is constructed in O(1) amortized, O(n) total.
  • The left subtree's preorder pointer must advance past all left-subtree nodes before building the right — building left first handles this naturally.
  • Use index ranges, not array slices, to keep it O(n) and avoid off-by-one errors in subarray bounds.
Watch Out: Off-by-one on the subarray bounds. The left inorder window is [lo, mid-1] and the right is [mid+1, hi] — including mid (the root) in either half corrupts the recursion. A shared, monotonically advancing preorder pointer is the cleanest way to avoid index bugs.

Medium tree dfs

The Problem

Given the root of a binary tree and two nodes p and q (both guaranteed to exist in the tree), return their lowest common ancestor — the deepest node that has both p and q as descendants (a node can be a descendant of itself).

Brute Force — find both paths, compare, O(n) time + O(n) space

Find the root-to-p path and the root-to-q path as lists of nodes. Walk both lists in parallel; the last node where they agree is the lowest common ancestor.

Pythondef lowest_common_ancestor(root, p, q):
    def path(node, target, acc):
        if not node:
            return False
        acc.append(node)
        if node is target:
            return True
        if path(node.left, target, acc) or path(node.right, target, acc):

        acc.pop()
        return False

    path(root, q, qp)
    lca = None
    for a, b in zip(pp, qp):
        if a is b:
            lca = a
        else:
            break
    return lca

Brute force: O(n) time, O(n) space (stored paths).
Key Insight: A single DFS that returns a node when it is p or q, and otherwise recurses into both children. If a node receives non-null results from both sides, p and q lie in different subtrees, so that node is their LCA. If only one side is non-null, propagate it upward.

How to Think About It

  1. Define the recursion's return value: "the LCA found within this subtree, or p/q if exactly one was found here, or null if neither is present."
  2. Base case: a null node returns null; a node equal to p or q returns itself.
  3. Recurse into left and right children, capturing their returned nodes.
  4. If both children return non-null, the current node is the split point — it is the LCA, so return it.
  5. Otherwise return whichever child result is non-null (or null if both are). This bubbles the found node up toward the root.
  6. The first node where the two searches meet is the lowest common ancestor.
Why This Works: Each call answers "what relevant node lives below me?" If p and q are found in different child subtrees, the current node is the deepest node that contains both — the LCA. If they are on the same side, the LCA is deeper, and returning the single non-null result lets the ancestor that splits them detect the meeting point. Because both nodes are guaranteed present, the recursion is guaranteed to find them.
  • The solution assumes both p and q exist in the tree (as the problem guarantees). Without that guarantee you would need to confirm both were actually found.
  • Null propagation is the engine: returning the lone non-null child result is what carries the eventual answer up.
Watch Out: Returning early or returning a child result before checking both children. You must evaluate left and right, then decide: both non-null means current node is the LCA; do not short-circuit before recursing into both sides.

How to Think About It

  2. For Kahn's: compute in-degree of every node, push all zero-in-degree nodes onto a queue.
  3. Pop a node, append it to the topo order, and decrement each neighbor's in-degree; push any that hit zero.
  4. If the number of processed nodes equals numCourses, every course is reachable in a valid order → no cycle.
  5. For DFS instead: mark a node in-progress before recursing; if you ever reach an in-progress node, return false (cycle); mark done when its subtree finishes.
Why This Works: A directed graph has a valid topological order iff it is acyclic. Kahn's peels off nodes with no remaining dependencies; a cycle's nodes can never reach in-degree zero, so they remain unprocessed and the count falls short. DFS 3-coloring catches a cycle the moment it follows an edge back into the current recursion stack.
Watch Out: The pair direction is easy to flip — [a, b] means b → a (b before a). If you build edges backwards, your topo sort and in-degrees are inverted and tests fail subtly.
  • A 2-state visited set (only visited/unvisited) can't distinguish a cross-edge from a back-edge; you need the third "in-progress" state for DFS cycle detection.
  • Disconnected components: seed the queue with all zero-in-degree nodes, not just node 0.

#133 Clone Graph

Medium graph BFS DFS hashmap

The Problem

Given a reference to a node in a connected, undirected graph, return a deep copy (clone): a fresh set of nodes with the same values and the same neighbor structure, sharing nothing with the original.

Brute Force — O(N+E), and the trap

There's no meaningfully slower approach here. The natural "naive" version — DFS that clones a node then recurses into each neighbor without tracking what's already cloned — loops forever on any cycle (and undirected graphs are full of 2-cycles). The fix is the standard solution itself: memoize original → clone. The baseline below already includes the hashmap, framed as the straightforward approach.

Python# BASELINE: DFS with original->clone map (the memo is mandatory)
def cloneGraph(node):
    if not node: return None
    seen = {}
    def dfs(cur):
        if cur in seen: return seen[cur]
        copy = Node(cur.val)
        seen[cur] = copy            # record BEFORE recursing (handles cycles)
        for nb in cur.neighbors:
            copy.neighbors.append(dfs(nb))
        return copy
    return dfs(node)

How to Think About It

  1. Handle the empty case: a null input returns null.

#743 Network Delay Time

Medium graph Dijkstra shortest-path heap

The Problem

Given times (directed weighted edges [u, v, w]), n nodes labeled 1..n, and a source k, return the minimum time for a signal starting at k to reach all nodes, or -1 if some node is unreachable.

Brute Force — O(V*E) time (Bellman-Ford)

Bellman-Ford: initialize all distances to infinity except the source, then relax every edge V-1 times. After that pass, the farthest finalized distance is the answer (or -1 if any node is still infinity).

Python# BRUTE: Bellman-Ford, relax all edges n-1 times
def networkDelayTime(times, n, k):
    INF = float('inf')
    dist = [INF] * (n + 1)
    dist[k] = 0
    for _ in range(n - 1):
        for u, v, w in times:
            if dist[u] + w < dist[v]:
                dist[v] = dist[u] + w
    ans = max(dist[1:])
    return ans if ans < INF else -1

Bellman-Ford: O(V*E) time, O(V) space.
Key Insight: This is single-source shortest paths with non-negative weights — Dijkstra. A min-heap always pops the closest unsettled node; once popped, its distance is final. The answer is the maximum finalized distance (or -1 if any node was never reached).

How to Think About It

  1. Build an adjacency list: adj[u] = list of (v, w).
  2. Push (0, k) onto a min-heap keyed by distance; track which nodes are finalized.
  3. Pop the smallest (d, node). If already finalized, skip (stale entry). Otherwise finalize it.
  4. For each neighbor, push (d + w, neighbor) if it could improve that neighbor's distance.
  5. After the heap empties, if all n nodes are finalized, return the largest finalized distance; else -1.
Why This Works: With non-negative weights, the first time a node is popped from the min-heap its distance is provably minimal — any later path to it goes through a node already at least as far, so it can't be shorter. The signal reaches everyone exactly when the slowest finalized node is reached, hence the max.
Watch Out: The heap accumulates stale entries (a node can be pushed several times with different distances). Skip any popped entry whose node is already finalized, or you'll re-process and corrupt results.
  • Return -1 if fewer than n nodes are finalized — don't assume connectivity.
  • Nodes are 1-indexed; size arrays as n+1 or remap.
  • Dijkstra requires non-negative weights; negative edges need Bellman-Ford.

#684 Redundant Connection

Medium union-find cycle-detection graph

The Problem

A graph started as a tree with n nodes (labeled 1..n) and had exactly one extra edge added. Given edges in the order they were added, return the edge that can be removed so the result is a tree. If several qualify, return the one that appears last in the input.

Brute Force — O(n²) connectivity check per edge

For each edge, before adding it, run a DFS/BFS in the graph built so far to test whether its two endpoints are already connected. The first edge whose endpoints are already connected is redundant. Each check is O(n), giving O(n²).

Python# BRUTE: DFS-check connectivity before adding each edge
def findRedundantConnection(edges):
    from collections import defaultdict
    adj = defaultdict(set)
    def connected(a, b, seen):
        if a == b: return True
        seen.add(a)
        return any(connected(nb, b, seen) for nb in adj[a] if nb not in seen)
    for u, v in edges:
        if connected(u, v, set()):
            return [u, v]
        adj[u].add(v); adj[v].add(u)
    return []

Key Insight: Process edges in order with Union-Find. The first edge whose two endpoints already share a root closes a cycle — that's the redundant edge. Because we scan in input order, "first found" is automatically the last-added valid answer.

How to Think About It

  1. Initialize parent[i] = i for nodes 1..n (mind 1-indexing — size n+1).
  2. For each edge [u, v], call find(u) and find(v).
  3. If the roots are equal, u and v are already connected — adding this edge forms a cycle, so return it.
  4. Otherwise union them (link one root under the other, by rank) and continue.
  5. The tree-plus-one-edge guarantee means exactly one edge triggers the equal-roots case.
Why This Works: Adding an edge between two already-connected nodes is precisely what creates a cycle. A tree with n nodes has n-1 edges; the one extra edge is the unique one that connects two nodes already in the same DSU set. Scanning in order returns the latest such edge automatically.
Watch Out: Nodes are 1-indexed — allocate parent of size n+1 (or remap), or you'll index out of bounds / mis-root node 0.
  • Make union return false when the two are already joined; that boolean is your cycle signal.
  • Use path compression so repeated finds stay cheap on long chains.

Why optimize? The hashmap costs O(n) space. XOR folds the whole array into a single accumulator for O(1) space — strictly better.

Hashmap: O(n) time, O(n) space.
Key Insight: XOR every element together. Since a ^ a = 0 and a ^ 0 = a, all paired numbers cancel out and only the unique element survives the fold.

How to Think About It

  1. Start an accumulator at 0.
  2. XOR each array element into it.
  3. Paired duplicates cancel (a ^ a = 0) regardless of order.
  4. The lone element XORs against 0 and remains.
  5. Return the accumulator.
Why This Works: XOR is associative and commutative, so you can reorder the array as (a^a)(b^b)...(unique). Every pair becomes 0, and 0 ^ unique = unique. No matter how the duplicates are interleaved, they annihilate.
Watch Out: This trick relies on every other element appearing exactly twice. If elements can appear three times (#137) or there are two unique elements (#260), plain XOR doesn't isolate the answer — you need bit-counting or a split-by-bit approach.
  • Initialize the accumulator to 0, not nums[0], to keep the loop uniform (or start at nums[0] and skip index 0 consistently).

#338 Counting Bits

Easy bit-manipulation DP

The Problem

Given an integer n, return an array ans of length n+1 where ans[i] is the number of set bits (1s) in the binary representation of i, for every i from 0 to n.

Brute Force — O(n log n) time

For each i, count its bits independently by repeatedly testing the low bit and right-shifting (or call a popcount). Each number takes O(log i) work, so the whole array is O(n log n).

Python# BRUTE: count bits per number independently
def countBits(n):
    ans = []
    for i in range(n + 1):
        c = 0
        x = i
        while x:
            c += x & 1
            x >>= 1
        ans.append(c)
    return ans

Why optimize? Counting each number from scratch repeats work. A DP recurrence reuses an already-computed smaller answer, dropping each entry to O(1) for O(n) total.

Per-number: O(n log n) time, O(1) extra (besides output).
Key Insight: ans[i] = ans[i >> 1] + (i & 1). Number i has the same bits as i/2 (drop the lowest bit) plus its own lowest bit. Alternatively ans[i] = ans[i & (i-1)] + 1 (clear the lowest set bit).

How to Think About It

  1. Allocate ans of length n+1 with ans[0] = 0.
  2. For i from 1 to n, compute ans[i] = ans[i >> 1] + (i & 1).
  3. Since i >> 1 < i, that subproblem is already filled — build left to right.
  4. The i & 1 term adds the bit shifting lost.
  5. Alternative recurrence ans[i & (i-1)] + 1 also references a smaller, ready index.
Why This Works: Right-shifting i by one bit divides by two and discards the least significant bit. So i's popcount equals (i/2)'s popcount plus whether that discarded bit was set. The recurrence always depends on a strictly smaller index, making a single left-to-right pass correct.
Watch Out: Fill the array left to right so ans[i >> 1] (and ans[i & (i-1)]) is already computed when you read it. Iterating out of order reads uninitialized/garbage entries.
  • Don't forget index 0 has zero set bits — the base case.
  • The output array itself isn't counted as extra space, but avoid an O(log n) per-element popcount if O(n) is required.

The Problem

  • Handle the empty trie / no children gracefully so DFS doesn't dereference null.

#146 LRU Cache

Medium design hashmap linked-list

The Problem

Design an LRU (Least Recently Used) cache with a fixed capacity supporting get(key) and put(key, value), both in O(1). When inserting into a full cache, evict the least-recently-used entry first. Both get and put count as a use.

Brute Force — O(n) per operation

Keep entries in an ordered list/array, most-recently-used at the front. get scans for the key and moves it to front; put scans, updates or inserts, and evicts the back when full. Linear scans make every op O(n).

Python# BRUTE: ordered list, O(n) per op
class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.data = []            # list of [key, value], front = most recent
    def get(self, key):
        for i, (k, v) in enumerate(self.data):
            if k == key:
                self.data.pop(i); self.data.insert(0, [k, v])
                return v
        return -1
    def put(self, key, value):
        for i, (k, _) in enumerate(self.data):
            if k == key: self.data.pop(i); break
        self.data.insert(0, [key, value])
        if len(self.data) > self.cap: self.data.pop()  # evict LRU

Why optimize? The list scan and the shift on reorder are O(n). The problem demands O(1) — achievable by pairing a hashmap (lookup) with a doubly-linked list (reorder).

Ordered list: O(n) per get/put, O(capacity) space.
Key Insight: Combine a hashmap (key → node, O(1) lookup) with a doubly-linked list ordered by recency. Most-recently-used sits at the head; evict from the tail. The DLL gives O(1) unlink-and-move; the map gives O(1) find.

How to Think About It

  1. Use a doubly-linked list with dummy head and tail sentinels to avoid null edge cases.
  2. Map key → node for O(1) access.
  3. get: if present, unlink the node and move it to the head (mark most recent), return its value; else -1.
  4. put: if key exists, update value and move to head; else create a node, insert at head, add to map.
  5. If size exceeds capacity, remove the node before the tail sentinel and delete it from the map.
Why This Works: A doubly-linked node knows its neighbors, so unlinking and re-inserting at the head is O(1) pointer surgery — no scan. The hashmap turns "find this key's node" into O(1). Together every operation touches a constant number of nodes. The tail always holds the least-recently-used entry, making eviction O(1).
Watch Out: Use dummy head/tail sentinel nodes. Without them, inserting into an empty list or removing the only node forces null-checks everywhere and invites pointer bugs.
  • get must also update recency — reading an entry makes it most-recently-used.
  • On put of an existing key, update the value and move to head; don't double-insert.
  • Remember to delete the evicted key from the hashmap, not just the list.

#127 Word Ladder

Hard BFS graph shortest-path

The Problem

Given beginWord, endWord, and a wordList, transform beginWord into endWord by changing one letter at a time, where every intermediate word must be in wordList. Return the number of words in the shortest transformation sequence (including both ends), or 0 if impossible.

Brute Force — O(N² · L) graph build

Build an explicit graph by comparing every pair of words and adding an edge when they differ by exactly one letter, then BFS from beginWord. Comparing all pairs is O(N² · L).

Python# BRUTE: O(N^2 * L) pairwise graph build, then BFS
from collections import deque
def ladderLength(beginWord, endWord, wordList):
    words = set(wordList)
    if endWord not in words: return 0
    nodes = [beginWord] + list(words)
    def one_diff(a, b):
        return sum(x != y for x, y in zip(a, b)) == 1
    adj = {w: [] for w in nodes}
    for i in range(len(nodes)):
        for j in range(i + 1, len(nodes)):
            if one_diff(nodes[i], nodes[j]):
                adj[nodes[i]].append(nodes[j]); adj[nodes[j]].append(nodes[i])
    q = deque([(beginWord, 1)]); seen = {beginWord}
    while q:
        w, d = q.popleft()
        if w == endWord: return d
        for nb in adj[w]:
            if nb not in seen: seen.add(nb); q.append((nb, d + 1))
    return 0

Pairwise build: O(N² · L) time, O(N²) edges space.
Key Insight: BFS over words. From each word, generate all one-letter neighbors (for each position, try 'a'..'z') and keep those in the wordList set. BFS explores level by level, so the first time you reach endWord you've found the shortest sequence.

How to Think About It

  1. Put wordList into a set for O(1) membership; return 0 if endWord isn't in it.
  2. BFS from beginWord with level/distance 1 (counting words, not steps).
  3. For the current word, for each character position, substitute each letter a..z to form a candidate.
  4. If a candidate is in the set and unvisited, mark it visited and enqueue it at the next level.
  5. When you dequeue endWord, return its level — BFS guarantees it's the shortest.
  6. If the queue empties without reaching endWord, return 0.
Why This Works: Treat each valid word as a graph node with edges to its one-letter-apart neighbors. BFS expands all words at distance d before distance d+1, so the level at which endWord first appears equals the shortest transformation length. Generating variants is cheaper than discovering edges pairwise.
Watch Out: endWord must itself be in the wordList, otherwise no valid sequence ends there — check up front and return 0.
  • Remove (or mark visited) words as you enqueue them, or BFS revisits and can loop / blow up.
  • The answer counts words in the path (begin and end included), so start the count at 1, not 0.

#297 Serialize and Deserialize Binary Tree

Hard tree design DFS

The Problem

Design serialize(root) → string and deserialize(string) → root so that a binary tree can be encoded to a string and rebuilt exactly. Values and structure (including nulls) must round-trip.

Baseline — O(n) preorder with null markers

There's no slower "brute force" worth showing — the canonical approach is already O(n). Encode with a preorder traversal that writes a sentinel (e.g. #) for null children; decode by consuming tokens in the same preorder.

Python# BASELINE: preorder with '#' null markers
class Codec:
    def serialize(self, root):
        out = []
        def go(node):
            if not node: out.append('#'); return
            out.append(str(node.val)); go(node.left); go(node.right)
        go(root)
        return ','.join(out)
    def deserialize(self, data):
        vals = iter(data.split(','))
        def build():
            v = next(vals)
            if v == '#': return None
            node = TreeNode(int(v))
            node.left = build(); node.right = build()
            return node
        return build()

Key Insight: A preorder traversal with explicit null markers uniquely determines a binary tree. Deserialize by consuming tokens in the same preorder order: read a value, recursively build the left subtree, then the right.

How to Think About It

  1. Serialize: preorder DFS — emit the node's value, then recurse left, then right; emit a null sentinel for empty children.
  2. Join tokens with a consistent delimiter (comma or space).
  3. Deserialize: split into tokens and walk them with a shared index/iterator.

Why This Works: Without null markers, preorder alone is ambiguous.   • Always emit and consume the null sentinel; skipping it desynchronizes left/right.

#23 Merge k Sorted Lists

Hard heap linked-list divide-and-conquer

The Problem

Given an array of k sorted linked lists, merge them into one sorted linked list and return its head.

Brute Force — O(N log N) collect-and-sort

Walk every list, dump all node values into one array, sort it, then rebuild a single linked list. Simple, but ignores the fact the inputs are already sorted, costing O(N log N).

Python# BRUTE: collect all values, sort, rebuild
def mergeKLists(lists):
    vals = []
    for head in lists:
        while head: vals.append(head.val); head = head.next
    vals.sort()
    dummy = tail = ListNode()
    for v in vals:
        tail.next = ListNode(v); tail = tail.next
    return dummy.next

Why optimize? Sorting throws away the per-list sortedness, costing O(N log N). A min-heap of the k current heads always knows the global minimum, giving O(N log k) — better since k ≤ N.

Collect-and-sort: O(N log N) time, O(N) space.
Key Insight: Keep a min-heap holding the current head of each list. Pop the smallest, append it to the result, then push that node's next. This is a k-way merge: the heap always surfaces the global minimum among the k frontiers.

How to Think About It

  1. Push the head of each non-empty list into a min-heap keyed by node value.
  2. Use a dummy head to simplify appending to the result list.
  3. Pop the smallest node, append it to the result tail.
  4. If the popped node has a next, push that next node into the heap.
  5. Repeat until the heap is empty; return dummy.next.
Why This Works: Each list is sorted, so its smallest unmerged element is its current head. The global minimum across all lists must be one of these k heads — exactly what the min-heap returns. Popping and pushing the successor maintains the invariant, producing a fully sorted merge in O(N log k).
Watch Out: Heaps can't compare raw nodes — provide a comparator or wrap entries as (value, tiebreaker, node). With equal values, comparing the node objects throws; the tiebreaker (a unique index) avoids it.
  • Skip empty lists when seeding the heap, and guard null heads.
  • Push the popped node's next only when it's non-null.
