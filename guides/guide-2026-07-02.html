Key InsightThe brute force re-scans after each removal. Instead push (char, run_count) pairs on a stack: when the incoming char matches the top, bump its count; the instant a count hits k, pop it. Removing a run automatically lets the neighbors on either side become adjacent — no re-scan needed.
How to Think About It

  2. For each character: if it equals the top's char, increment the top's count; otherwise push a new [char, 1].
  3. Whenever a top's count reaches k, pop that entry entirely.
  4. After the pass, rebuild the answer by repeating each surviving char by its stored count.
Why This WorksMerging into run-counts is exact bookkeeping: two equal characters are adjacent in the reduced string iff nothing different sits between them, which the stack guarantees. Watch Out For
  • Storing individual characters instead of run-counts forces you to compare the last k entries every step — store the count.
  • Forgetting to reconstruct with repetition (char * count) at the end.
  • Off-by-one: pop when count equals k, not after it exceeds k.

Key InsightMinimum depth is a shortest-path-to-a-leaf question, and BFS explores by increasing depth. The first leaf BFS dequeues is at the minimum depth — you can return immediately without visiting the rest of the tree.
How to Think About It
  1. Handle the empty tree: depth 0.

Why optimize? Re-summing greater values per node is O(n²); a single reverse in-order pass carrying a running total does it in O(n).

Key InsightA normal in-order BST traversal (left, node, right) visits values ascending. Reverse it — right, node, left — and you visit values descending. Carry a running sum of everything seen so far; when you reach a node, that sum is exactly the total of all greater values, so node.val += running, then update running.
How to Think About It
  1. Keep a single accumulator running = sum of already-visited (greater) values, starting at 0.
  2. Recurse right first — those are the largest values.
  3. At the node: running += node.val, then set node.val = running.
  4. Recurse left last — smaller values, which now include everything to their right.
Why This WorksReverse in-order is a strictly descending visit of a BST's keys. By the time you process a node, every value larger than it has already been added to running, so running holds precisely "sum of all greater values" — adding the node's own value then gives the required new value. The single-accumulator invariant makes it one linear pass.
Watch Out For
  • Traversing left-first (ascending) — that sums smaller values; you need descending order.
  • Resetting the accumulator inside the recursion; it must persist across the whole traversal (closure or reference).
  • Assigning node.val = running before adding the node itself — add first.

Key InsightA graph on n nodes is a tree iff it is connected and has exactly n-1 edges. Key InsightWords are graph nodes; an edge connects words differing by one letter. Shortest sequence = shortest path = BFS. The trick is neighbor generation: instead of diffing against every other word, substitute each of the 26 letters at each position and test membership in a set — O(L·26) candidates per word, no pairwise scan.
How to Think About It
  1. Put the word list in a hash set for O(1) membership; bail out early if endWord isn't present.
  2. BFS from beginWord with distance 1, tracking visited words to avoid revisits.
  3. To expand a word, loop over each position and each letter a..z, forming a candidate.
  4. If the candidate is in the set and unseen, mark it visited and enqueue with distance + 1.
  5. Return the distance when you dequeue endWord; return 0 if the queue drains.
Why This WorksBFS explores nodes in order of increasing distance from the start, so the first time endWord is reached, that distance is minimal. Generating neighbors by substitution enumerates exactly the words at Hamming distance 1 that exist in the set — the same edges as the brute force, just found without scanning all N words.

  • Returning 0 when endWord is not in the list — check up front.
  • Marking visited on dequeue instead of enqueue can push duplicates and blow up the queue.
  • Off-by-one on the count: the sequence length includes both begin and end words (start distance at 1).

Key InsightEvery ugly number is a smaller ugly number multiplied by 2, 3, or 5. So the sorted sequence is the merge of three streams — 2×, 3×, 5× the ugly numbers found so far. Keep three pointers into your growing dp array and always append the smallest of dp[i2]*2, dp[i3]*3, dp[i5]*5, advancing every pointer that produced that minimum (to skip duplicates like 6).
How to Think About It
  1. Start dp = [1] and pointers i2 = i3 = i5 = 0.
  2. Each step, the next ugly number is min(dp[i2]*2, dp[i3]*3, dp[i5]*5).
  3. Append it to dp.
  4. Advance each pointer whose product equals that minimum — this dedupes numbers reachable two ways (e.g. 6 = 2×3 = 3×2).
  5. After n-1 appends, dp[n-1] is the answer.
Why This WorksBy induction, dp holds the ugly numbers in sorted order. Any ugly number greater than 1 factors as 2, 3, or 5 times a strictly smaller ugly number, which is already in dp at or after the corresponding pointer. Why optimize? The BFS explores many redundant states; a number-theoretic observation collapses the whole thing to summing n's prime factors in O(√n).

Key InsightTo reach a length that is a multiple of a block you already have, the cheapest move is one Copy All followed by (factor − 1) Pastes — costing exactly factor operations to multiply the length by factor. So dp[i] = min over divisors j of (dp[j] + i/j), and the minimum is achieved by prime factors: the answer is simply the sum of the prime factors of n.
How to Think About It
  1. Realize you can only ever multiply the current length by an integer factor (Copy once, Paste factor−1 times), at a cost equal to that factor.
  2. So building n means factoring it into a product of integers; total cost is the sum of those factors.
  3. Summing is minimized by breaking every composite factor into primes (since a+b ≤ a·b for a,b ≥ 2).
  4. Therefore: repeatedly divide n by its smallest prime factor d, adding d each time, until n = 1.
Why This WorksEach "copy then paste" burst multiplies the screen length by some integer f ≥ 2 at cost f. Reaching n factorizes it as f₁·f₂·… with total cost f₁+f₂+…. Because splitting any composite factor f = a·b replaces cost ab with the smaller a+b, the optimum uses only prime factors — giving cost equal to the sum of n's prime factorization (with multiplicity).
Watch Out For
  • n = 1 needs 0 operations (you already have one A).
  • Prime n costs exactly n (one copy, n-1 pastes) — don't special-case it away.
  • Remember to keep dividing by the same prime while it divides (factors with multiplicity, e.g. 8 = 2+2+2 = 6).

Key InsightFix the first digit: the permutations split into n blocks of (n-1)! each, sorted. So the first digit's index is k / (n-1)!; the remainder k % (n-1)! recurses into the block. This is the factorial number system — each digit is chosen by dividing by the appropriate factorial, no enumeration.
How to Think About It
  1. Convert to 0-indexed: k -= 1.
  2. Keep the still-available digits in a sorted list [1..n].
  3. For position from n down to 1, let f = (i-1)!; the chosen digit index is k / f.
  4. Append that digit (removing it from the list), then set k = k % f.
  5. Repeat until the list is empty — the collected digits are the answer.
Why This WorksLexicographic order groups permutations by their leading digit into equal blocks of (n-1)!. Integer division k / (n-1)! selects which block (hence which leading digit), and the remainder is the rank within that block — an identical subproblem on the remaining digits. Peeling one digit per step reconstructs exactly the k-th permutation, because factorials tile the index space with no gaps or overlaps.
Watch Out For
  • Off-by-one: the problem is 1-indexed, so subtract 1 from k before the divisions.
  • Removing the used digit from the candidate list each step — forgetting this reuses digits.
  • Precompute factorials to avoid recomputing (i-1)! repeatedly.

        let prev: ListNode | null = null, cur = head!;
        while (cur.next) { prev = cur; cur = cur.next; }
        prev!.next = null;
        cur.next = head;
        head = cur;
    }
    return head;
}

        for _ in 0..k { let last = vals.pop().unwrap(); vals.insert(0, last); }

        next
    }

Why optimize? With k up to 2×10⁵, repeating single rotations is O(n·k); one length pass plus a single cut is O(n).

Key InsightRotating right by k is the same as rotating by k mod L (where L is the length). Close the list into a ring by joining tail to head, then walk to the new tail at position L - k%L and break the ring there — the node after the break is the new head. One and a half passes total.
How to Think About It
  1. Traverse once to find the length L and the tail node.
  2. Reduce k = k % L; if it's 0, the list is unchanged.
  3. Connect tail.next = head to form a cycle.
  4. The new tail is L - k steps from the old head; walk there.
  5. Set newHead = newTail.next, then cut with newTail.next = null.
Why This WorksRotating by L returns the original list, so only k mod L matters. Forming a cycle makes every node reachable in order; the last k nodes should move to the front, which means the break belongs right after the first L - k nodes. Cutting there yields a list whose head is the old (L-k)-th node — exactly a right rotation by k.
Watch Out For
  • Not reducing k modulo L — large k otherwise loops needlessly (or breaks the index math).
  • Empty list or single node: return as-is.
  • Forgetting to sever newTail.next — leaving the cycle intact creates an infinite list.

Two Pointers

Directly extending #15 (3Sum) and #167: fix outer indices and collapse the inner search to a two-pointer sweep. The dedup discipline is the whole game. Interleaved

#18 4Sum MediumTwo PointersSorting

The ProblemGiven an array nums and a target, return all unique quadruplets [a, b, c, d] (distinct indices) such that a + b + c + d == target. The solution set must not contain duplicate quadruplets.
Brute Force — O(n⁴) time

Four nested loops over all index combinations; whenever the four values sum to target, add the sorted quadruplet to a set to dedupe. Correct but quartic.

Python
def fourSum(nums, target):
    n = len(nums)
    res = set()
    for i in range(n):
        for j in range(i+1, n):
            for k in range(j+1, n):
                for l in range(k+1, n):
                    if nums[i]+nums[j]+nums[k]+nums[l] == target:
                        res.add(tuple(sorted((nums[i], nums[j], nums[k], nums[l]))))
    return [list(t) for t in res]

Why optimize? With n up to 200, O(n⁴) is 1.6×10⁹ — borderline, and the set-of-tuples dedup is memory-heavy. Sorting plus two pointers drops it to O(n³) with cheap skip-based dedup.

Key InsightSort the array, then fix the outer two values with nested loops and reduce the inner search to the classic two-pointer two-sum on the remaining suffix. Sorting also makes deduplication trivial: skip over equal neighbors at each of the four levels. This is #15 (3Sum) with one more outer loop.
How to Think About It
  1. Sort nums ascending.
  2. Loop i over the first index, skipping duplicates; loop j from i+1, skipping duplicates.
  3. Set lo = j+1, hi = n-1; compute sum = nums[i]+nums[j]+nums[lo]+nums[hi].
  4. If sum < target advance lo; if sum > target retreat hi; if equal, record the quad and move both inward, skipping duplicate values.
  5. Optional pruning: break when the smallest possible sum exceeds target, or the largest is below it.
Why This WorksOn a sorted array, for fixed i, j the two-sum target target - nums[i] - nums[j] is monotone under the two pointers: increasing lo only raises the sum, decreasing hi only lowers it, so every pair is considered without missing any. Skipping equal adjacent values at each level guarantees each distinct quadruplet is emitted exactly once, since duplicates are always contiguous after sorting.
Watch Out For
  • Integer overflow: four values near 10⁹ can exceed 32-bit range — accumulate in 64-bit.
  • Dedup at all four levels; skipping duplicates only at i and j still yields repeated quads.
  • Advancing only one pointer on a match — move both, then skip equals, or you re-emit the same quad.
