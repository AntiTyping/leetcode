Quick ReminderTwo pointers from both ends: the sortedness lets you move left up when the sum is too small and right down when it is too large.

class Solution(object):
    def lengthOfLIS(self, nums):
        dp = [1] * len(nums)
        for r in range(1, len(nums)):

Key InsightBecause the array is sorted, the predicate “nums[i] ≥ target” is false on a prefix and then true forever after; the answer is the first true index. Brute inspects every element on the false side one at a time, whereas binary search jumps straight to that boundary.
How to Think About It
  1. Use a half-open window lo = 0, hi = n; setting hi = n lets the answer represent “insert past the last element.”
  2. While lo < hi, compute mid = lo + (hi - lo) / 2.
  3. If nums[mid] ≥ target, the boundary is at mid or to its left, so set hi = mid; otherwise set lo = mid + 1.

Watch Out For
  • Initialize hi = n (not n - 1) so inserting after the last element is representable.
  • Pick one interval convention (half-open here) and keep the lo/hi update rules consistent with it.
  • Return lo (which equals hi) after the loop; there is no “not found” sentinel to return.

Key InsightA valid subarray is simply one that contains at most k zeros. Brute recomputes heavily overlapping ranges from scratch for every start; a single window whose left edge only ever moves forward re-examines each element at most twice, because when it holds too many zeros you advance the left edge instead of restarting.
How to Think About It

  • Only decrement the zero count when the element actually leaving the window is a zero.
  • Handle the boundaries k = 0 (no flips allowed) and k = nums.length (the whole array is valid).

Key InsightBrute rescans and rebuilds the entire string after every single removal; processing characters left to right against a stack lets each character be pushed once and cancelled at most once against the current top, so one pass suffices.
How to Think About It
  1. Traverse the string once, maintaining a stack of surviving characters.
  2. For each incoming character, if it equals the top of the stack, pop the top (the pair annihilates); otherwise push the character.
  3. Because a pop exposes a new top, the very next comparison can immediately cancel again with the character beneath — the stack chains these removals automatically.
  4. Join the remaining stacked characters, in bottom-to-top order, into the answer string.
Why This WorksRemoving an adjacent pair only ever exposes the two characters that flanked it, which are precisely the new stack top and the incoming character. The stack therefore reproduces the repeated-removal process in a single pass, and whatever remains on it is the unique fully reduced string.
Watch Out For
  • Only compare the incoming character to the top when the stack is non-empty.
  • Build the result from the stack in order (bottom to top); reversing it gives the wrong answer.
  • Use a character stack (bytes/chars), not repeated string slicing or erasing, to keep the pass linear.

#617 Merge Two Binary Trees EasyTreeDFS

The ProblemGiven the roots of two binary trees, merge them: wherever both trees have a node, the merged node's value is the sum of the two; wherever only one tree has a node, that whole subtree is used as-is. Return the root of the merged tree. Reuses the same idea of lockstep traversal seen in #100 Same Tree.
Brute Force — O(n) time, O(n) space

Instead of clean recursion, walk both trees together with an explicit stack of node pairs, mutating the first tree in place. When both children exist, push the pair to process later; when the first tree lacks a child, splice in the second tree's subtree directly. It produces the correct merged tree but is bookkeeping-heavy compared with the recursive form.

Python
class Solution:
    def mergeTrees(self, t1, t2):
        if not t1:
            return t2
        stack = [(t1, t2)]
        while stack:
            a, b = stack.pop()
            if not b:
                continue
            a.val += b.val
            if not a.left:
                a.left = b.left
            else:
                stack.append((a.left, b.left))
            if not a.right:
                a.right = b.right
            else:
                stack.append((a.right, b.right))
        return t1

#513 Find Bottom Left Tree Value MediumTreeBFS

The ProblemGiven the root of a binary tree, return the leftmost value in the last (deepest) row of the tree. "Leftmost" means the first node encountered when scanning that deepest level from left to right.
Brute Force — O(n) time, O(n) space

Run a depth-first traversal that records the value of every node grouped by its depth in a map from depth to a list of values. After the traversal, find the maximum depth and return the first value stored for that level. It works but materializes every value at every level even though only one value per level is ever needed.

Python
class Solution:
    def findBottomLeftValue(self, root):
        from collections import defaultdict
        levels = defaultdict(list)
        def dfs(node, depth):
            if not node:
                return
            levels[depth].append(node.val)
            dfs(node.left, depth + 1)
            dfs(node.right, depth + 1)
        dfs(root, 0)
        deepest = max(levels)
        return levels[deepest][0]

The ProblemIn a town of n people labeled 1..n, there may be a judge who is trusted by everyone else but trusts no one. The ProblemGiven a directed graph as an adjacency list graph where graph[i] lists the nodes reachable in one step from i, a node is safe if every possible path starting from it eventually reaches a terminal node (a node with no outgoing edges) — that is, no path from it can get stuck in a cycle. Brute Force — O(n · (n + e)) time, O(n) space

For each node independently, run a fresh depth-first search that tracks the nodes currently on the recursion path. If the search ever revisits a node already on the path, that node can reach a cycle and is unsafe; otherwise it is safe. Because it restarts from scratch for every node with no memoization, the same subgraphs are explored again and again.

class Solution:
    def eventualSafeNodes(self, graph):
        n = len(graph)
        def dfs(node, on_path):
            if node in on_path:
                return True  # found a cycle
            on_path.add(node)
            for nxt in graph[node]:
                if dfs(nxt, on_path):
                    return True
            on_path.remove(node)
            return False
        return [i for i in range(n) if not dfs(i, set())]

The ProblemYou are given routes, where routes[i] is the ordered, repeating list of bus stops served by the i-th bus. Starting at stop source (not on any bus yet), return the fewest number of buses you must take to reach stop target, or -1 if it is impossible. Each bus can be ridden any number of stops, and you may transfer between buses that share a stop.
Brute Force — O(R · S · L) time, O(S) space

Do a level-by-level BFS over stops where each new level represents boarding one more bus. From the frontier of reachable stops, scan every route to find any that passes through a frontier stop, then enqueue all of that route's stops. A visited-stops set stops infinite loops, but because routes are never marked as used, the same routes are rescanned on every level — correct but slow.

Python
class Solution:
    def numBusesToDestination(self, routes, source, target):
        if source == target:
            return 0
        from collections import deque
        visited = {source}
        q = deque([source])
        buses = 0
        while q:
            buses += 1
            for _ in range(len(q)):
                stop = q.popleft()
                for route in routes:          # no route pruning
                    if stop in route:
                        for nxt in route:
                            if nxt == target:
                                return buses
                            if nxt not in visited:
                                visited.add(nxt)
                                q.append(nxt)
        return -1

#118 Pascal's Triangle EasyDynamic Programming

The ProblemGiven an integer numRows, return the first numRows rows of Pascal's Triangle. Brute Force — O(n^3) time, O(n^2) space

The value at row r, column c equals the binomial coefficient C(r, c). A naive approach computes each entry independently using the factorial formula C(r,c) = r! / (c! (r-c)!). This recomputes factorials from scratch for every single cell and can overflow for large rows, but it produces the correct triangle.

        r = 1
        for i in range(2, n + 1):
            r *= i
        return r
    def comb(n, k):
        return fact(n) // (fact(k) * fact(n - k))
    res = []
    for r in range(numRows):
        res.append([comb(r, c) for c in range(r + 1)])
    return res

        for (let c = 0; c <= r; c++) row.push(comb(r, c));
        res.push(row);

Why optimize? Recomputing factorials for every cell wastes work and overflows quickly, when each entry can be built directly from the row above using only addition.

Key InsightYou never need factorials at all: each interior entry is simply the sum of the two entries above it. Build every row from the previous one, so the triangle grows additively with no large intermediate numbers.
How to Think About It
  1. Start the result with the first row [1].
  2. For each new row, begin and end with 1; every interior slot is the sum of two adjacent values from the previous row.
  3. Read the previous row's neighbors pairwise to fill the middle of the new row.
  4. Append each completed row and repeat until you have numRows rows.
Why This WorksPascal's Triangle is defined by the recurrence C(r,c) = C(r-1,c-1) + C(r-1,c), with the edges equal to 1. Summing adjacent entries of the previous row is exactly this recurrence, so the additive construction is provably correct.
Watch Out For
  • Off-by-one when indexing the previous row; the interior loop must stop before running past its last element.
  • Forgetting that every row starts and ends with 1, including the single-element row 0.
  • Mutating the previous row in place while reading it, which corrupts the sums.

#120 Triangle MediumDynamic Programming

The ProblemGiven a triangle array where row r has r+1 elements, find the minimum path sum from the top to the bottom. From index c on one row you may step to index c or index c+1 on the next row. Return the smallest total sum of values along any such top-to-bottom path.
Brute Force — O(2^n) time, O(n) space

Recurse from the top: at each cell, try both legal moves (straight down and diagonally down-right) and take the cheaper of the two subpaths. This explores an exponentially branching tree of paths because overlapping subproblems are recomputed. It is correct but blows up for large triangles.

Python
def minimumTotal(triangle):
    n = len(triangle)
    def dfs(row, col):
        if row == n - 1:
            return triangle[row][col]
        down = dfs(row + 1, col)
        diag = dfs(row + 1, col + 1)
        return triangle[row][col] + min(down, diag)
    return dfs(0, 0)

Why optimize? The same (row, col) subproblem is solved over and over, so caching or reusing results collapses the exponential tree into linear-in-cells work.

Key InsightWork from the bottom up: the best cost to reach the bottom from any cell equals that cell's value plus the smaller best-cost of the two cells beneath it. Compute one row at a time using only the results of the row directly below, so you can even overwrite in place.
How to Think About It
  1. Recognize the last row's best cost is just each cell's own value.
  2. Process rows from second-to-last upward; for each cell add the minimum of its two children's best costs.
  3. Store only one running array of best costs and update it as you move up (no full 2D table needed).
  4. After processing the top row, the single remaining value is the answer.
Why This WorksEvery path's cost decomposes into a cell plus the best path from one of its two children. Solving children before parents guarantees each parent already has correct child answers, so the bottom-up sweep computes the global minimum exactly once per cell.
Watch Out For
  • Mixing up the two valid moves: only index c and c+1 on the next row are reachable.
  • Iterating rows top-down for the in-place version, which would clobber values you still need.
  • Assuming values are non-negative; negatives are allowed, so always compare, never shortcut on sign.

Greedy

A locally optimal choice that provably stays globally optimal — spend the least flexible resource first, or fix the largest deficit first. Interleaved.

#860 Lemonade Change EasyGreedy

The ProblemCustomers line up to buy lemonade costing $5 each, paying with a $5, $10, or $20 bill given in order by the array bills. You start with no change and must give correct change to every customer using only bills you have received. Return true if you can serve everyone in order, otherwise false.
Brute Force — O(n · 2^n) time, O(n) space

Keep every bill you have received in a wallet list. For each customer who overpays, run a recursive subset search that tries including or excluding each held bill to reach the exact change owed. If no subset works, fail; otherwise remove those bills. This exhaustively explores subsets and is exponential per customer, but it never assumes which bills to hand back.

Python
def lemonadeChange(bills):
    wallet = []
    def give(amount, idx):
        if amount == 0:
            return []
        if idx >= len(wallet) or amount < 0:
            return None
        used = give(amount - wallet[idx], idx + 1)
        if used is not None:
            return [wallet[idx]] + used
        return give(amount, idx + 1)
    for b in bills:
        change = b - 5
        if change > 0:
            picked = give(change, 0)
            if picked is None:
                return False
            for p in picked:
                wallet.remove(p)
        wallet.append(b)
    return True

Why optimize? There are only two change scenarios ($10 owes $5, $20 owes $15), so a fixed rule beats searching every subset of your wallet.

Key InsightYou only ever owe $5 or $15, and you should always hand back the least flexible bills first. A $10 can only be used to make change for a $20, whereas a $5 is useful everywhere, so prefer spending tens before fives when giving $15.
How to Think About It
  1. Track only how many $5 and $10 bills you currently hold; you never give $20 as change.
  2. A $5 payment needs no change — just keep it.
  3. A $10 payment needs one $5 back; fail if you have none.
  4. A $20 payment needs $15: give a $10+$5 if possible, else three $5; fail if neither is available.
Why This WorksHanding out a $10 before three $5 bills preserves your most versatile currency for future customers. Since a $5 can satisfy both change amounts but a $10 only satisfies the $15 case, spending tens first never forecloses a future option that spending fives first would have kept open — the greedy choice is safe.
Watch Out For
  • Forgetting that the very first customer must pay with a $5, or you fail immediately.
  • Giving three $5 bills when a $10+$5 combination was available, wasting scarce fives.
  • Never needing to hand out a $20 as change — don't track it as spendable.

#1005 Maximize Sum Of Array After K Negations EasyGreedy

The ProblemGiven an integer array nums and an integer k, you must perform exactly k operations. Each operation picks an index i and replaces nums[i] with -nums[i]; the same index may be chosen more than once. Return the largest possible sum of the array after applying exactly k negations.
Brute Force — O(n · k) time, O(1) space

Perform the operations one at a time. On each of the k steps, scan the whole array to find the current minimum element and negate it. Flipping the smallest value each round pushes the sum up, but rescanning for the minimum on every step makes this O(n · k). Finally sum the array.

Python
def largestSumAfterKNegations(nums, k):
    nums = list(nums)
    for _ in range(k):
        i = nums.index(min(nums))
        nums[i] = -nums[i]
    return sum(nums)

Why optimize? Rescanning for the minimum every step is wasteful; ordering the elements once by magnitude lets you decide all negations in a single pass.

Key InsightSpend your negations on the most negative numbers first, because flipping a negative to positive gains the most. Process values in order of magnitude, and if operations remain after all negatives are gone, any leftover flips should land on the single smallest-magnitude value — and only matter if their count is odd.
How to Think About It
  1. Sort or order the values by absolute value so the largest-magnitude negatives are addressed first.
  2. Flip each negative to positive while you still have operations left, since that strictly increases the sum.
  3. If you run out of negatives with operations remaining, all further flips hit the smallest-magnitude element.
  4. Those leftover flips cancel in pairs, so only an odd remainder flips the smallest value once; then total the array.
Why This WorksTurning a negative into a positive of the same magnitude raises the sum by twice its magnitude, and doing the largest magnitudes first maximizes that gain. Once nothing is negative, flipping the same smallest element back and forth is optimal because two flips are a no-op, so only the parity of the leftover operations matters — an odd leftover sacrifices the least by hitting the smallest value.
Watch Out For
  • Ignoring that k may exceed the number of negatives, leaving extra operations you must still spend.
  • Forgetting parity: an even leftover changes nothing, an odd leftover flips exactly one element.
  • Applying leftover flips to the wrong element — it must be the smallest by absolute value, which may be a now-positive number.
