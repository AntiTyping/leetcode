    def solveNQueens(self, n):
        res, cols, d1, d2 = [], set(), set(), set()
        board = [['.'] * n for _ in range(n)]

            if r == n:

        bt(0)
        return res

Key InsightThe predicate “i*i <= x” is monotonic: true for all small i, then false forever after. That boundary is exactly what binary search finds — you want the largest i where the predicate still holds.
How to Think About It
  1. Recognize the search space of candidate roots is [0, x] (or tighter, [0, x/2 + 1]), and the test mid*mid <= x partitions it into a true-block then a false-block.
  2. Binary search for the boundary: keep lo and hi, compute mid, and decide which half to keep based on the predicate.
  3. When the predicate is true, mid is a valid floor candidate — move lo up but remember mid. When false, move hi down.
  4. To avoid mid*mid overflowing a 32-bit int, compare with division: test mid <= x / mid instead (guarding mid == 0).
  5. Continue until the range collapses; the last mid that passed the predicate is the floor square root.
Why This WorksBecause i ↦ i*i is strictly increasing on non-negative integers, the set of i with i*i <= x is a contiguous prefix [0, r]. Binary search provably converges to its right endpoint r, which by definition is floor(sqrt(x)).

  • Integer overflow: mid*mid can exceed 32-bit range even when the answer fits. Use mid <= x/mid or 64-bit math.
  • Edge cases x = 0 and x = 1 must return 0 and 1 respectively.
  • Off-by-one at the boundary: be clear whether you return hi, lo-1, or a stored candidate.
  • Setting hi = x is fine but hi = x/2 + 1 is tighter for large x.

Key InsightBecause arr is sorted, the k closest elements are always a contiguous length-k slice. So the real unknown is just the left index of that window, which ranges over [0, n-k] — and that index is itself binary-searchable.
How to Think About It
  1. Reframe: instead of picking k elements, pick the single left boundary lo of the window arr[lo .. lo+k-1], with lo in [0, n-k].
  2. Binary search that boundary. For a midpoint window starting at mid, compare the element just outside on the left, arr[mid], against the element just outside on the right, arr[mid+k].
  3. The comparison is on distances to x: weigh x - arr[mid] (how far the left edge is) against arr[mid+k] - x (how far the right edge is).
  4. If the left edge is farther than the right neighbor, the window should slide right (lo = mid+1); otherwise slide left (hi = mid).
  5. When the search converges, lo is the window start; return the slice arr[lo .. lo+k], which is already ascending.
  6. The tie rule (prefer smaller) falls out of using a strict inequality in the right direction — equal distances keep the left window.
Why This WorksFor sorted arr, sliding the window right trades away its leftmost element for a new rightmost one. The function “is the right neighbor closer than the left edge?” is monotonic in the window start, so binary search converges on the first position where sliding right no longer helps — the optimal, tie-preferring-smaller window.
Watch Out For
  • Comparison direction: use x - arr[mid] > arr[mid+k] - x. Getting the sign backwards silently returns a valid-looking but wrong window.
  • Ties: the comparison must be strict so equal distances keep the smaller (left) element, matching the tie rule.
  • Index bounds: mid+k must stay in range, which is guaranteed only if hi starts at n-k, not n-1.
  • Do not compare absolute values here — within the window the left edge is ≤ x-side and the right neighbor is ≥ x-side, so signed subtraction is intentional.
  • Return a slice of length exactly k; off-by-one on the upper index drops or adds an element.

Why optimize? A blind traversal visits every node even though the BST ordering tells you exactly which half to skip — on a balanced tree that is O(n) instead of O(log n).

Key Insight In a BST every left descendant is smaller and every right descendant is larger than a node, so a single comparison of val against the current node reveals which one child could possibly hold it.
How to Think About It
  1. Start at the root; if it is null the value is absent — return null.
  2. Compare val with the current node's value.
  3. If equal you have found the node — return it as the subtree root.
  4. If val is smaller, only the left subtree can contain it; move left.
  5. If val is larger, only the right subtree can contain it; move right.
  6. Repeat until you match or fall off the tree.
Why This Works The BST invariant guarantees the target, if present, lies on exactly one side of each node, so discarding the other side can never lose it — each step halves the remaining tree on average.
Watch Out For
  • Empty tree or not-found must cleanly return null.
  • Branch on the comparison — do not fall back to scanning both children.
  • Iterative and recursive walks are both fine, but keep the null base case.
  • Return the whole subtree node, not just its value.

Key Insight There is exactly one leaf position consistent with the ordering. Walk down choosing left or right by comparison until a child pointer is null, then hang the new node there — just like a search that overshoots by one step.
How to Think About It

Key InsightChoose digits in strictly increasing order so each combination is generated exactly once, and abandon a branch the moment the running sum exceeds n or there are too few remaining digits to reach length k.
How to Think About It
  1. Recurse with a start digit, the current path, and the remaining target.
  2. When path has exactly k digits, accept it only if the remaining target is 0.
  3. Loop the next digit from start to 9, always moving forward so combinations stay sorted and unique.
  4. Subtract the chosen digit from the target and recurse with start = digit + 1.
  5. Prune: if the digit already exceeds the remaining target, break the loop — larger digits are hopeless too.

Keep a stack of candidate "2" values and a variable third holding the largest value you have already confirmed can serve as the middle-of-the-pattern "2" (because a taller "3" sits to its right). If the current element drops below third, a valid "1" completes the pattern.
How to Think About It
  1. Think of the triple as roles: a low "1", a high "3", and a middle "2" between them, in that index order.
  2. Fix the "2" as the value you most want to beat, and remember the best one found so far as third.
  3. Iterate from the right so every element you examine is a potential "1".
  4. Before using an element as "1", pop stack values smaller than it, promoting each into third (they had a taller value to their right).
  5. If the current element is strictly below third, you have a "1" under a valid "2" under a "3" — return true.
  6. Otherwise push the element as a future "3" candidate and continue.
Why This Works Anything popped from the stack had a larger element to its right, so it is a legitimate "2" backed by a "3". Raising third to the largest such popped value maximizes your chance of finding a qualifying "1" later, and the stack stays monotdecreasing so each element is pushed and popped once.
Watch Out For
  • The scan direction is right-to-left — going left-to-right does not give the same guarantee.
  • All three comparisons are strict inequalities.
  • Initialize third to negative infinity so the first comparison is safe.
  • Pop while the top is smaller than the current value, updating third each time.

#767 Reorganize String MediumHeap / Greedy

The Problem Given a string s, rearrange its characters so that no two adjacent characters are the same. Return any valid arrangement, or an empty string if none exists.
Brute Force — O(n! ) time, O(n) space

Backtrack: at each position try any character that still has remaining count and differs from the previously placed one. If the full length is built, that arrangement works; otherwise undo and try another.

Python
def reorganizeString(s):
    from collections import Counter
    n = len(s)
    counts = Counter(s)
    res = []
    def backtrack():
        if len(res) == n:
            return True
        for ch in counts:
            if counts[ch] > 0 and (not res or res[-1] != ch):
                counts[ch] -= 1
                res.append(ch)
                if backtrack():
                    return True
                res.pop()
                counts[ch] += 1
        return False
    return "".join(res) if backtrack() else ""

Why optimize? Backtracking explores a factorial search space. A greedy rule — always place the character with the most remaining copies that is not the one you just placed — needs only a priority queue and runs in near-linear time.

Key Insight The character with the highest remaining count is the most constrained; delay it and it gets stuck adjacent to itself. A max-heap keyed by frequency lets you always emit the most frequent available character while holding back the one just used.
How to Think About It
  1. Count how many times each character appears.
  2. Feasibility check: if any count exceeds (n + 1) / 2, no valid arrangement exists.
  3. Push all (count, char) pairs into a max-heap ordered by count.
  4. Pop the most frequent character, append it, and set the previous one aside so it cannot be picked next.
  5. After placing, decrement its count and push the previously-held character back in if it still has copies.
  6. Repeat until the heap and the held slot are drained.
Why This Works Emitting the most frequent character at every step keeps the largest count as low as possible relative to remaining length, so a gap always exists to separate duplicates — provided the initial feasibility bound holds. Holding the just-used character one turn guarantees adjacency is never violated.
Watch Out For
  • Return early with an empty string when the max count exceeds (n + 1) / 2.
  • Only re-insert the held character after placing a different one.
  • Most languages give a max-heap by negating counts or by comparator; Python's heapq is a min-heap.
  • Tie-breaking between equal counts does not affect correctness.

#1049 Last Stone Weight II MediumDP / Subset Sum

The Problem You repeatedly smash any two stones; if weights differ, a stone of the difference remains. Return the smallest possible weight of the last remaining stone (or 0). Each smash is a choice of signs, so the answer is the minimum achievable |S1 - S2| over all ways to split the stones into two groups.
Brute Force — O(2^n) time, O(1) space

Enumerate every subset as one group; its complement is the other. For each split compute |total - 2 * subsetSum| and keep the minimum.

Why optimize? Enumerating all subsets is exponential. Because only the reachable sums matter — not which stones produce them — a boolean DP over achievable totals collapses the search into a pseudo-polynomial table.

Key Insight Minimizing |S1 - S2| is the same as choosing a subset whose sum is as close as possible to total / 2 without exceeding it. Track which sums up to total / 2 are reachable, take the largest reachable s, and the answer is total - 2 * s.
How to Think About It
  1. Rewrite the problem: every stone gets a plus or minus sign, and you want the smallest non-negative total.
  2. That equals splitting into two groups and minimizing their difference.
  3. Cap the target at total / 2 since the two halves are symmetric.
  4. Build a boolean "reachable sum" set, seeded with 0, adding each stone in turn (0/1 knapsack).
  5. Scan reachable sums downward from total / 2 for the largest achievable s.
  6. Return total - 2 * s.
Why This Works Any assignment of signs corresponds to a subset sum s, giving difference total - 2 * s. Making s as large as possible while <= total / 2 minimizes that difference. The DP records exactly the set of achievable subset sums, so the optimum is always found.
Watch Out For
  • Iterate sums downward when using a 1-D boolean array so each stone is used at most once.
  • Cap the DP width at total / 2 (integer division).
  • The answer is total - 2 * s, not s itself.
  • All-equal or single-stone edge cases still fit the same formula.

Brute Force — O(n^2) time, O(n) space

Sort by width, then run a classic longest-increasing-subsequence DP: dp[i] is the longest chain ending at i, extended from any earlier envelope strictly smaller in both dimensions.

def maxEnvelopes(envelopes):
    envelopes.sort()
    n = len(envelopes)
    dp = [1] * n
    best = 0
    for i in range(n):
        for j in range(i):
            if envelopes[j][0] < envelopes[i][0] and envelopes[j][1] < envelopes[i][1]:
                dp[i] = max(dp[i], dp[j] + 1)
        best = max(best, dp[i])
    return best

Key Insight Sort by width ascending, and for ties in width sort height descending. That tie-break makes it impossible to chain two envelopes of equal width (their heights now decrease), so a plain longest-increasing-subsequence on the heights gives the answer.
How to Think About It
  1. You need both dimensions strictly increasing — a 2-D chain.
  2. Sort by width so the first dimension is handled left-to-right.
  3. Break width ties by descending height so equal widths cannot both appear in an increasing height run.
  4. Now the whole problem is: longest strictly increasing subsequence of the height array.
  5. Run patience-sorting LIS, using binary search to place each height on the leftmost tail it can replace.
  6. The number of "piles" (tails length) is the answer.
Why This Works After the sort, any increasing height subsequence automatically has strictly increasing widths too — except equal-width pairs, which the descending-height tie-break prevents from forming an increasing run. So one-dimensional LIS on heights faithfully counts valid nestings.
Watch Out For
  • The equal-width, descending-height tie-break is the crux — ascending height here would over-count.
  • Both dimensions require strict inequality.
  • Use a leftmost binary search (lower bound) so the LIS stays strict.
  • Do not run 2-D comparisons after the trick; that reintroduces the O(n^2) cost.

A prefix trie storing the 3 smallest completions per node captures the same idea.
How to Think About It
  1. Sort products so lexicographic order is baked into the array positions.
  2. Think of each prefix as selecting a contiguous window of the sorted array where every entry starts with that prefix.
  3. As you append the next character, the window can only narrow — the left edge moves right when short/smaller words fall away, the right edge moves left when the prefix no longer matches.
  4. From the current window, the first (up to) 3 entries are already the lexicographically smallest completions.
  5. Alternatively, insert each sorted product into a trie; at every node keep a bounded list of the first 3 words that pass through it, then walk the search word down the trie.
  6. Stop contributing suggestions for a prefix as soon as the window empties or the trie path breaks.
Why This Works Sorting guarantees that words sharing a prefix are adjacent, and a longer prefix is always a refinement of a shorter one, so the valid window is monotonically nested. Because we only ever need the 3 smallest, reading from the front of the window (or a per-node cap of 3 in the trie) is sufficient and never re-examines discarded products.
Watch Out For
  • Lexicographic (dictionary) order, not length — sort strings, not by size.
  • Cap each inner list at 3 even when many products match.
  • A prefix that matches nothing yields an empty list, and all longer prefixes also match nothing.
  • Prefix comparison must respect the exact prefix length; a partial character match is not a full match.
  • Case sensitivity and empty product strings can trip up naive comparisons.

Bit Manipulation

Bitwise counting isolates the one element that breaks a k-fold repetition pattern — no extra hash needed. Interleaved.

#137 Single Number II MediumBit Manipulation

The Problem Every integer in nums appears exactly three times except for one integer that appears exactly once. Return that single integer. You must design an algorithm with linear runtime and, ideally, use only constant extra space.
Brute Force — O(n) time, O(n) space

Count how many times each value occurs with a hash map, then scan the counts and return the value whose count is 1.

Python
from collections import Counter

def singleNumber(nums):
    counts = Counter(nums)
    for num, c in counts.items():
        if c == 1:
            return num

Why optimize? The hash map is linear time but uses O(n) extra memory to store a count for every distinct value. The problem explicitly invites constant extra space, so we want a technique that reads the answer directly out of the numbers themselves.

Key Insight Look at one bit position across all numbers. Every value that repeats three times contributes that bit three times, so summing a single bit column over the whole array and taking the result modulo 3 cancels the triples and leaves exactly the bit of the unique number. Do that for all 32 bit positions and you reconstruct the answer with no map.
How to Think About It
  1. Forget the numbers as wholes; think of them as columns of bits, one column per bit position.
  2. In any column, each tripled value adds either 0 or 3 to the column sum, so the tripled contributions are all multiples of 3.
  3. Taking the column sum modulo 3 wipes out those multiples and leaves only the unique number's bit (0 or 1) in that column.
  4. Repeat across every bit position (32 for standard integers) and OR the surviving bits back together.
  5. Alternatively, model the same counting with two rolling bitmasks (ones/twos) that advance through the states "seen once / seen twice / reset on third".
  6. Handle the sign bit carefully so a negative answer is reconstructed correctly.
Why This Works Addition in a single bit column is just counting how many numbers have that bit set. Because every value except one appears a multiple of three times, every column's count is congruent modulo 3 to the unique number's bit. Modulo 3 is the exact inverse of the k-fold (k=3) repetition, so the leftover bits reassemble precisely the element that broke the pattern.
Watch Out For
  • The sign bit: in fixed-width languages the 32nd bit carries the sign, so reconstruct as a signed 32-bit integer.
  • Python has arbitrary-precision ints — mask with 0xFFFFFFFF and sign-extend when the top bit is set.
  • Shifts must be unsigned/logical when reading bit values, not sign-propagating arithmetic shifts.
  • Use modulo 3, not modulo 2 — this is the k=3 variant, not classic XOR.
  • Overflow: summing a bit column across a large array can exceed a small int type if you are not careful.
