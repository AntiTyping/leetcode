The ProblemYou climb a staircase of n steps, taking either 1 or 2 steps at a time. Return the number of distinct ways to reach the top.

#101 Symmetric Tree EasyTree / DFS-BFS

The ProblemGiven the root of a binary tree, return true if the tree is a mirror of itself around its center — i.e. the left subtree is the reflection of the right subtree.
Brute Force — O(n) time, O(n) space

Walk the tree level by level with BFS. For each level, collect the node values, inserting an explicit placeholder (e.g. None) for missing children so structure is preserved. Then check that each level's list of values reads the same forwards and backwards (is a palindrome). If every level is symmetric, the tree is symmetric.

Why optimize? The level-buffering with null placeholders is correct but clumsy — the recursion below expresses the mirror property directly and uses only the call stack.

Key InsightSymmetry is not a property of a single node — it is a property of a pair of nodes. Instead of comparing a level against its reverse, compare the two subtrees against each other, walking one left-to-right and the other right-to-left in lockstep.
How to Think About It
  1. Define a helper isMirror(a, b) that answers: are these two subtrees reflections of each other?
  2. Base case: if both a and b are null they mirror (true); if exactly one is null they cannot (false).
  3. Otherwise the values must match, AND a.left must mirror b.right, AND a.right must mirror b.left — note the crossed recursion.
  4. Kick it off with isMirror(root.left, root.right) (an empty tree is trivially symmetric).
Why This WorksA tree is symmetric exactly when its left and right subtrees are mirror images. Two subtrees are mirror images precisely when their roots are equal and each subtree's outer child mirrors the other's outer child (left-vs-right) and inner mirrors inner (right-vs-left). The recursion asserts exactly this pair of conditions at every level, so it returns true only when the full reflection holds structurally and by value.
Watch Out For
  • Comparing a.left to b.left instead of the crossed a.left to b.right — that checks equality, not mirroring.
  • The one-null-one-nonnull case: you must return false, not treat it as equal.
  • Forgetting that structure matters, not just the multiset of values — two null placeholders differ from two real nodes.

The ProblemGiven the root of a binary tree and an integer targetSum, return true if there exists a root-to-leaf path such that adding up all the values along the path equals targetSum. A leaf is a node with no children.
Brute Force — O(n) time, O(n²) worst-case work

Key InsightYou never need the whole path. Subtract each node's value from the target as you descend; the question "does a path sum to targetSum?" becomes "does a path in this subtree sum to remaining?" — a smaller instance of the same problem.
How to Think About It

  2. At a leaf (both children null), the path is complete: return whether remaining == 0.
  3. At an internal node, return true if either child subtree yields a valid path for the reduced remaining.

Why This WorksSubtracting node values as you descend keeps an invariant: remaining is exactly the amount the rest of the current path must still contribute. #542 01 Matrix MediumGraph / BFS / DP

The ProblemGiven an m × n binary matrix mat, return a matrix of the same size where each cell holds the distance to the nearest cell containing 0. Distance is the number of steps between adjacent cells (up/down/left/right).
Brute Force — O((mn)²) time

For each cell, run an independent BFS outward until you hit the first 0, recording how many steps that took. Cells that are already 0 get distance 0 immediately. This repeats a full grid search from every starting cell.

Python
from collections import deque
def updateMatrix(mat):
    m, n = len(mat), len(mat[0])
    res = [[0]*n for _ in range(m)]
    for i in range(m):
        for j in range(n):
            if mat[i][j] == 0: continue
            seen = {(i, j)}
            q = deque([(i, j, 0)])
            while q:
                r, c, d = q.popleft()
                if mat[r][c] == 0:
                    res[i][j] = d; break
                for dr, dc in ((1,0),(-1,0),(0,1),(0,-1)):
                    nr, nc = r+dr, c+dc
                    if 0<=nr<m and 0<=nc<n and (nr,nc) not in seen:
                        seen.add((nr,nc)); q.append((nr,nc,d+1))
    return res

Why optimize? Re-running BFS from every one of mn cells is O((mn)²) — with a large grid that is far too slow. One combined search covers all cells at once.

Key InsightReverse the direction of thought: instead of asking each 1 to search for a 0, let all the 0s search outward together. Seed a single BFS queue with every zero at distance 0; the wavefront that expands from them reaches each 1 at exactly its nearest-zero distance.
How to Think About It
  1. Initialize distances: every 0 cell is 0; every 1 cell is "unknown" (mark as infinity / unvisited).
  2. Push all zero cells into one queue simultaneously — this is the multi-source seed.
  3. Run standard BFS: pop a cell, and for each unvisited neighbor set its distance to current+1 and enqueue it.
  4. Because BFS expands in distance order from the whole set of sources, the first time a cell is assigned is its true minimum distance.
Why This WorksConceptually, add a virtual super-source connected to every zero with a zero-cost edge; the nearest-zero distance for any cell equals its BFS distance from that super-source. Multi-source BFS simulates this: since all sources start at level 0 and BFS processes cells in nondecreasing distance, the moment a cell is first dequeued/marked it has been reached by the closest source. Marking-on-enqueue prevents any later, longer path from overwriting it.
Watch Out For
  • Seeding the queue with only one zero, or running per-cell BFS — you must enqueue all zeros first.
  • Marking a cell visited only when you pop it (instead of when you enqueue it) can let duplicates inflate the queue; mark on enqueue.
  • Off-by-one on bounds and forgetting all four directions.
  • If you use the two-pass DP alternative, remember it needs both a top-left and a bottom-right sweep.

Why optimize? Backtracking DFS re-explores the same cells along countless path prefixes; on an open maze that is exponential, whereas BFS visits each cell once.

Key Insight"Minimum steps in an unweighted grid" is the textbook signal for BFS. Because BFS expands strictly in increasing distance, the first border exit it pops is guaranteed to be the nearest one — no need to enumerate or compare full paths.
How to Think About It
  1. Start a single-source BFS from the entrance with distance 0; mark the entrance visited immediately.
  2. Pop a cell, and for each of the four in-bounds, non-wall, unvisited neighbors, enqueue it at distance+1 and mark visited.
  3. When you enqueue (or pop) a cell, test if it is on the border and not the entrance — if so, its distance is the answer; return it right away.
  4. If the queue empties with no exit found, return -1.
Why This WorksIn an unweighted graph, BFS processes vertices in nondecreasing order of distance from the source, so the distance recorded when a cell is first reached is its shortest-path length. Every reachable empty cell is enqueued exactly once (marking on enqueue prevents revisits), making the search linear in the grid. The first border-exit cell encountered therefore has the minimum possible step count; exhausting the queue without finding one proves no exit is reachable.
Watch Out For
  • Counting the entrance itself as an exit — it is on the border but explicitly excluded.
  • Not marking cells visited (or marking on pop rather than enqueue), which lets BFS revisit and even loop.
  • Treating walls '+' as passable, or reading entrance as [col, row] instead of [row, col].
  • Returning the entrance's own distance (0) — the first level you should accept as an exit is at least 1 step away.

  2. For each spell s, compute the minimum passing potion value need = ceil(success / s), using integer math: (success + s - 1) / s.
  3. Binary-search (bisect_left / lower_bound) for the first potion >= need; call its index idx.
  4. The count of qualifying potions is m - idx. Append and move on.
Why This WorksSorting turns "product ≥ success" into a threshold on a single monotone axis. Every potion at index ≥ idx is ≥ need, so its product with s is ≥ success; every potion before idx falls short. The ceiling division ensures we don't over- or under-count the exact boundary value. Total work is dominated by n binary searches, each O(log m).
Watch Out For
  • Overflow: spell * potion can exceed 32-bit range — use 64-bit (Go int64, C++ long long, Rust i64).
  • Ceiling division, not floor: use (success + s - 1) / s, or bisect on the product directly to sidestep the rounding.
  • Comparing against need uses >=; bisect_left / lower_bound gives the first index where potion >= need, which is what you want.

Key InsightUse two stacks with dedicated roles: an in stack for pushes and an out stack for pops/peeks. Only transfer from in to out when out is empty. Each element moves between stacks at most once, giving amortized O(1) per operation.
How to Think About It

The ProblemGiven pushed and popped, both permutations of the same distinct values, return true if some sequence of push and pop operations on an initially empty stack could produce popped as the pop order.

def validateStackSequences(pushed, popped):
    def bt(i, j, stack):
        if j == len(popped):
            return True
        # option 1: pop if top matches
        if stack and stack[-1] == popped[j]:
            if bt(i, j + 1, stack[:-1]):

        # option 2: push next
        if i < len(pushed):
            if bt(i + 1, j, stack + [pushed[i]]):
                return True
        return False
    return bt(0, 0, [])

Key InsightSimulate with one real stack. Push values from pushed in order; after each push, greedily pop while the stack top equals the next expected value in popped. If the whole simulation consumes both arrays and leaves an empty stack, the sequence is valid.
How to Think About It
  1. Maintain a stack and a pointer j into popped (next value we must pop).
  2. Iterate over pushed: push each value onto the stack.

Why optimize? The count of a range depends only on its size, so recomputing it for every recursion path is wasted work that grows like the Catalan numbers.

Key InsightThe number of unique BSTs depends only on how many nodes there are, not which specific values, so f(size) can be tabulated once. Choosing root k splits size i into a left of size k-1 and a right of size i-k, giving the Catalan recurrence.
How to Think About It
  1. Define dp[i] as the number of unique BSTs holding i nodes; set dp[0] = dp[1] = 1.
  2. For each size i from 2 to n, try every position k (1..i) as the root.
  3. A root at k leaves k-1 nodes on the left and i-k on the right, contributing dp[k-1] * dp[i-k].
  4. Sum those products into dp[i]; the answer is dp[n].
Why This WorksEvery BST has exactly one root, and the left and right subtrees are independent BSTs of fixed sizes. Summing over all root choices counts each distinct structure exactly once, matching the Catalan number C(n).
Watch Out For
  • The base case dp[0] = 1 (an empty subtree is one valid arrangement), not 0.
  • Off-by-one: a root at k splits into k-1 and i-k, so indices must align precisely.
  • Values only affect ordering, never the count — do not overthink which numbers go where.

How to Think About It

  2. For each i, try every first part j from 1 to i-1.
  3. Compare j*(i-j) (leave the remainder whole) against j*dp[i-j] (break the remainder further).

  • The "at least two parts" rule means dp[i] itself must not equal i for the final answer — you must split at least once.
  • Small cases: n=2 gives 1, n=3 gives 2, because you are forced to split even though the whole number is larger.
  • Greedy remainder handling: when n mod 3 == 1, back off one 3 and use a 4 (2+2), not a lone 1.

Why optimize? Each start re-scans overlapping runs; a single left-to-right pass can reuse the length of the arithmetic run ending at the previous index.

Key InsightIf the arithmetic run ending at index i-1 contributed some count, then extending it by one more element with the same difference adds exactly that many new slices plus one. Track a running "current" that grows while the difference holds and resets when it breaks.
How to Think About It
  1. Keep a running variable cur for the number of arithmetic slices ending exactly at the current index.
  2. Scan from index 2 onward. If nums[i]-nums[i-1] equals nums[i-1]-nums[i-2], the run extends.
  3. On extension, do cur += 1 and add cur to the total; otherwise reset cur to 0.
  4. Return the accumulated total after the single pass.
Why This WorksA new element that continues the arithmetic progression forms a new slice ending here for every slice that ended at the previous index, plus the fresh length-3 slice. So cur increments by one and each cur is the count of valid slices ending at i; summing all cur values counts every arithmetic subarray exactly once.
Watch Out For
  • Reset cur to 0 the moment the difference breaks, or stale counts leak forward.
  • Arrays shorter than 3 yield 0 — guard the loop bounds.
  • Compare consecutive differences (nums[i]-nums[i-1] vs nums[i-1]-nums[i-2]), not a fixed global difference.

#83 Remove Duplicates from Sorted List EasyLinked List

The ProblemGiven the head of a sorted linked list, delete all duplicates so that each value appears only once. Return the modified list, which remains sorted.
Brute Force — O(n) time, O(n) space

Walk the list once collecting the first occurrence of each value into a set, and build a brand-new list (or array) of the unique values in order. Because the list is sorted, a value is a duplicate exactly when it equals the previously kept value. This uses extra memory for the set and new nodes.

Python
def deleteDuplicates(head):
    seen = set()
    dummy = ListNode(0)
    tail = dummy
    node = head
    while node:
        if node.val not in seen:
            seen.add(node.val)
            tail.next = ListNode(node.val)
            tail = tail.next
        node = node.next
    return dummy.next

Why optimize? The list is already sorted, so duplicates are adjacent — you can splice them out in place with no set and no new nodes.

Key InsightBecause the list is sorted, equal values sit next to each other. A single pointer at the current kept node can look at its next node: if they match, unlink the next node by skipping over it; otherwise advance.
How to Think About It
  1. Start a pointer cur at head.
  2. While cur and cur.next both exist, compare their values.
  3. If cur.next.val equals cur.val, set cur.next = cur.next.next to drop the duplicate (do not advance cur yet).
  4. Otherwise advance cur = cur.next. Return the original head.
Why This WorksSorting guarantees all copies of a value form one contiguous block. Repeatedly skipping cur.next while it equals cur removes the entire block, leaving one representative. Not advancing after a removal ensures a run of three or more equal values is fully collapsed.
Watch Out For
  • Do not advance cur on the same step you remove a node, or you skip past a possible third duplicate.
  • Handle an empty list (head is null) up front.
  • Always guard cur.next before reading cur.next.val to avoid a null dereference.

#328 Odd Even Linked List MediumLinked List

The ProblemGiven the head of a singly linked list, group all nodes at odd positions together followed by nodes at even positions (positions are 1-indexed by node, not by value). Preserve relative order within each group and use O(1) extra space.
Brute Force — O(n) time, O(n) space

Walk the list once, pushing nodes at odd positions into one array and even positions into another. Then relink: chain the odd-position nodes in order, followed by the even-position nodes. This ignores the O(1) space requirement by materializing helper arrays.

Python
def oddEvenList(head):
    if not head:
        return head
    odd, even = [], []
    i = 1
    node = head
    while node:
        (odd if i % 2 else even).append(node)
        node = node.next
        i += 1
    order = odd + even
    for a, b in zip(order, order[1:]):
        a.next = b
    order[-1].next = None
    return order[0]

Why optimize? The problem demands O(1) space, and you can weave the two chains together with a handful of pointers instead of storing every node.

Key InsightMaintain two travelling pointers, odd and even, plus a saved evenHead. Each step, odd jumps over the even node to the next odd node, and even jumps over that to the next even node. When both chains are built, hook the odd tail onto evenHead.
How to Think About It
  1. Set odd = head, even = head.next, and remember evenHead = even.
  2. While even and even.next exist: odd.next = even.next, then advance odd = odd.next.
  3. Then even.next = odd.next, and advance even = even.next.
  4. After the loop, connect odd.next = evenHead and return head.
Why This WorksEach iteration relinks one odd node to the following odd node and one even node to the following even node, preserving original relative order within each group. The loop stops when no further even node exists, and attaching evenHead after the last odd node yields all odds followed by all evens.
Watch Out For
  • Save evenHead before you start rewiring, or you lose the start of the even chain.
  • Loop condition must check even and even.next; getting it wrong causes null dereferences on odd-length lists.
  • Empty or single-node lists should return immediately unchanged.

Slide a variable-width window across the array, advancing the left edge to restore an invariant instead of restarting. Interleaved.

#713 Subarray Product Less Than K MediumSliding Window

The ProblemGiven an array of positive integers nums and an integer k, count the contiguous subarrays whose product of all elements is strictly less than k. Values are positive; return the count.
Brute Force — O(n^2) time, O(1) space

Fix each start index and extend to the right, multiplying elements into a running product. Every time the product stays below k, increment the count; stop extending as soon as it reaches k (since positive values only grow the product). Repeating for every start gives quadratic work.

Python
def numSubarrayProductLessThanK(nums, k):
    count = 0
    n = len(nums)
    for i in range(n):
        prod = 1
        for j in range(i, n):
            prod *= nums[j]
            if prod < k:
                count += 1
            else:
                break
    return count

Why optimize? Because all values are positive, the product is monotonic as the window grows, so a two-pointer window can amortize the scan to linear time.

Key InsightMaintain a window [left, right] with a running product. As right advances, multiply in nums[right]; while the product is at least k, divide out nums[left] and move left forward. Every valid window ending at right contributes (right - left + 1) new subarrays.
How to Think About It
  1. Handle the trivial case: if k ≤ 1 no product of positives can be below it, so return 0.
  2. Slide right from 0 to n-1, multiplying nums[right] into prod.
  3. While prod ≥ k, divide prod by nums[left] and increment left to shrink the window.
  4. Add (right - left + 1) to the count — the number of subarrays ending at right with product below k.
Why This WorksWith positive numbers the product only increases when you extend right and only decreases when you shrink left, so the window boundary moves monotonically. For each right, the window [left, right] is the largest valid one ending at right, and every subarray ending at right that starts at or after left is valid — there are exactly (right - left + 1) of them, counted once each.
Watch Out For
  • The k ≤ 1 guard is essential; otherwise the while loop can push left past right and misbehave.
  • Use strict less-than (< k), and shrink while prod ≥ k, matching the "strictly less" requirement.
  • This window trick relies on all-positive values; it breaks if zeros or negatives are allowed.
