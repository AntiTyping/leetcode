Quick ReminderBinary search comparing the midpoint against the right endpoint tells you which half contains the rotation point (the minimum).

class MinHeap {
    private a: number[] = [];
    size(): number { return this.a.length; }
    peek(): number { return this.a[0]; }
    push(v: number): void {
        this.a.push(v);
        let i = this.a.length - 1;
        while (i > 0) {
            const p = (i - 1) >> 1;
            if (this.a[p] <= this.a[i]) break;
            [this.a[p], this.a[i]] = [this.a[i], this.a[p]];
            i = p;
        }
    }
    pop(): number {
        const top = this.a[0], last = this.a.pop()!;
        if (this.a.length) {
            this.a[0] = last;
            let i = 0;
            const n = this.a.length;
            while (true) {
                let s = i, l = 2 * i + 1, r = 2 * i + 2;
                if (l < n && this.a[l] < this.a[s]) s = l;
                if (r < n && this.a[r] < this.a[s]) s = r;
                if (s === i) break;
                [this.a[s], this.a[i]] = [this.a[i], this.a[s]];
                i = s;
            }
        }
        return top;
    }
}

Why optimize? The hash map allocates O(n) memory; the problem explicitly demands constant extra space.

Key InsightXOR is its own inverse: a ^ a == 0 and a ^ 0 == a, and it is commutative and associative. If you XOR every element together, each pair cancels to zero and only the lone element survives. The hash map's redundancy was storing counts you never needed; XOR carries the exact information required in a single accumulator.
How to Think About It
  1. Recall the algebraic properties of XOR that let identical values annihilate.
  2. Decide on a single integer accumulator initialized to 0.
  3. Fold every array element into the accumulator with XOR in one pass.
  4. Return the accumulator; the paired elements have all cancelled.
Why This WorksBecause XOR is commutative and associative, you can reorder the fold so each duplicate pair is adjacent. Each pair XORs to 0, and XORing 0 leaves the running value unchanged, so after processing everything only the unique element remains combined with a chain of zeros, which equals the element itself.
Watch Out For
  • Initializing the accumulator to something other than 0 (e.g. nums[0]) and then also XORing nums[0] again, double-counting it.
  • Assuming the trick works when elements appear three times instead of twice — plain XOR only handles even multiplicities.
  • Overflow worries are unfounded here since XOR is bitwise, not arithmetic.

Why optimize? The hash map costs O(n) memory, which the constant-space requirement forbids.

Key InsightXOR the whole array to get a ^ b, where a and b are the two unique numbers. Since a ≠ b, that combined value has at least one set bit — any set bit is a position where a and b differ. Use that single distinguishing bit as a partition: split every element into two buckets by whether that bit is set. Each unique number lands in a different bucket, and within each bucket the duplicates still pair off, so you have reduced the problem to two independent instances of #136.
How to Think About It
  1. XOR all elements to obtain xorAll = a ^ b.
  2. Isolate one set bit of xorAll, e.g. the lowest with xorAll & (-xorAll).
  3. Partition elements into those where that bit is set versus not, XORing each group into its own accumulator.
  4. The two accumulators are exactly a and b; return them.
Why This WorksThe chosen bit differs between a and b, so they fall into opposite groups. Every duplicated value has the same bit in both copies, so both copies land in the same group and cancel via XOR. Thus each group's XOR reduces to the single unique element it contains, recovering a and b independently.

Key InsightStore words in a trie keyed by character (exactly the structure from #208). Exact-letter searches walk a single path. Key InsightThis problem combines a Trie (the structure from #208/#211) with grid DFS backtracking. Build one trie from all the words, then DFS the board just once, walking the trie in lockstep with your path: at each cell descend to the trie child for that letter. If no child exists, prune the entire branch immediately — that shared-prefix pruning is exactly what the per-word brute force lacked. When you reach a trie node marked as a word end, record that word.
How to Think About It
  1. Insert every word into a trie, storing the full word (or an end flag) at terminal nodes.
  2. DFS from each board cell, carrying the current trie node alongside the cell.
  3. From a cell, only step to a neighbor whose letter is a child of the current trie node; otherwise prune.
  4. Mark cells visited during the path and unmark on backtrack; when a trie node is a word end, add that word to the results (and de-duplicate, e.g. by clearing the end marker).
Why This WorksEvery valid board path corresponds to walking down the trie one letter at a time, so a path can only extend a stored word's prefix. Because a missing trie child means no word continues with that letter, pruning there discards only paths that could never complete a word. Visiting the board once while the trie encodes all words replaces W independent searches with a single shared traversal.
Watch Out For
  • Adding a word multiple times when it can be formed via several paths — clear the end marker (or use a set) after finding it.
  • Forgetting to restore the board cell / visited marker on backtrack, corrupting other paths.
  • Not pruning dead trie branches (an optional but important optimization: removing leaf nodes after a word is found keeps the search fast).
  • Off-by-one or bounds errors on the four-directional moves.

Key InsightGrow a combination one digit at a time in increasing order, carrying the remaining target and the count still needed. The moment the running choices cannot possibly succeed — the next smallest digit already exceeds the remaining sum, or too few digits are left to reach k — abandon that branch. How to Think About It
  1. Define state: current partial list, next allowed start digit, remaining sum, remaining count k'.
  2. Base case: if remaining count is 0 and remaining sum is 0, record the combination; if either alone is exhausted, stop.
  3. Loop the next digit from the start value upward, but stop early once the digit exceeds the remaining sum.
  4. Choose the digit (append, recurse with digit+1, reduced sum and count), then un-choose it (pop) before trying the next.
Why This WorksBecause digits are chosen strictly increasing, every combination is generated in exactly one canonical order, so there are no duplicates and no need to reconsider used digits. The pruning only discards branches whose partial sum or remaining capacity already makes success impossible, so no valid combination is ever skipped.

  • Allowing a digit to repeat or restarting the loop from 1, which produces duplicates and out-of-range values above 9.
  • Forgetting to un-choose (pop) after recursion, corrupting sibling branches.
  • Checking only the sum and not that exactly k numbers were used (or vice versa).
  • Capping digits above 9 — the choice range is fixed to 1..9.

            if (t[i] != t[j]) return false;

        parts.push_back(s.substr(prev));

Why optimize? It validates a full partition only after building all pieces, so it never prunes a bad first piece and re-tests overlapping substrings repeatedly.

Key InsightBuild the partition left to right: from the current index, try each prefix that is a palindrome as the next piece, then recurse on the remainder. The brute force wastes effort validating whole splits whose very first piece is not a palindrome; here, if a prefix fails the palindrome test you never explore any split that starts with it. Optionally precompute a palindrome table so each check is O(1).
How to Think About It
  1. Track a start index and the current list of chosen palindromic pieces.
  2. Base case: if start reaches the end of s, record a copy of the current pieces.
  3. For each end index from start onward, test whether s[start..end] is a palindrome.
  4. If it is, add that substring, recurse from end+1, then remove it and try a longer prefix.
Why This WorksEvery partitioning is a sequence of contiguous pieces, and the recursion considers every possible first palindromic piece, then solves the strictly smaller remaining suffix the same way. Because a non-palindromic prefix is never chosen, only valid partitions are ever extended, and every valid partition is reachable by exactly one sequence of prefix choices.
Watch Out For
  • Storing references to the mutable path instead of a copy, so all results alias the same list.
  • Off-by-one on substring bounds — be consistent about inclusive/exclusive end.
  • Re-checking the same substring's palindrome-ness repeatedly; a precomputed DP table of isPal[i][j] avoids O(L) checks inside the recursion.
  • Handling single characters (always palindromes) and the empty remainder correctly.

Why optimize? It explores all n^n column assignments and only rejects conflicts at the leaves, so it wastes exponential work on partial rows that were already invalid.

Key InsightPlace queens one row at a time and check conflicts as you place, not at the end. Track which columns and which two diagonal families are occupied; a queen at (r,c) uses column c, the “down-right” diagonal r-c, and the “down-left” diagonal r+c. If placing here conflicts, skip the column immediately — this prunes the whole subtree the brute force would still have expanded.
How to Think About It
  1. Recurse row by row; the row index is the recursion depth so no two queens ever share a row automatically.
  2. Maintain three occupancy sets: used columns, used r-c diagonals, used r+c anti-diagonals.
  3. For the current row, try each column that is free in all three sets; mark it, recurse to the next row, then unmark.
  4. When you place a queen in the last row successfully, translate the column assignment into the board strings and record it.
Why This WorksEvery pair of queens attacks along a row, column, or one of the two diagonal directions. Placing one per row removes row conflicts by construction; the three sets guarantee no column or diagonal is reused. Because a conflicting placement is rejected before recursing, only configurations that are valid so far are ever extended, and every complete valid board is reachable by choosing a legal column in each row.
Watch Out For
  • Mixing up the two diagonal keys; r-c can be negative, so offset it (add n) if you use array indices instead of a hash set.
  • Forgetting to un-mark all three sets on backtrack.
  • Building the output board incorrectly — each row string must have exactly one 'Q'.
  • Handling n=1 (one solution) and n=2,3 (zero solutions) correctly.

Why optimize? Sorting is O(n log n) and orders elements you will throw away; when k is much smaller than n you are doing far more work than the task requires.

Key InsightYou do not need the k closest points in sorted order, only the set of them. Keep a max-heap of size k keyed by squared distance: push each point, and whenever the heap exceeds k, pop the farthest. The heap's root is always the current k-th closest, so each point is compared only against that boundary rather than against all others. Compare squared distances to avoid floating-point square roots. (QuickSelect gives an even better average O(n), but the heap is the natural streaming choice.)
How to Think About It
  1. Define distance as x*x + y*y (no square root needed for comparison).
  2. Maintain a max-heap holding at most k points, ordered so the farthest is on top.
  3. For each point, push it; if the heap size exceeds k, pop the top (the current farthest).
  4. After processing all points, the heap contains exactly the k closest; return its contents.
Why This WorksAt every step the heap holds the k smallest distances seen so far, because whenever a (k+1)-th element arrives you discard the largest of the group, which can never be among the k smallest overall. After all points are processed, the retained k are exactly the globally closest, independent of insertion order.
Watch Out For
  • Using a min-heap and popping the closest by mistake — you must evict the farthest, so the size-k heap must be a max-heap.
  • Taking real square roots, which wastes time and can introduce precision issues; squared distance suffices.
  • Integer overflow in x*x + y*y for large coordinates in fixed-width languages — use a 64-bit type.
  • Handling k equal to n (return everything).

The ProblemGiven a list of CPU tasks (letters) and an integer cooldown n, each task takes one time unit and two identical tasks must be separated by at least n intervals. The CPU may sit idle. Return the minimum total time units needed to finish all tasks.
Brute Force — O(total · 26) simulation

Simulate tick by tick. At each time unit pick the available task with the highest remaining count that is off cooldown; if none is available, idle. Track each task's next-allowed time. This works but steps through every single time unit including idles.

Python
from collections import Counter
def leastInterval(tasks, n):
    counts = Counter(tasks)
    ready = {t: 0 for t in counts}     # next time each task may run
    time = 0
    remaining = dict(counts)
    while any(v > 0 for v in remaining.values()):
        cand = [t for t in remaining if remaining[t] > 0 and ready[t] <= time]
        if cand:
            t = max(cand, key=lambda x: remaining[x])
            remaining[t] -= 1
            ready[t] = time + n + 1
        time += 1
    return time

Why optimize? The answer can be much larger than the number of tasks due to idles, and stepping through every tick (rescanning 26 tasks each) is wasteful when a formula or batched heap can skip idle runs.

Key InsightThe most frequent task dictates the skeleton. With max frequency f, arrange those f copies into f-1 gaps each of width n+1, giving a frame of (f-1)*(n+1) plus the tasks that tie for the max in the final block. The heap/greedy view: always run the highest-remaining-count ready task, processing a full window of n+1 slots at a time and putting tasks back into the heap after cooldown. The redundancy the brute force has is re-picking tick by tick; here you reason about whole cooldown windows.
How to Think About It
  1. Count each task's frequency; identify the maximum frequency and how many tasks share it.
  2. Greedy/heap view: put counts in a max-heap; each round, pop up to n+1 tasks, decrement, and hold them aside until the round's cooldown passes.
  3. Re-add tasks that still have remaining count back into the heap after the window.
  4. Sum the time; the answer is the max of the total number of tasks and the computed frame width (idles fill the rest).
Why This WorksSpacing the most frequent task optimally forces a minimum number of slots: no schedule can be shorter than the frame built around it, because those copies alone require (f-1) gaps of size n between them. Filling each gap greedily with the next most frequent tasks never increases length, and if there are more tasks than the frame can absorb, they pack in without adding idle time, so the answer is the larger of the two quantities.
Watch Out For
  • Forgetting the tie case: several tasks sharing the max frequency each add one to the final block.
  • Returning the frame width when the total task count is actually larger (n small, many distinct tasks) — take the maximum of the two.
  • In the heap simulation, re-adding a task before its cooldown elapses.
  • Off-by-one in the window size — a cooldown of n means a window of n+1 slots.

Why optimize? Sorting all N values is O(N log N) and discards the per-list sortedness; using that structure you can merge in O(N log k), which is better since k ≤ N.

Key InsightAt any moment the next node of the merged list is the smallest among the current heads of the k lists. A min-heap holding one node per list gives that smallest in O(log k): pop it, append it, and push its successor. You only ever compare k candidates at a time instead of re-sorting everything. (Divide-and-conquer pairwise merging achieves the same O(N log k) without a heap.)
How to Think About It
  1. Seed a min-heap with the head node of every non-empty list, keyed by node value.
  2. Repeatedly pop the smallest node, append it to the output tail.
  3. If the popped node has a next, push that successor into the heap.
  4. Continue until the heap is empty; the appended nodes form the fully merged sorted list.
Why This WorksEach list is sorted, so a list's smallest unmerged value is always its current head. The heap therefore always contains the candidate minimums, and its root is the global minimum of all unmerged values. Popping it and pushing its successor preserves this invariant, so nodes are emitted in nondecreasing order until every node is consumed.
Watch Out For
  • Pushing null/empty list heads into the heap, or dereferencing a null successor.
  • Needing a tie-breaker so the heap never tries to compare node objects directly (include a unique index in the key).
  • Losing the tail pointer — use a dummy head to simplify appending.
  • Forgetting to terminate the final node's next (it may still point into an old list).

The ProblemYou are given an n x n matrix isConnected where isConnected[i][j] == 1 means city i and city j are directly connected, and 0 means they are not. Brute Force — O(n^2) time via flood fill

Treat the matrix as an adjacency matrix. Walk every city; when you find one not yet visited, run a DFS/BFS that marks every city reachable from it, and count one province. Repeat until all cities are visited. Each cell of the matrix is examined once during traversal.

Why optimize? The flood fill is fine here, but if connections streamed in as edges you would rerun the whole traversal on every query; a structure that merges groups incrementally answers "how many groups" in near-constant amortized time.

Key InsightCounting connected components is exactly what Union-Find is built for. Start with n separate components; every edge that joins two different components decreases the count by one. Instead of re-exploring the graph, maintain a parent pointer per city and merge sets as you scan the matrix. The final answer is the number of distinct set roots.
How to Think About It
  1. Initialize each city as its own parent (n singleton sets) and a component counter equal to n.
  2. Scan only the upper triangle of the matrix (j > i) since it is symmetric.
  3. For every pair with isConnected[i][j] == 1, find the roots of i and j; if they differ, union them and decrement the counter.
  4. Return the counter (or, alternatively, count distinct roots at the end).
Why This WorksEach union merges two previously separate groups into one, so the invariant "counter == number of connected components among processed edges" is preserved. Because union only fires when roots differ, indirect connectivity (i-j and j-k) collapses transitively into one root, matching the definition of a province. Path compression and union by rank keep find nearly O(1) amortized.
Watch Out For
  • The diagonal isConnected[i][i] == 1 is a self-loop; skip it or it wastes a no-op union.
  • Only decrement the counter when the two roots are actually different, otherwise you overcount merges.
  • Forgetting path compression on 200 nodes still passes, but the habit matters when n is large and edges are many.

#684 Redundant Connection MediumUnion-FindGraphCycle

The ProblemYou start with a tree of n nodes labeled 1..n (a connected acyclic graph) and one extra edge was added, creating exactly one cycle. You are given edges where edges[i] = [a, b]. Return the one edge that can be removed so the result is a tree of n nodes. If multiple answers exist, return the edge that occurs last in the input. Constraints: n == edges.length, 3 <= n <= 1000.
Brute Force — O(n^2) time via connectivity check

For each edge, tentatively add it and test whether its two endpoints were already connected using the edges seen so far. Do that connectivity test with a fresh DFS/BFS over the graph built from prior edges. The first edge whose endpoints are already reachable from each other is the redundant one; because we process edges in order, this is the last such edge.

Python
def findRedundantConnection(edges):
    n = len(edges)
    adj = {i: [] for i in range(1, n + 1)}
    def connected(a, b):
        stack, seen = [a], {a}
        while stack:
            node = stack.pop()
            if node == b:
                return True
            for nxt in adj[node]:
                if nxt not in seen:
                    seen.add(nxt)
                    stack.append(nxt)
        return False
    for a, b in edges:
        if connected(a, b):
            return [a, b]
        adj[a].append(b)
        adj[b].append(a)

Why optimize? Each edge triggers a full graph traversal costing O(n), making the whole scan O(n^2); a structure that answers "are these already connected?" in near-constant amortized time collapses this to nearly linear.

Key InsightAn edge is redundant precisely when its two endpoints already belong to the same connected component. Union-Find answers "already connected?" as a byproduct of the union call: process edges in input order and attempt to union each edge's endpoints. The first union that fails (both endpoints share a root) is the redundant edge, and because you scan in order it is automatically the last valid answer. No repeated traversal needed.
How to Think About It
  1. Create a DSU over nodes 1..n, each its own parent.
  2. Iterate edges in the given order; for edge [a, b] call find on both.
  3. If find(a) == find(b), the endpoints are already connected — this edge closes the cycle, so return it immediately.
  4. Otherwise union them and continue. The single guaranteed cycle ensures exactly one edge triggers the return.
Why This WorksProcessing edges of a would-be tree, every edge normally links two distinct components (one merge, no cycle). Since the graph has exactly one extra edge and thus exactly one cycle, exactly one edge will connect two nodes already in the same set. Returning that edge removes the cycle, leaving n-1 tree edges over n nodes. Input-order scanning guarantees it is the last-occurring valid removal.
Watch Out For
  • Nodes are 1-indexed; size your parent array as n+1 or offset by one to avoid off-by-one bugs.
  • Return the edge as given (the [a, b] pair from input), not the roots.
  • Do not union first and then check — check whether the roots match before merging, otherwise you lose the signal.

Key InsightEvery [ opens a new scope whose result depends on a repeat count and a partial string built so far.   2. On a digit, fold it into k with k = k*10 + digit (handles multi-digit counts).

Key InsightPrecedence without parentheses reduces to one rule: * and / must combine with their immediate neighbor before any +/- is resolved. How to Think About It
  1. Track the current number being built and the operator that preceded it (initialize the previous operator to +).
  2. Scan characters, accumulating digits into the current number; skip spaces.
  3. When you hit an operator (or the end of string), act on the previous operator: + push num, - push -num, * pop and push top*num, / pop and push trunc(top/num).
  4. Then set the previous operator to the current one, reset the number to 0, and continue. At the end, sum the stack.

Watch Out For

  • Spaces can appear anywhere; skip them without resetting state.

Why optimize? The two-path approach makes two full traversals and stores O(h) path lists; a single post-order pass finds the LCA in one traversal with only the recursion stack.

Key InsightSolve it in a single post-order recursion. Ask each subtree: "do you contain p or q?" A node is the LCA exactly when p and q are found on different sides beneath it (one in the left subtree, one in the right), or when the node itself is one of them and the other lies below. The redundancy the brute force has is materializing full paths; instead, let each recursive call return a compact signal (found node or null) and let the parent combine the two child signals.
How to Think About It
  1. Define a recursion that returns a node: null if neither target is in this subtree, otherwise a "found" signal bubbling up.
  2. Base case: if the current node is null, or equals p, or equals q, return the current node (you found a target or hit the bottom).
  3. Recurse into left and right and capture both results.
  4. If both left and right returned non-null, the current node is the split point — return it as the LCA. If only one side is non-null, propagate that side upward.
Why This WorksPost-order guarantees children are resolved before the parent decides. The first node (deepest, since recursion returns bottom-up) that sees both targets in different subtrees is by definition the lowest common ancestor. If a node is itself a target, returning it lets an ancestor detect the second target on the opposite side, correctly handling the "a node is a descendant of itself" case. Because both nodes are guaranteed to exist, exactly one node satisfies the split condition.
Watch Out For
  • The "a node can be a descendant of itself" rule — returning the node on match handles the ancestor-descendant case; do not require the split on both sides unconditionally.
  • Compare node identity (references/pointers), not values, since this is a general tree; though here values are unique either works.
  • Do not stop early at the first found target — you still need to check the other subtree so the parent can combine both signals.

Why optimize? Recomputing the downward gain for every apex re-traverses subtrees repeatedly, costing O(n^2); a single post-order pass computes each node's gain once and folds the apex check in along the way.

Key InsightDistinguish two quantities at every node. The gain a node can contribute upward to its parent is a single straight chain: node.val + max(0, gain(left), gain(right)) — you can only extend through one child because a path cannot fork and still go up. The path-through value, where the node is the apex, is node.val + max(0, gain(left)) + max(0, gain(right)) — this may use both children but cannot be passed upward. Compute gains once bottom-up, and update a global best with the path-through value at each node. Clamping negative gains to 0 means "skip this subtree if it hurts."
How to Think About It
  1. Write one post-order recursion that returns the upward gain of the node it visits.
  2. Inside, recurse to get left and right gains, then clamp each to at least 0 (a negative branch is better dropped).
  3. Update a global maximum with node.val + leftGain + rightGain (the apex/path-through value at this node).
  4. Return node.val + max(leftGain, rightGain) to the parent — only one branch may continue upward.
Why This WorksEvery path in the tree has a unique highest node (apex). By evaluating the path-through value at every node, you consider every possible apex exactly once, so the global maximum captures the true best path. Returning only a single-branch gain upward respects the constraint that a node used by its parent's path can extend in just one direction. Clamping to 0 correctly handles all-negative regions, and the single traversal ensures each gain is computed once.
Watch Out For
  • Return the single-branch gain to the parent, but update the global answer with the two-branch (through) value — mixing these is the classic bug.
  • Initialize the global best to negative infinity, not 0 — a tree of all-negative values must return the largest single node, and the path must be non-empty.
  • Clamp child gains with max(0, gain), but never clamp node.val itself.
  • Watch integer overflow when summing many large values in fixed-width languages.
