#5 · Longest Palindromic Substring

Medium String DP 393 days overdue · last reviewed 2025-05-13

The Problem: Given a string s, return the longest palindromic substring in s. A palindrome reads the same forward and backward.

Quick Reminder: Expand around every possible center. For each index, try both odd-length (single center) and even-length (double center) expansions. Track the longest palindrome found. This runs in O(n²) time with O(1) space.

Example 1: s = "babad" → "bab" (or "aba")

Example 2: s = "cbbd" → "bb"

Example 3: s = "a" → "a"

Example 4: s = "racecar" → "racecar"

Time: O(n²)  |  Space: O(1)
Challenge: You solved this in Python and Go. Try implementing it in TypeScript or Rust to build multi-language fluency.

The Problem: A happy number is defined by repeatedly replacing it with the sum of squares of its digits until it equals 1, or loops endlessly in a cycle. Return true if n is a happy number.

class Solution(object):
    def isHappy(self, n):
        """
        :type n: int
        :rtype: bool
        """
        visited = set()

        while n not in visited:
            visited.add(n)
            n = self.sumOfSquares(n)

            if n == 1:
                return True

        return False

    def sumOfSquares(self, n):

            a = a + (n % 10) * (n % 10)

#740 · Delete and Earn

Medium DP 393 days overdue · last reviewed 2025-05-13

The Problem: Given an integer array nums, you can pick any nums[i] to earn nums[i] points, but you must delete every element equal to nums[i]-1 and nums[i]+1. Return the maximum points you can earn.

Quick Reminder: Reduce to the House Robber problem. Aggregate the total points for each value, then iterate from 0 to max_value deciding whether to “rob” (take) or “skip” each value. You cannot take adjacent values, just like houses on a street.

Example 1: nums = [3,4,2] → 6 (delete 4, earn 4; then delete 2, earn 2; total 6)

Example 2: nums = [2,2,3,3,3,4] → 9 (delete all 3s, earn 9)

Example 3: nums = [1,1,1,2,4,5,5,5,6] → 18

Time: O(n + k) where k = max(nums)  |  Space: O(k)
Challenge: You solved this in Python with memoized recursion. Try the bottom-up iterative DP approach in Rust.

#309 · Best Time to Buy and Sell Stock with Cooldown

Medium DP 391 days overdue · last reviewed 2025-05-15

The Problem: Given an array prices where prices[i] is the price on day i, find the max profit with as many transactions as you like, but after selling you must cooldown for one day before buying again.

Quick Reminder: Model three states: held (holding stock), sold (just sold, cooldown next), and rest (not holding, free to buy). Transition: held = max(held, rest - price), sold = held + price, rest = max(rest, sold). This is a state machine DP with O(1) space.

Example 1: prices = [1,2,3,0,2] → 3 (buy@1, sell@3, cooldown, buy@0, sell@2)

Example 2: prices = [1] → 0

Example 3: prices = [1,2,4] → 3 (buy@1, sell@4)

Time: O(n)  |  Space: O(1) with state machine, O(n) with memoization
Challenge: Your Python solution uses top-down memoization with 3 parameters. Try the O(1) space state machine approach in Go.
Previous Solution (Python) 
class Solution:
    def maxProfit(self, prices: List[int]) -> int:
        # i, holding, cooldown
        @lru_cache
        def dp(i, holding, cooldown):
            if i == len(prices):
                return 0

            do_nothing = dp(i + 1, holding, 0)
            do_something = 0

            if holding:
                do_something = prices[i] + dp(i + 1, 0, 1)
            else:
                do_something = -prices[i] + dp(i + 1, 1, 0)

            if cooldown:
                return do_nothing
            else:
                return max(do_nothing, do_something)

        return dp(0, 0, 0)

#4 · Median of Two Sorted Arrays

The Problem: Given two sorted arrays nums1 and nums2, return the median of the combined sorted array. The overall run time complexity should be O(log(m+n)).

The Problem: Given the root of a binary tree, determine if it is a valid binary search tree (BST). A BST is valid if for every node, all values in its left subtree are strictly less than the node’s value, and all values in its right subtree are strictly greater.

Step-by-Step Thinking (Optimal: Inorder with Early Stop)

  1. Initialize a counter at 0 and a result variable.
  2. Perform iterative inorder traversal using an explicit stack (or recursive with a mutable counter).
  3. Push all left children onto the stack first (go left as far as possible).
  4. Pop a node, increment counter. If counter equals k, that node is the answer.
  5. Move to the right child and repeat.
  6. Stop as soon as the kth element is found — no need to visit remaining nodes.
Why This Works: In a BST, the inorder traversal visits nodes in sorted (ascending) order. By counting nodes as we visit them in-order, the kth node we visit is exactly the kth smallest element. Using a stack gives us explicit control to stop early.
Watch Out: k is 1-indexed, not 0-indexed. Also, if you use recursion without a way to short-circuit (like a mutable counter), you will visit all n nodes even after finding the answer. The iterative stack approach or a flag-based early return avoids this.

#236 · Lowest Common Ancestor of a Binary Tree

Medium Tree DFS LCA

The Problem: Given a binary tree and two nodes p and q, find their lowest common ancestor (LCA). The LCA is the deepest node that has both p and q as descendants (a node can be a descendant of itself).

Key Insight: Recurse on left and right subtrees. If the current node is p or q, return it. If one target is found in the left subtree and the other in the right, the current node is the LCA. If both are found on the same side, the LCA is on that side. This single-pass DFS is both the brute force and the optimal approach.
Brute Force — Path Finding

Find the path from root to p and root to q. Compare the paths and the last common node is the LCA. This requires two DFS passes and storing full paths.

Python

class Solution:
    def lowestCommonAncestor(self, root: 'TreeNode', p: 'TreeNode', q: 'TreeNode') -> 'TreeNode':
        def find_path(node, target, path):
            if not node:
                return False
            path.append(node)
            if node == target:
                return True
            if find_path(node.left, target, path) or find_path(node.right, target, path):
                return True
            path.pop()
            return False

        path_p, path_q = [], []
        find_path(root, p, path_p)
        find_path(root, q, path_q)

        lca = root
        for a, b in zip(path_p, path_q):
            if a == b:
                lca = a
            else:
                break
        return lca

#133 · Clone Graph

Medium Graph DFS BFS Hash Map

The Problem: Given a reference of a node in a connected undirected graph, return a deep copy (clone) of the graph. Each node contains a value and a list of its neighbors.

Key Insight: Use a hash map from original node to cloned node. DFS or BFS through the graph. For each node, if it is not yet cloned, create a clone and recursively/iteratively clone its neighbors. The map serves dual purpose: it prevents revisiting (cycle detection) and stores the mapping from old to new nodes.
Brute Force — BFS with Map

Use BFS with a queue. Clone each node when first discovered and store the mapping. Then wire up neighbors. This is already optimal at O(V+E), but BFS is conceptually the “brute” iterative approach vs the recursive DFS.

from collections import deque

class Solution:
    def cloneGraph(self, node: 'Node') -> 'Node':
        if not node:
            return None
        clones = {node: Node(node.val)}
        queue = deque([node])
        while queue:
            curr = queue.popleft()
            for neighbor in curr.neighbors:
                if neighbor not in clones:
                    clones[neighbor] = Node(neighbor.val)
                    queue.append(neighbor)
                clones[curr].neighbors.append(clones[neighbor])
        return clones[node]

Medium Graph Topological Sort BFS DFS

The Problem: There are numCourses courses (0 to numCourses-1). You are given prerequisite pairs [a, b] meaning you must take course b before course a. Return true if you can finish all courses (i.e., no cycle in the prerequisite graph).

Key Insight: This is cycle detection in a directed graph. Use either Kahn’s algorithm (BFS topological sort using in-degrees) or DFS with three-color marking (unvisited/visiting/visited). If a topological ordering exists (all nodes processed), there is no cycle and courses can be completed.
Brute Force — DFS Cycle Detection

Build an adjacency list. For each unvisited node, run DFS. Mark nodes as “visiting” (in current path) and “visited” (fully processed). If we encounter a “visiting” node, there is a cycle.

class Solution:
    def canFinish(self, numCourses: int, prerequisites: list[list[int]]) -> bool:
        graph = [[] for _ in range(numCourses)]
        for a, b in prerequisites:
            graph[b].append(a)

            if state[node] == 1:
                return True   # cycle found
            if state[node] == 2:

            state[node] = 1
            for neighbor in graph[node]:

            state[node] = 2
            return False

                return False
        return True

  2. Add all nodes with in-degree 0 to a queue (these have no prerequisites).

Watch Out: The prerequisite direction matters: [a, b] means b → a (b is a prerequisite for a). Build the edge from b to a, and increment in-degree of a. Reversing this direction will give wrong results.

The Problem: Given an n x n adjacency matrix isConnected where isConnected[i][j] = 1 means city i and city j are directly connected, return the number of provinces (connected components).

Step-by-Step Thinking (Optimal: Union-Find)

  1. Initialize Union-Find: each city is its own parent, component count = n.
  2. Iterate over the upper triangle of the matrix (i < j) to avoid duplicates.
  3. If isConnected[i][j] == 1, union cities i and j.
  4. On each successful union (different roots), decrement the component count.
  5. Return the final component count.
  6. Use path compression and union by rank for near O(1) per operation.
Why This Works: Initially each city is its own province. Every connection merges two provinces into one (unless they are already in the same province). Union-Find efficiently tracks these merges. After processing all connections, the remaining count of distinct sets equals the number of provinces.
Watch Out: The adjacency matrix is symmetric (isConnected[i][j] == isConnected[j][i]), so only iterate the upper triangle to avoid redundant unions. Also, the diagonal is always 1 (a city is connected to itself) — skip it.

Stack problems test your ability to maintain order-dependent state. #739 · Daily Temperatures

Medium Stack Monotonic Stack

The Problem: Given an array of integers temperatures where temperatures[i] is the daily temperature on day i, return an array answer such that answer[i] is the number of days you have to wait after the i-th day to get a warmer temperature. If there is no future day with a warmer temperature, set answer[i] = 0.

Example 1: temperatures = [73,74,75,71,69,72,76,73] → [1,1,4,2,1,1,0,0]

Example 2: temperatures = [30,40,50,60] → [1,1,1,0]

Example 3: temperatures = [30,60,90] → [1,1,0]

Brute Force — Nested Loop O(n²)

For each day i, scan forward through all subsequent days to find the first day j with a higher temperature. Set answer[i] = j - i.

Key Insight: Use a monotonic decreasing stack that stores indices. Iterate left to right. For each new temperature, pop all stack entries whose temperature is lower — each popped index has found its "next warmer day." The key realization: when we encounter a warmer day, we resolve ALL pending cooler days at once, because the stack maintains decreasing order. Each element is pushed and popped at most once, giving O(n) total.

How to Think About It

  1. Initialize an empty stack and a result array of zeros.
  2. Iterate through each day i from left to right.
  3. While the stack is non-empty AND temperatures[i] > temperatures[stack.top()]: pop the top index prev, set answer[prev] = i - prev.
  4. Push the current index i onto the stack.
  5. After processing all days, any indices remaining on the stack have no warmer future day (answer stays 0).
Why This Works: The stack always contains indices in decreasing temperature order. When a warmer day arrives, it "resolves" all the cooler days waiting on the stack. Each index enters the stack exactly once and leaves exactly once, so despite the nested while loop inside the for loop, the total work across all iterations is O(n). This is the amortized analysis key to monotonic stack efficiency.
Watch Out For:
  • Store indices, not values: The stack must hold indices (not temperatures) because you need to compute the distance i - prev. Use the index to look up the temperature when comparing.
  • Strictly greater: The problem asks for a warmer temperature, not warmer-or-equal. Use > not ≥ in the comparison.
  • Decreasing vs. increasing: For "next greater element" problems, use a monotonic decreasing stack. For "next smaller element," use a monotonic increasing stack.

#155 · Min Stack

Medium Stack Design

The Problem: Design a stack that supports push, pop, top, and retrieving the minimum element, all in O(1) time. Implement the MinStack class: MinStack() initializes the stack. push(val) pushes element val onto the stack. pop() removes the element on top. top() gets the top element. getMin() retrieves the minimum element in the stack.

Example:

MinStack minStack = new MinStack();

minStack.push(-2); minStack.push(0); minStack.push(-3);

minStack.getMin(); → -3

minStack.pop();

minStack.top(); → 0

minStack.getMin(); → -2

Brute Force — Scan for Min O(n)

Use a normal stack for push/pop/top. For getMin(), scan the entire stack to find the minimum. This makes getMin O(n) while other operations are O(1).

Key Insight: Maintain a parallel min-stack (or store tuples). Every time you push a value, also push the current minimum onto the min-stack. When you pop, pop from both stacks. The top of the min-stack always holds the current minimum. This works because the minimum can only change when elements are added or removed, and the stack’s LIFO order means undoing a push always restores the previous minimum.

How to Think About It

  1. Maintain two stacks: main_stack for values and min_stack for tracking minimums.
  2. On push(val): push val to main_stack. Push min(val, min_stack.top()) to min_stack (or just val if min_stack is empty).
  3. On pop(): pop from both stacks simultaneously.
  4. On top(): return main_stack.top().
  5. On getMin(): return min_stack.top(). This is O(1) because the min-stack always has the answer at its top.
Why This Works: The min-stack mirrors the main stack in height. At any point in time, min_stack[i] holds the minimum of main_stack[0..i]. When we pop an element, the new minimum is simply min_stack[new_top], which was computed when that element was pushed. The LIFO ordering of a stack guarantees that we always “undo” operations in reverse order, so the minimum is always correct.
Watch Out For:
  • Alternative: store tuples: Instead of two separate stacks, you can store (value, current_min) pairs in a single stack. Same idea, slightly simpler code.
  • Empty stack checks: When the stack is empty, getMin() and top() are undefined. The problem guarantees these are not called on an empty stack, but defensive coding adds a check.
  • Duplicate minimums: If the same minimum value is pushed multiple times, each push records the minimum independently. Popping one does not affect the other. This is handled automatically by the parallel stack approach.

Key Insight: Build a trie for efficient prefix matching. For addWord, insert character by character (identical to basic trie insertion). For search, traverse the trie with DFS. When you encounter a '.', branch into ALL existing children at that level and continue the search recursively. If any branch leads to a complete match, return true. This turns a linear scan into a tree-guided search that prunes impossible paths early.

How to Think About It

  1. Build a standard trie with a children map and an isEnd flag at each node.
  2. addWord: traverse/create nodes character by character. Mark the final node as isEnd = true.
  3. search: define a recursive DFS function dfs(node, index).
  4. Base case: if index == len(word), return node.isEnd.
  5. If word[index] == '.': iterate all children of the current node. If any dfs(child, index + 1) returns true, return true. Otherwise return false.
  6. If word[index] is a letter: check if that child exists. If yes, recurse. If no, return false.
Why This Works: The trie organizes words by shared prefixes, so non-matching prefixes are pruned immediately. The '.' wildcard triggers branching, but only into children that actually exist in the trie — not all 26 letters, just the ones with words going through them. For words without wildcards, search is O(L) (word length). Wildcards add branching, but in practice the trie structure limits the fan-out dramatically compared to checking every stored word.
Watch Out For:
  • Worst case with all dots: Searching "..." (all wildcards) explores the entire trie, which can be O(26^L) in the worst case. This is the trade-off: wildcards can degrade performance, but typical inputs are much better.
  • Must check isEnd: A trie path existing does not mean a word ends there. Always verify isEnd at the final character. "ba" should not match if only "bad" was added.
  • DFS not BFS: Use DFS (not BFS) for the search. DFS naturally handles the recursive character-by-character matching and short-circuits on the first match found.

Binary Search

Binary search on the answer space is one of the most powerful and often-missed patterns. Instead of searching an array for a value, you binary search on the answer itself, checking a feasibility predicate at each midpoint. #875 is the perfect introduction: the predicate is monotonic (if Koko can finish at speed k, she can finish at any speed > k), enabling binary search over the solution space.

#875 · Koko Eating Bananas

Medium Binary Search Greedy

The Problem: Koko loves eating bananas. There are n piles of bananas, where piles[i] is the number of bananas in the i-th pile. The guards have gone and will come back in h hours. Koko can decide her bananas-per-hour eating speed k. Each hour, she chooses a pile and eats k bananas from it. If the pile has fewer than k bananas, she eats all of them and does not eat any more bananas during that hour. Return the minimum integer k such that she can eat all the bananas within h hours.

Example 1: piles = [3,6,7,11], h = 8 → 4

Example 2: piles = [30,11,23,4,20], h = 5 → 30

Example 3: piles = [30,11,23,4,20], h = 6 → 23

Brute Force — Try Every Speed O(max(piles) * n)

Try every possible speed from 1 to max(piles). For each speed, compute the total hours needed. Return the first speed where total hours ≤ h.

            if total_hours <= h:

Key Insight: Binary search on the answer space. The answer k ranges from 1 to max(piles). For a given k, you can compute in O(n) whether Koko finishes in time. The key observation: if she can finish at speed k, she can also finish at any speed > k. This monotonicity means you can binary search for the minimum k where the predicate "can finish in h hours" flips from false to true.

How to Think About It

  1. Set lo = 1, hi = max(piles). The minimum possible speed is 1, the maximum needed is max(piles) (finishes any pile in 1 hour).
  2. Binary search: compute mid = (lo + hi) / 2.
  3. Feasibility check: for each pile, compute hours needed as ceil(pile / mid). Sum all hours.
  4. If total hours ≤ h: mid is fast enough. Try smaller: hi = mid.
  5. If total hours > h: mid is too slow. Need faster: lo = mid + 1.
  6. When lo == hi, return lo. This is the minimum speed that satisfies the constraint.
Why This Works: The predicate "can Koko finish in h hours at speed k" is monotonic: once it becomes true at some k, it stays true for all larger k. This is the hallmark of "binary search on the answer." Instead of searching through a sorted array, we search through the answer space [1, max(piles)] guided by a yes/no predicate. The binary search template is identical to searching a sorted array — only the "comparison" is replaced by a feasibility check.
Watch Out For:
  • Ceiling division: ceil(pile / k) is critical. Use (pile + k - 1) / k for integer ceiling division to avoid floating-point errors. In Python, math.ceil(pile / k) works but -(-pile // k) is a clever integer-only alternative.
  • lo = 1, not 0: Speed of 0 would cause division by zero. The minimum meaningful speed is 1.
  • Overflow in sum: In C++ and Rust, the sum of hours can overflow int32 for large inputs. Use long long or i64.

Bit Manipulation

Bit manipulation problems test your understanding of how numbers are represented at the binary level. XOR is the foundational operation: it is associative, commutative, and self-inverse (a ^ a = 0). #136 is the classic introduction that demonstrates how XOR can solve a problem in O(1) space that would otherwise require a hash map. Understanding XOR opens the door to problems like #137 (Single Number II), #260 (Single Number III), and many bit-masking DP problems.

#136 · Single Number

Easy Bit Manipulation XOR

The Problem: Given a non-empty array of integers nums, every element appears exactly twice except for one element which appears exactly once. Find that single element. You must implement a solution with linear runtime complexity and use only constant extra space.

Example 1: nums = [2,2,1] → 1

Example 2: nums = [4,1,2,1,2] → 4

Example 3: nums = [1] → 1

Brute Force — Hash Map Counting O(n) time, O(n) space

Count the frequency of each element using a hash map. Return the element with count 1. This uses O(n) space, violating the constant-space requirement.

from collections import Counter

class Solution:

        for num, count in counts.items():
            if count == 1:
                return num
        return -1  # unreachable

Key Insight: XOR is self-inverse: a ^ a = 0 and a ^ 0 = a. XOR all elements together. Every duplicate pair cancels out (a ^ a = 0), and the single element remains (0 ^ single = single). This gives O(n) time and O(1) space with a single pass through the array.

How to Think About It

  1. Initialize result = 0.
  2. Iterate through every element in the array.
  3. XOR each element with result: result ^= num.
  4. After processing all elements, result holds the single number because all paired elements cancelled to 0.
  5. Return result.
Why This Works: XOR has three key properties: (1) Commutative: a ^ b = b ^ a, so order does not matter. (2) Associative: (a ^ b) ^ c = a ^ (b ^ c), so grouping does not matter. (3) Self-inverse: a ^ a = 0. Together, these mean XOR-ing all elements is equivalent to XOR-ing all pairs (which each become 0) and then XOR-ing with the single element (which is 0 ^ single = single). The beauty is that this works regardless of the order of elements.
Watch Out For:
  • Only works for pairs: This trick requires every duplicate to appear exactly twice. If elements appear three times (#137), XOR alone does not work — you need bit counting or a more advanced technique.
  • Negative numbers: XOR works on the bit representation, which handles negative numbers correctly in two's complement. No special handling needed.
  • Don’t confuse with sum trick: The "2 * sum(set) - sum(array)" approach also works but uses O(n) space for the set and risks integer overflow. XOR is cleaner.

Dynamic Programming — Knapsack

The 0/1 knapsack is one of the most important DP patterns. #416 reduces a seemingly combinatorial problem (partition into equal-sum subsets) to a classical knapsack question: can you fill a knapsack of capacity sum/2? This transforms an exponential brute force into a polynomial DP solution. The subset sum / knapsack pattern recurs in #494 Target Sum, #474 Ones and Zeroes, and coin change variants.

#416 · Partition Equal Subset Sum

Medium DP Knapsack

The Problem: Given an integer array nums, return true if you can partition the array into two subsets such that the sum of elements in both subsets is equal. Otherwise, return false.

Example 1: nums = [1,5,11,5] → true (subsets: [1,5,5] and [11])

Example 2: nums = [1,2,3,5] → false

Brute Force — Generate All Subsets O(2^n)

Generate all possible subsets using recursion/backtracking. For each subset, check if its sum equals totalSum / 2. If any such subset exists, return true.

Python
class Solution:
    def canPartition(self, nums: list[int]) -> bool:
        total = sum(nums)
        if total % 2 != 0:
            return False
        target = total // 2

        def backtrack(index: int, current_sum: int) -> bool:
            if current_sum == target:
                return True
            if current_sum > target or index == len(nums):
                return False
            # Include nums[index] or skip it
            return (backtrack(index + 1, current_sum + nums[index]) or
                    backtrack(index + 1, current_sum))

        return backtrack(0, 0)

How to Think About It

  2. Set target = total / 2. Create a boolean array dp of size target + 1, initialized to false. Set dp[0] = true (empty subset has sum 0).
  3. For each number num in the array:
  4. Iterate j from target down to num (reverse order is critical): set dp[j] = dp[j] || dp[j - num].
  5. After processing all numbers, return dp[target].
  6. The reverse iteration ensures each number is used at most once (0/1 constraint).
Why This Works: dp[j] tracks whether sum j is achievable. For each new number num, if sum j - num was previously achievable, then sum j becomes achievable by adding num. The right-to-left iteration ensures we do not use num twice in the same "round" (this is the 0/1 knapsack constraint). If we iterated left-to-right, we could count num multiple times (which would be the unbounded knapsack variant, like coin change).
Watch Out For:
  • Odd total sum → immediate false: This is the most common early-exit optimization. If the total is odd, equal partition is mathematically impossible.
  • Iterate RIGHT to LEFT: This is the critical detail. Left-to-right iteration would allow reusing the same element, turning this into an unbounded knapsack (wrong answer). Right-to-left ensures each element is considered at most once.
  • Space optimization: The 1D DP array is already the space-optimized version. The 2D version uses dp[i][j] = "can sum j be achieved using elements [0..i]?" but the 1D version suffices since each row only depends on the previous row.

Python Features

  • Boolean list: [False] * (target + 1) creates the DP array. Python booleans are integers (True = 1, False = 0), so bitwise OR also works. 
    dp = [False] * (target + 1)
dp[0] = True
# Update: dp[j] = dp[j] or dp[j - num]
  • range(target, num - 1, -1): Iterates from target down to num (inclusive). The third argument is the step (-1 for reverse). 
    for j in range(target, num - 1, -1):
    dp[j] = dp[j] or dp[j - num]
  • Set-based alternative: Instead of a boolean array, maintain a set of achievable sums. For each num, add s + num for each s in the set. 
    achievable = {0}
for num in nums:
    achievable = achievable | {s + num for s in achievable}
return target in achievable
  • Bitset trick with integers: Python’s arbitrary-precision integers can act as bitsets. Shift and OR to update all sums at once. 
    bits = 1  # bit 0 is set = sum 0 is achievable
for num in nums:
    bits |= bits << num
return bool(bits & (1 << target))

  • Boolean slice: make([]bool, target+1) initializes all elements to false (Go zero value for bool). 
    dp := make([]bool, target+1)
dp[0] = true
  • Reverse loop: for j := target; j >= num; j-- iterates right to left. 
    for _, num := range nums {
    for j := target; j >= num; j-- {
        dp[j] = dp[j] || dp[j-num]
    }
}
  • Early termination: Check dp[target] after each number. If already true, return immediately. 
    if dp[target] {
    return true  // found a valid subset early
}

  • Boolean vec: vec![false; target + 1] creates the DP array.

    for &num in &nums {
    let num = num as usize;

C++ Features

  • vector<bool>: C++ vector<bool> is a space-optimized specialization that stores one bit per element. Good for this use case. 
    vector<bool> dp(target + 1, false);
dp[0] = true;
  • accumulate for sum: accumulate(nums.begin(), nums.end(), 0) computes the total.

return dp[target];
Brute: O(2^n) time, O(n) space → Optimal (DP): O(n * sum/2) time, O(sum/2) space

Connection: This is the canonical 0/1 knapsack problem in disguise. The right-to-left iteration trick for 0/1 knapsack (vs. left-to-right for unbounded knapsack in #322 Coin Change) is a fundamental distinction you will use repeatedly.
