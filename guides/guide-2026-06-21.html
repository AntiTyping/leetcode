#198 · House Robber

The Problem. Given an array of house values along a street, determine the maximum money you can rob without robbing two adjacent houses.

Quick Reminder. Classic DP. At each house choose max(skip, rob + best-non-adjacent). A two-variable bottom-up sweep is the cleanest form.
Key Insight. The choice at house i only depends on the best from i-1 (skip) versus i-2 + current (rob). This collapses the whole table into two rolling variables.
O(n) time, O(1) space

Input: nums = [1,2,3,1]

Output: 4

Explanation: Rob house 0 (1) and house 2 (3) = 4.

Input: nums = [2,7,9,3,1]

Output: 12

Explanation: Rob houses 0, 2, 4 = 2+9+1 = 12.

Input: nums = [0]

Output: 0

Explanation: Single house with value 0.

Pythonclass Solution:
    def rob(self, nums):
        memo = {}
        def dp(i):
            if i == 0: return nums[0]
            if i == 1: return max(nums[0], nums[1])
            if i not in memo:
                memo[i] = max(dp(i-1), nums[i] + dp(i-2))
            return memo[i]
        return dp(len(nums) - 1)

#139 · Word Break

Mediumdynamic-programmingstring3 days since last review &middot; rep 3

The Problem. Given a string s and a dictionary of words, return true if s can be segmented into a space-separated sequence of one or more dictionary words.

Quick Reminder. Bottom-up DP: dp[i] is true if s[:i] is segmentable. For each i, check every word that could end at i.
Key Insight. Position i is reachable if there exists a dictionary word w with dp[i-len(w)] true and s[i-len(w):i] == w. Reuse of subproblem answers avoids the exponential re-splitting of brute force.
O(n*m*k) time, O(n) space (n=len(s), m=words, k=avg word len)

Input: s = "leetcode", wordDict = ["leet","code"]

Output: true

Explanation: "leet" + "code".

Input: s = "applepenapple", wordDict = ["apple","pen"]

Output: true

Explanation: "apple" + "pen" + "apple" (words reusable).

Input: s = "catsandog", wordDict = ["cats","dog","sand","and","cat"]

Output: false

Explanation: No segmentation covers the whole string.

Pythonclass Solution:
    def wordBreak(self, s, wordDict):
        memo = {}
        def dp(i):
            if i >= len(s): return True
            for word in wordDict:
                if (word, i) not in memo:
                    if s[i:i+len(word)] == word:
                        memo[(word, i)] = dp(i + len(word))
                if (word, i) in memo and memo[(word, i)]:
                    return True
            return False
        return dp(0)

    def largestRectangleArea(self, heights):
        stack = []           # indices of increasing bars

Why optimize? With n up to 3×10⁴, the O(n²) double loop is ~10⁹ operations — over the time limit. You need to stop re-scanning subarrays.

Key Insight. Instead of summing per subarray, sum per element: count how many subarrays have arr[i] as their minimum. A monotonic stack finds, for each i, the distance to the Previous Less Element (left) and Next Less Element (right). Then arr[i] contributes arr[i] * left * right.

How to Think About It

  1. For element i, let left[i] = number of consecutive positions ending at i where arr[i] is the minimum (distance to the previous strictly-less element).
  2. Let right[i] = number of consecutive positions starting at i where arr[i] is the minimum (distance to the next less-or-equal element).
  3. The count of subarrays in which arr[i] is the minimum is left[i] * right[i] — choose any valid left endpoint and any valid right endpoint.
  4. Use a monotonic increasing stack to compute the boundaries: when the incoming value is smaller, pop and record spans.
  5. Sum arr[i] * left[i] * right[i] over all i, taking the result modulo 10^9+7.
Why This Works. Partitioning all subarrays by the index of their minimum element covers every subarray exactly once (with the strict/non-strict tie-break ensuring no subarray is counted under two different minima). The stack computes each boundary in amortized O(1), so the total work is linear.
Watch Out. Use strict < on one side and <= on the other for the less-element comparisons. If both sides use the same strictness, subarrays whose minimum repeats get counted twice (or zero times).
  • Forgetting the modulo on the running sum lets it overflow (especially in C++/Go with int).
  • Mixing up which side uses strict < versus <= — this is what prevents double-counting equal elements.

Key Insight. Before the single element, each pair occupies (even, odd) indices: nums[0]==nums[1], nums[2]==nums[3], and so on. After the single element, this parity flips and pairs start at odd indices. Binary search on which side still has the 'even-index-starts-a-pair' property.

  4. Continue until lo == hi; that index is the single element.
  5. No need for a separate equality check — the loop converges on the unique unpaired position.
Why This Works. In the prefix before the answer, every pair starts on an even index, so an even mid with nums[mid]==nums[mid+1] proves the anomaly is to the right. Past the answer, that equality fails at even indices. Binary search discards a balanced half each step, preserving the O(log n) elimination guarantee on the well-defined parity boundary.
Watch Out. Always normalize mid to an even index before comparing. If mid lands on an odd index, the 'compare with mid+1' logic no longer matches the even-starts-a-pair invariant and the search drifts to the wrong half.

Key Insight. Each child's candy count is governed by two independent constraints: their left neighbor and their right neighbor. Satisfy them separately. How to Think About It

  1. Initialize every child to 1 candy.
  2. Left-to-right: if ratings[i] > ratings[i-1], set left[i] = left[i-1] + 1; otherwise left[i] = 1.
  3. Right-to-left: if ratings[i] > ratings[i+1], set right[i] = right[i+1] + 1; otherwise right[i] = 1.
  4. The final candy for child i is max(left[i], right[i]) — the tighter of the two constraints.
  5. Sum the maxima for the answer. You can keep a single rolling 'right' counter to use O(1) extra space.
Why This Works. The left pass guarantees every up-slope from the left is strictly increasing; the right pass does the same from the right. Taking the maximum at each index satisfies both neighbor constraints simultaneously, and because each pass only ever increments by 1 over a smaller neighbor, no candy count is larger than necessary — hence the total is minimal.
Watch Out. At each index the answer is max(left[i], right[i]), not left[i] + right[i]. Adding the two passes is the classic mistake — it violates minimality and over-counts the shared baseline of 1.
  • Resetting to 1 only when the rating is not greater than the neighbor is essential; carrying the previous count forward over a 'valley' over-allocates.
  • Summing left and right instead of taking their max double-counts.

Key Insight. Instead of committing to one meaning per '*', track the range [lo, hi] of possible open-paren counts. lo is the count if every '*' so far were ')' or empty (minimum opens); hi is the count if every '*' were '(' (maximum opens). How to Think About It

  1. Maintain two counters: lo (minimum possible open parens) and hi (maximum possible open parens), both starting at 0.
  2. On '(': increment both lo and hi. On ')': decrement both.
  3. On '*': decrement lo (treat as ')') and increment hi (treat as '(') — this widens the range.
  4. If hi < 0 at any point, even the most generous interpretation has too many ')' — return false immediately.
  5. Clamp lo to 0 (it can never sensibly go below 0, since extra ')' beyond balance is just invalid for that branch).
  6. At the end, return whether lo == 0 — meaning some interpretation balanced perfectly.
Why This Works. lo and hi bound the interval of achievable open-paren counts under all '*' assignments. If hi ever drops below 0, every assignment is already unbalanced. Clamping lo at 0 discards branches that already had unmatched ')'. If at the end 0 lies within the final [lo, hi] interval (i.e., lo == 0), a valid assignment exists.
Watch Out. Bail out only when hi < 0, never when lo < 0. A negative lo simply means one over-eager interpretation died; clamp lo back to 0 and keep going. Bailing on lo < 0 rejects valid strings.
  • Forgetting to clamp lo to 0 lets it go negative, which corrupts the final lo == 0 test.
  • Checking lo < 0 to bail early is wrong — only hi < 0 proves invalidity; lo going negative just means that branch is dead, not the whole string.

Key Insight. This is an unbounded-coin problem where the coins are perfect squares and you want the minimum count. dp[i] = 1 + min over all squares j*j ≤ i of dp[i - j*j]. Equivalently, BFS from n where each edge subtracts a square; the first time you reach 0, the level number is the answer.

How to Think About It

  1. Define dp[i] = fewest squares summing to i, with dp[0] = 0.
  2. For each i from 1 to n, try every square j*j ≤ i and set dp[i] = min(dp[i], dp[i - j*j] + 1).
  3. Build dp bottom-up so each dp[i - j*j] is already computed.
  4. Return dp[n].
  5. Alternative: BFS treating each number as a node; subtract each square to find neighbors. The level at which you first hit 0 is the minimum count — BFS naturally finds the shortest path.
Why This Works. Every way to write i as a sum of squares ends with some final square j*j, leaving i - j*j to be formed optimally. Taking the minimum over all valid final squares gives the optimal substructure, and bottom-up evaluation guarantees each dependency is solved first. BFS works for the same reason: the shortest sequence of square-subtractions is a shortest path in an unweighted graph.
Watch Out. Initialize dp[i] for i >= 1 to a large value (infinity), not 0. If you leave them at 0, the min(dp[i-j*j]+1) comparison never improves and you get wrong (zero) answers.
  • Off-by-one in the square loop — the condition is j*j <= i, and j starts at 1, not 0.
  • Initializing dp with 0 instead of a large sentinel makes the min meaningless; use infinity for i >= 1.

Why optimize? The same remainder is recomputed exponentially often. With target up to 1000 you must memoize on the remaining sum, collapsing it to a 1D DP.

Key Insight. Because order matters, this counts sequences, not sets. dp[t] = number of ordered ways to reach t. To build t, the sequence ends in some num, preceded by an ordered way to reach t - num. So dp[t] = sum over num of dp[t - num], with target as the OUTER loop (this is what makes order matter, unlike Coin Change II).

How to Think About It

  1. Define dp[0] = 1 (one empty way to reach 0) and dp[t] = 0 for t > 0 initially.
  2. Loop t from 1 to target (the OUTER loop is the target).
  3. For each t, add dp[t - num] for every num <= t — each num can be the LAST element of the sequence.
  4. Return dp[target].
  5. Key distinction: iterating target outer / nums inner counts ordered sequences (permutations); swapping the loops (nums outer) would count unordered combinations instead.
Why This Works. Any ordered sequence summing to t has a well-defined last element num; removing it yields an ordered sequence summing to t - num, counted by dp[t - num]. Summing over all possible last elements counts every ordered sequence exactly once. Putting target in the outer loop ensures every ordering is generated, because each num is considered as the final step for every t.
Watch Out. Loop order is the entire problem. target-outer / nums-inner counts permutations (what this problem wants); nums-outer / target-inner counts combinations (#518). Get this backwards and every test with reorderable elements fails.
  • Swapping the loop order (nums outer, target inner) silently switches you to counting combinations, not permutations — the #518 Coin Change II behavior.
  • The answer can overflow 32-bit ints for some inputs even though the problem guarantees it fits in a signed int; use a wide accumulator in C++/Go to be safe.

Why This Works. Think of each non-majority element cancelling out one majority element. Watch Out. You MUST verify both candidates in a second pass. Unlike #169 where a majority is guaranteed, here a candidate can survive voting without truly exceeding n/3 (e.g., [1,2,3]). Output only verified survivors.

Why optimize? The doubly-recursive brute force re-walks each subtree from every node, giving O(n²) on skewed trees (up to ~10⁶ ops at n=1000). A prefix-sum hash map cuts it to O(n).

Key Insight. Adapt the array prefix-sum trick (#560 Subarray Sum Equals K) to a root-to-current-node path. Track the running prefix sum down the path and a hash map of how many times each prefix sum has occurred. A path ending at the current node sums to target iff appeared earlier on the path.

How to Think About It

  1. DFS from the root carrying a running sum (prefix sum from root to the current node).
  2. Keep a hash map mapping each prefix-sum value to its count among ancestors on the current path; initialize {0: 1} for paths starting at the root.
  3. At each node, the number of valid paths ending here is map[currentPrefix - targetSum].
  4. Add the current prefix to the map before recursing into children, then DECREMENT it after returning (backtracking) so sibling branches do not see this node's contribution.
  5. Sum the per-node counts for the total.
Why This Works. Any downward path ending at the current node corresponds to two prefix sums on the root-to-node chain whose difference is targetSum. The hash map records how many earlier prefix sums equal currentPrefix - targetSum, which is exactly the number of valid start points. Backtracking the map entry on return keeps the counts scoped to the current path, ensuring correctness across branches.
Watch Out. You must remove (decrement) the current prefix sum from the map AFTER recursing into both children. Skip this backtracking step and a node's prefix leaks into unrelated branches, inflating the count.
  • Forgetting to seed the map with {0: 1} misses paths that start at the root.
  • Not decrementing (backtracking) the prefix-sum count after visiting a node leaks counts into sibling subtrees and overcounts.
  • Sums can exceed 32-bit range; use a 64-bit accumulator.

How to Think About It

  5. If s == target, return immediately — distance 0 is optimal.

  • Not initializing best with the first valid triple (or a sentinel) can leave it uninitialized.

#332 · Reconstruct Itinerary

Hardgraphdfseulerian-path

The Problem. Given a list of airline tickets [from, to], reconstruct the itinerary that uses every ticket exactly once, starting from "JFK". If multiple valid itineraries exist, return the lexicographically smallest one.

Brute Force — O(E!) time worst case

Backtracking: try each unused ticket from the current airport in lexicographic order, recurse, and undo if the path fails to use all tickets. Exponential in the worst case.

Pythondef findItinerary(tickets):
    from collections import defaultdict
    graph = defaultdict(list)
    for a, b in sorted(tickets): graph[a].append(b)
    used = {a: [False]*len(v) for a, v in graph.items()}
    route = ["JFK"]
    def dfs(node):
        if len(route) == len(tickets) + 1: return True
        for i, nxt in enumerate(graph[node]):
            if not used[node][i]:
                used[node][i] = True; route.append(nxt)
                if dfs(nxt): return True
                used[node][i] = False; route.pop()
        return False
    dfs("JFK")
    return route

Why optimize? Naive backtracking can explore a factorial number of orderings before finding the valid one. With up to 300 tickets that is intractable — Hierholzer's algorithm finds the Eulerian path in O(E log E).

Key Insight. This is an Eulerian path problem: a trail using every edge exactly once. Hierholzer's algorithm builds it greedily: from each node, always take the lexicographically smallest unused outgoing edge; when a node has no more edges, prepend it to the result (post-order). Because you append on the way back, dead-end airports settle at the end of the route, and the final reversed order is a valid Eulerian path.

How to Think About It

  1. Build an adjacency map from airport to a sorted (or min-heap) list of destinations so you always pick the lexicographically smallest next edge.
  2. Start a DFS at "JFK".
  3. Greedily consume the smallest unused edge out of the current node, recursing into the destination.
  4. When the current node has no remaining outgoing edges, it is a dead end — add it to the FRONT of the route (post-order append, then reverse at the end).
  5. When the recursion fully unwinds, the route built in reverse post-order is the lexicographically smallest Eulerian path.
  6. Use a pointer/index per adjacency list (or pop from a heap) so each edge is consumed exactly once in O(E log E).
Why This Works. Hierholzer's algorithm works because a connected graph with the right degree balance has an Eulerian path, and greedily following edges always extends the trail until it gets stuck only at the path's true endpoint. Post-order insertion places stuck nodes at the tail; reversing yields a valid traversal. Choosing the smallest edge first makes the result lexicographically minimal without backtracking.
Watch Out. Add a node to the result only AFTER its outgoing edges are exhausted (post-order), then reverse at the end. Building the route in forward/pre-order order produces wrong itineraries whenever the greedy choice dead-ends before consuming all tickets.
  • Appending in pre-order instead of post-order produces an invalid route when a branch dead-ends early.
  • Not consuming edges exactly once (e.g., re-using a ticket) breaks the Eulerian guarantee.
  • Using a plain sorted list and removing from the front is O(n) per removal — use an index pointer or heap.

#131 · Palindrome Partitioning

Mediumbacktrackingstringdynamic-programming

The Problem. Given a string s, partition it so that every contiguous substring in the partition is a palindrome. Return all possible such partitionings.

Brute Force — O(2^n · n) time, O(n) space per candidate

Generate every possible way to cut the string (2^(n-1) cut patterns), then keep only those where all pieces are palindromes.

Pythondef partition(s):
    res = []
    n = len(s)
    def is_pal(t): return t == t[::-1]
    for mask in range(1 << (n - 1)):
        parts, prev = [], 0
        for i in range(n - 1):
            if mask & (1 << i):
                parts.append(s[prev:i+1]); prev = i + 1
        parts.append(s[prev:])
        if all(is_pal(p) for p in parts):
            res.append(parts)
    return res

Why optimize? Enumerating all 2^(n-1) cut patterns and validating each is wasteful: it builds many partitions only to discard them. Backtracking that only extends with valid palindrome prefixes prunes the search dramatically.

Key Insight. Backtrack over partition points instead of enumerating all cuts. At position start, try every end such that s[start:end] is a palindrome; if it is, add it to the current path and recurse from end. When start reaches the string's length, the current path is a complete valid partition. Pruning on the palindrome check avoids exploring impossible branches.

How to Think About It

  1. Maintain a current path (list of chosen palindrome pieces) and a start index.
  2. If start == len(s), the path is complete — record a copy of it.
  3. Otherwise, for each end from start+1 to len(s), check if s[start:end] is a palindrome.
  4. If it is, push that substring, recurse with start = end, then pop it (backtrack) to try the next length.
  5. Optionally precompute an n×n palindrome table (DP) so each palindrome check is O(1) instead of O(n).
Why This Works. Every valid partition is a sequence of palindrome prefixes: pick the first palindrome piece, then recursively partition the rest. By only recursing when the current prefix is a palindrome, the search constructs exactly the set of valid partitions and nothing else. The backtracking (push/recurse/pop) ensures each branch explores one choice at a time without corrupting siblings.
Watch Out. When recording a completed partition, store a COPY of the current path, not the path itself. Appending the live list means subsequent pops/pushes mutate your saved results, leaving the output corrupted.
  • Appending the path by reference instead of a copy means later mutations corrupt already-recorded results.
  • Forgetting to backtrack (pop) after the recursive call mixes pieces across branches.
  • Recomputing palindrome checks naively makes each O(n); a precomputed DP table or two-pointer expansion helps.

#621 · Task Scheduler

Mediumgreedyheapmathdesign

The Problem. Given a list of task labels and an integer n (the cooldown), where the same task must be separated by at least n intervals, return the minimum number of intervals (including idles) needed to finish all tasks. Each task takes one interval.

Brute Force — O(total · log 26) time, O(26) space

Simulate the schedule: at each tick, pick the available task with the highest remaining count that is off cooldown (use a heap and a waiting queue); count idles when nothing is available.

Pythondef leastInterval(tasks, n):
    import heapq
    from collections import Counter, deque
    heap = [-c for c in Counter(tasks).values()]
    heapq.heapify(heap)
    time = 0
    wait = deque()   # (ready_time, count)
    while heap or wait:
        time += 1
        if heap:
            c = heapq.heappop(heap) + 1
            if c < 0: wait.append((time + n, c))
        if wait and wait[0][0] == time:
            heapq.heappush(heap, wait.popleft()[1])
    return time

Why optimize? Tick-by-tick simulation works but is verbose and can be slow when total tasks are large. A closed-form formula computes the answer in O(total) with no simulation.

Key Insight. The most frequent task dictates the schedule's skeleton. If its count is maxFreq, it creates (maxFreq - 1) gaps, each needing n+1 slots (the task plus n cooldown). Other tasks fill the idle slots. The answer is max(len(tasks), (maxFreq - 1) * (n + 1) + numTasksWithMaxFreq) — the max guards against the case where there are so many distinct tasks that no idling is ever needed.

How to Think About It

  1. Count the frequency of each task; find maxFreq, the largest frequency.
  2. Count how many tasks share that maxFreq (call it numMax).
  3. Build the skeleton from the most frequent task: (maxFreq - 1) blocks of size (n + 1), then a final block holding the numMax most-frequent tasks: (maxFreq - 1) * (n + 1) + numMax.
  4. If there are many distinct tasks, the gaps fill up completely and there are no idles — in that case the answer is simply len(tasks).
  5. Return max(len(tasks), (maxFreq - 1) * (n + 1) + numMax).
Why This Works. The schedule length is lower-bounded by both the total number of tasks (every task needs a slot) and by the cooldown structure of the most frequent task, which forces (maxFreq - 1) gaps of size n+1 plus a trailing block for all tasks tied at maxFreq. When enough distinct tasks exist, those gaps are entirely filled by real tasks (no idles), so the total collapses to len(tasks). Taking the max captures both regimes exactly.
Watch Out. The final answer is max(len(tasks), (maxFreq - 1) * (n + 1) + numMax). Dropping the max() guard breaks every case where distinct tasks fully pack the cooldown gaps and zero idles are needed.
  • Forgetting the trailing + numMax term undercounts when several tasks share the maximum frequency.
  • Omitting the max(len(tasks), ...) guard overcounts when there are no idle slots (many distinct tasks).
  • Using (maxFreq - 1) * n instead of (maxFreq - 1) * (n + 1) forgets that each block includes the task itself plus n cooldown slots.
