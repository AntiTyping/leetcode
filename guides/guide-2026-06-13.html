#7 · Reverse Integer

Medium Math

The Problem: Given a signed 32-bit integer x, return x with its digits reversed. Return 0 if the reversed integer overflows the 32-bit signed integer range.

Example 1: x = 123 → 321

Example 2: x = -123 → -321

Example 3: x = 120 → 21

Key Insight: Pop digits with x % 10, push with result * 10 + digit. Check overflow before multiplying — if result > INT_MAX / 10 (or equals it and digit > 7), return 0.
Brute Force Approach

Convert to string, reverse, and parse back. Uses extra space for the string and relies on language-level big integers or 64-bit intermediates to detect overflow after the fact.

    sign := 1

How to Think About It

  1. Initialize result = 0.
  2. While x != 0: extract digit = x % 10, then x = x / 10.
  3. Check overflow before pushing: if result > INT_MAX / 10 (or boundary digit check), return 0.
  4. Push digit: result = result * 10 + digit.
  5. Return result.
Why This Works: Checking overflow before the multiply prevents undefined behavior. The digit-by-digit extraction with % 10 and / 10 handles the sign naturally in languages where negative modulo preserves the sign (Go, C++, Rust). Each iteration peels one digit from the input and appends it to the result.
Watch Out For: Different languages handle negative modulo differently. INT_MAX = 2147483647, INT_MIN = -2147483648. Python has arbitrary precision integers, so you must manually enforce the 32-bit constraint. In C++, signed integer overflow is undefined behavior — always check before multiplying.

How to Think About It

  1. Find the largest index i such that nums[i] < nums[i+1]. If no such index exists, the entire array is descending — reverse it and return.
  2. Find the largest index j > i such that nums[j] > nums[i].
  3. Swap nums[i] and nums[j].
  4. Reverse the suffix starting at position i + 1.

Watch Out For: Off-by-one errors when searching backwards. Must handle the all-descending case (no ascent found). After the swap, the suffix is still guaranteed to be in descending order, so reversing it is correct.

The Problem: Given an array nums with values 0, 1, and 2 representing red, white, and blue, sort them in-place so that objects of the same color are adjacent, in the order 0, 1, 2. Solve in one pass.

#40 · Combination Sum II

The Problem: Find all unique combinations in candidates where the chosen numbers sum to target. Each number may only be used once.

Two-dimensional DP on string pairs. Understanding the recurrence relation is key.

#115 · Distinct Subsequences

Hard DP String

The Problem: Given strings s and t, return the number of distinct subsequences of s which equals t.

Example 1: s = "rabbbit", t = "rabbit" → 3

Example 2: s = "babgbag", t = "bag" → 5

Key Insight: dp[i][j] = number of ways to form t[0..j-1] from s[0..i-1]. If s[i-1] == t[j-1]: dp[i-1][j-1] + dp[i-1][j] (use this char or skip it). Otherwise: dp[i-1][j] (must skip).
Brute Force Approach

Recursive DFS without memoization. At each position in s, decide whether to use the current character (if it matches) or skip it. Exponential time due to overlapping subproblems.

How to Think About It

  2. Iterate through each character of s (outer loop).
  3. For each s[i], iterate j from len(t) down to 1 (backwards to avoid using updated values).
  4. If s[i-1] == t[j-1], add dp[j-1] to dp[j].
  5. Return dp[len(t)].
Why This Works: At each character of s, we decide whether to use it to match the current character of t or skip it. The DP accumulates all possible paths. The Problem: Given a list of words sorted in the alien language’s lexicographic order, derive the character ordering. Return "" if the order is invalid.

Example 1: words = ["wrt","wrf","er","ett","rftt"] → "wertf"

Example 2: words = ["z","x"] → "zx"

Brute Force Approach

The Problem: Given an m × n integer matrix, return the length of the longest strictly increasing path. You can move in four directions (up, down, left, right) to cells with strictly greater values.

How to Think About It

  1. Build a weighted adjacency list: for each equation a/b = k, add edge a→b with weight k and edge b→a with weight 1/k.
  2. For each query (src, dst), run BFS from src to dst.
  3. Track the running product of edge weights along the path.
  4. If dst is reached, return the accumulated product. Otherwise return -1.0.
Why This Works: Division chains are transitive: a/c = (a/b) · (b/c). A path in the weighted graph from src to dst multiplies the edge weights, which correspond to successive division ratios. BFS finds this path if it exists.
Watch Out For: Both query variables must exist in the graph. Self-division (a/a) always equals 1.0 if the variable exists. Floating-point precision is generally fine for this problem since values are multiplied, not subtracted.

How to Think About It

  1. Maintain a global timestamp that increments with each tweet.
  2. HashMap: userId → tweet list (each tweet is a (timestamp, tweetId) pair).
  3. HashMap: userId → followee set.
  4. postTweet: append (timestamp++, tweetId) to the user's tweet list.
  5. getNewsFeed: collect tweets of user + all followees, sort by timestamp descending, return top 10.
  6. follow/unfollow: add/remove from the followee set.
Why This Works: The global timestamp gives every tweet a unique ordering. Sorting all collected tweets by timestamp descending produces the correct most-recent-first feed. The user always sees their own tweets because they are implicitly included in their own followee set.
Watch Out For: User must see their own tweets (add self to followees). unfollow should not crash if the user isn't following anyone. At most 10 tweets are returned. The brute sort approach is accepted but a heap-based k-way merge is optimal for large-scale feeds.

#460 · LFU Cache

Hard Design Hash Map Linked List

The Problem: Design an LFU (Least Frequently Used) cache with O(1) get and put operations. When the cache reaches capacity, evict the least frequently used key. If there is a tie, evict the least recently used among them.

Example: capacity=2, put(1,1), put(2,2), get(1) → 1, put(3,3) evicts key 2, get(2) → -1

Key Insight: Three HashMaps: key→value, key→freq, freq→OrderedSet. Track minFreq to find eviction candidates in O(1). When a key is accessed, move it from its current frequency bucket to the next one.
Brute Force Approach

Store value, frequency, and timestamp in a single map. On eviction, linear scan all entries to find the one with minimum frequency (tie-break by oldest timestamp). This gives O(n) per put when eviction is needed.

How to Think About It

  1. Maintain three maps: keyVal, keyFreq, and freqKeys (frequency → ordered set of keys).
  2. Track minFreq to locate eviction candidates in O(1).
  3. get: increase the key's frequency, move it between frequency buckets, update minFreq if the old bucket is now empty.
  4. put (existing key): update value and apply the same frequency-increase logic as get.
  5. put (new key, at capacity): evict the oldest key from the minFreq bucket, then insert with freq=1 and set minFreq=1.
Why This Works: The freq → orderedSet map allows O(1) eviction by always removing from the minFreq bucket. Moving keys between frequency groups is O(1) because we have direct pointers via the key maps. New inserts always set minFreq = 1 since a fresh key has frequency 1.
Watch Out For: Update minFreq when a frequency bucket becomes empty. New inserts always set minFreq = 1. Handle capacity = 0 as an edge case. When updating an existing key, don't forget to increase its frequency.

public:
    int findMinArrowShots(vector<vector<int>>& points) {
        sort(points.begin(), points.end(), [](auto& a, auto& b) {
            return a[1] < b[1];
        });
        int arrows = 1;
        long long arrowPos = points[0][1];
        for (int i = 1; i < (int)points.size(); i++) {
            if ((long long)points[i][0] > arrowPos) {
                arrows++;
                arrowPos = points[i][1];
            }
        }
        return arrows;
    }
};

Step-by-Step

  1. Sort intervals by end coordinate.
  2. Initialize arrows = 1 and arrow_pos = first interval’s end.
  3. For each subsequent interval: if start > arrow_pos, increment arrows and update arrow_pos = end.
  4. Return arrows.

Why it works: Shooting at the end of the earliest-ending balloon maximizes overlap with subsequent balloons. Any balloon with start <= arrow_pos is already burst.

Common Pitfalls: Sort by END coordinate, not start — sorting by start does not yield the optimal greedy solution. In C++, use long long for arrowPos to avoid overflow with INT_MIN/INT_MAX edge cases. Touching intervals (e.g., [1,2] and [2,3]) count as overlapping — one arrow at x=2 bursts both.

Language-Specific Features

  • Python: lambda x: x[1] as sort key for clean end-coordinate sorting.
  • Go: sort.Slice with custom comparison function for flexible sorting.
  • TypeScript: Arrow function (a, b) => a[1] - b[1] as comparator.
  • Rust: sort_by_key(|p| p[1]) for concise sort by extracted key.
  • C++: long long arrowPos to avoid overflow with INT_MIN/INT_MAX values.
