#739 Daily Temperatures MediumMonotonic StackArray~6 days since review · rep 2 → 3, interval → 15

The ProblemGiven a list temperatures, return an array ans where ans[i] is the number of days you must wait after day i for a warmer temperature. If no future day is warmer, put 0.
Quick ReminderKeep a stack of indices whose answer is still pending, with temperatures strictly decreasing. When a warmer day arrives it resolves every colder index on top.
Key InsightThe brute force rescans the future for every day. The insight: a monotonic decreasing stack of unresolved indices lets each warmer day pop and resolve all pending colder days in one pass — each index is pushed and popped once.
Try to recall: Can you do it in one pass in under 10 minutes? What exactly does the stack hold — temperatures or indices — and why indices?

class Solution(object):
    def dailyTemperatures(self, temperatures):
        ans = []
        # O(n^2)
        for i in range(len(temperatures)):
            d = 0
            for j in range(i + 1, len(temperatures)):
                if temperatures[j] > temperatures[i]:
                    d = j - i
                    break
            ans.append(d)
        return ans

The ProblemA sorted array of distinct integers was rotated at an unknown pivot. Given nums and a target, return its index or -1, in O(log n) time.
Quick ReminderAt each midpoint, one side is always sorted. Decide which half is sorted, test whether the target lies inside its range, then discard the other half.

#918 Maximum Sum Circular Subarray MediumDynamic ProgrammingKadane~418 days since review · rep 0 → 1, interval → 1 (refresh)

The ProblemGiven a circular integer array nums (the last element wraps to the first), return the maximum possible sum of a non-empty subarray. A subarray may wrap around but may not include any element more than once.
Quick ReminderAnswer is either a normal max subarray, or the total minus the minimum subarray (the "wrap" case). Run Kadane for both max and min in one pass.
Key InsightA wrapping subarray equals total − (some middle subarray). To maximize the wrap, minimize the middle. So compute both the max subarray (standard Kadane) and the min subarray; the answer is max(maxSub, total − minSub). The one trap: if all numbers are negative, total − minSub = 0 (empty), which is illegal, so fall back to maxSub.
Try to recall: Why must you special-case the all-negative array, and what value would the wrap formula wrongly produce there?

class Solution(object):
    def maxSubarraySumCircular(self, nums):
        max_current = nums[0]
        min_current = nums[0]
        max_sum = nums[0]
        min_sum = nums[0]
        total = nums[0]
        for i in range(1, len(nums)):
            num = nums[i]
            total += nums[i]
            max_current = max(max_current + num, num)
            max_sum = max(max_sum, max_current)
            min_current = min(min_current + num, num)
            min_sum = min(min_sum, min_current)
        if max_sum < 0:
            return max_sum
        return max(total - min_sum, max_sum)

The ProblemA robot at the top-left of an m × n grid can only move right or down. Count the distinct paths to the bottom-right corner.
Quick ReminderPaths into a cell = paths from above + paths from the left. First row and first column are all 1. A single rolling row suffices for O(n) space.
Key InsightRecomputing each cell recursively re-solves the same subproblems exponentially. The insight: dp[r][c] = dp[r-1][c] + dp[r][c-1] fills bottom-up; since a row only depends on the row above and the cell to the left, you can collapse the 2D table into a single reused 1D array.

class Solution(object):
    def uniquePaths(self, m, n):
        dp = [[0]*n for _ in range(m)]
        dp[0][0] = 1
        for r in range(m):
            for c in range(n):
                if r > 0:
                    dp[r][c] += dp[r-1][c]
                if c > 0:
                    dp[r][c] += dp[r][c-1]
        return dp[-1][-1]

The ProblemGiven two strings text1 and text2, return the length of their longest common subsequence (characters in order, not necessarily contiguous), or 0 if none exists.
Quick Reminder2D DP over prefixes: if the two current characters match, extend the diagonal by 1; otherwise take the better of dropping one character from either string.

The ProblemGiven the root of a binary tree, return the length of its diameter: the number of edges on the longest path between any two nodes. This path may or may not pass through the root. The node count is in [1, 10000] and each val is in [-100, 100].

Brute Force — O(n) time

Traverse the whole tree and group every node's value by its depth into per-level lists. After visiting all nodes, walk the levels top to bottom and take the last value recorded at each depth. This stores every node even though only one per level survives.

    def rightSideView(self, root):

            dfs(node.right, depth + 1)

Why optimize? Buffering every value per level uses O(n) extra space to keep information that is immediately thrown away—only one node per depth is ever part of the answer.

Key InsightThe brute force keeps every value at every level then discards all but the last; but you only ever need the rightmost node per depth. Do a BFS and record the last node dequeued in each level, or a DFS that visits the right child first and records the first node seen at each new depth.
How to Think About It
  1. Recognize the answer has exactly one entry per depth: the node farthest right.
  2. With BFS, process the tree level by level, snapshotting the level size before expanding.
  3. The last node you pop within a level is the one visible from the right—append its value.
  4. Alternatively DFS right-child-first and, the first time you reach a new depth, that node is the rightmost; record it.
Why This WorksWithin a single level, nodes are ordered left to right, so the final node processed by a left-to-right BFS is the rightmost—exactly what a viewer on the right sees. The right-first DFS reaches each depth's rightmost node before any node to its left, so the first sighting per depth is correct and complete.
Watch Out For
  • Handling the empty tree, which must return an empty list.
  • In BFS, reading the queue length after you start enqueuing children, which corrupts the per-level boundary.
  • In right-first DFS, recording every node instead of only the first one seen at each new depth.

Why optimize? The loop is already fast, but it touches every bit sequentially in 32 steps; a divide-and-conquer swap of whole bit-groups reaches the same answer in a handful of constant masked operations, which matters when the function is called repeatedly (the follow-up).

Key InsightThe brute force moves one bit per iteration; but reversal can be done by swapping entire halves at once. Swap the high 16 bits with the low 16 bits, then swap 8-bit groups, then 4-bit, 2-bit, and 1-bit groups—each step uses a fixed mask like 0xffff0000 / 0x0000ffff and shifts, reversing all 32 bits in only 5 masked steps.
How to Think About It
  1. Notice that reversing a sequence can proceed by swapping symmetric halves recursively rather than element by element.
  2. First swap the top 16 bits with the bottom 16 bits using masks and a 16-bit shift.
  3. Within each half swap adjacent 8-bit bytes, then 4-bit nibbles, then 2-bit pairs, then adjacent single bits.
  4. Each level uses a complementary mask pair (isolate the group to move left vs. the group to move right) combined with |; after the finest level the word is fully reversed.
Why This WorksReversal is symmetric: bit i ends at position 31 - i. Swapping groups of size 16, then 8, 4, 2, 1 composes into exactly that permutation—each pass sends every bit halfway toward its mirror position, and after the log-sized number of passes every bit has arrived at its reversed index. The masks guarantee no two groups collide as they move.
Watch Out For
  • Sign extension: in languages without native unsigned 32-bit ints, use an unsigned right shift and coerce back to 32 bits.
  • Getting the two masks of a pair backwards, which mixes groups instead of swapping them.
  • Using the wrong shift distance for a level (it must equal the group size being swapped).

The ProblemGiven an m x n binary grid where 1 is land and 0 is water, an island is a group of 1s connected 4-directionally. Return the area (number of cells) of the largest island, or 0 if none exists. Cells outside the grid count as water.
Brute Force — O(m·n) time

Scan every cell. When a fresh land cell is found, flood-fill its whole island with DFS, counting cells and marking them visited so they are never recounted. Track the maximum area seen across all islands.

Key InsightA naive scan might recount cells across overlapping traversals; the insight is that marking each visited land cell as water the moment you enter it guarantees each cell contributes to exactly one island count, turning the whole grid into a single linear pass.
How to Think About It
  1. Iterate over every cell of the grid in row-major order.
  2. On an unvisited land cell, launch a flood fill (DFS or BFS) that returns the connected area.

Key InsightThe brute force rescans all cells each minute to find the rot frontier; the insight is multi-source BFS — seed a queue with every initially rotten orange and expand layer by layer, where each BFS level equals one minute, so the frontier is maintained incrementally instead of rediscovered.
How to Think About It
  1. Count fresh oranges and enqueue every already-rotten orange as the starting frontier.
  2. Process the queue one full level per minute, rotting fresh neighbors and enqueuing them.
  3. Decrement the fresh count as each orange rots; increment a minute counter per completed level.

Key InsightThe brute force re-runs a full path search from every course, re-traversing shared subgraphs; the insight is that "can finish" is exactly "the graph is a DAG," which a topological sort settles in one pass — either Kahn's algorithm repeatedly removing zero-indegree nodes, or a DFS that permanently marks fully-explored nodes so they are never re-examined.
How to Think About It
  1. Model courses as nodes and each prerequisite as a directed edge, building an adjacency list.
  2. Recognize the question reduces to detecting whether that directed graph is acyclic.
  3. Compute indegrees and repeatedly remove nodes with indegree zero (Kahn), or DFS with a three-state (unvisited / on-path / done) coloring.
  4. If every node is eventually removed (or no back-edge is found), it is a DAG — return true; otherwise a cycle blocks completion.
Why This WorksA set of courses is completable iff the prerequisite graph contains no directed cycle. Why optimize? Generating all 2ⁿ masks and deduping through a hash set wastes work building then discarding many identical subsets; a backtracking traversal that skips duplicate branches never generates a repeat in the first place.

Key InsightBrute force produces duplicates and filters them afterward; the insight is to sort first and, at each recursion depth, skip any element equal to the previous one that was not chosen at this level — this prunes duplicate subsets before they are ever formed.
How to Think About It
  1. Sort nums so equal values are adjacent.
  2. Backtrack over start indices, appending the current partial subset to the results at every node.
  3. In the loop over the next index, skip an element when it equals its predecessor and that predecessor is at the same level (i > start).
  4. Recurse with the element included, then undo the choice and continue.
Why This WorksAfter sorting, duplicate subsets differ only by which copies of equal values they use. Fixing that at each level the first available duplicate must be chosen before any later identical one removes the ambiguity: skipping nums[i] == nums[i-1] when i > start forbids selecting a later duplicate without its earlier twin, so each distinct multiset of choices is generated exactly once.
Watch Out For
  • Forgetting to sort — the duplicate-skip test relies on equal elements being adjacent.
  • Using the wrong skip condition (i > 0 instead of i > start), which wrongly drops valid subsets containing repeated values.
  • Appending a reference to the mutable working list instead of a copy, so later mutations corrupt stored results.

The ProblemGiven an m x n grid board of characters and a string word, return true if word can be formed from sequentially adjacent (horizontally or vertically) cells, where each cell may be used at most once in a single path. Otherwise return false.
Brute Force — O(m·n·4ᴱ) time

If the whole word is matched along some path, return true; the extra visited set (versus in-place marking) is what makes it the naive variant.

class Solution:
    def exist(self, board: List[List[str]], word: str) -> bool:
        m, n = len(board), len(board[0])
        def dfs(r, c, k, used):
            if k == len(word):
                return True
            if (r < 0 or r >= m or c < 0 or c >= n
                    or (r, c) in used or board[r][c] != word[k]):
                return False
            used.add((r, c))
            for dr, dc in ((0,1),(0,-1),(1,0),(-1,0)):
                if dfs(r+dr, c+dc, k+1, used):

            used.discard((r, c))
            return False
        for r in range(m):
            for c in range(n):
                if dfs(r, c, 0, set()):
                    return True
        return False

            if let Some(n) = node {
                n.val = v;
                node = n.next.as_mut();

class Solution {
public:
    void reorderList(ListNode* head) {
        if (!head) return;
        vector<ListNode*> nodes;
        for (ListNode* cur = head; cur; cur = cur->next)
            nodes.push_back(cur);
        int i = 0, j = nodes.size() - 1;
        while (i < j) {
            nodes[i]->next = nodes[j];
            i++;
            if (i == j) break;
            nodes[j]->next = nodes[i];
            j--;
        }
        nodes[i]->next = nullptr;
    }
};

Why optimize? Storing every node in an array costs O(n) extra memory; the structure of a linked list allows the same reordering with only pointer manipulation and O(1) space.

Key InsightBrute force buys random access with an O(n) array. Instead, achieve the same front-to-back weaving using three linked-list techniques with no array: find the middle with slow/fast pointers, reverse the second half in place, then merge the two halves alternately. The reversed back half lets you consume nodes from the tail without indexing.
How to Think About It
  1. Use a slow pointer advancing one step and a fast pointer two steps to locate the midpoint.
  2. Split the list and reverse the second half by flipping next pointers.
  3. Merge the first half and the reversed second half, alternating one node from each.
  4. Stop when the second half is exhausted, leaving the list correctly interleaved.
Why This WorksReversing the back half turns the desired pairing (L0, Ln), (L1, Ln-1), ... into a simple alternating merge of two sequences read front to back. The slow/fast split guarantees the halves differ by at most one node, so the merge consumes them evenly and every original node is placed exactly once, all through pointer rewiring in constant extra space.
Watch Out For
  • Sever the link between the two halves before reversing, or you create a cycle.
  • Handle odd and even lengths: the middle node belongs to the first half.
  • During the merge, save each node's next before overwriting it, or you lose the rest of the list.

#142 Linked List Cycle II MediumLinked ListTwo PointersFloyd

The ProblemGiven the head of a linked list, return the node where a cycle begins, or null if there is no cycle. A cycle exists if some node can be reached again by continuously following next pointers. You must not modify the list, and an O(1)-space solution is expected.
Brute Force — O(n) time, O(n) space

Traverse the list while recording every node already seen in a hash set. The first node encountered that is already in the set is the entrance of the cycle. If traversal reaches a null pointer without any repeat, the list is acyclic and the answer is null.

pub fn detect_cycle(head: Option<Box<ListNode>>) -> bool {
    let mut seen: HashSet<*const ListNode> = HashSet::new();

        unordered_set<ListNode*> seen;

            seen.insert(cur);

    }
};

Why optimize? The hash set stores up to O(n) node references; Floyd's cycle detection finds the same entrance using only two pointers and constant space.

Key InsightBrute force spends O(n) memory remembering every visited node. Floyd's tortoise and hare instead detects a cycle by moving a slow pointer one step and a fast pointer two steps until they meet. The number-theoretic relation between the distances then lets you reset one pointer to the head and walk both at equal speed to the cycle's start, using no extra memory.
How to Think About It
  1. Advance slow by one and fast by two; if fast reaches null, there is no cycle.
  2. If they meet inside the loop, a cycle exists.
  3. Reset one pointer to head and leave the other at the meeting point.
  4. Move both one step at a time; the node where they meet again is the cycle's entrance.
Why This WorksLet the distance from head to the cycle start be a and the meeting point be b nodes into the loop of length c. When the pointers meet, the fast pointer has traveled twice the slow pointer's distance, which algebraically forces a to equal the remaining distance from the meeting point back to the entrance (modulo c). Walking one pointer from the head and one from the meeting point at equal speed therefore lands both on the entrance simultaneously.
Watch Out For
  • Check fast and fast.next before dereferencing to avoid null-pointer errors on acyclic lists.
  • Start both pointers at the head, and only reset after a confirmed meeting.
  • Distinguish "no cycle" (fast hits null) from "cycle found" (pointers meet) before the second phase.
