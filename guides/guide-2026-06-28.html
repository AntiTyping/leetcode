#121 Best Time to Buy and Sell Stock EasyArraysDP15 days since solve · rep 3 → 4, interval → 38

The ProblemGiven an array prices where prices[i] is the price of a stock on day i, find the maximum profit from buying on one day and selling on a later day. Return 0 if no profit is possible.
Quick ReminderSingle pass tracking the cheapest price seen so far; the best profit ending today is today's price minus that minimum.
Key InsightBrute force compares every buy/sell pair. Instead, sweep left to right holding the minimum price so far — each day's best sale is price - min_so_far, so one pass suffices.
Try to recall: Can you do it in a single pass without a nested loop, keeping only two scalars? What is the time/space complexity?
Language challenge: Last time you solved this in Python. Try it in Go this time.
Brute: O(n^2)→Optimal: O(n) time, O(1) space
Input: prices = [7,1,5,3,6,4]
Output: 5
Explanation: Buy on day 1 (price 1), sell on day 4 (price 6): profit 5.
Input: prices = [7,6,4,3,1]
Output: 0
Explanation: Prices only fall, so no profitable transaction exists.
Input: prices = [2,4,1]
Output: 2
Explanation: Buy at 2, sell at 4 for profit 2. The later low of 1 is useless since nothing follows it.

Output: false

class Solution(object):
    def minSubArrayLen(self, target, nums):
        p = [0] * len(nums)
        p[0] = nums[0]
        for i in range(1, len(nums)):
            p[i] = p[i - 1] + nums[i]
        min_l = len(nums) + 1
        i, j = 0, 0

                    min_l = min(min_l, j - i + 1)
                    i += 1
                else:
                    j += 1

func isPalindrome(s string) bool {
    isAlnum := func(c byte) bool {
        return (c >= 'a' && c <= 'z') || (c >= 'A' && c <= 'Z') || (c >= '0' && c <= '9')

        if c >= 'A' && c <= 'Z' {
            return c + 32
        }
        return c

            i++
        }
        for i < j && !isAlnum(s[j]) {
            j--

            return false
        }
        i++
        j--
    }

}

Why optimize? With n up to ~2 billion, a linear scan can make billions of API calls and time out.

Key InsightThe brute force rechecks every version even though the predicate isBadVersion is monotonic: it is false...false then true...true with a single flip point. Binary search finds that boundary in O(log n) calls instead of re-testing each version one by one.
How to Think About It
  1. Recognize the answer is the leftmost index where the predicate becomes true.
  2. Keep a search window [lo, hi] that always contains the first bad version.
  3. Compute the midpoint as lo + (hi - lo) / 2 to dodge integer overflow.
  4. If mid is bad, the boundary is at mid or earlier, so move hi = mid.
  5. If mid is good, the boundary is strictly after mid, so move lo = mid + 1.
  6. Stop when lo == hi; that index is the first bad version.
Why This WorksBecause badness is monotonic, testing the midpoint splits the window into a half that cannot contain the boundary and a half that must. Each step preserves the invariant that [lo, hi] still brackets the first bad version, while halving the window size, so the loop converges on exactly the flip point.
Watch Out For
  • Computing (lo + hi) / 2 directly overflows when both are near 2^31 - 1; use lo + (hi - lo) / 2.
  • When mid is bad, do not do hi = mid - 1; that can skip the boundary. Use hi = mid.
  • Getting the loop condition wrong (lo < hi vs lo <= hi) leads to an off-by-one or extra API call.

How to Think About It

  2. Note the predicate is false for tiny k and true for large k — a monotonic flip.
  3. Set the search window to lo = 1, hi = max(piles) (speed above max never helps).
  4. Binary search: if hours(mid) ≤ h, this speed is feasible, so try smaller (hi = mid).
  5. Otherwise it is too slow, so raise the floor (lo = mid + 1).
  6. Converge until lo == hi; that is the minimum speed.
Why This WorksEating faster can never increase the hours for any pile, so the total hours(k) is non-increasing in k; the feasibility predicate therefore flips exactly once from false to true. Key InsightThe brute force recomputes the minimum from scratch every call, throwing away work. Instead, remember the running minimum as of each push — either by pairing each value with the min-so-far, or by keeping a parallel stack of minimums — so getMin is just reading the top in O(1).
How to Think About It
  1. Observe that the minimum can only change on push or pop, not on getMin.
  2. Alongside each pushed value, store the minimum of the stack at that moment.
  3. On push(val), the stored min is min(val, current_min).
  4. On pop, discard both the value and its paired min so the previous min resurfaces.
  5. top reads the value part of the top entry; getMin reads the min part.
  6. Every operation touches only the top of the stack, so all are O(1).
Why This WorksEach entry records the minimum over exactly the prefix of pushes that are still present when that entry is on top. Because a stack removes elements in reverse order of insertion, popping an entry restores the exact state (including the recorded min) that existed before its push, so the top's stored minimum is always the true minimum of the current contents.
Watch Out For
  • If you use a separate min-stack, remember to pop it in lockstep with the main stack.
  • Duplicate minimums: pushing a value equal to the current min must still be tracked, or an early pop loses it — use ≤ semantics if you optimize the min-stack.
  • Values can be negative and up to 32-bit limits; do not initialize the min with 0 or a wrong sentinel.

The ProblemGiven an array asteroids, each element's sign is its direction (positive = moving right, negative = moving left) and its absolute value is its size. Asteroids move at the same speed. Return the state of the array after all collisions resolve. Constraints: 2 ≤ asteroids.length ≤ 10^4, -1000 ≤ asteroids[i] ≤ 1000, asteroids[i] != 0.
Brute Force — O(n^2) time, O(n) space

Repeatedly scan the array for an adjacent colliding pair — a positive value immediately followed by a negative value. Resolve that one collision (remove the loser, or both on a tie), then restart the scan. Repeat until a full pass finds no colliding pair, meaning the array is stable.

Python

            if a[i] > 0 and a[i + 1] < 0:      # right meets left

                else:

                break
    return a

Key InsightThe brute force rescans and shifts the whole array after every explosion. A stack of surviving asteroids resolves everything in one pass: each new asteroid interacts only with the current top, so cascading collisions unwind naturally without ever re-scanning from the start.
How to Think About It
  1. Sweep left to right, maintaining a stack of asteroids that have survived so far.
  2. A right-moving (positive) or into-empty-space asteroid can never collide with survivors on its left, so just push it.
  3. A left-moving (negative) asteroid may collide with a positive top; enter a resolve loop.
  4. While the top is positive and smaller, pop it (it explodes) and keep checking the new top.
  5. If the top is positive and equal, pop it and stop (both explode); if the top is bigger, the incoming one explodes — stop without pushing.
  6. If the resolve loop empties the positives (or the top is negative), push the incoming asteroid as a survivor.
Why This WorksOnly a positive asteroid on top followed by an incoming negative one can collide; all other adjacencies are stable. Processing collisions against the top mirrors the physical order in which they would actually happen, and each asteroid is pushed once and popped at most once, so every collision outcome is applied exactly as the rules dictate while total work stays linear.
Watch Out For
  • The equal-size tie destroys both asteroids — pop the top and do not push the incoming one.
  • A single incoming negative can destroy several stacked positives; use a loop, not a single comparison.
  • Only positive-top vs. negative-incoming collides; a negative top or an incoming positive never collides, so push directly.

Why optimize? Re-sorting the counts every round makes the simulation slow when the answer (total intervals, inflated by idles) is large.

Key InsightThe simulation re-derives the schedule step by step, but the answer is determined by the most frequent task alone. That task forms a skeleton of (maxFreq - 1) gaps each n + 1 wide, plus a final block for every task tied at maxFreq: (maxFreq - 1) * (n + 1) + (#ties). The result is max(len(tasks), that formula) — no simulation needed.
How to Think About It
  1. Count frequencies and find maxFreq, the largest count.
  2. Lay out the busiest task as anchors: maxFreq copies create maxFreq - 1 gaps.
  3. Each gap must span n + 1 slots (the anchor plus its cooldown), giving (maxFreq - 1) * (n + 1).
  4. Add one extra slot for each other task type also at maxFreq (they ride in the final column).
  5. If there are enough distinct tasks to fill all idle gaps, the schedule has no idles — the answer is just len(tasks).
  6. Return max(len(tasks), formula).
Why This WorksThe most frequent task cannot be scheduled tighter than one copy every n + 1 intervals, so the formula is a hard lower bound on the timeline length. Filling the gaps with other tasks (or idles when tasks run out) achieves exactly that bound, unless there are so many distinct tasks that they overflow the gaps — in which case no idle is ever needed and the count of tasks itself is the minimum. Taking the max of both captures every case.
Watch Out For
  • Forgetting to max with len(tasks): when many task types exist and n is small, the formula can undercount.
  • Count all task types tied at maxFreq, not just one, for the final block.
  • When n = 0 the answer is always len(tasks) — make sure the formula degrades to that.

Key InsightThe brute force re-walks an entire candidate subtree at every node, redoing overlapping comparisons. How to Think About It
  1. Write a helper same(a, b): both null → true; exactly one null → false; else compare values and recurse on left and right pairs.
  2. Traverse root (DFS). At each visited node, ask same(node, subRoot).
  3. If any node returns true, the whole answer is true — short-circuit up the recursion.
  4. Handle the empty root case: nothing to match against, so return false.
  5. For the linear approach, serialize each tree with delimiters and sentinel nulls, then run a substring search (e.g. KMP) of subRoot’s encoding inside root’s.
Why This WorksA subtree match requires identical structure and identical values at every position, which is exactly what a synchronized dual-recursion on (a, b) checks: it only returns true when both trees run out of nodes at the same places with equal values everywhere. Because we test that predicate at every node of root, if any node roots an identical subtree we are guaranteed to find it; if none does, every test fails and we correctly return false.
Watch Out For
  • Using value equality alone without a matching null/non-null shape check — a shorter tree can falsely “match” a prefix of a taller one.
  • Forgetting that subRoot must match an entire subtree down to the leaves, not just a connected fragment.
  • When serializing, using a plain in-order/pre-order without null markers or delimiters — different trees can produce the same token sequence, and value 12 vs 1,2 can collide without separators.

The ProblemGiven a binary search tree and two of its nodes p and q, return their lowest common ancestor (LCA): the deepest node that has both p and q as descendants, where a node is allowed to be a descendant of itself. All node values are unique, and both p and q are guaranteed to exist in the tree.
Brute Force — O(n) time, O(n) space

        def path(node, target, acc):

Why optimize? Building and storing two full paths costs O(n) extra space and throws away the sorted structure the BST hands you for free.

Key InsightThe brute force reconstructs full root-to-node paths and never uses the ordering that makes a BST a BST. The insight: at any node, the BST invariant tells you exactly where p and q live. If both values are smaller than the current node, both descendants are in the left subtree; if both are larger, both are on the right. The first node where they diverge (one on each side, or one equals the node) is the split point — and that split point is the LCA. This walks a single root-to-leaf path in O(h) with O(1) space.
How to Think About It
  1. Start at the root as the current node.
  2. If both p.val and q.val are less than node.val, move to node.left.
  3. Else if both are greater than node.val, move to node.right.
  4. Otherwise they split here (or one equals node) — return the current node.
  5. Repeat with a simple loop; no recursion or extra arrays are required.
Why This WorksIn a BST every value in the left subtree is strictly less than the node and every value in the right subtree is strictly greater. So as long as both targets sit on the same side, their common ancestor must be deeper on that side, and descending is safe. The moment the two targets fall on opposite sides of the current node — or one of them is the current node — no deeper node can contain both, so the current node is the lowest node that does. Because we only ever descend toward the region containing both, the first such split is guaranteed to be the LCA.
Watch Out For
  • Assuming p.val < q.val; the inputs are not ordered, so test both < and > conditions rather than a single range check that assumes an order.
  • Forgetting that a node can be its own ancestor — if node.val equals p or q, you have already found the LCA and must stop.
  • Accidentally solving the general binary-tree LCA (#236) recursion here — it works but wastes the BST property and its O(h) guarantee.

        int leftSize = mid - is;
        root->left = build(pre, ps + 1, ps + leftSize, in, is, mid - 1);
        root->right = build(pre, ps + leftSize + 1, pe, in, mid + 1, ie);
        return root;
    }
    TreeNode* buildTree(vector<int>& preorder, vector<int>& inorder) {
        return build(preorder, 0, preorder.size() - 1, inorder, 0, inorder.size() - 1);
    }
};

Why optimize? For a degenerate (skewed) tree the linear inorder search plus per-level slicing sums to O(n^2) — too slow and memory-heavy once n reaches a few thousand.

Key InsightThe brute force wastes time two ways: it linear-scans inorder to locate the root on every call, and it copies fresh subarrays through slicing. Two changes remove both costs. First, precompute a hashmap value → index-in-inorder so root location is O(1). Second, thread a single moving pointer through preorder (roots come out in exactly the order the recursion needs them) and pass index ranges instead of slices. That yields a single O(n) pass.
How to Think About It
  1. Build a hashmap from each inorder value to its index once, up front.
  2. Keep a shared cursor into preorder; the next root is preorder[cursor++].
  3. Recurse with an (inLeft, inRight) range describing the current subtree in inorder.
  4. For the current root, look up its inorder index; that splits the range into left and right subranges.
  5. Build the left subtree first (preorder visits the whole left subtree before the right), then the right.
  6. Stop when inLeft > inRight — that subrange is empty, return null.
Why This WorksA preorder traversal always emits a node before either of its subtrees, so the current front of preorder is always the root of the subtree being built. In an inorder traversal a node’s left subtree appears entirely before it and its right subtree entirely after it, so the root’s inorder index cleanly partitions the remaining values into the two subtrees. Because all values are unique, that index is unambiguous, and consuming preorder left-first keeps the cursor aligned with each subtree the recursion enters — so every node is placed exactly once in its correct position.
Watch Out For
  • Building the right subtree before the left — the preorder cursor must consume the entire left subtree first or every subsequent root is wrong.
  • Mixing up which array the split index applies to: the hashmap index is a position in inorder, and the left-subtree size (not the raw index) is what advances the preorder range.
  • Off-by-one errors in the empty-range base case (inLeft > inRight) or in the inclusive/exclusive bounds of the subranges.

Why optimize? The triple loop is O(n^3) and the closure matrix is O(n^2) space — wasteful when a single sweep that floods each component already answers the question.

Key InsightThe brute force computes reachability between every pair of cities before counting, which is far more information than needed. A province is simply a connected component, and you can count components by flooding: pick any unvisited city, mark its entire component reachable via DFS/BFS, and increment a counter. Alternatively, union every connected pair with a union-find structure and count the distinct roots at the end. Either way you touch each connection a constant number of times.
How to Think About It
  1. Keep a visited array of size n and a province counter at zero.
  2. Iterate over cities 0..n-1; skip any already visited.
  3. When you hit an unvisited city, increment the counter and start a DFS/BFS from it.
  4. From a city, follow every j with isConnected[i][j] == 1 to visit the whole component, marking each as visited.
  5. The counter ends equal to the number of connected components.
  6. Union-find variant: for each pair i < j that is connected, union them; the answer is the number of roots where find(x) == x.
Why This WorksEach connected component is exactly one province, and a flood-fill started from any of its members visits precisely that component and no other — because it only ever crosses edges that exist. By only starting a new flood (and only then incrementing the counter) from a city that is still unvisited, every component is counted once and only once: the first time its flood begins. The union-find variant reaches the same conclusion because unioning all connected pairs collapses each component to a single root, so the number of distinct roots equals the number of components.
Watch Out For
  • Counting every visited city instead of every started flood — you must increment the province count once per component, not once per node.
  • Re-scanning already-visited cities or forgetting to mark a city visited before/at enqueue, causing infinite loops on cycles.
  • Union-find without path compression or union by rank can degrade to O(n) per operation on adversarial inputs; also remember the diagonal isConnected[i][i] is always 1 and should not create a self-referential surprise.

The ProblemA tree on n nodes labelled 1..n had exactly one extra edge added, producing a graph with n edges given as edges, where edges[i] = [ai, bi]. Return an edge that can be removed so the result is a tree of n nodes. If several answers exist, return the one that occurs last in the input.
Brute Force — O(n^2) time, O(n) space

Try removing each edge (scanning from last to first so the latest candidate wins), rebuild the graph from the remaining n-1 edges, and DFS from node 1: if the removal leaves all n nodes connected, that edge is removable. Why optimize? Rebuilding the graph and running a fresh DFS for every one of the n edges is O(n^2); a single union-find pass answers it in near-linear time.

Key InsightThe brute force keeps rebuilding the whole graph to test connectivity edge by edge. But adding an edge to a tree creates exactly one cycle, and the redundant edge is any edge on that cycle — specifically, when we process edges left to right, the first edge whose two endpoints already belong to the same connected set is the one that closes the cycle. Union-find detects this in one pass: union the endpoints of each edge; the first union that fails (both endpoints share a root) is the answer.
How to Think About It
  1. Initialize a union-find over nodes 1..n, each its own parent.
  2. Process the edges in input order.
  3. For edge (u, v), compute find(u) and find(v).
  4. If the two roots are equal, u and v are already connected — this edge closes a cycle, so return it.
  5. Otherwise union the two sets and continue.
Why This WorksBecause the input is a tree plus one edge, the graph has exactly one cycle, and every edge except those on the cycle connects two previously separate components. Processing edges in order, each non-cycle edge merges two distinct sets and a valid union succeeds; the redundant edge is the unique one whose endpoints are already in the same set, which is exactly the condition find(u) == find(v) detects. Scanning left to right and returning at the first such edge yields the cycle-closing edge that appears last among the cycle’s edges in the input, satisfying the tie-break rule.
Watch Out For
  • Nodes are 1-indexed — size the parent array as n + 1 (or offset by one) or you will index out of range.
  • Unioning before checking whether the endpoints already share a root — do the find comparison first, then union only if they differ.
  • Returning too early or iterating in reverse: the correct answer is the first edge (in input order) that closes a cycle, which already respects the “last in input” tie-break.

Backtracking explores a decision tree depth-first: make a choice, recurse, undo. The new skills here are de-duplicating choices at each level and in-place visited marking on a grid. Interleaved.

Why optimize? The dedupe set stores up to 2^n subsets and hashes each in O(n), doubling the work — and it only papers over duplicates instead of never generating them.

Key InsightBrute force generates duplicate subsets and then throws them away redundantly. Instead, sort so equal values are adjacent, and during backtracking skip a value equal to its left neighbor at the same recursion level (if i > start and nums[i] == nums[i-1]: continue) — so duplicate branches are never explored at all.
How to Think About It
  1. Sort nums so identical numbers sit next to each other.
  2. Backtrack with a start index; at each entry, record the current path as one subset.
  3. Loop i from start to n-1, choosing nums[i] next.
  4. If i > start and nums[i] == nums[i-1], skip it — that value was already tried at this level.
  5. Append nums[i], recurse with start = i + 1, then pop to undo.
  6. Collect every path visited; those are all unique subsets.
Why This WorksAfter sorting, all copies of a value form a contiguous run. The first copy at a given level launches every branch that begins with that value; any later copy at the same level would launch an identical branch, so skipping it removes exactly the duplicates while preserving branches that use multiple copies (those come from deeper levels where i == start). Every distinct subset is therefore produced exactly once.
Watch Out For
  • The skip condition is i > start, not i > 0 — using the wrong bound wrongly drops subsets that legitimately reuse duplicate values.
  • Forgetting to sort first: the adjacency the skip relies on disappears.
  • Storing a reference to the mutable path instead of a copy, so all recorded subsets end up identical.

Key InsightSwapping values sidesteps the real task. Instead, put a dummy node before the head and repeatedly rewire pointers for each pair: prev->second->first->rest, then advance prev two nodes forward — the nodes themselves move, values never change.
How to Think About It
  1. Create a dummy whose next is head; keep a prev pointer starting at dummy.
  2. While prev.next and prev.next.next both exist, name them first and second.
  3. Point first.next to second.next (the rest of the list).
  4. Point second.next to first, and prev.next to second.
  5. Advance prev to first (now the second node of the swapped pair).
  6. Return dummy.next as the new head.

  • Rewiring the three pointers in the wrong order and losing the reference to second.next.
  • Advancing prev by only one node, which re-swaps or corrupts the next pair.
  • Returning head instead of dummy.next — head is now the second node.

  • Looping the wrong number of times — it is right - left insertions, not right - left + 1.
  • Forgetting the dummy when left == 1, which otherwise needs a special case for the head.

Why optimize? With up to 200 elements, 2^n branching is astronomically slow — but many recursive calls repeat the same (index, remaining) state.

Key InsightThe recursion recomputes the same "can I still reach sum s?" questions exponentially. Reframe it as 0/1 knapsack: the answer is yes iff some subset sums to total/2. Track a boolean set/array dp[s] = "sum s is reachable", updating it once per element — collapsing the exponential search into O(n · sum).
How to Think About It
  1. Compute total; if it is odd, return false — no equal split is possible.
  2. Set target = total / 2; you now need a subset summing to target.
  3. Keep a boolean array dp of size target + 1 with dp[0] = true.
  4. For each number, mark newly reachable sums by adding it to sums already reachable.
  5. Iterate the capacity from high to low so each item is used at most once.
  6. The answer is dp[target].
Why This Worksdp[s] is true exactly when some subset of the elements processed so far sums to s; it starts true only at 0 (the empty subset). Processing an element x makes s reachable if s - x was already reachable, which is precisely the include/exclude choice the recursion made — but each distinct sum is computed once. Sweeping the capacity downward guarantees x contributes to any sum at most once, matching 0/1 (not unbounded) selection. If dp[target] ends true, that subset and its complement form the two equal halves.
Watch Out For
  • Skipping the odd-total early exit, which wastes work and can mis-size the array.
  • Iterating the capacity ascending, which lets one element be reused and turns it into unbounded knapsack.
  • Assuming values fit a fixed bitset without sizing it to target (here up to 200 · 100 / 2 = 10000).

Hard Challenge

A monotonic deque maintains the window maximum in amortized O(1) per element — the capstone of the sliding-window family. Stretch.

#239 Sliding Window Maximum HardSliding WindowMonotonic DequeHeap

The ProblemYou are given an array nums and a sliding window of size k moving from the very left to the very right; you can only see the k numbers in the window, and it slides one position right each step. Return an array of the maximum of each window. Constraints: 1 ≤ nums.length ≤ 10^5, 1 ≤ k ≤ nums.length.
Brute Force — O(n · k) time, O(1) extra space

For every window start position, scan all k elements of that window and take the maximum. Simple and correct, but re-scans overlapping elements for every shift.

Python
def maxSlidingWindow(nums, k):

    for i in range(len(nums) - k + 1):
        res.append(max(nums[i:i + k]))
    return res

    vector<int> res;
    for (int i = 0; i + k <= (int)nums.size(); i++) {
        int m = nums[i];
        for (int j = i + 1; j < i + k; j++) m = max(m, nums[j]);
        res.push_back(m);
    }
    return res;
}

Why optimize? With n up to 10^5 and k up to n, the O(n · k) rescan can be ~10^10 operations — far too slow.

Key InsightBrute force re-finds the max from scratch even though windows overlap heavily. Maintain a monotonic decreasing deque of indices: before pushing a new index, pop all smaller values off the back (they can never be the max while the new one is present), and drop the front when it slides out of the window. The front index always holds the current window's max — each element is pushed and popped at most once, giving O(n).
How to Think About It
  1. Keep a deque holding indices, values in decreasing order.
  2. Before adding index i, pop indices off the back whose value is ≤ nums[i].
  3. Push i onto the back.
  4. If the front index is ≤ i - k, it left the window — pop it from the front.
  5. Once i ≥ k - 1, the front index's value is this window's max; record it.
  6. Slide to the next i and repeat.
Why This WorksWhen a new element enters, any earlier element that is smaller is now useless: it is both older and less than the newcomer, so it can never be the maximum of any window containing the newcomer — hence popping it from the back is safe. The deque therefore stays sorted by value in decreasing order, so its front is always the largest live candidate; evicting the front when its index falls out of range keeps that front valid for the current window. Because each index is added once and removed once, total work is linear.
Watch Out For
  • Storing values instead of indices — you need indices to detect when the front expires.
  • Recording a result before the first full window forms (i < k - 1).
  • Using < vs ≤ when popping the back inconsistently, or forgetting equal values still shift the front out by index.
