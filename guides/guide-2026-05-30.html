class Solution(object):
    def permute(self, nums):
        out = []
        def backtrack(nums, used):
            if len(used) == len(nums):
                o = []
                for i in used:
                    o.append(nums[i])
                out.append(o)
            for i in range(len(nums)):
                if i in used:
                    continue
                backtrack(nums, used + [i])
        backtrack(nums, [])
        return out

class Solution(object):
    def lengthOfLIS(self, nums):
        dp = [1] * len(nums)
        for r in range(1, len(nums)):
            for l in range(0, r):
                if nums[r] > nums[l]:

function findKthLargest(nums: number[], k: number): number {
    // Min-heap implementation using array
    const heap: number[] = [];

    function siftUp(i: number): void {
        while (i > 0) {
            const parent = (i - 1) >>> 1;
            if (heap[parent] > heap[i]) {
                [heap[parent], heap[i]] = [heap[i], heap[parent]];
                i = parent;
            } else break;
        }
    }

    function siftDown(i: number): void {
        const n = heap.length;
        while (true) {
            let smallest = i;
            const left = 2 * i + 1, right = 2 * i + 2;
            if (left < n && heap[left] < heap[smallest]) smallest = left;
            if (right < n && heap[right] < heap[smallest]) smallest = right;
            if (smallest === i) break;
            [heap[i], heap[smallest]] = [heap[smallest], heap[i]];
            i = smallest;
        }
    }

    for (const num of nums) {
        heap.push(num);
        siftUp(heap.length - 1);
        if (heap.length > k) {
            heap[0] = heap.pop()!;
            siftDown(0);
        }
    }
    return heap[0];
}

#20 Valid Parentheses Easy Stack String

The Problem: Given a string containing just the characters '(', ')', '{', '}', '[', and ']', determine if the input string is valid. An input string is valid if: (1) open brackets are closed by the same type of bracket, (2) open brackets are closed in the correct order, and (3) every close bracket has a corresponding open bracket of the same type.

Brute Force — O(n²) time

Repeatedly scan the string and remove adjacent matching pairs "()", "[]", "{}". Keep scanning until no more pairs can be removed. If the string is empty at the end, it was valid.

Python — Brute Force
def isValid(s: str) -> bool:
    prev = ""
    while s != prev:
        prev = s
        s = s.replace("()", "")
        s = s.replace("[]", "")
        s = s.replace("{}", "")
    return s == ""

Time: O(n²) · Space: O(n)

Each scan is O(n) and up to n/2 scans may be needed for deeply nested strings like "(((())))". With a stack, we can validate in a single pass.

Key Insight: Use a stack. Push every opening bracket. When you encounter a closing bracket, pop from the stack and check that it matches the expected opening bracket. If the stack is empty at the end, the string is valid.
How to Think About It
  1. Initialize an empty stack.
  2. Iterate through each character in the string.
  3. If the character is an opening bracket ((, [, {), push it onto the stack.
  4. If it is a closing bracket, first check if the stack is empty — if so, return false (nothing to match against).
  5. Pop the top element. If it does not match the expected opening bracket for this closing bracket, return false.
  6. After processing all characters, return true only if the stack is empty (all brackets were matched).
Why This Works: The stack enforces LIFO order, which exactly matches the nesting requirement of brackets — the most recently opened bracket must be the first one to close. Each bracket is pushed once and popped once, giving O(n) total work.
Watch Out For:
  • Empty stack pop: Trying to pop when the stack is empty (e.g., input starts with a closing bracket like "]"). Always check before popping.
  • Leftover elements: A string like "((" passes all character checks but the stack is not empty at the end — must return false.
  • Interleaved brackets: "([)]" is invalid even though each type has matching counts. The stack catches this because ] will try to match (.
Alternative Approach — Mapping: Use a hash map to store closing-to-opening bracket mappings. This avoids a chain of if/else checks and makes the code cleaner. The map approach is essentially the same algorithm, just more maintainable.

#155 Min Stack Medium Stack Design

The Problem: Design a stack that supports the following operations, all in constant time O(1): push(val) — pushes an element onto the stack, pop() — removes the top element, top() — returns the top element, and getMin() — retrieves the minimum element currently in the stack.

Brute Force — O(n) getMin

Use a regular stack for push, pop, and top. For getMin(), scan all elements in the stack to find the minimum. This gives O(1) for push/pop/top but O(n) for getMin.

Python — Brute Force
class MinStack:
    def __init__(self):
        self.stack = []

    def push(self, val: int) -> None:
        self.stack.append(val)

    def pop(self) -> None:
        self.stack.pop()

    def top(self) -> int:
        return self.stack[-1]

    def getMin(self) -> int:
        return min(self.stack)

Time: O(1) push/pop/top, O(n) getMin · Space: O(n)

If getMin() is called frequently (e.g., after every push/pop), the O(n) scan dominates. We need a way to track the minimum that updates in O(1) as elements are added and removed.

Key Insight: Store (value, current_min) tuples on the stack. Each entry remembers what the minimum was at the time it was pushed. When you pop, the previous entry automatically holds the correct minimum for the remaining stack.
How to Think About It
  1. Instead of storing just values, store pairs: each entry is (value, min_so_far).
  2. When pushing a new value, compute the new minimum as min(val, current_top_min). If the stack is empty, the minimum is the value itself.
  3. Push the pair (val, new_min).
  4. For top(), return the value component of the top pair.
  5. For getMin(), return the min component of the top pair.
  6. For pop(), just remove the top pair — the pair below it already has the correct min for the remaining stack.
Why This Works: Because each stack entry captures the minimum of all elements from the bottom up to and including that entry, popping always reveals the correct minimum for the remaining stack. The minimum is never lost because it is redundantly stored at every level. This trades O(n) extra space for O(1) getMin.
Watch Out For:
  • Global minimum trap: Do not track the minimum as a single variable — when you pop the element that was the min, you have no way to know the previous min.
  • Empty stack edge case: When pushing to an empty stack, the min is the value itself. Handle this before trying to access the top element's min.
  • Two-stack variant: An alternative uses a separate min-stack that only pushes when a new minimum arrives. This saves space but is trickier to get right — the tuple approach is safer for interviews.
Alternative — Two Stacks: Use one stack for values and a second stack for minimums. Only push onto the min-stack when the new value is less than or equal to the current min. Pop from the min-stack only when the popped value equals the current min. This uses less space on average but requires careful handling of duplicate minimums.

#739 Daily Temperatures Medium Stack Monotonic Stack

The Problem: Given an array of integers temperatures where temperatures[i] is the daily temperature on day i, return an array answer where answer[i] is the number of days you have to wait after day i to get a warmer temperature. If there is no future day with a warmer temperature, answer[i] = 0.

Brute Force — O(n²) time

For each day i, scan forward through all subsequent days to find the first day with a strictly higher temperature. Record the gap, or 0 if no warmer day is found.

Python — Brute Force
def dailyTemperatures(temperatures: list[int]) -> list[int]:
    n = len(temperatures)
    answer = [0] * n
    for i in range(n):
        for j in range(i + 1, n):
            if temperatures[j] > temperatures[i]:
                answer[i] = j - i
                break
    return answer

Time: O(n²) · Space: O(n) for the output array

For n = 100,000 days, O(n²) means up to 10 billion comparisons. A monotonic stack solves this in a single pass because each element is pushed and popped at most once.

Key Insight: Use a monotonic decreasing stack that stores indices. Iterate through the temperatures. When the current temperature is warmer than the temperature at the index on top of the stack, pop and record the gap. Each element is pushed and popped at most once, giving O(n) total.
How to Think About It
  1. Initialize an answer array of zeros and an empty stack.
  2. Iterate through each day i from left to right.
  3. While the stack is not empty AND temperatures[i] > temperatures[stack.top()]: pop the top index, call it prev. Set answer[prev] = i - prev.
  4. Push the current index i onto the stack.
  5. After the loop, any indices remaining on the stack have no warmer future day — their answer stays 0.
  6. The key invariant: the stack always holds indices in decreasing temperature order (monotonic decreasing). A warmer day resolves all pending cooler days in the stack.
Why This Works: The stack maintains a decreasing sequence of unresolved temperatures. When a warmer day arrives, it resolves all pending cooler days in one pass. Each element is processed exactly twice (one push + one pop), giving O(n) total work despite the nested while loop. Indices that are never popped correctly retain their default answer of 0.
Watch Out For:
  • Store indices, not temperatures: You need indices to compute the gap i - prev. Use the index to look up the temperature: temperatures[stack.top()].
  • Strictly greater: The problem asks for a warmer day, meaning strictly greater. Use >, not >=.
  • No warmer day: Indices left on the stack at the end have no warmer future day. Their answer is already 0 from initialization — do not forget to initialize the answer array to zeros.
  • Amortized O(n): The while loop inside the for loop looks like O(n²), but each index is pushed and popped at most once across the entire execution, so total work is O(n).
Alternative — Backward Iteration: Iterate from right to left. For each day, use the answer array itself to "jump" forward: if day j is not warmer, skip ahead by answer[j] positions. This also achieves O(n) amortized but is harder to reason about. The monotonic stack approach is more intuitive and more broadly applicable.

Time: O(n) · Space: O(1)

The problem explicitly requires O(log n). The array is "mostly sorted" — rotation preserves sorted order within each half, which is enough structure for binary search.

Key Insight: At any midpoint in a rotated sorted array, at least one half (left or right) is guaranteed to be sorted. Check if the target falls within the sorted half. If yes, search there. If no, search the other half. This single additional check adapts standard binary search to rotated arrays.
How to Think About It
  1. Set left = 0, right = len(nums) - 1.
  2. While left <= right, compute mid = left + (right - left) / 2.
  3. If nums[mid] == target, return mid.
  4. Determine which half is sorted: if nums[left] <= nums[mid], the left half is sorted.
  5. If left half is sorted: check if nums[left] <= target < nums[mid]. If yes, search left (right = mid - 1). Otherwise, search right (left = mid + 1).
  6. If right half is sorted: check if nums[mid] < target <= nums[right]. If yes, search right (left = mid + 1). Otherwise, search left (right = mid - 1).
  7. If the loop ends without finding the target, return -1.
Why This Works: Rotation preserves sorted order in at least one half. By identifying which half is sorted, we can reliably determine whether the target is in that half (a simple range check on sorted data) or must be in the other half. This maintains the binary search guarantee of eliminating half the candidates each step, giving O(log n).
Watch Out For:
  • Use <= in the sorted-half check: nums[left] <= nums[mid] must use <=, not <, to handle the case where left == mid (subarray of size 1 or 2).
  • Boundary inclusion: When checking if the target is in the sorted half, include the boundary: nums[left] <= target (could equal the left element).
  • Do not confuse with #153: Here you search for a target value, not the minimum. The binary search decision logic is different.
  • All elements are distinct: This simplifies the problem. With duplicates (#81), worst case degrades to O(n).
Alternative — Find Pivot First: First binary search for the rotation pivot (minimum element, like #153), then determine which sorted half the target is in, then binary search that half. This is two binary searches = O(2 log n) = O(log n). The single-pass approach above is more elegant but the two-pass approach can be easier to reason about.

Time: O(n) · Space: O(1)

The rotated sorted structure gives us enough information to eliminate half the search space at each step, just like standard binary search. We should exploit this for O(log n).

Key Insight: Compare nums[mid] with nums[right]. If nums[mid] > nums[right], the rotation point (minimum) must be in the right half. Otherwise, the minimum is in the left half (including mid). This is a binary search where the decision criterion is "which side of the rotation point am I on?"
How to Think About It
  1. Set left = 0, right = len(nums) - 1.
  2. While left < right (note: strict less-than, not <=).
  3. Compute mid = left + (right - left) / 2.
  4. If nums[mid] > nums[right]: the minimum is in the right half. Set left = mid + 1.
  5. Else: the minimum could be mid itself or to the left. Set right = mid (not mid - 1).
  6. When left == right, that index is the minimum. Return nums[left].
Why This Works: If nums[mid] > nums[right], the array "drops" somewhere between mid+1 and right — the minimum must be there. If nums[mid] <= nums[right], the subarray from mid to right is sorted, so the minimum is at or before mid. Each step halves the search space, converging on the single minimum element.
Watch Out For:
  • Compare with nums[right], NOT nums[left]: Comparing with nums[left] fails on non-rotated arrays like [1,2,3,4,5] where nums[left] <= nums[mid] always holds but the minimum is on the left.
  • Use right = mid, not right = mid - 1: When nums[mid] <= nums[right], mid itself could be the minimum. Setting right = mid - 1 could skip it.
  • Loop condition left < right: Use strict less-than. When left == right, we have converged on the answer. Using <= would cause an infinite loop.
  • Non-rotated array: An array rotated by its full length is back to sorted order. The algorithm handles this correctly — nums[mid] <= nums[right] always, so right moves left until convergence at index 0.
Alternative — Find the Drop: The minimum element is the only one where nums[i] < nums[i-1] (or nums[0] if the array is not rotated). You could binary search for this "drop point." This is conceptually the same algorithm expressed differently.

The matrix's sorted properties mean the entire matrix is effectively one long sorted array. We can binary search over it without actually flattening it, achieving O(log(m × n)).

Key Insight: Treat the 2D matrix as a virtual 1D sorted array. For a virtual index i, map it to row i // cols and column i % cols. Then apply standard binary search on indices 0 to m*n - 1. No extra space or actual flattening needed.
How to Think About It
  1. Get the dimensions: rows = len(matrix), cols = len(matrix[0]).
  2. Set left = 0, right = rows * cols - 1.
  3. While left <= right:
  4. Compute mid = left + (right - left) / 2.
  5. Convert mid to 2D coordinates: row = mid // cols, col = mid % cols.
  6. Compare matrix[row][col] with target: if equal, return true. If less, left = mid + 1. If greater, right = mid - 1.
  7. If the loop ends, return false.
Why This Works: Since rows are sorted and each row's first element exceeds the previous row's last element, the entire matrix is one sorted sequence when read row by row. The mapping i → (i // cols, i % cols) lets us treat it as a 1D array without materializing it, applying textbook binary search for O(log(m × n)).
Watch Out For:
  • Divide by cols, not rows: The mapping is row = mid // cols, col = mid % cols. A common mistake is dividing by the number of rows.
  • Use matrix[0].length for cols: Not matrix.length (that gives rows). Mixing these up gives wrong indices.
  • Empty matrix: Check if the matrix or its rows are empty before computing rows * cols.
  • Do not confuse with #240: Problem #240 (Search a 2D Matrix II) has a weaker guarantee — rows and columns are individually sorted, but the first element of a row is NOT necessarily greater than the last element of the previous row. That problem requires a different approach (staircase search).
Alternative — Two Binary Searches: First binary search to find which row the target could be in (compare target with first/last element of each row). Then binary search within that row. This is also O(log m + log n) = O(log(m × n)) but requires two separate searches and more edge-case handling. The virtual 1D approach is cleaner.

Time: O(n) · Space: O(h) where h = tree height (recursion stack)

This recursive solution is already optimal at O(n). Every node must be visited exactly once to swap its children, so no further optimization is possible.

Key Insight: At every node, swap its left and right children, then recurse on both subtrees. Base case: null node returns null. The swap-then-recurse pattern ensures every level of the tree gets mirrored.
How to Think About It
  1. Recognize this is a tree transformation — every node needs to be visited, so DFS or BFS both work.
  2. At the current node, swap left and right child pointers (not values — entire subtree references).
  3. Recurse on the (now-swapped) left and right children.
  4. Base case: if the node is null, return null — nothing to swap.
  5. Return the root so the caller gets back the same tree, now inverted.
Why This Works: If every node's children are swapped, then every path that originally went left now goes right and vice versa — producing the mirror image. Recursion ensures every level is processed. The order of traversal (pre-order, post-order, or level-order) does not matter because swapping at each node is independent of other swaps.
Watch Out For:
  • Don't try to swap values — swap the actual subtree references (pointers). Swapping values only works for leaf nodes and breaks the structure for internal nodes.
  • Remember the null base case. Without it, you get a null pointer dereference.
  • The tree is modified in-place. If you need the original tree preserved, clone it first.

When going right, update low to the current node's value. If any node's value falls outside its inherited range, the tree is not a valid BST.
How to Think About It
  1. Start with the root. Its valid range is (-infinity, +infinity) since it can be any value.
  2. When moving to a left child, the child must be less than the parent. Update the upper bound: range becomes (low, parent.val).
  3. When moving to a right child, the child must be greater than the parent. Update the lower bound: range becomes (parent.val, high).
  4. At each node, check: is low < node.val < high? If not, return false immediately.
  5. Recurse on both children. If both return true, this subtree is valid.
  6. The range bounds accumulate constraints from ALL ancestors, not just the immediate parent — this is what makes it correct.
Why This Works: The range bounds propagate the BST invariant downward through the tree. Each node inherits constraints from ALL its ancestors, not just its parent. For example, in the right subtree of the root, every node must be greater than the root — this constraint propagates even to deeply nested left children within that right subtree. If every node satisfies its inherited range, the global BST property is guaranteed.
Watch Out For:
  • A common mistake is only checking immediate children (left.val < node.val < right.val). This misses cases like [5,1,6,null,null,3,7] where 3 is in the right subtree of 5 but is less than 5. The full subtree bounds must be checked.
  • Use strict inequality (<, not <=). BSTs require strictly less/greater, not equal.
  • Initialize bounds with -infinity and +infinity (or use None/null with special handling). Using INT_MIN/INT_MAX fails if the tree contains those extreme values.
  • An alternative approach: in-order traversal should produce a strictly increasing sequence. Track the previous value and verify each node is greater than it.
Alternative — In-Order Traversal: Perform an in-order DFS traversal. If the tree is a valid BST, the values will appear in strictly increasing order. Keep a "previous value" variable (initialized to -infinity) and at each node, check that the current value is strictly greater than the previous. This also runs in O(n) time and O(h) space.

#236 Lowest Common Ancestor of a Binary Tree Medium Tree DFS Recursion

The Problem: Given a binary tree and two distinct nodes p and q that are guaranteed to exist in the tree, find their lowest common ancestor (LCA). The LCA is the deepest node that has both p and q as descendants. A node is allowed to be a descendant of itself — so if p is an ancestor of q, then p is the LCA.

Brute Force — O(n²) time

For each node in the tree, check if p is in its subtree AND q is in its subtree. The deepest node for which both checks return true is the LCA. Each "contains" check traverses up to O(n) nodes, and we check each of the n nodes, giving O(n²).

Time: O(n²) · Space: O(h) recursion stack

The brute force re-traverses subtrees for each node to check containment. The optimal approach finds the LCA in a single O(n) pass by using post-order traversal to propagate results upward.

Key Insight: Use post-order DFS. If the current node is p or q, return it. Recursively search left and right subtrees. If both subtrees return non-null results, the current node is the LCA (both targets are in different subtrees). If only one subtree returns non-null, propagate that result upward (both targets are in the same subtree).
How to Think About It
  1. Base case: if the current node is null, return null. If it IS p or q, return it immediately (no need to search deeper).
  2. Recurse on the left subtree. Recurse on the right subtree. Each returns either null (target not found) or a node (target found, or LCA found).
  3. If BOTH left and right returned non-null: p is in one subtree and q is in the other. The current node is where their paths diverge — it is the LCA. Return it.
  4. If only LEFT returned non-null: both p and q are in the left subtree (or one of them is the result). Propagate the left result upward.
  5. If only RIGHT returned non-null: same logic, propagate right result upward.
  6. If both returned null: neither p nor q is in this subtree. Return null.
Why This Works: Post-order traversal processes children before the parent, so information flows upward. When we find p or q, we signal upward. At the first node that receives signals from both subtrees, both paths converge — making it the deepest common ancestor by definition. The early-return when a node IS p or q also handles the case where one target is an ancestor of the other (e.g., p is the parent of q — p itself is returned before searching deeper).
Watch Out For:
  • Do not forget the case where one node is an ancestor of the other. For example, if p is the parent of q, p itself is the LCA. The "if current node is p or q, return it" base case handles this correctly.
  • Both p and q are guaranteed to exist in the tree. If they were not guaranteed, you would need additional logic to verify both were found.
  • Compare node references (identity), not values. Two different nodes could have the same value in a general binary tree (this is not a BST).
  • The problem uses a general binary tree, not a BST. Do not try to use BST properties to guide the search direction.

How to Think About It
  1. Define a TrieNode with two fields: a dictionary/map of children (char → TrieNode) and a boolean is_end flag.
  2. The Trie class holds a root TrieNode (which represents the empty prefix).
  3. Insert: Start at root. For each character in the word, check if a child exists for that character. If not, create one. Move to that child. After the last character, set is_end = true.
  4. Search: Start at root. For each character, check if a child exists. If any character is missing, return false. After the last character, return node.is_end (must be a complete word, not just a prefix).
  5. StartsWith: Exactly like search, but after the last character of the prefix, return true (don't check is_end — any word with this prefix counts).
  6. Factor out a shared helper: _find(prefix) that walks the trie and returns the final node (or null). Search calls it and checks is_end. StartsWith calls it and checks non-null.
Why This Works: The trie's tree structure naturally represents all prefixes. Shared prefixes share paths — "apple" and "app" share the first 3 nodes. This means lookup time depends only on the length of the query string (m), not on how many words are stored (n). The end-of-word flag distinguishes between a stored word ("app") and a prefix that was never inserted independently.

Every search scans all stored words. A trie skips irrelevant words by following character edges, and only fans out at '.' wildcards. For most practical patterns, the trie search is dramatically faster because non-wildcard characters prune most branches.

Key Insight: Build a trie (same as #208) for addWord. For search, walk the trie character by character. When you encounter a '.', recursively try ALL children at that position (DFS branching). For regular characters, follow the specific child edge. Return true if any branch reaches the end of the word with an end-of-word marker.
How to Think About It
  1. Use the exact same TrieNode and insert logic from #208 for addWord.
  2. For search, write a recursive helper that takes the current node and the remaining characters to match.
  3. At each character position: if the character is a letter, follow the matching child (or return false if it does not exist).
  4. If the character is '.', iterate over ALL children of the current node and recursively search each one. If ANY child leads to a match, return true.
  5. Base case: when all characters are consumed, check if the current node has is_end = true.
  6. If no children exist when processing '.', return false for that branch (dead end).
Why This Works: The trie provides the same prefix structure as #208 for exact-character matches. The wildcard '.' adds a branching search — when we see '.', we fan out to all possible children, trying every character at that position. This is DFS on the trie with backtracking. Non-wildcard characters prune the search to a single path, so performance is excellent when most characters are concrete.
Watch Out For:
  • When processing '.', you must try ALL children, not just the first match. Use a loop that returns true if any recursive call succeeds.
  • If a node has no children and you encounter '.', return false for that branch — there are no characters to match against.
  • Do not forget to check end-of-word at the final character. "b.." should only match 3-letter words, not prefixes of longer words.
  • Worst case with many dots (e.g., "...") can be expensive — it degenerates to checking all words of that length. This is expected and unavoidable.
  • The addWord method is identical to #208 insert — no wildcard handling is needed during insertion, only during search.
