#2 · Add Two Numbers

Medium Linked List 6 days since last review · rep 2 · interval 6

The Problem: Given two non-empty linked lists representing non-negative integers in reverse order (each node contains a single digit), add the two numbers and return the sum as a linked list in reverse order.

Quick Reminder: Walk both lists simultaneously with a carry variable. Create new nodes for each digit of the sum. Since digits are stored in reverse order, you can add from the least significant digit forward — exactly like manual addition. A carry variable handles overflow.
Challenge: Can you implement this in under 10 minutes? What is the time complexity? Previously solved in Python — try it in Go or TypeScript.

Example 1: l1=[2,4,3], l2=[5,6,4] → [7,0,8] (342+465=807)

Example 2: l1=[0], l2=[0] → [0]

Example 3: l1=[9,9,9,9], l2=[9,9,9] → [8,9,9,0,1] (9999+999=10998)

Time: O(max(m,n))  |  Space: O(max(m,n))
Previous Solution (Python) 
class ListNode(object):
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next
class Solution(object):
    def addTwoNumbers(self, l1, l2):
        head = ListNode(0)
        current = head
        carry = 0
        while l1 or l2 or carry != 0:
            x = l1.val if l1 else 0
            y = l2.val if l2 else 0
            sum = x + y + carry
            carry = sum // 10
            newNode = ListNode(sum % 10)
            current.next = newNode
            current = newNode
            l1 = l1.next if l1 else None
            l2 = l2.next if l2 else None
        return head.next

The Problem: Given an array of intervals where intervals[i] = [starti, endi], merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the intervals in the input.

Quick Reminder: Sort by start time, then merge consecutive overlapping intervals by extending the end time. After sorting by start, intervals that overlap must be adjacent. Compare each interval's start with the previous merged interval's end.
Challenge: Can you implement this in under 5 minutes? What edge cases exist? Previously solved in Python — try it in Rust or C++.

#198 · House Robber

Medium Dynamic Programming 6 days since last review · rep 2 · interval 6

The Problem: You are a robber planning to rob houses along a street. Each house has a certain amount of money. Adjacent houses have connected security systems — if two adjacent houses are broken into on the same night, the police are alerted. Return the maximum amount of money you can rob without alerting the police.

Quick Reminder: dp[i] = max(dp[i-1], nums[i] + dp[i-2]). At each house, the decision is binary: rob it (take its value + best from 2 houses back) or skip it (best from 1 house back). Two variables suffice for O(1) space.
Challenge: Can you write the O(1) space solution from memory? Previously solved in Python with memoization — try the iterative two-variable approach.

Example 1: [1,2,3,1] → 4 (rob house 1 and 3: 1+3)

Example 2: [2,7,9,3,1] → 12 (rob house 1, 3, 5: 2+9+1)

Edge: [5] → 5 (single house)

Time: O(n)  |  Space: O(1) (optimized)
Previous Solution (Python — memoization) 
class Solution:
    def rob(self, nums):
        memo = {}
        def dp(i):
            if i == 0: return nums[0]
            if i == 1: return max(nums[0], nums[1])
            if i not in memo:
                memo[i] = max(dp(i-1), nums[i] + dp(i-2))
            return memo[i]
        return dp(len(nums) -1)

The Problem: Given an m×n 2D binary grid where '1' represents land and '0' represents water, count the number of islands. An island is surrounded by water and is formed by connecting adjacent lands horizontally or vertically.

Quick Reminder: Iterate through the grid. When you find an unvisited '1', increment the island count and BFS/DFS to mark all connected land as visited. Each connected component of '1's is one island.

Example 2: grid=[["1","0","0"],["0","0","0"]] → 1

Edge: grid=[["0","0"],["0","0"]] → 0

Time: O(m×n)  |  Space: O(m×n)

    def numIslands(self, grid):
        rows, cols = len(grid), len(grid[0])
        visited = set()
        islands = 0
        def bfs(r, c):
            q = collections.deque()
            visited.add((r, c))
            q.append((r, c))

                row, col = q.popleft()
                d = [[0, 1], [1, 0], [-1, 0], [0, -1]]

            for c in range(cols):

        return islands

#295 · Find Median from Data Stream

Hard Heap Design 6 days since last review · rep 2 · interval 6

The Problem: Design a data structure that supports two operations: addNum(num) adds an integer from the data stream, and findMedian() returns the median of all elements so far. If the count is even, return the average of the two middle values.

Quick Reminder: Use two heaps — a max-heap for the lower half and a min-heap for the upper half. Keep them balanced within 1 element. The max of the lower half and min of the upper half give the median. Heaps give O(log n) insertion.
Challenge: Can you implement both methods correctly with proper balancing? Previously solved in Python — try it in Go or C++.

Example: addNum(1), addNum(2), findMedian() → 1.5, addNum(3), findMedian() → 2.0

Edge: addNum(5), findMedian() → 5.0 (single element)

addNum: O(log n)  |  findMedian: O(1)
Previous Solution (Python) 
class MedianFinder(object):
    def __init__(self):
        self.small, self.large = [], []
    def addNum(self, num):
        heapq.heappush(self.small, -num)
        if self.small and self.large and -self.small[0] > self.large[0]:
            a = - heapq.heappop(self.small)
            heapq.heappush(self.large, a)
        if len(self.small) > len(self.large) + 1:
            a = - heapq.heappop(self.small)
            heapq.heappush(self.large, a)

            a = - heapq.heappop(self.large)
            heapq.heappush(self.small, a)

            return -self.small[0]
        if len(self.small) < len(self.large):
            return self.large[0]
        return (-self.small[0] + self.large[0]) / float(2)

import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []  # max-heap (negate values)
        self.hi = []  # min-heap

    def addNum(self, num: int) -> None:
        heapq.heappush(self.lo, -num)
        # ensure max of lo <= min of hi
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        # balance sizes: lo can have at most 1 more than hi
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self) -> float:
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2.0

class MedianFinder {
    private lo: number[] = []; // max-heap via negation
    private hi: number[] = []; // min-heap

    private pushHeap(heap: number[], val: number): void {
        heap.push(val);
        let i = heap.length - 1;
        while (i > 0) {
            const parent = (i - 1) >> 1;
            if (heap[parent] <= heap[i]) break;
            [heap[parent], heap[i]] = [heap[i], heap[parent]];
            i = parent;
        }
    }

    private popHeap(heap: number[]): number {
        const top = heap[0];
        const last = heap.pop()!;
        if (heap.length > 0) {
            heap[0] = last;
            let i = 0;
            while (true) {
                let smallest = i;
                const l = 2 * i + 1, r = 2 * i + 2;
                if (l < heap.length && heap[l] < heap[smallest]) smallest = l;
                if (r < heap.length && heap[r] < heap[smallest]) smallest = r;
                if (smallest === i) break;
                [heap[i], heap[smallest]] = [heap[smallest], heap[i]];
                i = smallest;
            }
        }
        return top;
    }

    addNum(num: number): void {
        this.pushHeap(this.lo, -num);
        this.pushHeap(this.hi, -this.popHeap(this.lo));
        if (this.hi.length > this.lo.length) {
            this.pushHeap(this.lo, -this.popHeap(this.hi));
        }
    }

    findMedian(): number {
        if (this.lo.length > this.hi.length) {
            return -this.lo[0];
        }
        return (-this.lo[0] + this.hi[0]) / 2;
    }

Key Insight: Use two pointers (prev and curr). At each step, save the next node, point curr.next to prev, then advance both pointers. Each node's pointer is flipped exactly once, and when done, prev points to the former tail (new head).

How to Think About It

  1. Initialize prev = None (or null) and curr = head.
  2. While curr is not null, save next_node = curr.next (so you do not lose the rest of the list).
  3. Point curr.next = prev to reverse the link.
  4. Advance: prev = curr, then curr = next_node.
  5. When the loop ends, curr is null and prev is the new head. Return prev.
Why This Works: Each node's pointer is redirected from pointing forward to pointing backward. By saving the next pointer before overwriting, no nodes are lost. After processing all nodes, prev points to what was the last node — which is now the first.
Watch Out For: Forgetting to save curr.next before overwriting it — you lose the rest of the list. Returning curr instead of prev — curr is null when the loop ends, prev is the new head.

Key Insight: Use two pointers separated by n nodes. When the fast pointer reaches the end, the slow pointer is right before the node to remove. A dummy node before head handles the edge case of removing the head.

How to Think About It

  1. Create a dummy node with dummy.next = head. This handles the edge case where we need to remove the head itself.
  2. Set both fast and slow to dummy.

  5. Skip the target: slow.next = slow.next.next. Return dummy.next.

Key Insight: Three steps: (1) find the middle with slow/fast pointers, (2) reverse the second half, (3) merge the two halves by alternating nodes. This is a capstone problem combining #876 (find middle), #206 (reverse), and list merging.

How to Think About It

  1. Find the middle: Use slow/fast pointers. When fast reaches the end, slow is at the middle. For even-length lists, slow points to the end of the first half.
  2. Split the list: Save second = slow.next, then set slow.next = None to terminate the first half.
  3. Reverse the second half: Apply the #206 reversal technique to second. Now you have two independent lists.
  4. Merge alternately: Take one node from the first half, then one from the reversed second half, alternating until both are exhausted.
  5. The merge loop: save next pointers for both lists, rewire first.next = second and second.next = first_next, then advance both pointers.
Why This Works: Reversing the second half lets us interleave from "both ends" without extra space. The first half provides L0, L1, L2... and the reversed second half provides Ln, Ln-1, Ln-2... By merging them alternately, we achieve the desired L0 → Ln → L1 → Ln-1 ordering.
Watch Out For: Not properly terminating the first half (set slow.next = None) — without this, you create a cycle. Forgetting to save .next during the merge step causes you to lose nodes. Off-by-one with even vs. odd length lists — the first half should be equal or one longer than the second.

        let mut len = 0;
        let mut p = head.as_ref();
        while let Some(node) = p {
            len += 1;
            p = node.next.as_ref();
        }
        if len <= 2 {
            return;
        }

        // Split at middle
        let mid = (len - 1) / 2;
        let mut curr = head.as_mut().unwrap();
        for _ in 0..mid {
            curr = curr.next.as_mut().unwrap();
        }
        let mut second = curr.next.take();

        // Reverse second half
        let mut prev: Option<Box<ListNode>> = None;
        while let Some(mut node) = second {
            second = node.next.take();
            node.next = prev;
            prev = Some(node);
        }
        let mut second = prev;

        // Merge alternately
        let mut curr = head;
        while second.is_some() {
            let mut s_node = second.unwrap();
            second = s_node.next.take();
            let first_next = curr.as_mut().unwrap().next.take();
            s_node.next = first_next;
            curr.as_mut().unwrap().next = Some(s_node);
            curr = &mut curr.as_mut().unwrap().next.as_mut().unwrap().next;
        }
    }
}

class MinHeap {
    private data: ListNode[] = [];
    size() { return this.data.length; }
    push(node: ListNode) {
        this.data.push(node);
        this.bubbleUp(this.data.length - 1);
    }
    pop(): ListNode {
        const top = this.data[0];
        const last = this.data.pop()!;
        if (this.data.length > 0) {
            this.data[0] = last;
            this.sinkDown(0);
        }
        return top;
    }
    private bubbleUp(i: number) {
        while (i > 0) {
            const parent = (i - 1) >> 1;
            if (this.data[parent].val <= this.data[i].val) break;
            [this.data[parent], this.data[i]] = [this.data[i], this.data[parent]];
            i = parent;
        }
    }
    private sinkDown(i: number) {
        const n = this.data.length;
        while (true) {
            let smallest = i;
            const l = 2 * i + 1, r = 2 * i + 2;
            if (l < n && this.data[l].val < this.data[smallest].val) smallest = l;
            if (r < n && this.data[r].val < this.data[smallest].val) smallest = r;
            if (smallest === i) break;
            [this.data[smallest], this.data[i]] = [this.data[i], this.data[smallest]];
            i = smallest;
        }
    }
}

How to Think About It

  2. Push (distance, point) into a max-heap.

    // Max-heap by distance
    const heap: number[][] = [];

    function dist(p: number[]): number {
        return p[0] * p[0] + p[1] * p[1];
    }
    function swap(i: number, j: number) {
        [heap[i], heap[j]] = [heap[j], heap[i]];

            const parent = (i - 1) >> 1;

Key Insight: Use two pointers from both ends. The shorter side determines the water level. Move the shorter pointer inward, tracking a running max from each side. When height[left] < height[right], we know maxRight ≥ height[right] > height[left], so water at the left position depends only on leftMax.

How to Think About It

  1. Initialize left = 0, right = n - 1, leftMax = 0, rightMax = 0, water = 0.
  2. While left < right: if height[left] < height[right], process the left side.
  3. Update leftMax = max(leftMax, height[left]). Add leftMax - height[left] to water. Increment left.
  4. Otherwise, process the right side symmetrically: update rightMax, add rightMax - height[right], decrement right.
  5. Continue until the pointers meet.
Why This Works: Water at any position is determined by min(maxLeft, maxRight) - height[i]. So the bottleneck is maxLeft, which we already track. The same logic applies symmetrically to the right pointer.
Watch Out For: Moving the wrong pointer — always move the side with the smaller height. Off-by-one when leftMax/rightMax should include the current position. Forgetting that water can never be negative (leftMax ≥ height[left] is guaranteed by the max update).

Alternative: Precompute leftMax[] and rightMax[] arrays in two passes, then compute water in a third pass. Same O(n) time but O(n) space instead of O(1).

Key Insight: Use a sliding window with two pointers. Expand the right pointer to include characters until the window satisfies the requirement (contains all of t). Then contract the left pointer to minimize the window. Track a “have” counter (number of characters fully satisfied) vs a “need” count (number of distinct characters required) to avoid rescanning the frequency map.

How to Think About It

  1. Build a frequency map of t. Set need = len(t_count) (number of distinct characters needed).
  2. Expand the right pointer. For each character, update the window frequency map. If a character’s count matches its required count, increment have.
  3. When have == need, the window is valid. Try to contract from the left to shrink it.
  4. For each contraction, if removing the left character drops its count below the requirement, decrement have.
  5. Track the minimum valid window seen throughout the process.
Why This Works: The right pointer only moves forward, and for each position of the right pointer, we contract the left as far as possible. This means we visit each character at most twice (once by right, once by left), giving O(n) time. The have/need counters let us check window validity in O(1) instead of scanning the entire frequency map.
Watch Out For: Character frequency tracking requires exact counts, not just presence — if t = “AAB”, the window needs 2 A’s and 1 B. Updating the “have” counter correctly: increment only when window[c] == need[c], decrement only when window[c] < need[c] after removal. Return empty string when no valid window exists.

How to Think About It

  1. Initialize a frequency array of size 26, left = 0, maxFreq = 0, best = 0.
  2. Expand right: increment freq[s[right]], update maxFreq = max(maxFreq, freq[s[right]]).
  3. Check validity: if (right - left + 1) - maxFreq > k, the window is invalid.
  4. Shrink from left: decrement freq[s[left]], increment left. Do NOT decrease maxFreq.
  5. Update best = max(best, right - left + 1).
Why This Works: The window only grows when we find a character with a higher frequency (maxFreq increases). When maxFreq does not increase, the window cannot grow beyond its current best, so we slide it forward. This means maxFreq acts as a high-water mark — we only care about windows that could beat the current best, and those require a higher maxFreq.
Watch Out For: Do not decrease maxFreq when shrinking the window — it is unnecessary and complicates the logic. The window size may not always shrink when invalid (it slides forward instead). Connection to #76: similar sliding window structure but with a simpler validity condition.

#323 · Number of Connected Components in an Undirected Graph

Medium Union-Find Graph BFS/DFS

The Problem: You have n nodes labeled from 0 to n - 1 and a list of undirected edges. Return the number of connected components in the graph.

Brute Force Approach

Build an adjacency list and run BFS/DFS from each unvisited node. Each new traversal starting point represents a new connected component. This is O(V + E) and works well, but Union-Find provides a different mental model useful for dynamic connectivity.

Python Brute Force (DFS)

class Solution:
    def countComponents(self, n: int, edges: List[List[int]]) -> int:
        adj = [[] for _ in range(n)]
        for u, v in edges:
            adj[u].append(v)
            adj[v].append(u)
        visited = set()
        components = 0
        for node in range(n):
            if node not in visited:
                components += 1
                stack = [node]
                while stack:
                    curr = stack.pop()
                    if curr in visited:
                        continue
                    visited.add(curr)
                    for neighbor in adj[curr]:
                        if neighbor not in visited:
                            stack.append(neighbor)
        return components

Key Insight: Union-Find tracks connected components by maintaining a “parent” array. Initially each node is its own parent (n components). For each edge (u, v), union them — if they share the same root, they are already connected; otherwise, merge their sets and decrement the component count. Path compression and union by rank keep operations nearly O(1).

How to Think About It

  1. Initialize parent[i] = i and rank[i] = 0 for all nodes. Set components = n.
  2. Implement find(x): follow parent pointers to root, with path compression (set each visited node’s parent directly to root).
  3. Implement union(x, y): find roots of x and y. If different, merge by rank (attach shorter tree under taller). Decrement components.
  4. For each edge (u, v), call union(u, v).
  5. Return components.
Why This Works: Each union operation merges two components into one. After processing all edges, the remaining count is the number of connected components. Path compression ensures that find operations are nearly O(1) amortized (inverse Ackermann function α(n)), making the total time O(E · α(n)) ≈ O(E).
Watch Out For: Path compression is critical for efficiency — without it, find() degrades to O(n) in the worst case (a chain). Union by rank prevents the tree from becoming unbalanced. Always find the root of each node before comparing (do not compare parents directly).

Example 1: n = 5, edges = [[0,1],[1,2],[3,4]] → 2 (components: {0,1,2} and {3,4})

Example 2: n = 5, edges = [[0,1],[1,2],[2,3],[3,4]] → 1 (all connected)

Edge: n = 3, edges = [] → 3 (no edges, each node is its own component)

BFS/DFS: O(V + E) time, O(V + E) space → Union-Find: O(E · α(n)) time, O(n) space

#684 · Redundant Connection

Medium Union-Find Graph

The Problem: A tree is a connected graph with no cycles and n - 1 edges for n nodes. You are given a graph that started as a tree with nodes 1 to n, but one additional edge was added. Return the edge that, if removed, would result in a tree. If there are multiple answers, return the one that occurs last in the input.

Brute Force Approach

For each edge, temporarily remove it and check if the remaining graph is still connected (using BFS/DFS). The last removable edge that keeps the graph connected is the answer. O(E²) time.

Python Brute Force

class Solution:
    def findRedundantConnection(self, edges: List[List[int]]) -> List[int]:
        n = len(edges)
        # Try removing each edge from last to first
        for i in range(n - 1, -1, -1):
            adj = [[] for _ in range(n + 1)]
            for j, (u, v) in enumerate(edges):
                if j == i:
                    continue
                adj[u].append(v)
                adj[v].append(u)
            # Check connectivity from node 1
            visited = set()
            stack = [1]
            while stack:
                curr = stack.pop()
                if curr in visited:
                    continue
                visited.add(curr)
                for nb in adj[curr]:
                    if nb not in visited:
                        stack.append(nb)
            if len(visited) == n:
                return edges[i]
        return []

Key Insight: Process edges one by one with Union-Find. For each edge (u, v), check if u and v already share the same root. If they do, this edge creates a cycle — it is the redundant edge. If not, union them. The first edge that connects two already-connected nodes is the answer.

How to Think About It

  1. Initialize Union-Find with parent[i] = i for nodes 1 to n.
  2. For each edge (u, v) in input order:
  3. Call find(u) and find(v). If they return the same root, return [u, v] — this edge is redundant.
  4. Otherwise, call union(u, v) to merge their components.
  5. The problem guarantees exactly one redundant edge, so the loop always finds it.
Why This Works: A tree with n nodes has exactly n - 1 edges. The input has n edges, so exactly one edge creates a cycle. When we process edges sequentially, the edge that first connects two nodes already in the same component must be the one creating the cycle. Since the problem asks for the last such edge in the input, and there is exactly one cycle, the first detected redundancy (by Union-Find) is the answer.
Watch Out For: Nodes are 1-indexed (not 0-indexed) — size your parent array accordingly. The problem guarantees a unique answer. This problem builds directly on the #323 Union-Find template — the only change is returning the edge instead of counting components.

#763 · Partition Labels

Medium Greedy Hash Map Two Pointers

The Problem: You are given a string s. Partition the string into as many parts as possible so that each letter appears in at most one part. Return a list of integers representing the size of each part.

Brute Force Approach

Try all possible partition points. For each combination, verify that no character spans two partitions. This is exponential in the number of possible split points.

Python Brute Force

class Solution:
    def partitionLabels(self, s: str) -> List[int]:
        # Try all partition points using backtracking
        n = len(s)
        best = []

        def valid(parts: List[str]) -> bool:
            seen = set()
            for part in parts:
                chars = set(part)
                if chars & seen:
                    return False
                seen |= chars
            return True

        def backtrack(idx: int, current: List[str]):
            nonlocal best
            if idx == n:
                if valid() and len() > len(best):
                    best = [len(p) for p in current]
                return
            # Extend last partition
            if current:
                current[-1] += s[idx]
                backtrack(idx + 1, current)
                current[-1] = current[-1][:-1]
            # Start new partition
            current.append(s[idx])
            backtrack(idx + 1, current)
            current.pop()

        backtrack(0, [])
        return best

func partitionLabels(s string) []int {

    // For each starting position, extend until all chars are contained
    n := len(s)

    i := 0

                    end = k
                    break
                }
            }
        }

        i = end + 1
    }
    return result

Key Insight: Precompute the last occurrence of each character. Then scan left to right: for each character, extend the current partition’s end to at least the last occurrence of that character. When you reach the partition end, you know no character in the current partition appears later — cut here.

How to Think About It

  1. Build a map of the last index where each character appears in s.
  2. Initialize start = 0, end = 0.
  3. For each position i, update end = max(end, last[s[i]]).
  4. When i == end, a partition boundary is found. Record end - start + 1. Set start = i + 1.
  5. Continue until the string is exhausted.
Why This Works: A partition must include every occurrence of every character it contains. By extending end to the last occurrence of each character we encounter, we guarantee the partition is self-contained. The moment we reach end, we know we have included the last occurrence of every character in the partition — it is safe to cut.
Watch Out For: Forgetting to update end for each new character encountered while scanning. The greedy approach works because characters can only extend a partition forward, never backward. Only 26 lowercase letters, so the map is O(1) space.

Example 1: "ababcbacadefegdehijhklij" → [9, 7, 8]

Example 2: "eccbbbbdec" → [10] (all chars interleave)

Edge: "a" → [1]

Brute: O(2n) or O(n²) time → Optimal: O(n) time, O(1) space

#435 · Non-overlapping Intervals

Medium Greedy Sort Intervals

The Problem: Given an array of intervals where intervals[i] = [starti, endi], return the minimum number of intervals you need to remove to make the rest of the intervals non-overlapping.

Brute Force Approach

Try all subsets of intervals, find the largest subset where no two intervals overlap, and return n - largest_subset_size. This is O(2n) which is completely impractical for large inputs.

Python Brute Force

class Solution:
    def eraseOverlapIntervals(self, intervals: List[List[int]]) -> int:
        intervals.sort()
        n = len(intervals)
        # DP: longest non-overlapping subsequence
        dp = [1] * n  # dp[i] = max non-overlapping ending at i
        for i in range(1, n):
            for j in range(i):
                if intervals[j][1] <= intervals[i][0]:
                    dp[i] = max(dp[i], dp[j] + 1)
        return n - max(dp)

Key Insight: Sort intervals by END time, then greedily keep intervals that do not overlap with the last kept interval. This is the classic activity selection problem: choosing the earliest-ending compatible activity at each step maximizes the total number of non-overlapping intervals, minimizing removals.

How to Think About It

  1. Sort intervals by their end time (ascending).
  2. Initialize last_end = -infinity (or the end of the first interval) and removals = 0.
  3. For each interval: if start ≥ last_end, keep it and update last_end = end.
  4. Otherwise, this interval overlaps — remove it (increment removals). Do NOT update last_end.
  5. Return removals.
Why This Works: Sorting by end time and greedily selecting the earliest-ending non-overlapping interval is optimal because it leaves the maximum possible room for subsequent intervals. Any other selection strategy would end later, potentially blocking more future intervals. This is a well-known result from the activity selection problem (provable by exchange argument).
Watch Out For: Sorting by START time instead of END time — this does not work (consider [[1,100],[2,3],[3,4]]). Using strict > vs ≥ for the overlap check — intervals touching at a point ([1,2] and [2,3]) are NOT overlapping. Connection to #56 Merge Intervals: both require sorted intervals, but the goal is different (merge vs remove).

Example 1: [[1,2],[2,3],[3,4],[1,3]] → 1 (remove [1,3])

Example 2: [[1,2],[1,2],[1,2]] → 2 (keep one, remove two)

Example 3: [[1,2],[2,3]] → 0 (touching intervals are not overlapping)

Brute: O(n²) DP / O(2n) subset → Optimal: O(n log n) time, O(1) space
