Review — 6 Concepts
R1. WeakRef & FinalizationRegistry Meta-Programming rep 0 · overdue 1 day · last reviewed 2026-06-08
WeakRef holds a reference that does not prevent garbage collection. .deref() returns the object or undefined if collected. FinalizationRegistry registers a callback to run after an object is GC'd — but timing is non-deterministic.
Key Pattern 
class WeakCache {
  #cache = new Map();
  set(key, value) { this.#cache.set(key, new WeakRef(value)); }
  get(key) {
    const ref = this.#cache.get(key);
    const val = ref?.deref();
    if (ref && val === undefined) this.#cache.delete(key);
    return val;
  }
}
Challenge

Build a WeakCache with auto-eviction: use FinalizationRegistry to automatically remove Map entries when the cached object is garbage-collected. Log each cleanup event.

Pitfall: Relying on FinalizationRegistry for critical cleanup — it is non-deterministic. Always provide explicit close()/dispose() methods for resources that must be released.
R2. Closures Deep Dive Core Language rep 1 · last reviewed 2026-06-09
Functions capture variables from their defining scope by reference, not by value. A closure is a function bundled with its lexical environment. If the outer variable changes, the closure sees the new value.
Key Pattern 
function once(fn) {
  let called = false, result;
  return (...args) => {
    if (!called) { called = true; result = fn(...args); }
    return result;
  };
}
const init = once(() => Math.random());
console.log(init() === init()); // true
Challenge

Implement once(fn) that only calls fn the first time. Subsequent calls should return the cached result without invoking fn again.

Pitfall: var in for loops — all closures share the same variable because var is function-scoped. Use let (block-scoped) or an IIFE to capture per-iteration values.
R3. Prototypes & Inheritance Core Language rep 1 · last reviewed 2026-06-09
Every object has a [[Prototype]] chain. Property lookup walks up the chain until the property is found or null is reached. Object.create(proto) sets up inheritance. ES6 class is syntactic sugar over prototypes.
Key Pattern 
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return `${this.name} speaks`; };

function Dog(name) { Animal.call(this, name); }
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;

const rex = new Dog("Rex");
console.log(rex.speak()); // "Rex speaks"
console.log(rex instanceof Dog);    // true
console.log(rex instanceof Animal); // true
Challenge

Implement classical inheritance without the class keyword: create Animal and Dog constructor functions where Dog inherits from Animal. Ensure instanceof works for both.

Pitfall: __proto__ is legacy — use Object.getPrototypeOf() to inspect and Object.create(proto) to set up inheritance chains.
R4. this Binding Core Language rep 1 · last reviewed 2026-06-09
this is determined by the call site: default (global/undefined), implicit (dot), explicit (call/apply/bind), new, or arrow (lexical). Priority: new > explicit > implicit > default.
Key Pattern 
const obj = { name: "Widget", onClick() { console.log(this.name); } };

// Lost: const fn = obj.onClick; fn(); // undefined
// Fix 1: setTimeout(() => obj.onClick(), 0);
// Fix 2: setTimeout(obj.onClick.bind(obj), 0);
// Fix 3: class field arrow: onClick = () => { ... }
Challenge

Predict what this refers to in 5 different contexts: (1) standalone function call, (2) obj.method(), (3) fn.call(otherObj), (4) new Constructor(), (5) arrow function inside a method. Write code demonstrating each.

Pitfall: Extracting a method from an object as a callback loses this. const g = obj.greet; g(); — this is no longer obj. Fix with .bind() or an arrow wrapper.
R5. Symbols & Well-Known Symbols Meta-Programming rep 1 · last reviewed 2026-06-09
Symbols are unique, collision-free property keys. Well-known symbols like Symbol.iterator, Symbol.toPrimitive, and Symbol.hasInstance customize built-in language behavior. Symbols are invisible to Object.keys, for...in, and JSON.stringify.
Key Pattern 
class Pair {
  constructor(a, b) { this.a = a; this.b = b; }
  *[Symbol.iterator]() {
    yield [this.a, this.b];
  }
}
for (const [k, v] of new Pair("x", 1)) console.log(k, v); // "x" 1
Challenge

Make an object iterable that yields key-value pairs: [...new Entries({a:1, b:2})] should return [["a",1],["b",2]]. Use Symbol.iterator.

Pitfall: "text" + symbol throws TypeError. Use symbol.toString() or symbol.description for string representation.
R6. Property Descriptors & Object.defineProperty Meta-Programming rep 1 · last reviewed 2026-06-09
Property descriptors control writable, enumerable, and configurable per property. Accessor descriptors use get/set. Object.freeze makes all own properties non-writable and non-configurable — but it is shallow.
Key Pattern 
"use strict";
const obj = {};
Object.defineProperty(obj, "id", {
  value: 42,
  writable: false,
  enumerable: false,
  configurable: false
});
console.log(obj.id);           // 42
console.log(Object.keys(obj)); // [] (non-enumerable)
// obj.id = 99;                // TypeError (non-writable)
Challenge

Create a sealed object with one read-only property: use Object.defineProperty to make version non-writable, then Object.seal the object. Verify that version cannot be changed and no new properties can be added.

Pitfall: Object.freeze is shallow. Object.freeze({nested: {x: 1}}).nested.x = 99 silently succeeds. Use recursive deepFreeze for nested immutability.

#1. Event Loop Deep Dive Async Mastery
The event loop is the runtime model that makes JavaScript non-blocking. Understanding the call stack, microtask queue, and macrotask queue is essential for predicting execution order and avoiding subtle async bugs.
  • Call stack: synchronous code executes here, one frame at a time (single-threaded)
  • Microtask queue: Promise.resolve().then(), queueMicrotask(), MutationObserver — drained completely after each macrotask
  • Macrotask queue: setTimeout, setInterval, setImmediate (Node.js), I/O callbacks, UI rendering
  • Execution order: call stack empties → drain ALL microtasks → pick ONE macrotask → repeat
  • setTimeout(fn, 0) does NOT mean "immediately" — it schedules a macrotask (minimum ~1ms browser, ~1ms Node)
  • Promise.resolve().then(fn) always runs before setTimeout(fn, 0) because microtasks drain first
  • Node.js phases: timers → pending callbacks → idle/prepare → poll → check (setImmediate) → close callbacks
  • process.nextTick() (Node.js) runs before microtasks — highest priority after current operation
JavaScript — Microtasks vs Macrotasks Ordering
console.log("1: sync start");

setTimeout(() => console.log("2: setTimeout"), 0);

Promise.resolve().then(() => console.log("3: promise.then"));

queueMicrotask(() => console.log("4: queueMicrotask"));

console.log("5: sync end");
1: sync start 5: sync end 3: promise.then 4: queueMicrotask 2: setTimeout

Common Mistakes
Assuming setTimeout(fn, 0) runs immediately — it schedules a macrotask, which waits for all microtasks to drain first
Use queueMicrotask(fn) or Promise.resolve().then(fn) for "as soon as possible" after current synchronous code
Starving the macrotask queue by recursively scheduling microtasks — setTimeout callbacks never run
Be careful with recursive queueMicrotask or .then() chains; they block rendering and I/O
Confusing process.nextTick() (Node.js only, pre-microtask priority) with queueMicrotask() (standard, microtask queue)
Prefer queueMicrotask() for portability; process.nextTick is Node-specific and higher priority
Key Rules
  • Microtasks always drain completely before the next macrotask
  • Promise.then() and queueMicrotask() are microtasks; setTimeout is a macrotask
  • Long synchronous code blocks the event loop — breaks timers, I/O, and UI
  • In Node.js, process.nextTick() fires before Promise microtasks

#2. Promise Internals Async Mastery
Promises are the async primitive in JavaScript. Understanding their states, microtask scheduling, and combinators is critical for writing correct async code and debugging unexpected execution order.
  • Three states: pending → fulfilled (with value) or rejected (with reason). Transitions are one-way and permanent.
  • .then(onFulfilled, onRejected) returns a new Promise — enables chaining. The returned value becomes the next Promise's resolved value.
  • Thenable objects: any object with a .then() method is treated as Promise-like by the runtime
  • Promise.all(iterable) — resolves when ALL settle successfully; rejects on first rejection
  • Promise.allSettled(iterable) — resolves when ALL settle (never rejects), returns [{status, value/reason}]
  • Promise.race(iterable) — resolves/rejects with the first to settle
  • Promise.any(iterable) — resolves with the first to fulfill; rejects with AggregateError if ALL reject
  • .then() callbacks are always scheduled as microtasks, never synchronous
JavaScript — Promise States & Chaining
const p = new Promise((resolve, reject) => {
  console.log("executor runs synchronously");
  resolve(42);
  // resolve/reject after first call is ignored
  resolve(99);  // no effect
  reject("err"); // no effect
});

p.then(val => {
  console.log("fulfilled:", val); // 42
  return val * 2;                 // creates a new fulfilled promise
})
.then(val => {
  console.log("chained:", val);   // 84
  throw new Error("oops");       // creates a rejected promise
})
.catch(err => {
  console.log("caught:", err.message); // "oops"
  return "recovered";
})
.then(val => console.log("after catch:", val)); // "recovered"
executor runs synchronously fulfilled: 42 chained: 84 caught: oops after catch: recovered
JavaScript — Promise.all vs Promise.allSettled
const fast = Promise.resolve("fast");
const slow = new Promise(r => setTimeout(() => r("slow"), 100));
const fail = Promise.reject(new Error("fail"));

// Promise.all: rejects on first failure
Promise.all([fast, slow, fail])
  .then(vals => console.log("all:", vals))
  .catch(err => console.log("all rejected:", err.message));

// Promise.allSettled: always resolves with all results
Promise.allSettled([fast, slow, fail])
  .then(results => {
    results.forEach(r => {
      if (r.status === "fulfilled") console.log("ok:", r.value);
      else console.log("err:", r.reason.message);
    });
  });
all rejected: fail ok: fast ok: slow err: fail
JavaScript — Promise.race vs Promise.any
const slow = new Promise(r => setTimeout(() => r("slow"), 200));
const fast = new Promise(r => setTimeout(() => r("fast"), 50));
const fail = new Promise((_, rej) => setTimeout(() => rej(new Error("err")), 10));

// race: first to settle wins (even if it rejects)
Promise.race([slow, fast, fail])
  .then(v => console.log("race fulfilled:", v))
  .catch(e => console.log("race rejected:", e.message));

// any: first to FULFILL wins (ignores rejections unless ALL reject)
Promise.any([slow, fast, fail])
  .then(v => console.log("any fulfilled:", v))
  .catch(e => console.log("any rejected:", e.errors));
race rejected: err any fulfilled: fast

JavaScript — Microtask Scheduling Proof
// .then() callbacks are always asynchronous (microtask), even if
// the promise is already resolved
const resolved = Promise.resolve("done");

console.log("before then");
resolved.then(val => console.log("then:", val));
console.log("after then");

// It runs as a microtask after all synchronous code completes

Common Mistakes
Forgetting to return a promise in a .then() chain — the next .then() receives undefined
Always return from .then() callbacks to pass values down the chain
Using Promise.all when you need all results regardless of failure
Use Promise.allSettled when any individual failure should not abort the batch
Creating a promise in .then() without returning it: .then(() => { fetchData(); }) — the chain does not wait
Return the promise: .then(() => { return fetchData(); }) or .then(() => fetchData())
Key Rules
  • Promise state transitions are one-way and permanent (pending → fulfilled or rejected)
  • .then() always returns a new Promise; always return values in chains
  • Promise.all = fail-fast; Promise.allSettled = wait for all; Promise.race = first to settle; Promise.any = first to fulfill
  • .then() callbacks are microtasks — never synchronous, even for already-resolved promises

#3. async/await Patterns Async Mastery
async/await makes asynchronous code read like synchronous code, dramatically reducing callback nesting. Knowing when to run sequentially vs in parallel is the difference between a fast and a slow application.
  • async function always returns a Promise; the return value is wrapped in Promise.resolve()
  • await pauses the function until the Promise settles; the function resumes as a microtask
  • Error handling: use try/catch around await — rejected promises throw into the catch block
  • Sequential vs parallel: await a; await b; is sequential; await Promise.all([a, b]) is parallel
  • Top-level await: available in ES modules (.mjs or "type": "module" in package.json)
  • Async IIFE: (async () => { ... })(); for top-level await in non-module scripts
  • Returning vs awaiting: return await p vs return p — only matters for try/catch (the await catches rejections)
JavaScript — Sequential vs Parallel Execution
function delay(ms, value) {
  return new Promise(r => setTimeout(() => r(value), ms));
}

async function sequential() {
  const start = Date.now();
  const a = await delay(100, "a"); // 100ms
  const b = await delay(100, "b"); // +100ms = 200ms total
  console.log(`sequential: ${a}, ${b} in ${Date.now() - start}ms`);
}

async function parallel() {
  const start = Date.now();
  const [a, b] = await Promise.all([
    delay(100, "a"),
    delay(100, "b")
  ]); // Both start at the same time = 100ms total
  console.log(`parallel: ${a}, ${b} in ${Date.now() - start}ms`);
}

sequential().then(() => parallel());
sequential: a, b in 200ms parallel: a, b in 100ms

async function fetchUser(id) {
  if (id < 0) throw new Error("Invalid ID");

JavaScript — return vs return await in try/catch
async function fails() {
  throw new Error("boom");
}

// BUG: return without await - error is NOT caught here
async function withoutAwait() {
  try {
    return fails(); // the promise is returned, not awaited
  } catch (err) {
    console.log("caught in withoutAwait"); // never reached!
  }
}

// CORRECT: return await - error IS caught
async function withAwait() {
  try {
    return await fails(); // awaited, so throw is caught
  } catch (err) {
    console.log("caught in withAwait:", err.message);
    return "recovered";
  }
}

withAwait().then(val => console.log("result:", val));

JavaScript — Async Map with Concurrency Control
async function asyncMap(items, fn, concurrency = Infinity) {
  const results = new Array(items.length);
  let i = 0;

  async function worker() {
    while (i < items.length) {
      const index = i++;
      results[index] = await fn(items[index], index);
    }
  }

  const workers = Array.from(
    { length: Math.min(concurrency, items.length) },
    () => worker()
  );
  await Promise.all(workers);
  return results;
}

// Process 6 items with max 2 concurrent
const items = [1, 2, 3, 4, 5, 6];
asyncMap(items, async (n) => {
  await new Promise(r => setTimeout(r, 50));
  return n * 10;
}, 2).then(r => console.log(r));

Common Mistakes
Running await in a loop when tasks are independent: for (const url of urls) { await fetch(url); } — runs sequentially

#4. AbortController & AbortSignal Async Mastery
AbortController is the standard cancellation mechanism for async operations in JavaScript. It prevents wasted work, avoids stale responses in UIs, and enables timeout patterns that are cleaner than Promise.race with timers.
  • new AbortController() — .signal is the AbortSignal, .abort(reason) cancels
  • Pass { signal } to fetch() to make it cancellable — throws AbortError on abort
  • AbortSignal.timeout(ms) — creates a signal that auto-aborts after ms milliseconds
  • AbortSignal.any([signal1, signal2]) — aborts when any signal fires (like Promise.race for signals)
  • signal.addEventListener("abort", handler) — react to cancellation in custom async code
  • signal.aborted — boolean, check if already cancelled
  • signal.reason — the reason passed to .abort(reason)
JavaScript — Cancelling fetch with AbortController
async function fetchWithCancel(url, timeoutMs = 3000) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort("timeout"), timeoutMs);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);
    return await response.json();
  } catch (err) {
    clearTimeout(timeoutId);
    if (err.name === "AbortError") {
      console.log("Request cancelled:", controller.signal.reason);
      return null;
    }
    throw err; // re-throw non-abort errors
  }
}

// Usage:
// fetchWithCancel("https://api.example.com/data", 2000);
JavaScript — AbortSignal.timeout() (Built-In Timeout)
// AbortSignal.timeout creates a signal that auto-aborts
async function fetchWithTimeout(url, ms) {
  try {
    const response = await fetch(url, {
      signal: AbortSignal.timeout(ms)
    });
    return await response.json();
  } catch (err) {
    if (err.name === "TimeoutError") {
      console.log(`Request timed out after ${ms}ms`);
      return null;
    }
    if (err.name === "AbortError") {
      console.log("Request was aborted");
      return null;
    }
    throw err;
  }
}

// No manual setTimeout/clearTimeout needed!
JavaScript — Cancelling Custom Async Operations
function cancellableDelay(ms, signal) {
  return new Promise((resolve, reject) => {
    // Check if already aborted
    if (signal?.aborted) {
      return reject(new DOMException("Aborted", "AbortError"));
    }

    const timeoutId = setTimeout(() => {
      resolve(`completed after ${ms}ms`);
    }, ms);

    // Listen for abort
    signal?.addEventListener("abort", () => {
      clearTimeout(timeoutId);
      reject(new DOMException("Aborted", "AbortError"));
    }, { once: true });
  });
}

// Test: cancel after 50ms

  .then(v => console.log(v))
  .catch(e => console.log("Cancelled:", e.name));

JavaScript — Abort-Aware Async Generator
async function* pollEndpoint(url, intervalMs, signal) {
  while (!signal.aborted) {
    try {
      const response = await fetch(url, { signal });
      yield await response.json();
    } catch (err) {
      if (err.name === "AbortError") return; // clean exit
      yield { error: err.message };
    }
    await new Promise(r => setTimeout(r, intervalMs));
  }
}

// const controller = new AbortController();

Common Mistakes
Forgetting to pass { signal } to fetch() — the request is not cancellable
Always pass signal when you need cancellation: fetch(url, { signal: controller.signal })
Not distinguishing AbortError from real errors in catch blocks
Check err.name === "AbortError" to handle cancellation differently from failures
Reusing an aborted controller — once aborted, the signal stays aborted forever
Create a new AbortController for each cancellable operation
Key Rules
  • One AbortController per cancellable operation — do not reuse after abort
  • Always check err.name === "AbortError" in catch blocks
  • Use AbortSignal.timeout(ms) for timeouts — cleaner than manual setTimeout
  • Use AbortSignal.any() to combine user cancellation with timeout

#5. Streams API Async Mastery
Streams allow processing data incrementally — start working with the first bytes before the entire payload arrives. This is essential for large files, real-time feeds, and keeping memory usage constant regardless of data size.
  • ReadableStream — a source of data you can read from chunk by chunk
  • WritableStream — a sink you can write chunks to
  • TransformStream — both readable and writable; transforms data as it passes through
  • readableStream.pipeThrough(transformStream).pipeTo(writableStream) — composable pipeline
  • fetch responses have response.body as a ReadableStream
  • TextDecoderStream / TextEncoderStream — built-in transforms for encoding
  • Backpressure: if the consumer is slow, the stream automatically slows down the producer
  • Readable streams are async iterables — use for await...of
JavaScript — Creating a ReadableStream
const stream = new ReadableStream({
  start(controller) {
    controller.enqueue("Hello ");
    controller.enqueue("from ");
    controller.enqueue("stream!");
    controller.close();
  }
});

// Read using the reader protocol
const reader = stream.getReader();
let result = "";
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  result += value;
}
console.log(result);
Hello from stream!
JavaScript — Streaming a Fetch Response
// Process a large response as it arrives, chunk by chunk
async function streamResponse(url) {
  const response = await fetch(url);
  const reader = response.body
    .pipeThrough(new TextDecoderStream())
    .getReader();

  let totalChars = 0;
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    totalChars += value.length;
    console.log(`Chunk: ${value.length} chars (total: ${totalChars})`);
  }
  console.log(`Stream complete: ${totalChars} chars`);
}

// Usage: streamResponse("https://example.com/large-file.txt");

JavaScript — ReadableStream as Async Iterable
async function* generateNumbers(n) {
  for (let i = 1; i <= n; i++) {
    await new Promise(r => setTimeout(r, 10));
    yield i;
  }
}

// Convert async generator to ReadableStream
function asyncGenToStream(asyncGen) {
  return new ReadableStream({
    async pull(controller) {
      const { value, done } = await asyncGen.next();
      if (done) { controller.close(); return; }
      controller.enqueue(value);
    }
  });
}

const stream = asyncGenToStream(generateNumbers(5));

// Streams are async iterable in modern runtimes
const reader = stream.getReader();
const results = [];
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  results.push(value);
}
console.log(results);

Common Mistakes
Calling response.json() when you need streaming — it buffers the entire response into memory
Use response.body.getReader() for chunk-by-chunk processing of large responses
Forgetting to call controller.close() in a custom ReadableStream — the consumer hangs forever
Always controller.close() when there is no more data, and controller.error(err) on failure
Trying to read from a stream after it has been locked by another reader or piped
A stream can only have one reader at a time. Use .tee() to split a stream for multiple consumers.
Key Rules
  • ReadableStream, TransformStream, WritableStream — compose with pipeThrough/pipeTo
  • fetch().body is already a ReadableStream — use it for incremental processing
  • Always close() or error() the controller when done
  • Streams handle backpressure automatically — slow consumers slow down producers

Node.js streams are similar but use a different API; Web Streams are the standard. Service Workers (concept #7) use streams for response construction.
#6. Web Workers Async Mastery
JavaScript is single-threaded. Web Workers provide true parallelism by running code on a separate OS thread. Use them for CPU-intensive work (parsing, compression, image processing) to keep the main thread responsive.
  • new Worker("worker.js") or new Worker(new URL("worker.js", import.meta.url)) — creates a dedicated worker
  • Communication: postMessage(data) sends; onmessage receives (structured clone algorithm)
  • Transferable objects: ArrayBuffer, MessagePort, OffscreenCanvas — zero-copy transfer (ownership moves, no cloning)
  • SharedArrayBuffer + Atomics — shared memory between threads (requires COOP/COEP headers)
  • Dedicated vs Shared Workers: dedicated workers have one parent; shared workers can be accessed by multiple tabs/frames
  • Workers have no access to DOM, window, or document — but they can use fetch, IndexedDB, WebSocket
  • worker.terminate() kills the worker immediately from the main thread
JavaScript — Basic Worker (Main Thread)
// main.js
const worker = new Worker("worker.js");

worker.postMessage({ type: "compute", data: [1, 2, 3, 4, 5] });

worker.onmessage = (event) => {
  console.log("Result from worker:", event.data);
};

worker.onerror = (err) => {
  console.error("Worker error:", err.message);
};
JavaScript — Basic Worker (Worker Thread)
// worker.js
self.onmessage = (event) => {
  const { type, data } = event.data;

  if (type === "compute") {
    // Heavy computation that won't block the main thread
    const sum = data.reduce((a, b) => a + b, 0);
    self.postMessage({ type: "result", sum });
  }
};

// Workers can also use:
// fetch(), IndexedDB, WebSocket, setTimeout, etc.
// But NOT: document, window, DOM APIs
JavaScript — Transferable Objects (Zero-Copy)
// main.js
const buffer = new ArrayBuffer(1024 * 1024); // 1MB
console.log("Before transfer:", buffer.byteLength); // 1048576

// Transfer ownership to worker (zero-copy, instant)
worker.postMessage({ buffer }, [buffer]); // second arg = transfer list

console.log("After transfer:", buffer.byteLength); // 0 (detached!)

// The worker now owns the buffer
// This is MUCH faster than cloning for large data

// Create a worker from a string (useful for demos/tests)
function createInlineWorker(fn) {

JavaScript — Promisified Worker Communication
function workerRequest(worker, data) {
  return new Promise((resolve, reject) => {
    const id = Math.random().toString(36).slice(2);
    const handler = (event) => {
      if (event.data.id === id) {
        worker.removeEventListener("message", handler);
        if (event.data.error) reject(new Error(event.data.error));
        else resolve(event.data.result);
      }
    };
    worker.addEventListener("message", handler);
    worker.postMessage({ id, ...data });
  });
}

// Now you can use async/await with workers:
// const result = await workerRequest(worker, { type: "fib", n: 40 });
Common Mistakes
Sending large objects via postMessage without transferring — triggers expensive structured clone
Transfer ArrayBuffers using the transfer list: postMessage(data, [buffer])
Accessing DOM APIs from a worker — document, window, localStorage are not available
Workers can use fetch, IndexedDB, WebSocket, and caches — not DOM
Forgetting to worker.terminate() when done — the thread stays alive consuming resources
Always terminate workers when no longer needed; or reuse them with a pool pattern
Key Rules
  • Workers run on a separate OS thread — true parallelism, not just async
  • Communication via postMessage uses structured clone (deep copy by default)
  • Use Transferable objects for large data to avoid clone overhead
  • No DOM access in workers — use them for computation, parsing, and I/O

SharedArrayBuffer + Atomics enables lock-free data structures between threads. Transferable objects relate to ArrayBuffer and typed arrays. Service Workers (next concept) are a special type of worker for network interception.
#7. Service Workers Async Mastery
Service Workers act as a programmable network proxy between your app and the server. They enable offline-first apps, background sync, and push notifications — the foundation of Progressive Web Apps (PWAs).
  • Lifecycle: install → activate → fetch (idle between events, terminated when inactive)
  • Registration: navigator.serviceWorker.register("/sw.js") — scope defaults to the directory of the script
  • install event: pre-cache critical assets using caches.open() and cache.addAll()
  • activate event: clean up old caches; call self.clients.claim() to take control immediately
  • fetch event: intercept every network request; return cached or network response
  • Caching strategies:
    • 2026-07-09 (Day 28)
    • Cache-first: serve from cache, fall back to network
    • Network-first: try network, fall back to cache
    • Stale-while-revalidate: serve from cache immediately, update cache from network in background
  • self.skipWaiting() — skip the waiting phase, activate immediately (use with caution)
  • Service workers require HTTPS (except localhost for development)
JavaScript — Registering a Service Worker
// main.js (page script)
if ("serviceWorker" in navigator) {
  navigator.serviceWorker.register("/sw.js", { scope: "/" })
    .then(reg => {
      console.log("SW registered, scope:", reg.scope);

      reg.addEventListener("updatefound", () => {
        const newWorker = reg.installing;
        newWorker.addEventListener("statechange", () => {
          console.log("SW state:", newWorker.state);
          // states: installing -> installed -> activating -> activated
        });
      });
    })
    .catch(err => console.error("SW registration failed:", err));
}
JavaScript — Install Event (Pre-Caching)
// sw.js
const CACHE_NAME = "app-v1";
const PRECACHE_URLS = [
  "/",
  "/index.html",
  "/styles.css",
  "/app.js",
  "/offline.html"
];

self.addEventListener("install", (event) => {
  console.log("[SW] Install");
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => {
        console.log("[SW] Pre-caching", PRECACHE_URLS.length, "assets");
        return cache.addAll(PRECACHE_URLS);
      })
      .then(() => self.skipWaiting()) // activate immediately
  );
});
JavaScript — Activate Event (Cache Cleanup)
// sw.js
self.addEventListener("activate", (event) => {
  console.log("[SW] Activate");
  event.waitUntil(
    caches.keys().then(cacheNames => {
      return Promise.all(
        cacheNames
          .filter(name => name !== CACHE_NAME) // delete old versions
          .map(name => {
            console.log("[SW] Deleting old cache:", name);
            return caches.delete(name);
          })
      );
    })
    .then(() => self.clients.claim()) // take control of all pages
  );
});
JavaScript — Cache-First Strategy
// sw.js - serve from cache, fall back to network
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      if (cached) {
        console.log("[SW] Cache hit:", event.request.url);
        return cached;
      }
      console.log("[SW] Cache miss, fetching:", event.request.url);
      return fetch(event.request).then(response => {
        // Cache the new response for future requests
        const clone = response.clone();
        caches.open(CACHE_NAME).then(cache => {
          cache.put(event.request, clone);
        });
        return response;
      });
    })
    .catch(() => caches.match("/offline.html")) // fallback
  );
});
JavaScript — Stale-While-Revalidate Strategy
// sw.js - serve stale cache immediately, update in background
self.addEventListener("fetch", (event) => {
  if (event.request.url.includes("/api/")) {
    event.respondWith(
      caches.open(CACHE_NAME).then(async (cache) => {
        const cached = await cache.match(event.request);

        // Fetch fresh copy in background
        const fetchPromise = fetch(event.request).then(response => {
          cache.put(event.request, response.clone());
          return response;
        });

        // Return cached immediately, or wait for network
        return cached || fetchPromise;
      })
    );
  }
});
Common Mistakes
Forgetting response.clone() before caching — a Response body can only be consumed once
Always response.clone(): one copy for the cache, one for the browser
Using self.skipWaiting() without considering that it activates the new SW while old pages are still running
Use skipWaiting carefully; combine with clients.claim() and version your caches
Not versioning the cache name — stale assets persist after deployments
Use a versioned cache name ("app-v2") and clean old caches in the activate event
Key Rules
  • Service Workers require HTTPS (except localhost)
  • Lifecycle: install (pre-cache) → activate (clean up) → fetch (intercept requests)
  • Always clone() responses before caching — bodies can only be consumed once
  • Choose your caching strategy deliberately: cache-first for assets, network-first for API data, stale-while-revalidate for feeds
