R1. Intl API Functions & Iteration rep 0 · overdue 2 days · last reviewed 2026-06-09
Intl.NumberFormat, Intl.DateTimeFormat, and Intl.Collator provide locale-aware formatting for numbers, dates, and string comparison. Create a formatter once, call .format() many times — formatters are reusable and cached by the engine.
Key Pattern 
const usd = new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' });
console.log(usd.format(1234.5)); // "$1,234.50"

const date = new Intl.DateTimeFormat('en-GB', {
  dateStyle: 'long', timeStyle: 'short'
});
console.log(date.format(new Date('2026-06-12'))); // "12 June 2026 at 00:00"

const collator = new Intl.Collator('de', { sensitivity: 'base' });
console.log(collator.compare('a', 'A')); // 0 (equal under base sensitivity)
Challenge

Create an Intl.NumberFormat that formats numbers as compact notation (e.g., 1.5K, 2.3M). Format [999, 1500, 23400000] and log the results.

Pitfall: Constructing a new Intl.NumberFormat inside a loop is expensive. Create the formatter once outside the loop and reuse it — the engine caches locale data but not the formatter instance itself.
R2. Optional Chaining & Nullish Coalescing Functions & Iteration rep 0 · overdue 2 days · last reviewed 2026-06-09
?. short-circuits to undefined if any link in the chain is null or undefined. ?? returns the right operand only for null/undefined, not for falsy values like 0, "", or false. This distinction from || is critical.
Key Pattern 
const user = { profile: { address: { city: "NYC" } } };

// Safe deep access
console.log(user?.profile?.address?.city);   // "NYC"
console.log(user?.settings?.theme);           // undefined (no throw)

// Optional method call
const arr = [1, 2, 3];
console.log(arr?.at?.(-1));  // 3
console.log(null?.at?.(-1)); // undefined

// ?? vs || — the critical difference
const count = 0;
console.log(count || 10);  // 10 (WRONG — 0 is falsy)
console.log(count ?? 10);  // 0  (CORRECT — 0 is not null/undefined)
Challenge

Write a function safeGet(obj, path, fallback) that uses optional chaining logic to traverse a dot-separated path string (e.g., "a.b.c"). If any segment is null/undefined, return fallback ?? undefined.

Pitfall: You cannot write to an optional chain: obj?.a?.b = 5 is a SyntaxError. Optional chaining is read-only. Also, ?? cannot be mixed with &&/|| without explicit parentheses.
R3. Event Loop Deep Dive Async Mastery rep 0 · overdue 1 day · last reviewed 2026-06-10
The event loop processes the call stack first, then drains all microtasks (Promise callbacks, queueMicrotask, MutationObserver), then picks one macrotask (setTimeout, setInterval, I/O). Microtasks can starve macrotasks if they keep enqueuing more microtasks.
Key Pattern 
console.log("1: sync");

setTimeout(() => console.log("2: macrotask"), 0);

Promise.resolve().then(() => console.log("3: microtask"));

queueMicrotask(() => console.log("4: microtask"));

console.log("5: sync");

// Order: 1: sync, 5: sync, 3: microtask, 4: microtask, 2: macrotask
Challenge

Predict the exact output order of: setTimeout(fn, 0), Promise.resolve().then(fn), queueMicrotask(fn), requestAnimationFrame(fn) (browser), and a synchronous console.log. Write the code and verify.

Pitfall: A microtask that enqueues another microtask creates an infinite loop that blocks the event loop entirely. function spin() { queueMicrotask(spin); } will freeze the page — setTimeout callbacks will never fire.
R4. Promise Internals Async Mastery rep 0 · overdue 1 day · last reviewed 2026-06-10
Promises have 3 states: pending, fulfilled, rejected. Once settled, the state is immutable. .then() always returns a new Promise. Promise.allSettled() waits for all to settle (never short-circuits). Promise.any() resolves with the first fulfilled, throws AggregateError if all reject.
Key Pattern 
const results = await Promise.allSettled([
  Promise.resolve("ok"),
  Promise.reject(new Error("fail")),
  Promise.resolve(42)
]);

const fulfilled = results
  .filter(r => r.status === "fulfilled")
  .map(r => r.value);

console.log(fulfilled); // ["ok", 42]
Challenge

Implement promiseAllSettled(promises) from scratch without using Promise.allSettled. It should return a promise that resolves with an array of {status, value} or {status, reason} objects.

Pitfall: Promise.all() rejects immediately when any promise rejects, but the other promises keep running — they are not cancelled. Use AbortController to actually cancel in-flight work when one fails.
R5. AbortController & AbortSignal Async Mastery rep 0 · overdue 1 day · last reviewed 2026-06-10
AbortController creates a signal that can cancel fetch requests, event listeners, and custom async operations. AbortSignal.timeout(ms) creates a signal that auto-aborts after a timeout. AbortSignal.any([...signals]) combines multiple signals.
Key Pattern 
// Timeout a fetch
const res = await fetch("/api/data", {
  signal: AbortSignal.timeout(5000) // auto-abort after 5s
});

// Manual abort with cleanup
const controller = new AbortController();
const { signal } = controller;

signal.addEventListener("abort", () => {
  console.log("Aborted:", signal.reason);
});

// Pass signal to fetch, addEventListener, etc.
setTimeout(() => controller.abort("User cancelled"), 3000);
Challenge

Write a fetchWithRetry(url, retries, timeoutMs) that retries on failure up to retries times, using AbortSignal.timeout(timeoutMs) for each attempt. Distinguish between timeout aborts and network errors.

Pitfall: Calling controller.abort() after the fetch already completed is harmless but wasteful. Check signal.aborted before starting expensive work. Also, AbortSignal.timeout() throws a TimeoutError (DOMException), not a generic AbortError.
R6. WeakRef & FinalizationRegistry Meta-Programming rep 1 · overdue 1 day · last reviewed 2026-06-10
WeakRef holds a weak reference to an object — the GC can reclaim it at any time. .deref() returns the object or undefined if collected. FinalizationRegistry schedules a callback when a registered object is garbage-collected. Use for caches, not for correctness logic.
Key Pattern 
class WeakCache {
  #cache = new Map();
  #registry = new FinalizationRegistry(key => {
    this.#cache.delete(key); // cleanup entry when value is GC'd
  });

  set(key, value) {
    this.#cache.set(key, new WeakRef(value));
    this.#registry.register(value, key); // track for cleanup
  }

  get(key) {
    const ref = this.#cache.get(key);
    return ref?.deref(); // undefined if GC'd
  }
}
Challenge

Build a WeakCache with a getOrCreate(key, factory) method: return the cached value if still alive, otherwise call factory(), cache the result, and return it.

Pitfall: Never rely on FinalizationRegistry for critical cleanup (closing files, releasing locks). GC timing is non-deterministic and callbacks may never run if the program exits. Use explicit dispose()/[Symbol.dispose]() patterns instead.
1 Modules
#1. ES Modules Deep Dive Modules & Patterns
ES Modules are the standard module system for JavaScript. Understanding static vs dynamic imports, tree-shaking, circular dependencies, and import.meta is essential for writing efficient, maintainable code that bundles well and loads fast.
  • 2026-07-09 (current — Day 28)
  • Static imports (import/export) are analyzed at parse time — hoisted, evaluated once, and cached. This enables tree-shaking by bundlers.
  • Named vs default exports: Named exports are bound by name; default export is syntactic sugar for a special default name. Prefer named exports for discoverability.
  • Dynamic import(): Returns a promise resolving to the module namespace. Enables lazy loading, conditional loading, and code splitting.
  • import.meta: Contains module metadata — import.meta.url (file URL of the module), import.meta.resolve() (resolve a specifier).
  • Top-level await: Modules can use await at the top level. The importing module waits for the awaited value before executing.
  • Circular dependencies: ES modules handle cycles by returning the partially-evaluated namespace — exports accessed before they are initialized will be undefined, not an error.
  • Tree-shaking: Works because static imports are statically analyzable. Barrel files (index.js re-exporting everything) and side effects break tree-shaking.
JavaScript — Named vs Default Exports
// math.js — named exports
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }

// main.js — named imports (tree-shakeable)
import { add, PI } from "./math.js"; // multiply is NOT included in bundle
console.log(add(2, 3), PI); // 5 3.14159

// default export — one per module
// logger.js
export default class Logger {
  log(msg) { console.log(`[LOG] ${msg}`); }
}

// main.js — import default (any name works)
import Logger from "./logger.js";
new Logger().log("hello"); // [LOG] hello

// Re-export (barrel file) — use sparingly, hurts tree-shaking
// index.js
export { add, multiply } from "./math.js";
export { default as Logger } from "./logger.js";

JavaScript — Circular Dependencies
// a.js
import { b } from "./b.js";
export const a = "A";
console.log("a.js:", b); // "B" — b.js evaluated first

// b.js
import { a } from "./a.js";
export const b = "B";
console.log("b.js:", a); // undefined! — a.js not fully evaluated yet

// Execution order when importing a.js:
// 1. Engine discovers a.js imports b.js
// 2. b.js is evaluated first (dependency)
// 3. b.js tries to read `a` from a.js — but a.js hasn't run yet → undefined
// 4. b.js finishes, exports b = "B"
// 5. a.js runs, reads b = "B" successfully

// FIX: Use functions to defer access
// b-fixed.js

JavaScript — Tree-Shaking: What Helps and What Breaks It
// GOOD: Named exports — bundler can drop unused ones
export function used() { return 42; }
export function unused() { return 99; } // dropped if not imported

// BAD: Side effects prevent tree-shaking
export const config = (() => {
  console.log("I run on import!"); // side effect — bundler can't remove
  return { debug: true };
})();

// BAD: Barrel files re-export everything
// components/index.js
export { Button } from "./Button.js";
export { Modal } from "./Modal.js";    // imported even if unused
export { Table } from "./Table.js";    // by some bundlers

// FIX: Import directly from the source
import { Button } from "./components/Button.js"; // bypasses barrel

// package.json — mark side-effect-free for better tree-shaking
// { "sideEffects": false }
// or: { "sideEffects": ["*.css", "./src/polyfills.js"] }
Common Mistakes
Using require() and import in the same file — they are different module systems (CJS vs ESM) with different semantics
Choose one: use "type": "module" in package.json for ESM, or .mjs/.cjs extensions to be explicit
Expecting a barrel index.js to tree-shake perfectly — many bundlers pull in the entire barrel
Import directly from source files for optimal tree-shaking: import { fn } from "./utils/fn.js"
Accessing a circular dependency's export at module evaluation time and getting undefined
Wrap the access in a function so it runs after both modules are fully initialized
Key Rules
  • Prefer named exports over defaults — better for tree-shaking, auto-imports, and refactoring
  • Use import() for lazy loading, code splitting, and optional dependencies
  • Avoid barrel files in performance-critical paths
  • Mark your package "sideEffects": false to enable full tree-shaking
  • Top-level await blocks the entire import chain — use judiciously

2 Design & Functional Patterns
#2. Design Patterns: Observer, Strategy, Command Modules & Patterns
Design patterns solve recurring problems with proven structures. In JavaScript, first-class functions and closures make many patterns lighter than in classical OOP. Knowing when to reach for Observer, Strategy, or Command prevents over-engineering and under-engineering alike.
  • Observer (Pub/Sub): Decouples producers from consumers. An object maintains a list of subscribers and notifies them on state changes. The foundation of event systems.
  • Strategy: Encapsulates interchangeable algorithms behind a common interface. In JS, functions are strategies — pass different functions to change behavior without changing the caller.
  • Command: Wraps an action as an object with execute() and optionally undo(). Enables queuing, logging, and undo/redo.
  • Chain of Responsibility: Passes a request through a chain of handlers. Each handler decides to process or pass along. Express middleware is this pattern.
  • Classes vs closures: Use closures when the pattern is simple (Strategy). Use classes when you need state, inheritance, or instanceof checks (Command with undo).
JavaScript — Observer: Building a Pub/Sub from Scratch
class EventEmitter {
  #listeners = new Map();

  on(event, fn) {
    if (!this.#listeners.has(event)) {
      this.#listeners.set(event, new Set());
    }
    this.#listeners.get(event).add(fn);
    return () => this.off(event, fn); // return unsubscribe function
  }

  off(event, fn) {
    this.#listeners.get(event)?.delete(fn);
  }

  emit(event, ...args) {
    for (const fn of this.#listeners.get(event) ?? []) {
      fn(...args);
    }
  }

  once(event, fn) {
    const wrapper = (...args) => {
      this.off(event, wrapper);
      fn(...args);
    };
    return this.on(event, wrapper);
  }
}

const bus = new EventEmitter();
const unsub = bus.on("user:login", user => console.log(`Welcome, ${user}!`));
bus.emit("user:login", "Alice"); // "Welcome, Alice!"
unsub(); // unsubscribe
bus.emit("user:login", "Bob");   // (nothing — unsubscribed)
Welcome, Alice!
JavaScript — Strategy: Swappable Algorithms
// Strategies are just functions
const strategies = {
  bubble: (arr) => {
    const a = [...arr];
    for (let i = 0; i < a.length; i++)
      for (let j = 0; j < a.length - i - 1; j++)
        if (a[j] > a[j+1]) [a[j], a[j+1]] = [a[j+1], a[j]];
    return a;
  },
  native: (arr) => [...arr].sort((a, b) => a - b),
  insertion: (arr) => {
    const a = [...arr];
    for (let i = 1; i < a.length; i++) {
      let j = i;
      while (j > 0 && a[j-1] > a[j]) {
        [a[j-1], a[j]] = [a[j], a[j-1]];
        j--;
      }
    }
    return a;
  }
};

// The context doesn't know which algorithm it uses
function sortData(data, strategyName = "native") {
  const strategy = strategies[strategyName];
  if (!strategy) throw new Error(`Unknown strategy: ${strategyName}`);
  return strategy(data);
}

console.log(sortData([3, 1, 4, 1, 5], "bubble"));    // [1, 1, 3, 4, 5]
console.log(sortData([3, 1, 4, 1, 5], "insertion"));  // [1, 1, 3, 4, 5]
console.log(sortData([3, 1, 4, 1, 5], "native"));     // [1, 1, 3, 4, 5]
[ 1, 1, 3, 4, 5 ] [ 1, 1, 3, 4, 5 ] [ 1, 1, 3, 4, 5 ]
JavaScript — Command: Undo/Redo System
class Command {
  execute() { throw new Error("Implement execute()"); }
  undo() { throw new Error("Implement undo()"); }
}

class AddTextCommand extends Command {
  constructor(doc, text) {
    super();
    this.doc = doc;
    this.text = text;
  }
  execute() { this.doc.content += this.text; }
  undo() { this.doc.content = this.doc.content.slice(0, -this.text.length); }
}

class CommandHistory {
  #undoStack = [];
  #redoStack = [];

  execute(cmd) {
    cmd.execute();
    this.#undoStack.push(cmd);
    this.#redoStack.length = 0; // clear redo on new action
  }

  undo() {
    const cmd = this.#undoStack.pop();
    if (cmd) { cmd.undo(); this.#redoStack.push(cmd); }
  }

  redo() {
    const cmd = this.#redoStack.pop();
    if (cmd) { cmd.execute(); this.#undoStack.push(cmd); }
  }
}

const doc = { content: "" };
const history = new CommandHistory();

history.execute(new AddTextCommand(doc, "Hello"));
history.execute(new AddTextCommand(doc, " World"));
console.log(doc.content); // "Hello World"

history.undo();
console.log(doc.content); // "Hello"

history.redo();
console.log(doc.content); // "Hello World"
Hello World Hello Hello World
JavaScript — Chain of Responsibility: Middleware Pipeline
function createPipeline(...middlewares) {
  return function run(context) {
    let index = 0;

    function next() {
      if (index < middlewares.length) {
        const middleware = middlewares[index++];
        middleware(context, next);
      }
    }

    next();
  };
}

// Middleware functions — each calls next() to continue or stops the chain
const logger = (ctx, next) => {
  console.log(`[${new Date().toISOString()}] ${ctx.method} ${ctx.path}`);
  next();
};

const auth = (ctx, next) => {
  if (!ctx.token) {
    console.log("401 Unauthorized");
    return; // stop chain
  }
  ctx.user = "Alice";
  next();
};

const handler = (ctx, next) => {
  console.log(`200 OK — Hello, ${ctx.user}!`);
};

const pipeline = createPipeline(logger, auth, handler);

pipeline({ method: "GET", path: "/api", token: "abc123" });
pipeline({ method: "GET", path: "/api", token: null });
[2026-06-12T00:00:00.000Z] GET /api 200 OK — Hello, Alice! [2026-06-12T00:00:00.000Z] GET /api 401 Unauthorized
Common Mistakes
Using the Observer pattern without ever unsubscribing — leads to memory leaks and stale callbacks
Always return an unsubscribe function from on() and call it when the subscriber is no longer needed
Over-engineering with classes when a simple function would suffice (e.g., wrapping a Strategy in a class with one method)
In JS, a function is a strategy. Use objects/classes only when you need state or multiple methods
Forgetting to clear the redo stack when executing a new command after undoing
Always reset the redo stack on new execute() — the old redo path is no longer valid
Key Rules
  • Observer: decouple publishers from subscribers; always provide an unsubscribe mechanism
  • Strategy: pass behavior as functions; avoid if/else chains for algorithm selection
  • Command: encapsulate operations for undo, queuing, and logging
  • Chain of Responsibility: each handler can process or forward — great for middleware

#3. Functional Patterns: Compose, Pipe, Curry Modules & Patterns
Functional patterns let you build complex data transformations from small, reusable pieces. Compose and pipe create pipelines, currying enables partial application, and point-free style removes boilerplate — all without mutation or shared state.
  • Function composition: compose(f, g)(x) means f(g(x)) — right-to-left evaluation. Mathematical composition.
  • Pipe: pipe(f, g)(x) means g(f(x)) — left-to-right, more natural reading order. Same power, different direction.
  • Currying: Transform f(a, b, c) into f(a)(b)(c). Each call returns a new function until all arguments are provided.
  • Partial application: Fix some arguments and return a function for the rest. partial(f, a)(b, c) calls f(a, b, c).
  • Point-free style: Define functions without naming their arguments: const getNames = map(prop('name')). Improves readability when the pipeline is clear; hurts when it gets cryptic.
JavaScript — compose and pipe
// compose: right-to-left — f(g(h(x)))
const compose = (...fns) =>
  (x) => fns.reduceRight((acc, fn) => fn(acc), x);

// pipe: left-to-right — h(g(f(x)))
const pipe = (...fns) =>
  (x) => fns.reduce((acc, fn) => fn(acc), x);

// Small, reusable functions
const double = x => x * 2;
const increment = x => x + 1;
const square = x => x * x;

// compose: square first, then increment, then double
const transform = compose(double, increment, square);
console.log(transform(3)); // double(increment(square(3))) = double(increment(9)) = double(10) = 20

// pipe: double first, then increment, then square (different order!)
const pipeline = pipe(double, increment, square);
console.log(pipeline(3)); // square(increment(double(3))) = square(increment(6)) = square(7) = 49

JavaScript — Currying
// Generic curry function
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args);
    }
    return function (...moreArgs) {
      return curried.apply(this, [...args, ...moreArgs]);
    };
  };
}

const add = curry((a, b, c) => a + b + c);

console.log(add(1)(2)(3));    // 6 — fully curried
console.log(add(1, 2)(3));    // 6 — partial then rest
console.log(add(1)(2, 3));    // 6 — partial then rest
console.log(add(1, 2, 3));    // 6 — all at once

// Practical: create reusable, configured functions
const multiply = curry((a, b) => a * b);
const double = multiply(2);
const triple = multiply(3);

console.log([1, 2, 3, 4].map(double)); // [2, 4, 6, 8]
console.log([1, 2, 3, 4].map(triple)); // [3, 6, 9, 12]
6 6 6 6 [ 2, 4, 6, 8 ] [ 3, 6, 9, 12 ]
JavaScript — Partial Application
// partial: fix some arguments from the left
function partial(fn, ...fixed) {
  return function (...rest) {
    return fn(...fixed, ...rest);
  };
}

function log(level, timestamp, message) {
  console.log(`[${level}] ${timestamp}: ${message}`);
}

// Create specialized loggers
const warn = partial(log, "WARN");
const error = partial(log, "ERROR");

warn("2026-06-12", "Disk space low");  // [WARN] 2026-06-12: Disk space low
error("2026-06-12", "DB connection failed"); // [ERROR] 2026-06-12: DB connection failed

// Built-in: Function.prototype.bind also does partial application
const info = log.bind(null, "INFO", "2026-06-12");
info("Server started"); // [INFO] 2026-06-12: Server started
[WARN] 2026-06-12: Disk space low [ERROR] 2026-06-12: DB connection failed [INFO] 2026-06-12: Server started
JavaScript — Real-World Data Pipeline
// Building a data transformation pipeline
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

// Reusable transformers
const filter = (predicate) => (arr) => arr.filter(predicate);
const map = (fn) => (arr) => arr.map(fn);
const sortBy = (key) => (arr) => [...arr].sort((a, b) =>
  a[key] < b[key] ? -1 : a[key] > b[key] ? 1 : 0
);

const users = [
  { name: "Alice", age: 32, active: true },
  { name: "Bob", age: 25, active: false },
  { name: "Charlie", age: 28, active: true },
  { name: "Diana", age: 35, active: true },
  { name: "Eve", age: 22, active: true },
];

// Declarative pipeline: active users, sorted by age, top 3 names
const getTopActiveNames = pipe(
  filter(u => u.active),

Common Mistakes
Confusing compose (right-to-left) with pipe (left-to-right) — the function order is reversed
compose(f, g)(x) = f(g(x)) vs pipe(f, g)(x) = g(f(x)). Pipe reads like English, compose reads like math.
Currying variadic functions (those using ...args) — curry relies on fn.length, which is 0 for rest params
Only curry functions with a fixed number of parameters. For variadic functions, use partial application instead.
Making point-free code so dense that only the author can read it
If the pipeline is longer than 3-4 steps, name the intermediate functions for readability
Key Rules
  • pipe is left-to-right (data flows naturally); compose is right-to-left (math notation)
  • Curry when you want to create specialized functions from general ones
  • Use partial application when you want to fix some arguments without changing the function signature
  • Point-free is a readability tool, not a goal — stop when it gets cryptic

#4. Error Handling Patterns Modules & Patterns
Robust error handling separates production-quality code from prototypes. Custom errors enable typed catch blocks, the Result pattern eliminates unexpected throws, error chaining preserves context, and proper async error handling prevents silent failures.
  • Custom error classes: Extend Error for domain-specific errors with structured data (HTTP status, error codes, metadata).
  • cause property (ES2022): Chain errors by passing { cause: originalError } to preserve the full error trail.
  • Result type pattern: Return {ok: true, value} or {ok: false, error} instead of throwing. The caller must handle both cases. Common in Rust, Go, and robust JS libraries.
  • Error boundaries: Catch errors at well-defined boundaries (API route handler, event handler, component) rather than at every call site.
  • Library rule: Libraries should return errors, not throw them. Throwing forces callers into try/catch; returning gives them a choice.
  • Async errors: Unhandled promise rejections crash Node.js. Always attach .catch() or use try/catch in async functions. Listen for unhandledrejection as a safety net.
JavaScript — Custom Error Classes
class AppError extends Error {
  constructor(message, { code, statusCode, cause } = {}) {
    super(message, { cause });
    this.name = "AppError";
    this.code = code;
    this.statusCode = statusCode;
  }
}

class NotFoundError extends AppError {
  constructor(resource, id, { cause } = {}) {
    super(`${resource} with id "${id}" not found`, {
      code: "NOT_FOUND",
      statusCode: 404,
      cause
    });
    this.name = "NotFoundError";
    this.resource = resource;
    this.resourceId = id;
  }
}

class ValidationError extends AppError {
  constructor(field, message, { cause } = {}) {
    super(`Validation failed for "${field}": ${message}`, {
      code: "VALIDATION_ERROR",
      statusCode: 400,
      cause
    });
    this.name = "ValidationError";
    this.field = field;
  }
}

JavaScript — Result Type Pattern (No Throwing)
// Result type: the caller MUST handle success and failure
function ok(value) { return { ok: true, value }; }
function err(error) { return { ok: false, error }; }

function parseJSON(text) {
  try {
    return ok(JSON.parse(text));
  } catch (e) {
    return err(`Invalid JSON: ${e.message}`);
  }
}

function validateAge(data) {
  if (typeof data.age !== "number") return err("age must be a number");
  if (data.age < 0 || data.age > 150) return err("age out of range");
  return ok(data);
}

// Composing Results — chain operations, stop on first error
function chain(result, fn) {
  return result.ok ? fn(result.value) : result;
}

const input = '{"name": "Alice", "age": 30}';

const result = chain(
  parseJSON(input),
  data => validateAge(data)
);

if (result.ok) {
  console.log("Valid:", result.value);
} else {
  console.log("Error:", result.error);
}
// Valid: { name: 'Alice', age: 30 }
Valid: { name: 'Alice', age: 30 }

async function fetchAll(urls) {
  const results = await Promise.allSettled(
    urls.map(url => fetch(url).then(r => r.json()))
  );

  const successes = results
    .filter(r => r.status === "fulfilled")
    .map(r => r.value);

  const failures = results
    .filter(r => r.status === "rejected")
    .map(r => r.reason.message);

  if (failures.length) {
    console.warn(`${failures.length} requests failed:`, failures);
  }

  return successes;
}
JavaScript — Error Boundaries: Centralized Handling
// Express-style error boundary
function errorBoundary(handler) {
  return async (req, res) => {
    try {
      await handler(req, res);
    } catch (error) {
      if (error instanceof ValidationError) {
        res.status(400).json({ error: error.message, field: error.field });
      } else if (error instanceof NotFoundError) {
        res.status(404).json({ error: error.message });
      } else {
        console.error("Unexpected error:", error);
        res.status(500).json({ error: "Internal server error" });
      }
    }
  };
}

// Generic error boundary for any async operation
function withErrorBoundary(fn, fallback) {
  return async (...args) => {
    try {
      return await fn(...args);
    } catch (error) {
      console.error(`Error in ${fn.name || "anonymous"}:`, error);
      return typeof fallback === "function" ? fallback(error) : fallback;
    }
  };
}

  (err) => ({ error: err.message, data: null })
);
Common Mistakes
Catching errors and swallowing them: catch (e) {} — silent failures are the worst kind of bugs
Always log, re-throw, or return an error result. If you intentionally ignore an error, add a comment explaining why.
Throwing strings: throw "something went wrong" — no stack trace, no instanceof, no cause
Always throw Error instances: throw new Error("something went wrong")
Using try/catch around every single function call — clutters code and hides the happy path
Catch at boundaries (route handlers, event handlers, entry points). Let errors propagate naturally.
Key Rules
  • Extend Error for custom errors — include code, statusCode, and structured data
  • Use { cause } to chain errors and preserve the original context
  • Prefer the Result pattern in library code — never force callers into try/catch
  • Catch at boundaries, not at every call site
  • Never swallow errors silently

#5. Pub/Sub & EventTarget Modules & Patterns
EventTarget is the browser's built-in Observer pattern. Understanding CustomEvent, event delegation, listener options (once, passive, signal), and how to extend EventTarget for your own classes enables decoupled, event-driven architecture without third-party libraries.
  • EventTarget: The base class for all event-emitting objects in the browser (DOM elements, AbortController, etc.). You can extend it for custom classes.
  • CustomEvent: Create custom events with arbitrary data via the detail property. Use new CustomEvent("name", { detail: data }).
  • addEventListener options: once (auto-remove after fire), passive (promise not to call preventDefault), capture (fire during capture phase), signal (auto-remove when AbortController aborts).
  • Event delegation: Attach one listener on a parent to handle events from all children. Uses event.target to identify the actual source. Fewer listeners, works for dynamically added elements.
  • Type-safe event emitter: Use JSDoc or TypeScript to constrain event names and payload types.
JavaScript — Extending EventTarget for Custom Classes
class Store extends EventTarget {
  #state;

  constructor(initialState) {
    super();
    this.#state = initialState;
  }

  get state() { return this.#state; }

  setState(updater) {
    const prev = this.#state;
    this.#state = typeof updater === "function"
      ? updater(prev)
      : { ...prev, ...updater };

    this.dispatchEvent(new CustomEvent("change", {
      detail: { prev, next: this.#state }
    }));
  }
}

JavaScript — Event Delegation
// Instead of one listener per button, one listener on the parent
// <ul id="list">
//   <li data-id="1">Item 1 <button data-action="delete">X</button></li>
//   <li data-id="2">Item 2 <button data-action="delete">X</button></li>
//   <li data-id="3">Item 3 <button data-action="delete">X</button></li>
// </ul>

// Pure JS event delegation
function delegate(parent, selector, eventType, handler) {
  parent.addEventListener(eventType, (event) => {
    const target = event.target.closest(selector);
    if (target && parent.contains(target)) {
      handler(event, target);
    }
  });
}

JavaScript — Type-Safe Event Emitter (JSDoc)
/**
 * @typedef {Object} AppEvents
 * @property {{ user: string, timestamp: number }} "auth:login"
 * @property {{ reason: string }} "auth:logout"
 * @property {{ key: string, prev: any, next: any }} "state:change"
 */

class TypedEventEmitter extends EventTarget {
  /**
   * @template {keyof AppEvents} K
   * @param {K} type
   * @param {(event: CustomEvent<AppEvents[K]>) => void} handler
   * @param {AddEventListenerOptions} [options]
   */
  on(type, handler, options) {
    this.addEventListener(type, handler, options);
    return () => this.removeEventListener(type, handler, options);
  }

  /**
   * @template {keyof AppEvents} K
   * @param {K} type
   * @param {AppEvents[K]} detail
   */
  fire(type, detail) {
    this.dispatchEvent(new CustomEvent(type, { detail }));
  }
}

const emitter = new TypedEventEmitter();

// IDE provides autocomplete for event names and payload types
const unsub = emitter.on("auth:login", (e) => {
  console.log(`${e.detail.user} logged in at ${e.detail.timestamp}`);
});

emitter.fire("auth:login", { user: "Alice", timestamp: Date.now() });
unsub();
Alice logged in at 1749686400000
Common Mistakes
Using removeEventListener with an anonymous function — it requires the same function reference
Store the handler in a variable, or use { once: true }, or use { signal } with an AbortController for cleanup
Forgetting { bubbles: true } on CustomEvent when you need it to bubble up the DOM tree
CustomEvent does not bubble by default. Pass { bubbles: true } in the options if parent elements need to catch it.
Adding passive: true and then calling event.preventDefault() inside the handler
passive: true is a contract that you will NOT call preventDefault(). The browser ignores the call and logs a warning.
Key Rules
  • Extend EventTarget instead of writing custom pub/sub in browser environments
  • Use CustomEvent with detail for custom data payloads
  • Prefer { signal } for listener cleanup — one abort() removes all listeners tied to that signal
  • Use event delegation for lists and dynamic content — one listener on the parent
  • passive: true for scroll/touch handlers improves scrolling performance
