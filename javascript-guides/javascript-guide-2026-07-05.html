Typed Arrays & ArrayBuffer Data & Collections 7 days since review
An ArrayBuffer is a fixed-length block of raw bytes; you never touch it directly — you overlay a view. Typed arrays (Uint8Array, Int32Array, Float64Array…) are same-endianness views giving numeric access, while a DataView lets you read/write mixed types at byte offsets with explicit endianness. Multiple views can share one buffer, so a write through one is visible through another.
JavaScript
const buf = new ArrayBuffer(8);
const u32 = new Uint32Array(buf);      // view A: two 32-bit slots
const bytes = new Uint8Array(buf);      // view B: eight bytes, SAME buffer
u32[0] = 0x11223344;
console.log(bytes[0].toString(16));     // '44' on little-endian (byte order!)

const dv = new DataView(buf);
dv.setUint32(0, 0x11223344, false);     // false = big-endian, explicit
console.log(dv.getUint8(0).toString(16)); // '11'
Challenge:
Parse a 6-byte record — a big-endian uint16 id followed by a little-endian float32 price — from a Uint8Array using a single DataView, returning { id, price }. Then explain why reading the same bytes through a Uint32Array gives a different number on your machine than the big-endian value.
#1 Pitfall: Assuming a byte order. Typed arrays use the platform's native endianness (usually little-endian), so serializing with Uint32Array and reading elsewhere corrupts data — use DataView with an explicit endianness flag for any wire/file format. Also: slice() copies, subarray() shares the buffer.
Deno & Bun: Alternative Runtimes Node.js Runtime 2 days since review
Deno runs TypeScript directly, is secure-by-default (explicit --allow-read/--allow-net permissions), ships web-standard APIs (fetch, Deno.serve) plus URL/npm imports, and bundles a formatter/linter/test runner. Bun is a speed-focused all-in-one (runtime + bundler + package manager + test runner) on JavaScriptCore, largely Node-compatible with fast bun install and native TS. Both shrink the toolchain versus Node + separate tools.
Challenge:
Without notes: list three defaults Deno changes from Node (permissions, module resolution, built-in TS), name one reason bun install beats npm install, and give one web-standard API that behaves identically in Deno, Bun, and the browser. Then state what a Deno script does if it reads a file without --allow-read.

Package Management: npm, pnpm & yarn Node.js Runtime 2 days since review
npm uses a flat, hoisted node_modules. pnpm stores each package once in a global content-addressable store and links it in via a symlinked, non-flat layout — saving disk and preventing "phantom dependencies" (importing something you never declared). yarn (Berry) offers Plug'n'Play (no node_modules) and zero-installs. Always commit the lockfile; in CI use npm ci (not install) for a clean, lockfile-exact, reproducible tree.
Challenge:
Define a "phantom dependency" and explain why pnpm's non-flat layout prevents it. Then contrast npm install vs npm ci for CI, and state exactly what ^1.2.3 and ~1.2.3 each permit to change on the next install.
#1 Pitfall: Not committing the lockfile (or deleting it to "fix" an install) — you lose reproducibility and each machine can resolve different versions. Also relying on a hoisted package you never listed in dependencies: it works under npm's flat tree but breaks under pnpm/PnP and the moment a dependency drops it.
V8 Internals: Hidden Classes, Inline Caching & JIT Engine Internals 18 days since review
V8 gives each object a hidden shape (hidden class / "map") describing its property layout; objects created with the same properties in the same order share a shape, letting inline caches (ICs) at a property-access site skip the lookup. Adding properties in varying orders, or deleting them, forks shapes and deoptimizes hot code. Hot functions are JIT-compiled by TurboFan on the assumption that types stay stable; feeding mixed types triggers a deopt back to the interpreter.
JavaScript
// ✅ same property order → both objects share ONE hidden class
function makePoint(x, y) { return { x, y }; }   // stable shape
const a = makePoint(1, 2), b = makePoint(3, 4); // monomorphic access site

// ❌ different order / late props → new shapes, polymorphic IC
const c = { y: 2 }; c.x = 1;          // shape differs from {x, y}
delete c.y;                            // deletion deopts to dictionary mode
Challenge:
Explain why initializing every field in the constructor (in a fixed order) makes a class faster than assigning fields conditionally later. Then describe what turns a monomorphic call site (one shape) into a megamorphic one, and why passing both number and string to the same hot function forces a deopt.
#1 Pitfall: Micro-optimizing blindly. These internals matter only for genuinely hot paths; premature "shape-friendly" contortions hurt readability. But do avoid delete on hot objects (use obj.x = undefined or a Map) and keep property init order consistent — those two habits are free.
Polyfills & Transpilation: Babel & Feature Detection Engine Internals 18 days since review
Transpilation (Babel/tsc) rewrites new syntax (arrow fns, optional chaining) into older equivalents at build time. A polyfill supplies a missing runtime API (Array.prototype.at, Promise) by defining it if absent — syntax can't be polyfilled, only transpiled. Detect features, never sniff the user-agent string: if (!('at' in Array.prototype)) { /* install */ }. core-js + @babel/preset-env with a browserslist target injects only the polyfills your audience needs.
JavaScript
// ✅ feature detection: ask if the capability exists
if (typeof structuredClone !== 'function') {
  globalThis.structuredClone = (v) => JSON.parse(JSON.stringify(v)); // crude fallback
}

// ❌ user-agent sniffing: brittle, lies, breaks on new browsers
if (/Chrome\/8/.test(navigator.userAgent)) { /* guess capabilities */ }
Challenge:
Classify each as needing a polyfill, a transpile, or neither: optional chaining ?., Array.prototype.flat, Object.hasOwn, and top-level await. Then explain why @babel/preset-env driven by a browserslist target ships less code than importing all of core-js.
#1 Pitfall: Confusing the two: trying to "polyfill" syntax (impossible — the parser fails before any code runs) or shipping the entire core-js to every user regardless of target. Also, polyfilling by overwriting a native method that already exists can mask a spec-compliant implementation with a buggy one — always guard with a presence check.
Web Animations API (WAAPI) Modern Web APIs 17 days since review
element.animate(keyframes, options) runs CSS-quality animations from JavaScript, returning an Animation object you can pause(), reverse(), seek (currentTime), rate-adjust (playbackRate), and await via its finished promise. Keyframes are an array of style objects (or an object of arrays); options set duration, easing, iterations, fill, delay. Animations run on the compositor where possible, so transforms/opacity stay smooth off the main thread.
JavaScript
const anim = box.animate(
  [{ transform: 'translateX(0)' }, { transform: 'translateX(200px)' }],
  { duration: 400, easing: 'ease-out', fill: 'forwards' }
);
anim.pause();
anim.playbackRate = 2;          // play twice as fast
anim.play();
await anim.finished;            // a promise — resolves when done
console.log('animation complete');
Challenge:
Animate an element fading and sliding in, then chain a second animation to start only after the first's finished promise resolves. Explain what fill: 'forwards' does to the element's rendered state after the animation ends, and why animating transform/opacity is smoother than animating left/top.
#1 Pitfall: Animating layout-triggering properties (width, left, margin) — they force reflow every frame and jank. Prefer transform and opacity, which the compositor handles. Also, without fill: 'forwards' the element snaps back to its pre-animation style when the animation finishes.

🆕 Numbers & Math Utilities

The static helpers on Number and Math that replace fragile global functions and hand-rolled formulas — safer integer checks, correct rounding, and precision-aware comparisons.

Number Static Methods Core Language

The global parseInt, isNaN, and isFinite coerce their argument to a string/number first, producing surprising truthy results (isNaN('') is false!). The Number.* statics are the non-coercing, predictable replacements, plus they expose the constants (MAX_SAFE_INTEGER, EPSILON) you need to reason about integer and float precision correctly.

1Type-safe checks that don't coerce. Number.isNaN, Number.isFinite, and Number.isInteger return false for any non-number argument instead of coercing it — so a string never sneaks through.

JavaScript
// Global versions coerce first — surprising results:
console.log(isNaN('hello'));          // true  ('hello' → NaN)
console.log(isFinite('42'));          // true  ('42' → 42)

// Number.* versions never coerce:
console.log(Number.isNaN('hello'));   // false (a string is not NaN)
console.log(Number.isFinite('42'));   // false (a string is not finite)
console.log(Number.isInteger(4.0));   // true
console.log(Number.isInteger(4.5));   // false
true true false false true false

2Safe integers: the 2^53 boundary. JavaScript numbers are IEEE-754 doubles, so only integers up to Number.MAX_SAFE_INTEGER (2^53 − 1) are representable exactly. Beyond it, distinct integers collapse onto the same double. Number.isSafeInteger tells you whether a value is in the trustworthy range.

JavaScript
console.log(Number.MAX_SAFE_INTEGER);        // 9007199254740991
console.log(9007199254740992 === 9007199254740993); // true (!) — both round to the same double
console.log(Number.isSafeInteger(2 ** 53));  // false — past the safe range
console.log(Number.isSafeInteger(2 ** 53 - 1)); // true

// Past the boundary you need BigInt to stay exact:
console.log(2n ** 53n + 1n);                 // 9007199254740993n — exact
9007199254740991 true false true 9007199254740993n

3Non-coercing parsers and the float tolerance. Number.parseInt/Number.parseFloat are identical to the globals (moved onto Number for modularity). Number.EPSILON is the smallest gap between 1 and the next double — the correct tolerance for "are these floats effectively equal?" comparisons.

JavaScript
console.log(Number.parseInt('0xFF', 16));    // 255
console.log(Number.parseFloat('3.14abc'));   // 3.14 (stops at first invalid char)

console.log(0.1 + 0.2 === 0.3);              // false — 0.1+0.2 is 0.30000000000000004
const nearlyEqual = (a, b) => Math.abs(a - b) < Number.EPSILON;
console.log(nearlyEqual(0.1 + 0.2, 0.3));    // true — within one ULP
255 3.14 false true
Common Mistakes:
❌ Trusting the coercing globals and exact float equality: if (isNaN(userInput)) ... // 'abc' and '' behave inconsistently if (total === 0.3) ... // fails for 0.1 + 0.2 if (Number.isInteger(id)) ... // silently false for the STRING "5" from a form
✅ Use Number.* checks and an epsilon tolerance: if (Number.isNaN(x)) ... // only true for an actual NaN number if (Math.abs(total - 0.3) < Number.EPSILON) ... // float-safe compare const n = Number(str); if (Number.isInteger(n)) ... // parse THEN check
Key Rules:
  • Prefer Number.isNaN/Number.isFinite/Number.isInteger over the coercing globals — they return false for non-numbers instead of guessing.
  • Only integers up to Number.MAX_SAFE_INTEGER (2^53 − 1) are exact; guard with Number.isSafeInteger and reach for BigInt beyond it (IDs, currency in cents, timestamps in ns).
  • Never compare floats with ===; compare Math.abs(a - b) against Number.EPSILON (or a domain tolerance) instead.
Practice Exercises:
Write isPlainInteger(v) that returns true only for a real integer number (not a numeric string, not NaN, not Infinity) — and prove it rejects '5', 5.5, and NaN while accepting 5 and -0.
Given two prices as floats, write equalMoney(a, b) that treats them as equal when within a cent, and show why Number.EPSILON alone is too tight for values around 1000 (hint: scale the tolerance or work in integer cents).
Demonstrate the safe-integer boundary: find the smallest integer n for which n + 1 === n is true, confirm Number.isSafeInteger(n) is false, then redo the arithmetic with BigInt to get the correct distinct result.
Connections: The safe-integer boundary is exactly why typed arrays and BigInt exist for precise binary/large-number work, and these checks pair with Object.is for distinguishing NaN and -0. Number.EPSILON float-comparison shows up again in Math utilities below.
Math Utility Methods Core Language

Math hides a lot of correctness beyond round/floor: Math.trunc and Math.sign remove the classic "which way does it round for negatives?" bugs, Math.hypot computes distances without overflow, and clz32/fround/cbrt exist for bit-twiddling and numeric work. Knowing them means fewer hand-rolled, subtly-wrong formulas.

1Directional rounding done right. Math.trunc drops the fractional part (toward zero) — unlike Math.floor, which goes toward −∞ and mis-handles negatives. Math.sign returns −1, 0, or 1 (and preserves -0/NaN), replacing brittle x > 0 ? 1 : -1 ternaries.

JavaScript
console.log(Math.trunc(4.9));    //  4
console.log(Math.trunc(-4.9));   // -4  (toward zero)
console.log(Math.floor(-4.9));   // -5  (toward -Infinity — the classic trap)

console.log(Math.sign(-3));      // -1
console.log(Math.sign(0));       //  0
console.log(Math.sign(2));       //  1
4 -4 -5 -1 0 1

2Roots and overflow-safe distance. Math.cbrt gives a true cube root (works for negatives, where x ** (1/3) yields NaN). Math.hypot(...args) is √(a²+b²+…) computed to avoid intermediate overflow — the correct way to get a Euclidean distance or vector length.

JavaScript
console.log(Math.cbrt(-27));           // -3   (real cube root)
console.log((-27) ** (1 / 3));         // NaN  (fractional power of a negative)

console.log(Math.hypot(3, 4));         // 5    (√(9+16))
console.log(Math.hypot(3e200, 4e200)); // 5e200 — no overflow to Infinity
console.log(Math.sqrt(3e200**2 + 4e200**2)); // Infinity — naive version overflows
-3 NaN 5 5e+200 Infinity

3Bit-level and float32 helpers. Math.clz32(x) counts leading zero bits in a 32-bit representation (handy for bit-width / log2 tricks). Math.fround(x) rounds a double to the nearest 32-bit float — matching what a Float32Array would store, useful when reconciling JS numbers with float32 data.

JavaScript
console.log(Math.clz32(1));            // 31  (…0001 → 31 leading zeros)
console.log(Math.clz32(0xFFFF));      // 16  (16-bit value in a 32-bit word)
console.log(32 - Math.clz32(1000));   // 10  → bits needed to hold 1000

console.log(Math.fround(1.1));        // 1.100000023841858 (nearest float32)
console.log(Math.fround(1.1) === new Float32Array([1.1])[0]); // true
31 16 10 1.100000023841858 true
Common Mistakes:
❌ Hand-rolled rounding, sign, and distance: const whole = n < 0 ? Math.ceil(n) : Math.floor(n); // reinventing Math.trunc const s = x > 0 ? 1 : x < 0 ? -1 : 0; // reinventing Math.sign (loses -0) const d = Math.sqrt(dx*dx + dy*dy); // can overflow for large components
✅ Use the built-ins: const whole = Math.trunc(n); // toward zero, negatives handled const s = Math.sign(x); // -1 / 0 / 1, preserves -0 and NaN const d = Math.hypot(dx, dy); // overflow-safe Euclidean distance
Key Rules:
  • Math.trunc (toward zero) ≠ Math.floor (toward −∞): they differ for negatives — pick deliberately.
  • Use Math.hypot for distances/magnitudes (overflow-safe) and Math.cbrt for cube roots of possibly-negative numbers; x ** (1/3) returns NaN for negatives.
  • Math.fround mirrors Float32Array storage and Math.clz32 operates on the 32-bit integer view — both bridge JS doubles to lower-precision/binary representations.
Practice Exercises:
Write bitLength(n) for a non-negative integer using Math.clz32 (returning 0 for 0), and verify it against n.toString(2).length for several values.
Implement distance(p1, p2) for 3-D points with Math.hypot, then show a pair of large-magnitude points where a naive Math.sqrt(sum of squares) overflows to Infinity but Math.hypot stays finite.
Using Math.trunc and Math.sign, write splitNumber(n) returning { sign, whole, fraction } such that sign * (whole + fraction) reconstructs n for both positive and negative inputs — and confirm it round-trips -4.25.
Connections: Math.fround and clz32 tie directly to typed arrays and 32-bit integer semantics, while the float-precision theme continues from Number statics and Number.EPSILON above.

🆕 String Utility Methods

Everyday string helpers that eliminate hand-rolled loops and error-prone regexes — fixed-width padding, edge trimming, and literal-safe global replacement.

String Padding & Trimming Data & Collections

Formatting tabular output, zero-padding time components, building fixed-width IDs, or aligning a CLI table used to mean manual loops or regex hacks. padStart/padEnd and the one-sided trimStart/trimEnd do it declaratively and correctly, including a custom pad string.

1Pad to a target length. str.padStart(targetLen, padStr) prepends copies of padStr (default a space) until the string reaches targetLen; padEnd appends. The pad string is truncated to fit exactly. If the string is already at least targetLen, it's returned unchanged.

JavaScript
console.log('5'.padStart(3, '0'));        // '005'  — zero-pad
console.log('42'.padEnd(5, '.'));         // '42...'
console.log('abc'.padStart(2, '0'));      // 'abc'  — already long enough, unchanged
console.log('7'.padStart(6, 'ab'));       // 'ababa7' — pad string repeats & truncates
005 42... abc ababa7

2Real formatting: times and tables. Combine padStart for zero-padded numeric fields and padEnd for left-aligned text columns.

JavaScript
const hh = 9, mm = 5, ss = 3;
const time = [hh, mm, ss].map(n => String(n).padStart(2, '0')).join(':');
console.log(time);                        // '09:05:03'

const rows = [['Alice', 30], ['Bob', 7]];
for (const [name, score] of rows) {
  console.log(name.padEnd(8) + String(score).padStart(4));
}
09:05:03 Alice 30 Bob 7

3One-sided trimming. trimStart/trimEnd strip whitespace from just one edge (versus trim, which does both) — useful when trailing indentation is significant or you're normalizing only leading whitespace. They only remove whitespace; to strip other characters you still need a regex or replaceAll.

JavaScript
console.log(JSON.stringify('   hi   '.trimStart()));  // '"hi   "'
console.log(JSON.stringify('   hi   '.trimEnd()));    // '"   hi"'
console.log(JSON.stringify('   hi   '.trim()));       // '"hi"'

// trimStart only removes whitespace, not arbitrary chars:
console.log('xxhi'.trimStart());          // 'xxhi' (x is not whitespace)
"hi " " hi" "hi" xxhi
Common Mistakes:
❌ Manual padding loops and empty pad strings: let s = String(n); while (s.length < 3) s = '0' + s; // verbose, off-by-one prone '5'.padStart(3, ''); // '' pad → no padding at all, returns '5' unchanged ' hi '.replace(/^\s+/, '') // reinventing trimStart with a regex
✅ Use the built-ins: String(n).padStart(3, '0'); // '005' '5'.padStart(3, '0'); // give a NON-empty pad string ' hi '.trimStart(); // clear intent, no regex
Key Rules:
  • padStart/padEnd take a target total length, not a count of pad characters; a string already that long is returned unchanged (never truncated).
  • An empty pad string means no padding; the default pad is a single space. Length is measured in UTF-16 code units, so astral chars/emoji count as 2 — pad plain digits, not arbitrary Unicode.
  • trimStart/trimEnd remove only whitespace from one end; for arbitrary leading/trailing characters use replaceAll or a regex.
Practice Exercises:
Write formatDuration(sec) that turns a number of seconds into HH:MM:SS using padStart(2, '0') on each component; verify 3723 → '01:02:03'.
Given an array of { name, price } objects, print a right-aligned price column and left-aligned name column so every row lines up, using padEnd/padStart sized to the longest entry.

Connections: The UTF-16 code-unit caveat is the same one behind string .length vs byte length and grapheme segmentation. For stripping non-whitespace edges you'll reach for replaceAll, next.
String.prototype.replaceAll Data & Collections

The classic footgun: str.replace('a', 'b') replaces only the first match, so people reach for a global regex — and get bitten when their target contains regex metacharacters. replaceAll with a string argument replaces every occurrence literally, no escaping needed, making it the correct default for plain find-and-replace.

1String argument = replace every literal occurrence. Unlike replace(string, …) (first match only), replaceAll(string, …) replaces all of them, treating the search as a literal — dots, slashes, and parens are not special.

JavaScript
console.log('a-b-c'.replace('-', '_'));      // 'a_b-c'  — first only
console.log('a-b-c'.replaceAll('-', '_'));   // 'a_b_c'  — all of them

// metacharacters are literal — no escaping needed:
console.log('1.2.3'.replaceAll('.', '-'));   // '1-2-3'
console.log('a.b.c'.replace(/./g, '-'));     // '-----'  — regex '.' matched EVERYTHING
a_b-c a_b_c 1-2-3 -----

2Regex argument must be global. You can pass a regex to replaceAll, but it must carry the g flag or it throws a TypeError — a deliberate guard against the "replaceAll but forgot global" mistake. With g, it behaves like replace(/…/g, …).

JavaScript
console.log('aAbB'.replaceAll(/[a-z]/g, '_'));  // '_A_B' (global regex OK)

try {
  'aAbB'.replaceAll(/[a-z]/, '_');              // no 'g' flag
} catch (e) {
  console.log(e.name);                          // 'TypeError'
}
_A_B TypeError

3Replacement can be a function; $& patterns still work. The second argument may be a replacer function called for each match, or a string using $& (whole match), $1 (group), etc. This makes replaceAll a full-power global transform, not just literal swapping.

JavaScript
// wrap every word in brackets via a replacer function:
console.log('one two'.replaceAll(/\w+/g, (m) => `[${m}]`));  // '[one][two]'

// $& refers to the matched substring:
console.log('cat cat'.replaceAll('cat', '<$&>'));            // '<cat> <cat>'

// build a tiny template filler:
const fill = (tpl, data) => tpl.replaceAll(/\{(\w+)\}/g, (_, k) => data[k] ?? '');
console.log(fill('Hi {name}, you have {n} msgs', { name: 'Al', n: 3 }));
[one][two] <cat> <cat> Hi Al, you have 3 msgs
Common Mistakes:
❌ First-match replace, or unescaped dynamic regex: url.replace('/', '_') // only the FIRST slash changes str.replace(new RegExp(userInput, 'g'), x) // userInput '.' or '(' breaks / injects '1.2'.replaceAll(/./g, '-') // regex '.' = any char → all replaced
✅ Use replaceAll with a literal string: url.replaceAll('/', '_') // every slash, literally str.replaceAll(userInput, x) // literal — no regex escaping needed '1.2'.replaceAll('.', '-') // '.' is a literal dot here
Key Rules:
  • Prefer a string argument for plain find-and-replace — it's literal, so regex metacharacters and user input need no escaping.
  • A regex argument must have the g flag or replaceAll throws TypeError; otherwise it's equivalent to replace(/…/g, …).
  • The replacement can be a function or a $&/$1 pattern string — for splitting into an array instead, remember split/join is another literal-replace idiom.
Practice Exercises:
Write slugify(title) that lowercases, trims, and uses replaceAll to turn runs of spaces into single hyphens and strip a fixed set of punctuation — no manual escaping of the punctuation characters.
Implement censor(text, badWords) that replaces each banned word (taken from user data, possibly containing . or () with asterisks of equal length using the literal-string form of replaceAll — explaining why a new RegExp approach would be unsafe here.
Using the function-replacer form, write highlight(text, term) that wraps every case-insensitive occurrence of term in **…**, and note why this case requires a global regex rather than a plain string argument.
Connections: replaceAll's literal-vs-regex distinction is the string-level mirror of the escaping problem RegExp.escape solves, and its replacer-function form is the same callback style used by replace and array map. For non-whitespace edge stripping it complements trimStart/trimEnd.

🆕 Global Environment

One name for "the global object" that works in every environment — the portable replacement for window, self, and global.

globalThis Core Language

Before ES2020, referencing "the global object" meant guessing: window in browsers, self in workers, global in Node, frames in some contexts. Isomorphic code (libraries, polyfills) had to sniff which existed. globalThis is the single standard reference that resolves to the right global object everywhere — the correct way to feature-detect and install globals portably.

1One name, every runtime. globalThis is window in a browser, self in a Web/Service Worker, and global in Node — all the same object, reached by one identifier. Properties you read/write on it are the true globals.

JavaScript
// works in browser, worker, Node, Deno, Bun — no environment sniffing:
console.log(typeof globalThis);              // 'object'

globalThis.APP_VERSION = '1.4.0';            // define a real global
console.log(APP_VERSION);                    // '1.4.0' (bare reference resolves)

// in Node these are the same object:
// console.log(globalThis === global);       // true (in Node)
// in a browser: globalThis === window        // true
object 1.4.0

2Portable feature detection. Test for an API on globalThis instead of assuming it exists, then optionally install a fallback. This is the canonical polyfill guard — it works the same whether the code runs in a browser tab, a worker, or Node.

JavaScript
// is fetch available here?
if (typeof globalThis.fetch === 'function') {
  console.log('fetch is available');
} else {
  console.log('no fetch — load a polyfill');
}

// install a global only if missing (never clobber a native one):
if (typeof globalThis.structuredClone !== 'function') {
  globalThis.structuredClone = (v) => JSON.parse(JSON.stringify(v));
}
console.log(typeof globalThis.structuredClone);   // 'function'
fetch is available function

3Modules don't leak; this at top level differs. Inside an ES module, top-level this is undefined (not the global), and const/let/class declarations do not become properties of globalThis. To expose something globally from a module you must assign it explicitly. In non-strict scripts, top-level var/function declarations do attach to the global object.

JavaScript
// --- inside an ES module ---
const secret = 42;
console.log(globalThis.secret);        // undefined — module scope, not global
console.log(this);                     // undefined at module top level

globalThis.shared = secret;            // explicit opt-in to go global
console.log(globalThis.shared);        // 42

// --- classic non-module script (sloppy mode) ---
// var legacy = 1;  →  globalThis.legacy === 1  (var DOES attach)
undefined undefined 42
Common Mistakes:
❌ Environment sniffing and clobbering natives: const g = typeof window !== 'undefined' ? window : global; // brittle, misses workers globalThis.fetch = myFetch; // overwrites the real fetch unconditionally const x = this.someGlobal; // 'this' is undefined at ES-module top level
✅ Use globalThis and guard before assigning: const g = globalThis; // one portable reference if (!('fetch' in globalThis)) globalThis.fetch = myFetch; // only if missing const x = globalThis.someGlobal; // explicit global access
Key Rules:
  • Use globalThis as the one portable reference to the global object; it replaces window/self/global sniffing in isomorphic code and polyfills.
  • In ES modules, const/let/class stay module-scoped and top-level this is undefined — expose globals only by explicitly assigning to globalThis.
  • Prefer module imports over global state; when you must touch a global, feature-detect with 'name' in globalThis and never overwrite an existing native API unconditionally.
