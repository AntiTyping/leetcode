Closures Deep Dive Core Language 15 days since review
A closure is a function bundled with references to its surrounding lexical scope — the inner function keeps access to outer variables even after the outer function returns. Closures capture variables (live bindings), not snapshots of values, which is why they power data privacy (module pattern), factory functions, currying, and memoization. The classic trap: var in a loop shares one binding across all iterations; let creates a fresh binding per iteration.
JavaScript
function counter() {
  let n = 0;                       // private state, captured by the closure
  return { inc: () => ++n, get: () => n };
}
const c = counter();
c.inc(); c.inc();
console.log(c.get());              // 2  — n survives after counter() returned

// let vs var in a loop:
const fns = [];
for (let i = 0; i < 3; i++) fns.push(() => i);
console.log(fns.map(f => f()));    // [0, 1, 2]  (with var: [3, 3, 3])
Challenge:
Write once(fn) that runs fn at most once and caches its result, using a closure over a called flag. Then explain, in terms of live bindings, why replacing let with var in a for loop that pushes arrow functions makes them all print the same final number.
#1 Pitfall: Accidental retention — a closure keeps its entire captured scope alive, so a long-lived callback that closes over a large object prevents that object from being garbage-collected. Capture only what you need.
Prototypes & Inheritance Core Language 15 days since review
Every object has an internal [[Prototype]] link (read via Object.getPrototypeOf). Property lookups that miss on the object walk the prototype chain until found or reaching null. A function's .prototype object becomes the [[Prototype]] of instances created with new. class syntax is sugar over this prototype machinery — methods live on Class.prototype, shared by all instances. Object.create(proto) makes a new object with a chosen prototype.
JavaScript
const animal = { speak() { return `${this.name} makes a sound`; } };

const dog = Object.create(animal);      // dog.[[Prototype]] === animal
dog.name = 'Rex';
console.log(dog.speak());               // 'Rex makes a sound'  (found up the chain)
console.log(dog.hasOwnProperty('speak'));   // false — it's inherited
console.log(Object.getPrototypeOf(dog) === animal);   // true
Challenge:
Build a two-level chain with Object.create (grandparent → parent → child), shadow one inherited method on the child, and show that lookup stops at the first match. Then explain why methods defined with class live on the prototype (shared) while fields set in the constructor are per-instance.
#1 Pitfall: Mutating a shared prototype (e.g. adding to Array.prototype, or putting a mutable object as a class field on the prototype) leaks state across every instance. Never assign reference-type defaults to a shared prototype.
this Binding Core Language 15 days since review
this is determined by how a function is called, not where it's defined. Four rules, in priority order: new binding (this = the fresh instance), explicit (call/apply/bind), implicit (obj.method() → obj), and default (undefined in strict mode, else the global object). Arrow functions have no this of their own — they capture the lexical this from where they were defined, which is why they're ideal for callbacks.
JavaScript
const obj = {
  name: 'timer',
  startBad() { setTimeout(function () { console.log(this.name); }, 0); },  // undefined
  startGood() { setTimeout(() => console.log(this.name), 0); },           // 'timer'
};
obj.startGood();   // 'timer' — arrow captured obj's this

const unbound = obj.startGood;
// unbound();  // implicit binding lost — `this` would be undefined here
Challenge:
Predict this for the same function invoked four ways: standalone, as a method, via call, and with new. Then explain why passing obj.method as a callback "loses" this, and two fixes (bind vs wrapping in an arrow).
#1 Pitfall: Using an arrow function as an object method or a class prototype method — it captures the outer this (often the module/global), not the instance, so this won't refer to the object you called it on.
Symbols & Well-Known Symbols Meta-Programming 15 days since review
Symbol(desc) creates a unique, immutable primitive — perfect as a collision-proof property key (two symbols with the same description are still different). Symbol-keyed properties are skipped by for...in and JSON.stringify. Well-known symbols hook into language protocols: Symbol.iterator (for...of), Symbol.asyncIterator, Symbol.toPrimitive (coercion — today's new topic), Symbol.hasInstance (instanceof), and Symbol.toStringTag. Symbol.for(key) uses a cross-realm global registry.
JavaScript
const id = Symbol('id');
const user = { name: 'a', [id]: 42 };
console.log(Object.keys(user));        // ['name']  — symbol key hidden
console.log(user[id]);                 // 42

// make an object iterable via the well-known Symbol.iterator:
const range = {
  *[Symbol.iterator]() { yield 1; yield 2; yield 3; }
};
console.log([...range]);               // [1, 2, 3]
Challenge:
Use a Symbol to attach hidden metadata to an object and prove it doesn't show up in Object.keys or JSON.stringify. Then explain the difference between Symbol('x') === Symbol('x') and Symbol.for('x') === Symbol.for('x').
#1 Pitfall: Symbols are not truly private — Object.getOwnPropertySymbols and Reflect.ownKeys expose them. For real privacy use # private class fields. Also, symbol keys silently vanish in JSON serialization.
Generators & Iterators Functions & Iteration 15 days since review
An iterator has a next() returning { value, done }; an iterable exposes [Symbol.iterator](). A generator function (function*) is the easy way to build both: yield pauses execution and emits a value, resuming where it left off on the next next(). Communication is two-way — next(v) makes v the result of the paused yield. Generators enable lazy/infinite sequences, custom iteration, and cooperative control flow.
JavaScript
function* take(n, iterable) {
  let i = 0;
  for (const x of iterable) {
    if (i++ >= n) return;
    yield x;                          // pause & emit; resume on next()
  }
}
function* naturals() { let n = 1; while (true) yield n++; }   // infinite, lazy

console.log([...take(3, naturals())]);   // [1, 2, 3]  — only pulls what's needed
Challenge:
Write an infinite fibonacci() generator and a take(n, gen) helper, then collect the first 10 values. Then explain what next(value) does — how a value passed into next becomes the result of the yield expression inside the generator.
#1 Pitfall: A generator object is a one-shot iterator — once fully consumed (or after return) it's exhausted; iterating it again yields nothing. Re-call the generator function to get a fresh iterator.
Event Loop Deep Dive Async Mastery 15 days since review
JS runs on a single thread with two queues: macrotasks (timers, I/O, message events) and microtasks (promise reactions, queueMicrotask, await continuations). The rule: after the current synchronous run-to-completion finishes, the engine drains the entire microtask queue before taking the next macrotask (and before rendering). That's why a resolved Promise.then callback always runs before a setTimeout(…, 0) scheduled earlier.
JavaScript
console.log('1 sync');
setTimeout(() => console.log('4 macrotask'), 0);
Promise.resolve().then(() => console.log('3 microtask'));
console.log('2 sync');

// order: 1 sync → 2 sync → 3 microtask → 4 macrotask
// all microtasks drain before ANY macrotask runs
1 sync 2 sync 3 microtask 4 macrotask
Challenge:
Given a mix of console.log, setTimeout(…, 0), Promise.resolve().then, and queueMicrotask, write the exact output order and justify each step by naming the queue it belongs to. Then explain why an infinite chain of microtasks can starve the macrotask queue (and block rendering).
#1 Pitfall: Assuming setTimeout(fn, 0) runs "immediately" — it's a macrotask, so every pending microtask (and often a render) happens first. And a recursive queueMicrotask loop never yields to the browser, freezing the UI.

🆕 Object Coercion & Construction

Two hooks into the language's own object machinery: control exactly how your object turns into a primitive, and detect how a function was invoked. Both are plain-JS, every-runtime features.

Symbol.toPrimitive Meta-Programming

When an object appears where a primitive is needed — +obj, `${obj}`, obj < other, obj == 5 — JS runs an internal ToPrimitive algorithm that normally falls back to valueOf then toString (or the reverse, depending on context). Defining [Symbol.toPrimitive](hint) lets you take full control with a single method that knows which kind of primitive is being requested.

1One method, three hints. The engine calls your method with a hint of 'number', 'string', or 'default'. Unary + and comparisons pass 'number'; template literals and String() pass 'string'; binary + and == pass 'default'.

JavaScript
const money = {
  cents: 1299,
  [Symbol.toPrimitive](hint) {
    if (hint === 'number') return this.cents;                       // 1299
    if (hint === 'string') return `$${(this.cents / 100).toFixed(2)}`;  // '$12.99'
    return `Money(${this.cents})`;                                  // default
  },
};

console.log(+money);            // 1299          (hint 'number')
console.log(`${money}`);        // '$12.99'      (hint 'string')
console.log(money + '!');       // 'Money(1299)!'(hint 'default')
1299 $12.99 Money(1299)!

2It wins over valueOf and toString. If [Symbol.toPrimitive] exists, it is the only coercion path consulted — the legacy pair is ignored. That makes behavior predictable instead of depending on the subtle valueOf-first vs toString-first ordering.

JavaScript
const temp = {
  celsius: 20,
  valueOf() { return this.celsius; },        // legacy path...
  toString() { return `${this.celsius}°C`; },
  [Symbol.toPrimitive](hint) {               // ...is shadowed by this
    return hint === 'string' ? `${this.celsius}°C` : this.celsius;
  },
};
console.log(temp * 2);          // 40      (number hint → 20)
console.log(String(temp));      // '20°C'  (string hint)
console.log(temp > 15);         // true    (number hint → 20)
40 20°C true

3The 'default' hint is for ambiguous operators. Binary + (could be numeric add or string concat) and loose equality == use 'default'. Treat 'default' like 'number' unless your type is fundamentally string-like (as the built-in Date treats it as string).

JavaScript
const qty = {
  n: 3,
  [Symbol.toPrimitive](hint) {
    return hint === 'string' ? `${this.n} items` : this.n;   // default → number branch
  },
};
console.log(qty + 1);           // 4      ('default' → 3, numeric add)
console.log(qty == 3);          // true   ('default' → 3)
console.log(`Cart: ${qty}`);    // 'Cart: 3 items'  ('string')
4 true Cart: 3 items
Common Mistakes:
❌ Relying on valueOf/toString ordering, or ignoring the hint: const o = { valueOf(){return 5}, toString(){return 'five'} } `${o}` // 'five' via toString, but o+'' // '5' via valueOf — easy to confuse [Symbol.toPrimitive]() { return this.n } // ignores hint → wrong string form
✅ One method that branches on the hint: [Symbol.toPrimitive](hint) { if (hint === 'string') return this.label; // template literals, String() return this.value; // 'number' and 'default' }
Key Rules:
  • Hints: 'number' (unary +, -, *, <), 'string' (template literals, String()), 'default' (binary +, ==).
  • If present, [Symbol.toPrimitive] completely overrides valueOf and toString for coercion.
  • Always handle all three hints (usually fold 'default' into the number branch) so no operator surprises you.
Practice Exercises:
Build a Temperature object storing Celsius that returns the number for arithmetic and a "NN°C" string in template literals — using a single [Symbol.toPrimitive]. Verify all three hints fire correctly.
Show a case where valueOf-based coercion and toString-based coercion disagree, then add [Symbol.toPrimitive] to make the outcome deterministic. Explain which built-in type treats 'default' as string.
Explain why [Symbol.toPrimitive] is a well-known symbol and how that connects to the protocol idea you reviewed today — what other operator would you hook with Symbol.hasInstance?
Connections: This is a concrete member of the well-known symbols you reviewed today, and it's the coercion counterpart to Symbol.iterator's iteration protocol. The built-in Date uses it (string-preferring), which is why date + '' yields an ISO-ish string, not a number.
new.target Core Language

Sometimes a function or constructor must know how it was invoked — was it called with new, or as a plain function? And in a class hierarchy, which concrete subclass is actually being instantiated? new.target answers both: it's a meta-property that is undefined under a normal call and the constructor reference under a new call.

1Detect (or require) new. In a function, new.target is undefined when called without new, letting you enforce construction and avoid the classic bug where forgetting new silently pollutes the global object or returns undefined.

JavaScript
function User(name) {
  if (!new.target) throw new TypeError('User must be called with new');
  this.name = name;
}

const a = new User('Ada');       // ok
console.log(a.name);             // 'Ada'

try { User('Bob'); }            // called without new
catch (e) { console.log(e.message); }   // 'User must be called with new'
Ada User must be called with new

2In a base constructor, it's the most-derived class. When you new Derived(), the Base constructor sees new.target === Derived. This lets a base class read the real runtime class — useful for registries, factories, or reflecting the instance's constructor name.

JavaScript
class Shape {
  constructor() {
    console.log('constructing:', new.target.name);   // the concrete subclass
  }
}
class Circle extends Shape {}
class Square extends Shape {}

new Circle();   // 'constructing: Circle'
new Square();   // 'constructing: Square'
new Shape();    // 'constructing: Shape'
constructing: Circle constructing: Square constructing: Shape

3Enforce an abstract base class. Compare new.target to the class itself to forbid direct instantiation of a base that's only meant to be extended — a common OOP guard that JS doesn't provide natively.

JavaScript
class AbstractRepo {
  constructor() {
    if (new.target === AbstractRepo)
      throw new Error('AbstractRepo is abstract — extend it');
  }
}
class UserRepo extends AbstractRepo {}

console.log(new UserRepo() instanceof AbstractRepo);   // true
try { new AbstractRepo(); }
catch (e) { console.log(e.message); }   // 'AbstractRepo is abstract — extend it'
true AbstractRepo is abstract — extend it
Common Mistakes:
❌ Reaching for fragile alternatives, or using it where it's undefined: if (!(this instanceof User)) ... // breaks with subclassing / bound fns const arrow = () => new.target // SyntaxError — arrows have no new.target // assuming new.target is the class you wrote, not the subclass being built
✅ Use the meta-property directly: if (!new.target) throw new TypeError('use new') // require construction if (new.target === Base) throw ... // enforce abstract base const name = new.target.name // the concrete subclass
Key Rules:
  • new.target is undefined for normal calls and the invoked constructor for new calls (and for Reflect.construct).
  • Inside a superclass constructor it holds the most-derived class being instantiated, not the class where the code is written.
  • Arrow functions don't have their own new.target — they inherit it lexically, like this; you can't new an arrow.
Practice Exercises:
Write a function that works as both a constructor and a factory: when called without new, it internally does new for the caller (use new.target to branch). Verify both call styles produce a proper instance.
Create an abstract Component base that throws if instantiated directly but allows subclasses, using new.target === Component. Confirm instanceof still works for the subclass.
Explain why this instanceof Foo is a weaker guard than new.target for "was I called with new?", especially once inheritance or .call() is involved.
Connections: new.target ties directly into the prototype/construction and this-binding models you reviewed — it distinguishes the new binding rule from the others. It pairs with import.meta below as the language's two x.y "meta-properties".

🆕 Module-Level Execution

Two abilities unique to ES modules: awaiting asynchronous work at the top of a file, and reading metadata about the current module. Both are syntax errors in classic scripts — they only exist in module context.

Top-Level await Async Mastery

Inside an ES module you can write await at the top level — outside any async function. The module itself becomes asynchronous: anything that imports it waits until its top-level awaits settle before running. This cleanly handles startup work — loading config, opening a DB connection, conditionally choosing a dependency — without wrapping everything in an IIFE or exporting a promise.

1await at module scope. No surrounding async function needed. The awaited value is available synchronously to the rest of the module body once it resolves.

JavaScript
// config.mjs  (an ES module)
const res = await fetch('https://example.com/config.json');
export const config = await res.json();   // top-level await, no async wrapper
export const region = config.region;
// config.mjs finishes evaluating only after both awaits settle

2Importers wait for the awaited module. A module that imports one using top-level await is blocked until that dependency's async evaluation completes — the module graph resolves the promises for you, so consumers see fully-initialized exports.

JavaScript
// app.mjs
import { config } from './config.mjs';   // waits for config.mjs to finish
console.log('region is', config.region); // config is guaranteed ready here

// dynamic dependency selection at load time:
const { default: driver } =
  await import(process.env.DB === 'pg' ? './pg.mjs' : './sqlite.mjs');
region is us-east (only logs after config.mjs settled)

3It sequences the module graph — mind the cost. Sibling modules that don't depend on each other can still initialize in parallel, but a chain of top-level awaits serializes startup. And a circular dependency where each side awaits the other can deadlock. Reserve it for genuine startup I/O, not per-call work.

JavaScript
// ✅ good: one-time startup resource
export const db = await openDatabase();

// ⚠️ this blocks every importer for 2s — usually you want a lazy getter instead
await new Promise(r => setTimeout(r, 2000));
export const ready = true;
// importers of this module don't run until the 2s await resolves
Common Mistakes:
❌ Using it outside a module, or for hot-path work: <script>await fetch(...)</script> // SyntaxError — classic script, not a module // (Node) top-level await in a .cjs / CommonJS file → SyntaxError await heavyThing() // at module top for something only SOME runs need → blocks all importers
✅ Modules only, for genuine init: // in an .mjs / type="module" / "type":"module" package export const config = await loadConfig() // one-time startup I/O // for on-demand cost, export an async function or use import defer instead
Key Rules:
  • Only valid in ES modules (.mjs, <script type="module">, "type":"module") — never in classic scripts or CommonJS.
  • The module becomes async; its importers wait for the top-level awaits to settle before executing.
  • Great for startup config / connections / conditional imports; avoid for repeated work, and watch for deadlocks in circular graphs.
Practice Exercises:
Write data.mjs that top-level-awaits a (simulated) fetch and exports the result, and main.mjs that imports it and logs a value — confirm the log only appears after the fetch resolves, with no async wrapper anywhere.
Use top-level await import(...) to pick between two implementation modules based on an environment flag, exporting a unified interface. Explain why this is cleaner than a runtime if around two eager imports.
Describe how top-level await could deadlock a circular dependency, and contrast top-level await (defers importers on real async work) with import defer (defers only synchronous evaluation until first access).
Connections: Top-level await builds on the promise/event-loop model you reviewed — the module's evaluation is itself a microtask-scheduled continuation. It complements import defer (defer evaluation) and dynamic import() (defer fetch) from recent days.
import.meta Modules & Patterns

ES modules have no __dirname, __filename, or require. import.meta is the standard replacement: a module-scoped object carrying metadata about the current module — most importantly import.meta.url, from which you can resolve sibling files, and import.meta.resolve() to turn a specifier into a URL. Hosts (Node, browsers, bundlers) can add their own properties too.

1import.meta.url — where am I? It's the fully-qualified URL of the current module (a file:// URL in Node, an http(s):// URL in the browser). This is the anchor for locating files relative to the module.

JavaScript
console.log(import.meta.url);
// browser:  'https://example.com/js/app.mjs'
// node:     'file:///Users/me/project/app.mjs'
file:///Users/me/project/app.mjs

2Resolve sibling paths with the URL constructor. Combine import.meta.url as the base with a relative specifier to build a correct path to a data file or asset — the modern, portable replacement for path.join(__dirname, ...).

JavaScript
// point at ./data/seed.json next to this module:
const seedUrl = new URL('./data/seed.json', import.meta.url);
console.log(seedUrl.pathname);          // '/Users/me/project/data/seed.json'

// recover a __dirname-like value in an ES module (Node):
import { fileURLToPath } from 'node:url';
const __dirname = fileURLToPath(new URL('.', import.meta.url));
/Users/me/project/data/seed.json

3import.meta.resolve(spec) — module resolution as a string. It applies the host's full module-resolution algorithm (including node_modules and import maps) and returns the resolved URL without importing the module. Great for locating a dependency's asset or worker entry.

JavaScript
// resolve a bare specifier through node_modules / import maps:
const url = import.meta.resolve('lodash');
console.log(url);        // 'file:///.../node_modules/lodash/lodash.js'

// resolve a relative worker file to a URL you can hand to new Worker():
const workerUrl = import.meta.resolve('./worker.js');
file:///Users/me/project/node_modules/lodash/lodash.js
Common Mistakes:
❌ Old CommonJS habits, or string-munging URLs: const p = __dirname + '/data.json' // __dirname is undefined in ES modules const f = import.meta.url + '/x.json' // wrong — concatenates onto a file, not a dir import.meta.url.replace('file://','') // fragile on Windows / encoded chars
✅ Anchor with new URL() / fileURLToPath: new URL('./data.json', import.meta.url) // correct relative resolution fileURLToPath(new URL('.', import.meta.url)) // a real filesystem dir path import.meta.resolve('./worker.js') // host-accurate resolution
Key Rules:
  • import.meta is only valid inside ES modules — a SyntaxError in classic scripts or CommonJS.
  • Always resolve relative paths with new URL(relative, import.meta.url); never string-concatenate onto import.meta.url.
  • import.meta.resolve() returns a URL string using the real resolver (node_modules, import maps) without loading the module; bundlers may inject extra import.meta.* fields.
Practice Exercises:
In an ES module, build a portable path to a JSON file sitting next to it using new URL(..., import.meta.url), and (in Node) reconstruct __dirname via fileURLToPath. Explain why plain string concatenation breaks.
Use import.meta.resolve to get the URL of a dependency or a relative worker script, then explain how it differs from actually import()-ing that specifier.
Describe two host-specific import.meta extensions you might encounter (e.g. a bundler's hot-reload handle or env object) and why import.meta is designed to be extensible.
Connections: import.meta is the module sibling of new.target — both are x.y meta-properties, not real objects you can create. It leans on the URL API and the same ES-module system behind import defer, dynamic import(), and the Blob URLs used for workers.

🆕 Locale Display Names

The Intl tool that turns opaque codes — 'zh-Hant', 'US', 'EUR' — into human-readable names in any language. It pairs directly with yesterday's Intl.Locale. Ships everywhere today.

Intl.DisplayNames Internationalization

You have a language tag, a country code, a currency code, or a script code — and you need to show it to a human, translated into their language. Hard-coding a lookup table is a maintenance nightmare and never complete. Intl.DisplayNames uses the runtime's built-in CLDR data to render localized names for languages, regions, scripts, currencies, and calendars.

1Pick a type, then call .of(code). Construct with a locale and { type } ('language', 'region', 'script', 'currency', 'calendar', or 'dateTimeField'), then translate each code with .of().

JavaScript
const langs = new Intl.DisplayNames('en', { type: 'language' });
console.log(langs.of('zh-Hant'));     // 'Traditional Chinese'
console.log(langs.of('de'));          // 'German'

const regions = new Intl.DisplayNames('en', { type: 'region' });
console.log(regions.of('US'));        // 'United States'
console.log(regions.of('JP'));        // 'Japan'
Traditional Chinese German United States Japan

2The output is localized to the display locale. The first argument is the language the names are rendered in — so the same 'US' code becomes 'United States', 'Estados Unidos', or 'États-Unis' depending on the display locale.

JavaScript
const code = 'US';
console.log(new Intl.DisplayNames('en', { type: 'region' }).of(code));   // 'United States'
console.log(new Intl.DisplayNames('es', { type: 'region' }).of(code));   // 'Estados Unidos'
console.log(new Intl.DisplayNames('fr', { type: 'region' }).of(code));   // 'États-Unis'

// currencies and scripts too:
console.log(new Intl.DisplayNames('en', { type: 'currency' }).of('EUR'));  // 'Euro'
console.log(new Intl.DisplayNames('en', { type: 'script'   }).of('Cyrl')); // 'Cyrillic'
United States Estados Unidos États-Unis Euro Cyrillic

3Control the fallback for unknown codes. By default (fallback: 'code') an unrecognized code is returned as-is; with fallback: 'none' you get undefined instead — handy for validating input. languageDisplay: 'dialect' vs 'standard' tunes how language names read.

JavaScript
const strict = new Intl.DisplayNames('en', { type: 'region', fallback: 'none' });
console.log(strict.of('US'));         // 'United States'
console.log(strict.of('ZZ'));         // undefined  (invalid → no fallback)

const loose = new Intl.DisplayNames('en', { type: 'region' });  // fallback:'code'
console.log(loose.of('ZZ'));          // 'ZZ'  (echoes the code back)
United States undefined ZZ
Common Mistakes:
❌ Hand-rolled tables and wrong argument order: const NAMES = { US: 'United States', JP: 'Japan' /* ...forever */ } new Intl.DisplayNames({ type:'region' }) // missing locales arg → TypeError new Intl.DisplayNames('en', { type:'region' }).of('usa') // wrong code ('US', not 'usa')
✅ Let the runtime's CLDR data do it: const dn = new Intl.DisplayNames(userLocale, { type: 'region' }) dn.of('US') // 'United States' (or localized) new Intl.DisplayNames('en',{type:'region',fallback:'none'}).of(x) ?? 'Unknown'
Key Rules:
  • Always pass both arguments: a locale (the language the names appear in) and { type }; the constructor throws without a valid type.
  • .of(code) expects canonical codes — ISO 3166 regions ('US'), BCP 47 languages ('zh-Hant'), ISO 4217 currencies ('EUR'), ISO 15924 scripts ('Latn').
  • Use fallback: 'none' to get undefined for unknown codes (validation); the default 'code' echoes the input back.
Practice Exercises:
Build a describeLocale(tag, displayLocale) that parses tag with Intl.Locale, then uses Intl.DisplayNames to print the language, region, and script as human-readable names in displayLocale. Test with 'zh-Hant-TW' shown in both 'en' and 'fr'.

Connections: Intl.DisplayNames is the natural partner to yesterday's Intl.Locale — parse a tag into subtags with Locale, then render each subtag as a readable name with DisplayNames. It draws on the same CLDR data as Intl.NumberFormat's currency display and the rest of the Intl family.
