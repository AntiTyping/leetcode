DOM Manipulation & DocumentFragment Browser APIs & DOM 6 days since review
Every insertion into the live DOM can trigger layout/reflow, so appending N nodes one at a time is O(N) reflows. A DocumentFragment is a lightweight, off-screen container: build your subtree inside it, then insert it once — its children move in and the fragment empties. Prefer modern insertion methods (append, prepend, before, after, replaceChildren) which accept multiple nodes and strings.
JavaScript
const frag = document.createDocumentFragment();
for (const item of items) {
  const li = document.createElement('li');
  li.textContent = item.name;
  frag.append(li);                 // no reflow — frag is off-document
}
list.append(frag);                 // ONE insertion, one reflow; frag now empty

// replaceChildren clears + sets in a single call:
list.replaceChildren(...newNodes); // no innerHTML='' then loop
Challenge:
Render a 1,000-row table two ways — appending each <tr> to the live <tbody> in a loop, versus building a DocumentFragment and appending once — and measure both with performance.now(). Explain why the fragment version triggers far fewer reflows, and when batching wouldn't help.
#1 Pitfall: Building HTML by string concatenation and assigning innerHTML with untrusted data — that is an XSS vector; prefer textContent / createElement. Also, a node is moved, not copied, when appended elsewhere (use cloneNode(true) to duplicate), and a DocumentFragment is emptied once inserted.
IntersectionObserver Browser APIs & DOM 6 days since review
IntersectionObserver asynchronously reports when a target element enters or leaves a root's viewport, without scroll-handler thrashing. You give it a callback and options (root, rootMargin, threshold); it batches entries off the main thread. Classic uses: lazy-loading images, infinite scroll, and firing analytics when content becomes visible.
JavaScript
const io = new IntersectionObserver((entries) => {
  for (const e of entries) {
    if (e.isIntersecting) {
      e.target.src = e.target.dataset.src;   // lazy-load
      io.unobserve(e.target);                // one-shot: stop watching
    }
  }
}, { rootMargin: '200px', threshold: 0 });   // preload 200px early

document.querySelectorAll('img[data-src]').forEach((img) => io.observe(img));
Challenge:
Build an infinite-scroll loader: observe a sentinel <div> at the bottom of a list, and when it intersects, fetch the next page and insert rows (reusing a DocumentFragment). Ensure you don't fire duplicate fetches while one is in flight, and disconnect the observer when there are no more pages.
#1 Pitfall: Expecting synchronous, pixel-exact positions — the callback is asynchronous and coalesced, so it is not for animations tied to precise scroll offsets. Also forgetting to unobserve/disconnect leaks observers, and threshold: 1 may never fire for a target taller than the root.
ResizeObserver & PerformanceObserver Browser APIs & DOM 6 days since review
ResizeObserver fires when an element's box changes size (not just the window), enabling true container-based responsiveness; read entry.contentBoxSize / contentRect. PerformanceObserver subscribes to performance-timeline entries (paint, largest-contentful-paint, longtask, navigation) as they are recorded — the modern replacement for polling performance.getEntries().
JavaScript
const ro = new ResizeObserver((entries) => {
  for (const e of entries) {
    const { inlineSize } = e.contentBoxSize[0];
    e.target.classList.toggle('narrow', inlineSize < 400);   // container query in JS
  }
});
ro.observe(document.querySelector('.card'));

const po = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) console.log(entry.name, entry.startTime);
});
po.observe({ type: 'largest-contentful-paint', buffered: true });
Challenge:
Use a PerformanceObserver with type: 'longtask' to log any task blocking the main thread over 50ms, and a ResizeObserver to switch a chart between compact and full layouts based on its own width. Explain why buffered: true matters for entries that fired before your observer was attached.
#1 Pitfall: The "ResizeObserver loop limit exceeded" warning — resizing an observed element inside its own callback creates a feedback loop; guard against it or defer the mutation. Also, reading layout in the callback then writing can cause layout thrashing.
Fetch Advanced: Interceptors, Streaming & Cloning Browser APIs & DOM 6 days since review
fetch resolves as soon as headers arrive — the body is a stream you drain with .json()/.text()/.body. A Response body can be read once; use res.clone() to read it twice (e.g. cache + parse). There is no built-in interceptor, so you wrap fetch yourself. fetch only rejects on network failure — a 404/500 still resolves, so check res.ok.
JavaScript
// interceptor pattern: wrap fetch to inject auth + check status
async function apiFetch(url, opts = {}) {
  const res = await fetch(url, {
    ...opts,
    headers: { ...opts.headers, Authorization: `Bearer ${token}` },
  });
  if (!res.ok) throw new Error(`HTTP ${res.status}`);   // fetch won't do this
  return res;
}

// read a body twice via clone (cache the raw text, also parse it):
const res = await apiFetch('/data.json');
cache.set(url, await res.clone().text());
const data = await res.json();
Challenge:
Write fetchWithRetry(url, { retries, signal }) that retries on network errors and 5xx responses with exponential backoff, honors an AbortSignal to cancel mid-flight, and does not retry on 4xx. Then stream a large response with res.body.getReader(), reporting progress from the Content-Length header.
#1 Pitfall: Assuming a rejected promise on HTTP errors — fetch resolves for 4xx/5xx; only network/CORS failures reject. Also, reading a body twice without clone() throws "body stream already read," and forgetting credentials: 'include' drops cookies on cross-origin requests.
Web Storage & IndexedDB Web Storage & Persistence 6 days since review
localStorage/sessionStorage are synchronous, string-only key/value stores (~5MB) — fine for small settings, but blocking and easy to corrupt with un-parsed JSON. IndexedDB is an asynchronous, transactional object database that stores structured-cloneable values (objects, Blobs, ArrayBuffers), supports indexes and cursors, and scales to large data. Reach for IndexedDB when data is big, binary, or queried.
JavaScript
// localStorage: strings only — serialize yourself
localStorage.setItem('prefs', JSON.stringify({ theme: 'dark' }));
const prefs = JSON.parse(localStorage.getItem('prefs') ?? '{}');

// IndexedDB: async, transactional (promise-wrapped)
const db = await new Promise((resolve, reject) => {
  const req = indexedDB.open('app', 1);
  req.onupgradeneeded = () => req.result.createObjectStore('todos', { keyPath: 'id' });
  req.onsuccess = () => resolve(req.result);
  req.onerror = () => reject(req.error);
});
const tx = db.transaction('todos', 'readwrite');
tx.objectStore('todos').put({ id: 1, text: 'ship it' });   // stores an object, no JSON
Challenge:
Wrap the callback-based IndexedDB API in a small promise helper (idbGet/idbPut) around a single object store, then use the storage event on window to sync a localStorage-backed setting across two open tabs. Explain why IndexedDB writes belong in a readwrite transaction and what happens if you reuse a transaction after it auto-commits.
#1 Pitfall: Storing objects directly in localStorage — setItem('x', obj) stores the string "[object Object]"; you must JSON.stringify. Also, Web Storage is synchronous (janky for large data) and per-origin; and an IndexedDB transaction auto-commits when the microtask queue drains, so awaiting unrelated promises mid-transaction can close it.
Web Components Browser APIs & DOM 6 days since review
Three native primitives: Custom Elements (customElements.define a class extending HTMLElement with lifecycle callbacks), Shadow DOM (an encapsulated subtree with scoped styles via attachShadow), and <template> (inert, cloneable markup). Lifecycle: constructor → connectedCallback → attributeChangedCallback (for names in observedAttributes) → disconnectedCallback.
JavaScript
class StarRating extends HTMLElement {
  static observedAttributes = ['value'];
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });   // encapsulated, scoped styles
  }
  connectedCallback() { this.render(); }    // in the DOM — safe to read attrs
  attributeChangedCallback(name, oldV, newV) { this.render(); }
  render() {
    this.shadowRoot.innerHTML = `<style>:host{color:gold}</style>
      ${'★'.repeat(Number(this.getAttribute('value')) || 0)}`;
  }
}
customElements.define('star-rating', StarRating);   // name MUST contain a hyphen
Challenge:
Build a <toggle-switch> custom element with Shadow DOM: reflect a boolean checked attribute, dispatch a bubbling change CustomEvent on toggle, and expose a styleable part via ::part(). Explain why you should set up event listeners in connectedCallback and remove them in disconnectedCallback.
#1 Pitfall: Doing DOM work in the constructor — the element may not be connected and attributes may not be set yet; defer to connectedCallback. Also, custom-element names must contain a hyphen, and forgetting to list an attribute in observedAttributes means attributeChangedCallback never fires for it.

🆕 Errors & Async Combinators

Three features that make failures traceable and concurrency correct — error chaining, the four Promise combinators, and precise microtask scheduling.

Error Cause & AggregateError Core Language

Re-throwing an error with a fresh message used to destroy the original stack and context — you either lost the root cause or leaked low-level details up the stack. ES2022 added the { cause } option so you can wrap an error while preserving the one beneath it, and AggregateError to carry many errors at once (as Promise.any does).

1Chain errors with { cause }. The second argument to the Error constructor attaches the underlying error, readable via err.cause — devtools print the whole chain.

JavaScript
async function loadUser(id) {
  try {
    return await db.query('SELECT ...', id);
  } catch (dbErr) {
    // wrap with a domain-level message, KEEP the original:
    throw new Error(`Failed to load user ${id}`, { cause: dbErr });
  }
}

try {
  await loadUser(7);
} catch (e) {
  console.log(e.message);        // Failed to load user 7
  console.log(e.cause.message);  // e.g. connection timeout — root cause preserved
}
Failed to load user 7 connection timeout

2AggregateError bundles multiple failures. It holds an .errors array — used when several operations can fail independently and you want to report them all rather than the first.

JavaScript
const errors = [];
for (const rule of validators) {
  try { rule(input); } catch (e) { errors.push(e); }
}
if (errors.length) {
  throw new AggregateError(errors, `${errors.length} validation error(s)`);
}
// caller:
catch (e) {
  if (e instanceof AggregateError) e.errors.forEach((x) => console.log(x.message));
}
2 validation error(s) -> each message listed

3Walk the chain to find the root. Because cause can itself have a cause, you can follow it to the deepest error — handy for logging or matching on a low-level error type.

JavaScript
function rootCause(err) {
  while (err.cause instanceof Error) err = err.cause;
  return err;
}

const wrapped = new Error('request failed', {
  cause: new Error('socket closed', { cause: new TypeError('ECONNRESET') }),
});
console.log(rootCause(wrapped).constructor.name);   // TypeError
console.log(rootCause(wrapped).message);            // ECONNRESET
TypeError ECONNRESET
Common Mistakes:
❌ Swallowing the original error when re-throwing: catch (e) { throw new Error('load failed'); } // root cause LOST throw new Error('x', dbErr) // wrong: 2nd arg must be { cause }
✅ Pass the cause in the options object: catch (e) { throw new Error('load failed', { cause: e }); } // e.cause now points at the original error
Key Rules:
  • new Error(msg, { cause }) stores the original error on err.cause; the second argument is an options object, not the cause directly.
  • AggregateError(iterableOfErrors, msg?) exposes all failures via .errors; Promise.any rejects with one when every input rejects.
  • cause can be any value and can nest arbitrarily deep — walk err.cause to reach the root, and match with instanceof as needed.
Practice Exercises:
Write a parseConfig(text) that catches a low-level JSON.parse error and re-throws new Error('invalid config', { cause }). In the caller, print both the friendly message and e.cause.message, proving the original is preserved.
Implement validateAll(input, rules) that runs every rule, collects thrown errors, and throws a single AggregateError if any failed — then have the caller list every message. Show it reports all problems, not just the first.
Write rootCause(err) that follows the cause chain to the deepest Error, and use it to detect whether the ultimate cause is a TypeError anywhere three levels down.
Connections: AggregateError is exactly what Promise.any rejects with when all inputs fail, so these two concepts pair directly. Error chaining also complements the error-handling patterns and AbortController cancellation you have studied.
Promise Combinators: all, race, allSettled, any Async Mastery

Awaiting promises one-by-one serializes work that could run in parallel. The four static combinators express common concurrency shapes declaratively — but each has different success and failure semantics, and picking the wrong one is a frequent source of silent bugs (e.g. one failure sinking a whole batch you wanted to keep).

1all is all-or-nothing; allSettled never rejects. all resolves with an array once every promise fulfills, but rejects on the first rejection. allSettled waits for all and reports each as {status, value} or {status, reason}.

JavaScript
const urls = ['/a', '/b', '/bad'];

// all: rejects as soon as ANY fails (you lose the successful ones)
try { await Promise.all(urls.map(fetchJson)); }
catch (e) { console.log('all failed:', e.message); }

// allSettled: always resolves; inspect each outcome
const results = await Promise.allSettled(urls.map(fetchJson));
const ok = results.filter((r) => r.status === 'fulfilled').map((r) => r.value);
console.log('succeeded:', ok.length);
all failed: HTTP 404 succeeded: 2

2race settles on the first settled; any on the first fulfilled. race adopts whichever promise settles first — fulfilled or rejected (great for timeouts). any ignores rejections and resolves with the first success, rejecting only if all fail.

JavaScript
// race: first to settle wins — a timeout guard
const timeout = (ms) => new Promise((_, rej) => setTimeout(() => rej(new Error('timeout')), ms));
const data = await Promise.race([fetchJson('/slow'), timeout(3000)]);

// any: first SUCCESS wins; fastest working mirror
try {
  const fastest = await Promise.any([fetchJson('/mirror1'), fetchJson('/mirror2')]);
} catch (e) {
  // only here if BOTH rejected -> e is an AggregateError
  console.log(e instanceof AggregateError, e.errors.length);   // true 2
}
true 2

3They all start work immediately. Passing an array of already-started promises runs them concurrently; the combinator only decides how to wait. Map to promises first, then combine — don't await inside the .map.

JavaScript
// ✅ concurrent: all fetches in flight before we await
const users = await Promise.all(ids.map((id) => fetchUser(id)));

// ❌ accidental serialization — awaiting each before starting the next:
const slow = [];
for (const id of ids) slow.push(await fetchUser(id));   // one at a time!
concurrent version finishes in ~max(latency); serial in ~sum(latency)
Common Mistakes:
❌ Using Promise.all when partial success is acceptable: const all = await Promise.all(tasks); // ONE failure discards every result // or unhandled rejections from the "losing" promises in race/any
✅ Match the combinator to the semantics you want: Promise.allSettled(tasks) // keep every outcome, success or failure Promise.any(mirrors) // first success; AggregateError only if all fail
Key Rules:
  • all: fulfilled array, or reject on first rejection. allSettled: always fulfills with per-item {status,...}.
  • race: first to settle (fulfill or reject) wins. any: first to fulfill wins; rejects with an AggregateError only if all reject.
  • All accept any iterable of values (non-promises are wrapped) and start pending work eagerly — build the promise array first, then pass it in.
Practice Exercises:
Fetch a list of URLs with Promise.allSettled and return { succeeded, failed } arrays, so one bad URL never sinks the batch. Contrast the behavior with Promise.all on the same input.
Implement withTimeout(promise, ms) using Promise.race against a rejecting timer, and verify a slow operation rejects with your timeout error while a fast one resolves normally.
Given three redundant API mirrors, use Promise.any to return the first successful response, and when you force all three to fail, inspect the resulting AggregateError.errors to confirm every failure is captured.
Connections: Promise.any rejects with the AggregateError above, tying these two topics together. Combine race with an AbortController for cancelable timeouts, and remember the concurrency-vs-serialization lesson from async/await patterns.
queueMicrotask & Microtask Scheduling Async Mastery

Sometimes you need code to run after the current synchronous block but before the browser renders or handles the next timer — for consistent async timing without the ~4ms clamp and macrotask delay of setTimeout(fn, 0). queueMicrotask(fn) schedules a microtask directly, the same queue that Promise callbacks drain.

1Microtasks run before the next macrotask. The event loop drains the entire microtask queue after each task and before rendering. So a microtask beats any setTimeout, regardless of delay.

JavaScript
console.log('sync start');
setTimeout(() => console.log('timeout (macrotask)'), 0);
queueMicrotask(() => console.log('microtask'));
Promise.resolve().then(() => console.log('promise (also microtask)'));
console.log('sync end');
sync start sync end microtask promise (also microtask) timeout (macrotask)

2Use it to defer work while staying "in the same tick." queueMicrotask is clearer than the Promise.resolve().then() hack and avoids the throttling/clamping of timers — ideal for batching or making a normally-sync callback consistently async.

JavaScript
// coalesce many sync mutations into ONE async flush per tick:
let dirty = false;
function invalidate() {
  if (dirty) return;              // already scheduled this tick
  dirty = true;
  queueMicrotask(() => { dirty = false; render(); });   // runs once, after sync code
}
invalidate(); invalidate(); invalidate();   // render() called exactly once
render() runs a single time, after the current synchronous code

3Danger: an infinitely re-queuing microtask starves the loop. Because the queue is drained to empty before any rendering or I/O, a microtask that schedules another microtask forever will freeze the page. Timers (macrotasks) yield between iterations; microtasks do not.

JavaScript
// ❌ NEVER: the loop can't reach rendering or events — page hangs
function loop() { queueMicrotask(loop); }
// loop();   // do not run — starves the event loop

// ✅ for repeated deferred work that must yield, use a macrotask:
function tick() { doWork(); setTimeout(tick, 0); }   // yields each round
use setTimeout / requestAnimationFrame for work that must let the page breathe
Common Mistakes:
❌ Reaching for setTimeout(fn, 0) to "run later" when you mean "after this tick": setTimeout(flush, 0) // macrotask: clamped, runs after render + other timers // and: recursively queuing microtasks -> event loop starvation
✅ Use queueMicrotask for same-tick deferral; timers for yielding work: queueMicrotask(flush) // before next render / macrotask setTimeout(loopBody, 0) // when you must yield between iterations
Key Rules:
  • The microtask queue is fully drained after every macrotask and before rendering; microtasks always run before the next setTimeout.
  • queueMicrotask(fn) is the explicit, readable equivalent of Promise.resolve().then(fn) — without creating a promise or swallowing errors into a rejection.
  • Never recursively re-queue microtasks unboundedly — it starves rendering and I/O; use setTimeout/requestAnimationFrame for work that must yield.
Practice Exercises:
Predict, then verify, the exact console order of a script mixing synchronous logs, two setTimeout(…,0), one queueMicrotask, and one Promise.then. Explain the rule that produces the ordering.
Build a batch(fn) scheduler that collects synchronous calls and flushes them with a single queueMicrotask per tick (like a reactive framework), and prove fn runs once even after 100 synchronous calls.
Demonstrate microtask starvation safely by counting how many times a re-queuing microtask runs before a setTimeout callback ever fires (spoiler: the timer is starved) — then fix it by switching to a macrotask.
Connections: This is the practical lever on the Event Loop Deep Dive — the microtask queue that Promise callbacks and async/await continuations use. Contrast it with requestAnimationFrame (before paint) and setTimeout (macrotask) for scheduling.

🆕 Ergonomic Built-ins

Two clusters of small, standardized methods that replace clumsy idioms with correct, readable one-liners.

Relative Indexing: at, findLast, findLastIndex Data & Collections

Getting the last element used to mean arr[arr.length - 1] — verbose and impossible to chain. Searching from the end meant reversing first (which mutates or copies). ES2022/ES2023 added at() for negative indexing and findLast/findLastIndex for end-first searches, on both arrays and strings.

1at(i) accepts negative indices. at(-1) is the last element, at(-2) the second-to-last — no length arithmetic. It works on arrays, strings, and typed arrays, returning undefined when out of range.

JavaScript
const arr = [10, 20, 30, 40];
console.log(arr.at(-1));       // 40   (last)
console.log(arr.at(-2));       // 30
console.log(arr.at(0));        // 10   (same as arr[0])
console.log(arr.at(99));       // undefined

console.log('hello'.at(-1));   // 'o'
// chainable, unlike [.length - 1]:
console.log([1, 2, 3].map((x) => x * 2).at(-1));   // 6
40 30 10 undefined o 6

2findLast / findLastIndex search from the end. Same predicate signature as find/findIndex, but they walk right-to-left and return the last match — no reversing required.

JavaScript
const nums = [1, 4, 2, 8, 3];

console.log(nums.findLast((n) => n % 2 === 0));      // 8  (last even)
console.log(nums.findLastIndex((n) => n % 2 === 0)); // 3

// vs the old reverse-and-search dance (copies + flips the index):
const idx = [...nums].reverse().findIndex((n) => n % 2 === 0);
console.log(nums.length - 1 - idx);                 // 3  — same, but clumsy
8 3 3

3Both return sentinel values, not exceptions. findLast returns undefined and findLastIndex returns -1 when nothing matches — mirroring their forward counterparts, so existing guard patterns still apply.

JavaScript
const log = [
  { level: 'info', msg: 'start' },
  { level: 'error', msg: 'boom' },
  { level: 'info', msg: 'retry' },
];

const lastError = log.findLast((e) => e.level === 'error');
console.log(lastError?.msg ?? 'no errors');         // boom

const none = log.findLastIndex((e) => e.level === 'fatal');
console.log(none);                                  // -1
boom -1
Common Mistakes:
❌ length arithmetic and destructive reversal: const last = arr[arr.length - 1] // breaks on chains; off-by-one prone arr.reverse().find(pred) // MUTATES arr, and flips the index meaning
✅ Use the built-ins — non-mutating and chainable: arr.at(-1) // last element, any expression arr.findLast(pred) // last match, no reverse/copy
Key Rules:
  • at(i) supports negative indices (from the end) on arrays, strings, and typed arrays; out-of-range returns undefined.
  • findLast/findLastIndex iterate from the end and return the last match — undefined / -1 when none, just like find/findIndex.
  • None of these mutate the receiver, and all are chain-friendly — unlike reverse(), which mutates and reindexes.
Practice Exercises:
Write lastWhere(arr, pred) that returns the last matching element using findLast, and compare its output to a hand-written reverse-loop version on the same data to confirm they agree.
Implement tail(arr, n) returning the last n elements, then use at(-1) to grab the final one — all without touching arr.length directly. Handle n larger than the array.
Given a list of log entries, find the message of the most recent error with findLast, and its position with findLastIndex, safely handling the "no error" case with ?. and a -1 check.
Connections: at() pairs with the copying array methods (toSorted, with) as part of the modern non-mutating array toolkit, and its negative-index convention matches with(-1, …). findLast complements the forward find/findIndex you already know.
Object Utilities: Object.hasOwn, fromEntries, entries Data & Collections

Two everyday object chores had sharp edges: checking for an own property safely (obj.hasOwnProperty breaks on null-prototype objects and can be shadowed), and transforming objects (no map/filter for plain objects). ES2022's Object.hasOwn and the entries↔fromEntries round-trip fix both cleanly.

1Object.hasOwn(obj, key) is the safe own-property check. It works even when obj has no prototype or has shadowed hasOwnProperty, and it ignores inherited properties — the recommended replacement for obj.hasOwnProperty(key).

JavaScript
const obj = { a: 1, b: undefined };

console.log(Object.hasOwn(obj, 'a'));         // true
console.log(Object.hasOwn(obj, 'b'));         // true  ('in'-like, even if undefined)
console.log(Object.hasOwn(obj, 'toString'));  // false (inherited, not own)

// safe on a null-prototype object, where .hasOwnProperty doesn't exist:
const bag = Object.create(null);
bag.x = 1;
// bag.hasOwnProperty('x')      // TypeError: not a function
console.log(Object.hasOwn(bag, 'x'));         // true
true true false true

2entries → transform → fromEntries gives objects a map/filter. Convert to [key, value] pairs, use array methods, then rebuild the object — the idiomatic way to remap or prune keys.

JavaScript
const prices = { apple: 1, banana: 2, cherry: 3 };

// map values (apply a discount):
const discounted = Object.fromEntries(
  Object.entries(prices).map(([k, v]) => [k, v * 0.9])
);
console.log(discounted);        // { apple: 0.9, banana: 1.8, cherry: 2.7 }

// filter keys (drop the cheap ones):
const pricey = Object.fromEntries(
  Object.entries(prices).filter(([, v]) => v >= 2)
);
console.log(pricey);            // { banana: 2, cherry: 3 }
{ apple: 0.9, banana: 1.8, cherry: 2.7 } { banana: 2, cherry: 3 }

3fromEntries also converts Maps and query strings. It accepts any iterable of pairs — a Map, or URLSearchParams — making it the bridge from key/value structures back to plain objects.

JavaScript
const m = new Map([['id', 42], ['role', 'admin']]);
console.log(Object.fromEntries(m));            // { id: 42, role: 'admin' }

// parse a query string into an object:
const params = new URLSearchParams('page=2&sort=asc');
console.log(Object.fromEntries(params));       // { page: '2', sort: 'asc' }
{ id: 42, role: 'admin' } { page: '2', sort: 'asc' }
Common Mistakes:
❌ Calling hasOwnProperty off the instance, or the 'in' trap for own keys: obj.hasOwnProperty('k') // breaks if obj is null-proto or shadows the method 'toString' in obj // true! 'in' includes inherited props
✅ Use Object.hasOwn for a robust own-property test: Object.hasOwn(obj, 'k') // own-only, works on any object shape
Key Rules:
  • Object.hasOwn(obj, key) tests own (non-inherited) properties safely, even on null-prototype objects — prefer it over obj.hasOwnProperty.
  • Object.fromEntries(iterable) builds an object from any iterable of [key, value] pairs (arrays, Map, URLSearchParams); it is the inverse of Object.entries.
  • The entries → map/filter → fromEntries round-trip only sees string-keyed, enumerable own properties — symbol keys and non-enumerables are dropped.
Practice Exercises:
Write pick(obj, keys) and omit(obj, keys) using Object.entries/fromEntries and Object.hasOwn, and confirm they never pull in inherited properties.
Implement mapValues(obj, fn) that returns a new object with each value transformed by fn (keys untouched), proving the original object is not mutated.
Parse a query string with URLSearchParams into a plain object via Object.fromEntries, then handle the repeated-key case (e.g. tag=a&tag=b) by collecting duplicates into an array — explain why the naive fromEntries keeps only the last value.
Connections: Object.fromEntries is the inverse of Object.entries and the bridge between Map-style structures and plain objects. Object.hasOwn is the robust cousin of the in operator and the brand-check idea you have seen with private fields.
