Typed Arrays & ArrayBuffer Data & Collections 10 days since review
An ArrayBuffer is a fixed-length block of raw bytes you can't read directly. A typed array view (Uint8Array, Float64Array, ...) interprets those bytes as numbers of a fixed width; a DataView reads/writes at arbitrary offsets with explicit endianness. Multiple views can share one buffer, so writing through one is visible through another.
JavaScript
const buf = new ArrayBuffer(8);        // 8 raw bytes
const u8  = new Uint8Array(buf);       // view as 8 unsigned bytes
const f64 = new Float64Array(buf);     // view the SAME bytes as one double

u8[0] = 255;
console.log(f64[0]);                   // some float — same memory reinterpreted

// DataView: explicit offset + endianness
const dv = new DataView(buf);
dv.setUint32(0, 0x01020304, /*littleEndian*/ false);  // big-endian
console.log(u8[0], u8[1]);             // 1 2

// clamping vs wrapping
const c = new Uint8ClampedArray(1);
c[0] = 300; console.log(c[0]);         // 255 (clamped, not 300 % 256)
Challenge:
Write readRGBA(buffer, pixelIndex) that treats an ArrayBuffer of bytes as a flat RGBA image and returns { r, g, b, a } for the Nth pixel using a Uint8Array view. Then write toFloatLE(bytes) that reads a little-endian 32-bit float from a 4-byte array using a DataView.
#1 Pitfall: Assuming typedArray[i] = x stores x exactly. Out-of-range integers wrap modulo the element width (Uint8Array stores 257 as 1), non-clamped, and floats are truncated to ints for integer arrays. Use Uint8ClampedArray for pixel-style clamping, and never assume host endianness — use DataView when the byte order matters (files, network).
Structured Clone Data & Collections 8 days since review
structuredClone(value) deep-copies using the structured clone algorithm: it handles Map, Set, Date, RegExp, typed arrays, and cyclic references — things JSON.parse(JSON.stringify(x)) silently breaks. It cannot clone functions, DOM nodes, or class prototypes (you get a plain object), and throws DataCloneError on functions/symbols.
JavaScript
const original = {
  when: new Date(),
  tags: new Set(['a', 'b']),
  lookup: new Map([['k', 1]]),
};
original.self = original;              // cycle!

const copy = structuredClone(original);
console.log(copy.self === copy);       // true  (cycle preserved)
console.log(copy.tags instanceof Set); // true  (JSON would have lost this)
console.log(copy.when !== original.when, copy.when.getTime() === original.when.getTime());
                                       // true true (new Date, same instant)

// transfer: move ownership of a buffer (zero-copy) instead of cloning it
const buf = new ArrayBuffer(1024);
const moved = structuredClone(buf, { transfer: [buf] });
console.log(buf.byteLength);           // 0  (detached after transfer)
Challenge:
Write deepFreezeClone(obj) that returns a structuredClone of obj with every nested object recursively Object.freezed. Then demonstrate one value that structuredClone handles but JSON round-tripping destroys, and one value that makes structuredClone throw.
#1 Pitfall: Reaching for structuredClone on objects that contain methods or class instances. It copies own enumerable data only — the result is a plain object with no prototype, so methods and instanceof checks vanish. For those, clone fields manually or rebuild the instance.
JSON Advanced Data & Collections 8 days since review
JSON.stringify takes a replacer (function or allow-list array) and an indent third arg; objects with a toJSON() method serialize via its return value (that's why Date becomes an ISO string). JSON.parse takes a reviver to transform values on the way in. undefined, functions, and symbols are dropped from objects (and become null in arrays); BigInt throws.
JavaScript
const data = { name: 'Ada', secret: 42, born: new Date('1815-12-10') };

// replacer as allow-list + pretty indent
console.log(JSON.stringify(data, ['name', 'born'], 2));
// { "name": "Ada", "born": "1815-12-10T00:00:00.000Z" }

// reviver: revive ISO strings back into Dates
const json = '{"born":"1815-12-10T00:00:00.000Z"}';
const out = JSON.parse(json, (key, val) =>
  key === 'born' ? new Date(val) : val);
console.log(out.born instanceof Date); // true

// toJSON hook controls a type's own serialization
class Money { constructor(c){ this.cents = c; } toJSON(){ return this.cents/100; } }
console.log(JSON.stringify({ price: new Money(1999) })); // {"price":19.99}
Challenge:
Write stringifySafe(obj) that serializes an object containing BigInt values (encode each as { "$bigint": "123" }) and a parseSafe(json) reviver that restores them. Then write a replacer that redacts any key named password or token at any depth, replacing its value with "[redacted]".
#1 Pitfall: Forgetting that the reviver/replacer is called bottom-up / top-down respectively and that returning undefined from a replacer removes the key. Also: the replacer's this is the holder object and the value passed may already be post-toJSON (a Date arrives as a string), which trips people doing instanceof Date checks inside the replacer.
Error Handling Patterns Modules & Patterns 8 days since review
Throw Error instances, never strings (you lose the stack). Subclass Error for typed domain errors and set this.name. Use the { cause } option to wrap a low-level error while preserving the original. AggregateError bundles many (it's what Promise.any rejects with). A finally block runs on both success and throw — ideal for cleanup.
JavaScript
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = 'ValidationError';     // so error.name / logs are meaningful
    this.field = field;
  }
}

async function loadUser(id) {
  try {
    return await db.get(id);
  } catch (err) {
    // wrap: keep the original via cause, add context
    throw new Error(`loadUser(${id}) failed`, { cause: err });
  }
}

try {
  throw new ValidationError('email required', 'email');
} catch (e) {
  if (e instanceof ValidationError) console.log(e.field, e.name); // email ValidationError
  console.log(e.cause);                // undefined here; set when wrapping
}
Challenge:
Build a retry(fn, { tries }) that retries an async fn, and on final failure throws an AggregateError containing every attempt's error. Then write a custom HttpError extends Error with a status field and a type-guard isHttpError(x), and show wrapping a parse failure with { cause }.
#1 Pitfall: Swallowing errors with an empty catch {}, or re-throwing as throw new Error(e.message) — which discards the original stack and type. Preserve context with throw new Error('...', { cause: e }) and only catch errors you can actually handle; let the rest propagate.
Node.js fs/promises & path Node.js Runtime 8 days since review
Import the promise API as import { readFile, writeFile } from 'node:fs/promises' to await file I/O instead of callbacks. Build paths with path.join/path.resolve (cross-platform separators) rather than string concatenation, and prefer fs.readFile(url, 'utf8') or a URL from import.meta.url over relative cwd-dependent paths.
JavaScript
import { readFile, writeFile, mkdir, readdir } from 'node:fs/promises';
import path from 'node:path';
import { fileURLToPath } from 'node:url';

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const dataDir = path.join(__dirname, 'data');

await mkdir(dataDir, { recursive: true });             // no error if it exists
await writeFile(path.join(dataDir, 'out.json'),
  JSON.stringify({ ok: true }), 'utf8');

const text = await readFile(path.join(dataDir, 'out.json'), 'utf8');
console.log(JSON.parse(text).ok);                      // true

const entries = await readdir(dataDir, { withFileTypes: true });
console.log(entries.filter(e => e.isFile()).map(e => e.name)); // [ 'out.json' ]
Challenge:
Write readJson(relPath) that resolves a path relative to the current module (via import.meta.url), reads and parses it, and on a missing file (err.code === 'ENOENT') returns null instead of throwing. Then write walk(dir) that recursively yields every file path under a directory using readdir(..., { withFileTypes: true }).
#1 Pitfall: Concatenating paths with '/' (breaks on Windows) or building paths from process.cwd() (depends on where the process was launched, not where the file lives). Use path.join/path.resolve and anchor module-relative paths to import.meta.url. Also check err.code (e.g. 'ENOENT') rather than parsing error messages.
Testing with Node Test Runner Testing & Quality 8 days since review
Node ships a built-in runner: import test and describe/it from node:test and assertions from node:assert/strict. Run with node --test (auto-discovers *.test.js). Tests can be async (return a promise or use the callback), nest with t.test, and use mock for spies/stubs — no external dependency required.
JavaScript
import { test, describe, it, mock, before } from 'node:test';
import assert from 'node:assert/strict';

describe('math', () => {
  it('adds', () => {
    assert.equal(2 + 2, 4);
  });

  it('awaits async work', async () => {
    const v = await Promise.resolve(42);
    assert.equal(v, 42);
  });

  it('spies with mock.fn', () => {
    const fn = mock.fn((x) => x * 2);
    fn(3);
    assert.equal(fn.mock.calls.length, 1);
    assert.equal(fn.mock.calls[0].result, 6);
  });
});
// run:  node --test          (discovers *.test.js, exits non-zero on failure)
Challenge:
Write a test file for a parseQuery(str) function with at least one async test, one that asserts a thrown error via assert.throws, and one using mock.fn to verify a callback is invoked exactly once with the right argument. Add a before hook that sets up shared fixture data.
#1 Pitfall: Forgetting to await (or return) async work inside a test — the test passes before assertions run. Either mark the test async and await everything, or return the promise. Also: importing assertions from node:assert (loose) instead of node:assert/strict means assert.equal uses == and hides type bugs.

🆕 Modern Collections & Arrays

ES2023–2025 additions that replace hand-rolled set algebra and mutation-safe copying — fewer helpers, fewer bugs.

Set Methods (union, intersection, difference) Data & Collections

For years Set had no way to combine two sets — you converted to arrays, filtered, and rebuilt, which was verbose and O(n·m). ES2025 adds seven composable set-algebra methods directly on Set.prototype, each returning a brand-new Set (or a boolean) without mutating either operand.

1The combining methods. union, intersection, difference, and symmetricDifference each take another set-like and return a new Set. The receiver and argument are never modified.

JavaScript
const a = new Set([1, 2, 3, 4]);
const b = new Set([3, 4, 5, 6]);

console.log([...a.union(b)]);               // [1, 2, 3, 4, 5, 6]
console.log([...a.intersection(b)]);        // [3, 4]
console.log([...a.difference(b)]);          // [1, 2]      (in a, not in b)
console.log([...a.symmetricDifference(b)]); // [1, 2, 5, 6] (in exactly one)

console.log([...a]);                        // [1, 2, 3, 4]  — a is unchanged
[ 1, 2, 3, 4, 5, 6 ] [ 3, 4 ] [ 1, 2 ] [ 1, 2, 5, 6 ] [ 1, 2, 3, 4 ]

2The predicate methods. isSubsetOf, isSupersetOf, and isDisjointFrom return booleans — no allocation, useful for permission/tag checks.

JavaScript
const granted = new Set(['read', 'write', 'delete']);
const needed  = new Set(['read', 'write']);

console.log(needed.isSubsetOf(granted));    // true  — user has all needed perms
console.log(granted.isSupersetOf(needed));  // true
console.log(needed.isDisjointFrom(new Set(['admin']))); // true (no overlap)
true true true

3"Set-like", not just Set. The argument only needs a numeric size, a has method, and a keys method — so a Map works as an argument (its keys are used). The receiver, however, must be a real Set.

JavaScript
const tags = new Set(['js', 'ts', 'go']);
const counts = new Map([['ts', 5], ['rust', 2]]);   // Map is "set-like" via its keys

console.log([...tags.intersection(counts)]);        // [ 'ts' ]
console.log([...tags.difference(counts)]);          // [ 'js', 'go' ]
[ 'ts' ] [ 'js', 'go' ]
Common Mistakes:
❌ Expecting these to mutate in place, or passing a plain array: a.union(b); // returns a NEW set — the result is discarded here a.union([3, 4]); // TypeError — an array has no `size`/`has`/`keys` as a set-like
✅ Capture the returned set; wrap arrays in a Set first: const merged = a.union(new Set([3, 4])); // every method returns a fresh Set (or boolean); originals are untouched
Key Rules:
  • All seven methods are non-mutating: combiners return a new Set, predicates return a boolean.
  • The argument must be set-like (size + has + keys) — a Set or Map qualifies, a raw array does not. The receiver must be an actual Set.
  • difference is directional (a.difference(b) !== b.difference(a)); symmetricDifference is not. Result order follows the receiver, then the argument.
Practice Exercises:
Given required and userRoles sets, write missingRoles(required, userRoles) returning the set of roles the user lacks, and hasAllRoles(...) using isSubsetOf. Verify the originals are unmodified afterward.
Implement Jaccard similarity jaccard(a, b) = |a ∩ b| / |a ∪ b| for two sets using only the new methods (no array round-trips), and test it on two overlapping word sets.
Write diffChange(before, after) that returns { added, removed, kept } as three sets describing how a set of selected ids changed between two snapshots, using difference and intersection.
Connections: These build on the Map & Set reference-equality model and pair well with non-mutating array methods below — both favor returning fresh collections over in-place edits. Convert results to arrays with spread to feed them into array methods or iterator helpers.
Change Array by Copy: toSorted, toReversed, with Data & Collections

sort, reverse, and splice mutate the array in place — a constant source of bugs in React/Redux and anywhere shared state must stay immutable. ES2023 adds copying counterparts (toSorted, toReversed, toSpliced, with) that return a new array and leave the original untouched.

1Sort and reverse without mutating. toSorted(cmp) and toReversed() mirror their in-place versions but return a fresh array.

JavaScript
const nums = [3, 1, 2];

const sorted = nums.toSorted((a, b) => a - b);
console.log(sorted);   // [1, 2, 3]
console.log(nums);     // [3, 1, 2]  — original untouched

const rev = nums.toReversed();
console.log(rev);      // [2, 1, 3]
console.log(nums);     // [3, 1, 2]  — still intact
[ 1, 2, 3 ] [ 3, 1, 2 ] [ 2, 1, 3 ] [ 3, 1, 2 ]

2with(index, value) — immutable single-element update. Replaces one element by index and returns the new array. Negative indices count from the end. Replaces the clumsy [...arr.slice(0,i), v, ...arr.slice(i+1)] idiom.

JavaScript
const row = ['a', 'b', 'c', 'd'];

const updated = row.with(1, 'B');
console.log(updated);  // ['a', 'B', 'c', 'd']
console.log(row);      // ['a', 'b', 'c', 'd']

console.log(row.with(-1, 'Z'));  // ['a', 'b', 'c', 'Z']  (negative index)
// row.with(10, 'x')  -> RangeError: index out of range
[ 'a', 'B', 'c', 'd' ] [ 'a', 'b', 'c', 'd' ] [ 'a', 'b', 'c', 'Z' ]

3toSpliced(start, deleteCount, ...items). The non-mutating splice: returns the resulting array (not the removed elements). Great for inserting/removing immutably.

JavaScript
const list = [1, 2, 3, 4];

const inserted = list.toSpliced(2, 0, 'x', 'y');  // insert at index 2, delete 0
console.log(inserted);  // [1, 2, 'x', 'y', 3, 4]

const removed = list.toSpliced(1, 2);             // remove 2 starting at index 1
console.log(removed);   // [1, 4]
console.log(list);      // [1, 2, 3, 4]  — original preserved
[ 1, 2, 'x', 'y', 3, 4 ] [ 1, 4 ] [ 1, 2, 3, 4 ]
Common Mistakes:
❌ Mixing mutating and copying versions, or assuming toSpliced returns removed items: const removed = arr.splice(1, 2); // mutates arr; returns the REMOVED items const r = arr.toSpliced(1, 2); // does NOT mutate; returns the RESULTING array
✅ Use the to*/with copies whenever the source must stay immutable: const next = state.items.toSorted(byDate); // safe for React state const patched = state.items.with(i, newItem);
Key Rules:
  • toSorted, toReversed, toSpliced, with all return a new array; the originals sort/reverse/splice mutate. Reach for the copies in any shared/immutable state.
  • with and the others do a shallow copy — nested objects are shared by reference, so don't mutate those.
  • with throws RangeError on an out-of-bounds index (unlike assignment, which would create a sparse hole). They're also available on typed arrays.
Practice Exercises:
Given an immutable todos array of { id, done }, write toggle(todos, id) that returns a new array with the matching todo's done flipped using findIndex + with (do not mutate the input or its objects).
Implement moveItem(arr, from, to) that returns a new array with the element moved from one index to another, using toSpliced twice and never mutating arr.
Sort a list of records by a key without disturbing the original, then confirm via a deep check that the source array's element order and identities are unchanged.
Connections: Same immutability philosophy as the non-mutating Set methods above, and the natural pairing for Object.freeze/records-and-tuples style state. The returned arrays feed straight into relative indexing and iterator helpers.
Relative Indexing .at() & findLast Data & Collections

Getting the last element meant arr[arr.length - 1] (repeated, error-prone) and searching from the end meant reversing first. ES2022 added .at() for negative indexing and findLast/findLastIndex for end-first search — small, but they remove a class of off-by-one bugs.

1.at(index) with negative indices. Available on arrays, strings, and typed arrays. -1 is the last element; out of range returns undefined (not a throw).

JavaScript
const arr = [10, 20, 30, 40];
console.log(arr.at(-1));    // 40   (last)
console.log(arr.at(-2));    // 30
console.log(arr.at(0));     // 10
console.log(arr.at(99));    // undefined

console.log('hello'.at(-1)); // 'o'  (strings too)
40 30 10 undefined o

2findLast & findLastIndex. Like find/findIndex but scan from the end — return the last matching element/index, or undefined/-1.

JavaScript
const events = [
  { type: 'login',  ok: true },
  { type: 'fetch',  ok: false },
  { type: 'fetch',  ok: true },
  { type: 'logout', ok: false },
];

const lastFailure = events.findLast(e => !e.ok);
console.log(lastFailure.type);                 // 'logout'

const lastOkIndex = events.findLastIndex(e => e.ok);
console.log(lastOkIndex);                      // 2

console.log([1, 2, 3].findLast(n => n > 5));   // undefined
logout 2 undefined

3Why .at() beats bracket-with-math. It reads clearly, never produces an accidental string key, and chains nicely off methods that return arrays (so you don't store an intermediate just to index it).

JavaScript
const csv = 'a,b,c,d';
console.log(csv.split(',').at(-1));            // 'd'  — no temp variable

// bracket math fails silently on a wrong sign; .at(-1) is explicit:
const empty = [];
console.log(empty[empty.length - 1]);          // undefined (length-1 === -1 → arr[-1])
console.log(empty.at(-1));                      // undefined (clearly "last of empty")
d undefined undefined
Common Mistakes:
❌ Using bracket access with a negative index expecting wraparound: arr[-1]; // undefined — reads a string property "-1", NOT the last element arr[arr.length-1] // works, but repeats length and breaks on empty arrays cleanly?
✅ Use .at(-1) for end-relative access, findLast for end-first search: arr.at(-1); arr.findLast(pred); // instead of [...arr].reverse().find(pred)
Key Rules:
  • .at() accepts negative indices (counting from the end) and returns undefined out of range; bracket arr[-1] does not — it reads a named property.
  • findLast/findLastIndex iterate right-to-left and short-circuit on the first match from the end — cheaper and clearer than reverse-then-find.
  • All work on strings and typed arrays too; .at() on a string returns a single-character string, not a code unit.
Practice Exercises:
Write last(arr, n = 1) returning the nth-from-last element using .at(), returning undefined when out of range. Then write tail(arr, k) returning the final k elements as a new array.
Given an array of log lines, use findLast to return the most recent line matching a level (e.g. last 'ERROR') and findLastIndex to truncate the array up to and including it.
Implement isPalindrome(str) using .at(i) and .at(-1 - i) comparisons (no reverse, no extra array), handling the empty and single-character cases.
Connections: .at() composes naturally after copying array methods and split/toSorted results. The end-first scan of findLast complements the lazy forward iteration of iterator helpers — pick the direction the data needs.

🆕 Syntax & Concurrency

A terse assignment shorthand that respects short-circuiting, and the low-level primitives behind true multi-threaded shared memory.

Logical Assignment Operators (&&=, ||=, ??=) Core Language

Patterns like obj.x = obj.x || default and if (!cfg.y) cfg.y = z are everywhere. ES2021's &&=, ||=, and ??= compress them — but crucially they short-circuit, so the assignment (and any setter/side effect) only runs when needed.

1The three operators. a ||= b assigns only if a is falsy; a &&= b assigns only if a is truthy; a ??= b assigns only if a is null/undefined.

JavaScript
let count = 0;
count ||= 10;      // 0 is falsy → assigns
console.log(count);// 10

let name = 'Ada';
name ??= 'Anon';   // not nullish → no assignment
console.log(name); // 'Ada'

let token = 'abc';
token &&= token.trim();   // truthy → assigns trimmed value
console.log(token);       // 'abc'

let missing;       // undefined
missing ??= 'fallback';
console.log(missing);     // 'fallback'
10 Ada abc fallback

2Short-circuit means the write is conditional. Unlike a = a ?? b, which always assigns, a ??= b skips the assignment entirely when not needed — so setters don't fire and you don't trigger reactive/proxy writes pointlessly.

JavaScript
const obj = {
  _v: 1,
  get v() { return this._v; },
  set v(x) { console.log('setter ran'); this._v = x; },
};

obj.v ??= 99;   // getter returns 1 (not nullish) → setter is NEVER called
console.log(obj.v);          // 1, no "setter ran" logged

// compare the naive version, which always writes:
// obj.v = obj.v ?? 99;      // would log "setter ran" even though value is kept
1

3Lazy default initialization. Combine ??= with object/array literals to lazily build nested structures — a clean "create the bucket if absent" idiom.

JavaScript
function addToGroup(groups, key, value) {
  (groups[key] ??= []).push(value);   // create the array only the first time
  return groups;
}

const g = {};
addToGroup(g, 'a', 1);
addToGroup(g, 'a', 2);
addToGroup(g, 'b', 3);
console.log(g);   // { a: [1, 2], b: [3] }
{ a: [ 1, 2 ], b: [ 3 ] }
Common Mistakes:
❌ Using ||= for a config that legitimately accepts 0, '', or false: options.retries ||= 3; // a passed-in 0 is falsy → wrongly overwritten with 3
✅ Use ??= when only null/undefined should trigger the default: options.retries ??= 3; // an explicit 0 is kept; only missing → 3
Key Rules:
  • ||= triggers on any falsy value (0, '', false, NaN, null, undefined); ??= triggers only on null/undefined. Prefer ??= for defaults unless you truly want all falsy values replaced.
  • They short-circuit: the right side is evaluated and the assignment happens only when the condition is met — so side effects and setters are skipped otherwise.
  • They're sugar for a || (a = b) etc., so they read/write the target — fine on a plain property, but be mindful with proxies/getters that have side effects.
Practice Exercises:
Write withDefaults(opts) that fills in timeout, retries, and verbose using ??= so that a caller passing { retries: 0, verbose: false } keeps those exact values. Show why ||= would be wrong here.
Build a memoization map where cache[key] ??= expensive(key) computes a value at most once per key; prove expensive runs only once for repeated keys with a call counter.
Use &&= to normalize an optional string field only when it exists (e.g. user.bio &&= user.bio.trim()), leaving null/undefined bios untouched. Verify with both present and absent inputs.
Connections: ??= shares its nullish semantics with the optional chaining & nullish coalescing family — the same "only null/undefined counts as missing" rule. The setter-skipping behavior matters when the target is a Proxy or accessor property.
Atomics & SharedArrayBuffer Concurrency

Normal data passed to a worker is copied (or transferred). A SharedArrayBuffer is genuinely shared memory visible to multiple threads at once — but concurrent reads/writes race. The Atomics object provides race-free, indivisible operations (and a tiny wait/notify lock) over that shared memory.

1Shared memory + atomic ops. Back a typed array with a SharedArrayBuffer, then use Atomics.add/load/store so increments from different threads don't lose updates.

JavaScript
const sab = new SharedArrayBuffer(4);     // 4 bytes = one Int32
const counter = new Int32Array(sab);      // view over shared memory

Atomics.store(counter, 0, 0);             // counter[0] = 0, atomically
Atomics.add(counter, 0, 5);               // += 5, indivisibly
Atomics.add(counter, 0, 3);
console.log(Atomics.load(counter, 0));    // 8  (no lost updates)

// non-atomic counter[0]++ from two threads could lose increments;
// Atomics.add guarantees read-modify-write happens as one step.
8

2Compare-and-exchange: build a spinlock. Atomics.compareExchange(arr, i, expected, replacement) swaps only if the current value equals expected, returning the old value — the primitive for lock-free algorithms.

JavaScript
const lock = new Int32Array(new SharedArrayBuffer(4));  // 0 = free, 1 = held

function acquire() {
  // try to flip 0 → 1; spin until we win the swap
  while (Atomics.compareExchange(lock, 0, 0, 1) !== 0) { /* busy-wait */ }
}
function release() {
  Atomics.store(lock, 0, 0);
}

acquire();
// ... critical section: only one thread holds the lock ...
release();
console.log(Atomics.load(lock, 0));   // 0 (released)
0

3wait / notify — block without busy-spinning. A worker thread can Atomics.wait until another thread Atomics.notifys that index. (wait is not allowed on the main thread — it would freeze the UI; use Atomics.waitAsync there.)

JavaScript
const sab = new SharedArrayBuffer(4);
const flag = new Int32Array(sab);     // shared with a worker

// --- in a worker thread ---
// blocks until flag[0] !== 0, or 1000ms passes:
const res = Atomics.wait(flag, 0, 0, 1000);   // 'ok' | 'not-equal' | 'timed-out'

// --- in the producer thread ---
Atomics.store(flag, 0, 1);
Atomics.notify(flag, 0, 1);          // wake up to 1 waiter on index 0
console.log(res);                    // 'ok' once notified (in the worker)
ok
Common Mistakes:
❌ Mutating shared memory with plain operators across threads: counter[0]++; // read+write is NOT atomic — concurrent updates are lost Atomics.wait(arr, 0, 0); // throws on the MAIN thread (would block the event loop)
✅ Use Atomics for every shared access; wait only off-main-thread: Atomics.add(counter, 0, 1); // main thread: Atomics.waitAsync(arr, 0, 0).value.then(...)
Key Rules:
  • SharedArrayBuffer is shared (not copied) across workers; every concurrent access must go through Atomics to avoid torn reads / lost writes. It requires cross-origin isolation (COOP/COEP headers) in browsers.
  • Atomics works only on integer typed arrays backed by a SharedArrayBuffer (Int32Array, BigInt64Array, etc.) — not on floats or regular buffers.
  • Atomics.wait blocks and is forbidden on the main thread; use it in workers, or Atomics.waitAsync (returns a promise) anywhere.
Practice Exercises:
Set up a main thread + one Worker sharing a SharedArrayBuffer-backed Int32Array. Have the worker increment index 0 ten thousand times with Atomics.add while the main thread does the same, then confirm the total is exactly 20000 (try plain [0]++ to observe lost updates).
Implement a single-slot bounded buffer (producer/consumer) using Atomics.wait/notify: the consumer waits for a "full" flag, reads the value, then signals "empty".
Write a lock-free increment using Atomics.compareExchange in a loop (load current, compute next, CAS, retry on failure) and explain when it would livelock under high contention.
Connections: This is the low-level foundation under Web Workers / worker_threads message-passing — use shared memory + Atomics when copying is too slow. The buffer itself is the same ArrayBuffer machinery from today's review, just shared and access-synchronized.
