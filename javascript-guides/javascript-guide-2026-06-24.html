Performance Optimization Performance & Security 9 days since review
Defer non-urgent work with requestIdleCallback; measure with the User Timing API (performance.mark/measure); avoid recomputation with memoization and lazy evaluation. Measure before optimizing — guesses about hot paths are usually wrong.
JavaScript
performance.mark('start');
const memo = new Map();
function fib(n) {
  if (n < 2) return n;
  if (memo.has(n)) return memo.get(n);
  const v = fib(n - 1) + fib(n - 2);
  memo.set(n, v);
  return v;
}
fib(40);
performance.mark('end');
performance.measure('fib', 'start', 'end');
console.log(performance.getEntriesByName('fib')[0].duration.toFixed(2), 'ms');
Challenge:
Write a generic memoize(fn) that supports multiple arguments (keyed by a serialized arg list), then use performance.measure to prove the second call to an expensive function is at least 100x faster than the first. Bonus: schedule the warm-up call inside requestIdleCallback so it doesn't block first paint.
#1 Pitfall: Memoizing a function with object/array arguments using identity as the key — two structurally-equal objects are different references, so the cache never hits. Serialize the key (e.g. JSON.stringify) or use a WeakMap keyed on the actual object reference.
Security: XSS Prevention, CSP & Trusted Types Performance & Security 9 days since review
Never build HTML by concatenating untrusted input. Insert text with textContent (never innerHTML). Lock down what can execute with a Content-Security-Policy header, and require sanitized values for dangerous sinks with Trusted Types.
JavaScript
// ❌ XSS sink
el.innerHTML = '<p>' + userInput + '</p>';

// ✅ text is escaped automatically
el.textContent = userInput;

// CSP (sent as a response header) blocks inline & eval'd script:
// Content-Security-Policy: default-src 'self'; script-src 'self'

// Trusted Types force sanitization before assignment to innerHTML:
const policy = trustedTypes.createPolicy('app', {
  createHTML: (s) => DOMPurify.sanitize(s)
});
el.innerHTML = policy.createHTML(userInput);
Challenge:
Write escapeHtml(str) that converts & < > " ' to their HTML entities, then build a render(template, data) tag function that auto-escapes every interpolated value (so render`<b>${userInput}</b>` is XSS-safe). Explain why a CSP header would still be a valuable second layer of defense.
#1 Pitfall: Trusting client-side escaping alone. Escaping must happen at the correct sink and context (HTML vs attribute vs URL vs JS), and a single missed sink reopens XSS. Defense in depth — escaping plus a strict CSP — is the standard, not either/or.
WebAssembly Interop Performance & Security 9 days since review
Stream-compile and instantiate a .wasm module, then call exported functions like plain JS functions. WASM and JS share a single WebAssembly.Memory (an ArrayBuffer); pass complex data by writing bytes into that memory and passing offsets.
JavaScript
// Browser: stream + compile + instantiate in one call
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('add.wasm'),
  { env: { log: (x) => console.log('wasm:', x) } } // imports JS gives to WASM
);
console.log(instance.exports.add(2, 3)); // call an exported function -> 5

// Shared linear memory is just an ArrayBuffer view:
const mem = new Uint8Array(instance.exports.memory.buffer);
mem[0] = 42; // JS and WASM see the same byte
Challenge:
Given a WASM module that exports sumBytes(ptr, len) and a memory, write the JS glue that copies a Uint8Array into the module's linear memory at some offset, calls sumBytes with the pointer and length, and returns the sum. Account for the fact that JS and WASM share the same buffer (no copy back needed).
#1 Pitfall: Caching a typed-array view of memory.buffer and reusing it after the WASM module grows its memory. Growing memory detaches the old ArrayBuffer, so the stale view throws or reads garbage. Re-create the view after any memory.grow().
Advanced Debugging Performance & Security 9 days since review
Go beyond console.log: console.table for arrays of objects, console.group for nested structure, console.time/timeEnd for ad-hoc timing, console.trace for a stack snapshot, and console.assert to log only when an invariant fails.
JavaScript
console.table([{ id: 1, name: 'Ada' }, { id: 2, name: 'Linus' }]);

console.group('request');
console.log('url', '/api/users');
console.time('fetch');
// ... work ...
console.timeEnd('fetch');     // "fetch: 12.3ms"
console.groupEnd();

console.assert(1 + 1 === 3, 'math is broken'); // logs ONLY when assertion fails
console.trace('how did we get here?');         // prints the call stack
Challenge:
Build a debugWrap(fn, name) that returns a wrapped function which, on each call, opens a console.group(name), logs the arguments as a console.table, times the execution with console.time/timeEnd, logs the return value, and closes the group — even if fn throws (clean up the group in a finally).
#1 Pitfall: Logging a mutable object reference (console.log(obj)) and reading the console later. Many consoles show the object's current (mutated) state when you expand it, not its state at log time. Log a snapshot — console.log(structuredClone(obj)) or console.log(JSON.stringify(obj)) — when you need a point-in-time value.
Node.js Event Loop & Timers Node.js Runtime 7 days since review
process.nextTick callbacks run before any other microtasks/timers, after the current operation. Promise microtasks drain next. setImmediate fires in the check phase (after I/O); setTimeout(fn, 0) in the timers phase. Inside an I/O callback, setImmediate always beats setTimeout(0).
JavaScript
console.log('sync');
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
console.log('sync end');

// Order: sync, sync end, nextTick, promise, (timeout/immediate order varies at top level)
Challenge:
Predict the exact output order of a script that mixes process.nextTick, a resolved Promise .then, setImmediate, and setTimeout(fn, 0) — first at the top level, then with all four scheduled inside a fs.readFile callback. Explain why the setImmediate vs setTimeout ordering becomes deterministic inside the I/O callback.
#1 Pitfall: Recursively scheduling process.nextTick (or chaining microtasks) so the queue never drains. Because nextTick callbacks run before the loop continues, this starves I/O and timers entirely — the program appears hung. Use setImmediate for "run after I/O gets a turn" work.
Node.js Streams Node.js Runtime 7 days since review
Process data incrementally instead of buffering it all in memory. Readable → Transform → Writable, connected with pipeline() (which propagates errors and cleans up, unlike raw .pipe()). Backpressure is handled for you: a slow writable pauses the readable.
JavaScript
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { Transform } from 'node:stream';

const upper = new Transform({
  transform(chunk, _enc, cb) { cb(null, chunk.toString().toUpperCase()); }
});

await pipeline(
  createReadStream('in.txt'),
  upper,
  createWriteStream('out.txt')
);                       // resolves when done; rejects (and cleans up) on error
console.log('done');
Challenge:
Write a Transform stream that splits an incoming text stream into lines and emits a JSON object { line, num } per line (handle chunk boundaries that split a line in the middle). Wire it with pipeline() from a file read stream to process.stdout, and ensure errors reject the pipeline promise.
#1 Pitfall: Using raw readable.pipe(writable) and forgetting that .pipe() does not forward errors or auto-destroy the source on failure — a mid-stream error leaks file descriptors. Prefer stream.pipeline() (or stream/promises), which destroys all streams and surfaces the error.

The binary, cryptographic, and URL-parsing primitives every Node service leans on.

Buffer & Binary Data Node.js Runtime

Buffer is Node's view over raw bytes — every file read, socket payload, and hash digest is a Buffer. Knowing how to allocate, slice, encode, and decode them is unavoidable in any I/O or networking code.

A Buffer is a subclass of Uint8Array with extra encoding/decoding helpers. It's a fixed-length sequence of bytes living outside the V8 heap.

1Create & convert. Buffer.from builds from a string (with an encoding), an array, or another buffer. Decode back to a string with .toString(encoding).

JavaScript
const buf = Buffer.from('Café', 'utf8');
console.log(buf);              // raw bytes
console.log(buf.length);      // BYTE length, not character length
console.log(buf.toString('hex'));
console.log(buf.toString('base64'));
console.log(Buffer.from('SGk=', 'base64').toString('utf8')); // decode base64
<Buffer 43 61 66 c3 a9> 5 436166c3a9 Q2Fmw6k= Hi

2Allocate. Use Buffer.alloc(n) for a zero-filled buffer. Buffer.allocUnsafe(n) skips zeroing for speed but may contain old memory — only use it if you overwrite every byte.

JavaScript
const b = Buffer.alloc(4);     // safe: 00 00 00 00
b.writeUInt16BE(0x1234, 0);    // write a big-endian 16-bit int at offset 0
b.writeUInt16LE(0x1234, 2);    // little-endian at offset 2
console.log(b.toString('hex'));
console.log(b.readUInt16BE(0).toString(16));
12343412 1234

3Concatenate & slice. Buffer.concat joins chunks (e.g. accumulated from a stream). Note .subarray() shares memory with the original — it is not a copy.

JavaScript
const chunks = [Buffer.from('Hello, '), Buffer.from('world')];
const joined = Buffer.concat(chunks);
console.log(joined.toString());            // Hello, world

const view = joined.subarray(0, 5);        // SHARES memory with joined
view[0] = 0x4a;                            // mutates joined too!
console.log(joined.toString());            // Jello, world
Hello, world Jello, world
Common Mistakes:
❌ Confusing byte length with character length: const b = Buffer.from('Café'); b.length; // 5, not 4 — 'é' is two UTF-8 bytes
✅ For character count use the string: 'Café'.length // 4 Buffer.length / byteLength('Café') gives the BYTE count, which is what matters for buffers, offsets, and Content-Length.
Key Rules:
  • Always pass an explicit encoding to Buffer.from(str, enc) and buf.toString(enc) — the default 'utf8' can silently corrupt binary data.
  • Buffer.alloc is safe (zero-filled); Buffer.allocUnsafe is faster but may leak old memory — only use it when you immediately overwrite the whole buffer.
  • .subarray()/.slice() share the underlying memory; use Buffer.from(buf) to get an independent copy.
Practice Exercises:
Write toBase64Url(str) and fromBase64Url(b64) that encode/decode using URL-safe base64 (replace +/ with -_ and strip = padding). Verify a round trip restores the original string.
Implement concatStream(readable) that collects all chunks from a Readable stream into a single Buffer (push chunks into an array, then Buffer.concat at the end) and returns it via a Promise.
Given a 12-byte buffer, write a function that parses it as three big-endian 32-bit unsigned integers and returns them as an array. Then write the inverse that packs an array of three numbers back into a 12-byte buffer.
Connections: Buffer extends Uint8Array, so everything you learned about Typed Arrays & ArrayBuffer applies. Stream chunks (reviewed today in Node Streams) arrive as Buffers. The crypto and worker_threads sections below both pass data as Buffers/typed arrays.
node:crypto — Hashing, HMAC & Random Node.js Runtime

Password hashing, request signing, token generation, and integrity checks all run through node:crypto. Getting these right (and knowing the safe defaults) is the difference between secure and trivially-breakable code.

1Hashing & HMAC. createHash produces a one-way digest (e.g. for content addressing). createHmac adds a secret key — use it to sign and verify payloads (webhooks, tokens).

JavaScript
import { createHash, createHmac } from 'node:crypto';

const digest = createHash('sha256').update('hello').digest('hex');
console.log(digest.slice(0, 16), '...');

const sig = createHmac('sha256', 'secret-key')
  .update('payload')
  .digest('hex');
console.log('sig:', sig.slice(0, 16), '...');
2cf24dba5fb0a30e ... sig: aa54f... (depends on key)

2Secure randomness. Never use Math.random() for tokens. randomUUID() gives a v4 UUID; randomBytes gives raw entropy you can encode however you like.

JavaScript
import { randomUUID, randomBytes, randomInt } from 'node:crypto';

console.log(randomUUID());                       // e.g. 3b12...-...-...
console.log(randomBytes(16).toString('base64url')); // 22-char URL-safe token
console.log(randomInt(1, 7));                    // unbiased dice roll 1..6
a3f1c2d4-5e6f-4a7b-8c9d-0e1f2a3b4c5d Xy9_aB3cD7eF1gH2iJ3kL 4

3Password hashing & constant-time compare. Use scrypt (slow, salted) for passwords — never a plain SHA. Compare secrets with timingSafeEqual to avoid timing attacks.

JavaScript
import { scryptSync, randomBytes, timingSafeEqual } from 'node:crypto';

function hashPassword(pw) {
  const salt = randomBytes(16);
  const key = scryptSync(pw, salt, 32);
  return salt.toString('hex') + ':' + key.toString('hex');
}
function verify(pw, stored) {
  const [saltHex, keyHex] = stored.split(':');
  const key = scryptSync(pw, Buffer.from(saltHex, 'hex'), 32);
  return timingSafeEqual(key, Buffer.from(keyHex, 'hex'));
}
const h = hashPassword('hunter2');
console.log(verify('hunter2', h), verify('wrong', h));
true false
Common Mistakes:
❌ Comparing secrets with === or a plain hash for passwords: if (userMac === expectedMac) { ... } // timing attack leaks the answer storedHash = createHash('sha256').update(pw).digest('hex'); // unsalted, fast
✅ Use timingSafeEqual for MAC/token comparison, and a slow salted KDF (scrypt / argon2 / bcrypt) for passwords: crypto.timingSafeEqual(Buffer.from(a), Buffer.from(b));
Key Rules:
  • Use randomBytes/randomUUID/randomInt for anything security-sensitive — Math.random() is predictable.
  • Passwords need a slow, salted KDF (scrypt); fast hashes (SHA-256) are for integrity/content addressing, not passwords.
  • Compare secrets/MACs with timingSafeEqual (equal-length buffers) so comparison time doesn't leak how many bytes matched.
Practice Exercises:
Write signWebhook(body, secret) returning an HMAC-SHA256 hex signature, and verifyWebhook(body, secret, sig) that recomputes the HMAC and compares it with timingSafeEqual (guard against length mismatch first).
Implement a contentHash(buffer) that returns the first 12 hex chars of the SHA-256 digest — a short content-addressed id. Verify identical input yields an identical id and a one-byte change yields a different one.
Build a genToken(bytes = 32) that returns a URL-safe random token using randomBytes(...).toString('base64url'), and explain why you'd pick this over randomUUID() when you need more than 122 bits of entropy.
Connections: Digests and keys are returned as Buffers. This is the Node counterpart to the browser Web Crypto API (note: Web Crypto's subtle methods are async/Promise-based, while these *Sync variants block). HMAC verification is the server side of the webhook signing you'd protect against in XSS/Security.
URL & URLSearchParams Node.js Runtime

Parsing URLs and query strings by hand with split('?') and regex is a perennial source of bugs (encoding, repeated keys, missing parts). The WHATWG URL API — identical in Node and the browser — does it correctly and safely.

1Parse & build. The URL constructor splits a URL into typed parts. A second base argument resolves relative URLs.

JavaScript
const u = new URL('https://ex.com:8080/path/page?q=js&sort=new#sec');
console.log(u.protocol, u.hostname, u.port);
console.log(u.pathname, u.hash);
console.log(u.searchParams.get('q'));

// Resolve a relative path against a base:
console.log(new URL('../about', 'https://ex.com/a/b/c').href);
https: ex.com 8080 /path/page #sec js https://ex.com/a/about

2Query strings. URLSearchParams handles encoding, multiple values per key, and iteration. Mutating url.searchParams updates url.href live.

JavaScript
const params = new URLSearchParams('tag=js&tag=node&page=2');
console.log(params.getAll('tag'));   // repeated keys -> array
console.log(params.get('page'));

params.append('tag', 'wasm');
params.set('page', '3');
console.log(params.toString());      // auto-encoded
[ 'js', 'node' ] 2 tag=js&tag=node&page=2&tag=wasm

3Build from an object & encode safely. Pass a plain object to URLSearchParams and it encodes special characters for you — no manual encodeURIComponent juggling.

JavaScript
const qs = new URLSearchParams({ q: 'a b&c', lang: 'en' });
console.log(qs.toString());          // spaces/& encoded automatically

const api = new URL('https://api.ex.com/search');
api.search = qs.toString();
console.log(api.href);
q=a+b%26c&lang=en https://api.ex.com/search?q=a+b%26c&lang=en
Common Mistakes:
❌ Hand-rolling query parsing / concatenation: const q = url.split('?')[1]; const val = q.split('&')[0].split('=')[1]; // breaks on encoding, order, repeats const href = base + '?q=' + userInput; // unencoded -> broken/unsafe URL
✅ Let the platform parse and encode: const val = new URL(url).searchParams.get('q'); u.searchParams.set('q', userInput); // encodes correctly
Key Rules:
  • new URL(input) throws on an invalid absolute URL — wrap in try/catch or use URL.canParse(input) to validate user input.
  • Use getAll() for keys that can repeat; get() returns only the first value.
  • Build query strings with URLSearchParams (it encodes for you) instead of string concatenation + encodeURIComponent.
Practice Exercises:
Write addParams(url, obj) that returns a new URL string with the key/value pairs from obj merged into the existing query string (overwriting existing keys), leaving the rest of the URL untouched.
Implement parseQuery(qs) that turns a query string into an object, collapsing single-value keys to a string and repeated keys to an array (e.g. tag=a&tag=b&p=1 → { tag: ['a','b'], p: '1' }).
Write isSameOrigin(a, b) that returns true only when two URLs share the same protocol, hostname, and port — using the parsed URL fields, not string matching.
Connections: The same URL/URLSearchParams objects are used by fetch in both Node and the browser, so this knowledge is fully portable. Building request URLs here feeds directly into the Fetch Advanced patterns from earlier days.

🆕 Concurrency & Testing

Run CPU-bound work off the main thread, and lock behavior in with a fast test runner.

Node.js worker_threads Node.js Runtime

Node's event loop is single-threaded, so a heavy synchronous computation (parsing, hashing, image work) blocks every other request. worker_threads moves that work to a real OS thread, keeping the main thread responsive.

1Spawn a worker. The main thread creates a Worker and exchanges data via postMessage / the message event. Inside the worker, parentPort is the channel back.

JavaScript — main.mjs
import { Worker } from 'node:worker_threads';

const worker = new Worker(new URL('./worker.mjs', import.meta.url));
worker.postMessage(40);                         // send input
worker.on('message', (result) => {
  console.log('fib(40) =', result);
  worker.terminate();
});
worker.on('error', (err) => console.error(err));
console.log('main thread is NOT blocked');
main thread is NOT blocked fib(40) = 102334155
JavaScript — worker.mjs
import { parentPort } from 'node:worker_threads';

function fib(n) { return n < 2 ? n : fib(n - 1) + fib(n - 2); }

parentPort.on('message', (n) => {
  parentPort.postMessage(fib(n));               // CPU work happens off-main
});
// runs on a separate OS thread

2workerData & one-shot tasks. Pass startup data via the workerData option, and wrap a worker in a Promise for a clean async API.

JavaScript
import { Worker } from 'node:worker_threads';

function runTask(file, data) {
  return new Promise((resolve, reject) => {
    const w = new Worker(file, { workerData: data });
    w.once('message', resolve);
    w.once('error', reject);
    w.once('exit', (code) => {
      if (code !== 0) reject(new Error('worker exit ' + code));
    });
  });
}
// const sum = await runTask('./sum-worker.mjs', [1, 2, 3]);
// resolves with the worker's first posted message

3Zero-copy transfer. Messages are structured-cloned (copied) by default. Pass an ArrayBuffer in the transfer list to move ownership without copying — the sender can no longer use it.

JavaScript
const big = new Uint8Array(1_000_000);
// Transfer the underlying buffer (no copy); 'big' becomes detached after this:
worker.postMessage(big, [big.buffer]);
console.log(big.byteLength);   // 0 — ownership moved to the worker
0
Common Mistakes:
❌ Spawning a worker per request for tiny tasks: for (const job of jobs) new Worker('./w.mjs'); // thread startup cost dwarfs the work
✅ Reuse workers via a pool (e.g. a fixed set of N workers, or the 'piscina' library). Only offload work that is genuinely CPU-bound and long enough to amortize the messaging/startup overhead.
Key Rules:
  • Use workers for CPU-bound work; for I/O-bound work, async APIs on the single thread are already non-blocking and simpler.
  • Messages are copied (structured clone) — functions, DOM nodes, and class instances don't survive; use the transfer list for large ArrayBuffers to avoid the copy.
  • Workers have startup and per-message cost — pool and reuse them instead of spawning per task.
Practice Exercises:
Write a runTask(file, data) helper (Promise-wrapped worker) and a worker that sums an array passed via workerData, posting the total back. Confirm the main thread logs a message before the worker's result.
Build a minimal fixed-size worker pool: given N worker files and a queue of jobs, dispatch jobs to idle workers and resolve each job's Promise when its worker replies. Ensure no more than N workers run concurrently.
Transfer a large Uint8Array to a worker that reverses its bytes in place and transfers it back. Verify the original buffer is detached (length 0) on the main side after sending.
Connections: This is the Node analogue of browser Web Workers — same postMessage/transfer model. Transferable ArrayBuffers build on Typed Arrays and Buffer. Offloading CPU work is the structural fix for the event-loop-starvation pitfall in today's Node Event Loop review.
Testing with Vitest & Jest Testing & Quality

Vitest and Jest are the dominant JS test runners. They share a near-identical API (describe/it/expect), so learning one transfers to the other — Vitest is the modern, ESM-native, faster choice for new projects.

1Structure & assertions. Group with describe, define cases with it/test, assert with expect(value).matcher().

JavaScript — math.test.js
import { describe, it, expect } from 'vitest'; // or 'from "@jest/globals"'
import { add } from './math.js';

describe('add', () => {
  it('sums two numbers', () => {
    expect(add(2, 3)).toBe(5);            // strict ===
  });
  it('matches objects by value', () => {
    expect({ a: 1 }).toEqual({ a: 1 });   // deep equality
  });
});
✓ add > sums two numbers ✓ add > matches objects by value 2 passed

2Async & errors. Return or await a Promise; assert rejections/throws with .rejects / .toThrow().

JavaScript
it('awaits async results', async () => {
  await expect(Promise.resolve(42)).resolves.toBe(42);
});

it('throws on bad input', () => {
  expect(() => JSON.parse('{bad')).toThrow();
  expect(() => JSON.parse('{bad')).toThrow(SyntaxError);
});

it('rejects', async () => {
  await expect(Promise.reject(new Error('nope'))).rejects.toThrow('nope');
});
✓ awaits async results ✓ throws on bad input ✓ rejects

3Mocks & spies. vi.fn() (Vitest) / jest.fn() creates a spy you can assert calls on. Lifecycle hooks (beforeEach) reset state between tests.

JavaScript
import { it, expect, vi, beforeEach } from 'vitest';

let onSave;
beforeEach(() => { onSave = vi.fn(); });

it('calls the callback once with the payload', () => {
  function save(data, cb) { cb({ ok: true, data }); }
  save('hi', onSave);
  expect(onSave).toHaveBeenCalledTimes(1);
  expect(onSave).toHaveBeenCalledWith({ ok: true, data: 'hi' });
});
✓ calls the callback once with the payload
Common Mistakes:
❌ Forgetting to await an async assertion: it('x', () => { expect(Promise.reject('e')).rejects.toThrow(); }); // test PASSES even though it never waited — the rejection is unhandled
✅ await (or return) the assertion so the runner waits for it: it('x', async () => { await expect(p).rejects.toThrow(); });
Key Rules:
  • Use toBe for primitives/identity (===) and toEqual for deep structural equality — mixing them causes confusing failures.
  • Always await or return async assertions (resolves/rejects), or the test can pass without actually checking anything.
  • Reset mocks/state between tests with beforeEach (or vi.clearAllMocks()) so test order never affects results.
Practice Exercises:
Write tests for a clamp(n, min, max) function covering: a value inside the range, below the min, above the max, and the boundary values. Use toBe for each.
Test an async fetchUser(id) that returns a Promise: assert it resolves to the right object for a valid id, and rejects.toThrow('not found') for a missing one — remembering to await both.
Test an EventEmitter-style once(event, cb): use a spy (vi.fn()/jest.fn()) as the callback, emit the event twice, and assert the spy was called exactly once with the emitted payload.
Connections: Tests are the safety net for everything else in this curriculum — especially the async assertions, which lean on your Promise / async-await knowledge. Mocks/spies are closures tracking call records. Vitest runs your ES modules natively, no transpile step.
