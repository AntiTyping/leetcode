Destructuring Advanced Functions & Iteration 11 days since review
Destructuring binds by shape. You can nest, supply defaults (applied only when the value is undefined), rename to a new binding, pull a rest of the remaining keys, and even use computed keys. Defaults can reference earlier bindings in the same pattern.
JavaScript
const config = { db: { host: 'localhost' }, retries: 0 };

// nested + rename + default (default fires only on undefined, NOT on 0):
const { db: { host, port = 5432 }, retries = 3, mode = 'dev' } = config;
console.log(host, port, retries, mode);   // localhost 5432 0 dev

// rest gathers the leftover own-enumerable keys:
const { db: _db, ...rest } = config;
console.log(rest);                        // { retries: 0 }

// computed key + default referencing an earlier binding:
const key = 'x';
const { [key]: x = 1, y = x * 10 } = { x: 4 };
console.log(x, y);                        // 4 40
Challenge:
Write parseArgs(argv, defaults) that destructures a settings object out of argv, applying defaults only for missing keys, renaming o→output and v→verbose, and collecting any unknown flags into a rest object. It must preserve a passed verbose: false and a passed count: 0 rather than overriding them with defaults.
#1 Pitfall: Assuming defaults apply to any falsy value — they apply only to undefined. A stored null, 0, '', or false is kept as-is. Also, destructuring a null/undefined source (const { a } = null) throws a TypeError — guard with = {} on the parameter.
Spread & Rest Functions & Iteration 11 days since review
... is spread when it expands (in an array/object literal or a call) and rest when it collects (in a parameter list or destructuring pattern). Object spread copies own-enumerable properties; later keys win. Both spreads are shallow — nested objects are shared by reference.
JavaScript
const base = { a: 1, nested: { k: 1 } };
const merged = { ...base, b: 2, a: 9 };    // later 'a' overrides earlier
console.log(merged);                       // { a: 9, nested: { k: 1 }, b: 2 }

merged.nested.k = 99;                       // shallow copy: shared reference!
console.log(base.nested.k);                // 99  (mutated the original)

function sum(...nums) {                      // rest: gather args into an array
  return nums.reduce((a, b) => a + b, 0);
}
console.log(sum(...[1, 2, 3], 4));         // spread into call -> 10
Challenge:
Write deepMerge(a, b) that recursively merges two plain-object trees (arrays replaced, not concatenated; scalars from b win) and returns a new object without mutating either input — proving you understand why a single spread is not enough for nested data.
#1 Pitfall: Believing { ...obj } or [...arr] gives you a deep copy. It copies only the top level; nested objects/arrays are still shared. Mutating copy.nested mutates the original. For a true deep clone use structuredClone(obj).
AbortController & AbortSignal Async Mastery 11 days since review
An AbortController owns a signal you pass into cancelable operations; calling controller.abort(reason) flips the signal to aborted and fires its 'abort' event. Built-ins like fetch reject with an AbortError. AbortSignal.timeout(ms) and AbortSignal.any([...]) compose signals.
JavaScript
const ac = new AbortController();
const timer = setTimeout(() => ac.abort(new Error('too slow')), 100);

try {
  // combine a manual controller with a hard timeout:
  const res = await fetch('/api/data', {
    signal: AbortSignal.any([ac.signal, AbortSignal.timeout(5000)]),
  });
  clearTimeout(timer);
  console.log(await res.json());
} catch (err) {
  if (err.name === 'AbortError') console.log('cancelled:', ac.signal.reason);
  else throw err;
}
Challenge:
Write abortable(work, signal) where work is a long loop; it should check signal.aborted each iteration and reject with signal.reason as soon as it's set, and also attach an 'abort' listener (using { once: true }) so a mid-iteration abort rejects promptly. Ensure the listener is removed on normal completion to avoid a leak.
#1 Pitfall: Reusing one AbortController for multiple sequential operations. Once aborted, a signal stays aborted forever — a fresh operation started with it rejects immediately. Create a new controller per cancelable unit of work, and always check err.name === 'AbortError' so you don't treat cancellation as a real failure.
Intl API Functions & Iteration 9 days since review
The Intl namespace does locale-aware formatting: NumberFormat (currency, units, compact), DateTimeFormat, Collator (correct sorting), RelativeTimeFormat, and PluralRules. Create the formatter once and reuse it — construction is the expensive part.
JavaScript
const money = new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' });
console.log(money.format(1234.5));        // $1,234.50

const rel = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });
console.log(rel.format(-1, 'day'));       // yesterday
console.log(rel.format(3, 'hour'));       // in 3 hours

// locale-correct sorting (Collator), not code-unit order:
const words = ['zebra', 'apple', 'Ähre', 'ábaco'];
console.log(words.sort(new Intl.Collator('de').compare));
// [ 'ábaco', 'Ähre', 'apple', 'zebra' ]
Challenge:
Write formatList(items, locale) using Intl.ListFormat to render ['A','B','C'] as "A, B, and C" (en) and pick the correct conjunction for another locale. Then add a pluralized count prefix using Intl.PluralRules so 1 item reads "1 item" and 3 read "3 items" — driven by the locale's plural categories, not a hand-written === 1 check.
#1 Pitfall: Re-creating an Intl formatter inside a hot loop or on every render. Construction resolves locale data and is comparatively slow; the .format call is cheap. Build the formatter once (module scope or memoized) and reuse it.
Promise Internals Async Mastery 9 days since review
A promise is a state machine: pending → fulfilled or rejected, settling exactly once. .then returns a new promise, enabling chains; a returned promise is adopted (flattened). The combinators differ: all (fail-fast), allSettled (never rejects), race (first to settle), any (first to fulfill).
JavaScript
const tasks = [
  Promise.resolve(1),
  Promise.reject(new Error('boom')),
  Promise.resolve(3),
];

console.log(await Promise.allSettled(tasks));
// [ {status:'fulfilled', value:1},
//   {status:'rejected', reason:Error: boom},
//   {status:'fulfilled', value:3} ]

// any() resolves with the first FULFILLED, ignoring rejections
// (rejects with AggregateError only if ALL reject):
console.log(await Promise.any(tasks));      // 1

// chaining: each then returns a new promise; a returned promise is awaited
console.log(await Promise.resolve(2).then(x => Promise.resolve(x * 10)));  // 20
Challenge:
Implement promiseAllLimit(tasks, limit) that runs at most limit of the async tasks (functions returning promises) concurrently, resolves with the results in the original order, and rejects on the first failure like Promise.all. Do it without awaiting tasks sequentially — keep the pool saturated.
#1 Pitfall: Confusing Promise.all (rejects the moment any input rejects, losing the other results) with Promise.allSettled (waits for all, never rejects). Also: an unhandled rejection in a combinator's input still fires unhandledrejection unless the input promise is observed — attach handlers or use allSettled.
async/await Patterns Async Mastery 9 days since review
await pauses the async function until the awaited value settles, unwrapping it (or throwing on rejection, catchable with try/catch). Awaiting sequentially serializes independent work needlessly; start the promises first, then await, to run them concurrently.
JavaScript
// ❌ SEQUENTIAL: b doesn't start until a finishes (total = tA + tB)
async function slow() {
  const a = await fetchA();
  const b = await fetchB();
  return [a, b];
}

// ✅ CONCURRENT: both start now, then await both (total = max(tA, tB))
async function fast() {
  const pA = fetchA(), pB = fetchB();      // kick off immediately
  return [await pA, await pB];
  // or: return Promise.all([fetchA(), fetchB()]);
}

// per-item error isolation with allSettled:
async function fetchAll(urls) {
  const results = await Promise.allSettled(urls.map(u => fetch(u)));
  return results.filter(r => r.status === 'fulfilled').map(r => r.value);
}
Challenge:
Write retry(fn, { attempts, baseDelay }) that calls the async fn, and on rejection retries up to attempts times with exponential backoff (baseDelay * 2**n) plus jitter, rethrowing the last error if all attempts fail. Make it accept an optional signal so an abort stops further retries immediately.
#1 Pitfall: awaiting inside a for/forEach loop when the iterations are independent — this serializes them. Note array.forEach(async ...) doesn't await at all (it ignores the returned promises), so errors go unhandled. Use for...of + await for true sequencing, or Promise.all(map(...)) for concurrency.

Promise.try Async Mastery

When you call a function that might be sync or async — and might throw synchronously before it ever returns a promise — that synchronous throw escapes your .catch/await chain. Promise.try(fn) runs fn and funnels both a synchronous throw and an async rejection into one rejected promise, unifying your error handling.

1The problem it solves. A function that validates its input synchronously can throw before returning. Wrapping the call in Promise.try turns that throw into a rejection you can .catch.

JavaScript
function parse(input) {
  if (typeof input !== 'string') throw new TypeError('need a string'); // SYNC throw
  return JSON.parse(input);                                            // may also throw
}

// ❌ the sync throw is NOT caught by .catch — it explodes at the call site:
// parse(42).catch(...)   // Uncaught TypeError

// ✅ Promise.try captures sync throws AND async rejections uniformly:
Promise.try(parse, '{"ok":true}')
  .then((v) => console.log('ok', v))
  .catch((e) => console.log('caught', e.message));

Promise.try(parse, 42)
  .then((v) => console.log('ok', v))
  .catch((e) => console.log('caught', e.message));
ok { ok: true } caught need a string

2It forwards arguments. Promise.try(fn, a, b, c) calls fn(a, b, c) — no need for an extra arrow wrapper, and no accidental closure captures.

JavaScript
const compute = (x, y) => x / y;

const result = await Promise.try(compute, 10, 2);   // args passed straight through
console.log(result);                                 // 5

// works whether the function is sync or async — the caller stays uniform:
const asyncCompute = async (x, y) => x + y;
console.log(await Promise.try(asyncCompute, 3, 4));  // 7
5 7

3Uniform handling of unknown callbacks. Ideal for plugin systems / middleware where a user-supplied handler could be either sync or async — you don't know which, and you want one error path.

JavaScript
async function runHandler(handler, req) {
  // handler might return a value, a promise, or throw synchronously:
  return Promise.try(handler, req)
    .catch((err) => ({ error: err.message }));   // one place to normalize failures
}

console.log(await runHandler(() => 'sync-ok', {}));            // sync-ok
console.log(await runHandler(async () => 'async-ok', {}));     // async-ok
console.log(await runHandler(() => { throw new Error('x'); })); // { error: 'x' }
sync-ok async-ok { error: 'x' }
Common Mistakes:
❌ Calling the function first, then wrapping — the sync throw already escaped: Promise.resolve(parse(42)) // parse(42) throws BEFORE resolve runs Promise.try(parse(42)) // same bug: you invoked parse, passed its result
✅ Pass the function and its args to Promise.try; let IT do the call: Promise.try(parse, 42) // parse is invoked inside the try, throw -> rejection
Key Rules:
  • Pass a reference plus arguments: Promise.try(fn, ...args). Never Promise.try(fn()) — that calls fn outside the guard.
  • Use it at boundaries where a callback's sync-vs-async nature is unknown, so a synchronous throw becomes a rejection instead of crashing the caller.
  • Inside your own async function you rarely need it — a throw there already becomes a rejection. It shines when invoking external functions.
Practice Exercises:
Write safeInvoke(fn, ...args) that returns a Promise resolving to { ok: true, value } or { ok: false, error }, working correctly whether fn returns a value, returns a promise, or throws synchronously. Use Promise.try.
Build a pipeline(...steps) runner where each step may be sync or async and may throw synchronously. Run them in order, feeding each result to the next, and reject the whole pipeline on the first failure — using Promise.try so a step's sync throw is handled just like a rejection.
Show, with a small demo, why Promise.resolve().then(() => risky()) and Promise.try(risky) behave the same for error capture but differ in microtask timing — log a marker before each to observe that .then defers to a later microtask while Promise.try invokes risky synchronously.
Connections: This completes the Promise combinator family and pairs with Promise.withResolvers for building robust wrappers. The sync-throw-vs-rejection distinction is the same hazard you manage with try/catch in async functions.
RegExp.escape Data & Collections

Building a regex from user input (a search box, a config value) has always been a footgun: characters like ., *, or ( are interpreted as regex syntax, changing the meaning or throwing. For years everyone copy-pasted the same hand-rolled escape function. ES2025's RegExp.escape is the standard, correct one.

1Escape a literal string for embedding in a pattern. Every regex metacharacter is backslash-escaped so the result matches the input literally.

JavaScript
console.log(RegExp.escape('$5.00 (USD)'));   // \$5\.00\x20\(USD\)
console.log(RegExp.escape('a+b?'));          // a\+b\?  ... see rule below

// Safe "find literal substring" — user input can't inject regex syntax:
function highlight(text, query) {
  const re = new RegExp(RegExp.escape(query), 'gi');
  return text.replace(re, (m) => `[${m}]`);
}
console.log(highlight('cost is $5.00 today', '$5.00'));  // cost is [$5.00] today
\$5\.00\x20\(USD\) \x61\+b\? cost is [$5.00] today

2The leading-character rule. To guarantee the result is safe in every position (even right after another token), RegExp.escape hex-escapes the first code point when it's an ASCII letter or digit — that's why 'a+b?' above starts with \x61 (the code for a) rather than a bare a. Spaces are also encoded (\x20) so escaped strings survive the x (extended) flag.

JavaScript
console.log(RegExp.escape('hello'));   // \x68ello   (first char h -> \x68)
console.log(RegExp.escape('.hello'));  // \.hello    (first char '.' isn't alnum)

// It always round-trips to a literal match, regardless of the hex escaping:
const q = 'C++';
console.log(new RegExp(RegExp.escape(q)).test('I love C++'));   // true
\x68ello \.hello true
Common Mistakes:
❌ Interpolating raw user input into a RegExp: new RegExp(userInput) // 'a.*' matches too much; '(' throws SyntaxError new RegExp(userInput + '$') // metacharacters change the meaning entirely
✅ Escape first, then build the pattern: new RegExp(RegExp.escape(userInput) + '$') // matches the literal string, anchored
Key Rules:
  • Use it whenever a runtime string must be matched literally inside a constructed RegExp — search, replace-all-of-a-word, splitting on a user-chosen delimiter.
  • The output is meant to be embedded in a pattern, not read by humans; the \x escaping of the first alnum char is intentional and harmless.
  • You don't need it for regex literals you write by hand — only for patterns assembled from dynamic strings.
Practice Exercises:
Write countOccurrences(haystack, needle) that counts non-overlapping literal occurrences of needle (which may contain regex metacharacters like . or *) using a global RegExp built with RegExp.escape. Verify countOccurrences('a.b.c', '.') is 2.
Implement replaceAllLiteral(text, find, replaceWith) using a RegExp.escape-built global regex (do NOT use String.prototype.replaceAll with a string arg). Confirm it treats find = '(x)' as the literal three characters.
Given a list of forbidden words that may include punctuation, build a single alternation regex (word1|word2|...) by escaping each word with RegExp.escape and joining with |, then test that it flags exact literal matches without regex injection.
Connections: This is the safe-construction counterpart to the advanced RegExp features (named groups, lookarounds) you studied — those describe patterns you author; RegExp.escape protects patterns you assemble from data. It shares the "escape the interpolation, not the template" philosophy with tagged-template HTML/SQL builders.
Uint8Array Base64 & Hex Data & Collections

Converting bytes to/from Base64 or hex was awkward: btoa/atob work on binary strings (and mishandle non-Latin1), and hex needed manual map(...toString(16)) loops. ES2025 adds native, correct methods directly on Uint8Array, so encoding binary for JSON, data URLs, or tokens is a one-liner.

1Bytes → string and back. Instance methods .toBase64() / .toHex() encode; static Uint8Array.fromBase64() / fromHex() decode into a fresh array.

JavaScript
const bytes = new Uint8Array([72, 105]);      // 'H', 'i'

console.log(bytes.toBase64());                // 'SGk='
console.log(bytes.toHex());                   // '4869'

console.log(Uint8Array.fromBase64('SGk='));   // Uint8Array(2) [ 72, 105 ]
console.log(Uint8Array.fromHex('4869'));      // Uint8Array(2) [ 72, 105 ]
SGk= 4869 Uint8Array(2) [ 72, 105 ] Uint8Array(2) [ 72, 105 ]

2Options: URL-safe alphabet & padding. toBase64 accepts { alphabet: 'base64url', omitPadding: true } — exactly what you want for URL/JWT-style tokens (uses -/_ instead of +// and drops =).

JavaScript
const data = new Uint8Array([251, 255, 191]);

console.log(data.toBase64());                              // '+/+/'
console.log(data.toBase64({ alphabet: 'base64url' }));     // '-_-_'
console.log(data.toBase64({ alphabet: 'base64url', omitPadding: true }));  // '-_-_'

// decode must know the alphabet it was encoded with:
const url = data.toBase64({ alphabet: 'base64url' });
console.log(Uint8Array.fromBase64(url, { alphabet: 'base64url' }));   // [251,255,191]
+/+/ -_-_ -_-_ Uint8Array(3) [ 251, 255, 191 ]

3Round-tripping text. Pair with TextEncoder/TextDecoder to Base64-encode arbitrary Unicode safely — no more btoa Latin1 breakage on emoji or accents.

JavaScript
const text = 'café ☕';
const encoded = new TextEncoder().encode(text).toBase64();
console.log(encoded);                                  // 'Y2Fmw6kg4piV'

const decoded = new TextDecoder().decode(Uint8Array.fromBase64(encoded));
console.log(decoded);                                  // 'café ☕'
console.log(decoded === text);                         // true
Y2Fmw6kg4piV café ☕ true
Common Mistakes:
❌ Using btoa/atob for binary or Unicode: btoa('café') // InvalidCharacterError (non-Latin1) btoa(String.fromCharCode(...bytes)) // fragile, corrupts high bytes
✅ Encode text to bytes first, then use the typed-array methods: new TextEncoder().encode('café').toBase64() // correct for any Unicode
Key Rules:
  • .toBase64()/.toHex() are instance methods; fromBase64()/fromHex() are static and return a new Uint8Array.
  • For URLs/tokens use { alphabet: 'base64url' } on both encode and decode — the alphabets are not interchangeable.
  • These operate on bytes. For text, go through TextEncoder/TextDecoder so multi-byte characters survive.
Practice Exercises:
Write toDataURL(bytes, mime) that returns data:<mime>;base64,<...> using toBase64(), and fromDataURL(url) that parses it back to a { mime, bytes } pair with fromBase64.
Implement hmacToHex(keyBytes, msgBytes)-style formatting: given a Uint8Array digest, produce a lowercase hex string with toHex(), and a verifier that compares two hex strings for equality. Confirm fromHex(toHex(x)) round-trips.
Build a minimal URL-safe token codec: encodeToken(obj) JSON-stringifies, encodes to base64url without padding, and decodeToken(str) reverses it. Verify a token with emoji in it round-trips exactly.
Connections: These methods build on typed arrays & ArrayBuffer and complement Web Crypto / node:crypto, which produce raw byte digests you then hex- or base64-encode. The Unicode-safe encoding path is the same reason TextEncoder exists instead of btoa.
Import Attributes Modules & Patterns

JavaScript modules can now import non-JS resources — most importantly JSON — with an inline attribute that tells the host how to interpret the file. The with { type: 'json' } syntax is a security feature too: it forces the runtime to treat the resource as data, never executing it as a module.

1Static JSON import. Append with { type: 'json' } to a normal import. The default export is the parsed object.

JavaScript
// config.json  ->  { "name": "app", "version": 3 }
import config from './config.json' with { type: 'json' };

console.log(config.name, config.version);   // app 3
// Note: JSON modules have ONLY a default export — no named imports.
app 3

2Dynamic import with attributes. The import() function takes a second options argument carrying a with object — handy for loading config lazily or conditionally.

JavaScript
async function loadLocale(lang) {
  const mod = await import(`./locales/${lang}.json`, {
    with: { type: 'json' },
  });
  return mod.default;                        // the parsed JSON object
}

const en = await loadLocale('en');
console.log(en.greeting);                    // e.g. 'Hello'
Hello

3Why the attribute is required (not optional). Hosts must know a resource is JSON before fetching, so a server that mislabels a .json URL as JavaScript can't get it executed. Omitting the attribute on a JSON module is an error — this is the evolution of the earlier assert { type: 'json' } syntax, which is now deprecated in favor of with.

JavaScript
// ❌ old, deprecated keyword:
// import data from './data.json' assert { type: 'json' };

// ✅ current standard keyword:
import data from './data.json' with { type: 'json' };
console.log(Array.isArray(data) || typeof data);   // 'object' (or true for arrays)
object
Common Mistakes:
❌ Importing JSON without the attribute, or expecting named exports: import config from './config.json'; // TypeError: needs type attribute import { version } from './config.json' with { type: 'json' }; // no named exports
✅ Use the attribute and take the default, then destructure in code: import config from './config.json' with { type: 'json' }; const { version } = config;
Key Rules:
  • Syntax is with { type: 'json' } for static imports and { with: { type: 'json' } } as the 2nd arg to dynamic import().
  • JSON modules expose a default export only (the parsed value); there are no named exports.
  • Prefer with over the deprecated assert keyword. Attributes are validated before execution, which is what makes them a safe way to load data resources.
Practice Exercises:
Create a package.json-style file and statically import it with with { type: 'json' }; log its name and the number of dependencies keys using Object.keys(...).length.
Write loadConfig(env) that dynamically imports ./config.<env>.json with the import-attributes options object, falling back to ./config.default.json if the first import rejects. Return the merged object.
Explain in a short comment block why with { type: 'json' } is a security boundary: contrast what could happen if a mislabeled endpoint returned executable JS versus data, and why the runtime refuses to run a JSON module as code.
Connections: Import attributes extend ES modules deep dive (static vs dynamic import(), import.meta) to non-JS resources and feed naturally into module patterns like lazy locale/config loading. The default-only shape mirrors how bundlers have long exposed JSON.

🆕 Internationalization

Correctly counting and splitting human text — where "length" and "split on spaces" quietly break.

Intl.Segmenter Internationalization

string.length counts UTF-16 code units, and [...string] counts code points — neither matches what a human calls a "character." An emoji flag or a family emoji is one grapheme made of several code points. Intl.Segmenter splits text into locale-correct graphemes, words, or sentences — the right tool for cursor movement, truncation, and word counting.

1Grapheme segmentation = real "characters." Create a segmenter with granularity: 'grapheme' and iterate the result; each entry is a { segment, index, input } object.

JavaScript
const flag = '🇺🇸';
console.log(flag.length);          // 4  (two surrogate pairs)
console.log([...flag].length);     // 2  (two regional-indicator code points)

const seg = new Intl.Segmenter('en', { granularity: 'grapheme' });
console.log([...seg.segment(flag)].length);   // 1  (one visible character)

const family = '👨‍👩‍👧';
console.log([...family].length);              // 5  (code points incl. ZWJ)
console.log([...seg.segment(family)].length); // 1  (one grapheme cluster)
4 2 1 5 1

2Word segmentation with isWordLike. With granularity: 'word', each segment is flagged isWordLike (letters/numbers) vs. not (spaces, punctuation) — so counting real words is a filter, not a fragile split(/\s+/).

JavaScript
const words = new Intl.Segmenter('en', { granularity: 'word' });
const text = "Don't stop—now!";

const wordList = [...words.segment(text)]
  .filter((s) => s.isWordLike)
  .map((s) => s.segment);

console.log(wordList);            // [ "Don't", 'stop', 'now' ]
console.log(wordList.length);    // 3   (apostrophe kept, em-dash/! excluded)
[ "Don't", 'stop', 'now' ] 3

3Locale awareness. Languages without spaces (Japanese, Chinese, Thai) can't be word-split by whitespace at all. A locale-aware segmenter uses the language's rules; sentence granularity likewise respects locale punctuation.

JavaScript
// Japanese has no spaces between words — split() is useless here:
const ja = new Intl.Segmenter('ja', { granularity: 'word' });
const parts = [...ja.segment('私は学生です')]
  .filter((s) => s.isWordLike)
  .map((s) => s.segment);
console.log(parts);       // e.g. [ '私', 'は', '学生', 'です' ]  (dictionary-based)

const sentences = new Intl.Segmenter('en', { granularity: 'sentence' });
console.log([...sentences.segment('Hi there. How are you?')].length);  // 2
[ '私', 'は', '学生', 'です' ] 2
Common Mistakes:
❌ Using .length or .slice for user-visible characters: '🇺🇸'.length // 4, not 1 name.slice(0, 5) // can cut through a surrogate pair -> '�' mojibake text.split(/\s+/) // wrong word count; breaks for CJK, keeps punctuation
✅ Segment by grapheme/word for anything human-facing: [...new Intl.Segmenter('en',{granularity:'grapheme'}).segment(name)] [...seg.segment(text)].filter(s => s.isWordLike).length // real word count
Key Rules:
  • granularity is 'grapheme' (visible characters), 'word', or 'sentence'. Iterate segment(str) to get { segment, index, isWordLike } entries.
  • Use grapheme segmentation for length limits, truncation, and cursor/backspace logic; word segmentation (with isWordLike) for word counts and search.
  • Always pass a locale — word/sentence boundaries and CJK handling depend on it. Construct the segmenter once and reuse it, like other Intl objects.

Implement wordCount(text, locale) using a word-granularity segmenter and the isWordLike flag. Compare its result against text.split(/\s+/).length on a string with punctuation and an em-dash to show where split is wrong.
Write reverseGraphemes(str) that reverses a string by grapheme (so flags and emoji stay intact), and demonstrate that a naive [...str].reverse().join('') corrupts a flag emoji while your version does not.
