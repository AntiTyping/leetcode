Structured Clone Data & Collections 14 days since review
structuredClone(value) makes a deep copy using the structured-clone algorithm — the same one postMessage and IndexedDB use. Unlike JSON.parse(JSON.stringify(x)) it preserves Map, Set, Date, RegExp, typed arrays, and cyclic references. It cannot clone functions, DOM nodes, or class prototypes (you get a plain object), and throws DataCloneError on functions/symbols.
JavaScript
const original = { when: new Date(), tags: new Set(['a']), self: null };
original.self = original;                    // cycle

const copy = structuredClone(original);
copy.tags.add('b');
console.log(original.tags.has('b'));         // false — deep, independent
console.log(copy.self === copy);             // true — cycle preserved

// transfer (move, don't copy) an ArrayBuffer for zero-copy handoff:
const buf = new ArrayBuffer(8);
const moved = structuredClone(buf, { transfer: [buf] });
console.log(buf.byteLength);                 // 0 — detached after transfer
Challenge:
Build a nested object containing a Map, a Date, a typed array, and a self-reference, then deep-clone it two ways — structuredClone vs JSON round-trip — and enumerate every difference in the result (lost types, thrown cycle error, dropped undefined).
#1 Pitfall: Expecting functions, class instances, or DOM nodes to survive — functions throw DataCloneError and class instances come back as plain objects (prototype lost). Also, structuredClone is a deep clone; for a shallow copy use spread or Object.assign.
JSON Advanced Data & Collections 14 days since review
JSON.stringify takes two optional args: a replacer (function or allow-list array) and a space for pretty-printing. JSON.parse takes a reviver to transform values on the way in. A value's toJSON() method is called automatically (that is how Date serializes). undefined, functions, and symbols are dropped from objects and become null in arrays.
JavaScript
const data = { pw: 'secret', name: 'Ada', created: new Date(0) };

// replacer as allow-list + 2-space indent:
console.log(JSON.stringify(data, ['name', 'created'], 2));

// reviver: revive ISO strings back into Date objects
const text = '{"created":"1970-01-01T00:00:00.000Z"}';
const obj = JSON.parse(text, (k, v) =>
  k === 'created' ? new Date(v) : v);
console.log(obj.created instanceof Date);    // true
Challenge:
Serialize an object holding a Map and a BigInt (which JSON.stringify can't handle natively) by writing a replacer that encodes them into tagged objects, and a matching reviver that reconstructs them — a mini custom-serialization round-trip.
#1 Pitfall: JSON.stringify(BigInt(1)) throws a TypeError, and undefined/functions silently vanish (or become null inside arrays). Also, keys are visited depth-first and a toJSON method overrides the replacer's view of that value.
Error Handling Patterns Modules & Patterns 14 days since review
Prefer throwing Error subclasses (never strings) so stacks and instanceof work. Custom errors extend Error and set this.name. In async code, an unhandled rejection is the async analog of an uncaught throw. Centralize handling: let low-level code throw typed errors and let a boundary decide how to respond. Use { cause } to wrap without losing the root.
JavaScript
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = 'ValidationError';    // so instanceof + name are correct
    this.field = field;
  }
}

async function handler(req) {
  try {
    return await process(req);
  } catch (e) {
    if (e instanceof ValidationError) return { status: 400, field: e.field };
    throw new Error('processing failed', { cause: e });   // wrap unknowns
  }
}
Challenge:
Design a small error hierarchy (AppError → NotFoundError, ValidationError) and a single boundary function that maps each type to an HTTP status, wrapping anything unrecognized with { cause }. Show that instanceof AppError still catches the subclasses.
#1 Pitfall: throw 'oops' — a thrown string has no stack and fails instanceof. Also, an async function that isn't awaited swallows its rejection (unhandled), and a bare catch (e) {} that discards the error hides real failures.
Regular Expressions Advanced Data & Collections 13 days since review
Modern regex features: named groups (?<year>\d{4}) read via match.groups; lookbehind (?<=\$)\d+ and negative (?<!…); the s (dotAll) and u/v (unicode) flags; and String.prototype.matchAll for every match with groups. The g flag makes a regex stateful via lastIndex — a classic bug source when reused.
JavaScript
const re = /(?<y>\d{4})-(?<m>\d{2})/g;
const text = '2026-07 and 2025-01';

for (const m of text.matchAll(re)) {
  console.log(m.groups.y, m.groups.m);       // 2026 07 / 2025 01
}

// lookbehind: dollar amounts without capturing the $
console.log('$42 and $9'.match(/(?<=\$)\d+/g));   // ['42','9']
Challenge:
Write a template interpolator that replaces {{name}} tokens using named groups and replaceAll with a replacer function, then explain the lastIndex pitfall by reusing one /g regex across two .test() calls and observing the alternating result.
#1 Pitfall: Reusing a /g (or /y) regex literal across calls — lastIndex persists, so test()/exec() alternate true/false. Also, building regexes from untrusted input invites ReDoS (catastrophic backtracking); escape or bound the input.
Design Patterns: Observer, Strategy, Command Modules & Patterns 13 days since review
Observer: subjects notify a list of subscribers on change (the basis of event systems and reactivity). Strategy: swap an algorithm at runtime by passing a function/object — replaces sprawling if/else or switch. Command: encapsulate an action as an object with execute()/undo(), enabling queues, macros, and undo stacks. In JS, first-class functions make these lightweight.
JavaScript
// Strategy: pick behavior by key, no branching in the caller
const shipping = {
  standard: (w) => w * 1.0,
  express:  (w) => w * 2.5,
};
const cost = (method, w) => shipping[method](w);

// Command with undo:
const history = [];
function run(cmd) { cmd.execute(); history.push(cmd); }
function undo() { history.pop()?.undo(); }
Challenge:
Implement a tiny drawing app command stack: AddShapeCommand with execute/undo, a run/undo controller, and an Observer that redraws a status label whenever the stack changes. Show that undo restores exactly the previous state.
#1 Pitfall: Observer memory leaks — subscribers that never unsubscribe keep subjects (and their closures) alive; always return/expose an unsubscribe. Also, overusing patterns where a plain function or object literal would do adds ceremony without benefit.
Functional Patterns: Compose, Pipe, Curry Modules & Patterns 13 days since review
compose(f, g)(x) = f(g(x)) (right-to-left); pipe is the same left-to-right and usually reads better. Curry turns f(a, b, c) into f(a)(b)(c), enabling partial application and point-free style. These rely on pure, unary-ish functions; keep side effects at the edges so transformations compose predictably.
JavaScript
const pipe = (...fns) => (x) => fns.reduce((acc, f) => f(acc), x);
const compose = (...fns) => (x) => fns.reduceRight((acc, f) => f(acc), x);

const clean = pipe(
  (s) => s.trim(),
  (s) => s.toLowerCase(),
  (s) => s.replaceAll(/\s+/g, '-'),
);
console.log(clean('  Hello  World '));       // 'hello-world'

const add = (a) => (b) => a + b;             // curried
console.log([1, 2, 3].map(add(10)));         // [11, 12, 13]
Challenge:
Write a generic curry(fn) that supports both f(1)(2)(3) and f(1, 2)(3) by checking fn.length, then build a data-cleaning pipe of four small pure functions and prove compose of the reversed list produces the identical result.
#1 Pitfall: Composing functions with side effects or wrong arity — reduce passes a single accumulator, so multi-arg stages break. Also mind order: compose runs right-to-left, pipe left-to-right; swapping them silently reverses your transformation.

🆕 Numbers, Strings & Equality

Three foundations people think they know: exact big integers, Unicode-correct strings, and the equality that === gets subtly wrong.

BigInt & Numeric Separators Core Language

JS numbers are IEEE-754 doubles, so integers above 2^53 silently lose precision — fatal for database IDs, timestamps in nanoseconds, or cryptographic values. BigInt is a separate primitive for arbitrary-precision integers, and numeric separators (1_000_000) make large literals readable without changing their value.

1Past Number.MAX_SAFE_INTEGER, doubles lie. Create a BigInt with the n suffix or BigInt(x). It holds exact integers of any size.

JavaScript
console.log(Number.MAX_SAFE_INTEGER);        // 9007199254740991
console.log(9007199254740993 === 9007199254740992);  // true (!) — precision lost

const big = 9007199254740993n;               // trailing n = BigInt literal
console.log(big + 2n);                        // 9007199254740995n — exact
console.log(BigInt(Number.MAX_SAFE_INTEGER) + 2n);   // 9007199254740993n
9007199254740991 true 9007199254740995n 9007199254740993n

2BigInt and Number don't mix in arithmetic. You cannot add a BigInt to a Number — convert explicitly. Division truncates toward zero (integer math, no fractions).

JavaScript
// console.log(1n + 1);        // TypeError: Cannot mix BigInt and other types
console.log(1n + BigInt(1));   // 2n
console.log(Number(10n) + 1);  // 11

console.log(7n / 2n);          // 3n  — truncated, NOT 3.5
console.log(typeof 10n);       // 'bigint'
console.log(10n == 10);        // true  (loose) ...
console.log(10n === 10);       // false (strict — different types)
2n 11 3n bigint true false

3Numeric separators improve any numeric literal. Underscores group digits in decimal, hex, binary, and BigInt literals; they are purely cosmetic and stripped by the parser.

JavaScript
const budget = 1_000_000;              // === 1000000
const bytes  = 0xFF_FF_FF_FF;          // hex grouping
const card   = 1234_5678_9012_3456n;   // BigInt with separators
console.log(budget, bytes);            // 1000000 4294967295
// invalid placements throw at parse time:
// 1_000_  /  _1000  /  1__000  → SyntaxError
1000000 4294967295
Common Mistakes:
❌ Mixing types or JSON-serializing a BigInt: 1n + 1 // TypeError: Cannot mix BigInt and other types JSON.stringify({ id: 5n }) // TypeError: Do not know how to serialize a BigInt BigInt(0.5) // RangeError — not an integer
✅ Convert explicitly and serialize as a string: 1n + BigInt(1) // 2n JSON.stringify({ id: String(5n) }) // {"id":"5"} const back = BigInt("5") // parse it back
Key Rules:
  • BigInt is a distinct primitive (typeof x === 'bigint') for exact integers; write literals with an n suffix or BigInt(value).
  • Never mix BigInt and Number in arithmetic or bitwise ops — convert one side; / truncates and there are no fractional BigInts.
  • Numeric separators (_) are legal between digits in any numeric literal and are ignored by the parser — no leading, trailing, or doubled underscores.
Practice Exercises:
Compute 50! (factorial) with BigInt and confirm the exact result differs from the Number version, which loses precision. Print both to show where they diverge.
Write safeParseId(str) that returns a BigInt for a numeric string and safely JSON-round-trips it as a string, proving no precision is lost for a 19-digit ID.
Implement a nanosecond timestamp counter using BigInt arithmetic, and demonstrate that the equivalent Number arithmetic collides two distinct timestamps above 2^53.
Connections: BigInt is why JSON.stringify throws and needs a replacer, and it interacts with Object.is and === type rules below. It also underpins node:crypto and Web Crypto big-number work you have seen.
String Unicode: normalize, isWellFormed & toWellFormed Data & Collections

JS strings are sequences of UTF-16 code units, not characters — so 'é' can be one code point or two, and .length counts units, not glyphs. Two visually identical strings can fail ===. normalize() (ES6) fixes canonical mismatches, and isWellFormed/toWellFormed (ES2024) detect and repair lone surrogates before they corrupt output.

1Visually equal, byte-unequal. 'café' can be composed (one é) or decomposed (e + combining accent). normalize('NFC') canonicalizes so comparisons and dedup work.

JavaScript
const composed   = 'café';        // é as one code point
const decomposed = 'café';       // e + combining acute accent
console.log(composed === decomposed);  // false
console.log(composed.length, decomposed.length);   // 4 5

console.log(composed.normalize('NFC') === decomposed.normalize('NFC')); // true
false 4 5 true

2Iterate by code point, not code unit. Bracket indexing and .length operate on UTF-16 units, splitting astral characters (emoji). Spread / for…of / Intl.Segmenter respect code points.

JavaScript
const s = '😀a';
console.log(s.length);          // 3 — emoji is a surrogate PAIR (2 units)
console.log(s[0]);              // '\uD83D' — half of the emoji (mojibake)
console.log([...s]);            // ['😀', 'a'] — spread splits by code point
console.log([...s].length);    // 2 — the real glyph count here
console.log(s.codePointAt(0).toString(16));   // '1f600'
3 '\uD83D' ['😀', 'a'] 2 '1f600'

3isWellFormed / toWellFormed guard lone surrogates. Slicing a string mid-surrogate leaves an unpaired half that corrupts JSON, URLs, and TextEncoder. Check with isWellFormed(); repair (to U+FFFD ) with toWellFormed().

JavaScript
const broken = '😀'.slice(0, 1);       // one lone high surrogate
console.log(broken.isWellFormed());    // false
console.log(broken.toWellFormed());    // '�' — replacement char
console.log('hi'.isWellFormed());      // true

// encodeURIComponent throws on lone surrogates — sanitize first:
console.log(encodeURIComponent(broken.toWellFormed()));   // '%EF%BF%BD'
false '�' true '%EF%BF%BD'
Common Mistakes:
❌ Treating .length and s[i] as characters, and comparing un-normalized text: 'café'.length // may be 4 or 5 depending on composition userInput === stored // fails on NFC/NFD mismatch bigString.slice(0, 100) // may cut an emoji in half → lone surrogate
✅ Normalize before comparing, iterate by code point, validate surrogates: a.normalize('NFC') === b.normalize('NFC') [...str] or Intl.Segmenter // real characters if (!s.isWellFormed()) s = s.toWellFormed()
Key Rules:
  • Strings are UTF-16 code units; .length and index access count units, so astral chars (emoji, many CJK) span two — iterate with spread/for…of for code points.
  • normalize(form) ('NFC' default is the usual choice) canonicalizes equivalent sequences so comparison, sorting, and dedup are reliable.
  • str.isWellFormed() returns false when a lone surrogate exists; str.toWellFormed() replaces each with U+FFFD — do this before encodeURIComponent, JSON, or byte encoding.
Practice Exercises:
Write sameText(a, b) that compares two strings for canonical equality using normalize('NFC'), and prove it returns true for composed vs decomposed 'ñ' while === returns false.
Implement glyphLength(str) that counts user-perceived characters (grapheme clusters) using Intl.Segmenter, and show it returns 1 for a flag emoji where .length returns 4.
Write safeTruncate(str, n) that cuts a string to n UTF-16 units but never leaves a lone surrogate — verify the result with isWellFormed() and repair with toWellFormed() when needed.
Connections: Well-formedness matters right before encoding text to bytes and for Intl.Segmenter (grapheme counting). Canonicalization pairs with locale-aware Intl.Collator sorting from the Intl API you studied.
Object.is & Value Comparison Core Language

JavaScript has four equality algorithms, and the two you reach for — == and === — both get two edge cases wrong: they say NaN !== NaN and -0 === +0. Object.is implements SameValue, the semantics that Map/Set keys, Array.includes, and React's re-render checks actually use.

1Object.is fixes the two === anomalies. It treats NaN as equal to itself and distinguishes -0 from +0 — otherwise it matches ===.

JavaScript
console.log(NaN === NaN);          // false
console.log(Object.is(NaN, NaN));  // true

console.log(-0 === +0);            // true
console.log(Object.is(-0, +0));    // false — distinguishes signed zero

console.log(Object.is(1, 1));      // true
console.log(Object.is({}, {}));    // false — different references
false true true false true false

2Collections use SameValueZero — a third algorithm. Map/Set keys and Array.includes treat NaN as equal (like Object.is) but -0 and +0 as the same (unlike Object.is). This is why includes can find NaN but indexOf cannot.

JavaScript
const arr = [NaN, -0];
console.log(arr.indexOf(NaN));      // -1  (indexOf uses ===)
console.log(arr.includes(NaN));     // true (includes uses SameValueZero)

const set = new Set([NaN, NaN, 0, -0]);
console.log(set.size);              // 2  — NaN deduped, -0 and +0 merged
console.log(new Map().set(-0, 'x').get(+0));   // 'x' — keys use SameValueZero
-1 true 2 x

3Pick the algorithm deliberately. Use === for everyday checks, Object.is when signed zero or NaN identity matters (e.g. change detection), and remember collections silently apply SameValueZero for you.

JavaScript
// change detection that must treat NaN→NaN as "no change":
function changed(a, b) { return !Object.is(a, b); }
console.log(changed(NaN, NaN));    // false — correctly "unchanged"
console.log(changed(0, -0));       // true  — a real sign flip is a change

// avoid == entirely — its coercions are the fourth (loose) algorithm:
console.log(0 == '');              // true (!)   0 == false, '' == false ...
console.log(Object.is(0, ''));     // false
false true true false
Common Mistakes:
❌ Using === for NaN identity or ignoring signed zero: x === NaN // ALWAYS false — can never match Object.is(-0, 0) // false — surprising if you expected true arr.indexOf(NaN) // -1 — indexOf can't find NaN
✅ Match the algorithm to the need: Number.isNaN(x) || Object.is(x, NaN) // detect NaN arr.includes(NaN) // SameValueZero finds it === for normal comparisons; avoid ==
Key Rules:
  • Object.is = SameValue: like === but Object.is(NaN, NaN) is true and Object.is(-0, +0) is false.
  • SameValueZero (used by Map/Set keys, Array.includes, Array.from dedup) equals Object.is except it treats -0 and +0 as the same.
  • Objects compare by reference under every algorithm — Object.is is not a deep or structural equality check.
Practice Exercises:
Write a table that, for the inputs (NaN, NaN), (-0, +0), (0, ''), and ({}, {}), prints the result of ==, ===, and Object.is side by side, and explain each divergence.
Implement dedupe(arr) using a Set and confirm it collapses duplicate NaNs (which a naive indexOf-based dedupe cannot), then note how it treats -0 vs +0.
Build a shallow-equal helper shallowEqual(a, b) for flat objects that compares each value with Object.is, and show why using it for change detection avoids false "changed" on NaN fields.
Connections: SameValueZero is the key-identity rule behind Map & Set membership, and Object.is pairs with the BigInt/Number type distinctions above. Deep equality (beyond references) is what structured clone and JSON round-trips address.

🆕 Collections & File Data

Flattening nested arrays cleanly, and the browser's primitives for handling raw file and binary payloads.

Array flat & flatMap Data & Collections

Flattening nested arrays used to mean [].concat(...arr) (one level only) or a reduce/recursion hand-roll. ES2019's flat(depth) and flatMap(fn) do it declaratively — and flatMap is the idiomatic way to map-and-filter in a single pass, since returning [] drops an element.

1flat(depth = 1) flattens nested arrays. Pass a depth, or Infinity to fully flatten. It also removes empty slots in sparse arrays.

JavaScript
console.log([1, [2, 3], [4, [5]]].flat());          // [1, 2, 3, 4, [5]]
console.log([1, [2, [3, [4]]]].flat(2));            // [1, 2, 3, [4]]
console.log([1, [2, [3, [4]]]].flat(Infinity));     // [1, 2, 3, 4]

// flat drops holes in sparse arrays:
console.log([1, , 3].flat());                        // [1, 3]
[1, 2, 3, 4, [5]] [1, 2, 3, [4]] [1, 2, 3, 4] [1, 3]

2flatMap = map then flat(1), more efficiently. The callback returns an array for each element, and the results are flattened one level. Return a multi-element array to expand, or [] to remove.

JavaScript
// expand: split each sentence into words
console.log(['a b', 'c'].flatMap((s) => s.split(' ')));   // ['a','b','c']

// map + filter in one pass: keep evens, doubled; drop odds with []
console.log([1, 2, 3, 4].flatMap((n) => n % 2 ? [] : [n * 2]));   // [4, 8]

// only ONE level of flattening — nested arrays survive:
console.log([1, 2].flatMap((n) => [[n]]));                // [[1], [2]]
['a', 'b', 'c'] [4, 8] [[1], [2]]

3Non-mutating and skips holes. Both return new arrays and leave the source untouched, and their callbacks skip empty slots — handy for building pairs, adjacency lists, or Cartesian-style expansions.

JavaScript
const nums = [1, 2, 3];
// pair every element with a label (expand 1 → many):
console.log(nums.flatMap((n) => [`${n}a`, `${n}b`]));
//   ['1a','1b','2a','2b','3a','3b']

const src = [1, 2];
src.flatMap((n) => [n, n]);
console.log(src);                     // [1, 2] — original unchanged
['1a', '1b', '2a', '2b', '3a', '3b'] [1, 2]
Common Mistakes:
❌ Expecting deep flattening or chaining map().flat() needlessly: [1,[2,[3]]].flat() // only ONE level → [1, 2, [3]] arr.map(fn).flat() // two passes; flatMap does it in one [1,2].flatMap(n => [[n]]) // still nested — flatMap flattens only depth 1
✅ Use the right depth and flatMap for map+filter: nested.flat(Infinity) // fully flatten arr.flatMap(fn) // single pass arr.flatMap(x => cond(x) ? [x] : []) // filter + map together
Key Rules:
  • flat(depth) flattens depth levels (default 1, use Infinity for all) and removes holes; it returns a new array.
  • flatMap(fn) equals map(fn).flat(1) — flattening exactly one level — so return [] to drop and a multi-element array to expand.
  • Both are non-mutating and skip empty slots in sparse arrays; neither flattens beyond their stated depth.
Practice Exercises:
Write flattenDeep(arr) two ways — once with flat(Infinity) and once with a manual recursive reducer — and confirm they produce identical output on a 4-level-nested array.
Use flatMap to implement compactMap(arr, fn) that maps each element but omits any for which fn returns null/undefined, in a single pass.
Given an array of { name, tags } objects, use flatMap to produce a flat, de-duplicated list of all tags, then explain why map alone would leave a nested array.
Connections: flatMap is the eager array analog of the lazy Iterator.prototype.flatMap from Iterator Helpers, and its map+filter power complements reduce and the copying array methods (toSorted, with).
Blob, File & FileReader Browser APIs & DOM

Any time you handle an upload, generate a download, or move binary data around the browser, you meet Blob (immutable raw bytes + a MIME type), File (a Blob with a name and modified time), and the ways to read them. Modern async Blob methods largely replace the old event-based FileReader, but you still meet both.

1A Blob is immutable bytes with a type; a File adds metadata. Build one from strings, ArrayBuffers, or other blobs. <input type="file"> gives you File objects (which are Blobs).

JavaScript
const blob = new Blob(['hello, ', 'world'], { type: 'text/plain' });
console.log(blob.size, blob.type);          // 12 'text/plain'

const file = new File([blob], 'greeting.txt', { type: 'text/plain' });
console.log(file.name, file instanceof Blob);   // 'greeting.txt' true

// slice() is a cheap view — no copy — useful for chunked uploads:
console.log(await blob.slice(0, 5).text());     // 'hello'
12 'text/plain' 'greeting.txt' true 'hello'

2Prefer the promise-based Blob reader methods. blob.text(), blob.arrayBuffer(), and blob.bytes() (returns a Uint8Array) are async and far cleaner than FileReader. Use blob.stream() for large data.

JavaScript
const b = new Blob([JSON.stringify({ ok: true })], { type: 'application/json' });

const text = await b.text();
console.log(JSON.parse(text).ok);           // true

const buf = await b.arrayBuffer();
console.log(new Uint8Array(buf)[0]);        // 123  → '{'  (byte value)

// stream large blobs chunk by chunk:
const reader = b.stream().getReader();
const { value } = await reader.read();      // first Uint8Array chunk
true 123

3FileReader is the legacy event-based path — still needed for data URLs. readAsDataURL yields a data: URI (great for image previews) that the newer methods don't produce directly. For downloads/previews, prefer URL.createObjectURL and revoke it after.

JavaScript
// FileReader: callback/event based (older API)
const fr = new FileReader();
fr.onload = () => console.log(fr.result.slice(0, 30));  // 'data:text/plain;base64,aGVsbG8...'
fr.readAsDataURL(new Blob(['hello'], { type: 'text/plain' }));

// trigger a download without a server round-trip:
const url = URL.createObjectURL(new Blob(['report'], { type: 'text/plain' }));
const a = Object.assign(document.createElement('a'), { href: url, download: 'r.txt' });
a.click();
URL.revokeObjectURL(url);                   // free the memory when done
data:text/plain;base64,aGVsbG8=
Common Mistakes:
❌ Reaching for FileReader for everything, and leaking object URLs: new FileReader() ... onload ... // verbose when blob.text() would do URL.createObjectURL(blob) // never revoked → memory leak blob.text = '...' // Blobs are IMMUTABLE — can't mutate
✅ Use async Blob methods; always revoke object URLs: const s = await blob.text() // promise-based const buf = await file.arrayBuffer() // binary const u = URL.createObjectURL(blob); ...; URL.revokeObjectURL(u)
Key Rules:
  • Blob is immutable raw bytes + a MIME type; File extends it with name/lastModified. slice() makes a cheap, copy-free view.
  • Prefer async blob.text()/arrayBuffer()/bytes()/stream() over FileReader; use FileReader.readAsDataURL only when you specifically need a data: URI.
  • URL.createObjectURL(blob) creates a temporary URL you must revokeObjectURL to avoid leaks; object URLs are per-document.
Practice Exercises:
Wire up an <input type="file"> that reads the selected image with file.arrayBuffer(), logs its byte length and MIME type, and shows a preview via URL.createObjectURL — remembering to revoke the URL on the image's load.
Generate a CSV string in memory, wrap it in a Blob of type text/csv, and trigger a client-side download with a temporary <a download> — no server involved.
Split a large Blob into 64KB chunks with slice() and read each chunk's bytes with await chunk.arrayBuffer(), reconstructing the original and verifying the byte length matches — the basis of resumable uploads.
Connections: Blob.arrayBuffer() hands you the typed-array/ArrayBuffer world, and blob.stream() is a ReadableStream like the Streams API. Encoding a well-formed string to bytes (TextEncoder) is the counterpart of decoding a Blob back to text.
