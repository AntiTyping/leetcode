Console Advanced & Debugging Testing & Quality 9 days since review
The console object is far more than .log: .table renders arrays/objects as a grid, .group/.groupEnd nest output, .dir shows an object's property tree, .assert(cond, ...) logs only when the condition is false, .count/.countReset tally labels, .time/.timeEnd measure spans, and %c applies CSS in browsers. In Node, .error/.warn go to stderr, the rest to stdout.
JavaScript
console.table([{ id: 1, name: 'ada' }, { id: 2, name: 'lin' }]);

console.group('request');
console.log('method', 'GET');
console.assert(1 + 1 === 3, 'math broke');   // logs an Assertion failed
console.groupEnd();

console.time('work');
for (let i = 0; i < 1e6; i++) {}
console.timeEnd('work');                     // work: 1.2ms

for (const x of ['a', 'a', 'b']) console.count(x);  // a:1 a:2 b:1
Challenge:
Write a trace(label, fn) helper that wraps a synchronous call with console.group(label), logs the arguments and return value inside the group, times it with console.time/timeEnd, and uses console.assert to flag when the result is undefined. Make nested trace calls indent correctly.
#1 Pitfall: Logging an object logs a live reference — when you expand it later in DevTools you see its current state, not the state at log time. To snapshot, log structuredClone(obj) or JSON.parse(JSON.stringify(obj)). Also, leaving console.log in hot loops is a real performance and memory cost.
Regular Expressions Advanced Data & Collections 8 days since review
Beyond the basics: named groups (?<year>\d{4}) surface on match.groups; lookbehind (?<=\$) / (?<!\$) mirror lookahead; the s (dotAll) flag lets . match newlines; the u flag enables \p{...} Unicode property escapes; the sticky y flag anchors at lastIndex; and str.matchAll(/.../g) yields every match with its groups.
JavaScript
const re = /(?<y>\d{4})-(?<m>\d{2})-(?<d>\d{2})/;
const { groups } = '2026-06-26'.match(re);
console.log(groups.y, groups.m, groups.d);    // 2026 06 26

const price = '$42 and €9'.match(/(?<=\$)\d+/);
console.log(price[0]);                         // 42  (only the dollar amount)

for (const m of 'a1b2c3'.matchAll(/(?<ch>[a-z])(?<n>\d)/g)) {
  console.log(m.groups.ch, m.groups.n);
}
console.log(/\p{Emoji_Presentation}/u.test('🎯'));  // true
Challenge:
Write template(str, data) that replaces {{name}} placeholders using a named-group regex and replaceAll with a replacer function, leaving unknown keys untouched. Then write a tokenizer that uses a single sticky (y) regex in a loop over lastIndex to split "12 + 34 * 5" into number/operator tokens.
#1 Pitfall: A regex with the g or y flag is stateful — it carries lastIndex between calls. Reusing one shared /.../g across .test/.exec calls gives intermittent, position-dependent failures. Create a fresh regex per scan, or reset lastIndex = 0.
Design Patterns: Observer, Strategy, Command Modules & Patterns 8 days since review
Observer: a subject keeps a list of subscribers and notifies them on change (push). Strategy: swap an algorithm at runtime by passing a function/object that implements a shared interface. Command: wrap an action (and its inverse) as an object so it can be queued, logged, undone, or replayed.
JavaScript
// Strategy: behavior chosen at call time
const sorters = {
  asc:  (a, b) => a - b,
  desc: (a, b) => b - a,
};
const sort = (arr, how) => [...arr].sort(sorters[how]);
console.log(sort([3, 1, 2], 'desc'));   // [3, 2, 1]

// Command: action + undo, ready to queue
function makeAdd(list, item) {
  return { do: () => list.push(item),
           undo: () => { list.splice(list.indexOf(item), 1); } };
}
const xs = [], cmd = makeAdd(xs, 'a');
cmd.do(); console.log(xs);   // ['a']
cmd.undo(); console.log(xs); // []
Challenge:
Build an Observable with subscribe(fn) (returns an unsubscribe function) and set(value) that notifies all subscribers. Then build a CommandStack with execute(cmd), undo(), and redo() backed by two arrays, where each cmd has do/undo. Wire them together so undoing a command re-notifies observers.
#1 Pitfall: Observer leaks. If subscribe doesn't return (and callers don't call) an unsubscribe, dead listeners pile up — especially in UIs where components mount/unmount. Always hand back a disposer, and remove listeners on teardown.
Memory Management & Garbage Collection Performance & Security 8 days since review
JS uses a tracing GC: objects reachable from roots (the global object, the current call stack, active closures) survive; everything else is reclaimed (mark-and-sweep, generational in V8). You don't free memory — you make it unreachable. The classic leaks: detached DOM nodes still referenced by JS, forgotten setIntervals, closures capturing a large scope, and unbounded caches / module-level arrays that only grow.
JavaScript
// LEAK: every call keeps `huge` alive via the returned closure
function makeLeaky() {
  const huge = new Array(1e6).fill('*');
  return () => huge.length;        // closure pins the whole array
}

// FIX: capture only what you need
function makeLean() {
  const huge = new Array(1e6).fill('*');
  const len = huge.length;         // keep the number, drop the array
  return () => len;
}

// WeakMap: entries vanish when the key object is collected
const meta = new WeakMap();
let node = { id: 1 };
meta.set(node, { seen: Date.now() });
node = null;                       // entry becomes eligible for GC
Challenge:
Identify and fix the leak in a "cache" implemented as a plain Map keyed by request objects that is never pruned. Rewrite it two ways: (a) an LRU with a max size that evicts the oldest entry, and (b) a WeakMap keyed by the request object so entries drop automatically. Explain which one you can iterate and why.
#1 Pitfall: Closures that capture far more than they use. A handler returned from a big function pins every variable in that scope, not just the ones it references in obvious cases — and a single retained closure can hold megabytes. Null out large locals you no longer need, or factor them out of the closed-over scope.
V8 Internals: Hidden Classes, Inline Caching & JIT Engine Internals 8 days since review
V8 gives each object a hidden class (a "shape"/Map) describing its property layout. Objects built with the same properties in the same order share a shape, so property access compiles to a fixed offset. A call site that always sees one shape is monomorphic and gets an inline cache; seeing many shapes makes it polymorphic/megamorphic and slow. Small integers (SMIs) are tagged inline; mixing types or changing shapes triggers deoptimization.
JavaScript
// SAME shape: both objects share one hidden class -> fast property access
function Point(x, y) { this.x = x; this.y = y; }
const a = new Point(1, 2);
const b = new Point(3, 4);

// SHAPE DIVERGENCE: adding a property later forks the hidden class
const c = new Point(5, 6);
c.z = 7;                 // c now has a different hidden class than a, b

// DEOPT: a function specialized for numbers gets a string
function add(p) { return p.x + p.y; }
add(a); add(b);          // optimized for the Point shape (monomorphic)
add({ x: '1', y: '2' }); // different shape + string add -> deopt
Challenge:
Write two versions of a function that builds 1,000,000 record objects: one that initializes all fields in the constructor in a fixed order, and one that conditionally adds fields afterward (so shapes diverge). Sum a numeric field across each set and compare timings with performance.now(). Explain the gap in terms of hidden classes and inline caches.
#1 Pitfall: Treating objects like dictionaries — adding properties in varying orders, or using delete — forces V8 out of fast shapes into slow dictionary mode and deoptimizes call sites that touch them. Initialize every property up front (even to null) and keep types stable per property.
Node.js child_process: exec, spawn & fork Node.js Runtime 8 days since review
exec runs a command through a shell and buffers all output into a callback (handy for short output, dangerous with untrusted input). execFile runs a binary directly with an args array — no shell. spawn streams stdout/stderr, ideal for large or long-running output. fork is spawn specialized for a child Node process with an IPC channel (child.send / process.on('message')).
JavaScript
import { spawn, execFile } from 'node:child_process';

// stream output line by line, no shell:
const ls = spawn('ls', ['-la'], { shell: false });
ls.stdout.on('data', (chunk) => process.stdout.write(chunk));
ls.on('close', (code) => console.log('exit', code));

// safe: args passed as an array, never concatenated into a shell string:
execFile('git', ['log', '-1', '--format=%H'], (err, stdout) => {
  if (!err) console.log(stdout.trim());
});
Challenge:
Write run(cmd, args) that wraps spawn in a Promise resolving to { code, stdout, stderr }, collecting the streamed chunks. Then build a parent/child pair using fork where the parent sends numbers and the child replies with their squares over IPC, and the parent resolves once it has all answers.
#1 Pitfall: Shell injection. exec(`convert ${userInput} out.png`) hands user input to a shell — input like ; rm -rf ~ executes. Prefer execFile/spawn with an args array and shell: false so arguments are never re-parsed by a shell.

🆕 Newer Array, Object & String Methods

Recent additions (ES2022–ES2024) that replace error-prone idioms with safe, intention-revealing one-liners.

findLast & findLastIndex Data & Collections

Searching an array from the end used to mean reverse() (which mutates, or forces a copy) plus an index flip, or a manual backwards for loop. ES2023's findLast/findLastIndex scan from the right directly — no copy, no mutation, and they short-circuit on the first match from the end.

1Find the last matching element / its index. Same callback signature as find/findIndex, but iteration starts at the last element.

JavaScript
const nums = [1, 8, 3, 10, 5, 6];

console.log(nums.findLast(n => n % 2 === 0));      // 6  (last even)
console.log(nums.findLastIndex(n => n % 2 === 0)); // 5
console.log(nums.findLast(n => n > 100));          // undefined
console.log(nums.findLastIndex(n => n > 100));     // -1
6 5 undefined -1

2Why not filter().pop(). filter walks the whole array and builds an intermediate array just to take one element; findLast stops at the first hit from the end and allocates nothing.

JavaScript
const log = [
  { level: 'info',  msg: 'start' },
  { level: 'error', msg: 'boom' },
  { level: 'info',  msg: 'retry' },
  { level: 'error', msg: 'gave up' },
];

// the most recent error, without copying or scanning past it:
const lastError = log.findLast(e => e.level === 'error');
console.log(lastError.msg);        // 'gave up'

// also available on typed arrays:
console.log(Int8Array.from([5, 4, 9, 2]).findLastIndex(n => n > 3));  // 2
gave up 2
Common Mistakes:
❌ Reversing to search from the end — mutates and flips indices: const i = arr.reverse().findIndex(pred); // arr is now reversed in place! const realIndex = arr.length - 1 - i; // easy to get wrong
✅ Search from the end directly, no mutation: const item = arr.findLast(pred); const i = arr.findLastIndex(pred); // -1 if none, like findIndex
Key Rules:
  • Identical callback to find/findIndex — (element, index, array) — only the direction changes (right to left).
  • findLast returns the element or undefined; findLastIndex returns the index or -1.
  • Available on Array and all TypedArrays. Prefer it over reverse() tricks or filter().pop() when you want a single match from the end.
Practice Exercises:
Given an array of { ts, status } events sorted by time, write lastSuccessBefore(events, cutoff) that returns the most recent event with status === 'ok' and ts < cutoff, using findLast.
Implement trimTrailing(arr, pred) that removes trailing elements matching pred (e.g. trailing zeros) by finding the last non-matching index with findLastIndex and slicing — without reversing the array.
Write your own findLast polyfill with a single backwards for loop and confirm it matches the native method (including the undefined result and early exit) on a few inputs.
Connections: These round out the find/findIndex family and complement the copy-based array methods (toSorted / toReversed / with) from a recent guide. Reaching from the end relates to relative indexing with at(-1).
Object.hasOwn Core Language

obj.hasOwnProperty(key) is unsafe: it breaks on objects with a null prototype (no method to call) and on objects that shadow hasOwnProperty with their own property. The defensive idiom Object.prototype.hasOwnProperty.call(obj, key) is verbose and easy to forget. ES2022's Object.hasOwn(obj, key) is the safe, readable replacement.

1The safe own-property check. A static method, so the object never needs to inherit anything.

JavaScript
const o = { a: 1, b: undefined };

console.log(Object.hasOwn(o, 'a'));   // true
console.log(Object.hasOwn(o, 'b'));   // true  (own, even though value is undefined)
console.log(Object.hasOwn(o, 'c'));   // false
console.log(Object.hasOwn(o, 'toString'));  // false (inherited, not own)
true true false false

2Works where hasOwnProperty throws or lies. Null-prototype objects (like the result of Object.groupBy) and objects that override the method name are handled correctly.

JavaScript
const bag = Object.create(null);   // no prototype, no methods
bag.x = 1;
// bag.hasOwnProperty('x');        // TypeError: not a function
console.log(Object.hasOwn(bag, 'x'));   // true

const tricky = { hasOwnProperty: () => 'lol' };
console.log(tricky.hasOwnProperty('z'));   // 'lol'  (shadowed — wrong!)
console.log(Object.hasOwn(tricky, 'z'));   // false  (correct)
true lol false

3hasOwn vs the in operator. in also walks the prototype chain; Object.hasOwn checks only own properties — use the one that matches your intent.

JavaScript
const arr = [10, 20];
console.log('length' in arr);            // true  (inherited/own intrinsic)
console.log(Object.hasOwn(arr, 'push')); // false (push is on the prototype)
console.log('push' in arr);              // true  (found up the chain)
console.log(Object.hasOwn(arr, 0));      // true  (index 0 is an own property)
true false true true
Common Mistakes:
❌ Calling hasOwnProperty directly off an arbitrary object: if (data.hasOwnProperty('id')) ... // throws on null-proto, lies if shadowed
✅ Use the static, prototype-independent check: if (Object.hasOwn(data, 'id')) ... // safe regardless of prototype
Key Rules:
  • Object.hasOwn(obj, key) returns true only for own properties (including those whose value is undefined); inherited keys are false.
  • It is the drop-in replacement for Object.prototype.hasOwnProperty.call(obj, key) — use it for null-prototype objects and untrusted data.
  • Use in when you intentionally want to include inherited properties; use hasOwn when you mean "is this the object's own key".
Practice Exercises:
Write pick(obj, keys) that returns a new object containing only the keys that are own properties of obj, using Object.hasOwn so inherited keys are never copied.
Given a null-prototype config object from Object.groupBy, write countKeys(o) that safely reports how many of a candidate list of keys are present, and show that o.hasOwnProperty would have thrown.
Demonstrate the difference between Object.hasOwn(obj, 'x') and 'x' in obj for a value that is undefined versus genuinely absent versus inherited from the prototype.
Connections: The null-prototype gotcha is the same one from Object.groupBy's result and ties back to prototypes & the prototype chain. It pairs with Object.keys/entries when iterating untrusted data safely.
Well-Formed Strings: isWellFormed & toWellFormed Core Language

JS strings are UTF-16 and can contain lone surrogates — half of a surrogate pair with no partner. Those break encodeURIComponent (it throws), corrupt fetch bodies, and fail to clone across postMessage. ES2024's String.prototype.isWellFormed detects them and toWellFormed sanitizes them, so you can guard the boundary before encoding or sending data over the wire.

1Detect lone surrogates. isWellFormed() returns false if the string contains an unpaired surrogate (code units in 0xD800–0xDFFF without a partner).

JavaScript
const ok   = 'hello 🎯';            // 🎯 is a valid surrogate PAIR
const bad  = 'hi \uD83D';           // a lone high surrogate

console.log(ok.isWellFormed());     // true
console.log(bad.isWellFormed());    // false
true false

2Sanitize with toWellFormed. It returns a copy with every lone surrogate replaced by the Unicode replacement character U+FFFD (�), leaving valid text untouched.

JavaScript
const bad = 'price: 100\uD83D';
const fixed = bad.toWellFormed();
console.log(fixed);                 // "price: 100�"
console.log(fixed.isWellFormed());  // true
price: 100� true

3Guard the boundary. encodeURIComponent throws on lone surrogates; sanitize (or reject) first so a single bad byte from user input can't crash your request building.

JavaScript
const raw = 'q=\uD83D';

try {
  encodeURIComponent(raw);          // throws URIError
} catch (e) {
  console.log(e.name);              // URIError
}

// safe: clean it first
console.log(encodeURIComponent(raw.toWellFormed()));  // q%3D%EF%BF%BD
URIError q%3D%EF%BF%BD
Common Mistakes:
❌ Assuming any JS string is safe to encode or send: const url = base + '?q=' + encodeURIComponent(userText); // throws on lone surrogate
✅ Validate or sanitize at the boundary: if (!userText.isWellFormed()) userText = userText.toWellFormed(); const url = base + '?q=' + encodeURIComponent(userText);
Key Rules:
  • isWellFormed() → boolean; toWellFormed() → a new string with lone surrogates replaced by U+FFFD (never mutates).
  • Run them at I/O boundaries: before encodeURIComponent, building request bodies, postMessage/structuredClone, or persisting text.
  • Valid surrogate pairs (real emoji/astral characters) are well-formed and pass through unchanged — only unpaired halves are flagged.
Practice Exercises:
Write safeQuery(params) that builds a query string from an object, sanitizing each value with toWellFormed before encodeURIComponent, and returns both the string and a list of keys that needed fixing.
Construct a string with a lone low surrogate (\uDC00) and a valid pair, confirm isWellFormed is false, then verify toWellFormed fixes exactly one position and leaves the pair intact.
Without using isWellFormed, write a checker that scans charCodeAt values to detect an unpaired surrogate, then validate it against the native method on several inputs.
Connections: This complements JSON Advanced and structured-clone semantics (lone surrogates also trip up serialization) and the Unicode-aware side of regular expressions (\p{...}, the u flag).

🆕 Numbers, Globals & Modules

Two foundational features for correctness and module initialization: arbitrary-precision integers, and awaiting at module scope.

BigInt Core Language

JS Number is an IEEE-754 double, so it can only represent integers exactly up to 2^53 − 1 (Number.MAX_SAFE_INTEGER). Past that, arithmetic silently loses precision — a real problem for database bigint IDs, nanosecond timestamps, snowflake IDs, and big factorials. BigInt (ES2020) is an arbitrary-precision integer type that never loses digits.

1Where Number breaks, and the fix. Create a BigInt with the n literal suffix or the BigInt() function.

JavaScript
console.log(Number.MAX_SAFE_INTEGER);     // 9007199254740991
console.log(9007199254740991 + 2);        // 9007199254740992  ❌ wrong (should be ...993)

console.log(9007199254740991n + 2n);      // 9007199254740993n  ✅ exact
console.log(BigInt('123456789012345678901234567890') * 2n);
// 246913578024691357802469135780n
9007199254740991 9007199254740992 9007199254740993n 246913578024691357802469135780n

2No mixing with Number in arithmetic. 1n + 1 throws — convert explicitly. Division truncates toward zero (integers only), and there is no Math support.

JavaScript
// 1n + 1;                 // TypeError: Cannot mix BigInt and other types
console.log(1n + BigInt(1));     // 2n
console.log(Number(10n) + 1);    // 11

console.log(7n / 2n);            // 3n   (truncated, no fraction)
console.log(7n % 2n);            // 1n
console.log(typeof 10n);         // "bigint"
2n 11 3n 1n bigint

3Comparison vs serialization. Loose equality and ordering work across types (10n == 10), but strict equality does not (10n !== 10). JSON.stringify throws on BigInt — serialize as a string.

JavaScript
console.log(10n == 10);          // true   (loose, cross-type)
console.log(10n === 10);         // false  (different types)
console.log(5n < 10);            // true   (ordering is cross-type)

// JSON.stringify({ id: 10n });  // TypeError: Do not know how to serialize a BigInt
console.log(JSON.stringify({ id: 10n.toString() }));   // {"id":"10"}
true false true {"id":"10"}
Common Mistakes:
❌ Mixing types or JSON-stringifying a BigInt: total = bigCount + 1; // TypeError if bigCount is a BigInt JSON.stringify({ id: userId }); // TypeError when userId is 10n
✅ Convert at the boundary, serialize as string: total = bigCount + 1n; // stay in BigInt JSON.stringify({ id: userId.toString() }); // and parse with BigInt() on read
Key Rules:
  • Use BigInt for exact integers beyond 2^53 − 1 (DB ids, counters, crypto, nanosecond time). It is integer-only — no decimals, no Math.
  • Never mix BigInt and Number in arithmetic; convert with BigInt(x) or Number(x) first (mind precision loss going to Number).
  • ==/ordering are cross-type; === is not. JSON can't serialize BigInt — store/transmit it as a string.
Practice Exercises:
Write factorial(n) using BigInt and compute 50! exactly; confirm the same function with Number loses precision well before that.
Implement a jsonReplacer/jsonReviver pair so that objects containing BigInt round-trip through JSON.stringify/parse (e.g. encode as { "$bigint": "10" }).
Write parseId(value) that accepts a number or numeric string and returns a BigInt, throwing if the input would have lost precision as a Number (hint: compare against Number.MAX_SAFE_INTEGER).
Connections: BigInt is the precision counterpart to floating-point Number and underpins BigInt64Array in typed arrays. The non-serializability gotcha ties back to JSON Advanced's replacer/reviver hooks.
Top-Level await Async Mastery

Before ES2022, await was only legal inside an async function, so module initialization that needed async work (load config, connect a DB, conditional dynamic import) had to be wrapped in an IIFE — which meant the module's exports weren't ready when importers used them. Top-level await lets an ES module await directly at module scope, and importers transparently wait for it.

1Await at module scope. In an ES module (.mjs, or "type": "module"), you can await directly — the module becomes asynchronous.

JavaScript
// config.mjs
const res = await fetch('https://example.com/config.json');
export const config = await res.json();   // exports are ready only after this settles

// app.mjs
import { config } from './config.mjs';
console.log(config.featureFlag);          // importer waited for config.mjs to finish

2Conditional / dynamic dependencies. Pick a dependency at load time and await its dynamic import — impossible to express cleanly with static imports alone.

JavaScript
const strings = await import(
  navigator.language.startsWith('fr') ? './fr.mjs' : './en.mjs'
);
export const t = strings.translate;

// fall back to a polyfill only when needed:
if (!('structuredClone' in globalThis)) {
  await import('./structured-clone-polyfill.mjs');
}

3It blocks the module graph. A module with top-level await delays every module that depends on it until it resolves. That's the point — but it also means a slow top-level await stalls startup, and the feature only exists in modules.

JavaScript
// db.mjs — connection established once, before any importer runs
import { connect } from './driver.mjs';
export const db = await connect(process.env.DATABASE_URL);

// any module that imports { db } is guaranteed a live connection,
// with NO need for an async init() call or a "ready" promise to await later
Common Mistakes:
❌ Using top-level await in a CommonJS file or a classic script: // require()-based .js / <script> (no type=module): SyntaxError const data = await fetch(url);
✅ Use it only in ES modules; keep it fast, or defer heavy work: // .mjs / type=module / <script type="module"> export const data = await (await fetch(url)).json();
Key Rules:
  • Only valid in ES modules (.mjs, "type": "module", <script type="module">) — never in CommonJS or classic scripts.
  • A module's top-level await suspends every importer until it settles; the dependency graph awaits it for you (no manual "ready" promise).
  • Keep it short — heavy top-level awaits slow startup, and cyclic dependencies that both top-level-await each other can deadlock.
Practice Exercises:
Build a config.mjs that loads settings from a (mocked async) source with top-level await and exports them, plus an app.mjs that imports and uses them — confirm the importer sees fully-loaded config without any init call.
Write a module that conditionally await import()s one of two locale modules based on an environment value and re-exports a unified translate function.
Demonstrate the blocking behavior: create slow.mjs with await new Promise(r => setTimeout(r, 200)) before its export, import it from a module that logs timestamps, and observe that the importer's code runs only after the 200ms.
Connections: Top-level await builds on ES Modules evaluation order and the same promise machinery as async/await patterns. It pairs with dynamic import() and the globalThis feature-detection idiom shown above.
