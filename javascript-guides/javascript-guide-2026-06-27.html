Six meta-programming and iteration concepts are due for retrieval practice today. Recall first, verify second.

Property Descriptors & Object.defineProperty Meta-Programming 11 days since review
Every property has a descriptor: data properties carry value + writable, accessor properties carry get/set. All carry enumerable and configurable. Properties created by defineProperty default every flag to false — the opposite of a normal assignment.
JavaScript
const o = {};
Object.defineProperty(o, 'id', { value: 42, enumerable: false });
o.id = 99;                 // silently ignored (writable:false) in sloppy mode
console.log(o.id, Object.keys(o));  // 42 []  (hidden from enumeration)

let _t = 20;
Object.defineProperty(o, 'celsius', {
  get() { return _t; },
  set(v) { _t = v; },
  enumerable: true,
});
o.celsius = 100;
console.log(o.celsius);    // 100
console.log(Object.getOwnPropertyDescriptor(o, 'id'));
Challenge:
Write readonly(obj, key, value) that defines a non-writable, non-configurable, enumerable property and returns obj. Then write observable(obj, key) that replaces an existing data property with an accessor that logs every read and write while preserving the current value — without losing the original enumerability.
#1 Pitfall: Assuming a plain assignment and defineProperty behave the same. Assignment creates a writable/enumerable/configurable property; defineProperty defaults all three to false. Forgetting to set enumerable: true makes the property invisible to Object.keys, spread, and JSON.stringify.
Proxy & Reflect Meta-Programming 10 days since review
A Proxy wraps a target and intercepts fundamental operations via traps (get, set, has, deleteProperty, apply, ...). Inside a trap, forward the default behavior with the matching Reflect method so you don't reimplement it (and so receiver/getters work correctly).
JavaScript
const target = { a: 1 };
const logged = new Proxy(target, {
  get(t, key, recv) {
    console.log('get', key);
    return Reflect.get(t, key, recv);     // default behavior, preserves receiver
  },
  set(t, key, value, recv) {
    if (typeof value !== 'number') throw new TypeError(key + ' must be a number');
    return Reflect.set(t, key, value, recv);
  },
});
logged.a;          // logs "get a" -> 1
logged.a = 5;      // ok
// logged.a = 'x'; // throws TypeError
Challenge:
Build withDefault(obj, factory) returning a Proxy where reading a missing key lazily creates it via factory(key), stores it, and returns it (a defaultdict). Then extend it so 'has' and Object.keys still only report keys that were actually materialized.
#1 Pitfall: Returning the wrong invariant from a trap. The set trap must return a truthy value on success — returning undefined makes assignments throw in strict mode. Always return Reflect.set(...), which yields the correct boolean.
WeakRef & FinalizationRegistry Meta-Programming 9 days since review
A WeakRef holds an object without preventing its garbage collection; .deref() returns the object or undefined if it's been collected. A FinalizationRegistry fires a cleanup callback sometime after a registered object is collected. Both are best-effort and non-deterministic — never rely on timing.
JavaScript
let user = { id: 1 };
const ref = new WeakRef(user);

const registry = new FinalizationRegistry((heldValue) => {
  console.log('collected:', heldValue);   // runs at some later GC, maybe never
});
registry.register(user, 'user#1');

console.log(ref.deref()?.id);   // 1  (still alive)
user = null;                    // now eligible for GC
// later: ref.deref() -> undefined, registry callback may log "collected: user#1"
Challenge:
Implement a WeakCache for expensive-to-compute objects keyed by string, where values are held via WeakRef so the GC can reclaim them under memory pressure. On a miss or a dead ref, recompute. Use a FinalizationRegistry to delete the stale map entry once its value is collected, so the map doesn't leak empty refs.
#1 Pitfall: Using WeakRef/FinalizationRegistry for program logic. Collection timing is unspecified — the callback may run late, in a batch, or never (e.g. on page unload). Use them only for caches and cleanup hints, never for correctness-critical resource release (use using/try-finally for that).
Optional Chaining & Nullish Coalescing Functions & Iteration 8 days since review
?. short-circuits to undefined the moment the left side is null/undefined (works for property access, calls ?.(), and indexing ?.[i]). ?? falls back only on null/undefined — unlike ||, it keeps falsy-but-valid values like 0 and ''. ??= assigns only when the target is nullish.
JavaScript
const cfg = { timeout: 0, retries: null };

console.log(cfg.timeout || 3000);   // 3000  ❌ wrongly overrides a valid 0
console.log(cfg.timeout ?? 3000);   // 0     ✅ keeps the explicit 0
console.log(cfg.retries ?? 5);      // 5

const user = { profile: null };
console.log(user.profile?.name);            // undefined (no throw)
console.log(user.getName?.());              // undefined (method may not exist)
cfg.label ??= 'default';                    // sets only because label was missing
Challenge:
Write get(obj, path, fallback) where path is a dotted string (e.g. 'a.b.c'). Walk the path safely so a missing intermediate yields the fallback (using ?? semantics, so a stored 0/''/false is returned as-is, not replaced). Bonus: support array indices like 'items.0.id'.
#1 Pitfall: Combining ?? with ||/&& in one expression without parentheses — it's a syntax error by design (a ?? b || c won't parse). Also: ?. only guards the access immediately after it, so a?.b.c still throws if a.b is null. Chain every hop: a?.b?.c.
Async Generators & for await...of Functions & Iteration 6 days since review
An async function* can await between yields and produces an async iterable (implements Symbol.asyncIterator). Consume it with for await...of, which awaits each value. Ideal for paginated APIs and streams where each item arrives asynchronously and you want lazy, backpressured consumption.
JavaScript
async function* pages(url) {
  let next = url;
  while (next) {
    const res = await fetch(next);
    const { items, nextUrl } = await res.json();
    yield* items;            // delegate: yield each item
    next = nextUrl;          // stop when the API returns no next page
  }
}

for await (const item of pages('/api/items?page=1')) {
  console.log(item.id);      // processes one at a time, fetching pages lazily
  if (item.id === 'stop') break;   // break also closes the generator
}
Challenge:
Write an async generator take(asyncIterable, n) that yields at most the first n values from any async iterable and then stops (closing the source). Then write merge(...asyncIterables) that yields values from several async iterables as they arrive (not strictly in order), completing when all sources are exhausted.
#1 Pitfall: Reaching for .forEach(async ...) or Promise.all(map(...)) when you actually want sequential, backpressured iteration. for await...of awaits each turn before requesting the next value; Promise.all launches everything at once and buffers all results in memory.
Tagged Template Literals Functions & Iteration 6 days since review
A tag is a function called with the literal's static string parts (an array, with a .raw property) followed by each interpolated value as a separate argument. The tag decides how to combine them — the basis for safe HTML/SQL builders, i18n, styled-components, and DSLs.
JavaScript
function safe(strings, ...values) {
  const esc = (s) => String(s).replace(/[&<>"]/g,
    c => ({ '&': '&amp;', '<': '&lt;', '>': '&gt;', '"': '&quot;' }[c]));
  return strings.reduce((out, str, i) =>
    out + str + (i < values.length ? esc(values[i]) : ''), '');
}

const userInput = '';
console.log(safe`<p>Hello ${userInput}</p>`);
// <p>Hello &lt;img src=x onerror=alert(1)&gt;</p>   (interpolation escaped, static parts not)

console.log(String.raw`C:\new\tab`);   // C:\new\tab  (backslashes preserved)
Challenge:
Write a sql tag that builds a parameterized query: it returns { text, values } where text has the interpolations replaced by positional placeholders ($1, $2, ...) and values is the array of interpolated values in order — so user input is never concatenated into the SQL string.
#1 Pitfall: Escaping the whole assembled string instead of only the interpolated values, or forgetting that strings.length === values.length + 1 (there's always one more static part than there are values). Off-by-one here drops your last literal segment.

🆕 Modern Iteration & Collections

ES2024/2025 additions that make working with iterators and grouping data ergonomic — no more lodash for these.

Iterator Helpers Functions & Iteration

Until now, to map/filter a generator you had to spread it into an array first — eagerly materializing everything, which is impossible for infinite or huge sequences. Iterator helpers add lazy .map/.filter/.take directly on iterators, so you process items one at a time.

Any iterator (including the one a generator returns) now has chainable methods on Iterator.prototype. They are lazy: nothing runs until you pull values, and only as many source items as needed are consumed.

1Lazy chaining over an infinite source. take bounds an otherwise-endless generator; toArray is the terminal step that actually drains it.

JavaScript
function* naturals() { let n = 1; while (true) yield n++; }  // infinite

const result = naturals()
  .map(n => n * n)            // lazy
  .filter(n => n % 2 === 1)   // lazy
  .take(5)                    // stop after 5 — keeps the pipeline finite
  .toArray();                 // terminal: pull values now

console.log(result);          // [1, 9, 25, 49, 81]
[ 1, 9, 25, 49, 81 ]

2Terminal helpers. Besides toArray, there are short-circuiting reducers — find, some, every, reduce, and forEach — that consume only as far as they must.

JavaScript
const firstBig = naturals().map(n => n * n).find(n => n > 50);
console.log(firstBig);        // 64  (stops at 8*8; never computes further)

const sum = naturals().take(100).reduce((a, b) => a + b, 0);
console.log(sum);             // 5050

console.log(naturals().take(3).some(n => n > 2));  // true
64 5050 true

3Adapt any iterable with Iterator.from(). Wrap a built-in iterator (a Set, a Map's entries, matchAll results) to get the helper methods, and use drop to skip a prefix.

JavaScript
const set = new Set([10, 20, 30, 40, 50]);

const labels = Iterator.from(set)
  .drop(1)                       // skip the first value
  .map(n => `#${n}`)
  .toArray();
console.log(labels);             // [ '#20', '#30', '#40', '#50' ]

// flatMap expands each item into 0+ items, still lazily:
const pairs = Iterator.from([1, 2, 3]).flatMap(n => [n, -n]).toArray();
console.log(pairs);              // [ 1, -1, 2, -2, 3, -3 ]
[ '#20', '#30', '#40', '#50' ] [ 1, -1, 2, -2, 3, -3 ]
Common Mistakes:
❌ Spreading an iterator to use array methods, breaking laziness: [...naturals()].map(...).slice(0, 5); // hangs forever — spread never ends
✅ Chain the helpers and bound the stream BEFORE materializing: naturals().map(...).take(5).toArray(); // only 5 items are ever produced
Key Rules:
  • Helpers are lazy and single-use: an iterator is consumed as you pull from it — you can't iterate the same one twice.
  • Always bound an infinite source with take (or a short-circuiting terminal like find) before a terminal that drains it.
  • Use Iterator.from(x) to give helper methods to a built-in iterator or a hand-written one; reach for them instead of spreading into an array when the sequence is large or unbounded.
Practice Exercises:
Using a naturals() generator and iterator helpers only (no arrays until toArray), produce the first 10 numbers divisible by 3 or 5. Expected: [3, 5, 6, 9, 10, 12, 15, 18, 20, 21].
Write zip(a, b) as a generator, then use iterator helpers to take the first 5 zipped pairs of two infinite sequences (e.g. squares and cubes) and map them to their sums.
Given a large Map of userId → score, use Iterator.from(map) with filter and take to return the first 3 [id, score] entries whose score exceeds 90 — without building an intermediate array of all entries.
Connections: Helpers build directly on the iterator protocol behind generators and Symbol.iterator. The async sibling Array.fromAsync and async iterator helpers extend the same laziness to for await. Lazy take/drop are the eager functional pipe/compose patterns made memory-efficient.
Object.groupBy & Map.groupBy Data & Collections

Bucketing a list by some key is one of the most common data tasks (group orders by status, log lines by level). ES2024 made it a one-liner, replacing the verbose reduce-into-an-object boilerplate that everyone wrote slightly differently.

1Object.groupBy. Pass an iterable and a callback returning a key; you get back a null-prototype object whose values are arrays of the items that produced each key. Keys are coerced to strings (or symbols).

JavaScript
const items = [
  { name: 'apple',  type: 'fruit' },
  { name: 'carrot', type: 'veg' },
  { name: 'pear',   type: 'fruit' },
  { name: 'kale',   type: 'veg' },
];

const byType = Object.groupBy(items, (item) => item.type);
console.log(byType.fruit.map(i => i.name));  // [ 'apple', 'pear' ]
console.log(byType.veg.length);              // 2
console.log(Object.getPrototypeOf(byType));  // null  (no inherited keys)
[ 'apple', 'pear' ] 2 null

2Map.groupBy for non-string keys. When the grouping key is an object or you want to preserve key types/order, use Map.groupBy — it returns a Map, so keys keep their identity and you avoid string coercion.

JavaScript
const nums = [1, 2, 3, 4, 5, 6, 7];

const parity = Map.groupBy(nums, (n) => n % 2 === 0 ? 'even' : 'odd');
console.log(parity.get('even'));    // [ 2, 4, 6 ]
console.log(parity.get('odd'));     // [ 1, 3, 5, 7 ]

// group by a derived OBJECT key — only Map can do this safely:
const teamA = { id: 'A' }, teamB = { id: 'B' };
const players = [{ team: teamA }, { team: teamB }, { team: teamA }];
const byTeam = Map.groupBy(players, (p) => p.team);
console.log(byTeam.get(teamA).length);   // 2
[ 2, 4, 6 ] [ 1, 3, 5, 7 ] 2

3The callback's index argument. Like array iterators, the grouping callback also receives the element's index, useful for positional bucketing.

JavaScript
const data = ['a', 'b', 'c', 'd', 'e'];
const cols = Object.groupBy(data, (_, i) => `col${i % 3}`);
console.log(cols);
// [Object: null prototype] { col0: ['a','d'], col1: ['b','e'], col2: ['c'] }
[Object: null prototype] { col0: [ 'a', 'd' ], col1: [ 'b', 'e' ], col2: [ 'c' ] }
Common Mistakes:
❌ Treating Object.groupBy's result like a normal object: const g = Object.groupBy(items, fn); g.hasOwnProperty('fruit'); // TypeError — null prototype has no methods for (const k in g) { ... } // works, but g.toString etc. are gone
✅ Use static helpers / Object methods, or Map.groupBy: Object.hasOwn(g, 'fruit'); Object.keys(g); // safe const m = Map.groupBy(items, fn); m.has('fruit');
Key Rules:
  • Object.groupBy → null-prototype object, string/symbol keys; great for JSON-shaped output. Map.groupBy → Map, any key type, preserves insertion order.
  • The result's arrays are in source order; the callback gets (element, index).
  • Reach for Map.groupBy whenever the key is an object/derived value, or when you need .has/.size/ordered iteration.
Practice Exercises:
Given an array of transactions { amount, currency }, use Object.groupBy to bucket by currency, then map each bucket to its total amount, producing { USD: 120, EUR: 80 }.
Use Map.groupBy to split an array of numbers into three buckets keyed by the functions isNegative/isZero/isPositive (the key is the predicate's name string). Confirm order is preserved within each bucket.
Write your own groupBy(iterable, keyFn) polyfill that mimics Object.groupBy (null-prototype object, arrays in source order) using a single reduce, then verify it matches the native result on a sample input.
Connections: Map.groupBy leans on Map & Set reference-equality keys. The null-prototype gotcha ties back to prototypes & the prototype chain. Grouping pairs naturally with iterator helpers — group, then lazily transform each bucket.
Array.fromAsync Async Mastery

Collecting everything an async generator or stream produces into an array used to require a manual for await loop. Array.fromAsync is the async counterpart to Array.from — one call drains any async iterable, awaiting each value in sequence.

1Drain an async iterable. It returns a Promise of an array, so you await it. Each yielded value is awaited before the next is requested — sequential, not concurrent.

JavaScript
async function* stream() {
  yield 1;
  await new Promise(r => setTimeout(r, 10));
  yield 2;
  yield 3;
}

const all = await Array.fromAsync(stream());
console.log(all);            // [ 1, 2, 3 ]
[ 1, 2, 3 ]

2Optional mapping function. Like Array.from, a second arg maps each item; if it returns a Promise, that's awaited too — handy for resolving each element through an async transform.

JavaScript
const ids = [1, 2, 3];
async function fetchName(id) { return `user-${id}`; }   // pretend network

const names = await Array.fromAsync(ids, (id) => fetchName(id));
console.log(names);          // [ 'user-1', 'user-2', 'user-3' ]

// also accepts a sync iterable of promises (awaits each, in order):
const vals = await Array.fromAsync([Promise.resolve('a'), Promise.resolve('b')]);
console.log(vals);           // [ 'a', 'b' ]
[ 'user-1', 'user-2', 'user-3' ] [ 'a', 'b' ]

3Sequential vs concurrent — choose deliberately. fromAsync awaits items one at a time (good for backpressure / rate limits). For maximum concurrency, map to promises and use Promise.all instead.

JavaScript
// SEQUENTIAL: each fetchName waits for the previous (gentle on the server):
const seq = await Array.fromAsync(ids, fetchName);

// CONCURRENT: all requests fire at once, then collect:
const conc = await Promise.all(ids.map(fetchName));

console.log(seq.length === conc.length);   // true (same result, different timing)
true
Common Mistakes:
❌ Expecting concurrency from Array.fromAsync: // these run ONE AT A TIME — total time is the SUM of each delay await Array.fromAsync(urls, fetchJson);
✅ For parallel I/O, map to promises and Promise.all: await Promise.all(urls.map(fetchJson)); // use Array.fromAsync when you WANT sequential/backpressured consumption
Key Rules:
  • Returns a Promise<Array> — always await it. Works on async iterables, sync iterables, and array-likes.
  • It is sequential: items (and the optional mapper's promises) are awaited in order. Use Promise.all when you want them concurrent.
  • Use it to materialize an async generator/stream into an array; avoid it for unbounded streams (it buffers everything).
Practice Exercises:
Write an async generator that yields the numbers 1..n with a small delay between each, then use Array.fromAsync with a mapper to collect their squares. Confirm the result is [1, 4, 9, ...].
Given an array of URLs, write two collectors: collectSequential(urls) using Array.fromAsync and collectConcurrent(urls) using Promise.all. Add timing with performance.now() and observe the difference when each fetch takes ~50ms.
Implement a fromAsyncLimit(asyncIterable, mapper, limit) that is like Array.fromAsync with a mapper but runs at most limit mappers concurrently (a bounded pool). Return the results in source order.
Connections: This is the async twin of iterator helpers / Array.from, consuming the same async-iterator protocol as async generators and for await...of. The sequential-vs-concurrent tradeoff is the same one from Promise.all / async-await patterns.

🆕 Async Ergonomics

Two modern tools that remove long-standing async boilerplate: externally-resolvable promises, and implicit context that survives await.

Promise.withResolvers Async Mastery

The classic "deferred" pattern — creating a promise and hoisting its resolve/reject out of the executor — appears constantly (event-to-promise bridges, queues, request maps). ES2024's Promise.withResolvers gives you all three in one destructure, no closure gymnastics.

1The old way vs the new way. Previously you declared let resolve outside, then captured it inside the executor. Now one call returns { promise, resolve, reject }.

JavaScript
// OLD: leak resolve/reject out of the executor
let resolveOld;
const pOld = new Promise((res) => { resolveOld = res; });

// NEW: one line
const { promise, resolve, reject } = Promise.withResolvers();

resolve(42);
console.log(await promise);   // 42
42

2Bridge an event into a promise. Resolve from an event handler that fires later — a clean one-shot "wait for this event" helper.

JavaScript
function once(emitter, event) {
  const { promise, resolve } = Promise.withResolvers();
  emitter.once(event, resolve);     // resolve when the event fires
  return promise;
}

// const data = await once(socket, 'data');   // awaits the next 'data' event

// Timeout guard built the same way:
function delay(ms) {
  const { promise, resolve } = Promise.withResolvers();
  setTimeout(resolve, ms);
  return promise;
}
await delay(5);
console.log('done waiting');
done waiting

3A signal/latch you resolve from anywhere. Hand resolve to one part of the code and promise to another — a tiny synchronization primitive.

JavaScript
function createLatch() {
  const { promise, resolve } = Promise.withResolvers();
  return { ready: promise, open: resolve };   // share each half separately
}

const latch = createLatch();
setTimeout(() => latch.open('go!'), 10);      // released by some other code path

console.log('waiting...');
console.log(await latch.ready);               // 'go!'
waiting... go!
Common Mistakes:
❌ Doing real work inside a deferred and swallowing failures: const { promise, resolve } = Promise.withResolvers(); doAsyncThing().then(resolve); // if doAsyncThing rejects, promise hangs forever
✅ Wire up reject too, so failures propagate: doAsyncThing().then(resolve, reject); // and prefer plain async/await unless you genuinely need external control
Key Rules:
  • Use it when something outside the promise's creation site decides when it settles (events, callbacks, cross-module signaling) — not for ordinary sequential async, where async/await is clearer.
  • Calling resolve/reject more than once is a no-op after the first — the promise is already settled.
  • Always provide a path to reject (e.g. a timeout or error handler), or a stuck producer leaves the consumer awaiting forever.
Practice Exercises:
Implement withTimeout(promise, ms) using Promise.withResolvers for the timeout branch: it rejects with a 'timeout' error if ms elapses first, otherwise resolves/rejects with the original promise's outcome. Clear the timer either way.
Build a single-slot Channel with send(value) and receive(): receive() returns a promise that resolves with the next sended value. Use Promise.withResolvers to park a waiting receiver until a value arrives.
Write nextEvent(target, eventName, { signal }) that resolves on the next matching event but rejects if an AbortSignal fires first — wiring both the event listener and the abort listener to the same deferred, and removing both listeners on settle.
Connections: This is a formalization of the deferred you may have hand-rolled while studying Promise internals. It pairs naturally with AbortController (resolve on event, reject on abort) and is the building block for queues feeding async generators.
AsyncLocalStorage (async context) Node.js Runtime

In a server you often need request-scoped data (a request id, the current user, a trace span) available deep in the call tree — without threading it through every function signature. AsyncLocalStorage propagates a value through an entire async call chain implicitly, like thread-local storage for the event loop.

1run + getStore. als.run(store, fn) executes fn with store as the active context; anywhere inside (even after awaits and in nested calls) als.getStore() returns that same store.

JavaScript
import { AsyncLocalStorage } from 'node:async_hooks';

const als = new AsyncLocalStorage();

function log(msg) {
  const store = als.getStore();        // reads the ambient context
  console.log(`[req ${store?.reqId}] ${msg}`);
}

async function handler() {
  log('start');
  await new Promise(r => setTimeout(r, 5));
  log('after await');                  // context survives the await
}

als.run({ reqId: 'abc' }, handler);
als.run({ reqId: 'xyz' }, handler);
[req abc] start [req xyz] start [req abc] after await [req xyz] after await

2Isolation between concurrent runs. Each run gets its own store; interleaved async operations never see each other's context. That's what makes it safe for concurrent requests on one event loop.

JavaScript
// Simulate two overlapping requests:
function deep() {
  return als.getStore().user;          // no parameter passing needed
}
async function request(user) {
  return als.run({ user }, async () => {
    await new Promise(r => setTimeout(r, Math.random() * 10));
    return deep();                     // still sees THIS request's user
  });
}

const [a, b] = await Promise.all([request('ada'), request('linus')]);
console.log(a, b);                     // ada linus  (never crossed over)
ada linus

3Mutable per-request scratch space. The store is just an object (or a Map) — middleware can stash values into it that later layers read, without a global.

JavaScript
const ctx = new AsyncLocalStorage();

function middleware(next) {
  ctx.run(new Map([['startedAt', Date.now()]]), next);
}

function controller() {
  const store = ctx.getStore();
  store.set('userId', 7);              // add to the same per-request map
  return `user ${store.get('userId')} in ` +
         `${Date.now() - store.get('startedAt')}ms`;
}

middleware(() => console.log(controller()));
user 7 in 0ms
Common Mistakes:
❌ Reading getStore() outside any run() — or after detaching from the chain: setInterval(() => als.getStore().reqId, 1000); // undefined: timer isn't in a run const cb = als.getStore(); someEmitter.on('x', () => cb...) // may be stale/null
✅ Snapshot what you need while still inside run(), or re-enter context: const reqId = als.getStore()?.reqId; // capture the value now // or use als.run(store, () => ...) to re-establish context for later callbacks
Key Rules:
  • getStore() returns the store of the nearest enclosing run(), or undefined outside one — always guard with ?..
  • Context follows the async chain (promises, await, timers scheduled inside run), but not work detached from it; capture values before leaving the chain.
  • Use it for cross-cutting concerns (request id, auth, tracing, transactions) — not as a general-purpose global; overusing it hides data flow.
Practice Exercises:
Build a withRequestId(fn) wrapper that runs fn inside an AsyncLocalStorage.run with a fresh random request id, and a logger.info(msg) that prefixes every line with the current request id. Fire two concurrent calls and confirm logs don't cross over.
Implement a tiny tracing helper: trace(name, fn) reads a per-request spans array from the store, pushes { name, start, end } around await fn(), and returns the result. After a request completes, print the collected spans.
Demonstrate the detachment pitfall: store a reqId, schedule a setTimeout callback that reads getStore() — first from outside any run (observe undefined), then fix it by scheduling the timer inside the run callback.
Connections: This is the Node analogue of context propagation; it relies on the same async-task tracking as the async machinery and the Node event loop phases you reviewed. Capturing a value before a callback detaches uses the ?. guard from today's review.
