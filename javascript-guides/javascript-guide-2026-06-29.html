🔁 Review Section

Six data, module, and concurrency concepts are due for retrieval practice today. Recall first, verify second.

Map & Set Data & Collections 10 days since review
Map keys can be any value (objects, functions, NaN), preserve insertion order, and it exposes a .size plus direct iteration (for..of, .entries(), .keys(), .values()). Set stores unique values using SameValueZero (so NaN is deduped). Prefer them over plain objects when keys are dynamic, non-string, or you need cheap size/iteration.
JavaScript
const idOf = new Map();
const alice = { name: 'Alice' };
idOf.set(alice, 1).set('str-key', 2).set(NaN, 3);   // set() chains
console.log(idOf.get(alice), idOf.size);            // 1 3

// dedupe with Set, keep insertion order:
const seen = new Set([1, 1, 2, NaN, NaN]);
console.log([...seen]);                             // [1, 2, NaN]

// frequency count is a classic Map use:
const freq = new Map();
for (const ch of 'banana') freq.set(ch, (freq.get(ch) ?? 0) + 1);
console.log([...freq]);                             // [['b',1],['a',3],['n',2]]
Challenge:
Implement an LRUCache(capacity) using a single Map, exploiting its insertion-order guarantee: on get, move the key to the most-recent position (delete + re-set); on put past capacity, evict the oldest via map.keys().next().value. Explain why a plain object could not back this cleanly.
#1 Pitfall: Using object identity keys and expecting value equality — map.set({a:1}, 'x'); map.get({a:1}) is undefined because they are different objects. Also, Map is not JSON-serializable (JSON.stringify(map) gives {}); convert with [...map] / Object.fromEntries first.
ES Modules Deep Dive Modules & Patterns 10 days since review
ESM import/export are static (analyzable before execution) and hoisted; bindings are live (an imported name reflects later changes in the exporter). Each module is evaluated once and cached. import() is dynamic and returns a promise. Modules run in strict mode, have their own scope, and support top-level await.
JavaScript
// counter.js
export let count = 0;
export function inc() { count++; }

// main.js
import { count, inc } from './counter.js';
console.log(count);   // 0
inc();
console.log(count);   // 1  — LIVE binding, not a copied snapshot

// dynamic + conditional loading:
if (userWantsCharts) {
  const { renderChart } = await import('./charts.js');   // top-level await
  renderChart();
}
Challenge:
Given two modules that import from each other (a circular dependency), predict what each sees at evaluation time versus after both finish loading, and rewrite one so a function-scoped (lazy) access avoids the undefined-at-init hazard. Explain how live bindings make the fix work.
#1 Pitfall: Treating imports as mutable copies — you cannot reassign an imported binding (count = 5 throws; it is read-only in the importer). Also mixing CommonJS require and ESM import in the same file, or forgetting file extensions in Node ESM, breaks resolution.
Streams API Async Mastery 7 days since review
The Web Streams API models data that arrives over time: ReadableStream (source), WritableStream (sink), and TransformStream (a readable+writable pair). Backpressure is automatic — a slow consumer pauses the producer. Compose with pipeThrough(transform) and terminate with pipeTo(sink); a ReadableStream is async-iterable.
JavaScript
const upper = new TransformStream({
  transform(chunk, controller) { controller.enqueue(chunk.toUpperCase()); },
});

const source = new ReadableStream({
  start(c) { ['a', 'b', 'c'].forEach((x) => c.enqueue(x)); c.close(); },
});

// pipe through a transform, then async-iterate the result:
for await (const chunk of source.pipeThrough(upper)) {
  console.log(chunk);            // A  B  C
}

// stream a fetch body without buffering the whole response:
const res = await fetch('/big.txt');
const reader = res.body.pipeThrough(new TextDecoderStream()).getReader();
Challenge:
Build a lineSplitter() TransformStream that receives text chunks and emits one line at a time, buffering partial lines across chunk boundaries and flushing the remainder in flush(). Pipe a fetch body through TextDecoderStream then your splitter, and log each line.
#1 Pitfall: Reading a stream twice — a ReadableStream is single-use and, once a reader is acquired, it is locked. To consume the same data two ways, use stream.tee() to fork it. Forgetting to reader.releaseLock() or to close() a controller can hang the pipeline.
Web Workers Async Mastery 7 days since review
A Worker runs script on a separate thread, so CPU-heavy work never blocks the UI. It has no DOM access and communicates only via postMessage/onmessage, with arguments copied by the structured clone algorithm. Pass a transferable (e.g. an ArrayBuffer) as the second arg to move ownership with zero copy. Call worker.terminate() to stop it.
JavaScript
// main.js
const worker = new Worker('./crunch.js', { type: 'module' });
worker.onmessage = (e) => console.log('result:', e.data);

const buf = new ArrayBuffer(1024);
worker.postMessage({ cmd: 'hash', buf }, [buf]);   // TRANSFER buf (buf now detached here)

// crunch.js
self.onmessage = (e) => {
  const { cmd, buf } = e.data;
  // ...CPU-bound work off the main thread...
  self.postMessage(computeHash(buf));
};
Challenge:
Wrap a worker in a promise-based callWorker(worker, payload) that resolves with the reply to that call. Tag each message with a unique id, store pending resolvers in a Map keyed by id, and resolve the matching one on reply — so concurrent calls to the same worker don't cross their responses.
#1 Pitfall: Expecting shared memory — by default the worker gets a copy, so mutating e.data inside the worker never affects the main thread. Also, functions, DOM nodes, and class instances don't survive structured clone; and after transferring an ArrayBuffer it is detached (unusable) on the sending side.
Service Workers Async Mastery 7 days since review
A service worker is a network proxy that sits between your app and the network, enabling offline support, caching, and push. Its lifecycle is install → activate → fetch; it runs even with no page open, but only over HTTPS (or localhost) and is scoped to its directory. It cannot touch the DOM and must not rely on global state (it can be killed and restarted anytime).
JavaScript
// sw.js — cache-first strategy
const CACHE = 'v1';
self.addEventListener('install', (e) => {
  e.waitUntil(caches.open(CACHE).then((c) => c.addAll(['/', '/app.js'])));
});
self.addEventListener('fetch', (e) => {
  e.respondWith(
    caches.match(e.request).then((hit) => hit ?? fetch(e.request))
  );
});

// page: register (scoped to /)
if ('serviceWorker' in navigator) {
  await navigator.serviceWorker.register('/sw.js');
}
Challenge:
Implement a stale-while-revalidate fetch handler: respond immediately from cache if present, and in parallel fetch a fresh copy to update the cache for next time (falling back to network on a cache miss). Add an activate handler that deletes old cache versions so a new deploy cleans up.
#1 Pitfall: Forgetting event.waitUntil() — without it the browser may kill the worker before your async cache work finishes. Also, a new service worker waits by default until all old tabs close before activating; skipping skipWaiting()/clients.claim() surprises people who expect an instant update.
Typed Arrays & ArrayBuffer Data & Collections 14 days since review
An ArrayBuffer is a fixed-length block of raw bytes; you never read it directly. Instead you lay a view over it — Uint8Array, Float64Array, etc. — which interprets the bytes as numbers. Multiple views can share one buffer (a union/overlay). Use a DataView when you need explicit control over endianness or mixed types at chosen byte offsets.
JavaScript
const buf = new ArrayBuffer(4);            // 4 raw bytes
const u8 = new Uint8Array(buf);
const u32 = new Uint32Array(buf);         // SAME 4 bytes, one 32-bit view

u8[0] = 0xFF; u8[1] = 0x00; u8[2] = 0x00; u8[3] = 0x00;
console.log(u32[0].toString(16));         // ff  (little-endian on most machines)

// DataView = explicit endianness, no ambiguity:
const dv = new DataView(buf);
console.log(dv.getUint32(0, false));      // big-endian read: 4278190080
u8[300] = 1;                              // out-of-range write is SILENTLY ignored
Challenge:
Parse a little-endian binary record — [u16 id][f32 value][u8 flags] — out of a single ArrayBuffer using one DataView, returning { id, value, flags }. Then write the reverse encoder, and confirm a round-trip reproduces the original bytes. Explain why you must pass littleEndian=true explicitly.
#1 Pitfall: Assuming a platform endianness — bare typed-array reads use the host's byte order, which varies; only DataView with an explicit endian flag is portable across machines and network protocols. Also, out-of-bounds typed-array indexing doesn't throw — reads give undefined and writes are dropped silently.

🆕 Modern Collections

Three additions that make working with sets and arrays declarative and immutable — including brand-new ES2025 set algebra.

Set Methods (ES2025) Data & Collections

For years Set could only add, delete, and test membership — any set algebra (union, intersection, difference) meant hand-rolled loops or spreading through arrays. ES2025 ships seven native methods that turn Set into a proper mathematical set, correct and fast, on Node 24+ and current browsers.

1The three combining methods return a new Set. union, intersection, and difference never mutate either operand.

JavaScript
const a = new Set([1, 2, 3]);
const b = new Set([2, 3, 4]);

console.log([...a.union(b)]);              // [1, 2, 3, 4]
console.log([...a.intersection(b)]);      // [2, 3]
console.log([...a.difference(b)]);        // [1]        (in a, not in b)
console.log([...a.symmetricDifference(b)]); // [1, 4]   (in exactly one)

console.log([...a]);                       // [1, 2, 3]  — a is untouched
[ 1, 2, 3, 4 ] [ 2, 3 ] [ 1 ] [ 1, 4 ] [ 1, 2, 3 ]

2The three predicates answer subset questions. isSubsetOf, isSupersetOf, and isDisjointFrom return a boolean without building an intermediate set.

JavaScript
const perms = new Set(['read', 'write']);
const granted = new Set(['read', 'write', 'delete']);

console.log(perms.isSubsetOf(granted));        // true  (all of perms in granted)
console.log(granted.isSupersetOf(perms));      // true
console.log(perms.isDisjointFrom(new Set(['exec']))); // true (no overlap)
true true true

3The argument may be any "set-like". The methods only require the argument to expose size, has(), and keys() — so a Map works directly (its keys are used), not just another Set. The receiver, though, must be a real Set.

JavaScript
const users = new Set(['ann', 'bob', 'cy']);
const scores = new Map([['bob', 10], ['dan', 4]]);   // set-like: size/has/keys

// intersect a Set with a Map's keys — no manual conversion needed:
console.log([...users.intersection(scores)]);        // ['bob']
[ 'bob' ]
Common Mistakes:
❌ Reaching for arrays + filter for basic set algebra: const inter = [...a].filter(x => b.has(x)); // works but verbose, O(n) rebuild const uni = [...new Set([...a, ...b])]; // extra allocations
✅ Use the native methods — clearer and optimized: a.intersection(b); a.union(b); a.difference(b);
Key Rules:
  • All seven methods are non-mutating: the four algebra methods return a fresh Set; the three is… predicates return a boolean.
  • difference is directional (a.difference(b) ≠ b.difference(a)); symmetricDifference and union/intersection are order-independent in membership.
  • The argument can be any object with size, has, and keys (e.g. a Map); the receiver must be an actual Set.
Practice Exercises:
Given required and userSkills as sets, use difference to report the missing skills and isSubsetOf to decide if the user qualifies. Confirm neither input set is mutated.
Write jaccard(a, b) — the size of intersection over the size of union — using only the new methods and .size. Verify it returns 0 for disjoint sets and 1 for equal sets.
Given a Map of tag → count and a Set of banned tags, produce the set of allowed tags with difference, passing the Map directly as the set-like operand. Explain why the receiver, not the argument, must be the Set.
Connections: These extend the Map & Set collection you reviewed today with real set algebra, and complement WeakMap/WeakSet for membership tracking. The set-like duck-typing (size/has/keys) mirrors how iteration protocols keep built-ins interoperable.
Change Array by Copy Data & Collections

sort, reverse, and splice mutate the array in place — a constant source of bugs when the array is shared, is a prop, or sits in immutable state (React, Redux). ES2023 added copying counterparts — toSorted, toReversed, toSpliced, and with — that return a new array and leave the original untouched.

1The mutation trap. The classic footgun is [...arr].sort() forgetting the spread, or calling .sort() on a prop.

JavaScript
const scores = [3, 1, 2];

const sorted = scores.toSorted((a, b) => a - b);
console.log(sorted);        // [1, 2, 3]
console.log(scores);        // [3, 1, 2]   — ORIGINAL untouched

// contrast with the mutating version:
const oops = [3, 1, 2];
oops.sort((a, b) => a - b);
console.log(oops);          // [1, 2, 3]   — mutated in place
[ 1, 2, 3 ] [ 3, 1, 2 ] [ 1, 2, 3 ]

2with(index, value) = immutable single-element replace. It replaces one element by index and returns a new array — the copying counterpart of arr[i] = value. Negative indices count from the end.

JavaScript
const row = ['a', 'b', 'c'];

console.log(row.with(1, 'B'));    // ['a', 'B', 'c']
console.log(row.with(-1, 'Z'));   // ['a', 'b', 'Z']   (last element)
console.log(row);                 // ['a', 'b', 'c']    — unchanged

// great for updating immutable state without a manual map:
const state = [10, 20, 30];
const next = state.with(2, 99);   // [10, 20, 99], state kept
[ 'a', 'B', 'c' ] [ 'a', 'b', 'Z' ] [ 'a', 'b', 'c' ]

3toReversed and toSpliced. toSpliced(start, deleteCount, ...items) returns a new array with the splice applied — and, unlike splice, it returns the result, not the removed elements.

JavaScript
const q = ['a', 'b', 'c', 'd'];

console.log(q.toReversed());              // ['d', 'c', 'b', 'a']
console.log(q.toSpliced(1, 2, 'X'));      // ['a', 'X', 'd']  (remove b,c; insert X)
console.log(q);                           // ['a','b','c','d']  — original intact

// note: splice() RETURNS removed items and mutates; toSpliced RETURNS the new array
const removed = ['a','b','c','d'].splice(1, 2);
console.log(removed);                     // ['b', 'c']  (different return value!)
[ 'd', 'c', 'b', 'a' ] [ 'a', 'X', 'd' ] [ 'a', 'b', 'c', 'd' ] [ 'b', 'c' ]
Common Mistakes:
❌ Mutating shared/state arrays, or conflating the return values: props.items.sort() // mutates a prop -> render bugs const r = arr.toSpliced(0, 1) // r is the NEW array, not the removed item
✅ Use the copying methods for shared data; know what each returns: const sorted = props.items.toSorted(); // new array, prop untouched const updated = state.with(i, value); // immutable element update
Key Rules:
  • Four copying methods: toSorted, toReversed, toSpliced, with — each returns a new array and never mutates the receiver.
  • They are shallow copies: nested objects are still shared by reference (same caveat as spread).
  • toSpliced returns the resulting array (not the removed elements like splice); with throws RangeError on an out-of-bounds index (unlike bracket assignment, which would extend/skip).
Practice Exercises:
Write a pure moveItem(arr, from, to) that returns a new array with one element repositioned, using only toSpliced (and at) — no mutation of arr. Verify the input is unchanged.
Implement updateAt(state, index, updater) for immutable state that returns state.with(index, updater(state[index])), and show it leaves state intact while producing the new array.
Given an array of records, return a version sorted by a key and with the top record's rank field set to 1, using toSorted then with, proving the original array is never mutated at any step.
Connections: These are the array-level analogue of immutable update patterns and pair with structuredClone when you need deep (not shallow) copies. The shared-reference caveat is the same one from spread/rest you have seen with objects.
WeakMap & WeakSet Data & Collections

A normal Map holding an object as a key keeps that object alive forever — a classic memory leak when you attach metadata to DOM nodes or instances. WeakMap/WeakSet hold their keys/values weakly: once nothing else references the object, the garbage collector reclaims it and the entry, automatically.

1Keys must be objects (or symbols), and are held weakly. There is no way to leak memory by caching per-object data — drop the object and the entry disappears.

JavaScript
const metadata = new WeakMap();

let node = { id: 1 };
metadata.set(node, { clicks: 0 });      // attach data without touching the object

console.log(metadata.get(node));        // { clicks: 0 }
console.log(metadata.has(node));        // true

node = null;   // last strong ref gone -> the WeakMap entry becomes eligible for GC
// (you can't observe the removal directly — that's the point)
{ clicks: 0 } true

2The trade-off: no enumeration, no size. Because entries can vanish at any moment, WeakMap/WeakSet are not iterable, have no .size, and no .clear() in the spec sense of listing. You can only get/set/has/delete (and add/has/delete for WeakSet).

JavaScript
const wm = new WeakMap();
wm.set({}, 1);
// wm.size            // undefined — no size
// for (const x of wm) {}   // TypeError: wm is not iterable
console.log(typeof wm.keys);   // 'undefined' — cannot list keys

const ws = new WeakSet();
const el = document.createElement('div');
ws.add(el);
console.log(ws.has(el));       // true  — membership test only
undefined true

3Canonical uses: private state and "seen" tracking. Store per-instance private data keyed by this, or mark objects as already-processed without mutating them or leaking references.

JavaScript
// truly private fields via closure + WeakMap (pre-# fields pattern):
const _balance = new WeakMap();
class Account {
  constructor(n) { _balance.set(this, n); }
  deposit(n) { _balance.set(this, _balance.get(this) + n); }
  get balance() { return _balance.get(this); }
}
const acc = new Account(100);
acc.deposit(50);
console.log(acc.balance);       // 150  — _balance is not a visible property

// visited-set to avoid reprocessing (and avoid cycles):
const visited = new WeakSet();
function walk(obj) {
  if (visited.has(obj)) return;   // cycle / repeat guard
  visited.add(obj);
  for (const v of Object.values(obj)) if (typeof v === 'object' && v) walk(v);
}
150
Common Mistakes:
❌ Using primitives as keys, or expecting to iterate: new WeakMap().set('str', 1) // TypeError: Invalid value used as weak map key [...weakset] // TypeError: not iterable weakmap.size // undefined — there is no size
✅ Key by object; use a regular Map/Set when you need iteration or size: wm.set(objKey, data) // objects (or symbols) only // need .size / iteration? that requires strong refs -> use Map/Set
Key Rules:
  • Keys (WeakMap) / values (WeakSet) must be objects — or, in current engines, registered symbols; primitives throw.
  • References are weak: entries disappear when the key is otherwise unreachable. That means no iteration, size, or clear — the contents aren't observable.
  • Reach for them to associate data with an object's lifetime (metadata, caches, private state, cycle guards) without extending that lifetime.
Practice Exercises:
Build memoizeByObject(fn) that caches results of a one-object-argument function in a WeakMap, so cached entries are collected when their key object is. Explain why a plain Map would leak here.
Implement a once(obj, fn) helper using a WeakSet that runs fn the first time it sees a given object and is a no-op afterward — without ever preventing that object from being garbage-collected.
Rewrite a class that currently stores a secret on this._secret to instead keep it in a module-level WeakMap keyed by this, and show the secret no longer appears in Object.keys(instance) or JSON.stringify.
Connections: WeakMap/WeakSet share the memory-lifetime theme with WeakRef/FinalizationRegistry but are simpler and safer (no finalizers). The private-state use case is now often served by private class fields — compare the two approaches below.

🆕 Operators & Classes

Two core-language features that tighten everyday code: short-circuiting assignment and real encapsulation in classes.

Logical Assignment Operators Core Language

ES2021 added &&=, ||=, and ??= — compound operators that combine a logical test with assignment. Crucially they short-circuit: the assignment (and any side effects, including setters) only happens when the logic says so. They collapse common "set a default" and "guard an update" idioms into one clear line.

1??= assigns only when the current value is nullish (null or undefined) — the correct "default if missing" operator that preserves 0, '', and false.

JavaScript
const opts = { count: 0, label: '' };

opts.count ??= 10;      // count is 0 (not nullish) -> stays 0
opts.label ??= 'n/a';   // '' is not nullish       -> stays ''
opts.title ??= 'Untitled';  // undefined            -> assigned

console.log(opts);      // { count: 0, label: '', title: 'Untitled' }
{ count: 0, label: '', title: 'Untitled' }

2||= assigns on any falsy value; &&= assigns only when the current value is truthy. Use ||= when every falsy value should be replaced; use &&= to transform a value only if it already exists.

JavaScript
let name = '';
name ||= 'anonymous';        // '' is falsy -> 'anonymous'
console.log(name);           // 'anonymous'

let user = { token: 'abc' };
user.token &&= user.token.trim();   // token is truthy -> reassign trimmed
let guest = {};
guest.token &&= guest.token.trim(); // token is undefined (falsy) -> NOT run, no crash
console.log(user.token, guest.token); // 'abc' undefined
anonymous abc undefined

3Short-circuit means no write on skip. Unlike x = x ?? y (which always assigns), x ??= y performs the assignment only when needed. That matters for getters/setters, proxies, and reactive frameworks that observe writes.

JavaScript
const target = {
  _v: 5,
  get v() { return this._v; },
  set v(n) { console.log('setter ran with', n); this._v = n; },
};

target.v ??= 99;   // getter returns 5 (not nullish) -> setter NEVER runs
console.log(target.v);   // 5

target.v ||= 99;   // 5 is truthy -> setter still does NOT run
target._v = 0;
target.v ||= 99;   // now falsy -> setter RUNS once
setter ran with 99 5
Common Mistakes:
❌ Using ||= for defaults when 0/''/false are valid values: config.retries ||= 3 // a real retries:0 gets clobbered to 3! count ||= 10 // count:0 becomes 10 — usually a bug
✅ Use ??= for "default only if null/undefined": config.retries ??= 3 // keeps 0, only fills in missing count ??= 10 // 0 preserved
Key Rules:
  • a ??= b assigns when a is null/undefined; a ||= b when a is falsy; a &&= b when a is truthy.
  • All three short-circuit — the right side is evaluated and the write happens only if the condition triggers (no needless setter/proxy calls).
  • For "fill in a default", prefer ??= so legitimate 0/''/false survive; reserve ||= for when all falsy values truly should be replaced.
Practice Exercises:
Write withDefaults(opts) that fills in missing config (timeout, retries, verbose) using ??=, and prove that passing { retries: 0, verbose: false } keeps those exact values rather than overwriting them.
Implement a memo cache cache[key] ??= expensiveCompute(key) and demonstrate, with a logging expensiveCompute, that the computation runs at most once per key (short-circuit prevents recomputation).
Given an object with a setter that records every write, use ??=, ||=, and &&= to show exactly which assignments trigger the setter and which are skipped — contrast each with the equivalent x = x ?? y form that always writes.
Connections: ??= is the assignment form of the nullish coalescing / optional-chaining family you studied, and its short-circuit semantics are the same lazy-evaluation idea behind &&/||. Pair it with the copying array methods above for immutable-friendly default handling.
Private Class Fields, Methods & Static Blocks Core Language

Before # private members, "private" was a convention (_name) or a closure/WeakMap trick. Modern classes have hard private state: fields and methods prefixed with # are inaccessible — even undetectable — outside the class body. Static blocks add a place to run one-time class setup. Together they give real encapsulation with clean syntax.

1Private fields and methods. A # member is only visible inside the class. Accessing it from outside is a syntax error, not undefined — the name doesn't exist beyond the class.

JavaScript
class Counter {
  #count = 0;                       // private field
  #clamp(n) { return Math.max(0, n); }   // private method

  increment(by = 1) { this.#count = this.#clamp(this.#count + by); }
  get value() { return this.#count; }
}

const c = new Counter();
c.increment(3);
console.log(c.value);     // 3
// console.log(c.#count); // SyntaxError: Private field '#count' must be
//                        // declared in an enclosing class
console.log(Object.keys(c));   // []  — private fields don't enumerate
3 []

2Static private members and static blocks. static #x is per-class private state; a static { } block runs once when the class is defined, with access to the private statics — ideal for computed setup that a field initializer can't express.

JavaScript
class Registry {
  static #instances = 0;
  static #config;

  static {                         // runs once, at class definition
    Registry.#config = { max: 100, loadedAt: Date.now() };
  }

  constructor() { Registry.#instances++; }
  static get count() { return Registry.#instances; }
  static get max() { return Registry.#config.max; }
}

new Registry(); new Registry();
console.log(Registry.count, Registry.max);   // 2 100
2 100

3Brand checks with #field in obj. You can test whether an object is a genuine instance carrying your private field using #x in obj — a reliable "is this really one of mine?" check that faked look-alikes can't pass.

JavaScript
class Money {
  #amount;
  constructor(a) { this.#amount = a; }
  add(other) {
    if (!(#amount in other))        // brand check: real Money instance?
      throw new TypeError('not a Money');
    return new Money(this.#amount + other.#amount);   // can read other's #amount
  }
  get amount() { return this.#amount; }
}

const total = new Money(5).add(new Money(10));
console.log(total.amount);              // 15
console.log(#amount in { amount: 1 }); // (inside class) false — a fake fails
15
Common Mistakes:
❌ Treating #private like the _name convention, or accessing it dynamically: this['#count'] // NOT the private field — just a normal "#count" string key obj.#count // SyntaxError outside the class (can't be reached at all) class A { #x; } class B extends A { m(){ return this.#x; } } // SyntaxError: not B's
✅ Access #members only inside the declaring class, by literal name: this.#count // fine within the class (#x in obj) // safe presence/brand test, no throw
Key Rules:
  • # members (fields, methods, accessors, static) are truly private to the declaring class — not inherited into subclasses, not reachable via bracket/dynamic access, not enumerable.
  • Use #field in obj for a non-throwing brand check; accessing an absent #field via obj.#field would otherwise throw a TypeError.
  • static { } blocks run once at class-definition time (in source order with static fields) and can initialize private static state; they replace the old "assign after the class" hack.
Practice Exercises:
Write a Stack class whose backing array is a private #items field with a private #assertNotEmpty() method, exposing only push/pop/peek/size. Confirm the array is invisible to Object.keys and JSON.stringify.
Build a Temperature class with a static block that precomputes a conversion table into a private static field, and static getters that read it. Verify the block runs exactly once regardless of how many instances you create.
Add a static Money.isMoney(x) that uses a #brand in x check to distinguish real instances from plain objects that merely copy the shape, and show a duck-typed { amount: 5 } fails the check.
Connections: Private class fields are the modern replacement for the WeakMap private-state pattern shown above — compare the ergonomics. Brand checks via #x in obj are a sturdier alternative to instanceof, which can be fooled across realms or by prototype tampering.
