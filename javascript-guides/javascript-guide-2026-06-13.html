R1. async/await Patterns Async Mastery rep 0 · overdue 2 days
An async function always returns a Promise. await pauses execution until the promise settles. Use try/catch for error handling. Promise.all runs promises concurrently (not parallel — still single-threaded). Sequential await in a loop serializes requests unnecessarily.
Key Pattern 
// Sequential — each waits for the previous (SLOW)
async function sequential(urls) {
  const results = [];
  for (const url of urls) {
    const res = await fetch(url);    // waits before next iteration
    results.push(await res.json());
  }
  return results;
}

// Concurrent — all requests fire at once (FAST)
async function concurrent(urls) {
  const promises = urls.map(url => fetch(url).then(r => r.json()));
  return Promise.all(promises); // all in-flight simultaneously
}

// Error handling with try/catch
async function safeFetch(url) {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error("Fetch failed:", err.message);
    return null;
  }
}
Challenge

Write a fetchWithConcurrencyLimit(urls, limit) function that fetches all URLs but allows at most limit concurrent requests at a time. Use a pool pattern, not Promise.all on the full array.

Pitfall: Using await inside .forEach() does not work as expected — forEach ignores the returned promise, so all iterations fire concurrently. Use a for...of loop for sequential awaiting, or Promise.all(arr.map(...)) for concurrent.
R2. Generators & Iterators Functions & Iteration rep 1 · overdue 1 day
function* defines a generator that yields values lazily. Each yield pauses execution, and .next() resumes it. yield* delegates to another iterable. Objects become iterable by implementing [Symbol.iterator](). Generators are both iterators and iterables.
Key Pattern 
function* range(start, end, step = 1) {
  for (let i = start; i < end; i += step) {
    yield i;
  }
}

// Lazy — values produced on demand
for (const n of range(0, 10, 3)) {
  console.log(n); // 0, 3, 6, 9
}

// Spread into array
console.log([...range(1, 6)]); // [1, 2, 3, 4, 5]

// yield* delegates to another iterable
function* concat(...iterables) {
  for (const it of iterables) {
    yield* it;
  }
}

console.log([...concat([1, 2], [3, 4])]); // [1, 2, 3, 4]

// Custom iterable with Symbol.iterator
const fibonacci = {
  *[Symbol.iterator]() {
    let [a, b] = [0, 1];
    while (true) {
      yield a;
      [a, b] = [b, a + b];
    }
  }
};
Challenge

Write a generator function* zip(...iterables) that yields arrays pairing the n-th element from each iterable, stopping at the shortest one. Test with arrays and other generators.

Pitfall: Generators are single-use. Once exhausted (or .return() is called), they cannot be restarted. To iterate multiple times, create a new generator each time. Wrapping in an object with [Symbol.iterator] makes it re-iterable.
R3. Event Loop Deep Dive Async Mastery rep 1 · due today
The event loop processes the call stack first, then drains all microtasks (Promise callbacks, queueMicrotask, MutationObserver), then picks one macrotask (setTimeout, setInterval, I/O). Microtasks can starve macrotasks if they keep enqueuing more microtasks.
Key Pattern 
console.log("1: sync");

setTimeout(() => console.log("2: macrotask"), 0);

Promise.resolve().then(() => {
  console.log("3: microtask");
  queueMicrotask(() => console.log("4: nested microtask"));
});

queueMicrotask(() => console.log("5: microtask"));

console.log("6: sync");

// Order: 1: sync, 6: sync, 3: microtask, 5: microtask,
//        4: nested microtask, 2: macrotask
Challenge

Write code that logs numbers 1-5 using a mix of setTimeout, Promise.resolve().then(), queueMicrotask, and synchronous console.log. Arrange them so the output order is exactly 1, 2, 3, 4, 5 despite being written out of order in the source.

Pitfall: setTimeout(fn, 0) does not mean "run immediately" — it means "run in the next macrotask, after all pending microtasks are drained." In browsers, the minimum timeout is clamped to ~4ms after 5 nested calls.
R4. ES Modules Deep Dive Modules & Patterns rep 0 · due today
Static import/export are analyzed at parse time, evaluated once, and cached. Dynamic import() returns a Promise for lazy loading. import.meta.url gives the module's URL. Top-level await blocks the importing module. Barrel files and side effects break tree-shaking.
Key Pattern 
// Static named exports — tree-shakeable
import { add, PI } from "./math.js";

// Dynamic import — lazy loading
const { BarChart } = await import("./charts/bar.js");

// import.meta for module metadata
const configPath = new URL("./config.json", import.meta.url);

// Top-level await — blocks importers until resolved
const db = await connectToDatabase();
export { db };

// Circular deps: wrap access in functions to defer evaluation
import { a } from "./a.js";
export const b = "B";
export function getA() { return a; } // safe — called after init
Challenge

Write a pluginLoader(pluginNames) function that dynamically imports each plugin from ./plugins/${name}.js using import(), collects their default exports, and returns them. Handle the case where a plugin file does not exist (skip it gracefully).

Pitfall: A circular dependency where module A reads an export from module B at the top level, and B reads from A, will cause one of them to see undefined. The module that is evaluated second reads an uninitialized binding. Always use functions to defer access in circular setups.
R5. Optional Chaining & Nullish Coalescing Functions & Iteration rep 1 · due today
?. short-circuits to undefined if any link in the chain is null or undefined. ?? returns the right operand only for null/undefined, not for falsy values like 0, "", or false. This distinction from || is critical.
Key Pattern 
const config = { timeout: 0, retries: null, debug: false };

// ?? vs || — the critical difference
console.log(config.timeout || 3000);  // 3000 (WRONG — 0 is falsy)
console.log(config.timeout ?? 3000);  // 0    (CORRECT)
console.log(config.retries ?? 5);     // 5    (null → use default)
console.log(config.debug ?? true);    // false (false is NOT null)

// ?. with methods, bracket access, and function calls
const user = { greet: () => "hi" };
console.log(user?.greet?.());         // "hi"
console.log(user?.missing?.());       // undefined (no throw)
console.log(user?.["greet"]?.());     // "hi"

// Combining ?. and ??
const port = server?.config?.port ?? 8080;
Challenge

Given an API response that may have nested data.users[0].address.zipCode, write a one-liner using ?. and ?? that extracts the zip code or defaults to "00000". Then write the same logic using plain if checks to appreciate the brevity.

Pitfall: ?? cannot be mixed with && or || without explicit parentheses: a ?? b || c is a SyntaxError. Write (a ?? b) || c or a ?? (b || c) to clarify intent.
R6. Map & Set Data & Collections rep 0 · overdue 1 day
Map preserves insertion order, supports any key type (not just strings), and has O(1) get/set/has. Set stores unique values. Both use reference equality for objects. WeakMap/WeakSet allow GC of keys and are not iterable.
Key Pattern 
// Map — any key type, insertion order preserved
const cache = new Map();
const objKey = { id: 1 };
cache.set(objKey, "cached value");
cache.set(42, "number key");
console.log(cache.get(objKey));  // "cached value"
console.log(cache.size);        // 2

// Set — unique values, fast membership test
const tags = new Set(["js", "ts", "js", "node"]);
console.log(tags.size);            // 3 (duplicate "js" removed)
console.log(tags.has("ts"));       // true
console.log([...tags]);            // ["js", "ts", "node"]

// Set operations
const a = new Set([1, 2, 3, 4]);
const b = new Set([3, 4, 5, 6]);
const union = new Set([...a, ...b]);           // {1,2,3,4,5,6}
const intersection = new Set([...a].filter(x => b.has(x))); // {3,4}
const difference = new Set([...a].filter(x => !b.has(x)));  // {1,2}

// WeakMap — keys must be objects, allows GC
const metadata = new WeakMap();
(function() {
  const el = document.createElement("div");
  metadata.set(el, { clicks: 0 });
  // when el goes out of scope and is GC'd, the entry is auto-removed
})();
Challenge

Implement a LRUCache class using a Map (leveraging insertion order). It should have get(key) and put(key, value) methods with a max capacity. On access, move the entry to the end. On overflow, delete the oldest (first) entry.

Pitfall: Two identical-looking objects are different keys in a Map/Set: new Map().set({a:1}, "x").get({a:1}) returns undefined because {a:1} !== {a:1}. You must use the exact same reference. For value-based equality, stringify the key or use a custom hash.
1 DOM Manipulation
#1. DOM Manipulation Deep Dive Browser APIs & DOM
Every front-end framework ultimately manipulates the DOM. Understanding the raw DOM API — selectors, element creation, fragments, mutation observation, and the performance tradeoffs of innerHTML vs. DOM methods — lets you debug framework behavior, write efficient vanilla JS, and understand what React/Vue do under the hood.
  • 2026-07-09 (current — Day 28)
  • querySelector / querySelectorAll: Accept any CSS selector. querySelectorAll returns a static NodeList (snapshot) — new elements matching the selector are not automatically included.
  • getElementsBy*: Returns a live HTMLCollection that updates when the DOM changes. Useful for tracking, but iterating a live collection while mutating the DOM causes bugs.
  • createElement / createTextNode: Build elements programmatically. Safer than innerHTML because user data is treated as text, not HTML (no XSS).
  • DocumentFragment: An off-screen container for batching DOM updates. Append many elements to a fragment, then insert the fragment in one operation — triggers only one reflow.
  • MutationObserver: Watches for DOM changes (childList, attributes, characterData). More performant than deprecated MutationEvent. Callbacks fire as microtasks.
  • dataset: Access data-* attributes as a DOMStringMap. el.dataset.userId reads data-user-id.
  • innerHTML vs DOM API: innerHTML is fast for bulk HTML but is an XSS vector if user input is interpolated. DOM APIs are safer and allow finer control.
JavaScript — Selectors: Static vs Live Collections
// querySelectorAll — static NodeList (snapshot)
const items = document.querySelectorAll(".item");
console.log(items.length); // e.g., 3

// Adding a new .item does NOT update the static NodeList
const newItem = document.createElement("div");
newItem.className = "item";
document.body.append(newItem);
console.log(items.length); // still 3 — static snapshot

// getElementsByClassName — live HTMLCollection
const liveItems = document.getElementsByClassName("item");
console.log(liveItems.length); // 4 — includes the new element!

// querySelector with complex selectors
const el = document.querySelector('input[type="email"]:not(.disabled)');
const rows = document.querySelectorAll("table.data > tbody > tr:nth-child(odd)");

// Scope queries to a parent element
const nav = document.querySelector("nav");
const navLinks = nav.querySelectorAll("a.nav-link"); // only within nav

// SAFE: createElement — user input is text, not HTML
function createUserCard(user) {
  const card = document.createElement("div");
  card.className = "user-card";

  const name = document.createElement("h3");
  name.textContent = user.name; // safe — even if name is "<script>alert(1)</script>"

  const bio = document.createElement("p");
  bio.textContent = user.bio;

  card.append(name, bio);
  return card;
}

// DANGEROUS: innerHTML — user input becomes live HTML
function createUserCardUnsafe(user) {
  const card = document.createElement("div");
  // XSS vulnerability! If user.name is "<img src=x onerror=alert(1)>"
  card.innerHTML = `<h3>${user.name}</h3><p>${user.bio}</p>`;
  return card;
}

// COMPROMISE: innerHTML with sanitization
function escapeHTML(str) {
  const div = document.createElement("div");
  div.textContent = str;
  return div.innerHTML;
}
card.innerHTML = `<h3>${escapeHTML(user.name)}</h3>`; // safe now
JavaScript — DocumentFragment for Batch Updates
// BAD: appending in a loop triggers reflow on each append
function renderListSlow(items, container) {
  items.forEach(item => {
    const li = document.createElement("li");
    li.textContent = item;
    container.appendChild(li); // reflow on each iteration!
  });
}

// GOOD: batch with DocumentFragment — one reflow
function renderListFast(items, container) {
  const fragment = document.createDocumentFragment();

  items.forEach(item => {
    const li = document.createElement("li");
    li.textContent = item;
    fragment.appendChild(li); // no reflow — fragment is off-screen
  });

  container.appendChild(fragment); // one single reflow
}

// Even better with modern API:
function renderListModern(items, container) {
  const elements = items.map(item => {
    const li = document.createElement("li");
    li.textContent = item;
    return li;
  });
  container.append(...elements); // append accepts multiple nodes
}

const list = document.querySelector("#todo-list");
renderListFast(["Buy groceries", "Walk dog", "Code review"], list);

Common Mistakes
Iterating a live HTMLCollection while adding/removing elements — the collection changes mid-loop, causing skipped or double-processed items
Convert to an array first: [...document.getElementsByClassName("x")], or use querySelectorAll which returns a static NodeList
Using innerHTML with user-supplied data: el.innerHTML = userInput — classic XSS vulnerability
Use textContent for text, createElement for structure, or sanitize with a library like DOMPurify
Appending 1000 elements in a loop directly to the DOM — triggers 1000 reflows
Batch with DocumentFragment or container.append(...elements) for a single reflow
Key Rules
  • querySelectorAll = static snapshot; getElementsBy* = live collection
  • Never interpolate user input into innerHTML — use textContent or createElement
  • Batch DOM mutations with DocumentFragment to minimize reflows
  • MutationObserver callbacks fire as microtasks — they run before the next paint
  • dataset auto-converts camelCase to kebab-case: .dataset.userName = data-user-name

#2. IntersectionObserver: Lazy Loading & Infinite Scroll Browser APIs & DOM
IntersectionObserver replaces expensive scroll event listeners with a performant, browser-native way to detect when elements enter or leave the viewport. It enables lazy loading images, infinite scroll, analytics tracking, and reveal-on-scroll animations without jank or layout thrashing.
  • Basic usage: Create an observer with a callback and options. Call observer.observe(element) to start watching. The callback receives an array of IntersectionObserverEntry objects.
  • threshold: A number or array of numbers between 0 and 1. 0 means "any pixel visible," 1 means "fully visible," [0, 0.5, 1] fires at each milestone.
  • rootMargin: Expands or shrinks the root bounds. "100px" triggers 100px before the element enters the viewport (pre-loading). Accepts CSS-style margin values.
  • root: The element to use as the viewport. Default is null (browser viewport). Set to a scrollable container for local intersection detection.
  • isIntersecting: Boolean on the entry — true when the target is within the root bounds (per the threshold).
  • Cleanup: observer.unobserve(target) stops watching one element. observer.disconnect() stops watching everything.
JavaScript — Basic IntersectionObserver
const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach(entry => {
      console.log(entry.target.id, {
        isIntersecting: entry.isIntersecting,
        ratio: entry.intersectionRatio.toFixed(2),
        boundingRect: entry.boundingClientRect
      });

      if (entry.isIntersecting) {
        entry.target.classList.add("visible");
      } else {
        entry.target.classList.remove("visible");
      }
    });
  },
  {
    root: null,          // viewport
    rootMargin: "0px",   // no margin
    threshold: [0, 0.5, 1] // fire at 0%, 50%, 100% visibility
  }
);

// Observe multiple elements
document.querySelectorAll(".animate-on-scroll").forEach(el => {
  observer.observe(el);
});
JavaScript — Lazy Loading Images
// HTML: <img data-src="photo.jpg" class="lazy" alt="Photo">

function lazyLoadImages() {
  const images = document.querySelectorAll("img.lazy");

  const imageObserver = new IntersectionObserver(
    (entries, observer) => {
      entries.forEach(entry => {
        if (!entry.isIntersecting) return;

        const img = entry.target;
        img.src = img.dataset.src;         // swap placeholder → real src
        img.classList.remove("lazy");
        img.classList.add("loaded");
        observer.unobserve(img);            // done — stop watching this image
      });
    },
    {
      rootMargin: "200px 0px", // start loading 200px before visible
      threshold: 0
    }
  );

  images.forEach(img => imageObserver.observe(img));

  return () => imageObserver.disconnect(); // cleanup function
}

// Initialize
const cleanup = lazyLoadImages();

JavaScript — Infinite Scroll with Sentinel Element
// HTML: <div id="feed">...items...</div><div id="sentinel"></div>

function setupInfiniteScroll(feedEl, sentinelEl, loadMore) {
  let loading = false;
  let page = 1;

  const observer = new IntersectionObserver(
    async (entries) => {
      const entry = entries[0];
      if (!entry.isIntersecting || loading) return;

      loading = true;
      try {
        const newItems = await loadMore(++page);

        if (newItems.length === 0) {
          observer.disconnect();
          sentinelEl.textContent = "No more items";
          return;
        }

        const fragment = document.createDocumentFragment();
        newItems.forEach(item => {
          const div = document.createElement("div");
          div.className = "feed-item";
          div.textContent = item.title;
          fragment.appendChild(div);
        });
        feedEl.appendChild(fragment);
      } finally {
        loading = false;
      }
    },
    {
      rootMargin: "300px", // trigger 300px before reaching bottom
      threshold: 0
    }
  );

  observer.observe(sentinelEl);
  return () => observer.disconnect();
}

function setupScrollAnimations() {
  const observer = new IntersectionObserver(
    (entries) => {
      entries.forEach(entry => {
        if (entry.isIntersecting) {
          // Add animation class when element enters viewport
          entry.target.classList.add("animate-in");
          // Optional: unobserve for one-time animations
          observer.unobserve(entry.target);
        }
      });
    },
    {
      threshold: 0.15,      // trigger when 15% visible
      rootMargin: "0px 0px -50px 0px" // slight offset from bottom
    }
  );

  document.querySelectorAll("[data-animate]").forEach(el => {
    observer.observe(el);
  });

  return () => observer.disconnect();
}

Common Mistakes
Using scroll event listeners for visibility detection: window.addEventListener("scroll", checkVisibility) fires on every pixel of scroll and causes jank
Use IntersectionObserver — it runs off the main thread and batches notifications efficiently
Forgetting to unobserve one-shot targets (like lazy-loaded images) — the observer keeps tracking them uselessly
Call observer.unobserve(entry.target) inside the callback once the action is complete
Setting threshold: 1 for lazy loading — the callback only fires when 100% of the element is visible, which may never happen for tall elements
Use threshold: 0 (any pixel visible) for lazy loading. Reserve higher thresholds for analytics or animation triggers.
Key Rules
  • IntersectionObserver is asynchronous and off-main-thread — no scroll jank
  • Use rootMargin to pre-load content before it enters the viewport
  • Always unobserve one-time targets; disconnect when the observer is no longer needed
  • threshold: 0 = "any pixel"; threshold: 1 = "fully visible"
  • The callback receives an array — always iterate entries, even if you observe one element

2 Browser Performance APIs
#3. ResizeObserver & PerformanceObserver Browser APIs & DOM
ResizeObserver lets you build responsive components that react to their own size changes — independent of the viewport, without media queries. PerformanceObserver gives programmatic access to Web Vitals (LCP, FID, CLS) and custom performance marks, enabling you to measure what matters and catch regressions in production.
  • ResizeObserver: Fires when an element's content box or border box changes size. Unlike window.resize, it works per-element. Use for responsive components, chart resizing, and container queries polyfills.
  • contentRect: The entry's content box dimensions (width, height, x, y). Available on every ResizeObserverEntry.
  • borderBoxSize / contentBoxSize: Newer properties with inlineSize and blockSize for writing-mode-aware sizing. Returns an array (for fragmented elements).
  • PerformanceObserver: Observes performance entries like paint, largest-contentful-paint, first-input, layout-shift, and custom mark/measure entries.
  • performance.mark() / performance.measure(): Create named timestamps and measure durations between them. Visible in DevTools Performance panel.
  • PerformanceNavigationTiming: Detailed page load waterfall — DNS, TCP, TLS, request, response, DOM processing, load event.

Common Mistakes

  • PerformanceObserver with { buffered: true } catches past + future entries
  • Use performance.mark()/measure() for custom timing — visible in DevTools

#4. Fetch Advanced: Interceptors, Streaming & Cloning Browser APIs & DOM
The Fetch API is more than fetch(url). Building interceptors enables consistent auth/logging/retry logic. Streaming responses lets you process data as it arrives (like ChatGPT-style token streaming). Cloning allows reading a body multiple times. These patterns are the foundation of robust HTTP layers in production apps.
  • Interceptor pattern: Wrap fetch with a function that modifies requests (add headers, auth tokens) and responses (parse, log, retry) consistently.
  • response.clone(): Response bodies are streams — they can only be read once. clone() creates a copy so you can read the body multiple times (e.g., cache one copy, return another).
  • Streaming: response.body is a ReadableStream. Use .getReader() to read chunks as they arrive. Essential for large downloads and server-sent streaming (SSE, NDJSON).
  • Progress tracking: Pipe the stream through a TransformStream that counts bytes to report download progress.
  • Request object: Create reusable request configurations. Modify headers, method, and body without mutating the original.
  • AbortController integration: Pass signal to cancel in-flight requests. Combine with AbortSignal.timeout() for automatic timeouts.
JavaScript — Building a Fetch Interceptor
function createFetchClient({ baseURL = "", headers = {}, onRequest, onResponse, onError } = {}) {
  return async function fetchClient(path, options = {}) {
    let request = new Request(`${baseURL}${path}`, {
      ...options,
      headers: { ...headers, ...options.headers }
    });

    // Pre-request interceptor
    if (onRequest) {
      request = await onRequest(request);
    }

    try {
      let response = await fetch(request);

      // Post-response interceptor
      if (onResponse) {
        response = await onResponse(response, request);
      }

      return response;
    } catch (error) {
      if (onError) {
        return onError(error, request);
      }
      throw error;
    }
  };
}

// Create a configured client
const api = createFetchClient({
  baseURL: "https://api.example.com",
  headers: { "Content-Type": "application/json" },

  onRequest(req) {
    const token = localStorage.getItem("auth_token");
    if (token) {
      const headers = new Headers(req.headers);
      headers.set("Authorization", `Bearer ${token}`);
      return new Request(req, { headers });
    }
    return req;
  },

  onResponse(res, req) {
    if (res.status === 401) {
      localStorage.removeItem("auth_token");
      window.location.href = "/login";
    }
    return res;
  },

  onError(error, req) {
    console.error(`Fetch failed: ${req.method} ${req.url}`, error);
    throw error;
  }
});

  // const data = await response.json(); // reads the body

}

JavaScript — Streaming Responses: Reading Chunks
// Stream a large response chunk by chunk
async function fetchStream(url) {
  const response = await fetch(url);
  const reader = response.body.getReader();
  const decoder = new TextDecoder();

  let result = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value, { stream: true });
    result += chunk;
    console.log("Received chunk:", chunk.length, "chars");
  }

  return result;
}

// Stream NDJSON (newline-delimited JSON) — like server-sent events
async function* streamNDJSON(url) {
  const response = await fetch(url);
  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split("\n");
    buffer = lines.pop(); // keep incomplete last line

    for (const line of lines) {
      if (line.trim()) {
        yield JSON.parse(line);
      }
    }
  }

  // Process remaining buffer
  if (buffer.trim()) {
    yield JSON.parse(buffer);
  }
}

JavaScript — Download Progress with TransformStream
async function fetchWithProgress(url, onProgress) {
  const response = await fetch(url);
  const contentLength = +response.headers.get("Content-Length") || 0;
  let loaded = 0;

  // Create a TransformStream that tracks bytes
  const progressStream = new TransformStream({
    transform(chunk, controller) {
      loaded += chunk.length;
      if (contentLength) {
        onProgress({ loaded, total: contentLength, percent: (loaded / contentLength) * 100 });
      } else {
        onProgress({ loaded, total: null, percent: null });
      }
      controller.enqueue(chunk); // pass chunk through
    }
  });

  // Pipe response through the progress tracker
  const trackedResponse = new Response(
    response.body.pipeThrough(progressStream),
    response
  );

  return trackedResponse;
}

// Timeout + manual abort combined
async function fetchWithTimeout(url, timeoutMs = 5000) {
  const controller = new AbortController();

  // Combine manual abort with timeout
  const timeoutId = setTimeout(() => controller.abort("Timeout"), timeoutMs);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);
    return response;
  } catch (error) {
    clearTimeout(timeoutId);
    if (error.name === "AbortError") {
      throw new Error(`Request to ${url} timed out after ${timeoutMs}ms`);
    }
    throw error;
  }
}

// Or use the built-in AbortSignal.timeout (simpler)
const response = await fetch(url, {
  signal: AbortSignal.timeout(5000)
});
Common Mistakes
Reading response.json() and then response.text() on the same response — the body stream is consumed on first read
Use response.clone() before reading the body if you need to read it more than once
Forgetting that fetch does not reject on HTTP errors (404, 500) — only on network failures
Always check response.ok or response.status before reading the body: if (!response.ok) throw new Error(...)
Not handling AbortError separately from other errors in a try/catch — treating cancellation as a failure
Check error.name === "AbortError" and handle gracefully (e.g., ignore in UI, don't show error toast)
Key Rules
  • Response bodies are one-shot streams — clone() before reading if you need multiple reads
  • Build interceptors around fetch for consistent auth, logging, and error handling
  • Use response.body.getReader() for streaming large responses chunk by chunk
  • fetch only rejects on network errors — always check response.ok
  • Combine AbortController with fetch for cancellation and timeouts

#5. Web Storage & IndexedDB Browser APIs & DOM
Knowing when to use localStorage, sessionStorage, IndexedDB, cookies, or the Cache API determines whether your app is fast, secure, and reliable offline. Each has different capacity limits, persistence behavior, and API ergonomics. Choosing wrong leads to quota errors, security holes, or unnecessary complexity.
  • localStorage: Synchronous key-value store. Persists until explicitly cleared. ~5-10MB limit. Shared across all tabs on the same origin. Values are always strings (use JSON.stringify/parse).
  • sessionStorage: Same API as localStorage, but data is scoped to the tab/window and cleared when it closes. Not shared between tabs.
  • Storage event: When one tab modifies localStorage, other tabs on the same origin receive a storage event. Enables cross-tab communication without WebSockets.
  • IndexedDB: Asynchronous, transactional, object-oriented database. Stores structured data, blobs, files. No practical size limit (browser asks for permission above ~50MB). Supports indexes for fast queries.
  • IDB with Promises: The native IndexedDB API uses callbacks/events. Wrap it with Promises for cleaner async/await code, or use a library like idb.
  • When to use what: localStorage for small config/preferences. sessionStorage for per-tab state. IndexedDB for large/structured data. Cookies for server-sent tokens. Cache API for HTTP response caching (Service Workers).

JavaScript — IndexedDB: Basic CRUD with Promises
// Promise wrapper for IndexedDB
function openDB(name, version, onUpgrade) {
  return new Promise((resolve, reject) => {
    const request = indexedDB.open(name, version);
    request.onerror = () => reject(request.error);
    request.onsuccess = () => resolve(request.result);
    request.onupgradeneeded = (event) => {
      onUpgrade(event.target.result, event.oldVersion);
    };
  });
}

// Setup schema

// Generic transaction helper
function idbTransaction(db, storeName, mode, callback) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction(storeName, mode);
    const store = tx.objectStore(storeName);
    const result = callback(store);

    tx.oncomplete = () => resolve(result);
    tx.onerror = () => reject(tx.error);
  });
}

JavaScript — IndexedDB: Cursors and Bulk Operations
// Iterate all records with a cursor
async function getAllUsers(db) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction("users", "readonly");
    const store = tx.objectStore("users");
    const request = store.openCursor();
    const results = [];

    request.onsuccess = (event) => {
      const cursor = event.target.result;
      if (cursor) {
        results.push(cursor.value);
        cursor.continue(); // move to next record
      } else {
        resolve(results); // no more records
      }
    };
    request.onerror = () => reject(request.error);
  });
}

// Cursor with range — query users by name range
async function getUsersInRange(db, from, to) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction("users", "readonly");
    const index = tx.objectStore("users").index("name");
    const range = IDBKeyRange.bound(from, to);
    const request = index.openCursor(range);
    const results = [];

    request.onsuccess = (event) => {
      const cursor = event.target.result;
      if (cursor) {
        results.push(cursor.value);
        cursor.continue();
      } else {
        resolve(results);
      }
    };
    request.onerror = () => reject(request.error);
  });
}

// Bulk insert in a single transaction
async function bulkInsert(db, users) {
  return new Promise((resolve, reject) => {
    const tx = db.transaction("users", "readwrite");
    const store = tx.objectStore("users");

    users.forEach(user => store.put(user)); // put = upsert

    tx.oncomplete = () => resolve();
    tx.onerror = () => reject(tx.error);
  });
}
JavaScript — When to Use Which Storage
// Decision guide:

// localStorage (~5-10MB, sync, string-only, persists forever)
//   - User preferences (theme, language, sidebar state)
//   - Auth tokens (if not using httpOnly cookies)
//   - Small caches with manual expiration

// sessionStorage (~5-10MB, sync, string-only, per-tab, clears on close)
//   - Form drafts, wizard state, per-tab UI state
//   - Temporary data that shouldn't persist across sessions

// IndexedDB (large, async, structured data, persists)
//   - Offline app data (full database)
//   - Large file/blob storage
//   - Complex queries with indexes
//   - Data that exceeds localStorage limits

// Cookies (~4KB, sent with every HTTP request, server-accessible)
//   - Session tokens (httpOnly, secure, sameSite)
//   - Server-side preferences
//   - NEVER for large data (sent on EVERY request!)

// Cache API (used with Service Workers, stores Response objects)
//   - HTTP response caching for offline support
//   - Precaching static assets
//   - API response caching with custom strategies

// Summary table:
// | Storage        | Size    | API   | Persists  | Shared  | Use Case           |
// |----------------|---------|-------|-----------|---------|--------------------|
// | localStorage   | ~5MB    | sync  | forever   | tabs    | prefs, small cache |
// | sessionStorage | ~5MB    | sync  | tab close | no      | per-tab state      |
// | IndexedDB      | large   | async | forever   | tabs    | structured data    |
// | Cookies        | ~4KB    | sync  | expiry    | server  | auth tokens        |
// | Cache API      | large   | async | forever   | SW      | HTTP caching       |
Common Mistakes
Storing sensitive data (passwords, tokens) in localStorage without understanding XSS risk — any script on the page can read it
Use httpOnly cookies for auth tokens when possible. If using localStorage, ensure strong XSS protections (CSP, input sanitization).
Forgetting that localStorage only stores strings — storing an object directly gives "[object Object]"
Always JSON.stringify() when storing and JSON.parse() when reading. Wrap in try/catch for corrupt data.
Using localStorage for large datasets (>5MB) and hitting QuotaExceededError
Use IndexedDB for anything larger than a few KB of structured data. It has virtually no size limit.
Key Rules
  • localStorage is synchronous and blocks the main thread — keep stored data small
  • Always wrap JSON.parse(localStorage.getItem(...)) in try/catch
  • The storage event only fires in other tabs, not the tab that made the change
  • IndexedDB is transactional — group related operations in a single transaction
  • Use cookies for auth tokens (httpOnly), localStorage for preferences, IndexedDB for data
