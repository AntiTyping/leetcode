this Binding Core Language 7 days since review · rep 2
Four rules, in priority order: new binding > explicit (call/apply/bind) > implicit (method call obj.fn()) > default (globalThis, or undefined in strict mode). Arrow functions ignore all four and inherit this lexically from the enclosing scope.
Key Pattern
JavaScript
const obj = { x: 42, get() { return this.x; } };
const loose = obj.get;       // implicit binding lost
const bound = obj.get.bind(obj); // re-attach this
console.log(bound());        // 42
Challenge

Write an EventEmitter class whose on(event, fn) and emit(event, ...args) methods still work when destructured: const { on, emit } = new EventEmitter();. (Hint: arrow-function class fields, or bind in the constructor.)

#1 Pitfall: Passing obj.method as a callback (e.g. arr.forEach(obj.method) or setTimeout(obj.method, 0)) detaches it from obj, so this becomes undefined/globalThis.
Prototypes & Inheritance Core Language 7 days since review · rep 2
Every object has an internal [[Prototype]] link. Property lookup walks that chain until found or it hits null. Object.create(proto) sets the prototype directly; class is sugar over the same machinery (Child.prototype.__proto__ === Parent.prototype).
Key Pattern
JavaScript
const animal = { speak() { return `${this.name} makes a sound`; } };
const dog = Object.create(animal, {
  name: { value: "Rex", enumerable: true }
});
console.log(dog.speak()); // "Rex makes a sound"
Challenge

Implement mixin(target, ...sources) that copies own method definitions from each source's prototype onto target.prototype without breaking target's existing prototype chain. Verify with two source classes that each contribute a method.

#1 Pitfall: Monkey-patching built-ins like Array.prototype.foo = ... — it leaks into every array everywhere, breaks for...in, and collides with future language additions.
Closures Deep Dive Core Language 7 days since review · rep 2
A function closes over the variable bindings of its lexical scope, not snapshots of values. Each function instance created in a scope gets its own live reference to those bindings.
Key Pattern
JavaScript
function counter() {
  let n = 0;
  return () => ++n; // closes over n
}
const next = counter();
console.log(next(), next(), next()); // 1 2 3
Challenge

Write memoize(fn) that caches results in a Map held by a closure. It must support any number of arguments — use JSON.stringify(args) as the cache key. Prove it works by counting how many times the wrapped function actually executes.

#1 Pitfall: Closing over a var loop variable — all callbacks share the same i and print the final value. Use let (fresh binding per iteration) or an IIFE.
Symbols & Well-Known Symbols Core Language 7 days since review · rep 2
Symbol() mints a unique, non-string key — useful for collision-free properties. Well-known symbols hook into language behavior: Symbol.iterator (for...of), Symbol.toPrimitive (coercion), Symbol.hasInstance (instanceof).
Key Pattern
JavaScript
const range = {
  [Symbol.iterator]() {
    let i = 1;
    return { next: () => i <= 3 ? { value: i++, done: false }
                              : { value: undefined, done: true } };
  }
};
console.log([...range]); // [1, 2, 3]
Challenge

Create a Range(start, end) class implementing both Symbol.iterator (yields each integer) and Symbol.toPrimitive (returns the element count when coerced to a number). Verify: [...new Range(2, 5)] is [2,3,4,5] and +new Range(2, 5) is 4.

#1 Pitfall: Assuming the description means equality — Symbol('x') !== Symbol('x'). The description is purely for debugging. Use Symbol.for('x') if you need a shared, registry-backed symbol.
Generators & Iterators Functions & Iteration 4 days since review · rep 2
function* creates a generator; yield pauses and emits a value, resuming on the next .next(). Generators are lazy iterators — nothing runs until pulled. yield* delegates to another iterable.
Key Pattern
JavaScript
function* range(start, end) {
  for (let i = start; i < end; i++) yield i;
}
console.log([...range(0, 4)]); // [0, 1, 2, 3]
Challenge

Write a generator flatten(arr, depth = Infinity) that lazily yields the elements of a nested array, flattened to the given depth. It should use yield* to recurse and must not build an intermediate flat array. Test with [...flatten([1, [2, [3, [4]]]], 2)] → [1, 2, 3, [4]].

#1 Pitfall: Generators are single-use. Once exhausted, iterating again yields nothing — you must call the generator function again to get a fresh iterator.
Event Loop Deep Dive Async Mastery 4 days since review · rep 2
Order: run synchronous code to completion → drain the entire microtask queue (Promise callbacks, queueMicrotask) → run one macrotask (setTimeout, I/O) → drain microtasks again, repeat. Microtasks always finish before the next macrotask.
Key Pattern
JavaScript
console.log("A");
setTimeout(() => console.log("D"), 0);     // macrotask
Promise.resolve().then(() => console.log("C")); // microtask
console.log("B");
// Order: A B C D
Challenge

Predict the exact output order, then run it to check:

JavaScript
console.log(1);
setTimeout(() => console.log(2), 0);
queueMicrotask(() => console.log(3));
Promise.resolve().then(() => console.log(4)).then(() => console.log(5));
console.log(6);
#1 Pitfall: Treating setTimeout(fn, 0) as "run now." It is a macrotask — it runs only after the current synchronous code and all pending microtasks have finished.
1 Browser APIs & DOM
DOM Manipulation & DocumentFragment Browser APIs & DOM

Direct DOM manipulation is the bedrock under every UI framework. Understanding it deeply lets you debug React/Vue render bugs, write framework-free widgets, and avoid the reflow storms that make pages janky.

The core verbs are small. You find nodes, you create nodes, and you insert them. Modern APIs make all three ergonomic:

  • 2026-07-09 (Day 28)
  • Finding: document.querySelector(sel) returns the first match (or null); querySelectorAll(sel) returns a static NodeList you can spread into an array.
  • Creating: document.createElement('li'), then set el.textContent / el.className / el.dataset.x. Use append() (accepts multiple nodes and strings) and prepend().
  • Batching: A DocumentFragment is a lightweight, off-screen container. You build everything inside it, then insert it once — the browser does a single reflow instead of one per node.
  • Watching: A MutationObserver asynchronously reports added/removed nodes and attribute changes, batched into microtask-timed callbacks.

The progression below goes: query & modify → create dynamically → batch with a fragment → observe changes.

JavaScript — query and modify
// HTML: <ul id="todos"><li>Buy milk</li><li>Walk dog</li></ul>
const list = document.querySelector("#todos");
const items = [...list.querySelectorAll("li")]; // static NodeList -> array

items.forEach((li, i) => {
  li.textContent = `${i + 1}. ${li.textContent}`;
  li.dataset.index = i;            // sets data-index attribute
});
console.log(items.map(li => li.textContent));
[ '1. Buy milk', '2. Walk dog' ]
JavaScript — create elements dynamically
function makeCard(title, body) {
  const card = document.createElement("div");
  card.className = "card";

  const h = document.createElement("h3");
  h.textContent = title;

  const p = document.createElement("p");
  p.textContent = body;

  card.append(h, p);   // append accepts multiple children at once
  return card;
}

const node = makeCard("Hello", "World");
console.log(node.outerHTML);
<div class="card"><h3>Hello</h3><p>World</p></div>
JavaScript — batch insert with DocumentFragment
const data = ["Alpha", "Beta", "Gamma"];
const frag = document.createDocumentFragment(); // off-screen, no reflow

for (const name of data) {
  const li = document.createElement("li");
  li.textContent = name;
  frag.append(li);            // mutating the fragment is cheap
}

document.querySelector("#todos").append(frag); // ONE reflow here
// The fragment itself is now empty; its children moved into the DOM.
<ul> now ends with <li>Alpha</li><li>Beta</li><li>Gamma</li>
JavaScript — MutationObserver watching for child additions
const container = document.querySelector("#todos");

const observer = new MutationObserver((mutations) => {
  for (const m of mutations) {
    m.addedNodes.forEach(n => console.log("added:", n.textContent));
  }
});

observer.observe(container, { childList: true }); // watch direct children

const li = document.createElement("li");
li.textContent = "Delta";
container.append(li);
// Callback fires asynchronously (microtask), AFTER current sync code.
// Later: observer.disconnect();
added: Delta
Common Mistakes
❌ Appending nodes one at a time inside a loop directly into the live DOM: for (...) container.append(makeNode()) — each append can trigger layout/reflow, which is O(n) reflows.
✅ Build into a DocumentFragment (or an array passed to a single append(...nodes)) and insert once — a single reflow.
Key Rules
  • Use textContent for plain text (safe, fast). Reserve innerHTML for trusted markup — it is an XSS vector with user input.
  • Batch DOM writes with a DocumentFragment whenever you insert more than a couple of nodes in a loop.
  • Always disconnect() a MutationObserver when you no longer need it — otherwise it keeps the target alive and keeps firing.
Practice Exercises
1 Write renderList(items) that takes an array of strings and returns a <ul> element with one <li> per item, built using a DocumentFragment. Input: ["a","b","c"] → an element whose outerHTML is <ul><li>a</li><li>b</li><li>c</li></ul>.
2 Set up a MutationObserver on a container that console.logs a message every time a new child element is added. Then append three children in a loop and confirm how many times (and when) the callback fires.
3 Harder: write replaceChildren(container, items) that clears container and re-renders it from an array, doing only a single DOM insertion. (Look into the native Element.replaceChildren() after attempting it yourself.)
Connections: DocumentFragment batching is the manual version of what virtual-DOM diffing automates. MutationObserver is the DOM cousin of ResizeObserver/PerformanceObserver and IntersectionObserver — all async, all batched. It also pairs with Web Components, whose lifecycle callbacks observe the same mutations natively.
IntersectionObserver Browser APIs & DOM

It efficiently tells you when an element enters or leaves the viewport — the engine behind lazy-loaded images, infinite scroll, sticky-header detection, and impression analytics. It replaces hand-rolled scroll listeners that fire hundreds of times a second.

You create an observer with a callback and an options object, then call observe(element) for each element you care about:

  • root: the scroll container to measure against (null = the browser viewport).
  • rootMargin: grows/shrinks the root box, e.g. "200px" fires before the element is actually visible (pre-loading).
  • threshold: a ratio (or array of ratios) of visibility at which the callback fires. 0 = any pixel visible; 1 = fully visible; [0, 0.5, 1] = three checkpoints.

The callback receives an array of entries. Each entry has target, the boolean isIntersecting, and a numeric intersectionRatio. Crucially, it is asynchronous and batched — the browser computes intersections off the main render path, so it does not stall scrolling.

JavaScript — basic visibility detection
const io = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    console.log(entry.target.id, entry.isIntersecting
      ? "entered view"
      : "left view");
  }
}, { threshold: 0 });

io.observe(document.querySelector("#section-a"));
// Scrolling #section-a into view -> "section-a entered view"
section-a entered view section-a left view
JavaScript — lazy loading images
// HTML: <img data-src="/big.jpg" class="lazy" />
const lazyIO = new IntersectionObserver((entries, observer) => {
  for (const entry of entries) {
    if (!entry.isIntersecting) continue;
    const img = entry.target;
    img.src = img.dataset.src;     // swap placeholder -> real src
    img.removeAttribute("data-src");
    observer.unobserve(img);       // load once, then stop watching
  }
}, { rootMargin: "200px" });       // start loading 200px early

document.querySelectorAll("img.lazy").forEach(img => lazyIO.observe(img));
Each image's src is set just before it scrolls into view.
JavaScript — infinite scroll trigger
// A zero-height sentinel sits at the bottom of the list.
const sentinel = document.querySelector("#load-more");
let page = 0;

const scrollIO = new IntersectionObserver(async ([entry]) => {
  if (!entry.isIntersecting) return;
  page++;
  console.log(`loading page ${page}...`);
  // const rows = await fetch(`/api/items?page=${page}`).then(r => r.json());
  // appendRows(rows);  // append, then the sentinel moves down again
});

scrollIO.observe(sentinel);
loading page 1... loading page 2... (as the user keeps scrolling)
JavaScript — multiple thresholds for progress tracking
const progressIO = new IntersectionObserver((entries) => {
  for (const e of entries) {
    const pct = Math.round(e.intersectionRatio * 100);
    console.log(`${e.target.id} is ${pct}% visible`);
  }
}, { threshold: [0, 0.25, 0.5, 0.75, 1] });

progressIO.observe(document.querySelector("#hero"));
hero is 25% visible hero is 50% visible hero is 100% visible
Common Mistakes
❌ Using a scroll event listener to detect visibility: window.addEventListener("scroll", check) — it fires continuously on the main thread, forces synchronous layout reads (getBoundingClientRect), and needs manual throttling.
✅ Let IntersectionObserver do it — async, batched, off the main render path, and it self-throttles via threshold.
Key Rules
  • Call unobserve(el) (or disconnect()) once you no longer need an element — e.g. after a lazy image has loaded.
  • Use rootMargin to pre-trigger work before the user sees the element (smoother lazy loading).
  • Reach for multiple threshold values only when you need progress; a single 0 is enough for "is it on screen?"
Practice Exercises
1 Implement lazy image loading: given several <img class="lazy" data-src="...">, swap data-src into src when each becomes visible, and unobserve it afterward.
2 Build an infinite scroll: observe a sentinel <div> at the end of a list; when it intersects, log "load next page" and (conceptually) append more rows so the sentinel moves down.
3 Harder: write onVisibleOnce(el, fn) that calls fn exactly once the first time el is at least 50% visible, then cleans up its observer. Return a function that cancels it early.
Connections: Same observer family as ResizeObserver/PerformanceObserver and the DOM's MutationObserver — learn the create/observe/disconnect shape once and it transfers. Pairs naturally with IndexedDB caching for offline-friendly infinite scroll.
ResizeObserver & PerformanceObserver Browser APIs & DOM

ResizeObserver detects when an element (not just the window) changes size — essential for container-driven responsive components. PerformanceObserver surfaces real-user metrics (LCP, layout shifts, resource timing) so you can measure performance in production instead of guessing.

ResizeObserver watches one or more elements and fires when their content-box (or border-box) dimensions change, for any reason — window resize, sibling layout, content reflow, CSS changes. Each entry exposes contentRect and the newer borderBoxSize / contentBoxSize arrays.

PerformanceObserver subscribes to entries on the performance timeline by type: "largest-contentful-paint", "layout-shift", "resource", "longtask", plus your own "mark"/"measure" entries created with performance.mark() and performance.measure().

JavaScript — ResizeObserver on a container
const box = document.querySelector("#panel");

const ro = new ResizeObserver((entries) => {
  for (const entry of entries) {
    const { width, height } = entry.contentRect;
    console.log(`panel: ${Math.round(width)} x ${Math.round(height)}`);
  }
});

ro.observe(box);
// Resizing the panel (or the window) logs new dimensions.
// Later: ro.unobserve(box);
panel: 600 x 200 panel: 480 x 200 (after the layout narrowed)
JavaScript — responsive component that adapts layout
const grid = document.querySelector("#cards");

new ResizeObserver(([entry]) => {
  const w = entry.contentRect.width;
  // container queries in JS: switch layout by the element's own width
  grid.classList.toggle("compact", w < 500);
  grid.classList.toggle("wide", w >= 500);
}).observe(grid);
grid gets class "compact" below 500px, "wide" at/above — independent of window size.
JavaScript — PerformanceObserver for LCP
const lcpObserver = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log("LCP candidate:", Math.round(entry.startTime), "ms",
                entry.element?.tagName);
  }
});

// buffered: true replays entries that fired before we subscribed
lcpObserver.observe({ type: "largest-contentful-paint", buffered: true });
LCP candidate: 1240 ms IMG
JavaScript — custom marks and measures
const po = new PerformanceObserver((list) => {
  for (const e of list.getEntries()) {
    console.log(`${e.name}: ${e.duration.toFixed(1)}ms`);
  }
});
po.observe({ entryTypes: ["measure"] });

performance.mark("parse-start");
// ... expensive work ...
performance.mark("parse-end");
performance.measure("parse", "parse-start", "parse-end");
parse: 18.4ms
Common Mistakes
❌ Using window.onresize to react to a component's size change — it only fires on viewport resize and misses changes caused by content, siblings, or CSS.
✅ Use ResizeObserver, which fires on any element's actual dimension change, regardless of cause.
Common Mistakes
❌ Writing to an element's size inside a ResizeObserver callback in a way that re-triggers it — this causes the "ResizeObserver loop limit exceeded" warning and can thrash layout.
✅ Avoid resizing the observed element from within the callback, or guard the write so it only happens when the value actually needs to change.
Key Rules
  • Prefer ResizeObserver over window.onresize for any element-level responsiveness.
  • Use { buffered: true } with PerformanceObserver so you don't miss entries (like LCP) that occur before your code runs.
  • Pair performance.mark() + performance.measure() to time your own code paths in real users, not just devtools.
Practice Exercises
1 Create a component that displays its own current width × height as text, updating live via a ResizeObserver as the element is resized.
2 Use a PerformanceObserver with type: "resource" to log the name and duration of every resource that took longer than 500ms to load.
3 Harder: wrap an async function with timed(name, fn) that surrounds the call with performance.mark/measure and reports the duration through a single shared PerformanceObserver.
Connections: Completes the observer trio with IntersectionObserver and MutationObserver. PerformanceObserver marks/measures are the production counterpart to console.time from the earlier debugging lesson. Container-query behavior here previews CSS container queries.
2 Web Storage & Persistence
Web Storage & IndexedDB Web Storage & Persistence

Client-side storage enables offline experiences, response caching, and fewer server round-trips. Picking the right API — quick key/value vs. structured queryable data — is a core architectural decision for any serious web app.

Two families, different jobs:

  • Web Storage (localStorage, sessionStorage): synchronous string-only key/value, ~5–10MB. localStorage persists across sessions; sessionStorage lives for one tab/session. Great for small flags, theme, draft form state.
  • IndexedDB: asynchronous, transactional, object-oriented database. Effectively unlimited (quota-based), supports multiple object stores, indexes, and range queries. Great for large or structured datasets, offline caches, and anything you need to query.

Web Storage stores only strings, so you serialize with JSON.stringify on the way in and JSON.parse on the way out. IndexedDB's native API is event-based and verbose, so it is almost always wrapped in promises.

JavaScript — localStorage with JSON serialization
const user = { id: 7, name: "Ada", roles: ["admin"] };

localStorage.setItem("user", JSON.stringify(user)); // must serialize
const saved = JSON.parse(localStorage.getItem("user"));

console.log(saved.name, saved.roles);
console.log(typeof localStorage.getItem("user")); // always "string"
Ada [ 'admin' ] string
JavaScript — sessionStorage for form state
const form = document.querySelector("#signup");

// Restore on load
const draft = JSON.parse(sessionStorage.getItem("draft") || "{}");
if (draft.email) form.email.value = draft.email;

// Persist on every input (survives reload, not new tab)
form.addEventListener("input", () => {
  sessionStorage.setItem("draft", JSON.stringify({ email: form.email.value }));
});
Reloading the tab keeps the half-filled form; closing it clears the draft.
JavaScript — IndexedDB: open and create a store
function openDB() {
  return new Promise((resolve, reject) => {
    const req = indexedDB.open("app", 1);
    req.onupgradeneeded = () => {        // runs on version change / first open
      const db = req.result;
      const store = db.createObjectStore("notes", { keyPath: "id" });
      store.createIndex("byTag", "tag", { unique: false });
    };
    req.onsuccess = () => resolve(req.result);
    req.onerror = () => reject(req.error);
  });
}
Resolves to an IDBDatabase with a "notes" store keyed by id and an index on tag.
JavaScript — IndexedDB CRUD with a promise wrapper
function tx(db, store, mode, fn) {
  return new Promise((resolve, reject) => {
    const t = db.transaction(store, mode);
    const req = fn(t.objectStore(store));
    t.oncomplete = () => resolve(req?.result);
    t.onerror = () => reject(t.error);
  });
}

const db = await openDB();
await tx(db, "notes", "readwrite", s => s.put({ id: 1, tag: "todo", text: "ship" }));
const note = await tx(db, "notes", "readonly", s => s.get(1));
console.log(note);
{ id: 1, tag: 'todo', text: 'ship' }
Common Mistakes
❌ Storing an object directly: localStorage.setItem("u", user) — it is coerced via String() and you get the literal "[object Object]".
✅ Always JSON.stringify on write and JSON.parse on read. Guard the read with a fallback (|| "{}") in case the key is missing.
Key Rules
  • Reach for localStorage/sessionStorage for small, simple, synchronous key/value needs. Never put large blobs there — it blocks the main thread.
  • Use IndexedDB for large, structured, or queryable data, and for anything you want available offline. Wrap its event API in promises.
  • Web Storage is per-origin and synchronous; assume reads/writes can fail (private mode, quota) and handle the throw.
Practice Exercises
1 Build a Storage wrapper with set(key, value, ttlMs) and get(key) that auto-serializes JSON and supports TTL expiry: get returns null (and deletes the key) once the stored timestamp + TTL has passed.
2 Use IndexedDB to store 1000 objects of the shape { id, name, dept } and query all records in a given dept using a secondary index.
3 Harder: write a cacheFetch(url, ttlMs) that returns cached JSON from IndexedDB if fresh, otherwise fetches, stores it with a timestamp, and returns the new data.
Connections: IndexedDB pairs with IntersectionObserver-driven infinite scroll for offline caching, and is the persistence layer Service Workers use for offline-first apps. JSON.stringify/parse here ties back to the JSON Advanced lesson (replacer/reviver, toJSON).
3 Browser APIs & DOM — Web Components
Web Components Browser APIs & DOM

Web Components are framework-agnostic, reusable elements native to the platform. A <user-card> you author works unchanged in React, Vue, Svelte, or plain HTML — no build step, no framework lock-in, with real style and DOM encapsulation.

Three platform features combine:

  • Custom Elements: customElements.define("user-card", UserCard) registers a class that extends HTMLElement. Lifecycle callbacks fire automatically: connectedCallback (inserted into DOM), disconnectedCallback (removed), attributeChangedCallback (a watched attribute changed).
  • Shadow DOM: this.attachShadow({ mode: "open" }) gives the element a private subtree. Styles inside it don't leak out, and outside styles don't bleed in — true encapsulation.
  • Templates & Slots: <template> holds inert markup you clone; <slot> projects light-DOM children (the content the consumer puts between your tags) into the shadow tree.

To observe attributes you must declare them in a static observedAttributes getter; only those trigger attributeChangedCallback.

JavaScript — minimal custom element
class HelloTag extends HTMLElement {
  connectedCallback() {              // fires when added to the DOM
    this.textContent = "Hello from a custom element!";
  }
}
customElements.define("hello-tag", HelloTag);

// Usage in HTML:  <hello-tag></hello-tag>
document.body.append(document.createElement("hello-tag"));
<hello-tag>Hello from a custom element!</hello-tag>
JavaScript — Shadow DOM with scoped styles
class BadgeChip extends HTMLElement {
  constructor() {
    super();                              // MUST be first
    const root = this.attachShadow({ mode: "open" });
    root.innerHTML = `
      <style>
        span { background: #58a6ff; color: #000; padding: 2px 8px;
               border-radius: 12px; font: 12px sans-serif; }
      </style>
      <span><slot></slot></span>`;   // styles can't leak out of shadow root
  }
}
customElements.define("badge-chip", BadgeChip);
// <badge-chip>New</badge-chip>  -> styled pill; page CSS for span is unaffected
Renders a self-contained blue pill; the page's "span" rules never touch it.
JavaScript — slots for content projection
class CardBox extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: "open" }).innerHTML = `
      <header><slot name="title">Untitled</slot></header>
      <div><slot></slot></div>`;   // unnamed slot = default content
  }
}
customElements.define("card-box", CardBox);
// <card-box>
//   <span slot="title">Hi</span>   -> goes into the "title" slot
//   Body text                       -> goes into the default slot
// </card-box>
"Hi" renders in the header slot; "Body text" fills the default slot.
JavaScript — reactive attribute observation
class CounterTag extends HTMLElement {
  static get observedAttributes() { return ["count"]; } // only these trigger

  attributeChangedCallback(name, oldVal, newVal) {
    if (name === "count") this.render(newVal);
  }
  connectedCallback() { this.render(this.getAttribute("count") ?? "0"); }
  render(value) { this.textContent = `Count: ${value}`; }
}
customElements.define("counter-tag", CounterTag);

const c = document.createElement("counter-tag");
document.body.append(c);
c.setAttribute("count", "5");   // triggers attributeChangedCallback
Count: 0 Count: 5
Common Mistakes
❌ Forgetting super() in the constructor, or touching this / setting attributes before calling it — throws a ReferenceError and the element fails to upgrade.
✅ Call super() as the very first statement, then attach the shadow root and use this. Defer DOM-dependent setup to connectedCallback.
Key Rules
  • Custom element names must contain a hyphen (user-card, not usercard) to avoid collisions with future HTML tags.
  • Do real DOM work in connectedCallback, not the constructor — the element may not be in the document yet when constructed.
  • List every attribute you want to react to in observedAttributes; unlisted attributes will silently never call attributeChangedCallback.
Practice Exercises
1 Create a <user-card> component with Shadow DOM that renders a name, avatar image, and email read from its name, avatar, and email attributes. Scope all styling inside the shadow root.
2 Build a <tab-panel> component that uses named slots so the consumer supplies tab labels and tab bodies as light-DOM children, and clicking a label shows the matching body.
3 Harder: make <counter-tag> fully reactive — render from an observed count attribute, expose increment()/decrement() methods, and dispatch a CustomEvent("change", { detail: { count } }) on every change.
Connections: Lifecycle callbacks build on the DOM and MutationObserver machinery; the CustomEvent in exercise 3 ties back to Pub/Sub & EventTarget. Shadow-DOM encapsulation is the platform-native version of the scoped styles that frameworks emulate. Pairs with ResizeObserver for self-responsive components.
