Node.js Event Loop & Timers Node.js Runtime 9 days since review
Node's event loop runs in ordered phases: timers (setTimeout/setInterval) → pending → poll (I/O) → check (setImmediate) → close. Two microtask queues drain between every callback: process.nextTick first, then the Promise queue. So nextTick beats a resolved Promise, which beats a setTimeout(…, 0). Inside an I/O callback, setImmediate always fires before a setTimeout(…, 0).
JavaScript
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
console.log('sync');
// sync → nextTick → promise → timeout/immediate (order of last two varies at top level)
Challenge:
Predict the exact output when the four scheduling calls above are placed inside an fs.readFile callback instead of at the top level, and explain why setImmediate now deterministically precedes setTimeout(…, 0). Then show how a recursive process.nextTick can starve the event loop.
#1 Pitfall: Believing setTimeout(fn, 0) runs "immediately" — it is clamped (≥1ms) and yields to microtasks and I/O. For "run after this stack, before more I/O" use queueMicrotask/process.nextTick; for "run after the current I/O phase" use setImmediate.
Node.js Streams Node.js Runtime 9 days since review
Streams process data in chunks instead of buffering it all in memory. The four kinds: Readable (source), Writable (sink), Duplex (both), Transform (Duplex that maps input→output). Prefer pipeline() over .pipe() — it propagates errors and cleans up every stream. Backpressure is automatic under pipeline: a slow writable pauses the readable.
JavaScript
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

await pipeline(
  createReadStream('big.log'),
  createGzip(),                       // Transform stream
  createWriteStream('big.log.gz'),
);                                    // resolves when fully flushed; rejects on any error
Challenge:
Write a custom Transform stream (via Transform or the newer async-generator form with Readable.from) that upper-cases each line of text, and run it through pipeline from a file to stdout. Explain what backpressure prevents if the destination is 100× slower than the source.
#1 Pitfall: Using .pipe() and forgetting error handling — an error on any stream in a .pipe() chain leaks file descriptors and never rejects. pipeline() (or stream/promises) fixes both. Also, mixing manual data/readable listeners with pipe corrupts flow.
Node.js fs/promises & path Node.js Runtime 16 days since review
Import the promise API as import { readFile, writeFile } from 'node:fs/promises' — no callbacks, works with await. Never build paths with string concatenation: path.join normalizes separators, path.resolve makes an absolute path, and path.basename/dirname/extname parse one. Read text with an encoding argument ('utf8'), otherwise you get a Buffer.
JavaScript
import { readFile, writeFile, mkdir } from 'node:fs/promises';
import { join, dirname } from 'node:path';

const file = join(import.meta.dirname, 'data', 'out.json');
await mkdir(dirname(file), { recursive: true });   // no error if it exists
await writeFile(file, JSON.stringify({ ok: true }));
const text = await readFile(file, 'utf8');         // string, not Buffer
console.log(JSON.parse(text));                     // { ok: true }
Challenge:
Write readJson(path) and writeJson(path, obj) using fs/promises that create parent directories on write and return a typed default ({}) when the file is missing — distinguishing a genuine "missing file" (err.code === 'ENOENT') from a real read error, which you rethrow.
#1 Pitfall: Concatenating paths with '/' (breaks on Windows, doubles separators) and forgetting the encoding argument (getting a Buffer where you expected a string). Also, swallowing all errors instead of checking err.code — an EACCES should not be treated like an ENOENT.
Testing with Node Test Runner Testing & Quality 16 days since review
Node ships a built-in test runner — no dependency. Import test (and describe/it) from node:test and assertions from node:assert/strict. Run with node --test (auto-discovers *.test.js). Async tests just return a promise or use an async function; subtests via t.test(). Built-in mocking lives on t.mock.
JavaScript
import { test, describe, it } from 'node:test';
import assert from 'node:assert/strict';

describe('math', () => {
  it('adds', () => assert.equal(1 + 1, 2));
  it('awaits', async () => {
    const v = await Promise.resolve(42);
    assert.equal(v, 42);
  });
});
// run:  node --test        (or  node --test --watch)
Challenge:
Write a test file for a fetchUser(id) function that uses t.mock.method (or mock.fn) to stub globalThis.fetch, asserts the mock was called once with the right URL, and includes one failing assertion to confirm the runner reports the diff. Explain how node --test-name-pattern narrows the run.
#1 Pitfall: Using loose assert instead of assert/strict — assert.equal from the default module uses ==, so '1' equals 1. Always import node:assert/strict. Also, forgetting to await/return an async assertion means the test passes before it actually checks anything.
Console Advanced & Debugging Testing & Quality 16 days since review
The console is a toolkit: console.table(rows) renders a grid, group/groupEnd nest output, time/timeEnd measure a span, count tallies, assert(cond, …) logs only on failure, and trace() dumps a stack. Prefer conditional/logpoint breakpoints in DevTools over scattering logs, and load source maps to debug original source.
JavaScript
console.table([{ id: 1, name: 'Ada' }, { id: 2, name: 'Lin' }]);

console.group('request');
console.time('fetch');
// ...work...
console.timeEnd('fetch');            // fetch: 12.4ms
console.groupEnd();

console.assert(1 + 1 === 3, 'math broke');   // logs only because false
console.trace('who called me');              // prints call stack
Challenge:
Instrument a small pipeline so it prints each stage's rows with console.table, wraps each stage in a group, times the slow stage, and uses console.assert to flag rows violating an invariant — without changing the return values. Then describe when a conditional breakpoint beats all of this.
#1 Pitfall: Logging a live object reference then reading the console later — it may show the mutated current state, not the value at log time. Snapshot with structuredClone or JSON for a frozen view. And don't ship console.log debugging to production.
Node.js child_process: exec, spawn & fork Node.js Runtime 15 days since review
exec/execFile buffer the whole output into a callback (convenient for small output; exec runs a shell so it's injection-prone). spawn streams stdout/stderr (use for large or long-lived output). fork spawns another Node process with an IPC channel for child.send()/'message'. Prefer execFile/spawn with an args array over exec with an interpolated string.
JavaScript
import { spawn } from 'node:child_process';

const child = spawn('git', ['log', '--oneline', '-n', '5']);   // no shell, args array
child.stdout.on('data', (chunk) => process.stdout.write(chunk));
child.stderr.on('data', (chunk) => process.stderr.write(chunk));
child.on('close', (code) => console.log('exited', code));
Challenge:
Wrap spawn in a promise-returning run(cmd, args) that resolves with collected stdout on exit code 0 and rejects with stderr otherwise. Then explain the exact security difference between exec(`ls ${userInput}`) and execFile('ls', [userInput]), and why the second is safe from shell injection.
#1 Pitfall: Passing untrusted input into exec's command string — it runs through /bin/sh, so ; rm -rf executes. Use execFile/spawn with an argument array (no shell) for anything involving user data, and set maxBuffer for exec to avoid truncation on large output.

🆕 Node.js Built-in Modules

Two batteries-included modules that remove classic dependencies: an embedded SQL database and a zero-dep CLI argument parser.

node:sqlite — Built-in SQLite Node.js Runtime

Node 22.5+ ships a synchronous SQLite driver in core (node:sqlite), so a local database, test fixture, or CLI cache needs zero npm install and no native build step. It replaces reaching for better-sqlite3 for the common "I just need a real SQL store on disk" case.

1Open a database, run DDL with exec. new DatabaseSync(path) opens (or creates) a file; pass ':memory:' for an ephemeral in-RAM DB. exec() runs one or more statements with no parameters — ideal for schema setup.

JavaScript
import { DatabaseSync } from 'node:sqlite';   // Node 22.5+ (ExperimentalWarning)

const db = new DatabaseSync(':memory:');      // or 'app.db' for a file
db.exec(`
  CREATE TABLE users (
    id   INTEGER PRIMARY KEY,
    name TEXT NOT NULL
  )
`);
console.log('table created');
table created

2Prepare statements; bind with placeholders. db.prepare(sql) returns a reusable StatementSync. .run(...params) executes writes and returns { changes, lastInsertRowid }; always bind values with ? (or named $name) — never string-interpolate user input.

JavaScript
const insert = db.prepare('INSERT INTO users (name) VALUES (?)');
const r1 = insert.run('Ada');
const r2 = insert.run('Grace');
console.log(r1);                 // { changes: 1, lastInsertRowid: 1 }
console.log(r2.lastInsertRowid); // 2

// named parameters work too:
db.prepare('INSERT INTO users (name) VALUES ($name)').run({ name: 'Lin' });
{ changes: 1, lastInsertRowid: 1 } 2

3Query with .get (one row) and .all (every row). A prepared SELECT exposes .get(...params) → the first row object or undefined, and .all(...params) → an array of row objects. Because the API is synchronous, there is no await.

JavaScript
const byId  = db.prepare('SELECT * FROM users WHERE id = ?');
const above = db.prepare('SELECT * FROM users WHERE id > ? ORDER BY id');

console.log(byId.get(1));        // { id: 1, name: 'Ada' }
console.log(byId.get(999));      // undefined  — no match
console.log(above.all(1));       // [ { id: 2, name: 'Grace' }, { id: 3, name: 'Lin' } ]

db.close();                      // release the file handle
{ id: 1, name: 'Ada' } undefined [ { id: 2, name: 'Grace' }, { id: 3, name: 'Lin' } ]
Common Mistakes:
❌ Interpolating input or expecting a callback/Promise API: db.prepare(`SELECT * FROM users WHERE name = '${name}'`) // SQL injection await db.prepare('SELECT 1').get() // it's SYNC — no promise to await insert.get('Ada') // .get on an INSERT returns undefined, not the id
✅ Bind parameters; read the sync return values: db.prepare('SELECT * FROM users WHERE name = ?').get(name) const rows = db.prepare('SELECT * FROM users').all(); // no await const { lastInsertRowid } = insert.run('Ada'); // id from .run()
Key Rules:
  • DatabaseSync and StatementSync are synchronous — they block the event loop, which is fine for CLIs/startup/tests but not for a hot request path in a busy server.
  • Always parameterize with ? or $named placeholders; never build SQL by string concatenation.
  • .run() returns { changes, lastInsertRowid }; .get() returns one row or undefined; .all() returns an array. Still experimental — pin your Node version.
Practice Exercises:
Create an in-memory DB with a todos(id, task, done) table, insert three rows via a single prepared statement, then query and print only the rows where done = 0.
Write upsertUser(db, name) that inserts a user and returns the new lastInsertRowid, then fetch that exact row back with a .get query to confirm the round-trip.
Demonstrate the injection difference: write one query that interpolates a name containing ' OR 1=1 -- and one that binds it as a parameter, and show that only the parameterized version returns the intended single row.
Connections: node:sqlite joins the core-module family with node:fs/promises and node:crypto, and its synchronous, event-loop-blocking nature is a concrete case of the event-loop tradeoffs — reach for worker threads if you must keep a busy server responsive while querying.
util.parseArgs — CLI Argument Parsing Node.js Runtime

Every CLI needs to turn process.argv into structured options. parseArgs from node:util (stable since Node 20) gives you typed flags, short aliases, defaults, and positionals with no dependency — replacing minimist/yargs for the common cases.

1Declare an options schema. Each option has a type ('string' or 'boolean'), an optional short alias, a default, and multiple: true to collect repeats into an array. The result is { values, positionals }.

JavaScript
import { parseArgs } from 'node:util';

const { values, positionals } = parseArgs({
  args: ['--name', 'Ada', '-v', 'build.txt'],   // defaults to process.argv.slice(2)
  allowPositionals: true,
  options: {
    name:    { type: 'string',  short: 'n' },
    verbose: { type: 'boolean', short: 'v', default: false },
  },
});

console.log(values);        // { verbose: true, name: 'Ada' }
console.log(positionals);   // [ 'build.txt' ]
{ verbose: true, name: 'Ada' } [ 'build.txt' ]

2Repeatable options with multiple, and --flag=value syntax. With multiple: true the option always resolves to an array (empty if never passed). Both --tag a --tag b and --tag=a are accepted.

JavaScript
const { values } = parseArgs({
  args: ['--tag=frontend', '--tag', 'urgent'],
  options: {
    tag: { type: 'string', multiple: true },
  },
});
console.log(values.tag);    // [ 'frontend', 'urgent' ]
[ 'frontend', 'urgent' ]

3Use tokens: true for full control. Adding tokens: true returns a low-level token stream alongside values — useful to detect which options were explicitly passed (vs defaulted), or to implement custom rules like "last flag wins".

JavaScript
const { values, tokens } = parseArgs({
  args: ['--color', '--no-cache'],
  options: { color: { type: 'boolean' }, 'no-cache': { type: 'boolean' } },
  tokens: true,
});

const passed = tokens
  .filter((t) => t.kind === 'option')
  .map((t) => t.name);
console.log(passed);        // [ 'color', 'no-cache' ]
console.log(values.color);  // true
[ 'color', 'no-cache' ] true
Common Mistakes:
❌ Wrong args slice or unknown options: parseArgs({ args: process.argv }) // includes node + script path parseArgs({ args: ['pos'] }) // throws: positionals not allowed by default parseArgs({ args: ['--x'], options: {} })// throws: Unknown option '--x'
✅ Slice correctly and declare everything you accept: parseArgs({ options }) // defaults to process.argv.slice(2) parseArgs({ allowPositionals: true, options }) options: { x: { type: 'boolean' } } // every accepted flag must be declared
Key Rules:
  • type is only 'string' or 'boolean' — there is no 'number'; parse numeric strings yourself with Number().
  • allowPositionals: true is required to collect non-option args; unknown options throw unless you set strict: false.
  • Args default to process.argv.slice(2); multiple: true yields an array; tokens: true exposes the raw parse for advanced logic.
Practice Exercises:
Build a mini greet CLI schema: a required-feeling --name (string, short -n), a --loud boolean, and one positional. Print a greeting, upper-cased when --loud is set.
Add a repeatable --include option (multiple: true) and show that passing it zero times yields [] while --include a --include b yields ['a','b'].
Because there is no number type, accept --port as a string and validate it with Number.isInteger(Number(values.port)), throwing a clear error for --port abc.
Connections: parseArgs reads the same process.argv that drives child_process invocations, and its values/positionals output pairs naturally with node:sqlite and fs/promises to build dependency-free CLI tools.

🆕 Cancellation & Concurrency

Composable abort signals for timeouts and combined cancellation, plus lock-free shared memory across threads.

AbortSignal.timeout & AbortSignal.any Async Mastery

You already know AbortController for manual cancellation. The static helpers AbortSignal.timeout(ms) and AbortSignal.any([...signals]) let you express "cancel after N ms" and "cancel if any of these fire" declaratively — the clean way to add a timeout to fetch without wiring up your own timer + controller.

1AbortSignal.timeout(ms) auto-aborts. It returns a signal that aborts after ms with a DOMException named 'TimeoutError'. No controller, no clearTimeout to remember — pass it straight to any abortable API.

JavaScript
try {
  const res = await fetch('https://example.com/slow', {
    signal: AbortSignal.timeout(5000),      // abort after 5s
  });
  console.log(await res.text());
} catch (e) {
  if (e.name === 'TimeoutError') console.log('request timed out');
  else throw e;
}
request timed out

2AbortSignal.any([...]) combines signals. It returns a signal that aborts as soon as any input signal aborts, adopting that signal's reason. Perfect for "user clicked cancel or the timeout elapsed".

JavaScript
const userCancel = new AbortController();

const signal = AbortSignal.any([
  userCancel.signal,            // manual cancel button
  AbortSignal.timeout(10_000),  // hard 10s ceiling
]);

const p = fetch('/api/report', { signal });

// later, if the user clicks "stop":
userCancel.abort(new Error('user cancelled'));
// `signal` is now aborted; its .reason is the user's Error

3Read reason and check aborted. An aborted signal exposes why via signal.reason. In your own async code, poll signal.aborted or subscribe with signal.addEventListener('abort', …) to bail out cooperatively.

JavaScript
function delay(ms, signal) {
  return new Promise((resolve, reject) => {
    if (signal.aborted) return reject(signal.reason);
    const id = setTimeout(resolve, ms);
    signal.addEventListener('abort', () => {
      clearTimeout(id);
      reject(signal.reason);           // reject with the abort reason
    }, { once: true });
  });
}

await delay(1000, AbortSignal.timeout(200));   // rejects with TimeoutError after 200ms
Uncaught (in promise) DOMException [TimeoutError]: The operation timed out.
Common Mistakes:
❌ Rolling your own timeout, or mis-detecting the error: const c = new AbortController(); setTimeout(() => c.abort(), 5000); // manual, leaks the timer on success catch (e) { if (e.message === 'aborted') ... } // brittle string match AbortSignal.any(ac) // needs an ARRAY of signals
✅ Use the built-ins and check .name: fetch(url, { signal: AbortSignal.timeout(5000) }) catch (e) { if (e.name === 'TimeoutError') ... } AbortSignal.any([ac.signal, AbortSignal.timeout(5000)])
Key Rules:
  • AbortSignal.timeout(ms) aborts with a 'TimeoutError' DOMException — detect it via e.name, not a string message.
  • AbortSignal.any([signals]) takes an array and aborts on the first input to abort, inheriting its reason.
  • Any abortable operation should reject with signal.reason; register the abort listener with { once: true } and clean up timers to avoid leaks.
Practice Exercises:
Write fetchWithTimeout(url, ms) using AbortSignal.timeout that resolves to the response text or throws a clear "timed out after Nms" error, distinguishing a timeout from a network failure via e.name.
Combine a manual AbortController with a 3-second AbortSignal.timeout using AbortSignal.any, and log which source won by inspecting signal.reason after abort.
Make the delay(ms, signal) helper above and a companion race that runs two delays under a shared combined signal, proving that aborting once cancels both.
Connections: These extend the AbortController/AbortSignal pattern and the { signal } option used by streams, addEventListener, and fetch. AbortSignal.any is conceptually a cancellation-flavored Promise.race.
Atomics & SharedArrayBuffer Data & Collections

Workers normally communicate by copying messages. A SharedArrayBuffer is one block of memory visible to multiple threads at once — and Atomics provides the lock-free, tear-free reads/writes and thread coordination (wait/notify) needed to touch it safely. This is the foundation of high-performance parallel JS (WASM threads, ring buffers, ML workloads).

1A SharedArrayBuffer backs a typed-array view. Unlike a normal ArrayBuffer, it can be shared between threads without transfer. You always access it through a typed array (e.g. Int32Array), never the raw buffer.

JavaScript
const sab = new SharedArrayBuffer(8);   // 8 bytes = two Int32 slots
const view = new Int32Array(sab);

Atomics.store(view, 0, 42);             // atomic write to slot 0
console.log(Atomics.load(view, 0));     // 42  — atomic read
42

2Read-modify-write operations return the old value. Atomics.add, sub, and, or, xor, and exchange apply the update as one indivisible step — no other thread can interleave — and return the value that was there before.

JavaScript
const view = new Int32Array(new SharedArrayBuffer(4));
Atomics.store(view, 0, 100);

console.log(Atomics.add(view, 0, 5));    // 100  — returns PREVIOUS value
console.log(Atomics.load(view, 0));      // 105

// compareExchange: set to 200 only if it currently equals 105
console.log(Atomics.compareExchange(view, 0, 105, 200));  // 105 (the old value)
console.log(Atomics.load(view, 0));      // 200
100 105 105 200

3wait/notify coordinate threads. On a worker thread, Atomics.wait(view, i, expected) blocks until another thread calls Atomics.notify(view, i) — a zero-CPU way to build locks and signals. wait is forbidden on the main thread (it would freeze the UI); use Atomics.waitAsync there.

JavaScript
// --- inside a Worker thread ---
// block until slot 0 changes away from 0, then read the payload:
Atomics.wait(view, 0, 0);          // sleeps the thread, no busy-loop
console.log('woken, value =', Atomics.load(view, 0));

// --- on the producer side ---
Atomics.store(view, 0, 1);         // publish
Atomics.notify(view, 0, 1);        // wake one waiter
woken, value = 1
Common Mistakes:
❌ Non-atomic access or blocking the main thread: view[0] += 1; // read+write race across threads (lost updates) Atomics.wait(view, 0, 0); // throws on the main thread — UI freeze guard new Int32Array(new ArrayBuffer(4)) // plain buffer can't be shared with a worker
✅ Use Atomics on a SharedArrayBuffer; waitAsync on main: Atomics.add(view, 0, 1); // atomic, tear-free increment await Atomics.waitAsync(view, 0, 0).value; // non-blocking on main thread new Int32Array(new SharedArrayBuffer(4)) // shareable memory
Key Rules:
  • Atomics only works on integer typed arrays over a SharedArrayBuffer (or a normal buffer, but sharing across threads needs SharedArrayBuffer); RMW ops return the previous value.
  • Atomics.wait blocks and is banned on the main thread — use it in workers, and Atomics.waitAsync (Promise-based) on the main thread.
  • SharedArrayBuffer requires cross-origin isolation in browsers (COOP+COEP headers); in Node it works with worker_threads out of the box.
Practice Exercises:
Create a shared Int32Array counter and simulate two increments via Atomics.add, printing the returned previous values and the final total to show no update is lost.
Implement a one-slot spinlock-free "publish once" using Atomics.compareExchange so that only the first caller succeeds in flipping 0 → 1 and every later caller observes it was already claimed.
Sketch (conceptually, with two worker_threads) a producer that stores a value and calls Atomics.notify, and a consumer that Atomics.waits then reads it — and explain why Atomics.waitAsync would be required if the consumer were the main thread.
Connections: SharedArrayBuffer is the shareable sibling of the ArrayBuffer/typed-array family (including today's resizable buffers), and Atomics.wait/notify is the low-level primitive beneath worker_threads coordination and WebAssembly threads.

🆕 Flexible Binary Buffers

ES2024 gives ArrayBuffer two long-missing abilities: growing in place, and transferring ownership without a copy.

Resizable & Transferable ArrayBuffer Data & Collections

Historically an ArrayBuffer was fixed-size — growing a buffer meant allocating a new one and copying. ES2024 adds in-place resize() (up to a declared maxByteLength) and transfer() (zero-copy hand-off that detaches the original), which matter for streaming parsers, growable ring buffers, and cheaply moving buffers to a worker.

1Declare maxByteLength to make a buffer resizable. Pass it in the options; then resize(n) grows or shrinks in place up to that ceiling. Typed-array views over the buffer automatically track the new length (auto-length views).

JavaScript
const buf = new ArrayBuffer(8, { maxByteLength: 16 });
console.log(buf.resizable, buf.byteLength, buf.maxByteLength);  // true 8 16

const view = new Uint8Array(buf);      // auto-length view (no length passed)
console.log(view.length);              // 8

buf.resize(12);
console.log(buf.byteLength, view.length);   // 12 12  — view grew with the buffer

buf.resize(4);
console.log(view.length);              // 4   — and shrank
true 8 16 8 12 12 4

2You cannot exceed maxByteLength. resize beyond the declared max throws a RangeError. Choose a max up front based on the worst case; the engine may reserve that address space, so don't declare a gigantic ceiling you won't use.

JavaScript
const buf = new ArrayBuffer(8, { maxByteLength: 16 });
buf.resize(16);                        // ok — exactly at the max
try {
  buf.resize(20);                      // over the ceiling
} catch (e) {
  console.log(e.constructor.name);     // RangeError
}
RangeError

3transfer() moves ownership and detaches the source. It returns a new ArrayBuffer owning the same bytes with no copy; the original becomes detached (byteLength === 0, detached === true), and views over it throw. transferToFixedLength() does the same but drops resizability.

JavaScript
const a = new Uint8Array([1, 2, 3, 4]).buffer;
console.log(a.detached);               // false

const b = a.transfer();                // zero-copy move
console.log(a.detached, a.byteLength); // true 0   — source is now dead
console.log(b.byteLength);             // 4        — bytes live here now
console.log(new Uint8Array(b)[0]);     // 1

// reading the detached source throws:
try { new Uint8Array(a)[0]; } catch (e) { console.log(e.name); }  // TypeError
false true 0 4 1 TypeError
Common Mistakes:
❌ Growing without a max, or using a detached buffer: new ArrayBuffer(8).resize(16) // TypeError: not resizable (no maxByteLength) const v = new Uint8Array(buf, 0, 8); // FIXED-length view won't track resize() const b = a.transfer(); a.byteLength // 0 — 'a' is detached; reads throw
✅ Declare maxByteLength; use auto-length views; drop the old ref: new ArrayBuffer(8, { maxByteLength: 64 }).resize(16) const v = new Uint8Array(buf); // auto-length — grows/shrinks with buffer const b = a.transfer(); // then use b only; a is intentionally dead
Key Rules:
  • A buffer is only resizable if constructed with { maxByteLength }; resize() stays within [0, maxByteLength] or throws RangeError.
  • A typed array created without an explicit length is auto-tracking and follows the buffer's size; a fixed-length view does not.
  • transfer() is a zero-copy ownership move that detaches the original (like postMessage transferables) — never read the source afterward; check buffer.detached if unsure.
Practice Exercises:
Build a growable byte buffer: start at 8 bytes with maxByteLength: 32, write via an auto-length Uint8Array view, and resize to 16 then 32 — confirming the view length tracks each resize and that resizing to 40 throws.
Write moveBuffer(buf) that transfer()s a buffer and returns the new one, then assert the original is detached (byteLength === 0, detached === true) and that reading it throws.
Contrast transfer() with transferToFixedLength() on a resizable buffer: show that the first preserves resizable === true while the second returns a non-resizable buffer.
Connections: Resizable buffers round out the ArrayBuffer/DataView/typed-array family (including Float16Array), and transfer()'s detach semantics are the same mechanism behind postMessage transferables and the SharedArrayBuffer ownership model.
