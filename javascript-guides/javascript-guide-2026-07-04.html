Node.js fs/promises & path Node.js Runtime 1 day since review
Import the promise API as import { readFile, writeFile } from 'node:fs/promises' — no callbacks, works with await. Never build paths with string concatenation: path.join normalizes separators, path.resolve makes an absolute path, and path.basename/dirname/extname parse one. Read text with an encoding argument ('utf8'), otherwise you get a Buffer.
Challenge:
Write readJson(path) and writeJson(path, obj) that create parent directories on write (mkdir with recursive: true) and return a default ({}) when the file is missing — distinguishing err.code === 'ENOENT' from a real read error, which you rethrow.
#1 Pitfall: Concatenating paths with '/' (breaks on Windows) and forgetting the encoding argument (getting a Buffer where you expected a string). Also, swallowing all errors instead of checking err.code — an EACCES should not be treated like an ENOENT.
Testing with Node Test Runner Testing & Quality 1 day since review
Node ships a built-in test runner — no dependency. Import test (and describe/it) from node:test and assertions from node:assert/strict. Run with node --test (auto-discovers *.test.js). Async tests just return a promise or use an async function; subtests via t.test(). Built-in mocking lives on t.mock.
Challenge:
Write a test file for a fetchUser(id) function that uses t.mock.method (or mock.fn) to stub globalThis.fetch, asserts the mock was called once with the right URL, and includes one failing assertion to confirm the runner reports the diff. Explain how node --test-name-pattern narrows the run.
#1 Pitfall: Using loose assert instead of assert/strict — the default assert.equal uses ==, so '1' equals 1. Always import node:assert/strict. Also, forgetting to await/return an async assertion means the test passes before it actually checks anything.
Console Advanced & Debugging Testing & Quality 1 day since review
The console is a toolkit: console.table(rows) renders a grid, group/groupEnd nest output, time/timeEnd measure a span, count tallies, assert(cond, …) logs only on failure, and trace() dumps a stack. Prefer conditional/logpoint breakpoints in DevTools over scattering logs, and load source maps to debug original source.
Challenge:
Instrument a small pipeline so it prints each stage's rows with console.table, wraps each stage in a group, times the slow stage, and uses console.assert to flag rows violating an invariant — without changing the return values. Then describe when a conditional breakpoint beats all of this.
#1 Pitfall: Logging a live object reference then reading the console later — it may show the mutated current state, not the value at log time. Snapshot with structuredClone or JSON for a frozen view. And don't ship console.log debugging to production.
Node.js child_process: exec, spawn & fork Node.js Runtime 1 day since review
exec/execFile buffer the whole output into a callback (convenient for small output; exec runs a shell so it's injection-prone). spawn streams stdout/stderr (use for large or long-lived output). fork spawns another Node process with an IPC channel for child.send()/'message'. Prefer execFile/spawn with an args array over exec with an interpolated string.
Challenge:
Wrap spawn in a promise-returning run(cmd, args) that resolves with collected stdout on exit code 0 and rejects with stderr otherwise. Then explain the exact security difference between exec(`ls ${userInput}`) and execFile('ls', [userInput]), and why the second is safe from shell injection.
#1 Pitfall: Passing untrusted input into exec's command string — it runs through /bin/sh, so ; rm -rf executes. Use execFile/spawn with an argument array (no shell) for anything involving user data, and set maxBuffer for exec to avoid truncation on large output.
Deno & Bun: Alternative Runtimes Node.js Runtime 16 days since review
Deno runs TypeScript directly, is secure-by-default (explicit --allow-read/--allow-net permissions), ships web-standard APIs (fetch, Deno.serve) and URL/npm imports, with a built-in formatter/linter/test runner. Bun is a speed-focused all-in-one (runtime + bundler + package manager + test runner) built on JavaScriptCore, largely Node-compatible with fast bun install and native TS. Both aim to reduce toolchain sprawl versus Node + separate tools.
Challenge:
Name three things Deno does differently from Node by default (permissions, module resolution, TS support) and one reason Bun's bun install is faster than npm install. Then describe what "web-standard first" means and give one API that behaves the same in Deno, Bun, and the browser.
#1 Pitfall: Assuming 100% Node compatibility. Some Node built-ins/native addons and edge behaviors differ across Bun/Deno; verify libraries against the target runtime rather than trusting "it's Node-compatible", and remember Deno needs explicit permission flags or scripts fail with a permission error.
Package Management: npm, pnpm & yarn Node.js Runtime 16 days since review
npm uses a flat, hoisted node_modules. pnpm stores every package once in a global content-addressable store and links it in via a symlinked, non-flat layout — saving disk and preventing "phantom dependencies" (importing something you never declared). yarn (Berry) offers Plug'n'Play (no node_modules) and zero-installs. Always commit the lockfile; use npm ci (not install) in CI for a clean, lockfile-exact, reproducible install.
Challenge:
Explain what a "phantom dependency" is and why pnpm's non-flat layout prevents it. Then contrast npm install vs npm ci for CI, and state what semver ranges ^1.2.3 and ~1.2.3 each allow to change.
#1 Pitfall: Not committing the lockfile (or deleting it to "fix" an install) — you lose reproducibility and every machine can resolve different versions. Also relying on a hoisted package you never listed in dependencies: it works under npm's flat tree but breaks under pnpm/PnP and after a dependency drops it.

🆕 Node.js Event & Timer APIs

The event-driven core of Node — the emitter every stream and server is built on — plus the promise-native timers that let you await a delay.

EventEmitter (node:events) Node.js Runtime

EventEmitter is the pattern underneath nearly every Node object — streams, servers, sockets, and processes all are emitters. Where the browser has EventTarget/addEventListener, Node has emitter.on()/emitter.emit(). Understanding it lets you both consume Node APIs fluently and build your own decoupled, event-driven modules.

1Subscribe with on, publish with emit. Handlers registered with on(name, fn) run synchronously, in registration order, every time emit(name, ...args) is called. once(name, fn) auto-removes after the first fire; off (alias removeListener) detaches.

JavaScript
import { EventEmitter } from 'node:events';

const bus = new EventEmitter();
bus.on('order', (id, total) => console.log(`order ${id}: $${total}`));
bus.once('ready', () => console.log('ready fired once'));

bus.emit('ready');
bus.emit('ready');                 // ignored — 'once' already consumed
bus.emit('order', 7, 42);          // args are passed positionally
ready fired once order 7: $42

2Subclass to make your own type an emitter. Extend EventEmitter and emit named events from your methods. Consumers wire up behavior without your class knowing about them — the essence of loose coupling.

JavaScript
class Job extends EventEmitter {
  run() {
    this.emit('start');
    this.emit('progress', 50);
    this.emit('progress', 100);
    this.emit('done', { ok: true });
  }
}

const job = new Job();
job.on('progress', (p) => console.log(`progress ${p}%`));
job.on('done', (r) => console.log('done', r));
job.run();
progress 50% progress 100% done { ok: true }

3Handle 'error', and await events with helpers. An emitted 'error' with no listener throws and can crash the process — always attach one. The events module also gives once(emitter, name) (a promise that resolves on the next event) and on(emitter, name) (an async iterator over events).

JavaScript
import { EventEmitter, once } from 'node:events';

const em = new EventEmitter();
em.on('error', (e) => console.log('handled:', e.message));  // prevents a crash
em.emit('error', new Error('boom'));

const em2 = new EventEmitter();
setTimeout(() => em2.emit('tick', 1), 10);
const [value] = await once(em2, 'tick');   // await the next 'tick'
console.log('awaited tick', value);
handled: boom awaited tick 1
Common Mistakes:
❌ Leaking listeners and ignoring 'error': for (const s of sockets) em.on('data', handler); // never removed → leak + warning em.emit('error', new Error('x')); // no 'error' listener → THROWS, may crash em.on('x', async () => { throw e }) // a rejected async listener is unhandled
✅ Clean up, always listen for 'error', guard async: em.on('error', (e) => log(e)); // attach BEFORE anything can emit it em.once('data', handler); // or off() when done → no leak controller = new AbortController(); em.on('x', fn, { signal }); // auto-remove
Key Rules:
  • Listeners fire synchronously in registration order; emit returns true if there were listeners. Use once for one-shot, off/removeListener to detach.
  • A 'error' event with no listener throws — attach an 'error' handler on every emitter that can fail.
  • More than 10 listeners for one event logs a memory-leak warning; raise it deliberately with setMaxListeners(n) only when you truly need many, and prefer removing listeners you no longer need.
Practice Exercises:
Build a tiny Timer extends EventEmitter that emits 'tick' every 100ms with an incrementing count and 'stop' when told; subscribe, log five ticks, then stop and confirm no further ticks fire.
Use events.once(emitter, 'done') to await the completion of an emitter-based task, and separately trigger an 'error' to prove that awaiting rejects when the emitter emits 'error' instead.
Demonstrate the max-listeners warning: attach 11 listeners for one event, observe the warning, then use setMaxListeners(20) to silence it — and explain why silencing can hide a real leak.
Connections: EventEmitter is the Node-side counterpart of EventTarget-style pub/sub, and it underlies child_process, streams, and servers you've already reviewed. Its once/on promise + async-iterator helpers bridge events into async/await code.
node:timers/promises Node.js Runtime

Callback setTimeout doesn't compose with async/await. The node:timers/promises module gives promise-returning timers — setTimeout, setImmediate, and an async-iterator setInterval — that you can await directly, cancel with an AbortSignal, and use to write clean delays, polling loops, and backoff without callback nesting.

1setTimeout(ms) returns a promise you await. No callback — the promise resolves after ms (optionally to a passed value). This is the idiomatic "sleep" in modern Node.

JavaScript
import { setTimeout as sleep } from 'node:timers/promises';

console.log('start');
await sleep(100);                    // pause without blocking the event loop
console.log('after 100ms');

const value = await sleep(50, 'done');  // resolves TO a value
console.log(value);                  // 'done'
start after 100ms done

2Cancel with an AbortSignal. Pass { signal }; aborting rejects the pending timer promise with an AbortError. This composes with AbortSignal.timeout and AbortSignal.any for cancellable waits.

JavaScript
import { setTimeout as sleep } from 'node:timers/promises';

const ac = new AbortController();
setImmediate(() => ac.abort());          // cancel almost immediately

try {
  await sleep(1000, undefined, { signal: ac.signal });
  console.log('slept');
} catch (e) {
  console.log(e.name);                   // 'AbortError'
}
AbortError

3setInterval is an async iterator. The promise-flavored setInterval(ms, startValue) yields on each tick inside a for await…of loop — a clean, cancellable polling loop with no dangling clearInterval. setImmediate() likewise returns a promise that resolves after the current poll phase.

JavaScript
import { setInterval } from 'node:timers/promises';

let n = 0;
for await (const _ of setInterval(50)) {   // ticks every 50ms
  console.log('tick', ++n);
  if (n === 3) break;                      // break stops the interval — no clearInterval
}
console.log('done');
tick 1 tick 2 tick 3 done
Common Mistakes:
❌ Wrapping the callback API by hand, or forgetting cancellation: await new Promise(r => setTimeout(r, ms)); // reinventing timers/promises const t = setInterval(fn, ms); // callback version: needs clearInterval await sleep(10000) // un-cancellable long wait blocks shutdown
✅ Use the promise API and an AbortSignal: import { setTimeout as sleep } from 'node:timers/promises'; await sleep(ms); // clean, awaitable await sleep(ms, undefined, { signal }); // cancellable on shutdown
Key Rules:
  • Import from node:timers/promises — distinct from the global callback setTimeout/setInterval. It's Node-only (browsers use the callback globals or Scheduler).
  • Every function accepts an options object with a signal; aborting rejects with an AbortError — pair it with AbortSignal.timeout for deadlines.
  • setInterval here is an async iterator: drive it with for await…of and break to stop — there is no timer id to clear.
Practice Exercises:
Write retry(fn, { attempts, baseMs }) that retries an async fn with exponential backoff using await sleep(baseMs * 2 ** i), rethrowing the last error after the final attempt.
Build a poll(check, everyMs, signal) using the async-iterator setInterval that resolves when check() returns truthy and rejects with AbortError if signal aborts first.
Race a real task against sleep(timeoutMs) so that whichever settles first wins; then reimplement the timeout with AbortSignal.timeout passed into sleep, and compare which approach leaves no dangling timer.
Connections: These timers are the promise-native complement to the callback timers behind the Node event loop, and their { signal } option is the same AbortController/AbortSignal.timeout cancellation pattern used by fetch and streams.

🆕 Real-Time Networking

Two ways to push data as it happens: full-duplex WebSocket for interactive channels, and one-way Server-Sent Events for simple live streams over plain HTTP.

WebSocket Browser APIs & DOM

WebSocket upgrades an HTTP connection to a persistent, bidirectional channel — the server can push to the client and vice-versa over one long-lived socket, with far less overhead than polling. It's the transport behind chat, live dashboards, collaborative editing, and multiplayer games.

1Open a connection and wire the four events. Construct with a ws:///wss:// URL. It's event-based: open (ready to send), message (data arrived, in event.data), error, and close. Only send after open.

JavaScript
const ws = new WebSocket('wss://example.com/socket');

ws.addEventListener('open', () => {
  console.log('connected');
  ws.send(JSON.stringify({ type: 'hello' }));   // send only after 'open'
});
ws.addEventListener('message', (e) => {
  console.log('recv', e.data);                  // string or binary
});
ws.addEventListener('close', (e) => console.log('closed', e.code));
ws.addEventListener('error', () => console.log('socket error'));
connected recv {"type":"welcome"}

2Send text or binary; close cleanly. send() accepts a string, Blob, ArrayBuffer, or typed array. Set ws.binaryType = 'arraybuffer' to receive binary as an ArrayBuffer. Call ws.close(code, reason) to shut down; 1000 is a normal closure.

JavaScript
ws.binaryType = 'arraybuffer';

ws.send('plain text');
ws.send(JSON.stringify({ type: 'move', x: 3 }));
ws.send(new Uint8Array([1, 2, 3]).buffer);      // binary frame

console.log(ws.readyState);   // 0 CONNECTING, 1 OPEN, 2 CLOSING, 3 CLOSED
ws.close(1000, 'done');       // graceful close with a status code
1

3Reconnect with backoff — the socket won't do it for you. A dropped connection fires close; the app must reconnect. Use increasing delays (and heartbeats/ping to detect dead links) so a flapping server isn't hammered.

JavaScript
function connect(url, attempt = 0) {
  const ws = new WebSocket(url);
  ws.addEventListener('open', () => { attempt = 0; });
  ws.addEventListener('close', () => {
    const delay = Math.min(1000 * 2 ** attempt, 30_000);  // capped backoff
    setTimeout(() => connect(url, attempt + 1), delay);
  });
  return ws;
}
connect('wss://example.com/socket');
Common Mistakes:
❌ Sending before open, wrong scheme, or no reconnect: const ws = new WebSocket('ws://…'); ws.send('hi'); // throws: still CONNECTING new WebSocket('http://example.com') // wrong scheme (use ws/wss) ws.onclose = null // drop → dead app, no reconnect
✅ Send in 'open', use wss, reconnect with backoff: ws.addEventListener('open', () => ws.send('hi')); new WebSocket('wss://example.com') // TLS in production ws.addEventListener('close', () => scheduleReconnect());
Key Rules:
  • Only send() when readyState === WebSocket.OPEN; sending during CONNECTING throws. Queue outbound messages until open if needed.
  • Use wss:// (TLS) in production — ws:// is blocked from HTTPS pages as mixed content. WebSocket has no same-origin restriction, so authenticate/authorize the upgrade server-side.
  • The browser does not auto-reconnect; implement backoff + heartbeats yourself. event.data is a string unless you set binaryType.
Practice Exercises:
Wrap a WebSocket in a class that buffers send() calls made before open and flushes them once connected, exposing an onMessage(cb) method.
Implement reconnect-with-exponential-backoff (capped at 30s) that resets the delay after a successful open, and add a ping/pong heartbeat that closes the socket if no pong arrives within a timeout.
Send and receive binary: set binaryType = 'arraybuffer', transmit a Uint8Array, and on the receiving handler reconstruct and log the bytes — explaining when you'd choose binary frames over JSON.
Connections: WebSocket is full-duplex where Server-Sent Events is one-way; its binary frames use the same ArrayBuffer/typed-array data you've seen with buffers, and cancellation/heartbeats reuse the timer patterns above.
Server-Sent Events (EventSource) Browser APIs & DOM

When you only need the server to push to the client — live scores, notifications, log tails, streaming LLM tokens — EventSource is simpler than WebSocket: it's plain HTTP, text-only, and auto-reconnects with built-in event IDs for resuming. One-way, but batteries included.

1Open an EventSource and listen for message. Point it at an endpoint that streams text/event-stream. Default messages arrive on the message event with the payload in event.data (always a string).

JavaScript
const es = new EventSource('/events');

es.addEventListener('open', () => console.log('stream open'));
es.addEventListener('message', (e) => {
  console.log('data:', e.data);        // always a string
});
es.addEventListener('error', () => {
  // fired on disconnect; the browser will auto-reconnect
  console.log('disconnected — retrying automatically');
});
stream open data: {"price":101.2}

2Named events and the wire format. The server sends UTF-8 lines: data:, optional event: (a custom type), id:, and retry:, each message terminated by a blank line. A named event: is delivered to a matching addEventListener(name, …) instead of message.

JavaScript
// --- server writes this text/event-stream body ---
// event: price
// id: 42
// data: {"symbol":"ACME","px":101.2}
//
// (blank line = end of one event)

const es = new EventSource('/events');
es.addEventListener('price', (e) => {   // matches "event: price"
  const { symbol, px } = JSON.parse(e.data);
  console.log(`${symbol} @ ${px}, lastEventId=${e.lastEventId}`);
});
ACME @ 101.2, lastEventId=42

3Auto-reconnect and resume with Last-Event-ID. On a drop the browser reconnects after the server's retry: interval and sends the last seen id back in a Last-Event-ID request header, so the server can resume the stream. Call es.close() to stop for good.

JavaScript
const es = new EventSource('/events');
let count = 0;
es.onmessage = (e) => {
  console.log(e.data);
  if (++count === 10) es.close();   // stop reconnecting entirely
};
console.log(es.readyState);         // 0 CONNECTING, 1 OPEN, 2 CLOSED
0
Common Mistakes:
❌ Wrong content type, binary hopes, or duplicate manual reconnect: res.setHeader('Content-Type', 'application/json') // must be text/event-stream es.send(...) // EventSource is RECEIVE-ONLY es.onerror = () => new EventSource(url) // it ALREADY auto-reconnects
✅ Correct stream, receive-only, let it reconnect: res.setHeader('Content-Type', 'text/event-stream') // send from client via a normal fetch/POST; SSE is one-way down es.onerror = () => log('will auto-retry'); es.close() // only to stop for good
Key Rules:
  • SSE is one-way (server→client) and text-only (UTF-8); need bidirectional or binary? Use WebSocket. To send up, make a separate HTTP request.
  • The response must be Content-Type: text/event-stream; each event is field: value lines ended by a blank line. Named events use event:; id: enables resume.
  • Reconnection is automatic — don't create a new EventSource in onerror. Over HTTP/1.1 browsers cap ~6 connections per origin; HTTP/2 multiplexing avoids that limit.
Practice Exercises:
Consume an /events stream that emits a named tick event carrying JSON; parse each payload, render the latest value, and stop after 20 ticks with es.close().
Handle resume: log e.lastEventId on every message and describe what the browser sends after a disconnect (the Last-Event-ID header) and how a server would use it to replay missed events.
Compare transports: write two short paragraphs deciding between SSE and WebSocket for (a) a live notifications feed and (b) a collaborative cursor, justifying each choice by directionality, payload type, and reconnection needs.
Connections: EventSource is the one-way, auto-reconnecting sibling of WebSocket, and its event-driven API mirrors EventEmitter/EventTarget. The text/event-stream body is exactly the kind of chunked text a ReadableStream or TextDecoder processes on the wire.

🆕 Binary Text Encoding

The standard bridge between JavaScript strings and raw UTF-8 bytes — needed for streams, fetch bodies, WebSocket frames, and crypto.

TextEncoder & TextDecoder Data & Collections

Strings are UTF-16 in memory, but the network, files, and crypto all speak bytes. TextEncoder/TextDecoder are the web-standard, cross-platform (browser + Node + Deno) way to convert between a string and a UTF-8 Uint8Array — the correct alternative to old, lossy tricks like btoa/escape for non-ASCII text.

1TextEncoder → UTF-8 bytes. new TextEncoder().encode(str) returns a Uint8Array of UTF-8 bytes. It's always UTF-8 (no other encoding). Note that non-ASCII characters take multiple bytes, so byte length ≥ string length.

JavaScript
const enc = new TextEncoder();
const bytes = enc.encode('héllo €');

console.log(bytes);            // Uint8Array of UTF-8 bytes
console.log('chars:', 'héllo €'.length);   // 7  (UTF-16 code units)
console.log('bytes:', bytes.length);       // 11 (é=2, €=3 bytes)
Uint8Array(11) [104, 195, 169, 108, 108, 111, 32, 226, 130, 172] chars: 7 bytes: 11

2TextDecoder → string. new TextDecoder().decode(bytesOrBuffer) turns bytes back into a string (UTF-8 by default; it can decode other labels like 'utf-16le' or 'latin1'). Pass { fatal: true } to throw on malformed input instead of inserting replacement characters.

JavaScript
const dec = new TextDecoder();                 // utf-8 by default
console.log(dec.decode(bytes));                // 'héllo €'

// strict mode rejects invalid byte sequences:
const strict = new TextDecoder('utf-8', { fatal: true });
try {
  strict.decode(new Uint8Array([0xff, 0xff]));  // not valid UTF-8
} catch (e) {
  console.log(e.name);                         // 'TypeError'
}
héllo € TypeError

3Streaming decode across chunks. Bytes arriving in chunks may split a multi-byte character at a boundary. Create one TextDecoder and call decode(chunk, { stream: true }) per chunk — it buffers the incomplete tail and emits it once completed; a final decode() flushes.

JavaScript
const dec = new TextDecoder();
// the 3 bytes of '€' (226,130,172) are split across two chunks:
let out = dec.decode(new Uint8Array([226, 130]), { stream: true });  // ''
out    += dec.decode(new Uint8Array([172]),      { stream: true });  // '€'
out    += dec.decode();                                              // flush tail
console.log(JSON.stringify(out));    // "€"  — not corrupted at the split
"€"
Common Mistakes:
❌ Lossy legacy conversions or per-chunk decoders: btoa(unescape(encodeURIComponent(s))) // fragile ritual for UTF-8→base64 String.fromCharCode(...bytes) // breaks on multi-byte chars, stack blows up chunks.map(c => new TextDecoder().decode(c)) // splits corrupt boundary chars
✅ Use the standard codecs, stream across chunks: new TextEncoder().encode(s) // → Uint8Array (then base64 if needed) new TextDecoder().decode(bytes) // correct multi-byte handling const d = new TextDecoder(); d.decode(chunk, { stream: true }) // one decoder
Key Rules:
  • TextEncoder only produces UTF-8; TextDecoder can consume many labels (utf-8, utf-16le, latin1…) but defaults to UTF-8. Both are web standards available in browsers, Node, and Deno.
  • String .length counts UTF-16 code units, not bytes — always measure encoded byte length via encode(str).length.
  • For chunked/streaming input reuse one decoder with { stream: true } so characters split across chunk boundaries aren't corrupted; use { fatal: true } to reject invalid bytes.
Practice Exercises:
Write utf8ByteLength(str) using TextEncoder and prove it differs from str.length for a string containing an emoji and an accented letter.
Round-trip a string through encode → Uint8Array → decode and assert equality; then corrupt one byte and show how { fatal: true } turns the silent replacement character into a thrown error.
Given a fetch(url).body ReadableStream, read it with a reader and decode each chunk with a single streaming TextDecoder, concatenating into the full text — confirming no character is broken at a chunk boundary.
Connections: TextEncoder/TextDecoder produce and consume the Uint8Array data used by WebSocket binary frames, fetch/ReadableStream bodies, node:crypto hashing, and base64/hex helpers — making it the connective tissue between strings and the binary APIs across today's guide.
