Structured Clone Data & Collections 6 days since review
structuredClone(value) performs a true deep copy using the structured clone algorithm — it follows the same rules as postMessage. It correctly clones nested objects, arrays, Map, Set, Date, RegExp, ArrayBuffer, and typed arrays, and it handles cyclic references (unlike the JSON.parse(JSON.stringify(x)) hack). It cannot clone functions, DOM nodes, or class-instance prototypes (you get a plain object). Pass { transfer: [buf] } to move an ArrayBuffer instead of copying it.
JavaScript
const original = { d: new Date(), set: new Set([1, 2]), nested: { a: 1 } };
original.self = original;                    // cycle

const copy = structuredClone(original);
copy.nested.a = 99;
console.log(original.nested.a);             // 1  — deep, not shared
console.log(copy.set instanceof Set);       // true — Set preserved
console.log(copy.self === copy);            // true — cycle preserved
Challenge:
Deep-clone an object containing a Map, a cyclic reference, and a class instance. Predict what happens to the class instance's prototype and methods after cloning, then explain one thing structuredClone can do that JSON.parse(JSON.stringify(x)) cannot, and one thing the JSON hack allows that structuredClone rejects.
#1 Pitfall: Cloning objects that contain functions, symbols-as-values, or DOM nodes throws DataCloneError. And a class instance survives structurally but loses its prototype — it becomes a plain object with no methods and no working instanceof.
JSON Advanced Data & Collections 6 days since review
JSON.stringify(value, replacer, space): the replacer can be a function (key, val) => ... (return undefined to drop a key) or an array of allowed keys; space pretty-prints. Any object with a toJSON() method controls its own serialized form (that's why Date becomes an ISO string). On the way back, JSON.parse(text, reviver) transforms each value (e.g. revive ISO strings into Dates). Gotchas: undefined/functions are dropped from objects (become null in arrays), and BigInt throws.
JavaScript
const data = { name: 'a', secret: 'x', when: new Date('2026-07-07') };

// replacer array = allow-list of keys; toJSON turns Date into an ISO string:
console.log(JSON.stringify(data, ['name', 'when']));
// {"name":"a","when":"2026-07-07T00:00:00.000Z"}

// reviver revives ISO date strings back into Date objects:
const revived = JSON.parse('{"when":"2026-07-07T00:00:00.000Z"}', (k, v) =>
  /^\d{4}-\d\d-\d\dT/.test(v) ? new Date(v) : v);
console.log(revived.when instanceof Date);   // true
Challenge:
Serialize an object that contains a BigInt and a Map without throwing, by giving each a custom representation via a replacer (and reversing it with a reviver). Then explain the difference between the function replacer and the array replacer, and why JSON.stringify([undefined, () => {}]) yields [null,null].
#1 Pitfall: Silent data loss. undefined, functions, and symbol keys vanish from objects; NaN/Infinity become null; and a stray BigInt throws a TypeError. Never assume a round-trip is lossless.
Regular Expressions Advanced Data & Collections 6 days since review
Named capture groups (?<year>\d{4}) land on match.groups; lookahead (?=...)/(?!...) and lookbehind (?<=...)/(?<!...) assert context without consuming characters. The d flag adds match .indices, the s (dotAll) flag lets . match newlines, and u/v enable full Unicode mode. Pair with matchAll to iterate every match with its groups.
JavaScript
// named groups + lookbehind (grab the number after a '$', without the '$'):
const m = '$1299'.match(/(?<=\$)(?<amount>\d+)/);
console.log(m.groups.amount);        // '1299'

// lookahead: match a digit only if followed by 'px'
console.log('12px 30em'.match(/\d+(?=px)/)[0]);   // '12'
Challenge:
Write one regex with a lookbehind and a lookahead that extracts just the number from "total: $1,299.00 USD" (the digits+comma+decimal between $ and  USD). Then explain why lookaround assertions are "zero-width" and how that lets adjacent matches overlap in position.
#1 Pitfall: Forgetting that . does not match a newline unless you add the s flag, and reusing a single g-flagged regex across calls — its lastIndex carries over and silently skips or repeats matches. Prefer matchAll or a fresh regex per scan.
Design Patterns: Observer, Strategy, Command Modules & Patterns 6 days since review
Observer: a subject keeps a list of subscribers and notify()s them on change (the model behind EventEmitter/EventTarget). Strategy: encapsulate interchangeable algorithms behind one interface and swap them at runtime — in JS, usually just passing a function. Command: package an action (and often its undo) as an object with execute(), enabling queues, macros, and undo stacks.
JavaScript
// Strategy — swap the algorithm, not the caller:
const strategies = {
  asc:  (a, b) => a - b,
  desc: (a, b) => b - a,
};
const sortBy = (arr, name) => [...arr].sort(strategies[name]);
console.log(sortBy([3, 1, 2], 'desc'));    // [3, 2, 1]

// Command — action + undo as an object:
const cmd = { execute: () => stack.push(1), undo: () => stack.pop() };
Challenge:
Build a tiny Observable with subscribe(fn) returning an unsubscribe function, and emit(value). Then contrast it with the Strategy pattern: which one is about reacting to change over time and which is about selecting behavior at a point in time?
#1 Pitfall: Observer memory leaks — subscribers that never unsubscribe keep the subject (and their captured closures) alive forever. Always return and call a teardown function, or use AbortSignal/WeakRef for auto-cleanup.
Functional Patterns: Compose, Pipe, Curry Modules & Patterns 6 days since review
compose applies functions right-to-left: compose(f, g)(x) === f(g(x)). pipe is the same left-to-right (reading order): pipe(g, f)(x) === f(g(x)). Curry turns f(a, b, c) into f(a)(b)(c), enabling partial application — fix some arguments now, supply the rest later — which powers point-free style (defining functions by composition, without naming their data argument).
JavaScript
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const clean = pipe(
  (s) => s.trim(),
  (s) => s.toLowerCase(),
  (s) => s.replace(/\s+/g, '-'),
);
console.log(clean('  Hello World  '));      // 'hello-world'

const add = (a) => (b) => a + b;             // curried
const inc = add(1);                          // partial application
console.log([1, 2, 3].map(inc));            // [2, 3, 4]
Challenge:
Implement both compose and pipe with reduce/reduceRight, and prove compose(f, g)(x) equals pipe(g, f)(x). Then write a curried prop(key)(obj) and use it point-free with map to pluck a field from an array of objects.
#1 Pitfall: Reversing the order — compose runs its last argument first (right-to-left), which surprises people who read left-to-right; use pipe when reading order matters. And composition assumes pure functions: a step that mutates its input breaks reuse and makes the pipeline order-dependent.
Error Handling Patterns Modules & Patterns 6 days since review
Subclass Error for domain errors — set this.name so messages and instanceof checks read well. Preserve the original failure with the ES2022 cause option (new Error('msg', { cause: err })) instead of swallowing it. Always throw real Error objects (never strings), so a stack trace is captured. This dovetails with today's new Error.captureStackTrace, which trims constructor noise from that trace.
JavaScript
class ValidationError extends Error {
  constructor(field, cause) {
    super(`Invalid field: ${field}`, { cause });
    this.name = 'ValidationError';
    this.field = field;
  }
}
try {
  try { JSON.parse('{bad'); }
  catch (e) { throw new ValidationError('config', e); }  // wrap, keep cause
} catch (e) {
  console.log(e.name, '|', e.field, '|', e.cause.name);  // ValidationError | config | SyntaxError
}
Challenge:
Create a NotFoundError and a TimeoutError that both extend a shared AppError base. Wrap a low-level failure with { cause }, then walk the cause chain to print the root cause. Explain why throw 'oops' (a string) is worse than throw new Error('oops').
#1 Pitfall: Throwing non-Error values (strings, plain objects) — they carry no stack trace and break instanceof. And forgetting to set this.name in a subclass leaves it reading as a generic "Error" in logs.

🆕 Numbers & Error Diagnostics

Two grounded upgrades to bread-and-butter code: a correctly-rounded sum that beats reduce, and precise control over the stack trace attached to your custom errors. Both run in Node/V8 today.

Math.sumPrecise Core Language

Summing floats with reduce((a, b) => a + b, 0) accumulates rounding error at every step, and the result depends on the order of the terms — a real bug in money totals, statistics, and physics. Math.sumPrecise(iterable) returns the single correctly-rounded IEEE-754 sum of all the values, as if computed with infinite precision and rounded once at the end.

1Correctly-rounded vs naive accumulation. Adding ten copies of 0.1 left-to-right drifts to 0.9999999999999999; the true sum rounds to exactly 1. Math.sumPrecise gives you that exact answer.

JavaScript
const tenth = Array(10).fill(0.1);

console.log(tenth.reduce((a, b) => a + b, 0));   // 0.9999999999999999  (drift)
console.log(Math.sumPrecise(tenth));             // 1                    (exact rounding)
0.9999999999999999 1

2Immune to catastrophic cancellation and intermediate overflow. When a huge term swamps a tiny one, naive summation loses the small value entirely; sumPrecise keeps it. It also avoids overflowing to Infinity on intermediate partial sums that cancel out.

JavaScript
// the 0.1 is lost: 1e20 + 0.1 === 1e20, then minus 1e20 === 0
console.log([1e20, 0.1, -1e20].reduce((a, b) => a + b, 0));   // 0
console.log(Math.sumPrecise([1e20, 0.1, -1e20]));            // 0.1

// naive overflows to Infinity mid-sum; sumPrecise does not:
console.log([1e308, 1e308, -1e308, -1e308].reduce((a, b) => a + b, 0)); // 0 here, but
console.log(Math.sumPrecise([1e308, 1e308, -1e308, -1e308]));           // 0 (no Infinity step)
0 0.1 0 0

3Takes any iterable; numbers only. It consumes an iterable (array, Set, generator), but every element must be a Number — a BigInt or string throws a TypeError. An empty iterable returns -0.

JavaScript
function* prices() { yield 9.99; yield 4.25; yield 0.01; }
console.log(Math.sumPrecise(prices()));   // 14.25  (works on any iterable)

console.log(Object.is(Math.sumPrecise([]), -0));   // true  (empty → -0)

try { Math.sumPrecise([1, 2n]); }         // BigInt not allowed
catch (e) { console.log(e.name); }        // 'TypeError'
14.25 true TypeError
Common Mistakes:
❌ Trusting reduce for precision-sensitive sums: values.reduce((a, b) => a + b, 0) // order-dependent, accumulates error let t = 0; for (const v of values) t += v; // same drift as reduce Math.sumPrecise([total, bigIntCount]) // TypeError — mixing BigInt
✅ One correctly-rounded pass: Math.sumPrecise(values) // order-independent, single rounding Math.sumPrecise(nums.map(Number)) // coerce to Number first if needed
Key Rules:
  • Reach for it when order-independence and precision matter (money, aggregates, scientific data); for a few small integers, plain + is fine.
  • Every element must be a Number — no BigInt, no coercion of strings; an empty iterable yields -0.
  • It rounds once at the end, so the result never depends on the ordering of the inputs — unlike reduce.
Practice Exercises:
Sum the array [0.1, 0.2, 0.3] with both reduce and Math.sumPrecise and report which (if either) equals 0.6 exactly. Then explain why the answer is subtle: precision depends on the stored double values, not the decimal literals.
Write average(nums) using Math.sumPrecise and confirm it is more stable than a reduce-based version for an array mixing very large and very small magnitudes.
Demonstrate order-dependence: find an array whose reduce sum differs when you reverse the array, and show Math.sumPrecise gives the same value both ways.
Connections: Math.sumPrecise targets the same floating-point pitfalls as Number.EPSILON comparisons and Number.toFixed rounding, and it pairs naturally with Intl.NumberFormat for displaying the total. It consumes any iterable — the same protocol behind generators and Set.
Error.captureStackTrace Core Language

When you build custom errors or an error factory, the stack trace often starts inside your own error-creation code — noise that hides where the problem actually occurred. Error.captureStackTrace(target, constructorOpt) installs a clean .stack that begins at the caller, so logs point at the real culprit. It's the standard trick inside every good custom Error subclass.

1Attach a stack to any object. The one-argument form defines a lazy .stack getter on the target — it doesn't even have to be an Error. This is how libraries capture "where was this created?" diagnostics.

JavaScript
const marker = {};
Error.captureStackTrace(marker);        // give a plain object a .stack
console.log(typeof marker.stack);       // 'string'
console.log(marker.stack.split('\n')[1].trim().startsWith('at'));  // true
string true

2The second argument hides frames. Passing a function as constructorOpt omits that function and everything above it from the trace. In a subclass constructor, pass the subclass so the stack starts at the caller, not inside new.

JavaScript
class AppError extends Error {
  constructor(message) {
    super(message);
    this.name = 'AppError';
    // hide the AppError constructor frame — stack begins where the error was thrown:
    if (Error.captureStackTrace) Error.captureStackTrace(this, AppError);
  }
}

function doWork() { throw new AppError('boom'); }
try { doWork(); }
catch (e) { console.log(e.stack.split('\n')[1].includes('doWork')); }  // true
true

3Clean traces from a factory function. If a helper builds and returns errors, capture with the helper as constructorOpt so callers never see the factory in their traces. Error.stackTraceLimit caps how many frames are captured.

JavaScript
function httpError(status) {
  const err = new Error(`HTTP ${status}`);
  Error.captureStackTrace(err, httpError);   // omit httpError itself from the trace
  return err;
}

Error.stackTraceLimit = 5;                    // keep traces short
const e = httpError(404);
console.log(e.stack.split('\n')[1].includes('httpError'));  // false — factory hidden
false
Common Mistakes:
❌ Noisy traces and non-portable assumptions: class E extends Error { constructor(m){ super(m); } } // stack may include constructor noise Error.captureStackTrace(this) // no constructorOpt → own frame leaks in Error.captureStackTrace(this, E) // called unconditionally on Firefox/Safari → crash (undefined)
✅ Feature-guard and pass the constructor: if (Error.captureStackTrace) // V8-only API — guard it Error.captureStackTrace(this, MyError); // trims the constructor frame this.name = 'MyError'; // and always set the name
Key Rules:
  • Pass the current constructor/factory as the second argument to strip its frame — the trace then starts at the code that actually triggered the error.
  • It's a V8 API (Node, Chrome, Deno, Bun) — feature-detect it (if (Error.captureStackTrace)) because Firefox/Safari don't implement it.
  • Error.stackTraceLimit (a number) controls capture depth globally; the target's .stack is computed lazily on first read.
Practice Exercises:
Write a DomainError subclass that sets name, accepts a { cause } option, and calls Error.captureStackTrace(this, DomainError) behind a feature guard. Throw it from a nested function and confirm the first stack frame is the thrower, not the constructor.
Build an assert(cond, msg) helper that throws an Error whose stack omits assert itself (so failures point at the calling line). Verify by inspecting err.stack.
Explain what Error.stackTraceLimit = 0 does, and why capturing stacks has a performance cost worth bounding in hot paths that create many errors.
Connections: This is the low-level tool behind today's reviewed Error Handling Patterns — custom Error classes with name and cause become production-grade once their traces are clean. The lazy .stack getter is itself a property-descriptor trick.

🆕 Isolation & Deferred Loading

Two TC39 Stage 3 features that change when and where code runs: a same-thread sandbox with its own globals, and a way to fetch a module now but pay its evaluation cost only on first use. Support is still landing (recent Node behind flags, bundler/polyfill support) — study the semantics.

ShadowRealm Modern JavaScript

Sometimes you need to run code in a clean global environment — a plugin, a template expression, user-supplied logic — without it polluting or reading your globals, and without spinning up a Worker (which is async and message-passing only). ShadowRealm gives you a fresh set of intrinsics and globals on the same thread, with a deliberately narrow, synchronous boundary.

1A fresh global environment. new ShadowRealm() has its own globalThis, its own Array, Object, etc. realm.evaluate(sourceString) runs a string of code in that realm and returns the result — synchronously.

JavaScript
const realm = new ShadowRealm();

console.log(realm.evaluate('1 + 1'));                 // 2
console.log(realm.evaluate('[1,2,3].map(x => x*2).join()'));  // '2,4,6'

// its globals are isolated from ours:
globalThis.secret = 42;
console.log(realm.evaluate('typeof globalThis.secret'));      // 'undefined'
2 2,4,6 undefined

2Only primitives and functions cross the boundary. The realm can return a primitive, or a wrapped function (a callable that marshals primitive args/returns across). Attempting to pass or return a plain object or array directly throws a TypeError — that restriction is what keeps the two realms isolated.

JavaScript
// a function comes back as a callable wrapper:
const upper = realm.evaluate('(s) => s.toUpperCase()');
console.log(upper('hi'));                 // 'HI'

// returning an object across the boundary is rejected:
try { realm.evaluate('({ a: 1 })'); }
catch (e) { console.log(e.name); }        // 'TypeError'  — objects can't cross

// so serialize instead — pass primitives (e.g. JSON strings):
const run = realm.evaluate('(json) => JSON.stringify(JSON.parse(json).a)');
console.log(run('{"a":7}'));              // '7'
HI TypeError 7

3Import a module into the realm. realm.importValue(specifier, bindingName) asynchronously loads a module inside the realm and hands back the named export (again, only if it's a primitive or function). Great for loading untrusted logic in isolation.

JavaScript
// math.js exports:  export const triple = (n) => n * 3;
const triple = await realm.importValue('./math.js', 'triple');
console.log(triple(5));                    // 15  — ran in the isolated realm

// each realm has its OWN intrinsics, so cross-realm instanceof is false:
const makeArr = realm.evaluate('() => [1,2]');
// (the returned value can only be a primitive/function — this shows the isolation model)
console.log(realm.evaluate('[] instanceof Array'));   // true (its own Array)
15 true
Common Mistakes:
❌ Treating a realm like a Worker or a shared scope: const obj = realm.evaluate('({config: 1})') // TypeError — objects don't cross realm.evaluate('while(true){}') // SAME thread → blocks your main thread arr instanceof realm.Array // cross-realm instanceof is false
✅ Marshal via primitives/functions; use Workers for parallelism: const fn = realm.evaluate('(json) => run(json)') // pass strings/numbers across fn(JSON.stringify(data)) // serialize objects yourself // heavy/parallel work → Web Worker (separate thread), not ShadowRealm
Key Rules:
  • A realm is same-thread and synchronous — it isolates globals, but it does not give you parallelism or protect against infinite loops (use a Worker for that).
  • Only primitives and (wrapped) functions cross the boundary; pass objects by serializing them (e.g. JSON) and rebuilding on the other side.
  • Each realm has its own intrinsics, so instanceof and shared prototypes don't work across realms — it's Stage 3, so guard with a feature check or polyfill.
Practice Exercises:
Create a ShadowRealm, define a global inside it via evaluate, and prove that global is invisible to the outer globalThis (and vice-versa).
Build a safeEval(exprString, inputObj) that runs an arithmetic expression in a realm: serialize inputObj to JSON, pass it as a string argument to a wrapped function, and return a primitive result. Explain each place the boundary forces you to use primitives.
Explain why ShadowRealm is not a security sandbox against denial-of-service (CPU/loops) even though it isolates globals, and when you'd choose a Worker or an out-of-process VM instead.
Connections: ShadowRealm contrasts with Web Workers (separate thread, async postMessage, structured-clone boundary) and with the module system it can import into — see today's deferred module evaluation. Its primitive-only boundary echoes the clone limits you reviewed in Structured Clone.
Deferred Module Evaluation: import defer Modules & Patterns

A module that's expensive to evaluate at startup (builds big tables, registers things, runs setup) slows every launch — even when a given run never uses it. import defer * as ns from '...' lets the module be fetched and parsed up front but delays running its top-level code until the first time you touch the namespace. You get static-import ergonomics with pay-for-what-you-use evaluation.

1Evaluation is deferred to first access. The module's side effects and top-level code run the moment you read any property of the namespace object — not when the import is reached.

JavaScript
// heavy.js
console.log('[heavy] evaluating');
export const compute = (n) => n * n;

// main.js
import defer * as heavy from './heavy.js';
console.log('before first access');
const r = heavy.compute(6);          // <-- evaluation happens HERE
console.log('result', r);
before first access [heavy] evaluating result 36

2Only the namespace form works. You must use import defer * as ns. Named or default imports (import { x } / import d) cannot be deferred — binding a specific value would force the module to evaluate immediately, defeating the purpose.

JavaScript
// ✅ allowed — deferred until a property is read:
import defer * as utils from './utils.js';

// ❌ syntax error — you cannot defer named/default bindings:
// import defer { helper } from './utils.js';
// import defer helper from './utils.js';

// access triggers evaluation once; subsequent reads are cheap:
utils.a();   // evaluates now
utils.b();   // already evaluated — no re-run
(no runtime output — illustrates allowed vs disallowed syntax)

3Deferral is of evaluation, not of the network. The module (and its dependency graph) is still fetched and parsed eagerly — only the synchronous top-level execution is postponed. That's the key difference from dynamic import(), which is async, returns a Promise, and can defer the network fetch / enable code-splitting.

JavaScript
// dynamic import(): async, defers the FETCH, splits bundles
const mod = await import('./heavy.js');   // Promise; network + eval on demand
mod.compute(6);

// import defer: sync-on-access, fetch happens up front, only EVAL is deferred
import defer * as heavy from './heavy.js';
heavy.compute(6);                          // no await; evaluates synchronously on first touch
(conceptual — contrasts async import() with sync import defer)
Common Mistakes:
❌ Wrong expectations about what's deferred: import defer { fn } from './m.js' // SyntaxError — must be `* as ns` // expecting the network request to be skipped (it isn't — only eval is deferred) // expecting an await (access is synchronous, not a Promise)
✅ Use the right tool for the goal: import defer * as ns from './m.js'; ns.fn(); // skip startup EVAL cost, still fetched const m = await import('./m.js'); // skip FETCH / code-split (async)
Key Rules:
  • Use it for modules that are expensive to evaluate but only sometimes used — top-level cost is paid on first namespace access, not at startup.
  • Syntax is restricted to import defer * as ns; named/default deferred imports are a syntax error by design.
  • It defers evaluation only — the module is still fetched/parsed eagerly and access is synchronous. For deferring the fetch or code-splitting, use dynamic import() instead.
Practice Exercises:
Write a heavy.js with a top-level console.log and a main.js that import defer * as it, then predict the exact order of log lines before and after the first property access. Confirm the module's log appears after "before access".
Explain, in one sentence each, when you'd choose import defer vs dynamic import() vs a plain static import — keyed on the three axes fetch cost / evaluation cost / async-ness.
Show that accessing a second property of a deferred namespace does not re-run the module's top-level code (modules evaluate at most once).
Connections: This slots between the eager static import and the fully-async dynamic import() you learned in ES Modules, and it interacts with tree-shaking and barrel files from the reviewed module patterns. A realm's importValue is the isolated cousin of ordinary module loading.

🆕 Locale Metadata

The one Intl primitive that isn't a formatter: a structured, manipulable representation of a locale identifier — the thing you pass into all the formatters. Ships in every modern runtime today.

Intl.Locale Internationalization

A locale tag like 'zh-Hant-TW-u-nu-hanidec' is more than a string — it encodes language, script, region, and calendar/numbering preferences. Splitting it on '-' by hand is fragile and wrong. Intl.Locale parses it into a proper object with typed accessors, lets you derive or override subtags, and can be passed directly to every Intl formatter.

1Parse a tag into structured fields. new Intl.Locale(tag) exposes .language, .region, .script, and .baseName — no manual string slicing.

JavaScript
const loc = new Intl.Locale('en-US');
console.log(loc.language, loc.region);       // 'en' 'US'
console.log(loc.baseName);                   // 'en-US'

const zh = new Intl.Locale('zh-Hant-TW');
console.log(zh.language, zh.script, zh.region);   // 'zh' 'Hant' 'TW'
en US en-US zh Hant TW

2Override subtags via the options bag. Pass a second argument to set or replace parts — region, calendar, numbering system, hour cycle — and toString() rebuilds a valid canonical tag (Unicode extensions land in the -u- section).

JavaScript
const custom = new Intl.Locale('zh', {
  region: 'TW',
  numberingSystem: 'hanidec',
  hourCycle: 'h24',
});
console.log(custom.toString());              // 'zh-TW-u-hc-h24-nu-hanidec'
console.log(custom.numberingSystem);         // 'hanidec'
console.log(custom.hourCycle);               // 'h24'
zh-TW-u-hc-h24-nu-hanidec hanidec h24

3maximize / minimize, and feed it to formatters. maximize() fills in the likely script/region; minimize() strips redundant subtags. A Locale object can be passed straight into any Intl constructor in place of a string.

JavaScript
console.log(new Intl.Locale('en').maximize().toString());          // 'en-Latn-US'
console.log(new Intl.Locale('en-Latn-US').minimize().toString());  // 'en'

// pass the Locale object directly to a formatter:
const loc = new Intl.Locale('de', { region: 'DE' });
console.log(new Intl.NumberFormat(loc).format(1234.5));            // '1.234,5'
en-Latn-US en 1.234,5
Common Mistakes:
❌ Hand-parsing and hand-building locale strings: const [lang, region] = tag.split('-') // breaks on 'zh-Hant-TW', '-u-' extensions const t = `${lang}-${region}-u-nu-${ns}` // easy to produce an invalid/incanonical tag if (tag.startsWith('en')) // fragile substring sniffing
✅ Let Intl.Locale parse and rebuild: const loc = new Intl.Locale(tag); loc.language; loc.region; loc.script; // typed accessors new Intl.Locale('zh', { region:'TW', numberingSystem:'hanidec' }).toString() // valid tag
Key Rules:
  • Use it to parse, inspect, and modify locale identifiers instead of string-splitting — subtags and -u- extensions are easy to get wrong by hand.
  • maximize() adds likely subtags, minimize() removes redundant ones; toString()/baseName give you canonical strings back.
  • A Locale object is accepted anywhere a locale string is — pass it directly to NumberFormat, DateTimeFormat, Collator, etc.
Practice Exercises:
Given the tag 'sr-Cyrl-RS', use Intl.Locale to print its language, script, and region separately, then produce the minimized form and explain what minimize() dropped (and why it was safe to drop).
Build a withCalendar(tag, calendar) helper that returns a new canonical tag with the calendar subtag set (e.g. 'buddhist'), using the options bag rather than string concatenation.
Construct a Locale with a non-default numberingSystem and pass it directly to an Intl.NumberFormat; confirm the digits render in that numbering system, showing that formatters accept a Locale object.
Connections: Intl.Locale is the shared input to the whole Intl formatter family — NumberFormat, RelativeTimeFormat, PluralRules, and Collator from recent days all accept it. Its -u- extension parsing mirrors the BCP 47 tags used by document.documentElement.lang and the Accept-Language header.
