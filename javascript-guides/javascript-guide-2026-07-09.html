Pub/Sub & EventTarget Modules & Patterns 7 days since review
The publish/subscribe pattern decouples emitters from listeners: publishers fire named events, subscribers register callbacks, neither knows the other. The platform gives you this for free — extend the built-in EventTarget and dispatch CustomEvents (whose payload rides in event.detail). addEventListener options like { once: true }, { signal }, and { passive: true } control lifecycle and removal.
JavaScript
class Bus extends EventTarget {
  emit(type, detail) { this.dispatchEvent(new CustomEvent(type, { detail })); }
}
const bus = new Bus();

const ac = new AbortController();
bus.addEventListener('login', e => console.log('hi', e.detail.user),
                     { signal: ac.signal });        // auto-removes on abort
bus.emit('login', { user: 'Ada' });                 // hi Ada
ac.abort();                                          // unsubscribe every listener at once
bus.emit('login', { user: 'Bob' });                 // (nothing — listener gone)
Challenge:
Build a typed EventBus on top of EventTarget with on, once, and off methods, where on returns an unsubscribe function. Then explain why { signal } is a cleaner mass-unsubscribe than tracking every handler reference by hand.
#1 Pitfall: Forgetting to remove listeners — a subscriber that closes over a large object (or a detached DOM node) keeps it alive forever. Prefer { once: true } or an AbortSignal so removal is automatic.
Memory Management & Garbage Collection Performance & Security 7 days since review
JS memory is reclaimed by a mark-and-sweep GC: objects reachable from roots (globals, the stack, live closures) survive; everything else is collected. You don't free(), but you do cause leaks by keeping things reachable unintentionally. The four classic leaks: forgotten timers/listeners, detached DOM nodes held in JS, growing caches/maps with strong keys, and closures capturing more than they need. WeakMap/WeakRef hold values without preventing collection.
JavaScript
// ❌ strong Map keeps every element alive even after it leaves the DOM
const strong = new Map();
function tag(el, meta) { strong.set(el, meta); }   // el can never be GC'd

// ✅ WeakMap: entry disappears when the element is garbage-collected
const meta = new WeakMap();
function tagWeak(el, m) { meta.set(el, m); }        // no leak — key held weakly
Challenge:
Sketch a component that adds a resize listener and a setInterval in its constructor. Show the two lines needed in a destroy() method to prevent leaks, and explain why the interval alone can keep the entire component (and its closed-over DOM) alive indefinitely.
#1 Pitfall: Treating a plain Map or module-level array as a cache without eviction — it only grows. Use a WeakMap (object keys) or a bounded LRU so entries can be collected.
Performance Optimization Performance & Security 7 days since review
Measure before you optimize: performance.mark() / performance.measure() give real timings (never guess). Keep the main thread free — debounce/throttle high-frequency events, defer non-urgent work with requestIdleCallback, and batch DOM reads/writes to avoid layout thrashing. Memoize pure expensive functions. For heavy CPU work, move it off-thread (a Worker) rather than chunking on the main thread.
JavaScript
function memoize(fn) {
  const cache = new Map();
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const v = fn(...args);
    cache.set(key, v);
    return v;
  };
}
performance.mark('a');
const slow = memoize(n => { let s = 0; for (let i=0;i<n;i++) s+=i; return s; });
slow(1e7); slow(1e7);                       // 2nd call is a cache hit
performance.mark('b');
performance.measure('work', 'a', 'b');
console.log(performance.getEntriesByName('work')[0].duration.toFixed(2), 'ms');
Challenge:
Implement debounce(fn, ms) and throttle(fn, ms), then describe a concrete UI case for each (e.g. search-as-you-type vs scroll handler). Explain why a memoize cache keyed by JSON.stringify(args) can itself become a memory leak.
#1 Pitfall: Optimizing by intuition instead of measurement, and chunking heavy CPU work on the main thread with setTimeout — it still blocks input between chunks. Profile first; offload true CPU work to a Worker.
Security: XSS Prevention, CSP & Trusted Types Performance & Security 7 days since review
XSS happens when untrusted data is interpreted as code. Defense in depth: never build HTML by concatenation — use textContent (auto-escapes) or a vetted sanitizer for rich HTML; set a Content-Security-Policy to disallow inline scripts and restrict sources; enable Trusted Types so dangerous sinks (innerHTML, eval) reject raw strings and demand a policy-created object. Treat every input — URLs, JSON, postMessage data — as hostile.
JavaScript
// ❌ injects markup — a <script> or onerror handler runs
el.innerHTML = 'Hi ' + userName;

// ✅ text can never become markup
el.textContent = 'Hi ' + userName;

// ✅ Trusted Types: innerHTML now REQUIRES a policy-sanitized object
const policy = trustedTypes.createPolicy('app', {
  createHTML: (s) => DOMPurify.sanitize(s),
});
el.innerHTML = policy.createHTML(richUserHtml);   // raw strings are rejected
Challenge:
Given renderComment(text) that currently does div.innerHTML = text, rewrite it two ways: safe plain-text output, and safe rich HTML via a sanitizer. Then explain what a CSP header of script-src 'self' blocks that sanitization alone does not.
#1 Pitfall: Blocklist "sanitizing" by stripping <script> — attackers use <img onerror>, javascript: URLs, SVG, and more. Use an allowlist-based, well-tested sanitizer and escape by context (HTML vs attribute vs URL).
WebAssembly Interop Performance & Security 7 days since review
WebAssembly is a fast, portable binary instruction format that runs alongside JS. Load it with WebAssembly.instantiateStreaming(fetch(...)); the result's instance.exports exposes wasm functions callable from JS. Values crossing the boundary are limited to numbers (i32/i64/f32/f64) — strings, arrays, and objects don't pass directly; you read/write them through the shared WebAssembly.Memory (an ArrayBuffer) using typed-array views, which is what the toolchain's JS "glue" automates.
JavaScript
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('math.wasm'),
  { env: { log: (x) => console.log('wasm says', x) } }   // JS imported INTO wasm
);
const { add, memory } = instance.exports;
console.log(add(20, 22));                 // 42 — direct numeric call, near-native speed

// strings live in linear memory, not the arguments:
const bytes = new Uint8Array(memory.buffer, ptr, len);   // view a wasm-allocated buffer
const text = new TextDecoder().decode(bytes);
Challenge:
Explain the full round-trip to pass a JS string into a wasm function and get a string back: who allocates the memory, how the pointer/length are exchanged, and why a TextEncoder/TextDecoder pair sits on the JS side. Then name one case where wasm is not worth it over plain JS.
#1 Pitfall: Assuming you can pass a JS string or object straight to an exported wasm function — only numbers cross directly. Everything else must be marshalled through shared memory, and a stale typed-array view breaks if memory grows (its ArrayBuffer is detached/replaced).
Advanced Debugging Performance & Security 7 days since review
Go beyond console.log: console.table renders arrays/objects as a grid, console.group/groupEnd nest related logs, console.time/timeEnd measure a block, console.assert logs only on failure, and console.trace prints a stack trace. console.dir shows an object's properties (not its DOM rendering). For live inspection use conditional breakpoints, debugger;, and %c CSS formatting. Source maps map minified frames back to original source.
JavaScript
const users = [{ id: 1, name: 'Ada' }, { id: 2, name: 'Bob' }];
console.table(users);                     // tidy grid with id/name columns

console.group('request');
console.log('url', '/api');
console.assert(users.length > 5, 'expected >5 users, got', users.length);  // fires
console.groupEnd();

console.time('parse');
JSON.parse('[1,2,3]');
console.timeEnd('parse');                 // parse: 0.03ms
Challenge:
Take a function that logs several related values in a loop and reorganize its output using console.group, console.table, and console.time. Then explain the difference between console.log(obj) and console.dir(obj) for a DOM element, and when a conditional breakpoint beats a console.log.
#1 Pitfall: Shipping stray console.logs (they leak data and cost time in hot loops) and trusting a logged object's later expanded state — the console often shows a live reference, so a mutated object can look wrong at the moment you inspected it. Log a clone or a primitive snapshot for point-in-time truth.

🆕 Meta-Programming Hooks

Two ways to reach into the language's identity and access machinery: share a symbol by name across every realm, and hand out object access you can later switch off. Both are plain-JS, every-runtime features.

Global Symbol Registry: Symbol.for & Symbol.keyFor Meta-Programming

A plain Symbol('x') is unique — two of them are never equal, which is great for collision-proof private keys but useless when separate pieces of code (different libraries, an iframe, a worker, a module loaded twice) need to agree on the same symbol. Symbol.for(key) looks a symbol up in a process-wide, cross-realm registry by string key, creating it once and returning that same shared symbol everywhere.

1Symbol.for is memoized by string key. The first call with a given key mints a symbol and stores it; every later call with the same key returns the identical symbol — unlike Symbol(), which always makes a new one.

JavaScript
Symbol('app.id') === Symbol('app.id');        // false — two distinct unique symbols
Symbol.for('app.id') === Symbol.for('app.id'); // true  — same registry entry

const a = Symbol.for('app.id');
const b = Symbol.for('app.id');
console.log(a === b);                          // true
false true true

2Symbol.keyFor reverses it. Given a registered symbol it returns the string key; given an ordinary Symbol() it returns undefined — a reliable test for "did this come from the global registry?"

JavaScript
const shared = Symbol.for('color');
const local  = Symbol('color');

console.log(Symbol.keyFor(shared));    // 'color'      (in the registry)
console.log(Symbol.keyFor(local));     // undefined    (not registered)
console.log(shared.description);       // 'color'      (both have a description...)
console.log(local.description);        // 'color'      (...but only one is registered)
color undefined color color

3Real use: a cross-realm protocol key. The registry is shared across realms (main window, workers, iframes), so a well-known-style hook keyed by Symbol.for('org.myapp.serialize') lets independent modules cooperate without importing a shared constant — the string key is the contract.

JavaScript
const SERIALIZE = Symbol.for('org.myapp.serialize');   // any module can recreate this

class Money {
  constructor(cents) { this.cents = cents; }
  [SERIALIZE]() { return { $: this.cents / 100 }; }
}

function pack(obj) {
  return typeof obj[SERIALIZE] === 'function' ? obj[SERIALIZE]() : obj;
}
console.log(pack(new Money(1299)));    // { '$': 12.99 }  — matched by shared key
{ '$': 12.99 }
Common Mistakes:
❌ Mixing registered and unregistered symbols, or leaking the namespace: const KEY = Symbol('cfg') // unique — an importer's Symbol('cfg') won't match obj[Symbol.for('id')] = 1 // global key — another lib using 'id' collides Symbol.keyFor(Symbol('x')) // undefined — easy to assume it echoes 'x'
✅ Namespace registry keys; use unique symbols for privacy: const KEY = Symbol.for('org.myapp.cfg') // collision-proof, cross-realm shared const PRIVATE = Symbol('cfg') // truly local, never in the registry if (Symbol.keyFor(sym)) { /* it's a shared, registered symbol */ }
Key Rules:
  • Symbol.for(key) returns a shared symbol from a global, cross-realm registry; Symbol(desc) always returns a fresh unique one.
  • Symbol.keyFor(sym) returns the registry key for registered symbols, else undefined — use it to detect registry membership.
  • Namespace your registry keys ('org.app.thing') to avoid collisions; reach for a plain Symbol() when you want privacy/uniqueness, not sharing.
Practice Exercises:
Prove the core difference: show that Symbol('k') === Symbol('k') is false but Symbol.for('k') === Symbol.for('k') is true, and that Symbol.keyFor distinguishes the two. Explain when each behavior is the one you want.
Define a shared "hook" symbol via Symbol.for('yourns.observe') and write a function that calls obj[HOOK]() if present. Simulate a second, independently-authored module recreating the same symbol from the string key and confirm it matches.
Explain why the global registry is a namespace-collision risk that plain unique symbols are not, and how Symbol.for relates to the well-known symbols (Symbol.iterator, Symbol.toPrimitive) you studied recently.
Connections: This completes the symbol picture: unique symbols are for private keys, the registry is for shared ones. The [SERIALIZE]() pattern is exactly how well-known symbols like Symbol.iterator and Symbol.toPrimitive work — a protocol keyed by a symbol — and pairs with the Proxy traps below for full meta-programming reach.
Proxy.revocable Meta-Programming

A normal Proxy lives as long as anyone holds a reference — you can never take the access back. Proxy.revocable(target, handler) returns { proxy, revoke }: hand out the proxy, keep the revoke function, and calling revoke() permanently severs the link. Every subsequent operation on the proxy throws. This is the foundation of capability-based security and safe teardown: give code temporary, revocable access to an object.

1Split the proxy from its kill switch. The factory returns two things: the usable proxy and a zero-arg revoke. Until you revoke, the proxy behaves like any other proxy over the target.

JavaScript
const target = { secret: 42 };
const { proxy, revoke } = Proxy.revocable(target, {
  get(t, key) { return key in t ? t[key] : `<no ${String(key)}>`; },
});

console.log(proxy.secret);     // 42        — works normally
console.log(proxy.missing);    // '<no missing>'  — trap runs
revoke();                      // sever access, irreversibly
42 <no missing>

2After revoke(), everything throws. Any trap — get, set, has, function call — on a revoked proxy raises a TypeError. The underlying target is untouched and can still be used directly; only this proxy handle is dead.

JavaScript
try {
  console.log(proxy.secret);   // proxy was revoked above
} catch (e) {
  console.log(e.constructor.name, '-', e.message);
}
// the real object is fine — revocation only kills the proxy handle:
console.log(target.secret);    // 42
TypeError - Cannot perform 'get' on a proxy that has been revoked 42

3Real use: time- or scope-limited access. Give a plugin, callback, or third-party module a revocable view of your state, then revoke it the moment its job is done — so a leaked or long-held reference can't keep poking at your internals afterward. It's also a clean way to invalidate a resource handle on teardown.

JavaScript
function grantScopedAccess(obj, fn) {
  const { proxy, revoke } = Proxy.revocable(obj, {});
  try {
    return fn(proxy);           // plugin can use it only during this call
  } finally {
    revoke();                   // access is dead the instant fn returns
  }
}

const store = { count: 0 };
grantScopedAccess(store, (p) => { p.count++; });   // allowed here
console.log(store.count);       // 1
// any reference the plugin stashed is now inert — further use throws
1
Common Mistakes:
❌ Handing out a plain Proxy, or losing the revoke handle: const p = new Proxy(obj, handler) // no way to ever revoke it const { proxy } = Proxy.revocable(obj, h) // dropped revoke → can never cut access revoke(); revoke() // fine (idempotent) but re-using proxy after → throws
✅ Keep revoke, revoke in a finally, treat post-revoke use as fatal: const { proxy, revoke } = Proxy.revocable(obj, handler) try { use(proxy) } finally { revoke() } // guaranteed teardown // callers must handle the TypeError if they cling to a revoked proxy
Key Rules:
  • Proxy.revocable(target, handler) returns { proxy, revoke }; revoke() permanently disables the proxy (idempotent to call again).
  • After revocation every operation on the proxy throws TypeError; the underlying target is unaffected and still usable directly.
  • Use it for capability security and scoped/temporary access (revoke in a finally); a non-revocable new Proxy can never be withdrawn.
Practice Exercises:
Wrap an object in a revocable proxy, read a property successfully, then revoke() and confirm the next access throws a TypeError while the original target still works. Verify calling revoke() twice does not throw.
Write lease(obj, ms) that returns a proxy and auto-revokes it after ms via setTimeout. Demonstrate that access works before the timeout and throws after.
Explain how Proxy.revocable underpins "capability-based security" — why handing a revocable proxy is safer than handing the raw object, and how it complements the memory-leak concern from today's review (a revoked proxy no longer retains reachability through its handler).
Connections: Revocable proxies extend the Proxy/Reflect trap model into lifecycle control, and echo the teardown discipline from today's Memory Management review and the AbortController-based unsubscription in Pub/Sub. The "revoke in a finally" shape mirrors explicit resource management (using/DisposableStack).

🆕 Text & Data Precision

Two places where the "obvious" API silently corrupts data: JSON round-tripping large numbers, and treating a string as an array of characters. Both have precise, standardized fixes.

JSON Source Text Access: JSON.rawJSON & the reviver context Data & Collections

JSON.parse turns numbers into IEEE-754 doubles, so a 64-bit ID like 12345678901234567890 loses precision before your reviver ever sees it — the damage is done during parsing. The JSON source-access feature fixes both directions: on parse, the reviver's third context argument exposes the exact source text of each value; on stringify, JSON.rawJSON(str) emits a pre-formatted literal verbatim (no re-quoting, no float rounding).

1Read the raw source in the reviver. The reviver now receives (key, value, context); context.source is the original substring for primitive values — so you can build a BigInt from the untouched digits instead of the already-rounded value.

JavaScript
const json = '{ "id": 12345678901234567890, "n": 42 }';

const parsed = JSON.parse(json, (key, value, context) => {
  if (key === 'id') return BigInt(context.source);   // exact digits, no rounding
  return value;
});
console.log(parsed.id);            // 12345678901234567890n  (precise BigInt)
console.log(JSON.parse(json).id);  // 12345678901234568000   (lossy double!)
12345678901234567890n 12345678901234568000

2JSON.rawJSON injects a literal verbatim. Wrap a numeric string in JSON.rawJSON and JSON.stringify writes those exact characters into the output — bypassing the fact that JS has no BigInt JSON representation and no way to force 1.0 to keep its decimal.

JavaScript
const big = 12345678901234567890n;

// ❌ BigInt can't be stringified directly:
// JSON.stringify({ id: big })            // TypeError: Do not know how to serialize a BigInt

// ✅ emit the exact digits as a raw JSON number literal:
const out = JSON.stringify({ id: JSON.rawJSON(big.toString()), price: JSON.rawJSON('1.0') });
console.log(out);                          // {"id":12345678901234567890,"price":1.0}
{"id":12345678901234567890,"price":1.0}

3JSON.isRawJSON and the guardrails. JSON.rawJSON validates that its argument is a single well-formed JSON primitive (number, string, boolean, or null — not an object/array, not with leading whitespace). JSON.isRawJSON(x) tests whether a value is one of these frozen raw-JSON objects.

JavaScript
const raw = JSON.rawJSON('1.0');
console.log(JSON.isRawJSON(raw));      // true
console.log(JSON.isRawJSON(1.0));      // false — a normal number, not a raw literal

try { JSON.rawJSON('{"a":1}'); }       // objects/arrays are NOT allowed
catch (e) { console.log(e.constructor.name); }   // SyntaxError
true false SyntaxError
Common Mistakes:
❌ Trusting the parsed value for big numbers, or hand-editing JSON strings: const id = JSON.parse(s).id // already rounded to a double — data lost BigInt(JSON.parse(s).id) // too late: BigInt of an already-wrong number str.replace(/"id":(\d+)/, ...) // regex-patching JSON to keep precision — fragile
✅ Use source text on the way in, rawJSON on the way out: JSON.parse(s, (k, v, ctx) => k === 'id' ? BigInt(ctx.source) : v) JSON.stringify({ id: JSON.rawJSON(bigId.toString()) }) // exact, no rounding if (JSON.isRawJSON(x)) { /* it's a verbatim literal */ }
Key Rules:
  • The reviver signature is now (key, value, { source }); context.source holds the exact source text for primitive values (undefined for objects/arrays).
  • JSON.rawJSON(text) produces a frozen object that stringifies verbatim; text must be one valid JSON primitive with no surrounding whitespace, or it throws.
  • This is the standard way to round-trip BigInt and precision-sensitive numbers through JSON; JSON.isRawJSON detects raw-JSON values.
Practice Exercises:
Parse {"account": 90071992547409910000} two ways — once plainly, once with a reviver using context.source to produce a BigInt — and show that only the second preserves every digit.
Serialize an object containing a BigInt and a value that must render as 1.0 (not 1) using JSON.rawJSON, and verify the exact output string. Explain why JSON.stringify alone can't do either.
Explain why the precision loss for large integers happens during JSON.parse itself (not in your code afterward), and how context.source sidesteps it. Connect this to the BigInt numeric type you studied earlier.
Connections: This is the JSON counterpart to BigInt and floating-point precision — the same "doubles can't hold every 64-bit integer" problem, solved at the serialization boundary. It complements the replacer/reviver and toJSON mechanics from JSON Advanced, and matters most for APIs returning Snowflake-style IDs.
Unicode Code Points: codePointAt, fromCodePoint & iteration Data & Collections

JS strings are sequences of UTF-16 code units, not characters. Anything outside the Basic Multilingual Plane — emoji, many CJK extensions, math symbols — is stored as a surrogate pair (two code units). So str.length, str[i], charCodeAt, and split('') all cut those characters in half. codePointAt/fromCodePoint and code-point-aware iteration (spread, for...of) handle full Unicode scalars correctly.

1The surrogate-pair trap. An astral character counts as two in .length, and indexing returns a lone, meaningless surrogate half. This is the root cause of "🎉".length === 2 and mojibake when slicing user text.

JavaScript
const s = 'a🎉b';
console.log(s.length);            // 4  — the emoji is 2 code units, not 1
console.log(s[1]);               // '\uD83C'  — half a surrogate pair (garbage)
console.log(s.charCodeAt(1));    // 55356     — just the high surrogate
console.log([...s]);             // ['a', '🎉', 'b']  — spread iterates by code POINT
4 � 55356 [ 'a', '🎉', 'b' ]

2codePointAt and fromCodePoint speak full scalars. codePointAt(i) reads a whole code point (combining a surrogate pair into one number up to 0x10FFFF); String.fromCodePoint(n) builds a string from scalar values — unlike the BMP-only charCodeAt/fromCharCode.

JavaScript
console.log('🎉'.codePointAt(0).toString(16));   // '1f389'  (one scalar, not two units)
console.log('A'.codePointAt(0));                 // 65
console.log(String.fromCodePoint(0x1f389));      // '🎉'
console.log(String.fromCharCode(0x1f389));       // '烉' — WRONG: truncated to 16 bits
1f389 65 🎉 烉

3Iterate and count by code point. for...of, the spread operator, and Array.from all use the string iterator, which walks code points — so [...str].length is the code-point count and reversing via spread won't shred emoji. (True user-perceived "characters" — grapheme clusters like a family emoji or flag — still need Intl.Segmenter.)

JavaScript
const s = 'a🎉b';
console.log([...s].length);                    // 3   — code points, not 4 units
console.log([...s].reverse().join(''));        // 'b🎉a'  — emoji survives

// ❌ split('') / reverse by units mangles the pair:
console.log(s.split('').reverse().join(''));   // 'ba'  (broken)

for (const ch of s) process.stdout?.write?.(ch + ' ');   // a 🎉 b
3 b🎉a b��a
Common Mistakes:
❌ Treating length/index/split as character-accurate: '🎉'.length // 2 — not one character str[i] / str.charAt(i) // may return half a surrogate pair str.split('').reverse().join('') // shreds every astral character String.fromCharCode(0x1f389) // truncates to 16 bits → wrong glyph
✅ Use code-point APIs and iterator-based ops: [...str].length // code-point count str.codePointAt(i) // full scalar value String.fromCodePoint(0x1f389) // correct astral char [...str] / Array.from(str) / for...of // safe per-code-point iteration
Key Rules:
  • Strings are UTF-16 code units; astral characters (emoji, rare CJK) are surrogate pairs, so .length/[i]/charCodeAt/split('') are code-unit operations that split them.
  • Use codePointAt/fromCodePoint and iterator-based operations ([...s], for...of, Array.from) to work in whole code points.
  • Code points still aren't user-perceived characters — combining marks and ZWJ sequences (flags, family emoji) need Intl.Segmenter with granularity: 'grapheme'.
Practice Exercises:
Write codePointLength(str) and a reverseString(str) that both stay correct for input containing emoji (e.g. 'a🎉b'). Show that the naive .length and split('').reverse() versions fail.
Given a code point in hex (e.g. 0x1F600), print the character with String.fromCodePoint and read it back with codePointAt(0).toString(16) to confirm a clean round-trip. Then show why fromCharCode gives the wrong glyph.
Explain the difference between a code unit, a code point, and a grapheme cluster, and pick the right tool for each of: truncating a tweet to N visible characters, validating a single-emoji input, and hashing exact bytes.
Connections: This underlies the Intl.Segmenter grapheme work and the UTF-8 byte handling of TextEncoder/TextDecoder — three layers of "what is a character?" It also explains why the u/v regex flags exist (to match by code point) and why normalize() matters before comparison.

🆕 Node.js Runtime Security

A process-level sandbox baked into the runtime itself: restrict what your Node program (and its dependencies) can touch — the filesystem, child processes, native addons — with a startup flag.

Node.js Permission Model Node.js Runtime

A single malicious or compromised npm dependency runs with your program's full authority — it can read ~/.ssh, spawn processes, or open sockets. Node's Permission Model flips the default from "trusted" to "deny": start with --permission and the process can only touch resources you explicitly grant with flags like --allow-fs-read. It's a runtime sandbox, conceptually similar to Deno's permissions, that limits blast radius without changing your code.

1Opt in at startup; everything is denied by default. Run with node --permission app.js and filesystem, child_process, native addons, and worker_threads are all off until granted. A blocked operation throws an ERR_ACCESS_DENIED error rather than silently failing.

Shell + JavaScript
$ node --permission app.js
# app.js: const fs = require('node:fs');
#         fs.readFileSync('/etc/passwd');
# → throws:
#   Error: Access to this API has been restricted
#     code: 'ERR_ACCESS_DENIED',
#     permission: 'FileSystemRead',
#     resource: '/etc/passwd'
Error [ERR_ACCESS_DENIED]: Access to this API has been restricted permission: 'FileSystemRead' resource: '/etc/passwd'

2Grant narrow scopes with --allow-* flags. Whitelist exactly what the program needs: specific paths for reads/writes, or child processes / addons. Paths can use * wildcards; granting --allow-fs-read=* re-opens all reads (so keep it tight).

Shell
# read only ./config, write only ./tmp, nothing else:
$ node --permission \
       --allow-fs-read=./config \
       --allow-fs-write=./tmp \
       app.js

# other useful grants:
#   --allow-child-process     spawn/exec/fork
#   --allow-worker            worker_threads
#   --allow-addons            native .node addons
#   --allow-fs-read=/etc/app/*   wildcard path scope
# app.js can now read ./config and write ./tmp — every other resource stays denied

3Query permissions at runtime with process.permission. Check before attempting a restricted op (avoids a thrown error) via process.permission.has(scope[, resource]) — useful for degrading gracefully or logging what a plugin would need.

JavaScript
// under: node --permission --allow-fs-read=./config app.js
console.log(process.permission.has('fs.read'));              // false (not ALL reads)
console.log(process.permission.has('fs.read', './config'));  // true  (this path)
console.log(process.permission.has('fs.write', './tmp'));    // false (not granted)
console.log(process.permission.has('child'));                // false

if (process.permission.has('fs.write', './tmp')) {
  // safe to write here
}
false true false false
Common Mistakes:
❌ Over-granting or assuming access, defeating the point: node --permission --allow-fs-read=* app.js // re-opens ALL reads — no protection fs.readFileSync(path) // no guard → throws ERR_ACCESS_DENIED // forgetting --permission entirely → model is OFF, everything allowed as before
✅ Least privilege + runtime checks: node --permission --allow-fs-read=./config --allow-fs-write=./out app.js if (process.permission.has('fs.write', p)) writeFile(p, data) // guard first // grant the narrowest path/scope each dependency actually needs
Key Rules:
  • --permission enables the model and denies filesystem, child_process, worker_threads, and native addons until explicitly granted; blocked calls throw ERR_ACCESS_DENIED.
  • Grant the minimum with --allow-fs-read/--allow-fs-write (path-scoped, wildcards allowed), --allow-child-process, --allow-worker, --allow-addons.
  • Query at runtime with process.permission.has(scope, resource?) to branch before a restricted op; permissions can't be re-granted from inside the process.
Practice Exercises:
Write a small script that reads ./config/app.json and writes ./out/result.txt. Run it under --permission with only the two narrow --allow-fs-* path grants it needs, and confirm an attempt to read an ungranted path throws ERR_ACCESS_DENIED.
Use process.permission.has to gate a child_process.exec call: log a clear "feature disabled" message when --allow-child-process was not passed, instead of letting it throw.
Explain how the permission model limits the blast radius of a supply-chain attack in a transitive dependency, and contrast it with Deno's permission model (which prompts/opts-in per capability). Why is it enforced at process start and not toggleable at runtime?
Connections: This is the runtime-level sibling of today's web security review and the capability thinking behind Proxy.revocable — both are about handing out the least authority needed. It directly constrains child_process, worker_threads, and fs/promises from earlier days, and mirrors the sandboxing story of Deno & Bun.
