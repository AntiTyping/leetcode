Broadcast Channel API Modern Web APIs 17 days since review
new BroadcastChannel(name) creates a named bus for same-origin contexts — tabs, iframes, and workers — to talk to each other. Any context that opens a channel of the same name receives every postMessage sent on it (payloads are structured-cloned), except its own posts. It's fire-and-forget pub/sub with no connection setup, ideal for "log out everywhere" or syncing a theme toggle across tabs. Call close() to detach.
JavaScript
// tab A and tab B both run this:
const bus = new BroadcastChannel('auth');
bus.onmessage = (e) => {
  if (e.data.type === 'logout') location.reload();
};

// tab A triggers it — every OTHER tab's handler fires (not A's own):
bus.postMessage({ type: 'logout' });
Challenge:
Sync a "dark mode" toggle across all open tabs with a BroadcastChannel. Then explain two things: why the tab that sends the message does not receive its own message, and why BroadcastChannel cannot reach a tab on a different origin.
#1 Pitfall: Treating it as persistence or cross-origin transport. It carries no history — a tab opened after a message was sent never sees it (use localStorage/IndexedDB for state), and it is strictly same-origin. Also remember to close() channels in long-lived pages to avoid leaking handlers.
File System Access API Modern Web APIs 17 days since review
showOpenFilePicker() / showSaveFilePicker() / showDirectoryPicker() return FileSystemHandles that let a web app read and write back to real user files — a true "Save" (not just download). Each call requires a user gesture and grants scoped, revocable permission. Get a File for reading via handle.getFile(); write via await handle.createWritable() then write()/close(). Handles can be persisted in IndexedDB to re-open a file later (re-prompting for permission).
JavaScript
const [handle] = await window.showOpenFilePicker();
const file = await handle.getFile();
const text = await file.text();          // read

const writable = await handle.createWritable();
await writable.write(text.toUpperCase()); // edit in place
await writable.close();                   // commit to disk
Challenge:
Sketch a "open → edit → save back to the same file" flow using a single FileSystemFileHandle. Then explain why createWritable() writes to a temp file that only replaces the original on close(), and why the picker calls must be triggered by a click rather than on page load.
#1 Pitfall: Assuming universal support and silent access. The API is Chromium-centric (Safari/Firefox lag or expose only the origin-private file system), requires a secure context and user gesture, and permission is per-session unless you persist the handle and re-request. Never expect to write files without an explicit user grant.
Web Crypto API Security & Communication 17 days since review
crypto.subtle provides real, native primitives — hashing (digest), symmetric/asymmetric encryption, signing, and key derivation (deriveKey, PBKDF2) — all async and operating on ArrayBuffers. crypto.getRandomValues() fills a typed array with cryptographically strong randomness, and crypto.randomUUID() returns a v4 UUID. It's available only in secure contexts (HTTPS/localhost). Never hand-roll crypto or use Math.random() for anything security-sensitive.
JavaScript
// SHA-256 a string:
const data = new TextEncoder().encode('hello');
const digest = await crypto.subtle.digest('SHA-256', data);
const hex = [...new Uint8Array(digest)]
  .map(b => b.toString(16).padStart(2, '0')).join('');
console.log(hex);                     // 2cf24dba5fb0a30e...

const token = crypto.randomUUID();    // strong random id
const salt = crypto.getRandomValues(new Uint8Array(16));
Challenge:
Write an async sha256Hex(str) using TextEncoder + crypto.subtle.digest, returning a lowercase hex string. Then explain why getRandomValues/randomUUID are safe for tokens and salts but Math.random() is not, and why crypto.subtle is unavailable over plain http://.
#1 Pitfall: Using crypto.subtle for password storage directly, or forgetting it's async. A raw SHA-256 is not a password hash — use a slow KDF (PBKDF2 via deriveKey, or a server-side bcrypt/argon2). And every subtle method returns a promise operating on ArrayBuffers, not strings — encode first, await always.
Navigation API Security & Communication 17 days since review
The modern navigation object replaces the awkward history + popstate combo for SPA routing. A single navigate event fires for all same-document navigations (link clicks, navigation.navigate(url), back/forward), and event.intercept({ handler }) lets you take over rendering while the URL updates cleanly. It exposes a full navigation.entries() history list, navigation.currentEntry, and per-navigation state — no more manual pushState bookkeeping.
JavaScript
navigation.addEventListener('navigate', (e) => {
  if (!e.canIntercept || e.hashChange) return;
  const url = new URL(e.destination.url);
  e.intercept({
    async handler() {
      const html = await fetchRoute(url.pathname);
      document.querySelector('#app').innerHTML = html;
    }
  });
});
Challenge:
Outline a client-side router using the navigate event and intercept() that handles link clicks and back/forward uniformly. Then explain what e.canIntercept guards against, and one advantage of navigation.entries() over the classic opaque history object.
#1 Pitfall: Assuming full cross-browser support. The Navigation API is Chromium-first; Safari/Firefox still need a history.pushState + popstate fallback. Also, intercept() only applies to same-document navigations — cross-document (full page loads) and canIntercept === false cases must be left alone.
Canvas & requestAnimationFrame Browser APIs 14 days since review
A <canvas> gives you an immediate-mode 2D bitmap via getContext('2d') — you clear and redraw every frame (clearRect, fillRect, arc, paths). requestAnimationFrame(cb) schedules your draw for the next repaint (~60fps, paused in background tabs) and passes a high-resolution timestamp. Drive motion by elapsed time (delta), not a fixed per-frame step, so speed is frame-rate independent. For crisp rendering, scale the canvas backing store by devicePixelRatio.
JavaScript
const ctx = canvas.getContext('2d');
let last = performance.now(), x = 0;
function frame(now) {
  const dt = (now - last) / 1000; last = now;   // seconds since last frame
  x += 100 * dt;                                // 100 px/sec, fps-independent
  ctx.clearRect(0, 0, canvas.width, canvas.height);
  ctx.fillRect(x, 50, 30, 30);
  requestAnimationFrame(frame);
}
requestAnimationFrame(frame);
Challenge:
Animate a ball bouncing off the canvas edges using a delta-time loop. Then explain why multiplying velocity by dt keeps speed constant across a 30fps and a 144fps display, and why setInterval is a poor substitute for requestAnimationFrame here.
#1 Pitfall: Assuming a fixed 16.67ms per frame (moving by a constant pixel step) — motion then runs faster on high-refresh screens and stutters when frames drop. Always integrate by the timestamp delta. Also forgetting to clearRect leaves smears, and ignoring devicePixelRatio gives blurry output on HiDPI screens.
Module Patterns: Barrel Files, Circular Deps & Tree-Shaking Modules & Patterns 14 days since review
A barrel (index.js re-exporting a folder) tidies imports but can defeat tree-shaking and create load-order surprises. Tree-shaking (dead-code elimination) relies on static import/export and side-effect-free modules — mark "sideEffects": false in package.json and avoid top-level side effects so bundlers can drop unused exports. Circular dependencies resolve (ESM live bindings) but a binding may be read before it's initialized (temporal dead zone) — restructure shared code into a third module.
JavaScript
// ✅ named re-exports let bundlers drop what you don't import
export { Button } from './Button.js';
export { Modal } from './Modal.js';

// ⚠️ circular: a.js imports b, b.js imports a
// import { helper } from './b.js';  // may be undefined if read at module top-level
// → move `helper` into a shared c.js that both import
Challenge:
Explain how a barrel file can accidentally pull an entire library into a bundle even when you import one symbol, and what "sideEffects": false tells the bundler. Then describe how to break a circular dependency between two modules without changing their public behavior.
#1 Pitfall: Relying on dynamic require/computed imports or re-exporting modules with side effects — both block tree-shaking, so unused code ships anyway. And a barrel that eagerly imports everything turns "import one thing" into "load the whole folder"; prefer deep imports for heavy, rarely-used modules.

🆕 Intl: Numbers & Time

Locale-aware formatting that replaces hand-rolled currency strings and brittle "3 days ago" math — correct grouping, currency symbols, units, and relative-time phrasing for any locale.

Intl.NumberFormat Internationalization

Formatting money, percentages, or measurements by hand means guessing at thousands separators, decimal marks, symbol placement, and rounding — all of which vary by locale (1,234.50 vs 1.234,50). Intl.NumberFormat does it correctly for any locale and currency, and its reusable formatter is far faster than repeated toLocaleString calls in a loop.

1Currency, percent, and grouping in one call. Pass a locale and an options object; style: 'currency' (with a currency code) places the symbol and decimals correctly, style: 'percent' multiplies by 100 and appends %, and grouping separators are locale-correct automatically.

JavaScript
const usd = new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' });
console.log(usd.format(1234.5));             // '$1,234.50'

const de = new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' });
console.log(de.format(1234.5));              // '1.234,50 €'  (dot/comma swapped)

console.log(new Intl.NumberFormat('en-US', { style: 'percent' }).format(0.256)); // '26%'
console.log(new Intl.NumberFormat('en-IN').format(1234567));  // '12,34,567' (Indian grouping)
$1,234.50 1.234,50 € 26% 12,34,567

2Units, compact notation, and fraction control. style: 'unit' formats measurements, notation: 'compact' gives "1.2K"/"3M", and minimum/maximumFractionDigits control rounding precision.

JavaScript
console.log(new Intl.NumberFormat('en-US', { style: 'unit', unit: 'kilometer-per-hour' })
  .format(88));                              // '88 km/h'

console.log(new Intl.NumberFormat('en-US', { notation: 'compact' }).format(1_250_000)); // '1.3M'

console.log(new Intl.NumberFormat('en-US', {
  minimumFractionDigits: 2, maximumFractionDigits: 2
}).format(3.14159));                         // '3.14'
88 km/h 1.3M 3.14

3Reuse the formatter (and formatToParts). Constructing a formatter is expensive; build it once and reuse it across many values. formatToParts returns the pieces (currency symbol, integer, decimal, fraction) so you can style them individually.

JavaScript
const money = new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' });
const prices = [9.99, 1000, 42.5];
console.log(prices.map(p => money.format(p)));   // reuse — build once

console.log(money.formatToParts(1234.5).map(p => `${p.type}:${p.value}`));
// ['currency:$','integer:1','group:,','integer:234','decimal:.','fraction:50']
[ '$9.99', '$1,000.00', '$42.50' ] [ 'currency:$', 'integer:1', 'group:,', 'integer:234', 'decimal:.', 'fraction:50' ]
Common Mistakes:
❌ Hand-rolling money/percent and rebuilding the formatter each call: '$' + n.toFixed(2) // wrong for de-DE, no grouping, wrong symbol place (ratio * 100).toFixed(0) + '%' // reinvents style:'percent' items.map(n => new Intl.NumberFormat('en-US', opts).format(n)) // rebuilds N times
✅ One reusable, locale-aware formatter: const fmt = new Intl.NumberFormat(locale, { style: 'currency', currency }); fmt.format(n); // correct symbol, grouping, decimals const pct = new Intl.NumberFormat(locale, { style: 'percent' }); items.map(n => fmt.format(n)); // build once, reuse
Key Rules:
  • Grouping separators, decimal marks, and currency-symbol placement are locale-specific — never concatenate them by hand.
  • style: 'percent' multiplies the input by 100, so pass a ratio (0.26), not 26.
  • Construct the formatter once and reuse it; use formatToParts when you need to style or extract individual pieces.
Practice Exercises:
Write formatMoney(amount, currency, locale) that returns a correctly formatted string, and show that the same amount renders differently for 'en-US'/'USD' vs 'de-DE'/'EUR' (symbol placement and separator swap).
Using notation: 'compact', format an array of view counts (950, 12_400, 3_100_000) and confirm they render as 950, 12K, 3.1M.
With formatToParts, build an HTML string that wraps the currency symbol in a <span class="sym"> and the fraction in a smaller font — without hard-coding $ or the decimal point.
Connections: NumberFormat is the numeric sibling of the date/list/duration formatters in the Intl family, and its rounding behavior connects to the float-precision concerns of Number.EPSILON. Pair it with Intl.PluralRules for correct "1 item" vs "2 items" phrasing.
Intl.RelativeTimeFormat Internationalization

"3 days ago", "in 2 hours", "yesterday" — the strings under every social feed and notification. Hand-rolling them means English-only phrasing, manual singular/plural, and special-casing "1 day ago" → "yesterday". Intl.RelativeTimeFormat produces correct, localized relative phrases from a signed number and a unit, in any language.

1Signed value + unit → localized phrase. format(value, unit) uses the sign to pick past (negative) vs future (positive). With numeric: 'auto', it swaps in idiomatic words like "yesterday"/"tomorrow" where the locale has them.

JavaScript
const rtf = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });
console.log(rtf.format(-1, 'day'));      // 'yesterday'  (numeric:auto → idiomatic)
console.log(rtf.format(1, 'day'));       // 'tomorrow'
console.log(rtf.format(-3, 'day'));      // '3 days ago'
console.log(rtf.format(2, 'hour'));      // 'in 2 hours'

const always = new Intl.RelativeTimeFormat('en', { numeric: 'always' });
console.log(always.format(-1, 'day'));   // '1 day ago'  (no 'yesterday')
yesterday tomorrow 3 days ago in 2 hours 1 day ago

2Any locale, for free. The same call yields correct phrasing (and pluralization/word order) in other languages — no template strings to translate.

JavaScript
console.log(new Intl.RelativeTimeFormat('es', { numeric: 'auto' }).format(-1, 'day')); // 'ayer'
console.log(new Intl.RelativeTimeFormat('fr').format(3, 'month'));   // 'dans 3 mois'
console.log(new Intl.RelativeTimeFormat('de').format(-2, 'week'));   // 'vor 2 Wochen'
ayer dans 3 mois vor 2 Wochen

3You pick the unit — write a small chooser. The API formats one unit; deciding which unit (seconds vs days) from a millisecond delta is up to you. A tiny threshold table turns any time delta into the largest sensible unit.

JavaScript
const rtf = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });
const UNITS = [['year',31536e6],['day',864e5],['hour',36e5],['minute',6e4],['second',1e3]];
function ago(from, to = Date.now()) {
  const diff = from - to;                       // ms; negative = past
  for (const [unit, ms] of UNITS) {
    if (Math.abs(diff) >= ms || unit === 'second')
      return rtf.format(Math.round(diff / ms), unit);
  }
}
console.log(ago(Date.now() - 2 * 36e5));         // '2 hours ago'
console.log(ago(Date.now() + 3 * 864e5));        // 'in 3 days'
2 hours ago in 3 days
Common Mistakes:
❌ Manual relative-time strings and sign mistakes: `${n} days ago` // English-only, breaks '1 days ago', no future rtf.format(3, 'day') // wanting PAST but passing positive → 'in 3 days' const n = Math.floor(diffMs / 86400000) // truncation → off-by-one near boundaries
✅ Let the API phrase and pluralize: rtf.format(-n, 'day') // sign = direction; 'auto' gives 'yesterday' Math.round(diff / ms) // round, then let format() handle plural new Intl.RelativeTimeFormat(locale, { numeric: 'auto' })
Key Rules:
  • The sign of the value selects past vs future — negative is past, positive is future.
  • numeric: 'auto' yields idiomatic words ("yesterday", "next week") where available; 'always' forces the numeric form ("1 day ago").
  • The formatter handles wording and pluralization for one unit; choosing the unit from a raw delta is your job (round, don't floor, near boundaries).
Practice Exercises:
Implement timeAgo(date, locale) that picks the largest sensible unit (year → second) and returns a localized relative string; verify "yesterday" for −1 day with numeric: 'auto'.
Show the difference between numeric: 'auto' and numeric: 'always' for ±1 of day, week, and month, and note which contexts (e.g., a precise audit log) prefer 'always'.
Format the same −2 week delta in three locales ('en', 'ja', 'ar') and observe how word order and direction differ, all from one format(-2, 'week') call.
Connections: RelativeTimeFormat pairs with Intl.DateTimeFormat for absolute timestamps and complements the duration-oriented Intl.DurationFormat; the millisecond deltas it consumes come from Date/performance.now(), the same source used by requestAnimationFrame loops.

🆕 Intl: Plurals & Sorting

The two Intl tools you reach for when text depends on quantity or must be ordered — correct plural categories and locale-aware string comparison that < gets wrong.

Intl.PluralRules Internationalization

n === 1 ? 'item' : 'items' only works for English's two-way split. Many languages have three to six plural categories (Polish, Arabic, Russian), and even English ordinals need four ("1st, 2nd, 3rd, 4th"). Intl.PluralRules tells you which category a number falls into so you can pick the right message form correctly, for any locale.

1select(n) returns a plural category. It returns one of 'zero', 'one', 'two', 'few', 'many', 'other' — the CLDR category for that number in that locale. English cardinals use only 'one' and 'other'.

JavaScript
const en = new Intl.PluralRules('en-US');
console.log(en.select(0));     // 'other'
console.log(en.select(1));     // 'one'
console.log(en.select(2));     // 'other'

const forms = { one: 'item', other: 'items' };
const label = (n) => `${n} ${forms[en.select(n)]}`;
console.log(label(1), '/', label(5));   // '1 item / 5 items'
other one other 1 item / 5 items

2Other languages have more categories. Polish splits into one/few/many; a hard-coded English === 1 check would produce broken grammar. PluralRules gives you the correct bucket to index a per-locale message map.

JavaScript
const pl = new Intl.PluralRules('pl-PL');
console.log(pl.select(1));     // 'one'
console.log(pl.select(3));     // 'few'   ('3 pliki')
console.log(pl.select(5));     // 'many'  ('5 plików')
console.log(pl.select(22));    // 'few'   ('22 pliki')
one few many few

3Ordinals need type: 'ordinal'. The default is cardinal ("2 items"); ordinals ("2nd") use different categories. English ordinals need four suffixes keyed by category.

JavaScript
const ord = new Intl.PluralRules('en-US', { type: 'ordinal' });
const SUFFIX = { one: 'st', two: 'nd', few: 'rd', other: 'th' };
const ordinal = (n) => `${n}${SUFFIX[ord.select(n)]}`;
console.log(ordinal(1), ordinal(2), ordinal(3), ordinal(4), ordinal(21));
// '1st 2nd 3rd 4th 21st'
1st 2nd 3rd 4th 21st
Common Mistakes:
❌ Hard-coded English pluralization and naive ordinals: n === 1 ? 'file' : 'files' // wrong for pl/ru/ar (3+ categories) n + 'th' // '1th', '2th', '3th' — wrong ordinals messages[n] // no message for every possible count
✅ Select the category, then index a message map: const pr = new Intl.PluralRules(locale); forms[pr.select(n)] // 'one' | 'few' | 'many' | 'other' ... const op = new Intl.PluralRules(locale, { type: 'ordinal' }); n + SUFFIX[op.select(n)] // '1st','2nd','3rd','4th'
Key Rules:
  • select(n) returns a category (one/few/many/other…), not a formatted string — you map the category to your own message forms.
  • English cardinals use only one/other, but other locales use up to six categories — never assume a two-way singular/plural split.
  • Ordinals require { type: 'ordinal' } and use different categories than cardinals; pair it with Intl.NumberFormat for the digits.
Practice Exercises:
Write pluralize(n, forms, locale) where forms is a category→string map, and demonstrate it giving grammatically correct output for 1, 3, and 5 in both 'en' and 'pl'.
Build an English ordinal(n) using type: 'ordinal' and verify the tricky cases 11, 12, 13 map to 'th' (not 'st/nd/rd') while 21, 22, 23 map to 'st/nd/rd'.
Explain why Intl.PluralRules does not itself insert the number into the string, and how it composes with Intl.NumberFormat to build a fully localized "N files" message.
Connections: PluralRules is the correctness layer over Intl.NumberFormat for count-dependent text, and its category output is exactly the kind of lookup key you'd store in a message-catalog / i18n system.
Intl.Collator Internationalization

The default Array.prototype.sort compares strings by UTF-16 code unit, so 'Z' < 'a' (uppercase sorts first) and accented letters land in the wrong place — a classic bug in any user-facing list. Intl.Collator provides locale-correct, case- and accent-aware comparison, plus natural numeric ordering, and its compare method is fast enough to pass straight to sort.

1Locale-aware comparison beats <. collator.compare(a, b) returns −1/0/1 like a sort comparator, but ordered by the locale's rules — so accented characters and case are handled correctly instead of by raw code point.

JavaScript
const words = ['zebra', 'Apple', 'ábaco', 'banana'];

console.log([...words].sort());                    // ['Apple','banana','zebra','ábaco'] — wrong
                                                    // 'A' < 'b' by code unit, accents dumped last

const collator = new Intl.Collator('en');
console.log([...words].sort(collator.compare));    // ['ábaco','Apple','banana','zebra'] — correct
[ 'Apple', 'banana', 'zebra', 'ábaco' ] [ 'ábaco', 'Apple', 'banana', 'zebra' ]

2Numeric sorting and sensitivity options. numeric: true orders embedded numbers by value ("file2" before "file10"), and sensitivity controls whether case/accents are considered equal — 'base' treats a, A, and á as the same.

JavaScript
const files = ['file10', 'file2', 'file1'];
console.log([...files].sort());                            // ['file1','file10','file2'] — lexical
console.log([...files].sort(new Intl.Collator('en', { numeric: true }).compare));
                                                           // ['file1','file2','file10'] — natural

const ci = new Intl.Collator('en', { sensitivity: 'base' });
console.log(ci.compare('café', 'CAFE'));                   // 0  (equal: case + accent ignored)
[ 'file1', 'file10', 'file2' ] [ 'file1', 'file2', 'file10' ] 0

3Reuse one collator; it also powers localeCompare. Building a collator is costly, so create it once and reuse its compare. String.prototype.localeCompare accepts the same options and is fine for one-off comparisons, but a shared Collator is faster for sorting large arrays.

JavaScript
const byName = new Intl.Collator('sv', { sensitivity: 'variant' }).compare;
const users = [{ name: 'Öберг' }, { name: 'Anna' }, { name: 'Zoe' }];
users.sort((a, b) => byName(a.name, b.name));   // Swedish order: ...Z, then Ö

// one-off comparisons can use localeCompare with the same options:
console.log('café'.localeCompare('cafe', 'en', { sensitivity: 'base' })); // 0
0
Common Mistakes:
❌ Default sort and lexical number ordering: names.sort() // code-unit order: 'Z' < 'a', accents last files.sort() // 'file10' < 'file2' (string, not numeric) arr.sort((a,b) => a < b ? -1 : 1) // same broken code-unit comparison
✅ Sort through a locale-aware collator: const cmp = new Intl.Collator(locale, { numeric: true }).compare; names.sort(cmp); // locale order, accent/case correct files.sort(cmp); // 'file2' before 'file10'
Key Rules:
  • Never sort user-facing strings with the default comparator — it orders by UTF-16 code unit, mishandling case and accents.
  • numeric: true gives natural number ordering; sensitivity ('base'/'accent'/'case'/'variant') sets what counts as equal.
  • Reuse one Collator.compare for sorting large arrays; localeCompare (same options) is fine for occasional one-off comparisons.
Practice Exercises:
Sort ['résumé', 'Resume', 'react', 'Réact'] with an Intl.Collator('en') and compare the result to the default sort(); explain each difference.
Use numeric: true to sort ['v1.9', 'v1.10', 'v1.2'] in natural version order, and show why the default sort puts v1.10 before v1.2.
With sensitivity: 'base', write a case- and accent-insensitive includesName(list, query) that finds 'JOSÉ' when searching 'jose', using compare(...) === 0.
Connections: Collator fixes the same code-unit ordering trap that bites raw < and default sort; combined with NumberFormat and PluralRules it rounds out the locale-correct display layer. Its comparison logic underpins String.prototype.localeCompare.

🆕 String Pattern Matching

Iterating over every regex match — with capture groups and positions — without the exec-in-a-while-loop boilerplate that so easily infinite-loops.

String.prototype.matchAll Data & Collections

To get every match with its capture groups and index, the old way was a while ((m = re.exec(str))) loop that silently infinite-loops if you forget the g flag or mutate lastIndex. str.matchAll(re) returns a clean iterator of full match objects — groups, index, and named captures included — with no shared mutable state to mismanage.

1An iterator of rich match objects. Each yielded item is an array like String.match returns for a single match: [0] is the whole match, [1..] are capture groups, plus .index and .input. Spread it into an array or loop with for...of.

JavaScript
const str = 'a1b2c3';
for (const m of str.matchAll(/([a-z])(\d)/g)) {
  console.log(`${m[0]} @ ${m.index}  letter=${m[1]} digit=${m[2]}`);
}
// full match, position, and each capture group per iteration
a1 @ 0 letter=a digit=1 b2 @ 2 letter=b digit=2 c3 @ 4 letter=c digit=3

2vs match and named groups. With the g flag, String.match returns only the match strings and discards groups/indices. matchAll keeps everything, including named capture groups on m.groups.

JavaScript
const log = '2026-07-06 2026-01-01';
console.log(log.match(/(\d{4})-(\d{2})-(\d{2})/g));  // ['2026-07-06','2026-01-01'] — no groups

const dates = [...log.matchAll(/(?<y>\d{4})-(?<m>\d{2})-(?<d>\d{2})/g)]
  .map(m => ({ ...m.groups }));
console.log(dates);   // [{y:'2026',m:'07',d:'06'}, {y:'2026',m:'01',d:'01'}]
[ '2026-07-06', '2026-01-01' ] [ { y: '2026', m: '07', d: '06' }, { y: '2026', m: '01', d: '01' } ]

Common Mistakes:
❌ The exec-loop footguns and group-losing match: let m; while ((m = re.exec(s))) {...} // infinite loop if re has no 'g' flag str.match(/(\d+)/g) // drops capture groups AND .index re.lastIndex left dirty between calls // stateful shared regex → skipped/repeated matches
✅ Iterate matches declaratively: for (const m of str.matchAll(/(\d+)/g)) use(m[1], m.index); const all = [...str.matchAll(re)]; // groups + index preserved // fresh iterator each call — no lastIndex bookkeeping
Key Rules:
  • matchAll requires a global (g) regex or it throws TypeError; each yielded item is a full match array with .index, .input, and .groups.
  • Prefer it over a re.exec while loop — no shared lastIndex state to corrupt and no infinite-loop risk.
  • The returned iterator is lazy and single-use; spread into an array ([...it]) if you need to iterate more than once.

Connections: matchAll is the read/iterate counterpart to the replacer-function form of replaceAll, and it returns lazy iterators — the same protocol behind generators and iterator helpers. Named groups here mirror the destructuring you'd apply to each match object.
