R1. Streams API Async Mastery rep 0 · last reviewed 2026-06-10 · 5 days ago
ReadableStream, WritableStream, and TransformStream provide a standard interface for processing data incrementally. Use .pipeThrough() to chain transforms and .pipeTo() to write. Backpressure is handled automatically through the internal queue.
Key Pattern 
const response = await fetch("https://example.com/large-file");
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  console.log(decoder.decode(value, { stream: true }));
}
Challenge

Create a TransformStream that converts incoming text chunks to uppercase, then pipe a ReadableStream through it and collect the output.

Pitfall: Forgetting { stream: true } in TextDecoder.decode() for streaming — partial multi-byte characters get corrupted.
R2. Web Workers Async Mastery rep 0 · last reviewed 2026-06-10 · 5 days ago
Web Workers run JS in a separate thread. Communicate via postMessage/onmessage. Use Transferable objects (ArrayBuffer) for zero-copy data transfer. Workers have no DOM access.
Key Pattern 
// main.js
const worker = new Worker("worker.js");
worker.postMessage({ type: "compute", data: [1, 2, 3] });
worker.onmessage = (e) => console.log("Result:", e.data);

// worker.js
self.onmessage = (e) => {
  const sum = e.data.data.reduce((a, b) => a + b, 0);
  self.postMessage(sum);
};
Challenge

Create an inline Worker using new Blob() and URL.createObjectURL() that computes the nth Fibonacci number without blocking the main thread.

Pitfall: Trying to access document or window inside a Worker — they don't exist in the worker scope.
R3. Service Workers Async Mastery rep 0 · last reviewed 2026-06-10 · 5 days ago
Service Workers are persistent background scripts that intercept network requests. Lifecycle: install → waiting → activate. Use cache strategies (cache-first, network-first, stale-while-revalidate) for offline support.
Key Pattern 
// Register
navigator.serviceWorker.register("/sw.js");

// sw.js — cache-first strategy
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then(cached =>
      cached || fetch(event.request).then(response => {
        const clone = response.clone();
        caches.open("v1").then(cache => cache.put(event.request, clone));
        return response;
      })
    )
  );
});
Challenge

Implement a "stale-while-revalidate" strategy: return the cached response immediately but also fetch a fresh one and update the cache for next time.

Pitfall: Not calling response.clone() before caching — a Response body can only be read once.
R4. Async Generators & for await...of Functions & Iteration rep 1 · last reviewed 2026-06-11 · 4 days ago
async function* yields Promises. for await...of consumes them sequentially. Great for paginated APIs, event streams, and any async iteration.
Key Pattern 
async function* fetchPages(baseUrl) {
  let page = 1;
  while (true) {
    const res = await fetch(`${baseUrl}?page=${page}`);
    const data = await res.json();
    if (data.items.length === 0) return;
    yield data.items;
    page++;
  }
}

for await (const items of fetchPages("/api/users")) {
  console.log(`Got ${items.length} items`);
}
Challenge

Write an async generator pollEndpoint(url, intervalMs) that fetches a URL every N ms and yields the JSON response, stopping when it gets { done: true }.

Pitfall: Using for await...of on a non-async iterable of Promises — it works but awaits each one sequentially, which may not be what you want.
R5. Tagged Template Literals Functions & Iteration rep 1 · last reviewed 2026-06-11 · 4 days ago
A tag function receives (strings[], ...values) and can transform the template. Used for SQL sanitization, HTML escaping, i18n, styled-components, and DSLs.
Key Pattern 
function html(strings, ...values) {
  return strings.reduce((result, str, i) => {
    const val = values[i - 1];
    const escaped = String(val).replace(/&/g, "&amp;").replace(/</g, "&lt;");
    return result + escaped + str;
  });
}

const userInput = '<script>alert("xss")</script>';
console.log(html`<p>Hello, ${userInput}!</p>`);
// <p>Hello, &lt;script&gt;alert(&quot;xss&quot;)&lt;/script&gt;!</p>
Challenge

Write a sql tag function that replaces interpolated values with $1, $2, etc. and collects the values into a params array — returning { text: string, params: any[] }.

Pitfall: Forgetting that strings has one more element than values — the array is [before, between, ..., after].
R6. Typed Arrays & ArrayBuffer Data & Collections rep 0 · last reviewed 2026-06-11 · 4 days ago
ArrayBuffer is raw binary data. Typed arrays (Uint8Array, Int32Array, Float64Array) provide views into the buffer. DataView handles mixed types and endianness. Essential for WebGL, audio, binary protocols, and WebSocket binary frames.
Key Pattern 
const buffer = new ArrayBuffer(16);
const int32View = new Int32Array(buffer);
int32View[0] = 42;
int32View[1] = 100;

const uint8View = new Uint8Array(buffer);
console.log(uint8View[0]); // 42 (little-endian, first byte of int32)

const view = new DataView(buffer);
view.setFloat64(8, 3.14);
console.log(view.getFloat64(8)); // 3.14
Challenge

Write a function that takes a string and encodes it as a Uint8Array using TextEncoder, then decodes it back with TextDecoder. Verify round-trip with Unicode characters.

Pitfall: Typed arrays share the underlying ArrayBuffer — modifying one view's data changes what other views see.

#1 Memory Management & Garbage Collection Performance & Security
Understanding JavaScript's garbage collection helps you avoid memory leaks that degrade performance over time. In SPAs, leaked event listeners, detached DOM nodes, and forgotten closures are the top causes of memory bloat.

Example 1 — How GC works (mark-and-sweep):

JavaScript
// Objects become unreachable → GC collects them
function createData() {
  const obj = { data: new Array(1000000).fill("x") };
  return obj.data.length; // obj is unreachable after return
}

const result = createData();
console.log(result); // 1000000
// The 1M-element array is now eligible for GC
1000000

Example 2 — Memory leak: forgotten event listeners:

JavaScript
// ❌ Leak: listener references the element, element references the listener
function setupHandler() {
  const element = document.getElementById("btn");
  const heavyData = new Array(1000000).fill("leak");

  element.addEventListener("click", () => {
    console.log(heavyData.length); // closure captures heavyData
  });
  // Even if element is removed from DOM, heavyData persists
}

// ✅ Fix: use AbortController or removeEventListener
function setupHandlerFixed() {
  const controller = new AbortController();
  const element = document.getElementById("btn");
  const heavyData = new Array(1000000).fill("data");

  element.addEventListener("click", () => {
    console.log(heavyData.length);
  }, { signal: controller.signal });

  // Later: controller.abort() removes the listener and frees heavyData
  return controller;
}

Example 3 — Memory leak: detached DOM nodes:

JavaScript
// ❌ Detached DOM tree leak
let detachedNodes = [];
function replaceContent() {
  const old = document.getElementById("content");
  detachedNodes.push(old); // keeps reference to removed node!
  const newEl = document.createElement("div");
  newEl.id = "content";
  old.parentNode.replaceChild(newEl, old);
}

// ✅ Fix: don't store references to removed DOM nodes
function replaceContentFixed() {
  const old = document.getElementById("content");
  const newEl = document.createElement("div");
  newEl.id = "content";
  old.parentNode.replaceChild(newEl, old);
  // old is now unreachable → GC collects the entire subtree
}

Example 4 — WeakRef and FinalizationRegistry for cache:

JavaScript
const cache = new Map();
const registry = new FinalizationRegistry((key) => {
  console.log(`Cache entry "${key}" was garbage collected`);
  cache.delete(key);
});

function cacheObject(key, obj) {
  const ref = new WeakRef(obj);
  cache.set(key, ref);
  registry.register(obj, key);
}

function getCached(key) {
  const ref = cache.get(key);
  if (!ref) return undefined;
  const obj = ref.deref();
  if (!obj) {
    cache.delete(key); // already collected
    return undefined;
  }
  return obj;
}

let bigData = { values: new Array(1000000) };
cacheObject("data", bigData);
console.log(getCached("data") !== undefined); // true
bigData = null; // now the original can be GC'd when needed
true
Common Mistakes
❌ Setting up intervals/timeouts that reference closures without clearing them
✅ Always store the interval ID and call clearInterval(id) in cleanup
❌ Adding event listeners in a loop without removal logic
✅ Use AbortController with { signal } option for bulk listener cleanup
❌ Storing DOM references in long-lived objects (global arrays, Maps)
✅ Use WeakRef or WeakMap for caches that shouldn't prevent GC
Key Rules
  • 2026-07-09 (current — Day 28)
  • The #1 leak source in JS: closures that capture large objects + event listeners that are never removed.
  • Use Chrome DevTools → Memory tab → Heap Snapshot to diagnose leaks. Look for "Detached" DOM trees.
  • WeakRef/WeakMap/WeakSet hold references without preventing GC — use for caches and metadata.
Exercises
1 Write a function createLeakyTimer() that demonstrates a closure-based memory leak via setInterval, then write createSafeTimer() that properly cleans up. Explain why the first leaks.
2 Implement a WeakValueMap class that stores values as WeakRefs: set(key, value), get(key) (returns undefined if collected), has(key).
3 Using Chrome DevTools (conceptually), describe the steps to take a heap snapshot, trigger a leak, take another snapshot, and compare them to find the leaked objects.

#2 Performance Optimization Performance & Security
JavaScript runs on a single thread. Long-running code blocks rendering and user interaction. Understanding requestIdleCallback, performance.mark/measure, and memoization lets you write responsive, measurably fast apps.

Example 1 — performance.mark and measure:

JavaScript
performance.mark("start-processing");

const data = Array.from({ length: 100000 }, (_, i) => i);
const sum = data.reduce((a, b) => a + b, 0);

performance.mark("end-processing");
performance.measure("processing-time", "start-processing", "end-processing");

const measure = performance.getEntriesByName("processing-time")[0];
console.log(`Processing took: ${measure.duration.toFixed(2)}ms`);
Processing took: 2.34ms

Example 2 — requestIdleCallback for non-urgent work:

JavaScript
const tasks = Array.from({ length: 10000 }, (_, i) => () => {
  // simulate work
  return Math.sqrt(i) * Math.random();
});

function processTasksWhenIdle(tasks) {
  let index = 0;

  function doWork(deadline) {
    while (index < tasks.length && deadline.timeRemaining() > 1) {
      tasks[index]();
      index++;
    }

    if (index < tasks.length) {
      requestIdleCallback(doWork);
    } else {
      console.log("All tasks complete!");
    }
  }

  requestIdleCallback(doWork);
}

processTasksWhenIdle(tasks);
console.log("UI remains responsive during processing");
UI remains responsive during processing All tasks complete!

Example 3 — Memoization:

JavaScript
function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}

const memoFib = memoize(function fib(n) {
  if (n <= 1) return n;
  return memoFib(n - 1) + memoFib(n - 2);
});

console.time("naive");
console.log(fibonacci(35)); // 9227465 (slow!)
console.timeEnd("naive");   // ~100ms+

console.time("memoized");
console.log(memoFib(35));   // 9227465 (instant)
console.timeEnd("memoized"); // ~0.1ms
9227465 naive: ~100ms+ 9227465 memoized: ~0.1ms

Example 4 — Debounce and throttle:

JavaScript
function debounce(fn, ms) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), ms);
  };
}

function throttle(fn, ms) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall >= ms) {
      lastCall = now;
      return fn.apply(this, args);
    }
  };
}

// Usage
const search = debounce((query) => console.log("Searching:", query), 300);
search("h"); search("he"); search("hel"); search("hello");
// Only logs: "Searching: hello" (after 300ms)

const scroll = throttle(() => console.log("Scroll!"), 100);
// Called at most once per 100ms during rapid scrolling
Searching: hello
Common Mistakes
❌ Using JSON.stringify for memoization keys with object args (order matters, cycles crash)
✅ Use a WeakMap for single-object-arg memoization, or a stable key function
❌ Debouncing with setTimeout and forgetting to clearTimeout the previous call
✅ Always clear the previous timer before setting a new one
❌ Measuring performance with Date.now() (only millisecond precision)
✅ Use performance.now() for sub-millisecond timing
Key Rules
  • Use requestIdleCallback for non-critical background work. Use requestAnimationFrame for visual updates.
  • Memoize pure functions with expensive computation. Use WeakMap-based memoization for object arguments.
  • debounce = wait until calm (search input). throttle = at most once per interval (scroll handler).
Exercises
1 Implement memoize(fn) that handles any number of primitive arguments. Test with a recursive Fibonacci.
2 Write debounce(fn, ms, { leading: boolean }) — when leading is true, call immediately on the first invocation, then wait ms before allowing the next.
3 Use performance.mark() and performance.measure() to measure the execution time of sorting 100,000 random numbers with .sort().
Connections: Relates to event_loop_deep_dive (understanding when tasks run). Memoization connects to functional_patterns (pure functions). Debounce/throttle use closures from Week 1.

#3 Security — XSS Prevention, CSP & Trusted Types Performance & Security
Cross-Site Scripting (XSS) is the most common web vulnerability. Understanding how to prevent it in JavaScript — not just on the server — is essential. CSP and Trusted Types are the modern browser defenses.

Example 1 — XSS vectors and escaping:

JavaScript
// ❌ Direct innerHTML injection
const userInput = '<img src=x onerror="alert(document.cookie)">';
// element.innerHTML = userInput; // XSS!

// ✅ Use textContent for plain text
const div = document.createElement("div");
div.textContent = userInput; // Safe — rendered as text
console.log(div.innerHTML);
// &lt;img src=x onerror="alert(document.cookie)"&gt;

// ✅ Or escape HTML explicitly
function escapeHTML(str) {
  const div = document.createElement("div");
  div.textContent = str;
  return div.innerHTML;
}

console.log(escapeHTML('<script>alert("xss")</script>'));
// &lt;script&gt;alert("xss")&lt;/script&gt;
&lt;img src=x onerror="alert(document.cookie)"&gt; &lt;script&gt;alert("xss")&lt;/script&gt;

Example 2 — Content Security Policy (CSP):

JavaScript
// CSP is set via HTTP header or meta tag:

// <script>console.log("blocked by CSP")</script> — blocked!

Example 3 — Trusted Types API:

JavaScript
// Trusted Types prevent DOM XSS by requiring sanitized inputs
// Enable: Content-Security-Policy: require-trusted-types-for 'script'

if (window.trustedTypes) {
  const policy = trustedTypes.createPolicy("myPolicy", {
    createHTML: (input) => {
      // Sanitize the input
      const div = document.createElement("div");
      div.textContent = input;
      return div.innerHTML;
    },
    createScriptURL: (input) => {
      const url = new URL(input, location.origin);
      if (url.origin !== location.origin) {
        throw new Error("Cross-origin script blocked");
      }
      return url.toString();
    },
  });

  // Now innerHTML ONLY accepts TrustedHTML
  const safe = policy.createHTML('<b>bold</b> <script>alert(1)</script>');
  console.log(safe); // &lt;b&gt;bold&lt;/b&gt; &lt;script&gt;alert(1)&lt;/script&gt;
  // element.innerHTML = safe; // works
  // element.innerHTML = "<script>alert(1)</script>"; // throws TypeError!
}
&lt;b&gt;bold&lt;/b&gt; &lt;script&gt;alert(1)&lt;/script&gt;

Example 4 — Subresource Integrity (SRI):

JavaScript
// When loading external scripts, verify their hash:
// <script src="https://cdn.example.com/lib.js"
//   integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8w"
//   crossorigin="anonymous"></script>

// Generate hash for a script:
async function generateSRI(url) {
  const response = await fetch(url);
  const buffer = await response.arrayBuffer();
  const hash = await crypto.subtle.digest("SHA-384", buffer);
  const base64 = btoa(String.fromCharCode(...new Uint8Array(hash)));
  return `sha384-${base64}`;
}

// Usage
// const integrity = await generateSRI("https://cdn.example.com/lib.js");
// console.log(integrity);
Common Mistakes
❌ Using innerHTML with any user-controlled data
✅ Use textContent for text, createElement + setAttribute for structured content
❌ Using eval(), new Function(), or setTimeout("string") with user input
✅ Never pass strings to eval-like functions. Use CSP script-src to block them.
❌ Setting CSP to 'unsafe-inline' (defeats the purpose)
✅ Use nonce-based or hash-based CSP for inline scripts
Key Rules
  • Default to textContent over innerHTML. When you need HTML, use a sanitization library (DOMPurify) or Trusted Types.
  • CSP is a defense-in-depth layer — it mitigates XSS even when your code has bugs.
  • Never use eval() or document.write() in modern code. They're XSS footguns.

#4 WebAssembly Interop Performance & Security
WebAssembly (Wasm) runs compiled code at near-native speed in the browser. It's used for image processing, games, crypto, and any CPU-heavy work. JS acts as the glue layer to load, instantiate, and communicate with Wasm modules.

Example 1 — Loading and running Wasm:

JavaScript
// Assume we have a simple add.wasm module that exports an `add` function
async function loadWasm() {
  const response = await fetch("add.wasm");
  const buffer = await response.arrayBuffer();
  const { instance } = await WebAssembly.instantiate(buffer);

  console.log(instance.exports.add(2, 3)); // 5
}

// Or use the streaming API (more efficient):
async function loadWasmStreaming() {
  const { instance } = await WebAssembly.instantiateStreaming(
    fetch("add.wasm")
  );
  console.log(instance.exports.add(10, 20)); // 30
}
5 30

Example 2 — Sharing memory between JS and Wasm:

JavaScript
// Create shared memory
const memory = new WebAssembly.Memory({ initial: 1 }); // 1 page = 64KB

// Pass memory to Wasm module
const importObject = {
  env: {
    memory,
    log: (ptr, len) => {
      const bytes = new Uint8Array(memory.buffer, ptr, len);
      console.log(new TextDecoder().decode(bytes));
    },
  },
};

async function run() {
  const { instance } = await WebAssembly.instantiateStreaming(
    fetch("module.wasm"),
    importObject
  );

  // Write data to shared memory from JS
  const view = new Uint8Array(memory.buffer);
  const data = new TextEncoder().encode("Hello from JS!");
  view.set(data, 0);

  // Wasm can read memory.buffer[0..13]
  instance.exports.processData(0, data.length);
}

Example 4 — Real-world: image processing:

JavaScript
async function processImage(imageData) {
  const { instance } = await WebAssembly.instantiateStreaming(
    fetch("image-processor.wasm")
  );

  const { memory, grayscale, malloc, free } = instance.exports;

  // Allocate memory in Wasm for the image
  const pixels = imageData.data;
  const ptr = malloc(pixels.length);

  // Copy pixels into Wasm memory
  new Uint8Array(memory.buffer, ptr, pixels.length).set(pixels);

  // Process in Wasm (fast!)
  grayscale(ptr, imageData.width, imageData.height);

  // Copy result back
  const result = new Uint8ClampedArray(
    memory.buffer, ptr, pixels.length
  );
  const output = new ImageData(
    new Uint8ClampedArray(result),
    imageData.width,
    imageData.height
  );

  free(ptr);
  return output;
}
Common Mistakes
❌ Using WebAssembly.instantiate with a Response instead of ArrayBuffer
✅ Use WebAssembly.instantiateStreaming(fetch(...)) for responses (streams compilation)
❌ Accessing memory.buffer after the Wasm module grows memory (buffer detaches)
✅ Re-read memory.buffer after any call that might grow memory
❌ Trying to pass JS objects directly to Wasm (only numbers and memory)
✅ Serialize data to shared memory (ArrayBuffer), pass pointer + length
Key Rules
  • Wasm only works with numbers (i32, i64, f32, f64) and linear memory. Complex data must be serialized.
  • Use instantiateStreaming over instantiate — it compiles while downloading.
  • Wasm excels at CPU-bound tasks (image processing, physics, crypto). It won't speed up DOM or I/O.
Exercises
1 Write a function that creates a WebAssembly.Memory with 1 page, writes the string "Hello, Wasm!" using TextEncoder, then reads it back with TextDecoder.
2 Using WebAssembly.validate(), explain what it checks and write a test that validates a minimal valid Wasm module (magic bytes + version).
3 Describe how you would pass a JSON object from JS to a Wasm function. What are the steps for serialization, memory allocation, and deserialization?
Connections: Extends typed_arrays (SharedArrayBuffer, DataView). Relates to web_workers (Wasm can run in workers). Memory concepts connect to memory_management_js.

// Source maps connect compiled/minified code back to the original
// Generated automatically by bundlers (Webpack, Vite, esbuild)

// Generates .map files but doesn't add the //# comment

Common Mistakes
❌ Leaving console.log and debugger statements in production code
✅ Use a linter rule (no-console, no-debugger) and strip in build
❌ Using console.log(obj) to snapshot an object (it's a live reference!)
✅ Use console.log(JSON.parse(JSON.stringify(obj))) or console.log({...obj}) for a snapshot
❌ Disabling source maps in production (makes debugging impossible)
✅ Use hidden source maps — uploaded to error tracking but not served to browsers
Key Rules
  • console.table for arrays of objects, console.group for nested output, console.time for timing.
  • Use debugger for programmatic breakpoints — but always remove before committing.
  • Source maps are essential for debugging bundled/transpiled code. Use "hidden" mode in production.
