R1 — Destructuring Advanced 5 days overdue · rep 1
Quick reminder: nested destructuring, defaults, computed property keys, rest in destructuring patterns. These let you extract deeply nested values from objects and arrays in a single declaration.
Key Pattern
JavaScript
// Nested destructuring with rename and default
const response = {
  data: {
    user: {
      name: 'Alice',
      address: { city: 'NYC' }
    }
  },
  meta: { page: 1 }
};

const {
  data: {
    user: {
      name: userName,
      address: { city, zip = '00000' } // rename + default
    }
  },
  ...rest  // captures { meta: { page: 1 } }
} = response;

console.log(city);      // 'NYC'
console.log(zip);       // '00000' (default applied)
console.log(rest);      // { meta: { page: 1 } }

R4 — AbortController 4 days overdue · rep 1
Quick reminder: cancel fetch requests, custom async operations, and implement timeouts. AbortController provides a signal that listeners can check for abort.
Key Pattern
JavaScript
// Cancellable fetch with timeout
async function fetchWithTimeout(url, timeoutMs = 5000) {
  const controller = new AbortController();

  // AbortSignal.timeout() is even simpler (Node 18+):
  // const signal = AbortSignal.timeout(timeoutMs);

  const timeoutId = setTimeout(() => controller.abort(), timeoutMs);

  try {
    const response = await fetch(url, { signal: controller.signal });
    clearTimeout(timeoutId);
    return await response.json();
  } catch (err) {
    clearTimeout(timeoutId);
    if (err.name === 'AbortError') {
      throw new Error(`Request timed out after ${timeoutMs}ms`);
    }
    throw err;
  }
}

// Combining multiple signals (Node 20+)
const userCancel = new AbortController();
const combined = AbortSignal.any([
  userCancel.signal,
  AbortSignal.timeout(10000)
]);

1 Node.js Runtime
1. Node.js Event Loop & Timers new
Understanding when callbacks execute is essential for Node.js performance. The event loop is the heart of Node.js — it determines the order in which your asynchronous code actually runs, and getting it wrong leads to subtle bugs, starvation, and unresponsive servers.

Node.js runs on a single thread with an event loop that cycles through multiple phases. Each phase has a FIFO queue of callbacks to execute. The main phases, in order:

  1. Timers — executes setTimeout() and setInterval() callbacks whose threshold has elapsed
  2. Pending callbacks — I/O callbacks deferred from the previous iteration
  3. Idle/prepare — internal use only
  4. Poll — retrieves new I/O events; executes I/O callbacks (almost everything except timers, close, and setImmediate)
  5. Check — executes setImmediate() callbacks
  6. Close callbacks — e.g., socket.on('close', ...)

Between every phase transition, Node.js drains the microtask queues: first process.nextTick(), then resolved Promises. This is the critical detail most developers get wrong.

Example 1a: Execution Order

node event-loop-order.js
// Run this in Node.js to see the execution order

console.log('1. Synchronous - start');

setTimeout(() => {
  console.log('5. setTimeout (macrotask - timers phase)');
}, 0);

setImmediate(() => {
  console.log('6. setImmediate (macrotask - check phase)');
});

process.nextTick(() => {
  console.log('3. process.nextTick (microtask - highest priority)');
});

Promise.resolve().then(() => {
  console.log('4. Promise.then (microtask - after nextTick)');
});

console.log('2. Synchronous - end');

// Note: setTimeout and setImmediate order is NOT deterministic
// when called from the main module (outside an I/O cycle).
// Inside an I/O callback, setImmediate always fires first.
1. Synchronous - start 2. Synchronous - end 3. process.nextTick (microtask - highest priority) 4. Promise.then (microtask - after nextTick) 5. setTimeout (macrotask - timers phase) 6. setImmediate (macrotask - check phase)
Gotcha: setTimeout vs setImmediate order is not guaranteed When called from the main module (not inside an I/O callback), the order of setTimeout(fn, 0) vs setImmediate(fn) depends on process performance and timer resolution. Inside an I/O callback, setImmediate always fires before setTimeout.

  console.log('D: setTimeout 2');

Example 1c: Why nextTick Can Starve I/O

node nextTick-starvation.js
// WRONG: recursive nextTick starves I/O
// This blocks the event loop from ever reaching the poll phase

function badRecursion() {
  process.nextTick(() => {
    console.log('nextTick executing...');
    badRecursion(); // schedules another nextTick BEFORE I/O can run
  });
}

// If you uncomment badRecursion(), setTimeout NEVER fires:
// badRecursion();

setTimeout(() => {
  console.log('This will NEVER execute if badRecursion is running');
}, 100);

// RIGHT: use setImmediate for recursive async work
// It yields to the event loop between iterations
function goodRecursion(count) {
  if (count <= 0) return;
  setImmediate(() => {
    console.log(`setImmediate iteration ${count}`);
    goodRecursion(count - 1);
  });
}

goodRecursion(3);
setTimeout(() => {
  console.log('setTimeout still fires! I/O is not starved.');
}, 0);
setTimeout still fires! I/O is not starved. setImmediate iteration 3 setImmediate iteration 2 setImmediate iteration 1
Common Mistakes
WRONG: Assuming setTimeout(fn, 0) runs immediately. It runs in the next iteration of the event loop, after synchronous code and all microtasks. The minimum delay is 1ms (clamped by Node.js).
RIGHT: Use queueMicrotask(fn) or Promise.resolve().then(fn) when you need something to run after the current synchronous code but before I/O. Use setImmediate(fn) to run after the current I/O poll phase completes.
WRONG: Recursive process.nextTick() calls. The microtask queue is drained completely before moving to the next event loop phase, so recursive nextTick prevents I/O from ever executing.
RIGHT: Use setImmediate() for recursive async patterns. It schedules work in the check phase, allowing I/O to process between iterations.
Key Rules
  • 2026-07-09 (current — Day 28)
  • process.nextTick() > Promise.then() > setTimeout(fn, 0) > setImmediate() (from main module)
  • Microtasks (nextTick + Promises) drain between every phase and between every macrotask callback
  • setImmediate always fires before setTimeout(fn, 0) inside I/O callbacks
  • process.nextTick can starve I/O — prefer setImmediate or queueMicrotask
  • Use --inspect and Chrome DevTools to visualize event loop behavior

2. Node.js Streams new
Streams let you process data incrementally without loading everything into memory. A 10GB log file? Stream it line-by-line and use only a few KB of RAM. Without streams, you are limited by available memory and your server will crash on large inputs.

Node.js has four stream types:

  • Readable — source of data (e.g., fs.createReadStream, http.IncomingMessage)
  • Writable — destination for data (e.g., fs.createWriteStream, http.ServerResponse)
  • Duplex — both readable and writable (e.g., TCP socket)
  • Transform — duplex that modifies data passing through (e.g., zlib compression)

All streams are EventEmitter instances. The critical events are data, end, error, close, and drain (for backpressure).

Example 2a: Reading a File with Streams

node read-stream.js
import { createReadStream } from 'node:fs';
import { stat } from 'node:fs/promises';

// Compare: loading entire file vs streaming
// BAD for large files:
// const content = await readFile('huge.log', 'utf-8'); // loads ALL into memory

// GOOD: stream processes chunks incrementally
const filePath = 'server.log';

const stream = createReadStream(filePath, {
  encoding: 'utf-8',
  highWaterMark: 64 * 1024  // 64KB chunks (default is 64KB)
});

let lineCount = 0;
let remainder = '';

stream.on('data', (chunk) => {
  // chunk may split a line, so we handle partial lines
  const lines = (remainder + chunk).split('\n');
  remainder = lines.pop();  // last element may be partial
  lineCount += lines.length;
});

stream.on('end', () => {
  if (remainder.length > 0) lineCount++;
  console.log(`Total lines: ${lineCount}`);
});

stream.on('error', (err) => {
  console.error('Stream error:', err.message);
});
Total lines: 48231

Example 2b: Transform Stream — Line-by-Line Uppercase

node transform-stream.js
import { Transform } from 'node:stream';
import { createReadStream, createWriteStream } from 'node:fs';

// Custom Transform: uppercase each line
class UpperCaseTransform extends Transform {
  constructor() {
    super({ encoding: 'utf-8' });
    this._remainder = '';
  }

  _transform(chunk, encoding, callback) {
    const text = this._remainder + chunk.toString();
    const lines = text.split('\n');
    this._remainder = lines.pop();  // save incomplete last line

    for (const line of lines) {
      this.push(line.toUpperCase() + '\n');
    }
    callback();  // signal we're done processing this chunk
  }

  _flush(callback) {
    // Called when the readable side ends
    if (this._remainder) {
      this.push(this._remainder.toUpperCase() + '\n');
    }
    callback();
  }
}

// Usage:
// createReadStream('input.txt')
//   .pipe(new UpperCaseTransform())
//   .pipe(createWriteStream('output.txt'));

// Standalone test
const upper = new UpperCaseTransform();
upper.on('data', (chunk) => process.stdout.write(chunk));
upper.write('hello world\n');
upper.write('streams are powerful\n');
upper.end();

Example 2c: pipeline() — Safe Piping with Error Handling

node pipeline-example.js
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

// WRONG: .pipe() swallows errors and doesn't clean up
// createReadStream('input.txt')
//   .pipe(createGzip())
//   .pipe(createWriteStream('output.gz'));
// If the read stream errors, the write stream is left open!

// RIGHT: pipeline() propagates errors and cleans up ALL streams
async function compressFile(input, output) {
  try {
    await pipeline(
      createReadStream(input),
      createGzip(),
      createWriteStream(output)
    );
    console.log(`Compressed ${input} -> ${output}`);
  } catch (err) {
    console.error('Pipeline failed:', err.message);
    // All streams are automatically destroyed/cleaned up
  }
}

// compressFile('data.json', 'data.json.gz');

Example 2d: Async Iteration Over Streams

node async-stream-iteration.js
import { createReadStream } from 'node:fs';
import { createInterface } from 'node:readline';

// Modern approach: async iteration with readline
async function processLogFile(filePath) {
  const fileStream = createReadStream(filePath, { encoding: 'utf-8' });
  const rl = createInterface({ input: fileStream, crlfDelay: Infinity });

  const errorCounts = new Map();

  for await (const line of rl) {
    const match = line.match(/\[ERROR\]\s+(\w+)/);
    if (match) {
      const errorType = match[1];
      errorCounts.set(errorType, (errorCounts.get(errorType) || 0) + 1);
    }
  }

  // Stream and readline are automatically closed after the loop

  return errorCounts;
}

// Readable streams are async iterables since Node 10
// This also works:
async function countBytes(filePath) {
  const stream = createReadStream(filePath);
  let totalBytes = 0;

  for await (const chunk of stream) {
    totalBytes += chunk.length;
  }

  console.log(`Total bytes: ${totalBytes}`);
  return totalBytes;
}
Common Mistakes
WRONG: Not handling 'error' events on streams. An unhandled error event crashes the entire Node.js process with an uncaught exception.
RIGHT: Always attach an 'error' handler to every stream, or use pipeline() which propagates errors through the chain and cleans up resources.
WRONG: Ignoring backpressure. Piping a fast readable into a slow writable without respecting .write() return value causes unbounded memory growth.
RIGHT: Check the return value of writable.write(chunk). If it returns false, stop writing until the 'drain' event fires. Or use pipeline() which handles backpressure automatically.
Key Rules
  • Use pipeline() (from node:stream/promises) instead of .pipe()
  • Always handle 'error' events or use pipeline()
  • Respect backpressure: check .write() return value, wait for 'drain'
  • highWaterMark is the buffer threshold, NOT a hard limit
  • Readable streams are async iterables — use for await...of
  • _transform(chunk, enc, callback) must always call callback()

3. Node.js fs/promises & path new
File system operations are the bread and butter of Node.js scripting, CLI tools, and build systems. The fs/promises API gives you clean async/await file operations, and path ensures your code works across Windows, macOS, and Linux.

Node.js provides three flavors of fs APIs:

  • fs — callback-based (legacy, avoid in new code)
  • fs/promises — Promise-based (preferred)
  • fs.*Sync — synchronous (blocks the event loop, fine for scripts and CLI tools, never in servers)

The path module handles path manipulation in a platform-agnostic way. Never concatenate path segments with + or template literals.

Example 3a: Basic File Operations

node fs-basics.js

Example 3b: Directory Traversal

node directory-traversal.js
import { readdir, stat } from 'node:fs/promises';
import { join, extname } from 'node:path';

// Node 18.17+: recursive readdir
async function findAllFiles(dir) {
  const entries = await readdir(dir, {
    withFileTypes: true,  // returns Dirent objects, not strings
    recursive: true       // Node 18.17+ only
  });

  return entries
    .filter(entry => entry.isFile())
    .map(entry => join(entry.parentPath ?? entry.path, entry.name));
}

// Filter by extension
async function findByExtension(dir, ext) {
  const allFiles = await findAllFiles(dir);
  return allFiles.filter(f => extname(f) === ext);
}

// Node 22+: glob support
// import { glob } from 'node:fs/promises';
// const jsFiles = await glob('src/**/*.js');

// Example: list JS files in a project
// const jsFiles = await findByExtension('./src', '.js');

// Manual recursive traversal (works in all Node versions)
async function walkDir(dir, callback) {
  const entries = await readdir(dir, { withFileTypes: true });

  for (const entry of entries) {
    const fullPath = join(dir, entry.name);
    if (entry.isDirectory()) {
      await walkDir(fullPath, callback);
    } else {
      await callback(fullPath, entry);
    }
  }
}

Common Mistakes
WRONG: Using sync methods in HTTP servers. fs.readFileSync() blocks the entire event loop — while it reads, no other request can be processed. One slow disk read stalls all clients.
RIGHT: Use fs/promises in servers: const data = await readFile(path, 'utf-8'). Reserve sync methods for one-shot CLI scripts or module initialization.
WRONG: Building paths with string concatenation: const p = dir + '/' + file. Breaks on Windows (backslash separator) and creates double-slash bugs.

WRONG: Using path.join() with URLs: path.join('https://api.com', 'users') outputs 'https:/api.com/users' (strips a slash). Use the URL constructor instead.

4. Testing with Node Test Runner new
Node.js has a built-in test runner since version 18 — no need for Jest, Mocha, or Vitest for basic testing. Zero dependencies means faster CI, simpler setup, and one fewer supply-chain risk. It supports describe/it, subtests, mocking, snapshots, and code coverage.

The built-in test runner lives in node:test. Key features:

  • describe() and it() for BDD-style organization
  • test() for top-level tests with subtests via t.test()
  • test.todo() and test.skip() for planning and conditional skipping
  • mock.fn() and mock.method() for spying and stubbing
  • mock.module() for mocking entire modules (experimental)
  • Built-in assert module for assertions

Run tests with: node --test (auto-discovers **/*.test.js) or node --test src/ for a specific directory.

Example 4a: Basic Tests

// The function we're testing
function clamp(value, min, max) {
  if (min > max) throw new RangeError('min must be <= max');
  return Math.min(Math.max(value, min), max);
}

  it('returns the value when within range', () => {
    assert.strictEqual(clamp(5, 0, 10), 5);
  });

  it('clamps to minimum when value is below range', () => {
    assert.strictEqual(clamp(-3, 0, 10), 0);
  });

  it('clamps to maximum when value is above range', () => {
    assert.strictEqual(clamp(15, 0, 10), 10);
  });

  it('works with equal min and max', () => {
    assert.strictEqual(clamp(5, 3, 3), 3);
  });

  it('throws when min > max', () => {
    assert.throws(
      () => clamp(5, 10, 0),
      {
        name: 'RangeError',
        message: 'min must be <= max'
      }
    );
  });

  it('handles edge cases with Infinity', () => {
    assert.strictEqual(clamp(5, -Infinity, Infinity), 5);
    assert.strictEqual(clamp(Infinity, 0, 10), 10);
  });

node --test cache.test.js
import { test, describe, it, before, after, beforeEach } from 'node:test';
import assert from 'node:assert/strict';

// Simple in-memory cache for testing
class Cache {
  #store = new Map();
  #ttl;

  constructor(ttlMs = 5000) {
    this.#ttl = ttlMs;
  }

  set(key, value) {
    this.#store.set(key, { value, expires: Date.now() + this.#ttl });
  }

  get(key) {
    const entry = this.#store.get(key);
    if (!entry) return undefined;
    if (Date.now() > entry.expires) {
      this.#store.delete(key);
      return undefined;
    }
    return entry.value;
  }

  get size() { return this.#store.size; }
  clear() { this.#store.clear(); }
}

describe('Cache', () => {
  let cache;

  beforeEach(() => {
    cache = new Cache(1000); // 1 second TTL
  });

  it('stores and retrieves values', () => {
    cache.set('name', 'Alice');
    assert.strictEqual(cache.get('name'), 'Alice');
  });

  it('returns undefined for missing keys', () => {
    assert.strictEqual(cache.get('missing'), undefined);
  });

  // Subtests: group related assertions inside a test
  test('TTL behavior', async (t) => {
    await t.test('returns value before TTL expires', () => {
      const fastCache = new Cache(10000);
      fastCache.set('key', 'value');
      assert.strictEqual(fastCache.get('key'), 'value');
    });

    await t.test('returns undefined after TTL expires', async () => {
      const shortCache = new Cache(50); // 50ms TTL
      shortCache.set('key', 'value');
      await new Promise(resolve => setTimeout(resolve, 60));
      assert.strictEqual(shortCache.get('key'), undefined);
    });
  });

  // Mark planned but not-yet-implemented tests
  it.todo('should support LRU eviction');
  it.todo('should emit events on set/delete');

  // Skip tests conditionally
  it('uses structuredClone for deep copies', { skip: typeof structuredClone === 'undefined' }, () => {
    const obj = { nested: { value: 42 } };
    cache.set('obj', structuredClone(obj));
    obj.nested.value = 0;

import { performance, PerformanceObserver } from 'node:perf_hooks';

// performance.mark / performance.measure: precise timing
function expensiveOperation() {
  performance.mark('op-start');

  // Simulate work
  let sum = 0;
  for (let i = 0; i < 10_000_000; i++) {
    sum += Math.sqrt(i);
  }

  performance.mark('op-end');
  performance.measure('expensive-operation', 'op-start', 'op-end');

  const [entry] = performance.getEntriesByName('expensive-operation');
  console.log(`Operation took: ${entry.duration.toFixed(2)}ms`);

  // Clean up
  performance.clearMarks();
  performance.clearMeasures();

  return sum;
}

expensiveOperation();

// PerformanceObserver: react to performance entries automatically
const obs = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(`[perf] ${entry.name}: ${entry.duration.toFixed(2)}ms`);
  }
});
obs.observe({ entryTypes: ['measure'] });

// Now any performance.measure() will be logged by the observer
performance.mark('A');
// ... some work ...
performance.mark('B');
performance.measure('work-chunk', 'A', 'B');

Common Mistakes
WRONG: Leaving console.log() statements in production code. They are synchronous in Node.js (when writing to a terminal) and can significantly impact performance under high throughput.
RIGHT: Use a proper logging library (pino, winston) for production. Gate debug output behind environment variables: if (process.env.DEBUG) console.log(...). Or use Node.js built-in util.debuglog('section').
WRONG: Using console.log(obj) and assuming you see a snapshot. In browsers, the console displays a live reference — expanding the object later shows the CURRENT state, not the state at log time.
RIGHT: Use console.log(JSON.stringify(obj, null, 2)) or console.log(structuredClone(obj)) to capture the state at the time of logging. In Node.js, console.log serializes immediately, so this is mainly a browser issue.
Key Rules
  • console.table() for arrays/objects; console.dir(obj, {depth: null}) for deep objects
  • console.time(label)/timeEnd(label) for quick benchmarks
  • console.trace() shows "who called me" without throwing
  • console.count(label) tracks invocation frequency
  • console.assert(condition, msg) only logs on failure (does NOT throw)
  • Use performance.mark()/measure() for precise sub-millisecond timing
  • Gate debug output behind process.env.DEBUG or use util.debuglog()
  • node --inspect + Chrome DevTools for breakpoints, profiling, and memory analysis

import { createReadStream, createWriteStream } from 'node:fs';

// WRONG: if readable errors, writable stays open (resource leak)
createReadStream('missing-file.txt')
  .pipe(createWriteStream('output.txt'));
// Throws uncaught error, writable stream is NOT cleaned up

// RIGHT: use pipeline
import { pipeline } from 'node:stream/promises';

try {
  await pipeline(
    createReadStream('missing-file.txt'),
    createWriteStream('output.txt')
  );
} catch (err) {
  console.error('Pipeline handled error:', err.code);
  // Both streams are properly destroyed
}

Gotcha 3: assert vs assert/strict

import assert from 'node:assert';           // legacy mode
import strictAssert from 'node:assert/strict'; // strict mode

// Legacy assert.equal uses == (loose equality)
assert.equal(1, '1');  // PASSES! (1 == '1' is true)

// Strict assert.equal uses === (strict equality)

// Always use assert/strict in tests

Gotcha 5: setTimeout Minimum Delay

// The minimum delay is clamped to 1ms in Node.js

Gotcha 6: fs.watch Fires Multiple Times

import { watch } from 'node:fs';

// fs.watch fires multiple events for a single file save
// (because editors often write to a temp file, then rename)
watch('config.json', (eventType, filename) => {
  console.log(eventType, filename);
});
// Saving config.json might print:
// rename config.json
// change config.json
// change config.json   <-- duplicate!

// Fix: debounce the handler
function debounced(fn, delayMs) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delayMs);
  };
}

watch('config.json', debounced((eventType, filename) => {
  console.log('File changed (debounced):', filename);
}, 100));

Gotcha 7: Unhandled Stream Errors Crash the Process

import { createReadStream } from 'node:fs';

// WRONG: no error handler = process crash
// const stream = createReadStream('nonexistent.txt');
// stream.on('data', chunk => console.log(chunk));
// ^ throws: Error: ENOENT: no such file or directory

// RIGHT: always handle errors
const stream = createReadStream('nonexistent.txt');
stream.on('data', chunk => console.log(chunk));
stream.on('error', err => {
  console.error('Stream error handled:', err.code);
  // ENOENT, EACCES, etc.
});

// BEST: use pipeline which handles errors for you
import { pipeline } from 'node:stream/promises';
try {
  await pipeline(
    createReadStream('nonexistent.txt'),
    process.stdout
  );
} catch (err) {
  console.error('Caught:', err.code);
}

Gotcha 8: readFile Loads Entire File Into Memory

import { readFile } from 'node:fs/promises';
import { createReadStream } from 'node:fs';
import { createInterface } from 'node:readline';

// WRONG: for large files, this crashes with out-of-memory
// const bigLog = await readFile('10gb-access.log', 'utf-8');
// const lines = bigLog.split('\n');  // even more memory!

// RIGHT: stream the file
async function countErrors(filePath) {
  const rl = createInterface({
    input: createReadStream(filePath, { encoding: 'utf-8' }),
    crlfDelay: Infinity
  });

  let errorCount = 0;
  for await (const line of rl) {
    if (line.includes('[ERROR]')) errorCount++;
  }
  return errorCount;
}

// Memory usage stays constant regardless of file size!
M Today's Mental Model

Think of Node.js as a Restaurant Kitchen

The event loop is the head chef: there is only one, and they execute tasks one at a time. But they are incredibly fast because they never wait — while a dish is in the oven (I/O), they start the next one.

Streams are the conveyor belt: ingredients arrive in small batches, get processed, and move to the next station. You never need to pile all ingredients on the counter at once (memory). If the next station is busy (backpressure), the belt pauses.

fs/promises is the pantry: the chef sends an assistant to get ingredients (async), never goes to get them personally (sync). The async version keeps the kitchen running; the sync version makes everyone wait.

Tests are the food critic: they verify each dish meets the recipe before it leaves the kitchen. mock.fn() is like using rubber food for the photo shoot — you test the plating without needing real ingredients.

console methods are your kitchen cameras and timers: console.time is the egg timer, console.trace is the security camera showing who touched the dish last, and console.table is the order board showing all tickets at a glance.
