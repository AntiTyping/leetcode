Pub/Sub & EventTarget Modules & Patterns 14 days since review
Any object can become an event emitter by extending the built-in EventTarget. Fire named events with dispatchEvent(new CustomEvent(type, { detail })) and subscribe with addEventListener(type, handler, options). The options object supports { once: true } (auto-remove after first call) and { signal } (remove when an AbortController aborts) — the cleanest way to avoid listener leaks.
JavaScript
class Store extends EventTarget {
  #state = {};
  set(key, value) {
    this.#state[key] = value;
    this.dispatchEvent(new CustomEvent('change', { detail: { key, value } }));
  }
}

const store = new Store();
const ctrl = new AbortController();
store.addEventListener('change',
  (e) => console.log('changed:', e.detail),
  { signal: ctrl.signal });        // auto-unsubscribe on ctrl.abort()

store.set('user', 'Ada');          // changed: { key: 'user', value: 'Ada' }
ctrl.abort();                      // every listener added with this signal is removed
Challenge:
Build a typed EventBus on top of EventTarget exposing on(type, fn) that returns an unsubscribe function (implement it via an internal AbortController per subscription), plus emit(type, detail). Prove that calling unsubscribe() stops delivery and that a { once: true } handler fires exactly once.
#1 Pitfall: Passing a fresh arrow function to removeEventListener — it must be the same reference you added, so anonymous handlers can never be removed by hand. Prefer { signal } or { once: true }. Also, the payload lives on event.detail, not on the event object directly.
Memory Management & Garbage Collection Performance & Security 14 days since review
JS uses a tracing garbage collector (mark-and-sweep): an object is collected when it is no longer reachable from a root (globals, the stack, active closures). Leaks come from unintended reachability — detached DOM nodes still referenced by JS, forgotten setIntervals, event listeners never removed, and unbounded caches/Maps that only grow. WeakMap/WeakSet/WeakRef hold references that do not keep their target alive.
JavaScript
// LEAK: strong Map keyed by DOM nodes keeps detached nodes alive forever
const metaLeak = new Map();
function tag(node) { metaLeak.set(node, { seen: Date.now() }); }

// FIX: WeakMap lets the node (and its entry) be collected once detached
const meta = new WeakMap();
function tagSafe(node) { meta.set(node, { seen: Date.now() }); }

// LEAK: timer captures `bigData` in a closure that never clears
let bigData = new Array(1e6).fill('x');
const id = setInterval(() => console.log(bigData.length), 1000);
// FIX: clearInterval(id); bigData = null;   // break the reference chain
Challenge:
Describe (and sketch code for) three distinct leak sources — a detached-DOM cache, a growing module-level array, and a subscription whose listener is never removed — then give the exact fix for each (WeakMap, an eviction bound, and an AbortController). Explain how you would confirm the fix using a heap snapshot's "Detached" filter.
#1 Pitfall: Assuming WeakRef/FinalizationRegistry give deterministic cleanup — GC timing is unspecified, finalizers may never run, so never use them for critical resource release (use using/try…finally for that). WeakMaps are for associating data, not for scheduling cleanup.
Performance Optimization Performance & Security 8 days since review
Measure before optimizing: performance.mark() / performance.measure() give real numbers instead of guesses. Cut redundant work with memoization; throttle high-frequency events with debounce/throttle; defer non-urgent work with requestIdleCallback (and rendering work with requestAnimationFrame). In the DOM, batch reads then writes to avoid layout thrashing (forced synchronous reflow).
JavaScript
const memoize = (fn) => {
  const cache = new Map();
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const val = fn(...args);
    cache.set(key, val);
    return val;
  };
};

performance.mark('start');
// ...expensive work...
performance.mark('end');
performance.measure('work', 'start', 'end');
console.log(performance.getEntriesByName('work')[0].duration);   // ms
Challenge:
Implement debounce(fn, ms) and throttle(fn, ms) and articulate exactly when each is correct (search-as-you-type vs scroll handler). Then take a loop that reads offsetHeight and writes style.height for 100 elements and rewrite it to batch all reads before all writes — explain why the original forces 100 reflows.
#1 Pitfall: Premature or unmeasured optimization — micro-tuning a function that isn't the bottleneck while a layout-thrashing loop or an O(n²) render dominates. Also, an unbounded memoization cache is itself a memory leak; bound it (LRU) for hot paths with unbounded key spaces.
Security: XSS Prevention, CSP & Trusted Types Performance & Security 8 days since review
XSS happens when untrusted data reaches an HTML/JS sink. Default to text, not markup: textContent and setAttribute never execute. Avoid innerHTML, outerHTML, and insertAdjacentHTML with user input; if you must render HTML, sanitize (e.g. DOMPurify). Defense in depth: a Content-Security-Policy header blocks inline scripts and untrusted origins, and Trusted Types force all dangerous-sink assignments through a vetted policy.
JavaScript
// ❌ el.innerHTML = userInput;            // script/onerror executes
el.textContent = userInput;               // ✅ inert text, always safe

// Trusted Types: make raw string → innerHTML a TypeError unless vetted
if (window.trustedTypes) {
  const policy = trustedTypes.createPolicy('sanitize', {
    createHTML: (s) => DOMPurify.sanitize(s),
  });
  el.innerHTML = policy.createHTML(userInput);   // only path allowed under CSP
}
// Response header: Content-Security-Policy: require-trusted-types-for 'script'
Challenge:
Take a comment widget that renders comment.body via innerHTML and make it XSS-safe two ways: (1) switch to textContent, and (2) if limited formatting is required, route through a sanitizer wrapped in a Trusted Types policy. Then write the CSP header that disables inline scripts and enforces require-trusted-types-for 'script', and explain what each directive stops.
#1 Pitfall: Trusting a blocklist or a hand-rolled "escape" regex — attackers bypass naive filters with obscure vectors (SVG, javascript: URLs, mutation XSS). Allow-list and use a maintained sanitizer. Also, escaping for HTML is not the same as escaping for a JS string, a URL, or a CSS context.
WebAssembly Interop Performance & Security 8 days since review
WebAssembly runs a compact binary at near-native speed. Load with WebAssembly.instantiateStreaming(fetch(url), imports) (streaming compile) to get an instance whose exports are callable functions and a Memory. Wasm only speaks numbers (i32/i64/f32/f64) — strings and arrays cross the boundary as offsets into the shared WebAssembly.Memory, which JS reads via a typed-array view over memory.buffer.
JavaScript
const imports = { env: { log: (x) => console.log('from wasm:', x) } };
const { instance } = await WebAssembly.instantiateStreaming(
  fetch('add.wasm'), imports);

console.log(instance.exports.add(2, 3));      // 5 — direct numeric call

// read bytes the module wrote into shared linear memory:
const mem = new Uint8Array(instance.exports.memory.buffer);
const view = mem.subarray(0, 4);              // typed-array window over wasm memory
console.log([...view]);
Challenge:
Explain the full round-trip for passing a JS string into a wasm function and getting a string back: allocate space in the module's linear memory, copy the UTF-8 bytes via a Uint8Array view, call the export with the pointer + length, then decode the returned pointer/length with TextDecoder. Note why growing memory can invalidate an old buffer view.
#1 Pitfall: Caching a typed-array view over memory.buffer and reusing it after the module calls memory.grow() — growth detaches the old ArrayBuffer, so the stale view is empty/throws. Re-create the view after any potential growth.
Advanced Debugging Performance & Security 8 days since review
The console is more than log: console.table(rows) renders arrays/objects as a grid, group/groupEnd nest related output, time/timeEnd measure a span, count tallies calls, assert(cond, ...) logs only on failure, and trace() prints a stack. In DevTools, prefer conditional/logpoint breakpoints over scattering console.log, and load source maps to debug original code.
JavaScript
console.table([{ id: 1, name: 'Ada' }, { id: 2, name: 'Lin' }]);

console.group('request');
console.log('url', '/api');
console.time('fetch');
// ...work...
console.timeEnd('fetch');            // fetch: 12.4ms
console.groupEnd();

console.assert(1 + 1 === 3, 'math broke');   // logs only because it's false
console.trace('who called me');              // prints call stack
Challenge:
Instrument a small data pipeline so it prints each stage's rows with console.table, wraps each stage in a group, times the slow stage with time/timeEnd, and uses console.assert to flag any row that violates an invariant — without changing the pipeline's return values. Then describe when a conditional breakpoint or logpoint beats all of this.
#1 Pitfall: Shipping console.log debugging to production (noise, minor perf cost, and leaking internal data), and logging a live object reference then reading it later — the console may show the mutated current state, not the value at log time. Snapshot with structuredClone or JSON if you need a frozen view.

🆕 Locale-Aware Formatting

Two Intl formatters that turn raw values into correct human text in any language — without hand-built string glue.

Intl.ListFormat Internationalization

Joining a list into prose — "apples, oranges, and pears" — is deceptively hard: the separator, the "and"/"or" word, and the Oxford comma all vary by language. Intl.ListFormat (ES2021) produces grammatically correct lists for any locale, replacing the arr.join(', ') + manual "and" hack that breaks the moment you localize.

1Pick a locale and a list type. 'conjunction' means "and", 'disjunction' means "or", and 'unit' joins measurements. The formatter inserts the right words and commas.

JavaScript
const items = ['apples', 'oranges', 'pears'];

const and = new Intl.ListFormat('en', { type: 'conjunction' });
console.log(and.format(items));        // 'apples, oranges, and pears'

const or = new Intl.ListFormat('en', { type: 'disjunction' });
console.log(or.format(items));         // 'apples, oranges, or pears'

// two items: no comma, just the connector word
console.log(and.format(['tea', 'coffee']));   // 'tea and coffee'
apples, oranges, and pears apples, oranges, or pears tea and coffee

2Style and locale change everything. style can be 'long' (default), 'short', or 'narrow'; switching the locale swaps separators and connector words automatically — no code changes.

JavaScript
const items = ['A', 'B', 'C'];

console.log(new Intl.ListFormat('es', { type: 'conjunction' }).format(items));
//   'A, B y C'          (Spanish uses 'y', no Oxford comma)

console.log(new Intl.ListFormat('de', { type: 'disjunction' }).format(items));
//   'A, B oder C'       (German 'oder')

console.log(new Intl.ListFormat('en', { style: 'narrow' }).format(items));
//   'A, B, C'           (narrow style drops the word)
A, B y C A, B oder C A, B, C

3formatToParts lets you style pieces. It returns an array of { type: 'element' | 'literal', value } segments — use it to wrap each element in a link or span while leaving the separators as plain text.

JavaScript
const lf = new Intl.ListFormat('en', { type: 'conjunction' });
const parts = lf.formatToParts(['Ada', 'Grace']);
console.log(parts);
// [ { type: 'element', value: 'Ada' },
//   { type: 'literal', value: ' and ' },
//   { type: 'element', value: 'Grace' } ]

const html = parts
  .map((p) => p.type === 'element' ? `<b>${p.value}</b>` : p.value)
  .join('');
console.log(html);        // '<b>Ada</b> and <b>Grace</b>'
[ { type: 'element', value: 'Ada' }, { type: 'literal', value: ' and ' }, { type: 'element', value: 'Grace' } ] <b>Ada</b> and <b>Grace</b>
Common Mistakes:
❌ Hand-building localized lists with join + "and": arr.join(', ') + ' and ' + last // wrong comma rules, English-only items.join(' or ') // no Oxford comma logic, no locale new Intl.ListFormat('en').format([1, 2, 3]) // TypeError: not strings
✅ Let Intl.ListFormat handle grammar, feed it strings: new Intl.ListFormat('en', { type: 'conjunction' }).format(arr) new Intl.ListFormat(navigator.language).format(arr) lf.format([1, 2, 3].map(String)) // coerce non-strings first
Key Rules:
  • type selects grammar: 'conjunction' (and), 'disjunction' (or), 'unit' (measurements); style is 'long'/'short'/'narrow'.
  • Inputs must be strings — non-strings throw a TypeError; coerce with .map(String) first.
  • Reuse a formatter instance (construction is relatively expensive) and use formatToParts when you need to style individual elements.
Practice Exercises:
Write humanList(arr, locale) that returns a conjunction-formatted string, and show it produces the correct output for ['a'], ['a','b'], and ['a','b','c'] in both 'en' and 'es'.
Use formatToParts to render a list of author names where each name becomes an <a> link but the separators stay as plain text, and confirm the literals are untouched.
Compare type: 'conjunction' vs 'disjunction' vs 'unit' on ['5 hours', '3 minutes'] and explain when 'unit' (which omits "and") is the right choice.
Connections: This joins the Intl family alongside NumberFormat, DateTimeFormat, and Intl.Segmenter, and pairs naturally with Intl.DurationFormat below when you need to list several durations.
Intl.DurationFormat Internationalization

Rendering "1 hr, 46 min, 40 sec" or "1:46:40" correctly in every language used to require a table of unit words and pluralization rules. Intl.DurationFormat (ES2025) formats a plain duration object — { hours, minutes, seconds } — into localized text or a digital clock string, closing a long-standing gap in the Intl suite.

1Pass a duration record; choose a style. The value is an object with any of years … nanoseconds. style selects verbosity: 'long', 'short', 'narrow', or 'digital' (a clock).

JavaScript
const d = { hours: 1, minutes: 46, seconds: 40 };

console.log(new Intl.DurationFormat('en', { style: 'long' }).format(d));
//   '1 hour, 46 minutes, 40 seconds'

console.log(new Intl.DurationFormat('en', { style: 'short' }).format(d));
//   '1 hr, 46 min, 40 sec'

console.log(new Intl.DurationFormat('en', { style: 'digital' }).format(d));
//   '1:46:40'
1 hour, 46 minutes, 40 seconds 1 hr, 46 min, 40 sec 1:46:40

2Locale drives words and pluralization. Switch the locale and the unit names, plural forms, and separators all adjust — the exact problem that made hand-rolled duration strings fragile.

JavaScript
const d = { hours: 2, minutes: 5 };

console.log(new Intl.DurationFormat('fr', { style: 'long' }).format(d));
//   '2 heures et 5 minutes'

console.log(new Intl.DurationFormat('ja', { style: 'long' }).format(d));
//   '2 時間 5 分'

// per-unit overrides: force 2-digit minutes in digital style
const df = new Intl.DurationFormat('en', {
  style: 'digital', minutesDisplay: 'always',
});
console.log(df.format({ hours: 3, minutes: 5 }));   // '3:05:00'
2 heures et 5 minutes 2 時間 5 分 3:05:00

3It does not normalize — you supply the fields. DurationFormat renders exactly the units you give it; it will not roll 90 minutes into 1h 30m. Compute/normalize the components yourself (or with Temporal.Duration) before formatting.

JavaScript
const fmt = new Intl.DurationFormat('en', { style: 'short' });

console.log(fmt.format({ minutes: 90 }));       // '90 min'  — NOT '1 hr 30 min'

// normalize seconds → h/m/s yourself before formatting:
function breakdown(totalSec) {
  return {
    hours: Math.floor(totalSec / 3600),
    minutes: Math.floor((totalSec % 3600) / 60),
    seconds: totalSec % 60,
  };
}
console.log(fmt.format(breakdown(5400)));        // '1 hr, 30 min, 0 sec'
90 min 1 hr, 30 min, 0 sec
Common Mistakes:
❌ Expecting normalization or passing a raw number: new Intl.DurationFormat('en').format(5400) // TypeError: needs an object fmt.format({ minutes: 90 }) // stays '90 min', no rollover fmt.format({ seconds: 40, minutes: 46, hours: 1 }) // order in object is fine, // but you still normalize values
✅ Normalize into fields, then format the object: fmt.format({ hours: 1, minutes: 46, seconds: 40 }) fmt.format(breakdown(totalSeconds)) // your own h/m/s split fmt.format(temporalDuration) // Temporal.Duration works too
Key Rules:
  • The argument is a duration record ({ hours, minutes, seconds, … } or a Temporal.Duration), never a bare number of milliseconds.
  • style picks verbosity: 'long'/'short'/'narrow' for prose, 'digital' for a h:mm:ss clock; per-unit *Display options fine-tune which fields show.
  • It formats but does not normalize — { minutes: 90 } renders as 90 minutes; roll units up yourself (or via Temporal.Duration.round) first.
Practice Exercises:
Write formatMs(ms, locale) that converts milliseconds into { hours, minutes, seconds } and renders them with style: 'long'; verify 3_725_000 ms reads as "1 hour, 2 minutes, 5 seconds".
Render the same duration in 'long', 'short', and 'digital' styles for both 'en' and 'fr', and note where the digital style ignores the locale's unit words.
Demonstrate the no-normalization rule: format { minutes: 125 } directly, then normalize it to hours+minutes and format again, showing both outputs side by side.
Connections: Intl.DurationFormat is the display layer for a Temporal.Duration (which does the arithmetic and rounding), and it rounds out the Intl family with Intl.ListFormat, NumberFormat, and RelativeTimeFormat.

🆕 Patterns & Binary Precision

The modern regex flag that adds set algebra to character classes, and a compact 16-bit float type for graphics and ML workloads.

RegExp v Flag (Unicode Sets) Data & Collections

The u flag gave regex proper Unicode, but character classes stayed flat — you couldn't say "letters that are also ASCII" or match a multi-code-point emoji as one unit. The ES2024 v flag (unicodeSets) upgrades character classes with set operations (intersection, subtraction) and string literals, making complex Unicode matching readable.

1Set intersection && and subtraction --. Inside [...] under v, combine \p{...} property escapes: && keeps only members in both sets, -- removes members of the right set.

JavaScript
// intersection: a letter AND ASCII
const asciiLetter = /[\p{Letter}&&\p{ASCII}]/v;
console.log(asciiLetter.test('a'));    // true
console.log(asciiLetter.test('é'));    // false — a letter, but not ASCII

// subtraction: decimal digits that are NOT ASCII 0-9
const nonAsciiDigit = /[\p{Decimal_Number}--[0-9]]/v;
console.log(nonAsciiDigit.test('5'));   // false
console.log(nonAsciiDigit.test('٥'));   // true — Arabic-Indic five
true false false true

2Match multi-character strings inside a class with \q{…}. Classic classes match single code points only. Under v, [\q{…|…}] lets a class alternate over whole strings — perfect for emoji sequences (ZWJ-joined) that span many code points.

JavaScript
// match either the family emoji (a ZWJ sequence) or a plain letter
const re = /[\q{👨‍👩‍👧}a]/v;
console.log(re.test('👨‍👩‍👧'));      // true — whole sequence as one class member
console.log(re.test('a'));            // true
console.log('👨‍👩‍👧a'.match(/[\q{👨‍👩‍👧}a]/gv).length);   // 2
true true 2

3v is stricter and mostly a superset of u. It forbids some ambiguous syntax (certain literal [, (, { must be escaped) and enables case-insensitive matching that respects the set semantics. You cannot combine u and v on the same regex.

JavaScript
// nested/unioned classes read clearly under v
const vowelOrDigit = /[[aeiou][0-9]]/v;      // union of two classes
console.log('e'.match(vowelOrDigit)?.[0]);   // 'e'
console.log('7'.match(vowelOrDigit)?.[0]);   // '7'

// try { new RegExp('[a-z]', 'uv'); } catch (e) { ... }
//   SyntaxError: Invalid flags — u and v are mutually exclusive
e 7
Common Mistakes:
❌ Using u-flag classes for set logic, or mixing flags: /[\p{Letter}&&\p{ASCII}]/u // '&&' is literal under u, not intersection /[\q{ab}]/u // \q is a syntax error under u new RegExp('...', 'uv') // SyntaxError: can't combine u and v
✅ Use the v flag for set notation and string classes: /[\p{Letter}&&\p{ASCII}]/v // real intersection /[\p{Decimal_Number}--[0-9]]/v // real subtraction /[\q{ab|cd}]/v // multi-char string members
Key Rules:
  • The v (unicodeSets) flag enables in-class set operators: && (intersection), -- (subtraction), nested [...] unions.
  • \q{a|bc} lets a class match whole strings, not just single code points — essential for ZWJ emoji and other grapheme sequences.
  • v is stricter than u (more escaping required) and is mutually exclusive with u — pick one.
Practice Exercises:
Write a regex under v that matches a character which is a letter but not ASCII (i.e. accented/non-Latin letters), and test it against 'a', 'é', and 'あ'.
Use \q{…} to build a class that matches any of three country-flag emoji as single units, and count how many appear in a mixed string with matchAll under the g and v flags.
Demonstrate that /[\p{Emoji}--\p{ASCII}]/v excludes ASCII characters that carry the Emoji property (like # and digits) while still matching a real emoji.
Connections: The v flag builds on the Unicode property escapes from Regular Expressions Advanced, and its grapheme-aware string matching complements Intl.Segmenter and the well-formedness concerns of Unicode strings.
Float16Array & Math.f16round Data & Collections

Half-precision (16-bit) floats halve the memory and bandwidth of Float32Array — the native format of GPU textures, WebGPU buffers, and many machine-learning tensors. ES2025 adds Float16Array, Math.f16round, and DataView float16 accessors so JS can produce and consume this data without manual bit-twiddling.

1A typed array of 2-byte floats. Float16Array behaves like the other typed arrays but stores each element in 16 bits, so values are rounded to the nearest representable half-float on write.

JavaScript
const a = new Float16Array([1.5, 2.25, 0.1]);
console.log(a.BYTES_PER_ELEMENT);   // 2  (vs 4 for Float32Array)
console.log(a[0], a[1]);            // 1.5 2.25  — exactly representable
console.log(a[2]);                  // 0.0999755859375 — 0.1 rounded to nearest half

console.log(a.length, a.byteLength);   // 3 6
2 1.5 2.25 0.0999755859375 3 6

2Math.f16round(x) previews the rounding. Just as Math.fround rounds to float32, Math.f16round returns the nearest half-precision value — useful to check precision loss before committing data to a Float16Array.

JavaScript
console.log(Math.f16round(1.1));    // 1.099609375
console.log(Math.f16round(0.1));    // 0.0999755859375
console.log(Math.f16round(1.5));    // 1.5 — exact

// values beyond ~65504 overflow to Infinity in float16:
console.log(Math.f16round(70000));  // Infinity
console.log(new Float16Array([70000])[0]);   // Infinity
1.099609375 0.0999755859375 1.5 Infinity Infinity

3Read/write half-floats in binary buffers via DataView. getFloat16/setFloat16 (with an endianness flag) let you pack and parse half-precision fields in binary protocols and GPU-bound ArrayBuffers.

JavaScript
const buf = new ArrayBuffer(2);
const dv = new DataView(buf);
dv.setFloat16(0, 3.14, /* littleEndian */ true);
console.log(dv.getFloat16(0, true));   // 3.140625 — nearest half of pi

// share the same bytes through a Float16Array view:
console.log(new Float16Array(buf)[0]); // 3.140625
3.140625 3.140625
Common Mistakes:
❌ Assuming full precision/range, or comparing with ===: new Float16Array([0.1])[0] === 0.1 // false — 0.1 was rounded new Float16Array([100000])[0] // Infinity — exceeds ~65504 max // ~3 significant decimal digits only — not for money or precise math
✅ Expect rounding; use half only where it fits: Math.f16round(x) // preview the stored value Math.abs(a[0] - 0.1) < 1e-3 // tolerance-based compare // reserve Float16 for GPU/ML/graphics; use Float64 for precise work
Key Rules:
  • Float16Array stores 2-byte floats (~3 significant decimal digits, max finite ≈ 65504); writes round to the nearest half and can overflow to Infinity.
  • Math.f16round(x) returns the nearest float16 value (the analog of Math.fround for float32) — use it to inspect precision loss.
  • DataView.getFloat16/setFloat16 read/write half-floats with an endianness flag; reach for float16 only in memory/bandwidth-bound graphics and ML work, never for exact or financial math.
Practice Exercises:
Fill a Float16Array and a Float32Array with [0.1, 0.2, 0.3] and print each element, quantifying how much more precision the 32-bit version retains.
Write fitsInFloat16(x) that returns whether a number survives a round-trip through Math.f16round within a small tolerance, and test it on 0.5, 0.1, and 70000.
Pack three half-precision values into a 6-byte ArrayBuffer with DataView.setFloat16, then read them back with a Float16Array view and confirm the bytes are shared.
Connections: Float16Array joins the typed-array / ArrayBuffer / DataView family you studied, and its rounding behavior mirrors the IEEE-754 precision issues behind Object.is(-0, +0) and floating-point equality. It is the data format for WebGPU and WebAssembly numeric pipelines.

🆕 Robust Error Checks

A small ES2025 helper that finally makes "is this really an Error?" reliable across realms.

Error.isError Core Language

err instanceof Error lies in two common situations: an error thrown from another realm (iframe, worker, Node vm) has a different Error constructor and fails the check, while a plain object like { name, message, stack } can masquerade as one. Error.isError(x) (ES2025) inspects an internal slot instead of the prototype chain, so it answers correctly in both cases — the error-world analog of Array.isArray.

1It brands genuine errors, ignoring the prototype chain. Real Error objects (including all subclasses) return true; look-alike plain objects return false.

JavaScript
console.log(Error.isError(new Error('boom')));       // true
console.log(Error.isError(new TypeError('bad')));    // true — subclass
console.log(Error.isError(new DOMException('x')));   // true

console.log(Error.isError({ name: 'Error', message: 'fake', stack: '' })); // false
console.log(Error.isError('an error string'));       // false
console.log(Error.isError(null));                    // false
true true true false false false

2Why instanceof is not enough: realms. Each realm (iframe, worker) has its own Error constructor. An error crossing that boundary is a real error but fails instanceof against your realm's Error. Error.isError checks the internal brand, so it still returns true.

JavaScript
// Conceptual cross-realm example (e.g. an iframe's window):
const iframeError = otherRealm.eval('new Error("from iframe")');

console.log(iframeError instanceof Error);   // false — different Error constructor
console.log(Error.isError(iframeError));     // true  — brand check crosses realms

// same failure mode you already know from arrays:
// [] instanceof otherRealm.Array === false, which is why Array.isArray exists
false true

3Use it at boundaries in catch. Because catch (e) can receive anything (a thrown string, number, or object), gate error-specific handling on Error.isError(e) before reading .message/.stack, then fall back for non-errors.

JavaScript
function describe(e) {
  if (Error.isError(e)) return `${e.name}: ${e.message}`;
  return `non-error thrown: ${String(e)}`;
}

try { throw new RangeError('too big'); }
catch (e) { console.log(describe(e)); }        // 'RangeError: too big'

try { throw 'just a string'; }
catch (e) { console.log(describe(e)); }        // 'non-error thrown: just a string'
RangeError: too big non-error thrown: just a string
Common Mistakes:
❌ Relying on instanceof or duck-typing across boundaries: e instanceof Error // false for cross-realm / cross-worker errors 'stack' in e // true for fakes; false for some real errors e.name === 'TypeError' // trivially spoofable, and misses subclasses
✅ Brand-check with Error.isError, then narrow by type: if (Error.isError(e)) { /* safe to read .message/.stack */ } if (Error.isError(e) && e instanceof MyAppError) { ... } // same-realm subtype else { /* handle non-error throw */ }
Key Rules:
  • Error.isError(x) returns true only for genuine Error instances (and subclasses, and DOMException) by checking an internal slot — not the prototype chain.
  • Prefer it over instanceof Error when values may cross realms (iframes, workers, vm) or come from untrusted structured data.
  • It only answers "is this an error object" — to distinguish which error within your own realm, still use instanceof on your custom subclasses.
Practice Exercises:
Write safeMessage(x) that returns x.message when Error.isError(x) is true and String(x) otherwise, and test it with an Error, a string, and a { message } plain object.
Contrast Error.isError with instanceof Error on a fake { name: 'TypeError', message: 'x', stack: '...' } object, and explain which one the fake fools.
Build a catch boundary that logs the .stack for real errors, wraps non-error throws (strings/numbers) into a new Error(String(thrown)), and uses Error.isError to decide which path to take.
Connections: Error.isError mirrors Array.isArray's cross-realm brand check and strengthens the Error Handling Patterns and Error cause/AggregateError concepts — it is the reliable guard to run before inspecting .cause or matching custom error subclasses.
