Mocking: Interfaces for Testability

Testing & Tooling • overdue • rep 1

Quick Reminder

Go ships no mocking framework and rarely needs one. You make code testable by having it depend on a small interface rather than a concrete type, then in tests you pass a hand-written fake (stub/spy) that implements that interface. The guiding idiom is accept interfaces, return structs: define the interface where it is consumed, keep it narrow (one or two methods), and the real and test implementations both satisfy it.

Key Pattern

package main

import "fmt"

// Consumer defines a narrow interface it depends on.
type Notifier interface {
    Send(to, msg string) error
}

type Service struct{ n Notifier }

func (s Service) Alert(user string) error {
    return s.n.Send(user, "alert!")
}

// Test double: a spy that records calls and returns a canned result.
type fakeNotifier struct {
    sent []string
    err  error
}

func (f *fakeNotifier) Send(to, msg string) error {
    f.sent = append(f.sent, to+":"+msg)
    return f.err
}

func main() {
    f := &fakeNotifier{}
    _ = Service{n: f}.Alert("ada")
    fmt.Println(f.sent) // [ada:alert!]
}
Output
[ada:alert!]
Challenge Exercise

Write a Charger that depends on a PaymentGateway interface with one method Charge(cents int) (txID string, err error). Provide a fake that can be scripted to succeed or fail, and assert both the returned id and that exactly one call was made. Then explain why defining the interface in the consumer package (not the implementation package) keeps dependencies pointing inward, and why a one-method interface is easier to fake than a fat one.

Pitfall: Defining a huge interface next to the concrete implementation and re-exporting it everywhere. Fat interfaces force your fakes to stub dozens of unused methods and couple every consumer to methods it never calls. Declare the smallest interface each consumer actually needs, on the consumer side.

go vet & staticcheck

Testing & Tooling • overdue • rep 1

Quick Reminder

go vet is the compiler's suspicious-code detector: it flags constructs that compile but are almost certainly bugs — Printf format/argument mismatches, struct tags with bad syntax, unreachable code, copying a value that contains a sync.Mutex, and loop-variable capture. It ships with the toolchain and runs automatically as part of go test. staticcheck (a third-party linter) adds hundreds more checks and is the de-facto standard in CI.

Key Pattern

# vet the whole module; also runs during `go test`
go vet ./...

# A classic catch: format verb doesn't match the argument type
#   fmt.Printf("%d\n", "hello")
# vet reports:
#   Printf format %d has arg "hello" of wrong type string

# staticcheck: install once, then run across packages
go install honnef.co/go/tools/cmd/staticcheck@latest
staticcheck ./...
#   e.g. SA4006: value assigned to x is never read
#        S1005: unnecessary assignment to the blank identifier
Challenge Exercise

Write a tiny program with three latent bugs vet catches: a Printf("%d", someString), a struct literal copied while it embeds a sync.Mutex, and an unreachable return. Run go vet and read each diagnostic. Then wire go vet ./... and staticcheck ./... into a CI step and explain why they are complementary rather than redundant, and why vet is deliberately conservative (few false positives) while staticcheck is broader.

Pitfall: Assuming "it compiled, so it's fine." The compiler accepts fmt.Printf("%d", name), a self-assignment, or a mutex copied by value — all real bugs that only vet/staticcheck surface. Treat their output as build-breaking in CI, not advisory.

Builder Pattern

Advanced Patterns • overdue • rep 1

Quick Reminder

The builder assembles a complex value step by step through chained methods that each return the receiver, finishing with a terminal Build(). It shines when a struct has many optional fields or requires validation before construction. In Go it competes with the functional-options pattern; builders are fluent and stateful, options are variadic and composable — pick one per API.

Key Pattern

package main

import "fmt"

type Request struct {
    method, url string
    headers     map[string]string
}

type Builder struct{ r Request }

func NewBuilder(url string) *Builder {
    return &Builder{r: Request{method: "GET", url: url, headers: map[string]string{}}}
}
func (b *Builder) Method(m string) *Builder    { b.r.method = m; return b }
func (b *Builder) Header(k, v string) *Builder { b.r.headers[k] = v; return b }
func (b *Builder) Build() Request              { return b.r }

func main() {
    req := NewBuilder("/api").
        Method("POST").
        Header("Accept", "json").
        Build()
    fmt.Println(req.method, req.url, req.headers) // POST /api map[Accept:json]
}
Output
POST /api map[Accept:json]
Challenge Exercise

Build a QueryBuilder that composes a SQL-ish string with chained Select, From, Where methods and a Build() (string, error) that fails if From was never called. Return the error from the terminal method rather than panicking mid-chain. Then compare it against a functional-options constructor for the same type and explain which reads better for a required-vs-optional field split.

Pitfall: Returning the builder by value from each step instead of a pointer. With a value receiver, each chained call mutates a copy and the accumulated state is lost. Use a pointer receiver (*Builder) so the chain threads one instance — or fully embrace value semantics and reassign at every step.

Code Generation

Advanced Patterns • overdue • rep 1

Quick Reminder

Go leans on generated code instead of runtime reflection for repetitive boilerplate: enum stringers, mocks, protobuf bindings, marshalers. A //go:generate comment records the command; running go generate ./... executes every such directive. The generated .go file is committed to the repo so builds stay reflection-free and hermetic — go generate is never run automatically by go build.

Key Pattern

package main

import "fmt"

//go:generate stringer -type=Pill
type Pill int

const (
    Placebo Pill = iota
    Aspirin
    Ibuprofen
)

// Running `go generate ./...` invokes stringer, which writes
// pill_string.go containing:
//   func (p Pill) String() string { ... }

func main() {
    fmt.Println(Aspirin) // "Aspirin" after generation (else "1")
}
Challenge Exercise

Add a //go:generate stringer -type=Direction directive to a Direction enum, run go generate ./..., and inspect the produced _string.go file. Then write your own trivial generator: a small program that reads a list of type names from a flag and emits a .go file with a header comment // Code generated by ...; DO NOT EDIT.. Explain why generated files carry that marker and why you commit them rather than generating at build time.

Pitfall: Expecting go build or go test to run your //go:generate directives. They never do — generation is a manual step (go generate ./...), usually enforced by a CI check that regenerates and fails if the working tree changed. Forget it and your committed generated code silently drifts from its source.

Repository Pattern

Advanced Patterns • overdue • rep 1

Quick Reminder

A repository hides where and how data is persisted behind a domain-shaped interface (Get, Save, List), so business logic depends on the abstraction, not on SQL, HTTP, or a specific driver. You get a swappable backend: a real Postgres implementation in production, an in-memory map in tests. It is a specific application of "accept interfaces" to the persistence boundary.

Key Pattern

package main

import (
    "errors"
    "fmt"
)

type User struct{ ID, Name string }

type UserRepo interface {
    Get(id string) (User, error)
    Save(u User) error
}

// In-memory implementation — ideal for fast, hermetic tests.
type memRepo struct{ m map[string]User }

func newMemRepo() *memRepo { return &memRepo{m: map[string]User{}} }

func (r *memRepo) Get(id string) (User, error) {
    u, ok := r.m[id]
    if !ok {
        return User{}, errors.New("not found")
    }
    return u, nil
}
func (r *memRepo) Save(u User) error { r.m[u.ID] = u; return nil }

func main() {
    var repo UserRepo = newMemRepo()
    _ = repo.Save(User{ID: "1", Name: "Ada"})
    u, _ := repo.Get("1")
    fmt.Println(u.Name) // Ada
}
Output
Ada
Challenge Exercise

Define a ProductRepo interface and give it two implementations: an in-memory map and one backed by database/sql (sketch the queries). Write a service that takes the interface and add a sentinel ErrNotFound that both repos return, so callers use errors.Is regardless of backend. Explain why leaking a driver-specific error (like sql.ErrNoRows) through the interface breaks the abstraction.

Pitfall: Letting storage details bleed through the interface — returning *sql.Rows, exposing a *gorm.DB, or surfacing sql.ErrNoRows directly. The whole point is that callers can't tell which backend they're on. Translate driver errors into domain sentinels and return plain domain types.

unsafe Package

Advanced Patterns • overdue • rep 1

Quick Reminder

unsafe steps outside Go's type and memory safety. unsafe.Pointer converts between arbitrary pointer types; Sizeof, Offsetof, and Alignof introspect memory layout; and (Go 1.17+/1.20+) unsafe.Slice/unsafe.String build slices and strings from a pointer and length for zero-copy conversions. Reach for it only for FFI, layout tricks, or hot-path conversions — it voids portability and can defeat the garbage collector if misused.

Key Pattern

package main

import (
    "fmt"
    "unsafe"
)

type Point struct {
    X int32
    Y int32
}

func main() {
    var p Point
    fmt.Println(unsafe.Sizeof(p))     // 8  (two int32s)
    fmt.Println(unsafe.Offsetof(p.Y)) // 4  (bytes into the struct)
    fmt.Println(unsafe.Alignof(p))    // 4

    // Zero-copy []byte -> string without allocating (Go 1.20+).
    b := []byte("hi")
    s := unsafe.String(&b[0], len(b))
    fmt.Println(s) // hi
}
Output
8 4 4 hi
Challenge Exercise

Use unsafe.Sizeof/Offsetof to observe struct padding: compare struct{ a bool; b int64; c bool } against a reordered struct{ b int64; a, c bool } and explain the size difference. Then write a benchmarked zero-copy []byte↔string conversion with unsafe.String/unsafe.Slice, and describe the invariant you must uphold (the backing bytes must not be mutated afterward) and why violating it corrupts an immutable string.

Pitfall: Storing the result of uintptr(unsafe.Pointer(&x)) in a variable and dereferencing it later. A uintptr is just a number — the GC doesn't track it, so it can move or collect x out from under you. Conversions through uintptr must happen in a single expression; never park a pointer's address in a uintptr across statements.

Section 1 — Networking, Filesystems & Bits

net/netip: Modern IP Address Types

Std Lib

Why it matters: the old net.IP is a []byte slice — it allocates, is not comparable (can't be a map key), and carries no notion of IPv4-vs-IPv6 cleanly. net/netip (Go 1.18) introduces small, immutable, comparable value types: Addr, AddrPort, and Prefix. They fit in a register-friendly struct, work as map keys, and make CIDR containment a one-liner. Prefer them for any new address handling.

Parsing addresses and ports

Addr is a value you compare with == and use directly as a map key. MustParseAddr/MustParseAddrPort panic on bad input (great for constants and tests); use the ParseAddr form to get an error at runtime.

package main

import (
    "fmt"
    "net/netip"
)

func main() {
    a := netip.MustParseAddr("192.168.0.42")
    fmt.Println(a.Is4())    // true
    fmt.Println(a.Is6())    // false
    fmt.Println(a.String()) // 192.168.0.42

    ap := netip.MustParseAddrPort("[2001:db8::1]:443")
    fmt.Println(ap.Addr(), ap.Port()) // 2001:db8::1 443

    // Addr is comparable -> usable as a map key with no allocation.
    seen := map[netip.Addr]int{}
    seen[a]++
    fmt.Println(seen[a]) // 1
}
Output
true false 192.168.0.42 2001:db8::1 443 1

CIDR prefixes and containment

Prefix models a network like 10.0.0.0/8. Contains tests membership, and Masked zeroes the host bits — the canonical form of the network.

package main

import (
    "fmt"
    "net/netip"
)

func main() {
    pfx := netip.MustParsePrefix("10.0.0.0/8")
    fmt.Println(pfx.Contains(netip.MustParseAddr("10.1.2.3"))) // true
    fmt.Println(pfx.Contains(netip.MustParseAddr("11.0.0.1"))) // false
    fmt.Println(pfx.Masked())                                  // 10.0.0.0/8
}
Output
true false 10.0.0.0/8

Common Mistakes

Wrong
// net.IP is a []byte: not comparable, allocates,
// and can't be a map key.
ip := net.ParseIP("10.0.0.1")
seen := map[string]bool{}
seen[ip.String()] = true // stringify just to key it
Right
// netip.Addr is a comparable value type.
ip := netip.MustParseAddr("10.0.0.1")
seen := map[netip.Addr]bool{}
seen[ip] = true // direct, allocation-free key

Key Rules

  • Addr, AddrPort, and Prefix are small immutable values — compare with ==, use as map keys, and pass by value freely.
  • Use ParseAddr/ParsePrefix for untrusted input (returns an error); the MustParse* variants panic and belong in constants/tests only.
  • Prefix.Contains answers CIDR membership; Masked() normalizes to the network address; Addr.Is4()/Is6()/Unmap() handle the family.
  • Bridge to the older API when needed: netip.AddrFromSlice, Addr.AsSlice(), and net.TCPAddrFromAddrPort convert to/from net.IP/net.TCPAddr.
Practice Exercise

Build an allowlist checker: parse a set of CIDR Prefixes once, then classify a stream of client Addrs as allowed/denied with Contains. Store per-address hit counts in a map[netip.Addr]int to prove the type keys directly. Then convert an incoming *net.TCPAddr (from a real listener) into a netip.AddrPort and back, and explain why netip's value semantics reduce allocations on a hot connection-accept path compared with net.IP.

Pitfall: Comparing a netip.Addr parsed from an IPv4 literal against one that came through an IPv4-in-IPv6 mapping (::ffff:10.0.0.1). They are not == even though they denote the same host. Normalize with Addr.Unmap() before comparing or using as a key when both forms can occur.
Connections: The modern counterpart to the net types behind net/http and TCP servers; its comparable value semantics echo the "small immutable value" design of time.Time, and it pairs with containers and maps as first-class keys.

io/fs: The Filesystem Abstraction

Std Lib

Why it matters: io/fs (Go 1.16) defines fs.FS — a read-only filesystem interface — so code that reads files no longer hard-depends on the real disk. The same function works over the OS (os.DirFS), an embedded bundle (//go:embed), a zip archive, or an in-memory fstest.MapFS for tests. Helpers like fs.ReadFile, fs.WalkDir, fs.Glob, and fs.Sub operate on any fs.FS.

Walk and read any fs.FS

Write your logic against fs.FS; supply an in-memory fstest.MapFS in tests and a real os.DirFS("/some/dir") in production. fs.WalkDir visits entries in lexical order.

package main

import (
    "fmt"
    "io/fs"
    "testing/fstest"
)

func main() {
    fsys := fstest.MapFS{
        "README.md":  {Data: []byte("hi")},
        "docs/a.txt": {Data: []byte("alpha")},
        "docs/b.txt": {Data: []byte("bravo")},
    }

    data, _ := fs.ReadFile(fsys, "docs/a.txt")
    fmt.Println(string(data)) // alpha

    fs.WalkDir(fsys, ".", func(p string, d fs.DirEntry, err error) error {
        if !d.IsDir() {
            fmt.Println(p)
        }
        return nil
    })
}
Output
alpha README.md docs/a.txt docs/b.txt

Glob and sub-filesystems

fs.Glob matches patterns (sorted results); fs.Sub returns an fs.FS rooted at a subdirectory, so callers see a trimmed namespace.

package main

import (
    "fmt"
    "io/fs"
    "testing/fstest"
)

func main() {
    fsys := fstest.MapFS{
        "img/a.png": {Data: []byte("x")},
        "img/b.png": {Data: []byte("y")},
        "img/c.txt": {Data: []byte("z")},
    }

    pngs, _ := fs.Glob(fsys, "img/*.png")
    fmt.Println(pngs) // [img/a.png img/b.png]

    sub, _ := fs.Sub(fsys, "img") // root at img/
    entries, _ := fs.ReadDir(sub, ".")
    for _, e := range entries {
        fmt.Println(e.Name())
    }
}
Output
[img/a.png img/b.png] a.png b.png c.txt

Common Mistakes

Wrong
// Hard-wired to the OS disk -> untestable without
// real files, and paths use OS separators.
func Load(dir string) ([]byte, error) {
    return os.ReadFile(dir + "/config.yaml")
}
Right
// Depends on the fs.FS interface; test with fstest.MapFS,
// run against os.DirFS or an embed.FS in production.
func Load(fsys fs.FS) ([]byte, error) {
    return fs.ReadFile(fsys, "config.yaml")
}

Key Rules

  • Accept an fs.FS parameter instead of a directory string — the same code then serves OS dirs (os.DirFS), embeds (embed.FS), zips, and test doubles.
  • Paths in fs.FS are always slash-separated, unrooted, and relative to the FS root — no leading /, no .., no OS-specific separators.
  • Use the package helpers fs.ReadFile, fs.WalkDir, fs.Glob, fs.ReadDir, and fs.Sub; a minimal fs.FS only needs one Open method.
  • fstest.MapFS is a ready-made in-memory filesystem for tests, and fstest.TestFS validates that a custom implementation obeys the contract.
Practice Exercise

Write CountLines(fsys fs.FS, glob string) (int, error) that walks all files matching a glob and sums their line counts, then test it entirely with an fstest.MapFS — no disk touched. Run the same function against os.DirFS(".") to prove one implementation serves both. Finally, embed a directory with //go:embed assets/* and pass the embed.FS to the same function, explaining why slash-only, unrooted paths make the abstraction portable across all three backends.

Pitfall: Passing OS-style or rooted paths to an fs.FS — fs.ReadFile(fsys, "/etc/x") or fs.ReadFile(fsys, "dir\\file"). The fs contract requires clean, slash-separated, relative paths; a leading slash or backslash yields fs.ErrInvalid (or simply "file does not exist"). Convert with filepath.ToSlash and strip any root before handing a path to fs.
Connections: The interface behind //go:embed's embed.FS, http.FS for serving static assets, and os.DirFS; its "accept the interface" design mirrors the repository pattern and interface-based testing from today's review.

math/bits: Bit-Level Primitives

Std Lib

Why it matters: math/bits exposes the CPU's bit-twiddling instructions as portable, inlinable functions — population count, leading/trailing zeros, rotations, reversals, and overflow-aware add/subtract/multiply/divide. These power hash tables, bitsets, compression, and fixed-point math, and the compiler lowers most of them to a single machine instruction. Reach for it instead of hand-rolled loops that the optimizer can't recognize.

Counting and locating bits

OnesCount is population count; Len gives the index of the highest set bit (so it doubles as an integer log2); LeadingZeros/TrailingZeros come in sized variants (8/16/32/64).

package main

import (
    "fmt"
    "math/bits"
)

func main() {
    x := uint(0b1011_0100) // 180

    fmt.Println(bits.OnesCount(x))               // 4  set bits
    fmt.Println(bits.Len(x))                     // 8  high bit position
    fmt.Println(bits.LeadingZeros8(0b0001_0000)) // 3  in an 8-bit value
    fmt.Println(bits.TrailingZeros(x))           // 2  low zero run
    fmt.Println(bits.Reverse8(0b0000_0001))      // 128 bit-reversed
}
Output
4 8 3 2 128

Rounding to powers of two and wide multiply

Len gives a branch-free "round up to the next power of two"; Mul64 returns the full 128-bit product as (hi, lo), so you never silently lose the overflow.

package main

import (
    "fmt"
    "math/bits"
)

func main() {
    nextPow2 := func(n uint) uint {
        if n <= 1 {
            return 1
        }
        return 1 << bits.Len(n-1)
    }
    fmt.Println(nextPow2(17)) // 32
    fmt.Println(nextPow2(16)) // 16

    // Full 64x64 -> 128-bit product; hi holds the overflow.
    hi, lo := bits.Mul64(1<<40, 1<<40) // 2^80
    fmt.Println(hi, lo)               // 65536 0
}
Output
32 16 65536 0

Common Mistakes

Wrong
// Hand-rolled popcount the optimizer won't
// turn into a POPCNT instruction.
count := 0
for x != 0 {
    count += int(x & 1)
    x >>= 1
}
Right
// One intrinsic, lowered to a single CPU instruction.
count := bits.OnesCount64(x)

Key Rules

  • Functions come in width-specific flavors (OnesCount8/16/32/64) plus a word-width default (OnesCount) that matches uint.
  • Len(x) is the bit length (index of the top set bit, 0 for 0) — a cheap floor(log2)+1; Len(x-1) rounds up to a power of two.
  • Add64/Sub64/Mul64/Div64 expose carry/borrow and the full wide result for big-integer and checksum code without overflow loss.
  • Everything operates on unsigned integers; convert signed values explicitly and mind the width you pass to sized variants.
Practice Exercise

Implement a compact bitset over []uint64 with Set, Clear, Test, and a Count() that sums bits.OnesCount64 across words. Add a Next(i) that finds the next set bit at or after index i using bits.TrailingZeros64 on the masked word. Then write a power-of-two allocator size class using bits.Len, and benchmark your bitset's Count against a naive per-bit loop to see the intrinsic win.

Pitfall: Passing a value wider than the sized function expects — e.g. bits.LeadingZeros8(x) where x exceeds 8 bits, or assuming bits.Len counts a fixed 64. LeadingZeros8 only considers the low 8 bits, and Len returns a variable count (0 for 0). Match the function width to your data, and remember these operate on unsigned integers — a negative int won't behave as you expect until converted.
Connections: The low-level engine under hash bucketing (FNV/CRC-style sharding), container/heap sizing, and math/big's wide arithmetic; its intrinsics are the kind of hot-path win you verify with the benchmarking and escape-analysis tooling from earlier days.

Section 2 — Encoding & Data Structures

encoding/xml: Reading & Writing XML

Std Lib

Why it matters: XML still runs the enterprise world — SOAP, RSS/Atom feeds, config formats, SAML, sitemaps, office document parts. encoding/xml mirrors the familiar encoding/json model: struct tags map fields to elements and attributes, Marshal/Unmarshal handle whole documents, and a streaming Decoder processes huge inputs token by token. The tag syntax adds an XML-specific twist for attributes and nested paths.

Marshaling with element and attribute tags

An xml:"...,attr" tag makes a field an attribute; xml:"parent>child" nests it under an intermediate element; XMLName sets the root element name.

package main

import (
    "encoding/xml"
    "fmt"
)

type Book struct {
    XMLName xml.Name `xml:"book"`
    ID      int      `xml:"id,attr"`
    Title   string   `xml:"title"`
    Authors []string `xml:"authors>author"`
}

func main() {
    b := Book{ID: 7, Title: "Go in Action",
        Authors: []string{"Kennedy", "Ketelsen"}}
    out, _ := xml.MarshalIndent(b, "", "  ")
    fmt.Println(string(out))
}
Output
<book id="7"> <title>Go in Action</title> <authors> <author>Kennedy</author> <author>Ketelsen</author> </authors> </book>

Unmarshaling back into a struct

xml.Unmarshal fills exported fields by matching tags; attributes and nested paths round-trip symmetrically with marshaling.

package main

import (
    "encoding/xml"
    "fmt"
)

type Config struct {
    Env   string   `xml:"env,attr"`
    Hosts []string `xml:"hosts>host"`
}

func main() {
    data := `<config env="prod">
  <hosts>
    <host>a.example.com</host>
    <host>b.example.com</host>
  </hosts>
</config>`

    var c Config
    if err := xml.Unmarshal([]byte(data), &c); err != nil {
        panic(err)
    }
    fmt.Println(c.Env)   // prod
    fmt.Println(c.Hosts) // [a.example.com b.example.com]
}
Output
prod [a.example.com b.example.com]

Common Mistakes

Wrong
// json tags are ignored by encoding/xml, and
// unexported fields are never (un)marshaled.
type T struct {
    id    int    `json:"id"`   // lowercase + wrong tag
    Title string `json:"title"`
}
Right
type T struct {
    ID    int    `xml:"id,attr"`
    Title string `xml:"title"`
}

Key Rules

  • Only exported fields are processed; use xml:"name" for an element, xml:"name,attr" for an attribute, and xml:",chardata" for the element's text content.
  • xml:"a>b>c" maps a field through nested elements; XMLName xml.Name pins the wrapping element's name (and namespace).
  • Use xml.MarshalIndent for readable output and a streaming xml.NewDecoder(r).Token() loop for large documents you can't hold in memory.
  • XML has no native map type and no unmarshal-into-any; model documents as structs (or handle raw tokens), and add ,omitempty to skip zero-valued fields on marshal.
Practice Exercise

Model a minimal RSS feed (channel with a title and a slice of items) and round-trip it: unmarshal a sample document, then marshal it back with MarshalIndent and diff. Add an attribute and a ,chardata field to capture mixed content. Then rewrite the reader as a streaming Decoder that calls DecodeElement only on <item> start tokens, and explain why streaming matters for a multi-gigabyte feed where Unmarshal would exhaust memory.

Pitfall: Expecting encoding/xml to honor json tags or to unmarshal arbitrary XML into a map[string]any like JSON does. It reads only xml: tags, ignores unexported fields silently, and has no generic map target — unmatched elements are simply skipped with no error. Define a struct that mirrors the document, and reach for a streaming decoder when the shape is irregular.
Connections: Same tag-driven marshal/unmarshal model as encoding/json and encoding/gob, and the streaming Decoder mirrors the csv.Reader/json.Decoder token loops; the struct-tag reflection underneath is the same machinery vet checks for malformed tags.

container/list: Doubly Linked Lists

Std Lib

Why it matters: container/list is Go's ready-made doubly linked list — O(1) insert and remove at any position given the element, which slices can't do without shifting. It's the classic backbone of an LRU cache (move-to-front on use, evict from the back) and any queue where you splice elements out of the middle. It predates generics, so values are stored as any.

Building and traversing a list

list.New() returns an empty *List. PushBack/PushFront return the inserted *Element; walk with Front() and e.Next() until nil.

package main

import (
    "container/list"
    "fmt"
)

func main() {
    l := list.New()
    l.PushBack(1)
    l.PushBack(2)
    l.PushFront(0)

    for e := l.Front(); e != nil; e = e.Next() {
        fmt.Print(e.Value, " ")
    }
    fmt.Println()
    fmt.Println("len:", l.Len())
}
Output
0 1 2 len: 3

O(1) splicing: the LRU move

Holding the *Element lets you MoveToFront (mark as recently used) or Remove (evict) in constant time — no scan, no shifting.

package main

import (
    "container/list"
    "fmt"
)

func main() {
    l := list.New()
    a := l.PushBack("a")
    l.PushBack("b")
    c := l.PushBack("c")

    l.MoveToFront(c) // promote most-recently-used
    l.Remove(a)      // evict least-recently-used

    for e := l.Front(); e != nil; e = e.Next() {
        fmt.Print(e.Value, " ")
    }
    fmt.Println()
}
Output
c b

Common Mistakes

Wrong
// Value is `any` -> forgetting the type assertion,
// or using the zero *List value.
var l list.List        // usable, but easy to misuse
e := l.PushBack(42)
n := e.Value + 1       // compile error: any + int
Right
l := list.New()
e := l.PushBack(42)
n := e.Value.(int) + 1 // assert back to the concrete type

Key Rules

  • PushBack/PushFront return the *Element — keep it if you'll later Remove, MoveToFront, or InsertBefore in O(1).
  • Element.Value is any; type-assert on read. For a single element type, a generic wrapper or a slice-backed structure is often cleaner and faster.
  • Traverse with for e := l.Front(); e != nil; e = e.Next() (or Back()/Prev()); removing the current element mid-loop is safe if you capture e.Next() first.
  • The zero list.List is a valid empty list, but list.New() is the idiomatic constructor; don't move an *Element between different lists.
Practice Exercise

Build an LRU cache with capacity N: a map[string]*list.Element for O(1) lookup plus a *list.List for recency order; on Get move the node to front, on Put insert at front and evict from the back when over capacity. Store a {key, value} struct as the element's Value so eviction can delete the map entry. Then benchmark it against a slice-based version and explain where the linked list's O(1) splice actually pays off — and when a plain slice or generic ring buffer would be faster due to cache locality.

Pitfall: Reaching for container/list as a default sequence type. Because every node is a separately heap-allocated *Element holding an any, it fragments memory, defeats CPU cache locality, and boxes every value — a plain []T beats it for iteration and append-heavy workloads. Use it only when you genuinely need O(1) removal from the middle while holding element references (e.g. an LRU).
Connections: Complements container/heap and container/ring as the standard-library container trio; the LRU pattern pairs a list with a map keyed by comparable values, and its any-typed storage is exactly the boxing that generics (from earlier days) were introduced to avoid.
