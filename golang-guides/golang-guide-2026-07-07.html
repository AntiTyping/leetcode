Init Functions

Functions & Control Flow • 15 days since review • rep 3

Quick Reminder

init() runs automatically after all package-level variables are initialized and before main. A file may declare several init() functions; within a package they run in the order the compiler sees files (alphabetical by filename), and each file's in declaration order. Imported packages are fully initialized first, depth-first. Its main legitimate uses are registration (blank imports like SQL drivers or image decoders), validating package-level state, and precomputing lookup tables.

Key Pattern

var registry = map[string]Handler{}

func init() {              // runs once, before main()
    registry["png"] = pngHandler
}

// A blank import triggers ANOTHER package's init() for its side effect:
import _ "image/png"       // registers the PNG decoder with image.Decode
import _ "github.com/lib/pq" // registers the "postgres" sql driver
Challenge Exercise

Build a codec registry where several files each register a codec in their own init(), and prove the ordering rules (package-level vars first, then init()s, files alphabetical) with strategic prints. Explain why import _ "github.com/lib/pq" is the idiom for wiring a database/sql driver, and why doing slow, failing, or ordering-dependent work (network calls, reading config) inside init() is a design smell that hurts testability.

#1 Pitfall: Relying on cross-file init() ordering, or doing I/O that can fail inside init() — it runs before main, cannot return an error (only panic), and silently makes the package import-order sensitive and awkward to test. Keep init() trivial and side-effect-registration only.

Variadic Functions

Functions & Control Flow • 15 days since review • rep 3

Quick Reminder

A trailing ...T parameter collects zero or more arguments into a []T (a nil slice when none are passed). You can spread an existing slice into the call with f(s...) — but that passes the same backing array, not a copy, so appends within capacity are visible to the caller. append itself is variadic. The ...T parameter must be last.

Key Pattern

func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

sum(1, 2, 3)          // 6  -> nums is []int{1,2,3}
sum()                 // 0  -> nums is nil
xs := []int{4, 5, 6}
sum(xs...)            // 15 -> spread; shares xs's backing array
Challenge Exercise

Write a logf(level, format string, args ...any) that forwards to fmt.Sprintf(format, args...), and a maxOf(first int, rest ...int) int that makes "at least one argument" a compile-time guarantee. Then demonstrate the aliasing gotcha: pass s... to a function that appends to it and show the caller's slice mutate when spare capacity exists — then defend with slices.Clone or a fresh append(([]T)(nil), s...).

#1 Pitfall: Assuming f(s...) copies the slice — it doesn't. A variadic callee that appends within the existing capacity mutates the caller's backing array. Also you cannot place any parameter after ...T, and passing both spread and extra args (f(s..., x)) is a compile error.

Mocking: Interfaces for Testability

Testing & Tooling • 6 days since review • rep 2

Quick Reminder

Make code testable by depending on a small interface (defined at the consumer) rather than a concrete type; in tests, substitute a hand-written fake, stub, or generated mock. Go idiom strongly favors tiny interfaces plus simple fakes over heavyweight mocking frameworks.

Key Pattern

type Store interface {
    Get(id string) (User, error)
}

// A fake implementation for tests -- no database required.
type fakeStore struct{ users map[string]User }

func (f fakeStore) Get(id string) (User, error) {
    u, ok := f.users[id]
    if !ok {
        return User{}, ErrNotFound
    }
    return u, nil
}
// Inject fakeStore in tests, the real *sql.DB-backed store in prod.
Challenge Exercise

Design a service that accepts a Store and a Clock interface. Write table-driven tests using a fake store and a fixed clock, covering both the success and the ErrNotFound paths with zero real I/O. Add a //go:generate directive for a generated mock and explain the cases where a generated mock (call-count and argument assertions, call ordering) earns its keep over a plain fake.

#1 Pitfall: Defining the interface at the implementation instead of the consumer, or making it huge — you end up mocking a 12-method interface when the caller only needs one method. Accept interfaces, return concrete types, and keep interfaces to one or two methods.

go vet & staticcheck

Testing & Tooling • 6 days since review • rep 2

Quick Reminder

go vet flags code that compiles but is suspect: wrong Printf verbs, malformed struct tags, copying a lock by value, unreachable code. staticcheck (and golangci-lint, which bundles many linters) goes further — dead code, inefficient idioms, and bug-prone patterns. None of these are caught by the compiler, so wire them into CI.

Key Pattern

go vet ./...          # ships with the toolchain
staticcheck ./...     # go install honnef.co/go/tools/cmd/staticcheck@latest
golangci-lint run     # aggregates vet + staticcheck + more
fmt.Printf("%d\n", "hello") // compiles fine; go vet: wrong verb %d for string
var mu sync.Mutex
useLock(mu)                 // go vet: call copies lock value sync.Mutex
Challenge Exercise

Plant three defects — a Printf verb mismatch, a sync.Mutex copied by value, and a shadowed err that swallows an error — and record which tool flags each. Add a .golangci.yml that enables a curated linter set, and a CI step that fails the build on any finding. Explain why go vet runs automatically as part of go test but staticcheck does not.

#1 Pitfall: Assuming the compiler catches these — it doesn't. %d with a string, a copied lock, or an inner-scope err that shadows the outer one all compile cleanly. If vet/staticcheck aren't in CI, the entire class of bug silently regresses over time.

Builder Pattern

Advanced Patterns • 6 days since review • rep 2

Quick Reminder

A builder accumulates configuration through methods that each return the builder (pointer receiver!), then a final Build() produces the finished value. It gives fluent, readable construction for objects with many optional fields — in Go it competes with the functional-options pattern.

Key Pattern

type QueryBuilder struct {
    table string
    conds []string
}

func (b *QueryBuilder) From(t string) *QueryBuilder { b.table = t; return b }
func (b *QueryBuilder) Where(c string) *QueryBuilder { b.conds = append(b.conds, c); return b }

func (b *QueryBuilder) Build() (string, error) {
    if b.table == "" {
        return "", errors.New("missing FROM")
    }
    q := "SELECT * FROM " + b.table
    if len(b.conds) > 0 {
        q += " WHERE " + strings.Join(b.conds, " AND ")
    }
    return q, nil
}

// new(QueryBuilder).From("users").Where("age > 18").Build()
Challenge Exercise

Build a SQL query builder — Select(cols...).From(t).Where(c).OrderBy(c).Limit(n).Build() returning the query string plus arguments — using pointer-receiver chaining. Make Build() validate (missing FROM returns an error) rather than panicking. Then contrast it with the functional-options pattern and state when each reads more naturally in Go.

#1 Pitfall: Chaining on value receivers — each call mutates a copy that is then discarded, so only the last set field survives. Use pointer receivers. Also, letting Build() silently emit an invalid object instead of validating and returning an error.

Repository Pattern

Advanced Patterns • 6 days since review • rep 2

Quick Reminder

Hide data access behind an interface so business logic depends on FindByID/Save, not on SQL or a specific database. This lets you swap a Postgres implementation for an in-memory fake in tests and keeps persistence concerns out of the domain.

Key Pattern

type UserRepo interface {
    FindByID(ctx context.Context, id string) (User, error)
    Save(ctx context.Context, u User) error
}

type pgUserRepo struct{ db *sql.DB } // production implementation
type memUserRepo struct{ m map[string]User } // test implementation

func (r *memUserRepo) FindByID(_ context.Context, id string) (User, error) {
    u, ok := r.m[id]
    if !ok {
        return User{}, fmt.Errorf("user %s: %w", id, ErrNotFound)
    }
    return u, nil
}
Challenge Exercise

Define a UserRepo interface, implement a database/sql-backed version and an in-memory version, and test the service against the in-memory one. Thread context.Context through every method and wrap a sentinel ErrNotFound with %w. Then discuss the central trade-off: the abstraction vs. leaking query flexibility, and decide where transactions should live (the "seam" problem).

#1 Pitfall: Letting database types leak through the interface — returning *sql.Rows or raw driver errors defeats the whole abstraction. Return domain types and sentinel errors, and always thread context.Context for cancellation and timeouts.

Section 1 — Logging & Runtime Introspection

log: The Standard Library Logger

Logging

Why it matters: before log/slog there was log, and it is still the right tool for CLIs, small services, and quick diagnostics. It writes timestamped lines to stderr, offers Fatal/Panic helpers, and lets you build custom loggers with prefixes and flags — all in the standard library with zero dependencies.

The package-level logger

log.Print/Printf/Println write to the default logger, which targets os.Stderr with the LstdFlags (date + time) prefix.

package main

import "log"

func main() {
    log.Println("server starting")     // date + time + message, to stderr
    log.Printf("listening on :%d", 8080)
}
Output (to stderr; timestamp is the wall clock)
2026/07/07 10:00:00 server starting 2026/07/07 10:00:00 listening on :8080

A custom logger with prefix and flags

log.New(out, prefix, flags) builds an independent *log.Logger. Combine flag bits with |; Lshortfile appends the caller's file and line.

package main

import (
    "log"
    "os"
)

func main() {
    logger := log.New(os.Stdout, "[api] ", log.Ldate|log.Ltime|log.Lshortfile)
    logger.Println("request handled")
}
Output
[api] 2026/07/07 10:00:00 main.go:10: request handled

Deterministic output for tests (flags = 0)

Point a logger at any io.Writer and set flags to 0 to drop the timestamp — the output becomes stable and easy to assert on.

package main

import (
    "bytes"
    "fmt"
    "log"
)

func main() {
    var buf bytes.Buffer
    logger := log.New(&buf, "TEST: ", 0) // no timestamp
    logger.Print("hello")
    fmt.Printf("%q\n", buf.String())
}
Output
"TEST: hello\n"

Common Mistakes

Wrong
// In library / request-handler code:
func load(path string) Config {
    b, err := os.ReadFile(path)
    if err != nil {
        log.Fatalf("read: %v", err) // calls os.Exit(1)!
    }
    // deferred cleanups NEVER run; caller can't recover
    return parse(b)
}
Right
func load(path string) (Config, error) {
    b, err := os.ReadFile(path)
    if err != nil {
        return Config{}, fmt.Errorf("read %s: %w", path, err)
    }
    return parse(b), nil
}
// Reserve log.Fatal for main()/startup, never libraries.

Key Rules

  • log.Fatal* calls os.Exit(1) (deferred functions do not run) and log.Panic* calls panic — keep both out of libraries; return errors instead.
  • The default logger writes to os.Stderr; redirect with log.SetOutput or build isolated loggers with log.New.
  • Compose flags with | (log.Ldate|log.Ltime|log.Lshortfile); set flags to 0 for deterministic, timestamp-free test output.
  • *log.Logger is safe for concurrent use — each Print is written with a single locked Write, so lines never interleave.
Practice Exercise

Build a tiny leveled logger on top of log.New: Info, Warn, and Error each write with a distinct prefix to a shared io.Writer. Capture output into a bytes.Buffer in a test (flags 0) and assert exact strings. Then show that log.Fatal inside a deferred-cleanup function skips the cleanup, and rewrite the code path to return an error. Finally, explain when you would graduate from log to log/slog.

Pitfall: Calling log.Fatal/log.Fatalf anywhere but top-level startup — it terminates the process via os.Exit, bypassing every defer and giving callers no chance to handle the failure. It also makes the function impossible to unit-test. Libraries should return errors; only main gets to exit.

Connections: The direct predecessor to log/slog (structured logging, today's broader theme); writes through the io.Writer interface; pairs with runtime.Caller (next concept) for custom source annotations and with bytes.Buffer for testable output.

runtime: Introspecting the Running Program

Runtime

Why it matters: sometimes you need to ask the runtime about itself — how many CPUs and goroutines exist, how much heap is live, where a call came from. The runtime package powers diagnostics, custom logging, health metrics, and careful tuning. Used sparingly it's invaluable; reached for reflexively it's usually a code smell.

Environment and goroutine counts

NumCPU, GOMAXPROCS(0) (0 queries without changing), NumGoroutine, and Version report the execution environment.

package main

import (
    "fmt"
    "runtime"
)

func main() {
    fmt.Println("CPUs:", runtime.NumCPU())
    fmt.Println("GOMAXPROCS:", runtime.GOMAXPROCS(0)) // 0 = read, don't set
    fmt.Println("goroutines:", runtime.NumGoroutine())
    fmt.Println("version:", runtime.Version())
}
Output (representative — varies by machine)
CPUs: 8 GOMAXPROCS: 8 goroutines: 1 version: go1.24.0

Memory statistics

ReadMemStats fills a MemStats with live-heap and GC counters. runtime.GC() forces a collection — for diagnostics and tests only, never as a production optimization.

package main

import (
    "fmt"
    "runtime"
)

func main() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    fmt.Printf("HeapAlloc = %d KB\n", m.HeapAlloc/1024)
    fmt.Printf("NumGC = %d\n", m.NumGC)

    runtime.GC() // force a GC cycle (diagnostics only)
    runtime.ReadMemStats(&m)
    fmt.Printf("NumGC after = %d\n", m.NumGC)
}
Output (representative)
HeapAlloc = 110 KB NumGC = 0 NumGC after = 1

Where am I? runtime.Caller

runtime.Caller(skip) returns the program counter, file, and line of a frame up the stack — the building block for source-annotated log helpers.

package main

import (
    "fmt"
    "path/filepath"
    "runtime"
)

func where() string {
    _, file, line, ok := runtime.Caller(1) // 1 = the caller of where()
    if !ok {
        return "unknown"
    }
    return fmt.Sprintf("%s:%d", filepath.Base(file), line)
}

func main() {
    fmt.Println("called from", where())
}
Output
called from main.go:17

Common Mistakes

Wrong
// Finalizers are NOT guaranteed to run -- and may run late.
f, _ := os.Open(name)
runtime.SetFinalizer(f, func(f *os.File) {
    f.Close() // may never fire; file descriptors leak
})
Right
// Deterministic cleanup via defer.
f, err := os.Open(name)
if err != nil {
    return err
}
defer f.Close()

Key Rules

  • GOMAXPROCS(0) reads the current value without changing it; pass a positive number only when you have measured a reason to.
  • Use ReadMemStats for diagnostics, but prefer runtime/metrics for stable, structured production metrics.
  • Never rely on runtime.SetFinalizer for cleanup — finalizers may run late or not at all. Use defer Close() for deterministic release.
  • Don't call runtime.GC() in hot paths as an optimization — it stalls the program; let the collector schedule itself.
Practice Exercise

Write a memSnapshot() that prints HeapAlloc, NumGC, and NumGoroutine, then allocate a large slice, drop the reference, call runtime.GC(), and observe the numbers move. Build a caller() helper using runtime.Caller that annotates each log line with file:line, and compare it to the Lshortfile flag from the log concept. Finally, demonstrate empirically that a SetFinalizer is not a reliable substitute for defer Close().

Pitfall: Treating finalizers as destructors — they run (if ever) at the GC's discretion, long after the object became unreachable, so resources like file descriptors and sockets leak under load. And runtime.NumGoroutine/GC are diagnostics, not control knobs: shaping program behavior around them usually hides a real design problem.
Connections: Neighbours runtime/debug and runtime/metrics; runtime.Caller underlies the Lshortfile behavior of the log package (previous concept); MemStats and GOMAXPROCS tie into the escape-analysis, sync.Pool, and profiling (pprof) material from earlier days.

Section 2 — Web & Data Formats

mime: Media Types & Content-Type Headers

Web

Why it matters: every HTTP response needs a correct Content-Type, and every upload arrives with one you must parse. The mime package maps file extensions to MIME types and parses/formats media-type headers (with their charset, boundary, and other parameters) correctly — string-splitting a Content-Type by hand gets quoting and whitespace wrong.

Extensions and types

TypeByExtension looks up the type for a dotted extension; ExtensionsByType goes the other way.

package main

import (
    "fmt"
    "mime"
)

func main() {
    fmt.Println(mime.TypeByExtension(".html")) // text/html; charset=utf-8
    fmt.Println(mime.TypeByExtension(".json")) // application/json
    exts, _ := mime.ExtensionsByType("image/png")
    fmt.Println(exts)                          // [.png]
}
Output
text/html; charset=utf-8 application/json [.png]

Parsing a Content-Type with parameters

ParseMediaType splits the base type from its parameters and lowercases/normalizes them — the correct way to read a charset or multipart boundary.

package main

import (
    "fmt"
    "mime"
)

func main() {
    mt, params, err := mime.ParseMediaType("text/html; charset=utf-8")
    fmt.Println(mt, params, err)

    _, p2, _ := mime.ParseMediaType(`multipart/form-data; boundary="xYz"`)
    fmt.Println("boundary:", p2["boundary"])
}
Output
text/html map[charset:utf-8] <nil> boundary: xYz

Formatting and registering types

FormatMediaType builds a well-formed header (quoting when needed); AddExtensionType teaches the lookup table a new mapping.

package main

import (
    "fmt"
    "mime"
)

func main() {
    ct := mime.FormatMediaType("text/plain", map[string]string{"charset": "utf-8"})
    fmt.Println(ct) // text/plain; charset=utf-8

    _ = mime.AddExtensionType(".md", "text/markdown; charset=utf-8")
    fmt.Println(mime.TypeByExtension(".md"))
}
Output
text/plain; charset=utf-8 text/markdown; charset=utf-8

Common Mistakes

Wrong
// Hand-splitting a Content-Type header:
parts := strings.Split(header, ";")
mt := parts[0] // ignores params, mishandles quotes,
               // and keeps stray whitespace/casing
Right
mt, params, err := mime.ParseMediaType(header)
if err != nil {
    return err
}
// mt normalized; params["charset"], params["boundary"], ...

Key Rules

  • TypeByExtension needs the leading dot (".png", not "png") and returns "" for unknown extensions.
  • Parse incoming Content-Type headers with ParseMediaType — never split on ";" yourself; it handles quoting, casing, and whitespace.
  • Build outgoing headers with FormatMediaType so parameters are correctly quoted and ordered.
  • The extension table is seeded from the OS plus built-ins; extend it deterministically with AddExtensionType rather than trusting the host.
Practice Exercise

Write a contentTypeFor(filename string) string that returns the MIME type via TypeByExtension, defaulting to application/octet-stream for unknowns. Then write charsetOf(header string) string that uses ParseMediaType to extract the charset parameter (defaulting to utf-8), and prove it handles a quoted, oddly-spaced header that a naive split would mangle. Finally, register a custom extension and round-trip a header through FormatMediaType then ParseMediaType.

Pitfall: Parsing Content-Type by splitting on ";" and "=" — it breaks on quoted parameter values (a boundary may contain ;), inconsistent casing, and optional whitespace. Also assuming TypeByExtension is populated identically everywhere: it partly depends on the host's MIME database, so register the mappings you rely on.
Connections: The base package beneath mime/multipart (file uploads) and the Content-Type handling in net/http; complements net/url and http.FileServer; the boundary parameter it parses is exactly what multipart form handling consumes.

net/mail: Parsing Addresses & Messages

Email

Why it matters: validating and normalizing email addresses with a regex is a well-known trap — the real grammar (RFC 5322) is far more subtle than \S+@\S+. net/mail parses addresses and message headers correctly, so you can validate input, extract display names, and read raw messages without reinventing the spec.

Parsing a single address

ParseAddress accepts a display-name form and returns the name plus the bare address, erroring on genuinely invalid input.

package main

import (
    "fmt"
    "net/mail"
)

func main() {
    a, err := mail.ParseAddress("Alice Example <alice@example.com>")
    if err != nil {
        panic(err)
    }
    fmt.Printf("name=%q addr=%q\n", a.Name, a.Address)
}
Output
name="Alice Example" addr="alice@example.com"

Lists and canonical formatting

ParseAddressList handles comma-separated headers; Address.String() renders a canonical, correctly-quoted form.

package main

import (
    "fmt"
    "net/mail"
)

func main() {
    list, _ := mail.ParseAddressList("a@x.com, Bob <b@y.com>")
    for _, a := range list {
        fmt.Println(a.String())
    }
}
Output
<a@x.com> "Bob" <b@y.com>

Reading a whole message

ReadMessage parses the header block and leaves the body as an io.Reader. Header.AddressList parses an address-bearing header for you.

package main

import (
    "fmt"
    "io"
    "net/mail"
    "strings"
)

func main() {
    raw := "From: Alice <alice@example.com>\r\n" +
        "To: bob@example.com\r\n" +
        "Subject: Hi\r\n" +
        "\r\n" +
        "Hello Bob!\r\n"

    msg, err := mail.ReadMessage(strings.NewReader(raw))
    if err != nil {
        panic(err)
    }
    fmt.Println("Subject:", msg.Header.Get("Subject"))
    from, _ := msg.Header.AddressList("From")
    fmt.Println("From addr:", from[0].Address)
    body, _ := io.ReadAll(msg.Body)
    fmt.Printf("Body: %q\n", string(body))
}
Output
Subject: Hi From addr: alice@example.com Body: "Hello Bob!\r\n"

Common Mistakes

Wrong
// Regex "validation" -- rejects valid addresses,
// accepts invalid ones, and misses display names.
ok, _ := regexp.MatchString(`^\S+@\S+$`, s)
if !ok {
    return errInvalid
}
Right
// RFC 5322-aware parsing.
addr, err := mail.ParseAddress(s)
if err != nil {
    return fmt.Errorf("invalid address: %w", err)
}
_ = addr.Address // normalized address

Key Rules

  • Validate and normalize addresses with ParseAddress/ParseAddressList, not regexes — the grammar is genuinely complex.
  • Address.String() produces a canonical, correctly-quoted rendering — use it when re-emitting headers.
  • ReadMessage parses only the header block; the body stays a streaming io.Reader (decode MIME parts with mime/multipart if needed).
  • net/mail parses; it does not send mail (that's net/smtp) or verify that an address is deliverable.
Practice Exercise

Write normalizeEmail(s string) (string, error) that parses with ParseAddress and returns the lower-cased bare Address, rejecting invalid input. Then parse a To header containing three recipients (one with a quoted display name) via ParseAddressList and print each canonical form. Finally, feed a small raw message into ReadMessage, extract Subject, Date (via Header.Date()), and the From address, and explain why a regex is the wrong validation tool.

Pitfall: Rolling your own email validation with a regex — you will both reject valid RFC 5322 addresses (quoted local parts, display names, comments) and admit malformed ones. Delegate to mail.ParseAddress. Also remember parsing an address says nothing about deliverability — that requires DNS/SMTP checks well beyond this package.
Connections: Sits alongside net/smtp (sending) and mime/mime/multipart (decoding message bodies and attachments); ReadMessage returns a body as an io.Reader, tying into io/bufio; header values thread into time parsing via Header.Date().

image: Generating & Encoding Raster Images

Images

Why it matters: pure-Go image generation covers a surprising amount of real work — thumbnails, charts, badges, QR-style grids, deterministic test fixtures — with no external libraries. The image package models pixels and colors; image/png, image/jpeg, and image/gif encode and decode. A key gotcha is that a format is only usable once its package is imported for its init() registration.

Draw pixels and encode a PNG

image.NewRGBA allocates a mutable image over a rectangle; Set writes one pixel; png.Encode streams it to any io.Writer.

package main

import (
    "bytes"
    "fmt"
    "image"
    "image/color"
    "image/png"
)

func main() {
    img := image.NewRGBA(image.Rect(0, 0, 100, 50))
    for x := 0; x < 100; x++ {
        img.Set(x, 25, color.RGBA{R: 255, A: 255}) // a red horizontal line
    }
    var buf bytes.Buffer
    png.Encode(&buf, img)
    fmt.Println("encoded PNG:", buf.Len() > 0)
    fmt.Println("bounds:", img.Bounds())
}
Output
encoded PNG: true bounds: (0,0)-(100,50)

Fill a region with draw.Draw

The image/draw package composites: an image.Uniform is an infinite solid color, and draw.Src copies it into the destination rectangle.

package main

import (
    "fmt"
    "image"
    "image/color"
    "image/draw"
)

func main() {
    img := image.NewRGBA(image.Rect(0, 0, 4, 4))
    blue := color.RGBA{0, 0, 255, 255}
    draw.Draw(img, img.Bounds(), &image.Uniform{C: blue}, image.Point{}, draw.Src)
    fmt.Println(img.At(2, 2).(color.RGBA))
}
Output
{0 0 255 255}

Read dimensions cheaply with DecodeConfig

image.DecodeConfig reads only the header to report width, height, and format — no full decode. It needs the format package imported for its side-effect registration.

package main

import (
    "bytes"
    "fmt"
    "image"
    "image/png"
)

func main() {
    var buf bytes.Buffer
    png.Encode(&buf, image.NewRGBA(image.Rect(0, 0, 320, 240)))

    cfg, format, err := image.DecodeConfig(bytes.NewReader(buf.Bytes()))
    fmt.Println(cfg.Width, cfg.Height, format, err)
}
Output
320 240 png <nil>

Common Mistakes

Wrong
import "image" // no decoder registered

func size(r io.Reader) {
    _, format, err := image.Decode(r)
    // err: "image: unknown format" -- PNG init() never ran
    fmt.Println(format, err)
}
Right
import (
    "image"
    _ "image/png"  // blank import: init() registers the decoder
    _ "image/jpeg"
)

_, format, err := image.Decode(r) // now recognizes png/jpeg

Key Rules

  • Import a format package (often as a blank import _ "image/png") so its init() registers the decoder before you call image.Decode/DecodeConfig.
  • An image's coordinate space is its Bounds() rectangle — it need not start at (0,0), and Set outside the bounds is a silent no-op.
  • color.RGBA is alpha-premultiplied; a fully transparent-looking color with A: 0 stores no visible channel — set A: 255 for opaque colors.
  • Use DecodeConfig to get dimensions cheaply (header only) instead of a full Decode when you just need the size.
Practice Exercise

Generate a 256×256 PNG containing a diagonal gradient (compute each pixel's color.RGBA from its coordinates) and write it to a file with os.Create + png.Encode. Then write imageSize(path string) (int, int, error) using image.DecodeConfig with the correct blank imports. Finally, deliberately drop the _ "image/png" import and show the exact "unknown format" error — connecting this to today's init() review — then restore it.

Pitfall: Forgetting the blank import of the format package — image.Decode then fails with "unknown format" even though the file is a valid PNG, because the decoder self-registers in its init() and nothing pulled that package in. The mirror gotcha is color.RGBA{255,0,0,0}: with A: 0 it renders as transparent, not red.
Connections: Decoder registration is a live example of today's init() review and the blank-import idiom; encoding streams through io.Writer (the io and bufio days); it pairs with net/http for serving generated images and with bytes.Buffer for in-memory rendering and tests.
