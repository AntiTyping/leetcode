Multiple Return Values

Functions & Control Flow • due today • rep 3

Quick Reminder

Go functions return any number of values, and two idioms dominate: (value, error) for operations that can fail, and the comma-ok form (value, bool) for map lookups, type assertions, and channel receives. Named return values document intent and enable a bare return, while the blank identifier _ discards results you don't need.

Key Pattern

package main

import (
    "errors"
    "fmt"
)

// Named returns (q, r, err) document the result shape.
func divmod(a, b int) (q, r int, err error) {
    if b == 0 {
        return 0, 0, errors.New("division by zero")
    }
    return a / b, a % b, nil
}

func main() {
    q, r, _ := divmod(17, 5)
    fmt.Println(q, r)

    counts := map[string]int{"go": 3}
    if n, ok := counts["go"]; ok { // comma-ok: distinguishes 0 from absent
        fmt.Println("found:", n)
    }
    if _, ok := counts["rust"]; !ok {
        fmt.Println("rust missing")
    }
}
Output
3 2 found: 3 rust missing
Challenge Exercise

Write parsePair(s string) (k, v string, ok bool) that splits "key=value" on the first = and returns ok=false when there is no =. Then write a function with named returns that uses a defer closure to wrap an error with extra context (err = fmt.Errorf("load: %w", err)) only when the bare return path produced a non-nil error. Explain how the deferred closure can observe and mutate the named return.

Pitfall: Using := in an inner scope and accidentally shadowing a named return, so the value you set never reaches the caller. With named returns, assign with = in nested blocks, and never silently discard the error result with _ unless you have truly handled the failure.

Custom Constraints

Functions & Control Flow • due today • rep 3

Quick Reminder

A constraint is an interface used as a type-parameter bound. Beyond methods, a constraint can list a type set: a union of types joined with |, where ~T admits every named type whose underlying type is T. The compiler permits an operator inside the generic body only if it is valid for every type in the set, so a numeric union enables + and an ordered union enables <.

Key Pattern

package main

import "fmt"

// ~ admits named types (like Celsius) whose underlying type matches.
type Numeric interface {
    ~int | ~int64 | ~float64
}

func Sum[T Numeric](xs []T) T {
    var total T // zero value of T
    for _, x := range xs {
        total += x // legal: every type in the set supports +
    }
    return total
}

type Celsius float64

func main() {
    fmt.Println(Sum([]int{1, 2, 3}))
    fmt.Println(Sum([]Celsius{1.5, 2.5})) // Celsius has underlying float64
}
Output
6 4
Challenge Exercise

Define Ordered (ints, floats, strings) and write Min[T Ordered](xs []T) (T, bool) returning ok=false for an empty slice. Then build a constraint that requires both a method (fmt.Stringer) and a type set, and explain when the intersection of a method element and a type set is empty — making the constraint impossible to satisfy. Finally, show why a plain any type parameter cannot use < while an Ordered one can.

Pitfall: Listing a concrete type (int) instead of ~int in the union. Without the tilde, a named type such as Celsius or type ID int will not satisfy the constraint, and instantiation fails with a confusing "does not implement" error even though the underlying type matches.

sync Package

Concurrency • due today • rep 3

Quick Reminder

The sync package guards shared state when channels are overkill. Mutex/RWMutex protect critical sections (always defer Unlock()); WaitGroup joins a set of goroutines via Add/Done/Wait; and Once runs an initializer exactly once even under concurrent callers. None of these may be copied after first use — pass them by pointer.

Key Pattern

package main

import (
    "fmt"
    "sync"
)

func main() {
    var (
        mu    sync.Mutex
        wg    sync.WaitGroup
        count int
    )
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            mu.Lock()
            count++ // protected critical section
            mu.Unlock()
        }()
    }
    wg.Wait() // block until all 100 goroutines finish

    var once sync.Once
    for i := 0; i < 3; i++ {
        once.Do(func() { fmt.Println("init once") }) // runs a single time
    }
    fmt.Println("count:", count)
}
Output
init once count: 100
Challenge Exercise

Build a concurrency-safe Counter struct embedding a sync.Mutex with Inc() and Value() methods, and prove it is race-free under go test -race. Then convert a read-heavy lookup cache to sync.RWMutex so many readers share RLock while writers take the exclusive Lock. Finally, replace a lazy if cfg == nil init with sync.Once and explain why the manual check has a data race.

Pitfall: Copying a struct that contains a sync.Mutex (e.g. passing it by value or appending it to a slice). The copy has its own independent lock, so two goroutines can enter the "protected" section simultaneously. go vet flags this as a lock copy — always hold sync types behind a pointer.

errgroup

Concurrency • due today • rep 3

Quick Reminder

golang.org/x/sync/errgroup is a WaitGroup that also collects errors. g.Go(func() error) launches each task; g.Wait() blocks and returns the first non-nil error. errgroup.WithContext returns a context that is canceled the moment any task fails, so siblings can bail out early, and g.SetLimit(n) bounds concurrency.

Key Pattern

package main

import (
    "context"
    "errors"
    "fmt"

    "golang.org/x/sync/errgroup"
)

func main() {
    g, ctx := errgroup.WithContext(context.Background())

    g.Go(func() error { return nil })
    g.Go(func() error { return errors.New("task 2 failed") }) // cancels ctx
    g.Go(func() error {
        <-ctx.Done()       // observes the cancellation from task 2
        return ctx.Err()
    })

    if err := g.Wait(); err != nil {
        fmt.Println("first error:", err)
    }
}
Output
first error: task 2 failed
Challenge Exercise

Fetch a slice of URLs concurrently with errgroup.WithContext, writing each body into a pre-sized results []string by index (disjoint slots, no mutex). Add g.SetLimit(4) to cap in-flight requests, and confirm that when one fetch returns an error the others observe ctx.Done() and stop. Explain why g.Wait() returns only the first error and how you would collect all of them with errors.Join instead.

Pitfall: Ignoring the context that WithContext hands back. If your tasks never select on ctx.Done() (or pass it to http.NewRequestWithContext), a failure in one task can't actually stop the others — you lose the early-cancellation benefit and keep doing doomed work.

httptest

Testing & Tooling • due today • rep 3

Quick Reminder

net/http/httptest tests HTTP code without a real port. To test a handler, drive it with httptest.NewRequest plus a ResponseRecorder and inspect rec.Code/rec.Body. To test a client, stand up an ephemeral httptest.NewServer (real loopback socket) and point your code at ts.URL — always defer ts.Close().

Key Pattern

package main

import (
    "fmt"
    "io"
    "net/http"
    "net/http/httptest"
)

func greet(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "hello %s", r.URL.Query().Get("name"))
}

func main() {
    // 1) Handler test: recorder, no network.
    req := httptest.NewRequest("GET", "/?name=Ada", nil)
    rec := httptest.NewRecorder()
    greet(rec, req)
    fmt.Println("status:", rec.Code)
    fmt.Println("body:", rec.Body.String())

    // 2) Client test: real ephemeral server.
    ts := httptest.NewServer(http.HandlerFunc(greet))
    defer ts.Close()
    resp, _ := http.Get(ts.URL + "?name=Bo")
    b, _ := io.ReadAll(resp.Body)
    resp.Body.Close()
    fmt.Println("server said:", string(b))
}
Output
status: 200 body: hello Ada server said: hello Bo
Challenge Exercise

Write a table-driven test for a handler that returns 400 on a missing query parameter and 200 otherwise, asserting on rec.Code and rec.Body. Then test a client wrapper against an httptest.NewServer that returns 500, verifying your wrapper surfaces a typed error. Finally, swap to httptest.NewTLSServer and use ts.Client() so the client trusts the test certificate.

Pitfall: Forgetting defer ts.Close() (and resp.Body.Close()). Each leaked test server keeps a listener and goroutine alive, and unclosed bodies leak connections — across a large suite this exhausts file descriptors and produces flaky, order-dependent failures.

Functional Options Pattern

Advanced Patterns • due today • rep 3

Quick Reminder

The functional options pattern gives constructors a clean, extensible, backward-compatible API. Each option is a function that mutates a config struct (type Option func(*Server)); the constructor seeds sensible defaults, then applies the variadic opts ...Option. Callers set only what they care about, and adding a new option never breaks existing call sites.

Key Pattern

package main

import "fmt"

type Server struct {
    host string
    port int
    tls  bool
}

type Option func(*Server)

func WithHost(h string) Option { return func(s *Server) { s.host = h } }
func WithPort(p int) Option    { return func(s *Server) { s.port = p } }
func WithTLS() Option          { return func(s *Server) { s.tls = true } }

func NewServer(opts ...Option) *Server {
    s := &Server{host: "localhost", port: 8080} // defaults
    for _, opt := range opts {
        opt(s)
    }
    return s
}

func main() {
    s := NewServer(WithPort(9090), WithTLS()) // host keeps its default
    fmt.Printf("%s:%d tls=%v\n", s.host, s.port, s.tls)
}
Output
localhost:9090 tls=true
Challenge Exercise

Add a WithTimeout(d time.Duration) option and a WithLogger option, then design one that can fail validation: change Option to func(*Server) error so NewServer can return (*Server, error) and reject, say, a negative port. Discuss the trade-off versus a plain config struct, and explain why keeping the struct fields unexported is what makes options the only configuration path.

Pitfall: Exporting the config struct's fields. If callers can set s.Port directly, the options become optional decoration, validation can be bypassed, and you lose the encapsulation that justified the pattern. Keep fields unexported so the constructor is the single source of truth.

Section 1 — Modern Standard Library

os.Root: Directory-Scoped Filesystem Access

Std Lib

Why it matters: os.Root (Go 1.24) confines all file operations to a single directory tree. Every path is resolved relative to the root and any attempt to escape it — via .., an absolute path, or a symlink pointing outside — is rejected by the OS-level checks. It is the robust answer to path-traversal bugs that hand-rolled filepath.Clean guards keep getting wrong.

Open a root and read inside it

os.OpenRoot(dir) returns an *os.Root whose methods (Open, Create, Mkdir, Stat, ...) behave like the os equivalents but can never reach outside dir.

package main

import (
    "fmt"
    "io"
    "os"
)

func main() {
    dir, _ := os.MkdirTemp("", "demo")
    defer os.RemoveAll(dir)
    os.WriteFile(dir+"/note.txt", []byte("inside"), 0o644)

    root, err := os.OpenRoot(dir)
    if err != nil {
        fmt.Println("open root:", err)
        return
    }
    defer root.Close()

    f, err := root.Open("note.txt") // resolved relative to the root
    if err != nil {
        fmt.Println("open:", err)
        return
    }
    data, _ := io.ReadAll(f)
    f.Close()
    fmt.Println("read:", string(data))

    // A path that climbs out of the root is refused.
    if _, err := root.Open("../../../etc/passwd"); err != nil {
        fmt.Println("traversal blocked:", true)
    }
}
Output
read: inside traversal blocked: true

Writes are confined too

The same containment applies to Create, Mkdir, and Remove — an escaping path fails before any file is touched, so an attacker-controlled filename can't write outside your upload directory.

package main

import (
    "fmt"
    "os"
)

func main() {
    dir, _ := os.MkdirTemp("", "uploads")
    defer os.RemoveAll(dir)

    root, _ := os.OpenRoot(dir)
    defer root.Close()

    f, err := root.Create("report.txt") // stays inside the root
    if err != nil {
        fmt.Println("create:", err)
        return
    }
    f.WriteString("ok")
    f.Close()

    _, err = root.Create("../escape.txt") // rejected, nothing written
    fmt.Println("blocked climb-out:", err != nil)
}
Output
blocked climb-out: true

Common Mistakes

Wrong
// Manual guard: filepath.Join cleans
// ".." but does NOT stop a symlink
// inside dir from pointing elsewhere.
p := filepath.Join(dir, userPath)
os.ReadFile(p)
Right
root, _ := os.OpenRoot(dir)
defer root.Close()
f, err := root.Open(userPath)
// .. , absolute paths, and escaping
// symlinks are all rejected

Key Rules

  • os.OpenRoot(dir) returns an *os.Root; always defer root.Close() like any file handle.
  • Methods mirror os: Open, OpenFile, Create, Mkdir, Stat, Remove — all relative to the root.
  • Escapes via .., absolute paths, and symlinks resolving outside the root are rejected, not silently clamped.
  • Use it anywhere you handle untrusted path components: uploads, archive extraction (zip-slip), plugin directories.
Practice Exercise

Write a safe archive extractor: given a *zip.Reader and a destination dir, open an os.Root on the dir and create each entry through the root, so a malicious entry named ../../etc/cron.d/x fails instead of escaping (the classic "zip slip"). Then create a symlink inside the root that points to /etc and confirm that opening a path through that symlink is blocked. Compare your code's safety to a filepath.Join + strings.HasPrefix guard.

Pitfall: Treating os.Root as merely "cleaner filepath.Join" and still concatenating paths yourself. The whole point is to pass the untrusted component straight to root.Open/root.Create; if you pre-join with the real directory and call plain os functions, you reintroduce exactly the traversal hole you were trying to close.
Connections: Builds on the os & filepath material from earlier guides and is the security-hardened counterpart to io/fs. It pairs naturally with archive handling and any handler that accepts a user-supplied filename.

weak Pointers & the weak Package

Std Lib

Why it matters: a weak.Pointer[T] (Go 1.24) references an object without keeping it alive. The garbage collector may reclaim the target at any time, after which Value() returns nil. This is the building block for memory-sensitive caches, canonicalization (interning) maps, and associating side data with objects — structures that must not, by themselves, prevent collection.

A weak reference goes nil after collection

weak.Make(p) wraps a *T. While a strong reference still exists, Value() returns the pointer; once the last strong reference is gone and the GC runs, it returns nil.

package main

import (
    "fmt"
    "runtime"
    "weak"
)

type Config struct{ Name string }

func main() {
    c := &Config{Name: "prod"}
    wp := weak.Make(c) // weak.Pointer[Config]

    fmt.Println("alive:", wp.Value() != nil)
    fmt.Println("name:", wp.Value().Name)

    c = nil      // drop the only strong reference
    runtime.GC() // make the object collectable now

    fmt.Println("after GC, nil:", wp.Value() == nil)
}
Output
alive: true name: prod after GC, nil: true

A cache that doesn't pin its entries

Storing weak.Pointer values lets a cache hand back a live object on a hit, yet never keep an otherwise-unused object alive — bounded memory for free.

package main

import (
    "fmt"
    "weak"
)

type Sprite struct{ Name string }

// cache holds weak refs, so cached sprites can still be collected.
var cache = map[string]weak.Pointer[Sprite]{}

func get(name string) *Sprite {
    if wp, ok := cache[name]; ok {
        if s := wp.Value(); s != nil {
            return s // hit, still alive
        }
    }
    s := &Sprite{Name: name} // (re)load
    cache[name] = weak.Make(s)
    return s
}

func main() {
    a := get("hero")
    b := get("hero") // served from the cache while a is still live
    fmt.Println("same instance:", a == b)
    fmt.Println("name:", a.Name)
}
Output
same instance: true name: hero

Common Mistakes

Wrong
// Race: the object can be collected
// between the check and the deref.
if wp.Value() != nil {
    use(wp.Value()) // may be nil now!
}
Right
// Capture once into a strong local,
// then test and use that.
if s := wp.Value(); s != nil {
    use(s) // s keeps it alive here
}

Key Rules

  • weak.Make(p) builds a weak.Pointer[T]; wp.Value() returns a strong *T or nil if collected.
  • Always capture Value() into a local before testing/using it — the local is a strong reference that pins the object for that scope.
  • Weak pointers are comparable and hashable even after the target dies, so they work as map keys.
  • Pair with runtime.AddCleanup to run finalization logic (e.g. evict the dead map entry) when the target is reclaimed.
Practice Exercise

Build a string interner: map[string]weak.Pointer[string] where Intern(s) returns a canonical *string, reusing the existing one on a hit and storing a fresh weak pointer otherwise. Use runtime.AddCleanup to delete the map entry once a canonical value is collected, then write a test that forces a GC and asserts the entry is gone. Explain why a plain map[string]*string would leak every interned string forever.

Pitfall: Calling wp.Value() twice and assuming both calls return the same liveness. Between the two calls the GC can reclaim the object, so the second can be nil even though the first was not. Dereference only a single captured local, and never write logic that depends on a weak pointer staying alive across function calls.
Connections: Complements sync.Map for building concurrent caches, and the bounded-memory idea contrasts with the container/heap / LRU approach — weak pointers let the GC decide eviction, an LRU decides it explicitly.

testing/synctest: Deterministic Concurrency Tests

Std Lib

Why it matters: testing timeouts, tickers, and context deadlines normally means real time.Sleep calls — slow and flaky. testing/synctest (stable in Go 1.25) runs your test inside a bubble with a fake clock: time only advances when every goroutine in the bubble is blocked, so a one-hour timeout test finishes in microseconds and is fully deterministic.

A timeout test that runs instantly

synctest.Test(t, func) starts the bubble. Inside it, time.Sleep advances the fake clock rather than waiting, and synctest.Wait() blocks until all other bubbled goroutines are durably parked.

package cache

import (
    "testing"
    "testing/synctest"
    "time"
)

// expireAfter closes the returned channel once d has elapsed.
func expireAfter(d time.Duration) <-chan struct{} {
    ch := make(chan struct{})
    go func() {
        time.Sleep(d)
        close(ch)
    }()
    return ch
}

func TestExpiry(t *testing.T) {
    synctest.Test(t, func(t *testing.T) {
        ch := expireAfter(time.Hour) // no real wait

        select {
        case <-ch:
            t.Fatal("expired too early")
        default: // not yet
        }

        time.Sleep(time.Hour) // jumps the fake clock to the deadline
        synctest.Wait()       // let the timer goroutine close ch

        select {
        case <-ch: // expired exactly on schedule
        default:
            t.Fatal("should have expired")
        }
    })
}
Output (go test)
ok example/cache 0.00s

Deterministic context deadlines

Because context timers also use the bubble's clock, you can assert deadline behavior without sleeping for real.

package worker

import (
    "context"
    "testing"
    "testing/synctest"
    "time"
)

func TestContextTimeout(t *testing.T) {
    synctest.Test(t, func(t *testing.T) {
        ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()

        synctest.Wait()
        if ctx.Err() != nil {
            t.Fatal("cancelled too soon")
        }

        time.Sleep(5 * time.Second) // advance to the deadline
        synctest.Wait()

        if ctx.Err() != context.DeadlineExceeded {
            t.Fatalf("want DeadlineExceeded, got %v", ctx.Err())
        }
    })
}
Output (go test)
ok example/worker 0.00s

Common Mistakes

Wrong
// Real sleeps: slow, and the margins
// make the test flaky on a busy CI box.
go worker()
time.Sleep(2 * time.Second)
checkResult(t)
Right
synctest.Test(t, func(t *testing.T) {
    go worker()
    synctest.Wait() // block until worker parks
    checkResult(t)  // deterministic
})

Key Rules

  • synctest.Test(t, fn) runs fn in a bubble; all goroutines it starts join the same bubble.
  • The fake clock advances only when every bubbled goroutine is durably blocked — so timers fire in deterministic order.
  • synctest.Wait() blocks the caller until all other bubbled goroutines are parked; use it before assertions.
  • Stable in Go 1.25 (was synctest.Run behind GOEXPERIMENT=synctest in 1.24). It is part of the standard go test.
Practice Exercise

Test a rate limiter that allows one event per second: inside a bubble, fire two events back-to-back, assert the second is rejected, then time.Sleep(time.Second) and assert it is now allowed — all in zero real time. Next, write a flaky-looking test for a context.WithTimeout worker the old way with real sleeps, then port it to synctest and measure the wall-clock difference. Explain what "durably blocked" means and why a goroutine doing real network I/O breaks the bubble's assumptions.

Pitfall: Performing real, non-fake-clock blocking inside the bubble — e.g. a real network read or a blocking syscall. The fake clock only advances when all bubbled goroutines are blocked on bubble-aware operations (channels, mutexes, timers); a goroutine stuck in real I/O never counts as "durably blocked," so time never advances and the test deadlocks.
Connections: Pairs with the sync primitives and context-based cancellation from the review section — synctest is how you test that goroutine choreography deterministically, replacing the brittle time.Sleep assertions of earlier testing guides.

Section 2 — Concurrency & Data Structures

sync.Map: When (and When Not) to Use It

Concurrency

Why it matters: sync.Map is a concurrent map you use without your own mutex. It shines in exactly two access patterns: (1) a key is written once and read many times, or (2) goroutines operate on disjoint sets of keys. For general read/write workloads a plain map guarded by a sync.Mutex is faster — knowing which to reach for is the whole lesson.

Atomic get-or-insert with LoadOrStore

Methods take and return any. LoadOrStore inserts only if the key is absent and tells you which happened via loaded; Range walks the map (returning false stops early).

package main

import (
    "fmt"
    "sync"
)

func main() {
    var m sync.Map

    actual, loaded := m.LoadOrStore("a", 1)
    fmt.Println("first:", actual, loaded) // 1 false (we inserted)

    actual, loaded = m.LoadOrStore("a", 99)
    fmt.Println("second:", actual, loaded) // 1 true (already present)

    m.Store("b", 2)
    if v, ok := m.Load("b"); ok {
        fmt.Println("b =", v)
    }

    total := 0
    m.Range(func(key, value any) bool {
        total += value.(int) // values are any: type-assert
        return true          // return false to stop iterating
    })
    fmt.Println("total:", total)
}
Output
first: 1 false second: 1 true b = 2 total: 3

Atomic take-and-remove

LoadAndDelete removes a key and returns its old value in one atomic step — handy for "claim a job" style hand-offs.

package main

import (
    "fmt"
    "sync"
)

func main() {
    var seen sync.Map
    seen.Store("job-1", "done")

    if v, ok := seen.LoadAndDelete("job-1"); ok {
        fmt.Println("took:", v)
    }
    _, ok := seen.Load("job-1")
    fmt.Println("still there:", ok)
}
Output
took: done still there: false

Common Mistakes

Wrong
// Reaching for sync.Map as a default
// for a general high-write workload.
var m sync.Map // slower than a guarded
// plain map here, and untyped (any).
Right
// Read-mostly or disjoint keys -> sync.Map.
// Otherwise: a typed map + a mutex.
var mu sync.Mutex
m := map[string]int{}

Key Rules

  • Use it for read-mostly caches or when goroutines touch disjoint keys; otherwise prefer map + sync.Mutex.
  • Core API: Load, Store, LoadOrStore, LoadAndDelete, Delete, Range; atomic variants Swap, CompareAndSwap, CompareAndDelete (Go 1.20+).
  • Keys and values are any — you lose static typing and must type-assert on read.
  • Range is a weakly-consistent snapshot: it need not reflect concurrent stores and may visit a key at most once.
Practice Exercise

Build a concurrent memoizer for an expensive pure function using LoadOrStore to cache results, then run it from many goroutines under -race and confirm the function still computes each key only once (hint: store a *sync.Once or a future, not the raw value, to avoid duplicate computation on a miss). Then benchmark a 90%-read / 10%-write workload as sync.Map versus map+RWMutex and report where each wins.

Pitfall: Assuming LoadOrStore prevents duplicate work. It only deduplicates the stored value — if you compute an expensive result and then call LoadOrStore, every racing goroutine still computes before one wins. To dedup the computation itself, store a once-guarded placeholder (or use singleflight).
Connections: Extends the sync primitives from the review section and pairs with weak pointers for caches whose entries shouldn't outlive their last user. For deduplicating concurrent computation, it connects to the singleflight pattern.

container/heap: Priority Queues

Data Structures

Why it matters: container/heap turns any slice type into a binary heap, giving you a priority queue with O(log n) push and pop. It's the backbone of Dijkstra, event simulators, task schedulers, and "top-K" problems — anywhere you repeatedly need the current minimum (or maximum) without sorting the whole collection each time.

A minimal min-heap

You implement heap.Interface (which embeds sort.Interface plus Push/Pop) on a slice type. Less decides the ordering: < makes a min-heap. You then call the package functions heap.Init, heap.Push, heap.Pop — not the methods directly.

package main

import (
    "container/heap"
    "fmt"
)

type IntHeap []int

func (h IntHeap) Len() int           { return len(h) }
func (h IntHeap) Less(i, j int) bool { return h[i] < h[j] } // min-heap
func (h IntHeap) Swap(i, j int)      { h[i], h[j] = h[j], h[i] }
func (h *IntHeap) Push(x any)        { *h = append(*h, x.(int)) }
func (h *IntHeap) Pop() any {
    old := *h
    n := len(old)
    v := old[n-1]
    *h = old[:n-1]
    return v
}

func main() {
    h := &IntHeap{5, 2, 8, 1}
    heap.Init(h)     // O(n) heapify
    heap.Push(h, 3)  // O(log n)

    for h.Len() > 0 {
        fmt.Print(heap.Pop(h), " ") // always the current minimum
    }
    fmt.Println()
}
Output
1 2 3 5 8

A priority queue of tasks

Flip Less to > for a max-priority queue. Storing structs lets each element carry a payload alongside its priority.

package main

import (
    "container/heap"
    "fmt"
)

type Task struct {
    name     string
    priority int
}

type PQ []Task

func (pq PQ) Len() int           { return len(pq) }
func (pq PQ) Less(i, j int) bool { return pq[i].priority > pq[j].priority } // max first
func (pq PQ) Swap(i, j int)      { pq[i], pq[j] = pq[j], pq[i] }
func (pq *PQ) Push(x any)        { *pq = append(*pq, x.(Task)) }
func (pq *PQ) Pop() any {
    old := *pq
    n := len(old)
    t := old[n-1]
    *pq = old[:n-1]
    return t
}

func main() {
    pq := &PQ{}
    heap.Init(pq)
    heap.Push(pq, Task{"email", 1})
    heap.Push(pq, Task{"deploy", 5})
    heap.Push(pq, Task{"lunch", 3})

    for pq.Len() > 0 {
        t := heap.Pop(pq).(Task)
        fmt.Printf("%s(%d) ", t.name, t.priority)
    }
    fmt.Println()
}
Output
deploy(5) lunch(3) email(1)

Common Mistakes

Wrong
// Calling the methods directly skips
// the sift-up/down: the heap invariant
// is now broken.
h.Push(3)
v := h.Pop()
Right
// The package functions maintain order.
heap.Push(h, 3)
v := heap.Pop(h)

Key Rules

  • Implement Len, Less, Swap (value receiver fine) and Push/Pop (pointer receiver, they resize the slice).
  • Call heap.Init, heap.Push, heap.Pop, heap.Fix, heap.Remove — never the bare methods, which don't re-heapify.
  • Less with < is a min-heap; with > a max-heap. heap.Pop returns the root each time.
  • To change an element's priority in place, mutate it then call heap.Fix(h, index) — cheaper than remove + re-push.
Practice Exercise

Implement Dijkstra's shortest path on a small weighted graph using a container/heap priority queue of (node, dist) items, and use heap.Fix to decrease a node's key when a shorter path is found. Then solve "k-th largest element" by maintaining a min-heap of size k. Finally, explain why Push/Pop operate on the slice's tail even though heap.Pop conceptually removes the root (hint: the package swaps root to the end first).

Pitfall: Defining Push/Pop on a value receiver. They must mutate the slice header (append / re-slice), which only a pointer receiver can do; with a value receiver your appends vanish and the heap silently loses elements. Always use *T for Push and Pop, and pass &h to the heap functions.
Connections: The priority-ordering primitive complements the doubly linked list (container/list) from the previous guide — together they cover the "ordered by priority" and "ordered by recency/insertion" needs that a plain slice can't serve efficiently.
