Multiple Return Values

Functions & Control Flow • 4 days ago • rep 0

Quick Reminder

Go functions can return multiple values — most commonly (result, error). Named return values allow bare return statements. The comma-ok idiom is used for maps (val, ok := m[key]) and type assertions (val, ok := i.(Type)).

Key Pattern

func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("division by zero")
    }
    return a / b, nil
}
Challenge Exercise

Write a function parseConfig(path string) (Config, error) that reads a JSON file and returns a parsed struct or an error with context wrapping. Use fmt.Errorf with %w to wrap errors from os.Open and json.NewDecoder.

#1 Pitfall: Ignoring errors with _ — always handle or explicitly discard with a comment explaining why it is safe to ignore.

Interfaces

Go Foundations • 3 days ago • rep 1

Quick Reminder

Go interfaces are satisfied implicitly — no implements keyword. The empty interface any (alias for interface{}) accepts all types. Type assertions use val, ok := i.(ConcreteType). Interfaces should be small (1-2 methods) and defined where they are consumed, not where they are implemented.

Key Pattern

type Writer interface {
    Write(p []byte) (n int, err error)
}
// Any type with a Write method satisfies Writer automatically
Challenge Exercise

Define a Shape interface with Area() float64 and Perimeter() float64, implement it for Circle and Rectangle, then write a function that accepts []Shape and prints a summary table of each shape's area and perimeter.

#1 Pitfall: Pointer receivers don't satisfy interfaces for value types — if a method is defined on *T, then only *T implements the interface, not T.

Custom Constraints

Functions & Control Flow • 3 days ago • rep 0

Quick Reminder

Interface constraints limit what type parameters can be. The ~int syntax means "any type whose underlying type is int". Type sets define the union of allowed types using |. Combine method requirements with type sets for precise constraints.

Key Pattern

type Number interface {
    ~int | ~float64
}

func Sum[T Number](vals []T) T {
    var s T
    for _, v := range vals {
        s += v
    }
    return s
}
Challenge Exercise

Create a Stringable constraint that requires both a String() string method AND underlying type ~string | ~int, then write a generic Format function that calls String() on each element and joins them with commas.

#1 Pitfall: Forgetting ~ for underlying types — a bare int constraint will not match type MyInt int. Always use ~int when you want to accept named types.

Goroutines

Concurrency • 3 days ago • rep 0

Quick Reminder

go func() launches a goroutine. They are lightweight (~2KB initial stack). Use sync.WaitGroup to wait for completion. Goroutine leaks happen when goroutines block forever on a channel or lock that is never released.

Key Pattern

var wg sync.WaitGroup
for i := 0; i < 5; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        fmt.Println(id)
    }(i)
}
wg.Wait()
Challenge Exercise

Launch 10 goroutines that each compute the sum of a slice partition (divide a 100-element slice into 10 chunks). Use a sync.WaitGroup and a results slice to collect partial sums, then compute the total.

#1 Pitfall: Loop variable capture — in pre-Go 1.22 code, you must pass i as a parameter or shadow it with i := i inside the loop. Go 1.22+ fixes this with per-iteration loop variables.

Channels

Concurrency • 3 days ago • rep 0

Quick Reminder

Channels are typed conduits: ch := make(chan int). Unbuffered channels block until both sender and receiver are ready. Buffered channels make(chan int, 10) block only when full. Close with close(ch) and range over with for v := range ch.

Key Pattern

ch := make(chan int, 5)
go func() {
    for i := 0; i < 5; i++ {
        ch <- i
    }
    close(ch)
}()
for v := range ch {
    fmt.Println(v)
}
Challenge Exercise

Implement a producer-consumer pattern where 3 producers each send 10 ints to a shared channel and 1 consumer goroutine sums them all. Use a sync.WaitGroup to know when all producers are done, then close the channel.

#1 Pitfall: Sending on a closed channel panics. Only the sender should close the channel. Never close from the receiver side.

Select Statement

Concurrency • 3 days ago • rep 0

Quick Reminder

select multiplexes channel operations. If multiple cases are ready, one is chosen at random. A default case makes it non-blocking. Use time.After for timeouts.

Key Pattern

select {
case v := <-ch1:
    fmt.Println("from ch1:", v)
case v := <-ch2:
    fmt.Println("from ch2:", v)
case <-time.After(5 * time.Second):
    fmt.Println("timeout")
}
Challenge Exercise

Write a function that reads from two channels and returns whichever responds first, with a configurable timeout. The function signature should be func Race[T any](ch1, ch2 <-chan T, timeout time.Duration) (T, string, error) returning the value, which channel won, or a timeout error.

#1 Pitfall: time.After in a loop creates a new timer each iteration (memory leak). Use time.NewTimer and Reset() instead for repeated timeouts.

net/http: Handlers, ServeMux, Middleware, Client

Standard Library Mastery • New

Go is a top choice for HTTP services. The stdlib net/http package is production-grade — many projects use it without external frameworks. Understanding http.Handler, http.ServeMux, middleware patterns, and the HTTP client is essential for any Go backend developer.

Why It Matters

The http.Handler interface has a single method: ServeHTTP(w ResponseWriter, r *Request). The http.HandlerFunc adapter turns any func(w, r) into a Handler. Go 1.22+ added enhanced pattern matching to http.NewServeMux(). Middleware wraps handlers for cross-cutting concerns. The HTTP client must always have timeouts set.

Example 1: Simplest HTTP Server

package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "Hello, Go HTTP!")
    })

    http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        fmt.Fprintf(w, `{"status":"ok"}`)
    })

    fmt.Println("Server starting on :8080")
    http.ListenAndServe(":8080", nil)
}
Output (when visiting localhost:8080)
Hello, Go HTTP!

Example 2: Custom ServeMux with Multiple Routes

package main

import (
    "fmt"
    "net/http"
)

func main() {
    mux := http.NewServeMux()

    mux.HandleFunc("GET /api/users", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        fmt.Fprintf(w, `[{"name":"Alice"},{"name":"Bob"}]`)
    })

    mux.HandleFunc("GET /api/users/{id}", func(w http.ResponseWriter, r *http.Request) {
        id := r.PathValue("id")
        w.Header().Set("Content-Type", "application/json")
        fmt.Fprintf(w, `{"id":%q,"name":"User"}`, id)
    })

    mux.HandleFunc("POST /api/users", func(w http.ResponseWriter, r *http.Request) {
        w.WriteHeader(http.StatusCreated)
        fmt.Fprintf(w, `{"created":true}`)
    })

    fmt.Println("Server starting on :8080")
    http.ListenAndServe(":8080", mux)
}
Output (GET /api/users/42)
{"id":"42","name":"User"}

Example 3: Logging Middleware

package main

import (
    "fmt"
    "log"
    "net/http"
    "time"
)

// Middleware wraps an http.Handler and returns a new one
func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %s", r.Method, r.URL.Path, time.Since(start))
    })
}

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "Hello with logging!")
    })

    // Wrap the mux with middleware
    handler := LoggingMiddleware(mux)
    fmt.Println("Server starting on :8080")
    http.ListenAndServe(":8080", handler)
}
Output (server log)
2026/06/07 10:30:00 GET / 42.125µs

Example 4: JSON API Handler with Request Decoding

package main

import (
    "encoding/json"
    "fmt"
    "net/http"
    "time"
)

type StatusResponse struct {
    Status string `json:"status"`
    Time   string `json:"time"`
}

type CreateUserRequest struct {
    Name  string `json:"name"`
    Email string `json:"email"`
}

func main() {
    mux := http.NewServeMux()

    mux.HandleFunc("GET /status", func(w http.ResponseWriter, r *http.Request) {
        resp := StatusResponse{
            Status: "ok",
            Time:   time.Now().Format(time.RFC3339),
        }
        w.Header().Set("Content-Type", "application/json")
        json.NewEncoder(w).Encode(resp)
    })

    mux.HandleFunc("POST /users", func(w http.ResponseWriter, r *http.Request) {
        var req CreateUserRequest
        if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
            http.Error(w, `{"error":"invalid JSON"}`, http.StatusBadRequest)
            return
        }
        w.Header().Set("Content-Type", "application/json")
        w.WriteHeader(http.StatusCreated)
        json.NewEncoder(w).Encode(map[string]string{
            "message": fmt.Sprintf("Created user: %s", req.Name),
        })
    })

    fmt.Println("Server starting on :8080")
    http.ListenAndServe(":8080", mux)
}
Output (GET /status)
{"status":"ok","time":"2026-06-07T10:30:00Z"}

Example 5: HTTP Client with Timeout

package main

import (
    "fmt"
    "io"
    "net/http"
    "time"
)

func main() {
    // ALWAYS set a timeout on the client
    client := &http.Client{
        Timeout: 10 * time.Second,
    }

    resp, err := client.Get("https://httpbin.org/get")
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    defer resp.Body.Close() // ALWAYS close the body

    body, err := io.ReadAll(resp.Body)
    if err != nil {
        fmt.Println("Read error:", err)
        return
    }

    fmt.Printf("Status: %d\n", resp.StatusCode)
    fmt.Printf("Body length: %d bytes\n", len(body))
}
Output
Status: 200 Body length: 364 bytes

Common Mistakes

Do: Set timeouts on client and server
client := &http.Client{Timeout: 10 * time.Second}
server := &http.Server{
    Addr:         ":8080",
    ReadTimeout:  5 * time.Second,
    WriteTimeout: 10 * time.Second,
}
Don't: Use default client (no timeout!)
// Default client has NO timeout
resp, err := http.Get("https://example.com")
// Can hang forever if the server never responds
Do: Always close response body
resp, err := client.Get(url)
if err != nil { return err }
defer resp.Body.Close()
// ... use resp.Body
Don't: Forget to close the body
resp, _ := client.Get(url)
body, _ := io.ReadAll(resp.Body)
// Leaked connection! No Close() called

Key Rules

  • Always set timeouts on both HTTP clients and servers — the default client has no timeout
  • Always defer resp.Body.Close() after checking the error from the request
  • Use middleware for cross-cutting concerns: logging, auth, CORS, rate limiting
  • Go 1.22+ ServeMux supports method-based routing: "GET /api/users/{id}"
  • Use json.NewEncoder(w).Encode() for JSON responses — it writes directly to the ResponseWriter
Connection: Handlers use the Interfaces pattern (implicit satisfaction of http.Handler). JSON request/response handling ties to encoding/json. Middleware is a direct application of First-class Functions. Timeouts connect to the time Package and Context for cancellation.

Exercise 2

Write a logging middleware that logs the HTTP method, path, and request duration. Chain it with a simple handler and verify the log output.

Exercise 3

sort Package: Slice, SliceStable, Search, sort.Interface

Standard Library Mastery • New

Sorting is fundamental. Go's sort package offers both simple slice sorting and the sort.Interface for full control over custom types. Go 1.21+ added the slices package with generic, simpler alternatives.

Why It Matters

Every real application sorts data: leaderboards, search results, logs by timestamp, users by name. Understanding sort.Slice for quick sorts, sort.Interface for named sortable types, and sort.Search for binary search gives you the full toolkit.

Example 1: Sort Basic Slices

package main

import (
    "fmt"
    "sort"
)

func main() {
    nums := []int{5, 2, 8, 1, 9, 3}
    sort.Ints(nums)
    fmt.Println("Sorted ints:", nums)

    words := []string{"banana", "apple", "cherry", "date"}
    sort.Strings(words)
    fmt.Println("Sorted strings:", words)

    // Check if sorted
    fmt.Println("Is sorted?", sort.IntsAreSorted(nums))
}
Output
Sorted ints: [1 2 3 5 8 9] Sorted strings: [apple banana cherry date] Is sorted? true

Example 2: Sort Structs by Field

package main

import (
    "fmt"
    "sort"
)

type Person struct {
    Name string
    Age  int
}

func main() {
    people := []Person{
        {"Charlie", 30},
        {"Alice", 25},
        {"Bob", 35},
        {"Diana", 28},
    }

    // Sort by age ascending
    sort.Slice(people, func(i, j int) bool {
        return people[i].Age < people[j].Age
    })
    fmt.Println("By age:", people)

    // Sort by name alphabetically
    sort.Slice(people, func(i, j int) bool {
        return people[i].Name < people[j].Name
    })
    fmt.Println("By name:", people)
}
Output
By age: [{Alice 25} {Diana 28} {Charlie 30} {Bob 35}] By name: [{Alice 25} {Bob 35} {Charlie 30} {Diana 28}]

Example 3: Multi-key Sort and Stable Sort

package main

import (
    "fmt"
    "sort"
)

type Employee struct {
    Dept string
    Name string
    Age  int
}

func main() {
    employees := []Employee{
        {"Eng", "Charlie", 30},
        {"Eng", "Alice", 25},
        {"Sales", "Bob", 35},
        {"Eng", "Alice", 22},
        {"Sales", "Diana", 28},
    }

    // Multi-key sort: by department, then name, then age
    sort.SliceStable(employees, func(i, j int) bool {
        if employees[i].Dept != employees[j].Dept {
            return employees[i].Dept < employees[j].Dept
        }
        if employees[i].Name != employees[j].Name {
            return employees[i].Name < employees[j].Name
        }
        return employees[i].Age < employees[j].Age
    })

    for _, e := range employees {
        fmt.Printf("  %-6s %-8s %d\n", e.Dept, e.Name, e.Age)
    }
}
Output
Eng Alice 22 Eng Alice 25 Eng Charlie 30 Sales Bob 35 Sales Diana 28

Example 4: Binary Search with sort.Search

package main

import (
    "fmt"
    "sort"
)

func main() {
    data := []int{10, 20, 30, 40, 50, 60, 70, 80}

    // Find the insertion point for 45
    target := 45
    idx := sort.Search(len(data), func(i int) bool {
        return data[i] >= target
    })
    fmt.Printf("Insert %d at index %d\n", target, idx)
    fmt.Printf("Neighbors: ...%d, [%d here], %d...\n", data[idx-1], target, data[idx])

    // Find an existing element
    target = 30
    idx = sort.Search(len(data), func(i int) bool {
        return data[i] >= target
    })
    if idx < len(data) && data[idx] == target {
        fmt.Printf("Found %d at index %d\n", target, idx)
    }
}
Output
Insert 45 at index 4 Neighbors: ...40, [45 here], 50... Found 30 at index 2

Example 5: slices.SortFunc (Go 1.21+)

package main

import (
    "cmp"
    "fmt"
    "slices"
)

type Student struct {
    Name string
    GPA  float64
}

func main() {
    students := []Student{
        {"Alice", 3.8},
        {"Bob", 3.5},
        {"Charlie", 3.9},
        {"Diana", 3.5},
    }

    // Sort by GPA descending, then name ascending for ties
    slices.SortFunc(students, func(a, b Student) int {
        if c := cmp.Compare(b.GPA, a.GPA); c != 0 {
            return c // descending GPA
        }
        return cmp.Compare(a.Name, b.Name) // ascending name
    })

    for _, s := range students {
        fmt.Printf("  %-8s %.1f\n", s.Name, s.GPA)
    }
}
Output
Charlie 3.9 Alice 3.8 Bob 3.5 Diana 3.5

Common Mistakes

Do: Use strict less-than in comparators
sort.Slice(s, func(i, j int) bool {
    return s[i] < s[j] // strict <, never <=
})
Don't: Use less-than-or-equal
sort.Slice(s, func(i, j int) bool {
    return s[i] <= s[j] // WRONG: not strict weak
    // Can cause infinite loops or panics
})

Key Rules

  • Use slices.Sort/slices.SortFunc for new code (Go 1.21+) — they are generic and simpler
  • Use sort.Slice for older Go versions or when you need index-based access
  • Implement sort.Interface only when you need a named sortable type (rare)
  • sort.Search requires the slice to be sorted first — it performs binary search
  • The Less function must be a strict weak ordering: never use <=, always <
Connection: Sorting comparators are First-class Functions passed as arguments. sort.Interface uses the Interfaces pattern (three methods: Len, Less, Swap). slices.SortFunc leverages Type Parameters (Generics) from Go 1.18+. The cmp package ties to Custom Constraints with cmp.Ordered.
Exercise 1

Exercise 2

Implement sort.Interface for a ByLength type (a []string) that sorts strings by length ascending, breaking ties alphabetically. Verify it works with sort.Sort(ByLength(words)).

Exercise 3

Use sort.Search to find the insertion point for a value in a sorted slice of 1000 random ints. Verify the result is correct by checking that the element before the index is less than the target and the element at the index is greater or equal.

time Package: Duration, Timer, Ticker, Formatting

Standard Library Mastery • New

Time handling in Go is unique. Formatting uses a reference time (Mon Jan 2 15:04:05 MST 2006) instead of format strings like %Y-%m-%d. The digits in the reference time are specifically chosen: month=1, day=2, hour=15 (3pm), minute=04, second=05, year=2006, timezone=MST.

Why It Matters

Every application deals with time: timestamps, durations, scheduling, timeouts, rate limiting. Go's time package covers all of these. The reference-time formatting system is initially confusing but becomes natural once you memorize the magic numbers: 01/02 03:04:05PM '06 -0700.

Example 1: Measure Elapsed Time

package main

import (
    "fmt"
    "time"
)

func main() {
    start := time.Now()

    // Simulate some work
    total := 0
    for i := 0; i < 10_000_000; i++ {
        total += i
    }

    elapsed := time.Since(start)
    fmt.Printf("Computed sum: %d\n", total)
    fmt.Printf("Elapsed: %v\n", elapsed)
    fmt.Printf("Milliseconds: %.2f\n", elapsed.Seconds()*1000)
}
Output
Computed sum: 49999995000000 Elapsed: 4.208ms Milliseconds: 4.21

Example 2: Formatting and Parsing

package main

import (
    "fmt"
    "time"
)

func main() {
    now := time.Date(2026, 6, 7, 14, 30, 0, 0, time.UTC)

    // Format with reference time: Mon Jan 2 15:04:05 MST 2006
    fmt.Println("RFC 3339:", now.Format(time.RFC3339))
    fmt.Println("Custom:  ", now.Format("2006-01-02 15:04:05"))
    fmt.Println("Pretty:  ", now.Format("January 2, 2006"))
    fmt.Println("12-hour: ", now.Format("3:04 PM"))

    // Parse a date string
    t, err := time.Parse("2006-01-02", "2026-06-07")
    if err != nil {
        fmt.Println("Parse error:", err)
        return
    }
    fmt.Println("Parsed:  ", t)
}
Output
RFC 3339: 2026-06-07T14:30:00Z Custom: 2026-06-07 14:30:00 Pretty: June 7, 2026 12-hour: 2:30 PM Parsed: 2026-06-07 00:00:00 +0000 UTC

Example 3: Duration Arithmetic

package main

import (
    "fmt"
    "time"
)

func main() {
    // Duration constants
    d := 2*time.Hour + 30*time.Minute + 15*time.Second
    fmt.Println("Duration:", d)
    fmt.Printf("Hours: %.1f\n", d.Hours())
    fmt.Printf("Minutes: %.1f\n", d.Minutes())
    fmt.Printf("Seconds: %.0f\n", d.Seconds())

    // Time arithmetic
    now := time.Date(2026, 6, 7, 10, 0, 0, 0, time.UTC)
    later := now.Add(d)
    fmt.Println("Now:  ", now.Format("15:04:05"))
    fmt.Println("Later:", later.Format("15:04:05"))

    // Duration between two times
    deadline := time.Date(2026, 6, 10, 0, 0, 0, 0, time.UTC)
    remaining := time.Until(deadline) - time.Since(now)
    fmt.Printf("Remaining: ~%d hours\n", int(remaining.Hours()))
}
Output
Duration: 2h30m15s Hours: 2.5 Minutes: 150.2 Seconds: 9015 Now: 10:00:00 Later: 12:30:15 Remaining: ~62 hours

Example 4: Ticker for Periodic Work

package main

import (
    "fmt"
    "time"
)

func main() {
    ticker := time.NewTicker(500 * time.Millisecond)
    defer ticker.Stop() // ALWAYS stop the ticker

    done := make(chan bool)
    go func() {
        time.Sleep(2200 * time.Millisecond)
        done <- true
    }()

    count := 0
    for {
        select {
        case t := <-ticker.C:
            count++
            fmt.Printf("Tick %d at %s\n", count, t.Format("15:04:05.000"))
        case <-done:
            fmt.Printf("Done after %d ticks\n", count)
            return
        }
    }
}
Output
Tick 1 at 10:00:00.500 Tick 2 at 10:00:01.000 Tick 3 at 10:00:01.500 Tick 4 at 10:00:02.000 Done after 4 ticks

Example 5: Timer with Reset (Exponential Backoff)

package main

import (
    "fmt"
    "time"
)

func main() {
    backoff := 100 * time.Millisecond
    maxBackoff := 2 * time.Second
    timer := time.NewTimer(backoff)
    defer timer.Stop()

    for attempt := 1; attempt <= 5; attempt++ {
        <-timer.C
        fmt.Printf("Attempt %d after %v backoff\n", attempt, backoff)

        // Double the backoff, capped at max
        backoff *= 2
        if backoff > maxBackoff {
            backoff = maxBackoff
        }
        timer.Reset(backoff)
    }
    fmt.Println("All attempts complete")
}
Output
Attempt 1 after 100ms backoff Attempt 2 after 200ms backoff Attempt 3 after 400ms backoff Attempt 4 after 800ms backoff Attempt 5 after 1.6s backoff All attempts complete

Common Mistakes

Do: Use the reference time exactly
// Mon Jan 2 15:04:05 MST 2006
t.Format("2006-01-02")  // correct
t.Format("01/02/2006")  // correct
t.Format("15:04:05")    // correct (24h)
Don't: Use wrong reference digits
t.Format("2025-01-02")  // WRONG year
t.Format("2006-02-01")  // WRONG: month/day swapped
t.Format("14:04:05")    // WRONG hour digit
Do: Stop Tickers and Timers
ticker := time.NewTicker(time.Second)
defer ticker.Stop() // prevents goroutine leak
Don't: time.After in loops
for {
    select {
    case <-time.After(time.Second):
        // New timer EVERY iteration = memory leak
    }
}

Key Rules

  • Always use the reference time Mon Jan 2 15:04:05 MST 2006 for formatting — memorize it as 01/02 03:04:05 '06
  • Always defer ticker.Stop() — an unstopped ticker leaks a goroutine
  • Prefer time.NewTimer + Reset over time.After in loops to avoid timer leaks
  • Use time.Since(start) and time.Until(deadline) for readable duration calculations
  • Never store time.Duration as an integer — it is already a int64 of nanoseconds
Connection: Tickers and timers use Channels (ticker.C). Timeout patterns combine time.After with the Select Statement. HTTP server/client timeouts (net/http) are time.Duration values. Context deadlines wrap time.Time. Benchmarks (Benchmarks) use time.Duration for measurement.

Exercise 2

Exercise 3

Write a Ticker-based poller that calls a function every 5 seconds and stops after 30 seconds. Use time.NewTicker, time.After for the deadline, and a select loop. Count and print how many times the function was called.

testing Package: Table-Driven Tests, Subtests

Testing & Tooling • New

Go has a built-in testing framework — no external libraries needed. The go test command discovers and runs test functions automatically. Table-driven tests are THE idiomatic Go testing pattern, used extensively in the standard library itself.

Why It Matters

Tests are first-class citizens in Go. The test file lives alongside the code (foo_test.go), uses the same package, and requires only import "testing". Table-driven tests make it trivial to add new cases. Subtests with t.Run let you run individual cases and see clear failure messages.

Example 1: Simple Test

// math.go
package mathutil

func Add(a, b int) int {
    return a + b
}

func Abs(n int) int {
    if n < 0 {
        return -n
    }
    return n
}

// math_test.go
package mathutil

import "testing"

func TestAdd(t *testing.T) {
    got := Add(2, 3)
    want := 5
    if got != want {
        t.Errorf("Add(2, 3) = %d, want %d", got, want)
    }
}

func TestAbs(t *testing.T) {
    if Abs(-5) != 5 {
        t.Errorf("Abs(-5) = %d, want 5", Abs(-5))
    }
    if Abs(5) != 5 {
        t.Errorf("Abs(5) = %d, want 5", Abs(5))
    }
    if Abs(0) != 0 {
        t.Errorf("Abs(0) = %d, want 0", Abs(0))
    }
}

// fibonacci.go
package mathutil

func Fibonacci(n int) int {
    if n <= 0 {
        return 0
    }
    if n == 1 {
        return 1
    }
    a, b := 0, 1
    for i := 2; i <= n; i++ {
        a, b = b, a+b
    }
    return b
}

Example 3: Parallel Subtests

// string_test.go
package stringutil

import (
    "strings"
    "testing"
)

func Reverse(s string) string {
    runes := []rune(s)
    for i, j := 0, len(runes)-1; i < j; i, j = i+1, j-1 {
        runes[i], runes[j] = runes[j], runes[i]
    }
    return string(runes)
}

func TestReverse(t *testing.T) {
    tests := []struct {
        name  string
        input string
        want  string
    }{
        {"empty", "", ""},
        {"single", "a", "a"},
        {"word", "hello", "olleh"},
        {"palindrome", "racecar", "racecar"},
        {"unicode", "Hello, 世界", "界世 ,olleH"},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            t.Parallel() // run this subtest in parallel
            got := Reverse(tt.input)
            if got != tt.want {
                t.Errorf("Reverse(%q) = %q, want %q", tt.input, got, tt.want)
            }
        })
    }

    // Also test the roundtrip property
    t.Run("roundtrip", func(t *testing.T) {
        t.Parallel()
        input := "Go is great"
        if Reverse(Reverse(input)) != input {
            t.Error("double reverse should return original")
        }
    })

    _ = strings.Contains // just to show the import is used
}

Common Mistakes

Do: Use t.Helper() in test helpers
func assertEqual(t *testing.T, got, want int) {
    t.Helper() // error line points to caller
    if got != want {
        t.Errorf("got %d, want %d", got, want)
    }
}
Don't: Forget t.Helper()
func assertEqual(t *testing.T, got, want int) {
    // Without t.Helper(), errors point HERE
    // instead of the actual test line
    if got != want {
        t.Errorf("got %d, want %d", got, want)
    }
}
Do: Use t.Error in goroutines
t.Run("concurrent", func(t *testing.T) {
    go func() {
        // t.Error is safe from goroutines
        if result != expected {
            t.Error("mismatch")
        }
    }()
})
Don't: Use t.Fatal in goroutines
go func() {
    // t.Fatal calls runtime.Goexit()
    // Only safe in the test goroutine!
    if err != nil {
        t.Fatal(err) // PANIC or wrong goroutine exit
    }
}()

Key Rules

  • Use table-driven tests as the default pattern for any function with multiple input/output combinations
  • Use t.Run("name", ...) for every test case — it gives named subtests and allows -run filtering
  • Use t.Helper() in ALL helper functions so error lines point to the calling test
  • Use t.Parallel() when tests are independent — it speeds up test suites significantly
  • Use t.Error (not t.Fatal) in goroutines — t.Fatal calls runtime.Goexit and is only safe in the test goroutine
Connection: Test helpers use Generics for assertEq[T comparable]. t.Cleanup is conceptually similar to Defer, Panic, Recover (LIFO execution). Table-driven tests iterate with range over a slice of Structs. Parallel subtests connect to Goroutines — each subtest runs in its own goroutine.

Benchmarks: b.N, b.ResetTimer, b.ReportAllocs

Testing & Tooling • New

Go's built-in benchmarking lets you measure performance precisely. go test -bench=. runs benchmarks automatically. The framework controls iteration count (b.N) to get statistically stable results.

Why It Matters

Performance claims without benchmarks are opinions. Go makes benchmarking trivially easy — same file, same tool, same workflow as tests. Use benchmarks to compare algorithms, detect regressions, and measure memory allocations.

Example 1: Basic Benchmark

// concat_test.go
package stringutil

import (
    "strings"
    "testing"
)

func ConcatPlus(strs []string) string {
    result := ""
    for _, s := range strs {
        result += s
    }
    return result
}

func ConcatBuilder(strs []string) string {
    var b strings.Builder
    for _, s := range strs {
        b.WriteString(s)
    }
    return b.String()
}

func BenchmarkConcatPlus(b *testing.B) {
    strs := []string{"hello", " ", "world", " ", "from", " ", "go"}
    for i := 0; i < b.N; i++ {
        ConcatPlus(strs)
    }
}

    strs := []string{"hello", " ", "world", " ", "from", " ", "go"}
    for i := 0; i < b.N; i++ {

    }

    "fmt"

    "testing"

func BenchmarkSort(b *testing.B) {
    sizes := []int{100, 1000, 10000}

    for _, size := range sizes {
        b.Run(fmt.Sprintf("size=%d", size), func(b *testing.B) {
            // Create the source data once
            src := make([]int, size)
            for i := range src {
                src[i] = size - i // reverse-sorted (worst case)
            }

            b.ResetTimer()
            for i := 0; i < b.N; i++ {
                // Copy so each iteration sorts unsorted data
                data := make([]int, size)
                copy(data, src)
                sort.Ints(data)
            }
        })
    }
}

Key Rules

  • Always use b.N for the loop count — the framework determines the right number of iterations
  • Use b.ResetTimer() after expensive setup to exclude it from the measurement
  • Store results to a package-level variable to prevent the compiler from optimizing away dead code
  • Run with -count=5 or more for statistical significance — use benchstat to compare runs
  • Use b.ReportAllocs() or -benchmem flag to always include allocation data
Connection: Benchmarks live in _test.go files alongside testing Package tests. b.ResetTimer relates to time Package measurement. Sub-benchmarks mirror subtests from t.Run. Test Fixtures: TestMain, t.TempDir, t.Cleanup

Testing & Tooling • New

Complex tests need setup and teardown. Go provides TestMain for package-level setup, t.TempDir() for auto-cleaned temporary directories, t.Cleanup() for test-scoped resource management, and t.Setenv() for safe environment variable testing.

Why It Matters

Real-world tests need databases, temp files, mock servers, and environment variables. Go's built-in fixtures make setup and teardown reliable and automatic — no need for external frameworks. t.TempDir and t.Cleanup guarantee resources are freed even when tests fail.

Example 1: TestMain for Package-level Setup/Teardown

// main_test.go
package myapp

import (
    "fmt"
    "net/http"
    "net/http/httptest"
    "os"
    "testing"
)

var testServer *httptest.Server

func TestMain(m *testing.M) {
    // Package-level setup: start a mock server
    fmt.Println("Setting up test server...")
    testServer = httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, `{"status":"mock"}`)
    }))

    // Run all tests in this package
    code := m.Run()

    // Package-level teardown
    fmt.Println("Tearing down test server...")
    testServer.Close()

    os.Exit(code) // MUST call os.Exit with the test result
}

func TestServerReachable(t *testing.T) {
    resp, err := http.Get(testServer.URL)
    if err != nil {
        t.Fatalf("server not reachable: %v", err)
    }
    defer resp.Body.Close()
    if resp.StatusCode != 200 {
        t.Errorf("got status %d, want 200", resp.StatusCode)
    }
    fmt.Printf("Test server at: %s\n", testServer.URL)
}
Output (go test -v -run TestServer)

Example 2: t.TempDir for Auto-Cleaned Temp Files

// fileutil.go
package fileutil

import "os"

func WriteFile(path, content string) error {
    return os.WriteFile(path, []byte(content), 0644)
}

func ReadFile(path string) (string, error) {
    data, err := os.ReadFile(path)
    return string(data), err
}

// fileutil_test.go
package fileutil

import (
    "path/filepath"
    "testing"
)

func TestWriteAndRead(t *testing.T) {
    // t.TempDir() creates a temp dir that is automatically
    // cleaned up when the test finishes (even on failure)
    dir := t.TempDir()

    path := filepath.Join(dir, "test.txt")
    content := "hello, temp file"

    if err := WriteFile(path, content); err != nil {
        t.Fatalf("WriteFile failed: %v", err)
    }

    got, err := ReadFile(path)
    if err != nil {
        t.Fatalf("ReadFile failed: %v", err)
    }

    if got != content {
        t.Errorf("got %q, want %q", got, content)
    }
    // No cleanup needed — dir is auto-removed!
}

func TestMultipleFiles(t *testing.T) {
    dir := t.TempDir() // each test gets its own temp dir

    files := map[string]string{
        "a.txt": "alpha",
        "b.txt": "beta",
        "c.txt": "gamma",
    }

    for name, content := range files {
        path := filepath.Join(dir, name)
        if err := WriteFile(path, content); err != nil {
            t.Fatalf("WriteFile(%s) failed: %v", name, err)
        }
    }

    for name, want := range files {
        path := filepath.Join(dir, name)
        got, err := ReadFile(path)
        if err != nil {
            t.Fatalf("ReadFile(%s) failed: %v", name, err)
        }
        if got != want {
            t.Errorf("file %s: got %q, want %q", name, got, want)
        }
    }
}

package renderer

    "flag"
    "fmt"
    "os"
    "path/filepath"
    "testing"

var update = flag.Bool("update", false, "update golden files")

func Render(name string, items []string) string {

        result += fmt.Sprintf("%d. %s\n", i+1, item)

}

        t.Log("Updated golden file")
        return
    }

    want, err := os.ReadFile(golden)
    if err != nil {
        t.Fatalf("Failed to read golden file (run with -update to create): %v", err)
    }

    if got != string(want) {
        t.Errorf("output mismatch:\ngot:\n%s\nwant:\n%s", got, string(want))
    }
}

Common Mistakes

Do: Call os.Exit in TestMain
func TestMain(m *testing.M) {
    // setup...
    code := m.Run()
    // teardown...
    os.Exit(code) // REQUIRED
}
Don't: Forget os.Exit
func TestMain(m *testing.M) {
    // setup...
    m.Run()
    // Tests silently PASS even if they fail!
    // Without os.Exit, the exit code is lost
}
Do: Use t.TempDir()
dir := t.TempDir() // auto-cleaned
path := filepath.Join(dir, "test.txt")
Don't: Create temp files manually
dir, _ := os.MkdirTemp("", "test")
// Forgot defer os.RemoveAll(dir)
// Leaves garbage in /tmp!

Key Rules

  • Use TestMain only for expensive package-level setup (databases, servers) — not for per-test setup
  • Always call os.Exit(m.Run()) in TestMain — without it, failures are silently ignored
  • Use t.TempDir() over manual temp files — it auto-cleans even on test failure
  • Use t.Cleanup() over defer for test-scoped resources — it is more explicit and composable
  • Store test data in a testdata/ directory — go tool ignores it during builds
  • Cleanup functions run in LIFO order, just like defer
Connection: TestMain often starts mock servers using net/http's httptest. t.TempDir works with os & filepath for file-based tests. t.Cleanup mirrors the LIFO semantics of Defer, Panic, Recover. Golden file testing uses encoding/json or io Package for structured output comparison.
Exercise 1

Write a TestMain that sets up a mock HTTP server and tears it down. Use httptest.NewServer and have two test functions that make requests to it.

Exercise 2

Write a test that creates files in t.TempDir() and verifies a file-processing function (e.g., one that counts lines across multiple files). Ensure no temp files remain after the test.

Exercise 3

Implement golden file testing for a template renderer. The renderer should take a name and a map of key-value pairs and produce formatted output. Support the -update flag to regenerate golden files.
