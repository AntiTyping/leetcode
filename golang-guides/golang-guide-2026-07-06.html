Module Management: go.mod, Workspaces & Versioning

Production Go • due today • rep 1

Quick Reminder

A module is a versioned set of packages rooted at a go.mod file (module path + dependency requirements). go get adds/updates a requirement, go mod tidy prunes unused ones and syncs go.sum checksums. Semantic Import Versioning puts the major in the path for v2+ (.../v2). For local multi-module work, a go.work file (Go 1.18+) joins several modules without touching each go.mod or using replace.

Key Pattern

# module lifecycle
go mod init example.com/app          # create go.mod
go get github.com/google/uuid@v1.6.0 # pin a version
go mod tidy                          # sync require + go.sum
go list -m all                       # full dependency graph

# a workspace over two local modules (go.work, git-ignored)
go work init ./app ./lib
go work use ./newmod
Challenge Exercise

Publish a library at v1.2.0, then make a breaking change and cut v2: update the module path to end in /v2, tag v2.0.0, and fix an importing app so both a v1 and a v2 of the same library can coexist in one build. Use go mod graph and go mod why to explain how a transitive dependency got pulled in. Finally, contrast a committed replace directive against a local go.work file, and state when each is the right tool.

#1 Pitfall: Committing a go.work file to a shared repo — it silently overrides everyone's module resolution and breaks CI expecting tagged versions. Keep it local (git-ignore it). Also, forgetting the /v2 suffix on a major bump makes Go treat it as the same module and produces baffling "ambiguous import" errors.

Cross-Compilation & Build Tags

Production Go • due today • rep 1

Quick Reminder

Go cross-compiles to any target by setting GOOS/GOARCH — no cross toolchain needed as long as you stay pure Go (CGO_ENABLED=0). Build constraints pick which files compile for a target: a filename suffix (_windows.go, _linux.go) or a //go:build line at the top of a file. That's how OS-specific code stays isolated while you still ship one binary per target.

Key Pattern

//go:build linux || darwin

package app
// Compiled only on Linux/macOS. A sibling file with
// "//go:build windows" holds the Windows version.
# static Linux arm64 binary from any host
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -o app .
go tool dist list        # every supported GOOS/GOARCH pair
go build -tags prod .     # enable files gated by //go:build prod
Challenge Exercise

Write a ConfigDir() split across config_windows.go and config_unix.go using //go:build constraints so it returns %APPDATA% on Windows and $XDG_CONFIG_HOME (falling back to ~/.config) elsewhere. Add a debug tag that enables extra logging only under -tags debug. Cross-compile for linux/amd64, darwin/arm64, and windows/amd64, verify each with file, and explain why CGO_ENABLED=0 makes cross-builds painless.

#1 Pitfall: Leaving CGO_ENABLED=1 while cross-compiling — the build then needs a target C cross-compiler and often fails or yields a dynamically-linked binary that won't run in a scratch container. Also, a malformed //go:build line (or the old // +build without a trailing blank line) is silently ignored, so the file compiles everywhere unexpectedly.

WebSocket in Go

Advanced Networking • due today • rep 1

Quick Reminder

WebSockets upgrade an HTTP request into a persistent, full-duplex TCP connection so both sides can push messages any time. The standard library has no WebSocket server, so you use github.com/coder/websocket (formerly nhooyr) or gorilla/websocket. Inside an http.HandlerFunc you call Accept to complete the upgrade, then read and write typed (text/binary) frames until either side closes.

Key Pattern

func echo(w http.ResponseWriter, r *http.Request) {
    c, err := websocket.Accept(w, r, nil) // upgrade
    if err != nil {
        return
    }
    defer c.Close(websocket.StatusInternalError, "bye")

    ctx, cancel := context.WithTimeout(r.Context(), time.Minute)
    defer cancel()
    for {
        typ, data, err := c.Read(ctx) // blocks for next frame
        if err != nil {
            return // client closed or timed out
        }
        c.Write(ctx, typ, data) // echo it back
    }
}
Challenge Exercise

Build a broadcast chat hub: one central goroutine owns the set of connections and fans each incoming message out to every client through a buffered per-client send channel, so one slow reader can't stall everyone. Add ping/keepalive and drop clients that miss a pong deadline. Handle graceful shutdown by closing all connections with websocket.StatusGoingAway. Explain why you must never write to a single connection from two goroutines at once.

#1 Pitfall: Writing to one WebSocket connection from multiple goroutines simultaneously — frames interleave and corrupt the stream. Funnel all writes for a connection through a single owner goroutine or send channel. Skipping an origin check or a read deadline also opens you to cross-site hijacking and slow-loris resource exhaustion.

Profiling with pprof

Observability & Profiling • due today • rep 1

Quick Reminder

pprof collects and visualizes CPU, heap, goroutine, block, and mutex profiles. In a server, blank-import net/http/pprof to expose profiles under /debug/pprof/; for one-off runs, use the benchmark flags -cpuprofile/-memprofile. Analyze with go tool pprof: top finds the hottest functions, list Func gives line-level cost, and -http=:0 opens an interactive flame graph.

Key Pattern

import _ "net/http/pprof" // registers /debug/pprof/ handlers

func main() {
    // profiles at http://localhost:6060/debug/pprof/
    go http.ListenAndServe("localhost:6060", nil)
    // ... rest of the server ...
}
# 30s CPU profile from a live server, then a flame graph
go tool pprof -http=:0 \
    http://localhost:6060/debug/pprof/profile?seconds=30

# heap profile from a benchmark
go test -bench=. -memprofile=mem.out
go tool pprof -http=:0 mem.out   # (pprof) top10 ; list MyFunc
Challenge Exercise

Write a deliberately wasteful function (string concatenation in a loop plus needless allocations), expose net/http/pprof, and drive load from a goroutine. Capture a CPU and a heap profile, use top and list to find the hot line, fix it (strings.Builder, preallocation), and confirm the profile improved. Then take a goroutine profile while goroutines block on a channel and explain how it reveals a leak. Note why block/mutex profiles need runtime.SetBlockProfileRate/SetMutexProfileFraction.

#1 Pitfall: Exposing /debug/pprof/ on a public interface — profiles leak internals and let anyone trigger expensive captures (a DoS vector). Bind it to localhost or an internal port. Also, profiling an unrepresentative workload (too short, no load) gives noise; profile under realistic traffic long enough to gather meaningful samples.

OpenTelemetry & Tracing

Observability & Profiling • due today • rep 1

Quick Reminder

Distributed tracing follows one request across services as a trace of nested spans. OpenTelemetry is the vendor-neutral standard: configure a TracerProvider with an exporter (OTLP, Jaeger, stdout for dev), get a Tracer, and wrap work with tracer.Start(ctx, "name"). The returned context.Context carries the active span, so propagating ctx down the call chain (and across HTTP/gRPC via propagators) is what stitches spans into one trace.

Key Pattern

func handleOrder(ctx context.Context, id string) error {
    ctx, span := otel.Tracer("orders").Start(ctx, "handleOrder")
    defer span.End() // always end the span

    span.SetAttributes(attribute.String("order.id", id))
    if err := charge(ctx, id); err != nil { // child span sees ctx
        span.RecordError(err)
        span.SetStatus(codes.Error, err.Error())
        return err
    }
    return nil
}
Challenge Exercise

Set up a TracerProvider with the stdout exporter and instrument a two-function chain so the second span nests under the first — prove nesting by threading ctx through. Add attributes and an event, and record an error path with span.RecordError + span.SetStatus. Then wrap an http.Handler with otelhttp and its matching client transport so trace context propagates across a real HTTP call. Explain why forgetting defer span.End() leaks unfinished spans.

#1 Pitfall: Not propagating ctx — calling the next function with context.Background() instead of the span-carrying context starts a brand-new trace and severs the parent/child link. Also, forgetting defer span.End() means spans never flush, and a span per trivial call floods the backend — match span granularity to meaningful units of work.

Embedding & go:embed

Observability & Profiling • overdue (23d) • rep 0

Quick Reminder

The //go:embed directive bakes files or whole directories into the compiled binary at build time — no runtime file I/O, single self-contained artifact. The directive comment must sit directly above a package-level variable of type string, []byte, or embed.FS, and you must import embed (blank import _ "embed" is enough when embedding into string/[]byte). Paths are relative to the .go file, always use forward slashes, and cannot escape the module with ...

Key Pattern

import "embed"

//go:embed templates/*.html
var tmplFS embed.FS   // a read-only fs.FS of matched files

//go:embed version.txt
var version string    // file contents as a string

func main() {
    b, _ := tmplFS.ReadFile("templates/index.html")
    _ = b
    fmt.Println("version:", version)
}
Challenge Exercise

Embed a directory of static assets into an embed.FS and serve it with http.FileServer(http.FS(assets)); embed a set of HTML templates and parse them with template.ParseFS. Then embed a build version string and print it at startup. Explain why //go:embed paths must use forward slashes even on Windows (tie this to today's path vs path/filepath concept), and describe the difference between embedding into a string/[]byte versus an embed.FS.

#1 Pitfall: A blank line or another comment between the //go:embed directive and the var declaration — the directive is then ignored or rejected at compile time; it must be immediately adjacent. Also, embedding with a .. or absolute path is disallowed, and forgetting to import embed when using embed.FS fails to compile.

Section 1 — Binary, Numbers & Paths

encoding/binary: Byte Order & Binary Framing

Wire Formats

Why it matters: network protocols, file formats, and on-disk records need a fixed, agreed byte layout — not Go's self-describing gob or JSON. encoding/binary reads and writes fixed-width numeric values in a chosen byte order, and encodes compact variable-length integers (varints). It's how you build a custom TCP protocol, parse a binary file header, or length-prefix messages on a stream.

Fixed-width values with an explicit byte order

binary.Write/binary.Read serialize fixed-size values through a ByteOrder. Use BigEndian (network order) or LittleEndian — both ends must agree.

package main

import (
    "bytes"
    "encoding/binary"
    "fmt"
)

func main() {
    var buf bytes.Buffer
    binary.Write(&buf, binary.BigEndian, uint16(1000))  // 2 bytes
    binary.Write(&buf, binary.BigEndian, uint32(70000)) // 4 bytes
    fmt.Printf("% x\n", buf.Bytes())

    var a uint16
    var b uint32
    binary.Read(&buf, binary.BigEndian, &a)
    binary.Read(&buf, binary.BigEndian, &b)
    fmt.Println(a, b)
}
Output
03 e8 00 01 11 70 1000 70000

Packing a fixed-layout struct

A struct of fixed-size fields serializes field-by-field in order — perfect for a binary header. binary.Size reports the encoded length.

package main

import (
    "bytes"
    "encoding/binary"
    "fmt"
)

type header struct {
    Magic   uint32
    Version uint16
    Flags   uint16
}

func main() {
    h := header{Magic: 0xCAFEBABE, Version: 2, Flags: 0x0001}
    var buf bytes.Buffer
    binary.Write(&buf, binary.BigEndian, h)
    fmt.Printf("% x\n", buf.Bytes())
    fmt.Println("size:", binary.Size(h))
}
Output
ca fe ba be 00 02 00 01 size: 8

Compact integers with varints

When most values are small, a varint uses fewer bytes than a fixed width. binary.AppendUvarint encodes, binary.Uvarint decodes and returns how many bytes it consumed.

package main

import (
    "encoding/binary"
    "fmt"
)

func main() {
    buf := make([]byte, 0, 8)
    for _, v := range []uint64{1, 300, 70000} {
        buf = binary.AppendUvarint(buf, v)
    }
    fmt.Printf("% x\n", buf)

    rest := buf
    for len(rest) > 0 {
        v, n := binary.Uvarint(rest)
        fmt.Print(v, " ")
        rest = rest[n:]
    }
    fmt.Println()
}
Output
01 ac 02 f0 a2 04 1 300 70000

Common Mistakes

Wrong
type rec struct {
    ID   int      // platform-sized: rejected
    Name string   // variable length: rejected
}
var buf bytes.Buffer
// Returns an error you might ignore -> nothing written.
binary.Write(&buf, binary.BigEndian, rec{1, "x"})
Right
type rec struct {
    ID uint64      // fixed width
}
var buf bytes.Buffer
binary.Write(&buf, binary.BigEndian, rec{ID: 1})
// Encode strings yourself: length prefix + raw bytes.
binary.Write(&buf, binary.BigEndian, uint16(len(name)))
buf.WriteString(name)

Key Rules

  • binary.Write/Read handle only fixed-size data: fixed-width numeric types, arrays, and structs of those — never int, string, slice, or map.
  • Always choose an explicit ByteOrder: BigEndian for on-the-wire/network formats, LittleEndian for most CPU-native file formats — and use the same on both ends.
  • For small, self-delimiting integers use Uvarint (unsigned) or Varint (zig-zag signed) instead of a fixed width.
  • binary.Size(v) returns the encoded byte length, or -1 for a non-fixed type — use it to preallocate buffers.
Practice Exercise

Implement length-prefixed framing: a writer that emits a uint32 big-endian length followed by the payload, and a reader that decodes one frame at a time from an io.Reader until io.EOF. Then define a fixed-layout record struct, round-trip it through binary.Write/Read, and add a varint-encoded variant — compare the byte sizes for small vs large values. Finally, explain when you'd choose big-endian over little-endian and why passing a struct with an int field to binary.Write fails.

Pitfall: Passing a non-fixed-size type (a struct containing int, string, or a slice) to binary.Write returns an error rather than encoding — int is rejected because its width is platform-dependent. Ignoring that error means you silently write nothing. Use explicit widths (int32/uint64) and length-prefix any variable data yourself.
Connections: The fixed-layout counterpart to self-describing encoding/gob and text-based JSON/hex; builds on bytes.Buffer, io.Reader/Writer, and bufio; it's the foundation for custom TCP protocols (from the TCP servers day) and for parsing binary file headers.

math: Floating-Point, Inf, NaN & Rounding

Numbers

Why it matters: any numeric code beyond integer arithmetic — geometry, statistics, scoring, graphics — leans on math for constants, roots, powers, rounding modes, and the IEEE-754 special values infinity and NaN. Handling those specials correctly (you cannot compare NaN with ==) separates code that quietly produces garbage from code that stays correct.

Constants and common functions

math provides Pi, E, Sqrt, Pow, Abs, and Max/Min (which operate on float64 and propagate NaN, unlike the Go 1.21 builtins).

package main

import (
    "fmt"
    "math"
)

func main() {
    fmt.Println(math.Sqrt(144))                 // 12
    fmt.Println(math.Pow(2, 10))                // 1024
    fmt.Printf("%.4f\n", math.Pi)               // 3.1416
    fmt.Println(math.Max(3, 9), math.Min(3, 9)) // 9 3
    fmt.Println(math.Abs(-4.5))                 // 4.5
}
Output
12 1024 3.1416 9 3 4.5

Rounding modes matter

There is no single "round" — pick the mode deliberately. Round goes half away from zero; RoundToEven is banker's rounding; Floor/Ceil/Trunc are directional.

package main

import (
    "fmt"
    "math"
)

func main() {
    fmt.Println(math.Floor(2.5), math.Ceil(2.5))       // 2 3
    fmt.Println(math.Round(2.5), math.Round(-2.5))     // 3 -3
    fmt.Println(math.Trunc(2.9))                        // 2
    fmt.Println(math.RoundToEven(2.5), math.RoundToEven(3.5)) // 2 4
}
Output
2 3 3 -3 2 2 4

Inf and NaN: detect, never compare

Dividing a nonzero float by zero yields ±Inf; 0.0/0.0 (via variables) yields NaN. NaN is not equal to anything — even itself — so you must use math.IsNaN/math.IsInf.

package main

import (
    "fmt"
    "math"
)

func main() {
    zero := 0.0
    fmt.Println(math.IsInf(1/zero, 1)) // true  (+Inf)

    n := math.NaN()
    fmt.Println(n == n)         // false! NaN != NaN
    fmt.Println(math.IsNaN(n))  // true
    fmt.Println(math.MaxInt64)  // 9223372036854775807
}
Output
true false true 9223372036854775807

Common Mistakes

Wrong
// NaN never equals anything, so this misses it:
if result == math.NaN() { /* never true */ }

// Exact equality on computed floats is fragile:
if a*b == 0.06 { /* often false */ }
Right
if math.IsNaN(result) { /* handle it */ }

// Compare within a tolerance:
const eps = 1e-9
if math.Abs(a*b-0.06) < eps { /* close enough */ }

Key Rules

  • Detect specials with math.IsNaN and math.IsInf — never ==, since NaN compares false to everything including itself.
  • math.Round rounds half away from zero; math.RoundToEven does banker's rounding — choose deliberately for money/statistics.
  • math.Max/Min take and return float64 with NaN semantics; for ordered non-float types prefer the Go 1.21 builtins min/max.
  • Floats are binary approximations: never use == on computed results — compare within an epsilon, and use integers or a decimal library for currency.
Practice Exercise

Write nearlyEqual(a, b, eps float64) bool using math.Abs, and a classify(x float64) string that returns "nan", "+inf", "-inf", "zero", or "normal" using the math predicates. Then implement a numerically careful mean of a slice that returns an error (not NaN) for an empty slice, and detect integer overflow of a running sum against math.MaxInt64. Finally, print why 0.1 + 0.2 != 0.3 in Go and how you'd test that sum safely.

Pitfall: Comparing a result to math.NaN() with == (always false, so the branch never runs) or relying on exact == for computed floats. Also assuming math.Max/Min behave like the integer builtins — they're float64-only and treat NaN specially. Use the predicates and an epsilon, and reach for the builtins on integers.
Connections: Complements math/bits (integer bit ops), math/big (arbitrary precision), and math/rand/v2; pairs with the Go 1.21 builtins min/max/clear and with fmt float verbs (%f, %g, %e) for formatting.

path: Slash-Separated Path Manipulation

Std Lib

Why it matters: URLs, io/fs and embed.FS keys, and import-style paths are always slash-separated — on every OS. The path package manipulates those forward-slash paths, distinct from path/filepath, which adapts to the local OS separator. Using filepath to build a URL or an fs.FS key works on Unix but silently breaks on Windows (backslashes).

Core operations

Base, Dir, Ext, Join, and Clean all operate lexically on slash paths — no filesystem access.

package main

import (
    "fmt"
    "path"
)

func main() {
    fmt.Println(path.Base("/a/b/c.txt"))        // c.txt
    fmt.Println(path.Dir("/a/b/c.txt"))         // /a/b
    fmt.Println(path.Ext("/a/b/c.txt"))         // .txt
    fmt.Println(path.Join("a", "b", "..", "c")) // a/c
    fmt.Println(path.Clean("/a//b/../c/"))      // /a/c
}
Output
c.txt /a/b .txt a/c /a/c

Split, IsAbs, and glob matching

Split divides into dir + final element; IsAbs tests a leading slash; Match does shell-style globbing on a single path element.

package main

import (
    "fmt"
    "path"
)

func main() {
    dir, file := path.Split("/images/logo.png")
    fmt.Printf("%q %q\n", dir, file)                 // "/images/" "logo.png"
    fmt.Println(path.IsAbs("/x"), path.IsAbs("x"))   // true false
    ok, _ := path.Match("*.png", "logo.png")
    fmt.Println(ok)                                  // true
}
Output
"/images/" "logo.png" true false true

path vs path/filepath

Same call, different rule: path always uses /; filepath uses the OS separator (\ on Windows). Pick by what the path represents, not where your code runs.

package main

import (
    "fmt"
    "path"
    "path/filepath"
)

func main() {
    // For URLs and io/fs / embed keys, ALWAYS use path:
    fmt.Println(path.Join("static", "css", "app.css"))
    // -> static/css/app.css  (every OS)

    // For real local files, use filepath (OS-specific):
    fmt.Println(filepath.Join("static", "css", "app.css"))
    // -> static/css/app.css on Unix, static\css\app.css on Windows
}
Output
static/css/app.css static/css/app.css

Common Mistakes

Wrong
// Building an fs.FS / embed key with filepath:
key := filepath.Join("templates", name)
// "templates\index.html" on Windows -> lookup fails
tmplFS.ReadFile(key)
Right
// fs.FS keys are always slash paths:
key := path.Join("templates", name)
// "templates/index.html" everywhere
tmplFS.ReadFile(key)

Key Rules

  • Use path for slash-separated virtual paths: URLs, io/fs / embed.FS keys, and import-like paths — always forward slashes on every OS.
  • Use path/filepath for the local filesystem — it adapts to the platform separator. Never mix the two.
  • path.Clean/Join resolve . and .. lexically without touching disk — useful to sanitize untrusted URL paths (but still validate against a root).
  • There's no path.Abs or volume handling — "absolute" and drive letters are OS concepts that live in filepath.
Practice Exercise

Write a handler that takes an untrusted URL path, runs path.Clean, and rejects anything that would escape a served root (a directory-traversal guard) — then serve the file from an embed.FS using path.Join keys. Contrast it with a local disk walker built on filepath. Finally, show what path.Join("a", "..", "..") returns and explain why lexical cleaning alone is not a complete traversal defense.

Pitfall: Using filepath to construct keys for embed.FS or io/fs (which require forward slashes) — it works on Unix and silently fails on Windows. And remember path.Clean is purely lexical: it collapses .. but won't stop a cleaned ../../etc/passwd from escaping if you concatenate onto a root instead of validating the result stays within it.
Connections: The slash-path sibling of path/filepath (the OS filesystem days); essential for io/fs, go:embed (today's review), net/url paths, and http.FileServer/http.FS serving.

Section 2 — Concurrency & Networking

sync.Cond: Condition Variables

Concurrency

Why it matters: sometimes goroutines must wait for a shared state to change — a queue becomes non-empty, a resource frees up, a "start" flag flips — without burning CPU on a polling loop. A sync.Cond pairs a mutex with a wait/notify mechanism: waiters sleep until another goroutine signals a change. It's the classic tool when many waiters observe complex shared state that doesn't map cleanly onto a single channel.

Wait and Signal

cond.Wait() atomically unlocks the mutex, blocks, and re-locks before returning. Signal wakes one waiter. Always guard the predicate in a for loop.

package main

import (
    "fmt"
    "sync"
)

func main() {
    var mu sync.Mutex
    cond := sync.NewCond(&mu)
    ready := false

    go func() {
        mu.Lock()
        ready = true
        cond.Signal() // wake one waiter
        mu.Unlock()
    }()

    mu.Lock()
    for !ready {      // re-check in a loop, not an if
        cond.Wait()   // unlocks, sleeps, re-locks on wake
    }
    mu.Unlock()
    fmt.Println("ready:", ready)
}
Output
ready: true

Broadcast to many waiters

Broadcast wakes all waiting goroutines — ideal for a one-shot "start gate" that releases a pool of workers at once.

package main

import (
    "fmt"
    "sync"
)

func main() {
    var mu sync.Mutex
    cond := sync.NewCond(&mu)
    started := false
    var wg sync.WaitGroup

    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            mu.Lock()
            for !started {
                cond.Wait()
            }
            mu.Unlock()
            fmt.Println("worker", id, "started")
        }(i)
    }

    mu.Lock()
    started = true
    cond.Broadcast() // release ALL workers
    mu.Unlock()
    wg.Wait()
}
Output
worker 0 started worker 2 started worker 1 started (order varies between runs)

Common Mistakes

Wrong
mu.Lock()
if !ready {      // if, not for: proceeds on a stale wake
    cond.Wait()
}
mu.Unlock()

// Or calling Wait without holding cond.L -> undefined.
cond.Wait() // panic / corruption
Right
mu.Lock()
for !ready {     // re-check after every wake
    cond.Wait()  // must hold the lock here
}
// ... use the guarded state ...
mu.Unlock()

Key Rules

  • Always hold cond.L (the mutex) when calling Wait; it atomically unlocks, blocks, and re-locks before returning.
  • Re-check the predicate in a for loop, never a single if — after Broadcast every waiter runs but the condition may already be consumed by another.
  • Signal wakes one waiter, Broadcast wakes all; mutate the guarded state under the lock before signaling to avoid a missed wakeup.
  • Prefer channels for simple hand-offs; reach for sync.Cond when many goroutines wait on non-trivial shared state and channels get awkward.
Practice Exercise

Implement a bounded blocking queue with a single sync.Cond: Put waits while the buffer is full, Get waits while it is empty, and each mutation broadcasts so waiters re-check. Then build a one-shot start gate that releases N workers simultaneously with Broadcast. Deliberately swap the for guard for an if and show how a spurious/lost wake corrupts the queue. Finally, rewrite the bounded queue using buffered channels and explain which version you'd ship and why.

Pitfall: Guarding Wait with if instead of for — after a Broadcast all waiters wake, but only one may still satisfy the condition; the rest must loop and wait again. Also calling Wait without the lock held, or signaling before updating the guarded state (a missed-wakeup race where the signal is lost before a waiter parks).
Connections: Extends the sync package (Mutex, WaitGroup, Once) into wait/notify territory; an alternative to channel- and select-based signaling. The bounded-queue pattern underlies worker pools and backpressure from the concurrency-patterns days.

net.Dialer & net.Resolver: Custom Dialing & DNS

Networking

Why it matters: the default HTTP client and raw net.Dial hide connection behavior, but production code needs bounded timeouts, TCP keep-alives, context cancellation, and sometimes custom DNS. net.Dialer configures how outbound connections are made; net.Resolver controls name resolution. Wiring a tuned dialer into http.Transport is how you stop a single hung server from leaking goroutines forever.

A dialer with timeout and context

DialContext honors both the dialer's own Timeout and the caller's context deadline — prefer it over Dial.

package main

import (
    "context"
    "fmt"
    "net"
    "time"
)

func main() {
    d := net.Dialer{
        Timeout:   3 * time.Second,  // cap DNS + connect
        KeepAlive: 30 * time.Second, // TCP keep-alive probes
    }
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()

    conn, err := d.DialContext(ctx, "tcp", "example.com:80")
    if err != nil {
        fmt.Println("dial failed:", err)
        return
    }
    defer conn.Close()
    fmt.Println("connected to", conn.RemoteAddr())
}
Output (representative — remote IP varies)
connected to 93.184.216.34:80

A custom resolver

A net.Resolver with PreferGo and a custom Dial forces DNS through a chosen server — useful for testing, split-horizon DNS, or pinning a resolver.

r := &net.Resolver{
    PreferGo: true, // use Go's resolver, not cgo
    Dial: func(ctx context.Context, network, address string) (net.Conn, error) {
        d := net.Dialer{Timeout: 2 * time.Second}
        return d.DialContext(ctx, network, "1.1.1.1:53") // force Cloudflare DNS
    },
}
ips, err := r.LookupHost(context.Background(), "example.com")
fmt.Println(ips, err)

Plugging the dialer into http.Transport

Reuse one *http.Client with a tuned Transport: bound every layer — dial, TLS handshake, idle connections, and the overall request.

package main

import (
    "fmt"
    "net"
    "net/http"
    "time"
)

func main() {
    tr := &http.Transport{
        DialContext: (&net.Dialer{
            Timeout:   5 * time.Second,
            KeepAlive: 30 * time.Second,
        }).DialContext,
        MaxIdleConns:        100,
        IdleConnTimeout:     90 * time.Second,
        TLSHandshakeTimeout: 5 * time.Second,
    }
    client := &http.Client{Transport: tr, Timeout: 10 * time.Second}
    _ = client
    fmt.Println("client configured with custom dialer")
}
Output
client configured with custom dialer

Common Mistakes

Wrong
// No timeout: one hung server blocks forever.
resp, _ := http.Get(url) // uses DefaultClient (Timeout: 0)

// New client per request: defeats connection pooling.
for _, url := range urls {
    c := &http.Client{}
    c.Get(url)
}
Right
// One shared, fully-bounded client, reused everywhere.
var client = &http.Client{
    Timeout: 10 * time.Second,
    Transport: &http.Transport{
        DialContext: (&net.Dialer{Timeout: 5 * time.Second}).DialContext,
        TLSHandshakeTimeout: 5 * time.Second,
    },
}
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req)

Key Rules

  • Never use http.DefaultClient/http.Get in production without a Timeout — the default waits forever. Give it a bounded Timeout and a Transport with a configured net.Dialer.
  • Prefer DialContext over Dial so dials honor the caller's cancellation and deadline.
  • Dialer.Timeout bounds a single dial (DNS + connect); http.Client.Timeout bounds the whole request including reading the body — set both, plus TLSHandshakeTimeout.
  • Reuse one *http.Client/Transport across goroutines (they're safe and pool connections); use a custom net.Resolver to control DNS.
Practice Exercise

Build an http.Client with a fully bounded lifecycle — dial timeout, TLS handshake timeout, response-header timeout, and overall Timeout — and prove a deliberately slow endpoint returns a deadline error instead of hanging (use httptest with a handler that sleeps). Wrap the dialer to log each connection's target and latency. Then configure a net.Resolver pointing at an unreachable DNS server and observe the resolution error. Explain why sharing a single client across goroutines is both correct and preferred.

Pitfall: Creating a fresh http.Client per request (defeats connection pooling and can exhaust sockets under load) or using one with no timeout (a single unresponsive host stalls goroutines indefinitely, a classic production outage). Configure timeouts at every layer — dial, TLS, headers, overall — and reuse one client.
