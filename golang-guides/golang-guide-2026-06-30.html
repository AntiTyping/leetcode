Maps

Go Foundations • due today • rep 3

Quick Reminder

A map[K]V is a reference to a hash table; the zero value is nil and reading it is fine but writing to a nil map panics — always make(map[K]V) or use a literal first. Indexing a missing key returns the value's zero value, so use the comma-ok form v, ok := m[k] to tell "absent" from "present-but-zero". Iteration order is deliberately randomized.

Key Pattern

package main

import "fmt"

func main() {
    counts := make(map[string]int)
    for _, w := range []string{"go", "rust", "go"} {
        counts[w]++ // missing key reads as 0, then increments
    }

    if n, ok := counts["go"]; ok {
        fmt.Println("go:", n) // go: 2
    }
    fmt.Println(counts["zig"]) // 0 (absent -> zero value)

    delete(counts, "rust")     // safe even if key is absent
    fmt.Println(len(counts))   // 1
}
Challenge Exercise

Build a map[string][]int that groups numbers by a bucket name, appending to the slice for each key (note that append to a missing key works because the nil slice is a valid starting point). Then write a function that returns the keys sorted, since map iteration order is random. Finally, explain why you cannot take the address of a map element (&m[k] is a compile error) and how a map[K]*Struct works around it.

Pitfall: Assuming you can mutate a struct value stored directly in a map: m[k].Field = x is a compile error because map values are not addressable. Either store a pointer (map[K]*T) and mutate through it, or read the whole struct out, modify the copy, and assign it back with m[k] = v.

Strings & Runes

Go Foundations • due today • rep 3

Quick Reminder

A Go string is an immutable, read-only slice of bytes (UTF-8 by convention), not characters. Indexing s[i] yields a byte; ranging over a string decodes runes (Unicode code points) and gives you the byte offset of each. len(s) is the byte count, not the character count — use utf8.RuneCountInString for the latter.

Key Pattern

package main

import (
    "fmt"
    "unicode/utf8"
)

func main() {
    s := "Héllo" // 'é' is two bytes in UTF-8

    fmt.Println(len(s))                    // 6 bytes
    fmt.Println(utf8.RuneCountInString(s)) // 5 runes

    for i, r := range s { // r is a rune, i is a BYTE index
        fmt.Printf("%d:%c ", i, r)
    }
    fmt.Println()

    // Convert to []rune to index by character:
    rs := []rune(s)
    fmt.Printf("%c\n", rs[1]) // é
}
Output
6 5 0:H 1:é 3:l 4:l 5:o é
Challenge Exercise

Write a Reverse(s string) string that correctly reverses by rune, not byte, so multi-byte characters survive. Then write a function that counts how many bytes each of the first N runes occupies. Explain why s[0] gives a byte (uint8) rather than a rune, and why building a string with += in a loop is quadratic while strings.Builder is linear.

Pitfall: Slicing a string by byte offset (s[i:j]) in the middle of a multi-byte rune produces invalid UTF-8 — a garbled character. Byte-slicing is only safe at rune boundaries. When you need character-based indexing, convert to []rune once (it costs an allocation) rather than slicing bytes blindly.

Structs & Methods

Go Foundations • due today • rep 3

Quick Reminder

A struct groups named fields; methods attach behavior via a receiver. A value receiver (t T) operates on a copy — mutations don't stick; a pointer receiver (t *T) can mutate the original and avoids copying large structs. Go auto-takes the address for you when calling a pointer method on an addressable value. Keep the receiver type consistent across a type's method set.

Key Pattern

package main

import "fmt"

type Counter struct{ n int }

func (c Counter) Value() int { return c.n } // value receiver: read
func (c *Counter) Inc()      { c.n++ }       // pointer receiver: mutate

func main() {
    c := Counter{}
    c.Inc() // Go rewrites this as (&c).Inc()
    c.Inc()
    fmt.Println(c.Value()) // 2

    // Embedding: promoted fields & methods
    type Labeled struct {
        Counter // anonymous field -> Inc/Value are promoted
        Label string
    }
    l := Labeled{Label: "hits"}
    l.Inc()
    fmt.Println(l.Label, l.Value()) // hits 1
}
Challenge Exercise

Define a Rectangle with Area() (value receiver) and Scale(f float64) (pointer receiver), and show why calling Scale on an element of a []Rectangle works but calling it on a map[string]Rectangle value does not (addressability). Then embed Rectangle in a NamedShape and override Area(), demonstrating method promotion and shadowing.

Pitfall: Mixing value and pointer receivers on the same type. If some methods need a pointer, make them all pointer receivers — otherwise only *T satisfies the full method set, and storing a T value (e.g. in an interface) silently drops the pointer methods, so mutations appear to vanish or the type fails to implement the interface.

Graceful Shutdown

Production Go • due today • rep 2

Quick Reminder

A production server must stop cleanly: catch SIGINT/SIGTERM, stop accepting new connections, and let in-flight requests finish before exiting. signal.NotifyContext gives you a context cancelled on those signals; http.Server.Shutdown(ctx) drains active requests up to the context's deadline, then returns. Pair them so a Ctrl-C triggers an orderly drain instead of a hard kill.

Key Pattern

package main

import (
    "context"
    "errors"
    "net/http"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    srv := &http.Server{Addr: ":8080"}

    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    go func() {
        if err := srv.ListenAndServe(); err != nil &&
            !errors.Is(err, http.ErrServerClosed) {
            panic(err)
        }
    }()

    <-ctx.Done() // block until a signal arrives
    // give in-flight requests up to 10s to finish
    shCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    _ = srv.Shutdown(shCtx)
}
Challenge Exercise

Extend the pattern so shutdown also closes a database pool and flushes a metrics buffer, running those teardowns only after Shutdown returns. Add a hard-timeout fallback: if the drain context expires, call srv.Close() to force-close remaining connections. Explain why ListenAndServe returns http.ErrServerClosed on a clean shutdown and why you must not treat that as a fatal error.

Pitfall: Calling srv.Shutdown with a context.Background() that has no deadline. If a client holds a slow or hung connection open, Shutdown blocks forever and the process never exits. Always pass a bounded context, and consider a Close() fallback when the deadline is hit.

Configuration Management

Production Go • due today • rep 2

Quick Reminder

Twelve-factor config lives in the environment, not in code. Read values with os.Getenv (or os.LookupEnv to distinguish unset from empty), parse them into a typed struct, apply sane defaults, and fail fast at startup if a required value is missing or malformed. Centralize this in one LoadConfig() so the rest of the program depends on validated data, not raw strings.

Key Pattern

package main

import (
    "fmt"
    "os"
    "strconv"
    "time"
)

type Config struct {
    Port    int
    Timeout time.Duration
}

func LoadConfig() (Config, error) {
    c := Config{Port: 8080, Timeout: 5 * time.Second} // defaults

    if v, ok := os.LookupEnv("PORT"); ok {
        p, err := strconv.Atoi(v)
        if err != nil {
            return c, fmt.Errorf("PORT: %w", err) // fail fast
        }
        c.Port = p
    }
    if v, ok := os.LookupEnv("TIMEOUT"); ok {
        d, err := time.ParseDuration(v)
        if err != nil {
            return c, fmt.Errorf("TIMEOUT: %w", err)
        }
        c.Timeout = d
    }
    return c, nil
}
Challenge Exercise

Write a generic helper envOr[T any](key string, def T, parse func(string) (T, error)) (T, error) that reads, parses, and defaults a single variable, then compose LoadConfig from it. Add a Validate() that rejects a zero port or a negative timeout, aggregating all problems with errors.Join so the operator sees every misconfiguration at once, not just the first.

Pitfall: Using os.Getenv and treating "" as "unset". An explicitly empty variable (FOO=) is indistinguishable from a missing one this way, so a required-but-blank value silently falls back to a default. Use os.LookupEnv's ok boolean when the difference between "absent" and "empty" matters.

Escape Analysis & Stack vs Heap

Memory & Performance • due today • rep 2

Quick Reminder

The compiler decides at compile time whether a value lives on the goroutine's stack (cheap, auto-freed on return) or escapes to the heap (GC-managed). A value escapes when its lifetime outlives the function — most commonly by returning a pointer to a local, storing it in an interface, or capturing it in a closure that outlives the call. Fewer escapes mean fewer allocations and less GC pressure. Inspect with go build -gcflags='-m'.

Key Pattern

package main

// Returning &x forces x onto the heap: it must outlive newInt.
func newInt() *int {
    x := 42
    return &x // "moved to heap: x"
}

// Stays on the stack: sum never leaves the function as a pointer.
func total(xs []int) int {
    sum := 0
    for _, v := range xs {
        sum += v
    }
    return sum // returned by value -> no escape
}

// go build -gcflags='-m' main.go
//   ./main.go: moved to heap: x
//   ./main.go: total ... does not escape
Challenge Exercise

Take a function that builds a []byte and passes it to fmt.Sprintf, run -gcflags='-m', and explain each "escapes to heap" line. Then rewrite a hot path to accept a caller-provided buffer (func(dst []byte) []byte) so the allocation moves to the caller and can be pooled. Explain why passing a value to an interface{} parameter (like fmt.Println) usually forces an escape.

Pitfall: "Optimizing" by returning a pointer to avoid copying a small struct. Returning &x forces a heap allocation and a GC scan, whereas returning the struct by value often stays on the stack and is faster for small types. Measure with -gcflags='-m' and -benchmem before assuming pointers are cheaper.

Section 1 — Randomness, Hashing & Text Formatting

math/rand/v2: Modern Pseudo-Randomness

Std Lib

Why it matters: math/rand/v2 (Go 1.22, the standard library's first v2 package) fixes the ergonomic warts of the original math/rand. The global generator is now automatically seeded and safe for concurrent use, the API drops the error-prone default-Source boilerplate, and it adds handy helpers like N and IntN. Use it for shuffling, jitter, sampling, and load generation — but never for security tokens.

Package-level helpers, no seeding required

The top-level functions use a global, auto-seeded, goroutine-safe source. IntN(n) returns a value in [0, n); the generic N[T] works for any integer type including time.Duration.

package main

import (
    "fmt"
    "math/rand/v2"
    "time"
)

func main() {
    fmt.Println(rand.IntN(6) + 1)   // dice roll: 1..6
    fmt.Println(rand.Float64())     // 0.0 <= x < 1.0

    // rand.N is generic over integer types:
    jitter := rand.N(500 * time.Millisecond) // 0..500ms
    fmt.Println(jitter < 500*time.Millisecond) // true

    xs := []string{"a", "b", "c", "d"}
    rand.Shuffle(len(xs), func(i, j int) {
        xs[i], xs[j] = xs[j], xs[i]
    })
    fmt.Println(len(xs)) // 4 (order randomized)
}
Output (varies)
4 0.6122... true 4

Deterministic runs with an explicit source

For reproducible tests, construct your own generator from a seeded Source. NewPCG is a modern, fast, high-quality algorithm; a fixed seed yields the same sequence every run.

package main

import (
    "fmt"
    "math/rand/v2"
)

func main() {
    // Two 64-bit seed words -> fully reproducible stream.
    r := rand.New(rand.NewPCG(1, 2))
    for i := 0; i < 3; i++ {
        fmt.Println(r.IntN(100))
    }
    // Same seeds -> identical output on every run & machine.
}
Output (stable for these seeds)
1 30 53

Common Mistakes

Wrong
// v1 habit: manual Seed with the clock,
// and Intn spelled the old way.
rand.Seed(time.Now().UnixNano()) // no Seed in v2!
n := rand.Intn(6)                // renamed to IntN
Right
// v2 global source is auto-seeded & concurrent-safe.
n := rand.IntN(6)
// Reproducible? build your own:
r := rand.New(rand.NewPCG(seed1, seed2))

Key Rules

  • The global functions are auto-seeded and safe for concurrent use — there is no Seed function in v2; delete old rand.Seed calls.
  • Methods are renamed to IntN/Int64N/Uint64N; the generic rand.N[T](n) covers any integer type (great for durations).
  • For reproducible sequences, build rand.New(rand.NewPCG(s1, s2)) — a per-instance generator is not goroutine-safe, so give each goroutine its own.
  • Never use math/rand/v2 for tokens, keys, or nonces — it is predictable; use crypto/rand for security.
Practice Exercise

Write a retry helper that sleeps base * 2^attempt plus a random jitter drawn with rand.N, capping the total. Then write a weighted sampler: given items with integer weights, pick one proportional to its weight using a single IntN(total) draw. Finally, prove reproducibility by seeding two NewPCG(1, 2) generators and asserting they emit identical sequences — then contrast with why crypto/rand would be required for a session token.

Pitfall: Sharing one *rand.Rand instance across goroutines. Unlike the top-level functions, a generator you create with rand.New is not safe for concurrent use — concurrent calls corrupt its internal state and can panic or bias output. Either use the package-level helpers or give each goroutine its own seeded generator.
Connections: The randomness source behind shuffling, jittered backoff, and load testing; contrasts with crypto/rand for security; its rand.N generic echoes the type-parameter machinery from the generics topics.

Non-Cryptographic Hashing: hash/fnv & hash/crc32

Std Lib

Why it matters: not every hash needs to be cryptographic. For sharding keys across buckets, deduplicating, building hash tables, or a cheap checksum, you want a fast, well-distributed, non-secure hash — not the cost of SHA-256. hash/fnv and hash/crc32 implement Go's common hash.Hash interface (an io.Writer you feed bytes, then read a Sum), so they slot into the same plumbing as the crypto hashers.

FNV for fast key hashing

FNV-1a is a tiny, fast hash ideal for bucketing. Create a hasher, Write your bytes (it never errors), and read the result with Sum32/Sum64.

package main

import (
    "fmt"
    "hash/fnv"
)

func bucket(key string, n uint32) uint32 {
    h := fnv.New32a()
    h.Write([]byte(key)) // io.Writer; error is always nil
    return h.Sum32() % n
}

func main() {
    for _, k := range []string{"alice", "bob", "carol"} {
        fmt.Printf("%s -> shard %d\n", k, bucket(k, 4))
    }
}
Output
alice -> shard 3 bob -> shard 1 carol -> shard 2

CRC-32 for checksums

crc32.ChecksumIEEE is a one-shot checksum over a byte slice — the same polynomial used by zip/gzip — great for detecting accidental corruption (not tampering).

package main

import (
    "fmt"
    "hash/crc32"
)

func main() {
    data := []byte("go is fun")

    sum := crc32.ChecksumIEEE(data)
    fmt.Printf("%08x\n", sum)

    // Streaming form (implements hash.Hash32):
    h := crc32.NewIEEE()
    h.Write([]byte("go "))
    h.Write([]byte("is fun"))
    fmt.Printf("%08x\n", h.Sum32()) // same as one-shot
}
Output
6f8f9f0d 6f8f9f0d

Common Mistakes

Wrong
// Reusing a hasher without Reset carries
// the previous bytes into the next sum.
h := fnv.New32a()
h.Write([]byte("a"))
sumA := h.Sum32()
h.Write([]byte("b")) // now hashing "ab", not "b"!
sumB := h.Sum32()
Right
h := fnv.New32a()
h.Write([]byte("a"))
sumA := h.Sum32()
h.Reset() // clear state before the next value
h.Write([]byte("b"))
sumB := h.Sum32()

Key Rules

  • Both packages implement hash.Hash — Write bytes (error is always nil), then read Sum32/Sum64; call Reset() to reuse a hasher.
  • These are not cryptographic: they are trivially reversible/collidable, so never use them for passwords, signatures, or integrity against an attacker.
  • Use fnv for in-memory bucketing/sharding; use crc32.ChecksumIEEE for accidental-corruption checksums (matches gzip/zip).
  • The one-shot helpers (crc32.ChecksumIEEE) avoid hasher allocation; the streaming form suits data arriving in chunks via io.Copy.
Practice Exercise

Build a consistent-ish shard router: hash each key with FNV-1a and map it into N buckets, then print the load distribution for 10k random keys to confirm it's roughly even. Then write a file-integrity checker that streams a file through crc32.NewIEEE() with io.Copy and compares against a stored checksum. Finally, benchmark fnv.New64a vs sha256.New on the same input and quantify the speed gap — and explain when the crypto cost is nonetheless mandatory.

Pitfall: Reaching for CRC-32 or FNV as a "hash" to protect data integrity against tampering. Both are easily forged — an attacker can craft a different input with the same checksum. They detect accidental corruption only. For security (integrity against an adversary), use crypto/sha256 or an HMAC.
Connections: Shares the hash.Hash/io.Writer contract with crypto hashers and the io plumbing; the streaming form pairs with bufio and io.Copy; contrast its speed/security trade-off with crypto/rand-grade primitives.

text/tabwriter: Aligned Columnar Output

Std Lib

Why it matters: printing tab-separated data with fmt leaves ragged columns because tab stops don't line up. text/tabwriter is an io.Writer filter that buffers your tab-delimited rows and, on Flush, pads each cell so columns align — the mechanism behind clean CLI tables (it's what go tooling and kubectl-style output use). You write once with \t separators and let it compute widths.

Elastic tabstops in a few lines

Wrap any io.Writer with tabwriter.NewWriter, write rows using \t between columns, and call Flush to emit the aligned result. Nothing prints until you flush.

package main

import (
    "fmt"
    "os"
    "text/tabwriter"
)

func main() {
    // minwidth=0, tabwidth=0, padding=2, padchar=' ', flags=0
    w := tabwriter.NewWriter(os.Stdout, 0, 0, 2, ' ', 0)

    fmt.Fprintln(w, "NAME\tLANG\tSTARS")
    fmt.Fprintln(w, "cobra\tGo\t38000")
    fmt.Fprintln(w, "gin\tGo\t79000")
    fmt.Fprintln(w, "ripgrep\tRust\t48000")

    w.Flush() // aligns all buffered rows now
}
Output
NAME LANG STARS cobra Go 38000 gin Go 79000 ripgrep Rust 48000

Right-align numbers with AlignRight

Flags tune behavior: tabwriter.AlignRight right-justifies cells (ideal for numbers), and tabwriter.Debug draws a | between columns to visualize boundaries.

package main

import (
    "fmt"
    "os"
    "text/tabwriter"
)

func main() {
    w := tabwriter.NewWriter(os.Stdout, 0, 0, 1, ' ',
        tabwriter.AlignRight)

    fmt.Fprintln(w, "ITEM\tQTY\tPRICE")
    fmt.Fprintln(w, "widget\t3\t9")
    fmt.Fprintln(w, "gadget\t12\t120")
    w.Flush()
}
Output
ITEM QTY PRICE widget 3 9 gadget 12 120

Common Mistakes

Wrong
w := tabwriter.NewWriter(os.Stdout, 0, 0, 2, ' ', 0)
fmt.Fprintln(w, "a\tb")
// forgot Flush -> nothing is ever printed
Right
w := tabwriter.NewWriter(os.Stdout, 0, 0, 2, ' ', 0)
fmt.Fprintln(w, "a\tb")
w.Flush() // or defer w.Flush() right after creating it

Key Rules

  • tabwriter is an io.Writer filter — separate columns with \t, then Flush() to emit aligned output (buffered until then).
  • Constructor params are (output, minwidth, tabwidth, padding, padchar, flags); the common table setup is (w, 0, 0, 2, ' ', 0).
  • Use the AlignRight flag for numeric columns and Debug to visualize column edges while developing.
  • A trailing tab starts a new cell, so a line ending in \t creates an empty final column — end data rows without one.
Practice Exercise

Render a process table from a slice of structs (name, PID, memory, status) with a header row, using AlignRight for the numeric columns. Then wrap the whole thing so it writes to a bytes.Buffer instead of stdout, letting you unit-test the exact aligned bytes. Finally, add a summary footer row and explain why you must Flush before reading the buffer, and how tabwriter handles a row with fewer columns than the header.

Pitfall: Forgetting to Flush (or flushing too early). Because tabwriter buffers every row to compute column widths, output is empty until Flush — and if you flush mid-table, later rows are aligned in a separate block with different widths. Write all rows, then flush once (a defer w.Flush() right after construction is the safe idiom for simple cases).
Connections: An io.Writer filter like the bufio writers, commonly targeting os.Stdout or a bytes.Buffer; pairs with fmt.Fprintf for formatting and with the os/exec / CLI topics for building command-line tools.

Section 2 — Processes & Data Formats

os/exec: Running External Commands

Std Lib

Why it matters: real tools shell out — running git, ffmpeg, a linter, or a subprocess worker. os/exec builds and runs external commands with full control over arguments, environment, working directory, stdin/stdout/stderr, and cancellation via context. Crucially it does not use a shell by default, so arguments are passed directly — sidestepping the shell-injection class of bugs.

Capture output the simple way

exec.Command(name, args...) builds a *Cmd; Output() runs it and returns stdout. On failure you get an *exec.ExitError whose Stderr holds the command's error text.

package main

import (
    "errors"
    "fmt"
    "os/exec"
)

func main() {
    out, err := exec.Command("echo", "hello", "go").Output()
    if err != nil {
        var ee *exec.ExitError
        if errors.As(err, &ee) {
            fmt.Printf("exit %d: %s", ee.ExitCode(), ee.Stderr)
        }
        return
    }
    fmt.Printf("%s", out) // hello go
}
Output
hello go

Context for timeouts, plus pipes and env

Use exec.CommandContext so a cancelled or timed-out context kills the process. Set cmd.Dir, cmd.Env, and stream I/O by assigning to cmd.Stdin/Stdout/Stderr.

package main

import (
    "bytes"
    "context"
    "fmt"
    "os/exec"
    "time"
)

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()

    cmd := exec.CommandContext(ctx, "go", "version")
    cmd.Dir = "."               // working directory
    cmd.Env = append(cmd.Environ(), "GOFLAGS=-mod=mod")

    var out bytes.Buffer
    cmd.Stdout = &out           // stream stdout into a buffer
    if err := cmd.Run(); err != nil {
        fmt.Println("run failed:", err)
        return
    }
    fmt.Print(out.String())
}
Output (varies)
go version go1.24 darwin/arm64

Common Mistakes

Wrong
// Passing a whole command line as ONE string
// runs a program literally named "ls -l /tmp".
exec.Command("ls -l /tmp").Run() // file not found

// Or worse, invoking a shell with user input:
exec.Command("sh", "-c", "cat "+userInput) // injection!
Right
// Program name and each arg passed separately:
exec.Command("ls", "-l", "/tmp").Run()

// No shell -> userInput is a plain argument, not code:
exec.Command("cat", userInput).Run()

Key Rules

  • exec.Command(name, arg1, arg2, ...) takes the program and each argument separately — there is no shell, so no globbing, pipes, or $VAR expansion unless you invoke one explicitly.
  • Output() captures stdout; CombinedOutput() merges stdout+stderr; Run() just waits; Start()+Wait() lets you do work while it runs.
  • Use exec.CommandContext for timeouts/cancellation — the process is killed when the context is done.
  • Check errors with errors.As(&exec.ExitError{}) to read the exit code and Stderr; a nil error means exit status 0.
Practice Exercise

Write runGit(ctx, args...) (string, error) that runs git in a given directory, returns trimmed stdout, and surfaces stderr on failure via errors.As. Then build a pipeline: connect the stdout of one Cmd to the stdin of another using StdoutPipe (emulating a | b) and run both concurrently with Start/Wait. Finally, add a 500ms CommandContext timeout to a sleep 5 and confirm the process is killed — and explain why never passing user input to sh -c eliminates shell injection.

Pitfall: Building a command by concatenating strings and running it through sh -c. That reintroduces shell injection — a value like ; rm -rf ~ becomes executable code. Pass arguments as separate slice elements to exec.Command (the default, shell-free path); only use a shell when you truly need shell features, and never with untrusted input.
Connections: Uses context for cancellation like the concurrency topics, streams through io.Writer/bytes.Buffer, and errors flow through errors.As; the backbone of CLI wrappers built with flag/Cobra and formatted via tabwriter.

encoding/csv: Reading & Writing CSV

Std Lib

Why it matters: CSV is the lingua franca of data exchange — exports, spreadsheets, ETL feeds. Doing it by hand with strings.Split(",") breaks the moment a field contains a comma, quote, or newline. encoding/csv handles RFC 4180 quoting, embedded newlines, and configurable delimiters correctly, exposing a streaming Reader (record by record) and a buffered Writer.

Streaming a reader row by row

csv.NewReader wraps any io.Reader. Read() returns one record (a []string) at a time until io.EOF — the memory-friendly way to process large files.

package main

import (
    "encoding/csv"
    "fmt"
    "io"
    "strings"
)

func main() {
    const data = `name,age,city
"Ada, L.",36,London
Bo,25,"New
York"`

    r := csv.NewReader(strings.NewReader(data))
    for {
        rec, err := r.Read()
        if err == io.EOF {
            break
        }
        if err != nil {
            fmt.Println("error:", err)
            return
        }
        fmt.Printf("%q\n", rec)
    }
}
Output
["name" "age" "city"] ["Ada, L." "36" "London"] ["Bo" "25" "New\nYork"]

Writing records with automatic quoting

csv.NewWriter quotes fields that need it and must be Flushed. Set Comma to change the delimiter (e.g. '\t' for TSV or ';').

package main

import (
    "encoding/csv"
    "os"
)

func main() {
    w := csv.NewWriter(os.Stdout)
    defer w.Flush() // buffered: nothing written until flush

    rows := [][]string{
        {"product", "note"},
        {"pen", "blue, fine"}, // comma -> auto-quoted
        {"pad", "3\"x5\""},     // quote -> escaped
    }
    for _, row := range rows {
        if err := w.Write(row); err != nil {
            return
        }
    }
}
Output
product,note pen,"blue, fine" pad,"3""x5"""

Common Mistakes

Wrong
// Naive split mangles quoted commas & newlines:
for _, line := range strings.Split(data, "\n") {
    fields := strings.Split(line, ",")
    // "Ada, L." becomes two broken fields
}
Right
r := csv.NewReader(strings.NewReader(data))
records, err := r.ReadAll() // or Read() in a loop
// quoting, embedded commas & newlines handled correctly

Key Rules

  • csv.NewReader streams records via Read() (stop on io.EOF) or slurps everything with ReadAll(); each record is a []string.
  • The Writer is buffered — you must Flush() (and check w.Error()) or the tail of your output is lost.
  • Set Comma for other delimiters and reader.FieldsPerRecord to enforce (or with -1, allow) a variable column count.
  • Enable reader.ReuseRecord = true in hot loops to avoid a per-row allocation — but then copy the slice if you keep it, since it's overwritten next Read.
Practice Exercise

Parse a CSV with a header row into a []map[string]string keyed by column name, validating that every row has the header's field count. Then transform and write it back out as TSV by setting Comma = '\t'. Finally, process a large file in a streaming loop with ReuseRecord = true, and demonstrate the bug you get if you append the reused record slice to a growing list without copying it.

Pitfall: Forgetting to Flush the csv.Writer (or ignoring w.Error()). Because writes are buffered, a program that exits or returns without flushing silently drops the last rows — the file looks truncated for no obvious reason. Always defer w.Flush() and check w.Error() afterward to catch write failures that Write itself didn't surface.
Connections: A streaming codec over io.Reader/io.Writer like encoding/json and encoding/gob; fields are []string you often feed to strconv; the buffered-Flush discipline mirrors tabwriter and bufio.
