Type Parameters (Generics)

Functions & Control Flow • 15 days since review • rep 3

Quick Reminder

Type parameters let a function or type abstract over types with full compile-time safety and no runtime boxing. Declare them in square brackets with a constraint: [T any], [T comparable], or a custom interface constraint. Arguments are usually inferred from the call, but inference works from parameters, not from the return type. To use an operator like + or < inside the body, the constraint must permit it (e.g. constraints.Ordered or a union with ~ for named types).

Key Pattern

func Map[T, U any](s []T, f func(T) U) []U {
    r := make([]U, len(s))
    for i, v := range s {
        r[i] = f(v)
    }
    return r
}

type Number interface{ ~int | ~float64 } // ~ admits named types (type Cents int)

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x // legal: the constraint permits +
    }
    return total
}
Challenge Exercise

Implement generic Filter, Map, and Reduce over slices, plus a Keys[K comparable, V any](m map[K]V) []K. Write a Min[T constraints.Ordered] and explain why it won't compile without the ordered constraint. Then demonstrate the inference limit: a call where Go cannot infer U from arguments alone and must be instantiated explicitly as Map[int, string](...).

#1 Pitfall: Reaching for generics where any+type switch or a plain interface reads better — and forgetting that methods cannot declare their own type parameters (only the type can). Also, you cannot use +, <, or range-over an operator on a bare [T any]; widen the constraint to a union or constraints.Ordered first.

Goroutines

Concurrency • 15 days since review • rep 3

Quick Reminder

go f() launches a lightweight, runtime-scheduled goroutine over an OS thread pool (starting stack of a few KB, grown on demand). They are cheap but not free: nothing waits for them automatically, and when main returns the process exits, killing every remaining goroutine. You must synchronize explicitly (a sync.WaitGroup, a channel, errgroup). Since Go 1.22 the loop variable is per-iteration, so the classic capture bug is gone.

Key Pattern

var wg sync.WaitGroup
for _, u := range urls {
    wg.Add(1)
    go func() {
        defer wg.Done()
        fetch(u) // Go 1.22+: u is a fresh variable each iteration
    }()
}
wg.Wait() // block until every goroutine calls Done
Challenge Exercise

Launch N goroutines that each process one item and coordinate completion with a sync.WaitGroup. Then deliberately create a goroutine leak: a goroutine blocked forever on <-ch for a channel nobody closes; detect it with runtime.NumGoroutine() and fix it by closing the channel or wiring a context. Finally, show why calling wg.Add inside the goroutine races with wg.Wait.

#1 Pitfall: Fire-and-forget without any synchronization — main returns and the goroutines never run to completion. The mirror bug is the goroutine leak: a goroutine parked on a send/receive that will never happen accumulates until you run out of memory. Always define how each goroutine terminates.

Channels

Concurrency • 15 days since review • rep 3

Quick Reminder

Channels are typed conduits. Unbuffered channels are a synchronous rendezvous (send blocks until a receiver is ready); a buffered channel decouples sender and receiver up to its capacity. close(ch) signals "no more values": receiving from a closed, drained channel yields the zero value with ok == false, and range ends. Sending on a closed channel panics, and closing twice panics. A nil channel blocks forever.

Key Pattern

ch := make(chan int) // unbuffered
go func() {
    for i := 0; i < 3; i++ {
        ch <- i
    }
    close(ch) // the SENDER closes, exactly once
}()

for v := range ch { // loop ends when ch is closed and drained
    fmt.Println(v)
}

v, ok := <-ch // ok == false, v == 0 on a closed+drained channel
fmt.Println(v, ok)
Challenge Exercise

Build a three-stage pipeline (generate → square → print) where each stage returns a <-chan int and closes its output when done. Then trigger a deadlock by sending on an unbuffered channel with no receiver and read the fatal error: all goroutines are asleep message. Finally, use the comma-ok receive to distinguish a real zero value from a closed channel.

#1 Pitfall: Closing a channel from the receiver side or closing it twice — both panic. The rule is "only the sender closes, and only once." Equally common: forgetting to close, so a range over the channel hangs forever, or leaving a sender blocked on a full/unreceived channel (a leak).

Select Statement

Concurrency • 15 days since review • rep 3

Quick Reminder

select waits on multiple channel operations and proceeds with one that is ready; if several are ready it picks uniformly at random. A default case makes the select non-blocking. It is the backbone of timeouts (time.After), cancellation (<-ctx.Done()), and multiplexing. A case on a nil channel is never selected — a handy way to dynamically disable a branch. An empty select{} blocks forever.

Key Pattern

select {
case v := <-work:
    handle(v)
case <-ctx.Done():
    return ctx.Err() // cancelled or deadline exceeded
case <-time.After(500 * time.Millisecond):
    return errTimeout
default:
    // runs immediately if nothing else is ready (non-blocking)
}
Challenge Exercise

Write a worker that ranges over a jobs channel but also exits promptly on <-ctx.Done(), and add a per-job timeout. Demonstrate the random selection by looping over a select with two always-ready channels and tallying the picks. Then set one channel variable to nil mid-loop and show its case goes dormant — the idiom for turning a branch off.

#1 Pitfall: case <-time.After(d) inside a hot loop allocates and leaks a fresh timer every iteration — use a single time.NewTimer/Ticker and reset it. And a default case inside a tight for turns a blocking wait into a CPU-burning busy loop.

Context

Concurrency • 15 days since review • rep 3

Quick Reminder

context.Context carries cancellation, deadlines, and request-scoped values across API boundaries. WithCancel/WithTimeout/WithDeadline return a derived context and a cancel function you must always call (defer it) to release resources. ctx.Done() returns a channel that closes on cancellation; ctx.Err() reports Canceled or DeadlineExceeded. Pass it as the first parameter, never store it in a struct, and reserve WithValue for genuinely request-scoped data with a private key type.

Key Pattern

ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel() // always, even on the happy path

req, _ := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
resp, err := http.DefaultClient.Do(req) // aborts if ctx times out
if err != nil {
    return err // context.DeadlineExceeded on timeout
}
defer resp.Body.Close()
Challenge Exercise

Thread a ctx through a three-function call chain and cancel the whole tree when any one leg fails (hint: errgroup.WithContext). Add a private type ctxKey struct{} and stash a request ID with WithValue, retrieving it deep in the stack. Prove that omitting defer cancel() leaks the timer goroutine by watching runtime.NumGoroutine().

#1 Pitfall: Forgetting to call cancel() — the derived context (and its timer goroutine) leaks until the parent is cancelled. Also: passing a nil context (use context.TODO()), storing a context in a struct field, and abusing WithValue as a bag for optional function parameters.

Concurrency Patterns

Concurrency • 15 days since review • rep 3

Quick Reminder

The core idioms compose from goroutines + channels + context: generators (a function returning <-chan T), pipelines (stages linked by channels), fan-out/fan-in (many workers feeding one collector), and worker pools (a fixed set of goroutines draining a jobs channel). For each channel you must answer two questions: who closes it, and how does every goroutine exit (usually a closed input channel or ctx.Done()).

Key Pattern

jobs := make(chan int)
results := make(chan int)

var wg sync.WaitGroup
for i := 0; i < 4; i++ { // fixed pool of 4 workers
    wg.Add(1)
    go func() {
        defer wg.Done()
        for j := range jobs { // exits when jobs is closed
            results <- j * j
        }
    }()
}

go func() { for _, n := range input { jobs <- n }; close(jobs) }()
go func() { wg.Wait(); close(results) }() // close results AFTER workers finish

for r := range results {
    collect(r)
}
Challenge Exercise

Build a bounded worker pool that propagates the first error and cancels the rest via errgroup.WithContext. Add a cancellable pipeline where each stage selects on <-ctx.Done() so a downstream abort unblocks upstream sends. Verify with runtime.NumGoroutine() that no goroutine leaks after completion or early cancellation.

#1 Pitfall: Closing the results channel before all workers are done — a worker's subsequent send panics. The correct shape is a separate goroutine doing wg.Wait(); close(results). Related: unbounded goroutine creation (one per request with no ceiling) and stages that block forever because no one wired a cancellation/done signal.

Section 1 — Web & HTTP Standard Library

net/http Cookies: SetCookie, Reading & cookiejar

Web

Why it matters: cookies carry sessions, auth tokens, CSRF secrets, and preferences. net/http models one as an http.Cookie struct, writes Set-Cookie with http.SetCookie, reads request cookies with r.Cookie, and — via a http.CookieJar — stores and replays them automatically on the client. Assembling the header string by hand mishandles attributes, ordering, and value encoding.

Writing a cookie on the response

http.SetCookie serializes the struct (with its security attributes) into a well-formed Set-Cookie header.

package main

import (
    "fmt"
    "net/http"
    "net/http/httptest"
)

func handler(w http.ResponseWriter, r *http.Request) {
    http.SetCookie(w, &http.Cookie{
        Name:     "session",
        Value:    "abc123",
        Path:     "/",
        MaxAge:   3600,
        HttpOnly: true,
        Secure:   true,
        SameSite: http.SameSiteLaxMode,
    })
    fmt.Fprintln(w, "ok")
}

func main() {
    rec := httptest.NewRecorder()
    handler(rec, httptest.NewRequest(http.MethodGet, "/", nil))
    fmt.Println(rec.Result().Header.Get("Set-Cookie"))
}
Output
session=abc123; Path=/; Max-Age=3600; HttpOnly; Secure; SameSite=Lax

Reading cookies from a request

r.Cookie(name) returns a single cookie (or http.ErrNoCookie); r.Cookies() returns all of them.

package main

import (
    "fmt"
    "net/http"
    "net/http/httptest"
)

func main() {
    r := httptest.NewRequest(http.MethodGet, "/", nil)
    r.AddCookie(&http.Cookie{Name: "session", Value: "abc123"})

    c, err := r.Cookie("session")
    if err != nil {
        panic(err)
    }
    fmt.Println(c.Name, "=", c.Value)
    fmt.Println("count:", len(r.Cookies()))
}
Output
session = abc123 count: 1

Automatic client cookies with a jar

Give an http.Client a cookiejar.Jar and it stores Set-Cookie responses and replays them on later requests to the same site — no manual bookkeeping.

package main

import (
    "fmt"
    "io"
    "net/http"
    "net/http/cookiejar"
    "net/http/httptest"
)

func main() {
    srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if _, err := r.Cookie("session"); err != nil {
            http.SetCookie(w, &http.Cookie{Name: "session", Value: "xyz"})
            fmt.Fprint(w, "set")
            return
        }
        fmt.Fprint(w, "seen")
    }))
    defer srv.Close()

    jar, _ := cookiejar.New(nil)
    client := &http.Client{Jar: jar}

    do := func() string {
        resp, _ := client.Get(srv.URL)
        b, _ := io.ReadAll(resp.Body)
        resp.Body.Close()
        return string(b)
    }
    fmt.Println(do()) // first call: server sets the cookie
    fmt.Println(do()) // second call: jar replayed it
}
Output
set seen

Common Mistakes

Wrong
// Hand-writing the header: no attributes,
// no sanitizing, no security flags.
w.Header().Set("Set-Cookie", "session="+val)
// A val with ';' or ',' silently breaks the header.
Right
http.SetCookie(w, &http.Cookie{
    Name:     "session",
    Value:    val, // sanitized by SetCookie
    HttpOnly: true,
    Secure:   true,
    SameSite: http.SameSiteLaxMode,
})

Key Rules

  • MaxAge > 0 sets an expiry in seconds; MaxAge < 0 deletes the cookie now; MaxAge == 0 omits Max-Age (a session cookie).
  • SetCookie sanitizes but does not URL-encode the value — encode anything with spaces, ;, or , yourself (base64 or url.QueryEscape).
  • Set HttpOnly (hide from JS), Secure (HTTPS only), and SameSite (CSRF defense) on every session cookie.
  • An http.Client with a Jar sends and stores cookies automatically per URL; without a jar you manage the Cookie header yourself.
Practice Exercise

Write a login handler that issues an HttpOnly, Secure, SameSite=Lax session cookie and a logout handler that expires it with MaxAge: -1. Test both with httptest.NewRecorder, asserting the exact Set-Cookie header. Then build a two-request client flow with a cookiejar.Jar and confirm the second request is recognized as authenticated. Finally, store a value containing a space and show it must be encoded to survive.

Pitfall: Hand-building Set-Cookie/Cookie headers — you lose attribute formatting, value sanitizing, and the security flags, and a value containing ; or a comma corrupts the header. Also easy to forget: without a Secure flag a session cookie leaks over plain HTTP, and without a Jar the http.Client throws away every Set-Cookie the server sends.
Connections: Builds directly on net/http handlers and Client; tested with httptest; values are encoded with encoding/base64 or net/url; and cookie handling is the substrate for session/auth middleware chains and CSRF defenses.

net/http/httptrace: Observing Client Requests

Diagnostics

Why it matters: when an outbound HTTP request is slow, you need to know where the time went — DNS lookup, TCP connect, TLS handshake, or waiting for the first response byte. httptrace hooks callbacks into the client request lifecycle, giving per-phase visibility (and per-phase timings) without a proxy or packet capture.

Attaching a trace via the request context

Build a ClientTrace with the hooks you care about and attach it to the request's context with WithClientTrace.

package main

import (
    "fmt"
    "net/http"
    "net/http/httptest"
    "net/http/httptrace"
)

func main() {
    srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprint(w, "hello")
    }))
    defer srv.Close()

    trace := &httptrace.ClientTrace{
        ConnectStart:         func(network, addr string) { fmt.Println("dialing", network) },
        GotConn:              func(i httptrace.GotConnInfo) { fmt.Println("reused:", i.Reused) },
        GotFirstResponseByte: func() { fmt.Println("first byte") },
    }

    req, _ := http.NewRequest(http.MethodGet, srv.URL, nil)
    ctx := httptrace.WithClientTrace(req.Context(), trace)
    resp, err := http.DefaultClient.Do(req.WithContext(ctx))
    if err != nil {
        panic(err)
    }
    resp.Body.Close()
}
Output (representative)
dialing tcp reused: false first byte

Measuring time-to-first-byte

Record timestamps inside the hooks to compute per-phase durations — here, TTFB from connection acquisition to the first response byte.

var start time.Time
var ttfb time.Duration

trace := &httptrace.ClientTrace{
    GetConn:              func(string) { start = time.Now() },
    GotFirstResponseByte: func() { ttfb = time.Since(start) },
}

req, _ := http.NewRequest(http.MethodGet, url, nil)
req = req.WithContext(httptrace.WithClientTrace(req.Context(), trace))

resp, _ := http.DefaultClient.Do(req)
resp.Body.Close()
fmt.Printf("TTFB = %v\n", ttfb)
Output (representative)
TTFB = 214µs

Detecting connection reuse

GotConnInfo.Reused tells you whether a keep-alive connection was recycled — a reused connection skips the DNS/Connect/TLS hooks entirely.

trace := &httptrace.ClientTrace{
    GotConn: func(i httptrace.GotConnInfo) {
        fmt.Printf("reused=%v idle=%v\n", i.Reused, i.WasIdle)
    },
}
// First request: reused=false. A second request on the same
// client (keep-alive) reports reused=true and fires no dial hooks.

Common Mistakes

Wrong
// Building a trace but never attaching it:
trace := &httptrace.ClientTrace{ /* hooks */ }
resp, _ := http.DefaultClient.Do(req) // trace unused -> no hooks fire
Right
ctx := httptrace.WithClientTrace(req.Context(), trace)
resp, _ := http.DefaultClient.Do(req.WithContext(ctx)) // now hooks fire

Key Rules

  • httptrace is client-side only — hooks fire during http.Client/Transport round-trips, never for server handlers.
  • Attach with req.WithContext(httptrace.WithClientTrace(ctx, trace)); a trace value that isn't in the request context does nothing.
  • Only relevant hooks fire: no DNSStart for an IP literal, no TLS hooks for plain HTTP, and none of the dial hooks on a reused keep-alive connection.
  • Hooks may run on internal goroutines — synchronize (or keep to simple timestamp writes) if they touch shared state.
Practice Exercise

Write a timeGet(url string) that records DNSStart/DNSDone, ConnectStart/ConnectDone, TLSHandshakeStart/TLSHandshakeDone, and GotFirstResponseByte, then prints a per-phase latency breakdown against a real HTTPS URL. Make a second request on the same client and show the dial phases vanish because the connection was reused. Explain why the trace must live in the request's context, not a package variable.

Pitfall: Hooks run on the transport's internal goroutines, so writing to shared maps/slices from several hooks without synchronization is a data race. And "missing" callbacks are usually normal — a reused keep-alive connection, an IP-literal host, or a plain-HTTP request each legitimately skip whole groups of hooks.
Connections: Extends net/http Client/Transport; context-driven like today's context review; complements httptest for a server under test and the runtime/trace / pprof tools for whole-program latency analysis.

net/textproto: Header-Style Text Protocols

Protocols

Why it matters: HTTP, SMTP, NNTP, and mail all share a line-oriented shape — CRLF-terminated Key: value header blocks with canonicalized keys, plus numeric response codes. net/textproto is the shared engine beneath net/http and net/mail that reads exactly this. Reach for it directly when you implement or parse such a protocol.

Canonicalizing header keys

CanonicalMIMEHeaderKey normalizes casing the way HTTP/MIME expect (Content-Type, X-Request-Id), which is how case-insensitive lookups work.

package main

import (
    "fmt"
    "net/textproto"
)

func main() {
    fmt.Println(textproto.CanonicalMIMEHeaderKey("content-TYPE"))
    fmt.Println(textproto.CanonicalMIMEHeaderKey("x-request-id"))
}
Output
Content-Type X-Request-Id

Reading a MIME header block

Reader.ReadMIMEHeader consumes lines up to the blank separator and returns a MIMEHeader (map[string][]string) with canonical keys; the reader is then positioned at the body.

package main

import (
    "bufio"
    "fmt"
    "net/textproto"
    "strings"
)

func main() {
    raw := "Content-Type: text/plain\r\n" +
        "X-Count: 3\r\n" +
        "\r\n"
    r := textproto.NewReader(bufio.NewReader(strings.NewReader(raw)))
    hdr, err := r.ReadMIMEHeader()
    if err != nil {
        panic(err)
    }
    fmt.Println(hdr.Get("content-type")) // case-insensitive lookup
    fmt.Println(hdr.Get("X-Count"))
}
Output
text/plain 3

Parsing a numeric response line

ReadResponse parses the code message form (and multi-line continuations) used by SMTP, FTP, and NNTP; the argument is the expected code class.

package main

import (
    "bufio"
    "fmt"
    "net/textproto"
    "strings"
)

func main() {
    r := textproto.NewReader(bufio.NewReader(strings.NewReader("220 smtp ready\r\n")))
    code, msg, err := r.ReadResponse(2) // expect a 2xx code
    fmt.Println(code, msg, err)
}
Output
220 smtp ready <nil>

Common Mistakes

Wrong
// Rolling your own header parse into a plain map:
m := map[string]string{}
for _, line := range strings.Split(block, "\n") {
    k, v, _ := strings.Cut(line, ":")
    m[k] = v // loses casing rules, multi-values, folded lines
}
Right
hdr, err := textproto.NewReader(br).ReadMIMEHeader()
if err != nil {
    return err
}
// hdr: map[string][]string, canonical keys, case-insensitive Get

Key Rules

  • Keys are canonicalized (Content-Type), so hdr.Get("content-type") is case-insensitive; a raw map keyed on original casing would miss.
  • ReadMIMEHeader reads up to the blank line and leaves the reader at the body — ideal for header-then-payload protocols.
  • ReadResponse(expectCode) parses code message plus multi-line continuations — the basis of SMTP/FTP/NNTP clients.
  • Protocol lines are CRLF-terminated (\r\n); wrap your input in a bufio.Reader as textproto.NewReader requires.
Practice Exercise

Parse a raw HTTP-style request line plus header block: use ReadLine for the request line and ReadMIMEHeader for the headers, then read the body. Show that Get("HOST") and Get("host") both resolve. Then simulate an SMTP greeting/reply exchange with ReadResponse, handling a multi-line 250- continuation. Explain why net/http and net/mail build on this package instead of re-implementing header parsing.

Pitfall: Forgetting that keys are stored canonically — iterating the underlying MIMEHeader map and matching against "content-type" fails because the key is "Content-Type"; always go through Get. And these protocols require \r\n line endings: a lone \n or a missing terminating blank line makes ReadMIMEHeader hang or error.
Connections: The engine under net/mail (yesterday's concept) and net/http header parsing; pairs with bufio for buffered reading and mime for media-type parsing; ReadResponse is what an SMTP/FTP client loop is built on.

Section 2 — Diagnostics & System Standard Library

runtime/trace: The Execution Tracer

Diagnostics

Why it matters: a CPU profile tells you where cycles are spent; the execution tracer tells you what every goroutine did over time — scheduling, blocking, syscalls, network waits, and GC pauses — with microsecond timestamps. It's the right tool for latency, lock contention, and "why isn't this parallel" questions that a flat profile can't answer, and you can annotate it with your own tasks and regions.

Capturing a trace to a file

trace.Start(w) begins recording to any io.Writer; trace.Stop() flushes it. Always pair them with defer.

package main

import (
    "os"
    "runtime/trace"
)

func main() {
    f, err := os.Create("trace.out")
    if err != nil {
        panic(err)
    }
    defer f.Close()

    if err := trace.Start(f); err != nil {
        panic(err)
    }
    defer trace.Stop() // flushes; a truncated file won't open

    work()
}

func work() {
    sum := 0
    for i := 0; i < 1_000_000; i++ {
        sum += i
    }
    _ = sum
}
Analyze with
go tool trace trace.out   # opens the interactive timeline in a browser

Annotating with tasks, regions, and logs

NewTask scopes a logical operation, WithRegion labels a span of work, and Log emits a point event — all appear on the timeline.

package main

import (
    "context"
    "os"
    "runtime/trace"
)

func main() {
    f, _ := os.Create("trace.out")
    defer f.Close()
    trace.Start(f)
    defer trace.Stop()

    ctx, task := trace.NewTask(context.Background(), "request")
    defer task.End()

    trace.WithRegion(ctx, "parse", func() {
        // ... parsing work shows up as a labeled region
    })
    trace.Log(ctx, "stage", "parsed")
}

Capturing a trace from a benchmark

go test can emit a trace of a benchmark run — often the easiest way to get a representative window.

go test -bench=. -trace=trace.out ./...
go tool trace trace.out

Common Mistakes

Wrong
trace.Start(f)
// ... long-running server, never stopped ...
os.Exit(0) // defers skipped: trace.Stop never runs -> unreadable file
Right
trace.Start(f)
defer trace.Stop() // flushes on return

// Capture a bounded window, then let the program continue.

Key Rules

  • Always pair trace.Start(w) with trace.Stop() (defer it); Stop flushes, and a truncated trace won't open in go tool trace.
  • Analyze with go tool trace file — an interactive timeline of goroutines, GC, syscalls, and network blocking.
  • Annotate your own spans with trace.NewTask, trace.WithRegion, and trace.Log for meaningful labels.
  • Use the tracer for scheduling/latency/blocking questions and pprof for "where is CPU/allocation spent" — they're complementary, not interchangeable.
Practice Exercise

Instrument a program that spins up a worker pool over a channel, wrap each job in a trace.WithRegion, and open the result in go tool trace to spot where goroutines block. Then capture a trace of a contended sync.Mutex workload and identify the serialization in the timeline. Finally, show that replacing defer trace.Stop() with an early os.Exit produces an unreadable trace, and explain why the tracer is unsuitable as an always-on profiler.

Pitfall: Leaving the tracer running unbounded — it carries real overhead and writes large files, so it's a targeted, bounded-window tool, not an always-on profiler. And because trace.Stop does the flush, any path that skips deferred calls (os.Exit, log.Fatal, a panic without recovery in the wrong place) leaves a truncated, unopenable trace.
Connections: Sibling to pprof (profiling) and runtime/metrics from earlier days; its tasks and regions are context-scoped, echoing today's context review and OpenTelemetry spans; ideal for diagnosing the goroutine/channel patterns reviewed above.

os/user: Users, Groups & Home Directories

System

Why it matters: CLIs and services routinely need the current user, their home directory (for ~/.config paths), or to resolve a UID to a name for logging and ownership. os/user does this portably across macOS, Linux, and Windows — no shelling out to id and no hand-parsing /etc/passwd.

The current user

user.Current() returns a *user.User with the process's identity and home directory.

package main

import (
    "fmt"
    "os/user"
)

func main() {
    u, err := user.Current()
    if err != nil {
        panic(err)
    }
    fmt.Println("username:", u.Username)
    fmt.Println("uid:", u.Uid)
    fmt.Println("home:", u.HomeDir)
}
Output (representative — varies by machine)
username: andy uid: 501 home: /Users/andy

Looking up by name or id — and handling misses

Lookup/LookupId resolve other users; a miss returns a typed UnknownUserError you can match with errors.As.

package main

import (
    "errors"
    "fmt"
    "os/user"
)

func main() {
    _, err := user.Lookup("nonesuch")
    var unknown user.UnknownUserError
    if errors.As(err, &unknown) {
        fmt.Println("no such user")
    }
}
Output

Group membership

GroupIds lists a user's group IDs; LookupGroupId resolves each to a name.

u, _ := user.Current()
gids, err := u.GroupIds()
if err != nil {
    // may fail in a static (CGO_ENABLED=0) build
}
for _, gid := range gids {
    if g, err := user.LookupGroupId(gid); err == nil {
        fmt.Println(g.Name)
    }
}

Common Mistakes

Wrong
// Assumes a Linux layout and integer UID:
home := "/home/" + username
uid, _ := strconv.Atoi(u.Uid) // Uid is a STRING id
Right
u, _ := user.Current()
home := u.HomeDir // /Users/... on macOS, a profile path on Windows
// Or, if you only need the home dir:
home, _ = os.UserHomeDir()

Key Rules

  • Current() gives the process's user; Lookup/LookupId resolve others; misses return UnknownUserError/UnknownUserIdError (match with errors.As).
  • Uid and Gid are strings, not ints — they must portably represent Windows SIDs as well as POSIX numeric IDs.
  • For just a directory, prefer os.UserHomeDir() or os.UserConfigDir() — simpler and dependency-free.
  • Some lookups (GroupIds, name resolution) may rely on cgo; a pure-Go CGO_ENABLED=0 build falls back to /etc/passwd parsing and can behave differently in minimal containers.
Practice Exercise

Write configDir(app string) (string, error) that builds <home>/.config/<app> using user.Current().HomeDir and filepath.Join, then rewrite it with os.UserConfigDir and compare. Handle a lookup of a nonexistent user with errors.As against user.UnknownUserError. Finally, print the current user's group names, and note what happens when you cross-compile with CGO_ENABLED=0.

Pitfall: Treating Uid as an integer or hardcoding /home/<name> — on macOS home is under /Users, and on Windows the "Uid" is a SID with a profile-path home. And GroupIds/name lookups can fail or return incomplete data in static (CGO_ENABLED=0) builds, which is exactly the configuration many minimal container images use.
Connections: Pairs with os.UserHomeDir/UserConfigDir and path/filepath (the os & filepath day) for building config paths; feeds the configuration/env-var material from Production Go; and its cgo/static-build caveat ties into cross-compilation and build tags.
