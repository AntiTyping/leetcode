gRPC & Protocol Buffers

Advanced Networking • overdue 12d • rep 0

Quick Reminder

gRPC uses Protocol Buffers as both the interface definition language and the compact binary wire format, carried over HTTP/2. You write a .proto, run protoc with the Go plugins to generate type-safe stubs, then the server implements the generated interface and the client calls methods like local functions. It supports unary plus three streaming modes (server, client, bidi).

Key Pattern

// greet.proto
syntax = "proto3";
package greet;
option go_package = "example/greet";

service Greeter {
  rpc SayHello(HelloRequest) returns (HelloReply);
}
message HelloRequest { string name = 1; }
message HelloReply   { string message = 1; }
// server.go — uses code generated by protoc-gen-go + protoc-gen-go-grpc
package main

import (
    "context"
    "log"
    "net"

    "google.golang.org/grpc"
    "example/greet" // generated package
)

type server struct {
    greet.UnimplementedGreeterServer // forward-compatible embedding
}

func (s *server) SayHello(ctx context.Context, in *greet.HelloRequest) (*greet.HelloReply, error) {
    return &greet.HelloReply{Message: "Hello, " + in.GetName()}, nil
}

func main() {
    lis, _ := net.Listen("tcp", ":50051")
    s := grpc.NewServer()
    greet.RegisterGreeterServer(s, &server{})
    log.Fatal(s.Serve(lis)) // a client calling SayHello{name:"Ada"} gets "Hello, Ada"
}
Output (client side)
client got: Hello, Ada
Challenge Exercise

Add a server-streaming RPC rpc ListGreetings(HelloRequest) returns (stream HelloReply) to the proto. Regenerate the stubs and implement the handler so it streams three replies, then write a client loop that calls stream.Recv() until io.EOF. Explain why you must never renumber or reuse an existing field tag when evolving the message.

Pitfall: Omitting the embedded UnimplementedGreeterServer in your server struct. Without it, adding a new RPC to the service later breaks compilation of every existing implementation; with it, unimplemented methods return Unimplemented and your code keeps building. Also never change a field's tag number — the tag, not the name, is the wire identity.

Generics Advanced: Type Sets & Constraints

Modern Go • overdue 12d • rep 0

Quick Reminder

A constraint interface can hold a type set (a union of types, with ~ admitting named types whose underlying type matches), method elements, or both. The compiler intersects all type sets; an operator is allowed only if it is valid for every type in the set — so < needs an ordered set, and comparable is the built-in constraint for ==. The core type of a constraint decides which built-in operations are legal.

Key Pattern

package main

import "fmt"

// All listed types support < and >, so Max can compare with >.
type Ordered interface {
    ~int | ~int64 | ~float64 | ~string
}

func Max[T Ordered](a, b T) T {
    if a > b {
        return a
    }
    return b
}

func main() {
    fmt.Println(Max(3, 7))
    fmt.Println(Max("apple", "banana"))
}
Output
7 banana
Challenge Exercise

Write Clamp[T Ordered](v, lo, hi T) T using the same constraint. Then design a constraint Stringer-plus-numeric that requires both a String() string method and an underlying integer type, and explain why such a constraint can still be useful even though its type set is small. Finally, demonstrate that comparable is required to use a type parameter as a map key.

Pitfall: Combining a method element with a type set whose members don't have that method — the intersection is the empty type set, so no type can ever satisfy the constraint and instantiation fails. Make sure the listed types actually implement any methods you also require.

Fuzz Testing

Modern Go • overdue 12d • rep 0

Quick Reminder

Native fuzzing (Go 1.18+) lives in go test. Write FuzzXxx(f *testing.F), seed it with f.Add(...), and put the property to check inside f.Fuzz(func(t *testing.T, in ...){...}). Run go test -fuzz=FuzzXxx; the engine mutates inputs to maximize coverage and saves any failing input to testdata/fuzz/ as a permanent regression case.

Key Pattern

// reverse.go
package main

func Reverse(s string) string {
    r := []rune(s)
    for i, j := 0, len(r)-1; i < j; i, j = i+1, j-1 {
        r[i], r[j] = r[j], r[i]
    }
    return string(r)
}

func main() {}
// reverse_test.go
func FuzzReverse(f *testing.F) {
    f.Add("hello") // seed corpus
    f.Fuzz(func(t *testing.T, s string) {
        if Reverse(Reverse(s)) != s { // round-trip invariant
            t.Errorf("round-trip failed for %q", s)
        }
    })
}
Output (the classic invalid-UTF-8 discovery)
fuzz: elapsed: 0s, gathering baseline coverage: 1/1 completed --- FAIL: FuzzReverse (0.02s) reverse_test.go:5: round-trip failed for "\x80" Failing input written to testdata/fuzz/FuzzReverse/...
Challenge Exercise

Fix Reverse so the round-trip holds for arbitrary bytes (hint: operate on bytes, or guard with utf8.ValidString), then re-run the fuzzer and confirm it passes. Next, write a fuzz target for a small ParseQuery function whose invariant is Format(Parse(s)) being re-parseable. Commit the auto-generated failing corpus file and explain why it belongs in version control.

Pitfall: Assuming a Go string is always valid UTF-8. Strings can hold arbitrary bytes, and []rune(s) replaces invalid bytes with U+FFFD — which is exactly why the double-reverse invariant breaks on input like "\x80". Fuzzing surfaces these edge cases that hand-written tests miss.

Range-over-func Iterators

Modern Go • overdue 12d • rep 0

Quick Reminder

Go 1.23 lets for range drive a function. An iterator is just a function matching iter.Seq[V] = func(yield func(V) bool) (or iter.Seq2[K,V]). The loop calls your function, handing it a yield; you call yield(v) for each element and must stop if it returns false (the consumer used break or return). This gives custom iteration with no intermediate slice allocation.

Key Pattern

package main

import (
    "fmt"
    "iter"
)

func Count(n int) iter.Seq[int] {
    return func(yield func(int) bool) {
        for i := 0; i < n; i++ {
            if !yield(i) { // consumer broke out: stop producing
                return
            }
        }
    }
}

func main() {
    for v := range Count(5) {
        if v == 3 {
            break
        }
        fmt.Print(v, " ")
    }
    fmt.Println("done")
}
Output
0 1 2 done
Challenge Exercise

Write a Filter[V any](seq iter.Seq[V], keep func(V) bool) iter.Seq[V] adaptor that wraps another iterator, forwarding yield's boolean correctly. Compose it with Count to print even numbers under 10. Then write an iter.Seq2[int,string] that yields index/word pairs from a slice, and consume it with a two-variable range.

Pitfall: Ignoring the boolean that yield returns. If you keep producing after yield reports false, a consumer's break won't actually stop your loop — you leak work and, worse, may run cleanup or hold resources past the point the caller abandoned iteration. Always return the moment yield is false.

Structured Concurrency Patterns

Modern Go • overdue 12d • rep 0

Quick Reminder

Structured concurrency means every goroutine has a bounded lifetime tied to the scope that started it — it must not outlive the function that spawned it. You enforce this by always joining (WaitGroup.Wait or errgroup.Wait) before returning, and by threading a context so cancellation propagates. The opposite — fire-and-forget goroutines — is the classic source of leaks.

Key Pattern

package main

import (
    "context"
    "fmt"
    "sync"
    "time"
)

func process(ctx context.Context, items []int) []int {
    out := make([]int, len(items)) // disjoint slots: no mutex needed
    var wg sync.WaitGroup
    for i, v := range items {
        wg.Add(1)
        go func() {
            defer wg.Done()
            select {
            case <-ctx.Done():
                return
            case <-time.After(time.Millisecond):
                out[i] = v * v
            }
        }()
    }
    wg.Wait() // join before returning: no goroutine outlives this call
    return out
}

func main() {
    fmt.Println(process(context.Background(), []int{1, 2, 3, 4}))
}
Output
[1 4 9 16]
Challenge Exercise

Rewrite process with errgroup.WithContext so the first worker returning an error cancels the rest, and have process return ([]int, error). Then add a deadline via context.WithTimeout and confirm that on timeout every goroutine observes ctx.Done() and the function still joins cleanly (verify with -race and a goroutine-leak check).

Pitfall: Launching a goroutine and returning without waiting for it (go work() with no join). The goroutine now outlives its caller, can write to freed scope, and leaks if it blocks forever. Every goroutine you start should have a clearly owned join point and a cancellation path.

Advanced Error Handling: errors.Join & Sentinel Errors

Modern Go • overdue 12d • rep 0

Quick Reminder

A sentinel error is a package-level var ErrX = errors.New(...) that callers test with errors.Is. Wrap with %w to preserve the chain so errors.Is/errors.As can still see the cause through layers of context. errors.Join (Go 1.20) bundles several errors into one value that Is-matches any of them — ideal for accumulating multiple validation or cleanup failures.

Key Pattern

package main

import (
    "errors"
    "fmt"
)

var ErrNotFound = errors.New("not found")

func lookup(id int) error {
    if id < 0 {
        return fmt.Errorf("lookup %d: %w", id, ErrNotFound) // %w wraps
    }
    return nil
}

func main() {
    err := errors.Join(lookup(-1), lookup(-2)) // combine; nils dropped
    fmt.Println(errors.Is(err, ErrNotFound))
    fmt.Println(err)
}
Output
true lookup -1: not found lookup -2: not found
Challenge Exercise

Write validate(u User) error that checks several fields and returns errors.Join of all violations at once (so the caller sees every problem, not just the first). Define a typed error type FieldError struct{ Field string } implementing Error(), add some to the join, and use errors.As to pull the first FieldError back out of the combined error.

Pitfall: Wrapping with %v instead of %w. %v flattens the cause into a plain string, so errors.Is(err, ErrNotFound) returns false even though the message mentions it. Use %w to keep the chain, and compare with errors.Is — never == on a wrapped error.

Section 1 — Modern Standard Library

math/rand/v2: The Redesigned Randomness API

Std Lib

Why it matters: math/rand/v2 (Go 1.22) is the standard library's first /v2 package. It fixes long-standing warts of math/rand: the global generator is automatically and securely seeded (no more rand.Seed footgun), the methods are renamed for clarity (IntN not Intn), and it ships modern algorithms like PCG and ChaCha8.

Auto-seeded global functions

The top-level functions use a per-process, automatically seeded source. You never call Seed — every run produces a different stream by default, which is what you want for non-cryptographic randomness.

package main

import (
    "fmt"
    "math/rand/v2"
)

func main() {
    // No rand.Seed needed — the global source is auto-seeded.
    n := rand.IntN(100) // uniform in [0, 100)
    fmt.Println("in range:", n >= 0 && n < 100)

    fmt.Println("perm len:", len(rand.Perm(4))) // a permutation of 0..3

    g := rand.N(int32(10)) // generic N works for any int/uint kind
    fmt.Println("generic ok:", g >= 0 && g < 10)
}
Output
in range: true perm len: 4 generic ok: true

Reproducible streams with an explicit source

For tests or simulations you want determinism: construct your own generator from a seeded source. Two generators built from the same PCG seed produce the identical sequence.

package main

import (
    "fmt"
    "math/rand/v2"
)

func main() {
    a := rand.New(rand.NewPCG(1, 2))
    b := rand.New(rand.NewPCG(1, 2)) // same seed pair

    same := true
    for i := 0; i < 5; i++ {
        if a.Uint64() != b.Uint64() {
            same = false
        }
    }
    fmt.Println("same seed, same stream:", same)
}
Output
same seed, same stream: true

Common Mistakes

Wrong
// Legacy v1 habits: manual global Seed
// and the old Intn name.
rand.Seed(time.Now().UnixNano())
n := rand.Intn(6)
Right
import "math/rand/v2"
// Global is auto-seeded; use IntN.
n := rand.IntN(6) // no Seed call at all

Key Rules

  • The global functions are auto-seeded; there is no Seed for the global generator.
  • Names changed: use IntN, Int64N, Uint64, and the generic N[T] (capital N) — the v1 Intn/Int63n are gone.
  • For reproducibility build rand.New(rand.NewPCG(seed1, seed2)) and use its methods.
  • It is fast but not cryptographically secure — use crypto/rand for secrets.
Practice Exercise

Build a WeightedPicker that, given map[string]int weights, returns keys proportionally using rand.IntN over the cumulative total. Make it accept an injected *rand.Rand so a test can seed a PCG source and assert a deterministic sequence. Then use rand.Shuffle to randomize a deck and confirm a fixed seed reproduces the same order.

Pitfall: Reaching for math/rand/v2 to generate tokens, passwords, or session IDs. Its output is predictable to anyone who can observe enough values — it is a statistical PRNG, not a secure one. Security-sensitive randomness must come from crypto/rand.
Connections: The reproducible-source idea pairs with deterministic testing (seed a PCG for stable test data), and the security caveat ties back to the crypto material from earlier guides.

encoding/csv: Robust Tabular I/O

Std Lib

Why it matters: CSV looks trivial until a field contains a comma, a quote, or a newline. encoding/csv handles RFC 4180 quoting, configurable delimiters, and column-count validation for you — replacing the fragile strings.Split(line, ",") that breaks on real-world data.

Reading records

A csv.Reader wraps any io.Reader. Read() returns one record (a []string); ReadAll() returns them all. Quoted fields keep their embedded commas intact.

package main

import (
    "encoding/csv"
    "fmt"
    "strings"
)

func main() {
    in := "name,score\n\"Doe, Jane\",91\nAda,100\n"
    r := csv.NewReader(strings.NewReader(in))

    records, err := r.ReadAll()
    if err != nil {
        fmt.Println("err:", err)
        return
    }
    for _, rec := range records {
        fmt.Printf("%d fields: %v\n", len(rec), rec)
    }
}
Output
2 fields: [name score] 2 fields: [Doe, Jane 91] 2 fields: [Ada 100]

Writing records (and remembering to Flush)

A csv.Writer buffers output and quotes fields automatically when they contain the delimiter or a newline. You must call Flush() and then check Error().

package main

import (
    "encoding/csv"
    "fmt"
    "os"
)

func main() {
    w := csv.NewWriter(os.Stdout)
    rows := [][]string{
        {"city", "note"},
        {"Paris", "caf, croissant"}, // the comma forces quoting
    }
    for _, row := range rows {
        w.Write(row)
    }
    w.Flush() // buffered rows are written here
    if err := w.Error(); err != nil {
        fmt.Println("flush error:", err)
    }
}
Output
city,note Paris,"caf, croissant"

Common Mistakes

Wrong
// Naive split: a quoted comma splits
// the row into the wrong number of fields.
fields := strings.Split(line, ",")
Right
r := csv.NewReader(src)
rec, err := r.Read() // honors quotes,
// embedded commas, and newlines

Key Rules

  • Reader: Read() for one record, ReadAll() for everything; Writer: Write() then Flush() and check Error().
  • Set Comma for other delimiters (e.g. '\t' for TSV); set FieldsPerRecord to enforce a fixed column count.
  • LazyQuotes and TrimLeadingSpace relax parsing for messy input.
  • For huge files, stream with Read() in a loop and set ReuseRecord = true to avoid per-row allocations.
Practice Exercise

Read a TSV (Comma = '\t') of name, dept, salary, sum the salary column, and write a filtered CSV of only the rows above the average into a bytes.Buffer. Set FieldsPerRecord = 3 and feed a ragged row to observe the error. Then stream a million-row file with Read() + ReuseRecord and compare allocations against ReadAll().

Pitfall: Forgetting w.Flush() (and ignoring w.Error()). The writer buffers rows, so without a flush your file ends up empty or truncated — and a write error that happened mid-buffer is only surfaced by Error() after the flush.
Connections: Builds directly on the io.Reader/io.Writer abstractions and pairs with bufio for buffering. For human-readable (rather than machine-parseable) tables, see text/tabwriter below.

hash/maphash: Fast, Seeded Hashing

Std Lib

Why it matters: When you need to hash bytes, strings, or comparable values for sharding, bloom filters, or your own hash tables, hash/maphash gives you a fast, well-distributed, randomly seeded hash — the same engine behind Go's built-in maps. The random seed makes it resistant to hash-flooding (HashDoS) attacks that fixed-seed hashes are vulnerable to.

Hashing strings under a shared seed

A maphash.Seed selects the hash function. Equal inputs hashed under the same seed give equal results; different inputs almost certainly differ.

package main

import (
    "fmt"
    "hash/maphash"
)

func main() {
    seed := maphash.MakeSeed() // random, per process

    a := maphash.String(seed, "hello")
    b := maphash.String(seed, "hello")
    c := maphash.String(seed, "world")

    fmt.Println("equal inputs, equal hash:", a == b)
    fmt.Println("different inputs differ:", a != c)
}
Output
equal inputs, equal hash: true different inputs differ: true

Hashing any comparable value for sharding

maphash.Comparable[T] (Go 1.24) hashes any comparable type — structs, arrays, interface values — so you can route a key to one of N shards without writing a hash function.

package main

import (
    "fmt"
    "hash/maphash"
)

type Key struct {
    Region string
    ID     int
}

func main() {
    seed := maphash.MakeSeed()
    const shards = 8

    k := Key{Region: "us", ID: 42}
    shard := maphash.Comparable(seed, k) % shards

    fmt.Println("shard in range:", shard < shards)
}
Output
shard in range: true

Common Mistakes

Wrong
// Fixed-seed hash on untrusted keys:
// an attacker can force collisions
// and degrade your map to O(n) (HashDoS).
h := fnv.New64a()
Right
seed := maphash.MakeSeed() // random per run
sum := maphash.String(seed, key)
// collisions can't be precomputed

Key Rules

  • Make one Seed with maphash.MakeSeed() and reuse it for all keys that must hash into the same space.
  • maphash.String/maphash.Bytes for strings/bytes; maphash.Comparable[T] for any comparable type.
  • Hashes are seed-dependent: they are not stable across process restarts — never persist or transmit them.
  • Fast and DoS-resistant, but not cryptographic — don't use it for signatures or integrity checks.
Practice Exercise

Build a sharded counter: N independently mutex-guarded map[string]int shards, where the shard index is maphash.String(seed, key) % N. Hammer it from many goroutines under -race and check the per-shard distribution is roughly even. Then implement a tiny open-addressing string set that uses maphash for the probe index.

Pitfall: Persisting a maphash value to disk or comparing values produced in different processes. Because the seed is randomized per process, the same input hashes differently each run — comparisons are only meaningful within a single process under one seed.
Connections: The sharding idea complements the locking trade-offs from the sync package and the sync.Map discussion, and the DoS angle ties back to security-minded standard-library use.

Section 2 — Data Structures & Text

container/list: Doubly Linked Lists

Data Structures

Why it matters: container/list is a ready-made doubly linked list with O(1) insertion and removal at any element you hold a handle to. That is exactly what an LRU cache or a queue needs — cheaply moving or evicting an arbitrary node — something a slice can't do without O(n) shifting.

Building and traversing a list

list.New() returns a *list.List. PushBack/PushFront return the *list.Element you keep for later O(1) operations. Iterate from Front() via Next().

package main

import (
    "container/list"
    "fmt"
)

func main() {
    l := list.New()
    a := l.PushBack("a")  // [a]
    l.PushBack("b")       // [a b]
    l.PushFront("z")      // [z a b]
    l.InsertAfter("a2", a) // [z a a2 b]

    for e := l.Front(); e != nil; e = e.Next() {
        fmt.Print(e.Value, " ")
    }
    fmt.Println()
    fmt.Println("len:", l.Len())
}
Output
z a a2 b len: 4

O(1) move and evict (the LRU primitive)

Promoting a recently used node to the front and evicting the least-recent from the back are both O(1) — the operations an LRU cache performs on every access.

package main

import (
    "container/list"
    "fmt"
)

func main() {
    l := list.New()
    l.PushBack(1)
    y := l.PushBack(2)
    l.PushBack(3) // [1 2 3]

    l.MoveToFront(y)   // promote 2 -> [2 1 3]
    l.Remove(l.Back()) // evict least-recent (3) -> [2 1]

    var got []int
    for e := l.Front(); e != nil; e = e.Next() {
        got = append(got, e.Value.(int)) // Value is any -> assert
    }
    fmt.Println(got)
}
Output
[2 1]

Common Mistakes

Wrong
// Slice as a queue, deleting the middle:
// O(n) shift on every removal.
q = append(q[:i], q[i+1:]...)
Right
// Hold the *list.Element; removal is O(1)
// with no shifting of other nodes.
l.Remove(elem)

Key Rules

  • list.New() gives a *list.List; PushBack/PushFront return the *list.Element to keep.
  • Traverse with for e := l.Front(); e != nil; e = e.Next() (or Back()/Prev()).
  • Element.Value is any — type-assert on read; consider a typed wrapper for safety.
  • It is not concurrency-safe — guard a shared list with a mutex.
Practice Exercise

Implement an LRUCache[K comparable, V any] with capacity c: keep a map[K]*list.Element plus a *list.List whose elements store key/value pairs. Get moves the node to the front and returns the value; Put inserts at front and, when over capacity, removes the back node and deletes its key from the map. Verify every operation is O(1).

Pitfall: Keeping and reusing an *list.Element after you've Removed it. A removed element's links are stale; moving or inserting it again corrupts the list's invariants with no error. Drop the reference (and any map entry pointing to it) as soon as you remove a node.
Connections: Complements container/heap (priority ordering) from the previous guide and the sync.Map caching discussion — an LRU built on container/list is the bounded-memory counterpart to those caches.

text/tabwriter: Aligned Column Output

Std Lib

Why it matters: Aligning columns by counting spaces is tedious and falls apart the moment a value's width changes. text/tabwriter turns tab-separated cells into neatly aligned columns automatically — it's the same elastic-tabstop engine gofmt and countless CLIs use for tabular output.

A self-aligning table

Wrap any io.Writer with tabwriter.NewWriter, separate cells with '\t', write your rows, then Flush(). The writer measures each column and pads to align.

package main

import (
    "fmt"
    "os"
    "text/tabwriter"
)

func main() {
    // minwidth=0, tabwidth=4, padding=2, padchar=' ', flags=0
    w := tabwriter.NewWriter(os.Stdout, 0, 4, 2, ' ', 0)
    fmt.Fprintln(w, "NAME\tLANG\tSTARS")
    fmt.Fprintln(w, "ripgrep\tRust\t40000")
    fmt.Fprintln(w, "fzf\tGo\t60000")
    w.Flush() // alignment is computed here
}
Output
NAME LANG STARS ripgrep Rust 40000 fzf Go 60000

Padding character and minimum width

The constructor's parameters tune the layout. Here a minimum cell width of 8 and a '.' pad character make the column boundaries visible (handy when debugging alignment). A trailing tab makes the second column a real, padded column.

package main

import (
    "fmt"
    "os"
    "text/tabwriter"
)

func main() {
    // minwidth=8, tabwidth=0, padding=0, padchar='.'
    w := tabwriter.NewWriter(os.Stdout, 8, 0, 0, '.', 0)
    fmt.Fprintln(w, "id\tstatus\t")
    fmt.Fprintln(w, "1\tok\t")
    fmt.Fprintln(w, "200\tfailed\t")
    w.Flush()
}
Output
id......status.. 1.......ok...... 200.....failed..

Common Mistakes

Wrong
// Hand-aligned with fixed spaces:
// breaks as soon as a value is longer.
fmt.Printf("%s    %s\n", name, lang)
Right
w := tabwriter.NewWriter(os.Stdout, 0, 4, 2, ' ', 0)
fmt.Fprintf(w, "%s\t%s\n", name, lang)
w.Flush() // columns align automatically

Key Rules

  • NewWriter(out, minwidth, tabwidth, padding, padchar, flags); separate cells with '\t', end the line, then Flush().
  • Forgetting Flush() means nothing is written — the writer buffers to measure column widths.
  • The final cell on a line (no trailing tab) is not part of an aligned column; add a trailing '\t' to align the last column.
  • tabwriter.AlignRight right-justifies cells; other flags like Debug and DiscardEmptyColumns tune behavior.
Practice Exercise

Render a ps-style process table from a slice of structs: left-align NAME, right-align the numeric PID and MEM columns (use tabwriter.AlignRight and remember the trailing tab so the last column aligns). Then switch padchar to '.' to produce a dotted-leader menu (item on the left, price on the right).

Pitfall: Interleaving unflushed tabwriter writes with direct prints to the same underlying writer. Column widths are computed per Flush, so mixing the two scrambles alignment and can emit rows out of order. Write all tabular rows, Flush once, then continue with other output.
Connections: Builds on the io.Writer abstraction and is the human-readable counterpart to encoding/csv — CSV for machines, tabwriter for terminals.
