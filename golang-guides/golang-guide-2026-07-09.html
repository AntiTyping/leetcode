os & filepath

Standard Library Mastery • 15 days since review • rep 3

Quick Reminder

os talks to the operating system — files (os.Open, os.ReadFile, os.WriteFile), metadata (os.Stat), environment (os.Getenv, os.LookupEnv), and process (os.Args, os.Exit). path/filepath manipulates paths using the OS separator (\ on Windows), unlike path which is always slash-based and meant for URLs. filepath.Join cleans as it joins; filepath.WalkDir (Go 1.16+) traverses a tree using cheap fs.DirEntry values instead of a full Stat per file.

Key Pattern

err := filepath.WalkDir(root, func(p string, d fs.DirEntry, err error) error {
    if err != nil {
        return err // a directory couldn't be read
    }
    if !d.IsDir() && filepath.Ext(p) == ".go" {
        fmt.Println(filepath.Base(p))
    }
    return nil
})

data, err := os.ReadFile(filepath.Join(dir, "config.json"))
if errors.Is(err, os.ErrNotExist) {
    // file is absent (portable check, not string matching)
}
Challenge Exercise

Write duLike(root string) that walks a directory with WalkDir and sums file sizes per extension. Use filepath.Join/Ext/Base and skip a directory named .git by returning filepath.SkipDir. Then read a missing file and branch on errors.Is(err, os.ErrNotExist). Finally, show why path.Join is wrong for a real filesystem path on Windows.

#1 Pitfall: Using the path package (or hand-concatenating with "/") for filesystem paths — it breaks on Windows; use filepath. Also: string-matching error text instead of errors.Is(err, os.ErrNotExist), and forgetting to check the err argument inside a WalkDir callback (an unreadable subdir silently aborts the walk).

net/http: Handlers, ServeMux, Middleware, Client

Standard Library Mastery • 15 days since review • rep 3

Quick Reminder

A handler is anything implementing ServeHTTP(w, r); http.HandlerFunc adapts a plain function. ServeMux routes by pattern — since Go 1.22 patterns take a method and wildcards ("GET /users/{id}", read with r.PathValue("id")). Middleware is just func(http.Handler) http.Handler, composed outside-in. On the client side, never use http.DefaultClient for real traffic (no timeout); build an http.Client{Timeout: ...}, use http.NewRequestWithContext, and always defer resp.Body.Close().

Key Pattern

func logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}

mux := http.NewServeMux()
mux.HandleFunc("GET /users/{id}", func(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "user %s", r.PathValue("id")) // Go 1.22 routing
})
srv := &http.Server{Addr: ":8080", Handler: logging(mux)}
Challenge Exercise

Build a mux with GET /health and POST /echo (echo the body back), then wrap it in a logging + panic-recovery middleware chain. On the client side, write a fetch(ctx, url) using an http.Client with a 2s timeout and NewRequestWithContext, and prove that omitting resp.Body.Close() leaks connections. Show that calling w.WriteHeader(200) then w.WriteHeader(500) logs a "superfluous WriteHeader" warning.

#1 Pitfall: http.DefaultClient and http.Get have no timeout — a slow server hangs the goroutine forever. Also: forgetting defer resp.Body.Close() (leaks the connection and defeats keep-alive), and writing to the body before WriteHeader, which implicitly sends 200 and makes your later status code a no-op.

sort Package

Standard Library Mastery • 15 days since review • rep 3

Quick Reminder

sort.Slice(s, less) sorts in place with a comparison closure (unstable); sort.SliceStable preserves the order of equal elements. sort.Search is a generic binary search returning the insertion index of the first element for which the predicate is true. Since Go 1.21 the generic slices.Sort/slices.SortFunc are faster and type-safe — prefer them for new code, and use cmp.Compare (returning -1/0/+1) inside SortFunc.

Key Pattern

people := []Person{{"Ann", 30}, {"Bo", 30}, {"Cy", 25}}

// Modern (Go 1.21+): compare by age, then name to break ties.
slices.SortFunc(people, func(a, b Person) int {
    if c := cmp.Compare(a.Age, b.Age); c != 0 {
        return c
    }
    return cmp.Compare(a.Name, b.Name)
})

// Binary search: first index with value >= 25 in a sorted []int.
i := sort.Search(len(xs), func(i int) bool { return xs[i] >= 25 })
Challenge Exercise

Sort a slice of structs by two keys (primary ascending, secondary descending) with both sort.Slice and slices.SortFunc + cmp.Compare. Demonstrate stability by sorting the same data with sort.Slice vs sort.SliceStable and observing when equal elements reorder. Then use sort.Search to find an insertion point and explain why the predicate must be monotonic (false...false true...true).

#1 Pitfall: A less func that isn't a strict weak ordering (e.g. returns true for equal elements, or uses <=) — it can panic or produce garbage. And sort.Search's predicate must be monotonic; a non-monotonic condition returns a meaningless index. Remember sort.Slice is unstable — reach for SliceStable/SortStableFunc when tie order matters.

unsafe Package

Advanced Patterns • 8 days since review • rep 2

Quick Reminder

unsafe.Pointer is the universal pointer that bridges otherwise-incompatible pointer types, bypassing the type system. Sizeof/Alignof/Offsetof report memory layout; Go 1.17+ adds unsafe.Slice, unsafe.String, unsafe.SliceData, and unsafe.StringData for safe(r) slice/string construction. The classic use is a zero-copy []byte↔string conversion. It is outside the Go 1 compatibility promise for layout and must follow the documented valid-conversion patterns.

Key Pattern

// Zero-copy []byte -> string (no allocation/copy).
func b2s(b []byte) string {
    if len(b) == 0 {
        return ""
    }
    return unsafe.String(unsafe.SliceData(b), len(b))
}

// The single-expression rule: pointer arithmetic must not be split.
p := unsafe.Pointer(uintptr(unsafe.Pointer(&s)) + unsafe.Offsetof(s.Field))
Challenge Exercise

Implement zero-copy b2s and s2b with unsafe.String/unsafe.Slice, then show the danger: mutate the backing array of a string produced by b2s and observe corruption (strings must be immutable). Use unsafe.Offsetof to reach a struct field via pointer arithmetic in a single expression, and explain why storing the intermediate uintptr in a variable is a bug the GC can bite.

#1 Pitfall: Splitting pointer arithmetic across statements — u := uintptr(unsafe.Pointer(p)); ... unsafe.Pointer(u+off) lets the GC reclaim or the object become untracked between the two lines. Convert in one expression. And a zero-copy b2s shares memory: mutating the []byte afterward violates string immutability and corrupts anything holding that string.

Code Generation

Advanced Patterns • 8 days since review • rep 2

Quick Reminder

A //go:generate comment directive attaches a command to a source file; running go generate ./... executes every such directive. Crucially, go generate is never triggered by go build or go test — you (or CI) run it deliberately and commit the output. Common generators: stringer (enum String() methods), mockgen, and protoc. Generated files carry the canonical header // Code generated ... DO NOT EDIT. so tools and reviewers recognize them.

Key Pattern

//go:generate stringer -type=Pill
type Pill int

const (
    Placebo Pill = iota
    Aspirin
    Ibuprofen
)
// Run:  go generate ./...
// Emits pill_string.go with:
//   // Code generated by "stringer -type=Pill"; DO NOT EDIT.
Challenge Exercise

Add a //go:generate stringer -type=State directive to an enum and run go generate ./...; confirm go build alone does not regenerate it. Write a tiny generator of your own (a main.go invoked by a directive) that emits a _gen.go file with the exact DO NOT EDIT header. Explain why the directive comment must have no space between // and go:generate.

#1 Pitfall: Expecting go build/go test to run generators — they don't; stale generated code ships unless you run go generate in CI. Also: writing // go:generate with a space (it's then an ordinary comment, silently ignored), and hand-editing a DO NOT EDIT file whose changes vanish on the next regeneration.

bufio: Buffered I/O

I/O & Networking • 7 days since review • rep 2

Quick Reminder

bufio wraps an io.Reader/io.Writer to batch small operations into fewer syscalls. bufio.Scanner is the ergonomic way to read line-by-line or token-by-token (ScanLines by default, ScanWords, or a custom SplitFunc); it strips the delimiter and stops on EOF. A bufio.Writer buffers writes — you must Flush (defer it) or buffered bytes are lost. Scanner has a default 64 KiB token cap; raise it with Scanner.Buffer.

Key Pattern

sc := bufio.NewScanner(r)
sc.Buffer(make([]byte, 0, 64*1024), 1024*1024) // allow up to 1 MiB lines
for sc.Scan() {
    line := sc.Text() // newline already stripped
    process(line)
}
if err := sc.Err(); err != nil { // EOF is NOT reported here
    log.Fatal(err)
}

w := bufio.NewWriter(f)
defer w.Flush() // without this, buffered output never reaches f
fmt.Fprintln(w, "buffered line")
Challenge Exercise

Read a file line-by-line with bufio.Scanner, count words with a second scanner using ScanWords, and always check sc.Err() after the loop. Feed it a line longer than 64 KiB to trigger bufio.ErrTooLong, then fix it with Scanner.Buffer. Finally, write output through a bufio.Writer and demonstrate that forgetting Flush silently drops the last buffered bytes.

#1 Pitfall: Forgetting w.Flush() on a bufio.Writer — the tail of your output silently disappears. On the read side, the default Scanner aborts with bufio.ErrTooLong on lines over 64 KiB (raise it with Buffer), and skipping the post-loop sc.Err() check hides real read errors, since a normal EOF ends the loop without setting one.

Section 1 — Symmetric Encryption & Key Formats

crypto/aes & cipher.GCM: Authenticated Symmetric Encryption

Security

Why it matters: to encrypt data at rest or a session token you want authenticated encryption — confidentiality and tamper detection in one step. AES is the block cipher; GCM is the AEAD mode that wraps it, producing a ciphertext that Open refuses to decrypt if a single bit was flipped. Rolling your own CBC/ECB by hand is the classic path to a silent, unauthenticated, forgeable mess.

Encrypting with AES-GCM

Derive a cipher.Block from a 32-byte key, wrap it in GCM, and Seal with a fresh random nonce. Prepending the nonce to the ciphertext is the idiomatic way to keep them together.

package main

import (
    "crypto/aes"
    "crypto/cipher"
    "crypto/rand"
    "encoding/hex"
    "fmt"
)

func encrypt(key, plaintext []byte) ([]byte, error) {
    block, err := aes.NewCipher(key) // key: 16, 24, or 32 bytes
    if err != nil {
        return nil, err
    }
    gcm, err := cipher.NewGCM(block)
    if err != nil {
        return nil, err
    }
    nonce := make([]byte, gcm.NonceSize()) // 12 bytes
    if _, err := rand.Read(nonce); err != nil {
        return nil, err
    }
    // Seal appends the ciphertext+tag to its first arg (here: the nonce).
    return gcm.Seal(nonce, nonce, plaintext, nil), nil
}

func main() {
    key := make([]byte, 32) // AES-256; use a real random key in production
    rand.Read(key)
    ct, _ := encrypt(key, []byte("attack at dawn"))
    fmt.Println("nonce+ciphertext bytes:", len(ct))
    fmt.Println("hex prefix:", hex.EncodeToString(ct[:12]) != "")
}
Output
nonce+ciphertext bytes: 42 hex prefix: true

Decrypting and detecting tampering

Open splits the nonce back off, verifies the authentication tag, and returns an error if the ciphertext (or the associated data) was altered.

package main

import (
    "crypto/aes"
    "crypto/cipher"
    "crypto/rand"
    "fmt"
)

func main() {
    key := make([]byte, 32)
    rand.Read(key)
    block, _ := aes.NewCipher(key)
    gcm, _ := cipher.NewGCM(block)

    nonce := make([]byte, gcm.NonceSize())
    rand.Read(nonce)
    ct := gcm.Seal(nil, nonce, []byte("secret"), nil)

    pt, err := gcm.Open(nil, nonce, ct, nil)
    fmt.Println(string(pt), err) // secret 

    ct[0] ^= 0xFF                // tamper with one byte
    _, err = gcm.Open(nil, nonce, ct, nil)
    fmt.Println("tampered:", err != nil) // authentication fails
}
Output
secret <nil> tampered: true

Common Mistakes

Wrong
// A fixed/zero nonce reused across messages:
nonce := make([]byte, gcm.NonceSize()) // all zeros, never randomized
ct := gcm.Seal(nil, nonce, msg, nil)
// Reusing a (key, nonce) pair under GCM is catastrophic.
Right
nonce := make([]byte, gcm.NonceSize())
if _, err := rand.Read(nonce); err != nil { // crypto/rand
    return err
}
ct := gcm.Seal(nonce, nonce, msg, nil) // unique nonce per Seal

Key Rules

  • AES keys are exactly 16, 24, or 32 bytes (AES-128/192/256); anything else makes aes.NewCipher error.
  • The GCM nonce must be unique per key — generate it from crypto/rand and store it alongside the ciphertext (prepending is idiomatic). Reusing a nonce breaks confidentiality and authenticity.
  • GCM is an AEAD: Open returns an error on any tampering — never use the plaintext when err != nil.
  • The 4th arg to Seal/Open is optional associated data — authenticated but not encrypted (e.g. a header); it must match on both sides.
Practice Exercise

Write encrypt/decrypt helpers that prepend a random 12-byte nonce and round-trip a message. Add associated data (a version byte) and show that changing it on decrypt makes Open fail. Deliberately reuse a fixed nonce for two different plaintexts and read about why that leaks the XOR of the keystreams. Finally, confirm aes.NewCipher rejects a 20-byte key.

Pitfall: Nonce reuse under GCM is catastrophic — a repeated (key, nonce) pair lets an attacker recover the authentication subkey and forge messages. Always draw the nonce from crypto/rand. And never hand-roll AES-CBC/ECB for confidentiality without a separate MAC: ECB leaks patterns and unauthenticated CBC is malleable — reach for GCM.
Connections: Builds on crypto/rand (secure nonces/keys) and pairs with encoding/hex/base64 for transport; the keys it consumes are exactly what encoding/pem and crypto/x509 (next) serialize; complements crypto/subtle's constant-time comparisons for MAC-style checks.

encoding/pem: PEM Blocks for Keys & Certs

Encoding

Why it matters: keys and certificates travel as PEM — the -----BEGIN X----- base64 text you see in .pem, .crt, and id_ed25519 files. PEM is just a base64 wrapper around binary DER bytes, with a type label and optional headers. encoding/pem encodes and decodes that envelope; the DER inside is what crypto/x509 actually parses.

Encoding a PEM block

A pem.Block has a Type label and raw Bytes; pem.Encode base64-wraps the bytes with the BEGIN/END markers.

package main

import (
    "bytes"
    "encoding/pem"
    "fmt"
)

func main() {
    block := &pem.Block{
        Type:  "MESSAGE",
        Bytes: []byte("hello"), // in real use: DER-encoded key/cert bytes
    }
    var buf bytes.Buffer
    pem.Encode(&buf, block)
    fmt.Print(buf.String())
}
Output
-----BEGIN MESSAGE----- aGVsbG8= -----END MESSAGE-----

Decoding — and the second return value

pem.Decode returns the first block and the rest of the input, so a file with several blocks is decoded in a loop.

package main

import (
    "encoding/pem"
    "fmt"
)

func main() {
    data := []byte("-----BEGIN MESSAGE-----\naGVsbG8=\n-----END MESSAGE-----\n")
    block, rest := pem.Decode(data)
    if block == nil {
        panic("no PEM data found")
    }
    fmt.Printf("type=%s bytes=%q rest=%d\n", block.Type, block.Bytes, len(rest))
}
Output
type=MESSAGE bytes="hello" rest=0

Common Mistakes

Wrong
// Feeding the PEM text straight into x509:
cert, err := x509.ParseCertificate(pemBytes) // fails: expects DER
// PEM is base64-wrapped DER, not DER.
Right
block, _ := pem.Decode(pemBytes)
if block == nil || block.Type != "CERTIFICATE" {
    return errors.New("not a certificate PEM")
}
cert, err := x509.ParseCertificate(block.Bytes) // DER

Key Rules

  • Block.Bytes is binary DER; the PEM text is only a base64 envelope with a Type label — hand the Bytes, not the whole file, to x509.
  • pem.Decode returns (*Block, rest); loop on rest to read multiple blocks (a cert chain, or a key + cert bundle).
  • A nil block means "no PEM found" — always check it before dereferencing.
  • Match on block.Type ("CERTIFICATE", "PRIVATE KEY", ...) so you don't feed a key where a cert is expected.
Practice Exercise

Write encodePEM(typ string, der []byte) string and a matching decodePEM that errors on a nil block or wrong Type. Build a file containing two concatenated blocks and decode both by looping on the rest return. Then take a real DER private key from x509.MarshalPKCS8PrivateKey, PEM-encode it, and decode it back — confirming the round-trip is byte-identical.

Pitfall: Passing PEM text where DER is expected (or vice-versa) — x509 functions want block.Bytes, and confusing the two yields opaque "asn1: structure error" failures. And ignoring pem.Decode's second return value means silently dropping every block after the first in a multi-entry file.
Connections: The text wrapper around the DER that crypto/x509 (next) marshals and parses; its base64 core is the same idea as encoding/base64; the keys it carries are consumed by crypto/aes, ed25519, and ecdsa in this guide.

crypto/x509: Parsing Certificates & Keys

Security

Why it matters: crypto/x509 is how Go turns keys and certificates between in-memory objects and the DER bytes that live inside PEM files. It marshals/parses private keys (prefer PKCS#8, which handles any key type), creates and parses X.509 certificates, and verifies chains against a CertPool. It underpins every TLS server and client in the standard library.

Marshaling a key to PKCS#8 DER (then PEM)

MarshalPKCS8PrivateKey serializes any supported key type to DER; wrap that in a PEM block and you have the familiar PRIVATE KEY file. Parsing reverses it.

package main

import (
    "crypto/ed25519"
    "crypto/rand"
    "crypto/x509"
    "encoding/pem"
    "fmt"
)

func main() {
    _, priv, _ := ed25519.GenerateKey(rand.Reader)

    der, err := x509.MarshalPKCS8PrivateKey(priv)
    if err != nil {
        panic(err)
    }
    pemBytes := pem.EncodeToMemory(&pem.Block{Type: "PRIVATE KEY", Bytes: der})

    block, _ := pem.Decode(pemBytes)
    parsed, err := x509.ParsePKCS8PrivateKey(block.Bytes)
    if err != nil {
        panic(err)
    }
    _, ok := parsed.(ed25519.PrivateKey)
    fmt.Println("round-tripped ed25519 key:", ok)
}
Output
round-tripped ed25519 key: true

Creating and parsing a self-signed certificate

CreateCertificate emits DER from a template; ParseCertificate reads it back. A self-signed cert uses the same template as its own parent.

tmpl := &x509.Certificate{
    SerialNumber: big.NewInt(1),
    Subject:      pkix.Name{CommonName: "example.local"},
    NotBefore:    time.Now(),
    NotAfter:     time.Now().Add(24 * time.Hour),
    KeyUsage:     x509.KeyUsageDigitalSignature,
}
pub, priv, _ := ed25519.GenerateKey(rand.Reader)
der, _ := x509.CreateCertificate(rand.Reader, tmpl, tmpl, pub, priv)

cert, _ := x509.ParseCertificate(der)
fmt.Println(cert.Subject.CommonName) // example.local
Output
example.local

Common Mistakes

Wrong
// Guessing the key format with the wrong parser:
key, err := x509.ParsePKCS1PrivateKey(der) // RSA-only!
// Fails for EC/Ed25519 keys, or for PKCS#8-wrapped RSA.
Right
// PKCS#8 handles RSA, ECDSA, and Ed25519 uniformly:
key, err := x509.ParsePKCS8PrivateKey(der)
if err != nil {
    return err
}
// type-switch on key to get the concrete type

Key Rules

  • x509 speaks DER ([]byte); combine it with encoding/pem for the text form.
  • Prefer PKCS#8 (MarshalPKCS8PrivateKey/ParsePKCS8PrivateKey) — it's key-type-agnostic, unlike PKCS#1 (RSA-only) and SEC1 (EC-only).
  • CreateCertificate(rand, template, parent, pub, priv) returns DER; a self-signed cert passes the same template as both template and parent.
  • A template needs a unique SerialNumber, NotBefore/NotAfter, and appropriate KeyUsage — omissions produce certs that fail verification.
Practice Exercise

Generate an Ed25519 key, marshal it to a PKCS#8 PEM file, and load it back with a pem.Decode + ParsePKCS8PrivateKey + type switch. Create a self-signed certificate with a 1-year validity and a DNSNames entry, then parse it and print the subject and expiry. Finally, build a x509.NewCertPool, add your cert, and call cert.Verify to see a chain succeed against your own root.

Pitfall: Mixing the key-format parsers — ParsePKCS1PrivateKey is RSA-only and ParseECPrivateKey is EC-only, so calling the wrong one on a PKCS#8 blob fails cryptically; default to PKCS#8. And a certificate template missing SerialNumber or with a NotAfter in the past parses fine but fails every verification.
Connections: Consumes the DER that encoding/pem wraps and the keys made by ed25519/ecdsa (this guide); it's the layer beneath crypto/tls from an earlier day; verification uses constant-time and hashing primitives from the broader crypto family.

Section 2 — Digital Signatures

crypto/ed25519: Modern Digital Signatures

Security

Why it matters: Ed25519 is the default "just works" signature scheme — small keys (32-byte public, 64-byte signature), fast, and misuse-resistant because signing is deterministic (no per-signature nonce to leak). Use it to sign tokens, releases, or messages when you control both ends. The API is refreshingly small: generate, Sign, Verify.

Generate, sign, verify

Sign takes the private key and message and returns a signature; Verify returns a plain bool — no error, no options for the common case.

package main

import (
    "crypto/ed25519"
    "crypto/rand"
    "fmt"
)

func main() {
    pub, priv, err := ed25519.GenerateKey(rand.Reader)
    if err != nil {
        panic(err)
    }
    msg := []byte("ship v1.4.0")

    sig := ed25519.Sign(priv, msg) // deterministic: no rand needed here
    fmt.Println("signature bytes:", len(sig))
    fmt.Println("valid:", ed25519.Verify(pub, msg, sig))

    fmt.Println("tampered:", ed25519.Verify(pub, []byte("ship v1.4.1"), sig))
}
Output
signature bytes: 64 valid: true tampered: false

Recovering the public key from the private key

An ed25519.PrivateKey embeds its public half; Public() returns it, so you only need to distribute the public key.

pub := priv.Public().(ed25519.PublicKey)
fmt.Println("public key bytes:", len(pub)) // 32
Output
public key bytes: 32

Common Mistakes

Wrong
// Pre-hashing then signing the digest with plain Sign:
h := sha256.Sum256(msg)
sig := ed25519.Sign(priv, h[:]) // now Verify(pub, msg, sig) is false
// Plain Ed25519 signs the FULL message, not a digest.
Right
sig := ed25519.Sign(priv, msg)          // sign the message itself
ok := ed25519.Verify(pub, msg, sig)     // verify against the same message
// (Use ed25519.Options{Hash: crypto.SHA512} + Ed25519ph only if you
//  specifically need the pre-hash variant on both sides.)

Key Rules

  • Keys are just byte slices: PublicKey is 32 bytes, PrivateKey 64, signatures 64 — easy to store and transmit.
  • Signing is deterministic — the same key + message always yields the same signature, and there's no nonce to reuse or leak (a key advantage over ECDSA).
  • Verify returns a bool, not an error; sign and verify the full message unless you deliberately use the Ed25519ph pre-hash variant on both sides.
  • PrivateKey.Public() gives the matching public key — distribute only the public half.
Practice Exercise

Build a sign/verify pair for release manifests, then show that flipping one byte of the message or the signature makes Verify return false. Marshal the private key to PKCS#8 PEM (tying back to x509) and reload it. Confirm signing is deterministic by signing the same message twice and comparing the bytes, and contrast that with ECDSA's randomized signatures below.

Pitfall: Assuming Ed25519 signs a hash the way ECDSA/RSA workflows do — plain ed25519.Sign takes the whole message, so pre-hashing on one side and not the other makes verification silently fail. And ed25519.Sign panics (not errors) if you pass a wrong-length key, so validate key material at your trust boundary.
Connections: Keys serialize through crypto/x509 + encoding/pem (this guide); it can sign the same certificates x509.CreateCertificate builds; contrasts directly with crypto/ecdsa (next) on determinism and nonce safety; uses crypto/rand only at key-generation time.

crypto/ecdsa: Elliptic-Curve Signatures

Security

Why it matters: ECDSA over NIST curves (P-256) is what you meet in TLS certificates, JWTs (ES256), and many existing systems — so you need it for interop even when you'd pick Ed25519 for greenfield work. Modern Go gives you SignASN1/VerifyASN1, which produce and consume the standard DER signature encoding. The critical difference from Ed25519: every signature needs fresh, high-quality randomness.

Generate, sign a digest, verify

ECDSA signs a hash of the message. Generate a key on a curve, hash with SHA-256, then SignASN1/VerifyASN1.

package main

import (
    "crypto/ecdsa"
    "crypto/elliptic"
    "crypto/rand"
    "crypto/sha256"
    "fmt"
)

func main() {
    priv, err := ecdsa.GenerateKey(elliptic.P256(), rand.Reader)
    if err != nil {
        panic(err)
    }
    digest := sha256.Sum256([]byte("transfer approved"))

    // rand.Reader supplies the per-signature nonce -- required.
    sig, err := ecdsa.SignASN1(rand.Reader, priv, digest[:])
    if err != nil {
        panic(err)
    }
    fmt.Println("valid:", ecdsa.VerifyASN1(&priv.PublicKey, digest[:], sig))

    bad := sha256.Sum256([]byte("transfer denied"))
    fmt.Println("wrong msg:", ecdsa.VerifyASN1(&priv.PublicKey, bad[:], sig))
}
Output
valid: true wrong msg: false

Why the nonce source matters

ECDSA's security depends on a unique, unpredictable nonce per signature. A repeated or biased nonce lets an attacker recover the private key from two signatures — always pass crypto/rand.Reader.

// Correct: fresh randomness for every signature.
sig, err := ecdsa.SignASN1(rand.Reader, priv, digest[:])

// NEVER do this: a fixed/predictable reader reuses the nonce k and
// leaks the private key (the classic PlayStation 3 / Sony break).

Common Mistakes

Wrong
// Signing the raw message instead of its hash:
sig, _ := ecdsa.SignASN1(rand.Reader, priv, msg) // msg, not a digest
// ECDSA expects a fixed-size hash; oversized input is silently truncated.
Right
digest := sha256.Sum256(msg)
sig, _ := ecdsa.SignASN1(rand.Reader, priv, digest[:])
ok := ecdsa.VerifyASN1(&priv.PublicKey, digest[:], sig)

Key Rules

  • Prefer SignASN1/VerifyASN1 (DER []byte signatures) over the legacy Sign that returns raw (r, s) big ints.
  • ECDSA signs a hash, not the raw message — pass a digest sized to the curve (SHA-256 for P-256).
  • Every signature needs fresh randomness from crypto/rand.Reader; a reused or predictable nonce leaks the private key.
  • Pick the curve with elliptic.P256()/P384()/P521(); VerifyASN1 returns a bool.
Practice Exercise

Sign a document digest with P-256 and verify it, then tamper with the message and the signature independently to see both fail. Marshal the key with x509.MarshalPKCS8PrivateKey to PEM and reload it. Sign the same digest twice and observe that the signatures differ (randomized), unlike Ed25519 — then explain in one sentence why that randomness must never repeat.

Pitfall: ECDSA is catastrophically broken by a repeated or biased per-signature nonce — two signatures with the same k reveal the private key (the real-world Sony PS3 hack). Never supply a custom or deterministic reader unless it's RFC 6979 deterministic ECDSA; use crypto/rand.Reader. And signing the raw message instead of a digest gives interop failures and silent truncation.
Connections: The direct counterpoint to crypto/ed25519 (nonce-based vs deterministic); hashes its input with the SHA-256 primitives from the crypto family; keys and certs flow through crypto/x509 + encoding/pem; and its randomness comes from crypto/rand, reviewed earlier.
