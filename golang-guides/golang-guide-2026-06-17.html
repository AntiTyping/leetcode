Builder Pattern

Design Patterns • 7 days overdue • rep 0

Quick Reminder

Fluent API with method chaining via pointer receivers. Each With* method sets a field and returns *Builder. The final Build() call constructs the target struct. Pointer receivers are essential — value receivers would chain on copies, losing earlier mutations.

Key Pattern

type ServerBuilder struct {
    name string
    port int
}

func (b *ServerBuilder) WithName(n string) *ServerBuilder {
    b.name = n
    return b
}

func (b *ServerBuilder) WithPort(p int) *ServerBuilder {
    b.port = p
    return b
}

func (b *ServerBuilder) Build() Server {
    return Server{Name: b.name, Port: b.port}
}

// Usage:
// s := (&ServerBuilder{}).WithName("api").WithPort(8080).Build()
Challenge Exercise

Write an HTTPClientBuilder with WithTimeout, WithRetries, and WithBaseURL methods. Add a Build() (*HTTPClient, error) method that validates: timeout must be positive, retries must be 0-10, and base URL must not be empty. Return an error if validation fails.

Pitfall: Using a value receiver on With* methods — the chain silently compiles but each call modifies a copy, so Build() returns a zero-value struct.

encoding/json

Standard Library • 7 days overdue • rep 1

Quick Reminder

Struct tags control field names (`json:"name"`), omitting empty values (omitempty), and skipping fields (json:"-"). json.Marshal/json.Unmarshal for byte slices; json.NewEncoder/json.NewDecoder for streams. json.RawMessage delays parsing. Custom marshalers via MarshalJSON()/UnmarshalJSON().

Key Pattern

// Decoding from an HTTP request body:
var req CreateUserRequest
if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
    http.Error(w, err.Error(), http.StatusBadRequest)
    return
}

// Struct tags:
type User struct {
    ID       int    `json:"id"`
    Name     string `json:"name"`
    Email    string `json:"email,omitempty"`
    Password string `json:"-"` // never serialized
}
Challenge Exercise

Create a Timestamp type wrapping time.Time. Implement MarshalJSON to output Unix seconds (integer) and UnmarshalJSON to parse it back. A struct with CreatedAt Timestamp should marshal to {"created_at": 1719619200}.

Pitfall: Unexported fields (lowercase) are silently skipped by encoding/json. You unmarshal into a struct, everything looks fine, but all values are zero. Always export fields you need to serialize.

Pointers

Go Foundations • 7 days overdue • rep 2

Quick Reminder

& takes the address, * dereferences. Pointer receivers (func (p *T) M()) modify the original; value receivers copy. Passing a pointer to a function allows mutation. Go has no pointer arithmetic.

Key Pattern

type Node struct {
    Val  int
    Next *Node
}

func (n *Node) Append(val int) {
    for n.Next != nil {
        n = n.Next
    }
    n.Next = &Node{Val: val}
}

// nil check is critical:
func safeDeref(p *int) int {
    if p == nil {
        return 0
    }
    return *p
}
Challenge Exercise

Implement a singly linked list with Insert(val int), Delete(val int) bool, and String() string. Use pointer-to-pointer (**Node) in Delete to avoid special-casing head removal.

Pitfall: Dereferencing a nil pointer causes a runtime panic. Always guard with a nil check before accessing p.Field or *p when p could be nil.

fmt Package

Standard Library • 7 days overdue • rep 1

Quick Reminder

%v default format, %+v with field names, %#v Go-syntax representation, %T type name. Implement fmt.Stringer (String() string) for custom print output. fmt.Fprintf writes to any io.Writer. fmt.Errorf with %w wraps errors.

Key Pattern

type Point struct{ X, Y float64 }

func (p Point) String() string {
    return fmt.Sprintf("(%0.1f, %0.1f)", p.X, p.Y)
}

func (p Point) GoString() string {
    return fmt.Sprintf("Point{X: %0.1f, Y: %0.1f}", p.X, p.Y)
}

// %v  -> (3.0, 4.0)     (uses String())
// %#v -> Point{X: 3.0, Y: 4.0} (uses GoString())
// %T  -> main.Point
Challenge Exercise

Create a Matrix type (2D slice of float64). Implement both Stringer and GoStringer. The String() should produce a formatted grid, and GoString() should produce valid Go literal syntax.

Pitfall: Calling fmt.Sprintf("%s", p) inside String() where p is the receiver causes infinite recursion. Use fmt.Sprintf with the underlying fields instead.

io Package

Standard Library • 7 days overdue • rep 1

Quick Reminder

io.Reader and io.Writer are the fundamental I/O interfaces. io.Copy(dst, src) streams data. io.TeeReader reads and writes simultaneously. io.LimitReader caps bytes. Composition via io.MultiReader, io.MultiWriter.

Key Pattern

// Copy from reader to writer (streaming, no full buffer):
n, err := io.Copy(dst, src)

// TeeReader: read from src, and everything read is also written to buf
var buf bytes.Buffer
tee := io.TeeReader(resp.Body, &buf)
io.ReadAll(tee) // reads body AND fills buf

// LimitReader: read at most N bytes
limited := io.LimitReader(r, 1024*1024) // 1 MB max
Challenge Exercise

Implement a CountingReader struct that wraps an io.Reader and tracks the total number of bytes read. It should satisfy io.Reader and have a BytesRead() int64 method.

Pitfall: io.EOF is not an error — it signals the end of a stream. Do not wrap it with fmt.Errorf or treat it as a failure. Check if err == io.EOF or errors.Is(err, io.EOF) separately.

Production Patterns

Rate Limiting & Throttling

concept_id: rate_limiting Production

Rate limiting protects your services from overload and helps you respect upstream API limits. Go's concurrency primitives — tickers, channels, and the golang.org/x/time/rate package — make it straightforward to implement multiple rate limiting strategies.

Why It Matters

  • Prevents your service from being overwhelmed by bursts of requests
  • Respects third-party API rate limits (avoid getting banned)
  • Ensures fair resource distribution across clients
  • Required in virtually every production Go service

Example 1 — Simple rate limiter with time.Ticker

The simplest approach: use a ticker to allow one operation per interval. This creates a fixed-rate limiter with no burst capacity.

package main

import (
    "fmt"
    "time"
)

func main() {
    // Allow 1 request every 200ms = 5 requests/second
    limiter := time.NewTicker(200 * time.Millisecond)
    defer limiter.Stop() // CRITICAL: always stop tickers

    requests := []string{"req-1", "req-2", "req-3", "req-4", "req-5"}

    start := time.Now()
    for _, req := range requests {
        <-limiter.C // block until the next tick
        fmt.Printf("[%6dms] Processing %s\n",
            time.Since(start).Milliseconds(), req)
    }
    fmt.Printf("\nTotal time: %dms (expected ~800ms for 5 reqs at 200ms)\n",
        time.Since(start).Milliseconds())
}
Output
[ 200ms] Processing req-1 [ 400ms] Processing req-2 [ 600ms] Processing req-3 [ 800ms] Processing req-4 [ 1000ms] Processing req-5 Total time: 1000ms (expected ~800ms for 5 reqs at 200ms)
Pitfall: Forgetting defer limiter.Stop() leaks the goroutine backing the ticker. In long-running servers, this causes a slow memory leak that is hard to debug.

Example 2 — Token bucket with buffered channel

A buffered channel simulates a token bucket. The channel capacity is the burst size. A background goroutine refills tokens at a steady rate.

package main

import (
    "fmt"
    "time"
)

// TokenBucket allows burst requests up to capacity, then rate-limits.
type TokenBucket struct {
    tokens chan struct{}
    quit   chan struct{}
}

// NewTokenBucket creates a limiter: rate tokens/sec, burst capacity.
func NewTokenBucket(rate int, burst int) *TokenBucket {
    tb := &TokenBucket{
        tokens: make(chan struct{}, burst),
        quit:   make(chan struct{}),
    }

    // Pre-fill the bucket to capacity
    for i := 0; i < burst; i++ {
        tb.tokens <- struct{}{}
    }

    // Refill goroutine
    go func() {
        ticker := time.NewTicker(time.Second / time.Duration(rate))
        defer ticker.Stop()
        for {
            select {
            case <-ticker.C:
                select {
                case tb.tokens <- struct{}{}:
                    // Token added
                default:
                    // Bucket full, discard token
                }
            case <-tb.quit:
                return
            }
        }
    }()

    return tb
}

// Wait blocks until a token is available.
func (tb *TokenBucket) Wait() {
    <-tb.tokens
}

// Stop shuts down the refill goroutine.
func (tb *TokenBucket) Stop() {
    close(tb.quit)
}

func main() {
    // 5 tokens/sec, burst of 3
    bucket := NewTokenBucket(5, 3)
    defer bucket.Stop()

    start := time.Now()

    // First 3 should be instant (burst), then rate-limited
    for i := 1; i <= 7; i++ {
        bucket.Wait()
        fmt.Printf("[%6dms] Request %d processed\n",
            time.Since(start).Milliseconds(), i)
    }
}
Output
[ 0ms] Request 1 processed [ 0ms] Request 2 processed [ 0ms] Request 3 processed [ 200ms] Request 4 processed [ 400ms] Request 5 processed [ 600ms] Request 6 processed [ 800ms] Request 7 processed

Notice requests 1-3 are instant (burst), then 4-7 are spaced at 200ms (5/sec).

Common Mistake — Goroutine leak from abandoned ticker

WRONG — Goroutine leak
func rateLimited() chan struct{} {
    ch := make(chan struct{}, 1)
    go func() {
        // This ticker is never stopped!
        ticker := time.NewTicker(100 * time.Millisecond)
        for range ticker.C {
            select {
            case ch <- struct{}{}:
            default:
            }
        }
        // UNREACHABLE: range ticker.C never ends
    }()
    return ch
}
// If caller abandons the channel, the
// goroutine runs forever.
RIGHT — Quit channel for cleanup
func rateLimited() (chan struct{}, func()) {
    ch := make(chan struct{}, 1)
    quit := make(chan struct{})
    go func() {
        ticker := time.NewTicker(100 * time.Millisecond)
        defer ticker.Stop()
        for {
            select {
            case <-ticker.C:
                select {
                case ch <- struct{}{}:
                default:
                }
            case <-quit:
                return // clean exit
            }
        }
    }()
    stop := func() { close(quit) }
    return ch, stop
}

Example 3 — golang.org/x/time/rate Limiter with burst

The standard library extension rate.Limiter is the production choice. It implements a token bucket with configurable rate and burst size, plus context-aware waiting.

package main

import (
    "context"
    "fmt"
    "time"

    "golang.org/x/time/rate"
)

func main() {
    // 10 events per second, burst of 3
    limiter := rate.NewLimiter(rate.Limit(10), 3)

    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()

    start := time.Now()
    for i := 1; i <= 15; i++ {
        // Wait blocks until a token is available or context expires
        if err := limiter.Wait(ctx); err != nil {
            fmt.Printf("[%6dms] Request %d: %v\n",
                time.Since(start).Milliseconds(), i, err)
            break
        }
        fmt.Printf("[%6dms] Request %d: allowed\n",
            time.Since(start).Milliseconds(), i)
    }

    // Non-blocking check:
    fmt.Println("\nAllow() (non-blocking):", limiter.Allow())

    // Reserve for future use:
    r := limiter.Reserve()
    fmt.Println("Reservation delay:", r.Delay())
}
Output

Example 4 — Per-client rate limiting with sync.Map

In HTTP servers, you often need per-client (per-IP) rate limits. Use sync.Map to store a limiter per client key.

package main

import (
    "fmt"
    "sync"
    "time"

    "golang.org/x/time/rate"
)

// PerClientLimiter manages individual rate limiters per key.
type PerClientLimiter struct {
    limiters sync.Map
    rate     rate.Limit
    burst    int
}

func NewPerClientLimiter(r rate.Limit, burst int) *PerClientLimiter {
    return &PerClientLimiter{rate: r, burst: burst}
}

// GetLimiter returns (or creates) a limiter for the given key.
func (pcl *PerClientLimiter) GetLimiter(key string) *rate.Limiter {
    if v, ok := pcl.limiters.Load(key); ok {
        return v.(*rate.Limiter)
    }
    limiter := rate.NewLimiter(pcl.rate, pcl.burst)
    actual, _ := pcl.limiters.LoadOrStore(key, limiter)
    return actual.(*rate.Limiter)
}

func main() {
    pcl := NewPerClientLimiter(2, 3) // 2/sec, burst 3

    clients := []string{"client-A", "client-B", "client-A", "client-A",
        "client-B", "client-A", "client-A"}

    start := time.Now()
    for i, client := range clients {
        limiter := pcl.GetLimiter(client)
        allowed := limiter.Allow()
        fmt.Printf("[%6dms] %s request %d: allowed=%v\n",
            time.Since(start).Milliseconds(), client, i+1, allowed)
    }

    // client-A made 5 requests: first 3 burst allowed, then denied
    // client-B made 2 requests: both within burst, both allowed
    fmt.Println("\nEach client has an independent rate limit.")
}
Output
[ 0ms] client-A request 1: allowed=true [ 0ms] client-B request 2: allowed=true [ 0ms] client-A request 3: allowed=true [ 0ms] client-A request 4: allowed=true [ 0ms] client-B request 5: allowed=true [ 0ms] client-A request 6: allowed=false [ 0ms] client-A request 7: allowed=false Each client has an independent rate limit.

Common Mistake — Blocking vs. dropping

WRONG — Blocking all requests
// In an HTTP handler: blocking ties up
// the goroutine serving the request.
func handler(w http.ResponseWriter, r *http.Request) {
    limiter.Wait(r.Context()) // blocks!
    // If 1000 clients hit this, you have
    // 1000 goroutines sleeping.
    processRequest(w, r)
}
RIGHT — Drop with 429 status
func handler(w http.ResponseWriter, r *http.Request) {
    if !limiter.Allow() {
        http.Error(w,
            "Too Many Requests",
            http.StatusTooManyRequests)
        return // fast reject
    }
    processRequest(w, r)
}
Connections: Rate limiting builds on channels (token bucket pattern), context (cancellation-aware waiting), and sync.Map (concurrent per-client state). The rate.Limiter.Wait(ctx) method integrates with Go's context system for clean cancellation.

Circuit Breaker Pattern

concept_id: circuit_breaker Production

A circuit breaker prevents your service from repeatedly calling a failing dependency. Like an electrical circuit breaker, it "trips" after too many failures, returning errors immediately without making the call. After a timeout, it allows a test request through to see if the dependency has recovered.

Why It Matters

  • Prevents cascading failures in microservice architectures
  • Stops wasting resources on calls that will likely fail
  • Gives failing services time to recover
  • Provides fast failure feedback to callers

The Three States

  • Closed (normal): requests pass through. Failures are counted. If failures exceed a threshold, transition to Open.
  • Open (tripped): requests are rejected immediately with an error. After a timeout, transition to HalfOpen.
  • HalfOpen (testing): one test request is allowed. If it succeeds, transition to Closed. If it fails, transition back to Open.

Example 1 — Simple state machine implementation

package main

import (
    "errors"
    "fmt"
    "sync"
    "time"
)

// State represents the circuit breaker state.
type State int

const (
    StateClosed   State = iota // Normal operation
    StateOpen                  // Failing, reject all
    StateHalfOpen              // Testing recovery
)

func (s State) String() string {
    switch s {
    case StateClosed:
        return "CLOSED"
    case StateOpen:
        return "OPEN"
    case StateHalfOpen:
        return "HALF-OPEN"
    default:
        return "UNKNOWN"
    }
}

var ErrCircuitOpen = errors.New("circuit breaker is open")

// CircuitBreaker implements the circuit breaker pattern.
type CircuitBreaker struct {
    mu               sync.Mutex
    state            State
    failures         int
    maxFailures      int
    timeout          time.Duration
    lastFailureTime  time.Time
}

// NewCircuitBreaker creates a breaker that opens after maxFailures
// consecutive failures and stays open for timeout duration.
func NewCircuitBreaker(maxFailures int, timeout time.Duration) *CircuitBreaker {
    return &CircuitBreaker{
        state:       StateClosed,
        maxFailures: maxFailures,
        timeout:     timeout,
    }
}

// Execute runs the given function through the circuit breaker.
func (cb *CircuitBreaker) Execute(fn func() error) error {
    cb.mu.Lock()

    switch cb.state {
    case StateOpen:
        // Check if timeout has elapsed
        if time.Since(cb.lastFailureTime) > cb.timeout {
            cb.state = StateHalfOpen
            cb.mu.Unlock()
            fmt.Printf("  [breaker] State: OPEN -> HALF-OPEN (timeout elapsed)\n")
            return cb.doCall(fn)
        }
        cb.mu.Unlock()
        return ErrCircuitOpen

    case StateHalfOpen:
        cb.mu.Unlock()
        return cb.doCall(fn)

    default: // Closed
        cb.mu.Unlock()
        return cb.doCall(fn)
    }
}

func (cb *CircuitBreaker) doCall(fn func() error) error {
    err := fn()

    cb.mu.Lock()
    defer cb.mu.Unlock()

    if err != nil {
        cb.failures++
        cb.lastFailureTime = time.Now()

        if cb.state == StateHalfOpen {
            cb.state = StateOpen
            fmt.Printf("  [breaker] State: HALF-OPEN -> OPEN (test call failed)\n")
        } else if cb.failures >= cb.maxFailures {
            cb.state = StateOpen
            fmt.Printf("  [breaker] State: CLOSED -> OPEN (%d failures)\n", cb.failures)
        }
        return err
    }

    // Success: reset
    cb.failures = 0
    if cb.state == StateHalfOpen {
        cb.state = StateClosed
        fmt.Printf("  [breaker] State: HALF-OPEN -> CLOSED (test call succeeded)\n")
    }
    return nil
}

// State returns the current state.
func (cb *CircuitBreaker) CurrentState() State {
    cb.mu.Lock()
    defer cb.mu.Unlock()
    return cb.state
}

func main() {
    cb := NewCircuitBreaker(3, 500*time.Millisecond)

    callCount := 0
    // Simulated service: fails for the first 5 calls, then succeeds
    unreliableService := func() error {
        callCount++
        if callCount <= 5 {
            return fmt.Errorf("service unavailable (call %d)", callCount)
        }
        return nil
    }

    for i := 1; i <= 10; i++ {
        err := cb.Execute(unreliableService)
        fmt.Printf("Call %d: state=%s, err=%v\n\n", i, cb.CurrentState(), err)

        if i == 5 {
            fmt.Println("--- Waiting for timeout to elapse ---\n")
            time.Sleep(600 * time.Millisecond)
        }
        time.Sleep(50 * time.Millisecond)
    }
}
Output
Call 1: state=CLOSED, err=service unavailable (call 1) Call 2: state=CLOSED, err=service unavailable (call 2) [breaker] State: CLOSED -> OPEN (3 failures) Call 3: state=OPEN, err=service unavailable (call 3) Call 4: state=OPEN, err=circuit breaker is open Call 5: state=OPEN, err=circuit breaker is open --- Waiting for timeout to elapse --- [breaker] State: OPEN -> HALF-OPEN (timeout elapsed) [breaker] State: HALF-OPEN -> OPEN (test call failed) Call 6: state=OPEN, err=service unavailable (call 4) Call 7: state=OPEN, err=circuit breaker is open Call 8: state=OPEN, err=circuit breaker is open Call 9: state=OPEN, err=circuit breaker is open Call 10: state=OPEN, err=circuit breaker is open

Example 2 — Thread-safe breaker with timer-based reset

A more production-like version uses time.AfterFunc to automatically transition from Open to HalfOpen, avoiding the need to check elapsed time on every call.

package main

import (
    "errors"
    "fmt"
    "sync"
    "time"
)

type BreakerState int

const (
    Closed   BreakerState = iota
    Open
    HalfOpen
)

var ErrBreakerOpen = errors.New("circuit breaker is open")

type Breaker struct {
    mu           sync.Mutex
    state        BreakerState
    failures     int
    maxFailures  int
    resetTimeout time.Duration
    resetTimer   *time.Timer
}

func NewBreaker(maxFail int, resetTimeout time.Duration) *Breaker {
    return &Breaker{
        state:        Closed,
        maxFailures:  maxFail,
        resetTimeout: resetTimeout,
    }
}

func (b *Breaker) Call(fn func() error) error {
    b.mu.Lock()
    if b.state == Open {
        b.mu.Unlock()
        return ErrBreakerOpen
    }
    isHalfOpen := b.state == HalfOpen
    b.mu.Unlock()

    err := fn()

    b.mu.Lock()
    defer b.mu.Unlock()

    if err != nil {
        b.failures++
        if isHalfOpen || b.failures >= b.maxFailures {
            b.trip()
        }
        return err
    }

    // Success
    b.failures = 0
    b.state = Closed
    return nil
}

func (b *Breaker) trip() {
    b.state = Open
    // Automatically transition to HalfOpen after timeout
    if b.resetTimer != nil {
        b.resetTimer.Stop()
    }
    b.resetTimer = time.AfterFunc(b.resetTimeout, func() {
        b.mu.Lock()
        defer b.mu.Unlock()
        if b.state == Open {
            b.state = HalfOpen
        }
    })
}

func main() {
    b := NewBreaker(2, 300*time.Millisecond)

    // Simulate failures
    fail := func() error { return fmt.Errorf("timeout") }
    succeed := func() error { return nil }

    fmt.Println("--- Causing failures ---")
    fmt.Println("Call 1:", b.Call(fail))
    fmt.Println("Call 2:", b.Call(fail))      // trips breaker
    fmt.Println("Call 3:", b.Call(succeed))    // rejected (open)

    fmt.Println("\n--- Waiting for auto-reset ---")
    time.Sleep(400 * time.Millisecond) // wait for HalfOpen

    fmt.Println("Call 4:", b.Call(succeed))    // succeeds, closes breaker
    fmt.Println("Call 5:", b.Call(succeed))    // normal operation
}
Output
--- Causing failures --- Call 1: timeout Call 2: timeout Call 3: circuit breaker is open --- Waiting for auto-reset --- Call 4: <nil> Call 5: <nil>

Example 3 — Using the gobreaker library

In production, use Sony's gobreaker library instead of rolling your own. It handles edge cases like concurrent half-open requests.

package main

import (
    "fmt"
    "time"

    "github.com/sony/gobreaker/v2"
)

func main() {
    cb := gobreaker.NewCircuitBreaker[string](gobreaker.Settings{
        Name:        "my-service",
        MaxRequests: 1,                   // max requests in half-open
        Interval:    10 * time.Second,    // cyclic period for clearing counts (closed)
        Timeout:     5 * time.Second,     // open -> half-open timeout
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            return counts.ConsecutiveFailures >= 3
        },
        OnStateChange: func(name string, from, to gobreaker.State) {
            fmt.Printf("[%s] %s -> %s\n", name, from, to)
        },
    })

    // Simulated call
    result, err := cb.Execute(func() (string, error) {
        // Your actual HTTP/gRPC call here
        return "response-data", nil
    })

    if err != nil {
        fmt.Println("Error:", err)
    } else {
        fmt.Println("Result:", result)
    }

    fmt.Println("State:", cb.State())
    fmt.Println("Counts:", cb.Counts())
}
Output
Result: response-data State: closed Counts: {1 1 0 0 1 0}

Common Mistakes

WRONG — No half-open state
// Open forever once tripped!
func (b *Breaker) Execute(fn func() error) error {
    if b.isOpen {
        return ErrOpen // never recovers
    }
    err := fn()
    if err != nil {
        b.failures++
        if b.failures >= b.max {
            b.isOpen = true
        }
    }
    return err
}
RIGHT — HalfOpen allows recovery
func (b *Breaker) Execute(fn func() error) error {
    switch b.state {
    case Open:
        if time.Since(b.lastFail) > b.timeout {
            b.state = HalfOpen
            // Fall through to try call
        } else {
            return ErrOpen
        }
    }
    err := fn()
    if err == nil && b.state == HalfOpen {
        b.state = Closed // recovered!
        b.failures = 0
    }
    return err
}
WRONG — Not resetting on success
if err != nil {
    b.failures++
}
// Missing: b.failures = 0 on success
// After 100 successful calls and 3 failures
// spread over hours, breaker still trips!
RIGHT — Reset counter on success
if err != nil {
    b.failures++
} else {
    b.failures = 0 // Reset on success!
}
// Only consecutive failures count.
Connections: Circuit breakers work with error handling (sentinel errors like ErrCircuitOpen), context (propagating cancellation when the breaker is open), and retry with backoff (the next concept). In production, you often wrap calls in a circuit breaker that wraps a retry loop.

Retry with Backoff

concept_id: retry_backoff Production

Transient failures — network hiccups, temporary 503s, database lock contention — are a fact of life in distributed systems. Retrying with exponential backoff and jitter is the standard approach to handle them gracefully without overwhelming the failing service.

Why It Matters

  • Network calls fail transiently; a simple retry often succeeds
  • Without backoff, retries create a thundering herd that makes the problem worse
  • Without jitter, synchronized retries from many clients all hit at the same instant
  • Without a max, broken calls retry forever, wasting resources

Example 1 — Simple retry with fixed delay

The simplest pattern: retry up to N times with a fixed sleep between attempts.

package main

import (
    "errors"
    "fmt"
    "time"
)

// Retry calls fn up to maxAttempts times with a fixed delay between attempts.
func Retry(maxAttempts int, delay time.Duration, fn func() error) error {
    var lastErr error
    for attempt := 1; attempt <= maxAttempts; attempt++ {
        lastErr = fn()
        if lastErr == nil {
            return nil // success
        }
        fmt.Printf("  Attempt %d failed: %v\n", attempt, lastErr)
        if attempt < maxAttempts {
            time.Sleep(delay)
        }
    }
    return fmt.Errorf("all %d attempts failed, last error: %w",
        maxAttempts, lastErr)
}

func main() {
    callCount := 0
    unreliable := func() error {
        callCount++
        if callCount < 3 {
            return errors.New("connection refused")
        }
        return nil // succeeds on 3rd try
    }

    start := time.Now()
    err := Retry(5, 200*time.Millisecond, unreliable)
    elapsed := time.Since(start)

    if err != nil {
        fmt.Println("Final error:", err)
    } else {
        fmt.Printf("Success after %dms (%d calls)\n",
            elapsed.Milliseconds(), callCount)
    }
}
Output
Attempt 1 failed: connection refused Attempt 2 failed: connection refused Success after 400ms (3 calls)

Example 2 — Exponential backoff with jitter

Production retries use exponential backoff (doubling delay) with random jitter (prevents thundering herd). This is the most important retry pattern to know.

package main

import (
    "errors"
    "fmt"
    "math"
    "math/rand"
    "time"
)

// BackoffConfig configures exponential backoff.
type BackoffConfig struct {
    InitialDelay time.Duration
    MaxDelay     time.Duration
    MaxAttempts  int
    Multiplier   float64 // typically 2.0
}

// RetryWithBackoff retries with exponential backoff and full jitter.
func RetryWithBackoff(cfg BackoffConfig, fn func() error) error {
    var lastErr error
    delay := cfg.InitialDelay

    for attempt := 1; attempt <= cfg.MaxAttempts; attempt++ {
        lastErr = fn()
        if lastErr == nil {
            return nil
        }

        if attempt == cfg.MaxAttempts {
            break // don't sleep after the last attempt
        }

        // Full jitter: sleep a random duration between 0 and delay
        jittered := time.Duration(rand.Int63n(int64(delay)))
        fmt.Printf("  Attempt %d failed: %v (sleeping %dms)\n",
            attempt, lastErr, jittered.Milliseconds())
        time.Sleep(jittered)

        // Exponential increase, capped at MaxDelay
        delay = time.Duration(math.Min(
            float64(delay)*cfg.Multiplier,
            float64(cfg.MaxDelay),
        ))
    }

    return fmt.Errorf("all %d attempts failed: %w", cfg.MaxAttempts, lastErr)
}

func main() {
    callCount := 0
    flaky := func() error {
        callCount++
        if callCount <= 4 {
            return errors.New("service unavailable")
        }
        return nil
    }

    cfg := BackoffConfig{
        InitialDelay: 100 * time.Millisecond,
        MaxDelay:     2 * time.Second,
        MaxAttempts:  6,
        Multiplier:   2.0,
    }

    start := time.Now()
    err := RetryWithBackoff(cfg, flaky)
    fmt.Printf("\nResult: err=%v, elapsed=%dms, calls=%d\n",
        err, time.Since(start).Milliseconds(), callCount)
}
Output
Attempt 1 failed: service unavailable (sleeping 47ms) Attempt 2 failed: service unavailable (sleeping 124ms) Attempt 3 failed: service unavailable (sleeping 281ms) Attempt 4 failed: service unavailable (sleeping 614ms) Result: err=<nil>, elapsed=1066ms, calls=5

Each retry sleeps a random amount up to the current delay, which doubles each time (100ms, 200ms, 400ms, 800ms). The jitter makes the actual sleep times vary.

Common Mistake — No jitter (thundering herd)

WRONG — Fixed exponential (no jitter)
delay := 100 * time.Millisecond
for attempt := 0; attempt < max; attempt++ {
    if err := fn(); err == nil {
        return nil
    }
    time.Sleep(delay) // deterministic!
    delay *= 2
}
// 1000 clients all retry at exactly
// 100ms, 200ms, 400ms, 800ms...
// creating synchronized thundering herd
RIGHT — Full jitter
delay := 100 * time.Millisecond
for attempt := 0; attempt < max; attempt++ {
    if err := fn(); err == nil {
        return nil
    }
    // Random sleep in [0, delay)
    jitter := time.Duration(
        rand.Int63n(int64(delay)))
    time.Sleep(jitter)
    delay = min(delay*2, maxDelay)
}
// 1000 clients spread retries across
// the full window, reducing load spikes

Example 3 — Context-aware retry with cancellation

Production retries must respect context cancellation. If the parent context is cancelled (e.g., HTTP request timeout), retries should stop immediately.

package main

import (
    "context"
    "errors"
    "fmt"
    "math"
    "math/rand"
    "time"
)

// RetryOption configures retry behavior.
type RetryOption struct {
    MaxAttempts  int
    InitialDelay time.Duration
    MaxDelay     time.Duration
    Multiplier   float64
    // IsRetryable determines if an error should be retried.
    // If nil, all errors are retried.
    IsRetryable  func(error) bool
}

// DoWithRetry executes fn with exponential backoff, respecting ctx.
func DoWithRetry(ctx context.Context, opt RetryOption, fn func(ctx context.Context) error) error {
    delay := opt.InitialDelay
    var lastErr error

    for attempt := 1; attempt <= opt.MaxAttempts; attempt++ {
        // Check context before each attempt
        select {
        case <-ctx.Done():
            return fmt.Errorf("retry cancelled: %w", ctx.Err())
        default:
        }

        lastErr = fn(ctx)
        if lastErr == nil {
            return nil
        }

        // Check if the error is retryable
        if opt.IsRetryable != nil && !opt.IsRetryable(lastErr) {
            return fmt.Errorf("non-retryable error: %w", lastErr)
        }

        if attempt == opt.MaxAttempts {
            break
        }

        // Jittered backoff with context-aware sleep
        jittered := time.Duration(rand.Int63n(int64(delay)))
        fmt.Printf("  Attempt %d: %v (backoff %dms)\n",
            attempt, lastErr, jittered.Milliseconds())

        select {
        case <-time.After(jittered):
            // Continue to next attempt
        case <-ctx.Done():
            return fmt.Errorf("retry cancelled during backoff: %w", ctx.Err())
        }

        delay = time.Duration(math.Min(
            float64(delay)*opt.Multiplier,
            float64(opt.MaxDelay),
        ))
    }

    return fmt.Errorf("exhausted %d attempts: %w", opt.MaxAttempts, lastErr)
}

// Example: retryable vs non-retryable errors
var ErrNotFound = errors.New("not found")       // NOT retryable
var ErrTimeout = errors.New("request timeout")   // retryable

func main() {
    callCount := 0
    service := func(ctx context.Context) error {
        callCount++
        if callCount <= 2 {
            return ErrTimeout
        }
        return nil
    }

    opt := RetryOption{
        MaxAttempts:  5,
        InitialDelay: 50 * time.Millisecond,
        MaxDelay:     1 * time.Second,
        Multiplier:   2.0,
        IsRetryable: func(err error) bool {
            // Only retry timeouts, not 404s
            return errors.Is(err, ErrTimeout)
        },
    }

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    err := DoWithRetry(ctx, opt, service)
    fmt.Printf("Result: %v (calls: %d)\n", err, callCount)

    // Demo: non-retryable error stops immediately
    callCount = 0
    failService := func(ctx context.Context) error {
        callCount++
        return ErrNotFound // non-retryable
    }

    err = DoWithRetry(ctx, opt, failService)
    fmt.Printf("Non-retryable result: %v (calls: %d)\n", err, callCount)
}
Output
Attempt 1: request timeout (backoff 23ms) Attempt 2: request timeout (backoff 67ms) Result: <nil> (calls: 3) Non-retryable result: non-retryable error: not found (calls: 1)

Common Mistake — Not checking context in the retry loop

WRONG — Ignoring context
func retryIgnoringCtx(fn func() error) error {
    for i := 0; i < 10; i++ {
        if err := fn(); err == nil {
            return nil
        }
        time.Sleep(time.Duration(i) * time.Second)
    }
    return errors.New("failed")
}
// If the parent HTTP request is cancelled,
// this keeps retrying for up to 45 seconds!
// The client already gave up.
RIGHT — Respecting context
func retryWithCtx(ctx context.Context, fn func() error) error {
    for i := 0; i < 10; i++ {
        select {
        case <-ctx.Done():
            return ctx.Err()
        default:
        }
        if err := fn(); err == nil {
            return nil
        }
        select {
        case <-time.After(backoff(i)):
        case <-ctx.Done():
            return ctx.Err()
        }
    }
    return errors.New("failed")
}
Connections: Retry logic relies on context for cancellation, error handling for classifying retryable vs. non-retryable errors, and pairs with circuit breakers to avoid retrying when a dependency is known to be down. The select on time.After and ctx.Done() is a core Go concurrency pattern.

Data Processing

Streaming JSON & Line Protocols

concept_id: streaming_json Data

When processing large JSON files or streams (log files, API responses with thousands of records, NDJSON feeds), loading everything into memory with json.Unmarshal is wasteful or impossible. Go's json.Decoder and bufio.Scanner let you process records one at a time with constant memory.

Why It Matters

  • Processing a 10GB JSON array with json.Unmarshal requires 10GB+ of RAM — streaming uses constant memory
  • NDJSON (newline-delimited JSON) is the standard format for log streams, metrics, and data pipelines
  • Streaming lets you start processing before the entire response has arrived
  • Critical for ETL pipelines, log processing, and data ingestion services

Example 1 — json.Decoder with Token() for streaming arrays

Use Decoder.Token() to navigate JSON structure and Decoder.Decode() to parse individual elements, consuming only one element's worth of memory at a time.

package main

import (
    "encoding/json"
    "fmt"
    "strings"
)

type Record struct {
    ID   int    `json:"id"`
    Name string `json:"name"`
    Amt  float64 `json:"amount"`
}

func main() {
    // Imagine this is a multi-gigabyte file
    jsonStream := `[
        {"id": 1, "name": "Alice", "amount": 100.50},
        {"id": 2, "name": "Bob", "amount": 250.75},
        {"id": 3, "name": "Charlie", "amount": 50.00},
        {"id": 4, "name": "Diana", "amount": 300.25}
    ]`

    dec := json.NewDecoder(strings.NewReader(jsonStream))

    // Read opening bracket '['
    tok, err := dec.Token()
    if err != nil {
        panic(err)
    }
    fmt.Printf("Opening token: %v (type %T)\n", tok, tok)

    // Process each element one at a time
    var total float64
    count := 0
    for dec.More() { // returns true if there are more elements
        var rec Record
        if err := dec.Decode(&rec); err != nil {
            panic(err)
        }
        total += rec.Amt
        count++
        fmt.Printf("  Record %d: %s = $%.2f\n", rec.ID, rec.Name, rec.Amt)
    }

    // Read closing bracket ']'
    tok, err = dec.Token()
    if err != nil {
        panic(err)
    }
    fmt.Printf("Closing token: %v\n", tok)
    fmt.Printf("\nProcessed %d records, total: $%.2f\n", count, total)
}
Output
Opening token: [ (type json.Delim) Record 1: Alice = $100.50 Record 2: Bob = $250.75 Record 3: Charlie = $50.00 Record 4: Diana = $300.25 Closing token: ] Processed 4 records, total: $701.50

Example 2 — Processing NDJSON (newline-delimited JSON)

NDJSON has one JSON object per line. It is trivial to parse with json.Decoder — just call Decode in a loop without the Token/More ceremony.

package main

import (
    "encoding/json"
    "fmt"
    "io"
    "strings"
)

type LogEntry struct {
    Timestamp string `json:"ts"`
    Level     string `json:"level"`
    Message   string `json:"msg"`
    Service   string `json:"service"`
}

func main() {
    // NDJSON: one JSON object per line
    ndjson := `{"ts":"2026-06-17T10:00:00Z","level":"INFO","msg":"Server started","service":"api"}
{"ts":"2026-06-17T10:00:01Z","level":"DEBUG","msg":"Health check passed","service":"api"}
{"ts":"2026-06-17T10:00:02Z","level":"ERROR","msg":"Database connection timeout","service":"worker"}
{"ts":"2026-06-17T10:00:03Z","level":"WARN","msg":"Retry attempt 2","service":"worker"}
{"ts":"2026-06-17T10:00:04Z","level":"INFO","msg":"Connection restored","service":"worker"}
`

    dec := json.NewDecoder(strings.NewReader(ndjson))

    var errors []LogEntry
    total := 0

    for {
        var entry LogEntry
        err := dec.Decode(&entry)
        if err == io.EOF {
            break // end of stream
        }
        if err != nil {
            fmt.Println("Parse error:", err)
            continue // skip malformed lines
        }

        total++
        if entry.Level == "ERROR" {
            errors = append(errors, entry)
        }
        fmt.Printf("[%s] %s: %s (%s)\n",
            entry.Timestamp, entry.Level, entry.Message, entry.Service)
    }

    fmt.Printf("\nTotal entries: %d, Errors: %d\n", total, len(errors))
    for _, e := range errors {
        fmt.Printf("  ERROR: %s\n", e.Message)
    }
}
Output
[2026-06-17T10:00:00Z] INFO: Server started (api) [2026-06-17T10:00:01Z] DEBUG: Health check passed (api) [2026-06-17T10:00:02Z] ERROR: Database connection timeout (worker) [2026-06-17T10:00:03Z] WARN: Retry attempt 2 (worker) [2026-06-17T10:00:04Z] INFO: Connection restored (worker) Total entries: 5, Errors: 1 ERROR: Database connection timeout

Example 3 — Streaming JSON array elements with filtering

Combining streaming with filtering and transformation: process a large JSON array, filter records, and write results to an output stream without holding the full input or output in memory.

package main

import (
    "encoding/json"
    "fmt"
    "os"
    "strings"
)

type Transaction struct {
    ID       string  `json:"id"`
    Amount   float64 `json:"amount"`
    Currency string  `json:"currency"`
    Status   string  `json:"status"`
}

type Summary struct {
    ID     string  `json:"id"`
    Amount float64 `json:"amount_usd"`
}

func processStream(input string) error {
    dec := json.NewDecoder(strings.NewReader(input))
    enc := json.NewEncoder(os.Stdout)
    enc.SetIndent("", "  ")

    // Skip opening '['
    if _, err := dec.Token(); err != nil {
        return fmt.Errorf("expected '[': %w", err)
    }

    rates := map[string]float64{
        "USD": 1.0, "EUR": 1.08, "GBP": 1.27, "JPY": 0.0067,
    }

    fmt.Println("Filtered high-value USD transactions:")
    count := 0
    for dec.More() {
        var tx Transaction
        if err := dec.Decode(&tx); err != nil {
            return fmt.Errorf("decode error: %w", err)
        }

        // Skip non-completed transactions
        if tx.Status != "completed" {
            continue
        }

        // Convert to USD
        rate, ok := rates[tx.Currency]
        if !ok {
            continue
        }
        usdAmount := tx.Amount * rate

        // Filter: only high-value transactions
        if usdAmount >= 100.0 {
            count++
            enc.Encode(Summary{ID: tx.ID, Amount: usdAmount})
        }
    }

    // Skip closing ']'
    if _, err := dec.Token(); err != nil {
        return fmt.Errorf("expected ']': %w", err)
    }

    fmt.Printf("\nProcessed stream: %d high-value transactions found\n", count)
    return nil
}

func main() {
    transactions := `[
        {"id":"tx-001","amount":150.00,"currency":"USD","status":"completed"},
        {"id":"tx-002","amount":50.00,"currency":"EUR","status":"completed"},
        {"id":"tx-003","amount":200.00,"currency":"GBP","status":"pending"},
        {"id":"tx-004","amount":90.00,"currency":"EUR","status":"completed"},
        {"id":"tx-005","amount":25000,"currency":"JPY","status":"completed"},
        {"id":"tx-006","amount":500.00,"currency":"USD","status":"completed"}
    ]`

    if err := processStream(transactions); err != nil {
        fmt.Fprintf(os.Stderr, "Error: %v\n", err)
        os.Exit(1)
    }
}
Output
Filtered high-value USD transactions: { "id": "tx-001", "amount_usd": 150 } { "id": "tx-006", "amount_usd": 500 } Processed stream: 2 high-value transactions found

Note: tx-002 ($54 USD) and tx-005 ($167.50 USD from 25000 JPY) were filtered by the $100 threshold or other criteria. tx-003 was pending. tx-004 at 90 EUR = $97.20 was just under the threshold.

Common Mistakes

WRONG — Loading entire file into memory
// For a 5GB file, this needs 5GB+ RAM

// Even io.ReadAll is dangerous:

RIGHT — Stream from file/reader

defer f.Close()

dec.Token() // skip '['
for dec.More() {
    var rec Record
    dec.Decode(&rec) // one record at a time
    process(rec)
}
dec.Token() // skip ']'
// Constant memory regardless of file size
WRONG — Ignoring decoder errors
for dec.More() {
    var rec Record
    dec.Decode(&rec) // error ignored!
    results = append(results, rec)
}
// Malformed JSON silently produces
// zero-value records in your results
RIGHT — Check every error
for dec.More() {
    var rec Record
    if err := dec.Decode(&rec); err != nil {
        log.Printf("skip malformed: %v", err)
        continue // or return err
    }
    results = append(results, rec)
}
Connections: Streaming JSON builds on io.Reader (the decoder wraps any reader), encoding/json (struct tags, custom unmarshalers), and bufio (for line-based NDJSON processing). Combine with worker pools (next concept) to parallelize record processing.

Worker Pool with Error Collection

concept_id: worker_pool_errors Concurrency

Worker pools bound concurrency to a fixed number of goroutines, preventing resource exhaustion. The real challenge is collecting results and errors from workers correctly: handling partial failures, avoiding goroutine leaks, and ensuring all workers finish before returning.

Why It Matters

  • Unbounded goroutine creation (one per task) can exhaust memory and file descriptors
  • Worker pools let you process 100,000 tasks with only 10 goroutines
  • Proper error collection lets you report partial success (90% of files processed)
  • This is the standard concurrency pattern in production Go services

Example 1 — Simple worker pool with channels

The classic Go worker pool: a jobs channel feeds workers, a results channel collects output.

package main

import (
    "fmt"
    "math/rand"
    "sync"
    "time"
)

type Job struct {
    ID    int
    Input string
}

type Result struct {
    JobID  int
    Output string
    Err    error
}

func worker(id int, jobs <-chan Job, results chan<- Result, wg *sync.WaitGroup) {
    defer wg.Done()
    for job := range jobs {
        // Simulate work with random duration
        time.Sleep(time.Duration(50+rand.Intn(100)) * time.Millisecond)

        // Simulate occasional failures
        if job.ID%5 == 0 {
            results <- Result{
                JobID: job.ID,
                Err:   fmt.Errorf("job %d: simulated failure", job.ID),
            }
            continue
        }

        results <- Result{
            JobID:  job.ID,
            Output: fmt.Sprintf("processed-%s", job.Input),
        }
    }
}

func main() {
    const numWorkers = 3
    const numJobs = 12

    jobs := make(chan Job, numJobs)
    results := make(chan Result, numJobs)

    // Start workers
    var wg sync.WaitGroup
    for w := 1; w <= numWorkers; w++ {
        wg.Add(1)
        go worker(w, jobs, results, &wg)
    }

    // Send jobs
    for j := 1; j <= numJobs; j++ {
        jobs <- Job{ID: j, Input: fmt.Sprintf("item-%d", j)}
    }
    close(jobs) // Signal no more jobs

    // Close results channel when all workers done
    go func() {
        wg.Wait()
        close(results)
    }()

    // Collect results
    var successes, failures int
    for r := range results {
        if r.Err != nil {
            failures++
            fmt.Printf("  FAIL: %v\n", r.Err)
        } else {
            successes++
            fmt.Printf("  OK:   job %d -> %s\n", r.JobID, r.Output)
        }
    }

    fmt.Printf("\nSummary: %d succeeded, %d failed out of %d jobs\n",
        successes, failures, numJobs)
}
Output
OK: job 2 -> processed-item-2 OK: job 1 -> processed-item-1 OK: job 3 -> processed-item-3 OK: job 4 -> processed-item-4 FAIL: job 5: simulated failure OK: job 6 -> processed-item-6 OK: job 7 -> processed-item-7 OK: job 8 -> processed-item-8 OK: job 9 -> processed-item-9 FAIL: job 10: simulated failure OK: job 11 -> processed-item-11 OK: job 12 -> processed-item-12 Summary: 10 succeeded, 2 failed out of 12 jobs

Example 2 — Worker pool with errgroup and semaphore

The modern Go approach: errgroup manages goroutine lifecycle and error propagation. SetLimit acts as a semaphore for bounded concurrency.

package main

import (
    "context"
    "fmt"
    "math/rand"
    "sync"
    "time"

    "golang.org/x/sync/errgroup"
)

func main() {
    urls := []string{
        "https://api.example.com/users",
        "https://api.example.com/orders",
        "https://api.example.com/products",
        "https://api.example.com/inventory",
        "https://api.example.com/reports",
        "https://api.example.com/analytics",
        "https://api.example.com/settings",
        "https://api.example.com/notifications",
    }

    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(3) // Max 3 concurrent goroutines

    // Thread-safe result collection
    var mu sync.Mutex
    results := make(map[string]int)

    for _, url := range urls {
        url := url // capture loop variable (needed before Go 1.22)
        g.Go(func() error {
            // Simulate HTTP fetch
            time.Sleep(time.Duration(50+rand.Intn(150)) * time.Millisecond)

            // Check context before processing
            select {
            case <-ctx.Done():
                return ctx.Err()
            default:
            }

            // Simulate response size
            size := 100 + rand.Intn(900)

            mu.Lock()
            results[url] = size
            mu.Unlock()

            fmt.Printf("  Fetched %s (%d bytes)\n", url, size)
            return nil
        })
    }

    // Wait for all goroutines to finish
    if err := g.Wait(); err != nil {
        fmt.Printf("Error: %v\n", err)
    }

    fmt.Printf("\nFetched %d/%d URLs successfully\n", len(results), len(urls))
}

Common Mistake — Goroutine leak on error

WRONG — Goroutine leak
results := make(chan string) // unbuffered!
for _, url := range urls {
    go func(u string) {
        resp := fetch(u) // blocks
        results <- resp  // blocks if nobody reads!
    }(url)
}
// If we return early on first error,
// remaining goroutines block forever on
// results <- resp (channel send)
result := <-results
if isError(result) {
    return // LEAK: other goroutines stuck
}
RIGHT — Buffered channel or context
// Option 1: Buffered channel (never blocks)
results := make(chan string, len(urls))
for _, url := range urls {
    go func(u string) {
        results <- fetch(u) // never blocks
    }(url)
}

// Option 2: Context cancellation
g, ctx := errgroup.WithContext(ctx)
for _, url := range urls {
    url := url
    g.Go(func() error {
        return fetchWithCtx(ctx, url)
        // ctx cancelled = goroutines exit
    })
}
return g.Wait()

Example 3 — Worker pool with result collection and partial failure

A production worker pool that collects both successes and failures, reports partial progress, and supports cancellation.

package main

import (
    "context"
    "fmt"
    "math/rand"
    "sync"
    "time"
)

// Task represents a unit of work.
type Task struct {
    ID   int
    Name string
}

// TaskResult holds the outcome of a task.
type TaskResult struct {
    TaskID int
    Name   string
    Data   string
    Err    error
}

// PoolResult holds the aggregate outcome.
type PoolResult struct {
    Succeeded []TaskResult
    Failed    []TaskResult
}

// RunPool processes tasks with bounded concurrency.
func RunPool(ctx context.Context, tasks []Task, concurrency int) PoolResult {
    taskCh := make(chan Task, len(tasks))
    resultCh := make(chan TaskResult, len(tasks))

    // Start workers
    var wg sync.WaitGroup
    for i := 0; i < concurrency; i++ {
        wg.Add(1)
        go func(workerID int) {
            defer wg.Done()
            for task := range taskCh {
                select {
                case <-ctx.Done():
                    resultCh <- TaskResult{
                        TaskID: task.ID,
                        Name:   task.Name,
                        Err:    ctx.Err(),
                    }
                    continue
                default:
                }

                result := processTask(task)
                resultCh <- result
            }
        }(i)
    }

    // Send all tasks
    for _, task := range tasks {
        taskCh <- task
    }
    close(taskCh)

    // Close results when all workers done
    go func() {
        wg.Wait()
        close(resultCh)
    }()

    // Collect results
    var pool PoolResult
    for r := range resultCh {
        if r.Err != nil {
            pool.Failed = append(pool.Failed, r)
        } else {
            pool.Succeeded = append(pool.Succeeded, r)
        }
    }
    return pool
}

func processTask(t Task) TaskResult {
    // Simulate variable processing time
    time.Sleep(time.Duration(30+rand.Intn(70)) * time.Millisecond)

    // Simulate 20% failure rate
    if rand.Intn(5) == 0 {
        return TaskResult{
            TaskID: t.ID,
            Name:   t.Name,
            Err:    fmt.Errorf("processing failed for %q", t.Name),
        }
    }

    return TaskResult{
        TaskID: t.ID,
        Name:   t.Name,
        Data:   fmt.Sprintf("result-of-%s", t.Name),
    }
}

func main() {
    tasks := make([]Task, 15)
    for i := range tasks {
        tasks[i] = Task{ID: i + 1, Name: fmt.Sprintf("task-%d", i+1)}
    }

    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    start := time.Now()
    result := RunPool(ctx, tasks, 4) // 4 workers
    elapsed := time.Since(start)

    fmt.Println("=== Succeeded ===")
    for _, r := range result.Succeeded {
        fmt.Printf("  [%d] %s -> %s\n", r.TaskID, r.Name, r.Data)
    }

    fmt.Println("\n=== Failed ===")
    for _, r := range result.Failed {
        fmt.Printf("  [%d] %s -> %v\n", r.TaskID, r.Name, r.Err)
    }

    total := len(result.Succeeded) + len(result.Failed)
    fmt.Printf("\nCompleted %d/%d tasks in %dms (%d succeeded, %d failed)\n",
        total, len(tasks), elapsed.Milliseconds(),
        len(result.Succeeded), len(result.Failed))
    fmt.Printf("Success rate: %.1f%%\n",
        float64(len(result.Succeeded))/float64(total)*100)
}

Common Mistake — Unbounded goroutine creation

WRONG — One goroutine per task
// With 100,000 URLs, this creates 100,000
// goroutines, each with its own stack and
// potentially its own HTTP connection.
for _, url := range urls { // len=100000
    go func(u string) {
        resp, _ := http.Get(u)
        // 100K concurrent connections!
        // OS runs out of file descriptors
        process(resp)
    }(url)
}
// Also: no way to know when they finish!
RIGHT — Bounded worker pool
g, ctx := errgroup.WithContext(ctx)
g.SetLimit(20) // max 20 concurrent

for _, url := range urls { // len=100000
    url := url
    g.Go(func() error {
        req, _ := http.NewRequestWithContext(
            ctx, "GET", url, nil)
        resp, err := http.DefaultClient.Do(req)
        if err != nil { return err }
        defer resp.Body.Close()
        return process(resp)
    })
}
err := g.Wait() // blocks until all done

Common Mistake — Deadlock from full channels

WRONG — Deadlock
ch := make(chan int) // unbuffered
for i := 0; i < 10; i++ {
    ch <- i // blocks on first send!
    // No goroutine is reading yet.
}
// This is a deadlock. The main goroutine
// blocks on ch <- 0 because nobody is
// reading from ch.
RIGHT — Producer and consumer in separate goroutines
ch := make(chan int, 10) // buffered
// OR: start consumer first
go func() {
    for v := range ch {
        process(v)
    }
}()
for i := 0; i < 10; i++ {
    ch <- i // won't block (consumer is reading)
}
close(ch)
Connections: Worker pools combine goroutines (concurrent execution), channels (job distribution and result collection), errgroup (lifecycle management), and context (cancellation propagation). They pair naturally with streaming JSON (previous concept) for processing records from a stream in parallel.
