Slices Deep Dive

Go Foundations • New

Slices are Go's workhorse collection type. Unlike arrays (which have a fixed size baked into their type), slices are dynamic, flexible views into underlying arrays. Understanding the relationship between a slice's length, capacity, and its backing array is essential to writing correct Go.

Creating Slices

There are several ways to create a slice, each with different implications:

package main

import "fmt"

func main() {
    // 1. Slice literal
    s1 := []int{1, 2, 3}
    fmt.Println("Literal:", s1, "len:", len(s1), "cap:", cap(s1))

    // 2. make(type, length, capacity)
    s2 := make([]int, 3, 10) // length 3, capacity 10
    fmt.Println("Make:", s2, "len:", len(s2), "cap:", cap(s2))

    // 3. make(type, length) — capacity defaults to length
    s3 := make([]int, 5)
    fmt.Println("Make (no cap):", s3, "len:", len(s3), "cap:", cap(s3))

    // 4. nil slice — no allocation, but perfectly valid
    var s4 []int
    fmt.Println("Nil slice:", s4, "len:", len(s4), "cap:", cap(s4))
    fmt.Println("s4 == nil?", s4 == nil) // true

    // 5. Empty slice — allocated but empty
    s5 := []int{}
    fmt.Println("Empty slice:", s5, "len:", len(s5), "cap:", cap(s5))
    fmt.Println("s5 == nil?", s5 == nil) // false!
}
Output
Literal: [1 2 3] len: 3 cap: 3 Make: [0 0 0] len: 3 cap: 10 Make (no cap): [0 0 0 0 0] len: 5 cap: 5 Nil slice: [] len: 0 cap: 0 s4 == nil? true Empty slice: [] len: 0 cap: 0 s5 == nil? false
nil slice vs empty slice: Both var s []int and s := []int{} have length 0, but var s []int is nil while []int{} is not. For most operations (append, len, range) they behave identically. The difference matters when encoding to JSON: a nil slice encodes as null, while an empty slice encodes as [].

Length vs Capacity & How Append Grows

A slice has three components: a pointer to the backing array, a length (number of accessible elements), and a capacity (total space available before a new array is needed).

package main

import "fmt"

func main() {
    s := make([]int, 0, 4)
    fmt.Printf("Start: len=%d cap=%d %v\n", len(s), cap(s), s)

    for i := 1; i <= 10; i++ {
        s = append(s, i)
        fmt.Printf("After append(%d): len=%d cap=%d\n", i, len(s), cap(s))
    }
}
Output
Start: len=0 cap=4 [] After append(1): len=1 cap=4 After append(2): len=2 cap=4 After append(3): len=3 cap=4 After append(4): len=4 cap=4 After append(5): len=5 cap=8 After append(6): len=6 cap=8 After append(7): len=7 cap=8 After append(8): len=8 cap=8 After append(9): len=9 cap=16 After append(10): len=10 cap=16

When append exceeds capacity, Go allocates a new, larger backing array and copies elements over. For small slices, capacity roughly doubles. For larger slices (>256 elements in Go 1.21+), it grows by ~25%.

Slice Expressions & the Three-Index Slice

package main

import "fmt"

func main() {
    a := []int{10, 20, 30, 40, 50}

    // Standard slice: s[low:high]
    b := a[1:4]
    fmt.Println("b:", b, "len:", len(b), "cap:", cap(b))
    // b shares memory with a! cap extends to end of a's backing array.

    // Three-index slice: s[low:high:max] — limits capacity
    c := a[1:4:4]
    fmt.Println("c:", c, "len:", len(c), "cap:", cap(c))
    // c also shares memory, but cap is limited to 3 (4-1).
    // Any append to c will allocate a NEW backing array.

    // This is how you make a "safe" sub-slice
    c = append(c, 99)
    fmt.Println("After append to c:")
    fmt.Println("  a:", a) // a is NOT modified
    fmt.Println("  c:", c) // c has its own array

    // Without three-index, append would modify a!
    b = append(b, 99)
    fmt.Println("After append to b:")
    fmt.Println("  a:", a) // a[4] is now 99!
    fmt.Println("  b:", b)
}
Output
b: [20 30 40] len: 3 cap: 4 c: [20 30 40] len: 3 cap: 3 After append to c: a: [10 20 30 40 50] c: [20 30 40 99] After append to b: a: [10 20 30 40 99] b: [20 30 40 99]
Shared backing array: When you slice a slice, the new slice shares the same underlying array. Modifications to one are visible in the other. Use the three-index form s[low:high:max] to limit capacity and force a new allocation on append.

append() Must Be Reassigned

append may or may not return the same slice. If the backing array has capacity, it modifies in place and returns the original header. If not, it allocates a new array and returns a different header. You must always reassign:

Do
s = append(s, val)
Don't
append(s, val) // return value discarded!

copy() for Independent Copies

package main

import "fmt"

func main() {
    src := []int{1, 2, 3, 4, 5}

    // copy(dst, src) — copies min(len(dst), len(src)) elements
    dst := make([]int, len(src))
    n := copy(dst, src)
    fmt.Println("Copied", n, "elements:", dst)

    // Modify dst — src is unaffected
    dst[0] = 999
    fmt.Println("src:", src)
    fmt.Println("dst:", dst)

    // copy into a smaller slice — only copies what fits
    small := make([]int, 2)
    copy(small, src)
    fmt.Println("small:", small)
}
Output
Copied 5 elements: [1 2 3 4 5] src: [1 2 3 4 5] dst: [999 2 3 4 5] small: [1 2]

Slices as Function Parameters

package main

import "fmt"

func double(s []int) {
    for i := range s {
        s[i] *= 2
    }
}

func tryAppend(s []int) {
    s = append(s, 999) // This append is NOT visible to caller
    fmt.Println("Inside tryAppend:", s)
}

func main() {
    nums := []int{1, 2, 3}

    double(nums)
    fmt.Println("After double:", nums) // [2 4 6] — modifications visible!

    tryAppend(nums)
    fmt.Println("After tryAppend:", nums) // [2 4 6] — append NOT visible!
}
Output
After double: [2 4 6] Inside tryAppend: [2 4 6 999] After tryAppend: [2 4 6]

The slice header (pointer, length, capacity) is passed by value. Element modifications go through the shared pointer and are visible. But append may change the header (new pointer, new length), and those changes are local to the function.

Real-World: Stack Implementation

package main

import "fmt"

type Stack struct {
    items []int
}

func (st *Stack) Push(val int) {
    st.items = append(st.items, val)
}

func (st *Stack) Pop() (int, bool) {
    if len(st.items) == 0 {
        return 0, false
    }
    top := st.items[len(st.items)-1]
    st.items = st.items[:len(st.items)-1]
    return top, true
}

func (st *Stack) Peek() (int, bool) {
    if len(st.items) == 0 {
        return 0, false
    }
    return st.items[len(st.items)-1], true
}

func (st *Stack) Size() int {
    return len(st.items)
}

func main() {
    s := &Stack{}
    s.Push(10)
    s.Push(20)
    s.Push(30)
    fmt.Println("Size:", s.Size())

    if val, ok := s.Pop(); ok {
        fmt.Println("Popped:", val)
    }
    if val, ok := s.Peek(); ok {
        fmt.Println("Peek:", val)
    }
    fmt.Println("Size:", s.Size())
}
Output
Size: 3 Popped: 30 Peek: 20 Size: 2

Common Mistakes

Do: Reassign append
s = append(s, 42)
Don't: Discard append return
append(s, 42) // compiles but does nothing!
Do: Use copy or three-index slice
b := make([]int, len(a))
copy(b, a) // independent copy
Don't: Assume slices are independent
b := a[1:3]
b[0] = 99 // also changes a[1]!

Key Rules

  • Always reassign the result of append: s = append(s, ...)
  • Use copy() or the three-index slice s[lo:hi:hi] when you need an independent slice
  • Prefer var s []int (nil slice) over s := []int{} unless you specifically need a non-nil empty slice (e.g., JSON encoding)
  • Pre-allocate with make([]T, 0, expectedLen) when you know the approximate size to avoid repeated allocations
  • Remember: len() is how many elements, cap() is how many it can hold before growing
Connection: Slices are closely related to Pointers (the slice header contains a pointer to the backing array) and Maps (maps of slices are a common pattern for grouping). You will see slices used everywhere in Go — function parameters, return values, and as the backbone of algorithms.

Maps

Go Foundations • New

Maps are Go's built-in hash table. They provide O(1) average-case lookups, inserts, and deletes. Maps are reference types — passing a map to a function gives that function full access to the same data.

Creating Maps

package main

import "fmt"

func main() {
    // 1. Map literal
    m1 := map[string]int{
        "alice": 90,
        "bob":   85,
        "carol": 92,
    }
    fmt.Println("Literal:", m1)

    // 2. make()
    m2 := make(map[string]int)
    m2["dave"] = 88
    fmt.Println("Make:", m2)

    // 3. nil map — reads work, writes PANIC
    var m3 map[string]int
    fmt.Println("nil map read:", m3["anything"]) // returns zero value, no panic
    fmt.Println("m3 == nil?", m3 == nil)
    // m3["key"] = 1 // PANIC: assignment to entry in nil map
}
Output
Literal: map[alice:90 bob:85 carol:92] Make: map[dave:88] nil map read: 0 m3 == nil? true
nil map trap: A nil map allows reads (returning the zero value) but panics on write. Always initialize maps with make() or a literal before writing. This is one of the most common Go panics in production code.

Lookup with Comma-Ok Pattern

Reading a missing key returns the zero value. To distinguish "key exists with zero value" from "key missing," use the comma-ok idiom:

package main

import "fmt"

func main() {
    scores := map[string]int{"alice": 0, "bob": 95}

    // Bad: can't tell if alice has score 0 or doesn't exist
    fmt.Println("alice:", scores["alice"])   // 0
    fmt.Println("missing:", scores["carol"]) // 0 — same!

    // Good: comma-ok pattern
    if val, ok := scores["alice"]; ok {
        fmt.Println("alice exists, score:", val)
    }
    if _, ok := scores["carol"]; !ok {
        fmt.Println("carol does NOT exist")
    }
}
Output
alice: 0 missing: 0 alice exists, score: 0 carol does NOT exist

Iteration Order Is Random

package main

import "fmt"

func main() {
    m := map[string]int{"a": 1, "b": 2, "c": 3, "d": 4}

    // Each run may print in a different order!
    for k, v := range m {
        fmt.Printf("%s=%d ", k, v)
    }
    fmt.Println()

    // If you need sorted output, sort the keys first
    // (We'll skip the import for brevity, but use sort.Strings)
}
Random iteration: Go deliberately randomizes map iteration order to prevent code from accidentally depending on it. If you need deterministic order, extract keys into a slice and sort them.

Delete & Maps as Function Parameters

package main

import "fmt"

func addEntry(m map[string]int, key string, val int) {
    m[key] = val // Modifies the caller's map — maps are reference types
}

func main() {
    m := map[string]int{"a": 1, "b": 2, "c": 3}
    fmt.Println("Before delete:", m)

    delete(m, "b")        // Remove key "b"
    delete(m, "missing")  // No-op, no panic
    fmt.Println("After delete:", m)

    addEntry(m, "d", 4)
    fmt.Println("After addEntry:", m)
}
Output
Before delete: map[a:1 b:2 c:3] After delete: map[a:1 c:3] After addEntry: map[a:1 c:3 d:4]

Map of Slices — Grouping Pattern

package main

import "fmt"

func groupByLength(words []string) map[int][]string {
    groups := make(map[int][]string)
    for _, w := range words {
        groups[len(w)] = append(groups[len(w)], w)
    }
    return groups
}

func main() {
    words := []string{"go", "is", "fun", "maps", "are", "great", "hi"}
    groups := groupByLength(words)
    for length, ws := range groups {
        fmt.Printf("Length %d: %v\n", length, ws)
    }
}
Output (order may vary)
Length 2: [go is hi] Length 3: [fun are] Length 4: [maps] Length 5: [great]

Set Implementation with Maps

package main

import "fmt"

func main() {
    // Option 1: map[T]bool — slightly more readable
    setBool := map[string]bool{}
    setBool["apple"] = true
    setBool["banana"] = true
    if setBool["apple"] {
        fmt.Println("apple is in setBool")
    }

    // Option 2: map[T]struct{} — zero memory for values
    setEmpty := map[string]struct{}{}
    setEmpty["apple"] = struct{}{}
    setEmpty["banana"] = struct{}{}
    if _, ok := setEmpty["apple"]; ok {
        fmt.Println("apple is in setEmpty")
    }

    // struct{} takes 0 bytes; bool takes 1 byte per entry
    // For large sets, struct{} saves memory
    fmt.Printf("bool map: %v\n", setBool)
}
Output
apple is in setBool apple is in setEmpty bool map: map[apple:true banana:true]

Concurrent Access Warning

Concurrent map read/write is a fatal error: Unlike a panic (which can be recovered), concurrent map access causes a fatal error: concurrent map read and map write that kills the entire program. Protect maps with sync.RWMutex or use sync.Map for concurrent access. This will be covered in depth when we study goroutines and concurrency.

Common Mistakes

Do: Initialize before writing
m := make(map[string]int)
m["key"] = 1
Don't: Write to nil map
var m map[string]int
m["key"] = 1 // PANIC!
Do: Use comma-ok for existence check
if val, ok := m[key]; ok {
    // key exists
}
Don't: Check zero value for existence
if m[key] != 0 {
    // fails for keys with value 0
}

Key Rules

  • Always initialize maps before writing: make(map[K]V) or a map literal
  • Use the comma-ok pattern to distinguish "missing" from "zero value"
  • Never rely on map iteration order — it is random by design
  • Maps are reference types: passing to a function shares the data
  • For concurrent access, use sync.RWMutex or sync.Map
  • Use map[T]struct{} for memory-efficient sets
Connection: Maps pair naturally with Slices for grouping patterns (map[K][]V). The comma-ok pattern you learned here is the same pattern used in type assertions for Interfaces and channel receives in concurrency.

Strings & Runes

Go Foundations • New

Go strings are immutable sequences of bytes, encoded as UTF-8. This is a fundamental design choice that has deep implications: a single character (like a Chinese character or an emoji) may span multiple bytes. Go provides the rune type (an alias for int32) to represent individual Unicode code points.

Bytes vs Runes

package main

import (
    "fmt"
    "unicode/utf8"
)

func main() {
    s := "Hello, World!"
    fmt.Println("len(s):", len(s))                          // 14 bytes
    fmt.Println("rune count:", utf8.RuneCountInString(s))   // 14 runes

    // Now with multi-byte characters
    s2 := "Go is fun!"
    fmt.Println("\nlen(s2):", len(s2))                        // 17 bytes (emoji is 4 bytes)
    fmt.Println("rune count:", utf8.RuneCountInString(s2))   // 14 runes

    // Indexing gives BYTES, not characters
    fmt.Printf("\ns[0] = %d (byte: %c)\n", s[0], s[0])

    // Ranging gives RUNES
    fmt.Println("\nRanging over runes:")
    for i, r := range "café" { // "cafe" + combining accent = "café"
        fmt.Printf("  index=%d rune=%c (U+%04X)\n", i, r, r)
    }
}
Output
len(s): 13 rune count: 13 len(s2): 14 rune count: 11 s[0] = 72 (byte: H) Ranging over runes: index=0 rune=c (U+0063) index=1 rune=a (U+0061) index=2 rune=f (U+0066) index=3 rune=e (U+0065) index=4 rune=́ (U+0301)
len() counts bytes, not characters: len("hello") is 5 because each ASCII character is 1 byte. But len("Go") could be more than the number of visible characters because multi-byte UTF-8 characters are counted as their byte length. Use utf8.RuneCountInString() for character count.

[]byte vs []rune Conversions

package main

import "fmt"

func main() {
    s := "Hello"

    // Convert to byte slice — for byte-level processing
    bytes := []byte(s)
    fmt.Println("bytes:", bytes)

    // Convert to rune slice — for character-level processing
    runes := []rune(s)
    fmt.Println("runes:", runes)

    // Reverse a string (Unicode-safe)
    original := "Hello"
    r := []rune(original)
    for i, j := 0, len(r)-1; i < j; i, j = i+1, j-1 {
        r[i], r[j] = r[j], r[i]
    }
    fmt.Println("reversed:", string(r))
}
Output
bytes: [72 101 108 108 111] runes: [72 101 108 108 111] reversed: olleH

The strings Package

package main

import (
    "fmt"
    "strings"
)

func main() {
    s := "  Hello, Go World!  "

    fmt.Println(strings.TrimSpace(s))              // "Hello, Go World!"
    fmt.Println(strings.Contains(s, "Go"))          // true
    fmt.Println(strings.HasPrefix(s, "  Hello"))    // true
    fmt.Println(strings.HasSuffix(s, "!  "))        // true
    fmt.Println(strings.ToUpper("hello"))           // "HELLO"
    fmt.Println(strings.ToLower("HELLO"))           // "hello"
    fmt.Println(strings.Replace("aabaa", "a", "x", 2)) // "xxbaa" (replace first 2)
    fmt.Println(strings.ReplaceAll("aabaa", "a", "x")) // "xxbxx"
    fmt.Println(strings.Index("hello", "ll"))       // 2
    fmt.Println(strings.Repeat("Go! ", 3))          // "Go! Go! Go! "

    // Split and Join
    parts := strings.Split("a,b,c,d", ",")
    fmt.Println("Split:", parts)
    fmt.Println("Join:", strings.Join(parts, " | "))
}
Output
Hello, Go World! true true true HELLO hello xxbaa xxbxx 2 Go! Go! Go! Split: [a b c d] Join: a | b | c | d

strings.Builder for Efficient Concatenation

package main

import (
    "fmt"
    "strings"
)

func main() {
    // BAD: += creates a new string each time — O(n^2) for n concatenations
    result := ""
    for i := 0; i < 5; i++ {
        result += fmt.Sprintf("item%d ", i)
    }
    fmt.Println("+=:", result)

    // GOOD: strings.Builder is O(n) total
    var b strings.Builder
    for i := 0; i < 5; i++ {
        fmt.Fprintf(&b, "item%d ", i)
    }
    fmt.Println("Builder:", b.String())
}
Output
+=: item0 item1 item2 item3 item4 Builder: item0 item1 item2 item3 item4

strconv: Number/String Conversions

package main

import (
    "fmt"
    "strconv"
)

func main() {
    // Int to string
    s := strconv.Itoa(42)
    fmt.Println("Itoa:", s, fmt.Sprintf("(type: %T)", s))

    // String to int
    n, err := strconv.Atoi("123")
    if err != nil {
        fmt.Println("Error:", err)
    } else {
        fmt.Println("Atoi:", n, fmt.Sprintf("(type: %T)", n))
    }

    // Atoi with bad input
    _, err = strconv.Atoi("not_a_number")
    fmt.Println("Atoi error:", err)

    // FormatFloat / ParseFloat
    f := strconv.FormatFloat(3.14159, 'f', 2, 64)
    fmt.Println("FormatFloat:", f)
}
Output
Itoa: 42 (type: string) Atoi: 123 (type: int) Atoi error: strconv.Atoi: parsing "not_a_number": invalid syntax FormatFloat: 3.14

Common Mistakes

Do: Use runes for character ops
r := []rune(s)
for i, j := 0, len(r)-1; i < j; i, j = i+1, j-1 {
    r[i], r[j] = r[j], r[i]
}
reversed := string(r)
Don't: Index bytes as characters
// s[i] gives a BYTE, not a character
for i := 0; i < len(s); i++ {
    fmt.Println(s[i]) // byte values!
}
Do: Use strings.Builder for loops
var b strings.Builder
for _, s := range items {
    b.WriteString(s)
}
result := b.String()
Don't: Concatenate with += in loops
result := ""
for _, s := range items {
    result += s // O(n^2) allocations!
}

Key Rules

  • len(s) returns byte count; use utf8.RuneCountInString(s) for character count
  • Convert to []rune for character-level operations (reversing, indexing by character)
  • Use strings.Builder for building strings in loops — never +=
  • Ranging over a string with for i, r := range s iterates runes, not bytes
  • Use strconv.Atoi/strconv.Itoa for number/string conversions
Connection: Understanding byte vs rune is essential for Slices (since strings are essentially []byte). The io.Reader interface you will learn in Interfaces reads bytes, so knowing how UTF-8 encoding works helps you process text streams correctly.

Structs & Methods

Go Foundations • New

Structs are Go's primary way to define custom types with named fields. Combined with methods, they give you the ability to build clean abstractions. Go does not have classes — instead, it uses structs with methods and composition via embedding.

Defining and Creating Structs

package main

import "fmt"

type Point struct {
    X float64
    Y float64
}

type Person struct {
    Name string
    Age  int
    Email string `json:"email,omitempty"` // struct tag for JSON
}

func main() {
    // Named field initialization (preferred)
    p1 := Point{X: 3.0, Y: 4.0}
    fmt.Println("p1:", p1)

    // Positional initialization (fragile — avoid!)
    p2 := Point{1.0, 2.0}
    fmt.Println("p2:", p2)

    // Zero value — all fields get zero values
    var p3 Point
    fmt.Println("p3 (zero):", p3) // {0 0}

    // Pointer to struct
    p4 := &Point{X: 5, Y: 6}
    fmt.Println("p4:", *p4)
    fmt.Println("p4.X:", p4.X) // Go auto-dereferences!

    // new() returns a pointer to zero-valued struct
    p5 := new(Point)
    fmt.Println("p5:", *p5) // {0 0}
}
Output
p1: {3 4} p2: {1 2} p3 (zero): {0 0} p4: {5 6} p4.X: 5 p5: {0 0}

Value Receivers vs Pointer Receivers

package main

import (
    "fmt"
    "math"
)

type Circle struct {
    Radius float64
}

// Value receiver — gets a COPY of the struct
func (c Circle) Area() float64 {
    return math.Pi * c.Radius * c.Radius
}

// Pointer receiver — can MODIFY the original struct
func (c *Circle) Scale(factor float64) {
    c.Radius *= factor
}

// Value receiver — cannot modify (changes are lost)
func (c Circle) ScaleBroken(factor float64) {
    c.Radius *= factor // modifies the copy, not the original!
}

func main() {
    c := Circle{Radius: 5}
    fmt.Printf("Area: %.2f\n", c.Area())

    c.ScaleBroken(2)
    fmt.Println("After ScaleBroken(2):", c.Radius) // Still 5!

    c.Scale(2)
    fmt.Println("After Scale(2):", c.Radius) // Now 10
    fmt.Printf("New Area: %.2f\n", c.Area())
}
Output
Area: 78.54 After ScaleBroken(2): 5 After Scale(2): 10 New Area: 314.16
Value receiver trap: A value receiver method works on a copy of the struct. Any modifications inside the method are lost when the method returns. If your method needs to modify the struct, you must use a pointer receiver.

Method Set Consistency

The Go convention is: if any method on a type needs a pointer receiver, all methods should use pointer receivers. This keeps the method set consistent and avoids subtle bugs with interface satisfaction.

Do: Consistent receivers
func (c *Circle) Area() float64 { ... }
func (c *Circle) Scale(f float64) { ... }
func (c *Circle) String() string { ... }
Don't: Mixed receivers
func (c Circle) Area() float64 { ... }
func (c *Circle) Scale(f float64) { ... }
// Confusing method sets!

Struct Embedding (Composition)

package main

import "fmt"

type Logger struct {
    Prefix string
}

func (l Logger) Log(msg string) {
    fmt.Printf("[%s] %s\n", l.Prefix, msg)
}

type Server struct {
    Logger  // Embedded — promotes Logger's methods
    Host string
    Port int
}

func main() {
    s := Server{
        Logger: Logger{Prefix: "SERVER"},
        Host:   "localhost",
        Port:   8080,
    }

    // Log() is promoted from Logger — can call directly
    s.Log("Starting up")
    s.Log(fmt.Sprintf("Listening on %s:%d", s.Host, s.Port))

    // Can also access the embedded type explicitly
    s.Logger.Log("Explicit access")
}
Output
[SERVER] Starting up [SERVER] Listening on localhost:8080 [SERVER] Explicit access

Embedding is composition, not inheritance. The outer type has a Logger, not is a Logger. The embedded type's methods are promoted to the outer type for convenience, but there is no "super" or "parent" concept.

Struct Tags

package main

import (
    "encoding/json"
    "fmt"
)

type User struct {
    Name     string `json:"name"`
    Email    string `json:"email,omitempty"`
    Password string `json:"-"` // never serialize
    Age      int    `json:"age,omitempty"`
}

func main() {
    u := User{Name: "Alice", Email: "alice@example.com", Password: "secret", Age: 0}

    data, _ := json.MarshalIndent(u, "", "  ")
    fmt.Println(string(data))
    // Note: Password excluded (-), Age excluded (omitempty + zero value)
}
Output
{ "name": "Alice", "email": "alice@example.com" }

Key Rules

  • Use named fields in struct literals: Point{X: 1, Y: 2} not Point{1, 2}
  • Use pointer receivers when: method modifies state, struct is large, or for consistency when any method needs a pointer
  • Use value receivers when: method is read-only and struct is small (a few fields)
  • Keep receivers consistent: if one method uses pointer receiver, all should
  • Embedding is composition: use it to share behavior, not to model "is-a" relationships
  • Struct tags control serialization — use json:"-" to exclude sensitive fields
Connection: Structs and methods are the foundation for implementing Interfaces. The pointer vs value receiver distinction is directly tied to Pointers. Struct embedding provides a form of polymorphism that pairs with Interface composition.

Pointers

Go Foundations • New

Pointers hold the memory address of a value. Go's pointers are simpler than C/C++ pointers — there is no pointer arithmetic. Pointers let you share data efficiently, modify values in-place, and signal optional values with nil.

Pointer Basics

package main

import "fmt"

func main() {
    x := 42
    p := &x          // p is a *int, pointing to x
    fmt.Println("x:", x)
    fmt.Println("p:", p)     // memory address
    fmt.Println("*p:", *p)   // dereference: value at address

    *p = 100                 // modify x through the pointer
    fmt.Println("x after *p = 100:", x)

    // new() allocates and returns a pointer
    q := new(int)            // *int, points to 0 (zero value)
    fmt.Println("*q:", *q)
    *q = 7
    fmt.Println("*q after assignment:", *q)
}
Output
x: 42 p: 0xc0000b6010 *p: 42 x after *p = 100: 100 *q: 0 *q after assignment: 7

Pointers for Modifying Values

package main

import "fmt"

// Without pointer — changes are lost
func incrementBroken(n int) {
    n++
}

// With pointer — changes the original
func increment(n *int) {
    *n++
}

// Classic swap function
func swap(a, b *int) {
    *a, *b = *b, *a
}

func main() {
    x := 10
    incrementBroken(x)
    fmt.Println("After incrementBroken:", x) // Still 10

    increment(&x)
    fmt.Println("After increment:", x) // 11

    a, b := 3, 7
    swap(&a, &b)
    fmt.Println("After swap:", a, b) // 7 3
}
Output
After incrementBroken: 10 After increment: 11 After swap: 7 3

Pointer to Struct

package main

import "fmt"

type Config struct {
    Debug   bool
    Verbose bool
    MaxConn int
}

// Returns a pointer — common "constructor" pattern
func NewConfig() *Config {
    return &Config{
        Debug:   false,
        Verbose: false,
        MaxConn: 100,
    }
}

func enableDebug(c *Config) {
    c.Debug = true   // Go auto-dereferences: c.Debug == (*c).Debug
    c.Verbose = true
}

func main() {
    cfg := NewConfig()
    fmt.Println("Before:", *cfg)

    enableDebug(cfg)
    fmt.Println("After:", *cfg)
}
Output
Before: {false false 100} After: {true true 100}

When to Use (and Not Use) Pointers

Use pointers when:

  • You need to modify the original value (mutation)
  • The struct is large and copying would be expensive
  • You want to express "optional" / "nullable" with nil
  • You want to share a value between goroutines (with proper synchronization)

Do NOT use pointers for:

  • Small structs (a few fields) — copying is cheap and avoids aliasing bugs
  • Maps, slices, channels — they already contain internal pointers
  • Strings and interfaces — they are small headers, cheap to copy

nil Pointer Safety

package main

import "fmt"

type Node struct {
    Value int
    Next  *Node
}

func (n *Node) String() string {
    if n == nil {
        return "<nil>"
    }
    return fmt.Sprintf("Node(%d)", n.Value)
}

func main() {
    var n *Node // nil pointer
    fmt.Println(n.String()) // Safe: method checks for nil

    // fmt.Println(n.Value) // PANIC: nil pointer dereference!
}
Output
<nil>
nil pointer dereference: Accessing a field or calling a method on a nil pointer panics at runtime. Always check for nil before dereferencing, or design your methods to handle nil receivers gracefully.

Pointer to Loop Variable Gotcha

package main

import "fmt"

func main() {
    // GOTCHA: In Go < 1.22, this captures the same variable
    // In Go >= 1.22, loop variables are per-iteration (fixed!)
    nums := []int{1, 2, 3}
    ptrs := make([]*int, len(nums))

    // Safe approach (works in all Go versions)
    for i, v := range nums {
        v := v // shadow with a new variable
        ptrs[i] = &v
    }

    for _, p := range ptrs {
        fmt.Println(*p)
    }
}
Output
1 2 3

Go 1.22+ fixed this by making loop variables per-iteration. But the shadowing idiom (v := v) is still good practice for clarity and backwards compatibility.

Common Mistakes

Do: Check for nil before dereferencing
if p != nil {
    fmt.Println(*p)
}
Don't: Blindly dereference
fmt.Println(*p) // panic if p is nil!
Do: Use value types for small data
type Point struct { X, Y int }
func dist(a, b Point) float64 { ... }
Don't: Pointer for everything
// Unnecessary — Point is tiny
func dist(a, b *Point) float64 { ... }

Key Rules

  • & takes the address of a variable; * dereferences a pointer
  • Go has no pointer arithmetic — pointers are safe by design
  • Use pointers for mutation and large structs; use values for small, immutable data
  • Maps, slices, and channels already contain pointers — no need to use *map or *[]T
  • Always check for nil before dereferencing
  • Return *T from constructors is idiomatic: func NewFoo() *Foo
Connection: Pointers underpin Structs & Methods (pointer receivers), Slices (slice header contains a pointer), and are essential for building data structures like linked lists and trees. When we cover Interfaces, the pointer vs value distinction determines which methods are in a type's method set.

Error Handling

Go Foundations • New

Go does not have exceptions. Instead, functions that can fail return an error value as their last return. This explicit error handling is a core design philosophy — errors are values, not control flow. You check them, wrap them with context, and pass them up the call stack.

The error Interface

package main

import (
    "errors"
    "fmt"
)

func main() {
    // The error interface is simple:
    // type error interface { Error() string }

    // Creating errors
    err1 := errors.New("something went wrong")
    fmt.Println("err1:", err1)

    err2 := fmt.Errorf("failed to process item %d: %s", 42, "bad format")
    fmt.Println("err2:", err2)
}
Output
err1: something went wrong err2: failed to process item 42: bad format

Basic Error Handling Pattern

package main

import (
    "fmt"
    "strconv"
)

func parseAge(s string) (int, error) {
    age, err := strconv.Atoi(s)
    if err != nil {
        return 0, fmt.Errorf("parseAge: invalid input %q: %w", s, err)
    }
    if age < 0 || age > 150 {
        return 0, fmt.Errorf("parseAge: age %d out of range [0, 150]", age)
    }
    return age, nil
}

func main() {
    // Happy path
    age, err := parseAge("25")
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    fmt.Println("Age:", age)

    // Error path
    age, err = parseAge("abc")
    if err != nil {
        fmt.Println("Error:", err)
    }

    // Out of range
    age, err = parseAge("200")
    if err != nil {
        fmt.Println("Error:", err)
    }
}
Output
Age: 25 Error: parseAge: invalid input "abc": strconv.Atoi: parsing "abc": invalid syntax Error: parseAge: age 200 out of range [0, 150]

Error Wrapping with %w

The %w verb in fmt.Errorf wraps an error, preserving the error chain. This lets callers inspect the original error using errors.Is and errors.As.

package main

import (
    "errors"
    "fmt"
)

var ErrNotFound = errors.New("not found")

func findUser(id int) (string, error) {
    if id != 1 {
        return "", fmt.Errorf("findUser(%d): %w", id, ErrNotFound)
    }
    return "Alice", nil
}

func getProfile(id int) (string, error) {
    name, err := findUser(id)
    if err != nil {
        return "", fmt.Errorf("getProfile: %w", err)
    }
    return fmt.Sprintf("Profile: %s", name), nil
}

func main() {
    // Success
    profile, err := getProfile(1)
    if err != nil {
        fmt.Println("Error:", err)
    } else {
        fmt.Println(profile)
    }

    // Error — check the chain with errors.Is
    _, err = getProfile(99)
    fmt.Println("Error:", err)
    fmt.Println("Is ErrNotFound?", errors.Is(err, ErrNotFound))
}
Output
Profile: Alice Error: getProfile: findUser(99): not found Is ErrNotFound? true
%w vs %v: Using %v instead of %w converts the error to a string, breaking the error chain. After wrapping with %v, errors.Is will return false. Always use %w when you want callers to be able to inspect the original error.

Custom Error Types

package main

import (
    "errors"
    "fmt"
)

type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed on %q: %s", e.Field, e.Message)
}

func validateEmail(email string) error {
    if len(email) == 0 {
        return &ValidationError{Field: "email", Message: "cannot be empty"}
    }
    return nil
}

func main() {
    err := validateEmail("")
    fmt.Println("Error:", err)

    // Use errors.As to extract the custom type through the chain
    var ve *ValidationError
    if errors.As(err, &ve) {
        fmt.Println("Field:", ve.Field)
        fmt.Println("Message:", ve.Message)
    }
}
Output
Error: validation failed on "email": cannot be empty Field: email Message: cannot be empty

Sentinel Errors & errors.Is vs ==

Do: Use errors.Is (chain-aware)
if errors.Is(err, ErrNotFound) {
    // handles wrapped errors too
}
Don't: Use == (breaks on wrapping)
if err == ErrNotFound {
    // fails if error was wrapped!
}

Error Handling Best Practices

Do: Handle every error
val, err := doSomething()
if err != nil {
    return fmt.Errorf("context: %w", err)
}
// use val
Don't: Ignore errors
val, _ := doSomething()
// if doSomething failed, val is zero
// and you have no idea why

Key Rules

  • Check errors immediately: if err != nil { return ..., err }
  • Wrap errors with context: fmt.Errorf("doing X: %w", err)
  • Use %w (not %v) to preserve the error chain
  • Use errors.Is(err, target) to check for sentinel errors through the chain
  • Use errors.As(err, &target) to extract custom error types through the chain
  • Define sentinel errors as package-level variables: var ErrNotFound = errors.New("not found")
  • Never ignore returned errors in production code
Connection: The error type is itself an Interface. Custom error types implement this interface. Error wrapping chains are a practical application of Go's interface and type assertion mechanisms. You will see error handling in every Go program you write.

Interfaces

Go Foundations • New

Interfaces are Go's mechanism for polymorphism. Unlike Java or C#, Go interfaces are satisfied implicitly — if a type has all the methods an interface requires, it automatically implements that interface. No implements keyword needed. This makes Go interfaces incredibly flexible and enables loose coupling.

Implicit Interface Satisfaction

package main

import (
    "fmt"
    "math"
)

type Shape interface {
    Area() float64
    Perimeter() float64
}

type Circle struct {
    Radius float64
}

func (c Circle) Area() float64 {
    return math.Pi * c.Radius * c.Radius
}

func (c Circle) Perimeter() float64 {
    return 2 * math.Pi * c.Radius
}

type Rectangle struct {
    Width, Height float64
}

func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

func (r Rectangle) Perimeter() float64 {
    return 2 * (r.Width + r.Height)
}

// Works with ANY Shape — polymorphism!
func printShapeInfo(s Shape) {
    fmt.Printf("Area: %.2f, Perimeter: %.2f\n", s.Area(), s.Perimeter())
}

func main() {
    c := Circle{Radius: 5}
    r := Rectangle{Width: 3, Height: 4}

    printShapeInfo(c)
    printShapeInfo(r)

    // Slice of interfaces
    shapes := []Shape{c, r, Circle{Radius: 1}}
    for _, s := range shapes {
        printShapeInfo(s)
    }
}
Output
Area: 78.54, Perimeter: 31.42 Area: 12.00, Perimeter: 14.00 Area: 78.54, Perimeter: 31.42 Area: 12.00, Perimeter: 14.00 Area: 3.14, Perimeter: 6.28

The Empty Interface & any

package main

import "fmt"

// interface{} (or "any" since Go 1.18) accepts ANY value
func describe(i interface{}) {
    fmt.Printf("Value: %v, Type: %T\n", i, i)
}

func main() {
    describe(42)
    describe("hello")
    describe(true)
    describe([]int{1, 2, 3})
    describe(nil)
}
Output
Value: 42, Type: int Value: hello, Type: string Value: true, Type: bool Value: [1 2 3], Type: []int Value: <nil>, Type: <nil>

Type Assertions & Type Switches

package main

import "fmt"

func process(i interface{}) {
    // Type assertion with comma-ok (safe)
    if s, ok := i.(string); ok {
        fmt.Println("It's a string:", s)
        return
    }

    // Type switch — the idiomatic way
    switch v := i.(type) {
    case int:
        fmt.Println("It's an int:", v*2)
    case float64:
        fmt.Println("It's a float64:", v)
    case string:
        fmt.Println("It's a string:", v)
    case nil:
        fmt.Println("It's nil")
    default:
        fmt.Printf("Unknown type: %T\n", v)
    }
}

func main() {
    process("hello")
    process(42)
    process(3.14)
    process(nil)
    process([]int{1})
}
Output
It's a string: hello It's an int: 84 It's a float64: 3.14 It's nil Unknown type: []int

The Stringer Interface

package main

import "fmt"

// fmt.Stringer interface:
// type Stringer interface { String() string }

type Color struct {
    R, G, B uint8
}

func (c Color) String() string {
    return fmt.Sprintf("#%02x%02x%02x", c.R, c.G, c.B)
}

func main() {
    red := Color{255, 0, 0}
    blue := Color{0, 0, 255}

    // fmt.Println automatically calls String() if available
    fmt.Println("Red:", red)
    fmt.Println("Blue:", blue)
    fmt.Printf("Custom color: %s\n", Color{128, 64, 200})
}
Output
Red: #ff0000 Blue: #0000ff Custom color: #8040c8

Interface Composition

package main

import (
    "fmt"
    "io"
    "strings"
)

// Small, focused interfaces
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

// Composed interface (this is how io.ReadWriter is defined)
type ReadWriter interface {
    Reader
    Writer
}

func main() {
    // strings.NewReader implements io.Reader
    r := strings.NewReader("Hello, Go interfaces!")
    buf := make([]byte, 5)

    for {
        n, err := r.Read(buf)
        if n > 0 {
            fmt.Print(string(buf[:n]))
        }
        if err == io.EOF {
            break
        }
    }
    fmt.Println()
}
Output
Hello, Go interfaces!

The nil Interface Pitfall

This is one of Go's most subtle traps. An interface value is nil only if both its type and value are nil:

package main

import "fmt"

type MyError struct {
    Code int
}

func (e *MyError) Error() string {
    return fmt.Sprintf("error code %d", e.Code)
}

func doSomething(fail bool) error {
    var err *MyError // nil *MyError

    if fail {
        err = &MyError{Code: 500}
    }

    return err // TRAP: returns non-nil interface even when err is nil!
}

func main() {
    err := doSomething(false)
    if err != nil {
        fmt.Println("Error is NOT nil:", err) // This prints!
        // The interface holds type=*MyError, value=nil
        // Since the type is set, the interface is non-nil
    }

    // The correct pattern:
    fmt.Println("\n--- Correct pattern ---")
    err2 := doSomethingCorrect(false)
    if err2 != nil {
        fmt.Println("Error:", err2)
    } else {
        fmt.Println("No error (correct!)")
    }
}

func doSomethingCorrect(fail bool) error {
    if fail {
        return &MyError{Code: 500}
    }
    return nil // Return nil directly, not a typed nil pointer
}
Output
Error is NOT nil: error code 0 --- Correct pattern --- No error (correct!)
nil interface vs interface holding nil: An interface has two components: a type and a value. var err error = (*MyError)(nil) is a non-nil interface (type is set to *MyError, value is nil). Only var err error = nil is a truly nil interface. Always return nil directly from error-returning functions instead of a typed nil pointer.

Accept Interfaces, Return Structs

A Go proverb: functions should accept interfaces (for flexibility) and return concrete types (for clarity). This makes your APIs composable without over-abstracting.

Do: Accept interface
func Process(r io.Reader) error {
    // works with files, network, strings...
}
Don't: Accept concrete type
func Process(f *os.File) error {
    // only works with files!
}

Common Mistakes

Do: Keep interfaces small
type Sizer interface {
    Size() int64
}
// 1-2 methods is ideal
Don't: Giant interfaces
type DoEverything interface {
    Read() ...
    Write() ...
    Close() ...
    Flush() ...
    // 10 more methods...
}
Do: Return nil for no error
func f() error {
    return nil // truly nil interface
}
Don't: Return typed nil pointer
func f() error {
    var err *MyError
    return err // non-nil interface!
}

Key Rules

  • Interfaces are satisfied implicitly — no implements keyword
  • Keep interfaces small: 1-3 methods is ideal (the Go standard library averages 1-2)
  • Accept interfaces, return concrete types
  • Use type assertions with the comma-ok pattern for safety
  • Use type switches for handling multiple possible types
  • Beware the nil interface pitfall: always return nil directly for error returns
  • Compose interfaces by embedding smaller ones
Connection: Interfaces tie together everything in this guide. Error Handling is built on the error interface. Structs & Methods provide the implementations. Pointers determine the method set (value vs pointer receiver affects which interfaces a type satisfies). The io.Reader and io.Writer interfaces you learned here are used throughout Go's standard library for Strings processing, file I/O, and network programming.
