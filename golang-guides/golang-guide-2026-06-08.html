sync Package

Concurrency • overdue 3 days • rep 0

Quick Reminder

sync.Mutex protects shared state with Lock()/Unlock(). sync.RWMutex allows multiple concurrent readers. sync.WaitGroup waits for a collection of goroutines. sync.Once ensures a function runs exactly once. sync.Map is a concurrent-safe map for specific use cases (many reads, few writes).

Key Pattern

var mu sync.Mutex

func increment(counter *int) {
    mu.Lock()
    defer mu.Unlock()
    *counter++
}
Challenge Exercise

Implement a thread-safe counter struct using sync.Mutex. It should have Inc(), Dec(), and Value() int methods. Write a benchmark that increments the counter from 100 goroutines and verify the final count is correct.

#1 Pitfall: Copying a Mutex (pass by value) — always use a pointer receiver or embed the mutex in a struct. The go vet tool detects this.

Context

Concurrency • overdue 3 days • rep 0

Quick Reminder

context.Background() is the root context. WithCancel, WithTimeout, and WithDeadline derive child contexts. WithValue attaches request-scoped data. Always pass context as the first parameter (ctx context.Context). Check ctx.Err() or <-ctx.Done() for cancellation.

Key Pattern

ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
defer cancel()

select {
case result := <-doWork(ctx):
    fmt.Println(result)
case <-ctx.Done():
    fmt.Println("timeout:", ctx.Err())
}
Challenge Exercise

Write a function that makes 3 HTTP requests concurrently using goroutines. Use a context with cancellation — when the first successful response arrives, cancel the remaining requests. Return the fastest result.

#1 Pitfall: Forgetting to call cancel() — leaks goroutines and resources. Always defer cancel() immediately after creating a cancellable context.

First-class Functions

Functions & Control Flow • overdue 2 days • rep 1

Quick Reminder

Functions in Go are first-class values: you can assign them to variables, pass them as arguments, and return them from other functions. Closures capture variables from their enclosing scope. Function types like type Predicate func(int) bool make signatures reusable.

Key Pattern

type Predicate func(int) bool

func Filter(nums []int, pred Predicate) []int {
    var result []int
    for _, n := range nums {
        if pred(n) {
            result = append(result, n)
        }
    }
    return result
}

evens := Filter([]int{1, 2, 3, 4}, func(n int) bool {
    return n%2 == 0
})
Challenge Exercise

Implement a Compose function that takes two functions f and g (both func(int) int) and returns a new function that computes f(g(x)). Test it by composing double and addOne.

#1 Pitfall: Goroutine closure capturing loop variable by reference — in Go <1.22, the loop variable is shared across iterations. Use Go 1.22+ (which fixes this) or rebind: v := v.

Defer, Panic, Recover

Functions & Control Flow • overdue 2 days • rep 1

Quick Reminder

Deferred calls execute in LIFO order when the surrounding function returns. Named return values can be modified in deferred functions. recover() only works inside a deferred function and catches panics. Panics unwind the stack, running deferred functions at each level.

Key Pattern

func safeExecute(fn func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r)
        }
    }()
    fn()
    return nil
}
Challenge Exercise

Write a safeExecute(fn func()) error wrapper that recovers from panics and returns the panic value as an error. Test it with a function that panics and one that does not.

#1 Pitfall: Defer evaluates arguments immediately — defer fmt.Println(x) captures x's current value at the defer statement, not the value at function exit. Use a closure to capture the final value.

Concurrency Patterns

Concurrency • overdue 2 days • rep 0

Quick Reminder

Fan-out launches multiple goroutines to process work in parallel. Fan-in collects results from multiple channels into one. Pipelines chain stages connected by channels, each stage a goroutine. Worker pools use a fixed number of goroutines pulling from a shared job channel.

Key Pattern

// Fan-out / Fan-in
func fanOut(in <-chan int, workers int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for v := range in {
                out <- v * v
            }
        }()
    }
    go func() { wg.Wait(); close(out) }()
    return out
}
Challenge Exercise

Implement a 3-stage pipeline: stage 1 generates numbers 1-20, stage 2 squares them, stage 3 filters to keep only odd results. Connect stages with channels and print the final output.

#1 Pitfall: Not closing channels when producers are done — receivers using range will block forever waiting for more values. Always close the channel from the sender side.

errgroup

Concurrency • overdue 2 days • rep 0

Quick Reminder

golang.org/x/sync/errgroup manages a group of goroutines with error propagation. g.Go(func() error { ... }) launches work. g.Wait() returns the first non-nil error. errgroup.WithContext(ctx) returns a derived context that is cancelled when any goroutine fails.

Key Pattern

g, ctx := errgroup.WithContext(ctx)

for _, url := range urls {
    url := url // rebind for closure
    g.Go(func() error {
        req, err := http.NewRequestWithContext(ctx, "GET", url, nil)
        if err != nil {
            return err
        }
        resp, err := http.DefaultClient.Do(req)
        if err != nil {
            return err
        }
        defer resp.Body.Close()
        return nil
    })
}

if err := g.Wait(); err != nil {
    log.Fatal(err)
}
Challenge Exercise

Use errgroup to fetch 5 URLs concurrently with a shared context. If any request fails, the rest should be cancelled. Collect successful response sizes and return the total bytes, or the first error.

#1 Pitfall: Not using the derived context from errgroup.WithContext — if you pass the original context to HTTP requests, cancellation does not propagate when one goroutine fails.

httptest — Testing HTTP Handlers

Testing & Tooling • New

Why It Matters

The net/http/httptest package lets you test HTTP handlers without starting a real server, making tests fast and deterministic. Use httptest.NewRecorder for unit tests and httptest.NewServer for integration tests.

Core Concepts

  • httptest.NewRecorder() — captures response status, headers, and body without any network I/O
  • httptest.NewRequest(method, target, body) — creates an *http.Request suitable for testing
  • httptest.NewServer(handler) — starts a local HTTP server on a random port for integration tests
  • Table-driven HTTP tests — test multiple methods, paths, and bodies with a shared test structure
  • Testing middleware chains — wrap handlers and assert side effects

Simple Handler Test with NewRecorder

package main

import (
    "encoding/json"
    "net/http"
    "net/http/httptest"
    "testing"
)

func HealthHandler(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "ok"})
}

func TestHealthHandler(t *testing.T) {
    req := httptest.NewRequest("GET", "/health", nil)
    w := httptest.NewRecorder()

    HealthHandler(w, req)

    resp := w.Result()
    if resp.StatusCode != http.StatusOK {
        t.Errorf("got status %d, want %d", resp.StatusCode, http.StatusOK)
    }

    var body map[string]string
    json.NewDecoder(resp.Body).Decode(&body)
    if body["status"] != "ok" {
        t.Errorf("got status %q, want %q", body["status"], "ok")
    }
}

Testing JSON API Endpoints

package main

import (
    "bytes"
    "encoding/json"
    "net/http"
    "net/http/httptest"
    "testing"
)

type User struct {
    Name  string `json:"name"`
    Email string `json:"email"`
}

func CreateUserHandler(w http.ResponseWriter, r *http.Request) {
    if r.Method != http.MethodPost {
        http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
        return
    }
    var u User
    if err := json.NewDecoder(r.Body).Decode(&u); err != nil {
        http.Error(w, "bad request", http.StatusBadRequest)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusCreated)
    json.NewEncoder(w).Encode(u)
}

func TestCreateUserHandler(t *testing.T) {
    body := bytes.NewBufferString(`{"name":"Alice","email":"alice@example.com"}`)
    req := httptest.NewRequest("POST", "/users", body)
    req.Header.Set("Content-Type", "application/json")
    w := httptest.NewRecorder()

    CreateUserHandler(w, req)

    resp := w.Result()
    if resp.StatusCode != http.StatusCreated {
        t.Fatalf("got status %d, want %d", resp.StatusCode, http.StatusCreated)
    }

    var u User
    json.NewDecoder(resp.Body).Decode(&u)
    if u.Name != "Alice" {
        t.Errorf("got name %q, want %q", u.Name, "Alice")
    }
}

NewServer for Integration Tests

package main

import (
    "io"
    "net/http"
    "net/http/httptest"
    "testing"
)

func TestWithServer(t *testing.T) {
    handler := http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        w.WriteHeader(http.StatusOK)
        w.Write([]byte("hello from test server"))
    })

    ts := httptest.NewServer(handler)
    defer ts.Close() // Always close!

    resp, err := http.Get(ts.URL)
    if err != nil {
        t.Fatalf("unexpected error: %v", err)
    }
    defer resp.Body.Close()

    body, _ := io.ReadAll(resp.Body)
    if string(body) != "hello from test server" {
        t.Errorf("got %q, want %q", string(body), "hello from test server")
    }
}

Table-Driven HTTP Test Pattern

package main

import (
    "net/http"
    "net/http/httptest"
    "testing"
)

func StatusHandler(w http.ResponseWriter, r *http.Request) {
    switch r.URL.Path {
    case "/ok":
        w.WriteHeader(http.StatusOK)
    case "/not-found":
        http.NotFound(w, r)
    case "/error":
        http.Error(w, "internal error", http.StatusInternalServerError)
    default:
        http.NotFound(w, r)
    }
}

func TestStatusHandler(t *testing.T) {
    tests := []struct {
        name       string
        method     string
        path       string
        wantStatus int
    }{
        {"ok path", "GET", "/ok", http.StatusOK},
        {"not found", "GET", "/not-found", http.StatusNotFound},
        {"server error", "GET", "/error", http.StatusInternalServerError},
        {"unknown path", "GET", "/unknown", http.StatusNotFound},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            req := httptest.NewRequest(tt.method, tt.path, nil)
            w := httptest.NewRecorder()
            StatusHandler(w, req)

            if w.Code != tt.wantStatus {
                t.Errorf("got status %d, want %d", w.Code, tt.wantStatus)
            }
        })
    }
}
Output (go test -v -run TestStatusHandler)

Testing with Request Context

package main

import (
    "context"
    "fmt"
    "net/http"
    "net/http/httptest"
    "testing"
)

type contextKey string

func UserHandler(w http.ResponseWriter, r *http.Request) {
    userID := r.Context().Value(contextKey("userID"))
    if userID == nil {
        http.Error(w, "unauthorized", http.StatusUnauthorized)
        return
    }
    fmt.Fprintf(w, "Hello, user %s", userID)
}

func TestUserHandlerWithContext(t *testing.T) {
    req := httptest.NewRequest("GET", "/user", nil)
    ctx := context.WithValue(req.Context(), contextKey("userID"), "42")
    req = req.WithContext(ctx)

    w := httptest.NewRecorder()
    UserHandler(w, req)

    if w.Code != http.StatusOK {
        t.Errorf("got status %d, want %d", w.Code, http.StatusOK)
    }
    if w.Body.String() != "Hello, user 42" {
        t.Errorf("got body %q, want %q", w.Body.String(), "Hello, user 42")
    }
}

Common Mistakes

Do: Close test server
ts := httptest.NewServer(handler)
defer ts.Close()
Don't: Leak test servers
ts := httptest.NewServer(handler)
// Forgot ts.Close()!
// Port leaks, tests may fail
Do: Read and close response body
resp, _ := http.Get(ts.URL)
defer resp.Body.Close()
body, _ := io.ReadAll(resp.Body)
Don't: Ignore response body
resp, _ := http.Get(ts.URL)
// Never read/closed body
// Leaks connections!

Key Rules

  • Use NewRecorder for fast unit tests of individual handlers
  • Use NewServer for integration tests that need real HTTP connections
  • Always test both success and error response paths
  • Always close test servers with defer ts.Close()
  • Always read and close response bodies in client-side tests
Connection: httptest builds directly on net/http handlers and testing Package patterns. Table-driven HTTP tests combine the table-driven technique from Day 5 with HTTP-specific assertions. NewServer pairs naturally with Context for timeout-aware integration tests.

Mocking: Interfaces for Testability

Testing & Tooling • New

Why It Matters

Go's implicit interfaces enable clean mocking without frameworks. Define a small interface, implement it for production, and swap in a mock for tests. This follows the Go proverb: “Accept interfaces, return structs.”

Core Concepts

  • Accept interfaces, return structs — functions take interface parameters but return concrete types
  • Consumer-site interfaces — define interfaces where they are used, not where they are implemented
  • Manual mock implementation — a struct that satisfies the interface and records calls
  • //go:generate mockgen — auto-generate mocks with gomock/mockgen tooling
  • Interface segregation — keep interfaces small: io.Reader (1 method) vs io.ReadWriteCloser (3 methods)

Service with Interface and Mock

package main

import "fmt"

// UserStore is defined where it's CONSUMED, not where it's implemented.
type UserStore interface {
    GetByID(id int) (string, error)
}

// UserService depends on the interface, not a concrete type.
type UserService struct {
    store UserStore
}

func NewUserService(store UserStore) *UserService {
    return &UserService{store: store}
}

func (s *UserService) Greet(id int) (string, error) {
    name, err := s.store.GetByID(id)
    if err != nil {
        return "", fmt.Errorf("greeting user %d: %w", id, err)
    }
    return fmt.Sprintf("Hello, %s!", name), nil
}

func main() {
    // In production, you'd pass a real database store.
    // In tests, you pass a mock.
    fmt.Println("UserService accepts any UserStore implementation")
}
Output
UserService accepts any UserStore implementation

Testing with a Mock that Tracks Calls

package main

import (
    "errors"
    "testing"
)

// MockUserStore implements UserStore for testing.
type MockUserStore struct {
    GetByIDFunc func(id int) (string, error)
    Calls       []int // tracks which IDs were requested
}

func (m *MockUserStore) GetByID(id int) (string, error) {
    m.Calls = append(m.Calls, id)
    return m.GetByIDFunc(id)
}

func TestUserService_Greet(t *testing.T) {
    tests := []struct {
        name      string
        id        int
        mockFunc  func(int) (string, error)
        want      string
        wantErr   bool
    }{
        {
            name: "success",
            id:   1,
            mockFunc: func(id int) (string, error) {
                return "Alice", nil
            },
            want: "Hello, Alice!",
        },
        {
            name: "user not found",
            id:   999,
            mockFunc: func(id int) (string, error) {
                return "", errors.New("not found")
            },
            wantErr: true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            mock := &MockUserStore{GetByIDFunc: tt.mockFunc}
            svc := NewUserService(mock)

            got, err := svc.Greet(tt.id)
            if (err != nil) != tt.wantErr {
                t.Fatalf("Greet() error = %v, wantErr %v", err, tt.wantErr)
            }
            if got != tt.want {
                t.Errorf("Greet() = %q, want %q", got, tt.want)
            }
            if len(mock.Calls) != 1 || mock.Calls[0] != tt.id {
                t.Errorf("expected GetByID(%d) to be called once, got calls: %v", tt.id, mock.Calls)
            }
        })
    }
}

Interface at Consumer vs Provider

// BAD: interface defined in the "database" package (provider)
// package database
// type Store interface { ... }  // Too broad, hard to mock

// GOOD: interface defined where it's used (consumer)
// package userservice
type UserStore interface {
    GetByID(id int) (string, error)
}

// The consumer only needs what it uses.
// The database package returns a concrete struct that happens to satisfy this.

Table-Driven Tests with Different Mock Behaviors

package main

import (
    "errors"
    "testing"
    "time"
)

type Cache interface {
    Get(key string) (string, bool)
    Set(key string, value string, ttl time.Duration)
}

type MockCache struct {
    data map[string]string
}

func (m *MockCache) Get(key string) (string, bool) {
    v, ok := m.data[key]
    return v, ok
}

func (m *MockCache) Set(key, value string, ttl time.Duration) {
    m.data[key] = value
}

type CachedService struct {
    store UserStore
    cache Cache
}

func (s *CachedService) GetUser(id int) (string, error) {
    key := fmt.Sprintf("user:%d", id)
    if v, ok := s.cache.Get(key); ok {
        return v, nil // cache hit
    }
    name, err := s.store.GetByID(id)
    if err != nil {
        return "", err
    }
    s.cache.Set(key, name, 5*time.Minute)
    return name, nil
}

func TestCachedService_GetUser(t *testing.T) {
    tests := []struct {
        name      string
        cacheData map[string]string
        storeFunc func(int) (string, error)
        want      string
        wantErr   bool
    }{
        {
            name:      "cache hit",
            cacheData: map[string]string{"user:1": "Alice"},
            storeFunc: func(id int) (string, error) {
                t.Fatal("store should not be called on cache hit")
                return "", nil
            },
            want: "Alice",
        },
        {
            name:      "cache miss, store success",
            cacheData: map[string]string{},
            storeFunc: func(id int) (string, error) { return "Bob", nil },
            want:      "Bob",
        },
        {
            name:      "cache miss, store error",
            cacheData: map[string]string{},
            storeFunc: func(id int) (string, error) {
                return "", errors.New("db down")
            },
            wantErr: true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            svc := &CachedService{
                store: &MockUserStore{GetByIDFunc: tt.storeFunc},
                cache: &MockCache{data: tt.cacheData},
            }
            got, err := svc.GetUser(1)
            if (err != nil) != tt.wantErr {
                t.Fatalf("error = %v, wantErr %v", err, tt.wantErr)
            }
            if got != tt.want {
                t.Errorf("got %q, want %q", got, tt.want)
            }
        })
    }
}

Common Mistakes

Do: Small, focused interfaces
type Reader interface {
    Read(p []byte) (n int, err error)
}

// 1 method = easy to mock
Don't: God interfaces
type Database interface {
    GetUser(id int) (User, error)
    CreateUser(u User) error
    DeleteUser(id int) error
    ListUsers() ([]User, error)
    GetOrder(id int) (Order, error)
    // 15 more methods...
}
Do: Define at consumer
// In userservice package:
type UserStore interface {
    GetByID(id int) (string, error)
}
Don't: Define at provider
// In database package:
type Store interface {
    // Every consumer forced to depend
    // on the full interface
}

Key Rules

  • Keep interfaces small — 1 to 3 methods is ideal
  • Define interfaces where they are used, not where they are implemented
  • Prefer real implementations in tests when practical — only mock external dependencies (databases, APIs, clocks)
  • Use function fields in mock structs for flexible per-test behavior
  • Track mock calls to verify interaction patterns
Connection: Mocking with interfaces builds on Go's Interfaces concept (implicit satisfaction). It pairs with httptest — you can mock an HTTP client by defining a Doer interface. The Functional Options pattern (later in this guide) often uses interfaces for pluggable dependencies.

go vet & staticcheck

Testing & Tooling • New

Why It Matters

Static analysis catches bugs that tests might miss — unreachable code, incorrect printf verbs, unused results, mutex copies, and race conditions. Running these tools before committing prevents entire categories of bugs from reaching production.

Core Concepts

  • go vet — built-in tool that checks for suspicious constructs the compiler does not catch
  • Common go vet checks: printf format mismatches, unreachable code, mutex copy, invalid struct tags
  • staticcheck — more comprehensive analysis: deprecated APIs, unnecessary conversions, style issues
  • golangci-lint — meta-linter that orchestrates multiple linters in a single pass
  • .golangci.yml — configuration file for enabling/disabling specific linters and rules

Code That go vet Catches

package main

import (
    "fmt"
    "sync"
)

func main() {
    // BUG 1: Printf format mismatch
    name := "Alice"
    age := 30
    fmt.Printf("Name: %d, Age: %s\n", name, age)
    // go vet: fmt.Printf format %d has arg name of wrong type string

    // BUG 2: Copying a sync.Mutex
    var mu sync.Mutex
    mu2 := mu  // go vet: assignment copies lock value
    _ = mu2
}

Code That staticcheck Catches

package main

import (
    "fmt"
    "strings"
)

func main() {
    // ISSUE 1: Unnecessary type conversion
    var x int = 42
    y := int(x) // staticcheck: unnecessary conversion (SA4000)

    // ISSUE 2: Using deprecated API
    s := "hello world"
    idx := strings.Index(s, " ")
    _ = strings.Title(s) // staticcheck: SA1019: strings.Title is deprecated

    // ISSUE 3: Result of pure function not used
    strings.Replace(s, "hello", "hi", 1) // staticcheck: SA4017: result not used

    fmt.Println(y, idx)
}

Functional Options Pattern

Advanced Patterns • New

Why It Matters

The functional options pattern provides a clean, extensible API for configuring structs. Instead of constructors with many parameters (or breaking changes when adding new ones), you pass variadic option functions. This is idiomatic Go and used extensively in the standard library and popular packages.

Core Concepts

  • Problem: constructors with many parameters are hard to use, hard to read, and break when you add parameters
  • Solution: type Option func(*Config) with With* functions that modify the config
  • Default configuration with option overrides — sensible defaults, callers only specify what they need
  • Variadic options: func NewServer(opts ...Option) *Server
  • Validation in option functions — options can return errors for invalid values
  • Comparison with builder pattern: functional options are more idiomatic in Go

Basic Functional Options

package main

import (
    "fmt"
    "log"
    "os"
    "time"
)

type Server struct {
    host       string
    port       int
    timeout    time.Duration
    maxRetries int
    logger     *log.Logger
}

// Option is a function that configures a Server.
type Option func(*Server)

// WithHost sets the server host.
func WithHost(host string) Option {
    return func(s *Server) {
        s.host = host
    }
}

// WithPort sets the server port.
func WithPort(port int) Option {
    return func(s *Server) {
        s.port = port
    }
}

// WithTimeout sets the request timeout.
func WithTimeout(d time.Duration) Option {
    return func(s *Server) {
        s.timeout = d
    }
}

// WithMaxRetries sets the maximum retry count.
func WithMaxRetries(n int) Option {
    return func(s *Server) {
        s.maxRetries = n
    }
}

// WithLogger sets a custom logger.
func WithLogger(l *log.Logger) Option {
    return func(s *Server) {
        s.logger = l
    }
}

// NewServer creates a Server with sensible defaults and applies options.
func NewServer(opts ...Option) *Server {
    // Sensible defaults
    s := &Server{
        host:       "localhost",
        port:       8080,
        timeout:    30 * time.Second,
        maxRetries: 3,
        logger:     log.New(os.Stdout, "[server] ", log.LstdFlags),
    }

    // Apply each option
    for _, opt := range opts {
        opt(s)
    }

    return s
}

func main() {
    // Use defaults
    s1 := NewServer()
    fmt.Printf("Default: %s:%d timeout=%v retries=%d\n",
        s1.host, s1.port, s1.timeout, s1.maxRetries)

    // Override specific options
    s2 := NewServer(
        WithHost("0.0.0.0"),
        WithPort(9090),
        WithTimeout(10*time.Second),
    )
    fmt.Printf("Custom:  %s:%d timeout=%v retries=%d\n",
        s2.host, s2.port, s2.timeout, s2.maxRetries)
}
Output
Default: localhost:8080 timeout=30s retries=3 Custom: 0.0.0.0:9090 timeout=10s retries=3

Options with Validation

package main

import (
    "errors"
    "fmt"
)

type Config struct {
    Workers int
    QueueSize int
}

// OptionErr is an option that can fail validation.
type OptionErr func(*Config) error

func WithWorkers(n int) OptionErr {
    return func(c *Config) error {
        if n < 1 || n > 1000 {
            return fmt.Errorf("workers must be 1-1000, got %d", n)
        }
        c.Workers = n
        return nil
    }
}

func WithQueueSize(n int) OptionErr {
    return func(c *Config) error {
        if n < 0 {
            return errors.New("queue size must be non-negative")
        }
        c.QueueSize = n
        return nil
    }
}

func NewPool(opts ...OptionErr) (*Config, error) {
    c := &Config{Workers: 4, QueueSize: 100} // defaults
    for _, opt := range opts {
        if err := opt(c); err != nil {
            return nil, fmt.Errorf("invalid option: %w", err)
        }
    }
    return c, nil
}

func main() {
    // Valid options
    pool, err := NewPool(WithWorkers(8), WithQueueSize(500))
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    fmt.Printf("Pool: workers=%d queue=%d\n", pool.Workers, pool.QueueSize)

    // Invalid option
    _, err = NewPool(WithWorkers(-1))
    if err != nil {
        fmt.Println("Error:", err)
    }
}
Output
Pool: workers=8 queue=500 Error: invalid option: workers must be 1-1000, got -1

Real-World Example: HTTP Client Configuration

package main

import (
    "fmt"
    "net/http"
    "time"
)

type HTTPClient struct {
    baseURL    string
    httpClient *http.Client
    headers    map[string]string
    retries    int
}

type ClientOption func(*HTTPClient)

func WithBaseURL(url string) ClientOption {
    return func(c *HTTPClient) { c.baseURL = url }
}

func WithHTTPTimeout(d time.Duration) ClientOption {
    return func(c *HTTPClient) { c.httpClient.Timeout = d }
}

func WithHeader(key, value string) ClientOption {
    return func(c *HTTPClient) { c.headers[key] = value }
}

func WithRetries(n int) ClientOption {
    return func(c *HTTPClient) { c.retries = n }
}

func NewHTTPClient(opts ...ClientOption) *HTTPClient {
    c := &HTTPClient{
        baseURL:    "https://api.example.com",
        httpClient: &http.Client{Timeout: 30 * time.Second},
        headers:    map[string]string{"Content-Type": "application/json"},
        retries:    1,
    }
    for _, opt := range opts {
        opt(c)
    }
    return c
}

func main() {
    client := NewHTTPClient(
        WithBaseURL("https://api.myapp.com"),
        WithHTTPTimeout(5*time.Second),
        WithHeader("Authorization", "Bearer token123"),
        WithRetries(3),
    )

    fmt.Printf("Base URL: %s\n", client.baseURL)
    fmt.Printf("Timeout: %v\n", client.httpClient.Timeout)
    fmt.Printf("Headers: %v\n", client.headers)
    fmt.Printf("Retries: %d\n", client.retries)
}
Output
Base URL: https://api.myapp.com Timeout: 5s Headers: map[Authorization:Bearer token123 Content-Type:application/json] Retries: 3

Common Mistakes

Do: Use for complex configuration
// 5+ config params, API will grow
srv := NewServer(
    WithHost("0.0.0.0"),
    WithPort(9090),
    WithTimeout(10*time.Second),
)
Don't: Overengineer simple cases
// Only 2 params? Just use a constructor.
// func NewPoint(x, y int) *Point
// NOT:
p := NewPoint(WithX(1), WithY(2))
Do: Keep options independent
// Each option works regardless of order
NewServer(WithPort(9090), WithHost("0.0.0.0"))
NewServer(WithHost("0.0.0.0"), WithPort(9090))
// Same result
Don't: Options that depend on order
// BAD: WithTLS must come after WithPort
// because it reads s.port
// Options should be order-independent

Key Rules

  • Use functional options when you have 4+ configuration parameters or when the API will grow over time
  • Always provide sensible defaults — calling NewServer() with no options should work
  • Keep options independent of each other — the result should not depend on application order
  • Name option functions With* for consistency: WithTimeout, WithLogger, WithRetries
  • Use the error-returning variant (func(*Config) error) when options need validation
Connection: Functional options use First-class Functions and closures as their core mechanism. They pair naturally with Middleware Chains (next concept) — both compose behavior through function wrapping. The pattern is used in net/http server configuration and throughout the Go ecosystem.

Middleware Chains

Advanced Patterns • New

Why It Matters

HTTP middleware lets you compose cross-cutting concerns — logging, authentication, rate limiting, panic recovery — as reusable, stackable handlers. Middleware is the primary pattern for building production HTTP services in Go.

Core Concepts

  • http.Handler interface: ServeHTTP(w http.ResponseWriter, r *http.Request) — the building block
  • Middleware signature: func(http.Handler) http.Handler — takes a handler, returns a wrapped handler
  • Wrapping pattern: outer middleware calls next.ServeHTTP(w, r) to pass control to the inner handler
  • Composing multiple middleware: a Chain function that applies them in order
  • Common middleware: logging, recovery (catches panics), CORS, auth, request ID
  • http.HandlerFunc adapter: converts a function to an http.Handler

Simple Logging Middleware

package main

import (
    "fmt"
    "log"
    "net/http"
    "net/http/httptest"
    "time"
)

// Logging middleware wraps a handler and logs each request.
func Logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}

func helloHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Hello, World!")
}

func main() {
    handler := Logging(http.HandlerFunc(helloHandler))

    // Test it
    req := httptest.NewRequest("GET", "/hello", nil)
    w := httptest.NewRecorder()
    handler.ServeHTTP(w, req)

    fmt.Println("Response:", w.Body.String())
}
Output
2026/06/08 10:00:00 GET /hello 125ns Response: Hello, World!

Recovery Middleware (Catches Panics)

package main

import (
    "fmt"
    "log"
    "net/http"
    "net/http/httptest"
)

func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("PANIC recovered: %v", err)
                http.Error(w, "Internal Server Error", http.StatusInternalServerError)
            }
        }()
        next.ServeHTTP(w, r)
    })
}

func panicHandler(w http.ResponseWriter, r *http.Request) {
    panic("something went terribly wrong")
}

func main() {
    handler := Recovery(http.HandlerFunc(panicHandler))

    req := httptest.NewRequest("GET", "/panic", nil)
    w := httptest.NewRecorder()
    handler.ServeHTTP(w, req)

    fmt.Printf("Status: %d\n", w.Code)
    fmt.Printf("Body: %s", w.Body.String())
}
Output
2026/06/08 10:00:00 PANIC recovered: something went terribly wrong Status: 500 Body: Internal Server Error

Auth Middleware with Context

package main

import (
    "context"
    "fmt"
    "net/http"
    "net/http/httptest"
    "strings"
)

type ctxKey string

const userIDKey ctxKey = "userID"

func Auth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        token := r.Header.Get("Authorization")
        if !strings.HasPrefix(token, "Bearer ") {
            http.Error(w, "unauthorized", http.StatusUnauthorized)
            return // Don't call next!
        }

        // In real code, validate the token and extract user ID
        userID := strings.TrimPrefix(token, "Bearer ")
        ctx := context.WithValue(r.Context(), userIDKey, userID)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

func protectedHandler(w http.ResponseWriter, r *http.Request) {
    userID := r.Context().Value(userIDKey).(string)
    fmt.Fprintf(w, "Welcome, user %s!", userID)
}

func main() {
    handler := Auth(http.HandlerFunc(protectedHandler))

    // Request without auth
    req1 := httptest.NewRequest("GET", "/protected", nil)
    w1 := httptest.NewRecorder()
    handler.ServeHTTP(w1, req1)
    fmt.Printf("No auth: %d %s\n", w1.Code, strings.TrimSpace(w1.Body.String()))

    // Request with auth
    req2 := httptest.NewRequest("GET", "/protected", nil)
    req2.Header.Set("Authorization", "Bearer user42")
    w2 := httptest.NewRecorder()
    handler.ServeHTTP(w2, req2)
    fmt.Printf("With auth: %d %s\n", w2.Code, w2.Body.String())
}
Output
No auth: 401 unauthorized With auth: 200 Welcome, user user42!

Chain Function Composing Multiple Middleware

package main

import (
    "fmt"
    "log"
    "net/http"
    "net/http/httptest"
    "time"
)

type Middleware func(http.Handler) http.Handler

// Chain applies middleware in order: first middleware is outermost.
// Chain(Logging, Recovery, Auth)(handler)
// Request flow: Logging -> Recovery -> Auth -> handler
func Chain(middlewares ...Middleware) Middleware {
    return func(final http.Handler) http.Handler {
        for i := len(middlewares) - 1; i >= 0; i-- {
            final = middlewares[i](final)
        }
        return final
    }
}

func LoggingMW(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        log.Printf("--> %s %s", r.Method, r.URL.Path)
        next.ServeHTTP(w, r)
        log.Printf("<-- %s %s (%v)", r.Method, r.URL.Path, time.Since(start))
    })
}

func RecoveryMW(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("PANIC: %v", err)
                http.Error(w, "Internal Server Error", 500)
            }
        }()
        next.ServeHTTP(w, r)
    })
}

func RequestIDMW(next http.Handler) http.Handler {
    counter := 0
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        counter++
        w.Header().Set("X-Request-ID", fmt.Sprintf("req-%d", counter))
        next.ServeHTTP(w, r)
    })
}

func apiHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "API response")
}

func main() {
    // Compose the middleware stack
    stack := Chain(LoggingMW, RecoveryMW, RequestIDMW)
    handler := stack(http.HandlerFunc(apiHandler))

    req := httptest.NewRequest("GET", "/api/data", nil)
    w := httptest.NewRecorder()
    handler.ServeHTTP(w, req)

    fmt.Printf("Status: %d\n", w.Code)
    fmt.Printf("Request-ID: %s\n", w.Header().Get("X-Request-ID"))
    fmt.Printf("Body: %s", w.Body.String())
}
Output
2026/06/08 10:00:00 --> GET /api/data 2026/06/08 10:00:00 <-- GET /api/data (42us) Status: 200 Request-ID: req-1 Body: API response

package main

import (
    "fmt"
    "net/http"
)

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/api/users", usersHandler)

Common Mistakes

Do: Call next.ServeHTTP
func MyMW(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // pre-processing
        next.ServeHTTP(w, r) // pass control
        // post-processing
    })
}
Don't: Forget to call next
func MyMW(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        log.Println("logging...")
        // Forgot next.ServeHTTP(w, r)!
        // Handler never executes
    })
}
Do: Respect middleware order
// Recovery BEFORE auth, so panics in
// auth middleware are caught
Chain(Logging, Recovery, Auth)(handler)
Don't: Wrong order
// Auth BEFORE Recovery means a panic
// in Auth crashes the server
Chain(Logging, Auth, Recovery)(handler)

Key Rules

  • Middleware order matters — outermost middleware runs first on the request and last on the response
  • Use context to pass data between middleware and handlers (not global variables)
  • Keep each middleware focused on one concern — logging, auth, and recovery should be separate
  • Always call next.ServeHTTP(w, r) unless you intentionally want to short-circuit (e.g., auth failure)
  • Place recovery middleware early in the chain so it catches panics from all downstream handlers
Connection: Middleware chains use First-class Functions (functions returning functions) and the http.Handler interface from net/http. Recovery middleware directly applies Defer, Panic, Recover. Auth middleware uses Context to propagate user data. Test middleware with httptest. Configuration of middleware stacks can use the Functional Options pattern.
