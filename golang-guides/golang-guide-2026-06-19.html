Repository Pattern

Advanced Patterns • 9 days since review

Quick Reminder

Define a data-access interface (e.g. UserRepository), implement it with concrete storage, and inject it via the constructor. The business logic depends only on the interface, so tests can swap in a mock implementation without touching a real database.

Key Pattern

type UserRepository interface {
    FindByID(ctx context.Context, id string) (*User, error)
    Save(ctx context.Context, u *User) error
}

type pgUserRepo struct { db *sql.DB }

func (r *pgUserRepo) FindByID(ctx context.Context, id string) (*User, error) {
    var u User
    row := r.db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id=$1", id)
    if err := row.Scan(&u.ID, &u.Name); err != nil {
        return nil, err
    }
    return &u, nil
}
Challenge Exercise

Implement a CachingRepository that wraps any UserRepository, checks an in-memory cache first on FindByID, and falls back to the wrapped repo on a miss (populating the cache). Save should write through and invalidate the cache entry.

Pitfall: Leaking implementation details through the interface — keep it minimal (CRUD methods only) and never expose *sql.DB, transaction handles, or SQL strings. If a caller needs the DB, the abstraction has failed.

unsafe Package

Advanced Patterns • 9 days since review

Quick Reminder

unsafe.Pointer bypasses type safety for raw memory access. unsafe.Sizeof, unsafe.Offsetof, and unsafe.Alignof introspect struct layout (including padding). Almost never needed in application code — reach for it only at FFI boundaries or for very specific zero-copy tricks.

Key Pattern

import "unsafe"

type S struct {
    a int32
    b int64
}

fmt.Println(unsafe.Sizeof(S{}))     // 16 (4 bytes a + 4 padding + 8 b)
fmt.Println(unsafe.Offsetof(S{}.b)) // 8
fmt.Println(unsafe.Alignof(S{}))    // 8
Challenge Exercise

Use unsafe.Pointer (or unsafe.String / unsafe.Slice from Go 1.20+) to convert a []byte to a string without allocating a copy. Then explain in a comment why mutating the original slice afterwards is undefined behavior.

Pitfall: Breaking garbage-collector invariants — never store a uintptr in a variable and later convert it back to unsafe.Pointer. The GC can move or free the object between statements; conversions must happen in a single expression.

time Package

Standard Library Mastery • 8 days since review

Quick Reminder

time.Now(), time.Since(), and the time.Duration type cover most timing needs. Parse and format using the reference time Mon Jan 2 15:04:05 MST 2006. Use time.NewTicker for periodic work and time.After for timeouts.

Key Pattern

t, _ := time.Parse("2006-01-02", "2026-06-19")
fmt.Println(t.Format("January 2, 2006")) // June 19, 2026

ticker := time.NewTicker(5 * time.Second)
defer ticker.Stop()
for range ticker.C {
    doPeriodicWork()
}
Challenge Exercise

Build a rate limiter using time.Ticker that allows N operations per second. Block callers until a tick is available, and expose a Stop() that releases the ticker. Bonus: support bursts by buffering up to N ready tokens.

Pitfall: Not calling ticker.Stop() — a leaked ticker keeps an internal goroutine/timer alive and never gets garbage collected, slowly accumulating in long-running services.

testing Package

Testing & Tooling • 8 days since review

Quick Reminder

Table-driven tests use []struct{name string; ...} plus t.Run(tt.name, ...) for named subtests. t.Parallel() runs subtests concurrently. t.Helper() marks helper functions so failures point at the caller, not the helper.

Key Pattern

tests := []struct {
    name  string
    input int
    want  int
}{
    {"positive", 5, 25},
    {"zero", 0, 0},
}
for _, tt := range tests {
    t.Run(tt.name, func(t *testing.T) {
        if got := Square(tt.input); got != tt.want {
            t.Errorf("Square(%d) = %d, want %d", tt.input, got, tt.want)
        }
    })
}
Challenge Exercise

Write a table-driven test with t.Parallel() for a concurrency-safe function, ensuring each subtest is independent. Add a shared assertEqual helper using t.Helper() and verify the failure message points at the subtest line.

Pitfall: Go < 1.22 loop-variable capture in parallel tests — calling t.Parallel() defers the subtest body, so all closures see the final tt. Add tt := tt before t.Parallel() (unnecessary on Go 1.22+).

Benchmarks

Testing & Tooling • 8 days since review

Quick Reminder

func BenchmarkXxx(b *testing.B) runs the body b.N times, with b.N chosen automatically. b.ResetTimer() excludes setup from the measurement; b.ReportAllocs() adds per-op allocation columns.

Key Pattern

func BenchmarkConcat(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        _ = strings.Join(data, ",")
    }
}
Challenge Exercise

Write sub-benchmarks (via b.Run) comparing fmt.Sprintf, strings.Builder, and the + operator for building a string from 100 parts. Report allocations and rank the three by ns/op and allocs/op.

Pitfall: Including setup time in the measurement — always call b.ResetTimer() after any expensive setup, or the one-time cost is averaged into every reported ns/op.

bufio: Buffered I/O

I/O & Networking • 8 days since review

Quick Reminder

bufio.NewScanner(r) reads line-by-line. bufio.NewReader(r) wraps any io.Reader with buffering. A custom SplitFunc handles non-line delimiters. bufio.NewWriter(w) buffers writes — remember to Flush.

Key Pattern

scanner := bufio.NewScanner(file)
for scanner.Scan() {
    line := scanner.Text()
    // process line
}
if err := scanner.Err(); err != nil {
    log.Fatal(err)
}
Challenge Exercise

Implement a custom bufio.SplitFunc that splits input on \0 (null bytes) instead of newlines, correctly handling the final token at EOF and requesting more data when no delimiter is found yet.

Pitfall: The default scanner buffer is 64KB — for lines longer than that, Scan() stops and scanner.Err() returns bufio.ErrTooLong. Call scanner.Buffer(buf, maxSize) to raise the limit before scanning.

maps & slices Packages (Go 1.21+)

Modern Go Standard Library

Why It Matters

Go 1.21 added the slices and maps packages: generic utility functions that eliminate years of hand-rolled boilerplate. Sorting, searching, comparing, cloning, and transforming slices and maps are now type-safe one-liners — no more interface{} and no more copy-pasted helper loops.

What to Cover

  • slices.Sort, slices.SortFunc — generic sort replacing sort.Slice
  • slices.BinarySearch, slices.BinarySearchFunc — typed binary search
  • slices.Contains, slices.Index — membership and lookup
  • slices.Compact, slices.CompactFunc — remove consecutive duplicates
  • maps.Keys, maps.Values, maps.Clone, maps.Equal
  • slices.Grow, slices.Clip — capacity management

Example 1 — Basic: Sort a Slice of Structs by Field

package main

import (
    "cmp"
    "fmt"
    "slices"
)

type User struct {
    Name string
    Age  int
}

func main() {
    users := []User{
        {"Charlie", 30},
        {"Alice", 25},
        {"Bob", 30},
    }
    // SortFunc takes a comparator returning -1, 0, or +1.
    slices.SortFunc(users, func(a, b User) int {
        return cmp.Compare(a.Name, b.Name)
    })
    for _, u := range users {
        fmt.Printf("%s (%d)\n", u.Name, u.Age)
    }
}
Output
Alice (25) Bob (30) Charlie (30)

cmp.Compare (from the new cmp package) returns -1/0/+1 for any ordered type, exactly the contract SortFunc expects — no more i, j int index juggling.

Example 2 — Intermediate: Binary Search on a Sorted Slice

package main

import (
    "fmt"
    "slices"
)

func main() {
    nums := []int{2, 4, 6, 8, 10}

    idx, found := slices.BinarySearch(nums, 6)
    fmt.Println(idx, found) // 2 true

    // For a missing value, idx is the insertion point.
    idx, found = slices.BinarySearch(nums, 7)
    fmt.Println(idx, found) // 3 false

    // Membership without caring about position:
    fmt.Println(slices.Contains(nums, 8)) // true
    fmt.Println(slices.Index(nums, 10))   // 4
}
Output
2 true 3 false true 4

BinarySearch returns both the index and a found bool, so you never have to re-check nums[idx] == target the way the old sort.Search required.

Example 3 — Advanced: Deterministic Map Iteration

package main

import (
    "fmt"
    "maps"
    "slices"
)

func main() {
    scores := map[string]int{
        "charlie": 92,
        "alice":   88,
        "bob":     75,
    }
    // maps.Keys returns an iterator (Go 1.23); collect into a slice.
    keys := slices.Collect(maps.Keys(scores))
    slices.Sort(keys)

    for _, k := range keys {
        fmt.Printf("%s: %d\n", k, scores[k])
    }
}
Output
alice: 88 bob: 75 charlie: 92

Map iteration order is deliberately randomized in Go. The idiom slices.Sort(slices.Collect(maps.Keys(m))) gives you stable, reproducible output — essential for golden-file tests and deterministic logs.

Example 4 — Real-World: Deduplicate by Field with CompactFunc

package main

import (
    "cmp"
    "fmt"
    "slices"
)

type Event struct {
    UserID string
    Action string
}

func main() {
    events := []Event{
        {"u1", "login"},
        {"u1", "click"},
        {"u2", "login"},
        {"u2", "login"},
    }
    // Compact only removes CONSECUTIVE dups, so sort by the key first.
    slices.SortFunc(events, func(a, b Event) int {
        return cmp.Compare(a.UserID, b.UserID)
    })
    unique := slices.CompactFunc(events, func(a, b Event) bool {
        return a.UserID == b.UserID
    })
    for _, e := range unique {
        fmt.Printf("%s %s\n", e.UserID, e.Action)
    }
}
Output
u1 login u2 login

One row per UserID. Because CompactFunc only collapses adjacent equal elements, sorting by the dedup key first is mandatory — otherwise duplicates that aren't neighbors survive.

Common Mistakes

Don't
// Reaching for the old reflection-
// based sort.Slice for generic data.
sort.Slice(users, func(i, j int) bool {
    return users[i].Name < users[j].Name
})

// Expecting Compact to remove ALL dups:
u := slices.Compact([]int{1, 2, 1})
// u == [1 2 1] -- the trailing 1
// was never adjacent to the first.
Do
// Type-safe SortFunc + cmp.Compare.
slices.SortFunc(users, func(a, b User) int {
    return cmp.Compare(a.Name, b.Name)
})

// Sort first, THEN compact for a
// full deduplication.
nums := []int{1, 2, 1}
slices.Sort(nums)
nums = slices.Compact(nums) // [1 2]
  • Using sort.Slice (reflection-based) when slices.SortFunc is available — the generic version is faster and type-checked.
  • Assuming slices.Compact removes all duplicates — it only removes consecutive ones; sort first for global dedup.
  • Mutating a slice while a maps.Keys/slices.Values iterator is still being consumed — collect to a slice first if you need to modify.

Key Rules

  • Prefer slices.SortFunc + cmp.Compare over sort.Slice for new code.
  • BinarySearch returns (index, found) — use the bool, don't re-check by hand.
  • Sort before Compact when you want a fully deduplicated slice.
  • Use slices.Clone / maps.Clone for shallow copies instead of manual loops.
Exercise 1

Exercise 2

Write a function SortedKeys[K cmp.Ordered, V any](m map[K]V) []K that returns the map's keys deduplicated (they already are) and sorted, using only the maps and slices packages.

Exercise 3

Implement a type-safe filter using slices.DeleteFunc: given []int, remove every element that fails a predicate, in place, and return the trimmed slice.

Connections: These packages build directly on generics (type parameters and the cmp.Ordered constraint) and supersede the older sort package reviewed earlier. maps.Keys/slices.Values return range-over-func iterators (Go 1.23), tying into modern iterator patterns.

Advanced slog: Custom Handlers & Contextual Logging

Modern Go Standard Library

Why It Matters

Beyond basic slog.Info(), production systems need custom handlers for JSON or logfmt output, per-package log levels, request-scoped context (trace IDs, user IDs), and redaction of sensitive fields. Mastering the slog.Handler interface is what turns logging from noise into observability.

What to Cover

  • The slog.Handler interface: Handle, WithAttrs, WithGroup, Enabled
  • Writing a custom handler (e.g. add hostname/pid to every record)
  • slog.NewLogLogger for bridging to the legacy log.Logger
  • Context-aware logging: pulling trace/request IDs from a context.Context
  • HandlerOptions.ReplaceAttr for redacting sensitive fields and Level for filtering

Example 1 — Basic: A Custom Handler that Adds Hostname and PID

package main

import (
    "context"
    "log/slog"
    "os"
)

// hostHandler wraps another handler and injects host/pid on every record.
type hostHandler struct {
    slog.Handler
    host string
    pid  int
}

func (h *hostHandler) Handle(ctx context.Context, r slog.Record) error {
    r.AddAttrs(slog.String("host", h.host), slog.Int("pid", h.pid))
    return h.Handler.Handle(ctx, r)
}

func main() {
    host, _ := os.Hostname()
    base := slog.NewJSONHandler(os.Stdout, nil)
    logger := slog.New(&hostHandler{Handler: base, host: host, pid: os.Getpid()})

    logger.Info("server started", "port", 8080)
}
Output (one JSON line)
{"time":"2026-06-19T10:00:00Z","level":"INFO","msg":"server started","port":8080,"host":"web-1","pid":4242}

Embedding slog.Handler gives you Enabled, WithAttrs, and WithGroup for free — you only override Handle to inject the extra attributes before delegating.

Example 2 — Intermediate: Context-Aware Request ID Logging

type ctxKey struct{}

func WithRequestID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, ctxKey{}, id)
}

// ctxHandler copies a request_id from the context onto the record.
type ctxHandler struct{ slog.Handler }

func (h ctxHandler) Handle(ctx context.Context, r slog.Record) error {
    if id, ok := ctx.Value(ctxKey{}).(string); ok {
        r.AddAttrs(slog.String("request_id", id))
    }
    return h.Handler.Handle(ctx, r)
}

func middleware(logger *slog.Logger, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        rid := r.Header.Get("X-Request-ID")
        ctx := WithRequestID(r.Context(), rid)
        // Use the context-aware variants: InfoContext, ErrorContext...
        logger.InfoContext(ctx, "request received", "path", r.URL.Path)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

Because the handler reads from ctx, every *Context log call inside the request automatically carries the request_id — no need to thread it through every function signature.

Example 3 — Advanced: Redacting Sensitive Fields with ReplaceAttr

package main

import (
    "log/slog"
    "os"
    "strings"
)

func redact(groups []string, a slog.Attr) slog.Attr {
    key := strings.ToLower(a.Key)
    for _, secret := range []string{"password", "token", "secret"} {
        if strings.Contains(key, secret) {
            return slog.String(a.Key, "[REDACTED]")
        }
    }
    return a
}

func main() {
    h := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        ReplaceAttr: redact,
    })
    slog.New(h).Info("login attempt",
        "user", "ada",
        "password", "hunter2",
        "api_token", "sk-abc123")
}
Output
{"time":"...","level":"INFO","msg":"login attempt","user":"ada","password":"[REDACTED]","api_token":"[REDACTED]"}

ReplaceAttr runs for every attribute before it's written. Matching on key substrings catches password, api_token, and friends in one place — far safer than relying on every call site to remember.

Example 4 — Real-World: Derived Loggers and Level Filtering

package main

import (
    "log/slog"
    "os"
)

func main() {
    // Only WARN and above for this handler.
    h := slog.NewTextHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelWarn,
    })
    base := slog.New(h)

    // Derive a logger with fixed attrs instead of re-creating per request.
    reqLogger := base.With("request_id", "r-99", "user", "ada")

    reqLogger.Info("this is dropped")           // below WARN -> filtered out
    reqLogger.Warn("disk almost full", "pct", 92)
}
Output
level=WARN msg="disk almost full" request_id=r-99 user=ada pct=92

logger.With(...) returns a derived logger that carries the attrs cheaply — it calls WithAttrs on the handler once, not on every log line. The Info call is dropped because the handler's level is Warn.

Common Mistakes

Don't
// New logger per request: allocates
// and re-parses attrs every time.
func handle(r *http.Request) {
    log := slog.New(handler)
    log = log.With("request_id", rid)
    log.Info("...")
}

// Logging secrets in plain fields.
log.Info("auth", "password", pw)
Do
// Derive once with With().
reqLog := baseLogger.With(
    "request_id", rid,
)
reqLog.Info("...")

// Redact centrally via ReplaceAttr
// so no call site can leak secrets.
opts := &slog.HandlerOptions{
    ReplaceAttr: redact,
}
  • Creating a brand-new slog.Logger per request instead of deriving with Logger.With(...).
  • Logging sensitive data (passwords, tokens) in structured fields — redact centrally with ReplaceAttr.
  • Forgetting WithGroup semantics — ReplaceAttr's groups argument tells you the nesting path; ignoring it can mis-redact nested keys.

Key Rules

  • Embed slog.Handler in a custom handler so you only override Handle.
  • Use the *Context log methods so handlers can read request-scoped values.
  • Redact secrets once, centrally, via ReplaceAttr — never per call site.
  • Derive per-request loggers with With() rather than constructing new ones.
Exercise 1

Write a custom slog.Handler that outputs logs in logfmt format (key=value pairs on one line), correctly quoting values that contain spaces.

Exercise 2

Create HTTP middleware that extracts an X-Request-ID header (generating one if absent), stores it in the context, and ensures every log line within that request includes it.

Exercise 3

Connections: Advanced slog combines interfaces (the Handler contract), struct embedding (wrapping a base handler), context (request-scoped IDs), and net/http middleware. It extends the structured-logging basics and pairs with OpenTelemetry tracing for full request observability.

Production Testing & Architecture — 3 Concepts

Integration Testing Patterns

Production Testing & Architecture

Why It Matters

Unit tests with mocks can't catch real database bugs, SQL typos, migration mistakes, or API contract mismatches. Integration tests verify that components actually work together against a real (or realistic) dependency — the confidence you need before shipping a data-access layer.

What to Cover

  • TestMain for shared setup/teardown (start DB, seed data, cleanup)
  • Test database setup: in-memory SQLite or a dockerized PostgreSQL
  • t.Cleanup for per-test resource management
  • Build tags //go:build integration to separate slow tests from the unit fast path
  • testcontainers-go for Docker-based test dependencies

Example 1 — Basic: TestMain Starting an In-Memory SQLite

// file: repo_test.go
package storage

import (
    "database/sql"
    "os"
    "testing"

    _ "modernc.org/sqlite" // pure-Go driver, no cgo
)

var testDB *sql.DB

func TestMain(m *testing.M) {
    db, err := sql.Open("sqlite", ":memory:")
    if err != nil {
        panic(err)
    }
    if _, err := db.Exec(`CREATE TABLE users (id TEXT PRIMARY KEY, name TEXT)`); err != nil {
        panic(err)
    }
    testDB = db

    code := m.Run() // run all tests in this package

    db.Close()
    os.Exit(code)
}

TestMain runs once for the whole package. Set up shared state, call m.Run(), then tear down and pass its exit code to os.Exit — deferred functions don't run after os.Exit, so close explicitly.

Example 2 — Intermediate: A Save → FindByID Round-Trip

func TestUserRepo_RoundTrip(t *testing.T) {
    repo := &sqlUserRepo{db: testDB}

    want := &User{ID: "u1", Name: "Ada"}
    if err := repo.Save(t.Context(), want); err != nil {
        t.Fatalf("Save: %v", err)
    }

    got, err := repo.FindByID(t.Context(), "u1")
    if err != nil {
        t.Fatalf("FindByID: %v", err)
    }
    if got.Name != want.Name {
        t.Errorf("got %q, want %q", got.Name, want.Name)
    }

    // Clean up this test's row so other tests stay isolated.
    t.Cleanup(func() {
        testDB.Exec("DELETE FROM users WHERE id = ?", "u1")
    })
}

This exercises the real driver, real SQL, and real scanning — the exact path a mock would have hidden. t.Cleanup guarantees the row is removed even if an assertion fails.

Example 3 — Advanced: Isolated DB Per Test via a Helper

// newTestDB returns a fresh in-memory DB and registers its cleanup.
func newTestDB(t *testing.T) *sql.DB {
    t.Helper()
    db, err := sql.Open("sqlite", ":memory:")
    if err != nil {
        t.Fatalf("open: %v", err)
    }
    if _, err := db.Exec(schema); err != nil {
        t.Fatalf("migrate: %v", err)
    }
    t.Cleanup(func() { db.Close() })
    return db
}

func TestSomething(t *testing.T) {
    db := newTestDB(t) // each test gets its own isolated database
    repo := &sqlUserRepo{db: db}
    // ... no cross-test interference, safe to t.Parallel()
    _ = repo
}

Giving each test its own database (or a transaction that rolls back) removes shared-state coupling entirely — tests can run in parallel and in any order without flakiness.

Example 4 — Real-World: Build Tag to Separate Integration Tests

//go:build integration

// file: postgres_integration_test.go
package storage

import "testing"

func TestAgainstRealPostgres(t *testing.T) {
    // Spins up a dockerized Postgres (e.g. via testcontainers-go),
    // runs migrations, and exercises the repo against it.
    // ...
}
Makefile / CLI
# fast path (unit tests only) -- the default go test ./... # include integration tests go test -tags=integration ./...

The //go:build integration tag (it must be the first line, followed by a blank line) excludes the file from normal go test. CI runs the fast unit path on every push and the tagged integration suite on a slower lane.

Common Mistakes

Don't
// All tests share one mutable DB
// and depend on execution order.
func TestA(t *testing.T) {
    db.Exec("INSERT ...") // leaks
}
func TestB(t *testing.T) {
    // assumes TestA's row exists -- flaky!
}
Do
// Each test isolates its own data
// and cleans up after itself.
func TestA(t *testing.T) {
    db := newTestDB(t)
    // ... t.Cleanup closes db
}
func TestB(t *testing.T) {
    db := newTestDB(t) // independent
}
  • Running integration tests against a shared or production database — each test should get an isolated DB or a transaction that rolls back.
  • Not cleaning up resources — leaked connections, temp files, and dangling rows cause flaky neighbors. Use t.Cleanup(func(){ db.Close() }).
  • Forgetting the blank line after a //go:build tag — without it the line is treated as an ordinary comment and the tag is ignored.

Key Rules

  • Use TestMain for package-wide setup; t.Cleanup for per-test teardown.
  • Isolate each test's data — fresh DB or rolled-back transaction.
  • Gate slow integration tests behind a //go:build integration tag.
  • Prefer a pure-Go driver (e.g. modernc.org/sqlite) so tests need no cgo or external services.

Dependency Injection with Wire

Production Testing & Architecture

Why It Matters

As an application grows, manually wiring dependencies (NewServer(NewService(NewRepo(db)))) becomes error-prone and tedious. Google's Wire generates this code at compile time — no runtime reflection, fully type-safe, and the generated file is plain readable Go you can inspect.

What to Cover

  • The problem: deep manual constructor chains become unwieldy
  • Wire concepts: providers, injectors, provider sets
  • Provider functions: func NewRepo(db *sql.DB) *Repo
  • Injector functions with wire.Build (generates wire_gen.go)
  • Interface bindings with wire.Bind; grouping with wire.NewSet

Example 1 — Basic: Providers → Injector → Generated Code

// providers.go -- ordinary constructors, no Wire magic.
func NewDB(cfg Config) (*sql.DB, error) {
    return sql.Open("postgres", cfg.DSN)
}
func NewRepo(db *sql.DB) *Repo        { return &Repo{db: db} }
func NewService(r *Repo) *Service     { return &Service{repo: r} }
func NewServer(s *Service) *Server    { return &Server{svc: s} }
//go:build wireinject

// wire.go -- this file is the BLUEPRINT; Wire reads it to generate code.
package main

import "github.com/google/wire"

func InitServer(cfg Config) (*Server, error) {
    wire.Build(NewDB, NewRepo, NewService, NewServer)
    return nil, nil // body is ignored; Wire generates the real one
}

Running wire in the package reads the wire.Build call, topologically sorts the providers by their parameter/return types, and emits a real InitServer in wire_gen.go.

Example 2 — Intermediate: The Generated Output

// wire_gen.go -- GENERATED. Plain, readable, debuggable Go.
//go:build !wireinject

func InitServer(cfg Config) (*Server, error) {
    db, err := NewDB(cfg)
    if err != nil {
        return nil, err
    }
    repo := NewRepo(db)
    service := NewService(repo)
    server := NewServer(service)
    return server, nil
}

This is exactly what you'd have written by hand — Wire just keeps it correct as the graph grows. The complementary //go:build tags ensure the blueprint and the generated file are never compiled together.

Example 3 — Advanced: Interface Binding for Testability

// Service depends on the interface, not *Repo.
type UserStore interface {
    FindByID(id string) (*User, error)
}

func NewService(s UserStore) *Service { return &Service{store: s} }

var RepoSet = wire.NewSet(
    NewRepo,
    // Tell Wire: when a UserStore is needed, provide *Repo.
    wire.Bind(new(UserStore), new(*Repo)),
)

func InitServer(cfg Config) (*Server, error) {
    wire.Build(NewDB, RepoSet, NewService, NewServer)
    return nil, nil
}

wire.Bind maps an interface to a concrete implementation. In a test, swap RepoSet for a set that binds UserStore to a mock — the rest of the graph is untouched.

Example 4 — Real-World: Provider Sets and Cleanup Functions

// NewDB returns a cleanup func; Wire threads it through automatically.
func NewDB(cfg Config) (*sql.DB, func(), error) {
    db, err := sql.Open("postgres", cfg.DSN)
    if err != nil {
        return nil, nil, err
    }
    return db, func() { db.Close() }, nil
}

var InfraSet = wire.NewSet(NewDB, NewRepo)
var AppSet   = wire.NewSet(NewService, NewServer)

func InitServer(cfg Config) (*Server, func(), error) {
    wire.Build(InfraSet, AppSet)
    return nil, nil, nil
}

// Caller:
// srv, cleanup, err := InitServer(cfg)
// defer cleanup() // closes the DB on shutdown

When a provider returns (T, func(), error), Wire collects the cleanup functions and returns a single aggregate cleanup that runs them in reverse order — perfect for graceful shutdown.

Common Mistakes

Don't
// Add a provider but forget to
// re-run wire -- wire_gen.go is
// now stale and won't compile or
// will inject the wrong graph.
func InitServer(cfg Config) (*Server, error) {
    wire.Build(NewDB, NewRepo, NewCache,
        NewService, NewServer) // NewCache added...
    return nil, nil
}
// ...but `wire` never re-run.
Do
// Add a //go:generate directive and
// regenerate after any change.
//go:generate wire

// Then:
//   go generate ./...
// keeps wire_gen.go in sync.
  • Forgetting to re-run wire after adding or changing a provider — add //go:generate wire and run go generate ./....
  • Introducing a circular dependency — Wire errors at generation time, but the message can be cryptic; break cycles with an interface.
  • Providing the same type twice in one injector — Wire can't decide which to use and fails; give distinct types or use named providers.

Key Rules

  • Providers are plain constructors; injectors are blueprints behind a wireinject build tag.
  • Group related providers with wire.NewSet and bind interfaces with wire.Bind.
  • Return (T, func(), error) from providers that own resources so Wire threads cleanup.
  • Always go generate after changing the dependency graph — never hand-edit wire_gen.go.

Connections: Wire formalizes the constructor injection behind the repository pattern and plugin architecture. wire.Bind relies on interfaces, and the generation step uses the same code-generation / go:generate tooling seen earlier. Cleanup threading complements graceful shutdown.

Advanced HTTP Patterns: chi Router & Middleware

Production Testing & Architecture

Why It Matters

While net/http is powerful, chi adds lightweight routing with URL parameters, middleware groups, and sub-routers — all built on the standard http.Handler interface. It's the most popular choice for RESTful APIs in Go and adds zero non-stdlib dependencies of its own.

What to Cover

  • Route patterns: r.Get("/users/{id}", h) with chi.URLParam(r, "id")
  • Middleware: r.Use(middleware.Logger), r.Use(middleware.Recoverer)
  • Route groups: r.Route("/api/v1", func(r chi.Router) { ... })
  • Sub-routers: r.Mount("/admin", adminRouter)
  • Custom middleware and passing values through context

Example 1 — Basic: Parameterized Routes

package main

import (
    "fmt"
    "net/http"

    "github.com/go-chi/chi/v5"
)

func main() {
    r := chi.NewRouter()
    r.Get("/users/{id}", func(w http.ResponseWriter, req *http.Request) {
        id := chi.URLParam(req, "id")
        fmt.Fprintf(w, "user %s", id)
    })
    http.ListenAndServe(":8080", r)
}
curl localhost:8080/users/42
user 42

chi.URLParam pulls a named path segment. Because chi.Router satisfies http.Handler, you pass it straight to http.ListenAndServe — no adapter needed.

Example 2 — Intermediate: Middleware Stack

r := chi.NewRouter()
r.Use(middleware.RequestID)
r.Use(middleware.Logger)    // logs method, path, status, duration
r.Use(middleware.Recoverer) // recovers panics -> 500 instead of crash

r.Get("/health", func(w http.ResponseWriter, req *http.Request) {
    w.Write([]byte("ok"))
})

Middleware registered with r.Use wraps every route below it, in registration order. Recoverer turns a panicking handler into a clean 500 rather than taking down the server.

Example 3 — Advanced: RESTful Resource with a Sub-Router

func todoRouter() chi.Router {
    r := chi.NewRouter()
    r.Get("/", listTodos)            // GET    /todos
    r.Post("/", createTodo)          // POST   /todos
    r.Route("/{id}", func(r chi.Router) {
        r.Get("/", getTodo)          // GET    /todos/{id}
        r.Delete("/", deleteTodo)    // DELETE /todos/{id}
    })
    return r
}

func main() {
    root := chi.NewRouter()
    root.Use(middleware.Logger)
    root.Mount("/todos", todoRouter()) // mount the sub-router
    http.ListenAndServe(":8080", root)
}

Mount grafts a self-contained sub-router under a prefix. The /todos resource owns its own routes and can be tested in isolation — a clean way to organize a growing API.

Example 4 — Real-World: Scoped Auth Middleware via Group

// Custom middleware: reject requests without an Authorization header.
func auth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        if r.Header.Get("Authorization") == "" {
            http.Error(w, "unauthorized", http.StatusUnauthorized)
            return
        }
        next.ServeHTTP(w, r)
    })
}

func main() {
    r := chi.NewRouter()

    // Public routes.
    r.Get("/", home)
    r.Post("/login", login)

    // Protected routes: auth applies ONLY inside this group.
    r.Group(func(r chi.Router) {
        r.Use(auth)
        r.Get("/profile", profile)
        r.Post("/orders", createOrder)
    })

    http.ListenAndServe(":8080", r)
}

r.Group creates an inline scope with its own middleware stack but the same URL space — perfect when some routes are public and others require auth, without splitting into separate routers.

Common Mistakes

Don't
// Mixing net/http patterns with chi
// and reading params from the URL by
// hand.
r.Get("/users/{id}", func(w, req) {
    id := strings.TrimPrefix(
        req.URL.Path, "/users/")
})

// Applying auth globally even though
// /login must be public.
r.Use(auth) // locks everyone out!
Do
// Use chi's param accessor.
r.Get("/users/{id}", func(w, req) {
    id := chi.URLParam(req, "id")
})

// Scope auth to the protected group.
r.Group(func(r chi.Router) {
    r.Use(auth)
    r.Get("/profile", profile)
})
  • Parsing path parameters by hand instead of using chi.URLParam — fragile and breaks on nested routes.
  • Applying auth middleware globally with r.Use when some routes (login, health) must stay public — scope it with r.Group or a sub-router instead.
  • Registering routes after mounting/serving — chi builds its routing tree as you declare; register all routes before ListenAndServe.

Key Rules

  • Use r.Get/Post/... and chi.URLParam — never hand-parse the URL path.
  • Global concerns (RequestID, Logger, Recoverer) go in r.Use at the top.
  • Scope auth and other selective middleware with r.Group or r.Route.
  • Compose large APIs from sub-routers with r.Mount for isolation and testability.
