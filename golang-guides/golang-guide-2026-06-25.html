Module Management: go.mod, Workspaces & Versioning

Production Go • overdue 13d • rep 0

Quick Reminder

A module is a tree of packages rooted at a go.mod file. go.mod records the module path, the language version, and dependency requires (transitive ones are marked // indirect). Semantic Import Versioning means major version 2+ must appear in the import path (.../widget/v2). go mod tidy keeps go.mod/go.sum in sync with the code; replace swaps a dependency for a local fork during development.

A typical go.mod

module github.com/acme/widget

go 1.24

require (
    github.com/google/uuid v1.6.0
    golang.org/x/sync v0.10.0
)

require github.com/davecgh/go-spew v1.1.1 // indirect

// Swap in a local checkout while debugging upstream
replace github.com/google/uuid => ../uuid-fork

The commands you reach for daily

go mod init github.com/acme/widget   # create a module
go get github.com/google/uuid@v1.6.0 # add / pin a dependency
go mod tidy                          # add missing, drop unused, fix go.sum
go mod why github.com/davecgh/go-spew # explain why it is needed
go list -m all                       # full build list (MVS-selected versions)
Output of go mod why
# github.com/davecgh/go-spew/spew github.com/acme/widget github.com/stretchr/testify/assert github.com/davecgh/go-spew/spew
Challenge Exercise

Create a module example.com/mathx, add an external dependency, then run go mod tidy and inspect the resulting go.sum. Next, simulate a breaking v2 release: explain what the import path and go.mod module line must become, and why a consumer can import both v1 and v2 of the same module simultaneously. Finally, use a replace directive to point one dependency at a sibling directory and confirm with go list -m.

Pitfall: Hand-editing require versions and forgetting go.sum. Go verifies every download against go.sum checksums; if they drift you get "checksum mismatch" or "missing go.sum entry" errors. Always change versions with go get pkg@version followed by go mod tidy rather than editing the file by hand.

Cross-Compilation & Build Tags

Production Go • overdue 13d • rep 0

Quick Reminder

Go cross-compiles by simply setting GOOS and GOARCH — no cross toolchain to install. Set CGO_ENABLED=0 for a fully static binary that runs in a scratch container. Build constraints select which files compile for which target, either via a //go:build line at the top of a file or via filename suffixes like _linux.go / _amd64.go.

A build-tagged file

//go:build linux

package main

import "fmt"

// Only compiled into linux builds; provide a _windows.go twin.
func platform() string { return "linux" }

func main() { fmt.Println("running on", platform()) }

Building for another OS/arch

go tool dist list                 # every supported GOOS/GOARCH pair

# static linux/arm64 binary, built from macOS, no C toolchain:
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 go build -o app .

file app
Output of file app
app: ELF 64-bit LSB executable, ARM aarch64, version 1 (SYSV), statically linked, Go BuildID=..., not stripped
Challenge Exercise

Write two files, greet_linux.go and greet_darwin.go, each defining greeting() string differently, plus a shared main.go that prints it. Build for both targets without changing any source and confirm the output differs. Then convert the filename-based selection into explicit //go:build constraints with a custom tag (//go:build premium) and build with -tags premium.

Pitfall: Putting the //go:build line anywhere but the very top. The constraint must appear before the package clause, followed by a blank line; a stray comment or import above it silently disables the constraint, so the file compiles on every platform. Run go vet — it flags malformed build tags.

WebSocket in Go

Advanced Networking • overdue 12d • rep 0

Quick Reminder

WebSockets give you a persistent, full-duplex connection over a single HTTP upgrade. The standard library has no WebSocket server, so most code uses github.com/gorilla/websocket: an Upgrader turns an incoming HTTP request into a *websocket.Conn, then you loop on ReadMessage/WriteMessage. Each connection should be handled in its own goroutine.

A minimal echo server

package main

import (
    "log"
    "net/http"

    "github.com/gorilla/websocket"
)

var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool { return true }, // dev only
}

func echo(w http.ResponseWriter, r *http.Request) {
    conn, err := upgrader.Upgrade(w, r, nil)
    if err != nil {
        log.Println("upgrade:", err)
        return
    }
    defer conn.Close()
    for {
        mt, msg, err := conn.ReadMessage()
        if err != nil {
            break // client closed or read error
        }
        if err := conn.WriteMessage(mt, msg); err != nil {
            break
        }
    }
}

func main() {
    http.HandleFunc("/ws", echo)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
Transcript (client ↔ server on ws://localhost:8080/ws)
client -> "hello" server -> "hello" client -> "again" server -> "again"
Challenge Exercise

Extend the echo server into a tiny chat hub: keep a set of connected *websocket.Conn guarded by a mutex, and broadcast every received message to all clients. Add a write deadline (conn.SetWriteDeadline) and a ping/pong keep-alive so dead connections are detected and evicted. Why must all writes to a single connection be serialized through one goroutine?

Pitfall: Concurrent writes to one *websocket.Conn. The connection supports one concurrent reader and one concurrent writer only — calling WriteMessage from multiple goroutines corrupts frames. Funnel all outbound messages through a single writer goroutine (often fed by a channel), and never forget CheckOrigin in production to avoid cross-site hijacking.

Profiling with pprof

Observability & Profiling • overdue 12d • rep 0

Quick Reminder

pprof samples where your program spends CPU and where it allocates. Import net/http/pprof for the side-effect — it registers /debug/pprof/ handlers on the default mux — or use runtime/pprof to write a profile from code or a benchmark. Then analyze with go tool pprof (top, list, web).

Capturing a CPU profile in code

package main

import (
    "os"
    "runtime/pprof"
)

func main() {
    f, _ := os.Create("cpu.prof")
    defer f.Close()

    pprof.StartCPUProfile(f) // begin sampling
    defer pprof.StopCPUProfile()

    hotLoop() // the work you want to measure
}

func hotLoop() {
    sum := 0
    for i := 0; i < 1_000_000_000; i++ {
        sum += i % 7
    }
    _ = sum
}
Analyzing it: go tool pprof -top cpu.prof
Showing nodes accounting for 1.42s, 100% of 1.42s total flat flat% sum% cum cum% 1.42s 100% 100% 1.42s 100% main.hotLoop 0 0% 100% 1.42s 100% main.main
Challenge Exercise

Add import _ "net/http/pprof" and start http.ListenAndServe("localhost:6060", nil) in a goroutine. Capture a live 30-second CPU profile with go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30, then use list hotLoop to see line-level cost and web for a call graph. Separately, grab a heap profile (/debug/pprof/heap) and identify the top allocation site.

Pitfall: Exposing net/http/pprof on a public listener. Its import auto-registers handlers on http.DefaultServeMux, so if your main API also uses the default mux, /debug/pprof/ becomes internet-reachable — an information-disclosure and DoS risk. Bind pprof to localhost (or a separate, firewalled port) only.

OpenTelemetry & Tracing

Observability & Profiling • overdue 12d • rep 0

Quick Reminder

Distributed tracing follows one request across services as a tree of spans. With OpenTelemetry you configure a TracerProvider (with an exporter), obtain a Tracer, and call tracer.Start(ctx, name) to open a span — always defer span.End(). The returned ctx carries the span, so child spans nest automatically and context propagates across RPC boundaries.

One span with the stdout exporter

package main

import (
    "context"

    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/exporters/stdout/stdouttrace"
    sdktrace "go.opentelemetry.io/otel/sdk/trace"
    "go.opentelemetry.io/otel/trace"
)

func main() {
    exp, _ := stdouttrace.New(stdouttrace.WithPrettyPrint())
    tp := sdktrace.NewTracerProvider(sdktrace.WithBatcher(exp))
    defer tp.Shutdown(context.Background())
    otel.SetTracerProvider(tp)

    tr := otel.Tracer("checkout")
    ctx, span := tr.Start(context.Background(), "PlaceOrder")
    span.SetAttributes(attribute.Int("items", 3))
    doWork(ctx, tr) // child span nests under this one
    span.End()
}

func doWork(ctx context.Context, tr trace.Tracer) {
    _, span := tr.Start(ctx, "ChargeCard")
    defer span.End()
}
Exported span (abridged JSON)
{ "Name": "PlaceOrder", "SpanContext": { "TraceID": "7b1c...", "SpanID": "a14f..." }, "Attributes": [ { "Key": "items", "Value": { "Type": "INT64", "Value": 3 } } ], "ChildSpanCount": 1 }
Challenge Exercise

Instrument a two-handler HTTP service with otelhttp middleware so each request opens a root span automatically. Inside a handler, start a child span around a simulated DB call, record an error with span.RecordError(err) and set span.SetStatus(codes.Error, ...). Then swap the stdout exporter for an OTLP exporter pointed at a local collector and confirm the trace tree appears intact.

Pitfall: Dropping the context. If you call tracer.Start(context.Background(), ...) inside a handler instead of threading the request's ctx, the new span becomes a disconnected root — the parent/child relationship is lost and traces fragment. Always start child spans from the incoming ctx and pass the returned ctx downward.

Embedding & go:embed

Observability & Profiling • overdue 12d • rep 0

Quick Reminder

The //go:embed directive bakes files into the compiled binary at build time, so you ship a single self-contained executable — no sidecar assets to deploy. A single file can embed into a string or []byte; one or more files (with globs) embed into an embed.FS, which satisfies fs.FS and plugs straight into http.FileServer or template.ParseFS.

Embedding a string and a filesystem

package main

import (
    "embed"
    "fmt"
    "net/http"
)

//go:embed version.txt
var version string

//go:embed static/*
var staticFS embed.FS

func main() {
    fmt.Println("version:", version)

    // Serve the embedded static/ directory.
    http.Handle("/static/", http.FileServer(http.FS(staticFS)))
    // http.ListenAndServe(":8080", nil)
}
Output (with version.txt containing 1.4.2)
version: 1.4.2
Challenge Exercise

Embed a directory of HTML templates with //go:embed templates/*.tmpl and parse them via template.ParseFS(tmplFS, "templates/*.tmpl"), then render one to os.Stdout. Next, embed a static/ folder and serve it, but strip the prefix so /assets/app.css maps to static/app.css using fs.Sub + http.StripPrefix. What happens at build time if the embed pattern matches no files?

Pitfall: Expecting //go:embed to pick up files outside the package directory or with leading dots. The pattern is relative to the source file's directory, cannot use .. to escape it, and (by default) ignores files beginning with . or _. A non-matching pattern is a compile-time error, which is good — but it surprises people who typo a path.

Section 1 — Language & Standard Library Additions (Go 1.21+)

The min, max, and clear Builtins

Language

Why it matters: Go 1.21 added three builtins — min, max, and clear — that retire a pile of tiny helper functions. They need no import, work generically over any ordered type, and clear finally gives a one-liner for emptying a map or zeroing a slice.

min and max: variadic and generic

min and max take one or more arguments of any ordered type (integers, floats, strings) and return the smallest/largest. They are evaluated like calls but resolved by the compiler, and even work in constant expressions.

package main

import "fmt"

func main() {
    fmt.Println(min(3, 1, 2))             // ints
    fmt.Println(max(3.5, 2.1, 9.9))       // floats
    fmt.Println(min("banana", "apple"))   // strings, lexicographic

    const limit = max(8, 16, 4)           // usable as a constant
    fmt.Println("limit:", limit)
}
Output
1 9.9 apple limit: 16

clear: empty a map, zero a slice

clear deletes every entry from a map (length becomes 0) or sets every element of a slice to its zero value (length is unchanged). Reusing a cleared buffer avoids a fresh allocation.

package main

import "fmt"

func main() {
    m := map[string]int{"a": 1, "b": 2}
    clear(m)
    fmt.Println("map len:", len(m)) // 0

    s := []int{1, 2, 3}
    clear(s)                        // length stays 3
    fmt.Println("slice:", s, "len:", len(s))
}
Output
map len: 0 slice: [0 0 0] len: 3

Common Mistakes

Wrong
// Hand-rolled helper, only for ints,
// shadowed by every other package.
func maxInt(a, b int) int {
    if a > b { return a }
    return b
}
m := maxInt(x, y)
Right
// Builtin: generic, variadic,
// no import, no helper to maintain.
m := max(x, y)
hi := max(a, b, c, d) // any arity

Key Rules

  • min/max accept one or more args of any ordered type and need no import.
  • They work in constant expressions when all arguments are constants.
  • clear(map) removes all keys; clear(slice) zeroes elements but keeps len and cap.
  • For floats, if any argument is NaN the result is NaN (NaN propagates).
Practice Exercise

Write clamp[T cmp.Ordered](v, lo, hi T) T implemented purely with min and max (no if). Then write a function that reuses a single []byte scratch buffer across iterations by calling clear instead of re-allocating, and confirm with -benchmem that allocations drop to zero. Finally, demonstrate the NaN propagation rule for max(math.NaN(), 1.0).

Pitfall: Assuming clear shrinks a slice. It does not change len — it only writes zero values into the existing elements. If you want an empty slice, use s = s[:0] (keep capacity) or s = nil; reach for clear(s) only when you specifically need the old elements zeroed (e.g., to release pointers for GC).
Connections: min/max rely on the same ordering notion formalized by the cmp package's cmp.Ordered constraint, and clear pairs naturally with buffer-reuse patterns like sync.Pool.

The cmp Package: Ordered, Compare & Or

Std Lib

Why it matters: The cmp package (Go 1.21, extended in 1.22) is the small but essential glue for generic ordering. cmp.Ordered is the constraint behind min/max and slices.Sort; cmp.Compare gives the three-way result that slices.SortFunc wants; and cmp.Or collapses fallback chains into a single expression.

Multi-key sorting with Compare + Or

cmp.Compare(a, b) returns -1, 0, or +1. cmp.Or returns its first non-zero argument, so chaining comparisons gives a clean "sort by X, then by Y" tiebreaker without nested ifs.

package main

import (
    "cmp"
    "fmt"
    "slices"
)

type Person struct {
    Name string
    Age  int
}

func main() {
    people := []Person{
        {"Bob", 30}, {"Alice", 30}, {"Carol", 25},
    }
    slices.SortFunc(people, func(a, b Person) int {
        return cmp.Or(
            cmp.Compare(a.Age, b.Age),   // primary key
            cmp.Compare(a.Name, b.Name), // tiebreaker
        )
    })
    fmt.Println(people)
}
Output
[{Carol 25} {Alice 30} {Bob 30}]

cmp.Or for default values

Because cmp.Or returns the first non-zero value, it doubles as a concise "use this, else the fallback" for strings, numbers, or any comparable type.

package main

import (
    "cmp"
    "fmt"
)

func main() {
    var fromFlag string // empty: user passed nothing
    fromEnv := ""       // also empty

    port := cmp.Or(fromFlag, fromEnv, "8080") // first non-empty wins
    fmt.Println("port:", port)

    timeout := cmp.Or(0, 0, 30) // first non-zero int
    fmt.Println("timeout:", timeout)
}
Output
port: 8080 timeout: 30

Common Mistakes

Wrong
// Returning a bool from a SortFunc
// comparator -- wrong signature.
slices.SortFunc(xs, func(a, b T) bool {
    return a.Age < b.Age // does not compile
})
Right
// SortFunc wants an int: -1/0/+1.
slices.SortFunc(xs, func(a, b T) int {
    return cmp.Compare(a.Age, b.Age)
})

Key Rules

  • cmp.Ordered is the constraint for types supporting < <= >= > — use it to bound your own generics.
  • cmp.Compare(a,b) returns -1/0/+1; cmp.Less(a,b) returns a bool when you need one.
  • cmp.Or(vals...) returns the first non-zero value, or the zero value if all are zero.
  • cmp.Compare orders NaN as less than everything, unlike the raw < operator — handy for deterministic float sorts.
Practice Exercise

Sort a slice of Order{Priority int, Created time.Time, ID string} by descending priority, then oldest first, then ID, using a single cmp.Or chain (negate the priority comparison for descending). Then write a generic coalesce[T comparable](vals ...T) T and compare it to cmp.Or. Finally, show how cmp.Compare and the bare < operator differ when sorting a slice containing math.NaN().

Pitfall: Treating cmp.Or as a null-coalescing operator that skips "missing" values. It tests against the type's zero value, so a legitimately-zero entry (e.g., an int count of 0 or an explicit empty string) is treated as "absent" and skipped. When zero is a valid value, use an explicit ok check instead.
Connections: cmp underpins the min/max builtins and the slices/maps packages; its three-way comparator is exactly the shape container/heap and sort.Slice ultimately rely on.

sync.OnceFunc, OnceValue & OnceValues

Concurrency

Why it matters: Go 1.21 added three helpers that wrap the sync.Once boilerplate into a function you can just call. They make thread-safe lazy initialization a one-liner — no package-level Once variable, no manually-shared result field — and they correctly propagate panics.

OnceValue: cache an expensive result

sync.OnceValue(f) returns a function that runs f exactly once and caches its return value; every later call returns the cached value without re-running f, safely under concurrency.

package main

import (
    "fmt"
    "sync"
)

type Config struct{ Port int }

// loadConfig runs its body at most once, on first call.
var loadConfig = sync.OnceValue(func() Config {
    fmt.Println("loading config...") // side effect: happens once
    return Config{Port: 8080}
})

func main() {
    fmt.Println(loadConfig().Port)
    fmt.Println(loadConfig().Port) // cached: no reload
}
Output
loading config... 8080 8080

OnceValues and OnceFunc

sync.OnceValues caches a two-value (T, error) result — perfect for "load once, reuse the same error too." sync.OnceFunc wraps a side-effecting function with no return value.

package main

import (
    "fmt"
    "sync"
)

func main() {
    load := sync.OnceValues(func() ([]byte, error) {
        return []byte("data"), nil
    })
    b, err := load()
    _, _ = load() // same cached (value, error)
    fmt.Println(string(b), err)

    shutdown := sync.OnceFunc(func() { fmt.Println("closing pool") })
    shutdown()
    shutdown() // no-op the second time
}
Output
data <nil> closing pool

Common Mistakes

Wrong
// Manual Once + shared field:
var once sync.Once
var cfg Config
func get() Config {
    once.Do(func() { cfg = load() })
    return cfg
}
Right
// One line, no shared mutable state.
var get = sync.OnceValue(load)
// get() returns the cached Config

Key Rules

  • OnceFunc(f) → func(); OnceValue(f) → func() T; OnceValues(f) → func() (T1, T2).
  • The wrapped function runs at most once even under concurrent calls; later calls return instantly.
  • If f panics, the panic is re-raised on the first call and every subsequent call also panics with the same value.
  • Assign the wrapper to a package-level var for lazy singletons — it captures no external mutable state.
Practice Exercise

Replace a hand-written sync.Once singleton (e.g., a database handle) with sync.OnceValues that returns (*sql.DB, error), so callers get the same connection and the same initialization error. Launch 100 goroutines that all call it and assert with -race that the init body ran exactly once. Then make the init function panic and confirm that every caller observes the panic, not a zero value.

Pitfall: Re-creating the wrapper on every call. sync.OnceValue(f) only memoizes if you call the same returned function repeatedly. Writing sync.OnceValue(load)() inline each time builds a brand-new once-wrapper per call, so load runs every time. Store the wrapper once (package var or struct field) and reuse it.
Connections: These build directly on the sync.Once primitive from the sync package family and are the idiomatic modern way to express lazy initialization that older code did with package-level Once variables.

Section 2 — Networking & Security Tooling

net/netip: Modern IP Address Types

Networking

Why it matters: The classic net.IP is a []byte slice — it allocates, is not comparable, can't be a map key, and has an ambiguous zero value. net/netip (Go 1.18+) replaces it with small, allocation-free value types: Addr, AddrPort, and Prefix are comparable and usable directly as map keys.

Parsing and inspecting an address

netip.Addr is an immutable value. ParseAddr returns an error; MustParseAddr panics on bad input (use it only for constants/tests). Rich predicates like Is4, IsPrivate, and IsLoopback come for free.

package main

import (
    "fmt"
    "net/netip"
)

func main() {
    ip := netip.MustParseAddr("192.168.1.10")
    fmt.Println(ip.Is4(), ip.IsPrivate(), ip.IsLoopback())

    // Addr is comparable -> works as a map key, no allocation.
    seen := map[netip.Addr]int{}
    seen[ip]++
    seen[ip]++
    fmt.Println("hits:", seen[ip])
}
Output
true true false hits: 2

CIDR membership with Prefix

netip.Prefix models a CIDR block; Contains answers "is this address inside the subnet?" — the workhorse of allow-lists and routing tables.

package main

import (
    "fmt"
    "net/netip"
)

func main() {
    private := netip.MustParsePrefix("10.0.0.0/8")

    fmt.Println(private.Contains(netip.MustParseAddr("10.1.2.3"))) // inside
    fmt.Println(private.Contains(netip.MustParseAddr("11.0.0.1"))) // outside

    // AddrPort bundles an address with a port.
    ap := netip.MustParseAddrPort("[2001:db8::1]:443")
    fmt.Println(ap.Addr(), ap.Port())
}
Output
true false 2001:db8::1 443

Common Mistakes

Wrong
// net.IP: a []byte slice. Not comparable,
// can't be a map key, allocates on parse.
ip := net.ParseIP("10.0.0.1")
m := map[string]int{ip.String(): 1} // stringify to key
Right
// netip.Addr: a comparable value type.
ip := netip.MustParseAddr("10.0.0.1")
m := map[netip.Addr]int{ip: 1} // direct key

Key Rules

  • netip.Addr/AddrPort/Prefix are small, immutable, comparable value types — no allocation, usable as map keys.
  • Use ParseAddr (returns error) for untrusted input; reserve MustParseAddr for constants and tests.
  • Prefix.Contains does CIDR membership; Addr.Is4/Is6/IsPrivate/IsLoopback classify addresses.
  • Convert to/from the legacy API with netip.AddrFromSlice and Addr.AsSlice only at the boundary with older net.IP code.
Practice Exercise

Build an IP allow-list checker that loads a set of CIDR strings into []netip.Prefix and reports whether a request's netip.Addr matches any block. Then write a small rate-limiter keyed by netip.Addr (map of address → counter) and note how using the value type avoids the per-request string allocation of net.IP.String(). Handle IPv4-in-IPv6 forms with Addr.Unmap.

Pitfall: Comparing a 4-in-6 mapped address (::ffff:10.0.0.1) directly against a pure IPv4 Addr. They are distinct Addr values and won't be equal, so map lookups miss. Normalize with Addr.Unmap (or Is4In6 checks) before comparing or using addresses as keys.
Connections: netip.Addr's comparability makes it a natural map key alongside the concurrency caches from earlier guides, and it is the recommended address type for any new net/net/http code you write.

govulncheck: Vulnerability Scanning

Security

Why it matters: Knowing a dependency contains a CVE is noisy; knowing your code actually calls the vulnerable function is actionable. govulncheck uses call-graph analysis against the official Go vulnerability database, so it reports only vulnerabilities reachable from your code — dramatically cutting false positives compared to manifest-only scanners.

Install and run

It is a standalone tool, not part of the go command. Install it once, then point it at your packages. ./... scans the whole module.

go install golang.org/x/vuln/cmd/govulncheck@latest

govulncheck ./...            # scan source (call-graph aware)
govulncheck -mode=binary app # scan an already-built binary

Reading the output

Each finding links to a GO-... advisory and shows the exact call stack that reaches the vulnerable symbol — that trace is what tells you the issue is real for your program, not just present in a dependency.

=== Symbol Results ===

Vulnerability #1: GO-2024-2687
    HTTP/2 CONTINUATION flood enables DoS in net/http
  More info: https://pkg.go.dev/vuln/GO-2024-2687
  Standard library
    Found in: net/http@go1.21.4
    Fixed in: net/http@go1.21.9
    Example trace:
      main.go:28:21: yourapp.serve calls http.ListenAndServe
Summary line
Your code is affected by 1 vulnerability from the Go standard library. Fix by upgrading the Go toolchain to go1.21.9 or later.

Common Mistakes

Wrong
# Manifest-only scan: flags every CVE
# in the dependency tree, even unused code.
# Hundreds of "vulns" -> alert fatigue,
# real issues get ignored.
Right
# Call-graph aware: only reports vulns
# your code can actually reach.
govulncheck ./...
# few, high-signal findings to triage

Key Rules

  • Install separately (go install golang.org/x/vuln/cmd/govulncheck@latest); it is not bundled with go.
  • Source mode (./...) is call-graph aware and reports only reachable vulnerabilities; binary mode is coarser.
  • It covers both third-party modules and the standard library / toolchain version you build with.
  • A non-zero exit code on findings makes it a clean CI gate — run it on every PR.
Practice Exercise

Pin an old, known-vulnerable dependency version in a throwaway module, run govulncheck ./..., and read the reported call trace to confirm reachability. Then add the function call that touches the vulnerable symbol and re-run to see the finding change from "imported" to "called." Finally, wire govulncheck ./... into a GitHub Actions step and verify the job fails when a vulnerability is reachable.

Pitfall: Treating a clean govulncheck run as proof of total safety. It only knows about vulnerabilities published in the Go vuln database and analyzes static reachability — it can't see logic bugs, secrets in code, or issues reached only via reflection. Use it as one layer alongside go vet, static analysis, and dependency review, not a substitute for them.
Connections: govulncheck complements module management (it tells you which upgrade to make) and the go vet/staticcheck tooling, rounding out a production-grade CI quality gate.
