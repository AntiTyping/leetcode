TCP Servers

I/O & Networking • overdue (13d) • rep 1

Quick Reminder

A raw TCP server listens with net.Listen("tcp", addr), then loops on ln.Accept(), handing each net.Conn to its own goroutine. A net.Conn is both an io.Reader and io.Writer, so bufio, io.Copy, and encoders all compose with it. Set deadlines (SetReadDeadline) to avoid stuck connections, and close the listener to unblock Accept during graceful shutdown.

Key Pattern

package main

import (
    "bufio"
    "fmt"
    "net"
    "strings"
)

func main() {
    ln, err := net.Listen("tcp", "127.0.0.1:0") // :0 = any free port
    if err != nil {
        panic(err)
    }
    defer ln.Close()

    for {
        conn, err := ln.Accept()
        if err != nil {
            return // listener closed
        }
        go handle(conn) // one goroutine per connection
    }
}

func handle(conn net.Conn) {
    defer conn.Close()
    sc := bufio.NewScanner(conn)
    for sc.Scan() { // read line by line
        line := sc.Text()
        fmt.Fprintf(conn, "echo: %s\n", strings.ToUpper(line))
    }
}
Challenge Exercise

Build a line-based chat broadcast server: keep a map[net.Conn]struct{} guarded by a sync.Mutex, and when any client sends a line, fan it out to every other connection. Add a per-connection read deadline that drops idle clients after 30s, and implement graceful shutdown: on SIGINT close the listener, then close all live connections. Explain why writing to a slow client can block the broadcast loop and how a per-connection buffered send channel fixes it.

#1 Pitfall: Forgetting defer conn.Close() in the handler (leaking file descriptors), or blocking the accept loop with per-connection work instead of launching a goroutine. Also, a shared net.Conn is not safe for concurrent writes from multiple goroutines — serialize writes through a single owner goroutine or a mutex.

Configuration: Environment Variables & envconfig

Production Go • overdue (13d) • rep 1

Quick Reminder

12-factor apps read config from the environment, not baked-in files. os.Getenv returns "" for a missing key; os.LookupEnv distinguishes "unset" from "empty". Libraries like kelseyhightower/envconfig map env vars onto a struct via tags, with defaults, required fields, and type parsing (durations, ints, bools) for free. Load config once at startup into an immutable struct and pass it down — don't call os.Getenv scattered through the codebase.

Key Pattern

package main

import (
    "fmt"
    "time"

    "github.com/kelseyhightower/envconfig"
)

type Config struct {
    Port     int           `envconfig:"PORT" default:"8080"`
    LogLevel string        `envconfig:"LOG_LEVEL" default:"info"`
    Timeout  time.Duration `envconfig:"TIMEOUT" default:"5s"`
    DBURL    string        `envconfig:"DB_URL" required:"true"`
}

func main() {
    var cfg Config
    // Reads APP_PORT, APP_LOG_LEVEL, ... (prefix "APP").
    if err := envconfig.Process("APP", &cfg); err != nil {
        panic(err) // e.g. required DB_URL missing
    }
    fmt.Printf("%+v\n", cfg)
}
Challenge Exercise

Without any third-party library, write a Load() (Config, error) that reads PORT (int, default 8080), DEBUG (bool), and TIMEOUT (a time.Duration parsed with time.ParseDuration), returning a clear error naming the offending variable on a parse failure. Make missing-but-defaulted fields succeed while a required DATABASE_URL errors when unset (use os.LookupEnv). Then explain why loading config into one immutable struct at startup beats calling os.Getenv throughout the code, especially for testability.

#1 Pitfall: Using os.Getenv and treating "" as "unset" — an explicitly empty value is indistinguishable, so a deliberately blank setting silently falls back to a default. Use os.LookupEnv when the distinction matters. Also avoid re-reading env vars at request time; parse once at boot so a misconfiguration fails fast rather than mid-flight.

Structured Logging with log/slog

Production Go • overdue (12d) • rep 1

Quick Reminder

log/slog (Go 1.21) is the standard structured logger: log key/value attributes, not formatted strings, so machines can parse them. A *slog.Logger writes through a slog.Handler — slog.NewJSONHandler for production, slog.NewTextHandler for humans. Handlers carry a level threshold (via slog.HandlerOptions{Level: ...}). Set a process-wide default with slog.SetDefault so the top-level slog.Info helpers use it.

Key Pattern

package main

import (
    "log/slog"
    "os"
)

func main() {
    // JSON handler at Info level, to stderr.
    h := slog.NewJSONHandler(os.Stderr, &slog.HandlerOptions{
        Level: slog.LevelInfo,
    })
    logger := slog.New(h)
    slog.SetDefault(logger) // package-level slog.* now use it

    slog.Info("server started",
        "port", 8080,
        "env", "prod",
    )
    slog.Debug("not shown") // below threshold
    slog.Error("db failed", "err", "timeout", "retry", 3)
}
Output (stderr, JSON)
{"time":"2026-07-04T09:00:00Z","level":"INFO","msg":"server started","port":8080,"env":"prod"} {"time":"2026-07-04T09:00:00Z","level":"ERROR","msg":"db failed","err":"timeout","retry":3}
Challenge Exercise

Configure a logger whose level is driven by a LOG_LEVEL env var (map debug/info/warn/error to the slog.Level constants) using a *slog.LevelVar so you can change it at runtime. Log the same event through both a JSON and a Text handler and compare the output. Then write a helper that logs the duration of an operation as a time.Duration attribute, and explain why passing key/value pairs beats fmt.Sprintf into a single message string for downstream log processors.

#1 Pitfall: Passing an odd number of loose arguments (slog.Info("msg", "key") with no value) — slog logs a !BADKEY entry rather than failing. Prefer the typed slog.Int/slog.String/slog.Any attribute constructors in hot or important paths to keep pairs balanced and avoid per-call allocation of the interface slice.

Memory Management: Escape Analysis & sync.Pool

Production Go • overdue (12d) • rep 1

Quick Reminder

The Go compiler decides via escape analysis whether a value can live on the cheap, per-goroutine stack or must be heap-allocated (adding GC pressure). A value "escapes" when its lifetime outlives the function — returned by pointer, stored in an interface, captured by a closure, or sent on a channel. Inspect decisions with go build -gcflags=-m. When you can't avoid heap churn in a hot path, sync.Pool recycles objects across calls.

Key Pattern

package main

import "fmt"

// x does NOT escape: value copied out, stays on stack.
func stackAlloc() int {
    x := 42
    return x
}

// p ESCAPES: address returned, must live on the heap.
func heapAlloc() *int {
    p := new(int)
    *p = 42
    return p
}

func main() {
    fmt.Println(stackAlloc(), *heapAlloc())
}
// $ go build -gcflags='-m' main.go
//   ./main.go:12:10: new(int) escapes to heap
Challenge Exercise

Write two versions of a function that builds a 1 KB byte buffer per call: one that allocates a fresh make([]byte, 1024) each time, and one that borrows from a sync.Pool and Puts it back (resetting length with buf[:0]). Benchmark both with -benchmem and report allocs/op. Run go build -gcflags=-m on a function that returns a pointer versus a value and explain each escape decision. Finally, describe one case where a closure capturing a large struct forces a heap allocation you didn't expect.

#1 Pitfall: Reaching for sync.Pool everywhere — it only helps for short-lived, frequently-allocated objects in a genuine hot path, and pooled items can be reclaimed by the GC at any time (never assume something is still there). Micro-optimizing before profiling with -gcflags=-m and -benchmem usually costs readability for no measurable win.

Structured Logging: Attrs, With & Groups

Production Go • overdue (11d) • rep 1

Quick Reminder

Beyond basic slog setup, the power is in contextual logging: logger.With("request_id", id) returns a child logger that stamps that attribute on every subsequent line. slog.Group nests related attributes under one key. Use logger.InfoContext(ctx, ...) so a custom handler can pull values (like a trace ID) out of the context.Context. Typed constructors (slog.String, slog.Int, slog.Duration) avoid the loose key/value slice.

Key Pattern

package main

import (
    "log/slog"
    "os"
)

func main() {
    base := slog.New(slog.NewJSONHandler(os.Stdout, nil))

    // Child logger carries request_id on every line.
    reqLog := base.With("request_id", "abc-123")

    reqLog.Info("handling",
        slog.Group("http",
            slog.String("method", "GET"),
            slog.Int("status", 200),
        ),
    )
}
Output
{"time":"...","level":"INFO","msg":"handling","request_id":"abc-123","http":{"method":"GET","status":200}}
Challenge Exercise

Write a middleware that generates a request ID, stores it in the request context.Context, and derives a per-request logger with base.With("request_id", id) that handlers pull from the context. Implement a custom slog.Handler (wrapping NewJSONHandler) whose Handle reads a trace ID out of the context and adds it as an attribute automatically. Then use the ReplaceAttr option to redact a password field, and explain how WithAttrs/WithGroup let a handler pre-format shared attributes once.

#1 Pitfall: Rebuilding the same attributes on every log call instead of creating a With-derived logger once and reusing it — you repeat work and risk inconsistent keys. Also, plain Info ignores any context values; use the *Context variants (InfoContext) when a handler needs to enrich lines from the context.Context.

sync.Pool & Reducing Allocations

Memory & Performance • overdue (11d) • rep 1

Quick Reminder

sync.Pool is a free list of reusable, temporary objects that cuts allocation and GC pressure in hot paths. Get returns a pooled object (calling New if the pool is empty); Put returns it for reuse. Objects may be dropped by the GC between cycles, so a pool is a cache, not a guarantee. The canonical use is pooling *bytes.Buffer or []byte scratch space; always reset a borrowed object before use.

Key Pattern

package main

import (
    "bytes"
    "fmt"
    "sync"
)

var bufPool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func render(name string) string {
    b := bufPool.Get().(*bytes.Buffer)
    b.Reset()                 // MUST reset borrowed state
    defer bufPool.Put(b)      // return it for reuse

    fmt.Fprintf(b, "hello, %s!", name)
    return b.String()         // copy out before Put reclaims it
}

func main() {
    fmt.Println(render("gopher"))
    fmt.Println(render("world"))
}
Output
hello, gopher! hello, world!
Challenge Exercise

Pool []byte scratch buffers for a function that hex-encodes messages: Get a buffer, grow it if too small, use it, then Put it back after copying the result out. Benchmark with -benchmem to show allocs/op dropping versus a fresh make each call. Then demonstrate the classic bug: return a slice that still points into a pooled buffer, Put the buffer, and show the data getting corrupted when the buffer is reused. Explain why you must copy any data you keep beyond the Put.

#1 Pitfall: Retaining a reference to a pooled object (or a slice into it) after Put — the next Get may hand that same object to another goroutine, causing data races and corruption. Also, forgetting to Reset a borrowed bytes.Buffer leaks stale bytes from the previous user into your new output.

Section 1 — Context & Lifecycle Management (Go 1.21+)

context.AfterFunc & context.WithoutCancel (Go 1.21)

Concurrency

Why it matters: a huge amount of concurrency glue is "run this cleanup when the context is cancelled" or "start a background task that must outlive the request that spawned it". Before Go 1.21 you wrote a select { case <-ctx.Done(): ... } goroutine by hand, and detaching from a parent's cancellation meant a bare context.Background() that lost the parent's values. context.AfterFunc and context.WithoutCancel package both patterns correctly.

AfterFunc: run a callback when a context ends

context.AfterFunc(ctx, f) schedules f to run in its own goroutine after ctx is cancelled or times out. It returns a stop function; calling it cancels the registration if f hasn't started yet (returning true if it prevented the call).

package main

import (
    "context"
    "fmt"
    "time"
)

func main() {
    ctx, cancel := context.WithCancel(context.Background())

    // f runs once, in its own goroutine, after ctx ends.
    stop := context.AfterFunc(ctx, func() {
        fmt.Println("cleanup: context ended")
    })

    time.Sleep(10 * time.Millisecond)
    cancel()                       // triggers the AfterFunc
    time.Sleep(10 * time.Millisecond)

    // Stop reports whether it prevented f from running.
    fmt.Println("stopped in time?", stop()) // false: already ran
}
Output
cleanup: context ended stopped in time? false

WithoutCancel: keep the values, drop the cancellation

context.WithoutCancel(ctx) returns a context that carries all of ctx's values but is never cancelled by it. Perfect for a fire-and-forget task (audit log, metrics flush) that must finish even after the originating request returns.

package main

import (
    "context"
    "fmt"
)

type ctxKey string

func main() {
    parent := context.WithValue(context.Background(), ctxKey("user"), "alice")
    parent, cancel := context.WithCancel(parent)

    // Detached: still sees "user", ignores parent cancellation.
    bg := context.WithoutCancel(parent)

    cancel() // parent is now done...
    fmt.Println("parent err:", parent.Err())        // context canceled
    fmt.Println("detached err:", bg.Err())          // 
    fmt.Println("value survives:", bg.Value(ctxKey("user"))) // alice
}
Output
parent err: context canceled detached err: value survives: alice

Common Mistakes

Wrong
// Hand-rolled goroutine that leaks if you forget cleanup,
// and races on shared state.
go func() {
    <-ctx.Done()
    doCleanup() // no way to cancel this registration
}()

// Detaching by losing the parent's values entirely:
bg := context.Background() // trace IDs, user, all GONE
Right
// One call; returns a stop() to unregister.
stop := context.AfterFunc(ctx, doCleanup)
defer stop() // cancel if no longer needed

// Detach cancellation but KEEP request-scoped values.
bg := context.WithoutCancel(ctx)
go flushMetrics(bg) // survives request return

Key Rules

  • context.AfterFunc(ctx, f) calls f exactly once in a new goroutine after ctx is done; if ctx is already done it runs immediately (still in a goroutine).
  • The returned stop unregisters f; it returns true only if it stopped f before it started. Call it to avoid a lingering registration.
  • context.WithoutCancel(ctx) keeps ctx's values but never inherits its cancellation or deadline — give the detached task its own timeout if it needs one.
  • f in AfterFunc runs concurrently: it must be goroutine-safe, and it should be short (offload heavy work rather than blocking the scheduler goroutine).
Practice Exercise

Use context.AfterFunc to implement a "connection reaper": given a per-request ctx, register a callback that closes a net.Conn when the request is cancelled, and verify calling stop() before cancellation prevents the close. Then build a request handler that, on success, kicks off an audit-log write on a context.WithoutCancel(ctx) so it completes even after the HTTP response is sent — but give that detached context its own WithTimeout. Finally, compare AfterFunc with a manual select { case <-ctx.Done(): } goroutine and explain what the stop function buys you.

Pitfall: Assuming WithoutCancel also strips the deadline "for safety" while forgetting the task now has no timeout at all — a detached goroutine with no deadline can run forever and leak. Always wrap a detached context in a fresh WithTimeout. And remember the AfterFunc callback runs in its own goroutine even when the context is already cancelled, so it can race with code that assumed cleanup happens synchronously.
Connections: Builds directly on the Context package (cancellation, deadlines, values) from the concurrency days and pairs with signal.NotifyContext below for shutdown flows; the detach pattern is the correct replacement for the context.Background() hack often seen in fire-and-forget tasks and graceful-shutdown drains.

signal.NotifyContext: Graceful Signal Handling

Std Lib

Why it matters: every long-running server needs to shut down cleanly on SIGINT (Ctrl-C) or SIGTERM (from Kubernetes / systemd) — drain in-flight requests, flush buffers, close the DB. The old recipe wired a chan os.Signal to signal.Notify and blocked on it. signal.NotifyContext (Go 1.16) collapses that into a context that cancels when a signal arrives, so the same ctx.Done() your handlers already watch drives shutdown.

The canonical shutdown skeleton

signal.NotifyContext(parent, sigs...) returns a derived ctx plus a stop function. The context is cancelled on the first matching signal; stop unregisters the handler (restoring default behavior, e.g. so a second Ctrl-C force-quits).

package main

import (
    "context"
    "fmt"
    "os"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    // ctx cancels on SIGINT or SIGTERM.
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    fmt.Println("running; press Ctrl-C to stop")

    select {
    case <-ctx.Done():
        fmt.Println("\nsignal received, shutting down")
    case <-time.After(2 * time.Second):
        fmt.Println("finished normally")
    }
}
Output (no signal within 2s)
running; press Ctrl-C to stop finished normally

Driving an HTTP server's graceful shutdown

Pair it with http.Server.Shutdown: serve in a goroutine, block on ctx.Done(), then give in-flight requests a bounded grace period to finish.

package main

import (
    "context"
    "errors"
    "log"
    "net/http"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    srv := &http.Server{Addr: ":8080"}

    go func() {
        if err := srv.ListenAndServe(); err != nil &&
            !errors.Is(err, http.ErrServerClosed) {
            log.Fatal(err)
        }
    }()

    <-ctx.Done() // wait for signal
    stop()       // restore default: 2nd Ctrl-C force-quits

    shutdownCtx, cancel := context.WithTimeout(
        context.Background(), 10*time.Second)
    defer cancel()
    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Printf("forced shutdown: %v", err)
    }
    log.Println("bye")
}

Common Mistakes

Wrong
// Manual channel plumbing, easy to leak, and no way
// for downstream code to observe cancellation.
sig := make(chan os.Signal, 1)
signal.Notify(sig, syscall.SIGINT)
<-sig
os.Exit(0) // slams the door: no draining, no cleanup
Right
ctx, stop := signal.NotifyContext(context.Background(),
    syscall.SIGINT, syscall.SIGTERM)
defer stop()

<-ctx.Done()
// graceful: drain with a bounded shutdown context
srv.Shutdown(shutdownCtx)

Key Rules

  • signal.NotifyContext(parent, sigs...) returns a context cancelled on the first listed signal — watch ctx.Done() like any other cancellation.
  • Always defer stop() (or call it explicitly): it releases the signal handler and restores default OS behavior, so a second signal can force-terminate a hung shutdown.
  • Use a separate, fresh context.WithTimeout for the actual drain (srv.Shutdown) — the signal context is already cancelled, so reusing it would abort shutdown instantly.
  • Prefer SIGTERM handling for containers (that's what orchestrators send); SIGKILL cannot be caught — nothing you write runs on a kill -9.
Practice Exercise

Write a worker that pulls jobs from a channel and processes them; on SIGINT/SIGTERM, stop accepting new jobs but finish the current one before exiting, using signal.NotifyContext and a sync.WaitGroup. Add the "second signal force-quits" behavior by calling stop() after the first signal. Then wrap an http.Server with a 10-second Shutdown grace window and prove that an in-flight slow request completes while new connections are refused. Explain why you must build the shutdown context from context.Background() rather than the already-cancelled signal context.

Pitfall: Passing the (already cancelled) signal ctx into srv.Shutdown(ctx) — shutdown then returns immediately with a deadline-exceeded error and drops in-flight requests. Build a new timeout context for draining. Also, forgetting defer stop() leaves the signal handler installed, so a stuck graceful shutdown can't be interrupted by an impatient second Ctrl-C.
Connections: The lifecycle bookend to context.AfterFunc/WithoutCancel above and a direct upgrade to the Graceful Shutdown production pattern; the drain step reuses context.WithTimeout, and the detached-task idea from WithoutCancel is how you let post-shutdown flushes complete.

Section 2 — Runtime Introspection & Text Tokenizing

expvar: Exposing Runtime Variables

Std Lib

Why it matters: sometimes you want to expose a few operational counters (requests served, cache hits, queue depth) over HTTP without pulling in a full metrics stack. The standard expvar package publishes named variables as JSON at /debug/vars — zero dependencies, goroutine-safe counters, and it auto-registers Go's memory stats. It's the quickest way to make a service introspectable in a pinch.

Publishing and updating counters

Importing expvar registers an HTTP handler at /debug/vars on the default mux. expvar.NewInt/NewMap/NewFloat create published, atomic variables; mutate them with Add/Set.

package main

import (
    "expvar"
    "fmt"
    "net/http"
    "net/http/httptest"
)

// Registered under these names in /debug/vars.
var (
    hits   = expvar.NewInt("cache_hits")
    byPath = expvar.NewMap("requests_by_path")
)

func main() {
    hits.Add(1)
    hits.Add(1)                 // atomic, goroutine-safe
    byPath.Add("/home", 1)
    byPath.Add("/home", 1)
    byPath.Add("/api", 1)

    // expvar's handler is on http.DefaultServeMux at /debug/vars.
    req := httptest.NewRequest("GET", "/debug/vars", nil)
    rec := httptest.NewRecorder()
    http.DefaultServeMux.ServeHTTP(rec, req)

    fmt.Println(hits.Value())          // 2
    fmt.Println(byPath.Get("/home"))   // 2
}
Output
2 2

Publishing computed values with Func

expvar.Publish(name, expvar.Func(f)) exposes a value computed on each scrape — ideal for gauges like goroutine count or queue length that you don't want to track incrementally.

package main

import (
    "expvar"
    "fmt"
    "runtime"
)

func main() {
    // Evaluated fresh every time /debug/vars is read.
    expvar.Publish("goroutines", expvar.Func(func() any {
        return runtime.NumGoroutine()
    }))

    // Read it back through the exported Var.
    v := expvar.Get("goroutines")
    fmt.Println(v.String() != "") // true: JSON-encoded int
}
Output
true

Common Mistakes

Wrong
// Registering the same name twice PANICS at startup.
var a = expvar.NewInt("hits")
var b = expvar.NewInt("hits") // panic: reuse of "hits"

// A plain int64 counter you mutate with ++ is NOT
// goroutine-safe and won't appear in /debug/vars.
var count int64
count++ // data race under concurrency
Right
// One published var per name; share it.
var hits = expvar.NewInt("hits")
hits.Add(1) // atomic

// For a derived/read-only value, publish a Func.
expvar.Publish("queue_depth", expvar.Func(func() any {
    return q.Len()
}))

Key Rules

  • Blank-importing or using expvar auto-registers /debug/vars on http.DefaultServeMux and publishes memstats + cmdline for free.
  • NewInt/NewFloat/NewString/NewMap return goroutine-safe published vars; names must be unique or registration panics.
  • Use expvar.Func for values computed at scrape time (gauges); use counters (Add) for values you increment as events happen.
  • It's a lightweight introspection tool, not a full metrics system — there's no histograms, labels, or aggregation; reach for Prometheus/OpenTelemetry when you need those.
Practice Exercise

Instrument a tiny HTTP server: publish a requests_total *expvar.Int incremented in a middleware, and a requests_by_status *expvar.Map keyed by status code. Add an expvar.Func gauge for runtime.NumGoroutine(). Hit a few endpoints, then curl /debug/vars and confirm your counters and Go's memstats both appear as JSON. Finally, discuss why exposing /debug/vars publicly is a mild information-disclosure risk and how you'd gate it behind auth or a private port.

Pitfall: Registering a name more than once (or importing two packages that both register the same name) — expvar panics at init, taking down the whole binary. Also, mounting the default mux (which now carries /debug/vars) on a public interface leaks internal counters and memory stats; keep debug endpoints on a separate, internal listener.
Connections: A companion to net/http/pprof (both live under /debug/ on the default mux) and a stepping stone toward the OpenTelemetry metrics you saw in the observability days; the Func gauge pattern reads the same runtime data that runtime/metrics below exposes far more richly.

runtime/metrics: The Stable Metrics Interface

Std Lib

Why it matters: runtime.ReadMemStats is the old way to inspect the GC and heap, but it stops the world and its fields are frozen for compatibility. runtime/metrics (Go 1.16+) is the modern, stable, low-overhead replacement: a self-describing catalog of metrics (GC pauses, heap size, goroutine count, scheduler latency) addressed by string names like /gc/heap/allocs:bytes. It's what production dashboards and the Go team recommend for observability.

Reading specific metrics by name

You build a []metrics.Sample with the names you want, then call metrics.Read to fill in their values. Each value has a Kind (Uint64, Float64, Float64Histogram) you switch on.

package main

import (
    "fmt"
    "runtime/metrics"
)

func main() {
    // Ask for two well-known metrics by name.
    samples := []metrics.Sample{
        {Name: "/memory/classes/heap/objects:bytes"},
        {Name: "/sched/goroutines:goroutines"},
    }
    metrics.Read(samples) // fills in .Value

    for _, s := range samples {
        switch s.Value.Kind() {
        case metrics.KindUint64:
            fmt.Printf("%s = %d\n", s.Name, s.Value.Uint64())
        case metrics.KindFloat64:
            fmt.Printf("%s = %f\n", s.Name, s.Value.Float64())
        }
    }
}
Output (values vary)
/memory/classes/heap/objects:bytes = 195840 /sched/goroutines:goroutines = 1

Discovering the catalog & reading a histogram

metrics.All() returns every supported metric with a description — the interface is self-documenting. Latency/pause metrics come back as *metrics.Float64Histogram (buckets + counts).

package main

import (
    "fmt"
    "runtime/metrics"
)

func main() {
    all := metrics.All() // catalog of supported metrics
    fmt.Println("total metrics:", len(all) > 0) // true

    s := []metrics.Sample{{Name: "/gc/pauses:seconds"}}
    metrics.Read(s)
    if s[0].Value.Kind() == metrics.KindFloat64Histogram {
        h := s[0].Value.Float64Histogram()
        var total uint64
        for _, c := range h.Counts {
            total += c // number of recorded GC pauses
        }
        fmt.Println("have pause histogram:", len(h.Buckets) > 0)
        fmt.Println("counts collected:", total >= 0)
    }
}
Output
total metrics: true have pause histogram: true counts collected: true

Common Mistakes

Wrong
// Typo / unsupported name -> value Kind is KindBad,
// silently ignored if you don't check.
s := []metrics.Sample{{Name: "/memory/heap:bytes"}}
metrics.Read(s)
v := s[0].Value.Uint64() // panics: wrong Kind (KindBad)

// Calling runtime.ReadMemStats in a hot loop:
// it stops the world every call.
runtime.ReadMemStats(&m) // expensive, per request
Right
// Verify the metric exists / check its Kind first.
s := []metrics.Sample{{Name: "/memory/classes/heap/objects:bytes"}}
metrics.Read(s)
if s[0].Value.Kind() == metrics.KindUint64 {
    use(s[0].Value.Uint64())
}
// Sample periodically (e.g. every 10s), reuse the slice.

Key Rules

  • Metric names are stable strings with a :unit suffix (e.g. /gc/heap/allocs:bytes); discover them via metrics.All(), which also carries human descriptions.
  • Always switch on Value.Kind() (KindUint64, KindFloat64, KindFloat64Histogram, KindBad) before reading — the wrong accessor panics.
  • Prefer runtime/metrics over runtime.ReadMemStats: it's lower overhead, doesn't stop the world for most metrics, and won't freeze its schema.
  • Build the []metrics.Sample once and reuse it across periodic reads; sample on an interval, not per request.
Practice Exercise

Write a background goroutine that samples /memory/classes/heap/objects:bytes, /sched/goroutines:goroutines, and /gc/heap/allocs:bytes every few seconds and logs them with slog. Then enumerate metrics.All() and print each metric's name, kind, and description to build a reference of what your Go version exposes. Extract the p99 from the /gc/pauses:seconds histogram by walking its buckets and cumulative counts. Finally, compare a metric read against runtime.ReadMemStats and explain why the latter's stop-the-world cost makes it unsuitable for continuous scraping.

Pitfall: Reading a sample with the wrong typed accessor — a mistyped or unsupported name yields KindBad, and calling .Uint64()/.Float64() on it panics. Always branch on Value.Kind(). Also don't assume a metric exists on every Go version; the catalog grows over time, so validate names against metrics.All() rather than hard-coding and hoping.
Connections: The rich counterpart to the expvar gauges above and the runtime data behind pprof; feed these samples into your OpenTelemetry/Prometheus exporters, and use the heap/alloc metrics alongside the escape-analysis and sync.Pool techniques from today's memory-management review to confirm optimizations actually reduced allocations.

text/scanner: Tokenizing Source-Like Text

Std Lib

Why it matters: when you need to tokenize input that looks like a programming language or config DSL — identifiers, numbers, quoted strings, comments — hand-rolling a lexer is fiddly and bufio.Scanner only splits on lines/words. text/scanner is a ready-made tokenizer with Go-like lexical rules: it recognizes idents, ints, floats, chars, strings, and comments, tracks line/column positions, and lets you tune what counts as a token via a bitmask.

Scanning tokens with positions

Initialize a scanner.Scanner from an io.Reader, then loop calling Scan() until scanner.EOF. TokenText() returns the current token's text; the scanner carries its Position (line/column).

package main

import (
    "fmt"
    "strings"
    "text/scanner"
)

func main() {
    const src = `x = 42 + foo("hi")`
    var s scanner.Scanner
    s.Init(strings.NewReader(src))
    s.Filename = "example"

    for tok := s.Scan(); tok != scanner.EOF; tok = s.Scan() {
        fmt.Printf("%s: %q (%s)\n",
            s.Position, s.TokenText(), scanner.TokenString(tok))
    }
}
Output
example:1:1: "x" (Ident) example:1:3: "=" ("=") example:1:5: "42" (Int) example:1:8: "+" ("+") example:1:10: "foo" (Ident) example:1:13: "(" ("(") example:1:14: "\"hi\"" (String) example:1:18: ")" (")")

Tuning the token modes

The Mode bitmask selects which token classes to recognize; Whitespace controls which bytes are skipped. Here we scan only identifiers and integers and treat everything else as single runes.

package main

import (
    "fmt"
    "strings"
    "text/scanner"
)

func main() {
    var s scanner.Scanner
    s.Init(strings.NewReader("width: 100 height: 250"))
    // Recognize idents and ints only.
    s.Mode = scanner.ScanIdents | scanner.ScanInts

    var idents, ints int
    for tok := s.Scan(); tok != scanner.EOF; tok = s.Scan() {
        switch tok {
        case scanner.Ident:
            idents++
        case scanner.Int:
            ints++
        }
    }
    fmt.Printf("idents=%d ints=%d\n", idents, ints)
}
Output
idents=2 ints=2

Common Mistakes

Wrong
// Forgetting to Init -> zero scanner reads from nil,
// Scan returns EOF immediately (silent no-op).
var s scanner.Scanner
for s.Scan() != scanner.EOF { /* never runs */ }

// Assuming text/scanner parses YOUR grammar. It only
// TOKENIZES with Go-like rules; nesting/precedence is
// still your job.
Right
var s scanner.Scanner
s.Init(strings.NewReader(src)) // always Init first
// Optional: custom error hook instead of stderr.
s.Error = func(_ *scanner.Scanner, msg string) {
    fmt.Println("lex error:", msg)
}
for tok := s.Scan(); tok != scanner.EOF; tok = s.Scan() {
    // build your parser on top of these tokens
}

Key Rules

  • Always call s.Init(reader) before scanning; the zero Scanner has no source and returns EOF immediately.
  • Loop for tok := s.Scan(); tok != scanner.EOF; tok = s.Scan(); read the token via s.TokenText() and its location via s.Position.
  • Control recognition with the Mode bitmask (ScanIdents, ScanInts, ScanFloats, ScanStrings, ScanComments, …) and skipped bytes with Whitespace.
  • It's a tokenizer, not a parser: it hands you a token stream with Go-like lexical rules; grammar, precedence, and semantics are up to you (or use go/scanner for real Go source).
Practice Exercise

Write a tiny calculator lexer with text/scanner that tokenizes "3 + 4 * (2 - 1)" into numbers and operators, then feed the token stream into a recursive-descent parser that respects precedence and evaluates the result. Use s.Position to report a helpful "unexpected token at line:col" error for malformed input like "3 + * 4". Then switch the Mode to also capture ScanComments and skip #-style comments. Finally, contrast text/scanner with go/scanner and bufio.Scanner, explaining when each is the right tool.

Pitfall: Expecting text/scanner to understand a non-Go grammar — it applies Go-like lexical conventions (e.g. how it treats // comments, numeric literals, and string escapes), so a DSL with different rules may tokenize surprisingly. Configure Mode deliberately, and for input that isn't remotely Go-like, a custom bufio.Scanner split function or a hand-written lexer may be clearer than fighting the defaults.
Connections: Shares the io.Reader source model with bufio.Scanner (line/word splitting) and the UTF-8 rune handling from the unicode/utf8 concept; it's the informal cousin of go/scanner+go/ast used for parsing actual Go source, and its position tracking mirrors what regexp and CSV parsing give you for error reporting.
