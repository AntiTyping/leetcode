Error Handling

Go Foundations • due today • rep 3

Quick Reminder

An error is any value implementing interface{ Error() string }. Idiomatic Go returns errors as the last value and checks them immediately with if err != nil. Create them with errors.New or fmt.Errorf; the %w verb wraps an error so callers can later unwrap it. errors.Is(err, target) walks the wrap chain to match a sentinel value; errors.As(err, &dst) walks it to find a value of a concrete type. Prefer wrapping with context at each layer over discarding and re-creating.

Key Pattern

package main

import (
    "errors"
    "fmt"
)

var ErrNotFound = errors.New("not found") // sentinel

type ValidationError struct{ Field string }

func (e *ValidationError) Error() string {
    return "invalid field: " + e.Field
}

func lookup(id int) error {
    if id == 0 {
        return fmt.Errorf("lookup %d: %w", id, ErrNotFound) // wrap
    }
    return &ValidationError{Field: "id"}
}

func main() {
    err := lookup(0)
    fmt.Println(errors.Is(err, ErrNotFound)) // true (unwraps %w)

    var ve *ValidationError
    fmt.Println(errors.As(lookup(1), &ve))   // true
    fmt.Println(ve.Field)                     // id
}
Output
true true id
Challenge Exercise

Define a sentinel ErrClosed and a struct error *QueryError{Query string; Err error} whose Unwrap() returns its inner Err. Wrap ErrClosed inside a QueryError, wrap that again with fmt.Errorf("run: %w", ...), and show that errors.Is(top, ErrClosed) is still true and errors.As(top, &qe) recovers the QueryError. Then combine two independent failures with errors.Join and explain how errors.Is handles the multi-error tree.

#1 Pitfall: Using %v instead of %w when you meant to wrap — the message looks identical but the chain is broken, so errors.Is/errors.As can no longer see the underlying error. Also avoid comparing error strings (err.Error() == "not found"); compare against a sentinel with errors.Is instead.

First-class Functions

Functions & Control Flow • due today • rep 3

Quick Reminder

Functions in Go are values: you can store them in variables, pass them as arguments, return them, and put them in slices or maps. A closure is a function literal that captures variables from its enclosing scope by reference — it keeps them alive and sees their later mutations. Higher-order functions (take or return functions) power callbacks, strategies, middleware, and lazy generators.

Key Pattern

package main

import "fmt"

// Higher-order: returns a closure capturing next.
func counter(start int) func() int {
    next := start
    return func() int {
        n := next
        next++
        return n
    }
}

// Function as a parameter (a strategy).
func apply(xs []int, f func(int) int) []int {
    out := make([]int, len(xs))
    for i, x := range xs {
        out[i] = f(x)
    }
    return out
}

func main() {
    c := counter(10)
    fmt.Println(c(), c(), c()) // 10 11 12

    double := func(x int) int { return x * 2 }
    fmt.Println(apply([]int{1, 2, 3}, double)) // [2 4 6]
}
Output
10 11 12 [2 4 6]
Challenge Exercise

Write a memoize(f func(int) int) func(int) int that caches results in a captured map. Then build a tiny event system: a map[string][]func(payload string) where on(event, handler) registers callbacks and emit(event, payload) invokes them all. Finally, demonstrate the classic closure-capture bug by spawning goroutines in a loop and explain how Go 1.22's per-iteration loop variables changed the outcome versus taking x := x yourself.

#1 Pitfall: Assuming a closure captures a copy of the loop variable. Before Go 1.22 a single variable was shared across iterations, so all deferred/goroutine closures saw the final value. In Go 1.22+ each iteration gets its own variable, but if you build for an older language version the old sharing still bites — capture explicitly when in doubt.

Defer, Panic, Recover

Functions & Control Flow • due today • rep 3

Quick Reminder

defer schedules a call to run when the surrounding function returns, in LIFO order — ideal for cleanup (Close, Unlock). Deferred arguments are evaluated immediately, but the call itself runs last. panic unwinds the stack running deferred calls; recover, called inside a deferred function, stops the unwinding and returns the panic value. Use panic for truly unrecoverable states, not ordinary error flow. A deferred closure can even mutate a named return value.

Key Pattern

package main

import "fmt"

func safeDivide(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // set named return
        }
    }()
    result = a / b // panics on b == 0
    return result, nil
}

func main() {
    fmt.Println(safeDivide(10, 2)) // 5 
    fmt.Println(safeDivide(10, 0)) // 0 recovered: runtime error...

    for i := 0; i < 3; i++ {
        defer fmt.Print(i, " ") // args eval now; runs LIFO at return
    }
    fmt.Println("counting down:")
}
Output
5 0 recovered: runtime error: integer divide by zero counting down: 2 1 0
Challenge Exercise

Write a withLock(mu *sync.Mutex, f func()) helper that locks, defer mu.Unlock()s, and runs f even if it panics. Then write a goroutine-safe guard(fn func() error) (err error) that turns any panic in fn into an error via a deferred recover. Explain why a panic in one goroutine is not caught by a recover in another, and why deferred arguments being evaluated eagerly matters when you write defer f(expensiveCall()).

#1 Pitfall: Calling recover() directly (not inside a deferred function) — it returns nil and does nothing. Also, deferring inside a tight loop accumulates calls until the function returns, which can hold resources far longer than expected; move the loop body into its own function so each defer fires per iteration.

Advanced HTTP Patterns: chi Router

Production Architecture • overdue • rep 1

Quick Reminder

chi is a lightweight, net/http-compatible router: handlers are plain http.HandlerFuncs and middleware are plain func(http.Handler) http.Handler. It adds URL parameters (chi.URLParam(r, "id")), composable middleware stacks (r.Use(...)), sub-routers (r.Route/r.Mount), and per-group middleware via r.Group. Because it builds on the standard interfaces, everything you learned about http.Handler and middleware chains still applies.

Key Pattern

package main

import (
    "net/http"

    "github.com/go-chi/chi/v5"
    "github.com/go-chi/chi/v5/middleware"
)

func main() {
    r := chi.NewRouter()
    r.Use(middleware.Logger)               // stack-wide middleware

    r.Route("/users", func(r chi.Router) { // sub-router
        r.Get("/", listUsers)
        r.Get("/{id}", getUser)            // URL param
    })

    http.ListenAndServe(":8080", r)
}

func listUsers(w http.ResponseWriter, r *http.Request) {
    w.Write([]byte("all users"))
}

func getUser(w http.ResponseWriter, r *http.Request) {
    id := chi.URLParam(r, "id") // pull the path variable
    w.Write([]byte("user " + id))
}
Challenge Exercise

Build a router with a public /health route and an authenticated /api group: use r.Group to attach an auth middleware only to the API routes, and r.Route("/api/posts", ...) with GET /{id}, POST /, and DELETE /{id}. Add a middleware that injects a request ID into the context.Context and read it in a handler. Then compare chi's {id} params with Go 1.22's enhanced net/http ServeMux patterns and explain when the standard mux is now enough.

#1 Pitfall: Registering routes or calling r.Use after the router is already serving, or mixing r.Use (applies to routes defined afterward) with the expectation that it retroactively wraps earlier routes. Declare middleware before the routes it should cover, and use r.Group/r.With for per-route stacks rather than one giant global chain.

regexp: Regular Expressions

I/O & Networking • overdue • rep 1

Quick Reminder

Go's regexp uses the RE2 engine: linear-time matching with no catastrophic backtracking (and therefore no backreferences or lookahead). Compile a pattern once with regexp.MustCompile (panics on a bad pattern at init) or regexp.Compile (returns an error). The common methods come in a naming grid: Find/FindAll, with String, Submatch, and Index variants. Named groups (?P<name>...) pair with SubexpNames().

Key Pattern

package main

import (
    "fmt"
    "regexp"
)

// Compile once (package level or with sync.Once), reuse everywhere.
var dateRe = regexp.MustCompile(`(?P\d{4})-(?P\d{2})-(?P\d{2})`)

func main() {
    s := "log 2026-07-03 event"
    fmt.Println(dateRe.MatchString(s)) // true

    m := dateRe.FindStringSubmatch(s)
    fmt.Println(m[0]) // 2026-07-03 (whole match)
    fmt.Println(m[1]) // 2026        (group 1 = y)

    // Map named groups.
    names := dateRe.SubexpNames() // ["", "y", "m", "d"]
    for i, name := range names {
        if name != "" {
            fmt.Printf("%s=%s ", name, m[i])
        }
    }
    fmt.Println()

    out := dateRe.ReplaceAllString(s, "$y/$m/$d")
    fmt.Println(out) // log 2026/07/03 event
}
Output
true 2026-07-03 2026 y=2026 m=07 d=03 log 2026/07/03 event
Challenge Exercise

Write a log-line parser: given "127.0.0.1 - GET /path 200 13ms", use a single named-group regexp to extract IP, method, path, status, and latency into a struct. Use FindAllStringSubmatch to parse a whole multi-line log. Then benchmark compiling inside the hot loop versus once at package scope, and explain why RE2 guarantees linear time and what that means for accepting untrusted patterns (ReDoS-safe) compared with PCRE engines.

#1 Pitfall: Calling regexp.MustCompile (or Compile) inside a hot function instead of once at package level — compilation is expensive and dominates the cost. Also remember Go string literals: use raw backtick strings for patterns so \d isn't interpreted as a Go escape, and reach for strings functions when you don't actually need a regex.

database/sql

I/O & Networking • overdue • rep 1

Quick Reminder

database/sql is a driver-agnostic layer: you import a driver for its side effect (_ "github.com/mattn/go-sqlite3") and talk to an *sql.DB, which is a connection pool, not a single connection — keep one for the app's lifetime. Use QueryContext/QueryRowContext for reads, ExecContext for writes, and always pass placeholders (? or $1) rather than string-concatenating — that's your SQL-injection defense. Scan nullable columns into sql.NullString et al., and close every *sql.Rows.

Key Pattern

package main

import (
    "context"
    "database/sql"
    "fmt"

    _ "github.com/mattn/go-sqlite3" // driver, registered via init()
)

type User struct {
    ID   int
    Name string
}

func getUser(ctx context.Context, db *sql.DB, id int) (User, error) {
    var u User
    // Placeholder ? prevents SQL injection.
    row := db.QueryRowContext(ctx,
        "SELECT id, name FROM users WHERE id = ?", id)
    if err := row.Scan(&u.ID, &u.Name); err != nil {
        if err == sql.ErrNoRows {
            return u, fmt.Errorf("user %d: %w", id, err)
        }
        return u, err
    }
    return u, nil
}
Challenge Exercise

Open an in-memory SQLite DB, create a users table, and write insertUser (using ExecContext + LastInsertId) and listUsers (iterating *sql.Rows with rows.Next()/rows.Scan, then checking rows.Err()). Wrap two inserts in a transaction with db.BeginTx, commit on success, and defer tx.Rollback() as a safety net. Configure SetMaxOpenConns/SetConnMaxLifetime and explain why *sql.DB is safe for concurrent use while a single *sql.Conn or *sql.Tx is not.

#1 Pitfall: Building SQL with fmt.Sprintf and user input (injection), or forgetting to rows.Close() / check rows.Err() — leaked rows hold a pooled connection until the GC finalizer runs, starving the pool. Always use placeholders, defer rows.Close(), and treat sql.ErrNoRows from QueryRow as an expected "not found", not a hard failure.

Section 1 — Modern Go Language Features (Go 1.21+)

min, max & clear: New Universe Builtins (Go 1.21)

Language

Why it matters: for years every Go codebase hand-rolled an intMin helper because the language had none. Go 1.21 added min, max, and clear as universe-scope built-in functions — no import, generic over any ordered type. min/max take one or more arguments and return the smallest/largest; clear empties a map or zeroes a slice in place. They replace a whole category of boilerplate and subtle off-by-one helpers.

min and max over ordered values

They are variadic and work on any type whose values are ordered (integers, floats, strings). Mixed constant types are fine as long as they're compatible.

package main

import "fmt"

func main() {
    fmt.Println(min(3, 1, 2))       // 1
    fmt.Println(max(3, 1, 2))       // 3
    fmt.Println(min(4.5, 2.0))      // 2
    fmt.Println(max("apple", "pear")) // pear (lexicographic)

    // Clamp a value into [lo, hi] with no helper function.
    lo, hi, x := 0, 10, 15
    clamped := max(lo, min(x, hi))
    fmt.Println(clamped)            // 10
}
Output
1 3 2 pear 10

clear on maps and slices

clear(m) deletes every key from a map (length becomes 0). clear(s) sets every element of a slice to the zero value without changing its length — useful for reusing a buffer while dropping references so the GC can reclaim them.

package main

import "fmt"

func main() {
    m := map[string]int{"a": 1, "b": 2}
    clear(m)
    fmt.Println(len(m)) // 0 (all keys removed)

    s := []string{"x", "y", "z"}
    clear(s)            // zeroes elements; len stays 3
    fmt.Println(len(s), s) // 3 [  ]

    // Idiom: drop pointer references before pooling a slice.
    ptrs := []*int{new(int), new(int)}
    clear(ptrs) // now [nil nil] -> pointees are collectable
    fmt.Println(ptrs[0] == nil) // true
}
Output
0 3 [ ] true

Common Mistakes

Wrong
// clear on a slice does NOT shorten it.
s := []int{1, 2, 3}
clear(s)
// s is [0 0 0], not []! len(s) == 3 still.
if len(s) == 0 { /* never runs */ }

// min/max on a map's NaN floats is order-undefined
_ = max(math.NaN(), 1.0) // result is NaN, surprising
Right
// To empty a slice, reslice to zero length.
s := []int{1, 2, 3}
s = s[:0] // len 0, keeps capacity for reuse

// clear when you specifically want to zero elements
// (e.g. free pointers) but keep the length.
buf := make([]*T, n)
clear(buf) // drop references, reuse buf

Key Rules

  • min/max are builtins (Go 1.21+), variadic, and generic over ordered types — no import, no hand-written helper.
  • They evaluate all arguments; for exactly two args they're the direct replacement for the old if a < b helper.
  • clear(map) removes all keys (len → 0); clear(slice) zeroes elements but keeps the length — reslice with s[:0] to actually shrink.
  • With floating-point NaN, min/max follow the spec's NaN rules — sanitize inputs if NaN is possible.
Practice Exercise

Rewrite an old codebase helper set (IntMin, IntMax, Clamp) to use the builtins, and write a generic clamp[T cmp.Ordered](x, lo, hi T) T for the cases where you still want a named function. Then benchmark clear(m) versus a for k := range m { delete(m, k) } loop and versus reassigning m = make(map[K]V), explaining when reusing the cleared map (preserving capacity) beats allocating a fresh one. Finally, show a slice-buffer pool that uses clear to avoid leaking pointers.

Pitfall: Expecting clear(slice) to make the slice empty like clear(map) makes the map empty. On a slice it zeroes in place and the length is unchanged — a common source of "why is my slice still length 3?" bugs. Use s = s[:0] (or s = nil) to empty a slice; use clear only when you deliberately want zeroed elements at the same length.
Connections: Pairs with the generic cmp.Ordered constraint and the slices/maps packages from earlier days; clear complements the sync.Pool allocation-reduction pattern, and min/max often replace comparator helpers used with slices.SortFunc.

sync.OnceFunc, OnceValue & OnceValues (Go 1.21)

Std Lib

Why it matters: the classic var once sync.Once; once.Do(func(){...}) pattern is everywhere — lazy initialization of an expensive singleton, one-time setup, cached config. Go 1.21 added three helpers that wrap that boilerplate into a returned function: sync.OnceFunc, sync.OnceValue, and sync.OnceValues. They give you a goroutine-safe closure that runs the underlying work exactly once and caches nothing-or-the-result for all later calls.

OnceValue: lazily compute and cache a result

sync.OnceValue(f) returns a func() T; the first call runs f, every later call returns the same cached value — safe under concurrency, with no Once variable to manage.

package main

import (
    "fmt"
    "sync"
)

// f runs at most once, even across many goroutines.
var config = sync.OnceValue(func() map[string]string {
    fmt.Println("loading config...") // prints only once
    return map[string]string{"env": "prod"}
})

func main() {
    fmt.Println(config()["env"]) // loading config... then prod
    fmt.Println(config()["env"]) // prod (cached, no reload)

    var wg sync.WaitGroup
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func() { defer wg.Done(); _ = config() }()
    }
    wg.Wait() // still only one "loading config..." total
}
Output
loading config... prod prod

OnceFunc and OnceValues

OnceFunc wraps a side-effecting function with no result; OnceValues caches a two-value return — perfect for a (T, error) initializer.

package main

import (
    "fmt"
    "sync"
)

var setup = sync.OnceFunc(func() {
    fmt.Println("one-time setup")
})

var load = sync.OnceValues(func() (int, error) {
    return 42, nil // computed once, cached with its error
})

func main() {
    setup()
    setup() // no second print

    v, err := load()
    fmt.Println(v, err) // 42 
    v2, _ := load()
    fmt.Println(v2)     // 42 (same cached value)
}
Output
one-time setup 42 42

Common Mistakes

Wrong
// Recreating the wrapper each call defeats "once":
func getConfig() Config {
    // new OnceValue every call -> f runs every time
    return sync.OnceValue(loadConfig)()
}

// Panicking f: the panic is re-thrown on EVERY
// later call, but you assumed it retries.
var x = sync.OnceValue(mustLoad) // mustLoad panics
Right
// Build the wrapper ONCE (package var or in init).
var getConfig = sync.OnceValue(loadConfig)

// Prefer OnceValues to carry an error instead of
// panicking, so callers can handle failure.
var getConfig2 = sync.OnceValues(func() (Config, error) {
    return loadConfig()
})

Key Rules

  • Create the wrapper once (a package var or in init) and call the returned function many times — not a fresh wrapper per call.
  • OnceFunc → no result; OnceValue → one cached value; OnceValues → two (typically (T, error)).
  • All are goroutine-safe: concurrent first callers block until f finishes, then share the result.
  • If f panics, the wrapper caches the panic and re-panics on every subsequent call — it does not retry. Prefer returning an error for recoverable failures.
Practice Exercise

Replace a sync.Once-guarded lazy database handle with sync.OnceValues(func() (*sql.DB, error) {...}) and show that concurrent callers share one pool. Then write a small experiment proving the panic-caching behavior: wrap a function that panics, call the wrapper twice inside recover blocks, and observe both calls panic. Finally, compare OnceValue against a plain package-level var x = expensive() initializer and explain when lazy (deferred to first use) wins over eager package initialization.

Pitfall: Calling sync.OnceValue(f)() inline on every access — because you construct a brand-new wrapper each time, f runs every time and you get none of the caching or "once" guarantee. Store the returned function in a long-lived variable. Also beware that a panicking f is cached: every future call re-panics rather than retrying.
Connections: A modern, ergonomic layer over sync.Once from the concurrency days; pairs with lazy-singleton and configuration-loading patterns, and complements init functions by deferring cost until first use rather than at program start.

Section 2 — Text, Hashing & Testing Standard Library

unicode & unicode/utf8: Correct Character Handling

Std Lib

Why it matters: a Go string is a read-only sequence of bytes (UTF-8 encoded), not characters. Indexing s[i] gives you a byte, and len(s) counts bytes — both wrong for multibyte text like "café" or emoji. The unicode/utf8 package decodes bytes into runes (Unicode code points), and unicode classifies them (IsLetter, IsDigit, IsSpace, ToUpper). Getting this right is the difference between code that works on ASCII and code that works on the world.

Bytes vs runes

utf8.RuneCountInString counts characters; a for range over a string yields (byteIndex, rune) pairs, decoding UTF-8 for you.

package main

import (
    "fmt"
    "unicode/utf8"
)

func main() {
    s := "café" // "café" — é is 2 bytes in UTF-8

    fmt.Println(len(s))                    // 5 bytes
    fmt.Println(utf8.RuneCountInString(s)) // 4 runes

    // range decodes runes; i is the BYTE offset of each.
    for i, r := range s {
        fmt.Printf("%d:%c ", i, r)
    }
    fmt.Println()

    // Decode the first rune manually.
    r, size := utf8.DecodeRuneInString(s)
    fmt.Printf("%c is %d byte(s)\n", r, size)
}
Output
5 4 0:c 1:a 2:f 3:é c is 1 byte(s)

Classifying and transforming runes

The unicode package answers "is this a letter/digit/space?" and case-folds correctly across scripts. Convert to []rune when you need true random access or reversal.

package main

import (
    "fmt"
    "unicode"
)

func main() {
    letters, digits := 0, 0
    for _, r := range "Go 1.21!" {
        switch {
        case unicode.IsLetter(r):
            letters++
        case unicode.IsDigit(r):
            digits++
        }
    }
    fmt.Println(letters, digits) // 2 3

    // Reverse by runes so multibyte chars stay intact.
    rs := []rune("café")
    for i, j := 0, len(rs)-1; i < j; i, j = i+1, j-1 {
        rs[i], rs[j] = rs[j], rs[i]
    }
    fmt.Println(string(rs)) // éfac
}
Output
2 3 éfac

Common Mistakes

Wrong
// Byte indexing splits multibyte runes -> garbage.
s := "café"
first := s[0:1]          // "c" (ok by luck)
last := s[len(s)-1:]     // half of é -> invalid UTF-8
n := len(s)              // 5, not 4 "characters"
for i := 0; i < len(s); i++ {
    fmt.Printf("%c", s[i]) // prints bytes, mojibake on é
}
Right
s := "café"
n := utf8.RuneCountInString(s) // 4 real chars
for _, r := range s {          // decodes runes
    fmt.Printf("%c", r)
}
// Random access / reversal: convert once.
rs := []rune(s)
last := string(rs[len(rs)-1])  // "é" intact

Key Rules

  • len(s) and s[i] are byte operations; use for range or utf8.RuneCountInString/DecodeRuneInString for runes.
  • for i, r := range s gives the byte offset i and decoded rune r — the idiomatic way to walk text.
  • Convert to []rune for random access, indexing, or reversal; it costs an allocation but keeps multibyte characters whole.
  • Use unicode.IsLetter/IsDigit/IsSpace/IsPunct and unicode.ToUpper/ToLower for correct, script-aware classification instead of ASCII range checks.
Practice Exercise

Write reverse(s string) string that reverses by runes (test it on "Héllo, 世界") and a truncate(s string, n int) string that cuts to at most n runes without splitting a character, appending "…" if truncated. Then write a validator using utf8.ValidString that rejects malformed input, and a slugify that lowercases with unicode.ToLower and keeps only letters/digits. Explain why utf8.RuneCountInString is O(n) and when you'd cache a []rune instead of recounting.

Pitfall: Treating string length or indexing as character-based. len("café") == 5, and slicing at an arbitrary byte offset can cut a rune in half, producing invalid UTF-8 (which renders as the U+FFFD replacement char). Iterate with for range, count with utf8.RuneCountInString, and slice on rune boundaries (or via []rune) whenever the text may be non-ASCII.
Connections: Deepens the Strings & Runes foundation; the strings and bytes packages share these UTF-8 semantics, bufio.Scanner with ScanRunes uses the same decoder, and correct classification underpins parsing tasks in encoding/csv and tokenizers.

hash/maphash: Fast Hashing of Strings & Bytes

Std Lib

Why it matters: when you need a fast, well-distributed hash of strings or byte slices — for a custom hash table, sharding keys across workers, a bloom filter, or consistent bucketing — hash/maphash is the tool. It's the same non-cryptographic hash Go's built-in maps use, is faster than FNV/CRC for this purpose, and is randomly seeded per process to resist hash-flooding denial-of-service attacks. It is not for checksums or anything that must be stable across runs.

Hashing with a shared seed

All values you want to compare must use the same maphash.Seed. Go 1.19+ adds the convenient maphash.String/maphash.Bytes one-call helpers.

package main

import (
    "fmt"
    "hash/maphash"
)

func main() {
    // One seed shared by every value you hash together.
    seed := maphash.MakeSeed()

    h1 := maphash.String(seed, "gopher")
    h2 := maphash.String(seed, "gopher")
    h3 := maphash.String(seed, "rustacean")

    fmt.Println(h1 == h2) // true  (same seed + input)
    fmt.Println(h1 == h3) // false (different input)

    // Shard a key across N workers deterministically
    // within this process.
    const workers = 4
    bucket := h1 % workers
    fmt.Println(bucket < workers) // true
}
Output
true false true

Streaming with the Hash type

For incremental hashing, maphash.Hash implements io.Writer; write pieces, then read the 64-bit Sum64. Reset reuses it (keeping the seed).

package main

import (
    "fmt"
    "hash/maphash"
)

func main() {
    var h maphash.Hash
    h.SetSeed(maphash.MakeSeed())

    h.WriteString("hello ")
    h.Write([]byte("world"))
    sum := h.Sum64()
    fmt.Println(sum != 0) // true (some 64-bit digest)

    h.Reset()             // same seed, fresh accumulator
    h.WriteString("hello world")
    fmt.Println(h.Sum64() == sum) // true: same bytes, same seed
}
Output
true true

Common Mistakes

Wrong
// Different seeds -> different hashes for equal keys.
a := maphash.String(maphash.MakeSeed(), "k")
b := maphash.String(maphash.MakeSeed(), "k")
// a != b almost always! Can't compare across seeds.

// Persisting a maphash value to disk/DB:
save(a) // meaningless next run — seed is random per process
Right
seed := maphash.MakeSeed() // make ONE, share it
a := maphash.String(seed, "k")
b := maphash.String(seed, "k") // a == b

// Need a STABLE, cross-run hash? Use crc32/fnv/sha256,
// not maphash.
stable := crc32.ChecksumIEEE([]byte("k"))
_ = stable

Key Rules

  • Use one maphash.Seed (from maphash.MakeSeed) for every value you compare or bucket together; different seeds give different, incomparable hashes.
  • Values are randomly seeded per process and not stable across runs or machines — never persist or wire-transmit them, and never use them as a checksum.
  • maphash.String/maphash.Bytes are the fast one-shot API; maphash.Hash (an io.Writer) is for streaming, with Reset to reuse.
  • Its random seeding defends the runtime's maps against hash-flooding DoS — that same property makes it unsuitable when you need reproducibility.
Practice Exercise

Build a tiny sharded counter: N goroutines each own a map[string]int, and a key is routed to shard maphash.String(seed, key) % N to avoid a global lock. Verify that the same key always lands in the same shard within a run. Then contrast maphash with hash/fnv and crc32: hash the same key with each, restart the program, and observe which outputs change. Explain precisely why maphash's per-process randomness both thwarts hash-flooding and rules it out for a content-addressed store.

Pitfall: Assuming maphash output is stable. Because the seed is randomized at process start, the same string hashes to a different 64-bit value on the next run — so persisting it, sending it between machines, or using it as a cache key across restarts silently breaks. Reach for crc32, fnv, or a crypto hash when you need determinism; use maphash only for in-process, single-run bucketing.
Connections: The fast in-process counterpart to hash/fnv & hash/crc32; its io.Writer streaming mirrors the hex/gzip composition pattern, and its anti-DoS seeding is the same idea behind the runtime's built-in map implementation you reviewed under Maps.

testing/quick: Property-Based Testing

Std Lib

Why it matters: table-driven tests check the cases you thought of; property-based tests check thousands of random inputs against an invariant you assert always holds. The standard library's testing/quick generates random arguments for your check function and fails if the property is ever false. It's a low-friction way to flush out edge cases (empty input, huge values, unusual byte sequences) that hand-written tables miss — and it needs no third-party dependency.

Checking a property

quick.Check(f, cfg) calls f with randomly generated arguments; f returns bool. A classic property: encoding then decoding must round-trip to the original.

package main

import (
    "fmt"
    "strconv"
    "testing/quick"
)

func main() {
    // Property: Atoi(Itoa(n)) == n for every int.
    roundTrip := func(n int) bool {
        got, err := strconv.Atoi(strconv.Itoa(n))
        return err == nil && got == n
    }

    if err := quick.Check(roundTrip, nil); err != nil {
        fmt.Println("property failed:", err)
    } else {
        fmt.Println("round-trip holds for all sampled ints")
    }
}
Output
round-trip holds for all sampled ints

Catching a real bug & tuning generation

When a property is false, quick.Check reports the failing input. Use a quick.Config to raise MaxCount or plug in a custom generator via the Values field.

package main

import (
    "fmt"
    "testing/quick"
)

// Buggy: claims to be its own inverse, but it isn't.
func brokenNegate(n int8) int8 { return -n } // -(-128) overflows!

func main() {
    involutive := func(n int8) bool {
        return brokenNegate(brokenNegate(n)) == n
    }
    cfg := &quick.Config{MaxCount: 100000} // sample harder
    if err := quick.Check(involutive, cfg); err != nil {
        fmt.Println("found a counterexample")
    }
}
Output
found a counterexample

Common Mistakes

Wrong
// Ignoring the returned error means failures pass silently.
func TestRoundTrip(t *testing.T) {
    quick.Check(roundTrip, nil) // result discarded!
}

// A check func that panics instead of returning false
// aborts the run rather than reporting a clean failure.
Right
func TestRoundTrip(t *testing.T) {
    if err := quick.Check(roundTrip, nil); err != nil {
        t.Error(err) // prints the failing input
    }
}
// Return false for a violated property; reserve panics
// for genuinely impossible states.

Key Rules

  • Your check function takes generatable arguments and returns bool; quick.Check returns a non-nil error (containing the failing input) when the property breaks.
  • Always inspect the returned error — in a test call t.Error(err); discarding it makes the property test a no-op.
  • Tune with *quick.Config: MaxCount for how many samples, Rand for a fixed seed (reproducibility), and Values for a custom generator of complex types.
  • Implement quick.Generator (a Generate method) on your own types so quick can produce valid domain values instead of arbitrary junk.
Practice Exercise

Write property tests for a reverse[T any]([]T) []T: reversing twice yields the original, and length is preserved. Add a JSON round-trip property (Unmarshal(Marshal(x)) == x) for a small struct, implementing quick.Generator so only valid structs are produced. Then deliberately introduce an off-by-one in a merge of two sorted slices and let quick.Check find the counterexample. Finally, set Config.Rand to a fixed seed and explain why reproducibility matters when a CI run flags a rare failure.

Pitfall: Discarding the error from quick.Check — the test then passes no matter what, giving false confidence. Also, a property that's merely expensive can time out at the default 100 samples, while a property that panics on bad input aborts instead of reporting a tidy counterexample. Always check and surface the error, keep the property total (return false, don't panic), and set a fixed Rand seed so a failing case can be reproduced.
Connections: Complements table-driven testing and fuzzing from the Testing & Tooling days — fuzzing mutates a corpus toward crashes, while testing/quick asserts algebraic properties; round-trip checks pair naturally with encoding/json, encoding/gob, and the hex/base64 codecs.
