Error Handling

Go Foundations • 13 days since review

Quick Reminder

Go uses explicit error returns — no exceptions. The error interface has one method: Error() string. Wrap errors with fmt.Errorf("context: %w", err) to preserve the chain. Check with errors.Is(err, target) for sentinel errors and errors.As(err, &target) for typed errors.

Key Pattern

if err := doSomething(); err != nil {
    if errors.Is(err, ErrNotFound) {
        return defaultValue, nil
    }
    return zero, fmt.Errorf("doSomething failed: %w", err)
}
Challenge Exercise

Create a custom ValidationError type that holds multiple field errors. Implement Error(), make it work with errors.As, and wrap it to preserve the chain through multiple layers.

Pitfall: Using %v instead of %w in fmt.Errorf — this wraps the message but breaks the error chain, so errors.Is/errors.As won't match the original error.

First-class Functions

Functions & Control Flow • 10 days since review

Quick Reminder

Functions in Go are values — store them in variables, pass as arguments, return from functions. Function types are declared as type Transformer func(int) int. Closures capture variables from the enclosing scope by reference.

Key Pattern

type Predicate[T any] func(T) bool

func Filter[T any](items []T, pred Predicate[T]) []T {
    var result []T
    for _, item := range items {
        if pred(item) {
            result = append(result, item)
        }
    }
    return result
}

evens := Filter(nums, func(n int) bool { return n%2 == 0 })
Challenge Exercise

Implement a Compose function that takes two functions f: B→C and g: A→B and returns A→C. Then chain three transformations: trim whitespace, lowercase, and remove punctuation.

Pitfall: Closure over loop variables — in Go < 1.22, for _, v := range items { go func() { use(v) }() } captures the same v. Fix: go func(v T) { use(v) }(v) or use Go 1.22+ which scopes per-iteration.

Defer, Panic, Recover

Functions & Control Flow • 10 days since review

Quick Reminder

defer schedules a call for when the function returns (LIFO order). panic unwinds the stack; recover() in a deferred function catches it. Named returns + defer can modify return values. Common: defer f.Close(), defer mu.Unlock().

Key Pattern

func safeDiv(a, b float64) (result float64, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic: %v", r)
        }
    }()
    return a / b, nil
}
Challenge Exercise

Write a WithLock helper that takes a sync.Locker and a function, locks before calling, and guarantees unlock via defer even if the function panics.

Pitfall: defer f.Close() ignores the error from Close — for writable files, use defer func() { err = f.Close() }() with a named return to capture write-flush errors.

os & filepath

Standard Library • 9 days since review

Quick Reminder

os provides OS interaction: file I/O, env vars, process info. filepath handles cross-platform paths. Key: os.ReadFile/os.WriteFile for simple I/O, filepath.Walk/filepath.WalkDir for traversal, filepath.Join for path construction (never use string concat).

Key Pattern

entries, _ := os.ReadDir(".")
for _, e := range entries {
    info, _ := e.Info()
    fmt.Printf("%s %d bytes\n", e.Name(), info.Size())
}

// Walk directory tree
filepath.WalkDir(".", func(path string, d fs.DirEntry, err error) error {
    if err != nil { return err }
    if filepath.Ext(path) == ".go" { fmt.Println(path) }
    return nil
})
Challenge Exercise

Write a function that finds all duplicate files in a directory tree (by content hash), returning a map of hash → list of paths. Use filepath.WalkDir and crypto/sha256.

Pitfall: Using filepath.Walk instead of filepath.WalkDir — Walk calls os.Lstat on every file (slower). WalkDir is more efficient and should be preferred.

net/http

Standard Library • 8 days since review

Quick Reminder

http.Handler interface: ServeHTTP(w ResponseWriter, r *Request). http.HandlerFunc adapts functions. http.ServeMux routes requests. http.Client with timeouts for outbound requests. Always set timeouts on both server and client.

Key Pattern

mux := http.NewServeMux()
mux.HandleFunc("GET /users/{id}", getUser)
mux.HandleFunc("POST /users", createUser)
srv := &http.Server{Addr: ":8080", Handler: mux, ReadTimeout: 5 * time.Second}

// Middleware
func logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}
Challenge Exercise

Build a REST API with /items (GET list, POST create) and /items/{id} (GET one, DELETE). Include a logging middleware and a 10-second request timeout.

Pitfall: Not setting http.Client timeout — the default client has no timeout, so a slow server can hang your goroutine forever. Always: client := &http.Client{Timeout: 10 * time.Second}.

sort Package

Standard Library • 8 days since review

Quick Reminder

sort.Slice(s, less) sorts a slice with a custom comparator. sort.SliceStable preserves equal-element order. sort.Search(n, f) does binary search. For repeated sorting, implement sort.Interface (Len, Less, Swap).

Key Pattern

// Sort by age, then name
sort.Slice(people, func(i, j int) bool {
    if people[i].Age != people[j].Age {
        return people[i].Age < people[j].Age
    }
    return people[i].Name < people[j].Name
})

// Binary search
idx := sort.Search(len(sorted), func(i int) bool {
    return sorted[i] >= target
})
Challenge Exercise

Implement a TopK function using sort.Slice that returns the k largest elements from an unsorted slice. Then implement a more efficient version using a min-heap (container/heap).

Pitfall: sort.Search returns the insertion point, not the index of the found element. You must check if idx < len(s) && s[idx] == target after search.

Singleflight: Deduplicating Concurrent Requests

Concurrency Patterns

Why It Matters

When many goroutines request the same resource simultaneously (cache miss, expensive computation), singleflight ensures only one executes while others wait for the result. This prevents thundering herd / cache stampede problems where a popular key expiring triggers thousands of identical database queries at once.

The API

  • golang.org/x/sync/singleflight — the package (install with go get golang.org/x/sync)
  • Do(key, fn) — deduplicates by key, returns the shared result (v interface{}, err error, shared bool)
  • DoChan(key, fn) — non-blocking, returns a channel that delivers the result
  • Forget(key) — removes a key from the in-flight map so the next call re-executes
  • When to use: cache population, expensive lookups, API request deduplication

Example 1 — Basic: Only One Goroutine Executes

package main

import (
    "fmt"
    "sync"
    "sync/atomic"
    "time"

    "golang.org/x/sync/singleflight"
)

func main() {
    var g singleflight.Group
    var calls int64

    fetch := func() (interface{}, error) {
        atomic.AddInt64(&calls, 1)
        time.Sleep(100 * time.Millisecond) // simulate slow work
        return "result-for-user-42", nil
    }

    var wg sync.WaitGroup
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            v, err, shared := g.Do("user-42", fetch)
            fmt.Printf("goroutine %d got %v (shared=%v err=%v)\n", id, v, shared, err)
        }(i)
    }
    wg.Wait()

    fmt.Printf("fetch was actually called %d time(s)\n", atomic.LoadInt64(&calls))
}
Output
goroutine 3 got result-for-user-42 (shared=true err=<nil>) goroutine 0 got result-for-user-42 (shared=true err=<nil>) ... (all 10 goroutines receive the same value) fetch was actually called 1 time(s)

All ten goroutines block on the same key. Only the first runs fetch; the rest wait and receive a copy of the result. The shared flag is true when more than one caller received this value.

Example 2 — Intermediate: Cache-Miss Handler

type Cache struct {
    mu    sync.RWMutex
    data  map[string]string
    group singleflight.Group
    db    func(key string) (string, error)
}

func (c *Cache) Get(key string) (string, error) {
    c.mu.RLock()
    if v, ok := c.data[key]; ok {
        c.mu.RUnlock()
        return v, nil
    }
    c.mu.RUnlock()

    // Cache miss: dedupe the expensive DB lookup across goroutines.
    v, err, _ := c.group.Do(key, func() (interface{}, error) {
        val, err := c.db(key)
        if err != nil {
            return "", err
        }
        c.mu.Lock()
        c.data[key] = val
        c.mu.Unlock()
        return val, nil
    })
    if err != nil {
        return "", err
    }
    return v.(string), nil
}

On a miss, a hundred concurrent Get("popular") calls collapse into a single DB query. The first writes the cache; the rest read the deduplicated return value.

Example 3 — Advanced: DoChan with a Timeout

func GetWithTimeout(g *singleflight.Group, key string,
    fn func() (interface{}, error), d time.Duration) (interface{}, error) {

    ch := g.DoChan(key, fn)
    select {
    case res := <-ch:
        return res.Val, res.Err
    case <-time.After(d):
        // We stop waiting, but fn keeps running for other callers.
        g.Forget(key) // optional: let the next caller restart
        return nil, fmt.Errorf("singleflight timed out after %v", d)
    }
}

DoChan returns a channel so you can select against a timeout. Note that abandoning the wait does not cancel fn — it keeps running for any other waiters. Calling Forget lets the next caller start fresh instead of joining the slow in-flight call.

Example 4 — Real-World: HTTP Handler Deduping Identical Calls

type API struct {
    group  singleflight.Group
    client *http.Client
}

func (a *API) handleQuote(w http.ResponseWriter, r *http.Request) {
    symbol := r.PathValue("symbol")

    v, err, shared := a.group.Do("quote:"+symbol, func() (interface{}, error) {
        // Expensive upstream call, deduplicated per symbol.
        resp, err := a.client.Get("https://upstream/quote/" + symbol)
        if err != nil {
            return nil, err
        }
        defer resp.Body.Close()
        return io.ReadAll(resp.Body)
    })
    if err != nil {
        http.Error(w, err.Error(), http.StatusBadGateway)
        return
    }
    w.Header().Set("X-Cache-Shared", strconv.FormatBool(shared))
    w.Write(v.([]byte))
}

If 500 clients ask for AAPL in the same instant, only one upstream request goes out. The X-Cache-Shared header lets you observe deduplication in practice.

Common Mistakes

Don't
// Side effects in Do: a write is
// run once but N callers think
// they each performed it.
g.Do(orderID, func() (any, error) {
    return chargeCard(orderID) // !!
})
Do
// Only deduplicate idempotent,
// read-only work.
g.Do("price:"+sym, func() (any, error) {
    return fetchPrice(sym)
})
  • Not handling errors properly — all waiters receive the same error, so a single transient failure fails every concurrent caller.
  • Using singleflight for operations with side effects (writes, charges, emails) — the operation runs once but every caller believes it succeeded for them.
  • Keeping stale results too long — without Forget, a result is cached only for the in-flight window, but a slow fn can serve outdated data to a burst of callers.

Key Rules

  • Use singleflight for read-only / idempotent operations only.
  • Always handle the shared return value — it tells you whether the result was deduplicated.
  • Consider Forget(key) for time-sensitive data so the next caller re-fetches.
  • Errors propagate to all waiters — design for the fact that one failure fails the whole group.
Exercise 1

Implement a cache with singleflight protection: concurrent gets for the same key should trigger only one DB lookup. Verify with an atomic counter that the lookup ran exactly once under 50 concurrent goroutines.

Exercise 2

Add a timeout: if the singleflight call takes more than 2 seconds, cancel the wait and return an error to that caller while leaving the in-flight call running for others. Use DoChan and select.

Connections: Singleflight builds on goroutines, channels (via DoChan), and sync.Mutex internally. It pairs naturally with the cache-aside pattern and the graceful degradation concept later in this guide — dedupe the primary lookup, fall back when it fails.

Health Checks & Readiness Probes

Production Patterns

Why It Matters

Kubernetes uses health endpoints to decide a pod's lifecycle. A liveness probe verifies the process isn't deadlocked — if it fails, the pod is restarted. A readiness probe verifies the service can actually handle traffic (DB connected, dependencies reachable) — if it fails, the pod is removed from the load balancer but not killed.

What to Cover

  • /healthz (liveness) vs /readyz (readiness) endpoints
  • Dependency checking: DB ping, Redis ping, external API check
  • Response format: JSON with per-component statuses
  • Concurrent dependency checks with a timeout
  • Graceful degradation when non-critical dependencies fail

Example 1 — Basic: Liveness Always Returns 200

func healthz(w http.ResponseWriter, r *http.Request) {
    // Liveness only checks "is this process alive and serving?"
    // It must NOT depend on external services.
    w.WriteHeader(http.StatusOK)
    w.Write([]byte("ok"))
}

mux := http.NewServeMux()
mux.HandleFunc("GET /healthz", healthz)

Example 2 — Intermediate: Readiness Checks DB and Cache

type Server struct {
    db    *sql.DB
    cache *redis.Client
}

func (s *Server) readyz(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 2*time.Second)
    defer cancel()

    if err := s.db.PingContext(ctx); err != nil {
        http.Error(w, "db down: "+err.Error(), http.StatusServiceUnavailable)
        return
    }
    if err := s.cache.Ping(ctx).Err(); err != nil {
        http.Error(w, "cache down: "+err.Error(), http.StatusServiceUnavailable)
        return
    }
    w.WriteHeader(http.StatusOK)
    w.Write([]byte("ready"))
}

Readiness returns 503 when a required dependency is unreachable, so Kubernetes pulls the pod out of rotation until the DB recovers — without restarting a perfectly healthy process.

Example 3 — Advanced: Concurrent Checks with Partial Failure

type Check struct {
    Name     string
    Critical bool
    Probe    func(context.Context) error
}

type HealthChecker struct {
    checks []Check
}

type componentStatus struct {
    Status string `json:"status"`           // "up" or "down"
    Error  string `json:"error,omitempty"`
}

func (h *HealthChecker) Run(ctx context.Context) (int, map[string]componentStatus) {
    ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
    defer cancel()

    type result struct {
        name     string
        critical bool
        err      error
    }
    results := make(chan result, len(h.checks))

    for _, c := range h.checks {
        go func(c Check) {
            results <- result{c.Name, c.Critical, c.Probe(ctx)}
        }(c)
    }

    statuses := make(map[string]componentStatus, len(h.checks))
    code := http.StatusOK
    for range h.checks {
        r := <-results
        if r.err != nil {
            statuses[r.name] = componentStatus{Status: "down", Error: r.err.Error()}
            if r.critical {
                code = http.StatusServiceUnavailable // any critical failure => 503
            }
        } else {
            statuses[r.name] = componentStatus{Status: "up"}
        }
    }
    return code, statuses
}

All probes run concurrently and share one 2-second deadline. A non-critical failure (say, a metrics sink) is reported as down but still returns 200; a critical failure forces 503.

Example 4 — Real-World: Structured Response with Version

func (h *HealthChecker) Handler(version string) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        code, statuses := h.Run(r.Context())
        resp := map[string]any{
            "status":     map[bool]string{true: "ready", false: "degraded"}[code == http.StatusOK],
            "version":    version,
            "components": statuses,
        }
        w.Header().Set("Content-Type", "application/json")
        w.WriteHeader(code)
        json.NewEncoder(w).Encode(resp)
    }
}
Sample /readyz response (503)
{ "status": "degraded", "version": "1.4.2", "components": { "postgres": {"status": "up"}, "redis": {"status": "down", "error": "dial tcp: timeout"} } }

Common Mistakes

Don't
// Liveness depends on the DB.
// If the DB blips, K8s RESTARTS
// every healthy pod -> outage.
func healthz(w, r) {
    db.Ping() // wrong probe!
}
Do
// Liveness = process alive.
// Dependency checks belong in
// the readiness probe instead.
func healthz(w, r) {
    w.WriteHeader(200)
}
  • A liveness check that depends on external services — a transient DB outage then restarts every pod simultaneously, turning a blip into an outage.
  • A health check that's too expensive — probes run every few seconds; a heavy query adds constant load.
  • Not setting timeouts shorter than the Kubernetes probe timeout — the probe times out before your handler returns.

Key Rules

  • Liveness = process alive. Readiness = can serve traffic. Keep them distinct.
  • Keep liveness cheap and dependency-free.
  • Set check timeouts shorter than the Kubernetes probe timeout.
  • Distinguish critical from non-critical dependencies for graceful degradation.
Exercise 1

Create /healthz and /readyz handlers. /healthz always returns 200. /readyz checks a DB ping with a 2-second timeout and returns 503 if the ping fails.

Exercise 2

Build a HealthChecker that runs multiple dependency checks concurrently, returns 503 if any critical check fails, and 200 with warnings if only non-critical checks fail. Render the result as the structured JSON shown above.

Connections: Health checks combine net/http (handlers, mux routing), context (per-probe timeouts), goroutines + channels (concurrent probes), and encoding/json (structured response). The critical/non-critical split is the entry point to the graceful degradation concept that follows.

Graceful Degradation & Fallbacks

Production Patterns

Why It Matters

In distributed systems, dependencies fail. Instead of returning 500 to users, a resilient service serves stale data, uses defaults, or disables non-critical features. Combined with circuit breakers and retries, graceful degradation keeps a service useful even when parts of it are broken.

What to Cover

  • Fallback chains: try primary → try cache → return default
  • Stale-while-revalidate pattern
  • Feature flags for degraded mode
  • Timeout-based fallback (serve cached if primary is slow)
  • Combining with a circuit breaker

Example 1 — Basic: Try Primary, Return Cached on Error

func (s *Service) GetProfile(id string) (Profile, error) {
    p, err := s.primary.Fetch(id)
    if err != nil {
        log.Printf("primary failed for %s, serving cache: %v", id, err)
        if cached, ok := s.cache.Get(id); ok {
            return cached, nil // degraded but usable
        }
        return Profile{}, err
    }
    s.cache.Set(id, p)
    return p, nil
}

Note the log line: a fallback that happens silently is invisible degradation. Always record when you fall back.

Example 2 — Intermediate: Timeout-Based Fallback

func (s *Service) GetWithDeadline(ctx context.Context, id string) Profile {
    ctx, cancel := context.WithTimeout(ctx, 500*time.Millisecond)
    defer cancel()

    type res struct {
        p   Profile
        err error
    }
    ch := make(chan res, 1)
    go func() {
        p, err := s.primary.FetchContext(ctx, id)
        ch <- res{p, err}
    }()

    select {
    case r := <-ch:
        if r.err == nil {
            s.cache.Set(id, r.p)
            return r.p
        }
    case <-ctx.Done():
        log.Printf("primary too slow for %s, serving stale", id)
    }
    if cached, ok := s.cache.Get(id); ok {
        return cached
    }
    return defaultProfile
}

If the primary doesn't respond within 500ms, we stop waiting and serve stale cache. The user sees a fast (if slightly old) response instead of a hung request.

Example 3 — Advanced: Fallback Chain with Context

type Source[T any] func(ctx context.Context) (T, error)

func FirstSuccess[T any](ctx context.Context, sources ...Source[T]) (T, error) {
    var lastErr error
    for i, src := range sources {
        if err := ctx.Err(); err != nil {
            var zero T
            return zero, err // caller cancelled the whole chain
        }
        v, err := src(ctx)
        if err == nil {
            return v, nil
        }
        log.Printf("source %d failed: %v", i, err)
        lastErr = err
    }
    var zero T
    return zero, fmt.Errorf("all sources failed: %w", lastErr)
}

// Usage: try live API, then read replica, then static default.
profile, err := FirstSuccess(ctx,
    fromLiveAPI(id),
    fromReadReplica(id),
    fromStaticDefault(id),
)

Each source is tried in order; the first success wins. Context cancellation short-circuits the chain so a cancelled request doesn't keep probing dead backends.

Example 4 — Real-World: Feature Degradation by Dependency

type Page struct {
    Articles      []Article
    Recommended   []Article // optional, degrades gracefully
    Degraded      bool
}

func (s *Site) Home(ctx context.Context) Page {
    page := Page{Articles: s.mustArticles(ctx)} // critical

    recs, err := s.recommender.For(ctx, "home")
    if err != nil {
        // Non-critical: hide the section, mark degraded, keep serving.
        log.Printf("recommendations unavailable: %v", err)
        page.Degraded = true
        return page
    }
    page.Recommended = recs
    return page
}

The article list is critical — if it fails, the page fails. Recommendations are a bonus; when the recommender is down, the section disappears and the page still renders.

Common Mistakes

Don't
// Silent fallback: nobody knows
// the service is degraded.
if v, ok := cache.Get(id); ok {
    return v // no log, no metric
}
Do
// Observable fallback.
if v, ok := cache.Get(id); ok {
    fallbackRate.Inc()
    log.Printf("serving stale %s", id)
    return v
}
  • Not logging fallback activation — invisible degradation means you find out from users, not dashboards.
  • The fallback itself failing with no fallback for the fallback — always have a final default.
  • Not monitoring the degradation rate — a slowly climbing fallback rate is an early warning you'll miss without a metric.

Key Rules

  • Always log when you fall back.
  • Monitor fallback rate as a metric, not just an error.
  • Test fallback paths regularly — an untested fallback is a latent bug.
  • Set aggressive timeouts on the primary path when a fallback is available.
Exercise 1

Implement a WithFallback[T any](primary, fallback func() (T, error)) (T, error) function that tries primary first and falls back on error. Log which path served the result.

Exercise 2

Connections: Graceful degradation builds on context (timeouts, cancellation), goroutines + channels (timeout races), and generics (reusable fallback chains). It complements singleflight (dedupe the primary), health checks (the critical/non-critical split), and the circuit-breaker / retry patterns from Day 13.

Advanced Testing — 2 Concepts

Table-Driven Benchmarks & Allocation Tracking

Testing

Why It Matters

Go's testing.B includes allocation tracking built in. b.ReportAllocs() shows allocations per operation. Sub-benchmarks with table-driven patterns let you compare approaches side-by-side. This is critical for hot-path optimization, where a single avoidable allocation per call multiplies across millions of requests.

What to Cover

  • b.ReportAllocs() for per-op allocation stats
  • b.ResetTimer() after expensive setup
  • Table-driven benchmarks with b.Run
  • testing.AllocsPerRun() for exact allocation counting
  • Comparing implementations: string concat vs strings.Builder vs bytes.Buffer

Example 1 — Basic: Benchmark with ReportAllocs

// file: build_test.go
package main

import (
    "strings"
    "testing"
)

func joinBuilder(parts []string) string {
    var b strings.Builder
    for _, p := range parts {
        b.WriteString(p)
    }
    return b.String()
}

var parts = []string{"a", "b", "c", "d", "e", "f", "g", "h"}

func BenchmarkJoinBuilder(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        _ = joinBuilder(parts)
    }
}
go test -bench=. -benchmem
BenchmarkJoinBuilder-8 12000000 96.3 ns/op 24 B/op 2 allocs/op

-benchmem (or b.ReportAllocs()) adds the B/op and allocs/op columns — the bytes and number of heap allocations per single operation.

Example 2 — Intermediate: Table-Driven Comparison

func concatPlus(parts []string) string {
    s := ""
    for _, p := range parts {
        s += p
    }
    return s
}

func concatBuffer(parts []string) string {
    var buf bytes.Buffer
    for _, p := range parts {
        buf.WriteString(p)
    }
    return buf.String()
}

func BenchmarkConcat(b *testing.B) {
    impls := map[string]func([]string) string{
        "Plus":    concatPlus,
        "Builder": joinBuilder,
        "Buffer":  concatBuffer,
    }
    for name, fn := range impls {
        b.Run(name, func(b *testing.B) {
            b.ReportAllocs()
            for i := 0; i < b.N; i++ {
                _ = fn(parts)
            }
        })
    }
}
Comparison output
BenchmarkConcat/Plus-8 4000000 305 ns/op 120 B/op 7 allocs/op BenchmarkConcat/Builder-8 12000000 96 ns/op 24 B/op 2 allocs/op BenchmarkConcat/Buffer-8 9000000 128 ns/op 64 B/op 2 allocs/op

The + operator allocates a fresh string every iteration (7 allocs); strings.Builder grows one backing array (2 allocs). Sub-benchmarks named via b.Run show up as BenchmarkConcat/Plus etc., perfect for side-by-side comparison.

Example 3 — Advanced: Setup with ResetTimer

func BenchmarkLookup(b *testing.B) {
    // Expensive setup that must NOT count toward the benchmark.
    m := make(map[int]string, 1_000_000)
    for i := 0; i < 1_000_000; i++ {
        m[i] = strconv.Itoa(i)
    }

    b.ResetTimer()   // discard setup time and allocations
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        _ = m[i%1_000_000]
    }
}

Without b.ResetTimer(), the one-second map build would be averaged into every reported ns/op, wildly inflating the number. Reset right before the measured loop.

Example 4 — Real-World: Exact Allocation Counting

func TestMarshalAllocations(t *testing.T) {
    u := User{ID: 1, Name: "Ada", Email: "ada@example.com"}

    // testing.AllocsPerRun returns the average allocations per call.
    avg := testing.AllocsPerRun(1000, func() {
        b, _ := json.Marshal(u)
        _ = b
    })
    t.Logf("json.Marshal allocates %.0f times per call", avg)
    if avg > 5 {
        t.Errorf("regression: %.0f allocs/op exceeds budget of 5", avg)
    }
}

testing.AllocsPerRun gives an exact average (not a benchmark estimate), so you can assert an allocation budget in a regular test and fail CI if a change regresses it.

Common Mistakes

Don't
func BenchmarkX(b *testing.B) {
    setup() // counted!
    for i := 0; i < b.N; i++ {
        result := f()  // compiler may
                       // discard this
    }
}
Do
var sink Result
func BenchmarkX(b *testing.B) {
    setup()
    b.ResetTimer()
    for i := 0; i < b.N; i++ {
        sink = f() // escapes -> kept
    }
}
  • Not using b.ResetTimer() — setup time leaks into the benchmark and skews ns/op.
  • Benchmarking with too-small input — tiny inputs are dominated by loop overhead and give misleading rankings.
  • The compiler optimizing away unused results — assign to a package-level sink var so the work can't be eliminated.

Key Rules

  • Always b.ReportAllocs() for performance-sensitive code.
  • Use b.ResetTimer() after any non-trivial setup.
  • Assign the result to a package-level var to prevent compiler elimination.
  • Use testing.AllocsPerRun when you need an exact, assertable allocation count.

Plugin Architecture with Interfaces

Architecture

Why It Matters

Go's interfaces enable plugin-style architectures where components can be swapped without changing core logic. Storage backends, auth providers, notification channels — all benefit from interface-based pluggability. The result is code that's open for extension (add a new backend) but closed for modification (the core never changes).

What to Cover

  • Define narrow interfaces for extension points
  • Registry pattern: a map of name → implementation
  • Factory functions and constructors
  • Config-driven plugin selection
  • Init-time registration with init()

Example 1 — Basic: Storage Interface with Two Implementations

// The consumer package defines the narrow interface it needs.
type Storage interface {
    Get(key string) ([]byte, error)
    Put(key string, data []byte) error
}

type FileStorage struct{ dir string }

func (f *FileStorage) Get(key string) ([]byte, error) {
    return os.ReadFile(filepath.Join(f.dir, key))
}
func (f *FileStorage) Put(key string, data []byte) error {
    return os.WriteFile(filepath.Join(f.dir, key), data, 0o644)
}

type S3Storage struct{ bucket string /* ... client ... */ }

func (s *S3Storage) Get(key string) ([]byte, error) { /* ... */ return nil, nil }
func (s *S3Storage) Put(key string, data []byte) error { /* ... */ return nil }

// Core logic depends only on the interface, never the concrete type.
func Save(s Storage, key string, data []byte) error {
    return s.Put(key, data)
}

Example 2 — Intermediate: Registry with Register/Get

type Factory func(config map[string]string) (Storage, error)

var (
    registryMu sync.RWMutex
    registry   = map[string]Factory{}
)

func Register(name string, f Factory) {
    registryMu.Lock()
    defer registryMu.Unlock()
    if _, dup := registry[name]; dup {
        panic("storage: duplicate registration for " + name)
    }
    registry[name] = f
}

func New(name string, config map[string]string) (Storage, error) {
    registryMu.RLock()
    f, ok := registry[name]
    registryMu.RUnlock()
    if !ok {
        return nil, fmt.Errorf("storage: unknown backend %q", name)
    }
    return f(config)
}

The registry decouples the caller from concrete types entirely — you select a backend by string name, exactly like sql.Open("postgres", ...) does in the standard library.

Example 3 — Advanced: Config-Driven Selection at Startup

type Config struct {
    Backend string            `json:"backend"` // "file" or "s3"
    Options map[string]string `json:"options"`
}

func BuildStorage(cfg Config) (Storage, error) {
    s, err := New(cfg.Backend, cfg.Options)
    if err != nil {
        return nil, fmt.Errorf("build storage: %w", err)
    }
    return s, nil
}

// init() in each backend's file registers it automatically.
func init() {
    Register("file", func(c map[string]string) (Storage, error) {
        dir := c["dir"]
        if dir == "" {
            return nil, errors.New("file backend requires 'dir'")
        }
        return &FileStorage{dir: dir}, nil
    })
}

Each backend registers itself in its own init(). Importing the package (even with a blank _ "myapp/storage/s3" import) is enough to make it selectable — the core never references the concrete type.

Example 4 — Real-World: Notification System

type Notifier interface {
    Notify(ctx context.Context, msg string) error
}

type EmailNotifier struct{ to string }
func (e *EmailNotifier) Notify(ctx context.Context, msg string) error { /* ... */ return nil }

type SlackNotifier struct{ webhook string }
func (s *SlackNotifier) Notify(ctx context.Context, msg string) error { /* ... */ return nil }

type WebhookNotifier struct{ url string }
func (w *WebhookNotifier) Notify(ctx context.Context, msg string) error { /* ... */ return nil }

// Fan out to every configured channel; collect failures.
func NotifyAll(ctx context.Context, msg string, channels ...Notifier) error {
    var errs []error
    for _, ch := range channels {
        if err := ch.Notify(ctx, msg); err != nil {
            errs = append(errs, err)
        }
    }
    return errors.Join(errs...)
}

Adding a new channel (PagerDuty, SMS) means writing one type that satisfies Notifier — no change to NotifyAll or any caller. errors.Join aggregates partial failures.

Common Mistakes

Don't
// God interface: forces every
// backend to implement methods
// it doesn't need.
type Storage interface {
    Get(k string) ([]byte, error)
    Put(k string, d []byte) error
    List() ([]string, error)
    Backup() error
    Migrate() error
}
Do
// Small interfaces, composed.
type Getter interface {
    Get(k string) ([]byte, error)
}
type Putter interface {
    Put(k string, d []byte) error
}
type Storage interface {
    Getter
    Putter
}
  • Defining interfaces too wide (God interfaces) — every implementer is forced to stub methods it doesn't use.
  • Putting the interface definition in the implementation package — it belongs in the consumer package so the consumer owns its requirements.
  • Tight coupling through concrete types in function signatures — accept the interface, not *S3Storage.

Key Rules

  • Define interfaces where they're used, not where they're implemented.
  • Keep interfaces small — ideally 1–3 methods.
  • Accept interfaces, return structs.
  • Register implementations at init time so adding a backend never touches core code.
