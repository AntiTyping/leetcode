os & filepath

Standard Library Mastery • due today • rep 2

Quick Reminder

For simple whole-file I/O, reach for os.ReadFile and os.WriteFile — no manual open/close needed. Build paths with filepath.Join so they use the right separator on every OS. Walk directory trees with filepath.WalkDir (faster than the older Walk because it avoids a Stat per entry).

Key Pattern

package main

import (
    "fmt"
    "io/fs"
    "path/filepath"
)

func main() {
    root := filepath.Join("some", "dir") // OS-correct separator
    _ = root

    err := filepath.WalkDir(".", func(path string, d fs.DirEntry, err error) error {
        if err != nil {
            return err // surface permission / I/O errors
        }
        if !d.IsDir() && filepath.Ext(path) == ".go" {
            fmt.Println(path)
        }
        return nil
    })
    if err != nil {
        fmt.Println("walk failed:", err)
    }
}
Output (example)
main.go internal/util/util.go
Challenge Exercise

Write dirSize(root string) (int64, error) that returns the total byte size of all regular files under root, using filepath.WalkDir. Call d.Info() to get the file size, and make sure you skip directories. Bonus: build the input path portably with filepath.Join and verify it works whether the caller passes a trailing slash or not.

Pitfall: Building paths with string concatenation (dir + "/" + name) breaks on Windows and double-slashes when dir already ends in a separator. Always use filepath.Join, which cleans the result and uses the OS separator.

net/http: Handlers, ServeMux, Middleware, Client

Standard Library Mastery • due today • rep 2

Quick Reminder

An http.Handler is anything with ServeHTTP(w, r); http.HandlerFunc adapts a plain function to that interface. Since Go 1.22, http.ServeMux supports method-and-pattern routing like "GET /api/{id}" with r.PathValue("id"). On the client side, always construct an http.Client with an explicit Timeout.

Key Pattern

package main

import (
    "fmt"
    "net/http"
    "time"
)

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("GET /api/{id}", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "id = %s\n", r.PathValue("id"))
    })

    // Client with an explicit timeout
    client := &http.Client{Timeout: 5 * time.Second}
    _ = client
    _ = mux

    fmt.Println("routes registered")
}
Output
routes registered
Challenge Exercise

Write a logging middleware with signature func(http.Handler) http.Handler that records the method, path, status code, and duration of each request. Wrap http.ResponseWriter in a small struct that captures the status passed to WriteHeader (default 200). Register it around a mux that has a GET /health route.

Pitfall: Using http.Get or a bare &http.Client{} with no Timeout. The zero-value client has no timeout — a slow or hung server can block your goroutine forever. Always set Timeout (or use a per-request context deadline).

sort Package

Standard Library Mastery • due today • rep 2

Quick Reminder

sort.Slice(s, less) sorts in place with a custom comparator; sort.SliceStable preserves the relative order of equal elements. sort.Search(n, f) does a binary search and returns the smallest index where f is true — an insertion point, not necessarily a match. (In Go 1.21+ the generic slices.Sort / slices.SortFunc are often cleaner for ordered types.)

Key Pattern

package main

import (
    "fmt"
    "sort"
)

func main() {
    nums := []int{5, 2, 9, 1, 7}
    sort.Slice(nums, func(i, j int) bool { return nums[i] < nums[j] })
    fmt.Println(nums)

    // Binary search: first index where nums[i] >= 6
    i := sort.Search(len(nums), func(i int) bool { return nums[i] >= 6 })
    fmt.Printf("insertion point for 6: index %d (value %d)\n", i, nums[i])
}
Output
[1 2 5 7 9] insertion point for 6: index 3 (value 7)
Challenge Exercise

Given a sorted []int and a target, write contains(s []int, target int) bool using sort.Search. Remember that Search returns where the target would go — you must then check that i < len(s) && s[i] == target. Then sort a slice of structs by two fields (primary ascending, tiebreak descending) with sort.Slice.

Pitfall: Treating sort.Search's return value as "found index." It returns the insertion point even when the element is absent (possibly len(s)). You must verify i < len(s) && s[i] == target before trusting it — otherwise you risk an out-of-range index or a false positive.

Graceful Shutdown

Production Go • due today • rep 1

Quick Reminder

Catch OS signals with signal.NotifyContext, which returns a context that is cancelled on SIGINT/SIGTERM. On cancellation, call server.Shutdown(ctx) — it stops accepting new connections and drains in-flight requests, returning when they finish or ctx expires. Always wrap Shutdown with its own timeout context.

Key Pattern

package main

import (
    "context"
    "fmt"
    "net/http"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    srv := &http.Server{Addr: ":8080"}
    go func() { _ = srv.ListenAndServe() }()

    <-ctx.Done() // wait for signal
    fmt.Println("shutting down...")

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    if err := srv.Shutdown(shutdownCtx); err != nil {
        fmt.Println("forced shutdown:", err)
    }
    fmt.Println("done")
}
Output (after Ctrl-C)
shutting down... done
Challenge Exercise

Extend the pattern so that, in addition to the HTTP server, you also gracefully stop a background worker goroutine that reads from a job channel. The worker should finish its current job, then exit when the shutdown context is cancelled. Use a sync.WaitGroup to ensure main does not return until both the server and the worker have fully stopped.

Pitfall: Calling srv.Shutdown(context.Background()) with no timeout. Shutdown blocks until every in-flight connection closes — a single slow client (or a hijacked/streaming connection) can hang shutdown indefinitely. Always pass a context with a deadline.

Configuration Management

Production Go • due today • rep 1

Quick Reminder

Load config once, eagerly, at startup into a typed struct, then validate it. A common layering is defaults < flags < environment. Read env vars with os.Getenv / os.LookupEnv (the latter distinguishes "unset" from "empty"), or a library like envconfig driven by struct tags. Fail fast if a required value is missing.

Key Pattern

package main

import (
    "fmt"
    "os"
)

type Config struct {
    Port    string
    DBURL   string
    Verbose bool
}

func Load() (Config, error) {
    cfg := Config{Port: "8080"} // default
    if v, ok := os.LookupEnv("PORT"); ok {
        cfg.Port = v
    }
    cfg.DBURL = os.Getenv("DATABASE_URL")
    cfg.Verbose = os.Getenv("VERBOSE") == "true"

    if cfg.DBURL == "" {
        return cfg, fmt.Errorf("DATABASE_URL is required")
    }
    return cfg, nil
}

func main() {
    cfg, err := Load()
    if err != nil {
        fmt.Println("config error:", err)
        return
    }
    fmt.Printf("%+v\n", cfg)
}
Output (with DATABASE_URL=postgres://... PORT=9090)
{Port:9090 DBURL:postgres://... Verbose:false}
Challenge Exercise

Add a Validate() method to Config that checks the port is numeric and in range 1–65535, and that DBURL starts with postgres://. Then add a flag-based override layer so that a -port command-line flag beats the environment variable, which beats the default. Verify all three precedence levels with manual test runs.

Pitfall: Reading config lazily at request time (e.g., calling os.Getenv inside a handler). This hides a missing/invalid variable until traffic arrives instead of at boot, and repeats the parse cost on every request. Load and validate everything once at startup, then pass the struct around.

Escape Analysis & Stack vs Heap

Memory & Performance • due today • rep 1

Quick Reminder

The compiler decides whether a value lives on the (cheap, auto-freed) stack or the (GC-managed) heap. A value escapes to the heap when its lifetime outlives the function — most commonly when you return a pointer to a local, or store it in an interface. Inspect the decisions with go build -gcflags="-m".

Key Pattern

package main

import "fmt"

// x escapes: a pointer to it is returned
func newCounter() *int {
    x := 42
    return &x
}

// stays on the stack: never leaves the function
func sum(a, b int) int {
    total := a + b
    return total
}

func main() {
    c := newCounter()
    fmt.Println(*c, sum(2, 3)) // fmt.Println also forces escapes
}
go build -gcflags="-m" (excerpt)
./main.go:6:2: moved to heap: x ./main.go:18:14: ... argument does not escape 42 5
Challenge Exercise

Write two versions of a function that builds and returns a 1024-byte buffer: one that returns []byte (the backing array escapes) and one that accepts a caller-provided buf []byte and writes into it (no escape). Run go build -gcflags="-m" on both and explain, in a comment, which lines escape and why. Then benchmark both to see the allocation difference.

Pitfall: Assuming passing a value to fmt.Println (or any ...any function) is free. Converting a concrete value to an interface{} argument forces it to the heap, so even small ints "escape" when printed. Don't micro-optimize blindly — profile and read the -m output before reacting.

Section 1 — Go Performance & Optimization

Profile-Guided Optimization (PGO)

Performance

Why it matters: Since Go 1.21, you can feed a real CPU profile back into the compiler. It uses that data to make smarter inlining and devirtualization choices on your actual hot paths — typically a 2–7% speedup for free, no code changes.

What PGO actually does

Normally the compiler optimizes with static heuristics. PGO adds dynamic evidence: a profile that says "this branch runs 90% of the time" or "this interface call almost always dispatches to type *Foo." Armed with that, it inlines hotter functions more aggressively and turns hot interface calls into direct (devirtualized) calls.

Step 1 — Collect a CPU profile

Use runtime/pprof for a one-shot binary, or net/http/pprof for a running server. The goal is a profile that reflects a representative production workload.

package main

import (
    "os"
    "runtime/pprof"
)

func main() {
    f, _ := os.Create("default.pgo")
    defer f.Close()

    pprof.StartCPUProfile(f)
    defer pprof.StopCPUProfile()

    hotWork() // exercise the real, representative workload
}

func hotWork() {
    sum := 0
    for i := 0; i < 50_000_000; i++ {
        sum += i % 7
    }
    _ = sum
}
Result
writes a CPU profile to ./default.pgo

Step 2 — Build with the profile

Point the build at the profile explicitly, or let auto-PGO find it.

$ go build -pgo=default.pgo -o app .

# Auto-PGO: if a file named default.pgo sits next to
# the main package, `go build` uses it automatically:
$ ls
default.pgo  main.go
$ go build -o app .   # PGO applied automatically
Output
build uses default.pgo for profile-guided inlining & devirtualization

Step 3 — Measure

Compare benchmarks with and without the profile to confirm a real win. The iterate-and-repeat loop matters: collect on the PGO-built binary, rebuild, and the profile stays stable.

$ go test -bench=. -count=10 > old.txt          # no PGO
$ go test -bench=. -count=10 -pgo=default.pgo > new.txt
$ benchstat old.txt new.txt
benchstat (example)
name old time/op new time/op delta Process-8 1.42ms 1.34ms -5.6% (p=0.000 n=10)

Common Mistakes

Wrong
// Profile a synthetic micro-benchmark,
// then ship that PGO build to prod.
// The hot paths don't match real traffic.
go build -pgo=bench.pgo
Right
// Profile representative production load,
// commit default.pgo to the main package.
go build -pgo=default.pgo
// rebuild periodically as workload shifts

Key Rules

  • PGO is available in Go 1.21+ and is opt-in via -pgo or an auto-detected default.pgo.
  • The profile must come from a workload that matches production, or you optimize the wrong paths.
  • PGO is iterative: collect → build → deploy → collect again. The build is stable across iterations.
  • Typical gains are modest (2–7%); treat it as a free top-up, not a substitute for algorithmic fixes.
Practice Exercise

Take a CPU-bound program (e.g., a JSON parser over a large file, or a hash-heavy loop). Collect a profile with runtime/pprof, commit it as default.pgo, and use benchstat over 10 runs to quantify the speedup. Then deliberately profile a different workload and observe how the measured benefit changes — record both numbers.

Pitfall: Reusing a stale or mismatched profile. A profile captured from a different workload steers the compiler toward paths your production traffic doesn't take, and can even slightly regress the real hot paths. Re-collect whenever your traffic mix changes.
Connections: PGO consumes the same profiles you gather for performance investigation and pairs with GC tuning — together they cover compile-time and runtime performance knobs.

Garbage Collector Tuning: GOGC & GOMEMLIMIT

Performance

Why it matters: GC pauses and throughput are tunable. GOMEMLIMIT (Go 1.19+) is a game-changer for containers because it lets the GC respect a hard memory ceiling instead of only reacting to heap growth percentage.

GOGC — the classic knob

GOGC=100 (the default) means the GC triggers a cycle once the live heap has grown 100% since the last collection. Lower it for less memory at the cost of more frequent GC; raise it for more throughput at the cost of memory.

package main

import (
    "fmt"
    "runtime/debug"
)

func main() {
    old := debug.SetGCPercent(50) // GC twice as often, lower peak heap
    fmt.Println("previous GOGC:", old)
    debug.SetGCPercent(off()) // disable percentage trigger (see below)
}

func off() int { return -1 } // GOGC=off equivalent
Output
previous GOGC: 100

GOMEMLIMIT — a soft memory ceiling

Set a total-memory target; the GC will run more aggressively as the heap approaches it, trading CPU to avoid blowing the limit. The common container recipe is GOGC=off + GOMEMLIMIT just under the cgroup limit.

# Leave ~10% headroom under a 1 GiB container limit:
$ GOGC=off GOMEMLIMIT=900MiB ./app

# Or set it at runtime (bytes):
# debug.SetMemoryLimit(900 << 20)
Effect
GC paces itself to keep total memory near 900 MiB instead of OOM-killing

Inspecting GC behavior

runtime.ReadMemStats exposes live stats; for a cheap continuous view, run with GODEBUG=gctrace=1.

package main

import (
    "fmt"
    "runtime"
)

func main() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    fmt.Printf("HeapAlloc=%d KiB  NumGC=%d\n", m.HeapAlloc/1024, m.NumGC)
}
Output (example)
HeapAlloc=312 KiB NumGC=0

Common Mistakes

Wrong
// GOMEMLIMIT equal to the container limit.
// No headroom -> the OOM killer wins the
// race before the GC can react.
GOMEMLIMIT=1GiB  # container limit is 1GiB
Right
// Leave ~10% headroom so the GC has room
// to do its work under pressure.
GOMEMLIMIT=900MiB  # container limit is 1GiB

Key Rules

  • GOGC is a percentage trigger; GOMEMLIMIT is a soft byte ceiling — they can be combined.
  • In containers, prefer GOMEMLIMIT set below the cgroup limit, with ~10% headroom.
  • Adjust at runtime with debug.SetGCPercent and debug.SetMemoryLimit.
  • GOMEMLIMIT is a soft limit: the GC will burn CPU to honor it but cannot magically free unreachable-but-live memory.
Practice Exercise

Write a program that allocates a growing slice of slices in a loop and prints runtime.MemStats.NumGC each iteration. Run it three times: default GOGC=100, GOGC=20, and GOGC=off GOMEMLIMIT=64MiB. Record how NumGC and peak HeapAlloc differ across the three runs and explain the trade-off you observe.

Pitfall: Setting GOMEMLIMIT too close to (or above) the container's hard memory limit. Because it is a soft limit and accounts for more than just the Go heap (stacks, runtime metadata), leaving no headroom means the OOM killer fires before the GC can rescue you.
Connections: GC tuning works alongside escape analysis (fewer heap allocations = less GC pressure) and the allocation-reduction techniques in string performance.

String Performance: Builders, Interning & Conversions

Performance

Why it matters: String operations are one of the most common hidden sources of allocations in Go. Knowing exactly where copies happen — and where the compiler avoids them — lets you cut garbage without contorting your code.

strings.Builder — build without re-allocating

Concatenating with + in a loop allocates a fresh string every iteration. strings.Builder accumulates into a growable buffer and hands you the final string once, with zero extra copies. Use Grow(n) when you can estimate the size.

package main

import (
    "fmt"
    "strings"
)

func main() {
    var b strings.Builder
    b.Grow(32) // pre-allocate to avoid reallocations

    for i := 0; i < 3; i++ {
        b.WriteString("go")
        b.WriteByte('-')
    }
    fmt.Println(b.String())
}
Output
go-go-go-

Conversion costs & the map-lookup optimization

Converting []byte to string normally copies the data. But the compiler recognizes a special case: m[string(b)] as a map key does not allocate — it looks up using the bytes directly.

package main

import "fmt"

func main() {
    counts := map[string]int{"hello": 1}
    key := []byte("hello")

    // No allocation: string(key) used directly as a map key
    fmt.Println(counts[string(key)])
}
Output
1

strings.Clone & comparisons

A substring shares the backing array of its parent. strings.Clone(s) makes a minimal independent copy so a tiny piece doesn't pin a huge string in memory. For case-insensitive comparison, strings.EqualFold avoids the two allocations that ToLower(a) == ToLower(b) would cost.

package main

import (
    "fmt"
    "strings"
)

func main() {
    huge := strings.Repeat("x", 1<<20) + "tail"
    small := strings.Clone(huge[len(huge)-4:]) // detach "tail"

    fmt.Println(small)
    fmt.Println(strings.EqualFold("Go-LANG", "golang")) // no allocs
}
Output
tail true

Common Mistakes

Wrong
// O(n^2) allocations: each += builds
// a brand-new string.
s := ""
for _, p := range parts {
    s += p
}
Right
var b strings.Builder
b.Grow(estimatedLen)
for _, p := range parts {
    b.WriteString(p)
}
s := b.String()

Key Rules

  • Use strings.Builder (with Grow) for any non-trivial concatenation; never += in a loop.
  • []byte↔string copies — except m[string(b)] map lookups, which the compiler optimizes to zero allocation.
  • strings.Clone detaches a small substring so it stops pinning a large backing array.
  • strings.EqualFold beats ToLower+== for case-insensitive equality — no allocations.
  • unsafe.String / unsafe.StringData give zero-copy conversion, but are advanced and easy to misuse — reach for them rarely.
Practice Exercise

Write two functions that join a []string into a single comma-separated string: one using += in a loop and one using strings.Builder with Grow. Benchmark both with -benchmem over a 10,000-element slice and report allocs/op for each. Then demonstrate the substring-pinning problem: hold a 4-byte slice of a 1 MiB string with and without strings.Clone, and reason about which one keeps the megabyte alive.

Pitfall: Holding a small substring of a large string keeps the entire backing array alive. If you parse one short token out of a multi-megabyte response and stash it in a long-lived map, you may unintentionally pin megabytes of memory. Use strings.Clone to detach it.
Connections: Reducing string allocations directly lowers GC pressure, and the copy-vs-no-copy behavior is exactly what escape analysis surfaces. Builders also pair well with buffered I/O.

Section 2 — Advanced Generic Patterns

Generic Data Structures: Stack, Queue, Set

Generics

Why it matters: Go's generics let you build type-safe, reusable containers without the runtime boxing and per-call type assertions that interface{}-based structures required. You get one implementation that works for every element type.

A generic Stack[T]

The type parameter T any threads through every method, so a Stack[int] and a Stack[string] are distinct, fully type-checked types.

package main

import "fmt"

type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(v T) { s.items = append(s.items, v) }

func (s *Stack[T]) Pop() (T, bool) {
    var zero T
    if len(s.items) == 0 {
        return zero, false
    }
    v := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return v, true
}

func main() {
    var s Stack[int]
    s.Push(1)
    s.Push(2)
    v, ok := s.Pop()
    fmt.Println(v, ok)
}
Output
2 true

A generic Set[T comparable]

A set needs its elements as map keys, so the constraint must be comparable, not any.

package main

import "fmt"

type Set[T comparable] map[T]struct{}

func (s Set[T]) Add(v T)           { s[v] = struct{}{} }
func (s Set[T]) Contains(v T) bool { _, ok := s[v]; return ok }

func (s Set[T]) Union(other Set[T]) Set[T] {
    out := Set[T]{}
    for v := range s {
        out.Add(v)
    }
    for v := range other {
        out.Add(v)
    }
    return out
}

func main() {
    a := Set[string]{}
    a.Add("x")
    a.Add("y")
    b := Set[string]{}
    b.Add("y")
    b.Add("z")
    fmt.Println(len(a.Union(b)), a.Contains("x"))
}
Output
3 true

Constraints for ordering

To use < on a generic type (e.g., a sorted insert), constrain it with constraints.Ordered from golang.org/x/exp/constraints (or cmp.Ordered in the standard library since Go 1.21).

package main

import (
    "cmp"
    "fmt"
)

func Max[T cmp.Ordered](a, b T) T {
    if a > b {
        return a
    }
    return b
}

func main() {
    fmt.Println(Max(3, 9))
    fmt.Println(Max("apple", "banana"))
}
Output
9 banana

Common Mistakes

Wrong
type Set[T any] map[T]struct{}
// compile error: T (any) is not
// comparable, cannot be a map key
Right
type Set[T comparable] map[T]struct{}
// comparable allows == and use as map key

Key Rules

  • Use comparable when the type must be a map key or compared with ==.
  • Use cmp.Ordered (1.21+) / constraints.Ordered when you need <, >, sorting.
  • Return the zero value with var zero T for "empty" results from generic methods.
  • Generics vs codegen vs interfaces: generics give type safety + one source; interfaces add runtime dispatch; codegen is a last resort.
Practice Exercise

Implement a generic Queue[T any] backed by a slice with Enqueue, Dequeue (returning (T, bool)), and Len. Then add an Intersection method to the Set[T comparable] above. Finally, build an OrderedMap[K comparable, V any] that remembers insertion order by keeping a parallel []K of keys alongside the map — expose Set, Get, and an in-order Keys().

Pitfall: Trying to use <, >, or + on a value of a type parameter constrained only by any. The compiler rejects it because not all types support those operators. Add cmp.Ordered (for comparison) or a custom constraint with the right operations.
Connections: Builds directly on sort (ordered constraints mirror comparator logic) and the generics fundamentals reviewed earlier. The dispatch trade-offs continue in type switches & any.

Type Switches & any with Generics

Generics

Why it matters: Generics and type switches solve different problems — one resolves types at compile time, the other inspects them at runtime. Knowing which to reach for (and how to combine them) unlocks clean, flexible dispatch.

Type switches operate on any, not on T

You cannot type-switch on a constrained type parameter T directly — the compiler has already resolved T to a single concrete type per instantiation, so there is nothing to switch on. Instead, accept any and switch on the dynamic type.

package main

import "fmt"

func describe(x any) string {
    switch v := x.(type) {
    case int:
        return fmt.Sprintf("int doubled: %d", v*2)
    case string:
        return fmt.Sprintf("string of len %d", len(v))
    case nil:
        return "nil"
    default:
        return fmt.Sprintf("unhandled %T", v)
    }
}

func main() {
    fmt.Println(describe(21))
    fmt.Println(describe("hello"))
    fmt.Println(describe(3.14))
}
Output
int doubled: 42 string of len 5 unhandled float64

Pattern: generic wrapper, type-switch inside

A common hybrid: a generic function for ergonomics and inference, which converts T to any and dispatches on the runtime type for type-specific logic.

package main

import "fmt"

func Format[T any](v T) string {
    switch x := any(v).(type) { // bridge T -> any to inspect it
    case fmt.Stringer:
        return x.String()
    case error:
        return "error: " + x.Error()
    default:
        return fmt.Sprintf("%v", x)
    }
}

type Point struct{ X, Y int }

func (p Point) String() string { return fmt.Sprintf("(%d,%d)", p.X, p.Y) }

func main() {
    fmt.Println(Format(Point{1, 2}))
    fmt.Println(Format(42))
}
Output
(1,2) 42

comparable has no ordering

The comparable constraint permits == and != only — it does not give you <. If you need ordering, use cmp.Ordered instead.

package main

import "fmt"

func AllEqual[T comparable](items []T) bool {
    for i := 1; i < len(items); i++ {
        if items[i] != items[0] { // == / != allowed; < would NOT compile
            return false
        }
    }
    return true
}

func main() {
    fmt.Println(AllEqual([]string{"a", "a", "a"}))
    fmt.Println(AllEqual([]int{1, 1, 2}))
}
Output
true false

Common Mistakes

Wrong
func f[T any](v T) {
    switch v.(type) { // compile error:
        // cannot type-switch on T
    case int:
    }
}
Right
func f[T any](v T) {
    switch any(v).(type) { // bridge to any
    case int:
    }
}

Key Rules

  • Type-switch on any (or an interface) — never on a bare type parameter T.
  • Convert with any(v) to bridge a generic value into a type switch.
  • comparable gives ==/!= only; use cmp.Ordered for </>.
  • Choose generics for compile-time type safety, interfaces for open-ended dynamic dispatch, type switches for a closed set of known types.
Practice Exercise

Write a generic Encode[T any](v T) string that converts T to any and type-switches: render int/float64 as numbers, bool as true/false, anything implementing fmt.Stringer via its String(), and fall back to %v. Then explain in a comment why you could not write the same dispatch by switching on T directly, and what the any(v) conversion costs (hint: it may force an allocation).

Pitfall: Expecting switch v.(type) to work on a type parameter T. It won't compile — within a generic function T is one fixed type per instantiation, not a runtime union. You must convert with any(v) (or constrain T to an interface) before switching.
Connections: Extends the constraint reasoning from generic data structures and complements the runtime fmt.Stringer / interface dispatch seen across earlier guides. The any(v) bridge ties back to escape analysis — interface conversions can allocate.
