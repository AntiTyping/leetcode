Type Parameters (Generics)

Functions & Control Flow • 7 days overdue • rep 2

Quick Reminder

Generics (Go 1.18+) let functions and types operate over a set of types via type parameters, written in square brackets before the regular parameter list: func Map[T any, U any](s []T, f func(T) U) []U. A constraint restricts which types are allowed. Use any for "any type" and comparable when the type must support ==/!= (required for map keys).

Key Pattern

package main

import "fmt"

// Map transforms []T into []U using f
func Map[T any, U any](s []T, f func(T) U) []U {
    out := make([]U, len(s))
    for i, v := range s {
        out[i] = f(v)
    }
    return out
}

// Filter keeps only elements where pred returns true
func Filter[T any](s []T, pred func(T) bool) []T {
    out := make([]T, 0, len(s))
    for _, v := range s {
        if pred(v) {
            out = append(out, v)
        }
    }
    return out
}

func main() {
    nums := []int{1, 2, 3, 4, 5}

    // T inferred as int, U inferred as string
    strs := Map(nums, func(n int) string {
        return fmt.Sprintf("#%d", n)
    })

    evens := Filter(nums, func(n int) bool { return n%2 == 0 })

    fmt.Println(strs)  // [#1 #2 #3 #4 #5]
    fmt.Println(evens) // [2 4]
}
Output
[#1 #2 #3 #4 #5] [2 4]
Challenge Exercise

Write a generic GroupBy[T any, K comparable](s []T, key func(T) K) map[K][]T that groups slice elements by a key function. For example, grouping a slice of words by their length should return map[int][]string. Note that the key type K must be comparable because it is used as a map key.

Pitfall: Using any where comparable is needed. A type parameter used as a map key or compared with == must be constrained by comparable, otherwise the code won't compile: map[K][]T requires K comparable.

Select Statement

Concurrency • 6 days overdue • rep 2

Quick Reminder

select multiplexes channel operations: it blocks until one of its cases is ready, then executes that case. If several are ready, one is chosen at random. A default case makes the select non-blocking — it runs immediately when no other case is ready. Combine with time.After for timeouts.

Key Pattern

package main

import (
    "fmt"
    "time"
)

func main() {
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() { time.Sleep(50 * time.Millisecond); ch1 <- "from ch1" }()
    go func() { time.Sleep(80 * time.Millisecond); ch2 <- "from ch2" }()

    for i := 0; i < 2; i++ {
        select {
        case msg := <-ch1:
            fmt.Println(msg)
        case msg := <-ch2:
            fmt.Println(msg)
        case <-time.After(200 * time.Millisecond):
            fmt.Println("timeout")
        }
    }
}
Output
from ch1 from ch2
Challenge Exercise

Implement merge(a, b <-chan int) <-chan int that combines two input channels into one output channel using select. When both inputs are closed, close the output. Handle the case where one channel closes before the other (a closed channel is always ready, so you must set its variable to nil to disable that case).

Pitfall: Forgetting to handle channel close in select. Reading from a closed channel returns the zero value immediately and is always ready, so a closed channel will spin the select in a tight loop. Detect close with v, ok := <-ch and set the channel variable to nil to disable that case.

Goroutines

Concurrency • 6 days overdue • rep 2

Quick Reminder

A goroutine is a lightweight thread managed by the Go runtime, launched with go func(){}(). They're multiplexed onto a small number of OS threads. Use sync.WaitGroup to wait for a set of goroutines to finish: wg.Add(n) before launching, defer wg.Done() inside each, and wg.Wait() to block until all complete.

Key Pattern

package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup
    results := make([]int, 5)

    for i := 0; i < 5; i++ {
        wg.Add(1)
        go func(idx int) {
            defer wg.Done()
            results[idx] = idx * idx
        }(i) // pass i as an argument to capture its value
    }

    wg.Wait()
    fmt.Println(results)
}
Output
[0 1 4 9 16]
Challenge Exercise

Write a parallel URL fetcher that limits concurrency to 3 goroutines using a buffered semaphore channel (sem := make(chan struct{}, 3)). Acquire a slot with sem <- struct{}{} before fetching and release it with <-sem in a deferred call. Collect results with a WaitGroup.

Pitfall: Goroutine leak — launching goroutines that block forever (e.g., on a channel send no one receives) without any way to stop them. Always give long-lived goroutines a cancellation path via context.Context or a done channel.

Channels

Concurrency • 6 days overdue • rep 2

Quick Reminder

Channels are typed conduits for communication. Unbuffered channels (make(chan int)) synchronize sender and receiver — a send blocks until a receive happens. Buffered channels (make(chan int, 10)) have capacity and only block when full/empty. close(ch) signals no more values; for v := range ch receives until the channel is closed and drained.

Key Pattern

package main

import "fmt"

func main() {
    ch := make(chan int, 3)

    // Producer
    go func() {
        for i := 1; i <= 3; i++ {
            ch <- i * 10
        }
        close(ch) // signal completion; only the sender closes
    }()

    // Consumer: range stops automatically when ch is closed
    for v := range ch {
        fmt.Println("received:", v)
    }

    // Receiving from a closed channel: ok is false
    v, ok := <-ch
    fmt.Printf("after close: v=%d ok=%v\n", v, ok)
}
Output
received: 10 received: 20 received: 30 after close: v=0 ok=false
Challenge Exercise

Implement a pipeline with four stages, each running in its own goroutine and connected by channels: generator (emits 1..N) → square (n → n*n) → filter (keeps even values) → printer. Each stage should close its output channel when its input is drained so downstream range loops terminate cleanly.

Pitfall: Sending on a closed channel panics. Only the sender should call close, and only once. If multiple goroutines send on the same channel, coordinate closing through a separate WaitGroup + a single closer goroutine, never close from a sender.

Context

Concurrency • 5 days overdue • rep 2

Quick Reminder

context.Context carries deadlines, cancellation signals, and request-scoped values across API boundaries. By convention it's the first parameter: func DoWork(ctx context.Context, ...). Derive children with context.WithCancel, WithTimeout, or WithDeadline. The returned cancel function must always be called (usually via defer) to release resources.

Key Pattern

package main

import (
    "context"
    "fmt"
    "time"
)

func slowOp(ctx context.Context) error {
    select {
    case <-time.After(2 * time.Second):
        return nil // finished work
    case <-ctx.Done():
        return ctx.Err() // context.DeadlineExceeded or Canceled
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 500*time.Millisecond)
    defer cancel() // always release resources

    if err := slowOp(ctx); err != nil {
        fmt.Println("operation failed:", err)
        return
    }
    fmt.Println("operation succeeded")
}
Output
operation failed: context deadline exceeded
Challenge Exercise

Write an HTTP handler that runs a simulated slow database query in a goroutine and cancels it if the client disconnects. Use r.Context() (which is canceled when the client goes away) and select on ctx.Done() versus a result channel. Return 503 with a useful message on cancellation.

Pitfall: Not calling cancel(). Even when a context expires naturally via its deadline, the associated timer and goroutine resources are only released when cancel() is called. go vet flags the "lostcancel" case — always defer cancel() immediately after creating the context.

Concurrency Patterns

Concurrency • 5 days overdue • rep 2

Quick Reminder

Three building blocks: Fan-out — multiple goroutines read from one channel to parallelize work. Fan-in — multiple channels merge into one. Worker pool — a fixed number of goroutines pull jobs from a shared jobs channel and push to a results channel. Pools cap resource usage while keeping throughput high.

Key Pattern

package main

import (
    "fmt"
    "sync"
)

func main() {
    const workers = 3
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    var wg sync.WaitGroup
    for w := 0; w < workers; w++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for job := range jobs {
                results <- job * job
            }
        }()
    }

    // Feed jobs
    for i := 1; i <= 9; i++ {
        jobs <- i
    }
    close(jobs)

    // Close results once all workers are done
    go func() { wg.Wait(); close(results) }()

    sum := 0
    for r := range results {
        sum += r
    }
    fmt.Println("sum of squares:", sum)
}
Output
sum of squares: 285
Challenge Exercise

Implement a rate-limited worker pool that processes at most 10 requests per second. Combine a fixed worker pool with a time.Ticker (or golang.org/x/time/rate.Limiter) so each job waits for a token before being processed. Verify the throughput cap by timing how long 30 jobs take (should be ~3 seconds).

Pitfall: Not closing the results channel. The consumer's for r := range results blocks forever if nothing closes results. Use a dedicated goroutine: go func() { wg.Wait(); close(results) }() — close only after every worker has finished sending.

Graceful Shutdown

Production Go

Why this matters: Production servers must drain in-flight requests and clean up resources (database connections, file handles, message acknowledgements) before stopping. An abrupt exit drops live connections, corrupts partially written data, and produces noisy errors for clients mid-request.

The Idea

Graceful shutdown is a three-step dance: (1) catch the OS termination signal (SIGINT from Ctrl+C, SIGTERM from an orchestrator like Kubernetes), (2) stop accepting new work, and (3) wait for in-flight work to finish, bounded by a timeout so a stuck request can't hang shutdown forever. Go 1.16+ provides signal.NotifyContext, which returns a context that is canceled when the named signals arrive — this lets you thread cancellation through your whole program with the same context.Context you already use.

Example 1: Basic Signal Handling

package main

import (
    "context"
    "fmt"
    "os"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    // ctx is canceled when SIGINT or SIGTERM arrives
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop() // restore default signal behavior

    fmt.Println("running... press Ctrl+C to stop")

    select {
    case <-ctx.Done():
        fmt.Println("\nsignal received, shutting down")
    case <-time.After(30 * time.Second):
        fmt.Println("ran to completion")
    }
}
Output (after Ctrl+C)
running... press Ctrl+C to stop ^C signal received, shutting down

Example 2: HTTP Server Graceful Shutdown

package main

import (
    "context"
    "errors"
    "log"
    "net/http"
    "os/signal"
    "syscall"
    "time"
)

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        time.Sleep(2 * time.Second) // simulate slow work
        w.Write([]byte("done\n"))
    })

    srv := &http.Server{Addr: ":8080", Handler: mux}

    // Run the server in a goroutine so main can wait for signals
    go func() {
        if err := srv.ListenAndServe(); err != nil &&
            !errors.Is(err, http.ErrServerClosed) {
            log.Fatalf("listen: %v", err)
        }
    }()
    log.Println("server listening on :8080")

    // Block until a signal arrives
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()
    <-ctx.Done()
    log.Println("shutting down...")

    // Give in-flight requests up to 10s to finish
    shutdownCtx, cancel := context.WithTimeout(context.Background(),
        10*time.Second)
    defer cancel()

    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Printf("forced shutdown: %v", err)
    }
    log.Println("server stopped cleanly")
}

srv.Shutdown stops accepting new connections, then waits for active requests to complete. If shutdownCtx expires first, it returns an error and any still-active connections are closed abruptly.

Example 3: Draining a Worker Pool

package main

import (
    "context"
    "fmt"
    "os/signal"
    "sync"
    "syscall"
)

func worker(ctx context.Context, id int, jobs <-chan int, wg *sync.WaitGroup) {
    defer wg.Done()
    for {
        select {
        case <-ctx.Done():
            fmt.Printf("worker %d: stop signal, draining\n", id)
            // Finish remaining buffered jobs, then exit
            for job := range jobs {
                fmt.Printf("worker %d drained job %d\n", id, job)
            }
            return
        case job, ok := <-jobs:
            if !ok {
                return
            }
            fmt.Printf("worker %d did job %d\n", id, job)
        }
    }
}

func main() {
    ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT)
    defer stop()

    jobs := make(chan int, 10)
    var wg sync.WaitGroup
    for i := 1; i <= 3; i++ {
        wg.Add(1)
        go worker(ctx, i, jobs, &wg)
    }

    for i := 1; i <= 5; i++ {
        jobs <- i
    }
    close(jobs)
    wg.Wait()
    fmt.Println("all workers drained")
}

Example 4: Coordinating Multiple Subsystems

package main

import (
    "context"
    "log"
    "os/signal"
    "syscall"
    "time"
)

type Subsystem struct {
    Name     string
    Shutdown func(context.Context) error
}

func main() {
    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    subsystems := []Subsystem{
        {"http-server", func(c context.Context) error {
            time.Sleep(200 * time.Millisecond); return nil
        }},
        {"db-pool", func(c context.Context) error {
            time.Sleep(100 * time.Millisecond); return nil
        }},
        {"metrics", func(c context.Context) error {
            time.Sleep(50 * time.Millisecond); return nil
        }},
    }

    <-ctx.Done()
    log.Println("shutting down subsystems (reverse order)")

    shutdownCtx, cancel := context.WithTimeout(context.Background(),
        15*time.Second)
    defer cancel()

    // Shut down in reverse order of startup
    for i := len(subsystems) - 1; i >= 0; i-- {
        s := subsystems[i]
        if err := s.Shutdown(shutdownCtx); err != nil {
            log.Printf("%s shutdown error: %v", s.Name, err)
        } else {
            log.Printf("%s stopped", s.Name)
        }
    }
}

Common Mistakes

Don't
// Hard exit drops in-flight requests
func handleSignal() {
    <-sigChan
    os.Exit(0) // connections killed!
}
Do
// signal -> context cancel ->
// Shutdown(timeoutCtx) -> clean exit
<-ctx.Done()
srv.Shutdown(timeoutCtx)
// os.Exit happens naturally as
// main returns
Pitfall: Calling os.Exit() directly from a signal handler skips all deferred cleanup and aborts in-flight requests. Let the signal cancel a context, perform a bounded Shutdown, then return from main so deferred cleanup runs.
Exercises
  • Write a TCP echo server (net.Listen("tcp", ...)) with graceful shutdown: on signal, stop accepting new connections via listener.Close() but let active connections finish their current echo loop. Track active conns with a WaitGroup.
  • Add a shutdown hook that flushes a buffered writer (bufio.Writer) to disk before exit, ensuring no buffered bytes are lost when the process stops.
Connections: Graceful shutdown ties together Context (cancellation), Goroutines + WaitGroup (waiting for in-flight work), and Select (racing ctx.Done() against work). It's the production endgame for everything in the concurrency review above.

Configuration Management

Production Go

Why this matters: Hard-coded values make deployment painful and unsafe — you can't change a database URL or a log level without recompiling, and secrets end up in source control. Proper configuration cleanly supports dev/staging/prod environments and follows the 12-factor principle of storing config in the environment.

The Idea

Load all configuration once at startup into a single typed struct, validate it, and pass it explicitly to the components that need it. The canonical sources, in increasing sophistication: plain environment variables with defaults, struct-tag binding (envconfig), and layered config that merges files + env + flags (viper). Whichever you choose, fail fast: if required config is missing or invalid, the process should refuse to start with a clear error.

Example 1: os.Getenv with Defaults

package main

import (
    "fmt"
    "os"
    "strconv"
)

func getEnv(key, fallback string) string {
    if v, ok := os.LookupEnv(key); ok {
        return v
    }
    return fallback
}

func getEnvInt(key string, fallback int) int {
    if v, ok := os.LookupEnv(key); ok {
        if n, err := strconv.Atoi(v); err == nil {
            return n
        }
    }
    return fallback
}

func main() {
    port := getEnvInt("PORT", 8080)
    logLevel := getEnv("LOG_LEVEL", "info")
    fmt.Printf("port=%d logLevel=%s\n", port, logLevel)
}
Output (PORT unset, LOG_LEVEL=debug)
port=8080 logLevel=debug

Example 2: Config Struct with envconfig Tags

package main

import (
    "fmt"
    "log"

    "github.com/kelseyhightower/envconfig"
)

type Config struct {
    Port     int    `envconfig:"PORT" default:"8080"`
    DBURL    string `envconfig:"DB_URL" required:"true"`
    LogLevel string `envconfig:"LOG_LEVEL" default:"info"`
}

func main() {
    var cfg Config
    // Reads APP_PORT, APP_DB_URL, APP_LOG_LEVEL from the environment
    if err := envconfig.Process("APP", &cfg); err != nil {
        log.Fatalf("config: %v", err)
    }
    fmt.Printf("%+v\n", cfg)
}

The required:"true" tag makes startup fail with a clear error if APP_DB_URL is missing — exactly the fail-fast behavior you want.

Example 3: Viper with YAML + Env Override

package main

import (
    "fmt"
    "log"

    "github.com/spf13/viper"
)

func main() {
    viper.SetConfigName("config") // config.yaml
    viper.SetConfigType("yaml")
    viper.AddConfigPath(".")

    // Defaults
    viper.SetDefault("port", 8080)
    viper.SetDefault("log_level", "info")

    // Env vars override file values: APP_PORT, APP_LOG_LEVEL
    viper.SetEnvPrefix("APP")
    viper.AutomaticEnv()

    if err := viper.ReadInConfig(); err != nil {
        log.Printf("no config file, using defaults/env: %v", err)
    }

    fmt.Println("port:", viper.GetInt("port"))
    fmt.Println("log_level:", viper.GetString("log_level"))
}

Precedence (highest first): explicit Set > flags > env > config file > defaults. This layering is what lets the same binary behave differently per environment.

Example 4: Config Validation at Startup

package main

import (
    "fmt"
    "log"
)

type Config struct {
    Port     int
    DBURL    string
    LogLevel string
}

func (c Config) Validate() error {
    if c.Port < 1 || c.Port > 65535 {
        return fmt.Errorf("port %d out of range 1-65535", c.Port)
    }
    if c.DBURL == "" {
        return fmt.Errorf("DB_URL is required")
    }
    switch c.LogLevel {
    case "debug", "info", "warn", "error":
    default:
        return fmt.Errorf("invalid log level %q", c.LogLevel)
    }
    return nil
}

func main() {
    cfg := Config{Port: 99999, DBURL: "", LogLevel: "trace"}
    if err := cfg.Validate(); err != nil {
        log.Fatalf("invalid config: %v", err)
    }
    fmt.Println("config OK")
}
Output
invalid config: port 99999 out of range 1-65535

Common Mistakes

Don't
// os.Getenv scattered everywhere
func handler() {
    db := os.Getenv("DB_URL")
    timeout := os.Getenv("TIMEOUT")
    // untyped, unvalidated,
    // re-read on every call
}
Do
// Load once into a typed struct
cfg := MustLoadConfig()
cfg.Validate()
// pass cfg explicitly to
// components that need it
srv := NewServer(cfg)
Pitfall: Reading os.Getenv scattered throughout the codebase makes config invisible and untestable, and silently treats a missing variable as the empty string. Load everything once at startup into a typed, validated struct.
Exercises
  • Create a Config struct for a web app with Port int, DBURL string, and LogLevel string. Load it from environment variables (with sensible defaults) using either hand-rolled os.LookupEnv helpers or envconfig.
  • Add a Validate() method that enforces the port range, requires DBURL, and restricts LogLevel to a known set. Make main exit with a clear message when validation fails.
Connections: A validated config struct is the natural place to derive the slog level (next concept) and the timeouts you pass into Context and Graceful Shutdown. Config is the single source of truth that wires the whole program together.

Structured Logging with log/slog

Production Go

Why this matters: Structured logging emits machine-parseable key/value records (typically JSON) instead of free-form strings, which makes logs searchable, filterable, and alertable in tools like Loki, Datadog, or CloudWatch. Since Go 1.21, log/slog brings this into the standard library — no third-party logger required.

The Idea

A slog.Logger writes records through a Handler that controls format and minimum level. Log with key/value pairs: slog.Info("user created", "user_id", id, "email", email). Build a contextual logger with logger.With("request_id", id) so those fields appear on every subsequent line. Pick slog.NewTextHandler for human-readable dev output and slog.NewJSONHandler for production.

Example 1: Basic slog Usage

package main

import "log/slog"

func main() {
    slog.Info("server started", "port", 8080, "env", "prod")
    slog.Warn("disk space low", "free_mb", 512)
    slog.Error("db connection failed", "attempt", 3)
}
Output (default text handler)
2026/06/23 10:00:00 INFO server started port=8080 env=prod 2026/06/23 10:00:00 WARN disk space low free_mb=512 2026/06/23 10:00:00 ERROR db connection failed attempt=3

Example 2: JSON Handler with Level from Config

package main

import (
    "log/slog"
    "os"
)

func main() {
    handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelDebug, // minimum level to emit
    })
    logger := slog.New(handler)
    slog.SetDefault(logger) // make it the package-level default

    slog.Debug("cache miss", "key", "user:42")
    slog.Info("user created", "user_id", 42, "email", "a@b.com")
}
Output (JSON)
{"time":"2026-06-23T10:00:00Z","level":"DEBUG","msg":"cache miss","key":"user:42"} {"time":"2026-06-23T10:00:00Z","level":"INFO","msg":"user created","user_id":42,"email":"a@b.com"}

Example 3: Adding Context with slog.With

package main

import (
    "log/slog"
    "os"
)

func main() {
    base := slog.New(slog.NewJSONHandler(os.Stdout, nil))

    // Derive a logger that stamps every line with these fields
    reqLog := base.With("request_id", "abc-123", "method", "GET")

Example 4: Grouping and Typed Attributes

package main

import (
    "log/slog"
    "os"
    "time"
)

func main() {
    logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))

    // slog.Group nests related fields; typed attrs avoid alloc
    logger.Info("http request",
        slog.Group("req",
            slog.String("method", "POST"),

Example 5: Request-Scoped Logger via Context

package main

import (
    "context"
    "log/slog"
    "net/http"
    "os"
)

type ctxKey struct{}

func withLogger(ctx context.Context, l *slog.Logger) context.Context {
    return context.WithValue(ctx, ctxKey{}, l)
}

func loggerFrom(ctx context.Context) *slog.Logger {
    if l, ok := ctx.Value(ctxKey{}).(*slog.Logger); ok {
        return l
    }
    return slog.Default()
}

// Middleware injects a request-scoped logger with a request_id
func requestLogger(base *slog.Logger) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            l := base.With("request_id", r.Header.Get("X-Request-ID"))
            ctx := withLogger(r.Context(), l)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

func handler(w http.ResponseWriter, r *http.Request) {
    loggerFrom(r.Context()).Info("handling", "path", r.URL.Path)
    w.Write([]byte("ok\n"))
}

func main() {
    base := slog.New(slog.NewJSONHandler(os.Stdout, nil))
    h := requestLogger(base)(http.HandlerFunc(handler))
    http.ListenAndServe(":8080", h)
}

Common Mistakes

Don't
// Unstructured, unsearchable
fmt.Printf("user %d created with "+
    "email %s\n", id, email)
Do
// Structured key/value pairs
slog.Info("user created",
    "user_id", id,
    "email", email)
Pitfall: Passing an odd number of arguments after the message produces a dangling key with no value, which slog records as a !BADKEY attribute. Always pass complete key/value pairs, or use typed attrs (slog.Int, slog.String) to make the pairing explicit and avoid the mistake entirely.
Exercises
  • Set up slog with a JSON handler whose minimum level is read from an environment variable (LOG_LEVEL=debug|info|warn|error). Map the string to a slog.Level and pass it via HandlerOptions.
  • Create HTTP middleware that generates a request ID, attaches a slog.With("request_id", id) logger to the request context, and logs the method, path, status, and duration of every request.
Connections: The request-scoped logger pattern reuses Context value passing, and the middleware mirrors the middleware-chain pattern from prior days. The minimum level comes straight from Configuration Management. Together these three Production Go concepts form the operational backbone of a real Go service.
New Concepts

Memory & Performance

Escape Analysis & Stack vs Heap

Memory & Performance

Why this matters: Stack allocations are essentially free — they're reclaimed automatically when a function returns and never touch the garbage collector. Heap allocations cost a GC scan and add pressure that can dominate the runtime of a hot path. Understanding which is which lets you write faster code without guessing.

The Idea

The Go compiler decides at compile time whether each variable can live on the goroutine's stack or must "escape" to the heap. A value escapes when the compiler can't prove its lifetime ends with the function — most commonly when you return a pointer to a local, store it in an interface, capture it in a closure that outlives the call, or send it through a channel. Inspect the compiler's decisions with go build -gcflags='-m'.

Example 1: Stack-Allocated Local

package main

import "fmt"

// sum's locals never escape: total and the loop var
// stay on the stack.
func sum(nums []int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total // returning a value (copy), not a pointer
}

func main() {
    fmt.Println(sum([]int{1, 2, 3, 4})) // 10
}

Example 2: Heap-Escaped Pointer Return

package main

type User struct {
    Name string
    Age  int
}

// The returned *User must outlive newUser, so u escapes
// to the heap.
func newUser(name string, age int) *User {
    u := User{Name: name, Age: age} // u escapes
    return &u
}

func main() {
    _ = newUser("Alice", 30)
}
Output (go build -gcflags='-m')
./main.go:11:2: moved to heap: u

Even though u is a local variable, returning its address forces it onto the heap so the caller's pointer stays valid.

Example 3: Interface Boxing Escape

package main

import "fmt"

func main() {
    x := 42
    // Passing x to a function that takes ...any boxes it
    // into an interface value, which escapes to the heap.
    fmt.Println(x) // x escapes to heap
}
Output (go build -gcflags='-m')
./main.go:9:13: ... argument does not escape ./main.go:9:14: x escapes to heap

Converting a concrete value to an interface (here, any for the variadic Println) typically allocates, because the interface must hold a pointer to the value.

Example 4: Analyzing with -gcflags='-m'

$ go build -gcflags='-m' ./...
# example/app
./main.go:11:2: moved to heap: u
./main.go:18:13: inlining call to fmt.Println
./main.go:9:14: x escapes to heap
./main.go:9:13: ... argument does not escape

# Add -m -m for the reasoning behind each decision
$ go build -gcflags='-m -m' ./...

Read the output bottom-up per function. "moved to heap" and "escapes to heap" are the lines to hunt for in a hot path; "does not escape" confirms a stack allocation.

Common Mistakes

Don't
// "Pointers are faster" cargo cult
func add(a, b *int) *int {
    r := *a + *b
    return &r // forces heap alloc
}
Do
// Value types stay on the stack
func add(a, b int) int {
    return a + b // no alloc
}
// Use pointers only when you need
// to mutate or share large data
Pitfall: Reaching for pointers "for performance" by default. For small value types, copying is cheaper than the heap allocation + GC work a pointer can trigger. Measure with -gcflags='-m' and benchmarks before optimizing; use pointers when you genuinely need mutation or want to avoid copying a large struct.
Exercises
  • Write two versions of a function that produces a small struct: one that returns the struct by value and one that returns *Struct. Predict which allocates on the heap, then run go build -gcflags='-m' to confirm.
  • Take a function with a []byte local and observe how it changes from "does not escape" to "escapes to heap" when you return a slice of it versus copy into a freshly allocated slice.
Connections: Escape analysis explains why the sync.Pool pattern (next concept) pays off — it targets exactly the heap-escaping objects this analysis reveals. It also informs the value-vs-pointer receiver choice you make on every method.

sync.Pool & Reducing Allocations

Memory & Performance

Why this matters: High-throughput servers can spend a surprising fraction of their time in the garbage collector. sync.Pool recycles short-lived objects across calls so they don't have to be reallocated and re-collected on every request, cutting allocation pressure and GC pauses on hot paths.

The Idea

sync.Pool is a concurrency-safe set of temporary, interchangeable objects. Get() returns an object from the pool, or calls the pool's New function if it's empty. Put() returns an object for later reuse. Crucially, the pool may discard its contents at any GC cycle, so it's only suitable for objects you can recreate cheaply, never for caching or for anything whose loss matters.

Example 1: Basic Pool with bytes.Buffer

package main

import (
    "bytes"
    "fmt"
    "sync"
)

var bufPool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func render(name string) string {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset()          // clear any leftover state from prior use
    defer bufPool.Put(buf)

    buf.WriteString("Hello, ")
    buf.WriteString(name)
    return buf.String()
}

func main() {
    fmt.Println(render("Alice"))
    fmt.Println(render("Bob"))
}
Output
Hello, Alice Hello, Bob

Example 2: Pool for JSON Encoding

package main

import (
    "bytes"
    "encoding/json"
    "fmt"
    "sync"
)

var encPool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func marshal(v any) ([]byte, error) {
    buf := encPool.Get().(*bytes.Buffer)
    buf.Reset()
    defer encPool.Put(buf)

    if err := json.NewEncoder(buf).Encode(v); err != nil {
        return nil, err
    }
    // Copy out: the buffer goes back to the pool, so we can't
    // return its internal bytes directly.
    out := make([]byte, buf.Len())
    copy(out, buf.Bytes())
    return out, nil
}

func main() {
    b, _ := marshal(map[string]int{"a": 1, "b": 2})
    fmt.Print(string(b))
}
Output
{"a":1,"b":2}

Example 3: Benchmark Showing Pool vs No-Pool

package main

import (
    "bytes"
    "sync"
    "testing"
)

var pool = sync.Pool{New: func() any { return new(bytes.Buffer) }}

func BenchmarkNoPool(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        buf := new(bytes.Buffer)
        buf.WriteString("hello world")
        _ = buf.String()
    }
}

func BenchmarkPool(b *testing.B) {
    b.ReportAllocs()
    for i := 0; i < b.N; i++ {
        buf := pool.Get().(*bytes.Buffer)
        buf.Reset()
        buf.WriteString("hello world")
        _ = buf.String()
        pool.Put(buf)
    }
}
Output (go test -bench=. -benchmem)
BenchmarkNoPool-8 20000000 62.1 ns/op 64 B/op 1 allocs/op BenchmarkPool-8 50000000 24.8 ns/op 0 B/op 0 allocs/op

The pooled version eliminates the per-iteration buffer allocation, roughly halving the time and dropping to zero allocations for the buffer itself.

Example 4: Reset Before Put for a Custom Struct

package main

import (
    "fmt"
    "sync"
)

type Request struct {
    Headers map[string]string
    Body    []byte
}

func (r *Request) reset() {
    // Clear fields so a recycled Request carries no stale data
    for k := range r.Headers {
        delete(r.Headers, k)
    }
    r.Body = r.Body[:0]
}

var reqPool = sync.Pool{
    New: func() any {
        return &Request{Headers: make(map[string]string)}
    },
}

func handle(payload string) {
    req := reqPool.Get().(*Request)
    defer func() {
        req.reset()      // reset BEFORE returning to the pool
        reqPool.Put(req)
    }()

    req.Headers["Content-Type"] = "text/plain"
    req.Body = append(req.Body, payload...)
    fmt.Printf("handled %d bytes, %d headers\n",
        len(req.Body), len(req.Headers))
}

func main() {
    handle("first")
    handle("second")
}
Output
handled 5 bytes, 1 headers handled 6 bytes, 1 headers

Common Mistakes

Don't
// Put back without resetting:
// next Get sees stale data
buf := pool.Get().(*bytes.Buffer)
buf.WriteString(data)
pool.Put(buf) // still full!
Do
// Reset state before reuse
buf := pool.Get().(*bytes.Buffer)
buf.Reset()
buf.WriteString(data)
// ... use buf ...
buf.Reset()
pool.Put(buf)
Pitfall: Putting objects back without resetting their state. The next Get will hand out an object still carrying the previous user's data — leftover bytes in a buffer or stale map entries — causing subtle data corruption or even leaking one request's data into another. Always reset (e.g. buf.Reset()) right before Put.
Exercises
  • Create a pool of []byte buffers (e.g. make([]byte, 0, 4096)) used inside an HTTP handler that reads and echoes the request body. Write a -benchmem benchmark proving the pooled handler allocates fewer bytes per request than a naive version.
  • Implement a pool for a custom struct with a slice and a map field, including a correct reset() that truncates the slice (s[:0]) and clears the map. Verify with a test that recycled objects never leak prior state.
Connections: sync.Pool is the direct remedy for the heap escapes surfaced by Escape Analysis, and its concurrency safety builds on the Goroutines and channel discipline from the review. In a real server it pairs naturally with the per-request buffers used in structured logging and JSON response encoding.
