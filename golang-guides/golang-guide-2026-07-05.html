gRPC & Protocol Buffers

Advanced Networking • due today • rep 2

Quick Reminder

Protocol Buffers are a compact, schema-first binary serialization format: you declare messages and RPC services in a .proto file, then protoc (with protoc-gen-go and protoc-gen-go-grpc) generates Go structs plus strongly-typed client and server stubs. gRPC runs those services over HTTP/2 with four call styles — unary, server-streaming, client-streaming, and bidirectional. The server implements the generated interface; the client dials a *grpc.ClientConn and calls methods as if they were local.

Key Pattern

// greeter.proto
syntax = "proto3";
package greet;
option go_package = "example.com/greet";

message HelloRequest  { string name = 1; }
message HelloReply    { string message = 1; }

service Greeter {
  rpc SayHello(HelloRequest) returns (HelloReply);
}

// generate:
//   protoc --go_out=. --go-grpc_out=. greeter.proto
// server.go — implement the generated interface
type server struct {
    greet.UnimplementedGreeterServer // forward compatibility
}

func (s *server) SayHello(ctx context.Context,
    r *greet.HelloRequest) (*greet.HelloReply, error) {
    return &greet.HelloReply{Message: "hello, " + r.Name}, nil
}

func main() {
    lis, _ := net.Listen("tcp", ":50051")
    gs := grpc.NewServer()
    greet.RegisterGreeterServer(gs, &server{})
    gs.Serve(lis)
}
Challenge Exercise

Extend the Greeter service with a rpc SayHelloStream(HelloRequest) returns (stream HelloReply) server-streaming method that emits five greetings, and write a client that ranges over the stream until io.EOF. Add a unary interceptor that logs each method name and latency with slog, and propagate a deadline with context.WithTimeout so a slow server returns codes.DeadlineExceeded. Finally, explain why embedding UnimplementedGreeterServer matters when you add a new RPC to the .proto.

#1 Pitfall: Editing the generated *.pb.go files by hand — they are overwritten on every protoc run. Also, reusing field numbers or changing a field's type in the .proto breaks wire compatibility with already-deployed clients; only ever add new fields with fresh numbers and treat old ones as reserved.

Module Management: go.mod, Workspaces & Versioning

Production Go • overdue (23d) • rep 0

Quick Reminder

A module is a versioned collection of packages rooted at a go.mod file that names the module path and its dependency requirements. go get adds/updates requirements; go mod tidy prunes unused ones and fills in the go.sum checksums. Semantic Import Versioning means v2+ majors carry the version in the import path (.../v2). For local multi-module development, a go.work file (Go 1.18+) stitches several modules together without editing each go.mod.

Key Pattern

// go.mod
module example.com/app

go 1.24

require (
    github.com/google/uuid v1.6.0
    golang.org/x/sync v0.10.0
)
# everyday module commands
go mod init example.com/app     # create go.mod
go get github.com/google/uuid@latest
go mod tidy                     # sync require + go.sum
go list -m all                  # full dependency graph

# a workspace spanning two local modules
go work init ./app ./lib
go work use ./newmod            # add another module
Challenge Exercise

Create two local modules — a library and an app that imports it — and wire them together with a go.work file so edits to the library are picked up without publishing or a replace directive. Then bump the library to a breaking v2, update its module path to end in /v2, and fix the importing app. Explain the difference between a replace directive in go.mod (committed, affects everyone) and a go.work file (local, usually git-ignored), and when each is appropriate.

#1 Pitfall: Committing a go.work file to a shared repo — it silently overrides everyone's module resolution and breaks CI that expects tagged versions. Keep go.work local (git-ignore it). Also, forgetting the /v2 suffix on a major bump makes Go treat it as the same module, causing baffling "ambiguous import" errors.

Cross-Compilation & Build Tags

Production Go • overdue (23d) • rep 0

Quick Reminder

Go cross-compiles from any host to any target by setting GOOS and GOARCH — no cross toolchain needed, as long as you stay pure-Go (CGO_ENABLED=0). Build constraints select which files compile for a target: a filename suffix like _linux.go / _windows.go, or a //go:build line at the top of a file. This is how you keep OS-specific code (syscalls, path handling) isolated and still ship a single cross-platform binary per target.

Key Pattern

//go:build linux || darwin

package main

// Compiled only on Linux/macOS. A matching
// file with "//go:build windows" holds the
// Windows implementation of the same funcs.
# cross-compile a static Linux arm64 binary from any host
CGO_ENABLED=0 GOOS=linux GOARCH=arm64 \
    go build -o app-linux-arm64 .

# list every supported target pair
go tool dist list

# build with a custom tag (go build -tags prod)
#   //go:build prod
Challenge Exercise

Write a platform.go pair using //go:build constraints so that ConfigDir() returns %APPDATA% on Windows and $XDG_CONFIG_HOME (falling back to ~/.config) elsewhere, and confirm each file only compiles for its target with go vet. Then add a custom debug build tag that enables extra logging only when built with -tags debug. Finally, cross-compile the same program for linux/amd64, darwin/arm64, and windows/amd64, and explain why CGO_ENABLED=0 is usually required for painless cross-builds.

#1 Pitfall: Leaving CGO_ENABLED=1 (the default when a C toolchain is present) while cross-compiling — the build then needs a target C cross-compiler and often fails or produces a dynamically-linked binary that won't run in a scratch container. Also, a malformed //go:build line (or the old // +build without a blank line after it) is silently ignored, so the file compiles everywhere unexpectedly.

WebSocket in Go

Advanced Networking • overdue (22d) • rep 0

Quick Reminder

WebSockets upgrade a normal HTTP request into a persistent, full-duplex TCP connection so server and client can push messages at any time. The standard library has no WebSocket server, so you use a library such as github.com/coder/websocket (formerly nhooyr) or gorilla/websocket. The server calls Accept inside an http.HandlerFunc to complete the upgrade, then reads and writes typed messages (text/binary) on the resulting connection until either side closes it.

Key Pattern

import (
    "context"
    "net/http"
    "time"

    "github.com/coder/websocket"
)

func echo(w http.ResponseWriter, r *http.Request) {
    c, err := websocket.Accept(w, r, nil) // upgrade
    if err != nil {
        return
    }
    defer c.Close(websocket.StatusInternalError, "bye")

    ctx, cancel := context.WithTimeout(r.Context(), time.Minute)
    defer cancel()

    for {
        typ, data, err := c.Read(ctx) // blocks for next frame
        if err != nil {
            return // client closed or timed out
        }
        c.Write(ctx, typ, data) // echo it back
    }
}
Challenge Exercise

Build a broadcast chat hub: a central goroutine owns a map[*websocket.Conn]struct{} and fans each incoming message out to all connected clients over a buffered per-client send channel (so one slow reader can't stall everyone). Add a ping/keepalive using the library's ping helper and drop clients that don't pong within a deadline. Then handle graceful shutdown by closing every connection with websocket.StatusGoingAway. Explain why you must never write to the same connection from two goroutines concurrently.

#1 Pitfall: Writing to a single WebSocket connection from multiple goroutines at once — frames interleave and corrupt the stream. Funnel all writes for a connection through one owner goroutine (or a send channel). Also, forgetting to enforce an origin check or a read deadline leaves you open to cross-site hijacking and slow-loris style resource exhaustion.

Profiling with pprof

Observability & Profiling • overdue (22d) • rep 0

Quick Reminder

pprof collects and visualizes CPU, heap, goroutine, block, and mutex profiles. In a server, blank-import net/http/pprof to expose profiles at /debug/pprof/; for one-off analysis, add CPU/memory profile flags to testing benchmarks (-cpuprofile, -memprofile). Analyze with go tool pprof — use top to find the hottest functions, list Func for line-level cost, and -http=:0 for an interactive flame graph in the browser.

Key Pattern

import _ "net/http/pprof" // registers /debug/pprof/ handlers

func main() {
    // profiles now live under http://localhost:6060/debug/pprof/
    go http.ListenAndServe("localhost:6060", nil)
    // ... rest of your server ...
}
# 30-second CPU profile from a running server, then flame graph
go tool pprof -http=:0 \
    http://localhost:6060/debug/pprof/profile?seconds=30

# heap profile from a benchmark
go test -bench=. -memprofile=mem.out
go tool pprof -http=:0 mem.out
#   (pprof) top10        # hottest allocators
#   (pprof) list MyFunc  # line-by-line cost
Challenge Exercise

Write a deliberately wasteful function (e.g. string concatenation in a loop plus needless allocations), expose net/http/pprof, and capture both a CPU and a heap profile under load generated with a simple driver goroutine. Use go tool pprof's top and list to pinpoint the hot line, fix it (strings.Builder, preallocation), and confirm the profile improved. Then capture a goroutine profile while goroutines are blocked on a channel and explain how it reveals a leak. Note why block/mutex profiles require explicit runtime.SetBlockProfileRate/SetMutexProfileFraction.

#1 Pitfall: Exposing /debug/pprof/ on a public interface — profiles leak internal details and let anyone trigger expensive captures (a DoS vector). Bind it to localhost or an internal-only port. Also, profiling a workload that isn't representative (too short, no load) gives noise; profile under realistic traffic for long enough to gather meaningful samples.

OpenTelemetry & Tracing

Observability & Profiling • overdue (22d) • rep 0

Quick Reminder

Distributed tracing follows one request across services as a trace made of nested spans. OpenTelemetry (OTel) is the vendor-neutral standard: you configure a TracerProvider with an exporter (OTLP to a collector, Jaeger, stdout for dev), get a Tracer, and wrap units of work with tracer.Start(ctx, "name"). The returned context.Context carries the active span, so propagating ctx down the call chain (and across HTTP/gRPC via propagators) is what stitches spans into one trace.

Key Pattern

import (
    "context"
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
)

func handleOrder(ctx context.Context, id string) error {
    tr := otel.Tracer("orders")
    ctx, span := tr.Start(ctx, "handleOrder")
    defer span.End() // always end the span

    span.SetAttributes(attribute.String("order.id", id))

    if err := charge(ctx, id); err != nil { // child span sees ctx
        span.RecordError(err)
        return err
    }
    return nil
}
Challenge Exercise

Set up a TracerProvider with the stdout exporter and instrument a two-function call chain so the second function's span nests under the first — prove nesting by passing ctx through. Add attributes and an event to a span, and record an error path with span.RecordError plus span.SetStatus(codes.Error, ...). Then wrap an http.Handler with otelhttp and a client with the matching transport so the trace context propagates across a real HTTP call. Explain why forgetting defer span.End() leaks unfinished spans.

#1 Pitfall: Not propagating ctx — if you call the next function with context.Background() instead of the span-carrying context, the child span starts a brand-new trace and the connection is lost. Also, forgetting defer span.End() means spans never flush; and creating a span per trivial call floods your backend, so span granularity should match meaningful units of work.

Section 1 — Cancellation, Pipes & Timers

context.WithCancelCause & context.Cause (Go 1.20)

Concurrency

Why it matters: plain context.WithCancel tells you a context was cancelled but not why — ctx.Err() only ever returns the generic context.Canceled or context.DeadlineExceeded. When a pipeline stage aborts, you usually want the real reason (which stage failed, with which error) to bubble up. context.WithCancelCause lets the canceller attach an explanatory error, and context.Cause(ctx) retrieves it — turning cancellation into a proper error channel.

Cancelling with a cause

context.WithCancelCause returns a CancelCauseFunc that takes an error. After cancellation, ctx.Err() still reports context.Canceled, but context.Cause(ctx) returns the error you passed (or ctx.Err() if you passed nil).

package main

import (
    "context"
    "errors"
    "fmt"
)

var errQuotaExceeded = errors.New("quota exceeded")

func main() {
    ctx, cancel := context.WithCancelCause(context.Background())

    // Some worker decides to abort with a specific reason.
    cancel(errQuotaExceeded)

    fmt.Println("Err:  ", ctx.Err())              // context canceled
    fmt.Println("Cause:", context.Cause(ctx))     // quota exceeded
    fmt.Println("match:", errors.Is(context.Cause(ctx), errQuotaExceeded))
}
Output
Err: context canceled Cause: quota exceeded match: true

Deadlines carry a cause too

context.WithTimeoutCause / WithDeadlineCause (Go 1.21) attach a cause that surfaces when the timeout fires, so you can distinguish "this specific operation's SLA" from a generic deadline.

package main

import (
    "context"
    "errors"
    "fmt"
    "time"
)

var errSlowUpstream = errors.New("upstream SLA exceeded")

func main() {
    ctx, cancel := context.WithTimeoutCause(
        context.Background(), 10*time.Millisecond, errSlowUpstream)
    defer cancel()

    <-ctx.Done()
    fmt.Println("Err:  ", ctx.Err())            // context deadline exceeded
    fmt.Println("Cause:", context.Cause(ctx))   // upstream SLA exceeded
}
Output
Err: context deadline exceeded Cause: upstream SLA exceeded

Common Mistakes

Wrong
// Smuggling the reason through a shared variable and
// a plain WithCancel -- racy and easy to desync.
var failReason error
ctx, cancel := context.WithCancel(parent)
failReason = errQuotaExceeded // data race vs. readers
cancel()
// downstream can only see context.Canceled, not why
Right
ctx, cancel := context.WithCancelCause(parent)
// The reason travels WITH the context, safely.
cancel(errQuotaExceeded)

// Anywhere downstream:
if err := context.Cause(ctx); err != nil {
    return fmt.Errorf("aborted: %w", err)
}

Key Rules

  • ctx.Err() is unchanged (still Canceled/DeadlineExceeded); the specific reason lives in context.Cause(ctx).
  • Calling the CancelCauseFunc with nil makes the cause equal to ctx.Err(); only the first cancel's cause is recorded — later calls are no-ops.
  • If a parent context is cancelled first, Cause reports the parent's cause — causes propagate down the tree like cancellation itself.
  • You still must call the cancel function (defer it) to release resources, exactly as with WithCancel; the cause is an addition, not a replacement.
Practice Exercise

Build a fan-out search across three mock backends where the first one to fail cancels the others with WithCancelCause, passing a sentinel error naming the failed backend; have the coordinator return context.Cause(ctx) wrapped with %w. Then convert a fixed-timeout HTTP call to WithTimeoutCause so a caller can tell an SLA breach from an ordinary deadline. Finally, demonstrate that once a context is cancelled, a second cancel(differentErr) does not change the recorded cause, and explain why that "first cause wins" rule matters.

Pitfall: Branching on ctx.Err() and expecting your custom error — it will always be the generic context.Canceled/DeadlineExceeded. Read context.Cause(ctx) for the real reason. Also remember causes only exist if the context was created with a *Cause constructor; on a plain WithCancel, context.Cause just returns ctx.Err().
Connections: Extends the Context package cancellation/deadline model and complements context.AfterFunc/WithoutCancel from yesterday; the cause makes cancellation compose with the errors.Is/%w wrapping conventions and is a natural fit for errgroup-style fan-out where the first error should stop the group.

io.Pipe: An In-Memory Reader/Writer Bridge

Std Lib

Why it matters: sometimes you have a producer that writes to an io.Writer but a consumer that needs an io.Reader — for example, streaming a generated report as the body of an http.Request without first buffering the whole thing in memory. io.Pipe connects the two: bytes written to the *io.PipeWriter become readable on the *io.PipeReader, synchronously and without a temporary buffer. It's the idiomatic way to turn "push" code into a streamable "pull" source.

Streaming producer to consumer

io.Pipe returns a connected reader/writer. Writes block until a reader consumes them (it's unbuffered), so the producer must run in its own goroutine. Close the writer to signal EOF.

package main

import (
    "fmt"
    "io"
    "strings"
)

func main() {
    pr, pw := io.Pipe()

    // Producer writes in a goroutine; Write blocks until Read.
    go func() {
        defer pw.Close() // Close -> reader sees io.EOF
        for i := 1; i <= 3; i++ {
            fmt.Fprintf(pw, "line %d\n", i)
        }
    }()

    // Consumer pulls the stream as an io.Reader.
    var sb strings.Builder
    io.Copy(&sb, pr)
    fmt.Print(sb.String())
}
Output
line 1 line 2 line 3

Propagating errors with CloseWithError

If the producer fails mid-stream, pw.CloseWithError(err) makes the reader's next Read return that error instead of io.EOF — so failures surface on the consumer side.

package main

import (
    "errors"
    "fmt"
    "io"
)

func main() {
    pr, pw := io.Pipe()
    go func() {
        pw.Write([]byte("partial data..."))
        pw.CloseWithError(errors.New("source failed"))
    }()

    buf := make([]byte, 64)
    for {
        n, err := pr.Read(buf)
        if n > 0 {
            fmt.Printf("read: %q\n", buf[:n])
        }
        if err != nil {
            fmt.Println("err:", err)
            return
        }
    }
}
Output
read: "partial data..." err: source failed

Common Mistakes

Wrong
pr, pw := io.Pipe()
// Writing and reading in the SAME goroutine deadlocks:
// Write blocks waiting for a Read that never runs.
pw.Write([]byte("hi"))     // blocks forever
io.ReadAll(pr)

// Forgetting to Close the writer -> reader hangs on
// the final Read, never getting io.EOF.
Right
pr, pw := io.Pipe()
go func() {
    defer pw.Close()        // always close to signal EOF
    json.NewEncoder(pw).Encode(bigStruct)
}()
// Reader side streams it, e.g. as a request body:
http.Post(url, "application/json", pr)

Key Rules

  • An io.Pipe is synchronous and unbuffered: each Write blocks until one or more Reads consume all its bytes — so producer and consumer must run in different goroutines.
  • Close the writer (pw.Close()) to make the reader observe io.EOF; use pw.CloseWithError(err) to make it observe a specific error instead.
  • Closing the reader (pr.Close()) makes subsequent Writes fail with io.ErrClosedPipe — useful to signal the producer to stop.
  • It's safe for one reader and one writer concurrently; it is not a fan-out or a buffer — reach for bytes.Buffer or a channel if you need buffering or multiple consumers.
Practice Exercise

Use io.Pipe to stream a large CSV you generate on the fly as the body of an http.Post, so the whole file is never held in memory at once; encode rows with encoding/csv writing into the pipe writer from a goroutine. Then add error propagation: if row generation fails, call CloseWithError and confirm the HTTP client's request fails cleanly. Finally, wire a gzip.Writer in front of the pipe writer to stream compressed output, and explain why doing the write in the same goroutine as the read would deadlock.

Pitfall: Treating io.Pipe like a buffer and writing/reading from one goroutine — because writes block until read, you deadlock instantly. Always run the producer in a separate goroutine and always close the writer (with Close or CloseWithError) so the reader terminates; a forgotten close leaves the consumer blocked on a final Read forever.
Connections: Builds on the io.Reader/io.Writer interfaces and io.Copy from the io package day; it's the glue that lets streaming compression (compress/gzip), encoding/json/csv encoders, and net/http request bodies compose without intermediate buffers, and it pairs naturally with goroutines and channel-based coordination.

time.Timer & time.Ticker: One-Shot vs Repeating

Std Lib

Why it matters: time.Sleep is fine for a blocking pause, but real concurrent code needs timers you can select on, stop, and reset: a per-request timeout, a heartbeat, a periodic flush. time.Timer fires once; time.Ticker fires repeatedly on a channel. Using them correctly — especially stopping a Ticker so it doesn't leak, and understanding Reset semantics — is a common source of subtle goroutine and resource leaks.

Timer: fire once, and cancel if not needed

time.NewTimer(d) sends the current time on timer.C after d. timer.Stop() cancels it (returns false if it already fired). time.AfterFunc(d, f) is the callback form — it runs f in its own goroutine after d.

package main

import (
    "fmt"
    "time"
)

func main() {
    t := time.NewTimer(20 * time.Millisecond)

    select {
    case <-t.C:
        fmt.Println("timer fired")
    case <-time.After(5 * time.Millisecond):
        // Beat the timer: stop it so it doesn't leak.
        if t.Stop() {
            fmt.Println("work finished early; timer stopped")
        }
    }

    // AfterFunc: run a callback after a delay (own goroutine).
    done := make(chan struct{})
    time.AfterFunc(10*time.Millisecond, func() {
        fmt.Println("deferred callback ran")
        close(done)
    })
    <-done
}
Output
work finished early; timer stopped deferred callback ran

Ticker: repeat until you stop it

time.NewTicker(d) ticks every d on ticker.C. You must call ticker.Stop() (usually deferred) — unlike a timer, a ticker keeps its goroutine and channel alive until stopped, so forgetting is a leak.

package main

import (
    "fmt"
    "time"
)

func main() {
    ticker := time.NewTicker(10 * time.Millisecond)
    defer ticker.Stop() // REQUIRED: otherwise it leaks

    ticks := 0
    timeout := time.After(35 * time.Millisecond)
    for {
        select {
        case <-ticker.C:
            ticks++
        case <-timeout:
            fmt.Println("ticks observed:", ticks) // ~3
            return
        }
    }
}
Output
ticks observed: 3

Common Mistakes

Wrong
// Ticker leak: no Stop(), goroutine + channel live on.
for range time.NewTicker(time.Second).C {
    poll() // ticker can never be stopped -> leak
}

// Pre-Go-1.23: Reset on a timer whose value wasn't
// drained could deliver a stale tick. Drain first.
Right
ticker := time.NewTicker(time.Second)
defer ticker.Stop()
for {
    select {
    case <-ctx.Done():
        return          // clean exit
    case <-ticker.C:
        poll()
    }
}

Key Rules

  • A Ticker must be stopped with ticker.Stop() or it leaks its goroutine and channel; a fired Timer doesn't need stopping, but stop an unfired one you no longer need.
  • Stop() does not close or drain the channel; it just prevents future sends. A tick already sitting in C is still deliverable.
  • In Go 1.23+, unreferenced timers/tickers are GC-eligible even without Stop, and Reset/Stop no longer risk delivering a stale value — but calling Stop on tickers is still the clear, portable habit.
  • Reset(d) reuses a timer; only call it on a stopped or expired timer to avoid ambiguity on older Go versions, and never share one Timer across goroutines without synchronization.
Practice Exercise

Write a debouncer: given a stream of events on a channel, use a single time.Timer with Reset so a callback fires only once the events have been quiet for 100ms. Then build a periodic batch flusher with a time.Ticker that also flushes immediately when a buffer reaches capacity, exiting cleanly on ctx.Done() with a deferred Stop. Finally, demonstrate a ticker leak by starting one without Stop in a loop and observing goroutine growth via runtime.NumGoroutine(), then fix it.

Pitfall: Ranging directly over time.NewTicker(d).C without keeping a reference to call Stop() — on pre-1.23 Go this leaks a goroutine per ticker for the life of the process. Also, assuming Stop() drains the channel: a value may already be buffered in C, so a subsequent read can still see one last tick. Keep the *Ticker/*Timer handle so you can control it.
Connections: Deepens the time package basics into concurrency territory, pairing with select, channels, and context cancellation from the concurrency days; time.AfterFunc mirrors context.AfterFunc, and these primitives underpin the rate limiting, retry-with-backoff, and graceful-shutdown patterns you've seen in production Go.

Section 2 — Security Standard Library

crypto/rand: Cryptographically Secure Randomness

Security

Why it matters: anything security-sensitive — session tokens, API keys, password salts, nonces, CSRF tokens — must be unpredictable. math/rand (even rand/v2) is fast but predictable: given a couple of outputs an attacker can recover the seed and forecast the rest. crypto/rand reads from the operating system's CSPRNG (/dev/urandom, getrandom(2), CryptGenRandom) and is the only correct source for secrets.

Generating secure random bytes and tokens

rand.Read fills a byte slice from the OS CSPRNG. Encode the raw bytes (base64/hex) to get a URL-safe token. Aim for at least 16–32 bytes of entropy for tokens.

package main

import (
    "crypto/rand"
    "encoding/base64"
    "encoding/hex"
    "fmt"
)

func main() {
    // 32 bytes = 256 bits of entropy.
    b := make([]byte, 32)
    if _, err := rand.Read(b); err != nil {
        panic(err) // extremely rare: OS entropy failure
    }

    token := base64.RawURLEncoding.EncodeToString(b)
    fmt.Println("token len:", len(token)) // 43 chars
    fmt.Println("hex len:  ", len(hex.EncodeToString(b))) // 64
}
Output
token len: 43 hex len: 64

Unbiased integers in a range

For a secure integer in [0, n), use rand.Int(rand.Reader, big) — it avoids the modulo bias you'd get from b[0] % n. Go 1.22+ also adds crypto/rand.Text() for a random alphanumeric string.

package main

import (
    "crypto/rand"
    "fmt"
    "math/big"
)

func main() {
    // Secure, unbiased dice roll in [1, 6].
    n, err := rand.Int(rand.Reader, big.NewInt(6))
    if err != nil {
        panic(err)
    }
    roll := n.Int64() + 1
    fmt.Println("roll in range:", roll >= 1 && roll <= 6) // true
}
Output
roll in range: true

Common Mistakes

Wrong
import "math/rand/v2"

// PREDICTABLE: never for secrets/tokens/salts.
func token() string {
    b := make([]byte, 16)
    for i := range b {
        b[i] = byte(rand.IntN(256)) // recoverable
    }
    return hex.EncodeToString(b)
}

// Modulo bias with raw bytes:
digit := secureByte % 10 // skews toward low digits
Right
import "crypto/rand"

func token() string {
    b := make([]byte, 16)
    rand.Read(b) // OS CSPRNG, unpredictable
    return hex.EncodeToString(b)
}

// Unbiased bounded int:
n, _ := rand.Int(rand.Reader, big.NewInt(10))
digit := n.Int64()

Key Rules

  • Use crypto/rand (not math/rand) for anything an attacker shouldn't predict: tokens, keys, salts, nonces, IVs, password-reset codes.
  • rand.Read either fills the whole slice or returns an error — check it (a failure means the OS entropy source is broken, which is worth crashing over).
  • Encode raw bytes for transport with base64.RawURLEncoding (URL-safe, no padding) or hex; don't print raw bytes as a "string".
  • For bounded integers use rand.Int(rand.Reader, max) to avoid modulo bias; for random text, crypto/rand.Text() (Go 1.24) gives a ready-made secure string.
Practice Exercise

Write a SecureToken(nBytes int) (string, error) that reads nBytes from crypto/rand and returns a base64.RawURLEncoding string, and a SecureCode(n int) that produces an n-digit numeric code with no modulo bias using rand.Int. Then generate a 16-byte salt and combine it with a password using bcrypt or argon2, and explain why the salt must be unique and unpredictable per user. Finally, contrast the throughput of crypto/rand vs math/rand/v2 and state exactly when each is the correct choice.

Pitfall: Reaching for math/rand because it's simpler or faster when generating something secret — predictable tokens are a classic, severe vulnerability (session hijacking, guessable reset links). Also, ignoring the error from rand.Read, or reducing secure bytes with % n and reintroducing bias. Use crypto/rand for secrets and its unbiased helpers for ranges.
Connections: The security counterpart to math/rand/v2 (fast, seedable, non-secret) from the randomness day; feeds token generation that gets encoded with encoding/base64/hex, underpins TLS and the wider crypto packages, and pairs with crypto/subtle below for safely comparing the secrets you generate here.

crypto/subtle: Constant-Time Comparisons

Security

Why it matters: comparing secrets with == or bytes.Equal is a timing side channel. Those comparisons return as soon as the first differing byte is found, so an attacker measuring response time can recover a secret byte-by-byte (a classic attack against HMAC/API-key checks). crypto/subtle.ConstantTimeCompare always examines every byte, taking the same time regardless of where (or whether) the inputs differ — closing that leak.

Comparing secrets without leaking timing

subtle.ConstantTimeCompare(x, y) returns 1 if the two equal-length byte slices are identical, 0 otherwise — in time independent of the contents. Use it for MAC tags, API keys, and password-hash outputs.

package main

import (
    "crypto/hmac"
    "crypto/sha256"
    "crypto/subtle"
    "fmt"
)

func validMAC(msg, key, tag []byte) bool {
    mac := hmac.New(sha256.New, key)
    mac.Write(msg)
    expected := mac.Sum(nil)
    // Constant-time: no early-exit timing leak.
    return subtle.ConstantTimeCompare(expected, tag) == 1
}

func main() {
    key := []byte("secret-key")
    msg := []byte("transfer $100")

    mac := hmac.New(sha256.New, key)
    mac.Write(msg)
    good := mac.Sum(nil)

    fmt.Println("valid:  ", validMAC(msg, key, good))
    fmt.Println("forged: ", validMAC(msg, key, []byte("wrong-tag")))
}
Output
valid: true forged: false

hmac.Equal & other constant-time helpers

hmac.Equal wraps the same constant-time comparison for MAC tags. subtle also offers ConstantTimeSelect, ConstantTimeByteEq, and ConstantTimeCopy for branch-free conditional logic on secrets.

package main

import (
    "crypto/subtle"
    "fmt"
)

func main() {
    a := []byte("api-key-abc123")
    b := []byte("api-key-abc123")
    c := []byte("api-key-XXXXXX")

    fmt.Println(subtle.ConstantTimeCompare(a, b) == 1) // true
    fmt.Println(subtle.ConstantTimeCompare(a, c) == 1) // false

    // Branch-free select: pick x if v==1, else y.
    x, y := 10, 20
    fmt.Println(subtle.ConstantTimeSelect(1, x, y)) // 10
}
Output
true false 10

Common Mistakes

Wrong
// Early-exit comparison leaks how many leading bytes
// matched -> attacker recovers the token over time.
func check(got, want string) bool {
    return got == want        // timing side channel
}

func checkMAC(a, b []byte) bool {
    return bytes.Equal(a, b)  // also early-exit
}
Right
func check(got, want []byte) bool {
    // Constant-time regardless of contents.
    return subtle.ConstantTimeCompare(got, want) == 1
}

// For MAC tags specifically:
if hmac.Equal(expectedTag, providedTag) {
    // authenticated
}

Key Rules

  • Use subtle.ConstantTimeCompare (or hmac.Equal) for any secret comparison: API keys, MAC/HMAC tags, session tokens, hashed-password digests.
  • It returns 1/0 (an int), not a bool — compare == 1. Inputs of different lengths return 0 immediately (length itself isn't secret here).
  • Constant-time only matters against remote attackers when the value is secret; comparing two public strings can stay a normal ==.
  • Don't roll your own crypto around it — use it to compare, but rely on crypto/hmac, bcrypt/argon2, and TLS for the actual primitives.
Practice Exercise

Implement an API-key middleware that reads a bearer token from the Authorization header and compares it against the configured key using subtle.ConstantTimeCompare, returning 401 on mismatch. Then verify a webhook signature by computing an HMAC-SHA256 over the raw body and checking it with hmac.Equal. Write a micro-benchmark that times == vs ConstantTimeCompare on strings that differ at the first byte vs the last byte, and explain what the timing difference in the == case reveals to an attacker.

Pitfall: Assuming a constant-time compare fixes everything while leaking length or short-circuiting elsewhere — e.g. checking len(a) == len(b) with an early return before comparing, or logging how far a match got. Also, remember ConstantTimeCompare returns an int; treating a nonzero-looking value as truthy without == 1 is a bug. Use it strictly for the comparison, and keep the surrounding logic branch-free on secret data.
Connections: The natural partner to crypto/rand above (generate secrets securely, then compare them securely) and to crypto/hmac/crypto/sha256; the same timing-safety concern shows up in TLS certificate verification and in the auth middleware patterns from the production-Go and net/http days.
