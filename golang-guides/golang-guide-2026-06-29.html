testing Package: Table-Driven Tests & Subtests

Testing & Tooling • overdue 4d • rep 2

Quick Reminder

A test is a func TestXxx(t *testing.T) in a _test.go file. The idiomatic shape is a table of cases fed to t.Run(name, ...) so each row is an isolated subtest with its own name in the output. Call t.Errorf to record a failure and keep going; t.Fatalf stops the current test. Loop-scoped subtests can even run in parallel with t.Parallel().

Key Pattern

package math_test

import "testing"

func Abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}

func TestAbs(t *testing.T) {
    cases := []struct {
        name string
        in   int
        want int
    }{
        {"positive", 5, 5},
        {"negative", -5, 5},
        {"zero", 0, 0},
    }
    for _, c := range cases {
        t.Run(c.name, func(t *testing.T) {
            if got := Abs(c.in); got != c.want {
                t.Errorf("Abs(%d) = %d, want %d", c.in, got, c.want)
            }
        })
    }
}
Challenge Exercise

Write a table-driven test for a ParseRange("1-5") ([]int, error) function covering valid input, reversed bounds, and malformed strings. Give each row a descriptive name, assert on both the slice and whether an error was expected, and use t.Run so a single failing row is named precisely in the output. Then add t.Parallel() and explain why you must be careful with a shared loop variable in older Go versions.

Pitfall: Using t.Fatalf inside a goroutine started by the test. FailNow/Fatal must be called from the goroutine running the test function — from another goroutine it only calls runtime.Goexit on the wrong stack and the test won't stop as intended. Send the error back over a channel and fail on the main test goroutine instead.

Benchmarks

Testing & Tooling • overdue 4d • rep 2

Quick Reminder

A benchmark is func BenchmarkXxx(b *testing.B) with a for i := 0; i < b.N; i++ loop — the framework raises b.N until timings stabilize. Do expensive setup before the loop and call b.ResetTimer(); add b.ReportAllocs() to print allocations per op. Run with go test -bench=. -benchmem.

Key Pattern

package str_test

import (
    "strings"
    "testing"
)

func BenchmarkJoin(b *testing.B) {
    parts := make([]string, 1000)
    for i := range parts {
        parts[i] = "x"
    }
    b.ReportAllocs()
    b.ResetTimer() // discard setup cost above

    for i := 0; i < b.N; i++ {
        _ = strings.Join(parts, ",")
    }
}
// go test -bench=Join -benchmem
// BenchmarkJoin-10  380000  3120 ns/op  2048 B/op  1 allocs/op
Challenge Exercise

Benchmark three ways to concatenate 10k strings: +=, strings.Builder, and bytes.Buffer. Use sub-benchmarks via b.Run so they appear side by side, report allocations with b.ReportAllocs(), and explain the B/op and allocs/op gap. Finally, show why forgetting to assign the result to a package-level sink can let the compiler optimize the whole loop away.

Pitfall: The compiler may eliminate a benchmark body whose result is unused ("dead code"), giving an impossibly fast ~0.3 ns/op. Assign the result to an exported/package-level variable (a "sink") so the work is observably needed, and always ResetTimer after setup so fixture cost isn't charged to the measured operation.

Test Fixtures: TestMain, t.TempDir, t.Cleanup

Testing & Tooling • overdue 2d • rep 2

Quick Reminder

func TestMain(m *testing.M) runs once per package for global setup/teardown — you must call m.Run() and os.Exit with its code. Per-test, t.TempDir() hands you an auto-removed directory and t.Cleanup(fn) registers teardown that runs in LIFO order even if the test fails — cleaner than defer because it survives helper boundaries.

Key Pattern

package store_test

import (
    "os"
    "path/filepath"
    "testing"
)

func TestMain(m *testing.M) {
    // one-time setup for the whole package
    code := m.Run()
    // one-time teardown
    os.Exit(code)
}

func TestWrite(t *testing.T) {
    dir := t.TempDir() // removed automatically after the test
    path := filepath.Join(dir, "data.txt")

    t.Cleanup(func() { t.Log("cleanup ran") }) // LIFO, always runs

    if err := os.WriteFile(path, []byte("hi"), 0o644); err != nil {
        t.Fatalf("write: %v", err)
    }
    got, _ := os.ReadFile(path)
    if string(got) != "hi" {
        t.Errorf("got %q", got)
    }
}
Challenge Exercise

Write a test that spins up a temporary SQLite-style file DB in t.TempDir(), registers t.Cleanup to close the handle, and seeds a fixture row. Then use TestMain to build a shared read-only fixture once for the whole package. Explain why t.Cleanup is safer than a bare defer inside a test helper, and how cleanup ordering interacts with subtests.

Pitfall: Writing a TestMain that calls m.Run() but forgets os.Exit(code) (or discards the code). The process then exits 0 regardless of failures, so a red suite reports green in CI. Always os.Exit(m.Run()), and never run teardown after os.Exit — it never executes.

Middleware Chains

Advanced Patterns • overdue 1d • rep 2

Quick Reminder

HTTP middleware is a function func(http.Handler) http.Handler that wraps a handler to run code before/after it — logging, auth, recovery. Compose several by folding a slice from the innermost out, so the first in the list is the outermost layer. Each wrapper decides whether to call next.ServeHTTP (continue) or short-circuit.

Key Pattern

package main

import "net/http"

type Middleware func(http.Handler) http.Handler

func Chain(h http.Handler, mws ...Middleware) http.Handler {
    // apply in reverse so mws[0] is the OUTERMOST layer
    for i := len(mws) - 1; i >= 0; i-- {
        h = mws[i](h)
    }
    return h
}

func Logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // ... log r.Method, r.URL.Path ...
        next.ServeHTTP(w, r) // call next, or return early to block
    })
}
Challenge Exercise

Build three middlewares — a panic Recover (500 on panic), a RequireAuth that 401s when a header is missing, and a Timing that records elapsed time — then compose them with a Chain helper so ordering is Recover → Timing → Auth → handler. Verify with httptest that a blocked request never reaches the handler. Explain why Recover must be outermost.

Pitfall: Writing to the response body before setting the status code or headers. Once any bytes are written, WriteHeader is implicitly called with 200 and later w.WriteHeader(401) / header edits are ignored (with a "superfluous WriteHeader" warning). Middleware that may reject a request must set status/headers before calling next or before writing anything.

reflect Package

Advanced Patterns • overdue 1d • rep 2

Quick Reminder

reflect.TypeOf and reflect.ValueOf inspect a value's dynamic type and contents at runtime — the machinery behind encoding/json and ORMs. To set a field you need an addressable Value, obtained via a pointer: reflect.ValueOf(&x).Elem(). Struct tags are read with field.Tag.Get("json"). It's powerful but slow and unchecked — reach for it only when types truly aren't known at compile time.

Key Pattern

package main

import (
    "fmt"
    "reflect"
)

type User struct {
    Name string `json:"name"`
    Age  int    `json:"age"`
}

func main() {
    u := User{"Ada", 36}
    t := reflect.TypeOf(u)
    v := reflect.ValueOf(u)
    for i := 0; i < t.NumField(); i++ {
        f := t.Field(i)
        fmt.Printf("%s (tag=%s) = %v\n",
            f.Name, f.Tag.Get("json"), v.Field(i))
    }

    // Mutate through an addressable Value (needs a pointer).
    p := reflect.ValueOf(&u).Elem()
    p.FieldByName("Name").SetString("Bo")
    fmt.Println(u.Name)
}
Challenge Exercise

Write structToMap(v any) map[string]any that returns exported fields keyed by their json tag (falling back to the field name). Then write the inverse: populate a struct pointer from a map[string]any, skipping keys with no matching field. Handle the "cannot set unexported / unaddressable field" cases gracefully, and explain why reflect.ValueOf(x) (no pointer) is never settable.

Pitfall: Calling Set* on a Value obtained from a non-pointer, or on an unexported field — both panic (using unaddressable value / cannot set). Always start from a pointer and .Elem(), and check v.CanSet() before writing. Remember reflection errors surface at runtime, not compile time, so cover these paths with tests.

gRPC & Protocol Buffers

Advanced Networking • overdue 15d • rep 1

Quick Reminder

You define services and messages in a .proto file, then generate Go types and client/server stubs with protoc (+ protoc-gen-go / protoc-gen-go-grpc). gRPC rides HTTP/2 with binary protobuf payloads and supports four call styles: unary, server-streaming, client-streaming, and bidirectional. Every RPC takes a context.Context for deadlines and cancellation.

Key Pattern

// greeter.proto
syntax = "proto3";
package greet;
option go_package = "example/greet";

message HelloRequest  { string name = 1; }
message HelloReply    { string message = 1; }

service Greeter {
  rpc SayHello(HelloRequest) returns (HelloReply);
}

// generate:
//   protoc --go_out=. --go-grpc_out=. greeter.proto
// server implements the generated GreeterServer interface;
// client calls: reply, err := client.SayHello(ctx, &HelloRequest{Name: "Ada"})
Challenge Exercise

Sketch a .proto for a PriceService with a unary GetPrice and a server-streaming WatchPrice that pushes updates. Describe the generated Go types, then outline a server handler for the streaming RPC (looping on stream.Send until ctx.Done()). Explain how a client sets a 2-second deadline with context.WithTimeout and how the server observes that cancellation. Note why field numbers, not names, must stay stable across schema versions.

Pitfall: Reusing or reordering protobuf field numbers when evolving a message. The wire format is keyed by tag number, not field name — renaming a field is safe, but changing its number (or recycling a deleted one) silently corrupts decoding for older peers. Reserve retired numbers with reserved, and never change a field's type in place.

Section 1 — Text & Numbers Standard Library

strconv: String ↔ Number Conversions

Std Lib

Why it matters: parsing user input, config files, and wire formats means turning text into numbers and back — and doing it with error handling, not a silent guess. strconv is the correct, allocation-light tool for exactly that: Atoi/Itoa for the base-10 common case, ParseInt/ParseFloat/ParseBool when you need bases, bit sizes, or overflow detection.

The everyday conversions

Atoi (ASCII-to-int) and Itoa (int-to-ASCII) are base-10 shortcuts. Every parse returns an error you must check — that is the whole point over a raw cast.

package main

import (
    "fmt"
    "strconv"
)

func main() {
    n, err := strconv.Atoi("42")
    fmt.Println(n, err)              // 42 <nil>

    fmt.Println(strconv.Itoa(-7))    // "-7"

    f, _ := strconv.ParseFloat("3.14159", 64)
    fmt.Println(f)                   // 3.14159

    b, _ := strconv.ParseBool("true")
    fmt.Println(b)                   // true

    _, err = strconv.Atoi("12x")     // not a number
    fmt.Println(err)
}
Output
42 <nil> -7 3.14159 true strconv.Atoi: parsing "12x": invalid syntax

Bases, bit sizes, and quoting

ParseInt(s, base, bitSize) takes a base (0 auto-detects 0x/0o/0b prefixes) and a bitSize that bounds the value for overflow checks. FormatInt renders into any base; Quote produces a Go-syntax escaped literal.

package main

import (
    "fmt"
    "strconv"
)

func main() {
    v, _ := strconv.ParseInt("0xff", 0, 64) // base 0 -> detects hex
    fmt.Println(v)                           // 255

    fmt.Println(strconv.FormatInt(255, 16))  // "ff"
    fmt.Println(strconv.FormatInt(255, 2))   // "11111111"

    fmt.Println(strconv.Quote("tab\there"))  // escaped Go literal

    _, err := strconv.ParseInt("300", 10, 8) // int8 max is 127
    fmt.Println(err)
}
Output
255 ff 11111111 "tab\there" strconv.ParseInt: parsing "300": value out of range

Common Mistakes

Wrong
n := 65
// string(int) is a RUNE conversion:
s := string(rune(n)) // "A", not "65"
// bare string(n) even warns via go vet
Right
n := 65
s := strconv.Itoa(n) // "65"
// or strconv.FormatInt(int64(n), 10)

Key Rules

  • Atoi/Itoa are the base-10 int shortcuts; ParseInt/FormatInt when you need another base or int64.
  • Always check the returned error — a failed parse yields the zero value, which silently corrupts logic if ignored.
  • bitSize (8/16/32/64) only bounds overflow detection; the return type is still int64/float64 — you convert down yourself.
  • Quote/Unquote handle Go-literal escaping; use strconv.AppendInt to format into an existing []byte without allocating.
Practice Exercise

Parse a config of key=value lines where values may be ints, floats, or bools, returning a map[string]any with the correctly typed value (try each parser in order). Then write toBase(n int64, base int) string and its inverse using FormatInt/ParseInt, round-tripping bases 2–36. Finally, build a hot-loop CSV row formatter using strconv.AppendInt into a reused buffer and confirm it allocates less than fmt.Sprintf.

Pitfall: Converting an int to a string with string(n). That is a rune conversion — string(65) is "A", the Unicode code point, not the digits "65". Modern go vet flags it, but the bug still compiles; use strconv.Itoa (or fmt.Sprint) to render the digits.
Connections: The primitive under flag's typed parsing and cmp-based sorting of numeric strings; pairs with bytes/strings when tokenizing text, and hands off to math/big when a value overflows 64 bits.

bytes: The []byte Toolkit

Std Lib

Why it matters: the bytes package mirrors strings but operates on []byte, so you can search, split, and transform raw data without converting to a string (which copies). bytes.Buffer is a growable, read/write byte sink that implements io.Reader and io.Writer — the go-to accumulator whenever you assemble bytes destined for a file, socket, or hash.

bytes.Buffer as an io.Writer

A zero-value bytes.Buffer is ready to use. It grows on demand and works with fmt.Fprintf, io.Copy, and anything that takes an io.Writer.

package main

import (
    "bytes"
    "fmt"
)

func main() {
    var buf bytes.Buffer // zero value is usable
    buf.WriteString("Hello")
    buf.WriteByte(',')
    buf.WriteString(" Go")
    fmt.Fprintf(&buf, "! %d", 2026) // Buffer is an io.Writer

    fmt.Println(buf.String())
    fmt.Println("len:", buf.Len())
}
Output
Hello, Go! 2026 len: 15

Search and transform without allocating strings

Every strings helper has a bytes twin — Contains, Split, ToUpper, Count, Equal — operating directly on []byte read from a file or network.

package main

import (
    "bytes"
    "fmt"
)

func main() {
    data := []byte("go,rust,zig,go")

    fmt.Println(bytes.Contains(data, []byte("rust"))) // true
    parts := bytes.Split(data, []byte(","))
    fmt.Println(len(parts))                            // 4
    fmt.Printf("%s\n", bytes.ToUpper(parts[0]))        // GO
    fmt.Println(bytes.Count(data, []byte("go")))       // 2
    fmt.Println(bytes.Equal([]byte("a"), []byte("a"))) // true
}
Output
true 4 GO 2 true

Common Mistakes

Wrong
a := []byte("hi")
b := []byte("hi")
// slices are not comparable with ==
if a == b { /* compile error */ }
Right
a := []byte("hi")
b := []byte("hi")
if bytes.Equal(a, b) { // correct
    // ...
}

Key Rules

  • bytes is the []byte mirror of strings — same function names, no string conversion (and no copy).
  • bytes.Buffer implements both io.Reader and io.Writer; the zero value is ready — no make needed.
  • Compare byte slices with bytes.Equal/bytes.Compare; == is a compile error on slices.
  • Use bytes.NewReader(b) for a read-only io.Reader over an existing slice; prefer strings.Builder when the result is ultimately a string.
Practice Exercise

Read a file into a []byte and count lines by tallying '\n' with bytes.Count — without converting to a string. Then write a redactor that replaces every match of a keyword with **** using bytes.ReplaceAll, streaming the result into a bytes.Buffer. Finally, feed a bytes.NewReader to io.Copy and to a sha256 hasher, proving the same slice can act as an io.Reader.

Pitfall: Retaining the slice returned by buf.Bytes() and then writing more to the buffer. Bytes() exposes the buffer's internal storage; a subsequent write may append in place or reallocate, so your saved slice can be overwritten or point to stale data. Copy it (append([]byte(nil), buf.Bytes()...)) if you need to keep it past the next write.
Connections: bytes.Buffer is the sibling of strings.Builder and the classic sink for gob/json encoders and the io plumbing; bytes.NewReader plugs a slice into any io.Reader consumer, complementing strconv when tokenizing.

cmp: Ordering & Tie-Breaking Helpers

Std Lib

Why it matters: since Go 1.21 the tiny cmp package gives you a generic, type-safe way to order any ordered type and — with cmp.Or (Go 1.22) — to express multi-key "sort by last name, then first name" comparators declaratively. It's the natural comparator source for slices.SortFunc and slices.BinarySearchFunc, replacing hand-rolled if a < b { return -1 } ladders.

Compare and Less over any ordered type

cmp.Compare(a, b) returns -1, 0, or +1 for anything satisfying cmp.Ordered (ints, floats, strings). cmp.Less gives the boolean form.

package main

import (
    "cmp"
    "fmt"
)

func main() {
    fmt.Println(cmp.Compare(3, 9))     // -1
    fmt.Println(cmp.Compare(9, 9))     //  0
    fmt.Println(cmp.Compare("b", "a")) //  1
    fmt.Println(cmp.Less(2, 5))        // true
}
Output
-1 0 1 true

cmp.Or for defaults and multi-key sorts

cmp.Or returns its first non-zero argument — perfect for "value or fallback" and for chaining comparators so a tie on one key falls through to the next.

package main

import (
    "cmp"
    "fmt"
    "slices"
)

type Person struct {
    Last, First string
    Age         int
}

func main() {
    fmt.Println(cmp.Or("", "", "fallback")) // first non-zero

    people := []Person{
        {"Vance", "Ada", 30},
        {"Vance", "Bo", 25},
        {"Adler", "Cy", 40},
    }
    slices.SortFunc(people, func(a, b Person) int {
        return cmp.Or(
            cmp.Compare(a.Last, b.Last),   // primary key
            cmp.Compare(a.First, b.First), // tie-breaker
        )
    })
    for _, p := range people {
        fmt.Println(p.Last, p.First)
    }
}
Output
fallback Adler Cy Vance Ada Vance Bo

Common Mistakes

Wrong
// SortFunc wants an int comparator,
// not a bool "less" like sort.Slice.
slices.SortFunc(xs, func(a, b int) bool {
    return a < b // compile error
})
Right
slices.SortFunc(xs, func(a, b int) int {
    return cmp.Compare(a, b)
})

Key Rules

  • cmp.Compare returns -1/0/+1 for any cmp.Ordered type; cmp.Less is the boolean variant.
  • cmp.Or(a, b, ...) yields the first non-zero argument — use it for default values and for cascading multi-key comparators.
  • Pair with slices.SortFunc/BinarySearchFunc, which expect an int comparator (not the bool "less" that legacy sort.Slice uses).
  • cmp.Ordered is the reusable constraint for your own generic min/max/clamp helpers.
Practice Exercise

Sort a slice of Order{Priority int; Created time.Time; ID string} by priority descending, then oldest first, then ID — expressed as one cmp.Or chain (negate the priority compare for descending). Then write a generic Clamp[T cmp.Ordered](x, lo, hi T) T. Finally, demonstrate cmp.Or as a config resolver: pick the first non-empty of flagVal, envVal, defaultVal.

Pitfall: Assuming cmp.Compare orders floating-point NaN intuitively. By definition it treats NaN as less than any non-NaN value and equal to itself, so a slice of floats containing NaN sorts them all to the front — which may not be what you want. Filter or special-case NaN before sorting when it matters.
Connections: The comparator engine behind the slices/maps generic packages; its cmp.Ordered constraint generalizes the generics and custom-constraint topics, and it consumes the numeric values produced by strconv.

Section 2 — Big Numbers & Runtime Control

math/big: Arbitrary-Precision Arithmetic

Std Lib

Why it matters: int64 tops out around 9.2×1018; the moment you compute a large factorial, an RSA modulus, or exact money-as-fractions, you overflow or lose precision. math/big provides big.Int (unbounded integers), big.Rat (exact rationals), and big.Float (arbitrary-precision floats). The catch: Go has no operator overloading, so all arithmetic is method calls that mutate a receiver.

Unbounded integers: factorials and big powers

Construct with big.NewInt or new(big.Int). Methods like Mul, Add, and Exp write the result into the receiver and return it, so you can chain or accumulate.

package main

import (
    "fmt"
    "math/big"
)

func main() {
    // 100! overflows int64 by a mile.
    f := big.NewInt(1)
    for i := int64(2); i <= 100; i++ {
        f.Mul(f, big.NewInt(i)) // f = f * i  (mutates f)
    }
    fmt.Println(len(f.String()), "digits")

    // 2^200 via modular Exp (nil modulus = plain power).
    p := new(big.Int).Exp(big.NewInt(2), big.NewInt(200), nil)
    fmt.Println(p)
}
Output
158 digits 1606938044258990275541962092341162602522202993782792835301376

Exact rationals and primality tests

big.Rat stores fractions exactly — no floating-point drift — and auto-reduces. big.Int.ProbablyPrime runs a Miller-Rabin test, the backbone of key generation.

package main

import (
    "fmt"
    "math/big"
)

func main() {
    // 1/3 + 1/6 = 1/2, exactly.
    r := new(big.Rat).Add(big.NewRat(1, 3), big.NewRat(1, 6))
    fmt.Println(r.String()) // 1/2

    // Equal fractions compare as 0 regardless of form.
    fmt.Println(big.NewRat(1, 3).Cmp(big.NewRat(2, 6))) // 0

    n := big.NewInt(1000003)
    fmt.Println(n.ProbablyPrime(20)) // true
}
Output
1/2 0 true

Common Mistakes

Wrong
a := big.NewInt(6)
b := big.NewInt(7)
// no operator overloading:
z := a * b        // compile error
fmt.Println(a > b) // compile error
Right
a := big.NewInt(6)
b := big.NewInt(7)
z := new(big.Int).Mul(a, b) // 42
fmt.Println(a.Cmp(b))        // -1

Key Rules

  • No operators — use methods (Add, Sub, Mul, Div, Mod, Exp, Cmp); compare with Cmp returning -1/0/+1.
  • Methods write into the receiver and return it: z.Add(x, y) sets z; allocate a fresh new(big.Int) as the destination when you must preserve operands.
  • big.Int for exact integers, big.Rat for exact fractions (money, ratios), big.Float for tunable-precision reals.
  • Construct with big.NewInt(int64)/big.NewRat(a, b); parse arbitrary-length values with SetString(s, base).
Practice Exercise

Compute the 500th Fibonacci number with a rolling pair of big.Int (reuse a temp to avoid per-iteration allocations). Then sum a slice of prices represented as big.Rat cents and print the exact total with no rounding. Finally, write isProbablyPrime(decimal string) bool that uses SetString(s, 10) and ProbablyPrime, and explain why the aliasing form x.Add(x, y) is safe but sharing one big.Int across goroutines is a data race.

Pitfall: Forgetting that big-number methods mutate the receiver. total.Add(total, x) is fine, but a.Mul(a, b) silently overwrites a when you meant to keep it. Always pick a clear destination — often a fresh new(big.Int) — and never share one *big.Int between goroutines without synchronization, since a method call is a concurrent write.
Connections: The escape hatch when strconv-parsed values exceed 64 bits; Cmp mirrors the -1/0/+1 contract of cmp; underpins the crypto package's key math.

runtime/debug: Build Info & Runtime Control

Std Lib

Why it matters: runtime/debug is the standard-library control panel for a running Go program — capturing a stack trace when you recover from a panic, stamping your binary with its version and VCS revision (ReadBuildInfo), and tuning the garbage collector at runtime (SetGCPercent, SetMemoryLimit). These are the hooks production services use for observability and memory-limit safety.

Capture a stack trace on recover

debug.Stack() returns the current goroutine's trace as []byte. Grab it inside the recover handler — that's the only place the panic's stack is still available.

package main

import (
    "bytes"
    "fmt"
    "runtime/debug"
)

func main() {
    defer func() {
        if r := recover(); r != nil {
            trace := debug.Stack()
            first := bytes.SplitN(trace, []byte("\n"), 2)[0]
            fmt.Printf("recovered: %v\n", r)
            fmt.Printf("stack starts: %s\n", first)
        }
    }()
    panic("kaboom")
}
Output
recovered: kaboom stack starts: goroutine 1 [running]:

Tune the garbage collector at runtime

SetGCPercent trades memory for CPU (higher = fewer GCs); SetMemoryLimit sets a soft cap (Go 1.19+); each returns the previous value. FreeOSMemory forces a collection and hands freed pages back to the OS.

package main

import (
    "fmt"
    "runtime/debug"
)

func main() {
    prev := debug.SetGCPercent(200) // default is 100
    fmt.Println("old GOGC:", prev)

    old := debug.SetMemoryLimit(-1) // -1 reads without changing
    fmt.Println("mem limit was:", old)

    debug.FreeOSMemory() // force GC + return memory to OS
    fmt.Println("done")
}
Output
old GOGC: 100 mem limit was: 9223372036854775807 done

Common Mistakes

Wrong
defer func() {
    if r := recover(); r != nil {
        // no stack -> nearly useless in prod
        log.Println("panic:", r)
    }
}()
Right
defer func() {
    if r := recover(); r != nil {
        log.Printf("panic: %v\n%s",
            r, debug.Stack())
    }
}()

Key Rules

  • debug.Stack()/PrintStack() capture the calling goroutine's trace — call them inside a recover handler for a usable post-mortem.
  • debug.ReadBuildInfo() exposes module versions and the VCS revision — but only for a go build binary, not go run.
  • SetGCPercent and SetMemoryLimit tune the GC live and return the previous value; FreeOSMemory forces release of freed pages.
  • Prefer the GOGC/GOMEMLIMIT environment variables for deployment defaults; reach for the functions only when a value must change at runtime.
Practice Exercise

Write a SafeGo(fn func()) helper that runs fn in a goroutine, recovers any panic, and logs the value plus debug.Stack() so one crashing worker never takes down the process. Then add a -version flag that prints info.Main.Version and the vcs.revision setting from ReadBuildInfo (test it with go build, not go run). Finally, experiment: set debug.SetMemoryLimit low and observe GC frequency change under an allocation loop.

Pitfall: SetMemoryLimit is a soft limit — the runtime will exceed it rather than crash, and setting it too close to available memory makes the GC thrash (spinning near 100% CPU) trying to stay under the cap. Set it below your container's hard limit with headroom, and don't rely on it to prevent every OOM. Also note ReadBuildInfo returns empty VCS data under go run, so version stamping only works on built binaries.
Connections: The runtime companion to the defer/panic/recover flow and the pprof/GC-tuning performance topics; ReadBuildInfo pairs with build-time -ldflags for version stamping, and its stack capture complements structured logging with log/slog.
